Apple's retail analytics team labels every order with a frequency bucket — daily, weekly, or monthly — based on how often that customer typically buys. They want a customer-by-bucket grid: one row per customer, three columns of counts. This is the canonical pivot question, and the interviewer is watching whether you build the columns by hand with CASE or reach for a feature Postgres does not have.

1. The source data — orders

customer_idfreq_bucketorder_idamount
1daily110
1daily210
1weekly310
2monthly410
2weekly510
3daily610

Customer 1 spans two buckets (daily and weekly) — the bucket is a per-order label, not a per-customer one. Customer 3 has only a single order, so two of its three output cells will be 0. Those zeros are the point: a pivot must show 0, never a blank or a missing column.

2. The SQL — CASE pivot

SELECT customer_id,
  SUM(CASE WHEN freq_bucket='daily' THEN 1 ELSE 0 END)   AS daily,
  SUM(CASE WHEN freq_bucket='weekly' THEN 1 ELSE 0 END)  AS weekly,
  SUM(CASE WHEN freq_bucket='monthly' THEN 1 ELSE 0 END) AS monthly
FROM orders GROUP BY customer_id;

One GROUP BY customer_id collapses each customer to a single row; three conditional SUMs spread that row across three named columns.

3. The resulting analysis

1210
2011
3100

Each customer is now one row. Customer 1's three orders split 2 / 1 / 0; customer 3's single order is 1 / 0 / 0. The ELSE 0 is what guarantees a real zero in the empty cells — without it those cells would be NULL.

4. Strategic explanations

Why SUM, and why the ELSE 0 matters

The pattern is "fan one column into many": for each target column, a CASE emits 1 when the row belongs there and 0 otherwise, and SUM tallies. Drop the ELSE 0 and the CASE returns NULL for non-matching rows. SUM would still skip those NULLs, so a customer with no daily orders would get daily = NULL rather than 0. For a grid that downstream code adds up or charts, NULL is a landmine — always keep the ELSE 0.

There is no PIVOT in Postgres — the dialect trap

SQL Server and Oracle have a PIVOT operator; Postgres, MySQL, and SQLite do not. Reaching for PIVOT here is an instant dialect mismatch. The portable, universally-supported answer is exactly this conditional-aggregation idiom. Postgres also has crosstab() in the tablefunc extension, but it needs the extension installed and a rigid column definition — for three fixed buckets, hand-written CASE is simpler and clearer.

The follow-up: what if the buckets are not known in advance?

This query hardcodes three buckets. If a fourth (quarterly) appears, the query silently ignores it — no error, just a missing column. True dynamic pivots cannot be done in a single static SQL statement; you either generate the SQL programmatically or fall back to a long (un-pivoted) result:

SELECT customer_id, freq_bucket, COUNT(*) AS orders
FROM orders
GROUP BY customer_id, freq_bucket;

Saying "the wide form needs the bucket set frozen at query-write time; the long form survives new buckets" is the senior observation.

Dirty bucket values — the equality is fragile

freq_bucket='daily' is an exact match. A row with 'Daily', ' daily', or NULL matches none of the three CASEs and vanishes from every column — the customer's row stays, but their order count silently drops. Defend with LOWER(TRIM(freq_bucket)), and add a "catch-all" column or a HAVING sanity check that daily + weekly + monthly = COUNT(*) so unrecognised labels surface instead of leaking away.

Edge case: a customer with zero orders

GROUP BY only produces customers that appear in orders. A customer with no orders never shows up — usually fine. But if the grid must list every customer (including all-zero rows), start from the customers table and LEFT JOIN orders; the CASE/SUM then naturally yields a 0/0/0 row for the inactive customer.

Senior signal: recognise this as conditional aggregation, justify the ELSE 0 as the thing that produces real zeros instead of NULLs, name that Postgres has no PIVOT operator, and explain that a static CASE pivot silently drops any bucket value not hardcoded — offering the long-form result as the new-category-safe alternative.
