A gaming team wants each player's "first device" — the device they used on their earliest recorded session. The twist the interviewer plants: a player can have several sessions on that same earliest date, on different devices. The tie-break rule ("smallest device_id") is the whole point — it forces a deterministic ROW_NUMBER ordering rather than an ambiguous MIN.

1. The source data — activity

player_iddevice_idevent_date(narrative)

Player 2 is the edge case: two sessions, both on 2025-02-01, devices 30 and 40. "Earliest date" alone is a tie, so the rule "smallest device_id" decides — device 30 must win. Player 3 (a single session) tests the single-member-group path.

2. The SQL — ROW_NUMBER with a two-key ordering

WITH ranked AS (
  SELECT player_id, device_id, event_date,
         ROW_NUMBER() OVER (
           PARTITION BY player_id
           ORDER BY event_date, device_id
         ) AS rn
  FROM activity
)
SELECT player_id, device_id
FROM ranked
WHERE rn = 1
ORDER BY player_id;

PARTITION BY player_id restarts the numbering per player. ORDER BY event_date, device_id sorts each player's sessions earliest-date-first, then smallest-device-first; rn = 1 plucks the single winning row.

Player 2 returns device 30, not 40: both sessions share the earliest date, so the second sort key device_id breaks the tie toward the smaller value, exactly as the spec demands.

4. Strategic explanations

Why ROW_NUMBER, not RANK or MIN

The task wants exactly one row per player. ROW_NUMBER guarantees that — even a tie gets distinct numbers 1, 2, 3…, so rn = 1 is always a single row. RANK would give both of player 2's tied-date sessions rk = 1 (if the order only considered date) and return two rows. And a naive MIN(device_id) WHERE event_date = MIN(event_date)-style aggregate can be written, but it is clumsier and easy to get wrong; ROW_NUMBER with a compound ORDER BY states the tie-break declaratively.

The second ORDER BY key is the tie-break rule

ORDER BY event_date, device_id encodes two rules in priority order: first the earliest date, then — only among rows that tie on date — the smallest device_id. Drop the second key and player 2's result becomes non-deterministic: the engine could return device 30 or 40 on different runs. Any "earliest / latest, break ties by X" requirement must carry X as a trailing sort key, or the answer is silently unstable.

Why the CTE — you cannot filter a window function in WHERE

ROW_NUMBER() is computed after WHERE in SQL's logical evaluation order, so WHERE rn = 1 in the same query level is a syntax error — rn does not exist yet. The CTE (or a subquery) materialises rn as an ordinary column that the outer WHERE can then filter. This two-step shape — assign row numbers in a CTE, filter rn = 1 outside — is the standard "top-N-per-group" idiom.

NULL dates — the trap this clean data hides

Every row here has a real event_date, but production data may not. In Postgres ORDER BY event_date puts NULLs last by default for ascending order — so a player whose only session has a NULL date still gets rn = 1 on that row, which may or may not be intended. If a NULL date should never win, be explicit: ORDER BY event_date NULLS LAST documents intent, or filter WHERE event_date IS NOT NULL upstream. Naming this is the senior signal.

Senior signal: use ROW_NUMBER (not RANK) when you need exactly one row per group, encode the "break ties by smallest device" rule as a trailing ORDER BY key so the result is deterministic, wrap it in a CTE because window functions cannot be filtered in WHERE, and state how NULL dates sort.
