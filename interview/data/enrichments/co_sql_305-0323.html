Uber's ETA model is one of the most-watched ML systems in the company. Riders cancel when the ETA shifts; drivers route inefficiently when it's wrong. The query computes MAE (mean absolute error), MAPE (mean absolute percentage error), and bias per city.

2. The SQL — three error metrics in one pass

SELECT city,
  COUNT(*) AS rides,
  ROUND(AVG(ABS(predicted_minutes - actual_minutes)), 1) AS mae_minutes,
  ROUND(100.0 * AVG(ABS(predicted_minutes - actual_minutes) * 1.0
                  / NULLIF(actual_minutes, 0)), 1) AS mape_pct,
  ROUND(AVG(predicted_minutes - actual_minutes), 1) AS mean_bias_minutes
FROM trip_eta
GROUP BY city
ORDER BY mape_pct DESC;

The bias signal — over- vs under-promise

Bias = AVG(predicted - actual). Positive bias = ETAs longer than reality (rider waits less than told — pleasant surprise). Negative bias = ETAs shorter than reality (rider waits MORE than told — complaint trigger). Production targets +0.5 to +1 minute positive bias; pure zero is suspicious (model overfits validation).

The percentile follow-up

Mean error hides the catastrophic tail. The metric Uber actually monitors is P95 absolute error — 95% of rides have error below this number. A model with MAE=2min but P95=15min has a long tail of rider experience disasters. Use NTILE or window functions to estimate.

The traffic-condition confounder

Rush-hour ETAs are systematically harder. Splitting by daypart shows the model's actual struggle:

GROUP BY city, CASE WHEN CAST(strftime('%H', started_at) AS INT) BETWEEN 17 AND 20
  THEN 'evening_rush' ELSE 'offpeak' END
The retraining trigger

If MAPE drifts up > 10% over 4 weeks, the model retraining job fires. The query result feeds an alerting Airflow DAG, not just a dashboard.
