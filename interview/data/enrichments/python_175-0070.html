Meta frames "Koko Eating Bananas" as a capacity-planning question: given a deadline of h hours, what is the minimum throughput rate that still finishes the job? The insight the interviewer is hunting for is that you can binary-search the answer itself — the feasibility function is monotone, so the space of speeds is searchable even though it is not a sorted array.

1. The input data

piles = [3, 6, 7, 11]
h     = 8

Four piles, an 8-hour limit. At speed k, a pile of size p takes ceil(p / k) hours — Koko finishes a pile then idles the rest of that hour.

2. The approach

The candidate speeds form the range [1, max(piles)] — never slower than 1, never usefully faster than the biggest pile (one pile per hour is already maximal). Crucially, feasibility is monotone: if speed k finishes in time, so does every speed > k. That monotonicity is what licenses binary search.

The loop bisects on m. The hours needed are sum((p + m - 1) // m for p in piles) — the (p + m - 1) // m idiom is ceiling division without floats. If the total is <= h, speed m works, so m might be the answer — keep it by setting hi = m. Otherwise m is too slow, so lo = m + 1. The lo < hi loop converges on the smallest feasible speed (a lower-bound bisection), and lo is returned.

At speed 4: piles take 1, 2, 2, 3 hours = 8 total. Speed 3 needs 10 hours — too slow. So 4 is the minimum.

4. Strategic explanations

Complexity

O(n log M) where n is the pile count and M is max(piles) — the bisection does log M iterations, each summing over n piles. O(1) space. The brute-force "try every speed from 1 upward" is O(n · M) — the log factor is the entire payoff of the question.

The idiomatic alternative — math.ceil and bisect

(p + m - 1) // m can read as math.ceil(p / m) — clearer, at the cost of float division. More elegantly, the whole loop is a bisect over a predicate:

import bisect
class Feasible:
    def __getitem__(self, k):
        return sum(-(-p // k) for p in piles) <= h
lo = bisect.bisect_left(Feasible(), True, 1, max(piles))

The -(-p // k) trick is another ceiling-division idiom. Most seniors keep the explicit loop — it is more readable than abusing bisect — but naming "search on answer" out loud is the key move.

The edge case — tight and loose deadlines

If h == len(piles), every pile gets exactly one hour, so the answer is max(piles) — and the bisection correctly never goes below it. The problem guarantees h >= len(piles) (otherwise infeasible). A single pile reduces to ceil(pile / k) <= h, solved directly. lo starting at 1, not 0, avoids a division-by-zero in the rate computation.

The follow-up — "Capacity to Ship Packages in D Days"

The identical pattern appears in ship-capacity, "split array into k subarrays minimizing the max sum," and minimum-time scheduling problems. The interviewer wants to confirm you recognize the family: a monotone feasibility predicate plus binary search on the parameter.

The trap — float ceiling, and searching h instead of speed

Using math.ceil(p / m) with huge pile sizes risks float-precision error at the boundary — integer ceiling division is exact. The bigger trap is binary-searching the wrong axis: candidates sometimes try to search over hours, but hours are an output, not a tunable input. The searchable variable is the speed.

Senior signal: identify "binary search on the answer" because feasibility is monotone, justify the search bounds [1, max(piles)] rather than guessing them, and use exact integer ceiling division — then connect it to the ship-capacity / array-split family unprompted.
