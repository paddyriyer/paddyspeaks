A sales compensation system needs to split a fixed commission pool proportionally among salespeople based on their contribution to the team's revenue. The interviewer is probing whether you can combine a grouped aggregate CTE with a window-function running total over that CTE — a pattern that arises any time you need each entity's share of a partition-level total: budget allocation, cost attribution, revenue splits.

Note that salesperson 101 appears in both team 1 and team 2 — the question deliberately tests that such a person gets separate rows per team, not a merged aggregate. Team 3 has exactly one salesperson (105) and should receive 100 % of the pool.

2. The SQL — per-person CTE + window SUM for team total

WITH per_person AS (
  SELECT team_id, salesperson_id, SUM(amount) AS total_sales
  FROM sale
  GROUP BY team_id, salesperson_id
)
SELECT team_id,
       salesperson_id,
       total_sales,
       ROUND((10000.0 * total_sales / SUM(total_sales) OVER (PARTITION BY team_id))::numeric, 2)
         AS commission_share
FROM per_person
ORDER BY team_id, salesperson_id;

The CTE collapses all rows for the same (team_id, salesperson_id) pair into one total_sales figure. The outer SUM(total_sales) OVER (PARTITION BY team_id) then provides the team-level denominator without a second GROUP BY, preserving each salesperson as a separate row in the output.

4. Strategic explanations

Why a CTE then a window function — not a single query

You could write the denominator as a correlated subquery, but that executes once per output row. The two-step approach — aggregate first, then window-sum over the aggregated result — is cleaner and typically cheaper: the optimizer scans the base table once for the GROUP BY, then the window SUM scans the small CTE result rather than the raw fact table. It also keeps the formula readable: total_sales / SUM(total_sales) OVER (...) directly expresses the fraction.

The 10000.0 literal — type safety

If amount is NUMERIC the division is already exact. But when columns are integers, 10000 * total_sales / team_total can truncate to zero for small values. Writing 10000.0 forces a numeric promotion before division. The explicit ::numeric cast before ROUND is required in PostgreSQL because ROUND(double precision, n) does not exist — only ROUND(numeric, n) does.

Salesperson 101 across multiple teams

The GROUP BY is (team_id, salesperson_id), not just salesperson_id. That is what keeps person 101's team-1 row and team-2 row independent. If you accidentally grouped by salesperson only, you would merge their sales across teams and break the proportional calculation for both.

Single-member team edge case

When a team has exactly one salesperson, total_sales = SUM(total_sales) OVER (PARTITION BY team_id), so the share is exactly 10 000.00. Division by zero cannot occur here — a salesperson row implies at least one sale, so the team total is always positive.

Rounding and the pool sum

Rounding individual shares to 2 decimal places can leave the team total a few cents above or below 10 000.00. In production systems this is often handled by giving the residual cent(s) to the top earner ("largest-remainder method"). The question asks only for per-person rounded shares, so the solution is correct as written — but naming the rounding artefact in an interview demonstrates real-world awareness.

Dialect notes

The pattern is portable across PostgreSQL, Snowflake, BigQuery, and SQL Server. BigQuery uses ROUND(expr, 2) natively on FLOAT64; no cast is needed. Snowflake and SQL Server also handle the rounding without the ::numeric cast. MySQL 8+ supports window functions; in MySQL 5.7 or earlier you must fall back to a correlated subquery or a self-join for the team total.

Senior signal: use a GROUP BY CTE to produce one row per (team, person), then apply SUM(...) OVER (PARTITION BY team_id) on that CTE to get the team denominator without a second scan; cast to numeric before ROUND(…, 2) in PostgreSQL; and call out the largest-remainder rounding issue for production commission engines.
