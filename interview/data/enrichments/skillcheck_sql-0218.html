Counting someone’s direct referrals is a simple GROUP BY. Counting their whole referral network — everyone they brought in directly, plus everyone those people brought in, to any depth — needs recursion. A WITH RECURSIVE CTE walks the referral tree and rolls trip volume up to each referrer.

trips has 16 rows; trips per user are Ava 2, Ben 2, Cara 1, Dan 3, Eli 0, Fay 3, Gus 1, Hana 4. There are two referral trees: Ava’s (Ben, Cara, and everyone below) and Gus’s (Hana).

2. The recursive solution

WITH RECURSIVE downline AS (
  SELECT referred_by_id AS ancestor_id, user_id AS member_id
  FROM users
  WHERE referred_by_id IS NOT NULL
  UNION ALL
  SELECT d.ancestor_id, u.user_id
  FROM downline d
  JOIN users u ON u.referred_by_id = d.member_id
),
trips_per_user AS (
  SELECT user_id, COUNT(*) AS trip_count
  FROM trips
  GROUP BY user_id
)
SELECT a.user_name AS referrer,
       COUNT(*) AS downline_size,
       COALESCE(SUM(tpu.trip_count), 0) AS downline_trips
FROM downline d
JOIN users a ON a.user_id = d.ancestor_id
LEFT JOIN trips_per_user tpu ON tpu.user_id = d.member_id
GROUP BY a.user_id, a.user_name
ORDER BY downline_trips DESC, referrer ASC;

The downline CTE produces one row for every (ancestor, member) pair — every user paired with everyone beneath them in the referral tree. Trips are pre-aggregated per user, joined on the member, and summed up to the ancestor.

4. Strategic explanations

Anchor term, then recursive term

A recursive CTE has two halves joined by UNION ALL. The anchor — the first SELECT — seeds the working set with direct referrals: (referred_by_id, user_id) pairs. The recursive term — the second SELECT, which references downline itself — takes each known pair and reaches one level deeper, joining back to users to find who the member referred. The engine repeats the recursive term until it yields no new rows.

Why the recursion terminates — and when it would not

Each user has exactly one referred_by_id, so the referral graph is a tree: finite and acyclic. The recursive term eventually reaches leaves (users who referred no one), produces nothing, and stops. If the data allowed a cycle — A referred B who referred A — the recursion would never terminate; PostgreSQL would need a CYCLE clause or an explicit depth guard.

Pre-aggregate the fact table to avoid fan-out

trips has one row per trip. Joining it straight onto downline would multiply rows — a member with 3 trips would appear 3 times, inflating downline_size into a trip count rather than a head count. trips_per_user collapses trips to one row per user first, so each member contributes exactly one row: downline_size counts people, downline_trips sums their trips.

LEFT JOIN keeps members who took no trips

Eli took zero trips, so trips_per_user has no row for him. A LEFT JOIN keeps Eli in Ben’s downline — downline_size for Ben stays 2 — and COALESCE turns his missing trip sum into 0. An INNER JOIN would silently drop zero-trip members and undercount every network. Notice too that size and value diverge: Ava’s network is 5 people, Gus’s is 1, yet Gus’s lone referral out-rides Ben’s two.

Senior signal: a referral network is a tree — reach the whole of it with a WITH RECURSIVE CTE whose anchor seeds direct referrals and whose recursive term steps one level down. Pre-aggregate the fact table before joining to avoid fan-out, and LEFT JOIN it so zero-activity members still count toward network size.
