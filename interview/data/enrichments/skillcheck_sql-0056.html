A customer-marketing team wants to send milestone gifts: "as of 2025-05-14, which customers have a 5-, 10-, or 25-year signup anniversary landing in the next 30 days?" The interviewer is probing date anchoring — can you project a signup date forward by N years, handle the leap-day customer who signed up on 29 February, and filter against a rolling window?

Dan is the leap-day edge case: 2020-02-29 + 5 years is 2025-02-29, which does not exist. Gwen sits just outside the 30-day window. Both test the boundary precisely.

2. The SQL — cross join milestones, then anchor and filter

WITH base AS (
  SELECT id, name, signup_date, m AS milestone_years
  FROM customer, (VALUES (5),(10),(25)) AS milestones(m)
),
anniversaries AS (
  SELECT id, name, milestone_years,
         (signup_date + (milestone_years * INTERVAL '1 year'))::date AS anniversary_date
  FROM base
)
SELECT id, name, milestone_years, anniversary_date
FROM anniversaries
WHERE anniversary_date BETWEEN DATE '2025-05-14' AND DATE '2025-05-14' + 30
ORDER BY anniversary_date, id;

The VALUES (5),(10),(25) table cross-joined to customer produces three rows per customer — one per milestone. Each anniversary is the signup date plus that many year-intervals. The BETWEEN keeps anniversaries from the reference date through 30 days later.

4. Strategic explanations

Why the VALUES cross join

The rule has three milestone offsets and any customer might hit any of them. Rather than three near-identical UNION branches, the (VALUES (5),(10),(25)) AS milestones(m) inline table is cross-joined so each customer fans out into three candidate rows. Adding a 50-year milestone later is a one-token edit. This "small dimension as VALUES" pattern is the clean way to enumerate cases.

How leap-day arithmetic actually resolves

Dan signed up 2020-02-29. signup_date + INTERVAL '5 years' targets 2025-02-29, which does not exist, so PostgreSQL clamps to the last valid day — 2025-02-28. That matches the question's stated rule ("treat the anniversary as Feb 28"). This behaviour is engine-specific: do not assume it; verify it. The clamp is why Dan correctly falls out of a May window rather than throwing an error.

Adding years: INTERVAL vs raw integer

signup_date + (5 * INTERVAL '1 year') does calendar-aware year addition — it lands on the same month/day (modulo the leap clamp). Adding a raw integer instead (signup_date + 1825) adds days, drifting off the calendar date because of leap years in between. For anniversaries you want the calendar-aware INTERVAL arithmetic, then ::date to drop any time component.

The window boundary — inclusive both ends

BETWEEN a AND b is inclusive of both endpoints. The window is [2025-05-14, 2025-06-13]: an anniversary exactly on the reference date counts, and one on day 30 counts, but Gwen's day-32 anniversary does not. State the inclusivity explicitly — "next 30 days" is ambiguous about whether today and day 30 are in, and an off-by-one here changes who gets a gift. Using a half-open form (>= start AND < start + 31) is the safer idiom when the boundary matters.

Past milestones are correctly dropped

Eve's 5-year anniversary (2024-05-14) is in the past, so it falls below the window and is filtered out; her next is the 10-year in 2029. The single BETWEEN handles "only upcoming" for free — there is no need for a separate "is it in the future" clause, because anything earlier than the reference date fails the lower bound.

Senior signal: enumerate the milestone offsets as an inline VALUES table cross-joined to customers, add years with calendar-aware INTERVAL arithmetic (not raw day counts), know that PostgreSQL clamps a non-existent Feb 29 to Feb 28, and state the BETWEEN window's inclusivity explicitly so the 30-day boundary is unambiguous.
