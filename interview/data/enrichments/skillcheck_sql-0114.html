A workflow engine declares that each task has subtasks numbered 1..subtask_count, and a log records which subtasks actually ran. The goal is the gap report: which expected subtasks never executed. The interviewer adds a constraint — generate the universe of expected subtasks with a recursive CTE, not generate_series — to test whether you can build a counted sequence by hand and then anti-join it.

1. The source data — tasks and executed

131, 2, 3

341, 2, 3, 4
task_idsubtask_id(narrative)

The instructive edge case is task 3: subtask 3 executed but subtask 2 did not, so the missing set is non-contiguous (2 and 4). The solution cannot assume executions are a prefix — it must compare the full expected universe against the actual log, slot by slot.

2. The SQL — recursive countdown CTE, then anti-join

WITH RECURSIVE expand AS (
  SELECT task_id, subtask_count AS subtask_id FROM tasks
  UNION ALL
  SELECT task_id, subtask_id - 1 FROM expand WHERE subtask_id > 1
)
SELECT e.task_id, e.subtask_id
FROM expand e
LEFT JOIN executed x
  ON x.task_id = e.task_id AND x.subtask_id = e.subtask_id
WHERE x.task_id IS NULL
ORDER BY e.task_id, e.subtask_id;

The recursive CTE seeds each task at its highest subtask number, then the recursive arm subtracts 1 each step, stopping at 1 — producing every id from subtask_count down to 1. The LEFT JOIN + WHERE x.task_id IS NULL is an anti-join: it keeps expected subtasks with no matching row in executed.

Task 3 correctly yields two rows, 2 and 4 — the anti-join catches non-contiguous gaps because it tests each expected slot independently rather than assuming a prefix ran. The output is ordered by task then subtask.

4. Strategic explanations

How the recursive CTE manufactures the 1..n universe

A recursive CTE has two arms joined by UNION ALL: the anchor (the seed) and the recursive arm (which references the CTE itself). Here the anchor emits one row per task at subtask_id = subtask_count. The recursive arm takes each current row and emits subtask_id - 1, but only WHERE subtask_id > 1 — that predicate is the termination condition. Iteration stops when every branch has counted down to 1. The result is the complete expected set per task.

Why the WHERE subtask_id > 1 guard is non-negotiable

Drop it and the recursive arm keeps subtracting forever — 0, -1, -2 — an infinite loop the engine eventually aborts. Every recursive CTE must have a recursive arm that provably produces fewer rows each step and a predicate that eventually yields the empty set. Counting toward a fixed bound (1) is the textbook safe pattern. State the termination argument explicitly; interviewers want to hear it.

The anti-join — LEFT JOIN ... IS NULL vs NOT IN vs NOT EXISTS

"Expected but not executed" is a set difference. The LEFT JOIN keeps every expected row and attaches the matching execution if one exists; rows where the right side is all NULL had no match, so WHERE x.task_id IS NULL isolates the gaps. NOT EXISTS is equally correct and often clearer. NOT IN is the dangerous one: if the executed subquery ever yielded a NULL, NOT IN returns unknown for everything and the result set collapses to empty. Prefer LEFT JOIN/NOT EXISTS for anti-joins.

Why generate_series was banned — and the trade-off

In Postgres the easy answer is SELECT t.task_id, g FROM tasks t, generate_series(1, t.subtask_count) g. It is shorter and the planner handles it well. The interviewer bans it because generate_series is Postgres-specific (SQL Server has no equivalent; you would use a numbers table or a recursive CTE there anyway). The recursive CTE is the portable sequence generator — knowing it means you can solve this on any engine that supports WITH RECURSIVE.

Dialect notes

WITH RECURSIVE is standard SQL — PostgreSQL, SQLite, MySQL 8+, and Oracle (which also accepts the keyword) support it. SQL Server supports recursive CTEs but without the RECURSIVE keyword (just WITH) and caps recursion at 100 levels unless you add OPTION (MAXRECURSION n). For tasks with thousands of subtasks, raise that limit or revert to a numbers table.

Senior signal: generate a counted 1..n universe with a recursive CTE (anchor + decrementing recursive arm + a > 1 termination guard), find the gaps with a LEFT JOIN ... IS NULL anti-join that catches non-contiguous misses, avoid NOT IN against a possibly-NULL subquery, and know the recursive CTE is the portable stand-in for generate_series.
