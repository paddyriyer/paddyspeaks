Vercel stores each user's interests as a single comma-separated string — tags_csv — and the growth team wants a tag-frequency table: how many users carry each tag. A CSV column is a denormalisation smell, and the fix is to split it into rows and aggregate. The query is compact, but the trimming and the empty-segment filter are doing real defensive work that an interviewer will probe.

1. The source data — users

user_idtags_csv
1sql, python, data
2python,ml
3sql,  ,data
4(empty string)
5data, sql

Three deliberate messes: user 1 has leading spaces after each comma (" python"), user 3 has an empty middle segment (sql,  ,data — a stray comma), and user 4's string is completely empty. Real CSV columns always look like this.

2. The SQL — UNNEST(string_to_array(...))

SELECT trim(tag) AS tag, COUNT(*) AS users
FROM users, UNNEST(string_to_array(tags_csv, ',')) AS tag
WHERE trim(tag) <> ''
GROUP BY trim(tag);

string_to_array splits the CSV on commas into a Postgres array; UNNEST in the FROM clause is a lateral set-returning function that expands the array to one row per element, joined back to its user. trim strips whitespace; the WHERE drops empties.

3. The resulting analysis

tagusers
ml1
data3
sql3
python2

sql and data each appear for 3 users, python for 2, ml for 1. User 4 (empty string) and user 3's stray comma contribute nothing — the WHERE trim(tag) <> '' swept them out. Result rows are unordered: no ORDER BY was given.

4. Strategic explanations

Why trim appears — and why it appears twice

string_to_array('sql, python', ',') yields {'sql', ' python'} — the space after the comma is preserved. Without trim, 'python' and ' python' are different tags and the count fragments. Note trim(tag) is written three times (SELECT, WHERE, GROUP BY) and all three must agree, or you group by one expression and display another. Cleaner: trim once in a subquery or LATERAL, then reference the clean column:

SELECT tag, COUNT(*) AS users
FROM users
CROSS JOIN LATERAL UNNEST(string_to_array(tags_csv, ',')) AS raw(t)
CROSS JOIN LATERAL (SELECT trim(raw.t) AS tag) cleaned
WHERE tag <> ''
GROUP BY tag;
The empty-string and NULL edge cases

User 4's empty string is the subtle one. string_to_array('', ',') returns {''} — an array with one empty element, not an empty array — so UNNEST still emits one row, with an empty tag. The WHERE trim(tag) <> '' is what discards it. A genuine NULL tags_csv behaves differently again: string_to_array(NULL, ',') is NULL, UNNEST(NULL) yields zero rows, and that user drops out silently — usually fine here, but worth stating.

The comma-in-the-form trap

This whole approach assumes commas only ever separate tags. The moment a tag itself contains a comma — or the delimiter is inconsistent (some rows comma-, some semicolon-separated) — string_to_array shreds it wrongly. There is no quoting in a raw CSV column. The honest senior answer is that storing multi-valued data as a delimited string is the actual bug; the durable fix is a user_tags(user_id, tag) junction table, which makes this query a trivial GROUP BY with no parsing at all.

Counting users vs counting occurrences — the dedup question

COUNT(*) counts tag occurrences. If one user's CSV were "sql,sql" (a duplicate within the string), they would be counted twice for sql — almost never what "how many users have this tag" means. Make the dedup explicit:

SELECT tag, COUNT(DISTINCT user_id) AS users
FROM users, UNNEST(string_to_array(tags_csv, ',')) AS tag_raw(tag_raw)
CROSS JOIN LATERAL (SELECT trim(tag_raw) AS tag) c
WHERE tag <> ''
GROUP BY tag;

On clean data COUNT(*) and COUNT(DISTINCT user_id) agree; on dirty data only the second answers the question asked.

Case and ordering polish

SQL, Sql, and sql would count as three tags — fold case with lower(trim(tag)) if the taxonomy is case-insensitive. And the result has no ORDER BY, so the rows come back in an arbitrary order; add ORDER BY users DESC, tag so the most popular tag leads and ties break deterministically.

Senior signal: know that string_to_array keeps post-delimiter whitespace and empty segments, justify the trim and <> '' guards, distinguish the empty-string (one empty element) from NULL (zero rows) cases, use COUNT(DISTINCT user_id) against intra-string duplicates, and name the junction-table redesign as the real fix.
