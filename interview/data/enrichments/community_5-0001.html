A Community marketing analyst is asked the weekly question: which campaigns are pulling their weight? The cut-off is "more than 100 conversions". It looks like a one-line GROUP BY — but the interviewer is watching whether you count off the right table, whether you use a LEFT JOIN for a reason, and whether you know that HAVING and WHERE are not interchangeable.

1. The source data — conversions (joined to campaigns)

campaign_nameconversion rowsnote
Spring Launch150over the line
Summer Sale220over the line
Quiet Drip30below 100 — excluded
Fall Push110just over

"Quiet Drip" has real conversions but only 30 of them — it is the row the HAVING clause must drop. "Fall Push" at 110 is the edge case: a strict > 100 keeps it; a hypothetical >= 100 would also keep an exactly-100 campaign, so the boundary wording matters.

2. The SQL — GROUP BY + HAVING COUNT

SELECT
  c.campaign_name,
  COUNT(co.conversion_id) AS conversion_count
FROM conversions co
LEFT JOIN campaigns c ON co.campaign_id = c.campaign_id
GROUP BY c.campaign_name
HAVING COUNT(co.conversion_id) > 100;

Conversions is the driving table; each row is one conversion. Grouping by campaign_name collapses them per campaign, and HAVING filters the groups — not the rows — down to the busy ones.

3. The resulting analysis

campaign_nameconversion_count
Fall Push110
Summer Sale220
Spring Launch150

Three campaigns clear the bar; "Quiet Drip" with 30 conversions is gone. Note the result is unordered — there is no ORDER BY, so the engine returns groups in whatever order is convenient. If a report needs Summer Sale on top, that is a missing ORDER BY conversion_count DESC, not a bug in the aggregation.

4. Strategic explanations

Why HAVING and not WHERE

WHERE filters individual rows before grouping; HAVING filters groups after aggregation. You cannot put WHERE COUNT(*) > 100 — the count does not exist until rows are grouped. The pipeline is fixed: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. Per-row predicates (a date range, a status) belong in WHERE; predicates on an aggregate belong in HAVING.

The LEFT JOIN vs INNER JOIN question

Here the join direction barely matters — every conversion references a real campaign, and the HAVING > 100 threshold would discard any campaign with zero conversions anyway. But the interviewer may flip it: "list campaigns with FEWER than 100 conversions, including those with none." Then you must drive from campaigns and LEFT JOIN conversions, because a campaign with zero conversions has no rows on the conversion side:

SELECT c.campaign_name, COUNT(co.conversion_id) AS conversion_count
FROM campaigns c
LEFT JOIN conversions co ON c.campaign_id = co.campaign_id
GROUP BY c.campaign_name
HAVING COUNT(co.conversion_id) < 100;

The subtle part: COUNT(co.conversion_id) counts non-NULL values, so a campaign with no matching conversions correctly counts 0. Writing COUNT(*) there would count the one all-NULL row the LEFT JOIN manufactures — giving 1 instead of 0. Count the column, not the star, whenever an outer join is involved.

The grouping-key trap — group by id, not name

Grouping by campaign_name is fine only because names are unique here. If two campaigns shared a name (re-run of "Spring Launch" next year), GROUP BY campaign_name would silently merge them. The robust form groups by the primary key and carries the name along:

GROUP BY c.campaign_id, c.campaign_name

In strict-GROUP BY engines (Postgres) you must list every non-aggregated SELECT column anyway; MySQL's looser default lets a bare campaign_name slip through, which is exactly how the duplicate-name bug hides.

Edge case: counting conversions in a date window

"Conversions" often means conversions within the campaign's run dates, not all-time. If a conversion can land outside start_date–end_date, add a per-row filter in WHERE before the count: WHERE co.conversion_date BETWEEN c.start_date AND c.end_date. Confirm the definition — "100 conversions ever" and "100 conversions during the campaign" are different metrics.

Senior signal: place per-row predicates in WHERE and aggregate predicates in HAVING without hesitation, count the joined column (not COUNT(*)) so outer-join NULLs read as zero, group by the primary key to survive duplicate names, and ask whether the threshold is all-time or windowed before writing the query.
