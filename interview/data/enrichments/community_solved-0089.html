A logistics platform stores city adjacency as directed rows: each city points at one neighbour with a recorded distance. The ask is to surface only the mutual pairs — A names B as its neighbour, B names A back, and both rows agree on the distance. The interviewer is probing whether you can self-join a table to itself and express a bidirectional relationship, rather than naively returning every one-way edge.

Chennai ↔ Bengaluru is the edge case: they point at each other, but Chennai records 300 km while Bengaluru records 350 km. The relationship is reciprocal yet the distance disagrees — a correct query must drop this pair.

2. The SQL — self-join on a reciprocal condition

SELECT src.cityname AS source_city,
    dest.cityname AS destination_city,
    src.distance AS distance
FROM cities AS src
LEFT JOIN cities AS dest
  ON src.neighborcityid = dest.cityid
 AND src.cityid = dest.neighborcityid
WHERE src.distance = dest.distance;

Chennai/Bengaluru is correctly absent — the join matches them, but WHERE src.distance = dest.distance fails because 300 ≠ 350. Note every surviving pair appears twice, once per direction.

4. Strategic explanations

Why the self-join works

Aliasing one table as src and dest lets you treat each row's neighbour pointer as a lookup into a second copy. The two-part ON clause encodes "reciprocal": src.neighborcityid = dest.cityid finds the city src points to, and src.cityid = dest.neighborcityid demands that city point back. Without the second predicate you would return every one-way edge.

The LEFT JOIN here is effectively an INNER JOIN

The query writes LEFT JOIN but then filters WHERE src.distance = dest.distance. Any unmatched src row gets dest.distance = NULL, and 150 = NULL evaluates to unknown, so the row is dropped — the same outcome as an INNER JOIN. A WHERE predicate on the right-side table almost always collapses an outer join back to an inner one. Writing INNER JOIN states the intent honestly.

The duplicate-rows trap

Each mutual pair appears in both orientations (Mumbai→Pune and Pune→Mumbai). If the spec wants each unordered pair once, add AND src.cityid < dest.cityid to the join — that keeps only the orientation where the smaller id is the source and halves the output.

NULL neighbour pointers

A city with no neighbour (neighborcityid IS NULL) never matches anything: NULL = dest.cityid is unknown. With the WHERE filter that row simply vanishes, which is correct here, but it is worth saying aloud — an interviewer wants to hear that you considered the islands in the graph.

Senior signal: name this a self-join expressing a bidirectional edge, point out that the WHERE clause on dest turns the LEFT JOIN into an effective INNER JOIN, and pre-empt the duplicate-pair problem with src.cityid < dest.cityid when only one row per unordered pair is wanted.
