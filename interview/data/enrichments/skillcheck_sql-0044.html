A payments reconciliation team needs a per-order ledger: how much was billed, how much actually arrived, how much was sent back, and whether the order balances. Money lives across three tables — orders, payments, refunds — and an order can have many payment and many refund rows, or none. The interviewer is probing fan-out awareness: join the three tables naively and you double-count. The safe pattern is aggregate-then-join.

1. The source data — orders, payments, refunds

tablerows(narrative)
orders(1,100,50) (2,100,30) (3,101,80) (4,102,25) (5,102,100)order 4 will have no payments & no refunds
payments(1,50) (2,15) (2,15) (3,80) (3,5) (5,100)order 2 & 3 have multiple payments; order 3 overpays
refunds(3,20) (5,30) (5,70)order 5 has two refund rows summing to its full amount

The edge cases are deliberate: order 4 appears in neither child table (must still show with zeros), order 3 has multiple payments and a refund, and order 5 is fully paid then fully refunded. A direct three-way join would multiply order 3's 2 payments by its 1 refund into 2 rows — corrupting every sum.

2. The SQL — pre-aggregate each side, then LEFT JOIN

WITH p AS (
  SELECT order_id, COALESCE(SUM(paid_amount),0) AS total_paid
  FROM payments GROUP BY order_id
),
r AS (
  SELECT order_id, COALESCE(SUM(refund_amount),0) AS total_refunded
  FROM refunds  GROUP BY order_id
)
SELECT o.order_id,
       o.amount AS order_amount,
       COALESCE(p.total_paid, 0)      AS total_paid,
       COALESCE(r.total_refunded, 0)  AS total_refunded,
       COALESCE(p.total_paid, 0) - COALESCE(r.total_refunded, 0) AS net,
       CASE
         WHEN COALESCE(p.total_paid,0) > o.amount THEN 'overpaid'
         WHEN COALESCE(p.total_paid,0) < o.amount THEN 'underpaid'
         WHEN COALESCE(r.total_refunded,0) > 0    THEN 'refunded'
         ELSE 'paid'
       END AS status
FROM orders o
LEFT JOIN p USING (order_id)
LEFT JOIN r USING (order_id)
ORDER BY o.order_id;

Each child table is collapsed to one row per order in its own CTE before any join happens. orders then LEFT JOINs those pre-aggregated CTEs, so every order survives and each contributes exactly one row. COALESCE converts the NULLs from unmatched LEFT JOINs into 0.

4. Strategic explanations

The fan-out trap — why you cannot join all three raw

orders JOIN payments JOIN refunds produces a Cartesian product within each order: order 3's 2 payments paired with its 1 refund give 2 rows, and SUM(paid_amount) over those 2 rows is correct by luck — but SUM(refund_amount) would count the 20 twice (40). With order 5's 2 payments and 2 refunds it is worse: 4 rows, both sums doubled. Pre-aggregating each child to one-row-per-order before joining is the only way each total stays honest.

Why LEFT JOIN — "return EVERY order" is a hard requirement

An INNER JOIN to payments would silently drop order 4 (no payment rows). The spec says every order must appear, so orders is the driving table and both child CTEs hang off it via LEFT JOIN. The unmatched side yields NULL, which COALESCE(..., 0) turns into a real zero so arithmetic and the CASE behave.

Two layers of COALESCE — they guard different things

The COALESCE inside the CTEs (COALESCE(SUM(...),0)) is technically redundant: GROUP BY order_id only emits a row when at least one child row exists, so SUM is never NULL there. The COALESCE in the outer query is the one that matters — it catches orders that produced no CTE row at all (order 4) and the LEFT JOIN handed back NULL. Knowing which guard does the real work is the senior read.

CASE branch order encodes the business priority

The CASE tests overpaid and underpaid first, so an amount mismatch always wins over refund state — order 3 is overpaid even though it has a refund. Only when paid equals amount do the last two branches distinguish refunded (refunds > 0) from clean paid. CASE is first-match; reorder the branches and the labels change. The spec's wording — "'paid' if equal AND refunds = 0" — dictates this exact ordering.

Dialect notes — USING and the alternative shapes

JOIN ... USING (order_id) is standard and works in Postgres, MySQL, BigQuery, and SQLite; SQL Server has no USING — use ON o.order_id = p.order_id. The aggregate-then-join pattern can equivalently be written with correlated scalar subqueries ((SELECT COALESCE(SUM(paid_amount),0) FROM payments WHERE order_id = o.order_id)), which sidesteps fan-out by construction but can be slower without good indexing. Both are valid; the CTE form is usually clearer and the planner-friendlier choice.

Senior signal: spot that joining orders to two one-to-many child tables fans out and corrupts sums, so pre-aggregate each child to one row per order before the join, drive with LEFT JOIN + COALESCE so zero-activity orders still surface, and order the CASE branches to match the stated business priority.
