TikTok's growth team measures activation: the share of new users who do something meaningful within their first 7 days. The query left-joins users to activity with a date condition, then divides distinct activated users by distinct total users. The interviewer is watching one subtle, decisive thing — that the 7-day window lives in the ON clause, not the WHERE clause — and the integer-division guard.

Four users. User 1 acted on day 3 (inside the 7-day window), user 3 on day 5 (inside). User 2 acted on day 20 (outside — activity, but too late to count as activation). User 4 never acted. So 2 of 4 activated. User 2 is the trap: they have an activity row, but it must not count.

2. The SQL — LEFT JOIN with dated ON

SELECT 100.0 * COUNT(DISTINCT a.user_id) / COUNT(DISTINCT u.user_id) AS activation_rate
FROM users u LEFT JOIN activity a
  ON u.user_id=a.user_id AND a.action_date <= (u.signup_date + INTERVAL '+7 days');

The join keeps every user; the ON clause matches an activity row only if it belongs to the user and falls within 7 days of signup. The ratio counts distinct activated users over distinct total users.

3. The resulting analysis

activation_rate
50.0000000000000000

Numerator: COUNT(DISTINCT a.user_id) = 2 (users 1 and 3). User 2's late activity row joined but the date condition failed, so a.user_id is NULL for that pairing — and COUNT(DISTINCT) ignores NULLs. User 4 had nothing to join. Denominator: 4 total users. 100.0 × 2 / 4 = 50%.

4. Strategic explanations

The whole question: the date filter must be in ON, not WHERE

This is the trap the interviewer set. With a LEFT JOIN, a condition in the ON clause decides whether a row matches; a non-matching user is still kept, just with NULLs on the activity side. Move that same condition to WHERE and it filters the joined result — and because un-activated users have a.action_date = NULL, the predicate NULL <= date is unknown, so those users are dropped entirely:

-- WRONG: condition in WHERE silently turns the LEFT JOIN into an INNER JOIN
SELECT 100.0 * COUNT(DISTINCT a.user_id) / COUNT(DISTINCT u.user_id)
FROM users u LEFT JOIN activity a ON u.user_id = a.user_id
WHERE a.action_date <= u.signup_date + INTERVAL '7 days';
-- denominator becomes 2, not 4 -> reports 100%, wildly wrong

User 4 vanishes from the denominator, and the rate jumps from 50% to 100%. Filter the right side of a LEFT JOIN in ON; reserve WHERE for the left side. Being able to articulate this is the entire senior signal here.

The 100.0 dodges integer division

COUNT(DISTINCT ...) returns a bigint on both sides. 2 / 4 in integer arithmetic is 0, not 0.5 — every activation rate below 100% would collapse to 0. Leading with the numeric literal 100.0 promotes the expression to numeric so the division keeps its fraction. Equivalent guards: CAST(... AS numeric), or ... * 1.0.

Window-boundary semantics — is day 7 in or out?

action_date <= signup_date + 7 days makes the window inclusive of the 7th day, and <= also counts an action on the signup day itself (day 0). Is signup-day activity "activation", or must it be a later action? And should it be the first 7 calendar days or the first 7×24 hours? If action_date is a timestamp, date arithmetic can clip a few hours. Pin the boundary down rather than letting <= default it.

The "first action" subtlety — COUNT(DISTINCT) already handles it

A user with three actions inside the window produces three joined rows, but COUNT(DISTINCT a.user_id) collapses them to one — correct, activation is per-user not per-action. Drop the DISTINCT and a hyperactive user would be counted many times and the rate could exceed 100%. The DISTINCT on both counts is essential, not decorative.

Edge case: zero users

If users is empty, the denominator COUNT(DISTINCT u.user_id) is 0 and the query throws division by zero. A cohort query that might run on an empty segment should guard it: / NULLIF(COUNT(DISTINCT u.user_id), 0), which yields NULL ("no cohort") instead of crashing.

Senior signal: explain precisely why the 7-day predicate belongs in ON — moving it to WHERE demotes the LEFT JOIN to an inner join and silently shrinks the denominator — keep DISTINCT on both counts, force numeric division, and guard the empty-cohort divide-by-zero with NULLIF.
