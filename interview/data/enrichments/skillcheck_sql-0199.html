Reporting dashboards must show every time-period for every dimension, even when the underlying data has gaps. A naive GROUP BY silently drops (region, month) combinations with no sales. The interviewer is probing whether you can build a complete grid by CROSS JOIN-ing the dimension set against a generated date spine, then LEFT JOINing the data — a pattern called "sparse-to-dense" or "date spine join".

1. The source data — sales

regionsold_onamount
North2025-01-10100
North2025-03-0540
South2025-02-2070

Two regions, three months. Missing combinations: North-February (no sales), South-January, South-March — all three must appear with total = 0, not be absent.

2. The SQL — CROSS JOIN of regions × date spine, then LEFT JOIN sales

SELECT
  r.region,
  gs::date AS month_start,
  COALESCE(SUM(s.amount), 0) AS total
FROM (SELECT DISTINCT region FROM sales) r
CROSS JOIN generate_series(DATE '2025-01-01', DATE '2025-03-01', INTERVAL '1 month') AS gs
LEFT JOIN sales s
  ON s.region = r.region
 AND date_trunc('month', s.sold_on) = gs
GROUP BY r.region, gs
ORDER BY r.region, month_start;

generate_series produces three month-start dates (Jan 1, Feb 1, Mar 1). The CROSS JOIN with the two distinct regions yields 6 (region, month) combinations. The LEFT JOIN attaches matching sales rows; where there are none, SUM(s.amount) aggregates over no rows and returns NULL, which COALESCE(..., 0) converts to 0.

4. Strategic explanations

Why CROSS JOIN + LEFT JOIN, not just GROUP BY

A plain SELECT region, DATE_TRUNC('month', sold_on), SUM(amount) FROM sales GROUP BY 1,2 returns only 3 rows — the (region, month) pairs that actually have data. The dashboard consumer sees North with no February entry and assumes data is missing or the report is broken. The CROSS JOIN creates the full grid of expected cells up front; the LEFT JOIN then fills in data where it exists and leaves NULLs (zeroed by COALESCE) where it does not. This is the canonical "sparse-to-dense" pattern.

generate_series as the date spine

generate_series(start, stop, interval) is a PostgreSQL set-returning function (SRF) that produces one row per step. Here it yields 2025-01-01, 2025-02-01, 2025-03-01. The gs::date cast converts the default TIMESTAMPTZ output to plain DATE. If the date range were dynamic (e.g., derived from MIN/MAX of sold_on), you would pass those expressions as the start and stop arguments. In BigQuery the equivalent is GENERATE_DATE_ARRAY; in SQL Server, a recursive CTE or a numbers table; Snowflake has GENERATOR().

Matching sales to the spine with date_trunc

The JOIN predicate date_trunc('month', s.sold_on) = gs maps each sale to its month-start bucket. This is equivalent to s.sold_on >= gs AND s.sold_on < gs + INTERVAL '1 month' but more concise. Either form works; the range predicate is friendlier to an index on sold_on. Using date_trunc in a JOIN condition typically prevents index use unless there is a functional index on date_trunc('month', sold_on).

COALESCE on a SUM, not on a column

COALESCE(SUM(s.amount), 0) wraps the aggregate, not the individual column. When no sales rows join, SUM over an empty set returns NULL (SQL standard); COALESCE converts it to 0. A common mistake is writing SUM(COALESCE(s.amount, 0)) — that correctly handles NULL amount values in the source but still returns 0 only if there is at least one joined row; an empty join set still gives a NULL SUM. You need both layers if the source contains NULLable amounts and the grid must show 0 for missing months.

Dialect and portability notes

generate_series is PostgreSQL and DuckDB. BigQuery: GENERATE_DATE_ARRAY('2025-01-01', '2025-03-01', INTERVAL 1 MONTH) inside an UNNEST. SQL Server: recursive CTE or a calendar dimension table. Snowflake: DATEADD inside a GENERATOR. Redshift: generate_series is available. Knowing the alternatives is essential because "give me a date spine" is a recurring pattern in any data warehouse.

Senior signal: build the complete (dimension × time) grid with CROSS JOIN generate_series(…) before touching the data table, then use LEFT JOIN + COALESCE(SUM(…), 0) to fill gaps with zero; explain why a plain GROUP BY silently drops absent cells; and name the equivalent spine-generation function in BigQuery, SQL Server, and Snowflake.
