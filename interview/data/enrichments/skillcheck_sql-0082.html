A retention analyst needs the classic month-over-month retention number: of the users who were "monthly active" in June, how many were also active in July? There is a twist in the activity definition — a post action does not count. The interviewer is probing whether you can express a cohort overlap (active in both periods) and whether you apply the activity rule consistently to both months.

1. The source data — events

The two post rows are the trap. User 2 did something in July, but it was a post, so they are not July-active. User 5 was present in June, but only via a post, so they are not June-active — their July sign-in cannot count as retention because they were never in the June cohort.

2. The SQL — build a user-month activity set, then self-overlap

WITH active AS (
  SELECT DISTINCT user_id,
         DATE_TRUNC('month', event_date)::date AS month_start
  FROM events
  WHERE action <> 'post'
)
SELECT 7 AS month,
       COUNT(DISTINCT j.user_id) AS mau,
       COUNT(DISTINCT j.user_id) FILTER (
         WHERE j.user_id IN (SELECT user_id FROM active WHERE month_start = DATE '2025-06-01')
       ) AS retained
FROM active j
WHERE j.month_start = DATE '2025-07-01';

The active CTE applies the action <> 'post' rule once and reduces the log to distinct (user, month) pairs. The outer query restricts to July rows for mau, and the FILTER clause re-counts only those July-active users who also appear in the June slice of the same CTE.

3. The resulting analysis

752

July's active set is {1, 3, 4, 5, 6} — five users (user 2 excluded, July action was a post). June's active set is {1, 2, 4} (user 5 excluded, June action was a post). The overlap is {1, 4} — retained = 2. User 5 illustrates the subtlety perfectly: July-active, but not in the June cohort, so not "retained."

4. Strategic explanations

Why a normalised activity CTE first

The active CTE does the conceptual heavy lifting: it applies the activity rule (action <> 'post') exactly once and de-duplicates to one row per user per month. Every downstream count then just reads this clean set. If you instead filtered post separately for June and for July, you risk applying the rule inconsistently — the single CTE guarantees both months use identical semantics, which is precisely what the interviewer is checking.

FILTER vs CASE inside the aggregate

COUNT(...) FILTER (WHERE ...) is the SQL-standard way to count a conditional subset. The portable equivalent is COUNT(DISTINCT CASE WHEN ... THEN user_id END) — CASE returns the id when the user is in June's cohort and NULL otherwise, and COUNT ignores NULLs. Both compute the overlap; FILTER reads more clearly and is the right idiom in PostgreSQL.

<> 'post' and the NULL-action hazard

action <> 'post' keeps every action except post — but if action could ever be NULL, the predicate evaluates to UNKNOWN and that row is dropped, even though a NULL action is not a post. If NULL actions should count as activity, write action IS DISTINCT FROM 'post', which treats NULL as "not equal to post" and keeps the row. Naming this is a senior reflex around any <> filter.

Retention is directional — June ∩ July, not just July

"Retained" is defined relative to a base cohort: it is the intersection of June-active and July-active. A user active only in July (users 3 and 6) is an acquisition, not a retention. The query encodes direction by anchoring on July rows and filtering to membership in the June set. If the question wanted "resurrected" users (active in June, dormant in some month, back in July) the logic would differ — always confirm which cohort relationship is being measured.

Dialect notes

DATE_TRUNC('month', ...) and the FILTER clause are PostgreSQL (and standard) syntax. MySQL lacks FILTER — use the COUNT(DISTINCT CASE WHEN ...) form — and would bucket months with DATE_FORMAT(event_date, '%Y-%m-01'). BigQuery uses DATE_TRUNC(event_date, MONTH) and also supports COUNTIF for the conditional count.

Senior signal: normalise the event log into one clean (user, month) activity set in a CTE so the post-exclusion rule is applied identically to both months, compute retention as the intersection of the June and July cohorts (not just July activity), and guard the <> 'post' filter against NULL actions with IS DISTINCT FROM.
