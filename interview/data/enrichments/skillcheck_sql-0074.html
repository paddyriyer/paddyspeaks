A messaging platform wants its two most prolific senders for August 2025 — a leaderboard query. The catch: if several senders tie at the second spot, all of them must appear. The interviewer is probing whether you reach for DENSE_RANK rather than LIMIT 2 or ROW_NUMBER, because only ranking-with-ties returns "all tied users" without arbitrarily truncating the list.

1. The source data — messages

The scenario is engineered to break naive solutions. Three senders (10, 11, 14) tie at 4 messages. Sender 13's only messages are in July, so the date filter must zero them out — they should not appear at all. The genuine "rank 2" sender is 12, with 3 messages, sitting directly below a three-way tie.

2. The SQL — count in range, DENSE_RANK, keep rank ≤ 2

WITH cnt AS (
  SELECT sender_id, COUNT(*) AS message_count
  FROM messages
  WHERE sent_at >= TIMESTAMP '2025-08-01' AND sent_at < TIMESTAMP '2025-09-01'
  GROUP BY sender_id
),
rk AS (
  SELECT sender_id, message_count,
         DENSE_RANK() OVER (ORDER BY message_count DESC) AS r
  FROM cnt
)
SELECT sender_id, message_count
FROM rk
WHERE r <= 2
ORDER BY message_count DESC, sender_id;

cnt counts each sender's August messages. rk assigns a dense rank by descending count. The filter r <= 2 keeps the top two count levels; because DENSE_RANK gives every tied sender the same rank, all three top senders fall under rank 1 and survive together.

4. Strategic explanations

Why DENSE_RANK beats LIMIT 2 and ROW_NUMBER

ORDER BY message_count DESC LIMIT 2 would return exactly two rows — arbitrarily two of the three tied leaders — and never reach sender 12. ROW_NUMBER has the same flaw: it assigns 1, 2, 3, 4 unconditionally, breaking the tie by physical order. DENSE_RANK assigns tied rows the same rank, so a "top 2" filter naturally expands to include every tied member. The phrase "return ALL tied users" is the signal that a ranking-with-ties function is required.

DENSE_RANK vs RANK — the gap matters here

With three senders tied at rank 1, RANK would assign the next sender rank 4 (it skips 2 and 3), so WHERE r <= 2 would return only the three tied leaders and miss sender 12 entirely. DENSE_RANK assigns no gaps — the next sender is rank 2 — so the query correctly captures the top two distinct volumes. For a "top 2 levels" requirement, DENSE_RANK is the right choice; RANK would silently shrink the result.

The tie-break instruction is partly redundant — say so

The prompt says "break ties by lower sender_id first". With DENSE_RANK() OVER (ORDER BY message_count DESC), tied senders genuinely share a rank — there is no tie to break for inclusion, and all of them are returned. The sender_id ordering only affects the display order of the final rows, handled by the outer ORDER BY message_count DESC, sender_id. A senior candidate names this: the tie-break is a presentation rule here, not a selection rule, because we are told to keep all ties anyway.

The half-open month range

sent_at >= '2025-08-01' AND sent_at < '2025-09-01' is the correct way to bound a month on a TIMESTAMP column. BETWEEN '2025-08-01' AND '2025-08-31' would drop any message timestamped after midnight on Aug 31 — a real and common bug. The half-open [month-start, next-month-start) form catches every August instant and excludes July (so sender 13 vanishes) and September.

COUNT(*) is correct here — and so is the grouping

COUNT(*) counts message rows per sender, which is exactly "message volume". There is no distinct-counting subtlety — each message is one row. The GROUP BY sender_id in cnt runs before the window function in rk; you must aggregate to one row per sender first, then rank those counts. Window functions execute after GROUP BY, so the two-CTE split (aggregate, then rank) is the natural structure and also lets you filter on the rank in the outer query.

Senior signal: a leaderboard that must "return all tied users" calls for DENSE_RANK filtered by r <= N — LIMIT and ROW_NUMBER truncate ties, and RANK's gaps can drop a real runner-up. Recognise when a stated tie-break only governs display order, and bound the month with a half-open timestamp range so boundary messages land correctly.
