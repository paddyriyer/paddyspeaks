A risk-operations team gates a win-back campaign on a fraud check: before re-engaging churned customers with an incentive, score each one's recent fraud signals and route them to eligible, manual_review, or hard_excluded. The question (campaign win_back_q2_2025, snapshot 2025-05-01) layers severity-weighted L7/L28/L90 windows, a consecutive-day "burst" detector, and a campaign-eligibility join. The interviewer is probing conditional aggregation with mixed time windows, a gaps-and-islands burst, and string-rank tier filtering.

2. The SQL — severity-weighted windows, burst islands, tier-rank filter

WITH params AS (SELECT DATE '2025-05-01' AS snapshot),
cmp AS (SELECT * FROM campaigns WHERE name = 'win_back_q2_2025'),
sig AS (
    SELECT f.customer_id, f.severity, f.signal_ts,
           ((SELECT snapshot FROM params) - f.signal_ts) AS days_back
    FROM fraud_signals f
    WHERE f.signal_ts >= (SELECT snapshot FROM params) - INTERVAL '90 days'
),
sig_days AS (
    SELECT customer_id, signal_ts AS signal_day
    FROM sig GROUP BY customer_id, signal_ts
),
sig_indexed AS (
    SELECT customer_id, signal_day,
           (signal_day - DATE '2000-01-01')
             - ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY signal_day) AS island_id
    FROM sig_days
),
sig_islands AS (
    SELECT customer_id, island_id, COUNT(*) AS streak_len
    FROM sig_indexed GROUP BY customer_id, island_id
),
burst_per_cust AS (
    SELECT customer_id, MAX(streak_len) AS longest_burst
    FROM sig_islands GROUP BY customer_id
),
fraud_summary AS (
    SELECT customer_id,
        SUM(CASE WHEN days_back < 7  THEN 1 ELSE 0 END) AS l7_signals,
        SUM(CASE WHEN days_back < 28 THEN 1 ELSE 0 END) AS l28_signals,
        SUM(CASE WHEN days_back < 90 THEN 1 ELSE 0 END) AS l90_signals,
        SUM(CASE WHEN severity='HIGH'   AND days_back <= 90 THEN 1 ELSE 0 END) AS high_l90,
        SUM(CASE WHEN severity='MEDIUM' AND days_back <= 30 THEN 1 ELSE 0 END) AS medium_l30,
        SUM(CASE WHEN severity='LOW'    AND days_back <= 28 THEN 1 ELSE 0 END) AS low_l28,
        SUM(CASE WHEN severity='HIGH'   AND days_back <= 90 THEN 10
                 WHEN severity='MEDIUM' AND days_back <= 30 THEN 5
                 WHEN severity='LOW'    AND days_back <= 28 THEN 1
                 ELSE 0 END) AS risk_score
    FROM sig GROUP BY customer_id
)
SELECT c.customer_id, c.tier, c.lifecycle_state,
    COALESCE(fs.l7_signals,0)  AS l7_signals,
    COALESCE(fs.l28_signals,0) AS l28_signals,
    COALESCE(fs.l90_signals,0) AS l90_signals,
    COALESCE(fs.high_l90,0)    AS high_l90,
    COALESCE(fs.medium_l30,0)  AS medium_l30,
    COALESCE(fs.low_l28,0)     AS low_l28,
    COALESCE(bp.longest_burst,0) AS longest_burst,
    COALESCE(fs.risk_score,0)  AS risk_score,
    CASE
        WHEN COALESCE(fs.high_l90,0) > 0
          OR COALESCE(fs.risk_score,0) >= 15
          OR COALESCE(bp.longest_burst,0) >= 5 THEN 'hard_excluded'
        WHEN COALESCE(fs.medium_l30,0) > 0
          OR COALESCE(fs.risk_score,0) >= 5
          OR COALESCE(bp.longest_burst,0) >= 3 THEN 'manual_review'
        ELSE 'eligible'
    END AS status
FROM customers c
JOIN cmp ON c.lifecycle_state = cmp.target_state
LEFT JOIN fraud_summary  fs ON fs.customer_id = c.customer_id
LEFT JOIN burst_per_cust bp ON bp.customer_id = c.customer_id
WHERE CAST(substr(c.tier, 6) AS INTEGER) <= CAST(substr(cmp.min_tier, 6) AS INTEGER)
ORDER BY risk_score DESC, c.customer_id ASC;

4. Strategic explanations

Each severity has its own time window — the windows differ on purpose

HIGH counts over 90 days, MEDIUM over 30, LOW over 28. The conditional aggregation encodes three different lookbacks in one SUM(CASE ...) per metric, and the risk_score bundles weight (10/5/1) and window into a single CASE. The design rationale: a severe signal stays relevant longer, a minor one decays fast. Customer 4's expired MEDIUM (36 days back) is the live demonstration — same signal, wrong window, zero contribution.

The burst detector: distinct days, then signal_day - ROW_NUMBER

longest_burst is "longest run of consecutive calendar days with at least one signal" — a gaps-and-islands problem. sig_days first collapses to one row per (customer, day) so two signals on the same day do not desync the count. Then for each customer the day is converted to an integer (signal_day - '2000-01-01') and numbered with ROW_NUMBER; for a consecutive run both increase by 1, so their difference is a constant island id. MAX(COUNT(*)) over islands gives the longest streak.

String-rank tier filtering — clever but fragile

CAST(substr(c.tier, 6) AS INTEGER) <= CAST(substr(cmp.min_tier, 6) AS INTEGER) rips the digit out of 'Tier 2' (character 6 onward) and compares numerically. The <= is correct because a lower tier number is a better tier — "ranks at or above min_tier" means tier number ≤ min_tier number. It works, but it is brittle: rename a tier to 'Platinum' or 'Tier 10' and substr(...,6) breaks. A senior would flag that a proper tier_rank lookup table is the maintainable design and the string surgery is interview-grade only.

LEFT JOIN + COALESCE: clean customers must still appear

Customers 1 and 8 have no rows in fraud_summary (no signals in the 90-day window). The query starts from customers, inner-joins the campaign to apply the state filter, then LEFT JOINs the fraud aggregates. Every fraud column is wrapped in COALESCE(..., 0) so a missing match reads as zero risk → eligible. An inner join would silently drop the cleanest, most campaign-worthy customers — the exact opposite of intent.

The status CASE is OR-of-thresholds, severity-ordered

hard_excluded fires if any of three conditions hold (a HIGH signal, score ≥ 15, or a 5-day burst); manual_review if any of a softer set holds. Because the branches are evaluated top-down, a customer meeting both tiers is correctly hard-excluded — the strictest verdict wins. Each branch is an independent veto, which is why customer 2 (one HIGH, score only 10) is excluded while a higher-scoring-but-no-HIGH customer might merely hit review.

Senior signal: encode per-severity lookback windows in conditional aggregation and explain why severe signals decay slower, build the burst metric with distinct-day collapsing before the signal_day - ROW_NUMBER islands trick, justify the <= direction of the string-rank tier filter while flagging it as fragile versus a tier_rank table, and drive the result from customers with LEFT JOIN+COALESCE so zero-signal customers still surface as eligible.
