A point-in-time (PIT) feature store table is one of the trickiest analytical schemas to get right because it must serve two orthogonal use cases from a single table: training-time joins (which must use only features known before the label timestamp to avoid leakage) and serving-time lookups (which must return the latest feature value in milliseconds). The interviewer is probing whether you understand the event_ts / label_ts join predicate, how feature_set_ver enables safe backfills, and why is_backfill is a first-class column rather than an afterthought.

1. The requirements — what the design must capture

  • Point-in-time correctness: for training, the join must retrieve the feature value whose event_ts is the most recent one that does not exceed the label’s timestamp — no future information may leak into training data.
  • Entity key: user_id identifies the entity for which the feature is computed.
  • Feature value: the computed feature (user’s 7-day purchase count), stored as a DOUBLE.
  • Temporal columns: event_ts (when the feature value became true) and ingest_ts (when the row was written — audit).
  • Versioning: source_version (pipeline code version) and feature_set_ver (feature definition version) allow multiple definitions to coexist and backfills to be isolated.
  • Backfill flag: is_backfill distinguishes rows generated by a historical backfill job from live production rows, enabling selective retraining without contaminating live serving.
  • PK and partitioning: (user_id, event_ts, feature_set_ver) as PK; partition by event_ts for time-range pruning during training joins.

2. The DDL — the CREATE TABLE

CREATE TABLE feature_user_7d_purchase_count (
  user_id          BIGINT       NOT NULL,                  -- entity key
  feature_value    DOUBLE PRECISION NOT NULL,              -- the feature value
  event_ts         TIMESTAMP    NOT NULL,                  -- when this feature value became true
  ingest_ts        TIMESTAMP    NOT NULL,                  -- when the row was written (audit)
  source_version   VARCHAR(32)  NOT NULL,                  -- pipeline / code version that produced it
  feature_set_ver  VARCHAR(32)  NOT NULL,                  -- feature definition version
  is_backfill      BOOLEAN      NOT NULL DEFAULT FALSE,    -- distinguishes initial backfill from live updates
  PRIMARY KEY (user_id, event_ts, feature_set_ver)
)
PARTITION BY (event_ts);

4. Strategic explanations

Grain and the PIT join predicate

The grain is one row per (user, event_ts, feature_set_ver). Multiple rows can exist for the same user at different event timestamps because the feature is recomputed periodically (e.g., hourly). The PIT training join retrieves, for each label row, the latest feature row whose event_ts ≤ label.label_ts. The QUALIFY / ROW_NUMBER pattern in the pseudo-SQL achieves this: filter to rows that predate the label, then pick the most recent one per label. Using strict ≤ (not <) is deliberate: a feature computed at exactly the label time is still safe to use.

event_ts vs. ingest_ts — the leakage boundary

Training leakage occurs when a model is trained on information that was not actually available at prediction time. Using ingest_ts instead of event_ts in the PIT join predicate would allow features that were physically written to the table after the label timestamp to leak into training — for example, a feature computed at 14:00 but not ingested until 16:00 would leak if the label was at 15:00. event_ts is the analytically correct boundary; ingest_ts is only for auditing pipeline timeliness.

feature_set_ver: safe backfills without breaking models

Suppose model v3 was trained on feature definition ‘v3’. The ML team wants to recompute the feature with a bug fix (‘v4’). Without a version column, a backfill with the fixed definition would overwrite the ‘v3’ rows, silently changing the training data for model v3 without retraining — a subtle data integrity violation. By including feature_set_ver in the PK, ‘v4’ rows coexist with ‘v3’ rows. Model v3 specifies feature_set_ver = ‘v3’ in its training job and is unaffected by the backfill.

is_backfill: separating live and historical feature rows

A backfill computes features for historical dates where no live feature rows existed. These rows are valid for training but should not be served to the online feature store (which expects only live-computed values, validated by the live pipeline). The is_backfill = FALSE filter in the serving layer enforces this separation. It also enables monitoring: if the fraction of live rows served drops to zero (all rows are backfill), the live pipeline has silently failed.

Partition strategy and serving latency

Partitioning by event_ts prunes the expensive cross-time scans in training joins: a training job for the last 90 days of labels only scans partitions within that window. For online serving (latest value per user), the query is SELECT feature_value FROM … WHERE user_id = :uid AND feature_set_ver = ‘v3’ AND is_backfill = FALSE ORDER BY event_ts DESC LIMIT 1. This benefits from a secondary index on (user_id, feature_set_ver, event_ts DESC) rather than a partition scan. In production, the online store is typically a separate low-latency system (Redis, DynamoDB) populated from this offline table, not the offline table itself.

Warehouse dialect notes

In BigQuery, use PARTITION BY DATE(event_ts) and add is_backfill as a clustering column to prune serving queries. In Snowflake, CLUSTER BY (user_id, feature_set_ver, event_ts) supports both training (scan by event_ts) and serving (scan by user_id). In Redshift, use SORTKEY(user_id, event_ts) and DISTKEY(user_id) so each user’s rows are co-located for the serving query.

Senior signal: distinguish event_ts (the analytically correct PIT boundary) from ingest_ts (audit only) and explain why confusing them causes training leakage; justify feature_set_ver in the PK as the mechanism that lets backfills coexist with live rows without invalidating already-trained models; and articulate the is_backfill flag as a production guard that keeps serving infrastructure from accidentally serving historically-computed values — these three moves reveal that you understand feature stores as a data correctness problem, not just a schema exercise.
