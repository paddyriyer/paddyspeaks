UBS's operations desk runs a warehouse and wants the average stock cycle — how many days a product sits between arriving and being sold — per product. The query has to turn two timestamps into a duration, average that duration, and express it in days. The arithmetic looks more elaborate than it is: the solution multiplies by 86400 and divides by 86400, which cancels out. The real interview content is the WHERE sold_at IS NOT NULL filter and what it silently does to products still sitting on the shelf.

Product 1 has two completed cycles (5 days and 10 days). Product 2 has one completed cycle (3 days) and one unsold unit — a NULL sold_at. Product 3's cycle is 1.25 days, exercising the fractional-day path: 30 hours, not a whole number of days.

2. The SQL — AVG of timestamp diff in days

SELECT product_id,
       AVG(((EXTRACT(EPOCH FROM (sold_at - received_at)) / 86400.0) * 86400)/86400)
         AS avg_cycle_days
FROM inventory
WHERE sold_at IS NOT NULL
GROUP BY product_id;

sold_at - received_at is an interval; EXTRACT(EPOCH FROM …) turns it into total seconds; dividing by 86400 converts seconds to days. The extra * 86400 then /86400 cancel — harmless, but noise.

Product 1 averages (5 + 10) / 2 = 7.5 days. Product 2 shows 3 days from its single completed cycle — the unsold unit was filtered out entirely, so it neither raised nor lowered the average. Product 3 is 1.25 days, the fractional value preserved because the epoch math is floating-point.

4. Strategic explanations

The redundant arithmetic — what the formula actually is

(epoch / 86400.0) * 86400 / 86400 simplifies to epoch / 86400.0. The two extra operations cancel and contribute nothing but rounding noise. Write the intent plainly:

SELECT product_id,
       AVG(EXTRACT(EPOCH FROM (sold_at - received_at)) / 86400.0) AS avg_cycle_days
FROM inventory
WHERE sold_at IS NOT NULL
GROUP BY product_id;

Even cleaner, skip epoch entirely — subtract the timestamps as date values and the result is already an integer count of days (loses sub-day precision, so only if hours do not matter):

AVG(sold_at::date - received_at::date) AS avg_cycle_days
The WHERE sold_at IS NOT NULL filter — correct, but say why

A NULL sold_at means "not sold yet" — there is no cycle to measure. The filter is right. But note the consequence: a product that is entirely unsold drops out of the result set altogether (it has zero qualifying rows, so GROUP BY produces nothing for it). If the dashboard must list every product, that needs a LEFT JOIN from a product dimension, leaving avg_cycle_days NULL for never-sold items. Also flag the survivorship bias: averaging only sold units understates true holding time, because the slowest-moving stock is exactly what is still unsold.

The interviewer's likely follow-up — overall vs per-product

Expect "what is the average across the whole warehouse?" Dropping GROUP BY gives a single grand average — but a row-weighted one, where a product with many cycles dominates. If the business wants each product weighted equally, average the per-product averages:

SELECT AVG(avg_cycle_days) FROM (
  SELECT product_id, AVG(EXTRACT(EPOCH FROM (sold_at - received_at))/86400.0) AS avg_cycle_days
  FROM inventory WHERE sold_at IS NOT NULL GROUP BY product_id
) p;
Dirty-data edge case — sold_at before received_at

If a data-entry error puts sold_at earlier than received_at, the interval is negative and the average is silently dragged down — no error is raised. Guard it: WHERE sold_at > received_at, or surface the bad rows with a COUNT(*) FILTER (WHERE sold_at < received_at) data-quality check before trusting the metric.

Why EXTRACT(EPOCH ...) over date subtraction

Subtracting two timestamp values yields an interval, and you cannot directly AVG an interval into a clean day count — well, Postgres can AVG an interval, but the result is itself an interval (7 days 12:00:00), not the number 7.5. EXTRACT(EPOCH ...) drops to a scalar number of seconds, which divides and averages into a plain numeric. That scalar is what a BI tool wants on an axis.

Senior signal: recognise the *86400/86400 as a no-op and simplify it, justify the sold_at IS NOT NULL filter while naming the survivorship bias and the dropped never-sold products, distinguish row-weighted from product-weighted overall averages, and add a guard for negative (bad-data) cycles.
