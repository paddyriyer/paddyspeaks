A merchandising team has a curated "must-have" product list and wants the users whose cart contains every one of those products. This is relational division — the SQL operation that answers "for all" questions — and the interviewer explicitly wants the classic NOT EXISTS-inside-NOT EXISTS formulation, because SQL has no native "for all" quantifier.

The edge cases that decide the interview: user 1 has a superset (extra product 4 must not disqualify), user 5 has a duplicate row (must still count as one user, not two), and user 2 is the near-miss — two of three targets is not enough.

2. The SQL — double NOT EXISTS (relational division)

SELECT DISTINCT u.user_id
FROM cart u
WHERE NOT EXISTS (
  SELECT 1 FROM target_products t
  WHERE NOT EXISTS (
    SELECT 1 FROM cart c
    WHERE c.user_id = u.user_id AND c.product_id = t.product_id
  )
)
ORDER BY u.user_id;

Read it inside-out. The innermost query asks "does this user have this target product?" The middle NOT EXISTS turns that into "is there a target product this user is missing?" The outer NOT EXISTS negates again: "this user is missing no target product" — i.e. has them all.

Users 1, 3, 5 qualify. User 1's extra product 4 is irrelevant — division checks containment, not equality. User 5 appears once despite its duplicate cart row, thanks to SELECT DISTINCT. Users 2 and 4 are correctly absent: each is missing at least one target.

4. Strategic explanations

Why two negations express "for all"

SQL has EXISTS ("there is at least one") but no FOR ALL quantifier. The logical identity that rescues you is: "X holds for all t" is equivalent to "there is no t for which X fails". Apply it here: "user has every target" becomes "there is no target the user lacks." Each clause — "lacks" and "there is no" — is one NOT EXISTS, so the construct nests two of them. This double-negation is the canonical SQL pattern for universal quantification; recognising it on sight is the point of the question.

The correlation chain is what makes it work

Three query levels, two correlations. The middle query is correlated to the outer via c.user_id = u.user_id — "the user we are currently testing." The inner is correlated to the middle via c.product_id = t.product_id — "the target we are currently checking." If either correlation is dropped, the logic collapses: lose the user correlation and you ask whether anyone has the product, not this user. Trace both correlations explicitly when explaining.

Superset and duplicate behaviour — by design

Relational division tests "cart ⊇ targets", not "cart = targets". User 1's product 4 is simply never examined — the query only ever iterates over target_products, so anything in the cart beyond the targets is invisible. Duplicates are equally harmless to the logic: the inner EXISTS only cares whether at least one matching row exists, so user 5's repeated product 1 changes nothing. The duplicate does matter for the output though — FROM cart u would emit user 5 once per cart row, hence SELECT DISTINCT is required to collapse them.

The empty-divisor trap

A famous quirk: if target_products were empty, the middle NOT EXISTS would be vacuously true for everyone (no target rows means no missing target), so the query would return every user. Mathematically correct — everyone trivially owns all zero required products — but rarely the intended business answer. On a real system, guard against an empty divisor explicitly. Also note: a NULL product_id in either table would make the inner equality UNKNOWN and could distort the result; with a NOT NULL column this is moot, but worth stating.

Dialect notes and the alternative

The double-NOT EXISTS form is fully ANSI-standard and runs identically on PostgreSQL, MySQL, SQL Server, and Oracle. The common alternative is the counting formulation: join cart to targets, GROUP BY user_id, and keep groups where COUNT(DISTINCT c.product_id) = (SELECT COUNT(*) FROM target_products). It is often easier to read — but the DISTINCT inside the count is mandatory, precisely to neutralise user 5's duplicate. The question pins you to the NOT EXISTS form, but knowing the count-based equivalent (and why it needs DISTINCT) is the senior signal.

Senior signal: recognise "contains every X" as relational division, implement it as NOT EXISTS ("no target...") wrapping NOT EXISTS ("...that the user lacks"), keep both correlation predicates intact, add SELECT DISTINCT to collapse duplicate cart rows, and know the empty-divisor returns-everyone trap plus the count-based alternative.
