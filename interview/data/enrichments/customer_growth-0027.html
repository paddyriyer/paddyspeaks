The canonical hot-reducer rescue. The same pattern shows up at every layer of a distributed stack — Spark shuffle, Flink keyed state, DynamoDB write throttling, even a Postgres aggregation against a single partition. SQL is the cleanest place to internalise it because you can SEE every partial in a result table. Once you have the pattern in your head, the Spark / Flink / sharding versions become the same shape with different syntax.

0. Why salting exists — the whale / dolphin / minnow problem

At Uber, Netflix, Stripe, Meta scale, customer / merchant / content data is heavily Pareto-shaped (see the vocabulary page):

  • 🐋 Whales (top 1-5%) — one corporate Stripe account with 50M charges, one Netflix Original with 200M plays in launch week, one Uber market with 10M trips/day. This single key is the hot reducer.
  • 🐬 Dolphins (middle 65-75%) — normal accounts with 10K-100K rows. They land in roughly-balanced reducers under default hash partitioning.
  • 🐟 Minnows (bottom 20-30%) — long-tail accounts with 1-10 rows. So small that any reducer can swallow them in microseconds.

The default GROUP BY customer_id sends the whale's 50M rows to ONE reducer (because hash(whale) % N picks a single bucket). That reducer runs for hours while the others finish in seconds. Adding more reducers doesn't help — the hash still concentrates. Salting fixes it by spreading the whale's rows across N reducers via a small random suffix, then folding the partials back up.

The interview framing: "Salting is the standard rescue for the hot-key problem at any company that has a Pareto-shaped customer base. Naive aggregations break because one whale lands in one reducer. The fix is two-stage: GROUP BY (key, salt) first, GROUP BY key second."

3. The fix — add a salt, do two passes

Pass 1 — partial aggregation by (customer_id, salt)
SELECT customer_id, FLOOR(RANDOM() * 16)::INT AS salt, amount
FROM orders

Each row gets a uniform-random salt in [0, 16). Customer 1's 30 rows now split across (approximately) 16 different (customer_id, salt) groups. The partial aggregation:

SELECT customer_id, salt,
       SUM(amount) AS partial_sum,
       COUNT(*)    AS partial_rows
FROM salted
GROUP BY customer_id, salt;

Sample output for customer 1 (the whale):

The 30 rows are now spread across 16 groups instead of 1. The shuffle for pass 1 sends 16 tiny groups per customer to the cluster, not one giant group.

Reducer 1  ─── (c1, salt=1) (c1, salt=5) (c2, salt=3) (c4, salt=8)   ━━━━ ~5 rows  ✓
Reducer 2  ─── (c1, salt=2) (c1, salt=6) (c3, salt=12)               ━━━━ ~5 rows  ✓
...                                                                  every reducer ~5 rows
Pass 2 — roll the partials back up to customer_id
SELECT customer_id,
       COUNT(*)               AS salt_buckets_seen,
       SUM(partial_sum)       AS total_spend
FROM partial_agg
GROUP BY customer_id;

Pass 2 sees at most (N_customers × 16) rows — a tiny dataset. It runs in milliseconds. The hot key is no longer hot at this stage because each customer contributes only 16 small rows.

4. Why two passes — the associativity requirement

Salting only works for aggregations that are associative and commutative: (a + b) + c = a + (b + c). You can sum sub-totals to get the total. The aggregations that work:

AggregationSalt-friendly?How to combine partials
SUM✓ YesSUM(partial_sum)
COUNT / COUNT(*)✓ YesSUM(partial_count)
MIN / MAX✓ YesMIN/MAX(partial_min/max)
AVG⚠ IndirectKeep sum, count in pass 1; SUM(sum)/SUM(count) in pass 2
COUNT(DISTINCT x)✗ Not naivelyUse HLL sketches per salt bucket; merge in pass 2 (see customer_growth-0028)
PERCENTILE_CONT✗ NoNeeds the full sorted distribution; salting can't accelerate it. Use APPROX_PERCENTILE (sketch-based)
STRING_AGG⚠ Order-sensitiveWorks if order doesn't matter; tricky if it does (ORDER BY breaks across partials)

5. The four salting strategies — pick consciously

The runnable solution above uses random salting. That's the demo form. Production has four named variants — each with a different idempotency / cost / scope trade-off. Knowing which to reach for in a given context is the L5 signal.

Strategy A — Random salting (the demo form)
SELECT customer_id,
       FLOOR(RANDOM() * 16)::INT AS salt,    -- non-deterministic
       amount
FROM orders;

Use when: One-shot ad-hoc query; no need to reproduce the same partial breakdown twice. Trap: re-running the query produces different per-bucket partials, so any debugging or downstream verification gets confused. The total is identical (SUM is associative) but per-row provenance is not.

Strategy B — Hash-based / deterministic salting (the production form)
WITH salted AS (
    SELECT customer_id,
           ABS(HASHTEXT(order_id::text)) % 16  AS salt,   -- Postgres
           -- ABS(HASH(order_id))           % 16          -- Snowflake
           -- MD5(CAST(order_id AS STRING))  % 16         -- BigQuery (after CRC32)
           amount
    FROM orders
),
partial_agg AS (
    SELECT customer_id, salt, SUM(amount) AS partial_sum
    FROM salted GROUP BY customer_id, salt
)
SELECT customer_id, ROUND(SUM(partial_sum), 2) AS total_spend
FROM partial_agg GROUP BY customer_id ORDER BY total_spend DESC;

Use when: Production ETL pipeline, dbt model, anything that re-runs nightly. The hash of order_id always produces the same bucket assignment, so partial breakdowns are reproducible across runs. Idempotency is the win — if you debug a bad partial, you can re-run and see the exact same bucket pattern.

Strategy C — Filtered / skew-only salting (the cost-aware form)

Salting EVERY key — including the 10K minnows — adds overhead. For 10K small customers you force the database to manage 160K partial-aggregation groups (10K × 16 buckets) when 10K would suffice. The fix: only salt the keys you know are hot.

WITH heavy_hitters AS (
    -- Identify hot keys via threshold OR a pre-computed top-K table
    SELECT customer_id FROM orders
    GROUP BY customer_id
    HAVING COUNT(*) > 1000
),
salted AS (
    SELECT o.customer_id,
           CASE
             WHEN h.customer_id IS NOT NULL
                THEN ABS(HASHTEXT(o.order_id::text)) % 16   -- whales: spread across 16
             ELSE 0                                          -- minnows: untouched
           END AS salt,
           o.amount
    FROM orders o
    LEFT JOIN heavy_hitters h ON h.customer_id = o.customer_id
),
partial_agg AS (
    SELECT customer_id, salt, SUM(amount) AS partial_sum
    FROM salted GROUP BY customer_id, salt
)
SELECT customer_id, ROUND(SUM(partial_sum), 2) AS total_spend
FROM partial_agg GROUP BY customer_id ORDER BY total_spend DESC;

Use when: Pareto distribution with very few hot keys + large cold tail (the most common real-world shape). The cold tail's pass 1 is a no-op — every minnow row goes to salt=0, so the per-customer group count is unchanged for them. Only the whale's groups multiply by 16. This is the pattern most production Spark jobs use.

Strategy D — Logarithmic / MOD salting (the streaming form)

If you don't have RANDOM() available (some SQL engines, most streaming engines), or want zero CPU overhead per row, use the modulo of a sequential ID:

SELECT customer_id,
       (order_id % 16) AS salt,    -- distributes rows sequentially across buckets
       amount
FROM orders;

Use when: High-throughput streaming systems (Kafka producer key prefix, Flink keyed-state assignment). Trap: if order_id is monotonic AND batches arrive in chunks of 16, you get perfectly uniform distribution; if batches arrive in chunks of 17, bucket 0 gets all the extras. Hash-based (Strategy B) is more robust against this; MOD wins on raw cycle cost.

6. Salting a JOIN — the "Final Boss" of skew problems

Salting an aggregation is easy: spread → partial → roll up. Salting a JOIN is harder because the salted side has 16 versions of the key, and the unsalted side only has 1. You must explode the dim side so all 16 salt values find their match.

Scenario: Joining a billion-row orders fact (whale-skewed by customer_id) to a 1M-row customers dim. Default sort-merge join shuffles the entire fact table by hash(customer_id) — the whale's billion rows land on one reducer. 4-hour stage; everything else finishes in seconds.

-- Three-step rescue. Result is identical to the naive JOIN — only the
-- shuffle distribution differs.

-- Step 1: Salt the BIG side (orders). Each row gets a salt 0..15.
WITH orders_salted AS (
    SELECT order_id, customer_id, amount,
           ABS(HASHTEXT(order_id::text)) % 16 AS salt
    FROM orders
),

-- Step 2: EXPLODE the SMALL side (customers). Replicate each customer
-- row 16 times — once per possible salt value. Now every salted order
-- has a matching dim row available within its salt bucket.
salts AS (
    SELECT generate_series(0, 15) AS salt          -- Postgres
    -- SELECT EXPLODE(SEQUENCE(0, 15)) AS salt    -- Snowflake/Databricks
),
customers_exploded AS (
    SELECT c.customer_id, c.name, c.segment, s.salt
    FROM customers c
    CROSS JOIN salts s
)

-- Step 3: JOIN on (customer_id, salt). The whale's billion rows now
-- spread across 16 reducers, each with the matching customer row available.
SELECT o.customer_id, c.name, c.segment,
       SUM(o.amount) AS total
FROM orders_salted o
JOIN customers_exploded c
  ON o.customer_id = c.customer_id AND o.salt = c.salt
GROUP BY o.customer_id, c.name, c.segment;

The cost: the dim side gets 16× duplicated. For a 1M-row customers table, that's 16M rows in the join — still tiny vs. the fact-table shuffle savings. Pick salt count N such that N × dim_table_rows still fits in cluster memory.

Modern alternative: Spark 3+'s Adaptive Query Execution (AQE) auto-detects skewed join partitions and splits them. spark.sql.adaptive.skewJoin.enabled = true handles the 80% case for free. Reach for manual salt-and-explode only when AQE can't handle the case (very high skew, custom UDF aggregates, non-Spark engines).

8. Edge cases the interviewer will probe

Probe — "What if the customer_id is NULL?"

hash(NULL) = NULL in standard SQL. NULL % 16 = NULL. All NULL-customer rows land in the "NULL salt" bucket — which is itself a hot bucket. Two fixes: either filter NULLs before salting (if business semantics allow), or coalesce: ABS(HASHTEXT(COALESCE(order_id::text, RANDOM()::text))) % 16. The first is cleaner; surface the NULL rows separately.

Probe — "What if the order_id used as the hash basis isn't unique?"

Then identical order_ids land in the same salt bucket — the whale concentrates again, but in 1 bucket of 16 instead of 1 of 1. Better to use a guaranteed-unique row identifier (PK, sequence, ROW_NUMBER) for the hash input. If no unique field exists in the source, materialise a row_id column upstream.

Probe — "What if you have 10M unique customer_ids and only 5 are whales?"

Use Strategy C (filtered). Salting all 10M wastes ~150M groups in pass 1 and slows the cold tail by ~16×. Identify the 5 whales via a heavy-hitter query (the HAVING COUNT(*) > threshold in the Strategy C example) and apply the salt only to them.

9. The same pattern at every layer

Spark — salting the join / aggregate
# SQL-style two-stage Spark — same shape as the runnable SQL above
val SALT_BUCKETS = 16
val salted = orders.withColumn("salt", floor(rand() * SALT_BUCKETS).cast("int"))

val pass1 = salted.groupBy("customer_id", "salt")
                  .agg(sum("amount").as("partial_sum"))

val total = pass1.groupBy("customer_id")
                 .agg(sum("partial_sum").as("total_spend"))

For Spark JOIN skew, salting is more involved — you salt the LEFT side and explode the RIGHT side so all salt buckets find their matching dim row. customer_growth-0027 covers the aggregation case; Spark's full join-salt is in the Hot Shards design page, §03.

Modern alternative: Spark 3+'s Adaptive Query Execution (AQE) with spark.sql.adaptive.skewJoin.enabled = true auto-detects skewed partitions and splits them. Try AQE first; reach for manual salting when AQE can't handle the case (very high skew, custom UDF aggregates).

Flink — pre-aggregation before keyBy
// One whale = one operator subtask if you keyBy(user_id) directly.
// Add a salt to pre-aggregate, then keyBy on the user_id alone.

events
  .map(e => (e.user_id, ThreadLocalRandom.current().nextInt(16), e.amount))
  .keyBy { case (uid, salt, _) => (uid, salt) }     // pass 1: spreads whale across 16 subtasks
  .window(TumblingProcessingTimeWindows.of(Time.minutes(1)))
  .sum(2)
  .map { case ((uid, _), partial) => (uid, partial) }
  .keyBy(_._1)                                       // pass 2: small fan-in per user
  .sum(1)

Same algorithm: salt + partial + roll-up. In Flink the win is on RocksDB hotspot avoidance — without salting, the whale's keyed-state operator grows 100× larger than the others and triggers checkpoint storms. With salting, every subtask has roughly the same state size.

Sharding / write distribution — DynamoDB & Cassandra
// DynamoDB partition key: instead of pk = user_id, use a salted suffix
const SHARDS = 16;
const writePK = `${user_id}_${Math.floor(Math.random() * SHARDS)}`;
// Reads scatter across SHARDS partition keys and merge results client-side
const readKeys = Array.from({length: SHARDS}, (_, i) => `${user_id}_${i}`);

Cassandra's composite-partition-key bucketing is the same trick: PRIMARY KEY ((user_id, bucket), event_time) where bucket = hash(event_id) % 16. Spreads a hot user's writes across 16 physical partitions.

The read-side cost: every read becomes a scatter-gather across all salt buckets. You trade write-side concentration for read-side fan-out. Pick this when writes >> reads (event stores, time-series), avoid when reads dominate (look-up tables).

Streaming aggregation — Kafka partition skew

Kafka's default key-hash partitioning concentrates a whale's events on one partition; the downstream consumer for that partition lags. Salt the key on the producer:

// Instead of producer.send(new ProducerRecord(topic, user_id, event))
const salt = Math.floor(Math.random() * 16);
producer.send(new ProducerRecord(topic, `${user_id}_${salt}`, event));
// Downstream consumer must NOT assume "all events for user X are on one partition"

Ordering guarantee trade-off: per-key ordering across the whale's events is now broken. If you need it (e.g. for event-sourcing-style state), salt is the wrong tool — split the whale into a dedicated topic instead.

10. More interviewer edge-case probes

Probe 1 — "Why 16 buckets? Could you use 4? 256?"

Three forces in tension: (a) more buckets = better balance for the whale; (b) more buckets = more rows in pass 2 (each customer contributes N buckets); (c) more buckets = more state in keyed-state systems. Rule of thumb: N ≈ workers × 2, or N = next_power_of_two(max_keys_per_worker). 16 fits a typical 8-core worker; FAANG-scale shops use 128 or 1024.

Probe 2 — "What if the salt is non-uniform? Hash of timestamp instead of RANDOM?"

Works as long as the hash distribution is uniform with respect to the hot key. FLOOR(EXTRACT(EPOCH FROM order_ts) * 1000)::INT % 16 is deterministic (same row → same salt), which helps with idempotent re-runs. RANDOM() is non-deterministic but evens out faster on small datasets. Pick deterministic for reprocessing, random for one-shot jobs.

Probe 3 — "Does salting change the answer?"

No — that's the point. Pass 2's SUM(partial_sum) reconstructs the exact total. The result is identical to the naive GROUP BY customer_id; only the shuffle distribution differs. You can prove it with an SQL test:

SELECT salt_buckets_seen, total_spend FROM (this query)
EXCEPT
SELECT 16, SUM(amount) FROM orders GROUP BY customer_id;
-- Result: zero rows (modulo the salt_buckets_seen column, which is salting-specific)
Probe 4 — "What if I only have one hot key out of millions? Salting all of them is wasteful."

Yes — conditional salting is the production answer. Salt only the known-hot keys; leave the cold tail alone:

WITH heavy_hitters AS (SELECT customer_id FROM (top-K query)),
salted AS (
  SELECT customer_id,
         CASE WHEN customer_id IN (SELECT customer_id FROM heavy_hitters)
              THEN FLOOR(RANDOM() * 16)::INT
              ELSE 0
         END AS salt,
         amount
  FROM orders
)
-- pass 1, pass 2 as before

The cold tail's pass 1 is a no-op (every row goes to salt=0); the whale's pass 1 spreads across 16. Pass 2 still rolls up cleanly. This is the pattern most production Spark jobs use — discovered hot keys via a heavy-hitter sketch (Count-Min) and salt only those.

Probe 5 — "Could you avoid the two passes entirely with COMBINER / map-side aggregation?"

Sometimes — Spark/MapReduce can do a combine (map-side partial aggregation) before the shuffle, which reduces the row count before hashing to reducers. If your data is dense at the source (many rows per map task per key), the combine eliminates most of the skew without explicit salting. But for streaming systems and for SQL warehouses (Snowflake, BigQuery, Redshift), salting is the explicit knob.

Probe 6 — "How would you detect skew that needs salting in the first place?"

Detection metrics by layer (see Hot Shards design page, §07):

  • Spark: task duration P99 / P50 per stage. P99 > 10× P50 ⇒ salt.
  • Flink: subtask state size variance. Any subtask > 3× median ⇒ salt.
  • Postgres: EXPLAIN ANALYZE showing one HashAggregate node with skewed memory usage; or pg_stat_statements showing high shared_blks_hit on one query.
  • Kafka: per-partition consumer lag. Max partition lag > 5× P50 lag ⇒ salt the producer key.

Don't pre-emptively salt every aggregation. Salting has costs (two passes, more state, broken ordering) — apply it when detection metrics flag the hot key.

11. Production checklist for the salt pattern

  • ✓ Identify the hot key with a heavy-hitter detection job
  • ✓ Pick N salt buckets ≈ 2× cluster worker count, power of two for hash uniformity
  • ✓ Choose deterministic salt (hash) for reprocessability, random for one-shot
  • ✓ Verify the aggregation is associative (SUM/COUNT/MIN/MAX trivially; AVG needs sum+count split)
  • ✓ Apply salt conditionally — only to known hot keys, not the cold tail
  • ✓ Document the shuffle-size reduction in the query comment so the next engineer doesn't "simplify" it
  • ✓ Monitor: salt_buckets_seen per customer should match N for whales, 1-2 for tail (signals conditional salting is working)
