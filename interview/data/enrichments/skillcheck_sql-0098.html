A finance team keeps monthly metrics in tall (key-value) form — one row per month per metric — and wants a board-ready table: month down the side, revenue and cost as columns, plus a derived profit and a month-over-month profit delta. This combines a pivot with a window function, and the interviewer is checking that you can do both in one pass and get the first-month NULL right.

The edge cases: January is the first month, so its mom_profit_delta has no prior month and must be NULL — not 0. And March's profit falls (revenue down, cost up), so the delta must come out negative — a real sign test.

2. The SQL — pivot in a CTE, LAG for the delta

WITH p AS (
  SELECT month_start,
         SUM(value) FILTER (WHERE metric='revenue') AS revenue,
         SUM(value) FILTER (WHERE metric='cost')    AS cost
  FROM monthly_metric
  GROUP BY month_start
)
SELECT month_start, revenue, cost,
       ROUND((revenue - cost)::numeric, 2) AS profit,
       ROUND(((revenue - cost) - LAG(revenue - cost) OVER (ORDER BY month_start))::numeric, 2)
         AS mom_profit_delta
FROM p
ORDER BY month_start;

The p CTE pivots tall to wide with two filtered SUMs — one per metric. The outer query derives profit, and LAG(revenue - cost) OVER (ORDER BY month_start) reaches back one month so the delta is this month's profit minus last month's.

4. Strategic explanations

Pivot first, then window — why the CTE is essential

You cannot pivot and apply LAG in the same SELECT: LAG needs one row per month, but the raw table has two rows per month. The CTE p does the pivot — collapsing each month to a single row with revenue and cost columns — and only then can a window function order months and reach back. This "reshape in a CTE, analyse in the outer query" layering is the standard pattern when a pivot and a window must compose.

SUM(value) FILTER (WHERE metric=...) is conditional aggregation

Each filtered SUM lets through only the rows for one metric, turning a metric value into a column. The portable equivalent is SUM(CASE WHEN metric='revenue' THEN value END). SUM (not MAX) is the safe aggregate: if a month ever had two revenue rows they would add up rather than one silently winning. The column list is static — a new metric needs a new line — which is the inherent limit of a SQL pivot.

Why the first month is NULL, not 0 — and why that is correct

LAG(expr) OVER (ORDER BY month_start) with no default returns NULL for the first row. profit - NULL is NULL, so January's delta is NULL — precisely what the question demands. Had you written LAG(revenue - cost, 1, 0) with a default of 0, January's delta would be 4000 - 0 = 4000, falsely implying profit jumped from zero. The absence of a prior month is genuinely unknown, and NULL models that honestly.

LAG over a self-join — the performance and correctness win

The pre-window way to get "previous month" is a self-join: p joined to p on month = prev_month + interval '1 month'. That is fragile (it breaks on gaps in the month sequence) and slower. LAG with ORDER BY month_start takes the immediately preceding row in sort order regardless of calendar gaps — if March were missing, February's delta would compare to January, which is usually the desired "previous present month". One sorted pass, no join.

ORDER BY inside OVER vs the final ORDER BY

Two orderings appear and they are independent. OVER (ORDER BY month_start) defines which row LAG treats as "previous" — it controls the computation. The trailing ORDER BY month_start controls the order rows are returned to the client. Omitting the window's ORDER BY would make LAG non-deterministic; omitting the final one would leave output order unspecified. Both are needed, for different reasons.

Senior signal: layer the query — pivot tall-to-wide in a CTE with SUM(value) FILTER (WHERE metric=...), then apply LAG in the outer query — let LAG's default NULL give the first month a correct empty delta rather than a misleading 0, and prefer LAG over a calendar self-join that breaks on gaps.
