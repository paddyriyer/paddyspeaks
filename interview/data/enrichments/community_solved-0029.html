A Community web product logs raw page_load and page_exit events and wants each user's average session length. The catch buried in the data: a user can fire several page_load events in one day (reloads, multiple tabs) and several exits too. You must decide which load pairs with which exit before any duration arithmetic makes sense — and the interviewer is watching that decision, not the AVG.

User 2 has two loads on the same day — the query must pick one. User 3 loaded a page but never exited — an incomplete session that should be dropped, not counted as zero.

2. The SQL — ROW_NUMBER picks one event per action per day

WITH t1 AS (
  SELECT user_id, action, timestamp,
         ROW_NUMBER() OVER (PARTITION BY user_id, action, CAST(timestamp AS date) ORDER BY timestamp DESC) AS flag
  FROM users_web_log
  WHERE action IN ('page_load','page_exit')
), t2 AS (
  SELECT user_id, CAST(timestamp AS date) AS date_dd,
         MAX(CASE WHEN action = 'page_load' THEN timestamp END) AS login_time,
         MAX(CASE WHEN action = 'page_exit' THEN timestamp END) AS logout_time
  FROM t1
  WHERE flag = 1
  GROUP BY user_id, CAST(timestamp AS date)
), t3 AS (
  SELECT user_id, date_dd,
         EXTRACT(EPOCH FROM (logout_time - login_time)) AS session_time
  FROM t2
  WHERE login_time IS NOT NULL AND logout_time IS NOT NULL
)
SELECT user_id, AVG(session_time) AS session_time
FROM t3
GROUP BY user_id;

ROW_NUMBER partitioned by user, action and day, ordered DESC, tags the latest event of each action per day as flag = 1. The next CTE pivots that single load and single exit onto one row; t3 turns the pair into seconds; the final query averages per user.

User 1 had two clean days — 600 s and 1800 s — averaging 1200 s. User 2's two loads were 10:00 and 10:05; flag = 1 kept the later one (10:05), so the session is 10:05→10:20 = 900 s. User 3 has a load but no exit, so logout_time IS NULL filters them out entirely — they never reach the average.

4. Strategic explanations

The pairing decision is the whole question

With multiple loads per day, "session length" is ambiguous. ORDER BY timestamp DESC + flag = 1 encodes the rule "the session starts at the last load and ends at the last exit." A defensible alternative is first load to last exit (the full presence window) — just flip to ORDER BY timestamp ASC for the load. Either is fine; stating which rule you chose, and why, is the senior behaviour.

Why the incomplete session is dropped, not zeroed

User 3's missing exit yields logout_time = NULL from the MAX(CASE ...). The WHERE login_time IS NOT NULL AND logout_time IS NOT NULL guard discards that day. This matters: counting it as a 0-second session would drag the user's average down with a fabricated value. NULL-then-filter is the correct treatment for "we don't know."

The cross-midnight trap

Partitioning by CAST(timestamp AS date) assumes a session never crosses midnight. A load at 23:55 and an exit at 00:10 land on different calendar days, so the load's day has no exit and the exit's day has no load — both get filtered, and a real 15-minute session vanishes. If overnight sessions exist, pair events by sequence (a LEAD from each load to the next exit) rather than by date bucket.

An interleaved-events alternative

The robust general pattern ignores days entirely and matches each load to the next exit in time order:

WITH ordered AS (
  SELECT user_id, action, timestamp,
         LEAD(timestamp) OVER (PARTITION BY user_id ORDER BY timestamp) AS next_ts,
         LEAD(action)    OVER (PARTITION BY user_id ORDER BY timestamp) AS next_action
  FROM users_web_log
)
SELECT user_id,
       AVG(EXTRACT(EPOCH FROM (next_ts - timestamp))) AS session_time
FROM ordered
WHERE action = 'page_load' AND next_action = 'page_exit'
GROUP BY user_id;

This handles multiple sessions per day and overnight spans, at the cost of assuming events strictly alternate load/exit.

Negative durations — a data-quality check

If an exit timestamp predates its load (clock skew, bad logging), EXTRACT(EPOCH ...) goes negative and silently corrupts the average. A senior answer adds AND logout_time > login_time or surfaces a count of such anomalies rather than trusting the feed blindly.

Senior signal: name the load/exit pairing rule explicitly instead of letting it be implicit, drop incomplete sessions via NULL filtering rather than zeroing them, flag the cross-midnight bug that date-bucketing introduces, and validate that durations are non-negative before averaging.
