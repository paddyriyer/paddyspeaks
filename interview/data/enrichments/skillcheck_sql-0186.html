A compensation analyst wants to surface the lowest-paid employees within departments that are large enough for relative ranking to be statistically meaningful. The interviewer is probing whether you know that PERCENT_RANK returns exactly 0 for the lowest-ranked row(s) in every partition, and whether you can combine it cleanly with a COUNT(*) OVER (PARTITION BY dept) filter in one CTE pass — rather than a more cumbersome multi-join.

Eng (4 members, qualifies) — lowest salary is emp 3 at 100. Ops (4 members, qualifies) — lowest salary is emp 10 at 90. Sales (2 members) and HR (1 member) are excluded by the dept_size > 2 rule.

2. The SQL — PERCENT_RANK + dept-size guard in one CTE

WITH r AS (
  SELECT emp_id, dept, salary,
         PERCENT_RANK() OVER (PARTITION BY dept ORDER BY salary) AS pr,
         COUNT(*) OVER (PARTITION BY dept) AS dept_size
  FROM employee
)
SELECT emp_id, dept, salary
FROM r
WHERE pr = 0 AND dept_size > 2
ORDER BY dept, emp_id;

4. Strategic explanations

PERCENT_RANK definition and the guaranteed 0

PERCENT_RANK is defined as (rank - 1) / (N - 1) where rank is the 1-based row rank within the partition and N is the partition size. The formula guarantees that the lowest-ranked row always has rank = 1, giving (1 - 1) / (N - 1) = 0 exactly — regardless of N. This makes pr = 0 a clean predicate for "the minimum salary in this partition," including all tied minimums because PERCENT_RANK assigns the same value to ties (like RANK, not ROW_NUMBER). For a department with 1 member, (N - 1) = 0 would produce a 0/0 division; most engines define it as 0 in that case (single-row partition), but the question's dept_size > 2 guard makes this moot.

Why not use MIN(salary) OVER (PARTITION BY dept) instead

You could filter salary = MIN(salary) OVER (PARTITION BY dept). That works but requires materialising an extra column. pr = 0 is more semantically self-documenting — "this person is at the bottom of their dept's salary distribution" — and is the natural use of PERCENT_RANK. In an interview, framing the choice between the two and explaining the subtle 0/0 edge case on single-member departments is a strong differentiator.

Computing dept_size in the same window pass

COUNT(*) OVER (PARTITION BY dept) is a constant per partition — every row in the same department gets the same dept_size value. Adding it to the CTE avoids a separate subquery or self-join to find department sizes. The engine can compute both window functions in a single partitioned pass.

Edge case — tied minimums

If two employees in the same department share the lowest salary, both receive pr = 0 and both are returned. The question asks for the lowest-paid employees (plural is natural), and PERCENT_RANK handles this correctly without any special casing.

Dialect notes

PERCENT_RANK is ANSI SQL and is available in PostgreSQL, SQL Server, Oracle, Snowflake, BigQuery, and MySQL 8+. It is notably absent from SQLite (which has no PERCENT_RANK implementation). In Snowflake the integer equality check pr = 0 on a FLOAT column may occasionally fail due to floating-point rounding; use pr < 1e-9 as a defensive alternative. BigQuery returns 0.0 as FLOAT64 and the equality is safe.

Senior signal: PERCENT_RANK = 0 is the exact, tie-aware predicate for the lowest-ranked row(s) in a partition; pair it with COUNT(*) OVER (PARTITION BY dept) in the same CTE to filter small departments in one table pass; and mention the 0/0 definition for single-member partitions and potential float equality issues in Snowflake.
