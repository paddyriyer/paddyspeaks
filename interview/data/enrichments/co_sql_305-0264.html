Expedia's growth team tracks monthly active members of its travel community and wants the month-over-month growth rate on the same dashboard row as the raw count. The query looks like a textbook LAG exercise — until you notice the reference solution wraps every value in a regexp_replace. That is the tell: the members column is not clean numbers. An interviewer asking this wants to see that you handle the data you actually get, not the data you wish you had.

1. The source data — network

2024-0311500
2024-0413800
2024-0512_420

Two things make this interview. March is flat against February — a 0% month, which must come out as exactly 0, not NULL. And May arrives as 12_420 — a thousands separator (or an import artefact) that would make a plain ::numeric cast throw. The members column is text-typed and dirty.

2. The SQL — LAG growth over a scrubbed column

SELECT month, members,
  100.0*(NULLIF(regexp_replace(members::text, '[^-.0-9]', '', 'g'), '')::numeric - LAG(NULLIF(regexp_replace(members::text, '[^-.0-9]', '', 'g'), '')::numeric) OVER (ORDER BY month))/NULLIF(LAG(NULLIF(regexp_replace(members::text, '[^-.0-9]', '', 'g'), '')::numeric) OVER (ORDER BY month), 0) AS mom_growth
FROM network;

regexp_replace(…, '[^-.0-9]', '', 'g') deletes every character that is not a digit, minus, or dot — so 12_420 becomes 12420. NULLIF(…, '') turns an emptied-out string into NULL instead of a cast error, and the outer NULLIF(…, 0) guards against dividing by a zero baseline.

January has no prior month, so LAG returns NULL and the growth is NULL — correct, there is no "growth from nothing". February grows 15%, March is exactly 0% (flat, not NULL), April +20%, and May drops to 12420 for -10%. Note May's value was parsed through the underscore: the regex did its job.

4. Strategic explanations

Why regexp_replace at all — and what it costs

If members were a clean INTEGER column the whole query collapses to a readable line:

SELECT month, members,
  100.0 * (members - LAG(members) OVER (ORDER BY month))
        / NULLIF(LAG(members) OVER (ORDER BY month), 0) AS mom_growth
FROM network;

The scrubbing exists only because the column is text with junk in it. The senior point: scrubbing inside the query is a symptom. The real fix is upstream — clean the data at ingestion and store members as a typed integer, so every downstream query is simple and a malformed value fails loudly at load time instead of silently becoming a wrong number.

The divide-by-zero trap

If a month legitimately had 0 members, the denominator LAG(members) is 0 and x/0 raises division by zero. NULLIF(denominator, 0) converts that into x/NULL = NULL — the growth rate is undefined when you start from zero, and NULL says exactly that. Dropping the NULLIF is the most common way this query crashes in production.

Gaps in the month series

LAG takes the previous row, not the previous calendar month. If March is missing entirely, April's LAG reaches back to February and the "month-over-month" rate silently spans two months. If the series can have holes, generate a complete month spine and left-join the data onto it:

SELECT m.month, n.members,
       n.members - LAG(n.members) OVER (ORDER BY m.month) AS delta
FROM generate_series('2024-01-01'::date, '2024-12-01', '1 month') m(month)
LEFT JOIN network n ON to_char(m.month,'YYYY-MM') = n.members;
NULL members rows

If a row's members is NULL (or becomes NULL after scrubbing an all-junk value), that month's growth is NULL and the next month's growth is NULL too — because LAG hands the next row a NULL baseline. One bad row poisons two output rows. Decide deliberately: skip NULL months with a WHERE, or carry the last good value forward with a gap-fill before computing the rate.

Ordering must be deterministic

ORDER BY month here works because month sorts lexically as YYYY-MM. If months were stored as 'Jan 2024' the ordering would be alphabetical nonsense. Always order a window by a value that sorts chronologically — a real date, or a zero-padded YYYY-MM string.

Senior signal: recognise the regexp_replace as a data-quality band-aid and name the upstream fix (typed column, fail-at-load), keep the NULLIF(denominator, 0) guard, distinguish "previous row" from "previous month" when the series can have gaps, and ensure the window's ORDER BY is genuinely chronological.
