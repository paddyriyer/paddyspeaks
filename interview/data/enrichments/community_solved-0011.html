A Community string-processing question, Google-flavoured: a file store holds documents in a single contents column, and the task is to count how often the words "bull" and "bear" appear across all of them. The trick is that the words live inside a text blob — you cannot count rows, you have to count tokens, which means turning one string into many rows.

1. The source data — google_file_store

filenamecontents
news1.txtthe bull is strong and the bear is weak bull bull
news2.txtbear market continues bear bear bull
news3.txtno animal words here

Each row is a whole document. "bull" appears three times in news1 and once in news2; "bear" once in news1 and three times in news2. news3 has neither word — it must contribute nothing, not a zero row.

2. The SQL — split to tokens, then aggregate

SELECT word AS value, COUNT(*) AS word_count
FROM google_file_store, UNNEST(string_to_array(contents, ' ')) AS word
WHERE word IN ('bull', 'bear')
GROUP BY word;

string_to_array(contents, ' ') splits each document on spaces into an array of words; UNNEST explodes that array so each word becomes its own row. The comma is a lateral cross join — every document row fans out into one row per token. The WHERE keeps only the two words of interest, and GROUP BY counts them.

3. The resulting analysis

valueword_count
bear4
bull4

"bear" totals 4 (1 + 3), "bull" totals 4 (3 + 1) — counted across all documents, not per file. news3 contributed nothing: its tokens were all filtered out before the GROUP BY, so it produces no row rather than a 0. Note also there is no ORDER BY — the two rows could come back in either order.

4. Strategic explanations

Why UNNEST is the heart of the answer

The data is "one document per row" but the question is "per word". That mismatch — a collection packed into a single cell — is exactly what UNNEST exists for: it converts array elements into rows so ordinary aggregation can see them. string_to_array + UNNEST is the canonical Postgres tokeniser. Other engines spell it differently — Spark's explode(split(...)), Snowflake's LATERAL FLATTEN(SPLIT(...)), SQL Server's STRING_SPLIT — but the shape is identical.

The case-sensitivity trap

word IN ('bull', 'bear') is an exact, case-sensitive match. A document containing "Bull" at the start of a sentence, or "BEAR" in a headline, would be missed entirely. News text is full of capitalised words. Normalise before matching:

WHERE lower(word) IN ('bull', 'bear')

and GROUP BY lower(word) so "Bull" and "bull" land in the same bucket.

The punctuation trap — the bigger one

Splitting on a single space leaves punctuation glued to words: "bull,", "bear.", "(bull)" are all different tokens from "bull", and the IN list misses every one. Real prose breaks this query badly. Two fixes: split on a regex of non-word characters, or strip punctuation from each token:

FROM google_file_store,
     regexp_split_to_table(lower(contents), '\\W+') AS word
WHERE word IN ('bull', 'bear')

regexp_split_to_table splits and explodes in one step, and \\W+ treats any run of non-word characters as the delimiter. This is the version that survives contact with actual documents.

Substring matching is the wrong instinct

A tempting shortcut is WHERE contents LIKE '%bull%' — but that counts rows, not occurrences (it cannot tell one "bull" from three), and it fires on "bullish", "bulldozer", "bullet". Whole-word tokenising is the only correct approach for an occurrence count; LIKE answers a different question.

Edge cases: missing words and empty contents

If a target word appears nowhere, it simply will not appear in the output — there is no row to count. If a dashboard needs an explicit bull = 0, drive the query from a small literal list of target words LEFT JOINed to the token counts. Also, a NULL or empty contents yields an empty array, UNNEST produces no rows for it, and it harmlessly drops out — no special handling needed.

Senior signal: reach for UNNEST/regexp_split_to_table to turn packed text into countable rows, pre-empt the case-sensitivity and punctuation traps that wreck a naive space-split on real prose, and explain why LIKE '%word%' counts the wrong thing.
