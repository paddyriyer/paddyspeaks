Google's commerce-analytics team needs every order placed in a reporting window — here, the first two months of 2024 — to feed a quarterly revenue review. BETWEEN makes this a one-liner. But BETWEEN has a famous edge: it is inclusive on both ends, and when the column is a timestamp rather than a pure date, that inclusivity quietly drops the last day of the range. An interviewer asking a "simple" date filter is usually testing exactly this.

Six orders straddling the window. Note the deliberate boundary rows: order 1 on 2023-12-31 (just before), order 2 on 2024-01-01 (the lower bound), order 4 on 2024-02-28 (the literal upper bound), and order 5 on 2024-02-29 — 2024 is a leap year, so Feb has 29 days.

2. The SQL — WHERE ... BETWEEN

SELECT *
FROM orders
WHERE order_date BETWEEN '2024-01-01' AND '2024-02-28';

x BETWEEN a AND b is shorthand for x >= a AND x <= b — inclusive at both endpoints.

But order 5 on Feb 29 is missing: the window literally stops at Feb 28. Whether that is correct depends entirely on whether "the first two months" was meant to include the leap day.

4. Strategic explanations

The timestamp trap — why BETWEEN on a date column is risky

Here order_date is a pure DATE, so BETWEEN '2024-01-01' AND '2024-02-28' captures all of Feb 28. But if the column were a TIMESTAMP, the literal '2024-02-28' is interpreted as 2024-02-28 00:00:00 — midnight. An order placed at 2024-02-28 14:30:00 is after that instant and gets silently dropped. The entire last day vanishes. The robust pattern for timestamp columns is a half-open interval:

SELECT *
FROM orders
WHERE order_date >= '2024-01-01'
  AND order_date <  '2024-03-01';   -- note: < the day AFTER the range

>= start and < next-day is timezone-safe, leap-year-safe, and time-of-day-safe. Many senior engineers use this form even on pure DATE columns out of habit, so the query survives a future schema change to timestamp.

The leap-year boundary

Order 5 sits on Feb 29, 2024. The reference query's upper bound of '2024-02-28' excludes it. If the requirement is "all orders in January and February", the bound is wrong — it should be the last day of February, which in a leap year is the 29th. The half-open form < '2024-03-01' sidesteps the leap-year arithmetic entirely: you never have to know how many days February has.

SELECT * is a code-review smell

SELECT * is fine in an interview scratchpad but flag it for production: it breaks if columns are reordered, ships unnecessary bytes over the wire, and defeats covering indexes. Name the columns the report actually needs — SELECT order_id, order_date, amount.

Edge cases: NULL dates and parameterised ranges

An order with a NULL order_date — perhaps a draft never submitted — is excluded by BETWEEN, because NULL >= '2024-01-01' evaluates to NULL (not TRUE). That is usually right, but worth saying aloud. And in real code the range is rarely a literal — it comes from a parameter or a date dimension. Parameterise it: WHERE order_date >= $1 AND order_date < $2, and compute $2 as start_of_next_period so the half-open invariant holds regardless of which month or quarter the user picks.

Senior signal: know that BETWEEN is inclusive on both ends and that on a timestamp column an upper bound of '2024-02-28' means midnight — silently dropping the last day — so prefer the half-open >= start AND < next-day form, which is also leap-year-proof, and replace SELECT * with named columns for production.
