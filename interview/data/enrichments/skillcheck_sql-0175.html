Computing the gap between the top and second-highest salary per department is a classic window-function question with a subtle wrinkle: departments where every employee earns the same salary have no genuine second-place value, and the gap must surface as NULL rather than zero or disappearing from the result. The interviewer is probing whether you use DENSE_RANK (so tied top earners still allow a true #2 distinct salary), deduplicate to distinct salaries first, and use conditional aggregation rather than a correlated subquery.

Eng has two people tied at rank 1 (salary 200); despite the tie, salary 170 is genuinely the second-distinct salary. Ops has only one distinct salary value so there is no second; same for HR with one employee. Both must produce a NULL gap.

2. The SQL — DENSE_RANK on distinct salaries then conditional aggregation

WITH d AS (
  SELECT dept, salary,
         DENSE_RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS rk
  FROM employee
  GROUP BY dept, salary
)
SELECT dept,
       MAX(salary) FILTER (WHERE rk = 1) AS top_salary,
       MAX(salary) FILTER (WHERE rk = 2) AS second_salary,
       (MAX(salary) FILTER (WHERE rk = 1) - MAX(salary) FILTER (WHERE rk = 2)) AS gap
FROM d
WHERE rk <= 2
GROUP BY dept
ORDER BY dept;

The CTE first deduplicates to distinct (dept, salary) pairs, then ranks them with DENSE_RANK. The outer query pivots ranks 1 and 2 into separate columns using FILTER; when no rank-2 row exists, MAX(…) FILTER (WHERE rk = 2) returns NULL, and arithmetic with NULL propagates NULL to the gap column.

4. Strategic explanations

Why GROUP BY dept, salary before DENSE_RANK

Without deduplication, DENSE_RANK on the raw employee table would give both emp_id 1 and emp_id 2 (both earning 200) rank 1, and emp_id 3 (earning 170) rank 2 — which is actually the correct ranking. However, the outer aggregation would then see two rank-1 rows for Eng, making MAX(salary) FILTER (WHERE rk = 1) still correct. But for correctness guarantees across all inputs, pre-deduplicating to distinct (dept, salary) pairs before ranking is cleaner and more explicit. It also reduces the window-function input size.

DENSE_RANK vs RANK vs ROW_NUMBER

DENSE_RANK assigns consecutive integers to distinct values: ties at rank 1 do not create a gap to rank 3. RANK would skip: two rank-1 rows would make the next row rank 3, skipping rank 2 — so a WHERE rk = 2 filter would find nothing. ROW_NUMBER assigns unique numbers regardless of value, so two identical salaries at the top would get row numbers 1 and 2, making the “second” row another instance of the same salary — wrong. DENSE_RANK is the only correct choice here.

NULL arithmetic for the "no second salary" case

Any arithmetic with NULL returns NULL in SQL: 200 - NULL = NULL. This is exactly the desired behaviour: the gap for HR and Ops is genuinely undefined (not zero), and NULL communicates that clearly. Using COALESCE(gap, 0) would be wrong: a zero gap would imply the top and second salary are equal, which is different from "there is no second salary".

Alternative: correlated subquery

A correlated subquery can find the second-distinct salary: SELECT MAX(salary) FROM employee e2 WHERE e2.dept = e.dept AND e2.salary < top_salary. This is O(n²) in the number of departments × employees, whereas the window-function CTE approach is O(n log n). For small tables either works; the window function approach is the canonical modern answer.

Dialect portability

DENSE_RANK() and the FILTER clause are both PostgreSQL/SQL:2003 features. FILTER is not available in MySQL 5.x or SQL Server before 2022; the equivalent is MAX(CASE WHEN rk = 1 THEN salary END). DENSE_RANK itself is supported in MySQL 8+, SQL Server 2008+, Snowflake, and BigQuery.

Senior signal: deduplicate to distinct (dept, salary) before ranking; use DENSE_RANK (not RANK or ROW_NUMBER) so ties at the top do not swallow the rank-2 slot; rely on NULL arithmetic to produce a NULL gap when no second salary exists (never coerce to 0); and note FILTER vs CASE WHEN dialect portability.
