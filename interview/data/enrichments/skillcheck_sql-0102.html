An HR analytics request asks for the three highest distinct salary levels in each department — and crucially, if several people share one of those top-three levels, every one of them must appear. This is the canonical "top-N with ties" question, and the interviewer has even hinted at it: it is the exact case where DENSE_RANK() ≤ 3 is correct and ROW_NUMBER() ≤ 3 is wrong.

2. The SQL — DENSE_RANK partitioned by department

WITH ranked AS (
  SELECT e.dept_id, e.name, e.salary,
         DENSE_RANK() OVER (PARTITION BY e.dept_id ORDER BY e.salary DESC) AS rk
  FROM employee e
)
SELECT d.dept_name, r.name, r.salary
FROM ranked r
JOIN department d ON d.dept_id = r.dept_id
WHERE r.rk <= 3
ORDER BY d.dept_name, r.salary DESC, r.name;

DENSE_RANK() assigns a rank per department by descending salary; tied salaries get the same rank and no rank is skipped. Filtering rk <= 3 keeps the three highest distinct salary levels — and everyone sitting on them.

4. Strategic explanations

Why DENSE_RANK — "three distinct levels" means dense ranking

The question asks for the three highest distinct salaries, including everyone who earns them. DENSE_RANK ranks by distinct value: in Eng the salaries 200, 180, 160 get ranks 1, 2, 3 regardless of how many people sit on 200. Filtering rk <= 3 therefore captures exactly the top three pay levels. It is the function whose semantics match the phrase "top N distinct values with ties".

Why ROW_NUMBER() ≤ 3 is wrong — the hinted trap

ROW_NUMBER assigns 1, 2, 3, 4, 5 with no regard for ties. In Eng it would number Ann=1, Bo=2, Cy=3, Di=4 — and ≤ 3 would return Ann, Bo, Cy, dropping Di even though 160 is genuinely the third-highest salary. Worse, the tie between Ann and Bo would be broken arbitrarily by physical row order. ROW_NUMBER answers "give me exactly 3 rows"; the question asks "give me 3 salary levels", which is a different thing.

RANK would also be wrong — and differently

RANK shares ranks for ties but skips afterwards: in Eng, Ann and Bo both rank 1, then Cy ranks 3 (not 2), Di ranks 4. RANK ≤ 3 would return Ann, Bo, Cy and drop Di — only the top two distinct levels plus a partial third. RANK counts rows above you; DENSE_RANK counts distinct values above you. For "top N distinct values" you need the dense one.

PARTITION BY restarts the ranking per department

PARTITION BY e.dept_id makes DENSE_RANK reset to 1 at each department boundary, so Sales' 300 is rank 1 independently of Eng's 200. Without the partition you would rank all employees globally and return the top-three salaries company-wide — a completely different report. The partition is what makes this "per department".

Window functions cannot be filtered in WHERE — hence the CTE

DENSE_RANK() is computed after WHERE in SQL's evaluation order, so WHERE DENSE_RANK() OVER (...) ≤ 3 is a syntax error. The rank must be materialised in a CTE (or subquery) first, then the outer query can filter on the named column rk. This layering — rank in a CTE, filter outside — is the universal shape of every top-N-per-group query.

Senior signal: read "top three distinct salaries including ties" as a DENSE_RANK problem — ROW_NUMBER ≤ 3 drops genuine top earners and breaks ties by row order, RANK ≤ 3 skips levels — partition by department, and filter the rank in an outer query because window functions cannot appear in WHERE.
