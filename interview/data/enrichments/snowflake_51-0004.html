A social-media platform must ship a GDPR-compliant data export to a third-party analytics vendor without exposing raw PII. The task is to mask or hash every sensitive field in one SELECT — no temporary tables, no application code. The interviewer is probing knowledge of hashing functions, regex-based character masking, string arithmetic (LEFT, REPEAT, LENGTH), and date extraction, while expecting a discussion of the reversibility risk of MD5 for email addresses.

2. The SQL — multi-function PII anonymization

SELECT
  user_id,
  MD5(email)                                                       AS email_hash,
  REGEXP_REPLACE(phone, '\d(?=\d{4})', '*', 'g')                  AS phone_masked,
  LEFT(first_name, 1) || REPEAT('*', LENGTH(first_name) - 1)      AS first_masked,
  LEFT(last_name,  1) || REPEAT('*', LENGTH(last_name)  - 1)      AS last_masked,
  EXTRACT(YEAR FROM dob::date)                                     AS birth_year_only
FROM users;

Note on dialect: this query uses PostgreSQL syntax. In Snowflake the MD5 function is identical; REGEXP_REPLACE uses a different flag style (no 'g' flag — use REGEXP_REPLACE(phone, '\\d(?=\\d{4})', '*', 1, 0)); EXTRACT(YEAR FROM …) is portable as written; || for concatenation works in both. Snowflake's preferred hashing is SHA2(email, 256) for stronger security.

Phone masking: the regex replaces every digit that is followed by exactly 4 more digits with *, preserving only the last 4 digits. Name masking: the first character is preserved, all remaining characters become *. For "Li" (length 2), REPEAT('*', 1) = '*', giving L*. Birth year collapses the full date to just the year.

4. Strategic explanations

MD5 is a hash, not encryption — and it is reversible for common emails

MD5 is a one-way hash with no key, but email addresses have very low entropy. A lookup table of the ~1B most common email formats can reverse the hash in seconds. For true anonymization, pair the hash with a secret salt: MD5(email || 'secret_salt_value'), or use SHA2(email || salt, 256) (Snowflake). Without a salt, an adversary who knows a user's email can trivially confirm whether that user is in the dataset — defeating pseudonymization under GDPR Article 4.

The regex phone-masking pattern

The pattern \d(?=\d{4}) is a lookahead: it matches any digit that is immediately followed by exactly four more digits. For a 10-digit phone number, this matches digits 1 through 6 (positions 1–6 when the last 4 are positions 7–10). The replacement is *, applied globally ('g' flag in PostgreSQL; the sixth argument 0 meaning "all occurrences" in Snowflake). A simpler approach that is easier to explain in interviews: LPAD(RIGHT(phone::text, 4), LENGTH(phone::text), '*') — replace all but the last 4 characters with asterisks.

Name masking: LEFT + REPEAT + LENGTH

LEFT(name, 1) extracts the initial. REPEAT('*', LENGTH(name) - 1) produces as many asterisks as there are remaining characters. Concatenation with || glues them together. Edge case: a name of length 1 gives REPEAT('*', 0) = '', producing only the initial — still correct. A name of length 0 (empty string) gives REPEAT('*', -1) — PostgreSQL returns '', Snowflake raises no error either, but it is worth guarding with NULLIF(name, '') if empty names are possible.

Date granularity reduction

EXTRACT(YEAR FROM dob::date) reduces a full date to just the year, which cannot re-identify the user on its own. For ages over 90, even the year can be identifying in a small population — GDPR guidance recommends suppressing or further bucketing very old birth years. In Snowflake, DATE_PART('year', dob) and YEAR(dob) are equivalent alternatives.

Dialect portability matrix
  • Snowflake: SHA2(email, 256) preferred over MD5; REGEXP_REPLACE(phone, '\\d(?=\\d{4})', '*', 1, 0); YEAR(dob) or DATE_PART('year', dob)
  • PostgreSQL: MD5(email); regex with 'g' flag; EXTRACT(YEAR FROM …)
  • BigQuery: MD5(email) returns BYTES — wrap in TO_HEX(MD5(email)); REGEXP_REPLACE(phone, r'\d(?=\d{4})', '*') (no flags argument); EXTRACT(YEAR FROM …)
  • MySQL: MD5(email); REGEXP_REPLACE available since 8.0, no flag argument; YEAR(dob)
Senior signal: flag immediately that unsalted MD5 on email is reversible via preimage lookup tables — a GDPR/CCPA risk — and recommend SHA2(email || salt, 256) in Snowflake; explain the lookahead regex for phone masking and offer the simpler LPAD(RIGHT(phone, 4), …, '*') alternative; note the empty-string edge case for REPEAT; and call out BigQuery's TO_HEX(MD5(…)) portability gotcha.
