repartition and coalesce look like the same knob — they both change partition count. They are not. One does a full shuffle and can both grow and shrink; the other is shuffle-free and can only shrink. Picking the wrong one is the #1 cause of "my Spark job got slower after I added repartition(1)". This question separates engineers who memorise API signatures from engineers who can read a Spark DAG.

1. The source data — three real workloads

workloadinput shapegoal
A — Daily user-events ETL50 GB, 4 input files (oversized), current partitions = 4Increase parallelism before a heavy groupBy(user_id) step. 4 tasks on a 200-core cluster = 196 idle cores.
B — End-of-pipeline export2 GB DataFrame, current partitions = 2,000 (after multiple wide ops)Reduce to ~10 files for downstream BI tool that struggles with directories > 50 files.
C — Pre-join co-partitioning500 GB orders, 500 GB shipments, both partitioned on different columnsHash-co-partition both on order_id so the join shuffles only once.

Each workload picks a different operator. Workload A → repartition(200, "user_id"). Workload B → coalesce(10). Workload C → repartition(N, "order_id") on both sides. Wrong picks below.

2. The problem without the right choice

Bug 1 — coalesce(200) when the input has 4 partitions
Input: 4 partitions
                       coalesce only MERGES, never SPLITS
Request: coalesce(200) ────────────────────────────────────► Result: 4 partitions (unchanged)

coalesce can only collapse existing partitions into bigger ones — it cannot create new partitions. Your groupBy(user_id) still runs on 4 tasks. Cluster idle. Job takes 6× longer than expected. The senior tell: coalesce(N) where N > current partition count is a silent no-op.

Bug 2 — repartition(1) before a big write
Input: 2,000 partitions, ~1 MB each = 2 GB total
                                                                   Driver / one executor
Request: repartition(1)  ─── full shuffle, 2 GB lands on ────►  ┌─────────────────────┐
                                                                │  ONE task, 2 GB heap │ ← OOM risk
                                                                └─────────────────────┘

repartition(1) does a full shuffle that funnels every row to a single task. If your DataFrame is bigger than that executor's memory, the job OOMs. coalesce(1) is no shuffle and is usually the right pick to produce a single output file — but watch the upstream stage: coalesce back-propagates its partition count, so the upstream computation also runs on 1 partition. See Probe 3.

Bug 3 — random repartition(200) before a groupBy(user_id)

Random repartition spreads rows uniformly. The subsequent groupBy does another shuffle to hash-distribute by user_id. Two shuffles, one wasted. repartition(200, "user_id") uses the same hash partitioner, so the groupBy sees data already co-located and may skip the shuffle entirely (the Catalyst optimiser is partition-aware).

3. The fix — the runnable example, three workloads

from pyspark.sql import SparkSession, functions as F
from pyspark.sql import Row

spark = SparkSession.builder.appName("repartition-vs-coalesce").getOrCreate()

# A small seeded sample that exhibits all three patterns.
events = spark.createDataFrame([
    Row(user_id=1,  action="login",  amount=10.0),
    Row(user_id=1,  action="buy",    amount=42.0),
    Row(user_id=2,  action="login",  amount=10.0),
    Row(user_id=3,  action="buy",    amount=99.0),
    Row(user_id=4,  action="login",  amount=10.0),
    Row(user_id=4,  action="buy",    amount=18.0),
    Row(user_id=5,  action="buy",    amount=77.0),
    Row(user_id=5,  action="buy",    amount=55.0),
])
print("initial partitions:", events.rdd.getNumPartitions())

# ─── Workload A: increase parallelism before a heavy groupBy ───────────────
# repartition(N, col) = FULL SHUFFLE, hash-partitioned by `col`.
# Gives downstream groupBy a co-located layout — Catalyst skips its own shuffle.
events_for_agg = events.repartition(8, F.col("user_id"))
agg = events_for_agg.groupBy("user_id").agg(F.sum("amount").alias("total"))

# ─── Workload B: collapse to few files before write ────────────────────────
# coalesce(N) = NO SHUFFLE. Merges adjacent partitions on the same executor.
# Cheap. Use when N < current partition count and you don't need balance.
agg.coalesce(1).write.mode("overwrite").parquet("/tmp/user_totals")

# ─── Workload C: pre-join co-partitioning (the senior trick) ───────────────
orders    = spark.createDataFrame([Row(order_id=i, amount=i*10.0) for i in range(20)])
shipments = spark.createDataFrame([Row(order_id=i, carrier="UPS")  for i in range(20)])

# Hash-co-partition BOTH sides on the join key with the SAME N.
# The join now sees identical partitioners on both sides → no shuffle exchange.
N = 8
co_orders    = orders.repartition(N,    F.col("order_id"))
co_shipments = shipments.repartition(N, F.col("order_id"))
joined = co_orders.join(co_shipments, "order_id", "inner")

joined.explain()  # Look for "AQEShuffleRead" / no "Exchange" on the join's children.

4. Where this appears in production

  • Netflix re-partitions viewing-event tables nightly by (region, content_id) with N = 2 × executor_count before training per-region recommendation models. They explicitly avoid coalesce at this step because skewed regions need balancing.
  • Uber's surge-pricing pipeline repartitions trip data by (city_id, hour) ahead of per-city pricing models — same shape as workload A.
  • Databricks' own demo notebooks use coalesce(1) for "save to single CSV for the data scientist", explicitly warning that this collapses upstream parallelism — exactly the trap in Bug 2.
  • Spotify's Scio pipelines use repartition(N, "user_id") before any per-user windowed aggregation so the window operator inherits the partitioning and avoids a second shuffle.
  • Pinterest's content-ranking ETL pre-co-partitions pins and boards on board_id (workload C) before a daily join — saved them 40% of cluster time.

5. Interviewer probes — the edge cases

Probe 1 — "Why pick 200 partitions specifically?"

200 is the default value of spark.sql.shuffle.partitions. The right number is workload-dependent. Two rules of thumb:

  • Target partition size = 128–256 MB. So N ≈ total_data_size / 200 MB. For 50 GB → 250 partitions.
  • N ≥ total_cores × 2-3. So a 100-core cluster wants at least 200-300 partitions to keep cores fed.

Both rules should agree within 2×. If they don't, your data is unusually small or large for the cluster.

Probe 2 — "What's the cost of over-partitioning to 10,000?"

Three costs: (a) Task scheduling overhead — every task carries ~1-5 ms of driver overhead; 10K tasks = 10-50s of pure scheduling. (b) Small Parquet files on write — each partition writes its own file, metadata reads dominate downstream. (c) Shuffle metadata pressure — every map task writes one file per reduce task; with M=10K, R=10K, you get 100M shuffle files. The shuffle manager memory-maps each one. Sweet spot: 100s to low 1000s of partitions per stage.

Probe 3 — "coalesce(1) back-propagates — what does that mean?"

Spark's planner pushes coalesce upstream into the previous stage. So:

df.filter(...).map(...).coalesce(1).write...

doesn't run filter/map on N tasks and then merge — it runs the entire chain on one task. To force parallelism upstream, insert a barrier:

df.filter(...).map(...).repartition(200).coalesce(1).write...
# OR use a separate action / cache:
df.filter(...).map(...).cache().count()   # materialise with parallelism
df.coalesce(1).write...                    # only the final write is sequential
Probe 4 — "How does AQE change this picture?"

Adaptive Query Execution (Spark 3.0+, default on in 3.2+) does three things that overlap with manual repartition:

  • Coalesce post-shuffle partitions: If you set spark.sql.shuffle.partitions = 1000 but the data only needs 50, AQE merges them automatically. So you can pick N on the high side and let AQE shrink it.
  • Skew join handling: AQE splits skewed partitions (those > skewedPartitionFactor × median) into sub-partitions and replicates the matching side. Replaces some manual salting (see customer_growth-0027).
  • Dynamic join strategy: Promotes a shuffle-sort-merge join to a broadcast join if it sees the build side is small enough post-shuffle. Reduces the need for explicit broadcast() hints.

The interview-grade answer: AQE handles "the cluster is misshaping the data" cases. Manual repartition is still needed when YOU know the downstream layout you want (co-partitioning, output file count, hot-key salting).

Probe 5 — "What about repartitionByRange?"

Range-partitions data by sampling. Two use cases: (a) writing sorted Parquet files for predicate pushdown — readers using WHERE col BETWEEN ... skip whole files; (b) avoiding hash skew when one hash bucket would be huge. Cost: a sampling pass over the data (~1% sample) to pick split points. Don't use it inside a tight loop.

Probe 6 — "Is repartition guaranteed to balance perfectly?"

No. repartition(N) without a column does a round-robin and is near-perfectly balanced. repartition(N, col) uses hash partitioning — if your column has skew (one whale customer), one task gets the whale's rows and the others split the rest. Hash partitioning is uniform on keys, not on row counts. Detection: df.groupBy(spark_partition_id()).count().show() — if max/min > 3, you have skew.

Pick repartition(N, col) before a wide op so Catalyst skips the implicit shuffle; pick coalesce(N) for the final write to control file count, with a repartition barrier upstream if needed. With AQE on, lean on Spark to coalesce post-shuffle and split skew; use manual repartition only when you know a layout downstream needs."
