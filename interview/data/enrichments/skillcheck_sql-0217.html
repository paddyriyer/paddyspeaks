The CROSS JOIN LATERAL unpivot is elegant, but LATERAL is not available on every engine or every version. The lowest-common-denominator unpivot — the one that runs literally everywhere — uses nothing but UNION ALL: one SELECT per source column, stacked.

1. The source data — category_watch_summary

main_categorysuperhero_hoursspy_hourscomedy_hoursdrama_series_hours
Film180601000
Series000160

The same wide report as the LATERAL exercise — the target output is identical too.

2. The query — one SELECT per column, stacked

SELECT main_category, 'Superhero' AS sub_category, superhero_hours AS watch_hours
FROM category_watch_summary WHERE superhero_hours > 0
UNION ALL
SELECT main_category, 'Spy', spy_hours
FROM category_watch_summary WHERE spy_hours > 0
UNION ALL
SELECT main_category, 'Comedy', comedy_hours
FROM category_watch_summary WHERE comedy_hours > 0
UNION ALL
SELECT main_category, 'Drama Series', drama_series_hours
FROM category_watch_summary WHERE drama_series_hours > 0
ORDER BY main_category, watch_hours DESC;

4. Strategic explanations

One branch per column

Each SELECT projects three things: the key column unchanged, a string literal naming the genre, and one metric column aliased to watch_hours. Stacking four such branches with UNION ALL turns four columns into four groups of rows.

UNION ALL, never UNION

UNION removes duplicate rows, which forces a sort or hash across the whole result. That work is wasted here — the branches cannot collide — and it is outright wrong if two genres legitimately share a value, since one of the rows would vanish. UNION ALL simply concatenates, which is both correct and cheaper.

The cost: one scan per column

Each branch has its own FROM category_watch_summary, so the base table is read four times — once per column. The LATERAL form reads it once. On this two-row table the difference is nothing; on a wide table with many metric columns, or a large base table, that multiplier is the reason LATERAL or CROSS APPLY exists.

Only the first SELECT names the output

In a UNION chain the result column names come solely from the first SELECT; later branches are matched by position. That is why only branch one writes AS sub_category and AS watch_hours — the other three rely on column order alone. Mismatched ordering across branches is a classic silent bug.

Why this is the portable choice

UNION ALL, string literals and WHERE exist in every SQL engine and every version — no LATERAL, no VALUES table constructor, no vendor UNPIVOT. When the query must run on an unknown or old target, this is the unpivot that is guaranteed to work.

Senior signal: the universal unpivot is UNION ALL — one SELECT per source column, each projecting a literal label and that column. It is portable to every engine, at the cost of one base-table scan per column; reach for LATERAL / CROSS APPLY only when that scan count actually matters.
