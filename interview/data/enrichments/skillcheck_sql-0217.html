This is the compact form: one self-joined reporting chain, GROUP BY the manager, and a two-line HAVING clause that encodes both halves of the rule. It is the version to reach for in production — a single statement the planner optimizes in one shot — and it generalizes neatly to any tier of the org chart.

2. The GROUP BY / HAVING solution

WITH team_metrics AS (
  SELECT e1.manager_id AS senior_mgr_id,
         e1.emp_id     AS report_id,
         CASE WHEN e2.emp_id IS NOT NULL THEN 1 ELSE 0 END AS report_is_manager,
         CASE WHEN e3.emp_id IS NOT NULL THEN 1 ELSE 0 END AS report_is_senior_manager
  FROM employees e1
  LEFT JOIN employees e2 ON e2.manager_id = e1.emp_id
  LEFT JOIN employees e3 ON e3.manager_id = e2.emp_id
  WHERE e1.manager_id IS NOT NULL
)
SELECT m.emp_name AS senior_manager,
       COUNT(DISTINCT tm.report_id) AS direct_reports
FROM team_metrics tm
JOIN employees m ON m.emp_id = tm.senior_mgr_id
GROUP BY m.emp_id, m.emp_name
HAVING SUM(report_is_manager) >= 1
   AND SUM(report_is_senior_manager) = 0
ORDER BY direct_reports DESC, senior_manager ASC;

team_metrics walks two levels below each employee: report_is_manager marks a report who has a report; report_is_senior_manager marks a report who has a grand-report. Grouping by the manager and testing those two sums with HAVING settles the question in one aggregation.

4. Strategic explanations

HAVING with a threshold — the mixed-team fix

SUM(report_is_manager) >= 1 asks for at least one managerial report. A tempting but wrong condition instead demands that every report be a manager (COUNT(managers) = COUNT(reports)). That silently drops a Senior Manager whose team mixes Managers and ICs — exactly Bob, who manages Dave and Eve (Managers) plus Karl (an IC). The threshold is correct; the equality is the bug.

The e3 join is the grandparent fix

report_is_senior_manager depends on the third join, e3 — the report’s report. Checking only report_is_manager (one level down) would wrongly keep a VP sitting above a Director: the VP does manage a Manager. The e3 join looks one level deeper, so a manager whose report is itself a Senior Manager is correctly excluded.

The same roll-up classifies every tier

Add a fourth join and a report_is_exec flag and the aggregated profile classifies any layer by changing only the HAVING: SUM(report_is_manager) >= 1 AND SUM(report_is_senior_manager) = 0 gives Senior Managers; SUM(report_is_senior_manager) >= 1 AND SUM(report_is_exec) = 0 gives Directors; SUM(report_is_manager) = 0 gives front-line managers. One query shape, a tunable predicate.

“Single statement” is real; “single scan” is not

This is one statement, and the planner runs it as a hash self-join feeding one aggregation — efficient and readable. It is not, however, a “single scan of the table.” Determining who manages whom inherently relates employees to itself; wrapping the table in a CTE, or writing the self-join as a FULL OUTER JOIN, does not collapse that to one read. The honest win is one statement with no procedural steps — not one disk pass. In production, also wrap the display name in COALESCE so managers with a missing name are not all folded into a single NULL group.

Senior signal: GROUP BY + HAVING over a self-joined chain states a hierarchy rule in one optimizer-friendly statement. Use threshold tests (SUM(flag) >= 1), not equalities, so mixed teams survive — and be precise about “single-pass”: it means one statement, not one table scan.
