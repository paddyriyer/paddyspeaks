A brokerage's analytics team wants a running profit-and-loss line for each user's holdings: after every trade, what is the cumulative cash position for that user and ticker? The interviewer is testing a running total within a partition — a window SUM with the right PARTITION BY, the right ordering, and a sign convention that makes buys outflows and sells inflows.

Note user 1's AAPL trades are not inserted in time order (trade 3 is dated after trade 2 but they are adjacent); the window's ORDER BY executed_at must impose chronology, not row insertion order.

2. The SQL — signed window SUM partitioned by user and ticker

SELECT user_id, ticker, executed_at,
       ROUND(
         SUM(CASE WHEN action='sell' THEN qty*price ELSE -qty*price END)
           OVER (PARTITION BY user_id, ticker
                 ORDER BY executed_at
                 ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW),
         2) AS running_pnl
FROM trade
ORDER BY user_id, ticker, executed_at;

The CASE assigns each trade a signed cash flow: a sell adds +qty*price (cash in), a buy adds -qty*price (cash out). The window SUM accumulates those signed flows within each (user_id, ticker) partition, ordered by execution time, from the partition's first row through the current one.

4. Strategic explanations

Why the running total needs a window SUM, not a self-join

A pre-window-function approach would self-join each trade to all earlier trades of the same user/ticker and sum — an O(n²) pattern. SUM(...) OVER (PARTITION BY ... ORDER BY ...) does it in a single ordered pass per partition, O(n log n) dominated by the sort. The window function is both clearer and the only scalable option for a real trade table.

The partition key defines what "running" means

PARTITION BY user_id, ticker is the heart of the query. It restarts the accumulator for every distinct user/ticker combination, so user 1's TSLA P&L is computed in total isolation from their AAPL P&L. Drop ticker from the partition and you would get a per-user blended cash position across all instruments — a different, usually wrong, metric. The partition key is the grain of the running total.

ORDER BY executed_at — chronology, not row order

A running total is only meaningful in a defined sequence. Without ORDER BY inside the OVER clause, the SUM would be an unordered total over the whole partition — every row would show the same final number. ORDER BY executed_at imposes chronological accumulation. It also means the result does not depend on physical row insertion order: the engine sorts by timestamp regardless of how rows were stored.

ROWS vs RANGE — and the timestamp-tie subtlety

ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW sums row-by-row. The default frame for an ordered window is RANGE, which treats rows with an equal executed_at as peers and gives them the same cumulative value. If a user fired two AAPL trades at the identical timestamp, RANGE would show both with the post-both total — not a true step-by-step P&L. Writing ROWS explicitly guarantees each row reflects exactly the trades up to and including itself. Adding a tiebreaker like ORDER BY executed_at, trade_id makes the ordering fully deterministic.

The cash-flow sign convention — state it explicitly

The CASE encodes the definition the question gave: a sell is positive (cash in), a buy is negative (cash out), so the running sum is cumulative cash_in - cash_out. This is a simple cash-flow P&L — not a cost-basis or FIFO realized P&L, which would require matching sold lots to specific buy lots. A senior candidate names which definition is in use; ROUND(..., 2) finally trims the result to cents.
