A Community retail dataset has a customer table and an order table. The ask is oddly specific: among customers who placed exactly one order, find the one with the highest customer id, and return that customer's full record. It is a two-step problem — filter to single-order customers, then pick the max — and the interviewer wants to see you handle the join direction and the empty-result case.

1. The source data — customer_tab and order_tab

order_idcustomer_iditem
1011Pen
1021Book
1032Lamp
1043Chair
1054Desk
1064Mat

Customer 1 placed two orders and customer 4 placed two — both disqualified. Customers 2 and 3 each placed exactly one. Among those single-order customers the maximum id is 3. (Customer dimension: 1 Ann, 2 Bob, 3 Cara, 4 Dan.)

2. The SQL — HAVING count = 1, then MAX

with t1 as(
select max(a.customer_id) as customer_id
from(
select customer_id
from order_tab
group by customer_id
having count(*) = 1
) a
)
select c.* from t1
left join customer_tab c
on t1.customer_id = c.customer_id;

The innermost query groups orders by customer and keeps groups of size 1 — the single-order customers. The next layer takes MAX(customer_id) of that set. The outer query joins that single id back to customer_tab to fetch the full row.

3. The resulting analysis

customer_idnamecountrystate
3CaraIndiaMH

Single-order customers are {2, 3}; MAX picks 3; the join resolves 3 to Cara's full record. Customers 1 and 4 are excluded because HAVING count(*) = 1 rejects their two-order groups. Exactly one row out — as the question demands.

4. Strategic explanations

Why HAVING, not WHERE, for "exactly one order"

"Placed one order" is a statement about a count per group, and a count only exists after GROUP BY. WHERE runs before grouping and cannot see count(*). HAVING count(*) = 1 is the correct — and only — way to express it. This is the same WHERE-vs-HAVING reflex interviewers probe repeatedly.

The collapse-then-join structure

MAX(customer_id) over the single-order set guarantees the inner CTE returns exactly one row. That is what makes the final join safe: joining a one-row table to customer_tab can only yield one row. A LEFT JOIN and an INNER JOIN behave identically here — the max id provably exists in customer_tab if referential integrity holds. An INNER JOIN states that expectation more honestly.

A flatter equivalent

The triple nesting is heavier than the problem needs. One subquery does it:

SELECT c.*
FROM customer_tab c
WHERE c.customer_id = (
  SELECT customer_id
  FROM order_tab
  GROUP BY customer_id
  HAVING COUNT(*) = 1
  ORDER BY customer_id DESC
  LIMIT 1
);

Same result, one level of nesting, and the intent — "the customer record whose id equals the largest single-order id" — reads straight off the page.

The empty-result edge case

If no customer placed exactly one order, the inner query returns no rows and MAX(...) over an empty set returns NULL. The outer LEFT JOIN ... ON t1.customer_id = c.customer_id then joins on NULL = c.customer_id, which is never true, so c.* comes back all-NULL — one row of NULLs rather than zero rows. If the consumer expects "no row" when there is no answer, switch the outer join to INNER JOIN so the NULL id matches nothing and the result is genuinely empty.

Customers with zero orders

"Exactly one order" deliberately excludes customers with no orders — they never appear in order_tab, so they form no group and are absent from the count. That is correct for this question. But it is worth confirming with the interviewer that zero-order customers should not count as "single order" — the wording occasionally trips people.

Ties are impossible here — know why

MAX(customer_id) can only ever return one value, so unlike a "max order amount" question there is no tie to resolve. If the ask were instead "the customer with the most orders", ties would be possible and you would need RANK() or a "= (SELECT MAX(cnt))" pattern. Recognising when ties can and cannot arise saves needless defensive code.

Senior signal: express "exactly one order" with HAVING COUNT(*) = 1, flatten the nesting to a single subquery, and call out the empty-set behaviour — a LEFT JOIN on a NULL max id yields a row of NULLs, so use INNER JOIN if "no answer" should mean "no row".
