Bloomberg's pricing desk stores product prices one row per region — long format — but the analyst wants them side by side: one row per product with a US, EU, and APAC column to eyeball regional spreads. This is the classic long-to-wide pivot, and the interviewer is checking whether you reach for the conditional-aggregation idiom and understand why the MAX wrapper is load-bearing.

1. The source data — prices

product_idregionprice
1US100
1EU110
1APAC95
2US200
2EU220
3APAC50

Product 1 is priced in all three regions; product 2 is missing APAC; product 3 exists only in APAC. Those gaps are the interview — a pivot must leave the missing cells as NULL, not drop the product or invent a price.

2. The SQL — conditional aggregation pivot

SELECT product_id,
  MAX(CASE WHEN region='US'   THEN price END) AS us_price,
  MAX(CASE WHEN region='EU'   THEN price END) AS eu_price,
  MAX(CASE WHEN region='APAC' THEN price END) AS apac_price
FROM prices
GROUP BY product_id;

One row per product via GROUP BY. Each CASE picks out the price for one region (and yields NULL for the other regions' rows); MAX then collapses the group, keeping the single non-NULL value.

3. The resulting analysis

3NULLNULL50

Three products, three columns. Product 2's apac_price is NULL — correct, there is no APAC row for it. Product 3 shows up with only apac_price filled — the pivot did not drop it just because two regions are missing. The NULLs are honest "no price here" markers, exactly what the analyst needs to spot coverage gaps.

4. Strategic explanations

Why MAX and not the bare CASE

This is the detail interviewers probe. CASE WHEN region='US' THEN price END on its own returns one value per row: 100 for product 1's US row, NULL for its EU and APAC rows. Without an aggregate you would get three rows back per product, each with two NULLs — not a pivot at all. MAX (inside the GROUP BY) crushes those three rows into one, and because MAX ignores NULLs it surfaces the single real price. MIN or SUM would work identically here — one value per region per product — but see the next point.

The duplicate-price trap

The whole idiom assumes one price per (product, region). If a product had two US rows — 100 and 105 — MAX silently returns 105 and the other is lost. SUM would return 205, a number that was never a real price. Neither is "right"; the data is ambiguous. A senior answer flags it: "this assumes region is unique per product — if not, I need a rule: latest price? average? and I'd add price_date to the key." Pick the aggregate that matches the business rule, do not let MAX hide a data-quality problem.

The native crosstab — and why CASE often wins anyway

Postgres ships a crosstab function in the tablefunc extension that pivots more declaratively. But it requires the extension to be installed, has a fiddly column-definition signature, and still needs the region list spelled out. The conditional-aggregation pattern is pure standard SQL, portable to every engine, and easy to read — which is why it remains the interview-preferred answer. Mention crosstab exists; reach for CASE.

Dynamic regions — the honest limitation

The query hard-codes three regions. A new region — LATAM — will not appear; its rows are simply ignored by every CASE. SQL cannot produce a variable number of columns from data: the column list must be known at query-compile time. If regions are open-ended you either (a) generate the SQL dynamically in application code after a SELECT DISTINCT region, or (b) keep the data long and let the BI tool pivot at render time. Stating this limitation out loud is the senior signal — juniors promise a query that "adapts" and there is no such thing.

Comparing the pivoted columns

Once wide, regional analysis is trivial — but mind the NULLs. us_price - apac_price is NULL whenever either region is missing (product 2, product 3), so a "biggest US-vs-APAC gap" query must decide whether a missing region means "skip this product" or counts as a known zero. COALESCE only if a missing price genuinely means free; usually it means unknown, and NULL should propagate.

Senior signal: use conditional aggregation for the pivot and explain that the MAX wrapper collapses the per-row CASE values into one row per product; flag that the idiom assumes one price per region and that MAX would silently mask duplicates; and state plainly that the region columns are fixed at compile time — SQL cannot pivot an open-ended set without dynamic SQL.
