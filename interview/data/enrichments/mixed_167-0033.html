A finance team wants every customer with their order count and lifetime spend — including customers who have never ordered, so the report covers the whole base. The reference solution LEFT JOINs orders onto customers and aggregates. The interviewer is probing the most-tested asymmetry in SQL: on a LEFT JOIN with no matches, COUNT and SUM behave differently, and a cancelled order with a NULL amount adds a second twist.

Two customers carry the interview. Dave has no orders — he must still appear, with a count of 0. Carol has one order with a NULL amount — her count and her total must not contradict each other.

2. The SQL — customer aggregation over a LEFT JOIN

SELECT c.customer_id, c.name,
       COUNT(o.order_id) AS orders,
       SUM(o.amount)     AS total
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
GROUP BY c.customer_id, c.name;

The LEFT JOIN keeps every customer; non-ordering customers get a synthetic all-NULL orders row. GROUP BY then collapses each customer's rows and the aggregates summarise them.

Dave shows orders = 0 but total = NULL — the headline asymmetry. COUNT(o.order_id) counts non-null order_ids, and Dave has none, so it returns the integer 0. SUM over zero rows (or all-NULL rows) returns NULL, not 0. Carol shows the same total = NULL for a different reason: she has one real order, but its amount is NULL, and SUM of a single NULL is NULL.

4. Strategic explanations

Why COUNT says 0 but SUM says NULL

This is the single most-tested LEFT JOIN fact. For a non-matching customer the join fabricates one row with every orders column NULL. COUNT(o.order_id) counts non-null values of that column — the fabricated NULL is not counted, so the result is a true 0. SUM(o.amount) adds non-null values, and when there are none the SQL standard defines the result as NULL (not 0 — "the sum of nothing" is undefined, not zero). The two aggregates disagree by design.

COUNT(o.order_id) vs COUNT(*) — a critical distinction on a LEFT JOIN

COUNT(*) counts rows, including the fabricated all-NULL row, so Dave would wrongly show 1. COUNT(o.order_id) counts non-null order_ids, correctly giving Dave 0. On a LEFT JOIN you must count a column from the right (optional) table — ideally its primary key, which is never NULL for a real row. Using COUNT(*) here is the classic silent bug.

Make total read 0 with COALESCE

If the report should show 0.00 rather than NULL for non-spending customers, wrap the aggregate: COALESCE(SUM(o.amount), 0) AS total. COALESCE goes outside SUM — it converts the final NULL result to 0. Putting it inside, SUM(COALESCE(o.amount, 0)), also yields 0 here but means something subtly different (it treats each NULL amount as a 0 contribution); for a pure "show 0 not NULL" display, the outer form is the clear choice.

The Carol case — a count/total mismatch that looks like a bug

Carol has orders = 1 but total = NULL, which a reader may flag as inconsistent. It is not a join artefact — it is a genuine NULL amount on a real (cancelled) order. The business question is whether a cancelled order should count at all. If only completed revenue matters, filter inside the join condition — LEFT JOIN orders o ON c.customer_id = o.customer_id AND o.status = 'shipped' — not in a WHERE clause, because WHERE o.status = 'shipped' would discard Dave's NULL row and silently turn the LEFT JOIN back into an inner join.

GROUP BY must list every non-aggregated column

The query groups by c.customer_id, c.name — both selected non-aggregated columns. PostgreSQL would reject GROUP BY c.customer_id alone while selecting c.name (some engines like MySQL permit it and pick an arbitrary name). Because customer_id is the primary key, grouping by it alone is functionally sufficient, and Postgres actually allows GROUP BY c.customer_id to also carry c.name if customer_id is the declared PK — but listing both is the portable, unambiguous habit.

Senior signal: know that on a LEFT JOIN with no match COUNT(right_pk) returns 0 while SUM returns NULL — use COUNT(o.order_id) not COUNT(*), wrap with COALESCE(SUM(...), 0) for a 0 display, and put any status filter in the ON clause so a WHERE predicate does not silently demote the LEFT JOIN to an inner join.
