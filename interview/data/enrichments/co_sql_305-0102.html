JPMorgan's card-issuance team tracks month-over-month change — how many cards were issued this month versus last. The metric is "variance vs previous month", and the natural tool is LAG. The interview is about the first row (which has no previous month), about sorting a month column correctly, and about not confusing absolute change with percentage change.

1. The source data — cards_issued

issue_monthcount
2026-011000
2026-021200
2026-03900
2026-041500
2026-051500

Two edge cases. January is the first month — there is no prior row to compare against. And April and May are flat at 1500 — the variance there should be exactly 0, distinct from a NULL "no comparison".

2. The SQL — LAG over ordered months

SELECT issue_month, count,
  count - LAG(count) OVER (ORDER BY issue_month) AS variance_vs_prev_month
FROM cards_issued;

LAG(count) OVER (ORDER BY issue_month) fetches the previous month's count for each row; subtracting gives the month-over-month delta.

3. The resulting analysis

issue_monthcountvariance_vs_prev_month
2026-011000NULL
2026-021200200
2026-03900-300
2026-041500600
2026-0515000

January's variance is NULL — LAG has no prior row, so it returns NULL and 1000 - NULL is NULL. That is correct: there genuinely is no comparison. February rose 200, March fell 300, April jumped 600, and May is 0 — flat, not NULL. The distinction matters: 0 means "measured, no change"; NULL means "could not measure".

4. Strategic explanations

The first-row NULL — keep it or default it, on purpose

NULL for January is honest. But if a dashboard cannot render NULL, decide what to substitute — and say which you chose:

-- treat the first month's variance as 0
count - LAG(count, 1, count) OVER (ORDER BY issue_month)
-- LAG's 3rd argument is the default when no prior row exists;
-- passing `count` makes (count - count) = 0

-- or COALESCE the whole expression
COALESCE(count - LAG(count) OVER (ORDER BY issue_month), 0)

Both yield 0 for January — but that now looks identical to a genuine flat month. Whether that conflation is acceptable is a product call; flag it rather than silently COALESCE-ing.

Sorting a month column — why '2026-05' as text still works, until it doesn't

ORDER BY issue_month drives the whole window. If issue_month is a TEXT column in YYYY-MM form, lexical sort happens to match chronological sort — '2026-01' < '2026-02' — so it works. But the moment the format varies ('2026-1', 'Jan 2026', '1/2026'), text sort scrambles the timeline and LAG compares the wrong months. Robust code sorts on a real date:

... OVER (ORDER BY to_date(issue_month, 'YYYY-MM'))

Stating "this relies on the month string being zero-padded and ISO-formatted" is the senior catch.

Absolute change vs percentage change

variance_vs_prev_month is an absolute delta. A +600 swing means very different things off a base of 100 versus 100,000. Executives usually want percent:

SELECT issue_month, count,
  ROUND(100.0 * (count - LAG(count) OVER (ORDER BY issue_month))
        / LAG(count) OVER (ORDER BY issue_month), 1) AS pct_change
FROM cards_issued;

And watch the percentage trap: if a prior month's count is 0, the percent change divides by zero. Guard with NULLIF(LAG(count) OVER (...), 0).

Gaps in the month series

LAG means "the previous row", not "the previous calendar month". If March is missing from the table entirely, LAG on April silently reaches back to February and the "month-over-month" delta secretly spans two months. If the series can have holes, generate a complete month spine (generate_series) and LEFT JOIN the data onto it before applying LAG.

Senior signal: explain that LAG returns NULL for the first row and decide deliberately whether to keep that or default it (knowing 0 then looks like a flat month), recognise the query depends on issue_month being ISO-formatted text or be explicit and sort on a date, and distinguish absolute from percentage change — guarding the divide-by-zero when a base month is 0.
