Anthropic's revenue-analytics team wants every product ranked by total sales for a leaderboard. The query aggregates sales per product and applies RANK() over the summed total — but the interviewer is watching whether you understand you can rank over an aggregate in one pass, what RANK does at a tie, and how it differs from DENSE_RANK and ROW_NUMBER.

1. The source data — sales

Each product has several sale rows, so we sum first. The deliberate trap: products 3 and 4 each total 500 — an exact tie at the top. How the ranking handles that tie is the whole question.

2. The SQL — RANK() over SUM(amount)

SELECT product_id, SUM(amount) AS total,
  RANK() OVER (ORDER BY SUM(amount) DESC) AS rk
FROM sales GROUP BY product_id;

GROUP BY product_id collapses each product to one row; the window function then ranks those grouped rows by their SUM(amount), descending.

Products 3 and 4 both get rank 1 — neither outsells the other. Then rank 2 is skipped: the next products land at rank 3. Products 1 and 2 also tie (300 each) and share rank 3, so rank 4 is skipped too. That gap-after-ties behaviour is exactly what RANK means.

4. Strategic explanations

Ranking over an aggregate — one pass, no subquery

The elegant part is RANK() OVER (ORDER BY SUM(amount) DESC): the window function's ORDER BY can reference an aggregate of the same query. The engine applies GROUP BY first, producing one row per product, then evaluates the window over those grouped rows. No nested subquery is needed. A junior often writes a CTE that sums, then a second query that ranks — correct, but unnecessarily verbose.

RANK leaves gaps after ties (competition ranking — "joint 1st, then 3rd"). DENSE_RANK never skips (1, 1, 2, 2, 3). ROW_NUMBER ignores ties entirely and assigns distinct numbers — which one of the tied 500-products gets 1 vs 2 is arbitrary. Name which semantics the leaderboard wants; for a sales leaderboard, RANK matching real-world competition ranking is usually right, but say so.

The ROW_NUMBER trap for "top product"

If the follow-up is "give me the single best-selling product," ROW_NUMBER() OVER (ORDER BY SUM(amount) DESC) = 1 returns exactly one row — but it would pick product 3 or 4 non-deterministically and hide the tie. If the business needs to know there is a tie for first, filter on RANK() = 1 instead and accept multiple rows. Choosing the wrong window function silently drops a real tie.

Determinism within a tie

Even with RANK, the row order among the two 500-products is unspecified — the engine may emit 3 then 4 or 4 then 3. If the output feeds a UI that must be stable, add a tiebreaker to the window and the final query: ORDER BY SUM(amount) DESC, product_id. Ranks stay tied; presentation order becomes reproducible.

NULL amounts and the missing ORDER BY

SUM skips NULL amount values; a product whose every sale is NULL sums to NULL, and NULL sorts last under DESC by default in Postgres — landing it at the bottom rank, which is probably fine but worth a deliberate NULLS LAST. Also: the query has no final ORDER BY, so although ranks are correct, the rows may not arrive in rank order — add ORDER BY rk for a presentable leaderboard.

Senior signal: rank directly over SUM(amount) in a single pass without a redundant subquery, articulate that RANK skips numbers after ties while DENSE_RANK does not and ROW_NUMBER ignores ties, pick the window function to match whether tied products should share a position, and add an explicit tiebreaker plus ORDER BY rk for a stable leaderboard.
