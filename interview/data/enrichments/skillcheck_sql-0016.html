A gaming product wants each game annotated with the player's points carried into that game — the running total of everything they scored before tonight, excluding tonight itself. That "before" is the trap: a standard cumulative sum includes the current row, and the interviewer wants to see whether you can shift the window frame to end one row early instead of subtracting the current value by hand.

1. The source data — score

player_idgame_datepoints(narrative)

12025-01-038before = 10 + 5

Two edge cases earn the marks. Every player's first game has no prior rows, and player 2 has exactly one game — a single-member partition. Both must yield points_before = 0, not NULL.

2. The SQL — cumulative sum with a frame ending one row early

SELECT
  player_id,
  game_date,
  points,
  COALESCE(SUM(points) OVER (
    PARTITION BY player_id ORDER BY game_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING
  ), 0) AS points_before
FROM score
ORDER BY player_id, game_date;

The frame UNBOUNDED PRECEDING AND 1 PRECEDING covers every earlier row in the partition and stops one row short of the current row. PARTITION BY player_id restarts the total per player; COALESCE converts the empty-frame NULL of the first row into 0.

Player 1's last game shows 23, not 35 — tonight's 12 is deliberately outside the frame. Player 2's lone game shows 0: PARTITION BY isolates them, and with no prior rows the frame is empty.

4. Strategic explanations

The 1 PRECEDING upper bound is the exclusion mechanism

A default cumulative sum — SUM(points) OVER (PARTITION BY player_id ORDER BY game_date) — uses the frame UNBOUNDED PRECEDING AND CURRENT ROW, which includes tonight. Changing the upper bound to 1 PRECEDING ends the window one row early, so the current game's points are structurally excluded. This is cleaner than computing the inclusive total and subtracting points — the frame expresses the intent directly, with no arithmetic to get wrong.

The empty frame returns NULL, hence COALESCE

For the first row of every partition the frame UNBOUNDED PRECEDING AND 1 PRECEDING selects zero rows. SUM over zero rows is NULL, not 0 — a sum of nothing is undefined, not empty. COALESCE(..., 0) makes the business meaning explicit: "no games yet" means zero points carried in. Skip the COALESCE and every player's debut game shows NULL, which a downstream calculation may propagate or a chart may drop.

PARTITION BY is what stops totals bleeding across players

Without PARTITION BY player_id the running sum would accumulate across all players in date order — player 2's total would inherit player 1's points. The partition restarts the frame at each player boundary. The single-row partition for player 2 is the proof: it correctly shows 0 because its frame contains no earlier rows from its own partition.

The tie-on-date trap with ROWS

ROWS counts physical rows, so if a player had two games on the same game_date, the frame would treat them as distinct ordered positions — the second same-day game would "see" the first, and which one is first is arbitrary unless you add a tiebreaker. If same-day games should both exclude each other, switch to RANGE BETWEEN UNBOUNDED PRECEDING AND ... or add a unique column to the ORDER BY. For strictly one-game-per-day data this never bites, but say so.

Senior signal: exclude the current row from a running total by ending the frame at 1 PRECEDING rather than subtracting it, wrap the result in COALESCE(..., 0) because an empty frame sums to NULL, and use PARTITION BY so totals never leak between players.
