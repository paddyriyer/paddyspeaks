A subscriptions team wants to find loyal customers: "who ordered in at least three consecutive calendar months?" Jan–Feb–Mar qualifies; Jan, Mar, May does not. The interviewer is probing the gaps-and-islands technique — identifying runs of consecutive values, the canonical hard SQL pattern that has no single built-in.

1. The source data — orders

customer_idorder months (distinct)longest streak
1Jan, Feb, Mar 20253 → qualifies
2Jan, Feb, [gap], Apr, May2 then 2 → fails
3Jan, Mar, May 20251, 1, 1 → fails
4Jul, Aug, Sep, Oct 20254 → qualifies

The edge cases are deliberate. Customer 2 has two separate 2-month runs — neither reaches 3, and they must not be added together. Customer 1 lands exactly on the threshold. Multiple orders in the same month must collapse to one (handled by DISTINCT).

2. The SQL — gaps-and-islands by date offset

WITH months AS (
  SELECT DISTINCT customer_id,
         date_trunc('month', order_date)::date AS m
  FROM orders
),
numbered AS (
  SELECT customer_id, m,
         (m - (ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY m) || ' months')::interval)::date AS streak_key
  FROM numbered_input
),
streaks AS (
  SELECT customer_id, streak_key, COUNT(*) AS streak_len
  FROM numbered
  GROUP BY customer_id, streak_key
)
SELECT DISTINCT customer_id
FROM streaks
WHERE streak_len >= 3
ORDER BY customer_id;

(In the actual solution the second CTE reads FROM months.) Distinct order-months are numbered 1, 2, 3… per customer. Subtracting that many months from each month value makes every month inside one consecutive run collapse to the same anchor date — the streak_key. Grouping by it counts each run's length.

3. The resulting analysis

customer_idsignal
1Jan/Feb/Mar collapse to one streak_key, length 3 — qualifies
4Jul–Oct collapse to one streak_key, length 4 — qualifies

Customers 2 and 3 are absent. Customer 2's two runs (Jan–Feb, Apr–May) each get a different streak_key and a length of 2 — below the threshold, and correctly not summed. Customer 3's three isolated months produce three distinct keys, each length 1.

4. Strategic explanations

The core trick: row number as a "consecutiveness ruler"

If months are truly consecutive, they advance by exactly one month per step — and so does ROW_NUMBER(). Subtract the row number (as months) from the month value and every member of a consecutive run maps to the same constant: month and counter move in lockstep, so their difference is invariant within a run. A gap makes the month jump ahead while the counter does not, shifting the difference — a new run, a new key. Grouping by that difference is the island detection.

Why DISTINCT on months is essential first

Two orders in the same month would otherwise be numbered as two consecutive steps, fabricating a streak from a single month. The first CTE does SELECT DISTINCT customer_id, date_trunc('month', order_date) so each customer/month appears once before numbering. Collapsing to monthly granularity up front is what makes "consecutive months" — rather than "consecutive orders" — the thing being measured.

Why the offset is in months, not days

Months have unequal lengths, so you cannot subtract a fixed number of days. The expression (ROW_NUMBER() || ' months')::interval builds a genuine month interval ('3 months'::interval) and subtracts that. Because every m is already the first of a month (from date_trunc), subtracting whole-month intervals keeps the anchor on a clean month boundary and the lockstep logic exact regardless of February or 31-day months.

Why SELECT DISTINCT at the end

A customer with two qualifying runs (say Jan–Mar and Jul–Oct) would produce two rows in streaks with streak_len >= 3. The question wants the customer listed once, so the final SELECT DISTINCT customer_id deduplicates. Forgetting it would list such a customer twice — a subtle correctness bug that only shows up for customers with multiple long streaks.

The modern alternative — and why this form still matters

This is the classic "row-number difference" gaps-and-islands solution. A more readable variant uses LAG to flag where a new island starts, then a running SUM of those flags as a group id. Either is fine; both beat any attempt to do this with self-joins. Knowing the technique by name — "gaps and islands" — and that no engine has a built-in for consecutive-run detection is the senior signal.

Senior signal: recognise "consecutive months" as a gaps-and-islands problem — collapse to distinct months first, then subtract a per-customer ROW_NUMBER() (as a month interval) so every month in a run shares an anchor key, group to measure run length, and SELECT DISTINCT at the end so a customer with two long streaks is listed once.
