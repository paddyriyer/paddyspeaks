Caching in Spark is the second-most-misused feature after collect(). Half of "I cached the DataFrame and it got slower" reports are cache-on-once-used data, the other half are forgetting to materialise the cache before reading from it. This question separates engineers who think df.cache() is magic from those who can reason about Spark's lazy evaluation and the storage-level trade-offs.

1. The source data — a real ETL with a fan-out

propertyvalue
base DataFrameenriched_events = 18 stages of joins + parsing on raw event logs
compute cost to materialise~22 minutes on a 50-executor cluster
downstream consumers in the same job4 outputs: by_country, by_device, by_hour, fraud_signals
size in memory (cached)~38 GB (Parquet on disk would be ~12 GB compressed)
cluster memory available200 GB across executors

The trap: without caching, Spark re-executes the 22-minute lineage 4 times, once per consumer = 88 minutes of pure recomputation. With caching, the lineage runs once, then 4 cheap scans of cached data ≈ 26 minutes total. The 62-minute savings is the entire reason cache exists. But cache also has costs (memory, eviction, GC) that bite if you cache the wrong thing.

2. The problem without cache

                     ┌─── by_country.write    (recomputes 22 min lineage)  ── 22 min
   raw_events ───┐   ├─── by_device.write     (recomputes 22 min lineage)  ── 22 min
                 │   │
   18-stage  ────┴───┤
   lineage           │
                     ├─── by_hour.write       (recomputes 22 min lineage)  ── 22 min
                     └─── fraud_signals.write (recomputes 22 min lineage)  ── 22 min

   Wall-clock (no cache):  ~88 minutes (linear in consumer count)

Spark is lazy. Every write (an action) triggers the full lineage from the start. There's no automatic memoisation across actions. The fix is to insert an explicit checkpoint — either in-memory (cache/persist) or on-disk (checkpoint).

The other failure mode — caching without materialising
# BUG: cache() is lazy. It only registers intent. The cache is empty until an action triggers it.
df.cache()
df.filter(...).write(...)   # ← First action: lineage runs AND cache fills, but in parallel.
                            #   The filter sees the live RDD, not the cache.
df.groupBy(...).write(...)  # ← This one DOES read from the warm cache.
# Net: you got cache benefit only for the 2nd consumer, not the 1st. 30% of the win is lost.

The fix: materialise the cache deliberately with an action that reads every row (df.count()) before branching to consumers.

3. The fix — cache, persist, materialise, unpersist

from pyspark.sql import SparkSession, functions as F
from pyspark.sql import Row
from pyspark import StorageLevel

spark = SparkSession.builder.appName("cache-and-performance").getOrCreate()

# ─── Seeded sample so the example is runnable. In production this would be the 18-stage lineage. ───
raw_events = spark.createDataFrame([
    Row(event_id=1, user_id=10, country="US", device="ios",     ts="2026-01-01 09:01", amount=12.0),
    Row(event_id=2, user_id=11, country="US", device="android", ts="2026-01-01 09:02", amount=88.0),
    Row(event_id=3, user_id=12, country="DE", device="ios",     ts="2026-01-01 09:05", amount=5.0),
    Row(event_id=4, user_id=10, country="US", device="ios",     ts="2026-01-01 10:01", amount=110.0),
    Row(event_id=5, user_id=13, country="IN", device="android", ts="2026-01-01 10:15", amount=3.5),
    Row(event_id=6, user_id=11, country="US", device="android", ts="2026-01-01 11:00", amount=44.0),
])

# Pretend this is the expensive 18-stage lineage.
enriched = (raw_events
            .withColumn("ts", F.to_timestamp("ts"))
            .withColumn("hour", F.hour("ts"))
            .withColumn("amount_usd", F.col("amount") * 1.0))   # currency conversion etc.

# ─── Step 1: choose storage level deliberately ────────────────────────────
# MEMORY_AND_DISK = the safe default. Fits in RAM when possible; spills to disk on pressure.
# Why not the default cache() (= MEMORY_ONLY)? If executor RAM fills, MEMORY_ONLY EVICTS partitions
# and Spark silently recomputes them from lineage on the next read. MEMORY_AND_DISK spills instead.
enriched.persist(StorageLevel.MEMORY_AND_DISK)

# ─── Step 2: materialise the cache with a forcing action ──────────────────
# .count() touches every partition, populating the cache before any consumer runs.
# Without this, the first consumer "races" the cache fill and may not benefit.
enriched.count()

# ─── Step 3: fan out — every consumer below reads from the warm cache ─────
enriched.groupBy("country").agg(F.sum("amount_usd").alias("total")).write.mode("overwrite").parquet("/tmp/by_country")
enriched.groupBy("device").agg(F.count("*").alias("events")).write.mode("overwrite").parquet("/tmp/by_device")
enriched.groupBy("hour").agg(F.avg("amount_usd").alias("avg_amount")).write.mode("overwrite").parquet("/tmp/by_hour")
enriched.filter(F.col("amount_usd") > 100).write.mode("overwrite").parquet("/tmp/fraud_signals")

# ─── Step 4: release the memory ──────────────────────────────────────────
# Without this, the cache lives until the session ends. If the same job runs another stage
# that needs memory, you'll OOM. blocking=True waits for eviction (safer in scripts).
enriched.unpersist(blocking=True)

LevelRAMDiskSerialised?Replicated?When to use
MEMORY_ONLY (= cache())YesNoNo1×Small DF, lots of RAM, OK with silent recompute on eviction
MEMORY_AND_DISKYesYesNo1×Default for PySpark cache(); safe everywhere
MEMORY_ONLY_SERYes (serialised)NoYes1×RAM pressure, Java objects bigger than serialised; trade CPU for memory
MEMORY_AND_DISK_SERYes (serialised)YesYes1×Big DF, RAM-bound cluster

For PySpark specifically, Java objects are NOT a concern — Python rows live in the Python worker, while the cached DataFrame is in the JVM Tungsten binary format. So MEMORY_ONLY vs MEMORY_ONLY_SER is less meaningful in PySpark than in Scala. The senior PySpark default is MEMORY_AND_DISK.

4. Where this appears in production

  • Netflix's ad-attribution job caches the joined events × impressions DataFrame (~250 GB in memory) because it's consumed by 7 downstream reports. Skipping cache cost them 6× cluster-hours per night.
  • Uber's surge model training pipeline caches the feature DataFrame between feature engineering and model fitting. They use MEMORY_AND_DISK_SER because the DF is 800 GB and the cluster has 600 GB RAM — spilling is expected, recomputation is not.
  • Stripe's reconciliation job caches the daily matched_transactions DataFrame, then runs 3 different audit queries against it. The team explicitly added a .count() after .cache() in their template after a junior dev's PR caused intermittent doubled run-times.
  • Databricks' notebook UX caches DataFrames when you hit "Run All" in a notebook with downstream cells — but their support docs flag that the cache lives across notebook re-runs and can starve later cells. spark.catalog.clearCache() on notebook entry is the typical fix.
  • Iterative ML in MLlib (k-means, ALS) caches the training data because every iteration scans it. Without caching, each of N iterations re-reads the source → N× IO cost.

5. Interviewer probes — the edge cases

Probe 1 — "Cache or persist — what's the difference?"

df.cache() is exactly df.persist(StorageLevel.MEMORY_AND_DISK) in PySpark (and MEMORY_ONLY in Scala — a notorious cross-language gotcha). persist() lets you pick any storage level; cache() uses the default. Use persist when you want explicit control or you're reading code that should be self-documenting. Senior-grade code uses persist(StorageLevel.X) with an inline comment naming the trade-off.

Probe 2 — "When should you NOT cache?"
  • Used only once. Cache writes the partitions to memory (cost), then the consumer reads them back. If there's only one consumer, the round-trip is wasted work. Rule: cache only DataFrames used 2+ times.
  • Cheap to recompute. A simple spark.read.parquet("...").filter("ds = '2026-01-01'") on a partition-pruned source might run in 30 seconds. Caching that takes more memory than recomputing costs CPU. Cache the expensive lineage, not the cheap one.
  • Memory pressure. If the cluster is already RAM-bound, adding a cache evicts other partitions or spills to disk, hurting more than it helps. Check the Storage tab in Spark UI: green = fully cached, red = thrashing.
  • Single-column projections. Caching a 100-column DataFrame and then only reading 1 column = 100× memory waste. Project columns BEFORE caching: df.select("a","b","c").cache().
Probe 3 — "What's the difference between cache() and checkpoint()?"

Cache stores the DataFrame in memory/disk and keeps the lineage — if a partition is lost, Spark recomputes from the lineage. checkpoint() writes the DataFrame to a reliable storage (HDFS/S3) and truncates the lineage — Spark forgets how to recompute it. Use checkpoint when:

  • Lineage is too long (100+ stages) — the DAG itself becomes expensive to serialise to executors.
  • Iterative algorithms (PageRank, ALS) — each iteration's lineage references the previous, growing without bound. Periodic checkpoint truncates it.
  • You want job restart safety — cached data is lost on driver crash; checkpoint survives.
Probe 4 — "You forgot unpersist(). What happens?"

The cache lives until either (a) the SparkSession ends, (b) memory pressure causes LRU eviction, or (c) spark.catalog.clearCache() is called. In short jobs, no harm. In long-lived sessions (notebooks, structured streaming, Spark Connect servers), un-released caches are the #1 cause of "the cluster slowly fills up and then OOMs at 3 AM". Treat persist/unpersist like malloc/free: pair them in the same code block.

Probe 5 — "How do you verify the cache is actually working?"

Three checks:

  • Spark UI → Storage tab. Shows the cached RDD/DataFrame, fraction cached, size in memory, size on disk. Look for "100% cached" and reasonable size.
  • SQL UI → Query plan. The cached DataFrame appears as InMemoryTableScan instead of the original source/lineage. If you still see the source scan, the cache isn't being used.
  • Wall-clock comparison. Time the second action; it should be 10-100× faster than the first if cache is working.
Probe 6 — "What about spark.catalog.cacheTable for SQL tables?"

For temporary views (createOrReplaceTempView), use spark.catalog.cacheTable("view_name"). Same semantics as df.cache(), but you reference by name from SQL. Useful in notebooks where you mix SQL and DataFrame API. spark.catalog.isCached("view_name") tells you if it's hot. Unpersist with spark.catalog.uncacheTable("view_name").
