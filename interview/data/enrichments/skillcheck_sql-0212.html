“Average rating per course” sounds trivial — until the naive arithmetic SUM(stars) / COUNT(*) quietly returns the wrong number. Both operands are integers, so the division is integer division: the fractional part is discarded before you ever see it. An average of 4.5 prints as 4.

2. The trap — SUM(stars) / COUNT(*)

SELECT course_id, SUM(stars) / COUNT(*) AS avg_rating
FROM ratings GROUP BY course_id;

SUM(stars) and COUNT(*) are both integer types, and integer ÷ integer in SQL truncates toward zero. Course 100 computes 9 / 2 = 4 (not 4.5); course 200 computes 7 / 3 = 2 (not 2.33). The query runs without error and returns plausible-looking whole numbers — the bug is invisible unless you know the true averages.

3. The correct query

SELECT course_id, ROUND(AVG(stars), 2) AS avg_rating
FROM ratings
GROUP BY course_id
ORDER BY course_id;

4. Strategic explanations

Why AVG is safe and manual division is not

AVG() over an integer column returns an exact numeric in PostgreSQL — it does not inherit the integer type of its input. SUM() / COUNT(), by contrast, divides one integer by another, and SQL’s / operator stays in integer arithmetic when both sides are integers. Prefer the built-in AVG precisely because it sidesteps the coercion question.

Forcing fractional division when you must divide manually

If a hand-rolled ratio is unavoidable (e.g. a weighted average), cast one operand: SUM(stars)::numeric / COUNT(*), or multiply by 1.0: 1.0 * SUM(stars) / COUNT(*). Casting after the division ((SUM(stars) / COUNT(*))::numeric) is too late — the truncation has already happened. The cast must reach the operands.

ROUND for presentation

ROUND(AVG(stars), 2) trims the long numeric tail to two decimals for display. ROUND(numeric, int) is well defined in PostgreSQL; if the value were a double precision, the two-argument ROUND would not exist and you would cast to numeric first.

Dialect notes

Integer division is the default for / in PostgreSQL, SQL Server, Oracle, and SQLite. MySQL is the exception — its / always yields a decimal (use DIV for integer division). BigQuery’s / returns a float. Because the bank targets PostgreSQL, the truncation trap is live here.

Senior signal: in PostgreSQL integer / integer truncates, so SUM(x)/COUNT(*) silently floors the average. Use AVG() (returns exact numeric), or, if dividing by hand, cast an operand — SUM(x)::numeric / COUNT(*) — before the division, not after.
