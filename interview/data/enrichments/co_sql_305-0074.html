The CFO opens the daily revenue dashboard and immediately scans for the cumulative line: "are we ahead of last quarter's pace at this point in the month?" Running totals are how every finance dashboard answers that question. The query is a one-line window function — the part that goes wrong in the real world is the date-spine and the partition.

1. The source data — orders

order_dateamount

Notice April 5 is missing — a weekend with zero orders. This is where naïve queries silently produce wrong charts.

2. The SQL — SUM as window function over date

SELECT
  order_date,
  amount,
  SUM(amount) OVER (ORDER BY order_date) AS running_total
FROM orders;

Each row's running_total = sum of amount from the start through this row.

4. Strategic explanations

Why SUM() OVER (ORDER BY date) works without PARTITION BY

Omitting PARTITION BY means the entire result set is one partition. ORDER BY order_date ranks rows within it. The default frame for ORDER BY with no explicit frame clause is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — i.e. "everything from the beginning up to and including this row." Sum that frame → running total.

Explicit frame — be precise

Implicit frames are a portability gotcha. Make it explicit:

SUM(amount) OVER (
  ORDER BY order_date
  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
) AS running_total

ROWS = count physical rows, fast. RANGE = include all rows with the same ORDER BY value (matters when there are ties on order_date). For most running totals on date columns, ROWS is what you actually want and is faster.

The missing-day trap — date spine before SUM

April 5 had zero orders. The running total jumps from $4,430 to $6,080 with no intermediate row. If the dashboard plots running_total on a continuous date axis, April 5 is missing — the chart line connects April 4 directly to April 6, which looks like nothing happened. Production fix: build a date spine first, LEFT JOIN the orders, then run the cumulative:

WITH dates AS (
  SELECT generate_series(DATE '2026-04-01', DATE '2026-04-07', '1 day'::interval)::DATE AS d
  -- Or in SQLite: WITH RECURSIVE dates(d) AS (
  --   SELECT '2026-04-01' UNION ALL SELECT date(d, '+1 day') FROM dates WHERE d < '2026-04-07'
  -- )
)
SELECT d.d, COALESCE(o.amount, 0) AS amount,
  SUM(COALESCE(o.amount, 0)) OVER (ORDER BY d.d) AS running_total
FROM dates d
LEFT JOIN orders o ON o.order_date = d.d
ORDER BY d.d;

Now April 5 shows as amount = $0, running_total = $4,430 — the chart has a flat segment showing the actual zero day.

Per-region or per-product running total — PARTITION BY

The CFO follow-up: "break down the running total by region — I want to see how each is pacing."

SUM(amount) OVER (
  PARTITION BY region
  ORDER BY order_date
  ROWS UNBOUNDED PRECEDING
) AS running_total_by_region

Each region resets to zero at its first row. Without PARTITION BY, regions all share one accumulator — wrong.

"Last 7 days running" — sliding window, not cumulative

Drop UNBOUNDED PRECEDING and put a numeric anchor instead — you have a sliding window:

SUM(amount) OVER (
  ORDER BY order_date
  ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
) AS rolling_7d_total

Useful for "how much did we sell in the last 7 days?" trailing-window dashboards.

Year-to-date that resets at January

The CFO's actual ask is usually YTD: cumulative from Jan 1 of the current year, then resets. Two PARTITION BYs:

SUM(amount) OVER (
  PARTITION BY EXTRACT(YEAR FROM order_date)
  ORDER BY order_date
  ROWS UNBOUNDED PRECEDING
) AS ytd_total
Senior signal: articulate ROWS vs RANGE and pick ROWS with explicit frame, name the missing-day trap and bring up the date-spine fix unprompted, and show the YTD partition variant. The sliding-window vs cumulative-window distinction is the senior detail.
