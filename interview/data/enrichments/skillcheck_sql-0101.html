A tennis-stats system stores Grand Slam winners in a wide layout — one row per year, with four columns (wimbledon, fr_open, us_open, au_open) each holding the winning player's id. The task is to total each player's titles across all years and all four tournaments. The interviewer is checking whether you can unpivot four columns into a single stream of wins, then aggregate — the inverse of a pivot.

1. The source data — players and championships

players maps ids to names: 1 Federer, 2 Nadal, 3 Djokovic, 4 Murray, 5 Wawrinka. The edge cases: player 5 (Wawrinka) wins exactly one title — the smallest non-zero total — and players 1 (Federer) and 4 (Murray) win nothing across all five years, so they must be entirely absent from the output, not shown with a zero.

2. The SQL — UNION ALL to unpivot, then GROUP BY

WITH wins AS (
  SELECT wimbledon AS player_id FROM championships
  UNION ALL SELECT fr_open  FROM championships
  UNION ALL SELECT us_open  FROM championships
  UNION ALL SELECT au_open  FROM championships
)
SELECT p.player_id, p.player_name, COUNT(*)::int AS grand_slam_count
FROM players p
JOIN wins w ON w.player_id = p.player_id
GROUP BY p.player_id, p.player_name
ORDER BY grand_slam_count DESC, p.player_id;

Four SELECTs stacked with UNION ALL turn the four winner columns into one column of player_ids — one row per title. An inner JOIN to players attaches names, and COUNT(*) per player totals the titles.

4. Strategic explanations

This is an unpivot — UNION ALL is the portable tool

The four winner columns are really one logical fact ("a player won a slam") spread across columns. To aggregate it you must first unpivot: collapse the four columns into one. Stacking four SELECTs with UNION ALL — each projecting one column under a shared alias — is the universal way to do that. It is the exact inverse of the SUM(CASE) pivot, and works in every SQL engine.

UNION ALL, never UNION — this is the bug that loses the offer

UNION de-duplicates its combined result; UNION ALL keeps every row. Here the rows are deliberately repetitive — Djokovic's id 3 appears many times. UNION would collapse all those 3s into a single row, and COUNT(*) would return 1 for every player. You must use UNION ALL so each title survives as its own row. UNION also pays an unnecessary sort/hash cost to dedupe. When counting, UNION ALL is both correct and faster.

Why the inner JOIN correctly drops zero-win players

The spec says players with zero titles must not appear. An inner JOIN between players and wins emits a row only when a player id matches a win — Federer (1) and Murray (4) never appear in wins, so they produce nothing. This is the desired behaviour for free. A LEFT JOIN would keep them with grand_slam_count = 0 (after a COUNT of the nullable joined column) — correct only if the spec wanted zeros. Match the join type to the requirement.

COUNT(*) vs COUNT(column) after the join

Post-join, every row represents one real title, so COUNT(*) is right. Be careful with COUNT(w.player_id): it would also be correct here (the joined id is never NULL), but if you had used a LEFT JOIN, COUNT(*) would count 1 for an unmatched player (the row still exists) while COUNT(w.player_id) would correctly count 0. The two diverge precisely on outer-join NULLs — know which you need.

The type tag is misleading

The question is tagged "SQL / Window Functions", but there is no window function here — it is unpivot + GROUP BY. Do not force a RANK() or SUM() OVER () where a plain aggregate suffices. (You could add RANK() OVER (ORDER BY COUNT(*) DESC) as a "title rank" column if asked, but the question does not ask for it.) Solve the question, not the label.

Senior signal: recognise four winner columns as needing an unpivot, do it with UNION ALL — never UNION, which would dedupe the repeated ids and break the count — then total with COUNT(*), and use an inner JOIN so zero-win players drop out without an explicit filter.
