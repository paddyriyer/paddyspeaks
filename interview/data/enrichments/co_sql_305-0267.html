Hulu's recruiting-ops team reports "time to fill" — the average number of days a job requisition stays open before someone is hired. It is one AVG over a date difference, but the interview is in the WHERE clause: requisitions that are still open have no filled_at date, and if you let them in, the metric quietly breaks.

1. The source data — requisitions

req_idopened_atfilled_at
12024-01-012024-01-31
22024-02-012024-02-15
32024-03-012024-05-01
42024-04-01NULL

Three reqs were filled; req 4 is still open — its filled_at is NULL. That open req is the whole question: a half-finished requisition has no cycle time yet, so it must be excluded from the average, not counted as zero.

2. The SQL — AVG(filled_at - opened_at)

SELECT AVG(filled_at - opened_at) AS avg_cycle_days
FROM requisitions
WHERE filled_at IS NOT NULL;

Subtracting two date values in Postgres yields an integer number of days. AVG over those integers gives the mean cycle time; the WHERE keeps only completed requisitions.

3. The resulting analysis

avg_cycle_days
35.0000000000000000

The three filled reqs took 30, 14, and 61 days; (30 + 14 + 61) / 3 = 35. Req 4 never entered the arithmetic. The result is a high-precision numeric — AVG of integers promotes to numeric so the mean is not truncated — which a dashboard would round to 35 days.

4. Strategic explanations

Why the WHERE is non-negotiable

Drop the filter and req 4 contributes NULL - '2024-04-01' = NULL. AVG skips NULLs, so the number stays 35 — which feels safe but is a trap. The danger is the opposite mistake: someone "fixes" open reqs by coalescing the gap to 0, or by measuring against today:

-- WRONG: counts a still-open req as a 0-day fill
SELECT AVG(COALESCE(filled_at, opened_at) - opened_at) FROM requisitions;
-- pulls the average DOWN to 26.25 with a fake zero

An open requisition has no cycle time — not a zero one. The WHERE filled_at IS NOT NULL states that explicitly. Say it out loud in the interview: "I'm averaging only filled reqs; open ones have no cycle time yet."

The AVG-skips-NULL subtlety

Even without the WHERE, AVG(filled_at - opened_at) returns 35, because the NULL difference is ignored by AVG in both numerator and denominator. So the filter is not strictly needed for correctness of the number — but it is needed for clarity of intent, and it matters the instant you also want COUNT(*): with the filter, COUNT(*) is "reqs filled"; without it, COUNT(*) is "all reqs" and disagrees with the average's denominator.

"Aging" open reqs — the follow-up

Recruiters care about open reqs too: how long has each been sitting? That is a different metric — measure against the current date, and only for unfilled reqs:

SELECT req_id, CURRENT_DATE - opened_at AS days_open
FROM requisitions
WHERE filled_at IS NULL
ORDER BY days_open DESC;

Never blend the two: completed cycle time uses filled_at, current age uses CURRENT_DATE. Averaging them together compares finished races with runners still on the track.

Timestamp columns change the units

If opened_at and filled_at were timestamp rather than date, subtraction yields an interval, not an integer — and AVG(interval) returns an interval like 35 days 04:00:00. To force whole days, extract first:

AVG(EXTRACT(EPOCH FROM (filled_at - opened_at)) / 86400) AS avg_cycle_days
Median beats mean for cycle time

One six-month executive search drags the mean up and misrepresents the typical req. Recruiting metrics are usually reported as a median:

SELECT PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY filled_at - opened_at)
         AS median_cycle_days
FROM requisitions
WHERE filled_at IS NOT NULL;

Offering the median unprompted shows you understand the metric, not just the syntax.

Senior signal: exclude open requisitions because they have no cycle time yet — and never paper over them with a coalesced zero — know that date - date is integer days while timestamp - timestamp is an interval, keep completed-cycle and aging metrics separate, and suggest a median for a skew-prone duration.
