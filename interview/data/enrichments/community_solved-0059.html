An order-management system needs to roll up many order statuses into a single quote status. The rules are a priority ladder — Delivered beats Submitted beats Created — with special wording when statuses are mixed. This is the "aggregate child states into a parent state" pattern behind every order-tracking, ticket-triage, or pipeline-stage dashboard. The interviewer wants to see you translate a tangle of business rules into deterministic SQL.

1. The source data — order_status

Quote 2 is the key case — a mix of Delivered and Submitted. The rules say one-or-more (but not all) Delivered means "in delivery", so the presence of any Delivered must outrank the Submitted sibling.

2. The SQL — priority ranking joined to a per-quote status count

WITH priorities AS (
    SELECT quote_id, order_id, order_status,
        CASE order_status
            WHEN 'Delivered' THEN 1
            WHEN 'Submitted' THEN 2
            WHEN 'Created'   THEN 3
            ELSE 4
        END AS priority
    FROM order_status
),
ranked AS (
    SELECT p.*,
        ROW_NUMBER() OVER (PARTITION BY quote_id ORDER BY priority) AS rn
    FROM priorities p
),
per_quote AS (
    SELECT quote_id, COUNT(DISTINCT order_status) AS distinct_status_count
    FROM order_status
    GROUP BY quote_id
)
SELECT r.quote_id,
    CASE
        WHEN r.order_status = 'Delivered' AND q.distinct_status_count = 1 THEN 'complete'
        WHEN r.order_status = 'Delivered' AND q.distinct_status_count > 1 THEN 'in delivery'
        WHEN r.order_status = 'Submitted'                                THEN 'awaiting for submission'
        ELSE 'awaiting for entry'
    END AS quote_status
FROM ranked r
JOIN per_quote q ON q.quote_id = r.quote_id
WHERE r.rn = 1
ORDER BY r.quote_id;

Each order gets a numeric priority; ROW_NUMBER picks the top-priority order per quote (rn = 1); a separate GROUP BY counts how many distinct statuses the quote has, which the final CASE uses to tell "all delivered" from "mixed".

4. Strategic explanations

Why a numeric priority plus ROW_NUMBER works

The rules form a strict ladder, so the natural model is a rank: Delivered=1, Submitted=2, Created=3. ROW_NUMBER() OVER (PARTITION BY quote_id ORDER BY priority) sorts each quote's orders by that rank and tags the winner rn = 1. Filtering to rn = 1 reduces every quote to its single highest-priority order — the row whose status drives the verdict.

Why the separate distinct_status_count join is needed

Knowing the top-priority status alone cannot distinguish "every order Delivered" from "one Delivered among others" — both have a Delivered winner. You need a second fact: how many distinct statuses the quote has. COUNT(DISTINCT order_status) = 1 means uniform (all Delivered → "complete"); > 1 means mixed (→ "in delivery"). The two CTEs answer two different questions and are joined back together.

The single-pass alternative — conditional aggregation

The whole thing collapses to one GROUP BY with boolean aggregates, no window function, no join:

SELECT quote_id,
  CASE
    WHEN BOOL_AND(order_status = 'Delivered')                          THEN 'complete'
    WHEN BOOL_OR(order_status = 'Delivered')                           THEN 'in delivery'
    WHEN BOOL_OR(order_status = 'Submitted')                           THEN 'awaiting for submission'
    ELSE 'awaiting for entry'
  END AS quote_status
FROM order_status
GROUP BY quote_id;

BOOL_AND is "all", BOOL_OR is "at least one", and the CASE branches are evaluated top-down so the order encodes the priority. This is shorter, one pass, and reads almost like the spec. In MySQL use MIN/MAX over the boolean expression; in SQL Server use CASE with COUNT.

The unknown-status and ordering traps

The ELSE 4 priority branch quietly absorbs any status that is not one of the three known values — a typo like 'Delivred' or a new 'Cancelled' state lands in priority 4 and silently maps to "awaiting for entry". A defensive design alerts on unknown statuses instead of bucketing them. Also: if two orders tie on priority, which one ROW_NUMBER labels rn = 1 is arbitrary — harmless here because tied rows share the same status, but a habit worth noticing. The BOOL_AND/BOOL_OR version sidesteps the tie issue entirely.

Senior signal: recognise the priority ladder and collapse the three-CTE window solution into a single GROUP BY with BOOL_AND/BOOL_OR — "all" vs "at least one" maps directly onto the business rules — and flag that the ELSE branch silently swallows unknown statuses instead of alerting.
