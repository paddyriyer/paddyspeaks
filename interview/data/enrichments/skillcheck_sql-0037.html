A banking team needs a running account balance: a ledger of deposits, withdrawals, and refunds, with a cumulative balance after every transaction. The interviewer is probing whether you can sign each transaction by type with a CASE, then carry a running sum with a partitioned window — and whether your ordering is deterministic when two transactions share a timestamp.

1. The source data — txn

idaccounttskindamount
1A2025-01-01 09:00deposit100
2A2025-01-02 10:00withdrawal30
3A2025-01-03 11:00withdrawal50
4A2025-01-05 12:00refund20
5A2025-01-06 13:00withdrawal10
6B2025-01-02 09:00deposit500
7B2025-01-04 10:00refund25
8B2025-01-04 14:00withdrawal100

Two details matter. refund is a positive credit, easy to mis-sign as negative because it sounds like money leaving. And account B has two transactions on 2025-01-04 — the window must order them deterministically or the running balance is unstable.

2. The SQL — signed delta + partitioned running sum

SELECT id, account, ts, kind,
       CASE WHEN kind = 'withdrawal' THEN -amount ELSE amount END AS signed_amount,
       SUM(CASE WHEN kind = 'withdrawal' THEN -amount ELSE amount END)
         OVER (PARTITION BY account ORDER BY ts, id
               ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_balance
FROM txn
ORDER BY account, ts, id;

The CASE turns each transaction into a signed delta: withdrawals negative, deposits and refunds positive. The window SUM of that same expression, PARTITION BY account, accumulates per account; ORDER BY ts, id with an explicit ROWS frame produces the cumulative balance after each row.

The refund at id 4 raises A's balance from 20 to 40 — it is a credit, not a debit. Account B's two Jan-4 rows are sequenced refund-then-withdrawal by the ts, id tiebreak, giving a stable 525 then 425.

4. Strategic explanations

Why the CASE normalises before summing

The three transaction kinds have different effects on the balance, but a running SUM only knows how to add. The CASE converts every kind into a single signed delta — the universal currency the window function can accumulate. Note the polarity carefully: only withdrawal is negative; deposit and refund are both positive credits. Mis-signing refund is the most common error here, because the word sounds like an outflow.

Why PARTITION BY account is essential

The running sum must restart for each account — account A's balance must never bleed into account B's. PARTITION BY account gives each account an independent window: the UNBOUNDED PRECEDING boundary resets at every partition. Drop the partition and you would get one global running total across all accounts — meaningless as a per-account balance.

Why ORDER BY ts, id — determinism

Account B has two transactions stamped 2025-01-04 (one 10:00, one 14:00 — here the time component already differs, but if both were the same instant the order would be ambiguous). Ordering by ts alone risks a non-deterministic sequence within a tie group, so the running balance could flip between query runs. Adding id — a unique, monotonic key — gives a total order and a reproducible balance. A ledger that is not deterministic is not a ledger.

Why the explicit ROWS frame

ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW states the frame literally: every row from the partition start through this one. Without an explicit frame, an ordered window defaults to RANGE, and any tied ORDER BY values would lump tied rows into one shared total. With id in the sort the keys are unique so RANGE and ROWS would coincide here — but writing ROWS makes the intent explicit and bug-proof against future edits.

Repeating the CASE — readability

The signed-delta CASE appears twice: once for the signed_amount column, once inside SUM. You cannot reference the output alias signed_amount inside the window in the same SELECT level. To avoid the duplication, compute signed_amount in a CTE or subquery, then SUM(signed_amount) OVER (...) over it — cleaner and a single source of truth for the sign logic.

Senior signal: build a running balance by first normalising each transaction to a signed delta with CASE (watch the refund polarity), then SUM it over a window PARTITION BY account ORDER BY ts, id with an explicit ROWS frame — the unique-key tiebreak makes the ledger deterministic and the partition keeps accounts isolated.
