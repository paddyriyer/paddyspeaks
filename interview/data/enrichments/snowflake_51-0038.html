Intel’s core banking team needs a simple enriched transaction ledger: join each transaction to its account holder’s name and sort the result by account then newest-first within that account. This is the foundational one-to-many join pattern — every transaction belongs to exactly one account, and the join inflates the account row into one row per transaction. The question tests correct join key selection, column projection, and meaningful result ordering.

Carol Kim (A3) has no transactions. An INNER JOIN will exclude her from the result.

2. The SQL — INNER JOIN with account-then-time sort

SELECT a.account_id, a.holder_name, t.txn_id, t.txn_type, t.amount, t.txn_ts
FROM accounts     a
JOIN transactions t ON a.account_id = t.account_id
ORDER BY a.account_id, t.txn_ts DESC;

4. Strategic explanations

INNER JOIN semantics — accounts without transactions are excluded

The JOIN keyword (equivalent to INNER JOIN) only returns rows where a match exists in both tables. Carol Kim has no matching row in transactions, so she is silently dropped. This is the correct behavior for a "show me transactions" report, but wrong for a "show me all accounts" report. If Carol should appear with a NULL or empty transaction section, use LEFT JOIN accounts a … (keeping all accounts) or LEFT JOIN transactions t … (keeping all accounts, NULLing the transaction columns where absent).

One account row fans into multiple transaction rows

This is the canonical one-to-many join result: for each account row, the join produces one output row per matching transaction. Alice’s single account row becomes three rows. The account columns (account_id, holder_name) are repeated verbatim on each row — this is not an error, it is the expected structure of a denormalized join output. If the output were to be aggregated (total deposits per account), you would add GROUP BY a.account_id, a.holder_name.

The two-level ORDER BY

ORDER BY a.account_id, t.txn_ts DESC produces a bank-statement view: all transactions for each account grouped together, newest first within each account. The first key (account_id ASC) clusters by account; the second key (txn_ts DESC) inverts time within each cluster. Mixed sort directions are common and underappreciated in interviews — ASC and DESC are per-column in ORDER BY.

Column projection — what to select and why

The query selects exactly the columns needed for a transaction ledger view: account identity (account_id, holder_name), transaction identity (txn_id), and transaction detail (txn_type, amount, txn_ts). Avoid SELECT * in production: both tables likely contain columns irrelevant to the report (plan, signup_date, status, etc.), and joining on overlapping column names (both tables have account_id) can cause ambiguity or doubled columns in SELECT *.

Performance and indexing

An index on transactions(account_id, txn_ts DESC) serves both the join condition and the ORDER BY in one index scan. Without it, the engine performs a hash join on account_id followed by a full sort on the result. On large transaction tables this index is standard for any account-based ledger query.

Dialect notes

JOIN ... ON is universal ANSI SQL. The ORDER BY col1 ASC, col2 DESC mixed-direction syntax is supported identically in Snowflake, Postgres, BigQuery, MySQL, and SQL Server. In Snowflake you can also use QUALIFY with a window function to limit to the most recent transaction per account without a subquery, e.g. QUALIFY ROW_NUMBER() OVER (PARTITION BY a.account_id ORDER BY t.txn_ts DESC) = 1.
