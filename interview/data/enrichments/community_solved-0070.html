An online retailer logs every purchase as mobile or desktop, and product wants a daily breakdown: how many users — and how much spend — came from mobile-only, desktop-only, and users who shopped on both platforms that day. The interviewer is probing whether you can classify each user's daily behaviour before aggregating, and whether you can synthesise the "both" bucket, which is not a value in the data but a derived category.

On 2024-01-01, user 1 appears twice (both platforms), user 2 once (mobile), user 3 once (desktop). User 2 also reappears on 2024-01-02 — a single-member day for that date. This mix of a "both" user, two single-platform users, and a lone-day user covers every classification branch.

2. The SQL — classify by daily row count, then aggregate per bucket

WITH combination_cte AS (
  SELECT *, COUNT(1) OVER (PARTITION BY user_id, spend_date) AS test_count
  FROM spending
),
both_cte AS (
  SELECT spend_date, 'both' AS platform,
         SUM(amount) AS total_amount,
         COUNT(DISTINCT user_id) AS total_users
  FROM combination_cte
  WHERE test_count = 2
  GROUP BY spend_date
),
mobile_desktop_users AS (
  SELECT spend_date, platform,
         SUM(amount) AS total_amount,
         COUNT(DISTINCT user_id) AS total_users
  FROM combination_cte
  WHERE test_count = 1
  GROUP BY spend_date, platform
)
SELECT * FROM both_cte
UNION ALL
SELECT * FROM mobile_desktop_users
ORDER BY spend_date ASC;

User 1's two rows collapse into one "both" user spending 200. Users 2 and 3 land in mobile-only / desktop-only. On 2024-01-02 only mobile appears — the query emits no zero rows for the absent buckets.

4. Strategic explanations

The classification trick — COUNT(*) OVER (PARTITION BY user_id, spend_date)

The clever step is the windowed count: it tags every row with how many rows that user has on that day, without collapsing the rows. A count of 2 means the user transacted on two platforms (one mobile + one desktop) — the "both" group. A count of 1 means a single platform. This lets one CTE feed two differently-grouped downstream queries.

The hidden assumption — one row per user / platform / day

This is the most important thing to say out loud. test_count = 2 only means "mobile and desktop" if a user can have at most one row per platform per day. If user 1 made two separate mobile purchases on the same day, test_count would also be 2 — and they would be misclassified as a "both" user when they only used mobile. The robust version does not count rows; it counts distinct platforms: COUNT(DISTINCT platform) OVER (PARTITION BY user_id, spend_date). Then 2 genuinely means two different platforms regardless of how many transactions each had. Naming this fragility — and the DISTINCT fix — is the senior signal.

Why UNION ALL and the matching column shape

The "both" rows come from one grouping (GROUP BY spend_date) and the single-platform rows from another (GROUP BY spend_date, platform); they cannot be produced by one query. UNION ALL stitches them — ALL, not UNION, because the two sets are already disjoint (test_count = 2 vs = 1), so deduplication is wasted work. Both arms must project the same four columns in the same order; the literal 'both' AS platform manufactures the synthetic category so the shapes line up.

The missing-bucket trap

On 2024-01-02 the result shows only a mobile row — no desktop = 0, no both = 0. If the dashboard needs an explicit zero for every (date, bucket) pair, the query must CROSS JOIN a list of dates with the three bucket labels and LEFT JOIN the aggregates on. The current query reports only buckets that actually had activity — fine for some consumers, wrong for a fixed-shape report. Clarify which the stakeholder wants.

The modern alternative — conditional aggregation in one pass

Instead of two CTEs plus a union, classify each user once and pivot with FILTER:

WITH per_user AS (
  SELECT user_id, spend_date,
         CASE WHEN COUNT(DISTINCT platform) = 2 THEN 'both'
              ELSE MIN(platform) END AS bucket,
         SUM(amount) AS amt
  FROM spending GROUP BY user_id, spend_date
)
SELECT spend_date, bucket,
       COUNT(*) AS total_users, SUM(amt) AS total_amount
FROM per_user GROUP BY spend_date, bucket;

One scan, no union, and the "both" logic is explicit. MIN(platform) safely yields the sole platform when the count is 1.

Senior signal: recognise the windowed COUNT() OVER (PARTITION BY user_id, spend_date) as a classify-before-aggregate trick — then immediately flag that counting rows assumes one row per platform per day, and switch to COUNT(DISTINCT platform) so two mobile purchases are not mistaken for "both". Folding the two CTEs into a single conditional-aggregation pass is the over-the-bar finish.
