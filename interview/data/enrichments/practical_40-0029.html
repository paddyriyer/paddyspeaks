A pricing system stores product_dim as a slowly-changing dimension: every price change appends a new row stamped with an effective_date, and old rows are kept for history. A downstream report needs only the current state — one row per product, the one with the latest effective_date. The interviewer is testing the latest-per-entity pattern over an SCD, and whether you treat the effective-dated history correctly.

1. The source data — product_dim

product_idnamepriceeffective_date(narrative)
1Widget9.992024-01-01original price
1Widget11.992024-06-01first hike
1Widget Pro14.992025-01-01current — rename + hike
2Gadget19.992024-03-01original price
2Gadget17.992024-09-01current — price cut
3Gizmo4.992024-02-15single row — never changed

Product 1 has three versions — the latest is also a rename (Widget → Widget Pro), so picking the wrong row corrupts the name, not just the price. Product 3 is the edge case: a product whose price never changed has a single row, and it must still appear in the current snapshot.

2. The SQL — ROW_NUMBER() per product, keep the newest

SELECT product_id, name, price, effective_date
FROM (
    SELECT *,
           ROW_NUMBER() OVER (
               PARTITION BY product_id
               ORDER BY effective_date DESC
           ) AS rn
    FROM product_dim
)
WHERE rn = 1
ORDER BY product_id;

PARTITION BY product_id restarts numbering for each product; ORDER BY effective_date DESC puts the most recent version first, giving it rn = 1. The outer query keeps only those current rows.

3. The resulting analysis

product_idnamepriceeffective_datesignal
1Widget Pro14.992025-01-01latest of 3 — name and price updated
2Gadget17.992024-09-01latest of 2 — the price cut
3Gizmo4.992024-02-15single row passes straight through

One row per product. Product 1 correctly carries Widget Pro / 14.99 — the whole row from the latest version, so the rename rides along with the price. Product 3's lone row survives because a one-member partition still has an rn = 1. History rows are discarded but the table itself is untouched.

4. Strategic explanations

Picking the whole row, not just the max date

The naive instinct is SELECT product_id, MAX(effective_date) GROUP BY product_id — but that gives you only the id and date; you cannot also select name and price, because they are not in the GROUP BY and not aggregated. ROW_NUMBER ranks whole rows and lets you keep every column of the winning row intact. That is why the rename travels with the price: you select the row, not a recomputed scalar.

The self-join alternative and its multi-row trap

The commented form joins back to a MAX(effective_date) subquery — valid on legacy MySQL without window functions. But it has a defect: if a product had two rows sharing the same effective_date (a same-day double-write), the join matches both and the product appears twice. ROW_NUMBER with a tie-breaking secondary sort returns exactly one. For an SCD that should have unique effective dates, either is fine — but name the assumption.

The tie-break gap on duplicate effective dates

ORDER BY effective_date DESC alone leaves two rows with an identical latest effective_date in undefined order — the engine may pick either, and the result is non-deterministic. A well-formed SCD enforces one row per (entity, effective_date), so this should not arise; if it can, add a tie-breaker that varies within the partition, such as a surrogate id or a load timestamp: ORDER BY effective_date DESC, id DESC.

"Current" vs "current as of a date"

This query returns the absolute latest version. Real SCD queries often need "the version effective on a given as-of date" — the row where effective_date <= :as_of with the maximum such date. The senior extension adds WHERE effective_date <= :as_of inside the subquery before ranking. And if the table also has an end_date (a true SCD Type 2), "current" is simply WHERE end_date IS NULL — no window needed. Knowing which SCD shape you are on is the differentiator.

Dialect notes

Window functions are standard in Postgres, MySQL 8+, SQLite 3.25+, Snowflake, and SQL Server. Postgres requires the derived table to have an alias. The Postgres-idiomatic shortcut is SELECT DISTINCT ON (product_id) * FROM product_dim ORDER BY product_id, effective_date DESC, which picks the first row per product_id directly — terser, but Postgres-only.

Senior signal: get the current SCD snapshot with ROW_NUMBER() OVER (PARTITION BY entity ORDER BY effective_date DESC) + rn = 1 so the whole winning row (rename and all) is kept, prefer it over a MAX self-join which can return duplicates on tied dates, add a within-partition tie-breaker for determinism, and recognise the "as-of date" and SCD-Type-2 end_date IS NULL variants of "current".
