Okta's people-analytics team tracks a productivity score per employee per period. A single period is noisy — one bad sprint, one heroic week — so managers want a 3-period rolling average to see the trend, not the jitter. The interviewer is checking whether you know the difference between ROWS and RANGE framing, and what the average does at the start of an employee's history.

1. The source data — productivity

emp_idperiodscore

12024-0290
12024-0370
12024-04100

22024-0260

Employee 1 has four periods; employee 2 has only two. The interesting parts: the first one or two periods of each employee (where a 3-period window cannot be full) and the fact that the window must not bleed across employees.

2. The SQL — AVG OVER (… ROWS 2 PRECEDING)

SELECT emp_id, period, score,
  AVG(score) OVER (
    PARTITION BY emp_id
    ORDER BY period
    ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
  ) AS rolling3
FROM productivity;

PARTITION BY emp_id walls off each employee; ORDER BY period sequences their history; the ROWS frame averages the current row and the two before it.

Employee 1's first row averages just itself (80). The second averages two rows ((80+90)/2 = 85). Only from the third row on is the window genuinely 3-wide ((80+90+70)/3 = 80, then (90+70+100)/3 = 86.67). Employee 2's window resets — it never sees employee 1's numbers.

4. Strategic explanations

The ramp-up: a "3-period average" that averages fewer than 3

The first two rows of each employee do not have two predecessors, so the frame is partial — the average is over 1 then 2 rows. That is usually fine, but it means early values are computed on a different basis than later ones, and a dashboard plotting them side by side can look misleadingly smooth at the start. If the business wants a rolling average only once a full window exists, gate it with a count:

CASE WHEN COUNT(*) OVER w = 3
     THEN AVG(score) OVER w END AS rolling3
-- w = (PARTITION BY emp_id ORDER BY period ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)
-- first two periods now return NULL instead of a partial average
The ROWS vs RANGE trap — and why it bites here

If you omit the frame and write only ORDER BY period, the default frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — a running average from the start, not a 3-period window. And ROWS counts physical rows, while RANGE groups by value: if an employee had two rows with the same period, a RANGE frame would lump those duplicates together and a ROWS frame would not. For "last 3 records", you want ROWS — explicitly. The reference solution does this correctly.

The gap problem — "3 periods" vs "3 months"

ROWS 2 PRECEDING means the three most recent rows, not the three most recent months. If an employee skips 2024-02 entirely, the window silently spans Jan, Mar, Apr and still calls itself a 3-period average. If the metric must be calendar-accurate, you either pre-fill missing periods with a calendar spine, or switch to a RANGE frame over a real date with an interval. Naming this distinction is the senior detail.

NULL scores inside the window

AVG skips NULLs — a NULL score in the frame is dropped from both numerator and denominator, so a 3-row window with one NULL averages 2 values. If a missing score should count as a zero, wrap it: AVG(COALESCE(score, 0)) OVER …. Decide on purpose.

ORDER BY period only works because the format sorts

'2024-01' < '2024-02' as text because zero-padded YYYY-MM strings sort chronologically. If period were 'Jan-2024' or an unpadded '2024-1', the text sort would be wrong and the window would slide over the wrong rows. Confirm the period column's type and format before trusting the order.

Senior signal: use an explicit ROWS BETWEEN 2 PRECEDING AND CURRENT ROW frame (not the default RANGE, which gives a running total), call out that the first two periods average a partial window, distinguish "3 rows" from "3 calendar months" when gaps exist, and verify the period column actually sorts chronologically.
