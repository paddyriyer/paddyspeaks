Trapping Rain Water is the L5 interview's favourite "you-cannot-brute-force-this" geometry problem because it has three valid solutions that compose a tidy interview script: O(n²) brute force (rejected), O(n) space-O(n) precomputed-max arrays (passable), and O(n) space-O(1) two-pointer sweep (the senior answer). The underlying primitive — "capacity bounded by min(left_max, right_max)" — is the abstraction at the heart of every grid-storage capacity problem at scale: GPU memory packing on heterogeneous-shaped kernels, container loading in shipping logistics, ML reservoir capacity in feature stores, and terrain-flood simulation in disaster modelling. Visa lists this question in their senior-engineer loop because their fraud-detection pipeline literally computes a "trapped-anomaly-score" over an elevation map of per-merchant baseline rates.

1. The problem in a sentence

Given an array of non-negative integers representing the heights of unit-width bars in an elevation map, compute how many units of water are trapped after rain. The naive O(n²) approach — for each cell, scan left for max and right for max, then add min(L, R) - h[i] — is what a junior writes. The constraint that rules it out: arrays of length 10⁵. The trick is the two-pointer sweep: maintain running lmax and rmax, and advance the pointer on the shorter side because that side's running max is the binding constraint for the next cell (the other side can only be taller).

The key intuition: when h[l] < h[r], we KNOW rmax ≥ h[r] > h[l], so for position l the binding constraint is lmax (the right side will always have a wall of at least h[r]). We don't need to know what rmax actually is — just that it's at least h[r], which is enough.

3. The algorithm — naming the data structure

def trap(h):
    """Two-pointer sweep, O(n) time / O(1) space.
       The pointer on the SHORTER side is the binding constraint
       for the next cell -- we don't need both maxes precomputed."""
    if not h: return 0
    l, r = 0, len(h) - 1
    lmax = rmax = 0
    total = 0
    while l < r:
        if h[l] < h[r]:                       # left wall is the shorter wall
            lmax = max(lmax, h[l])             # update running max from the left
            total += lmax - h[l]               # water above position l is lmax - h[l]
            l += 1
        else:                                  # right wall is the shorter (or equal)
            rmax = max(rmax, h[r])             # update running max from the right
            total += rmax - h[r]               # water above position r is rmax - h[r]
            r -= 1
    return total

Three variants you should be ready to name out loud, in order of increasing sophistication: (1) brute force — O(n²), for-each-cell-scan-both-sides; (2) precomputed max arrays — O(n) time, O(n) space, build left_max[] and right_max[] in two sweeps, sum min(L[i], R[i]) - h[i]; (3) two-pointer sweep — O(n) time, O(1) space, this implementation. There's also (4) monotonic stack — O(n) time, O(n) space, fills water layer-by-layer; same complexity as (2) but slicker code.

4. Complexity

Time: O(n) — each index is visited exactly once across the two pointers. Space: O(1) — five integers (l, r, lmax, rmax, total). The trick that gets you there: instead of computing both running maxes up-front (which forces O(n) space), we exploit the asymmetry that the shorter side's max is the binding constraint and only update the side we know about.

5. The interviewer's edge-case probes

  • Empty or single-element input — return 0. No walls means no water.
  • All-monotonic input (strictly increasing or decreasing) — return 0. No right wall (or left wall) to hold water.
  • Flat profile (all equal) — return 0. No depressions to fill.
  • One peak ([1,2,3,2,1]) — return 0. Water needs a depression bounded on both sides.
  • Big bowl ([5,0,0,0,0,5]) — return 20. lmax=5 dominates; 4 cells × 5 units each.
  • 2D variant (Trapping Rain Water II, LeetCode 407) — the 1D two-pointer doesn't generalise. The 2D solution is a min-heap-based BFS from the boundary inward, processing the lowest border cell first (because that's the binding constraint for the cells inside). This is the killer follow-up at L6.
  • Negative heights / non-integer heights — algorithm works unchanged; the formula is just max(0, min(L, R) - h[i]). Most interviewers stipulate non-negative integers to avoid this discussion.

6. Real-world appearance — what runs this in production

GPU memory packing — NVIDIA CUDA, AMD ROCm

When a CUDA kernel launches with heterogeneous-shaped allocations (workgroup shared memory, register spills, scratchpad), the runtime packs them into a fixed-size SRAM region. Internal padding ("water") is wasted bandwidth; CUDA's __shared__ allocator uses a variant of trapping-rain-water to compute the minimum padding needed. NVIDIA's PTX compiler has a documented optimisation pass that minimises this trapped capacity.

Container loading — Maersk, FedEx, Amazon FBA inbound

Loading goods of varying heights into a fixed-width container is exactly the trapping-rain-water problem in 2D (height profile of items vs. the container's roof). FedEx's "cube optimisation" engine and Maersk's stowage planner both run a 2D variant on shipment manifests. Amazon FBA's inbound-shipment placement uses the 1D form per shelf to decide which SKUs get fast-pick vs. reserve.

Terrain-flood simulation — USGS, FEMA, climate-risk models

"How much water is trapped in this watershed after X inches of rain?" is the FEMA flood-zone classification primitive. USGS's NHDPlus river-network model runs a 2D priority-flood algorithm (LeetCode 407 variant) on digital elevation models at 10m resolution to compute trapped-water capacity per watershed. Munich Re's catastrophe-modelling team uses the same primitive for residential flood insurance pricing.

Feature store reservoir sizing — Tecton, Feast

When a feature store buffers events before flushing to long-term storage, the "trapped capacity" of the buffer is the integral of incoming-rate minus drain-rate. Tecton's auto-tuning sizer uses a 1D trapping-rain-water computation on historical traffic profiles to compute the minimum reservoir size that survives a 99th-percentile burst without dropping events.

Fraud detection — Visa, Mastercard, Stripe Radar

"Anomalous-transaction-volume over baseline" can be modelled as the area between an observed-rate curve and a baseline-rate curve. The "trapped" anomaly area uses the same min-of-bounding-walls trick to set the alert threshold. Visa's VAA (Visa Advanced Authorization) ML pipeline uses this as a feature engineering step.
