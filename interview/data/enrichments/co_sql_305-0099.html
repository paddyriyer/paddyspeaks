Click-through rate is the single most-stared-at metric in product analytics. Every PM dashboard has it; every growth experiment moves on it. The SQL is a one-liner — but the gotchas (zero-impression denominators, mis-counted clicks, multi-impression sessions) are where every shipping CTR query goes wrong.

8 events: 5 impressions, 3 clicks. Notice U_45's click without an impression — possibly a logging order issue, possibly a pre-loaded card, possibly a producer bug.

2. The SQL — conditional aggregation in a single pass

SELECT
  100.0 * SUM(CASE WHEN event_type = 'click'      THEN 1 ELSE 0 END) /
  NULLIF(  SUM(CASE WHEN event_type = 'impression' THEN 1 ELSE 0 END), 0) AS ctr_pct
FROM app_events;

3. The resulting analysis

ctr_pct
60.0%

3 clicks ÷ 5 impressions × 100 = 60% — extraordinarily high (real CTR is usually <5%). The orphan click from U_45 inflated it.

4. Strategic explanations

Why NULLIF(... , 0) matters more than it looks

If a particular surface had zero impressions on a given day (rare but real — a feature flag killed it, an outage, a brand-new feature with no traffic yet), the denominator is 0 and the query crashes with a division-by-zero error. NULLIF turns the 0 into NULL; dividing by NULL gives NULL; the dashboard shows "—" instead of erroring. Always wrap denominators of rate metrics with NULLIF.

The "orphan click" problem — should U_45 count?

Click-without-impression usually means one of:

  • Logging race condition — the click event landed before the impression event due to network reordering. Production systems usually accept these as legit.
  • Pre-loaded card — the impression was on a previous session that wasn't logged.
  • Producer bug — the impression beacon failed but the click beacon succeeded.

The query above counts orphan clicks. To exclude them, switch to a join-based version that requires a matching impression first:

WITH paired AS (
  SELECT user_id,
    MAX(CASE WHEN event_type = 'impression' THEN 1 END) AS had_impression,
    MAX(CASE WHEN event_type = 'click'      THEN 1 END) AS had_click
  FROM app_events
  GROUP BY user_id
)
SELECT 100.0 * COUNT(*) FILTER (WHERE had_click = 1)
       / NULLIF(COUNT(*) FILTER (WHERE had_impression = 1), 0) AS ctr_pct
FROM paired
WHERE had_impression = 1;
-- Now CTR is per-user (not per-event) and orphan clicks are excluded.
Per-event vs per-user CTR — different metrics

The naïve query is per-event: 3 clicks / 5 impressions = 60%. The senior version is per-user: of the 3 users who saw an impression, how many clicked? 2 out of 3 = 66.7%. They tell different stories — make sure you know which one the PM asked for.

Multi-impression sessions and click attribution

U_42 had 2 impressions before clicking. Should that count as 1 click / 2 impressions (50%) or 1 click / 1 impression (100%)? Production systems define impression-to-click attribution windows — usually first-impression credit (CTR = clicks / first-impressions). Under that model, U_42 = 1 click / 1 first-impression = 100%.

Daily / weekly / per-surface breakdowns

The single-number CTR is dashboard wallpaper. The useful version slices by time and surface:

SELECT
  DATE(event_ts)                                                          AS day,
  surface,                                                                  -- e.g. feed, story, reel
  100.0 * SUM(CASE WHEN event_type='click'      THEN 1 ELSE 0 END) /
  NULLIF( SUM(CASE WHEN event_type='impression' THEN 1 ELSE 0 END), 0)    AS ctr_pct
FROM app_events
GROUP BY DATE(event_ts), surface
ORDER BY day, surface;
Senior signal: name the per-event vs per-user distinction unprompted, mention NULLIF as the division-safety contract, and pre-empt the breakdown follow-up. The orphan-click handling is the over-the-bar move — it's the kind of edge case that comes up in real production CTR audits.
