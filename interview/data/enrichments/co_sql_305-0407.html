Salesforce's enterprise customers organize into hierarchies (parent company → subsidiary → division). Revenue rollups must aggregate up the tree. The query is a recursive CTE.

2. The SQL — recursive CTE + rollup

WITH RECURSIVE chain AS (
  SELECT account_id, name, parent_account_id, arr, account_id AS root_id, 1 AS depth
  FROM accounts WHERE parent_account_id IS NULL
  UNION ALL
  SELECT a.account_id, a.name, a.parent_account_id, a.arr, c.root_id, c.depth + 1
  FROM accounts a JOIN chain c ON a.parent_account_id = c.account_id
)
SELECT root_id, MAX(depth) AS tree_depth, COUNT(*) AS total_accounts,
  ROUND(SUM(arr), 2) AS rolled_up_arr
FROM chain GROUP BY root_id ORDER BY rolled_up_arr DESC;

3. Strategic explanations

The root_id trick

The anchor sets root_id = account_id (self). The recursive part propagates: each descendant inherits its grandparent's root. After traversal, every account knows its top-level ancestor.

Cycle-protection — depth cap

If the data has a cycle (A → B → A), recursive CTE loops infinitely. Add WHERE c.depth < 100 as a circuit breaker.

Materialized path alternative

Instead of recursing per query, store the path: '/JPM/Chase/Investment/'. Subtree queries become WHERE path LIKE '/JPM/%' — O(1) with a prefix index. Re-orgs require rewriting paths nightly.

The double-count avoidance

If arr is stored at every level (parent + child both have $$), summing gives 2x. Production stores arr only at leaf accounts; rollup sums only leaves. Or: store arr only at parent + use a flag.

The "report at any level" requirement

The CFO might want JPM total OR JPM-Chase total OR JPM-Chase-Investment total. The query above only rolls to root; production parameterizes the rollup level.

Senior signal: propagate root_id through the recursion for clean rollup, depth-cap as cycle protection, materialized path as a perf alternative, and pin where arr lives (leaf-only) to avoid double-count.
