Mining and natural resources companies need to aggregate extraction volumes by site and mineral: "For each mine location and mineral type, how much total tonnage was extracted and how many extraction events occurred?" This is a clean JOIN + GROUP BY pattern, but it includes a defensive NULLIF(regexp_replace(...)) cast guard on the tonnage column, signalling awareness that production schemas often have dirty numeric fields stored as text.

2. The SQL — join + aggregate with defensive numeric cast

SELECT m.location, e.mineral,
       SUM(NULLIF(regexp_replace(e.tonnage::text, '[^-.0-9]', '', 'g'), '')::numeric) AS total_tonnage,
       COUNT(*) AS extraction_events
FROM extractions e
JOIN mines m ON e.mine_id = m.mine_id
GROUP BY m.location, e.mineral
ORDER BY m.location, total_tonnage DESC;

4. Strategic explanations

The regexp_replace / NULLIF / ::numeric defensive cast pipeline

The tonnage column is cast through three steps: regexp_replace(e.tonnage::text, '[^-.0-9]', '', 'g') strips any non-numeric characters (e.g., " mt", "t", commas in European notation). NULLIF(..., '') converts empty strings (fully non-numeric values) to NULL rather than throwing a cast error. ::numeric completes the conversion. This pipeline is the production-grade pattern for any column that might contain units, commas, or stray characters. Without it, a single dirty value like "12.5 mt" would cause a runtime error on SUM.

INNER JOIN semantics — mines with no extractions are excluded

The INNER JOIN means any mine that has no extraction records will not appear in the output. In an audit context (e.g., "list all mines and their production") a LEFT JOIN from mines to extractions with COALESCE(total_tonnage, 0) would include idle mines. Always clarify with the interviewer whether "mines with no extractions" should appear in the report.

ORDER BY within groups — the two-column sort

ORDER BY m.location, total_tonnage DESC produces a within-group descending sort: all rows for a given location are sorted by total tonnage descending. This is the natural output for a "top minerals per mine" report. An alternative would be ORDER BY total_tonnage DESC globally, which would interleave locations. The choice depends on the reporting consumer (dashboards usually prefer the grouped form).

Extension: percentage of total tonnage per location

A window function over the grouped result computes each mineral’s share of its mine’s total production — a common analytical extension:

SUM(total_tonnage) OVER (PARTITION BY location) AS location_total,
total_tonnage * 100.0 / SUM(total_tonnage) OVER (PARTITION BY location) AS pct_of_location

This wraps the GROUP BY output in another SELECT with window functions — the double-aggregation pattern (GROUP BY first, then window on the aggregate).

Dialect portability of the regex cast
  • Postgres (solution dialect): regexp_replace(col::text, pattern, '', 'g') with the 'g' flag for global replace.
  • Snowflake: REGEXP_REPLACE(col::text, '[^-.0-9]', '') — no flag needed; Snowflake replaces all matches by default. Then NULLIF(..., '')::numeric.
  • BigQuery: REGEXP_REPLACE(CAST(col AS STRING), r'[^0-9.-]', '').
  • MySQL: REGEXP_REPLACE(col, '[^0-9.-]', '') (MySQL 8.0+). Note: no 'g' flag; it replaces all occurrences by default.
Senior signal: name the three-step cast pipeline (strip non-numeric → NULLIF empty → cast) as the standard guard for dirty numeric text columns, state that INNER JOIN hides idle mines and offer the LEFT JOIN alternative for complete coverage, and show the window-over-aggregate pattern for computing each mineral’s share of its location’s total tonnage.
