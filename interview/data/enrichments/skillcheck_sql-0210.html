Labeling every employee as CEO, Manager, or Individual Contributor looks like a simple CASE. The trap is identifying who manages nobody. The intuitive test — emp_id NOT IN (SELECT manager_id FROM employees) — silently returns nothing: the manager_id column contains a NULL (the CEO has no manager), and NOT IN against any list containing a NULL can never evaluate to TRUE.

Alice has no manager (the CEO). Bob manages Dan and Eve. Carol, Dan, and Eve manage no one.

2. The trap — NOT IN over a nullable column

-- intended: an Individual Contributor manages nobody
... WHERE emp_id NOT IN (SELECT manager_id FROM employees) ...

The subquery returns {NULL, 1, 1, 2, 2}. For Carol, 3 NOT IN (NULL, 1, 2) expands to 3 <> NULL AND 3 <> 1 AND 3 <> 2. The term 3 <> NULL is UNKNOWN, and UNKNOWN AND TRUE AND TRUE is UNKNOWN — not TRUE — so the row is rejected. This happens for every employee: with a NULL in the list, NOT IN yields either FALSE or UNKNOWN, never TRUE. The Individual Contributor branch matches zero rows.

3. The correct query

SELECT e.emp_id, e.emp_name,
       CASE
         WHEN e.manager_id IS NULL THEN 'CEO'
         WHEN EXISTS (SELECT 1 FROM employees s WHERE s.manager_id = e.emp_id) THEN 'Manager'
         ELSE 'Individual Contributor'
       END AS role
FROM employees e
ORDER BY e.emp_id;

4. Strategic explanations

Why NOT IN + NULL collapses to no rows

x NOT IN (list) is defined as x <> e1 AND x <> e2 AND ... for every element. If any element is NULL, that conjunct is UNKNOWN. A conjunction can only be TRUE if every term is TRUE, so the presence of a single NULL caps the whole expression at UNKNOWN (or FALSE). NOT IN with a nullable subquery is one of the most common silent-wrong-answer bugs in SQL.

NOT EXISTS is the null-safe choice

EXISTS / NOT EXISTS test row presence, not value equality, so NULLs never poison them. NOT EXISTS (SELECT 1 FROM employees s WHERE s.manager_id = e.emp_id) correctly identifies employees nobody reports to, regardless of NULLs in manager_id. It is also typically the fastest form, since the optimiser can stop at the first matching row.

If you must use NOT IN

Filter the NULLs out explicitly: emp_id NOT IN (SELECT manager_id FROM employees WHERE manager_id IS NOT NULL). This restores correct behaviour, but NOT EXISTS is preferred — it cannot be broken by forgetting the filter, and it states the intent (“no such subordinate”) directly. Note the asymmetry: plain IN tolerates NULLs in the list fine; only NOT IN is dangerous.

Dialect notes

This three-valued-logic behaviour is ANSI SQL and identical across PostgreSQL, MySQL, SQL Server, Oracle, Snowflake, and BigQuery — NOT IN with a NULL is universally hazardous. No engine warns about it; the query runs and quietly returns an empty or short result.

Senior signal: NOT IN against a subquery that can yield NULL never returns TRUE for any row — a single NULL turns the whole predicate UNKNOWN. Reach for NOT EXISTS (null-safe and usually faster), or, if keeping NOT IN, add an explicit WHERE col IS NOT NULL to the subquery.
