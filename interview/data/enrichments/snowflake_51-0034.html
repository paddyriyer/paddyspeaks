Meta’s engagement pipeline receives user interactions from three independent tables — likes, comments, and shares — each recording who interacted, with what target, and when. The analytics team needs a single chronological feed of all interaction events with an interaction-type label. This is the canonical UNION ALL fan-in pattern: bring heterogeneous event streams into one schema and sort the combined stream by time.

2. The SQL — UNION ALL fan-in with interaction-type label

SELECT user_id, target_id, ts, 'like'    AS interaction FROM likes
UNION ALL
SELECT user_id, target_id, ts, 'comment'         FROM comments
UNION ALL
SELECT user_id, target_id, ts, 'share'           FROM shares
ORDER BY ts;

Six rows in chronological order, drawn from all three source tables. U2 shared Post_A at 08:55 (earliest), U3 commented on Post_C at 09:10 (latest). The interaction column is a string literal added by each branch — not a column from any table.

4. Strategic explanations

UNION ALL vs UNION — the critical difference

UNION ALL concatenates all rows from every branch, including exact duplicates. UNION (without ALL) performs a deduplication step across the combined set — essentially a SELECT DISTINCT on the result — which requires a sort or hash and is significantly more expensive. For event-stream fan-in you almost always want UNION ALL: each event is a distinct occurrence and deduplication would silently drop legitimate simultaneous events. The only time UNION is appropriate is when the source tables genuinely overlap and you need exactly one copy of each unique row.

Column alignment: position, name, and type must match

In a multi-branch UNION ALL the engine uses the first branch’s column names and aligns subsequent branches by position, not name. All branches must have the same number of columns, and corresponding columns must have compatible types. Here the literal 'like', 'comment', and 'share' are all TEXT — compatible. If likes.ts were a DATE and comments.ts a TIMESTAMP, most engines would cast up to TIMESTAMP automatically; explicit casts are safer for portability.

The ORDER BY applies to the entire union, not per-branch

A single ORDER BY ts at the end sorts the combined 6-row result set. It is not legal to put ORDER BY inside individual branches (most engines reject this). If you need per-branch ordering before combining, wrap each branch in a subquery with its own order, but note that standard SQL does not guarantee the order of subquery rows in a UNION ALL anyway — the final ORDER BY is the only reliable sort.

Adding a secondary tiebreaker

If two events share the same ts, the row order within the tie is non-deterministic. Add a secondary sort to make it stable: ORDER BY ts, interaction or ORDER BY ts, user_id, target_id. In production pipelines that feed downstream aggregations, non-deterministic ordering causes flaky tests and inconsistent reports.

Scaling: partitioned UNION ALL

With billions of events across three tables, a full-union sort is expensive. Common patterns: (1) store all events in a single partitioned table with an interaction_type column from the start; (2) use incremental materialized views that fan in only the delta since last run. The UNION ALL query is the conceptual blueprint; production systems denormalize it away.

Dialect notes

UNION ALL is universally supported (ANSI SQL). String literals as column values are universal. Snowflake, Postgres, BigQuery, MySQL, and SQL Server all handle this identically. The only gotcha is MySQL’s historic 64KB limit on UNION result sets in older versions (pre-8.0) — irrelevant in modern deployments.

Senior signal: immediately distinguish UNION ALL (no dedup, O(n)) from UNION (dedup, O(n log n)), explain that column alignment is positional not by name, note that ORDER BY belongs on the outermost query, and advocate for a denormalized single-table design at scale. The tiebreaker discussion shows production awareness.
