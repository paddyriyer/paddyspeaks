AMD's order-management team wants a flat line-item report — one row per product on every order, with the customer's name spelled out. The data is normalised across four tables, so the report is a chain of joins. The interview question is not "can you write three joins" — it is "do you understand that joining through a bridge table multiplies rows, and is that what the report wants?"

1. The source data

orders (one row per order) and order_items (the bridge — one row per product on an order):

2. The SQL — chained INNER JOINs

SELECT o.order_id, c.name AS customer, p.name AS product
FROM orders o
JOIN customers c   ON o.customer_id = c.customer_id
JOIN order_items oi ON o.order_id    = oi.order_id
JOIN products p     ON oi.product_id = p.product_id;

Four tables, three join conditions. The chain goes orders → customers (to get the name), then orders → order_items (the bridge) → products (to get the product name).

Four rows from three orders. Order 5001 appears twice — once per line item. This is not a bug; it is the intended grain of a line-item report. The crucial thing is to know it happens: if you summed an order-level amount column across these rows, you would double-count order 5001's revenue.

4. Strategic explanations

The fan-out trap — the heart of this question

Joining orders through the order_items bridge fans out: a one-row order becomes one row per product. That is correct for a line-item report, but it makes order-level aggregates wrong. If the query also selected o.amount and you ran SUM(o.amount), order 5001's amount would be added twice. To total order revenue after a fan-out, deduplicate first:

-- WRONG: double-counts multi-item orders
SELECT SUM(o.amount) FROM orders o JOIN order_items oi ON o.order_id = oi.order_id;

-- RIGHT: aggregate at the order grain, then there is nothing to fan out
SELECT SUM(amount) FROM orders;

The rule: know the grain of every row before you aggregate. A line-item row is not an order.

Join order is cosmetic; join type is not

The optimiser is free to execute these joins in any order — writing customers before order_items does not change the plan or the result. What does change the result is the join type. Every join here is INNER, so the report silently drops:

  • an order with no rows in order_items (an empty cart) — gone;
  • a line item whose product_id points at a deleted product — gone;
  • an order whose customer_id is NULL or orphaned — gone.

If the report must show every order even when a join fails, switch the relevant links to LEFT JOIN and decide what the missing side should display.

Including order status — the filter the interviewer expects

Order 5003 is pending. A revenue or fulfilment report usually wants only realised orders. The filter belongs in WHERE (it filters a non-outer-joined table):

SELECT o.order_id, c.name AS customer, p.name AS product
FROM orders o
JOIN customers c    ON o.customer_id = c.customer_id
JOIN order_items oi ON o.order_id    = oi.order_id
JOIN products p     ON oi.product_id = p.product_id
WHERE o.status = 'completed';

A subtlety worth voicing: if any join were a LEFT JOIN, a status predicate in WHERE would quietly turn it back into an INNER JOIN — the condition on a join's nullable side belongs in ON, not WHERE.

The duplicate-key edge case

A line-item report is supposed to fan out on order_items. The unwanted fan-out is on the dimension tables: if customers or products had two rows with the same id — a dirty dimension — every row would duplicate again, and that fan-out is a genuine bug. Dimension keys must be unique; verify with a GROUP BY id HAVING COUNT(*) > 1 before trusting the row count.

Senior signal: recognise that joining through the order_items bridge fans an order into one row per line item — correct for this report but fatal to any order-level SUM — reason about grain before aggregating, know that INNER JOIN silently drops empty carts and orphaned keys, and place the status filter in WHERE only because no link here is an outer join.
