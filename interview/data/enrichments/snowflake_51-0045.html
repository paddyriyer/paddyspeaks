A lab information system needs a quick summary view: "For each thermodynamics experiment, what were the average temperature and pressure across all runs, and how many measurements were recorded?" The interviewer is probing a straightforward one-to-many JOIN followed by grouped aggregation, with the added nuance of the NULLIF(regexp_replace(...)::numeric) defensive cast for numeric columns stored as text — a realistic pattern when instrument firmware dumps mixed-format readings into a VARCHAR column.

2. The SQL — JOIN and grouped aggregation with defensive casts

SELECT e.exp_id, e.name,
       AVG(NULLIF(regexp_replace(m.temperature_k::text, '[^-.0-9]', '', 'g'), '')::numeric) AS avg_temp_k,
       AVG(NULLIF(regexp_replace(m.pressure_pa::text,   '[^-.0-9]', '', 'g'), '')::numeric) AS avg_pressure_pa,
       COUNT(m.meas_id) AS measurement_count
FROM experiments e
JOIN measurements m ON e.exp_id = m.exp_id
GROUP BY e.exp_id, e.name
ORDER BY e.exp_id;

4. Strategic explanations

INNER JOIN drops experiments with no measurements

The query uses an INNER JOIN, which means any experiment in the experiments table that has no rows in measurements will not appear in the results. If the business requirement is "list all experiments, even those not yet run," replace with a LEFT JOIN. With a LEFT JOIN, avg_temp_k and avg_pressure_pa would be NULL and COUNT(m.meas_id) would be 0 for experiments without measurements (because COUNT on a specific non-null column returns 0, not NULL, when all values are NULL due to the outer join). This COUNT(m.meas_id) vs COUNT(*) distinction is precisely why the query counts a specific column rather than *.

COUNT(m.meas_id) vs COUNT(*) — NULL sensitivity

COUNT(*) counts every row, including those where all columns are NULL (which happens with a LEFT JOIN when there is no matching measurement). COUNT(m.meas_id) counts only rows where meas_id IS NOT NULL. With a LEFT JOIN and no matching measurements, COUNT(m.meas_id) correctly returns 0, while COUNT(*) would incorrectly return 1 (counting the placeholder NULL row the LEFT JOIN introduces). Using a specific column in COUNT is the correct idiom for LEFT JOIN aggregation.

Defensive cast pattern — NULLIF(regexp_replace(...)::numeric)

The measurement columns may be stored as TEXT because instrument firmware can emit values like "300.0K", "N/A", or "". The pipeline: regexp_replace(val::text, '[^-.0-9]', '', 'g') strips non-numeric characters globally; NULLIF(..., '') converts the empty string (from a fully non-numeric value) to NULL; ::numeric parses the clean string. AVG ignores NULLs, so corrupt readings drop out of the average. In Snowflake, TRY_CAST(m.temperature_k AS FLOAT) achieves the same result with less syntax.

GROUP BY includes non-aggregated SELECT columns

The query groups by both e.exp_id and e.name. If exp_id is the primary key, e.name is functionally dependent on it and strictly speaking the GROUP BY could list only exp_id in engines that support functional-dependency detection (Postgres does, with a PK declaration; MySQL, Snowflake, and BigQuery generally require all non-aggregated SELECT expressions in GROUP BY). Including e.name explicitly is portable and self-documenting.

Dialect portability
  • Snowflake: TRY_CAST(temperature_k AS FLOAT) replaces the regex chain. Works for AVG identically.
  • BigQuery: SAFE_CAST(temperature_k AS FLOAT64).
  • MySQL: No regexp_replace with global flag natively (MySQL 8.0 has REGEXP_REPLACE but applies globally by default); use CAST(... AS DECIMAL) for clean numeric columns or a custom UDF for dirty ones.
Senior signal: distinguish INNER from LEFT JOIN and explain what happens to experiments without measurements in each case, use COUNT(m.meas_id) (not COUNT(*)) to correctly return 0 for outer-joined non-matching rows, and explain the NULLIF(regexp_replace(...)::numeric) sanitisation pattern alongside the Snowflake TRY_CAST equivalent.
