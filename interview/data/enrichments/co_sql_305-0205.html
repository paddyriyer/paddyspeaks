JPMorgan launches a new credit card product each month and tracks how many applications each launch pulls in. The analyst wants the month-over-month change in application volume. The twist: the applications column arrived as text, with thousands separators baked in ("1,850"), and some months have no figure at all. The reference cleans the text inline and uses LAG for the delta. The interviewer is watching how you handle the dirty type and the missing rows.

1. The source data — cards

52026-05-012,750

The applications values are inconsistent text: some clean ("1200"), some with a comma ("1,850"), and April's is NULL — the launch metrics were never filed. That NULL row in the middle of the series is the whole interview.

2. The SQL — regex-clean then LAG delta

SELECT launch_date, applications,
  NULLIF(regexp_replace(applications::text, '[^-.0-9]', '', 'g'), '')::numeric
  - LAG(NULLIF(regexp_replace(applications::text, '[^-.0-9]', '', 'g'), '')::numeric)
      OVER (ORDER BY launch_date) AS delta
FROM cards;

regexp_replace(..., '[^-.0-9]', '', 'g') strips every character that is not a digit, minus, or dot — so "1,850" becomes "1850". NULLIF(..., '') turns an empty result into NULL, then ::numeric casts. LAG over launch_date fetches the previous month's cleaned value, and the subtraction gives the change.

3. The resulting analysis

February's delta is 1850 − 1200 = 650; March's is 2100 − 1850 = 250 — both correct. January is NULL because LAG has no prior row. But look at the last two rows: April is NULL because its own value is missing — expected. May is also NULL, even though May (2750) is known — because LAG returns the immediately preceding row, which is April's NULL. The missing month poisons the month after it.

4. Strategic explanations

The NULL-propagation trap — LAG reaches back exactly one row

Plain LAG(x) takes the value from the physically previous row, NULL or not. So a single missing month creates two NULL deltas: April's (its own gap) and May's (April was its predecessor). For a month-over-month chart that is usually wrong — May genuinely grew over the last known reading. If the metric should compare against the last non-NULL month, that is a different query. Postgres 16+ supports the SQL-standard option directly:

-- Postgres 16+: skip NULL predecessors
... - LAG(apps_num) IGNORE NULLS OVER (ORDER BY launch_date) ...

On older Postgres, drop the missing rows before windowing so LAG only ever sees real values:

WITH parsed AS (
  SELECT launch_date, applications,
    NULLIF(regexp_replace(applications::text,'[^-.0-9]','','g'),'')::numeric AS apps_num
  FROM cards
)
SELECT launch_date, applications,
       apps_num - LAG(apps_num) OVER (ORDER BY launch_date) AS delta
FROM parsed WHERE apps_num IS NOT NULL;

With the missing month filtered out, May's delta becomes 2750 − 2100 = 650. Which behaviour is right is a product question — "should a gap break the series, or should we compare to the last figure we have?" — and naming it is the senior move. The reference's output (delta NULL on the post-gap month) is defensible only if a gap is meant to interrupt the trend.

Why the data is text — and why NULLIF guards the cast

A numeric column would never need regex scrubbing. applications being text with thousands separators screams a bad ETL load — the real fix is upstream: store it as integer. Inline, regexp_replace strips the commas; the NULLIF(..., '') is the safety catch. If a value were 'N/A' or '—', the regex strips it to the empty string, and ''::numeric would throw. NULLIF converts that empty string to NULL first, so the cast succeeds and the bad row simply has no number. Without it, one junk value aborts the whole query.

Ordering ties — two launches in the same month

LAG needs a total order. If two cards launched on the same launch_date, ORDER BY launch_date alone leaves their relative order to the planner, so the delta could pair them in either direction. Add a deterministic tiebreaker: ORDER BY launch_date, card_id. For a strict monthly series this is unlikely, but a real launch calendar can have two products on the same day.

LAG vs a self-join

The pre-window way to get "previous month" is a self-join on launch_date = prev.launch_date + interval '1 month'. That is brittle — it breaks the instant a month is skipped, and it assumes exact monthly spacing. LAG OVER (ORDER BY launch_date) simply takes the prior row regardless of spacing, which is more robust and the source of the NULL-propagation behaviour above. One tool, one trade-off to articulate.

Edge cases: first row, percentage change, negative deltas

The first row's delta is always NULL — no predecessor — which is correct, not a bug. If the analyst wants percent change rather than absolute, guard the division: (curr - prev) / NULLIF(prev, 0) so a zero-application month does not divide-by-zero. And deltas can legitimately be negative when a launch underperforms the prior one — do not ABS() them away; the sign is the signal.

Senior signal: recognise that plain LAG propagates a missing month into the next month's delta, and decide deliberately between letting a gap break the series or comparing to the last non-NULL value (IGNORE NULLS, or filter-then-window); explain that NULLIF before ::numeric is what stops a junk string from aborting the query; and flag that the text column is an upstream ETL defect, not something to scrub forever inline.
