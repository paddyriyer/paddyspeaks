Zoom's pricing team wants to know how much discounting actually costs per product category — original price, discounted price, and the savings, averaged within each category. The query is three AVGs over a GROUP BY, but the interviewer is probing whether you understand that AVG(a) − AVG(b) equals AVG(a − b) here, how NULLs can break that identity, and whether an average is even the right lens on discount impact.

1. The source data — orders

Three categories. The interesting row is order 2 — an Electronics order with no discount at all (original = discounted = 500). A zero-discount order inside a discounted category drags the category's average savings down, which is exactly the signal pricing wants to see.

2. The SQL — three AVGs per category

SELECT category,
  AVG(original_price) AS avg_orig,
  AVG(discounted_price) AS avg_disc,
  AVG(original_price - discounted_price) AS avg_savings
FROM orders GROUP BY category;

One group per category; within each, the average list price, the average paid price, and the average per-order savings.

4. Strategic explanations

Why AVG(a) − AVG(b) equals AVG(a − b) — and when it stops

The average is linear: AVG(a) − AVG(b) = (Σa − Σb)/n = Σ(a−b)/n = AVG(a−b) — provided both columns have the same set of non-NULL rows. The identity breaks if, say, discounted_price is NULL on some rows: AVG(discounted_price) divides by fewer rows, while AVG(original_price - discounted_price) drops the whole expression (any arithmetic with NULL is NULL) on a possibly different row set. With clean data they agree; with NULLs they diverge. AVG(original_price - discounted_price) is the more honest "average savings" because it only averages orders where both prices exist.

NULL discounted prices — the subtle bug

If a NULL discounted_price really means "no discount applied," then original - discounted yields NULL and that order is excluded from avg_savings entirely — understating how often the category sells at full price. Decide deliberately:

AVG(original_price - COALESCE(discounted_price, original_price)) AS avg_savings
-- treats a missing discount as zero savings, keeping the order in the average
Average savings vs discount rate — the better metric

Absolute savings is dominated by expensive categories — Electronics "saves 100," Books "saves 6," but that comparison is meaningless across price tiers. Pricing usually wants the rate:

SELECT category,
  SUM(original_price - discounted_price) / NULLIF(SUM(original_price), 0) AS discount_rate
FROM orders GROUP BY category;

Note this is a weighted rate (total savings ÷ total list), not the average of per-order rates — large orders count more, which is what margin impact actually means. Offering both numbers and naming the difference is the senior move.

Average of ratios vs ratio of sums

AVG(savings / original_price) weights a $10 order and a $10,000 order equally; SUM(savings) / SUM(original_price) weights by dollar value. They can differ a lot. Neither is "wrong" — but for revenue/margin impact the ratio-of-sums is right, and you should say which one you computed.

Negative savings and category hygiene

If discounted_price > original_price (a price correction, a surcharge), savings go negative and quietly pull the average down — worth a sanity filter or a CHECK constraint. And 'Books' vs 'books' vs NULL category each form separate groups; dirty category labels fragment the report.

Senior signal: explain that AVG(orig) − AVG(disc) equals AVG(orig − disc) only when both columns share the same non-NULL rows, handle a NULL discounted price on purpose with COALESCE, and recommend a weighted discount rate (SUM/SUM) over absolute savings for comparing impact across price tiers.
