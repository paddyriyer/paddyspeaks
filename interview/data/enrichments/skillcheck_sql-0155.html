A product analyst wants a per-user lifecycle summary: each user's first and last event on a single row — first timestamp, first action, last timestamp, last action. The interviewer is probing FIRST_VALUE / LAST_VALUE window functions, and specifically the notorious default-frame trap that makes LAST_VALUE return the wrong row unless you set the frame explicitly.

1. The source data — event

user_idtsaction(narrative)
12025-01-01 10:00signupuser 1 first event
12025-01-02 11:00viewmiddle event
12025-01-05 12:00purchaseuser 1 last event
22025-02-01 09:00signupuser 2 first event
22025-02-10 14:00churnuser 2 last event — only two rows

User 1 has a clear middle event (view) that must not leak into the first/last columns. User 2 has just two events — first and last are different rows, a useful boundary for the frame logic. Both users exercise whether LAST_VALUE sees the whole partition.

2. The SQL — FIRST_VALUE / LAST_VALUE with an explicit full-partition frame

SELECT DISTINCT
  user_id,
  FIRST_VALUE(ts)     OVER w AS first_ts,
  FIRST_VALUE(action) OVER w AS first_action,
  LAST_VALUE(ts)      OVER w AS last_ts,
  LAST_VALUE(action)  OVER w AS last_action
FROM event
WINDOW w AS (PARTITION BY user_id ORDER BY ts
             ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING)
ORDER BY user_id;

The named window w partitions by user, orders by ts, and — critically — sets the frame to the entire partition. FIRST_VALUE reads the earliest row, LAST_VALUE the latest. The window function repeats those values on every row of the user, so DISTINCT collapses each user to one summary row.

3. The resulting analysis

user_idfirst_tsfirst_actionlast_tslast_action
12025-01-01 10:00signup2025-01-05 12:00purchase
22025-02-01 09:00signup2025-02-10 14:00churn

User 1's last is purchase at Jan 5 — the middle view event is correctly ignored. User 2 spans signup to churn. Each user is one row. With the default frame, last_action would instead have echoed each row's own action — the bug this query is built to avoid.

4. Strategic explanations

The default-frame trap — why LAST_VALUE usually lies

When a window has an ORDER BY but no explicit frame, the default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. The frame therefore ends at the current row. FIRST_VALUE still works — the first row is always in that frame — but LAST_VALUE returns the last row of the frame, which is the current row itself. So LAST_VALUE(action) without an explicit frame just echoes each row's own action: user 1 would show view on the view row, purchase on the purchase row. Setting ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING extends the frame to the full partition so LAST_VALUE genuinely sees the final row.

Why FIRST_VALUE seems immune — and is not, conceptually

FIRST_VALUE appears to work even with the default frame, because the partition's first row is in every growing frame. That is luck, not correctness — it relies on the frame starting at UNBOUNDED PRECEDING. The disciplined habit is to always state the frame explicitly when using FIRST_VALUE/LAST_VALUE/NTH_VALUE, so the reader does not have to reason about defaults and a later edit cannot quietly break it.

SELECT DISTINCT — why it is needed and its alternatives

A window function does not collapse rows: it computes a value per row and leaves the row count unchanged. User 1 has three event rows, each now carrying the identical first/last quad, so without deduplication you would get three identical summary rows. DISTINCT folds them to one. Cleaner alternatives: filter to one row per user with ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY ts) = 1, or skip windows entirely and use conditional aggregation — MIN(ts), plus (array_agg(action ORDER BY ts))[1] for the first action.

Ties on ts — an unstated risk

If a user had two events at the exact same timestamp, "first" and "last" become ambiguous — FIRST_VALUE/LAST_VALUE would pick whichever the engine ordered first, non-deterministically. The fixture has distinct timestamps so it does not bite, but a senior answer adds a tie-break to the window ORDER BY — ORDER BY ts, event_id — to make first/last deterministic. Always ask whether the ordering key is unique.

Dialect — FIRST/LAST aggregates elsewhere

The WINDOW clause, FIRST_VALUE, LAST_VALUE, and explicit frames are standard in Postgres, SQL Server, Oracle, MySQL 8+, and SQLite 3.25+. Some engines offer terser aggregate forms — BigQuery has ANY_VALUE with HAVING MIN/MAX, Oracle has FIRST/LAST with KEEP (DENSE_RANK FIRST ORDER BY …), and Postgres can do (array_agg(x ORDER BY ts))[1]. The explicit-frame window form shown here is the most portable and the version interviewers want you to be able to defend.

Senior signal: when using LAST_VALUE, always set the frame explicitly to ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING — the default RANGE … CURRENT ROW frame makes LAST_VALUE echo the current row — remember windows do not collapse rows so you need DISTINCT or a row filter, and add a tie-break to the window ORDER BY when the timestamp is not unique.
