Apple's retail-ops team runs a "value pick" shelf near the till — the single cheapest item in the catalogue, refreshed nightly. The query behind it looks trivial: sort by price, take one. But "the cheapest product" hides a question the interviewer is really asking: what happens when two products tie at the lowest price? A one-row answer that silently picks one of them is a decision, and a senior candidate makes it on purpose.

1. The source data — products

One product — the Power Adapter at $8.99 — is cleanly the cheapest. But notice the tie at $12.99 between the USB-C Cable and the Screen Wipe. It is not the answer here, but it is exactly the case the interviewer will reach for next.

2. The SQL — ORDER BY price LIMIT 1

SELECT product_id, name, price
FROM products
ORDER BY price ASC
LIMIT 1;

Sort the whole table ascending by price, then keep only the first row. No aggregation, no grouping — just a sort and a cut.

3. The resulting analysis

product_idnameprice
5Power Adapter8.99

One row. Because $8.99 is strictly below every other price, the answer is unambiguous and deterministic here. The fragility only shows up when the minimum is shared — which is the whole point of the follow-ups below.

4. Strategic explanations

The tie: LIMIT 1 is non-deterministic

If the cheapest price were shared by two rows, ORDER BY price LIMIT 1 would return one of them — and which one is up to the planner. Re-run after an index change or a vacuum and the answer can flip. To make it stable, add a tiebreak column so the sort is total:

SELECT product_id, name, price
FROM products
ORDER BY price ASC, product_id ASC
LIMIT 1;

If the business actually wants every product at the minimum price, LIMIT 1 is the wrong tool entirely — use a subquery:

SELECT product_id, name, price
FROM products
WHERE price = (SELECT MIN(price) FROM products);
MIN(price) vs ORDER BY ... LIMIT 1

A frequent "equivalent" is SELECT MIN(price) FROM products. It is not equivalent: MIN returns the price only, not the product_id or name. The moment you need the identity of the product, you either sort-and-limit or filter against the MIN subquery. The subquery form also generalises cleanly to ties, which is why it is the safer default in a real codebase.

The window-function approach — and why it earns "top N cheapest"

RANK() answers the tie question and extends to "show me the five cheapest" with a one-character edit:

SELECT product_id, name, price
FROM (
  SELECT product_id, name, price,
         RANK() OVER (ORDER BY price ASC) AS rk
  FROM products
) t
WHERE rk = 1;

RANK assigns rank 1 to all rows at the minimum price, so ties are kept. Swap WHERE rk = 1 for WHERE rk <= 5 and you have the cheapest five — the natural next request from a merchandiser.

Edge cases: empty table, NULL prices

On an empty table, ORDER BY ... LIMIT 1 returns zero rows — not a NULL row, not an error. Code that expects exactly one row must handle the empty result. NULLs matter too: price should never be NULL for a sellable product, but if it is, ORDER BY price ASC in Postgres sorts NULLs last, so a NULL-priced row won't be mistaken for the cheapest. MIN(price) likewise skips NULLs. Worth a sentence to the interviewer: "I'm assuming price is non-NULL and non-negative; if returns or credits can make it negative, the 'cheapest' row may not be a real product."

Senior signal: recognise that ORDER BY price LIMIT 1 is non-deterministic under ties and add a tiebreak, know when to switch to a MIN subquery or RANK() to return all tied rows, and call out the empty-table zero-row result and the NULL-price ordering before the interviewer has to probe.
