A finance team wants a running total of revenue: "for each sale, show the cumulative dollars booked up to and including that day." The reference solution is a windowed SUM with ORDER BY — the canonical running-total pattern. The interviewer is probing whether you understand the default window frame and what it does when two sales share a date.

1. The source data — sales

sale_dateamount(narrative)

2026-04-01$200same-day second sale — a tie
2026-04-02-$50refund
2026-04-03$300sale
2026-04-04$80same-day pair
2026-04-04$450same-day pair

Two dates (Apr 1 and Apr 4) carry two rows each — ties in the ORDER BY key. That tie is the whole interview: it decides what the cumulative value is on a multi-sale day.

2. The SQL — windowed SUM with ORDER BY

SELECT sale_date, amount,
  SUM(amount) OVER (ORDER BY sale_date) AS cumulative
FROM sales;

3. The resulting analysis

The surprise: the first Apr-1 row shows $320, not $120. Both same-date rows get the cumulative through the end of that date. Same on Apr 4 — both rows read $1100. That is the default frame at work.

4. Strategic explanations

The default frame is RANGE, not ROWS — this is the key insight

When a window has ORDER BY but no explicit frame clause, the implicit frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. Under RANGE, "current row" means every peer that ties on the ORDER BY key. So all rows sharing a sale_date see the same cumulative — the sum through that entire date. If you intended a strict row-by-row running total, you must say so:

SUM(amount) OVER (ORDER BY sale_date
  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)

With ROWS, Apr 1 would read 120 then 320 — but the order between the two tied rows is then arbitrary unless you add a tiebreaker to ORDER BY.

RANGE is actually correct for a daily running total

If the deliverable is "end-of-day cumulative revenue," the default RANGE behaviour is exactly right — you genuinely want one cumulative number per day, and intra-day row order is meaningless. The senior move is to recognise that the "weird" 320 is a feature, not a bug, and to name it. Many candidates "fix" it with ROWS without realising they have just made the result order-dependent.

Make the order total when you need a row-level total

For a true per-transaction running balance, sort on something unique — a timestamp or an id — so no two rows tie: OVER (ORDER BY sale_date, sale_id ROWS UNBOUNDED PRECEDING). A non-deterministic ORDER BY plus ROWS gives results that can change between runs.

NULL amounts and output ordering

SUM skips NULL amounts, so a NULL amount leaves the cumulative flat for that row rather than blanking it. Separately, the query has no outer ORDER BY — the window orders the computation, but row display order is not guaranteed. Add a final ORDER BY sale_date if the consumer needs sorted output.

Senior signal: state that ORDER BY with no frame defaults to RANGE UNBOUNDED PRECEDING, so tied dates all show the end-of-day cumulative; explain that this is correct for a daily total but wrong for a per-row running balance, where you need ROWS plus a unique tiebreaker in ORDER BY.
