The monthly cohort retention triangle is a visual staple of growth dashboards: a matrix where rows are signup cohorts and columns are calendar months, with each cell showing what fraction of the original cohort was still active. This question tests whether you can (1) compute the correct month offset using year-month arithmetic rather than raw date subtraction, and (2) deduplicate daily activity rows into distinct monthly presence before grouping.

2. The SQL — first_month CTE + monthly CTE + year-month offset

WITH first_month AS (
  SELECT user_id, DATE_TRUNC('month', MIN(activity_date))::date AS first_month
  FROM activity GROUP BY user_id
),
monthly AS (
  SELECT DISTINCT DATE_TRUNC('month', activity_date)::date AS month, user_id FROM activity
)
SELECT f.first_month, m.month,
       (EXTRACT(YEAR FROM m.month) * 12 + EXTRACT(MONTH FROM m.month))
       - (EXTRACT(YEAR FROM f.first_month) * 12 + EXTRACT(MONTH FROM f.first_month)) AS months_since_first,
       COUNT(DISTINCT m.user_id) AS active_users
FROM monthly m JOIN first_month f ON m.user_id = f.user_id
GROUP BY 1,2,3
ORDER BY 1,2;

The month-offset formula converts each date to an absolute month index (year × 12 + month), then subtracts — giving an exact integer offset regardless of month length differences.

4. Strategic explanations

Why raw date subtraction gives wrong offsets

Subtracting two DATE values in PostgreSQL gives a number of days, not months. February has 28 days, March has 31 — so "one month later" measured in days varies. The formula year * 12 + month converts both dates to a linear month index and the difference is always an exact integer count of calendar months. This is the central trap in the question; candidates who try (m.month - f.first_month) / 30 will get wrong offsets for many pairs.

DATE_TRUNC to the month

DATE_TRUNC('month', …) maps any date to the first day of its month — the canonical representation for both cohort and activity month. Casting to ::date strips the timestamp, giving a clean DATE. In BigQuery the equivalent is DATE_TRUNC(date_col, MONTH); in MySQL it is DATE_FORMAT(date_col, '%Y-%m-01').

DISTINCT in the monthly CTE

A user active on 10 days in a month should count as one active user for that month — SELECT DISTINCT … month, user_id enforces this at the CTE level. Without it, the final COUNT(DISTINCT m.user_id) would still deduplicate, but the DISTINCT in the CTE is a self-documenting declaration of intent: "each (month, user) pair appears exactly once." Both work; the CTE DISTINCT is marginally cheaper because it avoids duplicating join output rows.

Building the retention triangle in the application layer

This query produces the raw data (first_month, month, offset, users). In practice a BI tool (Looker, Tableau) or Python (pandas pivot_table) pivots the offsets into columns, colours cells by a retention % relative to month-0, and displays the triangle heat-map. The SQL intentionally stays in the "tall" format, which is the right separation of concerns — dynamic pivoting belongs in the presentation layer.

Dialect notes

PostgreSQL, BigQuery, Snowflake, and DuckDB all support DATE_TRUNC. SQL Server uses DATEFROMPARTS(YEAR(date_col), MONTH(date_col), 1). MySQL uses DATE_FORMAT. The year×12+month arithmetic is universally portable — the only "portability" risk is the cast ::date, which is PostgreSQL syntax; standard SQL uses CAST(… AS DATE).

Senior signal: compute month offsets as (year * 12 + month) differences — never raw date subtraction, which yields days; use DATE_TRUNC to collapse daily rows to month-level before joining; add DISTINCT in the monthly CTE to deduplicate multi-day activity; and explain that the "tall" SQL output is the right format, with the triangle pivot delegated to the BI or application layer.
