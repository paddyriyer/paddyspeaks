Plus500's analytics team wants each customer's purchases ranked biggest-spend first — "this was your largest order, this your second largest." The reference solution reaches straight for ROW_NUMBER. It runs and looks right, but the interviewer has seeded a tie: two purchases at the identical amount. How ROW_NUMBER handles that tie — arbitrarily — is the whole conversation.

Customer 101 has two purchases tied at 500 — purchases 2 and 3. That tie decides whether the ranking is deterministic, and it is the entire interview.

2. The SQL — ROW_NUMBER per customer

SELECT *,
  ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY amount DESC) AS rn
FROM purchases;

PARTITION BY customer_id restarts the numbering for each customer; ORDER BY amount DESC ranks within the partition; ROW_NUMBER hands out 1, 2, 3… with no gaps and no ties.

The numbering restarts at customer 102 — that part is solid. But the two tied 500s get rn 1 and rn 2: ROW_NUMBER was forced to pick a winner, and it broke the tie arbitrarily. Here purchase 2 happened to land first, but amount DESC alone does not specify which of two equal amounts comes first — the result is non-deterministic.

4. Strategic explanations

The non-deterministic tie — ROW_NUMBER's sharp edge

When the ORDER BY does not uniquely order the rows, ROW_NUMBER still produces distinct integers — the engine just chooses, and the choice can change between runs, after a re-index, or on a parallel plan. Make the order total by adding a tiebreaker:

ROW_NUMBER() OVER (
  PARTITION BY customer_id
  ORDER BY amount DESC, purchase_date DESC, purchase_id DESC)

Now "second largest" is reproducible: equal amounts break by recency, then by id. A senior answer never ships a ROW_NUMBER whose ORDER BY is not unique.

ROW_NUMBER vs RANK vs DENSE_RANK — which does the question want?

The three differ entirely on ties:

If the two 500s are genuinely "joint largest", ROW_NUMBER is the wrong tool — it lies by assigning 1 and 2. RANK gives 1, 1, 3 (tie shares, gap after); DENSE_RANK gives 1, 1, 2 (no gap). Pick by intent: ROW_NUMBER only when you truly need exactly one row per position.

The "pick one row per customer" pattern

ROW_NUMBER shines for top-N-per-group. Wrap it and filter:

SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (
    PARTITION BY customer_id
    ORDER BY amount DESC, purchase_id DESC) AS rn
  FROM purchases
) t
WHERE rn = 1;

You cannot put WHERE rn = 1 in the same SELECT — window functions are evaluated after WHERE — so the subquery (or a CTE) is mandatory. Some engines offer QUALIFY rn = 1 as sugar; Postgres does not, so the subquery is the portable form.

NULL amounts and the sort

A purchase with a NULL amount sorts first under ORDER BY amount DESC in Postgres — so a priceless row would be crowned the customer's "largest" purchase. Decide deliberately: ORDER BY amount DESC NULLS LAST to sink them, or filter WHERE amount IS NOT NULL if unpriced purchases should not be ranked at all.

Single-purchase customers

A customer with exactly one purchase gets a single row with rn = 1 — the partition logic handles it for free, no special case. Worth a sentence in an interview: confirm the edge behaves, then move on.

Senior signal: add a tiebreaker column so ROW_NUMBER is deterministic, articulate ROW_NUMBER vs RANK vs DENSE_RANK by how each treats ties, know the window result needs a subquery wrapper before it can be filtered, and handle NULL amounts that would otherwise sort to the top.
