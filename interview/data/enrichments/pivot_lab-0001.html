A growth analyst inherits a finance export where each product is one fat row with four quarterly revenue columns. The charting library they want to feed expects tidy data — one row per observation. The ask is an unpivot: rotate those four side-by-side columns down into rows, so q1_revenue…q4_revenue become values of a single quarter column. The interviewer is checking whether you can flip a wide table to long format in portable SQL, and whether you notice what happens to a NULL cell along the way.

1. The source data — quarterly_kpi

Two cells decide the interview. Gadget's Q2 is a genuine 0 — the product really earned nothing that quarter. Gizmo's Q3 is NULL — the figure is missing or unreported. After the unpivot those two must stay distinguishable: 0 is a measurement, NULL is the absence of one.

2. The SQL — unpivot via stacked UNION ALL

SELECT product, 'Q1' AS quarter, q1_revenue AS revenue FROM quarterly_kpi
UNION ALL
SELECT product, 'Q2', q2_revenue FROM quarterly_kpi
UNION ALL
SELECT product, 'Q3', q3_revenue FROM quarterly_kpi
UNION ALL
SELECT product, 'Q4', q4_revenue FROM quarterly_kpi
ORDER BY product, quarter;

Each branch reads one quarter's column, hard-codes the matching label as a string literal, and aliases the metric to a common name revenue. UNION ALL stacks the four result sets vertically. The column count and types must line up across every branch — that alignment is the unpivot.

4. Strategic explanations

Why this is an unpivot — the mirror image of a pivot

A pivot rotates row values up into column headers. An unpivot does the reverse: it takes column names (q1_revenue …) and turns them into the data of a new column (quarter), while the cells line up under a single value column. Wide-to-long is the shape every charting library, time-series model, and GROUP BY wants — you can aggregate over quarter once it is a column, but you cannot aggregate over four separate columns without naming all four.

UNION ALL, never UNION

Use UNION ALL. Plain UNION adds a global de-duplication pass, and that can silently delete real data: if two products both had (NULL revenue) in the same quarter, or two genuinely equal rows existed, UNION would collapse them to one. The branches here are disjoint by construction — each carries a different quarter literal — so dedup buys nothing and only costs a sort. UNION ALL is both correct and faster.

The performance cost — four scans of one table

This query names quarterly_kpi four times, so the planner reads the table four times, once per branch. For three rows that is irrelevant; for a hundred-million-row table it is four full scans plus the stitch. The single-scan alternative is CROSS JOIN LATERAL (VALUES …) in Postgres, or the dedicated UNPIVOT operator elsewhere — same result, one pass. Knowing the UNION ALL form re-scans is the detail that separates a junior answer from a senior one.

NULL is carried, not coerced — decide if that is what you want

Gizmo's Q3 arrives as NULL because the branch simply selects the column and the cell was already NULL. That is usually correct: "no Q3 figure" should not masquerade as "£0 of Q3 revenue". But if a downstream SUM or chart treats NULL badly, wrap the column — COALESCE(q3_revenue, 0) — only if the business rule truly says missing means zero. Conflating the two is the classic data-quality bug; make the choice explicitly and say so.

Dialect — the dedicated UNPIVOT operator

Oracle, SQL Server, and Snowflake ship an UNPIVOT clause that does this in one scan and far less text — e.g. SELECT product, quarter, revenue FROM quarterly_kpi UNPIVOT (revenue FOR quarter IN (q1_revenue AS 'Q1', q2_revenue AS 'Q2', q3_revenue AS 'Q3', q4_revenue AS 'Q4')). Note one gotcha: SQL Server's UNPIVOT drops NULL input rows by default, so Gizmo's Q3 would simply vanish — the UNION ALL form keeps it. PostgreSQL, MySQL, and SQLite have no UNPIVOT keyword, so the stacked UNION ALL shown here is the portable answer.

Senior signal: recognise unpivot as the inverse of a pivot — column names becoming row values — implement it portably with stacked UNION ALL (never UNION), call out that this re-scans the table once per branch and a LATERAL VALUES or native UNPIVOT does it in one, and note that a plain unpivot preserves the 0-vs-NULL distinction whereas SQL Server's UNPIVOT silently drops NULL cells.
