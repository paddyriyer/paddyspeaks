IoT sensor streams have gaps. ML models break on NaN. The wrong fill strategy fabricates plausible-but-fake data and pollutes the training set. VMware's vCenter datacenter sensors poll temperature, humidity, fan RPM at 5-minute nominal intervals but the real-world data is full of dropouts — failed pings, sensor reboots, network jitter. Before the data feeds the capacity-planning model or the auto-throttle controller, somebody has to fill the gaps; choosing time-weighted over straight linear is the difference between a model that respects physics and one that doesn't.

2. The business question — what the consumer of clean data decides on

  • ML model training data quality. If you feed NaN-containing data to scikit-learn or PyTorch, the model errors out (best case) or trains on a tiny subset of rows that happened to be all-non-null (worst case — silently biased model). Interpolation is the pre-step.
  • Capacity-planning forecasts. VMware's vROps predicts rack thermal capacity 6 hours ahead. The forecast model needs evenly-spaced inputs; gaps must be filled deterministically so the next training run produces the same model.
  • Auto-throttle control. If the rack temperature controller misses a reading and the next-best fill is "yesterday's mean", it would massively over-correct. Interpolation between the surrounding two readings gives a plausible point estimate that keeps the controller stable.

3. The pandas implementation — annotated

def interpolate_irregular(df, ts_col, value_col, method='time'):
    # Don't mutate the caller. .copy() is cheap; bug-on-shared-frames
    # is one of the top pandas misuses.
    out = df.copy()
    out[ts_col] = pd.to_datetime(out[ts_col])

    # DatetimeIndex is REQUIRED for method='time' — pandas inspects
    # the index dtype to decide how to weight the gaps. sort_index()
    # is non-optional: interpolate on unordered data corrupts silently.
    out = out.set_index(ts_col).sort_index()

    if method == 'time':
        # Weight each NaN by ACTUAL time elapsed. A 2-minute gap fills
        # closer to its previous neighbour than a 15-minute gap does.
        # This is the only correct choice for irregularly-sampled data.
        out[value_col] = out[value_col].interpolate(method='time')
    elif method == 'linear':
        # Equal weight per row — IGNORES the timestamp spacing. Only
        # correct when the cadence is uniform. Faster, but wrong for
        # real sensor data.
        out[value_col] = out[value_col].interpolate(method='linear')
    elif method == 'spline':
        # Cubic spline — smooth curve through the points. Good for
        # physical sensors (smooth signal); BAD for step-shaped signals
        # like network packets/sec, where it over-shoots between steps.
        out[value_col] = out[value_col].interpolate(method='spline', order=3)

    # Edge handling — interpolate() can't fill leading or trailing
    # NaN (no neighbour on one side). ffill().bfill() carries the
    # nearest non-null value out to the edges. Acceptable for short
    # edge gaps; for long ones, prefer to truncate instead.
    out[value_col] = out[value_col].ffill().bfill()
    return out.reset_index()

The "time" vs "linear" trap. On evenly-spaced data they produce identical output. On irregular data they diverge — and irregular data is the only kind that needs interpolation in the first place. A candidate who writes method='linear' without qualification has missed the central insight of the problem.

Why ffill().bfill() in that order? If both leading and trailing NaN exist (sensor came online mid-day, went offline mid-day), ffill first carries the first-known value backward (no-op on the leading edge — there's nothing to forward-fill from), then bfill handles the trailing edge. Swapping the order doesn't change the result in the two-edge case, but the convention is forward-then-back because trailing data is more often the user-facing alert lag.

The 00:40 row is the demonstration. Linear interpolation thinks the missing point is "1 step away from the previous and 1 step away from the next" so it averages. Time-weighted knows the missing point is timestamped at 00:40 — 23 minutes after 00:17, only 2 minutes before 00:42 — so the right answer is much closer to 22.9 than to 22.4. Physics agrees: temperature trends are continuous in time, not in row-index.

5. Production patterns — how this scales

  • pandas → 1 GB before OOM. One sensor for one year at 5-min cadence ≈ 100k rows — trivial. The full VMware fleet (~10k racks × 50 sensors × 105k readings/year ≈ 50 B rows/year) is way past pandas. Production splits: Flink for streaming gap-fill (state: last-seen value + timestamp per sensor), Spark for backfill / re-interpolation when the model is re-trained.
  • Kalman filter for high-stakes sensors. When the sensor is noisy AND has gaps, pandas' deterministic interpolation amplifies noise. A Kalman filter (or Gaussian Process) models the true signal + measurement noise + missing data jointly. Industrial IoT (oil rigs, wind turbines) uses Kalman; consumer telemetry usually uses time-weighted because it's 100× cheaper.
  • Multi-sensor cross-imputation. When one sensor's reading is missing but a nearby sensor in the same rack measured 22.7, that's a stronger signal than the time-weighted interpolation of the missing sensor's own history. Production at scale uses a per-rack matrix factorisation. The pandas function above only sees one sensor's history.
  • Don't interpolate "long" gaps. A 6-hour sensor outage shouldn't be interpolated — the values are guesses, not measurements, and feeding them to a controller is dangerous. Production rule: cap interpolation at 3× nominal cadence; longer gaps go to a quarantine table for human review. The demo doesn't enforce this; production must.
  • Datadog metric backfill uses time-weighted interpolation for sparse metric series. AWS CloudWatch and GCP Cloud Monitoring both expose interpolated curves on their dashboards — the API even lets you pick the method (linear, step, spline).

6. Interviewer probes

Probe 1 — "What's the difference between method='time' and method='linear'?"

This is the gateway probe — if the candidate can't answer it, they don't understand interpolation. linear = equal weight per row index, ignores timestamps. time = weight by timestamp distance, respects the actual elapsed time between known points. On uniformly-spaced data they're identical; on irregular data only time respects physics.

Probe 2 — "Spline interpolation can produce values OUTSIDE the surrounding two known points. Is that ever a problem?"

Yes — for step-shaped signals (network packet count, request rate, integer-valued metrics) cubic spline overshoots and creates impossible values like negative requests/sec. Spline is right for continuously-varying physical signals (temperature, pressure, voltage). The senior candidate names the assumption: spline fits assume smoothness; check the signal supports that.

Probe 3 — "What's wrong with .fillna(mean)?"

It pretends the sensor reading at every gap is the global mean. For time-series, that's almost always wrong — temperature at 3 AM is not the mean of 24h temperature. It also kills the variance: training data with mean-filled rows has artificially low std, which biases downstream models. Only acceptable for pre-EDA quick looks; never for production ML pipelines.

Probe 4 — "How would you incrementalise this for streaming?"

Flink keyed state per sensor: state = (last_value, last_ts). On a new event, emit (this_event, this_value) and update state. When the next event arrives, you can retroactively fill the in-between gap by emitting a derived "interpolated" event. The trade-off: emitting interpolated events introduces synthetic data into the stream that downstream consumers can't distinguish from real measurements unless tagged. Always tag interpolated values with an is_interpolated column.

Probe 5 — "Cap on interpolation length — how do you implement it?"

.interpolate(method='time', limit=N) where N is the max consecutive NaN run length to fill. Anything beyond limit stays NaN. Then a final .dropna() or routing to quarantine. Common production pattern: cap at 3× the nominal cadence — a sensor offline for 15 min is interpolatable; offline for 3 hours is not.

Probe 6 — "Edge cases — leading NaN, trailing NaN, all-NaN sensor?"

(a) Leading NaN: ffill can't help, bfill carries the first known value backwards — debatable physics but safe default. (b) Trailing NaN: ffill carries the last value forward — fine for short trailing gaps, dangerous for long ones (the dashboard shows "current temperature" stuck at yesterday's). (c) All-NaN sensor: interpolate leaves it NaN, ffill().bfill() can't fix it, the row goes to quarantine. The function above returns the all-NaN frame as-is — caller has to handle.

Probe 7 — "Hot sensor — one sensor with 10 M readings, 30 % NaN. Will groupby(sensor_id).apply(interpolate_irregular) work at scale?"

In pandas, 10 M rows in one group is on the edge of OOM but workable. The bigger issue is the apply-per-group is Python-level — Spark's pandas-UDF (applyInPandas) is the production path: each sensor's group ships to one executor and runs the pandas interpolation locally. The per-group OOM risk is real; cap with repartition(sensor_id) and a salt for the whale sensors. See customer_growth-0027.
