Every B2C product team has a funnel chart on the wall: Landing → Plan Select → Payment Info → Confirm. Each step loses some users; the question is always "where's the biggest leak?". The query computes per-step user counts and the percent-of-top so the leakiest step pops out instantly.

5 users hit Step 1, 4 hit Step 2, 2 hit Step 3, 1 reached Step 4 (Confirm). The biggest drop-off is Step 2 → Step 3 (4 → 2 = 50% loss).

2. The SQL — distinct count per step + percent-of-top via window function

SELECT
  step,
  COUNT(DISTINCT user_id) AS users,
  100.0 * COUNT(DISTINCT user_id) /
    FIRST_VALUE(COUNT(DISTINCT user_id)) OVER (ORDER BY step) AS pct_of_top
FROM funnel
GROUP BY step
ORDER BY step;

4. Strategic explanations

Why FIRST_VALUE over step

FIRST_VALUE(COUNT(DISTINCT user_id)) OVER (ORDER BY step) grabs the count from the lowest-numbered step (Landing) and replicates it on every row — so every step can divide by the top. Without this, you'd need a self-join to the step-1 count or a CTE — much more code for the same answer.

The "leaky funnel" follow-up — step-to-step ratio

The query above shows percent-of-top, which is good for the headline. The actionable view is step-to-step (how much did THIS step lose vs the previous step), which highlights where the friction is:

WITH step_counts AS (
  SELECT step, COUNT(DISTINCT user_id) AS users
  FROM funnel GROUP BY step
)
SELECT step, users,
  LAG(users) OVER (ORDER BY step) AS prev_step_users,
  ROUND(100.0 * users / NULLIF(LAG(users) OVER (ORDER BY step), 0), 1) AS step_to_step_pct
FROM step_counts
ORDER BY step;
The strict-funnel trap — does the user need to complete previous steps?

The query above counts any user who hit step N, regardless of whether they hit steps 1 through N-1. In a real funnel this is wrong: a user who jumped to "Confirm" via a deep link without doing "Plan Select" should not count toward Step 4. Strict funnel:

WITH per_user AS (
  SELECT user_id,
    MAX(CASE WHEN step = 1 THEN 1 END) AS hit_1,
    MAX(CASE WHEN step = 2 THEN 1 END) AS hit_2,
    MAX(CASE WHEN step = 3 THEN 1 END) AS hit_3,
    MAX(CASE WHEN step = 4 THEN 1 END) AS hit_4
  FROM funnel
  GROUP BY user_id
)
SELECT
  SUM(hit_1)                                                AS step_1,
  SUM(CASE WHEN hit_1 = 1 AND hit_2 = 1                              THEN 1 END) AS step_2,
  SUM(CASE WHEN hit_1 = 1 AND hit_2 = 1 AND hit_3 = 1                  THEN 1 END) AS step_3,
  SUM(CASE WHEN hit_1 = 1 AND hit_2 = 1 AND hit_3 = 1 AND hit_4 = 1    THEN 1 END) AS step_4
FROM per_user;
Time-bounded funnel — "did they finish within 7 days?"

Production funnels add a per-step time window: a user who hit Plan Select 30 days after Landing isn't really in the same funnel session. Add timestamp checks:

MAX(CASE WHEN step = 2 AND ts < landing_ts + INTERVAL '7 days' THEN 1 END) AS hit_2_within_7d
Cohort funnel — break by signup week

The PM's actual question is usually "is the funnel improving or getting worse over time?" Slice by acquisition cohort:

WITH cohorts AS (
  SELECT user_id, DATE_TRUNC('week', MIN(ts)) AS cohort_week
  FROM funnel
  WHERE step = 1
  GROUP BY user_id
)
SELECT c.cohort_week, f.step, COUNT(DISTINCT f.user_id) AS users
FROM cohorts c JOIN funnel f USING (user_id)
GROUP BY c.cohort_week, f.step
ORDER BY c.cohort_week, f.step;
Senior signal: name FIRST_VALUE as a clean alternative to a self-join, articulate the strict-vs-loose funnel distinction unprompted, layer in the time-window per step, and pre-empt the cohort-funnel question. The strict funnel via per-user MAX(CASE) is the over-the-bar move.
