An HR data team has an employees table carrying a denormalised dept_name column that has drifted out of sync with the authoritative departments dimension. The task is a MERGE-style sync: overwrite each employee's dept_name from the matching department. The interviewer is probing whether you know Postgres's UPDATE ... FROM idiom, how it handles non-matching and stale rows, and the join-multiplicity hazard that makes UPDATE FROM dangerous if you are careless.

1. The source data — employees and departments

Dave is the trap: his dept_id 9 does not exist in departments. The interview turns on what happens to a row whose key finds no match in the source — does it error, blank out, or stay untouched?

2. The SQL — the MERGE-equivalent UPDATE ... FROM

-- Setup: departments dimension
CREATE TABLE departments (dept_id INT PRIMARY KEY, dept_name TEXT NOT NULL);
INSERT INTO departments VALUES (1,'Engineering'),(2,'Sales'),(3,'Marketing'),(4,'Finance');

CREATE TABLE employees (emp_id INT PRIMARY KEY, name TEXT, dept_id INT, dept_name TEXT);
INSERT INTO employees VALUES
  (101,'Alice',1,NULL),
  (102,'Bob',  2,NULL),
  (103,'Carol',3,'Mktg'),   -- stale
  (104,'Dave', 9,NULL);     -- dept_id 9 absent from departments

-- The sync
UPDATE employees e
   SET dept_name = d.dept_name
  FROM departments d
 WHERE e.dept_id = d.dept_id;

SELECT emp_id, name, dept_id, dept_name FROM employees ORDER BY emp_id;

UPDATE ... FROM joins the target employees to departments on dept_id; the WHERE is the join condition. Only rows that find a partner are touched.

Carol's stale 'Mktg' is corrected to 'Marketing'; Alice and Bob's NULLs are filled. Dave's row is not modified at all — dept_id 9 joins to nothing, so the WHERE excludes his row from the update. His dept_name stays NULL; the statement did not error and did not blank a different column.

4. Strategic explanations

The WHERE in UPDATE ... FROM is a join filter — non-matches are skipped, not erased

This is the key insight. Rows of employees that fail the join condition are simply not part of the update set — they keep their existing values. So a missing dimension key (Dave) leaves the row alone; it does not set dept_name to NULL. If the business rule were "blank out unknown departments", you would need a separate UPDATE ... WHERE dept_id NOT IN (SELECT dept_id FROM departments). UPDATE FROM is an inner-join update by nature.

The hidden danger — a non-unique join key silently corrupts data

UPDATE ... FROM has no protection against fan-out. If departments had two rows with dept_id = 3, each target row would join twice and Postgres would apply one arbitrary matched value — non-deterministic, with no error. Here dept_id is the primary key of departments, so each employee matches at most one row and the update is well-defined. Always confirm the source side of an UPDATE FROM is unique on the join key; this is the senior caveat.

ANSI MERGE — the same intent, more explicit

SQL Server, Oracle, and PostgreSQL 15+ support standard MERGE, which spells out each branch: MERGE INTO employees e USING departments d ON e.dept_id = d.dept_id WHEN MATCHED THEN UPDATE SET dept_name = d.dept_name;. MERGE lets you also handle WHEN NOT MATCHED explicitly — insert a row, or set a sentinel — which is exactly the Dave case. On older Postgres or for a pure update, UPDATE ... FROM is the established equivalent.

Idempotency — this sync is safe to re-run

Running the statement twice produces the same table state: the second run re-sets each dept_name to the value it already holds. That makes it safe in a scheduled job. A senior answer adds AND e.dept_name IS DISTINCT FROM d.dept_name to the WHERE — this skips rows that already match, so the statement only writes (and only fires triggers / generates WAL) for rows that genuinely changed.

Dialect — the FROM-in-UPDATE syntax is not portable

Postgres and SQL Server both allow a FROM clause in UPDATE, but the syntax differs (SQL Server: UPDATE e SET ... FROM employees e JOIN departments d ...). MySQL uses a multi-table UPDATE with the join in the UPDATE clause itself. Oracle historically used a correlated subquery: SET dept_name = (SELECT d.dept_name FROM departments d WHERE d.dept_id = e.dept_id) — but that form sets NULL for non-matches, unlike the join-filtered Postgres version. Know that the "obvious" syntax changes per engine.

Senior signal: understand that UPDATE ... FROM is an inner-join update — unmatched target rows are left untouched, not nulled — insist that the source table be unique on the join key (a non-unique key applies an arbitrary value with no error), reach for ANSI MERGE when you also need a NOT MATCHED branch, and add IS DISTINCT FROM to make the sync write only genuinely-changed rows.
