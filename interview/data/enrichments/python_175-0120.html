Google opens many phone screens with "Maximum Depth of Binary Tree" as a warm-up — but it is also a fast filter. In one line of recursion it checks whether you instinctively define the base case first, whether you trust recursion to compose, and whether you can immediately discuss the iterative rewrite when the interviewer says "now assume the tree is a million nodes deep."

1. The input data

A small binary tree. None marks an absent child:

        3
       / \
      9   20
         /  \
        15   7

# nodes 9, 15, 7 are leaves; expected max depth = 3

2. The approach

A textbook post-order recursion, written as a single expression. The structure is "base case or recursive case":

  • Base case: 0 if not root — an empty subtree contributes depth 0. This also handles the entirely-empty-tree input.
  • Recursive case: 1 + max(maxDepth(root.left), maxDepth(root.right)) — the depth of a node is 1 (for itself) plus the depth of its deeper child.

The recursion is post-order because both children must return before the parent can combine them. max is the right combiner: depth is the longest root-to-leaf path, so the shallower branch is discarded. Each call frame does O(1) work; correctness rests on trusting that the recursive calls answer the same question for smaller trees.

3. The output

subtree rooted atleft depthright depthreturns
9 (leaf)001
15 (leaf)001
7 (leaf)001
20112
3 (root)123

Final answer: 3. The path 3 → 20 → 15 (or 3 → 20 → 7) has three nodes.

4. Strategic explanations

Complexity

Time O(n) — every node is visited exactly once. Space O(h) where h is the tree height, for the recursion call stack. For a balanced tree h = O(log n); for a degenerate "linked-list" tree h = O(n).

The idiomatic alternative — iterative BFS

A level-order traversal counts depth by counting levels and never risks the recursion limit:

from collections import deque

def maxDepth(root):
    if not root: return 0
    q, depth = deque([root]), 0
    while q:
        depth += 1
        for _ in range(len(q)):       # drain exactly one level
            n = q.popleft()
            if n.left:  q.append(n.left)
            if n.right: q.append(n.right)
    return depth

The for _ in range(len(q)) snapshot is the idiom that processes one full level per outer iteration. Same O(n) time; space is O(w), the maximum level width.

The edge case

Empty tree — root is None → returns 0, the base case earns its keep. Single node → both children are None → 1 + max(0, 0) = 1. Skewed tree (every node has only a left child) — still correct, but the recursion depth equals n, which is what motivates the iterative version.

The follow-up

"Now find the minimum depth." It is not just swapping max for min — a node with one missing child must follow the existing child, not count the None side as depth 0. You must special-case "only one child present." Interviewers love this because the naive symmetric edit is wrong.

The trap

For very deep trees, Python's default recursion limit (~1000 frames) raises RecursionError. A junior's recursive one-liner crashes on a 100k-node skewed tree. The senior either names this limit and switches to the BFS/explicit-stack version, or mentions sys.setrecursionlimit as a fragile last resort.

Senior signal: writes the base case before the recursive case, explains why max (not sum) is the correct combiner, proactively names Python's recursion-limit risk on deep/skewed trees, and warns that the "minimum depth" follow-up is not a symmetric one-character change.
