Activision Blizzard's storefront team wants every product tagged into a price band — Cheap, Standard, Premium, Luxury — for merchandising filters and reporting buckets. A CASE expression is the obvious tool. The interview is not "can you write a CASE"; it is whether you get the boundary logic exactly right, what the ELSE branch quietly absorbs, and what happens to a product with no price.

Products 5 and 6 sit exactly on the band boundaries (10 and 100) — the probe for off-by-one errors. Product 7 has a NULL price — the probe for how the CASE handles missing data. Both will reveal whether the bands behave as a senior would expect.

2. The SQL — CASE price bands

SELECT product_id,
  CASE WHEN price < 10 THEN 'Cheap'
       WHEN price < 100 THEN 'Standard'
       WHEN price < 1000 THEN 'Premium'
       ELSE 'Luxury' END AS price_band
FROM products;

A searched CASE evaluates branches top to bottom and stops at the first true one. The order matters: each later WHEN implicitly assumes the earlier ones were false.

The boundary rows are revealing. Product 5 at exactly $10 is Standard, not Cheap — price < 10 is false at 10. Product 6 at exactly $100 is Premium, not Standard — same reason. And product 7, with a NULL price, lands in Luxury — almost certainly a bug, explained below.

4. Strategic explanations

The NULL falls through to ELSE — the real trap

This is the bug an interviewer is hunting for. For product 7, price IS NULL, so every comparison — NULL < 10, NULL < 100, NULL < 1000 — evaluates to unknown, never to true. No WHEN matches, and the row drops through to ELSE 'Luxury'. An unpriced product is silently labelled the most expensive band. The fix is an explicit NULL branch first:

CASE WHEN price IS NULL THEN 'Unknown'
     WHEN price < 10   THEN 'Cheap'
     WHEN price < 100  THEN 'Standard'
     WHEN price < 1000 THEN 'Premium'
     ELSE 'Luxury' END

Never let ELSE be the place NULLs and bad data quietly accumulate — make the catch-all branch mean exactly one thing.

Boundary semantics — < vs <=

Because the bands use strict <, the breakpoints 10, 100, 1000 belong to the upper band: $10 is Standard, $100 is Premium, $1000 is Luxury. That is a deliberate, consistent rule — the bands are [0,10), [10,100), [100,1000), [1000,∞), with no gaps and no overlaps. If the spec says "$10 and under is Cheap", you would switch to price <= 10. Confirm which side the boundary lives on; a merchandising team usually has a firm answer.

Branch order is load-bearing

The WHENs only read correctly because they run top-down. price < 100 really means "≥10 and <100", because reaching it guarantees price < 10 already failed. Reorder them — put price < 1000 first — and every product under $1000 collapses into one band. A common interview misstep is writing the bands in the wrong order; the SQL still parses, the result is just wrong.

Negative or zero prices — another ELSE escape

What about a price of 0 (a freebie) or a negative price (a credit/refund line)? Zero satisfies price < 10 and lands in Cheap, which may be acceptable. A negative price also lands in Cheap. If those are not real products, filter them or give them their own branch. The point is the same: anticipate the values outside the happy path before ELSE swallows them.

Maintainability — bands as data, not code

Hard-coded thresholds in a CASE are fine for one query but rot when marketing rebands the catalogue. A more durable design stores bands in a small lookup table and joins:

SELECT p.product_id, b.band_name
FROM products p
JOIN price_bands b
  ON p.price >= b.min_price AND p.price < b.max_price;

Now rebanding is a row update, not a code change — and note this JOIN form naturally excludes NULL-priced products rather than misfiling them, which is often the safer default.

Senior signal: catch that a NULL price falls through every WHEN into ELSE and add an explicit price IS NULL branch first, state the boundary rule (< puts 10/100/1000 in the upper band) so the bands tile without gaps, and recognise that branch order is logic — reordering the WHENs silently breaks the classification.
