Snap's community team wants a quick read on its forum: who posts the most, and when did each of them last show up. It is two metrics per user — a count and a recency — from a single grouped scan of the post log. Easy on the surface, but the interviewer is checking whether you understand that COUNT(*) and MAX(posted_at) coexist happily in one GROUP BY, and whether you handle the user who has gone quiet.

User 10 is the heavy poster (3 posts), user 20 is moderate (2), and user 30 is a single-post user — the edge case. A single-member group still aggregates correctly: COUNT is 1 and MAX equals that one timestamp. Worth confirming, because grouping logic that quietly assumes “more than one row” breaks here.

2. The SQL — COUNT + MAX per user

SELECT user_id, COUNT(*) AS posts,
  MAX(posted_at) AS last_post_at
FROM posts GROUP BY user_id ORDER BY posts DESC;

One row per user_id. COUNT(*) is the post tally, MAX(posted_at) is the most recent post time, and ORDER BY posts DESC puts the most active users on top.

4. Strategic explanations

Two aggregates, one GROUP BY

A frequent beginner instinct is to compute the count and the recency in two separate queries and join them. Unnecessary — any number of aggregate functions can sit in one SELECT over the same GROUP BY. Each is computed independently across the group's rows. One scan produces both columns; the join is wasted work.

COUNT(*) vs COUNT(posted_at)

Here they agree, but they are not the same. COUNT(*) counts rows; COUNT(posted_at) counts rows where posted_at is non-NULL. If a post could have a NULL timestamp (a draft, a bad import), COUNT(*) still counts it as a post while COUNT(posted_at) would not. For "how many posts did this user make", COUNT(*) is right — a post exists whether or not its timestamp survived.

The user who never posted — the missing-zero trap

This query can only ever report users who appear in posts. A registered user with zero posts produces no rows, so they are absent — not shown with posts = 0. If the community team wants every user including the silent ones, you must start from the users table and LEFT JOIN:

SELECT u.user_id,
       COUNT(p.post_id)      AS posts,
       MAX(p.posted_at)      AS last_post_at
FROM users u
LEFT JOIN posts p ON p.user_id = u.user_id
GROUP BY u.user_id
ORDER BY posts DESC;

Note the count is COUNT(p.post_id), not COUNT(*): after a LEFT JOIN, a user with no posts has one all-NULL joined row, and COUNT(*) would wrongly report 1. COUNT(p.post_id) counts only real posts, yielding the correct 0.

Stabilising the sort

ORDER BY posts DESC leaves users with equal post counts in arbitrary order. For a deterministic leaderboard, add a tiebreaker — most-recently-active first reads naturally: ORDER BY posts DESC, last_post_at DESC. Without it, two equally-active users can swap places between runs.

Inactive-user follow-up — using MAX as a filter

The recency column doubles as a churn signal. "Which users haven't posted in 30 days?" is a HAVING on the same MAX:

SELECT user_id, MAX(posted_at) AS last_post_at
FROM posts
GROUP BY user_id
HAVING MAX(posted_at) < CURRENT_DATE - INTERVAL '30 days';

This is why MAX(posted_at) is more useful than it looks — the same aggregate that reports recency also identifies the at-risk accounts.

Senior signal: compute both aggregates in a single GROUP BY rather than joining two queries, distinguish COUNT(*) from COUNT(column), recognise that an inner-join-style summary silently omits zero-post users and switch to a LEFT JOIN with COUNT(post_id) when every user must appear, and add a deterministic tiebreaker to the leaderboard sort.
