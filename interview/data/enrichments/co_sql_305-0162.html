Uber's commerce team sells through two channels — an online store and physical retail — each landing in its own table. Finance wants one combined view: total sales per channel, stacked in a single result set. This is a UNION ALL of two aggregates, and the interviewer is checking whether you know why UNION ALL and not UNION, and how to stamp each branch with a channel label.

1. The source data

online_sales:

sameamount
x500
y300

retail_sales:

sameamount
a100
b250
c75

Two separate tables, identical shape. Online totals 800, retail totals 425. Note the retail total (425) and online total (800) are different numbers — but watch what would happen if they coincided: UNION (no ALL) would collapse the two identical totals into one row.

2. The SQL — UNION ALL of two aggregates

SELECT 'online' AS channel, SUM(amount) FROM online_sales
UNION ALL
SELECT 'retail', SUM(amount) FROM retail_sales;

Each SELECT aggregates one table to a single row and prepends a constant string naming the channel. UNION ALL stacks the two one-row results into a two-row output.

3. The resulting analysis

channelsum
online800
retail425

One row per channel, exactly as finance wants. The literal 'online' / 'retail' in each branch is what makes the rows identifiable — without it the result would be two anonymous numbers and you could not tell which channel was which.

4. Strategic explanations

UNION ALL vs UNION — the deduplication trap

UNION removes duplicate rows; UNION ALL keeps every row. With the channel label present, the two rows here ('online', 800 and 'retail', 425) differ and would survive either way. But suppose both channels happened to total the same amount — say 500 each. UNION would see two identical (label, 500)… no, the labels still differ. The real danger is dropping the label: SELECT SUM(amount) FROM online ... UNION SELECT SUM(amount) FROM retail with equal totals collapses to one row, silently losing a channel. Beyond correctness, UNION pays for a sort/hash to dedupe; UNION ALL skips it and is faster. Default to UNION ALL unless you specifically need duplicate removal — and say so.

The channel label is not optional

Each branch hard-codes its channel as a string literal. That constant is the only thing tying a number back to its source — it is data, not decoration. A frequent mistake is omitting it and assuming row order identifies the channel; row order from a set operation is not guaranteed. The label belongs in every branch.

Column count, order, and type must line up

Set operations match columns by position, not by name. Both branches here project two columns — (text, numeric) — in the same order, so they align. The result column names come from the first SELECT only; the second branch's names are ignored, which is why aliasing the second branch is pointless. If the branches had different column counts, or a text lined up against a number, the query would error. Alias the aggregate in the first branch so the output column has a real name:

SELECT 'online' AS channel, SUM(amount) AS total_sales FROM online_sales
UNION ALL
SELECT 'retail' AS channel, SUM(amount) AS total_sales FROM retail_sales;
The schema typo — same

Both tables have a column literally named same — almost certainly a typo for name or an id column. It does not affect this query (we only touch amount), but flag it: a column called same is a maintenance landmine, and noticing schema oddities is part of the job.

Scaling to more channels — and a grand total

UNION ALL extends naturally: add a third SELECT for a marketplace channel, and so on. If finance also wants the all-channels total, do not run a separate query — wrap and aggregate:

SELECT channel, total_sales FROM (
  SELECT 'online' AS channel, SUM(amount) AS total_sales FROM online_sales
  UNION ALL
  SELECT 'retail', SUM(amount) FROM retail_sales
) c
UNION ALL
SELECT 'ALL', SUM(total_sales) FROM (
  SELECT SUM(amount) AS total_sales FROM online_sales
  UNION ALL SELECT SUM(amount) FROM retail_sales
) g;

For many channels, the cleaner long-term design is one sales table with a channel column — then it is a plain GROUP BY. Mentioning that the table-per-channel layout is the real problem is the senior observation.

Senior signal: default to UNION ALL and explain that UNION's deduplication can silently merge legitimately-equal rows while also costing a sort, stamp every branch with its channel label, know that set operations align columns by position with names taken from the first SELECT, and note that a per-channel table layout is itself a design smell.
