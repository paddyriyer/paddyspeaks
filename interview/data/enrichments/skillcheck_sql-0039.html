A merchandising team wants the top 3 products by sales in each category — but with a fairness rule: if products tie at the cut-off, all of them stay, even if that means 4 rows. This is top-N with ties, and the interviewer is probing whether you know which ranking function honours ties and how it differs from a plain LIMIT.

Category A has a tie at the boundary (p3 and p4 both at 80, both rank 3) — both must appear, yielding 4 rows. Category C has just one product, a single-member-group edge case. Category B has a three-way tie that all shares rank 1.

2. The SQL — DENSE_RANK keeps boundary ties

SELECT category, name, sales
FROM (
  SELECT category, name, sales,
         DENSE_RANK() OVER (PARTITION BY category ORDER BY sales DESC) AS rk
  FROM products
) t
WHERE rk <= 3
ORDER BY category, sales DESC, name;

DENSE_RANK() assigns a rank per category by descending sales; tied rows get the same rank. Filtering rk <= 3 in the outer query keeps the top three rank levels — and because tied products share a rank, every product at the 3rd level survives the cut.

4. Strategic explanations

Why DENSE_RANK, not ROW_NUMBER

ROW_NUMBER() assigns a strict 1,2,3,… — it would give p3 number 3 and p4 number 4, and rk <= 3 would arbitrarily drop one of the tied products by physical order. That breaks the fairness rule. DENSE_RANK gives equal rows the same rank, so a boundary tie is kept or dropped as a unit — exactly the "include ties" requirement.

DENSE_RANK vs RANK — a subtle choice

Both honour ties, but they differ after a tie. RANK skips: ranks go 1, 1, 3 — so a category with a 2-way tie for 1st has its next product at rank 3, and rk <= 3 would yield only 3 rows (1,1,3) and stop. DENSE_RANK never gaps: 1, 1, 2 — "top 3" means three distinct sales levels. For "top 3 distinct values, ties included", DENSE_RANK is right. If the rule were "top 3 rows, plus ties at the 3rd row", that is RANK() <= 3. Naming which interpretation the product wants is the senior move.

Why a per-category LIMIT cannot do this

LIMIT 3 (or FETCH FIRST 3 ROWS) is purely positional — it counts rows and stops, blind to ties, and it has no notion of "per category" without a lateral join. Category A would lose p4. SQL Server's TOP 3 WITH TIES is the one built-in that does honour boundary ties — but only for the whole result, not per group. A windowed DENSE_RANK filter is the portable way to get tie-inclusive top-N within each partition.

Why the rank must be in a subquery

Window functions are computed after the WHERE clause, so you cannot write WHERE DENSE_RANK() OVER (...) <= 3 directly — the engine rejects it. Wrap the ranking in a subquery or CTE, then filter rk in the outer query. The final ORDER BY category, sales DESC, name is presentation only; name as the last key keeps tied rows in a stable, readable order.

Senior signal: for tie-inclusive top-N per group, rank with DENSE_RANK() OVER (PARTITION BY ... ORDER BY ... DESC) in a subquery and filter rk <= N — ROW_NUMBER would arbitrarily split a boundary tie, and a plain LIMIT is tie-blind; distinguish DENSE_RANK (no gaps, top-N distinct values) from RANK (gaps after ties).
