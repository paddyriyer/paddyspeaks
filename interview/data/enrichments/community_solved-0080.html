A building-management system needs a comma-separated manifest of which passengers can safely board each lift without exceeding its weight capacity, filling lightest-first. The interviewer is probing two things: can you build a running total with a window function and string-aggregate per group — and, more subtly, do you know that the default window frame mishandles tied weights, silently corrupting the answer.

2. The SQL — running weight, then string-aggregate per lift

Original solution (flagged — tie bug):

WITH lift_cte AS (
  SELECT lp.lift_id, lp.passenger_name, lp.weight_kg, l.capacity_kg,
    SUM(lp.weight_kg) OVER (PARTITION BY lp.lift_id ORDER BY lp.weight_kg ASC) AS running_weight,
    CASE WHEN SUM(lp.weight_kg) OVER (PARTITION BY lp.lift_id ORDER BY lp.weight_kg ASC) > l.capacity_kg
         THEN 1 ELSE 0 END AS flag
  FROM lift_passengers lp
  LEFT JOIN lift l ON lp.lift_id = l.id
)
SELECT lift_id,
       STRING_AGG(passenger_name, ',' ORDER BY passenger_name ASC) AS passenger_list
FROM lift_cte
WHERE flag <> 1
GROUP BY lift_id;

The window SUM(weight_kg) OVER (ORDER BY weight_kg) has no frame clause, so it defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. Under RANGE, all rows that tie on the order-by value are summed together as one peer group. Erin and Frank therefore both get running_weight = 160 (80 + 80), both exceed 150, both are flagged, and lift 2 vanishes from the output — even though a single 80 kg passenger fits comfortably.

Corrected solution — explicit ROWS frame:

WITH lift_cte AS (
  SELECT lp.lift_id, lp.passenger_name, lp.weight_kg, l.capacity_kg,
    SUM(lp.weight_kg) OVER (PARTITION BY lp.lift_id ORDER BY lp.weight_kg ASC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_weight
  FROM lift_passengers lp
  LEFT JOIN lift l ON lp.lift_id = l.id
)
SELECT lift_id,
       STRING_AGG(passenger_name, ',' ORDER BY weight_kg ASC) AS passenger_list
FROM lift_cte
WHERE running_weight <= capacity_kg
GROUP BY lift_id;

4. Strategic explanations

The bug — RANGE vs ROWS framing

When an OVER clause has an ORDER BY but no frame, SQL defaults to RANGE UNBOUNDED PRECEDING. RANGE defines the window by value: every row sharing the current order-by value is inside the frame together. So a running SUM ordered by weight_kg gives all tied-weight rows the same cumulative total — useless for a greedy lightest-first fill. ROWS defines the window by position: each row sees itself and the rows physically before it, so a tie still advances the total one passenger at a time. For any running-total-with-a-cutoff problem, write ROWS explicitly.

The greedy-fill semantics — and its limit

Ordering by ascending weight and cutting off at the first over-capacity row implements a greedy "pack the lightest passengers" heuristic. That maximises headcount, which is the natural reading of the question. It does not maximise total weight carried, and it is not a true knapsack optimum — worth stating so the interviewer knows you chose the heuristic deliberately rather than missing the optimisation problem.

Sort the output list by weight, not by name

The question explicitly says the passenger list must be ordered by increasing weight. The original STRING_AGG(passenger_name, ',' ORDER BY passenger_name) sorts alphabetically — a second, quieter deviation from the spec. The corrected query uses ORDER BY weight_kg inside STRING_AGG so the manifest reads lightest-to-heaviest as required. STRING_AGG is Postgres syntax; MySQL spells it GROUP_CONCAT, Oracle LISTAGG, SQL Server STRING_AGG (with WITHIN GROUP).

The LEFT JOIN and the NULL-capacity trap

LEFT JOIN lift keeps passengers even if their lift_id has no matching lift row — but then capacity_kg is NULL, and running_weight <= NULL is unknown, so every such passenger is dropped. That may or may not be intended. If an unknown lift should be treated as an error, an INNER JOIN makes the assumption explicit; if it should never happen, assert it.

What to say next — the cutoff must be monotonic

Filtering on running_weight <= capacity only works because the running total is monotonically increasing — once you cross the cap you never come back under it, so "keep rows under the cap" and "keep a contiguous prefix" coincide. Point this out: it is why a simple WHERE suffices and you do not need to find the cutoff row and slice. If weights could be negative (refunds, credits in the analogous money problem), that invariant breaks and you would need row-number ≤ cutoff logic instead.

Senior signal: spot that a frame-less SUM() OVER (ORDER BY weight) uses RANGE and double-counts tied weights — fix it with an explicit ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. Then honour the spec's weight ordering inside STRING_AGG, and note the greedy prefix only works because the running total is monotonic.
