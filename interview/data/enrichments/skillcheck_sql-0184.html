A sales analyst wants to find the inflection point where momentum has built enough to lift the cumulative average above the long-run mean — the first sale row (by date) where the running average crosses at or above the global average. The interviewer is probing window-function layering: computing two different window aggregates in the same CTE, then filtering and picking the first qualifying row with ROW_NUMBER.

Global average: (10+20+15+80+60+45+30) / 7 = 260 / 7 ≈ 37.14. The running average starts low (small early amounts) and spikes upward at row 4 when the 80-unit sale arrives.

2. The SQL — dual window aggregates + first-qualifying-row filter

WITH base AS (
  SELECT sale_date, amount,
         AVG(amount) OVER (ORDER BY sale_date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_avg,
         AVG(amount) OVER () AS global_avg
  FROM sale
),
flagged AS (
  SELECT sale_date, amount, running_avg, global_avg,
         ROW_NUMBER() OVER (PARTITION BY (running_avg >= global_avg) ORDER BY sale_date) AS first_above_rn
  FROM base
  WHERE running_avg >= global_avg
)
SELECT sale_date,
       amount,
       ROUND(running_avg::numeric, 2) AS running_avg,
       ROUND(global_avg::numeric, 2) AS global_avg
FROM flagged
WHERE first_above_rn = 1
ORDER BY sale_date;

4. Strategic explanations

Two window specs on the same table scan

The base CTE defines two window functions in one SELECT. The ROWS frame gives the cumulative running average; the frameless OVER () is a full-table aggregate producing the same global_avg on every row. Most optimizers can satisfy both in a single pass over the sorted data. Splitting them into separate CTEs or subqueries is unnecessary and can force two scans.

Why ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW instead of default

The default frame for ORDER BY in a window function is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. On dates, RANGE groups ties: if two sales share the same date, both get the cumulative average including both. ROWS counts physical rows regardless of value ties, which is the correct "running average up to and including this row" semantics the question specifies. Getting the frame clause right is a key senior signal.

The ROW_NUMBER inside flagged trick

After filtering to rows where running_avg ≥ global_avg, the remaining rows may be more than one (Jan 06 and Jan 07 both qualify). ROW_NUMBER() OVER (PARTITION BY (running_avg ≥ global_avg) ORDER BY sale_date) assigns 1 to the earliest such row. The PARTITION BY expression is a boolean constant within the filtered CTE — all rows have the same partition value (TRUE), so the window ranks them purely by date. An alternative is a simpler LIMIT 1 ORDER BY sale_date after the WHERE; the ROW_NUMBER approach is portable to dialects that don't support LIMIT in subqueries.

Floating-point precision

The global average 260/7 is irrational in decimal. The comparison running_avg ≥ global_avg is done on the raw double precision (or numeric) values before rounding — rounding happens only in the final SELECT. If both window functions run at the same precision the comparison is consistent. Applying ROUND before the comparison would introduce artefacts.

Dialect notes

The ROWS frame syntax is ANSI SQL and is supported by PostgreSQL, Snowflake, BigQuery, and SQL Server. MySQL 8+ supports window frames. SQLite supports ROWS frames since 3.25. BigQuery requires explicit ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — it does not infer the frame from ORDER BY alone. The boolean PARTITION BY expression works in PostgreSQL and Snowflake; in SQL Server wrap it in a CASE expression or filter via a sub-CTE instead.

Senior signal: use ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW (not the default RANGE frame) for a true row-by-row cumulative average; compute global_avg in the same CTE with a frameless OVER (); and isolate the first crossing with ROW_NUMBER rather than a fragile outer LIMIT 1 so the logic survives in subquery contexts.
