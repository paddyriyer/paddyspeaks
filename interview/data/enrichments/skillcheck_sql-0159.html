An account-management dashboard wants a "recent activity" strip: for every customer, show their two latest orders side by side. The naive instinct is a window function with ROW_NUMBER(), but the question pins you to a LATERAL subquery — a correlated derived table that runs once per outer row. The interviewer is checking whether you understand that LATERAL lets a subquery reference columns from the table to its left, and why that produces a clean per-group Top-N without a wrapping filter.

Two edge cases steer the answer. C02 has only one order — a LIMIT 2 must gracefully return just that one row, not error or pad. And every customer here has at least one order, so the JOIN LATERAL form is fine; a customer with zero orders would be silently dropped (see below).

2. The SQL — per-customer Top-N via JOIN LATERAL

SELECT c.cust_code, c.cust_name, o.ord_num, o.ord_amount, o.ord_date
FROM customer c
JOIN LATERAL (
  SELECT ord_num, ord_amount, ord_date
  FROM orders
  WHERE cust_code = c.cust_code
  ORDER BY ord_date DESC, ord_num DESC
  LIMIT 2
) o ON TRUE
ORDER BY c.cust_code, o.ord_date DESC, o.ord_num;

The keyword LATERAL grants the subquery the right to see c.cust_code from the row currently being processed. For each customer the inner query sorts that customer's orders newest-first and keeps two. ON TRUE is the join predicate — the real filtering already happened inside via the correlation, so there is nothing left to match on.

4. Strategic explanations

What LATERAL actually unlocks

An ordinary subquery in the FROM clause is evaluated once, independently, and cannot reference other FROM items. LATERAL lifts that restriction: the derived table is re-evaluated for every row of the table on its left, and may use that row's columns. It is the relational equivalent of a for loop calling a parameterised query. That is exactly what per-group Top-N needs — "for each customer, run this little ranked query."

LATERAL vs the window-function approach

The window form — ROW_NUMBER() OVER (PARTITION BY cust_code ORDER BY ord_date DESC) then WHERE rn <= 2 — computes a rank for every row, then throws most away. LATERAL with LIMIT can stop early per customer. With a supporting index on orders(cust_code, ord_date DESC) the lateral does an index range scan and grabs the first two entries per customer — often dramatically faster on a wide table with few rows wanted per group. The window function sorts the whole partition regardless.

The trap: JOIN LATERAL drops empty groups

JOIN LATERAL ... ON TRUE is an inner join: if a customer has no orders, the lateral yields zero rows and that customer vanishes from the output. The question explicitly says "customers with no orders are omitted", so JOIN is correct here. If the requirement were to keep every customer (orders or not), you would switch to LEFT JOIN LATERAL ... ON TRUE, which preserves the outer row and fills the order columns with NULL. Knowing which join keyword encodes which business rule is the senior distinction.

The deterministic tiebreak

Sorting by ord_date DESC alone is ambiguous if a customer placed two orders on the same day — LIMIT 2 would pick an arbitrary pair. The inner ORDER BY ord_date DESC, ord_num DESC adds ord_num as a tiebreaker so the result is reproducible run to run. The outer ORDER BY then re-sorts the final five rows for presentation; note it uses ord_num ascending as its last key, which is the requested display order, distinct from the inner descending pick order.

Dialect notes

LATERAL is SQL-standard and supported by PostgreSQL, Oracle (12c+), and SQL Server (as CROSS APPLY / OUTER APPLY — same idea, different keyword). MySQL gained LATERAL in 8.0.14. Engines without it fall back to the window-function pattern. CROSS APPLY maps to JOIN LATERAL ... ON TRUE; OUTER APPLY maps to LEFT JOIN LATERAL.

Senior signal: recognise LATERAL as a correlated, per-outer-row subquery that enables index-friendly per-group Top-N via LIMIT, choose JOIN vs LEFT JOIN LATERAL based on whether empty groups should survive, add a deterministic tiebreak to the inner ORDER BY, and know it surfaces as CROSS APPLY in SQL Server.
