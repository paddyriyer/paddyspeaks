A Community HR system stores employee attributes in an entity-attribute-value (EAV) table — one row per attribute, infinitely flexible but unreadable. The reporting team wants the opposite: one row per employee with named columns. This is a pivot. The interviewer is watching whether you reach for conditional aggregation, why MAX is the right aggregator, and how a missing attribute behaves.

Employees 401, 402, 404 each have all four attributes. Employee 403 is a recent hire whose salary row simply does not exist yet — only three rows for them. That missing row is the edge case: the pivot must still produce a 403 row, with a NULL salary, not drop the employee.

2. The SQL — MAX(CASE WHEN ...) conditional aggregation

SELECT
  emp_id,
  MAX(CASE WHEN attr_name = 'full_name'  THEN attr_value END) AS full_name,
  MAX(CASE WHEN attr_name = 'department' THEN attr_value END) AS department,
  CAST(MAX(CASE WHEN attr_name = 'years_exp' THEN attr_value END) AS INTEGER)
      AS years_exp,
  CAST(MAX(CASE WHEN attr_name = 'salary'    THEN attr_value END) AS INTEGER)
      AS salary
FROM emp_attributes
GROUP BY emp_id
ORDER BY emp_id;

Group by the row identity (emp_id); for each target column, a CASE exposes attr_value only on the matching attribute row and NULL on every other; MAX then collapses the group, skipping the NULLs and surfacing the one real value.

4. Strategic explanations

Why MAX and not MIN, SUM, or first value

For each output column, exactly one row in the group has a non-NULL CASE result; every other row is NULL. MAX ignores NULLs and returns that single real value — and MIN would do equally well, since there is only one non-NULL to choose from. The aggregator is required purely because GROUP BY demands one: you cannot select a bare attr_value alongside a GROUP BY emp_id. SUM works only for numeric attributes and breaks on text; MAX is the portable choice that handles both. The one assumption: each (emp_id, attr_name) pair is unique. If an employee had two salary rows, MAX would quietly pick the larger and hide the duplicate — worth a sanity-check.

The CASE without ELSE — deliberate

CASE WHEN attr_name = 'salary' THEN attr_value END has no ELSE, so non-matching rows yield NULL. That is essential: those NULLs are exactly what MAX needs to skip. Adding ELSE '' or ELSE 0 would feed real values into the aggregate and corrupt the pivot — a MAX over ('82000', '', '', '') still gives '82000' by luck of string ordering, but the pattern is fragile and breaks for other aggregators. Leave the ELSE off.

The typing problem — EAV stores everything as text

Because one attr_value column must hold names, departments, years, and salaries, it is typed TEXT. years_exp and salary are genuinely numbers, so the solution wraps them in CAST(... AS INTEGER) — without it, sorting or arithmetic on those columns would be lexical ('82000' < '9'). The cast sits outside the MAX here, which is fine. One hazard: if any salary value were non-numeric ('TBD', 'N/A'), the cast throws for the whole query. A defensive version uses a safe cast — in Postgres, NULLIF(regexp_replace(val,'\\D','','g'),'')::int, or simply validate the EAV table upstream.

The dialect-specific PIVOT operator

SQL Server, Oracle, and Snowflake offer a dedicated PIVOT clause:

SELECT emp_id, full_name, department, years_exp, salary
FROM emp_attributes
PIVOT (MAX(attr_value)
       FOR attr_name IN (full_name, department, years_exp, salary)) AS p;

It is the same conditional aggregation under the hood — note it still takes a MAX aggregator. The portable MAX(CASE) form runs in every dialect (Postgres, MySQL, SQLite included) and is the safer interview answer; mention PIVOT as the platform-specific sugar.

The fundamental limit: pivot columns must be known in advance

Every approach here hard-codes the four attribute names. If a new attribute ('location') is added to the EAV table, it simply will not appear — the query has no column for it. A truly dynamic pivot, where the columns are discovered from the data, cannot be done in a single static SQL statement: it requires generating the SQL text from a SELECT DISTINCT attr_name and executing it dynamically. Naming this limitation — "static pivot needs a fixed, known set of columns" — is the senior observation.

Senior signal: recognise EAV-to-wide as conditional aggregation, use MAX(CASE WHEN ... THEN ... END) with no ELSE so absent attributes become NULL cells (not dropped rows), cast text-typed numeric attributes explicitly while guarding against non-numeric values, and state that a static pivot can only project a fixed, pre-known column set.
