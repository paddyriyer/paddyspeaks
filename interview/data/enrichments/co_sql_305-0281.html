GoDaddy's catalogue team wants every product stamped with a position when the list is sorted cheapest-first — a stable sequence number for pagination and "rank 1–10 by price" widgets. It looks like a trivial ROW_NUMBER() call, and it is. The interview lives in one detail: the data has two products at the same price, and ROW_NUMBER handles a tie very differently from how a human would expect a "rank" to behave.

1. The source data — products

product_idnameprice
1Domain Basic$9.99
2Domain Premium$49.99
3SSL Cert$49.99
4Hosting Lite$2.99
5Hosting Pro$19.99

Domain Premium and SSL Cert are tied at $49.99. That tie is the whole question: ROW_NUMBER must still hand them two different numbers, and which product gets the lower one is not determined by price.

2. The SQL — ROW_NUMBER() OVER (ORDER BY price)

SELECT product_id, name, price,
  ROW_NUMBER() OVER (ORDER BY price ASC) AS price_rank
FROM products;

ROW_NUMBER() assigns a strictly increasing integer, 1..n, walking the rows in ascending price order — no two rows ever share a value.

3. The resulting analysis

product_idnamepriceprice_rank
4Hosting Lite2.991
1Domain Basic9.992
5Hosting Pro19.993
2Domain Premium49.994
3SSL Cert49.995

The two $49.99 products get ranks 4 and 5 — distinct numbers for an identical price. Domain Premium landed at 4 and SSL Cert at 5, but that ordering is arbitrary: nothing in ORDER BY price distinguishes them, so the engine is free to swap them between runs.

4. Strategic explanations

The non-determinism trap — an ORDER BY with no tiebreaker

ORDER BY price alone does not specify what happens within a price group. The plan, the index used, even insert order can change which tied row gets 4 vs 5. For a "sequence number" that must be stable across runs — pagination keys, reproducible reports — that is a real bug. Fix it with a deterministic tiebreaker:

ROW_NUMBER() OVER (ORDER BY price ASC, product_id ASC)

Adding the unique product_id guarantees a total order, so the same product always gets the same number.

ROW_NUMBER vs RANK vs DENSE_RANK — pick by intent
productpriceROW_NUMBERRANKDENSE_RANK
Hosting Pro$19.99333
Domain Premium$49.99444
SSL Cert$49.99544

If the title's "sequence" means a unique 1..n line number, ROW_NUMBER is right. If two products at the same price should share a position, that is RANK (ties share, next value skips) or DENSE_RANK (ties share, no gap). The question says "sequence", so ROW_NUMBER fits — but naming the distinction shows you chose it rather than reached for the first window function you remembered.

Top-N pagination — why the subquery wrapper is mandatory

You cannot filter on a window function in WHERE — WHERE price_rank <= 10 fails, because WHERE runs before window functions are computed. Wrap it:

SELECT * FROM (
  SELECT product_id, name, price,
         ROW_NUMBER() OVER (ORDER BY price ASC, product_id) AS price_rank
  FROM products
) ranked
WHERE price_rank <= 10;

Snowflake/BigQuery offer QUALIFY price_rank <= 10 as sugar for exactly this; Postgres has no QUALIFY, so the subquery (or a CTE) is the idiom.

NULL prices and the descending case

If price can be NULL, ascending order puts NULLs last by default in Postgres, so unpriced products get the highest ranks — rarely intended. Be explicit: ORDER BY price ASC NULLS FIRST, or filter WHERE price IS NOT NULL. (Note other engines default NULLs differently — never rely on the default across databases.)

Per-category sequencing — the likely follow-up

"Sequence within each category" just needs a PARTITION BY — the counter restarts at 1 per group:

ROW_NUMBER() OVER (PARTITION BY category ORDER BY price ASC, product_id)
Senior signal: add a unique tiebreaker (product_id) to the ORDER BY so the sequence is deterministic across runs, articulate ROW_NUMBER vs RANK vs DENSE_RANK by how each treats the $49.99 tie, and know that filtering a window result needs a subquery/CTE because WHERE cannot see price_rank.
