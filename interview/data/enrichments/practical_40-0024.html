After a data migration, an engineer needs to reconcile orders_old against orders_new and prove the move was lossless. The deliverable is a single result set listing every order_id that exists on only one side — rows dropped in the migration and rows that appeared from nowhere — with a side column saying which. The interviewer is checking whether you can do a two-way diff and label each discrepancy in one query.

1. The source data — orders_old and orders_new

The overlap (3–6) is the part that must not appear in the output. The edge case is the asymmetry: a clean diff has to catch losses (1, 2, 7) and gains (8, 9, 10) — a one-directional check would miss half the problem.

2. The SQL — two anti-joins stitched with UNION ALL

SELECT 'old_only' AS side, o.order_id
FROM orders_old o
LEFT JOIN orders_new n ON n.order_id = o.order_id
WHERE n.order_id IS NULL

UNION ALL

SELECT 'new_only' AS side, n.order_id
FROM orders_new n
LEFT JOIN orders_old o ON o.order_id = n.order_id
WHERE o.order_id IS NULL

ORDER BY 1, 2;

The top half is an anti-join finding order_ids in orders_old with no match in orders_new; the bottom half does the mirror image. UNION ALL concatenates them, and the literal side column tags which direction each row came from.

Six discrepancies, zero false positives. The matched rows 3–6 are absent — each found its counterpart, so neither anti-join emitted it. The side column turns a raw list into a triage report: old_only rows need investigation for data loss, new_only rows for unexpected inserts.

4. Strategic explanations

Why two queries, not one

A diff is inherently asymmetric: "in A not in B" and "in B not in A" are different questions, and a clean migration is verified only when both are empty. A single LEFT JOIN from old to new finds only the losses; you would silently miss every spurious insert. Computing both directions and labelling each is the whole point — the side column is what makes the result actionable.

UNION ALL vs UNION — pick ALL deliberately

UNION de-duplicates its combined output, which costs a sort or hash and is pure waste here: the two halves are disjoint by construction (an id cannot be both old_only and new_only), so there is nothing to dedupe. UNION ALL skips that step and is faster. Reach for plain UNION only when duplicates are genuinely possible and unwanted.

The set-operator alternative — EXCEPT

Postgres, SQL Server, and SQLite offer SELECT order_id FROM orders_old EXCEPT SELECT order_id FROM orders_new — a built-in set difference that is terser than the anti-join. Two caveats worth voicing: EXCEPT removes duplicates within each side (usually fine for keys), and it treats NULL = NULL as a match, unlike a join's ON. MySQL only added EXCEPT in 8.0.31, so the LEFT JOIN form remains the most portable. Oracle spells it MINUS.

NULL ids would break the join — and that is correct

If either table had a NULL order_id, the ON n.order_id = o.order_id would never match it (NULL never equals NULL), so a NULL id is reported as "only on its side". For a primary-key reconciliation that is the right behaviour — a NULL key is itself a data-quality defect worth surfacing. Note this is exactly where EXCEPT diverges: it would pair two NULLs.

Ordering by position

ORDER BY 1, 2 sorts by the first and second output columns — side then order_id. Positional ORDER BY is handy after a UNION because the column names come from the first SELECT only; referring to position sidesteps any naming ambiguity. It is standard SQL, though some teams consider explicit names clearer.

Senior signal: treat a table diff as two directional anti-joins, combine them with UNION ALL (not UNION, since the halves are disjoint), carry a side label so each discrepancy is triageable, and know that EXCEPT/MINUS is the terser equivalent but differs on NULL matching and dedup — details that decide which to use.
