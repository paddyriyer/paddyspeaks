A data team wants a smoothed version of a noisy daily metric for a dashboard sparkline — a centered 3-day moving average that damps single-day spikes. The subtlety the interviewer is hunting for: a centered window peeks one row into the future, and the first and last days have no neighbour on one side, so the window is asymmetric there. Getting the frame and the edge behaviour right is the whole question.

1. The source data — daily

dv(narrative)
2025-01-0110first day — no preceding row
2025-01-0220full 3-row window
2025-01-0330full 3-row window
2025-01-0440full 3-row window
2025-01-0550last day — no following row

The edge cases sit at both ends. The first row has only itself + 1 following (2 rows); the last row has only 1 preceding + itself (2 rows). The interesting question is what the average should be when the window cannot be the promised three rows wide.

2. The SQL — AVG over a centered ROWS frame

SELECT
  d,
  v,
  ROUND(AVG(v) OVER (ORDER BY d ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING)::numeric, 2)::float AS ma
FROM daily
ORDER BY d;

ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING defines a frame of the row before, the current row, and the row after — centered on "now". AVG averages whatever rows actually fall inside that frame; at the table edges the frame simply contains fewer rows, with no error.

3. The resulting analysis

dvmawindow contents
2025-01-011015(10, 20) — 2 rows, no preceding
2025-01-022020(10, 20, 30) — full window
2025-01-033030(20, 30, 40) — full window
2025-01-044040(30, 40, 50) — full window
2025-01-055045(40, 50) — 2 rows, no following

The first row's ma is 15 — the mean of just 10 and 20, because there is no day before it. The last row's ma is 45 — the mean of 40 and 50. The window shrinks at the edges; it does not pad with zeros or NULLs.

4. Strategic explanations

Why ROWS and not RANGE

ROWS counts physical rows — exactly one before and one after — which is what "3-row window" means. RANGE counts by the value of the ORDER BY column instead, so RANGE BETWEEN 1 PRECEDING AND 1 FOLLOWING over a date would mean "within one unit of this date" and, more dangerously, would lump all rows sharing the current date into one frame. If two readings landed on the same day, RANGE would average them together; ROWS keeps them as distinct positions. For a row-count window, ROWS is the correct and intended choice.

The frame shrinks at the edges — by design

A window frame is clipped to the partition's bounds. At the first row there is no preceding row, so the frame is just two rows; AVG divides by 2, not 3. This is usually what you want for a sparkline. If the business rule were "only show a moving average once a full window exists", you would wrap it: CASE WHEN COUNT(*) OVER (...) = 3 THEN AVG(...) END, yielding NULL at the edges. Knowing both behaviours — and that the default is the shrinking window — is the senior distinction.

An explicit ORDER BY in the window is mandatory

"Preceding" and "following" are meaningless without an ordering. OVER (ORDER BY d ...) is what makes the centered frame well-defined. Omit it and the engine either errors or treats the whole partition as one unordered frame. Note also that if two rows shared a date, the tie-break between them would be arbitrary — add a unique tiebreaker (ORDER BY d, id) when determinism matters.

The rounding cast chain

AVG of integers in PostgreSQL returns numeric; the explicit ::numeric makes ROUND(x, 2) unambiguous, and the final ::float matches the column type the grader expects. A frequent mistake is ROUND(AVG(v), 2) with no cast in engines where AVG of an int truncates to an integer — you would silently lose the fractional part. Be explicit about numeric types around averages.

Senior signal: implement a centered moving average with AVG(v) OVER (ORDER BY d ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING), choose ROWS over RANGE deliberately, and know that the frame shrinks at the partition edges — volunteer whether the edge rows should average fewer values or return NULL.
