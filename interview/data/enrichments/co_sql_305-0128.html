A payments processor sees millions of charges per day. Most are routine — "$47.20 at the gas station, $9.99 Spotify subscription" — but a tiny fraction are statistical outliers: card-testing fraud, stolen-card spending, or a producer bug double-charging a merchant. The risk team needs an automated flag that surfaces those anomalies for human review without requiring hand-tuned thresholds per merchant.

2. The SQL — three-sigma outlier detection

The classic statistical-outlier rule: anything more than 3 standard deviations from the mean is flagged. No hand-tuned thresholds, no per-merchant tables — pure math derived from the data itself.

SELECT txn_id, amount
FROM transactions
WHERE amount > (SELECT AVG(amount) + 3 * STDDEV(amount) FROM transactions)
   OR amount < (SELECT AVG(amount) - 3 * STDDEV(amount) FROM transactions);

-- Engine note: STDDEV is Postgres / Snowflake / BigQuery (and Oracle).
-- SQLite calls it STDEV() via an extension; some Spark dialects use STDDEV_SAMP.

4. Strategic explanations

Why 3σ, not 2σ — false-positive cost

2σ catches roughly 5% of legitimate transactions (the shoulders of a normal distribution). At a million transactions per day, that's 50,000 false flags — a triage team can't process those. 3σ catches ~0.27%, or ~2,700 — an order of magnitude more workable. Risk teams routinely tune up to 4σ or even 5σ for high-volume merchants.

Why STDDEV not range

Range (max − min) is brittle: a single outlier blows it up. Standard deviation gives every value a vote weighted by distance from the mean — robust to a few outliers in either tail and proportional to the natural spread of the data.

The "Friday-night surge" problem — when the model lies

This query has a real failure mode: legitimate spikes look like fraud. Black Friday, payday weekends, and concert ticket sales all create real outliers. Production systems usually compute σ over a rolling 7-day window per merchant per hour-of-week, not the global table — so "Friday at 8pm" gets compared to recent Fridays at 8pm, not to Tuesday morning.

The fix that ships next: per-merchant baselines

One global threshold misses small-merchant fraud. A $300 charge is normal for a furniture store, suspicious for a coffee shop. Production version of this query partitions:

WITH baselines AS (
  SELECT merchant_id,
         AVG(amount)    AS mu,
         STDDEV(amount) AS sigma
  FROM transactions
  WHERE created_at >= NOW() - INTERVAL '7 days'
  GROUP BY merchant_id
)
SELECT t.txn_id, t.amount, b.mu, b.sigma,
       (t.amount - b.mu) / NULLIF(b.sigma, 0) AS z_score
FROM transactions t
JOIN baselines b USING (merchant_id)
WHERE t.amount > b.mu + 3 * b.sigma
   OR t.amount < b.mu - 3 * b.sigma;
Senior signal: the interviewer wants to hear that 3σ is a starting point, not the answer. The follow-ups are: per-merchant baseline, rolling window, handle the bimodal distributions (legit big-ticket vs fraud big-ticket — usually solved with merchant category code joins and supervised models on top of the σ filter).
