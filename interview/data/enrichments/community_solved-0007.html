The same Community brief as its sibling — return brands that grew every year — but this solution takes a different route: tag each row with a 1/0 flag for "did this year beat the last", then keep brands that never recorded a 0. It is a cleaner encoding for one reason worth naming: a flag built on amount > LAG is strict, so it correctly rejects flat years where the delta-sign version does not.

1. The source data — brand_amount

yearbrandamount
2020apple100
2021apple150
2022apple180
2020nike80
2021nike60
2022nike90
2021sony200

apple rises every year; nike dips in 2021 and recovers; sony has one year only — a single-row group whose first (and only) row gets a flag from the LAG default.

2. The SQL — per-row 1/0 flag, then NOT EXISTS a 0

WITH t1 AS (
    SELECT *,
           CASE WHEN amount > LAG(amount, 1, 0) OVER (PARTITION BY brand ORDER BY year ASC)
                THEN 1 ELSE 0 END AS flag
    FROM brand_amount
)
SELECT * FROM brand_amount
WHERE NOT EXISTS (
    SELECT brand FROM t1
    WHERE flag = 0 AND t1.brand = brand_amount.brand
);

flag = 1 when the year strictly beat the prior year, else 0. The outer query keeps a brand only if no row of that brand has flag = 0 — i.e. every year was a strict increase.

3. The resulting analysis

yearbrandamount
2022apple180
2021apple150
2020apple100
2021sony200

apple is returned (all three rows) and so is sony. nike is excluded: its 2021 row (60 < 80) has flag = 0, so the NOT EXISTS probe matches and removes every nike row. Note the apple rows come back in year-descending order — the query has no ORDER BY, so this ordering is the planner's choice, not a guarantee.

4. Strategic explanations

Why the strict > flag beats the delta-sign version

This solution's sibling rejects a brand on amount_change < 0 — a strict decrease. That tolerates a flat year (delta exactly 0 is not < 0). Here the flag is amount > LAG(amount), also strict: a flat year (amount = LAG) is not greater, so flag = 0, and the brand is excluded. If "increased every year" means each year must be strictly higher than the last — the usual reading — this flag formulation is the more faithful one. Naming that difference between the two sibling solutions is the senior move.

The first-year flag — a lucky LAG default

LAG(amount, 1, 0) returns 0 for each brand's debut year. Since amounts are positive, amount > 0 is true, so the first year always gets flag = 1 — it never triggers a false rejection. That is correct here, but it is correct by accident of the default. If a brand could legitimately start at amount = 0 or negative, the debut row would get flag = 0 and the brand would be wrongly dropped even if it climbed forever after. The robust version uses LAG(amount, 1) with no default and treats the first row specially:

CASE WHEN LAG(amount) OVER w IS NULL THEN 1          -- first year: no down-move yet
     WHEN amount > LAG(amount) OVER w THEN 1
     ELSE 0 END AS flag

so the first year is never the reason a brand fails.

Single-row groups — sony again

sony's only row gets flag = 1 (200 > default 0), so it has no flag = 0 row and passes. A one-data-point brand "increasing every year" is vacuously true. If the business wants at least two years before a brand can qualify, guard it:

HAVING COUNT(*) > 1

in a GROUP BY brand reformulation, or a correlated count in the outer WHERE.

The tighter rewrite — GROUP BY ... HAVING MIN(flag)

A brand is fully increasing iff every flag is 1, i.e. the minimum flag is 1:

WITH t1 AS ( ... flag ... )
SELECT brand FROM t1 GROUP BY brand HAVING MIN(flag) = 1;

This avoids the correlated NOT EXISTS subquery entirely — one pass, one aggregate. MIN(flag) = 1 reads as "no zero anywhere", which is exactly the intent.

Output shape and ordering

SELECT * returns every row of the qualifying brands; if only the names are wanted, SELECT DISTINCT brand. Either way add ORDER BY brand, year — the demo output above came back year-descending purely because nothing pinned the order.

Senior signal: explain that an amount > LAG flag is strict and so rejects flat years (unlike a delta < 0 test), notice the LAG default of 0 only saves the first year because amounts happen to be positive, and offer the GROUP BY brand HAVING MIN(flag) = 1 rewrite as the cleanest "every year" encoding.
