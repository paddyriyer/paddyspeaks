Amazon asks this because moving averages are everywhere in their stack — smoothing latency metrics, demand forecasts, fraud signals — and they all arrive as unbounded streams. The interviewer is checking whether you can keep a rolling window in constant time and memory, rather than recomputing a sum over the whole window on every tick. It looks trivial; the trap is the naive O(k)-per-call solution.

1. The input data

A window size = 3, then a stream of calls to next(val):

m = MovingAverage(3)
m.next(1)
m.next(10)
m.next(3)
m.next(5)

2. The approach

The class keeps two pieces of state: a collections.deque created with maxlen=size, and a running integer/float sum self.s. The maxlen argument is the key trick — a bounded deque automatically discards the oldest element when it overflows. On each next(val), the code checks whether the deque is already full (len(self.q) == self.q.maxlen); if so, it subtracts the element about to fall off the left (self.q[0]) from the running sum before appending. Then it appends the new value and adds it to the sum. The average is self.s / len(self.q). Because the deque length tracks how many values have actually arrived, the early calls (before the window fills) divide by the correct smaller count.

3. The output

callwindow contentsrunning sumreturned average
next(1)[1]11.0
next(10)[1, 10]115.5
next(3)[1, 10, 3]144.666...
next(5)[10, 3, 5]186.0

On the fourth call the window is full, so 1 is evicted: the sum goes 14 − 1 + 5 = 18, divided by 3.

4. Strategic explanations

Complexity

next is O(1) time — one pop-check, one append, two arithmetic ops — and O(k) space for a window of size k. That O(1) per call is the whole point: it is what makes the structure usable on an infinite stream.

The idiomatic alternative

You could store the deque and compute sum(self.q) / len(self.q) each call, dropping the running-sum bookkeeping. It is shorter and less bug-prone, but it is O(k) per call. Naming this tradeoff — "I can trade the running sum for clarity if k is tiny" — shows judgment. At interview scale either passes; at Amazon scale the running sum wins.

The edge case — the empty window

Before any next call the deque is empty; calling the average then would be 0 / 0 → ZeroDivisionError. The given API only divides inside next after an append, so len is always ≥ 1 — safe by construction. Worth stating aloud so the interviewer knows you saw it.

The follow-up — floating-point drift

"This runs for months — any problem?" Yes: continually adding and subtracting floats lets rounding error accumulate, so self.s slowly diverges from the true window sum. Fixes: periodically recompute sum(self.q) to resync, use math.fsum, or keep integer inputs as integers. For a sliding median instead of mean, the answer changes entirely — two heaps.

The trap — a plain list

Using a Python list and pop(0) to evict looks fine but is O(k): removing index 0 shifts every remaining element. A deque popping from the left is O(1). Picking the wrong container quietly turns an O(1) design into O(k).

Senior signal: a senior reaches for deque(maxlen=...) so eviction is automatic, keeps a running sum for true O(1) updates, and pre-empts long-run float drift — a junior recomputes sum() every call or evicts from a list with pop(0).
