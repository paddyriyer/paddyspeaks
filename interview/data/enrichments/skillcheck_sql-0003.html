A product team wants to flag "sticky" users — anyone who has ever logged in on three or more consecutive calendar days. There is no built-in "consecutive run" operator, so the interviewer is probing whether you know the gaps-and-islands trick: the difference between a date and its row number is constant inside a run, which turns "consecutive" into a plain GROUP BY.

1. The source data — login

user_idlogin_date(narrative)
12025-01-01 / 01-02 / 01-03a 3-day run — qualifies
12025-01-05isolated, one day after a gap
22025-02-01 / 02-02run of 2
22025-02-04 / 02-05another run of 2 — longest streak still only 2
32025-03-10 … 03-13a 4-day run — qualifies
42025-04-01single login — streak of 1

The traps: user 2 logs in on four days but never three in a row (a one-day gap splits it into 2+2), and user 4 has just one login. A correct query must reject both while accepting user 1, whose qualifying run sits alongside a separate isolated day.

2. The SQL — gaps-and-islands via date minus row number

WITH dt AS (
  SELECT DISTINCT user_id, login_date::date AS d FROM login
),
grouped AS (
  SELECT user_id, d,
         d - (ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY d))::integer AS streak_key
  FROM dt
),
streaks AS (
  SELECT user_id, streak_key, COUNT(*) AS streak_len
  FROM grouped
  GROUP BY user_id, streak_key
)
SELECT DISTINCT user_id FROM streaks WHERE streak_len >= 3 ORDER BY user_id;

The dt CTE de-duplicates dates per user. In grouped, each user's dates are numbered 1, 2, 3… by date order; subtracting that counter from the date yields streak_key — constant for every date in an unbroken run, because date and counter both step by exactly 1. streaks then counts dates per streak_key; any group of 3+ is a qualifying run.

3. The resulting analysis

user_id(why it qualifies)
1Jan 1–3 share one streak_key → streak_len 3
3Mar 10–13 share one streak_key → streak_len 4

Users 2 and 4 are correctly absent. User 2's four logins fall into two streak_keys (Feb 1–2 and Feb 4–5), each only length 2 — the one-day gap on Feb 3 shifts the row counter relative to the date, breaking the constant difference. User 4's single login is a streak of 1. User 1 qualifies on the Jan 1–3 island even though Jan 5 forms a separate length-1 island.

4. Strategic explanations

Why "date minus row number" isolates runs

Walk a consecutive run: the dates increase by 1 each step, and ROW_NUMBER also increases by 1 each step. Their difference therefore stays fixed across the whole run. The instant there is a gap — a missing day — the date jumps by 2 or more while the row number still only advances by 1, so the difference changes and a new island begins. That difference is a synthetic key that is identical within a run and different between runs — exactly what GROUP BY needs.

The DISTINCT on dates is essential

If a user logs in twice on the same calendar day, two rows share a date but ROW_NUMBER still gives them different counters — so date - rownum differs, and the duplicate silently fractures the run. SELECT DISTINCT user_id, login_date::date in the first CTE collapses same-day logins to one row before numbering, so the counter advances once per distinct day. Skipping this is the single most common bug in gaps-and-islands solutions.

Why date arithmetic works — and the dialect catch

In PostgreSQL, date - integer yields a date, and date_a - date_b yields an integer number of days — clean arithmetic that makes this idiom terse. Other engines differ: SQL Server needs DATEADD(day, -rn, d), and MySQL needs DATE_SUB(d, INTERVAL rn DAY). The technique is universal; only the date-math syntax is dialect-specific. An alternative that sidesteps date math entirely is DENSE_RANK() OVER (ORDER BY d) as the counter, or ROW_NUMBER over a contiguous integer sequence.

Edge cases: single logins, same-day repeats, and "at least 3"

A user with one login forms a length-1 island and is correctly excluded by >= 3. The threshold is inclusive — "at least 3 consecutive days" is >= 3, not > 3; a strict > would demand four-day runs and wrongly drop user 1. Note also that the final DISTINCT user_id matters: a user could have several qualifying islands and would otherwise appear multiple times.

What to say next — weekly streaks and gap tolerance

Real "engagement streak" features rarely mean strictly back-to-back days. If the rule were "consecutive weeks", you would group by date_trunc('week', d) first. If a one-day lapse should be forgiven, the row-number trick no longer applies directly — you move to a comparison against LAG(d) with a tolerance, marking a new island only when the gap exceeds N days. Naming how the idiom generalises (and where it stops) is the senior signal.

Senior signal: recognise "consecutive days" as a gaps-and-islands problem, key each run by date - ROW_NUMBER() which stays constant within an unbroken run, de-duplicate same-day logins before numbering, use an inclusive >= 3 threshold, and know the technique extends to weekly streaks or gap-tolerant runs via LAG.
