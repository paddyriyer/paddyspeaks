Citi's people-analytics team runs an attrition dashboard, and the core panel is a monthly count of employee terminations. The query is a DATE_TRUNC and a GROUP BY — but the interviewer is checking two things: that you exclude employees who are still on staff, and that you understand a month with zero departures produces no row at all.

1. The source data — employees

Employee 4 has a NULL term_date — still employed. And note February: nobody left in 2024-02, so that month will simply be absent from the output, not zero.

2. The SQL — DATE_TRUNC + GROUP BY

SELECT DATE_TRUNC('month', term_date)::DATE AS month,
  COUNT(*) AS terminations
FROM employees WHERE term_date IS NOT NULL
GROUP BY DATE_TRUNC('month', term_date)::DATE ORDER BY month;

DATE_TRUNC('month', ...) snaps every termination to the first of its month; WHERE term_date IS NOT NULL drops still-employed staff; GROUP BY + COUNT(*) tallies each month.

January and March each show 2 terminations. February is missing entirely — not 2024-02-01 | 0, just gone — because no row had a February termination, so GROUP BY never created a February group. Employee 4, still active, never entered the count. The numbers are right; the gap is the subtle part.

4. Strategic explanations

The IS NOT NULL filter is load-bearing

Drop WHERE term_date IS NOT NULL and DATE_TRUNC('month', NULL) returns NULL for every active employee. Those NULLs collapse into one GROUP BY bucket, producing a phantom month = NULL row whose count is "number of people still employed" — meaningless on an attrition chart. The filter is not optional politeness; it is what keeps active staff out of a terminations metric.

The missing-month gap — the real trap

A time series with holes breaks line charts and rolling averages. The fix is to generate the full month spine and LEFT JOIN the counts onto it:

SELECT m.month, COUNT(e.emp_id) AS terminations
FROM generate_series('2024-01-01'::date, '2024-03-01'::date, '1 month') AS m(month)
LEFT JOIN employees e
  ON DATE_TRUNC('month', e.term_date)::date = m.month
GROUP BY m.month
ORDER BY m.month;

Now February appears as 2024-02-01 | 0. Note COUNT(e.emp_id), not COUNT(*) — on an unmatched spine row the left join yields one all-NULL row, and COUNT(*) would wrongly count it as 1.

DATE_TRUNC vs TO_CHAR — keep it a date

You could group with TO_CHAR(term_date, 'YYYY-MM') instead. It works, but the result is a string — it sorts lexically (which happens to match chronologically for YYYY-MM) and cannot do date arithmetic. DATE_TRUNC(...)::date stays a real date: it sorts correctly across year boundaries and joins to a date spine. For a dashboard axis, prefer the date.

Termination count is not the same as turnover rate

The question title says "turnover", but the query returns a raw count. True turnover is terminations divided by average headcount that month — 5 departures means very different things at a 50-person and a 5000-person company. If the dashboard label says "rate", you owe a denominator: a headcount snapshot or a count of active employees per month. Clarify count vs rate before shipping.

Time-zone and boundary care

If term_date were a timestamptz, DATE_TRUNC bucket boundaries depend on the session time zone — a termination at 11pm on Jan 31 in one zone is Feb 1 in another. With a pure date column this is moot, but on timestamp data, truncate in an explicit zone: DATE_TRUNC('month', term_date AT TIME ZONE 'UTC').

Senior signal: keep the term_date IS NOT NULL filter that excludes active staff, recognise that zero-termination months vanish and fix it with a generate_series spine plus COUNT(column), prefer a real date bucket over a string, and flag that a raw count is not a turnover rate without a headcount denominator.
