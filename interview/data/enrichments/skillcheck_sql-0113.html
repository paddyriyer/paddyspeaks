A reporting team has long-format daily sales — one row per date per product — and needs it reshaped into a wide grid: one row per date, one column per product. That rotation of distinct row values into column headers is a pivot. The interviewer is checking whether you can express a pivot in portable SQL and whether you fill missing date/product combinations with 0 rather than NULL.

1. The source data — daily_sales

date_idproductqty(narrative)
2025-06-01apples10day 1 has apples and bananas
2025-06-01bananas3… but no cherries → cherry cell must be 0
2025-06-02apples7day 2 has apples and cherries
2025-06-02cherries5… but no bananas → banana cell must be 0
2025-06-03bananas8day 3 has only bananas → apples and cherries both 0

The edge case threaded through every row: no date has all three products. Each output row will have at least one missing combination, and the pivot must turn each of those gaps into a 0, not a NULL — a sales grid reads cleaner with explicit zeros.

2. The SQL — pivot via filtered conditional aggregation

SELECT date_id,
       COALESCE(SUM(qty) FILTER (WHERE product = 'apples'),0)::int   AS apples,
       COALESCE(SUM(qty) FILTER (WHERE product = 'bananas'),0)::int  AS bananas,
       COALESCE(SUM(qty) FILTER (WHERE product = 'cherries'),0)::int AS cherries
FROM daily_sales
GROUP BY date_id
ORDER BY date_id;

GROUP BY date_id collapses each date to one row. Each SUM(qty) FILTER (WHERE product = '...') totals only that product's quantity within the date's group. When a date has no rows for a product, the filtered SUM sees nothing and returns NULL; COALESCE(..., 0) converts that to 0.

3. The resulting analysis

Every missing combination shows as 0, exactly as required. Without the COALESCE, those cells would be NULL — technically a different meaning ("no data") and uglier in a dashboard. The pivot rotates the three distinct product values into three columns.

4. Strategic explanations

Why this is a pivot and a plain GROUP BY is not

SELECT date_id, product, SUM(qty) GROUP BY date_id, product returns the same numbers in long form — one row per date/product pair. A pivot turns the product values into column headers: it is a rotation. The conditional-aggregation idiom — one aggregate per target column — is the portable way to do that, because CASE/FILTER work in every engine.

Why SUM and not MAX for the cell value

The data here has one row per date/product, so SUM, MAX, and MIN would all return the same cell. But SUM is the correct choice of intent: if a date ever had two apples rows, SUM adds them (the day's total apple quantity) while MAX would silently keep only the larger. The aggregate you pick is the business rule — "total quantity" means SUM.

COALESCE placement — outside the aggregate, not inside

COALESCE(SUM(qty) FILTER (...), 0) wraps the result of the aggregate. Writing SUM(COALESCE(qty, 0)) instead would not help: when a product is absent there are zero rows to sum, so SUM still returns NULL — the COALESCE inside never even runs. The empty-group NULL can only be caught after the aggregate. This is a subtle but common mistake.

The hard limit — columns are static

The three product columns are hard-coded. Add a fourth product ('grapes') to the table and it lands in no column — it silently vanishes from the report. Pure SQL cannot grow columns from data: a truly dynamic pivot needs a procedural step (build the SQL text from SELECT DISTINCT product, then execute it) or PostgreSQL's crosstab() from the tablefunc extension. Calling out this limitation unprompted is the senior signal.

Dialect notes — FILTER, CASE, and PIVOT

FILTER (WHERE ...) is standard SQL, supported by PostgreSQL and SQLite 3.30+. MySQL and SQL Server lack it; the portable form is SUM(CASE WHEN product='apples' THEN qty ELSE 0 END) — and with the ELSE 0 the COALESCE becomes unnecessary because non-matching rows contribute 0 directly. Snowflake, Oracle, and SQL Server also ship a dedicated PIVOT clause, which still makes you enumerate the values in an IN list — the same static-column constraint.

Senior signal: recognise reshaping long-to-wide as a pivot, implement it portably with one SUM(... FILTER) or SUM(CASE ...) per column, wrap the aggregate (not the inner column) in COALESCE(..., 0) so empty-group NULLs become zeros, and flag unprompted that the column list is static — a data-driven pivot needs dynamic SQL or crosstab().
