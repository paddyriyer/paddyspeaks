A marketing team wants to run a win-back campaign aimed at registered users who have never placed an order. The query must return every users row with no matching row in orders (joined on user_id = customer_id). The interviewer is testing the anti-join — finding the absence of a match — and whether you know the three ways to write it and which one is NULL-safe.

1. The source data — users and orders

101Bobo@x.comnone — target

The orders table holds rows for customers 100, 102 (twice), and 104. The edge case is customer 102 with two orders — a naive join would return that user twice; a correct anti-join must not emit 102 at all, and must not be confused by the multiplicity.

2. The SQL — LEFT JOIN + IS NULL

SELECT u.user_id, u.name, u.email
FROM users u
LEFT JOIN orders o ON o.customer_id = u.user_id
WHERE o.order_id IS NULL
ORDER BY u.user_id;

The LEFT JOIN keeps every user row; users with no order get all-NULL columns on the orders side. WHERE o.order_id IS NULL then keeps only those unmatched rows — the users who never appeared in orders.

3. The resulting analysis

Three users. Customer 102, despite having two orders, does not appear — both of its matched rows have a non-NULL order_id and are filtered out. The anti-join never multiplies: it returns a user zero times or once, never twice.

4. Strategic explanations

Why filter on a column that is never NULL in real data

The WHERE tests o.order_id — the orders primary key. That column is never NULL for a real order row, so the only way it can be NULL after a LEFT JOIN is when there was no matching row and the join padded the right side with NULLs. Filtering on a nullable-but-actually-populated column (say o.status) is the classic bug: a real order with a NULL status would be misread as "no order". Always test the join key or the PK.

NOT EXISTS — often the better choice

The commented NOT EXISTS form expresses the same intent more directly: "keep users for whom no order exists." On large tables it frequently out-performs the LEFT JOIN, because the engine can stop scanning a user's orders the instant it finds one match — it does not build the full padded join. Semantically it is also clearer: you are asking a yes/no question, not joining and then discarding.

The NOT IN trap with NULLs

The tempting one-liner is WHERE user_id NOT IN (SELECT customer_id FROM orders). It is a landmine: if any customer_id in orders is NULL, the NOT IN evaluates to unknown for every row and the query returns zero rows — silently, no error. This happens because x NOT IN (1, 2, NULL) is x≠1 AND x≠2 AND x≠NULL, and the last term is never true. LEFT JOIN / IS NULL and NOT EXISTS both handle NULLs correctly; NOT IN only when the subquery column is guaranteed non-NULL.

Why the anti-join does not multiply rows

An inner join with a one-to-many relationship multiplies the left row by the number of matches. The anti-join keeps only the zero-match rows, so multiplicity is impossible — each surviving user matched nothing and appears exactly once. That is why customer 102's two orders are irrelevant here, and why you do not need DISTINCT.

Dialect notes

LEFT JOIN, IS NULL, and NOT EXISTS are universal. Modern Postgres and SQL Server optimisers recognise all three anti-join forms and often compile them to the identical "anti-join" physical operator, so the performance gap has narrowed — but NOT IN still carries the NULL semantics, regardless of optimiser, so the safety argument stands in every dialect.

Senior signal: implement an anti-join with LEFT JOIN … WHERE pk IS NULL or NOT EXISTS, filter on a column that is genuinely never NULL (the join key/PK), know the anti-join cannot multiply rows so DISTINCT is unneeded, and reject NOT IN on a nullable subquery because a single NULL makes it return nothing.
