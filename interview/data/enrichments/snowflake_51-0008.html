A customer-support operations team wants a daily call-center dashboard: how many distinct customers called each day, and what was the total call time in both seconds and minutes? The interviewer is probing correct use of DATE() for timestamp truncation, COUNT(DISTINCT) for unique callers (not total calls), and integer arithmetic for unit conversion — a deceptively simple aggregation that trips candidates on the DISTINCT vs plain COUNT distinction.

April 1st has 3 calls from 2 distinct customers (C01 called twice). April 2nd has 3 calls from 2 distinct customers (C01 and C03, with C03 calling twice). The distinction between "3 calls" and "2 customers" on each day is the core of this question.

2. The SQL — daily aggregation with DISTINCT customer count

SELECT DATE(call_ts)             AS day,
       COUNT(DISTINCT customer_id) AS unique_customers,
       SUM(duration_sec)           AS total_duration_sec,
       SUM(duration_sec) / 60      AS total_duration_min
FROM calls
GROUP BY DATE(call_ts)
ORDER BY day;

4. Strategic explanations

DATE() — extracting the calendar day from a timestamp

DATE(call_ts) truncates the timestamp to the date portion, collapsing all calls within a calendar day into the same group. The same expression appears in both SELECT and GROUP BY — this is required because the GROUP BY must match the aggregation key exactly. An alternative is DATE_TRUNC('day', call_ts), which returns a timestamp at midnight; DATE() returns a date type, which is preferable for a date-only report. In Snowflake: DATE(call_ts) or CAST(call_ts AS DATE) — both work. In BigQuery: DATE(call_ts) requires a DATE column or explicit timezone: DATE(call_ts, 'America/New_York').

COUNT(DISTINCT) vs COUNT(*) — the semantic fork

COUNT(*) would return 3 for each day (total calls placed), not 2 (unique callers). The business metric here is unique customers who contacted support — a customer who called three times on the same day counts once for the "how many people did we serve" KPI. COUNT(DISTINCT customer_id) correctly deduplicates. This is also a common performance bottleneck at scale: COUNT(DISTINCT) requires maintaining a set of seen values, whereas COUNT(*) is a simple counter. Snowflake's HLL_COUNT.DISTINCT (HyperLogLog) offers an approximate but very fast alternative at scale.

Integer division truncation for minutes

SUM(duration_sec) / 60 uses integer division in most SQL engines, truncating toward zero. 900 / 60 = 15 (exact). 901 / 60 = 15, not 15.02. If you want fractional minutes, cast first: SUM(duration_sec)::NUMERIC / 60 (PostgreSQL / Snowflake) or SUM(duration_sec) / 60.0 (any dialect). For a dashboard displaying "minutes," truncation is usually fine; for a billing system, fractional accuracy matters.

Missing days in the result

The query only returns days on which at least one call was made. If you need a row for every calendar day (including days with zero calls, e.g., weekends), you would generate a date spine and LEFT JOIN the calls aggregation to it. In Snowflake: GENERATOR(ROWCOUNT => n) or a date dimension table. In PostgreSQL: generate_series. In BigQuery: UNNEST(GENERATE_DATE_ARRAY(...)).

Dialect notes
  • Snowflake: DATE(call_ts) works; TO_DATE(call_ts) is equivalent.
  • PostgreSQL: DATE(call_ts) works; call_ts::date also common.
  • BigQuery: DATE(call_ts) requires timezone awareness for TIMESTAMP columns.
  • MySQL: DATE(call_ts) works identically.
Senior signal: immediately distinguish COUNT(DISTINCT customer_id) (unique people) from COUNT(*) (total calls); note that integer division truncates and explain when to cast for fractional minutes; and mention the date-spine LEFT JOIN pattern for filling in days with zero calls — a frequent follow-up requirement in operational dashboards.
