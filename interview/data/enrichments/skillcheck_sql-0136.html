An e-commerce team wants each customer's "favourite product" — the one they order most often — for a personalised re-order widget. The twist the interviewer is probing: a customer can have two products tied for most-ordered, and the spec explicitly says return all tied products. That single word, "all", decides whether you reach for RANK or ROW_NUMBER.

Customer 2 is the whole interview: products 300 and 400 are each ordered twice — a genuine tie at the top. Customer 3 has exactly one order, the degenerate single-row group. The spec wants both of customer 2's products returned, not an arbitrary one.

2. The SQL — aggregate, then RANK the counts

WITH counts AS (
  SELECT customer_id, product_id, COUNT(*) AS times
  FROM orders
  GROUP BY customer_id, product_id
),
ranked AS (
  SELECT customer_id, product_id, times,
         RANK() OVER (PARTITION BY customer_id ORDER BY times DESC) AS rk
  FROM counts
)
SELECT customer_id, product_id, times::int AS times
FROM ranked
WHERE rk = 1
ORDER BY customer_id, product_id;

The counts CTE collapses raw orders into one row per customer/product with an order tally. RANK() OVER (PARTITION BY customer_id ORDER BY times DESC) then ranks products within each customer by that tally, and WHERE rk = 1 keeps every product holding the top rank — including ties.

4. Strategic explanations

Why RANK, not ROW_NUMBER — the spec said "all tied"

ROW_NUMBER() would assign customer 2's two products the distinct numbers 1 and 2, and WHERE rn = 1 would silently drop one — arbitrarily, by physical row order. RANK() gives tied rows the same rank, so WHERE rk = 1 returns every product at the maximum count. The choice is dictated by the requirement: "return all tied products" means ties must share a rank, which is exactly what RANK does and ROW_NUMBER refuses to.

Two-stage shape — aggregate first, rank second

You cannot rank by COUNT(*) in one pass, because the count and the window function operate at different granularities. The counts CTE first reduces orders to per-product tallies; only then can a second window rank those tallies. Trying to write RANK() OVER (... ORDER BY COUNT(*) DESC) directly is possible in some engines but muddier — the explicit two-CTE form makes "count, then rank" legible, which is what an interviewer wants to see.

"Same combo on different dates" is the point of the COUNT

The question stresses that a customer-product pair can recur across dates — that is precisely why GROUP BY customer_id, product_id with COUNT(*) is correct: it tallies how many times the pair appears regardless of date. If the metric were "distinct days a product was ordered" you would switch to COUNT(DISTINCT order_date); if it were "distinct orders" the current COUNT(*) is right because each row is one order. The aggregate encodes the business definition — state which one you mean.

RANK gaps do not matter here — but know they exist

RANK skips numbers after a tie: if customer 2 had a third, less-ordered product it would be rank 3, not 2. That gap is irrelevant to this query because we only ever keep rk = 1. But if the task were "top two distinct order-counts" you would need DENSE_RANK instead, which never leaves gaps. Knowing the difference — and that it is harmless here — is the senior nuance.

The type label — "ASOF" is a red herring

The question is tagged "SQL / ASOF / Window Functions", but there is no as-of join, no point-in-time lookup, no temporal nearest-match here — it is a tie-aware top-N-per-group. Do not be lured into ASOF JOIN (a DuckDB/ClickHouse construct) or interval gymnastics; recognise the problem as a ranked aggregation and solve that. Mislabelled tags happen — answer the question, not the category.

Senior signal: recognise "return all tied products" as a hard requirement for RANK over ROW_NUMBER, structure the query as aggregate-then-rank in two CTEs, justify COUNT(*) as the chosen definition of "ordered most often", and ignore the misleading ASOF tag — it is a tie-aware top-N-per-group, nothing more.
