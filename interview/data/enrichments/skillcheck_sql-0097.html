A compensation analyst wants the median base salary per department, with one twist: rows where the salary is NULL must be ignored, and any department where every employee has a NULL salary should not appear at all. The interviewer is probing whether you know SQL's ordered-set aggregate for median — not AVG — and whether you handle the all-NULL group correctly.

Three edge cases drive the answer. Department 1 has a NULL salary that must not skew the median. Department 3 has only NULL salaries and must vanish from the output. Department 4 has an even count, so the median sits between two values.

2. The SQL — PERCENTILE_CONT ordered-set aggregate

SELECT dept_id,
       ROUND(PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY base_salary)::numeric, 2) AS median_salary
FROM salary
WHERE base_salary IS NOT NULL
GROUP BY dept_id
ORDER BY dept_id;

PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY base_salary) is the SQL-standard ordered-set aggregate for the continuous 50th percentile — the median. The WHERE base_salary IS NOT NULL filter both excludes NULL rows from the median and removes all-NULL departments before grouping.

Department 3 is absent: every salary was NULL, the WHERE filter removed all its rows, so no group formed. Department 4's median is 50000 — PERCENTILE_CONT interpolates the midpoint of the two middle values (30000 + 70000) / 2.

4. Strategic explanations

Why PERCENTILE_CONT, and what WITHIN GROUP means

SQL has no MEDIAN() aggregate in the standard. PERCENTILE_CONT(0.5) is an ordered-set aggregate: the WITHIN GROUP (ORDER BY base_salary) clause tells it how to sort the group before picking the 50th-percentile position. It is fundamentally different from a normal aggregate — the ordering is part of the function call, not an OVER clause. This is the correct, set-based way to get a median.

PERCENTILE_CONT vs PERCENTILE_DISC

For an even-count group the two diverge. PERCENTILE_CONT (continuous) interpolates — department 4 yields 50000, a value no employee actually earns. PERCENTILE_DISC (discrete) returns an actual data value — it would return 30000 for department 4. The textbook statistical median is the interpolating one, so PERCENTILE_CONT is the right default; mention PERCENTILE_DISC if the business needs a real observed salary.

The all-NULL department: why WHERE makes it disappear

The requirement "departments where every row is NULL must not appear" is satisfied for free by WHERE base_salary IS NOT NULL. The filter runs before GROUP BY; department 3's rows are all removed, so there is no surviving row to seed a group — GROUP BY only creates groups for keys that have rows. If instead you filtered NULLs inside the aggregate and kept the rows, department 3 would appear with a NULL median, violating the spec.

AVG is the wrong tool — know why

A tempting shortcut is AVG(base_salary). It is not the median: for a skewed distribution the mean and median differ sharply, and the question explicitly says median. AVG also happens to ignore NULLs automatically, which can lull a candidate into thinking it is "close enough". It is not — median is a positional statistic, mean is not. Use the aggregate the question names.

The ::numeric cast before ROUND

PERCENTILE_CONT returns double precision. PostgreSQL's two-argument ROUND(value, scale) is only defined for numeric, not double precision — calling ROUND(double, 2) raises "function round(double precision, integer) does not exist". The explicit ::numeric cast bridges the type so the 2-dp rounding works. This is a real PostgreSQL gotcha worth naming.

Senior signal: compute median with the ordered-set aggregate PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) not AVG, know PERCENTILE_CONT interpolates while PERCENTILE_DISC returns a real value, drop NULLs with a WHERE so all-NULL departments never form a group, and cast to numeric before two-arg ROUND.
