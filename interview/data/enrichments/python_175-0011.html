Google asks this as the "trie with a twist" question. A plain prefix tree is mechanical; the twist is the . wildcard in search, which matches any single character. That one rule turns a deterministic walk down the tree into a branching search — and the interviewer is watching to see whether you recognize it as a depth-first search and prune it correctly.

1. The input data

Three words added, then a mix of exact and wildcard searches:

wd = WordDictionary()
wd.addWord('bad')
wd.addWord('dad')
wd.addWord('mad')
wd.search('pad')   # exact, no match
wd.search('bad')   # exact, match
wd.search('.ad')   # wildcard first char
wd.search('b..')   # two wildcards

2. The approach

The trie is a nested plain dict: each node maps a character to a child dict. addWord walks the word, using n.setdefault(c, {}) to create-or-descend in one step, then plants a sentinel key '#' at the end to mark "a word terminates here" — without it, 'bad' would be indistinguishable from a mere prefix of a longer word. search delegates to a recursive dfs(i, n) over index i and current node n. Base case: i == len(w) → the word matched iff '#' is in the node. Otherwise, if the current character is '.', it must try every real child — any(k != '#' and dfs(i+1, n[k]) for k in n) — excluding the sentinel so it is never treated as a letter. For a normal character it is a single deterministic step: c in n and dfs(i+1, n[c]). The and/any short-circuit on the first match.

4. Strategic explanations

Complexity

addWord is O(L) for word length L. An exact search is also O(L). A search with wildcards is the catch: each . can branch into up to 26 children, so the worst case is O(26^d · L) where d is the number of dots — an all-dots query explores the whole tree. Space is O(total characters across all words), shared aggressively via common prefixes.

The idiomatic alternative

A defaultdict-based TrieNode class with an explicit is_word boolean is the more conventional, readable form — the '#' sentinel-in-a-plain-dict trick is compact but mixes a marker into the same namespace as letters, which is exactly why every wildcard branch must remember to filter k != '#'. For a small fixed word list, a regex (. already means "any char") is a legitimate shortcut worth naming.

The edge case — the empty word and the sentinel collision

addWord('') places '#' directly at the root, and search('') then returns True — consistent, worth confirming with the interviewer. The subtler bug: if you forget k != '#', a . would try to "descend into" the sentinel, and dfs(i+1, True) crashes since True is not a dict. The filter is load-bearing, not cosmetic.

The follow-up — a * wildcard

"Now support * meaning zero or more characters." That breaks the one-index-per-character recursion: * must try consuming 0, 1, 2, ... characters, so dfs needs a branch that recurses without advancing i and another that advances the node. It becomes regex-style matching — a noticeably harder DFS with memoization to avoid re-exploring states.

The trap — treating . as a literal

The naive instinct is to walk the trie character by character as if . were just another letter, finding no child '.' and returning False. The wildcard fundamentally is a search, not a lookup — recognizing that search needs recursion while addWord does not is the whole point of the question.

Senior signal: a senior immediately frames the wildcard as a DFS with 26-way branching, prunes the sentinel out of the branch loop, states the O(26^d) wildcard cost honestly, and contrasts it with the O(L) exact path — a junior treats . as a literal character.
