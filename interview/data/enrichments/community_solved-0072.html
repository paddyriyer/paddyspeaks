A retail chain wants a per-store conversion rate: of the people who walked in (visitors), what fraction actually placed an order. The interviewer is probing two things — can you relate a footfall table to an orders table, and do you understand integer-division pitfalls when computing a percentage. The boundary case is a store with visitors but few or no conversions, where the ratio must still come out sensibly rather than as 0 or a divide-by-zero error.

2. The SQL — full join footfall to orders, then a ratio

WITH visitor_customer AS (
  SELECT v.*, o.customer_id, o.order_id, o.order_date
  FROM visitors v
  FULL JOIN store_orders o
    ON v.visitor_id = o.visitor_id
   AND v.visit_date = o.order_date
   AND v.store_id   = o.store_id
)
SELECT store_id,
       COUNT(customer_id) AS customer_count,
       COUNT(visitor_id)  AS visitor_count,
       CAST(CAST(COUNT(customer_id) AS DECIMAL(10,2))
            / COUNT(visitor_id) * 100 AS DECIMAL(10,2)) AS con_rate
FROM visitor_customer
GROUP BY store_id;

4. Strategic explanations

Why COUNT(column) ignoring NULLs does the real work

This is the engine of the query. After the join, a visitor who never ordered has a NULL customer_id. COUNT(customer_id) counts only non-NULL values, so it counts converted visitors; COUNT(visitor_id) counts all visitors. The ratio of the two is the conversion rate. COUNT(*) would have counted every row including the non-converters — the distinction between COUNT(*) and COUNT(column) is exactly what makes this work.

The integer-division trap — why the double CAST exists

COUNT(...) returns an integer. In integer arithmetic 2 / 3 truncates to 0, so 0 * 100 = 0 — every store would report a 0% conversion rate. Casting the numerator to DECIMAL first forces floating-point division, yielding 0.6667; multiplying by 100 and casting again rounds the display to two places. The lesson: in any percentage formula, promote to a decimal/float type before the division, never after.

The divide-by-zero a senior anticipates

If a store had orders logged but somehow zero visitor rows, COUNT(visitor_id) would be 0 and the division would throw. Production-grade SQL guards it: ... / NULLIF(COUNT(visitor_id), 0) — NULLIF turns a zero denominator into NULL, and the result becomes NULL (no rate) instead of erroring. Mentioning this unprompted is a strong signal even though the sample data never triggers it.

FULL JOIN — intent, and a subtle grouping risk

A FULL JOIN keeps visitors with no order and orders with no matching visitor. The first half is essential — non-converting visitors must stay in the denominator (a plain INNER JOIN would drop them and every store would show 100%). But the second half is a quiet hazard: an order with no matching visitor row contributes a row whose store_id may come out NULL (or, if you read store_id only from visitors, is NULL), creating a stray group. For a pure "visitor → conversion" rate, a LEFT JOIN from visitors is cleaner: it keeps exactly the visitor population and never invents orphan-order rows. Use COALESCE(v.store_id, o.store_id) in the SELECT if you keep the full join.

The join key — matching on date is a design decision

The join requires visit_date = order_date: a visit converts only if the order happened the same day. That is a deliberate, narrow definition. If a visitor browses Monday and buys Tuesday, this query counts it as a non-conversion. Real conversion analytics often allow an attribution window (order_date BETWEEN visit_date AND visit_date + 7). Surface this assumption — "are we counting same-day conversions only?" — rather than silently accepting it.

Dialect note — DECIMAL(10,2) in a cast

The CAST(... AS DECIMAL(10,2)) syntax is portable (Postgres, SQL Server, MySQL all accept it). Postgres also offers ROUND(x::numeric, 2) which is more idiomatic for "round to 2 places". The double-cast works fine; just know ROUND expresses the rounding intent more directly.

Senior signal: explain that COUNT(customer_id) vs COUNT(visitor_id) — NULL-skipping vs total — is what computes the rate, and that the double CAST exists solely to dodge integer-division truncation. Adding NULLIF(..., 0) for the empty-store divide-by-zero, and questioning the same-day join window, is the senior finish.
