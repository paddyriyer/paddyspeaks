A Community warm-up on self-joins: a routes table holds both A→B and B→A as separate rows, and the task is to find the pairs that have a mirror. The solution joins the table to itself with the endpoints swapped. It works — but the interviewer is watching whether you notice that this query matches reverse pairs without actually deduplicating them.

1. The source data — src_dest

Three routes appear in both directions (Mumbai–Delhi, Bangalore–Chennai, Hyderabad–Pune). Kolkata–Patna appears only once — it has no reverse partner. That asymmetric row is the edge case that proves the join is doing real work.

2. The SQL — self-join on swapped endpoints

select * from
src_dest t1
join
src_dest t2
on t1.source = t2.destination and t1.destination = t2.source;

The table is joined to a second copy of itself. The condition pairs each row t1 with the row t2 whose source/destination are reversed. A route with no mirror finds no t2 match and, under the inner JOIN, drops out.

Six rows — and that number is the lesson. Kolkata–Patna is correctly absent (no mirror). But each of the three two-way routes appears twice: once as Mumbai→Delhi paired with Delhi→Mumbai, and again as Delhi→Mumbai paired with Mumbai→Delhi. The join finds the partner from both sides, so the result is symmetric — not the deduplicated route list a reader might expect.

4. Strategic explanations

"Finds reverse pairs" ≠ "removes reverse pairs"

This is the trap. The query identifies routes that have a mirror, but it returns both orientations — 3 routes become 6 rows. If the goal is one canonical row per undirected route, add a tie-breaker that keeps only one side:

select t1.* from src_dest t1
join src_dest t2
  on t1.source = t2.destination and t1.destination = t2.source
where t1.source < t1.destination;

t1.source < t1.destination keeps exactly the orientation where the first city sorts earlier — one row per pair. State which output the question wants before writing the join.

The canonical-key alternative — no self-join at all

Self-joins are O(n²)-ish without the right index. The set-based way to dedupe undirected pairs is to sort the endpoints into a key and group:

SELECT LEAST(source, destination) AS a,
       GREATEST(source, destination) AS b,
       MIN(distance) AS distance, COUNT(*) AS directions
FROM src_dest
GROUP BY 1, 2;

directions = 2 marks routes present both ways; = 1 marks one-way routes like Kolkata–Patna. One pass, no join, and it scales.

Interviewer follow-up: find the routes missing a reverse

The mirror of "has a reverse" is often what is really wanted — data-quality gaps. Swap the inner join for an anti-join:

SELECT t1.* FROM src_dest t1
LEFT JOIN src_dest t2
  ON t1.source = t2.destination AND t1.destination = t2.source
WHERE t2.source IS NULL;

That returns Kolkata–Patna — the route entered in only one direction.

Edge cases: mismatched distances and self-loops

The join matches on city names only, not distance — so Mumbai→Delhi 1400 would still pair with Delhi→Mumbai 1390. If a reverse pair must also agree on distance, add AND t1.distance = t2.distance. And a self-loop row (source = destination) trivially satisfies the swap condition and joins to itself — exclude it with WHERE t1.source <> t1.destination if such rows exist.

Senior signal: recognise that the swapped self-join matches reverse pairs but returns both orientations — n routes become 2n rows — and add a source < destination filter to deduplicate, or skip the join entirely with a LEAST/GREATEST group-by key that also scales.
