Intel's manufacturing floor tracks every part rolling off a production line. The ops team needs two counters on every row: "what is this part's position in its own line's sequence today?" and "what is its position across the entire factory today?" This is the dual-sequencing pattern — one partitioned ROW_NUMBER for local rank and one unpartitioned for global rank — with no aggregation and no join.

2. The SQL — dual ROW_NUMBER: partitioned and global

SELECT part_id, manufactured_at, line_id,
       ROW_NUMBER() OVER (PARTITION BY line_id ORDER BY manufactured_at) AS line_seq,
       ROW_NUMBER() OVER (ORDER BY manufactured_at)                      AS global_seq
FROM parts;

L1 has three parts (line_seq 1–3), L2 has three (1–3), L3 has one (1). The global_seq reflects the wall-clock order across all lines: P001 came first factory-wide (1), P006 was sixth (6) even though it was line L3's very first part (line_seq = 1). The two counters are independent.

4. Strategic explanations

The structural difference between the two windows

ROW_NUMBER() OVER (PARTITION BY line_id ORDER BY manufactured_at) resets the counter to 1 for every new line_id. ROW_NUMBER() OVER (ORDER BY manufactured_at) has no PARTITION BY, so the entire result set is one window — the counter climbs from 1 to N without ever resetting. Both share the same sort key, so the engine resolves the ordering once and applies both counters in a single pass.

ROW_NUMBER vs RANK vs DENSE_RANK in this context

ROW_NUMBER is the right choice here because every part is physically distinct — no two parts share the exact same manufactured_at timestamp in production. If two parts on the same line were stamped at the identical millisecond, ROW_NUMBER assigns them consecutive numbers (tie-breaking is deterministic but arbitrary). RANK would give them the same number and skip the next one; DENSE_RANK would give them the same number without skipping. For assembly-line serial numbering you almost always want ROW_NUMBER — every unit must have a unique position regardless of timestamp ties.

The no-partition global window — a common interview stumble

Candidates sometimes write PARTITION BY 1 or forget the clause entirely and accidentally add a PARTITION BY line_id. Omitting PARTITION BY deliberately is the correct way to say "one partition = the whole table." This is equivalent to a traditional SELECT ROW_NUMBER() without any grouping — useful for adding a surrogate sequence to a result set, paginating, or assigning global priority order.

Determinism: tie-breaking in timestamps

If two parts share manufactured_at, the ORDER BY is not a total order and the database may assign line_seq or global_seq values in any order between them on repeated executions. For true determinism add a tiebreaker: ORDER BY manufactured_at, part_id. In a production setting, using a timestamp column that could have duplicates without a tiebreaker is a latent non-determinism bug that surfaces silently under parallelism or query plan changes.

Performance note

Both windows share the same ORDER BY manufactured_at, though one also includes a PARTITION BY. Most engines (Snowflake, Postgres, BigQuery) detect the shared sort and compute both in one merge. An index on (line_id, manufactured_at) serves the partitioned window optimally; an index on (manufactured_at) alone serves the global window. On very large parts tables consider a composite index.

Dialect notes

ROW_NUMBER() OVER (...) is ANSI SQL and supported identically in Snowflake, Postgres, BigQuery, SQL Server, and Oracle. MySQL added it in v8.0. Snowflake also supports the QUALIFY clause for filtering on window function results without a subquery — e.g. QUALIFY line_seq = 1 returns only each line's first part, a concise pattern unavailable in Postgres or MySQL.
