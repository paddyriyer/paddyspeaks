Visa's people-analytics team runs an annual bonus cycle: an employee qualifies when their performance score is at least 90 and they have served at least two years. The rule is a one-line CASE with two conditions joined by AND. The wrinkle — and what makes the reference solution look intimidating — is that performance and tenure_yrs arrive as messy text, so the query scrubs them to numbers before comparing.

Each row probes one corner of the rule: Ann passes both; Ben fails on performance; Cy has a great score but only one year (fails tenure); Di clears both comfortably; Ed has a NULL performance — the edge case that decides how missing data is treated.

2. The SQL — CASE with two ANDed conditions

SELECT emp_id,
  CASE WHEN NULLIF(regexp_replace(performance::text, '[^-.0-9]', '', 'g'), '')::numeric >= 90
        AND NULLIF(regexp_replace(tenure_yrs::text, '[^-.0-9]', '', 'g'), '')::numeric >= 2
       THEN 'Eligible' ELSE 'Not Eligible' END AS bonus_status
FROM employees;

The regexp_replace(…, '[^-.0-9]', '', 'g') strips every character that is not a digit, minus sign, or dot — turning "95%" or " 3 yrs" into a clean numeric string. NULLIF(…, '') converts an emptied-out string back to NULL so the ::numeric cast does not blow up.

4. Strategic explanations

The AND short-circuit is the whole rule

Eligibility needs both conditions true. Cy is the instructive case: a strong score is worthless without tenure. Candidates sometimes write OR by reflex — that would qualify anyone with either a high score or long tenure, a far looser policy. Reading the requirement carefully and choosing AND is the first checkpoint. State the policy back: "both conditions, not either."

NULL in a boolean — why Ed is "Not Eligible", correctly

SQL booleans are three-valued: true, false, unknown. NULL >= 90 is unknown, and unknown AND anything is at best unknown — never true. So a NULL feeds the ELSE branch and Ed is "Not Eligible". Here that is the safe outcome (no data, no bonus), but it is happening by accident of NULL semantics. A senior answer makes it deliberate — e.g. a distinct flag so HR can chase the missing review rather than silently denying it:

CASE WHEN performance IS NULL OR tenure_yrs IS NULL THEN 'Incomplete Data'
     WHEN ... >= 90 AND ... >= 2 THEN 'Eligible'
     ELSE 'Not Eligible' END
Why scrub the text at all — and the NULLIF guard

If performance is stored as "95%" or tenure_yrs as "3 years", a direct ::numeric cast throws invalid input syntax and the whole query dies. regexp_replace keeps only [-.0-9], salvaging the number. The NULLIF(…, '') matters: a value like "N/A" scrubs down to the empty string, and ''::numeric is itself an error — NULLIF turns that empty string into NULL, which casts harmlessly. Skipping NULLIF is a real bug, not a style choice.

If the columns are already numeric, drop the machinery

The schema is ambiguous about the stored type. If performance and tenure_yrs are genuine numeric columns, all the regex/cast scaffolding is noise and the query is simply:

SELECT emp_id,
  CASE WHEN performance >= 90 AND tenure_yrs >= 2
       THEN 'Eligible' ELSE 'Not Eligible' END AS bonus_status
FROM employees;

The defensive version is the right call only when the data type is dirty or unknown. Naming that trade-off — robustness vs readability — is the senior move.

Boundary values — exactly 90, exactly 2

Both comparisons use >=, so an employee at precisely 90 or precisely 2 years qualifies. Confirm that is the intended policy — "is exactly 2 years in or out?" If the rule were "more than two years", it would be > 2 and a 2.0-year employee would fail. An off-by-one here quietly mis-pays people.

Senior signal: choose AND deliberately and read the policy back, explain that a NULL input makes the boolean unknown so it falls to ELSE (and consider flagging incomplete data instead), justify the regexp_replace/NULLIF guard as protection against a cast error rather than decoration, and pin down whether the >= boundaries are correct.
