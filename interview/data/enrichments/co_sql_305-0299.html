Salesforce's catalogue team is auditing how products are classified — not how many products exist, but how many distinct classification values are in use. It is a one-line query, and that is precisely why it is an interview question: COUNT(DISTINCT classification) hides a decision about NULLs, and a decision about whether "distinct values" and "the set of categories you intended" are actually the same thing.

1. The source data — products

product_idclassification
1Hardware
2Software

4Service

7Service

Seven products, but only three real classification labels — Hardware, Software, Service — each appearing twice. Product 5 has a NULL classification: it was never categorised. That NULL is the entire interview, because the answer depends on whether you count it.

2. The SQL — COUNT(DISTINCT classification)

SELECT COUNT(DISTINCT classification) AS distinct_classes FROM products;

DISTINCT inside COUNT de-duplicates the values first, then counts how many unique ones remain. No GROUP BY, so the result is a single number.

3. The resulting analysis

distinct_classes
3

The answer is 3 — Hardware, Software, Service. The four duplicate rows are folded away by DISTINCT, and the NULL is not counted. That is correct standard behaviour, but it is a behaviour you should be able to state out loud rather than be surprised by.

4. Strategic explanations

COUNT(DISTINCT col) ignores NULL — the one fact to say aloud

Every aggregate COUNT(column) skips NULLs, and COUNT(DISTINCT column) is no exception. Product 5's NULL contributes nothing, so the count is 3, not 4. If the audit question is really "how many distinct states does this column take, including uncategorised", you have to add the NULL back yourself:

SELECT COUNT(DISTINCT classification)
       + COUNT(*) FILTER (WHERE classification IS NULL AND FALSE) AS bad,
       COUNT(DISTINCT classification)
       + (CASE WHEN COUNT(*) FILTER (WHERE classification IS NULL) > 0
               THEN 1 ELSE 0 END) AS distinct_including_null
FROM products;

The clean idiom is the CASE form: add 1 if any NULL exists. Decide deliberately — "3 categories" and "3 categories plus an unclassified bucket" are different answers to different questions.

COUNT(DISTINCT) vs COUNT(*) vs COUNT(col)
expressionresultwhat it counts
COUNT(*)7all rows
COUNT(classification)6rows with a non-NULL value
COUNT(DISTINCT classification)3unique non-NULL values

Three different numbers from one column. An interviewer may ask all three at once to see whether you can articulate the difference crisply. The gap between 7 and 6 is the NULL row; the gap between 6 and 3 is the duplicates.

Surfacing the unclassified backlog

The audit's real follow-up is usually "how complete is our classification?" — which the bare count cannot answer. Report coverage alongside it:

SELECT COUNT(DISTINCT classification) AS distinct_classes,
       COUNT(*) FILTER (WHERE classification IS NULL) AS unclassified,
       ROUND(100.0 * COUNT(classification) / COUNT(*), 1) AS pct_classified
FROM products;

This turns a trivia number into something a data-quality owner can act on: the count of categories and the size of the uncategorised backlog.

Distinct values, but messy ones — the data-quality trap

COUNT(DISTINCT) compares values exactly and case-sensitively. If the column contains 'Hardware', 'hardware', and 'Hardware ' (trailing space), they count as three distinct classifications even though a human sees one. That inflates the count and is a real catalogue problem. A more honest audit normalises first:

SELECT COUNT(DISTINCT LOWER(TRIM(classification))) AS distinct_classes
FROM products;

Mentioning this shows you treat "distinct" as a data-quality question, not just a keyword.

Performance at scale

COUNT(DISTINCT) must de-duplicate, which means a hash or sort over the column — cheap here, but on a billion-row table it is heavier than a plain COUNT(*). If an approximate answer is acceptable for a dashboard, some engines offer APPROX_COUNT_DISTINCT (or HyperLogLog); Postgres has no built-in for it but the trade-off is worth naming. For an exact audit, COUNT(DISTINCT) is correct — just know it is not free.

Senior signal: state that COUNT(DISTINCT) drops NULLs and decide on purpose whether "uncategorised" deserves its own bucket, distinguish COUNT(*) / COUNT(col) / COUNT(DISTINCT col) in one breath, and treat case and whitespace variants as a real data-quality risk that can inflate the distinct count.
