Stripe's growth-accounting framework: every month, customers fall into 4 mutually-exclusive buckets — new / existing / churned / resurrected. The query uses LAG to compare current vs previous month MRR per customer.

1. The source data — monthly_revenue

2. The SQL — LAG-based state classification

WITH classified AS (
  SELECT customer_id, month, mrr,
    LAG(mrr) OVER (PARTITION BY customer_id ORDER BY month) AS prev_mrr
  FROM monthly_revenue
)
SELECT month,
  SUM(CASE WHEN prev_mrr IS NULL  AND mrr > 0          THEN 1 ELSE 0 END) AS new_customers,
  SUM(CASE WHEN prev_mrr > 0      AND mrr > 0          THEN 1 ELSE 0 END) AS existing,
  SUM(CASE WHEN prev_mrr > 0      AND (mrr = 0 OR mrr IS NULL) THEN 1 ELSE 0 END) AS churned,
  SUM(CASE WHEN prev_mrr = 0      AND mrr > 0          THEN 1 ELSE 0 END) AS resurrected
FROM classified
GROUP BY month
ORDER BY month;

3. Strategic explanations

The four mutually-exclusive states

Churnedprev_mrr > 0, mrr = 0lost
Resurrectedprev_mrr = 0, mrr > 0winback
The waterfall reconciliation

Each month's count: existing + new + resurrected = active_customers. churned goes the OTHER way. The CFO loves this because it reconciles the customer-base growth equation.

The expansion / contraction extension

Existing customers split: expansion (mrr grew), contraction (mrr shrank), flat. Real growth-accounting models all three:

SUM(CASE WHEN prev_mrr > 0 AND mrr > prev_mrr THEN 1 END) AS expansion
SUM(CASE WHEN prev_mrr > 0 AND mrr < prev_mrr AND mrr > 0 THEN 1 END) AS contraction
Net dollar retention from the waterfall
SUM(mrr WHERE existing OR resurrected OR expansion) /
SUM(prev_mrr WHERE existing OR churned OR contraction) = NDR

NDR > 100% = expansion outpaces churn (Snowflake-grade). NDR ≈ 100% = healthy. NDR < 90% = product-market-fit problem.

Resurrected — the leading indicator

Resurrected count growing over time = winback campaigns are working OR product is improving. Watch this trend; it predicts churn-rate inflection 3-6 months out.

The first-month edge

Month 1 of the data has no LAG → all customers look "new". Filter out the bootstrap row, OR pre-populate with a synthetic month-0 starting point.

Senior signal: name the 4-bucket waterfall (new/existing/churned/resurrected), expand existing into expansion/contraction/flat for full growth accounting, compute NDR from the waterfall, and watch resurrected as the leading indicator. The first-month bootstrap edge case is the over-the-bar guard.
