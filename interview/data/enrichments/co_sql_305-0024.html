Snap's fulfillment team gives every order a payment-due date seven days after it was placed. It is the simplest possible date arithmetic — add 7 days — but the interviewer is checking whether you reach for proper interval arithmetic instead of treating dates like integers, and whether you know what happens at month and year boundaries.

1. The source data — orders

order_idorder_datecustomer_idstatus
12024-01-25101completed
22024-02-27102completed

42024-02-29104completed

Three deliberate edge dates: Jan 25 (+7 crosses into February), Dec 28 (+7 crosses into the next year), and Feb 29 of a leap year. If the arithmetic is real date arithmetic, all three just work.

2. The SQL — order_date + INTERVAL '7 days'

SELECT order_id, order_date, (order_date + INTERVAL '+7 days') AS due_date FROM orders;

Adding an INTERVAL to a date shifts it by that span. The engine handles month lengths and leap years — you do not compute "what month is it" yourself.

3. The resulting analysis

Every boundary crossed correctly: Jan 25 + 7 rolls into February, Dec 28 + 7 rolls into 2025, and Feb 29 + 7 lands on Mar 7 — the engine knew 2024's February has 29 days. Note one subtlety: order_date is a DATE, but date + interval returns a timestamp, so due_date technically carries a (midnight) time component.

4. Strategic explanations

The '+7 days' string — harmless, but not idiomatic

The literal INTERVAL '+7 days' works, but the leading + is unusual; the conventional spelling is INTERVAL '7 days'. Cleaner still in Postgres is plain integer addition on a date:

SELECT order_id, order_date, order_date + 7 AS due_date FROM orders;

date + integer adds that many days and — unlike date + interval — returns a DATE, not a timestamp. So if you want due_date to stay a pure date, integer addition is actually the more precise choice. This is one of the few places a non-portable shortcut is also the better one.

The date-vs-timestamp return type — a real gotcha

order_date + INTERVAL '7 days' promotes the result to timestamp. Usually invisible, but it bites when you compare or join: due_date = some_date can behave unexpectedly if due_date ever carries a non-midnight time, and the column reports a different type to downstream tooling. If the schema expects a clean date, cast it: (order_date + INTERVAL '7 days')::DATE.

Dialect portability — this is the trap the question hides

Date arithmetic is the least portable corner of SQL. The same "+7 days" is spelled four different ways:

order_date + INTERVAL '7 days'                 -- Postgres
DATE_ADD(order_date, INTERVAL 7 DAY)           -- MySQL
DATEADD(day, 7, order_date)                    -- SQL Server
DATE_ADD(order_date, 7)  /  order_date + 7     -- BigQuery / others

State which engine you are targeting. Reaching for an interval (or integer addition) rather than something like order_date + 7 assuming a numeric date is the senior move — in many engines a date is not silently a number.

Edge cases: NULL dates and "business days"

A NULL order_date yields a NULL due_date — arithmetic propagates NULL — which is correct but worth a comment. The bigger lurking ambiguity: "7 days" almost always means 7 calendar days, but fulfillment SLAs are often quoted in business days. If order 1's Jan 25 (a Thursday) should skip the weekend, a flat +7 is wrong and you need a calendar table or a weekday-aware calculation. Confirm calendar-vs-business before writing the query.

Senior signal: use interval (or integer) date arithmetic instead of treating a date as a raw number, know that date + interval returns a timestamp while date + integer stays a DATE, name the per-dialect spelling differences, and surface the calendar-days-vs-business-days ambiguity before coding.
