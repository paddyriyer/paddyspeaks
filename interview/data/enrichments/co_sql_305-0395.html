Snapstreaks — consecutive days both users send a snap to each other — are Snap's signature engagement loop. Streak length predicts retention: 100+ day streaks have 95%+ retention. The query is gaps-and-islands on bidirectional pairs.

2. The SQL — per-pair gaps-and-islands

WITH islands AS (
  SELECT user_a, user_b, exchange_date,
    julianday(exchange_date) - ROW_NUMBER() OVER (PARTITION BY user_a, user_b ORDER BY exchange_date) AS island_id
  FROM snap_exchanges
)
SELECT user_a, user_b,
  MIN(exchange_date) AS streak_start,
  MAX(exchange_date) AS streak_end,
  COUNT(*) AS streak_days
FROM islands
GROUP BY user_a, user_b, island_id
ORDER BY streak_days DESC;

3. Strategic explanations

The mechanic — both sides, every 24h

Streak rule: BOTH users must exchange a snap within a 24-hour window. The data table assumes that's pre-filtered (one row per qualifying day). Gaps-and-islands picks up the streak structure.

The 100-day badge

Snap awards a "💯" emoji at 100 days. That visual reward drives streak preservation behavior — users send half-hearted "streak snaps" purely to maintain. Production has a quality filter on what counts.

The 24-hour grace

Snap allows a "streak save" — if you forget for 24h, you can pay or beg for a one-time grace. Production tracks streak-save activations as a separate metric.

Streak-loss intervention

When a 50+ day streak breaks, both users get notifications + an offer to "restart the streak". 30-40% accept. The query result feeds these alerts.

Bidirectional vs directional

The query uses (user_a, user_b) but doesn't enforce symmetry. Streaks ARE symmetric — should canonical-order: WHERE user_a < user_b to dedupe pairs.
