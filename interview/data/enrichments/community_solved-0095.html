A subscription business wants a per-customer profile: average subscription length, average revenue per subscription, and the longest dormant gap between one subscription ending and the next beginning — restricted to customers with at least two subscriptions. The reference solution uses LAG to reach the previous subscription's end date, but its HAVING filter is off by one: it demands more than two subscriptions instead of at least two.

2. The SQL

Original solution (flagged broken):

WITH t1 AS (
  SELECT *,
    (end_date::date - start_date::date) AS duration,
    LAG(end_date, 1, NULL) OVER (PARTITION BY customer_id ORDER BY end_date ASC) AS pre_subs_end_date
  FROM subscriptions
), t2 AS (
  SELECT *,
    (start_date::date - pre_subs_end_date::date) AS subscription_gap
  FROM t1
)
SELECT customer_id,
       AVG(duration) AS average_subscription_duration,
       AVG(amount_paid) AS average_amount_paid,
       MAX(subscription_gap) AS longest_gap_between_subscriptions
FROM t2
GROUP BY customer_id
HAVING COUNT(subscription_id) > 2
ORDER BY longest_gap_between_subscriptions DESC;

Requirement 4 says "at least two subscriptions." COUNT(subscription_id) > 2 means three or more, so customer 502 — who has exactly two — is silently dropped. The fix is a single character: >=.

Corrected solution:

WITH t1 AS (
  SELECT *,
    (end_date::date - start_date::date) AS duration,
    LAG(end_date, 1, NULL) OVER (PARTITION BY customer_id ORDER BY end_date ASC) AS pre_subs_end_date
  FROM subscriptions
), t2 AS (
  SELECT *,
    (start_date::date - pre_subs_end_date::date) AS subscription_gap
  FROM t1
)
SELECT customer_id,
       AVG(duration) AS average_subscription_duration,
       AVG(amount_paid) AS average_amount_paid,
       MAX(subscription_gap) AS longest_gap_between_subscriptions
FROM t2
GROUP BY customer_id
HAVING COUNT(subscription_id) >= 2
ORDER BY longest_gap_between_subscriptions DESC;

4. Strategic explanations

The off-by-one in HAVING

"At least two" is an inclusive lower bound: COUNT(*) >= 2. The strict > 2 means "at least three" and quietly excludes every two-subscription customer. Boundary phrasing — "at least", "more than", "no fewer than" — maps to >= vs >, and getting it wrong is a textbook silent bug. Always re-read the requirement and match the operator literally.

LAG reaches the previous subscription

LAG(end_date) OVER (PARTITION BY customer_id ORDER BY end_date) pulls each subscription's predecessor end date into the same row. The gap is then just start_date - pre_subs_end_date. The first subscription per customer has no predecessor, so LAG returns NULL and its gap is NULL — which MAX ignores, exactly the desired behaviour.

The ordering choice inside the window matters

The window orders by end_date. If subscriptions can overlap, ordering by start_date instead could give a different — possibly negative — gap. With clean non-overlapping data the two orderings agree; with messy data, pick the one the business defines as "consecutive" and say so. A negative gap is a signal of overlapping subscriptions worth surfacing rather than hiding.

AVG ignores NULLs — know what that means here

AVG(amount_paid) skips NULL amounts entirely, dividing by the count of non-NULL rows. If a NULL amount should count as a zero-revenue subscription, use AVG(COALESCE(amount_paid, 0)). Same for duration if either date can be NULL. State the assumption explicitly.

Date subtraction is a dialect minefield

In Postgres date - date yields an integer day count, so end_date::date - start_date::date works directly. MySQL needs DATEDIFF(end_date, start_date); SQL Server DATEDIFF(day, start_date, end_date); BigQuery DATE_DIFF(end_date, start_date, DAY). Translate per engine.

Senior signal: catch that "at least two" requires HAVING COUNT(*) >= 2 not > 2, explain that LAG ordered by end_date yields a NULL (correctly ignored) gap for each customer's first subscription, and flag that overlapping subscriptions can produce negative gaps that deserve surfacing rather than silent inclusion.
