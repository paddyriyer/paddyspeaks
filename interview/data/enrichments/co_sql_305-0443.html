Salesforce opportunities move through ranked stages (Qualified → Proposal → Negotiation → Closed Won), but real deals don't go forward only. A deal might fall back from Negotiation to Proposal when the prospect requests a redo. The query produces one row per stage period AND labels the direction (forward / backward / lateral / initial) using the stage_rank column.

2. The SQL — LAG of BOTH stage and rank, then direction CASE

WITH changes AS (
  SELECT opp_id, entered_at, stage, stage_rank,
    LAG(stage)      OVER (PARTITION BY opp_id ORDER BY entered_at) AS prev_stage,
    LAG(stage_rank) OVER (PARTITION BY opp_id ORDER BY entered_at) AS prev_rank
  FROM opp_stage_history
),
filtered AS (
  SELECT opp_id, entered_at, stage, stage_rank, prev_rank
  FROM changes
  WHERE stage != prev_stage OR prev_stage IS NULL
)
SELECT opp_id, stage,
  entered_at AS period_start,
  LEAD(entered_at) OVER (PARTITION BY opp_id ORDER BY entered_at) AS period_end,
  CASE
    WHEN prev_rank IS NULL                        THEN 'initial'
    WHEN stage_rank > prev_rank                   THEN 'forward'
    WHEN stage_rank < prev_rank                   THEN 'backward'
    ELSE 'lateral'
  END AS direction
FROM filtered
ORDER BY opp_id, period_start;

3. Strategic explanations

Two LAGs in one query

The query needs to know both the previous stage (for change detection) and the previous rank (for direction labelling). Two parallel LAG calls share the same window — the planner computes them together in one sort, no double work.

Why "lateral" is its own bucket

Some Salesforce orgs add custom stages at the same rank (e.g., "Proposal — Technical Review" and "Proposal — Pricing Review" both at rank 2). A transition between them is neither forward nor backward; it's lateral. Distinguishing the three categories matters for funnel-velocity analysis: "how often do deals go backward?" is a different question from "how often do they shuffle laterally?"

The backward-rate KPI

Backward-transition rate is the leading indicator of pipeline-health problems. Healthy pipeline: 5-10% of stage transitions are backward. Above 20% means something is broken — sales process unclear, qualification weak, deals re-opened post-loss.

SELECT
  100.0 * SUM(CASE WHEN direction = 'backward' THEN 1 ELSE 0 END) / COUNT(*) AS backward_rate_pct
FROM (/* this query */)
WHERE direction != 'initial';
Time-in-stage with direction

Two-month negotiation period that ENDS in backward = the negotiation collapsed. Two-month negotiation that ENDS in forward = the deal advanced normally. Same duration, opposite outcomes. Slice:

The "skipped stage" detection

The query catches stage skips implicitly: a transition from rank 1 to rank 4 still counts as "forward" but skips ranks 2 and 3. Production tracks this with an additional flag:

CASE WHEN stage_rank - prev_rank > 1 THEN 'skip' ELSE 'normal' END AS skip_flag

Skipped stages indicate bad data hygiene — usually rep entered the wrong stage and corrected it later.

Stage-rank as a dim, not a fact column

The stage_rank ideally lives in dim_stage, not denormalized on every opportunity row. Salesforce orgs change the stage list quarterly; if rank lives in the fact, every change requires backfill. SCD2 dim with effective_from/_to lets historical periods retain their then-correct rank.

Senior signal: two parallel LAGs (stage + rank) in one window, four-way direction classification (initial / forward / backward / lateral) for funnel-velocity analysis, backward-rate > 20% as a pipeline-health alarm, and SCD2 dim_stage to keep rank historically correct under stage-list churn.
