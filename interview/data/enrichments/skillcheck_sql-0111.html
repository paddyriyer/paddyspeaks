A payments dashboard wants the single highest-value transaction for each calendar day — a "daily top transaction" widget. The interesting part is the tie-break: when two transactions on the same day share the largest amount, the earlier timestamp wins, then the smaller id. The interviewer is checking whether you express a multi-key tie-break correctly inside a ROW_NUMBER window, rather than producing two rows for a tied day.

1. The source data — transactions

Day 1 has the headline edge case: two transactions tied at the max amount 250. Day 2 repeats it at a lower amount (both 75). Day 3 is the clean single-row day. A naive MAX(amount)-join would emit both 250 rows for day 1 — the tie-break is what makes the answer one row per day.

2. The SQL — ROW_NUMBER with a three-key ordering

WITH ranked AS (
  SELECT transaction_id, ts::date AS day, amount, ts,
         ROW_NUMBER() OVER (PARTITION BY ts::date ORDER BY amount DESC, ts, transaction_id) AS rn
  FROM transactions
)
SELECT day, transaction_id, amount
FROM ranked
WHERE rn = 1
ORDER BY day;

PARTITION BY ts::date groups rows by calendar day. The ORDER BY inside the window ranks within each day: amount DESC puts the largest first; ts ascending breaks amount-ties toward the earlier transaction; transaction_id ascending breaks any residual tie. rn = 1 keeps exactly the winner per day.

Exactly one row per day. For day 1 the tie at 250 is resolved by timestamp — transaction 2 at 11:30 beats transaction 3 at 14:00. The transaction_id key never had to fire here because the timestamps already differ, but it is the correct final guarantee of determinism.

4. Strategic explanations

Why ROW_NUMBER, not RANK, for "exactly one per group"

The requirement is one row per day. RANK() OVER (... ORDER BY amount DESC) would give both 250-transactions rank = 1, and WHERE rank = 1 would return both — defeating the purpose. ROW_NUMBER always produces a strict 1, 2, 3... sequence, so rn = 1 is guaranteed to match precisely one row. Top-1-per-group is the canonical ROW_NUMBER use case.

The tie-break keys must all live in the window's ORDER BY

A tie-break is just additional sort keys. ORDER BY amount DESC, ts, transaction_id reads as "largest amount; if tied, earliest; if still tied, lowest id." The order of keys is the priority order. Crucially, adding transaction_id last makes the ranking fully deterministic: with a unique id as the final key, no two rows can ever share a ROW_NUMBER, so the query returns the same answer on every run regardless of physical row order.

Why determinism matters — the silent non-bug

Without the transaction_id tie-break, two rows identical in both amount and ts would get rn = 1 and rn = 2 in an arbitrary order — the query "works" but returns different rows across executions, a flaky result that passes a demo and fails in production. A senior always terminates a ROW_NUMBER ordering with a unique column. Naming this unprompted is the signal.

ts::date — truncating timestamp to calendar day

Casting a timestamp to date drops the time component, so all of 2025-05-01's rows share one partition key. Beware time zones: if ts were timestamptz, ::date truncates in the session time zone — a transaction at 2025-05-01 23:30 UTC could land on May 1 or May 2 depending on SET timezone. For an audited daily report, truncate in an explicit zone: (ts AT TIME ZONE 'UTC')::date.

Dialect notes

ts::date is Postgres shorthand; portable forms are CAST(ts AS DATE), or DATE(ts) in MySQL/SQLite, or CONVERT(date, ts) in SQL Server. ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...) is standard everywhere. An alternative is Postgres's DISTINCT ON (ts::date) ... ORDER BY ts::date, amount DESC, ts, transaction_id — terser, but Postgres-only; the ROW_NUMBER form ports unchanged.

Senior signal: use ROW_NUMBER (not RANK) for one-row-per-group, encode every tie-break as an extra key in the window's ORDER BY in priority order, always end that ordering with a unique column so the result is deterministic, and watch time-zone semantics when truncating a timestamp to a calendar day.
