A Community question on a Gmail activity table: rank senders by how many emails they sent, with no ties — every sender gets a unique rank. The data team wants a strict leaderboard. The subtle part is the phrase "unique rank": two senders with the same volume must still get different numbers, which rules out RANK and DENSE_RANK and points squarely at ROW_NUMBER — with a deterministic tiebreaker.

1. The source data — google_gmail_emails

The table is a single column — one row per email sent. alice sent 3, bob and carol 2 each, dave 1. bob and carol are the tie, and "unique rank" says they cannot share a number — that tie is the whole question.

2. The SQL — ROW_NUMBER over a grouped count

select from_user,
    sent_email,
    row_number() over(order by sent_email desc, from_user asc) as unique_rank
from (
select from_user,
  count(*) as sent_email
from google_gmail_emails
group by from_user
) a

The inner query collapses the table to one row per sender with a sent_email count. The outer ROW_NUMBER orders those rows by volume descending, breaking ties on from_user ascending, and hands out 1, 2, 3, … with no gaps and no shared values.

3. The resulting analysis

bob and carol both sent 2, yet bob is rank 2 and carol rank 3 — the from_user ASC tiebreaker put "bob" before "carol" alphabetically. That is exactly the "unique rank" the question asked for. With RANK() they would both be 2 (then a skip to 4); with DENSE_RANK() both 2 (then 3). Only ROW_NUMBER guarantees distinct ranks.

4. Strategic explanations

Why ROW_NUMBER, and why the tiebreaker is mandatory

ROW_NUMBER() assigns a strictly increasing integer regardless of ties — that is what "unique rank" demands. But ROW_NUMBER only decides tied rows by whatever the ORDER BY says; if the ORDER BY were sent_email DESC alone, bob and carol's 2-vs-3 split would be arbitrary and could change between runs or engines. Adding from_user ASC makes the result deterministic. A unique rank with a non-deterministic tiebreak is a bug waiting to happen — the secondary key is not optional.

RANK vs DENSE_RANK vs ROW_NUMBER — pick by tie semantics
sendersentROW_NUMBERRANKDENSE_RANK
alice3111
bob2222
carol2322
dave1443

ROW_NUMBER — all distinct, no ties shared. RANK — ties share, next value skips. DENSE_RANK — ties share, no skip. "Unique rank" ⇒ ROW_NUMBER; "competition placing" ⇒ RANK; "tier number" ⇒ DENSE_RANK. Stating this trio fluently is the senior tell.

Why the count must be a subquery (or CTE)

ROW_NUMBER() OVER (ORDER BY sent_email …) needs sent_email to already exist as a column. You cannot both COUNT(*) and rank that count in one SELECT level — the window function would see individual email rows, not per-sender totals. The inner GROUP BY produces the aggregate; the outer level ranks it. A CTE reads more cleanly than the inline subquery but is logically identical:

WITH counts AS (
  SELECT from_user, COUNT(*) AS sent_email
  FROM google_gmail_emails GROUP BY from_user
)
SELECT from_user, sent_email,
       ROW_NUMBER() OVER (ORDER BY sent_email DESC, from_user) AS unique_rank
FROM counts;
Edge cases: NULL senders and a missing ORDER BY

If from_user can be NULL, GROUP BY collects all NULL senders into one group that gets its own rank — probably meaningless; filter WHERE from_user IS NOT NULL or decide what a "NULL sender" means. And note the final result has an ORDER BY only inside the window — the row order of the output itself is technically undefined. Add a top-level ORDER BY unique_rank so the leaderboard prints in rank order.

The dialect footnote

Some warehouses (Snowflake, BigQuery) let you write QUALIFY ROW_NUMBER() OVER (…) <= 10 to grab the top 10 in one statement. Postgres has no QUALIFY — you wrap the window in a subquery/CTE and filter in the outer WHERE. Same result, different syntax; mention it if the role is warehouse-specific.

Senior signal: map "unique rank" to ROW_NUMBER rather than RANK/DENSE_RANK, insist on a deterministic tiebreaker (from_user) so tied senders rank reproducibly, explain why the count must be aggregated before it can be ranked, and add the top-level ORDER BY the leaderboard needs.
