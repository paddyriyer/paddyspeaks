Instacart's identity team gets a routine ticket: "the new fraud model is firing on accounts that share an email with a different user_id." First step is finding all such pairs — users with the same email address but different user_ids. This is the canonical self-join with an inequality predicate: same table aliased twice, joined on the duplicate field, with < on the id to surface each pair exactly once.

1. The source data — users

user_idemail
U_001alex@gmail.com
U_002bailey@yahoo.com
U_003alex@gmail.com
U_004casey@hotmail.com
U_005alex@gmail.com
U_006bailey@yahoo.com

"alex@gmail.com" appears 3 times, "bailey@yahoo.com" 2 times.

2. The SQL — self-join with < ordering

SELECT a.user_id AS u1, b.user_id AS u2, a.email
FROM users a
JOIN users b ON a.email = b.email AND a.user_id < b.user_id;

3. The resulting analysis

4 unique pairs: 3 from alex's three-way collision (C(3,2)=3), 1 from bailey's pair.

4. Strategic explanations

Why < not <> — deduping pairs

<> (not equal) gives both directions — (U_001, U_003) and (U_003, U_001). For 3 colliding ids you'd get 6 rows instead of 3. Strict < picks a canonical order and emits each pair exactly once. This is the same trick as the pizza-toppings combinations problem, applied to dedup detection.

The simpler alternative — GROUP BY ... HAVING COUNT > 1

If all you need is the list of duplicated emails (not the user-id pairs), GROUP BY is shorter:

SELECT email, COUNT(*) AS dup_count, ARRAY_AGG(user_id) AS user_ids
FROM users
GROUP BY email
HAVING COUNT(*) > 1;

This collapses each duplicated email into one row, which is more compact for the fraud-investigation dashboard. The self-join form is preferred when you want to emit one row per pair (good for graph-traversal: connecting each pair as an edge).

Email normalization — the silent miss

a.email = b.email is a strict string equality. Real-world duplicates fall into the gap:

  • Alex@gmail.com vs alex@gmail.com — case difference
  • alex@gmail.com vs alex@gmail.com — trailing whitespace
  • alex+sale@gmail.com vs alex@gmail.com — Gmail-style plus addressing (same inbox)
  • alex.smith@gmail.com vs alexsmith@gmail.com — Gmail's dots-don't-matter rule
-- Normalize before joining
ON LOWER(TRIM(a.email)) = LOWER(TRIM(b.email))
-- For Gmail-aware dedup
ON LOWER(REGEXP_REPLACE(SPLIT_PART(a.email,'@',1), '\\+.*$', ''))
 = LOWER(REGEXP_REPLACE(SPLIT_PART(b.email,'@',1), '\\+.*$', ''))

For an interview answer, mention the normalization layer up front — it's the difference between catching 60% of fraud collisions and 95%.

The "find fully-merged identity clusters" follow-up

Pairs are useful, clusters are better. "Show me all groups of users that share an email." Just GROUP BY email and aggregate the ids:

SELECT email, ARRAY_AGG(user_id ORDER BY user_id) AS cluster
FROM users
GROUP BY email
HAVING COUNT(*) > 1;
The transitive-cluster trap — A=B, B=C, A=?C

Real identity matching uses multiple keys: same email or same phone or same payment_token. A=B by email, B=C by phone, A=C indirectly. Pure self-join only catches direct ties — the senior follow-up is to mention graph-traversal via recursive CTE or external tools (DBT-Identity, Splink, even pandas' connected components).

WITH RECURSIVE edges AS (
  SELECT a.user_id AS u1, b.user_id AS u2 FROM users a JOIN users b
   ON LOWER(a.email)=LOWER(b.email) AND a.user_id<>b.user_id
  UNION
  SELECT a.user_id, b.user_id FROM users a JOIN users b
   ON a.phone=b.phone AND a.user_id<>b.user_id
),
cluster AS (
  SELECT u1, u2 FROM edges
  UNION
  SELECT c.u1, e.u2 FROM cluster c JOIN edges e ON c.u2 = e.u1
)
SELECT u1, MIN(u2) AS canonical_id FROM cluster GROUP BY u1;
The unique-constraint follow-up — prevention > detection

If duplicate emails are a bug not a feature, the right fix is upstream: UNIQUE INDEX users(LOWER(email)) at create-time. Self-join queries are for the historical mess; the constraint stops the next one. Always close with "and we should add the constraint going forward."

Senior signal: name the strict-< dedup mechanism, surface email normalization (case, whitespace, plus-addressing, Gmail dots) as a first-class concern, contrast pair-emit (self-join) with cluster-emit (GROUP BY), and pre-empt the transitive-cluster problem with a recursive-CTE sketch. Closing on the unique-index prevention is the over-the-bar finish.
