A growth team wants a classic cohort retention grid: group users by the week they signed up, then for each week-offset 0–3 count how many were active in that 7-day window. The interviewer is probing whether you can bucket users into cohorts, compute a relative week offset from raw dates, and count distinct retained users without double-counting multiple activity rows.

Users 1, 2, 3 all land in the 2025-01-06 cohort even though they signed up on different days — date_trunc('week', ...) snaps each to the Monday. Edge case: user 2 has two activity rows in week 0 (Jan 8 and Jan 9); the count must treat that as one retained user, not two.

2. The SQL — truncate to cohort week, compute offset, count distinct

WITH cohorts AS (
  SELECT user_id,
         date_trunc('week', signup_date)::date AS cohort_week
  FROM users
),
joined AS (
  SELECT c.cohort_week,
         c.user_id,
         FLOOR((a.active_date - c.cohort_week)::numeric / 7)::int AS week_offset
  FROM cohorts c
  JOIN activity a USING (user_id)
)
SELECT cohort_week,
       week_offset,
       COUNT(DISTINCT user_id) AS retained_users
FROM joined
WHERE week_offset BETWEEN 0 AND 3
GROUP BY cohort_week, week_offset
ORDER BY cohort_week, week_offset;

Three steps. cohorts assigns each user a Monday-anchored cohort. joined attaches every activity row and converts the day-gap into a week index via integer division by 7. The final COUNT(DISTINCT user_id) collapses a user's multiple activity rows in the same offset to a single retained count.

Week 0 of the first cohort is 3, not 4: user 2's two activity rows collapse under COUNT(DISTINCT). Offset 3 of the 2025-01-13 cohort produces no row at all — nobody was retained, and a GROUP BY emits no row for an empty group, satisfying "skip 0-counts".

4. Strategic explanations

Why COUNT(DISTINCT user_id) is mandatory

The activity table has one row per active day, so a user active twice in the same week appears twice after the join. A plain COUNT(*) would report 4 for the first cohort's week 0 — counting visits, not users. Retention is a headcount metric: COUNT(DISTINCT user_id) answers "how many people", which is the question.

Why date_trunc('week') anchors the cohort

Users who signed up Mon–Sun of the same week belong to one cohort. date_trunc('week', d) maps any date to the Monday of its ISO week, giving a stable cohort key. Anchoring to a fixed weekday is essential — computing offsets from each user's individual signup day instead would make "week 1" mean a different calendar window per user, and the grid would no longer be comparable across the cohort.

The offset formula — FLOOR(day_gap / 7)

active_date - cohort_week yields an integer day count; dividing by 7 and flooring gives the week index. The ::numeric cast matters: integer division 13 / 7 would truncate to 1 directly, but casting to numeric first makes the intent (real division then explicit FLOOR) unambiguous and avoids surprises if the expression is edited. A user active on the cohort Monday is day-gap 0 → offset 0; day 13 → offset 1.

Why empty offsets disappear — and when that is wrong

GROUP BY only emits groups that have rows. Offset 3 of the second cohort has zero retained users, so there is no row — exactly what "skip 0-counts" asks for. But a true retention chart usually wants explicit zeros so the curve is continuous. To get them, CROSS JOIN the distinct cohorts to a generate_series(0,3) of offsets and LEFT JOIN the counts, coalescing NULL to 0. Know both shapes and which the consumer needs.

Dialect notes

date_trunc and date subtraction returning an integer are PostgreSQL/DuckDB. In BigQuery use DATE_TRUNC(signup_date, WEEK(MONDAY)) and DATE_DIFF(active_date, cohort_week, DAY); in Snowflake DATE_TRUNC('week', ...) with the WEEK_START session parameter controlling the anchor day; in MySQL there is no date_trunc, so compute the Monday with signup_date - INTERVAL WEEKDAY(signup_date) DAY. The cohort logic is identical — only the date functions change.

Senior signal: build cohort retention by anchoring signups to a fixed weekday with date_trunc, deriving a relative week_offset via FLOOR(day_gap / 7), and counting people with COUNT(DISTINCT user_id) so multiple activity rows never inflate the headcount; know that GROUP BY drops empty offsets and how to back-fill explicit zeros for a chart.
