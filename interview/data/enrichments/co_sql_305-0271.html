Visa's payments-analytics team wants to study how customers re-order — for every customer, pair each order with the orders that came after it and measure the gap in days. That is a self-join: the orders table joined to itself. The interview lives in the join condition, because get it slightly wrong and you produce every pair twice, or pair an order with itself.

Customer 100 has three orders — that yields three ordered pairs (1-2, 1-3, 2-3), the case that exposes whether your join condition double-counts. Customer 200 has two orders, one pair. The join must never cross customers: order 1 and order 4 belong to different people.

2. The SQL — self-join on order_id < order_id

SELECT a.order_id, b.order_id, b.order_date - a.order_date AS gap
FROM orders a
JOIN orders b
  ON a.customer_id = b.customer_id
 AND a.order_id < b.order_id;

Two aliases of the same table. a.customer_id = b.customer_id keeps pairs within one customer; a.order_id < b.order_id is the clever half — it pairs each order only with later-ID orders, so each pair appears once and no order joins itself.

Four pairs — three for customer 100, one for customer 200, none crossing customers. The gaps are integers because date - date yields whole days. Note both output columns are literally named order_id — the query never aliased them. That is legal in Postgres but a problem for any consumer that addresses columns by name; see below.

4. Strategic explanations

Why < and not <> — the double-count trap

Using a.order_id <> b.order_id (just "not the same row") returns both orderings of every pair: (1,2) and (2,1), (1,3) and (3,1) — six rows for customer 100 instead of three, with half the gaps negative. The strict < imposes a direction: a is always the earlier order. It does two jobs at once — eliminates the self-pair (an order is never < itself) and picks one ordering of each pair. This is the single detail the interviewer is checking.

Order by order_id or by order_date?

a.order_id < b.order_id assumes ID order matches chronological order. If IDs are assigned out of sequence (parallel inserts, back-dated orders), "earlier ID" is not "earlier date" and gap could be negative. The robust condition pairs on the date itself:

JOIN orders b
  ON a.customer_id = b.customer_id
 AND a.order_date < b.order_date

Mention the assumption. If two orders share a date you may then need order_id as a tiebreaker to keep each pair once.

The duplicate column name — alias your output

Both selected columns are order_id. A driver returning rows as a dictionary keeps only the last one; downstream code silently loses the earlier order. Always alias:

SELECT a.order_id AS earlier_order,
       b.order_id AS later_order,
       b.order_date - a.order_date AS gap_days
FROM orders a JOIN orders b
  ON a.customer_id = b.customer_id AND a.order_id < b.order_id;

The original solution is correct, but un-aliased duplicate names are a production hazard worth calling out.

"Gap to the next order" — the better question

This self-join produces all later pairs, so order 1 pairs with both 2 and 3. Usually the business wants only consecutive orders — the gap to the immediately next one. A self-join can do it, but LEAD is far cleaner and O(n log n) instead of O(n²):

SELECT order_id, customer_id,
       LEAD(order_date) OVER (PARTITION BY customer_id ORDER BY order_date)
         - order_date AS days_to_next
FROM orders;

The self-join is the right tool for all pairs (e.g. "any two orders within 7 days"); LEAD/LAG is right for adjacent pairs. Knowing which the question wants is the senior move.

Single-order customers vanish

A customer with one order produces no pair — an inner self-join needs two rows to match. If the report must list every customer (showing NULL for those who never re-ordered), that calls for a LEFT JOIN or the LEAD form, where a lone order simply gets a NULL days_to_next.

Senior signal: use the strict a.order_id < b.order_id to kill both the self-pair and the mirror-image duplicate in one stroke, question whether ID order equals date order, alias the two order_id outputs, and reach for LEAD when the question is really about consecutive orders rather than all pairs.
