A telecom analyst wants the chatty countries: countries whose average call duration — counting every call where the country is on either end — beats the global average across all call legs. The interviewer is probing two skills at once: deriving the country from a phone-number prefix via string parsing, and a HAVING filter that compares each group against a scalar global benchmark.

1. The source data — country, person, calls

relationrow(narrative)

calls(11→30, 60), (20→30, 10)USA↔India 60s, China↔India 10s

The edge case is the cross-border call. A call between a USA caller and a China callee must count once for USA and once for China — that is the "either caller or callee" rule. Splitting each call into two legs makes both endpoints countable; India ends up with only short calls, which tests a group that falls below the benchmark.

2. The SQL — explode calls into legs, tag with country, compare to global average

WITH legs AS (
  SELECT caller_id AS pid, duration FROM calls
  UNION ALL
  SELECT callee_id, duration FROM calls
),
tagged AS (
  SELECT c.country_name, l.duration
  FROM legs l
  JOIN person  p ON p.person_id = l.pid
  JOIN country c ON split_part(p.phone_number, '-', 1) = c.country_code
)
SELECT country_name
FROM tagged
GROUP BY country_name
HAVING AVG(duration) > (SELECT AVG(duration) FROM legs)
ORDER BY country_name;

legs turns each call into two rows — the caller's leg and the callee's leg. tagged attaches a country by parsing the prefix out of the phone number. The HAVING keeps countries whose per-country average exceeds the scalar global average over the same leg set.

The global average over all 8 legs is 100. China (155) and USA (105) strictly exceed it. India is correctly excluded: its legs (a 60s and two 10s calls) average 35 — well below the benchmark, so the HAVING rejects it.

4. Strategic explanations

Why explode calls into legs with UNION ALL

"Calls touching that country, as caller or callee" means each call must be attributable to up to two countries. Keeping calls one-row-each and trying to OR the two sides leads to double-handling bugs. Splitting into a caller leg and a callee leg — UNION ALL — makes every endpoint a first-class row, so a cross-border call cleanly contributes its duration to both countries' averages. UNION ALL not UNION: a call's two legs share the same duration, and UNION would dedupe them, destroying the count.

The benchmark must use the same leg universe

HAVING AVG(duration) > (SELECT AVG(duration) FROM legs) — the global benchmark averages over legs, the same doubled row set the per-country averages use. This is deliberate and the question says so ("global average across all call legs"). If you instead benchmarked against SELECT AVG(duration) FROM calls (one row per call) the two averages would actually coincide here, but in general they differ — comparing a per-leg average to a per-call average is an apples-to-oranges bug. Keep numerator and benchmark on the same grain.

split_part — parsing the country code, and its fragility

split_part(phone_number, '-', 1) takes the text before the first '-' — the dialling prefix — and joins it to country.country_code. It works because the data format is strictly code-rest. It is fragile: a number with no '-' yields the whole string, an unknown prefix joins to nothing (that leg silently drops from tagged), and a leading '+' would break the match. A senior flags that prefix-to-country is genuinely ambiguous in reality (variable-length codes) and that a proper schema would store country_id on person directly.

Dialect — split_part is not universal

split_part is PostgreSQL (and Redshift). MySQL has SUBSTRING_INDEX(phone_number,'-',1); SQL Server has no direct equivalent and needs LEFT(phone_number, CHARINDEX('-',phone_number)-1); SQLite needs substr(phone_number,1,instr(phone_number,'-')-1). The leg-explode and HAVING-vs-subquery logic is fully portable — only the string-splitting function changes. Knowing the per-dialect substitution, and that the rest of the query is unaffected, is the senior signal.

Senior signal: explode each call into caller and callee legs with UNION ALL so a cross-border call counts for both countries, benchmark the HAVING against the global average computed over the same leg grain, and treat the split_part prefix parse as fragile — naming the dialect variants and the cleaner "store country_id directly" schema.
