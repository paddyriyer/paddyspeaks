Deutsche Bank's digital-channels team reports an engagement rate for every piece of published content — engagements as a percentage of views. The catch is in the schema: engagements is not a clean integer. It arrived from an upstream feed as text, sometimes with units or labels glued on ("60 likes"). An interviewer posing this is testing whether you can sanitise a dirty numeric column and still guard the rate against zero-view content.

1. The source data — content

Four edge cases. Content 2 has the unit suffix "60 likes" that must be stripped. Content 3 has zero views — the divide-by-zero. Content 4's engagements are NULL. Content 5's engagements are an empty string, which is not the same as NULL and would explode a naive cast.

2. The SQL — regex-clean then divide

SELECT content_id,
  100.0 * NULLIF(regexp_replace(engagements::text, '[^-.0-9]', '', 'g'), '')::numeric / NULLIF(views, 0) AS engagement_rate_pct
FROM content;

regexp_replace strips every character that is not a digit, minus, or dot — turning "60 likes" into "60". The inner NULLIF(…, '') turns an empty result into NULL so the ::numeric cast never sees a blank. NULLIF(views, 0) guards the denominator.

The three NULL rows each fail for a different reason — content 3 from zero views, content 4 from NULL engagements, content 5 from the empty-string-to-NULL guard. None of them errors the query.

4. Strategic explanations

Reading the regex: a character-class delete

regexp_replace(text, '[^-.0-9]', '', 'g') means "globally replace every character not in the set {-, ., digits} with nothing". The ^ inside the brackets negates the class; the 'g' flag makes it replace all matches, not just the first. So "60 likes" → "60", "$1,200" → "1200". It is a blunt instrument — useful, but see the next two traps.

Why the inner NULLIF(…, '') is essential

After stripping, content 5's empty string stays empty, and content 2's text could in principle reduce to "" if it held no digits at all. Casting ''::numeric raises invalid input syntax for type numeric: "" — which aborts the whole query. NULLIF(stripped, '') converts the blank to NULL before the cast; NULL::numeric is harmlessly NULL. This guard is not optional defensive padding; without it the query dies on row 5.

The regex is too permissive — the subtle bug

Keeping all digits, dots, and minuses can produce garbage. "12.3.4" survives as "12.3.4" and still fails the cast. "likes: 5, shares: 7" collapses to "57" — a silently wrong number. The character-delete approach assumes the field holds exactly one number with cosmetic noise. If the feed is messier, extract the first numeric token instead:

SELECT content_id,
  100.0 * NULLIF((regexp_match(engagements, '[0-9]+(\.[0-9]+)?'))[1], '')::numeric
        / NULLIF(views, 0) AS engagement_rate_pct
FROM content;

regexp_match pulls the first well-formed number and ignores the rest — far safer than blindly concatenating every digit.

Zero views — NULL, not zero

Content 3 has views = 0. NULLIF(views, 0) makes the denominator NULL, and the rate is NULL. That is defensible — "engagement rate of content nobody saw" is genuinely undefined. But if the dashboard must show a number, decide: COALESCE(…, 0) for "0%", or filter zero-view content out of the report entirely. NULL is the honest default; 0% is a presentation choice.

The real fix is upstream

A senior would name it: engagements should be a numeric column, with the cleaning done once in the ingestion pipeline, not re-derived on every dashboard query. Regex-scrubbing in the reporting layer is fragile and slow at scale. The query is a workaround; the data contract is the bug.

Senior signal: sanitise the dirty text column with a regex but acknowledge its limits (multi-number garbage), keep the empty-string-to-NULL guard that stops the cast from aborting the whole query, treat zero views as NULL or 0% on purpose, and call out that the cleaning belongs in ingestion, not in every read.
