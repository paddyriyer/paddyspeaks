A Community e-commerce dataset wants the share of orders that are actually shippable — meaning they carry a real delivery address. It is a simple ratio: shippable orders over total orders, as a percentage. The interviewer is checking whether you get the numerator and denominator scoped correctly, and whether you notice that "blank address" is sneakier than "NULL address".

1. The source data — orders

Five orders. Orders 3 and 5 have a blank-space address — not NULL, an actual ' ' string. That distinction is the whole trap: a query checking address IS NOT NULL would count these as valid.

2. The SQL — filtered count over total count

SELECT CAST((1.0*COUNT(*)/(SELECT COUNT(*) FROM orders))*100 AS INT) AS shipable_order_percent
FROM orders o
LEFT JOIN customer c ON o.cust_id = c.id
WHERE o.address <> ' ';

The outer COUNT(*) counts orders that survive WHERE address <> ' ' — the shippable ones. The scalar subquery (SELECT COUNT(*) FROM orders) counts all orders, unfiltered. Their ratio × 100 is the percentage.

3. The resulting analysis

shipable_order_percent
60

Three of five orders have a non-blank address, so 3 / 5 = 0.6 → 60. The two blank-address orders are excluded from the numerator but still counted in the denominator — which is correct: they are orders, they just are not shippable.

4. Strategic explanations

Why the denominator must be a separate subquery

The WHERE clause filters the rows feeding the outer COUNT(*), so that count is already only the shippable orders. To get the total, you need a count that the WHERE does not touch — hence the independent (SELECT COUNT(*) FROM orders). A common bug is using COUNT(*) for both: numerator and denominator become identical and the answer is always 100%.

The cleaner one-pass alternative — conditional aggregation

Scanning the table twice is avoidable. Compute both counts in a single pass with a filtered aggregate:

SELECT CAST(
         100.0 * COUNT(*) FILTER (WHERE address <> ' ')
         / COUNT(*)
       AS INT) AS shipable_order_percent
FROM orders;

COUNT(*) FILTER (...) is the numerator, plain COUNT(*) the denominator — one scan, no correlated subquery, and the two counts are guaranteed to agree on what "all orders" means.

The trap: <> ' ' does not catch NULL or empty string

address <> ' ' excludes only the literal single-space string. It does not exclude a genuine empty string '', and crucially a NULL address makes NULL <> ' ' evaluate to unknown — so a NULL-address order is silently dropped from the numerator too, which may or may not be intended. A robust "has a real address" test handles all three:

WHERE address IS NOT NULL AND TRIM(address) <> ''

TRIM collapses any amount of whitespace to empty, so one space, three spaces, or tabs all count as "no address".

The unused join — recognise dead code

LEFT JOIN customer c contributes nothing: no customer column appears in the SELECT or WHERE. Because it is a left join it does not even change the row count, so the answer is unaffected — but it is wasted work and noise. If the real definition of shippable also required, say, the customer to exist or be in a serviced country, the join would matter; as written, drop it. Spotting and removing dead joins is a senior habit.

Integer division and the rounding choice

1.0 * promotes the division to numeric so the fraction is not truncated to 0 before the × 100. The final CAST(... AS INT) then truncates toward zero — 59.9% would report as 59, not 60. If the business wants nearest-integer, use ROUND(...) instead of an INT cast; if it wants one decimal, cast to NUMERIC(5,1). State which rounding rule you applied.

Senior signal: keep the denominator independent of the WHERE filter (or better, use one-pass COUNT(*) FILTER), recognise that <> ' ' misses NULL and empty strings and replace it with a TRIM-based test, prune the dead LEFT JOIN, and state whether the percentage truncates or rounds.
