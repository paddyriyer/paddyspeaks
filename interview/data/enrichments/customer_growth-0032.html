The single dashboard tile every consumer-platform exec asks for: "what % of our users are whales? what % are minnows? and how skewed are we overall?". The L5 answer surfaces all three named cohorts plus the skew ratio in one query. The vocabulary signals seniority more than the SQL itself.

For the full vocabulary (whales / dolphins / minnows · deciles / quartiles / percentiles · Pareto / long-tail / normal / skewed · mean vs median · LAD gap), see the Skew & Distributions design page.

1. The four archetypes — boundaries and intent

ArchetypeBoundaryWhat they need from the platform
🐋 WhaleTop 5% (NTILE(20) = 1)Reliability + edge cache; never let them have a bad experience
🐬 Heavy dolphinDeciles 2-3 (top 6-30%)Feature breadth — they're whales-in-the-making
🐬 DolphinDeciles 4-8 (mid 30-80%)Stability + content discovery; the platform backbone
🐟 MinnowBottom 20% (deciles 9-10)Onboarding / re-activation; move them up to dolphin

2. The L5 vocabulary moments inside the query

  • NTILE(20) for the whale boundary, NTILE(10) for the rest. Whales sit at the top 5% — that's NTILE(20) bucket 1. The other bands work fine at decile (NTILE(10)) granularity. Mixing two NTILE granularities in one CASE chain is the senior-IC pattern.
  • PERCENTILE_CONT(0.5) and PERCENTILE_CONT(0.99), not AVG and STDDEV. Engagement data is positively skewed — STDDEV-based outlier detection silently mis-classifies half the dolphins. PERCENTILE_CONT is the safe default.
  • skew_ratio = AVG / MEDIAN. One number that quantifies "how Pareto-shaped is the platform". Uniform ≈ 1.0; healthy long-tail = 2-4; extreme power-law (one viral hit) > 10.
  • p99_over_p50. The interview-friendly variant of skew_ratio. "Our top 1% watches K times more than the median user" — instantly understandable on a board deck.
  • is_metadata_complete = TRUE. The LAD guard. A whale watching a freshly-released show whose metadata is still propagating would have artificially low hours and get stamped as a minnow without this filter. Surface the filter explicitly in any interview answer.

3. The interviewer's edge-case probes

Probe 1 — "Why NTILE(20) for whales but NTILE(10) for everyone else?"

The whale cohort is conventionally "top 5%", which doesn't fall on a decile boundary. NTILE(20) bucket 1 = top 5%; bucket 2 = next 5%. Splitting the analysis at decile-level for the lower bands keeps the output legible — nobody wants a 20-row table when 4 cohorts is the operational decision space. The honest framing: granularity should match the decision being made, not the symmetry of the math.

Probe 2 — "What if your boundaries leave gaps? NTILE(20)=1 is 5% but NTILE(10) buckets 2-3 is 20%, leaving 5% unaccounted for."

Good catch — and it's deliberate. With NTILE(20)=1 grabbing the top 5%, NTILE(10)=1 (top 10%) overlaps the whale region. The CASE chain runs top-down — whales are claimed first by NTILE(20)=1; heavy_dolphin then claims NTILE(10) buckets 2-3 which together cover deciles 11-30% — the 5% gap (deciles 6-10% by NTILE(20)) overlaps with the whale boundary and is already claimed. The defensive rewrite uses a single ranking function and explicit bucket-number ranges so the boundary math is obvious:

-- More explicit alternative: PERCENT_RANK with discrete thresholds
SELECT
  user_id, hours,
  CASE
    WHEN pct <= 0.05 THEN 'whale'
    WHEN pct <= 0.30 THEN 'heavy_dolphin'
    WHEN pct <= 0.80 THEN 'dolphin'
    ELSE                  'minnow'
  END AS archetype
FROM (
  SELECT user_id, hours,
    PERCENT_RANK() OVER (ORDER BY hours DESC) AS pct
  FROM user_metrics
) p;
Probe 3 — "How does the platform health change if skew_ratio drifts from 2.5 to 4.0 over a quarter?"

Skew ratio drifting UP means the platform is concentrating — the whales are becoming more dominant relative to the median. Two interpretations:

  • Bull case: the platform's stickiness for high-end users is improving — they're watching more.
  • Bear case: the median is dropping — dolphins are shrinking toward minnow territory, churn risk is rising. The aggregate "hours per user" headline looks healthy because whales mask the dolphin slowdown.

The senior answer: always pair skew_ratio with median trend. If both go up, healthy. If skew_ratio up + median down, intervention needed.

Probe 4 — "At 250M users (Netflix scale), does this query work?"

PERCENTILE_CONT requires a full sort — at 250M rows that's hours. Production rewrites:

  • APPROX_PERCENTILE (Snowflake / BigQuery / Spark / Trino) — sketch-based, sub-second on billions of rows, ~1% error. Safe for dashboards.
  • Pre-computed daily histogram — bucket users into 100 hour-bins overnight; the dashboard queries the 100-row histogram, not the 250M-row table. The classification job becomes O(1).
  • Sample + extrapolate — run the exact PERCENTILE_CONT on a 1% random sample, multiply with confidence bands. Good for ad-hoc analysis.
Probe 5 — "What about the transition matrix — how many minnows became dolphins this quarter?"

The static archetype rollup is the snapshot. The interesting question is the flow:

-- The archetype transition matrix — pair this quarter's classification
-- with last quarter's, count user_id flow between buckets.
SELECT prev.archetype AS last_q_archetype,
       curr.archetype AS this_q_archetype,
       COUNT(*) AS users
FROM archetype_snapshot prev
JOIN archetype_snapshot curr USING (user_id)
WHERE prev.snapshot_date = '2025-01-01' AND curr.snapshot_date = '2025-04-01'
GROUP BY 1, 2 ORDER BY 1, 2;

"Whale → whale" = the persistent VIPs. "Minnow → dolphin" = the win the platform is built on. "Dolphin → minnow" = the leading-indicator churn risk before the financial metrics show it. The static rollup hides all three.

Senior signal: name all four archetypes explicitly (whale / heavy_dolphin / dolphin / minnow), surface skew_ratio + p99_over_p50 alongside the rollup, pre-empt the LAD-tolerant view, and propose the transition matrix as the actually-interesting follow-up. One sentence to close: "because engagement data is positively skewed toward whales, I'm using NTILE / PERCENTILE_CONT, never AVG ± STDDEV — the standard-deviation approach assumes a normal distribution that engagement data never is."
