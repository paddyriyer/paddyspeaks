Disputes (chargebacks) are Stripe's risk metric — too many disputes means the card networks penalize Stripe AND the merchant. The query computes per-merchant dispute rate and classifies into risk tiers.

2. The SQL — rate + tier-classification

SELECT merchant_id,
  COUNT(*) AS charges,
  SUM(disputed) AS disputes,
  ROUND(100.0 * SUM(disputed) / NULLIF(COUNT(*), 0), 2) AS dispute_rate_pct,
  CASE
    WHEN 100.0 * SUM(disputed) / NULLIF(COUNT(*), 0) >= 1.0 THEN 'high_risk'
    WHEN 100.0 * SUM(disputed) / NULLIF(COUNT(*), 0) >= 0.3 THEN 'monitor'
    ELSE 'normal'
  END AS risk_tier
FROM charges
GROUP BY merchant_id
HAVING COUNT(*) >= 5
ORDER BY dispute_rate_pct DESC;

3. Strategic explanations

The 1% threshold — the network-mandated bright line

Visa and Mastercard impose penalties on merchants with sustained > 1% dispute rate. Stripe pre-emptively flags merchants approaching this to give them time to fix.

The CASE-in-SELECT for tier classification

Inline CASE keeps the query single-pass. Alternative: CTE with the rate first, then classify. The inline form is more readable for reports.

The chargeback fight-back rate

Disputes can be contested by the merchant. Win-rate (disputes won / disputes filed) is a quality signal too — high dispute rate but high win-rate means buyer-fraud, not merchant-fraud.

SUM(CASE WHEN d.outcome = 'won' THEN 1 ELSE 0 END) * 1.0 / NULLIF(SUM(disputed), 0) AS win_rate
Time-to-dispute

Most disputes happen 30-90 days after charge. A merchant with disputes spread across 1-180 days has different risk profile from one with disputes clustered at day 1. Slice by time-since-charge.

Reason-code grouping

Visa's chargeback reason codes split into:

  • "Fraud" (4837) — card stolen
  • "Customer dispute" (4853) — service not as described
  • "Authorization" (4808) — auth failure
  • "Processing error" (4831) — operational

Different remediations. Production groups by reason-code top-N per merchant.

The Radar feedback loop

High-risk merchants get tighter Radar rules — more transactions blocked at the auth stage to prevent the dispute downstream. The query result feeds the Radar model retrain.

Senior signal: name the 1% network bright line, surface chargeback win-rate as a counter-signal, slice by time-since-charge and reason-code, and connect to the Radar feedback loop. Inline CASE for tier is the readability move.
