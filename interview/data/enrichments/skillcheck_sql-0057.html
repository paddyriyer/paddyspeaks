A product manager asks a deceptively simple question: "what's the average number of orders per customer?" The trap is the customers who ordered nothing — if they are dropped, the average is inflated. The interviewer is checking whether you compute a two-stage average: count per customer first (including zeros), then average those counts — rather than dividing total orders by the wrong denominator.

1. The source data — customer and orders

Customers 3 and 5 are the edge case: they exist in customer but have no rows in orders. They must contribute a count of 0 to the average — dropping them changes 6/5 into 6/3.

2. The SQL — per-customer counts, then average

SELECT ROUND(AVG(order_count)::numeric, 2) AS avg_orders_per_customer
FROM (
  SELECT c.id, COUNT(o.order_id) AS order_count
  FROM customer c
  LEFT JOIN orders o ON o.customer_id = c.id
  GROUP BY c.id
) t;

The inner query is stage one: a LEFT JOIN from customer keeps every customer, and COUNT(o.order_id) yields 0 for those with no matching orders. The outer query is stage two: AVG over those per-customer counts.

3. The resulting analysis

avg_orders_per_customersignal
1.20(3 + 1 + 0 + 2 + 0) / 5 = 6 / 5 = 1.20

The answer is 1.20, computed over all five customers. Drop the two zero-order customers and you would get 6/3 = 2.00 — a 67% overstatement. The single number is correct only because the zeros are in the denominator.

4. Strategic explanations

Why two stages, not one

The naive one-liner SELECT COUNT(*) / COUNT(DISTINCT customer_id) FROM orders divides total orders by customers who appear in orders — it cannot see customers 3 and 5 at all, because they have no rows there. The two-stage shape fixes the denominator: stage one materialises one row per customer (zeros included), stage two averages. The structure encodes "every customer counts, even the silent ones".

COUNT(o.order_id) vs COUNT(*) on a LEFT JOIN

This is the make-or-break detail. After a LEFT JOIN, an unmatched customer still produces one output row, but with all orders columns NULL. COUNT(*) counts rows, so it would return 1 for a zero-order customer — wrong. COUNT(o.order_id) counts non-NULL values of an orders-side column, so the all-NULL phantom row counts as 0. Always count a column from the right-hand table, never *.

The ::numeric cast before ROUND

Two reasons. First, integer division: if AVG operated on plain integers without care, 6 / 5 could truncate to 1 — AVG itself returns numeric here, but casting makes the intent explicit and safe. Second, PostgreSQL's two-argument ROUND(value, 2) only accepts numeric; calling it on a double precision raises "function round(double precision, integer) does not exist". The cast guarantees both correct division and a compilable ROUND.

An equivalent single-pass form — and its risk

You can also write COUNT(o.order_id)::numeric / COUNT(DISTINCT c.id) over the same LEFT JOIN without a subquery. It gives 1.20 too, but it is easier to get subtly wrong (which COUNT over which table) and harder to extend — the moment the PM also wants the median orders per customer, you need the per-customer rows anyway. The explicit two-stage form is more robust and more extensible.

Why AVG can't be nested directly

You might reach for AVG(COUNT(o.order_id)) in one query — it does not compile. SQL forbids nesting one aggregate inside another in the same query level: COUNT collapses rows within a group, and AVG would then need a second, outer level of grouping that does not exist. The subquery is that second level — the inner SELECT produces the per-customer counts as ordinary rows, and only then can the outer AVG aggregate them. Two passes are structurally required, not stylistic.

The result shape — one row, one column

The question asks for a single scalar. The outer query has no GROUP BY and one aggregate in the SELECT list, so it returns exactly one row with one column — the contract the prompt specifies. If a stray non-aggregate column crept into that outer SELECT it would either error ("must appear in GROUP BY") or fan the result back out to many rows. Keeping the final projection a lone aggregate is what guarantees the scalar shape.

Senior signal: recognise this as a two-stage average — LEFT JOIN from the customer table so zero-order customers survive, COUNT(o.order_id) (a right-side column, never COUNT(*)) so their count is 0, then AVG the per-customer counts — and cast to numeric before two-argument ROUND.
