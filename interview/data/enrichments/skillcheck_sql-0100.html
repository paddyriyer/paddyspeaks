A category manager wants products ranked by units sold within each category — but discontinued products are a special case: they must still appear in the listing, yet must not occupy a rank. Active products are densely ranked with no gaps. The interviewer is probing whether you can rank a subset of rows while keeping the excluded rows visible, and whether you understand how a window function treats NULL in its ORDER BY.

The edge cases are pointed. Product 4 has the highest units in category A (999) but is discontinued — it must get rank NULL, not rank 1. Products 2 and 3 are tied at 80 — they must share a rank. Category C has a single product. And category B's discontinued product sits in the middle of the sales range, testing that exclusion is by status, not by position.

2. The SQL — rank a subset with a CASE inside the window

SELECT product_id, category, units_sold, is_discontinued,
       CASE WHEN is_discontinued THEN NULL
            ELSE DENSE_RANK() OVER (
              PARTITION BY category
              ORDER BY CASE WHEN is_discontinued THEN NULL ELSE units_sold END DESC NULLS LAST
            ) END AS rank_in_category
FROM product
ORDER BY category, units_sold DESC, product_id;

The window ORDER BY maps every discontinued row's sort key to NULL and pushes those NULLs to the bottom with NULLS LAST, so they never take an early rank. The outer CASE then blanks the discontinued rows' rank to NULL outright.

4. Strategic explanations

Ranking a subset: two CASEs, two different jobs

The query has a CASE in two places and they are not redundant. The CASE inside the window's ORDER BY removes discontinued rows from the ranking computation — it makes their sort key NULL so they cannot earn a rank number. The CASE outside, wrapping DENSE_RANK(), sets the displayed value to NULL. Without the inner one, a discontinued product would still consume a rank slot; without the outer one, it would show whatever rank the NULL sort key landed on.

NULLS LAST is what keeps the gaps out

In PostgreSQL, ORDER BY ... DESC places NULLs first by default. If the discontinued rows' NULL sort keys sorted first, DENSE_RANK would assign them rank 1 and active products would start at 2 — gap introduced. DESC NULLS LAST forces the NULLs to the bottom of each partition, so active products are ranked first and densely from 1. The NULLS LAST is load-bearing, not decoration.

Why DENSE_RANK and not RANK or ROW_NUMBER

The spec says "densely ranked (1, 2, 3, ... ties share a rank, no gaps)". Products 2 and 3 tie at 80. DENSE_RANK gives 1, 2, 2 — and the next distinct value would be 3. RANK would give 1, 2, 2, then skip to 4. ROW_NUMBER would give 1, 2, 3 — breaking the tie arbitrarily. "No gaps after ties" is the precise definition of DENSE_RANK; the wording chooses the function.

An alternative: a filtered CTE then a join-back

You could rank only the active products in a CTE — SELECT ..., DENSE_RANK() OVER (PARTITION BY category ORDER BY units_sold DESC) FROM product WHERE NOT is_discontinued — then LEFT JOIN that back onto the full product list so discontinued rows pick up a NULL rank. Functionally equivalent and arguably clearer. The single-query CASE-in-ORDER-BY form is terser and avoids the join; both are defensible — saying so signals range.

The single-product partition still works

Category C has one product. PARTITION BY category makes a partition of one, DENSE_RANK assigns it rank 1, done — window functions handle singleton partitions cleanly. The discontinued-in-the-middle case (category B) is the real subtlety: exclusion is driven by is_discontinued, not by sales position, so an active product with lower units than a discontinued one still out-ranks it.

Senior signal: to rank a subset while keeping excluded rows visible, neutralise their window ORDER BY key with a CASE and push them down with NULLS LAST, blank their displayed rank with an outer CASE, and pick DENSE_RANK because "no gaps after ties" is exactly its semantics — RANK skips, ROW_NUMBER breaks ties.
