An analytics dashboard needs a smoothed sales line: alongside each day's raw amount, show the trailing 7-day total. The interviewer is probing window frames — specifically ROWS BETWEEN 6 PRECEDING AND CURRENT ROW — and whether you understand how the frame behaves on the first few rows, and the subtle difference between a ROWS frame and a RANGE frame.

The boundary case is the start of the series. Rows 1–6 do not have six days of history behind them, so the "7-day" sum is necessarily a partial sum. Row 7 is the first with a complete window; row 8 is the first where the window genuinely slides — gaining day 8, losing day 1.

2. The SQL — a windowed SUM with a row frame

SELECT
  sale_date,
  amount,
  SUM(amount) OVER (
    ORDER BY sale_date
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS rolling_7d
FROM sales
ORDER BY sale_date;

The OVER clause defines a window ordered by sale_date. The frame ROWS BETWEEN 6 PRECEDING AND CURRENT ROW says: for each row, sum the amount of itself plus the six rows immediately before it — seven rows in total. As the engine walks down the ordered rows, that seven-row window slides one step at a time.

4. Strategic explanations

Why the first six rows are "partial" — and that it is correct

A frame can never reach rows that do not exist. On 2025-01-01 there are zero preceding rows, so rolling_7d equals just that day's amount. This is the expected behaviour of a trailing window, not a bug. If the business wants those early days suppressed or shown as NULL until a full window exists, you add a guard like CASE WHEN COUNT(*) OVER (...same frame...) = 7 THEN SUM(...) END — mention this, because a stakeholder may not want a "7-day average" computed from 2 days.

ROWS vs RANGE — the trap the prompt pre-empts

ROWS counts physical rows: "6 preceding" means the six rows above, whatever dates they hold. RANGE counts logical values of the ORDER BY column. With exactly one row per day they coincide, which is why the prompt says "assume one row per day". But if a day were missing, ROWS would reach back six rows — possibly spanning nine calendar days — whereas a true date-based window needs RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW. Choosing ROWS here is correct given the assumption; saying so out loud shows you know the difference.

Gaps and duplicates would break the row-count assumption

If two sales landed on the same date, a ROWS frame would treat them as two separate steps, so "7 rows" might cover only six days. If a date were skipped, seven rows would span more than seven days. For a calendar-accurate rolling window on irregular data, the robust choice is the RANGE/INTERVAL frame, which is defined by date distance and is immune to both gaps and duplicates.

The ORDER BY inside OVER is mandatory here

A frame clause is only meaningful relative to an ordering — "6 preceding" has no meaning without knowing what "preceding" is. Omitting ORDER BY inside OVER makes the frame illegal (or collapses to summing the whole partition). Note this ORDER BY is independent of the final ORDER BY sale_date that sorts the output: one drives the window, the other drives presentation. Keeping them aligned is good practice but they are distinct clauses.

Senior signal: use ROWS BETWEEN 6 PRECEDING AND CURRENT ROW for a 7-row trailing sum, recognise that the first six rows yield correct partial windows, and explain that ROWS equals RANGE only because there is one row per day — gaps or duplicates would demand a RANGE ... INTERVAL '6 days' frame for a calendar-accurate window.
