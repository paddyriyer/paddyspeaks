A community sales dataset asks for the top-selling country in each continent — first summing sales to the country level, then surfacing the leader per continent. This is the universal "max-per-group" / "greatest-N-per-group" interview pattern, behind every "best region", "top product per category", "highest-paid employee per department" question. The interviewer is watching how you rank within groups and, above all, how you handle ties at the top.

1. The source data — salesinfo

2AsiaIndia300India row 2 — India totals 800

7AfricaKenya120Kenya — sole country in Africa

After summing, Asia has a three-way tie at 800 (India 500+300, China 800, Japan 800). Africa has a single country. Both decide the design: does the query return one winner per continent, or all tied winners?

2. The SQL — aggregate, then DENSE_RANK per continent

WITH t1 AS (
  SELECT continents, country,
         SUM(CASE WHEN sales::text ~ '^-?[0-9]+(\.[0-9]+)?$' THEN sales::numeric ELSE 0 END) AS total_sales
  FROM salesinfo
  GROUP BY continents, country
),
rnk_sales AS (
  SELECT *, DENSE_RANK() OVER (PARTITION BY continents ORDER BY total_sales DESC) AS rnk
  FROM t1
)
SELECT continents, country, total_sales, rnk
FROM rnk_sales
WHERE rnk = 1;

t1 rolls sales up to one row per (continent, country); DENSE_RANK ranks countries within each continent by descending total; filtering rnk = 1 keeps the leader(s).

Asia returns three rows — China, India, and Japan all tie at 800, so all three get rank 1. Africa returns Kenya, the only country there. This "ties produce multiple winners" behaviour is a deliberate property of DENSE_RANK, not a bug.

4. Strategic explanations

Why aggregate first, then rank

The question is about country totals, but the raw table has multiple rows per country (India appears twice). Ranking the raw rows would compare individual transactions, not countries. The GROUP BY continents, country in t1 collapses each country to one total before DENSE_RANK sees it — aggregate, then rank, in that order.

DENSE_RANK vs RANK vs ROW_NUMBER — the tie decision

All three rank within PARTITION BY continents, but treat the 800-way tie differently. ROW_NUMBER would pick one arbitrary winner — India or China or Japan, decided by physical order, non-deterministic. RANK and DENSE_RANK both give all tied countries rank 1, so all three are returned. The choice is a product question: "show the top country" (one row) → ROW_NUMBER with a deterministic tiebreaker; "show all countries that lead" → RANK/DENSE_RANK. Naming this trade-off is the senior signal — WHERE rnk = 1 silently returns a variable number of rows per continent.

The single-country edge — Africa

Africa has one country, Kenya, which trivially ranks 1. No ranking method drops it — a partition of size one always has a rank-1 row. Worth confirming out loud, because interviewers sometimes expect "top N" logic to mishandle a group of size one.

Correlated-subquery alternative and the cast guard

Pre-window-function, the same result comes from WHERE total_sales = (SELECT MAX(total_sales) FROM t1 t2 WHERE t2.continents = t1.continents) — correct, and it also returns all ties, but it re-scans per row. The window version sorts once. Separately, sales is TEXT, so the CASE ... ~ '^-?[0-9]+(\.[0-9]+)?$' regex validates each value before ::numeric; the ELSE 0 branch keeps a dirty 'N/A' from aborting the query, but silently under-counts that country — the column should really be NUMERIC upstream.

Senior signal: aggregate to the country grain before ranking, and make the tie decision explicit — DENSE_RANK/RANK with WHERE rnk = 1 returns every tied leader (Asia yields three rows here), whereas ROW_NUMBER forces a single arbitrary winner and needs a deterministic tiebreaker.
