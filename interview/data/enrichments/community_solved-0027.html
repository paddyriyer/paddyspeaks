A delivery team at a Community open-source project logs every task as a calendar day of work. The PM wants the project timeline reconstructed: which stretches of days were worked back-to-back, and how long each stretch lasted. The data has no "project" column — a project is a maximal run of consecutive dates. The interviewer is checking whether you reach for gaps-and-islands instead of trying to write a recursive crawl.

1. The source data — tasks

There are two gaps: Jan 3 to Jan 5 (Jan 4 missing) and Jan 6 to Jan 9. That splits the six rows into three islands — a 3-day run, a 2-day run, and a lone single day. The single day is the edge case: a "project" of one task still has duration 1.

2. The SQL — date − row_number islands

WITH numbered AS (
    SELECT start_date,
           ROW_NUMBER() OVER (ORDER BY start_date) AS rn
    FROM tasks
),
islands AS (
    SELECT start_date,
           start_date - (rn || ' days')::INTERVAL AS island_key
    FROM numbered
)
SELECT MIN(start_date)::DATE                      AS project_start,
       MAX(start_date)::DATE                      AS project_end,
       (MAX(start_date) - MIN(start_date) + 1)    AS duration_days
FROM islands
GROUP BY island_key
ORDER BY duration_days ASC, project_start ASC;

Number the rows in date order. Within a consecutive run both the date and rn increment by exactly 1, so date − rn stays constant. The moment a gap appears, the date jumps but rn does not — the difference shifts. That constant is the island key.

Three projects, sorted shortest-first as the ORDER BY asks. Notice the lone Jan 9 task: MIN and MAX are the same date, and MAX − MIN + 1 = 1 — the + 1 is what makes a one-day project read as duration 1 rather than 0.

4. Strategic explanations

Why subtract row_number at all?

The whole trick is that two monotonic sequences advancing at the same rate have a constant difference. Dates in a run go 1, 2, 3; rn goes 1, 2, 3; the difference is flat. Across a gap the date sequence skips a value while rn does not, so the difference steps up by the size of the gap. Grouping on that difference collapses each run into one row — no recursion, no self-join, one sort.

Integer dates make this cleaner

Postgres lets you subtract an INTERVAL from a date, which is why the solution builds (rn || ' days')::INTERVAL. A tidier idiom many engineers prefer is to anchor against day-zero arithmetic directly:

SELECT start_date,
       start_date - rn * INTERVAL '1 day' AS island_key
FROM numbered;

Same result. If start_date were an integer surrogate instead of a real date, it collapses to the textbook one-liner start_date - rn.

The trap: duplicate dates

This pattern assumes each date appears at most once. If two tasks share 2024-01-02, ROW_NUMBER gives them different rn values, so their date − rn differs and the run is wrongly split. If duplicates are possible, dedupe first (SELECT DISTINCT start_date) or rank with DENSE_RANK() over distinct dates so tied dates collapse to one ordinal.

HackerRank's two-column variant

The classic HackerRank "Project Planning" question ships Start_Date and End_Date where each task spans exactly one day (End = Start + 1). There the island key is Start_Date − rn over start dates, and you report MIN(Start_Date) and MAX(End_Date). The single-date framing here is the same problem with the redundant end column dropped — recognising the equivalence is the senior move.

Edge cases: one row, and all-consecutive

A single task yields one island, duration 1 — verified above. A table with no gaps at all yields exactly one island spanning everything; the GROUP BY still works, you just get one row. An empty table returns zero rows, which is correct — no projects.

Senior signal: recognise "consecutive runs" as gaps-and-islands and reach for date − row_number rather than recursion, explain why the difference is constant within a run, pre-empt the duplicate-date trap with DISTINCT or DENSE_RANK, and connect the single-date framing back to the canonical two-column HackerRank version.
