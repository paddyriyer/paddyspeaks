Citi's billing-reconciliation team computes a net position for every payment: what the customer paid minus what they owed. The arithmetic is trivial — one subtraction — but the payments table is full of NULLs, because a payment that has not cleared has no paid_amount yet, and a payment with no associated invoice has no due_amount. In SQL, any arithmetic touching a NULL collapses the whole expression to NULL. The interview is about defusing that.

1. The source data — payments

Every NULL shape is present: NULL on the left only (row 2), NULL on the right only (row 3), NULL on both sides (row 4). Row 4 is the sharpest edge case — it decides whether "no data at all" should produce 0 or stay NULL.

2. The SQL — COALESCE both sides

SELECT payment_id,
       COALESCE(paid_amount,0) - COALESCE(due_amount,0) AS net
FROM payments;

COALESCE returns its first non-NULL argument, so each NULL becomes 0 before the subtraction runs. Both operands are guaranteed numeric, so the result is always a real number.

Row 2: 0 − 50 = −50 (owed money, paid nothing). Row 3: 200 − 0 = 200 (paid, nothing due). Row 4: 0 − 0 = 0 — and that is the row to interrogate. A payment with no paid and no due amount is most likely a data gap, yet it now reports a perfectly balanced 0.

4. Strategic explanations

Why one NULL poisons the whole expression

Without COALESCE, paid_amount - due_amount for row 2 is NULL - 50, which is NULL; row 4 is NULL - NULL, also NULL. NULL means "unknown", and unknown minus anything is unknown. COALESCE on both operands is required — guarding only one side still leaves the other able to NULL out the result. Coalescing both is the deliberate, complete fix.

ISNULL is the wrong dialect — the title's trap

The question is titled "Null-Safe Calculation with ISNULL", but ISNULL is SQL Server syntax and is not valid in Postgres. The portable, ANSI-standard choice is COALESCE, which the solution correctly uses. Worth saying out loud:

-- SQL Server only:   ISNULL(paid_amount, 0)
-- MySQL only:        IFNULL(paid_amount, 0)
-- Oracle:            NVL(paid_amount, 0)
-- ANSI / portable:   COALESCE(paid_amount, 0)   <- prefer this

COALESCE also accepts more than two arguments and is standard everywhere, so it should be the default in any cross-engine codebase. Naming the dialect mismatch is itself a senior signal.

The "is 0 a real value or a missing value?" question

Coalescing to 0 conflates two different situations: a payment genuinely worth net 0 (row 5, paid 75 / due 75) and a payment with no data at all (row 4). Both now read 0. If downstream consumers must tell them apart, do not blanket-coalesce — keep the NULL and let it mean "unknown":

SELECT payment_id,
       paid_amount - due_amount AS net,         -- NULL when data missing
       (paid_amount IS NULL OR due_amount IS NULL) AS incomplete
FROM payments;

Whether 0 or NULL is correct is a business decision — surface it rather than defaulting silently.

COALESCE vs NULLIF — don't confuse them

A frequent mix-up: NULLIF(a, b) does the opposite — it returns NULL when a = b, otherwise a. It is for turning a sentinel value into NULL (e.g. NULLIF(due_amount, 0) to dodge a later divide-by-zero). COALESCE replaces NULL with a value; NULLIF creates NULL from a value. They are not interchangeable.

Aggregating the net — NULLs behave differently there

If the follow-up is "total net across all payments", note that aggregates already skip NULLs: SUM(paid_amount - due_amount) would silently drop rows 2, 3 and 4 and total only rows 1 and 5. To include the partial payments in the sum you still need the per-row COALESCE first: SUM(COALESCE(paid_amount,0) - COALESCE(due_amount,0)).

Senior signal: explain that a single NULL operand collapses the whole arithmetic expression so both sides need guarding, flag that ISNULL from the title is SQL Server syntax and COALESCE is the portable choice, and raise that coalescing to 0 hides genuine data gaps — making "0 vs NULL" an explicit business decision rather than a default.
