An academic-analytics team wants to flag high-variance students — those whose best exam score sits at least 20 points above their worst, a sign of uneven performance across subjects. The interviewer is probing the bread-and-butter of aggregation: GROUP BY with MIN/MAX, and crucially the difference between WHERE and HAVING when the filter is on an aggregate.

Three edge cases. Bob's spread is only 5 — he must be filtered out. Dan has a single exam, so his min equals his max and spread is 0 — also excluded. And Alice and Eve both have a spread of exactly 45 — a tie in the ORDER BY spread DESC.

2. The SQL — GROUP BY with HAVING on the spread

SELECT
  student,
  MIN(points) AS min_pts,
  MAX(points) AS max_pts,
  MAX(points) - MIN(points) AS spread
FROM score
GROUP BY student
HAVING MAX(points) - MIN(points) >= 20
ORDER BY spread DESC;

GROUP BY student collapses each student's exams into one row; MIN and MAX find their worst and best scores. HAVING filters those grouped rows, keeping only students whose MAX - MIN is at least 20. ORDER BY spread DESC ranks the survivors widest-first.

4. Strategic explanations

HAVING vs WHERE — the central distinction

WHERE filters rows before grouping; HAVING filters groups after aggregation. The condition here is on MAX(points) - MIN(points) — an aggregate that does not exist until the group is formed — so it must be in HAVING. Writing WHERE MAX(points) - MIN(points) >= 20 is a hard error: aggregates are not allowed in WHERE because WHERE runs before any group exists. Knowing exactly which clause an aggregate filter belongs in is the core competency this question tests.

The single-exam group — why Dan's spread is 0

For a group of one row, MIN and MAX both return that row's value, so MAX - MIN = 0. Dan therefore fails >= 20 and drops out — correct, since one exam carries no variance. This is worth saying aloud: aggregates are perfectly well-defined on single-member groups, they just collapse to that member. No special-casing is needed; the HAVING threshold handles it.

The tie in ORDER BY — add a tiebreaker for determinism

Alice and Eve both have spread 45. ORDER BY spread DESC places them adjacent but in an unspecified order — rerun the query and they may swap. If the output must be reproducible (a snapshot test, a paginated report), add a deterministic tiebreaker: ORDER BY spread DESC, student. Volunteering that a sort key with duplicate values is non-deterministic, and fixing it, is a senior signal.

NULL points and the empty edge

MIN and MAX silently skip NULLs, so a NULL exam score does not poison the spread — but a student whose scores are all NULL would have MIN and MAX both NULL, spread NULL, and NULL >= 20 is unknown, so they drop out. That is usually the right behaviour, but state it. Likewise, a student is in the result only if they have at least one row; students absent from score entirely never form a group.

Recomputing the aggregate vs aliasing it

The query repeats MAX(points) - MIN(points) in both the SELECT and the HAVING. That is required by standard SQL — HAVING cannot reference a SELECT-list alias (spread) because HAVING is logically evaluated before the SELECT projection. ORDER BY can use the alias, which is why ORDER BY spread works. PostgreSQL and MySQL tolerate aliases in HAVING as an extension, but the portable form recomputes the expression — the optimiser collapses the duplication anyway.

Senior signal: filter on an aggregate with HAVING, never WHERE (aggregates do not exist before grouping); know that MIN/MAX on a single-row group collapse to that value (spread 0); add a tiebreaker to ORDER BY when the sort key has duplicates; and recall that HAVING cannot see SELECT aliases while ORDER BY can.
