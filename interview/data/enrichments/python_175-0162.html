Dijkstra with max-edge instead of sum-edge. The same shape as flood-fill at a variable threshold, terrain-flood simulation in GIS, and any "minimise the worst link in the chain" problem (bottleneck shortest path). The classical Dijkstra minimises sum of edge weights; here we minimise the maximum elevation along the path. That single substitution — max instead of + in the priority key — turns the algorithm from "shortest path" into "minimum-bottleneck path". IBM asks this version because their resilient-network routing for legacy mainframe traffic prefers the path with the lowest peak congestion, not the shortest hop count — same problem, same algorithm.

1. The problem in one sentence + the key insight

An n × n grid of elevations; at time t every cell ≤ t is submerged and crossable; find the minimum t at which (0,0) and (n−1, n−1) are connected. Insight: the answer is the maximum elevation on the path that minimises this maximum. Run Dijkstra from (0,0); the "distance" to a cell is the max elevation along the best path to it. The priority queue pops the next-lowest-max-so-far cell; the first time we pop (n−1, n−1) is the answer. Same skeleton as Dijkstra, only the relaxation step changes from dist[v] = min(dist[v], dist[u] + w(u,v)) to dist[v] = min(dist[v], max(dist[u], grid[v])).

         heap = [(4,1,1), (5,1,2), (6,2,0)],  seen += {(2,0)}

iter 5:  pop (4, 1, 1).  ans = 4.  Not goal.
         no unseen neighbours (all four are already in seen)

iter 6:  pop (5, 1, 2).  ans = 5.  Not goal.
         push neighbour:  (8, 2, 2)  (down)
         heap = [(6,2,0), (7,2,1), (8,2,2)],  seen += {(2,2)}    [wait — (2,1) wasn't pushed yet]

         Correction: push (7, 2, 1)  -- actually (2,1) is pushed when (1,1) is later
         processed via (5,1,2)'s neighbours OR via (2,0)'s right neighbour.

iter 7:  pop (6, 2, 0).  ans = 6.  push neighbour (7, 2, 1).
iter 8:  pop (7, 2, 1).  ans = 7.  push neighbour (8, 2, 2) (already in seen, skip).
iter 9:  pop (8, 2, 2).  ans = 8.  IS GOAL -> return 8.

The route was effectively forced by the increasing row-major elevations; the max along it is 8. Notice that ans = max(ans, t) only ever grows — the heap pops in non-decreasing order, so by the time we pop (n−1, n−1), the running max IS the answer.

3. The algorithm — code with inline comments

import heapq

def swimInWater(grid):
    n = len(grid)
    # Heap entries: (elevation, row, col).
    # Python's heapq is a min-heap; we pop the next-lowest-elevation frontier cell.
    h = [(grid[0][0], 0, 0)]
    seen = {(0, 0)}     # cells whose entry-elevation has been finalised
    ans = 0             # running maximum elevation popped so far

    while h:
        t, r, c = heapq.heappop(h)
        ans = max(ans, t)            # this is the Dijkstra relaxation
        if (r, c) == (n - 1, n - 1): return ans
        for dr, dc in [(1,0),(-1,0),(0,1),(0,-1)]:
            nr, nc = r + dr, c + dc
            if 0 <= nr < n and 0 <= nc < n and (nr, nc) not in seen:
                seen.add((nr, nc))
                heapq.heappush(h, (grid[nr][nc], nr, nc))

Data structures: heapq as min-priority queue + a set for visited cells. Invariants: (1) every cell in seen has been heap-pushed exactly once; (2) the running ans is monotone non-decreasing because heapq pops in increasing order — so when (n−1, n−1) is finally popped, ans equals the max elevation on the chosen path.

4. Complexity

Time: O(n² log n²) = O(n² log n). Each of the n² cells is pushed and popped at most once; each heap op is O(log n²). Space: O(n²) for the seen set and the heap. The dominant term is the heap log factor — for n=50, n² = 2500 and the work is ~25k heap ops. Trivially fast.

5. The interviewer's probes

Probe 1 — 1×1 grid. Heap starts with (grid[0][0], 0, 0); first pop sets ans = grid[0][0] and immediately matches the goal. Returns grid[0][0]. No special case needed.

Probe 2 — All zeros. Every cell has elevation 0; the heap pops in arbitrary order, ans stays at 0, returns 0 when goal is popped. Confirms the algorithm doesn't degenerate.

Probe 3 — Generalisation: minimum-cost-max-edge path on a general graph. Same algorithm, replace grid[nr][nc] with the edge weight w(u, v) and the relaxation becomes max(dist[u], w(u, v)). Bottleneck shortest path — used in network reliability, maximum-flow Edmonds-Karp variants.

Probe 4 — Alternative algorithm: Union-Find + sorted cells. Sort all n² cells by elevation. Process in order: when a cell is processed, union it with already-processed neighbours. The answer is the elevation at which (0,0) and (n−1, n−1) become connected. O(n² log n) — same complexity but no heap. Some interviewers prefer this because it's more idiomatic for "merging islands".

Probe 5 — Alternative algorithm: binary search + BFS. Binary-search the answer t ∈ [0, n²−1]; for each candidate t, BFS over cells with elevation ≤ t and check connectivity. O(n² log(n²)) — same complexity, clearer monotonicity argument. Use this version when explaining "why is the answer monotone in t".

Probe 6 — Memory: n=10^4 grid? The algorithm's O(n²) memory becomes 10^8 cells × ~30 bytes per heap entry = 3 GB. Out-of-core: process cells in elevation order from disk, maintain a streaming union-find. This is exactly how GIS terrain-flood tools (ArcGIS, GRASS) handle billion-cell DEMs.

6. Where this pattern appears in production

  • GIS terrain-flood simulation (ArcGIS, GRASS, QGIS) — given a digital elevation model and a rising water level, find regions that become connected. Same algorithm at billion-cell scale (with streaming union-find).
  • Network reliability routing (Cisco, Juniper) — find the path between two routers that minimises the worst-case link latency or packet-loss rate. Bottleneck path == this algorithm.
  • Maximum spanning tree as bottleneck-path proxy — Prim's MST is a generalisation; the path between any two nodes in a maximum spanning tree is the bottleneck path. Used in robust distributed system design.
  • Game pathfinding for "swim through liquid" mechanics — in roguelikes and Minecraft-style worlds, an entity can pass through cells iff terrain ≤ liquid level; this algorithm finds the minimum liquid level for a unit to reach a target.
  • Power grid / supply chain resilience — find the route between two nodes that minimises the highest-stress link. Same min-of-max formulation.
