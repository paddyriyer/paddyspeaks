Dropbox's RevOps lead asks: "what's the 3rd-largest deal we've ever closed?" — the answer feeds a slide on average deal size at the top of the funnel. The question generalises to "Nth highest" and is one of the most-asked SQL interview questions on the planet, because it forces a candidate to choose among OFFSET, DENSE_RANK, and a correlated subquery — each with different tie behaviour, performance profiles, and dialect support.

Notice the tie at $850K — that's the senior conversation.

2. The SQL — DISTINCT + OFFSET

SELECT DISTINCT revenue
FROM orders
ORDER BY revenue DESC
OFFSET 2 LIMIT 1;
-- Replace 2 with N-1 for the Nth highest distinct revenue

3. The resulting analysis

revenue (3rd highest distinct)
$700,000

Without DISTINCT, the 3rd row is the second $850K — almost certainly not what RevOps wanted. The DISTINCT is the senior-detail moment.

4. Strategic explanations

Three idioms, three semantics — pick deliberately
-- A. OFFSET-based — relies on DISTINCT for tie-handling
SELECT DISTINCT revenue FROM orders
ORDER BY revenue DESC OFFSET 2 LIMIT 1;

-- B. DENSE_RANK — most readable, ties native
SELECT revenue FROM (
  SELECT revenue, DENSE_RANK() OVER (ORDER BY revenue DESC) AS rk
  FROM orders
) WHERE rk = 3;

-- C. Correlated subquery — slow but classic
SELECT MAX(revenue) FROM orders o1
WHERE 2 = (SELECT COUNT(DISTINCT revenue) FROM orders o2 WHERE o2.revenue > o1.revenue);

All three return $700,000 on this dataset. Differences:

Pick DENSE_RANK when window functions are available; OFFSET when they aren't (legacy MySQL/SQLite). The correlated subquery is the historical answer — never the production answer.

RANK vs DENSE_RANK — the trap

If two deals tie at the top, what's the "2nd highest"?

revenueRANKDENSE_RANK
$1,000,00011
$850,00022
$850,00022
$700,00043

With RANK there is no row at rank 3 — WHERE rk = 3 returns nothing. With DENSE_RANK the next distinct value gets rank 3. For "Nth highest distinct", always use DENSE_RANK.

The dialect cheat sheet for OFFSET
-- Postgres / SQLite / MySQL 8 / Oracle 12+
SELECT DISTINCT revenue FROM orders ORDER BY revenue DESC OFFSET 2 LIMIT 1;

-- SQL Server (must include ORDER BY)
SELECT DISTINCT revenue FROM orders
ORDER BY revenue DESC OFFSET 2 ROWS FETCH NEXT 1 ROWS ONLY;

-- Oracle 11 and earlier — no OFFSET, use ROWNUM in a nested query
SELECT revenue FROM (
  SELECT revenue, ROWNUM rn FROM (
    SELECT DISTINCT revenue FROM orders ORDER BY revenue DESC
  ) WHERE ROWNUM <= 3
) WHERE rn = 3;
Edge case: N exceeds the number of distinct revenues

For 3 distinct revenues, requesting the 5th highest:

  • OFFSET: returns zero rows.
  • DENSE_RANK: returns zero rows.
  • If the consumer expects NULL: SELECT (SELECT ... LIMIT 1) AS nth — outer scalar wrap converts "no row" into NULL.
Top-N per group — the senior generalization

The follow-up is always "now do it per region". DENSE_RANK with PARTITION BY:

SELECT region, revenue
FROM (
  SELECT region, revenue,
    DENSE_RANK() OVER (PARTITION BY region ORDER BY revenue DESC) AS rk
  FROM orders
) WHERE rk <= 3;

Now you've got the top 3 deals per region in one shot — the canonical "leaderboard" pattern.

The semantic confusion to flag

"3rd highest revenue" can mean three things; clarify before answering:

  1. 3rd highest distinct revenue value — what most people mean (DENSE_RANK = 3 or DISTINCT + OFFSET 2).
  2. 3rd-ranked deal allowing ties (RANK = 3) — rare but RevOps-style.
  3. 3rd row in descending order (ROW_NUMBER = 3) — arbitrary tiebreak, almost always wrong.
Senior signal: ask "ties: how do you want them counted?" before writing SQL, name DENSE_RANK as the default and OFFSET-DISTINCT as the dialect-portable fallback, contrast all three with their performance profiles, and pre-empt the top-N-per-group generalisation. The dialect cheat sheet (Oracle 11 ROWNUM!) is the over-the-bar finish.
