A factless fact table records the intersection of dimension keys at a point in time without carrying any numeric measure — the fact is the event itself (a student was in class on a given date). The interviewer is probing whether you know when to use this pattern and, critically, how to answer questions that sound like they need a measure but do not: “How many students attended?” is answered by COUNT(*) on the factless fact, not by summing a column. The composite PK and the partition column are also under scrutiny.

1. The requirements — what the design must capture

  • Grain: one row per (student, class, date) combination — one row per attendance event.
  • Dimension FKs: student, class, teacher, school, and date — five keys covering the full context of an attendance event.
  • No additive measure: the table records presence, not a count or dollar amount.
  • Attendance type: a classification flag (present / tardy / excused / unexcused) kept on the fact as a low-cardinality degenerate dimension.
  • Attended flag: a Boolean shortcut for fast “present vs. absent” filtering without parsing attendance_type.
  • Composite PK: (student_sk, class_sk, date_sk) enforces one record per student per class per day.
  • Partition: by date_sk to prune date-range queries efficiently.

2. The DDL — the CREATE TABLE

CREATE TABLE fact_attendance (
  student_sk        BIGINT      NOT NULL,           -- FK -> dim_student
  class_sk          BIGINT      NOT NULL,           -- FK -> dim_class
  teacher_sk        BIGINT      NOT NULL,           -- FK -> dim_teacher
  school_sk         BIGINT      NOT NULL,           -- FK -> dim_school
  date_sk           INT         NOT NULL,           -- FK -> dim_date
  attended_flag     BOOLEAN     NOT NULL DEFAULT TRUE,
  attendance_type   VARCHAR(20) NOT NULL,           -- present / tardy / excused / unexcused
  recorded_at       TIMESTAMP,
  ingested_at       TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (student_sk, class_sk, date_sk)
)
PARTITION BY (date_sk);

4. Strategic explanations

Why factless? The event IS the measure

In a traditional fact table, numeric measures give the row analytical value. In a factless fact, the existence of a row is the datum. “How many students attended on Tuesday?” → SELECT COUNT(*) FROM fact_attendance WHERE date_sk IN (…Tuesday dates…) AND attended_flag = TRUE. There is no SUM to perform. The pattern also answers questions about events that did not happen: “Which enrolled students did NOT attend class X on date Y?” requires joining an enrollment coverage table to fact_attendance and finding the complement — a query structure that dimensional designers call a coverage/event pair.

Grain declaration and the composite PK

The grain is one row per student per class per calendar day. The composite PK (student_sk, class_sk, date_sk) enforces this at the database level. A student who is both present for half a period and then absent is ambiguous under this grain; if the business needs sub-period granularity, the grain would change to (student, class, date, period_sk) and the PK would expand accordingly. Stating the grain explicitly before choosing the PK is the critical design step.

Absent rows: include or exclude?

Two design philosophies exist: (1) only record present rows and derive absences as missing rows against an enrollment table, or (2) record both present and absent rows explicitly. Approach 1 keeps the table smaller (absent rows can dominate). Approach 2 makes absence queries trivial (WHERE attended_flag = FALSE) without requiring an enrollment join. For a school district, approach 2 is usually preferred: absence is a first-class analytical event (truancy tracking, early-warning systems), and the enrollment roster is a separate data source whose join adds complexity and potential for orphaned records.

teacher_sk and school_sk: denormalisation on the fact

Strictly, teacher and school are attributes of the class, not the attendance event. A pure star schema would derive them by joining dim_class. They are denormalised onto the fact here for two reasons: (1) teacher-level and school-level roll-up queries are extremely common (district dashboards), and (2) the additional columns are a small cost (8 bytes each) compared to the join elimination benefit at hundreds of millions of rows.

Partition by date_sk

School-attendance queries almost always include a date range (“this week,” “this semester”). Partitioning on date_sk prunes these scans to the relevant time window. One consideration: date_sk is an integer (e.g., 20250516 for 2025-05-16), not a DATE. Partitioning on an integer is supported in PostgreSQL range partitioning but not natively in BigQuery, which prefers a DATE column. For BigQuery, add a date_actual DATE NOT NULL column and partition on that.

Warehouse dialect notes

In BigQuery, replace PARTITION BY (date_sk) with a DATE column and PARTITION BY date_actual. In Snowflake, use CLUSTER BY (date_sk, school_sk). In Redshift, use DISTKEY(student_sk) if most queries are student-centric and SORTKEY(date_sk, school_sk) for zone-map pruning.

Senior signal: explain that the factless fact table records presence by existence — COUNT(*) is the only “measure” — and then describe the coverage/event-pair pattern (“join the enrollment table to fact_attendance and take the set difference to find non-attenders”); justify the decision to store absent rows explicitly as a first-class event; and note that teacher_sk and school_sk are deliberately denormalised onto the fact for roll-up query performance even though they are technically class attributes.
