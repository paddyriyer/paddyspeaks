NVIDIA's finance team wants a customer-by-year revenue grid — one row per customer, one column per fiscal year — the shape a spreadsheet wants but a normalised orders table does not store. This is a pivot: turning row values (the year) into columns. Postgres has no native PIVOT keyword, so the idiom is conditional aggregation. The interview tests whether you know that idiom and what its empty cells mean.

1. The source data — orders

Customer 101 ordered in all three years. Customer 102 skipped 2025 entirely — the missing-cell case. Customer 103 ordered twice in 2025 ($400 and $600) and never in 2024 or 2026 — the case that tests whether the aggregate sums correctly within a cell.

2. The SQL — SUM(CASE…) conditional pivot

SELECT customer_id,
  SUM(CASE WHEN year=2024 THEN amount END) AS y2024,
  SUM(CASE WHEN year=2025 THEN amount END) AS y2025,
  SUM(CASE WHEN year=2026 THEN amount END) AS y2026
FROM orders GROUP BY customer_id;

One GROUP BY customer_id collapses to one row per customer. Each SUM(CASE…) is a filtered total: the CASE returns amount only for its target year and NULL otherwise, and SUM ignores those NULLs.

Three customers, three year columns. Customer 103's y2025 is 1000.00 — the two 2025 orders ($400 + $600) summed inside one cell, exactly as a pivot should. Customer 102's y2025 is NULL — no 2025 orders, so the CASE never fired and SUM over an all-NULL set returns NULL. The rows come back in no particular order.

4. Strategic explanations

The empty cell is NULL, not 0 — decide which you want

Customer 102's missing 2025 shows as NULL, because SUM of zero non-NULL inputs is NULL (not 0). For a finance grid that is usually wrong — "$0 of revenue" is a fact, "unknown" is not, and NULLs break downstream arithmetic like a row total. Wrap each cell:

SUM(CASE WHEN year=2024 THEN amount END) -- becomes
COALESCE(SUM(CASE WHEN year=2024 THEN amount END), 0) AS y2024

Now a year with no orders reads as 0. State the choice explicitly: a NULL says "no data", a 0 says "we looked and there was none".

SUM(CASE…) vs the cleaner FILTER form

Postgres offers an aggregate FILTER clause that says the same thing more directly:

SELECT customer_id,
  SUM(amount) FILTER (WHERE year=2024) AS y2024,
  SUM(amount) FILTER (WHERE year=2025) AS y2025,
  SUM(amount) FILTER (WHERE year=2026) AS y2026
FROM orders GROUP BY customer_id;

Identical result, but it reads as "sum the 2024 amounts" rather than as arithmetic over a CASE. SUM(CASE…) remains the portable idiom that works on every engine; FILTER is the idiomatic Postgres choice. Knowing both, and why, is the senior detail.

The fixed columns are a real limitation

The query hard-codes 2024, 2025, 2026. A 2027 order would be silently dropped from every cell — it matches no CASE branch — and a 2027 column would simply not exist. A static pivot can only show columns you wrote by hand. Genuinely dynamic year sets need either application code that generates the SQL, or Postgres's crosstab() from the tablefunc extension. Name this ceiling; do not pretend the static query auto-adapts.

SUM vs MAX inside the pivot — the duplicate-row trap

Customer 103 has two 2025 rows, and SUM correctly adds them to $1,000. A common pivot mistake is MAX(CASE WHEN year=2025 THEN amount END), which is right only when each (customer, year) pair has at most one row — here it would return $600 and quietly lose the $400. Choose the aggregate to match the question: SUM for "total revenue per year", MAX/MIN for "largest order", COUNT for "number of orders".

Trust year, or derive it?

The schema carries both a year column and an order_date. They can drift — a backfilled or mis-stamped row might have year = 2025 but an order_date in 2024. The robust pivot derives the year from the authoritative timestamp: EXTRACT(YEAR FROM order_date). Ask which column finance treats as truth, and consider whether a status filter (excluding cancelled orders) belongs in the CASE as well.

Senior signal: recognise conditional aggregation as the standard pivot idiom, wrap each cell in COALESCE(…,0) when an empty cell should read as $0 rather than NULL, pick SUM over MAX so duplicate (customer, year) rows aggregate instead of being lost, and flag that the hard-coded year columns cannot adapt to new years without regenerating the query.
