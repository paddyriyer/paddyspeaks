A Community question in the playbook-analytics style: for each country, count how many users touched an Apple device and how many users there were in total. Product wants to size the iOS/Mac install base by geography. The interesting part is doing it in one pass — a conditional COUNT(DISTINCT …) — and being honest about what the LEFT JOIN direction does to events whose user is unknown.

1. The source data

playbook_events:

event_iduser_iddevice
11macbook pro

32windows pc

playbook_users maps users to countries: users 1,2→USA; users 3,4→India; user 5→Brazil. The traps: user 1 has two Apple events (must count once), and user 2 used both a Windows PC and a MacBook — one user, two device classes — so a per-event count would double them.

2. The SQL — conditional COUNT(DISTINCT)

SELECT
    pu.country,
    COUNT(DISTINCT CASE
        WHEN pe.device IN ('macbook pro', 'iphone 5s', 'ipad air')
        THEN pe.user_id END) AS n_apple_users,
    COUNT(DISTINCT pe.user_id) AS n_total_users
FROM playbook_events pe
LEFT JOIN playbook_users pu ON pe.user_id = pu.user_id
GROUP BY pu.country
ORDER BY n_total_users DESC;

The CASE inside COUNT(DISTINCT) returns the user_id only for Apple devices and NULL otherwise; COUNT ignores the NULLs, so it counts distinct Apple users per country.

3. The resulting analysis

countryn_apple_usersn_total_users
India12
USA22
Brazil11

USA shows 2 Apple users of 2 total — user 1 (two Apple events, counted once) and user 2 (MacBook, also counted once despite his Windows event). India has user 3 on iPad and user 4 on Samsung → 1 of 2. The DISTINCT is doing the heavy lifting: without it, user 1's two Apple events alone would have inflated the count.

4. Strategic explanations

Why CASE inside COUNT(DISTINCT) — the conditional-aggregation pattern

COUNT(DISTINCT CASE WHEN cond THEN user_id END) is the single-pass idiom for "count distinct things that satisfy a condition". The CASE with no ELSE yields NULL for non-Apple rows; COUNT drops NULLs and de-duplicates the rest. One scan produces both metrics side by side — no subquery, no join of two grouped results. Putting the device IN (…) in a WHERE instead would corrupt n_total_users, because it would also filter the rows feeding the total count.

The LEFT JOIN direction and the NULL-country bucket

The join is events LEFT JOIN users, so every event survives even if its user_id is absent from playbook_users — such rows get country = NULL and GROUP BY gathers them into a NULL group. That can be intentional ("count unmapped traffic") or a bug ("users should always resolve"). Here every user maps, so no NULL row appears, but you should say what would happen. An INNER JOIN would instead silently drop unmapped events — a different, equally deliberate choice.

Counts come from events, not the user table — the coverage gap

n_total_users counts distinct users who appear in playbook_events. A registered user in playbook_users with zero events is invisible to this query. If "total users per country" means all registered users, the query must drive from playbook_users and LEFT JOIN events. "Active users" vs "registered users" is a real definitional fork — confirm which one product means.

The device-list trap — exact strings only

device IN ('macbook pro', 'iphone 5s', 'ipad air') is a hard-coded, case-sensitive allow-list. A new Apple device ('iphone 14'), a capitalised value ('MacBook Pro'), or trailing whitespace all fail to match and quietly undercount. More durable: normalise with lower(trim(device)), pattern-match (device ILIKE 'iphone%' OR device ILIKE 'ipad%' OR device ILIKE 'macbook%'), or keep the mapping in a reference table. Flag that the literal list is brittle.

The ordering tiebreak

ORDER BY n_total_users DESC leaves countries with equal totals in arbitrary order — here India and USA both have 2. Add a deterministic secondary key, e.g. ORDER BY n_total_users DESC, country, so the report is reproducible run to run.

Senior signal: use COUNT(DISTINCT CASE WHEN …) for one-pass conditional counting rather than a WHERE that would corrupt the total, state what the LEFT JOIN does to unmapped events, distinguish active-from-events vs registered users, and call out the brittle hard-coded device list.
