Venture capital and private equity data teams need to answer: "Which industry sectors are attracting the most capital, and how concentrated or fragmented are the deal sizes within each sector?" This question tests a straightforward but important pattern — joining a transaction table to a dimension table, then computing a full suite of aggregates (count, sum, mean, max) in a single GROUP BY pass, ordered by the most business-relevant metric.

2. The SQL — join + multi-aggregate GROUP BY

SELECT c.sector,
       COUNT(*)         AS deal_count,
       SUM(i.amount)    AS total_invested,
       AVG(i.amount)    AS avg_deal_size,
       MAX(i.amount)    AS largest_deal
FROM investments i
JOIN companies   c ON i.company_id = c.company_id
GROUP BY c.sector
ORDER BY total_invested DESC;

4. Strategic explanations

INNER JOIN semantics — what gets excluded

The INNER JOIN means any investment whose company_id does not exist in companies is silently dropped. Similarly, companies with no investments never appear in the output. In a financial audit context this could mask orphan transactions. A senior candidate names this and offers the alternative: LEFT JOIN companies c ON ... with COALESCE(c.sector, 'Unknown') ensures every investment is counted, even those lacking a company record.

Four aggregates, one scan

COUNT(*), SUM, AVG, and MAX are all computed in a single GROUP BY pass over the join result. The query engine never re-scans the data for each function. Knowing that these four are "free riders" on the same grouping operation is a query-cost awareness signal. Adding MIN(i.amount) or STDDEV(i.amount) would cost nothing extra in terms of scan passes.

AVG pitfalls: integer division and precision

If amount is stored as INTEGER, AVG(i.amount) in some engines (notably older SQL Server versions) returns an integer, truncating the decimal. In Postgres, Snowflake, and BigQuery, AVG of an integer returns a numeric/float result automatically. To be safe, cast explicitly: AVG(i.amount::numeric). This is especially important for financial data where fractional cents matter.

Ordering by alias — when it works

ORDER BY total_invested DESC references the column alias. This works in Snowflake, Postgres, BigQuery, and MySQL because ORDER BY is evaluated after SELECT. It does not work in the WHERE or HAVING clause — a common mistake is HAVING total_invested > 1000000 which only works in MySQL's non-strict mode; standard SQL requires HAVING SUM(i.amount) > 1000000.

Extending the analysis: sector-level concentration

A natural follow-on adds a window function to compute each sector's share of total capital:

SUM(i.amount) * 100.0 / SUM(SUM(i.amount)) OVER () AS pct_of_total

The nested SUM(SUM(...)) pattern — an aggregate inside a window function — computes the grand total using the group-level sum as input to an OVER() with no partition (full table). This is a senior-level pattern worth naming.

Senior signal: flag that INNER JOIN silently excludes orphan investments and offer the LEFT JOIN alternative, explain that all four aggregates are one-scan "free riders", warn about integer-AVG truncation and offer the explicit cast, and demonstrate the nested SUM(SUM(...)) OVER () pattern for computing sector-level share of total investment.
