HR wants the highest earner in each department — and if two people tie at the department's top salary, both should appear. The interviewer explicitly bans window functions, so they are testing whether you know the classic pre-window idiom: a tuple-match subquery that pairs each department with its max salary and selects the rows matching that pair.

Two edge cases. Department 1 has a tie: Bob and Cara both earn 120, and both must be returned. Department 3 is a single-member group: Finn is the only employee, so he is trivially the top earner — the query must not drop him.

2. The SQL — tuple-match against per-department maxima

SELECT dept_id, name, salary
FROM employee
WHERE (dept_id, salary) IN (
  SELECT dept_id, MAX(salary) FROM employee GROUP BY dept_id
)
ORDER BY dept_id, name;

The subquery yields one (dept_id, max_salary) pair per department. The outer WHERE (dept_id, salary) IN (…) keeps an employee only if their department-and-salary pair exactly matches one of those maxima — so every employee sitting at their department's top salary survives, ties included.

4. Strategic explanations

Why the row-value (tuple) IN, not a scalar comparison

(dept_id, salary) IN (SELECT dept_id, MAX(salary) …) compares a pair to a set of pairs. The naive scalar form WHERE salary IN (SELECT MAX(salary) FROM employee GROUP BY dept_id) is a classic bug: it would match anyone whose salary equals any department's max. If dept 2's max (80) coincidentally equalled some employee's salary in dept 1, that employee would be wrongly returned. Binding dept_id and salary together as a tuple ties the max to its own department.

Why this returns ties for free

The subquery produces one pair per department, but the outer WHERE keeps every employee row whose pair matches. Bob and Cara both have (1, 120), so both pass — no special tie handling needed. This is the behaviour the question wants. A correlated alternative — WHERE salary = (SELECT MAX(salary) FROM employee e2 WHERE e2.dept_id = employee.dept_id) — gives the identical result and also surfaces ties; both are valid pre-window idioms.

The window-function answer, and why it is banned here

The modern phrasing is RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) then filter rk = 1 — RANK (not ROW_NUMBER) so tied top earners share rank 1 and both appear. The question forbids window functions precisely to test the older tuple-match technique. A senior names the window-function solution as the production-preferred form — one table scan, no self-reference — while still delivering the requested subquery version.

NULL salaries and empty-department traps

MAX(salary) ignores NULLs, so a department where everyone has a NULL salary yields (dept_id, NULL), and (dept_id, salary) IN ((dept_id, NULL)) never matches — that department silently produces no winner (NULL is not equal to anything, even another NULL). Decide whether that is acceptable or whether such departments need surfacing separately. Also: the tuple-IN form needs the subquery's dept_id to be non-nullable to stay clean — another reason a senior reaches for the window function on messy data.

Senior signal: match the composite (dept_id, salary) tuple against per-department maxima — never a scalar salary IN (…), which leaks across departments — understand the pattern returns tied top earners for free, and name RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) as the production-preferred form the question deliberately rules out.
