A markets desk wants a 30-day moving average of each stock's closing price — the classic noise-smoothing line that sits under every price chart. The interviewer is checking one specific skill: whether you can express a sliding window with an explicit frame clause, and whether you understand what the average does at the start of the series when there are not yet 30 days of history.

Only five days of history for stock 1 — far fewer than 30. That is the deliberate edge case: the average over "the previous 30 days" when fewer than 30 exist.

2. The SQL — a windowed AVG with an explicit ROWS frame

SELECT stock_id,
    stock_date,
    price,
    avg(price) over(partition by stock_id
            order by stock_date asc
            rows between 29 preceding and current row) as moving_avg_30_days
from stock_prices;

4. Strategic explanations

Why the frame clause is the whole answer

ROWS BETWEEN 29 PRECEDING AND CURRENT ROW defines the sliding window: the current row plus the 29 rows before it — 30 rows total. Without an explicit frame, an ORDER BY in a window defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, which is a cumulative (running) average from the start of the series, not a 30-row trailing one. Stating the frame is mandatory for a true moving average.

"30 rows" is not "30 days" — the gap trap

ROWS counts rows. If a stock does not trade on weekends or holidays, 30 rows can span six calendar weeks. If the requirement is genuinely the last 30 calendar days, you need a range frame on the date itself: RANGE BETWEEN INTERVAL '29 days' PRECEDING AND CURRENT ROW. Naming this distinction — trading days vs calendar days — is the senior move; for finance the trading-day reading is usually what is wanted, but say so.

The warm-up period — partial windows are not wrong, just early

For the first 29 days the window simply contains fewer than 30 rows and AVG divides by however many it has. That is mathematically defensible but often undesirable on a chart, where an "MA30" implies 30 real data points. If the business wants the line blank until 30 days exist, gate it: CASE WHEN COUNT(*) OVER (...) = 30 THEN avg(...) END. Decide deliberately rather than letting partial windows leak out.

Duplicate dates and the ROWS vs RANGE nuance

ROWS treats each physical row independently, so two rows sharing a stock_date both occupy a slot — the window may "see" only one of a tied pair at a given step. RANGE, by contrast, pulls in all rows with peer order-by values at once. For a clean daily series with one row per stock per day this never bites, but if intraday ticks are possible, aggregate to one row per day first or the count of "days" becomes ambiguous.

What to say next — the leading bare SELECT *

The reference includes a throwaway select * from stock_prices; as an inspection step — harmless in a console, but it is not part of the answer. In an interview, present only the second statement. Mention that price is returned alongside the moving average precisely so the consumer can see raw vs smoothed side by side — the typical chart overlay.

Senior signal: name the frame clause as what makes it a moving (not cumulative) average, distinguish ROWS (30 rows) from RANGE INTERVAL (30 calendar days), and address the warm-up period explicitly — either accept partial windows or blank the line until 30 points exist.
