A finance team wants a department ranking that is robust to salary outliers — strip the highest and lowest earner from each department before computing the average, then rank departments by that trimmed mean. The interviewer is probing dual ROW_NUMBER exclusion (ascending and descending) to remove exactly one max and one min per department, handling small departments gracefully, and applying RANK over a nullable column.

1. The source data — employee

Eng (5 employees, qualifies): remove highest (200) and lowest (120), average of 180, 170, 160 = 170.00. Sales (4, qualifies): remove 300 and 150, average of 250 and 240 = 245.00. Ops (3, qualifies): remove 120 and 100, average of 110 = 110.00. HR (2, excluded — trimmed_avg = NULL).

2. The SQL — dual ROW_NUMBER exclusion with CASE WHEN for dept rank

WITH r AS (
  SELECT emp_id, dept, salary,
         ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn_desc,
         ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary ASC)  AS rn_asc,
         COUNT(*) OVER (PARTITION BY dept) AS headcount
  FROM employee
),
trimmed AS (
  SELECT dept, headcount,
         CASE WHEN headcount >= 3
              THEN ROUND(AVG(CASE WHEN rn_desc <> 1 AND rn_asc <> 1 THEN salary END)::numeric, 2)
              ELSE NULL
         END AS trimmed_avg
  FROM r
  GROUP BY dept, headcount
)
SELECT dept, headcount, trimmed_avg,
       CASE WHEN trimmed_avg IS NULL THEN NULL
            ELSE RANK() OVER (ORDER BY trimmed_avg DESC NULLS LAST)
       END AS dept_rank
FROM trimmed
ORDER BY dept;

4. Strategic explanations

Dual ROW_NUMBER for single max and min removal

Two separate ROW_NUMBER windows — one descending (highest salary gets rn_desc = 1) and one ascending (lowest salary gets rn_asc = 1) — identify the exact one row to exclude at each end. Crucially, ROW_NUMBER always assigns a unique ordinal even for ties, so when two employees share the maximum salary only one is excluded. If the question required removing all ties, RANK would be the right choice. The question says "the highest-paid and lowest-paid employee" (singular), so ROW_NUMBER is correct.

The nested CASE inside AVG

AVG(CASE WHEN rn_desc <> 1 AND rn_asc <> 1 THEN salary END) is a conditional aggregate. For excluded rows the CASE returns NULL, and AVG ignores NULLs. This pattern avoids a separate WHERE clause after a GROUP BY — everything stays in one aggregation step. The outer CASE (headcount ≥ 3) wraps this to produce NULL for small departments.

Why RANK over a nullable column needs a CASE wrapper

A plain RANK() OVER (ORDER BY trimmed_avg DESC NULLS LAST) would assign a numeric rank to HR too (it would rank last). The CASE expression WHEN trimmed_avg IS NULL THEN NULL ELSE RANK() OVER (...) makes HR's rank explicitly NULL rather than an uninformative position. This matches the requirement "departments with fewer than 3 employees produce a NULL … rank."

Edge case — all employees at the same salary in Ops

Ops has 3 employees. After removing the ROW_NUMBER-1 max (120) and ROW_NUMBER-1 min (100), the trimmed set has exactly one employee (110). AVG of one value is that value — correct. If all three had the same salary, ROW_NUMBER would still assign distinct ranks and two rows would be excluded, leaving one row at the common salary for the average.

Dialect notes

The dual ROW_NUMBER and conditional aggregate pattern is portable to PostgreSQL, Snowflake, BigQuery, and SQL Server. In BigQuery, ROUND(expr, 2) works on FLOAT64 without a cast. SQL Server uses ROUND(CAST(expr AS decimal(18,4)), 2). Snowflake follows PostgreSQL conventions. The NULLS LAST clause is ANSI SQL but missing from MySQL — in MySQL, add ISNULL(trimmed_avg) ASC as a sort key to push NULLs to the end.

Senior signal: use two ROW_NUMBER windows (DESC and ASC) to exclude exactly one max and one min per department regardless of salary ties; use a conditional AVG(CASE WHEN ... THEN salary END) rather than a post-GROUP-BY WHERE; and NULL-out the dept_rank explicitly via CASE rather than letting NULLs receive a trailing numeric rank from the window function.
