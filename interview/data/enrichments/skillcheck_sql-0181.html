Skip-level reporting — the practice of a manager meeting with employees two levels below them — is a common HR concept, and counting those skip-level reports is a useful org-chart metric. The interviewer is probing whether you can navigate a two-hop path in a self-referential adjacency-list table without a recursive CTE, using a plain double-join instead. Getting the join chain right and understanding why this is exactly two levels (not one, not three) is the crux.

2. The SQL — double-join for exactly two hops

SELECT m.emp_id AS manager_id,
       m.name AS manager_name,
       COUNT(skip.emp_id)::int AS skip_level_count
FROM employee m
JOIN employee mid ON mid.manager_id = m.emp_id
JOIN employee skip ON skip.manager_id = mid.emp_id
GROUP BY m.emp_id, m.name
ORDER BY m.emp_id;

The query aliases the employee table three times. m is the manager at the top. mid is the intermediate employee who reports directly to m. skip is the skip-level report who reports directly to mid. The chain m → mid → skip is exactly two hops. COUNT(skip.emp_id) counts skip-level employees per manager; using INNER JOIN means managers with no skip-level reports (like Carla and Erin) are excluded by the join, satisfying the "only include managers with at least one" requirement.

4. Strategic explanations

Why a double-join rather than a recursive CTE

The problem specifies exactly two hops. A recursive CTE traverses all depths and would require a depth column and a WHERE depth = 2 filter — more complex and slower for a fixed-depth query. Two INNER JOINs express the fixed depth directly and are typically optimised as a straightforward nested-loop or hash join rather than an iterative process. Use recursive CTEs for variable-depth traversal; use multiple joins for fixed-depth traversal.

INNER JOIN semantics automatically enforce "at least one"

The problem asks to include only managers with at least one skip-level report. Because all three joins are INNER JOINs, any m that has no matching mid, or any mid that has no matching skip, produces no rows in the join and is excluded from the GROUP BY output entirely. There is no need for a HAVING COUNT(…) ≥ 1 clause. If the requirement were to show all managers including zero-count ones, LEFT JOIN both joins and use COUNT(skip.emp_id) (which returns 0 for NULLs) rather than COUNT(*).

The ::int cast

In PostgreSQL, COUNT returns a bigint. The ::int cast (equivalent to CAST(COUNT(skip.emp_id) AS int)) narrows it to a 32-bit integer, which is consistent with the emp_id INT column type. This is a minor style choice but shows attention to type consistency. In most other engines (MySQL, SQL Server, BigQuery), COUNT returns a standard integer type and no cast is needed.

Adapting for exactly k levels

For exactly 3 hops (granddirect reports), add a fourth alias: JOIN employee skip2 ON skip2.manager_id = skip.emp_id. For variable depth, switch to the recursive CTE approach. The double-join idiom is elegant precisely because org-chart hop counts in HR use cases are typically small and fixed (skip-level is almost always exactly 2).

Dialect notes

The query uses no PostgreSQL-specific features beyond ::int. The core double-join with GROUP BY is portable to MySQL, SQL Server, BigQuery, Snowflake, Oracle, and SQLite. Replace ::int with CAST(… AS INT) for SQL Server and MySQL compatibility.

Senior signal: use a double INNER JOIN (not a recursive CTE) for a fixed two-hop traversal; explain that INNER JOIN implicitly filters to managers with at least one skip-level report; contrast with the LEFT JOIN + COALESCE pattern for including zero-count managers; and note the linear scalability advantage over recursive approaches for fixed-depth problems.
