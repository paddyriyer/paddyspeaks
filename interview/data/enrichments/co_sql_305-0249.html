EPAM's revenue team wants the single top-selling region for a board slide. GROUP BY, ORDER BY total DESC, LIMIT 1 — three lines, done. But "the top region" hides a question: what if two regions tie for first? The interviewer wants to see whether you notice that LIMIT 1 picks an arbitrary winner among ties, and what you would do about it.

1. The source data — sales

regionamount
West100
West150
East200
East60
North90
South300

Per-region totals: South 300, West 250, East 260, North 90. South is the clear winner here — but notice how close West and East are. If a few more East rows pushed it to 300 as well, LIMIT 1 would have to break a genuine first-place tie, and it would do so silently.

2. The SQL — GROUP BY then ORDER BY total DESC LIMIT 1

SELECT region, SUM(amount) AS total
FROM sales GROUP BY region
ORDER BY total DESC LIMIT 1;

Group by region, sum the amounts, sort descending so the biggest total floats to the top, and LIMIT 1 keeps only that first row.

3. The resulting analysis

regiontotal
South300

South wins with 300 — correct for this data, since no other region reaches it. One row, exactly what the board slide wants. The query is right; the discussion below is about the scenario it does not handle gracefully.

4. Strategic explanations

The tie trap — LIMIT 1 picks one winner arbitrarily

If South and (say) East both totalled 300, ORDER BY total DESC LIMIT 1 still returns exactly one row — and which one is non-deterministic, decided by the planner's scan order. The query would silently crown one of two tied regions. If ties must all be shown, rank instead of limiting:

SELECT region, total FROM (
  SELECT region, SUM(amount) AS total,
         RANK() OVER (ORDER BY SUM(amount) DESC) AS rk
  FROM sales GROUP BY region) t
WHERE rk = 1;

RANK() = 1 returns every region tied at the maximum — one row normally, several on a tie. Stating "do you want one region, or all regions tied for first?" before writing SQL is the senior move.

The HAVING-with-subquery alternative

Another tie-safe form keeps every region whose total equals the maximum total:

SELECT region, SUM(amount) AS total
FROM sales GROUP BY region
HAVING SUM(amount) = (
  SELECT MAX(t.total) FROM (
    SELECT SUM(amount) AS total FROM sales GROUP BY region) t);

It is wordier than the RANK version and aggregates the data twice, but it makes the "equals the max" logic explicit. HAVING filters after grouping — you cannot use a plain WHERE on an aggregate.

Why a deterministic tiebreaker matters even without ties

Even when there is no tie on the metric, LIMIT queries should be deterministic for reproducibility. Add a secondary sort key — ORDER BY total DESC, region — so that if a tie ever does occur, the same region wins every run instead of flickering between executions. Non-deterministic top-N queries are a classic source of "the dashboard changed but the data didn't" bugs.

NULL amounts and regions with no sales

SUM ignores NULL amounts, so a region with some unpriced rows is still totalled on its priced ones. A region whose every amount is NULL gets SUM = NULL; under ORDER BY ... DESC Postgres sorts NULLs first, so such a region could wrongly land at the top — guard with NULLS LAST or filter NULLs out. And a region with no rows in sales never appears at all; that is fine for "top region", but worth noting.

Returning the runner-up too

Boards often ask for the runner-up in the same breath. Generalise to top-N by dropping LIMIT for a window function: ORDER BY total DESC LIMIT 3 gives the top three but is still tie-blind, whereas RANK() OVER (ORDER BY SUM(amount) DESC) with rk <= 3 handles ties at every position. Designing for the inevitable follow-up signals seniority.

Senior signal: recognise that ORDER BY ... DESC LIMIT 1 silently picks one arbitrary winner when regions tie for first, offer RANK() = 1 or a HAVING = MAX form when all ties must show, add a deterministic tiebreaker for reproducibility, and handle NULL totals with NULLS LAST.
