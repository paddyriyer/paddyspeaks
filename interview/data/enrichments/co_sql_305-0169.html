Cognizant inherited a regional sales table built for a spreadsheet: one column per region — north, south, east, west — sitting side by side. The new BI tool wants the data long: one row per (product, region). This is an unpivot, and the interviewer wants to see whether you reach for the portable UNION ALL idiom and whether you understand what it does to a NULL cell.

1. The source data — regional_sales

productnorthsoutheastwest
Widget1008012090
Gadget50NULL7060
Gizmo200150180210

Three products across four region columns. Gadget has no South figure — the cell is NULL. That single NULL is the interview: it decides whether the long table gets a Gadget/South row at all, and what value it carries.

2. The SQL — UNION ALL unpivot

SELECT product, 'North' AS region, north AS amount FROM regional_sales
UNION ALL SELECT product, 'South', south FROM regional_sales
UNION ALL SELECT product, 'East',  east  FROM regional_sales
UNION ALL SELECT product, 'West',  west  FROM regional_sales;

One SELECT per region column, each emitting a literal region label and that column as amount. UNION ALL stacks the four result sets into one tall table.

Twelve rows — 3 products × 4 regions. The Gadget/South row still exists, carrying amount = NULL. The unpivot is structural: it produces a row per (product, region) cell regardless of whether the cell had a value.

4. Strategic explanations

UNION ALL, never UNION — the silent-row-loss trap

UNION deduplicates; UNION ALL does not. If two products had identical (region, amount) — say two products both with North = 100 — plain UNION would collapse them into one row and you would lose data. Worse, it triggers a sort/hash to find duplicates, so it is slower for no benefit. Unpivot is a structural reshape: every source cell must survive. Always UNION ALL.

Should the NULL row be kept or dropped?

That is a business question. Keeping Gadget/South as NULL faithfully records "no South sales data for Gadget". If the consumer wants only rows that represent real sales, filter the result:

SELECT * FROM ( … the UNION ALL … ) u
WHERE amount IS NOT NULL;

Do not assume — ask whether a missing cell should appear as a NULL row or vanish. And note: a NULL here is genuinely different from a zero. NULL means "unknown / not recorded"; 0 means "we sold none".

Postgres native: LATERAL + VALUES — one table scan

The UNION ALL form scans regional_sales four times. A tidier, single-scan alternative uses a lateral join:

SELECT r.product, v.region, v.amount
FROM regional_sales r
CROSS JOIN LATERAL (VALUES
  ('North', r.north), ('South', r.south),
  ('East',  r.east),  ('West',  r.west)
) AS v(region, amount);

It reads the table once and expands each row into four. Same result, better plan on a large table. (Other engines have a dedicated UNPIVOT operator; UNION ALL is the portable answer that runs everywhere.)

Column types must line up

Each branch of the UNION ALL must have compatible types in matching positions. Here all four region columns are numeric, so amount is clean. If one region column were text and another numeric, the union would either error or coerce to text — cast explicitly so the result type is intentional, not whatever the planner negotiates.

Adding a fifth region later

This pattern hard-codes the region list. Add a central column and you must add a fifth SELECT — the query does not adapt on its own. That is the inherent cost of unpivoting wide tables in SQL; the real fix is upstream, storing the data long in the first place so reshaping is never needed.

Senior signal: use UNION ALL (not UNION, which silently drops duplicate rows), recognise that the unpivot keeps a NULL cell as a NULL row and decide deliberately whether to filter it, offer the single-scan LATERAL VALUES form, and note that the hard-coded region list is a maintenance cost best solved by storing data long upstream.
