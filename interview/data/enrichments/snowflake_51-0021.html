The growth team's monthly question: "For each customer, how does their relationship with our top SKUs evolve? When did they buy this product the first time? The third time? What's the gap between purchases? What's their cumulative spend per product?" All four metrics drop out of one query that stacks four window functions on the same partition.

2. The SQL — four window functions, one partition

SELECT
  customer_id, product_id, purchase_ts, amount,

  -- 1. Which-purchase-number is this for this (customer, product)?
  ROW_NUMBER() OVER (
    PARTITION BY customer_id, product_id
    ORDER BY purchase_ts
  ) AS purchase_no,

  -- 2. Timestamp of the previous purchase of this product by this customer
  LAG(purchase_ts) OVER (
    PARTITION BY customer_id, product_id
    ORDER BY purchase_ts
  ) AS prev_ts,

  -- 3. Days since the last purchase (NULL on first)
  DATEDIFF(day,
    LAG(purchase_ts) OVER (PARTITION BY customer_id, product_id ORDER BY purchase_ts),
    purchase_ts
  ) AS days_since_last,

  -- 4. Cumulative spend on this product by this customer over time
  SUM(amount) OVER (
    PARTITION BY customer_id, product_id
    ORDER BY purchase_ts
    ROWS UNBOUNDED PRECEDING
  ) AS cumulative_spend
FROM purchases;

4. Strategic explanations

One PARTITION, four functions — the engine de-duplicates the sort

The partition (customer_id, product_id) ORDER BY purchase_ts appears in all four window functions. Modern engines (Postgres, Snowflake, BigQuery) compute the partition once and run all four functions in a single pass — not 4× the work. SQLite is the same. So this 4-window query is roughly the cost of the most expensive single window.

Why (customer_id, product_id) as the partition

The metric is per-customer-per-product. If you partitioned only by customer_id, the cumulative spend would mix products together; if only by product_id, it'd mix customers. The composite partition isolates each (customer × product) timeline cleanly. This is the most-asked variant in interviews — composite partitions with a time order.

The ROWS UNBOUNDED PRECEDING frame — explicit beats implicit

By default, SUM() OVER (... ORDER BY ...) uses RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. ROWS UNBOUNDED PRECEDING is the equivalent in row-counting mode (faster on most engines, identical answer when the order column has no ties). Always state the frame explicitly — implicit defaults are a portability gotcha.

Dialect note: DATEDIFF

The query uses DATEDIFF(day, prev, curr) — Snowflake / SQL Server / Oracle syntax. Other dialects:

  • Postgres / BigQuery: DATE_DIFF(curr, prev, DAY) or (curr - prev) for date-typed columns
  • SQLite: JULIANDAY(curr) - JULIANDAY(prev)
  • MySQL: DATEDIFF(curr, prev) — note: NOT 3 args
Related metrics that drop out of the same shape
  • First-purchase date: filter WHERE purchase_no = 1 — gives you each customer's first encounter with each product. Acquisition channel attribution starts here.
  • Inter-purchase interval (mean): AVG(days_since_last) GROUP BY customer_id, product_id — the customer's natural cadence per product. Re-engagement campaigns trigger when the gap exceeds 2× this mean.
  • "Bought this 5+ times" cohort: WHERE purchase_no >= 5 — high-affinity customers per product. Brand loyalty signal.
  • Lifetime value per (customer × product): WHERE purchase_no = MAX(purchase_no) OVER (PARTITION BY customer_id, product_id) — final cumulative_spend = lifetime spend on this SKU.
The over-the-bar finish: cohort retention by product

Add a NTILE(4) over time-since-first-purchase to bucket customers into "first-quarter buyers, second-quarter buyers, …" — feeds product-cohort retention curves directly:

NTILE(4) OVER (
  PARTITION BY product_id
  ORDER BY MIN(purchase_ts) OVER (PARTITION BY customer_id, product_id)
) AS acquisition_quartile
Senior signal: name the composite-partition pattern, explain why all four windows in one pass cost almost the same as one (engine de-duplicates the sort), state the frame explicitly (ROWS UNBOUNDED PRECEDING), and pre-empt the dialect difference for DATEDIFF. The cohort-retention extension is the over-the-bar move.
