BMW's customer-analytics team wants a one-row-per-customer summary — how many orders each placed and how much they spent. A textbook GROUP BY. The interviewer is not testing whether you can write it; they are testing whether you know the difference between COUNT(*) and COUNT(column), and what happens to the summary when an order has a NULL amount.

1. The source data — orders

Customer 103 has two orders, but one carries a NULL amount (a pending order not yet priced). That NULL is the whole interview — it splits COUNT(*) from COUNT(amount), and it determines whether the spend total is a number or NULL.

2. The SQL — GROUP BY customer

SELECT customer_id, COUNT(*) AS orders, SUM(amount) AS total
FROM orders GROUP BY customer_id;

One group per customer_id. COUNT(*) tallies rows in the group; SUM(amount) adds their amounts.

Customer 103 shows orders = 2 but total = 50. COUNT(*) counted both rows; SUM(amount) skipped the NULL and added only the 50. The two columns disagree by design — that is the point. Note also the rows come back in no particular order; a GROUP BY does not sort its output.

4. Strategic explanations

COUNT(*) vs COUNT(amount) — rows vs values

COUNT(*) counts rows regardless of NULLs. COUNT(amount) counts only rows where amount is non-NULL. For customer 103 that is 2 vs 1. Which one is "number of orders"? Almost always COUNT(*) — a pending order is still an order. But if the metric is "number of priced orders", COUNT(amount) is right. Say which you mean; the choice is a business decision, not a syntax preference.

SUM over a group of all-NULLs returns NULL, not 0

SUM ignores NULLs — here that quietly drops 103's pending order. Harmless when at least one row has a value, but if a customer's every order were unpriced, SUM(amount) returns NULL, not 0. A dashboard expecting a number then shows a blank. Guard it:

SELECT customer_id, COUNT(*) AS orders,
       COALESCE(SUM(amount), 0) AS total
FROM orders GROUP BY customer_id;

COALESCE wraps the aggregate, not the column — SUM(COALESCE(amount,0)) also works and additionally makes a partly-NULL group treat the NULL row as a $0 order, which may or may not be what you want.

The "every column not aggregated must be grouped" rule

You can SELECT customer_id because it is in the GROUP BY. Adding order_id or status to the select list without aggregating them is a syntax error in Postgres — the engine cannot pick one order_id for a group of several. If you need a representative value, aggregate it explicitly: MAX(order_date) AS last_order, STRING_AGG(status, ','), etc.

Customers with zero orders never appear

This query only sees customers who have at least one row in orders. A customer who registered but never ordered is simply absent — not shown as orders = 0. If the report must include them, start from the customers table and LEFT JOIN:

SELECT c.customer_id, COUNT(o.order_id) AS orders,
       COALESCE(SUM(o.amount), 0) AS total
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.customer_id
GROUP BY c.customer_id;

Crucially, count o.order_id here, not * — an unmatched customer produces one row of NULLs from the join, and COUNT(*) would wrongly report 1 order for them.

Should cancelled orders count?

Customer 102's only order is cancelled but still contributes to both the count and the total. "Total spent" usually means realised revenue, which excludes cancellations — add WHERE status <> 'cancelled' (or filter inside the aggregates with SUM(amount) FILTER (WHERE status = 'completed') if you want both the gross order count and net spend in one row).

Senior signal: articulate COUNT(*) (rows) versus COUNT(col) (non-NULL values) and pick the one the metric demands, wrap SUM in COALESCE so an all-NULL group reports 0 not NULL, switch to a LEFT JOIN with COUNT(o.order_id) when zero-order customers must appear, and question whether cancelled orders belong in the totals.
