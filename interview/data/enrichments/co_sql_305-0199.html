Tesla's battery-test rig writes a stream of sensor measurements, each with an auto-incrementing measurement_id. An engineer wants the total for odd-numbered readings versus even-numbered ones — a quick way to check the rig's two alternating sample channels are balanced. The query is a single-pass conditional aggregation, and the interviewer is checking whether you can pivot with CASE instead of two separate queries.

Seven readings, but measurement 6 has a NULL value — a sensor dropout. It is even-numbered, so it lands in the even bucket. How SUM treats that NULL is the subtle part of the question.

2. The SQL — conditional SUM with CASE

SELECT
  SUM(CASE WHEN measurement_id % 2 = 1 THEN value ELSE 0 END) AS odd_sum,
  SUM(CASE WHEN measurement_id % 2 = 0 THEN value ELSE 0 END) AS even_sum
FROM measurements;

Each CASE picks value for rows on its side of the parity test and 0 for the rest. SUM then totals each expression. One pass over the table produces both buckets — no self-join, no two queries, no GROUP BY.

3. The resulting analysis

Odd: 10.5 + 5.5 + 12.0 + 3.0 = 31.0 (ids 1, 3, 5, 7). Even: 20.0 + 8.0 = 28.0 (ids 2, 4 — id 6's NULL contributed nothing). Two scalars, one row. Notice the NULL did not blow up the even total or turn it NULL — that is the behaviour worth being able to explain.

4. Strategic explanations

Why the NULL row vanished cleanly

For measurement 6, CASE WHEN measurement_id % 2 = 0 THEN value ELSE 0 END evaluates to value, which is NULL. SUM ignores NULL inputs, so that row simply does not contribute — even_sum is 20.0 + 8.0 = 28.0. The ELSE 0 only fires for odd rows; it does not "fix" the NULL. If a dropout reading should count as 0 explicitly, wrap it: SUM(CASE WHEN ... THEN COALESCE(value,0) ELSE 0 END) — same result here, but it documents the intent and matters if you later divide by a count.

The negative-modulo trap — % on signed integers

This works only because measurement_id is positive. In Postgres the % operator follows the sign of the dividend: -3 % 2 = -1, not 1. If IDs could ever be negative, measurement_id % 2 = 1 would miss the odd negatives. The robust parity test is ABS(measurement_id) % 2 = 1, or measurement_id % 2 <> 0 for "odd". For an auto-increment surrogate key it is academic — but say it, because the interviewer may be probing exactly that.

The FILTER clause — the modern idiom

Postgres offers a cleaner way to write conditional aggregates — FILTER:

SELECT
  SUM(value) FILTER (WHERE measurement_id % 2 = 1) AS odd_sum,
  SUM(value) FILTER (WHERE measurement_id % 2 = 0) AS even_sum
FROM measurements;

This reads better and has a real semantic difference: FILTER excludes non-matching rows entirely, so there is no ELSE 0 padding. On an empty side FILTER yields NULL where the CASE form yields 0 — usually you want that distinction visible. Mentioning FILTER signals current-Postgres fluency; the CASE form is the portable fallback for engines without it.

Edge cases: empty table and integer values

On an empty table the CASE version returns one row of (0, 0) — because SUM over zero non-NULL inputs is NULL, but every row that would exist contributes 0... actually no: with zero rows there is nothing to sum, so SUM returns NULL and the result is (NULL, NULL). Wrap in COALESCE(SUM(...), 0) if a dashboard needs zeros. If value were an INTEGER column the sums stay integers — fine here, but cast to numeric if downstream math needs decimals.

Generalising to N buckets

Two buckets via two CASE columns is fine. If the ask grew to "sum by measurement_id % 5", do not write five columns — pivot to rows with a GROUP BY: SELECT measurement_id % 5 AS bucket, SUM(value) FROM measurements GROUP BY 1. The conditional-column trick is for a small, fixed set of buckets you want side by side.

Senior signal: reach for conditional aggregation (CASE inside SUM, or FILTER) to produce both buckets in one pass, explain that SUM silently skips the NULL reading, flag the signed-modulo gotcha, and know that FILTER returns NULL on an empty bucket where CASE ... ELSE 0 returns 0.
