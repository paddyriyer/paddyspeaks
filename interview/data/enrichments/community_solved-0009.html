A Community routes question: A→B and B→A describe the same trip, so collapse them and report the average distance per undirected route. Here the distance column is a clean numeric, which makes the question almost entirely about one idea — building a direction-independent key so the two halves of a route group together.

1. The source data — locations

sourcedestinationdistance
MumbaiDelhi1400
DelhiMumbai1390
BangaloreChennai350
ChennaiBangalore360
HyderabadPune560

Mumbai–Delhi and Bangalore–Chennai each appear in both directions, with slightly different recorded distances — the same road, measured twice. Hyderabad–Pune appears only once: a single-direction route. Both shapes must come out right.

2. The SQL — canonical route key, then average

SELECT
    CASE WHEN source > destination THEN concat(source, '-', destination)
         ELSE concat(destination, '-', source) END AS unique_route,
    AVG(NULLIF(regexp_replace(distance::text, '[^-.0-9]', '', 'g'), '')::numeric) AS average_distance
FROM locations
GROUP BY CASE WHEN source > destination THEN concat(source, '-', destination)
              ELSE concat(destination, '-', source) END;

The CASE places the alphabetically-larger city name first, so both Mumbai→Delhi and Delhi→Mumbai produce the identical key Mumbai-Delhi and land in one group. AVG then averages the distances within each group.

3. The resulting analysis

unique_routeaverage_distance
Mumbai-Delhi1395.0000000000000000
Pune-Hyderabad560.0000000000000000
Chennai-Bangalore355.0000000000000000

Five rows collapse to three routes. Mumbai-Delhi averages 1400 and 1390 to 1395; Chennai-Bangalore averages 350 and 360 to 355. Pune-Hyderabad had only one row, so its "average" is just 560. The long decimal tail is only the numeric display type — round it for a report.

4. Strategic explanations

The canonical key — the one idea this question tests

"A→B equals B→A" means you must group on an unordered pair. SQL groups on values, so you manufacture a value that is the same regardless of direction: sort the two endpoints. The CASE does it; Postgres does it more tersely with LEAST/GREATEST:

concat(LEAST(source, destination), '-', GREATEST(source, destination))

One expression instead of the four-line CASE repeated in SELECT and GROUP BY. You can also write GROUP BY 1 to point at the select-list position and avoid restating the key entirely.

Why the distance is wrapped in regexp_replace even though it is numeric

Here distance is a clean integer, so regexp_replace(distance::text, ...) is a no-op — defensive plumbing for the case where the column is actually dirty text like '350 km'. NULLIF(..., '') then turns any value that scrubs down to nothing into NULL instead of crashing on ''::numeric. On genuinely numeric data you can drop the whole wrapper and just write AVG(distance) — mention that you would, to show you are not cargo-culting.

The reverse-pair distances disagree — is averaging right?

Mumbai–Delhi is recorded as both 1400 and 1390. The query happily averages them to 1395, but that hides a data-quality issue: the same road should have one length. A senior surfaces it:

SELECT unique_route, MIN(distance) lo, MAX(distance) hi
FROM locations GROUP BY 1 HAVING MIN(distance) <> MAX(distance);

If the directions should always agree, the right move may be MIN or "flag for review", not AVG. If they can genuinely differ (one-way detours), averaging is fine — but that is a decision, not a default.

Edge cases: lone routes, NULLs, self-routes

A single-direction route averages to its one value — correct. A NULL distance is skipped by AVG (both numerator and denominator), so a route with one real reading and one NULL reports the real reading. A degenerate source = destination row keys as City-City via the ELSE branch — filter it with WHERE source <> destination if such rows are meaningless.

The simpler alternative: UNION the directions

Instead of a sort key, you can normalise by stacking both orientations and deduping — but for averaging, the GROUP BY on a canonical key is fewer moving parts and one pass. Reach for UNION only when you need the symmetric edge list itself.

Senior signal: build the direction-independent key with LEAST/GREATEST rather than a duplicated CASE, note that the regexp_replace wrapper is defensive and drop it on truly numeric data, and treat reverse-pair distances that disagree as something to surface rather than silently average.
