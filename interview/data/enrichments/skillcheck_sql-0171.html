Market-basket analysis — finding which items are frequently purchased together — is foundational to e-commerce recommendation engines and retail planogramming. The interviewer is probing the self-join with a canonical-pair guard pattern: joining a table to itself on basket_id while enforcing a.item < b.item to produce each unordered pair exactly once, then aggregating to find frequency. Getting the deduplication logic right and understanding why < rather than ≠ is the crux.

The top pair by frequency should be bread-milk at 4 occurrences.

2. The SQL — self-join with canonical-pair guard

SELECT a.item AS item_a, b.item AS item_b, COUNT(*) AS frequency
FROM basket_item a JOIN basket_item b
  ON a.basket_id = b.basket_id AND a.item < b.item
GROUP BY a.item, b.item
ORDER BY frequency DESC, a.item, b.item
LIMIT 3;

The table joins to itself on basket_id to generate all co-occurrence pairs within the same basket. The a.item < b.item guard does two things simultaneously: it eliminates self-pairs (an item paired with itself) and it fixes the canonical ordering so (A, B) and (B, A) collapse into the single row where item_a < item_b alphabetically. COUNT(*) counts baskets containing that pair; the result is ordered by descending frequency, then alphabetically for deterministic tie-breaking.

4. Strategic explanations

Why a.item < b.item rather than a.item ≠ b.item

≠ eliminates self-pairs but still emits both (bread, milk) and (milk, bread) as separate rows, each with their own count, requiring a later LEAST/GREATEST re-canonicalisation or a HAVING item_a < item_b post-filter that throws away half the work. The < predicate does the canonicalisation at join time: because string comparison is total and antisymmetric, exactly one of a.item < b.item and b.item < a.item is true for any two distinct items, so each unordered pair appears in precisely one join row. This is cleaner, cheaper, and harder to mis-implement.

What COUNT(*) is counting

COUNT(*) counts the number of rows in each (a.item, b.item) group after the self-join. Because each row in the self-join corresponds to one basket where both items appear, it is counting baskets — which is exactly the definition of support in market-basket analysis. If the same item appeared twice in one basket row (e.g. a basket listing ‘milk’ twice), self-join would inflate counts; the schema here has one row per (basket, item) so this is not a problem, but noting it shows depth.

Scaling beyond two items — the apriori principle

Pair mining with a self-join is O(k²) in the number of distinct items k per basket. For large catalogues (thousands of SKUs), the apriori algorithm prunes the candidate space: any frequent pair must have both members individually frequent. In SQL this means pre-filtering to items above a support threshold before the self-join, dramatically shrinking k. For triple-item sets, a second self-join or a recursive approach is needed; most production systems move to specialised libraries (Spark MLlib FPGrowth) beyond pairs.

Tie-breaking with secondary sorts

The ORDER BY frequency DESC, a.item, b.item is essential for deterministic LIMIT results. Without it, any row among the four frequency-1 pairs could appear as the third result — the output would be correct by the frequency criterion but non-deterministic across runs and engines. In interviews, always ask “how should ties be broken?” and encode the answer in the ORDER BY.

Dialect notes

The a.item < b.item string comparison relies on the database's default collation for alphabetical ordering, which is consistent across PostgreSQL, MySQL, SQL Server, BigQuery, and Snowflake for ASCII text. LIMIT 3 is PostgreSQL/MySQL/SQLite syntax; SQL Server uses TOP 3 in the SELECT clause; BigQuery and Snowflake also accept LIMIT. The rest of the query is fully portable.

Senior signal: use a.item < b.item — not ≠ — to deduplicate unordered pairs at join time; know that this produces each pair exactly once with O(items²) cost per basket; name support as the market-basket metric being computed; and mention the apriori pre-filter and FPGrowth as the scalable path for large catalogues.
