A social-graph follow table is one of the most read-heavy schemas in existence: Twitter-style products perform billions of edge lookups per day. The interviewer is probing whether you understand bidirectional index design (you need one index per direction of traversal), idempotent upserts under concurrency (ON CONFLICT DO NOTHING is the canonical pattern), a self-referential integrity constraint (no self-follows), and why storing the graph in a relational table — versus exporting to a dedicated graph database — is appropriate for many workloads.

1. The requirements — what the design must capture

  • Directed edge: following is asymmetric — A follows B does not imply B follows A; the table stores one row per directed edge.
  • Both traversal directions: "who does user X follow?" (forward: filter on follower_id) and "who follows user X?" (reverse: filter on followee_id) must both be fast.
  • Idempotent upserts: mobile and distributed systems frequently send duplicate follow events; the INSERT ... ON CONFLICT DO NOTHING pattern is the standard solution.
  • No self-follows: a CHECK (follower_id <> followee_id) constraint enforces this at the database level without application code.
  • Audit provenance: source records where the edge originated (in-app button, import, API) for audit and abuse analysis.
  • Bulk export: the table's flat structure (every row is a complete edge) makes it trivial to dump to a Parquet file or stream to a graph database like Neo4j or Neptune.

2. The DDL — the CREATE TABLE

CREATE TABLE follow_edge (
  follower_id     BIGINT       NOT NULL,                    -- the one doing the following
  followee_id     BIGINT       NOT NULL,                    -- the one being followed
  created_at      TIMESTAMPTZ  NOT NULL DEFAULT now(),
  source          VARCHAR(20)  NOT NULL DEFAULT 'app',      -- audit: app / import / api
  PRIMARY KEY (follower_id, followee_id),
  CHECK (follower_id <> followee_id)                        -- no self-follows
);

-- Reverse-direction lookup: followers of X.
CREATE INDEX ix_follow_by_followee
  ON follow_edge (followee_id, follower_id);

4. Strategic explanations

Why two indexes cover both traversal directions

The PK (follower_id, followee_id) covers the forward direction: "give me all followees of user X" is a range scan on the leading column. The reverse index (followee_id, follower_id) covers the backward direction: "give me all followers of user X." Without the reverse index, that query would require a full-table scan or a sequential scan through every row where followee_id = X. Critically, the reverse index includes follower_id as the second column, making it a covering index for the common query SELECT follower_id FROM follow_edge WHERE followee_id = ? — no heap access needed.

Idempotent upserts under heavy concurrency

INSERT ... ON CONFLICT (follower_id, followee_id) DO NOTHING is the correct pattern. It is atomic: two concurrent workers racing to insert the same edge will both succeed from the application's perspective — one inserts, the other receives a no-op — with no lost updates and no serialization errors. The alternative, a read-before-write check-then-insert, has a TOCTOU race. DO NOTHING is preferred over DO UPDATE here because a follow edge has no mutable fields worth updating on a re-insert.

The CHECK constraint for self-follows

A CHECK (follower_id <> followee_id) is evaluated on every INSERT and UPDATE by the database engine before the row is written. This is strictly stronger than application-level validation: a database migration script, a direct admin insert, or a bug in the API layer cannot accidentally create a self-follow row. It costs nothing at query time.

Fact-table type: a factless fact table

This is a classic factless fact table: the existence of a row is the fact (the follow relationship exists), and there are no numeric measures. The grain is one row per directed follow relationship. This maps cleanly to the relational model and is straightforward to export. Aggregations like follower count are computed at query time (COUNT(*) WHERE followee_id = X); if these counts are performance-critical, they are denormalized into a separate user_stats table and maintained via triggers or event-driven updates.

Scaling considerations and graph databases

For networks with billions of users, a relational follow table handles 1-hop queries (followers of X, followees of X) efficiently. It struggles with multi-hop traversal: "friends of friends," "shortest path," and "mutual followers" all require recursive CTEs or application-side BFS, which do not scale past 2-3 hops. At that point, bulk export to a dedicated graph engine (Neo4j, Amazon Neptune, Meta's TAO) is the right architectural move. The flat relational schema makes that export trivial — a full-table scan produces the edge list.

Senior signal: name both traversal directions explicitly and provide an index for each, make the reverse index covering by including follower_id in its second position, use ON CONFLICT DO NOTHING for atomic idempotent upserts rather than read-before-write, and articulate where a relational follow table breaks down (multi-hop traversal) and when to graduate to a graph engine.
