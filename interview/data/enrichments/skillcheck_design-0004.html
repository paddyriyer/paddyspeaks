A bridge table is the dimensional modeling solution to many-to-many (M:N) relationships between a fact table and a dimension. Here, a clinical visit can carry multiple diagnoses and a diagnosis applies to many visits — a many-to-many that cannot be collapsed into the fact row. The interviewer is probing whether you know when to introduce a bridge, how to define its composite PK, the semantics of the weight (apportionment) column, and the indexing needed to join efficiently from both sides.

1. The requirements — what the design must capture

  • M:N resolution: one visit maps to many diagnoses; one diagnosis appears in many visits.
  • Composite PK: (visit_sk, diagnosis_sk) — each pair appears at most once.
  • Rank: distinguishes primary from secondary diagnoses without a separate Boolean column for every position.
  • Primary flag: explicit Boolean so queries for primary diagnoses need no rank-comparison arithmetic.
  • Weight: an apportionment factor (0–1, sums to 1.0 per visit) so numeric fact measures (charges, RVUs) can be allocated across diagnoses in proportion.
  • Provenance: who coded the diagnosis and when — audit trail for clinical and billing compliance.
  • Index on diagnosis side: to support queries that start from a diagnosis (“all visits with ICD code X”) without a full bridge scan.

2. The DDL — the CREATE TABLE

CREATE TABLE bridge_visit_diagnosis (
  visit_sk         BIGINT       NOT NULL,           -- FK -> fact_clinical_visit
  diagnosis_sk     BIGINT       NOT NULL,           -- FK -> dim_diagnosis
  diagnosis_rank   SMALLINT     NOT NULL,           -- 1 = primary, 2 = secondary, ...
  is_primary       BOOLEAN      NOT NULL DEFAULT FALSE,
  weight           DECIMAL(6,4) NOT NULL DEFAULT 1.0, -- apportionment factor (sums to 1.0 per visit)
  coded_by         VARCHAR(64),                       -- ICD coder / system
  coded_at         TIMESTAMP,
  ingested_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (visit_sk, diagnosis_sk)
);

CREATE INDEX ix_bridge_diagnosis ON bridge_visit_diagnosis (diagnosis_sk);

4. Strategic explanations

Why a bridge table rather than a multi-valued fact column

Some designers are tempted to add diagnosis_sk_1, diagnosis_sk_2, … diagnosis_sk_N columns to the fact table. This breaks down immediately: N is unbounded (some visits have 15 diagnoses), most columns will be NULL, and adding a diagnosis requires an ALTER TABLE. A bridge table is the correct relational pattern: it has one row per (visit, diagnosis) pair and scales to any cardinality without schema changes.

Grain and composite PK

The grain of the bridge is one row per (visit, diagnosis) pair. The composite primary key (visit_sk, diagnosis_sk) enforces uniqueness at the database level — a given diagnosis can appear at most once per visit. If the business allows the same diagnosis code to be recorded twice (e.g., with different rank or coder), the PK would need to include diagnosis_rank.

The weight (apportionment) column

When a visit has three diagnoses with weights 0.5, 0.3, and 0.2, a charge of $1,000 can be attributed as $500, $300, and $200 across the respective diagnosis codes. This is critical in clinical billing, where payers reimburse by diagnosis. The default value of 1.0 means a single-diagnosis visit apportions 100% of its measures without special-casing. Equal apportionment (1/N) is another common default. The constraint that weights sum to 1.0 per visit should be enforced by an ETL-level assertion or a per-group CHECK (not natively supported in most engines, but testable in dbt).

Querying through the bridge

To roll up visit charges by diagnosis, the pattern is: JOIN bridge_visit_diagnosis b ON b.visit_sk = f.visit_sk JOIN dim_diagnosis d ON d.diagnosis_sk = b.diagnosis_sk, then SUM(f.charge_amount * b.weight). The weight multiplication ensures apportioned measures are used when grouping by diagnosis, rather than double-counting the full charge against every linked diagnosis.

Index strategy

The composite PK creates an index on (visit_sk, diagnosis_sk), which efficiently supports joins starting from the fact table (“given a visit, find its diagnoses”). The explicit ix_bridge_diagnosis index on diagnosis_sk supports the reverse direction (“given a diagnosis code, find all visits”) — this is the typical epidemiology query pattern and would be a full bridge scan without the index.

Star vs. snowflake considerations

The bridge table sits between the fact and the dimension; it is not itself a fact or a dimension. In a strict star schema the fact table would directly reference dimensions. The bridge adds a layer but preserves the star’s query simplicity by making the join pattern predictable: fact → bridge → dimension, always. This is sometimes called a fact constellation pattern when multiple fact tables share the bridge.

Senior signal: justify the bridge table by explaining why multi-valued FK columns on the fact are untenable, define the composite PK correctly as (visit_sk, diagnosis_sk), explain the weight apportionment pattern with the SUM(measure * weight) idiom and the 1.0-sum invariant, and mention that the reverse-direction index on diagnosis_sk is what makes diagnosis-first queries performant — these moves show you understand the bridge not just as a schema artifact but as a query-time mechanism.
