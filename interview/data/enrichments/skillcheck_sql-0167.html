A catalog team stores categories as a tree (each category points at its parent) and wants every product under "Electronics" — including products filed in deep sub-categories like Phones or Laptops. The interviewer is probing the recursive CTE: can you walk a parent/child hierarchy of unknown depth, collect the whole subtree of category ids, and then join products onto it?

The edge cases: product 104 is filed directly on Electronics (the "direct" half of "directly or indirectly"), and the Apparel branch is a sibling subtree that must not leak in — the recursion must stop at the right boundary.

2. The SQL — recursive subtree walk

WITH RECURSIVE elec AS (
  SELECT category_id FROM category WHERE name = 'Electronics'
  UNION ALL
  SELECT c.category_id FROM category c JOIN elec e ON c.parent_id = e.category_id
)
SELECT p.product_id, p.name
FROM product p
JOIN elec e ON e.category_id = p.category_id
ORDER BY p.product_id;

The anchor seeds the CTE with the Electronics category itself. The recursive term repeatedly finds categories whose parent_id is already in elec — pulling in children, then grandchildren, until nothing new appears. The final join keeps products whose category_id is anywhere in that collected subtree.

4. Strategic explanations

Anatomy of the recursive CTE

A WITH RECURSIVE CTE has two halves joined by UNION ALL. The anchor (before the UNION) runs once and supplies the starting rows — here, the single Electronics category. The recursive term (after the UNION) references the CTE name itself; the engine runs it repeatedly, each iteration seeing only the rows produced by the previous iteration, appending results until an iteration produces zero new rows. That fixpoint is what lets one query traverse a tree of arbitrary, unknown depth — something a fixed number of JOINs cannot do.

Why the anchor handles the "directly" case for free

The phrase is "directly or indirectly". The anchor SELECT category_id ... WHERE name = 'Electronics' puts Electronics' own id into elec, so a product filed straight onto Electronics (Charger) joins and is returned — that is the "directly" half. The recursive term then adds every descendant id, covering "indirectly". One CTE, both halves, no UNION with a separate direct-match query needed.

Direction of the join matters — descendants, not ancestors

The recursive join is c.parent_id = e.category_id: "find category c whose parent is something we already have." That walks downward — from Electronics to its children. Flip it to c.category_id = e.parent_id and you would walk upward toward the root instead, collecting ancestors. Getting this direction right is the single most common recursive-CTE mistake; state out loud which way you are traversing.

Cycle safety — UNION ALL vs UNION

This query uses UNION ALL, which does not deduplicate. That is fine for a true tree where each category has exactly one parent — no node is reachable twice. But if the data were corrupted into a cycle (A's parent is B, B's parent is A), UNION ALL would loop forever. Defenses: switch to UNION (deduplicates, so a revisited node stops the branch), add an explicit visited-path array and exclude already-seen ids, or use PostgreSQL 14+'s CYCLE clause. Mentioning cycle protection unprompted — even though this data is a clean tree — is the senior signal.

Dialect notes

WITH RECURSIVE is SQL-standard — PostgreSQL, MySQL 8+, SQLite, and recent MariaDB use exactly this syntax. SQL Server writes WITH (the RECURSIVE keyword is omitted, recursion is implicit) and caps recursion at 100 levels unless you add OPTION (MAXRECURSION n). Oracle supports the standard form and also a legacy proprietary CONNECT BY ... START WITH hierarchical syntax. The pattern — anchor, recurse on the parent link, join facts to the collected ids — is universal.

Senior signal: walk a category tree with a WITH RECURSIVE CTE whose anchor seeds the target node (covering the "direct" case) and whose recursive term joins child.parent_id = collected.id to descend, be deliberate about traversal direction, and call out cycle protection (UNION or a visited set) even when the data is a clean tree.
