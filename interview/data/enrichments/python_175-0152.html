Amazon asks "Walls and Gates" to test whether you recognise a multi-source shortest-path problem in disguise. A grid where -1 is a wall, 0 is a gate, and 2147483647 (i.e. INT_MAX) is an empty room. Fill every empty room with its distance to the nearest gate. The naive instinct — BFS outward from each room — is quadratic; the senior answer floods from all gates at once.

1. The input data

INF below is the literal 2147483647:

rooms = [
  [INF,  -1,   0,  INF],
  [INF, INF, INF,  -1],
  [INF,  -1, INF,  -1],
  [  0,  -1, INF, INF],
]
# two gates: (0,2) and (3,0).  -1 cells are walls.

2. The approach

Multi-source BFS. The crucial line builds the initial queue from a generator expression: deque((r,c) for r in range(R) for c in range(C) if rooms[r][c] == 0) — every gate enters the queue at distance 0 simultaneously. Because BFS expands all sources in lockstep, the first time any room is reached, it is reached by the closest gate — so the first write to a room is automatically its minimum distance.

The loop pops a cell with popleft() (FIFO → non-decreasing distances) and inspects its four neighbours. The neighbour test rooms[nr][nc] == 2147483647 does double duty: it skips walls (-1), skips gates (0), and skips already-filled rooms — an unfilled room is the only cell still holding INT_MAX. When a fresh room is found, rooms[nr][nc] = rooms[r][c] + 1 records distance and marks it visited in one stroke, then it's enqueued. In place, returns nothing.

3. The output

distance grid (W = wall)
3W01
221W
1W2W
0W34

Each room shows the steps to its nearest gate. Cell (0,0) is 3 away from gate (3,0) via the left column; cell (3,3) is 4 from gate (0,2). Any room walled off from all gates would keep INT_MAX.

4. Strategic explanations

Complexity

Time O(R·C) — every cell is enqueued and dequeued at most once because the == INT_MAX guard blocks re-visits. Space O(R·C) for the queue. Compare the brute force: BFS from each of up to R·C rooms is O((R·C)²).

The idiomatic alternative

The generator-expression queue seed is already the clean idiom. A direction-vector list DIRS = ((1,0),(-1,0),(0,1),(0,-1)) hoisted out of the loop avoids rebuilding it each iteration. The level-by-level form — for _ in range(len(q)) with an explicit dist counter — is equivalent and lets you stop early if you only need distances up to some cap.

The edge case

if not rooms guards an empty grid. A grid with no gates → the queue starts empty, the loop never runs, every room stays INT_MAX — correct (unreachable). A room fully enclosed by walls keeps INT_MAX too. A grid that is all gates → every cell is 0, nothing to fill. The magic-number 2147483647 is brittle — a senior would name it as INF or use float('inf') for readability.

The follow-up

"What if each gate has a different cost to traverse, or rooms have weights?" — uniform BFS no longer gives shortest paths; you switch to Dijkstra with a heapq, multi-source by pushing all gates onto the heap first. "Return the single farthest room" — track the max distance written.

The trap

Two classic mistakes. First, looping over rooms and running an independent BFS from each — O((R·C)²) and times out. Second, in the multi-source version, forgetting that writing rooms[nr][nc] is the visited mark; if you instead keep a separate visited set but check it after enqueuing, a cell can be enqueued by two gates before either marks it, and it gets a non-minimal distance. Mark on enqueue.

Senior signal: seeds the BFS queue with all gates at once so the first visit to a room is provably its nearest-gate distance, uses the == INT_MAX sentinel as the combined wall/gate/visited filter, marks rooms on enqueue not dequeue, and knows to escalate to multi-source Dijkstra the moment traversal costs stop being uniform.
