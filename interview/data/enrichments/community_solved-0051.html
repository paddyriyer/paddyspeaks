A Community member asks the deceptively simple question every SQL interviewer loves: given two tiny tables, how many rows does each kind of join return? It is not really about syntax — it is a probe of whether you understand what each join does with unmatched rows, duplicate keys, and NULLs.

1. The source data — tbl1 and tbl2

tbl1.id
1
2
2
3
NULL
tbl2.id
2
2
3
4
NULL

Deliberately nasty: tbl1 has 2 twice, tbl2 has 2 twice, each table has a NULL, and each has a value the other lacks (tbl1 has 1, tbl2 has 4). Five rows each — and the join row counts are anything but five.

2. The SQL — one count per join type

SELECT 'inner' AS join_type, COUNT(*) AS row_count
FROM tbl1 INNER JOIN tbl2 ON tbl1.id = tbl2.id
UNION ALL
SELECT 'left',  COUNT(*) FROM tbl1 LEFT  JOIN tbl2 ON tbl1.id = tbl2.id
UNION ALL
SELECT 'right', COUNT(*) FROM tbl1 RIGHT JOIN tbl2 ON tbl1.id = tbl2.id
UNION ALL
SELECT 'full',  COUNT(*) FROM tbl1 FULL  JOIN tbl2 ON tbl1.id = tbl2.id
UNION ALL
SELECT 'cross', COUNT(*) FROM tbl1 CROSS JOIN tbl2;

Each branch joins the two tables one way and counts the rows. UNION ALL stacks the five counts into a single comparison table.

3. The resulting analysis

join_typerow_count
inner5
left7
right7
full9
cross25

Not one of these is 5, the row count of either table. The inner join already multiplies — the duplicate 2s cross-pair into four rows on their own. The cross join, 5×5, is the only one whose count you can predict without thinking.

4. Strategic explanations

The inner join — why duplicates multiply to 5, not 3

Matching values are 2 and 3. The value 3 appears once on each side: 1×1 = 1 row. But 2 appears twice in tbl1 and twice in tbl2: every left 2 pairs with every right 2, so 2×2 = 4 rows. Total 4 + 1 = 5. The lesson interviewers hunt for: a join on a non-unique key is a partial Cartesian product — the count is the sum over each key of (left count × right count), not "number of matching keys".

The NULL trap — NULL = NULL is not true

Both tables contain a NULL id. They never match — in SQL, NULL = NULL evaluates to unknown, not true. So neither NULL contributes to the inner join, and in the outer joins each NULL row appears only as an unmatched row padded with NULLs. This is the single most common wrong answer to this question: candidates assume the two NULLs join to each other. They do not.

Left and right — inner plus the orphans

A LEFT JOIN count is the inner-join count plus every left row that matched nothing. Unmatched left rows: id = 1 and id = NULL — 2 orphans. So 5 + 2 = 7. RIGHT JOIN is the mirror: inner plus unmatched right rows, which are id = 4 and id = NULL — again 2 orphans, 5 + 2 = 7. The symmetry is a coincidence of this data, not a rule.

Full join — inner plus both sides' orphans

FULL JOIN keeps the inner matches and both sets of unmatched rows: 5 + 2 (left orphans) + 2 (right orphans) = 9. The clean identity to recite: full = inner + left_only + right_only, equivalently full = left + right − inner — here 7 + 7 − 5 = 9. That formula is the senior-level one-liner.

Cross join — the pure Cartesian baseline

CROSS JOIN ignores any condition and pairs every left row with every right row: 5 × 5 = 25. NULLs and duplicates are irrelevant — it is raw multiplication. It is also the ceiling: no INNER/LEFT/RIGHT/FULL join on these tables can ever exceed 25. A useful sanity bound when a join "explodes" unexpectedly — if your result tops n × m, something is structurally wrong.

The portability and dialect note

The reference is written as five separate SELECT * statements — fine in a console, but the single UNION ALL query above returns one tidy comparison table. One caveat: MySQL before 8.0 has no FULL JOIN — you emulate it with LEFT UNION RIGHT. And CROSS JOIN with a WHERE equality is logically identical to an INNER JOIN; the planner treats them the same.

Senior signal: compute an inner-join count as the sum of left×right per key (duplicates multiply), state firmly that NULL never matches NULL, and recite full = left + right − inner with cross = n×m as the hard ceiling that flags a runaway join.
