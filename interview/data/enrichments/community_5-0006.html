A Community subscription business wants to measure win-back: for each customer who lapsed, when did they go quiet and when did they come back? "Lapsed" here means a gap of more than 90 days between consecutive orders. The interviewer is probing whether you can detect gaps with LAG, why you keep only the first churn per customer, and what happens to customers who never lapsed or never returned.

1. The source data — customer_orders

Customer 102 is the control — orders never more than ~40 days apart, so they must not appear. Customer 103 lapsed twice; the query must report only their first churn. Customer 105's single gap is exactly 90 days — the test of whether the threshold is strict (> 90 excludes them) or inclusive.

2. The SQL — LAG gap + first-churn ROW_NUMBER

WITH gaps AS (
    SELECT customer_id, order_date,
           LAG(order_date) OVER (PARTITION BY customer_id ORDER BY order_date) AS previous_order_date,
           (order_date - LAG(order_date) OVER (PARTITION BY customer_id ORDER BY order_date)) AS days_since_last
    FROM customer_orders
),
churned_customers AS (
    SELECT customer_id, order_date, previous_order_date
    FROM gaps
    WHERE days_since_last > 90
),
first_churn AS (
    SELECT customer_id, previous_order_date, order_date,
           ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY previous_order_date ASC) AS flag
    FROM churned_customers
)
SELECT customer_id,
       previous_order_date AS first_churn_date,
       order_date          AS recovery_order_date,
       (order_date - previous_order_date) AS num_days_churn_recovery
FROM first_churn
WHERE flag = 1
ORDER BY customer_id;

Three stages: LAG attaches each order's previous order date and the day-gap; WHERE days_since_last > 90 keeps only orders that ended a lapse; ROW_NUMBER then numbers each customer's lapse-ending orders so flag = 1 picks the earliest.

4. Strategic explanations

Why LAG over a self-join

LAG(order_date) OVER (PARTITION BY customer_id ORDER BY order_date) pulls the previous order's date onto the current row in a single ordered pass — O(n log n). The pre-window-function equivalent is a correlated subquery ("max order_date for this customer that is less than the current one"), which re-scans the partition per row and goes quadratic. On a real orders table the window function is the only viable form. The same query appears as days_since_last being driven by a second LAG call — most planners compute the identical window once and reuse it.

The ROW_NUMBER — "first churn" needs deduplication

Customer 103 has two lapse-ending orders. Without the first_churn CTE the query would emit both, double-reporting them. ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY previous_order_date) ranks a customer's lapses chronologically; WHERE flag = 1 keeps the earliest. Note the choice of ROW_NUMBER over RANK: two lapses cannot share a previous_order_date (a customer has one order per date), so there is no tie to worry about — but ROW_NUMBER guarantees exactly one row even if duplicate-dated orders existed.

The strict-vs-inclusive boundary — customer 105

days_since_last > 90 excludes a gap of exactly 90 days. Customer 105's Jan→Apr gap is precisely 90, so they do not count as churned. If the business defines churn as "90 days or more", the predicate becomes >= 90 and customer 105 joins the result. This is a definition to confirm with the stakeholder, not a number to guess — a one-day shift in the threshold changes who counts as a churned customer.

Customers who churned but never came back

This query's structure has a built-in bias worth naming: a lapse is only detected when a later order exists to compute the gap against. A customer whose last-ever order was in January 2023 and who never returned has no next row, so LAG on a future order never fires — they are invisible here. That is correct for a "churn and recovery" report, but if the question were "who has churned (returned or not)", you would instead compare each customer's most recent order date against the report date:

SELECT customer_id, MAX(order_date) AS last_order
FROM customer_orders
GROUP BY customer_id
HAVING (DATE '2023-12-31' - MAX(order_date)) > 90;
The amount column and tied dates

The schema carries amount, unused here — a follow-up might ask for revenue lost during the lapse, or whether the win-back order was larger than the pre-churn order. Also note ORDER BY order_date inside the window: if a customer placed two orders on the same day, their relative order is non-deterministic. With DATE-grain data and a 90-day threshold that does not change the answer, but on timestamp data you would add a tiebreaker like order_id to the ORDER BY.

Senior signal: reach for LAG rather than a quadratic self-join to compute consecutive-order gaps, use ROW_NUMBER partitioned per customer to isolate the first lapse, confirm whether the 90-day churn threshold is strict or inclusive, and recognise that a LAG-based design only sees customers who eventually returned — a true churn census needs a max-date-vs-today comparison instead.
