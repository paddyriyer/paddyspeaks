Finding the first month where revenue growth exceeded a threshold and the prior month also grew is a compound sequential condition on a time series — a pattern used in finance for breakout detection and in SaaS for identifying when growth is accelerating, not just spiking. The interviewer is probing whether you can express multi-month lookback conditions concisely using double-LAG and strict inequality, without resorting to a self-join or multiple subqueries.

May is a near-miss: revenue grew 15.79% which is below the 20% threshold. June breaks through at 27.27%, and May (the prior month) had positive growth, satisfying both conditions. June is the earliest qualifying month.

2. The SQL — double LAG for two-month lookback

WITH g AS (
  SELECT month_start, revenue,
         LAG(revenue) OVER (ORDER BY month_start) AS prev_revenue,
         LAG(revenue, 2) OVER (ORDER BY month_start) AS prev2_revenue
  FROM monthly_revenue
)
SELECT month_start,
       prev_revenue,
       revenue,
       ROUND(((revenue - prev_revenue) / NULLIF(prev_revenue, 0) * 100)::numeric, 2) AS mom_growth_pct
FROM g
WHERE prev_revenue IS NOT NULL AND prev2_revenue IS NOT NULL
  AND prev_revenue > prev2_revenue
  AND (revenue - prev_revenue) / NULLIF(prev_revenue, 0) > 0.20
ORDER BY month_start
LIMIT 1;

LAG(revenue) is the previous month's revenue; LAG(revenue, 2) is two months back. The condition prev_revenue > prev2_revenue checks that the prior month itself had positive MoM growth. The threshold check > 0.20 applies to the current month. NULLIF(prev_revenue, 0) prevents division by zero if any month had zero revenue. LIMIT 1 returns only the earliest qualifying month.

4. Strategic explanations

Why two LAG calls rather than a self-join

The two-condition lookback (current month exceeds 20%, prior month also grew) requires values from three consecutive rows: the current row, one row back, and two rows back. A self-join would require joining monthly_revenue to itself twice (on month_start - interval '1 month' and month_start - interval '2 months'), which is verbose, error-prone around gap months, and typically less readable. The double-LAG approach computes all three values as columns in a single pass over the table.

NULLIF(prev_revenue, 0) — safe division

NULLIF(x, 0) returns NULL when x = 0, converting division-by-zero from a runtime error to a NULL result. The NULL propagates through the growth calculation, causing that row to fail all comparison conditions and be excluded from the output — the correct behaviour when a prior month had zero revenue. Using CASE WHEN prev_revenue = 0 THEN NULL ELSE … END achieves the same effect but is more verbose.

What "prior month had positive growth" means precisely

The condition prev_revenue > prev2_revenue means the revenue in month M-1 exceeded the revenue in month M-2. This is strictly positive MoM growth for the prior month (i.e., at least some growth, not flat). If the business definition of "positive growth" excluded extremely small increases, you might require a minimum percentage rather than strict inequality on revenue. Raising this distinction in the interview shows production experience.

Rounding and the display vs filter distinction

The threshold comparison uses raw division: (revenue - prev_revenue) / NULLIF(prev_revenue, 0) > 0.20. The ROUND function appears only in the output column mom_growth_pct, not in the filter. Never round before comparing to a threshold — rounding can push a value that is genuinely 20.004% (above the threshold) to 20.00% and a value that is 20.3% (which should qualify) to display cleanly. Keep the filter on the full-precision value and round only for presentation.

Dialect notes

The ::numeric cast in ROUND is PostgreSQL-specific; in other engines use CAST(value AS DECIMAL(10,2)). NULLIF is standard SQL. LIMIT 1 is PostgreSQL/MySQL/SQLite; SQL Server uses SELECT TOP 1; BigQuery and Snowflake accept LIMIT 1. The rest of the query is fully portable.

Senior signal: use double-LAG to express a two-month lookback cleanly in a single pass; apply NULLIF(prev_revenue, 0) to guard against division-by-zero; keep ROUND out of the filter predicate (round only for display); and verify the NULL guards that exclude the first two rows where no complete lookback window exists.
