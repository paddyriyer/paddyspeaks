IBM's compensation team runs an annual bonus cycle, and finance wants a single query that turns each employee's performance score into a bonus amount: top performers get 20% of salary, solid performers 10%, everyone else nothing. It looks like trivial arithmetic — but the interviewer is watching the boundary values and the rows where performance_score is missing.

1. The source data — employees

Two rows are the whole interview. Chetan sits exactly on the 75 boundary and Farhan exactly on 90 — do they get the higher tier? And Esha has a NULL score because her review is not yet filed.

2. The SQL — tiered CASE

SELECT emp_id,
  CASE WHEN performance_score >= 90 THEN salary * 0.20
       WHEN performance_score >= 75 THEN salary * 0.10
       ELSE 0 END AS bonus
FROM employees;

A searched CASE evaluates its WHEN clauses top to bottom and stops at the first one that is true. Order matters: the >= 90 test must come before >= 75, or every high performer would be caught by the 10% rule first.

Because >= is inclusive, Chetan at 75 lands in the 10% tier (8000) and Farhan at 90 in the 20% tier (24000) — the boundaries belong to the higher band. Deepa at 60 and Esha at NULL both fall to the ELSE and get 0. That NULL behaviour is the trap worth naming.

4. Strategic explanations

NULL never matches a WHEN — it always falls through to ELSE

NULL >= 90 evaluates to unknown, not true, so a NULL-score row matches no WHEN branch and lands in ELSE 0. That happens to be sensible here — an unreviewed employee gets no bonus — but it is silent. If a missing review should instead block the row entirely or raise a flag, make that explicit:

SELECT emp_id,
  CASE WHEN performance_score IS NULL THEN NULL  -- "not yet calculable"
       WHEN performance_score >= 90 THEN salary * 0.20
       WHEN performance_score >= 75 THEN salary * 0.10
       ELSE 0 END AS bonus
FROM employees;

A NULL bonus and a 0 bonus mean very different things to finance. Decide which on purpose.

The branch-ordering trap

Flip the two conditions and the logic quietly breaks:

CASE WHEN performance_score >= 75 THEN salary * 0.10   -- catches 95 too!
     WHEN performance_score >= 90 THEN salary * 0.20   -- never reached
     ELSE 0 END

A score of 95 satisfies >= 75 first, so it gets 10% and the 20% branch is dead code. With overlapping ranges, always order the WHEN clauses from most restrictive to least.

Inclusive vs exclusive boundaries

Is a score of exactly 75 a "solid performer" or not? >= 75 says yes; > 75 would say no. The spec rarely states it, so confirm with the interviewer and write the operator that matches their answer. Off-by-one at a tier boundary is the single most common bug in banding logic.

If a missing ELSE — the implicit NULL

A CASE with no ELSE returns NULL when nothing matches, not 0. Here the explicit ELSE 0 is what makes Deepa's bonus a number. Drop it and her bonus — and Esha's — would be NULL, which then poisons any SUM(bonus) downstream unless wrapped in COALESCE.

Integer division and the multiplier

salary * 0.20 is safe because 0.20 is a numeric literal, so the result is numeric. But salary * 20 / 100 on an integer salary column can truncate intermediate results in some engines. Keep the fractional multiplier, or cast: salary::numeric * 0.20.

Senior signal: state that NULL scores fall through to ELSE and decide whether that should be 0 or NULL, order overlapping WHEN branches most-restrictive-first, pin down whether tier boundaries are inclusive, and keep the explicit ELSE so the bonus is never an accidental NULL.
