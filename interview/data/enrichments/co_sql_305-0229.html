Zscaler's catalog team is auditing product copy and wants every product whose description contains a digit — specs like "40mm", "100W" or "750ml" that should arguably live in structured fields, not buried in free text. The brute-force solution chains ten LIKE clauses, one per digit. It works, but the interviewer wants to see whether you know the single-expression alternatives and the trap hiding in pattern matching.

1. The source data — products

product_iddescription
1Wireless headphones with 40mm drivers
2Soft cotton blanket, machine washable
3USB-C cable rated for 100W charging
4Ergonomic chair with lumbar support
5Stainless steel water bottle 750ml

Products 1, 3 and 5 carry numbers in their copy. Products 2 and 4 are pure prose. Note product 3's "USB-C" contains a hyphen and a letter but no digit there — the digit is in "100W". Substring matching does not care about word boundaries.

2. The SQL — OR-chain of LIKE per digit

SELECT product_id, description
FROM products
WHERE description LIKE '%0%' OR description LIKE '%1%' OR description LIKE '%2%'
   OR description LIKE '%3%' OR description LIKE '%4%' OR description LIKE '%5%'
   OR description LIKE '%6%' OR description LIKE '%7%' OR description LIKE '%8%'
   OR description LIKE '%9%';

Ten predicates joined by OR — a row passes if it contains any one of the digit characters 0 through 9 anywhere.

3. The resulting analysis

product_iddescription
1Wireless headphones with 40mm drivers
3USB-C cable rated for 100W charging
5Stainless steel water bottle 750ml

Exactly the three rows with numbers in them. The OR-chain is verbose but correct. The interview value is showing you can collapse it.

4. Strategic explanations

The single-expression rewrite — SIMILAR TO or regex

Ten LIKEs is a character class spelled out the long way. Postgres has two compact equivalents:

-- POSIX regex: ~ means "matches"
SELECT product_id, description
FROM products
WHERE description ~ '[0-9]';

-- SQL-standard SIMILAR TO (note the surrounding %)
SELECT product_id, description
FROM products
WHERE description SIMILAR TO '%[0-9]%';

The regex ~ '[0-9]' is the cleanest — it is an unanchored search, so no % wildcards needed. \d also works as shorthand for [0-9].

The trap: LIKE is anchored, regex is not

A frequent mistake is writing description LIKE '[0-9]' — in LIKE, square brackets are literal characters, not a class, so that hunts for the literal text "[0-9]" and matches nothing. Conversely, a regex like description ~ '0' matches a substring anywhere, but description SIMILAR TO '0' requires the whole string to equal "0". SIMILAR TO is implicitly anchored at both ends, which is why it needs the leading and trailing %. Mixing up the three matching dialects is the classic error.

Actually extracting the number, not just flagging it

The title says "pattern extraction". Flagging rows is one reading; pulling the numeric token out is the stronger one:

SELECT product_id,
       description,
       (regexp_match(description, '\d+'))[1] AS first_number
FROM products
WHERE description ~ '\d';

For product 1 that yields "40", for product 3 "100", for product 5 "750". If a description holds several numbers, regexp_matches(…, '\d+', 'g') returns them all as separate rows.

Case, performance and NULLs

Digits have no case, so ILIKE versus LIKE is irrelevant here — but if the audit were for letters it would matter. On a large table every one of these is a sequential scan; a leading-% pattern cannot use a normal B-tree index. For repeated digit-hunting, a pg_trgm GIN index or a precomputed boolean has_digit column pays off. And a NULL description fails every predicate — it is neither matched nor returned, which is the right behaviour but worth stating.

Senior signal: collapse the ten-way OR-chain into ~ '[0-9]' or SIMILAR TO '%[0-9]%', articulate that LIKE brackets are literal while regex brackets are a class and SIMILAR TO is whole-string anchored, and offer regexp_match to actually extract the number when the task is "extraction" rather than mere detection.
