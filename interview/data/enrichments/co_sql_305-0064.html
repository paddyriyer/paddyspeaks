DigitalOcean's growth team is segmenting the customer base and wants everyone whose name starts with "A" or ends in "son". It is a two-line WHERE with LIKE — but the interviewer is probing whether you know exactly what the wildcards match, that LIKE is case-sensitive in Postgres, and what an index can and cannot do with each pattern.

1. The source data — customers

customer_idnameemailcountry
1Andersonan@xUS
2Johnsonjo@xUS
3Aliceal@xUS
4Bobbo@xUS
5Watsonwa@xUS

Anderson matches both rules (starts with A, ends in "son"); Alice matches only the first; Johnson and Watson only the second; Bob matches neither. That overlap — one name satisfying both predicates — is the subtlety the interviewer is testing.

2. The SQL — two LIKE patterns, OR

SELECT * FROM customers
WHERE name LIKE 'A%' OR name LIKE '%son';

'A%' matches any string beginning with a capital A (% = zero-or-more characters). '%son' matches any string ending in "son". OR keeps a row satisfying either.

3. The resulting analysis

customer_idnameemailcountry
1Andersonan@xUS
2Johnsonjo@xUS
3Aliceal@xUS
5Watsonwa@xUS

Four of five rows match; only Bob is excluded. Crucially, Anderson appears exactly once even though it satisfies both predicates — OR in a WHERE clause is a row filter, not a join, so a row is returned once if it qualifies at all. No DISTINCT needed; double-matching never means double rows.

4. Strategic explanations

LIKE is case-sensitive — the silent miss

In Postgres, LIKE respects case. name LIKE 'A%' matches "Anderson" but not "anderson" or "ASHTON" in lowercase — and a name typed 'andrew' would slip through the segment. If the segmentation should be case-insensitive, use ILIKE:

SELECT * FROM customers
WHERE name ILIKE 'a%' OR name ILIKE '%son';

Or normalise both sides with LOWER(name) LIKE 'a%'. Names are exactly the column where casing is inconsistent in real data — confirm whether the match should be case-sensitive before assuming.

% vs _ — know both wildcards

% matches zero or more characters; _ matches exactly one. 'A%' matches "A" itself (zero following chars) and "Anderson" alike. If the requirement were "a three-letter name starting with A", that is 'A__', not 'A%'. And to match a literal percent or underscore in the data, escape it: LIKE '100\%'.

The leading-wildcard index trap

name LIKE 'A%' is sargable — a plain B-tree index on name can range-scan the "A" prefix. But name LIKE '%son' has a leading wildcard, and a B-tree cannot seek a suffix — that predicate forces a full scan. On a large customers table this matters. Two senior fixes: index the reversed string, CREATE INDEX ON customers (reverse(name)) and query reverse(name) LIKE reverse('%son'); or use a trigram index, CREATE INDEX ON customers USING gin (name gin_trgm_ops), which accelerates both prefix and suffix LIKE.

NULL names are never matched

A customer with a NULL name satisfies neither LIKE — any comparison with NULL is unknown, and unknown OR unknown is unknown, which WHERE treats as not-true. So NULL-name rows are silently excluded. Usually fine; but if "no name on file" is its own segment, handle it explicitly with OR name IS NULL.

SELECT * is a smell in production

SELECT * is acceptable for exploration, but a segmentation query feeding a campaign should name its columns — SELECT customer_id, name, email. Explicit columns survive schema changes (a new column will not silently widen the export), make the intent reviewable, and avoid shipping sensitive fields by accident.

Senior signal: know that LIKE is case-sensitive in Postgres and reach for ILIKE when matching messy name data, distinguish % from _, explain that a leading-wildcard '%son' defeats a B-tree index and propose a reverse or trigram index, and note that OR never produces duplicate rows so no DISTINCT is required.
