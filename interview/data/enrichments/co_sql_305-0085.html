Coinbase's growth team wants to know where its sign-ups come from at the email-provider level: how many users are on Gmail, on a corporate domain, on a disposable address. The raw material is the email column, and the task is pure string surgery — pull the part after the @, then group on it. Easy to write, easy to get subtly wrong on malformed input.

1. The source data — users

user_idemail
1a@gmail.com
2b@gmail.com
3c@yahoo.com
4d@coinbase.com
5e@gmail.com
6f@yahoo.com

Three Gmail users, two Yahoo, one corporate (coinbase.com). The interesting part is what these rows don't contain — a missing @, mixed case, a NULL — all of which the parsing logic must survive in production.

2. The SQL — substr after the @

SELECT substr(email, POSITION('@' IN email) + 1) AS domain,
       COUNT(*) AS users
FROM users
GROUP BY domain
ORDER BY users DESC;

POSITION('@' IN email) finds the 1-based index of the @; + 1 steps past it; substr with no length argument takes everything from there to the end. Group on that derived domain, order by population.

3. The resulting analysis

domainusers
gmail.com3
yahoo.com2
coinbase.com1

Gmail leads with 3, Yahoo 2, the corporate domain 1. The ORDER BY users DESC ranks providers by reach — the exact shape a growth team wants for a channel chart.

4. Strategic explanations

Grouping by an alias — portable or not?

GROUP BY domain references the SELECT-list alias. Postgres and MySQL allow this, but it is not standard SQL — some engines reject it because GROUP BY logically runs before SELECT. The fully portable forms repeat the expression or use a positional reference:

GROUP BY substr(email, POSITION('@' IN email) + 1)   -- repeat expression
GROUP BY 1                                            -- positional, terse

Knowing the alias shortcut works here but is non-portable is the kind of detail an interviewer rewards.

The "no @ sign" trap

If an email value has no @ at all (bad data, or a username typed without a domain), POSITION returns 0. Then substr(email, 0 + 1) = substr(email, 1) returns the entire string — the whole malformed value becomes a bogus "domain" and pollutes the grouping. Defend against it explicitly:

SELECT lower(split_part(email, '@', 2)) AS domain, COUNT(*) AS users
FROM users
WHERE email LIKE '%@%'
GROUP BY 1
ORDER BY users DESC;

split_part(email, '@', 2) is cleaner than substr+POSITION — it directly returns the second piece — and the WHERE email LIKE '%@%' filters junk rows before they can form a fake domain.

Case sensitivity splits the count

Email domains are case-insensitive, but substr is not. Gmail.com, GMAIL.COM and gmail.com would land in three separate groups and the headline Gmail number would be wrong. Normalise with lower(...) before grouping — cheap insurance that prevents a real reporting bug.

NULL emails and the off-by-one

A NULL email yields NULL through substr; GROUP BY then collects every NULL into a single group and COUNT(*) tallies it — a NULL "domain" row appears unless you filter email IS NOT NULL. Also note the + 1 is load-bearing: POSITION returns the index of the @, so without the increment the domain string would start with a literal @.

Sub-domain rollups — the likely follow-up

A growth team often wants to collapse mail.google.com and google.com together, or group every corporate address against the free providers. That means extracting the registrable domain (last two labels) — split_part from the right, or a CASE that buckets known free providers vs "corporate". Anticipating that the raw domain is rarely the final grain is the senior move.

Senior signal: reach for split_part over substr+POSITION for clarity, guard against the missing-@ case where POSITION returns 0 and leaks the whole string, lower-case the domain so case variants do not fragment the count, and note that grouping by a SELECT alias is a Postgres convenience rather than portable SQL.
