A music-streaming analyst is reconciling two stores: a historical table holding pre-aggregated play counts up to 2025-08-03, and a weekly table holding one row per individual play from 2025-08-04 onward. The task is a cumulative total through 2025-08-04 — which means counting weekly rows for that single day, then unioning with history. The interviewer is probing whether you can combine two differently-grained tables without losing pairs that appear in only one of them.

1. The source data — songs_history and songs_weekly

Three edge cases: (1,100) appears in both tables (must be summed), (3,300) exists only in history, (4,400) only in weekly. And (1,200)'s weekly play is on 08-05 — past the cutoff, so it must not be counted even though the pair has history.

2. The SQL — count weekly to the cutoff, UNION ALL, then sum

WITH new_plays AS (
  SELECT user_id, song_id, COUNT(*) AS play_count
  FROM songs_weekly
  WHERE played_at <= DATE '2025-08-04'
  GROUP BY user_id, song_id
)
SELECT user_id, song_id, SUM(play_count)::int AS cumulative_plays
FROM (
  SELECT user_id, song_id, play_count FROM songs_history
  UNION ALL
  SELECT user_id, song_id, play_count FROM new_plays
) all_plays
GROUP BY user_id, song_id
ORDER BY cumulative_plays DESC, user_id, song_id;

new_plays reduces the event-grain weekly table to the same (user, song, count) grain as history, filtered to plays on or before the cutoff. UNION ALL stacks both, and the outer GROUP BY ... SUM folds rows for the same pair into one cumulative total.

4. Strategic explanations

Why UNION ALL, never UNION

UNION deduplicates its result — and that is exactly wrong here. If history and weekly happened to produce an identical (user_id, song_id, play_count) triple, UNION would collapse the two into one and under-count. UNION ALL keeps every row, so the subsequent SUM adds both contributions. The rule of thumb: when you are about to aggregate the combined set, always UNION ALL — deduplication before a SUM destroys data.

Reconciling two grains before combining

songs_history is pre-aggregated (one row per pair); songs_weekly is event-grained (one row per play). You cannot union them directly — the columns would mean different things. The new_plays CTE first lifts weekly to the aggregate grain with COUNT(*) ... GROUP BY user_id, song_id. Only once both sides share the (user, song, count) shape is a union meaningful. Aligning grain before combining sources is the core senior move here.

The cutoff filter and the half-open question

The spec says "through and including 2025-08-04," so played_at <= DATE '2025-08-04' is correct for a DATE column. If played_at were a TIMESTAMP, <= '2025-08-04' would mean "≤ midnight" and drop the rest of the day — you would switch to the half-open < DATE '2025-08-05'. Here the column is DATE, so both forms agree, but stating which type you assumed shows you know the trap.

Full-outer behaviour without a FULL JOIN

A natural instinct is FULL OUTER JOIN on (user_id, song_id) with COALESCE to add the two counts. That works, but it is verbose and needs COALESCE(h.cnt,0) + COALESCE(w.cnt,0) plus COALESCE on the key columns themselves. UNION ALL + GROUP BY achieves the same "include pairs from either side" result far more simply, because grouping naturally gathers a pair whether it came from one source or both.

Dialect notes

UNION ALL, COUNT(*), and GROUP BY are universal — this query is fully portable to MySQL, SQL Server, Oracle, and SQLite. The only Postgres-flavoured touch is ::int; elsewhere use CAST(SUM(play_count) AS INT). Note SUM over an empty group never happens here, but in general SUM of no rows is NULL, not 0 — irrelevant given every pair has at least one contributing row.

Senior signal: when stitching two tables of different grain, first aggregate the event-level table up to the summary grain, then combine with UNION ALL (never UNION, which would dedupe and under-count before the SUM), and let the outer GROUP BY gather pairs that appear in only one source — the union plus group-by is a simpler full-outer than a FULL JOIN with COALESCE.
