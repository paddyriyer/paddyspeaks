An exoplanet catalogue team asks: "For every star, show how many planets orbit it, the average planet mass, and the shortest and longest orbital periods — including stars with no known planets." The "including stars with no known planets" clause is the key signal — it demands a LEFT JOIN from the star dimension to the planet fact table, so that solitary stars still appear in the output. The interviewer is also checking whether the candidate applies the NULLIF(regexp_replace(...)::numeric) defensive cast for text-stored numeric columns and sorts by planet count to surface the richest systems first.

Vega (star_id=4) has no planets. It should appear in the output with planet_count = 0 and NULL aggregates.

2. The SQL — LEFT JOIN with grouped multi-aggregate and defensive casts

SELECT s.name AS star, s.spectral_type,
       COUNT(p.planet_id) AS planet_count,
       AVG(NULLIF(regexp_replace(p.mass_earth::text,           '[^-.0-9]', '', 'g'), '')::numeric) AS avg_planet_mass,
       MIN(NULLIF(regexp_replace(p.orbital_period_days::text,  '[^-.0-9]', '', 'g'), '')::numeric) AS shortest_orbit_days,
       MAX(NULLIF(regexp_replace(p.orbital_period_days::text,  '[^-.0-9]', '', 'g'), '')::numeric) AS longest_orbit_days
FROM stars s
LEFT JOIN planets p ON s.star_id = p.star_id
GROUP BY s.name, s.spectral_type
ORDER BY planet_count DESC;

4. Strategic explanations

LEFT JOIN — retaining planetless stars

A LEFT JOIN returns all rows from the left table (stars) regardless of whether a matching row exists in planets. For Vega, the LEFT JOIN contributes a row where all planets columns are NULL. An INNER JOIN would silently exclude Vega, understating the number of stars in the catalogue and giving a skewed view that only well-studied systems appear. Whenever the requirement uses the phrase "including X with no Y," LEFT JOIN from the parent to the child is the required pattern.

COUNT(p.planet_id) vs COUNT(*) for outer joins

With a LEFT JOIN, the NULL-padded row for Vega has p.planet_id = NULL. COUNT(p.planet_id) ignores NULL values and returns 0. COUNT(*) counts the NULL row and returns 1, incorrectly suggesting Vega has one planet. Always count a specific NOT NULL column from the right (optional) side of a LEFT JOIN. This is the canonical LEFT JOIN aggregation trap tested at every level.

AVG / MIN / MAX of all-NULL groups return NULL

Standard SQL aggregate functions skip NULL values. When the entire group consists of NULL values (Vega's case), AVG, MIN, and MAX return NULL. This is semantically correct — there is no average mass or orbital period when there are no planets. If a sentinel like 0 or −1 is required for downstream systems, use COALESCE(AVG(...), 0), but prefer NULL as the truthful representation.

Defensive cast pattern

The mass_earth and orbital_period_days columns may be stored as VARCHAR with unit strings or measurement notes. The triple regexp_replace / NULLIF / ::numeric sanitises them: strip non-numeric characters globally, convert empty string to NULL, then parse as numeric. In Snowflake the equivalent is TRY_TO_NUMBER(p.mass_earth, 10, 4) or TRY_CAST(p.mass_earth AS FLOAT), which return NULL on any parse failure without the regex overhead.

GROUP BY functional dependency — (name, spectral_type)

Since star_id is the primary key, s.name and s.spectral_type are functionally determined by it. The query groups by both anyway for portability — Snowflake and BigQuery require all non-aggregated SELECT columns to appear in GROUP BY even when a PK dependency is obvious. If two stars happened to share the same name and spectral_type (unusual but possible), they would be merged into a single output row, which would be wrong; adding s.star_id to the GROUP BY prevents that edge case.

ORDER BY with tie-breaking

ORDER BY planet_count DESC leaves ties (e.g., two stars each with 1 planet) in undefined order. Add , s.name as a deterministic tiebreaker for stable, reproducible output.

Senior signal: articulate why LEFT JOIN is required to include planetless stars, use COUNT(p.planet_id) not COUNT(*) to return 0 for unmatched outer-join rows, explain that AVG/MIN/MAX of an all-NULL group returns NULL (semantically correct), and substitute TRY_CAST or TRY_TO_NUMBER for the Snowflake rewrite of the defensive numeric cast.
