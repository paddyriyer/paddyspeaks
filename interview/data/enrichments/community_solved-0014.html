A Community layout puzzle: a cookbook has one recipe title per page, and the task is to render it the way an open book reads — a left page beside its facing right page, one row per spread. The reference solution generates a page sequence with a recursive CTE and joins titles onto the even (left) and odd (right) slots. It works, but it quietly produces a "page 0" spread that an interviewer will ask about.

1. The source data — cookbook_titles

page_numbertitle
1Introduction
2Soups
3Salads
4Mains
5Desserts

Five pages, numbered 1–5. The catch: 5 is odd, so the last page has no facing partner — and the book starts at page 1, but the solution's generated sequence starts at 0. Both boundaries surface in the output.

2. The SQL — generate a page sequence, join left/right

WITH RECURSIVE seq AS (
    SELECT 0 AS page_num
    UNION ALL
    SELECT page_num + 1 FROM seq
    WHERE page_num < (SELECT MAX(NULLIF(regexp_replace(page_number::text, '[^-.0-9]', '', 'g'), '')::numeric::int) FROM cookbook_titles)
),
left_page AS (
    SELECT page_num AS left_page_num FROM seq WHERE page_num % 2 = 0
)
SELECT left_page.left_page_num,
       ct_left.title AS left_page_title,
       ct_right.title AS right_page_title
FROM left_page
LEFT JOIN cookbook_titles ct_left
       ON left_page.left_page_num = NULLIF(regexp_replace(ct_left.page_number::text, '[^-.0-9]', '', 'g'), '')::numeric::int
LEFT JOIN cookbook_titles ct_right
       ON (left_page.left_page_num + 1) = NULLIF(regexp_replace(ct_right.page_number::text, '[^-.0-9]', '', 'g'), '')::numeric::int;

seq generates every integer from 0 to the max page. left_page keeps the even numbers as the left page of each spread. The two LEFT JOINs attach the title at left_page_num and at left_page_num + 1 — the facing right page. LEFT JOIN (not inner) is what lets a half-empty spread survive.

3. The resulting analysis

left_page_numleft_page_titleright_page_title
2SoupsSalads
4MainsDesserts
0NULLIntroduction

The two real spreads are correct: pages 2–3 (Soups / Salads) and 4–5 (Mains / Desserts). But spread 0 is an artefact: seq started at 0, there is no page 0, so its left title is NULL — and it pulls page 1 ("Introduction") into the right slot. The LEFT JOIN kept this half-empty row instead of dropping it. The query also has no ORDER BY, which is why spread 0 lands last rather than first.

4. Strategic explanations

The page-0 artefact — the off-by-one the interviewer is waiting for

seq seeds at SELECT 0, so the even-page filter yields 0, 2, 4… and page 1 becomes the right half of a non-existent spread 0. Real books open with page 1 alone on the right (a recto), then pages 2–3 face each other. There are two clean fixes depending on the desired layout. If page 1 should stand alone first, seed the sequence at 2 and add a separate "page 1 alone" row. If the intent is simply pages paired (1,2), (3,4), (5), then the left page is odd, not even:

SELECT page_num AS left_page_num FROM seq WHERE page_num % 2 = 1
-- left = 1,3,5 ... right = left+1

Either way, starting seq at 0 and not filtering it out is the bug to name. The cleanest sequence generator avoids it entirely — generate_series(1, (SELECT MAX(page_number) FROM cookbook_titles)) — no recursion, no stray 0.

Why generate_series beats the recursive CTE here

The recursive CTE is just reinventing a number range. Postgres has a purpose-built function:

WITH seq AS (SELECT generate_series(1, (SELECT MAX(page_number) FROM cookbook_titles)) AS page_num)

It is shorter, faster, and cannot accidentally start at the wrong number. Recursive CTEs earn their keep for genuine graph/tree walking — not for "1 to N".

The odd-last-page edge case — why LEFT JOIN matters

With 5 pages, the final spread has a left page (5) but no page 6. An inner join on the right side would silently drop the entire last spread — the lone page vanishes. The LEFT JOIN keeps it with a NULL right title, which is correct: an open book's last leaf often has a blank facing page. Choosing LEFT over INNER here is a deliberate, explainable decision.

The regexp_replace wrapper — defensive, possibly unnecessary

page_number is joined through NULLIF(regexp_replace(page_number::text, '[^-.0-9]', '', 'g'), '')::numeric::int — scrubbing non-numeric characters in case page numbers arrive as dirty text ("p. 12", "12a"). If the column is a clean integer, this is a no-op and the join is just ON left_page.left_page_num = ct.page_number. Mention that you would drop the wrapper on clean data rather than carrying it by reflex.

Gaps and duplicate page numbers

If a page number is missing (the book skips 3), the LEFT JOIN yields a NULL title for that slot — a blank page, arguably correct. But a duplicate page_number is dangerous: the join would match both rows and that spread would appear twice. Real page numbers should be unique — a senior notes the assumption and would enforce it with a key.

Senior signal: spot that seeding the sequence at 0 manufactures a phantom "page 0" spread and fix the parity (or use generate_series(1, N)), explain that the LEFT JOIN is what preserves an odd last page instead of dropping it, and add an explicit ORDER BY so spreads render in reading order.
