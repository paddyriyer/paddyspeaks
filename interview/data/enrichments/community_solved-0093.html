A retail-banking team needs a daily balance ledger: for every account, the running balance after each day's transactions, with same-day credits and debits netted first, then laid out as one column per account. The interviewer is probing three layered skills — converting a transaction type into a signed amount, computing a running total with a window function, and pivoting the long result into a wide report.

Two edge cases: account 401 has a credit and a debit on the same day (Jan 1) that must net to +800; account 404 only ever has a debit, so its balance goes negative — the query must not assume balances stay positive.

2. The SQL — sign, running sum, then pivot

WITH account_txn AS (
  SELECT accountid, transactiondate, amount, transactiontype,
         CASE WHEN transactiontype = 'Credit' THEN amount ELSE -amount END AS credit_debit
  FROM bank_txn
),
account_balance AS (
  SELECT accountid, transactiondate, SUM(credit_debit) AS amount
  FROM account_txn
  GROUP BY accountid, transactiondate
),
account_running AS (
  SELECT accountid, transactiondate, amount,
         SUM(amount) OVER (PARTITION BY accountid ORDER BY transactiondate ASC) AS daily_account_balance
  FROM account_balance
)
SELECT transactiondate,
       MAX(CASE WHEN accountid = 401 THEN daily_account_balance END) AS account_401,
       MAX(CASE WHEN accountid = 402 THEN daily_account_balance END) AS account_402,
       MAX(CASE WHEN accountid = 403 THEN daily_account_balance END) AS account_403,
       MAX(CASE WHEN accountid = 404 THEN daily_account_balance END) AS account_404
FROM account_running
GROUP BY transactiondate
ORDER BY transactiondate;

Account 401 on Jan 1 nets 1000 − 200 = 800, then Jan 2 adds 500 → 1300. Account 404 sits at −150. The NULLs mark days an account had no transaction — the query reports a sparse grid, not a forward-filled one.

4. Strategic explanations

Step 1 — signing with CASE

CASE WHEN transactiontype = 'Credit' THEN amount ELSE -amount END turns the type/amount pair into a single signed number. Once every row is signed, "balance" is just a sum — no special-casing downstream. This is the move that makes the rest of the query trivial.

Step 2 — netting same-day rows before the running sum

The account_balance CTE does GROUP BY accountid, transactiondate first. This is essential: account 401's two Jan-1 rows collapse to one net row (+800) before the window function runs. Without it the running sum would still total correctly, but you would emit two output rows for the same account/date and the pivot's MAX would have to mask the duplicate.

Step 3 — the running balance window

SUM(amount) OVER (PARTITION BY accountid ORDER BY transactiondate) accumulates per account in date order. With an ORDER BY and no explicit frame, the default frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — exactly a running total. Each row shows the balance as of that day.

Step 4 — conditional aggregation as a pivot, and the NULL gaps

MAX(CASE WHEN accountid = 401 THEN daily_account_balance END) is the portable pivot idiom: per date, only the matching account's value survives the CASE, the rest are NULL, and MAX picks the one real value. The downside is hardcoded account ids — a fifth account silently won't appear. The NULL cells (e.g. 403 on Jan 1) mean "no activity that day"; if the report needs the balance carried forward, you would generate a full date spine per account and forward-fill with LAST_VALUE / a gaps-and-islands fill, which is a meaningfully harder query worth flagging.

Dialect note — native PIVOT

SQL Server, Oracle, and Snowflake offer a PIVOT operator that expresses step 4 declaratively. Postgres (this engine) and BigQuery rely on the CASE idiom shown. Either way the account list must be enumerated at query time; a truly dynamic column set needs procedural SQL or crosstab.

Senior signal: decompose this into sign → net same-day → running window → pivot, explain why the same-day GROUP BY must precede the window function, note that NULL cells mean "no activity" and forward-filling them is a separate harder problem, and flag the hardcoded-account-id fragility of the conditional-aggregation pivot.
