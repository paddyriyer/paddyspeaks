SEO teams routinely ask: "For each domain we crawl, which page has the highest SEO score, and when was it last seen?" When multiple pages tie for the top score within a domain, all tied pages must be returned — not just an arbitrary one. This question tests the candidate's ability to distinguish ROW_NUMBER (picks one winner arbitrarily) from RANK (preserves ties), and to filter the result correctly using either a subquery or Snowflake's first-class QUALIFY clause.

example.com has two pages tied at score 92 — the key edge case that distinguishes RANK from ROW_NUMBER. techblog.io has a clear winner at 95.

2. The SQL — RANK() in a subquery, filter on rk = 1

SELECT domain, page_url, seo_score, last_crawled FROM (
  SELECT domain, page_url, seo_score, last_crawled,
         RANK() OVER (PARTITION BY domain ORDER BY seo_score::numeric DESC) AS rk
  FROM pages
) t
WHERE rk = 1
ORDER BY domain;

Both tied example.com pages surface because RANK assigns rank 1 to both. techblog.io returns exactly one row. The about page (score 85) and the faq/intro pages are correctly excluded.

4. Strategic explanations

RANK vs ROW_NUMBER vs DENSE_RANK — the decision tree

All three are ranking functions, but they handle ties differently:

  • ROW_NUMBER(): assigns a unique integer to every row regardless of ties — the "winner" is arbitrary when scores are equal. Use when you genuinely want exactly one row per partition (e.g. deduplication).
  • RANK(): assigns the same rank to ties, then skips ranks. Rank 1 is shared by both 92-score pages; rank 3 would be the next distinct score (85). Filtering WHERE rk = 1 returns all co-winners.
  • DENSE_RANK(): same tie assignment but no gaps — after two rank-1 rows the next is rank 2, not 3. For a "top-1" filter the result is identical to RANK; the difference matters when filtering for rank 2, rank N.

The interviewer chose RANK to ensure ties are preserved. Switching to ROW_NUMBER would silently drop one of the two example.com winners — a data-correctness bug, not a performance choice.

The ::numeric cast on seo_score

The cast seo_score::numeric suggests the column is stored as text or a mixed type (common in NoSQL-derived schemas loaded into Snowflake). Without the cast, lexicographic ordering would make "92" < "95" correctly but "9" > "85" — a silent correctness failure. Always cast before numeric window-function ordering when column types are uncertain.

Snowflake's QUALIFY clause — the idiomatic alternative

The solution uses a subquery to filter on the window result, which is required in Postgres and most ANSI-SQL engines (window functions cannot appear in WHERE). Snowflake (and DuckDB) add a QUALIFY clause that filters on window results inline, eliminating the subquery wrapper:

SELECT domain, page_url, seo_score, last_crawled
FROM pages
QUALIFY RANK() OVER (PARTITION BY domain ORDER BY seo_score::numeric DESC) = 1
ORDER BY domain;

QUALIFY executes after the window computation but before ORDER BY, exactly mirroring HAVING's position relative to GROUP BY. It is cleaner and often marginally faster (one less materialisation). Naming this as the Snowflake-native pattern signals platform fluency.

Portability notes
  • Snowflake: both the subquery form and QUALIFY work. Prefer QUALIFY.
  • Postgres: subquery form only; no QUALIFY clause. Use a CTE or subquery.
  • BigQuery: supports QUALIFY (added 2021). The subquery form also works.
  • MySQL: no QUALIFY; subquery or CTE required.
Performance consideration

The window function scans the pages table once, computing the rank for every row. The outer WHERE rk = 1 then filters. An index on (domain, seo_score DESC) can help the optimiser if the table is large and domain cardinality is high. In Snowflake the micro-partition pruning architecture often makes full scans cheap; explicit indexes are not available, but clustering keys on domain can guide micro-partition elimination.

Senior signal: articulate the RANK vs ROW_NUMBER choice as a data-correctness decision (not a style preference), explain that the ::numeric cast prevents silent lexicographic-ordering bugs, and name Snowflake's QUALIFY clause as the idiomatic one-pass alternative to the subquery pattern. Bonus: note that QUALIFY is also available in BigQuery and DuckDB.
