A finance lead wants the 2026 quarterly scorecard: each business unit on a row, Q1 through Q4 as columns, and a cumulative full-year total at the end. The source is a transaction-level ledger — one row per sale — so the query must aggregate to quarters, run a cumulative total, then pivot the four quarters into columns. The interviewer is probing whether you can layer a window-function running total inside a pivot pipeline and surface it correctly through the final GROUP BY.

1. The source data — sales_ledger

business_unitsale_dateamountquarter
North2026-02-10100Q1
North2026-05-11200Q2
North2026-08-12150Q3
North2026-11-13250Q4
South2026-01-1580Q1
South2026-06-20120Q2

North transacts in all four quarters; South only in Q1 and Q2. South is the test case: its Q3 and Q4 columns must read 0 — not NULL, not absent — and its running total must stop climbing after Q2 rather than inventing later activity.

2. The SQL — aggregate, running total, then pivot

WITH quarterly AS (
  SELECT
    business_unit,
    EXTRACT(QUARTER FROM sale_date)::INT AS q,
    SUM(amount) AS revenue
  FROM sales_ledger
  WHERE EXTRACT(YEAR FROM sale_date) = 2026
  GROUP BY business_unit, EXTRACT(QUARTER FROM sale_date)
),
ranked AS (
  SELECT
    business_unit, q, revenue,
    SUM(revenue) OVER (
      PARTITION BY business_unit ORDER BY q
      ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total
  FROM quarterly
)
SELECT
  business_unit,
  SUM(CASE WHEN q = 1 THEN revenue ELSE 0 END) AS q1,
  SUM(CASE WHEN q = 2 THEN revenue ELSE 0 END) AS q2,
  SUM(CASE WHEN q = 3 THEN revenue ELSE 0 END) AS q3,
  SUM(CASE WHEN q = 4 THEN revenue ELSE 0 END) AS q4,
  MAX(running_total) AS full_year_total
FROM ranked
GROUP BY business_unit
ORDER BY business_unit;

Three stages. quarterly rolls transactions up to one row per unit-quarter. ranked adds a per-unit cumulative sum across quarters. The outer query pivots the four quarters into columns and lifts the final cumulative value with MAX.

4. Strategic explanations

Why the running total must precede the pivot

The cumulative sum walks across quarters in order — it is inherently a row-wise sequential operation. Once the four quarters are flattened into one wide row (q1…q4 columns) that ordering is gone; you can no longer say "sum everything up to Q3". So the window function must run while the data is still long — one row per quarter — in the ranked CTE. Pivot is the last step, after every sequential computation is done. Getting that order right is the core senior insight here.

The window frame — what makes it a running total

SUM(revenue) OVER (PARTITION BY business_unit ORDER BY q ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) is a textbook cumulative sum: PARTITION BY business_unit restarts the total for each unit, ORDER BY q sequences the quarters, and the explicit frame sums from the first row through the current one. The explicit ROWS frame is worth stating even though ORDER BY defaults to a similar range — the default is RANGE, which would lump tied quarters together; ROWS is unambiguous and is the habit to show.

Why MAX(running_total) survives the GROUP BY

The outer query groups by business_unit, so every non-grouped column needs an aggregate. running_total climbs monotonically — each quarter adds a non-negative revenue — so its largest value is the last quarter's, i.e. the full-year total. MAX picks exactly that. It works because the series is non-decreasing; if revenue could be negative (refunds), MAX would grab a mid-year peak instead, and you would carry the total differently — e.g. via the value at the maximum q. Knowing why MAX is valid here is the senior nuance.

ELSE 0 and the missing-quarter trap

South has no Q3 or Q4 rows in quarterly, so without ELSE 0 the SUM(CASE…) for those columns would see no matching rows and return NULL. ELSE 0 makes them a clean 0 — correct for a financial scorecard, where a quarter with no sales genuinely earned zero. Critically, the running total is unaffected: it is computed in ranked over only the quarters that exist, so South's total correctly stops at 200 rather than being dragged down or padded by phantom zero-quarters.

Dialect — window functions and the PIVOT operator

Window functions with explicit frames are SQL-standard and run unchanged in Postgres, SQL Server, Oracle, Snowflake, and BigQuery. The pivot is the portable SUM(CASE…) idiom. A native PIVOT operator (Snowflake, Oracle, SQL Server) could rotate the quarters, but it would not compute the running total — you would still need the window-function CTE first, then PIVOT the result. The pattern "window function in an early CTE, pivot at the end" is identical across dialects; only the pivot syntax of the final step changes.

Senior signal: sequence the pipeline correctly — aggregate, then run the window-function cumulative sum while data is still long, then pivot last — because a running total cannot be computed after quarters collapse into columns; use an explicit ROWS frame, and know that MAX(running_total) recovers the full-year total only because the series is monotonically non-decreasing.
