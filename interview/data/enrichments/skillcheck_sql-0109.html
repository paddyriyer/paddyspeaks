A telecom analytics team wants average call duration broken down by country code, where the code is the prefix of a stored phone number. The subtlety is that one call connects two people, so it must count toward both the caller's country and the callee's country. The interviewer is probing string extraction (pulling the prefix) and the "fan a row into two legs" union pattern.

The edge cases: code 880 is three digits, so a fixed SUBSTRING(phone,1,1) would mangle it — the split must be delimiter-based. And code 1 is shared by two people (Alice, Eve), so its average aggregates legs from multiple persons.

2. The SQL — fan calls into legs, join to country, average

WITH legs AS (
  SELECT caller AS who, duration_sec FROM calls
  UNION ALL
  SELECT callee, duration_sec FROM calls
)
SELECT split_part(p.phone_number, '-', 1) AS country_code,
       ROUND(AVG(l.duration_sec)::numeric, 2) AS avg_duration
FROM legs l
JOIN person p ON p.name = l.who
GROUP BY split_part(p.phone_number, '-', 1)
ORDER BY country_code;

The legs CTE turns each call into two rows — one for the caller, one for the callee — with UNION ALL (not UNION, so identical durations are not collapsed). Each leg is joined to its person to find the country, then grouped by the delimiter-split prefix and averaged.

Note 880 sorts between 86 and 91 — country_code is text, so ordering is lexicographic, not numeric. Code 1 aggregates three legs because two distinct people share that code; this is the "same call may bump two country averages" rule working as intended.

4. Strategic explanations

Why split_part beats fixed-width substring

Country codes are variable length (1, 86, 880). SUBSTRING(phone, 1, 2) would read 88 for Dan and 1- for Alice — both wrong. split_part(phone_number, '-', 1) takes everything up to the first hyphen regardless of length, which exactly matches "the code is the prefix". Delimiter-based parsing is the right tool whenever the field width varies.

UNION ALL vs UNION — a correctness issue, not style

The two legs of one call carry the same duration_sec. Plain UNION deduplicates, so if a caller and callee happened to be in the same country with the same duration, UNION would silently drop one leg and skew the average. UNION ALL keeps every leg. The rule: use UNION ALL unless you specifically want deduplication — here each leg is a distinct fact and must be preserved.

The ::numeric cast before ROUND

AVG over an integer column returns numeric in Postgres already, but ROUND(value, 2) with two arguments is only defined for numeric — calling ROUND(double precision, int) raises "function does not exist". The explicit ::numeric guarantees the two-argument ROUND resolves. It is defensive, and on a column typed double precision it would be mandatory.

Edge cases — single-leg codes and people who never call

Dan appears in exactly one call (as a callee), so code 880 averages a single value — correct, and the query needs no special case for it. A subtler point: the prompt restricts output to codes that "ever appeared in a call". Because the query is driven by legs (derived from calls) and only inner-joins to person, a person who never placed or received a call contributes nothing — their country silently never appears, which is exactly the requirement.

Dialect notes

split_part is PostgreSQL-specific. The portable equivalents: MySQL SUBSTRING_INDEX(phone_number, '-', 1); SQL Server has no direct function pre-2022, you slice with LEFT(phone, CHARINDEX('-', phone) - 1); Oracle SUBSTR(phone, 1, INSTR(phone,'-') - 1); BigQuery SPLIT(phone_number, '-')[OFFSET(0)]. The UNION ALL fan-out and the grouping are fully standard.

Senior signal: extract a variable-length prefix with delimiter-based split_part rather than fixed-width substring, fan each call into caller and callee legs with UNION ALL (never UNION, which would drop equal-duration legs), and remember that a text country code sorts lexicographically so 880 lands before 91.
