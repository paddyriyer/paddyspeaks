A growth team wants the cohort of users who got off to a strong start: their first event was a signup and their second event happened within 24 hours of it — a proxy for activation. The interviewer is probing whether you can pin down the first and second events per user by timestamp order, then compare them, and whether you correctly exclude users who never produced a second event at all.

The fixture is built around the traps: user 2's signup is second (disqualified), user 3's gap exceeds 24 h, user 4's gap is exactly 24 h (the boundary — the rule is "within 24 hours", so it qualifies), and user 5 has a signup but no second event at all.

2. The SQL — rank events per user, pivot the first two, compare

WITH ordered AS (
  SELECT
    user_id,
    event_type,
    ts,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY ts) AS rn
  FROM event
),
first_two AS (
  SELECT
    user_id,
    MAX(CASE WHEN rn=1 THEN event_type END) AS first_type,
    MAX(CASE WHEN rn=1 THEN ts END) AS first_ts,
    MAX(CASE WHEN rn=2 THEN ts END) AS second_ts
  FROM ordered
  WHERE rn <= 2
  GROUP BY user_id
)
SELECT user_id
FROM first_two
WHERE first_type = 'signup'
  AND second_ts IS NOT NULL
  AND EXTRACT(EPOCH FROM (second_ts::timestamp - first_ts::timestamp))/3600 <= 24
ORDER BY user_id;

ordered numbers each user's events by timestamp. first_two collapses the first two into one row per user — first_type, first_ts, second_ts. The final filter keeps users whose first event was a signup and whose second event exists and falls within 24 hours.

4. Strategic explanations

Why ROW_NUMBER to identify "first" and "second"

"First" and "second" are positional, defined by timestamp order — not by the row's id and not by physical storage. ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY ts) assigns each user's events a strict 1, 2, 3… in chronological order. The WHERE rn <= 2 then narrows to just the two events that matter, before the comparison logic runs. This is the standard way to reach the k-th event per group.

The conditional-aggregation pivot — getting two rows side by side

The first and second events are two separate rows; to compare them you need them in one row. MAX(CASE WHEN rn=1 THEN ...) and MAX(CASE WHEN rn=2 THEN ...), grouped by user_id, pivot them into columns. MAX is just the aggregate that survives the GROUP BY — within a user only one row has rn=1, so MAX simply returns its value (the others are NULL and ignored). The alternative is a self-join of ordered to itself on rn=1 vs rn=2; the pivot is tidier.

The second_ts IS NOT NULL guard is the user-5 fix

User 5 has only a signup. In first_two their second_ts is NULL — there was no rn=2 row. Without the explicit second_ts IS NOT NULL check, the time-difference expression would compute NULL - first_ts, the comparison would evaluate to unknown, and the WHERE would correctly drop the row anyway — but relying on that is fragile and unclear. Stating IS NOT NULL makes "must have a second event" an explicit, readable requirement rather than a side effect of three-valued logic.

The 24-hour boundary — <= 24 is deliberate

"Within 24 hours" includes the 24-hour mark, so the predicate is <= 24, not < 24. User 4's second event lands at exactly 24 hours — the test case that distinguishes the two operators. With <= they qualify; with strict < they would be wrongly excluded. Whenever a window is phrased "within X", confirm whether the endpoint counts and pick the operator to match — this is a classic off-by-one.

Dialect — timestamp difference and a LEAD alternative

ts is TEXT, so the cast to timestamp is needed; EXTRACT(EPOCH FROM interval)/3600 converts the gap to hours. SQL Server would write DATEDIFF(HOUR, first_ts, second_ts) (note DATEDIFF counts boundary crossings, which behaves differently — prefer comparing against an interval); MySQL has TIMESTAMPDIFF. A neat one-CTE variant skips the pivot entirely: keep only rn=1 rows and use LEAD(ts) OVER (PARTITION BY user_id ORDER BY ts) to pull the second timestamp onto the first row directly.

Senior signal: identify "first" and "second" events with ROW_NUMBER ordered by timestamp, pivot them onto one row with conditional MAX(CASE ...), explicitly guard second_ts IS NOT NULL so users with no second event are dropped clearly, and make the boundary operator <= 24 because "within 24 hours" includes the endpoint.
