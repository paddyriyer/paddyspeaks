An analytics engineer inherits a wide grades table — one column per subject — and needs it in long form: one row per (person, subject, score). Long form is what charts, joins, and aggregations want, because the subject becomes data instead of schema. This is an unpivot, and most engines have no portable UNPIVOT keyword, so the interviewer is checking whether you can do it with UNION ALL and whether you handle NULL scores correctly.

1. The source data — wide

personmathenglishscience(narrative)
Alice908592all three subjects present
Bob75NULL70no English score recorded

The edge case is Bob's NULL English score. An unpivot must decide: does that become a row with a NULL score, or does it disappear entirely? The two behaviours are different reports, and the choice should be conscious.

2. The SQL — unpivot via UNION ALL

SELECT person, 'english' AS subject, english AS score FROM wide
UNION ALL
SELECT person, 'math', math FROM wide
UNION ALL
SELECT person, 'science', science FROM wide
ORDER BY person, subject;

Each SELECT peels off one subject column, relabels it as a literal subject string, and renames the value column to score. UNION ALL stacks the three result sets vertically; the final ORDER BY sorts the combined output.

Six rows from two input rows — each subject column became its own row. Bob still has an english row; its score is NULL. UNION ALL copies every source row faithfully, including the missing value — the row is kept, the absence is recorded as NULL.

4. Strategic explanations

Why UNION ALL and not UNION

UNION deduplicates its result — it sorts and drops identical rows. If two people scored the same in the same subject, plain UNION would silently collapse them into one row, losing a real record. UNION ALL keeps every row, which is exactly what an unpivot needs: each subject value is its own fact. UNION ALL is also faster — no dedup sort. Using UNION here is a classic, quiet correctness bug.

Column names and types are taken from the first SELECT

The output column names — person, subject, score — come from the first branch; the later branches' aliases are ignored, so they must be in the same order and compatible types. All three value columns are integers here, so the stack is clean. Mixing a TEXT column into the same position would force an implicit cast or an error. Keep the column list identical in shape across every branch.

NULLs survive — decide if that is what you want

Bob's NULL English score becomes a row ('Bob','english',NULL). That is correct for a coverage report — you can see the subject was offered but not graded. If the requirement were "only actual scores", you would add WHERE english IS NOT NULL to that branch (and the equivalent to each). Both are valid; the point is to choose, not to let the default decide silently. Keeping the NULL row is the more informative default.

The ORDER BY applies to the whole union

A trailing ORDER BY after the last branch sorts the combined result, not just the final SELECT — this is what gives the required person-then-subject ordering. Putting ORDER BY inside an earlier branch is either ignored or an error depending on the engine. One sort, at the end, governs everything.

Dialect — dedicated unpivot tools

SQL Server and Oracle have an UNPIVOT operator; Snowflake too. PostgreSQL has no UNPIVOT keyword but offers a tidy alternative — a LATERAL join over a VALUES list: ... CROSS JOIN LATERAL (VALUES ('math', math), ('english', english), ('science', science)) AS u(subject, score), which scans the base table once instead of three times. The UNION ALL form shown here is the most portable and the clearest to read; mention LATERAL as the performance-conscious Postgres variant.

Senior signal: unpivot wide-to-long with UNION ALL (never plain UNION, which would dedupe real rows), keep every branch's column list identical in order and type, decide consciously whether NULL cells become NULL rows or are filtered out, and know LATERAL VALUES as the single-scan Postgres alternative.
