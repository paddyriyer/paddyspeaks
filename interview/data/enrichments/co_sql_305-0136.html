IBM's revenue desk wants the yearly sales report laid out the way a spreadsheet shows it: one row per year, one column per region. The source is long-format — one row per (year, region) — so this is a pivot. The interviewer wants to see the conditional-aggregation idiom and whether you notice what happens to a region with no data.

1. The source data — sales

yearregionamount
2023NA100
2023EU80
2023NA50
2024EU120
2024APAC90
2024NA70

Two things to notice. 2023 has two NA rows (100 and 50) that must be summed, not just picked. And 2023 has no APAC row at all — the edge case that decides whether that cell shows 0 or NULL.

2. The SQL — conditional aggregation

SELECT year,
  SUM(CASE WHEN region='NA'   THEN amount END) AS na,
  SUM(CASE WHEN region='EU'   THEN amount END) AS eu,
  SUM(CASE WHEN region='APAC' THEN amount END) AS apac
FROM sales
GROUP BY year;

Each SUM(CASE …) sums amount only for rows of one region and contributes NULL for the others. GROUP BY year collapses all rows of a year into a single output row — one column per region.

3. The resulting analysis

yearnaeuapac
20247012090
202315080NULL

2023's na is 150 — the two NA rows summed, exactly what conditional aggregation (not a lookup) gets right. And 2023's apac is NULL, not 0: with no APAC row, every CASE returned NULL, and SUM of nothing is NULL. Whether that should be 0 is a product decision.

4. Strategic explanations

The "empty cell" question — NULL or 0?

A NULL apac for 2023 means "no APAC sales were recorded" — which could be genuine ($0 of business) or a data gap. They display differently and behave differently in arithmetic (NULL + 5 is NULL). If the report should read 0, wrap each column:

SELECT year,
  COALESCE(SUM(CASE WHEN region='NA' THEN amount END), 0) AS na,
  COALESCE(SUM(CASE WHEN region='EU' THEN amount END), 0) AS eu,
  COALESCE(SUM(CASE WHEN region='APAC' THEN amount END), 0) AS apac
FROM sales GROUP BY year;

Equivalently, move the zero inside: SUM(CASE WHEN region='APAC' THEN amount ELSE 0 END). Subtle difference — the ELSE 0 form returns 0 even for a year with zero rows entirely, whereas COALESCE wraps the final sum. State which you mean.

Why SUM(CASE…) beats MAX(CASE…)

A common pivot variant uses MAX(CASE WHEN region='NA' THEN amount END). That works only when each (year, region) pair has at most one row. Here 2023 has two NA rows — MAX would return 100 and silently drop the 50. Use SUM when the cell is an additive measure; MAX only for a genuine one-row-per-cell lookup. Picking the wrong one is a silent data-loss bug.

The native FILTER clause — cleaner Postgres

Postgres offers a more readable form of the same idea:

SELECT year,
  SUM(amount) FILTER (WHERE region='NA')   AS na,
  SUM(amount) FILTER (WHERE region='EU')   AS eu,
  SUM(amount) FILTER (WHERE region='APAC') AS apac
FROM sales GROUP BY year;

FILTER is standard SQL and states intent directly. SUM(CASE…) is the portable fallback that works on every engine. Mention both and the trade-off.

The hard-coded-columns limitation

This query bakes NA, EU, APAC into the SQL. A new region — say LATAM — will not appear until someone edits the query; its sales just vanish from the report. SQL pivots are inherently static: the column list must be known at write time. A truly dynamic pivot needs either a code generator that inspects DISTINCT region first, or Postgres's crosstab() from the tablefunc extension. Naming this limitation is the senior move.

NULLs in the amount column itself

Separate from empty cells: if a recorded sales row has amount = NULL, SUM simply skips it — usually fine. But a NULL region is silently excluded from every region column, so its amount disappears from the report entirely. If region can be NULL, add a catch-all column SUM(amount) FILTER (WHERE region IS NULL) AS unknown so the totals still reconcile.

Senior signal: use SUM(CASE…) not MAX so multiple rows per cell aggregate correctly, decide deliberately whether an empty cell is NULL or 0, prefer the FILTER clause for readability in Postgres, and flag that the region columns are hard-coded so a new region needs a query change.
