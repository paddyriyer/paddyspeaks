This approach makes the hierarchy literal: a chain of LEFT JOINs lays the reporting line out sideways — manager, child, grandchild, great-grandchild. Once the tiers are columns, a Senior Manager is a simple shape: someone who has a grandchild but no great-grandchild.

Four tiers run Alice (CEO) → Bob and Carol → Dave, Eve and Frank → the individual contributors. Bob’s team is deliberately mixed — Dave and Eve are Managers, but Karl is an IC.

2. The tier-mapping solution

WITH chain AS (
  SELECT e1.emp_id   AS m_id,
         e1.emp_name AS m_name,
         e2.emp_id   AS child_id,
         e3.emp_id   AS grandchild_id,
         e4.emp_id   AS greatgrandchild_id
  FROM employees e1
  LEFT JOIN employees e2 ON e2.manager_id = e1.emp_id
  LEFT JOIN employees e3 ON e3.manager_id = e2.emp_id
  LEFT JOIN employees e4 ON e4.manager_id = e3.emp_id
)
SELECT m_name AS senior_manager,
       COUNT(DISTINCT child_id) AS direct_reports
FROM chain
WHERE m_id IN (SELECT m_id FROM chain WHERE grandchild_id IS NOT NULL)
  AND m_id NOT IN (SELECT m_id FROM chain WHERE greatgrandchild_id IS NOT NULL)
GROUP BY m_id, m_name
ORDER BY direct_reports DESC, senior_manager ASC;

Each LEFT JOIN steps one level down the org chart. The chain rows expose, for every manager, who their grandchildren and great-grandchildren are — or NULL where the line runs out. A manager stays in the result if some row has a grandchild and no row has a great-grandchild.

4. Strategic explanations

Senior Manager = has a grandchild, lacks a great-grandchild

Translate the definition into the chain. “Manages a Manager” means the manager has a grandchild — a report’s report. “A report is a Senior Manager” means that report manages a Manager — i.e. the manager has a great-grandchild. So a Senior Manager has at least one grandchild and no great-grandchild. Bob reaches Grace, Heidi and Ivan (grandchildren) but no further; Alice reaches a fourth tier, so she is out.

LEFT JOIN is mandatory — INNER would undercount

The chain must use LEFT JOIN. With INNER JOIN, a manager vanishes the moment the chain ends — Bob’s report Karl has no reports, so the Bob–Karl branch would be dropped, and Karl would never be counted among Bob’s direct reports. The NULLs that LEFT JOIN preserves are the signal that a tier is absent.

NOT IN is safe here — because m_id cannot be NULL

NOT IN is notorious for breaking when its subquery yields a NULL. Here it is safe: m_id is e1.emp_id, a non-nullable key, so the subquery list never contains NULL. Knowing why it is safe is the point — if that column could be NULL, this would have to become NOT EXISTS.

COUNT(DISTINCT child_id), not COUNT(*)

The join fans out: Bob→Dave appears twice in the chain, once via Grace and once via Heidi. COUNT(*) would count chain rows and badly inflate the team size. COUNT(DISTINCT child_id) counts each direct report once — the correct three for Bob.

Senior signal: a fixed-depth hierarchy question can be flattened into a chain of LEFT JOINs, turning “tiers” into columns you can test for NULL. Keep the joins LEFT so partial chains survive, and de-duplicate join fan-out with COUNT(DISTINCT ...).
