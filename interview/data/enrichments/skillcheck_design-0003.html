An accumulating snapshot fact table models a multi-stage business process where a single row tracks an entity (here, an order) through every milestone from birth to resolution. The interviewer is probing three things: whether you know when to choose an accumulating snapshot over a transaction fact, how to represent nullable milestone timestamps and their corresponding dimension date FKs, and how derived duration measures relate to those timestamps. Getting the “one row per order, updated in place” grain right is the crux.

1. The requirements — what the design must capture

  • Grain: one row per order, representing the order’s full lifecycle — not one row per event.
  • Milestones: placed → paid → shipped → delivered → returned. Each milestone may or may not have occurred.
  • Milestone date FKs: integer FKs into dim_date for each milestone, allowing calendar-dimension joins (e.g., “orders delivered on a Sunday”).
  • Milestone timestamps: full TIMESTAMP precision for sub-day analysis (e.g., “average hours from placed to shipped”).
  • Duration measures: computed elapsed-time measures between milestone pairs; NULL until both endpoints exist.
  • Additive measures: order amount and refunded amount.
  • Current status: a snapshot of the order’s latest state.
  • Row update timestamp: updated_at to track when the row was last mutated by the ETL.

2. The DDL — the CREATE TABLE

CREATE TABLE fact_order_fulfillment (
  order_sk            BIGINT      NOT NULL,         -- surrogate PK
  order_nk            VARCHAR(64) NOT NULL,         -- natural key (order_id)
  customer_sk         BIGINT      NOT NULL,
  product_sk          BIGINT      NOT NULL,
  -- milestone date FKs (NULL until milestone occurs)
  placed_date_sk      INT         NOT NULL,
  paid_date_sk        INT,
  shipped_date_sk     INT,
  delivered_date_sk   INT,
  returned_date_sk    INT,
  -- milestone timestamps for sub-day analysis
  placed_ts           TIMESTAMP   NOT NULL,
  paid_ts             TIMESTAMP,
  shipped_ts          TIMESTAMP,
  delivered_ts        TIMESTAMP,
  returned_ts         TIMESTAMP,
  -- duration measures (computed, NULL until both endpoints set)
  hours_to_pay        DECIMAL(10,2),
  hours_to_ship       DECIMAL(10,2),
  hours_to_deliver    DECIMAL(10,2),
  -- additive measures
  order_amount        DECIMAL(12,2),
  refunded_amount     DECIMAL(12,2),
  current_status      VARCHAR(20),                   -- placed / paid / shipped / delivered / returned
  updated_at          TIMESTAMP   NOT NULL,
  PRIMARY KEY (order_sk)
);

4. Strategic explanations

Why an accumulating snapshot, not a transaction fact

A transaction fact inserts one row per event (order placed, order paid, order shipped) and never updates existing rows. That design is ideal for event-level analysis but requires awkward self-joins or window functions to compute “time from placed to shipped” across separate rows. An accumulating snapshot consolidates all milestones for one business process into a single row that is updated in place as milestones occur. The tradeoff: ETL must find and update the existing row (no append-only simplicity), but queries that span milestones become trivial single-row reads.

Dual milestone representation: date SK + timestamp

Each milestone appears twice: as an INT FK into dim_date and as a full TIMESTAMP. The date SK enables fast joins to the date dimension for calendar-based grouping (quarter, weekday). The timestamp supports sub-day calculations (hours, minutes) and SLA breach detection. Storing both is not redundancy — they serve different query patterns. The date SK is derived from the timestamp during ETL.

Duration measures: semi-additive and pre-computed

hours_to_pay and siblings are semi-additive: averaging across orders is valid, summing is not (sum of durations is meaningless). They are stored pre-computed because duration calculation (TIMESTAMPDIFF) is non-trivial in columnar queries, and the values are needed in nearly every fulfillment report. Storing them trades a little storage for significant query simplicity. NULL while either endpoint is missing — ETL must handle the partial-update case.

Handling updates in the data warehouse

Most analytical platforms are append-optimised, but accumulating snapshots require in-place UPDATEs. In Redshift and BigQuery, the pattern is a MERGE (UPSERT) on the natural key. In Snowflake, MERGE works natively. In BigQuery streaming inserts, a common workaround is to maintain a separate events log and materialise the snapshot as a view or scheduled MERGE query. The updated_at column enables incremental MERGE by acting as a change watermark.

Partition strategy

Partition on placed_date_sk (or an equivalent date column derived from placed_ts) so that historical analysis of order cohorts scans only relevant partitions. Clustering on customer_sk or product_sk accelerates the most common analytical slice. In Snowflake, CLUSTER BY (placed_date_sk, current_status) also prunes scans for “all open orders from last month.”

Fact-table type in the Kimball taxonomy

Kimball names three fact-table types: transaction (one row per event), periodic snapshot (one row per entity per time period), and accumulating snapshot (one row per business process lifecycle, updated). This table is an accumulating snapshot. Interviewers expect you to name the type and justify it against the alternatives, not just produce DDL.

Senior signal: name the fact-table type (“accumulating snapshot”), justify it vs. a transaction fact (“milestone-spanning queries would require self-joins across event rows”), explain dual milestone representation (date SK for calendar joins, timestamp for duration math), classify duration measures as semi-additive, and discuss the MERGE/UPSERT requirement that accumulating snapshots impose on the ETL pipeline — these four points together show senior-level dimensional modeling depth.
