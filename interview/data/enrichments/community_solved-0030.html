This is the StrataScratch framing of the average-session-time problem from a Community question bank. Here the table is already shaped one row per visit, with page_load and page_exit sitting side by side as text columns. The work shifts from event-pairing to data hygiene: those text columns may hold non-timestamp junk, and casting them blindly is how the query crashes in front of the interviewer.

User 3's visit has a load but a NULL exit — an incomplete session. User 4's row is NULL on both sides — an empty visit. Both must be excluded before any subtraction happens.

2. The SQL — cast-and-filter per visit

SELECT user_id,
       AVG(EXTRACT(EPOCH FROM (page_exit::timestamp - page_load::timestamp))) AS avg_duration
FROM users_web_log
WHERE page_load IS NOT NULL AND page_exit IS NOT NULL
  AND page_load::text ~ '^[0-9]{4}-[0-9]{2}-[0-9]{2}'
  AND page_exit::text ~ '^[0-9]{4}-[0-9]{2}-[0-9]{2}'
GROUP BY user_id;

The first two predicates drop rows where either side is NULL. The two regex predicates are a defensive guard: only rows whose text actually starts like a date survive to be cast, so a stray label or empty string can never reach ::timestamp and abort the query.

User 1 has two visits — 600 s and 1800 s — averaging 1200 s. User 2 has one 1200 s visit. Users 3 and 4 produced no row at all: the NULL filter removed them, so they are absent from the output rather than appearing with a NULL or 0 average.

4. Strategic explanations

Why the regex guard exists

When timestamps are stored as TEXT, the column can accumulate anything — '', 'n/a', a malformed date. 'n/a'::timestamp does not return NULL; it raises an error that kills the whole statement. The pattern '^[0-9]{4}-[0-9]{2}-[0-9]{2}' screens each value before the cast, so only plausibly-castable rows proceed. It is belt-and-braces, and worth a sentence to the interviewer: "I'm assuming the text column isn't clean."

The cleaner fix — store timestamps as timestamps

The regex is a workaround for a schema smell. If you own the table, the real fix is a typed column:

ALTER TABLE users_web_log
  ALTER COLUMN page_load TYPE timestamp USING page_load::timestamp;

Then the cast and the regex both disappear and the database enforces validity on write. Postgres 12+ also offers a non-throwing path — a CASE wrapping the cast, or a helper that returns NULL on failure — if you cannot change the type.

Why the NULL filter is not redundant

page_exit IS NOT NULL looks superfluous next to the regex, but it is not: a NULL value never matches a regex (the comparison is unknown, not false), so the regex alone would already exclude NULLs — yet stating IS NOT NULL explicitly documents intent and makes the filter readable. Keep it; clarity beats terseness here.

Subtracting timestamps yields an interval

page_exit::timestamp - page_load::timestamp is an INTERVAL, not a number. EXTRACT(EPOCH FROM ...) converts that interval to total seconds — the only reliable way to average durations. Averaging intervals directly works in Postgres but is fragile across engines; epoch-seconds is portable. Divide by 60 for minutes if the dashboard wants those units.

Edge case: a user with only bad rows

If every one of a user's visits is filtered out, that user simply does not appear — GROUP BY produces no group for them. If the dashboard needs every user listed, even those with no valid session, start from a distinct user list and LEFT JOIN the durations onto it, leaving avg_duration NULL where there is nothing to average.

Senior signal: recognise that casting a TEXT timestamp column is unsafe and either guard it with a regex or fix the schema, explain why a NULL never matches a regex (so the explicit IS NOT NULL is for readability), and convert interval differences to epoch-seconds for a portable, averageable duration.
