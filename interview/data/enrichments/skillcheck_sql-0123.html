A social product wants a "Pages you might like" rail for user 1: surface every page that any of user 1's friends has liked, minus the pages user 1 already likes. The interviewer plants two traps. Friendships are stored only one direction (an undirected edge written once), so you must reconstruct user 1's full friend list from both columns; and you must subtract what the user already likes.

The edge cases: user 1's friends (2 and 3) both happen to sit in the user1_id column — but the query must still UNION the user2_id direction or it would miss any friendship written the other way. And page 101 is liked by friend 2 and by user 1 — it must be filtered out as already-liked.

2. The SQL — build the friend set, join likes, subtract own likes

WITH friends AS (
  SELECT user2_id AS friend FROM friendship WHERE user1_id = 1
  UNION
  SELECT user1_id        FROM friendship WHERE user2_id = 1
)
SELECT DISTINCT l.page_id AS recommended_page
FROM friends f
JOIN likes l ON l.user_id = f.friend
WHERE l.page_id NOT IN (SELECT page_id FROM likes WHERE user_id = 1)
ORDER BY recommended_page;

The friends CTE collects user 1's neighbours from both sides of the undirected edge. The main query gathers every page those friends like, then NOT IN removes pages user 1 already likes; DISTINCT collapses pages liked by several friends.

4. Strategic explanations

Why the friend set needs a UNION of both columns

An undirected friendship stored once can have user 1 on either side. WHERE user1_id = 1 finds friendships where 1 is listed first; WHERE user2_id = 1 finds the rest. Querying only one column would silently miss half of user 1's real friends. UNION (not UNION ALL) merges the two directions and also dedupes — harmless here, but it protects against a pathological (1,1) self-row or a duplicated edge.

Depth 1 only — this is not a recursive traversal

The recommendation is "pages my friends like" — one hop. User 5, a friend of friend 3, is deliberately excluded. The single friends CTE plus one join expresses exactly depth 1. If the spec said "friends-of-friends" you would need a second hop (another join, or a recursive CTE with a depth cap); recognising that this question stops at one hop — and not over-engineering a recursive solution — is part of the read.

The NOT IN NULL trap

NOT IN (SELECT page_id FROM likes WHERE user_id = 1) is correct here because page_id is non-nullable. But it is a known footgun: if that subquery ever returned a single NULL, the entire NOT IN would evaluate to NULL for every row and the query would return nothing. The NULL-safe rewrite is NOT EXISTS (SELECT 1 FROM likes l1 WHERE l1.user_id = 1 AND l1.page_id = l.page_id). Mentioning the trap — and that NOT EXISTS sidesteps it — is the senior move.

Why DISTINCT on the output

If two of user 1's friends both like the same page, the join yields that page twice. The rail wants each page once, so DISTINCT is required on page_id. An equivalent shape is to wrap the friend-likes in their own set and use EXCEPT against user 1's likes: (SELECT page_id FROM likes WHERE user_id IN (friends)) EXCEPT (SELECT page_id FROM likes WHERE user_id = 1) — EXCEPT dedupes and subtracts in one operator, and is NULL-safe, making it arguably the cleanest phrasing.

Senior signal: reconstruct an undirected friend list by UNION-ing both edge columns, keep the traversal at exactly one hop (friends, not friends-of-friends), DISTINCT the output because multiple friends can like the same page, and flag the NOT IN/NULL trap — offering NOT EXISTS or EXCEPT as the safe alternative.
