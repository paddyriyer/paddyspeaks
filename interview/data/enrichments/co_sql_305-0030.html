Shopify's fulfilment team tracks how long each order takes to leave the warehouse — "days to ship" is a core SLA metric. The query is a single subtraction, and that is exactly why an interviewer uses it: the arithmetic is trivial, so the conversation is entirely about what the data does to that arithmetic.

1. The source data — orders

Three rows are landmines. Order 4 has a NULL ship_date (not shipped yet). Order 2 shipped the same day it was ordered. Order 5 has a ship_date before its order_date — dirty data that should not physically exist.

2. The SQL — ship_date − order_date

SELECT order_id, ship_date - order_date AS days_to_ship FROM orders;

In PostgreSQL, subtracting one date from another returns an integer — a count of days — not an interval. No casting needed.

Order 2 returns 0 — correct, a same-day ship. Order 4 returns NULL: NULL minus a date is NULL, the right answer for "not yet shipped". Order 5 returns −1 — a negative SLA, which is a data-quality flag, not a real metric.

4. Strategic explanations

date − date is an integer; timestamp − timestamp is not

The clean integer here depends on both columns being date type. If they were timestamp, the same subtraction returns an interval like 3 days 04:30:00, and a downstream consumer expecting a number breaks. When you need whole days from timestamps, be explicit:

SELECT order_id,
       (ship_date::date - order_date::date)        AS days_to_ship,
       EXTRACT(DAY FROM ship_date - order_date)    AS days_alt
FROM orders;

Casting to date first truncates the time-of-day and restores integer subtraction. Naming the column types out loud is the senior tell.

The negative value — the trap the interviewer is fishing for

Order 5's −1 is the moment that matters. A negative "days to ship" is impossible in the real world, so it signals a bad ship_date (a typo, a timezone slip, a backfill error). Don't let it poison an average. Either filter it or surface it as a quality column:

SELECT order_id, ship_date - order_date AS days_to_ship
FROM orders
WHERE ship_date >= order_date;          -- drop impossible rows
-- or flag them, never silently average them in:
SELECT order_id,
       ship_date - order_date AS days_to_ship,
       (ship_date < order_date) AS is_suspect
FROM orders;
The NULL row and how it behaves in aggregates

Order 4's NULL is correct per-row, but watch what it does downstream. AVG(days_to_ship) skips NULLs — the average is over shipped orders only, which is usually what an SLA wants. But COUNT(*) still counts the pending row, so SUM(days_to_ship) / COUNT(*) would be wrong. If "unshipped" should read as zero days, use COALESCE(ship_date, CURRENT_DATE) - order_date to measure age-so-far instead.

Business days, not calendar days

A frequent follow-up: "the warehouse is closed weekends — can you exclude them?" Plain subtraction counts calendar days. Business-day logic needs a calendar/dimension table joined on the date range, or a generate-series trick counting non-weekend days. Naming this limitation — "this is calendar days; business-day SLA needs a date dimension" — shows you understand the metric, not just the operator.

Inclusive vs exclusive day counts

Order 2 ships the same day and scores 0. Some SLA definitions call that "1 day" (the day itself counts). Confirm the convention: if shipping must be reported as days + 1, the consumer needs to know whether the boundary is inclusive. It is a one-character change with a real reporting impact.

Senior signal: know that date − date yields an integer while timestamp − timestamp yields an interval, treat the negative result as a data-quality flag rather than a metric, explain how the NULL row flows through AVG versus COUNT(*), and raise the calendar-vs-business-day distinction before it is asked.
