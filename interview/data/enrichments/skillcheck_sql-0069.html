A CRM team is cleaning a contacts table before a marketing send. The raw data is messy — the same person was entered three times with different casing and stray spaces — and they want a deduplication report grouping records that are really the same after normalization. The interviewer is testing whether you can build a robust normalization key with string functions and a regex, then group on that derived key rather than the raw columns.

Row 2 carries every flavour of mess at once: uppercase, an internal double space in the name, and a trailing space. Carol is the negative control — a unique contact that the HAVING clause must filter out, proving the query reports only genuine duplicate groups.

2. The SQL — normalize in a CTE, then group

WITH n AS (
  SELECT id,
         LOWER(TRIM(email)) AS normalized_email,
         LOWER(REGEXP_REPLACE(TRIM(name), '\s+', ' ', 'g')) AS normalized_name
  FROM contacts
)
SELECT normalized_name, normalized_email,
       STRING_AGG(id::text, ',' ORDER BY id) AS ids,
       COUNT(*) AS n
FROM n
GROUP BY normalized_name, normalized_email
HAVING COUNT(*) >= 2
ORDER BY n DESC, normalized_email, normalized_name;

The n CTE builds the comparison key once. TRIM strips outer whitespace, LOWER kills case, and REGEXP_REPLACE(..., '\s+', ' ', 'g') collapses any run of internal whitespace to a single space. The outer query groups on those two normalized columns and keeps only groups of size 2 or more.

4. Strategic explanations

Why normalize in a CTE instead of inline

The normalized expressions are needed in three places: the SELECT list, the GROUP BY, and (implicitly) the ORDER BY. Computing them once in a CTE keeps the query DRY, guarantees all three sites use identical logic, and makes the intent readable. Repeating a long LOWER(REGEXP_REPLACE(TRIM(...))) expression in the GROUP BY invites a subtle typo where the grouping key drifts from the projected key.

The 'g' flag is essential

REGEXP_REPLACE in Postgres replaces only the first match unless you pass the 'g' (global) flag. Without it, a name with two separate double-spaces would only have the first one collapsed, leaving the key inconsistent. The pattern \s+ also matters: it matches a run of whitespace, so a triple space becomes one space in a single substitution — covering tabs and newlines too, not just literal spaces.

Order of operations — TRIM before collapse, LOWER last

TRIM first removes the outer padding so the regex only deals with interior whitespace; you could let \s+ handle the edges too, but trimming first keeps intent obvious. LOWER wraps the whole thing because casefolding doesn't interact with whitespace — the order of LOWER vs TRIM is interchangeable, but applying it to the already-collapsed string is tidy. The key principle: every record must pass through the exact same transformation pipeline.

Why COUNT(*) in HAVING and not COUNT(DISTINCT id)

Here id is unique per row, so COUNT(*) and COUNT(DISTINCT id) coincide. COUNT(*) is the right choice — it counts rows in the group, which is exactly "how many contacts share this normalized identity". The HAVING runs after grouping, which is why it can filter on the aggregate; a WHERE clause cannot reference COUNT(*) because WHERE is evaluated before rows are grouped.

Dialect — string aggregation has many spellings

STRING_AGG(id::text, ',' ORDER BY id) is Postgres. MySQL uses GROUP_CONCAT(id ORDER BY id SEPARATOR ','); Oracle uses LISTAGG(id, ',') WITHIN GROUP (ORDER BY id); SQL Server uses STRING_AGG(CONVERT(varchar,id), ',') WITHIN GROUP (ORDER BY id). The ORDER BY inside the aggregate is what makes the output deterministic — without it, the id list order is undefined and two runs could disagree.

Senior signal: build a normalization key (LOWER + TRIM + REGEXP_REPLACE(..., '\s+', ' ', 'g')) once in a CTE so every record passes the identical pipeline, group on the derived key not the raw columns, filter duplicate groups with HAVING COUNT(*) >= 2, and always put ORDER BY inside STRING_AGG for a deterministic id list.
