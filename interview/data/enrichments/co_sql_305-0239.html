Amazon's supply-chain team wants a reorder alert: for each product, how many days of stock remain at the current sales pace, surfacing only the ones about to run out (under 14 days). The query divides on-hand quantity by daily sales velocity. The arithmetic is trivial — but the reference solution wraps every number in a regexp_replace, and the interviewer wants to know why, and what happens when velocity is zero.

Product 4 is already out of stock (qty 0). Product 5 has a daily_avg of 0 — it does not sell, so "days left" is a division by zero. Those two rows are the interview: one is an honest 0, the other must not crash the query.

2. The SQL — qty / daily_avg, with regexp scrub

SELECT i.product_id,
  NULLIF(regexp_replace(i.qty::text, '[^-.0-9]', '', 'g'), '')::numeric
  / NULLIF(NULLIF(regexp_replace(s.daily_avg::text, '[^-.0-9]', '', 'g'), '')::numeric, 0)
  AS days_left
FROM inventory i
JOIN sales_velocity s ON i.product_id = s.product_id
WHERE NULLIF(regexp_replace(i.qty::text, '[^-.0-9]', '', 'g'), '')::numeric
  / NULLIF(NULLIF(regexp_replace(s.daily_avg::text, '[^-.0-9]', '', 'g'), '')::numeric, 0)
  < 14;

The regexp_replace(…, '[^-.0-9]', '', 'g') strips every character that is not a digit, minus or dot — defensive cleaning in case qty arrived as text like "50 units". The inner NULLIF(…, 0) on the divisor is the guard against division by zero.

3. The resulting analysis

Product 1: 50 / 10 = 5 days. Product 3: 12 / 3 = 4 days. Product 4: 0 / 5 = 0 days — correctly flagged, it is out of stock now. Product 2 (200 / 8 = 25 days) is comfortably stocked and excluded by the < 14 filter. Product 5 is silently dropped: NULLIF(daily_avg, 0) turned its divisor into NULL, the division yielded NULL, and NULL < 14 is UNKNOWN — so it fails the WHERE. Round the high-precision result with ROUND(…, 1) for a dashboard.

4. Strategic explanations

Division by zero — the NULLIF guard

Without the inner NULLIF(divisor, 0), product 5 would raise division by zero and abort the whole query. NULLIF(x, 0) converts a zero divisor to NULL, and dividing by NULL yields NULL rather than an error. The behavioural consequence: a non-selling product reports an unknown runway, not an infinite one — arguably correct (you cannot forecast a stockout for something that never sells), but the interviewer wants you to name that product 5 vanishes, and decide whether a non-seller should instead surface as "never runs out".

Why the regexp_replace — defensive or paranoid?

If qty and daily_avg are already numeric columns — as the schema implies — the regex scrubbing is dead weight: it casts a number to text, strips nothing, and casts back. It only earns its place if those columns are actually TEXT holding values like "50 units" or "1,200". On clean numeric columns the query simplifies dramatically:

SELECT i.product_id, i.qty::numeric / NULLIF(s.daily_avg, 0) AS days_left
FROM inventory i
JOIN sales_velocity s ON i.product_id = s.product_id
WHERE i.qty::numeric / NULLIF(s.daily_avg, 0) < 14;

Knowing when defensive parsing is justified versus cargo-culted is a senior judgement call — ask what type the columns really are.

Don't repeat the expression — CTE or lateral

The solution computes the identical division twice — once in SELECT, once in WHERE. That is error-prone (edit one, forget the other) and you cannot reference a SELECT alias in WHERE. Compute once, filter after:

SELECT * FROM (
  SELECT i.product_id, i.qty::numeric / NULLIF(s.daily_avg, 0) AS days_left
  FROM inventory i JOIN sales_velocity s ON i.product_id = s.product_id
) t
WHERE days_left < 14;
The join type — missing velocity rows

The INNER JOIN drops any product that exists in inventory but has no sales_velocity row — a brand-new SKU with no sales history. That product never gets a reorder alert, which is a real blind spot. A LEFT JOIN keeps it (with days_left NULL); decide whether unknown-velocity products should be reviewed manually rather than silently ignored.

Senior signal: guard the divisor with NULLIF(x, 0) and explicitly note that zero-velocity products then drop out of the alert, question whether the regexp_replace scrubbing is warranted or cargo-culted given the column types, compute the division once instead of duplicating it across SELECT and WHERE, and flag that an inner join blinds the alert to products with no sales history.
