Datadog's #1 customer pain: alert fatigue. Auto-resolved alerts = false positives. The query classifies monitors into noisy / flaky / actionable.

2. The SQL — noise classification

SELECT monitor_id, COUNT(*) AS fires, SUM(acknowledged) AS acknowledged,
  SUM(auto_resolved) AS auto_resolved,
  ROUND(100.0 * SUM(auto_resolved) / NULLIF(COUNT(*), 0), 1) AS noise_ratio_pct,
  CASE
    WHEN 100.0 * SUM(auto_resolved) / NULLIF(COUNT(*), 0) >= 70 THEN 'noisy'
    WHEN 100.0 * SUM(auto_resolved) / NULLIF(COUNT(*), 0) >= 30 THEN 'flaky'
    ELSE 'actionable'
  END AS quality
FROM alerts GROUP BY monitor_id HAVING COUNT(*) >= 3
ORDER BY noise_ratio_pct DESC;

3. Strategic explanations

The 70%/30% calibrated cutoffs

From Datadog's published guidelines: 70%+ auto-resolved = monitor needs threshold tuning OR removal. 30-70% = flaky underlying signal. Below 30% = actionable signal.

The acknowledged-but-not-fixed pattern

Acknowledged alerts that auto-resolve (someone noticed but didn't fix) ≠ true acknowledged. Production splits.

The "snooze creep"

Engineers snooze noisy alerts. Each snooze hides the issue but doesn't fix it. Production tracks "snooze count per monitor" — alarming if growing.

The MTTR cousin metric

Mean Time to Resolution = avg time from fire to ack-or-resolve. Combined with noise-rate gives full alert quality picture.

The on-call burden

Datadog dashboards show "alerts per on-call shift". Above 10 = burning out the team. Below 2 = monitors not catching real issues.

Senior signal: 70/30 noise cutoffs, distinguish ack-without-fix from true ack, watch snooze creep as a hidden problem, pair with MTTR for full picture, and tie to on-call burden metric.
