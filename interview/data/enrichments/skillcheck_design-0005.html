A junk dimension consolidates several low-cardinality flag columns into a single pre-enumerated lookup table, replacing multiple FK columns on the fact table with one. The interviewer is probing whether you know why junk dimensions exist (reduced fact-table width, centralised attribute management), what their cardinality ceiling is, and how to enforce completeness with a UNIQUE constraint on the combination. The question also asks for the referencing pattern, so your answer must include how fact_order uses it.

1. The requirements — what the design must capture

  • Flags to consolidate: channel (4 values), payment_status (4 values), is_gift (2 values), is_fraud_reviewed (2 values).
  • Maximum distinct combinations: 4 × 4 × 2 × 2 = 64 rows — a tiny, cacheable table.
  • Surrogate PK: order_junk_sk is the single FK placed on the fact table.
  • Uniqueness: the combination of all four flags must be unique; enforced by a UNIQUE constraint on the four columns.
  • Fact-table reference: one FK column replaces four separate columns on fact_order.

2. The DDL — the CREATE TABLE

CREATE TABLE dim_order_junk (
  order_junk_sk        BIGINT      NOT NULL,
  channel              VARCHAR(20) NOT NULL,        -- web / mobile / store / partner
  payment_status       VARCHAR(20) NOT NULL,        -- pending / paid / refunded / chargeback
  is_gift              BOOLEAN     NOT NULL,
  is_fraud_reviewed    BOOLEAN     NOT NULL,
  PRIMARY KEY (order_junk_sk),
  UNIQUE (channel, payment_status, is_gift, is_fraud_reviewed)
);

-- fact_order carries a single FK column: order_junk_sk BIGINT NOT NULL.
-- One row per distinct combination of the flags (~4 * 4 * 2 * 2 = 64 rows max).
-- Avoids four extra columns on every fact row; queries filter / group by
-- attributes on the junk dim instead.

4. Strategic explanations

Why junk dimensions exist

Without a junk dimension, each flag lands as its own column on the fact table: channel VARCHAR(20), payment_status VARCHAR(20), is_gift BOOLEAN, is_fraud_reviewed BOOLEAN. For a fact table with hundreds of millions of rows, those four columns add up to tens of gigabytes of redundantly repeated string values. The junk dimension replaces them with one 8-byte integer FK per fact row. Beyond storage, it also centralises the controlled vocabulary: new channel values are added to the dimension, not directly to every tool that reads the fact table.

Cardinality ceiling and when junk dimensions are appropriate

The pattern is only viable when the cross product of all flags is small enough to pre-populate. Here: 4 × 4 × 2 × 2 = 64 rows — trivially small. If one flag had 1,000 values the cross product would be 16,000 rows; still manageable. If you reach millions of combinations, the junk dimension is no longer efficient and you should revert to individual FK columns or a sparse attribute table. As a rule of thumb, junk dimensions work when the combination count stays below ~10,000.

Population strategy

The junk dimension can be fully pre-populated at load time (64 INSERT statements for all combinations), or populated lazily as the ETL encounters new combinations. The UNIQUE constraint enables the lazy approach: INSERT the combination, let the constraint reject duplicates, and use ON CONFLICT DO NOTHING RETURNING order_junk_sk in PostgreSQL (or an equivalent MERGE in other engines) to retrieve the SK for whichever path executed. Pre-population is simpler and avoids concurrency issues in high-throughput ETL.

Query pattern

Fact queries filter and group on junk-dimension attributes with a single join: JOIN dim_order_junk j ON j.order_junk_sk = f.order_junk_sk WHERE j.channel = ‘mobile’ AND j.is_fraud_reviewed = TRUE. Because the junk table has at most 64 rows, the join result set is tiny and the planner will choose a hash join or nested loop with an index scan — effectively free.

What should NOT go into a junk dimension

Attributes that change slowly over time (and need SCD2 versioning) do not belong in a junk dimension — put them in a proper slowly-changing dimension. High-cardinality attributes (customer region, product category) should get their own dimension. The junk dim is specifically for flags and codes that are: low-cardinality, not independently analysed, and not slowly changing.

Warehouse dialect notes

In BigQuery there are no enforced UNIQUE constraints; document the invariant and enforce it in ETL. In Redshift, declare the UNIQUE constraint for documentation but note it is not enforced; rely on a pre-load deduplication step. In Snowflake and PostgreSQL, the UNIQUE constraint is enforced and enables the UPSERT pattern reliably.

Senior signal: state the maximum combination count (4 × 4 × 2 × 2 = 64) to justify the pattern; explain the storage benefit (“one 8-byte integer per fact row replaces ~60 bytes of repeated string values”); describe the lazy-UPSERT population strategy enabled by the UNIQUE constraint; and articulate the limits of the pattern (“if the cross product exceeded ~10,000 rows I’d revert to individual FK columns or a sparse attribute table”) — knowing the pattern’s failure mode is as important as knowing the pattern itself.
