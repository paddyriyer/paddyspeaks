A merchandising analyst wants to know, for every product category, what the average unit price is, how many order line items have been placed, and how many units have been sold. The interviewer is testing whether you correctly use a LEFT JOIN to keep categories that have never been ordered, understand the difference between COUNT and SUM in this context, and can explain why AVG(p.price) is per product — not per order line.

Product 50 (T-Shirt, Apparel) has no matching rows in order_items. The LEFT JOIN keeps it in the result with NULL aggregates, which the query exposes without COALESCE — worth noting.

2. The SQL — LEFT JOIN aggregation by category

SELECT p.category,
       AVG(p.price)       AS avg_price,
       COUNT(oi.order_id) AS order_count,
       SUM(oi.qty)        AS units_sold
FROM products    p
LEFT JOIN order_items oi ON p.product_id = oi.product_id
GROUP BY p.category
ORDER BY p.category;

4. Strategic explanations

LEFT JOIN to preserve unordered categories

An INNER JOIN would drop the Apparel category entirely because product 50 has no matching order_item rows. The LEFT JOIN preserves all products (and therefore all categories) in the result. The GROUP BY p.category then collapses products within each category. This is the canonical "show all groups, even empty ones" pattern.

AVG(p.price) is a product-level average, not a sales-weighted average

There are 3 Electronics products (Widget $25, Gadget $50, Cable $8). AVG(p.price) is (25 + 50 + 8) / 3 = $27.67 — the simple average of the product catalogue prices, one row per product after the join. It is not weighted by sales volume. A sales-weighted average would be SUM(s.amount) / SUM(s.qty) from a sales table. Candidates who confuse these two are answering a different question.

COUNT vs SUM for different metrics

COUNT(oi.order_id) counts the number of order line items (one row per product per order). SUM(oi.qty) sums the quantities on each line. Both are correct but measure different things: order frequency vs units moved. Note that COUNT(oi.order_id) returns 0 when all oi.order_id values are NULL (the LEFT JOIN NULL case), whereas SUM(oi.qty) returns NULL. Wrapping the SUM in COALESCE(SUM(oi.qty), 0) makes both metrics consistently zero for empty categories.

GROUP BY on non-aggregated columns

The SELECT includes p.category, a non-aggregated expression. It must appear in the GROUP BY. A common mistake is to also put p.price in the SELECT without aggregating it — that would force grouping at the product level, breaking the category rollup. Always check: every column in SELECT that is not inside an aggregate function must be in GROUP BY.

Dialect notes

AVG, COUNT, SUM, and LEFT JOIN are ANSI SQL and identical across Snowflake, PostgreSQL, BigQuery, and MySQL. In BigQuery you must use FLOAT64 division for AVG on integer price columns (handled automatically by BigQuery's type inference). MySQL's GROUP BY historically allowed non-aggregated columns without including them in the clause (the infamous ONLY_FULL_GROUP_BY mode off by default pre-5.7) — always verify that mode is enabled.

Senior signal: explain that LEFT JOIN is required to keep zero-order categories, distinguish AVG(p.price) (catalogue average) from a sales-weighted average (SUM(revenue) / SUM(qty)), and call out that SUM returns NULL while COUNT returns 0 for empty groups — requiring COALESCE if you want consistent zeros across both columns.
