A data-quality audit needs to surface duplicate email addresses before a unique constraint can be added — and "duplicate" must be case-insensitive, because alice@x.com and Alice@X.com are the same mailbox. The interviewer is probing whether you normalise before grouping, and whether you know HAVING is where a per-group count filter belongs.

1. The source data — users

idemail(narrative)
1alice@x.comsame mailbox as id 3, different casing
2BOB@y.comsame mailbox as id 5, different casing
3Alice@X.comcase-variant duplicate of id 1
4carol@z.comunique — appears once
5bob@y.comcase-variant duplicate of id 2

The edge case is the casing itself. A naive exact-match GROUP BY email sees five distinct strings and reports zero duplicates — the bug the question is built to catch. Carol is the control: a genuinely unique address that must not appear in the output.

2. The SQL — group by the normalised email

SELECT
  LOWER(email) AS email_lower,
  COUNT(*) AS occurrences
FROM users
GROUP BY LOWER(email)
HAVING COUNT(*) > 1
ORDER BY occurrences DESC, email_lower;

LOWER(email) folds every address to a single canonical case before grouping, so alice@x.com and Alice@X.com land in the same bucket. GROUP BY LOWER(email) forms one group per distinct mailbox; COUNT(*) sizes each; HAVING COUNT(*) > 1 keeps only the buckets that hold a collision.

Two duplicate mailboxes surface; carol@z.com is correctly absent because its single occurrence fails HAVING COUNT(*) > 1. Both reported rows show the lowercased form — the GROUP BY key — not whichever mixed-case spelling happened to be stored, which is the right thing for a normalised audit report.

4. Strategic explanations

Normalise before grouping — not after

Grouping is an equality test on the group key. If the key is the raw email, equality is case-sensitive and Alice@X.com ≠ alice@x.com, so the duplicate hides. Applying LOWER inside the GROUP BY makes the key itself canonical, so case variants collapse into one bucket. The principle generalises: whenever "same" means something looser than byte-for-byte equality — trimmed whitespace, case, accent-folding — normalise in the group key, never as a cosmetic afterthought in the SELECT.

Why the count filter must be HAVING, not WHERE

WHERE filters individual rows before any grouping happens, so the aggregate COUNT(*) does not exist yet — WHERE COUNT(*) > 1 is a hard error. HAVING runs after grouping and aggregation, when each group's count is known, so it is the only place a per-group threshold can live. "More than once" is a property of a group, hence HAVING.

Dialect note — collation can do this too

Lowercasing in the query is the portable answer, but some engines can enforce case-insensitivity at the column level: PostgreSQL has the citext type and case-insensitive ICU collations; SQL Server columns carry a collation such as SQL_Latin1_General_CP1_CI_AS where CI already means case-insensitive, so a plain GROUP BY email would fold cases automatically. Relying on column collation is convenient but invisible in the query text — an explicit LOWER makes the intent obvious and behaves identically everywhere.

Edge cases: NULL emails, whitespace, and the real fix

LOWER(NULL) is NULL; all NULL emails group into one bucket, and if two or more rows have a NULL email this query would flag NULL as a "duplicate" — usually not what an email audit wants, so consider WHERE email IS NOT NULL. Real address data also carries stray spaces; LOWER(TRIM(email)) is the sturdier key. And once the duplicates are found, the fix is structural: dedupe the rows, then add CREATE UNIQUE INDEX ... ON users (LOWER(email)) — a functional index — so the database rejects future case-variant collisions outright.

Senior signal: normalise with LOWER (ideally LOWER(TRIM(...))) inside the GROUP BY key so case variants collapse, put the "more than once" test in HAVING because it filters groups not rows, exclude NULL emails deliberately, and close the loop by recommending a unique functional index on LOWER(email) to prevent recurrence.
