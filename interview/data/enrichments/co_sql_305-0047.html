Plus500's analytics team needs the overlap between two cohorts — say, ids that appear in both a "funded accounts" feed and a "placed a trade" feed. The reference answer reaches straight for INTERSECT. The interviewer asking this wants to see that you know what INTERSECT does to duplicates and NULLs, and that you can name the join-based equivalent and when it would behave differently.

1. The source data — table_a and table_b

table_a.id
1, 2, 3, 4, 5
table_b.id
3, 4, 5, 6, 7

The two sets overlap on 3, 4, 5. Ids 1 and 2 are only in A; ids 6 and 7 are only in B. The expected answer is the three-element intersection — the middle of the Venn diagram.

2. The SQL — INTERSECT

SELECT id FROM table_a INTERSECT SELECT id FROM table_b;

INTERSECT takes two result sets and returns only the rows present in both. It is a set operator, so it compares whole rows and — importantly — removes duplicates from the output.

3. The resulting analysis

id
3
5
4

Exactly the three shared ids. Note the rows come back 3, 5, 4 — not sorted. INTERSECT guarantees the set of results but makes no promise about order; if the consumer needs 3, 4, 5 you must add an explicit ORDER BY id at the end.

4. Strategic explanations

INTERSECT de-duplicates — the behaviour to call out

INTERSECT is set-semantic: even if table_a held id 3 five times and table_b held it twice, the result has one row for 3. That is usually what "find common ids" wants — but say it explicitly. If you need multiplicity preserved (the smaller of the two counts), the variant is INTERSECT ALL, which most engines including Postgres support.

The INNER JOIN equivalent — and where it diverges

The same intersection via a join:

SELECT DISTINCT a.id
FROM table_a a
JOIN table_b b ON a.id = b.id;

The DISTINCT is load-bearing. Without it, if id 3 appears twice in A and three times in B, the join produces 2×3 = 6 rows for id 3 — the dreaded fan-out. INTERSECT de-duplicates for free; the join does not. This is the key difference, and a common interview gotcha.

NULL handling — where set operators differ from joins

If both tables contained a NULL id, INTERSECT treats the two NULLs as equal and the result includes one NULL row. A join, by contrast, uses a.id = b.id, and NULL = NULL is unknown — so the join would not match the NULLs. Set operators compare with NULL-equal semantics; equality joins do not. If NULL ids are meaningful, this distinction matters.

Column count and type must line up

INTERSECT requires both SELECTs to expose the same number of columns with compatible types, matched by position — not by name. SELECT id FROM table_a INTERSECT SELECT user_id FROM table_b works fine despite the different column names; what matters is that each side projects one column of a comparable type.

EXISTS — the scalable alternative

On large tables, a semi-join via EXISTS can let the planner stop at the first match per row:

SELECT id FROM table_a a
WHERE EXISTS (SELECT 1 FROM table_b b WHERE b.id = a.id);

One subtlety: this returns one row per matching row of A, so duplicates in A survive — add DISTINCT if you want true set semantics like INTERSECT.

Senior signal: state that INTERSECT de-duplicates and returns rows in no guaranteed order, contrast it with an INNER JOIN (which fans out without DISTINCT), note that set operators treat NULLs as equal while equality joins do not, and add an explicit ORDER BY when the consumer needs sorted output.
