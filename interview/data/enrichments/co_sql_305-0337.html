When a user searches once and the results don't satisfy them, they refine the query and search again. Reformulation rate is the inverse of search-task-success: high reformulation = users aren't getting what they need. The query uses LAG within session to detect successive searches.

1. The source data — searches

search_idsession_idquery_textsearch_at
S_001SS_01"weather"10:00
S_002SS_01"weather sf"10:01

2. The SQL — LAG to compare consecutive searches in session

WITH ordered AS (
  SELECT session_id, search_id, query_text, search_at,
    LAG(query_text) OVER (PARTITION BY session_id ORDER BY search_at) AS prev_query
  FROM searches
)
SELECT session_id,
  COUNT(*) AS searches_in_session,
  SUM(CASE WHEN prev_query IS NOT NULL AND prev_query <> query_text THEN 1 ELSE 0 END) AS reformulations
FROM ordered
GROUP BY session_id
HAVING COUNT(*) >= 2
ORDER BY reformulations DESC;

3. Strategic explanations

The LAG window — same-session compare

LAG(query_text) OVER (PARTITION BY session_id ORDER BY search_at) reaches back to the previous search within the same session. The first search has prev_query IS NULL — that's not a reformulation, it's a fresh start. The predicate filters those out.

"Reformulation" — define carefully

Three flavors with different signals:

typepredicatesignal
Refinementnew query starts with old + extra wordsnarrowing — algorithm under-targeted
Replacementcompletely different termsreset — first attempt failed entirely
Spelling fixedit distance <= 2typo — algorithm or autocomplete bug

Production splits all three. The query above only catches the union "any change".

The session boundary

What's a session? Production uses 30-minute idle threshold (Google's published rule). Reformulations across sessions are different intents, not failed-task signals. The query relies on session_id being correctly bucketed upstream.

The success-via-click correlation

A reformulation that results in a click and 30+ second dwell on the destination = "we eventually got it". A session that ends in reformulations with no click = "abandoned task". Join to clicks and split:

LEFT JOIN clicks c ON c.session_id = s.session_id AND c.dwell_seconds >= 30
The reformulation-graph

Common reformulation pairs ("weather" → "weather sf" → "weather san francisco") reveal user-intent gaps. The training data for query-rewriting and "did you mean" comes from this. Build a graph of (prev_query → next_query) with frequencies; the top edges are auto-suggest candidates.

The Position-1 confound

Reformulation rate looks artificially low for queries where Google's first result is excellent (Wikipedia, official site). Removing those reveals the true struggle queries. Senior analysis stratifies by query difficulty.

Senior signal: name the LAG-within-session pattern, taxonomize reformulation into refinement/replacement/spelling-fix, pin the 30-min session boundary, and connect to the click+dwell success signal. The reformulation-graph for auto-suggest training is the over-the-bar reframe.
