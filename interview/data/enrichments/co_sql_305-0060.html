Walmart's merchandising team wants order volume bucketed by month for a trend chart. It is a one-line GROUP BY on a formatted date — but the interviewer is watching two things: whether you bucket on the month rather than the raw date, and whether you understand that the choice of YYYY-MM versus another format silently controls how the months sort.

1. The source data — orders

Six orders with distinct dates spread across three months — two in January, three in February, one in March. The interesting bit: every date differs, so the only way to get a clean three-row monthly chart is to collapse the day component.

2. The SQL — TO_CHAR month bucket

SELECT TO_CHAR(order_date, 'YYYY-MM') AS year_month, COUNT(*) AS orders
FROM orders GROUP BY TO_CHAR(order_date, 'YYYY-MM') ORDER BY year_month;

TO_CHAR(order_date, 'YYYY-MM') renders each date as a '2024-02' string, discarding the day; GROUP BY on that string collapses all of a month's orders into one row; COUNT(*) tallies them.

3. The resulting analysis

Three rows, one per month, correctly tallied. The output sorts chronologically — but only by luck of format: '2024-01' < '2024-02' as strings happens to match calendar order. Pick a format like 'MM-YYYY' and that coincidence breaks.

4. Strategic explanations

String sort vs date sort — the format is load-bearing

year_month is text, so ORDER BY year_month is a lexical sort. With 'YYYY-MM' the big-endian layout means lexical order equals chronological order — even across year boundaries ('2023-12' < '2024-01'). Switch the mask to 'MM-YYYY' and you get '01-2024', '12-2023', '01-2025' sorting nonsensically. YYYY-MM is not cosmetic; it is what makes the sort correct.

The robust alternative — DATE_TRUNC

To remove all doubt, bucket on a real date and format only for display:

SELECT DATE_TRUNC('month', order_date)::date AS month, COUNT(*) AS orders
FROM orders
GROUP BY DATE_TRUNC('month', order_date)::date
ORDER BY month;

DATE_TRUNC keeps the bucket a genuine date (the 1st of the month). It sorts chronologically by definition, supports date arithmetic, and joins cleanly to a calendar spine. Group by the date; apply TO_CHAR at the presentation edge if a string label is needed.

Missing months leave gaps

If no orders fall in a month, that month produces no row — GROUP BY only emits groups that exist. A trend chart with April absent draws a misleading line. Fix it with a generated month spine and a LEFT JOIN:

SELECT m.month, COUNT(o.order_id) AS orders
FROM generate_series('2024-01-01'::date,'2024-03-01'::date,'1 month') AS m(month)
LEFT JOIN orders o ON DATE_TRUNC('month', o.order_date)::date = m.month
GROUP BY m.month ORDER BY m.month;

Use COUNT(o.order_id), not COUNT(*) — an empty month's left join yields one all-NULL row, and COUNT(*) would miscount it as 1.

Counting orders vs counting customers

COUNT(*) counts order rows. If February's three orders came from one repeat customer, "orders" is 3 but "active customers" is 1. The question says orders, so COUNT(*) is right — but if a follow-up asks for monthly active customers, switch to COUNT(DISTINCT customer_id). Know which the chart is labelled.

Status filtering and time zones

The table carries a status; cancelled orders inflate the volume. If the chart means realised orders, add WHERE status = 'completed'. And if order_date were a timestamptz, the month a late-night order falls into depends on the session time zone — truncate in an explicit zone for a reproducible bucket.

Senior signal: explain that TO_CHAR returns a string whose lexical sort only matches the calendar because YYYY-MM is big-endian, prefer DATE_TRUNC to keep the bucket a real date, fill gaps with a generated month spine plus COUNT(column), and distinguish counting orders from counting distinct customers.
