Accenture's sales operations team runs a quarterly leaderboard: total bookings per rep, ranked best-to-worst, for the commission committee. The catch in any real sales org is that two reps land on exactly the same number, and how the leaderboard handles that tie — do they share a place, does the next rep get skipped — is the entire point of asking for DENSE_RANK by name.

1. The source data — sales

Each rep has several deals. Once summed, rep 1 totals 800 and rep 2 also totals 800 — a tie at the top. The interview question is which rank rep 3 (total 200) then receives: 2 or 3?

2. The SQL — DENSE_RANK over a grouped sum

SELECT rep_id, SUM(amount) AS total,
  DENSE_RANK() OVER (ORDER BY SUM(amount) DESC) AS rk
FROM sales GROUP BY rep_id;

The clever part: the window function's ORDER BY references SUM(amount). Aggregates are computed first, then the window ranks the grouped rows — you can rank by an aggregate without a subquery.

Reps 1 and 2 both get rank 1 — the tie shares a place. Then rep 3 gets rank 2, not 3: DENSE_RANK does not leave a gap after a tie. (Within the rank-1 tie, rep 2 prints before rep 1, but that order is arbitrary — both are rank 1.)

4. Strategic explanations

DENSE_RANK vs RANK — the gap after a tie

Both give tied rows the same rank. They differ on what comes next. After two reps tie at rank 1, RANK() resumes at 3 (it counts rows above you), while DENSE_RANK() resumes at 2 (it counts distinct values above you):

rep   total   RANK   DENSE_RANK
2     800     1      1
1     800     1      1
3     200     3      2      <- the difference
4     150     4      3

For a leaderboard people read as "1st, 2nd, 3rd place", DENSE_RANK matches intuition — there is no missing "2nd". For competition standings where skipping is expected (two golds, no silver), RANK is right. The question names DENSE_RANK, so the no-gap behaviour is the desired one.

Why ROW_NUMBER would be wrong here

ROW_NUMBER() would hand reps 1 and 2 different numbers — 1 and 2 — even though they sold identical amounts, breaking the tie on physical row order. Two reps with equal bookings getting different leaderboard places (and possibly different commission tiers) is indefensible. ROW_NUMBER is for "pick exactly one row per group"; genuine ranking with shared places needs RANK or DENSE_RANK.

Ranking by an aggregate without a subquery

People often assume you must first compute totals in a subquery, then rank the outer query. Not so — a window function's ORDER BY can take an aggregate directly, because window functions execute after GROUP BY. The subquery form is equivalent and sometimes clearer, but it is not required:

SELECT rep_id, total, DENSE_RANK() OVER (ORDER BY total DESC) AS rk
FROM (SELECT rep_id, SUM(amount) AS total FROM sales GROUP BY rep_id) t;
Top-3 reps — and the dialect note

To return only the podium, filter on the rank — which needs a wrapper, since WHERE cannot reference a window result:

SELECT * FROM (
  SELECT rep_id, SUM(amount) AS total,
         DENSE_RANK() OVER (ORDER BY SUM(amount) DESC) AS rk
  FROM sales GROUP BY rep_id
) ranked
WHERE rk <= 3;

With DENSE_RANK, "top 3" can return more than 3 reps if several tie at rank 3 — usually what the committee wants. Snowflake/BigQuery would let you write QUALIFY rk <= 3; Postgres has no QUALIFY, so the subquery is the idiom.

NULL amounts and a rep with no sales

SUM ignores NULL amounts, and a rep present only with NULL deals sums to NULL; ORDER BY … DESC places NULL totals first by default in Postgres — an unsold rep wrongly topping the board. And a rep with no rows at all in sales simply does not appear; if the leaderboard must list every rep, start from a reps table and LEFT JOIN, with COALESCE(SUM(amount), 0).

Senior signal: explain that DENSE_RANK shares a rank for ties and resumes with no gap (rep 3 is 2nd, not 3rd) whereas RANK skips, reject ROW_NUMBER because equal sellers must not get different places, and know window functions can rank an aggregate directly without a subquery.
