New Relic's finance team wants the annual sales summary in the shape executives expect: one row per year, with four columns — Q1, Q2, Q3, Q4. The raw table stores quarters as rows; the report needs them as columns. That reshaping is a pivot, and the interviewer asking this wants to see that you build it with conditional aggregation, and that you handle a year that is missing a quarter.

1. The source data — sales

yearquarteramount
2024Q1100
2024Q2150
2024Q3200
2024Q4250
2025Q1300
2025Q2350
2025Q4450

2024 is a complete year — all four quarters. 2025 is missing Q3 entirely (no row at all). That gap is the whole interview: it decides whether 2025's Q3 cell reads 0 or NULL.

2. The SQL — conditional SUM(CASE ...)

SELECT year,
  SUM(CASE WHEN quarter='Q1' THEN amount END) AS q1,
  SUM(CASE WHEN quarter='Q2' THEN amount END) AS q2,
  SUM(CASE WHEN quarter='Q3' THEN amount END) AS q3,
  SUM(CASE WHEN quarter='Q4' THEN amount END) AS q4
FROM sales GROUP BY year;

GROUP BY year makes one row per year. Within each year, every CASE passes amount through only for its quarter and yields NULL otherwise; SUM then collapses those into one value per quarter column.

3. The resulting analysis

yearq1q2q3q4
2025300350NULL450
2024100150200250

2024 fills all four columns. For 2025, the CASE WHEN quarter='Q3' never matches any row, every value handed to that SUM is NULL, and SUM over nothing-but-NULL is NULL — not 0. The two output rows also come back unordered; a real report needs ORDER BY year.

4. Strategic explanations

The CASE has no ELSE — on purpose

Each CASE omits ELSE, so a non-matching quarter produces NULL and SUM ignores it. If you wrote ELSE 0, the column would still be correct here — SUM of zeros plus the real value is unchanged. The difference shows in the missing quarter: with ELSE 0, 2025's Q3 would have no rows to sum at all and stay NULL anyway. The ELSE matters far more for AVG (where stray zeros distort the mean) than for SUM — a nuance worth knowing.

The missing-quarter NULL — 0 or NULL?

2025's Q3 being NULL is honest: there is no data. If the executive sheet must show a number, wrap each column — but decide deliberately:

SELECT year,
  COALESCE(SUM(CASE WHEN quarter='Q1' THEN amount END),0) AS q1,
  COALESCE(SUM(CASE WHEN quarter='Q3' THEN amount END),0) AS q3
  -- ... q2, q4 likewise
FROM sales GROUP BY year ORDER BY year;

A 0 reads as "zero sales that quarter", which may be wrong if the truth is "data not yet loaded". NULL preserves the distinction; COALESCE erases it. Pick consciously.

The Postgres-native way — FILTER

Conditional aggregation has a cleaner Postgres form, the aggregate FILTER clause:

SELECT year,
  SUM(amount) FILTER (WHERE quarter='Q1') AS q1,
  SUM(amount) FILTER (WHERE quarter='Q2') AS q2,
  SUM(amount) FILTER (WHERE quarter='Q3') AS q3,
  SUM(amount) FILTER (WHERE quarter='Q4') AS q4
FROM sales GROUP BY year ORDER BY year;

Same result, intent on the surface. CASE is the portable form for every engine; FILTER is the idiomatic Postgres one. (Some databases also have a dedicated PIVOT operator, but Postgres does not — conditional aggregation is the pivot here.)

Why the column list is fixed — the real limitation

This pivot hard-codes four quarters. That is fine for quarters (there will only ever be four), but the pattern does not generalise: pivoting an unknown set of values — say, one column per product — can't be done in pure SQL, because a query's column list must be known before execution. That needs dynamic SQL or application-side reshaping. Naming this boundary is the senior beat.

Edge case: duplicate rows per quarter

If sales can hold two rows for the same year and quarter (two regions, say), SUM correctly adds them — that is why the aggregate is SUM and not MAX or a bare CASE. Use MAX only when each year/quarter is guaranteed unique and you just want to surface the single value.

Senior signal: build the pivot with conditional SUM(CASE ...), explain that a year missing a quarter yields NULL (no data) not 0 and treat that gap deliberately, offer FILTER as the cleaner Postgres equivalent, and note that the hard-coded column list is why pivoting an open-ended value set needs dynamic SQL.
