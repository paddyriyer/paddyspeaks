A sales manager wants a "top deals per rep" board: for every agent, show their two biggest orders, with a stable tie-break so the report is deterministic run after run. The interviewer is probing whether you reach for ROW_NUMBER() in a partitioned window, and — more subtly — whether you understand that a multi-key ORDER BY inside the window is what makes the ranking reproducible when amounts tie.

1. The source data — orders

Three deliberate edge cases: A001 has only one order (must still appear), A002 ties on ord_amount (broken by date), and A004 ties on both amount and date (broken only by ord_num). A correct tie-break chain has to survive all three.

2. The SQL — ROW_NUMBER with a three-key window ORDER BY

WITH ranked AS (
  SELECT agent_code, ord_num, ord_date, ord_amount,
         ROW_NUMBER() OVER (PARTITION BY agent_code
                            ORDER BY ord_amount DESC, ord_date ASC, ord_num ASC) AS rk
  FROM orders
)
SELECT agent_code, ord_num, ord_date, ord_amount, rk
FROM ranked
WHERE rk <= 2
ORDER BY agent_code, rk;

PARTITION BY agent_code restarts the numbering for each agent; the three-key ORDER BY decides the sequence within the partition. The WHERE rk <= 2 filter must live in an outer query — you cannot reference a window alias in the same SELECT's WHERE, because window functions are evaluated after WHERE.

4. Strategic explanations

Why ROW_NUMBER and not RANK here

The question explicitly says "no ties surface a duplicate rank" — it wants exactly two distinct rows per agent. RANK() would give both A004 orders rank 1 (a tie group), so rk <= 2 could return three rows for an agent with a three-way tie. ROW_NUMBER() guarantees a strict 1, 2, 3, … sequence, so "top 2" means precisely two rows. The price is that you must decide the tie order yourself — which is what the three-key ORDER BY does.

The tie-break chain must be total

A tie-break is only deterministic if the final key is unique. ord_amount DESC, ord_date ASC alone still leaves A004's two orders tied — same amount, same date. The engine would then pick arbitrarily, and the "winner" could flip between runs. Ending the chain with ord_num ASC (a primary key, guaranteed unique) makes the ordering total. Senior tip: always anchor a tie-break with a unique column, or accept non-determinism explicitly.

Why the filter needs a CTE (or subquery)

SQL's logical evaluation order is FROM → WHERE → GROUP BY → window functions → SELECT → ORDER BY. The window function does not exist yet when WHERE runs, so WHERE rk <= 2 in the same query is a syntax error. Wrapping the windowed query in a CTE (or derived table) lets the outer query treat rk as an ordinary column. QUALIFY (Snowflake, BigQuery, DuckDB) is sugar for exactly this, but Postgres, MySQL, and SQLite require the explicit wrapper.

Single-member partitions are free

A001's partition has one row, which gets rk = 1 and passes the filter. There is no special case to write — "top N" via ROW_NUMBER naturally returns "all of them" when a group has fewer than N rows. Contrast a LIMIT-per-group hack or a hard-coded "two rows" join, which would need extra logic to not drop the single-order agent.

Performance and the NULLS footnote

The window sorts each partition once — O(n log n) overall — and a composite index on (agent_code, ord_amount DESC, ord_date, ord_num) can let the planner stream the rows pre-sorted, skipping the sort entirely. One trap: if ord_amount can be NULL, Postgres sorts NULLs first under DESC, so a NULL-amount order would wrongly rank #1. Add NULLS LAST or filter NULL amounts if that is possible in your data.

Senior signal: reach for ROW_NUMBER() OVER (PARTITION BY … ORDER BY …) for strict top-N-per-group, make the tie-break chain total by ending it on a unique key, know that the rk <= N filter must sit in an outer query because windows run after WHERE, and note that single-member groups need no special handling.
