A symbolic-math pipeline needs to decompose free-text arithmetic expressions into their constituent parts: "Extract all numbers, operators, and parentheses from each expression, and count how many numbers and operators it contains." The interviewer is probing advanced regex usage — correlated subqueries with regexp_matches and array_agg, non-capturing groups for decimal numbers, and the global ('g') flag. This is a Hard question precisely because the pattern requires multiple correlated subqueries and the regex syntax is non-trivial.

1. The source data — math_inputs

input_idexpression
13 + 4.5 * (2 - 1)
2(10 / 2) + 7
3100 * 0.05
442

2. The SQL — correlated subqueries with regexp_matches and array_agg

SELECT input_id, expression,
       (SELECT array_agg(m[1])
        FROM regexp_matches(expression, '(\d+(?:\.\d+)?)', 'g') m) AS numbers,
       (SELECT array_agg(m[1])
        FROM regexp_matches(expression, '([+\-*/])', 'g') m)      AS operators,
       (SELECT array_agg(m[1])
        FROM regexp_matches(expression, '([()])', 'g') m)          AS parentheses,
       (SELECT COUNT(*)
        FROM regexp_matches(expression, '\d+(?:\.\d+)?', 'g'))     AS number_count,
       (SELECT COUNT(*)
        FROM regexp_matches(expression, '[+\-*/]', 'g'))            AS operator_count
FROM math_inputs;

When regexp_matches finds no matches, it returns zero rows; array_agg of zero rows produces NULL (not an empty array). Row 3 has no parentheses → NULL. Row 4 has no operators and no parentheses → both NULL. COUNT(*) of zero rows returns 0 (not NULL), hence row 4 has operator_count = 0, not NULL.

4. Strategic explanations

The regex pattern for decimal numbers — \d+(?:\.\d+)?

\d+ matches one or more digits. (?:\.\d+)? is a non-capturing group ((?:...)) that optionally (?) matches a literal dot followed by more digits. Non-capturing vs capturing groups matter here: regexp_matches returns an array of captured groups. The outer capturing group (\d+(?:\.\d+)?) captures the whole number (integer or decimal) as group 1 (m[1]). Using a capturing group for the optional decimal part instead — (\d+)(\.\d+)? — would return two groups per match, breaking the m[1] reference.

Correlated subquery per expression — what regexp_matches returns

regexp_matches(text, pattern, 'g') is a Postgres set-returning function: it yields one row per match, where each row is an array of captured groups. Wrapping it in a correlated subquery with array_agg(m[1]) collapses the rows back to a single array value. This is the Postgres idiom for "collect all regex matches into an array." The 'g' flag means "global" — find all non-overlapping matches, not just the first.

array_agg vs COUNT(*) on zero rows

When no matches exist, array_agg returns NULL (because aggregating zero values with no GROUP BY produces NULL). COUNT(*) returns 0 (the count of zero rows is zero). This asymmetry is visible in the result: operator_count for row 4 is 0, while operators (the array) is NULL. If the consumer needs an empty array instead of NULL for the array columns, use COALESCE((...), ARRAY[]::text[]).

Character class for operators — [+\-*/]

Inside a character class [], the minus sign must be escaped as \- (or placed at the start/end of the class) to prevent it from being interpreted as a range indicator. [+-*/] would be parsed as "characters in the range + through *, then /" which is not what's intended. [+\-*/] is the correct literal match for the four arithmetic operators. The * inside a character class is literal; outside a class it is a quantifier.

Performance — five correlated subqueries per row

Each of the five correlated subqueries re-scans the expression column for every input row. For a small lookup table this is fine. At scale, a single pass using REGEXP_MATCHES in a lateral join (or a single large pattern with alternation) reduces the regex evaluation from 5×N to 1×N. Snowflake does not have regexp_matches as a set-returning function; use REGEXP_SUBSTR(expression, '\\d+(?:\\.\\d+)?', 1, n) iteratively or Snowflake's FLATTEN(REGEXP_SPLIT_TO_ARRAY(...)) pattern.

Dialect portability
  • Postgres: This query runs as written. regexp_matches with 'g' flag is native.
  • Snowflake: No direct equivalent of regexp_matches as a set-returning function. Use REGEXP_COUNT for counts and repeated REGEXP_SUBSTR with occurrence parameter for extraction.
  • BigQuery: REGEXP_EXTRACT_ALL(expression, r'\\d+(?:\\.\\d+)?') returns an array of all matches in a single call — cleaner than the correlated subquery pattern.
  • MySQL: REGEXP_EXTRACT_ALL is not available before MySQL 8; the correlated subquery pattern does not exist. Regex extraction typically requires UDFs.
Senior signal: explain the non-capturing group (?:...) vs capturing group distinction and why it matters for m[1], articulate that array_agg returns NULL on zero rows while COUNT(*) returns 0 (use COALESCE if an empty array is required), and immediately flag that this Postgres-specific pattern has no direct Snowflake equivalent — pivot to REGEXP_COUNT and REGEXP_SUBSTR for the Snowflake rewrite.
