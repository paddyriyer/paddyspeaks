Broadcom's sales operations team wants a single result set that delivers every grain of a sales breakdown at once: each region×product cell, every region subtotal, every product subtotal, and the grand total. The reference solution builds this by hand with four UNION ALL branches. The interview is whether you recognise it as a reimplementation of CUBE — and can name the trade-offs of doing it the long way.

1. The source data — sales

regionproductamount
EastWidget100
EastGadget200
WestWidget150
WestGadget50

A clean 2×2 grid — deliberately small so every total is verifiable by eye. The interesting part is the output: subtotal and grand-total rows carry NULL in the dimension they roll up, and that overloading of NULL is the whole interview.

2. The SQL — four UNION ALL grains

SELECT region, product, SUM(amount) AS total FROM sales GROUP BY region, product
UNION ALL
SELECT region, NULL, SUM(amount) FROM sales GROUP BY region
UNION ALL
SELECT NULL, product, SUM(amount) FROM sales GROUP BY product
UNION ALL
SELECT NULL, NULL, SUM(amount) FROM sales
ORDER BY region, product;

Branch 1 is the finest grain (region×product); branch 2 totals by region; branch 3 by product; branch 4 is the grand total. UNION ALL stacks them — no dedup, because the grains never collide.

Nine rows: 4 detail cells, 2 region subtotals (East 300, West 200), 2 product subtotals (Gadget 250, Widget 250), 1 grand total (500). ORDER BY sorts NULLs last in Postgres, so each region's subtotal trails its detail rows. The grand-total 500 reconciles every margin — that cross-check is how you prove the breakdown is complete.

4. Strategic explanations

This is GROUP BY CUBE — say so

Four branches scanning the same table is the long-hand of one clause:

SELECT region, product, SUM(amount) AS total
FROM sales
GROUP BY CUBE (region, product)
ORDER BY region, product;

CUBE (region, product) generates all 2² grouping combinations — (region, product), (region), (product), () — in a single pass over the table. The UNION ALL version scans sales four times; CUBE scans once. Identical output, dramatically better plan. Naming CUBE is the headline senior signal here.

ROLLUP vs CUBE — pick the right one

CUBE gives every combination. If the report is hierarchical — region, then product within region, then grand total, but no standalone product subtotals — that is ROLLUP (region, product), which yields only (region, product), (region), (). Choosing CUBE when the business wants a hierarchy produces extra rows that confuse the reader; choosing ROLLUP when they want the full cross-tab omits the product totals. Match the operator to the shape of the question.

The NULL-overloading trap — GROUPING()

Subtotal rows mark the rolled-up dimension with NULL. But what if region itself legitimately contains NULL values (sales with no region)? Then a row with region = NULL is ambiguous: is it the grand subtotal, or the "no region" detail cell? You cannot tell them apart by eye. The fix is the GROUPING() function, which returns 1 for a rolled-up column and 0 for a real value:

SELECT
  CASE WHEN GROUPING(region) = 1 THEN 'All regions' ELSE region END AS region,
  CASE WHEN GROUPING(product) = 1 THEN 'All products' ELSE product END AS product,
  SUM(amount) AS total
FROM sales GROUP BY CUBE (region, product);
Why UNION ALL, not UNION

The branches produce disjoint grains, so no duplicate rows exist to remove — UNION ALL is correct. Using plain UNION would force a needless dedup sort over the whole result, pure wasted work. And a subtle hazard: if two different grains ever did compute the same (region, product, total) triple, UNION would silently collapse them into one row and corrupt the totals. UNION ALL is both faster and safer here.

Column-type alignment across branches

A bare NULL literal is untyped. In branch 2, SELECT region, NULL, ... — Postgres infers the NULL's type from the union with the other branches' product column, so it works. But if the branches disagreed on types, or you fed this into a stricter engine, you would need an explicit CAST(NULL AS text) to align the column. The aggregate aliases also only need to be named in the first SELECT; later branches inherit the column names.

Senior signal: recognise the four-way UNION ALL as a hand-rolled CUBE and collapse it to a single-pass GROUP BY CUBE, choose ROLLUP vs CUBE by whether the report is hierarchical or a full cross-tab, and use GROUPING() to disambiguate subtotal-NULLs from genuine NULL data.
