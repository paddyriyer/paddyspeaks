Okta's compensation team wants to flag employees who out-earn their own department's average — a quick "above the line" list for the next review. The reference solution does it with a correlated subquery: for each employee, re-compute the department average. It works, but the interviewer is watching whether you see the per-row re-execution, and whether you spot the subtle "employee compared against an average that includes themselves" wrinkle.

1. The source data — employees

Dept 10 has three salaries around a $100K mean; dept 30 has two; dept 20 has a single employee. That one-person department is the edge case — an employee can never beat an average they alone define.

2. The SQL — correlated AVG subquery

SELECT emp_id, dept_id, salary
FROM employees e
WHERE salary > (SELECT AVG(salary) FROM employees WHERE dept_id = e.dept_id);

For every row in the outer employees e, the subquery runs against employees again, filtered to that row's dept_id. The e.dept_id reference is the correlation — it ties the inner query to the current outer row.

4. Strategic explanations

The single-employee department — correct, but worth naming

Emp 4 is dropped, and that is mathematically right: in a one-row group the salary equals the average, and the strict > excludes it. But an interviewer wants you to say it: "a sole department member can never exceed their own average." If the business actually wants such employees flagged (or excluded with an explanation), that is a requirement to clarify, not an accident to leave silent.

The "compared against themselves" subtlety

The department average here includes the employee being tested. Emp 6's $160K is measured against $140K — an average their own salary pulled upward. Sometimes the fairer comparison is against peers excluding the individual:

SELECT emp_id, dept_id, salary
FROM employees e
WHERE salary > (
  SELECT AVG(salary) FROM employees
  WHERE dept_id = e.dept_id AND emp_id <> e.emp_id
);

Self-inclusive vs leave-one-out averages give different lists. Neither is "wrong" — but a senior names the choice.

The window-function rewrite — one pass instead of N

The correlated subquery conceptually re-scans the table once per employee. A window function computes every department average in a single pass:

SELECT emp_id, dept_id, salary
FROM (
  SELECT emp_id, dept_id, salary,
         AVG(salary) OVER (PARTITION BY dept_id) AS dept_avg
  FROM employees
) t
WHERE salary > dept_avg;

The window must be wrapped in a subquery because you cannot reference a window result in WHERE — window functions are evaluated after WHERE. On large tables this is usually the faster, cleaner answer; a good optimizer may already de-correlate the original, but do not rely on it.

NULL salaries and the empty-department case

AVG ignores NULL salaries, so an employee with a NULL salary is never > anything (any comparison with NULL is unknown) and is silently dropped — fine, but state it. If a department somehow had only NULL salaries, the subquery returns NULL and again no rows qualify. The correlated form also naturally skips departments with no matching peers; the window form behaves identically.

Senior signal: recognise the correlated subquery re-evaluates per outer row and offer the single-pass AVG() OVER (PARTITION BY dept_id) rewrite (wrapped because windows run after WHERE), and surface the two genuine ambiguities — the lone-employee department and whether the average should include the employee being tested.
