A FP&A analyst is building the monthly board pack and needs the canonical growth table: for each business unit, April 2026 revenue alongside its month-over-month and year-over-year growth. The raw ledger has one row per unit-month, so the three reference periods — this month, last month, same month last year — are scattered across rows. The ask is a pivot: rotate three time periods into three columns so the deltas can be computed side by side. The interviewer is probing conditional aggregation and whether you defend every division against a zero denominator.

SaaS B2B has all three reference months — a full MoM and YoY story. Hardware has March and April 2026 but no April 2025 row at all. That gap is the interview: YoY for Hardware divides by a period that does not exist, and the query must yield a clean NULL rather than crash.

2. The SQL — pivot periods, then compute growth

WITH execution_pivot AS (
  SELECT
    business_unit,
    SUM(CASE WHEN year_num = 2026 AND month_num = 4 THEN revenue ELSE 0 END) AS current_month,
    SUM(CASE WHEN year_num = 2026 AND month_num = 3 THEN revenue ELSE 0 END) AS prior_month,
    SUM(CASE WHEN year_num = 2025 AND month_num = 4 THEN revenue ELSE 0 END) AS prior_year
  FROM monthly_revenue
  GROUP BY business_unit
)
SELECT
  business_unit,
  current_month AS rev_apr_2026,
  ROUND(((current_month - prior_month) * 100.0) / NULLIF(prior_month, 0), 2) AS mom_growth_pct,
  ROUND(((current_month - prior_year) * 100.0) / NULLIF(prior_year, 0), 2) AS yoy_growth_pct
FROM execution_pivot
ORDER BY business_unit;

The CTE pivots: three SUM(CASE…) expressions fold each unit's rows into one row with three period columns. The outer query then does the arithmetic — deltas over baselines — on those now-adjacent columns.

4. Strategic explanations

Why pivot first — growth needs periods side by side

Growth is a ratio between two periods, and SQL arithmetic operates within a row, not across rows. While both periods sit in separate rows you cannot subtract them without a self-join or window function. Pivoting the three periods into three columns of one row makes current_month - prior_month a plain in-row expression. The pivot is not decoration here — it is what makes the arithmetic expressible at all.

NULLIF — the load-bearing defence

Division by zero is a hard error in SQL: it aborts the whole statement, not just one row. NULLIF(prior_month, 0) returns NULL when the argument is 0 and the value otherwise; dividing by NULL yields NULL — a quiet, correct "growth is undefined" — instead of a crash. Every denominator in a growth query must be wrapped. The semantics are also right: if a unit had zero revenue last period, its percentage growth genuinely is not a number.

The ELSE 0 / NULLIF interplay — subtle but deliberate

The pivot uses ELSE 0, so a missing period becomes 0, not NULL. That makes current_month - prior_year compute as 60000 - 0 in the numerator — harmless — while NULLIF(prior_year, 0) catches the same 0 in the denominator and produces NULL. The two work as a pair: ELSE 0 keeps the numerator arithmetic clean, NULLIF rescues the denominator. Drop ELSE 0 and you would instead lean on SUM ignoring NULLs — same number, but you must still NULLIF the divisor regardless.

The * 100.0 trick — forcing real division

revenue is INT. In Postgres integer / integer truncates, so (150000-100000)/100000 would be 0, not 0.5. Multiplying by 100.0 — a numeric literal — promotes the numerator to numeric before the divide, so the result keeps its fractional part and ROUND(…, 2) has something to round. Writing 100 instead of 100.0 is a classic silent bug: every growth figure collapses to a rounded-down integer percentage or zero.

Dialect — portability of the pieces

NULLIF and CASE are SQL-standard and run unchanged in every engine. Snowflake, Oracle, and SQL Server have a PIVOT operator that could rotate the periods, but it pivots on a single column's values — here the period is a composite of year_num and month_num, so you would first concatenate them into one key, then pivot. The SUM(CASE…) form sidesteps that and handles the two-column predicate directly — which is exactly why it stays the portable default.

Senior signal: pivot the reference periods into columns so growth becomes in-row arithmetic, wrap every denominator in NULLIF(…, 0) so a missing baseline yields NULL instead of a divide-by-zero abort, and force floating-point division with a 100.0 literal — using 100 truncates every percentage to an integer.
