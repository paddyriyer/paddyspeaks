Apple frames "Rotting Oranges" as the canonical multi-source BFS question. A grid of cells: 0 empty, 1 fresh orange, 2 rotten. Each minute, every rotten orange rots its 4-directional fresh neighbours. Return the minutes until none are fresh, or -1 if some can never rot. It tests whether you see this as simultaneous spreading from many sources at once — not one DFS per orange.

1. The input data

grid = [
  [2, 1, 1],
  [1, 1, 0],
  [0, 1, 1],
]
# one rotten orange at (0,0); six fresh oranges; one empty cell

2. The approach

A breadth-first search seeded from all rotten cells simultaneously. The first scan does two jobs: push every rotten orange onto the deque as (r, c, 0) — the 0 is its rot-time — and count the fresh oranges so we know when we're done.

The BFS then pops cells with popleft() (FIFO — the property that makes it breadth-first and the times monotonically non-decreasing). For each fresh neighbour it rots the cell (grid[nr][nc] = 2), decrements fresh, and enqueues it with time t+1. Mutating the grid to 2 on enqueue is the visited-marker — a cell can't be rotted twice. minutes tracks the largest time seen, which is the last minute any orange rotted.

The final check is the elegant part: return minutes if fresh == 0 else -1. If a fresh orange was walled off (no path of adjacency from any rotten source), the BFS can never reach it, fresh stays positive, and the answer is -1.

3. The output

All six fresh oranges rot; the last one falls at minute 4, so the answer is 4.

4. Strategic explanations

Complexity

Time O(R·C) — the seeding scan is O(R·C), and the BFS enqueues each cell at most once. Space O(R·C) for the queue, which in the worst case (the whole grid rotten at minute 0) holds every cell.

The idiomatic alternative — level-by-level BFS

Instead of carrying a per-cell time in the tuple, drain one whole "minute" per outer loop and increment a counter:

minutes = 0
while q and fresh:
    minutes += 1
    for _ in range(len(q)):          # exactly one level
        r, c = q.popleft()
        for dr, dc in ((1,0),(-1,0),(0,1),(0,-1)):
            nr, nc = r+dr, c+dc
            if 0<=nr<R and 0<=nc<C and grid[nr][nc]==1:
                grid[nr][nc] = 2; fresh -= 1; q.append((nr,nc))
return minutes if fresh == 0 else -1

The for _ in range(len(q)) snapshot is the standard "process one BFS level" idiom — cleaner state, same O(R·C).

The edge case

Zero fresh oranges at the start &mdash> fresh == 0 from the outset; the answer must be 0 (nothing to rot). The shown solution handles this: if the grid has a rotten orange, minutes ends at 0 after the BFS; if the grid is entirely empty or all-empty cells, the queue is empty and minutes stays 0. The level-by-level variant needs the and fresh guard or it would wrongly return 1. A walled-off fresh orange → -1.

The follow-up

"Which orange rots last, and where?" — track the coordinate alongside minutes. "Oranges also rot diagonally" → eight direction deltas. "Stream a grid too big for memory" breaks the in-RAM queue and pushes toward a tiled approach.

The trap

Running a separate DFS/BFS from each rotten orange independently and taking the min distance per fresh cell — that's O((R·C)²) and easy to get wrong. The insight is that all sources spread at once, so they must share one queue, seeded together. The other trap: forgetting that "0 fresh at start" must yield 0, not whatever the loop counter happens to be.

Senior signal: recognises this as multi-source BFS and seeds all rotten oranges into one queue rather than searching from each separately, marks cells visited by mutating them to 2 on enqueue, returns -1 via a leftover-fresh check, and explicitly handles the "no fresh oranges → 0 minutes" edge that trips up the level-counting variant.
