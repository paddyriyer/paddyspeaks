Netflix's Content Strategy team wants the audience for a Stranger Things spin-off. The brief: "every subscriber who watched Stranger Things at least once but has never started The Witcher." Sounds simple. The bare-metal NOT IN solution looks correct, ships to production, and silently returns zero users — because one row in viewing_activity has NULL in content_name from a 2-year-old data-quality incident. NOT IN with a NULL in the subquery collapses everything to UNKNOWN, and WHERE UNKNOWN filters every row out. This is one of the top three bugs in real-world SQL.

The right answer is users 1 and 4 — they watched Stranger Things, never touched The Witcher. User 6's NULL row is invisible to a content-match filter… unless your query construction lets that NULL leak into a NOT IN subquery.

2. The three solution shapes — pick consciously

Shape 1 — NOT IN · the trap
-- Looks correct. Runs in dev with no NULLs in the test data.
-- Ships to prod. Returns ZERO rows.
SELECT DISTINCT user_id
FROM viewing_activity
WHERE content_name = 'Stranger Things'
  AND user_id NOT IN (
    SELECT user_id FROM viewing_activity
    -- Forgot to filter content_name here, so user_id 6 with NULL
    -- content gets included in the subquery
  );

The fatal step: x NOT IN (a, b, NULL) evaluates to UNKNOWN (because x != NULL is unknown, not true), and WHERE UNKNOWN excludes the row. The query silently returns zero rows — no error, no warning. Marketing wonders why the audience is empty for a week.

Shape 2 — NOT EXISTS · null-safe, usually fastest
SELECT DISTINCT v1.user_id
FROM viewing_activity v1
WHERE v1.content_name = 'Stranger Things'
  AND NOT EXISTS (
    SELECT 1 FROM viewing_activity v2
    WHERE v2.user_id = v1.user_id
      AND v2.content_name = 'The Witcher'
  );

Correlated anti-join. The inner subquery is a boolean existence test, not a value comparison, so NULLs in the inner table don't propagate. The planner can stop scanning the inner side the moment it finds one match (semi-/anti-join short-circuit). This is the safe production default.

Shape 3 — LEFT JOIN ... IS NULL · readable, sometimes slower
SELECT DISTINCT v1.user_id
FROM (SELECT DISTINCT user_id FROM viewing_activity
      WHERE content_name = 'Stranger Things') v1
LEFT JOIN viewing_activity v2
  ON v2.user_id = v1.user_id AND v2.content_name = 'The Witcher'
WHERE v2.user_id IS NULL;

The "anti-join via outer join" idiom — readable to analysts who haven't internalised NOT EXISTS. The planner usually rewrites this to the same anti-join as Shape 2, but on older engines it can materialise matched rows then discard them (slower at scale).

3. The interviewer's edge-case probes

Probe 1 — "Why does NOT IN behave like that? Walk me through three-valued logic."

SQL uses three-valued logic: TRUE, FALSE, UNKNOWN. NULL compared to anything returns UNKNOWN. WHERE keeps rows where the predicate is TRUE — UNKNOWN is treated like FALSE. So 5 NOT IN (1, 2, NULL) = (5 != 1) AND (5 != 2) AND (5 != NULL) = TRUE AND TRUE AND UNKNOWN = UNKNOWN → row excluded. This is the spec, not a bug — but it's the spec everyone forgets.

Probe 2 — "How would you defend against this in code review?"
  • Lint rule: ban NOT IN in production code; require NOT EXISTS or LEFT JOIN IS NULL.
  • If you must use NOT IN: explicitly filter the subquery: NOT IN (SELECT user_id FROM ... WHERE user_id IS NOT NULL). Reviewer-readable, NULL-safe.
  • Make NULLs impossible at the source: content_name TEXT NOT NULL + a backfill of the 2-year-old bad rows. This is the only fix that prevents the next incident.
Probe 3 — "What if 'never watched The Witcher' should also exclude users who watched 30 seconds and bailed?"
NOT EXISTS (
  SELECT 1 FROM viewing_activity v2
  WHERE v2.user_id = v1.user_id
    AND v2.content_name = 'The Witcher'
    AND v2.watch_seconds >= 600    -- 10-minute meaningful watch threshold
)

The threshold turns the question from "any view" to "meaningful engagement". This is the kind of definition the interviewer expects you to ask about: a real content team would never call a 30-second sample a "view".

Probe 4 — "Now generalise: find users who watched any show in genre A but never any show in genre B."

Same shape, joined to a content-metadata table:

SELECT DISTINCT v1.user_id
FROM viewing_activity v1
JOIN content_metadata c1 ON c1.content_name = v1.content_name
WHERE c1.genre = 'sci-fi'
  AND NOT EXISTS (
    SELECT 1 FROM viewing_activity v2
    JOIN content_metadata c2 ON c2.content_name = v2.content_name
    WHERE v2.user_id = v1.user_id AND c2.genre = 'romance'
  );
Probe 5 — "At Netflix scale (250M subscribers, 15B events / day), would you still write this as a correlated subquery?"

No. At that scale you'd materialise two daily-aggregated cohort tables — users_who_watched_X(user_id, last_watch_date) and users_who_watched_Y(user_id, last_watch_date) — and do the anti-join at the cohort grain, not the event grain. The query becomes a single hash-anti-join on ~50M rows instead of a correlated lookup on 15B. The aggregation pipeline runs incrementally on the daily partition, not the full history.
