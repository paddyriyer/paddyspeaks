A subscription business wants a daily active-users curve for the first work-week of January. Each row in subscription is one user's active interval, and the question asks: on each calendar day, how many distinct users had a live subscription? The interviewer is testing the calendar-spine pattern — you must report every day in the range, including days with zero users, which a plain GROUP BY over the data can never do.

1. The source data — subscription

Two edge cases are deliberately planted. User 3 appears in two rows — on Jan 3 both intervals could overlap the window, so a naive count would double-count them; COUNT(DISTINCT) is mandatory. User 4's interval starts in December but is end-exclusive at Jan 2, so they count only on Jan 1.

2. The SQL — calendar spine left-joined to intervals

SELECT d, COUNT(DISTINCT s.user_id) AS active_users
FROM GENERATE_SERIES(DATE '2025-01-01', DATE '2025-01-05', INTERVAL '1 day') AS gs(d)
LEFT JOIN subscription s ON d >= s.start_date AND d < s.end_date
GROUP BY d
ORDER BY d;

GENERATE_SERIES manufactures one row per day from Jan 1 to Jan 5 — this is the spine. Each spine day LEFT JOINs to every subscription whose half-open interval covers it (start_date <= d < end_date). The LEFT JOIN guarantees a day with no matches still survives as a row.

3. The resulting analysis

On Jan 3 the answer is 3, not 4: user 3's first interval covers Jan 3, and although their second interval (starting Jan 5) does not, COUNT(DISTINCT user_id) would still have suppressed any duplication. The half-open boundary shows up on Jan 4 — user 1's interval end_date of Jan 4 means they are not active that day.

4. Strategic explanations

Why a plain GROUP BY cannot do this

If you grouped the subscription rows directly, you could only ever emit days that some interval mentions. A day on which the entire customer base churned would simply be missing from the output — silently, with no zero row. The spine inverts the logic: you start from the complete set of days you must report and attach data to them. Any "include every period even when empty" requirement is a spine signal.

Why the LEFT JOIN direction matters

The spine must be the left table. A plain JOIN would drop spine days with no matching subscription, defeating the entire purpose. With LEFT JOIN, an unmatched day still produces one row whose s.user_id is NULL — and crucially, COUNT(DISTINCT s.user_id) ignores NULL, so that day correctly reports 0 rather than 1. Using COUNT(*) here would be a bug: it counts the NULL row and reports 1 for an empty day.

The half-open interval and the join predicate

The spec says active when start_date <= D < end_date, so the join uses d >= s.start_date AND d < s.end_date. The strict < on the upper bound is what makes Jan 4 exclude user 1. If the data used inclusive end dates you would write d <= s.end_date. Half-open intervals are the safer convention — adjacent intervals tile the timeline with no gap or overlap — but you must match the predicate to whichever convention the data uses.

Dialect — generating the spine elsewhere

GENERATE_SERIES with a DATE range is Postgres-specific. SQL Server uses a recursive CTE (WITH dates AS (SELECT @start ... UNION ALL SELECT DATEADD(day,1,d) ...)) or a permanent calendar/dimension table. BigQuery has GENERATE_DATE_ARRAY + UNNEST; Snowflake has GENERATOR with DATEADD. Data warehouses almost always keep a persistent dim_date table — mentioning that as the production pattern shows warehouse maturity.

Performance — the join can explode

A range-predicate join (d BETWEEN start AND end) cannot use a simple equality hash join; the planner may resort to a nested loop. For a wide date range against millions of intervals, index subscription(start_date, end_date) so each spine day range-scans only relevant intervals. An alternative for huge data is the "delta" trick — emit +1 at each start and -1 at each end, then take a running sum — which is O(n log n) instead of days×intervals.

Senior signal: reach for a calendar spine (GENERATE_SERIES or a dim_date table) whenever every period must appear including empty ones, LEFT JOIN data onto the spine so zero-rows survive, and pair it with COUNT(DISTINCT user_id) — not COUNT(*) — so unmatched NULL rows report 0 and multi-interval users are never double-counted.
