A data pipeline writes one daily_metrics row per calendar day — or it should. An on-call engineer suspects the loader skipped some days and needs to find every gap: the day after a gap starts and the day before it ends, plus how many days went missing. The interviewer is testing the gaps-and-islands idiom — reasoning about the row that comes next, not the current one.

1. The source data — daily_metrics

metric_datevalue(narrative)
2025-03-01100start of run
2025-03-02110consecutive — no gap
2025-03-03105then nothing until 03-07
2025-03-07130gap of 3 days (04, 05, 06)
2025-03-08120consecutive — no gap
2025-03-15140last row — gap of 6 days before it

Two edge cases. 2025-03-01→02 is a one-day step — not a gap, and must not be reported. And 2025-03-15 is the last row: it has no following day, so it cannot itself start a gap — the query must not invent a phantom gap after it.

2. The SQL — LEAD() to peek at the next date

WITH ordered AS (
    SELECT metric_date,
           LEAD(metric_date) OVER (ORDER BY metric_date) AS next_date
    FROM daily_metrics
)
SELECT metric_date AS gap_starts_after,
       next_date  AS gap_ends_before,
       (next_date::date - metric_date::date - 1) AS missing_days
FROM ordered
WHERE next_date IS NOT NULL
  AND (next_date::date - metric_date::date) > 1
ORDER BY metric_date;

LEAD(metric_date) pulls the following row's date onto the current row. If the difference between a date and its successor exceeds one day, the days in between were never loaded — that pair brackets a gap.

3. The resulting analysis

gap_starts_aftergap_ends_beforemissing_dayssignal
2025-03-032025-03-07304, 05, 06 missing
2025-03-082025-03-15609–14 missing

Two gaps. The 03-01→02 step has a difference of exactly 1, fails the > 1 test, and is correctly silent. 2025-03-15 has a NULL next_date (no row after it), is filtered by next_date IS NOT NULL, and produces no false gap. missing_days subtracts the extra 1 so a 4-day jump reports 3 absent days.

4. Strategic explanations

Why LEAD beats a self-join

The pre-window way to find "the next row" is a correlated subquery — SELECT MIN(metric_date) FROM daily_metrics d2 WHERE d2.metric_date > d1.metric_date — which scans the table once per row. LEAD sorts the data once and reads the neighbour for free in a single pass: O(n log n) instead of O(n²), and far clearer about intent. For any ordered-neighbour question, LEAD/LAG is the senior tool.

The last row and the NULL guard

LEAD returns NULL for the final row of the partition — there is no next date. Without WHERE next_date IS NOT NULL, the arithmetic next_date - metric_date would be NULL, the > 1 comparison would be unknown, and the row would be dropped anyway — but relying on that is fragile. Stating the guard explicitly documents the intent and protects against an engine that handles NULL comparisons differently.

The off-by-one in missing_days

From 03-03 to 03-07 the raw date difference is 4, but only 3 days (04, 05, 06) are actually missing — the endpoints exist. Subtracting the extra 1 converts "span between two present dates" into "count of absent dates". Forgetting that -1 over-reports every gap by one day; it is the single most common mistake on this question.

This finds gap boundaries, not the missing dates themselves

The output is one row per gap with its endpoints — it does not list 2025-03-04, 05, 06 individually. If the business wants the actual missing dates, the senior follow-up is to generate_series(min_date, max_date, '1 day') and LEFT JOIN the real table, keeping the unmatched calendar days. Knowing both shapes — gap boundaries vs enumerated missing days — and which the consumer needs is the differentiator.

Dialect notes

The date subtraction date - date → integer is Postgres-specific. SQLite would use julianday(next_date) - julianday(metric_date); MySQL DATEDIFF(next_date, metric_date); Snowflake DATEDIFF('day', metric_date, next_date). LEAD and the CTE are standard across all of them — only the date arithmetic needs translation.

Senior signal: solve sequence-gap detection with LEAD to compare each row to its successor, guard the final row with next_date IS NOT NULL, subtract one to turn a date span into a missing-day count, and know the generate_series + anti-join follow-up when the consumer needs the actual missing dates rather than just the gap boundaries.
