Stripe's billing-integrity team is hunting for accidental double charges — the same customer billed the same amount more than once. The naive read is "find duplicates", but a subscription that legitimately bills $9.99 every month is also the same user and the same amount, repeated. The interview question is whether you can tell a real duplicate from a recurring charge, and whether you ask before you accuse the billing system of a bug.

User 4 paid $100 twice one day apart — that smells like a genuine double charge. User 1's three $9.99 payments are spread across months — almost certainly a monthly subscription. User 2's two $49 payments are a month apart, same story. The query below treats all of them identically, which is exactly the tension to surface.

2. The SQL — GROUP BY user, amount HAVING COUNT > 1

SELECT user_id, amount, COUNT(*) AS times_paid
FROM payments GROUP BY user_id, amount HAVING COUNT(*) > 1;

Group every payment by the pair (user_id, amount), count the rows in each group, and keep only groups with more than one. HAVING filters on the aggregate, after grouping — a WHERE COUNT(*) > 1 would be a syntax error.

Three groups survive. The query is doing exactly what it says — but notice it cannot tell user 4's same-day double charge apart from user 1's monthly subscription. All three are flagged with equal confidence. That is the gap between "rows with the same value" and "suspicious payments".

4. Strategic explanations

The follow-up: "which of these are actually fraud?"

The senior answer is to add a time window. A real accidental double charge happens within minutes or hours; a subscription repeats monthly. Bring paid_at into the analysis and only flag payments that are close together:

SELECT user_id, amount, paid_at
FROM (
  SELECT user_id, amount, paid_at,
    paid_at - LAG(paid_at) OVER (PARTITION BY user_id, amount ORDER BY paid_at) AS gap
  FROM payments
) t
WHERE gap < INTERVAL '1 hour';

Against this data only user 4 would survive a one-hour window (his payments are a day apart, so widen to a day or use a business-defined threshold). User 1 and user 2, spaced a month apart, drop out. That is the duplicate-detection query; the reference query is merely a same-value report.

The schema is wider than the query — use payment_id

The table also has payment_id, customer_id, method, and status. A defensible duplicate check would group on customer_id rather than user_id if customers can have multiple user accounts, and would almost certainly exclude refunds and failures: WHERE status = 'succeeded'. A failed retry that later succeeds is not a double charge. Flagging it as one sends a real customer a false apology email.

COUNT(*) vs COUNT(amount) when amounts can be NULL

If amount could be NULL, two subtleties bite. First, GROUP BY treats all NULLs as one group, so NULL-amount payments collapse together — usually not meaningful. Second, COUNT(*) counts those rows while COUNT(amount) would not. Here COUNT(*) is right (we want row counts), but say why. Filter NULL amounts out explicitly if they are not real charges.

Floating-point amounts — the silent miss

GROUP BY on money compares values exactly. If amount were a FLOAT/DOUBLE column, 9.99 stored from two code paths might be 9.99000000001 and 9.99, landing in different groups and a duplicate slipping through. This is why payment amounts belong in NUMERIC/DECIMAL, never binary floating point. Mention it — it shows you think about how money is stored, not just queried.

Getting back the offending rows, not just the count

GROUP BY tells you a duplicate exists but throws away the individual payment_ids you would need to refund. To recover them, a window count keeps every row:

SELECT *
FROM (
  SELECT *, COUNT(*) OVER (PARTITION BY user_id, amount) AS n
  FROM payments
) t
WHERE n > 1;

Now you have each duplicated payment with its payment_id — actionable, where the grouped query is only diagnostic.

Senior signal: name the difference between "same user, same amount" and a genuine accidental double charge, propose a time-window check on paid_at so monthly subscriptions are not falsely flagged, exclude non-succeeded statuses, and switch to a windowed COUNT(*) OVER when the business needs the actual payment_ids rather than a count.
