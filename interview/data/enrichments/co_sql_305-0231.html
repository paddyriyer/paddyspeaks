Snowflake's revenue-operations team wants a regional scorecard: for each region, completed-order revenue in one column and cancelled-order revenue in another. The instinct of a junior is to run two separate queries and join them. The interviewer is watching for the one-pass idiom — SUM wrapped around a CASE — that pivots a status column into named measures.

1. The source data — orders

regionstatusamount
Westcompleted100
Westcancelled40
Westcompleted60
Eastcompleted200
Eastpending75
Northcancelled30

Three statuses appear, but the report only names two. East has a pending order — it belongs to neither bucket. North has only a cancelled order — its completed column will have to be 0, not absent.

2. The SQL — SUM(CASE WHEN …)

SELECT region,
  SUM(CASE WHEN status='completed' THEN amount ELSE 0 END) AS completed_amt,
  SUM(CASE WHEN status='cancelled' THEN amount ELSE 0 END) AS cancelled_amt
FROM orders
GROUP BY region;

Each CASE rewrites amount to either the real value or 0, depending on the row's status. SUM then totals the rewritten column per region — one scan, two pivoted measures.

3. The resulting analysis

regioncompleted_amtcancelled_amt
East2000
West16040
North030

West's completed column is 100 + 60 = 160; cancelled is 40. East's pending $75 order contributed to neither column — it fell through both CASE expressions to the ELSE 0. North shows completed_amt = 0: it has no completed orders, but the region still appears with an honest zero rather than dropping out.

4. Strategic explanations

The ELSE 0 vs ELSE NULL decision

Write ELSE 0 and a region with no matching orders sums to 0. Omit the ELSE (so it defaults to ELSE NULL) and SUM ignores the NULLs — for a region with zero completed orders, SUM over all-NULL returns NULL, not 0. Both are defensible: 0 says "we earned nothing", NULL says "no data". For a financial scorecard, 0 is almost always what stakeholders expect — but the difference surfaces a missing-data edge case worth naming.

SUM(CASE) vs COUNT(CASE) — revenue vs order count

This pattern generalises. To count cancelled orders rather than sum their revenue, two equivalent forms:

COUNT(CASE WHEN status='cancelled' THEN 1 END)        AS cancelled_orders,
SUM(CASE WHEN status='cancelled' THEN 1 ELSE 0 END)   AS cancelled_orders

Note the COUNT form deliberately has no ELSE — COUNT ignores NULL, so a bare THEN 1 counts only the matching rows. Adding ELSE 0 to a COUNT would count every row, a classic bug.

The modern idiom — FILTER

Postgres offers a cleaner spelling of the same conditional aggregate:

SELECT region,
  SUM(amount) FILTER (WHERE status='completed') AS completed_amt,
  SUM(amount) FILTER (WHERE status='cancelled') AS cancelled_amt
FROM orders
GROUP BY region;

One nuance: FILTER has no ELSE branch, so an empty group yields NULL just like the ELSE NULL case — wrap in COALESCE(…, 0) if you need zeros. FILTER is the senior tell; not every engine supports it, so SUM(CASE) remains the portable fallback.

Status typos and total reconciliation

The CASE matches the literal string 'completed'. A stray 'Completed' or 'COMPLETED' would silently fall into ELSE 0 — revenue quietly lost. Defend with LOWER(status) or, better, a constrained enum. And a good sanity check: completed_amt + cancelled_amt should be less than SUM(amount) here precisely because pending orders exist — if they were equal, you would know no other statuses existed.

Senior signal: pivot with a single-scan SUM(CASE) instead of joining separate queries, choose ELSE 0 vs ELSE NULL deliberately for empty groups, reach for the FILTER idiom where supported, and guard against status-string typos that would silently route revenue into the ELSE bucket.
