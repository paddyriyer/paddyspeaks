This is a sophisticated Trend & Momentum analysis — the SAP-style answer to a common Sales Performance Management problem: seasonality and volatility. By combining a 3-month rolling average with a month-over-month (MoM) growth %, the dashboard distinguishes a "lucky month" from a genuine upward trend. Two windows on the same axis: the smooth trendline and the velocity ticker.

2. The SQL — CTE + double window function

WITH monthly AS (
  SELECT channel, DATE_TRUNC('month', sale_date) AS mo, SUM(amount) AS total
  FROM sales
  GROUP BY channel, DATE_TRUNC('month', sale_date)
)
SELECT channel, mo, total,
  AVG(total) OVER (
    PARTITION BY channel ORDER BY mo
    ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
  ) AS rolling_3mo,
  100.0 * (total - LAG(total) OVER (PARTITION BY channel ORDER BY mo))
        / NULLIF(LAG(total) OVER (PARTITION BY channel ORDER BY mo), 0) AS mom_pct
FROM monthly
ORDER BY channel, mo;

3. The step-by-step logic

Step 1 — the monthly CTE (pre-aggregation)

Thousands of daily transactions become one row per (channel, month). Window functions like LAG and ROWS BETWEEN need a regular time grid — one row per month, no gaps. DATE_TRUNC('month', sale_date) is the canonical bucketing primitive (SQLite alternative: strftime('%Y-%m-01', sale_date)).

Step 2 — rolling_3mo (the trendline)

AVG(total) OVER (... ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) looks at the current month plus the two before. This smooths the data: a one-time spike (e.g. a Black Friday surge or a single mega-deal) gets averaged across three months instead of being mistaken for a permanent step-change.

Step 3 — mom_pct (the momentum)

LAG(total) OVER (PARTITION BY channel ORDER BY mo) reaches back exactly one month within the same channel. The percentage formula gives instant velocity: this month vs last month. Combined signals:

  • total > rolling_3mo AND mom_pct > 0 → channel is accelerating (above trend AND positive momentum)
  • total < rolling_3mo AND mom_pct < 0 → channel is decelerating (below trend AND negative momentum)
  • total > rolling_3mo BUT mom_pct < 0 → cooling spike (still elevated but losing steam)

5. Strategic explanations — SPM applications

Identifying channel fatigue / saturation

If mom_pct declines smoothly — say +20% → +10% → +2% → -1% — while total is still high, the channel is hitting saturation: the addressable market is exhausted. The Sales Ops playbook here is to shift marketing spend or sales headcount to a less-mature channel before the absolute number turns negative. The momentum signal leads the revenue signal.

Dynamic quotas via the rolling baseline

Instead of a fixed annual target, set the next-month quota as "110% of the trailing 3-month rolling average". This auto-adjusts to market conditions:

SELECT channel, mo,
  rolling_3mo * 1.10 AS next_month_target
FROM trend_table
WHERE mo = (SELECT MAX(mo) FROM trend_table);

Reps stop being penalised for hitting a fixed target during a downturn or sandbagging during a boom.

The NULLIF safety net — not optional

NULLIF(LAG(total) OVER (...), 0) handles the edge case where the previous month had zero revenue (newly-launched channel, paused campaign). Without it: division by zero → entire row errors (Postgres) or returns infinity/NaN (other engines). With it: the mom_pct column shows NULL for that month — the dashboard renders a gap, not a crash.

The first-month NULL is by design

Every channel's first row has mom_pct = NULL because LAG has nothing to look back at. Don't filter these out — they're the channel's birthday and the dashboard should render them as "start of data" not an error.

The seasonality cousin — YoY %, not just MoM %

For seasonal businesses (retail, travel, education), MoM is misleading: December always beats November because of holidays. The senior follow-up is YoY:

LAG(total, 12) OVER (PARTITION BY channel ORDER BY mo) AS same_month_last_year,
100.0 * (total - LAG(total, 12) OVER (...)) /
   NULLIF(LAG(total, 12) OVER (...), 0) AS yoy_pct

Production trend dashboards usually show MoM and YoY side by side — MoM for short-cycle businesses, YoY for seasonal ones.

Trend-direction CASE for management readability
SELECT channel, mo, total, rolling_3mo, mom_pct,
  CASE
    WHEN total > rolling_3mo AND mom_pct > 0 THEN 'Accelerating'
    WHEN total < rolling_3mo AND mom_pct < 0 THEN 'Decelerating'
    WHEN total > rolling_3mo AND mom_pct < 0 THEN 'Cooling Spike'
    WHEN total < rolling_3mo AND mom_pct > 0 THEN 'Recovering'
    ELSE 'Stable'
  END AS momentum_status
FROM /* the trend CTE */;

Four quadrants instead of two columns. The CFO scans the column for "Decelerating" once a month and asks Sales Ops about each.

The ROWS vs RANGE reminder

ROWS BETWEEN 2 PRECEDING AND CURRENT ROW is physical rows: even if April had no data and is missing from the CTE, the window picks the last 3 existing rows, which silently smooths over a real gap. If the data has missing months, build a month spine first (LEFT JOIN against generate_series) so missing months show as zeros and the window math is honest.

The dialect cheat sheet
-- Postgres / BigQuery / Snowflake (DATE_TRUNC supported)
DATE_TRUNC('month', sale_date)

-- SQLite (no DATE_TRUNC)
DATE(sale_date, 'start of month')   -- or strftime('%Y-%m-01', sale_date)

-- MySQL
DATE_FORMAT(sale_date, '%Y-%m-01')

-- SQL Server
DATEFROMPARTS(YEAR(sale_date), MONTH(sale_date), 1)

6. The over-the-bar finish — alerting from the trend table

Production SPM doesn't just render the trend table — it alerts on it. A daily job materialises the CTE, joins to the previous run, and posts to Slack when:

  • any channel transitions to "Decelerating" for the first time in 3 months (early-warning signal)
  • any channel's mom_pct falls more than 2 standard deviations below its 12-month mean (statistical anomaly)
  • the spread between total and rolling_3mo exceeds ±25% (unusual volatility)
Senior signal: articulate the two-window pattern (smooth trendline + momentum velocity) as solving seasonality vs lucky-month conflation, name the NULLIF safety net unprompted, and pre-empt the YoY follow-up for seasonal businesses. The four-quadrant momentum-status CASE and the trend-table-driven alerting design are the over-the-bar finish that turns the query into a system.
