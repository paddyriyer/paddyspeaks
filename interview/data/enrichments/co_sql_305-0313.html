The Search team's KPI sounds simple: "of all searches, how many led to a play?". Every search is a small bet by Netflix — surface the right thing or the user bounces. The query is a time-bounded LEFT JOIN: each search either matches a play of the searched title within the next 30 minutes, or it doesn't. The senior conversation is what counts as "the searched title" and how to handle abandoned searches.

2. The SQL — LEFT JOIN with time-bounded predicate

SELECT s.query_text,
  COUNT(*) AS searches,
  COUNT(p.title_id) AS resulting_plays,
  ROUND(100.0 * COUNT(p.title_id) / NULLIF(COUNT(*), 0), 1) AS search_to_play_pct
FROM searches s
LEFT JOIN plays p
  ON p.user_id  = s.user_id
 AND p.title_id = s.top_title_id
 AND p.play_ts BETWEEN s.search_ts AND s.search_ts + INTERVAL '30 minutes'
GROUP BY s.query_text
HAVING COUNT(*) >= 50
ORDER BY search_to_play_pct DESC;

3. Strategic explanations

The 30-minute attribution window — not arbitrary

Netflix's published session-timeout is 30 minutes — if the user's next play is more than 30 minutes after the search, attribution becomes ambiguous (was it the search that drove it or some unrelated session?). Tighter windows (5-min) miss legitimate "I'll watch later" cases; wider windows (24-hour) over-attribute. 30 minutes is the empirically-determined sweet spot.

Why LEFT JOIN is non-negotiable

An INNER JOIN drops searches that didn't lead to a play — the dropouts are the entire point of the metric. LEFT JOIN keeps every search row; the matched-or-NULL state of p.title_id drives the conversion count. COUNT(p.title_id) counts only non-NULL (matched) rows.

The "why <= and not =" thing

The query asks "was there any play within 30 minutes?" but the LEFT JOIN can match multiple plays (if the user replayed the title). Two solutions:

-- A. dedup by counting distinct
COUNT(DISTINCT p.user_id, p.title_id, p.play_ts) AS resulting_plays

-- B. only count the FIRST play per search
LEFT JOIN LATERAL (
  SELECT title_id FROM plays p
  WHERE p.user_id = s.user_id AND p.title_id = s.top_title_id
    AND p.play_ts BETWEEN s.search_ts AND s.search_ts + INTERVAL '30 minutes'
  ORDER BY play_ts LIMIT 1
) p ON TRUE
The top_title_id IS NULL wrinkle — "no results" searches

The "asdf" search has top_title_id = NULL (no result returned). The LEFT JOIN never matches (NULL = anything is NULL), so the search counts but the play doesn't. That's the right behavior: a no-results search can never lead to a play of "the searched title" because there isn't one. But you should split this in reporting:

SUM(CASE WHEN s.top_title_id IS NULL THEN 1 END) AS no_results_searches,
SUM(CASE WHEN s.top_title_id IS NOT NULL AND p.title_id IS NULL THEN 1 END) AS results_but_no_play,
COUNT(p.title_id) AS results_and_play
The "any play" vs "searched-title play" distinction

The query attributes only when the user plays the top result of their search. But maybe the user searched for "stranger" and ended up watching "Stranger in the Night" (different title that came up in the same search results). To capture that, drop the p.title_id = s.top_title_id predicate — but then you need a way to know which titles appeared in the search-result rail (a search_results table with one row per displayed title).

The query-grade leaderboard — what the search team optimizes

"stranger" at 85% search-to-play is a healthy query — the algorithm matches user intent. "comedies for tonight" at 12% is broken — users searched for a vibe and didn't find one. The query-text leaderboard drives the search-relevance backlog.

The session-attribution variant

The strict definition above attributes per-search. The session-aware version groups searches into sessions and credits "did this search session lead to a play?":

WITH search_sessions AS (
  SELECT user_id, search_ts,
    SUM(CASE WHEN search_ts - LAG(search_ts) OVER (PARTITION BY user_id ORDER BY search_ts)
              > INTERVAL '5 minutes' THEN 1 ELSE 0 END
    ) OVER (PARTITION BY user_id ORDER BY search_ts) AS session_id
  FROM searches
)
SELECT user_id, session_id, COUNT(*) AS query_refinements,
  MAX(CASE WHEN p.title_id IS NOT NULL THEN 1 ELSE 0 END) AS led_to_play
FROM search_sessions s LEFT JOIN plays p ON ...
GROUP BY user_id, session_id;

"Query refinements" = how many times the user refined the search before something worked. High refinement-count = poor first-result relevance.

The reverse causality trap

Some users play first then search for the next thing. The query above only looks forward in time, but the same data could be misused to compute "play-to-search %" which has reverse causality. Always pin the temporal direction explicitly in the predicate.

The personalization confounder

Search-to-play rates are confounded by personalization: the algorithm puts your most-likely watch as the top result, so a high rate may just mean "the algorithm guessed right". Diff vs a random-results control bucket (an A/B test) gives you the algorithm's incremental lift over a no-personalization baseline.

Empty-handed dropout — the dark-pattern follow-up

The opposite signal: users who searched, didn't find, and left the platform within the next session. That's a churn risk:

WHERE p.title_id IS NULL
  AND NOT EXISTS (SELECT 1 FROM plays p2 WHERE p2.user_id = s.user_id
                  AND p2.play_ts BETWEEN s.search_ts AND s.search_ts + INTERVAL '24 hours')
Senior signal: the LEFT-JOIN-with-time-bounded-predicate idiom in one breath, surface the no-results-vs-results-but-no-play split unprompted, and pre-empt the multi-match dedup with the LATERAL trick. The session-aware refinement count and the personalization-confounder discussion are the over-the-bar finish.
