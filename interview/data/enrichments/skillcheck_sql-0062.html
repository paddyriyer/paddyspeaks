A revenue-ops team wants month-over-month growth by region for a board deck: "each month's revenue, the prior month's revenue, and the percentage change." The subtle requirement: if a region is missing a month, the "prior month" must be a true calendar predecessor — not just whatever row happens to sort before it. The interviewer is probing LAG plus gap detection.

1. The source data — monthly_sales

The edge case is NA's missing April. The row before May (by sort order) is March — but March is not May's calendar predecessor. The question demands that growth for May be NULL, because there is no April to compare against.

2. The SQL — LAG with a calendar-gap check

Original solution (flagged — prior_month_revenue not nulled on a gap):

SELECT month_start, region, revenue,
       LAG(revenue) OVER (PARTITION BY region ORDER BY month_start) AS prior_month_revenue,
       CASE
         WHEN LAG(month_start) OVER (PARTITION BY region ORDER BY month_start)
              = (month_start - INTERVAL '1 month')::date
         THEN ROUND((revenue / LAG(revenue) OVER (PARTITION BY region ORDER BY month_start) - 1) * 100, 2)
         ELSE NULL
       END AS mom_growth_pct
FROM monthly_sales
ORDER BY region, month_start;

Corrected solution — gate prior_month_revenue on the same gap check:

SELECT month_start, region, revenue,
       CASE
         WHEN LAG(month_start) OVER (PARTITION BY region ORDER BY month_start)
              = (month_start - INTERVAL '1 month')::date
         THEN LAG(revenue) OVER (PARTITION BY region ORDER BY month_start)
         ELSE NULL
       END AS prior_month_revenue,
       CASE
         WHEN LAG(month_start) OVER (PARTITION BY region ORDER BY month_start)
              = (month_start - INTERVAL '1 month')::date
         THEN ROUND((revenue / LAG(revenue) OVER (PARTITION BY region ORDER BY month_start) - 1) * 100, 2)
         ELSE NULL
       END AS mom_growth_pct
FROM monthly_sales
ORDER BY region, month_start;

The question states that on a gap, both prior_month_revenue and mom_growth_pct must be NULL. The original nulls only the growth percentage; prior_month_revenue still leaks the wrong month's value (March's 1320 against May). The corrected version applies the identical calendar-gap CASE to both columns.

4. Strategic explanations

Why LAG alone is not enough

LAG returns the previous row in the window, not the previous month. When April is missing, LAG over NA's May row hands back March — the rows are adjacent in sort order but a month apart on the calendar. The fix is to also LAG the month_start and verify it equals current_month − 1 month. Only when that holds is the previous row a genuine calendar predecessor; otherwise the comparison is meaningless and the result must be NULL.

The bug: NULL the value, not just the percentage

The original solution correctly NULLs mom_growth_pct on a gap but leaves prior_month_revenue as the raw LAG(revenue). The spec is explicit that on a gap both columns are NULL — a report showing "prior month revenue: 1320" next to a NULL growth is internally inconsistent and invites a reader to assume 1320 belongs to April. Any gap-aware derived column must apply the gap test uniformly. The corrected query wraps both columns in the same CASE.

First observation vs mid-series gap — same handling, different cause

For the first month of any region, LAG returns NULL, so LAG(month_start) = month_start - 1 month is NULL = ..., which is unknown — the CASE falls to ELSE NULL. For a mid-series gap, LAG(month_start) is a real date but the wrong one, so the equality is false — again ELSE NULL. One CASE elegantly covers both "no prior at all" and "prior exists but isn't last month".

Repeating window functions and the cleaner form

LAG(revenue) OVER (...) and LAG(month_start) OVER (...) appear several times with an identical window. PostgreSQL lets you name it once with a WINDOW clause — WINDOW w AS (PARTITION BY region ORDER BY month_start) — then write LAG(revenue) OVER w. The optimizer already computes a shared window once, so this is about readability, not speed; a senior would factor it out.

Division-by-zero on the growth ratio

revenue / LAG(revenue) divides by the prior month's revenue. If a region ever posted 0 revenue in a month, the next month's growth raises division by zero. Guard with NULLIF(LAG(revenue) OVER w, 0) so a zero base yields NULL growth instead of an error. The demo data has no zeros, but a real revenue series eventually will — flag it.

Senior signal: use LAG for period-over-period, but verify the previous row is a true calendar predecessor by also lagging month_start — and apply that gap test to every derived column (prior value and growth), not just the percentage, so a gap never leaks a mislabelled neighbour. Guard the ratio with NULLIF(..., 0).
