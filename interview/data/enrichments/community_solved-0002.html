A Community HR tool needs an org-chart query: given a manager's name, list everyone beneath them — direct reports, their reports, all the way down. The depth is unknown, so a fixed number of joins cannot work. This is the canonical recursive-CTE problem. The interviewer is watching whether you structure the anchor and recursive members correctly, name the termination condition, and guard against infinite loops.

This is a four-level tree rooted at asha (CEO, NULL manager). The hierarchy branches: asha → {ravi, meena}; ravi → {kiran, sunil}; kiran → {priya, arjun}; meena → {divya}. Querying for asha should pull the whole company; querying for kiran should pull only kiran + priya + arjun — the test of whether the recursion correctly walks a subtree, not the whole table.

2. The SQL — WITH RECURSIVE downward walk

WITH RECURSIVE cte AS (
    SELECT name, designation, id, manager_id, 1 AS hier_level FROM emp_details
    WHERE name = 'asha'
    UNION ALL
    SELECT ed.name, ed.designation, ed.id, ed.manager_id, (cte.hier_level + 1) AS hier_level
    FROM cte
    JOIN emp_details ed ON cte.id = ed.manager_id
)
SELECT * FROM cte;

The anchor member (before UNION ALL) seeds the recursion with the named manager at level 1. The recursive member (after) joins the working set back to emp_details on cte.id = ed.manager_id — "find everyone whose manager is someone I have already found" — incrementing the level each hop. It repeats until a hop adds no new rows.

Rows emerge in breadth-first waves — the whole of level n before level n+1 — which is how UNION ALL recursion naturally iterates. The output order is not guaranteed without an explicit ORDER BY.

4. Strategic explanations

Anchor + recursive member — the two halves

Every recursive CTE has exactly this shape: an anchor that runs once to seed the result, UNION ALL, and a recursive member that references the CTE by name. Execution: run the anchor; then repeatedly run the recursive member against just the rows produced by the previous iteration; stop when an iteration yields zero rows. Here the anchor is "the employee named asha"; each iteration finds the direct reports of whoever was found last round. Termination is automatic and implicit: once the recursion reaches the leaf engineers, the next join finds nobody whose manager is an engineer, returns empty, and the loop ends.

Walk direction — the join condition is everything

The recursion walks down because the join is cte.id = ed.manager_id: "rows whose manager is in my current set." To walk up instead — an employee's chain of managers to the CEO — you flip the join to cte.manager_id = ed.id and seed the anchor with the employee. Same skeleton, reversed join, opposite direction. Knowing this is one toggle, not a rewrite, is the senior insight.

The infinite-loop risk — UNION ALL has no built-in cycle guard

UNION ALL never deduplicates, so if the data has a cycle — employee A reports to B and B reports to A, a corrupt record — the recursion never terminates. Real org tables should be acyclic, but defensive code handles it. Two standard guards:

-- (a) depth cap: stop after N levels
... WHERE cte.hier_level < 20

-- (b) Postgres CYCLE clause: detect and halt on revisited ids
WITH RECURSIVE cte AS ( ... )
  CYCLE id SET is_cycle USING path
SELECT * FROM cte WHERE NOT is_cycle;

Switching UNION ALL to UNION would dedupe identical rows but does not reliably stop a cycle, since the hier_level column makes each revisit a distinct row. A depth cap or an explicit cycle/path check is the real fix.

Including or excluding the manager themselves

The anchor selects asha, so the result includes the queried manager (at level 1). If the question is strictly "employees under the manager", exclude the root with a final WHERE hier_level > 1, or seed the anchor with the manager's direct reports instead. Confirm whether the manager belongs in their own subtree — a small ambiguity that changes the row count.

Edge cases: unknown name, leaf node, duplicate names

If no employee matches the anchor's WHERE name = '...', the anchor is empty, the recursion never starts, and the result is zero rows — correct, not an error. Querying a leaf (e.g. priya, who manages nobody) returns just that one row. And the anchor filters on name, which is not a key: two employees named "asha" would both seed the recursion and merge two subtrees into one result. Seed on the primary key (WHERE id = ?) whenever the caller can supply it.

Senior signal: structure the recursive CTE as anchor + UNION ALL + recursive member, articulate that termination happens when an iteration adds no rows, know the join condition alone flips the walk up vs down, and proactively guard against cycles with a depth cap or CYCLE clause rather than trusting the data to be acyclic.
