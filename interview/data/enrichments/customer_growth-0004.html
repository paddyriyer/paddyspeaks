A lifecycle-marketing team wants to study customers who have repeatedly lapsed and come back — not one-time churners, but accounts with multiple resurrection cycles. The question (snapshot 2025-04-01) asks for cycle counts, the idle-gap statistics between active phases, and crucially whether each comeback brings a bigger basket than the order before the lapse (the AOV uplift). The interviewer is testing the full gaps-and-islands toolkit: define islands by a 90-day gap, then compare adjacent islands with LAG.

Customer 4 is the edge case: a steady customer who orders every two to three months but never lets a gap exceed 90 days. They form exactly one island, have zero resurrection cycles, and must drop out of the result — "loyal-and-continuous" is the opposite of "resurrected".

2. The SQL — 90-day-gap islands with adjacent-island comparison

WITH params AS (SELECT DATE '2025-04-01' AS snapshot),
flagged AS (
    SELECT o.customer_id, o.order_date, o.amount,
        CASE
          WHEN LAG(o.order_date) OVER w IS NULL THEN 1
          WHEN (o.order_date - LAG(o.order_date) OVER w) > 90 THEN 1
          ELSE 0
        END AS opens_island,
        ((SELECT snapshot FROM params) - o.order_date) AS days_back
    FROM orders o
    WHERE o.order_date >= (SELECT snapshot FROM params) - INTERVAL '730 days'
    WINDOW w AS (PARTITION BY o.customer_id ORDER BY o.order_date)
),
with_island AS (
    SELECT customer_id, order_date, amount, days_back,
        SUM(opens_island) OVER (
            PARTITION BY customer_id ORDER BY order_date
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS island_id
    FROM flagged
),
windowed AS (
    SELECT customer_id, order_date, amount, island_id,
        FIRST_VALUE(amount) OVER w_island AS first_amount,
        LAST_VALUE(amount)  OVER w_island AS last_amount
    FROM with_island
    WINDOW w_island AS (PARTITION BY customer_id, island_id ORDER BY order_date
                        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING)
),
islands AS (
    SELECT customer_id, island_id,
        MIN(order_date) AS island_start, MAX(order_date) AS island_end,
        MAX(first_amount) AS first_amount, MAX(last_amount) AS last_amount
    FROM windowed GROUP BY customer_id, island_id
),
phase_pairs AS (
    SELECT customer_id, island_id,
        island_start AS comeback_date, first_amount AS comeback_aov,
        LAG(island_end)  OVER (PARTITION BY customer_id ORDER BY island_id) AS prev_island_end,
        LAG(last_amount) OVER (PARTITION BY customer_id ORDER BY island_id) AS prev_island_aov,
        ((SELECT snapshot FROM params) - island_start) AS days_back_comeback
    FROM islands
),
resurrections AS (
    SELECT customer_id, comeback_date, comeback_aov,
        prev_island_aov AS pre_churn_aov,
        (comeback_date - prev_island_end) AS gap_days, days_back_comeback
    FROM phase_pairs
    WHERE prev_island_end IS NOT NULL AND (comeback_date - prev_island_end) > 90
),
total_islands_per_cust AS (
    SELECT customer_id, MAX(island_id) AS total_islands
    FROM islands GROUP BY customer_id
),
per_customer AS (
    SELECT r.customer_id, ti.total_islands,
        COUNT(*) AS resurrection_cycles,
        ROUND(AVG(gap_days), 0) AS avg_gap_days, MAX(gap_days) AS max_gap_days,
        ROUND(AVG(pre_churn_aov), 2) AS avg_pre_churn_aov,
        ROUND(AVG(comeback_aov), 2)  AS avg_comeback_aov,
        SUM(CASE WHEN days_back_comeback < 365 THEN 1 ELSE 0 END) AS y1_resurrections,
        SUM(CASE WHEN days_back_comeback >= 365 AND days_back_comeback < 730 THEN 1 ELSE 0 END) AS y2_resurrections,
        MAX(comeback_date) AS most_recent_comeback
    FROM resurrections r
    JOIN total_islands_per_cust ti ON ti.customer_id = r.customer_id
    GROUP BY r.customer_id, ti.total_islands
)
SELECT customer_id, total_islands, resurrection_cycles,
    avg_gap_days, max_gap_days, avg_pre_churn_aov, avg_comeback_aov,
    ROUND(avg_comeback_aov - avg_pre_churn_aov, 2) AS aov_uplift,
    y1_resurrections, y2_resurrections,
    CASE
        WHEN y2_resurrections = 0 AND y1_resurrections > 0 THEN 'y1_only'
        WHEN y1_resurrections > y2_resurrections THEN 'rising'
        WHEN y1_resurrections < y2_resurrections THEN 'falling'
        ELSE 'stable'
    END AS resurrection_trend,
    most_recent_comeback
FROM per_customer
ORDER BY resurrection_cycles DESC, aov_uplift DESC;

4. Strategic explanations

Two layers of islands: the running-sum trick

The first layer flags an order that opens a new island — either the customer's first order, or one more than 90 days after the previous. A running SUM(opens_island) over the customer's timeline turns those 0/1 flags into a monotonically increasing island_id: every flag bumps the id, and orders in between inherit it. This "flag-then-cumulative-sum" is the gaps-and-islands variant for event streams (as opposed to the m_idx - rn variant used for dense calendar grids).

FIRST_VALUE/LAST_VALUE need the full-frame window

LAST_VALUE is the notorious trap: with the default frame (RANGE UNBOUNDED PRECEDING TO CURRENT ROW) it returns the current row, not the last row of the partition. The query explicitly sets ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING so FIRST_VALUE grabs the island's opening basket and LAST_VALUE its closing basket. Without that frame the pre-churn AOV would be garbage. (You could also use MIN/MAX by order_date, but the explicit frame is the idiomatic answer when the question says "first/last".)

Comparing adjacent islands — LAG on the collapsed grain

After collapsing to one row per island, LAG(island_end) and LAG(last_amount) ordered by island_id pull the previous island's end date and closing basket alongside the current island's start. A resurrection is then simply: previous island exists and the gap between previous-end and current-start exceeds 90 days. The first island per customer has LAG = NULL and is correctly not a resurrection — you cannot resurrect into your first-ever phase.

Why total_islands comes from a separate CTE

resurrection_cycles counts rows in resurrections (gaps > 90d), and that count should equal total_islands - 1. But the join is to a separately computed MAX(island_id) because a customer with only one island produces zero resurrection rows — they vanish from resurrections entirely. Computing total_islands independently and inner-joining it keeps the arithmetic honest for customers who do appear, while still excluding the single-island customers (customer 4) because the inner join finds no matching resurrection rows.

The resurrection_trend ordering of CASE branches matters

'y1_only' (Y2 = 0, Y1 > 0) is checked first. If you instead checked y1 > y2 first, a customer with 3 Y1 and 0 Y2 comebacks would be labelled 'rising' — technically true but it hides the more useful signal that this customer's entire resurrection history is recent. Customer 3 is exactly this case. Branch order in a CASE is semantics, not style.

Senior signal: distinguish the flag-then-running-SUM island pattern from the m_idx - rn variant, explicitly frame FIRST_VALUE/LAST_VALUE with ROWS UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING to avoid the LAST_VALUE default-frame trap, compute total_islands in its own CTE so single-island customers are cleanly excluded, and order the trend CASE branches so y1_only wins over a misleading rising.
