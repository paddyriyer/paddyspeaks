Amazon's category team wants the five products that generate the most revenue, to decide where to spend marketing budget. The query is a textbook GROUP BY with a top-N cut — but an interviewer is really checking two things: do you compute gross as qty × price per line and then sum, and do you notice that the schema ships a tempting amount column that may or may not mean the same thing.

1. The source data — orders

Item 101 appears on two order lines; item 102 on two as well. Order 4 is cancelled but still carries a quantity and price. That cancelled row is the interview — should it count toward gross revenue?

2. The SQL — SUM(qty*price) top 5

SELECT item_id, SUM(qty*price) AS gross FROM orders
GROUP BY item_id ORDER BY gross DESC LIMIT 5;

Per line the engine computes qty × price, then SUM rolls those line values up per item_id. ORDER BY gross DESC LIMIT 5 keeps the five biggest.

4. Strategic explanations

The cancelled-order filter the interviewer is waiting for

Gross revenue almost always means realised revenue. Order 4 is cancelled, so item 103's entire $50 is phantom. Add the filter and 103 drops to zero:

SELECT item_id, SUM(qty*price) AS gross FROM orders
WHERE status = 'completed'
GROUP BY item_id ORDER BY gross DESC LIMIT 5;

Confirm the definition before writing SQL: "top grossing — over all orders, or only completed ones?" Getting the filter right matters more than the aggregation.

qty*price per row vs SUM(qty)*SUM(price) — a real trap

It is tempting to write SUM(qty) * SUM(price). That is wrong: it multiplies total quantity by total price, mixing line items that may have different prices. For item 101 it would give (2+3) × (50+50) = 500, not 250. The multiplication must happen inside the row, before the sum: SUM(qty*price). The two only coincide when there is a single row per group.

The amount column — which is the source of truth?

The schema carries an amount column that, on this data, often equals qty*price — but not always (order 6: amount 400 = 2×200, yet order 4: amount 50 = 5×10). They can drift: one pre-discount, one post-discount; one with tax, one without. Do not guess. Ask which column the business treats as revenue. If amount is the authoritative post-discount figure, the query becomes SUM(amount) and the line-item multiplication disappears entirely.

The ties-at-the-cutoff problem

LIMIT 5 returns exactly five rows. If the 5th and 6th items have equal gross, the engine keeps one and drops the other arbitrarily — non-deterministic, and a stakeholder will notice an item missing. When ties at the boundary must all appear, use a ranking window:

SELECT item_id, gross FROM (
  SELECT item_id, SUM(qty*price) AS gross,
         RANK() OVER (ORDER BY SUM(qty*price) DESC) AS rk
  FROM orders GROUP BY item_id
) t WHERE rk <= 5;

This returns all items ranked 5th or better — possibly six or seven rows if there are ties. Add a deterministic secondary sort (ORDER BY gross DESC, item_id) if exactly five is required.

Edge cases: NULLs and an empty result

If qty or price is NULL on a line, qty*price is NULL and SUM simply skips it — that line contributes nothing, which may understate an item. Coalesce if the data is dirty. And if the table is empty the query returns zero rows, not a row of zeros — fine for a ranking, but worth stating.

Senior signal: multiply qty*price per row before summing (never SUM(qty)*SUM(price)), add the status = 'completed' filter that separates real revenue from cancelled orders, surface the amount-vs-computed ambiguity, and handle ties at the LIMIT boundary with a ranking window rather than an arbitrary cut.
