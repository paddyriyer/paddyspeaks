A growth team wants the re-engagement list: "which customers signed up but have never placed an order?" The query is the classic anti-join — a LEFT JOIN followed by WHERE ... IS NULL. The interviewer is checking whether you reach for the anti-join idiom rather than a subquery, and whether you pick the right column for the NULL test.

2. The SQL — anti-join via LEFT JOIN + IS NULL

SELECT c.customer_id, c.name
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL;

4. Strategic explanations

Why the LEFT JOIN + IS NULL pattern finds non-matches

A LEFT JOIN keeps every customers row. Where a customer has matching orders, the order columns are filled; where a customer has none, the join fabricates a row with every order column NULL. Filtering WHERE o.order_id IS NULL therefore keeps precisely the customers the join could not match — that is the anti-join.

The critical detail — test a NEVER-NULL column

The IS NULL check must target a column that is NULL only because the join failed — never because of real data. order_id is a primary key, so a real order can never have a NULL order_id; a NULL there unambiguously means "no match." If you tested a nullable column instead — say WHERE o.amount IS NULL — you would also catch real orders that happen to have a NULL amount, producing wrong results. Always anti-join on the primary key or join key.

The "cancelled order" subtlety

Noah ordered but it was cancelled. By row existence he is excluded from this list. If the business actually means "no successful order," push the status into the join's ON clause, not the WHERE:

LEFT JOIN orders o
  ON c.customer_id = o.customer_id
 AND o.status = 'completed'
WHERE o.order_id IS NULL

Putting o.status = 'completed' in WHERE instead would discard the fabricated all-NULL rows and break the anti-join entirely — a classic trap. Filter the right table in ON; keep WHERE for the IS NULL test.

Equivalent forms — NOT EXISTS and NOT IN

NOT EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id) is equally idiomatic, often the planner's preferred shape, and reads as a direct statement of intent. Avoid NOT IN (SELECT customer_id FROM orders): if any customer_id in orders is NULL, NOT IN returns no rows at all due to three-valued logic. NOT EXISTS and the anti-join are NULL-safe; NOT IN is the landmine.

Performance

The anti-join wants an index on orders(customer_id) — the join probe and the existence check both ride it. With that index, all three forms (LEFT JOIN/IS NULL, NOT EXISTS, anti-join) typically compile to the same anti-join operator, so choose the one your team finds most readable.

Senior signal: recognise this as an anti-join, insist the IS NULL test hit a never-null key like order_id, know that any status filter belongs in ON not WHERE, and warn that NOT IN breaks on NULLs while NOT EXISTS does not.
