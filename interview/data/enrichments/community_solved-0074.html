A logistics team wants delivery lead time: for each customer, how many days elapsed between their first shipment and a delivery. The interviewer is probing a two-step pattern — isolate a per-customer anchor date with a window function, then date-subtract every relevant event against that anchor — plus the NULL behaviour when a customer has a delivery but no shipment on record.

Three edge cases: customer 200 has two shipments (only the earliest is the anchor); customer 300 has a delivery but no shipment at all; customer 400 shipped but was never delivered. Each tests a different join/NULL path.

2. The SQL — FIRST_VALUE anchor, then date diff

WITH customer_frst_ship_date AS (
  SELECT DISTINCT customer_id,
         FIRST_VALUE(order_date) OVER (PARTITION BY customer_id
                                       ORDER BY order_date ASC) AS first_ship_date
  FROM customer_orders
  WHERE status = 'Shipped'
),
shipped_delivered_date AS (
  SELECT c.*, s.first_ship_date
  FROM customer_orders c
  LEFT JOIN customer_frst_ship_date s ON c.customer_id = s.customer_id
  WHERE c.status = 'Delivered'
    AND c.order_date > s.first_ship_date
)
SELECT customer_id,
       order_date AS first_delivered_date,
       first_ship_date,
       (order_date::date - first_ship_date::date) AS diff
FROM shipped_delivered_date;

4. Strategic explanations

Why FIRST_VALUE + DISTINCT isolates the anchor

FIRST_VALUE(order_date) OVER (PARTITION BY customer_id ORDER BY order_date ASC) evaluated over only the Shipped rows returns each customer's earliest shipment date. But a window function does not collapse rows — customer 200 with two shipments gets that same first date stamped on both rows. SELECT DISTINCT then dedupes to one row per customer. It works, but it is a slightly indirect idiom: MIN(order_date) ... GROUP BY customer_id expresses "earliest ship date" more directly and needs no DISTINCT. Naming that the aggregate is the cleaner equivalent is a senior observation.

The NULL filter that quietly excludes customer 300

Customer 300 has a delivery but no shipment, so the LEFT JOIN leaves first_ship_date as NULL. The predicate c.order_date > s.first_ship_date then becomes date > NULL, which evaluates to unknown — and WHERE keeps only rows that are true, so customer 300 is dropped. That is the desired outcome here (you cannot measure lead time without a ship date), but it is happening as a side effect of three-valued logic, not an explicit rule. A senior states it: "the > comparison silently filters delivered-but-never-shipped customers because comparing to NULL is unknown." If you wanted to surface them as anomalies instead, you would test first_ship_date IS NULL explicitly.

The LEFT JOIN is effectively an INNER JOIN here

Because the very next line filters order_date > first_ship_date — which is never true when first_ship_date is NULL — the unmatched (NULL) rows a LEFT JOIN preserves are immediately discarded. So this LEFT JOIN behaves like an INNER JOIN. That is not wrong, but it is worth recognising: if the intent were to keep no-shipment customers visible (with a NULL diff), the filter would have to move into the join's ON clause or be guarded.

Date subtraction — dialect-dependent

order_date::date - first_ship_date::date returns an integer day count in PostgreSQL — subtracting two date values yields days directly. Other engines differ: SQL Server needs DATEDIFF(day, first_ship_date, order_date); MySQL uses DATEDIFF(order_date, first_ship_date); BigQuery uses DATE_DIFF(order_date, first_ship_date, DAY). If the columns were TIMESTAMP rather than DATE, the Postgres subtraction would return an interval (with hours/minutes), not a plain integer — cast to date first, exactly as the solution does, to get whole days.

The misleading alias — first_delivered_date

The output column is aliased first_delivered_date, but customer 100 returns two delivery rows — the query lists every delivery after the ship date. If the spec truly wants only the first delivery per customer, add a ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date) and keep rank 1, or aggregate with MIN(order_date). Always check that an alias's promise matches the query's grain.

Senior signal: recognise FIRST_VALUE+DISTINCT as a per-customer anchor (and that MIN ... GROUP BY is the cleaner equivalent), then explain that order_date > NULL is unknown — which is what silently drops the delivered-but-never-shipped customer and makes the LEFT JOIN behave like an INNER JOIN. Flagging the over-promising first_delivered_date alias is the finishing touch.
