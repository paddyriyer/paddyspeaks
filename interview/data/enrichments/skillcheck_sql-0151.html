A sales-ops dashboard needs per-agent performance: total order value driven by each agent's customers, the commission they earned on it, and their average order size. The data spans three tables — agents own customers, customers place orders — so the query must chain two joins and aggregate at the agent grain. The interviewer is probing whether you can aggregate cleanly across a join chain without fan-out errors, and whether you handle the commission and rounding correctly.

The shape that tests you: A002 and A010 each have one customer with multiple orders and one with a single order. Aggregating across the join must total all orders per agent without an order being counted once per customer — and without losing the single-order customers.

2. The SQL — three-table join with grouped aggregation

SELECT a.agent_code,
       COUNT(o.ord_num) AS num_orders,
       SUM(o.ord_amount) AS total_amount,
       ROUND((SUM(o.ord_amount) * a.commission)::numeric, 2) AS commission_earned,
       ROUND(AVG(o.ord_amount)::numeric, 2) AS avg_order
FROM agent a
JOIN customer c ON c.agent_code = a.agent_code
JOIN orders o ON o.cust_code = c.cust_code
GROUP BY a.agent_code, a.commission
ORDER BY total_amount DESC, a.agent_code;

The two inner joins flatten agent → customer → order into one row per order (carrying the agent's code and commission). GROUP BY agent_code, commission collapses those order rows back to one row per agent; SUM, COUNT, and AVG aggregate over each agent's orders.

4. Strategic explanations

Aggregate at the join's leaf grain, group back up

After both joins, each result row is one order — the finest grain in the chain. SUM(o.ord_amount) then adds order amounts, and COUNT(o.ord_num) counts orders, with no double-counting: an order belongs to exactly one customer, and that customer to exactly one agent, so each order row is unique. The mental model — "join down to the leaf, then GROUP BY the level you want to report" — is the reliable way to reason about multi-table aggregation.

When this would fan out — and why it does not here

Fan-out happens when a row joins to many rows on a side you are also aggregating. If you joined a second child table to agent (say a visits table) in the same query, each order would pair with every visit and SUM(ord_amount) would inflate by the visit count. Here the chain is strictly linear — agent 1:N customer 1:N order, each step a single fan — so order rows stay distinct. Recognising the linear chain is what tells you a single grouped query is safe; two independent children would need separate aggregation (CTEs) before joining.

Inner join silently drops zero-order agents — matches the spec

The question says "exclude agents with zero orders", and inner joins do exactly that for free: an agent with no customers, or customers with no orders, produces no rows and disappears. If the requirement were instead "list every agent, zero where idle", you would switch to LEFT JOIN and wrap aggregates — COALESCE(SUM(o.ord_amount), 0) — because SUM over no rows is NULL. State which behaviour the spec wants; here inner join is correct.

GROUP BY must include commission — functional dependency

commission is referenced in the SELECT (inside the commission calculation) but is not aggregated, so it must appear in GROUP BY — or the query is ambiguous and errors in strict engines. Adding a.commission is safe because it is functionally dependent on agent_code (one commission per agent), so it does not split any group. Postgres 9.1+ can infer this when you group by a primary key; grouping both columns explicitly is the portable, unambiguous form.

Rounding, casts, and the ::numeric footnote

ROUND(…, 2) with a scale argument requires a numeric in Postgres — ROUND(double precision, int) does not exist — hence the explicit ::numeric casts on the commission product and the average. AVG already returns numeric for a numeric column, so the cast there is harmless insurance. Round at the presentation step only: rounding mid-calculation (e.g. rounding each order before summing) would accumulate error. Other engines (SQL Server, MySQL) accept ROUND(float, n) directly — the cast is a Postgres-specific detail.

Senior signal: aggregate a linear multi-table chain by joining down to the leaf grain then grouping back up, recognise that a single 1:N chain does not fan out (two independent children would), use inner joins to drop zero-order agents per the spec, and include the non-aggregated commission in GROUP BY as a functional dependency.
