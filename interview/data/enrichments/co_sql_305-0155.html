Netflix's hardware catalogue team wants the product list rendered the way a merchandiser thinks: grouped by category, with the priciest items first inside each group, and ties broken alphabetically by name. It is a single ORDER BY, but the interviewer is checking whether you understand that sort keys are applied in sequence — and that mixing ASC and DESC per column is not just legal, it is the point.

1. The source data — products

product_idcategorypricenamesku
1Audio199.00Studio HeadphonesSKU-A1
2Audio199.00Aero EarbudsSKU-A2
3Audio89.00Clip SpeakerSKU-A3
4Video499.004K MonitorSKU-V1
5Video499.00Action CamSKU-V2
6Cables12.00HDMI 2mSKU-C1

Products 1 and 2 share a category and a price (199.00) — the only thing left to order them is the third sort key, name. Products 4 and 5 do the same in Video. Those ties are exactly where a multi-column ORDER BY earns its keep.

2. The SQL — ORDER BY a, b, c

SELECT * FROM products ORDER BY category ASC, price DESC, name ASC;

Three sort keys, evaluated left to right. category ascending groups the rows; within each category, price descending; within an equal price, name ascending.

3. The resulting analysis

Inside Audio, the two 199.00 rows are tied on the first two keys, so the third key decides: "Aero Earbuds" sorts before "Studio Headphones", and product 2 comes first. Without that third key, the order of those two rows would be undefined — whatever the planner happened to emit.

4. Strategic explanations

Sort keys are lexicographic — later keys only break ties in earlier ones

A multi-column ORDER BY works like alphabetising words: the second letter only matters when the first letters match. price DESC is consulted only for rows that share a category; name ASC is consulted only for rows that share both category and price. If every category had distinct prices, the third key would never fire. State this ordering explicitly — interviewers want to hear "keys are applied in priority order", not "it sorts by all three".

Per-column direction is independent

ASC and DESC attach to each column individually. ORDER BY category ASC, price DESC means category low-to-high and price high-to-low simultaneously — not a single global direction. A frequent mistake is writing ORDER BY category, price DESC and assuming DESC covers both; it does not. category there defaults to ASC. If you wanted both descending you would write category DESC, price DESC.

The deterministic-order trap — add a tiebreaker

Even with three keys, products tied on category, price and name would still come back in arbitrary order. For a result that is byte-for-byte reproducible across runs — needed for pagination, snapshot tests, or diffing — append a guaranteed-unique final key, almost always the primary key:

SELECT * FROM products
ORDER BY category ASC, price DESC, name ASC, product_id ASC;

A senior answer mentions that ORDER BY without a unique trailing key is "stable enough for display, not stable enough for tests".

NULLs in a sort key

If price contained NULLs, Postgres places them last under ASC and first under DESC by default — so a NULL-priced item would jump to the top of its category here. That is rarely intended. Control it explicitly:

ORDER BY category ASC, price DESC NULLS LAST, name ASC;

Engines disagree on the default (MySQL sorts NULLs first under ASC; Postgres sorts them last), so NULLS LAST / NULLS FIRST is the portable, intentional choice.

SELECT * versus naming columns

SELECT * is fine for an ad-hoc pull, but in shipped code it is brittle: add or reorder a column and downstream consumers silently break. The sort keys do not have to appear in the select list — you could sort by price while only projecting name — so naming the projected columns and the sort keys separately is the production-grade habit.

Senior signal: explain that ORDER BY keys are applied in priority order with each later key breaking ties in the earlier ones, that ASC/DESC is per-column, and that a result is only reproducible when the final sort key is unique — then pre-empt the NULL-ordering default before it bites.
