Ubisoft's player-engagement team runs a re-activation campaign and needs the mailing list: every active customer who is not on the blacklist. It is the canonical "set difference" problem. The interviewer asking this is probing whether you reach for the right exclusion construct — and, above all, whether you know the silent way NOT IN breaks when the exclusion list contains a NULL.

1. The source data — active_customers and blacklist

The solution ships its own fixture, so the data is fixed:

active_customers.id
1, 2, 3, 4, 5, 6, 7, 8
blacklist.id
2, 5, 7

Eight active customers; three of them — 2, 5, 7 — are blacklisted. The expected mailing list is the other five. Crucially, here blacklist contains no NULLs — which is exactly why this version of the query is safe.

2. The SQL — NOT IN subquery

SELECT id FROM active_customers
WHERE id NOT IN (SELECT id FROM blacklist);

For each active customer, NOT IN keeps the row only when its id matches none of the values returned by the subquery.

3. The resulting analysis

id
1
3
4
6
8

Five customers — 1, 3, 4, 6, 8 — the active set minus the three blacklisted ids. The query is correct because the blacklist has no NULL. Change that one fact and the result collapses, as the next section shows.

4. Strategic explanations

The NOT IN + NULL trap — the entire point of the question

If blacklist ever contained a single NULL id, this query would return zero rows. NOT IN (2, 5, 7, NULL) expands to id <> 2 AND id <> 5 AND id <> 7 AND id <> NULL. That last comparison is never true — it is unknown — so the whole AND can never be true, and every row is filtered out. An empty mailing list, no error, campaign quietly dead. Always either guarantee no NULLs or filter them: NOT IN (SELECT id FROM blacklist WHERE id IS NOT NULL).

NOT EXISTS — the NULL-safe default

The robust idiom is NOT EXISTS, which is immune to the NULL problem:

SELECT a.id FROM active_customers a
WHERE NOT EXISTS (
  SELECT 1 FROM blacklist b WHERE b.id = a.id
);

NOT EXISTS asks "is there a matching row?" — a plain yes/no. A NULL in blacklist simply fails the b.id = a.id join and is ignored, rather than poisoning the whole predicate. For exclusion against a column that might be nullable, prefer NOT EXISTS.

LEFT JOIN / IS NULL — the anti-join

A third correct form, the classic anti-join:

SELECT a.id FROM active_customers a
LEFT JOIN blacklist b ON a.id = b.id
WHERE b.id IS NULL;

Keep every active customer, then discard any that found a blacklist match (its joined columns stay NULL only for non-matches). Equivalent here; some planners optimise it identically to NOT EXISTS.

EXCEPT — the set operator

Because this is literally set difference, EXCEPT expresses it directly:

SELECT id FROM active_customers
EXCEPT
SELECT id FROM blacklist;

One caveat that changes behaviour: EXCEPT also de-duplicates its inputs. If active_customers had the same id twice, EXCEPT collapses it to one row, whereas NOT IN and NOT EXISTS preserve both.

Edge case: empty blacklist

If blacklist were empty, NOT IN (empty set) is true for everyone, so all eight customers pass — the desired behaviour. (This is the one case where NOT IN is well-behaved with no rows; the danger is purely a NULL value, not an empty set.)

Senior signal: name the NOT IN + NULL trap unprompted — one NULL in the subquery empties the entire result — and default to NOT EXISTS (or a LEFT JOIN / IS NULL anti-join) for exclusion against any nullable column, reserving EXCEPT for true set difference while noting it de-duplicates.
