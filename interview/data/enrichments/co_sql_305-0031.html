Amazon's finance team keeps each fiscal year's sales in its own table — sales_2024, sales_2025 — and the year-over-year deck needs one tidy two-row total. The reference solution stitches the totals together with UNION ALL. It works, and the interview is about why it is UNION ALL and not UNION, and what happens when there are ten yearly tables instead of two.

1. The source data — sales_2024 and sales_2025

sales_2024a$100
sales_2024b$250
sales_2024c$75
sales_2025a$300
sales_2025b$120
sales_2025c$90
sales_2025d$60

Two tables with identical structure. 2024 totals $425 across three rows; 2025 totals $570 across four. The yearly totals happen to be different numbers — keep that in mind, because if they were equal the choice of UNION vs UNION ALL would suddenly matter.

2. The SQL — UNION ALL of two SUMs

SELECT '2024' AS yr, SUM(amount) FROM sales_2024
UNION ALL
SELECT '2025', SUM(amount) FROM sales_2025;

Each branch collapses a whole table to one row with a hard-coded year literal, then the two single-row results are stacked vertically.

3. The resulting analysis

Two rows, exactly as the deck wants. Note the second column is just called sum — only the first branch's column aliases survive, and that branch never named the aggregate.

4. Strategic explanations

Why UNION ALL, not UNION — and the trap if you pick wrong

UNION de-duplicates: it sorts the combined result and drops identical rows. UNION ALL concatenates blindly. Here the danger is real — suppose two years coincidentally posted the same total revenue. With UNION, the rows ('2024', 500) and ('2025', 500) stay distinct (the year literal differs), so you are safe this time. But the moment the branches select only SUM(amount) with no year label, two equal totals would silently collapse to one row. UNION ALL is correct because the intent is "keep every branch's row", and it skips the needless dedupe sort — faster, and free of accidental row loss.

Column names come from the first SELECT only

The output column is sum because the second branch's names are ignored entirely — a set operation takes its column names from the first query. Name it once, in the first branch:

SELECT '2024' AS yr, SUM(amount) AS total_sales FROM sales_2024
UNION ALL
SELECT '2025' AS yr, SUM(amount) AS total_sales FROM sales_2025;

The aliases on the second branch are documentation for the reader; the engine discards them.

Type and column-count alignment — the silent failure modes

Branches in a UNION must match on column count and have compatible types, matched by position, not by name. Selecting '2024' (text) above SUM(amount) (numeric) in different positions across branches would either error or coerce unexpectedly. The literal '2024' is text here; if a consumer needs a real year integer, cast it: 2024 AS yr or '2024'::int — consistently in every branch.

The design smell: year-per-table doesn't scale

The honest senior comment is that this schema is the actual problem. Ten years means a ten-branch UNION ALL that must be hand-edited every January. A single partitioned sales table with a year column reduces the whole report to:

SELECT year, SUM(amount) AS total_sales
FROM sales
GROUP BY year
ORDER BY year;

If you cannot change the schema, generate the union dynamically or wrap the per-year tables in a view. Flagging the data-model issue — while still delivering the asked-for query — is the over-the-bar move.

Edge case: an empty year table

SUM(amount) over an empty table returns NULL, not 0 — so a year with no sales rows shows a blank, which a chart may render as a gap. Wrap it if the deck needs a hard zero: COALESCE(SUM(amount), 0) AS total_sales.

Senior signal: justify UNION ALL over UNION by intent (keep every row, skip the dedupe sort), know that output column names come from the first branch and that branches align by position, and call out that one-table-per-year is a schema smell that a single partitioned table with a GROUP BY would fix.
