A trailing moving average is a common window-function task, and it hides two traps at the edges. First, the rows at the start of each group cannot have a full trailing window — and the engine does not return NULL or pad with zeros there, it simply averages the rows that exist. Second, PARTITION BY resets the frame at every group boundary, so the window never bleeds across groups. Forgetting either gives wrong numbers.

2. The query — a 3-day trailing average per region

SELECT region, day, cnt,
       ROUND(AVG(cnt) OVER (PARTITION BY region ORDER BY day
                            ROWS BETWEEN 2 PRECEDING AND CURRENT ROW), 2) AS moving_avg_3d
FROM signups
ORDER BY region, day;

The frame ROWS BETWEEN 2 PRECEDING AND CURRENT ROW asks for this row plus the two before it — a 3-row trailing window. PARTITION BY region scopes that window to one region at a time.

Day 1 of each region averages a single value; day 2 averages two. Only from day 3 onward is the window full. EU day 1 is 7.00, not a blend of US’s tail — the partition reset keeps the regions separate.

4. Strategic explanations

Partial windows at the edge — no NULL, no zero-padding

ROWS BETWEEN 2 PRECEDING AND CURRENT ROW requests up to three rows, but at the partition’s first row there are no preceding rows to include. SQL does not error, return NULL, or invent zero rows — the frame simply contains whatever rows exist (here, just the current one). So day 1’s “3-day average” is the day-1 value itself. Candidates often expect the first two rows to be NULL until a full window is available; they are not.

PARTITION BY resets the frame

The frame is always evaluated within the current partition. EU day 1 cannot reach back into the US rows, even though US day 4 physically precedes it in the table. Drop PARTITION BY region and the bug is silent but real: EU day 1 would average US day 3, US day 4, and EU day 1 — (30 + 40 + 7) / 3 — a meaningless cross-region blend.

If you want a strict full-window-only average

When a partial-window average is undesirable (an under-counted day-1 figure can mislead), make the partiality explicit: compute COUNT(*) OVER (same frame) and null out rows where the count is below 3, e.g. CASE WHEN COUNT(*) OVER (...) = 3 THEN AVG(...) END. That is a deliberate choice, not the default.

Dialect notes

The frame syntax and partial-edge behaviour are ANSI standard and identical in PostgreSQL, Snowflake, BigQuery, SQL Server, Oracle, and MySQL 8.0. AVG over an integer column returns an exact numeric in PostgreSQL, so ROUND(..., 2) applies cleanly; in engines where AVG yields a float, cast before rounding.

Senior signal: a trailing ROWS BETWEEN n PRECEDING AND CURRENT ROW frame silently shrinks at the partition edge — the first rows average a partial window rather than returning NULL — and PARTITION BY hard-stops the frame at group boundaries. Add PARTITION BY to prevent cross-group bleed, and gate on COUNT(*) OVER (...) if partial windows must be excluded.
