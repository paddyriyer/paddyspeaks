A manufacturing-ops analyst is chasing the work-in-progress board: "which parts are still on the line, and how many steps are left on each?" A part counts as finished only when every recorded assembly step has a real finish_at timestamp. The interviewer is probing whether you can express an "all rows satisfy a condition" test as its negation — "at least one row fails" — using a filtered aggregate, and whether you understand that NULL means "step not yet done".

1. The source data — parts_assembly

Three edge cases are baked in: charger has every step done (must be excluded), seat is a single-row group (one NULL step), and battery has more than one open step so the count must exceed 1.

2. The SQL — filtered count with a HAVING gate

SELECT part_name, COUNT(*) FILTER (WHERE finish_at IS NULL) AS incomplete_steps
FROM parts_assembly
GROUP BY part_name
HAVING COUNT(*) FILTER (WHERE finish_at IS NULL) > 0
ORDER BY part_name;

COUNT(*) FILTER (WHERE finish_at IS NULL) counts only the open steps inside each part's group. HAVING … > 0 keeps only the parts that have at least one — the negation of "all steps finished" — and the same expression in the SELECT reports how many remain.

charger is absent: every step has a timestamp, so its filtered count is 0 and HAVING drops it. seat appears with 1 even though it has a single row — a one-member group is still a group.

4. Strategic explanations

"Every row satisfies X" becomes "zero rows fail X"

SQL has no direct "for all rows in this group" quantifier, so you flip it: a part is finished iff no step is open, i.e. the count of open steps is 0. Asking for the unfinished parts is then simply count > 0. This negation trick — universal turned into existential — is the core idea, and it generalises: "all orders shipped" becomes "count of unshipped = 0".

Why FILTER beats SUM(CASE…) here

COUNT(*) FILTER (WHERE finish_at IS NULL) is the standard-SQL aggregate filter and reads as exactly what it does. The portable equivalent is SUM(CASE WHEN finish_at IS NULL THEN 1 ELSE 0 END) or COUNT(*) - COUNT(finish_at) — the latter exploiting that bare COUNT(column) already skips NULLs. All three give the same number; FILTER is clearest where the dialect supports it.

The NULL trap — = NULL never matches

You must write finish_at IS NULL, never finish_at = NULL. Equality against NULL evaluates to unknown, not true, so WHERE finish_at = NULL silently filters everything out and the report comes back empty. This is the single most common bug in "find the missing data" queries.

HAVING, not WHERE — the filter is on the aggregate

The condition "this part has an open step" is a property of the whole group, computed after GROUP BY, so it belongs in HAVING. You cannot move it to WHERE: WHERE finish_at IS NULL would discard the finished steps before grouping, corrupting the count and changing which parts even appear. WHERE filters rows, HAVING filters groups.

Dialect notes

FILTER is supported by PostgreSQL and SQLite. MySQL has no FILTER clause — use SUM(finish_at IS NULL) (MySQL treats the boolean as 1/0) or the portable SUM(CASE WHEN …). SQL Server likewise needs SUM(CASE…). The HAVING gate is identical everywhere.

Senior signal: reframe "every step finished" as "zero open steps", count open steps with COUNT(*) FILTER (WHERE finish_at IS NULL), gate unfinished parts with HAVING > 0, and never write = NULL — use IS NULL, because equality with NULL yields unknown and quietly empties the result.
