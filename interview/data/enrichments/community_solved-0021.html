A Community question on a simple customer/orders pair: list the orders placed by customers named Jill or Eva, showing first name, order date, details and cost. It is deliberately easy — but the reference joins orders to customer with a LEFT JOIN, and an interviewer will ask whether that direction was a choice or an accident, and what happens to an order whose customer is missing.

1. The source data

customer:

idfirst_namelast_namecity
1JillBrownBoston
2EvaStoneAustin
3MarkHillBoston
5JillCarterDenver

orders has seven rows linked by cust_id. Two subtleties: there are two different Jills (ids 1 and 5) — filtering on a first name is not the same as filtering on a person — and customer 5 (Jill Carter) has one order while customer 4 (Susan, not selected) also has one.

2. The SQL — orders LEFT JOIN customer, filter on name

select c.first_name, o.order_date, o.order_details, o.total_order_cost
from orders o left join customer c
on o.cust_id = c.id
where c.first_name in ('Jill','Eva')
order by c.id asc;

Start from orders, attach the matching customer, keep only rows whose customer first name is Jill or Eva, sort by customer id.

Five orders. The first two Jill rows are Jill Brown (id 1); the last is Jill Carter (id 5) — two different people collapsed under one first name, and order by c.id puts them at opposite ends. The output cannot tell them apart, which is the lurking data-quality point.

4. Strategic explanations

The LEFT JOIN here behaves like an INNER JOIN — and that is a trap

The join is LEFT, but the WHERE c.first_name IN ('Jill','Eva') clause references a right-table column. For any order with no matching customer, c.first_name is NULL, and NULL IN ('Jill','Eva') is never true — so those rows are filtered out anyway. The LEFT JOIN is silently downgraded to an INNER JOIN. If preserving orphan orders mattered, the predicate would have to move into the ON clause or allow NULLs. Here the inner-join behaviour is fine, but recognising the downgrade is the senior observation.

Filtering on a name is not filtering on a person

Two customers are named Jill (ids 1 and 5). WHERE first_name IN ('Jill','Eva') matches both, mixing their orders in one result. If the brief means a specific customer, filter on id or customer_id — a key, not a display name. Names are not unique and not stable; treating one as an identifier is a classic junior mistake. Surface the ambiguity: "Jill the person, or anyone whose first name is Jill?"

Why the join direction is mostly cosmetic here

Reading customer LEFT JOIN orders instead would, before the WHERE, also keep Jills/Evas with zero orders (as NULL-order rows) — but the same right-column filter strips those too. With this filter the result is identical either way. Still, choose the driving table on purpose: drive from orders if the unit of the answer is an order; drive from customer if you need every selected customer to appear even with no orders.

The duplicate-rows / fan-out reminder

Each order matches exactly one customer on cust_id = id, so no row multiplies. But if customer had two rows with the same id (a dirty table), every order would duplicate per match — a fan-out. Always know whether the join key is unique on the side you join to; here id is the primary key, so it is safe.

NULL handling and ordering stability

If cust_id is NULL on an order, the join finds nothing and the row is dropped by the name filter — consistent with the inner-join behaviour above. And order by c.id alone is not a total order: the two Jill-Brown orders share id 1, so their relative order is arbitrary. Add o.order_date as a tiebreaker for deterministic output.

Senior signal: spot that a WHERE on a right-table column silently turns a LEFT JOIN into an INNER JOIN, point out that filtering on first_name matches two different Jills so a key should be used for a specific person, and add a tiebreaker so the sort is deterministic.
