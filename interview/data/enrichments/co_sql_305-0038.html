Salesforce's workforce-management team needs to catch scheduling errors before payroll runs — the same employee booked into two shifts that overlap in time. The reference solution self-joins the shift table and applies the classic interval-overlap test. The logic is sound; the one defect is cosmetic but real, and an interviewer reviewing the output will spot it instantly.

Employee 100's shifts 1 and 2 overlap by an hour (11:00–12:00); shift 3 sits cleanly apart. Employee 200's shifts 4 and 5 overlap. Employee 300 has a single shift — the control row that must produce no match.

2. The SQL — self-join with the overlap predicate

SELECT a.shift_id AS shift_a, b.shift_id AS shift_b
FROM shifts a JOIN shifts b
  ON a.employee_id = b.employee_id AND a.shift_id < b.shift_id
 AND a.start_time < b.end_time AND b.start_time < a.end_time;

Each shift a is paired with another shift b of the same employee. a.shift_id < b.shift_id keeps each pair once and removes self-pairs; the last two conditions are the interval-overlap test. (The reference answer selected a.shift_id, b.shift_id with no aliases — see the explanations for why that needs fixing.)

4. Strategic explanations

The defect: two output columns named shift_id

The reference query selects a.shift_id, b.shift_id — producing two result columns with the identical name shift_id. Postgres permits this, but it is a genuine bug for any consumer: an application that reads rows into a keyed object (a JSON map, a Python dict) sees only one shift_id — the second silently overwrites the first. The conflict pair becomes unreadable. The fix is one word per column: a.shift_id AS shift_a, b.shift_id AS shift_b. Distinct aliases on a self-join are not style; they are correctness.

The overlap predicate — why a.start < b.end AND b.start < a.end

Two intervals overlap when each starts before the other ends. It is easier to derive from the negation: they don't overlap only if one finishes before the other begins — a.end <= b.start OR b.end <= a.start. Invert that (De Morgan) and you get exactly a.start < b.end AND b.start < a.end. Memorising the positive form without the derivation is the junior tell; deriving it on the spot is the senior one.

The boundary: do touching shifts count?

Strict < means a shift ending at 12:00 and another starting at exactly 12:00 do not conflict — back-to-back shifts are allowed. If the rule is "even touching is a violation", switch to <=: a.start_time <= b.end_time AND b.start_time <= a.end_time. This is a real policy question — confirm whether adjacency is a conflict before picking the operator.

Why a.shift_id < b.shift_id, not <>

Two jobs are done by one condition. It removes the trivial self-pair (a shift never conflicts with itself), and it de-duplicates: without it, an overlap of shifts 1 and 2 would show up twice — as (1,2) and (2,1). Using <> instead of < would let both orderings through. The strict inequality gives one canonical row per conflicting pair.

Edge cases: NULL end times and three-way pile-ups

An open-ended shift with a NULL end_time makes a.start_time < b.end_time evaluate to NULL — not true — so a genuine open shift conflict is silently missed. Treat an ongoing shift as ending "now": COALESCE(b.end_time, CURRENT_TIMESTAMP). And if an employee has three mutually overlapping shifts, this query reports three pairs (A-B, A-C, B-C) rather than one cluster — fine for "flag every conflict", but if the question is "how many employees have any conflict", wrap it: SELECT COUNT(DISTINCT employee_id) over the conflicting shifts.

Senior signal: alias both sides of a self-join so the output never has two identically-named columns, derive the interval-overlap predicate from its negation rather than reciting it, justify shift_id < shift_id as the combined de-dupe-and-self-exclude trick, and handle NULL end-times with COALESCE so open shifts are not silently exempt from conflict detection.
