A CDC sink table is the landing zone for Debezium change events before they are merged into a silver-layer table. The interviewer is probing whether you understand the Debezium event envelope (before/after images, op code, LSN), how to design an idempotent primary key that allows event replay without duplicates, how to store schema-flexible payloads (JSONB for the before/after rows), and how to index the primary_key column for efficient MERGE lookups. This pattern is the backbone of CDC-to-lakehouse pipelines.

1. The requirements — what the design must capture

  • Source coordinates: source_db, source_table, and source_lsn (or binlog position) pinpoint an exact position in the source changelog — and therefore uniquely identify one change event.
  • Operation type: op distinguishes creates, updates, deletes, and initial-snapshot reads; the MERGE step applies different logic per op code.
  • Before and after images: Debezium provides both the pre-change and post-change row; the before image is required for correct UPDATE processing and is essential for DELETE (it is the only data about the deleted row).
  • Idempotency on replay: Kafka exactly-once semantics are hard; the PK on (source_db, source_table, source_lsn) — the changelog position uniquely identifies one change event — ensures that replaying the same event twice is a no-op.
  • Schema versioning: Debezium event schemas evolve; schema_version records the Debezium schema registry ID, enabling version-aware MERGE logic.
  • GIN index on primary_key JSONB: the MERGE step looks up staging rows by the source row's primary key; a GIN index on the JSONB column allows containment queries.

2. The DDL — the CREATE TABLE

CREATE TABLE staging_orders_cdc (
  source_db          VARCHAR(64)  NOT NULL,
  source_table       VARCHAR(64)  NOT NULL,
  source_lsn         NUMERIC(20)  NOT NULL,                  -- e.g. Postgres LSN or MySQL binlog file+pos as integer
  source_txid        BIGINT,
  op                 CHAR(1)      NOT NULL,                  -- c = create, u = update, d = delete, r = read (initial snapshot)
  source_ts_ms       BIGINT       NOT NULL,                  -- event time at source
  ingest_ts          TIMESTAMPTZ  NOT NULL DEFAULT now(),
  primary_key        JSONB        NOT NULL,                  -- e.g. {"order_id": 1234}
  before_image       JSONB,                                  -- null for c, present for u/d
  after_image        JSONB,                                  -- null for d, present for c/u
  schema_version     INT          NOT NULL,                  -- Debezium schema id
  PRIMARY KEY (source_db, source_table, source_lsn)
);

CREATE INDEX ix_staging_orders_cdc_pk
  ON staging_orders_cdc USING GIN (primary_key);

4. Strategic explanations

Why source_lsn is NUMERIC(20)

A Postgres LSN is a 64-bit integer (two 32-bit parts). MySQL's binlog position is a (filename, offset) pair that is commonly serialized as a large integer for comparison purposes. NUMERIC(20) accommodates both representations without loss, while also preserving natural ordering — higher LSN = later in the log. Storing it as a string would break ORDER BY source_lsn comparisons.

JSONB for primary_key — and why it stays out of the PK

Using JSONB for the source row's primary key means one staging schema handles tables with simple integer PKs, composite PKs, and UUID PKs without alteration. It cannot, however, sit in the table's PRIMARY KEY: a PK is backed by a btree index and JSONB has no default btree operator class, so PRIMARY KEY (…, primary_key) fails at CREATE TABLE. It does not need to — (source_db, source_table, source_lsn) already identifies one change event uniquely (the LSN is a position in the source changelog), so that triple is the PK and delivers idempotent replay: re-inserting the same event is a no-op under INSERT ... ON CONFLICT DO NOTHING. The GIN index on the JSONB primary_key column is what serves the MERGE step, allowing WHERE primary_key @> '{"order_id": 1234}' containment lookups.

Before/after image nullability rules

The Debezium envelope has clear rules: creates have only an after image; deletes have only a before image; updates have both; initial snapshots (op 'r') have only an after image. A CHECK constraint can enforce this: CHECK (NOT (op = 'c' AND before_image IS NOT NULL)) and similarly for deletes. These constraints catch malformed events at ingest time rather than propagating them silently to the silver layer.

Silver-layer MERGE pattern

The MERGE step reads all unprocessed staging rows, orders them by source_lsn ASC, and applies each: UPSERT the after_image for 'c' and 'u' ops; DELETE the row identified by primary_key for 'd' ops. After processing, staging rows can be archived or truncated. Ordering by LSN is critical — an out-of-order MERGE could apply a DELETE before the UPDATE it logically follows.

Dialect notes

This design is Postgres-specific (JSONB, GIN indexes). In Snowflake the before/after images would be VARIANT columns and the GIN index is replaced by a FLATTEN-based query. In Delta Lake / Iceberg, the staging table is typically a raw Parquet landing zone and the MERGE is expressed as a Delta MERGE INTO statement triggered by a Spark job. The before/after image and op-code columns are universal across all CDC sink patterns.

Senior signal: make the PK (source_db, source_table, source_lsn) — the changelog position uniquely identifies each event, so replay is idempotent; keep primary_key as JSONB (GIN-indexed) for MERGE lookups but out of the PK, since JSONB has no btree operator class; store both before and after images; and order MERGE processing by source_lsn ASC — out-of-order application is a silent data-correctness bug.
