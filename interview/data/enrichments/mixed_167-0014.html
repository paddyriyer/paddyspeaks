A product manager wants to spot the premium half of the catalogue: "which SKUs are priced above the average product price?" The reference solution is a textbook scalar subquery in the WHERE clause — compute one number (the catalogue average), then compare every row against it. The interviewer is checking that you understand when the subquery runs and how it behaves when prices are NULL.

1. The source data — products

product_idnamecategoryskumanufacturerprice
1Widget AtoolsSKU1Acme$10.00
2Widget BtoolsSKU2Acme$20.00
3GizmotoolsSKU3Globex$50.00
4GadgetelectronicsSKU4Globex$30.00
5DoohickeyelectronicsSKU5Initech$5.00
6ThingyelectronicsSKU6NULLNULL

Product 6 has a NULL price — the edge case. It must neither inflate the average nor sneak into the result.

2. The SQL — scalar subquery in the WHERE clause

SELECT * FROM products
WHERE price > (SELECT AVG(price) FROM products);

3. The resulting analysis

product_idnamecategoryskumanufacturerpricenote
3GizmotoolsSKU3Globex$50.00well above avg
4GadgetelectronicsSKU4Globex$30.00just above avg

The average over the five non-NULL prices is (10+20+50+30+5)/5 = $23.00. Only Gizmo and Gadget clear that bar. Product 6 is absent — a NULL price fails price > 23 (the comparison is unknown, not true).

4. Strategic explanations

The subquery is uncorrelated — it runs once

(SELECT AVG(price) FROM products) references nothing from the outer query, so the planner evaluates it a single time, caches the scalar, and reuses it for every row comparison. This is an uncorrelated scalar subquery — cheap and predictable. Contrast with a correlated subquery (one that mentions the outer table), which conceptually re-runs per row. Saying "this runs once, not per row" is the senior tell.

NULL handling is silent but correct here

Two NULL interactions matter. First, AVG ignores NULL prices entirely — the denominator is 5, not 6 — so a missing price does not drag the average toward zero. Second, NULL > 23 evaluates to unknown, and WHERE keeps only rows that are strictly true, so the NULL-priced row is correctly excluded. The query happens to do the right thing, but you should be able to articulate why rather than assume it.

SELECT * is fine for a demo, risky in production

The solution returns every column. In an interview that is acceptable, but flag the production concern: SELECT * couples the query to table shape, breaks if columns are renamed, and ships unneeded bytes. Name the columns the consumer actually needs.

The strict-vs-inclusive boundary

price > AVG excludes products priced exactly at the average. If a product sat at precisely $23.00, it would drop out. Confirm with the interviewer whether "above average" means strictly greater (>) or at-least-average (>=) — boundary semantics are a classic place to lose a point.

The window-function alternative — average plus delta in one pass

If the ask grows to "show each product's price and how far it sits from the average," a window function avoids re-scanning:

SELECT *, price - AVG(price) OVER () AS delta_from_avg
FROM products
WHERE price > AVG(price) OVER ();   -- needs a subquery/CTE wrapper: window fns can't sit in WHERE

Window functions are evaluated after WHERE, so the filter form must wrap the window result in a CTE. The simple scalar subquery stays the cleanest answer for the literal question asked.

Senior signal: call the construct an uncorrelated scalar subquery that evaluates once, explain that AVG skips NULLs while NULL > x is unknown (so the NULL row is correctly absent on both counts), and clarify the strict-vs-inclusive boundary before committing to >.
