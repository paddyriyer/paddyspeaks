An operations dashboard wants one row per region with revenue broken out into paid, pending, and refunded columns plus a total. This is a pivot — rotating distinct values of status into column headers. The interviewer is checking whether you reach for the modern FILTER clause on aggregates, and whether you correctly turn a region's missing status into a 0 rather than a NULL.

1. The source data — orders

The edge cases that decide the interview: NA has no refunded order and EU has no pending order — those cells must read 0, not blank. And APAC has two paid orders that must sum to 400 in a single cell.

2. The SQL — pivot via FILTER-clause aggregates

SELECT region,
       COALESCE(SUM(amount) FILTER (WHERE status = 'paid'), 0) AS paid,
       COALESCE(SUM(amount) FILTER (WHERE status = 'pending'), 0) AS pending,
       COALESCE(SUM(amount) FILTER (WHERE status = 'refunded'), 0) AS refunded,
       SUM(amount) AS total
FROM orders
GROUP BY region
ORDER BY region;

SUM(amount) FILTER (WHERE status = 'paid') sums amount over only the rows in that group where the status matches — one filtered aggregate per output column. GROUP BY region collapses each region to one row. COALESCE(..., 0) converts the NULL that an empty filter produces into 0.

4. Strategic explanations

The FILTER clause vs SUM(CASE WHEN ...)

The classic portable pivot is SUM(CASE WHEN status = 'paid' THEN amount END). The FILTER clause — SUM(amount) FILTER (WHERE status = 'paid') — expresses the identical idea more declaratively: the predicate decides which rows the aggregate sees, instead of rewriting non-matching rows to NULL. They produce the same result. FILTER is SQL-standard and reads better; the CASE form is the fallback for engines that lack it. Both still require you to enumerate the statuses by hand.

Why COALESCE is needed — empty filter yields NULL

When a region has no order of a given status, that filtered SUM aggregates over zero rows. SUM of an empty set is NULL, not 0 — a deliberate SQL rule (you summed nothing, so there is no number). The question demands 0 for missing combinations, so each filtered column is wrapped in COALESCE(..., 0). Omit it and EU's pending would be NULL, which then poisons any downstream arithmetic. Note total is not wrapped: every region has at least one order, so the unfiltered SUM is never NULL here.

Why total is the unfiltered SUM, not paid+pending+refunded

total is computed as a plain SUM(amount) over the whole group. That is robust: if a future order arrived with a status outside the three named ones (say 'cancelled'), it would still count toward total but appear in no status column — and the three columns would no longer add up to total. Adding the three columns instead would silently hide such rows. Computing total independently is a deliberate integrity check; the mismatch, if any, is a signal worth surfacing.

The static-column limitation

The three statuses are hard-coded. A new status value lands in no column and is invisible except in total. Pure SQL cannot grow columns from data — a truly data-driven pivot needs dynamic SQL (build the column list from SELECT DISTINCT status, then execute the generated text) or PostgreSQL's crosstab() from the tablefunc extension. Naming this limit unprompted is the senior signal.

Dialect notes

The FILTER clause is supported by PostgreSQL and SQLite (3.30+). MySQL and SQL Server do not have it — there you write SUM(CASE WHEN status = 'paid' THEN amount ELSE 0 END), and the ELSE 0 makes the per-column COALESCE unnecessary because the aggregate then always sees a number. Snowflake, Oracle, and SQL Server additionally offer a dedicated PIVOT operator that is terser still but equally requires an explicit value list.

Senior signal: pivot status into columns with one SUM(amount) FILTER (WHERE ...) per status, wrap each in COALESCE(...,0) because an empty filtered aggregate returns NULL, compute total as an independent unfiltered SUM so stray statuses are not hidden, and note the column list is static unless you use dynamic SQL or crosstab().
