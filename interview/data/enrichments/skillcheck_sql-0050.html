This is a classic from Danny Ma's 8 Week SQL Challenge ("Pizza Runner"): does a bigger order take longer to prepare? customer_orders holds one row per pizza and runner_orders holds the delivery record. The metric is average prep time bucketed by order size. The interviewer is probing two things at once — aggregating a one-to-many join without fan-out, and excluding cancelled orders cleanly.

Two edge cases. Order 3 lists pizza_id = 1 twice — the spec defines size as the count of pizza rows, so order 3 is size 3 (duplicates count). Order 6 has a non-NULL cancellation and a NULL pickup_time — it must be filtered out before any time arithmetic.

2. The SQL — aggregate per order, then average by size

WITH per_order AS (
  SELECT co.order_id,
         COUNT(*) AS pizzas_in_order,
         EXTRACT(EPOCH FROM (ro.pickup_time - MIN(co.order_time)))/60.0 AS prep_minutes
  FROM customer_orders co
  JOIN runner_orders ro USING (order_id)
  WHERE ro.cancellation IS NULL OR ro.cancellation = ''
  GROUP BY co.order_id, ro.pickup_time
)
SELECT pizzas_in_order,
       ROUND(AVG(prep_minutes)::numeric, 2) AS avg_prep_minutes
FROM per_order
GROUP BY pizzas_in_order
ORDER BY pizzas_in_order;

The CTE collapses each order to one row: COUNT(*) is the pizza-row count, and prep time is pickup_time minus the order's MIN(order_time), converted to minutes. The outer query then groups those per-order rows by size and averages.

Prep time rises monotonically with order size — 9, 22.5, 35 minutes — the answer to "do bigger orders take longer". Order 6 contributes nothing: the cancellation filter removed it. Order 3's size is 3 because its duplicate pizza row was counted, exactly as the spec's "count of pizza rows" wording requires.

4. Strategic explanations

Why aggregate inside a CTE first — two grain levels

The question mixes two grains: per-order facts (size, prep time) and a per-size average. You cannot compute both in one GROUP BY. The CTE settles the order grain — one row per order — and the outer query rolls those up to the size grain. Trying to average prep time directly off the joined pizza rows would weight each order by its pizza count, skewing the mean.

MIN(order_time) — collapsing repeated order_time

Every pizza row of an order carries the same order_time, but it is not in the GROUP BY at the order grain, so it needs an aggregate. MIN(co.order_time) picks that single shared value — MAX or any_value would do equally since the values are identical. Prep time is then one well-defined interval per order: pickup_time - order_time.

The cancellation filter — NULL and empty string

The spec says exclude rows where cancellation is "non-NULL and non-empty". The dataset stores completed orders as cancellation = NULL, but real Pizza Runner data also has '' (empty string) for non-cancellations. WHERE ro.cancellation IS NULL OR ro.cancellation = '' keeps both as valid deliveries and drops only genuine cancellation text like 'Customer Cancellation'. Handling NULL and empty is the careful read of the requirement.

Why pickup_time is in the GROUP BY

ro.pickup_time appears in the CTE's SELECT (inside EXTRACT) but is a column of the joined runner_orders row, not an aggregate — so it must be in the GROUP BY. Since runner_orders has one row per order, grouping by (order_id, pickup_time) is the same grain as grouping by order_id alone; the extra key is just SQL bookkeeping. Alternatively MIN(ro.pickup_time) would let you drop it from the GROUP BY.

Dialect notes — interval-to-minutes is engine-specific

EXTRACT(EPOCH FROM (a - b))/60.0 is the Postgres idiom (timestamp subtraction yields an interval; EPOCH gives seconds). SQL Server uses DATEDIFF(MINUTE, b, a) (integer minutes, no fractions); BigQuery uses TIMESTAMP_DIFF(a, b, MINUTE); MySQL uses TIMESTAMPDIFF(MINUTE, b, a). The aggregate-then-average structure is portable; only the duration expression needs translating per engine.

Senior signal: when a query mixes per-order and per-size grains, settle the lower grain in a CTE before rolling up — averaging off the raw one-to-many join would weight orders by pizza count — and write the cancellation filter to catch both NULL and empty-string "not cancelled" values.
