A revenue dashboard needs two numbers side by side: total billings across all orders, and the portion that has actually been paid. The interviewer is probing whether you can compute a filtered aggregate and an unfiltered one in a single pass — using SUM(CASE ...) — instead of running two queries or self-joining the table to itself.

1. The source data — orders

order_idstatusamount(narrative)
1paid100counts toward both totals
2pending50in total, not in paid
3paid75counts toward both totals
4cancelled200in total, not in paid — a large non-paid order
5paid25counts toward both totals

The mix of statuses is the point: three paid, one pending, one cancelled. The cancelled order at 200 is a deliberate stressor — it is the single biggest amount, so any query that accidentally folds it into paid revenue would be off by a very visible 200.

2. The SQL — conditional aggregation in one pass

SELECT
  SUM(amount) AS total_revenue,
  SUM(CASE WHEN status = 'paid' THEN amount ELSE 0 END) AS paid_revenue
FROM orders;

SUM(amount) adds every row's amount — the unconditional total. The second column wraps each amount in a CASE: an order's amount passes through only when its status is 'paid', otherwise the CASE yields 0. SUM then adds those filtered contributions. One scan of the table produces both figures.

3. The resulting analysis

total_revenuepaid_revenuesignal
450200all five (100+50+75+200+25) vs only the three paid (100+75+25)

total_revenue = 450 includes the pending 50 and the cancelled 200. paid_revenue = 200 is just the three paid orders; the cancelled 200 and pending 50 each hit the ELSE 0 branch and contribute nothing. The single output row carries both metrics — no join, no second query.

4. Strategic explanations

Why one pass beats two queries

The instinct is to run SELECT SUM(amount) FROM orders and separately SELECT SUM(amount) FROM orders WHERE status = 'paid'. That reads the table twice and forces the caller to stitch the results together. Conditional aggregation computes both during a single scan — faster, atomic (both numbers reflect the same snapshot), and trivially extensible: add a column per status without re-reading anything.

SUM(CASE ...) vs COUNT vs FILTER

The CASE idiom generalises. SUM(CASE WHEN cond THEN amount ELSE 0 END) gives a conditional total; SUM(CASE WHEN cond THEN 1 ELSE 0 END) — or COUNT(CASE WHEN cond THEN 1 END) — gives a conditional count. PostgreSQL also offers cleaner syntax: SUM(amount) FILTER (WHERE status = 'paid') expresses the exact same thing more readably. The FILTER clause is ANSI SQL but not universal — SQL Server and MySQL lack it, so SUM(CASE) remains the portable form.

ELSE 0 vs omitting it — a NULL nuance

With ELSE 0, non-paid rows contribute a literal zero. Drop the ELSE and the CASE returns NULL for non-paid rows — but SUM skips NULLs, so the paid total is identical either way. The visible difference appears only in the all-non-matching case: if no order were paid, SUM of all-NULL is NULL, whereas ELSE 0 gives 0. For a revenue figure, 0 is the honest answer, so keeping ELSE 0 (or wrapping in COALESCE) is the safer habit.

Edge cases: NULL status, NULL amount, and an empty table

A row with status = NULL fails status = 'paid' (the comparison is unknown, not true), so it correctly lands in ELSE 0 — but it still counts in total_revenue, which is usually intended. A NULL amount is ignored by both SUMs. And on an empty table both columns return NULL, not 0 — if the dashboard must show zeros, wrap each with COALESCE(..., 0). Naming these three is the senior touch.

Senior signal: compute filtered and unfiltered aggregates in one scan with SUM(CASE WHEN ... THEN amount ELSE 0 END) rather than two queries, know the cleaner Postgres FILTER (WHERE ...) equivalent and that it is not portable, and wrap totals in COALESCE if an empty table must report 0 instead of NULL.
