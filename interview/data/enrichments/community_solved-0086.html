A fraud / risk team wants customers who fired off more than three transactions inside any one-hour window — a burst pattern that often signals card testing or account abuse. The interviewer is probing a sliding-window-count technique, and is watching closely for the off-by-one error that turns "more than 3" into "3 or more".

Customer 300 has exactly three transactions in an hour — the deliberate boundary. "More than 3" means four or more, so 300 must not appear; only customer 100 should.

2. The SQL — pair transactions, count per anchor

Original solution (flagged — wrong threshold):

WITH frequent_txn_info AS (
  SELECT ct1.customer_id,
         ct1.transaction_id AS transaction_1,
         ct2.transaction_id AS transaction_2,
         ct1.transaction_time AS transaction_ts1,
         ct2.transaction_time AS transaction_ts2,
         EXTRACT(EPOCH FROM (ct2.transaction_time - ct1.transaction_time))/60 AS interval_min
  FROM customer_transactions ct1
  INNER JOIN customer_transactions ct2
    ON ct1.customer_id = ct2.customer_id
   AND ct1.transaction_time <= ct2.transaction_time
   AND ct1.transaction_id <> ct2.transaction_id
),
frequent_txn AS (
  SELECT customer_id,
         COUNT(*) OVER (PARTITION BY customer_id, transaction_1) AS freq_txn
  FROM frequent_txn_info
  WHERE interval_min <= 60
)
SELECT DISTINCT customer_id
FROM frequent_txn
WHERE freq_txn >= 2;

The self-join pairs each transaction (ct1, the anchor) with every later transaction within 60 minutes. COUNT(*) OVER (PARTITION BY customer_id, transaction_1) counts those later partners per anchor. Because the join's ct1.transaction_id <> ct2.transaction_id clause excludes the anchor from its own count, an anchor with N partners represents a burst of N + 1 transactions. So freq_txn >= 2 admits bursts of only 3 — it answers "3 or more", not "more than 3". Customer 300 wrongly slips through.

Corrected solution — threshold raised to >= 3:

-- ...frequent_txn_info and frequent_txn CTEs unchanged...
SELECT DISTINCT customer_id
FROM frequent_txn
WHERE freq_txn >= 3;   -- 3 later partners + the anchor = 4+ transactions

4. Strategic explanations

The bug — counting partners, not transactions

The mental model must be precise. freq_txn is the number of transactions that occur at or after the anchor and within 60 minutes, excluding the anchor itself. A burst of K transactions, anchored on its earliest member, has K−1 such partners. "More than 3 transactions" means K ≥ 4, which means K−1 ≥ 3 — hence freq_txn >= 3. The original >= 2 corresponds to K ≥ 3, the "at least 3" question. A single misread of strict-vs-inclusive bound flips the answer.

Why anchor on the earliest transaction of a window

The join condition ct1.transaction_time <= ct2.transaction_time only pairs an anchor with transactions that come after it. So a window is correctly measured from its first transaction forward — the anchor sees the whole hour ahead of it. If any transaction (not just the earliest) could anchor a backward-and-forward window you would double-count; pinning the window's left edge to the anchor keeps each burst counted from one well-defined origin.

The self-join is O(n²) — the modern alternative

Pairing every transaction with every other transaction of the same customer is quadratic in transactions-per-customer. For a high-volume customer that is expensive. The cleaner, near-linear approach is a range-framed window count: COUNT(*) OVER (PARTITION BY customer_id ORDER BY transaction_time RANGE BETWEEN CURRENT ROW AND INTERVAL '1 hour' FOLLOWING) gives, per row, how many transactions fall in the next hour — then keep customers where that count ever reaches 4. One sort, one pass, no self-join.

Boundary semantics — <= 60 and inclusive windows

interval_min <= 60 makes the one-hour window inclusive of its endpoint: two transactions exactly 60 minutes apart count as "within the window". Whether the window is open or closed at 60 minutes is a business decision; state it. Also note EXTRACT(EPOCH FROM interval)/60 yields fractional minutes — fine for a <= comparison, but do not assume it is an integer.

What to say next — ties and the meaning of "any window"

If two transactions share an identical transaction_time, the <= join still pairs them (the <> on id keeps it from self-pairing), so simultaneous transactions are handled. The deeper senior point: "any 1-hour window" is a continuous sliding window, and anchoring on actual transaction timestamps is correct precisely because a burst's window can always be slid to start exactly on one of its transactions — you never need to test arbitrary clock times. Saying that shows you understand why the discrete anchor set suffices.

Senior signal: nail the off-by-one — an anchor's partner count excludes the anchor, so "more than 3 transactions" is freq_txn >= 3, not >= 2. Then flag the O(n²) self-join and offer the RANGE ... INTERVAL '1 hour' FOLLOWING window count as the linear replacement.
