A Community member poses a deceptively small puzzle: print all 26 letters of the English alphabet, A through Z, using only SQL — no source table at all. It is a classic recursive-CTE warm-up, and the interviewer uses it to see whether you can generate rows from nothing and, just as importantly, whether you place a termination condition that actually fires.

1. The source data — (none)

conceptvalue
ASCII of 'A'65
ASCII of 'Z'90
letters to emit90 − 65 + 1 = 26
recursion stepnext letter = char(ascii(letter) + 1)

There is no table to query — the rows are manufactured. The whole problem is a number line from 65 to 90, walked one ASCII code at a time and rendered back into characters. The interesting edge is the stopping point: the recursion must halt exactly at 'Z', not one past it.

2. The SQL — recursive CTE over ASCII codes

WITH alphabet AS (
  SELECT char(ascii('A')) AS letter
  UNION ALL
  SELECT char(ascii(letter) + 1) AS letter
  FROM alphabet
  WHERE letter <> 'Z'
)
SELECT * FROM alphabet;

The anchor seeds 'A'. The recursive member takes the last letter, advances its ASCII code by one, converts back to a character, and the WHERE letter <> 'Z' guard stops the walk once 'Z' has been produced.

26 rows, A through Z in order. The guard WHERE letter <> 'Z' is checked on each input row before the next is produced: once the row holding 'Z' is fed in, the predicate is false and no 91st character ('[') is generated. 'Z' itself is emitted because it was produced from 'Y', whose predicate was still true.

4. Strategic explanations

The dialect note — char() is not Postgres

The reference uses char(), which is the SQL Server / MySQL name for "ASCII code to character". Postgres spells it chr(), and standard SQL needs the RECURSIVE keyword:

WITH RECURSIVE alphabet AS (
  SELECT chr(ascii('A')) AS letter
  UNION ALL
  SELECT chr(ascii(letter) + 1)
  FROM alphabet WHERE letter <> 'Z'
)
SELECT * FROM alphabet;

The logic is identical; only char→chr and the RECURSIVE marker differ. Calling out that the function name is dialect-specific — rather than assuming one engine — is the senior tell.

The infinite-recursion trap

The termination predicate is the entire risk. WHERE letter <> 'Z' works because the sequence lands exactly on 'Z'. But guard on the wrong condition — say WHERE letter < 'Z' when you meant to include Z, or omit the WHERE entirely — and the CTE walks past 'Z' forever, hitting the engine's recursion-depth cap (Postgres errors; SQL Server stops at 100 by default). Always prove the guard is reached by an exact value, not approached.

The set-based alternative — generate_series

Recursion is the textbook answer, but a sequence generator is shorter and clearer where the engine has one:

-- Postgres
SELECT chr(g) AS letter
FROM generate_series(ascii('A'), ascii('Z')) AS g;

One declarative line, no anchor, no termination predicate to get wrong. SQL Server reaches the same goal via master..spt_values or a numbers table. Recursion is great practice; for "generate N rows" a series function is usually the production choice.

Case, collation, and the "next character" assumption

ascii(letter) + 1 assumes A–Z are 26 contiguous code points — true for ASCII and UTF-8. It is not portable to arbitrary alphabets: in Unicode, accented or non-Latin letters are not one step apart. Also, 'Z' (90) and 'z' (122) are different code points; this query emits uppercase only. For lowercase, seed 'a' and stop at 'z'. And beware case-insensitive collations — under one, 'Z' <> 'z' could behave unexpectedly in the guard.

Why this pattern matters beyond the toy

"Generate a contiguous sequence" is the backbone of real work: a date spine to fill calendar gaps, an integer range to left-join against for missing-row detection, buckets for a histogram. The alphabet is just the friendliest instance — master the recursive-CTE / generate_series pair here and the date-gap problems become routine.

Senior signal: generate rows from nothing with a recursive CTE whose termination predicate lands on an exact value (never an open inequality that risks infinite recursion), note that char vs chr is dialect-specific, and recognise that generate_series is the cleaner production tool for any contiguous sequence.
