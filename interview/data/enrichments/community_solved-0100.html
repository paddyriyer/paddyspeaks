An e-commerce company tracks product page views and wants the customers who viewed the same product on more than three consecutive days — a strong purchase-intent signal. The reference solution takes a fixed-window LAG approach: look back three rows and check every step is one day apart. The interviewer is probing whether you can detect a run of length 4+ and whether you see the limits of a hardcoded-window method.

2. The SQL — three-deep LAG window

WITH t1 AS (
  SELECT viewid, customerid, productid, viewdate,
    LAG(viewdate, 1) OVER (PARTITION BY customerid, productid ORDER BY viewdate ASC) AS previousday1,
    LAG(viewdate, 2) OVER (PARTITION BY customerid, productid ORDER BY viewdate ASC) AS previousday2,
    LAG(viewdate, 3) OVER (PARTITION BY customerid, productid ORDER BY viewdate ASC) AS previousday3
  FROM productviews
), diffs AS (
  SELECT *,
    viewdate::date     - previousday1::date AS d1,
    previousday1::date - previousday2::date AS d2,
    previousday2::date - previousday3::date AS d3
  FROM t1
)
SELECT DISTINCT customerid, productid
FROM diffs
WHERE d1 = 1 AND d2 = 1 AND d3 = 1;

(The original regex-guarded date casts are simplified here for clarity — with a typed DATE column the casts are no-ops and the run executes identically.)

4. Strategic explanations

How the fixed-window check works

Three LAGs, partitioned by (customerid, productid) and ordered by date, expose the current view date and its three predecessors on one row. If all three step-differences (d1, d2, d3) equal 1, that row is the 4th day of an unbroken run — meaning the customer viewed the product on four consecutive days. DISTINCT collapses the possibly-many qualifying rows of one run to a single pair.

"More than three" means a window of exactly three back-steps

Four consecutive days produce three one-day gaps. So "more than 3 consecutive days" maps to "3 prior days all one step apart" — the LAG(.., 3) depth. A common off-by-one is checking only d1 and d2 (a 3-day run) or going to LAG(.., 4) (a 5-day run). Re-derive the depth from the boundary: N consecutive days ⇒ N−1 back-steps.

The NULL behaviour at the start of each partition handles itself

For the first three rows of any (customer, product) partition, at least one LAG returns NULL, so the corresponding difference is NULL, and d1 = 1 AND d2 = 1 AND d3 = 1 evaluates to unknown — the row is excluded. That is correct: a run cannot be confirmed until a 4th day exists. No explicit NULL guard is needed.

The real weakness — the window is hardcoded

This solution answers "exactly > 3" but does not generalise. If the threshold becomes "> 7 consecutive days" you must bolt on four more LAGs and four more comparisons. The scalable answer is gaps-and-islands: assign ROW_NUMBER() OVER (PARTITION BY customerid, productid ORDER BY viewdate), compute viewdate - row_number * INTERVAL '1 day' as an island key (constant within a run), then GROUP BY customerid, productid, island_key HAVING COUNT(*) > 3. One query, any threshold. Naming this as the better approach is the senior move.

The duplicate-date trap

If a customer views the same product twice on one day, a step difference becomes 0, silently breaking the run — or, in the gaps-and-islands variant, ROW_NUMBER splits the island. Deduplicate first with SELECT DISTINCT customerid, productid, viewdate (or use DENSE_RANK). View tables almost always contain same-day repeats, so this caveat is not hypothetical.

Senior signal: derive the window depth from the boundary (4+ days ⇒ 3 one-day back-steps), note the leading-NULL rows exclude themselves correctly, then call out that the hardcoded triple-LAG does not scale — the gaps-and-islands form (viewdate - ROW_NUMBER() island key, HAVING COUNT(*) > 3) handles any threshold — and deduplicate same-day views first.
