Spotify wants a play counter that ticks up: for every listen, how many times has this user played this song so far, up to and including now. It powers "you've played this 50 times" badges and the obsession-tracking in Wrapped. The query is a running COUNT over a window — and the interviewer is checking whether you partition on the right keys and order on time.

1. The source data — plays

User 1 plays song 10 three times and song 20 once; user 2 plays each once. Two different users both play song 10 — the partition must keep those counters separate, or user 2's first play would read "4".

2. The SQL — running COUNT(*) OVER

SELECT user_id, song_id, played_at,
  COUNT(*) OVER (PARTITION BY user_id, song_id ORDER BY played_at) AS cumulative_plays
FROM plays;

PARTITION BY user_id, song_id gives each (user, song) pair its own counter. ORDER BY played_at turns COUNT(*) into a running count: with an ORDER BY present, the implicit frame is RANGE UNBOUNDED PRECEDING TO CURRENT ROW, so each row counts itself and every earlier row in its partition.

User 1's song 10 counter climbs 1 → 2 → 3 across the three plays. Their song 20 restarts at 1 — different partition. User 2's song 10 also reads 1, not 4: the two-column partition kept the two users' histories apart. Each row reports the count as of that play, which is exactly the badge number.

4. Strategic explanations

Why the implicit frame matters — the silent RANGE

With an ORDER BY in the window and no explicit frame, Postgres applies RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. That is what makes COUNT(*) cumulative rather than partition-wide. Drop the ORDER BY and the frame becomes the whole partition — every row would show the total play count (3 for user 1 / song 10), not the running one. The ORDER BY is not cosmetic; it changes the answer.

The duplicate-timestamp trap — RANGE vs ROWS

If a user plays the same song twice at the identical timestamp, the default RANGE frame treats them as peers and gives both rows the same cumulative count — the higher one. Two plays at 09:00 would both read 2, never 1 then 2. If you need a strict 1, 2, 3 even through ties, switch the frame to ROWS:

COUNT(*) OVER (PARTITION BY user_id, song_id
  ORDER BY played_at ROWS UNBOUNDED PRECEDING)

Or, if "Nth play" must be unique, use ROW_NUMBER() instead — it never ties, assigning 1, 2, 3 even at equal timestamps (breaking the tie arbitrarily).

COUNT(*) vs ROW_NUMBER() here

With distinct timestamps the two are interchangeable: both give 1, 2, 3. They diverge only on ties — COUNT(*) with a RANGE frame counts peers together, ROW_NUMBER() forces distinct positions. Pick based on whether two simultaneous plays should be "the 2nd play" together or "the 2nd and 3rd play" separately.

Partition keys: forgetting song_id

A common error is PARTITION BY user_id alone. That counts every play by the user across all songs — a per-user lifetime listen counter, not a per-song one. The question asks for plays of a specific song, so both user_id and song_id belong in the partition. Naming the grain explicitly ("plays of this song by this user") prevents this.

Edge cases: NULL timestamps and ordering

If played_at is NULL on some rows, ORDER BY played_at sorts NULLs last by default in a descending sort but first in this ascending one — an undated play would be counted as the earliest. Decide deliberately: filter NULL timestamps or add NULLS LAST. Also note the output here has no ORDER BY on the outer query, so row order is not guaranteed — add ORDER BY user_id, song_id, played_at if the consumer cares about presentation order.

Senior signal: partition on the full grain (user_id, song_id), know that the ORDER BY is what makes COUNT(*) cumulative via the implicit RANGE frame, and switch to ROWS or ROW_NUMBER() when duplicate timestamps would otherwise tie the running count.
