An HR system stores departments as an adjacency list — each row points to its parent. The CFO asks: "for every department, what is the total salary of that department plus all departments beneath it?" Engineering's number must include Backend and Frontend. The interviewer is probing recursive CTEs — can you walk a tree of arbitrary depth and roll a metric up it?

The structure itself is the edge case: only leaf departments have employees. Company, Engineering, and Sales employ nobody directly — their totals come entirely from descendants. A query that only sums direct employees would report 0 for them.

2. The SQL — recursive CTE building each subtree

WITH RECURSIVE subtree AS (
  SELECT id AS root_id, id AS dept_id FROM dept
  UNION ALL
  SELECT s.root_id, d.id
  FROM dept d JOIN subtree s ON d.parent_id = s.dept_id
)
SELECT d.id AS dept_id,
       d.name AS dept_name,
       COALESCE(SUM(e.salary), 0) AS total_salary
FROM dept d
LEFT JOIN subtree s ON s.root_id = d.id
LEFT JOIN employee e ON e.dept_id = s.dept_id
GROUP BY d.id, d.name
ORDER BY d.id;

The recursive CTE produces every (root_id, dept_id) pair where dept_id is the root itself or any descendant. Joining that to employee and grouping by the root collapses each whole subtree into one rolled-up salary.

4. Strategic explanations

How the recursive CTE walks the tree

A recursive CTE has two parts joined by UNION ALL. The anchor — SELECT id AS root_id, id AS dept_id FROM dept — seeds every department as the root of its own (so-far one-node) subtree. The recursive term repeatedly finds departments whose parent_id is a dept_id already in the set and adds them, carrying the original root_id forward. Iteration stops when a pass adds no new rows. The result: one row per ancestor/descendant relationship, including each node with itself.

Why carry root_id through the recursion

The trick is keeping root_id constant while dept_id descends. That is what lets the final GROUP BY root_id ask "everything under department X". Without the propagated root, the CTE would just enumerate the tree once and you could not attribute a descendant to every ancestor above it. This "carry the root" pattern is the standard idiom for subtree rollups.

The two LEFT JOINs and COALESCE

Both joins are LEFT for a reason. A leaf has a subtree of just itself; an ancestor has many descendants — both are covered. But a subtree may contain no employees at all (imagine an empty department branch): the employee join then yields all-NULL rows, SUM returns NULL, and COALESCE(..., 0) supplies the required 0. Inner joins would silently drop staff-less departments from the output entirely.

Cycle safety — the production caveat

This query assumes a true tree. If the data ever had a cycle (department A's ancestor points back to A through bad data), UNION ALL would loop forever. PostgreSQL 14+ offers UNION ALL ... CYCLE dept_id SET is_cycle USING path to detect and stop; older versions need a manual visited-path array. Mentioning that an adjacency list is only safe to recurse if cycles are impossible — or guarded — is the senior signal.

Counting rows vs summing — a duplication subtlety

Here each employee belongs to exactly one department, so joining through subtree and summing is correct. But if the metric were COUNT(DISTINCT employee) and an employee could appear under a department reachable by two paths (a DAG, not a tree), COUNT(*) would double-count. With a strict tree there is exactly one path root→descendant, so no duplication — but that guarantee is worth stating, because it is what makes the plain SUM safe.

Senior signal: roll a metric up a hierarchy with a recursive CTE that carries root_id through every descendant, join through that subtree mapping and GROUP BY the root, use LEFT JOIN + COALESCE(SUM(...), 0) so staff-less branches still appear, and flag that recursing an adjacency list is only safe when cycles are impossible or explicitly guarded.
