X's engagement team logs every interaction in one events stream and wants two headline numbers off it: how many clicks, how many views. It is a conditional count — and although the question is titled "with FILTER", the reference solution uses the SUM(CASE…) idiom. The interviewer wants to see you know both, and why both beat running two separate queries.

1. The source data — events

user_idevent_type
1click
1view
2view
2view
3click
3scroll

Six events: two clicks, three views — and one scroll. The scroll row is the deliberate edge case: it matches neither bucket and must be counted in neither, while still being a real row in the table.

2. The SQL — SUM(CASE) buckets

SELECT
  SUM(CASE WHEN event_type = 'click' THEN 1 ELSE 0 END) AS clicks,
  SUM(CASE WHEN event_type = 'view'  THEN 1 ELSE 0 END) AS views
FROM events;

Each CASE emits 1 for a matching row and 0 otherwise; SUM adds those up. No GROUP BY, so the whole table collapses to a single row with two totals computed in one pass.

3. The resulting analysis

clicksviews
23

Clicks = 2, views = 3. The scroll row contributed 0 to both sums — it is in the table but in neither bucket. Note clicks + views = 5, not 6: a conditional count does not have to cover every row, and that is correct here.

4. Strategic explanations

The FILTER clause — what the title is pointing at

Postgres has a purpose-built syntax for exactly this, and it is cleaner than CASE:

SELECT
  COUNT(*) FILTER (WHERE event_type = 'click') AS clicks,
  COUNT(*) FILTER (WHERE event_type = 'view')  AS views
FROM events;

COUNT(*) FILTER (WHERE …) counts only the rows the predicate keeps. It is standard SQL, reads as plain English, and produces the identical 2 / 3. SUM(CASE…) is the portable equivalent for engines without FILTER (MySQL, older SQL Server). Knowing both — and that they are equivalent — is the senior signal.

The SUM(CASE) vs COUNT(CASE) trap

A frequent rewrite drops the ELSE 0 and switches to COUNT:

COUNT(CASE WHEN event_type = 'click' THEN 1 END) AS clicks  -- also correct, on purpose

This works — but only because the CASE has no ELSE, so non-clicks yield NULL, and COUNT ignores NULLs. Write COUNT(CASE WHEN … THEN 1 ELSE 0 END) and it breaks: the ELSE 0 makes every row non-NULL, so COUNT returns 6 for both columns. Rule of thumb: SUM pairs with ELSE 0; COUNT pairs with no ELSE. Mixing them is a classic silent bug.

Why not two separate queries?

You could write SELECT COUNT(*) FROM events WHERE event_type='click' and another for views. That scans the table twice and returns two result sets the application must stitch together. The conditional-aggregation form scans once and returns one tidy row — faster and easier to consume. On a billion-row event stream the single-pass version is the only sensible answer.

The "counts per user" follow-up

The natural next ask is the same buckets per user — just add a GROUP BY:

SELECT user_id,
  COUNT(*) FILTER (WHERE event_type = 'click') AS clicks,
  COUNT(*) FILTER (WHERE event_type = 'view')  AS views
FROM events
GROUP BY user_id;

Now user 3 shows clicks = 1, views = 0 — and crucially still appears, because they have some event. A user with zero events of any kind is simply absent; if the report must list every user at 0/0, that needs a LEFT JOIN from a users table.

Edge cases: empty table, NULL event_type, and case sensitivity

On an empty table, SUM(CASE…) returns NULL (sum of no rows), whereas COUNT(*) FILTER(…) returns 0 — a real behavioural difference; wrap the SUM in COALESCE(…, 0) if a number is required. A NULL event_type matches neither predicate and is counted nowhere — usually fine, but worth a glance for data-quality. And 'click' vs 'Click': string equality is case-sensitive, so normalise with lower(event_type) if the stream is dirty.

Senior signal: compute both buckets in a single pass rather than running two queries, know that FILTER and SUM(CASE…) are equivalent (and which engines support which), pair SUM with ELSE 0 but COUNT with no ELSE, and flag the empty-table NULL-vs-0 difference.
