A retail analytics team wants the "completionist" customers — people who have bought every product in the catalog at least once — for a loyalty tier. This is the textbook relational division problem: divide the purchase set by the full product set. The interviewer is probing whether you can express "for all products" in SQL, which has no native universal quantifier, and whether your COUNT(DISTINCT) correctly tolerates repeat purchases.

1. The source data — product and purchase

11, 2, 3, 1all 3 products — product 1 bought twice

42, 3missing product 1 — does not qualify

The catalog (product) has exactly three items: 1, 2, 3. Customer 1 is the duplicate edge case — they bought product 1 twice, so a raw COUNT(*) of their rows would be 4, overstating their distinct coverage. Customers 2 and 4 each miss exactly one product.

2. The SQL — count distinct products vs catalog size

SELECT customer_id
FROM purchase
GROUP BY customer_id
HAVING COUNT(DISTINCT product_id) = (SELECT COUNT(*) FROM product)
ORDER BY customer_id;

Group purchases per customer; COUNT(DISTINCT product_id) counts the distinct products that customer ever bought. HAVING keeps a customer only when that distinct count equals (SELECT COUNT(*) FROM product) — the total catalog size. Equal distinct counts mean the customer's product set covers the whole catalog.

Customer 1 survives despite having four purchase rows: COUNT(DISTINCT product_id) folds the two product-1 rows into one, giving 3 — exactly the catalog size. Customers 2 and 4 have distinct counts of 2, short of 3, so HAVING drops them.

4. Strategic explanations

Why counting equals "for all" — the division trick

SQL has no FOR ALL quantifier, so relational division is done by counting. The logic: a customer's distinct products are always a subset of the catalog, so the subset equals the whole set if and only if the two have the same cardinality. COUNT(DISTINCT product_id) = catalog size is therefore an exact test for "bought every product" — provided purchase.product_id values are guaranteed to exist in the catalog. If purchase could contain a product not in product, that foreign product would inflate the count, and you would need a join to product first to filter it out.

COUNT(DISTINCT) is load-bearing — not COUNT(*)

Customer 1 bought product 1 twice; COUNT(*) would return 4 and 4 = 3 is false — the query would wrongly exclude a customer who genuinely owns every product. COUNT(DISTINCT product_id) measures coverage, not purchase volume. The question explicitly allows duplicates ("duplicates allowed"), which is the signpost that DISTINCT inside the count is mandatory, not optional polish.

The scalar subquery runs once

(SELECT COUNT(*) FROM product) is an uncorrelated scalar subquery: it depends on nothing in the outer query, so the planner evaluates it a single time and reuses the constant for every group. There is no per-customer re-execution and no performance concern — it is just a clean way to inline the catalog size. You could equally compute it in a CTE; the effect is identical.

The empty-catalog edge case

The question raises it: if product were empty, (SELECT COUNT(*) FROM product) is 0. A customer with any purchases has COUNT(DISTINCT product_id) >= 1, which never equals 0 — so this query would return nobody, contradicting the stated "every customer trivially qualifies". A purist division (using NOT EXISTS ... NOT EXISTS) returns all customers for an empty divisor. With a non-empty catalog the counting form is correct and simpler; flag the empty-catalog divergence as a known limitation if the interviewer pushes.

The double-NOT EXISTS alternative

The canonical relational-division form is "no product exists that this customer did not buy": WHERE NOT EXISTS (SELECT 1 FROM product p WHERE NOT EXISTS (SELECT 1 FROM purchase pu WHERE pu.customer_id = c.customer_id AND pu.product_id = p.product_id)). It is the rigorous, textbook answer — handles the empty catalog correctly and needs no assumption that purchases reference valid products — but it is verbose and harder to read. The counting form is the pragmatic choice when the catalog is known non-empty and purchases are clean; knowing both, and when each applies, is the senior signal.

Senior signal: recognise "bought every product" as relational division and implement it by comparing COUNT(DISTINCT product_id) to the catalog size, insist on DISTINCT so repeat purchases do not inflate coverage, and name both the counting form's assumptions (non-empty catalog, purchases reference real products) and the rigorous double-NOT EXISTS alternative that handles the empty-divisor edge case.
