A product-analytics team has a raw clickstream — one row per event — and needs it reconstructed into sessions: bursts of activity separated by 30 minutes of silence. The interviewer is probing whether you reach for the modern single-pass technique (flag every gap, then a running SUM gives each event a session id) instead of an O(n²) self-join, and whether you can then roll each session up into duration, event count, bounce, and a purchase flag.

2. The SQL — cumulative-flag sessionization with a 30-minute gap rule

WITH flagged AS (
    -- Step 1: mark every event whose gap from the previous event
    -- exceeds 30 minutes (or which has no previous event) as a boundary.
    SELECT e.event_id, e.user_id, e.event_ts, e.event_type,
        CASE
          WHEN LAG(e.event_ts) OVER w IS NULL                              THEN 1
          WHEN (julianday(e.event_ts) - julianday(LAG(e.event_ts) OVER w))
                  * 24 * 60 > 30                                           THEN 1
          ELSE 0
        END AS new_session
    FROM events e
    WINDOW w AS (PARTITION BY e.user_id ORDER BY e.event_ts)
),
sessionized AS (
    -- Step 2: a running SUM of the boundary flag, per user, gives every
    -- event a deterministic session_id -- no self-join, single pass.
    SELECT user_id, event_ts, event_type,
        SUM(new_session) OVER (PARTITION BY user_id ORDER BY event_ts
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS session_id
    FROM flagged
)
SELECT user_id, session_id,
    MIN(event_ts) AS session_start,
    MAX(event_ts) AS session_end,
    CAST((julianday(MAX(event_ts)) - julianday(MIN(event_ts))) * 86400 AS INTEGER)
        AS duration_seconds,
    COUNT(*)                   AS event_count,
    COUNT(DISTINCT event_type) AS distinct_event_types,
    MAX(CASE WHEN event_type = 'purchase' THEN 1 ELSE 0 END) AS has_purchase,
    CASE WHEN COUNT(*) = 1 THEN 1 ELSE 0 END AS bounce
FROM sessionized
GROUP BY user_id, session_id
ORDER BY user_id, session_id;

The solution is written in SQLite dialect — julianday() returns a fractional day count, so × 24 × 60 converts to minutes and × 86400 to seconds. PostgreSQL has no julianday(); the equivalent is EXTRACT(EPOCH FROM (event_ts - LAG(event_ts) OVER w)) / 60 for minutes. The logic is identical — this is a dialect translation, not a correction.

4. Strategic explanations

Why flag-then-running-SUM beats a self-join

The naive sessionizer joins each event to all earlier events to find session boundaries — O(n²) and unacceptable at scale. This pattern is O(n log n): one LAG to read the previous timestamp, one boolean flag, one cumulative SUM. The cumulative SUM is the clever bit — it increments by 1 exactly at each boundary and is otherwise flat, so every event inside a session carries the same id, and every session is numbered 1, 2, 3… per user. No join, single ordered pass.

The ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW frame is load-bearing

For a running total the frame must be explicit. With an ORDER BY and no frame clause, SQL defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — and RANGE groups all peer rows with the same ORDER BY value together. If two events share an identical timestamp, a RANGE frame would sum them as one step and assign them the same partial total. ROWS counts physical rows, which is what a deterministic running counter needs.

The edge cases: bounce, ties, and the first event

LAG(...) OVER w IS NULL handles the first event of every user — it has no predecessor, so it always opens a session. The bounce case (COUNT(*) = 1) falls out naturally and gets duration_seconds = 0. A real trap is two events at the identical timestamp: the gap is 0, so they correctly stay in one session, but ORDER BY event_ts alone leaves their relative order undefined — in production you would tie-break on event_id to make LAG deterministic.

Dialect notes — timestamp arithmetic is the portability fault line

The gap math is the least portable part. SQLite uses julianday() differences. PostgreSQL subtracts timestamps directly to get an interval, then EXTRACT(EPOCH FROM ...) yields seconds. MySQL has TIMESTAMPDIFF(MINUTE, prev, curr). BigQuery has TIMESTAMP_DIFF(curr, prev, MINUTE). The sessionization skeleton — LAG, flag, cumulative SUM — is identical across all of them; only the duration expression changes.

What a senior says next

"The 30-minute timeout is a business parameter, not a constant — I would lift it into a CTE or a parameter so analysts can A/B different thresholds. And for an incremental pipeline I would watch the session-boundary case at partition edges: an event at 23:58 and the next at 00:03 belong to one session, so the job cannot naively shard by calendar day."

Senior signal: describe sessionization as flag-the-gap then cumulative-SUM (O(n log n), no self-join), insist on an explicit ROWS frame so identical timestamps do not collapse the running counter, tie-break LAG on event_id, and note that the 30-minute timeout belongs in a parameter and that day-sharding breaks sessions straddling midnight.
