Meta's Growth team measures the longest unbroken streak of consecutive daily-active dates for a cohort — gaps matter, but the dates arrive unsorted and with duplicates. "Longest Consecutive Sequence" is famous because the obvious solution (sort, then scan) is O(n log n), and the interviewer explicitly wants O(n). That constraint forces the hash-set insight.

1. The input data

nums = [100, 4, 200, 1, 3, 2]

Scrambled on purpose. The hidden run is 1→2→3→4 (length 4); 100 and 200 are isolated singletons.

2. The approach

Dump everything into a set — this gives O(1) membership tests and silently drops duplicates. Then iterate the set, but only do real work at the start of a run: n is a run start exactly when n - 1 is absent. That single guard is the whole trick.

From a confirmed start, walk upward with while cur + 1 in s, counting length, and update best. Because the inner walk only ever fires from a start element, each value is visited by the while loop at most once across the entire run — making the algorithm linear despite the nested loop.

4. Strategic explanations

Complexity

O(n) time — the set build is O(n), and although there is a while inside the for, the start-guard ensures every element is the target of cur + 1 in s at most once. O(n) space for the set. Stating why the nested loop is still linear is the point of the question.

The idiomatic alternative — sort, when O(n log n) is acceptable
def longestConsecutive(nums):
    if not nums: return 0
    nums = sorted(set(nums))
    best = run = 1
    for i in range(1, len(nums)):
        run = run + 1 if nums[i] == nums[i-1] + 1 else 1
        best = max(best, run)
    return best

Shorter and cache-friendly, but O(n log n). The right call only when the interviewer relaxes the time bound — otherwise the hash-set version is the expected answer.

The edge case — empty input and duplicates

[] never enters the loop, so best stays 0 — correct. Duplicates like [1, 1, 1] collapse in the set, so the run length is 1, not 3 — consecutive means distinct consecutive integers. A single element returns 1.

The follow-up — return the actual range, or handle a stream

"Don't just give me the length — give me the start and end." Track best_start alongside best. The harder follow-up is an online stream where you must maintain longest-so-far with a union-find or an interval map keyed on endpoints — that probes whether you can drop the "all data up front" assumption.

The trap — walking from every element

Dropping the if n - 1 not in s guard turns the algorithm O(n²): every element of a length-L run re-walks the rest of that run (L + (L−1) + ... = O(L²)). The code still returns the correct answer, so the bug is invisible in tests — it only shows up as a timeout on large input. The guard is not an optimization; it is the algorithm.

Senior signal: pre-empt "but that's a nested loop, isn't it O(n²)?" by explaining the amortization — each value is walked once because walks only start at run heads. Knowing why it is O(n) beats knowing that it is.
