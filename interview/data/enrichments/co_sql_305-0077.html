Deutsche Bank's customer-360 team wants one view that stitches a customer to their transactions and to whoever referred them. The schema is a kitchen sink — every table carries half the columns of every other table — so the real skill is choosing the correct join keys and join type. An interviewer is watching whether you keep every customer in the result, and whether you understand what happens to the row count when a customer has many transactions.

Three deliberate cases: Anita has two transactions and a referral; Boris has a transaction but no referral; Chen has a referral but no transaction. Each tests a different corner of the join.

2. The SQL — customers LEFT JOIN both

SELECT c.customer_id, c.name, t.txn_id, t.amount, r.referrer_id
FROM customers c
LEFT JOIN transactions t ON c.customer_id = t.customer_id
LEFT JOIN referrals r ON c.customer_id = r.referred_id;

Start from customers as the spine. LEFT JOIN transactions on customer_id, then LEFT JOIN referrals matching the customer to the referred_id column. Left joins guarantee every customer appears even when one or both side tables have nothing.

4. Strategic explanations

Join fan-out — why 3 customers became 4 rows

A LEFT JOIN to a table with multiple matches per key produces one output row per match. Anita has two transactions, so she fans out to two rows, and the referral join then duplicates her single referrer across both. This is correct join behaviour, but it has a sharp consequence: COUNT(*) on this result is 4, not 3, and SUM(t.amount) is fine but COUNT(DISTINCT c.customer_id) is the only safe way to count customers. Always know the grain of your result — here it is "customer-transaction", not "customer".

The referrer repetition trap

Because the referral join happens after the transaction fan-out, r.referrer_id = 88 is stamped onto every Anita row. If you later SUM or COUNT anything from referrals off this result, you double-count: referrer 88 looks like it referred Anita twice. When you need per-customer referral facts, aggregate the referral table separately or join it to a de-duplicated customer set.

LEFT vs INNER — who would disappear

If the transaction join were INNER, Chen (no transaction) would vanish. If the referral join were INNER, Boris (no referral) would vanish. With both inner, only Anita survives. The customer-360 view exists precisely to show every customer regardless of gaps — so both joins must be LEFT, with customers as the preserved left side.

Choosing the right key — the schema is a trap

Every table redundantly carries customer_id, referrer_id, referred_id, txn_id, name. The join must use the semantically correct key: transactions join on customer_id, but referrals join customer-to-referred_id (the customer is the one who was referred). Joining on referrer_id instead would answer a completely different question — "transactions of people this customer referred". Pick the key by meaning, not by name-matching.

Collapsing to one row per customer

If the consumer wants exactly one row per customer — transaction summary, not transaction detail — aggregate the transactions before the join so there is no fan-out:

SELECT c.customer_id, c.name,
       t.txn_count, t.total_amount, r.referrer_id
FROM customers c
LEFT JOIN (
  SELECT customer_id, COUNT(*) AS txn_count, SUM(amount) AS total_amount
  FROM transactions GROUP BY customer_id
) t ON c.customer_id = t.customer_id
LEFT JOIN referrals r ON c.customer_id = r.referred_id;

Pre-aggregating makes t one row per customer, the join no longer multiplies, and the referrer is no longer repeated — three customers, three rows.

Senior signal: use LEFT JOIN from customers so no customer is dropped, recognise that joining to the multi-row transactions table fans Anita out to two rows and inflates any naive COUNT(*), pick the referral key by meaning (referred_id, not referrer_id), and pre-aggregate transactions when the consumer needs one row per customer.
