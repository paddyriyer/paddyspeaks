SaaS billing teams need monthly recurring revenue (MRR) sliced by country and plan: "For every active subscription as of today, count subscribers and sum up the plan’s monthly price, broken down by user country and plan name." This question tests the correct active-subscription filter (handling NULL ended_at meaning "never cancelled"), multi-table JOIN chaining, and GROUP BY on a joined dimension attribute.

2. The SQL — active subscription filter with NULL-safe ended_at check

SELECT u.country, p.name AS plan,
       COUNT(*) AS active_subs,
       SUM(p.monthly_price) AS mrr
FROM subscriptions s
JOIN users u ON s.user_id = u.user_id
JOIN plans p ON s.plan_id = p.plan_id
WHERE s.started_at <= CURRENT_DATE
  AND (s.ended_at IS NULL OR s.ended_at > CURRENT_DATE)
GROUP BY u.country, p.name
ORDER BY u.country, mrr DESC;

4. Strategic explanations

The active-subscription predicate — the NULL trap

The most common interview mistake is writing WHERE s.ended_at > CURRENT_DATE without the NULL branch. A NULL ended_at means "never cancelled" — an indefinitely active subscription. But NULL > CURRENT_DATE evaluates to NULL (not TRUE), so those subscriptions are silently dropped. The correct predicate is always:

s.ended_at IS NULL OR s.ended_at > CURRENT_DATE

This is the textbook NULL-handling interview trap for subscription/session data.

The future-start guard: started_at ≤ CURRENT_DATE

Pre-provisioned subscriptions may have a future started_at. Without the started_at guard, S5 (starting 2026-06-01) would count as active today because its ended_at is NULL. The two-sided filter correctly brackets the active window: [started_at, ended_at) semantics using started_at ≤ today AND (ended_at IS NULL OR ended_at > today).

MRR as SUM of plan price — the plan-based model

The query computes MRR as SUM(p.monthly_price) across active subscriptions. This is correct when all subscribers on the same plan pay the same price. If pricing is per-subscriber (discounts, trials, usage-based), the monthly_price should be a column on subscriptions rather than plans. Identifying this modelling assumption is a senior data-modelling signal.

GROUP BY on a joined dimension: country and plan name

The GROUP BY groups on u.country and p.name — both from joined dimension tables, not from the fact table. This is clean: each subscription row resolves to exactly one country and one plan name through the INNER JOINs, so aggregation is unambiguous. Grouping by p.plan_id and selecting p.name would also work in standard SQL (since plan_id functionally determines name), but most engines require p.name to appear in GROUP BY unless it is in a functional dependency that the engine can prove.

Dialect notes
  • Snowflake: CURRENT_DATE is a keyword (no parentheses needed). CURRENT_DATE() also works.
  • Postgres: CURRENT_DATE (no parentheses). The solution runs as-is.
  • BigQuery: CURRENT_DATE() (parentheses required).
  • MySQL: CURDATE() or CURRENT_DATE.
Senior signal: immediately name the NULL trap in subscription filters — ended_at > today silently drops never-cancelled subscriptions because NULL > date is NULL, not TRUE. Add the started_at guard to exclude future-provisioned subscriptions, and flag that the plan-price MRR model breaks when per-subscriber pricing exists (move the price column to the subscriptions table).
