A CRM report wants every customer's biggest order — one row per customer — and crucially must still list customers who have never ordered, with blank order fields. The interviewer is probing two things at once: top-1-per-group with a deterministic tie-break, and a LEFT JOIN that survives the "no rows on the right" case.

Two edge cases. C01 ties at 800 on two orders — the earlier date (ord 3, Jan 15) must win. And C04 has no orders at all: it must still appear, with NULL order columns. Lose C04 and you have used the wrong join.

2. The SQL — rank in a CTE, then LEFT JOIN on rk = 1

WITH ranked AS (
  SELECT cust_code, ord_num, ord_amount, ord_date,
         ROW_NUMBER() OVER (PARTITION BY cust_code
                            ORDER BY ord_amount DESC, ord_date ASC, ord_num ASC) AS rk
  FROM orders
)
SELECT c.cust_code, c.cust_name, r.ord_num, r.ord_amount, r.ord_date
FROM customer c
LEFT JOIN ranked r ON r.cust_code = c.cust_code AND r.rk = 1
ORDER BY c.cust_code;

The CTE numbers each customer's orders, largest first, with date then ord_num breaking ties. The outer query LEFT JOINs customer to that ranked set, keeping only the rk = 1 row — and because it is a left join, a customer with no matching ranked row is retained with NULLs.

4. Strategic explanations

Why the join condition, not a WHERE, carries rk = 1

The rk = 1 predicate sits in the ON clause, not a WHERE. That placement is essential. In a LEFT JOIN, ON conditions decide which right-side rows match; an unmatched left row is still kept with NULLs. If you moved rk = 1 to WHERE, the filter would run after the join — C04's all-NULL row has rk = NULL, NULL = 1 is unknown, and the row would be dropped. The classic bug: a WHERE on a right-table column silently converts a LEFT JOIN back into an inner join.

ROW_NUMBER guarantees exactly one row per customer

C01 and C03 each have a tie on amount. RANK() would give both tied orders rk = 1, and the join would return two rows for that customer — breaking "one row per customer". ROW_NUMBER() always produces a strict 1, 2, 3 sequence, so exactly one order gets rk = 1. The cost is that you must dictate the tie order yourself, which the multi-key ORDER BY does.

The tie-break chain must end on a unique key

ord_amount DESC, ord_date ASC resolves C01 (different dates) but consider two orders identical on amount and date — the chain must still decide. Ending with ord_num ASC, a unique key, makes the ordering total: there is always a definite winner and the result is reproducible run after run. A tie-break that bottoms out on a non-unique column leaves the engine free to pick arbitrarily.

Why a CTE + join beats a correlated subquery

An alternative is a correlated subquery picking MAX(ord_amount) per customer — but that reopens the tie problem (two orders share the max amount) and re-scans orders per customer. The CTE ranks all orders in a single windowed pass, then one join attaches the winner. It also keeps the tie-break logic in exactly one place. For top-N (N>1) the same CTE just changes the filter to rk <= N — the pattern scales.

Dialect and the NULLS footnote

ROW_NUMBER() OVER (PARTITION BY …) is standard in Postgres, SQL Server, Oracle, MySQL 8+, and SQLite 3.25+. Snowflake/BigQuery/DuckDB offer QUALIFY rk = 1 as sugar for the outer filter, but you would still need the left join for no-order customers, so the CTE form is the portable choice. One trap: if ord_amount can be NULL, ORDER BY ord_amount DESC sorts NULLs first in Postgres — a NULL-amount order would wrongly rank #1; add NULLS LAST if that is possible.

Senior signal: top-1-per-group with every customer retained means ROW_NUMBER() in a CTE plus a LEFT JOIN with rk = 1 kept in the ON clause — moving it to WHERE silently drops the no-order rows — and the tie-break chain must end on a unique key so the single winner is deterministic.
