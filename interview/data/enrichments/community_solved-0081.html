The task is a reshaping problem: a long table with one row per (class, subject) must become a wide table with one row per class and the subjects spread across numbered columns sub1…sub5. This is a pivot, and the interviewer is checking whether you can manufacture a per-group sequence number and then collapse the rows with conditional aggregation.

1. The source data — subject_info

classsubject(narrative)
Class AMath3 subjects
Class AScience
Class AEnglish
Class BHistoryonly 2 subjects
Class BGeography
Class CPhysicssingle-subject class — edge case

Classes have a different number of subjects — A has three, B has two, C has just one. The pivot must leave the unused columns NULL rather than mis-aligning rows.

2. The SQL — number the rows, then pivot with conditional MAX

with cte1 as(
select *,
    ROW_NUMBER() over(partition by class order by (select null)) as flag
from subject_info
), cte2 as(
select *,
    case when flag = 1 then subject else null end as sub1,
    case when flag = 2 then subject else null end as sub2,
    case when flag = 3 then subject else null end as sub3,
    case when flag = 4 then subject else null end as sub4,
    case when flag = 5 then subject else null end as sub5
from cte1)
select class,
    max(sub1) as sub1,
    max(sub2) as sub2,
    max(sub3) as sub3,
    max(sub4) as sub4,
    max(sub5) as sub5
from cte2
group by class;

3. The resulting analysis

Class CPhysicsNULLNULLNULLNULL

Each class collapses to one row; subjects fill the columns left-to-right and the surplus columns stay NULL. Class C's single subject lands in sub1 with the rest NULL — the single-member group behaves correctly.

4. Strategic explanations

The pivot mechanic — number, spread, collapse

Step 1 assigns each subject within a class a position 1–5 via ROW_NUMBER. Step 2 spreads each subject into exactly one of five columns using a CASE per slot — for any given row, four of the five columns are NULL. Step 3 GROUP BY class collapses the rows; MAX picks the one non-NULL value out of each column's stack of NULLs (any aggregate that ignores NULLs — MIN, MAX — works, since at most one real value exists per slot). This three-CTE shape is the textbook static pivot.

The fatal flaw — ORDER BY (SELECT NULL) is non-deterministic

ROW_NUMBER() OVER (PARTITION BY class ORDER BY (select null)) deliberately orders by a constant, which means the engine is free to assign positions in any order. Run the query twice and Class A might show Math/Science/English one time and Science/English/Math the next. For a stable pivot you need a real ordering key — ORDER BY subject (alphabetical) or an explicit subject_seq column. An interviewer will absolutely ask "is this output reproducible?"; the honest answer is no, and you should name the fix.

The hard-coded width — the 6th subject silently disappears

The query supports exactly five subjects because there are five CASE branches and five MAX columns. A class with a sixth subject gets flag = 6, which matches no CASE branch, so that subject is dropped from the output entirely — no error, no warning. Static pivots always bake in the column count; truly dynamic width needs procedural SQL (build the query string) or Postgres's crosstab, and you should flag the assumption out loud.

Dialect note — (SELECT NULL) is a T-SQL habit

ORDER BY (SELECT NULL) is a SQL Server idiom for "I need ROW_NUMBER but have no natural order." Postgres accepts it too. Several engines also offer a dedicated PIVOT operator (SQL Server, Oracle, Snowflake) that expresses this more declaratively; Postgres and MySQL rely on the conditional-aggregation pattern shown here. The technique is portable; the PIVOT keyword is not.

What to say next — pivots are inherently order-dependent

The deeper point: a pivot to numbered columns only makes sense if "which subject is #1" is well defined. If subjects are an unordered set, numbering them is arbitrary and the wide shape is questionable — an array (array_agg(subject)) or a delimited string (string_agg) may model the data more honestly. Raising that — "what should determine the column order?" — is the senior signal.

Senior signal: recognise the number–spread–collapse pivot pattern, but immediately flag that ORDER BY (SELECT NULL) makes the column assignment non-deterministic — demand a real ordering key. Note the hard-coded five-column width silently drops a sixth subject, and that truly dynamic pivots need procedural SQL.
