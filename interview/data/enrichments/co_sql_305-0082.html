Booking.com's revenue team wants every order tagged into a size band — Small, Medium, or Large — so dashboards can slice conversion and support load by order value. A CASE expression is the obvious tool. What the interviewer is actually testing is whether you understand how CASE evaluates top-to-bottom, where the band boundaries fall, and what happens at the exact threshold values.

Orders 3 (total = 50) and 6 (total = 500) sit exactly on the boundaries. Those two rows are the whole interview — they expose whether the cut-offs are inclusive or exclusive.

2. The SQL — CASE bucketing

SELECT order_id,
  CASE WHEN total < 50 THEN 'Small'
       WHEN total < 500 THEN 'Medium' ELSE 'Large' END AS bucket
FROM orders;

CASE tests branches in order and stops at the first true one. The ELSE catches everything that fell through. There is no WHERE, so every order is returned and labelled.

The bands are lower-bound-inclusive: Small is [0, 50), Medium is [50, 500), Large is [500, ∞). Order 5 at 499.99 stays Medium — one cent decides the label.

4. Strategic explanations

Order matters — CASE short-circuits

CASE WHEN ... WHEN ... is not a set of independent rules; it is a ladder. The first branch that evaluates true wins and the rest are skipped. That is why the Medium branch can be written as the bare total < 500 — any row reaching it has already failed total < 50, so the lower bound is implied. Re-order the branches and the logic breaks: put total < 500 first and every Small order would be mislabelled Medium.

The boundary-semantics trap

"Under 50 is Small, 50 to 500 is Medium" is ambiguous in English. This CASE resolves it as lower-inclusive, upper-exclusive. An interviewer may want the opposite — e.g. Medium is "50 up to and including 500". Be explicit and write it to match:

CASE WHEN total <= 50  THEN 'Small'
     WHEN total <= 500 THEN 'Medium'
     ELSE 'Large' END

With this version order 3 becomes Small and order 6 becomes Medium. State which convention you are using and confirm the boundary rows are where the business expects.

The NULL row falls into ELSE — usually wrong

If total is NULL, every comparison (NULL < 50, NULL < 500) is unknown, never true — so the row drops through to ELSE and is silently labelled 'Large'. A NULL-priced order tagged as the biggest band is a real reporting bug. Handle NULL on purpose with an explicit branch:

CASE WHEN total IS NULL  THEN 'Unknown'
     WHEN total < 50     THEN 'Small'
     WHEN total < 500    THEN 'Medium'
     ELSE 'Large' END

Put the IS NULL test first so it catches the row before any comparison can mis-route it.

Counting orders per bucket

The natural follow-up is "how many orders in each band". Wrap the CASE in a subquery and group, or group by the expression directly:

SELECT CASE WHEN total < 50 THEN 'Small'
            WHEN total < 500 THEN 'Medium' ELSE 'Large' END AS bucket,
       COUNT(*) AS orders
FROM orders
GROUP BY 1
ORDER BY 1;

For this data: Small 2, Medium 3, Large 2. Knowing you can GROUP BY a positional reference (or repeat the expression) to a derived label is a small but expected fluency.

The schema has both total and amount

The table ships total and amount as separate columns — classic near-duplicates that drift (pre- vs post-discount, with vs without tax). The solution buckets on total; confirm that is the value the business means by "order size" rather than assuming.

Senior signal: explain that CASE short-circuits so branch order encodes the lower bounds, state the inclusive/exclusive boundary convention and verify the on-threshold rows, and pre-empt the NULL row silently landing in ELSE by giving it an explicit first branch.
