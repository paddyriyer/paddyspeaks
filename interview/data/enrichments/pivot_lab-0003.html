A growth team at a consumer app reports engagement with the industry-standard L1–L28 metric: of the trailing 28 days, how many distinct days did each user open the app? An L28 user showed up all four weeks; an L1 user showed up once. The leadership dashboard wants a single snapshot row that buckets the whole user base into engagement tiers. That is a pivot: distinct bucket labels become columns, and the interviewer is checking whether you can collapse a per-user count into a one-row tiered summary.

1. The source data — user_activity

One row per user per active day. Distinct-day counts inside the 28-day window (2026-04-18 to 2026-05-15):

Five users span the full spread of buckets: 102 is exactly L1, users 101 and 103 sit in L2–L7, user 104 lands in L8–L27, and user 105 hit all 28 days. One user per tier — a clean test of the bucket boundaries.

2. The SQL — count per user, then pivot into buckets

WITH user_28d_counts AS (
  SELECT user_id, COUNT(DISTINCT activity_date) AS active_days_count
  FROM user_activity
  WHERE activity_date BETWEEN '2026-04-18' AND '2026-05-15'
  GROUP BY user_id
)
SELECT
  '2026-05-15' AS snapshot_date,
  COUNT(CASE WHEN active_days_count = 1 THEN 1 END) AS l1_users,
  COUNT(CASE WHEN active_days_count BETWEEN 2 AND 7 THEN 1 END) AS l2_l7_users,
  COUNT(CASE WHEN active_days_count BETWEEN 8 AND 27 THEN 1 END) AS l8_l27_users,
  COUNT(CASE WHEN active_days_count >= 28 THEN 1 END) AS l28_users
FROM user_28d_counts;

Stage one (user_28d_counts) reduces the event log to one row per user holding their distinct-day count. Stage two pivots: each COUNT(CASE…) tallies users whose count falls in one bucket. With no GROUP BY, the whole CTE collapses to a single summary row.

The five users partition cleanly: 1 in L1 (user 102), 2 in L2–L7 (users 101 and 103), 1 in L8–L27 (user 104), 1 in L28 (user 105). Every user lands in exactly one bucket and the counts sum to 5 — the partition is total and disjoint. The four input rows of the pivot stage collapse to a single output row because there is no GROUP BY: the whole CTE is one implicit group, which is exactly the shape a snapshot dashboard wants.

4. Strategic explanations

Why two stages — you cannot bucket what you have not counted

The buckets are defined over per-user distinct-day counts, and that count does not exist in the raw table — it must be derived first. The CTE produces the intermediate metric; the outer query pivots it. Trying to do both at once fails: COUNT(DISTINCT activity_date) is per user, but the bucket COUNT(CASE…) is over users — two different grains, so two stages. Naming the grain change is the senior framing.

This is a pivot — bucket labels become columns

A long-format answer would be SELECT bucket, COUNT(*) GROUP BY bucket — four rows. The dashboard wants four columns in one row, so the bucket dimension is rotated into the header. COUNT(CASE WHEN … THEN 1 END) per bucket is the portable pivot idiom: each CASE emits a non-NULL only for its bucket, and COUNT — which ignores NULLs — tallies just those.

Why COUNT with no ELSE, not SUM with ELSE 0

COUNT counts non-NULL values, so CASE WHEN cond THEN 1 END — which yields NULL when the condition is false — counts exactly the matching users. SUM(CASE WHEN cond THEN 1 ELSE 0 END) gives the identical number. Both are correct; COUNT without ELSE is the tighter idiom for counting rows, SUM with ELSE 0 reads better when you are adding amounts. Pick one and be consistent — do not write COUNT(CASE … ELSE 0 END), which counts every row because 0 is non-NULL.

The buckets must be exhaustive and non-overlapping

= 1, BETWEEN 2 AND 7, BETWEEN 8 AND 27, >= 28 tile the integers 1 and up with no gap and no overlap, so every user is counted once. Off-by-one errors hide here: write BETWEEN 1 AND 7 for L2–L7 and user 102 is double-counted; stop L8–L27 at 27 but start L28 at > 28 and a 28-day user falls through a crack. A user inactive in the window produces no CTE row at all, so 0-day users are simply absent — correct if "L0" is out of scope, a bug if the deck expects them. The boundary >= 28 also guards against dirty data: timestamps mis-bucketed could push a count above 28, and >= still catches them.

Dialect — FILTER and the PIVOT operator

PostgreSQL offers a cleaner spelling: COUNT(*) FILTER (WHERE active_days_count = 1) — the SQL-standard aggregate FILTER clause, more readable than CASE and identical in result. Snowflake, Oracle, and SQL Server have a PIVOT operator, but it pivots on discrete values, not ranges — so for bucketed counts like L1–L28 you would still pre-compute a bucket label, then pivot that. The CASE / FILTER form handles ranges directly and works everywhere.

Senior signal: split the work into a per-user count CTE and a bucket-pivot outer query because the two live at different grains, implement the pivot with COUNT(CASE WHEN … THEN 1 END) (or Postgres FILTER), and verify the range buckets are exhaustive and non-overlapping — off-by-one boundaries and silently-absent 0-day users are the traps an interviewer is hunting for.
