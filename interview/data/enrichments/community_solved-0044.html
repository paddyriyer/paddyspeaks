A Community member asks for every prime number between 1 and 10, computed purely in SQL. A prime is a natural number greater than 1 divisible only by 1 and itself. There is no table — the candidates must be generated — and the interviewer is checking whether you can express "has no smaller divisor" as a set operation rather than a loop.

The candidates 2–10 are generated, not stored. The edge that matters: the test must compare each number against divisors strictly smaller than itself — every number divides itself evenly, so an unguarded modulo test would reject every candidate, including the primes.

2. The SQL — generate candidates, exclude composites

WITH cte AS (
  SELECT 2 AS prime_num
  UNION ALL
  SELECT (prime_num + 1) AS prime_num
  FROM cte
  WHERE prime_num < 10
)
SELECT t1.prime_num
FROM cte t1
WHERE NOT EXISTS (
  SELECT 1 FROM cte t2
  WHERE 1 = 1
    AND t1.prime_num % t2.prime_num = 0
    AND t1.prime_num > t2.prime_num
);

The recursive CTE produces the integers 2 through 10. The outer query keeps a candidate only when NOT EXISTS a smaller number that divides it evenly — the definition of prime, expressed as a self-join.

4. Strategic explanations

Why t1.prime_num > t2.prime_num is load-bearing

This predicate is the whole trick. Drop it and t2 ranges over all candidates — including t2 = t1 itself. Since t1.prime_num % t1.prime_num = 0 is always true, the NOT EXISTS would fail for every candidate and the query would return nothing. The > restricts the divisor search to numbers strictly below the candidate, which also implicitly excludes the divisor 1 (since the candidates start at 2). The reference's commented-out != line was an earlier, weaker attempt at the same guard — > is the correct one.

The set-based mindset — NOT EXISTS is "for all"

"A prime has no divisor below it" is a universally-quantified statement. SQL has no FOR ALL, so you express it as NOT EXISTS (a counterexample) — "there does not exist a smaller divisor". Recognising that universal claims map to NOT EXISTS over the negation is a reusable pattern: "every order shipped", "no failed test", "all dependencies met" are all the same shape.

The dialect note — recursion keyword and modulo

Standard SQL and Postgres require WITH RECURSIVE cte AS (...); SQL Server and Oracle omit RECURSIVE. The % operator is the modulo in Postgres, SQL Server and MySQL; Oracle uses the MOD(a, b) function instead. The algorithm is identical — only the keyword and the modulo spelling change between engines.

The set-generation alternative

In Postgres the candidate CTE is just a series:

SELECT n FROM generate_series(2, 10) AS n
WHERE NOT EXISTS (
  SELECT 1 FROM generate_series(2, n - 1) AS d
  WHERE n % d = 0
);

Cleaner: no anchor, no termination predicate. Note the inner generator runs 2 .. n-1 — the upper bound itself enforces "strictly smaller", so no extra > guard is needed.

Scaling and the √n optimisation

This is O(n²): each candidate is tested against every smaller number. For 1–10 that is trivial; for 1–1,000,000 it is not. The number-theory fact is that a composite n always has a divisor ≤ √n, so the inner search can stop at d * d <= n — a quadratic-to-near-linear improvement. Mentioning the √n bound shows you see SQL puzzles as algorithms with a cost, not just syntax.

The boundary cases — 1 and 2

Two classic off-by-one traps. 1 is not prime — starting the candidate set at 2 handles that for free; starting at 1 would wrongly emit it (1 has no smaller divisor above 1). 2 is prime and is the only even prime — it has no candidate strictly between 1 and 2, so NOT EXISTS is trivially true and 2 is correctly kept. A naive "skip all even numbers" optimisation would wrongly drop 2.

Senior signal: express "prime" as NOT EXISTS (a smaller divisor) — the SQL idiom for a universal claim — with a strict > guard so a number is never tested against itself, handle the 1-is-not-prime and 2-is-prime boundaries deliberately, and note the √n bound that turns the quadratic scan near-linear at scale.
