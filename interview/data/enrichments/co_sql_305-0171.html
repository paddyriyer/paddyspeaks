Infosys runs demand-planning for a retail client and needs to grade each period's forecast against what actually happened. The metric is absolute percentage error: how far off was the forecast, as a percentage of actual? The twist the interviewer is hunting for — the forecast and actual columns arrived as messy text ("1,100", "$1000"), and a zero or missing actual will blow up the division.

1. The source data — forecasts

periodforecastactual
2024-Q111001000
2024-Q29001000
2024-Q310001000
2024-Q41200NULL

Q1 over-forecast, Q2 under-forecast, Q3 was spot-on. Q4 has no actual yet — the quarter has not closed. Dividing by that missing actual is the trap the query must survive.

2. The SQL — ABS(forecast − actual) / actual

SELECT period,
  ABS(
    NULLIF(regexp_replace(forecast::text, '[^-.0-9]', '', 'g'), '')::numeric
  - NULLIF(regexp_replace(actual::text,   '[^-.0-9]', '', 'g'), '')::numeric
  )
  / NULLIF(NULLIF(regexp_replace(actual::text, '[^-.0-9]', '', 'g'), '')::numeric, 0)
  * 100 AS error_pct
FROM forecasts;

Each value is stripped to digits, sign and decimal point with regexp_replace, cast to numeric, and the absolute difference is divided by actual and scaled to a percentage. Two NULLIFs guard the denominator.

Q1 and Q2 both score 10% error — ABS makes a +100 miss and a −100 miss look identical in magnitude. Q3's perfect forecast scores 0%. Q4 returns NULL, not a crash: with no actual, error percentage is genuinely undefined, and the query says so honestly.

4. Strategic explanations

The divide-by-zero defence — NULLIF(…, 0)

The outer NULLIF(actual_numeric, 0) is the safety valve. If a period's actual is exactly 0 — a genuine zero-sales quarter — dividing by it would raise division by zero and abort the whole query. NULLIF(x, 0) turns a 0 denominator into NULL, and division by NULL yields NULL: one bad row degrades to a NULL instead of killing the report. Percentage error against a zero actual is mathematically undefined anyway, so NULL is the correct answer, not a workaround.

The two roles of NULLIF here — do not conflate them

NULLIF appears three times doing two different jobs. The inner NULLIF(regexp_replace(…), '') handles a value that is empty after cleaning — a blank string, or text with no digits at all — so ''::numeric (which would error) never happens. The outer NULLIF(…, 0) handles a zero denominator. Same function, two distinct failure modes: empty-input and divide-by-zero.

Why regexp_replace at all — and the dialect note

The columns are typed as text and may hold "$1,100" or "1 200". regexp_replace(x, '[^-.0-9]', '', 'g') deletes every character that is not a digit, minus, or dot, leaving a clean numeric string. If the columns were already a numeric type, all of this scaffolding collapses to ABS(forecast - actual) / NULLIF(actual,0) * 100 — mention that the defensive parsing is a symptom of bad upstream typing. (The regexp_replace here is standard Postgres; the same idea in other engines uses REGEXP_REPLACE with minor syntax differences.)

The interviewer follow-up: APE vs MAPE

This query gives per-period absolute percentage error. The headline accuracy metric planners actually report is MAPE — the mean of those across periods:

SELECT AVG(error_pct) AS mape
FROM ( … the per-period query … ) p;

And AVG skipping the Q4 NULL is exactly right here: you cannot average in an error you could not compute. Be ready to also name bias — the signed mean error — which tells you if the forecast systematically over- or under-shoots, something ABS deliberately hides.

The asymmetry trap of percentage error

MAPE has a known flaw: under-forecasting is capped at 100% error, but over-forecasting is unbounded. Forecasting 2000 against an actual of 1000 is 100% error; forecasting 0 against 1000 is also 100%. This skews the metric toward penalising over-forecasts. Senior candidates mention symmetric MAPE or WAPE (weighted by volume) as the fix when the interviewer pushes on metric choice.

Senior signal: guard the denominator with NULLIF(actual, 0) so a zero actual degrades to NULL instead of crashing, recognise the inner NULLIF(…, '') as a separate empty-string defence, treat the text-parsing as a code smell of bad upstream typing, and aggregate to MAPE while naming its asymmetry weakness.
