Meta's data engineering interviews love this question because it sits exactly where analysts and engineers collide: a recruiter wants a quarterly regional-sales workbook, and you have a database. The task tests whether you can move data cleanly across three boundaries — SQL engine, in-process DataFrame, and a delivered file — without losing types or silently corrupting numbers. It is less about clever algorithms and more about knowing which tool owns which step.

1. The input data

A single sales table loaded into an in-memory SQLite database:

regionamount
NA100.0
NA50.0
APAC75.0
EMEA125.0
APAC25.0

2. The approach

The solution deliberately splits work between layers. SQLite does the aggregation — SUM(amount) ... GROUP BY region ORDER BY total DESC — because the database is far faster at folding millions of rows into a handful of groups than Python is. fetchall() then pulls back only the small result set (one row per region), not the raw table. In production the next line would be pd.DataFrame(rows, columns=['region','total']).to_excel('out.xlsx', index=False); pandas owns the file-format step because it knows how to write a typed .xlsx via openpyxl. The demo substitutes the csv module writing into an io.StringIO buffer so the snippet runs anywhere without the Excel dependency — same shape of output, no binary writer. Note defaultdict is imported but unused; a senior candidate would delete it.

3. The output

NA totals 150, EMEA 125, APAC 100, sorted descending:

region,total
NA,150.0
EMEA,125.0
APAC,100.0

4. Strategic explanations

Complexity — and where the work belongs

The GROUP BY is O(n) over the rows with O(g) groups; the database streams it. Materializing into pandas is O(g), trivial here. The senior point: do not SELECT * then group in Python — that ships every row over the connection and discards the engine's indexes and hash-aggregate.

The idiomatic alternative

If pandas is already a dependency, skip the manual cursor entirely: pd.read_sql_query(query, con) returns a typed DataFrame in one call, then .to_excel(...). That is the canonical two-liner an interviewer wants to hear named.

The edge case — the index column

The most common bug here is forgetting index=False on to_excel, which writes pandas' 0,1,2 row index as a phantom first column the recruiter has to delete by hand. Also: an empty sales table yields zero rows, so the workbook has headers only — usually correct, but confirm the consumer does not expect a "no data" sentinel.

The follow-up — multiple sheets

"Now put each region on its own tab." Switch to pd.ExcelWriter('out.xlsx') as a context manager and call df.to_excel(writer, sheet_name=region) per group. The SQL stays identical; only the delivery layer changes — which is the whole lesson of the question.

The trap — floats and money

Summing REAL columns accumulates binary-float error; 0.1 + 0.2 is the classic. For currency, aggregate as integer cents or cast to NUMERIC/Decimal before the workbook is published, or the totals will be off by fractions of a cent that auditors notice.

Senior signal: a senior candidate pushes aggregation into SQL, names read_sql_query + to_excel(index=False) as the clean path, and flags float-money precision before being asked — a junior pulls the whole table into Python and loops.
