A community payments dataset asks a classic finance question: "for every transaction, show the customer's largest transaction and how this one stacks up against it." The ratio turns raw dollars into a normalised 0–1 score — useful for spotting whether a charge is a customer's biggest spend or pocket change. The interviewer wants three implementations because each reveals whether you understand window functions, correlated subqueries, and joins as interchangeable tools, and which one scales.

Customer 101 has two transactions tied at 800 — both must report ratio 1.000. Customer 102 has a single transaction, so its ratio is trivially 1.000 (it is its own max). Those two cases are what the interviewer is checking.

2. The SQL — MAX as a window function

SELECT
    customer_id,
    transaction_id,
    transaction_amount,
    MAX(transaction_amount) OVER (PARTITION BY customer_id) AS max_customer_txn_amount,
    ROUND(
        (1.0 * transaction_amount
            / MAX(transaction_amount) OVER (PARTITION BY customer_id))::NUMERIC,
        3
    ) AS ratio
FROM transaction_tab
ORDER BY customer_id, transaction_id;

MAX(...) OVER (PARTITION BY customer_id) computes the per-customer maximum but, unlike GROUP BY, keeps every original row — so the max sits beside each transaction and the ratio is a plain inline division.

Both of customer 101's 800-dollar rows get ratio 1.000 — the window MAX does not collapse ties, it simply reports 800 against each. Customer 102's lone transaction is its own maximum, ratio 1.000.

4. Strategic explanations

Why the window form is the one to write

PARTITION BY customer_id defines a window per customer; MAX is evaluated over that window for every row, with no GROUP BY and therefore no loss of detail. One sort over the table, one pass — O(n log n). It is the shortest correct answer and the one a senior reaches for first.

The three methods the question wants

Method 1 — window function (shown above). Method 2 — correlated subquery: (SELECT MAX(t2.transaction_amount) FROM transaction_tab t2 WHERE t2.customer_id = t1.customer_id) — correct, but it re-scans the table once per row, O(n²). Method 3 — join to a derived table: JOIN (SELECT customer_id, MAX(transaction_amount) AS mx FROM transaction_tab GROUP BY customer_id) m USING (customer_id) — one aggregate pass plus a hash join, O(n). Methods 1 and 3 scale; method 2 is the teaching counter-example. Saying that out loud is the signal.

The division-by-zero trap

If a customer's transactions sum to a maximum of zero — e.g. a single 0.00 charge, or only refunds — the divisor is 0 and Postgres raises division by zero (error 22012), aborting the whole query. The robust form wraps the denominator: ... / NULLIF(MAX(transaction_amount) OVER (PARTITION BY customer_id), 0), which turns a zero max into NULL and yields a NULL ratio instead of an error. Mention this unprompted — real transaction tables contain zero-dollar and negative rows.

The cast and rounding

1.0 * forces numeric division (an integer-typed amount would otherwise truncate). The explicit ::NUMERIC is needed because ROUND(value, scale) in Postgres only accepts a numeric first argument — ROUND(double precision, int) does not exist and would error. In MySQL or SQL Server ROUND is more permissive, so the cast is Postgres-specific defensive coding.

Senior signal: deliver the window-function version as the primary answer, name the correlated subquery as the O(n²) trap among the three methods, and guard the ratio's denominator with NULLIF(..., 0) before the interviewer asks what happens to a zero-max customer.
