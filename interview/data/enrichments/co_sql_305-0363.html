Stripe's most-trusted-by-finance query: per-merchant charges − refunds = expected_payout, compared to actual_payout. The diff column shows reconciliation errors. Every accounting closing process runs this.

1. The source data — charges + refunds + payouts

chargesrefundspayouts
C_01M_01 / $100R_01C_01 / $20P_01M_01 / $80

2. The SQL — multi-table net + diff

WITH net AS (
  SELECT c.merchant_id,
    ROUND(SUM(c.amount), 2) AS gross_charges,
    ROUND(COALESCE(SUM(r.amount), 0), 2) AS gross_refunds,
    ROUND(SUM(c.amount) - COALESCE(SUM(r.amount), 0), 2) AS expected_payout
  FROM charges c LEFT JOIN refunds r ON r.charge_id = c.charge_id
  GROUP BY c.merchant_id
)
SELECT n.merchant_id, n.gross_charges, n.gross_refunds, n.expected_payout,
  ROUND(COALESCE(SUM(p.amount), 0), 2) AS actual_payout,
  ROUND(n.expected_payout - COALESCE(SUM(p.amount), 0), 2) AS reconciliation_diff
FROM net n LEFT JOIN payouts p ON p.merchant_id = n.merchant_id
GROUP BY n.merchant_id, n.gross_charges, n.gross_refunds, n.expected_payout
ORDER BY ABS(reconciliation_diff) DESC;

3. Strategic explanations

The double-LEFT-JOIN

charges → refunds (one-to-many; some charges aren't refunded) and net → payouts (one merchant, multiple payouts). Both must be LEFT to keep merchants who haven't been refunded or paid out yet.

The COALESCE around SUM

If a merchant has no refunds, SUM(r.amount) returns NULL — and charges - NULL = NULL. COALESCE(SUM(r.amount), 0) ensures arithmetic continues. Same for payouts.

The reconciliation_diff interpretation
diff signmeaning
0perfect match — expected
+ (positive)expected > actual: payout under-disbursed (Stripe owes merchant)
− (negative)actual > expected: payout over-disbursed (merchant owes Stripe)
The fee-and-tax adjustment

Real payout = gross − refunds − Stripe fees − reverse holds. The query above ignores fees. Production:

expected_payout = gross_charges - gross_refunds - SUM(stripe_fees) - SUM(reverse_holds)
The currency mismatch

Multi-currency merchants need FX-adjusted reconciliation. A charge in EUR and payout in USD must use the at-time-of-payout FX rate. The diff includes FX gain/loss, not just a real discrepancy.

The timing window

Charges arrive continuously; payouts run on a schedule (daily/weekly). At any moment there's a "pending" amount — charges not yet paid out. The reconciliation needs a time cutoff:

WHERE c.charged_at <= '2024-03-01' AND p.paid_at <= '2024-03-08'
The threshold that pages

diff < $1 = rounding noise. diff > $100 = investigate. Production paged on |diff| > $100 with a SLA to resolve within 24 hours.

Senior signal: double-LEFT-JOIN with COALESCE-around-SUM, interpret diff direction (under-pay vs over-pay), include fees + FX in real reconciliation, pin a time cutoff for in-flight charges, and define the "investigate" threshold. Currency-mismatch handling is the over-the-bar correctness move.
