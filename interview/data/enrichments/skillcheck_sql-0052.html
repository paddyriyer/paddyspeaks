An experimentation team wants to know whether two A/B tests overlap: which users were enrolled in both the feed_v2 and pricing_v3 experiments, and how their variant assignments cross-tabulate. The experiment_users table is one row per (experiment, user, variant). The interviewer is probing a self-join on a long-format table to produce a crosstab — and whether you understand that the join itself enforces the "both experiments" condition.

1. The source data — experiment_users

experimentuser_idvariant(narrative)
feed_v21, 2, 5, 6Ausers on the feed_v2 control
feed_v23, 4, 7Busers on the feed_v2 treatment
pricing_v31, 3, 6, 7Ausers on the pricing_v3 control
pricing_v32, 4Busers on the pricing_v3 treatment

The intersection is the point. Users 1, 2, 3, 4, 6, 7 appear in both experiments — they are the only ones counted. User 5 is in feed_v2 only and user 5's absence from pricing_v3 means the join drops them. The edge case to watch: every (variant_feed, variant_pricing) combination must surface, including thinly-populated cells.

2. The SQL — self-join on user_id, crosstab the variants

SELECT f.variant AS variant_feed,
       p.variant AS variant_pricing,
       COUNT(*) AS users
FROM experiment_users f
JOIN experiment_users p
  ON f.user_id = p.user_id
 AND f.experiment = 'feed_v2'
 AND p.experiment = 'pricing_v3'
GROUP BY f.variant, p.variant
ORDER BY variant_feed, variant_pricing;

The table is joined to itself: alias f is filtered to feed_v2 rows, alias p to pricing_v3 rows, and the join matches them on user_id. A row survives only if the same user exists in both filtered sets — that is the intersection. Grouping by the two variants produces the crosstab counts.

3. The resulting analysis

The four cells sum to 6 — exactly the six users in both experiments. User 5 contributed nothing: with no pricing_v3 row, the inner join found no partner and dropped them. The crosstab shows feed and pricing assignments are roughly balanced — no obvious correlation between the two experiments' bucketing.

4. Strategic explanations

The self-join is the intersection — no INTERSECT needed

An INNER JOIN only emits a row when a match exists on both sides. By filtering f to feed_v2 and p to pricing_v3 and joining on user_id, a user appears in the result if and only if they are in both experiments. The "enrolled in BOTH" requirement is satisfied structurally by the join — no separate INTERSECT, EXISTS, or HAVING COUNT is required.

Filters in the ON clause vs the WHERE clause

Here the experiment filters sit in the ON clause. For an INNER JOIN that is equivalent to putting them in WHERE — the optimiser treats them the same and the result is identical. The distinction only bites with OUTER JOINs, where an ON predicate filters before the join (preserving unmatched rows) and a WHERE predicate filters after (discarding them). Knowing the equivalence holds for INNER but not OUTER is the precise senior point.

This is a pivot in disguise — long to crosstab

GROUP BY f.variant, p.variant turns one row per (experiment, user) into a 2×2 grid of counts — a crosstab. It stays in long format (one row per cell), which is what the spec asks. To rotate it into a true wide grid (pricing variants as columns) you would add SUM(CASE WHEN p.variant = 'A' THEN 1 ELSE 0 END) per column — conditional aggregation — but the long form here is the cleaner answer to the question posed.

The duplicate-enrollment trap — when COUNT(*) lies

COUNT(*) is correct only if each user has at most one row per experiment. If a user were enrolled in feed_v2 twice (re-randomisation, logging bug), the self-join would emit two rows for them and the cell count would over-state real users. When uniqueness is not guaranteed, COUNT(DISTINCT f.user_id) is the safe expression. State the one-row-per-(experiment,user) assumption explicitly — experiment-enrollment tables are a classic place it silently fails.

Dialect notes — fully portable

INNER JOIN, predicates in ON, and GROUP BY on multiple columns are SQL-standard and behave identically across Postgres, SQL Server, MySQL, BigQuery, Snowflake, and Oracle. There is nothing engine-specific in this query. The only stylistic variation is the wide-pivot alternative: Snowflake/SQL Server/Oracle offer a PIVOT operator, but the portable SUM(CASE ...) idiom works everywhere.

Senior signal: express "enrolled in both experiments" as a self-join on user_id with each alias filtered to one experiment — the inner join enforces the intersection — produce the crosstab with a two-column GROUP BY, and use COUNT(DISTINCT user_id) when one-row-per-enrollment is not guaranteed.
