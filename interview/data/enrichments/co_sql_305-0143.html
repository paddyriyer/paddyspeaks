Activision Blizzard's people-analytics team wants the median salary for every job title, not the average — a handful of staff-level outliers drag the mean up and make every band look healthier than it is. Median has no built-in aggregate in standard SQL, so the candidate has to build it by hand from row numbers. That hand-rolled arithmetic is exactly where this question bites: the offset formula has to pick the middle row for odd-sized groups and the two middle rows for even-sized ones, and getting that boundary wrong is silent — the query still returns a number.

1. The source data — employees

Three deliberately different group sizes: Engineer has 4 (even — median is the mean of the two middle values), Analyst has 3 (odd — median is the single middle value, $90K), and Manager has 1 (the median is just that one salary). The odd group is the trap.

2. The SQL — corrected median offsets

WITH ranked AS (
  SELECT job_title, salary,
    ROW_NUMBER() OVER (PARTITION BY job_title ORDER BY salary) AS rn,
    COUNT(*)    OVER (PARTITION BY job_title)                  AS n
  FROM employees
)
SELECT job_title, AVG(salary) AS median_salary
FROM ranked
WHERE rn IN ((n+1)/2, (n+2)/2)
GROUP BY job_title
ORDER BY job_title;

The reference solution shipped a different predicate — rn IN (n/2, n/2 + 1, (n+1)/2) — and it is wrong for odd-sized groups (see the trap below). The corrected predicate above uses the two classic median offsets and, with integer division, collapses to a single row when n is odd.

4. Strategic explanations

The bug in the reference predicate

The shipped predicate rn IN (n/2, n/2 + 1, (n+1)/2) looks plausible but mishandles odd groups. With integer division, for Analyst n = 3: n/2 = 1, n/2+1 = 2, (n+1)/2 = 2 — the set is {1, 2}. It selects rows 1 and 2 (salaries 80K and 90K), averaging to 85K. The real median of {80K, 90K, 110K} is the middle value, 90K. For even groups it happens to land right (Engineer's set is {2, 3}), so the error hides until an odd-sized title appears. The correct predicate is just the two median offsets:

WHERE rn IN ((n+1)/2, (n+2)/2)
-- odd  n=3:  (4/2, 5/2) = (2, 2) -> one row,  the middle
-- even n=4:  (5/2, 6/2) = (2, 3) -> two rows, both middles

The same offsets are the standard hand-rolled median; integer truncation does the odd/even branching for free.

The one-line answer — percentile_cont

Postgres ships an ordered-set aggregate that is the median, interpolation and all:

SELECT job_title,
       percentile_cont(0.5) WITHIN GROUP (ORDER BY salary) AS median_salary
FROM employees
GROUP BY job_title;

Mention this first — it is shorter, correct by construction, and signals you know the modern toolset. The window-function version still matters because many interviewers ask you to build median without the aggregate, and engines like MySQL < 8 lack it.

percentile_cont vs percentile_disc

For even groups they differ. percentile_cont interpolates — Engineer returns 130K, the mean of 120K and 140K, matching AVG of the two middle rows. percentile_disc returns an actual data value — it would return 120K, the lower middle. The hand-rolled AVG approach matches percentile_cont. State which definition the business wants for even-sized bands.

Ties at the median

If two Analysts both earned 90K, ROW_NUMBER still assigns them distinct rn values, so the offset arithmetic is unaffected — the median is still 90K. That robustness is why ROW_NUMBER is the right window function here, not RANK: RANK would give tied rows the same number and break the "pick row k" logic.

NULL salaries

A NULL salary still gets a ROW_NUMBER and is counted by COUNT(*), so it inflates n and shifts the median offset — a phantom employee skewing the band. ORDER BY salary sorts NULLs last by default, so they typically land outside the middle, but the count is still wrong. Filter explicitly with WHERE salary IS NOT NULL in the CTE before ranking.

Senior signal: catch that the reference offset formula double-counts on odd-sized groups, fix it to the two standard median offsets (n+1)/2, (n+2)/2, reach first for percentile_cont(0.5) WITHIN GROUP, and distinguish cont (interpolated) from disc (actual value) while pre-empting the NULL-salary count skew.
