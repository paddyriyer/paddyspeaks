A growth team wants to count users who have not opted in to marketing — per country. The obvious filter WHERE opt_in != 'yes' looks right, but it silently drops everyone who never answered (opt_in IS NULL). The interviewer is probing whether you understand three-valued logic: a WHERE clause keeps a row only when the predicate evaluates to true, and NULL != 'yes' is not true — it is unknown.

1. The source data — users

idcountryopt_in(narrative)
1CAyesopted in
2CAnoexplicitly declined
3CANULLnever answered — the edge case
4FRyesopted in
5FRNULLnever answered
6JPnoexplicitly declined
7JPnoexplicitly declined

Two countries carry a NULL opt_in — CA (id 3) and FR (id 5). Whether those rows are counted as "not opted in" is the entire question. FR is the sharpest case: its only non-yes row is the NULL, so the naive query returns no FR row at all.

2. The SQL — non-opt-in count with IS DISTINCT FROM

SELECT country, COUNT(*) AS c
FROM users
WHERE opt_in IS DISTINCT FROM 'yes'
GROUP BY country
ORDER BY country;

IS DISTINCT FROM is a NULL-safe inequality. It always returns a clean boolean: 'no' IS DISTINCT FROM 'yes' is true, and crucially NULL IS DISTINCT FROM 'yes' is also true (a NULL is distinct from the value 'yes'). So every row that is not literally 'yes' — including the never-answered rows — passes the filter.

3. The resulting analysis

countryc(commentary)
CA2id 2 ('no') + id 3 (NULL)
FR1id 5 (NULL) — the naive query lost this row entirely
JP2id 6 + id 7, both 'no'

Compare the naive opt_in != 'yes': it returns only CA=1 and JP=2 — FR vanishes from the report, and CA undercounts by one. The corrected query restores all three NULL-bearing users.

4. Strategic explanations

Why != 'yes' drops the NULLs

SQL uses three-valued logic: true, false, unknown. Any comparison with NULL yields unknown — NULL != 'yes' is unknown, not true. A WHERE clause admits a row only on true, so the never-answered rows are filtered out. This is not a bug in the engine; it is the standard, and it bites whenever a nullable column appears under a negated predicate.

Why IS DISTINCT FROM is the clean fix

IS DISTINCT FROM treats NULL as just another value for comparison purposes: two NULLs are not distinct, and a NULL is distinct from any non-NULL. It never returns unknown, so it behaves predictably inside WHERE. The portable alternative is WHERE opt_in IS NULL OR opt_in != 'yes' — correct, but verbose and easy to forget. COALESCE(opt_in,'') != 'yes' also works but invents a sentinel; IS DISTINCT FROM states the intent directly.

The mirror trap: = 'yes' and IN

The same logic catches you on the positive side. WHERE opt_in = 'yes' correctly excludes the NULLs (you want only confirmed opt-ins), but WHERE opt_in NOT IN ('yes') behaves like != and drops them. And NOT IN with a subquery that can yield a NULL is famously worse — it can return zero rows. Whenever a column is nullable, audit every negated predicate touching it.

Dialect notes

IS DISTINCT FROM is standard SQL and supported by PostgreSQL, SQLite, and DuckDB. SQL Server lacks it before 2022, where you write EXCEPT-style logic or the explicit OR opt_in IS NULL; SQL Server 2022 adds the shorthand opt_in IS DISTINCT FROM 'yes'. MySQL spells the NULL-safe equality as <=>, so the inequality is NOT (opt_in <=> 'yes'). Same semantics, different surface syntax — mention this to show dialect awareness.

Senior signal: recognise that a negated predicate over a nullable column silently drops NULL rows because NULL != x is unknown, not true; reach for IS DISTINCT FROM as the NULL-safe inequality, know the portable OR ... IS NULL fallback, and audit every != / NOT IN against a nullable column.
