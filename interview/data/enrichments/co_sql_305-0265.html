PayPal's merchant-analytics team ranks products by profit margin to decide which to promote. "Rank by margin" sounds trivial, but the interviewer picked DENSE_RANK specifically — and the dataset almost certainly has a tie and a NULL margin. The real question is whether you know how this window function treats both, because the default behaviour is not what most people assume.

1. The source data — products

Products 1 and 3 are tied at 0.42 — that exercises the "dense" part. Product 5 has a NULL margin (never computed, maybe a new SKU). Where NULL lands in a DESC sort is the trap.

2. The SQL — DENSE_RANK() OVER (ORDER BY profit_margin DESC)

SELECT product_id, profit_margin,
  DENSE_RANK() OVER (ORDER BY profit_margin DESC) AS rk
FROM products;

No PARTITION BY, so the ranking is over the whole table. DENSE_RANK assigns consecutive integers with no gaps after a tie.

Look at the top row: the NULL-margin product ranks #1. In Postgres, ORDER BY … DESC places NULLs first by default — so a product with no known margin outranks every product that actually earns money. The two 0.42 products correctly share rank 2, and the next product is rank 3 (no gap, that is "dense"). But the NULL at the top is almost certainly a bug in the result, not a feature.

4. Strategic explanations

The NULL-ordering trap — the headline issue

SQL's rule: in an ASC sort NULLs come last, in a DESC sort NULLs come first. Nobody wants an unpriced product crowned the most profitable. Fix it explicitly with NULLS LAST:

SELECT product_id, profit_margin,
  DENSE_RANK() OVER (ORDER BY profit_margin DESC NULLS LAST) AS rk
FROM products;
-- now: 0.42 -> 1, 0.42 -> 1, 0.35 -> 2, 0.28 -> 3, NULL -> 4

Even better, decide whether NULL-margin products belong in the ranking at all — often you want WHERE profit_margin IS NOT NULL so they are excluded entirely rather than ranked last.

DENSE_RANK gives 1,1,2,3 — no gap, ideal when rank means "tier" or "Nth distinct value". RANK gives 1,1,3,4 — competition ranking, gap equals the size of the tie group. ROW_NUMBER gives 1,2,3,4 — ignores ties, breaks them arbitrarily. The interviewer's choice of DENSE_RANK signals they want tiers.

The non-deterministic tiebreak follow-up

Products 1 and 3 both get rank 2, but which prints first is undefined — the window only ordered by margin. If the output row order matters (it does for a UI list), add a deterministic tiebreaker:

DENSE_RANK() OVER (ORDER BY profit_margin DESC NULLS LAST)
-- and the outer query:
ORDER BY rk, product_id;

The tiebreaker does not change ranks — both products stay rank 2 — it just makes the listing reproducible.

"Top 3 margins" — the natural extension

Because DENSE_RANK has no gaps, WHERE rk <= 3 reliably means "the three best distinct margins" and returns every product at those margins, ties included. With RANK a tie could push real contenders past the cutoff. Wrap the window in a subquery — you cannot filter on a window function in the same SELECT's WHERE:

SELECT * FROM (
  SELECT product_id, profit_margin,
    DENSE_RANK() OVER (ORDER BY profit_margin DESC NULLS LAST) AS rk
  FROM products
) t WHERE rk <= 3;
Per-category ranking

The schema carries a category column, and "top margin within each category" is the more useful business question. Add PARTITION BY category and the ranking restarts per group — same window, one extra clause.

Senior signal: immediately flag that ORDER BY … DESC sorts NULLs first — so an unpriced product would falsely rank #1 — and fix it with NULLS LAST or a WHERE filter; contrast DENSE_RANK / RANK / ROW_NUMBER by tie behaviour; and add a deterministic tiebreaker for reproducible output.
