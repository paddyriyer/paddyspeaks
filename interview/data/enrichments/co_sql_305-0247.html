Churn rate is the second-most-stared-at SaaS metric after MRR. The CFO asks for it monthly; the board cares about it quarterly; growth teams build their entire roadmap around moving it. The query looks trivial — and it is, until you ask "what counts as churn?" and discover three different valid definitions.

8 subscriptions. 3 churned, 5 still active. ended_at IS NOT NULL = churned.

2. The SQL — conditional aggregation in one pass

SELECT
  100.0 * SUM(CASE WHEN ended_at IS NOT NULL THEN 1 ELSE 0 END)
        / COUNT(*) AS churn_rate
FROM subscriptions;

3 of 8 subscriptions ended → 37.5% lifetime churn. (Lifetime churn is a vanity number. The real metric is per-period — see below.)

4. Strategic explanations

The three definitions of churn — pick one and stick with it
  • Lifetime churn (the query above): of all subscriptions ever, what % ended? Useful for cohort analysis but useless for trend monitoring — the rate keeps climbing as time passes regardless of whether things are getting better or worse.
  • Period churn: of subscriptions active at the start of the period, what % ended during it? This is the one CFOs actually want.
  • Revenue churn: of MRR active at start of period, what % was lost (cancellations + downgrades)? Different number entirely — losing one $10K customer hurts more than losing 100 $50 customers.
The CFO version — monthly churn rate
SELECT
  DATE_TRUNC('month', ended_at) AS churn_month,
  COUNT(*) AS churned_count,
  -- denominator: subscriptions that were active at start of that month
  (SELECT COUNT(*) FROM subscriptions s2
   WHERE s2.started_at < DATE_TRUNC('month', s.ended_at)
     AND (s2.ended_at IS NULL OR s2.ended_at >= DATE_TRUNC('month', s.ended_at))
  ) AS active_at_month_start,
  ROUND(100.0 * COUNT(*) /
    (SELECT COUNT(*) FROM subscriptions s2
     WHERE s2.started_at < DATE_TRUNC('month', s.ended_at)
       AND (s2.ended_at IS NULL OR s2.ended_at >= DATE_TRUNC('month', s.ended_at))
    ), 2) AS monthly_churn_pct
FROM subscriptions s
WHERE ended_at IS NOT NULL
GROUP BY DATE_TRUNC('month', ended_at)
ORDER BY churn_month;

This gives the trend: "churn was 2.1% in Q1, 2.4% in Q2 — slight uptick, investigate."

The "involuntary churn" carve-out

Not all ended_at IS NOT NULL rows are equal. Some subscriptions end because the customer cancelled (voluntary); some end because the credit card expired and dunning failed (involuntary). The remediation is different — voluntary needs product fixes, involuntary needs payment retry logic. Production schema typically has ended_reason and the query branches:

SELECT
  100.0 * COUNT(*) FILTER (WHERE ended_reason = 'cancelled')   / COUNT(*) AS voluntary_churn_pct,
  100.0 * COUNT(*) FILTER (WHERE ended_reason = 'payment_fail') / COUNT(*) AS involuntary_churn_pct
FROM subscriptions;
"Active customers" snapshot — the denominator question

Many engineers compute churn as "churned this month / active this month" — but "active this month" is ambiguous. Active at start? End? Average? The cleanest answer is "active at start of period":

WHERE started_at < '2026-01-01'                              -- existed before period
  AND (ended_at IS NULL OR ended_at >= '2026-01-01')         -- still active at start
The Net Revenue Retention (NRR) follow-up

The metric that actually correlates with valuation: (starting MRR + expansion − contraction − churn) / starting MRR. NRR > 100% means existing-customer growth offsets churn — the SaaS holy grail. Senior candidates volunteer NRR as the "next-level" metric beyond simple churn rate.

Senior signal: name the three definitions of churn (lifetime / period / revenue), default to the period version, distinguish voluntary vs involuntary, and pre-empt the NRR follow-up. The "denominator = active-at-start" precision is the senior detail.
