AMD's resourcing team wants a staffing roster: each employee, the department they sit in, and the project they are assigned to. The data lives across four tables joined through a project_assignments bridge. The interviewer wants to see clean aliasing, the right join path, and awareness that an INNER JOIN chain silently drops anyone with a gap.

1. The source data — employees, departments, projects, project_assignments

The roster threads four tables: employees → departments (on dept_id), and employees → project_assignments → projects (on emp_id then project_id). Every employee here has both a department and an assignment — the interesting case is what happens when one is missing.

2. The SQL — four-table join via the bridge

SELECT e.name, d.dept_name, p.project_name
FROM employees AS e
JOIN departments AS d         ON e.dept_id = d.dept_id
JOIN project_assignments AS pa ON e.emp_id = pa.emp_id
JOIN projects AS p            ON pa.project_id = p.project_id;

Each JOIN adds one table and one ON condition. The project_assignments bridge is the key move: an employee and a project have no direct foreign key, so the join must pass through the assignment table that links them.

4. Strategic explanations

Aliases are not cosmetic — this schema has duplicate column names

Look at the schema: dept_name appears in employees, departments, projects, and project_assignments; name and project_name are similarly scattered. Without table aliases, SELECT name, dept_name raises "column reference is ambiguous". The aliases e, d, pa, p let every reference be unqualified-free: e.name, d.dept_name. Always qualify columns in a multi-table join — it is correctness, not style.

The INNER JOIN drop — the silent-data-loss trap

Every join here is an inner join, so a row survives only if it matches in all four tables. An employee with no row in project_assignments — on the bench, between projects — vanishes from the roster entirely. So does one whose dept_id points at a department that was deleted. If the roster must list everyone, switch to LEFT JOIN from employees:

SELECT e.name, d.dept_name, p.project_name
FROM employees AS e
LEFT JOIN departments AS d          ON e.dept_id = d.dept_id
LEFT JOIN project_assignments AS pa ON e.emp_id = pa.emp_id
LEFT JOIN projects AS p             ON pa.project_id = p.project_id;

Unassigned employees then appear with NULL project columns. Ask: "should the roster include people not on any project?" The answer picks inner vs left.

The bridge table fans out rows — one employee, many rows

project_assignments is many-to-many: if Aarav is staffed on two projects, he gets two rows in the output, one per assignment. That is correct for a staffing roster, but it means COUNT(*) on this result counts assignments, not employees. If a downstream metric needs distinct headcount, use COUNT(DISTINCT e.emp_id) — the fan-out is a classic source of inflated counts.

Join order does not change the result — but watch the path

Inner joins are associative and commutative, so the planner is free to reorder them; writing departments first would give the identical answer. What you cannot do is join projects directly to employees — there is no shared key. The join graph must stay connected: each new table needs an ON condition tying it to a table already in the join. Omit one (a missing ON, or ON against the wrong column) and you get an accidental cross join — a row explosion.

Trusting the dept_id path — two ways to get department

Notice projects also carries a dept_id. So an employee's department could be read two ways: the department they report to (employees.dept_id), or the department that owns their project (projects.dept_id). These can disagree — an Engineering employee loaned to a Design-owned project. The solution uses employees.dept_id; confirm that is the department the roster means, and do not silently mix the two.

Senior signal: qualify every column with a table alias because the schema has many duplicate names, recognise the INNER JOIN chain drops employees with any missing link (LEFT JOIN if the roster must be complete), know the many-to-many bridge fans out rows so headcount needs COUNT(DISTINCT), and resolve which dept_id path is the intended one.
