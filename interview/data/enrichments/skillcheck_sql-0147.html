A dependency or routing tool needs reachability: from node 1, which nodes can you get to, and what is the shortest hop count to each? The graph is directed and may contain cycles, so a naive recursive traversal would loop forever. The interviewer is probing whether you know the cycle-safe pattern — carry the visited path in an array and refuse to revisit — and whether you compute the minimum hops rather than just any path length.

2. The SQL — recursive walk with a visited-path array

WITH RECURSIVE walk(node, hops, path) AS (
  SELECT 1, 0, ARRAY[1]
  UNION ALL
  SELECT e.dst, w.hops + 1, w.path || e.dst
  FROM walk w JOIN edge e ON e.src = w.node
  WHERE NOT (e.dst = ANY(w.path))
)
SELECT node, MIN(hops) AS min_hops
FROM walk
WHERE node <> 1
GROUP BY node
ORDER BY node;

Each walk row carries the path array of nodes visited so far. The recursive arm follows an out-edge only if e.dst is not already in path — that guard is what breaks cycles. The traversal still enumerates multiple distinct paths to a node, so the outer MIN(hops) collapses them to the shortest.

4. Strategic explanations

The visited-path array is the cycle breaker

A plain recursive CTE on a cyclic graph never stops: 1→2→4→5→2→4… produces rows forever. Carrying path — the ordered list of nodes on the current branch — and filtering WHERE NOT (e.dst = ANY(w.path)) means a branch refuses to re-enter a node it has already stepped on. Each branch can therefore visit each node at most once, so its length is bounded by the node count and the recursion is guaranteed finite.

Per-path memory, not global — why MIN is still needed

The array tracks visited nodes for that branch only; it is not a global "seen" set. So node 4 is still discovered twice — once down the 1→2 branch, once down 1→3 — because neither branch's path contained 4 beforehand. That is intentional: a global visited set would prune some paths before their hop count was known and could miss the shortest one. The recursion enumerates all simple paths; the outer GROUP BY node + MIN(hops) then picks the shortest reach per node.

= ANY(array) and array append

e.dst = ANY(w.path) tests membership of a scalar in a Postgres array; w.path || e.dst appends an element. The anchor seeds ARRAY[1] so node 1 itself is "visited" from the start — that is what stops the 5→2… and 7→3… cycles from ever re-deriving node 1 and is why the final WHERE node <> 1 is about output, not termination. These are array operations; in engines without native arrays you would carry a delimited string and use LIKE, or use the CYCLE clause below.

This is BFS-by-result, not by traversal

A recursive CTE explores depth-first-ish, not breadth-first, so it does not naturally find shortest paths — it finds all simple paths and you reduce. For genuinely large graphs that is exponential in the worst case. Real shortest-path work belongs in a graph engine or an iterative algorithm; recursive-CTE reachability is correct and interview-appropriate but does not scale like Dijkstra. Naming that ceiling is the senior move.

Dialect — the CYCLE clause

Standard SQL and Postgres 14+ offer UNION ALL … CYCLE node SET is_cycle USING path_col, which builds and checks the visited array for you — terser and less error-prone than the manual ARRAY + ANY guard. SQL Server has no CYCLE clause and needs the manual pattern (often with MAXRECURSION as a safety cap); Oracle's CONNECT BY has NOCYCLE and CONNECT_BY_ISCYCLE. The manual array shown here is the most portable and the clearest to explain in an interview.

Senior signal: make recursive graph traversal cycle-safe by carrying the visited path array and filtering NOT (dst = ANY(path)), understand the array is per-branch so multiple paths still surface and an outer MIN(hops) is required, and acknowledge that recursive-CTE reachability enumerates all simple paths — fine for interviews, not a substitute for a real shortest-path algorithm at scale.
