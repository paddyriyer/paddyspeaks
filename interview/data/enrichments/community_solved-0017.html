A Community question modelled on a classic Microsoft star-schema problem: total downloads by date, split into paying and non-paying users, returning only the dates where non-payers out-downloaded payers. The catch is that ms_download_facts carries a user_id but the paying_customer flag lives two hops away on the account dimension — so the real test is whether you can navigate the snowflake without losing or duplicating fact rows.

1. The source data

ms_download_facts (the fact table):

ms_user_dimension maps user→account: users 1,2→acc 100; user 3→acc 200; users 4,5→acc 300. ms_acc_dimension holds the flag: acc 100 = yes, accs 200 and 300 = no. The flag is one join removed from the fact — that indirection is the question.

2. The SQL — fact → user dim → acc dim, conditional sums

WITH t1 AS (
    SELECT mdf.date, mdf.user_id, mdf.downloads, mad.acc_id, mad.paying_customer
    FROM ms_download_facts mdf
    LEFT JOIN ms_user_dimension mud ON mdf.user_id = mud.user_id
    LEFT JOIN ms_acc_dimension  mad ON mad.acc_id  = mud.acc_id
)
SELECT date,
       SUM(CASE WHEN paying_customer = 'no'  THEN NULLIF(regexp_replace(downloads::text, '[^-.0-9]', '', 'g'), '')::numeric ELSE 0 END) AS non_paying,
       SUM(CASE WHEN paying_customer = 'yes' THEN NULLIF(regexp_replace(downloads::text, '[^-.0-9]', '', 'g'), '')::numeric ELSE 0 END) AS paying
FROM t1
GROUP BY date
HAVING SUM(CASE WHEN paying_customer = 'no'  THEN NULLIF(regexp_replace(downloads::text, '[^-.0-9]', '', 'g'), '')::numeric ELSE 0 END)
     > SUM(CASE WHEN paying_customer = 'yes' THEN NULLIF(regexp_replace(downloads::text, '[^-.0-9]', '', 'g'), '')::numeric ELSE 0 END)
ORDER BY date ASC;

The CTE walks the snowflake. The regexp_replace scrubs any stray non-numeric characters from downloads before casting — defensive against a column that arrived as text.

4. Strategic explanations

Why join through the user dimension, not straight to accounts

The fact has user_id but no acc_id. There is no direct path to paying_customer — you must hop fact→ms_user_dimension to pick up acc_id, then acc_id→ms_acc_dimension for the flag. Skipping the bridge table is the most common wrong answer here. This two-step hop is the definition of a snowflake schema, and naming it shows you read the data model rather than pattern-matching one join.

Conditional aggregation beats two grouped subqueries

A junior instinct is to compute paying and non-paying totals separately and join them by date. The SUM(CASE WHEN …) pivot does it in a single GROUP BY — one scan, no join, and dates with only one customer type still appear (the missing side sums to 0, not NULL, thanks to the ELSE 0). That ELSE 0 is load-bearing: drop it and a one-sided date yields NULL, which then loses the HAVING comparison.

Why HAVING, not WHERE, for the comparison

The condition "non-paying total > paying total" is a statement about an aggregate, which does not exist until after GROUP BY. WHERE runs per-row, before grouping, and cannot see SUM(…). HAVING runs after. A cleaner-reading variant wraps the aggregation in a CTE and filters with a plain WHERE on the alias:

WITH agg AS (SELECT date, SUM(...) AS non_paying, SUM(...) AS paying FROM t1 GROUP BY date)
SELECT * FROM agg WHERE non_paying > paying ORDER BY date;
The LEFT JOIN orphan trap

The joins are LEFT, so a fact row whose user_id is missing from the dimension survives with paying_customer = NULL. That row matches neither 'yes' nor 'no', so it silently contributes 0 to both sums — its downloads vanish from the totals without error. Decide deliberately: is an unmapped user a data bug to surface, or downloads to drop? An INNER JOIN would discard them too, but at least visibly.

The regexp_replace tell — and the strict-string trap

Scrubbing downloads with a regex implies the column is text that should be numeric. If it is already a clean integer, the regex is harmless overhead. But watch the flag comparison: paying_customer = 'yes' is case- and whitespace-sensitive — 'Yes' or 'yes ' would match nothing. Normalise with lower(trim(paying_customer)) if the source is dirty.

Senior signal: recognise the snowflake and route fact→user→account rather than guessing a direct join, use single-pass conditional aggregation with an explicit ELSE 0, place the aggregate comparison in HAVING, and call out that LEFT JOIN orphans drop downloads into a silent NULL bucket.
