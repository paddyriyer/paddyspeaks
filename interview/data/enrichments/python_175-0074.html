Amazon ingests logs from dozens of services — application logs, Apache access logs, syslog, custom formats — each stamping time differently. "Extract and Normalize Timestamps from Multi-Format Log File" tests real data-engineering judgement: a regex to locate the timestamp, a fallback chain to parse it across formats, and graceful degradation when nothing matches. It is a parsing problem disguised as a string problem.

1. The input data — a log file

2024-03-01 09:15:42 INFO  user login ok
192.168.0.5 - - [01/Mar/2024:09:16:03 +0000] "GET /api"
2024-03-01T09:17:30Z WARN  retry scheduled
Mar  1 09:18:11 host sshd: connection closed
service started, no timestamp here

Four distinct timestamp dialects plus one line with none — the function must handle all five gracefully.

2. The approach

TS_RE is a single broad regex that locates an ISO-ish timestamp anywhere in a line — it tolerates - or / date separators, T or space before the time, optional fractional seconds ([.,]\d+), and an optional offset (+00:00 or Z). Locating is separate from parsing.

parse_ts then tries each entry in FORMATS in order with datetime.strptime, catching ValueError and falling through to the next — a classic try-each-pattern chain. The first format that parses wins; if all fail it returns None. normalize_log streams the file line-by-line (no full load into memory), runs the regex, parses the captured token, and emits {'iso': ..., 'raw': ...} — keeping the original line so nothing is lost when parsing fails.

3. The output

raw (truncated)iso
2024-03-01 09:15:42 INFO...2024-03-01T09:15:42
192.168.0.5 - - [01/Mar/2024:09:16:03 +0000]...None
2024-03-01T09:17:30Z WARN...2024-03-01T09:17:30+00:00
Mar 1 09:18:11 host sshd...None
service started, no timestamp hereNone

Only two lines fully normalize. The Apache and syslog lines yield None because TS_RE requires a numeric \d{4} year and \d{2} month — it never matches 01/Mar/2024 or Mar 1, so their FORMATS entries are unreachable through this regex. That mismatch between the regex and the format list is the question's hidden bug.

4. Strategic explanations

Complexity

O(L · F) where L is the number of lines and F the number of formats — each line tries up to F strptime calls. Memory is O(1) streaming for the file plus O(L) for the result list. The regex match itself is linear in line length.

The idiomatic alternative — dateutil or datetime.fromisoformat
from dateutil import parser
ts = parser.parse(token, fuzzy=True)   # handles most formats, no FORMATS list

dateutil.parser auto-detects almost every dialect, and Python 3.11+ datetime.fromisoformat finally accepts Z and offsets natively. A senior names these but also notes the tradeoff: dateutil is permissive to a fault and a hard-coded FORMATS list is more predictable for a known set of services.

The edge case — the regex/format mismatch and naive vs aware datetimes

As traced above, Apache and syslog timestamps are caught by FORMATS but never reach it — the regex gates them out. Fixing this means broadening TS_RE with month-name and bracketed-date alternatives. Separately, isoformat() output is aware (has an offset) when the source had one and naive otherwise — mixing the two in later comparisons raises TypeError. Syslog has no year at all, so even when parsed it defaults to year 1900.

The follow-up — normalize everything to UTC

"Convert all timestamps to UTC so they're comparable." That means attaching tzinfo to naive values (you must assume a source zone) and calling .astimezone(timezone.utc) on aware ones — the interviewer is probing whether you understand that a naive timestamp carries no zone and the assumption must be explicit.

The trap — one mega-regex with capture groups per format

Trying to parse and validate every format in a single monster regex is unmaintainable and brittle — each new log source means surgery on a 300-character pattern. The locate-then-try-parse split is the robust design: a loose regex to find a candidate, then a clean ordered list of strptime formats (or dateutil) to interpret it.

Senior signal: separate locating a timestamp from parsing it, stream the file instead of loading it, preserve the raw line for unparseable rows, and spot that the regex and the FORMATS list disagree — then push to normalize everything to UTC so the output is actually usable.
