A subscriptions team wants the trial-conversion breakdown: of everyone who started on a trial, what plan did they move to next — basic, pro, churn — and how many never converted at all? The subs table is an append-only log of plan changes. The interviewer is probing whether you can use LEAD to find each customer's next plan and, critically, whether you handle the customer whose trial has no next row.

The edge case is customer 5: a single trial row with nothing after it. Their LEAD returns NULL, and that NULL must be reported as the explicit bucket never_converted — not dropped, not left blank.

2. The SQL — LEAD to find the next plan, COALESCE the NULL bucket

WITH journeys AS (
  SELECT customer_id, plan,
         LEAD(plan) OVER (PARTITION BY customer_id ORDER BY start_date) AS next_plan
  FROM subs
)
SELECT COALESCE(next_plan, 'never_converted') AS next_plan,
       COUNT(*) AS customer_count
FROM journeys
WHERE plan = 'trial'
GROUP BY COALESCE(next_plan, 'never_converted')
ORDER BY customer_count DESC, next_plan;

LEAD(plan) looks one row forward in each customer's date-ordered timeline to find the plan they moved to. The outer query keeps only the trial rows, replaces a NULL next plan with the literal never_converted, and counts customers per destination.

4. Strategic explanations

Why LEAD — the next event is the transition target

A state-transition question asks "from X, where next?" In an append-only log, the answer is literally the next row for that customer. LEAD(plan) OVER (PARTITION BY customer_id ORDER BY start_date) reaches that row directly — no self-join, no correlated subquery. Pairing each row with its successor is the canonical pattern for transition and conversion analysis.

The NULL is the answer, not noise — COALESCE twice

For customer 5's only row, LEAD returns NULL — and that NULL means "no conversion happened", a result the business explicitly wants counted. COALESCE(next_plan, 'never_converted') turns it into a first-class bucket. It must appear in both the SELECT and the GROUP BY: grouping on the raw next_plan would create an unlabelled NULL group, and the projected label would not match the grouping key.

Filter on plan = 'trial' after the window, not before

LEAD must see every row to compute the successor correctly — if you filtered to plan = 'trial' before the window function, LEAD would look forward to the next trial row (skipping basic, pro, etc.) or find nothing. The window runs over the full log inside the CTE; the WHERE plan = 'trial' in the outer query then restricts to the transitions that start from a trial. Filter placement relative to the window is the subtle correctness point.

Counting customers vs counting rows

COUNT(*) is correct here only because each customer has exactly one trial row, so one row equals one customer. If a customer could re-enter trial (trial → churn → trial → basic), COUNT(*) would count two trial transitions for one person. If the metric is truly "how many customers", COUNT(DISTINCT customer_id) is the safer expression. State the assumption that trials are non-recurring.

Dialect notes — broadly portable

LEAD, COALESCE, and GROUP BY on an expression are SQL-standard and behave identically in Postgres, SQL Server, BigQuery, Snowflake, Oracle, and MySQL 8+/SQLite 3.25+. One minor portability note: some engines let you GROUP BY a SELECT-list alias, but repeating the COALESCE expression in the GROUP BY (as here) is the universally safe form.

Senior signal: model state transitions by pairing each row with its successor via LEAD, run the window over the full log and filter to the source state afterward, and treat the NULL from a missing next row as a meaningful never_converted bucket with COALESCE in both SELECT and GROUP BY.
