A lifecycle analyst needs the classic signup → confirm → purchase funnel for the October 2025 cohort, reported as conversion percentages. The subtle part: every downstream event must occur after the user's signup, and a purchase only counts if the user also confirmed first. The interviewer is checking whether you respect event ordering and cohort boundaries instead of naively counting "did this user ever confirm".

The edge cases are deliberate booby-traps. User 6 signed up in September and must be excluded from the cohort. User 5 has a confirm timestamp earlier than their signup — a data anomaly that an after-signup filter must reject. User 2 has a purchase before signup, which also must not count. Users 1, 2, 3 confirm correctly (after signup); only user 1 then purchases after confirming.

2. The SQL — ordered-event funnel with cohort fencing

WITH oct_signups AS (
  SELECT user_id, signed_up_at
  FROM signup_event
  WHERE signed_up_at >= TIMESTAMP '2025-10-01' AND signed_up_at < TIMESTAMP '2025-11-01'
),
confirmed AS (
  SELECT DISTINCT s.user_id
  FROM oct_signups s
  JOIN confirm_event c ON c.user_id = s.user_id AND c.confirmed_at > s.signed_up_at
),
purchased AS (
  SELECT DISTINCT s.user_id
  FROM oct_signups s
  JOIN confirm_event c ON c.user_id = s.user_id AND c.confirmed_at > s.signed_up_at
  JOIN purchase_event p ON p.user_id = s.user_id AND p.purchased_at > c.confirmed_at
)
SELECT (SELECT COUNT(*) FROM oct_signups)::int AS signups,
       ROUND(100.0 * (SELECT COUNT(*) FROM confirmed)
             / NULLIF((SELECT COUNT(*) FROM oct_signups),0), 2) AS confirm_pct,
       ROUND(100.0 * (SELECT COUNT(*) FROM purchased)
             / NULLIF((SELECT COUNT(*) FROM oct_signups),0), 2) AS purchase_pct;

Each funnel stage joins on user_id and a strict time inequality. DISTINCT collapses users who have several confirm or purchase rows to one. The final row divides each stage's headcount by the cohort size.

4. Strategic explanations

The funnel is built on ordered joins, not membership joins

A weak answer joins on user_id alone — "did this user ever confirm?". The question demands causality: confirm must follow signup, purchase must follow confirm. Encoding c.confirmed_at > s.signed_up_at in the join condition is what rejects user 5's anomalous early confirm. The funnel's correctness lives in those inequalities; drop them and you count noise as conversion.

Why purchased re-joins confirm_event

purchase_pct is defined as users who confirmed and then purchased — not all signups who purchased. So the purchased CTE must pass through the confirm stage too, and p.purchased_at > c.confirmed_at enforces the second ordering. A purchase that precedes the confirm does not count. Re-deriving the chain from scratch is clearer than trying to reuse confirmed and bolt purchases on.

DISTINCT guards against fan-out

If a user confirmed twice (or has multiple purchase rows), the join produces multiple rows for that user, and a bare COUNT(*) would over-count the funnel stage — conversion could exceed 100%. SELECT DISTINCT user_id reduces each CTE to one row per user, so the counts are genuine headcounts. This is the most common funnel bug: counting events when the metric is users.

The cohort fence: half-open interval beats BETWEEN on timestamps

signed_up_at >= '2025-10-01' AND signed_up_at < '2025-11-01' is a half-open range — it captures every instant in October including 23:59:59.999 and excludes November cleanly. BETWEEN '2025-10-01' AND '2025-10-31' is the classic trap: 2025-10-31 as a timestamp is midnight, so all of October 31st's daytime activity is silently dropped. For timestamp ranges, always use >= start AND < next_start.

NULLIF on the denominator and the 100.0 literal

NULLIF((SELECT COUNT(*) FROM oct_signups), 0) turns a zero cohort into NULL, so the division yields NULL instead of a divide-by-zero error — the report degrades gracefully for an empty month. And the multiplier is 100.0, not 100: a numeric literal forces floating-point division. With integer 100 and integer counts, 3/5 would truncate to 0 and every percentage would collapse to 0.00 or 100.00.

Senior signal: build funnels with ordered joins (confirmed_at > signed_up_at, purchased_at > confirmed_at) not bare membership joins, DISTINCT each stage to count users not events, fence the cohort with a half-open timestamp range, and force float division with 100.0 while guarding the denominator with NULLIF.
