Airbnb's growth-analytics team needs the simplest engagement metric there is — daily active users, the count of distinct users who did something each day. The query is one GROUP BY with a COUNT(DISTINCT). It is genuinely a one-liner, and the interviewer knows it: the test is whether you remember the DISTINCT, and whether you handle the days that produced no activity at all.

1. The source data — activity

On 05-01, user 1 has two rows — two actions, but one active user. On 05-03, user 2 also appears twice. Those duplicate rows are the trap: COUNT(*) would inflate the metric; only COUNT(DISTINCT user_id) is right.

2. The SQL — COUNT(DISTINCT) GROUP BY date

SELECT action_date, COUNT(DISTINCT user_id) AS dau
FROM activity
GROUP BY action_date;

GROUP BY action_date forms one bucket per day; COUNT(DISTINCT user_id) counts the unique users in each bucket — collapsing a user's repeated actions into a single contribution.

05-01 shows dau = 2 — users 1 and 2 — even though there were three action rows; user 1's two actions counted once. 05-03 shows 3 (users 1, 2, 4) from four rows. With plain COUNT(*) these would read 3 and 4 — the inflated, wrong numbers.

4. Strategic explanations

The COUNT(*) vs COUNT(DISTINCT) trap

This is the entire question. COUNT(*) counts events; DAU counts users. They diverge the instant a user acts twice in a day — which is the normal case. If someone hands you COUNT(*) for an "active users" metric, that is a bug, not a style choice. The only time they agree is when the table is guaranteed one row per user per day.

Zero-activity days disappear — the gap problem

GROUP BY only produces rows for dates that exist in activity. A day with no activity at all simply has no row — dau is not 0, it is missing. A line chart then connects across the gap and hides the outage. To get an honest 0, left-join a date spine:

SELECT d.day AS action_date,
       COUNT(DISTINCT a.user_id) AS dau
FROM generate_series(DATE '2026-05-01', DATE '2026-05-07', INTERVAL '1 day') AS d(day)
LEFT JOIN activity a ON a.action_date = d.day
GROUP BY d.day
ORDER BY d.day;

COUNT(DISTINCT a.user_id) over an all-NULL left-join group is correctly 0 — COUNT ignores NULLs. That is the right way to show a dead day.

Timestamp granularity — truncate before grouping

If action_date were really a TIMESTAMP, grouping on it raw would create one bucket per second, not per day — the metric would be nonsense. Truncate first: GROUP BY date_trunc('day', action_ts) or action_ts::date. And truncation is timezone-sensitive — "day" in UTC versus the user's local time can shift activity across the midnight boundary; pin the timezone explicitly.

The natural follow-ups — MAU and stickiness

DAU rarely ships alone. Interviewers extend to monthly active users and the DAU/MAU "stickiness" ratio. MAU needs a rolling 30-day distinct count, which a simple GROUP BY cannot do — it requires a self-join or a window over a date spine. Mentioning that DAU is the easy member of a family, and MAU is the hard one, signals range.

Senior signal: insist on COUNT(DISTINCT user_id) because DAU counts users not events, left-join a date spine so zero-activity days report 0 instead of vanishing, truncate timestamps to a day (timezone-aware) before grouping, and know that MAU and DAU/MAU stickiness are the harder follow-ups that need windows, not a plain group-by.
