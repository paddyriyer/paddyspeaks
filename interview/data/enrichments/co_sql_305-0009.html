A retail analyst at a multi-category store opens the weekly stand-up deck: "give me each category's sales broken out by day — one column per day, one row per category." That shape — categories down the side, days across the top — is a pivot: you are rotating distinct row values (the dates) up into column headers. The interviewer is checking whether you can express a pivot in portable SQL, because most engines have no universal PIVOT keyword.

1. The source data — sales

categorysale_dateamount(narrative)
Electronics2026-05-11150Monday
Electronics2026-05-12200Tuesday
Electronics2026-05-13100Wednesday
Clothing2026-05-1180Monday
Clothing2026-05-13120Wednesday
Clothing2026-05-1350Wednesday — second entry, same day

Two details decide the interview. Clothing has no Tuesday row at all, and Clothing has two Wednesday rows. A correct pivot must handle both — the missing cell and the colliding cells.

2. The SQL — pivot via conditional aggregation

SELECT
  category,
  SUM(CASE WHEN sale_date = '2026-05-11' THEN amount ELSE 0 END) AS monday_sales,
  SUM(CASE WHEN sale_date = '2026-05-12' THEN amount ELSE 0 END) AS tuesday_sales,
  SUM(CASE WHEN sale_date = '2026-05-13' THEN amount ELSE 0 END) AS wednesday_sales
FROM sales
GROUP BY category
ORDER BY category;

One CASE per target column: it lets a row's amount through only when that row belongs to the column's date, and contributes 0 otherwise. GROUP BY category collapses every category's rows into a single output row, and SUM adds up whatever each CASE let through.

3. The resulting analysis

Clothing800170no Tuesday → 0; two Wednesday rows folded together
Electronics150200100one row per day, clean

Clothing's Wednesday cell is 170, not 120 or 50: the two same-day rows (120 + 50) both satisfy the Wednesday CASE and SUM adds them. Tuesday is 0 because no Clothing row matches and ELSE 0 supplies the blank.

4. Strategic explanations

Why this is a pivot and a plain GROUP BY is not

A plain SELECT category, sale_date, SUM(amount) … GROUP BY category, sale_date returns the same numbers as rows — long format, one row per category/day pair. A pivot turns the sale_date values into column headers: it is a rotation, not just an aggregation. The conditional-aggregation idiom — one SUM(CASE WHEN …) per column — is the universal way to do that, because CASE works in every SQL engine.

ELSE 0 vs no ELSE — a deliberate choice

With ELSE 0, a category with no sales on a day shows 0. Drop the ELSE and non-matching rows become NULL; SUM ignores NULLs, so a category that sold nothing yields NULL rather than 0. Both are valid — choose by intent. A financial grid usually wants 0 (the sale genuinely was zero); a coverage report may want NULL to keep "no data" distinct from a real zero.

Duplicate rows fold in — by design, not a bug

Clothing's two Wednesday rows are not an error to dedupe; SUM over the group adds every matching row, so 120 + 50 = 170. If the business rule were "how many selling days" you would switch to COUNT(DISTINCT sale_date); if the duplicates were genuine data-quality noise you would dedupe in a CTE first. The aggregate you pick is the business rule — state which one you mean.

The hard limit: the columns are static

The three dates are hard-coded in the query. Sell something on 2026-05-14 and it lands in no column — it silently vanishes from the report. Pure SQL cannot grow columns from data: a truly dynamic pivot needs a procedural step (build the SQL text from SELECT DISTINCT sale_date, then execute it) or PostgreSQL's crosstab() from the tablefunc extension. Naming this limitation unprompted is the senior signal.

Dialect — the dedicated PIVOT operator

Snowflake, Oracle, and SQL Server ship a PIVOT clause that is terser than the CASE form — e.g. Snowflake: SELECT * FROM sales PIVOT(SUM(amount) FOR sale_date IN ('2026-05-11','2026-05-12','2026-05-13')). PostgreSQL, MySQL, and SQLite have no PIVOT keyword, so the SUM(CASE) idiom shown here is the portable answer — and it still makes you enumerate the values, exactly like that explicit IN list.

Senior signal: recognise a pivot as rotating row values into columns, implement it portably with one SUM(CASE WHEN …) per column, choose ELSE 0 vs NULL deliberately, know that SUM folds duplicate cells together by design, and call out unprompted that the column list is static — a data-driven pivot needs dynamic SQL or crosstab().
