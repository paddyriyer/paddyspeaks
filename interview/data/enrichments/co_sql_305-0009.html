Google Ads finance opens the daily reconciliation: "per day, what was the gross sale dollars and the txn count?" — the negative rows are refunds and shouldn't pollute the gross. The query is conditional aggregation: a CASE inside SUM that filters at row level, paired with COUNT(*) for total volume. This is the spreadsheet-pivot pattern translated to SQL.

2. The SQL — conditional aggregation with CASE

SELECT sale_date,
  SUM(CASE WHEN amount > 0 THEN amount END) AS positive_sales,
  COUNT(*) AS txn_count
FROM sales
GROUP BY sale_date
ORDER BY sale_date;

4. Strategic explanations

Why CASE with no ELSE works

CASE WHEN amount > 0 THEN amount END — with no ELSE, non-matching rows return NULL. SUM ignores NULLs (unlike COUNT(*), which counts every row). So the negative rows are filtered out of SUM but still counted in COUNT(*). This is the cleanest "filter inside aggregate" idiom in SQL.

FILTER (WHERE ...) — the modern, more readable form

PostgreSQL, BigQuery, Snowflake, and SQLite 3.30+ support FILTER:

SELECT sale_date,
  SUM(amount)         FILTER (WHERE amount > 0) AS positive_sales,
  SUM(amount)         FILTER (WHERE amount < 0) AS refunds,
  COUNT(*)                                      AS txn_count,
  COUNT(*)            FILTER (WHERE amount > 0) AS sale_count,
  COUNT(*)            FILTER (WHERE amount < 0) AS refund_count
FROM sales GROUP BY sale_date;

Same semantics, no nested CASE. MySQL still doesn't support FILTER — fall back to CASE there.

The "true pivot" — one column per channel/category

Google's actual finance ask is usually: "per day, columns for ads, search, cloud, hardware." That's a real pivot — one column per category value:

SELECT sale_date,
  SUM(CASE WHEN channel = 'ads'      THEN amount END) AS ads,
  SUM(CASE WHEN channel = 'search'   THEN amount END) AS search,
  SUM(CASE WHEN channel = 'cloud'    THEN amount END) AS cloud,
  SUM(CASE WHEN channel = 'hardware' THEN amount END) AS hardware
FROM sales GROUP BY sale_date;

Snowflake, Oracle, and SQL Server have a dedicated PIVOT operator; in Postgres / SQLite / BigQuery the conditional-aggregation idiom is the universal pattern. Either way, you must enumerate the categories at query-time — truly dynamic pivoting needs a procedural step (or crosstab in Postgres).

The unknown-categories trap

What happens when Google adds a new "YouTube Premium" channel and nobody updates the SQL? Those rows fall into none of the CASE branches and silently disappear from the pivot. Production-grade defenses:

  • Keep an other bucket: SUM(CASE WHEN channel NOT IN ('ads','search','cloud','hardware') THEN amount END) AS other — if it's ever non-zero, alert.
  • A row-level invariant: SUM(amount) = ads + search + cloud + hardware + other. If that fails, the categorization has a hole.
The unpivot follow-up — flip a wide table into long

Sometimes the source is the pivoted version (one row per day, columns per channel) and the consumer needs the long format. UNION ALL is the universal unpivot:

SELECT sale_date, 'ads' AS channel, ads AS amount FROM daily_pivot
UNION ALL
SELECT sale_date, 'search', search FROM daily_pivot
UNION ALL
SELECT sale_date, 'cloud',  cloud  FROM daily_pivot
UNION ALL
SELECT sale_date, 'hardware', hardware FROM daily_pivot;
The ratio follow-up — gross / net / refund-rate in one query
SELECT sale_date,
  SUM(CASE WHEN amount > 0 THEN amount END) AS gross,
  SUM(amount) AS net,
  -1.0 * SUM(CASE WHEN amount < 0 THEN amount END) /
   NULLIF(SUM(CASE WHEN amount > 0 THEN amount END), 0) AS refund_rate
FROM sales GROUP BY sale_date;
Senior signal: name SUM(CASE) as conditional aggregation, contrast it cleanly with FILTER (WHERE), and bring up the unknown-category trap unprompted with a defensive "other" bucket. The pivot ↔ unpivot pair plus the refund-rate ratio is the over-the-bar finish.
