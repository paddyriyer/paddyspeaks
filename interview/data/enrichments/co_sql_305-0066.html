SAP's commerce-cloud team runs a Monday morning revenue review, and the slide everyone looks at first is "this week vs last week". It sounds trivial — two sums — but the whole question lives in how you draw the two seven-day windows. Off by one day and the boundary sale lands in the wrong bucket, or in both, or in neither. An interviewer asking this is checking whether you can define a date range that is exactly seven days, with no gap and no overlap.

1. The source data — sales

CURRENT_DATE − 9300
CURRENT_DATE − 12175
CURRENT_DATE − 20500
CURRENT_DATE − 860

The interesting rows are the boundaries. The sale at CURRENT_DATE − 8 sits exactly on the edge between the two windows, and the sale at CURRENT_DATE − 20 is old enough to fall outside both. If the windows are drawn sloppily, the −8 sale double-counts or vanishes.

2. The SQL — conditional aggregation

SELECT
  SUM(CASE WHEN sale_date >= (CURRENT_DATE + INTERVAL '-7 days') THEN amount END) AS this_week,
  SUM(CASE WHEN sale_date BETWEEN (CURRENT_DATE + INTERVAL '-14 days') AND (CURRENT_DATE + INTERVAL '-8 days') THEN amount END) AS last_week
FROM sales;

Two CASE expressions inside two SUMs — the classic pivot-without-GROUP BY. Each row contributes its amount to at most one column. The whole table collapses to a single summary row.

3. The resulting analysis

this_weeklast_week
350535

This week is 100 + 250 = 350 (the −1 and −5 sales). Last week is 300 + 175 + 60 = 535 (the −9, −12 and the boundary −8). The −20 sale is excluded from both. Revenue is down week-over-week — 350 vs 535 — which is exactly the number the review wants.

4. Strategic explanations

Why the windows abut without overlapping

"This week" is sale_date >= CURRENT_DATE - 7 — everything from 7 days ago through today. "Last week" is BETWEEN CURRENT_DATE - 14 AND CURRENT_DATE - 8. The two ranges meet exactly: last week ends at −8, this week starts at −7. No day belongs to both, no day falls between them. That is the property to verbalise — "the windows tile the calendar with no seam".

The classic off-by-one trap

A tempting "fix" is to make last week BETWEEN CURRENT_DATE - 14 AND CURRENT_DATE - 7. Now day −7 is in both buckets and its revenue is counted twice. The opposite slip — starting this week at > CURRENT_DATE - 7 instead of >= — drops day −7 from everything. BETWEEN is inclusive on both ends in Postgres; pair it with a strict >= on the adjacent window so the shared edge is owned by exactly one side.

NULL when a window is empty

SUM over zero matching rows returns NULL, not 0. If last week had no sales at all, the dashboard would show a blank instead of a zero, and a downstream growth calculation (this_week - last_week) would yield NULL. Wrap each side: COALESCE(SUM(CASE … END), 0).

Timestamps vs dates — the silent boundary bug

If sale_date were actually a timestamp, CURRENT_DATE - 7 is midnight, so a sale at 09:00 on that day still qualifies — but a comparison like sale_date <= CURRENT_DATE would exclude everything after midnight today. When the column carries a time component, truncate it: sale_date::date, or compare against date_trunc('day', …). State the column's type before trusting the arithmetic.

Adding the growth percentage

The reviewer's next sentence is always "and what's the change?" Compute it in one pass without re-scanning:

SELECT this_week, last_week,
       ROUND(100.0 * (this_week - last_week) / NULLIF(last_week, 0), 1) AS wow_pct
FROM (
  SELECT
    COALESCE(SUM(amount) FILTER (WHERE sale_date >= CURRENT_DATE - 7), 0) AS this_week,
    COALESCE(SUM(amount) FILTER (WHERE sale_date BETWEEN CURRENT_DATE - 14 AND CURRENT_DATE - 8), 0) AS last_week
  FROM sales
) t;

The NULLIF(last_week, 0) guards the divide-by-zero when last week was a dead period.

Senior signal: draw the two seven-day windows so they abut with no overlap and no gap, justify the >= / BETWEEN boundary choice out loud, pre-empt the empty-window NULL with COALESCE, and flag the timestamp-truncation hazard before it silently corrupts the edge days.
