A sales-operations team wants a per-region commission scorecard: how many agents work each area, what their average, minimum, and maximum commission rates are, the spread between best and worst, and where each region ranks by average. The interviewer is probing whether you can combine grouped aggregates with a window-function ranking in the right order, and whether you pick the ranking function whose tie behaviour the spec actually names.

The fixture has 12 agents across 9 areas. The decisive edge case: three different regions — Bangalore, Chennai, London — all average exactly 0.14. That three-way tie is what tests the ranking function. Single-agent areas (Chennai, Toronto, Mumbai...) are the degenerate one-member groups, where min, max, and avg all coincide and the spread is 0.

2. The SQL — aggregate in a CTE, then DENSE_RANK

WITH g AS (
  SELECT working_area,
         COUNT(*) AS num_agents,
         ROUND(AVG(commission)::numeric, 4) AS avg_commission,
         MIN(commission) AS min_commission,
         MAX(commission) AS max_commission,
         MAX(commission) - MIN(commission) AS commission_spread
  FROM agent
  GROUP BY working_area
)
SELECT working_area, num_agents, avg_commission, min_commission, max_commission, commission_spread,
       DENSE_RANK() OVER (ORDER BY avg_commission DESC) AS rk
FROM g
ORDER BY rk, working_area;

The g CTE collapses agents into one row per area with all five aggregates. The outer query then applies DENSE_RANK() OVER (ORDER BY avg_commission DESC) to those per-area rows, assigning the rank, and orders the output by rank then area name.

4. Strategic explanations

Why aggregate first, then rank — the two-stage shape

You cannot rank regions by a per-region average in one pass: AVG(commission) is computed by GROUP BY, while DENSE_RANK is a window function that needs those averages as input rows. The g CTE produces one row per area; only then can the window function rank them. Trying to write DENSE_RANK() OVER (ORDER BY AVG(commission) DESC) directly is actually legal in PostgreSQL — window functions may take aggregates as arguments — but the CTE form is far clearer about the "summarise, then rank" intent and is what an interviewer wants to see.

DENSE_RANK vs RANK — the spec dictates the choice

The question says "ties get the same rank, no gaps — use DENSE_RANK". That clause is decisive. RANK() would also give the three 0.14 areas rank 2, but the next area would jump to rank 5 (1 + the number of rows at or above it), leaving a gap. DENSE_RANK() compresses ranks so the next distinct value is always exactly one higher — 1, 2, 2, 2, 3, 4, 4, 5, 5. And ROW_NUMBER() would be wrong outright: it would give the three tied areas different numbers, arbitrarily breaking a tie the business considers equal.

Why commission_spread is MAX - MIN, and the single-member case

MAX(commission) - MIN(commission) is the range — a one-line dispersion measure. For a single-agent area there is exactly one value, so MAX and MIN are that same value and the spread is 0 — correct, not a bug: a region with one agent has no internal variation. Recomputing MIN/MAX inside the subtraction (rather than referencing the aliases) is necessary because SQL does not let you use a select-list alias in a sibling select expression; the engine evaluates them independently and cheaply in the same group pass.

The ::numeric cast inside ROUND — a real correctness guard

AVG(commission) on a numeric column already returns numeric, but PostgreSQL's two-argument ROUND(value, scale) is only defined for numeric — there is no ROUND(double precision, int). If AVG ever produced a double precision (for instance if the column were float), ROUND(x, 4) would throw "function round(double precision, integer) does not exist". The explicit ::numeric cast makes the call type-safe regardless of the column's type — load-bearing, not decorative.

The num_agents >= 1 filter is a no-op — and the question admits it

The spec asks to "return only areas where num_agents >= 1 (i.e. all of them)". GROUP BY only ever produces a group when at least one row exists, so every grouped row already has COUNT(*) >= 1 — a HAVING COUNT(*) >= 1 would filter nothing. The phrasing is a deliberate tell: a region with zero agents cannot appear in a GROUP BY over agent at all. To list empty regions you would need a separate region table and a LEFT JOIN. Recognising that a >= 1 group filter is always vacuous is a small but real senior signal.

Senior signal: structure region scorecards as aggregate-in-a-CTE then rank-with-a-window, choose DENSE_RANK when the spec wants tied groups to share a rank with no gaps (vs RANK's gaps or ROW_NUMBER's arbitrary tie-break), keep the ::numeric cast that makes two-argument ROUND type-safe, and recognise that a num_agents >= 1 filter over a GROUP BY is always a no-op since empty groups never materialise.
