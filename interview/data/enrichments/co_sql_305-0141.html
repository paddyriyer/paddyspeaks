Target's L&D team runs internal certification exams — some count more than others (the in-store-safety exam carries 5x weight; the optional Excel quiz counts 1x). Final scores must be weighted averages, not simple averages, or someone passing five trivial quizzes ends up "tied" with someone who passed the heavyweight cert. The query is one line, but the senior conversation is about NULL-handling and what happens when weights are missing.

1. The source data — scores

Both students took 3 exams. The simple average gives S_01 = 80, S_02 = 85 — suggesting S_02 is the stronger learner. The weighted average tells a different story.

2. The SQL — SUM(score*weight) / SUM(weight)

SELECT student_id,
  SUM(score * weight) / SUM(weight) AS weighted_avg
FROM scores
GROUP BY student_id;

4. Strategic explanations

Why SUM(score*weight) / SUM(weight) — not AVG(score*weight)

A common mistake: writing AVG(score * weight). That returns "average weighted score-point" — a meaningless number scaled by the average weight. The correct identity for weighted mean:

weighted_avg = sum_of(score_i * weight_i) / sum_of(weight_i)

Test mentally: if all weights are equal (say all 1), SUM(score*1) / SUM(1) = SUM(score)/N = simple AVG. The formula generalises correctly to the simple-average case. AVG(score*weight) doesn't.

The integer-division trap

If both score and weight are integer columns, MySQL/SQLite may do integer division and lop off the decimal. Cast one operand:

SUM(score * weight) * 1.0 / SUM(weight)
-- or
CAST(SUM(score*weight) AS FLOAT) / SUM(weight)
The zero-weight trap

If a student has all weights zero, the denominator is zero and the query errors (Postgres) or returns NULL (SQLite). Production-grade form:

SUM(score * weight) * 1.0 / NULLIF(SUM(weight), 0) AS weighted_avg
Missing-exam handling — the silent zero

What if a student didn't take the safety exam? The simple form just averages what they took — effectively giving them a "skip" instead of a zero. If the policy is "a missed exam counts as zero", you need a LEFT JOIN against the full exam catalog:

SELECT s.student_id, SUM(COALESCE(sc.score, 0) * e.weight) * 1.0 / SUM(e.weight) AS weighted_avg
FROM students s
CROSS JOIN exams e
LEFT JOIN scores sc ON sc.student_id = s.student_id AND sc.exam = e.exam
GROUP BY s.student_id;

Always confirm the missing-exam policy with the L&D PM. "We just skip them" vs "they count as zero" are very different curricula.

The pass / fail / honour-roll buckets

The L&D dashboard's actual ask is segmentation:

WITH wa AS (
  SELECT student_id,
    SUM(score*weight) * 1.0 / NULLIF(SUM(weight), 0) AS weighted_avg
  FROM scores GROUP BY student_id
)
SELECT
  CASE
    WHEN weighted_avg >= 90 THEN 'honour_roll'
    WHEN weighted_avg >= 75 THEN 'pass'
    WHEN weighted_avg >= 60 THEN 'remedial'
    ELSE 'fail'
  END AS bucket,
  COUNT(*) AS students
FROM wa GROUP BY 1;
Time-decayed weights — the over-the-bar variant

Real-world certification renewals weight recent scores more than old ones. Multiply the static weight by an exponential decay on age:

SUM(score * weight * EXP(-0.05 * EXTRACT(DAY FROM CURRENT_DATE - exam_date))) /
SUM(weight * EXP(-0.05 * EXTRACT(DAY FROM CURRENT_DATE - exam_date))) AS time_decayed_avg

Used for "your most recent safety exam matters more than the one from 3 years ago" rolling-average dashboards.

The Simpson's paradox follow-up

If you compute weighted average per cohort and then average those, you can get a different number than computing the global weighted average directly — classic Simpson's paradox. Always weight at the lowest grain (per (student, exam)) and aggregate up; never re-average pre-aggregated weighted averages.

Senior signal: name the SUM(x*w)/SUM(w) identity and contrast against the AVG(x*w) mistake, pre-empt integer-division and zero-weight traps with explicit guards, and surface the missing-exam policy question unprompted. The time-decayed variant and the Simpson's-paradox warning are over-the-bar.
