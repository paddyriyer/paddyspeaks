Broadcast join is the single largest free lunch in Spark performance work. Replace a 4-hour shuffle-sort-merge join with a 4-minute hash lookup, no algorithmic complexity added. The senior question is never "what's the syntax" — it's "how do you decide WHEN to broadcast, what's the size limit, and what happens when AQE makes the decision for you." This question is a tell for whether the candidate has actually tuned a Spark job in anger.

1. The source data — the canonical asymmetric join

sidetablerowssizerole
LEFT (large fact)orders1,000,000,000~400 GB ParquetOne row per order, partitioned by date
RIGHT (small dim)dim_country249~30 KBISO country code → country name, region, currency

The ratio is 4 million to 1. The dim table fits in L2 cache of a modern CPU. Shuffling 400 GB of orders just to put rows next to a 30 KB lookup table is institutional malpractice. Yet this is exactly what Spark's default SortMergeJoin does when the planner doesn't know the right side is tiny.

2. The problem without broadcast — what SortMergeJoin actually costs

                  ┌────────────────────┐         ┌────────────────────┐
                  │ orders  (400 GB)   │         │ dim_country (30KB) │
                  │ partitioned by ds  │         │ 1 partition        │
                  └────────────────────┘         └────────────────────┘
                            │                              │
                  ┌─────────▼──────────┐         ┌─────────▼──────────┐
                  │ Shuffle: HASH(cc)  │         │ Shuffle: HASH(cc)  │
                  │ writes 400GB twice │         │ broadcast 30KB     │  ← but Spark doesn't know that
                  │ across the network │         │ shuffled anyway    │
                  └─────────┬──────────┘         └─────────┬──────────┘
                            │                              │
                            ▼                              ▼
                       ┌────────────────────────────────────────┐
                       │  SortMergeJoin on country_code         │  ← sorts BOTH sides per partition
                       │  Each partition: sort 400GB / 200      │  ← 2 GB sort per task = OOM risk
                       └────────────────────────────────────────┘

  Wall-clock time observed in production:  3h 48m on a 100-executor cluster

The waste: 400 GB of network shuffle, 400 GB of disk spill during sort, and a final merge phase — all to pair each order row with one of 249 country rows. The dim table could have been a Python dict copied to every executor.

3. The fix — broadcast hash join, step-by-step

from pyspark.sql import SparkSession, functions as F
from pyspark.sql.functions import broadcast
from pyspark.sql import Row

spark = (SparkSession.builder
         .appName("broadcast-join")
         # The threshold below which Spark auto-broadcasts. Default 10 MB.
         # Set explicitly so reviewers see the intent.
         .config("spark.sql.autoBroadcastJoinThreshold", "10485760")   # 10 MB
         .config("spark.sql.adaptive.enabled", "true")
         .getOrCreate())

# ─── Seeded sample so the example is runnable ──────────────────────────────
orders = spark.createDataFrame([
    Row(order_id=1, country_code="US", amount=49.99),
    Row(order_id=2, country_code="US", amount=12.50),
    Row(order_id=3, country_code="IN", amount=7.20),
    Row(order_id=4, country_code="DE", amount=88.00),
    Row(order_id=5, country_code="BR", amount=21.10),
    Row(order_id=6, country_code="XX", amount=5.00),     # unknown country → tests left-join behaviour
])
dim_country = spark.createDataFrame([
    Row(country_code="US", country_name="United States", region="NA",   currency="USD"),
    Row(country_code="IN", country_name="India",         region="APAC", currency="INR"),
    Row(country_code="DE", country_name="Germany",       region="EU",   currency="EUR"),
    Row(country_code="BR", country_name="Brazil",        region="LATAM",currency="BRL"),
])

# broadcast() = hint to the planner: ship this DataFrame to every executor as a hash map.
# The planner converts SortMergeJoin → BroadcastHashJoin.
# No shuffle on the large side. Each executor does a local hash-lookup per row.
joined = orders.join(broadcast(dim_country), "country_code", "left")

# Verify the plan: look for "BroadcastHashJoin" (good) vs "SortMergeJoin" (bad).
joined.explain()

joined.show(truncate=False)

What broadcast actually does under the hood
1. Driver collects the right side → in-memory HashMap[key → row]
2. Driver serialises HashMap, ships to every executor via torrent-like protocol
   (BitTorrent variant — peers share, not driver-fan-out)
3. Each executor caches the HashMap in BlockManager (off-heap if configured)
4. Per partition of LEFT side: stream rows, do a hash lookup, emit joined row
5. NO shuffle on the large side. NO sort. NO disk spill.

  Wall-clock time observed in production:  4m 12s   (~55× speedup over 3h 48m SMJ)

4. Where this appears in production

  • Stripe's charges-by-region pipeline broadcasts a ~50-row dim_payment_method table against a 100 GB charges fact every hour. The job is part of fraud-rule serving and has a 5-minute SLA — only broadcast meets it.
  • Netflix's playback-events enrichment broadcasts the device_taxonomy table (~3,000 rows) onto billions of playback events nightly. They explicitly set spark.sql.autoBroadcastJoinThreshold = 100MB in their cluster default to make this automatic.
  • Airbnb's pricing pipeline broadcasts the currency_rates table (~180 rows × 1 row per day) when normalising every booking's price to USD. Replaced a 90-minute SMJ with a 6-minute broadcast.
  • Uber's ETA model training broadcasts the dim_city_attributes table (~500 cities) onto trip records. They keep a guard: if broadcast().count() > 500_000, fall back to bucketed join — a runtime safety check against the dim table growing past the broadcast budget.
  • Pinterest's content ranking uses broadcast for dim_category (~10K rows) but switched to bucketBy for dim_user_segment (~50M rows) — the size split where broadcast stops being free.

5. Interviewer probes — the edge cases

Probe 1 — "When does broadcast hurt?"

Three failure modes:

  • Dim too big. Broadcast serialises the right side and ships it to every executor. A 5 GB "small" table on 200 executors = 1 TB of network + 5 GB of executor memory per node. OutOfMemoryError in the driver when collecting, or in executors when deserialising.
  • Driver OOM during collect. Broadcast goes through the driver. If the driver has 4 GB heap and you broadcast 3 GB, you crash mid-job. Symptom: SparkException: Cannot broadcast the table that is larger than 8GB (hard limit).
  • Right side is a complex DataFrame, not a table. broadcast(df.filter(...).join(...)) forces Spark to compute the entire sub-plan, collect it to the driver, and re-distribute. If the sub-plan is expensive, you've paid that cost once per query.

Practical limit: broadcast tables up to ~200 MB. Above that, use shuffle hash join with bucketing, or sort-merge with co-partitioning.

Probe 2 — "What if I don't use the broadcast() hint — will Spark figure it out?"

Sometimes. The auto-broadcast threshold (spark.sql.autoBroadcastJoinThreshold, default 10 MB) kicks in only when:

  • Spark can estimate the right side's size from table statistics. Run ANALYZE TABLE dim_country COMPUTE STATISTICS first — otherwise estimates use file-size heuristics that over-count compressed Parquet by 5-10×.
  • The right side is a base table or a simple scan, not a sub-query (no filter/agg/join in front of it).
  • With AQE on, Spark can switch SortMergeJoin → BroadcastHashJoin at runtime once it sees a stage's output is small (e.g. after an aggressive filter). This is the killer feature: you write a normal join, AQE picks broadcast when the data shape allows.

The broadcast() hint is for cases where stats are stale or wrong, or where you know the dim is small even though Spark doesn't.

Probe 3 — "What's the difference between BroadcastHashJoin and BroadcastNestedLoopJoin?"

BroadcastHashJoin: equi-join (ON a.key = b.key), uses hash lookup, O(N) on the streamed side. BroadcastNestedLoopJoin: non-equi joins (ON a.x BETWEEN b.lo AND b.hi or arbitrary predicates), no hash possible, O(N × M) — broadcasts B, then for each row in A scans all of B. Useful for tiny B (range lookups, IP-to-country) but catastrophic if B grows. Detection: BroadcastNestedLoopJoin in the physical plan = either tiny B by design, or you misread your data.

Probe 4 — "Can you broadcast the LEFT side in a LEFT JOIN?"

No. Broadcast hash join broadcasts the build side, which for a LEFT OUTER JOIN must be the right side (otherwise you can't enumerate unmatched left rows). Specifically:

  • INNER: either side can be broadcast.
  • LEFT OUTER: only RIGHT side can be broadcast.
  • RIGHT OUTER: only LEFT side can be broadcast.
  • FULL OUTER: cannot use BroadcastHashJoin at all.
  • LEFT SEMI / LEFT ANTI: only RIGHT side can be broadcast.

If you hint broadcast() on the wrong side, Spark silently ignores the hint and falls back to SMJ. Read explain() to verify.

Probe 5 — "How does this interact with AQE?"

AQE (Adaptive Query Execution, default on in 3.2+) adds two relevant behaviours:

  • Dynamic broadcast promotion: A SortMergeJoin can be rewritten to BroadcastHashJoin at runtime once AQE sees the actual size of a shuffle output (not the estimate). Threshold: spark.sql.adaptive.autoBroadcastJoinThreshold (separate from the static one).
  • Skew-join split: If one side has skew, AQE splits the skewed partitions and replicates the matching side. This obviates manual salting for the join case (but not for the agg case — see customer_growth-0027).

With AQE on, an explicit broadcast() hint is an override that says "I know better than the planner — broadcast this even if stats say otherwise." Use sparingly.

Probe 6 — "You said 200 MB is the practical limit. Could I push it to 1 GB?"

Yes, with caveats. Bump spark.sql.autoBroadcastJoinThreshold and spark.driver.maxResultSize together. The driver collects the dim, so driver heap must be 2× the broadcast size (Java serialised overhead). Executors must have room for the hash map. The break-even point is workload-specific: if the LEFT side is > 100 GB and the dim is 1 GB, broadcast (1 GB × 200 nodes = 200 GB network) might still beat SMJ (100 GB shuffle of LEFT). Use explain("formatted") and the SQL UI's runtime stats to decide empirically.
