Same retention question as its window-function sibling — identify "returning active users" who made a second purchase within 7 days of one of their others — but here the interviewer wants the self-join formulation. The probe is whether you can pair every purchase of a user with every other purchase of the same user, compute the day gap, and then recognise the performance and correctness pitfalls a self-join introduces that a window function avoids.

User 300 (single purchase) and user 400 (two purchases on the same date) are the edge cases — one tests the self-join's behaviour with no partner, the other tests the <> 0 filter.

2. The SQL — self-join on userid, gap via date subtraction

WITH ac AS (
  SELECT t1.userid AS user_1, t1.createdat AS createdat_1,
         t2.userid AS user_2, t2.createdat AS createdat_2,
    CASE WHEN t1.createdat::text ~ '^[0-9]{4}-[0-9]{2}-[0-9]{2}'
          AND t2.createdat::text ~ '^[0-9]{4}-[0-9]{2}-[0-9]{2}'
         THEN t1.createdat::date - t2.createdat::date
         ELSE NULL END AS diff_date
  FROM transactions_ecomm t1
  INNER JOIN transactions_ecomm t2 ON t1.userid = t2.userid
)
SELECT DISTINCT user_1 FROM ac
WHERE ABS(diff_date) <= 7 AND diff_date <> 0;

4. Strategic explanations

Why ABS on the difference is required here

The self-join pairs purchases both ways: row A joins row B (gap +4) and row B joins row A (gap −4). Only one direction is positive, so ABS(diff_date) <= 7 catches the pair regardless of join order. This is a structural consequence of the unconstrained self-join — it produces every ordered pair, including the symmetric mirror. The LEAD-based sibling solution never needs ABS because it only ever looks forward in sorted order.

The diff_date <> 0 clause does two jobs — one good, one debatable

Because the join has no t1.id <> t2.id guard, every purchase also joins to itself, yielding diff_date = 0. The <> 0 filter is doing necessary work removing those self-pairs. But it also removes user 400's genuine same-day second purchase. Those are two different zeros conflated into one filter. The cleaner design separates them: constrain the join with AND t1.id <> t2.id to drop true self-matches, then decide same-day policy independently — ABS(diff_date) BETWEEN 0 AND 7 if a same-day repeat counts as returning (it usually does). Surfacing that conflation is the senior observation.

The performance trap — O(n²) per user

A self-join on userid produces, for a user with k purchases, k² paired rows — the full cross product within each user. A power user with 10,000 orders generates 100 million pairs before the WHERE filters anything. The window-function approach sorts each user's purchases once and compares neighbours: O(n log n) total. For an interview, state plainly that the self-join is fine for correctness and small data but is the wrong tool at scale, and that LEAD is the production answer.

You can shrink the self-join with a directional predicate

Adding AND t2.createdat > t1.createdat (or t2.id > t1.id) to the join keeps only one direction of each pair — halving the output, eliminating self-matches, and removing the need for ABS entirely. Even better, tighten it to AND t2.createdat BETWEEN t1.createdat AND t1.createdat + 7 so the gap test happens in the join, letting an index on (userid, createdat) do the heavy lifting instead of materialising every pair.

The regex guard signals an untyped column

As in the sibling problem, the ~ '^[0-9]{4}-...' checks only make sense if createdat is stored as dirty text. With a real DATE column the CASE collapses to t1.createdat - t2.createdat. Validating date format mid-query is a code smell — the fix is a typed column enforced at ingestion.

Senior signal: explain why an unconstrained self-join forces ABS (symmetric pairs) and why diff_date <> 0 is silently doing two jobs — removing real self-matches and excluding genuine same-day repeats. Tightening the join to a directional, range-bounded predicate (and naming LEAD as the O(n log n) production answer) is the over-the-bar finish.
