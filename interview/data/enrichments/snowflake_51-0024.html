Data engineering teams at SaaS companies often receive churn signals from multiple upstream sources (CRM exports, billing systems, support tickets) that each record the same customer’s churn independently. The task: "Combine all three sources, deduplicate by customer, and return each customer’s first churn date and the source where they were first seen." This tests UNION ALL for source-tagged combination, ROW_NUMBER for cross-source deduplication, and CTE composition.

C001 appears in sources A (Jan 15) and B (Jan 20) — source A saw it first. C002 appears in sources A (Feb 10) and C (Feb 8) — source C actually has the earlier date. C003 and C004 appear in only one source each. These cases exercise the core deduplication logic: earliest date wins, not earliest source letter.

2. The SQL — UNION ALL + ROW_NUMBER cross-source dedup

WITH all_churn AS (
  SELECT customer_id, churned_on, 'A' AS src FROM source_a
  UNION ALL
  SELECT customer_id, churned_on, 'B'         FROM source_b
  UNION ALL
  SELECT customer_id, churned_on, 'C'         FROM source_c
),
deduped AS (
  SELECT customer_id, churned_on, src,
         ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY churned_on) AS rn
  FROM all_churn
)
SELECT customer_id, churned_on AS first_churn_date, src AS first_seen_in
FROM deduped
WHERE rn = 1
ORDER BY first_churn_date;

4. Strategic explanations

UNION ALL vs UNION — keep the source tag, keep duplicates

UNION ALL is used (not UNION) for two reasons: first, it preserves the src tag that identifies which source each row came from; second, it avoids the implicit deduplication that UNION applies (which would be wrong here — we want to see all versions of each customer across all sources before choosing the earliest). UNION without ALL also requires an extra sort pass to deduplicate, making it slower. Always prefer UNION ALL unless deduplication across the full row is intentional.

The ROW_NUMBER partition + ORDER BY — the cross-source earliest-date selector

ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY churned_on) ranks each customer’s rows across all three sources by date, earliest first. Filtering WHERE rn = 1 returns only the earliest row for each customer. The src column on that row identifies the winning source. This is the clean way to select "the row with the minimum value per group" without a correlated subquery or a JOIN on a MIN subquery.

Tie-breaking when two sources have identical dates

If C001 appeared with the same churned_on in both source A and source B, the ROW_NUMBER would assign rank 1 arbitrarily. To make the result deterministic, add a secondary tiebreaker: ORDER BY churned_on, src would favour source A over B over C alphabetically. The right tiebreaker is a business decision (e.g., "prefer the most authoritative source"), but always make it explicit in code.

CTE composition for readability and maintainability

The two CTEs split the logic cleanly: all_churn handles combination, deduped handles ranking, the final SELECT handles filtering and presentation. Each stage is independently readable and testable. A single nested subquery would achieve the same result but is harder to debug. In production code, always name the intent of each CTE.

Extension: flag customers seen in multiple sources

A natural follow-on adds a count of how many sources reported the churn — customers seen in all three sources are the most reliable churn signals:

COUNT(DISTINCT src) AS source_count

Add this to the deduped CTE as COUNT(DISTINCT src) OVER (PARTITION BY customer_id) to expose it on every row without an additional GROUP BY.

Dialect notes
  • Snowflake / Postgres / BigQuery / MySQL 8+: CTEs, UNION ALL, and ROW_NUMBER work identically.
  • Snowflake QUALIFY shorthand: the deduped CTE + WHERE rn = 1 can be replaced with QUALIFY ROW_NUMBER() OVER (...) = 1 directly on the all_churn CTE.
Senior signal: use UNION ALL (not UNION) to preserve source tags and avoid an implicit sort, note that the ORDER BY inside ROW_NUMBER chooses the earliest date across sources (not the alphabetically first source), and always add a deterministic tiebreaker for same-date conflicts. Bonus: extend with COUNT(DISTINCT src) OVER (...) to surface multi-source confirmation as a churn confidence score.
