The revenue team needs to highlight the top-3 products by revenue within each category — a leaderboard query. The interviewer is probing whether you know the difference between RANK, DENSE_RANK, and ROW_NUMBER for tie-breaking, whether you can correctly apply a window function on top of a GROUP BY aggregate, and whether you know that Snowflake supports QUALIFY to avoid the outer subquery.

2. The SQL — RANK over a per-category aggregate in a subquery filter

SELECT category, product_id, name, revenue, rk
FROM (
  SELECT p.category, p.product_id, p.name,
         SUM(s.amount) AS revenue,
         RANK() OVER (PARTITION BY p.category
                      ORDER BY SUM(s.amount) DESC) AS rk
  FROM sales s
  JOIN products p ON s.product_id = p.product_id
  GROUP BY p.category, p.product_id, p.name
)
WHERE rk <= 3
ORDER BY category, rk;

The inner query joins, aggregates revenue per product, and applies RANK() within each category. The outer query filters to the top 3. In Snowflake, the subquery wrapper can be replaced with QUALIFY RANK() OVER (…) <= 3 directly in the GROUP BY query — no derived table needed.

4. Strategic explanations

Window function on an aggregate: the subquery is required (except in Snowflake)

You cannot write RANK() OVER (PARTITION BY p.category ORDER BY SUM(s.amount) DESC) directly in a query alongside GROUP BY and then filter it in the same WHERE clause. The window function is evaluated after GROUP BY but before the outer WHERE, so filtering on the window result requires either a subquery/CTE or — in Snowflake — the QUALIFY clause, which is evaluated after window functions and avoids the derived table entirely.

RANK vs DENSE_RANK vs ROW_NUMBER

If two products in the same category had equal revenue, RANK() would assign the same rank to both and skip the next rank (e.g., two products at rank 1, nothing at rank 2). A WHERE rk <= 3 filter would then return 4 rows for that category — two tied at rank 1 plus two more. DENSE_RANK() would not skip: both ties get rank 1, next gets rank 2. ROW_NUMBER() assigns a unique rank to each row, breaking ties arbitrarily. For a strict "top 3 products" limit regardless of ties, use ROW_NUMBER; for "all products that are in top-3 revenue," use RANK. Knowing which one the business requires is the question to ask before writing any code.

Snowflake's QUALIFY clause

Snowflake (and BigQuery's QUALIFY extension) allows: GROUP BY p.category, p.product_id, p.name QUALIFY RANK() OVER (PARTITION BY p.category ORDER BY SUM(s.amount) DESC) <= 3. This eliminates the derived table and makes the intent clearer. PostgreSQL and MySQL do not support QUALIFY — the subquery wrapper is required. Mentioning this in a Snowflake interview is the "knows the platform" signal.

Products with zero sales

The INNER JOIN drops products with no sales rows. If the requirement includes "show all products in their category ranking even if they have $0 revenue," change to LEFT JOIN sales s ON s.product_id = p.product_id and use COALESCE(SUM(s.amount), 0). Zero-revenue products would rank last within their category.

Performance

In Snowflake, the optimizer pushes the GROUP BY into the distributed aggregation layer before the window function runs. The subquery (or QUALIFY) does not add a separate scan — it is a filter on the window output. A composite sort key on sales(product_id) and a cluster key on products(category) improve micro-partition pruning for the join and the PARTITION BY, respectively.

Senior signal: distinguish RANK (ties get same rank, next skipped), DENSE_RANK (ties share rank, no skip), and ROW_NUMBER (always unique) — and ask the business which tie-breaking rule applies; cite Snowflake's QUALIFY as the subquery-free alternative; and note that an INNER JOIN silently drops zero-sale products that might be relevant for the ranking.
