“Time to first reply” on a support inbox is a self-join — match each inbound message to the reply that answers it. The trap is the join predicate: a reply is not just “another message in the same thread.” It must travel in the opposite direction (sender and recipient swapped), arrive after the inbound message, and only the earliest such reply counts. Miss any one condition and the answer is silently wrong.

1. The source data — messages

2T1support@co.comalice@x.com2025-04-01 11:30
3T2bob@y.comsupport@co.com2025-04-02 14:00
4T2support@co.combob@y.com2025-04-02 14:45
5T2support@co.combob@y.com2025-04-02 16:00
6T3carol@z.comsupport@co.com2025-04-03 10:00

T2 has two support replies (msg 4 and msg 5). T3 has no reply at all.

2. The query — a direction-aware self-join

SELECT inbound.msg_id, inbound.thread,
       MIN(reply.sent_at) - inbound.sent_at AS response_time
FROM messages inbound
JOIN messages reply
  ON reply.thread    = inbound.thread
 AND reply.sender    = inbound.recipient   -- reply goes the other way
 AND reply.recipient = inbound.sender
 AND reply.sent_at   > inbound.sent_at     -- and strictly after
WHERE inbound.recipient = 'support@co.com'
GROUP BY inbound.msg_id, inbound.thread, inbound.sent_at
ORDER BY inbound.msg_id;

The aliases inbound and reply are the same table. The join pairs an inbound message with every later message flowing the opposite way in the thread; MIN(reply.sent_at) then picks the first of those.

3. The result

msg_idthreadresponse_time
1T102:30:00
3T200:45:00

msg 1 is answered by msg 2 (2h30m later). msg 3 has two candidate replies, msg 4 and msg 5; MIN takes msg 4 (45 minutes). msg 6 has no reply, so the inner join drops it — which is exactly the requirement (“exclude messages that received no response”).

4. Strategic explanations

The direction swap is the crux

Joining only on thread matches every message to every other message in the thread — including itself, and including same-direction messages. The pair of conditions reply.sender = inbound.recipient AND reply.recipient = inbound.sender enforces that the reply is a genuine response: it goes from the original recipient back to the original sender. Without the swap, an inbound message could “answer” itself or be answered by a parallel inbound message.

The time predicate must be strict

reply.sent_at > inbound.sent_at keeps only replies that came after. Using >= would let a message pair with itself if timestamps tie; omitting the predicate entirely would let an earlier message count as a “reply” to a later one, producing negative response times.

MIN handles the multi-reply thread

A thread often has several back-and-forth messages. The join produces one row per (inbound, later-reply) pair, so msg 3 joins to both msg 4 and msg 5. Aggregating with MIN(reply.sent_at) collapses those to the first reply — the response time. Taking MAX, or forgetting to aggregate, would measure the wrong thing.

INNER JOIN doubles as the “had a response” filter

Because the join is an inner join, an inbound message with no qualifying reply contributes no rows and disappears — msg 6 is excluded for free. If the task instead wanted all inbound messages with NULL for unanswered ones, a LEFT JOIN would be required, and the GROUP BY would still collapse correctly.

Senior signal: a “response” self-join must pin three things at once — same thread, swapped sender/recipient (the reply travels the other way), and a strict > on time — then MIN the reply timestamp to get the first response. The inner join naturally excludes unanswered messages; switch to LEFT JOIN only if unanswered rows must be kept.
