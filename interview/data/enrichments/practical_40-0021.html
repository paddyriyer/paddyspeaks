An auditor at an e-commerce company flags that the orders table looks bloated — a billing reconciliation found the same charge appearing several times. The task is to surface the offending composite keys: a row counts as a duplicate when (customer_id, order_date, order_total) already exists. The interviewer is checking whether you can define "duplicate" precisely against a multi-column key, and whether you return only the keys that actually repeat — not every row.

The edge case is the triple at customer 100: a duplicate group is not always exactly two rows. A correct answer must report the multiplicity (3), not just a yes/no flag, and must exclude the genuinely unique rows 6 and 9.

2. The SQL — GROUP BY the key, filter with HAVING

SELECT customer_id, order_date, order_total, COUNT(*) AS copies
FROM orders
GROUP BY customer_id, order_date, order_total
HAVING COUNT(*) > 1
ORDER BY copies DESC, customer_id;

The GROUP BY lists exactly the columns that define a duplicate, collapsing every matching row into one group. COUNT(*) measures the group size, and HAVING COUNT(*) > 1 keeps only groups that repeat — HAVING filters after aggregation, which WHERE cannot do.

4. Strategic explanations

Why HAVING and not WHERE

WHERE is evaluated before rows are grouped, so it cannot see COUNT(*) — the aggregate does not exist yet. HAVING runs after the GROUP BY has formed groups and computed aggregates, so it is the only clause that can filter on "group size > 1". Mixing them up — WHERE COUNT(*) > 1 — is a syntax error in every engine.

The key must match the business definition exactly

Duplicate is defined here as (customer_id, order_date, order_total) — deliberately not order_id, which is unique by construction and would put every row in its own group. Whatever columns the business calls "the same order" must be the exact GROUP BY list. Add a column and you under-report; drop one and you over-report. The GROUP BY list is the definition of duplicate.

NULLs in the key — a quiet trap

GROUP BY treats all NULLs in a column as one group (unlike =, which never matches NULL). So two rows with a NULL order_total but the same customer and date would be flagged as duplicates here. That is usually what an audit wants, but state it: if NULL means "unknown amount" you may not want to call two unknowns identical, in which case filter WHERE order_total IS NOT NULL first or treat the NULL rows as a separate review bucket.

Finding the keys vs deleting the rows

This query identifies duplicates; it does not remove them. Deletion is the harder follow-up because you must keep one row per group. The senior answer is ROW_NUMBER() OVER (PARTITION BY customer_id, order_date, order_total ORDER BY order_id) and delete where rn > 1 — that keeps the lowest order_id deterministically. Mentioning the keep-one strategy unprompted shows you have actually run a dedup, not just spotted one.

Dialect notes

This query is fully portable — GROUP BY, COUNT(*), and HAVING behave identically in Postgres, MySQL, SQLite, Snowflake, and SQL Server. The only variance is ordering by an alias: ORDER BY copies works because ORDER BY is evaluated after the SELECT list; if an engine objected you would repeat ORDER BY COUNT(*) DESC instead.

Senior signal: define "duplicate" as an exact composite GROUP BY key, filter repeating groups with HAVING COUNT(*) > 1 (never WHERE), report the multiplicity so cleanup knows how many rows are surplus, flag that GROUP BY folds NULL keys together, and know the ROW_NUMBER follow-up for actually deleting duplicates while keeping one.
