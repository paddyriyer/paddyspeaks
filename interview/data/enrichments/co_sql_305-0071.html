This question lands at Hulu but is pure airline-operations logic: find the flights that have been overbooked — more tickets sold than seats on the plane — and by how many. It is one of the simplest queries in the bank, which is exactly why an interviewer uses it: they want to see whether you pick the right comparison operator, surface the exactly-full edge case, and treat NULL capacity deliberately rather than letting it slip through.

1. The source data — flights

flight_idcapacitytickets_sold
UA100180185
UA200200200
UA300150142
UA400220235
UA500160161

The decisive row is UA200: tickets sold exactly equals capacity. A full flight is not an overbooked flight, and the choice of > versus >= is what separates the two. UA500 is the minimal overbook — just one seat over — and should still appear.

2. The SQL — strict-greater filter

SELECT flight_id, tickets_sold - capacity AS overbooked_by
FROM flights WHERE tickets_sold > capacity;

Keep only rows where tickets sold strictly exceeds capacity, and project the difference as overbooked_by. No grouping — each flight is already one row.

3. The resulting analysis

Three flights are overbooked. UA200 is correctly absent — 200 = 200 is full, not over. UA300 is under-sold and excluded. UA500 makes the cut at +1, confirming the filter does not accidentally require a minimum overbook size. UA400 at +15 is the operations team's biggest headache.

4. Strategic explanations

> vs >= — the one-character interview

"Overbooked" means more tickets than seats, strictly. tickets_sold > capacity excludes the exactly-full flight UA200; >= would wrongly include it with overbooked_by = 0. A flight at capacity is a success, not an alert. State the definition out loud — "full is not overbooked" — and let it justify the operator.

NULL capacity or NULL tickets — the silent drop

If a flight has NULL capacity (plane not yet assigned), then tickets_sold > capacity evaluates to NULL, which is not true, so the row is silently excluded. That may be wrong — an unassigned-aircraft flight with tickets sold is arguably a worse problem. Decide explicitly:

SELECT flight_id, tickets_sold - capacity AS overbooked_by
FROM flights
WHERE tickets_sold > capacity
   OR capacity IS NULL;   -- flag missing-capacity flights for review

And the projected overbooked_by would itself be NULL for such a row — NULL minus anything is NULL. Mention it; do not let NULLs vanish unremarked.

Ordering — worst first

An alert list is only useful if the biggest problems are at the top. The query as written returns rows in physical order. Add the sort the operations team actually wants:

SELECT flight_id, tickets_sold - capacity AS overbooked_by
FROM flights
WHERE tickets_sold > capacity
ORDER BY overbooked_by DESC;

You can sort by the alias overbooked_by — ORDER BY is evaluated after the SELECT list, so column aliases are visible there (unlike in WHERE).

Overbooking rate — the analytical follow-up

Operations rarely stops at "which flights"; they want "how badly, proportionally". A 1-seat overbook on a 50-seat regional jet is worse than 5 seats on a 400-seat widebody:

SELECT flight_id,
       tickets_sold - capacity AS overbooked_by,
       ROUND(100.0 * (tickets_sold - capacity) / NULLIF(capacity, 0), 1) AS overbook_pct
FROM flights
WHERE tickets_sold > capacity
ORDER BY overbook_pct DESC;

The NULLIF(capacity, 0) guards the impossible-but-possible zero-capacity row, and 100.0 keeps the division in numeric arithmetic.

Aggregate view — fleet health

A dashboard often wants one number: how much overbooking is there across the whole fleet? SELECT SUM(GREATEST(tickets_sold - capacity, 0)) AS total_overbooked_seats FROM flights; — GREATEST(…, 0) clamps under-sold flights to 0 so they do not subtract from the total.

Senior signal: choose strict > and articulate why a full flight is not overbooked, notice that NULL capacity silently drops rows and decide what to do about it, sort the alert worst-first using the projected alias, and offer the proportional overbooking rate as the metric operations actually act on.
