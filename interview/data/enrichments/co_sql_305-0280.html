Anthropic's people-data team is consolidating an HR system. The employees table carries a denormalised dept_name that has gone stale — some rows are NULL, some hold an old abbreviation — while the authoritative names live in a clean departments dimension. The task is a classic data-sync: refresh employees.dept_name from the dimension. Other engines call this MERGE; the interview is whether you know the Postgres way and what it does to rows that don't match.

1. The source data — employees (before) and departments

departments holds 1=Engineering, 2=Sales, 3=Marketing, 4=Finance. Two edge rows matter: Carol's dept_name is the stale Mktg (must be overwritten with Marketing), and Dave's dept_id 9 has no match in departments — an orphaned foreign key.

2. The SQL — UPDATE … FROM (the Postgres MERGE)

UPDATE employees e
   SET dept_name = d.dept_name
  FROM departments d
 WHERE e.dept_id = d.dept_id;

UPDATE … FROM joins the target to a source in one statement: every employee whose dept_id matches a department gets that department's name written back.

Alice and Bob get filled, Carol's stale Mktg is corrected to Marketing. Dave is the lesson: his dept_id 9 matched nothing, so the WHERE excluded his row and dept_name stayed at its prior value (NULL here). The UPDATE … FROM only touches matched rows — it is a MERGE's WHEN MATCHED branch, with no WHEN NOT MATCHED.

4. Strategic explanations

The mental model: FROM is an inner join to the target

UPDATE t … FROM s WHERE t.k = s.k behaves like an inner join — only rows of t that find a partner in s are updated; the rest are left exactly as they were. That is why Dave is untouched, not set to NULL or errored. If you wanted unmatched employees blanked out, that is a different statement (a second UPDATE … WHERE NOT EXISTS …).

The fan-out trap — a non-unique join key

This is safe only because departments.dept_id is a primary key, so each employee matches at most one department. If the source key were not unique — say two rows for dept_id 3 — Postgres picks one match arbitrarily and the result is non-deterministic. Standard MERGE raises an error in that case ("MERGE command cannot affect row a second time"); UPDATE … FROM does not warn you. Always confirm the source key is unique before trusting it.

True ANSI MERGE — Postgres 15+

Modern Postgres supports the standard statement, which is clearer about intent and can express insert/delete branches too:

MERGE INTO employees e
USING departments d ON e.dept_id = d.dept_id
WHEN MATCHED THEN UPDATE SET dept_name = d.dept_name;

For an update-only sync the two are equivalent; MERGE wins when you also need WHEN NOT MATCHED THEN INSERT. Naming both, and saying UPDATE … FROM predates MERGE and still works everywhere, is the senior answer.

The deeper question — should dept_name exist at all?

This whole exercise exists because dept_name is denormalised onto employees, which is precisely why it drifted stale. The principled fix is to drop the column and join to departments at read time, or expose a view:

CREATE VIEW employee_full AS
SELECT e.emp_id, e.name, e.dept_id, d.dept_name
FROM employees e LEFT JOIN departments d ON e.dept_id = d.dept_id;

A LEFT JOIN view also surfaces Dave's orphan cleanly as a NULL rather than freezing a stale value. Flagging the data-modelling root cause — not just executing the sync — is what is being tested.

Idempotency and the no-op write

Re-running this UPDATE is safe: it just rewrites the same names. But Postgres still issues a physical row update even when the value is unchanged, generating dead tuples and WAL. On a large table, guard it: add AND e.dept_name IS DISTINCT FROM d.dept_name so only genuinely stale rows are touched. IS DISTINCT FROM (not <>) is required because the existing value may be NULL.

Senior signal: recognise UPDATE … FROM as an inner-join sync that leaves unmatched rows untouched (Dave stays NULL), insist the source key is unique to avoid silent non-deterministic fan-out, name ANSI MERGE as the modern equivalent, and call out that the stale-dept_name problem is a denormalisation smell better cured with a join or view.
