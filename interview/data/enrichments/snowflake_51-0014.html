Private equity and venture capital analytics always starts with the same question: "Which of our portfolio companies has each fund invested in, how much total capital was deployed, and when did we first and last touch each company?" This question probes the candidate's ability to chain a four-table star-schema join correctly, apply per-group aggregation, and produce a clean output ordered by the natural hierarchy (firm → fund → company).

2. The SQL — four-table join with per-(firm, fund, company) aggregation

SELECT f.name        AS firm,
       fu.fund_name   AS fund,
       c.name         AS portfolio_company,
       c.sector,
       SUM(i.amount)  AS total_invested,
       MIN(i.invested_on) AS first_investment,
       MAX(i.invested_on) AS latest_investment
FROM firms       f
JOIN funds       fu ON f.firm_id     = fu.firm_id
JOIN investments i  ON fu.fund_id    = i.fund_id
JOIN companies   c  ON i.company_id  = c.company_id
GROUP BY f.name, fu.fund_name, c.name, c.sector
ORDER BY f.name, total_invested DESC;

4. Strategic explanations

The join chain and its directionality

The hierarchy is firms → funds → investments → companies. Each JOIN adds one level of the star: firms → funds on firm_id, then funds → investments on fund_id, then investments → companies on company_id. Using INNER JOINs means firms with no funds, funds with no investments, or orphan company IDs are silently excluded. In a production audit query you would often use LEFT JOINs to surface those gaps — that design choice should be stated explicitly.

GROUP BY must mirror every non-aggregate SELECT column

The GROUP BY lists f.name, fu.fund_name, c.name, c.sector — exactly the four non-aggregate columns in the SELECT. Omitting c.sector would violate the SQL standard (though MySQL's lenient mode silently picks an arbitrary sector value). In Snowflake and Postgres it is a hard error. A common interview mistake is grouping only by IDs and selecting display names without aggregating them; name the pattern: "GROUP BY must cover every unaggregated column in the SELECT list."

Why total_invested is an alias in ORDER BY

The ORDER BY clause references the column alias total_invested rather than repeating SUM(i.amount). Snowflake, Postgres, BigQuery, and MySQL all allow alias references in ORDER BY because ORDER BY executes after SELECT. This is a convenience not available in WHERE or GROUP BY — use the full expression there.

Aggregate semantics: SUM vs COUNT

The query uses SUM(i.amount) for capital deployed and MIN/MAX on the date column to bracket the investment window. If the team also needed deal count (useful for pace-of-deployment metrics), adding COUNT(*) AS deal_count inside the same GROUP BY is free — no additional scan. Knowing which aggregates are "free riders" on an existing group-by is a senior efficiency instinct.

Dialect notes
  • Snowflake: The query runs as-is. Alias in ORDER BY is fully supported.
  • Postgres: Identical syntax. ORDER BY total_invested DESC resolves the alias correctly.
  • BigQuery: Alias in ORDER BY supported; JOIN syntax identical.
  • MySQL: Works, but beware the lenient GROUP BY mode (sql_mode=ONLY_FULL_GROUP_BY must be set to enforce standard behaviour).
Senior signal: name the four-table star-schema chain and explain that INNER JOINs silently exclude firms/funds with no investments — LEFT JOINs are the audit-friendly alternative. State that every non-aggregate SELECT column must appear in GROUP BY, and call out that ORDER BY can reference aliases (unlike WHERE and GROUP BY). Knowing which aggregates are "free riders" on the same GROUP BY pass signals query-cost awareness.
