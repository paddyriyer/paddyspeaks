A courier company wants the drivers who have handled every package type in the catalogue — the classic "relational division" problem. The interviewer is probing whether you can express "covers all values of a set" without a brittle hardcoded list, using a HAVING count compared against the total distinct count.

1. The source data — deliveries

The catalogue has three types: Letter, Box, Pallet. Driver 13 is the edge case — two deliveries, but both are Pallet. Counting rows would say "2"; counting distinct types says "1". That distinction is the whole problem.

2. The SQL — relational division via HAVING

SELECT driverid
FROM deliveries
GROUP BY driverid
HAVING COUNT(DISTINCT packagetype) =
       (SELECT COUNT(DISTINCT packagetype) FROM deliveries);

3. The resulting analysis

driverid(narrative)

Driver 11 covers only 2 of 3 types and is excluded. Driver 13 has two deliveries but only the Pallet type — COUNT(DISTINCT) = 1, correctly dropped. A COUNT(*) would have mistaken its duplicate Pallets for coverage.

4. Strategic explanations

Why COUNT(DISTINCT) is mandatory here

The question asks about types covered, not deliveries made. Driver 13's two Pallet rows must count as one type. Plain COUNT(packagetype) counts rows and would inflate to 2; COUNT(DISTINCT packagetype) counts unique types and yields the correct 1. Getting this wrong is the single most common error on this pattern.

The subquery makes it self-maintaining

(SELECT COUNT(DISTINCT packagetype) FROM deliveries) computes the catalogue size at query time. When the company adds a fourth type (say "Crate"), the threshold updates automatically — no edit needed. Hardcoding HAVING COUNT(DISTINCT packagetype) = 3 would silently keep passing drivers who never carried a Crate.

The "available in the system" subtlety

This solution defines the universe of package types as whatever appears in deliveries. If the catalogue lives in a separate package_types table, a type with zero deliveries would still count toward "all types", and the subquery must read that master table instead: (SELECT COUNT(*) FROM package_types). Clarifying which table is the source of truth is a senior-level question to ask.

The anti-join alternative — "no type is missing"

Relational division can also be phrased as a double negation: a driver qualifies if there is no package type they have not delivered.

SELECT d.driverid
FROM (SELECT DISTINCT driverid FROM deliveries) d
WHERE NOT EXISTS (
  SELECT 1 FROM (SELECT DISTINCT packagetype FROM deliveries) pt
  WHERE NOT EXISTS (
    SELECT 1 FROM deliveries x
    WHERE x.driverid = d.driverid AND x.packagetype = pt.packagetype));

It is wordier, but it generalises when "all" is defined by a master table and it can short-circuit. The HAVING-count form is the one to write first; mention the anti-join to show you know the canonical alternative.

Senior signal: identify this as relational division, insist on COUNT(DISTINCT packagetype) so a driver's repeated same-type deliveries do not fake full coverage, compute the threshold with a subquery so it self-updates, and ask whether the type universe is the fact table or a separate catalogue.
