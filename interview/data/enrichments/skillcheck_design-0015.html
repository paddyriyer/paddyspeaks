A time-bucketed metrics table is the foundational pattern for observability backends: rather than storing one row per event, each row pre-aggregates all events that occurred within a fixed time window for a given service+endpoint combination. The interviewer is probing your understanding of time-series grain selection (1-minute buckets), pre-aggregation vs. raw storage, percentile approximation (why you cannot simply store p95 values and later re-aggregate them across buckets without a sketch), range partitioning for retention, and composite primary key design that also serves as the clustering key for dashboard reads.

1. The requirements — what the design must capture

  • Grain: one row per (service_name, endpoint, 1-minute bucket); the bucket timestamp is always a floored minute, never a raw event timestamp.
  • Additive count measures: request_count and error_count can be summed across buckets without loss of accuracy.
  • Non-additive latency percentiles: p50_ms, p95_ms, p99_ms are meaningful only within a single bucket; they cannot be naively averaged across buckets. A mergeable sketch (td_digest) enables accurate re-aggregation.
  • Retention: ~30-day window; old partitions must be droppable without expensive deletes.
  • Dashboard read pattern: queries filter on service_name, endpoint, and a recent time range; the PK must match this access pattern exactly.

2. The DDL — the CREATE TABLE

CREATE TABLE metric_request (
  service_name      VARCHAR(64)  NOT NULL,
  endpoint          VARCHAR(128) NOT NULL,
  bucket_ts         TIMESTAMP    NOT NULL,                  -- floor(event_ts, '1 minute')
  request_count     BIGINT       NOT NULL,
  error_count       BIGINT       NOT NULL,
  p50_ms            DOUBLE,
  p95_ms            DOUBLE,
  p99_ms            DOUBLE,
  -- (optionally a t-digest blob for re-aggregation across buckets)
  td_digest         BYTEA,
  ingested_at       TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (service_name, endpoint, bucket_ts)
)
PARTITION BY RANGE (bucket_ts);

4. Strategic explanations

Grain: the 1-minute bucket

The grain is one row per (service, endpoint, minute). This is a periodic-snapshot fact-table pattern: state is captured at regular intervals. The bucket timestamp must be computed as date_trunc('minute', event_ts) (or equivalent) at write time, not stored raw, so the PK uniqueness constraint can enforce exactly one row per combination and INSERT ... ON CONFLICT DO UPDATE can accumulate counts within the bucket transactionally.

Percentiles are non-additive — the sketch solution

This is the crux of the design. If you have p95 = 120 ms for minute 1 and p95 = 80 ms for minute 2, there is no arithmetic formula to compute the correct 2-minute p95 from those two scalars alone — you need the underlying distribution. The td_digest column stores a serialized t-digest or DDSketch, which are mergeable: you can merge any number of digests and compute an accurate percentile from the result. The scalar p-values remain useful for single-minute dashboards and are fast to read; the digest supports cross-bucket analysis.

Range partitioning + retention

Daily partitions make 30-day retention trivial: DROP TABLE metric_request_20250411 removes 24 hours of data instantly with no heap bloat, no vacuum pressure, and no DELETE lock. Creating partitions in advance (via a scheduled job or Postgres pg_partman extension) avoids insert failures on day boundaries. Contrast with a single-table design where a DELETE WHERE bucket_ts < now() - INTERVAL '30 days' runs row-by-row, generates dead tuples, and requires subsequent vacuum work.

Composite PK as the clustering key

The PK (service_name, endpoint, bucket_ts) is not just a uniqueness constraint — it is the physical sort order of each partition. The leading columns match the WHERE service_name = ? AND endpoint = ? AND bucket_ts >= ? predicate of every dashboard query, so Postgres can satisfy it with a single B-tree range scan on the PK index. No secondary index is needed for the primary access pattern.

Dialect notes

In Postgres, declarative partitioning with PARTITION BY RANGE (bucket_ts) and pg_partman for automation is the standard approach. In BigQuery, partition on bucket_ts (DATE or TIMESTAMP) and cluster by (service_name, endpoint); clustering handles the leading-column filter without a separate index. In Snowflake, define a CLUSTER BY (service_name, endpoint, DATE(bucket_ts)) micro-partition strategy rather than explicit partitions. In Redshift, set DISTKEY(service_name) and SORTKEY(service_name, endpoint, bucket_ts).

Senior signal: clearly articulate that percentiles are non-additive — scalar p-values cannot be merged across buckets — and that a mergeable sketch column (t-digest, DDSketch) is the correct fix; design the PK so its leading columns exactly match the dashboard filter pattern; and choose daily range partitions for zero-cost retention via DROP PARTITION rather than row-level deletes.
