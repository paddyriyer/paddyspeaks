A product analyst wants each user's longest stretch of inactivity — the biggest gap between leaving the app and coming back. This is a per-user, ordered-row problem: you need each session's end placed next to the following session's start. The interviewer is probing the LAG window function and whether you correctly clamp negative gaps to zero when sessions overlap.

1. The source data — app_session

user_idstarted_atended_at(narrative)

110:0010:3030-min gap after A
114:0014:30210-min gap after B — the longest
209:0009:30session A
209:2009:45starts before A ended — overlap, negative gap

Two edge cases. User 2 has an overlapping session (starts 09:20, before the 09:30 end) — the raw gap is negative and must be clamped to 0. User 3 has only one session, so there is no "next" session and they must be absent from the output entirely.

2. The SQL — LAG for the previous end, then clamp and aggregate

WITH gaps AS (
  SELECT user_id,
         LAG(ended_at) OVER (PARTITION BY user_id ORDER BY started_at) AS prev_end,
         started_at
  FROM app_session
),
clamped AS (
  SELECT user_id,
         GREATEST(EXTRACT(EPOCH FROM (started_at - prev_end))/60, 0) AS gap_min
  FROM gaps
  WHERE prev_end IS NOT NULL
)
SELECT user_id, MAX(gap_min)::int AS longest_gap_minutes
FROM clamped
GROUP BY user_id
ORDER BY longest_gap_minutes DESC, user_id;

LAG(ended_at) pulls the prior session's end into the current row. started_at - prev_end is the gap; EXTRACT(EPOCH ...)/60 converts the interval to minutes; GREATEST(..., 0) floors negative (overlap) gaps at zero. WHERE prev_end IS NOT NULL drops each user's first session, which has no predecessor.

3. The resulting analysis

User 3 is absent — their single session has prev_end IS NULL, so it is filtered out before the GROUP BY and no row survives. User 2's longest gap is 5, not a negative number: the 09:20-vs-09:30 overlap produced -10 minutes, which GREATEST raised to 0, leaving the genuine 5-minute gap as the maximum.

4. Strategic explanations

Why LAG is the right tool

The gap is between consecutive rows within a user, so you need each row to see the one before it. LAG(ended_at) OVER (PARTITION BY user_id ORDER BY started_at) does exactly that: partition isolates each user, the order defines "previous," and LAG reaches back one row. The self-join alternative — joining the table to itself on "next session" — is far clumsier and slower; LAG expresses ordered adjacency directly in one pass.

Clamping with GREATEST — overlap is not a negative gap

If session B starts before session A ends, started_B - ended_A is negative. A "gap of inactivity" cannot be negative — the user was never inactive. GREATEST(value, 0) floors it at zero, encoding the rule "overlapping sessions mean zero gap." Without the clamp, a heavily overlapping user could show a large negative number that MAX might still pick if every gap were negative — GREATEST makes the semantics correct, not just tidy.

Ordering by started_at — and a subtle overlap caveat

The window orders by started_at, so "previous" means "started earlier." For user 2 that gives the sequence A(09:00–09:30), B(09:20–09:45), C(09:50–10:00). Note B is compared to A's end (overlap, clamped) and C to B's end (5 min) — correct. But if sessions could start in one order and end in another, "previous by start" is a modelling choice; some definitions order by ended_at. State which ordering the gap definition assumes.

The prev_end IS NULL filter excludes single-session users for free

Every user's first session (by start order) has no predecessor, so LAG returns NULL. A user with only one session has only that first row, so after WHERE prev_end IS NOT NULL they have no rows at all and never reach the GROUP BY — satisfying "users with one session should not appear" without a special case. This is the same mechanism that makes user 3 vanish.

Dialect notes

LAG and PARTITION BY are standard window syntax (PostgreSQL, SQL Server, Oracle, MySQL 8+, modern SQLite). The interval maths differs: EXTRACT(EPOCH FROM interval)/60 is Postgres; SQL Server uses DATEDIFF(MINUTE, prev_end, started_at); MySQL TIMESTAMPDIFF(MINUTE, prev_end, started_at). GREATEST exists in Postgres and MySQL; SQL Server lacks it, so use CASE WHEN gap < 0 THEN 0 ELSE gap END.

Senior signal: reach for LAG(...) OVER (PARTITION BY user ORDER BY time) to compare consecutive rows instead of a self-join, clamp overlapping-session gaps to zero with GREATEST(gap, 0) so a negative interval is not mistaken for inactivity, and rely on the LAG-returns-NULL-for-the-first-row property to drop single-session users automatically.
