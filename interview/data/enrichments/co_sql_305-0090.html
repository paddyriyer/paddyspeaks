Tesla's manufacturing-execution team runs an assembly line where every station logs tasks with a status. The supervisor's morning question is blunt: which tasks are not finished? — the work-in-progress board for the shift. The query is a one-line filter, but the interview is in the details: how the predicate handles statuses other than the obvious ones, and what happens to NULLs.

1. The source data — tasks

task_idstatusname
1completeWeld frame
2in_progressAttach door
3pendingPaint body
4completeInstall seats
5blockedQA inspect

Three distinct non-complete statuses appear — in_progress, pending, blocked. That variety is the point: "incomplete" is not a single value, it is "everything that is not complete". A solution must catch all three.

2. The SQL — status <> 'complete'

SELECT * FROM tasks WHERE status <> 'complete';

The not-equal operator <> keeps every row whose status is anything other than the literal 'complete'. SELECT * returns the full task record for each.

The two complete tasks (1 and 4) drop out; the three unfinished ones remain — regardless of which incomplete status they carry. Defining the filter by exclusion ("not complete") rather than enumeration ("pending or in_progress or blocked") is what makes it robust.

4. Strategic explanations

Exclusion beats enumeration

A tempting alternative lists the incomplete states explicitly: WHERE status IN ('pending','in_progress','blocked'). It returns the same three rows today — but it is brittle. The day the line introduces a new status ('rework', 'awaiting_parts'), the IN-list silently misses it and tasks vanish from the board. status <> 'complete' needs no maintenance: anything that is not the single terminal state is incomplete by definition.

The NULL trap — <> silently drops NULL rows

This is the sharp edge. If a task has a NULL status, NULL <> 'complete' evaluates to unknown, not true — so that row is excluded from the result. A task with no status is almost certainly not complete and belongs on the board, yet the query quietly hides it. Make NULL handling explicit:

SELECT * FROM tasks
WHERE status IS DISTINCT FROM 'complete';

IS DISTINCT FROM is NULL-aware: it treats NULL as a normal, different value, so a NULL-status task is returned. (The verbose equivalent is status <> 'complete' OR status IS NULL.) Naming this trap is the single most important senior signal here.

Case and whitespace sensitivity

String equality is exact. 'Complete', 'COMPLETE', or 'complete ' with a trailing space would all fail the = 'complete' test and therefore pass the <> filter — a finished task wrongly shown as incomplete. If status values are not strictly controlled, normalise: lower(trim(status)) <> 'complete'. The real fix is an enum or a check constraint on the column so only canonical values can exist.

SELECT * — fine for a person, risky in code

SELECT * is convenient for an ad-hoc supervisor query. In a saved report or application code it is fragile: if the table gains or reorders columns, downstream consumers break, and it pulls more data than needed. Project the columns the board actually shows — task_id, name, status, assignee_id, due_date — in production.

Sharpening the board — the likely follow-up

"Incomplete" usually leads to "incomplete and overdue" or "ordered by urgency". The schema ships due_date and assignee_id, so the natural extension is:

SELECT task_id, name, status, assignee_id, due_date
FROM tasks
WHERE status IS DISTINCT FROM 'complete'
  AND due_date < CURRENT_DATE
ORDER BY due_date;

That turns a flat list into a prioritised, actionable shift board — anticipating it shows you read the schema, not just the question.

Senior signal: define "incomplete" by exclusion (<> 'complete') so new statuses need no code change, and above all flag that <> silently drops NULL-status rows — use IS DISTINCT FROM so a task with no status still appears on the board.
