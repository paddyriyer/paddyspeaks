Binary-search-on-the-partition is the trick. The naive merge-and-pick is O(m + n); the merge-then-stop-at-half is still O(m + n). To hit O(log(min(m, n))) you have to stop searching for a value and start searching for the partition index. The same pattern shows up wherever you compute a percentile over two pre-sorted streams without merging them: cross-region p50 aggregation in Datadog and Honeycomb, distributed quantile sketches in ClickHouse, federated histogram stitching in Prometheus remote-read. The hard part isn't the binary search — it's keeping the four boundary values straight (a_l, a_r, b_l, b_r) when one partition is at index 0 or m.

1. The problem in one sentence + the key insight

Given two sorted arrays, return the median of their union in O(log(min(m, n))). Insight: you don't search for the median value — you search for the partition index i in the shorter array such that a[:i] ∪ b[:half-i] is exactly the lower half of the merged array. The boundary condition max(a_l, b_l) ≤ min(a_r, b_r) is the only thing the binary search is checking. Binary-search the index, not the answer.

2. Worked example — m=4, n=5, trace 3 iterations

Let a = [1, 3, 8, 9] (m=4) and b = [7, 11, 18, 19, 21] (n=5). Total length 9 (odd), so half = (9 + 1) // 2 = 5 elements on the left. We binary-search i ∈ [0, 4].

iter 1:  lo=0, hi=4, i=2, j=half-i=3
         a = [1, 3 | 8, 9]            <- partition after index 2
         b = [7, 11, 18 | 19, 21]     <- partition after index 3
         a_l=3, a_r=8, b_l=18, b_r=19
         Check: a_l(3)  <= b_r(19)  YES
                b_l(18) <= a_r(8)   NO    -- b's left is too big
         => move LEFT in b means move RIGHT in a; lo = i+1 = 3

iter 2:  lo=3, hi=4, i=3, j=half-i=2
         a = [1, 3, 8 | 9]
         b = [7, 11 | 18, 19, 21]
         a_l=8, a_r=9, b_l=11, b_r=18
         Check: a_l(8)  <= b_r(18) YES
                b_l(11) <= a_r(9)  NO   -- still b's left too big
         => lo = i+1 = 4

iter 3:  lo=4, hi=4, i=4, j=half-i=1
         a = [1, 3, 8, 9 |]           <- i==m, a_r = +inf
         b = [7 | 11, 18, 19, 21]
         a_l=9, a_r=+inf, b_l=11, b_r=11
         Check: a_l(9) <= b_r(11) YES
                b_l(11) <= a_r(+inf) YES   <- partition found
         Total length odd -> median = max(a_l, b_l) = max(9, 11) = 11

Three iterations, log2(4) = 2 (plus the final check), zero merge work. The sentinel ±infinity values at the partition edges are what keep the boundary logic identical at i=0 and i=m.

3. The algorithm — code with inline comments

def findMedianSortedArrays(a, b):
    # Always binary-search the SHORTER array to keep the search space minimal
    # and to guarantee j = half - i stays within [0, n].
    if len(a) > len(b): a, b = b, a
    m, n = len(a), len(b)
    half = (m + n + 1) // 2            # size of the left side of the merged array
    lo, hi = 0, m
    while lo <= hi:
        i = (lo + hi) // 2             # partition in a after index i
        j = half - i                   # partition in b is forced by half
        # Sentinels make the boundary cases (i=0, i=m) uniform.
        a_l = a[i - 1] if i > 0 else float('-inf')
        a_r = a[i]     if i < m else float('inf')
        b_l = b[j - 1] if j > 0 else float('-inf')
        b_r = b[j]     if j < n else float('inf')
        if a_l <= b_r and b_l <= a_r:  # correct partition found
            if (m + n) % 2: return max(a_l, b_l)
            return (max(a_l, b_l) + min(a_r, b_r)) / 2
        if a_l > b_r: hi = i - 1       # a's left is too big, shrink right
        else:         lo = i + 1       # b's left is too big, grow i

Data structures: just the two input arrays — no heap, no merge buffer, no auxiliary storage. Invariant: at every step, exactly half elements lie strictly to the left of the two partitions combined; the binary search only adjusts where within a that boundary falls.

4. Complexity

Time: O(log(min(m, n))). Each iteration halves the search space on the shorter array; we never touch the longer one's interior. Space: O(1). No merge buffer, no recursion, no sentinel array — the ±inf comparisons are scalar.

For contrast: the textbook merge-then-pick is O(m + n) time / O(1) space; a 1B + 1B array pair would take ~2 seconds vs ~60 comparisons with this algorithm.

5. The interviewer's probes

Probe 1 — Empty input. If a = [], the algorithm enters with m=0, lo=0, hi=0, i=0; a_l = -inf, a_r = +inf, so the partition check trivially succeeds with b[half-1] as the answer. The sentinels are doing the work.

Probe 2 — All duplicates. a = b = [2, 2, 2]. Every partition is valid in the first iteration; we return 2. The ≤ in the boundary check (not strict <) is what makes ties work.

Probe 3 — Generalisation: k-th element of two sorted arrays. Replace half = (m + n + 1) // 2 with half = k and return max(a_l, b_l) unconditionally. Same algorithm, same complexity — useful for top-k queries over federated indexes.

Probe 4 — Three sorted arrays. Binary search no longer works because the partition isn't unique. Switch to a heap of size 3 (k-way merge) — O((m+n+p) log 3). Naming this gap is a senior-level move.

Probe 5 — Stream variant: median of a single growing stream. Wholly different algorithm — two heaps (max-heap for the lower half, min-heap for the upper half), O(log n) per insert, O(1) per median query. Worth naming.

Probe 6 — Numerical precision. Float division on huge integers can round; in Python 3 it's safe because integers are arbitrary precision until the final / 2. In Java/Go you'd use long arithmetic and only divide at the very end.

6. Where this pattern appears in production

  • Datadog / Honeycomb / New Relic — federated percentile aggregation: each region computes a local sorted summary; the global median across two regions uses this exact partition-based algorithm to avoid shipping the full data.
  • ClickHouse quantileExact — over sorted column shards, the cross-shard merge of medians uses partition-based selection, not full merge.
  • Prometheus remote-read federation — histogram quantile stitching between a local and a remote tier; the same "search the partition" idea avoids re-bucketing.
  • BigQuery APPROX_QUANTILES / Druid approxQuantile — t-digest internally, but the merge of two t-digests at query time uses a partition-search variant.
  • Spotify recommender system — selecting the k-th most relevant item across two pre-ranked candidate pools (e.g. user-collaborative + content-based) uses the k-th-element variant in Probe 3.
