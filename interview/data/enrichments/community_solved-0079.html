A trading-signals team wants every stretch of three or more consecutive days on which a stock rose every single day — a momentum streak. This is the canonical gaps-and-islands interview problem: the interviewer wants to see whether you can convert a row-by-row condition into stable group identifiers, then collapse each group into one summary row.

2. The SQL — canonical gaps-and-islands

WITH ordered AS (
    SELECT stock_id, stock_date, price,
           LAG(price) OVER (PARTITION BY stock_id ORDER BY stock_date) AS prev_price
    FROM stock_price
),
flagged AS (
    SELECT stock_id, stock_date, price,
           CASE WHEN price > prev_price THEN 1 ELSE 0 END AS is_increase
    FROM ordered
),
grouped AS (
    SELECT stock_id, stock_date, is_increase,
           SUM(CASE WHEN is_increase = 0 THEN 1 ELSE 0 END)
               OVER (PARTITION BY stock_id ORDER BY stock_date) AS group_id
    FROM flagged
),
runs AS (
    SELECT stock_id, group_id,
        MIN(stock_date) AS run_start,
        MAX(stock_date) AS run_end,
        COUNT(*) + 1    AS days_in_run
    FROM grouped
    WHERE is_increase = 1
    GROUP BY stock_id, group_id
    HAVING COUNT(*) >= 2
)
SELECT stock_id,
       (run_start - INTERVAL '1 day')::DATE AS run_start_date,
       run_end::DATE                        AS run_end_date,
       days_in_run
FROM runs
ORDER BY stock_id, run_start;

Stock 1's first run spans Jan 1–4 (three increases, 4 days). After the Jan-5 dip a new run climbs to Jan 7. Stock 2's lone run is reported once. Each increase row marks a day the price went up; a run of N increases therefore covers N+1 calendar days, which is why days_in_run = COUNT(*) + 1 and the start date is backdated one day.

4. Strategic explanations

The core trick — a cumulative count of "breaks" becomes a group id

Step 3 runs SUM(CASE WHEN is_increase = 0 THEN 1 ELSE 0 END) OVER (... ORDER BY stock_date). Every non-increase day bumps a running counter; every increase day leaves it unchanged. So all the increase rows between two breaks share an identical group_id — that constant value is the island label. This "running sum of a boolean" is the heart of gaps-and-islands and works in any SQL engine with window functions.

Why HAVING COUNT(*) >= 2, not 3

The grouped rows are the increase days only. A streak that spans 3 calendar days has exactly 2 day-over-day increases (day1→day2, day2→day3). So "3+ consecutive days" translates to "2+ increase rows" — HAVING COUNT(*) >= 2. Getting this off-by-one right is exactly what the interviewer is checking; the comment in the solution spells out the N-increases-span-N+1-days arithmetic.

Strict > — flat days break the run

price > prev_price is a strict inequality, so a day where the price is unchanged counts as a non-increase and ends the streak. That matches "increased every day". If the business wanted "non-decreasing", you would use >= — a one-character change with a real semantic difference worth confirming aloud.

The first-row NULL and the missing-day trap

For each stock's earliest date LAG returns NULL; price > NULL is unknown, so the CASE falls to ELSE 0 — the first day is correctly never an "increase". Separately, LAG compares the previous row, not the previous calendar day: if the stock skips a non-trading day, two rows three days apart are still treated as adjacent. If the requirement is strict calendar consecutiveness, add a check that stock_date = prev_date + 1 into the break condition.

What to say next — MATCH_RECOGNIZE and the row-number variant

Mention that newer engines (Oracle, Snowflake, Trino, DB2) express this directly with MATCH_RECOGNIZE — a pattern like UP+ over a row sequence — which is far more readable for runs. The other classic alternative is the ROW_NUMBER difference trick (subtract a per-group row number from a global one); both are worth naming to show you know the pattern has a family of solutions.

Senior signal: name this as gaps-and-islands, explain that a cumulative sum of break-flags manufactures the island id, justify HAVING COUNT(*) >= 2 via the N-increases-span-N+1-days off-by-one, and flag that LAG compares rows not calendar days. Bonus: cite MATCH_RECOGNIZE as the modern one-liner.
