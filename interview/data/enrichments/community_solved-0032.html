A Community automotive dataset records every car a manufacturer launched, by year. The product team wants a one-line sentence per company: did it launch more or fewer cars this year than last, and by how many. It is a conditional-aggregation question dressed up as a string-formatting one — and the interviewer is watching whether you count cleanly, handle the company that launched zero this year, and build the sentence safely.

Toyota launched 3 this year vs 1 last — net +2. Honda launched 1 vs 2 — net −1. Ford is the edge case: it launched nothing in 2024 — 0 vs 1 — and must still appear with a "less" message.

2. The SQL — conditional counts then concat_ws

with t1 as(
select company_name,
    count(case when year = 2024 then 1 else null end) as car_count_2024,
    count(case when year <> 2024 then 1 else null end) as car_count_2023,
    (count(case when year = 2024 then 1 else null end)
    - count(case when year <> 2024 then 1 else null end)) as diff
from car_launches
group by company_name
)
select t1.company_name,
    concat_ws('  ',company_name,'launched',abs(diff),
    case when diff > 0 then 'more'
      when diff < 0 then 'less' end,'car in the current year') as result_message
from t1

The CTE counts 2024 launches and non-2024 launches per company via COUNT(CASE ...), and stores the signed difference. The outer query takes the absolute value of that difference and picks "more" or "less" from its sign, then stitches the sentence with concat_ws.

ELSE NULL END) returns 0, not NULL, for a group with no matching rows, so the company survives the grouping. The double spaces are concat_ws's two-space separator showing between every field.

4. Strategic explanations

Why COUNT(CASE ... THEN 1 ELSE NULL END) works

COUNT ignores NULLs and counts everything else. The CASE emits 1 for matching rows and NULL for the rest, so COUNT tallies exactly the matches. COUNT(CASE WHEN cond THEN 1 END) — with the ELSE NULL left implicit — is identical and shorter. In Postgres the idiomatic form is the FILTER clause, which reads better:

count(*) FILTER (WHERE year = 2024) AS car_count_2024,
count(*) FILTER (WHERE year <> 2024) AS car_count_2023
The trap: year <> 2024 is "every other year", not "2023"

The "previous year" bucket is defined as not 2024. If the table also held 2022 or 2021 launches, they would all land in car_count_2023 and the comparison would silently become "2024 vs all-prior-years". The column alias even says 2023, which makes the bug invisible. If the question truly means last year specifically, write year = 2023. Naming this mismatch is a senior catch.

Hardcoded year vs a dynamic "current year"

2024 is baked into the SQL. The moment the calendar turns over, every "current year" sentence is wrong. A robust version derives it: EXTRACT(YEAR FROM CURRENT_DATE) for this year and that value minus one for last year. Hardcoding is acceptable only if the question explicitly pins the reference year — otherwise call out the maintenance hazard.

The diff = 0 gap in the CASE

The sentence CASE covers diff > 0 and diff < 0 but has no branch for diff = 0. A company that launched the same count both years gets a CASE result of NULL, and concat_ws — which skips NULL arguments — would quietly drop the word, producing "launched 0 car in the current year" with a missing adjective. Add WHEN diff = 0 THEN 'the same number of' (or similar) to close the gap.

Companies absent in one year entirely

This works only because every company appears in at least one year, so GROUP BY company_name creates a group for it. A brand that launched cars in 2024 but never before still appears (its 2023 count is 0). But a company that exists in some master table yet launched nothing in either year would be missing — if the report must list all known brands, drive the query from the company dimension and LEFT JOIN launches onto it.

Senior signal: use conditional aggregation (or FILTER) and know it returns 0 not NULL for empty groups, flag that year <> 2024 silently means "all other years", replace the hardcoded year with EXTRACT(YEAR FROM CURRENT_DATE), and close the missing diff = 0 branch so concat_ws never drops a word.
