A Community macro-economics dataset lists the world's largest companies, and an analyst wants the total market value per country — but only for countries that clear a $500B bar. It is a textbook GROUP BY, and the interviewer is checking one specific reflex: do you filter the aggregate with HAVING, or do you wrongly try to do it in WHERE?

1. The source data — global_companies

companycountrymarketvalue
AppleUSA400
MicrosoftUSA350
ToyotaJapan280
SonyJapan240
TataIndia150
Saudi AramcoSaudi Arabia600

The values are in billions. Note India: its only company is worth 150 — below the 500 threshold — so India must not appear in the result. Saudi Arabia is the single-company-country edge case: one row clears the bar on its own.

2. The SQL — GROUP BY with a HAVING filter

SELECT country, SUM(marketvalue) AS total_marketvalue
FROM global_companies
GROUP BY country
HAVING SUM(marketvalue) > 500
ORDER BY total_marketvalue DESC;

Group the rows by country, sum each group's market value, keep only the groups whose sum exceeds 500, and order the survivors largest-first.

3. The resulting analysis

countrytotal_marketvalue
USA750
Saudi Arabia600
Japan520

USA sums to 750, Japan to 520, Saudi Arabia to 600 — all three clear 500 and appear, sorted descending. India's 150 falls short and is gone. The threshold is applied to the group total, which is exactly what HAVING is for.

4. Strategic explanations

WHERE vs HAVING — the core distinction

WHERE filters individual rows before grouping; HAVING filters whole groups after aggregation. You cannot reference SUM(marketvalue) in WHERE — the sum does not exist yet at that stage of evaluation. This query needs a per-country total, which only exists post-GROUP BY, so the filter must be HAVING. Putting an aggregate in WHERE is a hard error: aggregate functions are not allowed in WHERE.

The trap: filtering rows when you meant to filter groups

A subtle wrong answer keeps companies worth more than 500 instead of countries:

-- WRONG intent: keeps individual big companies, then sums
SELECT country, SUM(marketvalue) AS total_marketvalue
FROM global_companies
WHERE marketvalue > 500
GROUP BY country;

On this data that returns only Saudi Arabia (600) — USA's 750 total vanishes because neither Apple nor Microsoft individually exceeds 500. WHERE marketvalue > 500 answers a completely different question. Always ask: is the threshold on a row, or on a group?

Combining both clauses

WHERE and HAVING coexist and run in order. To total only tech companies and then keep big-tech countries:

SELECT country, SUM(marketvalue) AS total_marketvalue
FROM global_companies
WHERE sector = 'Technology'   -- prune rows first
GROUP BY country
HAVING SUM(marketvalue) > 500 -- then prune groups
ORDER BY total_marketvalue DESC;

Filtering rows in WHERE first is also a performance win — fewer rows reach the grouping step.

NULL market values

If any marketvalue is NULL, SUM skips it — NULLs do not contribute and do not poison the total. A country whose companies are all NULL sums to NULL, and NULL > 500 is unknown, so HAVING drops that country. That is usually the desired behaviour, but worth saying aloud.

Ties on the threshold and the boundary

> 500 is strict: a country totalling exactly 500 is excluded. Confirm whether the business wants >=. And if two countries tie on total_marketvalue, ORDER BY total_marketvalue DESC leaves their relative order unspecified — add a tiebreaker like , country ASC for a deterministic, reproducible report.

Senior signal: articulate that WHERE filters rows pre-aggregation and HAVING filters groups post-aggregation, recognise that WHERE marketvalue > 500 silently answers a different question, and tighten the report with a deterministic tiebreaker and an explicit decision on the strict-vs-inclusive boundary.
