Manufacturing wants a parts explosion: given a bill of materials, how many of each raw material does one finished bike consume? Assemblies nest — a bike contains wheels, a wheel contains spokes — and quantities multiply down the tree. The interviewer is probing whether you can carry a running quantity product through a recursive CTE and correctly identify which parts are leaves (raw materials) versus sub-assemblies.

Two leaf materials — Bolt and SteelTube — are reached through more than one parent assembly. That is the edge case: their final quantity is the sum across all paths, not the figure from a single branch.

2. The SQL — recursive explosion with multiplied quantities

WITH RECURSIVE explode(part, qty) AS (
  SELECT child, qty FROM bom WHERE parent = 'Bike'
  UNION ALL
  SELECT b.child, e.qty * b.qty
  FROM explode e JOIN bom b ON b.parent = e.part
),
leaves AS (
  SELECT DISTINCT child FROM bom
  EXCEPT
  SELECT DISTINCT parent FROM bom
)
SELECT e.part AS leaf_material, SUM(e.qty) AS total_qty
FROM explode e JOIN leaves l ON l.child = e.part
GROUP BY e.part
ORDER BY e.part;

The anchor takes Bike's direct children at face quantity. The recursive arm descends one level and multiplies: a child needed b.qty times per parent, and the parent itself needed e.qty times, so the child is needed e.qty * b.qty times. leaves isolates raw materials — children that are never a parent — via set difference. The final SUM totals each leaf across every path.

Bolt totals 24: the Frame branch contributes 1×12, the Wheel branch 2×6 — SUM adds both. Spoke is 64 because 32 multiplied by the 2-wheel quantity. SteelTube sums 5 and 1 from its two parents. Sub-assemblies (Frame, Wheel, Handlebar) appear in explode but are filtered out by the leaves join.

4. Strategic explanations

Quantities multiply, they do not add, on the way down

The recursive arm's key expression is e.qty * b.qty. If a bike needs 2 wheels and each wheel needs 32 spokes, the bike needs 2×32 = 64 spokes — multiplication, because each level scales the level below it. A common mistake is to add or to forget the parent factor and report 32. The running product is the accumulator state carried through the recursion, exactly analogous to a running sum but with *.

Multiply down a branch, sum across branches

Two distinct operations are in play. Within one root-to-leaf path, quantities multiply (the recursive arm). Across different paths that reach the same leaf, quantities add (the outer SUM). Bolt arrives via Frame and via Wheel as two separate explode rows; neither is wrong, and the total demand is their sum. Conflating the two — multiplying across branches, or summing within one — is the classic BOM bug.

Defining "leaf" by set difference

A raw material is a part that is consumed but never itself decomposed: it appears as a child but never as a parent. SELECT child FROM bom EXCEPT SELECT parent FROM bom computes exactly that. Without the leaves filter the result would also list Frame, Wheel, and Handlebar — double-counting, since their materials are already exploded into the leaves. EXCEPT deduplicates automatically, so the explicit DISTINCTs are belt-and-braces.

DAG vs tree — why no double-count despite shared parts

The graph is a DAG: Bolt has two parents, so it is not a tree. That is fine here because each occurrence is a genuine, separate requirement — bolts for the frame and bolts for the wheels really are distinct demand and should sum. The danger a DAG does introduce is non-termination if it were cyclic; the question guarantees no cycles, so the recursion ends when it reaches leaves. State that you are relying on the acyclic guarantee.

Termination, performance, dialect

Recursion stops because leaves have no bom rows as parents, so the join produces nothing and the iteration drains. A cycle (part containing itself) would loop forever — defend with a visited-path array or a depth cap. WITH RECURSIVE is standard in Postgres, SQLite, MySQL 8+, and SQL Server (as WITH); Oracle also supports CONNECT BY. For deep BOMs, an index on bom(parent) keeps each recursive join fast.

Senior signal: in a BOM explosion, multiply quantities descending a branch and sum across branches that reach the same leaf, define a leaf as child EXCEPT parent so sub-assemblies are not double-counted, and note explicitly that the DAG must be acyclic for the recursion to terminate.
