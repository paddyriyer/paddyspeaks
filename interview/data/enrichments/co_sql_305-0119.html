X (formerly Twitter) runs an internal "spaces booking" system — conference rooms reserved by team. Operations gets a complaint: "two meetings double-booked the same room at the same time." The classic SQL problem: find pairs of bookings on the same room whose time ranges overlap. The trick is the two-condition overlap predicate; getting it wrong produces missed conflicts or false positives.

1. The source data — bookings

R_A has the conflicts: B_001/B_002 overlap, B_001/B_005 overlap, B_002/B_005 overlap. B_003 is clean. B_004 is on a different room.

2. The SQL — the two-condition overlap predicate

SELECT a.booking_id AS b1, b.booking_id AS b2
FROM bookings a
JOIN bookings b
  ON a.room_id = b.room_id
 AND a.booking_id < b.booking_id           -- dedup the pair
 AND a.start_dt  < b.end_dt                -- a starts before b ends
 AND b.start_dt  < a.end_dt;               -- b starts before a ends

4. Strategic explanations

The two-condition overlap rule — the only right way

Two intervals [a_start, a_end] and [b_start, b_end] overlap iff:

a.start_dt < b.end_dt   AND   b.start_dt < a.end_dt

The intuition: a starts before b ends AND b starts before a ends. If either fails, one finishes before the other begins. Junior engineers reach for "BETWEEN" or check just one endpoint — both are wrong. Memorize this two-line predicate.

Strict < vs <= — touching ranges

What about [9:00, 10:00] and [10:00, 11:00]? Are they "overlapping"?

  • Half-open intervals (industry standard, [start, end)): the second starts when the first ends — not overlapping. Use <.
  • Closed intervals ([start, end]): they share the boundary 10:00 — overlapping. Use <=.

Calendar systems are universally half-open (the 10:00 meeting "starts" at 10:00 sharp, the 9:00 meeting "ends" at 10:00 sharp, they don't conflict). Strict < is the calendar-correct choice.

The dedup trick — booking_id < booking_id

Without the a.booking_id < b.booking_id predicate, the join produces both directions (B_001/B_002 and B_002/B_001) plus self-pairs (B_001/B_001). The strict-less-than picks a canonical order and emits each pair exactly once — same trick as the duplicate-email and pizza-toppings problems.

The "find conflicts for one specific booking" variant

Real-world "is my new booking valid?":

SELECT * FROM bookings
WHERE room_id  = :new_room
  AND start_dt < :new_end
  AND end_dt   > :new_start;

If this returns any row, the new booking conflicts. The two-condition overlap, applied with the candidate as one side.

The naive "BETWEEN" trap — why it's wrong
-- WRONG — misses conflicts where a fully contains b
WHERE a.start_dt BETWEEN b.start_dt AND b.end_dt
   OR a.end_dt   BETWEEN b.start_dt AND b.end_dt

This catches endpoint-inside overlaps but misses the case where one interval fully contains the other (both endpoints are outside b's range, but b is wholly inside a). The two-condition rule handles all four overlap shapes (left-overlap, right-overlap, contained, contains) in a single predicate.

Performance — the index that matters

For large tables, a self-join on time ranges is O(n²) without help. Standard indexes don't accelerate range overlap; what does:

  • Partition the table by date and only join within the same day's partition.
  • Postgres: GIST index on tstzrange(start_dt, end_dt) + && overlap operator — sub-linear lookup.
  • BigQuery / Snowflake: clustering on (room_id, start_dt).
-- Postgres GiST — production-grade overlap query
SELECT a.booking_id, b.booking_id
FROM bookings a JOIN bookings b
  ON a.room_id = b.room_id
 AND a.booking_id < b.booking_id
 AND tstzrange(a.start_dt, a.end_dt) && tstzrange(b.start_dt, b.end_dt);
The three-or-more-overlap follow-up

"Find rooms that had three or more concurrent bookings at any moment." Sweep-line via window function:

WITH events AS (
  SELECT room_id, start_dt AS t, +1 AS delta FROM bookings
  UNION ALL
  SELECT room_id, end_dt,         -1        FROM bookings
)
SELECT room_id, t,
  SUM(delta) OVER (PARTITION BY room_id ORDER BY t ROWS UNBOUNDED PRECEDING) AS active
FROM events
ORDER BY room_id, t;

Filter WHERE active >= 3 for the moments of triple-booking. Sweep-line is the canonical algorithm for any "how many active simultaneously?" question.

Senior signal: nail the two-condition overlap predicate without hesitation, articulate strict-vs-non-strict for half-open intervals, contrast it cleanly against the BETWEEN trap, and pre-empt the perf question with a GiST/range-type sketch. The sweep-line for triple-booking is the over-the-bar move.
