A product analytics team wants to understand how users start a session. They define a session as a run of events with no gap longer than 30 minutes between consecutive events, then ask: "for each session, what was the user doing first — did they open the app, or land mid-flow on a view?" The interviewer is probing two skills at once: sessionization (turning a raw event stream into bounded sessions) and first-per-group selection. Both are gap-and-island problems, and doing them together is the real test.

The edge case hides in plain sight: the 09:55→10:30 gap is 35 minutes, which exceeds 30. The schema's inline comments only call out the 40m and 60m gaps, but a correct query must split here too. User 1 ends up with four sessions, not three.

2. The SQL — gap flag, running sum, first-per-session

WITH flagged AS (
  SELECT user_id, ts, event_type,
         CASE
           WHEN LAG(ts) OVER (PARTITION BY user_id ORDER BY ts) IS NULL THEN 1
           WHEN EXTRACT(EPOCH FROM (ts - LAG(ts) OVER (PARTITION BY user_id ORDER BY ts)))/60 > 30 THEN 1
           ELSE 0
         END AS is_new_session
  FROM event
),
sessioned AS (
  SELECT user_id, ts, event_type,
         SUM(is_new_session) OVER (PARTITION BY user_id ORDER BY ts
                                    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS session_id
  FROM flagged
),
first_in_session AS (
  SELECT user_id, session_id, ts, event_type,
         ROW_NUMBER() OVER (PARTITION BY user_id, session_id ORDER BY ts) AS rn
  FROM sessioned
)
SELECT user_id, session_id, ts, event_type
FROM first_in_session
WHERE rn = 1
ORDER BY user_id, session_id;

Three stages. flagged marks every row that opens a new session with a 1 (the first row of a user, or any row more than 30 minutes after its predecessor). sessioned turns those flags into a stable session number with a running SUM — each 1 bumps the counter. first_in_session ranks rows inside each session and keeps rn = 1.

User 1 produces four rows because four sessions formed. Session 2's first event is a close and session 3's is a view — the "first event" is whatever the user happened to do first, not necessarily an open. That is exactly the insight the interviewer wants surfaced.

4. Strategic explanations

Why the running sum produces stable session IDs

The trick is that a cumulative SUM of 0/1 flags is monotonic and only increments at boundaries. Every row between two boundaries shares the same partial sum, so rows naturally cluster into "islands" with identical session_id. This is the canonical gaps-and-islands pattern: detect the boundary, then run a sum to label the islands. No self-join, one pass after the sort.

Why ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW is explicit

An ordered window SUM defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. With distinct timestamps RANGE and ROWS agree, but if two events share an exact timestamp, RANGE treats them as peers and gives both the same running total — which can merge or mislabel sessions. Writing ROWS explicitly makes the frame row-by-row and removes that ambiguity. Stating this distinction unprompted is a senior tell.

The 30-minute boundary — strict vs inclusive

The rule is "gap longer than 30 minutes" so the code uses > 30. A gap of exactly 30 minutes stays in the same session. If the spec said "30 minutes or more ends the session" you would switch to >= 30. Computing the gap as EXTRACT(EPOCH FROM (ts - LAG(ts)))/60 converts an interval to minutes; comparing intervals directly (ts - LAG(ts) > INTERVAL '30 minutes') is equivalent and arguably cleaner.

ROW_NUMBER vs RANK for first-per-session

ROW_NUMBER is correct here: each session has exactly one chronologically-first event, and timestamps within a session are distinct, so rn = 1 picks one row deterministically. If two events could share the session's earliest timestamp, ROW_NUMBER would still pick exactly one (arbitrarily), whereas RANK would return both. Choose based on whether ties should yield one row or many — the question asks for "the first event", singular, so ROW_NUMBER fits.

Single-member groups and NULL gaps

User 2 and the very first row of each user exercise the boundary logic: LAG returns NULL for the first row, and the CASE explicitly catches that with IS NULL THEN 1. Without that branch the first event would get flag 0, the running sum would start at 0, and session numbering would be off by one (or zero-based). The IS NULL arm is what guarantees session IDs are 1-based.

Senior signal: recognise sessionization as gaps-and-islands — flag boundaries with LAG, label islands with a running SUM of those flags, then apply ROW_NUMBER for first-per-group. Handle the first-row NULL from LAG explicitly so IDs stay 1-based, write ROWS not RANGE to avoid timestamp-tie surprises, and read every gap in the data — the 35-minute split is easy to miss.
