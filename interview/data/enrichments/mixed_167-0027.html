An operations analyst wants every order whose SKU is either a promotional code or stamped with the 2025 vintage — a quick segmentation for a campaign post-mortem. The reference solution uses two LIKE patterns joined by OR: a prefix match and a suffix match. The interviewer is probing whether you know how anchored wildcards behave, how NULL SKUs are treated, and why this pattern cannot use a normal index.

Two rows decide the interview. Order 3's SKU matches both patterns (starts with PROMO- and ends with -2025). Order 6 has a NULL SKU. A correct answer must handle the double match without duplicating the row, and explain why NULL drops out.

2. The SQL — anchored LIKE with OR

SELECT * FROM orders WHERE sku LIKE 'PROMO-%' OR sku LIKE '%-2025';

'PROMO-%' matches any string that begins with PROMO- (% trails). '%-2025' matches any string that ends with -2025 (% leads). OR accepts a row if either pattern matches.

Four rows return. Order 3 satisfies both patterns but appears once — WHERE filters rows, it does not multiply them, and OR is a single boolean test. Order 4 (STD-4400) matches neither. Order 6's NULL SKU is excluded: NULL LIKE anything is unknown, and unknown OR unknown is still unknown — not true.

4. Strategic explanations

A double match does not duplicate the row

A frequent worry is that an order matching both LIKE clauses gets returned twice. It does not. WHERE evaluates one boolean per row; OR short-circuits to true and the row is emitted once. Duplication only happens with JOINs or UNION ALL — never from OR in a single-table filter. (If you had written this as two SELECTs combined with UNION ALL, order 3 would appear twice; plain UNION would dedupe it.)

NULL SKUs vanish — three-valued logic

Order 6's NULL SKU produces NULL LIKE 'PROMO-%' → unknown and NULL LIKE '%-2025' → unknown; unknown OR unknown → unknown, which WHERE treats as "exclude". So NULL-SKU orders silently disappear. That is usually correct here, but if "uncategorised" orders must be surfaced, add an explicit OR sku IS NULL. Never let NULL filtering be accidental.

Anchored vs unanchored wildcards — and the index consequence

'PROMO-%' is left-anchored: the pattern's fixed text is at the start, so a B-tree index on sku can range-scan the PROMO- prefix — fast. '%-2025' is right-anchored with a leading %: a B-tree index is useless because the engine cannot know where matching strings start, so it scans every row. The mixed query therefore degrades to a full scan overall. Knowing which wildcard placement preserves index usability is the senior point.

Making the suffix match index-friendly

If '%-2025' matters at scale, two Postgres options help. An expression index on the reversed string — CREATE INDEX ON orders (reverse(sku)) with the predicate rewritten as reverse(sku) LIKE reverse('-2025') || '%' — turns the suffix search into a prefix search. Or a trigram GIN index (pg_trgm) accelerates arbitrary substring and suffix matches directly. Either beats a repeated sequential scan.

Case sensitivity and the underscore wildcard

LIKE is case-sensitive in Postgres, so promo-winter would not match 'PROMO-%'; use ILIKE if SKUs are entered inconsistently. Also note _ is a single-character wildcard in LIKE — if a SKU literally contains an underscore you must escape it (LIKE '%\_2025'). SKUs here use hyphens, so it is moot, but it is the kind of detail an interviewer rewards you for raising.

Senior signal: know that OR in a WHERE clause filters without duplicating — a row matching both LIKE patterns returns once — that NULL SKUs drop out via three-valued logic, and that a left-anchored 'PROMO-%' can use a B-tree index while a leading-wildcard '%-2025' forces a scan unless you add a reverse-expression or trigram index.
