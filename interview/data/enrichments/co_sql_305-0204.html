Etsy's logistics team wants the typical basket size — the most common number of items per order — to size packaging and shipping bands. "Most common" means statistical mode. The reference computes it by counting how often each item_count appears and taking the top one. The interviewer is checking whether you reach for GROUP BY ... ORDER BY COUNT(*), and whether you notice the answer is unstable when two basket sizes tie.

Basket size 3 occurs three times, size 1 twice, size 2 once. There is a clear winner here — but section 4 shows what happens when 3 and 1 each occur the same number of times, which is when the query gets fragile.

2. The SQL — GROUP BY, order by frequency, LIMIT 1

SELECT item_count FROM orders GROUP BY item_count
ORDER BY COUNT(*) DESC LIMIT 1;

GROUP BY item_count makes one row per distinct basket size, COUNT(*) counts orders in each, ORDER BY COUNT(*) DESC sorts the most frequent first, and LIMIT 1 takes the winner. A clean way to compute a mode without a dedicated function.

3. The resulting analysis

item_count
3

One row: basket size 3 is the mode — it appears in three orders, more than any other size. The query answers the question. But it answers it with a confidence the data does not always justify: the moment two sizes are equally common, LIMIT 1 still returns exactly one, chosen arbitrarily.

4. Strategic explanations

The tie trap — LIMIT 1 hides a multimodal distribution

Suppose sizes 3 and 1 both appear twice. ORDER BY COUNT(*) DESC puts them adjacent with equal counts, and LIMIT 1 picks whichever the planner sorted first — non-deterministic, and it silently conceals that the distribution has two modes. To surface ties, rank and keep all winners:

SELECT item_count FROM (
  SELECT item_count, RANK() OVER (ORDER BY COUNT(*) DESC) AS rk
  FROM orders GROUP BY item_count
) t WHERE rk = 1;

This returns every basket size tied for most common — one row or several. Telling the interviewer "if it's bimodal, do you want both, or a tiebreak rule?" is the senior move.

Postgres has a built-in: MODE()

You do not have to hand-roll this. The ordered-set aggregate MODE() computes it directly:

SELECT MODE() WITHIN GROUP (ORDER BY item_count) AS typical_basket
FROM orders;

It returns the most frequent value, and on a tie it returns the first by the sort order — deterministic, unlike bare LIMIT 1. Knowing MODE() exists signals fluency; the GROUP BY form is the portable fallback for engines without it.

NULL item counts — are they a basket size?

If some orders have a NULL item_count, GROUP BY gathers them into their own group, and that NULL group could even win if NULLs are common — the query would return NULL as "the typical basket size", which is nonsense. MODE() ignores NULLs by definition. With the GROUP BY form, filter explicitly: WHERE item_count IS NOT NULL. Decide whether a NULL means "unknown" (exclude) or "zero items" (coalesce) before counting.

Mode vs average vs median — pick the right "typical"

"Typical basket size" is ambiguous. The mode is the most common value — great for discrete choices like packaging bands. But if the distribution is skewed, the business might actually want the median (PERCENTILE_CONT(0.5)) or the mean (AVG). The mode of a long-tail order-size distribution can be 1 even when most items ship in larger baskets. Confirm which statistic the decision needs.

Edge cases: empty table and a status filter

On an empty table the GROUP BY form returns zero rows and MODE() returns a single NULL row — a subtle difference worth knowing. Also consider whether cancelled orders belong in the mode: a cancelled order's item count may never have shipped, so a WHERE status = 'completed' filter is often the correct, unstated requirement.

Senior signal: compute the mode with GROUP BY ... ORDER BY COUNT(*) or the built-in MODE() WITHIN GROUP, recognise that bare LIMIT 1 hides a multimodal tie, exclude NULL counts so they cannot win spuriously, and question whether "typical" should be the mode, median, or mean.
