Stripe payments move through a state machine: pending → processing → succeeded on the happy path, or pending → failed on the error path. Some states are terminal (no further transitions allowed); others are intermediate. The query produces one row per state period, flags terminal states, and uses IS DISTINCT FROM for cleaner NULL handling.

1. The source data — payment_state_log

payment_idstatechanged_at
P_001pending10:00:00
P_001processing10:00:05
P_001succeeded10:00:08

2. The SQL — IS DISTINCT FROM + terminal-state flag

WITH changes AS (
  SELECT payment_id, changed_at, state,
    LAG(state) OVER (PARTITION BY payment_id ORDER BY changed_at) AS prev_state
  FROM payment_state_log
),
filtered AS (
  SELECT payment_id, changed_at, state
  FROM changes
  WHERE state IS DISTINCT FROM prev_state
)
SELECT payment_id, state,
  changed_at AS state_start,
  LEAD(changed_at) OVER (PARTITION BY payment_id ORDER BY changed_at) AS state_end,
  CASE WHEN state IN ('succeeded','failed','refunded','disputed') THEN 1 ELSE 0 END AS is_terminal
FROM filtered
ORDER BY payment_id, state_start;

3. Strategic explanations

The IS DISTINCT FROM elegance

state IS DISTINCT FROM prev_state treats NULL as a value: 'pending' IS DISTINCT FROM NULL evaluates to TRUE. Replaces the state != prev_state OR prev_state IS NULL pattern with a single null-safe expression. Postgres / SQLite / BigQuery / DuckDB all support it; MySQL has the <=> spaceship operator (use NOT (a <=> b)); SQL Server forces the COALESCE-to-sentinel workaround.

Terminal vs intermediate — why it matters

Stripe's payment state machine has a one-way valve: once succeeded, the state can't go back to processing (refunds are a NEW state, not a reverse transition). The is_terminal flag enables:

  • Latency analysis — only meaningful for non-terminal periods
  • Stuck-payment alarm — pending period > 5 min = something's wrong
  • Final-state distribution — WHERE is_terminal = 1 + GROUP BY state = success / fail breakdown
The latency metric — what production uses
SELECT AVG((julianday(state_end) - julianday(state_start)) * 86400) AS avg_seconds
FROM (/* this query */)
WHERE state = 'pending' AND state_end IS NOT NULL;

Average pending duration is the auth-rate-issue early warning. Spikes above baseline → bank-network problem.

The chargeback "extension" pattern

A succeeded payment can later receive a chargeback (60-90 days later). Architecturally that's a NEW row in the log with state='disputed' — not an update to the existing succeeded row. Append-only is the integrity contract.

The retry-storm detection

If a payment_id has more than 2 transitions through pending → failed → pending → failed, that's a retry storm — likely fraud or card-testing. Detection:

SELECT payment_id, COUNT(*) AS transitions
FROM (/* this query */)
GROUP BY payment_id HAVING COUNT(*) >= 4;
Senior signal: use IS DISTINCT FROM for null-safe inequality, flag terminal states inline so downstream queries can filter cleanly, name the append-only pattern for late-arriving chargebacks, and connect pending-duration to bank-network health monitoring. Retry-storm detection via COUNT-of-transitions is the fraud-side over-the-bar.
