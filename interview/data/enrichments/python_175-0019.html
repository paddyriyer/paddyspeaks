Apple's iCloud team de-duplicates device identifiers before a sync batch — if any UDID appears twice, the batch is malformed and gets rejected. "Contains Duplicate" is an Easy warm-up, but the interviewer is really checking one reflex: do you reach for a set immediately, or do you write nested loops? The one-liner is correct; the conversation around it is what scores.

1. The input data

nums = [1, 2, 3, 1]

Four integers, with 1 appearing at index 0 and index 3. A clean example also worth discussing: [1, 2, 3, 4] — all distinct.

2. The approach

The solution builds set(nums), which discards every repeated element by definition, then compares lengths. If len(nums) != len(set(nums)), at least one value collapsed during set construction — that is a duplicate. One pass to build the set, one cheap length read.

The elegance is that it never has to find the duplicate or count occurrences; the size difference is a sufficient witness. set construction is amortized O(1) per insertion via hashing.

4. Strategic explanations

Complexity

O(n) time to hash every element into the set, O(n) extra space for the set itself. The nested-loop alternative is O(n²) time / O(1) space — only defensible if the interviewer explicitly forbids extra memory.

The idiomatic alternative — early exit

The one-liner always consumes the whole list. To bail the instant a repeat is seen:

def containsDuplicate(nums):
    seen = set()
    for n in nums:
        if n in seen:
            return True
        seen.add(n)
    return False

Same O(n) worst case, but on [1, 1, <a million more>] it returns after two elements instead of hashing the entire input. For an interview, mention both and let the interviewer pick.

The edge case — empty and single-element lists

[] and [5] both yield len == len(set), so the function correctly returns False — no special-casing needed. Be careful with unhashable elements though: a list of lists raises TypeError on set construction. And nan is famously not equal to itself, so a list with two float('nan') values keeps both in the set and reports no duplicate.

The follow-up — "Contains Duplicate II / III"

The natural escalation is "duplicate within index distance k" (sliding-window set) or "near-duplicate within value distance t" (bucketing / SortedList). The interviewer wants to see that you can graduate from a global set to a windowed one without rewriting from scratch.

The trap — sorting first

Sorting and scanning adjacent pairs works but is O(n log n) — strictly worse than the hash set and it mutates or copies the input. Candidates who reach for sorted() here are signalling they default to comparison-based thinking when hashing is available.

Senior signal: state O(n)/O(n) instantly, then volunteer the early-exit variant and the unhashable-element / nan caveats unprompted — turning a trivial question into evidence you think about input shape and failure modes.
