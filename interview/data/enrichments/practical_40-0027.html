A merchandising team wants to know the two best-selling products inside each category, ranked by total revenue, with ties broken by product_id. That is a top-N-per-group problem — the classic test of whether you reach for a partitioned ranking window rather than a per-category subquery or a hand-written loop. The interviewer also wants to see you separate the aggregation step from the ranking step cleanly.

Two edge cases collide here. Electronics has a three-way revenue tie at 500 — ROW_NUMBER must pick exactly two and break the tie deterministically. And Books has a single product: top-2 of a one-member group must still return that one row, not error or pad.

2. The SQL — aggregate, then rank with ROW_NUMBER

WITH product_revenue AS (
    SELECT p.category, p.product_id,
           SUM(o.order_total) AS revenue
    FROM orders o
    JOIN products p ON p.product_id = o.product_id
    GROUP BY p.category, p.product_id
),
ranked AS (
    SELECT category, product_id, revenue,
           ROW_NUMBER() OVER (
               PARTITION BY category
               ORDER BY revenue DESC, product_id ASC
           ) AS rn
    FROM product_revenue
)
SELECT category, product_id, revenue
FROM ranked
WHERE rn <= 2
ORDER BY category, rn;

The first CTE collapses orders into one revenue total per product. The second ranks products within each category by revenue. The outer query keeps ranks 1 and 2 — the top two per category.

4. Strategic explanations

Why ROW_NUMBER here — and why it is a judgement call

The brief says "top 2, ties broken by product_id". ROW_NUMBER returns exactly two rows per category because it assigns distinct 1, 2, 3… even to revenue ties — the product_id ASC secondary sort decides who is 1 vs 2. Had the brief said "include all products tied for the top 2 revenue figures", you would switch to RANK or DENSE_RANK and Electronics would return all three 500-grossers. The choice of ranking function is the answer to "what do we do with ties" — name it explicitly.

Why two CTEs — aggregate before you rank

You cannot rank by revenue until revenue exists. The first CTE does the GROUP BY; the second applies the window over that aggregated result. Trying to fuse them — a window function over SUM in one SELECT — fights SQL's evaluation order and gets messy fast. Separating "compute the measure" from "rank the measure" into named steps is both correct and far more readable.

The JOIN hides products with zero orders

The inner JOIN drops any product that has never been ordered — it has no row in orders. If the report should show such products at revenue 0, switch to products LEFT JOIN orders and wrap the sum: COALESCE(SUM(o.order_total), 0), since SUM over zero rows yields NULL. Whether un-ordered products belong in a "top sellers" list is a business call — surface the assumption rather than letting the JOIN decide silently.

WHERE rn <= 2 must be in an outer query

A window function cannot be referenced in the same query's WHERE — WHERE ROW_NUMBER() OVER(…) <= 2 is illegal, because WHERE is evaluated before window functions. The rank must be produced in a subquery or CTE and filtered one level up. That ordering rule is why the ranked CTE exists; it is the most common beginner error on top-N-per-group.

Dialect notes

CTEs and ROW_NUMBER are standard in Postgres, MySQL 8+, SQLite 3.25+, Snowflake, and SQL Server. SQL Server and Snowflake also offer QUALIFY rn <= 2, which filters on a window function inline and removes the need for the outer query — terser, but not portable. Postgres has no QUALIFY, so the CTE-then-filter form shown here is the portable answer.

Senior signal: solve top-N-per-group by aggregating in one CTE then ranking with ROW_NUMBER() OVER (PARTITION BY group ORDER BY measure DESC, tiebreak) in the next, filter rn <= N in an outer query because WHERE precedes window functions, justify ROW_NUMBER vs RANK by the tie rule, and flag that an inner JOIN silently excludes never-ordered products.
