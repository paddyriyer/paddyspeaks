A growth analyst wants to know the longest stretch a user went without visiting — the biggest "engagement gap." The twist is the boundary: a user's last visit has no following visit, so the gap is measured against a fixed cap date 2025-04-01. The interviewer is probing whether you reach for LEAD with a default argument instead of fragile self-joins, and whether you handle the user who visited only once.

1. The source data — user_visits

user_idvisit_date(narrative)
12025-01-01three visits, irregular gaps
12025-01-054 days after the first
12025-01-20last visit → gap runs to the cap
22025-02-01two visits, one day apart
22025-02-02last visit → gap runs to the cap
32025-03-01single-visit user — the edge case

User 3 is the trap: they have no consecutive pair at all. A self-join on "next visit" would simply drop them. The cap date has to give every user, even a one-visit user, exactly one measurable window.

2. The SQL — LEAD with a default fallback

WITH paired AS (
  SELECT user_id, visit_date,
         LEAD(visit_date, 1, DATE '2025-04-01') OVER (PARTITION BY user_id ORDER BY visit_date) AS next_date
  FROM user_visits
)
SELECT user_id, MAX(next_date - visit_date)::int AS biggest_window
FROM paired
GROUP BY user_id
ORDER BY user_id;

The three-argument form of LEAD(expr, offset, default) returns default when there is no row offset positions ahead. Here every user's final row — including user 3's only row — gets next_date = 2025-04-01, so subtraction always yields a real interval.

3. The resulting analysis

user_idbiggest_windowsignal
171gaps 4, 15, then 20 Jan → 1 Apr = 71 days; the cap leg wins

331single visit: 1 Mar → 1 Apr = 31 days, supplied entirely by the default

User 3 produces a row at all only because of the LEAD default. Without it, their single visit would have next_date = NULL, the subtraction would be NULL, and MAX over a single NULL is NULL — or the row vanishes entirely under a self-join formulation.

4. Strategic explanations

Why LEAD beats a self-join here

The classic alternative pairs each visit with "the minimum later visit for the same user" via a correlated subquery or self-join — O(n²) comparisons and, worse, it structurally cannot emit the last visit's gap because there is no later row to join to. LEAD sorts each partition once, reads the neighbour in a single pass, and its default argument cleanly models "and if there is no neighbour, use the cap."

Date arithmetic returns an integer in Postgres

date - date yields an integer count of days in PostgreSQL, so next_date - visit_date is already a day count; the ::int cast is cosmetic insurance. In other engines this differs sharply — see the dialect note — so never assume the bare subtraction is portable.

The single-member partition is the whole point

Interviewers seed exactly one user with one visit to see if your answer collapses. Because LEAD's default fires per partition regardless of partition size, user 3 behaves identically to everyone else. If you had used LEAD(visit_date) without the third argument and then COALESCEd afterwards, you would get the same result — but stating the default inline is tighter and shows you know the full signature.

Dialect — date subtraction is not universal

MySQL needs DATEDIFF(next_date, visit_date); next_date - visit_date there does numeric coercion and gives wrong answers. SQL Server uses DATEDIFF(day, visit_date, next_date). BigQuery uses DATE_DIFF(next_date, visit_date, DAY). The LEAD(expr, 1, default) three-argument form, by contrast, is standard SQL and works across all of them — only the subtraction needs translating.

What to say next

Mention that the cap date is a hard-coded assumption: in production you would parameterise it (often CURRENT_DATE) so the report stays correct as time passes. And if two users could share a (user, date) row — ruled out here — you would first dedupe, because LEAD over duplicate dates yields a zero-length gap that muddies the maximum.

Senior signal: reach for LEAD(expr, 1, default) to pair each row with its successor and model the missing-successor boundary in one stroke, recognise that the single-visit user only survives because of that default, and flag that date - date is Postgres-specific while the windowing is portable.
