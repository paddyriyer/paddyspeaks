A social platform wants to measure engagement gaps: for users who posted more than once in 2025, how many days separated their first post from their last? This "hiatus" metric flags users who were active early then went quiet. The interviewer is checking three things at once — date-range filtering, the difference between distinct posting days and raw post rows, and date subtraction arithmetic.

The traps cluster around the word "distinct". User 103 has three rows but only two distinct days. User 105 has three rows that are all the same day — one distinct day, so 105 must be excluded. User 104 posted twice, but one post is in 2024; after the date filter only one 2025 day remains, so 104 is also excluded.

2. The SQL — range filter, then group with a distinct-day guard

SELECT user_id,
       (MAX(post_date) - MIN(post_date)) AS hiatus_days
FROM posts
WHERE post_date >= DATE '2025-01-01' AND post_date < DATE '2026-01-01'
GROUP BY user_id
HAVING COUNT(DISTINCT post_date) >= 2
ORDER BY user_id;

The WHERE clause keeps only 2025 dates using a half-open range. GROUP BY user_id collapses to one row per user; MAX - MIN on a DATE yields an integer day count. HAVING COUNT(DISTINCT post_date) >= 2 drops anyone whose 2025 activity landed on a single day.

4. Strategic explanations

Why COUNT(DISTINCT post_date) and not COUNT(*)

The question says "posted on at least two distinct dates". COUNT(*) counts rows, so user 105 (three rows, one day) would pass with a count of 3 — wrong. COUNT(DISTINCT post_date) counts unique calendar days, correctly yielding 1 for 105 and 2 for 103. Whenever a spec says "distinct days/values", the literal translation is COUNT(DISTINCT col); reading COUNT(*) instead is the classic mistake this question is built to catch.

Half-open date range — why < '2026-01-01' beats BETWEEN

Writing post_date >= '2025-01-01' AND post_date < '2026-01-01' is preferable to BETWEEN '2025-01-01' AND '2025-12-31'. For a pure DATE column they agree, but if the column were ever a TIMESTAMP, BETWEEN ... '2025-12-31' would silently drop everything after midnight on Dec 31. The half-open [start, next-period-start) idiom is timezone- and granularity-proof — a habit worth keeping uniform across all range filters.

Date subtraction returns an integer in Postgres

DATE - DATE in Postgres yields an integer count of whole days — exactly the "whole days between" the question asks for. No EXTRACT or DATEDIFF needed. Note this is special to date types: TIMESTAMP - TIMESTAMP yields an interval, not an integer, so if the column were a timestamp you would cast to ::date first or use EXTRACT(DAY FROM ...).

WHERE filters rows, HAVING filters groups

The two-stage filtering is deliberate. WHERE runs first and removes 2024 rows before grouping — this is why user 104 ends up with only one qualifying day. HAVING runs after grouping and can see aggregates like COUNT(DISTINCT ...). You cannot move the distinct-day test into WHERE (the aggregate doesn't exist yet) and you would not want the date filter in HAVING (it would be evaluated too late, after rows are already grouped).

Dialect — date difference elsewhere

Bare DATE - DATE arithmetic is a Postgres convenience. SQL Server needs DATEDIFF(day, MIN(post_date), MAX(post_date)); MySQL needs DATEDIFF(MAX(post_date), MIN(post_date)); BigQuery needs DATE_DIFF(MAX(post_date), MIN(post_date), DAY); Oracle allows the subtraction but returns a NUMBER. The grouping, range filter, and HAVING logic are identical across dialects — only the day-count function changes.

Senior signal: translate "distinct dates" literally into COUNT(DISTINCT post_date) rather than COUNT(*), filter the date range in WHERE (before grouping) and the group-size rule in HAVING (after), prefer the half-open >= start AND < next range so the filter survives a TIMESTAMP column, and know that DATE - DATE returns an integer day count only in Postgres.
