A payments analyst wants each customer's third transaction — a common "nth event per group" pattern used for things like measuring time-to-third-purchase or spotting where engagement stalls. The interviewer is probing whether you reach for ROW_NUMBER partitioned per user, and whether you correctly drop users who never reached a third transaction.

1. The source data — transaction

txn_iduser_idspendoccurred_at(narrative)
1–4110 / 20 / 30 / 40Jan 1–44 txns — 3rd is spend 30
5–6215 / 25Jan 1–2only 2 txns — must NOT appear
7–935 / 55 / 65Jan 1–3exactly 3 txns — 3rd is spend 65

The edge case is user 2: with only two transactions there is no "third," so user 2 must be excluded entirely — not returned with a NULL. User 3 sits exactly on the boundary with precisely three transactions, confirming the filter is = 3, not >= 3.

2. The SQL — rank per user, keep rank 3

WITH ranked AS (
  SELECT user_id, spend, occurred_at,
         ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY occurred_at) AS rn
  FROM transaction
)
SELECT user_id, spend, occurred_at
FROM ranked
WHERE rn = 3
ORDER BY user_id;

ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY occurred_at) numbers each user's transactions 1, 2, 3, ... in chronological order. The outer WHERE rn = 3 keeps only the third row per user. A user with fewer than three transactions simply has no row numbered 3, so they fall away.

3. The resulting analysis

User 2 is absent: their transactions are numbered only 1 and 2, so rn = 3 matches nothing for them — no row, no NULL. User 1's fourth transaction (spend 40) is numbered rn = 4 and filtered out; only the spend-30 row survives.

4. Strategic explanations

Why ROW_NUMBER and not RANK / DENSE_RANK

"The third transaction" means exactly one row — the third in time. ROW_NUMBER assigns a strict 1, 2, 3, ... with no ties: there is always exactly one rn = 3 per qualifying user. RANK would, on a timestamp tie, assign two rows the same number and skip the next, so rn = 3 might match two rows or none. DENSE_RANK would match every row sharing the third-place timestamp. For "pick the single nth row," ROW_NUMBER is the only correct ranking function.

The PARTITION BY makes the count per-user

Without PARTITION BY user_id, ROW_NUMBER would number all transactions 1..9 globally and rn = 3 would return a single row across the whole table. The partition restarts the counter at 1 for each user, so "third" is third within that user. Partitioning is what turns a global ranking into a per-group ranking — the heart of every "nth per group" problem.

Fewer-than-n users are excluded by construction

The requirement "if a user has fewer than 3 transactions, do not include them" needs no explicit count or HAVING. A user with two transactions only ever produces rn values 1 and 2; WHERE rn = 3 finds nothing for them and they vanish. This is cleaner than computing COUNT(*) per user and joining — the rank filter does the work. Contrast with "nth or last": for "third or their last if fewer," you would need COUNT(*) OVER (PARTITION BY user_id) too.

Tie-breaking the ORDER BY

ORDER BY occurred_at alone is non-deterministic if a user has two transactions at the identical timestamp — which of them is "second" vs "third" is arbitrary, and re-running the query could swap them. The data here has distinct timestamps so it is stable, but a production-grade answer adds a tiebreaker: ORDER BY occurred_at, txn_id. Naming the determinism gap unprompted is the senior signal.

Dialect notes

ROW_NUMBER() OVER (...) is standard and supported by PostgreSQL, SQL Server, Oracle, MySQL 8+, and modern SQLite. Older MySQL (< 8.0) lacks window functions — there you would emulate per-user numbering with a correlated subquery counting earlier transactions, or with session variables. The window-function form is vastly clearer and is the expected answer on any current engine.

Senior signal: solve "the nth row per group" with ROW_NUMBER() OVER (PARTITION BY group ORDER BY time) filtered to rn = n — ROW_NUMBER not RANK, because you want exactly one row — rely on the rank filter to drop groups with fewer than n rows for free, and add a tiebreaker to the ORDER BY so the result is deterministic under timestamp ties.
