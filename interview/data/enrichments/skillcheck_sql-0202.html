Set-returning functions like generate_series are a PostgreSQL superpower for building date spines and numbered sequences without application code. The WITH ORDINALITY modifier is a lesser-known SQL:2003 extension that appends an automatically incremented row number to any SRF output — the interviewer is testing whether you know this clean alternative to a ROW_NUMBER() window function on an already-generated series.

1. The source data — generated (no base table)

There is no source table; the query is self-contained. generate_series synthesises five month-start dates. The conceptual "data" before ordinality is attached:

m (from generate_series)
2025-01-01 00:00:00+00
2025-02-01 00:00:00+00
2025-03-01 00:00:00+00
2025-04-01 00:00:00+00
2025-05-01 00:00:00+00

WITH ORDINALITY appends an integer column ord starting at 1, giving exactly the 1-based position requested.

2. The SQL — generate_series WITH ORDINALITY

SELECT
  ord AS month_num,
  m::date AS month_start
FROM generate_series(DATE '2025-01-01', DATE '2025-05-01', INTERVAL '1 month')
     WITH ORDINALITY AS t(m, ord)
ORDER BY ord;

WITH ORDINALITY is placed immediately after the SRF call, before the alias. The AS t(m, ord) names the two output columns: m for the generated timestamp and ord for the ordinal counter. The cast m::date strips the time zone component, yielding clean date values.

Five rows, month_num 1 through 5, month_start the first of each month from January through May 2025. The ordinality counter starts at 1 by definition of the SQL standard; there is no way to choose a different start without wrapping in a CASE or arithmetic.

4. Strategic explanations

WITH ORDINALITY vs ROW_NUMBER()

The equivalent without WITH ORDINALITY:

SELECT ROW_NUMBER() OVER (ORDER BY m) AS month_num, m::date AS month_start
FROM generate_series(DATE '2025-01-01', DATE '2025-05-01', INTERVAL '1 month') AS t(m)
ORDER BY m;

Both produce the same result. WITH ORDINALITY is cleaner because it avoids a window function on a set that is already ordered by construction. It also computes the counter at the SRF level, before any filter or ORDER BY, which makes it semantically unambiguous: the ordinal reflects the position in the generated sequence, not a post-filter rank.

Column naming with AS t(m, ord)

When a SRF is used with WITH ORDINALITY, the output has two columns: the original SRF output(s) and the ordinality counter. Naming them via AS t(m, ord) is required to reference them in the SELECT list. If you omit the names, PostgreSQL falls back to default column names (generate_series and ordinality), which work but are less readable.

Data type of generate_series output

When called with DATE arguments and an INTERVAL, PostgreSQL returns TIMESTAMPTZ by default (the interval may cross DST boundaries). Casting to ::date is necessary to get a plain date. Calling generate_series with TIMESTAMP arguments instead avoids the timezone coercion, but in either case the cast is good practice for readability.

Portability of WITH ORDINALITY

WITH ORDINALITY is a SQL:2003 feature supported by PostgreSQL (since 9.4), DuckDB, and CockroachDB. MySQL, SQL Server, BigQuery, and Snowflake do not support WITH ORDINALITY on SRFs — because most of them lack native SRFs like generate_series entirely. On those platforms you use GENERATE_DATE_ARRAY (BigQuery) + WITH ORDINALITY-equivalent unnest syntax, or a recursive CTE with an explicit counter column. The portability gap is significant; knowing it demonstrates dialect awareness.

Zero-based vs one-based ordinality

The SQL standard mandates that ordinality starts at 1. If you need 0-based positions, subtract 1: ord - 1 AS offset. This is common when the ordinal represents a date offset (week 0, week 1, …) rather than a human-readable position.

Senior signal: place WITH ORDINALITY immediately after the SRF (before the alias) to get a 1-based counter from the function itself rather than a post-hoc window function; name both columns with AS t(m, ord); cast the timestamp to ::date; and note that WITH ORDINALITY is a SQL:2003 feature not available in MySQL, SQL Server, BigQuery, or Snowflake, where a recursive CTE with an explicit counter is the fallback.
