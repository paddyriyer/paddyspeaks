A retention analyst at a subscription business is asked for the classic cohort grid: group customers by the month of their first order, then track what fraction of each cohort comes back one month later, two months later, and so on. The triangular table that results is the single most-requested chart in any growth review. It is a pivot — "months since signup" rotates into columns — layered on cohort assignment. The interviewer is testing whether you can chain CTEs to compute a relative time offset and pivot it correctly.

2. The SQL — cohort, offset, size, then pivot

WITH first_order AS (
  SELECT customer_id, DATE_TRUNC('month', MIN(order_date)) AS cohort_month
  FROM orders
  GROUP BY customer_id
),
activity AS (
  SELECT DISTINCT
    f.cohort_month,
    o.customer_id,
    (EXTRACT(YEAR FROM o.order_date) - EXTRACT(YEAR FROM f.cohort_month)) * 12
      + (EXTRACT(MONTH FROM o.order_date) - EXTRACT(MONTH FROM f.cohort_month)) AS month_since
  FROM orders o
  JOIN first_order f ON f.customer_id = o.customer_id
),
cohort_size AS (
  SELECT cohort_month, COUNT(*) AS n
  FROM first_order
  GROUP BY cohort_month
)
SELECT
  TO_CHAR(a.cohort_month, 'YYYY-MM') AS cohort,
  c.n AS cohort_size,
  ROUND(100.0 * COUNT(DISTINCT CASE WHEN a.month_since = 0 THEN a.customer_id END) / c.n, 1) AS month_0,
  ROUND(100.0 * COUNT(DISTINCT CASE WHEN a.month_since = 1 THEN a.customer_id END) / c.n, 1) AS month_1,
  ROUND(100.0 * COUNT(DISTINCT CASE WHEN a.month_since = 2 THEN a.customer_id END) / c.n, 1) AS month_2
FROM activity a
JOIN cohort_size c ON c.cohort_month = a.cohort_month
GROUP BY a.cohort_month, c.n
ORDER BY cohort;

first_order stamps each customer with their cohort. activity computes, per order, how many whole months after signup it fell. cohort_size counts cohort members for the denominator. The final query pivots month_since into columns and divides by cohort size.

February's month_1 is 0.0 — customer 3 placed no order in March — yet month_2 rebounds to 100%. Retention is not monotonic, and the grid captures that honestly.

4. Strategic explanations

Why three CTEs — each computes a distinct fact

The grid needs three things the raw table does not hold: which cohort each customer belongs to, how far after signup each order fell, and how big each cohort is. first_order, activity, and cohort_size each isolate one. Stacking them keeps every step independently readable and testable — the senior habit of decomposing a hard query into named, verifiable stages rather than one nested monster.

The month-offset arithmetic — why not subtract dates

month_since is (year diff) * 12 + (month diff). You cannot just subtract dates: '2026-03-12' - '2026-01-05' is a day count (66), not 2 months. And you cannot subtract raw EXTRACT(MONTH) values alone — that breaks across a year boundary (December to January would read 1 - 12 = -11, not +1). The *12 term makes the offset correct across any year gap. DATE_TRUNC('month', …) on the cohort side ensures both sides compare at month granularity.

The pivot — and why COUNT(DISTINCT …) is mandatory

Each month_N column is a COUNT(DISTINCT CASE WHEN month_since = N THEN customer_id END): the CASE isolates orders at that offset, COUNT tallies them, and DISTINCT ensures a customer who ordered twice in the same offset month counts once — retention measures people who returned, not order volume. Drop DISTINCT and a heavy repeat buyer inflates the percentage above 100%. The SELECT DISTINCT already in the activity CTE is a second layer of the same guard.

month_0 is 100% by construction — and that is a sanity check

Every customer ordered in their own cohort month — that is the definition of the cohort — so month_0 is always 100%. Far from redundant, it is a built-in correctness probe: if your query ever shows month_0 below 100%, the cohort assignment or the offset math is wrong. The denominator c.n is the cohort's original size, frozen — never the count still active in month N — so each column reads as "percent of the starting cohort", which is what makes columns comparable down a cohort's row.

Dialect — date functions and the shape of the grid

DATE_TRUNC, EXTRACT, and TO_CHAR are PostgreSQL spellings; SQL Server uses DATETRUNC/DATEDIFF/FORMAT, BigQuery DATE_TRUNC/DATE_DIFF — the logic is identical, only the names move. The pivot itself is the universal COUNT(CASE…) idiom. A native PIVOT operator could rotate month_since, but real cohort grids run to 12 or 24 columns — at that width teams generate the column list with dynamic SQL or compute the grid long and pivot in the BI layer, because hand-typing 24 CASE branches does not scale.

Senior signal: decompose the cohort grid into CTEs for first-order month, per-order month offset, and cohort size; compute the offset as year_diff*12 + month_diff so it survives year boundaries; pivot with COUNT(DISTINCT CASE…) so repeat buyers cannot push retention past 100%; and treat the always-100% month_0 column as a correctness check on your cohort logic.
