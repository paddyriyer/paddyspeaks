Cisco's customer-success team wants a clean list: every customer who has placed an order but never requested a refund — the candidates for an upsell. It is an exclusion query, and the reference solution shows real maturity by pre-filtering the refunds list for NULLs. The interviewer asking this is testing whether you understand why that WHERE customer_id IS NOT NULL is not optional cleanup but a correctness fix.

1. The source data — orders and refunds

The solution provides its own fixture:

orders.customer_id
1, 2, 3, 4, 5, 6, 7, 8
refunds.customer_id
2, 4, 7, NULL

Eight customers have ordered; three — 2, 4, 7 — have refunds. The fourth refund row has a NULL customer_id (an orphaned or unattributed refund). That single NULL is the entire interview.

2. The SQL — NOT IN with a NULL guard

SELECT DISTINCT customer_id FROM orders
WHERE customer_id NOT IN (
  SELECT customer_id FROM refunds WHERE customer_id IS NOT NULL
);

The inner WHERE customer_id IS NOT NULL strips the NULL refund row before it can reach NOT IN. The outer query then keeps customers whose id matches none of the (now NULL-free) refunded ids.

3. The resulting analysis

customer_id
3
5
6
1
8

Five customers — 1, 3, 5, 6, 8 — ordered without ever refunding. The result is unordered (DISTINCT and NOT IN promise the set, not a sort). And it is correct — but only because of the NULL guard, as the next section shows.

4. Strategic explanations

Why the NULL guard is mandatory, not cosmetic

Delete WHERE customer_id IS NOT NULL and the query returns zero rows. NOT IN (2, 4, 7, NULL) means customer_id <> 2 AND <> 4 AND <> 7 AND <> NULL. That final <> NULL evaluates to unknown, never true, so the whole AND can never be true — every customer is filtered out. An empty upsell list, no error raised. The reference solution's guard is the fix for this exact failure.

NOT EXISTS — immune by construction

A more robust idiom that needs no guard at all:

SELECT DISTINCT o.customer_id FROM orders o
WHERE NOT EXISTS (
  SELECT 1 FROM refunds r WHERE r.customer_id = o.customer_id
);

NOT EXISTS asks a yes/no question: "is there a refund row for this customer?" The NULL refund row simply fails the r.customer_id = o.customer_id join and is harmlessly skipped — it cannot poison the predicate. For exclusion against a nullable column, NOT EXISTS is the safer default precisely because you don't have to remember the guard.

The LEFT JOIN / IS NULL anti-join

Equivalent, and also NULL-safe here:

SELECT DISTINCT o.customer_id FROM orders o
LEFT JOIN refunds r ON o.customer_id = r.customer_id
WHERE r.customer_id IS NULL;

Keep every order, then discard customers that matched a refund. The NULL refund row never equals any customer id, so it joins to nobody and changes nothing.

Why DISTINCT is here

orders has one row per order, so a customer with five orders appears five times. DISTINCT collapses the output to one row per customer — the question asks for a list of customers, not of orders. With the NOT EXISTS form the DISTINCT is still needed for the same reason.

Edge case: the orphaned refund itself

The NULL-customer_id refund is unattributable — it can't exclude anyone because we don't know whose it is. Dropping it (as the solution does) is the right call for this question, but in a data-quality review it's a flag: a refund with no customer is a broken record worth chasing upstream.

Senior signal: explain that the inner IS NOT NULL is a correctness fix — one NULL in a NOT IN subquery empties the whole result — and know that NOT EXISTS or a LEFT JOIN / IS NULL anti-join avoids the trap entirely, while remembering DISTINCT because orders is per-order not per-customer.
