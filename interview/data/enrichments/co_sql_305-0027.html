Wix's HR system stores names split into first_name and last_name, and the employee directory wants a single display name. It is the canonical string-concatenation question — one line with || — but the interviewer is checking whether you know what happens when one of those name parts is NULL, because incomplete name records exist in every HR table.

1. The source data — employees

emp_idfirst_namelast_name
1AnaPatel
2BenCohen
3CaraNULL
4DanSingh

Employee 3 has a first_name but a NULL last_name — a single-name record, or a data-entry gap. That one NULL decides whether the directory shows Cara or shows nothing at all.

2. The SQL — first_name || ' ' || last_name

SELECT emp_id, first_name || ' ' || last_name AS full_name FROM employees;

The || operator concatenates strings. Three operands here: the first name, a literal space, and the last name.

3. The resulting analysis

emp_idfull_name
1Ana Patel
2Ben Cohen
3NULL
4Dan Singh

Three names render fine — but employee 3 comes back with a completely NULL full_name, not Cara. One missing part nuked the whole string. The directory would show a blank row where Cara should be. That is the bug the question is built to expose.

4. Strategic explanations

Why || blanked the whole name — NULL propagation

In Postgres, || propagates NULL: if any operand is NULL, the entire expression is NULL. 'Cara' || ' ' || NULL is NULL — the good first name is discarded along with the missing last name. This is the single most common mistake on concatenation questions, and the demo data is designed to catch it.

The fix: COALESCE each part, or use CONCAT_WS

Two solid repairs. Defend each operand with COALESCE:

SELECT emp_id,
  COALESCE(first_name, '') || ' ' || COALESCE(last_name, '') AS full_name
FROM employees;

This keeps Cara — but leaves a trailing space ('Cara '), so wrap it in TRIM(…). Cleaner is CONCAT_WS, which skips NULLs and drops the orphaned separator:

SELECT emp_id, CONCAT_WS(' ', first_name, last_name) AS full_name
FROM employees;

For employee 3, CONCAT_WS returns a clean Cara — no NULL, no dangling space. It is the idiomatic answer once NULLs are in play.

Plain CONCAT — better than ||, still imperfect

CONCAT(first_name, ' ', last_name) treats NULL as an empty string, so it never blanks the whole row — an improvement on ||. But it cannot remove the separator next to a missing value: employee 3 would render 'Cara ' with a trailing space. Only CONCAT_WS omits both the value and its delimiter.

The redundant name column — ask before you build

The schema already ships a name column alongside first_name/last_name. Before deriving full_name at all, ask: is name the authoritative display name? If it is populated and trustworthy, the whole concatenation may be unnecessary — or worse, may disagree with name and confuse the directory. Surfacing that duplication is the senior instinct: do not compute what the table already stores.

Edge cases: empty strings and whitespace

An empty-string last_name ('', not NULL) is a real value: || keeps it and you get 'Cara ' with a trailing space; CONCAT_WS keeps it too and emits a doubled space. Dirty data often has '' where you expect NULL. Normalise with NULLIF(TRIM(last_name), '') so blanks become NULLs that CONCAT_WS will then drop, and the directory stays clean.

Senior signal: immediately flag that || propagates NULL and blanks the entire name when one part is missing, fix it with CONCAT_WS (which drops the NULL and its separator) or COALESCE+TRIM, and question whether the existing name column makes the derivation redundant in the first place.
