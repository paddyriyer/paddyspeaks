A Community-sourced Yelp question: surface the five states with the most 5-star businesses for a "best places to start a business" map. It reads like a trivial GROUP BY with a LIMIT 5 — but the schema column stars is a per-business average, the headline word is "top 5", and what happens when the sixth state ties the fifth is exactly the thing an interviewer is probing.

1. The source data — yelp_business

CA has a 4-star business that must not be counted; NV has a 2-star one likewise. The interesting part is the long tail: six states each end up with exactly one 5-star business, all tied for fifth place.

2. The SQL — RANK then filter rnk ≤ 5

WITH t1 AS (
    SELECT state, COUNT(DISTINCT business_id) AS n_businesses
    FROM yelp_business
    WHERE stars = 5
    GROUP BY state
),
t2 AS (
    SELECT state, n_businesses, RANK() OVER (ORDER BY n_businesses DESC) AS rnk FROM t1
)
SELECT state, n_businesses
FROM t2
WHERE rnk <= 5
ORDER BY rnk ASC, state ASC;

The WHERE stars = 5 filter runs before the aggregate, so only 5-star rows are counted. RANK() then orders states by count, and the outer filter keeps everything ranked 5 or better.

Eight rows, not five. CA and TX tie at rank 1; RANK() skips rank 2 and the next group lands at rank 3. Six states sit at count 1, all sharing rank 3 — and 3 ≤ 5, so every one of them survives the filter. "Top 5" by rank can return far more than five rows. That is a feature, not a bug, but you must say so.

4. Strategic explanations

Why RANK ≤ 5 is the defensible reading of "top 5"

If three states are tied for first, picking an arbitrary five and dropping the rest is indefensible — the cut would be decided by physical row order, not data. RANK() <= 5 keeps every state that is genuinely in the top five positions, ties included. The cost is an unbounded row count when the tail is flat, exactly as we see here. State the trade out loud: "ties are kept, so the result can exceed five rows — is that acceptable, or do you want a hard five?"

The LIMIT 5 alternative — and why it is tie-blind
SELECT state, COUNT(DISTINCT business_id) AS n_businesses
FROM yelp_business
WHERE stars = 5
GROUP BY state
ORDER BY n_businesses DESC, state ASC
LIMIT 5;

This returns exactly five rows — CA, TX, then three of the six tied states chosen by the state ASC tiebreak (AZ, FL, NV). It is deterministic only because of that secondary sort key; without it the planner is free to pick any three. LIMIT answers "give me five rows"; RANK answers "give me the top five tiers". Different questions — confirm which the map needs.

RANK vs DENSE_RANK here

With DENSE_RANK() the tiers would be 1, 1, 2 instead of 1, 1, 3 — the six count-1 states would rank 2, still inside the ≤ 5 cut, so the result is identical for this data. They diverge only when enough distinct count values stack up between rank 1 and the cutoff. RANK matches "competition" placement (Olympic medals); DENSE_RANK matches "how many tiers down am I". Know which the stakeholder means.

Why COUNT(DISTINCT business_id) and not COUNT(*)

The schema hints at messy joins (t1, t2, businesses columns). If yelp_business were ever a join result with one row per review, COUNT(*) would count reviews and inflate every state. COUNT(DISTINCT business_id) is defensive: it counts businesses no matter how the row grain drifts. On a clean one-row-per-business table the two agree, but the distinct version costs nothing in correctness and buys robustness.

The NULL-state edge case

If state is NULL for some businesses, GROUP BY state collapses them into a single NULL group that can itself rank in the top five — a meaningless "state". Decide deliberately: WHERE state IS NOT NULL to drop it, or surface it as a data-quality flag. Silence here is the junior tell.

Senior signal: recognise that RANK() <= 5 can return more than five rows when the tail ties, name that trade against a tie-blind LIMIT 5, justify COUNT(DISTINCT business_id) as grain-defensive, and flag the NULL-state group before it sneaks into the leaderboard.
