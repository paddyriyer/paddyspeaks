Robinhood's operations team works a queue of orders and wants it sorted the way the desk actually triages: High-priority first, then Med, then everything else — and within each tier, oldest order first. The catch is that priority is a text column, so a plain sort gives the wrong order. The interviewer wants the CASE-in-ORDER BY trick.

1. The source data — orders

order_idpriorityorder_dateamount
1Med2024-02-10$200
2High2024-02-12$400
3Low2024-02-01$90
4High2024-02-05$350
5Med2024-02-03$150

Two orders are High (2 and 4) and two are Med (1 and 5) — so the within-tier date sort has real work to do. The desired order is not alphabetical: alphabetically High < Low < Med, which is nonsense for triage.

2. The SQL — CASE in ORDER BY

SELECT * FROM orders
ORDER BY CASE priority WHEN 'High' THEN 1
                       WHEN 'Med'  THEN 2
                       ELSE 3 END,
         order_date;

The CASE maps each priority string to a sort weight — 1, 2, 3 — so the database sorts by that number, not the text. order_date is the second sort key: the tiebreaker within a tier.

3. The resulting analysis

order_idpriorityorder_dateamount
4High2024-02-05350
2High2024-02-12400
5Med2024-02-03150
1Med2024-02-10200
3Low2024-02-0190

High orders lead, Med follow, Low last — the business order, not the alphabet. Within High, order 4 (Feb 5) precedes order 2 (Feb 12); within Med, order 5 (Feb 3) precedes order 1 (Feb 10). The two-level sort — tier then date — reads exactly like the triage rule.

4. Strategic explanations

Why a plain ORDER BY priority is wrong

Sorting the text column directly sorts lexically: 'High', 'Low', 'Med' — so a Low-priority order would jump ahead of a Med one. Alphabetical order rarely equals business order. The CASE exists precisely to impose an arbitrary, intentional ordering that the raw values cannot express.

The cleaner long-term fix — a priority lookup or enum

An inline CASE hard-codes the ordering into every query that needs it; add a Critical tier and you must edit them all. Two more durable designs:

-- A. a lookup table carrying an explicit sort rank
SELECT o.* FROM orders o
JOIN priority_levels pl ON o.priority = pl.name
ORDER BY pl.sort_rank, o.order_date;

-- B. a Postgres ENUM, whose declared order IS the sort order
CREATE TYPE priority_t AS ENUM ('High','Med','Low');
-- then ORDER BY priority sorts High, Med, Low natively

An ENUM sorts by declaration order for free; a lookup table keeps the ordering as data, editable without code changes. The inline CASE is the right answer for a one-off query, the wrong one for a pattern repeated across a codebase.

The ELSE catch-all and unexpected values

ELSE 3 sweeps up 'Low' — but also 'Urgent', 'low' (wrong case), a typo, or NULL. They all collapse to weight 3 and land together at the bottom, indistinguishably. If new priority values are likely, list them explicitly and reserve ELSE for genuine unknowns — or sort unknowns out of the way deliberately rather than lumping them with Low.

Where do NULL priorities go?

CASE priority WHEN 'High'… uses equality, and NULL = 'High' is unknown, never true — so a NULL priority falls through to ELSE 3 and sorts with Low. That may or may not be intended; a NULL could mean "untriaged" and belong at the top. If so, give it its own branch: WHEN priority IS NULL THEN 0 (note: that needs the searched CASE form, CASE WHEN …, since the simple CASE priority WHEN form cannot test IS NULL).

SELECT * in production code

The solution uses SELECT *, fine for an interview. In a real service it is fragile: a new column changes the result shape and can break a consumer or an INSERT … SELECT. Name the columns the desk actually needs. Worth stating as a senior aside even though it does not affect this answer.

Senior signal: use a CASE map to impose a business ordering that text sort cannot, add order_date as an explicit within-tier tiebreaker, know that ELSE silently absorbs typos and NULLs, and mention that an ENUM or priority lookup table is the more maintainable design when the ordering is reused.
