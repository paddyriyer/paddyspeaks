LRU Cache is the most-asked design question at Google for a reason: every layer of a real system has a bounded cache — CPU, OS page cache, CDN, application memcache — and they all need the same thing: O(1) get, O(1) put, and an eviction policy that drops whatever was used least recently. The interviewer wants to see you reach both constant-time targets and the recency ordering at once.

1. The input data

A cache with capacity = 2, then a sequence of operations:

c = LRUCache(2)
c.put(1, 'A')
c.put(2, 'B')
c.get(1)         # touches key 1
c.put(3, 'C')    # capacity exceeded -> evict
c.get(2)
c.get(3)

2. The approach

The solution leans entirely on collections.OrderedDict, which maintains keys in insertion order and exposes O(1) reordering. The convention here: the front (left end) is least-recently-used, the back (right end) is most-recently-used. get(key) returns -1 on a miss; on a hit it calls move_to_end(key) to promote the key to the back — "just used" — then returns the value. put(key, value) first promotes the key with move_to_end if it already exists (so an update also counts as a use), assigns the value, and then, if the dict has grown past cap, calls popitem(last=False) to evict the item at the front — the genuine least-recently-used entry. Both move_to_end and popitem are O(1) because OrderedDict is backed by a hash map plus a doubly linked list.

3. The output

operationorder (LRU→MRU)returns
put(1,'A')1—
put(2,'B')1, 2—
get(1)2, 1'A'
put(3,'C')1, 3  (evicted 2)—
get(2)1, 3-1
get(3)1, 3'C'

The pivotal step is put(3,'C'): because get(1) had just promoted key 1, the least-recently-used entry was 2 — so 2 is evicted, not 1. A subsequent get(2) correctly misses.

4. Strategic explanations

Complexity

Both get and put are O(1), and that is the entire bar for this question — an interviewer will reject any solution that is O(n) on either. Space is O(capacity). The reason O(1) is even possible is the hash-map-plus-linked-list combination: the map finds the node, the linked list relinks it without shifting anything.

The idiomatic alternative — functools.lru_cache

For caching the results of a pure function, you would never hand-roll this — @functools.lru_cache(maxsize=2) does exactly this internally. Naming it shows you know the stdlib. The hand-built version is still the right interview answer because it demonstrates you understand the mechanism, and because a real cache needs explicit invalidation that the decorator does not offer.

The edge case — updating an existing key, and capacity 0

put on an existing key must also count as a use — the move_to_end before assignment handles that; forgetting it makes a freshly-updated key wrongly eligible for eviction. A capacity of 0 is a subtle case: every put immediately exceeds it and evicts, so the cache stays empty — technically correct here, but worth confirming the spec allows it.

The follow-up — LFU, or thread safety

"Now evict least-frequently-used instead." LFU is strictly harder: you must track per-key hit counts and find the minimum-count key in O(1), which needs a frequency-bucketed linked-list structure. The other common follow-up is "make it thread-safe" — OrderedDict operations are not atomic across the read-modify-write in put, so you wrap them in a threading.Lock.

The trap — a plain dict with a timestamp

A tempting design stores key → (value, last_used_time) in an ordinary dict and, on eviction, scans for the smallest timestamp. That scan is O(n) — it fails the headline requirement. The recency order must be maintained incrementally as a linked list, not reconstructed by searching.

Senior signal: a senior recognizes OrderedDict (or a hash-map-plus-doubly-linked-list) as the O(1) backbone, treats an update as a use, and names functools.lru_cache and the LFU/thread-safety follow-ups — a junior scans for the oldest timestamp and lands at O(n).
