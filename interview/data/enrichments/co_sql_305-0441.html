Uber driver-status pings fire every minute or so — the system records "online" repeatedly even when the driver hasn't done anything. Useful for telemetry; useless for "how many distinct online sessions did this driver have today?". The query collapses pings into session_id-labelled groups using the cumulative-flag trick.

1. The source data — driver_status_pings

2. The SQL — cumulative-flag SUM for session_id

WITH transitions AS (
  SELECT driver_id, ping_at, status,
    CASE WHEN status != COALESCE(LAG(status) OVER (PARTITION BY driver_id ORDER BY ping_at), '__none__')
         THEN 1 ELSE 0 END AS is_change
  FROM driver_status_pings
),
sessioned AS (
  SELECT driver_id, ping_at, status,
    SUM(is_change) OVER (PARTITION BY driver_id ORDER BY ping_at) AS session_id
  FROM transitions
)
SELECT driver_id, session_id, status,
  MIN(ping_at) AS session_start,
  MAX(ping_at) AS session_end,
  COUNT(*)    AS pings_in_session
FROM sessioned
GROUP BY driver_id, session_id, status
ORDER BY driver_id, session_start;

3. Strategic explanations

The cumulative-flag SUM idiom

The is_change flag is 1 when status differs from the previous ping, 0 otherwise. SUM(is_change) OVER (... ORDER BY ping_at) gives a monotonically-increasing label that bumps once per state change. Within a state run, all rows share the same session_id; across changes, the id increments.

Why COALESCE(LAG(status), '__none__') instead of OR IS NULL

Both forms work; the COALESCE-to-sentinel form is one expression instead of two predicates. Pick whichever your team finds clearer; the optimizer treats them identically.

The "online with no rides" diagnostic

Sessions with status='online' AND zero rides are the marketplace's idle-supply signal. Joining this query's output to fct_rides on overlapping time produces the "driver was online but un-utilized" minutes — input to surge-pricing and Peak Pay tuning.

The minimum-session-length filter

3-second online sessions (driver toggled accidentally) are noise. Production filter:

HAVING COUNT(*) >= 3 AND julianday(MAX(ping_at)) - julianday(MIN(ping_at)) >= 1.0/144   -- >= 10 min
The 30-minute-grace cousin (tolerant streak)

If pings are flaky and a driver might miss one or two, treat gaps < 30 sec as the same session even if there's no ping in that window. Modify the is_change flag to require both status difference AND time gap exceeding tolerance. Same pattern, richer condition.

Senior signal: name the cumulative-flag SUM idiom (LAG-detect-change + SUM-OVER-ORDER), filter minimum session length to remove toggle noise, and connect online-but-no-rides sessions to the marketplace's idle-supply metric. The 30-minute-grace tolerant variant is the over-the-bar generalization.
