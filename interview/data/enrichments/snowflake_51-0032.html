GitHub’s amusement park division (think enterprise analytics on ride-quality data) wants to flag reviews that are statistical outliers: ratings that fall more than two standard deviations from their ride’s mean. This is the z-score anomaly-detection pattern — compute per-group mean and standard deviation in a CTE, join back to the detail rows, filter on the absolute z-score, and guard the division by NULLIF to handle rides with only one rating (where stddev = 0).

2. The SQL — z-score anomaly detection with NULLIF guard

WITH stats AS (
  SELECT ride_id,
         AVG(rating::numeric) AS mu,
         STDDEV(rating::numeric) AS sd
  FROM ratings
  GROUP BY ride_id
)
SELECT r.rating_id, r.ride_id, r.user_id, r.rating, s.mu, s.sd,
       (r.rating::numeric - s.mu) / NULLIF(s.sd, 0) AS z_score
FROM ratings r
JOIN stats s USING (ride_id)
WHERE s.sd IS NOT NULL AND ABS(r.rating::numeric - s.mu) > 2 * s.sd
ORDER BY ABS((r.rating::numeric - s.mu) / NULLIF(s.sd, 0)) DESC;

The empty result here is in fact structural, not just a quirk of these numbers: in a sample of size n the largest possible absolute z-score (using sample stddev) is (n−1)/√n — for ride 1's n = 5 that ceiling is 4/√5 ≈ 1.79, below the 2σ cutoff no matter how extreme one rating is. The reason is self-correcting feedback: a lone outlier also pulls the mean toward itself and inflates the stddev, so it can never travel far in z-units. A genuinely non-empty result needs a ride with many more ratings, where one value can sit far from a mean the rest hold stable.

4. Strategic explanations

Why a CTE for stats — not a subquery or window function

The CTE approach computes one row per ride (aggregated stats) and joins to the detail table. A correlated subquery would recompute AVG and STDDEV for every rating row — O(n) per row = O(n²) total. A window function approach (AVG(...) OVER (PARTITION BY ride_id)) would work but requires a wrapping query to filter on the computed value, which is functionally equivalent to the CTE. The CTE is the clearest decomposition for interview purposes.

NULLIF(s.sd, 0) — two failure modes, one guard

NULLIF(s.sd, 0) converts sd=0 to NULL before dividing, preventing a division-by-zero error. But there is also the case where STDDEV of a single value returns NULL (sample standard deviation is undefined for n=1). The WHERE s.sd IS NOT NULL clause eliminates both: sd=0 rides with no variance pass the NULL check but their deviation ABS(...) is never strictly greater than 2 * 0 = 0 (unless they all equal the mean, which they do by definition). Rides with one rating get sd=NULL and are excluded by the IS NOT NULL filter. This two-layer defense prevents both errors.

STDDEV vs STDDEV_POP

STDDEV (also written STDDEV_SAMP) uses the sample formula — divides by n-1. STDDEV_POP divides by n. For anomaly detection on a sample of reviews, the sample stddev is more conservative (slightly larger), so the ±2σ threshold is slightly harder to breach — fewer false positives. For a complete census of all ratings ever recorded, STDDEV_POP would be appropriate. In practice the difference is negligible for large n but the distinction should be stated.

The 2-sigma threshold is a convention, not a law

Under a normal distribution, ±2σ captures ~95.4% of observations, so ~4.6% of ratings would be "anomalous" by this definition on any normally distributed dataset. Interviewers sometimes follow up: "What if the distribution of ratings is heavily skewed?" In that case a z-score cutoff is misleading; alternatives include IQR-based outlier detection (rating < Q1 - 1.5 × IQR OR rating > Q3 + 1.5 × IQR) or Median Absolute Deviation (MAD). Mentioning this shows statistical depth.

Dialect notes

STDDEV() is standard in Postgres, Snowflake, BigQuery, and MySQL (8.0+). In BigQuery it is written identically. SQL Server uses STDEV() (no second D). Snowflake supports both STDDEV and STDDEV_SAMP. The USING (ride_id) join shorthand is supported in Postgres and Snowflake but not in SQL Server (which requires ON r.ride_id = s.ride_id).

Senior signal: name the CTE-then-join pattern for per-group statistics as the O(n) alternative to correlated subqueries, explain the double-defense (NULLIF for zero-sd, IS NOT NULL for single-row sd=NULL), distinguish sample vs population stddev and when each applies, and note that z-score assumes normality — IQR or MAD are more robust for skewed rating distributions.
