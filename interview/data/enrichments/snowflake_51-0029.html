GitHub's finance team runs a quarterly review: for each department and fiscal year, how much did we budget, how much did we actually spend, and what is the variance — in absolute dollars and as a percentage of budget? The query also needs to surface departments that had no expenses at all (a LEFT JOIN) and must guard against division by zero when the budgeted amount is zero. This is a four-skill test: LEFT JOIN, COALESCE for NULL-safe aggregation, CASE WHEN for division-guard, and ORDER BY ABS(...) to surface the most alarming variances first.

2. The SQL — LEFT JOIN + COALESCE + CASE WHEN variance analysis

SELECT b.dept_id, b.fiscal_year,
       b.budgeted::numeric AS budgeted,
       COALESCE(SUM(e.amount::numeric), 0) AS actual,
       b.budgeted::numeric - COALESCE(SUM(e.amount::numeric), 0) AS variance,
       CASE WHEN b.budgeted::numeric = 0 THEN NULL
            ELSE 100.0 * (COALESCE(SUM(e.amount::numeric), 0) - b.budgeted::numeric) / b.budgeted::numeric
       END AS variance_pct
FROM budgets b
LEFT JOIN expenses e
       ON b.dept_id     = e.dept_id
      AND b.fiscal_year = e.fiscal_year
GROUP BY b.dept_id, b.fiscal_year, b.budgeted
ORDER BY ABS(b.budgeted::numeric - COALESCE(SUM(e.amount::numeric), 0)) DESC;

Dept 40 shows variance_pct = NULL because its budget is zero.

4. Strategic explanations

Why LEFT JOIN — not INNER JOIN

A department may exist in budgets with no rows in expenses (dept 40 here). An INNER JOIN would silently drop it. The LEFT JOIN keeps all budget rows; the expenses columns come back as NULL for unmatched departments. SUM(NULL) returns NULL, so without COALESCE the actual and variance columns would also be NULL for dept 40. COALESCE(SUM(e.amount::numeric), 0) converts that to zero, which is correct.

The JOIN condition spans two columns

The join uses both dept_id and fiscal_year in the ON clause. Without fiscal_year, dept 10’s 2024 expenses would match its 2023 budget rows (if they existed), producing wildly wrong aggregates. This multi-column join key is the natural key for the budget entity and must appear completely in the join.

COALESCE inside ORDER BY

The ORDER BY clause re-evaluates ABS(b.budgeted::numeric - COALESCE(SUM(...), 0)) rather than referencing the alias variance. Standard SQL allows alias references in ORDER BY (Snowflake, Postgres, MySQL all support it), so this could be written as ORDER BY ABS(variance) DESC for clarity. The long form works universally but is harder to read.

Division-by-zero guard with CASE WHEN

Dividing by a budget of zero produces a runtime error or infinity depending on the engine. The CASE WHEN b.budgeted::numeric = 0 THEN NULL guard makes the percentage NULL for zero-budget rows — semantically correct (a percentage of zero has no meaning) and avoids the error. An alternative is NULLIF(b.budgeted, 0) in the denominator: 100.0 * (...) / NULLIF(b.budgeted::numeric, 0). Both are valid; NULLIF is more concise.

The ::numeric cast

The casts (::numeric) suggest the columns may be stored as TEXT or an integer type and the author wants to ensure decimal precision. In Snowflake this is idiomatic; in standard SQL the equivalent is CAST(col AS NUMERIC). If budgeted and amount are already NUMERIC or FLOAT, the casts are redundant but harmless.

Dialect notes

The query uses Postgres/Snowflake ::numeric cast syntax and standard aggregation. In BigQuery use SAFE_DIVIDE(numerator, denominator) instead of the CASE WHEN guard — it returns NULL on divide-by-zero natively. MySQL does not have :: casts; use CAST(col AS DECIMAL(14,2)). The COALESCE pattern for NULL-safe aggregation is portable across all major engines.

Senior signal: immediately reach for LEFT JOIN when the driving table (budgets) must survive with no matches, wrap the aggregate in COALESCE(..., 0) so NULL rows produce a meaningful zero, guard division by zero with NULLIF or CASE WHEN, and sort by ABS(variance) to surface the most operationally significant rows first. The BigQuery SAFE_DIVIDE callout shows dialect awareness that matters at senior level.
