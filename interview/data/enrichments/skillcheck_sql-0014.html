An ops analyst is plotting a daily metric and the chart looks wrong — the line connects 2025-01-01 straight to 2025-01-03, hiding the fact that nothing happened on the 2nd. The table only stores days that have data, so the gaps are invisible. The ask is to manufacture a contiguous calendar spine from the earliest to the latest date and tag each day as present or missing. The interviewer is probing whether you can generate rows that do not exist in any table — the canonical job of a recursive CTE.

1. The source data — dates_with_gaps

d(narrative)
2025-01-01range start
2025-01-03the 2nd is missing — a one-day gap
2025-01-04consecutive with the 3rd
2025-01-07range end — the 5th and 6th are both missing

The edge case is the data itself: only four rows span a seven-day window. A correct answer must emit days that appear in no row at all — 01-02, 01-05, 01-06 — which a plain SELECT over the table can never do.

2. The SQL — recursive date spine with a presence flag

WITH RECURSIVE spine AS (
  SELECT (SELECT MIN(d::date) FROM dates_with_gaps) AS d,
         (SELECT MAX(d::date) FROM dates_with_gaps) AS max_d
  UNION ALL
  SELECT d + 1, max_d FROM spine WHERE d < max_d
)
SELECT
  to_char(s.d, 'YYYY-MM-DD') AS d,
  CASE WHEN g.d IS NOT NULL THEN 1 ELSE 0 END AS present
FROM spine s
LEFT JOIN dates_with_gaps g ON g.d::date = s.d
ORDER BY s.d;

The anchor seeds one row holding the minimum date and the maximum date. The recursive arm adds one day at a time, carrying max_d along, and stops when d < max_d is no longer true. A LEFT JOIN back to the real table then decides each day's flag — a match yields 1, a non-match yields a NULL that CASE turns into 0.

Seven rows out of four input rows. The three zero-flagged days are pure spine output — the recursion invented them, the LEFT JOIN found no partner, and CASE stamped them 0. This is the difference between a calendar that lies and one that tells the truth.

4. Strategic explanations

Why a recursive CTE and not a join

You cannot JOIN your way to rows that exist nowhere. The recursive CTE is a row generator: the anchor produces a starting row, the recursive arm feeds its own output back in, and each pass emits one more day until the stop condition fails. That self-referential loop is the only pure-SQL way to materialise a dense sequence — here, every calendar day between two bounds.

The termination condition is load-bearing

WHERE d < max_d is what makes the recursion finite. Drop it, get the comparison direction wrong, or forget to advance d with + 1, and the CTE never stops — PostgreSQL eventually errors or exhausts memory. Always be able to point at the exact predicate that guarantees the loop ends, and confirm the step monotonically moves toward it.

The LEFT JOIN direction is the whole trick

The spine must be the left table so every generated day survives the join; dates_with_gaps is the optional right side. Flip it to an INNER JOIN and the gap days vanish — you are back to the original sparse list. present is then derived purely from whether the right side came back NULL, which is exactly what CASE WHEN g.d IS NOT NULL tests.

Casts and the empty-table trap

d is stored as TEXT, so d::date is required before date arithmetic — '2025-01-01' + 1 is meaningless on text. One sharp edge: if dates_with_gaps were empty, MIN/MAX return NULL, the anchor produces a NULL-bounded row, and the result is empty rather than an error — usually acceptable, but worth stating. to_char at the end converts back to the YYYY-MM-DD string the spec asks for.

Dialect notes — generate_series is cleaner in Postgres

In PostgreSQL the idiomatic answer skips recursion entirely: generate_series(MIN(d)::date, MAX(d)::date, '1 day') produces the same spine in one set-returning function. SQL Server reaches for a numbers/tally table; MySQL 8 also supports WITH RECURSIVE. The recursive form shown here is the most portable, but if the engine offers generate_series, naming it signals you know the cleaner tool.

Senior signal: recognise "fill the gaps" as a row-generation problem, build the calendar with a recursive CTE (or generate_series), keep the generated spine on the left of a LEFT JOIN so missing days survive, and be able to point at the exact WHERE clause that guarantees the recursion terminates.
