A customer-success dashboard wants each customer's three most recent orders — the classic "top-N per group" pattern. The catch is the graceful fallback: a customer with fewer than three orders should return all of them, not be padded or dropped. The interviewer is checking whether you use a partitioned ROW_NUMBER with a rn <= 3 filter, and whether you order the final output stably.

Three deliberate cases: Aanya has more than 3 orders (the oldest, 101, must be cut), Bilal has fewer than 3 (both survive), and Chitra has exactly 1 (returned as-is). A correct top-N solution must not require padding rows or assume every group has three.

2. The SQL — partitioned ROW_NUMBER, keep rn <= 3

WITH ranked AS (
  SELECT o.order_id, o.customer_id, o.order_date, o.total,
         ROW_NUMBER() OVER (PARTITION BY o.customer_id ORDER BY o.order_date DESC, o.order_id DESC) AS rn
  FROM orders o
)
SELECT c.name, r.order_id, r.order_date, r.total
FROM ranked r
JOIN customers c ON c.customer_id = r.customer_id
WHERE r.rn <= 3
ORDER BY c.name, r.order_date DESC, r.order_id;

ROW_NUMBER numbers each customer's orders newest-first (order_date DESC, then order_id DESC as a tie-break). Filtering rn <= 3 keeps at most three per customer; a customer with one or two simply has no rn above their count, so all their rows pass. The join attaches the customer name.

4. Strategic explanations

Why rn <= 3 gives the "fewer than N returns all" behaviour for free

This is the elegant part. A customer with two orders gets rn values 1 and 2; both satisfy <= 3, so both pass. There is no rn = 3 row to miss. The filter is a ceiling, not an exact match — so it naturally degrades to "all rows" for small groups. Trying to special-case short customers with UNIONs or CASE logic is a classic over-engineering mistake.

ROW_NUMBER vs RANK vs DENSE_RANK for top-N

If two orders shared a date, RANK could assign 1, 1, 3 — and rn <= 3 would then return potentially four rows for a customer (two tied at 1, one at 3, and the gap). DENSE_RANK avoids the gap but still returns all tied rows, so a three-way tie at the top yields four rows. ROW_NUMBER guarantees a strict count — at most three rows, always. For "exactly N per group" you want ROW_NUMBER; for "all rows tied into the top N positions" you want RANK/DENSE_RANK. State which the product wants.

The tie-break on order_id — determinism again

ORDER BY order_date DESC, order_id DESC inside the window ensures two orders on the same date get a stable, repeatable ranking. Without the order_id key, a date tie at the rn=3 boundary would make it arbitrary which of two same-day orders is kept — a flaky result. Ending the window ORDER BY with a unique column is the habit that removes that nondeterminism.

Why filter in the outer query, not the OVER clause

ROW_NUMBER cannot be referenced in a WHERE in the same SELECT that computes it — window functions are evaluated after WHERE. Hence the CTE: compute rn in ranked, then filter rn <= 3 in the outer query. A subquery works identically. This two-step shape is mandatory for any "filter on a window result" task.

Dialect notes — and the LATERAL alternative

The ROW_NUMBER + CTE pattern is fully portable. Postgres also offers a LATERAL join — FROM customers c, LATERAL (SELECT ... FROM orders WHERE customer_id = c.customer_id ORDER BY order_date DESC LIMIT 3) o — which can be faster when there is an index on (customer_id, order_date) because it fetches only three rows per customer instead of ranking all of them. SQL Server's equivalent is CROSS APPLY with TOP 3. Mention the LATERAL/APPLY option as a scale optimisation.

Senior signal: implement top-N-per-group with a partitioned ROW_NUMBER filtered rn <= N in an outer query, recognise that the inequality filter gives "fewer than N returns all" with no special case, choose ROW_NUMBER over RANK for a strict count, and offer LATERAL/CROSS APPLY as the index-friendly optimisation at scale.
