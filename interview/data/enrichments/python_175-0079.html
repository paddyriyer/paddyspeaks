Google asks "Reverse Linked List" of nearly every candidate — it is the litmus test for pointer manipulation. The code is four lines, so the interview is entirely about whether you can mutate three references in the correct order without losing the rest of the list, and whether you can explain the in-place O(1)-space invariant.

1. The input data

class ListNode:
    def __init__(self, val, next=None):
        self.val, self.next = val, next

# list:  1 -> 2 -> 3 -> 4 -> None
head = ListNode(1, ListNode(2, ListNode(3, ListNode(4))))

A singly linked list of four nodes. The goal is 4 -> 3 -> 2 -> 1 -> None, reusing the same node objects.

2. The approach

The algorithm walks the list once, flipping each node's next to point backward. prev trails behind, starting at None — the new tail's next.

The single line head.next, prev, head = prev, head, head.next does three updates atomically. Python evaluates the entire right side first — prev, head, head.next are all read before any assignment — then binds left to right. So it (1) rewires the current node's next to prev, (2) advances prev to the current node, (3) advances head to the originally-next node. Capturing head.next on the right before overwriting it is what prevents losing the rest of the list. When head becomes None, prev holds the new head.

4. Strategic explanations

Complexity

O(n) time — exactly one visit per node. O(1) space — only the two pointers prev and head, no matter how long the list. This is the optimal answer and the reason interviewers prefer the iterative form.

The idiomatic alternative — the explicit three-pointer form
def reverseList(head):
    prev = None
    while head:
        nxt = head.next   # save before overwriting
        head.next = prev
        prev = head
        head = nxt
    return prev

Functionally identical, just spelling out the temp variable that the tuple-swap hides. Many interviewers prefer this version — the tuple trick is clever but obscures the "save the next pointer first" lesson. Knowing both, and articulating why the temp matters, is the senior move.

The edge case — empty list and single node

An empty list (head is None) skips the loop entirely and returns None — correct. A single node enters once: its next becomes None (already was), prev becomes that node, head becomes None — returns the same node. No special-casing needed; the loop condition handles both.

The follow-up — reverse a sublist, or do it recursively

"Reverse only nodes between positions m and n" (Reverse Linked List II) reuses this core inside a bounded segment with careful boundary stitching. The recursive form — recurse to the tail, then unwind setting head.next.next = head — is elegant but O(n) stack space, which the interviewer will make you state as the tradeoff.

The trap — overwriting next before saving it

Writing head.next = prev first and only then trying to advance head = head.next is the classic bug: head.next now points to prev, so you walk backward into the already-reversed part instead of forward — an infinite loop or truncated list. The whole discipline of the problem is: capture the forward pointer before you clobber it. The tuple swap enforces this automatically; the manual form requires the nxt temp.

Senior signal: explain that Python evaluates the tuple's right-hand side fully before assigning — that is why the one-liner is safe — state O(n)/O(1) crisply, and offer the explicit three-pointer version because it teaches the "save next first" rule the slick one-liner conceals.
