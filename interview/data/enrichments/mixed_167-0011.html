The salary-question's cousin: "find the 3rd-highest order revenue". Same three approaches — correlated subquery, DENSE_RANK, LIMIT/OFFSET — and the same edge cases the interviewer will probe.

1. The source data — orders

order_idcustomer_idrevenue
O01C1$1,200
O02C1$1,200
O03C2$980
O04C3$650
O05C4$650
O06C5$400

Three distinct revenue tiers: $1,200 (tied), $980, $650 (tied), $400. The "3rd highest distinct revenue" is $650. The "3rd row when sorted desc" is $980 (because O01 and O02 share rank 1 if you count rows).

2. The SQL — the three canonical answers

-- A. LIMIT / OFFSET with DISTINCT (the one in the solution)
SELECT DISTINCT revenue FROM orders ORDER BY revenue DESC LIMIT 1 OFFSET 2;

-- B. DENSE_RANK — explicit about distinct semantics
SELECT revenue FROM (
  SELECT DISTINCT revenue, DENSE_RANK() OVER (ORDER BY revenue DESC) AS rk
  FROM orders
) t WHERE rk = 3;

-- C. Correlated subquery — fine for small N, slow for large N
SELECT MAX(revenue) FROM orders WHERE revenue NOT IN (
  SELECT revenue FROM orders ORDER BY revenue DESC LIMIT 2
);

4. Strategic explanations — the interviewer's edge-case probes

Probe 1 — "What if there are fewer than N distinct revenues?"

Most prompts ask "find the 3rd-highest" assuming the data has at least 3 distinct values. If it doesn't, approaches A and B return zero rows; approach C returns a single NULL row. The consumer of the report cares about the difference. Wrap defensively when the downstream BI tool expects exactly one row:

SELECT (SELECT revenue FROM (
  SELECT DISTINCT revenue, DENSE_RANK() OVER (ORDER BY revenue DESC) AS rk
  FROM orders
) t WHERE rk = 3) AS third_highest_revenue;
-- Returns NULL if fewer than 3 distinct revenues; otherwise the value.
Probe 2 — "What about ties? Are O04 and O05 both at rank 3, or is O04 at 3 and O05 at 4?"

Depends on the ranking function:

FunctionRanks on (1200, 1200, 980, 650, 650, 400)Result for rk=3
RANK1, 1, 3, 4, 4, 6One row ($980)
DENSE_RANK1, 1, 2, 3, 3, 4Two rows (both $650)
ROW_NUMBER1, 2, 3, 4, 5, 6One row (whichever the engine picks)

For "Nth-highest distinct value", DENSE_RANK is the right pick. For "Nth row in sorted order", ROW_NUMBER. For "Nth place ignoring ties at lower positions", RANK.

Probe 3 — "Now generalise: top 3 highest revenues, not just the 3rd."

The LIMIT/OFFSET shortcut doesn't extend cleanly. The window-function answer does:

SELECT revenue FROM (
  SELECT DISTINCT revenue, DENSE_RANK() OVER (ORDER BY revenue DESC) AS rk
  FROM orders
) t WHERE rk <= 3;
-- Returns $1200, $980, $650 — three distinct tiers, regardless of ties.
Probe 4 — "What if N is a parameter, not a literal?"

In production reports, N comes from a config or a URL parameter. LIMIT / OFFSET handles parameterisation natively. DENSE_RANK does too via the WHERE rk = :n binding. The correlated-subquery approach gets ugly fast and should be avoided when N is dynamic.

The "Nth-highest per group" extension

The classic follow-up: "Find the 3rd-highest revenue per customer." Add PARTITION BY customer_id to the window:

SELECT customer_id, revenue FROM (
  SELECT DISTINCT customer_id, revenue,
    DENSE_RANK() OVER (PARTITION BY customer_id ORDER BY revenue DESC) AS rk
  FROM orders
) t WHERE rk = 3;

Same edge-case probes apply: customers with fewer than 3 distinct revenues silently disappear from the report unless you LEFT JOIN against the customer list.

Senior signal: show all three approaches, name DENSE_RANK as the safe default for "distinct Nth-highest", surface the "fewer than N distinct values" edge case before being asked, and pre-empt the partition-by-group generalisation. The interviewer wants to see you naming edge cases, not just writing the obvious SQL.
