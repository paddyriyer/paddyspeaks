The smallest possible building block of every production observability system. LogAggregator takes a firehose of (timestamp, key, value) tuples and folds them into fixed-width time buckets, exposing add, query, and top — the three primitives that Datadog, Honeycomb, New Relic, Splunk, Prometheus, and AWS CloudWatch all expose under different names. The class fits in 15 lines of Python; the production reality is that this exact data structure (a hash map keyed by (time_bucket, dimension)) is what backs every point-and-click "metrics over time" dashboard in your engineering org. Get the bucketing, the eviction, and the top-K right and you have a tiny version of a system that ingests trillions of events a day at Datadog.

1. The problem — concrete numbers

Picture a fleet of 10 application servers, each emitting log lines at 1,000/min — 10,000 lines/min total. The lines look like (timestamp, level, service, host, message), where level is one of INFO / WARN / ERROR. Most of the traffic is INFO (95%), but one of the 10 hosts (web-07) has a degraded disk and is throwing 95% of the ERROR lines. The on-call wants to answer three questions in <100ms:

  • "How many ERROR lines did service X have in the last 5 minutes?"
  • "Which host is producing the most ERRORs right now?"
  • "Is this minute-bucket abnormal compared to the prior 10?"

Naive approach: store every log line, grep | wc -l. For 10,000 lines/min retained 24h, that's 14.4M rows scanned per query — multi-second latency, no good for dashboards. The aggregator collapses every line into a counter keyed by (minute_bucket, (level, host)): 3 levels × 10 hosts × 1,440 minutes = 43,200 entries per day. The 10M-line firehose folds into a 43K-entry hash map. Queries become O(1) point look-ups or O(N_hosts) scans.

Storage approachPer-minute rowsQuery latency for "errors/min on host X"
Raw log lines (no aggregation)10,000O(10,000) — 50–500ms scan
Aggregator keyed by (minute, (level, host))30O(1) — <1µs hash look-up
Aggregator + top-K (current bucket)30 + sorted sliceO(N_keys_in_bucket · log K)

The pattern fits any "high-volume → low-cardinality" stream: HTTP status codes per endpoint, query types per database, error types per microservice. The dimensions you keep are the questions you'll be able to answer. Choose them carefully — every added dimension multiplies the key space.

2. The source code — with a runnable demo

from collections import defaultdict, Counter
import random
import time


class LogAggregator:
    """Folds a firehose of (ts, key, value) events into fixed-width time
    buckets. The primitives — add / query / top — back every "metrics over
    time" UI: Datadog gauges, Prometheus counters, CloudWatch dimensions."""

    def __init__(self, bucket_seconds=60):
        self.buckets = defaultdict(int)   # (bucket_id, key) -> sum
        self.bs = bucket_seconds

    def add(self, ts, key, val=1):
        self.buckets[(ts // self.bs, key)] += val

    def query(self, bucket_id, key):
        return self.buckets.get((bucket_id, key), 0)

    def top(self, bucket_id, k=10):
        items = [(key, v) for (b, key), v in self.buckets.items() if b == bucket_id]
        return sorted(items, key=lambda x: -x[1])[:k]


if __name__ == "__main__":
    random.seed(11)

    # Simulate 10 minutes of logs from a 10-host fleet, 1,000 lines/min each.
    # web-07 has a degraded disk and is the source of 95% of ERRORs.
    start_ts = 1_700_000_000        # arbitrary epoch second
    BUCKET = 60                     # 1-minute buckets
    agg = LogAggregator(bucket_seconds=BUCKET)

    hosts = [f"web-{i:02d}" for i in range(10)]
    sick_host = "web-07"

    for minute in range(10):
        for _ in range(10_000):                       # 10k lines/min cluster-wide
            ts = start_ts + minute * 60 + random.randint(0, 59)
            if random.random() < 0.95:                # 95% of lines are the sick host's
                host = sick_host if random.random() < 0.95 else random.choice(hosts)
                level = "ERROR" if random.random() < 0.20 else "INFO"
            else:
                host = random.choice(hosts)
                level = "INFO"
            agg.add(ts, key=(level, host))

    minute_5 = (start_ts + 5 * 60) // BUCKET

    # --- NAIVE: "errors on web-07 last minute?" without aggregation would
    # require scanning ~10k raw lines.  With the aggregator: O(1).
    print("ERROR on web-07 in minute 5:",
          agg.query(minute_5, ("ERROR", "web-07")))
    print("ERROR on web-02 in minute 5:",
          agg.query(minute_5, ("ERROR", "web-02")))

    # --- Surfacing the offender: who is the noisiest in this minute?
    print("\nTop 5 (level, host) pairs in minute 5:")
    for key, cnt in agg.top(minute_5, k=5):
        print(f"  {key}  ->  {cnt}")

    # --- Compare across minutes: is the sick host *getting worse*?
    print("\nERROR on web-07, per minute (rolling 10-minute window):")
    base = start_ts // BUCKET
    series = [agg.query(base + m, ("ERROR", "web-07")) for m in range(10)]
    print("  ", series)

    # --- Aggregate over the whole window
    total_errors = sum(v for (b, (lvl, h)), v in agg.buckets.items() if lvl == "ERROR")
    sick_errors = sum(v for (b, (lvl, h)), v in agg.buckets.items()
                      if lvl == "ERROR" and h == sick_host)
    print(f"\nTotal ERRORs (10 min): {total_errors}")
    print(f"  of which on {sick_host}: {sick_errors} "
          f"({100 * sick_errors / total_errors:.1f}%)")

3. Why this is the production pattern

Datadog / DogStatsD — counters with tags

The DogStatsD wire format is literally metric_name:value|c|#tag1,tag2; the server-side aggregator collapses ten thousand of those per second into a single bucket per (metric, tags, time_bucket) tuple. The Datadog agent does this locally on every host before shipping — pre-aggregation at the edge is what lets one Datadog cluster handle trillions of points/day. The class above is a faithful in-memory model of that exact behaviour.

Prometheus — counters, gauges, histograms

A Prometheus counter http_requests_total{method="GET", status="200"} is one entry in this hash map, keyed by (scrape_time_bucket, (method, status)). The scrape interval defines the bucket size. Histograms add bucket boundaries as another dimension but the data structure is the same. Prometheus query latency for rate(http_requests_total[5m]) is O(buckets_in_range) — fast because pre-aggregation already happened at scrape time.

AWS CloudWatch — embedded metric format

CloudWatch's Embedded Metric Format (EMF) lets you log JSON with a _aws.CloudWatchMetrics envelope; the ingestion pipeline does exactly this aggregation — keyed by (namespace, dimensions, minute_bucket). Lambda functions emit EMF instead of calling PutMetricData N times because the server-side fold is ~100× cheaper per metric.

Honeycomb / event-driven observability

Honeycomb keeps the raw events (high-cardinality is their differentiator) but runs the same aggregator on top for the "BubbleUp" feature — find the dimension value that's most over-represented in a slow trace. The top() method above is the kernel of BubbleUp: at query time, materialise (dimension, count) for the filtered set, sort, surface the outlier.

Splunk / Elastic — bucketed time-series aggregations

Splunk's stats count by host, level piped through bin _time span=1m, or Elasticsearch's date_histogram + terms aggregation, are this class wrapped in DSL. The on-disk format (TSDB-style columnar with run-length-encoded buckets) is more sophisticated, but the logical operation is identical.

Kafka Streams / Flink — windowed counts

Kafka Streams' groupBy(level, host).windowedBy(TimeWindows.of(Duration.ofMinutes(1))).count() compiles down to this exact pattern: a state store keyed by (window_start, (level, host)) mapping to a counter. The state store is RocksDB instead of a Python dict, and there's an eviction policy, but the algorithm is the LogAggregator class.

4. The trade-offs

Memory grows with cardinality, not volume

The aggregator's footprint is determined by distinct key tuples, not by event count. Adding 1M more events for the existing (level, host) pairs costs nothing; adding one new dimension with 10K distinct values multiplies the key space. Acceptable when: dimensions are low-cardinality (status codes, HTTP methods, service names — 10s to 100s of values). Not acceptable when: you tag with user_id or request_id (millions of values) — the map explodes. Production fix: either drop the high-cardinality dimension, or sample it, or use HLL for "count distinct" without keeping the values.

No eviction — the dict grows forever

The toy class never deletes old buckets. In production you need a ring buffer of buckets (keep last N minutes only) or a time-based eviction pass. Prometheus does this with its TSDB block rotation; Kafka Streams uses RocksDB TTL. Acceptable when: the process is short-lived (a query, a serverless function). Not acceptable when: the process runs for days — it will OOM.

top() is O(N · log K) per call

The naive top() scans every key in the bucket and sorts. For a 30-key bucket that's fine; for a 10M-key bucket (high-cardinality dimension) it's a multi-second scan. Production fix: maintain a per-bucket top-K heap incrementally during add() — or use a streaming algorithm (Misra-Gries, Space-Saving) for approximate top-K with bounded memory. Trade exactness for speed.

Single-process — no horizontal scaling

One Python process can ingest maybe 100K events/sec on a beefy box. Beyond that you need to shard the aggregator by key (the LogAggregator instances become Kafka Streams tasks or Flink subtasks), then roll up. And then you hit the skew problem — one hot dimension value (a noisy host) pins one subtask. The fix is the salt pattern from python_175-0002. Same class, fed from that partitioner.

5. Interviewer probes

Probe 1 — "How do you bound memory? The dict will grow forever."

Three production patterns:

  • Ring buffer of buckets: keep the last N buckets in a deque; drop the oldest when a new bucket is created. Simple, exact, bounded.
  • Per-bucket TTL: store buckets in a dict keyed by bucket_id; lazily evict any bucket older than now - retention on every add() or in a background thread.
  • Sketch-based: replace the hash map with a Count-Min Sketch for query() (approximate) and a Misra-Gries summary for top(). Constant memory regardless of cardinality, with a bounded error.
Probe 2 — "What if a noisy host is generating millions of unique error messages?"

That's a high-cardinality dimension hiding inside what looked like a low-cardinality one. Three fixes: (a) hash the message to a fixed-length fingerprint and bucket by fingerprint (lossless count, lossy message-recovery); (b) use a Count-Min Sketch with a per-bucket Misra-Gries top-K for the heavy-hitter messages only (lossy count, lossless heavy-hitter); (c) drop the message from the key entirely and surface it via separate sampled log search (Honeycomb / Splunk pattern).

Probe 3 — "How do you make top() fast for huge buckets?"

Maintain a min-heap of size K alongside the dict. On every add() that touches a key, check if its new count exceeds the heap's min — if so, swap. Top-K becomes O(log K) per add and O(K log K) per query. Streaming alternative: Misra-Gries (bounded memory, ε-approximate top-K) for the case where K ≪ distinct keys.

Probe 4 — "What about late-arriving events?"

If a log line arrives 5 minutes after its event time, the naive aggregator just routes it to its historical bucket — fine as long as you haven't evicted that bucket yet. With a ring buffer of, say, 15 minutes, you tolerate up to 15-minute lateness; beyond that you drop on the floor or write to a "late events" stream for offline correction. This is exactly Kafka Streams' grace() parameter and Flink's allowedLateness. Pick the window based on your event-time clock skew P99.

Probe 5 — "How do you scale this past one process?"

Shard the aggregator by key (hash(key) % N) — N aggregator instances, each owning a key range. Roll up at query time with a fan-out. Now you have a hot-shard problem: one hot key pins one shard. Apply the salt pattern from python_175-0002: salt the hot key on ingest, two-stage roll-up at query. Same algorithm as customer_growth-0027 — counters are associative, so salting is free.

Probe 6 — "Why bucket by time at all — couldn't you just keep a running counter?"

A running counter answers "total since the process started"; a bucketed counter answers "rate over the last 5 minutes". Rates are what alerts and SLOs care about. The bucket size is the temporal resolution of every downstream question: 1-second buckets for security/abuse (high cost, fine-grain), 1-minute for ops dashboards (Datadog/Prometheus default), 5-minute for billing.
