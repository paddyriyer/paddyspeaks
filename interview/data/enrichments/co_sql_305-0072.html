EY's media-analytics practice builds a popularity leaderboard for a streaming client: total plays per title, ranked. The play log has one row per play event, so a title appears many times — meaning you must aggregate first and rank the aggregate. An interviewer is watching two things: that you put SUM inside the RANK's ORDER BY correctly, and that you know what RANK does when two titles tie on total plays.

1. The source data — plays

The rows are play batches, not titles — content 1 and content 3 each appear twice. The deliberate trap: content 1 totals 800 (500+300) and content 3 also totals 800 (400+400). They tie. That tie is the whole interview — it decides whether the title below them is ranked 3 or 4.

2. The SQL — RANK over a grouped SUM

SELECT content_id, SUM(play_count) AS plays,
  RANK() OVER (ORDER BY SUM(play_count) DESC) AS popularity_rank
FROM plays GROUP BY content_id;

Group by title, sum the play counts, and rank by that sum descending. The window function is allowed to reference the aggregate SUM(play_count) directly — window functions are evaluated after grouping.

4. Strategic explanations

The window function runs after the GROUP BY

It looks circular — how can RANK() OVER (ORDER BY SUM(play_count)) reference an aggregate? Logical evaluation order is the answer: FROM → GROUP BY → aggregates → then window functions → SELECT → ORDER BY. By the time RANK is computed, each title is already one row with a known SUM. You may not, however, reference the alias plays inside the OVER clause — repeat the SUM(play_count) expression, as the solution does.

RANK leaves a gap after a tie (1, 2, 2, 4). DENSE_RANK does not (1, 2, 2, 3). ROW_NUMBER refuses to tie at all — it hands content 1 and content 3 different numbers (2 and 3) chosen arbitrarily by physical row order, which for a leaderboard is simply wrong. Pick the function to match the product: a sports-style "joint 2nd, no 3rd place" is RANK; a "2nd tier" grouping is DENSE_RANK.

The non-deterministic tie order

Content 1 and content 3 both show rank 2, but which one prints first is undefined — the query has no ORDER BY on the final result, only inside the window. For a stable, reproducible leaderboard, add an outer sort with a tiebreaker:

SELECT content_id, SUM(play_count) AS plays,
  RANK() OVER (ORDER BY SUM(play_count) DESC) AS popularity_rank
FROM plays
GROUP BY content_id
ORDER BY popularity_rank, content_id;

Without the content_id tiebreaker, two runs of the same query can list the tied titles in different orders.

Top-N — the natural follow-up

"Give me the top 3 most-played titles" is not LIMIT 3 if ties matter — that would arbitrarily cut one of the tied 800-play titles. Wrap and filter on the rank:

SELECT * FROM (
  SELECT content_id, SUM(play_count) AS plays,
    RANK() OVER (ORDER BY SUM(play_count) DESC) AS popularity_rank
  FROM plays GROUP BY content_id
) t
WHERE popularity_rank <= 3;

This returns content 2, 3, and 1 — and would return four rows if three titles tied for the cutoff, which is usually the desired "include all ties" semantics.

NULL play counts

SUM skips NULL inputs. A title whose every play row is NULL would sum to NULL, and ORDER BY SUM(...) DESC sorts NULL first or last depending on NULLS handling — potentially ranking a data-quality hole as the #1 title. If NULL play counts are possible, use SUM(COALESCE(play_count, 0)) or add NULLS LAST.

Senior signal: aggregate before ranking and explain that window functions run after GROUP BY, name the 1-2-2-4 gap as deliberate RANK semantics, contrast RANK/DENSE_RANK/ROW_NUMBER on the tie, and add a deterministic outer ORDER BY so the tied titles do not shuffle between runs.
