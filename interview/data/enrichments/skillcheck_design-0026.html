A SOC2-style audit log must answer two questions that a compliance auditor will ask: "who changed what, and when?" and "can you prove this log has not been tampered with?" The interviewer is probing whether you understand polymorphic entity references (one log table for all entity types), schema-flexible field diffs (JSONB), immutability enforcement (privilege revocation, append-only writes, hash chains), and why the standard approaches — REVOKE DELETE, WORM storage, and cryptographic hash chaining — are complementary rather than alternatives.

1. The requirements — what the design must capture

  • Who: actor type (user, service, system), actor ID, IP address, and user-agent for attribution.
  • What: entity type and ID (polymorphic: any entity in the system), action, and a field-diff showing exactly which fields changed from what to what.
  • When: occurred_at is the authoritative compliance timestamp; it may differ from ingested_at for event-sourced systems.
  • Tamper evidence: a hash chain (prev_log_hash → this_log_hash) allows any auditor to detect if a historical row was modified or deleted by verifying the chain is unbroken.
  • Multi-tenant scope: tenant_id allows per-tenant audit export and retention policy enforcement.
  • Immutability: the database role used by the application must not have UPDATE or DELETE privileges on this table.

2. The DDL — the CREATE TABLE

CREATE TABLE audit_log (
  log_id           BIGSERIAL    PRIMARY KEY,
  occurred_at      TIMESTAMPTZ  NOT NULL DEFAULT now(),
  actor_type       VARCHAR(16)  NOT NULL,                      -- 'user','service','system'
  actor_id         VARCHAR(128) NOT NULL,                      -- user_id, service principal, etc.
  actor_ip         INET,
  actor_user_agent TEXT,
  entity_type      VARCHAR(64)  NOT NULL,                      -- 'customer','order','config'
  entity_id        VARCHAR(128) NOT NULL,                      -- entity PK as string to keep it polymorphic
  action           VARCHAR(16)  NOT NULL,                      -- 'create','update','delete','read'
  field_diffs      JSONB,                                       -- e.g., {"email":{"old":"a@x","new":"b@x"}}
  request_id       VARCHAR(64),                                 -- trace correlation
  tenant_id        BIGINT,                                      -- multi-tenant scope
  prev_log_hash    CHAR(64),                                    -- SHA-256 of preceding row — hash chain
  this_log_hash    CHAR(64)     NOT NULL                        -- SHA-256(prev_log_hash || serialized(this row))
);

CREATE INDEX ix_audit_log_entity ON audit_log (entity_type, entity_id, occurred_at DESC);
CREATE INDEX ix_audit_log_actor ON audit_log (actor_type, actor_id, occurred_at DESC);
CREATE INDEX ix_audit_log_tenant_time ON audit_log (tenant_id, occurred_at DESC);
CREATE INDEX ix_audit_log_action_time ON audit_log (action, occurred_at DESC);

-- 3. prev_log_hash + this_log_hash form a hash chain: tampering with any historical row breaks

-- 4. Partition by month to manage retention without DELETEing live partitions:

4. Strategic explanations

Polymorphic entity references: one table for all

The alternative is one audit_log_orders, audit_log_customers, … table per entity type. That approach is easier to query for a specific entity but creates schema sprawl, inconsistent columns across tables, and a new table for every new auditable entity. The polymorphic pattern (entity_type + entity_id as VARCHAR) consolidates everything into one place. The cost: you cannot add a typed FK constraint from entity_id to the entity's PK, which is acceptable here because audit logs must survive entity deletion (the audit trail must not cascade-delete when an order is deleted).

Immutability: three complementary layers

Layer 1 — privilege revocation: REVOKE UPDATE, DELETE ON audit_log FROM app_role prevents the application from mutating rows. This stops accidental or malicious modification via the application's own database connection. It does not stop a database superuser. Layer 2 — WORM storage: streaming rows to S3 with Object Lock (WORM) or an immutable append-only data store (e.g., Amazon QLDB, Immudb) provides a second independent copy that survives even a database compromise. Layer 3 — hash chain: the SHA-256 chain means that even if an attacker gains superuser access and modifies a row, the chain breaks at that row, making the tampering detectable by the verifier. These three layers provide defense in depth.

Hash chain verification

A nightly verification job reads all rows in log_id order, recomputes SHA-256(prev_log_hash || serialize(row)) for each row, and compares it to this_log_hash. Any mismatch indicates tampering (or a software bug in the insert path). The verification job should run under a separate read-only role and write its result to an external compliance system. This pattern is similar to blockchain but lighter: a single-chain hash ledger, not a distributed consensus system.

Partitioning for retention without DELETE

SOC2 typically requires 12-month retention; HIPAA and GDPR may require 6–10 years. Partitioning by month on occurred_at allows expired partitions to be detached and archived rather than row-deleted: ALTER TABLE audit_log DETACH PARTITION audit_log_2024_01 removes January 2024's data instantly and without DELETE operations. The detached partition can be exported to WORM storage before being dropped. This is both faster and immutability-safe.

Four indexes for four query patterns

Each index serves a distinct compliance use case: entity index for "show me all changes to customer 12345"; actor index for "show me everything user alice@co.com did this week" (SOC2 access review); tenant+time index for per-tenant audit export; action+time index for "how many delete events occurred today?" (anomaly detection). All trailing columns are occurred_at DESC so the most recent events appear first without a sort step.

Senior signal: combine three immutability mechanisms (privilege revocation, WORM sidecar, hash chain) because each layer protects against a different threat model; use a polymorphic (entity_type, entity_id) pair rather than per-entity tables; and partition by month so retention policies are enforced by DETACH PARTITION rather than DELETE — which would itself require the DELETE privilege you just revoked.
