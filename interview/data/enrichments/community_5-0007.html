A Community streaming service wants an engagement signal: which users watched something on three consecutive days? A three-day "binge streak" is a retention leading indicator. The interviewer is watching whether you dedupe to one row per user-per-day first, whether you detect consecutiveness with LAG or with gaps-and-islands, and whether you notice the solution leans on a SQLite-only date function.

User 501 watches Dec 1–2–3: a clean streak. User 503 watches Dec 10–11–12: also a streak. User 502 watches Dec 1 and Dec 3 but skips Dec 2 — two days, not consecutive — and is the trap that separates "watched on three days" from "watched on three days in a row".

2. The SQL — LAG triple-check

WITH recent AS (
    SELECT user_id, watch_date
    FROM watch_history
    WHERE watch_date BETWEEN '2025-11-29' AND '2025-12-15'
),
daily AS (
    SELECT user_id, watch_date
    FROM recent
    GROUP BY user_id, watch_date
),
triplet AS (
    SELECT
        user_id,
        watch_date AS d3,
        LAG(watch_date, 1) OVER w AS d2,
        LAG(watch_date, 2) OVER w AS d1
    FROM daily
    WINDOW w AS (PARTITION BY user_id ORDER BY watch_date)
)
SELECT user_id
FROM triplet
WHERE julianday(d3) - julianday(d2) = 1
  AND julianday(d3) - julianday(d1) = 2
GROUP BY user_id
ORDER BY user_id;

The daily CTE collapses multiple same-day plays into one row. triplet lines up today (d3), yesterday (d2), and the day before (d1) using two LAG offsets. The final filter keeps users where those three dates are exactly one and two days apart.

Dialect note: julianday() is a SQLite function — it converts a date to a Julian day number so the subtraction yields an integer day-count. Postgres has no julianday(); there you cast to date and subtract directly: (d3::date - d2::date) = 1. The logic is identical; only the date-difference idiom changes.

4. Strategic explanations

Dedupe first — the daily CTE is not optional

If a user watches three shows on Dec 1, raw watch_history has three Dec-1 rows. LAG over those would compare Dec 1 to Dec 1 (gap 0) and never see a real streak — or worse, manufacture a false one. Collapsing to one row per (user_id, watch_date) first is mandatory. The solution uses GROUP BY user_id, watch_date rather than SELECT DISTINCT; they are semantically identical, and the inline comment's claim that GROUP BY is faster is a folk optimisation — on most planners they produce the same plan. Either is fine; the point is that some dedup must happen.

The gaps-and-islands alternative — the cleaner generalisation

The LAG triple-check is hard-wired to exactly 3. For a parameterised "N days in a row" the gaps-and-islands trick scales without adding more LAG columns:

WITH daily AS (
    SELECT DISTINCT user_id, watch_date::date AS d FROM watch_history
),
grp AS (
    SELECT user_id, d,
           d - (ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY d))::int AS streak_key
    FROM daily
)
SELECT user_id
FROM grp
GROUP BY user_id, streak_key
HAVING COUNT(*) >= 3;

The insight: within a run of consecutive dates, date - row_number is constant — each step adds 1 to both, so the difference holds. That constant becomes a group key; counting rows per key gives the streak length. Change >= 3 to >= N and it answers any streak length — that flexibility is the senior reason to know this pattern.

The WINDOW clause and the partition prune

Two details worth voicing. First, WINDOW w AS (...) names a window spec so both LAG calls reuse it — cleaner than repeating OVER (PARTITION BY user_id ORDER BY watch_date) twice, and a hint that the engine evaluates it once. Second, the recent CTE's date filter (BETWEEN '2025-11-29' AND '2025-12-15') is a deliberate scan-reduction: a 3-day streak detector only needs a short lookback, so on a partitioned billion-row table this prunes whole partitions. Just make sure the window is wide enough — trimming it too tight could chop the first day off a streak that straddles the boundary.

The final GROUP BY user_id — dedupe the output

A user who watches for five straight days produces three qualifying triplet rows (days 3, 4, and 5 each close a 3-streak). Without the trailing GROUP BY user_id the same user would be listed three times. The group collapses them to one. SELECT DISTINCT user_id would do the same job.

Edge cases: data types and same-day boundaries

watch_date is stored as TEXT here. String BETWEEN works only because ISO YYYY-MM-DD sorts lexically the same as chronologically — a non-ISO format (MM/DD/YYYY) would break the filter silently. Casting to a real date early is safer. Also: if watch_date were a full timestamp, two watches at 23:00 and 01:00 are different calendar days but close in time — you must truncate to the date (watch_date::date) before deduping, or the streak logic misfires.

Senior signal: collapse to one row per user-per-day before any consecutiveness logic, know both the LAG triple-check (fixed length) and the date - ROW_NUMBER gaps-and-islands trick (parameterised length), dedupe the final output so a long streak is not double-counted, and recognise julianday() as a SQLite idiom that becomes plain date subtraction in Postgres.
