Nintendo's e-commerce team wants every order flagged with whether it was returned. The returns table only lists order ids that came back — absence means "kept". This is the textbook use of a LEFT JOIN: keep all orders, attach return information where it exists, and turn the presence-or-absence of a match into a boolean. An interviewer is checking whether you reach for LEFT JOIN over INNER JOIN, and whether you know how to test "did the join find a row".

Only 2 of the 5 orders were returned. The interesting property is that the returns table is sparse — orders 1, 3, and 4 simply have no row there. An INNER JOIN would drop them entirely; the whole point is to keep them and mark them as not returned.

2. The SQL — LEFT JOIN + IS NOT NULL

SELECT o.order_id, o.customer_id, r.order_id IS NOT NULL AS was_returned
FROM orders o LEFT JOIN returns r ON o.order_id = r.order_id;

Every order from orders is kept. Where a matching returns row exists, r.order_id is populated; where none exists, the LEFT JOIN fills it with NULL. The expression r.order_id IS NOT NULL converts that into a clean boolean flag.

4. Strategic explanations

Why LEFT JOIN, not INNER JOIN

INNER JOIN emits a row only when both sides match. Here that would silently discard every order that was not returned — the majority — and the report would claim a 100% return rate. LEFT JOIN guarantees every left-table (orders) row survives; unmatched right-table columns come back NULL. Whenever the question is "all of X, plus optional info from Y", the answer is a left join.

Testing the match: IS NOT NULL, never = NULL

"Did the join find a returns row?" is answered by inspecting any non-nullable column from the right side. r.order_id works because in the source returns table that column is never NULL — so a NULL there can only mean "no match". The test must be IS NOT NULL: the expression r.order_id = NULL evaluates to unknown, never true, because nothing equals NULL in three-valued logic.

The duplicate-rows trap — join fan-out

This works cleanly because order_id is unique in returns — one return per order at most. If returns could hold multiple rows per order (a return, then a re-return, or a sloppy load), the LEFT JOIN would fan out: order 2 would appear twice and the order count would inflate. Guard against it — either confirm uniqueness, or use an existence test that cannot duplicate:

SELECT o.order_id, o.customer_id,
       EXISTS (SELECT 1 FROM returns r WHERE r.order_id = o.order_id) AS was_returned
FROM orders o;

EXISTS is the more robust pattern for a pure yes/no flag: it returns one row per order regardless of how many matching returns rows exist, and it can stop scanning at the first hit.

Putting the match test in WHERE — the anti-join

A common follow-up: "show only the orders that were not returned." Moving the NULL test from SELECT to WHERE turns the left join into an anti-join:

SELECT o.order_id, o.customer_id
FROM orders o
LEFT JOIN returns r ON o.order_id = r.order_id
WHERE r.order_id IS NULL;

Beware the classic mistake: putting a filter on the right table in the WHERE clause (e.g. WHERE r.something = 'x') silently converts a LEFT JOIN back into an INNER JOIN, because NULL rows fail the predicate. Right-table conditions belong in the ON clause.

Aggregating to a return rate

The boolean flag rolls straight up into a metric: SELECT AVG((r.order_id IS NOT NULL)::int) AS return_rate FROM orders o LEFT JOIN returns r ON o.order_id = r.order_id; — here 2/5 = 0.4. Casting the boolean to int lets AVG treat it as a 0/1 indicator.

Senior signal: choose LEFT JOIN so kept orders are not silently dropped, derive the flag with IS NOT NULL (never = NULL), recognise the fan-out risk if the returns table is not unique on order_id and reach for EXISTS when a duplicate-proof yes/no is needed, and know that a right-table predicate in WHERE quietly degrades a left join to an inner join.
