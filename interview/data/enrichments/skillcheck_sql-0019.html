A product analytics team needs to sessionize a raw event stream: group a user's events into sessions, starting a fresh session whenever they go quiet for more than 30 minutes. Sessionization is a staple of funnel and engagement analysis, and the interviewer is probing the canonical two-step pattern — flag the session boundaries, then turn those flags into running session numbers.

1. The source data — event

user_idts(narrative)
12025-01-01 09:00:00user 1's first event — session starts
12025-01-01 09:30:00gap = exactly 30 min — the boundary
12025-01-01 10:01:00gap = 31 min — new session
22025-01-01 08:00:00user 2's only event — single-event session
32025-01-01 07:00:00user 3 first event
32025-01-01 09:00:00gap = 120 min — new session

Two edge cases carry the question. User 1's middle gap is exactly 30 minutes — and the rule is "more than 30", so it must not start a new session. User 2 has a single event — the LAG is NULL and that lone event must still land in session 1.

2. The SQL — flag boundaries, then running-sum the flags

WITH flagged AS (
  SELECT
    user_id,
    ts,
    CASE
      WHEN LAG(ts) OVER (PARTITION BY user_id ORDER BY ts) IS NULL THEN 1
      WHEN EXTRACT(EPOCH FROM (ts::timestamp - LAG(ts) OVER (PARTITION BY user_id ORDER BY ts)::timestamp))/60 > 30 THEN 1
      ELSE 0
    END AS is_new_session
  FROM event
)
SELECT
  user_id,
  ts,
  SUM(is_new_session) OVER (PARTITION BY user_id ORDER BY ts ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)::int AS session_id
FROM flagged
ORDER BY user_id, ts;

LAG fetches the previous event's timestamp per user. The CASE marks is_new_session = 1 for the very first event (no LAG) or for any gap over 30 minutes. The outer query runs a cumulative SUM of those flags — every new-session marker bumps the count.

User 1's 09:30 event stays in session 1: a 30-minute gap is not more than 30, so the > 30 test is false. User 2's single event is session 1 — the NULL LAG branch handles the partition's first (and only) row.

4. Strategic explanations

The flag-then-cumulative-sum pattern

This is the standard sessionization shape. Step one converts an inter-row condition (a time gap) into a per-row 0/1 marker with LAG + CASE. Step two runs a cumulative SUM over those markers: because the count only increments where a flag is 1, every row inside a session inherits the same total, and the total is the session id. This same two-step idea solves "islands and gaps" problems generally — recognise it once and reuse it everywhere.

The boundary — > 30 is strict, and that matters

The rule says "more than 30 minutes", so the predicate is strictly > 30, not >= 30. User 1's exactly-30-minute gap is the test case: with > it stays in the same session; with >= it would wrongly split. Always pin down whether a threshold is inclusive and make the operator match the wording — off-by-one on a boundary is the most common sessionization bug.

The first event must seed session 1

For each user's earliest event, LAG returns NULL — there is no prior row. The explicit WHEN LAG(ts) ... IS NULL THEN 1 branch handles that: the first event always flags 1, so the cumulative sum starts at 1. Omit that branch and the first event flags 0, every session_id is off by one, and a single-event user like user 2 would land in session 0. The NULL branch is not optional.

PARTITION BY user_id appears in both windows

The LAG and the cumulative SUM both partition by user_id, so neither one's previous row nor running total ever crosses a user boundary. Drop the partition from the LAG and a user's first event would compare against the previous user's last event; drop it from the SUM and session numbers would accumulate globally. Sessionization is inherently per-user — every window must say so.

Dialect — timestamp arithmetic and gap windows

ts is stored as TEXT, so the cast to timestamp is required before subtraction; EXTRACT(EPOCH FROM interval)/60 converts the resulting interval to minutes — idiomatic PostgreSQL. SQL Server would use DATEDIFF(MINUTE, prev, ts); MySQL TIMESTAMPDIFF(MINUTE, prev, ts). Storing timestamps as proper timestamp columns rather than text would remove the cast entirely — a worthwhile schema note.

Senior signal: sessionize with the two-step flag-then-cumulative-sum pattern, make the gap operator strictly > to match "more than 30 minutes", explicitly seed the first event (NULL LAG) as a new session, and partition every window by user so neither gaps nor counts leak across users.
