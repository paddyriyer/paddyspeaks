The pandas version of the velocity rule — the same algorithm Robinhood's risk team and Stripe Radar run in production, just compressed into a single dataframe so you can debug it interactively. The senior signal isn't that the candidate writes rolling('10min').count(); it's that they explain (a) why time-offset rolling beats fixed-row rolling for fraud, (b) why pandas-on-bounded-batch is the offline twin of the streaming detector — not a replacement for it, and (c) which legitimate user pattern this rule will misfire on if you don't tune it.

1. The fraud pattern — what we're catching and what we're trying NOT to flag

Velocity rules detect bursts: an unnatural concentration of activity from a single account in a short time window. In equities trading, the canonical attack is stolen-account high-frequency liquidation — the attacker has minutes before the rightful owner gets the SMS alert and revokes session, so they fire trades as fast as the rate-limiter allows.

PatternFraud signatureLegitimate pattern that looks similar
5+ trades in 60sStolen-account liquidation — sell every position before the owner regains controlDay-trader during open — a known active retail trader at 9:30 AM ET firing 3-4 limit orders into volatility
$50K notional in 5 minAccount takeover draining margin balance into a small-cap pumpWhale customer rebalancing — a $2M portfolio doing scheduled monthly reallocation
10 orders / 10 min / single assetWash trading or layering to manipulate quoteAlgorithmic DCA bot — customer's own auto-invest cron firing into a thin-liquidity name

The false-positive cost is real: flagging a whale's monthly rebalance freezes their account, generates a Trustpilot one-star, and triggers a churn save call. Production tuning uses a cohort-aware threshold (see customer_growth-0032) so whales get a higher bar than minnows.

3. The query — walk through window mechanics

df['order_ts'] = pd.to_datetime(df['order_ts'])
df = df.sort_values(['customer_id', 'order_ts'])   # required for offset-rolling

for cust, g in df.groupby('customer_id'):
    g = g.set_index('order_ts')                    # DatetimeIndex = unlocks '10min' offset
    cnt = g['order_id'].rolling('10min').count()   # time-offset, NOT fixed window=5
    amt = g['amount'].rolling('10min').sum()
    bad = (cnt > 4) | (amt > 2_000)
    if bad.any():

The three pandas gotchas an interviewer will probe
  • Time-offset rolling requires a monotonic DatetimeIndex. If you skip the sort_values, pandas falls back to the slow numpy path or raises ValueError: index must be monotonic. The sort is non-optional.
  • rolling('10min') vs rolling(window=5). The string form is a time offset — look back 10 wall-clock minutes regardless of row count. The integer form is 5 rows — meaningless for fraud because a sparse user's 5 rows could span a week.
  • bad.idxmax() finds the first True. Emit only the first offending row per customer; otherwise the alert queue gets one row per event in the burst. Alert fatigue is a real on-call problem.
Why pandas-on-stream is bounded — what production actually runs

pandas can't ingest an unbounded Kafka stream — there's no checkpoint, no watermark, no incremental state. What it can do is reproduce the exact algorithm on a finite slice (yesterday's orders, a backfill window, a unit-test fixture). The production detector is PyFlink with SlidingEventTimeWindows + KeyedProcessFunction (see customer_growth-0012). The pandas version is the offline twin used for:

  • Threshold tuning — risk analysts replay last quarter's data with max_orders ∈ {3,4,5,6} and pick the precision-recall sweet spot.
  • Incident forensics — "show me every customer that would have fired the rule last Tuesday between 2-4 PM" is a pandas notebook, not a Flink redeploy.
  • Unit tests — the Flink job's ProcessFunction calls into pure-Python logic that pandas exercises in milliseconds.

4. Why event-time matters — the L5 trap

If you run this on order_ts (event-time) you get the deterministic answer regardless of when pandas saw the row. If you accidentally use received_at (processing-time / ingestion time), a mobile client that flushed an order 4 minutes late lands in the WRONG 10-minute window — and a borderline burst silently shifts above or below the threshold.

ScenarioEvent-time answerProcessing-time answer
5 trades placed at 09:30:00 – 09:30:31, all arrived on timeFires too_many_orders with trigger_ts = 09:30:31Same
Same 5 trades, but order #4 stuck in offline mobile queue, arrived at 09:34:50Fires too_many_orders with trigger_ts = 09:30:31 — the rule still sees 5 trades in 31 wall-clock secondsMisses — the 4th trade lands in a different processing window, count is 4 instead of 5

The auditor question: "Did this trade actually happen in the 60-second window?" — only event-time can answer that. Processing-time tells you when your servers happened to see the row, which has no regulatory meaning. Production rule: every fraud rule is event-time, full stop. Processing-time is acceptable only for ops dashboards.

6. Interviewer probes

Probe 1 — "What happens when a customer has 50,000 orders in the dataframe? Does groupby(customer_id).rolling('10min') scale?"

It scales linearly per customer because the offset-rolling implementation walks a sorted index with two pointers (the standard sliding-window two-pointer trick). The bottleneck is the Python-level for-loop over groupby; the production speed-up is to use df.groupby('customer_id').rolling('10min', on='order_ts').count() which keeps the work in a single C call. For 100M rows you've outgrown pandas — that's the cue to mention Polars (lazy, multi-threaded) or move to PyFlink.

Probe 2 — "How do you handle the whale customer who legitimately fires 10 orders/min?"

Cohort-aware threshold: classify customers into Whale / Dolphin / Minnow (see customer_growth-0032) and apply a different max_orders per cohort. Whales get max_orders=20; minnows get max_orders=4. The same rule with different parameters. Don't hard-code one threshold and patch with an allowlist — allowlists rot.

Probe 3 — "Backfill — can you re-run yesterday's stream through this rule with a new threshold?"

Yes, that's the killer feature of the pandas twin. Snapshot yesterday's events from the data lake (Parquet), call velocity_alerts(df, max_orders=3), diff against the production alerts. The Flink job can't easily do this because its state has already advanced past yesterday — you'd need a savepoint replay, which is expensive. The senior pattern: keep the rule logic in a pure function, call it from both Flink (for live) and pandas (for backfill/tuning).

Probe 4 — "What if order_ts has duplicates (two orders with the same timestamp)?"

Time-offset rolling tolerates duplicates — both rows fall into the same window. The idxmax() picks one of them as the trigger_ts (the first one in sort order). If you need a tiebreaker, sort by (customer_id, order_ts, order_id) instead of just (customer_id, order_ts).

Probe 5 — "Hot key — one customer (HFT prop account) generates 100K orders/sec. Will this break?"

In pandas, the whale's group becomes a giant rolling computation on one thread — wall-clock dominated by that single group. In the Flink twin, that whale's keyed state lives on one task manager's RocksDB and triggers a hot-shard problem. The fix is the same shape as for aggregations: salting + two-pass. See Hot Shards & Skew design page §03 and customer_growth-0027 for the canonical salt pattern.

Probe 6 — "Why emit only the first offending row per customer, not every row that breaches?"

Alert fatigue. A 20-order burst inside the window means 16 consecutive rows are "bad" — emitting all 16 floods the analyst queue with rows that all describe the same incident. The first row is the one with maximum signal ("here's the moment the rule fired"); subsequent rows are redundant until the burst clears and starts fresh. Production systems wrap this in a cooldown: don't re-alert the same customer within 1 hour of the last alert.
