Salesforce's Sales Ops dashboard: average days in each pipeline stage. Stages with high duration are the bottlenecks; coaching focuses there.

2. The SQL — LEAD-based duration

WITH transitions AS (
  SELECT opp_id, stage, entered_at,
    LEAD(entered_at) OVER (PARTITION BY opp_id ORDER BY entered_at) AS exited_at
  FROM opp_stages
)
SELECT stage, COUNT(*) AS transitions,
  ROUND(AVG(CASE WHEN exited_at IS NOT NULL
    THEN (julianday(exited_at) - julianday(entered_at)) END), 1) AS avg_days_in_stage
FROM transitions GROUP BY stage ORDER BY avg_days_in_stage DESC;

3. Strategic explanations

LEAD pattern for time-in-state

Each stage row has a entered_at. The next-row's entered_at = THIS row's exit. LEAD grabs it. The last stage of a deal has exited_at IS NULL — the deal is currently in that stage.

The current-stage exclusion

Open deals have unfinished stage durations. Excluding them with WHERE exited_at IS NOT NULL shows historical avg. Including them with COALESCE(exited_at, NOW()) shows current cycle health.

The "stage skipped" pattern

Some deals jump stage 2 → stage 5. The schema records the jump as one row. Production tracks "stage skip rate" as a separate metric — high skip rates mean the stage is non-binding.

Median > mean for duration

Two stuck deals (180 days) skew avg. Median is more representative. SQLite needs a workaround:

WITH ranked AS (SELECT stage, days, ROW_NUMBER() OVER (PARTITION BY stage ORDER BY days) AS rn,
  COUNT(*) OVER (PARTITION BY stage) AS n FROM ...)
SELECT stage, AVG(days) AS median FROM ranked WHERE rn IN (n/2, n/2+1) GROUP BY stage;
Senior signal: LEAD for time-in-state, distinguish historical-avg from current-cycle-health via NULL handling, surface stage-skip rate as a separate metric, and prefer median over mean for skewed durations.
