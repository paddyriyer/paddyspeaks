Google Maps ETA quality directly affects user trust — overestimating traffic loses users to Apple Maps; underestimating loses them to "Google was wrong again". The query measures ETA bias and MAE per travel mode (driving, walking, transit, cycling).

1. The source data — maps_eta

2. The SQL — bias + MAE + over-prediction rate per mode

SELECT mode,
  COUNT(*) AS trips,
  ROUND(AVG(predicted_minutes - actual_minutes), 2)             AS mean_bias,
  ROUND(AVG(ABS(predicted_minutes - actual_minutes)), 2)        AS mae,
  ROUND(100.0 * SUM(CASE WHEN predicted_minutes > actual_minutes THEN 1 ELSE 0 END)
              / NULLIF(COUNT(*), 0), 1) AS pct_overestimated
FROM maps_eta
GROUP BY mode
ORDER BY mae DESC;

3. Strategic explanations

Bias vs MAE — different stories

Bias is signed (over- or under-prediction); MAE is unsigned (magnitude only). A model with bias=0 but MAE=10 is consistently wrong but balanced. A model with bias=+5, MAE=5 is systematically over-predicting by exactly 5 minutes — easy to fix with a calibration step.

Why "over-estimate" is preferred — UX physics

Slightly over-predicted ETAs result in users arriving early ("nice surprise"). Under-predicted ETAs cause missed appointments and trust collapse. Production targets +2 to +5% positive bias. Pure zero is suspicious.

Sorting MAE descending should produce: transit > driving > cycling > walking. If your data shows otherwise, investigate.

The traffic-condition normalization

"Driving MAE 5 min" means very different things in rush hour vs midnight. Always slice by hour of day:

GROUP BY mode, CASE WHEN CAST(strftime('%H', started_at) AS INT) BETWEEN 17 AND 20 THEN 'rush' ELSE 'offpeak' END
The percentile tail

MAE is mean error. The metric Google actually monitors is P95 — 95th percentile error. A model with mean MAE 3 min but P95 = 25 min has long-tail user-disaster sessions. Use NTILE or PERCENT_RANK to estimate.

The destination-zone confound

ETAs to airports are systematically harder (multiple terminals, parking lots, security). ETAs to home addresses are easy. Mixing them in one number obscures both. Production splits by destination type.

The retraining trigger

Maps' real-time learning pipeline ingests this query nightly. Cells where MAE drift > 2x baseline trigger an investigation; sustained drift triggers retraining. The query result is feedstock for the model lifecycle, not just analytics.

Senior signal: distinguish bias (signed) from MAE (unsigned), pin the target as slight positive bias for UX physics, expect the mode hardness ordering (transit > driving > cycling > walking), and surface P95 as the production-monitored metric. The retraining-trigger feedstock framing is the over-the-bar tie-back.
