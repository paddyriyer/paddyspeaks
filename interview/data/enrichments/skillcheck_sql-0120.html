A payroll reconciliation job compares the HR roster against the salary-paid ledger — the two should always agree. The task is to surface the discrepancies: employee IDs that appear in exactly one table, meaning someone was enrolled but never paid, or paid but never enrolled. The interviewer wants a FULL OUTER JOIN, because that is the one join that keeps unmatched rows from both sides at once.

The two edge cases are mirror images: emp_id 3 exists only in enrolled_employee, emp_id 4 exists only in paid_salary. A LEFT or RIGHT join would catch one direction and miss the other — the FULL OUTER JOIN catches both in a single pass.

2. The SQL — FULL OUTER JOIN, keep the NULL-on-either-side rows

SELECT COALESCE(e.emp_id, p.emp_id) AS emp_id
FROM enrolled_employee e
FULL OUTER JOIN paid_salary p ON e.emp_id = p.emp_id
WHERE e.emp_id IS NULL OR p.emp_id IS NULL
ORDER BY emp_id;

The FULL OUTER JOIN emits one row per matched pair plus one row for every unmatched left row (right side all NULL) plus every unmatched right row (left side all NULL). WHERE e.emp_id IS NULL OR p.emp_id IS NULL discards the matched pairs and keeps only the orphans.

4. Strategic explanations

Why COALESCE is mandatory, not decorative

For an orphan row, exactly one of e.emp_id / p.emp_id holds the value — the other is NULL. Selecting just e.emp_id would return NULL for every paid-but-not-enrolled employee (id 4 would vanish into a NULL). COALESCE(e.emp_id, p.emp_id) picks whichever side is populated, so each orphan reports its real ID. This pairing — FULL OUTER JOIN plus COALESCE on the join key — is the canonical idiom.

This is a symmetric difference

The query computes the symmetric difference of the two ID sets: members of either set but not both. Drop the WHERE entirely and you get a full reconciliation (the union, with matches and mismatches side by side). Keep only p.emp_id IS NULL and you get a one-directional anti-join (enrolled but unpaid). The clause e.emp_id IS NULL OR p.emp_id IS NULL is what makes it symmetric — choose the variant the question actually asks for.

Test the join keys for NULL, not other columns

WHERE filters on emp_id, the join keys — correct, because a join key is NULL precisely when that side had no matching row. Testing a different nullable column (say e.name) would also work if that column happens to be non-nullable in real rows, but it is fragile: a genuine NULL name would be misread as a missing row. The join key is the reliable witness of "no match".

Dialect — MySQL has no FULL OUTER JOIN

PostgreSQL, SQL Server, and Oracle support FULL OUTER JOIN natively. MySQL does not — you emulate it with LEFT JOIN … UNION … RIGHT JOIN, or here more simply with the symmetric anti-join (SELECT emp_id FROM enrolled_employee WHERE emp_id NOT IN (SELECT emp_id FROM paid_salary)) UNION (SELECT emp_id FROM paid_salary WHERE emp_id NOT IN (SELECT emp_id FROM enrolled_employee)). Knowing the portable fallback — and that NOT IN needs non-nullable keys to be safe — is the senior signal.

Senior signal: reach for FULL OUTER JOIN when you need unmatched rows from both sides at once, always COALESCE the two join keys so orphans report a real ID, recognise e.key IS NULL OR p.key IS NULL as the symmetric difference, and know MySQL lacks the operator so you can offer the UNION fallback.
