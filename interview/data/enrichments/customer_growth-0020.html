A growth-analytics team needs the full life story of every engagement burst: not just "how many consecutive days did this user show up," but what happened after the streak ended — did they come back the next day, the next week, or never. The interviewer is probing two things at once: can you run gaps-and-islands to detect the streaks, and can you then layer a retention/exit state machine on top of the islands. The snapshot date matters — "no activity yet" means churned only if enough observation time has elapsed.

2. The SQL — gaps-and-islands streaks plus a 5-bucket exit state machine

WITH params(snapshot) AS (VALUES (DATE '2025-04-01')),
days AS (
    SELECT user_id, activity_date
    FROM user_activity
    WHERE activity_date >= (SELECT snapshot FROM params) - INTERVAL '540 days'
    GROUP BY user_id, activity_date
),
indexed AS (
    -- island id = date - row_number(): consecutive dates collapse to
    -- the same island_id; a gap breaks it.
    SELECT user_id, activity_date,
        activity_date - (ROW_NUMBER() OVER (PARTITION BY user_id
                         ORDER BY activity_date))::INT AS island_id
    FROM days
),
streaks AS (
    SELECT user_id, island_id,
        MIN(activity_date) AS streak_start,
        MAX(activity_date) AS streak_end,
        COUNT(*)           AS streak_days,
        ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY MIN(activity_date))
            AS streak_id
    FROM indexed
    GROUP BY user_id, island_id
),
next_activity AS (
    SELECT s.user_id, s.streak_id, s.streak_end,
        MIN(d.activity_date) AS next_activity_date
    FROM streaks s
    LEFT JOIN days d ON d.user_id = s.user_id
                    AND d.activity_date > s.streak_end
    GROUP BY s.user_id, s.streak_id, s.streak_end
),
enriched AS (
    SELECT s.user_id, s.streak_id, s.streak_start, s.streak_end, s.streak_days,
        ((SELECT snapshot FROM params) - s.streak_end) AS days_since_streak_end,
        n.next_activity_date,
        CASE WHEN (n.next_activity_date - s.streak_end) <= 1  THEN 1 ELSE 0 END AS d1_retained,
        CASE WHEN (n.next_activity_date - s.streak_end) <= 7  THEN 1 ELSE 0 END AS d7_retained,
        CASE WHEN (n.next_activity_date - s.streak_end) <= 28 THEN 1 ELSE 0 END AS d28_retained
    FROM streaks s
    LEFT JOIN next_activity n ON n.user_id = s.user_id AND n.streak_id = s.streak_id
)
SELECT user_id, streak_id, streak_start, streak_end, streak_days,
    d1_retained, d7_retained, d28_retained, next_activity_date,
    CASE
        WHEN d28_retained = 1                                         THEN 'continuing'
        WHEN next_activity_date IS NULL AND days_since_streak_end > 90 THEN 'churned'
        WHEN next_activity_date IS NULL                               THEN 'open'
        WHEN (next_activity_date - streak_end) > 90                   THEN 'long_resurrect'
        ELSE                                                               'resurrected'
    END AS exit_status
FROM enriched
ORDER BY user_id, streak_id;

4. Strategic explanations

Why date - row_number() detects islands

Inside a run of consecutive calendar days, activity_date increases by 1 each row and ROW_NUMBER() also increases by 1. Their difference is therefore constant for the whole run — that constant is the island id. The moment a day is skipped, the date jumps but the row number does not, so the difference shifts and a new island opens. The ::INT cast on ROW_NUMBER() is required because PostgreSQL will not subtract a bigint from a date directly.

The streak vs the next streak — a subtle definition trap

next_activity_date is the earliest activity day strictly after streak_end, which is usually the first day of the next streak. That is deliberate: D1/D7/D28 retention asks "did the user come back," and the start of the next burst is exactly that signal. A common mistake is to join streaks to streaks and measure gap-between-islands; joining back to the day grid is simpler and correct. Note d1_retained is 0 everywhere here — by construction a gap of exactly 1 day would have merged the two streaks into one island, so a separate streak can never be D1-retained.

NULL handling makes the exit ladder correct

When there is no follow-up activity, next_activity_date is NULL and (NULL - streak_end) is NULL, so all three retained flags fall to 0. The exit CASE then leans on days_since_streak_end: if more than 90 days have passed with no return it is churned; otherwise the observation window is still open, hence open. Order matters in that CASE — the d28_retained = 1 branch must come first so a returning user is never mislabelled, and the two NULL branches must precede the gap-size branches because subtracting NULL would otherwise never fire.

Dialect notes — date arithmetic is not portable

date - date returning an integer count of days is PostgreSQL behaviour. SQLite needs julianday() differences; MySQL needs DATEDIFF(); BigQuery needs DATE_DIFF(a, b, DAY); Snowflake needs DATEDIFF('day', a, b). The 540-day lookback window (snapshot - INTERVAL '540 days') is a partition-pruning guard so the scan never reads the entire history table — it just needs to reach far enough back to cover the longest plausible gap.

What a senior says next

"This is a per-streak fact table. In production I would materialise it incrementally — only streaks whose exit_status is still open can change, so each night you re-evaluate just those rows against the new snapshot. Everything labelled continuing, churned, or resurrected is frozen and never needs recomputing."

Senior signal: name the date - row_number() island trick, explain why a separate streak can never be D1-retained (a 1-day gap would have merged it), order the exit-status CASE so the retained branch and the NULL branches fire before the gap-size branches, and propose incrementally refreshing only the still-open streaks.
