Project management and ERP systems need total cost rollups: "For every project, compute the duration in days, the number of distinct employees assigned, total labor cost (hours × rate), total equipment cost, and the grand total — even for projects that have no labor or no equipment." This question probes LEFT JOIN handling, COALESCE for NULL-to-zero conversion, the dangerous SUM(DISTINCT ...) pattern, and correct date-arithmetic in SQL.

2. The SQL — multi-table LEFT JOIN with COALESCE and SUM(DISTINCT)

SELECT p.project_id, p.name,
       (p.end_date::date - p.start_date::date) AS duration_days,
       COUNT(DISTINCT a.emp_id) AS employees,
       COALESCE(SUM(a.hours::numeric * a.hourly_rate::numeric), 0) AS labor_cost,
       COALESCE(SUM(DISTINCT e.cost::numeric), 0) AS equipment_cost,
       COALESCE(SUM(a.hours::numeric * a.hourly_rate::numeric), 0)
         + COALESCE(SUM(DISTINCT e.cost::numeric), 0) AS total_cost
FROM projects p
LEFT JOIN assignments      a ON p.project_id = a.project_id
LEFT JOIN equipment_costs  e ON p.project_id = e.project_id
GROUP BY p.project_id, p.name, p.start_date, p.end_date
ORDER BY total_cost DESC;

4. Strategic explanations

Why both joins must be LEFT JOIN

If either join were INNER, projects with no employees or no equipment would vanish from the result. The question explicitly requires all projects. The pattern is: start with the anchor table (projects) and LEFT JOIN all optional fact tables so the anchor always appears. NULL values in LEFT JOIN non-matching rows are then neutralised by COALESCE(..., 0).

The SUM(DISTINCT e.cost) hazard

When both LEFT JOINs are active simultaneously, the Cartesian product of assignments and equipment rows inflates the row count: P1 has 2 assignment rows × 2 equipment rows = 4 intermediate rows. If SUM(e.cost) were used without DISTINCT, each equipment cost would be counted twice (once per assignment row). SUM(DISTINCT e.cost) deduplicates the costs before summing. This is a well-known multi-join aggregation trap. The production-safe alternative is to pre-aggregate each table in a CTE before joining, avoiding the row multiplication entirely.

The production-safe CTE alternative
WITH lab AS (
  SELECT project_id,
         COUNT(DISTINCT emp_id) AS employees,
         SUM(hours::numeric * hourly_rate::numeric) AS labor_cost
  FROM assignments GROUP BY project_id
),
equip AS (
  SELECT project_id, SUM(cost::numeric) AS equipment_cost
  FROM equipment_costs GROUP BY project_id
)
SELECT p.project_id, p.name,
       (p.end_date::date - p.start_date::date) AS duration_days,
       COALESCE(lab.employees, 0)       AS employees,
       COALESCE(lab.labor_cost, 0)      AS labor_cost,
       COALESCE(equip.equipment_cost, 0) AS equipment_cost,
       COALESCE(lab.labor_cost, 0) + COALESCE(equip.equipment_cost, 0) AS total_cost
FROM projects p
LEFT JOIN lab   ON p.project_id = lab.project_id
LEFT JOIN equip ON p.project_id = equip.project_id
ORDER BY total_cost DESC;

This removes the SUM(DISTINCT) danger and is cleaner to read. The CTE approach is preferred in production; knowing both patterns — and being able to explain why the direct form needs DISTINCT — is the senior signal.

Date arithmetic: Postgres vs Snowflake
  • Postgres (solution dialect): date - date returns an integer number of days directly — the ::date cast normalises timestamp columns.
  • Snowflake: DATEDIFF('day', start_date, end_date) or end_date::date - start_date::date (same subtraction works).
  • BigQuery: DATE_DIFF(end_date, start_date, DAY).
  • MySQL: DATEDIFF(end_date, start_date).
Senior signal: identify the multi-join row-multiplication problem that makes SUM(DISTINCT e.cost) necessary, explain why this is dangerous (duplicate cost values would be merged even if two different equipment items cost the same amount), and immediately offer the CTE pre-aggregation pattern as the correct production alternative. Also state that both LEFT JOINs are required so projects with no labor or no equipment still appear.
