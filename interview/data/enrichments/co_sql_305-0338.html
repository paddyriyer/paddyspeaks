Google Search Suggest (autocomplete) is one of the highest-leverage ML systems on the planet — every accepted suggestion saves a few hundred ms across billions of queries. Acceptance rate by prefix-length tells the team whether the model is firing too early (low quality, low acceptance) or too late (user already typed the full query).

1. The source data — autocomplete_events

2. The SQL — per-prefix-length acceptance

SELECT prefix_length,
  COUNT(*) AS shows,
  SUM(accepted) AS accepts,
  ROUND(100.0 * SUM(accepted) / NULLIF(COUNT(*), 0), 1) AS acceptance_pct
FROM autocomplete_events
GROUP BY prefix_length
ORDER BY prefix_length;

3. Strategic explanations

The expected curve shape
prefix lengthtypical acceptancesignal
1-2 chars5-15%too early — many possible queries
3-5 chars30-50%sweet spot
6-10 chars40-60%good — model has enough signal
11+ chars10-20%too late — user nearly done typing

The curve is bell-shaped. A flat curve means the suggestion model isn't adapting to confidence; a sharp peak at 5 chars means the model "decides" too rigidly.

Time saved — the latent metric

Acceptance rate alone hides the time savings. Accepting a 50-char suggestion at length 5 saves a lot more than accepting a 7-char suggestion at length 5. Production multiplies:

SUM(accepted * (suggestion_length - prefix_length))   -- chars saved

Total chars saved is the system's value-creation metric.

The acceptance-without-show ghost

Some users accept by pressing Enter on a query that happens to match a suggestion they didn't see. The instrumentation must distinguish:

  • Suggestion shown + user clicked it = explicit acceptance
  • Suggestion shown + user typed the same thing = passive acceptance
  • Suggestion shown + user typed something different = rejection
  • Suggestion not shown = no event
Prefix-length-N is per-CHAR, not per-WORD

"barack obama" at length 6 = "barack" — first word complete. The model's behavior at word boundaries differs from mid-word: word-end suggestions tend to accept more. Slice:

GROUP BY prefix_length, CASE WHEN substr(prefix, -1) = ' ' THEN 'word_boundary' ELSE 'mid_word' END
The mobile-vs-desktop divide

Mobile users accept suggestions much more readily (typing is harder). Desktop users often type through. Curve shape diverges: mobile peaks at length 3-4; desktop peaks at length 6-8. Production splits.

The personalization layer

Personalized autocomplete (using your search history) has 2x acceptance over generic. The query above doesn't differentiate. Join to autocomplete_personalization_flag to split.

Senior signal: describe the bell-shaped expected curve, name the chars-saved value metric beyond raw acceptance, distinguish active/passive/no-show acceptance variants, and pre-empt the mobile vs desktop split. The personalization-flag stratification is the over-the-bar finish.
