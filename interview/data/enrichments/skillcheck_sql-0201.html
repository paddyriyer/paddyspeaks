Log parsing in SQL is a practical requirement whenever structured application logs are stored in a text column — a common pattern in data warehouses that ingest raw log lines before downstream ETL. The interviewer is probing whether you can use a regex-based string function to extract a structured field from a semi-structured text, specifically using regexp_replace with a capture group backreference to isolate the level token.

1. The source data — log_line

idraw
1[2025-03-04] level=WARN msg=disk full
2[2025-03-04] level=INFO msg=startup ok
3[2025-03-05] level=ERROR msg=timeout

Each raw line has the format [date] level=TOKEN msg=…. The target token is the run of uppercase letters immediately after level= and before the next space.

2. The SQL — regexp_replace with a capture group

SELECT
  id,
  regexp_replace(raw, '^.*level=([A-Z]+).*$', '\1') AS level
FROM log_line
ORDER BY id;

The regex ^.*level=([A-Z]+).*$ matches the entire string. The capture group ([A-Z]+) captures the uppercase level token. The replacement string \1 emits only the first capture group, discarding everything else — so the function effectively extracts the token rather than replacing text.

3. The resulting analysis

idlevel
1WARN
2INFO
3ERROR

Each row's raw text is fully consumed by the regex (anchors ^ and $, greedy wildcards on both sides). The replacement \1 emits only the captured token. For row 1: WARN. Row 2: INFO. Row 3: ERROR.

4. Strategic explanations

regexp_replace as an extractor

The idiomatic use of regexp_replace for extraction relies on matching the whole string with anchors (^…$) and replacing the entire match with just the capture group via \1. This turns a replacement function into a pure extractor. The alternative — SUBSTRING(raw FROM 'level=([A-Z]+)') (PostgreSQL regex syntax) — is more directly an extractor: it returns the first match of the pattern, and if a capture group is present, it returns the captured group rather than the full match. Both approaches are correct; SUBSTRING … FROM is often cleaner for simple extraction.

Anchor discipline: why ^.* and .*$

The leading ^.* absorbs everything before level= (the timestamp and bracket). The trailing .*$ absorbs everything after the token (the space and msg field). Without these anchors the regex would match only the level=WARN portion and regexp_replace would leave the non-matching parts intact, returning the original string with only the matched substring replaced — giving [2025-03-04] WARN msg=disk full rather than WARN.

Character class [A-Z]+ and edge cases

The pattern [A-Z]+ captures only uppercase ASCII letters. If level tokens could be lowercase or mixed-case, you would use [A-Za-z]+ or \w+. If the token could contain digits or hyphens (e.g., level=HTTP-500), you would widen the class. Clarifying the token alphabet during the interview shows production awareness.

SUBSTRING vs regexp_replace — when to prefer each

SUBSTRING(raw FROM 'level=([A-Z]+)') returns NULL if the pattern does not match, which is often the safer default for nullable raw data. regexp_replace returns the original string unchanged if the pattern does not match — meaning a row with no level= field would return its entire raw text as "level", silently poisoning the result. For robustness, adding a WHERE raw ~ 'level=' guard or using SUBSTRING is preferable in production.

Dialect notes

PostgreSQL uses POSIX regex: regexp_replace(str, pattern, replacement) and SUBSTRING(str FROM pattern) with \1 backreferences. MySQL uses REGEXP_REPLACE (8.0+) with $1 backreferences in the replacement string. SQL Server uses PATINDEX + string functions (no regex replacement natively; use CLR or a JSON parsing trick). BigQuery has REGEXP_EXTRACT(str, pattern) which directly returns the first capture group — the cleanest form. DuckDB supports both POSIX and REGEXP_EXTRACT.

Senior signal: use regexp_replace(raw, '^.*level=([A-Z]+).*$', '\1') to extract via a full-string anchor + capture group replacement; explain why the anchors are necessary (without them non-matching text is left in place); note that SUBSTRING(raw FROM 'level=([A-Z]+)') is often cleaner for extraction and returns NULL on no-match rather than the raw string; and name BigQuery's REGEXP_EXTRACT as the most concise portable equivalent.
