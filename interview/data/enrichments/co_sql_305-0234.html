Deutsche Bank's risk-reporting desk charts a daily metric — say a P&L figure — and the raw line is too jagged to read a trend off. They want a 7-day moving average smoothed alongside the raw series. The query is a single windowed AVG with a row frame. It looks compact, but the interviewer is checking whether you understand what the window actually averages at the start of the series.

1. The source data — metrics

metric_datevalue
2026-01-0110
2026-01-0220
2026-01-0330
2026-01-0440
……
2026-01-0880

Eight consecutive days, values climbing 10, 20, 30 … 80. The first six rows are interesting: each has fewer than 7 prior days available, so the "7-day" average is computed over a shorter window. That partial-window behaviour is the heart of the question.

2. The SQL — AVG OVER (ROWS 6 PRECEDING)

SELECT metric_date, value,
  AVG(value) OVER (ORDER BY metric_date
                   ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS ma7
FROM metrics;

The frame ROWS BETWEEN 6 PRECEDING AND CURRENT ROW spans the current row plus the six before it — seven rows when seven exist. AVG averages value over whatever the frame currently holds.

On 01-01 there is one row in the frame, so ma7 = 10 (just itself). By 01-07 the frame is full — seven rows, average of 10…70 = 40. On 01-08 the window slides: it now covers 20…80, averaging 50. The first six values are not true 7-day averages — they are "average so far". The result type is high-precision numeric; round it for display with ROUND(…, 2).

4. Strategic explanations

The partial-window trap — the chart's first six points lie

A reader assumes every ma7 point averages seven days. The first six do not — they average 1 through 6. That makes the early curve look smoother and rise faster than a real 7-day average would. If the requirement is "only show the average once 7 full days exist", null out the warm-up:

CASE WHEN COUNT(*) OVER (ORDER BY metric_date
                         ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) = 7
     THEN AVG(value) OVER (ORDER BY metric_date
                           ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)
END AS ma7
ROWS vs RANGE — rows are not days

ROWS 6 PRECEDING counts six physical rows back. If the series has gaps — a missing weekend, a holiday — six rows might span ten calendar days, and the "7-day" average silently covers the wrong window. For a genuine 7-calendar-day average, switch to a RANGE frame on the date:

AVG(value) OVER (ORDER BY metric_date
                 RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW)

Naming the rows-vs-range distinction, and which one the data's gappiness demands, is the senior move. If the series is guaranteed dense (one row per day, no gaps), ROWS is fine and faster.

Duplicate dates and the RANGE peers caveat

If two rows share a metric_date, a RANGE frame treats them as peers — they enter and leave the frame together, and a CURRENT ROW boundary includes both. A ROWS frame would treat them as distinct positions. Confirm the grain is one row per day; if not, pre-aggregate to a daily total first.

Centred vs trailing, and ordering determinism

This is a trailing average — it only looks backward, which is correct for real-time dashboards (no future data leakage). A centred average uses ROWS BETWEEN 3 PRECEDING AND 3 FOLLOWING and is common in retrospective analysis. Also: the ORDER BY inside the window must be unambiguous — if dates can tie, add a tiebreaker so the frame is deterministic.

Senior signal: flag that the first six rows are partial windows that misrepresent the trend, distinguish ROWS (physical rows) from RANGE (calendar interval) and pick RANGE when the series can have gaps, choose a trailing frame to avoid future-data leakage, and round the high-precision numeric result for display.
