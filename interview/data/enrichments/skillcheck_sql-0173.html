Bucketing customers into LTV deciles is a standard segmentation step in CRM and marketing: it lets you identify the top 10%, the bottom 10%, and every stratum between them in a single query. The interviewer is probing whether you know NTILE(n) and understand its bucket-assignment semantics — particularly how it handles remainder rows that cannot be divided evenly across buckets.

Exactly 10 rows for 10 buckets — the clean case. Each customer lands in a unique decile (1 through 10). The schema is deliberately tidy to make the decile concept clear; real datasets have many more rows and ties, which is where NTILE's behaviour matters most.

2. The SQL — NTILE window function

SELECT cust_code, ltv,
       NTILE(10) OVER (ORDER BY ltv) AS decile
FROM customer
ORDER BY decile DESC, ltv DESC, cust_code;

NTILE(10) OVER (ORDER BY ltv) sorts all rows by ltv ascending and partitions them into 10 as-equal-as-possible groups, labelling each group 1 (lowest) through 10 (highest). There is no PARTITION BY because all customers share one global ranking. The ORDER BY in the output sorts descending so the most valuable segment (decile 10) appears first.

4. Strategic explanations

How NTILE(n) assigns rows to buckets

NTILE(n) divides the ordered row set into n groups as evenly as possible. If the row count does not divide evenly by n, the first row_count % n buckets each get one extra row (they are larger); the remaining buckets are one row smaller. For example, 11 rows into 10 buckets: buckets 1–1 get 2 rows, buckets 2–10 get 1 row each. The larger buckets always come first in the ordering, which matters when interpreting bucket sizes. There is no guarantee that a bucket labelled “decile 1” contains exactly 10% of the population when the row count is not a multiple of 10.

NTILE vs PERCENT_RANK vs CUME_DIST

NTILE assigns a discrete bucket number. PERCENT_RANK returns a value in [0, 1] representing relative rank (0 for the lowest, 1 for the highest); it is not a bucket. CUME_DIST returns the fraction of rows at or below the current row — useful for the question “what percentile is this row?” rather than “which bucket does this row belong to?”. Use NTILE when you need discrete segment labels for downstream GROUP BY or reporting; use CUME_DIST when you need a continuous percentile score.

Tie handling — the hard edge case

If two customers have the same LTV, NTILE can assign them to different buckets, splitting a tie group across a bucket boundary. This is often surprising: two customers with LTV 500 might end up in decile 5 and decile 6. If the business requirement is “ties must be in the same bucket,” NTILE is the wrong tool; use PERCENT_RANK or CUME_DIST with manual thresholds instead. Raising this in the interview demonstrates you have applied bucketing in a real production context.

Performance and dialect portability

NTILE is a standard SQL window function supported by PostgreSQL, SQL Server, Oracle, Snowflake, BigQuery, and MySQL 8.0+. SQLite does not support it natively. The function requires a full sort of the partition, so it is O(n log n) in the partition size. On very large tables a pre-aggregated materialized view of percentile breakpoints with a range join is often faster than a live window function pass.

Senior signal: know that NTILE(n) makes the first row_count % n buckets larger when the count is not evenly divisible; understand that ties can be split across bucket boundaries (and that CUME_DIST is the fix when ties must stay together); and confirm the output ORDER BY reflects the business requirement (decile DESC puts the most valuable segment first).
