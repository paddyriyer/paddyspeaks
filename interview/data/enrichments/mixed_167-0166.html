The same hot-partition rescue as python_175-0002, framed as a mixed-language problem so the discussion can drift from Python to Kafka, DynamoDB, Spark, or Flink. The class is a Python crystallisation of an algorithm that ships in every major distributed-data product: salt the known hot keys with a suffix, leave the cold tail unchanged, accept the read-side scatter-gather as the price of write-side balance.

1. The problem — concrete numbers

A B2B SaaS event stream, 16 Kafka partitions, three customer tiers:

TierCountEvents/sec eachTier total
Whale (user_42)11,0001,000
Dolphins (u000–u099)100101,000
Minnows (m0000–m0999)1,00011,000

Naive hash(key) % 16 sends every user_42 event to one partition. That partition ends up with ~1,125 ev/s; the other 15 partitions sit at ~125 ev/s. Skew ratio: 9×. The consumer for the hot partition lags forever; the other 15 idle.

Apply an 8-bucket salt to user_42 only: the whale's 1,000 ev/s spread across 8 partitions (~125 ev/s each), so the busiest partition tops out near 250 ev/s. Skew ratio falls to ~2×. Same data, same hash, same partition count — one config change.

2. The source code — with a runnable demo

import hashlib
import random
from collections import Counter

class SkewAwarePartitioner:
    """Hash-partitions events; salts known hot keys so a single producer key
    fans out across `salt_factor` partitions instead of pinning one."""
    def __init__(self, n_partitions, hot_keys, salt_factor=8):
        self.n = n_partitions
        self.hot = set(hot_keys)
        self.salt = salt_factor

    def partition(self, key, row_index=0):
        k = f"{key}#{row_index % self.salt}" if key in self.hot else key
        return int(hashlib.md5(k.encode()).hexdigest(), 16) % self.n


if __name__ == "__main__":
    random.seed(7)
    stream = (
        [("user_42", i) for i in range(1000)] +
        [(f"u{j:03d}", i) for j in range(100) for i in range(10)] +
        [(f"m{k:04d}", 0) for k in range(1000)]
    )
    random.shuffle(stream)

    naive = SkewAwarePartitioner(n_partitions=16, hot_keys=set())
    nl = Counter(naive.partition(k, i) for k, i in stream)
    print("Naive  -> max:", max(nl.values()), "min:", min(nl.values()),
          "skew:", round(max(nl.values())/min(nl.values()), 2), "x")

    aware = SkewAwarePartitioner(n_partitions=16, hot_keys={"user_42"}, salt_factor=8)
    al = Counter(aware.partition(k, i) for k, i in stream)
    print("Aware  -> max:", max(al.values()), "min:", min(al.values()),
          "skew:", round(max(al.values())/min(al.values()), 2), "x")
    # Naive  -> max: 1107  min:  81  skew: 13.67x
    # Aware  -> max:  390  min:  81  skew:  4.81x

Same total event count, same MD5, same modulo — only the hot key's input gets a salt suffix. The bottleneck partition's load drops 4.4×.

3. The same pattern at every layer

  • Kafka — Pinterest MemQ, Twitter EventBus, Shopify webhook ingest all ship hot-key salting on the producer. Trade: per-key ordering across the whale is broken; salting is only safe when downstream aggregation is associative.
  • DynamoDB — AWS's own partition-key whitepaper recommends PK = user_id + "_" + random(0..N) for any key that exceeds 1,000 WCU. Lyft re:Invent 2019 demo; Robinhood for per-symbol order books.
  • Spark — Spark 3 AQE (spark.sql.adaptive.skewJoin.enabled) automates this for joins; pre-AQE, Netflix and Uber shipped explicit "salt UDFs" before joins on hot keys.
  • Flink — two-stage keyBy: first on (user_id, salt) for pre-aggregation, then on user_id for roll-up. Alibaba's Flink Forward 2020 talk is the canonical reference.
  • Cassandra / ScyllaDB — Discord's "How Discord Stores Trillions of Messages" bucketing: PRIMARY KEY ((channel_id, bucket), message_id).
  • Kinesis — ExplicitHashKey is the salted-shard knob; Coinbase uses it to spread BTC/ETH market data.

4. The trade-offs

CostAcceptable when…Not acceptable when…
Reads become scatter-gather (N salt buckets per hot key)writes ≫ reads (event logs, audit, metrics)reads dominate (user-profile look-ups, dim tables)
Per-key ordering breaksaggregations are associative (counters, sums, sketches)downstream is an event-sourced state machine; use a dedicated topic instead
You need a heavy-hitter detector upstreamhot-key set is small and stable (10s of keys)hot-key set churns faster than config-push; switch to consistent hashing with bounded loads

5. Interviewer probes

  • "How do you discover hot keys?" — Count-Min Sketch + Misra-Gries top-K on the producer, flushed every minute. Or an offline daily heavy-hitters job for stable cases.
  • "How do you pick the salt factor?" — salt_factor ≈ whale_rate / cold_partition_rate. Here that's 1000/125 = 8.
  • "Does this break ordering?" — Yes; alternative for order-sensitive consumers is a dedicated topic / partition / shard for the whale.
  • "Hot keys change — how do you adapt?" — Rolling detection + config push; readers must scatter-gather over the salt buckets until the retention window expires (Stripe's "salt sunset").
  • "What about consistent hashing?" — Balances keys, not traffic. Use consistent hashing with bounded loads (Google, 2017) when hot keys are unknowable in advance.
  • "Can salting work on any aggregation?" — Only associative ones. See customer_growth-0027 §4 for the full table.

Senior signal: The same algorithm — salt + partial + roll-up — shows up in Kafka, DynamoDB, Spark, Flink, Cassandra, and Kinesis. The mechanical question is which layer to salt at; the senior question is whether to salt at all (vs. consistent hashing with bounded loads, vs. a dedicated topic for the whale, vs. trusting Spark AQE). Salting trades read-side scatter-gather and broken per-key ordering for write-side balance — accept it when writes dominate and aggregations are associative; reject it when reads dominate or causal order matters.
