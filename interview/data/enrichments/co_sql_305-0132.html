Spotify keeps active and churned listeners in two separate tables, and the lifecycle team wants one unified roster tagged with each user's state. It is a classic UNION — but the interviewer is watching whether you pick UNION or UNION ALL on purpose, and whether you have thought about a user who appears in both tables.

1. The source data — active_users and inactive_users

active_users.user_id inactive_users.user_id
1 4
2 5
3 2
3

Two things are deliberate. active_users lists user 3 twice (a duplicate row). And user 2 appears in both tables — active and inactive — which is the data-quality landmine the interviewer planted.

2. The SQL — UNION

SELECT user_id, 'active' AS status FROM active_users
UNION
SELECT user_id, 'inactive' FROM inactive_users;

UNION stacks the two result sets and then removes duplicate rows across the whole combined set. The literal columns 'active' and 'inactive' tag which table each row came from.

3. The resulting analysis

user_idstatus
5inactive
4inactive
3active

1active
2active

The duplicate (3, 'active') from active_users collapsed to one row — that is UNION de-duplicating. But user 2 appears twice: once as active, once as inactive. The rows are not duplicates — the status column differs — so UNION keeps both. The roster now contradicts itself, and that is the whole interview.

4. Strategic explanations

UNION vs UNION ALL — know the cost

UNION runs a global DISTINCT over the combined output — a sort or hash across every row. UNION ALL just concatenates, no de-dup, far cheaper. Here, because each branch tags a different status literal, no row from branch A can ever equal a row from branch B, so the de-dup pass cannot remove anything across branches — it only collapsed the within-table duplicate (3, 'active'). If the source tables are already clean, UNION ALL gives the identical answer for less work:

SELECT user_id, 'active' AS status FROM active_users
UNION ALL
SELECT user_id, 'inactive' FROM inactive_users;

Default to UNION ALL and only pay for UNION when you genuinely need duplicate removal. Saying that out loud is the senior signal.

The real bug: a user in both tables

User 2 being both active and inactive is a source-data contradiction that UNION faithfully reproduces — it cannot fix what the inputs got wrong. Surface it and ask the product question: should the tables be mutually exclusive? If so, decide a precedence rule. "Active wins" is expressible cleanly:

SELECT user_id, 'active' AS status FROM active_users
UNION
SELECT user_id, 'inactive' FROM inactive_users i
WHERE i.user_id NOT IN (SELECT user_id FROM active_users);

Now user 2 resolves to a single active row. A senior answer flags the contradiction before shipping the roster.

Column matching is positional, not by name

The two SELECT lists are matched by position: column 1 to column 1, column 2 to column 2. The names 'active' / 'inactive' and even the AS status alias on only the first branch are irrelevant to the pairing — the result takes its column names from the first query. Swap the order of columns in one branch and you would silently union user_id against a status string, which is a type error or, worse, a logic error.

Type coercion across branches

Corresponding columns must have compatible types; Postgres coerces to a common type if it can. 'active' and 'inactive' are both text, fine. But if one branch produced an integer status code and the other a string, the union would fail or coerce unexpectedly. Keep the tag columns the same type on both sides.

NULL handling — NULLs are equal for UNION

If both tables held a row with a NULL user_id, UNION treats the two NULL rows as duplicates and keeps one — unlike =, set operations consider NULL = NULL true. Usually fine, but if a NULL user_id is itself a data error, filter it (WHERE user_id IS NOT NULL) rather than letting it ride into the roster.

Senior signal: default to UNION ALL and justify UNION only when real duplicate removal is needed, recognise that the status literal makes cross-branch duplicates impossible anyway, and flag the user-in-both-tables contradiction with an explicit precedence rule instead of shipping a self-contradictory roster.
