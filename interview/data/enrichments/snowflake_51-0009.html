An academic citation system needs to display co-authors in a canonical, reproducible order for each paper — numbered 1, 2, 3 … according to a declared author_order field, with author_id as a deterministic tiebreaker when two authors share the same declared position. The interviewer is probing whether you can use ROW_NUMBER() with a composite ORDER BY inside a window partition, and whether you understand why ROW_NUMBER is the right function here (vs RANK or DENSE_RANK, which can produce gaps or ties).

Paper P1 has authors A20 and A30 tied at author_order=2. Paper P3 has authors A20 and A50 both at author_order=1. These ties exercise the secondary sort on author_id — A20 < A30 alphabetically, and A20 < A50, so they get the lower sequence number.

2. The SQL — ROW_NUMBER with composite ORDER BY per partition

SELECT paper_id, author_id,
       ROW_NUMBER() OVER (
         PARTITION BY paper_id
         ORDER BY author_order, author_id
       ) AS author_seq
FROM paper_authors
ORDER BY paper_id, author_seq;

4. Strategic explanations

Why ROW_NUMBER, not RANK or DENSE_RANK

The requirement is a gapless sequence starting at 1 for each paper, with no two authors sharing the same sequence number. RANK() would assign seq=2 to both A20 and A30 in paper P1 and then skip to seq=4 for the next author — gaps in the sequence. DENSE_RANK() would assign seq=2 to both and seq=3 next — no gaps but duplicate numbers. Only ROW_NUMBER() guarantees strictly unique, gapless integers, which is exactly what a citation system needs to display "Author 1, Author 2, Author 3."

Composite ORDER BY as a deterministic tiebreaker

ORDER BY author_order, author_id is a two-level sort. The primary key is the declared order; the secondary key (author_id) breaks ties deterministically. Without the secondary sort, ROW_NUMBER would still assign unique numbers to tied rows, but the assignment would be non-deterministic — the same query could return a different ordering on repeated runs depending on the physical storage order. A deterministic tiebreaker is required in any production system where the output must be reproducible.

PARTITION BY resets the counter per group

PARTITION BY paper_id means the ROW_NUMBER counter resets to 1 for each distinct paper_id. Without the partition, ROW_NUMBER would number all rows globally across all papers (1, 2, 3, 4, 5, 6, 7), which is not the intent. This is the most common beginner mistake with window functions: forgetting the PARTITION BY when the business context is per-group.

The column contribution_score is unused

The schema includes contribution_score but the query sorts by author_order and author_id. A follow-up interviewer question is often: "What if you wanted to rank by contribution score instead?" The answer is simply swap the ORDER BY: ORDER BY contribution_score DESC, author_id. Another follow-up: "How would you also show the author's contribution as a percentage of total paper contribution?" — contribution_score / SUM(contribution_score) OVER (PARTITION BY paper_id).

Dialect notes

ROW_NUMBER() OVER (PARTITION BY … ORDER BY …) is ANSI SQL:2003 and supported identically in Snowflake, PostgreSQL, BigQuery, MySQL 8+, and SQL Server. There are no dialect differences for this specific pattern. MySQL before version 8.0 does not support window functions at all — a workaround using user-defined variables or correlated subqueries was required, but that is now historical.

Senior signal: explain why ROW_NUMBER is the only function that guarantees a unique gapless sequence (vs RANK with gaps and DENSE_RANK with duplicates); stress that the secondary author_id tiebreaker is required for query reproducibility; and offer the contribution-percentage extension (score / SUM(score) OVER (PARTITION BY paper_id)) as a natural follow-on.
