Netflix's Content Engineering org runs the morning standup looking at one chart: viewership by device. Mobile vs TV vs web vs game-console isn't a vanity slice — it dictates encoding-ladder priorities, ad-tier rollouts, and the famous "next episode" auto-play timing. The query is a one-liner; the senior conversation is what "watch_time" actually means and why the answer is hardly ever just "sum it up by device".

2. The SQL — per-device users + total minutes

SELECT device_type,
  COUNT(DISTINCT user_id) AS users,
  SUM(watch_time)         AS total_watch
FROM viewership
GROUP BY device_type;

4. Strategic explanations

Why COUNT(DISTINCT user_id) — not COUNT(*)

Without DISTINCT, a user who fires three viewership events on mobile gets counted three times — you'd see "millions of mobile users" when really it's the same handful watching repeatedly. COUNT(DISTINCT user_id) dedupes within the group; COUNT(*) counts events, which is a different (and equally valuable) metric. Production dashboards usually emit both:

SELECT device_type,
  COUNT(*)                AS sessions,
  COUNT(DISTINCT user_id) AS users,
  SUM(watch_time)         AS total_minutes,
  AVG(watch_time)         AS avg_session_minutes,
  SUM(watch_time) * 1.0 / COUNT(DISTINCT user_id) AS minutes_per_user
FROM viewership
GROUP BY device_type;
The "what's a session?" trap

Each row in viewership probably represents one playback event — not one session. A user pausing for 15 minutes and resuming creates one or two rows depending on the instrumentation. Real-world Netflix queries pre-sessionize:

WITH sessionized AS (
  SELECT user_id, device_type,
    SUM(CASE WHEN started_at - LAG(ended_at) OVER (
                  PARTITION BY user_id, device_type ORDER BY started_at)
              > INTERVAL '30 minutes' THEN 1 ELSE 0 END
    ) OVER (PARTITION BY user_id, device_type ORDER BY started_at) AS session_id
  FROM viewership
)
SELECT device_type, COUNT(DISTINCT (user_id, session_id)) AS sessions
FROM sessionized GROUP BY device_type;

The 30-minute idle threshold is the Netflix-published default; mobile uses 5 minutes (different consumption pattern).

Watch-time semantics — minutes streamed vs minutes engaged

watch_time in raw form usually means "minutes the player was emitting heartbeats", which includes:

  • Background playback after the user walked away (very common on TV)
  • Skipped intros (counted as watched if heartbeats fired)
  • The auto-play preview reel on the home page (counted by some pipelines, not by others)

The "engaged minutes" metric subtracts these — usually computed in a downstream Spark job, not in this query. Always confirm with the PM which definition the dashboard expects. Engaged minutes is typically 70-85% of raw watch_time.

Per-user vs per-device deduplication

The query above dedupes users per device — U_001 contributes 1 to tv.users AND 1 to mobile.users — total across rows is 6, but distinct users is 5. To answer "how many users total?" you need a separate query:

SELECT COUNT(DISTINCT user_id) AS total_users FROM viewership;

The double-counting is correct (a multi-device user really does appear on each device they used), but reporting must be careful about adding column totals.

The cross-device follow-up — multi-device users

The actual product question: "what fraction of users watch on more than one device?" Multi-device users are the high-engagement segment.

WITH per_user AS (
  SELECT user_id, COUNT(DISTINCT device_type) AS devices
  FROM viewership GROUP BY user_id
)
SELECT devices, COUNT(*) AS users
FROM per_user
GROUP BY devices
ORDER BY devices;

Distribution typically: 1 device = 60%, 2 = 25%, 3+ = 15%. The 3+ segment converts to ad-free at much higher rates — ARPU-relevant.

The "primary device" classification

For each multi-device user, which device is dominant? Use ROW_NUMBER:

WITH per_user_device AS (
  SELECT user_id, device_type, SUM(watch_time) AS minutes
  FROM viewership GROUP BY user_id, device_type
),
ranked AS (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY minutes DESC) AS rk
  FROM per_user_device
)
SELECT device_type AS primary_device, COUNT(*) AS users
FROM ranked WHERE rk = 1
GROUP BY device_type;

Drives the home-screen layout: the TV-primary user sees the lean-back row layout; the mobile-primary user sees vertical-card layout.

The encoding-ladder follow-up — what content gets which ABR rung

Netflix's per-title encoding budget is set by device-mix forecasts. Game consoles and TVs need 4K HDR; mobile rarely justifies anything above 1080p. The query that answers "should we encode 4K for this title?":

SELECT title_id,
  100.0 * SUM(CASE WHEN device_type IN ('tv','game_console') THEN watch_time END)
        / NULLIF(SUM(watch_time),0) AS pct_big_screen
FROM viewership v JOIN sessions s USING (session_id)
GROUP BY title_id
HAVING SUM(watch_time) > 1000000  -- only well-watched titles
ORDER BY pct_big_screen DESC;

If pct_big_screen > 50% → encode 4K HDR. Otherwise skip the budget.

The QoE join — minutes weighted by quality

Senior follow-up: "raw watch_time treats a buffering-heavy session the same as a perfect playback. Adjust for QoE."

SELECT device_type,
  SUM(watch_time)                   AS raw_minutes,
  SUM(watch_time * (1 - rebuf_pct)) AS quality_adjusted_minutes,
  AVG(rebuf_pct) AS avg_rebuffer_rate
FROM viewership v JOIN qoe q USING (session_id)
GROUP BY device_type;

Quality-adjusted minutes is the metric the SRE team is on the hook for. Diverges most on game consoles (older Wi-Fi chipsets → rebuffering).

Cohort splits the dashboard never ships without

"Per-device" is rarely the slide on its own. The real chart is per-device-by-X for some X:

  • Country — mobile dominates in IN/BR/ID; TV dominates in US/EU.
  • Plan tier — mobile-only basic plan vs ad-tier vs premium.
  • Account age — new signups skew mobile (download-first), old accounts skew TV.
  • Daypart — mobile peaks 8am commute, TV peaks 9pm.
The denominator the analyst forgets

Watching minutes ↑ can mean (a) more users, (b) longer sessions per user, or (c) more sessions per user. Without splitting the lever, you can't know why. The composite metric is:

minutes_per_dau = SUM(watch_time) / COUNT(DISTINCT user_id, watch_date)

Always show users, sessions, minutes-per-session, minutes-per-DAU on the same slide so the eye lands on the changing variable.

Senior signal: dedupe users with COUNT(DISTINCT) and emit sessions/users/minutes/per-user in one go, raise the engaged-vs-raw watch_time distinction unprompted, and pre-empt the multi-device cross-tab. The encoding-ladder application (TV+console → 4K) and the QoE-adjusted minutes connect the query to actual Netflix decisions — that's the over-the-bar finish.
