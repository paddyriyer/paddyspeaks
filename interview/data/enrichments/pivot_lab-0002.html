A product analyst pulls a per-user device summary: one row per user, three columns counting sessions on mobile, desktop, and tablet. To compare engagement across device types — group, filter, rank by device — that wide shape is useless; they need one row per user-per-device. This is an unpivot, and the candidate already knows the stacked-UNION ALL trick. The interviewer is now probing the next level: can you unpivot without scanning the table once per column?

User 1's tablet count is a real 0 (they own a tablet, used the app zero times). User 3's desktop count is NULL (no desktop telemetry at all). The reshape must keep those apart, and it must emit a row for every user-device pair regardless — nine rows out, no matter the values.

2. The SQL — unpivot in one scan with CROSS JOIN LATERAL

SELECT m.user_id, v.device, v.sessions
FROM device_metrics m
CROSS JOIN LATERAL (VALUES
  ('mobile', m.mobile_sessions),
  ('desktop', m.desktop_sessions),
  ('tablet', m.tablet_sessions)
) AS v(device, sessions)
ORDER BY m.user_id, v.device;

For each row m, the LATERAL subquery builds a tiny three-row inline table: a literal device label paired with the matching column from that same row. CROSS JOIN then attaches all three rows to the user. One outer scan, three rows manufactured per user — the table is read exactly once.

Three rows became nine — 3 users × 3 devices. User 1's tablet row carries 0 and user 3's desktop row carries NULL: CROSS JOIN emits a row for every pair unconditionally, so neither a zero nor a NULL suppresses its row. The row count is fixed by the schema (users × named device columns), not by the data — a desirable property for a reshape, since the output shape is then predictable regardless of which sessions happen to be zero.

4. Strategic explanations

One scan vs four — the whole point of LATERAL

The stacked-UNION ALL unpivot names the source table once per column; with three device columns the planner does three full scans and stitches the pieces. The LATERAL VALUES form names device_metrics exactly once and manufactures the extra rows in memory from each scanned row. On a 5-row demo identical; on a billion-row events table the difference is one I/O pass versus three. That scan-count argument is the senior reason to reach for LATERAL.

What LATERAL actually unlocks

A normal subquery in the FROM clause is evaluated once, independent of the outer query — it cannot see m.mobile_sessions. LATERAL lifts that restriction: the subquery is re-evaluated for each outer row and may reference that row's columns. Here it lets the inline VALUES list pull m.mobile_sessions, m.desktop_sessions, m.tablet_sessions — the correlation is exactly what makes the per-row expansion work.

CROSS JOIN keeps every row — including zeros and NULLs

CROSS JOIN pairs each outer row with every row the LATERAL block produces, with no ON predicate to satisfy — so all nine rows survive, and a 0 or NULL in sessions never causes a row to disappear. That matters: if you later need only active devices you filter explicitly with WHERE v.sessions > 0 — and note > 0 also discards the NULLs, so decide whether "no telemetry" should be excluded too. The reshape stays lossless; filtering is a separate, deliberate step.

Type alignment inside the VALUES list

Every tuple in the VALUES list must agree column-by-column: first slot text, second slot integer. If one device column were NUMERIC and another INT, Postgres resolves a common type for the sessions column — usually fine, but mixing genuinely incompatible types (a text column with an int column in the same slot) is an error. When unpivoting columns of differing types, cast them to a shared type inside the VALUES list so the result column has one well-defined type.

Dialect — LATERAL, CROSS APPLY, and native UNPIVOT

LATERAL is standard SQL and works in PostgreSQL and MySQL 8+. SQL Server spells the same idea CROSS APPLY over a VALUES table. Oracle, SQL Server, and Snowflake also offer the dedicated UNPIVOT operator, which is the most readable single-scan option where available. SQLite has neither LATERAL nor UNPIVOT, so there the stacked UNION ALL remains the only route — accepting its repeated scans.

Senior signal: know that stacked UNION ALL re-scans the table once per column, and unpivot in a single scan with CROSS JOIN LATERAL (VALUES …) — explaining that LATERAL correlates the inline table to each outer row, that CROSS JOIN keeps every pair so zeros and NULLs are preserved, and that the equivalents are CROSS APPLY in SQL Server and the native UNPIVOT operator elsewhere.
