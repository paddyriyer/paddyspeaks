You need the IDs common to two datasets — say, customers who appear in both a marketing list and a purchase log. The reference solution uses the INTERSECT set operator. It is a one-liner, but the interviewer is probing whether you know how INTERSECT treats duplicates and NULLs, because both differ from a naive JOIN.

Three rows. Note what is not there: 1 and 2 (A only), 5 and 6 (B only). And note the two surprises — the 4 that was duplicated in table_a collapses to a single row, and NULL survives even though NULL = NULL is never true.

4. Strategic explanations

INTERSECT de-duplicates — it is set semantics

Plain INTERSECT (without ALL) returns distinct values present in both inputs. Even though 4 appears twice in table_a, the result has one 4. This is the defining difference from a join: an inner join on id would emit a row for every matching pair, multiplying duplicates. If you need that multiplicity preserved, use INTERSECT ALL, which returns min(count_in_a, count_in_b) copies of each value.

The NULL surprise — INTERSECT uses NOT-DISTINCT, not equality

This is the trap. Set operators compare rows with is-not-distinct-from semantics, not =. Under that comparison, NULL is considered equal to NULL. So because both tables contain a NULL id, the result contains a NULL row. A join written ON a.id = b.id would do the opposite — NULL = NULL is unknown, so NULLs never match and are dropped. Same intent, opposite NULL outcome. Knowing this divergence cold is the senior signal.

INTERSECT vs INNER JOIN — pick deliberately

Use INTERSECT when you want a clean set of common values, deduplicated, NULLs included. Use a join when you need columns from both sides, or when row multiplicity matters. A common rewrite is WHERE id IN (SELECT id FROM table_b) — but beware: IN with a subquery does not match NULL to NULL, so it behaves like the join, not like INTERSECT. Three constructs, two different NULL behaviours.

Column count, order, and type must line up

Set operators match columns positionally, not by name, and both SELECTs must have the same number of columns with compatible types. The output column name is taken from the first query. If the two sides had mismatched column counts the statement fails to parse — a frequent slip when intersecting wider rows.

Dialect note

INTERSECT is standard SQL and supported by PostgreSQL, SQL Server, Oracle, SQLite, and modern MySQL (8.0.31+). Older MySQL lacked it — the portable fallback there is SELECT DISTINCT a.id FROM table_a a JOIN table_b b ON a.id = b.id, which silently drops the NULL match. INTERSECT ALL is not universally available (SQL Server omits it).

Senior signal: state that INTERSECT returns distinct values and uses is-not-distinct-from comparison, so duplicates collapse and a NULL present in both inputs survives — the exact opposite of an ON a.id = b.id join. Reach for INTERSECT ALL only when multiplicity must be preserved.
