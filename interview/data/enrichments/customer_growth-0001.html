A retention team at a subscription-commerce company runs the quarterly RFM review off a fixed snapshot date of 2025-04-01. The ask is not just "who are the best customers" but which customers moved between tiers year-over-year — an interviewer is probing whether you can compute two independent RFM scorings (rolling-12 vs prior-12), compare them, and surface the "fell" cohort first because those are the accounts at risk of churning. This is a multi-CTE problem stacking conditional aggregation, PERCENT_RANK, and a self-comparison.

2. The SQL — dual RFM scoring with PERCENT_RANK and a tier self-comparison

WITH params AS (SELECT DATE '2025-04-01' AS snapshot),
paid AS (
    SELECT o.customer_id, o.order_date, o.amount,
           (p.snapshot - o.order_date) AS days_back
    FROM orders o, params p
    WHERE o.status = 'paid'
),
agg AS (
    SELECT customer_id,
        MIN(CASE WHEN days_back < 365 THEN days_back END) AS recency_days,
        SUM(CASE WHEN days_back < 365 THEN 1 ELSE 0 END) AS y1_orders,
        ROUND(SUM(CASE WHEN days_back < 365 THEN amount ELSE 0 END), 2) AS y1_spend,
        SUM(CASE WHEN days_back >= 365 AND days_back < 730 THEN 1 ELSE 0 END) AS y2_orders,
        ROUND(SUM(CASE WHEN days_back >= 365 AND days_back < 730 THEN amount ELSE 0 END), 2) AS y2_spend,
        SUM(CASE WHEN days_back < 7  THEN 1 ELSE 0 END) AS l7_orders,
        SUM(CASE WHEN days_back < 28 THEN 1 ELSE 0 END) AS l28_orders,
        SUM(CASE WHEN days_back < 90 THEN 1 ELSE 0 END) AS l90_orders
    FROM paid GROUP BY customer_id
    HAVING COUNT(*) > 1
),
scored_now AS (
    SELECT agg.*,
        PERCENT_RANK() OVER (ORDER BY recency_days DESC) AS r_pct,
        PERCENT_RANK() OVER (ORDER BY y1_orders ASC)    AS f_pct,
        PERCENT_RANK() OVER (ORDER BY y1_spend ASC)     AS m_pct
    FROM agg WHERE y1_orders > 0
),
scored_prior AS (
    SELECT customer_id,
        PERCENT_RANK() OVER (ORDER BY y2_orders ASC) AS f2_pct,
        PERCENT_RANK() OVER (ORDER BY y2_spend ASC)  AS m2_pct
    FROM agg WHERE y2_orders > 0
),
labelled AS (
    SELECT n.*,
        ROUND((n.r_pct + n.f_pct + n.m_pct)::numeric / 3.0, 3) AS rfm_pct,
        CASE WHEN (n.r_pct + n.f_pct + n.m_pct)/3.0 >= 0.90 THEN 'Tier 1'
             WHEN (n.r_pct + n.f_pct + n.m_pct)/3.0 >= 0.75 THEN 'Tier 2'
             ELSE 'Tier 3' END AS current_tier,
        CASE WHEN p.customer_id IS NULL THEN 'new_repeat'
             WHEN (p.f2_pct + p.m2_pct)/2.0 >= 0.75 THEN 'Tier 1'
             WHEN (p.f2_pct + p.m2_pct)/2.0 >= 0.50 THEN 'Tier 2'
             ELSE 'Tier 3' END AS prior_tier
    FROM scored_now n
    LEFT JOIN scored_prior p USING (customer_id)
)
SELECT customer_id, l7_orders, l28_orders, l90_orders,
    y1_orders, y1_spend, y2_orders, y2_spend,
    CASE WHEN y2_spend > 0 THEN ROUND(100.0*(y1_spend - y2_spend)/y2_spend, 1) END AS yoy_growth_pct,
    rfm_pct, current_tier, prior_tier,
    CASE WHEN prior_tier = 'new_repeat'      THEN 'new_repeat'
         WHEN current_tier = prior_tier      THEN 'stable'
         WHEN current_tier > prior_tier      THEN 'fell'
         ELSE 'climbed' END AS tier_shift
FROM labelled
ORDER BY CASE WHEN current_tier > prior_tier AND prior_tier <> 'new_repeat' THEN 0 ELSE 1 END,
         yoy_growth_pct ASC NULLS LAST;

4. Strategic explanations

Why PERCENT_RANK and not NTILE for RFM

PERCENT_RANK() returns a value in [0,1] giving each row's relative standing: (rank - 1) / (rows - 1). Averaging three percent-ranks (recency, frequency, monetary) produces a smooth composite score, and the 0.90 / 0.75 cutoffs map cleanly onto it. NTILE(n) would force buckets of equal size regardless of where the natural breaks are — fine for "give me deciles", wrong when the business defines a tier by a percentile threshold. Note the direction flips: recency uses DESC (fewer days back = better) while frequency and monetary use ASC (more is better), so a high percent-rank always means "good".

The two scorings are intentionally on different scales

current_tier averages three signals (r+f+m) and cuts at 0.90/0.75. prior_tier only has two signals available (f+m — you cannot compute a meaningful Y2 recency from a snapshot) and cuts at 0.75/0.50. A candidate who assumes both tiers use identical thresholds will mis-explain the result. This asymmetry is a common real-world wrinkle: historical comparisons rarely have the same feature set as the live score.

The NULLS LAST trap on the final sort

Customer 4 has yoy_growth_pct = NULL because it has no Y2 spend to divide by. In Postgres a plain ORDER BY yoy_growth_pct ASC sorts NULLs last by default, but that is the opposite of MySQL and SQL Server, where NULLs sort first. Spelling out NULLS LAST makes the intent portable and explicit — you do not want a "we cannot compute growth" customer jumping to the top of an urgency-ranked list.

Why HAVING COUNT(*) > 1 must run before scoring

The question scopes to repeat customers. Filtering single-order customers in the agg CTE (before PERCENT_RANK) is essential: percentile ranks are computed over the surviving population, so leaving one-time buyers in would shift every percentile and distort the tier cutoffs. Customer 5 (one paid order) and the refund-only path for customer 6 both demonstrate why row-level filtering precedes the window functions.

What a senior would say next

"The current_tier > prior_tier string comparison works only because the tier labels sort lexically the same way they rank ('Tier 1' < 'Tier 2' < 'Tier 3'). That is fragile — rename a tier to 'Platinum' and the comparison silently breaks. I would carry a numeric tier_rank alongside the label and compare on that. I'd also confirm the snapshot is a parameter, not a literal, so the same query backfills any quarter."

Senior signal: recognise this as two independent RFM scorings compared against each other, justify PERCENT_RANK over NTILE for percentile-threshold tiers, explain why the prior scoring legitimately uses fewer features and different cutoffs, handle the NULL growth row with explicit NULLS LAST, and flag the fragile string-based tier comparison with a numeric-rank fix.
