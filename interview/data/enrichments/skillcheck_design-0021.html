A restaurant reservation system is the canonical example for Postgres exclusion constraints: traditional UNIQUE constraints operate on equality, but a double-booking is two rows whose time intervals overlap, not two rows that are equal. The interviewer is probing whether you know that overlap-prevention requires the btree_gist extension, a TSTZRANGE column, and an EXCLUDE USING gist constraint — and crucially, whether you know to scope the exclusion to only active reservations via a WHERE predicate, so canceled reservations do not block re-booking the same slot.

1. The requirements — what the design must capture

  • Restaurant entity: includes timezone (IANA format) because reservation times must be interpreted in the restaurant's local timezone; overlapping reservations on the same physical table must be prevented at the database layer.
  • Physical table resource: the thing being booked is a specific physical table (a "4-top by window"), identified within a restaurant; two parties cannot be seated at the same table at the same time.
  • Half-open range: TSTZRANGE with [start, end) semantics models the reservation window; a party leaving at 21:00 does not conflict with a party arriving at 21:00.
  • Conditional exclusion: only confirmed and seated reservations are active; canceled and no_show reservations must not participate in the exclusion check so their slots can be re-booked.
  • Status lifecycle: reservations progress through confirmed → seated → completed; they can be canceled or marked no-show at any point before seating.

2. The DDL — the CREATE TABLE

CREATE EXTENSION IF NOT EXISTS btree_gist;

CREATE TABLE restaurant (
  restaurant_id    BIGSERIAL    PRIMARY KEY,
  name             VARCHAR(256) NOT NULL,
  timezone         VARCHAR(64)  NOT NULL,                  -- IANA tz, e.g., 'America/New_York'
  city             VARCHAR(128),
  created_at       TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE TABLE table_resource (
  table_id         BIGSERIAL    PRIMARY KEY,
  restaurant_id    BIGINT       NOT NULL REFERENCES restaurant(restaurant_id) ON DELETE CASCADE,
  label            VARCHAR(32)  NOT NULL,                  -- '4-top by window'
  seats            SMALLINT     NOT NULL CHECK (seats > 0),
  is_active        BOOLEAN      NOT NULL DEFAULT TRUE,
  UNIQUE (restaurant_id, label)
);
CREATE INDEX ix_table_resource_restaurant ON table_resource (restaurant_id);

CREATE TABLE reservation (
  reservation_id   BIGSERIAL    PRIMARY KEY,
  table_id         BIGINT       NOT NULL REFERENCES table_resource(table_id),
  party_id         BIGINT       NOT NULL,                  -- FK to guest/party table omitted
  party_size       SMALLINT     NOT NULL CHECK (party_size > 0),
  reservation_range TSTZRANGE   NOT NULL,                  -- [start_ts, end_ts) inclusive-exclusive
  status           VARCHAR(16)  NOT NULL DEFAULT 'confirmed',  -- 'confirmed','seated','canceled','no_show'
  created_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),
  canceled_at      TIMESTAMPTZ,
  -- Prevent overlapping confirmed reservations on the same physical table:
  EXCLUDE USING gist (
    table_id WITH =,
    reservation_range WITH &&
  ) WHERE (status IN ('confirmed','seated'))
);
CREATE INDEX ix_reservation_table_range ON reservation USING gist (table_id, reservation_range);
CREATE INDEX ix_reservation_party ON reservation (party_id);

-- Insertion path: the database refuses any reservation whose range overlaps an existing
-- confirmed/seated reservation on the same table; the loser gets a unique-violation that the
-- application maps to HTTP 409 Conflict.

4. Strategic explanations

Why btree_gist is required

Postgres's built-in GiST operator classes handle range types natively (TSTZRANGE && TSTZRANGE is a GiST-supported operation). However, the exclusion constraint needs to combine a scalar column (table_id WITH =) with a range column (reservation_range WITH &&) in a single GiST index. The scalar equality operator for integers is not in the default GiST operator class — btree_gist adds it. Without this extension the EXCLUDE USING gist statement fails with "operator class ... not found for access method gist."

The WHERE predicate on the exclusion constraint

This is the most important subtlety. Without WHERE (status IN ('confirmed','seated')), canceling a reservation and re-booking the same slot would fail because the canceled row would still participate in the overlap check. The partial exclusion constraint is the database equivalent of a partial index: it is smaller, faster, and semantically correct. When a reservation is canceled, the row remains in the table (for audit purposes) but its status change removes it from the exclusion predicate, freeing the slot for re-booking.

Application response to exclusion violation

A concurrent double-booking attempt raises a PostgreSQL exclusion violation (SQLSTATE 23P01, distinct from a unique violation at 23505). The application must catch this error and return HTTP 409 Conflict to the client, along with a "no availability" message. This is strict serializability for the resource acquisition pattern — no advisory locks, no SELECT-FOR-UPDATE check-then-insert race; the database enforces it atomically.

GiST index duality

The EXCLUDE USING gist constraint implicitly creates a GiST index that also serves availability queries: SELECT * FROM reservation WHERE table_id = ? AND reservation_range && tstzrange('2025-05-16 19:00', '2025-05-16 21:00', '[)') is an index scan on the same GiST structure. The additional explicit GiST index (ix_reservation_table_range) is shown for documentation clarity; in practice Postgres may merge these.

Dialect notes

TSTZRANGE and EXCLUDE USING gist are Postgres-specific. In MySQL and SQL Server, there are no range types or exclusion constraints; you must enforce non-overlap with a check-then-insert stored procedure under explicit locking, or with a serializable transaction. In CockroachDB, range types are supported as of v22.2 but exclusion constraints are not yet available; the recommended workaround is an advisory lock or an application-level serialized booking service.

Senior signal: know that preventing interval overlap requires EXCLUDE USING gist with btree_gist — not a UNIQUE constraint — and that the WHERE (status IN ('confirmed','seated')) predicate is not optional: without it, every canceled reservation permanently blocks its time slot, making the system unusable.
