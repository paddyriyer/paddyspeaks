LinkedIn's job-marketplace team has a data-quality alarm: the same employer keeps posting what looks like the same role over and over, inflating search results and annoying job-seekers. Your task is to surface those duplicate listings. The hard part is not the SQL — it is agreeing on what "duplicate" even means, because the obvious definition catches legitimate reposts too.

2. The SQL — GROUP BY + HAVING COUNT > 1

SELECT title, company_id, COUNT(*) AS dups FROM jobs
GROUP BY title, company_id HAVING COUNT(*) > 1;

Group rows by the (title, company) pair, count how many listings fall into each group, and keep only the groups with more than one. This is the canonical "find duplicates" shape.

4. Strategic explanations

"Duplicate" is a product decision, not a SQL fact

Grouping on (title, company_id) assumes any two identical titles from one company are duplicates. That over-reports: a company genuinely hiring two engineers, or reposting a role a quarter later after the first hire fell through, is not spam. Real definitions usually add a time window:

-- duplicates only if posted within 14 days of each other
SELECT a.job_id, b.job_id, a.title, a.company_id
FROM jobs a JOIN jobs b
  ON a.company_id = b.company_id
 AND a.title = b.title
 AND a.job_id < b.job_id
 AND b.posted_at - a.posted_at <= 14;

With a 14-day window, company 10's January pair (4 days apart) is a duplicate but the February posting is not. Surfacing that "duplicate depends on a time threshold" is the senior instinct — the bare GROUP BY would flag a legitimate repost.

Titles are dirty — exact-match misses near-duplicates

"Software Engineer", "Software Engineer ", "Sr. Software Engineer", "Software Engineer (Remote)" all hash to different groups under GROUP BY title, so genuine duplicates slip through. Production de-duplication normalises first: GROUP BY LOWER(TRIM(title)), company_id at minimum, and often a fuzzy match (similarity() from pg_trgm) or a hash of normalised text. Mention that exact string equality is a fragile key for human-entered data.

Finding the rows to delete, not just the counts

The interviewer's natural follow-up: "now keep the earliest posting and flag the rest for removal." That is a window function ranking each posting within its group:

SELECT job_id, title, company_id, posted_at
FROM (
  SELECT *,
    ROW_NUMBER() OVER (PARTITION BY title, company_id
                       ORDER BY posted_at) AS rn
  FROM jobs
) t
WHERE rn > 1;   -- everything except the first posting per group

ROW_NUMBER is the right tool here precisely because we want exactly one survivor per group — ties broken arbitrarily are fine, since both copies are duplicates anyway.

NULLs in the grouping key

If some rows have a NULL title or NULL company_id, GROUP BY collapses all NULLs into a single group (SQL treats NULLs as equal for grouping, unlike =). So two listings with missing titles get reported as duplicates of each other — possibly noise. Decide whether NULL-keyed rows should be excluded with a WHERE title IS NOT NULL guard.

Senior signal: treat "duplicate" as a definition to negotiate — raise the time-window and title-normalisation questions before the bare GROUP BY ... HAVING COUNT(*) > 1 ships, and know that finding the actual rows to delete needs ROW_NUMBER over a partition, not the group counts.
