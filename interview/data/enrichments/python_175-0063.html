Google uses "Evaluate Reverse Polish Notation" to test the canonical stack application: postfix expressions need no parentheses and no precedence rules, so a single left-to-right pass with a stack evaluates them. The interesting wrinkles are integer division semantics and operand order — the two places candidates quietly get it wrong.

1. The input data

tokens = ["2", "1", "+", "3", "*"]

This is infix (2 + 1) * 3. A second case worth tracing for division: ["10", "6", "9", "3", "+", "-11", "*", "/", "*", "17", "+", "5", "+"].

2. The approach

Iterate the tokens. A numeric token is pushed onto stack st as an int. An operator token pops the top two operands — and order matters: b = st.pop() comes off first (the right operand), a = st.pop() second (the left operand), because the right operand was pushed most recently.

For + - * the code builds the expression string and uses eval. For / it computes int(a / b) — true division then truncation, which rounds toward zero (so -7 / 2 becomes -3, not -4). The result is pushed back. After all tokens, the single remaining value st[0] is the answer — a well-formed RPN expression always reduces to exactly one.

3. The output

tokenactionstack after
2push 2[2]
1push 1[2, 1]
+b=1, a=2 → push 3[3]
3push 3[3, 3]
*b=3, a=3 → push 9[9]
 
evalRPN(["2","1","+","3","*"])  ->  9

4. Strategic explanations

Complexity

O(n) time — one pass, each token does O(1) stack work. O(n) space for the stack in the worst case (a long run of operands before any operator). Optimal; there is no sub-linear answer.

The idiomatic alternative — an operator dispatch table
import operator
OPS = {'+': operator.add, '-': operator.sub, '*': operator.mul,
       '/': lambda a, b: int(a / b)}
def evalRPN(tokens):
    st = []
    for t in tokens:
        if t in OPS:
            b, a = st.pop(), st.pop()
            st.append(OPS[t](a, b))
        else:
            st.append(int(t))
    return st[0]

This drops eval entirely — a real senior would never ship eval on tokenized input, even "trusted" input, because it is a code-execution surface. The dispatch table is faster and safe.

The edge case — operand order and negative truncation

Subtraction and division are non-commutative: swapping a and b computes b - a — a silent sign error. And Python's // floors (-7 // 2 == -4), whereas the problem demands truncation toward zero (-3); that is exactly why the code uses int(a / b) rather than a // b. The detail that "-11" parses fine as a negative operand via int() while "-" is the operator also matters.

The follow-up — validate, or convert infix to RPN

"What if the expression is malformed?" Then st.pop() raises IndexError on too few operands, and len(st) != 1 at the end signals too many — a senior adds those guards. The deeper follow-up is the shunting-yard algorithm: convert infix to RPN, which requires real precedence handling.

The trap — t.isdigit() to detect numbers

Branching on t.isdigit() instead of t in '+-*/' mis-classifies negative numbers: "-11".isdigit() is False, so -11 would be treated as an operator and crash. Test for the operator set explicitly — the operator alphabet is tiny and known, the operand space is not.

Senior signal: get the pop order right (right operand first), use truncating division deliberately for negatives, and replace eval with an operator dispatch table — shipping eval on input is the kind of thing that fails a security-conscious interviewer.
