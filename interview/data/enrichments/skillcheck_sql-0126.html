A pricing table stores the same product's price across three stores as three side-by-side columns. Downstream analytics — charts, joins, aggregations — want the data in long format: one row per (product, store, price). That rotation of columns into rows is an unpivot, and the interviewer is checking whether you can do it portably and whether you correctly drop the NULLs that mean "not sold here".

1. The source data — products

Two edge cases drive the interview. Product 4 has all-NULL prices — after unpivoting it must produce zero rows, not three NULL rows. Product 3 has the same price (5) in two stores — both legs are real data and must both survive; this is not a duplicate to dedupe.

2. The SQL — unpivot via UNION ALL, one branch per column

SELECT product_id, 'store1' AS store, store1 AS price FROM products WHERE store1 IS NOT NULL
UNION ALL
SELECT product_id, 'store2',        store2        FROM products WHERE store2 IS NOT NULL
UNION ALL
SELECT product_id, 'store3',        store3        FROM products WHERE store3 IS NOT NULL
ORDER BY product_id, store;

Each branch peels off one store column, labels it with a literal store name, and — via its own WHERE … IS NOT NULL — emits a row only where that store actually has a price. UNION ALL stacks the three branches into the long-format result.

Product 4 produces no rows at all — every branch's IS NOT NULL filter rejects its three NULLs. Product 3 contributes two rows even though both prices are 5, because each is a legitimate (store, price) fact.

4. Strategic explanations

Why this is an unpivot, and why UNION ALL

A pivot rotates row values into columns; an unpivot does the reverse — it turns the column headers store1/store2/store3 into values of a new store column. The portable way is one SELECT per source column, stacked with UNION ALL. It must be UNION ALL, never UNION: UNION deduplicates, and product 3's two rows (3,'store1',5) and (3,'store3',5) are distinct anyway — but if the spec only projected (product_id, price), UNION would silently merge two real prices into one. Default to UNION ALL and only dedupe deliberately.

The NULL filter belongs in each branch's WHERE

The spec says skip rows where price is NULL. Each branch carries its own WHERE storeN IS NOT NULL so a "not sold here" cell never becomes an output row — that is why product 4 disappears and product 1 yields two rows, not three. Filtering after the UNION ALL with one outer WHERE price IS NOT NULL also works and is equivalent, but per-branch filters let each branch be index-friendly and make the intent local to each column.

Column names and types must line up across branches

UNION ALL matches columns by position, and the result's column names come from the first branch — hence AS store and AS price appear only in branch one. The 'store1'/'store2'/'store3' literals must all be the same text type, and store1/2/3 are all NUMERIC, so the stacked types are consistent. A type clash between branches (say one column int, another text) would either fail or force a surprising implicit cast.

Dialect — native UNPIVOT and the modern alternatives

The question is tagged "Recursive CTEs / Pivoting", but no recursion is needed — recognise the unpivot for what it is. SQL Server and Oracle offer a native UNPIVOT operator; Postgres does not, so the UNION ALL idiom is the portable answer. Postgres can also unpivot more compactly with a lateral VALUES: SELECT product_id, v.store, v.price FROM products CROSS JOIN LATERAL (VALUES ('store1',store1),('store2',store2),('store3',store3)) AS v(store,price) WHERE v.price IS NOT NULL — one pass over products instead of three. Offering that is the senior signal.

Senior signal: name the operation an unpivot, implement it portably with one UNION ALL branch per column (never UNION — it would dedupe real prices), filter IS NOT NULL so "not sold here" cells vanish, and know the compact CROSS JOIN LATERAL (VALUES …) rewrite that scans the table once.
