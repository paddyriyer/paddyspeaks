A film database stores one row per (actor, director, movie) collaboration. Casting analysts want the recurring partnerships — actor/director pairs that have made at least three distinct movies together. The interviewer is probing one specific thing: do you reach for COUNT(DISTINCT …) rather than a plain COUNT(*), because the source table can contain duplicate rows for the same movie.

1. The source data — actor_director

actor_iddirector_idmovie_id(narrative)
1110, 11, 123 distinct movies → qualifies
1220, 21only 2 distinct movies → below threshold
2130, 31, 32, 334 distinct movies → qualifies
3340, 41, 423 distinct movies…
3342…but movie 42 is listed TWICE — duplicate row

The edge case is the last pair. Actor 3 / Director 3 has four rows but only three distinct movies — movie 42 appears twice. COUNT(*) would read 4 and qualify; COUNT(DISTINCT movie_id) reads 3 and qualifies for the right reason. Actor 1 / Director 2 (only 2 movies) tests the >= 3 boundary from below.

2. The SQL — GROUP BY the pair, HAVING on distinct movie count

SELECT actor_id, director_id
FROM actor_director
GROUP BY actor_id, director_id
HAVING COUNT(DISTINCT movie_id) >= 3
ORDER BY actor_id, director_id;

GROUP BY actor_id, director_id forms one group per partnership. COUNT(DISTINCT movie_id) counts the unique movies inside each group, and HAVING keeps only the groups whose count reaches 3.

3. The resulting analysis

actor_iddirector_idwhy it qualifies
11movies 10, 11, 12 — 3 distinct
21movies 30–33 — 4 distinct
33movies 40, 41, 42 — 3 distinct (the duplicate 42 counts once)

Actor 1 / Director 2 is correctly absent — two distinct movies, below the threshold. Actor 3 / Director 3 qualifies on 3, not 4: COUNT(DISTINCT movie_id) deduplicates movie 42 before counting. A COUNT(*) would have admitted it on a phantom fourth movie.

4. Strategic explanations

COUNT(DISTINCT) vs COUNT(*) — the question hinges on it

COUNT(*) counts rows; COUNT(DISTINCT movie_id) counts unique values. They diverge exactly when the table has duplicate (actor, director, movie) rows — which the fixture deliberately includes. The business question says "distinct movies", so the metric must dedupe. Whenever a count is supposed to mean "how many different things", reach for DISTINCT unless you can prove the source is already unique on that column.

HAVING vs WHERE — filter the right object

WHERE filters rows before grouping; HAVING filters groups after aggregation. A per-group condition like "count ≥ 3" cannot go in WHERE — the count does not exist until rows are grouped. HAVING COUNT(DISTINCT movie_id) >= 3 is the correct place. (If you also needed, say, only released movies, that row-level test would go in WHERE — the two clauses are complementary, not interchangeable.)

Composite grouping key — the pair, not either column alone

GROUP BY actor_id, director_id groups on the combination. Grouping by actor_id alone would merge an actor's work across all directors and miscount the partnership; grouping by director_id alone has the mirror flaw. The unit of analysis is the actor–director pair, so both columns form the key — and both are correctly projected in SELECT because both appear in GROUP BY.

What a senior says next — dedupe upstream, or accept the cost

The duplicate row for movie 42 is a data-quality smell. COUNT(DISTINCT) defends the query against it, but a senior also flags that the table probably should have a unique constraint on (actor_id, director_id, movie_id), and notes the trade-off: COUNT(DISTINCT) forces a per-group sort/hash and is measurably heavier than COUNT(*) on large tables. If the table were guaranteed unique by constraint, plain COUNT(*) would be both correct and faster — naming that conditional optimisation is the senior signal.

Senior signal: read "distinct movies" as a literal instruction to use COUNT(DISTINCT movie_id) so duplicate rows do not inflate the count, put the per-group threshold in HAVING (not WHERE), group on the composite (actor, director) key, and flag that a unique constraint upstream would make a cheaper COUNT(*) safe.
