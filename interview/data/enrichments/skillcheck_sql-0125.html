A football league stores one row per match with host and guest goals. The standings need total points per team — 3 for a win, 1 for a draw, 0 for a loss — where a team earns points whether it played at home or away. The interviewer is probing whether you can normalise a wide match row into two team-legs (one per side) and aggregate cleanly, instead of writing a tangle of CASE branches that double-count.

1. The source data — match

The edge cases: team 40 plays exactly one match (a draw — a single-match, never-won team that must still appear with 1 point), and team 20 loses every game (0 points, but it did play, so it must appear — not be dropped). Teams that never appear in any match must be excluded entirely.

2. The SQL — split each match into two legs, then aggregate

WITH legs AS (
  SELECT host_team AS team,
         CASE WHEN host_goals > guest_goals THEN 3
              WHEN host_goals = guest_goals THEN 1
              ELSE 0 END AS pts
  FROM match
  UNION ALL
  SELECT guest_team,
         CASE WHEN guest_goals > host_goals THEN 3
              WHEN host_goals = guest_goals THEN 1
              ELSE 0 END
  FROM match
)
SELECT team AS team_id, SUM(pts)::int AS points
FROM legs
GROUP BY team
ORDER BY points DESC, team_id;

The first SELECT emits the host's leg of every match; the second emits the guest's leg. UNION ALL stacks them into one row-per-team-per-match table, and a plain GROUP BY team sums each team's points.

Team 40 shows up with 1 point despite never winning, and team 20 shows up with 0 — both played, so both belong in the standings. No team absent from match appears, because legs only ever contains teams that took the field.

4. Strategic explanations

Why UNION ALL to unpivot the two sides

A match row mixes two teams' outcomes in one record. The clean move is to normalise it: produce one leg per participant, so each team-match becomes its own row. UNION ALL — not UNION — is essential: UNION would deduplicate, and two genuinely distinct legs that happen to share (team, pts) (e.g. a team drawing twice for 1 point each) would be wrongly collapsed, undercounting. After the split, the points logic is a trivial SUM over one tidy column.

The CASE must be written from each side's own perspective

In the host leg, a win is host_goals > guest_goals; in the guest leg it is guest_goals > host_goals — the inequality flips. The draw branch host_goals = guest_goals is symmetric, so it reads the same in both. The single most common bug here is copying the host's > into the guest leg, which awards the guest points for the host's win. Each leg's CASE is written from that team's own viewpoint.

Why never-played teams are correctly absent

The question says exclude teams that never played. This falls out for free: legs is built purely from host_team and guest_team values, so a team that appears in no match contributes no leg and no group. There is no separate teams table being LEFT JOINed in — if there were, you would need an INNER JOIN or an EXISTS filter to drop the non-players. Recognise that the data model already enforces the rule.

The label says "Window Functions" — but none are needed

The question is tagged window-functions, yet the natural solution is an unpivot plus GROUP BY — no OVER() anywhere. Do not force a window function to match the tag; solve the problem as posed. (A window function would enter if the spec also asked for each team's rank in the standings — RANK() OVER (ORDER BY SUM(pts) DESC) — but plain total points need only aggregation.) Reading past a misleading tag is itself a senior signal.

Senior signal: normalise a two-sided match row into one leg per team with UNION ALL (never UNION — it would dedupe equal legs), write each leg's win/draw/loss CASE from that team's own perspective so the inequality flips correctly, rely on the unpivot to exclude never-played teams for free, and don't bolt on a window function just because the tag says so.
