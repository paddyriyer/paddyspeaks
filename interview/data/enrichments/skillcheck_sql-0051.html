A social network builds a "People You May Know" panel: for user 1, surface friends-of-friends — people exactly two hops away, not already friends — ranked by how many mutual connections they share. The friendship table stores undirected edges once. The interviewer is probing graph traversal in SQL: symmetrising the edge list, doing a two-hop self-join, and excluding the user and their direct friends.

The standout edge case is user 5: reachable through friends 2, 3, and 4 — three distinct mutual connections, so it should top the ranking. User 9 is a trap: it sits three hops from user 1 (via 5) and must not appear. Direct friends 2, 3, 4 must also be excluded even though the join would otherwise reach them.

2. The SQL — symmetrise edges, two-hop join, exclude self & friends

WITH edges AS (
  SELECT a AS u, b AS v FROM friendship
  UNION ALL
  SELECT b, a FROM friendship
),
direct AS (
  SELECT v FROM edges WHERE u = 1
)
SELECT e2.v AS user_id, COUNT(DISTINCT e1.v) AS mutual_count
FROM edges e1
JOIN edges e2 ON e1.v = e2.u
WHERE e1.u = 1
  AND e2.v <> 1
  AND e2.v NOT IN (SELECT v FROM direct)
GROUP BY e2.v
ORDER BY mutual_count DESC, user_id;

The edges CTE doubles every undirected row into both directions so traversal works from either endpoint. direct is the set of user 1's friends. The self-join chains e1 (1 → friend) to e2 (friend → friend-of-friend); the WHERE drops user 1 and their direct friends; COUNT(DISTINCT e1.v) counts the bridging friends.

4. Strategic explanations

Symmetrising the edge list — the foundational step

An undirected graph stored as one row per pair ((1,2) but not (2,1)) cannot be traversed directionally. The UNION ALL with swapped columns builds a directed view containing both 1→2 and 2→1. Every traversal query on an undirected graph starts here — without it, you would only ever find neighbours stored as the a endpoint.

The two-hop join and why COUNT(DISTINCT) matters

edges e1 JOIN edges e2 ON e1.v = e2.u with e1.u = 1 walks: first hop 1 → e1.v (a friend), second hop e1.v → e2.v (a friend-of-friend). A friend-of-friend reachable through several friends produces several join rows — user 5 appears via 2, 3, and 4. COUNT(DISTINCT e1.v) counts the distinct bridging friends, which is exactly "mutual connections". Plain COUNT(*) would also work here but DISTINCT guards against duplicate edges.

The three exclusions — self, direct friends, and the implicit back-edge

Three things must be filtered. e2.v <> 1 drops paths that loop back to user 1 (1 → 2 → 1, which the symmetrised edges create). e2.v NOT IN (direct) drops people already friends with user 1 — "friends-of-friends only". Together they ensure the panel shows genuinely new two-hop people. Forgetting either is the most common bug: you would recommend user 1 to themselves, or recommend existing friends.

"Shortest path of length 2" — the subtle correctness point

The spec says shortest path exactly 2. The NOT IN (direct) filter is what enforces this: a node that is both a direct friend and a friend-of-friend has a shortest path of 1, so it is excluded. User 9, by contrast, has no two-hop path at all (its shortest path is 3) — the two-hop join simply never reaches it. The query returns precisely the distance-2 layer, not "distance ≤ 2".

Dialect notes — and the recursive-CTE alternative

UNION ALL, self-joins, and COUNT(DISTINCT) are universal. For a fixed depth of 2 this explicit double-join is the clearest form. For variable or deeper traversal (degrees of separation, shortest path of length N) you would switch to WITH RECURSIVE, carrying a hop counter and a visited-set to avoid cycles. Knowing that two hops is fine as a self-join but N hops needs recursion is the senior framing.

Senior signal: traverse an undirected graph by first symmetrising edges with UNION ALL, express a fixed two-hop reach as a self-join, count bridges with COUNT(DISTINCT), and apply all three exclusions — self, direct friends, and loop-back edges — so the result is exactly the shortest-path-2 layer.
