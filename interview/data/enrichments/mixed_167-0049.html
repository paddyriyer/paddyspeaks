This question masquerades as a one-liner CSV-to-Parquet conversion, but every clause is loaded. repartition(8, "region") controls file count per region at write time; partitionBy("region") controls directory layout on disk. Get either wrong and downstream readers either drown in 100,000 tiny files or scan 2 TB to answer a one-region query. This is the file-layout question every data-platform team asks on the senior screen.

1. The source data — a real Friday-night CSV drop

propertyvalue
file/data/in.csv — one fat CSV from an upstream RDBMS export
rows800,000,000
size on disk~120 GB gzipped
columnsorder_id, region, customer_id, amount, order_ts
region cardinality8 values — US-E, US-W, EU, UK, IN, APAC, LATAM, MEA
distributionskewed: US-E = 38%, US-W = 22%, others split the remaining 40%

The numbers force the design. 120 GB / 128 MB target block ≈ 1,000 ideal Parquet files. Eight regions × ~125 files per region. Per-region queries should touch only one directory, never scan the whole 120 GB.

2. The problem without this technique

Failure mode A — naive read.csv().write.parquet()
Input CSV (1 file, gzip)
        │
        ▼  Spark reads with ONE task (gzip is not splittable)
  ┌─────────────┐
  │  Task 0     │  ← reads ALL 120 GB on one executor
  └─────────────┘
        │
        ▼  No repartition, default 1 output partition
  /data/out/part-00000-xxx.parquet   ← single 120 GB Parquet file ✗

One executor reads the whole gzip. The write produces one giant Parquet file that no downstream reader can split sensibly. A query like WHERE region = 'EU' reads the full 120 GB — no pruning.

Failure mode B — partitionBy("region") alone, no repartition

You add partitionBy for directory pruning. Now every Spark task that holds rows from multiple regions opens N file handles (one per region in that task) and writes a tiny file to each. With default 200 shuffle partitions and 8 regions, you get up to 200 × 8 = 1,600 files, most under 1 MB. The Parquet small-file problem: metadata reads dominate, query planning balloons, S3 LIST throttles you.

Failure mode C — partitionBy("region") + repartition(200) on no column

Still 1,600 files. Random repartition doesn't co-locate rows of the same region on the same task. Every task still writes to every region directory.

3. The fix — step-by-step with WHY for each line

from pyspark.sql import SparkSession
from pyspark.sql.functions import col

spark = (SparkSession.builder
         .appName("csv-to-partitioned-parquet")
         .config("spark.sql.files.maxPartitionBytes", "134217728")  # 128 MB read splits
         .getOrCreate())

# 1) Read CSV. header=True + inferSchema=False to avoid a 2nd pass over 120GB.
#    Provide an explicit schema in production; inference doubles read cost.
df = (spark.read
      .option("header", True)
      .option("inferSchema", False)
      .schema("order_id BIGINT, region STRING, customer_id BIGINT, amount DECIMAL(12,2), order_ts TIMESTAMP")
      .csv("/data/in.csv"))

# 2) repartition(8, "region") = hash-partition rows by region into exactly 8 shuffle partitions.
#    WHY 8? It matches region cardinality. Every task ends up writing ONE region's rows only,
#    so partitionBy("region") at write produces ~1 file per task per region = clean file count.
df_r = df.repartition(8, col("region"))

# 3) partitionBy("region") = Hive-style directory layout on disk.
#    Output: /data/out/region=US-E/part-xxx.parquet, /data/out/region=EU/part-xxx.parquet, ...
#    Downstream readers prune at the directory level: WHERE region='EU' reads ONE folder.
(df_r.write
     .mode("overwrite")
     .partitionBy("region")
     .parquet("/data/out"))

The two knobs are doing different jobs:

KnobWhat it controlsVisible where?
repartition(N, col)In-memory shuffle: how many tasks write the output, and which rows go to which taskNumber of part-* files inside each region directory
partitionBy("region")On-disk directory layout for partition pruning at read timeThe region=XXX/ folders
Resulting layout
/data/out/
├── region=US-E/   part-00000.parquet   (~45 GB → split across multiple files by row-group size)
├── region=US-W/   part-00001.parquet   (~26 GB)
├── region=EU/     part-00002.parquet   (~14 GB)
├── region=UK/     part-00003.parquet   (~9 GB)
├── region=IN/     part-00004.parquet   (~9 GB)
├── region=APAC/   part-00005.parquet   (~7 GB)
├── region=LATAM/  part-00006.parquet   (~6 GB)
└── region=MEA/    part-00007.parquet   (~4 GB)

A query SELECT SUM(amount) FROM out WHERE region='EU' now reads 14 GB, not 120 GB. ~88% I/O reduction — that's the entire reason this pattern exists.

4. Where this appears in production

  • Airbnb partitions its fact_bookings table by (ds, country) — the ds (date string) prunes by day, country prunes by market. Their internal style guide forbids more than two partitionBy columns to keep file count bounded.
  • Netflix partitions raw playback events by (date, region, content_type) with target file sizes of 256 MB. Their Iceberg migration was driven explicitly by Hive's small-file problem when partition cardinality exploded past ~10,000 directories.
  • Uber's Hudi lake hash-buckets trip records by (city_id, hour) using exactly this repartition + partitionBy shape; the bucket count is set to 2 × executor_count for write-time parallelism.
  • Stripe writes the charges ledger partitioned by (ds, livemode): separating livemode=true from livemode=false at directory level avoids ever mixing test and production data in a query plan.

5. Interviewer probes — the edge cases

Probe 1 — "Why repartition(8, "region") and not just partitionBy("region")?"

Tests whether you know what each does. partitionBy is a write-side hint about directory layout, applied per task. Without an upstream repartition by the same column, every task ends up holding rows from many regions and writes a small fragment to each directory — the classic small-file problem. repartition(8, "region") guarantees that each task holds exactly one region's rows, so each task writes one file per region directory.

Probe 2 — "What if region has 50,000 distinct values, not 8?"

Don't partitionBy it. Hive partition pruning is metadata-driven — the metastore has to list every partition. At 50,000 partitions per table, query planning takes minutes. Rule of thumb: partition columns should have low-to-medium cardinality (≤ 10,000). For high-cardinality columns, use bucketing (.bucketBy(N, "col").sortBy("col")) or Iceberg/Delta hidden partitioning with truncate/bucket transforms.

Probe 3 — "US-E is 38% of data — won't task 0 finish 10× slower than task 7 (MEA)?"

Yes — that's data skew at write time. Three fixes:

  • Salt the hot region: withColumn("region_salt", when(col("region")=="US-E", concat(col("region"), lit("_"), floor(rand()*4))).otherwise(col("region"))) then repartition by region_salt. Trade-off: 4 sub-directories for US-E, downstream queries need to union them.
  • Enable AQE: spark.sql.adaptive.enabled = true and spark.sql.adaptive.skewJoin.enabled = true. Splits skewed shuffle partitions automatically since Spark 3.0.
  • Use repartitionByRange("amount") instead if you're co-partitioning for a downstream join. Range partitioning samples the data and picks balanced split points.

See customer_growth-0027 (salting) for the full pattern.

Probe 4 — "What's the relationship to spark.sql.files.maxPartitionBytes?"

That config controls read-side input splits — how many tasks Spark spawns to read the CSV. Default 128 MB. Irrelevant for our gzipped CSV (gzip is not splittable, you get one task regardless), but critical for splittable formats like Parquet, ORC, or bzip2. The write-side file count is controlled by the number of shuffle partitions feeding the write, which is what repartition(8) sets.

Probe 5 — "How do you avoid re-reading 120 GB if the job fails midway?"

Two complementary techniques. (a) Switch to Delta Lake or Iceberg — atomic commits via a transaction log mean a failed write leaves no orphan files. (b) Split the job: first checkpoint the parsed DataFrame to a temporary Parquet (cheap to re-read), then repartition and write the final layout. The checkpoint converts the 120 GB gzip to ~30 GB splittable Parquet, which any retry can re-read in parallel.

Probe 6 — "mode("overwrite") with partitionBy — what's the gotcha?"

By default, mode("overwrite") deletes the entire output directory before writing — including partitions that aren't in this batch. If today's job only has US-E, EU data, yesterday's APAC, LATAM directories disappear. The fix: spark.conf.set("spark.sql.sources.partitionOverwriteMode", "dynamic"). Now only the partitions present in the new DataFrame get overwritten; the rest are untouched. This is the single most common production bug with this pattern.
