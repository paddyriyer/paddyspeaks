A reporting pipeline feeds a daily chart, but the source table only stores days that had activity — so a quiet day produces no row, and the chart skips it instead of showing a zero. The ask is to densify the series: emit every calendar day from the first to the last, filling absent days with v = 0. The interviewer is checking whether you can generate a contiguous date spine and outer-join the sparse data onto it.

1. The source data — sparse

dv(narrative)
2025-03-015range start
2025-03-04803-02 and 03-03 are both missing — a two-day gap
2025-03-052range end — consecutive with 03-04

The edge case is the data: three rows, but a two-day hole between 03-01 and 03-04. A correct answer must emit 03-02 and 03-03 — days that exist in no row — each with v = 0, not NULL and not omitted.

2. The SQL — recursive date spine left-joined to the sparse table

WITH RECURSIVE spine(d, max_d) AS (
  SELECT (SELECT MIN(d::date) FROM sparse), (SELECT MAX(d::date) FROM sparse)
  UNION ALL
  SELECT d + 1, max_d FROM spine WHERE d < max_d
)
SELECT to_char(s.d, 'YYYY-MM-DD') AS d, COALESCE(p.v, 0) AS v
FROM spine s
LEFT JOIN sparse p ON p.d::date = s.d
ORDER BY s.d;

The recursive CTE builds one row per day from MIN(d) to MAX(d). A LEFT JOIN attaches the real value where a day has one; COALESCE(p.v, 0) turns the NULL of an unmatched day into the required 0.

3. The resulting analysis

Five contiguous rows from three input rows. The two zero-valued days are pure spine output: the recursion invented them, the LEFT JOIN found no partner, and COALESCE set them to 0. The chart now draws a continuous line that dips to zero rather than silently jumping the gap.

4. Strategic explanations

Why a recursive CTE generates the missing days

No join can produce rows that exist in no table. The recursive CTE is a row generator: the anchor seeds the first day, the recursive arm feeds its own output back in and adds one day per pass, stopping when d < max_d fails. That self-reference is the pure-SQL way to materialise a dense calendar — the spine, not the data, defines which days appear.

The LEFT JOIN direction is the densification

The spine must be the left table so every generated day survives; sparse is the optional right side. An INNER JOIN would drop the gap days and hand back the original three rows — the opposite of the goal. The whole densification lives in the choice of left join plus the spine being on the left.

COALESCE(p.v, 0) — the spec wants 0, not NULL

An unmatched LEFT JOIN row has p.v IS NULL. The requirement is explicitly v = 0 for gaps, so COALESCE(p.v, 0) substitutes 0. The distinction is real: NULL means "we have no measurement", 0 means "the measured value was zero". For a metric where a quiet day genuinely is zero activity, 0 is correct — but that is a business decision, and the spec made it. State which one you mean.

Termination, casts, and the empty-table edge

WHERE d < max_d bounds the recursion — lose it or get the direction wrong and it never stops. d is TEXT, so d::date is required before + 1 arithmetic, and to_char converts back to the YYYY-MM-DD string at the end. One sharp edge: if sparse were empty, MIN/MAX are NULL, the anchor row is NULL-bounded, and the output is empty rather than an error — usually fine, but worth naming.

Dialect — generate_series is the Postgres idiom

In PostgreSQL the cleaner spine skips recursion: SELECT generate_series(MIN(d)::date, MAX(d)::date, '1 day') AS d FROM sparse as a subquery, then the same LEFT JOIN + COALESCE. SQL Server uses a numbers/tally table; MySQL 8 supports WITH RECURSIVE as shown. The recursive form is the most portable answer; if the engine has generate_series, reaching for it signals fluency.

Senior signal: densify a sparse series by generating a contiguous date spine (recursive CTE or generate_series), keep that spine on the left of a LEFT JOIN so missing days survive, and use COALESCE(v, 0) deliberately — knowing that 0 and NULL carry different meanings and the spec chose 0.
