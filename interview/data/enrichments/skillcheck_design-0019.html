A feature-flag service is a foundational piece of SaaS infrastructure: it decouples feature releases from deployments and enables per-tenant customization. The interviewer is probing whether you can design a three-layer evaluation hierarchy — global flag default, ordered rules, tenant override — with the correct FK relationships and uniqueness constraints, and whether you understand the tradeoffs of a JSONB rule-payload column vs. a typed-columns approach. Three tables and their interactions need to be designed together coherently.

1. The requirements — what the design must capture

  • Flag registry: each flag has a name (unique, the public identifier), a global default on/off state, an archived flag for soft-delete, and timestamps.
  • Ordered rules: flags can have zero or more rules (cohort match, percentage rollout, attribute match); rules have a priority that determines evaluation order; the first matching rule wins.
  • Tenant overrides: a specific tenant can override a flag's computed state entirely, or pin evaluation to a specific rule, or both.
  • Referential integrity: rules belong to flags (cascade delete); overrides reference both flags and optionally a specific rule (restrict delete on the rule to avoid dangling references).
  • CHECK constraint on override: an override row must specify at least one of override_state or override_rule_id — a completely empty override row is invalid.

2. The DDL — the CREATE TABLE

CREATE TABLE flag (
  flag_id          BIGSERIAL    PRIMARY KEY,
  name             VARCHAR(128) NOT NULL UNIQUE,
  description      TEXT,
  default_state    BOOLEAN      NOT NULL DEFAULT FALSE,
  created_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),
  updated_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),
  is_archived      BOOLEAN      NOT NULL DEFAULT FALSE
);

CREATE TABLE flag_rule (
  rule_id          BIGSERIAL    PRIMARY KEY,
  flag_id          BIGINT       NOT NULL REFERENCES flag(flag_id) ON DELETE CASCADE,
  priority         INT          NOT NULL,                   -- evaluation order, lower = first
  rule_type        VARCHAR(32)  NOT NULL,                   -- 'cohort', 'percentage', 'attribute_match'
  payload          JSONB        NOT NULL,                   -- rule-specific parameters
  result_state     BOOLEAN      NOT NULL,                   -- state when this rule matches
  is_active        BOOLEAN      NOT NULL DEFAULT TRUE,
  created_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),
  UNIQUE (flag_id, priority)
);
CREATE INDEX ix_flag_rule_flag_active ON flag_rule (flag_id) WHERE is_active;

CREATE TABLE flag_tenant_override (
  flag_id          BIGINT       NOT NULL REFERENCES flag(flag_id) ON DELETE CASCADE,
  tenant_id        BIGINT       NOT NULL,
  override_state   BOOLEAN,                                 -- NULL means 'inherit, but override the rule set'
  override_rule_id BIGINT       REFERENCES flag_rule(rule_id),
  created_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),
  updated_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),
  PRIMARY KEY (flag_id, tenant_id),
  CHECK (override_state IS NOT NULL OR override_rule_id IS NOT NULL)
);
CREATE INDEX ix_flag_tenant_override_tenant ON flag_tenant_override (tenant_id);

4. Strategic explanations

The three-tier evaluation hierarchy

The evaluation algorithm is: (1) Check for a flag_tenant_override row for this tenant. If it exists and override_state IS NOT NULL, return that state immediately. If it exists and override_rule_id IS NOT NULL, evaluate only that specific rule. (2) If no tenant override, evaluate flag_rule rows for the flag in ascending priority order; return the first matching rule's result_state. (3) If no rule matches, return flag.default_state. This hierarchy is standard in LaunchDarkly-style systems.

UNIQUE (flag_id, priority) and its implications

The uniqueness constraint prevents two rules from competing at the same priority level, which would make evaluation non-deterministic. Reordering rules requires a multi-row update that avoids transient conflicts; the standard technique is a deferred unique constraint or a two-step update (shift priorities first, then set new values). In Postgres, UNIQUE ... DEFERRABLE INITIALLY DEFERRED handles this cleanly within a transaction.

JSONB payload vs. typed-column alternatives

The alternative to JSONB is a separate table per rule type (e.g., cohort_rule, percentage_rule). That approach gives column-level constraints and indexes on individual rule parameters at the cost of complex JOINs and a schema migration for every new rule type. JSONB is the pragmatic choice for a flag service where rule types evolve rapidly and the payload is only consumed by the evaluation engine — not queried ad hoc. If specific rule parameters need indexing (e.g., find all flags that target cohort 42), a GIN index on the payload column covers it.

Dialect notes

This schema is Postgres-native. In MySQL, use JSON instead of JSONB and note that MySQL JSON indexes require generated columns. In Snowflake, VARIANT replaces JSONB and there is no GIN equivalent — use FLATTEN in queries. The deferrable unique constraint on (flag_id, priority) is Postgres-specific; in other engines, wrap priority reordering in an application-level transaction that temporarily assigns out-of-range values.

Senior signal: articulate the three-tier evaluation hierarchy (tenant override → ordered rules → global default) explicitly, justify JSONB for the rule payload as the right tradeoff for a rapidly-evolving rule type set, flag the deferrable unique constraint on (flag_id, priority) as the safe way to reorder rules transactionally, and explain the CHECK on flag_tenant_override that prevents a vacuous override row.
