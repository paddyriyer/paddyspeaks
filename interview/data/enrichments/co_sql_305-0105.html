Meta's growth team wants a fast read on how engaged users are: not the average action count — an average is hostage to a handful of power users — but the shape of the distribution. How many users are dormant, casual, regular, or heavy? The question looks like a one-liner, and it is. The interview is whether your buckets are watertight: no user lands in two, and no user falls through a crack.

The interesting rows are the boundary values: user 4 sits exactly on 10 and user 6 sits exactly on 100. A bucket definition that uses < where it means ≤ would silently misfile them. Those two rows are the whole interview.

2. The SQL — CASE buckets + COUNT

SELECT
  CASE WHEN action_count = 0 THEN '0'
       WHEN action_count <= 10 THEN '1-10'
       WHEN action_count <= 100 THEN '11-100'
       ELSE '100+' END AS bucket,
  COUNT(*) AS users
FROM activity GROUP BY 1;

CASE evaluates top-down and stops at the first match, so the bands do not need to restate their lower edge — reaching the ≤ 100 arm already implies the value is above 10. GROUP BY 1 groups by the first select-list item, the computed bucket.

4. Strategic explanations

The ordering follow-up — buckets sort as strings

A dashboard wants buckets in ascending order, but ORDER BY bucket sorts the labels lexically: '0', '1-10', '100+', '11-100' — '100+' lands before '11-100' because '0' < '1' as text. Carry an explicit sort key:

SELECT bucket, users FROM (
  SELECT CASE WHEN action_count = 0 THEN '0'
              WHEN action_count <= 10 THEN '1-10'
              WHEN action_count <= 100 THEN '11-100'
              ELSE '100+' END AS bucket,
         MIN(action_count) AS sort_key,
         COUNT(*) AS users
  FROM activity GROUP BY 1
) t ORDER BY sort_key;
The empty-bucket trap

If no user has zero actions, the '0' bucket simply does not appear — GROUP BY only emits buckets that have rows. A dashboard expecting all four bars every week will silently lose one. The fix is to LEFT JOIN a tiny buckets dimension (a 4-row VALUES list) against the activity counts, so empty buckets render as zero rather than vanishing.

NULL action counts

action_count = 0 and every ≤ comparison return unknown for a NULL, so a user with a NULL count falls through every arm into the ELSE and is misfiled as '100+' — the worst possible bucket. If NULLs are possible, handle them first: WHEN action_count IS NULL THEN 'unknown', or filter them out with intent.

width_bucket — the built-in for equal-width bins

For evenly-sized numeric bins, Postgres ships width_bucket(value, low, high, count), which removes the hand-written ladder entirely. It does not fit here — our buckets are logarithmic (0, 1-10, 11-100, 100+), not equal-width — but naming it shows you know the histogram is a first-class operation, not always a CASE.

Negative counts and the open-ended top bucket

ELSE is a catch-all: a stray negative action_count would land in '100+', which is nonsense. If the column should never be negative, that is a data-quality check worth a sentence. And '100+' being unbounded is a deliberate product choice — whales and a user with 101 actions share a bucket; if the team cares about that tail, add a '1000+' arm.

Senior signal: get the ≤ boundaries exactly right so edge values land in one bucket only, prove exhaustiveness by checking the counts sum to the row total, anticipate that string-labelled buckets sort lexically and empty buckets disappear, and handle NULL counts before the ELSE arm swallows them.
