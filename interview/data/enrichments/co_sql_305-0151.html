Accenture's reporting team wants a year-over-year view: each month's sales next to the same month a year earlier, plus the delta. The data is one flat sales table, so "the same month last year" has to come from the table joined to itself, shifted twelve months. The query is short, but the join key — a.month = b.month + INTERVAL '1 year' — has a direction that is easy to get backwards, and the LEFT JOIN is doing something specific at the edges.

1. The source data — sales

monthamount
2024-01-01100

2025-03-01200

Two months exist in both years (Jan and Feb), so they have a prior-year comparison. The 2024 months have nothing before them, and 2025-03 has no 2024-03 partner. Those three rows with no match are the edge cases the LEFT JOIN governs.

2. The SQL — self-join shifted one year

SELECT a.month, a.amount,
  b.amount AS prev_year_amount,
  a.amount - b.amount AS yoy_diff
FROM sales a
LEFT JOIN sales b ON a.month = b.month + INTERVAL '1 year';

Alias a is the current month; alias b is the prior-year row. The condition reads "a's month equals b's month plus a year" — so b is always twelve months behind a.

2025-01 grew by 50 over 2024-01; 2025-02 fell 30. The three rows with no prior-year partner — both 2024 months and 2025-03 — show prev_year_amount NULL, and yoy_diff is therefore NULL too: any arithmetic with NULL is NULL.

4. Strategic explanations

The join direction — getting it backwards is silent

a.month = b.month + INTERVAL '1 year' means b is the older row. If you instead wrote a.month + INTERVAL '1 year' = b.month, then b would be the future row, prev_year_amount would actually be next year's number, and every delta would carry the wrong sign. The query still runs and still returns numbers — the bug is invisible without checking a known pair. Always sanity-check one row by hand: 2025-01 should pull 2024-01's 100, and it does.

Why LEFT JOIN, not INNER JOIN

With an INNER JOIN, the three unmatched rows vanish — you would see only the two months that have a prior-year partner. That is fine for a strict YoY-only report, but usually wrong: a dashboard wants every month listed, with the comparison blank where it cannot be computed. The LEFT JOIN keeps all of a and lets b's columns fall to NULL. Choosing LEFT here is the deliberate "show all months" decision.

The NULL yoy_diff — and the percentage version

For months without a partner, NULL is honest — "no comparison available" is not the same as a zero change. But if the consumer wants a number, COALESCE(a.amount - b.amount, 0). The far more common follow-up is the YoY percentage, which needs a divide-by-zero guard:

ROUND((a.amount - b.amount) * 100.0 / NULLIF(b.amount, 0), 1) AS yoy_pct

NULLIF(b.amount, 0) turns a zero prior-year base into NULL instead of raising an error.

The modern alternative — LAG with a year offset

Provided the series is dense (exactly one row per month, no gaps), a window function expresses the same idea without a self-join:

SELECT month, amount,
       LAG(amount, 12) OVER (ORDER BY month) AS prev_year_amount
FROM sales;

But note the catch: LAG(amount, 12) steps back twelve rows, not twelve months. If any month is missing, the offset lands on the wrong month. The self-join, keyed on the actual date arithmetic, is robust to gaps — it matches by calendar value, not by row position. That trade-off (gap-safety vs simplicity) is the senior point.

The grain trap — duplicate months fan out

The join assumes month is unique. If 2024-01 appeared twice (a data bug, or monthly rows that are really per-region), the self-join produces a cross product for that key — each current row matches every prior-year duplicate, multiplying rows and double-counting the delta. Confirm the table is one row per month; if it is per-region, the join key must include region on both sides.

Senior signal: verify the join direction against a known pair so the sign is right, justify LEFT JOIN as the "show every month" choice, guard the percentage form with NULLIF, and explain that LAG(amount, 12) is row-positional and breaks on gaps whereas the date-keyed self-join does not — while checking the table's grain to avoid a fan-out.
