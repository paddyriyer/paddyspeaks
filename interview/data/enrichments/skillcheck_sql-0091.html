An insurance-operations analyst wants to find policy holders who phoned in repeatedly across the year — specifically, those whose calls form at least three distinct "callback clusters," where calls within 7 days of each other count as one cluster. This is a sessionization problem: the interviewer is probing whether you can detect cluster boundaries with LAG and a gap threshold, then count clusters per policy.

1. The source data — call_log

Policy 2 sits exactly on the >= 3 threshold — it must be counted. Policy 4 is the chaining trap: Jan-01→Jan-04 is 3 days, Jan-04→Jan-08 is 4 days; each gap is ≤ 7, so all three calls collapse into a single cluster — the gaps are measured between consecutive calls, not against the cluster's first call.

2. The SQL — flag cluster starts with LAG, sum them per policy

WITH flagged AS (
  SELECT policy_id, called_at,
         CASE WHEN called_at - LAG(called_at) OVER (PARTITION BY policy_id ORDER BY called_at)
                   > INTERVAL '7 days' OR LAG(called_at) OVER (PARTITION BY policy_id ORDER BY called_at) IS NULL
              THEN 1 ELSE 0 END AS is_cluster_start
  FROM call_log
),
cluster_counts AS (
  SELECT policy_id, SUM(is_cluster_start) AS clusters
  FROM flagged
  GROUP BY policy_id
)
SELECT COUNT(*) AS policy_holders
FROM cluster_counts
WHERE clusters >= 3;

flagged marks a call with 1 when it begins a new cluster — either the gap from the prior call exceeds 7 days, or there is no prior call. cluster_counts sums those flags per policy (each 1 is one cluster). The final query counts policies with three or more.

The final answer is 2 — policies 1 and 4... no: policies 1 and 2, the only ones with clusters >= 3. Policy 4 illustrates the chaining rule: although it has three calls, consecutive gaps of 3 and 4 days are both within the window, so it forms a single cluster and is correctly excluded.

4. Strategic explanations

Sessionization — boundaries, not memberships

The elegant trick is to never explicitly build clusters. You only need to count them, and a cluster count equals the number of cluster starts. LAG gives each call its predecessor; a call starts a new cluster exactly when the gap exceeds the threshold (or there is no predecessor). Summing a 0/1 start-flag per policy counts clusters without ever materialising cluster ids. This "flag the boundary, then sum" idiom is the core sessionization pattern — it also underlies counting login streaks, trip legs, and visit sessions.

The gap is between consecutive calls — why policy 4 is one cluster

LAG compares each call to the immediately preceding call, not to the cluster's first call. Policy 4's calls are Jan-01, Jan-04, Jan-08: gaps of 3 and 4 days, both ≤ 7, so no boundary is created and all three are one cluster — even though Jan-01 to Jan-08 is 7 days total and a "from cluster start" rule might split differently. The spec says "the gap from the previous call," so consecutive-pair comparison is correct. If clusters had a maximum span you would need a recursive or running-anchor approach instead; naming that distinction is the senior move.

The first-call NULL must be an explicit cluster start

For a policy's earliest call, LAG returns NULL, and NULL > INTERVAL '7 days' evaluates to UNKNOWN — not TRUE. Without the explicit OR LAG(...) IS NULL, the very first call of every policy would be flagged 0 and you would under-count clusters by exactly one per policy. That single OR term is load-bearing: it ensures every policy's first call opens its first cluster. Forgetting it is the most common bug in this pattern.

> INTERVAL '7 days' — strict inequality matters at the boundary

The rule "within 7 days counts as the same cluster" means a gap of exactly 7 days is still the same cluster, and only a gap strictly greater than 7 days starts a new one — hence >, not >=. A call 7 days and one second after the previous one would, with timestamps, exceed the interval and split. Pin down whether the boundary is inclusive; here "within 7 days" reads as inclusive, so strict > is right.

Dialect notes

timestamp - timestamp > INTERVAL '7 days' and LAG are PostgreSQL/standard. SQL Server has no interval literal — write DATEDIFF(day, LAG(called_at) OVER (...), called_at) > 7. MySQL 8+ uses TIMESTAMPDIFF(DAY, LAG(...) OVER (...), called_at) > 7. Note DATEDIFF/TIMESTAMPDIFF in day units count calendar-day boundaries, which differs subtly from a 7×24h interval — for timestamped data, decide whether "7 days" means 168 hours or 7 date boundaries.

Senior signal: count sessions by flagging boundaries, not by building them — use LAG to compare each event to its predecessor, mark a 1 when the gap exceeds the threshold or the predecessor is NULL (the explicit NULL branch prevents under-counting every group's first cluster), then SUM the flags per group; know that consecutive-pair gaps differ from a maximum-span rule.
