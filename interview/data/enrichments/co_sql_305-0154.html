Deloitte's workforce-analytics team needs the median salary across a single flat list — the robust middle-of-the-pack number that staff-level outliers cannot distort the way they distort the mean. Standard SQL has no MEDIAN aggregate, so the candidate has to construct it: number the rows in salary order, then pick the middle one (odd count) or the two middle ones (even count). The whole interview lives in that odd/even branch and whether the offset arithmetic gets it exactly right.

1. The source data — salaries

salary
60,000
80,000
100,000
120,000
200,000

Five values — an odd count, so the median is the single middle value, 100,000. The 200,000 outlier is deliberate: it pulls the mean up to 112,000 while leaving the median untouched at 100,000. That gap — mean 112K vs median 100K — is exactly why anyone asks for a median.

2. The SQL — ROW_NUMBER with median offsets

SELECT AVG(salary) AS median FROM (
  SELECT salary,
         ROW_NUMBER() OVER (ORDER BY salary) AS rn,
         COUNT(*)     OVER ()                AS n
  FROM salaries
) t
WHERE rn IN ((n+1)/2, (n+2)/2);

The subquery numbers every salary in ascending order (rn) and tags each row with the total count (n). The outer query keeps the middle row(s) and averages them — the AVG matters only for the even case.

3. The resulting analysis

median
100000.000000000000

One row, one number: 100,000. With n = 5, integer division gives (n+1)/2 = 3 and (n+2)/2 = 3 — the same value, so the IN list selects only rn = 3, the single middle salary. AVG over that one row returns it unchanged. Cross-checked against percentile_cont(0.5): also 100,000.

4. Strategic explanations

Why the offsets (n+1)/2, (n+2)/2 handle both parities

This is the elegant core. With integer division: for odd n = 5, both offsets evaluate to 3 — the IN set is {3}, one row, the true middle. For even n = 6, (7)/2 = 3 and (8)/2 = 4 — the set is {3, 4}, the two middle rows, and AVG blends them. One formula, no CASE, because truncating division does the odd/even decision automatically. (Float division would break it — the truncation is load-bearing.)

The one-line answer — percentile_cont

Postgres ships the median as an ordered-set aggregate:

SELECT percentile_cont(0.5) WITHIN GROUP (ORDER BY salary) AS median
FROM salaries;

Lead with this — it is correct by construction and obviously expresses intent. Keep the ROW_NUMBER construction in your pocket: many interviewers explicitly ask you to build median without the aggregate (to test window-function fluency), and engines like MySQL < 8 have no percentile_cont.

percentile_cont vs percentile_disc — the even-count divergence

For an even count they differ. percentile_cont interpolates — with values {80K, 120K} as the middles it returns 100K, matching the AVG-of-two-rows approach. percentile_disc returns an actual data value — it would return 80K, the lower middle. The ROW_NUMBER + AVG solution matches cont. State which definition the business wants; "the median is 100K" is ambiguous for even-sized datasets.

The NULL-salary trap

A NULL salary still receives a ROW_NUMBER and is still tallied by COUNT(*) OVER (), so it inflates n and shifts the median offset — a phantom data point dragging the answer off. ORDER BY salary sorts NULLs last by default so they usually fall outside the middle band, but n is still wrong. Filter first: WHERE salary IS NOT NULL inside the subquery, before numbering. percentile_cont ignores NULLs automatically — another reason to prefer it.

The naive sort-and-pick that breaks at scale

A tempting non-window approach — ORDER BY salary LIMIT 1 OFFSET (count/2) — needs a separate query (or a subquery) just to get the count, returns the wrong thing for even counts unless special-cased, and still has to sort the whole set. The ROW_NUMBER + COUNT() OVER () form computes the count and the ordering in one pass over one sort, and the offset formula handles parity uniformly. It is both cleaner and the genuinely scalable choice.

Senior signal: explain that integer-division offsets (n+1)/2, (n+2)/2 collapse to one row for odd counts and two for even with no CASE, reach first for percentile_cont(0.5) WITHIN GROUP, distinguish cont (interpolated) from disc (actual value) on even-sized data, and filter NULL salaries before numbering so they cannot skew the count.
