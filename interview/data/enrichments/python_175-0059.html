Minimum Window Substring is the canonical "expand-then-contract" sliding-window problem, and it's in the L5 interview rotation because it tests three things at once: (1) can you reason about a sliding window whose two endpoints move at different rates, (2) can you turn a multi-character feasibility check into an O(1) per-step counter, and (3) can you cleanly handle duplicates in the target string. The same pattern is what runs inside Elasticsearch's match_phrase with slop (find the smallest text span that contains every query term), Bloomberg's news-feed term-cluster search, and every "find the shortest snippet containing all these highlights" feature in document viewers.

1. The problem in a sentence

Given strings s and t, return the minimum-length substring of s that contains every character of t including duplicates (or the empty string if no such substring exists). The naive approach — enumerate every (l, r) pair and Counter-check feasibility — is O(n²·|t|) and times out on the LeetCode hidden tests around n = 10⁵. The constraint that rules it out: each character must be visited a constant number of times across the whole sweep. The trick is a two-pointer sliding window where both pointers only advance, and a "satisfied distinct-character count" makes feasibility O(1).

Two invariants make this cheap: (a) we never re-examine a character — l and r only move forward; (b) the have_cnt counter tracks how many distinct required characters are at-or-above their required count, which is what feasibility means. A full Counter == Counter comparison every step would be O(26) — fine for ASCII but the principle scales: keep aggregate state, don't recompute it.

3. The algorithm — naming the data structure

from collections import Counter

def minWindow(s, t):
    if not s or not t: return ""
    need = Counter(t)         # target multiset
    have = {}                 # window's current multiset
    have_cnt, need_cnt = 0, len(need)
    res, res_len = [-1, -1], float('inf')
    l = 0
    for r, c in enumerate(s):                # expand pointer
        have[c] = have.get(c, 0) + 1
        # A required char just hit its required count for the first time.
        # Only the EQUALITY moment increments have_cnt -- subsequent surplus doesn't.
        if c in need and have[c] == need[c]:
            have_cnt += 1
        while have_cnt == need_cnt:          # feasible -> try to shrink
            if r - l + 1 < res_len:
                res, res_len = [l, r], r - l + 1
            have[s[l]] -= 1
            # Dropping STRICTLY below required count breaks feasibility.
            if s[l] in need and have[s[l]] < need[s[l]]:
                have_cnt -= 1
            l += 1                            # contract pointer
    return s[res[0]:res[1] + 1] if res_len < float('inf') else ""

The crucial detail is the EQUALITY check on the increment, versus the STRICT-LESS-THAN check on the decrement. If the target needs B:2 and the window has B:3, dropping one B keeps the window feasible — have[s[l]] -= 1 brings it to 2, which is not less than 2, so have_cnt stays. Get this asymmetry wrong and you'll either over-count satisfied chars (returning windows that aren't feasible) or under-count (skipping windows that are).

4. Complexity

Time: O(|s| + |t|). Each character of s is added to the window exactly once (by the for loop) and removed at most once (by the while loop). The t work is the initial Counter build. Space: O(|alphabet|) for the two counters; constant for ASCII, O(|t|) for arbitrary inputs. The trick that gets you there: a single integer have_cnt replaces an O(26) Counter-comparison every step, and the asymmetric equality/inequality check is what keeps the increment-decrement balanced under duplicates.

5. The interviewer's edge-case probes

  • t is longer than s — return "". The have_cnt == need_cnt condition will never trigger; the function falls through.
  • t has duplicates (e.g., t="aab") — the equality/inequality asymmetry above is what handles this. A window must contain at least 2 a's to be feasible. Junior implementations using a set(t) instead of Counter(t) fail this case.
  • s and t share no characters — return "". have_cnt stays 0 forever; the contract loop never runs.
  • The smallest window is at the very end — the worked example above. The contract loop must run on each feasible state, not just on the first one.
  • s == t — return s. The window grows to full length, becomes feasible, shrinks to length |t| = |s|, and exits.
  • Tied minimum windows (multiple windows of the same minimum length) — the algorithm returns the leftmost. The strict-less-than r - l + 1 < res_len is the tiebreaker. Some interviewers ask for the rightmost — change to ≤.

6. Real-world appearance — what runs this in production

Elasticsearch / OpenSearch — match_phrase with slop

Elasticsearch's span_near query (which backs match_phrase) finds the smallest text span containing all required terms within a slop distance. The underlying Lucene SpanScorer uses a multi-pointer sliding window that is structurally identical to this algorithm — expand pointers until all terms are present, contract from the trailing pointer to minimise span length. Bloomberg's BQuery internal search uses the same primitive for "find me the smallest news-article snippet mentioning all three of {Tesla, earnings, Q3}."

Document viewers — Adobe Acrobat, Google Docs "snippet" highlights

When you search a long PDF for multiple terms, the snippet shown above each result is the minimum window containing all of them. Adobe's PDXSearchHighlight and Google Docs' search-bar highlights both apply this algorithm character-stream-style on the indexed text.

NLP / RAG retrieval — Anthropic, OpenAI embeddings, LangChain

"Span extraction" — given a long document and a set of required keywords, return the shortest contiguous span containing all of them — is a baseline step in retrieval-augmented generation. The minimum-window algorithm is the deterministic fallback when semantic embeddings can't isolate the relevant span. LangChain's parent_document_retriever uses a length-bounded variant.

Log forensics — Splunk, Datadog Logs, Sumo Logic

"Find the smallest timestamp range where all of these events fired" — log forensics uses this primitive to bound an incident. Splunk's transaction command with multiple startswith/endswith patterns is a generalisation: minimum-window over event types instead of characters.

Genomics — short-read alignment

BWA and Bowtie2 use a sliding-window-with-feasibility-check during seed-and-extend alignment. The exact algorithm differs (uses an FM-index for the inner check) but the outer loop is structurally a min-window-with-must-include-all-seeds.
