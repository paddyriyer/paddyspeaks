A customer-success team tracks each account's RFM tier month by month and wants to know who is stable versus who oscillates: a customer who held Tier 1 for a year and just dropped is an urgent save; a customer who flickers between tiers every month is noise. The interviewer is probing the canonical "two ROW_NUMBERs" gaps-and-islands variant — the DIFFERENCE method — and then whether you can pivot the streak table into one summary row per customer covering current streak, longest streak, and total transition count.

2. The SQL — DIFFERENCE-method islands, then a per-customer summary

WITH diffed AS (
    -- streak_id = rn over full timeline  -  rn over (customer, tier) timeline.
    -- Inside a same-tier run both march up in lock-step, so the spread is
    -- constant; any tier change shifts it and opens a fresh streak.
    SELECT customer_id, ym, tier,
        ROW_NUMBER() OVER (PARTITION BY customer_id       ORDER BY ym) -
        ROW_NUMBER() OVER (PARTITION BY customer_id, tier ORDER BY ym) AS streak_id
    FROM monthly_tier
),
streaks AS (
    SELECT customer_id, tier, streak_id,
        COUNT(*) AS streak_months,
        MIN(ym)  AS streak_start,
        MAX(ym)  AS streak_end
    FROM diffed
    GROUP BY customer_id, tier, streak_id
),
longest AS (
    SELECT customer_id,
        tier          AS longest_tier,
        streak_months AS longest_streak_months,
        streak_start  AS longest_streak_start,
        streak_end    AS longest_streak_end,
        ROW_NUMBER() OVER (PARTITION BY customer_id
                           ORDER BY streak_months DESC, streak_end DESC) AS rn
    FROM streaks
),
current_streak AS (
    SELECT customer_id,
        tier          AS current_tier,
        streak_months AS current_streak_months,
        ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY streak_end DESC) AS rn
    FROM streaks
),
streak_count AS (
    SELECT customer_id, COUNT(*) AS total_distinct_streaks
    FROM streaks GROUP BY customer_id
)
SELECT l.customer_id, c.current_tier, c.current_streak_months,
    l.longest_tier, l.longest_streak_months,
    l.longest_streak_start, l.longest_streak_end,
    sc.total_distinct_streaks
FROM longest l
JOIN current_streak c ON c.customer_id = l.customer_id AND c.rn = 1
JOIN streak_count   sc ON sc.customer_id = l.customer_id
WHERE l.rn = 1
ORDER BY l.longest_streak_months DESC, l.customer_id ASC;

4. Strategic explanations

Why two ROW_NUMBERs identify a streak

rn_all counts the customer's months 1, 2, 3… over their whole timeline. rn_tier counts months 1, 2, 3… but only within rows of the same tier. While the tier does not change, both counters advance by exactly 1 each month, so rn_all - rn_tier stays fixed — that fixed value is the streak key. When the tier changes, rn_tier restarts its own count (or jumps to a different per-tier sequence) while rn_all keeps going, so the difference shifts and a new streak begins. The key is only unique within a (customer, tier) pair, which is why GROUP BY customer_id, tier, streak_id needs all three columns.

String ym sorts correctly — but only because of zero-padding

ym is TEXT in 'YYYY-MM' form, and ORDER BY ym works because that format is lexicographically monotonic — '2024-09' < '2024-10' < '2025-01'. This silently breaks if anyone stores '2024-9' without the leading zero. A senior would either enforce the padded format with a check constraint or, more robustly, store a real DATE (the first of the month) and order on that.

The current streak: ORDER BY streak_end DESC, not by recency of any row

"Current streak" must be the streak containing the customer's most recent month. Picking ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY streak_end DESC) and keeping rn = 1 grabs exactly that streak. The longest streak uses a different ordering — streak_months DESC with streak_end DESC as a deterministic tie-break — so two streaks of equal length resolve to the more recent one rather than to an arbitrary row. Without that tie-break the result is non-deterministic.

Dialect notes — QUALIFY collapses the boilerplate

This query needs three "pick rn = 1" CTEs (longest, current_streak) plus an aggregate. In BigQuery, Snowflake, or DuckDB, QUALIFY lets you filter on a window result inline — QUALIFY ROW_NUMBER() OVER (…) = 1 — eliminating the wrapping CTE entirely. PostgreSQL and SQLite have no QUALIFY, so the CTE-then-WHERE pattern shown here is the portable form. The DIFFERENCE method itself works identically everywhere.

What a senior says next

"total_distinct_streaks is effectively a stability score — one streak means a rock-solid account, ten means a flapper. I would expose streak_months / total_distinct_streaks as a single 'tier volatility' metric so success managers can sort their book by who actually needs intervention versus who just looks noisy."

Senior signal: explain that the two ROW_NUMBERs march in lock-step within a same-tier run so their difference is the streak key, flag that the 'YYYY-MM' string only orders correctly because it is zero-padded, give the longest-streak ROW_NUMBER a deterministic tie-break, and mention QUALIFY as the engine-specific way to drop the wrapper CTEs.
