The mirror image of the streak question: "find every user who stopped watching for more than 30 days — the churntential gap." Each gap is a candidate winback target. The query is one of the cleanest applications of LEAD: compare each viewing day to the next viewing day; if the difference exceeds the threshold, you've found a gap.

2. The SQL — LEAD + threshold filter

WITH user_activity AS (
  SELECT user_id, view_date,
    LEAD(view_date) OVER (PARTITION BY user_id ORDER BY view_date) AS next_view_date
  FROM (SELECT DISTINCT user_id, view_date FROM user_views) t
)
SELECT user_id,
  view_date AS gap_start_date,
  next_view_date AS gap_end_date,
  (next_view_date - view_date) AS gap_duration
FROM user_activity
WHERE (next_view_date - view_date) > 30;

3. Strategic explanations

Why LEAD not LAG

Both work; the choice is about how you want to label the gap:

  • LEAD: each row is "this view, and the next view is N days later". The gap is associated with the start of the inactive period.
  • LAG: each row is "this view, and the previous view was N days ago". The gap is associated with the end of the inactive period (the "comeback" day).

LEAD is more natural for "who's about to fall off?" reports. LAG is more natural for "who came back from long gaps?" reports.

The DISTINCT inner query is non-negotiable

Same as in the streak problem: a user with 5 events on the same day creates 5 rows, and LEAD then sees consecutive same-day events with gap=0 — junk rows. Always dedupe to one row per (user, day) before LEAD.

Open-ended gaps — the user who never came back

U_003 has only one row, so LEAD(view_date) IS NULL — the gap is "infinity". The query above silently drops these because NULL - date IS NULL and NULL > 30 IS NULL IS FALSE. To capture them as a separate bucket:

SELECT user_id,
  view_date AS last_seen,
  next_view_date,
  CASE
    WHEN next_view_date IS NULL THEN CURRENT_DATE - view_date
    ELSE next_view_date - view_date
  END AS gap_duration,
  CASE
    WHEN next_view_date IS NULL                        THEN 'open_gap'
    WHEN next_view_date - view_date > 90               THEN 'churn_recovered'
    WHEN next_view_date - view_date > 30               THEN 'extended_gap'
    ELSE 'normal'
  END AS gap_type
FROM user_activity
WHERE next_view_date IS NULL OR next_view_date - view_date > 30;
The 30-day threshold — calibrate from data

Why 30? Netflix's published "active subscriber" definition uses 28-30 days. Other businesses tune differently:

  • Spotify daily-listening: 7-day gap is concerning
  • Booking.com: 90-day gap is normal (annual travel cadence)
  • Fitness apps: 3-day gap is the early-churn signal

The senior approach: calibrate the threshold from the distribution of inter-event gaps among confirmed-active users. Take the 90th percentile of those gaps as the "concerning gap" threshold — data-driven, not Mixpanel-template.

The "gap-during-active" metric — multiple gaps per user

The query emits every qualifying gap. A user might have multiple "took a break" episodes. To get one row per user with their longest gap:

SELECT user_id, MAX(next_view_date - view_date) AS longest_gap
FROM user_activity
WHERE next_view_date IS NOT NULL
GROUP BY user_id;

Combining with streaks — the full lifecycle picture

Streaks (consecutive activity) and gaps (consecutive inactivity) are duals. The customer_growth-0020 question stitches them together with an exit_status CASE. Production retention dashboards always show both:

WITH streaks AS (/* ... */),
gaps AS (/* the LEAD query */)
SELECT u.user_id,
  COUNT(DISTINCT s.streak_id)                        AS lifetime_streaks,
  MAX(s.streak_days)                                 AS longest_streak,
  COUNT(DISTINCT g.gap_start_date)                   AS lifetime_gaps_over_30d,
  MAX(g.gap_duration)                                AS longest_gap_days
FROM users u
LEFT JOIN streaks s USING (user_id)
LEFT JOIN gaps g    USING (user_id)
GROUP BY u.user_id;
The "rolling" version — gaps in the last 90 days only

The original query looks across the user's entire history. For "who's at risk right now?", filter to a recent window:

WITH recent AS (
  SELECT DISTINCT user_id, view_date
  FROM user_views
  WHERE view_date >= CURRENT_DATE - INTERVAL '180 days'
)
/* then run the LEAD logic on `recent` instead of `user_views` */
The dialect note for date subtraction
-- Postgres / DuckDB: date - date = integer (days)
next_view_date - view_date

-- BigQuery: DATE_DIFF
DATE_DIFF(next_view_date, view_date, DAY)

-- Snowflake: DATEDIFF
DATEDIFF(DAY, view_date, next_view_date)

-- SQLite: julianday
julianday(next_view_date) - julianday(view_date)
Production-grade alerting

The query result feeds a daily PagerDuty rule:

  • Distinct users with new 30-day gaps starting today > 5% of yesterday's DAU → product-issue smoke alarm.
  • Distribution shift in gap_duration over 30 days → cohort regression in growth review.
  • Specific "high-LTV" users entering new gaps → CRM team manual outreach.
Senior signal: the LEAD-with-threshold idiom in one breath, surface the open-ended-gap edge case (no LEAD partner) and the dedup requirement, and split "gap" into multiple actionable buckets (open / extended / recovered). The data-driven threshold calibration (90th percentile of inter-event gaps) and combined streaks+gaps lifecycle stitch are the over-the-bar finish.
