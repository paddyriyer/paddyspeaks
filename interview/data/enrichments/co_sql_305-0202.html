"Streak" looks innocent: which customers shop several days in a row? But the engine that powers it — the gaps-and-islands trick — is one of the most-misunderstood SQL patterns. The query below is the textbook one-liner. The senior layer is what comes after: turning streaks into a lifecycle (continuing / resurrected / long-resurrect / churned / open) that drives retention dashboards.

2. The SQL — gaps-and-islands via date - ROW_NUMBER

WITH d AS (
  SELECT DISTINCT customer_id, order_date FROM orders
),
ranked AS (
  SELECT customer_id, order_date,
    order_date - (ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date))::int AS grp
  FROM d
)
SELECT customer_id, COUNT(*) AS streak
FROM ranked
GROUP BY customer_id, grp;

3. The trick — why date - row_number identifies an island

Within a streak, both order_date and row_number increment by 1 each step — so their difference is constant. The moment a date skips, order_date jumps but row_number only ticks by 1; the difference shifts. Group by that difference and you've grouped by streak.

Persona B — U_002 "Long Resurrect"

Active Dec 10-12, gone, returns Mar 26. The gaps-and-islands trick produces two streaks: (Dec 10-12, streak=3) and (Mar 26, streak=1). The 103-day gap between island_2.first and island_1.last is the lifecycle signal — you classify this user as long_resurrect after the streak detection completes.

Persona C — U_003 "Churned"

One order Feb 15, never returns. Just one island, streak=1. The lifecycle layer (next paragraph) flags this user as churned because today − last_order > 90 days and there's no next island to resurrect.

Persona D — U_004 "Open"

Two consecutive days Mar 28-29, no follow-up yet. One island, streak=2. Today − last_order ≤ 90 days → still in grace period. Renewal-marketing eligible but not panicking yet.

5. Strategic explanations — lifecycle on top of streaks

The "longest streak per user" first-pass
WITH islands AS (
  SELECT customer_id, order_date,
    order_date - (ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date))::int AS grp
  FROM (SELECT DISTINCT customer_id, order_date FROM orders) d
),
streaks AS (
  SELECT customer_id, grp,
    MIN(order_date) AS streak_start,
    MAX(order_date) AS streak_end,
    COUNT(*)        AS streak_len
  FROM islands GROUP BY customer_id, grp
)
SELECT customer_id, MAX(streak_len) AS longest_streak FROM streaks GROUP BY customer_id;
The lifecycle classification — what makes the dashboard land

The CRM team's actual question isn't "longest streak" — it's which retention bucket is each user in right now? Layer two more CTEs on top of streaks:

WITH /* islands + streaks as above */
next_activity AS (
  SELECT customer_id, grp, streak_start, streak_end, streak_len,
    LEAD(streak_start) OVER (PARTITION BY customer_id ORDER BY streak_start) AS next_start
  FROM streaks
),
enriched AS (
  SELECT *,
    CASE
      WHEN next_start IS NULL
        THEN CURRENT_DATE - streak_end          -- still inactive
      ELSE next_start - streak_end              -- gap before resurrection
    END AS gap_days
  FROM next_activity
)
SELECT customer_id, streak_start, streak_end, streak_len, gap_days,
  CASE
    WHEN next_start IS NOT NULL AND gap_days <= 28              THEN 'continuing'
    WHEN next_start IS NOT NULL AND gap_days <= 90              THEN 'resurrected'
    WHEN next_start IS NOT NULL AND gap_days  > 90              THEN 'long_resurrect'
    WHEN next_start IS NULL     AND gap_days  > 90              THEN 'churned'
    WHEN next_start IS NULL     AND gap_days <= 90              THEN 'open'
  END AS exit_status
FROM enriched;

The "calibrate the windows" senior follow-up

The 28-day and 90-day thresholds are not universal. They're business-cycle dependent:

  • Subscription B2C (Spotify, Netflix): 30 / 90 is fine.
  • Grocery ecommerce (Instacart): 7 / 30 — people shop weekly.
  • Travel booking (Booking.com): 90 / 365 — once-a-year users are normal.
  • Enterprise SaaS: 30 / 180 — account-cycle dictated.

Always derive the thresholds from the data — e.g., "the 80th percentile of recurring-customer inter-purchase gaps". Do not let a default like 30/90 leak from a Mixpanel template into your dashboard.

Streak breakers — what counts as "consecutive"?

The query treats consecutive calendar days as the streak unit. Real-world variants:

  • Consecutive weeks: bucket dates to DATE_TRUNC('week', ...) first — allows for Tue/Thu cadence to count as a "weekly streak".
  • "At least one purchase per period": bucket to month, then island-detect the buckets.
  • Allow a 1-day grace: this requires LAG-style logic, not gaps-and-islands — see the "tolerant streak" pattern.
The tolerant-streak variant (LAG-based)
WITH d AS (SELECT DISTINCT customer_id, order_date FROM orders),
flagged AS (
  SELECT customer_id, order_date,
    CASE WHEN LAG(order_date) OVER (PARTITION BY customer_id ORDER BY order_date)
              >= order_date - INTERVAL '2 days' THEN 0 ELSE 1 END AS new_streak
  FROM d
),
sessioned AS (
  SELECT customer_id, order_date,
    SUM(new_streak) OVER (PARTITION BY customer_id ORDER BY order_date) AS streak_id
  FROM flagged
)
SELECT customer_id, streak_id, COUNT(*) AS days, MIN(order_date) AS start, MAX(order_date) AS end
FROM sessioned GROUP BY customer_id, streak_id;

Replace 2 days with the tolerated gap. This pattern generalises to "session detection" — the same shape solves "30-minute browsing sessions in a clickstream".

The dialect note — date arithmetic
-- Postgres / DuckDB / Snowflake: date - integer = date (julian-style)
order_date - ROW_NUMBER() OVER (...)::int

-- BigQuery: must use DATE_SUB
DATE_SUB(order_date, INTERVAL CAST(ROW_NUMBER() OVER (...) AS INT64) DAY)

-- SQLite: julianday() math
julianday(order_date) - ROW_NUMBER() OVER (...)

-- MySQL: TO_DAYS
TO_DAYS(order_date) - ROW_NUMBER() OVER (...)
