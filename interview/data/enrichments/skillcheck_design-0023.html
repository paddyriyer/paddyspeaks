An Iceberg fact table for ride-share trips at 80 million rows per day is a data-warehouse scale challenge where the DDL is not just columns and types — it is a partition spec, a clustering/sort order, and table properties that govern file sizes and compression. The interviewer is probing whether you understand Iceberg's partition evolution model (partition without modifying query predicates), bucket partitioning on high-cardinality dimensions like driver_id, geo-clustering with H3 cells for spatial predicate pruning, and which measures are additive vs. non-additive.

1. The requirements — what the design must capture

  • Transaction fact grain: one row per completed (or attempted) trip; this is a transaction fact table — each row represents a discrete business event, not a period snapshot.
  • Common predicates: filter on trip_date (daily range queries), on driver_id (driver analytics), and on pickup H3 cell (geo-density heatmaps); the partition and sort spec must serve all three.
  • Volume: 80M rows/day × 36 months ≈ 87 billion rows; partitioning is mandatory for scan elimination, not optional.
  • Geo encoding: H3 cells at resolution 9 (≈ 105m hexagons) are string tokens; they enable spatial joins without a dedicated geo engine and support efficient clustering because nearby cells sort lexicographically.
  • Monetary and duration measures: fares, tips, payouts — all additive; duration and distance — additive; surge multiplier — non-additive (cannot be summed).

2. The DDL — the CREATE TABLE

CREATE TABLE fact_trip (
  trip_id            STRING        NOT NULL,                   -- natural key (UUID from app)
  rider_id           BIGINT        NOT NULL,
  driver_id          BIGINT        NOT NULL,
  vehicle_id         BIGINT,
  product_type       STRING,                                    -- 'standard','xl','black','pool'
  request_ts         TIMESTAMP     NOT NULL,                    -- when rider requested
  pickup_ts          TIMESTAMP,
  dropoff_ts         TIMESTAMP,
  pickup_h3_r9       STRING        NOT NULL,                    -- H3 resolution 9 (≈74K cells/sq km)
  dropoff_h3_r9      STRING,
  pickup_lat         DOUBLE        NOT NULL,
  pickup_lng         DOUBLE        NOT NULL,
  distance_km        DECIMAL(8,3),
  duration_sec       INT,
  base_fare_cents    BIGINT,
  surge_multiplier   DECIMAL(4,2),
  tip_cents          BIGINT,
  total_paid_cents   BIGINT,
  driver_payout_cents BIGINT,
  status             STRING        NOT NULL,                    -- 'completed','canceled','no_show'
  cancel_reason      STRING,
  trip_date          DATE          NOT NULL,                    -- truncated request_ts in driver-local tz
  ingested_at        TIMESTAMP     NOT NULL
)
USING iceberg
PARTITIONED BY (
  days(trip_date),
  bucket(16, driver_id)
)
TBLPROPERTIES (
  'write.target-file-size-bytes' = '536870912',          -- 512 MB target
  'write.parquet.compression-codec' = 'zstd',
  'write.distribution-mode' = 'hash',
  'sort-order' = 'pickup_h3_r9'                          -- file-level sort for geo predicates
);

-- Optional secondary cluster (engine-dependent): cluster by (driver_id, pickup_h3_r9)
-- for queries filtering by driver and pickup neighborhood.

4. Strategic explanations

Fact table type: transaction + accumulating-snapshot hybrid

At its core this is a transaction fact table: each row represents one discrete trip event. However, storing pickup_ts and dropoff_ts alongside request_ts borrows from the accumulating-snapshot pattern, where multiple milestone timestamps appear on the same row. The tradeoff: updates to the row (when the trip completes and we record dropoff_ts) require Iceberg's MERGE INTO capability — not all Iceberg engines support row-level updates efficiently. An alternative is a pure transaction design with one row per state change, which is append-only but requires more complex queries to reconstruct the trip lifecycle.

Two-dimensional partition spec: days + buckets

PARTITIONED BY (days(trip_date), bucket(16, driver_id)) creates one partition per (date, driver_id bucket). Date partitioning alone gives 36 × 30 = 1,080 partitions; bucket(16, driver_id) multiplies this by 16 for 17,280 total partitions over 36 months. At 80M rows/day this means roughly 5M rows per (day, bucket) partition — a reasonable Parquet file size. Driver-level analytics scan only 1/16 of each day's files. The bucket function is a consistent hash, so the same driver_id always lands in the same bucket, enabling efficient GROUP BY driver_id pushdown.

H3 as a sort key for geo-predicate pruning

H3 cell strings for nearby hexagons do not sort together (H3 uses a hilbert-curve-like ordering internally), but within a city or metro area the set of cells is predictably bounded, and sorting by pickup_h3_r9 within each Parquet file concentrates nearby trips together. Iceberg's file-level min/max statistics on the H3 column allow the engine to skip files that contain no matching cells for geo-bounded queries. This is coarse but effective: a query for "all trips originating in downtown SF" eliminates most non-SF files via min/max pruning.

Additive vs. non-additive measures

All cent amounts (base_fare_cents, tip_cents, total_paid_cents, driver_payout_cents) and duration/distance are fully additive: safe to SUM across any dimension. surge_multiplier is non-additive: it is a ratio, not a quantity. Summing surge multipliers is meaningless; the correct aggregation is a weighted average (SUM(surge_multiplier * base_fare_cents) / SUM(base_fare_cents)). Naming this distinction in the interview is a senior signal.

Iceberg vs. BigQuery / Snowflake dialect notes

In BigQuery: partition on trip_date as a DATE column; cluster by (driver_id, pickup_h3_r9) (BigQuery clustering supports up to 4 columns and drives automatic block elimination). In Snowflake: use CLUSTER BY (trip_date, driver_id); micro-partition pruning handles the rest without explicit bucket partitioning. In Redshift: set DISTKEY(driver_id) and COMPOUND SORTKEY(trip_date, pickup_h3_r9). Iceberg is the portable format; the partition spec and table properties translate to engine-native instructions when using Spark, Trino, or Flink as the compute engine.

Senior signal: choose a two-dimensional partition spec (days + bucket on driver_id) to serve both time-range and driver-level analytics without one dominating the other; identify surge_multiplier as a non-additive measure requiring a weighted average — not a SUM; explain H3 as a geo-clustering key that enables file-level min/max skipping; and know the difference between a pure transaction fact and an accumulating-snapshot fact and the Iceberg write implications of each.
