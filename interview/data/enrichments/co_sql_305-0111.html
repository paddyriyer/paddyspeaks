Deutsche Bank's recruiting team wants a funnel view of its applicant pipeline: how many candidates sit at each stage, and what share of the total each stage represents. The trick is the percentage — you need a per-group count and the grand total in the same query, without a second pass over the table.

1. The source data — applications

app_idstageapplied_at
1applied2024-01-01
2applied2024-01-02
3applied2024-01-03
4applied2024-01-04
5screen2024-01-05
6screen2024-01-06
7onsite2024-01-07
8offer2024-01-08

Eight applications across four stages — a classic narrowing funnel: 4 applied, 2 screened, 1 onsite, 1 offer. The interesting part is the denominator: every stage's percentage must be measured against the same total of 8, computed once.

2. The SQL — COUNT with a windowed total

SELECT stage, COUNT(*) AS apps,
  100.0 * COUNT(*) / SUM(COUNT(*)) OVER () AS pct
FROM applications GROUP BY stage;

The clever line is SUM(COUNT(*)) OVER (): COUNT(*) is the per-stage count, and the window SUM(... ) OVER () with an empty frame sums those group counts across all groups — the grand total. An aggregate of an aggregate, evaluated after the GROUP BY.

3. The resulting analysis

The percentages sum to 100 — the proof the denominator is consistent. Note the result has no ORDER BY, so the engine returns the four stages in whatever order it likes (here grouped, not funnel-ordered). The trailing zeros are numeric display precision; round in the presentation layer.

4. Strategic explanations

How SUM(COUNT(*)) OVER () actually works

Execution order is the key. GROUP BY runs first, collapsing the table to one row per stage with a COUNT(*). Then the window function runs over those four grouped rows: SUM(COUNT(*)) OVER () adds up 4 + 2 + 1 + 1 = 8. The empty OVER () means "the whole result set is one frame". This is why you can nest an aggregate inside a window function but not inside another plain aggregate.

The alternative — a subquery for the total

Before window functions, the total came from a scalar subquery:

SELECT stage, COUNT(*) AS apps,
  100.0 * COUNT(*) / (SELECT COUNT(*) FROM applications) AS pct
FROM applications GROUP BY stage;

It produces the same numbers but scans the table twice — once for the groups, once for the total. The window version scans once. On a large applications table that difference is real; on a small one it is style. Either is defensible if you can articulate the trade-off.

The ordering follow-up — funnels are not alphabetical

A funnel must read applied → screen → onsite → offer, but ORDER BY stage sorts alphabetically: applied, offer, onsite, screen — "offer" jumps to second. Stages are an ordinal sequence with no natural string order. Carry an explicit rank: a CASE mapping each stage to 1–4, or join a small stages(stage, ordinal) dimension table and ORDER BY ordinal.

The integer-division trap

The leading 100.0 is essential. With 100 * COUNT(*) / total — all integers — Postgres does integer division: 100 * 1 / 8 = 100 / 8 = 12, silently dropping the .5. Every non-round percentage would truncate. The 100.0 literal forces numeric arithmetic so the .5 survives. Equivalently, cast: COUNT(*)::numeric.

Missing stages and conversion rates

Two deeper points. First, a stage with zero applicants produces no row — GROUP BY only emits stages that exist; a true funnel view needs a LEFT JOIN to a stages dimension so empty stages show as 0. Second, this query gives share of total, not stage-to-stage conversion — recruiting usually wants "what % of screened candidates reached onsite", which is each stage over the previous stage's count (a LAG), not over the grand total. Clarify which the interviewer means.

Senior signal: explain that SUM(COUNT(*)) OVER () works because the window runs after the GROUP BY, prefer it to a double-scanning subquery while naming the trade-off, force float math to avoid truncated percentages, and distinguish share-of-total from stage-to-stage conversion — and add an explicit ordinal so the funnel does not sort alphabetically.
