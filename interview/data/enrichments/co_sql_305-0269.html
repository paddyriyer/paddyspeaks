DigitalOcean's revenue team wants year-over-year change in sales for each region. The raw sales table has many rows per region-year, so the work is two-staged: first roll up to one total per region-year, then compare each year to the one before. A CTE makes both stages readable — and the interviewer wants to see you keep the window function partitioned so regions never bleed into each other.

1. The source data — sales

regionyearamount
West2022100
West202250
West2023200
East2022300
East2023250

West 2022 has two rows ($100 + $50) — that is why we aggregate first; the YoY comparison must run on the $150 total, not on individual transactions. Each region's earliest year (2022) has no prior year, so its change will be NULL.

2. The SQL — CTE rollup, then LAG

WITH agg AS (
  SELECT region, year, SUM(amount) AS total
  FROM sales
  GROUP BY region, year
)
SELECT region, year, total,
  total - LAG(total) OVER (PARTITION BY region ORDER BY year) AS yoy_change
FROM agg;

The CTE agg collapses transactions to one total per region-year. The outer query's LAG reaches back one row — within the same region, ordered by year — and subtracts.

3. The resulting analysis

regionyeartotalyoy_change
East2022300NULL
East2023250-50
West2022150NULL
West202320050

West 2022's two transactions correctly summed to 150. Each region's 2022 row has a NULL change — there is no 2021 to compare to. East fell 50 (300 to 250); West rose 50 (150 to 200). Critically, West 2022's LAG did not pick up East 2023 — the PARTITION BY region resets the window at each region boundary.

4. Strategic explanations

PARTITION BY is the load-bearing clause

Drop PARTITION BY region and LAG walks the entire result set in year order. West 2022's "prior row" becomes East 2023, and the YoY change turns into nonsense — a cross-region subtraction. The partition is what makes "year over year" mean "this region's previous year". This is the single most common mistake on this pattern; name it explicitly.

Why a CTE and not LAG straight on sales

You cannot LAG directly over the raw table — West 2022 has two rows, so "the previous row" is ambiguous and the comparison would be transaction-to-transaction, not year-to-year. The aggregation must happen first. A CTE is the clean way; a subquery in the FROM is exactly equivalent:

SELECT region, year, total,
       total - LAG(total) OVER (PARTITION BY region ORDER BY year) AS yoy_change
FROM (SELECT region, year, SUM(amount) AS total
      FROM sales GROUP BY region, year) agg;

The CTE is preferred for readability and because it is reusable if the query grows.

The schema's agg column — a deliberate distractor

The table is sales(region, year, amount, agg) and the CTE is also named agg. The column agg is never referenced — the answer computes its own aggregate from amount. Do not assume a column called agg holds a pre-aggregated number you should use; trust the math you control. The name collision between the column and the CTE is harmless (different namespaces) but worth noticing so you do not get confused.

Percentage change — the inevitable follow-up

An absolute delta of -50 means little without scale. Stakeholders want a percentage, which needs a divide-by-zero guard:

WITH agg AS (
  SELECT region, year, SUM(amount) AS total
  FROM sales GROUP BY region, year
)
SELECT region, year, total,
  ROUND(100.0 * (total - LAG(total) OVER w)
      / NULLIF(LAG(total) OVER w, 0), 1) AS yoy_pct
FROM agg
WINDOW w AS (PARTITION BY region ORDER BY year);

Note the named WINDOW clause — the window spec is reused three times, so defining it once avoids copy-paste drift.

Missing years break "year over year"

LAG takes the previous row, not the previous calendar year. If East has 2022 and 2024 but no 2023, East 2024's LAG reaches 2022 and the "YoY" change silently spans two years. If gaps are possible, build a complete region×year spine with generate_series and left-join the totals, or compare on year - 1 with an explicit self-join so a missing year yields NULL rather than a wrong number.

Senior signal: aggregate to one row per region-year before the window function, keep PARTITION BY region so LAG never crosses regions, ignore the decoy agg column, guard the percentage variant with NULLIF(…, 0), and flag that LAG compares adjacent rows — not adjacent years — when the series has gaps.
