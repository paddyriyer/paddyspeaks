A mountaineering records platform needs to show, for each mountain, which climber summited it most recently. The task is a classic "latest event per group" query — one of the most frequently asked SQL interview patterns. The interviewer is probing whether you can select one row per partition ordered by time, using ROW_NUMBER() in a subquery, and whether you know Snowflake's native QUALIFY clause that eliminates the outer wrapper entirely.

1. The source data — climbs

climb_idmountain_idclimber_idsummit_ts

3M1C102025-04-10 07:45:00

Mountain M1 has three climbs — the most recent is C10 on 2025-04-10. Mountain M2 has two climbs — the most recent is C20 on 2025-02-28. Mountain M3 has exactly one climb. Climber C10 summited M1 twice (2025-01-15 and 2025-04-10) — only the later one should appear.

2. The SQL — ROW_NUMBER in subquery, filter on rn = 1

SELECT mountain_id, climber_id, summit_ts
FROM (
  SELECT mountain_id, climber_id, summit_ts,
         ROW_NUMBER() OVER (
           PARTITION BY mountain_id
           ORDER BY summit_ts DESC
         ) AS rn
  FROM climbs
) t
WHERE rn = 1
ORDER BY mountain_id;

Snowflake native shortcut: replace the subquery + WHERE with a single QUALIFY clause: SELECT mountain_id, climber_id, summit_ts FROM climbs QUALIFY ROW_NUMBER() OVER (PARTITION BY mountain_id ORDER BY summit_ts DESC) = 1 ORDER BY mountain_id;

4. Strategic explanations

The "latest row per group" pattern — three approaches

This pattern appears constantly in analytics. Three standard approaches:

  • ROW_NUMBER subquery (this query): assign rn=1 to the latest row per partition, filter in the outer query. Works universally.
  • Correlated subquery: WHERE summit_ts = (SELECT MAX(summit_ts) FROM climbs c2 WHERE c2.mountain_id = c.mountain_id). Simpler to write but can return multiple rows when two climbers summit at the exact same timestamp (a tie), and is less efficient on large tables.
  • DISTINCT ON (PostgreSQL only): SELECT DISTINCT ON (mountain_id) mountain_id, climber_id, summit_ts FROM climbs ORDER BY mountain_id, summit_ts DESC. Elegant and efficient in PostgreSQL but non-standard — not available in Snowflake, BigQuery, or MySQL.
ROW_NUMBER vs RANK for tie handling

If two climbers summited the same mountain at exactly the same timestamp, ROW_NUMBER() assigns rn=1 to one of them arbitrarily (implementation-defined ordering for ties). RANK() would assign rn=1 to both, so the outer WHERE rn = 1 would return both rows — not a single winner. If you need exactly one row even in a tie, ROW_NUMBER is correct. If you need to surface all tied "winners," use RANK() OVER … WHERE rk = 1. The choice depends on the business requirement; a senior candidate always asks before coding.

Why the subquery wrapper is needed (except in Snowflake)

The window function ROW_NUMBER() is evaluated after the WHERE clause in the logical query processing order. Therefore you cannot filter on rn = 1 in the same query that defines rn. A subquery (or CTE) that materializes the window result first, followed by an outer filter, is the portable solution. Snowflake's QUALIFY clause is a dedicated post-window filter, evaluated after window functions — it makes the subquery unnecessary and makes the intent crystal clear.

QUALIFY: Snowflake (and BigQuery) specific

QUALIFY is evaluated in the order: FROM → WHERE → GROUP BY → HAVING → window functions → QUALIFY → ORDER BY → LIMIT. It filters on window function results directly, removing the need for a subquery or CTE. It is available in Snowflake and BigQuery. PostgreSQL, MySQL, and SQL Server do not support it — use the subquery pattern there. Knowing which engines support QUALIFY is a reliable Snowflake-specific interview differentiator.

Performance

On a large climbs table, an index on (mountain_id, summit_ts DESC) allows the engine to seek to the top entry per mountain without scanning the full table. In Snowflake, a cluster key on mountain_id enables micro-partition pruning for the PARTITION BY. The ROW_NUMBER subquery approach typically performs better than the correlated subquery on large data because it makes a single pass rather than one correlated lookup per outer row.

Senior signal: name the "latest row per group" pattern, contrast ROW_NUMBER (exactly one winner even with ties) vs RANK (all tied winners) and explain when each is correct, immediately cite Snowflake's QUALIFY ROW_NUMBER() OVER (…) = 1 as the idiomatic subquery-free solution, and contrast with PostgreSQL's DISTINCT ON as the engine-specific alternative. Mentioning the cluster key on mountain_id for Snowflake performance is the over-the-bar move.
