Compute the median salary per department — but the interviewer has banned PERCENTILE_CONT and PERCENTILE_DISC, the built-ins that would make this trivial. This is a deliberate constraint to see whether you understand what a median is: rank the values, pick the middle one (odd group) or average the two straddling the middle (even group). The technique is ROW_NUMBER plus a group-size COUNT.

The three departments cover the three cases on purpose: an odd-sized group (dept 1), an even-sized group (dept 2), and a single-row group (dept 3). The single-row group is the trap — the median of one value is that value, and the formula must collapse to it cleanly.

2. The SQL — rank, group-size count, pick the middle position(s)

WITH ranked AS (
  SELECT dept_id, salary,
         ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary) AS rn_asc,
         COUNT(*)   OVER (PARTITION BY dept_id) AS cnt
  FROM employee
)
SELECT dept_id,
       ROUND(AVG(salary)::numeric, 1) AS median_salary
FROM ranked
WHERE rn_asc IN (FLOOR((cnt + 1) / 2.0)::int, CEIL((cnt + 1) / 2.0)::int)
GROUP BY dept_id
ORDER BY dept_id;

Each row gets its rank within the department (rn_asc) and the department's total size (cnt). The two "middle positions" are FLOOR((cnt+1)/2) and CEIL((cnt+1)/2): for odd cnt they collapse to the same single position; for even cnt they are the two straddling rows. Averaging the salaries at those positions gives the median in both cases.

Dept 2's median is 65 — the average of the 2nd and 3rd salaries — not any single stored value, which is the whole point of the even-group rule. Dept 3's single employee returns 500.0: both middle positions resolve to rank 1, the IN list dedupes them, and AVG of one value is that value.

4. Strategic explanations

Why (cnt + 1) / 2 with FLOOR and CEIL handles both parities

For odd cnt, (cnt+1)/2 is a whole number — FLOOR and CEIL both return it, so the IN list is a single position and the median is one value. For even cnt, (cnt+1)/2 is an "x.5", so FLOOR gives the lower middle and CEIL the upper middle — two positions, averaged. One formula, no CASE on parity. Writing 2.0 (not 2) forces real division so the .5 actually appears; integer division would silently truncate it.

ROW_NUMBER is correct here — RANK would be a bug

The median picks values at positions, and positions must be a dense, gap-free 1..n sequence — that is exactly ROW_NUMBER. RANK skips numbers after ties (1, 1, 3), so a tied salary at the median position could leave no row with the rank the formula expects, dropping the department entirely. DENSE_RANK collapses ties, distorting which physical row sits at position k. For positional selection, ROW_NUMBER is the only safe ranking function.

The single-row group — why it just works

Dept 3 has cnt = 1. Both middle positions evaluate to 1; the IN (1, 1) list is logically just {1}, so only rank-1 is selected and AVG returns that lone salary. No special case needed — but you should say you checked it, because a formula that divided by cnt - 1 or assumed at least two rows would crash here.

Dialect notes — and the banned built-in

The intended one-liner, PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary), exists in PostgreSQL, Oracle, and SQL Server — the interviewer banned it precisely to test the manual method. MySQL has no percentile function before 8.0, making this ROW_NUMBER technique the standard MySQL answer anyway. The window functions used here (ROW_NUMBER, windowed COUNT(*)) are fully portable; only the ::numeric/::int casts need CAST(...) syntax elsewhere.

Common mistakes

(1) Integer division — (cnt+1)/2 with an integer 2 truncates, so even groups silently pick one value instead of averaging two. (2) Using RANK and getting NULL departments when salaries tie. (3) Computing a per-department median in a correlated subquery — correct but O(n²); the windowed version is one sort per partition.

Senior signal: implement median as ROW_NUMBER ranking plus a windowed COUNT(*), select positions FLOOR and CEIL of (cnt+1)/2.0 so odd and even groups fall out of one formula, insist on ROW_NUMBER (never RANK) for positional selection, and force real division with a literal 2.0.
