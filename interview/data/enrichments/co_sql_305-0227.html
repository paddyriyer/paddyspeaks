Reddit's growth team logs a daily-events feed, and a dashboard charts it day by day. Then someone notices the line chart has invisible gaps — days with zero events simply don't render, so a flat stretch looks identical to a missing one. The fix is a query that names every calendar day that has no matching event row. The interviewer wants to see you reach for a complete date spine, not the events table itself.

1. The source data — calendar and events

calendar is a dense spine of all ten days, 05-01 through 05-10. events is sparse — only five days fired. The gap days 05-03, 05-04, 05-06, 05-07 and 05-09 exist in the spine but never appear in events. Those are exactly the rows we want to surface.

2. The SQL — anti-join via LEFT JOIN … IS NULL

SELECT c.d
FROM calendar c
LEFT JOIN events e ON c.d = e.d
WHERE e.d IS NULL;

The LEFT JOIN keeps every calendar day. Where a match exists, e.d is populated; where none does, the join pads e.d with NULL. Filtering on e.d IS NULL keeps precisely the unmatched days — the classic anti-join.

Five days with no events. Note these days can only come from the calendar table — the events table has no knowledge they exist. That is the whole reason a date spine is mandatory: you cannot find missing rows by querying the table that is missing them.

4. Strategic explanations

Why NOT IN is the dangerous answer here

A tempting rewrite is WHERE c.d NOT IN (SELECT d FROM events). It works — until events contains a single NULL date. NOT IN against a list that holds a NULL evaluates to UNKNOWN for every row, so the query silently returns zero rows. NOT EXISTS and the LEFT JOIN … IS NULL anti-join are both NULL-safe and should be the default:

SELECT c.d
FROM calendar c
WHERE NOT EXISTS (SELECT 1 FROM events e WHERE e.d = c.d);
What if there is no calendar table?

Many schemas don't ship a calendar table. Generate the spine inline rather than abandoning the approach — generate_series bounded by the data's own min and max:

SELECT g.d
FROM generate_series(
       (SELECT MIN(d) FROM events),
       (SELECT MAX(d) FROM events),
       INTERVAL '1 day') AS g(d)
LEFT JOIN events e ON e.d = g.d::date
WHERE e.d IS NULL;

One caveat: a series bounded by MIN/MAX of events can never detect a gap at the very start or end of the reporting window. If the business wants "missing days in May" specifically, hard-code the month boundaries instead.

Duplicates and granularity

If events has multiple rows per day, the LEFT JOIN would fan out matched calendar days into duplicates — harmless here because we only keep the unmatched side, which has exactly one row each. But if the column were a TIMESTAMP rather than a DATE, c.d = e.d would almost never match; you would join on e.ts::date = c.d. Always confirm both sides share the same grain.

Counting gaps, not just listing them

A natural follow-up: "how many days were missing, and what's the longest dry spell?" Listing is step one; quantifying is the senior finish:

SELECT COUNT(*) AS missing_days
FROM calendar c
LEFT JOIN events e ON c.d = e.d
WHERE e.d IS NULL;
Senior signal: recognise that missing rows can only be found against a complete date spine, reach for a NULL-safe anti-join (LEFT JOIN … IS NULL or NOT EXISTS) over the NOT IN trap, and know that a generate_series spine bounded by the data itself cannot catch gaps at the window edges.
