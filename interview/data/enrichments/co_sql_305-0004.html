Backblaze's people-ops team wants an org roster that reads in plain English: each employee next to their manager's name, not a cryptic manager_id. The data only stores the id, so the name has to come from the same table joined to itself. The interview question hiding inside this is small but sharp: what happens to the person at the top, whose manager_id is NULL?

1. The source data — employees

Five employees in a chain. Ada is the CEO — her manager_id is NULL, because she reports to nobody. That single NULL is the whole interview: get the join type wrong and Ada vanishes from the roster.

2. The SQL — self-join, LEFT

SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.emp_id;

The table is aliased twice: e for the employee, m for the manager. The join condition matches an employee's manager_id to another row's emp_id. LEFT JOIN keeps every e row even when no m matches.

4. Strategic explanations

Why LEFT and not INNER — the question the interviewer is testing

Swap LEFT JOIN for plain JOIN and the result loses a row:

SELECT e.name AS employee, m.name AS manager
FROM employees e
JOIN employees m ON e.manager_id = m.emp_id;
-- Ada is GONE: her manager_id is NULL, so e.manager_id = m.emp_id
-- is never true (NULL = anything is never TRUE), and INNER JOIN drops her.

An INNER JOIN keeps only rows where the condition is TRUE. NULL = m.emp_id is never TRUE for any m, so Ada has no match and is dropped. The CEO — the single most important person on an org roster — silently disappears. LEFT JOIN is mandatory whenever the "parent" pointer can be NULL.

Making the NULL human-readable

A NULL in the manager column looks like missing data. For the top of the org it is a fact, not a gap — label it:

SELECT e.name AS employee,
       COALESCE(m.name, '(top of org)') AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.emp_id;
Aliasing is not optional in a self-join

A self-join references one physical table twice, so the engine cannot tell which name you mean without aliases. employees e and employees m create two independent "copies" to reason about. Forget the aliases and you get an ambiguous column error — or worse, in a sloppy query, a silently wrong column. Always qualify every column (e.name, m.name) in a self-join.

Edge cases: dangling pointers and self-managers

Two data-quality cases worth naming. First, a dangling manager_id — an id pointing at an emp_id that no longer exists (the manager left). With LEFT JOIN that employee still appears, with a NULL manager — indistinguishable from the CEO. A diagnostic query finds them:

SELECT e.emp_id, e.name, e.manager_id
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.emp_id
WHERE e.manager_id IS NOT NULL AND m.emp_id IS NULL;

Second, a row where manager_id = emp_id — an employee managing themselves. The self-join happily matches it (the person appears as their own manager), and a recursive walk over the same data would loop forever. Both are integrity violations the join will not catch for you; flag them as checks the source system should enforce.

Senior signal: explain that LEFT JOIN is required because NULL = emp_id is never true and an INNER JOIN would drop the CEO, alias both copies of the table explicitly, dress the top-of-org NULL with COALESCE, and name the dangling-pointer and self-manager data-quality cases the join silently tolerates.
