Okta's GTM ops opens the quarterly board deck and the second slide is always "revenue by region, percent of total." The query is two aggregations stacked: per-region SUM for the numerator, and a window-function SUM across all groups for the denominator. The senior trick is recognising that you can do both in one pass — a window over the GROUP BY result.

2. The SQL — aggregate inside a window aggregate

SELECT region,
  SUM(amount) AS region_total,
  100.0 * SUM(amount) / SUM(SUM(amount)) OVER () AS pct_contribution
FROM sales
GROUP BY region;

4. Strategic explanations

The double-SUM mind-bender — SUM(SUM(amount)) OVER ()

The inner SUM(amount) is the GROUP BY aggregate — per-region total. The outer SUM(...) OVER () is the window aggregate, and at this point the input is the post-grouping rows. With no PARTITION BY or ORDER BY, the window covers all rows — so it sums the per-region totals into the grand total. One pass, no subquery.

Equivalent forms with explicit subquery (less elegant, but easier to read for juniors):

WITH region_totals AS (
  SELECT region, SUM(amount) AS region_total
  FROM sales GROUP BY region
)
SELECT region, region_total,
  100.0 * region_total / SUM(region_total) OVER () AS pct_contribution
FROM region_totals;
Beware the 100.0 * — integer division

If amount is an integer column and you write 100 * SUM(amount) / SUM(...), some databases (MySQL, SQLite) do integer division and you'll get values like 45 instead of 45.7. The 100.0 forces the dividend into floating-point. Always lead percentage formulas with a literal that has a decimal point.

The zero-denominator trap

What if the entire window has SUM(amount) = 0? You divide by zero and the query errors out (Postgres) or returns NULL (SQLite). Wrap with NULLIF:

100.0 * SUM(amount) / NULLIF(SUM(SUM(amount)) OVER (), 0) AS pct
Per-product within region — PARTITION BY

The board's follow-up is usually "break it down by product within each region." Add the partition:

SELECT region, product,
  SUM(amount) AS amt,
  100.0 * SUM(amount) / SUM(SUM(amount)) OVER (PARTITION BY region) AS pct_within_region
FROM sales
GROUP BY region, product;

Now each region's products sum to 100%, and you can spot product-mix shifts within geographies.

The "what's missing 5% from where?" follow-up — YoY contribution shift

The CFO's actual question is rarely the snapshot — it's "how did the contribution mix shift YoY?" Two windows, one for current year and one for prior:

WITH yearly AS (
  SELECT EXTRACT(YEAR FROM sale_date) AS yr, region, SUM(amount) AS amt
  FROM sales GROUP BY 1, 2
),
mix AS (
  SELECT yr, region, amt,
    100.0 * amt / SUM(amt) OVER (PARTITION BY yr) AS pct
  FROM yearly
)
SELECT region,
  MAX(CASE WHEN yr = 2025 THEN pct END) AS pct_2025,
  MAX(CASE WHEN yr = 2026 THEN pct END) AS pct_2026,
  MAX(CASE WHEN yr = 2026 THEN pct END) - MAX(CASE WHEN yr = 2025 THEN pct END) AS pct_shift
FROM mix GROUP BY region;
Pareto / 80-20 follow-up

Sort regions descending by contribution and take a running total — the first regions to cross 80% are the Pareto leaders, the rest are the long tail.

SELECT region, region_total, pct,
  SUM(pct) OVER (ORDER BY pct DESC ROWS UNBOUNDED PRECEDING) AS cumulative_pct
FROM regional_mix;
Senior signal: name the aggregate-of-aggregate idiom (SUM(SUM(...)) OVER ()) crisply, pre-empt integer-division and zero-denominator traps, and offer the per-region PARTITION BY form without being asked. The YoY mix-shift and Pareto follow-ups are over-the-bar.
