NVIDIA's order-operations team wants a working view of the fulfillment pipeline: every order whose status is shipped, delivered, or returned — the post-dispatch states. It looks like the simplest filter in SQL, and it nearly is. But the interviewer asking this is checking that you know how IN handles a value not in the list, how it behaves around NULL, and why SELECT * is a liability in anything but a quick check.

1. The source data — orders

order_idstatuscustomer_idamount
1shipped101100
2pending102200
3delivered103300
4returned104400
5cancelled105500
6shipped106600

Six orders across five distinct statuses. Two — pending and cancelled — are not in the target list; they are the rows the filter must exclude. Order 6 confirms a status can repeat.

2. The SQL — WHERE status IN (...)

SELECT * FROM orders WHERE status IN ('shipped','delivered','returned');

status IN (a, b, c) is exact shorthand for status = a OR status = b OR status = c. A row is kept when its status equals any one of the three listed values.

3. The resulting analysis

order_idstatuscustomer_idamount
1shipped101100
3delivered103300
4returned104400
6shipped106600

Four orders survive — the two shipped plus the one each delivered and returned. Orders 2 (pending) and 5 (cancelled) are dropped: their status matched none of the list, so every OR branch was false.

4. Strategic explanations

IN is just chained OR — and its NULL behaviour follows from that

Because IN expands to a chain of equality checks, a row with a NULL status is not returned: NULL = 'shipped' is unknown, never true, for every branch. That is fine for an inclusion list. It only bites you with the negation — NOT IN — where a NULL in the list can empty the whole result. Worth stating that the trap lives in NOT IN, not IN.

SELECT * — fine for a probe, risky in production

SELECT * is convenient at the prompt, but in a saved query or a view it is fragile: if someone adds a column to orders, the result shape changes silently and can break a downstream consumer or an INSERT ... SELECT. Name the columns you actually need:

SELECT order_id, status, customer_id, amount
FROM orders
WHERE status IN ('shipped','delivered','returned');
Case sensitivity — the silent miss

String comparison in Postgres is case-sensitive, so 'Shipped' or 'SHIPPED' would not match 'shipped' and that order would quietly disappear. If status casing isn't guaranteed clean, normalise both sides: WHERE LOWER(status) IN ('shipped','delivered','returned') — though note that wrapping the column in a function can prevent an index on status from being used.

IN with a subquery — keeping the list dynamic

A hard-coded list goes stale when the business adds a status. If valid statuses live in a reference table, drive the filter from it:

SELECT order_id, status, customer_id, amount
FROM orders
WHERE status IN (SELECT status_name FROM post_dispatch_statuses);

This is also where the NOT IN + NULL hazard would matter — if the subquery could yield a NULL, a negated form would need a WHERE ... IS NOT NULL guard or a switch to NOT EXISTS.

Edge case: an empty IN list

status IN () is a syntax error in most engines, and an IN fed by a subquery that returns no rows matches nothing — the query returns zero rows, correctly. If the status set is built dynamically in application code, guard against generating an empty list.

Senior signal: explain that IN is chained OR and that a NULL status simply fails to match (the real NULL trap belongs to NOT IN), replace SELECT * with an explicit column list for anything persisted, flag case sensitivity, and drive the list from a reference table when the valid statuses can change.
