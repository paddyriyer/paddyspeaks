Twilio bills international and domestic calls at very different rates, so the finance team wants the share of traffic that is international. Simple ratio — except the is_international flag arrived as a grab-bag of text values: 'true', '1', 't'. The reference normalises all the truthy spellings and divides. The interviewer is checking whether you spot the dirty-boolean problem and get the float-division detail right.

1. The source data — calls

Eight calls, and is_international is stored as text in three different truthy spellings — 'true', '1', 't'. Four rows are international (1, 3, 5, 6); four are domestic. That inconsistent encoding is the interview.

2. The SQL — normalise truthy values, then ratio

SELECT 100.0 * SUM(CASE WHEN is_international::text IN ('1','true','t','True','TRUE') THEN 1 ELSE 0 END)/COUNT(*) AS intl_pct
FROM calls;

The CASE maps every recognised truthy spelling to 1 and everything else to 0; SUM counts the international calls; COUNT(*) counts all calls. Multiplying by the literal 100.0 (not 100) forces the division into floating point.

3. The resulting analysis

intl_pct
50.000000

Four international out of eight → 50%. The CASE caught all three truthy spellings — 'true' (twice), '1', and 't' — so no international call was missed. Drop one spelling from the IN list and the percentage silently drifts down.

4. Strategic explanations

Why 100.0 and not 100 — the integer-division trap

If both operands of / are integers, Postgres does integer division and truncates. SUM(...)/COUNT(*) with 4 international and 8 total is 4/8 = 0 — the fraction is gone before the percentage is computed. Writing 100.0 * first promotes the whole expression to numeric, so the division keeps its decimal. The placement matters: 100.0 * SUM / COUNT works; 100 * (SUM/COUNT) would compute 0 inside the parentheses and then multiply zero by 100. An explicit cast — SUM(...)::numeric / COUNT(*) — is the clearest way to signal intent.

The dirty-boolean problem — an allow-list is fragile

The CASE hard-codes five truthy spellings. That works until a new spelling appears — 'TRUE ' with a trailing space, 'Y', 'yes' — and that call is silently miscounted as domestic. A sturdier normalisation lowercases and trims first:

SUM(CASE WHEN lower(trim(is_international::text)) IN ('1','true','t','y','yes')
         THEN 1 ELSE 0 END)

Better still, if the column is genuinely a Postgres boolean, none of this is needed — SUM(is_international::int) or COUNT(*) FILTER (WHERE is_international) just works. The text storage is an ETL defect; the real fix is to cast the column to boolean at load time.

What about the unrecognised values? The NULL question

The CASE's ELSE 0 lumps three groups together: genuine domestic calls, NULL flags, and unrecognised junk. So a NULL is_international is counted as domestic, and it still sits in the COUNT(*) denominator. If "unknown" should be excluded rather than assumed domestic, the denominator must change too:

SELECT 100.0 * COUNT(*) FILTER (WHERE lower(trim(is_international::text)) IN ('1','true','t'))
       / COUNT(*) FILTER (WHERE is_international IS NOT NULL) AS intl_pct
FROM calls;

Numerator and denominator must agree on how unknowns are treated — mixing "exclude from top, include in bottom" understates the rate.

The FILTER idiom — cleaner conditional counting

SUM(CASE WHEN cond THEN 1 ELSE 0 END) is the portable way to count matching rows; Postgres' COUNT(*) FILTER (WHERE cond) says the same thing more directly and is easier to read. Functionally identical here; FILTER signals current-Postgres fluency.

Edge case: the empty table

On a table with no calls, COUNT(*) is 0 and the division is 0/0 — Postgres raises a division-by-zero error, not NULL. If the dashboard can ever query an empty window, guard the denominator: ... / NULLIF(COUNT(*), 0), which yields NULL instead of crashing. Wrap the whole thing in COALESCE(..., 0) if a number is required.

Senior signal: multiply by 100.0 (or cast) so integer division does not truncate the ratio to zero, treat the multi-spelling text flag as an ETL defect and normalise defensively with lower(trim(...)), decide whether NULL flags count as domestic or are excluded (and keep numerator and denominator consistent), and guard 0/0 with NULLIF for the empty-table case.
