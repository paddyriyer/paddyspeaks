Every observability stack you have ever read about — Datadog, Prometheus, Honeycomb, New Relic, Grafana Cloud — reports a p50 latency. None of them re-sort a million-sample buffer every second to produce it. The streaming-median problem is the underlying primitive: keep a running summary of an unbounded numeric stream that answers "what's the middle value?" in constant time after each insert. The two-heap trick that solves it is the canonical L5-Python interview question for heap fluency, and the same shape (two complementary data structures keeping a balanced invariant) appears in t-digest, HDR histogram, and the GK quantile sketch that all the above products actually ship.

1. The problem in a sentence

Design a data structure that supports two operations on an unbounded stream of numbers: addNum(x) in O(log n) and findMedian() in O(1) (returning the middle element for odd counts, the average of the two middle elements for even counts). The constraint that rules out the naive approach: sort the buffer on every findMedian call is O(n log n) per query; insertion-sort on every addNum is O(n) per insert. At 1M samples/sec, neither fits in the time budget. The trick is to split the stream into two halves that we never need to fully sort, only to keep balanced.

2. Worked example with the seeded data

Stream: 5, 2, 8, 1, 9, 3, 7. State evolution of lo (max-heap, larger root means larger key) and hi (min-heap):

add        lo (max-heap)    hi (min-heap)    median
─────────────────────────────────────────────────────
add(5)     [5]              []                5
add(2)     [2]              [5]               (2+5)/2 = 3.5
add(8)     [5]              [8]               5            ← lo's top is the answer
add(1)     [2,1]            [5,8]             (2+5)/2 = 3.5
add(9)     [5,1,2]          [8,9]             5
add(3)     [3,1,2]          [5,8,9]           (3+5)/2 = 4.0
add(7)     [5,1,3,2]        [7,8,9]           5

Visual after the last add (7 numbers ingested):
  lo holds {1,2,3,5} (the 4 smallest)            ← max-heap, top = 5
  hi holds {7,8,9}   (the 3 largest)             ← min-heap, top = 7
                            ↑
                       median = lo's top = 5
                       (odd count → middle is lo's top by invariant)

The invariant we maintain: len(lo) == len(hi) or len(lo) == len(hi) + 1. When the count is odd, lo is one larger and its root is the median. When even, the median is the average of lo's root and hi's root. Both reads are O(1).

3. The algorithm — naming the data structure

import heapq

class MedianFinder:
    def __init__(self):
        self.lo = []   # max-heap of the smaller half (Python heapq is min-only,
                       # so we negate every value going in and going out)
        self.hi = []   # min-heap of the larger half

    def addNum(self, num):
        # Step 1: tentatively push into lo (negated for max-heap semantics).
        heapq.heappush(self.lo, -num)
        # Step 2: move lo's max into hi. This guarantees the heap-cross
        # invariant: max(lo) <= min(hi). After this step lo may be 1 SHORT.
        heapq.heappush(self.hi, -heapq.heappop(self.lo))
        # Step 3: rebalance so lo is either equal-size or 1 bigger than hi.
        if len(self.hi) > len(self.lo):
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def findMedian(self):
        if len(self.lo) > len(self.hi):
            return -self.lo[0]                       # odd count: lo's top
        return (-self.lo[0] + self.hi[0]) / 2        # even count: average of tops

The three-step addNum (push-to-lo → push-lo's-max-to-hi → rebalance) is the canonical formulation because each step preserves exactly one invariant. Pushing directly into lo or hi based on the value would also work but requires comparing to the current median first — that path branches more and is harder to get right under interview pressure.

4. Complexity

Time: addNum is O(log n) — exactly two heap pushes and one heap pop. findMedian is O(1) — both heaps expose their root in constant time. Space: O(n) — every value is held in one of the two heaps. The trick that gets you there: instead of one sorted structure, use two unsorted heaps whose roots bracket the median. Sorting is O(n log n); two heaps cost log-per-insert because we only need to know the top, not the order.

5. The interviewer's edge-case probes

  • findMedian on an empty stream — undefined; some specs return 0.0, some raise. State your convention out loud.
  • Negative numbers — the heap negation is independent of value sign. Don't second-guess yourself.
  • Duplicate values — fine; the heap handles ties internally. Median of [5, 5, 5] is 5.
  • The 3-step addNum looks redundant — why push then immediately pop? Because the simpler "push to lo if x < median else hi" requires you to know the median first, and bootstrapping the first two pushes from that is fiddly. The canonical form has no branches and is provably invariant-preserving.
  • Extension to p95 / p99 — heaps don't extend naturally. The answer is a t-digest or HDR histogram: bucketed approximate quantiles with O(1) merge across shards. Datadog uses DDSketch; Prometheus uses bucketed histograms with linear interpolation.
  • Removing a value (sliding-window median) — heaps don't support arbitrary delete. Use a multiset (Python: SortedList from sortedcontainers) keyed by value, or a two-heap with lazy deletion (mark-and-sweep).

6. Real-world appearance — what runs this in production

Observability — Datadog, Prometheus, Honeycomb, Grafana

None of these store every sample and re-sort on query — too expensive. Datadog ships DDSketch (relative-error quantile sketch), Prometheus uses pre-defined bucket histograms (histogram_quantile() with linear interpolation), Honeycomb stores raw events but computes quantiles via reservoir sampling, Grafana Cloud uses t-digest. All of these are streaming-quantile structures whose conceptual ancestor is the two-heap median. The two-heap is the interview-friendly canonical form; t-digest is what you reach for when you need p99 and merging across nodes.

HFT / market making — Jane Street, Citadel Securities, Optiver

"Median price over the last 1000 ticks" is a fundamental signal in market-microstructure models. The sliding-window variant of the two-heap (with lazy deletion of expired ticks) runs inside FPGA-paired trading kernels at sub-microsecond latency. Citadel's published HFT papers describe a "ring-buffer multiset" for the rolling median that is essentially this algorithm.

Real-time anomaly detection — Stripe Radar, Square Sift

Stripe's fraud-detection engine maintains streaming p50 of "merchant's typical transaction amount" per merchant per hour. When a single transaction exceeds 5× the streaming median, the rule fires. The two-heap (per merchant, in Redis as a sorted-set) is the persistence layer; the median read is O(1) hot-path.

Adaptive video bitrate — YouTube, Netflix, Twitch

The ABR algorithm picks a bitrate based on the streaming median (not mean — too sensitive to congestion spikes) of recent bandwidth measurements. Netflix's open-sourced Mezzanine and Twitch's Pied Piper ABR implementations both reference a rolling-median bandwidth estimator with two heaps as the underlying structure.

Database query planners — PostgreSQL ANALYZE, Snowflake automatic clustering

PostgreSQL's pg_statistic table stores most-common-values and histograms; Snowflake auto-collects approximate quantile sketches over every column. Both use streaming-quantile structures (Greenwald-Khanna for Postgres, GK-variant for Snowflake) — same family.
