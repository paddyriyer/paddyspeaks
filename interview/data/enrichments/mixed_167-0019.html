A sales dashboard needs a ranked list of orders by value, where equal-value orders share a position with no gaps afterwards. The reference solution uses DENSE_RANK(). It is a one-liner, but the interviewer is checking that you can distinguish DENSE_RANK from RANK and ROW_NUMBER — the three ranking functions diverge precisely on ties.

1. The source data — orders

Two ties: orders 1 and 2 at $500, orders 3 and 4 at $300. Those tie groups are the whole interview — they decide whether the next rank is 2 or 3.

2. The SQL — DENSE_RANK over amount

SELECT order_id, amount,
  DENSE_RANK() OVER (ORDER BY amount DESC) AS rk
FROM orders;

3. The resulting analysis

The ranks read 1, 1, 2, 2, 3. After the two-way tie at rank 1, the next value gets rank 2 — DENSE_RANK never skips. That is the defining behaviour.

4. Strategic explanations

DENSE_RANK — ties share a rank, no gaps

DENSE_RANK assigns the same rank to tied rows and then advances the counter by exactly one, regardless of how many rows tied. So a 17-way tie at rank 1 is still followed by rank 2. The result is a contiguous ranking with no holes — ideal when "rank" should mean "which distinct tier," like medal placings or pricing bands.

The contrast that wins the interview — RANK vs DENSE_RANK vs ROW_NUMBER

All three sort the same way; they differ only on ties. On this dataset:

ROW_NUMBER ignores ties — every row gets a unique number, the tie broken arbitrarily. RANK shares ranks but skips after a tie (1, 1, 3, 3, 5) — competition ranking. DENSE_RANK shares ranks and does not skip (1, 1, 2, 2, 3). Saying this comparison crisply is the senior tell.

The ORDER BY inside the window is non-deterministic on ties

Within a $500 tie, which physical order_id comes "first" is undefined — it does not matter for DENSE_RANK (both get rank 1 anyway), but it would matter for ROW_NUMBER. If you ever need a stable, reproducible ordering, add a tiebreaker: ORDER BY amount DESC, order_id.

NULL amounts sort to a default end

With ORDER BY amount DESC, Postgres places NULL amounts first (NULLs are largest by default in DESC), so a NULL-amount order would wrongly grab rank 1. Use ORDER BY amount DESC NULLS LAST to sink unscored rows to the bottom. Dialects differ — MySQL treats NULL as smallest — so be explicit.

No outer ORDER BY

The window orders the ranking computation, but the query has no final ORDER BY, so display order is not guaranteed. Add ORDER BY rk (or amount DESC) if the consumer expects sorted rows.

Senior signal: describe DENSE_RANK as "ties share a rank, counter advances by one, no gaps" and contrast all three functions on the same tied data — ROW_NUMBER unique, RANK skips after ties, DENSE_RANK does not — then handle NULL ordering with NULLS LAST.
