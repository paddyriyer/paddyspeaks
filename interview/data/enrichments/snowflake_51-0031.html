IBM’s e-commerce analytics team wants to isolate orders placed on weekends — a common demand-signal segmentation used in retail and logistics. The interviewer is probing whether you know how to extract a numeric day-of-week from a timestamp, which integer values represent Saturday and Sunday in the target engine, how to format it into a human-readable name, and whether you know the engine’s DOW numbering convention (0-indexed vs 1-indexed).

2025-05-12 is Monday (DOW=1), 2025-05-13 Tuesday, 2025-05-14 Wednesday, 2025-05-15 Thursday, 2025-05-16 Friday, 2025-05-17 Saturday (DOW=6), 2025-05-18 Sunday (DOW=0). The query should return only orders 1006 and 1007.

2. The SQL — EXTRACT(DOW) weekend filter with TO_CHAR label

SELECT order_id, order_ts, amount,
       TO_CHAR(order_ts::timestamp, 'Day') AS day_name,
       CASE WHEN EXTRACT(DOW FROM order_ts::timestamp) IN (0, 6) THEN 'weekend' ELSE 'weekday' END AS day_type
FROM orders
WHERE EXTRACT(DOW FROM order_ts::timestamp) IN (0, 6)
ORDER BY order_ts;

TO_CHAR(..., 'Day') returns a space-padded string (e.g. 'Saturday ') in Postgres; in Snowflake the padding may differ. The day_type column is always 'weekend' here because the WHERE clause already filters to DOW 0 or 6 — the CASE is redundant but included for display/reporting purposes.

4. Strategic explanations

Postgres/Snowflake DOW numbering: 0 = Sunday

EXTRACT(DOW FROM ...) uses the ISO-adjacent convention where Sunday = 0, Monday = 1, …, Saturday = 6. This is the Postgres convention and matches Snowflake’s EXTRACT(DAYOFWEEK_ISO) offset behavior. The key interview trap: SQL Server’s DATEPART(dw, ...) returns 1 = Sunday by default (or 1 = Monday depending on SET DATEFIRST), MySQL’s DAYOFWEEK() returns 1 = Sunday with the same mapping, while WEEKDAY() returns 0 = Monday. Knowing which engine you are on and which DOW convention it uses is the crux of this question.

The CASE WHEN is redundant given the WHERE — and that’s intentional

Because the WHERE already restricts to DOW IN (0, 6), every returned row has day_type = 'weekend'. The CASE WHEN is included for reporting completeness — downstream tools consuming this query can rely on the column existing without re-computing DOW. If the WHERE were relaxed to return all days, the CASE WHEN would correctly label each row. Interviewers sometimes ask "why is this column always ‘weekend’?" — the right answer is "because the WHERE pre-filters; the column would become meaningful if you removed the WHERE to get a full-week breakdown."

TO_CHAR for human-readable day names

TO_CHAR(ts, 'Day') is the Postgres/Snowflake idiom. It returns a space-padded fixed-width string. Use TRIM(TO_CHAR(ts, 'Day')) if you need the name without padding. Alternatives: TO_CHAR(ts, 'Dy') for abbreviated names (Mon, Tue …). The query casts order_ts::timestamp to ensure the function receives a timestamp type even if the column is stored as text.

Performance: filtering on a function

EXTRACT(DOW FROM order_ts) in the WHERE clause prevents index use on order_ts. For large tables this causes a full scan. Options to avoid it:

  • Add a generated/computed column order_dow TINYINT and index it (Postgres, MySQL, SQL Server).
  • Snowflake: use a SEARCH OPTIMIZATION service or cluster key, or maintain a materialized column.
  • Partition the table by calendar week or use a date dimension table joined on DATE(order_ts).
Dialect notes
  • Snowflake native: DAYOFWEEK(order_ts) returns 0 (Sun) – 6 (Sat) — identical to this query. Also DAYOFWEEKISO() returns 1 (Mon) – 7 (Sun).
  • BigQuery: EXTRACT(DAYOFWEEK FROM ts) returns 1 (Sun) – 7 (Sat); weekend = IN (1, 7).
  • MySQL: DAYOFWEEK(ts) = 1 (Sun) – 7 (Sat); weekend = IN (1, 7).
  • SQL Server: DATEPART(dw, ts) default = 1 (Sun) – 7 (Sat); controlled by SET DATEFIRST.
Senior signal: state the DOW numbering convention for the engine (0 = Sunday in Postgres/Snowflake), explain that the CASE WHEN is redundant but retained for reporting downstream, flag that EXTRACT(DOW FROM ...) in WHERE blocks index use on the timestamp column, and recite the cross-dialect DOW mapping table from memory — it decides correctness silently and most candidates get it wrong for at least one engine.
