A Community member models a cinema seat map: each seat has a status of occupied or empty, and the task is to find runs of consecutive occupied seats — and how many seats each run contains. A run of one does not count. This is gaps-and-islands again, and the interviewer wants the row-number trick plus a clear eye on the "more than one" boundary.

Seat 1 is occupied but isolated (seat 2 is empty) — a run of length 1. Seats 3–5 form a run of three; seats 7–8 a run of two. The interview hinges on the lone seat 1 being correctly excluded, and the two genuine runs surviving.

2. The SQL — row-number difference, then size filter

WITH t1 AS (
  SELECT seat_id, status,
         ROW_NUMBER() OVER (ORDER BY seat_id ASC) AS nw_id,
         (seat_id - ROW_NUMBER() OVER (ORDER BY seat_id ASC)) AS flg
  FROM cinema_tbl
  WHERE status::text = '1'
), t2 AS (
  SELECT *, SUM(1) OVER (PARTITION BY flg) AS seat_count
  FROM t1
)
SELECT seat_id, status, nw_id, flg, seat_count
FROM t2
WHERE seat_count > 1;

Filter to occupied seats, then subtract a dense row number from seat_id: that difference flg stays constant only while seat ids are contiguous. SUM(1) OVER (PARTITION BY flg) counts each run's size, and the final WHERE seat_count > 1 drops isolated seats.

4. Strategic explanations

Why the row-number difference isolates runs

After filtering to occupied seats only, walk their ids: 1, 3, 4, 5, 7, 8. The dense row number is 1, 2, 3, 4, 5, 6. Subtract: 0, 1, 1, 1, 2, 2. Within a contiguous block both counters step by exactly 1, so the difference is flat; the moment an empty seat creates a gap, seat_id jumps but the row number does not, and the difference increments. That constant flg is the run identifier — the canonical gaps-and-islands trick.

The filter-before-numbering subtlety

The ROW_NUMBER is computed after WHERE status = '1' — it numbers only occupied seats, 1..6. That is exactly what makes the difference work: empty seats must not consume row numbers, or the gaps would not register. If you numbered all seats first and filtered later, flg would be constant everywhere and every occupied seat would look like one giant run. Order of operations is the bug-or-not here.

The seat_count > 1 boundary — runs vs the lone seat

"Consecutive" implies at least two adjacent seats; a single occupied seat between two empties is not a run. WHERE seat_count > 1 encodes that. Confirm the definition with the interviewer though — some phrasings of this problem want all occupied seats with their run length annotated, lone seats included as runs of 1. The threshold is a one-character change (> 1 vs >= 1); getting the definition right matters more than the SQL.

The output shape question — per-seat vs per-run

This query returns one row per seat. Often the consumer wants one row per run — start seat, end seat, length:

SELECT MIN(seat_id) AS run_start,
       MAX(seat_id) AS run_end,
       COUNT(*)     AS run_length
FROM t1
GROUP BY flg
HAVING COUNT(*) > 1;

Same flg grouping, collapsed with GROUP BY and a HAVING filter. Knowing both the detail and the summarised shape — and asking which one is wanted — is the senior move.

The status type and the gap-vs-row assumption

The solution writes status::text = '1' — a defensive cast, because status might be stored as text, integer, or boolean depending on the source. Pin the actual type and compare cleanly (status = 1 or status IS TRUE). Also note "consecutive" here means consecutive seat_id values: this assumes seat ids are a dense integer sequence with no legitimate gaps. If seat ids skip numbers for non-seat reasons (an aisle), the difference trick would wrongly split a run — use ROW_NUMBER over the physical seat order instead of raw seat_id.

Senior signal: identify the seat-id-minus-row-number difference as the gaps-and-islands island key, understand that the ROW_NUMBER must be computed after filtering to occupied seats, treat seat_count > 1 as a definition to confirm (run vs lone seat), and offer the per-run summarised shape as the likely real deliverable.
