Apple poses "Pacific Atlantic Water Flow" to see whether you spot the reverse-the-search insight. A grid of cell heights; water flows from a cell to a neighbour of equal or lower height. The Pacific touches the top and left edges, the Atlantic the bottom and right. You must return every cell from which water can reach both oceans. The naive "from each cell, can it reach an ocean?" is O((R·C)²); the elegant answer flips the direction.

1. The input data

A 5×5 height map (the canonical example):

h = [
  [1, 2, 2, 3, 5],
  [3, 2, 3, 4, 4],
  [2, 4, 5, 3, 1],
  [6, 7, 1, 4, 5],
  [5, 1, 1, 2, 4],
]
# Pacific borders the top row + left column
# Atlantic borders the bottom row + right column

2. The approach

The key reframe: instead of asking "can this cell flow out to an ocean," ask "starting at an ocean, which cells can water climb back up to?" Reverse the flow rule — from an ocean border cell, move to a neighbour whose height is >= the current height.

Two sets, pac and atl. The code seeds DFS from every Pacific border cell into pac and every Atlantic border cell into atl. dfs(r, c, seen, prev) stops if the cell was already visited, is out of bounds, or — the reversed rule — h[r][c] < prev (water can't climb down when running the flow backwards). Otherwise it records the cell and recurses into all four neighbours, passing the current height as the new prev. Border seeds use prev=0 so the border cell itself always qualifies. The answer is the set intersection pac & atl — cells reachable from both.

3. The output

The cells that drain to both oceans (order is set-dependent):

(row, col)height
(0, 4)5
(1, 3)4
(1, 4)4
(2, 2)5
(3, 0)6
(3, 1)7
(4, 0)5

Seven cells. They form the ridge from which water spills toward both coasts.

4. Strategic explanations

Complexity

Time O(R·C) — each cell is visited at most once per ocean (the seen set blocks revisits), so two passes over the grid. Space O(R·C) for the two reachability sets plus recursion depth. This beats the brute-force O((R·C)²) of searching outward from every cell independently.

The idiomatic alternative

A multi-source BFS with deque per ocean is equally valid and dodges the recursion limit — seed the queue with all border cells at once. The intersection step itself is already idiomatic Python: list(pac & atl) is far cleaner than nested loops checking two boolean grids. Comprehensions can build the border seeds: [(r,0) for r in range(R)].

The edge case

if not h: return [] covers an empty grid. A 1×1 grid: the single cell borders all four sides, so it reaches both oceans → returned. A grid that is a single row or column: every cell touches both a Pacific and an Atlantic edge, so all cells qualify. Equal-height plateaus flow freely in the reversed search because the rule is >=, not strict >.

The follow-up

"Return the count of such cells" (trivial: len(pac & atl)) or "what if water also flows diagonally?" — add four diagonal deltas to the direction list. A deeper one: "stream the grid, you can't hold it all" — this breaks the in-memory set approach and pushes toward a tiled or external algorithm.

The trap

The fatal mistake is searching in the natural flow direction — from each cell, DFS downhill looking for an ocean. That recomputes overlapping paths for every start cell, giving O((R·C)²) and timing out on large grids. The whole point of the question is to invert the search: start at the oceans, climb uphill, intersect. Miss the reversal and you've missed the question.

Senior signal: reverses the flow — searches uphill from the oceans rather than downhill from every cell — turning O((R·C)²) into O(R·C), expresses the final answer as a clean set intersection, gets the >= climb rule and prev=0 border seeding exactly right, and can swap DFS for multi-source BFS when recursion depth is a concern.
