Citi's supply-chain analytics group runs a nightly replenishment check: which products have fallen to or below their reorder point and need a purchase order raised. The logic is a single comparison, but the reference solution wraps both columns in regex-scrubbing because the inventory feed delivers quantities as untrusted text — and the comparison must be numeric, not lexical.

Product 3 sits exactly at its reorder point (10 = 10) — the boundary case that decides whether the comparison is < or <=. Product 4 is fully depleted. Products 2 and 5 are comfortably stocked. The quantities are stored as text, which is the trap.

2. The SQL — numeric comparison after scrubbing

SELECT product_id FROM inventory
WHERE NULLIF(regexp_replace(qty_on_hand::text, '[^-.0-9]', '', 'g'), '')::numeric
   <= NULLIF(regexp_replace(reorder_point::text, '[^-.0-9]', '', 'g'), '')::numeric;

regexp_replace(…, '[^-.0-9]', '', 'g') strips any character that is not a digit, dot, or minus — so "10 units" becomes "10". NULLIF(…, '') turns a string that scrubbed down to nothing into NULL, then ::numeric casts to a real number for the comparison.

4. Strategic explanations

The boundary — <= vs < is a business decision

Product 3 appears only because the comparison is <=. The reorder point is the level at which you should reorder, so "stock equal to the reorder point" almost always means "reorder now" — <= is the right call. But it is a genuine policy choice: with <, product 3 waits until stock drops to 9. Confirm it: "does hitting the reorder point exactly trigger a PO, or only dropping below it?" An off-by-one here means either premature orders or stockouts.

The string-comparison trap — why the numeric cast is essential

This is the heart of the question. If qty_on_hand and reorder_point are text, comparing them directly does a lexical comparison, character by character. Lexically, '9' > '10' is true — '9' sorts after '1' — so a product with 9 units would be judged above a reorder point of 10 and the PO would never fire. String ordering is not numeric ordering. The ::numeric cast forces a real arithmetic comparison: 9 < 10. Diagnosing that "text columns compared directly give wrong answers" is the senior insight this question is built around.

Why regexp_replace and the NULLIF guard

A bare qty_on_hand::numeric throws invalid input syntax for type numeric the moment a value contains a stray unit suffix or whitespace — and one bad row aborts the whole query. regexp_replace salvages the digits first. The NULLIF(…, '') is not optional: a value like "unknown" scrubs to the empty string, and ''::numeric is itself an error. NULLIF turns that empty string into NULL, which casts cleanly. Skipping it reintroduces the very crash you were defending against.

The NULL outcome — a NULL-quantity product is silently excluded

If a product's qty_on_hand is NULL (or scrubs to NULL), the comparison NULL <= reorder_point evaluates to unknown, and WHERE keeps only true rows — so that product is dropped from the replenishment list. That is dangerous: a product with unknown stock is arguably the one most needing attention, yet it silently disappears. Surface it instead:

SELECT product_id,
  CASE WHEN qty IS NULL THEN 'Unknown stock — investigate'
       WHEN qty <= rop THEN 'Reorder'
       ELSE 'OK' END AS status
FROM ( ... cleaned columns ... ) c;
If the columns are already numeric, the query collapses

The schema does not state the type. If qty_on_hand and reorder_point are proper integer/numeric columns, every scrap of regex and casting is dead weight and the query is simply:

SELECT product_id FROM inventory WHERE qty_on_hand <= reorder_point;

The defensive form earns its complexity only against a dirty text feed. Stating that trade-off — and asking what the column type actually is — is the over-the-bar move.

Senior signal: pin down whether the trigger is <= or < at the reorder point, explain that comparing text columns does a lexical comparison where '9' > '10' — so the numeric cast is correctness, not cosmetics — justify the regexp_replace/NULLIF pair as crash protection, and flag that a NULL-stock product is silently dropped when it most needs review.
