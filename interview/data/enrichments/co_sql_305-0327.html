Uber's Marketplace team has one number on the wall: unfilled rate = % of ride requests that didn't match a driver. High unfilled rate means lost revenue + frustrated rider + drift to Lyft. The query is conditional aggregation; the senior layer is which areas consistently spike vs which are episodic.

1. The source data — ride_requests

2. The SQL — fulfillment rate per area

SELECT area_id,
  COUNT(*)                                       AS total_requests,
  SUM(CASE WHEN fulfilled = 1 THEN 1 ELSE 0 END) AS fulfilled,
  SUM(CASE WHEN fulfilled = 0 THEN 1 ELSE 0 END) AS unfilled,
  ROUND(100.0 * SUM(CASE WHEN fulfilled = 0 THEN 1 ELSE 0 END)
              / NULLIF(COUNT(*), 0), 1) AS unfill_rate_pct
FROM ride_requests
GROUP BY area_id
ORDER BY unfill_rate_pct DESC;

3. Strategic explanations

Why three buckets — fulfilled, unfilled, total

Reporting raw rate alone (e.g. "12% unfilled") loses the volume signal. An area with 1000 requests and 12% unfilled is a real problem. An area with 50 requests and 12% unfilled is statistical noise. Always emit the volume so the reader can weight the rate.

The NULLIF(COUNT(*), 0) safety net

If GROUP BY produces an area with no requests (shouldn't happen if filtered correctly), division by zero. NULLIF turns 0 into NULL, division returns NULL, dashboard renders "—" gracefully.

Episodic vs sustained — the senior diagnostic

One-day spike (concert lets out) needs no marketplace intervention — fix it next time. Sustained week-over-week unfill rate is a structural supply gap. The query above doesn't separate them; the time-series version does:

SELECT area_id, DATE(requested_at) AS day,
  ROUND(100.0 * SUM(CASE WHEN fulfilled=0 THEN 1 ELSE 0 END) / COUNT(*), 1) AS rate
FROM ride_requests
GROUP BY area_id, day
ORDER BY area_id, day;

An area where rate > 10% for 5+ consecutive days is "sustained" — that's the driver-incentive trigger.

The hour-of-day cut

Rush hour 5-7pm has structurally higher unfill than 11am offpeak. Don't compare across hours without normalizing — slice by hour:

GROUP BY area_id, CAST(strftime('%H', requested_at) AS INTEGER)
The "unfilled because" decomposition

An unfilled request can be:

  • No driver online — fix: incentive supply
  • All drivers busy — fix: surge to throttle demand or attract new supply
  • Driver declined — fix: matching algorithm tweak (route longer rides to nearer drivers)
  • Rider cancelled before match — fix: faster matching latency

Each diagnosis needs different data: drivers_online_at_request, match_attempts, cancel_reason. The unfilled rate alone is the symptom; the breakdown is the prescription.

The action threshold

Production ops dashboards page when unfill_rate > 8% for 30 minutes in a top-10 area. The query result feeds the alerting; the action is operations dispatching driver bonuses to the area within 15 minutes.

Senior signal: emit volume alongside rate, distinguish episodic from sustained, and decompose "unfilled" into actionable subcategories (no driver / all busy / declined / pre-match cancel). Hour-of-day normalization is the over-the-bar slice.
