DoorDash's #1 marketplace metric: dasher utilization = time-on-active-delivery ÷ time-online. High utilization = efficient marketplace but tired drivers; low = supply slack but lost income.

2. The SQL — julianday-based time ratio

SELECT s.dasher_id,
  ROUND(SUM(julianday(s.offline_at) - julianday(s.online_at)) * 24.0, 2) AS hours_online,
  ROUND(SUM(julianday(d.ended_at) - julianday(d.started_at)) * 24.0, 2) AS hours_delivering,
  ROUND(100.0 * SUM(julianday(d.ended_at) - julianday(d.started_at))
              / NULLIF(SUM(julianday(s.offline_at) - julianday(s.online_at)), 0), 1) AS utilization_pct
FROM dasher_shifts s
LEFT JOIN deliveries d ON d.dasher_id = s.dasher_id
GROUP BY s.dasher_id
ORDER BY utilization_pct DESC;

3. Strategic explanations

Three utilization tiers — pick the right one
tierwhat it counts
P0 — restaurant-to-customerstart-of-pickup → drop-off (this query)
P1 — accepted-to-drop-offP0 + drive-to-restaurant
P2 — engagedP1 + post-drop-off-rebalance time

P0 is the strictest. Production reports all three.

The 50% target

DoorDash internally targets 50% P0 utilization. Below 35% drivers earn too little; above 65% drivers burn out. The query result feeds the dispatch algorithm's load balancing.

Multi-stop bonus

Stacked deliveries (multiple orders, one trip) inflate utilization unfairly because ended_at includes all stops. Production breaks per-leg.

Per-hour utilization

Lunch peak 11-2 has higher utilization than 3-5 trough. Comparing dashers across hours is unfair. Slice:

GROUP BY dasher_id, CAST(strftime('%H', online_at) AS INTEGER)
Senior signal: name P0/P1/P2 tiers, pin the 50% target with directional implications, raise multi-stop double-count, and stratify by hour for fair comparison.
