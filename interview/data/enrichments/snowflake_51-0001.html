A streaming platform's content team asks: "Show us every published video that has crossed the one-million-view threshold, ranked by popularity." The interviewer is testing whether the candidate can correctly combine a categorical equality filter with a numeric range filter — and whether they default to AND over OR — while also noting that status is an application-controlled enum that should never be trusted to be clean without validation.

The table has three filtering traps: video 3 has high views but is a draft (must be excluded); video 6 is over the threshold but archived (excluded); video 4 is published but under the view threshold (excluded). Only the intersection of both conditions qualifies.

2. The SQL — dual-predicate filter with descending sort

SELECT video_id, title, views
FROM videos
WHERE status = 'published'
  AND views > 1000000
ORDER BY views DESC;

Two predicates joined with AND: both must be true. The result set is sorted descending on views so the most popular video appears first.

4. Strategic explanations

Why AND not OR — the most common beginner slip

A candidate who writes WHERE status = 'published' OR views > 1000000 would include the 5M-view draft (video 3) and the 3M-view archived concert (video 6). The business requirement is the intersection of popular and live, which maps to AND. Explicitly naming that the predicate expresses a set intersection is the first thing a senior interviewee says.

The threshold: strict > vs inclusive >=

The query uses views > 1000000 (strict greater-than). A video with exactly 1,000,000 views does not qualify. The word "popular" in the prompt is ambiguous — a senior candidate clarifies the boundary before writing any SQL. If the spec changes to "at least one million," the fix is >=. Noting this in an interview signals that you read requirements precisely.

Index strategy

On a large table, a composite index (status, views DESC) lets the engine seek directly to the 'published' bucket and scan only the rows with views above the threshold in reverse order, potentially satisfying ORDER BY views DESC without a separate sort step. In Snowflake, micro-partition pruning is more relevant: clustering the table on status (or adding a cluster key) lets Snowflake skip non-published micro-partitions entirely before evaluating the numeric filter.

Snowflake vs other dialects

The query is ANSI SQL and runs identically on Snowflake, PostgreSQL, BigQuery, and MySQL. In BigQuery you might add a LIMIT for cost control on very large tables. In Snowflake, BIGINT and NUMBER(38,0) are both valid for view counts; the comparison views > 1000000 is safe with either type since the literal is an integer. If views were stored as VARCHAR (a schema smell), you'd need an explicit cast before the numeric comparison.

Extending the query

Real content pipelines extend this shape in predictable ways: add AND uploaded_at >= DATEADD('day', -30, CURRENT_TIMESTAMP) to restrict to recent videos; use RANK() OVER (ORDER BY views DESC) to assign a public leaderboard position; or replace the hardcoded threshold with a subquery that computes the 90th-percentile view count from the same table. All of these are natural follow-up prompts an interviewer may give after the baseline.

Senior signal: flag the AND/OR distinction as a set-intersection vs set-union choice, clarify whether the threshold is strict or inclusive before writing code, and describe the indexing or Snowflake clustering strategy that avoids a full-table scan on a production-scale videos table.
