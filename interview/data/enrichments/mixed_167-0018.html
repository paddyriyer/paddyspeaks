An e-commerce analyst wants each customer's biggest order: "return the full order row with the highest amount for every customer." The reference solution is a correlated subquery — for each row, compare its amount to that customer's own maximum. It is correct, but the interviewer is watching for the tie behaviour and whether you can name the modern window-function alternative.

1. The source data — orders

Customer 101 has two orders tied at $400 — the edge case. Whether the answer returns one row or both is the entire interview question.

2. The SQL — correlated subquery on MAX

SELECT * FROM orders o
WHERE amount = (SELECT MAX(amount) FROM orders
                WHERE customer_id = o.customer_id);

4. Strategic explanations

The correlated subquery runs per customer-row

The inner query references o.customer_id from the outer query, so it is correlated — conceptually re-evaluated for each outer row, computing that customer's maximum. The outer row survives only if its amount equals that maximum. This is the pre-window-function idiom for "rows matching a per-group extreme."

The tie behaviour — = MAX returns ALL ties

This is the point the interviewer wants. Because the filter is an equality against the max value, every row equal to that value passes. Customer 101's two $400 orders both qualify. That may be exactly what you want ("show me all top orders") or a bug ("I expected one row per customer"). Contrast with ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY amount DESC) filtered to = 1, which picks exactly one row per customer and breaks the tie arbitrarily. RANK() = 1 behaves like the subquery — it keeps all ties. Naming which semantics the product needs is the senior move.

The window-function rewrite — one table scan

The subquery effectively scans orders twice. A window function does it in one pass:

SELECT order_id, customer_id, amount, order_date, status
FROM (
  SELECT *, RANK() OVER (PARTITION BY customer_id ORDER BY amount DESC) AS rk
  FROM orders
) t
WHERE rk = 1;

Swap RANK for ROW_NUMBER if you want strictly one row per customer. On large tables this is both clearer and faster than the correlated form.

NULL amounts and the empty-group case

If a customer's orders all have NULL amount, MAX returns NULL, and amount = NULL is unknown — that customer contributes no rows at all. Also, a row with a NULL amount can never equal a non-NULL max, so it is silently dropped. Decide whether NULL-amount orders should be excluded upstream or coalesced.

SELECT * caveat

Returning every column is fine for a demo. In production, name the columns — SELECT * in a subquery-filtered statement is brittle if the schema changes and ships unnecessary data.

Senior signal: recognise that amount = (SELECT MAX ...) returns every tied row, not one per customer; state that this matches RANK() = 1 but differs from ROW_NUMBER() = 1; and offer the single-scan window rewrite while clarifying with the interviewer whether ties should be kept.
