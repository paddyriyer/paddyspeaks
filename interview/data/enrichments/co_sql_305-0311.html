Netflix plans cap simultaneous streams — Basic-with-Ads = 1, Standard = 2, Premium = 4. The famous 2023 "password sharing crackdown" required answering one query at scale: which accounts exceeded their cap, ever? Each session has a start and end, and the question becomes "how many were active at the same instant?" — the canonical sweep-line concurrency problem.

From 19:28 to 19:30 there are 5 concurrent streams on this account — one over the Premium cap of 4.

2. The SQL — sweep-line via UNION ALL of start/end events

WITH events AS (
  SELECT account_id, started_at AS t, +1 AS delta FROM streams
  UNION ALL
  SELECT account_id, ended_at,           -1        FROM streams
),
active AS (
  SELECT account_id, t,
    SUM(delta) OVER (
      PARTITION BY account_id
      ORDER BY t
      ROWS UNBOUNDED PRECEDING
    ) AS concurrent
  FROM events
)
SELECT DISTINCT account_id
FROM active
WHERE concurrent > 4;

3. Strategic explanations

The sweep-line algorithm in 6 lines

This is one of the most elegant patterns in SQL. The mental model:

  1. For each session, emit two events: +1 at start, -1 at end. Now you have an event stream with deltas.
  2. Sort by time within each account.
  3. Running cumulative sum of the deltas = active count at any instant.
  4. Filter where the cumulative ever exceeded the cap.

The same shape solves any "how many simultaneous?" question: emergency rooms (patients in vs out), parking lots (cars in vs out), thread pools, lock contention.

The tie problem — ends and starts at the same timestamp

If session A ends at 19:30:00 exactly when session B starts, do they overlap? Half-open intervals (industry standard) say no. But sorting by t alone is non-deterministic when ties exist. Force ends before starts:

UNION ALL
SELECT account_id, started_at AS t, 1 AS delta, 1 AS sort_priority FROM streams
UNION ALL
SELECT account_id, ended_at,    -1,           0 FROM streams
-- then ORDER BY t, sort_priority
ORDER BY t, sort_priority

Ends process first → the count drops then the new session bumps it — correct half-open semantics. Without this fix, the count momentarily over-counts at the boundary and you get false positives.

Per-plan cap — join to the plan-tier table

The "> 4" hardcode is sloppy — different plans have different caps. Production form:

SELECT DISTINCT a.account_id
FROM active a
JOIN accounts ac USING (account_id)
JOIN plans p ON p.tier = ac.tier
WHERE a.concurrent > p.stream_cap;
The reverse query — max concurrent per account

Often more useful than the binary "exceeded" flag is the actual peak:

SELECT account_id, MAX(concurrent) AS peak_concurrent_streams
FROM active GROUP BY account_id ORDER BY peak_concurrent_streams DESC;

Distribution: 95% of accounts peak at 1, the next 4% at 2-3, then a fat tail of 5+ that drives the password-sharing campaign.

Why ROWS UNBOUNDED PRECEDING not the default RANGE

With RANGE default frame, ties on t get summed together — the running total sees both a +1 and -1 of a tie at once, and you can't observe the momentary peak. ROWS processes one event at a time, exactly what sweep-line needs.

The "duration over cap" follow-up

"Did the cap get exceeded at all?" is binary. "For how long was it exceeded?" is the policy-relevant version — a 2-second blip during plan-change vs 3 hours of household sharing are very different:

WITH spans AS (
  SELECT account_id, t, concurrent,
    LEAD(t) OVER (PARTITION BY account_id ORDER BY t) - t AS duration
  FROM active
)
SELECT account_id,
  SUM(CASE WHEN concurrent > 4 THEN duration END) AS over_cap_duration
FROM spans GROUP BY account_id
HAVING SUM(CASE WHEN concurrent > 4 THEN duration END) > INTERVAL '5 minutes';
Geo-aware sharing detection

Two streams from the same account with one in NYC and one in LA, simultaneously, is the canonical password-sharing signal — geographically impossible for one household. Add IP/geo to the events:

WITH events AS (
  SELECT account_id, location_state AS state, started_at AS t, +1 AS delta FROM streams
  UNION ALL
  SELECT account_id, location_state, ended_at, -1 FROM streams
)
-- Then for each account, look at moments where 2+ states were active.
The "household" exemption complexity

Netflix's TOS allows multiple users in one household — defined by IP-bucket / device-association / fingerprint. The query above flags everyone, including legitimate households. Production filtering exempts:

  • Streams from the "primary location" (the IP that paid for the plan)
  • Devices registered as "household devices" via MAC/fingerprint
  • Sessions during travel (verified by login flow)
The "4-stream-burst" false positive

A user pausing then resuming creates a new session — not 4 distinct viewers. Production data quality dedupes within a 30-second window before sweep-line.

Senior signal: name the sweep-line algorithm crisply (+1/-1 events, cumulative sum), pin the tie-breaking via the end-before-start sort priority, and pick ROWS UNBOUNDED PRECEDING over the default RANGE frame. The duration-over-cap follow-up and geo-aware household-exemption logic connect the query directly to the password-sharing crackdown decisioning — over-the-bar.
