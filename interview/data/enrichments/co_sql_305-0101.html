IBM's internal data-platform team monitors who is using the SQL console — for capacity planning and to spot analysts who have gone dormant. The ask: per employee, how many queries they have run and when they last ran one. It is a two-aggregate GROUP BY, and the interview probes whether you know what a single-row activity log can and cannot tell you.

Employee 2 has run exactly one query — their query_count is 1 and their last_query is that same lone event. Employee 3's first query is back in March, a long gap before April activity. These are the rows that test the edges.

2. The SQL — COUNT + MAX per group

SELECT emp_id, COUNT(*) AS query_count, MAX(query_at) AS last_query
FROM queries GROUP BY emp_id;

One row per employee; COUNT(*) is total queries run, MAX(query_at) is the most recent timestamp.

Employee 2's single query produces a clean query_count = 1 — COUNT and MAX handle a one-row group with no special case. Employees 1 and 3 both ran 3 queries, but MAX picks each one's latest correctly. Rows come back in no particular order — add ORDER BY last_query DESC for a human reader. Note what is missing: an employee who has never opened the console has zero rows here and is invisible.

4. Strategic explanations

The dormant analyst is the whole point — and they are missing

queries only contains employees who have run at least one query. The team's actual goal — spotting people who have gone dormant — cannot be served by this query, because someone with zero queries simply has no group. To include them, drive from the employees table:

SELECT e.emp_id,
       COUNT(q.query_at) AS query_count,   -- COUNT(col): 0 when no match
       MAX(q.query_at)   AS last_query     -- NULL when no match
FROM employees e
LEFT JOIN queries q ON q.emp_id = e.emp_id
GROUP BY e.emp_id;

Critically, use COUNT(q.query_at) not COUNT(*): with a LEFT JOIN, an employee with no queries still produces one row (all-NULL on the right), so COUNT(*) would report 1 for someone who ran nothing. COUNT(column) ignores the NULL and correctly returns 0. Naming this is a senior signal.

"Dormant" needs a clock — MAX alone is not a status

last_query is a timestamp, not a verdict. To flag dormancy, compare it to now:

SELECT emp_id, MAX(query_at) AS last_query,
       CURRENT_DATE - MAX(query_at)::date AS days_idle
FROM queries
GROUP BY emp_id
HAVING CURRENT_DATE - MAX(query_at)::date > 30;

That turns the raw MAX into the actual business question. Mentioning that the deliverable is "who is dormant", not "what is everyone's max timestamp", shows you read the intent.

COUNT(*) counts queries, not active days

COUNT(*) is total query volume. "Activity" might better mean distinct days active — an analyst who logs in daily is more engaged than one who ran 50 queries in a single afternoon. That metric is COUNT(DISTINCT query_at::date). Different question, different number; worth clarifying which the capacity model wants.

First-seen, span, and bot accounts

Two natural follow-ups. Add MIN(query_at) AS first_query for tenure, and the active span is MAX - MIN. Also watch for non-human emp_ids — service accounts or scheduled jobs can dominate query_count and distort "who is using the console"; you may need to filter them out.

Senior signal: recognise that employees with zero queries are absent — reach for a LEFT JOIN from the employees table with COUNT(column) (never COUNT(*), which returns 1 for an unmatched row), and turn the raw MAX(query_at) into the dormancy verdict the team actually asked for by comparing it against the current date.
