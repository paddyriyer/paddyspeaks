A directory export needs each employee's full name from separate first_name and last_name columns. The query uses the SQL string-concatenation operator ||. It is a one-liner, but the interviewer is watching for one specific thing: what happens to the result when one of the name parts is NULL.

1. The source data — employees

Carol's last_name is NULL. That single missing value is the entire interview — it does not just blank one field.

2. The SQL — the || concatenation operator

SELECT first_name || ' ' || last_name AS full_name
FROM employees;

3. The resulting analysis

Carol's row is the lesson: her first name is "Carol", perfectly valid — yet full_name is entirely NULL. A single NULL operand poisoned the whole expression. Her name vanished from the directory.

4. Strategic explanations

Why one NULL nulls the entire concatenation

In ANSI SQL, || with any NULL operand yields NULL — NULL is "unknown," and unknown joined to anything is still unknown. 'Carol' || ' ' || NULL evaluates to NULL, not to "Carol ". This is the single most common surprise with string building: the failure is silent and total, not partial.

The fix — coalesce every nullable part

Wrap each nullable column in COALESCE so a missing piece degrades to empty string instead of poisoning the row:

SELECT COALESCE(first_name, '') || ' ' || COALESCE(last_name, '') AS full_name
FROM employees;

Carol now reads "Carol " — with a trailing space, because the separator is still emitted. To drop the orphan space, wrap the whole thing: TRIM(COALESCE(first_name,'') || ' ' || COALESCE(last_name,'')).

The cleaner answer — CONCAT_WS

CONCAT_WS(' ', first_name, last_name) solves both problems at once: it skips NULL parts and does not emit a separator for a skipped part. Carol becomes "Carol" with no trailing space, no coalescing, no trim. When concatenating optionally-missing fields with a delimiter, CONCAT_WS is the senior default; reserve raw || for cases where every operand is guaranteed non-NULL.

Dialect notes — || is not universal

|| is the SQL-standard concatenation operator and works in PostgreSQL, Oracle, SQLite, and DB2. MySQL by default treats || as logical OR — 'a' || 'b' returns 0, not 'ab' — unless PIPES_AS_CONCAT is set in sql_mode; use CONCAT() there. SQL Server uses + for string concat (and + with NULL also yields NULL unless CONCAT_NULL_YIELDS_NULL is off). Know your engine.

Common mistakes

Assuming || behaves like MySQL CONCAT (which treats NULL as ''). Forgetting the literal space separator and getting "AliceNg". And ordering: this query has no ORDER BY, so the directory rows come back in physical-storage order — add ORDER BY last_name, first_name for a real export.

Senior signal: immediately flag that || with one NULL operand nulls the entire name, fix it with COALESCE per part or — better — CONCAT_WS, and note that || means OR in default-mode MySQL.
