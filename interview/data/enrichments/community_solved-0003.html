A Community-sourced org-chart question: given one employee, walk up the reporting chain and list every manager above them. It is the canonical recursive-CTE warm-up, but the interviewer is watching two things — do you anchor on the right row, and do you understand why the chain stops on its own when it reaches the CEO.

1. The source data — emp_details

idnamemanager_iddesignation
1jia2developer
2rohan3team lead
3priya4manager
4amitNULLdirector

The table is a self-referencing tree: each row points at its boss through manager_id. The interesting row is amit — manager_id is NULL, the signal "top of the org, nobody above me". That NULL is what makes the recursion terminate.

2. The SQL — WITH RECURSIVE walking up

WITH RECURSIVE cte AS (
    SELECT id, name, manager_id FROM emp_details WHERE name = 'jia'
    UNION ALL
    SELECT ed.id, ed.name, ed.manager_id
    FROM cte
    JOIN emp_details ed ON cte.manager_id = ed.id
)
SELECT cte.*, emp.name AS manager_name
FROM cte
JOIN emp_details emp ON cte.manager_id = emp.id;

The anchor selects jia. Each recursive step takes the current row's manager_id and fetches that employee — one hop up the tree per iteration. The final SELECT joins each row back to emp_details to print the manager's name alongside the id.

3. The resulting analysis

idnamemanager_idmanager_name
1jia2rohan
2rohan3priya
3priya4amit

Three rows: jia reports to rohan, rohan to priya, priya to amit. Notice amit himself is absent from the output even though the recursion did visit his row. The recursive step produced amit's row with manager_id = NULL; the final JOIN ... ON cte.manager_id = emp.id then drops it, because NULL never equals anything. The query lists "jia and her managers", not "jia, her managers, and the CEO's empty slot".

4. Strategic explanations

Why the recursion terminates

A recursive CTE stops when the recursive term returns zero new rows. After visiting amit (manager_id = NULL), the next iteration runs JOIN emp_details ed ON cte.manager_id = ed.id with cte.manager_id = NULL — the join condition is unknown, matches nothing, and the CTE closes. The NULL at the root is not a nuisance; it is the natural stop condition. If every employee had a non-NULL manager (a cycle, or a self-reference), the query would loop forever — see the cycle trap below.

The interviewer follow-up: include the employee's own depth

"How many levels up is each manager?" Carry a counter through the recursion:

WITH RECURSIVE cte AS (
    SELECT id, name, manager_id, 0 AS lvl
    FROM emp_details WHERE name = 'jia'
    UNION ALL
    SELECT ed.id, ed.name, ed.manager_id, cte.lvl + 1
    FROM cte JOIN emp_details ed ON cte.manager_id = ed.id
)
SELECT * FROM cte;

jia is level 0, rohan 1, priya 2, amit 3. The same counter trick answers "stop after N levels" — add WHERE cte.lvl < 3 to the recursive term.

The cycle trap — dirty hierarchy data

If the data is corrupt — A manages B and B manages A — the recursion never sees a NULL and loops until the engine aborts. Postgres lets you guard with CYCLE:

WITH RECURSIVE cte AS ( ... )
  CYCLE id SET is_cycle USING path
SELECT * FROM cte WHERE NOT is_cycle;

Or carry an array path and add WHERE ed.id <> ALL(cte.path) to the recursive term. Mentioning that real org tables do contain accidental cycles is a senior tell.

Anchoring by name vs. id

The anchor uses WHERE name = 'jia'. Names are not unique — two employees called "jia" would seed the recursion with both chains, silently merging them. In production, anchor on the primary key (WHERE id = 1); filter by name only in a throwaway analysis.

Walking down instead of up

Reverse the join to get jia's reports rather than her managers: anchor on the target manager, then JOIN emp_details ed ON ed.manager_id = cte.id. Same skeleton, opposite direction — knowing the join flip is the difference between "my chain of command" and "my whole sub-org".

Senior signal: explain that the NULL root is the termination condition, carry a level counter for depth-limited queries, anchor on the primary key rather than a non-unique name, and pre-empt the infinite loop that a cyclic hierarchy would cause with a CYCLE clause or path array.
