Red Hat's compensation team wants a per-department salary summary — total payroll, average pay, and headcount — for a budget review. It is a textbook GROUP BY with three aggregates. The interview is in the details: how the average treats NULL salaries, why COUNT(*) and COUNT(salary) can disagree, and what a single-person department does to the numbers.

1. The source data — employees

Department 30 has a single employee — its "average" is just that one salary, and the budget reviewer should know the figure is one data point, not a mean of many. Departments 10 and 20 each have two. That lone-member group is the edge case worth surfacing.

2. The SQL — three aggregates over dept_id

SELECT dept_id, SUM(salary) AS total, AVG(salary) AS avg, COUNT(*) AS headcount
FROM employees GROUP BY dept_id;

One row per dept_id. SUM totals payroll, AVG means it, COUNT(*) counts rows. Note avg is a SQL reserved word used as an alias here — legal but worth flagging.

Department 10 totals 200K across two people (avg 100K); department 30's "average" of 60K is simply Eve's salary — a mean of one. The rows arrive unordered (no ORDER BY), and the trailing zeros are numeric display precision; round in the presentation layer.

4. Strategic explanations

The COUNT(*) vs COUNT(salary) trap

The query uses COUNT(*) for headcount — correct, it counts people. But if any employee had a NULL salary, COUNT(*) still counts them while AVG(salary) and SUM(salary) silently skip them. So total / headcount would not equal the reported avg: the average divides by the count of non-NULL salaries, headcount divides by all rows. If you want the count that matches the average, that is COUNT(salary). Reporting both is the honest move:

SELECT dept_id, SUM(salary) AS total, AVG(salary) AS avg_salary,
       COUNT(*) AS headcount, COUNT(salary) AS paid_headcount
FROM employees GROUP BY dept_id;
NULL salary — skipped, not zero

This is the same NULL story from the other side. A NULL salary contributes nothing to SUM and is excluded from AVG's denominator — it does not pull the average toward zero. If a NULL salary should be treated as $0 (e.g. an unpaid intern), fold it in explicitly with SUM(COALESCE(salary,0)) and AVG(COALESCE(salary,0)). If a NULL means "unknown / pending", skipping it is right — but say which interpretation you chose.

Employees with no department

If an employee has dept_id IS NULL, they form their own group: GROUP BY treats NULL as a single distinct grouping value, so the result gets a row with a NULL dept_id. A budget review usually wants those flagged ("unassigned") rather than silently bucketed. Decide whether to WHERE dept_id IS NOT NULL, or COALESCE(dept_id, -1) into an explicit "unassigned" bucket.

The follow-up — department name, not just an ID

dept_id = 30 means nothing to a budget reviewer. The natural extension joins a departments table for the name. The senior detail: use a LEFT JOIN so a department with zero employees still appears (with NULL aggregates) — an inner join silently hides empty departments, which a headcount report specifically needs to see.

Single-member groups and integer division

Department 30's average being a single salary is not wrong, but a reviewer may misread it as a population mean — consider showing headcount prominently so a "1" is visible. Separately, if salary were an INTEGER column, AVG still returns numeric in Postgres so no truncation here — but a hand-rolled SUM(salary)/COUNT(*) on integer columns would truncate; another reason to use AVG directly.

Senior signal: distinguish COUNT(*) (people) from COUNT(salary) (the average's true denominator) and report both when NULL salaries are possible, decide deliberately whether a NULL salary is "skip" or "$0", flag NULL-dept_id employees rather than silently grouping them, and LEFT JOIN the departments table so empty departments still surface.
