Dashers can reject offered orders. Restaurants in tier-3 areas (low population density, far from kitchens) get higher rejection rates. The query exposes the disparity for marketplace investment decisions.

2. The SQL — rejection rate per tier

SELECT tier,
  COUNT(*) AS offers,
  SUM(CASE WHEN status = 'rejected' THEN 1 ELSE 0 END) AS rejected,
  ROUND(100.0 * SUM(CASE WHEN status = 'rejected' THEN 1 ELSE 0 END)
              / NULLIF(COUNT(*), 0), 1) AS rejection_rate_pct
FROM order_offers
GROUP BY tier
HAVING COUNT(*) >= 3
ORDER BY rejection_rate_pct DESC;

The reject-reason layer

"Rejected" hides:

  • Pay too low for distance
  • Restaurant too far for drive-time
  • Item-type unwanted (alcohol/large)
  • Tip too low

Production splits by reason for actionable diagnosis.

The downstream effect — customer wait

Each rejection adds ~3 min to customer wait while DoorDash searches for next dasher. Tier-3 with 50% reject rate → cumulative delays. Operational target: keep reject rate < 25% globally.

Pay-bonus as a counter-lever

DoorDash boosts Peak Pay in high-reject tiers to incentivize acceptance. The query's tier-rate feeds the dynamic pay-tuning algorithm.

Senior signal: tier as supply-heterogeneity proxy, decompose "rejected" into actionable reasons, name the 25% global target, and tie to pay-bonus tuning.
