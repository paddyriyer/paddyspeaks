Walmart's store-operations team tracks "average transaction value" (ATV) per store — a core retail KPI. The query is a single AVG with a GROUP BY. The interviewer is not testing the syntax; they are testing whether you know what AVG does with a NULL amount, whether refunds belong in the average, and what type the result comes back as.

1. The source data — transactions

Store 1 has three transactions, but one is a refund with a NULL amount. That row decides whether store 1's ATV is averaged over 2 transactions or 3 — and the answer is not obvious.

2. The SQL — AVG(amount) per store

SELECT store_id, AVG(amount) AS atv
FROM transactions GROUP BY store_id;

One group per store; AVG(amount) is the mean transaction amount within each group.

3. The resulting analysis

store_idatv
260.0000000000000000
1150.0000000000000000

Store 1's ATV is 150, not 100. AVG ignored the NULL refund row entirely: it summed 100 + 200 = 300 and divided by 2, the count of non-NULL amounts — not by 3. Store 2 is a clean (50 + 70) / 2 = 60.

4. Strategic explanations

AVG silently skips NULLs — the central fact

AVG(amount) is SUM(amount) / COUNT(amount), and COUNT(column) counts only non-NULL values. The NULL refund is dropped from both numerator and denominator. Usually that is right — an unpriced row should not drag the average toward zero — but it must be a conscious choice. If a NULL should count as a $0 transaction:

SELECT store_id, AVG(COALESCE(amount, 0)) AS atv
FROM transactions GROUP BY store_id;
-- store 1: (100 + 200 + 0) / 3 = 100

Same column, same table, a 50-point swing in the KPI — purely from how one NULL is treated.

The AVG vs SUM / COUNT(*) trap

A tempting "equivalent" rewrite is wrong whenever NULLs exist:

SELECT store_id, SUM(amount) / COUNT(*) AS atv
FROM transactions GROUP BY store_id;
-- store 1: 300 / 3 = 100, NOT 150

COUNT(*) counts rows (3); AVG divides by non-NULL values (2). They agree only when the column has no NULLs. If you ever hand-roll a mean, pair SUM(col) with COUNT(col) — never COUNT(*).

Should refunds be in the average at all?

Even setting the NULL aside: a refund is not a sale. ATV almost always means the average sale value. The refund row should probably be filtered out, not just averaged around:

SELECT store_id, AVG(amount) AS atv
FROM transactions
WHERE txn_type = 'sale'
GROUP BY store_id;

And if refunds carry real negative amounts elsewhere in the data, leaving them in would understate ATV. Confirm the definition — "sales only, or all transactions?" — before writing the query. The filter matters more than the aggregate.

Result type and rounding

Over a numeric column, AVG returns a high-scale numeric — hence the trailing zeros. Over an integer column, AVG still returns numeric in Postgres (no truncation), but other engines do integer division and truncate; a defensive AVG(amount::numeric) is portable. For a dashboard, round explicitly: ROUND(AVG(amount), 2).

Empty groups and stores with no transactions

AVG over a group with no non-NULL amounts returns NULL, not 0 — wrap in COALESCE(AVG(amount), 0) if the consumer needs a number. And a store with zero rows in transactions never appears in this GROUP BY at all; surfacing it as "ATV = 0" or "no data" requires a LEFT JOIN from the store dimension.

Senior signal: state that AVG ignores NULLs and decide on purpose whether a NULL means "skip" or "$0", never substitute COUNT(*) for COUNT(col) in a hand-rolled mean, add the txn_type = 'sale' filter that turns a raw average into a real ATV, and pre-empt the empty-group NULL and integer-division pitfalls.
