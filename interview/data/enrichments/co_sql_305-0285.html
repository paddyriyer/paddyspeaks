Search query frequency follows an extreme power-law distribution. The top 1% of queries (head terms like "weather", "youtube") account for 30%+ of all volume; the bottom 99% are the long tail of typos, niche queries, and one-shot searches. The mean is dominated by the head and lies; the median is the actual "typical" query frequency. Computing the median in SQL is a class of question every analytics engineer hits.

1. The source data — searches

10 query terms with their daily search counts. Most are small; a few head terms blow up the mean.

Mean = (sum) / 10 ≈ 1.22M. Median = (5th + 6th by sort) / 2. The mean is wildly misleading — most queries are run a handful of times.

2. The SQL — PERCENTILE_CONT

SELECT PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY freq) AS median_freq
FROM searches;

3. The resulting analysis

Sort the 10 frequencies: 3, 14, 22, 1.2K, 4.5K, 180K, 620K, 3.2M, 5.4M, 2.8M. Re-sorted: 3, 14, 22, 1200, 4500, 180000, 620000, 2800000, 3200000, 5400000. The 5th and 6th values are 4,500 and 180,000 → median = 92,250.

median_freq
92,250

Compare to mean (1,220,485): the median is 92,250 — 13× smaller. The mean told you "queries run on average 1.2M times"; the median tells the truth: "half of queries run fewer than 92K times". Both are correct; only one is honest about the long tail.

4. Strategic explanations

Why PERCENTILE_CONT not AVG

AVG(freq) is the mean — distorted by the head. PERCENTILE_CONT(0.5) is the median — robust to outliers. For any power-law distribution (search frequency, income, file sizes, transaction amounts), the median is what stakeholders actually want even when they ask for the "average". Senior analytics engineers volunteer this distinction unprompted.

The dialect note — PERCENTILE_CONT is Postgres / Snowflake / BigQuery

The query above runs in Postgres, Snowflake, BigQuery, Oracle, SQL Server. It does not run in SQLite (or older MySQL). The SQLite-portable workaround uses window-function ranking:

-- SQLite-friendly median via row ranking
WITH ranked AS (
  SELECT freq,
    ROW_NUMBER() OVER (ORDER BY freq)                 AS rn,
    COUNT(*) OVER ()                                   AS total
  FROM searches
)
SELECT AVG(freq) AS median_freq
FROM ranked
WHERE rn IN ((total + 1) / 2, (total + 2) / 2);
-- For 10 rows: rn IN (5, 6) → average of the 5th and 6th values.
-- For 11 rows: rn IN (6, 6) → just the 6th value.
PERCENTILE_CONT vs PERCENTILE_DISC

Same syntax, subtly different math:

  • PERCENTILE_CONT — interpolates between the two surrounding values when the rank is fractional. "continuous". Smooth.
  • PERCENTILE_DISC — picks the actual nearest value (no interpolation). "discrete". Always returns a real data point.

For 10 rows, the 50th percentile is technically between rank 5 and 6 — CONT interpolates the average; DISC just returns the 5th. For finance / regulatory reports requiring an actual data point (no synthesised values), use DISC. For everything else, CONT is fine.

Multiple percentiles in one pass — the dashboard query

Real reporting doesn't stop at the median — analytics teams want p10, p25, p50, p75, p90, p95, p99 in one row:

SELECT
  PERCENTILE_CONT(0.10) WITHIN GROUP (ORDER BY freq) AS p10,
  PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY freq) AS p25,
  PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY freq) AS p50,
  PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY freq) AS p75,
  PERCENTILE_CONT(0.90) WITHIN GROUP (ORDER BY freq) AS p90,
  PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY freq) AS p99,
  AVG(freq)                                             AS mean
FROM searches;

The gap between p99 and mean tells you how heavy-tailed the distribution is. p99 / median is the "skew ratio" data scientists watch.

The over-the-bar finish: per-group medians

Search frequency by category (medical / commercial / news / etc.) makes the long-tail comparison even more interesting:

SELECT
  category,
  PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY freq) AS median_freq
FROM searches
GROUP BY category;
Senior signal: name median is robust to outliers, mention the dialect difference (PERCENTILE_CONT works in Postgres/Snowflake/BigQuery, NOT SQLite — so this question is flagged non-sqlite in the playground), volunteer the SQLite workaround using ROW_NUMBER, and pre-empt the multi-percentile dashboard query. The PERCENTILE_CONT vs DISC distinction is the over-the-bar move.
