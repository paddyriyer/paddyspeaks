A loyalty program wants to identify its genuine long-standing customers off a 2025-04-01 snapshot — not just anyone with old orders, but accounts with real tenure, breadth of active months, and a sustained streak of consecutive activity. The interviewer is probing two things: can you compute a "longest run of consecutive months" with the gaps-and-islands pattern, and can you stack a dozen conditional-aggregation buckets (Y1/Y2, quarterly, L28) without the windows overlapping or leaking.

2. The SQL — gaps-and-islands streak plus stacked conditional aggregation

WITH params AS (SELECT DATE '2025-04-01' AS snapshot),
e AS (
    SELECT o.customer_id, o.order_date, o.amount,
           TO_CHAR(o.order_date, 'YYYY-MM') AS ym,
           (p.snapshot - o.order_date) AS days_back
    FROM orders o, params p
),
monthly AS (SELECT DISTINCT customer_id, ym FROM e),
indexed AS (
    SELECT customer_id, ym,
           (EXTRACT(YEAR FROM TO_DATE(ym||'-01','YYYY-MM-DD'))::INT - 2000) * 12
             + EXTRACT(MONTH FROM TO_DATE(ym||'-01','YYYY-MM-DD'))::INT AS m_idx,
           ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY ym) AS rn
    FROM monthly
),
islands AS (
    SELECT customer_id, (m_idx - rn) AS island_id, COUNT(*) AS streak_len
    FROM indexed GROUP BY customer_id, (m_idx - rn)
),
streak_per_cust AS (
    SELECT customer_id, MAX(streak_len) AS longest_streak
    FROM islands GROUP BY customer_id
),
idle_gaps AS (
    SELECT customer_id,
           m_idx - LAG(m_idx) OVER (PARTITION BY customer_id ORDER BY m_idx) - 1 AS idle_months
    FROM indexed
),
idle_per_cust AS (
    SELECT customer_id, COALESCE(MAX(idle_months), 0) AS max_idle_months
    FROM idle_gaps GROUP BY customer_id
),
agg AS (
    SELECT e.customer_id,
           MIN(order_date) AS first_order_date, MAX(order_date) AS last_order_date,
           COUNT(DISTINCT ym) AS months_active, COUNT(*) AS lifetime_orders,
           ROUND(SUM(amount), 2) AS lifetime_spend,
           (MAX(order_date) - MIN(order_date)) AS tenure_days,
           ROUND(SUM(CASE WHEN days_back < 365 THEN amount ELSE 0 END), 2) AS y1_spend,
           ROUND(SUM(CASE WHEN days_back >= 365 AND days_back < 730 THEN amount ELSE 0 END), 2) AS y2_spend,
           ROUND(SUM(CASE WHEN days_back < 90 THEN amount ELSE 0 END), 2) AS q1_spend,
           ROUND(SUM(CASE WHEN days_back >= 90 AND days_back < 180 THEN amount ELSE 0 END), 2) AS q2_spend,
           ROUND(SUM(CASE WHEN days_back >= 180 AND days_back < 270 THEN amount ELSE 0 END), 2) AS q3_spend,
           ROUND(SUM(CASE WHEN days_back >= 270 AND days_back < 365 THEN amount ELSE 0 END), 2) AS q4_spend,
           ROUND(SUM(CASE WHEN days_back < 28 THEN amount ELSE 0 END), 2) AS l28_spend,
           ROUND(SUM(CASE WHEN days_back >= 28 AND days_back < 56 THEN amount ELSE 0 END), 2) AS prior_l28_spend
    FROM e GROUP BY e.customer_id
)
SELECT a.customer_id, first_order_date, last_order_date,
       months_active, longest_streak, max_idle_months,
       lifetime_orders, lifetime_spend, y1_spend, y2_spend,
       CASE WHEN y2_spend > 0 THEN ROUND(100.0*(y1_spend - y2_spend)/y2_spend, 1) END AS yoy_growth_pct,
       q1_spend, q2_spend, q3_spend, q4_spend, l28_spend, prior_l28_spend,
       CASE WHEN prior_l28_spend > 0 THEN ROUND(l28_spend/prior_l28_spend, 2) END AS l28_velocity,
       DENSE_RANK() OVER (ORDER BY y1_spend DESC) AS loyalty_rank
FROM agg a
JOIN streak_per_cust sp ON sp.customer_id = a.customer_id
JOIN idle_per_cust   ip ON ip.customer_id = a.customer_id
WHERE tenure_days >= 730 AND months_active >= 12 AND longest_streak >= 6
ORDER BY y1_spend DESC, yoy_growth_pct DESC;

4. Strategic explanations

The gaps-and-islands trick: m_idx - rn

To find the longest run of consecutive calendar months, convert each year-month into a dense integer index (m_idx = year×12 + month) and number the distinct active months per customer with ROW_NUMBER() (rn). For any unbroken run, m_idx and rn both increase by exactly 1 each step, so their difference m_idx - rn stays constant. That constant becomes the island identifier; GROUP BY (m_idx - rn) then MAX(COUNT(*)) gives the longest streak. A one-month gap bumps the difference and starts a new island. This is the canonical pattern and saying its name is the senior move.

Why DISTINCT on months is mandatory

The monthly CTE collapses orders to one row per (customer, month) before indexing. If you skip the DISTINCT, two orders in the same month each get a separate rn, the m_idx - rn arithmetic desynchronises, and the streak count is silently wrong. Streak logic operates on the calendar grid, not on raw transactions.

The quarterly buckets are rolling 90-day slices, not calendar quarters

q1_spend…q4_spend are defined by days_back bands (0–89, 90–179, 180–269, 270–364) anchored on the snapshot. They are "the four quarters of the most recent rolling year", not Jan–Mar / Apr–Jun. The bands are half-open and non-overlapping by construction, so q1+q2+q3+q4 exactly equals y1_spend — a useful invariant to assert. An interviewer may ask you to prove the bands cannot double-count; the answer is that every >= lower bound equals the previous band's < upper bound.

NULL guards on the two ratios

yoy_growth_pct and l28_velocity both divide, and both are wrapped in CASE WHEN … > 0. A customer with no Y2 spend or no prior-L28 spend would otherwise trigger a division-by-zero error in Postgres (unlike some engines that return NULL). Returning NULL for "cannot be computed" is the correct, honest output — do not coalesce it to 0, which would falsely read as "flat".

Dialect note — the 2000 epoch and date math

m_idx subtracts 2000 from the year purely to keep the integer small and readable; any fixed epoch works. (MAX(order_date) - MIN(order_date)) yields an integer day count in Postgres; in SQL Server you would need DATEDIFF(day, ...) and in BigQuery DATE_DIFF(..., DAY). TO_CHAR(date,'YYYY-MM') is the Postgres month key; SQLite uses strftime('%Y-%m', ...). The streak logic itself is dialect-agnostic once you have an integer month index.

Senior signal: name the m_idx - rn gaps-and-islands pattern and explain why consecutive runs hold the difference constant, insist on DISTINCT months before indexing, clarify that the quarterly columns are rolling 90-day bands that sum to Y1, and guard every ratio against a zero denominator with an honest NULL rather than a misleading 0.
