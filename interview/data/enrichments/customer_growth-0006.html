A payments-fraud team flags account-takeover and reshipping rings by watching how many distinct shipping addresses an account uses in a short window. The question: find the first 30-day stretch in which a customer used 4 or more distinct addresses, and report that window with its address count and total spend. The interviewer is probing whether you can do a 30-day sliding-window count efficiently — and whether you collapse to one row per address before the window, so a customer who reuses one address ten times is not mistaken for ten addresses.

The dataset packs three traps: customer 2 (one address, five orders — the must-not-count-rows case), customer 3 (four addresses but never four within 30 days), and customer 4 sitting exactly on the > 3 boundary with three addresses. Only customers 1 and 5 should appear.

2. The SQL — collapse to first-seen, then a self-join 30-day window

The reference is written for PostgreSQL with native INTERVAL arithmetic (the original SQLite would use datetime(ts,'+30 days')). It runs as-is here.

WITH first_addr AS (
    SELECT customer_id, shipping_address,
           MIN(order_ts) AS first_ts,
           SUM(amount)   AS addr_spend
    FROM orders
    GROUP BY customer_id, shipping_address
),
windowed AS (
    SELECT a.customer_id,
           a.first_ts        AS window_start,
           MAX(b.first_ts)   AS window_end,
           COUNT(*)          AS distinct_addresses,
           SUM(b.addr_spend) AS total_amount
    FROM first_addr a
    JOIN first_addr b
      ON b.customer_id = a.customer_id
     AND b.first_ts BETWEEN a.first_ts
                        AND a.first_ts + INTERVAL '30 days'
    GROUP BY a.customer_id, a.first_ts
    HAVING COUNT(*) > 3
),
ranked AS (
    SELECT windowed.*,
           ROW_NUMBER() OVER (PARTITION BY customer_id
                              ORDER BY distinct_addresses DESC,
                                       window_start ASC) AS rn
    FROM windowed
)
SELECT customer_id, window_start, window_end,
       distinct_addresses, total_amount
FROM ranked
WHERE rn = 1
ORDER BY distinct_addresses DESC, customer_id ASC;

4. Strategic explanations

Collapse to first-seen before the window — the core insight

first_addr reduces orders to one row per (customer, address) at the moment that address was first used. This is what makes the count correct: customer 2 placed five orders to "12 Oak Dr", but after the collapse that is a single row, so COUNT(*) can never read it as five addresses. The fraud signal is "how many distinct addresses", and counting must happen on the distinct grain. Skipping this step is the most common wrong answer to the question.

Why anchor the window on each address's first_ts

The self-join treats every address-first-seen as a potential window start, then counts how many other addresses were first seen within the next 30 days. Anchoring on actual data points (not an arbitrary calendar grid) guarantees that if a qualifying 30-day window exists at all, one of the anchors finds it — the densest cluster always begins on some address's first appearance. This is the standard "event-anchored sliding window" technique and it is exact, not approximate.

The half-open versus closed boundary, and the > 3 threshold

BETWEEN a.first_ts AND a.first_ts + INTERVAL '30 days' is inclusive on both ends — an address first seen exactly 30 days later still counts. HAVING COUNT(*) > 3 means strictly more than 3, i.e. 4+. Customer 4 with exactly 3 addresses fails by one, which is the deliberate boundary test: a careless >= 3 would wrongly flag a customer who simply has a billing, a work, and a home address. Read the threshold operator precisely.

ROW_NUMBER picks the first qualifying window per customer

A customer can have several overlapping 30-day windows that all clear the threshold. The question asks for one row per offender — the first such window. ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY distinct_addresses DESC, window_start ASC) then WHERE rn = 1 deduplicates: it prefers the window with the most addresses, breaking ties toward the earliest start. ROW_NUMBER (not RANK) is correct here precisely because we want exactly one row even if two windows tie.

Dialect note — INTERVAL versus datetime()

Native Postgres does ts + INTERVAL '30 days'; SQLite needs datetime(ts, '+30 days'); SQL Server uses DATEADD(day, 30, ts); BigQuery uses TIMESTAMP_ADD(ts, INTERVAL 30 DAY). The window logic is identical — only the additive-interval syntax changes. Keeping order_ts as a real TIMESTAMP (rather than a text date) is what lets the arithmetic run natively without casts.

Senior signal: collapse to one row per (customer, address) first-seen before counting so repeated orders to one address cannot inflate the distinct count, anchor the 30-day window on real event timestamps so the densest cluster is always found, read the inclusive BETWEEN and strict > 3 threshold exactly (the 3-address customer must not flag), and use ROW_NUMBER to return exactly the first qualifying window per offender.
