This is the classic "report-shaped output" question: the customer table only has rows for months a customer actually spent in, but finance wants a dense grid — every customer against every month, with blanks where there was no spend. The interviewer is checking whether you reach for a CROSS JOIN to manufacture the full skeleton, then LEFT JOIN the facts back onto it. A naive GROUP BY on the raw table can never invent the missing rows.

Aarav never has a February row; Diya has two February rows. Both are deliberate: one tests the gap-filling, the other tests whether the join silently fans out.

month_tab: (1, Jan), (2, Feb), (3, Mar).

2. The SQL — cross join to build the grid, left join to fill it

WITH cte AS (
  SELECT c.cust_id AS customer_id, c.name AS customer_name,
         m.month_id, m.month_name
  FROM customer_tab c
  CROSS JOIN month_tab m
  GROUP BY c.cust_id, c.name, m.month_id, m.month_name
)
SELECT cte.customer_id, cte.customer_name, cte.month_name, c.amount_spend
FROM cte
LEFT JOIN customer_tab c
  ON cte.customer_id = c.cust_id
 AND cte.month_id    = c.month_id;

4. Strategic explanations

Why CROSS JOIN is the right move

A CROSS JOIN produces the Cartesian product: every customer paired with every month. That is exactly the dense skeleton a calendar-style report needs. The customer table can only describe months that happened; the cross join manufactures the months that should appear. This "spine table" pattern — build the complete dimension grid, then left-join facts onto it — is the standard fix for any report that must show zero/blank rows.

The GROUP BY with no aggregate is just a disguised DISTINCT

The CTE's GROUP BY c.cust_id, c.name, m.month_id, m.month_name has no SUM/COUNT — it is purely deduplicating the cross-join output. Writing SELECT DISTINCT would be clearer about intent. But here is the subtle bug: the dedup happens before the second join. The CTE collapses to one row per customer/month, then the LEFT JOIN back to customer_tab re-introduces Diya's duplicate Feb row. Deduplicating early does not protect you from a fan-out later.

The duplicate-row trap — and the real fix

If the spec is "one row per customer per month", the correct query aggregates the spend instead of joining raw rows:

SELECT c.cust_id AS customer_id, c.name AS customer_name,
       m.month_name,
       SUM(s.amount_spend) AS amount_spend
FROM customer_tab c
CROSS JOIN month_tab m
LEFT JOIN customer_tab s
  ON s.cust_id = c.cust_id AND s.month_id = m.month_id
GROUP BY c.cust_id, c.name, m.month_id, m.month_name;

Now Diya's February is a single row showing 450, and untouched months still show NULL (because SUM of zero rows is NULL). Wrap in COALESCE(SUM(...), 0) if the report wants explicit zeros. The lesson: when the source can have duplicates, you must decide — sum them, or pick one with DISTINCT ON/ROW_NUMBER — never leave the fan-out to chance.

Cross-join cost — know the size before you run it

CROSS JOIN output is rows(A) × rows(B). Customers × 12 months is harmless; customers × a daily calendar for 10 years is 3,650× the customer count. The pattern is fine, but always bound the smaller side — here, generate only the months you actually need rather than cross-joining a huge date dimension.

Senior signal: name the "spine table" pattern — CROSS JOIN to build the dense grid, LEFT JOIN facts onto it — and then catch that deduplicating inside the CTE does not survive the later join. The senior finish is replacing the raw left-join with SUM(amount_spend) ... GROUP BY so a duplicate source row collapses to one correct cell.
