Atlassian's account-management team wants a one-line summary of what each customer has bought — a comma-separated list of products on a single row, so a CSM can glance at it without scrolling an order history. The query is a group-by with a string aggregate. It looks trivial, but the interviewer is probing whether you handle ordering, duplicates, and the engine-specific name of the function.

1. The source data — orders

customer_idproductorder_idamount
101Mouse125
101Keyboard275
101Mouse325
102Monitor4300
103Cable512

Customer 101 ordered a Mouse twice (orders 1 and 3) — a duplicate product. Customers 102 and 103 each have a single order. The duplicate is the edge case: it decides whether the summary shows "Mouse" once or twice.

2. The SQL — STRING_AGG with ORDER BY

SELECT customer_id,
       STRING_AGG(product, ', ' ORDER BY product) AS products
FROM orders
GROUP BY customer_id;

STRING_AGG collapses all product values in a group into one delimited string. The inline ORDER BY product fixes the concatenation order so the output is stable across runs.

3. The resulting analysis

customer_idproducts
101Keyboard, Mouse, Mouse
102Monitor
103Cable

Customer 101's list reads "Keyboard, Mouse, Mouse" — the Mouse appears twice because they placed two Mouse orders, and STRING_AGG aggregates rows, not distinct values. For a "what has this customer ever bought" summary that repetition is usually unwanted noise.

4. Strategic explanations

De-duplicating — DISTINCT inside the aggregate

If the summary should list each product once, add DISTINCT:

SELECT customer_id,
       STRING_AGG(DISTINCT product, ', ' ORDER BY product) AS products
FROM orders
GROUP BY customer_id;
-- customer 101 -> "Keyboard, Mouse"

One constraint: when you use DISTINCT in STRING_AGG, the ORDER BY expression must match the aggregated expression — you can order by product but not by, say, order_date. Confirm with the interviewer whether duplicates are signal (bought it 5 times) or noise.

Why the ORDER BY is not optional

Without ORDER BY product, the concatenation order is whatever the executor happens to produce — it can change between runs, between plan choices, after a vacuum. For anything compared, diffed, or snapshot-tested, an unordered STRING_AGG is a flaky-test generator. Always pin the order.

The dialect map — same idea, four names
enginefunction
PostgresSTRING_AGG(product, ', ' ORDER BY product)
MySQLGROUP_CONCAT(product ORDER BY product SEPARATOR ', ')
Oracle / SnowflakeLISTAGG(product, ', ') WITHIN GROUP (ORDER BY product)
SQL ServerSTRING_AGG(product, ', ') WITHIN GROUP (ORDER BY product)

Naming that LISTAGG / GROUP_CONCAT are the same primitive under different syntax is a senior tell. Note LISTAGG also has a low default length cap on some engines — long lists can truncate or error.

NULL products and the array alternative

STRING_AGG skips NULL inputs — a NULL product simply does not appear in the list, with no stray delimiter. That is usually desirable. If a downstream consumer needs to process the items rather than just display them, aggregate to a real array instead of a string — ARRAY_AGG(DISTINCT product ORDER BY product) — which avoids fragile delimiter-splitting later. A string is for humans; an array is for code.

Senior signal: always pin STRING_AGG with an explicit ORDER BY for deterministic output, decide whether duplicates are signal or noise and reach for DISTINCT when they are noise, map the function to LISTAGG/GROUP_CONCAT across engines, and prefer ARRAY_AGG when the result feeds code rather than a human.
