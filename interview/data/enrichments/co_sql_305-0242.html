Apple's commerce team tags SKUs with structured prefixes and suffixes — PROMO- for promotional items, -2025 for the 2025 catalogue year. An analyst needs to pull every order whose SKU matches either pattern. It looks like a trivial LIKE query, and it is — but the interviewer is checking whether you understand wildcard semantics, anchoring, and the OR's effect on the index.

1. The source data — orders

order_idskucustomer_idorder_datestatusamount
1PROMO-ABC1012025-01-01completed50

3PROMO-XYZ-20251032025-03-01completed120
4STD-WIDGET1042025-04-01completed30
5GIFT-20241052024-05-01completed60

Order 3's SKU PROMO-XYZ-2025 matches both patterns — the OR must still return it exactly once. Order 5's GIFT-2024 ends in -2024, a near-miss that proves the suffix anchor is doing real work.

2. The SQL — LIKE with prefix and suffix wildcards

SELECT * FROM orders WHERE sku LIKE 'PROMO-%' OR sku LIKE '%-2025';

LIKE 'PROMO-%' matches any SKU that starts with the literal PROMO- (% is "zero or more characters"). LIKE '%-2025' matches any SKU that ends with -2025. The OR unions the two sets.

3. The resulting analysis

order_idskucustomer_idorder_datestatusamount
1PROMO-ABC1012025-01-01completed50
2STD-20251022025-02-01completed80
3PROMO-XYZ-20251032025-03-01completed120

Three of five rows match. Order 3 satisfies both predicates but appears once — OR is set membership, not concatenation, so no duplication. Order 4 (STD-WIDGET) matches neither; order 5 (GIFT-2024) is correctly excluded because its suffix is -2024, not -2025.

4. Strategic explanations

% vs _ — the two wildcards

% matches any run of characters including the empty string; _ matches exactly one character. So LIKE 'PROMO-%' would still match a SKU that is literally PROMO- with nothing after. If "PROMO- followed by at least one character" is required, write LIKE 'PROMO-_%' — one mandatory character, then any tail. Knowing the empty-match behaviour is the detail interviewers probe.

Anchoring — where the wildcards sit changes everything

'PROMO-%' is anchored at the start; '%PROMO-%' would match PROMO- anywhere in the string (a substring search). '%-2025' is anchored at the end. A frequent bug is writing '%PROMO%' when you meant a true prefix — it would also catch a SKU like OLD-PROMO-1. Decide deliberately: prefix, suffix, or contains.

The OR and the index trap

A leading-anchored LIKE 'PROMO-%' can use a B-tree index on sku — it is effectively a range scan. But LIKE '%-2025' has a leading %, so it cannot; it forces a full scan. And once two predicates are joined by OR, many planners scan the whole table anyway. On a large orders table, a trailing-wildcard filter is a known performance smell — the senior fix is to store the catalogue year as its own indexed column rather than parsing it out of the SKU.

Escaping literal % and _

If a SKU genuinely contains a percent sign or underscore, those characters are wildcards inside LIKE and must be escaped: LIKE 'A\_B%' ESCAPE '\' matches a literal underscore. This bites most often with _, since underscores are common in identifiers — LIKE 'PROMO_%' silently treats the underscore as "any one character", not a literal.

Case sensitivity and NULL SKUs

LIKE is case-sensitive in Postgres — 'promo-abc' would not match 'PROMO-%'. Use ILIKE for case-insensitive matching, or normalise with UPPER(sku) LIKE 'PROMO-%' (which also defeats the index). And a NULL sku makes LIKE return NULL, not true — NULL-SKU orders are silently excluded by both predicates, which is usually correct but worth stating.

SELECT * — acceptable here, flag it anyway

The solution uses SELECT *. For an ad-hoc analyst pull that is fine, but in shipping code it is fragile: a schema change reorders or adds columns and silently changes the result shape. Name the columns the consumer actually needs.

Senior signal: distinguish % from _ and know % matches the empty string, anchor patterns deliberately (prefix vs suffix vs contains), recognise that a leading-wildcard LIKE under an OR defeats the index and suggest a dedicated catalogue-year column, and remember ILIKE/escaping for case and literal-wildcard correctness.
