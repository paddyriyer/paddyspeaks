Google asks Min Stack because it is a perfect small test of one idea: can you make an O(1) query work by precomputing the answer instead of recomputing it? A stack that returns its minimum in constant time cannot scan its contents — that would be O(n). The interviewer wants to see you carry the minimum forward as the stack mutates, which is a microcosm of how real systems keep maintained aggregates.

1. The input data

A sequence of operations on a fresh MinStack:

st = MinStack()
st.push(5)
st.push(2)
st.push(7)
st.push(1)
st.getMin()
st.pop()
st.getMin()
st.top()

2. The approach

The class keeps two parallel lists. self.s is the actual value stack. self.m is a "minimum-so-far" stack: m[i] holds the smallest value among the first i+1 pushes. On push(x), the code appends x to s, and appends to m either x (if m is empty) or min(x, m[-1]) — the smaller of the new value and the previous running minimum. The two stacks therefore always have equal length and move in lockstep. pop() pops both. top() reads s[-1]; getMin() reads m[-1], which by construction is the minimum of every value currently in the stack. The minimum is never searched for — it is maintained.

3. The output

operationvalue stack smin stack mreturns
push(5)[5][5]—
push(2)[5, 2][5, 2]—
push(7)[5, 2, 7][5, 2, 2]—
push(1)[5, 2, 7, 1][5, 2, 2, 1]—
getMin()[5, 2, 7, 1][5, 2, 2, 1]1
pop()[5, 2, 7][5, 2, 2]—
getMin()[5, 2, 7][5, 2, 2]2
top()[5, 2, 7][5, 2, 2]7

The crucial line is the second getMin(): after popping 1, the min stack also pops, exposing 2 — the minimum correctly "rolls back" without any rescan.

4. Strategic explanations

Complexity

Every operation — push, pop, top, getMin — is O(1). Space is O(n), with a second stack the same height as the first. That extra stack is the price of constant-time minimum queries; it is a deliberate space-for-time trade.

The idiomatic alternative — store pairs or deltas

Instead of two lists you can push (value, current_min) tuples onto one stack — same complexity, but the min can never desync from its value because they live in the same slot. A more advanced variant stores only deltas from the running minimum, achieving O(1) extra space beyond the value stack. Mentioning the delta trick is a strong senior tell.

The edge case — popping an empty stack

pop(), top(), and getMin() all index [-1] with no guard, so calling them on an empty stack raises IndexError. Whether that is acceptable depends on the contract; many interviewers want an explicit check or a documented precondition. State the assumption rather than letting it surprise you.

The follow-up — a Max Stack, or popMax

"Now also support getMax." Add a third stack mirroring the min logic with max(...). The harder follow-up, popMax (remove the maximum element wherever it sits), breaks the pure-stack model — it needs a balanced BST or a doubly linked list plus a heap, because you can no longer assume the target is on top.

The trap — storing just a scalar minimum

Keeping a single self.min integer works for push but fails on pop: when you pop the current minimum, you have no idea what the previous minimum was — that history is gone. The minimum must be a stack precisely so it can unwind in step with the values.

Senior signal: a senior keeps the minimum as a synchronized stack (or tuples) so it rolls back on pop, can describe the O(1)-extra-space delta encoding, and names the empty-stack IndexError — a junior tracks one scalar min and corrupts it on the first pop.
