Payroll calculation is a classic SQL interview scenario for gig-economy and staffing platforms: "Compute each employee's gross pay for the week, splitting hours into regular (up to 40) and overtime (above 40, paid at 1.5× the hourly rate)." The interviewer is probing knowledge of LEAST and GREATEST — two scalar clamping functions that replace verbose CASE WHEN branches and make overtime arithmetic clean and readable.

1. The source data — timecards

emp_idweek_starthours_workedhourly_rate

E0032025-05-054830.00

The edge cases: E001 is under 40 hours (no overtime); E002 is exactly 40 (no overtime); E003 has 8 hours overtime; E004 has 15 hours overtime; E005 worked zero hours (both overtime columns should be 0, not negative).

2. The SQL — LEAST / GREATEST clamping for overtime pay

SELECT emp_id, week_start, hours_worked, hourly_rate,
       LEAST(hours_worked, 40)                    AS regular_hours,
       GREATEST(hours_worked - 40, 0)             AS overtime_hours,
       LEAST(hours_worked, 40) * hourly_rate
       + GREATEST(hours_worked - 40, 0) * hourly_rate * 1.5  AS gross_pay
FROM timecards;

4. Strategic explanations

Why LEAST / GREATEST beats CASE WHEN here

The equivalent CASE WHEN for regular hours would be:

CASE WHEN hours_worked < 40 THEN hours_worked ELSE 40 END

And for overtime hours:

CASE WHEN hours_worked > 40 THEN hours_worked - 40 ELSE 0 END

LEAST(hours_worked, 40) and GREATEST(hours_worked - 40, 0) express the same idea in half the tokens and are unambiguous about intent: clamp to a floor or ceiling. They also handle the boundary (exactly 40 hours) correctly without any explicit condition. In an interview, reaching for LEAST/GREATEST is a signal of SQL fluency beyond basic CASE.

The GREATEST(x, 0) floor pattern — prevents negative values

hours_worked - 40 is negative for any employee under 40 hours. Without the GREATEST(..., 0) floor, overtime hours would be negative and gross_pay would be understated. This is the key correctness guard: any "difference that can go negative but should read as zero" is the textbook use case for GREATEST(expr, 0). The same pattern appears in inventory deltas, refund calculations, and balance floor checks.

Arithmetic correctness — integer vs numeric types

If hours_worked and hourly_rate are stored as integers, the product is an integer and the multiplication by 1.5 (a float literal) triggers an implicit cast in most engines. In Snowflake and Postgres this works but results in a FLOAT column. For financial calculations, prefer explicit casts to NUMERIC/DECIMAL to preserve scale and avoid floating-point rounding: hours_worked::numeric * hourly_rate::numeric * 1.5.

Dialect portability
  • Snowflake: LEAST and GREATEST are built-in; accept multiple arguments.
  • Postgres: identical support.
  • BigQuery: identical support.
  • MySQL: identical support.
  • SQL Server: no LEAST/GREATEST before SQL Server 2022. Must use CASE WHEN or IIF on older versions. SQL Server 2022 added them.
Extending to double-time or tiered overtime

Many jurisdictions pay double-time above 12 hours in a day or 60 hours in a week. The three-tier version follows the same clamping pattern:

LEAST(hours_worked, 40)                      * hourly_rate          -- regular
+ LEAST(GREATEST(hours_worked - 40, 0), 20)  * hourly_rate * 1.5   -- OT tier 1 (40-60 hrs)
+ GREATEST(hours_worked - 60, 0)             * hourly_rate * 2.0    -- double time >60 hrs

Recognising and naming this nested-clamp pattern for multi-tier rates is the senior-level extension.

Senior signal: reach for LEAST/GREATEST instead of CASE WHEN for scalar clamping, explain that GREATEST(hours_worked - 40, 0) is the floor guard that prevents negative overtime values, flag the integer-vs-NUMERIC type risk for financial arithmetic, and extend the pattern to multi-tier overtime rates using nested clamps.
