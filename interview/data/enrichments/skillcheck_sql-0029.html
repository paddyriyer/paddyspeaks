An analyst builds a per-transaction running total of daily sales and is surprised: two rows that share the same date both show the same cumulative figure — the running total "lumps" instead of stepping row by row. The interviewer is probing whether you know the difference between the two default window frames, RANGE and ROWS, and how ORDER BY tie groups interact with them.

1. The source data — sales

damt(narrative)
2025-01-01100day 1, first sale
2025-01-0150day 1, second sale — tied date, the edge case
2025-01-0230day 2
2025-01-0370day 3

The two 2025-01-01 rows form a tie group on d. With a window ordered only by d, both rows are peers — and that is what makes the running total collapse them.

2. The SQL — deterministic running total with ROWS

SELECT d, amt,
       SUM(amt) OVER (ORDER BY d, amt
                      ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_total
FROM sales
ORDER BY d, amt;

Two fixes work together. ORDER BY d, amt adds a tiebreaker so every row has a unique ordering position. And ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW makes the frame physical — "all rows up to and including this one" — so each row gets its own distinct cumulative value.

3. The resulting analysis

Contrast the naive SUM(amt) OVER (ORDER BY d): both 2025-01-01 rows show 150, because the default RANGE frame sums every peer in the tie group. The corrected query produces a strict step of 50 then 150.

4. Strategic explanations

Why the default frame lumps — RANGE vs ROWS

When a window has ORDER BY but no explicit frame, SQL defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. The subtlety: under RANGE, "current row" means every row with the same ORDER BY value — the whole peer group. So both 2025-01-01 rows see a frame that includes both of them, and both get 100+50=150. ROWS instead counts physical rows, so "current row" is literally one row, and the cumulative steps cleanly.

The tiebreaker matters too

Even with ROWS, ordering only by d leaves the two day-1 rows in an arbitrary physical order — the engine may put 100 or 50 first, and the result is non-deterministic across runs or plans. Adding amt (or an id) to the ORDER BY makes the sequence reproducible. Determinism is a correctness property, not a nicety: a "running total" that flips between runs is a bug.

When RANGE is actually what you want

RANGE is not broken — it is correct when you genuinely want every tied row to share the group's cumulative value, e.g. a "total sales through end of this day" column where intra-day ordering is meaningless. The skill is choosing deliberately: ROWS for a strict per-row running total, RANGE for a per-tie-group total. State which one the requirement implies.

Dialect notes

PostgreSQL, SQL Server, Oracle, and DuckDB all default to RANGE and support explicit ROWS. MySQL 8 likewise. SQLite supports both. The trap is universal — this is standard SQL behaviour, not a quirk — so "just always write ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW for running totals" is portable, defensible advice. A subtle extra: RANGE with a numeric offset (RANGE BETWEEN 5 PRECEDING ...) requires a single numeric/date ORDER BY column, another reason teams default to ROWS.

Senior signal: know that an ORDER BY-only window defaults to a RANGE frame, which makes tied rows share one cumulative value; for a strict per-row running total specify ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW and add a tiebreaker to ORDER BY for determinism.
