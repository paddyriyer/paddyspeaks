A SaaS company stores its users table as an append-only edit log: every profile change writes a new row instead of updating in place, so a single user_id appears many times. The reporting team needs the current state — one row per user, the one with the latest updated_at. The interviewer is probing whether you reach for the canonical ROW_NUMBER() "latest-per-group" pattern and whether you make the tie-break deterministic.

1. The source data — users

User 100's three rows are out of insertion order on purpose — the latest updated_at is not the last-inserted row. User 102 is the edge case: a user with a single history row must still survive and pass through unchanged.

2. The SQL — ROW_NUMBER() per partition, keep rank 1

SELECT user_id, name, email, updated_at
FROM (
    SELECT *,
           ROW_NUMBER() OVER (
               PARTITION BY user_id
               ORDER BY updated_at DESC, user_id ASC
           ) AS rn
    FROM users
)
WHERE rn = 1;

PARTITION BY user_id restarts the numbering for each user; ORDER BY updated_at DESC puts the most recent edit first, so it gets rn = 1. The outer query keeps only those rows. The window function must live in a subquery because rn cannot be referenced in the same WHERE that produces it.

3. The resulting analysis

Exactly one row per user. User 100 collapses to its 2024-03-10 edit even though that was the third INSERT; ORDER BY updated_at DESC — not row order — decides the keeper. User 102's lone row is untouched: a partition of size one still has an rn = 1.

4. Strategic explanations

Why ROW_NUMBER, not RANK or DENSE_RANK

The requirement is "exactly one row per user". ROW_NUMBER guarantees that — it assigns 1, 2, 3… with no repeats, so rn = 1 selects precisely one row. RANK would assign the same rank to tied rows, so two edits with an identical updated_at would both get rank 1 and you would get duplicate users back. When the goal is deduplication, ROW_NUMBER is the correct family member.

The tie-break makes the result deterministic

If two rows share the same updated_at, ORDER BY updated_at DESC alone leaves their order undefined — the engine may return either, and the answer can change between runs. Adding user_id ASC as a secondary key is a reasonable gesture, but note: within one partition user_id is constant, so it does not actually break a within-user tie. A genuinely deterministic tie-break needs a column that varies inside the group — a surrogate id, an insert sequence, or a row version. Calling that out is the senior detail.

The pre-window-function alternative and its trap

The commented correlated-subquery form — WHERE updated_at = (SELECT MAX(updated_at) FROM users u2 WHERE u2.user_id = u.user_id) — works on legacy engines without window functions, but has a bug the ROW_NUMBER form does not: if a user has two rows tied at the max updated_at, the subquery returns both, re-introducing duplicates. The window approach with a proper tie-break is strictly safer.

NULL timestamps sink or float — decide

If updated_at can be NULL, ORDER BY updated_at DESC in Postgres places NULLs first (NULLs sort as largest under DESC by default), so a user whose only meaningful edit lacks a timestamp could have the NULL row chosen as "latest". Add NULLS LAST if a missing timestamp should never win. MySQL and SQLite differ on default NULL ordering, so be explicit.

Dialect notes

Window functions are standard in Postgres, modern MySQL (8.0+), SQLite (3.25+), Snowflake, and SQL Server. Postgres requires the derived table to have an alias — FROM (…) AS t; some engines tolerate its absence. A terser Postgres-specific alternative is DISTINCT ON (user_id) … ORDER BY user_id, updated_at DESC, which picks one row per user_id directly without the subquery.

Senior signal: use ROW_NUMBER() OVER (PARTITION BY key ORDER BY recency DESC) + WHERE rn = 1 for latest-per-group, choose ROW_NUMBER over RANK because dedup needs exactly one row, insist the tie-break uses a column that varies within the partition for true determinism, and handle NULL timestamps with explicit NULLS LAST.
