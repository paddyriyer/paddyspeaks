New Relic's logistics analytics team tracks how long shipments take versus their promised date. Leadership wants a monthly trend: is the average delay getting better or worse? The query is a one-liner, but it hides a subtraction that depends on column types, a grouping key that has to match the projection, and an AVG that quietly decides what to do with shipments that arrived early.

1. The source data — shipments

Shipment 2 arrived exactly on time (delay 0). Shipment 6 arrived a day early — its delay is -1. That negative value is the interesting one: it will offset a late shipment inside the same month's average.

2. The SQL — AVG(actual - scheduled) by month

SELECT DATE_TRUNC('month', scheduled_date)::DATE AS mo,
  AVG(actual_date - scheduled_date) AS avg_delay_days
FROM shipments GROUP BY DATE_TRUNC('month', scheduled_date)::DATE;

Subtracting two Postgres DATE values yields an integer count of days. DATE_TRUNC('month', …) snaps each scheduled date down to the first of its month, giving a clean grouping key.

3. The resulting analysis

January's three shipments delayed by 2, 0 and 5 days average to 2.33. February's 1, 10 and −1 average to 3.33 — note the early shipment (−1) partly cancels the badly-late one (10). The two-month picture says delays are rising. Also note the rows come back in no particular order: there is no ORDER BY.

4. Strategic explanations

The missing ORDER BY — a trend that isn't sorted

A "trend" query that returns months out of order is half-finished. The grouping key is computed, so add the matching sort:

... GROUP BY DATE_TRUNC('month', scheduled_date)::DATE
    ORDER BY mo;

Without it, the SQL is technically correct but the consumer has to re-sort. For a leadership trend chart, that omission reads as careless.

Early arrivals: should they count?

The subtraction makes shipment 6 a -1, and February's average benefits from it. If "delay" should never go below zero — an early arrival is just "on time", not "negative delay" — floor it:

AVG(GREATEST(actual_date - scheduled_date, 0)) AS avg_delay_days

That changes February to (1 + 10 + 0) / 3 = 3.67. Naming this choice — raw signed delay vs floored lateness — is the senior move; they measure different things.

Truncate by scheduled or by actual?

The query buckets on scheduled_date. A shipment scheduled Jan 28 but delivered Feb 2 lands in January. That is usually right — you are measuring the performance of January's commitments. But if the question is "how did the ops team do in February", you would truncate on actual_date instead. Same data, different story; confirm which month the metric belongs to.

Integer division and AVG

actual_date - scheduled_date is an integer, but AVG of integers in Postgres returns numeric — hence the long decimals above, not a truncated 2 or 3. That is Postgres being helpful; some engines (and integer SUM/COUNT hand-rolls) would truncate. Round for display: ROUND(AVG(actual_date - scheduled_date), 1).

NULL actual dates — shipments still in transit

If a shipment hasn't arrived, actual_date is NULL, the subtraction is NULL, and AVG silently skips it. That is defensible (you can't measure a delay that hasn't finished) but it means the metric covers only delivered shipments. Say so — or report a separate "still open" count so leadership isn't blind to the in-flight tail.

Senior signal: add the ORDER BY mo a trend chart needs, decide deliberately whether early arrivals are negative delays or floored to zero, justify bucketing on scheduled vs actual date, and flag that NULL actual dates make this a delivered-only metric.
