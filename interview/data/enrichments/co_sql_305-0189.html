Citi's finance desk reports profit on a smoothed basis — a 7-period rolling average that damps the noise of any single period. The query is one window function, but the interviewer is watching the frame specification (ROWS BETWEEN 6 PRECEDING AND CURRENT ROW), what happens at the start of the series, and the difference between ROWS and RANGE.

Eight periods. The interesting region is the first six: the window asks for 6 preceding rows, but period 1 has none, period 2 has one, and so on — the average "warms up" over a partial window before it ever sees a full 7.

2. The SQL — windowed AVG

SELECT period, amount,
  AVG(amount) OVER (ORDER BY period ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS rolling_7
FROM profit;

For each row, the window is "this row plus the 6 before it", ordered by period, and AVG averages the amounts inside that frame.

Period 1's "rolling 7" is just 100 — it averaged a single row. Period 7 is the first true 7-period average (100..350, sum 1750, ÷7 = 250). Period 8 averages periods 2..8. The first six rows are partial-window averages, and that is the subtlety most candidates miss.

4. Strategic explanations

The frame does not error at the edges — it shrinks

ROWS BETWEEN 6 PRECEDING AND CURRENT ROW does not fail when fewer than 6 prior rows exist; it simply uses whatever rows are there. So rolling_7 for period 3 is AVG(100, 200, 150) — a 3-row average labelled as if it were a 7. For a finance report this can mislead: the early periods look smoother than the methodology claims. If only genuine full windows should report a value, gate on the row count:

SELECT period, amount,
  CASE WHEN COUNT(*) OVER w = 7
       THEN AVG(amount) OVER w END AS rolling_7
FROM profit
WINDOW w AS (ORDER BY period ROWS BETWEEN 6 PRECEDING AND CURRENT ROW);

Now periods 1–6 return NULL and only period 7 onward shows a number. Stating this trade-off — "partial windows, or NULL until warmed up?" — is the senior move.

It is a 7-row window, not 6 — the off-by-one

6 PRECEDING plus CURRENT ROW is 7 rows. The classic mistake is writing 7 PRECEDING for a "7-period average", which yields an 8-row window. Count the current row in: an N-period rolling average is (N-1) PRECEDING AND CURRENT ROW.

ROWS vs RANGE — and the gap trap

ROWS counts physical rows: exactly 7 rows regardless of their period values. RANGE counts logical values of the ordering column. If period had a gap — say period 5 is missing — ROWS still grabs 7 rows (reaching further back in time), whereas a value-based RANGE BETWEEN '6' PRECEDING ... would span a fixed period-width window and include fewer rows. For a "last 7 data points" average, ROWS is correct. For "last 7 calendar units", RANGE is correct. Know which the report means; with gappy data they diverge.

The default frame trap — always specify it

If you write AVG(amount) OVER (ORDER BY period) with no frame clause, Postgres applies the default RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — a cumulative average from the start of the series, not a rolling one. Forgetting the explicit frame silently turns a rolling metric into a running one. The explicit ROWS BETWEEN ... in the solution is mandatory, not optional.

Edge case: ties in period, and NULL amounts

If two rows share a period, the ORDER BY cannot deterministically order them, so which one is "current" and which is "preceding" is arbitrary — profit should have one row per period. And if an amount is NULL, AVG skips it: the window still spans 7 rows but averages only the 6 non-NULL ones. If a missing period should count as 0 profit, COALESCE(amount, 0) first.

Senior signal: explain that the frame shrinks (not errors) at the series start so early rows are partial-window averages, get the off-by-one right (6 PRECEDING + current = 7 rows), contrast ROWS vs RANGE for gappy data, and know that omitting the frame clause silently yields a cumulative average instead of a rolling one.
