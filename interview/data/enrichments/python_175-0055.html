Amazon's data-engineering teams constantly land nested JSON from APIs and event streams into flat tables for the warehouse. "Flatten Nested JSON to CSV with Dot-Notation Columns" tests whether you can write a clean recursive traversal that handles dicts and lists, builds path-prefixed column names, and bottoms out on scalars — without losing data or crashing on irregular shapes.

1. The input data

[
  {"id": 1, "name": "Ann",
   "address": {"city": "NYC", "zip": "10001"},
   "tags": ["vip", "new"]},
  {"id": 2, "name": "Bob",
   "address": {"city": "LA", "zip": "90001"},
   "tags": ["new"]}
]

Two records with a nested object (address) and a list (tags) — the two recursion cases the function must handle.

2. The approach

flatten() recurses on three cases. For a dict, it walks each k, v and recurses with the prefix extended by {k}.. For a list, it recurses on each element using the index as the path segment (tags.0, tags.1). For a scalar (the base case), it writes one entry, stripping the trailing dot from the accumulated prefix with rstrip('.'). Each level out.update(...)s the child results, so a single flat dict bubbles up.

json_to_csv loads the file, normalizes a top-level object to a one-element list, flattens every row, and hands the list of flat dicts to pd.DataFrame. Pandas unions all keys into columns and fills missing cells with NaN — which matters when rows have different shapes.

Bob's tags.1 is empty because his tags list has only one element — pandas aligns columns and leaves the missing cell blank in CSV.

4. Strategic explanations

Complexity

O(N) where N is the total number of leaf scalars across all records — every node is visited once. Recursion depth equals the nesting depth, which is the space cost. The DataFrame holds all columns × all rows.

The idiomatic alternative — pandas.json_normalize

Pandas ships a built-in for exactly this:

df = pd.json_normalize(data)   # dicts -> dotted columns automatically

But note json_normalize by default flattens dicts only and leaves lists as cell values — the hand-rolled flatten here also explodes list indices into columns. Knowing where the stdlib/pandas shortcut stops, and why you'd still hand-roll, is the senior point.

The edge case — empty containers and key collisions

An empty {} or [] produces no entries — that column silently disappears for that row, which can drop data unexpectedly. Worse, a key literally containing a dot ({"a.b": 1}) is ambiguous against a real nested path {"a": {"b": 1}} — both yield column a.b and one overwrites the other. None values pass through as NaN.

The follow-up — index-based list columns are a schema landmine

"What happens when one record has 3 tags and another has 50?" You get 50 sparse columns. The interviewer wants you to recognize that arrays of homogeneous items usually belong in a long format (one row per tag) or a single delimited string, not index columns — i.e. when to not flatten.

The trap — treating lists as scalars

If flatten omitted the isinstance(obj, list) branch, a list would hit the else and be written as a single cell like ['vip', 'new'] — a stringified Python literal that no SQL loader can parse. Equally, falling into infinite recursion on a self-referential structure (rare in JSON, common in Python objects) crashes; JSON's acyclic guarantee is what makes the naive recursion safe.

Senior signal: handle dict and list recursion explicitly, then immediately question whether index-based list columns are the right model — a senior knows flattening is a schema decision, not just a code exercise, and names json_normalize while explaining its limits.
