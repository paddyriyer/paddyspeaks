An HR-analytics team wants the org chart flattened with a depth label — the CEO at depth 0, their directs at depth 1, and so on. A single self-join reaches only one level; an organisation of unknown depth needs a recursive CTE that walks the management tree until it runs out of employees. The interviewer is probing the anchor / recursive-member structure and what makes the recursion terminate.

1. The source data — employee

A four-deep tree. The NULL manager_id on the CEO is the edge case that matters: it is the only row with no parent, so it is the seed the recursion starts from. Mgr-A2 is a leaf with no reports — the recursion must not break when it reaches a node that has no children.

2. The SQL — WITH RECURSIVE

WITH RECURSIVE tree AS (
  SELECT id, name, 0 AS depth
  FROM employee
  WHERE manager_id IS NULL
  UNION ALL
  SELECT e.id, e.name, t.depth + 1
  FROM employee e
  JOIN tree t ON e.manager_id = t.id
)
SELECT id, name, depth
FROM tree
ORDER BY depth, id;

Two members joined by UNION ALL. The anchor seeds tree with the root — the employee whose manager_id IS NULL, stamped depth 0. The recursive member joins employee back to rows already in tree, finding everyone whose manager surfaced in the previous round, and labels them one level deeper.

4. Strategic explanations

How a recursive CTE executes — the termination question

The recursive member runs repeatedly. On each iteration, tree on the right of the join refers only to the rows produced by the previous iteration. The engine keeps iterating until an iteration adds zero new rows, then stops. That is the natural terminator: once the deepest employees are found, the next round's join matches nobody, the round is empty, and recursion ends. No loop counter is required — the tree's finite depth ends it.

UNION ALL vs UNION — not interchangeable

Recursive CTEs use UNION ALL deliberately. UNION deduplicates on every iteration — extra work, and more importantly it can mask a structural problem instead of surfacing it. In a clean tree each employee is reached exactly once, so there is nothing to dedupe and UNION ALL is both correct and faster. If the data were a graph with diamonds (an employee reachable by two paths), UNION ALL would reveal the duplication so you handle it consciously.

The infinite-loop trap — cycles in the data

Termination depends on the data being an actual tree. If a cycle exists — the CEO's manager_id pointed back at an IC, say — the recursive join keeps finding "new" rows forever and the query never terminates (or hits a recursion limit). Defensive recursion carries a visited-path array and a depth cap; PostgreSQL 14+ also offers a built-in CYCLE id SET is_cycle USING path clause for exactly this protection. Naming the cycle risk unprompted is the senior move.

Edge cases: depth 0 vs 1, multiple roots, and orphans

The anchor sets 0 AS depth because the prompt says the CEO is depth 0; seed it with 1 and every level shifts. Because the anchor is WHERE manager_id IS NULL, it seeds every root — correct for a company with several independent top-level executives (a forest). The opposite risk: an employee whose manager_id points to a non-existent or cyclic node never reaches a root and is silently absent. Reconcile COUNT(*) FROM tree against COUNT(*) FROM employee to catch any such orphan.

Senior signal: explain the anchor / recursive-member split and that recursion stops when an iteration yields zero new rows, justify UNION ALL over UNION, pre-empt the infinite loop a data cycle would cause with a visited-path guard or the CYCLE clause, and reconcile the row count against the table to catch orphans the anchor never reaches.
