Zoom's support analytics team wants to spot frustrated customers: callers who phoned back within 7 days of a previous call — a likely sign their first issue was not resolved. The data is just caller_id and call_at. The interview question is how you express "a later call within 7 days of an earlier one" without a window function, and the trap is a self-join that quietly counts a call against itself.

Caller 1 called back 3 days later — a repeat. Caller 2's two calls are 19 days apart — not a repeat. Caller 3 called once. Caller 4 is the edge case: Jan 2 then Jan 9 is exactly 7 days, and there are two calls on Jan 9 — testing whether the boundary is inclusive and whether same-day calls are handled.

2. The SQL — a self-join on the time window

SELECT DISTINCT a.caller_id FROM calls a JOIN calls b
ON a.caller_id = b.caller_id AND a.call_at < b.call_at
AND b.call_at <= a.call_at + INTERVAL '7 days';

The table is joined to itself: row a is the earlier call, row b a later one by the same caller, constrained to land within 7 days. DISTINCT collapses callers who qualify on multiple pairs.

Callers 1 and 4 are flagged. Caller 1's Jan 1 → Jan 4 pair is 3 days — inside the window. Caller 4 qualifies because Jan 2 → Jan 9 is exactly 7 days and the join uses <=, so the boundary is inclusive. Caller 2 (19-day gap) and caller 3 (one call) are correctly absent.

4. Strategic explanations

The self-join trap — a.call_at < b.call_at is load-bearing

If the join condition were a.call_at <= b.call_at or omitted the ordering entirely, every call would pair with itself (same row, same timestamp), making every caller a false positive. The strict < does two jobs: it enforces that b is genuinely after a, and it prevents the self-match. Getting < vs <= wrong here is the single most common mistake.

Same-timestamp calls — a subtler version of the trap

Caller 4 has two distinct calls at 14:00 and 15:00 on Jan 9. They are different rows, both satisfy a.call_at < b.call_at, fine. But if two rows shared the exact same call_at (a double-logged event), a.call_at < b.call_at is false for that pair, so they would not flag each other — usually correct (one event, not a repeat). If the table has a unique call_id, a more precise condition is a.call_id <> b.call_id AND a.call_at <= b.call_at, which handles ties explicitly.

The window-function alternative — O(n log n) vs O(n²)

The self-join compares every call against every other call by the same caller — quadratic within each caller. LAG sorts once and looks at just the previous call:

SELECT DISTINCT caller_id FROM (
  SELECT caller_id, call_at,
         call_at - LAG(call_at) OVER (PARTITION BY caller_id ORDER BY call_at) AS gap
  FROM calls
) t
WHERE gap <= INTERVAL '7 days';

One caveat: LAG only compares consecutive calls. That is fine for "called back within 7 days of the previous call", which is the natural reading. If the requirement were "within 7 days of any earlier call", consecutive comparison still suffices — if calls A and C are within 7 days, the intervening B is too — so LAG is both faster and correct here.

Counting NULL timestamps out

If call_at can be NULL, every inequality involving it is unknown, so a NULL-timestamp row neither flags nor gets flagged — it silently drops. That is probably acceptable, but state it; if not, filter WHERE call_at IS NOT NULL up front so the assumption is explicit.

"How many times" — the natural follow-up

To rank callers by how often they called back, drop DISTINCT and aggregate the qualifying pairs — or with the LAG version, COUNT(*) FILTER (WHERE gap <= INTERVAL '7 days') per caller. The window form extends to "average gap between calls" far more naturally than the self-join does.

Senior signal: recognise the strict a.call_at < b.call_at as essential — it both orders the pair and blocks the self-match — nail the inclusive <= 7-day boundary, and offer the LAG rewrite as the O(n log n) alternative while correctly arguing that consecutive comparison still answers the question.
