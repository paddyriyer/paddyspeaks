Intuit's procurement analytics team wants a simple catalog view: each product paired with the name of its supplier. The query is a textbook two-table join — but the interviewer dropped a product whose supplier_id points at a supplier that does not exist, and the choice of join type decides whether that orphaned product appears, vanishes silently, or gets flagged.

1. The source data — products and suppliers

Product 4, Orphan, has supplier_id = 99 — and there is no supplier 99. That dangling foreign key is the entire interview.

2. The SQL — INNER JOIN on supplier_id

SELECT p.name, s.supplier_name
FROM products p JOIN suppliers s ON p.supplier_id = s.supplier_id;

A plain JOIN (inner) emits a row only when a product's supplier_id finds a matching suppliers row.

3. The resulting analysis

Three rows — Orphan is gone. The inner join found no supplier 99, so it dropped the product without a word: no error, no NULL, no warning. The catalog shows 3 products when there are 4. If a row count or a revenue total depends on this view, the discrepancy is now invisible.

4. Strategic explanations

The silent row loss — INNER JOIN's quiet failure mode

An inner join answers "products that have a valid supplier", not "all products". When the requirement is the catalog — every product — that is a bug, and the nastiest kind: it produces a plausible-looking, smaller result. Use a LEFT JOIN to keep every product regardless of supplier match:

SELECT p.name, s.supplier_name
FROM products p
LEFT JOIN suppliers s ON p.supplier_id = s.supplier_id;

Now Orphan appears with supplier_name = NULL — the product is preserved and the missing supplier is visible rather than hidden.

Turning the NULL into a data-quality signal

A NULL supplier on a catalog row is itself a finding. Make it explicit:

SELECT p.name,
       COALESCE(s.supplier_name, '(unknown supplier)') AS supplier_name
FROM products p
LEFT JOIN suppliers s ON p.supplier_id = s.supplier_id;

Or invert it — WHERE s.supplier_id IS NULL after the left join lists exactly the products with broken supplier links, a useful report for the procurement team to fix the source data.

The duplicate-name column trap

suppliers ships both supplier_name and a plain name, and products also has a name. Select name unqualified in a join touching two tables that both have it and Postgres errors with column reference "name" is ambiguous. Always table-qualify — p.name, s.supplier_name — and confirm with the schema owner whether suppliers.name and suppliers.supplier_name are duplicates or hold different things (legal entity vs trading name).

Fan-out if supplier_id is not unique

The join is one-to-many only if suppliers.supplier_id is a true primary key. If the supplier table has duplicate ids (a dirty extract, or a versioned/SCD table), each product matches multiple supplier rows and the result fans out — more rows than products, silently double-counting. Confirm the join key is unique on the supplier side, or de-duplicate first with DISTINCT ON (supplier_id).

NULL supplier_id on the product side

A product whose own supplier_id is NULL never matches anything — NULL = anything is unknown — so an inner join drops it and a left join keeps it with a NULL supplier, same as the orphan. Either way, NULL join keys deserve a deliberate decision rather than a surprise.

Senior signal: recognise that an inner join silently drops the orphaned product and switch to LEFT JOIN to preserve the full catalog, turn the resulting NULL into a data-quality report, table-qualify every column when both tables expose a name, and verify the supplier-side key is unique so the join does not fan out.
