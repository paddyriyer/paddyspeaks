Uber's growth team keeps a history of which region each rider belongs to, appending a row every time a rider's region is reassigned. They want the riders who have ever moved — the region-switchers — because those accounts behave differently for pricing and marketing. The schema looks join-shaped, but the real question is set cardinality: a switcher is a customer whose region history contains more than one distinct region. An interviewer is checking whether you reach for COUNT(DISTINCT) and not a naive COUNT.

The trap is customer 102: two rows, same region. A re-confirmation of NA, or a duplicate load — either way 102 never actually switched. Customer 101 moved once, 103 moved twice, 104 has only a single row. A query that counts rows instead of distinct regions will wrongly flag 102.

2. The SQL — HAVING COUNT(DISTINCT region) > 1

SELECT customer_id FROM customer_regions
GROUP BY customer_id HAVING COUNT(DISTINCT region) > 1;

Group the history by customer, count how many distinct region values each one has, and keep only those with more than one. HAVING filters on the aggregate — it is the WHERE clause for grouped rows.

Exactly the two genuine switchers. Customer 102 is correctly excluded: it has two rows but COUNT(DISTINCT region) is 1. Customer 104 has a single row, count 1, also excluded. The DISTINCT is what makes 102 disappear — remove it and 102 falsely joins the list.

4. Strategic explanations

COUNT(*) vs COUNT(DISTINCT region) — the trap in one line

The wrong query is HAVING COUNT(*) > 1. That counts rows, so customer 102's duplicate NA entry pushes it to 2 and it gets reported as a switcher even though it never moved. COUNT(DISTINCT region) counts unique region values. The distinction is the entire question: "switched region" means the set of regions has size > 1, not the row count.

HAVING vs WHERE

You cannot write WHERE COUNT(DISTINCT region) > 1 — WHERE is evaluated before rows are grouped, so aggregates are not available there. HAVING runs after the GROUP BY, on the grouped result. The mental model: WHERE filters input rows, HAVING filters output groups.

NULL regions are silently ignored

COUNT(DISTINCT region) does not count NULL. A customer with history {EMEA, NULL} has COUNT(DISTINCT region) = 1 and is not flagged — even though a missing-then-known region might be a real transition. If a NULL should count as its own state, you need to handle it explicitly:

SELECT customer_id FROM customer_regions
GROUP BY customer_id
HAVING COUNT(DISTINCT COALESCE(region, '__none__')) > 1;

Decide whether NULL means "unknown, skip" or "a distinct state". The default behaviour silently picks the first.

Returning the switch path, not just the id

The interviewer's follow-up: "show me what they switched from and to." Order the history with a window function and read off the first and last region:

SELECT customer_id,
       string_agg(region, ' → ' ORDER BY effective_date) AS region_path
FROM customer_regions
GROUP BY customer_id
HAVING COUNT(DISTINCT region) > 1;

For 103 that yields EMEA → NA → APAC — the full journey, with the duplicate-free distinct test still gating who appears.

"Switched" vs "currently in a different region"

Subtle product question: a customer could move EMEA → APAC → EMEA and end up back where they started. COUNT(DISTINCT region) > 1 still flags them — they did switch, twice. If the business instead means "current region differs from original region", that is a different query: compare the earliest and latest rows by effective_date. Confirm which definition the growth team wants.

Senior signal: use COUNT(DISTINCT region) so duplicate same-region rows do not masquerade as switches, explain why the aggregate filter belongs in HAVING not WHERE, flag that COUNT(DISTINCT) drops NULLs, and clarify whether "switched" means "ever changed" or "ends up different from where it started".
