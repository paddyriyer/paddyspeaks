Datadog's billing team stores customer addresses as four separate columns — street, city, state, zip — and the invoice renderer wants one display string. The naive answer is to glue them with ||. The interviewer's real interest is what happens when one of those columns is NULL, because in any production customer table, some of them always are.

1. The source data — customers

customer_idstreetcitystatezip
1123 Main StBostonMA02101
2500 Oak AveAustinTX78701
3NULLDenverCO80201
49 Elm RdNULLNULL10001

Customer 3 is missing a street; customer 4 is missing both city and state. Those NULLs are the interview — they decide whether you get a clean address or one peppered with stray commas.

2. The SQL — CONCAT_WS

SELECT customer_id,
  CONCAT_WS(', ', street, city, state, zip) AS full_address
FROM customers;

CONCAT_WS — "concatenate with separator" — takes the delimiter as its first argument and joins the rest. Its defining behaviour: it skips NULL arguments entirely, and crucially does not emit a separator for a skipped value.

3. The resulting analysis

customer_idfull_address
1123 Main St, Boston, MA, 02101
2500 Oak Ave, Austin, TX, 78701
3Denver, CO, 80201
49 Elm Rd, 10001

Customer 3 produces Denver, CO, 80201 with no leading comma where the street would be. Customer 4 jumps straight from street to zip with a single separator. CONCAT_WS closed the gaps for you — no double commas, no dangling delimiters.

4. Strategic explanations

Why plain || is the trap answer

The instinctive solution is string concatenation:

SELECT customer_id,
  street || ', ' || city || ', ' || state || ', ' || zip AS full_address
FROM customers;

In Postgres, || propagates NULL: any NULL operand makes the entire expression NULL. Customer 3 and customer 4 would both come back with a completely blank full_address — one missing field nukes the whole string. That is the single most common mistake on this question.

Why CONCAT (no _WS) is also wrong here

Plain CONCAT(street, ', ', city, ', ', state, ', ', zip) does treat NULL as an empty string — so it never blanks the whole row. But it cannot drop the separator next to a missing value. Customer 3 would render , Denver, CO, 80201 with an ugly leading comma; customer 4 would show 9 Elm Rd, , , 10001. CONCAT_WS is the only one of the three that omits both the value and its delimiter.

The edge case CONCAT_WS still does not solve: empty strings

CONCAT_WS skips NULL, but an empty string '' is a real value — it is kept, and you get a doubled separator Denver, , CO. Dirty data often has '' instead of NULL. Normalise first:

CONCAT_WS(', ',
  NULLIF(TRIM(street), ''), NULLIF(TRIM(city), ''),
  NULLIF(TRIM(state), ''), NULLIF(TRIM(zip),  ''))

NULLIF(x, '') converts a blank (after TRIM removes whitespace-only values) back into a NULL that CONCAT_WS will then drop.

The all-NULL row

If every address column is NULL, CONCAT_WS returns an empty string '', not NULL. A consumer that tests full_address IS NULL to detect "no address on file" will miss those rows. If you need a true NULL for a fully-empty address, wrap it: NULLIF(CONCAT_WS(', ', …), '').

Senior signal: reach for CONCAT_WS immediately and explain that it is the only join that drops both a NULL value and its separator, name the || NULL-propagation trap that blanks an entire address, and pre-empt the empty-string case with NULLIF(TRIM(…), '') rather than assuming missing data is always NULL.
