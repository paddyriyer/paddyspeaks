This running-balance query is a realistic, well-written pattern — signed amounts, a windowed cumulative sum, LAG for the opening balance. It is also subtly broken: on any account with more than one transaction on the same date, balance_after is non-deterministic. The cause is a ROWS frame driven by an ORDER BY whose key, transaction_date, is not unique.

Three transactions land on 2025-03-02. A debit subtracts, a credit adds.

2. The trap — ordering by a non-unique key

SUM(signed_amount) OVER (
  PARTITION BY account_id
  ORDER BY transaction_date          -- not unique: 3 rows share 2025-03-02
  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
) AS balance_after

A ROWS frame sums every row from the partition start up to “this” one — but among the three 2025-03-02 rows, the query gives SQL no way to decide which is first, second, or third. The engine may visit them in any order, so each same-day row’s balance_after is an arbitrary partial sum. The opening balance (500) and the final same-day balance (300) are stable; the two values in between are not. balance_before, computed with LAG over the same ordering, inherits the same instability.

3. The correct query — add a unique tiebreaker

WITH signed AS (
  SELECT txn_id, account_id, transaction_date, amount, transaction_type,
         CASE WHEN transaction_type = 'debit' THEN -amount ELSE amount END AS signed_amount
  FROM transactions
),
running AS (
  SELECT txn_id, account_id, transaction_date, amount, transaction_type,
         SUM(signed_amount) OVER (
           PARTITION BY account_id
           ORDER BY transaction_date, txn_id
           ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
         ) AS balance_after
  FROM signed
)
SELECT txn_id, account_id, transaction_date, transaction_type, ABS(amount) AS amount,
       COALESCE(LAG(balance_after) OVER (PARTITION BY account_id
                                         ORDER BY transaction_date, txn_id), 0) AS balance_before,
       balance_after
FROM running
ORDER BY account_id, transaction_date, txn_id;

4. Strategic explanations

Why ROWS over a non-unique key is non-deterministic

A ROWS frame is defined by physical row position in the window’s sort order. When the ORDER BY key has ties, that sort order is only partially defined — the engine is free to arrange tied rows however it likes, and may even change between runs, plans, or after a vacuum. Every per-row cumulative value across a tie group is therefore unstable. Only the value at the last row of the tie group (and beyond) is well defined.

The fix: a tiebreaker that totally orders the rows

Appending a unique column — here txn_id — to the window’s ORDER BY makes the row order total and deterministic. A ledger table should always carry such a key: an autoincrement id or a per-account sequence number. Without one, a “running balance” is fundamentally ill-defined for same-day activity, and no window clause can rescue it.

RANGE would be deterministic — but wrong here

Switching the frame to RANGE removes the non-determinism, because RANGE assigns every tied row the same value: the balance through the end of that date. But that is an end-of-day balance, not a per-transaction one — all three 2025-03-02 rows would read 300. For a transaction-level ledger you need ROWS plus a tiebreaker, not RANGE.

Dialect notes

The non-determinism is standard behaviour everywhere — PostgreSQL, Snowflake, BigQuery, SQL Server, Oracle, MySQL 8.0. ROWS UNBOUNDED PRECEDING is accepted as shorthand for ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW in all of them. None will warn you about the missing tiebreaker; the query runs and returns plausible-looking numbers.

Senior signal: a windowed running total over a non-unique ORDER BY key is non-deterministic for the tied rows — the engine may order them arbitrarily, so intermediate balance_after (and LAG-derived balance_before) values are unstable. Always add a unique tiebreaker (a transaction id or sequence) to the window’s ORDER BY; switching to RANGE hides the symptom but yields end-of-day, not per-transaction, balances.
