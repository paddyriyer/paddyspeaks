A product team measuring engagement wants to flag "streak" users — people who logged in three or more days in a row — for a retention cohort. The interviewer pins the implementation: solve it with LAG by checking whether the two preceding login rows fall exactly one and two calendar days back. The probe is whether you understand that a gap of one missing day breaks the streak, and that LAG walks rows, not dates.

User 2 is the trap: three login rows, but the Feb 2 gap means the longest consecutive run is only two days. User 4 has just two rows — the degenerate too-short group. Only users 1 and 3 ever have three calendar days in a row.

2. The SQL — LAG over a 3-row window

WITH dt AS (
  SELECT DISTINCT user_id, login_date FROM login
),
shifted AS (
  SELECT user_id, login_date,
         LAG(login_date, 1) OVER (PARTITION BY user_id ORDER BY login_date) AS d1,
         LAG(login_date, 2) OVER (PARTITION BY user_id ORDER BY login_date) AS d2
  FROM dt
)
SELECT DISTINCT user_id
FROM shifted
WHERE login_date - d1 = 1 AND login_date - d2 = 2
ORDER BY user_id;

LAG(login_date, 1) and LAG(login_date, 2) pull the previous and second-previous login dates within each user, ordered by date. If the current row is the tip of a three-in-a-row run, those two neighbours are exactly 1 and 2 days behind — so login_date - d1 = 1 AND login_date - d2 = 2 marks a streak.

3. The resulting analysis

user_idsignal

3Mar 12 (and Mar 13) satisfy the 3-row pattern — streak

Users 2 and 4 are absent. User 2's Feb 3 row has d1 = Feb 1, so login_date - d1 = 2, not 1 — the missing Feb 2 row poisons the pattern. User 4 never has a second-previous row at all, so d2 is NULL and the predicate is never true.

4. Strategic explanations

Why LAG compares rows, and why DISTINCT first matters

LAG(col, n) returns the value n rows earlier in the ordered partition — it counts rows, not days. The arithmetic login_date - d1 then converts that row offset into a day difference. The dt CTE's SELECT DISTINCT is essential: if a user could log in twice on the same date, two identical rows would make LAG see a zero-day gap and the whole 3-row alignment would shift. The question says logins are distinct daily, but normalising first makes the query robust regardless.

The boundary rows are NULL — and that is correct

The first row of every partition has no predecessor, so d1 is NULL; the second row has no second-predecessor, so d2 is NULL. Any comparison against NULL yields unknown, which WHERE treats as false — so those rows simply cannot be flagged. That is exactly right: a streak of three needs a third row to even be evaluated. User 4 with two logins is filtered out for free, no special-casing required.

One streak day is enough — DISTINCT on the output

In user 3's 4-day run, both Mar 12 and Mar 13 satisfy the 3-row pattern, so the user matches twice. The outer SELECT DISTINCT user_id collapses that to one row. The question asks "which users", not "how many streak days", so de-duplication is the right final step — though a GROUP BY user_id would do the same job.

The gaps-and-islands alternative

The fixed 3-row LAG pattern answers "exactly 3 consecutive" cheaply, but it does not generalise: "5 or more in a row" would need five LAG calls. The scalable technique is gaps-and-islands — subtract ROW_NUMBER() from the date to give every consecutive run a constant group key, then GROUP BY that key and filter COUNT(*) >= N. Mention it as the general tool; the LAG form here is the pragmatic answer for a small fixed N.

Dialect — date subtraction is not portable

login_date - d1 = 1 works because PostgreSQL subtracts two dates into an integer number of days. SQL Server needs DATEDIFF(day, d1, login_date) = 1; MySQL uses DATEDIFF(login_date, d1) = 1; Oracle subtracts dates into a number like Postgres. The LAG window function itself is ANSI-standard and portable — only the date arithmetic needs translation.

Senior signal: use LAG(date, 1) and LAG(date, 2) to inspect the two preceding rows, convert the row offset to a day gap with date subtraction, rely on the NULL boundary rows to exclude users with too few logins automatically, de-duplicate the final user list, and name gaps-and-islands as the technique that scales when N is not fixed.
