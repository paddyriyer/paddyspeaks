SRE Monday morning. The infrastructure team committed to a 99.9% uptime SLO across the fleet. The CTO asks: "Which servers are missing the SLO this quarter? They're our top maintenance candidates." The query is conditional aggregation — but the operational nuance (sample interval, weighted uptime, scheduled vs unscheduled downtime) is what separates a junior answer from a Staff SRE answer.

2. The SQL — conditional-aggregation uptime ratio

SELECT
  server_id,
  100.0 * SUM(CASE WHEN status = 'up' THEN 1 ELSE 0 END) / COUNT(*) AS uptime_pct
FROM server_status
GROUP BY server_id;

4. Strategic explanations

What 99.9% actually means in minutes per year

SRE language uses "nines": 99% = 87.6 hours of downtime/year; 99.9% = 8.76 hours; 99.99% = 52.6 minutes; 99.999% = 5.26 minutes. Every additional nine costs ~10× more engineering. Knowing the conversion is a senior signal.

The big trap — sampling interval bias

The query above counts samples, not seconds. If samples are uniformly spaced (every 60 seconds), it's roughly correct. If samples are bursty — say the agent crashes during outages and only resumes after recovery — you'll overstate uptime because the down period had fewer samples logged. Production fleet-uptime queries weight by inter-sample duration:

WITH durations AS (
  SELECT server_id, status, ts,
    LEAD(ts) OVER (PARTITION BY server_id ORDER BY ts) AS next_ts
  FROM server_status
)
SELECT server_id,
  100.0 * SUM(CASE WHEN status = 'up'
                   THEN EXTRACT(EPOCH FROM (next_ts - ts)) END)
        / SUM(EXTRACT(EPOCH FROM (next_ts - ts))) AS time_weighted_uptime_pct
FROM durations
WHERE next_ts IS NOT NULL                 -- skip the open last-sample
GROUP BY server_id;

This is the real uptime — what the customer-facing SLA actually measures.

Scheduled vs unscheduled downtime

SLAs typically exclude scheduled maintenance windows (you announced them). Production queries join maintenance_windows and exclude samples inside those:

WHERE NOT EXISTS (
  SELECT 1 FROM maintenance_windows m
  WHERE m.server_id = s.server_id
    AND s.ts BETWEEN m.start_ts AND m.end_ts
)
Per-region / per-tier breakdowns

Single-server uptime is operational. Fleet uptime is what the CTO reports. Aggregate by region or tier for the executive dashboard:

SELECT region, server_tier,
  100.0 * SUM(CASE WHEN status = 'up' THEN 1 ELSE 0 END) / COUNT(*) AS uptime_pct
FROM server_status JOIN servers USING (server_id)
GROUP BY region, server_tier
HAVING 100.0 * SUM(CASE WHEN status = 'up' THEN 1 ELSE 0 END) / COUNT(*) < 99.9
ORDER BY uptime_pct ASC;
-- Lists every (region × tier) bucket missing the SLO. Sorted by worst.
Error budget — the SRE framing

SREs don't track uptime, they track error budget: time you're allowed to be down before breaking SLA. 99.9% × 30 days = 43.2 minutes of error budget per month. Burn it slowly with reliability work; burn it fast with a feature push that crashes things. The query becomes:

WITH up AS (
  SELECT server_id,
    SUM(CASE WHEN status = 'down' THEN 60 ELSE 0 END) AS down_seconds_this_month
  FROM server_status
  WHERE ts >= DATE_TRUNC('month', CURRENT_DATE)
  GROUP BY server_id
)
SELECT server_id, down_seconds_this_month,
  43 * 60 - down_seconds_this_month AS error_budget_remaining_seconds
FROM up;
-- Negative remaining = you blew the budget; freeze deploys.
Senior signal: recite the nines→time conversion (99.9% = 8.76h/yr), name the sampling-interval bias unprompted, distinguish scheduled vs unscheduled downtime, and pre-empt the error-budget framing. The time-weighted variant is the over-the-bar move — it's what real SRE on-call playbooks use.
