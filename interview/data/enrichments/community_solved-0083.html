A retention analyst wants the repeat customers — everyone who has placed more than one order — together with their lifetime spend. This is a deceptively simple GROUP BY + HAVING question, and the interviewer is watching whether you put the count condition in HAVING (not WHERE) and whether you understand how the join type and NULL amounts interact with COUNT and SUM.

Dara has two orders but one has a NULL amount, and Elena has none — two edge cases that probe how the aggregates behave.

2. The SQL — GROUP BY with a HAVING count filter

SELECT c.customerid, c.customername,
       SUM(o.amount) AS total_amount
FROM customer_info c
LEFT JOIN order_info o ON c.customerid = o.customerid
GROUP BY c.customerid, c.customername
HAVING COUNT(o.orderid) > 1;

Treating that NULL as $0 happens to give the same number here, but the two are not the same thing.

4. Strategic explanations

Why the filter belongs in HAVING, not WHERE

WHERE filters individual rows before grouping; HAVING filters whole groups after aggregation. "More than one order" is a property of a group, not a row — you cannot know a customer's order count until the rows are collapsed. Putting COUNT(...) > 1 in WHERE is a syntax error in every engine. This WHERE-vs-HAVING distinction is the single thing the question exists to test.

COUNT(o.orderid) vs COUNT(*) — the LEFT JOIN interaction

This matters because of the LEFT JOIN. Elena has no orders, so her joined row is one row of all-NULL order columns. COUNT(*) would count that phantom row as 1; COUNT(o.orderid) ignores it because orderid is NULL — Elena's count is 0, and 0 > 1 is false, so she is excluded. COUNT(*) would have wrongly given her a count of 1. With a LEFT JOIN, always count a column from the right-hand (optional) table.

SUM ignores NULL — the Dara trap

SUM(o.amount) skips NULL amounts rather than treating them as 0. Dara's total is $250, not "$250 plus an unknown". If a NULL amount means a genuinely free order, that is fine; if it means a data-quality hole, the total is understating reality. And a customer whose orders are all NULL-amount would get SUM = NULL, not 0. Decide explicitly: SUM(COALESCE(amount, 0)) if NULL should mean zero, or surface a COUNT(amount) alongside to expose how many amounts were missing.

Why LEFT JOIN is harmless here but worth a comment

An INNER JOIN would produce the identical result, because the HAVING COUNT > 1 filter discards every zero-order customer anyway — and those are exactly the rows LEFT JOIN adds. The LEFT JOIN is therefore redundant for this specific question. It only becomes meaningful if the requirement flips to "show all customers, including those with one or zero orders" — then the join type carries real weight. Naming when a join choice is and is not load-bearing is a senior tell.

What to say next — the correlated-subquery alternative and ties

The same result can be had with a correlated subquery in WHERE (WHERE (SELECT COUNT(*) FROM order_info o WHERE o.customerid = c.customerid) > 1), but that re-scans orders per customer — the GROUP BY/HAVING form aggregates in one pass and is the better answer. Also mention extensibility: the same HAVING slot can carry richer conditions like COUNT(*) > 1 AND SUM(amount) > 1000 for "repeat and high-value" customers.

Senior signal: place the count filter in HAVING (group-level), not WHERE (row-level); use COUNT(o.orderid) not COUNT(*) so a LEFT JOIN's phantom NULL row is not miscounted; and flag that SUM silently drops NULL amounts — COALESCE if zero is the intended meaning.
