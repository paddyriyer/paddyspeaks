Citi's reconciliation team is comparing two quarters of order books: which orders were placed in Q1 but did not recur in Q2? It is a textbook set-difference, and an interviewer asking it wants to see that you reach for EXCEPT rather than hand-rolling a join — and that you know what EXCEPT does with duplicates and NULLs.

1. The source data — orders_q1 and orders_q2

orders_q1.order_id orders_q2.order_id
1 2
2 4
3 6
4 8
5 9
6
7

Q1 holds orders 1–7; Q2 holds 2, 4, 6, 8, 9. The overlap is 2, 4, 6. Orders 8 and 9 are Q2-only — a deliberate distractor, because EXCEPT is directional and must ignore them.

2. The SQL — EXCEPT

SELECT order_id FROM orders_q1
EXCEPT
SELECT order_id FROM orders_q2;

EXCEPT returns every distinct row from the first query that does not appear in the second. It is one-directional: rows unique to orders_q2 (8 and 9) are simply not considered.

3. The resulting analysis

order_id
3
5
7
1

Orders 1, 3, 5, 7 — the Q1 entries with no Q2 match. Orders 2, 4, 6 were removed (they recur); 8 and 9 never appear because they are not in Q1. Note the result is unordered — EXCEPT guarantees no row order, so add ORDER BY if the report needs one.

4. Strategic explanations

EXCEPT is directional — and that is the point

Swapping the operands answers a different question:

SELECT order_id FROM orders_q2
EXCEPT
SELECT order_id FROM orders_q1;
-- returns 8, 9 — orders new in Q2

"In A not B" and "in B not A" are both legitimate; confirm which side the analyst means. If they want both — the full symmetric difference — that is a different construction (a FULL JOIN … WHERE one side IS NULL, or two EXCEPTs unioned).

EXCEPT de-duplicates — use EXCEPT ALL to keep multiplicity

Plain EXCEPT applies DISTINCT to both inputs: if Q1 listed order 3 twice, the result still shows a single 3. When row counts matter — reconciling quantities, not just identities — EXCEPT ALL does multiset subtraction: three copies in A minus one in B leaves two. Know which semantics the task wants before you pick the keyword.

The anti-join equivalent — and when to prefer it

EXCEPT is the cleanest expression of intent, but two rewrites do the same job and scale better when you need extra columns:

-- NOT EXISTS — usually the planner's favourite
SELECT q1.order_id FROM orders_q1 q1
WHERE NOT EXISTS (SELECT 1 FROM orders_q2 q2 WHERE q2.order_id = q1.order_id);

-- LEFT JOIN ... IS NULL — handy when you also want q1's other columns
SELECT q1.order_id FROM orders_q1 q1
LEFT JOIN orders_q2 q2 ON q1.order_id = q2.order_id
WHERE q2.order_id IS NULL;

Key difference: EXCEPT forces the output to be distinct and the two SELECT lists to match in shape; the anti-joins let you carry q1.customer_id, q1.amount, etc. through to the result. Reach for EXCEPT for pure set logic, the anti-join when you need a wider row.

The NULL trap — NOT IN is the wrong tool

A tempting fourth option is WHERE order_id NOT IN (SELECT order_id FROM orders_q2). If any row in orders_q2 has a NULL order_id, NOT IN returns no rows at all — the NULL makes every comparison unknown. EXCEPT sidesteps this: it treats two NULLs as equal for set purposes, so a NULL in Q1 is correctly removed if Q2 also has one. This NULL-safety is a genuine reason to prefer EXCEPT or NOT EXISTS over NOT IN.

Column compatibility and ordering

Both sides of EXCEPT must have the same number of columns with compatible types, matched by position, not name. And the result has no defined order — the 1, 3, 5, 7 above could come back in any sequence. Wrap with ORDER BY order_id for a stable report.

Senior signal: use EXCEPT for the set-difference, state that it de-duplicates (reach for EXCEPT ALL when counts matter), know it is directional, and call out that it is NULL-safe where NOT IN silently collapses to zero rows.
