A webhook delivery table is a persistent work queue backed by Postgres: it must handle exponential backoff, bounded retry counts, and eventual dead-lettering, all without a separate message broker. The interviewer is probing your understanding of polling-queue patterns (SELECT ... FOR UPDATE SKIP LOCKED is the key), exponential backoff math (the next_attempt_at column is a computed future timestamp), idempotency (the same event must not be delivered twice to the same endpoint), and partial indexes that make the scheduler query fast on a table that grows to millions of dead rows.

1. The requirements — what the design must capture

  • Delivery record per (endpoint, event): one row tracks all attempts to deliver one event to one endpoint; the UNIQUE constraint on (endpoint_id, event_id) prevents re-queuing the same delivery.
  • Retry state machine: status progresses pending → succeeded or pending → failed → dead; each failed attempt increments attempt_count and sets next_attempt_at to an exponentially growing future time.
  • Scheduler query: the worker polls for WHERE status = 'pending' AND next_attempt_at <= now(); a partial index on this predicate makes the poll fast regardless of how large the dead/succeeded tail is.
  • Dead-lettering: after max_attempts, the status transitions to 'dead'; the row itself is the dead-letter record — no separate DLQ table needed.
  • Last-N-hours query for monitoring: WHERE endpoint_id = ? AND status = 'failed' AND last_attempt_at >= now() - INTERVAL '1 day'; the second index on (endpoint_id, status, last_attempt_at DESC) serves this.

2. The DDL — the CREATE TABLE

CREATE TABLE webhook_delivery (
  delivery_id       BIGSERIAL    PRIMARY KEY,
  endpoint_id       BIGINT       NOT NULL,                       -- FK to webhook_endpoint(s) elsewhere
  event_id          VARCHAR(128) NOT NULL,                       -- the upstream business event being notified
  event_type        VARCHAR(64)  NOT NULL,
  payload_jsonb     JSONB        NOT NULL,
  attempt_count     INT          NOT NULL DEFAULT 0,
  max_attempts      INT          NOT NULL DEFAULT 8,             -- 2^8 minutes ≈ 4 hours max backoff
  status            VARCHAR(16)  NOT NULL DEFAULT 'pending',     -- 'pending','succeeded','failed','dead'
  last_attempt_at   TIMESTAMPTZ,
  next_attempt_at   TIMESTAMPTZ  NOT NULL DEFAULT now(),         -- scheduler picks rows where next_attempt_at <= now
  last_http_status  SMALLINT,
  last_error        TEXT,
  created_at        TIMESTAMPTZ  NOT NULL DEFAULT now(),
  completed_at      TIMESTAMPTZ,                                  -- terminal state timestamp
  UNIQUE (endpoint_id, event_id)                                  -- idempotency: don't re-deliver same event to same endpoint
);

CREATE INDEX ix_webhook_pending ON webhook_delivery (next_attempt_at)
  WHERE status = 'pending';
CREATE INDEX ix_webhook_endpoint_status_time ON webhook_delivery (endpoint_id, status, last_attempt_at DESC);

4. Strategic explanations

The partial index is the critical performance decision

The ix_webhook_pending partial index on (next_attempt_at) WHERE status = 'pending' is the heart of the design. Without it, the scheduler query must scan all rows — including potentially millions of succeeded and dead rows — to find the pending ones due for delivery. With the partial index, the index contains only pending rows, typically a tiny fraction of the total. The worker's poll query WHERE status='pending' AND next_attempt_at <= now() becomes an index range scan on this small structure, completing in microseconds regardless of historical row count.

FOR UPDATE SKIP LOCKED for concurrent workers

Multiple worker processes polling the same table simultaneously would compete to process the same rows. SELECT ... FOR UPDATE SKIP LOCKED causes each worker to atomically lock a batch of rows and skip any that are already locked by another worker. This distributes work across workers without serialization errors or starvation, and without an external coordination layer. After processing, the worker updates the row's status and next_attempt_at within the same transaction — on commit, the lock is released.

Backoff formula and jitter

next_attempt_at = now() + INTERVAL '1 minute' * power(2, attempt_count) + random_jitter(0..30s). The exponential component (1, 2, 4, 8, 16 … minutes) gives the endpoint time to recover. The random jitter is essential: without it, all deliveries that failed at the same time would retry simultaneously, creating a retry thundering herd that can overwhelm the endpoint. Adding a uniformly random 0–30 second jitter spreads retries over a window. This is the standard pattern used by AWS SQS, Stripe, and Twilio.

Dead-lettering: the row is the dead-letter record

When attempt_count >= max_attempts, the worker sets status = 'dead' and completed_at = now(). The row itself contains the full history (last_http_status, last_error, attempt_count, the original payload) — it is its own dead-letter record. A separate dead-letter table would duplicate this data. An alert on WHERE status = 'dead' AND completed_at >= now() - INTERVAL '1 hour' triggers on-call if dead-letter rate spikes.

Dialect notes

FOR UPDATE SKIP LOCKED is supported in Postgres (since 9.5), MySQL (since 8.0), and Oracle. Partial indexes are Postgres-specific; in MySQL, use a filtered index on a generated column. For very high throughput (millions of events/hour), consider partitioning webhook_delivery by day on created_at and moving old partitions to cold storage — the pending partial index naturally stays small but old terminal rows accumulate indefinitely without pruning.

Senior signal: the partial index on next_attempt_at WHERE status = 'pending' is the load-bearing decision — without it the scheduler degrades linearly with history; use FOR UPDATE SKIP LOCKED for concurrent-safe polling without a broker; and add random jitter to the backoff formula to prevent retry thundering-herds when a batch of deliveries fails simultaneously.
