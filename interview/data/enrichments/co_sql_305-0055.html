AMD's supply-chain analytics group tracks how fast each product line converts inventory into sales — the inventory turnover ratio, cost of goods sold divided by average inventory. It is a textbook division, and the interviewer knows it. What they are really testing is whether you defend the denominator: a product with zero average inventory will detonate a plain division.

1. The source data — inventory

product_idcogsavg_inventory

Product 3 has zero average inventory — a discontinued SKU, or a drop-ship line that never sits in a warehouse. That single 0 is the whole interview: it is the value that turns division into a runtime error.

2. The SQL — cogs / NULLIF(avg_inventory, 0)

SELECT product_id, cogs / NULLIF(avg_inventory,0) AS turnover_ratio
FROM inventory;

NULLIF(avg_inventory, 0) returns NULL when the inventory is exactly 0 and the value itself otherwise. Dividing by NULL yields NULL — no exception — so product 3 produces a blank instead of crashing the query.

Products 1 and 2 both turn over 4 times; product 4, twice. Product 3 returns NULL — the query completed and every other row is intact. Without NULLIF the whole statement would have aborted with division by zero, taking the three valid answers down with it.

4. Strategic explanations

Why NULLIF is the right guard

Postgres raises an error on integer or numeric division by zero — it does not return infinity or NULL on its own. NULLIF(x, 0) is the idiomatic one-token fix: it converts the dangerous 0 into a NULL, and NULL propagates harmlessly through arithmetic. The alternative, a CASE, is more verbose but identical in effect:

SELECT product_id,
  CASE WHEN avg_inventory = 0 OR avg_inventory IS NULL
       THEN NULL
       ELSE cogs / avg_inventory END AS turnover_ratio
FROM inventory;

Reach for NULLIF for brevity; reach for CASE when the zero case needs its own non-NULL sentinel.

Integer division — the silent truncation trap

If cogs and avg_inventory are INTEGER columns, cogs / avg_inventory does integer division: 50000 / 30000 becomes 1, not 1.67. The ratio is silently truncated and nobody errors. Cast one operand to force real division:

SELECT product_id,
  cogs::numeric / NULLIF(avg_inventory, 0) AS turnover_ratio
FROM inventory;

Here the columns are numeric so the result is exact — but on an unknown schema, casting is cheap insurance.

What does a NULL ratio mean to the consumer?

A NULL turnover is honest — "undefined, the divisor was zero" — but a dashboard may want a number. If zero inventory should read as zero turnover, COALESCE(cogs / NULLIF(avg_inventory,0), 0). If it should be flagged, keep the NULL and let the UI render "n/a". What you must not do is let it become a crash. Decide the semantics with the analyst.

Negative or NULL inputs

NULLIF guards zero, not negative. A negative avg_inventory (a data-entry error) divides fine and yields a nonsensical negative ratio that no guard catches — consider WHERE avg_inventory > 0 or a data-quality check upstream. And if avg_inventory is already NULL, the division is NULL anyway; NULLIF simply makes the zero case behave the same as the missing case.

Rounding for presentation

The raw result carries 16 decimal places. A turnover ratio is a business KPI read by humans — ROUND(cogs::numeric / NULLIF(avg_inventory,0), 2) trims it to two places. Keep full precision in the warehouse layer; round only at the presentation edge.

Senior signal: guard every division with NULLIF(divisor, 0) before an interviewer has to ask, explain that Postgres errors rather than returning NULL on divide-by-zero, pre-empt integer-division truncation with a cast, and decide deliberately whether an undefined ratio should surface as NULL or be coalesced to a number.
