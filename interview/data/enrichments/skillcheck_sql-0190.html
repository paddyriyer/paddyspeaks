A SaaS product team wants to understand how many users from each signup cohort came back exactly one month later — the M1 retention rate that is typically the most actionable early signal of product stickiness. The interviewer is probing conditional aggregation via LEFT JOIN on a date-arithmetic predicate, with the twist that cohorts with zero M1 retention must still appear in the output (no silent row drops).

2. The SQL — cohort size + LEFT JOIN on date arithmetic

WITH cohort AS (
  SELECT signup_month, COUNT(*) AS cohort_size FROM signup GROUP BY signup_month
),
retained AS (
  SELECT s.signup_month,
         COUNT(DISTINCT a.user_id) AS retained_m1
  FROM signup s
  LEFT JOIN activity a
    ON a.user_id = s.user_id
    AND a.activity_month = (s.signup_month + INTERVAL '1 month')::date
  GROUP BY s.signup_month
)
SELECT c.signup_month,
       c.cohort_size,
       r.retained_m1,
       ROUND((100.0 * r.retained_m1 / c.cohort_size)::numeric, 2) AS retention_pct_m1
FROM cohort c
JOIN retained r USING (signup_month)
ORDER BY c.signup_month;

4. Strategic explanations

Why LEFT JOIN (not INNER JOIN) in the retained CTE

The LEFT JOIN from signup to activity ensures that every user in the signup table appears in the GROUP BY, even if they have no matching M1 activity row. For unretained users the activity columns are NULL; COUNT(DISTINCT a.user_id) on a NULL column returns 0. An INNER JOIN would silently drop cohorts (or individual users within cohorts) that have zero M1 activity, causing the GROUP BY to produce a missing row for that cohort.

The date arithmetic predicate (signup_month + INTERVAL '1 month')::date

Adding an INTERVAL to a DATE in PostgreSQL yields a TIMESTAMP; the ::date cast normalizes it back to a date. This correctly handles month-end boundaries: adding 1 month to 2024-01-31 gives 2024-02-29 (a valid date in 2024), or 2024-02-28 in non-leap years — but since signup_month is always the first of the month, the result is always the first of the next month, avoiding edge cases.

COUNT(DISTINCT a.user_id) vs COUNT(*)

The activity table guarantees one row per (user_id, activity_month), so COUNT(*) and COUNT(DISTINCT a.user_id) are equivalent given the schema. Using DISTINCT is defensive: if a future schema change allows multiple activity rows per month, the query remains correct. In a live interview, noting this trade-off shows awareness of data quality assumptions.

100.0 multiplier for integer promotion

Both retained_m1 and cohort_size are integers. Writing 100 * retained_m1 / cohort_size would truncate to integer under integer division (PostgreSQL: 100 * 2 / 4 = 50, but 100 * 1 / 3 = 33, dropping the decimal). The 100.0 literal forces the multiplication to produce a NUMERIC, making the subsequent division exact before rounding.

Dialect notes

PostgreSQL + INTERVAL '1 month' with a ::date cast is clean. In BigQuery, use DATE_ADD(signup_month, INTERVAL 1 MONTH). In Snowflake, use DATEADD('month', 1, signup_month). In SQL Server, use DATEADD(month, 1, signup_month). All return a date type natively. MySQL 8 supports DATE_ADD(signup_month, INTERVAL 1 MONTH).

Senior signal: use a LEFT JOIN from the signup table to activity (not the other way) to preserve cohorts with zero retention; note that COUNT(DISTINCT a.user_id) correctly returns 0 for non-matching LEFT JOIN rows; and use 100.0 * integer / integer to force decimal division before rounding.
