LinkedIn's lifecycle-marketing team wants the list of customers who signed up but never placed an order — the audience for a "your first order is on us" win-back campaign. This is the canonical anti-join: rows in one table with no match in another. The interview question is whether you know the LEFT JOIN ... IS NULL idiom, and — more sharply — which column you check for NULL.

Five customers; only customers 1 and 3 have ever ordered (customer 1 twice). So Ben, Dan, and Eve — ids 2, 4, 5 — are the never-ordered set. That is the answer the query must produce.

2. The SQL — LEFT JOIN ... WHERE NULL

SELECT c.customer_id, c.name
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.order_id IS NULL;

The LEFT JOIN keeps every customer; customers with no order get NULLs in all o.* columns. The WHERE then keeps only those NULL-padded rows — the customers with no match.

Three customers, exactly the never-ordered set. Customers 1 and 3 matched real order rows, so their o.order_id was non-NULL and the WHERE filtered them out. Ben, Dan, and Eve produced only NULL-padded rows and survived the filter.

4. Strategic explanations

Why WHERE runs after the join — and why that is not a self-contradiction

It looks paradoxical: a LEFT JOIN exists to keep unmatched rows, then WHERE throws matched rows away. The order is what makes it work. The LEFT JOIN produces the full padded result first — matched rows with real values, unmatched rows with NULLs. Only then does WHERE o.order_id IS NULL run, selecting precisely the unmatched rows. Filtering for the NULL is how you isolate "no match". This LEFT JOIN + IS NULL pattern is the classic anti-join.

Check a column that cannot be NULL on its own — the subtle trap

The query checks o.order_id IS NULL. That works because order_id is the orders table's primary key — it is never NULL in a real order row, so a NULL there can only mean "no order matched". Check the wrong column and the query breaks:

-- TRAP: orders.amount can legitimately be NULL on a real order
WHERE o.amount IS NULL
-- This now returns never-ordered customers PLUS customers whose
-- order exists but has a NULL amount. Two different populations, merged.

Always test the NULL on the join key or the primary key of the right table — a column whose NULL is guaranteed to mean "row absent", never "value missing".

Three ways to write an anti-join
-- A. NOT EXISTS — usually the clearest, and NULL-safe
SELECT c.customer_id, c.name
FROM customers c
WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id);

-- B. NOT IN — concise, but DANGEROUS if the subquery can return NULL
SELECT customer_id, name
FROM customers
WHERE customer_id NOT IN (SELECT customer_id FROM orders);

-- C. LEFT JOIN ... IS NULL — the reference solution

The NOT IN form has a notorious failure: if any row of orders.customer_id is NULL, NOT IN returns no rows at all — because x NOT IN (1, 2, NULL) evaluates to NULL, never TRUE. NOT EXISTS and LEFT JOIN ... IS NULL are immune to this. Prefer NOT EXISTS as the default; reach for NOT IN only when the column is provably NULL-free.

Edge case: "never ordered" vs "no completed order"

The reference query treats any row in orders as "has ordered". But a customer whose only order was cancelled arguably still belongs in a win-back campaign. If the definition is "no completed order", the status filter must go in the ON clause, not WHERE — putting it in WHERE would demote the LEFT JOIN to an INNER JOIN and break the anti-join:

SELECT c.customer_id, c.name
FROM customers c
LEFT JOIN orders o
  ON c.customer_id = o.customer_id AND o.status = 'completed'
WHERE o.order_id IS NULL;
Senior signal: name the LEFT JOIN ... IS NULL anti-join pattern, insist on testing the NULL against the join key or PK (never a nullable attribute like amount), warn that NOT IN collapses to zero rows when the subquery contains a NULL, and put any status filter in ON so the outer join is preserved.
