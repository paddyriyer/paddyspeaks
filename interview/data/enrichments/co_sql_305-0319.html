The composite question that wraps everything Netflix Data Engineering cares about: binge detection that survives late-arriving dimensions. When Stranger Things S5 drops globally at midnight, millions of view events fire seconds later — before dim_series_metadata has finished propagating across regional shards. An INNER JOIN would silently drop those views and the launch-day binge dashboard would show "0 bingers". The defensive pattern: LEFT JOIN + most-recent-effective dim + COALESCE placeholder + per-day series aggregation. The fact is the truth, the dimension is just an attribute.

2. The SQL — defensive join + per-day series binge

WITH current_metadata AS (
  -- Step 1: dedupe SCD2 dim to most-recent row per series
  SELECT series_id, series_title, season_number
  FROM (
    SELECT *,
      ROW_NUMBER() OVER (PARTITION BY series_id ORDER BY effective_date DESC) AS rn
    FROM dim_series_metadata
  ) m
  WHERE rn = 1
),
daily_binges AS (
  -- Step 2: count episodes per (user, series, day); LEFT JOIN keeps orphan-fact users
  SELECT v.user_id, v.series_id, v.view_date,
    COUNT(v.episode_id) AS episodes_watched,
    COALESCE(m.series_title,
             'Series ID: ' || v.series_id || ' (Pending Metadata)') AS series_display_name
  FROM fact_viewing_history v
  LEFT JOIN current_metadata m USING (series_id)
  GROUP BY v.user_id, v.series_id, v.view_date, m.series_title
)
SELECT *
FROM daily_binges
WHERE episodes_watched >= 3
ORDER BY view_date DESC, episodes_watched DESC;

3. Strategic explanations

Why this is one query, not three

It composes three patterns that engineers usually meet separately:

  1. SCD2 latest-effective — ROW_NUMBER + WHERE rn = 1 picks the current dim row.
  2. LAD defensive join — LEFT JOIN + COALESCE placeholder so orphan facts aren't lost.
  3. Per-day binge aggregation — GROUP BY user, series, date + HAVING COUNT >= 3.
The crucial insight — the join key is the FK, not the title

The streak math doesn't depend on whether the metadata has arrived. series_id is the source-of-truth identifier; series_title is decoration. Even with zero dim rows, the binge-detection still works correctly because GROUP BY v.series_id groups on the FK. The COALESCE only affects the human-readable display column. Always join on IDs, not names.

The placeholder string is searchable on purpose
COALESCE(m.series_title, 'Series ID: ' || v.series_id || ' (Pending Metadata)')

Two design choices encoded:

  • The placeholder includes the actual series_id — the analyst can copy it into a metadata-investigate query.
  • The "(Pending Metadata)" suffix is grep-able — the BI tool's filter dropdown shows it as a real bucket and the platform team can alert on its prevalence.
Why GROUP BY ... m.series_title appears

In Postgres-strict GROUP BY mode, every non-aggregate column in SELECT must appear in GROUP BY. m.series_title appears in the COALESCE expression, so it must be grouped. If you forget this, you get "column must appear in GROUP BY". Some engines (MySQL non-strict, BigQuery with ANY_VALUE) tolerate it; portable code lists it explicitly.

The pre-aggregation performance trick

The query aggregates first, then filters. If you reversed it (filtered after window function over 100M raw rows), you'd sort the entire fact table. GROUP BY reduces the working set to ~100x fewer rows before the next step. The order is intentional and matters at scale.

Joining at fact-row grain vs aggregated grain

You could equivalently join AFTER aggregation:

WITH per_day AS (
  SELECT user_id, series_id, view_date, COUNT(*) AS eps
  FROM fact_viewing_history
  GROUP BY user_id, series_id, view_date
)
SELECT pd.*,
  COALESCE(m.series_title, 'Pending') AS title
FROM per_day pd
LEFT JOIN current_metadata m USING (series_id)
WHERE pd.eps >= 3;

This is even more performant for big fact tables — fewer rows enter the join. Either form is correct; pick the one that fits the surrounding pipeline conventions.

The "series streak across days" cousin

Per-day binge says "watched 3 episodes Tuesday". Series streak says "watched any episode of Stranger Things every day for 5 days". Both signals are valuable; combine them via gaps-and-islands on the (user_id, series_id, view_date) grain:

WITH series_active_days AS (
  SELECT user_id, series_id, view_date
  FROM fact_viewing_history
  GROUP BY user_id, series_id, view_date
),
series_islands AS (
  SELECT user_id, series_id, view_date,
    (view_date - DATE '1970-01-01') -
    ROW_NUMBER() OVER (PARTITION BY user_id, series_id ORDER BY view_date) AS island_id
  FROM series_active_days
)
SELECT user_id, series_id,
  MIN(view_date) AS streak_start,
  MAX(view_date) AS streak_end,
  COUNT(*)       AS days_in_streak
FROM series_islands
GROUP BY user_id, series_id, island_id
HAVING COUNT(*) >= 2
ORDER BY days_in_streak DESC;

The same-series-multiple-days streak is the "hooked-on-this-show" diagnostic. It's the input to the recommendation algorithm's "watch what's next" surface and the user-segment most likely to renew at price hikes.

Operational guard — alert on orphan rate
SELECT view_date,
  100.0 * SUM(CASE WHEN m.series_title IS NULL THEN 1 ELSE 0 END) / COUNT(*) AS pct_orphan_views
FROM fact_viewing_history v
LEFT JOIN current_metadata m USING (series_id)
GROUP BY view_date
ORDER BY view_date DESC;

SLO: pct_orphan_views < 0.5%. A spike above 1% pages the data-platform on-call — the dim-replication pipeline has stalled. This is exactly how Netflix instruments the LAD pattern in production.

Late-arriving facts — the dual problem

The opposite case (current dim, late fact) is the temporal-join problem. A fact arriving days late from offline-mode mobile must attribute against the dim version that was current at fact time, not now. SCD2 with effective_from / effective_to + a temporal join solves it. Same family of patterns; different direction.

Senior signal: articulate the fact-vs-dim asymmetry in one sentence ("the fact is the truth"), name the LAD pattern (LEFT JOIN + COALESCE placeholder + ROW_NUMBER for SCD2), aggregate before joining for performance, and extend the same shape to multi-day series streaks. The orphan-rate SLO and the late-arriving-fact temporal-join cousin are the over-the-bar finish.
