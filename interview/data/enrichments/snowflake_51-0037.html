Anthropic’s facilities team wants a maintenance cost leaderboard with two ranking dimensions simultaneously: which equipment in each location is the costliest (a partitioned rank), and which equipment across the entire fleet has been serviced most frequently (a global dense rank). The query combines aggregation and window functions on the same result set — a "grouped-then-ranked" pattern that many candidates stumble on because they try to apply the window function before the GROUP BY is resolved.

2. The SQL — LEFT JOIN + GROUP BY + dual window ranks

SELECT e.equipment_id, e.name, e.location,
       COUNT(m.maint_id)  AS maintenance_events,
       SUM(m.cost)        AS total_cost,
       MAX(m.maint_date)  AS last_maintenance,
       RANK() OVER (PARTITION BY e.location ORDER BY SUM(m.cost) DESC) AS cost_rank_in_location,
       DENSE_RANK() OVER (ORDER BY COUNT(m.maint_id) DESC)             AS frequency_rank_global
FROM equipment    e
LEFT JOIN maintenance m ON e.equipment_id = m.equipment_id
GROUP BY e.equipment_id, e.name, e.location
ORDER BY total_cost DESC;

4. Strategic explanations

Window functions applied to grouped output — why this works

The GROUP BY runs first, producing one row per equipment. The window functions (RANK, DENSE_RANK) then operate on this aggregated row set — not on the raw rows. This is why RANK() OVER (... ORDER BY SUM(m.cost) DESC) is legal: by the time the window function executes, SUM(m.cost) is a resolved column value in the group row, not a pending aggregate. Candidates who try to use a raw column like ORDER BY m.cost inside the window will get an error, because m.cost was aggregated away by the GROUP BY.

RANK vs DENSE_RANK — what the gap means

Within Plant North, RANK() OVER (PARTITION BY location ORDER BY SUM(cost) DESC): if two items tied at cost they would both get rank 1 and the next item would get rank 3 (a gap). DENSE_RANK() OVER (ORDER BY COUNT(maint_id) DESC) for frequency: E2 and E4 both have 1 event, both get frequency_rank=3; the next rank is 4 (no gap). The choice signals intent: RANK for "there are N-1 equipment that beat you" (gap shows how many ties at the top), DENSE_RANK for "there are N-1 distinct frequency levels above you." For leaderboard display, DENSE_RANK avoids confusing gaps.

LEFT JOIN to include unserviced equipment

E5 has no maintenance records. The LEFT JOIN keeps it in the result with NULL cost and zero events. COUNT(m.maint_id) correctly returns 0 for E5 (COUNT ignores NULLs); SUM(m.cost) returns NULL (SUM of NULLs). In Postgres a NULL sorts as larger than every non-NULL value — last in ASC, first in DESC — so under ORDER BY SUM(m.cost) DESC the unserviced E5 ranks first within Plant North, ahead of equipment that actually incurred cost. To rank it last instead (usually what a cost leaderboard wants), treat NULL as 0 with COALESCE(SUM(m.cost), 0) or append NULLS LAST.

NULLs in ORDER BY of a window function

By default in Postgres and Snowflake, NULLs sort last in ASC order and first in DESC order. Snowflake supports NULLS LAST / NULLS FIRST modifiers: ORDER BY SUM(m.cost) DESC NULLS LAST explicitly puts unserviced equipment at the bottom regardless of sort direction — the production-safe form.

Dialect notes

RANK() and DENSE_RANK() are ANSI SQL and supported in all major engines (Snowflake, Postgres, BigQuery, SQL Server, MySQL 8+). NULLS FIRST/NULLS LAST is supported in Snowflake and Postgres but not MySQL (which always puts NULLs first in ASC, last in DESC without override). BigQuery supports NULLS LAST in ORDER BY within window functions.

Senior signal: explain that window functions in this query operate on the already-grouped rows (not raw rows), articulate when to choose RANK vs DENSE_RANK (gap vs no-gap in ties), use COUNT(m.maint_id) not COUNT(*) so unserviced equipment returns 0, and add NULLS LAST to the window ORDER BY to correctly position equipment with no maintenance history.
