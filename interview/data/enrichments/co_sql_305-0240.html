Yahoo's logistics dashboard ranks suppliers by an "on-time reliability" score. The reference query looks plausible — it divides a SUM(CASE…) by COUNT(*) — but read the CASE condition closely. It tests on_time IS NOT NULL, not on_time = true. That single mismatch turns a reliability score into a "did we record an outcome at all" score, and an interviewer is watching whether you notice that the worst supplier ends up rated 100%.

Supplier 3's two shipments were both late — a true reliability score for them is 0%. Some rows carry a NULL on_time: shipments where the outcome was never logged. Those NULLs are exactly where the buggy and correct queries diverge.

2. The SQL — on-time rate per supplier

SELECT supplier_id,
  100.0 * SUM(CASE WHEN on_time THEN 1 ELSE 0 END) / COUNT(*) AS reliability
FROM shipments GROUP BY supplier_id ORDER BY reliability DESC;

The reference solution wrote CASE WHEN on_time IS NOT NULL THEN 1 ELSE 0 END — that counts shipments that have an outcome, late or not. Reliability has to count shipments that arrived on time: the CASE must test the boolean itself, CASE WHEN on_time THEN 1 ELSE 0 END.

4. Strategic explanations

The bug: IS NOT NULL measures the wrong thing

SUM(CASE WHEN on_time IS NOT NULL THEN 1 ELSE 0 END) counts every row where the column has any value — true and false both pass. So a supplier whose every shipment was late, but recorded, scores 100%. The fix is to make the CASE test the fact you actually care about: WHEN on_time (equivalently WHEN on_time = true). Late shipments and unlogged shipments then both fall to the ELSE 0 branch.

What should the denominator be? — the NULL decision

Dividing by COUNT(*) treats an unlogged shipment as a failure. That is defensible: "we have no proof it was on time." But a logistics team often wants reliability over graded shipments only, excluding the ones still in transit. That changes the denominator:

SELECT supplier_id,
  100.0 * COUNT(*) FILTER (WHERE on_time) / COUNT(on_time) AS reliability
FROM shipments GROUP BY supplier_id ORDER BY reliability DESC;

COUNT(on_time) counts non-NULL values only, so supplier 1 becomes 2/3 ≈ 66.7% and supplier 2 becomes 1/1 = 100%. Same numerator intent, different universe. Confirm with the interviewer: "does an unrecorded shipment count against the supplier, or is it excluded?"

Cleaner numerators — FILTER and AVG

The SUM(CASE…) idiom is portable, but Postgres has two tidier forms. COUNT(*) FILTER (WHERE on_time) reads as exactly what it does. And because on_time is already boolean, AVG((on_time)::int) gives the on-time fraction directly — multiply by 100 for a percentage. All three agree once the condition is correct; pick the one your team finds most readable.

Integer division — why the 100.0 matters

The literal is written 100.0, not 100, on purpose. If you wrote 100 * SUM(…) / COUNT(*) with all-integer operands, Postgres would do integer division: 100 * 2 / 4 evaluates left-to-right to 200 / 4 = 50 here, but 100 * 1 / 3 becomes 33, silently truncating. The 100.0 forces numeric arithmetic so fractional rates survive. Alternatively cast: SUM(…)::numeric / COUNT(*).

Edge cases: a supplier with zero shipments, and three-valued logic

A supplier present in some dimension table but with no rows in shipments simply won't appear in this GROUP BY — if the dashboard needs them shown as "no data", that requires a LEFT JOIN from the supplier list. Also note CASE WHEN on_time handles NULL gracefully: a NULL is neither true nor false, so it falls through to ELSE 0 — no need for an explicit IS NULL branch.

Senior signal: catch that on_time IS NOT NULL measures "outcome recorded" rather than "on time" — the tell is the all-late supplier scoring 100% — fix the CASE to test the boolean, decide deliberately whether unlogged shipments belong in the denominator, and keep the percentage in numeric arithmetic so rates don't truncate.
