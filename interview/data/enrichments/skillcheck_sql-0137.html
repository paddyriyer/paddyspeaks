A growth team building a year-end engagement report needs each user's most recent login within calendar 2025 — and users who never logged in during 2025 must drop off the report entirely. The interviewer is probing two things: whether you filter the year with a clean half-open range rather than a fragile EXTRACT(YEAR ...), and whether you understand that filtering before the aggregate naturally excludes the no-activity users.

1. The source data — logins

user_idtime_stamp(narrative)
12024-12-31 22:00:00just outside 2025 — must be excluded
12025-01-15 10:00:00inside 2025
12025-09-01 18:00:00inside 2025 — the latest for user 1
22025-06-06 06:06:00user 2's only login, inside 2025
32024-11-11 11:11:00user 3's only login — entirely in 2024

User 1's earliest row sits at 2024-12-31 22:00 — the boundary edge case, hours short of 2025. User 3 has no 2025 activity at all and must vanish from the output, not appear with a NULL.

2. The SQL — range-filter the year, then MAX per user

SELECT user_id, MAX(time_stamp) AS last_stamp
FROM logins
WHERE time_stamp >= TIMESTAMP '2025-01-01'
  AND time_stamp <  TIMESTAMP '2026-01-01'
GROUP BY user_id
ORDER BY user_id;

The WHERE keeps only rows inside the half-open interval [2025-01-01, 2026-01-01). GROUP BY user_id then buckets the survivors per user, and MAX(time_stamp) picks the latest within each bucket. A user with no surviving rows forms no group and produces no output line.

3. The resulting analysis

user_idlast_stampsignal
12025-09-01 18:00:00Sep beats Jan; the 2024-12-31 row was filtered out first
22025-06-06 06:06:00only login, trivially the max

User 3 is absent — correctly. Its single login is in 2024, so WHERE discards it before grouping; with no rows left, there is no group and no row. The report shows only users who were genuinely active in 2025.

4. Strategic explanations

Half-open range beats EXTRACT(YEAR ...)

time_stamp >= '2025-01-01' AND time_stamp < '2026-01-01' is sargable — the optimiser can use a B-tree index on time_stamp as a range scan. Wrapping the column in a function, EXTRACT(YEAR FROM time_stamp) = 2025, applies that function to every row and defeats a plain index. The range form is also correct at the edges: < '2026-01-01' excludes exactly midnight on New Year while including the last microsecond of 2025-12-31. Functionally equivalent on a 5-row demo, materially faster at scale.

Filter-then-aggregate is what drops the inactive users

The requirement "users with no 2025 logins must NOT appear" is satisfied for free by ordering: WHERE runs first and deletes all of user 3's rows, so GROUP BY never sees that user. Contrast a LEFT JOIN approach or a post-aggregation HAVING — those could surface user 3 with a NULL last_stamp that you would then have to filter again. Filtering individual rows before grouping is both simpler and exactly the semantics the spec wants.

Why this is MAX, not ROW_NUMBER

The task asks for a single scalar — the latest timestamp — not the whole latest row with its other columns. MAX(time_stamp) in a GROUP BY is the direct, cheap answer. If the spec needed the full row (say, the device used on the last login) you would switch to ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY time_stamp DESC) filtered to 1. Here, with only user_id and the timestamp requested, the aggregate is the right and lighter tool.

NULL timestamps and the strict comparisons

If a time_stamp were NULL, both >= and < comparisons evaluate to unknown, so the row fails WHERE and is dropped — a NULL-timestamp login simply never counts as "in 2025", which is the sensible default. And MAX itself ignores NULLs. The fixture has no NULLs, but stating that the range filter quietly handles them shows you have considered the dirty-data case.

Dialect — the year-filtering idiom elsewhere

The TIMESTAMP '2025-01-01' literal and half-open range are portable to every major engine. SQL Server would write '2025-01-01' as a plain string literal that implicitly casts; MySQL accepts the same. The anti-pattern YEAR(time_stamp) = 2025 exists in MySQL and SQL Server too — and is non-sargable everywhere. Prefer the range; it is the one form that is both correct and index-friendly across all dialects.

Senior signal: bound the year with a sargable half-open range [2025-01-01, 2026-01-01) instead of EXTRACT(YEAR ...), let WHERE-before-GROUP BY drop inactive users naturally rather than filtering a NULL afterwards, pick MAX over ROW_NUMBER when only a scalar is needed, and note that the range filter also disposes of NULL timestamps cleanly.
