A flooring installation company needs a job costing report: "For every job, show the customer name, the material used, the square footage, and the total cost — sorted by most expensive jobs first." This is a three-table JOIN exercise combining a transaction table (jobs) with two dimension tables (customers, materials). The interviewer is probing whether you can chain multiple JOINs correctly, derive a computed column from two table attributes, and handle the NULLIF(regexp_replace(...)::numeric) cast when the source columns may be stored as text.

2. The SQL — three-table JOIN with derived cost column

SELECT j.job_id, c.customer_name, m.material_name, j.sqft,
       NULLIF(regexp_replace(j.sqft::text,           '[^-.0-9]', '', 'g'), '')::numeric
       * NULLIF(regexp_replace(m.cost_per_sqft::text, '[^-.0-9]', '', 'g'), '')::numeric AS total_cost
FROM jobs j
JOIN customers c ON j.customer_id = c.customer_id
JOIN materials m ON j.material_id = m.material_id
ORDER BY total_cost DESC;

4. Strategic explanations

Chaining multiple INNER JOINs

Two INNER JOINs are applied in sequence: jobs → customers on customer_id, then the result → materials on material_id. The join order in the SQL text is a hint to the optimiser but the engine is free to reorder them. If any job's customer_id or material_id has no match in the respective dimension table, that job row is silently dropped from the result. In a production system, referential integrity constraints (foreign keys) prevent orphaned job rows at write time; in a data warehouse where FK enforcement is often disabled (Snowflake, BigQuery), the query can silently lose rows. A LEFT JOIN version with WHERE c.customer_id IS NULL OR m.material_id IS NULL surfaces orphans for data-quality investigation.

Derived column multiplication — NULLs propagate

The expression sqft_clean * cost_per_sqft_clean produces NULL if either operand is NULL. This arises when the regex cast returns NULL because the value was non-numeric. NULL propagation in arithmetic is correct behaviour — a cost computed from a missing price is undefined. The output row appears but with total_cost = NULL. In ORDER BY total_cost DESC, NULLs sort last in Postgres (NULLS LAST is the default for DESC); in Snowflake and BigQuery NULLs sort first by default for DESC — use ORDER BY total_cost DESC NULLS LAST for portable, explicit behaviour.

When to aggregate vs not

This query has no GROUP BY — it returns one row per job. If the requirement were "total revenue per customer," a GROUP BY c.customer_name with SUM(total_cost) would be needed. Recognising when aggregation is and is not required is a basic but frequently tested competency: the absence of GROUP BY is deliberate here because "list every job" is the requirement.

Defensive cast — NULLIF(regexp_replace(...)::numeric)

Both sqft and cost_per_sqft may be stored as text if imported from CSV or a legacy system. The regex strips non-numeric characters; NULLIF handles the empty-string case; ::numeric does the final parse. In Snowflake the equivalent is TRY_TO_NUMBER(j.sqft) * TRY_TO_NUMBER(m.cost_per_sqft). BigQuery uses SAFE_CAST(j.sqft AS FLOAT64).

ORDER BY NULL behaviour by dialect
  • Postgres: DESC sorts NULLs last by default.
  • Snowflake / BigQuery: DESC sorts NULLs first by default.
  • MySQL: NULLs sort first regardless of direction.
  • SQL Server: NULLs sort first with ASC, last with DESC.
  • Use ORDER BY total_cost DESC NULLS LAST for portable, explicit behaviour (supported by Postgres, Snowflake, and BigQuery).
Senior signal: explain that chained INNER JOINs silently drop rows with missing dimension matches (flag for data-quality checking), note that NULL propagates through the multiplication (a missing sqft or price yields NULL total_cost), and cite the cross-dialect difference in NULL ordering for DESC sorts — NULLS LAST makes it explicit and portable.
