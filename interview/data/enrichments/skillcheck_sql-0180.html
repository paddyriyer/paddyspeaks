Identifying employees who are globally well-paid but underperform within their own elite department surfaces a paradox common in high-compensation fields: someone earning the 60th percentile company-wide can still be in the bottom third of their team if that team is very well-paid. The interviewer is probing whether you can apply two CUME_DIST window functions with different partition scopes to the same row, and whether you understand the semantics of each — particularly that CUME_DIST returns the fraction of rows at or below the current row, which differs from PERCENT_RANK.

Dept A is elite: even its two lowest earners (emp 1 at 180, emp 2 at 200) are far above all of dept B and C. The question is whether any dept A employee who earns more than the bottom half of the company is nonetheless in the bottom 30% of dept A.

2. The SQL — dual CUME_DIST with different scopes

WITH r AS (
  SELECT emp_id, dept, salary,
         CUME_DIST() OVER (ORDER BY salary) AS company_pct,
         CUME_DIST() OVER (PARTITION BY dept ORDER BY salary) AS dept_pct
  FROM employee
)
SELECT emp_id, dept, salary,
       ROUND(company_pct::numeric, 2) AS company_pct,
       ROUND(dept_pct::numeric, 2) AS dept_pct
FROM r
WHERE company_pct > 0.50 AND dept_pct <= 0.30
ORDER BY emp_id;

The first CUME_DIST has no PARTITION BY, so it ranks across all 20 employees. The second is partitioned by dept, ranking within each department's own pay scale. The WHERE clause keeps only employees who clear the global median (company_pct > 0.50) but fall in the bottom 30% of their department (dept_pct ≤ 0.30).

4. Strategic explanations

CUME_DIST vs PERCENT_RANK

CUME_DIST for a row = (number of rows with value ≤ current row) / (total rows in partition). It is always in (0, 1] and the highest-valued row always gets 1.0. PERCENT_RANK = (rank − 1) / (total rows − 1); the lowest row gets 0.0 and the highest gets 1.0. For a "top half" threshold, CUME_DIST > 0.50 means more than half the partition earns ≤ this salary — i.e. this employee earns more than the median. Using PERCENT_RANK would give a slightly different boundary and would exclude the median exactly; the choice should match the business definition.

Boundary behaviour: ≤ 0.30 vs < 0.30

The filter uses dept_pct ≤ 0.30, so emp 3 (dept_pct exactly 0.30) is included. With < 0.30, emp 3 would be excluded. In a 10-person department, exactly the 3rd employee by salary lands at dept_pct = 0.30, so the operator choice matters. The question specifies "bottom 30%" which is conventionally inclusive, matching ≤ 0.30. Always confirm the boundary convention with the interviewer.

Why ROUND appears only in the outer SELECT, not in the WHERE clause

Rounding in the WHERE predicate would compare rounded percentages, which can introduce boundary errors (a true 0.305 rounds to 0.31 and is excluded; a true 0.304 rounds to 0.30 and is included). The WHERE clause compares the full-precision CUME_DIST value; ROUND is deferred to the display columns. The CTE materialises the unrounded values, so the outer query accesses them correctly.

Performance: two window functions on the same table

Most optimisers execute each distinct window specification (different PARTITION BY or ORDER BY) as a separate sort-and-scan pass. Here there are two specifications (no partition and partition-by-dept), so expect two sort operations. On very large employee tables, pre-aggregating company-wide salary percentiles in a separate CTE and joining can reduce redundant work, though the query is typically fast enough for the sizes where this pattern is used.

Senior signal: apply two CUME_DIST calls with different PARTITION BY scopes in one CTE; explain the difference from PERCENT_RANK (CUME_DIST never reaches 0, always reaches 1); keep ROUND out of the filter; and confirm with the interviewer whether the 30% boundary is inclusive or exclusive.
