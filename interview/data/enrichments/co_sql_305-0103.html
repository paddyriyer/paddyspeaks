Amazon's fulfillment analytics team wants a single number for basket size: on average, how many items are in an order? The phrasing "order items average" is deceptively loose — it could mean the average line, or the average order. The reference solution picks one interpretation, and the interview is about recognising the ambiguity and the two-level aggregation it forces.

1. The source data — order_items

Order 1 has two line items totalling 5 units; order 3 has three line items of 1 unit each, also totalling 3; orders 2 and 4 are single-line. The mix — some orders many lines, some one — is exactly what makes "average items per order" different from "average qty per line".

2. The SQL — aggregate of an aggregate

SELECT AVG(item_count) FROM (
  SELECT order_id, SUM(qty) AS item_count FROM order_items GROUP BY order_id) t;

The inner query collapses line items to one row per order, summing qty into a per-order item total. The outer query averages those order totals.

3. The resulting analysis

avg
5.7500000000000000

Per-order totals are 5, 5, 3, 10 — summing to 23, divided by 4 orders = 5.75 items per order. The output column is unnamed (Postgres defaults it to avg); add AS avg_items_per_order for a readable result. The high-scale numeric tail is normal for AVG. Crucially, the denominator is 4 — the number of orders — not 7, the number of line-item rows.

4. Strategic explanations

The two interpretations — this is the whole question

"Order items average" has two defensible readings, and they give different numbers:

-- A. average items per ORDER (the reference solution): 23 / 4 = 5.75
SELECT AVG(item_count) FROM (
  SELECT order_id, SUM(qty) AS item_count
  FROM order_items GROUP BY order_id) t;

-- B. average qty per LINE ITEM: (2+3+5+1+1+1+10) / 7 = 3.28...
SELECT AVG(qty) FROM order_items;

Reading A weights every order equally; reading B weights every line equally. They are not interchangeable. The senior move is to state both and ask which the business wants — for "basket size", reading A is almost always right.

Why the nested query is necessary

You cannot get reading A in one flat SELECT. AVG(SUM(qty)) is illegal — you cannot nest aggregates directly — and AVG(qty) silently gives reading B. The pattern "GROUP BY to one row per order, then aggregate that" requires a subquery (or a CTE). Equivalent and arguably clearer with a CTE:

WITH per_order AS (
  SELECT order_id, SUM(qty) AS item_count
  FROM order_items GROUP BY order_id)
SELECT AVG(item_count) AS avg_items_per_order FROM per_order;

There is also a shortcut for this specific case: SUM(qty) / COUNT(DISTINCT order_id) over the flat table gives the same 5.75, because total units over total orders is the average per order. Knowing why the shortcut works (and that it is integer division unless you cast) is a nice detail.

Lines per order vs units per order

Even within reading A there is a sub-choice. SUM(qty) averages units (an order of "3 of one SKU" counts as 3). If "items" means distinct products, the inner aggregate is COUNT(*) — number of line rows — not SUM(qty). For order 3 that is 3 lines either way, but for order 1 it is 2 lines vs 5 units. Confirm whether "items" means physical units or line entries.

Empty table and integer division

AVG over no orders returns NULL, not 0 — COALESCE if the dashboard needs a number. And if qty is an INTEGER column, the inner SUM stays integer, but AVG of integers in Postgres already returns numeric, so no truncation here — though in engines that keep integer division you would cast with AVG(item_count::numeric).

Orders with no line items

This query only sees orders that appear in order_items. An order placed but with zero line rows (a glitch, or a fully-cancelled order) is absent, so it never enters the denominator. If such orders exist and should count as 0 items, start from the orders table with a LEFT JOIN and COALESCE(SUM(qty), 0).

Senior signal: surface that "order items average" has two readings — per-order vs per-line — and name which the reference query computes (per-order, denominator = order count), explain that the nested aggregate is required because AVG(SUM(...)) is illegal, and clarify whether "items" means units or distinct lines before committing.
