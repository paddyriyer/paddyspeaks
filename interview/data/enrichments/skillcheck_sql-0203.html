Fetching the latest row per group is one of the most common SQL patterns in production systems — current price, most recent status, last known location. PostgreSQL's DISTINCT ON handles it elegantly in a single scan. This question sharpens the challenge with a tie-breaking rule: when two rows share the same timestamp, prefer the alphabetically last source — testing whether you understand the coupling between the DISTINCT ON columns, the ORDER BY key, and the tie-breaker.

1. The source data — price_change

12025-02-01 10:008.75manual
22025-01-15 09:0020.00feed

Product 1 has two rows tied at 2025-02-01 10:00: sources "feed" and "manual". Alphabetically, "manual" > "feed" (m > f), so "manual" wins the tie. Product 2 has a single row — no tie-breaking needed.

2. The SQL — DISTINCT ON with compound ORDER BY

SELECT DISTINCT ON (product_id)
  product_id, price::float8 AS price, source
FROM price_change
ORDER BY product_id, changed_at DESC, source DESC;

DISTINCT ON (product_id) keeps exactly one row per product_id — whichever row appears first after ORDER BY. The ORDER BY sorts by timestamp descending first (latest timestamp wins), then by source descending (alphabetically last source wins ties). The product_id must appear first in the ORDER BY to satisfy PostgreSQL's requirement that DISTINCT ON columns lead the sort.

The ::float8 cast converts the NUMERIC price to a double-precision float as the solution specifies.

4. Strategic explanations

DISTINCT ON — how it works

SELECT DISTINCT ON (col1, col2, …) is a PostgreSQL extension (not standard SQL) that keeps the first row for each distinct combination of the listed columns, where "first" is defined by the ORDER BY. PostgreSQL sorts the full result set by the ORDER BY key, then scans and emits only the first row per DISTINCT ON group. This is a single pass over an ordered result — no subquery, no window function, no self-join. Its cousins in standard SQL are ROW_NUMBER() OVER (PARTITION BY … ORDER BY …) = 1 and FIRST_VALUE.

The ORDER BY constraint: DISTINCT ON columns must lead

PostgreSQL requires that the DISTINCT ON expressions appear as the leading columns in ORDER BY. ORDER BY product_id, changed_at DESC, source DESC satisfies this because product_id (the DISTINCT ON column) comes first. Omitting it or reordering it causes a syntax error. This requirement exists because PostgreSQL's implementation sorts by the ORDER BY key and then picks the first row per group — without leading on the group key the "first" row would not be consistent.

Tie-breaking with DESC on string

Alphabetical DESC ordering puts "z" before "a". "manual" starts with 'm' and "feed" starts with 'f'; since m > f, "manual" precedes "feed" in DESC order. The rule "alphabetically last" maps directly to source DESC. A common mistake is writing source ASC (which would pick "feed") or omitting the tie-breaker entirely (which would give non-deterministic results when the optimiser sorts ties arbitrarily).

Standard SQL alternative: ROW_NUMBER()
SELECT product_id, price, source
FROM (
  SELECT product_id, price, source,
         ROW_NUMBER() OVER (PARTITION BY product_id
                            ORDER BY changed_at DESC, source DESC) AS rn
  FROM price_change
) t
WHERE rn = 1
ORDER BY product_id;

This is portable to every engine that supports window functions (PostgreSQL, SQL Server, BigQuery, Snowflake, MySQL 8+, SQLite 3.25+). It is logically identical to DISTINCT ON and typically compiled to the same execution plan in PostgreSQL. The tradeoff: more verbose, but universally portable and explicit about the tie-break ordering.

Performance notes

Both DISTINCT ON and ROW_NUMBER() = 1 benefit from an index on (product_id, changed_at DESC, source DESC). PostgreSQL can satisfy the query with an index scan, materialising no more than one row per product. On very large tables (millions of price changes, thousands of products) an index on this composite key is essential; without it, the planner falls back to a full sort, which is O(n log n).

Senior signal: DISTINCT ON (product_id) keeps the first row per product as defined by the trailing ORDER BY; the DISTINCT ON column must lead the ORDER BY; tie-break with source DESC for "alphabetically last" (DESC = largest first); and name the portable ROW_NUMBER() OVER (PARTITION BY … ORDER BY …) = 1 equivalent for non-PostgreSQL engines.
