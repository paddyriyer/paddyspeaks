Shopify’s trust-and-safety team wants to detect interactions where a user interacted with themselves — a common data-quality audit in social and marketplace systems, where self-loops indicate either a bug in the interaction pipeline, a test account, or in some cases a deliberate fraud pattern (self-reviews, self-referrals). The query is intentionally minimal: a single equality predicate on two columns of the same row, with no joins and no aggregation. The interviewer is testing whether you can write a clean predicate without over-engineering it.

I003, I004, and I006 are self-interactions (source = target). I001, I002, and I005 are normal cross-user interactions.

2. The SQL — same-row equality filter

SELECT *
FROM interactions
WHERE source_user_id = target_user_id;

Three self-interaction rows returned. The normal cross-user interactions (I001, I002, I005) are correctly excluded because source_user_id <> target_user_id for those rows.

4. Strategic explanations

The predicate: comparing two columns in the same row

WHERE source_user_id = target_user_id is an intra-row predicate — it compares two columns within the same tuple. No join is needed. This is conceptually different from a self-join (which would create pairs of rows from the same table). Intra-row predicates are evaluated once per row during the filter phase and are highly efficient: the engine can short-circuit any row where the two columns differ without further processing.

SELECT * in this context

SELECT * is appropriate for a diagnostic audit query where the goal is to surface the full record for manual review. In production reporting pipelines, prefer explicit column lists to avoid breaking when new columns are added, to control data exposure, and to improve query plan readability. For a trust-and-safety audit tool, however, SELECT * lets analysts see the complete row and route to remediation without needing to modify the query.

NULL behaviour: what if source or target is NULL?

If source_user_id IS NULL or target_user_id IS NULL, the equality NULL = NULL evaluates to UNKNOWN in SQL — not TRUE. Such rows are excluded by the WHERE filter. This is almost always correct (a NULL user ID indicates missing data, not a self-interaction), but worth stating explicitly in interview contexts where data quality is discussed. To also surface NULL-vs-NULL rows (if that is meaningful), add OR (source_user_id IS NULL AND target_user_id IS NULL).

Inverting the filter: excluding self-interactions from a feed

The mirror-image query — WHERE source_user_id <> target_user_id — is equally important: it is the standard guard added to social-graph queries to prevent self-loops from polluting recommendation models or friend-of-friend traversals. Knowing both sides of the predicate shows you understand the operational context, not just the syntax.

Indexing and performance

An index on (source_user_id) or (target_user_id) alone does not help this query: the predicate is not a range or equality filter on a single column but a cross-column comparison. A full table scan is typically required. For large interactions tables, a computed/generated column is_self_interaction BOOLEAN DEFAULT (source_user_id = target_user_id) with an index on it, or a partial index WHERE source_user_id = target_user_id (supported in Postgres), makes this audit query O(log n + k) where k is the number of self-interactions.

Downstream action: DELETE vs flag

In practice, detected self-interactions are typically flagged (update a flagged column or insert into a review queue) rather than immediately deleted. The detection query becomes the subquery of an UPDATE or INSERT INTO review_queue ... SELECT. Framing the detection as the first step in a remediation pipeline shows senior operational thinking.

Dialect notes

The predicate WHERE col_a = col_b is universal ANSI SQL. Postgres supports partial indexes (CREATE INDEX ON interactions(interaction_id) WHERE source_user_id = target_user_id) for fast audit queries. Snowflake does not support partial indexes but supports search optimization on equality predicates. MySQL and SQL Server also require a full scan without a generated column index.

Senior signal: recognize that this is an intra-row predicate (no join needed), explain NULL = NULL → UNKNOWN so NULL-ID rows are silently excluded, offer the inverted predicate (<>) as the equally important guard for downstream pipelines, and mention the partial index (Postgres) or generated-column index pattern for making audit scans fast at scale.
