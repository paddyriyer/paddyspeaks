Booking.com runs a learning catalogue, and the platform team needs a gap report: which courses currently have no instructor assigned, so they cannot be offered? It is an anti-join — "rows in A with no match in B" — and the interviewer wants the LEFT JOIN … IS NULL idiom plus an understanding of why NOT IN is the risky choice.

1. The source data — courses and instructors

courses.course_id instructors.course_id
1 1
2 3
3 5
4
5

Five courses; the instructors table has assignments for courses 1, 3, and 5. So courses 2 and 4 have no instructor — those are the gaps the report must surface.

2. The SQL — LEFT JOIN anti-join

SELECT c.course_id
FROM courses c
LEFT JOIN instructors i ON c.course_id = i.course_id
WHERE i.course_id IS NULL;

The LEFT JOIN keeps every course; for a course with no instructor, the i.* columns come back NULL. The WHERE i.course_id IS NULL then keeps only those unmatched rows — the anti-join pattern.

3. The resulting analysis

course_id
2
4

Exactly courses 2 and 4 — the two with no row on the instructors side. Every other course matched at least one instructor, so its i.course_id was non-NULL and the WHERE filtered it out. The result is unordered; add ORDER BY c.course_id for a stable report.

4. Strategic explanations

The mechanics — filter after the join, on the join key

The IS NULL test must target a column that is non-nullable in the real instructors table — the join key i.course_id is ideal. The only way that column is NULL in the output is the LEFT JOIN manufacturing an unmatched row. If you tested a column that can legitimately be NULL in the data (say i.notes), you would also catch matched courses whose instructor row happens to have a NULL note — a wrong answer. Always probe the join key.

NOT EXISTS — the equivalent the interviewer often prefers

The same gap report, expressed as a correlated subquery:

SELECT c.course_id
FROM courses c
WHERE NOT EXISTS (
  SELECT 1 FROM instructors i WHERE i.course_id = c.course_id
);

Many engineers find this reads more directly — "courses where no instructor exists" — and Postgres plans NOT EXISTS as an anti-join, identical performance to the LEFT JOIN form. Both are correct; knowing they are equivalent is the senior signal.

The NOT IN NULL trap — the headline danger

The tempting third form is broken in a way that is easy to miss:

SELECT course_id FROM courses
WHERE course_id NOT IN (SELECT course_id FROM instructors);

If the instructors table contains even one row with a NULL course_id, this returns zero rows — not an error, just a silently empty report. course_id NOT IN (1, 3, 5, NULL) is course_id <> NULL AND-ed in, which is unknown, so no row ever qualifies. The schema here lets instructors.course_id be NULL (no NOT NULL constraint), so this is a live hazard. LEFT JOIN … IS NULL and NOT EXISTS are both NULL-safe; NOT IN is not. This is the single most important point of the question.

Duplicate instructors — why this anti-join is robust

If a course has two instructors (co-teaching), it has two rows in instructors. The LEFT JOIN would produce two output rows for it — but both have a non-NULL i.course_id, so both are filtered out, and the course correctly does not appear in the gap report. The anti-join asks only "any match at all?", so duplicate matches do not distort it. (If you instead wanted the count of instructors per course, that is a GROUP BY, a different query.)

Edge cases: empty instructors and NULL course ids

If instructors is empty, every course is unmatched and the report lists all five — correct, every course is a gap. And if courses itself had a NULL course_id row, the LEFT JOIN never matches it (NULL equals nothing), so it appears in the gap report — arguably right, but a NULL course id is itself a data-quality issue worth flagging upstream.

Senior signal: express the gap report as LEFT JOIN … WHERE join_key IS NULL or NOT EXISTS, test the join key for NULL (not an arbitrary nullable column), and explicitly warn that NOT IN collapses to zero rows if the inner table has any NULL — a real risk given the unconstrained instructors.course_id.
