Infosys is building an engagement dashboard for a client's portal and the first tile is the simplest one: daily active users — how many distinct people logged in each day. The login table stores a full timestamp per event, and a single user logs in many times a day. Two things must go right: collapse timestamps to calendar days, and count people, not events.

1. The source data — logins

22024-06-03 07:00

User 1 logs in twice on 2024-06-01 (08:00 and 22:00). That repeat is the whole interview: a naive count would tally it twice and overstate the active-user number.

2. The SQL — COUNT(DISTINCT user_id) per day

SELECT DATE(login_at) AS day, COUNT(DISTINCT user_id) AS daily_logins
FROM logins GROUP BY DATE(login_at);

DATE(login_at) truncates each timestamp to its calendar date, so all of a day's events collapse into one group. COUNT(DISTINCT user_id) then counts unique people inside that group.

3. The resulting analysis

2024-06-01 has three login events but only 2 distinct users — user 1's two logins count once. That is the metric working correctly: this is daily active users, not daily login volume.

4. Strategic explanations

DISTINCT is the whole point — users vs events

Drop the DISTINCT and you get a different metric. COUNT(*) per day counts login events (3 on 2024-06-01); COUNT(DISTINCT user_id) counts unique people (2). The column name daily_logins is honestly a bit misleading — the query computes daily active users. Be precise about which one the dashboard tile actually wants; for a "DAU" tile it is the distinct count, for "login volume" it is the row count.

Why truncating the timestamp is mandatory

Grouping by the raw login_at would put every event in its own group — 08:00 and 22:00 are different timestamps even on the same date — so you would get one row per login, not one per day. DATE() (or the more explicit date_trunc('day', login_at)) is what bins events into calendar days. date_trunc returns a timestamp at midnight rather than a bare date, which matters if a downstream join expects a timestamp type.

The timezone trap

"Which day" depends on whose day. If login_at is stored in UTC, user 1's 22:00 UTC login might actually be the next calendar day in their local timezone — and DAU would be attributed to the wrong day. For a global portal, convert before truncating:

SELECT DATE(login_at AT TIME ZONE 'America/New_York') AS day,
       COUNT(DISTINCT user_id) AS dau
FROM logins
GROUP BY 1;

Naming the timezone assumption — "are these UTC, and whose midnight defines the day?" — is a senior reflex.

Missing days — gaps in the result

This query only emits rows for days that had a login. A day with zero activity simply will not appear — the dashboard would skip it rather than show a 0, which breaks a time series chart. To get a continuous axis, generate the date spine and LEFT JOIN:

SELECT d::date AS day, COUNT(DISTINCT l.user_id) AS dau
FROM generate_series(DATE '2024-06-01', DATE '2024-06-03', INTERVAL '1 day') d
LEFT JOIN logins l ON DATE(l.login_at) = d::date
GROUP BY d
ORDER BY d;
Ordering and NULL user_ids

The solution has no ORDER BY, so day order is not guaranteed — add ORDER BY day for a chart-ready result. And if user_id can be NULL (anonymous or logged-out hits), COUNT(DISTINCT user_id) ignores NULLs entirely — sometimes desirable, sometimes a silent undercount; decide deliberately.

Senior signal: use COUNT(DISTINCT user_id) so repeat logins do not inflate the count, explain that truncating the timestamp is what bins events into days, raise the timezone question for a global product, and note that zero-activity days vanish from the result unless you join against a generated date spine.
