A Community member shares a tournament-results table where each row is a played match: two teams and the winner. The ask is to "digitize" it into a standings table — matches played, won, tied, lost — per team. The catch the interviewer wants you to find: a team appears in two columns, and a tied match has no winner at all.

1. The source data — match_result

Team_1Team_2Result_won
IndiaAustraliaIndia
IndiaEnglandEngland
AustraliaEnglandNULL
IndiaAustraliaAustralia
EnglandIndiaIndia

Five matches. India plays in both Team_1 and Team_2 slots. The Australia–England match has a NULL Result_won — that is a tie, and it must be counted as a tie for both teams, not silently dropped.

2. The SQL — unpivot then aggregate

WITH t1 AS (
  SELECT team_1 AS team, result_won AS result FROM match_result
  UNION ALL
  SELECT team_2 AS team, result_won AS result FROM match_result
)
SELECT team,
       COUNT(1) AS match_count,
       SUM(CASE WHEN team = result THEN 1 ELSE 0 END) AS match_won,
       SUM(CASE WHEN result IS NULL THEN 1 ELSE 0 END) AS tied_match,
       SUM(CASE WHEN team <> result THEN 1 ELSE 0 END) AS lost_match
FROM t1
GROUP BY team;

The CTE stacks the two team columns into one team column — each match becomes two rows, one per participant, each carrying the same result. Then a single GROUP BY team with conditional sums splits the outcomes.

3. The resulting analysis

teammatch_countmatch_wontied_matchlost_match
Australia3111
England3111
India4202

India played 4 (no ties), won 2, lost 2 — and 2 + 0 + 2 reconciles to 4. Australia and England each played 3, and crucially each shows 1 tie: the single NULL-result match was counted for both sides. Won + tied + lost equals match_count for every team — the integrity check that proves the three buckets partition the matches with no double-count and no gap.

4. Strategic explanations

Why UNION ALL, never UNION

The CTE must use UNION ALL. Plain UNION de-duplicates, and that would silently collapse two genuinely distinct matches that happen to produce the same (team, result) pair — e.g. India winning two separate matches would become one row, and India's count would read 1 instead of 2. UNION ALL keeps every row; here it is correctness, not just speed.

The NULL trap — team <> result never matches a tie

This is the heart of the question. For the tied match, result is NULL. team = NULL is unknown, and team <> NULL is also unknown — neither the won bucket nor the lost bucket fires. So the tie falls only into the result IS NULL bucket, which is exactly right. But the logic is fragile: if you wrote lost_match as COUNT(*) - match_won - tied_match you would get the same answer, whereas team <> result only works because NULL comparisons are unknown. Know which one you are relying on.

The classic bug: forgetting ties don't satisfy <>

A common wrong instinct is to define a loss as "this team did not win": SUM(CASE WHEN team <> result THEN 1 ELSE 0 END) looks like "not a win" — but a tie is also "not a win", and <> against NULL is unknown so the tie is excluded here by luck. If a junior instead writes CASE WHEN team != result OR result IS NULL THEN 1 END for losses, ties get miscounted as losses. The three outcomes are mutually exclusive states; model them explicitly, do not infer one by negating another.

A team that only ever ties

Suppose a team's matches are all NULL-result. Its row would read match_count = N, won = 0, tied = N, lost = 0 — correct, and the reconciliation still holds. The conditional-sum pattern degrades gracefully to single-bucket teams; a JOIN-based approach that assumes at least one win would not.

Generalising the unpivot

UNION ALL of one SELECT per column is the portable unpivot — it works in every engine. SQL Server and Oracle also have a dedicated UNPIVOT operator, and Postgres can use LATERAL (VALUES ...) or unnest. With only two team columns the UNION ALL is clearest; with twenty columns the operator-based forms scale better.

Senior signal: insist on UNION ALL so distinct matches are never merged, recognise that a NULL Result_won is a tie that must land in its own bucket for both teams, model won/tied/lost as three explicit mutually exclusive cases, and sanity-check that they sum back to match_count.
