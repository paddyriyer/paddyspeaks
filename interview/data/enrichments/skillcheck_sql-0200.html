Finding "fully complete" projects is a universal completeness check in task-management and workflow systems. The twist here is two-part: (1) the boolean aggregate BOOL_AND must be used instead of a more common COUNT(done = false) = 0 approach, and (2) projects with zero tasks must be included as trivially complete — which requires an outer join from the project table and careful NULL handling so their NULL-joined done column does not accidentally disqualify them.

2. The SQL — LEFT JOIN + BOOL_AND with COALESCE

SELECT p.project_id
FROM project p
LEFT JOIN task t ON t.project_id = p.project_id
GROUP BY p.project_id
HAVING BOOL_AND(COALESCE(t.done, true))
ORDER BY p.project_id;

The LEFT JOIN keeps all projects, including project 4 with no tasks (it joins with NULL in every task column). COALESCE(t.done, true) converts the NULL done value to true, so for project 4 the BOOL_AND aggregates over a single synthetic true — correctly classifying it as complete. For projects with real tasks, COALESCE is a no-op.

4. Strategic explanations

BOOL_AND — the boolean aggregate

BOOL_AND(expr) returns true if every value in the group is true, false if any value is false, and NULL if the group is empty. It is the boolean analogue of MIN over a boolean column (and in fact PostgreSQL implements it the same way). The equivalent without BOOL_AND is HAVING SUM(CASE WHEN done = false THEN 1 ELSE 0 END) = 0 or HAVING COUNT(*) FILTER (WHERE NOT COALESCE(done, true)) = 0 — more verbose and easier to get wrong.

COALESCE to handle the NULL-task project

Without COALESCE(t.done, true), project 4's single NULL value causes BOOL_AND(NULL) to return NULL, and a HAVING clause filters out NULL — so project 4 would be silently excluded. The COALESCE argument is: a project with no tasks satisfies "all tasks done" vacuously, so the missing done value should be treated as true. This is the vacuous-truth convention from logic, and recognising it is the key to the correct answer.

Why LEFT JOIN from project, not from task

An INNER JOIN (or joining from task rather than project) would drop project 4 entirely before aggregation because it has no matching task rows. You must start from the authoritative list of projects (the project table) and LEFT JOIN tasks so that zero-task projects survive into the GROUP BY. This "left table is the master" pattern applies whenever you need to preserve entities that lack related rows.

Alternative approaches and their tradeoffs

Three common alternatives: (1) NOT EXISTS (SELECT 1 FROM task WHERE project_id = p.project_id AND NOT done) — readable but does not include zero-task projects without an additional OR NOT EXISTS (SELECT 1 FROM task WHERE …) clause; (2) two separate queries unioned (all-done projects UNION zero-task projects) — correct but verbose; (3) HAVING MIN(done::int) = 1 in PostgreSQL (casting bool to int: true=1, false=0) — works but less idiomatic than BOOL_AND. The LEFT JOIN + BOOL_AND + COALESCE form handles both cases in a single clean query.

Dialect notes

BOOL_AND is a PostgreSQL and DuckDB extension. In SQL Server and MySQL use HAVING MIN(CAST(done AS INT)) = 1. In BigQuery use LOGICAL_AND(done) with a similar COALESCE wrapper. Snowflake has BOOLOR_AGG (OR) but no native BOOL_AND — use HAVING MIN(IFF(done, 1, 0)) = 1.

Senior signal: drive the query from the project table with a LEFT JOIN so zero-task projects survive; use COALESCE(t.done, true) inside BOOL_AND to encode the vacuous-truth convention that an empty task set is trivially complete; and be ready to translate to LOGICAL_AND (BigQuery) or MIN(CAST(done AS INT)) = 1 (SQL Server/MySQL) for cross-dialect portability.
