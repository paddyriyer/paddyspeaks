Tax-software companies (TurboTax, H&R Block) live and die by filing-day cadence. The compliance team's deceptively simple ask: which users filed at least 3 days in a row? Underneath sits the same gaps-and-islands engine that powers every retention dashboard. The senior conversation isn't the count — it's classifying each user's exit from their most recent streak.

2. The SQL — gaps-and-islands + HAVING >= 3

WITH ranked AS (
  SELECT user_id, filed_date,
    filed_date - (ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY filed_date))::int AS grp
  FROM filings
)
SELECT user_id, COUNT(*) AS streak
FROM ranked
GROUP BY user_id, grp
HAVING COUNT(*) >= 3;

3. The mechanism — why filed_date − row_number labels each island

Within a streak, both filed_date and the user-partitioned row_number increment by 1 each row. Their difference is constant. As soon as a date skips, filed_date jumps forward but row_number only ticks 1 — the difference shifts to a new value. Distinct difference values = distinct streaks.

4. Persona walk-through — what each user emits

Persona A — U_001 "Habitual"

Two islands → the GROUP BY emits two rows: (streak=5) and (streak=3). Both pass HAVING >= 3 — this user shows up twice in the result, which is usually not what compliance wants. To get longest streak only:

-- Wrap the original query and aggregate again
SELECT user_id, MAX(streak) AS longest_streak
FROM (/* the gaps-and-islands query */) GROUP BY user_id;
Persona B — U_002 "Long Resurrect"

Streak 1: Jan 28-30 (3 days). Streak 2: Apr 18-20 (3 days). Both islands pass HAVING. Inter-streak gap is ~78 days — this is the lifecycle signal. The basic query returns both streaks but doesn't expose the gap. The lifecycle layer (next section) does.

Persona C — U_003 "Churned"

One streak, Mar 03-05 (3 days), passes HAVING. But today is May 1 — 57 days since last filing. The base query says "yes, qualifying streak"; the lifecycle layer says "churned" because today − streak_end > 28 with no resurrection.

Persona D — U_004 "Open"

Apr 28-29 = 2 consecutive days. Fails the HAVING >= 3 filter — doesn't appear in the result. But they're an active user; the dashboard shouldn't lose them. Lifecycle perspective: they're "open" / in grace period. Use the unfiltered streak table for retention dashboards; the HAVING filter is for the specific "3+" compliance ask.

5. Strategic explanations — lifecycle on top of streaks

The full pipeline — streaks + next_activity + exit_status
WITH d AS (
  SELECT DISTINCT user_id, filed_date FROM filings
),
ranked AS (
  SELECT user_id, filed_date,
    filed_date - (ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY filed_date))::int AS grp
  FROM d
),
streaks AS (
  SELECT user_id, grp,
    MIN(filed_date) AS streak_start,
    MAX(filed_date) AS streak_end,
    COUNT(*)        AS streak_days,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY MIN(filed_date)) AS streak_id
  FROM ranked GROUP BY user_id, grp
),
next_activity AS (
  SELECT s.user_id, s.streak_id, s.streak_end,
    MIN(d.filed_date) AS next_filed
  FROM streaks s
  LEFT JOIN d
    ON d.user_id = s.user_id
   AND d.filed_date > s.streak_end
  GROUP BY s.user_id, s.streak_id, s.streak_end
),
enriched AS (
  SELECT s.user_id, s.streak_id, s.streak_start, s.streak_end, s.streak_days,
    n.next_filed,
    CASE WHEN n.next_filed IS NOT NULL THEN n.next_filed - s.streak_end END AS gap_days,
    DATE '2026-05-01' - s.streak_end AS days_since_streak_end
  FROM streaks s LEFT JOIN next_activity n USING (user_id, streak_id, streak_end)
)
SELECT user_id, streak_id, streak_start, streak_end, streak_days, next_filed,
  CASE
    WHEN gap_days IS NOT NULL AND gap_days <= 28              THEN 'continuing'
    WHEN gap_days IS NOT NULL AND gap_days <= 90              THEN 'resurrected'
    WHEN gap_days IS NOT NULL AND gap_days  > 90              THEN 'long_resurrect'
    WHEN gap_days IS NULL     AND days_since_streak_end > 90  THEN 'churned'
    WHEN gap_days IS NULL                                     THEN 'open'
  END AS exit_status
FROM enriched
ORDER BY user_id, streak_id;

The threshold-calibration follow-up

The 28/90 split is a Mixpanel-template default, not a universal truth:

  • Tax software (this question): cycles are annual; "long resurrect" is anyone returning after the next April. Calibrate to 365/730.
  • Subscription B2C (Spotify, Netflix): 30/90 is fine.
  • Grocery (Instacart): 7/30 — weekly cadence.
  • Travel (Booking.com): 90/365 — once-a-year is normal.

Always derive thresholds from the data — e.g., the 80th percentile of recurring inter-event gaps among confirmed-active users.

D1 / D7 / D28 retention flags — the leaderboard metric

Beyond "did they ever come back?", the growth team usually wants the granular retention rates. Add three boolean flags to enriched:

CASE WHEN gap_days <= 1  THEN 1 ELSE 0 END AS d1_retained,
CASE WHEN gap_days <= 7  THEN 1 ELSE 0 END AS d7_retained,
CASE WHEN gap_days <= 28 THEN 1 ELSE 0 END AS d28_retained

Aggregate over all streaks to compute fleet-wide retention curves; aggregate over the most-recent streak per user for the "current state of the funnel".

The "tolerant streak" variant — allow a 1-day grace

For tax filing especially, weekends or single-day skips shouldn't break a streak. Use LAG instead of pure gaps-and-islands:

WITH d AS (SELECT DISTINCT user_id, filed_date FROM filings),
flagged AS (
  SELECT user_id, filed_date,
    CASE WHEN LAG(filed_date) OVER (PARTITION BY user_id ORDER BY filed_date)
              >= filed_date - INTERVAL '2 days' THEN 0 ELSE 1 END AS new_streak
  FROM d
),
sessioned AS (
  SELECT user_id, filed_date,
    SUM(new_streak) OVER (PARTITION BY user_id ORDER BY filed_date) AS streak_id
  FROM flagged
)
SELECT user_id, streak_id, COUNT(*) AS days
FROM sessioned GROUP BY user_id, streak_id HAVING COUNT(*) >= 3;

Replace 2 days with whatever tolerance the business defines. Same shape solves "30-minute web sessions" and "audio playback sessions with 5-second tolerance".

The HAVING-on-streaks-vs-on-users distinction

The original query's HAVING COUNT(*) >= 3 filters per island — "give me every streak of 3+". A user with one 5-day streak appears once; a user with two 3-day streaks appears twice. The follow-up question is usually one of:

  • "Users who have any 3+ streak" → SELECT DISTINCT user_id FROM (...)
  • "Users whose longest streak is 3+" → HAVING MAX(...) >= 3 on the outer aggregation
  • "Users with at least 5 separate streaks of 3+" → count islands per user

Always pin which interpretation the requester means before writing code.

Dialect cheat sheet for date arithmetic
-- Postgres / DuckDB / Snowflake — date - integer = date
filed_date - ROW_NUMBER() OVER (...)::int

-- BigQuery — DATE_SUB with INT64
DATE_SUB(filed_date, INTERVAL CAST(ROW_NUMBER() OVER (...) AS INT64) DAY)

-- SQLite — julianday
julianday(filed_date) - ROW_NUMBER() OVER (...)

-- MySQL — TO_DAYS
TO_DAYS(filed_date) - ROW_NUMBER() OVER (...)
