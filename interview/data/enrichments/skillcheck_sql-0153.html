An analytics dashboard needs running totals on a daily metric: as of each day, the cumulative sum, the count of days so far, and the running average. The question explicitly contrasts the modern window-function approach with the pre-2003 self-join. The interviewer is probing whether you can write a correct cumulative window frame — and whether you know the frame clause is what makes it "to date".

1. The source data — metric

dv(narrative)
2025-01-0110first day — running stats equal the row itself
2025-01-0220cumulative 30
2025-01-0315cumulative 45
2025-01-0425cumulative 70
2025-01-0530last day — cumulative 100

The boundary case is the first row: with no prior days, sum_to_date, count_to_date, and avg_to_date must equal the day's own values (10, 1, 10.00). A correct cumulative frame includes the current row, so day one is self-consistent rather than empty.

2. The SQL — window functions with a cumulative frame

SELECT d, v,
       SUM(v)   OVER (ORDER BY d ROWS UNBOUNDED PRECEDING) AS sum_to_date,
       COUNT(*) OVER (ORDER BY d ROWS UNBOUNDED PRECEDING) AS count_to_date,
       ROUND(AVG(v) OVER (ORDER BY d ROWS UNBOUNDED PRECEDING)::numeric, 2) AS avg_to_date_2dp
FROM metric
ORDER BY d;

Each window aggregate runs over a frame: ORDER BY d ROWS UNBOUNDED PRECEDING means "every row from the start of the partition up to and including the current row". As the engine walks the rows in date order, the frame grows by one each step, so the aggregates accumulate.

Day one's running stats are 10 / 1 / 10.00 — the row itself, because the frame includes the current row. Each later row adds v to the sum and 1 to the count; avg_to_date is always sum_to_date / count_to_date (e.g. 70 / 4 = 17.50).

4. Strategic explanations

The frame clause is the cumulation

A window function without a frame would aggregate over a different scope. ROWS UNBOUNDED PRECEDING is shorthand for ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — "all rows up to and including this one". That growing frame is precisely what turns a plain SUM into a running total. Change the frame to ROWS BETWEEN 2 PRECEDING AND CURRENT ROW and you get a 3-day moving sum instead. The aggregate function describes what; the frame describes over which rows.

Why ROWS and not the default RANGE — the tie trap

When you supply an ORDER BY but omit the frame, the SQL default is RANGE UNBOUNDED PRECEDING, not ROWS. RANGE frames by value: all rows whose d equals the current d collapse into one frame step. With one row per day that happens to match ROWS — but if two rows shared a date, RANGE would give them an identical running total (both seeing the full tied group), whereas ROWS increments strictly row by row. Writing ROWS explicitly makes the per-row cumulation unambiguous and tie-proof.

Window functions vs the pre-2003 self-join

The old approach was a correlated self-join: SELECT m1.d, m1.v, (SELECT SUM(v) FROM metric m2 WHERE m2.d <= m1.d) …. It returns the same numbers but rescans the table for every row — O(n²). The window function sorts once and sweeps the rows in a single pass maintaining a running accumulator — O(n log n), and O(n) if an index already supplies date order. On a large table the window form is not just cleaner, it is the only viable answer; the self-join is a historical artefact from before SQL:2003 standardised window functions.

COUNT(*) over the frame, and the WINDOW alias option

COUNT(*) OVER (…) counts rows in the frame — here the day ordinal, 1..5. The three aggregates repeat the identical OVER clause; that is correct but verbose. A named window — … WINDOW w AS (ORDER BY d ROWS UNBOUNDED PRECEDING), then SUM(v) OVER w — defines the frame once and is the cleaner senior style when several functions share a window.

NULLs, ordering, and dialect

SUM and AVG skip NULL v; COUNT(*) still counts the row, so on a NULL-valued day count_to_date advances but the running average is over fewer real values — usually fine, but know it. The outer ORDER BY d governs display order and is separate from the window's internal ORDER BY. Window functions and explicit ROWS frames are standard in Postgres, SQL Server, Oracle, modern MySQL 8+, and SQLite 3.25+; the ::numeric cast before two-argument ROUND is the Postgres-specific touch.

Senior signal: build a running total with a window aggregate over an explicit ROWS UNBOUNDED PRECEDING frame — the frame, not the function, is what cumulates — spell out ROWS to avoid the default-RANGE tie trap, and know the window form replaces the O(n²) pre-2003 correlated self-join with a single O(n log n) pass.
