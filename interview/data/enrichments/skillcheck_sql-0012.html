This is a deliberately minimal warm-up: produce the integers 1 through 10 with no source table at all. The interviewer is checking that you understand the bare mechanics of a recursive CTE — an anchor seed, a recursive step, and a termination predicate — stripped of any join or business logic. It is the "hello world" of WITH RECURSIVE.

1. The source data — none

note
There is no input table. The schema declares an unused _dummy placeholder; the result is generated entirely from the recursive CTE itself.
The "edge cases" here are the boundaries of the sequence: the first value (1, from the anchor) and the last value (10, where the recursion must stop and not overshoot to 11).

With no rows to read, every row in the output is synthesised. The whole question is whether the recursion starts at the right number and stops at the right number — an off-by-one in the predicate is the only way to get this wrong.

2. The SQL — a number-generating recursive CTE

WITH RECURSIVE nums(n) AS (
  SELECT 1
  UNION ALL
  SELECT n + 1 FROM nums WHERE n < 10
)
SELECT n FROM nums ORDER BY n;

The anchor is just SELECT 1 — it seeds the CTE with a single row, n = 1. The recursive member reads the previous row's n, adds 1, and emits it — but only WHERE n < 10. That predicate is the brake: once n reaches 10, the condition 10 < 10 is false, the recursive member produces no row, and the recursion halts.

3. The resulting analysis

n(produced by)
1the anchor
2iteration 1 — from n=1, since 1 < 10
……
9iteration 8 — from n=8
10iteration 9 — from n=9, since 9 < 10

Exactly ten rows, 1 through 10. The value 10 is produced — it comes from the row n = 9, which satisfies 9 < 10. But n = 10 itself fails 10 < 10, so no 11 is generated. The predicate n < 10 therefore yields an inclusive upper bound of 10 — the classic boundary subtlety.

4. Strategic explanations

Why the brake is n < 10 and not n <= 10

The predicate gates whether the next row is produced, so it is one step ahead of the value you want to stop at. WHERE n < 10 lets the row n = 9 generate 10, then the row n = 10 is blocked — final value 10. Use WHERE n <= 10 and the row n = 10 would generate 11; the sequence runs to 11. The rule: to make the last value k, gate on n < k. This off-by-one is the single thing the question tests.

The termination guarantee — and what removes it

Recursion stops when an iteration produces zero new rows; here that happens the instant the predicate fails. The danger in number generators is forgetting the predicate entirely — SELECT n + 1 FROM nums with no WHERE counts upward forever until the engine's recursion limit aborts it. Unlike a tree walk (which terminates naturally when it runs out of nodes), a synthetic sequence has no natural end, so the stop condition is mandatory, not optional.

Dialect notes — the idiomatic shortcuts

PostgreSQL rarely needs this hand-rolled: SELECT generate_series(1, 10) AS n does the same in one line and is the idiomatic choice. SQL Server has no generate_series before 2022, so the recursive CTE shown here is the standard trick there (often with OPTION (MAXRECURSION 0) to lift the default 100-level cap for longer ranges). Oracle offers SELECT level FROM dual CONNECT BY level <= 10. The recursive-CTE form is the most portable, which is why it is worth knowing cold.

What a generated sequence is actually for

Generating 1..10 is never the real goal — the technique matters because a contiguous number or date series is the scaffold for gap-filling. Need every calendar day in a month even when some have no data? Generate the dates, then LEFT JOIN the facts onto them so missing days surface as zero rows instead of vanishing. Recognising this warm-up as the foundation of "report all buckets, including empty ones" is the senior framing.

Senior signal: build a number series with an anchor (SELECT 1), a +1 recursive step, and — critically — a mandatory termination predicate, gating on n < k to make k the inclusive last value; know generate_series is the Postgres shortcut, and that the real use of a generated series is left-joining to fill reporting gaps.
