Elastic's catalog team wants to surface "premium" products — the ones priced above the average for their category. An electronics item at $300 might be cheap; a stationery item at $300 is absurd. The interviewer is probing whether you can compare each row against a moving target that depends on the row itself — the canonical use of a correlated subquery.

1. The source data — products

product_idpricecategory

450B
5150B

Category A averages 200 (100, 200, 300); category B averages 80 (50, 150, 40). Product 5 at 150 is below A's average but well above B's — proof the comparison must be per category, not against one global mean.

2. The SQL — correlated AVG subquery

SELECT product_id, category, price
FROM products p
WHERE price > (SELECT AVG(price) FROM products WHERE category = p.category);

The inner query is correlated: category = p.category ties it to the outer row. For each product it recomputes the average of that product's own category, then keeps the row only if its price beats that average.

3. The resulting analysis

product_idcategoryprice
3A300
5B150

Only two products clear their category bar: product 3 (300 > A's 200) and product 5 (150 > B's 80). Product 2 at 200 is equal to A's average, and a strict > excludes it. Product 1 (100) and the cheap B items fall below. The result correctly judges each item against its peers, not the whole catalog.

4. Strategic explanations

Why a global average would be wrong

The naive version computes one average over every product:

SELECT product_id, category, price FROM products
WHERE price > (SELECT AVG(price) FROM products);  -- global avg = 140

That uncorrelated subquery runs once and returns 140. It would keep products 2, 3 and 5 — promoting product 2, which is merely average for category A, and it would mishandle a catalog where one category dwarfs another. The correlation WHERE category = p.category is the single line that makes the comparison fair.

The window-function rewrite — one pass, not n

The correlated subquery re-scans the table once per row — O(n²). A window function computes every category average in a single pass:

SELECT product_id, category, price
FROM (
  SELECT product_id, category, price,
         AVG(price) OVER (PARTITION BY category) AS cat_avg
  FROM products
) t
WHERE price > cat_avg;

The window form needs a subquery because you cannot reference a window result in WHERE (windows are evaluated after WHERE). On a large catalog this is dramatically faster and the intent — "compare to my partition's average" — reads off the page.

The strict-inequality trap with ties

price > avg drops a product priced exactly at its category average — product 2 here. Is that intended? "Above average" usually means strictly above, so > is right. But if the spec is "at or above", it must be >=. With >= product 2 would join the result. Confirm the boundary; it is a one-character decision that changes the answer.

NULL prices and the category filter

AVG(price) skips NULL prices, so a category's average is over its priced items only. A product whose own price is NULL fails price > (...) (NULL comparisons are unknown) and is excluded — usually fine. More subtly, if category itself is NULL, category = p.category is never true, so the subquery averages an empty set, returns NULL, and that row is dropped. NULL categories silently vanish — flag that to the catalog owner.

The type tag says "Joins"

This question is tagged "SQL / Joins", but there is no join — it is a correlated subquery. The two are related (a correlated subquery is conceptually a per-row lookup, and the window rewrite avoids both), but do not force a literal JOIN here. Solve the question; the tag is a loose label.

Senior signal: recognise the correlation category = p.category as what makes the comparison per-category rather than global, offer the AVG() OVER (PARTITION BY ...) rewrite that turns O(n²) into one pass, and pin down whether "above average" is strict > or inclusive >= before committing.
