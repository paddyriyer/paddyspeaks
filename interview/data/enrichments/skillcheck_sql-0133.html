An observability team wants an event-rate chart with 15-minute granularity — the classic "bucket the timeline" task behind every dashboard histogram. The interviewer is checking whether you can floor each timestamp to a quarter-hour boundary with arithmetic alone, and whether you understand exactly which side of 09:15:00 a 09:15:00 event lands on.

1. The source data — events

event_idts(narrative)

22025-08-01 09:10:00bucket 09:00
32025-08-01 09:14:59bucket 09:00 — last second before the boundary

Events 3 and 4 straddle the 09:15 line one second apart — the boundary edge case. And the 09:45 bucket has zero events: a plain GROUP BY simply omits it rather than emitting a 0.

2. The SQL — floor the minute, multiply back into an interval

SELECT date_trunc('hour', ts)
       + (EXTRACT(MINUTE FROM ts)::int / 15) * INTERVAL '15 minutes' AS bucket_start,
       COUNT(*)::int AS event_count
FROM events
GROUP BY 1
ORDER BY 1;

date_trunc('hour', ts) snaps to the top of the hour. EXTRACT(MINUTE…)::int / 15 is integer division — minutes 0–14 give 0, 15–29 give 1, etc. Multiplying that 0–3 quotient by a 15-minute interval and adding it back reconstructs the bucket's start.

09:14:59 and 09:15:00 split cleanly into adjacent buckets. Note there is no 09:45 row: GROUP BY only produces buckets that have data.

4. Strategic explanations

Why integer division is the engine of the bucket

The whole floor-to-15-minutes trick is one integer division. In PostgreSQL, dividing two integers truncates toward zero, so 14/15 = 0 and 29/15 = 1. That quotient is the bucket index within the hour. If EXTRACT returned a numeric and you forgot ::int, the division would be exact (0.933…) and every event would collapse to the top of the hour — the cast is load-bearing, not decorative.

Buckets are half-open [start, start+15)

An event exactly on a boundary — 09:15:00 — has minute 15, quotient 1, and lands in the 09:15 bucket. So each bucket includes its start instant and excludes its end instant: the half-open interval [09:15, 09:30). State this convention explicitly; "which bucket owns the boundary" is the single most common source of off-by-one disagreement in time-series work.

Empty buckets are silently dropped

The 09:45–10:00 window has no events, so it produces no group and no row. If the chart needs a continuous axis — a visible 0 at 09:45 — you must generate_series the full set of buckets and LEFT JOIN the counts onto it, coalescing missing counts to 0. Knowing that aggregation cannot invent absent rows, and offering the gap-fill, is the senior move.

GROUP BY 1 — convenience with a caveat

GROUP BY 1 / ORDER BY 1 reference the first select-list item by position, avoiding repetition of the long bucket expression. It is concise and supported in PostgreSQL and MySQL, but positional grouping is fragile — reorder the select list and the query silently groups by the wrong thing. In shared code, repeating the expression or naming it via a CTE is safer.

Dialect — date_bin and the bucketing helpers

PostgreSQL 14+ offers date_bin('15 minutes', ts, TIMESTAMP '2025-08-01 00:00'), which buckets directly against an origin and is clearer than the arithmetic. TimescaleDB has time_bucket; BigQuery has TIMESTAMP_BUCKET; Snowflake uses TIME_SLICE. The date_trunc + integer-division form shown here is the most portable and works on every PostgreSQL version — mention date_bin as the modern one-liner.

Senior signal: floor a timestamp to an N-minute bucket with date_trunc plus integer division (and keep the explicit ::int cast that makes the division truncate), state the half-open [start, start+N) convention so boundary events are unambiguous, and proactively gap-fill empty buckets with generate_series — or reach for date_bin where available.
