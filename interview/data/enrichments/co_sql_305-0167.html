Qualcomm's finance team keeps quarterly revenue in a long, tall table — one row per quarter — but the board deck wants it wide: one row per year with Q1…Q4 as columns. This is the classic pivot, and the interviewer is watching whether you reach for conditional aggregation cleanly, and whether you know what happens to a quarter that simply has no data.

1. The source data — sales

yearquarteramount
2023Q1100
2023Q2150
2023Q3200
2023Q4250
2024Q1300
2024Q2350
2024Q4400

2023 is complete, but 2024 has no Q3 row at all — the quarter has not closed, or the data has not landed. That missing row is the whole interview: it decides whether the pivoted Q3 cell is NULL or 0.

2. The SQL — SUM(CASE WHEN …) pivot

SELECT year,
  SUM(CASE WHEN quarter='Q1' THEN amount END) AS q1,
  SUM(CASE WHEN quarter='Q2' THEN amount END) AS q2,
  SUM(CASE WHEN quarter='Q3' THEN amount END) AS q3,
  SUM(CASE WHEN quarter='Q4' THEN amount END) AS q4
FROM sales GROUP BY year;

Group by year, then for each quarter use a CASE that yields amount on a match and NULL otherwise. SUM collapses the four CASE columns into one row per year.

3. The resulting analysis

yearq1q2q3q4
2023100150200250
2024300350NULL400

Two years, four quarter columns. 2024's q3 comes back NULL, not 0: there was no Q3 row, so every CASE for that year produced NULL, and SUM over an all-NULL set is itself NULL. Whether that is right depends on what "no data" means to the business.

4. Strategic explanations

The NULL-vs-0 decision — the question behind the question

A missing quarter could mean "we genuinely sold nothing" (then 0 is right) or "the data has not arrived yet" (then NULL is honest). SUM(CASE…) gives you NULL by default; force a 0 with COALESCE only if the business says so:

SUM(CASE WHEN quarter='Q3' THEN amount ELSE 0 END) AS q3
-- now an empty quarter shows 0, indistinguishable from a real zero-sales quarter

The senior move is to flag the ambiguity, not to silently pick one. A NULL that means "unknown" is information; a 0 that hides it is a bug waiting to mislead the board.

The ELSE clause is doing nothing here — on purpose

CASE WHEN quarter='Q1' THEN amount END has no ELSE, so non-Q1 rows yield NULL. That is exactly what you want inside SUM: NULLs are skipped, so only the true Q1 rows are added. Writing ELSE 0 instead would still give the same total (adding zeros changes nothing) — but it changes the empty-group behaviour above. Know which one you are choosing.

Postgres has a native crosstab

The tablefunc extension offers crosstab(), and other dialects have PIVOT (SQL Server, Oracle). They are terser but rigid — the column list is fixed and the syntax is engine-specific. Conditional aggregation is portable and readable, and it composes with other aggregates. For an interview, SUM(CASE…) is the answer that travels.

Dynamic columns — when the quarters are not known in advance

This works because we know the four quarters. If the pivot columns were, say, an open-ended set of product codes, no single static query can produce them — SQL's result shape is fixed at parse time. That needs dynamic SQL: query the distinct values, build the CASE list as a string, execute it. Saying this out loud shows you understand the limit of the pattern.

Duplicate rows per quarter

If sales can hold more than one row for the same (year, quarter) — say one per region — the SUM correctly adds them. Had you reached for MAX(CASE…) instead (a common pivot variant), you would silently drop all but the largest. SUM is the safe aggregate when a year/quarter cell represents a total.

Senior signal: write the pivot as SUM(CASE WHEN…), explain that a missing quarter yields NULL not 0 and decide that on purpose with COALESCE, prefer portable conditional aggregation over engine-specific PIVOT/crosstab, and note that an unknown set of pivot columns requires dynamic SQL.
