An e-commerce analytics team wants each customer's longest streak of consecutive calendar days with at least one order — a loyalty/engagement metric. This is the classic "gaps and islands" problem, and the interviewer is probing whether you know the row-number-offset trick that turns a run of consecutive dates into a constant grouping key.

2. The SQL — row-number-offset gaps and islands

WITH t1 AS (
  SELECT customerid, orderdate, orderid,
         ROW_NUMBER() OVER (PARTITION BY customerid ORDER BY orderdate ASC) AS id
  FROM customer_order
), t2 AS (
  SELECT customerid, orderdate, orderid, id,
         orderdate::date - id * INTERVAL '1 day' AS streak_window
  FROM t1
), t3 AS (
  SELECT customerid, streak_window, COUNT(orderid) AS consecutive_orders
  FROM t2
  GROUP BY customerid, streak_window
)
SELECT customerid, MAX(consecutive_orders) AS max_consecutive_days
FROM t3
GROUP BY customerid;

Customer 1 correctly reports 3, taking the longest island, not the count of islands. Customer 2 reports 1 — the query never overstates a streak when days are not adjacent.

4. Strategic explanations

The core trick — date minus row number is constant within a run

Within a streak of consecutive dates, the date increases by exactly 1 each step, and ROW_NUMBER() also increases by exactly 1. So orderdate - id stays constant across the whole run — and changes the moment a gap appears. That constant becomes the island's identity key (streak_window). Grouping by it and counting rows gives each run's length.

Why id * INTERVAL '1 day'

id is a plain integer; subtracting it directly from a date is ambiguous across engines. Multiplying by INTERVAL '1 day' makes the subtraction an unambiguous date arithmetic operation in Postgres. The resulting streak_window values are not meaningful dates themselves — they are only labels — but rows in the same run share the same label.

The duplicate-date trap — ROW_NUMBER vs DENSE_RANK

If a customer places two orders on the same day, ROW_NUMBER still assigns them different sequence numbers, so orderdate - id differs and the run is wrongly split. The robust fix is to either deduplicate dates first (SELECT DISTINCT customerid, orderdate) or use DENSE_RANK() OVER (ORDER BY orderdate), which gives same-day rows the same rank. With the sample data every (customer, date) is unique so it does not bite here — but on real order tables it absolutely will. This is the single most important caveat to raise.

"Consecutive days" vs "consecutive orders"

The query counts orders per island. Once dates are deduplicated, that equals the number of consecutive days — which is what the question wants. Keep the distinction explicit: the metric is calendar-day span, not order volume. Naming it precisely shows you read the requirement carefully.

The window-function alternative

Modern engines can do this without the offset trick: flag each row where orderdate - LAG(orderdate) > 1 as a new-island boundary, then SUM those flags as a running total to label islands. It is more explicit about "a gap starts a new group" and extends naturally to "consecutive within 2 days" style rules. The row-number-offset form is the most compact; mention the LAG + running-sum version to show range.

Senior signal: name this gaps-and-islands, explain that orderdate - ROW_NUMBER() is invariant within a consecutive run and so labels each island, take the MAX island length per customer — and pre-empt the duplicate-date bug by deduplicating dates or switching ROW_NUMBER to DENSE_RANK.
