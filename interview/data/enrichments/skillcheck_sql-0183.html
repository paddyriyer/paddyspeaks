A product analytics team wants a cohort retention table: given users' first-ever activity month (their "cohort"), how many of them returned in each of the next three calendar months? The interviewer is probing three interlocking skills — computing cohort assignment with MIN, generating a complete month × offset grid with a CROSS JOIN to a VALUES list, and using a LEFT JOIN to guarantee zero counts for empty cells rather than omitting them.

Three cohorts: Jan (users 1, 2), Feb (users 3, 4), Mar (user 5). Each cohort is observed at offsets 0, 1, 2, and 3 months from its cohort month. User 4 never returned after Feb — a zero-count cell that must still appear in the output.

2. The SQL — cohort CTE + sparse activity + grid cross join

WITH first_event AS (
  SELECT user_id,
         DATE_TRUNC('month', MIN(event_date))::date AS cohort_month
  FROM event
  GROUP BY user_id
),
active AS (
  SELECT DISTINCT user_id,
         DATE_TRUNC('month', event_date)::date AS active_month
  FROM event
),
grid AS (
  SELECT DISTINCT f.cohort_month, m.months_since
  FROM first_event f
  CROSS JOIN (VALUES (0),(1),(2),(3)) AS m(months_since)
)
SELECT g.cohort_month,
       g.months_since,
       COUNT(DISTINCT a.user_id)::int AS active_users
FROM grid g
LEFT JOIN first_event f ON f.cohort_month = g.cohort_month
LEFT JOIN active a
  ON a.user_id = f.user_id
  AND a.active_month = (g.cohort_month + (g.months_since || ' months')::interval)::date
GROUP BY g.cohort_month, g.months_since
ORDER BY g.cohort_month, g.months_since;

The grid CTE manufactures every (cohort_month, offset) cell. The two LEFT JOINs bring in first_event to find which users belong to each cohort, and then active to test whether a user had any event in the target calendar month. COUNT(DISTINCT a.user_id) returns 0 for rows where no match was found.

The Jan cohort reads 2, 1, 2, 1 across offsets 0–3: user 2 has no February event (its next activity after January is Mar-01), so it drops out at offset 1 and reappears at offset 2. The Feb cohort's offsets 2 and 3 are genuine zeros — user 4 never returned after sign-up — yet those rows still appear because the grid manufactured them. Surfacing those zeros is the entire point of the pattern.

4. Strategic explanations

The grid pattern — why it is essential

Without the manufactured grid, a GROUP BY on actual events will silently drop any (cohort, offset) cell that has zero active users. The CROSS JOIN (VALUES (0),(1),(2),(3)) creates every legal cell first; the subsequent LEFT JOINs fill in whatever activity exists and leave NULL (which COUNT converts to 0) for empty cells. This is the standard pattern for any "include zeros" reporting requirement.

Why DISTINCT in the active CTE

A user can have multiple events in the same month. The active CTE deduplicates to one row per (user_id, active_month) so that COUNT(DISTINCT a.user_id) in the final SELECT does not over-count. Alternatively, COUNT(DISTINCT a.user_id) alone handles this — but the DISTINCT in the CTE makes the join cheaper since there are fewer rows to scan.

Date arithmetic — (cohort_month + (months_since || ' months')::interval)::date

PostgreSQL interval arithmetic on a date returns a timestamp, hence the outer ::date cast. The string concatenation months_since || ' months' converts the integer offset into an interval literal like '2 months'. An alternative is cohort_month + months_since * INTERVAL '1 month', which is slightly more explicit and avoids string casting.

Edge cases

A user who signs up at the end of January still has cohort_month = 2024-01-01 because DATE_TRUNC('month', ...) truncates to the first of the month. Offset 1 for a January cohort is always February 1st regardless of the user's actual signup day — cohort retention uses calendar months, not 30-day windows. This distinction matters and is worth stating explicitly in an interview.

Dialect notes

PostgreSQL's DATE_TRUNC and interval arithmetic are clean here. In BigQuery use DATE_TRUNC(date, MONTH) and DATE_ADD(cohort_month, INTERVAL months_since MONTH). In SQL Server use DATEFROMPARTS(YEAR(date),MONTH(date),1) and DATEADD(month, months_since, cohort_month). Snowflake supports DATE_TRUNC('month', date) and DATEADD('month', months_since, cohort_month). The VALUES list cross-join is ANSI SQL and works everywhere.

Senior signal: manufacture the complete (cohort, offset) grid with a CROSS JOIN (VALUES ...) before joining activity — the LEFT JOIN ensures zero counts appear; dedup active months in a CTE to keep the outer count clean; and explain the calendar-month vs. rolling-window distinction for cohort semantics.
