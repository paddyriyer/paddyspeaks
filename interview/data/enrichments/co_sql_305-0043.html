RedHat's revenue analysts want a category-by-category comparison: for each product category, the average sale in the North region beside the average sale in the South. It is a pivot — turning region values into columns — built with conditional aggregation. The interviewer asking this wants to see that you reach for AVG(CASE WHEN ...) rather than a join, and that you know what AVG returns when a category has no sales in one of the regions.

1. The source data — sales

categoryregionamount
BooksNorth100
BooksNorth200
BooksSouth150
ToysNorth300
ToysSouth50
ToysSouth70
GamesNorth400

Books and Toys sell in both regions, but Games sells only in the North — it has zero South rows. That asymmetric category is the whole interview: it decides what shows up in the avg_south column.

2. The SQL — AVG(CASE WHEN ...)

SELECT category,
  AVG(CASE WHEN region = 'North' THEN amount END) AS avg_north,
  AVG(CASE WHEN region = 'South' THEN amount END) AS avg_south
FROM sales GROUP BY category;

For each category, the CASE passes amount through only for the matching region and yields NULL otherwise. AVG then averages just the non-NULL values — effectively a region-filtered average, one per column.

3. The resulting analysis

categoryavg_northavg_south
Games400.0000000000000000NULL
Toys300.000000000000000060.0000000000000000
Books150.0000000000000000150.0000000000000000

Books North = (100+200)/2 = 150; Toys South = (50+70)/2 = 60. The key row is Games: with no South sales, every CASE for South returns NULL, and AVG over nothing-but-NULL returns NULL — not 0. That distinction (no data vs a real zero) is exactly what the interviewer is probing.

4. Strategic explanations

Why the CASE has no ELSE

The CASE deliberately omits ELSE, so a non-matching region yields NULL, and AVG skips it. Writing ELSE 0 instead would be a real bug: the zeros would enter the average and crush it. For Books' South average, ELSE 0 would give (0+0+150+0+0+0+0)/7 — meaningless. With AVG, "not this region" must map to NULL, never 0.

NULL vs 0 — making the empty cell explicit

Games' avg_south is NULL because there is genuinely no South data. If a dashboard would rather show a placeholder, make that choice on purpose:

SELECT category,
  COALESCE(AVG(CASE WHEN region='North' THEN amount END), 0) AS avg_north,
  COALESCE(AVG(CASE WHEN region='South' THEN amount END), 0) AS avg_south
FROM sales GROUP BY category;

But know what you've done: a 0 now means "no sales", indistinguishable from a true average of 0. Leaving NULL preserves that signal — the better default for analytics.

The FILTER alternative — clearer intent

Postgres offers a more readable form of the same pivot — the aggregate FILTER clause:

SELECT category,
  AVG(amount) FILTER (WHERE region = 'North') AS avg_north,
  AVG(amount) FILTER (WHERE region = 'South') AS avg_south
FROM sales GROUP BY category;

Identical results, but the intent — "average the rows where region is North" — reads off the page. CASE is the portable form that works on every engine; FILTER is the cleaner Postgres-native one.

Why not self-join the table twice?

One could join sales to itself, once filtered to North and once to South. It is slower (two passes plus a join), it explodes if either side has multiple rows per category, and it drops categories absent from one side unless you use outer joins. Conditional aggregation does it in a single GROUP BY pass — the right tool for pivoting a low-cardinality column.

Edge case: an unexpected third region

The query hard-codes North and South. A row with region = 'West' still forms a group and contributes to neither column — its amount silently disappears from both averages. If a category sells only in the West, it appears as a row with two NULLs. That's a correctness gap worth naming; a robust report enumerates every region or validates the domain.

Senior signal: pivot with AVG(CASE WHEN ...) and deliberately omit ELSE so non-matching rows become NULL not 0, explain that AVG over an all-NULL group returns NULL (no data, not zero), offer FILTER as the cleaner Postgres equivalent, and note that hard-coded region columns silently swallow any unlisted region.
