A compensation review asks: "for each employee, how does their salary compare to their department's average?" The reference solution names a CTE that pre-aggregates the per-department average, then joins it back to the employee rows and subtracts. The interviewer is probing whether you understand the join cardinality and whether you spot the cleaner window-function alternative.

1. The source data — employees

emp_idnamejob_titlesalarydept_id

Department 1 has two employees, department 2 has two, and department 3 has exactly one (Eve). The single-member department is the edge case — its average equals that one salary, so the employee's deviation must be exactly zero.

2. The SQL — CTE aggregation joined back

WITH dept_avg AS (
  SELECT dept_id, AVG(salary) AS avg_sal
  FROM employees
  GROUP BY dept_id
)
SELECT e.emp_id, e.salary, d.avg_sal,
  e.salary - d.avg_sal AS diff
FROM employees e
JOIN dept_avg d ON e.dept_id = d.dept_id;

Eve (emp 5) is the only person in department 3, so the department average is her salary and her diff is exactly $0. Every employee row survives the join because every dept_id has a matching aggregate.

4. Strategic explanations

The CTE pre-aggregates, then the join fans it back out

dept_avg computes one row per department — a small summary table. The main query joins each of the five employee rows to its single matching department-average row. The join is many-to-one (many employees, one average per department), so it does not multiply rows: five employees in, five rows out. Recognising that cardinality — and that it is safe — is the key reasoning step.

The window-function rewrite — no CTE, no join, one scan

The whole pattern collapses into a single windowed aggregate:

SELECT emp_id, salary,
  AVG(salary) OVER (PARTITION BY dept_id) AS avg_sal,
  salary - AVG(salary) OVER (PARTITION BY dept_id) AS diff
FROM employees;

AVG(salary) OVER (PARTITION BY dept_id) computes each department's average and attaches it to every row of that department — no grouping collapse, no join. This is the senior-preferred form: less code, one pass over the table, and no risk of the join accidentally fanning out if dept_avg ever produced duplicate dept_ids. The CTE-and-join version is fine and very readable; knowing the window equivalent is the differentiator.

The NULL dept_id trap — INNER JOIN silently drops rows

If an employee had a NULL dept_id, the GROUP BY dept_id would create a NULL-keyed average group, but the join condition e.dept_id = d.dept_id uses =, and NULL = NULL is unknown — so that employee would vanish from the result entirely. The window-function version keeps them (NULL becomes its own partition). With the join form, decide explicitly: use a LEFT JOIN if unassigned employees must still appear, or filter them out deliberately.

NULL salaries and integer division

AVG ignores NULL salaries, so a NULL salary shrinks the denominator rather than counting as zero. And a NULL-salary employee gets diff = NULL because NULL - avg is NULL. Also note AVG on an integer column can truncate in some dialects — cast to numeric (AVG(salary::numeric)) if fractional cents matter.

Why a CTE rather than an inline subquery

The CTE is purely for readability — dept_avg reads like a named step. It could equally be an inline derived table. Modern Postgres does not materialise a simple CTE by default, so there is no performance penalty; older versions (pre-12) treated CTEs as optimisation fences, which is worth a sentence if asked about query planning.

Senior signal: note the CTE-to-employee join is many-to-one so it cannot fan out, then offer AVG(salary) OVER (PARTITION BY dept_id) as the no-join single-scan rewrite — and flag that an INNER JOIN on dept_id silently drops employees with a NULL department, where the window form keeps them.
