Coinbase's operations team wants a single ops health number: on average, how many minutes does an order take from "received" to "processed"? Easy enough — subtract two timestamps. But the reference solution wraps the arithmetic in a baroque epoch conversion, and an order still sitting in the queue has a NULL processed_at. The interviewer wants to see you both simplify the math and reason about the unfinished orders.

1. The source data — orders

Three orders are processed (durations 30, 120, 15 minutes); order 4 is still pending with a NULL processed_at. That unfinished order is the interview: it must not count as a zero-minute order, and it must not count as anything at all.

2. The SQL — EPOCH diff to minutes

SELECT AVG(((EXTRACT(EPOCH FROM (processed_at - received_at)) / 86400.0) * 86400)/60)
         AS avg_minutes_to_process,
  SUM(CASE WHEN processed_at IS NOT NULL THEN 1 ELSE 0 END) AS processed_count
FROM orders;

processed_at - received_at is an interval; EXTRACT(EPOCH FROM ...) turns it into seconds. The / 86400.0 * 86400 pair cancels out (a no-op round-trip through days), leaving / 60 — seconds to minutes.

3. The resulting analysis

avg_minutes_to_processprocessed_count
55.03

Durations 30, 120 and 15 sum to 165; 165 ÷ 3 = 55 minutes. The pending order contributed nothing — processed_at - received_at is NULL when processed_at is NULL, and AVG skips NULLs. processed_count reports 3, the correct denominator. The query gets the right answer; it just takes a scenic route to get there.

4. Strategic explanations

The / 86400.0 * 86400 is dead code

Multiplying by 86400 immediately after dividing by it is an identity — it does nothing but obscure intent. The honest expression is just seconds-to-minutes:

SELECT AVG(EXTRACT(EPOCH FROM (processed_at - received_at)) / 60)
         AS avg_minutes_to_process,
       COUNT(processed_at) AS processed_count
FROM orders;

Same 55.0, far clearer. In an interview, calling out the redundant round-trip — and removing it — signals you read SQL critically rather than copying it.

COUNT(processed_at) beats the SUM(CASE...)

SUM(CASE WHEN processed_at IS NOT NULL THEN 1 ELSE 0 END) is a verbose way to count non-NULL values — which is exactly what COUNT(processed_at) does in one token. Both return 3. The CASE form is only needed for conditional counts on a different predicate; for "count the non-NULLs", use COUNT(column).

Why the NULL handling is actually correct

The crucial point: averaging duration over processed orders is the right denominator. Order 4 has no processing time yet — including it as 0 minutes would understate the true latency, and there is no honest non-NULL value to give it. Because the interval arithmetic yields NULL and AVG ignores NULLs, the pending order is excluded automatically. The two output columns agree: average over 3, count of 3. Make this explicit to the interviewer — it is a feature, not an accident.

The average-vs-median trap for latency

Order 2 took 120 minutes against 30 and 15 — one slow order drags the mean to 55, above two of the three values. Latency distributions are right-skewed, and operations teams usually care about the median or p95, not the mean:

SELECT percentile_cont(0.5) WITHIN GROUP (
         ORDER BY EXTRACT(EPOCH FROM (processed_at - received_at)) / 60)
         AS median_minutes
FROM orders;

Offering the percentile alongside the mean shows you understand the metric, not just the syntax.

Negative durations — the data-quality check

If a clock skew or a bad backfill makes processed_at earlier than received_at, the interval goes negative and silently pulls the average down. Defend with WHERE processed_at >= received_at, or surface the bad rows in a data-quality report. The mean of a metric is only as trustworthy as the rows feeding it.

Senior signal: spot and delete the /86400*86400 no-op, collapse the SUM(CASE...) into COUNT(column), explain that NULL processed_at correctly excludes pending orders from both numerator and denominator, and recommend a median or p95 for a skewed latency metric.
