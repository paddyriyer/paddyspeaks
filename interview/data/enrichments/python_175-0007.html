Google's data-infrastructure interviews ask this because exactly-once delivery is a myth — pub/sub systems redeliver, clients retry, and the same event arrives twice. A deduplicator with a time-to-live window is the standard defense: remember recent event IDs, but bound the memory so it does not grow forever. The question tests whether you can keep two data structures in sync — one for fast membership, one for ordered expiry.

1. The input data

A deduplicator with ttl_seconds = 10, then a stream of (event_id, timestamp) calls to is_new:

d = Deduplicator(ttl_seconds=10)
d.is_new('x', 100)
d.is_new('y', 105)
d.is_new('x', 108)
d.is_new('z', 115)
d.is_new('x', 116)

2. The approach

Two structures cooperate. self.seen is a dict event_id → timestamp giving O(1) "have I seen this?" checks. self.q is a deque holding (event_id, ts) in arrival order, which is also timestamp order, so the oldest entry is always at the left. Every call first runs an eviction loop: while the front entry's timestamp plus the TTL is strictly less than the current ts, it has expired — popleft() it from the deque and seen.pop(eid, None) it from the dict. After eviction, if event_id is still in seen it is a live duplicate, so return False; otherwise record it in both structures and return True. The deque gives ordered expiry; the dict gives fast lookup — neither alone suffices.

At ts 108, x's expiry is 100+10=110, not < 108, so x is still live → duplicate. At ts 115, x expired (110 < 115) but y (105+10=115) is not < 115, so y survives. By ts 116, x has long since aged out and counts as new again.

4. Strategic explanations

Complexity

Each call is amortized O(1): the lookup is O(1), and although the eviction loop can pop several entries in one call, every event is pushed once and popped once across the stream's lifetime. Space is O(w) where w is the number of distinct events inside the TTL window — the bound the design exists to guarantee.

The idiomatic alternative — the dict is ordered

Since Python 3.7 dicts preserve insertion order, the separate deque is technically redundant: you could iterate self.seen from the front and delete expired keys. The deque still earns its place because popleft() is a clean O(1) and the intent ("FIFO expiry queue") is explicit. Naming this tradeoff shows you know the language.

The edge case — a refreshed duplicate keeps the OLD timestamp

When a duplicate hits, the code returns False and does not update the stored timestamp. So the TTL counts from first sight, not last — a sliding-window-by-arrival policy. If the requirement were "expire 10s after the most recent occurrence," you would have to refresh seen[eid] and move the deque entry, which the current single-deque design cannot do cleanly.

The follow-up — out-of-order timestamps

"Events can arrive late, with a smaller ts than the previous call." The deque is then no longer sorted by timestamp, so front-eviction breaks. The fix is a min-heap keyed on expiry time instead of a deque, or bucketing events into fixed time windows. The interviewer is probing whether you assumed monotonic time without saying so.

The trap — an unbounded set

The tempting answer is if eid in seen: return False; seen.add(eid) with a plain set and no expiry. It is correct for correctness but leaks memory forever — on an infinite stream the set grows without bound. The TTL and the eviction loop are the entire reason this is a Medium, not an Easy.

Senior signal: a senior pairs a dict for lookup with an ordered structure for expiry, states the "TTL from first sight vs last sight" semantics out loud, and asks whether timestamps are monotonic before trusting deque-front eviction — a junior uses one unbounded set and leaks.
