A growth analyst inherits a users table where interests are crammed into one comma-separated tags_csv column — a denormalised mess — and asks: "how many users picked each tag?" The reference solution splits the string into an array, explodes it into rows, and aggregates. The interviewer is probing whether you can handle the dirty-data realities: stray whitespace, empty tokens, and NULL cells.

1. The source data — users

user_idnamecountrytags_csv
1AnnUSsql, python ,ml
2BobUKpython,devops
3CaraINsql,,python
4DanUS (empty string)
5EveCANULL

Four deliberate hazards: inconsistent spacing around Ann's tags, an empty token from the double comma in Cara's row, an empty string for Dan, and a NULL for Eve. A clean split must survive all four.

2. The SQL — split, unnest, aggregate

SELECT trim(tag) AS tag, COUNT(*) AS users
FROM users, UNNEST(string_to_array(tags_csv, ',')) AS tag
WHERE trim(tag) <> ''
GROUP BY trim(tag);

3. The resulting analysis

tagusersnote
devops1Bob only
ml1Ann — ml trimmed
python3Ann, Bob, Cara
sql2Ann, Cara

Four tags. The empty token from Cara's sql,,python is gone, Ann's spaced python and ml are normalised, and Dan (empty string) and Eve (NULL) contribute nothing — exactly the intended outcome.

4. Strategic explanations

The split-unnest pipeline — how it works

string_to_array(tags_csv, ',') turns the CSV text into a Postgres array. UNNEST(...) is a set-returning function: placed in the FROM clause it expands that array into one row per element, and a comma-join against users implicitly behaves like a LATERAL join — each user's row is paired with each of its exploded tags. The result is the normalised long form the original schema should have stored to begin with.

Why trim() appears three times — and the double-comma trap

Real CSV text has untidy spacing. trim(tag) in the SELECT normalises the displayed value, in the WHERE it ensures a token that is only whitespace counts as empty, and in GROUP BY it makes python and python aggregate together — without it they would be different groups. The double comma in sql,,python produces a genuine empty-string element; WHERE trim(tag) <> '' discards it. Skipping any of the three trims gives split or miscounted groups — the classic mistake here.

NULL and empty-string handling — the quiet edge cases

Two non-tag rows behave differently under the hood but land in the same place. Eve's tags_csv is NULL: string_to_array(NULL, ',') returns NULL, and UNNEST(NULL) yields zero rows, so Eve drops out before the filter. Dan's value is the empty string: it splits into a single empty-string element, which WHERE trim(tag) <> '' then removes. Note that COUNT(*) counts exploded tag-rows, not users — if one user listed the same tag twice it would be counted twice; use COUNT(DISTINCT user_id) for true per-user counts.

Dialect notes — this is Postgres-specific

string_to_array + UNNEST is Postgres. Elsewhere: BigQuery uses SPLIT() with UNNEST(); Snowflake has SPLIT_TO_TABLE (or LATERAL FLATTEN); SQL Server and modern MySQL/Oracle expose STRING_SPLIT / JSON_TABLE. The shape — split, explode, trim, filter, group — is universal even though the function names are not.

The real senior point — don't store CSV in a column

This whole query is a workaround for a denormalisation anti-pattern. A comma-separated column cannot be indexed per tag, cannot enforce a foreign key, and forces a string parse on every read. The proper design is a user_tags(user_id, tag) junction table — then this question is a trivial GROUP BY tag. Saying that unprompted is the over-the-bar finish.

Senior signal: describe the split-unnest pipeline, justify trim() in all three clauses (display, empty-token filter, and group key), explain that NULL yields zero unnested rows while the empty string yields one filtered-out token — then call out that storing CSV in a column is the underlying anti-pattern and a junction table is the fix.
