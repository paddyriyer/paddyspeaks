Podcasts are Spotify's growth lever. Completion rate per episode is the host-quality signal — high completion = engaging content, low = drop-off boredom. The query bands plays into completed/engaged/sampled/bounced.

2. The SQL — banded completion (95/50/10%)

WITH banded AS (
  SELECT episode_id,
    CASE
      WHEN listened_seconds * 1.0 / NULLIF(episode_seconds, 0) >= 0.95 THEN 'completed'
      WHEN listened_seconds * 1.0 / NULLIF(episode_seconds, 0) >= 0.50 THEN 'engaged'
      WHEN listened_seconds * 1.0 / NULLIF(episode_seconds, 0) >= 0.10 THEN 'sampled'
      ELSE 'bounced'
    END AS engagement_band
  FROM podcast_plays
)
SELECT episode_id, engagement_band, COUNT(*) AS plays
FROM banded GROUP BY episode_id, engagement_band
ORDER BY episode_id, plays DESC;

3. Strategic explanations

Why bands not raw % — actionability

Banded percentages map to host-feedback signals: "completed = great" / "engaged = strong middle / weak end" / "sampled = bad hook" / "bounced = thumbnail mismatch". Raw 0-100% values don't suggest interventions; bands do.

The intro-3-minutes metric

Podcasts live or die in the first 3 minutes. A separate metric: listened_seconds >= 180 = "passed the hook". 70%+ pass-rate is healthy; below 50% the intro needs reworking.

Length-bias caveat

Long podcasts (90+ min) have lower completion structurally. Comparing a 90-min interview to a 15-min daily news episode is unfair. Stratify by duration band.

The drop-off diagnostic

Per-episode aggregation hides WHERE in the episode users drop. The high-fidelity version aggregates per-second engagement curves across users — production has a separate "audio_heartbeats" table for this.

Senior signal: use 95/50/10 bands tied to host-feedback signals, surface the 3-min hook-pass rate, control for duration when comparing episodes, and acknowledge per-second drop-off curves as the higher-fidelity diagnostic.
