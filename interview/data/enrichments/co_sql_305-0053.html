TCS is running a retail-analytics engagement for a department-store chain, and the regional manager wants a single ranked list: every store and its total sales, biggest first. The query is four lines — but the interviewer planted a trap in the word "every". A store that opened last week and has not rung up a single sale still needs to appear on the report, and a naive join makes it vanish.

Four stores, but sales only mentions store_ids 1, 2 and 3. Store 4 — Outlet in Nagpur — has zero sales rows. That single missing store is the entire question.

2. The SQL — LEFT JOIN + SUM

SELECT s.name, SUM(sa.amount) AS total
FROM stores s LEFT JOIN sales sa ON s.store_id = sa.store_id
GROUP BY s.name ORDER BY total DESC;

The LEFT JOIN keeps every row of stores even when no sales row matches; GROUP BY name rolls each store into one line; ORDER BY total DESC ranks them.

Outlet survives the join — but its total is NULL, not 0, because SUM over zero rows returns NULL. And NULL sorts first under a plain DESC in Postgres, so the store with no sales floats to the top of a "biggest sellers" list. The query keeps the store but lies about its rank.

4. Strategic explanations

Why INNER JOIN would be silently wrong

Swap LEFT JOIN for a plain JOIN and Outlet disappears entirely — no matching sales row, no output row. The report would show three stores, the manager would never know a fourth exists, and nothing errors. This is the classic "missing rows" bug: an inner join answers "stores that have sales", not "every store". When the question says every, all, or including those with none, reach for an outer join.

The NULL total — coalesce it to zero

A store with no sales should report 0, not NULL. Wrap the aggregate:

SELECT s.name, COALESCE(SUM(sa.amount), 0) AS total
FROM stores s LEFT JOIN sales sa ON s.store_id = sa.store_id
GROUP BY s.name
ORDER BY total DESC;

COALESCE turns the empty-group NULL into 0, and now Outlet correctly sorts last. Alternatively keep the NULL but force its sort position with ORDER BY total DESC NULLS LAST — decide whether the consumer wants a number or a blank.

Group by the id, not the name

GROUP BY s.name works only while names are unique. Two stores both called "Outlet" in different cities would be merged into one inflated row. The robust form groups on the primary key and carries the name along:

SELECT s.store_id, s.name, COALESCE(SUM(sa.amount), 0) AS total
FROM stores s LEFT JOIN sales sa ON s.store_id = sa.store_id
GROUP BY s.store_id, s.name
ORDER BY total DESC;

Grouping on the key is correct regardless of name collisions, and Postgres lets you SELECT functionally-dependent columns like name once the key is in the GROUP BY.

Aggregate filters live in HAVING, not WHERE

If the follow-up is "only stores selling more than 600", that filter is on the aggregate and belongs in HAVING SUM(sa.amount) > 600. Putting amount > 600 in WHERE filters individual sale rows before grouping — a completely different, usually wrong, calculation.

The duplicate location columns

The schema ships both location and city. They are likely redundant — one a street address, one a city. The query needs neither, but if the manager wants per-city rollups, confirm which column is the analytic dimension before grouping on it.

Senior signal: reach for LEFT JOIN the moment the question says "every store", explain that an inner join would silently drop the no-sales store, COALESCE the empty-group NULL to 0 (and remember NULL sorts first under plain DESC), and group on the primary key rather than a name that might collide.
