An analytics team at a search company has a year-end summary table that is already aggregated: instead of one row per user, it stores "this many users each ran exactly this many searches." The task asks for the median number of searches across the real population those counts describe. The interviewer is probing one specific skill: can you compute a percentile over a weighted histogram — a frequency table — rather than over a raw event log?

1. The source data — search_frequency

searchesnum_users(narrative)
122 users ran 1 search each
222 users ran 2 searches each
333 users ran 3 searches each — the heaviest bucket
41single-user bucket — the edge case

The implied population is eight users, not four rows: [1, 1, 2, 2, 3, 3, 3, 4]. The (4, 1) bucket is the trap — a naive median over the four rows would average the values 2 and 3 and report 2.5 by luck, but for the wrong reason. The correct answer must weight every value by num_users.

2. The SQL — expand the histogram, then take the percentile

WITH expanded AS (
  SELECT s.searches
  FROM search_frequency s,
       generate_series(1, s.num_users)
)
SELECT ROUND(PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY searches)::numeric, 1) AS median
FROM expanded;

generate_series(1, s.num_users) is a lateral set-returning function: for a row with num_users = 3 it emits three rows, so the cross join multiplies that searches value into three physical rows. After the CTE, expanded holds the eight-row raw population, and PERCENTILE_CONT(0.5) computes the ordinary median over it.

3. The resulting analysis

median
2.5

With eight values the median is the mean of the 4th and 5th: in [1, 1, 2, 2, 3, 3, 3, 4] those are 2 and 3, so PERCENTILE_CONT interpolates to 2.5. The single-member (4, 1) bucket sits at the tail and never reaches the middle — but it still counts toward the population size of 8, which is what shifts the median position.

4. Strategic explanations

Why the histogram must be expanded

A percentile is defined over a population. The table is pre-aggregated, so each row is shorthand for num_users identical observations. Running PERCENTILE_CONT directly over the four rows answers a different question — "median of the four distinct search-counts" — which ignores that 3 searches is three times as common as 4. generate_series reconstitutes the raw population so the percentile sees the real distribution.

PERCENTILE_CONT vs PERCENTILE_DISC

PERCENTILE_CONT (continuous) interpolates between the two middle values — here 2 and 3 give 2.5, the textbook median for an even count. PERCENTILE_DISC (discrete) returns an actual value from the set, picking 2 (the value at or below the 0.5 cume). Both are "the median" by some definition; interviewers want you to name which one you used and why. For "the median," PERCENTILE_CONT is the conventional choice.

Performance — expansion is O(population), not O(rows)

generate_series materialises one row per user. If the histogram described 50 million users, the CTE produces 50 million rows and the sort behind WITHIN GROUP is over all of them — the compact histogram's size advantage is thrown away. At scale you would compute the median arithmetically from the cumulative num_users (a running SUM as a window) and find which bucket the (N+1)/2 position lands in — no expansion at all. For interview-sized data, expansion is clear and correct; mention the scaled alternative.

An empty or single-user table

If search_frequency were empty, expanded is empty and PERCENTILE_CONT returns NULL — not an error, but worth stating. A row like (4, 0) (zero users) contributes nothing because generate_series(1, 0) yields no rows. And a single-user population returns that user's value exactly, with no interpolation.

Dialect notes

PERCENTILE_CONT(...) WITHIN GROUP (ORDER BY ...) is standard SQL and works in PostgreSQL, Oracle, Snowflake, and BigQuery. MySQL has no ordered-set aggregate before 8.0 and still lacks PERCENTILE_CONT — you would emulate it with row numbering and an average of the middle one or two rows. SQL Server spells it PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY ...) OVER () as a window function, not a plain aggregate.

Senior signal: recognise a frequency table as a compressed population, expand it with generate_series before taking a percentile, distinguish PERCENTILE_CONT's interpolation from PERCENTILE_DISC, and note that at production scale you would compute the median from a cumulative count instead of materialising every user.
