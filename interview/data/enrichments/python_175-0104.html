Tesla's vehicle telemetry fleet streams a few billion sensor readings a day — battery cell voltages, motor temperatures, brake-pad wear, autopilot disengagement events. Each cell sends a reading every ~30 s with jitter; the fleet dashboard needs hourly summaries so the engineer on call sees one row per hour per metric per vehicle, not 120. The resample-and-aggregate pattern is how every IoT, observability, and high-frequency-finance company turns raw signal into oversight. Datadog rolls metrics into 10s, 1m, 5m, 1h buckets the same way; the algorithm is identical, just the cadences differ.

1. The data — irregular, jittered, occasionally spiky

The output schema (one row per window, columns count/mean/std/min/max/p50/p95/anomaly) is the Datadog metric-summary shape verbatim. The anomaly column is the on-call alert flag: a 3-sigma deviation from the trailing 24-window baseline.

2. The business question — what the on-call engineer decides on

  • Auto-throttle the affected cars. If hourly mean cell temperature jumps 15 °C, the vehicle firmware should reduce charge rate and notify the owner. The aggregation feeds the fleet-wide alert that triggers the OTA throttle command.
  • Capacity planning. p95 latency on autopilot stack tells SRE whether the inference cluster is sized for the next 6 months — feeds a quarterly procurement decision.
  • Warranty claims. The seasonal pattern (summer cars run hotter than winter cars) is normal; the per-hour deviation from that baseline is what triggers a warranty investigation. The trailing-24-window baseline implicitly handles the seasonality.

3. The pandas implementation — annotated

def windowed_aggregates(df, ts_col, value_col, freq='1h'):
    # ── Step 1. DatetimeIndex with monotonic ordering — required
    # for resample. .sort_index() costs O(N log N) but is non-negotiable;
    # resample on an unsorted index either errors or silently corrupts.
    s = df.set_index(pd.to_datetime(df[ts_col])).sort_index()
    r = s[value_col].resample(freq)

    # ── Step 2. Resample → DataFrameGroupBy-like object. We pull
    # each summary stat as its own column. Building the DataFrame
    # from a dict keeps the column order explicit (versus .agg([...])
    # which orders alphabetically post-pandas-2.0).
    agg = pd.DataFrame({
        'count': r.count(),
        'mean':  r.mean(),
        'std':   r.std(),
        'min':   r.min(),
        'max':   r.max(),
        'p50':   r.quantile(0.50),
        'p95':   r.quantile(0.95),
    })

    # ── Step 3. Z-score anomaly. Rolling(24) gives a 24-hour
    # trailing baseline — automatically deseasonalises for the
    # daily warming/cooling cycle. The first 24 windows have NaN
    # baseline (insufficient history) and get fillna(False).
    baseline_mean = agg['mean'].rolling(24).mean()
    baseline_std  = agg['mean'].rolling(24).std()
    agg['anomaly'] = (agg['mean'] - baseline_mean).abs() > 3 * baseline_std
    agg['anomaly'] = agg['anomaly'].fillna(False)
    return agg.reset_index()

Why resample('1h') and not groupby(df.index.floor('1h'))? Both produce the same buckets. resample is purpose-built: it handles the timestamp-arithmetic, knows about time-zone boundaries, and emits a clean DatetimeIndex on the result. The groupby(floor) form is a workaround when you can't get a DatetimeIndex — e.g. when grouping by a non-index column with groupby(pd.Grouper(key='ts', freq='1h')), which is what you'd use for "category × time" double aggregations (see python_175-0100).

Why z-score on mean and not on raw value? Anomalies in this domain are distributional shifts of the window, not single outlier readings. A spike at one timestamp is noise; a sustained shift across a window is the alarm. The mean is the right level of aggregation; the std gives you the noise floor against which to call "real shift". This is the canonical metric-anomaly pattern; Datadog Watchdog uses exactly this shape.

Critically, the seasonal warm-up at hour 5 does not trip the alarm — the rolling-24-window baseline already learned that 5 AM is warmer than midnight on this car. Only the unexpected spike fires. This is why metric anomaly detection rolls the baseline rather than comparing to a global mean.

5. Production patterns — how this scales

  • pandas → 10 M rows comfortable, 100 M painful. For one sensor on one car for a year (~10 M rows), pandas is fine. For the full Tesla fleet (~10⁹ rows/day), no. The production rewrite splits two ways:
    • Streaming live alerting: Flink with TumblingEventTimeWindows.of(Time.hours(1)) keyed by (vehicle_id, sensor_id). Watermark = 5 minutes for late events; events past the watermark go to a side-output for backfill. Same algorithm, unbounded data.
    • Batch backfill / historical analysis: Spark with window() + groupBy(window, vehicle_id). The pandas function in this question is the offline-twin / debugging tool — analysts replay yesterday's data with a different threshold and compare alert counts.
  • Datadog metric pipeline uses this exact shape for its 1m / 5m / 1h rollups. The 1m rollup runs on Lambda Architecture (Flink for live, Spark for backfill); the pandas pattern is the same algorithm. Cloudflare's request-rate monitoring is identical.
  • Hot key (single vehicle = 1 M readings/hour from an over-instrumented test car). The resample is fine in pandas (one big group). In Spark, the per-vehicle groupby would skew. Salt the vehicle_id with a salt-bucket for pass 1, roll up by vehicle_id in pass 2. See customer_growth-0027.
  • Watermark / late-arriving events. The pandas version sees all the data at once, so late events are a non-issue. In streaming, a reading that arrives 5 minutes after its window closed has to go somewhere — Flink's side-output pattern (route late events to a quarantine topic) is the production answer. Don't drop late events silently; known-known unknowns beat unknown unknowns.

6. Interviewer probes

Probe 1 — "How does resample('1h') handle a window with zero readings (sensor offline)?"

It emits a row with count=0, mean=NaN, all stats NaN. The downstream anomaly check then evaluates NaN > 3*NaN = False, so no false alert fires — but the NaN row is still in the output, which is exactly what you want: "the sensor was offline this hour" is information. Don't drop those rows. A senior candidate calls out the explicit None-handling here.

Probe 2 — "What's wrong with using resample('1h').apply(lambda x: x.quantile(0.95))?"

Two things. (a) .apply on a resample re-enters Python per group; .quantile(0.95) directly stays in C. (b) The lambda creates a function-call overhead per window. For 8760 windows (one year hourly), the apply form is ~50× slower. Use r.quantile(0.95) directly.

Probe 3 — "Tumbling vs sliding windows — when do you pick which?"

Tumbling (what we used): non-overlapping, each event belongs to exactly one window. Right for hourly summaries — the "5 PM hour" is well-defined. Sliding (e.g. "1h window, 10m stride"): overlapping, smooths out edge effects, gives 6× more rows. Right for "any 1h period with metric > X" alerts — catches spikes that straddle the tumbling-window boundary. The trade-off is row count: sliding produces 6× more rows for the same data, which can OOM the consumer dashboard.

Probe 4 — "Event-time vs processing-time — which does this use, and does it matter?"

The function uses the ts_col the caller passes in. If that's the event-time (timestamp from the sensor), the answer is deterministic and replay-safe. If it's the processing-time (when the row landed in the warehouse), the windows depend on ingestion latency — replay produces different results. For any audit-able use case, event-time is non-negotiable. The candidate who calls this out explicitly is showing senior signal — same theme as customer_growth-0010.

Probe 5 — "The 3-sigma threshold — is it tuned?"

Empirically, yes. A normal distribution puts 0.3 % of points beyond 3σ — for hourly windows over a year (~8760 hours per sensor), that's ~26 expected false positives. Across a 1000-sensor fleet, ~26 000/year ≈ 70/day. That's typically too many for on-call. Production raises the threshold to 4σ (~0.006 % = ~4 expected false positives/year/sensor) and requires the anomaly to persist for ≥ 2 consecutive windows. Tunable; document the trade-off.

Probe 6 — "Hot vehicle — a test car with 100k readings/hour. Does the resample OOM?"

In pandas, no — 100k rows per group is comfortable. The OOM risk is downstream (the dashboard showing one row per hour for 100 sensors over a year). In Spark, the resample-equivalent (groupBy(window, vehicle_id)) is fine because the aggregations (count/mean/std/min/max) are all associative and Spark's partial aggregation handles the skew. Quantiles (p50/p95) are the tricky case — not associative. Use approx_percentile (HLL/t-digest sketches) for the streaming version.

Probe 7 — "Why 24 windows for the baseline?"

It captures one full diurnal cycle. With hourly windows, 24 means "yesterday's same-hour mean is in the baseline" — so a 5 PM reading is compared to the last 24 5-PM-and-other-hour means, including yesterday's 5 PM. If you wanted same-hour-of-day baseline only, you'd use a shift(24)-based comparison instead of a rolling mean. Production at Datadog uses a weighted exponential moving average (EWMA) — gives more weight to recent windows while still smoothing.
