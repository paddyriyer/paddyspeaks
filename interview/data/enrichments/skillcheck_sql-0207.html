The two window expressions in this query differ by just three words — ORDER BY month_num — yet they return entirely different numbers. The trap is believing that ORDER BY inside an OVER() clause merely sorts. It does not: it installs a default frame, turning a whole-partition aggregate into a running one.

1. The source data — monthly_revenue

month_numrevenue
1100
2200
350
4150

2. The query — two SUMs that look alike

SELECT month_num, revenue,
       SUM(revenue) OVER (ORDER BY month_num) AS running_total,
       SUM(revenue) OVER ()                   AS period_total
FROM monthly_revenue
ORDER BY month_num;

Same aggregate, same column — the only difference is whether the OVER() clause carries an ORDER BY. That single change moves the frame, and the frame is what decides which rows the SUM sees.

3. The result

month_numrevenuerunning_totalperiod_total
1100100500
2200300500
350350500
4150500500

period_total is 500 on every row; running_total climbs 100, 300, 350, 500. Identical aggregate expression — the frame alone produces the difference.

4. Strategic explanations

OVER () — the whole partition

An empty OVER () has no ORDER BY, so its frame defaults to the entire partition: RANGE BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING. Every row sees every other row, so the result is the grand total, repeated on each row. This is the idiomatic way to put a static total beside detail rows — useful for percent-of-total: revenue * 100.0 / SUM(revenue) OVER ().

Adding ORDER BY changes the frame, not just the order

Once an ORDER BY appears in the window, the frame default becomes RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — from the start of the partition up to the current row. That is a running total. The ORDER BY did not sort the output (the outer query’s ORDER BY does that); it redefined which rows the aggregate accumulates.

The naive mistake

Reaching for SUM(revenue) OVER (ORDER BY month_num) as a period total returns a running total instead — month 1 would read 100, not 500. For a static grand total, the window must be frameless: OVER (). Conversely, an OVER () where a running figure was intended yields a flat grand total on every row.

Dialect notes

This frame-default behaviour is ANSI SQL and consistent across PostgreSQL, Snowflake, BigQuery, SQL Server, Oracle, and MySQL 8.0. Some engines reject an aggregate window without PARTITION BY or ORDER BY in older versions, but modern ones all accept the bare OVER () as a whole-partition frame.

Senior signal: OVER () aggregates the whole partition (a static total); adding ORDER BY to the window switches the default frame to UNBOUNDED PRECEDING .. CURRENT ROW, producing a running total. The ORDER BY inside OVER() is a frame directive, not a sort — use OVER () for grand totals and OVER (ORDER BY ...) for running ones, deliberately.
