Snowflake's onboarding team is normalising a user table and wants a quick worked example of the everyday string functions — uppercase the name, measure its length, grab the first letter. It is deliberately a warm-up, but an interviewer uses it to check whether you know the quiet inconsistencies between engines and how each function reacts to an empty string or a NULL.

1. The source data — users

user_idname
1alice
2Bob
3charlie
4Dee
5(empty string)

Names arrive in mixed case — some lowercase, some capitalised. User 5 has an empty string for a name (not NULL, but zero-length). That row is the edge case: it probes whether LENGTH and substr behave sanely on a string with nothing in it.

2. The SQL — UPPER / LENGTH / substr

SELECT UPPER(name)        AS upper_name,
       LENGTH(name)       AS len,
       substr(name, 1, 1) AS first_letter
FROM users;

Three scalar functions applied row by row. No aggregation — every input row produces one output row, the projection just transforms the name column three different ways.

User 5 returns cleanly: UPPER('') is '', LENGTH('') is 0, and substr('', 1, 1) is '' — no error. Notice first_letter preserves the original case ("a" stays lowercase) because substr slices before any case change is applied; upper_name is the only column that normalises case.

4. Strategic explanations

LENGTH vs CHAR_LENGTH vs OCTET_LENGTH

For plain ASCII these agree, but they diverge on multi-byte text. LENGTH/CHAR_LENGTH count characters; OCTET_LENGTH counts bytes. The name "Renee" with an accented e is 5 characters but 6 bytes in UTF-8:

SELECT LENGTH('Renée'), OCTET_LENGTH('Renée');  -- 5, 6

If the requirement is "display width" or "characters", use CHAR_LENGTH; OCTET_LENGTH only matters for storage sizing. Worth naming the distinction even on an easy question.

The NULL behaviour — empty string is not NULL

User 5 is an empty string and behaves predictably. A genuine NULL name behaves differently: UPPER(NULL), LENGTH(NULL) and substr(NULL,1,1) all return NULL, not '' or 0. If a dashboard needs a value, guard it: UPPER(COALESCE(name, '')) or COALESCE(LENGTH(name), 0). Conflating "empty" and "NULL" is the single most common string-handling mistake.

Dialect drift — the same idea, different spelling

This question is tagged Snowflake, and Snowflake accepts all three functions — but portable code should know the variants:

intentPostgresother engines
uppercaseUPPER(s)same everywhere
lengthLENGTH(s)LEN(s) in SQL Server
substringsubstr(s,1,1) or substring(s FROM 1 FOR 1)SUBSTRING, or LEFT(s,1)

For "first letter", LEFT(name, 1) reads more clearly than substr(name, 1, 1) and is widely supported — a small clarity win.

Edge cases with substr indexing

SQL substr is 1-indexed, unlike most programming languages — position 1 is the first character. Asking for more characters than exist is not an error: substr('Dee', 1, 99) simply returns "Dee". A start position past the end, or a non-positive start, yields an empty string rather than failing. That tolerance is why user 5 produced '' instead of an exception.

Senior signal: distinguish empty string from NULL and guard NULL with COALESCE, know that LENGTH counts characters while OCTET_LENGTH counts bytes, prefer the clearer LEFT(name,1) for a first character, and remember SQL string indexing is 1-based and out-of-range slices return empty rather than erroring.
