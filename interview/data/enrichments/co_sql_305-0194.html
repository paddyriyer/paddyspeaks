Twitter's growth team watches a single chart every morning: daily post engagement, smoothed into a 7-day rolling average so a viral spike or a quiet weekend doesn't make the line jump around. The raw daily number is noisy; the rolling average is what the exec deck shows. An interviewer asking this wants to see that you can aggregate and window in one query — and that you know what the window frame is actually measuring.

One post per day here, but that is an assumption — the table is keyed by post_id, not date. Day 5 has zero engagement (a dead post), and day 8 is the first day the window has a full 7 days behind it. Those two facts drive the result.

2. The SQL — GROUP BY then window

SELECT posted_date, SUM(likes+comments) AS engagement,
  AVG(SUM(likes+comments)) OVER (ORDER BY posted_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS rolling
FROM posts GROUP BY posted_date;

The inner SUM(likes+comments) collapses each day to one engagement number. The window function then averages that aggregate over the current row and the 6 rows before it — a 7-row sliding window. Nesting an aggregate inside a window function is legal precisely because the GROUP BY runs first.

Notice the ramp-up: on day 1 the rolling average equals the daily value (12), because the window has only one row to average. Day 3 averages three rows (12+25+6)/3 = 14.33. Only from day 7 onward does the window hold a full seven days. The early values are mathematically correct but not comparable to the later ones — a real dashboard often blanks the first six days.

4. Strategic explanations

ROWS vs RANGE — the frame that bites

ROWS BETWEEN 6 PRECEDING AND CURRENT ROW counts 7 physical rows. That equals 7 calendar days only if there is exactly one row per date with no gaps. If a day has no posts the row is simply missing, so the window silently reaches back to the 7th-most-recent row, which could be 10 calendar days ago. If you truly need 7 days regardless of gaps, switch to a range frame:

AVG(...) OVER (ORDER BY posted_date
  RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW)

The senior move is to say which one the metric needs and why — here, after the GROUP BY guarantees one row per date, ROWS is fine, but only because of that guarantee.

Why the aggregate can be nested in the window

SQL evaluates GROUP BY aggregation before window functions. So SUM(likes+comments) produces the per-day total, and AVG(...) OVER (...) sees those totals as its input rows. Writing AVG(SUM(...)) looks like a double aggregate but is not — the outer call is a window function. Try AVG(AVG(...)) without OVER and the engine rejects it.

The multiple-posts-per-day trap

If two posts share a date, GROUP BY posted_date sums them into one engagement figure — correct for a daily chart.

-- day with posts (10,2) and (5,3) -> engagement = 20, one row
posted_date | engagement | rolling
2026-01-01  | 20         | 20.000000

But if you forgot the GROUP BY and windowed over raw rows, each post would be its own window row and the "7-day" average would actually span far fewer than 7 days. Aggregate first, window second.

Edge cases: gaps, ties, and zero-engagement days

A zero day (day 5) correctly pulls the average down — that is real signal, not noise to discard. A genuinely missing day, by contrast, is invisible to a ROWS frame. If continuous dates matter, left-join against a generated date series so absent days appear as engagement 0. And SUM(likes+comments) returns NULL for that expression if either column is NULL on a row — coalesce the components (SUM(COALESCE(likes,0)+COALESCE(comments,0))) if the data is dirty.

Senior signal: aggregate to one row per day before applying the window, name ROWS vs RANGE as the difference between "7 rows" and "7 days", flag that the ramp-up period produces shorter (non-comparable) averages, and pre-empt the missing-date gap rather than assuming one row per date.
