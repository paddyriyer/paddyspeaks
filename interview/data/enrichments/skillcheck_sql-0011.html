A manufacturing-planning team needs the full parts explosion for one bicycle: starting from the root assembly "Bike", list every component required transitively, with the total quantity for a single unit. This is a recursive bill-of-materials — the interviewer is probing whether you can walk a multi-level tree and multiply quantities correctly down each path, not just collect part names.

1. The source data — bom

The edge case is multi-level multiplication. A Bearing is three levels down: Bike → Wheel(×2) → Hub(×1) → Bearing(×2). The per-bike quantity is the product of every qty on the path, 2×1×2 = 4 — a query that merely sums the leaf qty would wrongly report 2.

2. The SQL — recursive explosion with running multiplication

WITH RECURSIVE explode AS (
  SELECT child_part AS part, qty AS total_qty
  FROM bom
  WHERE parent_part = 'Bike'
  UNION ALL
  SELECT b.child_part, e.total_qty * b.qty
  FROM explode e
  JOIN bom b ON b.parent_part = e.part
)
SELECT part, SUM(total_qty) AS total_qty
FROM explode
GROUP BY part
ORDER BY part;

The anchor takes the direct children of Bike with their face quantities. The recursive member joins each discovered part to its own children and carries the running product e.total_qty * b.qty — so the quantity accumulates down the path. The final GROUP BY part with SUM folds together parts reached by more than one route.

Every quantity is the product down its path, not the raw edge qty. Spoke is 64 (two wheels × 32 spokes), Bearing is 4, Wheel appears both as a component (qty 2) and as a sub-assembly that gets exploded further. Summing leaf qty values alone would have produced 32 and 2 — visibly wrong.

4. Strategic explanations

Why the multiplication must accumulate, not reset

A bill-of-materials quantity is path-dependent: needing 2 Wheels, each needing 1 Hub, each needing 2 Bearings means 4 Bearings per bike. The recursive member encodes that by passing e.total_qty * b.qty — the parent's already-accumulated quantity times this edge's qty. Each level multiplies onto the running total. Carrying just b.qty (the bare edge quantity) would lose the upstream factors and undercount every deep part.

Why SUM at the end — the diamond case

If the same part were reachable through two different sub-assemblies — say both Frame and Wheel required a Bolt — the recursion would emit a separate explode row for each path, each with its own path-product. The outer GROUP BY part, SUM(total_qty) adds those contributions into one correct grand total per part. Skip the aggregation and a multi-parent part would appear several times with partial quantities. UNION ALL (not UNION) inside the CTE is essential here precisely so those distinct-path rows survive to be summed.

The cycle trap — a BOM must be acyclic

The recursion terminates only because the BOM is a DAG — eventually every path reaches a leaf with no children and the join returns nothing. If the data contained a cycle (part A requires part B which requires part A), the recursion would loop forever, multiplying quantities without bound. Real BOM schemas forbid this; a defensive query carries a visited-path array (WHERE b.child_part <> ALL(path)) or a depth cap, and PostgreSQL 14+ offers the CYCLE clause. Flagging this unprompted is the senior signal.

Edge cases: shared sub-assemblies and "where-used"

Hub shows a part that is simultaneously a component and a parent — the recursion handles it naturally, emitting a Hub row and exploding its children. The same engine answers the inverse "where-used" question (given a Bearing, which assemblies need it) by recursing on child_part = parent instead. And if the planner needs quantities for 50 bikes rather than 1, you multiply the final total_qty by 50 — or seed the anchor with qty * 50. Knowing the same recursion drives both directions and scales by order size is the depth an interviewer wants.

Senior signal: explode a BOM with a recursive CTE that carries a running product (parent_total * edge_qty) so quantities accumulate down each path, SUM by part at the end so a component reachable via multiple sub-assemblies is totalled correctly, use UNION ALL to preserve those per-path rows, and guard against cycles that would otherwise never terminate.
