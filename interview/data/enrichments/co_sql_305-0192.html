HR runs comp band reviews twice a year. The People-Analytics team needs a per-department salary leaderboard: who's the highest paid in each org, who's at risk of being underpaid relative to peers, and where pay-band overlaps create grievance risk. Different ranking functions answer different versions of these questions — picking the wrong one quietly corrupts the analysis.

1. The source data — employees

Notice E02 and E03 are tied at $180K — that's the interesting case the ranking function has to handle correctly.

2. The SQL — RANK() with PARTITION BY dept_id

SELECT emp_id, dept_id, salary,
  RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rk
FROM employees;

I included DENSE_RANK() and ROW_NUMBER() to make the ties explicit. RANK jumps from 2 to 4 (skipping 3) because two people are tied at position 2; DENSE_RANK doesn't skip; ROW_NUMBER picks an arbitrary tie-breaker.

4. Strategic explanations

RANK vs DENSE_RANK vs ROW_NUMBER — pick consciously
  • RANK — ties get the same number, then rank skips. Use when "how far behind first place are you?" matters. "3rd place finisher in a race" stays the 3rd best even if 2 people tied for 2nd.
  • DENSE_RANK — ties get the same number, no skip. Use for pay bands: HR cares about how many distinct salary tiers exist, not how many people sit at each.
  • ROW_NUMBER — ties broken arbitrarily by the engine. Use only when you need exactly one row per partition; never trust the tie-break unless you add a deterministic secondary ORDER BY.
The "top earner per dept" question — wrap in a subquery

The query as written returns every employee with their rank. To filter to just the top earners, wrap it:

SELECT * FROM (
  SELECT emp_id, dept_id, salary,
    RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rk
  FROM employees
) t WHERE rk = 1;
-- Returns 3 rows: E01 (ENG #1), E02 and E03 (both tied for ENG #1 ... wait, no)
-- Actually returns just E01 and E05 because E02/E03 are at rank 2.

If the goal is "all employees tied for first place" and ties should both qualify, switch to RANK() = 1 over a re-ordered partition or use DENSE_RANK. State the tie semantics in the interview.

The Snowflake / BigQuery shortcut: QUALIFY
-- Snowflake / BigQuery — collapses two CTEs into one
SELECT emp_id, dept_id, salary,
  RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rk
FROM employees
QUALIFY rk = 1;

Mention QUALIFY in the interview as the dialect-specific shortcut, but lead with the portable subquery — Postgres and SQLite don't have it.

What HR actually does next: pay-band overlap detection

The grievance-risk question is: "Is anyone in dept A earning less than the lowest-paid person at the same level in dept B?" That's a JOIN on dept-level aggregates, not a single ranking — but the ranking output above is the input. Once the rank column exists, HR can compare WHERE rk = N across departments to find pay parity issues.

5. Interviewer follow-up probes — the edge cases that separate seniors from juniors

Whatever rank-and-filter solution you write, the interviewer will start poking at it. These are the four probes you should preempt before being asked:

Probe 1 — "What if a department has only one employee? Will your query still return them as 'second-highest'?"

The trap: WHERE rk = 2 returns zero rows for that department. The hiring manager opens the report and sees "no second-highest in Marketing" without realising Marketing has only one person. Three defensive options — note the CTE is named and aliased properly so the query actually runs:

-- Option A: explicit COUNT-aware report — surfaces the "N/A" case
WITH ranked AS (
  SELECT e.dept_id, e.salary,
    DENSE_RANK() OVER (PARTITION BY e.dept_id ORDER BY e.salary DESC) AS rk
  FROM employees e
)
SELECT r.dept_id,
  COUNT(*)                                    AS headcount,
  MAX(CASE WHEN r.rk = 1 THEN r.salary END)   AS top_salary,
  MAX(CASE WHEN r.rk = 2 THEN r.salary END)   AS second_salary  -- NULL if headcount < 2
FROM ranked r
GROUP BY r.dept_id;

-- Option B: LEFT JOIN dept list to ranked so 1-person depts still appear
WITH ranked AS (
  SELECT e.dept_id, e.salary,
    DENSE_RANK() OVER (PARTITION BY e.dept_id ORDER BY e.salary DESC) AS rk
  FROM employees e
)
SELECT d.dept_id, r.salary AS second_highest
FROM departments d
LEFT JOIN ranked r
  ON r.dept_id = d.dept_id AND r.rk = 2;

-- ⚠ Option B caveat: when two people are tied at the second-highest
-- salary, this LEFT JOIN returns ONE ROW PER TIED EMPLOYEE (the join
-- multiplies). If you want exactly one row per department, fold the
-- tied salary up with conditional aggregation — that's Probe 1b
-- below, which uses GROUP BY + MAX(CASE WHEN rk = 2 THEN salary END).
Probe 1b — "Now list each department with BOTH the highest AND the second-highest salary side by side."

One row per department, two salary columns. The right shape pivots the ranked output via conditional aggregation, then LEFT JOINs from departments so single-person and empty departments still appear (with NULL in the second column):

WITH ranked AS (
  SELECT e.dept_id, e.salary,
    DENSE_RANK() OVER (PARTITION BY e.dept_id ORDER BY e.salary DESC) AS rk
  FROM employees e
)
SELECT
  d.dept_id,
  d.department_name,
  MAX(CASE WHEN r.rk = 1 THEN r.salary END) AS highest_salary,
  MAX(CASE WHEN r.rk = 2 THEN r.salary END) AS second_highest_salary
FROM departments d
LEFT JOIN ranked r ON r.dept_id = d.dept_id
GROUP BY d.dept_id, d.department_name
ORDER BY d.dept_id;

Two senior-signal moves baked in: (1) LEFT JOIN from departments guarantees every department appears even with 0 or 1 employee — no silent disappearance; (2) the conditional MAX(CASE WHEN rk = N THEN salary END) pivot avoids two separate queries with a UNION. Note we use DENSE_RANK so that if two people are tied at the top, the "second-highest" still surfaces the next distinct salary (not NULL).

Probe 1c — "Now ROTATE it: salaries as rows, departments as columns."

Same data, transposed. The output has two rows ("1st Highest", "2nd Highest") and one column per department. This is what a finance / HR dashboard wants — a tiny compact pivot they can paste into a slide. Two cleanly-equivalent shapes:

-- Shape A: dimension-table JOIN form (normalised data — production grade)
WITH ranked AS (
  SELECT d.department_name, e.salary,
    DENSE_RANK() OVER (PARTITION BY e.dept_id ORDER BY e.salary DESC) AS rk
  FROM employees e
  JOIN departments d ON d.dept_id = e.dept_id
)
SELECT
  CASE WHEN rk = 1 THEN '1st Highest'
       WHEN rk = 2 THEN '2nd Highest' END                            AS salary_position,
  MAX(CASE WHEN department_name = 'Engineering' THEN salary END)     AS engineering,
  MAX(CASE WHEN department_name = 'Sales'       THEN salary END)     AS sales,
  MAX(CASE WHEN department_name = 'Marketing'   THEN salary END)     AS marketing,
  MAX(CASE WHEN department_name = 'HR'          THEN salary END)     AS hr,
  MAX(CASE WHEN department_name IS NULL         THEN salary END)     AS unmapped_dept
FROM ranked
WHERE rk IN (1, 2)
GROUP BY rk
ORDER BY rk;

-- Shape B: FILTER clause (PostgreSQL / SQLite / BigQuery — cleaner syntax)
WITH ranked AS (
  SELECT d.department_name, e.salary,
    DENSE_RANK() OVER (PARTITION BY e.dept_id ORDER BY e.salary DESC) AS rk
  FROM employees e
  JOIN departments d ON d.dept_id = e.dept_id
)
SELECT
  CASE WHEN rk = 1 THEN '1st Highest'
       WHEN rk = 2 THEN '2nd Highest' END                AS salary_position,
  MAX(salary) FILTER (WHERE department_name = 'Engineering') AS engineering,
  MAX(salary) FILTER (WHERE department_name = 'Sales')       AS sales,
  MAX(salary) FILTER (WHERE department_name = 'Marketing')   AS marketing,
  MAX(salary) FILTER (WHERE department_name = 'HR')          AS hr,
  MAX(salary) FILTER (WHERE department_name IS NULL)         AS unmapped_dept
FROM ranked
WHERE rk IN (1, 2)
GROUP BY rk
ORDER BY rk;

Sample output (your dashboard tile):

salary_positionengineeringsalesmarketinghrunmapped_dept
1st Highest$185,000$140,000$135,000$95,000NULL
2nd Highest$170,000$125,000$110,000$80,000NULL

The how-it-works in three steps:

  1. Rank inside each department — DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) stamps every employee with their position within their own dept. Ties get the same rank (5 people tied at the top all contribute to "1st Highest").
  2. Rotate row values into columns — SQL can't pivot natively. The trick: one MAX(CASE WHEN dept = 'X' THEN salary END) column per department. MAX ignores NULLs, so each column picks up only the rows whose dept matches.
  3. Collapse to two rows — GROUP BY rk with WHERE rk IN (1, 2) produces one row per rank tier. Result: two rows, N+1 columns (N depts + the unmapped catch-all).

FILTER vs CASE WHEN: they produce identical results. FILTER (WHERE …) is ANSI-standard and clearer; CASE WHEN … THEN … END works on every engine including MySQL < 8.0 and SQL Server. Use FILTER where supported; the CASE form is the portable fallback.

Probe 1d — "What about the LAD catch-all column?"

If a high earner's dept_id is still propagating (late-arriving dimension), the JOIN to departments produces department_name = NULL for them. Without an explicit catch-all column they vanish from every named-department column AND don't appear anywhere — high earners silently dropped from the finance report. The unmapped_dept column above is the defensive pattern: MAX(salary) FILTER (WHERE department_name IS NULL). Surface them, then the data-quality team can chase the missing mapping. This is the L5 / senior-IC move the interviewer is testing for.

Probe 1e — "What if you don't know the department names upfront — generate columns dynamically?"

SQL is static — columns must be known at parse time. Three answers, none of them "just SQL":

  • PostgreSQL crosstab() (tablefunc extension) — accepts a row-based query and a column-name query; returns a wide result. Requires installing the extension.
  • Snowflake / Databricks PIVOT — native syntax: PIVOT (MAX(salary) FOR department_name IN ('Engineering', 'Sales', ...)). Still requires listing values, but the engine generates the CASE statements.
  • Two queries in the app layer — query distinct department names first; build the SQL string with each name as a column; run the second query. The pattern most BI tools use under the hood.
Probe 2 — "Two people are tied for second. Do you want both, or just one?"

This is the question the user always intends to ask but rarely does — and the SQL answer depends entirely on which ranking function you picked:

Probe 3 — "What if there's a tie at the top? Does your 'second-highest' return the next distinct salary, or somebody at the top salary?"

Consider {$320K, $320K, $280K, $240K}. With RANK: ranks are 1, 1, 3, 4 → rk = 2 returns NOTHING. With DENSE_RANK: ranks are 1, 1, 2, 3 → rk = 2 returns the $280K row. The user almost always means the latter. DENSE_RANK is the safe default for Nth-highest.

Probe 4 — "Now generalise it to Nth-highest. Does your approach work for N = 17?"

The cute one-liner approaches (correlated subquery, OFFSET tricks) all break or get slow at large N. The window-function approach scales O(n log n) regardless of N:

-- Works for any N, no rewrite needed:
SELECT salary FROM (
  SELECT salary, DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rk
  FROM employees
) t WHERE rk = :n;

If the interviewer asks for "the top 3 in each dept", change rk = :n to rk <= 3 — same query shape.

Senior signal: show all three ranking functions in one breath, name the tie-handling difference, mention QUALIFY, and preempt the four edge-case probes: empty-department case, "all ties or just one", tie-at-top semantics, and the generalisation to Nth. The interviewer is checking whether you've internalised the why of each function, not just the syntax of OVER.
