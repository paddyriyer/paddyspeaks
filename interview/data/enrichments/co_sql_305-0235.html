An on-call engineer is paged at 2am: "the daily ETL pipeline only loaded 4.3M rows last night instead of the usual 5M." The first question is always: "are we missing log_ids?" A row-by-row check is impossible at that scale — the canonical SQL answer is gap-and-island detection with LAG. Same pattern flags missing invoice numbers, missing transaction sequences, missing event IDs in any append-only stream.

1. The source data — logs

An ingest pipeline writes monotonic log_ids. In healthy operation, IDs are dense (1, 2, 3, 4...). When ingest is incomplete, gaps appear.

3OK

2. The SQL — LAG + arithmetic difference

WITH ids AS (
  SELECT log_id,
         LAG(log_id) OVER (ORDER BY log_id) AS prev
  FROM logs
)
SELECT
  prev + 1     AS gap_start,    -- first missing id
  log_id - 1   AS gap_end       -- last missing id
FROM ids
WHERE log_id - prev > 1;        -- non-adjacent rows = a gap

4. Strategic explanations

Why LAG beats a self-join here

The naïve solution is SELECT a.log_id+1 FROM logs a LEFT JOIN logs b ON b.log_id = a.log_id+1 WHERE b.log_id IS NULL. That works but does an O(n²) self-join in the worst case. LAG is a single sorted pass — same answer, far cheaper. On a million-row log table, the self-join takes minutes; LAG takes seconds.

The first row gotcha — LAG returns NULL

For the very first row in the partition, LAG returns NULL. Our WHERE log_id - prev > 1 filter excludes that row automatically (NULL arithmetic = NULL = filtered out). But if the first expected ID is 1 and the actual first row is 5, the gap from 1-4 is invisible. Fix: anchor with a known boundary:

WITH ids AS (
  SELECT log_id,
    COALESCE(
      LAG(log_id) OVER (ORDER BY log_id),
      0                              -- assume gap starts at id=1
    ) AS prev
  FROM logs
)
SELECT prev + 1 AS gap_start, log_id - 1 AS gap_end
FROM ids
WHERE log_id - prev > 1;
The "missing range" pattern is a gap-and-islands cousin

Same shape works on dates ("which days had no orders?"), transaction IDs ("which invoice numbers were skipped — possible voided invoices?"), and time-bucketed metrics ("which 5-minute windows had zero pings — outage?"). The function family is always: LAG for the previous expected anchor, arithmetic difference for the gap test, optional date-spine for buckets that should always exist.

The follow-up: detect gaps within a partition

Real systems have multiple log streams (one log_id sequence per service). The gap detection has to run per-service:

WITH ids AS (
  SELECT service_name, log_id,
    LAG(log_id) OVER (PARTITION BY service_name ORDER BY log_id) AS prev
  FROM logs
)
SELECT service_name, prev + 1 AS gap_start, log_id - 1 AS gap_end
FROM ids
WHERE log_id - prev > 1;
Why this query goes in the on-call playbook

Gap detection is the first triage step for any append-only data quality issue. Senior data engineers keep a parameterised version of this query in their on-call playbook — point it at any sequence column, get the missing ranges in seconds.
