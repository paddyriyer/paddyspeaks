A growth team wants to identify customers who convert to their second purchase faster than the typical buyer — a signal of high intent. The interviewer is probing three distinct skills in sequence: ROW_NUMBER to isolate ordinal purchases, a self-join to pair first and second, and PERCENTILE_CONT to compute a statistical median for filtering.

2. The SQL — ROW_NUMBER, self-join, PERCENTILE_CONT

WITH numbered AS (
  SELECT customer_id, purchased_at,
         ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY purchased_at) AS rn
  FROM purchase
),
paired AS (
  SELECT n1.customer_id,
         n1.purchased_at AS first_at,
         n2.purchased_at AS second_at,
         FLOOR(EXTRACT(EPOCH FROM (n2.purchased_at - n1.purchased_at)) / 86400)::int AS latency_days
  FROM numbered n1
  JOIN numbered n2
    ON n1.customer_id = n2.customer_id
    AND n1.rn = 1 AND n2.rn = 2
),
m AS (
  SELECT PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY latency_days) AS med FROM paired
)
SELECT customer_id, first_at, second_at, latency_days
FROM paired, m
WHERE latency_days < m.med
ORDER BY customer_id;

4. Strategic explanations

ROW_NUMBER for ordinal purchase isolation

ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY purchased_at) assigns 1 to the first purchase and 2 to the second for each customer, without any ambiguity. The self-join n1.rn = 1 AND n2.rn = 2 then cleanly extracts the first and second timestamps per customer. An alternative using LAG or LEAD would work but produces a result table with one row per purchase rather than one row per customer, requiring an extra filter step.

FLOOR(EXTRACT(EPOCH ...) / 86400) for truncated-day latency

Subtracting two PostgreSQL timestamps produces an interval. EXTRACT(EPOCH FROM interval) converts it to total seconds as a float. Dividing by 86400 (seconds per day) and applying FLOOR gives integer days truncated toward zero — exactly what "truncated toward zero" means for positive durations. An alternative is DATE_PART('day', second_at - first_at) in PostgreSQL, which extracts only the day component of the interval without accumulating hours; for timestamps at the same time of day this produces the same result, but for cross-midnight timestamps EPOCH/86400 is more accurate.

PERCENTILE_CONT vs PERCENTILE_DISC

PERCENTILE_CONT(0.5) returns a linearly interpolated value (10.5 in this example), which may not be an integer. PERCENTILE_DISC(0.5) returns the nearest actual data point (7 in this example). The question asks for "median latency" without specifying interpolation, but PERCENTILE_CONT is the statistical standard. Noting both options and their difference is a strong interview signal. The strict < (not ≤) in the filter means that customers exactly at the median are excluded; if you used PERCENTILE_DISC (returning 7), customer 1 would be excluded too.

The cross join to m

The comma in FROM paired, m is a CROSS JOIN — valid because m is guaranteed to return exactly one row. An explicit CROSS JOIN keyword is clearer. An alternative is a scalar subquery in the WHERE clause: WHERE latency_days < (SELECT med FROM m); that expresses the same logic but prevents the CTE from being referenced twice in complex plans.

Dialect notes

PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY ...) is ANSI SQL:2003 and is available in PostgreSQL, Snowflake, BigQuery, and SQL Server (as PERCENTILE_CONT inside OVER or as an aggregate). MySQL 8+ provides PERCENTILE_CONT as a window function only, not as an aggregate; computing a single-value median requires a workaround. SQLite does not support it natively.

Senior signal: use ROW_NUMBER + a self-join on rn=1/rn=2 to cleanly extract first-and-second-purchase pairs; prefer EPOCH/86400 + FLOOR over interval day-extraction for cross-midnight accuracy; and distinguish PERCENTILE_CONT (interpolated) from PERCENTILE_DISC (nearest data point) — the choice changes which customers fall exactly on the median boundary.
