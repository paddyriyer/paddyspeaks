LinkedIn member experience: each position has a started event when added, and an ended event when the member marks it done. Periods can overlap — current full-time job + part-time consulting at the same time. Position deduplication has a twist: the same (member, company) pair can have multiple distinct started/ended events if the member rejoined the same company.

2. The SQL — partition by (member, company) for per-tenure dedup

WITH paired AS (
  SELECT member_id, company_id, event_type, event_at,
    LAG(event_type) OVER (PARTITION BY member_id, company_id ORDER BY event_at) AS prev_event
  FROM member_company_events
),
filtered AS (
  SELECT member_id, company_id, event_type, event_at
  FROM paired
  WHERE event_type != prev_event OR prev_event IS NULL
)
SELECT member_id, company_id,
  MIN(CASE WHEN event_type = 'started'  THEN event_at END) AS started_at,
  MAX(CASE WHEN event_type = 'ended'    THEN event_at END) AS ended_at
FROM filtered
GROUP BY member_id, company_id
ORDER BY member_id, started_at;

3. Strategic explanations

Composite partition key

Most status-period queries partition by a single entity (account_id, order_id). LinkedIn position-tenure is per (member_id, company_id) — multiple companies per member, each with its own lifecycle. The composite partition key keeps each company's events siloed for proper dedup.

The "current position" semantic

A position with ended_at IS NULL is currently active. M_001's role at C_020 and C_030 are both current — the member holds both simultaneously (full-time + advising). LinkedIn's "Current Position" field on a profile is just WHERE ended_at IS NULL.

The conditional-aggregation idiom

MIN(CASE WHEN event_type = 'started' THEN event_at END) picks only started rows; MAX(... 'ended' ...) picks only ended rows. NULL-on-no-match flows through naturally. Cleaner than two separate filtered subqueries.

Re-joined company — the rejoiner pattern

What if M_001 was at C_010 from 2018-2020, left, and rejoined in 2024? Two distinct tenures. The query above collapses them into one row (started 2018, ended NULL). To preserve separate tenures, add a tenure_id by counting started events:

SUM(CASE WHEN event_type = 'started' THEN 1 ELSE 0 END)
  OVER (PARTITION BY member_id, company_id ORDER BY event_at) AS tenure_id
Why no LEAD here

Standard period queries use LAG for change-detection + LEAD for end_date. Here the schema already encodes "started" and "ended" explicitly — no need to compute end from a future row. The conditional MIN/MAX accomplishes the same thing without a window function. Cleaner when the lifecycle is binary-event-based.

The graph-of-employment use case

The output of this query feeds LinkedIn's "your colleagues" suggestions. Two members who overlapped at the same company in the same period are candidates for the recommendation. Self-join on overlapping (started_at, ended_at) intervals — the same overlap-detection pattern from the bookings / shifts questions.

Senior signal: composite (member, company) partition key, NULL ended_at as "current", conditional MIN/MAX for binary-event lifecycles (cleaner than LAG+LEAD), tenure_id via SUM-of-started-flag for rejoiner detection, and connect the output to overlap-based "colleagues" recommendations.
