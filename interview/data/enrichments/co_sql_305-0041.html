Elastic's growth team tracks one number above all others: how many new customers signed up each month. The query is a classic DATE_TRUNC grouping — but the interviewer asking this is probing whether you know why you must group by the truncated value rather than the raw timestamp, and whether you spot that months with zero signups will silently vanish from the result.

1. The source data — customers

Six customers spread over three months — two in January, three in February, one in March. Every signup_date is a distinct day; the whole point of the query is to bucket distinct days into shared months.

2. The SQL — DATE_TRUNC('month', ...)

SELECT DATE_TRUNC('month', signup_date)::DATE AS month, COUNT(*) AS signups
FROM customers GROUP BY DATE_TRUNC('month', signup_date)::DATE ORDER BY month;

DATE_TRUNC('month', d) snaps any date to the first day of its month, so 2026-01-05 and 2026-01-20 both become 2026-01-01 and collapse into one group. The ::DATE cast drops the time component that DATE_TRUNC returns.

3. The resulting analysis

Three rows, one per month with activity. January's two distinct signup days collapsed into a single 2026-01-01 bucket; February's three did the same. The month label is a real date (the first of the month), which sorts and filters cleanly — better than a text label like "2026-01".

4. Strategic explanations

Why group by the truncated value, not signup_date

A frequent mistake is GROUP BY signup_date with DATE_TRUNC only in the SELECT. That groups by the raw day, producing six one-row groups, and most engines reject it because the selected expression isn't functionally dependent on the grouping key. The grouping key and the projected month must be the same expression. Postgres also allows the shorthand GROUP BY 1 (group by the first select item).

Missing months — the gap the interviewer is waiting for

If no customer signs up in, say, April, there is no April row at all — GROUP BY only emits buckets that have data. A trend chart with a silently-absent month is misleading. Generate the calendar and LEFT JOIN the counts onto it:

SELECT m.month, COUNT(c.customer_id) AS signups
FROM generate_series(DATE '2026-01-01', DATE '2026-04-01', INTERVAL '1 month') AS m(month)
LEFT JOIN customers c
  ON DATE_TRUNC('month', c.signup_date) = m.month
GROUP BY m.month ORDER BY m.month;

Note COUNT(c.customer_id), not COUNT(*): a month with no signups still produces one join row with NULLs, and COUNT(*) would wrongly report 1 for it.

Time zones — where signup counts actually go wrong

If signup_date is really a TIMESTAMPTZ, DATE_TRUNC uses the session time zone, so a sign-up at 23:30 UTC on Jan 31 may land in January or February depending on the viewer. For a stable report, truncate in an explicit zone: DATE_TRUNC('month', signup_date AT TIME ZONE 'UTC').

Cumulative growth — the natural follow-up

Growth teams rarely stop at per-month counts; they want the running total. Layer a window function over the grouped result:

SELECT month, signups,
  SUM(signups) OVER (ORDER BY month) AS cumulative_signups
FROM (
  SELECT DATE_TRUNC('month', signup_date)::DATE AS month, COUNT(*) AS signups
  FROM customers GROUP BY 1
) t ORDER BY month;
Senior signal: group by the same DATE_TRUNC expression you project, return a real date label rather than text, pre-empt the missing-months gap with a generate_series calendar plus LEFT JOIN (and COUNT(column)), and raise the time-zone caveat for TIMESTAMPTZ sign-up data.
