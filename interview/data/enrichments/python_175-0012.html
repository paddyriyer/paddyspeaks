The trie is a Google staple because it is the data structure behind autocomplete, spell-check, and IP routing tables — anywhere you need fast prefix queries. The interview question tests one specific insight: the difference between "is this an exact word?" and "is this a prefix of some word?" Candidates who conflate the two write a trie that cannot tell app from apple.

1. The input data

A sequence of operations on a fresh Trie:

t = Trie()
t.insert('apple')
t.search('apple')      # exact word?
t.search('app')        # exact word?
t.startsWith('app')    # any word with this prefix?
t.insert('app')
t.search('app')        # exact word now?

2. The approach

The trie is a tree of nested dicts rooted at self.r; each level maps one character to a child node. insert walks the word, using n.setdefault(c, {}) to descend into an existing child or create one on the fly, then stores a sentinel key '#': True at the final node to mark a complete word. The private helper _walk(w) follows the path for w character by character, returning the node it lands on, or None the moment a character has no child. Both public queries are thin wrappers over _walk: search returns True only if _walk reaches a node and that node carries the '#' sentinel; startsWith returns True if _walk simply reaches any node at all. That single sentinel check is the entire distinction between the two operations.

The two search('app') calls bracket the lesson: same path both times, but the answer flips once a word actually terminates there.

4. Strategic explanations

Complexity

insert, search, and startsWith are each O(L) in the length of the input string — crucially independent of how many words the trie holds. That is the trie's headline advantage over a hash set, which is O(L) to hash but offers no prefix query at all. Space is O(total characters inserted), reduced by shared prefixes — apple and app share their first three nodes.

The idiomatic alternative

For exact-membership only, a set of strings is simpler and faster to write. The trie earns its complexity exclusively when you need startsWith — prefix enumeration, autocomplete, longest-common-prefix. A more conventional implementation uses a small TrieNode class with a children dict and an explicit is_end boolean, which reads more clearly than smuggling '#' into the children namespace.

The edge case — empty string and the sentinel as a key

insert('') puts '#' at the root, so search('') returns True and startsWith('') always returns True (the root always exists) — reasonable, but state it. The subtle hazard of the plain-dict design: a word containing the literal character '#' would collide with the sentinel. A dedicated boolean field sidesteps that entirely.

The follow-up — delete, and autocomplete

"Add a delete(word)." Deletion is the tricky operation: clear the '#' at the terminal node, then walk back up pruning nodes that have become childless — but stop the moment a node is still part of another word or marks a shorter word. The other classic follow-up, "return all words with prefix P," is a DFS from the node _walk(P) returns, collecting every path that ends in '#'.

The trap — no end-of-word marker

The most common failure is omitting '#' and making search just check that the path exists. That implementation reports search('app') as True after only apple was inserted — it has silently turned search into startsWith. The sentinel is not optional.

Senior signal: a senior nails the word-vs-prefix distinction with an explicit terminal marker, notes that O(L) operations are independent of dictionary size, and reaches for a trie only when prefix queries are actually required — a junior drops the end marker and conflates search with startsWith.
