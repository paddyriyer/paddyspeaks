Anthropic runs two products on separate billing systems, each with its own customer table. The growth team wants a single deduplicated list of every customer across both. It sounds like a one-line UNION — and it is — but the interviewer is checking whether you know the difference between UNION and UNION ALL, and what "duplicate" actually means when two tables disagree.

1. The source data — service_a_customers & service_b_customers

tablecustomer_idname
service_a1Ana
service_a2Ben
service_a3Cara
service_b2Ben
service_b3Cara
service_b4Dan

Customers 2 and 3 use both products — they appear in each table with identical (customer_id, name). Those overlaps are what UNION has to collapse.

2. The SQL — UNION

SELECT customer_id, name FROM service_a_customers
UNION
SELECT customer_id, name FROM service_b_customers;

UNION stacks the two result sets and then removes duplicate rows — a row is a duplicate only when every selected column matches. The two queries must have the same column count and compatible types, which they do.

3. The resulting analysis

Four distinct customers from six input rows. Ben and Cara appeared twice but surface once each. Note there is no ORDER BY — the rows above happen to come back sorted, but that ordering is incidental, decided by how the engine deduplicated, not guaranteed.

4. Strategic explanations

UNION vs UNION ALL — the question behind the question

UNION deduplicates; UNION ALL does not. With UNION ALL the result would have six rows — Ben and Cara listed twice. The deduplication is not free: it forces the engine to sort or hash the entire combined set. If you know the inputs are disjoint, UNION ALL is strictly faster:

SELECT customer_id, name FROM service_a_customers
UNION ALL
SELECT customer_id, name FROM service_b_customers;

The senior reflex: default to UNION ALL and add the DISTINCT-cost of plain UNION only when overlap is actually possible. Here it is, so UNION is right — but say why.

The dedup is by whole row — the dangerous edge

UNION compares the full row, not just customer_id. Suppose service B has customer 2 as (2, 'Benjamin') — same person, name spelled differently. UNION sees two distinct rows and keeps both, and now customer 2 appears twice. If customer_id is the true identity key, deduplicate on that, not the row:

SELECT DISTINCT ON (customer_id) customer_id, name
FROM (
  SELECT customer_id, name FROM service_a_customers
  UNION ALL
  SELECT customer_id, name FROM service_b_customers
) all_c
ORDER BY customer_id;

This is the trap: "deduplicate customers" and "deduplicate rows" are different requests, and conflicting name values across systems is exactly when they diverge.

How NULLs are treated in UNION

Unlike =, UNION's duplicate-detection treats two NULLs as equal. So (5, NULL) from each table collapses to a single row — consistent with the dedup of any other repeated value. Worth stating, because it contradicts the usual "NULL is never equal to anything" rule that governs WHERE and JOIN.

Column names, types, and ordering

The output column names come from the first SELECT; the second query's aliases are ignored. The two branches must be union-compatible — same arity, coercible types — or the query errors. And a single ORDER BY at the very end orders the combined result; you cannot put ORDER BY on an individual branch. If the consumer needs a stable list, append ORDER BY customer_id — never rely on the incidental ordering shown above.

Senior signal: distinguish UNION from UNION ALL by the cost of deduplication and default to ALL unless overlap is real, recognise that row-level dedup breaks when the same customer has a different name across systems (use DISTINCT ON the identity key), and add an explicit ORDER BY rather than trusting incidental ordering.
