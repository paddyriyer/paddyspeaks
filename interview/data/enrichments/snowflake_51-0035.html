Snowflake’s internal commerce team wants a daily revenue breakdown by product category — a three-table join that connects orders to their line items and then to the product catalogue, groups by day and category, and ranks categories within each day by revenue. The query tests clean multi-table join navigation, DATE() truncation of a timestamp, and the interaction between GROUP BY and computed expressions.

2. The SQL — three-table join, date truncation, SUM(qty × price)

SELECT DATE(o.order_ts) AS day, p.category,
       SUM(oi.qty * oi.price) AS revenue
FROM orders       o
JOIN order_items  oi ON o.order_id    = oi.order_id
JOIN products     p  ON oi.product_id = p.product_id
GROUP BY DATE(o.order_ts), p.category
ORDER BY day, revenue DESC;

4. Strategic explanations

The three-table join chain

The join pattern follows the natural foreign-key chain: orders → order_items (one order to many items) → products (many items to one product). Both joins use INNER JOIN implicitly, dropping any order with no line items (orphaned orders) and any line item with no matching product (referential-integrity violation). If either scenario can occur in practice, consider LEFT JOIN and account for NULL product categories.

Computing revenue at the line-item level: qty × price

SUM(oi.qty * oi.price) computes the line-item total first, then sums across all line items for that (day, category) group. This is correct because each line item has its own price (which may differ from the product’s catalogue price if discounts were applied at the order level). Avoid SUM(oi.qty) * SUM(oi.price) — that would multiply aggregate totals, producing wildly wrong results when there are multiple line items per group.

DATE() vs TRUNC vs CAST in GROUP BY

DATE(o.order_ts) extracts the calendar date from a timestamp. Alternatives by engine:

  • Snowflake: DATE(order_ts), order_ts::DATE, or DATE_TRUNC('day', order_ts)
  • Postgres: DATE_TRUNC('day', order_ts) (returns a timestamp at midnight) or order_ts::DATE
  • BigQuery: DATE(order_ts) or DATE_TRUNC(order_ts, DAY)
  • MySQL: DATE(order_ts)
  • SQL Server: CAST(order_ts AS DATE)

The important gotcha: DATE_TRUNC in Postgres returns a TIMESTAMP (not a DATE), which means the GROUP BY key has a time component (00:00:00) and displays differently. Both are correct for grouping; the display format differs.

ORDER BY day, revenue DESC — mixed sort directions

The sort is two-level: day ascending (default), then revenue descending within each day. Writing ORDER BY day ASC, revenue DESC makes both directions explicit and is better documentation. If an index exists on (order_ts, product_id), the optimizer may be able to avoid a sort for the day grouping, but sorting the final aggregate result typically requires a separate pass.

Adding a daily running total or rank

A common follow-up extends this query with a daily rank or running total using window functions over the aggregated result:

SELECT day, category, revenue,
       RANK() OVER (PARTITION BY day ORDER BY revenue DESC) AS rank_in_day,
       SUM(revenue) OVER (PARTITION BY day) AS day_total
FROM (
  SELECT DATE(o.order_ts) AS day, p.category, SUM(oi.qty * oi.price) AS revenue
  FROM orders o JOIN order_items oi ON o.order_id = oi.order_id
                JOIN products p ON oi.product_id = p.product_id
  GROUP BY DATE(o.order_ts), p.category
) agg
ORDER BY day, rank_in_day;
Senior signal: verify that SUM(qty * price) is computed at the line-item level — not SUM(qty) * SUM(price) which would be wrong for multi-item groups — explain the three-table FK chain, and give the cross-dialect DATE() / DATE_TRUNC / ::DATE equivalents. The window-function extension (rank within day, day total) is the over-the-bar follow-up that shows you’ve thought about the next reporting layer.
