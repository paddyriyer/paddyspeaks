Salesforce's marketing-cloud team gets a brand-safety complaint: "customers are saying we're hammering them with the same channel." The growth lead asks for the list of customers who got three or more consecutive touches on the same channel — e.g., email after email after email with no SMS or in-app in between. This is a textbook "consecutive run" pattern: LAG to compare to the previous row, then count.

C_01: 3 consecutive emails → FLAG. C_02: alternating channels → OK. C_03: 4 consecutive pushes → FLAG.

2. The SQL — LAG + HAVING COUNT(*) >= 3

SELECT customer_id
FROM (
  SELECT customer_id, channel, ts,
    LAG(channel) OVER (PARTITION BY customer_id ORDER BY ts) AS prev
  FROM touches
) t
WHERE channel = prev
GROUP BY customer_id
HAVING COUNT(*) >= 3;

4. Strategic explanations

The off-by-one trap — what does HAVING COUNT(*) >= 3 actually count?

The inner query produces rows where channel = prev: that is, rows where this touch matches the previous touch. For C_01's 3 emails, that's 2 matching rows (row 2 matches row 1; row 3 matches row 2). For 3 consecutive same-channel, HAVING COUNT(*) >= 2 is correct. The original query needs >= 3 to mean 4 consecutive same-channel. Always trace the off-by-one explicitly:

The senior version pins this down without ambiguity:

HAVING COUNT(*) >= 2  -- 3 consecutive same-channel (which is the actual ask)
Why LAG over a self-join

LAG(channel) OVER (PARTITION BY customer_id ORDER BY ts) reaches back exactly one row within each customer's chronologically-ordered touches. Equivalent self-join with ROW_NUMBER:

WITH n AS (
  SELECT customer_id, channel, ts,
    ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY ts) AS rn
  FROM touches
)
SELECT a.customer_id
FROM n a JOIN n b
  ON a.customer_id = b.customer_id
 AND b.rn = a.rn - 1
 AND a.channel = b.channel
GROUP BY a.customer_id
HAVING COUNT(*) >= 2;

LAG is shorter and the planner usually executes it as a single sort. Pick LAG.

The "any consecutive run of 3" gotcha — gaps-and-islands

The query above flags customers who have 2 same-channel pairs total. But those pairs could be spread across the timeline: 2 emails on Mon-Tue, then a gap, then 2 emails on Sat-Sun is also 2 pairs, but it's not a "run of 3". To find a customer with any single run of length ≥ 3, use the gaps-and-islands trick:

WITH labeled AS (
  SELECT customer_id, channel, ts,
    ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY ts) -
    ROW_NUMBER() OVER (PARTITION BY customer_id, channel ORDER BY ts) AS island_id
  FROM touches
)
SELECT DISTINCT customer_id
FROM labeled
GROUP BY customer_id, channel, island_id
HAVING COUNT(*) >= 3;

The two-row-number difference is constant within each island and changes when the channel breaks — the canonical pattern for "consecutive runs."

The time-window addition — "consecutive within 7 days"

Marketing's actual concern is "3 same-channel touches within a week", not 3 consecutive over any timespan. Add a time-gap check via LAG on the timestamp:

SELECT customer_id, channel, ts,
  LAG(channel) OVER w AS prev_ch,
  LAG(ts)      OVER w AS prev_ts
FROM touches
WINDOW w AS (PARTITION BY customer_id ORDER BY ts)
-- then filter: channel = prev_ch AND ts - prev_ts <= INTERVAL '7 days'
The do-not-disturb policy follow-up

Production marketing systems don't just find these customers — they have a frequency-cap rule upstream that blocks the third email from being sent in the first place. The query is for the audit / compliance side: which customers escaped the cap and why? The runbook is: query → compare to cap config → identify the broken segment / journey.

Senior signal: trace the off-by-one explicitly (HAVING >= 2 for 3 consecutive), choose LAG over a self-join, raise the gaps-and-islands flag for the "any run of 3" interpretation, and pre-empt the 7-day time-window variant. Linking back to the upstream frequency cap is the over-the-bar move.
