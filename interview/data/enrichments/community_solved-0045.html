A Community member models a bank ledger: every row is a transaction, tagged Credit or Debit, and the task is to derive the running net balance after each one. It is a signed running total — and the interviewer is watching whether you fold the sign into the accumulation cleanly, and whether you notice what an unexpected transaction type would do.

1. The source data — transactions

The Amount column is always stored positive — the direction lives in TranType. A debit must be subtracted, so the query has to translate the label into a sign before accumulating, not just sum the column.

2. The SQL — signed SUM() OVER (ORDER BY date)

SELECT trandate, tranid, trantype, amount,
       SUM(CASE WHEN trantype = 'Credit' THEN amount
                WHEN trantype = 'Debit'  THEN -amount END)
       OVER (ORDER BY trandate ASC) AS net_balance
FROM transactions;

The inner CASE turns each amount into a signed value — positive for a credit, negative for a debit. SUM() OVER (ORDER BY trandate) then accumulates those signed values into a running balance, one figure per transaction.

The balance walks 1000 → 700 (−300) → 1200 (+500) → 400 (−800) → 600 (+200). Each debit pulls the running figure down, each credit lifts it — the closing balance of 600 equals total credits (1700) minus total debits (1100).

4. Strategic explanations

The unmatched-type trap — the silent NULL

The CASE has branches for 'Credit' and 'Debit' but no ELSE. If a row carries any other TranType — 'Reversal', 'Fee', a lower-case 'credit', a stray space — the CASE returns NULL. SUM skips NULLs, so that transaction is silently omitted from the balance: the ledger no longer reconciles, and nothing errors. Defend it explicitly:

SUM(CASE WHEN trantype = 'Credit' THEN amount
         WHEN trantype = 'Debit'  THEN -amount
         ELSE 0 END) OVER (ORDER BY trandate)

Better yet, in a real pipeline add a CHECK constraint or a validation query so an unknown type fails loudly rather than quietly distorting the balance.

The ordering ambiguity — same-day transactions

ORDER BY trandate alone is ambiguous when two transactions share a date — and a bank balance is order-sensitive (a debit before a credit can mean an overdraft that the reverse order avoids). Two issues compound: the running total needs a deterministic order, and the default RANGE frame treats same-date rows as peers, giving them an identical balance. Order by a tiebreaker and switch to ROWS:

SUM(...) OVER (ORDER BY trandate, tranid
               ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)

tranid (or a transaction timestamp) makes the sequence reproducible; ROWS makes each transaction step the balance individually.

The opening-balance question

The query assumes the account starts at zero. Real accounts carry an opening balance from before the visible window. Seed it — either add a synthetic opening row, or add a constant: opening_balance + SUM(...) OVER (...). Confirm with the interviewer whether the first row is genuinely the account's first transaction ever.

The single-column alternative — store the sign

If you control the schema, a cleaner design stores Amount already signed (debits negative), or adds a generated signed_amount column. Then the running balance is simply SUM(signed_amount) OVER (ORDER BY ...) — no CASE, no chance of an unmatched type. Translating a label to a sign at query time is fine for a fixed two-value domain, but it is logic that belongs in the data model when you can put it there.

Per-account balances

The demo has one implicit account. A real ledger has many — the schema even hints at a customer_id. The balance must then reset per account: add PARTITION BY customer_id to the OVER clause. Without it, one customer's debits would wrongly eat into another's balance.

Senior signal: fold the credit/debit sign into the accumulation with a CASE that has an explicit ELSE (so an unknown type cannot silently drop a row), pin a deterministic tiebreaker with a ROWS frame for an order-sensitive balance, and add PARTITION BY the account plus an opening-balance seed for any real ledger.
