Instagram Stories expire after 24 hours by design. The Stories team monitors the engagement decay curve: if 80% of views happen in the first hour, the algorithm priorities matter; if views distribute uniformly, the timing matters less. The query bucketizes views by hour-since-post.

1. The source data — story_views

story_idposted_atviewed_at
S_00109:0009:15
S_00109:0011:30

2. The SQL — hourly bucketing via julianday

SELECT
  CAST((julianday(viewed_at) - julianday(posted_at)) * 24 AS INTEGER) AS hours_since_post,
  COUNT(*) AS views
FROM story_views
WHERE julianday(viewed_at) - julianday(posted_at) BETWEEN 0 AND 1
GROUP BY hours_since_post
ORDER BY hours_since_post;

3. Strategic explanations

The integer-cast bucketing trick

(julianday(viewed_at) - julianday(posted_at)) * 24 gives fractional hours. CAST(... AS INTEGER) truncates to whole hours: 0.5h → 0, 1.7h → 1, 3.99h → 3. That's the bucket. Cleaner than nested CASE statements.

The 24-hour cutoff

Stories expire at 24h, but the data table might have stale view events from clock-skew or replay attacks. The BETWEEN 0 AND 1 filter (julianday is in days, so 1 = 24h) bounds the window. Without it, the histogram has a long tail that's noise.

The cumulative-engagement follow-up

Per-hour counts are a histogram. Cumulative-share is more useful for product decisions:

SELECT hours_since_post, views,
  ROUND(100.0 * SUM(views) OVER (ORDER BY hours_since_post)
              / SUM(views) OVER (), 1) AS cumulative_pct
FROM /* the histogram CTE */;

If cumulative_pct(1h) = 65%, the first hour drives most engagement; algorithm changes that demote stories after hour 1 lose < 35% of value.

The "follower count" bias

A story from an account with 1M followers naturally accumulates views faster than one with 100. The decay curve shape normalizes this — both might peak in hour 1. But for absolute volume, weight by follower count.

Per-creator-tier split

Decay shape varies dramatically by creator tier: celebrity stories decay slowly (steady viewership across 24h); friend-of-friend decays fast (peak in 30 min, gone). Slice by creator-tier:

JOIN creators c ON c.creator_id = story_views.creator_id
GROUP BY hours_since_post, c.tier
The "view through" definition

Stories auto-advance after 5 sec. Does a 1-second peek count as a "view"? Production has multiple definitions:

  • Impression — story appeared on screen, any duration
  • View — >= 1 second visible
  • Complete view — watched to the end

The choice changes which decay curve you see. Check definition with the requester.

Senior signal: use the integer-cast bucketing trick, layer cumulative-share on top for product decisions, and surface the creator-tier-dependent decay-shape variation. The view-definition disambiguation (impression vs view vs complete) is the over-the-bar guard.
