JaneStreet's HR-comp team wants to publish the comp range for each level. The simplest possible question: what's the highest-paid employee's salary? — and yet it's the question that separates juniors from seniors, because the right answer depends on what the recruiter actually meant by "highest". One person? All people tied at the top? The Nth highest? The trick is to surface those interpretations before writing SQL.

Alex and Casey are tied at $320K. That tie is what makes this interesting.

2. The SQL — MAX(salary)

SELECT MAX(salary) AS highest_salary FROM employees;

3. The resulting analysis

highest_salary
$320,000

One row, one number. But this answer is incomplete for any real recruiter conversation — they want to know who, not just the number, and they want to handle ties.

4. Strategic explanations

The "give me the person, not the number" follow-up

Most interviewers will immediately ask: "Now show me the employee's name." Three correct ways, with very different semantics:

-- 1. ORDER BY + LIMIT 1 — returns ONE employee, arbitrary tiebreak
SELECT name, salary FROM employees ORDER BY salary DESC LIMIT 1;

-- 2. Subquery — returns ALL employees tied at the top
SELECT name, salary FROM employees
WHERE salary = (SELECT MAX(salary) FROM employees);

-- 3. RANK() — returns ALL ties AND extends to "top N salaries"
SELECT name, salary FROM (
  SELECT name, salary, RANK() OVER (ORDER BY salary DESC) AS rk
  FROM employees
) WHERE rk = 1;

For our data: option 1 returns Alex or Casey (whichever the planner picks — non-deterministic!). Options 2 and 3 return both. Always confirm with the interviewer: "if there's a tie, do you want one row or all of them?"

The Nth-highest salary — the LeetCode classic

The follow-up is always: "now find the 2nd highest." Three approaches:

-- A. DENSE_RANK — handles ties correctly (2nd highest = next distinct salary)
SELECT salary FROM (
  SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rk
  FROM employees
) WHERE rk = 2;

-- B. LIMIT/OFFSET — fast but tie-blind
SELECT DISTINCT salary FROM employees ORDER BY salary DESC LIMIT 1 OFFSET 1;

-- C. Correlated subquery — slow on large tables, avoid
SELECT MAX(salary) FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);

For our data: 2nd-highest distinct salary = $280K (Bailey). With RANK instead of DENSE_RANK, the 2nd highest gets skipped (ranks 1, 1, 3) — that's the trap.

RANK leaves gaps after ties. DENSE_RANK doesn't. ROW_NUMBER ignores ties entirely. For "Nth highest" interpretations: DENSE_RANK is almost always what you want.

The empty-table edge case

MAX on an empty table returns NULL, not an error or 0. If the consumer expects 0, wrap it: COALESCE(MAX(salary), 0). The Nth-highest patterns return zero rows when N exceeds the number of distinct salaries — recruiters often want NULL instead, which means a final SELECT (subquery LIMIT 1) wrapper.

The pay-equity follow-up

The recruiter's actual question is rarely just "highest" — it's usually pay-equity-adjacent: "highest by gender, by location, by level." Add PARTITION BY to RANK and pivot:

SELECT level, gender, MAX(salary) AS top_salary,
  COUNT(*) FILTER (WHERE salary >= MAX(salary) OVER (PARTITION BY level, gender)) AS num_at_top
FROM employees
GROUP BY level, gender;
Senior signal: ask "ties or no ties?" before writing SQL, name DENSE_RANK as the safe default for Nth-highest, contrast RANK / DENSE_RANK / ROW_NUMBER in one breath, and pre-empt the empty-table NULL behaviour. Pivoting to the pay-equity question is the over-the-bar finish.
