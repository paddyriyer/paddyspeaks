TikTok's growth analysts want to roll orders up by month for cohort charts, so they need year and month pulled out of order_date as plain integers. It is a one-liner — but "extract the month" hides a choice between half a dozen functions, and the interviewer wants to know you picked one deliberately and know what type it returns.

1. The source data — orders

Order 3 lands on 2024-12-31 — the last day of the year. That edge case checks that the split returns month 12, year 2024, and does not roll the year forward.

2. The SQL — TO_CHAR then cast to integer

SELECT order_id,
       CAST(TO_CHAR(order_date, 'YYYY') AS INTEGER) AS yr,
       CAST(TO_CHAR(order_date, 'MM') AS INTEGER) AS mo
FROM orders;

TO_CHAR formats the date as text using a pattern, then CAST(… AS INTEGER) turns '2024' and '01' back into numbers. The MM pattern is zero-padded, but the cast strips the leading zero so January becomes 1.

4. Strategic explanations

The idiomatic Postgres way is EXTRACT

TO_CHAR(…)-then-cast works, but it round-trips through text needlessly. EXTRACT pulls a numeric field straight out of the date:

SELECT order_id,
       EXTRACT(YEAR  FROM order_date)::int AS yr,
       EXTRACT(MONTH FROM order_date)::int AS mo
FROM orders;

EXTRACT returns numeric in modern Postgres, so the ::int cast keeps the column a clean integer. Same result, no string detour. Mentioning both and explaining the trade-off is the senior move.

The "month sorts as text" trap

If you skip the cast and select TO_CHAR(order_date, 'MM') directly, mo is text. Then ORDER BY mo sorts lexically: '01', '02', '10', '11', '12', '03' … actually '1' < '10' < '11' < '12' < '2'. Charts come out in nonsense order. The explicit CAST(… AS INTEGER) is what makes downstream GROUP BY/ORDER BY behave.

A month number alone is ambiguous — you usually want a month bucket

Splitting into yr and mo means January 2024 and January 2025 both have mo = 1. For a true monthly time series you want a single sortable key. DATE_TRUNC gives one:

SELECT DATE_TRUNC('month', order_date) AS month_bucket,
       COUNT(*) AS orders
FROM orders
GROUP BY 1
ORDER BY 1;

DATE_TRUNC('month', '2024-12-31') returns 2024-12-01 — one timestamp per month, naturally ordered, no two-column join key needed.

Dialect note — this is not portable as written

Functions for date parts differ sharply across engines. MySQL uses YEAR(order_date) and MONTH(order_date); SQL Server uses DATEPART(year, order_date); BigQuery uses EXTRACT(YEAR FROM order_date) like Postgres. TO_CHAR exists in Postgres and Oracle but not MySQL. Name the target engine before reaching for a date function.

Edge cases: NULL dates and time zones

A NULL order_date yields NULL for both yr and mo — harmless, but those rows then form a NULL group in any roll-up. And if order_date were a timestamptz, the extracted month depends on the session time zone: an order at 2024-12-31 23:00 UTC is January in UTC+02. Pin the zone with AT TIME ZONE when the boundary matters.

Senior signal: reach for EXTRACT over a TO_CHAR text round-trip, always cast month/year to integer so they sort numerically, suggest DATE_TRUNC when the real need is a monthly bucket, and flag the dialect differences and the time-zone dependency for timestamptz columns.
