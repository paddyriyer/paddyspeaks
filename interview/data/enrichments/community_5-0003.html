A Community education platform wants a pass-rate dashboard: for each course and semester, what fraction of students passed? The shape is conditional aggregation — count passes, divide by total. The interviewer is watching for one classic trap (integer division silently flooring every rate to 0) and one judgement call (how NULL grades and the pass threshold are handled).

1. The source data — student_courses

Two rows sit right on the boundary: student 4 at exactly 50 (passes, with >= 50) and student 5 at 49 (fails). That one-point gap is the whole interview — it decides whether CS101/Spring2025 is a 50% or a 100% pass rate, and it forces you to confirm whether 50 is a pass.

2. The SQL — conditional aggregation

SELECT
  course_id,
  semester,
  100.0 * SUM(CASE WHEN grade >= 50 THEN 1 ELSE 0 END) / COUNT(student_id)
    AS pass_rate_pct
FROM student_courses
GROUP BY course_id, semester
ORDER BY course_id, semester;

The SUM(CASE ...) is a filtered count of passes; COUNT(student_id) is the group size. Multiplying by the literal 100.0 — not 100 — is the detail that makes the result a real percentage.

The raw value carries a long decimal tail — a dashboard would wrap it in ROUND(…, 2), but the un-rounded number proves the division happened in floating point, not integer.

4. Strategic explanations

The integer-division trap — the one that fails the question

The single most common wrong answer drops the .0:

-- WRONG: every rate floors to 0
100 * SUM(CASE WHEN grade >= 50 THEN 1 ELSE 0 END) / COUNT(student_id)

With integer operands, 2 / 3 evaluates to 0 before the * 100 can help — so CS101/Fall2024 reports 0%. The fix is to force one operand to be non-integer: the literal 100.0, or SUM(...)::numeric, or CAST(... AS FLOAT). The reference solution does this correctly with 100.0. Always make the numerator floating-point before the division operator, never after.

SUM(CASE) vs COUNT(CASE) vs FILTER

Three equivalent ways to count the passes — know all three:

SUM(CASE WHEN grade >= 50 THEN 1 ELSE 0 END)   -- portable, every dialect
COUNT(CASE WHEN grade >= 50 THEN 1 END)         -- COUNT ignores the NULL else-branch
COUNT(*) FILTER (WHERE grade >= 50)             -- Postgres / standard SQL, cleanest

The trap in the middle form: COUNT(CASE WHEN ... THEN 1 ELSE 0 END) counts every row, because COUNT counts the literal 0 too — you must omit the ELSE so non-passes become NULL and are skipped. SUM with ELSE 0 is safe; COUNT with ELSE 0 is a silent bug.

NULL grades — numerator and denominator disagree

If a student is enrolled but ungraded, grade is NULL. CASE WHEN grade >= 50 is unknown, so the ELSE 0 branch fires — the NULL-grade student counts as a fail. Meanwhile COUNT(student_id) still counts them in the denominator. Is an ungraded student a fail, or should they be excluded? Decide explicitly:

-- exclude ungraded students from both numerator and denominator
100.0 * COUNT(*) FILTER (WHERE grade >= 50)
      / COUNT(*) FILTER (WHERE grade IS NOT NULL) AS pass_rate_pct

Using COUNT(grade) as the denominator achieves the same thing — it counts only non-NULL grades.

The pass threshold is a business rule, not a guess

The solution hard-codes grade >= 50. Is the pass mark 50, or 40, or does it vary by course? And is exactly-50 a pass (>= 50) or a fail (> 50)? Student 4 sits precisely on that line. Confirm the rule before writing SQL — an off-by-one in the comparison flips a real student's outcome and the whole course's headline number.

Empty groups and division by zero

A course/semester with zero enrolled students cannot appear here — GROUP BY only produces groups that have rows. But if the denominator were filtered (e.g. COUNT(grade) and every grade in a group is NULL), it becomes 0 and the division errors. Guard it with NULLIF(COUNT(grade), 0) so the rate returns NULL instead of throwing.

Senior signal: force floating-point division with 100.0 (or a cast) so rates do not floor to zero, prefer COUNT(*) FILTER or SUM(CASE) while knowing COUNT(CASE ... ELSE 0) is a trap, decide deliberately whether a NULL grade is a fail or an exclusion, and confirm the pass threshold and its boundary rather than hard-coding a guess.
