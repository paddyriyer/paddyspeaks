A community sales dataset asks for a running total of sales per product — each row carrying the cumulative amount for its product up to and including that row, in id order. Running totals are the most-requested window-function task in analytics: think account balances, cumulative revenue, burn-down charts. The interviewer wants to see SUM(...) OVER (PARTITION BY ... ORDER BY ...) and to confirm you understand what the ORDER BY inside the window does.

1. The source data — sales_tab

idproductsales(narrative)
1Pen10Pen, first
2Pen20Pen, second
3Book100Book, first
4Pen30Pen, third — non-adjacent to ids 1–2
5Book50Book, second
6Pencil15Pencil — single-row product

Two edge cases: Pen's rows are interleaved with other products (ids 1, 2, 4), so the running sum must follow the product partition, not physical adjacency; and Pencil is a single-row product, whose running total is just its own value.

2. The SQL — partitioned running SUM

SELECT id, product, sales,
  SUM(CASE WHEN sales::text ~ '^-?[0-9]+(\.[0-9]+)?$' THEN sales::numeric ELSE 0 END)
    OVER (PARTITION BY product ORDER BY id ASC) AS sales_products
FROM sales_tab;

SUM(...) OVER (PARTITION BY product ORDER BY id) accumulates within each product, in id order. The presence of ORDER BY inside the window is what turns a flat per-product total into a running one.

4. Strategic explanations

Why ORDER BY inside OVER creates the running total

A window SUM with an ORDER BY gets an implicit frame of RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — so each row sums itself plus everything before it in the ordering. Remove the ORDER BY and the frame becomes the whole partition: every Pen row would show 60, the flat product total. The ORDER BY is the single clause that distinguishes "running total" from "grand total".

PARTITION BY resets the accumulator per product

Each distinct product is its own window; the running sum starts fresh at the first row of each partition. That is why Book begins again at 100 even though Pen rows preceded it, and why interleaving in id order does not leak across products. The window function evaluates partition-by-partition, not by table layout.

The RANGE vs ROWS tie subtlety

The default frame is RANGE, which treats peer rows — rows with the same ORDER BY value — as a single step: tied rows all receive the same cumulative total. Here id is unique, so RANGE and ROWS behave identically. But if two Pen rows shared an id, RANGE would give both the post-addition total while ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW would step through them one at a time. For a true row-by-row running balance, write ROWS explicitly — and always order by a unique, deterministic key.

The TEXT-column guard and what it hides

sales is stored as TEXT, so the CASE ... ~ '^-?[0-9]+(\.[0-9]+)?$' regex validates each value is numeric before ::numeric; a stray 'N/A' would otherwise abort the whole query with a cast error. The ELSE 0 branch keeps the query alive but silently treats dirty rows as zero — the running total then under-reports with no warning. A stricter design routes non-numeric rows to a quarantine count or fails loudly. The real fix is upstream: sales should be a NUMERIC column, not text.

Senior signal: explain that the ORDER BY inside OVER is what makes the sum cumulative (drop it and you get the flat total), note that the default RANGE frame lumps tied rows so a true running balance needs explicit ROWS and a unique sort key, and flag that the ELSE 0 cast guard silently zeroes dirty data rather than surfacing it.
