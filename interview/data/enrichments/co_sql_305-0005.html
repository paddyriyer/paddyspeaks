Adobe's HR-systems team is building a directory page that pairs every employee with their department's human-readable name. The fact table only stores a dept_id; the label lives in a separate departments table. A two-table join — the most routine query there is. The interview question underneath: which join, and what happens to an employee whose dept_id is missing or wrong?

Five employees, but two are awkward: Dan has a NULL dept_id (a contractor not yet assigned) and Eve points at dept_id 99 — a department that does not exist in the departments table. Those two rows decide the answer.

2. The SQL — INNER JOIN on dept_id

SELECT e.name, d.dept_name
FROM employees e
JOIN departments d ON e.dept_id = d.dept_id;

A plain JOIN — an INNER JOIN — matching each employee's dept_id to a department row, returning only employees whose department resolves.

Three rows, not five. Dan and Eve are gone. Dan's dept_id is NULL, so NULL = d.dept_id is never true. Eve's dept_id 99 has no matching department row. An INNER JOIN keeps only rows that match on both sides — so both unmatched employees drop out silently.

4. Strategic explanations

The headcount question — is dropping Dan and Eve correct?

It depends entirely on the use case. For a directory page that only shows assigned staff, an INNER JOIN is fine. For a headcount — "how many employees do we have?" — silently losing two people is a bug. Use a LEFT JOIN to keep every employee:

SELECT e.name, COALESCE(d.dept_name, '(unassigned)') AS dept_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;
-- all 5 rows: Dan and Eve come back with dept_name = '(unassigned)'

The senior move is to ask the interviewer: "should employees with no resolvable department appear in this report?" The join type is the answer to that question, not a default.

NULL dept_id vs orphaned dept_id — two different bugs

Dan and Eve both disappear from the inner join, but for different reasons, and a good answer distinguishes them. Dan's dept_id is NULL — legitimately unassigned. Eve's is 99, a non-NULL value that should match but does not — an orphaned foreign key, a genuine data-integrity violation. A LEFT JOIN filtered to unmatched rows separates the two:

SELECT e.emp_id, e.name, e.dept_id
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id
WHERE d.dept_id IS NULL;
-- Dan (NULL — expected) and Eve (99 — orphaned FK, needs fixing)

Eve's row is the one to escalate: it means a department was deleted without reassigning its staff, or a typo went uncaught. A proper FOREIGN KEY constraint would have blocked it at write time.

The duplicate-rows trap

This join is one-to-many done right (many employees, one department), so no row multiplication. But if departments had two rows with dept_id = 10 — a dirty dimension table — every Engineering employee would appear twice. Before trusting a join, confirm the join key is unique on the dimension side. A quick SELECT dept_id, COUNT(*) FROM departments GROUP BY dept_id HAVING COUNT(*) > 1 proves it.

Qualify your columns

dept_id exists in both tables, so an unqualified SELECT dept_id is ambiguous and errors. name and dept_name happen to be unique here, but qualifying every column (e.name, d.dept_name) is the habit that survives schema changes — the day someone adds a name column to departments, an unqualified query breaks.

Senior signal: recognise that INNER JOIN silently drops employees with a NULL or orphaned dept_id, choose LEFT JOIN deliberately when the report is a headcount, distinguish a legitimately-NULL key from an integrity-violating orphaned FK, and check the dimension side for duplicate keys before trusting the row count.
