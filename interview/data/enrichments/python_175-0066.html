Largest Rectangle in Histogram is in the L5 canon because it's the monotonic-stack masterclass — three valid solutions (O(n²) brute force, O(n log n) divide-and-conquer, O(n) stack), and only the stack one survives at scale. The reason it appears in so many interviews is the follow-up: the same primitive solves Maximal Rectangle in a 0/1 Matrix (LeetCode 85), which runs inside every OCR-table-extraction pipeline (Google Document AI, AWS Textract, Adobe Acrobat's auto-recognise-tables) and every cluster job-packing scheduler that needs to find the biggest contiguous rectangular block of free resources. Vercel uses it inside their serverless cold-start packer to find the largest contiguous block of warm-pool capacity that can host a new isolate.

1. The problem in a sentence

Given an array of non-negative integers representing histogram bar heights (each of unit width), find the area of the largest axis-aligned rectangle that fits entirely inside the histogram. The naive brute force — for each bar, expand left and right while bars are ≥ current height — is O(n²); for n up to 10⁵ it doesn't fit in the time budget. The constraint that rules it out is O(n) amortised. The trick is a monotonic-increasing stack of (start_index, height) pairs: when a shorter bar arrives, all taller bars on the stack get their right boundary fixed at the current index, and the shorter bar "inherits" the leftmost extension of the bars it just popped.

                                finalise (0,1): 1 * (6-0) = 6       10

answer: 10  (rectangle 5×2 spanning indices 2,3)

The crucial moment is i=4 popping (2,5). When 2 arrives, both 5 and 6 are taller and get evicted — but the start_index of the popped 5 is 2 (it had been on the stack since i=2), and the rectangle stretches from 2 (inclusive) to 4 (exclusive) for a width of 2 and area of 10. The newly-pushed bar (the shorter 2) inherits start=2 because it could have extended left as far as the popped 5 could have. This start-inheritance is what lets us account for rectangles whose left boundary is earlier than the bar that started them.

3. The algorithm — naming the data structure

def largestRectangleArea(h):
    """Monotonic-increasing stack of (start_index, height). Each bar is
       pushed and popped at most once, so total work is O(n)."""
    st, best = [], 0
    for i, x in enumerate(h):
        start = i                              # default: this bar starts here
        # Step 1: pop every taller (or equal-tall) bar -- their right
        # boundary is fixed at i (the new bar is the first thing shorter).
        while st and st[-1][1] > x:
            j, y = st.pop()
            best = max(best, y * (i - j))      # area of the popped rect
            start = j                          # the new bar can extend back to here
        st.append((start, x))                  # carry-over: start may predate i
    # Step 2: drain the stack -- anything still on it can extend to the right edge.
    n = len(h)
    for j, y in st:
        best = max(best, y * (n - j))
    return best

The (start_index, height) variant is the cleanest formulation. The alternative — push only indices and compute width as i - stack[-1] - 1 after popping — is what most LeetCode solutions show, but the start-inheritance version is easier to reason about under interview pressure and easier to extend to the 2D follow-up (Maximal Rectangle).

4. Complexity

Time: O(n) amortised. Each index is pushed onto the stack exactly once (in the for loop) and popped at most once (either by the inner while or by the final drain). Total stack operations are bounded by 2n. Space: O(n) — the stack can hold at most n entries (worst case: strictly increasing input). The trick that gets you there: the start-index carry-over lets each rectangle be finalised exactly when its right boundary is known, and the monotonic invariant guarantees each bar contributes to exactly one finalised rectangle.

5. The interviewer's edge-case probes

  • Single bar — answer is the bar's height (1 × height).
  • Strictly increasing — no pops during the loop; the entire stack drains at the end. The widest rectangle wins, then height kicks in.
  • Strictly decreasing — every new bar pops the previous one. The stack stays size 1 after each step (start_index inherits).
  • All-equal bars — no pops during the loop (we pop only when strictly taller); drain at the end gives the full width × the height.
  • A zero in the middle — forces all previous bars to be finalised; nothing on the right side can extend left past the zero. This is exactly how the 2D Maximal Rectangle problem reduces to this one: each row's "histogram" is the column-wise consecutive 1s, and a zero in row R resets that column's height.
  • Empty input — return 0. The if not h guard isn't strictly necessary; the loop is a no-op and best stays 0.
  • 2D extension (Maximal Rectangle, LeetCode 85) — for each row, compute the column-wise running height of 1s, then run this algorithm on the resulting histogram. Repeat for all rows; max over the m runs. O(m·n) total.

6. Real-world appearance — what runs this in production

OCR table extraction — Google Document AI, AWS Textract, Adobe Acrobat

The first step of table extraction from a scanned page is binarisation (every pixel is 0 or 1). The second step is finding the largest white rectangle (table cell) inside a region of mostly-white pixels — this is Maximal Rectangle on a 0/1 matrix. Google's Document AI ships a SIMD-vectorised version of the algorithm in their tabular-data extractor; Adobe's "auto-recognise-tables" feature in Acrobat Pro uses it; AWS Textract's TABLE block detection is based on the same primitive.

Cluster job packing — Kubernetes scheduler, YARN, Slurm

When a multi-pod job requests a contiguous block of nodes (e.g., a 64-GPU all-reduce training job that needs locality), the scheduler searches for the largest contiguous rectangle of free capacity in the cluster's resource map. Kubernetes' Volcano scheduler ships a BinPack plugin that uses this primitive. YARN's FairScheduler uses a 2D variant for memory-and-cores joint packing. Slurm's SelectPlugin/cons_tres does the same for HPC clusters.

Serverless cold-start packing — Vercel, Cloudflare Workers, AWS Lambda

Vercel's serverless platform maintains a "warm pool" of pre-initialised V8 isolates. When a new function deploys, the scheduler needs to find the largest contiguous block of warm-pool capacity that matches the function's memory shape, to maximise the chance of a warm hit. The bin-packing inside is largest-rectangle. Cloudflare Workers' Durable Objects placement uses the same primitive across colos.

GIS / urban planning — ESRI ArcGIS, Mapbox

"Find the largest buildable rectangle inside a parcel with non-rectangular boundaries" is the developer-feasibility-study primitive. ESRI's Build Footprints tool and Mapbox's GeoSpatial Studio use a discretised grid form of this algorithm.

Manufacturing — sheet-metal nesting (PTC Creo, Solidworks)

Cutting parts from a sheet of steel — the "nesting" step — uses largest-rectangle as a building block in the heuristic stack. The 2D variant (Maximal Rectangle in a 0/1 matrix of "already-cut" vs "still-available") is what runs in PTC Creo's optimiser.
