A Community live-streaming platform wants its headline capacity number: what was the maximum number of viewers watching at the same instant? Each viewer has a join time and a leave time; the answer is the peak overlap of those intervals. The interviewer is watching whether you reach for the sweep-line trick, and crucially how you break the tie when one viewer leaves at the exact second another joins.

1. The source data — live_stream_viewers

The interesting instant is 10:07:00: viewer 3 leaves at exactly the second viewer 5 joins. Whether that counts as an overlap — do both occupy 10:07, or does one hand off to the other? — decides if the peak is 3 or 4. That tie is the entire interview.

2. The SQL — sweep-line running SUM

WITH event_log AS (
    SELECT start_time AS event_time,  1 AS indicator FROM live_stream_viewers
    UNION ALL
    SELECT end_time   AS event_time, -1 AS indicator FROM live_stream_viewers
),
running_total AS (
    SELECT
        event_time,
        SUM(indicator) OVER (ORDER BY event_time ASC, indicator DESC)
            AS current_viewers
    FROM event_log
)
SELECT MAX(current_viewers) AS peak_concurrent_viewers
FROM running_total;

Each session is exploded into two timeline events: +1 at the start, -1 at the end. Sorting and running a cumulative SUM gives the live audience after every event; the MAX of that running total is the peak.

The aggregate query returns a single row: peak_concurrent_viewers = 4. The peak is hit at 10:07:00 — viewers 1 and 2 long gone, but 2, 4, 5 plus the just-leaving 3 momentarily coexist. The indicator DESC tiebreak is what produces the 4: the +1 for viewer 5 is processed before the -1 for viewer 3.

4. Strategic explanations

The tie-break is the whole question — ORDER BY event_time, indicator DESC

At 10:07:00 there are two events: viewer 5 joins (+1) and viewer 3 leaves (-1). The secondary sort key indicator DESC forces +1 (which sorts after -1 descending) to be applied first, so the running total touches 4 before dropping to 3. This treats intervals as closed — a viewer present at exactly their end_time still counts. Flip to indicator ASC and the -1 goes first: the total never exceeds 3, treating intervals as half-open. Neither is "right" universally — you must ask the interviewer: "if a viewer leaves at 10:07:00 and another joins at 10:07:00, do they overlap?" The answer picks the sort direction.

Why the sweep-line beats a self-join

The brute-force answer asks, for each viewer, "how many other intervals contain my start time?" — a correlated count, O(n²), one full scan per viewer. The sweep-line touches each of the 2n events once after a single sort: O(n log n). On a stream with millions of concurrent sessions that is the difference between a query that returns and one that does not. Exploding intervals into +1/-1 events and running a cumulative sum is the canonical pattern for any "max overlap" / "peak concurrency" problem — meeting rooms, hotel occupancy, server connections.

The window frame — a subtle default

SUM(indicator) OVER (ORDER BY event_time ASC, indicator DESC) has no explicit frame. With an ORDER BY and no frame clause, the default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — a running total, which is what we want. But RANGE (as opposed to ROWS) treats peer rows — rows with an identical ORDER BY key — as a single block, summing them all at once. Here the tiebreak indicator DESC makes the two 10:07 events non-peers (different indicator), so each gets its own running value and we correctly see the intermediate 4. If two viewers shared the exact same event_time AND indicator, RANGE would lump them; switching to ROWS would step through one at a time. Worth knowing which frame you are getting.

Finding WHEN the peak happened

The question asks only for the number, but the natural follow-up is "when?". The aggregate MAX discards that. Order the running total instead:

SELECT event_time, current_viewers
FROM running_total
ORDER BY current_viewers DESC, event_time ASC
LIMIT 1;
-- => 10:07:00, 4

If the peak is sustained across several events, this returns the first instant it was reached; reporting the full peak interval needs a bit more work (filter to current_viewers = MAX and take min/max event_time).

Edge cases: zero-length sessions, NULLs, and text timestamps

A session where start_time = end_time contributes a +1 and a -1 at the same instant; with indicator DESC the count blips up by one then back — arguably correct (the viewer was there for an instant), arguably noise. A NULL end_time (still-watching viewer) produces a NULL event that sorts unpredictably and never decrements — coalesce it to the stream's current time or the report cutoff. And note the columns are TEXT: ISO YYYY-MM-DD HH:MM:SS sorts correctly as text, but casting to timestamp makes the ordering robust to any format drift.

Senior signal: recognise "peak concurrency" as the interval sweep-line (+1/-1 events, running SUM, MAX), explicitly justify the indicator DESC tiebreak as the closed-vs-half-open interval decision and ask which the business wants, explain why it is O(n log n) versus an O(n²) self-join, and offer the "when" variant before being asked.
