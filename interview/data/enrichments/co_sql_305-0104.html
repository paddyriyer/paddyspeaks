Meta's finance team — here looking at a pharma portfolio — wants the three most profitable drugs, where profit is revenue minus cost. It is an ORDER BY ... LIMIT, the shortest kind of ranking query. The interview is about what LIMIT 3 does when drugs are tied in profit, and whether "profit" should really be a raw subtraction.

1. The source data — drugs

drug_idnamerevenuecost
1Aspirin1000400
2Insulin50002000
3Statin30001000
4Antibiotic20001900
5Vaccine80005000

Compute profit and a tie appears: Insulin (5000 − 2000) and Vaccine (8000 − 5000) both net 3000. With LIMIT 3 and a fourth-place drug (Statin at 2000) below them, that tie sits comfortably inside the cut — but it sets up the trap for when a tie straddles the boundary.

2. The SQL — ORDER BY a computed column, LIMIT 3

SELECT name, revenue - cost AS profit FROM drugs
ORDER BY profit DESC LIMIT 3;

Profit is computed inline, the result is sorted highest-first, and the top three rows are returned.

3. The resulting analysis

Top three by profit. Insulin and Vaccine are tied at 3000 and the order between them is arbitrary — ORDER BY profit DESC alone does not say which 3000 comes first, so the engine is free to swap them. Here both are inside the top 3 so it does not change membership; but if the tie spanned positions 3 and 4, LIMIT 3 would keep one and drop the other unpredictably.

4. Strategic explanations

ORDER BY can reference a SELECT alias — WHERE cannot

Notice ORDER BY profit uses the alias defined in the SELECT list. That is allowed: ORDER BY is logically evaluated after the SELECT projection, so the alias exists. WHERE and GROUP BY run before projection, so WHERE profit > 0 would fail with "column does not exist" — you would repeat the expression: WHERE revenue - cost > 0. Knowing which clauses can see the alias is a senior detail.

The tie at the LIMIT boundary — the real bug

LIMIT 3 with an unstable sort means: if the 3rd and 4th drugs tie on profit, which one is "top 3" is undefined and can flip between runs. Two fixes, depending on intent. Make it deterministic with a tiebreaker:

SELECT name, revenue - cost AS profit FROM drugs
ORDER BY profit DESC, name ASC   -- stable: ties broken alphabetically
LIMIT 3;

Or, if the rule is "include everyone tied for 3rd", drop LIMIT and rank:

SELECT name, profit FROM (
  SELECT name, revenue - cost AS profit,
         RANK() OVER (ORDER BY revenue - cost DESC) AS rk
  FROM drugs) t
WHERE rk <= 3;

With RANK, two drugs tied for 1st both get rank 1, the next is rank 3, and "top 3 ranks" may return more than three rows. Asking "if there's a tie for last place, one row or all?" is the over-the-bar move.

Raw profit vs profit margin

revenue - cost is absolute profit. A blockbuster with huge revenue and huge cost can out-rank an efficient drug. Finance often cares about margin — profit as a fraction of revenue:

SELECT name, revenue - cost AS profit,
       ROUND(100.0 * (revenue - cost) / NULLIF(revenue, 0), 1) AS margin_pct
FROM drugs
ORDER BY margin_pct DESC LIMIT 3;

The NULLIF(revenue, 0) guards against a divide-by-zero for a drug with no revenue. "Most profitable" by dollars and by margin can be entirely different lists — clarify which the finance team means.

NULL revenue or cost

If any drug has a NULL revenue or cost, then revenue - cost is NULL. In Postgres, ORDER BY ... DESC places NULLs first by default — so a drug with missing data could be wrongly ranked as the most profitable. Defend with NULLS LAST, or filter incomplete rows: WHERE revenue IS NOT NULL AND cost IS NOT NULL.

Senior signal: spot that ORDER BY profit DESC LIMIT 3 has an undefined tie-break at the cut line — add a deterministic key or switch to RANK() depending on whether ties for last should be included — note that ORDER BY can use the SELECT alias while WHERE cannot, and offer profit margin (with a divide-by-zero guard) as the likely real metric.
