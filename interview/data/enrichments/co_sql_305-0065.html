Marketing wants to send three different email campaigns: a thank-you to VIPs, a loyalty offer to Regulars, and an onboarding nudge to New customers. The segmentation rules — VIP at $10K+ lifetime, Regular at $1K–$10K, New below that — come from the analytics team and need to be encoded once in SQL. CASE is the universal answer.

1. The source data — customers

Notice C_006 sits exactly on the $10K threshold and C_007 exactly on $1K — boundary cases the CASE has to handle deterministically.

2. The SQL — tiered CASE with explicit boundaries

SELECT
  customer_id,
  total_spent,
  CASE
    WHEN total_spent >= 10000 THEN 'VIP'
    WHEN total_spent >= 1000  THEN 'Regular'
    ELSE                            'New'
  END AS segment
FROM customers;

4. Strategic explanations

Order of CASE matters — most-restrictive first

Reverse the order and everything breaks. If you wrote WHEN total_spent >= 1000 THEN 'Regular' before the VIP rule, then C_001 (24K) and C_003 (11.8K) would land in 'Regular' because they match the first WHEN that fires — CASE evaluates top-down and short-circuits. Rule of thumb: most-restrictive first, fallback last.

The boundary-inclusion choice — >= vs >

$10,000 exactly: VIP or Regular? With >= 10000 they're VIP. With > 10000 they're Regular. Both are valid; pick consciously. Marketing usually wants the more generous interpretation (more VIPs = more retention spend = more revenue), so >= is conventional. Document the choice in a comment so finance can audit it.

ELSE matters more than it looks

Without ELSE 'New', customers below $1K would get NULL in the segment column. Most BI tools treat NULL as "no data" and silently drop those customers from the report — they vanish from the email list. Always include ELSE with an explicit fallback value. Empty ELSE is rarely what you want.

When CASE outgrows itself — refactor to a rules table

If marketing comes back with 6 tiers, then sub-tiers, then per-region rules, the CASE balloons unreadably. At that point, the rules belong in a dim_segment_rule table joined on the threshold:

SELECT c.customer_id, c.total_spent, r.segment_name
FROM customers c
JOIN dim_segment_rule r
  ON c.total_spent >= r.lower_bound
 AND c.total_spent <  r.upper_bound;

Now marketing can edit the rules table without a code deploy.

The follow-up: counts per segment

Inevitably the analyst asks "how many customers are in each segment?". Wrap it:

SELECT segment, COUNT(*) AS n, SUM(total_spent) AS revenue
FROM (
  SELECT customer_id, total_spent,
    CASE WHEN total_spent >= 10000 THEN 'VIP'
         WHEN total_spent >= 1000  THEN 'Regular'
         ELSE                            'New' END AS segment
  FROM customers
) t
GROUP BY segment;
The over-the-bar finish: RFM segmentation

"Total spent" is the simplest segmentation. The actual marketing-team standard is RFM — Recency × Frequency × Monetary. Each axis gets its own CASE, then the three are combined into a 555 / 555 / 555 grid with custom segment names ("Champions", "At-Risk", "Lost"). Mention RFM as the next-level evolution; senior candidates know the term.

Senior signal: name order-of-CASE matters upfront, articulate the boundary-inclusion choice consciously, and pre-empt the rules-table refactor at scale. RFM is the over-the-bar finish — it's what real marketing teams actually use.
