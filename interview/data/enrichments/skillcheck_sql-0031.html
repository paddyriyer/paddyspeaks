A pricing team builds a 2-period trailing sum over a price log and wants to see exactly which rows each window frame is adding. The data has two rows sharing one timestamp, which is the whole point: it exposes the difference between a ROWS frame (counts physical rows) and a GROUPS frame (counts peer groups of equal ORDER BY value). The interviewer is probing whether you know the third frame mode beyond ROWS and RANGE.

1. The source data — price_logs

event_timeprice(narrative)
2023-10-01 12:00:0010first reading
2023-10-01 12:05:0020tied timestamp — the edge case
2023-10-01 12:05:0020tied timestamp — same peer group
2023-10-01 12:10:0050last reading

The two 12:05:00 rows are peers — equal on the ORDER BY column. ROWS treats them as two separate units; GROUPS treats them as one. Watching the two formula columns side by side makes the distinction concrete.

2. The SQL — ROWS vs GROUPS, with a visible formula

SELECT event_time, price,
  SUM(price)                     OVER (ORDER BY event_time ROWS   BETWEEN 1 PRECEDING AND CURRENT ROW) AS rows_sum,
  STRING_AGG(price::text, ' + ') OVER (ORDER BY event_time ROWS   BETWEEN 1 PRECEDING AND CURRENT ROW) AS rows_formula,
  SUM(price)                     OVER (ORDER BY event_time GROUPS BETWEEN 1 PRECEDING AND CURRENT ROW) AS groups_sum,
  STRING_AGG(price::text, ' + ') OVER (ORDER BY event_time GROUPS BETWEEN 1 PRECEDING AND CURRENT ROW) AS groups_formula
FROM price_logs
ORDER BY event_time, price;

ROWS BETWEEN 1 PRECEDING AND CURRENT ROW spans this row and the one physical row before it. GROUPS BETWEEN 1 PRECEDING AND CURRENT ROW spans this peer group and the one peer group before it — and a peer group can contain several rows. STRING_AGG reuses the identical frame to print the operands.

The two 12:05:00 rows differ under ROWS (30 vs 40 — each sees a different physical predecessor) but are identical under GROUPS (both 50 — same two peer groups: {10} and {20,20}). That is the frame-mode difference made visible.

4. Strategic explanations

Three frame modes, not two — ROWS, RANGE, GROUPS

ROWS measures the frame in physical rows: "1 preceding" is exactly one row back. RANGE measures in value distance on the ordering column. GROUPS (SQL:2011) measures in peer groups: "1 preceding" is one whole group of tied rows back, however many rows that group holds. The tied 12:05:00 pair is the only data shape that distinguishes all three.

Why the two 12:05 rows diverge under ROWS

Under ROWS, row order within a tie group is positional. The first 12:05:00 row's "1 preceding" is the 12:00:00 row → 10+20=30. The second 12:05:00 row's "1 preceding" is the first 12:05:00 row → 20+20=40. The frame slides one physical row at a time, so tied rows get different sums — and which row is "first" depends on the ORDER BY tiebreaker, so add one for determinism.

Why GROUPS gives both 12:05 rows the same answer

GROUPS collapses tied rows into one unit. For either 12:05:00 row, the frame is "this peer group {20,20} plus one group back {10}" — all three values, 10+20+20=50. Every row in a peer group gets the identical frame and identical result. Choose GROUPS when "the previous N distinct timestamps" is the business unit, regardless of how many events landed on each.

STRING_AGG and explicit casts

STRING_AGG concatenates text, so price (an INT) needs ::text — Postgres will not implicitly coerce the argument and otherwise raises a type error. Using STRING_AGG as a window function (with OVER) is a neat debugging trick: it prints the exact operands a numeric aggregate is consuming, so you can audit a frame without guessing.

Dialect notes

GROUPS is the newest frame mode: PostgreSQL 11+, SQLite 3.28+, DuckDB, and Oracle support it; SQL Server does not (only ROWS/RANGE), and MySQL 8 omits it too. On those engines, emulate GROUPS by ranking peer groups with DENSE_RANK() and framing over RANGE of that rank. STRING_AGG is GROUP_CONCAT in MySQL/SQLite and LISTAGG in Oracle.

Senior signal: know all three window frame modes — ROWS counts physical rows, RANGE counts value distance, GROUPS counts peer groups of tied rows — and pick deliberately when the data has ties; use a STRING_AGG window column to make a frame's operands visible while debugging.
