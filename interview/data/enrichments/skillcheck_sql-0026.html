A colleague reports "logins per user" with a query that fans out across two joined tables, and the numbers come back inflated — user 1 shows 6 logins when they only logged in twice. This is the classic join fan-out bug. The interviewer is probing whether you can diagnose why joining two unrelated one-to-many tables multiplies rows, and whether you know the fix is COUNT(DISTINCT ...) on a primary key — or, better, pre-aggregation.

1. The source data — users, logins, sessions

tablerows(narrative)
users(1, Alice), (2, Bob)two users
logins(100,1), (101,1), (102,2)Alice has 2 logins, Bob has 1
sessions(500,1), (501,1), (502,1), (503,2)Alice has 3 sessions, Bob has 1

The edge case is Alice: she has 2 logins and 3 sessions — two different multiplicities. Joining both onto her one user row produces 2×3 = 6 combined rows, and that 6 is what a naive COUNT(*) reports as her login count. Bob, with 1 and 1, produces 1×1 = 1 and looks deceptively fine — the bug hides on single-child users.

2. The SQL — the broken query and the fix

Original (flagged broken — row fan-out inflates the counts):

SELECT u.id, u.name,
       COUNT(*)              AS logins,
       COUNT(DISTINCT s.id)  AS sessions
FROM users u
LEFT JOIN logins   l ON u.id = l.user_id
LEFT JOIN sessions s ON u.id = s.user_id
GROUP BY u.id, u.name;

Corrected — count distinct primary keys so the fan-out cannot inflate:

SELECT u.id, u.name,
       COUNT(DISTINCT l.id) AS logins,
       COUNT(DISTINCT s.id) AS sessions
FROM users u
LEFT JOIN logins   l ON u.id = l.user_id
LEFT JOIN sessions s ON u.id = s.user_id
GROUP BY u.id, u.name
ORDER BY u.id;

The two LEFT JOINs build a Cartesian product of Alice's logins and sessions. COUNT(*) counts those product rows; COUNT(DISTINCT l.id) counts distinct login primary keys, so each real login is tallied once no matter how many session rows it was paired with.

3. The resulting analysis

idnamelogins (broken)logins (fixed)sessions
1Alice623

The broken query gave Alice logins = 6 — her 2 logins times her 3 sessions. The corrected query gives 2: COUNT(DISTINCT l.id) sees login ids {100, 101} however many times the join duplicated them. Bob reads 1 in both — 1×1 hides nothing, which is exactly why fan-out bugs survive code review.

4. Strategic explanations

Why joining two one-to-many tables multiplies rows

users joins logins one-to-many (Alice → 2 rows), then that intermediate result joins sessions one-to-many (each of those 2 rows → 3 rows). The join is a Cartesian product within each user: 2×3 = 6 rows for Alice. Neither child table is at fault — the bug is fanning out two independent dimensions in one query. COUNT(*) then counts product rows, not entities. The mental model: every additional one-to-many join multiplies, it does not add.

Why COUNT(DISTINCT pk) repairs it — and its limit

Each logins row has a unique id. The fan-out duplicates whole login rows — id 100 appears 3 times, once per session — so COUNT(DISTINCT l.id) collapses the duplicates and recovers the true count of 2. The key word is primary key: COUNT(DISTINCT login_ts) would be wrong if two logins shared a timestamp. The limit: COUNT(DISTINCT) fixes counts, but SUM over a fanned-out column is still inflated and DISTINCT cannot save it — you would sum duplicated values.

The better fix — pre-aggregate before joining

COUNT(DISTINCT) patches the symptom; the cleaner design removes the fan-out entirely by aggregating each child before the join:

SELECT u.id, u.name,
       COALESCE(l.cnt, 0) AS logins,
       COALESCE(s.cnt, 0) AS sessions
FROM users u
LEFT JOIN (SELECT user_id, COUNT(*) AS cnt FROM logins   GROUP BY user_id) l ON l.user_id = u.id
LEFT JOIN (SELECT user_id, COUNT(*) AS cnt FROM sessions GROUP BY user_id) s ON s.user_id = u.id
ORDER BY u.id;

Each subquery yields one row per user, so the joins are now one-to-one — no Cartesian product, no DISTINCT needed, and crucially SUMs would be correct too. This is the answer that signals seniority: fix the structure, not just the count.

The LEFT JOIN + COUNT(*) NULL trap

There is a second latent bug. COUNT(*) counts rows, including rows a LEFT JOIN manufactured for a user with no children — a user with zero logins would still get COUNT(*) = 1 from the NULL-padded row. COUNT(DISTINCT l.id) avoids this too: COUNT ignores NULLs, so a childless user correctly counts 0. Whenever you COUNT after a LEFT JOIN, count a non-null column from the joined side, never *.

How to catch fan-out in review

The tell is two or more one-to-many joins feeding one aggregate. Sanity-check by counting each child independently — SELECT COUNT(*) FROM logins WHERE user_id = 1 — and compare; a mismatch screams fan-out. The inflation factor equals the product of the sibling cardinalities (here 3, Alice's session count). Single-child users mask it, so always test with a user that has multiple rows in both tables.

Senior signal: recognise inflated counts as join fan-out — two one-to-many joins form a Cartesian product per group; patch it with COUNT(DISTINCT primary_key), but prefer pre-aggregating each child to one row per user so the joins become one-to-one (which also keeps SUMs correct); and never COUNT(*) after a LEFT JOIN, since it counts NULL-padded rows.
