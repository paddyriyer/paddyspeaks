An FP&A team wants a quarterly board summary for 2025: "one row per customer, with columns Q1 through Q4 totalling that customer's sales in each calendar quarter." That shape — quarters across the top — is a pivot. The interviewer is probing whether you can pivot in portable SQL, scope the data to one year, and make empty quarters read 0 rather than NULL.

1. The source data — sales

Two edge cases. Customer 4's only sale is in 2024 — the whole customer should vanish from the report. Customers 1 and 3 have empty quarters that must show 0, not NULL.

2. The SQL — pivot via FILTERed aggregates

SELECT customer_id,
       COALESCE(SUM(amount) FILTER (WHERE EXTRACT(QUARTER FROM sale_date) = 1), 0) AS q1,
       COALESCE(SUM(amount) FILTER (WHERE EXTRACT(QUARTER FROM sale_date) = 2), 0) AS q2,
       COALESCE(SUM(amount) FILTER (WHERE EXTRACT(QUARTER FROM sale_date) = 3), 0) AS q3,
       COALESCE(SUM(amount) FILTER (WHERE EXTRACT(QUARTER FROM sale_date) = 4), 0) AS q4
FROM sales
WHERE sale_date >= DATE '2025-01-01' AND sale_date < DATE '2026-01-01'
GROUP BY customer_id
ORDER BY customer_id;

The WHERE clause restricts to 2025. One SUM(amount) FILTER (WHERE quarter = n) per output column adds only that quarter's sales. COALESCE(..., 0) turns an empty quarter's NULL into 0.

4. Strategic explanations

FILTER vs SUM(CASE WHEN ...)

The aggregate(...) FILTER (WHERE ...) clause is the modern SQL-standard way to conditionally aggregate. SUM(amount) FILTER (WHERE quarter = 1) is exactly equivalent to SUM(CASE WHEN quarter = 1 THEN amount END) but reads more clearly — the condition is visibly a row filter, not part of the value. Both produce NULL when no row qualifies, which is why the COALESCE is still needed.

Why COALESCE is required — the NULL vs 0 choice

When a customer has no sales in a quarter, SUM over zero rows returns NULL, not 0 — SUM of an empty set is undefined. The question explicitly wants 0 ("not NULL"), so each column is wrapped in COALESCE(..., 0). This is a deliberate business decision: a financial grid wants 0 (the customer genuinely sold nothing), whereas a coverage report might keep NULL to distinguish "no activity" from "real zero". State which you mean.

The half-open date range

sale_date >= '2025-01-01' AND sale_date < '2026-01-01' is the robust way to scope a year. It is preferable to BETWEEN '2025-01-01' AND '2025-12-31' because if sale_date were a timestamp, a sale at 2025-12-31 14:00 is > the date 2025-12-31 and BETWEEN would silently drop it. The half-open [start, next-year-start) pattern is correct for both date and timestamp columns — a senior habit.

Duplicate sales fold in by design

Customer 1's two Q1 rows are not an error to dedupe — SUM over the filtered group adds every matching row (100 + 50 = 150). That is the correct quarterly total. If the question had asked "how many quarters had activity" you would switch to COUNT(*) FILTER (...) or a distinct count. The aggregate you pick encodes the business rule.

The columns are static — the hard limit

Four quarters are hard-coded, which is fine because a year always has exactly four. But pure SQL pivots cannot grow columns from data — if the request were "one column per month present in the data", you would need dynamic SQL (build the statement text from SELECT DISTINCT, then execute) or PostgreSQL's crosstab() from the tablefunc extension. Snowflake/Oracle/SQL Server also offer a dedicated PIVOT operator, but it still makes you enumerate the values. Naming the static-column limitation unprompted is the senior signal.

Senior signal: pivot portably with one SUM(amount) FILTER (WHERE quarter = n) per column, wrap each in COALESCE(..., 0) because SUM of zero rows is NULL not 0, scope the year with a half-open >= start AND < next-year range (timestamp-safe), and know the four columns are static — a data-driven pivot needs dynamic SQL or crosstab().
