An HR analytics team wants the median salary per job title, not the mean — medians resist distortion from one outsized executive package. SQL has no portable MEDIAN aggregate, so the reference solution computes it from first principles: number the rows within each title, then average the one or two rows that straddle the middle. The interviewer is probing whether you understand the odd/even arithmetic and the single-member edge case.

Three group sizes on purpose: Engineer has 4 rows (even), Manager has 3 rows (odd), and Analyst has a single row. The single-member group is the edge case — its median must equal that one salary.

2. The SQL — median via ROW_NUMBER and COUNT

WITH ranked AS (
  SELECT job_title, salary,
    ROW_NUMBER() OVER (PARTITION BY job_title ORDER BY salary) AS rn,
    COUNT(*)    OVER (PARTITION BY job_title)                  AS n
  FROM employees
)
SELECT job_title, AVG(salary) AS median_salary
FROM ranked
WHERE rn IN ((n+1)/2, (n+2)/2)
GROUP BY job_title
ORDER BY job_title;

Engineer's four salaries have no single middle, so the median is the average of the two central values, (120k+140k)/2 = $130k. Manager's three salaries have a clean middle at $220k. Analyst's single row averages to itself.

4. Strategic explanations

The integer-division trick — how (n+1)/2 and (n+2)/2 pick the middle

This is the clever core. With n an integer, division truncates. For an odd count (n=3): (3+1)/2 = 2 and (3+2)/2 = 2 — both expressions equal 2, so rn IN (2,2) selects exactly the one middle row. For an even count (n=4): (4+1)/2 = 2 and (4+2)/2 = 3 — the two expressions differ, selecting rows 2 and 3, the central pair. One formula handles both parities with no CASE. For the single-row group (n=1): (1+1)/2 = 1 and (1+2)/2 = 1 — row 1 only, and AVG of one value is that value.

Why two window functions in the same CTE

ROW_NUMBER() orders rows within each title so position has meaning; COUNT(*) OVER (PARTITION BY job_title) attaches the group size n to every row, because the WHERE clause needs n available on the rows it filters. Computing n as a window aggregate (not a separate GROUP BY) avoids an extra join. The final AVG ... GROUP BY then collapses the one or two surviving rows per title into the median.

NULL salaries silently shift the median

ROW_NUMBER still numbers a NULL-salary row, but COUNT(*) counts it too — so a NULL salary inflates n and can drag a NULL into the selected middle, where AVG then ignores it and the result is subtly wrong. The robust fix is WHERE salary IS NOT NULL before the window, or COUNT(salary) instead of COUNT(*) so the denominator reflects only real salaries. Calling this out unprompted is the senior signal.

The modern alternative — PERCENTILE_CONT

Postgres, Oracle, SQL Server, BigQuery, and Snowflake expose an ordered-set aggregate that computes the median directly:

SELECT job_title,
  PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) AS median_salary
FROM employees
GROUP BY job_title;

PERCENTILE_CONT interpolates between the two middle values (same answer here), while PERCENTILE_DISC returns an actual data point. Prefer it where available — it is clearer and less error-prone. The ROW_NUMBER manual method is the portable fallback (older MySQL has no percentile aggregate).

Tie ordering is harmless here

If two engineers earned the same salary, ROW_NUMBER would order them arbitrarily — but since the median averages by value, the result is unaffected. A tiebreaker in ORDER BY only matters if you need reproducible row numbers for another purpose.

Senior signal: explain that integer-truncating (n+1)/2 and (n+2)/2 collapse to the same row for odd counts and straddle the middle pair for even counts — one formula, no CASE — then note NULL salaries corrupt n unless filtered, and offer PERCENTILE_CONT(0.5) as the modern one-liner.
