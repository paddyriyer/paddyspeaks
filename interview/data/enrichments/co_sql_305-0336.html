Google's most-watched ML signal: CTR by SERP position. Position 1 sees 30%+ CTR; position 10 sees under 1%; the curve is the headline product metric. The query is per-position aggregation; the senior conversation is what makes the curve flatten or steepen.

2. The SQL — straight per-position rate

SELECT position,
  COUNT(*) AS impressions,
  SUM(clicked) AS clicks,
  ROUND(100.0 * SUM(clicked) / NULLIF(COUNT(*), 0), 2) AS ctr_pct
FROM search_results
GROUP BY position
ORDER BY position;

This curve is the basis of every SEO/SEM bidding model. Cite it from memory.

The "feature" pollution

Google's SERP isn't 10 organic results anymore — it has Knowledge Panels, Featured Snippets, Maps, Shopping, Ads. These crowd organic results down, so "position 5 in 2010" and "position 5 in 2024" mean very different real-estate. Always join to serp_features and exclude when the analysis assumes pure organic.

Mobile vs desktop

Mobile screens show 3-4 results above the fold; desktop shows 5-7. Position-3 CTR diverges between devices. Always slice:

GROUP BY position, device_type
Branded vs generic queries

Branded queries (someone searched "facebook") have an extreme position-1 spike (often 95%+ CTR) because the user knew what they wanted. Generic queries (someone searched "social media") spread clicks across positions. Average CTR by position confounds these. Split by query intent.

The "pogo-sticking" follow-up

A click that returns to the SERP within 5 seconds is a bounce — the user clicked but found nothing useful. Production CTR analysis filters these out for the "satisfied click" metric:

SUM(CASE WHEN clicked = 1 AND time_on_target_seconds >= 30 THEN 1 ELSE 0 END)
The CTR-prediction model

The query result is the training-data validation for the ad-ranker's predicted CTR. If observed CTR at position 3 deviates from the model's predicted CTR, the model retrains. The query is a feedback-loop input.
