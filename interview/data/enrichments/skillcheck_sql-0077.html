A finance team wants a year-over-year growth report per product: each year's total spend, the prior year's total, and the percentage change. The interviewer is testing the LAG window function for fetching a previous period's value, plus two classic traps — the first year of any product has no prior to compare against, and a zero prior would divide by zero.

The edge cases are the first-year rows. Product 1's earliest year is 2023 — no 2022 to compare. Product 2 debuts in 2024. Product 3 exists in only 2025 and has no prior year at all. Each of these must appear in the output with prev_year_spend and yoy_growth as NULL, not omitted.

2. The SQL — yearly rollup, then LAG over years per product

WITH yearly AS (
  SELECT EXTRACT(YEAR FROM occurred_at)::int AS year,
         product_id,
         SUM(spend) AS curr_year_spend
  FROM txn
  GROUP BY EXTRACT(YEAR FROM occurred_at), product_id
)
SELECT year, product_id, curr_year_spend,
       LAG(curr_year_spend) OVER (PARTITION BY product_id ORDER BY year) AS prev_year_spend,
       ROUND(
         (curr_year_spend - LAG(curr_year_spend) OVER (PARTITION BY product_id ORDER BY year))
         / NULLIF(LAG(curr_year_spend) OVER (PARTITION BY product_id ORDER BY year), 0) * 100,
         2) AS yoy_growth
FROM yearly
ORDER BY year, product_id;

yearly rolls transactions up to one row per (product, year). LAG(curr_year_spend) OVER (PARTITION BY product_id ORDER BY year) reaches back to the previous year's total for the same product. The growth formula subtracts, divides by the prior total, and scales by 100 — with NULLIF(..., 0) guarding the denominator.

4. Strategic explanations

Why LAG partitioned by product is the right reach-back

Year-over-year means comparing a row to the same product's previous year. LAG(curr_year_spend) OVER (PARTITION BY product_id ORDER BY year) does exactly that: PARTITION BY product_id isolates each product's timeline, ORDER BY year defines "previous", and LAG with the default offset of 1 fetches the immediately prior year. No self-join on year = year - 1 is needed — and LAG is robust even if a product skips a year (it returns the previous row, not literally year minus one).

Why first-year rows naturally become NULL

LAG returns NULL when there is no preceding row in the partition — precisely the situation for each product's earliest year. The arithmetic then propagates that NULL: curr - NULL is NULL, and NULL / anything is NULL, so yoy_growth is NULL too. The query needs no special CASE for the first year — SQL's three-valued logic handles it. This is why the rows still appear instead of being filtered out: nothing removes them.

NULLIF(prev, 0) — the divide-by-zero guard

If a product had a real prior-year total of 0, dividing by it would raise a division-by-zero error and abort the whole query. NULLIF(LAG(...), 0) converts a zero denominator into NULL, so the division yields NULL instead of crashing. It is defensive: the sample data has no zero year, but a robust YoY query must assume one could occur. Note that "growth from zero" is mathematically undefined anyway — NULL is the honest answer.

The cost of repeating LAG three times

The query writes LAG(curr_year_spend) OVER (...) three times — once for the column, twice in the formula. It is correct (the optimizer typically computes the window once), but a cleaner form wraps yearly in a second CTE that materialises prev_year_spend as a named column, then the growth formula references that name. That removes the repetition, makes the divide-by-zero guard read as NULLIF(prev_year_spend, 0), and is the refactor a senior reviewer would suggest.

Dialect — year extraction and rounding

EXTRACT(YEAR FROM occurred_at) is standard and portable. SQL Server prefers YEAR(occurred_at) or DATEPART(year, ...); MySQL also has YEAR(). LAG and NULLIF are ANSI and work everywhere with window-function support. One caveat: integer division — if spend were an integer type, the growth ratio could truncate before * 100; here spend is NUMERIC so the division is exact, but multiplying by 100.0 would be a safe habit.

Senior signal: compute period-over-period change with LAG(...) OVER (PARTITION BY entity ORDER BY period) — it fetches the prior value without a self-join and returns NULL for the first period, so debut rows survive with null growth automatically. Always wrap the denominator in NULLIF(prev, 0) to turn a divide-by-zero into a clean NULL, and factor the repeated LAG into a named column for readability.
