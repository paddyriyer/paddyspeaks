Spotify's growth team separates two very different events: a brand-new user signing up, and a lapsed user coming back. The second is a "reactivation", and it is what retention campaigns are measured against. The definition hinges on a gap: how long must a user be silent before their next play counts as a reactivation? The reference solution picks "more than 30 days", and the interview is about whether you treat that threshold — and the day-arithmetic behind it — deliberately.

User 1's Jan 5→Jan 20 gap is 15 days — not a reactivation; Jan 20→Mar 10 is 49 days — a reactivation in March. User 2's Jan 8→Feb 25 gap is 48 days — reactivation in February. User 3 has a single play, so no gap exists at all (the edge case). User 4's Feb 1→Apr 15 gap is 73 days — reactivation in April.

2. The SQL — LAG gap > 30 days, counted per month

WITH gaps AS (
  SELECT user_id, action_date,
    LAG(action_date) OVER (PARTITION BY user_id ORDER BY action_date) AS prev
  FROM (SELECT DISTINCT user_id, action_date FROM activity) d
)
SELECT DATE_TRUNC('month', action_date)::DATE AS month, COUNT(DISTINCT user_id) AS reactivated
FROM gaps
WHERE (action_date - prev) > 30
GROUP BY DATE_TRUNC('month', action_date)::DATE;

The inner SELECT DISTINCT collapses multiple actions on the same day, LAG fetches each user's previous active date, and the filter keeps rows where more than 30 days elapsed. The outer query buckets those reactivation events by month.

January has no reactivations because every January play is a user's first. The answer is correct for this data; the discussion below is about where the definition could bend.

4. Strategic explanations

action_date - prev is integer day arithmetic — know the type

Subtracting two DATE values in Postgres yields an integer number of days, so > 30 compares cleanly. The same expression on TIMESTAMP columns would yield an interval, and interval > 30 is a type error — you would need > INTERVAL '30 days'. Always check whether the column is a date or a timestamp before writing the gap comparison; this is a frequent silent break when the schema changes under you.

The > 30 boundary is a product decision

Why 30 and not 28, or "one calendar month", or 60? A gap of exactly 30 days is currently not a reactivation (>, not >=). Spotify might define dormancy as a full calendar month, in which case January 5 to February 5 should count regardless of day count. State the threshold as an assumption and ask. The SQL is trivial to change; the wrong definition silently miscounts the headline retention number every month.

The single-action edge case is handled — say so

User 3 has exactly one play. LAG over a one-row partition returns NULL, the gap is NULL, and NULL > 30 is unknown, so the WHERE drops it. That is the correct outcome — a user who has only ever played once cannot have "come back". But it is correct by accident of NULL semantics; an interviewer wants you to point at it and confirm it is intended, not leave it implicit.

COUNT(DISTINCT user_id) guards against a double-count

If a user reactivated twice in the same month — lapsed, returned, lapsed again, returned again — they would produce two qualifying rows in that month. COUNT(DISTINCT user_id) counts them once; a plain COUNT(*) would count the user twice. Which is right depends on the metric: "how many users reactivated" wants DISTINCT; "how many reactivation events occurred" wants COUNT(*). Name the metric.

Gap months and ordering

The output only has rows for months that contained a reactivation — a month with zero reactivations is simply absent. For a dashboard you would left-join against a month spine so zero months show as 0 rather than vanishing. Also note there is no ORDER BY: GROUP BY does not guarantee sorted output, so add ORDER BY month if the consumer expects chronological rows.

The window must be partitioned

LAG(…) OVER (PARTITION BY user_id ORDER BY action_date) — the PARTITION BY user_id is essential. Without it, LAG would compare one user's first play to a different user's last play, manufacturing gaps that mean nothing. Per-user time series almost always need to partition by the entity before any LAG/LEAD/running aggregate.

Senior signal: state the >30-day threshold as a negotiable product definition, know that date - date returns an integer (and timestamp - timestamp an interval that needs different syntax), confirm the single-action user is correctly excluded via NULL semantics, and justify COUNT(DISTINCT user_id) over COUNT(*) by naming whether the metric is users or events.
