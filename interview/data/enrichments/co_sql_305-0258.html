Snap's product-research team watches customer satisfaction trend by week. The ask is a one-liner: average rating, bucketed weekly. The query is genuinely short, but it carries two senior-grade subtleties — what "week" actually means in Postgres, and the gap between projecting a truncated key and grouping by it.

1. The source data — feedback

feedback_daterating
2025-03-035
2025-03-053
2025-03-064
2025-03-102
2025-03-124
2025-03-175

March 3, 10 and 17, 2025 are all Mondays. Postgres weeks start on Monday, so these three dates each open a fresh week — the data deliberately straddles three weekly buckets so the grouping is visible.

2. The SQL — AVG(rating) by week

SELECT DATE_TRUNC('week', feedback_date)::date AS wk, AVG(rating) AS avg_rating
FROM feedback GROUP BY DATE_TRUNC('week', feedback_date);

DATE_TRUNC('week', …) snaps each date back to the Monday that begins its week. Rows sharing a Monday collapse into one group, and AVG averages the ratings within each.

3. The resulting analysis

wkavg_rating
2025-03-103.0000000000000000
2025-03-175.0000000000000000
2025-03-034.0000000000000000

Week of Mar 3: ratings 5, 3, 4 average to 4.0. Week of Mar 10: 2 and 4 average to 3.0. Week of Mar 17: a lone 5. Satisfaction dipped mid-month then recovered — but notice the rows came back in no chronological order: there is no ORDER BY.

4. Strategic explanations

The missing ORDER BY — a trend out of order

A weekly trend that returns Mar 10, Mar 17, Mar 3 is not a trend a human can read. Always sort the result:

... GROUP BY DATE_TRUNC('week', feedback_date)
    ORDER BY wk;

The query is correct without it, but for a trend report the omission is a real defect, not a nitpick.

What "week" means — the ISO vs locale trap

DATE_TRUNC('week', …) in Postgres always uses the ISO week: weeks start Monday. If the business reports Sunday-start weeks (common in US retail and ad-tech), this query is silently off by a day — a Sunday's feedback lands in the previous ISO week. To force a Sunday start, shift the date before and after truncation:

(DATE_TRUNC('week', feedback_date + INTERVAL '1 day') - INTERVAL '1 day')::date AS wk

Confirm the calendar convention before writing the query; "week" is not universal.

Projecting vs grouping — why this still works

The SELECT casts the key to ::date; the GROUP BY does not. They still match because the cast is a deterministic function of the grouped expression — Postgres allows a projected column that is functionally dependent on a GROUP BY term. Cleaner is to group by the exact alias or expression you project, so reviewers don't have to reason about functional dependence:

SELECT DATE_TRUNC('week', feedback_date)::date AS wk, AVG(rating) AS avg_rating
FROM feedback GROUP BY 1 ORDER BY 1;
AVG and NULL ratings

AVG(rating) skips NULL ratings — a customer who left a comment but no star score is dropped from both numerator and denominator. Usually fine, but if a missing rating should count as a sentiment signal, decide explicitly. Also: AVG over a week with only NULL ratings returns NULL, not 0.

Empty weeks — the gap the GROUP BY can't show

If a week receives zero feedback, it produces no row at all — the trend line has an invisible hole, and a chart will draw a straight line across it as if nothing happened. To make quiet weeks visible, generate_series a full week spine and LEFT JOIN the aggregated feedback onto it.

Senior signal: add the ORDER BY wk a trend needs, know that DATE_TRUNC('week') is ISO/Monday-start and adjust if the business reports Sunday weeks, group by the exact projected expression rather than relying on functional dependence, and surface zero-feedback weeks with a calendar spine so the trend has no invisible gaps.
