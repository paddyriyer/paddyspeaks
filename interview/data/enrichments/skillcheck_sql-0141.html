A registrar wants the students who have completed every required course — not merely enrolled, but passed: a grade that is present and not an F. This is relational division with a predicate twist: you divide by the course catalog, but only enrollments that count as a genuine pass should feed the division. The interviewer is probing whether you filter before counting, and whether you handle the NULL "in progress" grade correctly.

1. The source data — course and enrollment

The catalog (course) has three required courses. Student 2 has an F in Physics; student 3 has a NULL (still in progress) in Physics — two distinct ways to not complete a course. Student 5 has only one enrollment. Only students 1 and 4 have a genuine pass in all three.

2. The SQL — filter to passes, then divide

SELECT student_id
FROM enrollment
WHERE grade IS NOT NULL AND grade <> 'F'
GROUP BY student_id
HAVING COUNT(DISTINCT course_id) = (SELECT COUNT(*) FROM course)
ORDER BY student_id;

The WHERE keeps only rows that count as a pass — grade present (IS NOT NULL) and not an F. GROUP BY student_id then buckets the survivors, and HAVING COUNT(DISTINCT course_id) = (SELECT COUNT(*) FROM course) keeps students whose passed-course set covers the whole catalog.

Students 2, 3, 5 are absent. Student 2's F-row is removed by grade <> 'F', leaving only 2 passed courses. Student 3's NULL-row is removed by grade IS NOT NULL, also leaving 2. Student 5 only ever had one enrollment. Each falls short of the count of 3.

4. Strategic explanations

Filter first, then count — the WHERE defines "completed"

The whole correctness of this division hinges on filtering before the aggregate. The WHERE clause removes non-passing rows, so by the time COUNT(DISTINCT course_id) runs it is counting only completed courses. A student who enrolled in all three but failed one is left with two passing rows, and 2 <> 3 excludes them. Putting the grade test anywhere else — or omitting it — would count mere enrollment as completion, which the spec explicitly rejects.

grade IS NOT NULL — why <> 'F' alone is not enough

This is the NULL trap. grade <> 'F' evaluates to unknown when grade is NULL, and WHERE discards unknown rows — so it might seem the explicit IS NOT NULL is redundant. It is not redundant conceptually: it documents intent and guards against any engine or rewrite where the NULL handling differs. More importantly it makes the predicate's two cases explicit — "in progress" (NULL) and "failed" (F) are both non-completion, and the code should say so. Relying on three-valued logic implicitly is fragile; stating IS NOT NULL AND grade <> 'F' is self-documenting and unambiguous.

Why COUNT(DISTINCT course_id), not COUNT(*)

If a student could have two passing rows for the same course (a retake, say), COUNT(*) would over-count and a student missing a course could be wrongly admitted. COUNT(DISTINCT course_id) measures how many distinct required courses were passed — the coverage, not the row volume. The fixture has one row per student/course, so the two would agree here, but DISTINCT is the defensively correct choice for a division and costs nothing in correctness.

The subset assumption — courses must be required

The count-equality test works because a student's passed courses are assumed to be a subset of the required catalog: subset cardinality equals full-set cardinality only when the subset is the full set. If enrollment could reference an elective not in course, that passed elective would inflate the count and could falsely qualify a student. To be safe against that you would JOIN enrollment TO course on course_id first, so only required courses are counted. With a clean schema the simple form holds.

The double-NOT EXISTS alternative and the empty catalog

The rigorous division form is "no required course exists that the student has not completed", a nested NOT EXISTS over course and a passing-enrollment subquery. It handles the empty-catalog edge case correctly (every student qualifies when there are no required courses), whereas the counting form returns nobody if course is empty, since COUNT(DISTINCT course_id) >= 1 can never equal 0. With a non-empty catalog the counting form is simpler and correct; knowing the divergence is the senior nuance.

Senior signal: push the "is this a pass" predicate into WHERE so the division counts only completed courses, write grade IS NOT NULL AND grade <> 'F' explicitly so the NULL "in progress" and F "failed" cases are both visibly excluded, use COUNT(DISTINCT course_id) for coverage, and note the subset assumption and empty-catalog behaviour that separate the counting form from rigorous double-NOT EXISTS division.
