Revolut's resourcing team wants a headcount-utilisation view: for every employee, how many projects are they assigned to — including the people assigned to none, because bench capacity is the whole point. The query is a join plus a count, but the interviewer is watching whether you pick a LEFT JOIN over an inner one, and whether you count the right column so the unassigned employees show 0 rather than 1.

Ada is on two projects, Bao and Cleo on one each — and Dev has no assignment row at all. That benched employee is the interview: a correct utilisation report must show Dev with a count of 0, not omit him.

2. The SQL — LEFT JOIN + COUNT(column)

SELECT e.emp_id, e.name, COUNT(pa.project_id) AS projects
FROM employees e LEFT JOIN project_assignments pa ON e.emp_id = pa.emp_id
GROUP BY e.emp_id, e.name;

The LEFT JOIN keeps every employee even with no matching assignment; COUNT(pa.project_id) tallies their assignment rows per employee.

4. Strategic explanations

Why COUNT(pa.project_id) and not COUNT(*)

This is the single decision the question turns on. After the LEFT JOIN, Dev still produces one output row — with every project_assignments column NULL. COUNT(*) counts rows, so it would count that phantom row and report Dev as 1 — a benched employee wrongly shown as staffed. COUNT(pa.project_id) counts only non-NULL values, so Dev's NULL is skipped and he correctly shows 0. With a LEFT JOIN, always count a column from the right table.

LEFT JOIN vs INNER JOIN — the visible difference

Swap to an inner join and Dev vanishes entirely — no assignment row, no match, no output. For a utilisation report that is a real bug: the bench (your spare capacity) is precisely the rows you most need. The inner join would be correct only if the ask were "employees who are on a project." Read the requirement: "every employee, including zero" demands the outer join.

The double-count trap with DISTINCT

COUNT(pa.project_id) counts assignment rows. If the same employee can be assigned to the same project twice (a data-quality slip, or one row per project-role), that inflates the count. If "how many projects" means distinct projects, harden it:

SELECT e.emp_id, e.name, COUNT(DISTINCT pa.project_id) AS projects
FROM employees e LEFT JOIN project_assignments pa ON e.emp_id = pa.emp_id
GROUP BY e.emp_id, e.name;

COUNT(DISTINCT ...) still ignores NULLs, so Dev stays at 0. Confirm the assignment grain before choosing.

Grouping by the key, carrying the name

GROUP BY e.emp_id, e.name groups by the primary key and carries name along. Grouping by name alone would merge two different employees who happen to share a name into one row — group by the id. (Postgres also lets you GROUP BY e.emp_id alone and still select e.name, because emp_id is the table's primary key and functionally determines it; listing both is the portable habit.)

Finding the bench — the natural follow-up

"Show only unassigned employees" filters this with HAVING, not WHERE:

... GROUP BY e.emp_id, e.name HAVING COUNT(pa.project_id) = 0;

Equivalently, an anti-join: WHERE NOT EXISTS (SELECT 1 FROM project_assignments pa WHERE pa.emp_id = e.emp_id). Both isolate Dev; the anti-join skips the aggregation entirely and often plans better.

Senior signal: choose LEFT JOIN so benched employees are not silently dropped, count pa.project_id rather than COUNT(*) so a NULL-extended row reads as 0 instead of 1, reach for COUNT(DISTINCT ...) if assignments can repeat, and know that finding the bench is a HAVING ... = 0 or an anti-join.
