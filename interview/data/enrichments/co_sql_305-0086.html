Walmart's customer-analytics team wants a list of high-value customers for a loyalty campaign — specifically, anyone whose lifetime spend tops $1,000. The data is one row per order, so the work is: aggregate to the customer, then keep only the customers whose total clears the bar. The interview hinges on where that "clears the bar" filter goes — WHERE or HAVING.

Customer 104 is the edge case: no single order exceeds $1,000 (each is $400), yet three orders sum to $1,200. Any solution that filters on individual order amounts would wrongly drop them.

2. The SQL — GROUP BY + HAVING SUM

SELECT customer_id, COUNT(*) AS orders, SUM(amount) AS total
FROM orders
GROUP BY customer_id
HAVING SUM(amount) > 1000;

GROUP BY customer_id rolls orders up per customer; HAVING SUM(amount) > 1000 tests the aggregated total and keeps only the groups that pass.

4. Strategic explanations

WHERE vs HAVING — the central distinction

WHERE filters rows, before grouping. HAVING filters groups, after aggregation. You cannot put an aggregate in WHERE — WHERE SUM(amount) > 1000 is a syntax error, because at the row stage no SUM exists yet. The threshold is on a per-customer total, which only exists post-aggregation, so it must be HAVING. This is the single most important point of the question.

Why a WHERE amount > 1000 answer is wrong

A tempting but incorrect attempt filters individual orders first:

SELECT customer_id, COUNT(*), SUM(amount)
FROM orders
WHERE amount > 1000           -- WRONG: tests each order, not the total
GROUP BY customer_id;

This keeps only orders individually over $1,000, then sums those survivors. Customer 104 (three $400 orders) contributes nothing and vanishes; customer 101 (orders of 600 and 700) also vanishes. The result would list only customer 103 — a badly truncated campaign list. The filter must apply to the sum, not the parts.

WHERE and HAVING together — both have a job

They are not rivals; a real query often uses both. WHERE trims rows before aggregating (cheaper, and changes what goes into the sum); HAVING trims groups after. Counting only completed orders toward lifetime value:

SELECT customer_id, COUNT(*) AS orders, SUM(amount) AS total
FROM orders
WHERE status = 'completed'      -- row filter: which orders count
GROUP BY customer_id
HAVING SUM(amount) > 1000;       -- group filter: which customers qualify

The schema ships a status column — flagging that cancelled/refunded orders probably should not count toward "high value" is the senior observation.

Strict > vs >= on the boundary

HAVING SUM(amount) > 1000 excludes a customer whose total is exactly $1,000. "Tops $1,000" is ambiguous — if the campaign means "$1,000 or more", it must be >= 1000. With this dataset no total lands exactly on 1000 so it makes no visible difference, but confirm the boundary intent rather than guessing.

The NULL-amount edge

SUM ignores NULL amount values, so an order with a NULL amount silently contributes 0 to the total — usually fine, but if a NULL should disqualify or flag a customer, handle it explicitly. And a customer all of whose orders have NULL amounts gets SUM = NULL; NULL > 1000 is unknown, so they are excluded — the desired outcome here, but worth stating you know why.

Senior signal: articulate that HAVING filters post-aggregation groups while WHERE filters pre-aggregation rows, recognise that a per-order WHERE amount > 1000 wrongly drops customers who clear the bar via many small orders, and use WHERE and HAVING together — e.g. a status row filter alongside the SUM group filter.
