A digital-health platform wants to flag high-risk patients: those whose 7-day average heart rate is elevated and whose 7-day average step count is low. The interviewer is probing whether you can compute two rolling averages over a sliding window and combine the conditions — and whether you guard against the messy, dirty-typed activity data that real wearable feeds always carry.

2. The SQL — sanitise types, then dual rolling averages

WITH safe AS (
  SELECT patient_id, on_date,
    CASE WHEN steps::text ~ '^-?[0-9]+(\.[0-9]+)?$' THEN steps::numeric END AS steps,
    CASE WHEN heart_rate::text ~ '^-?[0-9]+(\.[0-9]+)?$' THEN heart_rate::numeric END AS heart_rate
  FROM patientactivity
),
patient_info AS (
  SELECT patient_id, on_date, steps, heart_rate,
    AVG(heart_rate) OVER (PARTITION BY patient_id ORDER BY on_date ASC
                          ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS seven_day_avg_hr,
    AVG(steps) OVER (PARTITION BY patient_id ORDER BY on_date ASC
                     ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS seven_day_avg_steps
  FROM safe
)
SELECT patient_id
FROM patient_info
WHERE seven_day_avg_hr > 100 AND seven_day_avg_steps < 4000
GROUP BY patient_id;

4. Strategic explanations

The defensive cast — regex-guarded type conversion

The steps and heart_rate columns are text. CASE WHEN steps ~ '^-?[0-9]+(\.[0-9]+)?$' THEN steps::numeric END casts only when the string actually looks numeric; anything else ('N/A', empty, garbage) falls through to NULL instead of raising an invalid input syntax for numeric error that aborts the whole statement. This regex-then-cast guard is how you survive real-world wearable feeds, and Postgres 16+ offers a cleaner equivalent: steps::numeric ... ON ERROR NULL via CAST's error-handling, or a safe-style wrapper.

The sliding-window frame — "6 PRECEDING" means 7 rows

ROWS BETWEEN 6 PRECEDING AND CURRENT ROW spans the current day plus the six before it — seven rows, hence "7-day average". An off-by-one here (writing 7 PRECEDING) silently averages eight days. As with any moving average, ROWS counts rows, not calendar days: if a patient misses a day of syncing, "7 rows" stretches across more than 7 calendar days. If strict calendar windows matter, use RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW.

AVG ignores NULL — helpful here, but know the denominator

After the guard, patient 3's bad steps row is NULL, and AVG excludes NULLs from both numerator and denominator — so the 7-day step average is computed over the two valid readings, not three. That is usually the right call (better than treating a missing reading as zero steps, which would falsely depress the average). But be aware the "7-day average" can quietly be a 6-day or 5-day average when data is sparse; if every reading in the window is NULL, the result is NULL and the WHERE comparison is unknown, excluding the patient.

The trailing-window trap — early rows are partial

For a patient's first six days the window holds fewer than seven rows, so the "7-day average" is really a 1-to-6-day average. A patient could be flagged high-risk on day 2 from a single alarming reading. Clinically that may be premature. If the rule should require a full week of history, gate it: only evaluate rows where COUNT(*) OVER (same window) = 7.

What to say next — the GROUP BY is doing dedup work

The window function produces a rolling average on every row, so a patient who is high-risk on several days would otherwise appear multiple times. The trailing GROUP BY patient_id (with no aggregates) collapses those to one row each — effectively a DISTINCT. SELECT DISTINCT patient_id would read more clearly. A senior answer also asks: should "high-risk" require the thresholds to be breached on the latest day, on any day, or on a sustained run? The query as written flags "any qualifying day", which is worth confirming.

Senior signal: guard the numeric cast with a regex so a 'N/A' reading becomes NULL instead of aborting the query, get the 6 PRECEDING = 7-rows frame right, and know that AVG drops NULLs so a "7-day" average can silently be shorter. Flag that the trailing GROUP BY is really a DISTINCT.
