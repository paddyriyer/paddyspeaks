A loyalty program (a Danny's Diner-style points engine) awards points per dollar spent, but with two stacking bonuses: a first-week welcome bonus and a sushi promotion. The interviewer is probing whether you can express layered, mutually-exclusive business rules with a single CASE ladder — getting the order of the branches right — and whether you handle non-members, who must fall back to the base rate.

1. The source data — members and purchases

customerjoin_dateorder_dateproductamount
A (member)2021-01-072021-01-01sushi 10 / curry 15before join — sushi bonus only
A2021-01-072021-01-07curry 15on join_date — first-week bonus

B (member)2021-01-092021-01-01..16curry, sushi, ramenspans before, during, after window
C (non-member)—2021-01-01/07ramen ×2no join row — base rate only

Customer C is the edge case: she has no members row, so the LEFT JOIN leaves join_date NULL and she must earn the flat 10 points/dollar — no bonus branch may fire for her.

2. The SQL — layered CASE inside a conditional SUM

SELECT p.customer_id,
       SUM(
         CASE
           WHEN m.join_date IS NOT NULL
                AND p.order_date BETWEEN m.join_date AND m.join_date + 6
             THEN p.amount * 20
           WHEN p.product = 'sushi' THEN p.amount * 20
           ELSE p.amount * 10
         END
       ) AS points
FROM purchases p
LEFT JOIN members m ON m.customer_id = p.customer_id
WHERE p.order_date >= DATE '2021-01-01'
  AND p.order_date <  DATE '2021-02-01'
GROUP BY p.customer_id
ORDER BY p.customer_id;

The CASE ladder is evaluated top to bottom and the first matching branch wins. First-week bonus (member-only, date-bounded) is checked first; sushi promotion second; base rate is the ELSE. SUM rolls every row's points into a per-customer total.

4. Strategic explanations

Why branch order is the whole problem

CASE short-circuits: the first WHEN that is true decides the row, and later branches never see it. A sushi bought during the first week is worth 20 either way, so order does not change the number there — but the principle matters: if the rates differed, putting the broader rule first would mask the narrower one. Encode the most specific / highest-priority rule first and let generality cascade down to ELSE.

Why the member guard must be explicit

For a non-member, m.join_date is NULL after the LEFT JOIN. Without m.join_date IS NOT NULL, the predicate p.order_date BETWEEN NULL AND NULL+6 evaluates to unknown — which is not true, so the branch happens not to fire here. But relying on NULL-propagation accidentally is fragile; an explicit IS NOT NULL states the intent ("members only") and survives a future rewrite to an INNER JOIN or an OR condition.

Inclusive 7-day window: join_date + 6

"First 7 days, inclusive of join_date" means days 0 through 6 — BETWEEN join_date AND join_date + 6, not + 7. Off-by-one here silently over- or under-pays bonuses. In Postgres, date + integer adds days; be explicit (+ INTERVAL '6 days') if the column is a timestamp, because date vs timestamp arithmetic differs.

Conditional aggregation vs a pre-computed column

Folding the rule into SUM(CASE ...) keeps everything in one pass and one place. The alternative — a CTE that computes row_points, then a plain SUM — is equally correct and often more readable when the rule is long; it also makes the per-row points inspectable for debugging. Both are fine; for a 3-branch rule the inline form is concise enough.

Half-open date range — and a note on the prompt

The month filter uses >= '2021-01-01' AND < '2021-02-01' — a half-open interval. This is the safe idiom: it captures every January timestamp including 2021-01-31 23:59:59 without the BETWEEN ... AND '2021-01-31' bug that drops late-in-day rows. (The prompt says "January 2025" while the fixture data and solution both use January 2021 — the query is internally consistent with its own dataset, which is what is graded.)

Senior signal: express stacked business rules as a single CASE ladder ordered most-specific-first because the first true branch wins; guard member-only logic with an explicit IS NOT NULL rather than relying on NULL propagation; get the inclusive-window arithmetic exactly right and filter months with a half-open >= / < range.
