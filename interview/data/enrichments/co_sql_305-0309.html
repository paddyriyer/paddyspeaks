"Are you still watching?" The auto-play prompt is the most-instrumented UI element in the Netflix product. Every impression is a decision moment: the user either clicks "Continue" (auto-play wins) or walks away (auto-play loses). The conversion rate per title is the single best proxy for narrative momentum — does the show pull the viewer through the seam? The query is conditional aggregation; the senior conversation is the statistical-significance gate.

2. The SQL — rate per title with NULLIF and HAVING

SELECT title_id,
  COUNT(*) AS impressions,
  SUM(CASE WHEN user_continued = 1 THEN 1 ELSE 0 END) AS continued,
  ROUND(100.0 * SUM(CASE WHEN user_continued = 1 THEN 1 ELSE 0 END)
              / NULLIF(COUNT(*), 0), 2) AS continue_rate_pct
FROM auto_play_events
GROUP BY title_id
HAVING COUNT(*) >= 100
ORDER BY continue_rate_pct DESC;

3. Strategic explanations

Why HAVING COUNT(*) >= 100 is non-negotiable

A title with 3 impressions and 2 continues looks like 67% conversion — topping the leaderboard. With 100 impressions you have a 95% confidence interval of roughly ±10pp around the true rate. With 1,000, ±3pp. With 10,000, ±1pp. The HAVING gate is the difference between a real signal and a sample-size artifact.

For "leaderboard"-style reports, push the gate higher (1000+) and add a confidence-interval column:

continue_rate_pct,
1.96 * SQRT(rate * (1 - rate) / impressions) * 100 AS ci_pp_at_95
The "1 ELSE 0" vs FILTER (WHERE) idiom
-- Verbose but portable (MySQL-friendly):
SUM(CASE WHEN user_continued = 1 THEN 1 ELSE 0 END)

-- Modern (Postgres / SQLite 3.30+ / Snowflake):
COUNT(*) FILTER (WHERE user_continued = 1)

-- Sneaky for boolean-like 0/1 columns:
SUM(user_continued)

The third form is the shortest but assumes user_continued is exactly 0 or 1 (no NULLs, no 2s). Production-grade code uses FILTER for clarity.

The seam-position trap — intra-episode vs end-of-episode

Auto-play impressions fire at multiple seams:

  • End-of-episode → next episode in same show (the high-conversion seam — same world, same characters)
  • End-of-show → recommended different show (much lower conversion — cold-start)
  • Mid-episode "you've been idle" (lowest — the user already disengaged)

Aggregating all three together blurs everything. Always include seam_type in GROUP BY for actionable analysis.

The episode-position gradient

Within a show, conversion at the end of E_01 is much higher than at the end of E_08 (some viewers always drop after a few episodes). The CASE when conversion drops sharply identifies narrative dead spots:

SELECT show_id, episode_number,
  100.0 * SUM(user_continued) / COUNT(*) AS continue_rate_pct
FROM auto_play_events e JOIN episodes ep USING (title_id)
GROUP BY show_id, episode_number
ORDER BY show_id, episode_number;

The "biggest cliff" identifies the episode after which retention collapses — that's the writers' room conversation.

Counterfactual — what would have happened without auto-play?

The conversion rate alone doesn't measure auto-play's incremental value — some users would have started the next episode anyway. The real question requires an A/B test:

SELECT bucket,                    -- 'autoplay-on' vs 'control'
  COUNT(*) AS users,
  AVG(next_ep_started_within_5min) AS conversion_rate
FROM ab_assignments a JOIN auto_play_events e USING (user_id)
GROUP BY bucket;

The diff between treatment and control is the auto-play uplift — what Netflix actually optimizes.

The "skip" wrinkle

Users can actively cancel auto-play (clicking "Cancel") vs passively let it expire. Both produce user_continued = 0 but tell different stories: active cancel = "I'm done with this show"; passive expire = "I walked away from the screen". Three buckets, not two:

SUM(CASE WHEN user_continued = 1                        THEN 1 END) AS continued,
SUM(CASE WHEN user_continued = 0 AND cancelled_at IS NOT NULL THEN 1 END) AS active_cancel,
SUM(CASE WHEN user_continued = 0 AND cancelled_at IS NULL     THEN 1 END) AS passive_expire
The completion-rate cousin metric

Auto-play conversion is most predictive when combined with episode completion. A title where 90% finish but only 30% auto-continue means the show ended cleanly — that's a compliment, not a problem (people are satisfied). The diagnostic isn't auto-play alone; it's the delta between completion and continuation.

Day-part normalization

Auto-play conversion at 11pm (sleep timer kicks in) is structurally lower than at 7pm. Don't compare titles across day-parts unless you normalize:

GROUP BY title_id, EXTRACT(HOUR FROM autoplay_shown_at)
Senior signal: pin the sample-size gate (HAVING COUNT >= 100) with a confidence-interval column, raise the seam-type confounder unprompted, and push the conversation toward A/B test uplift over raw rate. The episode-position gradient + active-vs-passive cancel split is the over-the-bar finish that connects the query to the writers' room.
