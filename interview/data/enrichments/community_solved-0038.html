A community dataset hands you a sequence of numbers split into named partitions and asks for the start and end of each unbroken run. This is the canonical gaps-and-islands problem dressed up: within each partition, find every maximal stretch of consecutive integers and report its min and max. The interviewer is probing whether you know the row-number-minus-value trick — the single most reused pattern in SQL interviews.

1. The source data — tab_sequence

partitionsequence(narrative)
A1run 1 starts
A2…
A3run 1 ends (gap: 4 missing)
A5run 2 starts
A6run 2 ends
B10run starts
B11run ends (gap: 12 missing)
B13single-element run

Partition B's 13 is a run of length one — an isolated value with gaps on both sides. The query must still emit it as an island where min equals max, not drop it.

2. The SQL — row-number-minus-value islands

WITH flag_query AS (
  SELECT "partition" AS grp,
         "sequence",
         ROW_NUMBER() OVER (PARTITION BY "partition" ORDER BY "sequence" ASC) AS flag,
         CASE WHEN "sequence"::text ~ '^-?[0-9]+$' THEN "sequence"::int ELSE 0 END
           - ROW_NUMBER() OVER (PARTITION BY "partition" ORDER BY "sequence" ASC) AS diff
  FROM tab_sequence
)
SELECT DISTINCT grp,
  MIN("sequence") OVER (PARTITION BY grp, diff) AS min_seq,
  MAX("sequence") OVER (PARTITION BY grp, diff) AS max_seq
FROM flag_query;

Inside a consecutive run the value and the row number both increase by 1 each step, so value - row_number stays constant; it jumps only at a gap. That constant diff is the island identifier.

3. The resulting analysis

grpmin_seqmax_seq
A13
A56
B1011
B1313

Four islands. Partition B's single value 13 surfaces as its own island with min_seq = max_seq = 13 — exactly the behaviour you want, and the reason the demo includes it.

4. Strategic explanations

Why value - row_number identifies an island

Order the values; assign each a 1,2,3… row number. Where the values are consecutive, both series advance in lockstep, so their difference is invariant. A missing value makes the value jump while the row number does not — the difference shifts. Rows sharing a difference therefore belong to the same unbroken run. PARTITION BY grp, diff turns that into a window, and MIN/MAX over it give the run's endpoints.

Why DISTINCT is required here

MIN/MAX as window functions (not aggregates) attach the same island min and max to every row of the island without collapsing rows. A 3-row island therefore produces three identical output rows; SELECT DISTINCT dedupes them down to one per island. The cleaner alternative is to aggregate instead: GROUP BY grp, diff with plain MIN/MAX — one row per island by construction, no DISTINCT needed.

The TEXT-column trap — lexicographic ordering

sequence is stored as TEXT, so ORDER BY "sequence" and MIN/MAX sort as strings: '10' sorts before '9', and '13' before '2'. With this dataset the strings happen to match numeric order, so the result is correct — but on a partition containing both '9' and '10' the islands would be wrong. The CASE ... ~ '^-?[0-9]+$' regex already extracts a clean integer for diff; the fix is to also sort and aggregate on that integer: ORDER BY "sequence"::int and MIN("sequence"::int). Naming this latent bug is the senior move.

The regex guard against dirty data

"sequence"::text ~ '^-?[0-9]+$' tests that the value is a pure (optionally negative) integer before casting. A stray 'N/A' would otherwise abort the whole query with an invalid input syntax for type integer error. The ELSE 0 branch keeps non-numeric rows from crashing the run — though it silently lumps them into the diff = -row_number bucket, which a stricter design would route to a NULL or an error instead.

Senior signal: recognise this instantly as gaps-and-islands, explain that value - row_number is constant within a run, prefer GROUP BY grp, diff over SELECT DISTINCT for one clean row per island, and flag that ordering a TEXT sequence column sorts lexicographically — cast to int before sorting.
