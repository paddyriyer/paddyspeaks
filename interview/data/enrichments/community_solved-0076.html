A growth team wants to know which customers spent the most in their first calendar month of ordering — an early-revenue signal that predicts lifetime value. The subtlety the interviewer is watching for: "first month" is not the first order, it is the calendar month of the first order, and every order in that same month must be summed. This forces a two-step "find the anchor, then aggregate within it" pattern.

2. The SQL — anchor month via MIN, then self-rejoin

WITH first_order AS (
  SELECT customer_id, MIN(order_date) AS first_order_date
  FROM order_details
  GROUP BY customer_id
),
first_month AS (
  SELECT customer_id, to_char(first_order_date, 'YYYY-MM') AS month_year
  FROM first_order
),
max_amount_spend AS (
  SELECT o.customer_id, SUM(o.order_amount) AS total_amount
  FROM order_details o
  INNER JOIN first_month f
    ON o.customer_id = f.customer_id
   AND f.month_year = to_char(o.order_date, 'YYYY-MM')
  GROUP BY o.customer_id
  ORDER BY total_amount DESC
  LIMIT 3
)
SELECT m.*, c.customer_name
FROM max_amount_spend m
LEFT JOIN customers c ON m.customer_id = c.customer_id;

4. Strategic explanations

Why two passes over the same table are needed

You cannot know which month is "first" until you have scanned all of a customer's orders, and you cannot sum the first month's orders until you know which month that is. The first CTE establishes the anchor (MIN(order_date)); the third CTE rejoins order_details to itself on customer + month string to pull only the in-month rows. This "establish anchor, then filter to anchor" shape recurs constantly — first purchase, cohort month, account-opening week.

The month-bucket join key — to_char vs date_trunc

Comparing to_char(date, 'YYYY-MM') on both sides buckets by calendar month as a string. It works, but date_trunc('month', order_date) is the more idiomatic Postgres form: it keeps a real date/timestamp, sorts and indexes correctly, and avoids locale surprises. String-month comparison is fine for equality but would mis-sort across year boundaries if you ever needed < / >.

The LIMIT-without-tie-break trap

ORDER BY total_amount DESC LIMIT 3 is non-deterministic when several customers tie on the third-place amount — the engine may return any of them. For a stable, defensible "top 3" add a deterministic tie-breaker (ORDER BY total_amount DESC, customer_id), or switch to RANK() OVER (ORDER BY total_amount DESC) and filter rank <= 3 if the business wants all tied customers included.

Why the final join is LEFT, and the ordering caveat

The closing LEFT JOIN customers only decorates the three winners with a display name; LEFT guards against an order referencing a customer missing from customers (the row survives with a NULL name rather than vanishing). But note the join does not preserve the ORDER BY from the CTE — SQL result order is undefined unless the outermost query has its own ORDER BY. Add ORDER BY total_amount DESC on the final SELECT for a presentation-ready ranking.

What to say next — window-function alternative

The whole thing collapses into one scan with a window function: tag each order with MIN(date_trunc('month', order_date)) OVER (PARTITION BY customer_id), keep rows whose month matches, sum per customer, and rank. Fewer self-joins, one pass — the senior framing is "this is a first-cohort problem, and window functions express cohorts most directly."

Senior signal: recognise "first month" as a calendar-month anchor (not the first order), prefer date_trunc over to_char string-month keys, add a deterministic tie-breaker to LIMIT 3, and remember that a join does not carry an inner ORDER BY — the outermost query needs its own.
