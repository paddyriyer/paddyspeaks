Shopify's merchant dashboard shows each customer's most recent order — one row per customer, the latest purchase. It feels trivial until you hit a customer who placed two orders on the same day: which one is "the latest"? The reference uses ROW_NUMBER() to pick exactly one. The interviewer is checking whether you know that ROW_NUMBER resolves ties arbitrarily — and whether you make that tiebreak deterministic.

1. The source data — orders

Customers 101 and 102 each have two orders on different days; customer 103 has just one. Distinct dates make this clean — the interesting case (two orders on the same date) is what section 4 stress-tests.

2. The SQL — ROW_NUMBER then filter rn = 1

SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC) rn FROM orders
) t WHERE rn=1;

The inner query numbers each customer's orders newest-first. The outer query keeps only rn = 1 — the latest order per customer. The window function must be assigned in a subquery because you cannot reference a window alias in a WHERE at the same level.

4. Strategic explanations

The same-day tie — the bug waiting to happen

ORDER BY order_date DESC alone is not a total order. If customer 101 placed two orders on 2026-01-05, ROW_NUMBER still assigns 1 and 2 — but which order gets rn 1 is arbitrary, decided by physical row order or the planner's whim. The dashboard would show a different "latest order" run to run. Watch:

-- customer 101: orders 1 and 2 BOTH on 2026-01-05
-- ROW_NUMBER picks one non-deterministically
customer_id | order_id | order_date
101         | 1        | 2026-01-05   <- could be order 2 next run

The fix is a deterministic tiebreaker — usually the surrogate key, which is monotonic with insertion time: ORDER BY order_date DESC, order_id DESC. Now the latest order is reproducible. If the table has a true timestamp column, order by that instead of the date.

ROW_NUMBER vs RANK — pick one, on purpose

If two orders genuinely tie on the latest date and the business wants both shown, ROW_NUMBER is wrong — it forces a single winner. Use RANK() and filter rk = 1: ties share rank 1, so both rows survive. The choice is semantic: ROW_NUMBER = "exactly one row per customer", RANK = "all rows tied at the top". The question says "the latest purchase" (singular), so ROW_NUMBER plus a tiebreaker is right here — but name the alternative.

The DISTINCT ON alternative

Postgres has a terser idiom for "one row per group, top by some order":

SELECT DISTINCT ON (customer_id) *
FROM orders
ORDER BY customer_id, order_date DESC, order_id DESC;

It avoids the subquery and never emits a stray rn column. The catch: the leading ORDER BY columns must start with the DISTINCT ON key, and the same tiebreaker rule applies. It is Postgres-specific — ROW_NUMBER is the portable choice across engines.

The correlated-subquery anti-pattern

A pre-window approach — WHERE order_date = (SELECT MAX(order_date) FROM orders o2 WHERE o2.customer_id = orders.customer_id) — works but runs a subquery per row and, crucially, returns two rows for a customer with two orders on the max date. It reintroduces the tie problem you were trying to solve and scales poorly. The window function is both cleaner and tie-controllable.

Edge cases: NULL dates and SELECT * hygiene

If order_date is NULL on some orders, ORDER BY order_date DESC sorts NULLs first in Postgres — an undated order would be crowned "latest". Add NULLS LAST or filter NULL dates. And the literal SELECT * here exposes the internal rn to consumers; in production, project the columns you actually want.

Senior signal: recognise that ORDER BY order_date DESC is not a total order, add a deterministic tiebreaker (order_id DESC) so "latest" is reproducible, distinguish ROW_NUMBER ("exactly one") from RANK ("all ties"), and avoid leaking the helper rn column via a careless SELECT *.
