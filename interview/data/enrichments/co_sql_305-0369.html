Listings with long blackout stretches signal hosts who manage poorly OR have offline-rented (illegal long-tail bookings). The query uses gaps-and-islands on the calendar to detect contiguous unavailable runs per listing.

1. The source data — calendar_nights

listing_idnightavailable
L_012024-01-010
L_012024-01-020

2. The SQL — gaps-and-islands on unavailable nights

WITH unavail AS (
  SELECT listing_id, night,
    julianday(night) - ROW_NUMBER() OVER (PARTITION BY listing_id ORDER BY night) AS island_id
  FROM calendar_nights
  WHERE available = 0
)
SELECT listing_id,
  MIN(night) AS blackout_start,
  MAX(night) AS blackout_end,
  COUNT(*) AS blackout_days
FROM unavail
GROUP BY listing_id, island_id
HAVING COUNT(*) >= 1
ORDER BY blackout_days DESC;

3. Strategic explanations

Gaps-and-islands — why date − rn = constant

Within a contiguous run of unavailable nights, both the date and the row_number tick by 1 each row. Their difference is constant. As soon as the calendar has a 1-day gap (an available night), the difference shifts. Distinct difference = distinct blackout stretch.

The DEDUP step (implicit here)

Calendar-as-fact is already 1-row-per-night, so no dedup needed. For event-stream tables (page_views, etc.), always DISTINCT first before ROW_NUMBER.

Why filter available = 0 AT THE START

The window function operates only on unavailable rows. The "gaps" between them in the calendar produce different island_ids — exactly what we want. Filtering after window doesn't work; the window enumeration would include available days.

The "30+ day" red flag

Production threshold: blackout_days > 30 is investigated. Reasons:

  • Host out of country (legitimate)
  • Long-term off-platform rental (Airbnb policy violation)
  • Property under repair
  • Listing should be deactivated

Trust + Safety team uses this to flag policy-violation listings.

The "every-weekend-blackout" pattern

Some hosts block all weekends to keep place for personal use. Detection: blackout_days = 2, recurring weekly. The query catches the streaks; the recurrence pattern needs follow-up:

... HAVING COUNT(*) >= 1 AND COUNT(DISTINCT strftime('%w', night)) <= 3
Calendar-vs-bookings reconciliation

Some "blackouts" are real bookings. Join to bookings for the diff. Pure blackouts (no booking explanation) are the suspicious ones.

Calendar-vs-bookings reconciliation is the over-the-bar policy-enforcement move.
