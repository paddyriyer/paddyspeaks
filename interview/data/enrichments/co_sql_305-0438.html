One of the most-asked window-function gotchas: "my MoM increase query returns NULL for some departments — why?" The answer is the boundary behavior of LAG(): at the first row of every partition, there's nothing to look back at. LAG() returns NULL, the subtraction propagates that NULL forward, and downstream RANK / MAX / WHERE all behave in surprising ways because of it.

2. The naïve query (and the trap)

A first-pass solution computes monthly totals per department, then subtracts the previous month via LAG:

WITH monthly_totals AS (
  SELECT o.department_id, d.department_name,
    strftime('%Y-%m', o.order_date) AS order_month,
    SUM(o.order_amount) AS monthly_amount
  FROM orders o JOIN departments d USING (department_id)
  WHERE o.order_date >= '2024-01-01' AND o.order_date < '2024-03-01'
  GROUP BY o.department_id, d.department_name, strftime('%Y-%m', o.order_date)
),
mom AS (
  SELECT *, monthly_amount - LAG(monthly_amount) OVER (
    PARTITION BY department_id ORDER BY order_month
  ) AS mom_increase
  FROM monthly_totals
)
SELECT department_id, department_name, mom_increase
FROM mom
WHERE order_month = '2024-02'
ORDER BY mom_increase DESC LIMIT 1;

This works only if every department has both January and February rows. Departments that started selling in February have only one row in their partition; LAG returns NULL; the subtraction yields NULL; the row gets pushed to the bottom of ORDER BY DESC (or the top, depending on dialect). The "winning" department might be the one that legitimately had no January at all.

3. The robust solution — cross-join + COALESCE + NULLS LAST

WITH months AS (
  SELECT '2024-01' AS order_month UNION ALL SELECT '2024-02'
),
dept_months AS (
  SELECT d.department_id, d.department_name, m.order_month
  FROM departments d CROSS JOIN months m
),
monthly_totals AS (
  SELECT dm.department_id, dm.department_name, dm.order_month,
    COALESCE(SUM(o.order_amount), 0) AS monthly_amount
  FROM dept_months dm
  LEFT JOIN orders o
    ON o.department_id = dm.department_id
   AND strftime('%Y-%m', o.order_date) = dm.order_month
  GROUP BY dm.department_id, dm.department_name, dm.order_month
),
mom AS (
  SELECT department_id, department_name, order_month,
    monthly_amount - LAG(monthly_amount) OVER (
      PARTITION BY department_id ORDER BY order_month
    ) AS mom_increase
  FROM monthly_totals
)
SELECT department_id, department_name, mom_increase
FROM mom
WHERE order_month = '2024-02'
ORDER BY mom_increase DESC NULLS LAST
LIMIT 1;

4. Strategic explanations

Why LAG returns NULL — partition boundaries are NULL by design

Window functions never invent data. When you ask for the previous row and there isn't one, the spec says: return NULL. This is consistent across every dialect (Postgres, SQLite, Snowflake, BigQuery, MySQL 8+) — and consistent with what FIRST_VALUE, LEAD, NTH_VALUE all return at boundaries. The fix is never to "make LAG do something else"; it's to handle the NULL where it surfaces.

How NULL propagates — three failure modes
OperationWhat happens with NULLVisible symptom
x − NULLNULLmom_increase column is NULL for one-month departments
NULL = MAX(...)NULL (never TRUE)The "use a subquery for max" pattern silently excludes NULL rows — sometimes that's what you want, sometimes it hides bugs

Three fixes — pick deliberately
Fix A — COALESCE inside LAG (treat missing prior month as zero)
monthly_amount - COALESCE(
  LAG(monthly_amount) OVER (PARTITION BY department_id ORDER BY order_month),
  0
) AS mom_increase

Semantic: "a department that didn't exist last month started at zero, so the entire current-month amount IS the increase." Right answer when "department launched mid-year and December's $50K is real growth." Wrong answer when launching a new department shouldn't be ranked above an existing one growing 10×.

Fix B — Cross-join to force both rows to exist (the cleanest)

The solution above. Generates a (department × month) Cartesian product, LEFT JOINs the actuals, COALESCEs the missing aggregate to zero. After this step, every department has exactly two rows; LAG always finds a predecessor; NULL can no longer leak. The verbosity buys total predictability.

Fix C — Filter the NULLs out (preserve the original "rank only departments with both months" intent)
WHERE order_month = '2024-02' AND mom_increase IS NOT NULL
ORDER BY mom_increase DESC LIMIT 1;

Equivalent to the = MAX(...) trick. Use when "departments that didn't exist last month are out of scope for the comparison" is the business rule.

The NULLS LAST tail

Even if you've COALESCEd most NULLs away, ORDER BY x DESC NULLS LAST is a free safety net. Without it, a single straggling NULL can sit at the top of your LIMIT 1 result — depending on the dialect's default — and quietly return a wrong winner. Three things to remember:

  • Postgres / Oracle / SQLite 3.30+ / DuckDB: support NULLS FIRST / NULLS LAST explicitly. Default for DESC is NULLS FIRST.
  • MySQL 8 / SQL Server: don't support NULLS LAST directly. Workaround: ORDER BY x IS NULL, x DESC — the boolean coerces to 0/1, NULLs sort last.
  • BigQuery: supports it; default for DESC is NULLS LAST. Different from Postgres — read the manual when porting.
The RANK equivalent — switching from = MAX to RANK() = 1

The = MAX(...) filter implicitly drops NULLs because NULL = NULL is NULL, not TRUE. If you switch to RANK() OVER (ORDER BY mom_increase DESC) = 1, NULLs are no longer auto-dropped — they get ranked. In Postgres / SQLite, NULLs end up at the BOTTOM of the rank (last), but the rank value is still computed. In SQL Server / MySQL, NULLs end up FIRST and rank=1 silently lands on the wrong row.

-- Cross-dialect-safe RANK on a NULL-able sort key
RANK() OVER (ORDER BY mom_increase DESC NULLS LAST) AS rk
-- Then: WHERE rk = 1 AND mom_increase IS NOT NULL
The diagnostic query — what to actually run when debugging

When a teammate complains "MoM is NULL for department X", the fastest debugging step is to look at the monthly_totals CTE before the LAG layer:

SELECT * FROM monthly_totals
WHERE department_id = X
ORDER BY order_month;

Three patterns appear:

Rows visibleWhat it means
Both Jan + FebLAG works; mom_increase has a real value

The "implicit filter" trap with = MAX(...)

The = (SELECT MAX(mom_increase) FROM mom) idiom looks elegant but does TWO things: finds the maximum AND silently filters NULLs. Switching to RANK() OVER ... = 1 changes only the first behavior; the second has to be re-implemented explicitly. Many "I just refactored to RANK and now my query is broken" bugs are exactly this implicit-filter loss.

Production-grade pattern — the date spine

The cross-join trick generalizes. For any "compare metric across N consecutive periods" question, prebuild a date / period spine and LEFT JOIN against it:

-- Generic period-spine template
WITH period_spine AS (
  SELECT '2024-01' AS p UNION ALL SELECT '2024-02' UNION ALL SELECT '2024-03'
  -- or: SELECT generate_series('2024-01-01'::date, '2024-12-01'::date, '1 month')
),
entity_period AS (
  SELECT e.entity_id, sp.p
  FROM entities e CROSS JOIN period_spine sp
),
metric_per_period AS (
  SELECT ep.entity_id, ep.p, COALESCE(SUM(f.value), 0) AS metric
  FROM entity_period ep
  LEFT JOIN fact f ON f.entity_id = ep.entity_id AND period_of(f) = ep.p
  GROUP BY ep.entity_id, ep.p
)

This is the single most-useful template for retention curves, cohort analysis, MoM / WoW / YoY trends, and any "missing period = real zero" comparison.

The senior reframe

The interviewer's actual signal isn't "do you know the LAG function?" — every junior does. It's: "do you anticipate the NULL boundary, and have you internalized that NULL is contagious through arithmetic, comparison, ordering, and ranking?". The candidate who reaches for COALESCE, NULLS LAST, and the cross-join spine reflexively is the one who's been bitten enough times to be senior.

Senior signal: name LAG's first-row-NULL as the boundary contract, articulate the four NULL-propagation paths (subtraction / equality / ranking / WHERE), pick between Fix A (COALESCE-zero), Fix B (cross-join spine), and Fix C (explicit IS NOT NULL) based on the actual business framing, and add NULLS LAST as a portable safety net. The "implicit-filter loss when refactoring = MAX to RANK" is the trap that catches even experienced engineers — flag it unprompted.
