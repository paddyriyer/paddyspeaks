An e-commerce analyst wants, per customer, how many products they bought on their most recent shopping day — a quick read on latest-visit basket size. The interviewer is probing a deceptively tricky "latest per group" case: a customer can have several transactions on that same most-recent date, and the spec says to sum across them rather than pick one arbitrarily.

1. The source data — user_txn

User 1 is the trap: two transactions share the most-recent date 2025-02-10 (2 and 4 products). The answer for user 1 must be 2 + 4 = 6, not 2 or 4. User 3 confirms only the true maximum date counts — the 04-19 row is one day short and must be excluded.

2. The SQL — find each user's max date, join back, sum the matches

WITH latest AS (
  SELECT user_id, MAX(txn_date) AS latest_date
  FROM user_txn
  GROUP BY user_id
)
SELECT u.user_id, u.txn_date AS latest_date, SUM(u.products_bought)::int AS total_products
FROM user_txn u
JOIN latest l ON l.user_id = u.user_id AND l.latest_date = u.txn_date
GROUP BY u.user_id, u.txn_date
ORDER BY u.user_id;

latest computes each user's most-recent txn_date. The join keeps only original rows that fall on that date (matching both user_id and the date). The final GROUP BY ... SUM then folds multiple same-day transactions into one total.

User 1 returns 6 — both 2025-02-10 rows match the latest date and SUM adds them. User 3 returns 7: the 04-19 transaction is one day before the maximum, so the join drops it. The SUM + GROUP BY is what correctly handles the same-day multiplicity.

4. Strategic explanations

Why ROW_NUMBER alone would be wrong here

The reflex for "latest per group" is ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY txn_date DESC) filtered to rn = 1 — but that returns exactly one row, so for user 1 it would pick either the 2-product or the 4-product transaction arbitrarily and report 2 or 4. The spec demands summing all transactions on the latest date. The max-date-and-join pattern naturally keeps every tied row; ROW_NUMBER deliberately discards them. Choosing the right "latest per group" variant is the core senior judgement.

The join condition must match on both keys

ON l.user_id = u.user_id AND l.latest_date = u.txn_date joins each user's transactions back to their own max date. Dropping the date predicate would re-attach every transaction; dropping the user predicate would let one user's transactions match another user's max date. Both keys together act as a filter: keep a transaction iff it occurred on its owner's most-recent day. This semi-join-by-equality is the heart of the pattern.

If a window approach is preferred — do it correctly

You can stay window-only: MAX(txn_date) OVER (PARTITION BY user_id) AS latest_date, then WHERE txn_date = latest_date, then GROUP BY user_id, txn_date with SUM. That avoids the self-join and gives the identical result — the key is the final GROUP BY/SUM, present in both forms, because that is what aggregates the tied same-day rows. The mistake to avoid is any approach that picks a single row before the sum.

Including latest_date in GROUP BY — and why it is safe

The output needs latest_date as a column, so it appears in the GROUP BY. That is harmless: within a single user, every surviving row already has the same txn_date (it is the max date by construction), so grouping by (user_id, txn_date) creates exactly one group per user — the same grouping as user_id alone. The date is along for the ride, not splitting groups.

Dialect notes

MAX, GROUP BY, JOIN, and SUM are universal — the CTE-and-join form ports unchanged to MySQL, SQL Server, Oracle, and SQLite. The window alternative needs MAX() OVER (...), available in MySQL 8+, SQL Server, Oracle, and SQLite 3.25+ but not legacy MySQL. ::int is Postgres; elsewhere use CAST(SUM(products_bought) AS INT).

Senior signal: for "latest day per group," recognise that the most-recent date can carry multiple rows — compute the per-group MAX date, join (or window-filter) back to keep every row on that date, then GROUP BY/SUM to fold them; a bare ROW_NUMBER() = 1 is a bug because it arbitrarily keeps just one of the tied same-day transactions.
