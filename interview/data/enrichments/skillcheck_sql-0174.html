Fetching the most recent row per group from an append-only log is one of the highest-frequency patterns in production analytics: current account status, latest order state, most recent device telemetry. The interviewer is probing whether you know PostgreSQL's DISTINCT ON extension — a concise, planner-friendly alternative to the window-function subquery — and whether you understand its precise semantics: the ORDER BY inside the clause both selects the winning row and drives the output order.

1. The source data — status_log

32025-03-01 09:00reopenedsame ts as above; "reopened" > "open" lexicographically

Entity 3 has two rows with identical timestamps — the tie-break requirement. Alphabetically, reopened > open, so ORDER BY … status DESC places reopened first, and DISTINCT ON selects it.

2. The SQL — Postgres DISTINCT ON

SELECT DISTINCT ON (entity_id) entity_id, ts, status
FROM status_log
ORDER BY entity_id, ts DESC, status DESC;

DISTINCT ON (entity_id) keeps exactly one row per entity_id — whichever comes first in the ORDER BY. With ORDER BY entity_id, ts DESC, status DESC, the row with the largest ts for each entity appears first; ties in ts are broken by status DESC (reverse alphabetical). The output is then sorted by entity_id ASC because that is the leftmost ORDER BY key.

3. The resulting analysis

Entity 1: three rows; ts DESC puts 2025-01-05 first → closed. Entity 2: two rows; 2025-02-10 > 2025-02-01 → pending. Entity 3: two rows with equal ts; status DESC puts reopened before open → reopened.

4. Strategic explanations

How DISTINCT ON works

PostgreSQL evaluates DISTINCT ON (cols) after sorting by the full ORDER BY clause: it scans the sorted result and emits only the first row seen for each distinct combination of (cols). The key rule: the DISTINCT ON columns must be the leftmost keys in ORDER BY. The remaining ORDER BY keys control which row within each group is selected; they do not impose a separate sort on the output beyond the first key. Violating this constraint — putting non-DISTINCT-ON keys before the DISTINCT-ON key in ORDER BY — is a PostgreSQL error.

Equivalent window-function approach (portable SQL)

The portable equivalent uses ROW_NUMBER():

SELECT entity_id, ts, status
FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY entity_id ORDER BY ts DESC, status DESC) AS rn
  FROM status_log
) t
WHERE rn = 1
ORDER BY entity_id;

This works on PostgreSQL, MySQL 8+, SQL Server, BigQuery, Snowflake, and Oracle. It is slightly more verbose and may require a subquery or CTE, but it is the correct answer on any engine that lacks DISTINCT ON. In an interview, knowing both forms and their trade-offs is the senior signal.

When DISTINCT ON beats ROW_NUMBER on performance

With an index on status_log(entity_id, ts DESC), the PostgreSQL planner can satisfy DISTINCT ON with an index-only scan: it reads one row per entity from the index tip and stops, never needing to materialise the full sorted set. ROW_NUMBER in a subquery forces the planner to compute all row numbers before filtering — it cannot short-circuit. For tables with millions of log entries and hundreds of entities, this difference can be an order of magnitude. The index should match the DISTINCT ON key and the leading ORDER BY direction.

Tie-breaking determinism

Append-only logs frequently have rows with the same timestamp (batch imports, clock skew). Without a stable tie-break in ORDER BY, the winner among tied rows is arbitrary and may change between query runs. The status DESC secondary sort makes the result deterministic. In production, a surrogate sequence column (e.g. log_id SERIAL) is often a safer tie-break than a business column, because it reflects insertion order unambiguously.

Senior signal: know that DISTINCT ON requires the DISTINCT ON columns to be leftmost in ORDER BY; explain the ROW_NUMBER portable equivalent; mention that an index on (entity_id, ts DESC) lets Postgres short-circuit with one index read per entity rather than sorting the whole table; and always include a deterministic tie-break column.
