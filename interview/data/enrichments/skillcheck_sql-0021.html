A monitoring team wants an anomaly score for a daily metric: the z-score of each day's value against the trailing 7-day window that includes that day. A z-score says "how many standard deviations from the recent norm is today" — the backbone of spike detection. The interviewer is probing rolling-window aggregates, the population-versus-sample variance distinction, and the divide-by-zero trap when a window is perfectly flat.

Two edge cases. The opening flat run means the first four windows have zero variance — the z-score formula would divide by zero. And 2025-01-08's value of 50 is a deliberate outlier whose z-score should jump.

2. The SQL — rolling mean and variance via the E[X²]−E[X]² identity

WITH stats AS (
  SELECT
    d,
    v,
    AVG(v)  OVER (ORDER BY d ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS mu,
    AVG(v*v) OVER (ORDER BY d ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)
      - AVG(v) OVER (ORDER BY d ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)
      * AVG(v) OVER (ORDER BY d ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS var_pop
  FROM daily
)
SELECT
  d,
  v,
  CASE WHEN var_pop = 0 THEN NULL
       ELSE ROUND(((v - mu) / sqrt(var_pop))::numeric, 3)::float END AS z
FROM stats
ORDER BY d;

ROWS BETWEEN 6 PRECEDING AND CURRENT ROW is the trailing 7-row window. The mean is a plain rolling AVG; the population variance uses the identity Var = E[X²] − (E[X])². The CASE guards against a zero-variance window before the division.

4. Strategic explanations

Why the E[X²] − E[X]² identity instead of STDDEV

Population variance equals the mean of the squares minus the square of the mean. Expressing it with two AVG windows means the variance is computed over the same rolling frame as the mean — no separate machinery. Many engines also expose VAR_POP/STDDEV_POP as window functions, which is cleaner where available: STDDEV_POP(v) OVER (ORDER BY d ROWS BETWEEN 6 PRECEDING AND CURRENT ROW). The identity form is the most portable and shows you understand what the function computes underneath.

Population vs sample variance — the spec is explicit

The question demands population standard deviation, "no Bessel correction". Population variance divides the summed squared deviations by n; sample variance divides by n−1 (Bessel's correction) and is what STDDEV/STDDEV_SAMP return by default in most engines. They give different numbers, especially for small windows. Using VAR_SAMP here would be a subtle, marks-losing mistake — match the requested estimator exactly, and say which one you used.

The divide-by-zero guard is mandatory

When a window is perfectly flat, variance is 0, sqrt(0) is 0, and (v - mu) / 0 is a division-by-zero error in PostgreSQL — the whole query fails. The CASE WHEN var_pop = 0 THEN NULL branch intercepts that: a flat window has no meaningful spread, so "z is undefined" (NULL) is the honest answer. The flat opening run exists precisely to force this path. Never divide by a computed statistic without first asking whether it can be zero.

The window shrinks at the start — and that interacts with the flat run

ROWS BETWEEN 6 PRECEDING AND CURRENT ROW wants seven rows but clips to whatever is available, so day 1's window is just itself, day 2's is two rows, and so on. Here every early window also happens to be flat, so they are NULL regardless; but in general a one-row window has variance 0 and would likewise yield NULL. If the rule were "only score once seven days exist", add COUNT(*) OVER (...) = 7 to the CASE. Knowing the frame shrinks — and that a single-row window is trivially zero-variance — is the senior detail.

Numeric hygiene around the rounding

ROUND(x, 3) needs a numeric argument to round to a fixed scale, hence the ::numeric cast; the final ::float matches the expected column type. Floating-point variance can also produce a tiny negative value instead of an exact 0 from accumulated error — if that risk is real, guard with var_pop <= 0 rather than = 0, or GREATEST(var_pop, 0) before the sqrt.

Senior signal: compute a rolling z-score with matched AVG windows for mean and variance (or STDDEV_POP where available), use the population estimator because the spec says so, and always guard the division with CASE WHEN var_pop = 0 THEN NULL — a flat window must yield NULL, never a divide-by-zero crash.
