Amazon is sizing fulfillment for a Prime sale and needs to flag warehouses running hot — anything above 80% utilization gets put on a capacity-watch list. The data is capacity and used per warehouse. The reference solution does something defensive and slightly surprising: it strips non-numeric characters out of each value before dividing. The interview question is why a senior engineer would write that, and where the approach is fragile.

1. The source data — warehouses

warehouse_idcapacityused
11000900
21000750
3500480
420001000
5800800

Utilization is used / capacity: warehouse 1 = 90%, 2 = 75%, 3 = 96%, 4 = 50%, 5 = 100%. Only 1, 3, and 5 clear the 80% threshold. Warehouse 5 is the edge case — exactly full, 100%, completely out of headroom.

2. The SQL — sanitise, then compute utilization

SELECT warehouse_id, capacity, used,
  NULLIF(regexp_replace(capacity::text, '[^-.0-9]', '', 'g'), '')::numeric
    - NULLIF(regexp_replace(used::text, '[^-.0-9]', '', 'g'), '')::numeric AS available,
  100.0*NULLIF(regexp_replace(used::text, '[^-.0-9]', '', 'g'), '')::numeric
    / NULLIF(NULLIF(regexp_replace(capacity::text, '[^-.0-9]', '', 'g'), '')::numeric, 0) AS utilization_pct
FROM warehouses
WHERE 100.0*NULLIF(regexp_replace(used::text, '[^-.0-9]', '', 'g'), '')::numeric
    / NULLIF(NULLIF(regexp_replace(capacity::text, '[^-.0-9]', '', 'g'), '')::numeric, 0) > 80;

regexp_replace(…, '[^-.0-9]', '', 'g') deletes every character that is not a digit, dot, or minus — so "1,000 units" would become 1000. NULLIF(…, '') turns an emptied string into NULL, and the inner NULLIF(…, 0) guards against divide-by-zero.

4. Strategic explanations

Why sanitise at all — defensive parsing of dirty columns

The regex scrub only does anything if capacity/used are text columns holding values like "1,000", "2000 sqft", or "950u" — a common reality when warehouse data is ingested from spreadsheets or legacy systems. Stripping to [-.0-9] then casting to numeric rescues the number. If the columns are already a proper numeric type (as here), the regex is dead weight: the real fix is to clean the schema so capacity is numeric and the query becomes 100.0 * used / NULLIF(capacity, 0). Say that out loud — sanitising in every query is a smell.

The double-NULLIF — two different jobs

There are two nested NULLIFs on the capacity term and they are not redundant. The inner NULLIF(regexp_replace(…), '') handles a value that scrubbed down to an empty string (e.g. "N/A") — casting '' to numeric would error, so it becomes NULL first. The outer NULLIF(…, 0) handles a genuine zero capacity, preventing divide-by-zero. Drop either and a specific dirty-data row crashes the report.

The repeated-expression trap — compute utilization once

The utilization formula is written three times — twice in SELECT, once in WHERE. That is error-prone (change one, forget another) and re-evaluates the regex per usage. You cannot reference a SELECT alias in WHERE, so lift it into a CTE or subquery and filter the clean column:

WITH w AS (
  SELECT warehouse_id, capacity, used,
         100.0 * used / NULLIF(capacity, 0) AS utilization_pct
  FROM warehouses
)
SELECT * FROM w WHERE utilization_pct > 80;

One definition, one place to fix it — the maintainability point an interviewer wants raised.

The boundary — is exactly 80% on the list?

The filter is > 80, strict, so a warehouse at precisely 80.0% is excluded. Whether the watch list is "above 80" or "80 and above" is a business decision — confirm it and switch to >= 80 if needed. With float-derived percentages, also beware that a value you think is exactly 80 may be 79.9999; numeric arithmetic (as used here) avoids that, which is another reason to cast to numeric not float.

The NULL-utilization row silently drops

If a warehouse has NULL or zero capacity, utilization_pct is NULL, and NULL > 80 is unknown — the row is filtered out. A warehouse whose capacity is unknown is arguably the one ops most needs flagged. Decide deliberately: surface those rows separately with WHERE utilization_pct > 80 OR capacity IS NULL, rather than letting them vanish.

Senior signal: explain the regex scrub as defensive parsing for dirty text columns — while noting the real fix is a clean numeric schema — distinguish the two nested NULLIFs (empty-string cast vs divide-by-zero), refactor the thrice-repeated formula into a CTE, and treat the strict > 80 boundary and the dropped NULL-capacity rows as explicit business decisions.
