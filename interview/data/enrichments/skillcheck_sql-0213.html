Finding pairs of states whose streaming totals are close is a self-join. The trap is the join predicate that removes duplicates. A naive self-join produces every state paired with itself, and every real pair twice — once as (A,B) and once as (B,A). The fix is one operator: join on a.state < b.state, not a.state <> b.state.

1. The source data — state_streams

statetotal_streams
NC34569
SC33999
CA98324
MA19345
GA34000

Goal: pairs of states whose totals differ by less than 1000, each pair listed once.

2. The query — self-join with an ordering predicate

SELECT a.state AS state_a, b.state AS state_b,
       ABS(a.total_streams - b.total_streams) AS stream_gap
FROM state_streams a
JOIN state_streams b ON a.state < b.state
WHERE ABS(a.total_streams - b.total_streams) < 1000
ORDER BY a.state, b.state;

a.state < b.state does two jobs at once: it drops self-pairs (a state is never < itself) and it keeps only one ordering of each real pair (the alphabetically-earlier state is always a).

3. The result

state_astate_bstream_gap
GANC569
GASC1
NCSC570

Three pairs. GA (34000) sits within 1000 of both NC (34569) and SC (33999); NC and SC are 570 apart. CA and MA are far from everything.

4. Strategic explanations

Why <> is not enough

An unrestricted self-join is a cross join: 5 states → 25 ordered pairs. a.state <> b.state removes only the 5 self-pairs, leaving 20 — still both (GA,NC) and (NC,GA). The result set is doubled. a.state < b.state admits exactly one ordering of each unordered pair, giving the n choose 2 distinct combinations and no self-pairs.

The key must be unique and totally ordered

The < trick relies on the join key having a stable, unique ordering. state is a unique text key, so < works. If states could repeat, < would wrongly drop the pair of two equal-keyed-but-distinct rows; you would then compare a unique surrogate id instead (a.id < b.id).

The predicate cannot live in the SELECT

The closeness test ABS(...) < 1000 belongs in WHERE (or the ON clause). Computing it as a SELECT column and hoping to filter later does not remove rows; and a window or aggregate cannot stand in for a row-level pair filter. Here it is a plain row predicate.

Dialect notes

The pattern is fully portable — PostgreSQL, MySQL, SQL Server, Oracle, Snowflake, BigQuery all evaluate a.state < b.state identically for text. An explicit CROSS JOIN ... WHERE a.state < b.state is equivalent to the JOIN ... ON form shown; pick whichever reads more clearly.

Senior signal: to enumerate unordered pairs from a self-join, join on a.key < b.key — the strict inequality removes self-pairs and the mirror duplicate in one stroke, where <> leaves every pair counted twice. Use a unique, totally-ordered key (a surrogate id if the natural key can repeat).
