A restaurant analytics team (think Danny's Diner) wants, for each customer, the first item they ever bought in each menu category — first dessert, first drink, first main. "First" is by date, with deterministic tiebreaks. This is the classic first-per-group problem, and the interviewer is checking whether you reach for ROW_NUMBER() over a fragile GROUP BY + self-join, and whether your tiebreak chain is fully specified.

1. The source data — sales joined to menu

Two edge cases. Customer A bought two mains on the same first date (sushi 10 and curry 15) — the tie must be broken by lower price → sushi. And customer C only ever bought a main, so C must produce one row, not three: a missing category yields no row.

2. The SQL — ROW_NUMBER per (customer, category)

SELECT customer_id, category, name AS product_name, order_date
FROM (
  SELECT s.customer_id, m.category, m.name, s.order_date,
         ROW_NUMBER() OVER (PARTITION BY s.customer_id, m.category
                            ORDER BY s.order_date, m.price, m.product_id) AS rn
  FROM sales s JOIN menu m USING (product_id)
) t
WHERE rn = 1
ORDER BY customer_id, category;

ROW_NUMBER() numbers each customer's purchases within each category, ordered earliest-first. Keeping rn = 1 in the outer query picks exactly one row per (customer, category) — the first purchase. Categories the customer never bought simply have no rows to number, so they vanish naturally.

4. Strategic explanations

Why ROW_NUMBER is the right tool here

"First per group" needs exactly one row per partition. ROW_NUMBER() guarantees a strict 1,2,3,… sequence, so rn = 1 always returns one row. RANK() would return multiple rows for the first position if the full ORDER BY ties — wrong for "the first product". ROW_NUMBER is the precise instrument: pick one, break all ties.

Why every tiebreak must be in the ORDER BY

The spec says: date, then lower price, then product_id. All three must appear in the window's ORDER BY. Stop at order_date alone and A's two same-day mains tie — the engine picks one arbitrarily, and the result is non-deterministic across runs and plans. Stop at order_date, price and you are safe here, but a future row with an equal price would re-introduce non-determinism; product_id (a unique key) is the final guarantee. A reproducible "first" needs a total order.

Why missing categories produce no row — for free

The inner query joins sales to menu, so it only contains categories the customer actually purchased. ROW_NUMBER partitions over those; a category with no purchases simply has no partition. No LEFT JOIN, no HAVING COUNT > 0 needed — the absence falls out of the data. If the spec wanted a row-with-NULL for un-purchased categories, you would instead CROSS JOIN customers to the distinct category list and LEFT JOIN the picks.

The classic alternatives — and why they are weaker

The pre-window idiom is GROUP BY customer, category with MIN(order_date), then a self-join back to recover name. It works but is verbose, and the self-join can return two products if two share the min date — you must re-apply the price/id tiebreak in the join, easy to forget. DISTINCT ON (customer_id, category) ... ORDER BY customer_id, category, order_date, price, product_id is the elegant Postgres-specific form — terse, but not portable. ROW_NUMBER is the portable, unambiguous standard.

Senior signal: solve first-per-group with ROW_NUMBER() OVER (PARTITION BY group ORDER BY ...) filtered to rn = 1; put every tiebreak — down to a unique key — in the ORDER BY so the pick is deterministic; let an inner join, not a LEFT JOIN, drop empty groups when the spec wants no row for them.
