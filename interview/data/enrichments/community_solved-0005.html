A Community window-function exercise: from a yearly sales table, list the brand-years where revenue went up versus the previous year. The reference solution uses LAG — the right tool — but it ships with a default-value choice that quietly invents a fake "increase" in every brand's first year. The interviewer wants to see whether you catch it.

1. The source data — brand_amount

yearbrandamount
2020apple100
2021apple150
2022apple120
2020nike80
2021nike60
2021sony200

apple rises then falls; nike only falls; sony has a single year — no previous year to compare against. That lone sony row is the edge case the LAG default mishandles.

2. The SQL — LAG then filter on the delta

with t1 as
(
select  brand
    ,year as current_year
    ,(year-1) as previous_year
    ,amount
    ,LAG(amount,1,0) OVER(PARTITION BY brand ORDER BY year asc) as amount_previous_yr
    ,(amount - LAG(amount,1,0) OVER(PARTITION BY brand ORDER BY year asc)) as amount_change
from brand_amount
)
select brand,concat(previous_year,'-',current_year) as years
    ,amount_change
from t1
where amount_change > 0

LAG(amount, 1, 0) fetches the prior year's amount within each brand; the 0 is the default used when there is no prior row. amount_change is the year-over-year delta, and the outer query keeps only positive deltas.

3. The resulting analysis

brandyearsamount_change
apple2019-2020100
apple2020-202150
nike2019-202080
sony2020-2021200

The genuine increase is real: apple 2020→2021 was 100→150, a +50. But look at the other three rows — apple 2019-2020, nike 2019-2020, sony 2020-2021 are all first-year artefacts. There is no year 2019 in the data. LAG returned its default 0, so the delta is the entire first-year amount, which is always positive, so every brand's debut row passes the > 0 filter. Three of the four "increases" are noise.

4. Strategic explanations

The LAG(..., 1, 0) default trap

The third argument to LAG is the value used when the offset row does not exist. Choosing 0 means "treat the year before a brand's first as $0 revenue" — which fabricates a 0→first-amount jump and labels it growth. The fix is to drop the default so the missing row yields NULL:

LAG(amount, 1) OVER (PARTITION BY brand ORDER BY year)

Then amount - NULL is NULL, NULL fails > 0, and first-year rows fall out cleanly. apple 2020→2021 stays; the three phantom rows vanish. Same data, correct answer — purely from one argument.

The previous_year column lies too

The solution computes previous_year as year - 1, then labels rows 2019-2020. But 2019 is not in the table — the label asserts a comparison that never happened. If years can have gaps (a brand skips 2021), year - 1 is simply wrong. Derive the real prior year from the window instead:

LAG(year) OVER (PARTITION BY brand ORDER BY year) AS previous_year

so the label always names the actual compared years. The reference even left this exact line commented out — it knew.

Interviewer follow-up: "show the percentage change"

Absolute deltas favour big brands. Recruiters usually want growth rate:

ROUND(100.0 * (amount - LAG(amount) OVER w)
       / LAG(amount) OVER w, 1) AS pct_change
FROM brand_amount
WINDOW w AS (PARTITION BY brand ORDER BY year)

Guard the denominator: if a prior amount can be 0, percentage change is undefined — NULLIF(LAG(amount) OVER w, 0).

Edge case: ties (flat years) and the > operator

A year where revenue is unchanged gives amount_change = 0, which fails > 0 — correctly excluded, since flat is not an increase. Be deliberate about the boundary: > 0 means "strictly grew", >= 0 would mean "did not shrink". The interviewer may be probing exactly that wording.

Sorting and determinism

The final query has no ORDER BY, so row order is whatever the planner returns. For a report, sort explicitly — ORDER BY brand, current_year — so the output is stable across runs and engines.

Senior signal: recognise that LAG(..., 1, 0) manufactures a fake first-year increase, switch the default to NULL so debut rows drop out, derive previous_year from LAG(year) rather than year - 1 to survive gaps, and state whether "increase" means strictly greater or merely non-decreasing.
