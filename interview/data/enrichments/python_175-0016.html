Subarray Sum Equals K is the question Google uses to find out if you truly understand prefix sums — not just "running total" but the trick of looking up a previous prefix sum in a hash map. It is also a deliberate trap for the sliding-window reflex: the array can contain negatives, which quietly invalidates the window technique most candidates reach for first.

1. The input data

An integer array (note the negative value) and a target k:

nums = [1, 2, 3, -1, 2]
k    = 3
subarraySum(nums, k)

The question: how many contiguous subarrays sum to exactly 3?

2. The approach

The identity: a subarray nums[i..j] sums to k exactly when prefix[j] - prefix[i-1] == k, i.e. prefix[i-1] == prefix[j] - k. So as the code sweeps left to right maintaining a running prefix sum s, the count of subarrays ending at the current index equals the number of earlier prefix sums equal to s - k. A defaultdict(int) named cnt stores how many times each prefix sum has occurred — a count, not just presence, because the same prefix sum can recur and each occurrence is a distinct valid start. It is seeded with cnt[0] = 1 to represent the empty prefix, which lets a subarray starting at index 0 be counted. Per element: add to s, add cnt[s - k] to the result, then increment cnt[s]. One pass, no nested loop.

3. The output

11cnt[-2] = 00{0:1, 1:1}

Final result: 2 — the subarrays [1, 2] (indices 0–1) and [3] (index 2) each sum to 3. The hit at cnt[0] catches [1,2]; the hit at cnt[3] catches the single [3].

4. Strategic explanations

Complexity

O(n) time — a single pass, O(1) hash operations per element — and O(n) space for the prefix-sum counts in the worst case (all sums distinct). The brute force, summing every (i, j) pair, is O(n²); for the input sizes Google tests, only the linear version passes.

The idiomatic detail — count, and the cnt[0]=1 seed

Two lines carry the whole solution. Using a counter rather than a set is essential: if prefix sum 5 occurred three times, a subarray ending now with s - k == 5 closes three distinct subarrays. And seeding cnt[0] = 1 encodes the empty prefix — without it, any subarray that starts at index 0 (like [1,2] here) goes uncounted. collections.Counter works equally well in place of defaultdict(int).

The edge case — negatives, zeros, and k = 0

Negative numbers are the point of the question — they make prefix sums non-monotonic, which is fine for this approach but fatal for sliding windows. Zeros and k = 0 are handled naturally: a run of zeros produces repeated prefix sums, and each repeat is correctly counted as a zero-sum subarray. An empty array returns 0.

The follow-up — return the subarrays, or longest one

"Return the actual subarrays, not the count." Store a list of indices per prefix sum instead of a bare count, then for each match emit the bounds. "Find the longest subarray summing to k" flips it: store the earliest index for each prefix sum and maximize j - i — the same prefix-sum-hashmap skeleton, retuned.

The trap — the sliding window

The instinctive approach — expand a window, shrink it when the sum exceeds k — is correct only for non-negative arrays, where the running sum increases monotonically. With a negative number, shrinking the window can never be justified, because a later element might pull the sum back down. Reaching for a window here, without first asking "can the values be negative?", is the single most common failure on this question.

Senior signal: a senior asks "can the numbers be negative?" up front, rejects the sliding window when the answer is yes, counts prefix-sum frequencies rather than presence, and seeds cnt[0] = 1 for subarrays anchored at index 0 — a junior writes a two-pointer window that silently misreports on negatives.
