E-commerce and retail data teams often store promotional details as free-text descriptions rather than structured fields: "A product description says 'Save 20% — was $89.99, now $71.99, offer valid until 2025-06-30.' Extract the discount percentage, original price, final price, and expiry date as typed columns." This question tests multi-pattern regex extraction, the substring(text FROM pattern) capture-group mechanism, and the NULLIF / cast chain for safe type conversion.

1. The source data — products

10420% discount, was $80.00, now $64.00, valid 2025-07-01

Product 103 has no discount percentage, no "was" price, no date — all four extracted columns should be NULL. Product 105 has a "now" price but no "was" price and no percentage. These NULL cases are the edge cases the interview is testing.

2. The SQL — four simultaneous regex extractions

SELECT product_id,
       description,
       NULLIF(substring(description from '(\d+)\s*%'), '')::numeric AS discount_pct,
       NULLIF(substring(description from 'was\s*\$([0-9.]+)'), '')::numeric AS original_price,
       NULLIF(substring(description from 'now\s*\$([0-9.]+)'), '')::numeric AS final_price,
       NULLIF(substring(description from '\d{4}-\d{2}-\d{2}'), '')::date AS discount_until
FROM products;

4. Strategic explanations

Capture groups in substring(text FROM pattern)

Postgres’s substring(text FROM pattern) with POSIX regex has two modes: if the pattern contains no capturing group (...), the entire match is returned; if a capturing group exists, only the content of the first group is returned. The patterns for discount_pct and prices use explicit groups — (\d+)\s*% captures just the digits, not the % sign; was\s*\$([0-9.]+) captures just the numeric part, not the "was $" prefix. The date pattern has no group, so the full YYYY-MM-DD match is returned directly. Understanding when to use a group vs no group is the regex literacy the interviewer is testing.

Why NULLIF(..., '') before casting

When a pattern does not match, substring returns NULL in Postgres. However, if the regex matches but the captured group is empty, an empty string is returned. NULLIF(expr, '') normalises both cases: NULL stays NULL, empty strings become NULL, so the subsequent ::numeric or ::date cast never receives an empty string (which would cause a runtime error). This is a defensive programming pattern worth naming explicitly.

Pattern design decisions — anchoring and specificity

The "was price" pattern was\s*\$([0-9.]+) is anchored to the keyword "was", so a price appearing without the keyword (like product 103’s "$29.99") is not mistakenly captured. Similarly, now\s*\$([0-9.]+) requires the "now" keyword before the price. This keyword-anchoring is essential when descriptions contain multiple numeric values. Greedy vs lazy quantifiers also matter: [0-9.]+ is greedy and would consume "49.99.00" incorrectly — using [0-9]+(?:\.[0-9]+)? would be more precise for prices.

Computed validation column — sanity check

A useful extension adds a computed validation: ROUND(original_price * (1 - discount_pct/100), 2) AS computed_final and then compares it to final_price. Discrepancies flag data-entry errors or rounding conventions in the source descriptions. This analytical extension turns an extraction query into a data-quality check.

Dialect notes: regex extraction
  • Postgres (solution dialect): substring(col FROM pattern) with POSIX regex and optional capture group. Works as written.
  • Snowflake: REGEXP_SUBSTR(col, pattern, 1, 1, 'e', group_num). The 'e' flag enables sub-expression extraction. Example: REGEXP_SUBSTR(description, '(\\d+)\\s*%', 1, 1, 'e', 1).
  • BigQuery: REGEXP_EXTRACT(col, r'(\d+)\s*%') — returns the first capture group.
  • MySQL 8+: REGEXP_SUBSTR(col, '[0-9]+(?:\\.[0-9]+)?') (no group extraction in MySQL 8.0; full-match only).

Add the validation column extension as a data-quality signal.
