Quarter-end at a SaaS company. The CRO needs the standard board-pack revenue table: months as rows, regions as columns. The raw revenue table is in narrow / long format — one row per (month × region). Tableau would struggle with that shape; the board wants wide. Conditional aggregation is the canonical SQL way to pivot without leaving the database.

2. The SQL — conditional aggregation pivot

One SUM(CASE WHEN ...) per output column. Group by the dimension that becomes the row (month). The CASE collapses to NULL for non-matching rows, which SUM ignores by default — exactly what you want.

SELECT
  month,
  SUM(CASE WHEN region = 'North' THEN amount END) AS north,
  SUM(CASE WHEN region = 'South' THEN amount END) AS south,
  SUM(CASE WHEN region = 'East'  THEN amount END) AS east,
  SUM(CASE WHEN region = 'West'  THEN amount END) AS west,
  SUM(amount)                                     AS total
FROM revenue
GROUP BY month
ORDER BY month;

4. Strategic explanations

Conditional aggregation vs PIVOT — portability wins

Snowflake, BigQuery and SQL Server all have a real PIVOT operator. Postgres and SQLite do not. Conditional aggregation works everywhere — it's the dialect-portable answer. Mention PIVOT in the interview as the engine-specific shortcut, but lead with SUM(CASE WHEN).

The "missing region" problem — NULL vs 0

If the revenue table has no East row for January (the team didn't operate there yet), the CASE returns NULL → SUM returns NULL → the board sees an empty cell. To force a zero, wrap with COALESCE(SUM(CASE WHEN region = 'East' THEN amount END), 0). Boards usually prefer 0 over blank; product analytics often prefer NULL to distinguish "no data" from "$0 in revenue". State your choice.

Why this beats wide-table joins

An alternative is to maintain a revenue_wide(month, north, south, east, west) table and JOIN. That's a denormalisation: you now have two sources of truth for the same data, and any region change requires a schema migration. The conditional aggregation produces the wide shape on demand — the revenue table stays normalised.

When to use a real PIVOT

If the region list is dynamic (users keep adding new regions), SUM(CASE WHEN) doesn't scale — every new region requires a code change. At that point, switch to a true PIVOT (Snowflake/BigQuery) with ... FOR region IN (ANY ORDER BY total DESC LIMIT 8), or generate the SQL dynamically in the BI layer.

The "growth columns" you'll be asked for next

Inevitably the CFO comes back: "What about month-over-month growth?" The clean answer is to layer a window function on the pivot result:

SELECT month, north, south, east, west, total,
  ROUND(100.0 * (total - LAG(total) OVER (ORDER BY month))
        / NULLIF(LAG(total) OVER (ORDER BY month), 0), 1) AS mom_growth_pct
FROM (
  SELECT month,
    SUM(CASE WHEN region = 'North' THEN amount END) AS north,
    SUM(CASE WHEN region = 'South' THEN amount END) AS south,
    SUM(CASE WHEN region = 'East'  THEN amount END) AS east,
    SUM(CASE WHEN region = 'West'  THEN amount END) AS west,
    SUM(amount)                                     AS total
  FROM revenue GROUP BY month
) ORDER BY month;
-- Mar growth = 8.9% MoM. The board chart now writes itself.
Senior signal: mention conditional aggregation by name, justify the choice over PIVOT with portability, and pre-empt the inevitable follow-up by knowing how to layer a LAG window on top for MoM growth. The interview almost always asks for that next.
