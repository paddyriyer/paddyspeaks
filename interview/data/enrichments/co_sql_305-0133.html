Hulu's CRM team wants the most recent order for every customer — the input to a "we miss you" winback email when the last order is older than 60 days. The naive answer (MAX(order_date) GROUP BY customer_id) loses the order_id and other columns. The senior question becomes: how do you keep the full row for the latest event per group?

4. Strategic explanations

Why the naive GROUP BY fails
-- WRONG — loses order_id
SELECT customer_id, MAX(order_date) FROM orders GROUP BY customer_id;

You get the date but not the order_id — and you can't just SELECT order_id, customer_id, MAX(order_date) ... GROUP BY customer_id because order_id isn't in the GROUP BY (Postgres errors; MySQL silently picks any value — the famous ONLY_FULL_GROUP_BY trap). The "max of one column, full row of another" need is what makes this a senior question.

Three production-grade idioms — pick deliberately
-- A. Correlated subquery — original solution, clear semantics, can be slow
SELECT * FROM orders o
WHERE order_date = (SELECT MAX(order_date) FROM orders WHERE customer_id = o.customer_id);

-- B. Window function (ROW_NUMBER) — modern default
SELECT order_id, customer_id, order_date FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC) AS rn
  FROM orders
) WHERE rn = 1;

-- C. Self-join via aggregate CTE
WITH latest AS (
  SELECT customer_id, MAX(order_date) AS max_dt FROM orders GROUP BY customer_id
)
SELECT o.* FROM orders o JOIN latest l
  ON o.customer_id = l.customer_id AND o.order_date = l.max_dt;

The tie problem — what if two orders share the latest date?

If C_01 had two orders on 2026-04-22, the correlated-subquery and self-join forms return both rows; ROW_NUMBER returns one (arbitrarily). Pick based on intent:

  • "Latest activity timestamp" → tie-emit is fine (correlated / self-join).
  • "The single most-recent order_id" → force determinism with ROW_NUMBER() OVER (... ORDER BY order_date DESC, order_id DESC).
  • "All ties as one row, comma-separated" → ARRAY_AGG(order_id ORDER BY order_date DESC)[1:].
Postgres' DISTINCT ON — the cleanest single-best-row idiom
SELECT DISTINCT ON (customer_id) *
FROM orders
ORDER BY customer_id, order_date DESC;

Postgres-only, but unbeatable for "exactly one row per group, ordered by ...". The companion to GROUP BY for "give me the row, not just the aggregate".

Top-K per group — ROW_NUMBER scales naturally

The follow-up: "now give me the 3 most recent orders per customer." Just change the filter:

SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC) AS rn
  FROM orders
) WHERE rn <= 3;

Try doing this with a correlated subquery and you get an O(n²) mess. ROW_NUMBER is the only sane choice past N=1.

The recency-bucket follow-up — the real winback ask

The CRM team's actual question is segmentation by recency:

WITH last_order AS (
  SELECT customer_id, MAX(order_date) AS last_dt FROM orders GROUP BY customer_id
)
SELECT
  CASE
    WHEN CURRENT_DATE - last_dt <= 30 THEN 'active'
    WHEN CURRENT_DATE - last_dt <= 60 THEN 'lapsing'
    WHEN CURRENT_DATE - last_dt <= 90 THEN 'winback'
    ELSE 'churned'
  END AS recency_bucket,
  COUNT(*) AS customers
FROM last_order GROUP BY 1
ORDER BY MIN(last_dt) DESC;
The index that makes correlated-subquery perform

The query above does N+1 lookups against orders. With a composite index on (customer_id, order_date DESC), the inner MAX is a single index-only seek — constant time per outer row. Without that index, it's a full scan per customer. Always check the explain plan when reaching for correlated subqueries.

Senior signal: diagnose the GROUP-BY-loses-columns problem in one sentence, contrast all three idioms (correlated, ROW_NUMBER, self-join CTE) with their performance and tie behaviour, and surface Postgres' DISTINCT ON as the cleanest dialect-specific form. Top-K-per-group via ROW_NUMBER and the recency-bucket segmentation are the over-the-bar finish.
