Netflix coined the term: bingewatching = three or more episodes of the same show within 24 hours. The metric drives the auto-play algorithm, the "Continue Watching" row priority, and the show-greenlighting model (a binge-friendly show like Stranger Things justifies a 9-figure budget; a slow-burn like The Crown does not). The query is a sliding-window count — one of the most underused window-function tricks in SQL.

2. The SQL — COUNT over a 24-hour RANGE window

WITH ranked AS (
  SELECT user_id, show_id, episode_id, started_at,
    COUNT(*) OVER (
      PARTITION BY user_id, show_id
      ORDER BY started_at
      RANGE BETWEEN INTERVAL '24 hours' PRECEDING AND CURRENT ROW
    ) AS eps_in_last_24h
  FROM episode_plays
)
SELECT DISTINCT user_id, show_id
FROM ranked
WHERE eps_in_last_24h >= 3;

3. Strategic explanations

The RANGE BETWEEN INTERVAL trick — sliding time windows

The default window frame counts physical rows (ROWS BETWEEN N PRECEDING). For "within 24 hours" you need a time-based window: RANGE BETWEEN INTERVAL '24 hours' PRECEDING AND CURRENT ROW. Behind the scenes the engine looks back from the current row to all rows whose ORDER BY column is within 24 hours — could be 3 rows, could be 30, could be 1.

Self-join workaround for SQLite / MySQL
SELECT DISTINCT a.user_id, a.show_id
FROM episode_plays a
JOIN episode_plays b
  ON a.user_id = b.user_id
 AND a.show_id = b.show_id
 AND b.started_at BETWEEN a.started_at - INTERVAL '24 hours' AND a.started_at
GROUP BY a.user_id, a.show_id, a.started_at
HAVING COUNT(*) >= 3;

O(n²) without index; in production add a composite index on (user_id, show_id, started_at).

The "3 distinct episodes" precision

If a user watches E_01 three times in 24 hours (re-watch), the original query counts that as 3 — technically a binge by row count, semantically not a binge by content. Fix:

COUNT(DISTINCT episode_id) OVER (PARTITION BY ...)

...except COUNT DISTINCT in a window function is not supported in most engines. Workaround: dedup first via DISTINCT in a CTE, then run the window count on the deduped rows.

The session-aware version — back-to-back not just within 24h

"3 episodes within 24 hours" includes "1 in the morning, 2 at night" — arguably not a binge. The truer binge is back-to-back: each episode starts within 30 minutes of the previous one ending. Use LAG and a sessionizer:

WITH sessions AS (
  SELECT user_id, show_id, episode_id, started_at,
    SUM(CASE WHEN started_at - LAG(started_at) OVER (
                  PARTITION BY user_id, show_id ORDER BY started_at)
              > INTERVAL '30 minutes' THEN 1 ELSE 0 END
    ) OVER (PARTITION BY user_id, show_id ORDER BY started_at) AS session_id
  FROM episode_plays
)
SELECT user_id, show_id, session_id, COUNT(*) AS eps_in_session
FROM sessions GROUP BY user_id, show_id, session_id
HAVING COUNT(*) >= 3;

This is the Netflix-published definition of "binge session". The 30-minute threshold is the auto-play timeout.

The "completion-aware" binge

If users skip-around (start E_01, hit "next episode", abandon, jump to E_05), is that 5 episodes started or 1 episode actually watched? Production binge-detection joins to episode_completion and only counts plays that crossed an 80% completion threshold:

WHERE completion_pct >= 0.80
The downstream metric — "binge-worthiness score"

Aggregate over titles to get a binge propensity score per show:

SELECT show_id,
  COUNT(DISTINCT (user_id, session_id)) AS binge_sessions,
  COUNT(DISTINCT user_id)               AS unique_bingers,
  AVG(eps_in_session)                   AS avg_eps_per_binge
FROM (... session-aware query ...)
GROUP BY show_id
ORDER BY binge_sessions DESC;

Stranger Things, You, Squid Game = top of leaderboard. The binge-score is what the content org uses to argue for season 2 budget.

The release-strategy implication

Binge-friendly shows get the all-episodes-day-one drop. Slow-burn shows get the weekly-release schedule. The query's output literally drives Netflix's launch-strategy decisions.

Cross-show binge — a different question

"Watched 3 episodes of any show in 24h" is a different metric — user-level engagement, not show-level. Drop the show_id partition:

OVER (PARTITION BY user_id ORDER BY started_at RANGE BETWEEN INTERVAL '24 hours' PRECEDING AND CURRENT ROW)
Senior signal: name the RANGE BETWEEN INTERVAL sliding-window pattern crisply, contrast against ROWS-based windows and against the self-join fallback for non-supporting dialects, then upgrade to the session-aware definition (LAG + cumulative-flag sessionization) to match Netflix's published "binge session" definition. The completion-pct gate and downstream binge-worthiness leaderboard tie the query to actual greenlight decisions — the over-the-bar finish.
