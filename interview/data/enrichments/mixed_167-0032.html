A pricing analyst wants the five product categories with the highest average price — a quick read on which segments anchor the premium end of the catalogue. The reference solution groups by category, averages the price, and takes the top five by descending average. It runs without error, but it has a subtle correctness flaw the interviewer is hunting for: a category whose prices are all NULL sorts to the top.

Collectibles is the trap: its only product has a NULL price. AVG over an all-NULL group returns NULL — and the interview turns on where that NULL lands once you sort descending.

2. The SQL — group, average, take top five

-- original solution (flagged: NULL avg sorts first under DESC)
SELECT category, AVG(price) AS avg_price FROM products
GROUP BY category ORDER BY avg_price DESC LIMIT 5;

-- corrected: push NULL averages to the bottom
SELECT category, AVG(price) AS avg_price FROM products
GROUP BY category ORDER BY avg_price DESC NULLS LAST LIMIT 5;

GROUP BY category forms one row per category; AVG(price) averages each group's non-null prices; ORDER BY ... DESC LIMIT 5 keeps the five highest.

With the original query, Collectibles (avg NULL) appeared first and Stationery was pushed out of the top five — because Postgres treats NULL as the largest value under DESC. The corrected NULLS LAST demotes the empty category, and Stationery correctly fills the fifth slot.

4. Strategic explanations

Why the original is broken — NULL is "largest" under DESC

AVG of a group whose prices are all NULL is NULL, not 0 — AVG ignores NULLs and has nothing left to average. Under ORDER BY avg_price DESC, Postgres sorts NULLs first (its default for descending). So a category with no priced products at all ranks above Jewelry — a clearly wrong "top categories" answer. The fix is one keyword, NULLS LAST, or a HAVING COUNT(price) > 0 to drop priceless categories outright.

AVG ignores NULL prices within a group too

The NULL hazard is not only about all-NULL groups. If Jewelry had a third product with a NULL price, AVG(price) would still return 2100 — it divides the sum of the two real prices by 2, not by 3. That is usually the desired behaviour ("average of known prices"), but if a NULL price should count as zero you must write AVG(COALESCE(price, 0)). State which definition of "average" the business means.

Single-product categories average to that one price

Apparel and Stationery each have exactly one product, so their AVG is simply that product's price — 25.00 and 3.25. That is mathematically fine but often not what a "top categories" report wants: a single-item category can dominate or vanish on the strength of one SKU. A senior answer adds a volume guard, e.g. HAVING COUNT(*) >= 3, so the ranking reflects categories with enough products to be meaningful.

LIMIT with ties at the boundary is non-deterministic

LIMIT 5 takes five rows, but if the 5th and 6th categories had an equal average, which one is kept is arbitrary — ORDER BY avg_price alone does not break that tie. Add a deterministic secondary key (ORDER BY avg_price DESC NULLS LAST, category) so the result is reproducible. For "include all ties at the cutoff" semantics, RANK() OVER (ORDER BY avg_price DESC) filtered to <= 5 is the correct tool instead of LIMIT.

NULL category — GROUP BY keeps it as its own bucket

If some products had a NULL category, GROUP BY category would collect them into a single group keyed by NULL (all NULLs group together, even though NULL = NULL is never true — grouping uses "not distinct" semantics). That phantom NULL-category row could occupy a top-five slot. If uncategorised products should be excluded, add WHERE category IS NOT NULL.

Senior signal: recognise that AVG over an all-NULL group returns NULL and Postgres sorts NULL first under DESC — so the naive query ranks a priceless category #1; fix it with NULLS LAST or HAVING COUNT(price) > 0, guard single-product categories with a HAVING COUNT(*) threshold, and add a tie-breaker so LIMIT is deterministic.
