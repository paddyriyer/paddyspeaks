Qualcomm's PMO wants an early-warning report: which projects have spent more than their approved budget, and by how much. It is a one-line filtered query. The interview is in the details that a one-liner hides — the strict-vs-inclusive boundary, what NULL budgets or spend do to the comparison, and whether "over budget" should be a percentage rather than an absolute.

1. The source data — projects

Three rows are deliberate edge cases. Cronos spent exactly its budget — over, or not? Eos is $1 under — the just-barely-safe row. Deimos is dramatically over. Whether Cronos appears is decided entirely by > vs ≥.

2. The SQL — filter on a computed difference

SELECT project_id, spent - budget AS over_budget FROM projects WHERE spent > budget;

The WHERE keeps only projects where spent strictly exceeds budget; the select list reports the overrun amount spent - budget. One row per over-budget project.

Two projects are flagged: Apollo $2,000 over, Deimos $7,500 over. Cronos — spent exactly to budget — is excluded because the operator is strict >; spending to the penny is "on budget", not "over". Eos at $1 under is correctly clear. No ORDER BY, so row order is not guaranteed.

4. Strategic explanations

The boundary — is "exactly on budget" over?

Cronos is the entire reason this question exists. spent > budget excludes it; spent >= budget would flag it. The business answer is almost always "spending exactly to budget is fine" — so strict > is right — but state the choice rather than letting the operator decide silently. The deeper PMO point: a project at 100% of budget with months of work left is arguably the bigger risk than one finished 1% over. "Over budget" and "at risk" are different questions.

The NULL trap — missing budgets vanish

If a project has a NULL budget (not yet approved) or NULL spent, then spent > budget is unknown, and WHERE keeps only true rows — so that project silently drops out of the report. A project with no approved budget but real spend is arguably the most urgent case, and it disappears. Surface those rows explicitly instead of losing them:

SELECT project_id, spent - budget AS over_budget,
       CASE WHEN budget IS NULL THEN 'no approved budget'
            WHEN spent  IS NULL THEN 'spend not reported'
            ELSE 'over budget' END AS flag
FROM projects
WHERE spent > budget OR budget IS NULL OR spent IS NULL;
Absolute overrun vs percentage overrun

The query reports raw dollars. But $2,000 over on a $10,000 project (Apollo, +20%) is a far worse signal than $2,000 over on a $2M project (+0.1%). A PMO usually wants severity as a ratio:

SELECT project_id,
       spent - budget AS over_budget,
       round(100.0 * (spent - budget) / NULLIF(budget, 0), 1) AS over_pct
FROM projects
WHERE spent > budget
ORDER BY over_pct DESC;

Note NULLIF(budget, 0): a project with a zero budget would otherwise divide by zero and abort the whole query. And 100.0 forces float math so the percentage is not integer-truncated.

The follow-up — total exposure

Leadership rarely wants the row list alone; they want the aggregate: "what is our total overrun across the portfolio?" That is SELECT SUM(spent - budget) FROM projects WHERE spent > budget — here 2000 + 7500 = 9500. A subtle point: only sum the over-budget projects; including under-budget projects would let Boreas's $2,000 underspend net against the overruns and understate the true exposure.

Should closed projects count?

The schema carries a status column. An overrun on an active project is actionable; an overrun on a completed project is history, and on a cancelled one it may be noise. An early-warning report probably wants WHERE spent > budget AND status = 'active'. Confirm the scope — the right filter on status matters as much as the budget comparison itself.

Senior signal: justify strict > so "exactly on budget" is not flagged, recognise that NULL budget or spend silently drops projects from a WHERE and surface them deliberately, offer percentage overrun (guarded by NULLIF) as the better severity measure, and scope the report by project status.
