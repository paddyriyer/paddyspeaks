Detecting a run of three or more strictly increasing values in time-ordered data is a sequence-pattern problem that appears in fraud detection (escalating transaction amounts), sports analytics (win streaks), and financial monitoring (sustained growth). The interviewer is probing the double-LAG technique: compute the previous two values as window columns, then filter for positions where all three form a strict ascending triple — a purely set-based approach that avoids recursive CTEs or self-joins.

2. The SQL — double-LAG window function

WITH w AS (
  SELECT customer_id,
         amount,
         LAG(amount, 1) OVER (PARTITION BY customer_id ORDER BY order_date) AS prev1,
         LAG(amount, 2) OVER (PARTITION BY customer_id ORDER BY order_date) AS prev2
  FROM orders
)
SELECT DISTINCT customer_id
FROM w
WHERE prev2 IS NOT NULL
  AND prev2 < prev1
  AND prev1 < amount
ORDER BY customer_id;

For each order row, the CTE computes the one-step-back amount (prev1) and two-steps-back amount (prev2). The outer query filters for any row where the triple (prev2, prev1, amount) is strictly ascending. prev2 IS NOT NULL guards against the first two rows per customer where no complete triple exists. DISTINCT deduplicates customers who have multiple qualifying triples.

4. Strategic explanations

Why double-LAG is the right tool

LAG(amount, 1) and LAG(amount, 2) bring the previous two amounts into the current row as regular column values. At that point the condition prev2 < prev1 AND prev1 < amount is a simple scalar comparison that the engine evaluates in a single pass. No self-join (which produces O(n²) intermediate rows), no recursive CTE (which iterates until convergence), just a sort once per partition and a predicate filter. This is exactly the pattern the question requires and the most efficient approach for detecting fixed-length runs.

The NULL guard and why it must be on prev2

The first row per customer has prev1 = NULL, prev2 = NULL; the second row has prev1 = value, prev2 = NULL. Comparing NULL with < returns NULL, not FALSE, so WHERE prev2 < prev1 would silently exclude those rows anyway. However, explicitly writing prev2 IS NOT NULL makes the intent clear and avoids relying on the three-valued logic of NULL comparisons. Checking only prev1 IS NOT NULL would be insufficient — the third row in the partition (the first with a complete triple) has both prev1 and prev2 non-null; the second row has prev1 non-null but prev2 null, and NULL < prev1 evaluates to NULL (excluded), so in practice just checking prev2 IS NOT NULL is the minimal correct guard.

Strict vs non-strict inequality

The question asks for strictly increasing, so < is correct. Customer 3's sequence 10, 10, 20 fails because the first pair is not strictly less than. If the requirement were "non-decreasing" (allowing plateaus), ≤ would be used instead. Always clarify whether ties break the run.

Extending to runs of length 4 or more

For a run of 4, add LAG(amount, 3) and extend the comparison chain. For a variable-length run of at least k, the double-LAG idiom reaches its limit (you would need k−1 LAG calls); at that point a recursive CTE or a gaps-and-islands approach using SUM(CASE WHEN … THEN 0 ELSE 1 END) OVER (…) to mark run boundaries becomes more maintainable. For k = 3 the two-LAG solution is idiomatic and preferred.

Dialect notes

LAG with an offset argument is standard SQL and supported by PostgreSQL, MySQL 8+, SQL Server 2012+, BigQuery, Snowflake, and Oracle. SQLite 3.25+ supports LAG. The query is fully portable.

Senior signal: reach for double-LAG to detect three-element sequences in a single pass without a self-join or recursive CTE; use prev2 IS NOT NULL explicitly to mark complete triples; use strict < and discuss the non-strict variant; and explain why this pattern is O(n log n) vs the O(n²) self-join alternative.
