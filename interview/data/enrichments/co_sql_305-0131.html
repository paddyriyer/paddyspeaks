The growth team wants to size the upsell ceiling for each customer. The single biggest order a customer has ever placed is a strong proxy for their willingness-to-pay — much better than average order value, which gets diluted by frequent small repeat purchases. The query has to return the full row (order_id + amount + product), not just the max value, so the marketing system can reference the actual SKU.

2. The SQL — correlated subquery

The classic pattern: keep rows whose amount equals the per-customer max. The correlated subquery references the outer row's customer_id, so it's recomputed for each customer.

SELECT customer_id, order_id, amount
FROM orders o
WHERE amount = (
  SELECT MAX(amount)
  FROM orders
  WHERE customer_id = o.customer_id        -- correlated
);

4. Strategic explanations

Correlated subquery vs ROW_NUMBER() — when each is faster

Two valid solutions, different performance profiles:

-- Alternative: ROW_NUMBER (returns exactly one row per customer, deterministic)
SELECT customer_id, order_id, amount
FROM (
  SELECT customer_id, order_id, amount,
    ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY amount DESC, order_id) AS rn
  FROM orders
) t
WHERE rn = 1;

For small tables, the correlated subquery is fine. For large tables it's quadratic — the subquery re-scans orders for each outer row. ROW_NUMBER() sorts once and assigns ranks in a single pass — typically 10–100× faster on millions of rows. Modern optimisers (Postgres 12+, Snowflake, BigQuery) sometimes rewrite the correlated form into a window-function plan automatically; older engines do not.

Tie-breaking — the C_102 problem

Customer C_102's two $220 orders are a real issue: the correlated subquery returns both, the marketing system sends two emails, the customer wonders why. ROW_NUMBER() with a deterministic secondary sort (ORDER BY amount DESC, order_id ASC) returns exactly one. RANK() returns both and tells you they're tied. Pick consciously — the right answer depends on whether "biggest order" should be unique per customer or whether genuine ties should both qualify.

Why MAX(amount) alone loses the row

The naïve attempt is SELECT customer_id, MAX(amount) FROM orders GROUP BY customer_id. That gives you the value, but you've lost order_id — and the marketing system needs the SKU on that order to personalise the upsell email. The correlated subquery (or ROW_NUMBER) returns the entire matching row, not just the aggregate.

"Biggest order" vs "first order" — different signals

Biggest order = upsell ceiling ("how much will they pay?"). First order = acquisition channel attribution ("how did they find us?"). The same query shape — change MAX(amount) to MIN(order_date) — answers a completely different growth question. Senior candidates volunteer this as the natural follow-up.

The follow-up the interviewer always asks: "second-biggest order"

Once you've got the biggest, they ask for the second-biggest (or top-3). Correlated subquery gets ugly fast; ROW_NUMBER handles it trivially:

WHERE rn <= 3   -- top-3 biggest orders per customer
Senior signal: show both solutions (correlated subquery + ROW_NUMBER), name the performance trade-off (quadratic vs single-pass), and call out the tie-handling question without being asked. The "first order vs biggest order" framing as different growth signals is the over-the-bar answer.
