Given a stream of order-status updates with timestamps, produce one row per actual status period with start and end dates. The catch the question doesn't tell you about: the source table contains "status pings" — repeated rows of the same status that come from periodic system updates, retries, and re-confirmations. They must be collapsed before any periods are computed. Two SQL ideas working together: LAG to detect change, then LEAD to compute the end of each period.

1. The source data — orders as a status log

One row per status update. The same order_id appears many times across its lifecycle.

order_idorder_datestatusnarrative
1Jan 1Pendingreal start
1Jan 2Pendingsystem re-ping — same status, no real change
1Jan 3Shippedreal change
1Jan 5Shippedanother ping
1Jan 7Deliveredreal change

Order 1 has 5 rows but only 3 real status periods: Pending, Shipped, Delivered. The naïve query produces 5 period rows; the correct query produces 3.

2. The two-CTE solution

WITH status_changes AS (
  SELECT order_id, order_date, status,
    LAG(status) OVER (PARTITION BY order_id ORDER BY order_date) AS prev_status
  FROM orders
),
filtered AS (
  SELECT order_id, order_date, status
  FROM status_changes
  WHERE status != prev_status OR prev_status IS NULL
)
SELECT order_id, status,
  order_date AS start_date,
  LEAD(order_date) OVER (PARTITION BY order_id ORDER BY order_date) AS end_date
FROM filtered
ORDER BY order_id, start_date;

3. The expected output

order_idstatusstart_dateend_date
1PendingJan 1Jan 3
1ShippedJan 3Jan 7
1DeliveredJan 7NULL

The last status of any order has end_date = NULL because LEAD has nothing to look forward to. That's the right semantic: the period is still open.

4. Strategic explanations

Why dedup BEFORE computing end_date — not after

If you applied LEAD(order_date) directly to the raw orders table, the Pending(Jan 1) period would end at Jan 2 (the next ping, not the next real change). Then you'd have to collapse consecutive periods with the same status downstream — much harder than dedup first. The trick is to do all the change-detection in one CTE, filter, then run period-computation on the clean stream.

The OR prev_status IS NULL safety net — easy to forget

The very first row of each order has prev_status IS NULL (nothing to look back at). The natural-looking predicate WHERE status != prev_status evaluates to:

'Pending' != NULL  →  NULL  (NOT TRUE!)

SQL's three-valued logic means the first row gets silently filtered out. The order's actual start vanishes; the first period appears to begin at the second ping. The OR prev_status IS NULL rescues the start row.

The cross-dialect NULL-safe alternative

Some teams prefer an explicit NULL-safe operator instead of the OR clause:

DialectNULL-safe inequality
SQLite / Postgresstatus IS DISTINCT FROM prev_status
MySQLNOT (status <=> prev_status)
BigQuerystatus IS DISTINCT FROM prev_status
SQL ServerNo native — use COALESCE(status, '') <> COALESCE(prev_status, '')

The IS DISTINCT FROM form treats NULLs as ordinary values — 'Pending' IS DISTINCT FROM NULL evaluates to TRUE. One-line replacement for the != + OR IS NULL idiom.

LAG vs LEAD — a deliberate pairing
DirectionWhat it answersNULL appears at
LAG (look backward)"What was the previous value?" — used to detect a changeFIRST row of each partition
LEAD (look forward)"When does this end?" — used to find the boundaryLAST row of each partition

Status-period detection needs both: LAG for "did this row start a new period?", LEAD for "when does this period end?". Each NULL is correct on its respective boundary — first row legitimately had no predecessor; last row legitimately has no successor.

The "first vs last NULL" specification trap

Many problem statements say "the first status's end_date should be NULL". This is almost always a misread of the real semantic — the LEAD-based approach gives the last status NULL end_date, which is the right interpretation (the period is open-ended; the order is still in that state). If a problem statement insists the FIRST status has NULL end_date, you'd be inverting the model — periods would be defined by where they ENDED, not where they started. Read carefully and clarify.

Gaps-and-islands — recognize the family

This is the "island start" variant of gaps-and-islands. The full family of patterns is worth memorizing:

VariantMechanismUse case
Island start detection (this query)WHERE val != LAG(val)State changes, status periods, price changes, plan-tier transitions
Island id labellingSUM(new_island_flag) OVER (...)Session-id assignment from raw events, run-length encoding
Consecutive-day streakdate - ROW_NUMBER() = constant within runLogin streaks, consecutive-day buys, snapshot-data continuity checks
Tolerant-streak (allow gaps)SUM(case when gap > threshold then 1 else 0 end) over (order)"Continuing streak even with one missed day", session-30min-gap rule

All four patterns share the same shape: "compare adjacent rows, mark where adjacency breaks, group by the marker". Once you've internalized one, the others are 10-line variations.

The "session boundary" cousin

For raw clickstream where you want to assign a session_id based on a 30-minute idle gap, the same shape applies:

WITH gaps AS (
  SELECT user_id, event_at,
    CASE WHEN event_at - LAG(event_at) OVER (PARTITION BY user_id ORDER BY event_at) > INTERVAL '30 min'
            OR LAG(event_at) OVER (PARTITION BY user_id ORDER BY event_at) IS NULL
         THEN 1 ELSE 0 END AS new_session
  FROM clicks
),
sessioned AS (
  SELECT *, SUM(new_session) OVER (PARTITION BY user_id ORDER BY event_at) AS session_id
  FROM gaps
)
SELECT user_id, session_id, MIN(event_at) AS session_start, MAX(event_at) AS session_end, COUNT(*) AS clicks
FROM sessioned
GROUP BY user_id, session_id;

Same LAG-detect-change pattern, plus SUM(...) OVER () to assign IDs. This is the most-used streaming-analytics SQL pattern outside of plain aggregations.

Why the data-quality of orders matters

If the source orders table has duplicate (order_id, order_date) rows — same order, same exact timestamp, two writes — the LAG ordering is undefined; the dedup may keep the wrong one. Production-grade pipelines either:

  • Add a tiebreaker column to the ORDER BY: ORDER BY order_date, ingest_seq
  • Pre-aggregate duplicates: SELECT DISTINCT order_id, order_date, status FROM orders as the first CTE
  • Enforce uniqueness upstream: a producer-side dedup key on (order_id, order_date)
Performance — when the CTE chain matters

For multi-billion-row order logs, the two-CTE approach is fine because both window functions partition by the same key (order_id) and the planner can stream through. The performance trap is when teams add an ORDER BY on a non-partitioned column in either CTE — that triggers a full sort of the dataset. Keep the partition key consistent across CTEs and you stay on the fast path.

The senior reframe

The interviewer isn't testing whether you can write LAG / LEAD. They're testing whether you ask "can the same status appear twice in a row?" when reading the prompt. The pattern recognition that "status history" → gaps-and-islands → LAG-detect-change → dedup → window-on-clean-stream is the senior signal. Junior engineers write the naïve LEAD-only query and ship it; the duplicate-status bug hits production a week later.

Senior signal: recognize "status periods" as a gaps-and-islands variant, dedup BEFORE computing periods (not after), pair LAG-for-detection with LEAD-for-end-boundary, never forget OR prev_status IS NULL (or use IS DISTINCT FROM), and clarify the first-vs-last-NULL specification with the interviewer rather than guessing. The "session boundary via cumulative-flag SUM" cousin is the over-the-bar generalization.
