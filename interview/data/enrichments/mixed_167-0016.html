A scheduling or booking system needs to detect double-bookings: "find every pair of date ranges that overlap." The reference solution is a self-join with the classic interval-overlap predicate, hardened with regex guards because the dates are stored as text. The interviewer is probing two things: do you know the overlap formula, and do you understand why a.id < b.id is essential.

1. The source data — ranges

Range 4 is fully contained inside ranges 1 and 2 — the edge case. A naive "do the endpoints fall between" check often misses containment; the proper formula catches it.

2. The SQL — self-join with the overlap predicate

SELECT a.id, b.id FROM ranges a JOIN ranges b
ON a.id < b.id
AND a.start_dt ~ '^\d{4}-' AND b.start_dt ~ '^\d{4}-'
AND a.end_dt ~ '^\d{4}-' AND b.end_dt ~ '^\d{4}-'
AND a.start_dt::timestamp < b.end_dt::timestamp
AND b.start_dt::timestamp < a.end_dt::timestamp;

3. The resulting analysis

a.idb.idnote
12Jan 1–10 vs Jan 5–15 — partial overlap
14Jan 8–9 sits inside Jan 1–10
24Jan 8–9 sits inside Jan 5–15

Three overlapping pairs. Range 3 (late Jan) and range 5 (February) touch nothing. Crucially, each pair appears once — (1,2) with no mirror (2,1) — and no range is paired with itself.

4. Strategic explanations

The overlap formula — memorise it

Two intervals overlap if and only if A.start < B.end AND B.start < A.end. It is far easier to reason about the negation: two ranges do not overlap when one ends before the other starts (A.end ≤ B.start OR B.end ≤ A.start). Negate that and you get the formula above. This single predicate handles partial overlaps, full containment, and identical ranges — no special cases. Enumerating "endpoint-in-range" conditions instead is where candidates miss the containment case (range 4 here).

Why a.id < b.id does double duty

Without it, a self-join produces every ordered pair: each range matches itself (a range trivially overlaps itself), and every real pair appears twice as (x,y) and (y,x). The strict inequality a.id < b.id kills both problems at once — it removes self-pairs and keeps exactly one orientation of each pair. Using <> would only remove self-pairs and still leave mirrored duplicates.

The text-date trap — what the regex guards are for

start_dt and end_dt are text, not date. Casting 'not-a-date'::timestamp throws and aborts the whole query. The ~ '^\d{4}-' regex checks each value starts with four digits and a hyphen before any cast is attempted, so malformed rows are skipped instead of crashing. The real senior fix is upstream: store dates in a date/timestamp column so the database enforces validity and comparisons are correct and indexable. ISO-8601 text happens to sort correctly, but that is luck, not a guarantee — '2026-1-5' would sort wrong.

Inclusive vs exclusive endpoints — do bookings touch?

Strict < means a range ending exactly when another begins (Jan 10 end, Jan 10 start) is not an overlap. For room bookings that is usually right — checkout and check-in can share a day. For sensor coverage you may want ≤ so touching counts. Always confirm endpoint semantics with the interviewer.

Performance — the self-join is O(n²)

A range-on-range join with inequality predicates cannot use a plain equality hash join; it tends toward a nested loop, roughly n² comparisons. For large tables, mention Postgres tstzrange with a GiST index and the && overlap operator — that turns interval overlap into an indexed lookup. Mentioning range types unprompted is the over-the-bar finish.

Senior signal: recall the overlap predicate as A.start < B.end AND B.start < A.end and derive it from the no-overlap negation; explain that a.id < b.id simultaneously removes self-pairs and mirrored duplicates; and call out that storing dates as text is the real bug — the regex guard is a workaround, range types with a GiST index are the fix.
