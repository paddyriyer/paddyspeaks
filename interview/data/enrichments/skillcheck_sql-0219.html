This is the capstone: pivot, then unpivot, in a single query. It chains every move from the earlier exercises — flatten the tree, pivot with conditional aggregation, melt the result with CROSS JOIN LATERAL — and it teaches the one thing a round trip reveals that neither half does alone: the transform is lossy.

1. The source data

The familiar categories tree and titles table. Leaf-genre totals: Superhero 180, Spy 60, Comedy 100 under Film; Drama Series 160 and Reality 25 under Series. Keep an eye on Reality.

2. The query — three named stages

WITH category_hierarchy AS (
  SELECT
    c1.category_id   AS root_id,
    c1.category_name AS root_name,
    COALESCE(c3.category_id,   c2.category_id,   c1.category_id)   AS descendant_id,
    COALESCE(c3.category_name, c2.category_name, c1.category_name) AS descendant_name
  FROM categories c1
  LEFT JOIN categories c2 ON c2.parent_id = c1.category_id
  LEFT JOIN categories c3 ON c3.parent_id = c2.category_id
  WHERE c1.parent_id IS NULL
),
wide AS (
  SELECT
    h.root_name AS main_category,
    COALESCE(SUM(CASE WHEN h.descendant_name = 'Superhero'    THEN t.watch_hours END), 0) AS superhero_hours,
    COALESCE(SUM(CASE WHEN h.descendant_name = 'Spy'          THEN t.watch_hours END), 0) AS spy_hours,
    COALESCE(SUM(CASE WHEN h.descendant_name = 'Comedy'       THEN t.watch_hours END), 0) AS comedy_hours,
    COALESCE(SUM(CASE WHEN h.descendant_name = 'Drama Series' THEN t.watch_hours END), 0) AS drama_series_hours
  FROM category_hierarchy h
  LEFT JOIN titles t ON t.category_id = h.descendant_id
  GROUP BY h.root_id, h.root_name
)
SELECT
  w.main_category,
  u.sub_category,
  u.watch_hours
FROM wide w
CROSS JOIN LATERAL (
  VALUES
    ('Superhero',    w.superhero_hours),
    ('Spy',          w.spy_hours),
    ('Comedy',       w.comedy_hours),
    ('Drama Series', w.drama_series_hours)
) AS u(sub_category, watch_hours)
WHERE u.watch_hours > 0
ORDER BY w.main_category, u.watch_hours DESC;

category_hierarchy flattens the tree; wide pivots watch hours into genre columns; the final SELECT unpivots wide straight back into rows.

4. Strategic explanations

Pivot and unpivot are inverses — but only over the named columns

unpivot(pivot(x)) does not return x. It returns x restricted to the columns the pivot chose to name. The wide CTE breaks out exactly four genres; the LATERAL step can only melt what those four columns hold. Anything else never makes the round trip.

Where Reality goes

Reality has 25 watch hours, but wide has no reality_hours column — and unlike the share-of-total report, this CTE does not even keep a total. So Reality's hours are aggregated into nothing: they are dropped at the pivot step and cannot reappear at the unpivot step. The output's Series row accounts for 160 hours; the real Series total is 185. The 25 are silently gone.

The round trip is an approximation of the original long form

titles is already a long-form table — (category_id, watch_hours), one row per title. The pivot-then-unpivot reconstructs a coarser version of it: aggregated to root level, and limited to four named genres. It is a reshape, not a faithful copy.

Why you would still do this

The pattern is legitimate when data arrives wide — a spreadsheet export, a vendor feed with one column per period — and a downstream tool wants it long, or vice versa. The query is fine. The senior move is to name the lossiness: state plainly that any category outside the hard-coded column list will not survive.

The zero-cell filter

WHERE u.watch_hours > 0 drops Film's empty Drama Series cell and Series' three empty genres — the same dense-ledger choice made in the standalone unpivot. Remove it and the result is a complete 2×4 matrix with explicit zeros.

Senior signal: pivot and unpivot are inverses only across the columns the pivot explicitly names. Any value outside that hard-coded list survives the pivot folded inside an aggregate and is gone after the unpivot — a round trip is faithful only when the pivot's column list is exhaustive. Always say so out loud.
