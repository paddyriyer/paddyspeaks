A title belongs to exactly one category, but a category’s real total should include every nested subcategory beneath it. Because the category tree can be any depth, rolling counts up it is a recursive job — a WITH RECURSIVE CTE that pairs each category with its whole subtree.

1. The source data

categories — parent_id points at the enclosing category (NULL = a top-level category):

titles has 11 rows, each tagged to one category_id with its watch_hours. Titles sit at every level — some directly under Film, some down in Superhero.

2. The recursive solution

WITH RECURSIVE subtree AS (
  SELECT category_id AS ancestor_id, category_id AS descendant_id
  FROM categories
  UNION ALL
  SELECT s.ancestor_id, c.category_id
  FROM subtree s
  JOIN categories c ON c.parent_id = s.descendant_id
)
SELECT cat.category_name,
       COUNT(t.title_id) AS total_titles,
       COALESCE(SUM(t.watch_hours), 0) AS total_watch_hours
FROM subtree s
JOIN categories cat ON cat.category_id = s.ancestor_id
LEFT JOIN titles t ON t.category_id = s.descendant_id
GROUP BY cat.category_id, cat.category_name
ORDER BY total_watch_hours DESC, cat.category_name ASC;

The subtree CTE pairs every category (ancestor_id) with every category in its subtree (descendant_id), itself included. Titles join on the descendant; the aggregate groups by the ancestor.

4. Strategic explanations

The anchor pairs each category with itself

The base case is SELECT category_id AS ancestor_id, category_id AS descendant_id — every category paired with itself. That self-pair is deliberate: a category’s rollup must include the titles tagged directly to it, not only those in its children. Seeding (c, c) makes subtree mean “this category and everything under it.” Contrast a referral downline, which excludes the referrer and so seeds its anchor with direct referrals only.

The recursive term descends one level at a time

JOIN categories c ON c.parent_id = s.descendant_id takes each known (ancestor, descendant) pair, finds the descendant’s direct children, and pairs them with the same ancestor. Film reaches Action, then on the next pass Superhero and Spy — the ancestor stays Film the whole way down.

Join the facts on the descendant, group by the ancestor

This swap is the rollup. titles joins on descendant_id — a title is physically in that category — but the query groups by ancestor_id. So a title in Superhero is counted once for Superhero, once for Action, and once for Film: each ancestor up the chain. That is precisely what “include nested subcategories” means.

LEFT JOIN so empty branches still report

A category whose entire subtree holds no titles should still appear with a total of 0, not vanish. LEFT JOIN titles plus COALESCE(SUM(...), 0) guarantees that. And note the ranking: Comedy has more titles than Superhero (3 vs 2) but far fewer watch hours (100 vs 180) — counting rows and summing a metric tell different stories.

Senior signal: to roll a metric up a tree, build an (ancestor, descendant) closure with a recursive CTE whose anchor pairs each node with itself, join the fact table on the descendant, and group by the ancestor. The self-pair in the anchor is what makes a node’s own rows count toward its rollup.
