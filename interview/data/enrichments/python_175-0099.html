Meta's data and growth teams constantly turn raw event logs into a multi-tab spreadsheet for non-technical stakeholders — PMs and marketers who live in Excel, not SQL. This question tests the everyday "analyst engineer" pipeline: pull from a DB, reshape with pandas, and emit a polished .xlsx with several sheets. It probes whether you know groupby / unstack / nlargest and — the senior detail — the ExcelWriter context manager that writes multiple sheets to one file.

1. The input data

An events table in events.db. Three columns; raw rows:

2. The approach

pd.read_sql runs the query over the live sqlite3 connection and materialises a DataFrame — SQL does the I/O, pandas does the shaping. pd.to_datetime parses the text timestamps, then df['ts'].dt.date extracts the calendar date so rows can be bucketed per day.

The with pd.ExcelWriter(...) as xl block is the crux: a single workbook, three .to_excel(xl, sheet_name) calls, each landing on its own tab.

  • daily_events: groupby(['date','event_type']).size() counts per (day, type), then .unstack(fill_value=0) pivots event_type into columns — a tidy date×type matrix, zeros where a type didn't occur.
  • top_users: groupby('user_id').size().nlargest(50) — total events per user, top 50 by count.
  • event_totals: groupby('event_type').size() — one grand total per event type.

3. The output

Sheet daily_events (date index, event types as columns):

dateclicklogin
2026-05-1012
2026-05-1121

Sheet top_users: u1 = 3, u2 = 2, u3 = 1. Sheet event_totals: click = 3, login = 3.

4. Strategic explanations

Complexity

Time O(n log n) dominated by the groupby sorts (hash-grouping is near-linear; nlargest on n rows for k items is O(n log k)). Space O(n) — the whole table is loaded into memory, which is the scaling weakness.

The idiomatic alternative

groupby(...).size() is the clean count idiom, but for the daily matrix pivot_table says the intent in one call:

df.pivot_table(index='date', columns='event_type',
               values='user_id', aggfunc='count', fill_value=0)

Also prefer size() over count() for pure row counts: count() is per-column and skips NaNs, size() counts every row including nulls — almost always what "how many events" means.

The edge case

If a date has zero rows for some event type, unstack(fill_value=0) fills 0 rather than leaving NaN — correct. But if the entire table is empty, read_sql returns an empty frame and .dt accessor on a non-datetime empty column can raise; guard with an early "if df.empty" return. Also: .dt.date yields Python date objects, losing timezone — fine for daily buckets, lossy if you later need hourly.

The follow-up

"The events table is 500M rows — this OOMs." Push the aggregation into SQL (GROUP BY date, event_type server-side, return only the small rollup), or stream with pd.read_sql(..., chunksize=100_000) and accumulate. The senior moves the groupby to where the data lives.

The trap

Calling .to_excel('report.xlsx') three times with separate paths — each call overwrites the file, leaving only the last sheet. Multiple tabs require one ExcelWriter shared across the writes. Forgetting the with block (or not calling .save()/.close()) produces a truncated, unopenable file.

Senior signal: uses a single ExcelWriter context manager for the multi-sheet workbook, picks size() over count() deliberately, knows unstack(fill_value=0) avoids ragged NaN holes, and — when the data won't fit in RAM — pushes the aggregation into SQL or switches to chunked reads instead of scaling up the box.
