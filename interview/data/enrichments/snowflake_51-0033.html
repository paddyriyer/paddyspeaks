Elastic’s aerospace-equipment catalogue needs a report that combines three tables: a piece of equipment belongs to exactly one category (inner join), and may have zero or more free-form tags (left join). The result must collapse each equipment’s tags into a single comma-separated string. The pattern tests multi-table join composition, the difference between JOIN and LEFT JOIN when multiplicity differs, and string aggregation — which is spelled differently across SQL dialects.

E01 has three tags, E02 has one, E03 has a tag row with an empty string, E04 has no tag rows at all.

2. The SQL — inner join + left join + STRING_AGG

SELECT e.equipment_id, e.name, c.label AS category,
       STRING_AGG(t.tag, ', ' ORDER BY t.tag) AS tags
FROM equipment   e
JOIN categories  c ON e.category_id = c.category_id
LEFT JOIN tags   t ON e.equipment_id = t.equipment_id
GROUP BY e.equipment_id, e.name, c.label;

4. Strategic explanations

Why inner JOIN for categories but LEFT JOIN for tags

A piece of equipment that has no category is a data-integrity violation — it should not appear in the report. The JOIN categories c acts as an implicit integrity check: equipment without a valid category is excluded. Tags, by contrast, are optional metadata. Using LEFT JOIN tags t keeps all equipment in the result even if it has no tags, correctly returning NULL in the tags column rather than dropping the row.

GROUP BY must include all non-aggregated SELECT columns

The GROUP BY clause lists e.equipment_id, e.name, c.label — every column that appears in SELECT outside of an aggregate function. If you omit any of these, most engines raise an error. Snowflake is strict here; MySQL with ONLY_FULL_GROUP_BY disabled is notoriously permissive (silently picks an arbitrary value), which is a known source of query bugs.

STRING_AGG with ORDER BY — the Postgres/Snowflake form

STRING_AGG(expr, delimiter ORDER BY sort_col) is the ANSI SQL 2016 form and is supported natively in Snowflake and Postgres. The dialect equivalent in Snowflake is also LISTAGG(t.tag, ', ') WITHIN GROUP (ORDER BY t.tag) — the form the dialect_token field hints at. These are functionally identical; Snowflake accepts both. MySQL uses GROUP_CONCAT(t.tag ORDER BY t.tag SEPARATOR ', '). BigQuery uses STRING_AGG(t.tag, ', ' ORDER BY t.tag) (same as Postgres). SQL Server uses STRING_AGG(t.tag, ', ') WITHIN GROUP (ORDER BY t.tag) from v2017+.

NULL vs empty-string in STRING_AGG

STRING_AGG skips NULL values — it aggregates only non-NULL inputs. An equipment with no tag rows has t.tag = NULL from the LEFT JOIN; STRING_AGG of all-NULLs returns NULL. An equipment with a tag row containing an empty string has a non-NULL input; STRING_AGG includes it. This distinction matters when tag rows exist but are blank — you may want to add a WHERE t.tag IS NOT NULL AND t.tag <> '' filter inside the aggregation or in the JOIN to clean upstream noise.

Performance on large tag tables

An index on tags(equipment_id, tag) serves the LEFT JOIN and provides the ORDER BY sort without a separate sort step. The GROUP BY requires a hash or sort aggregate; with the index already sorted by equipment_id, Postgres and Snowflake can use a streaming aggregate without materializing the full hash table.
