A growth team wants a "users seen so far" curve: for every date in the log, how many distinct users have been observed up to and including that day. A user who first appeared on Monday must not be re-counted on Tuesday. The interviewer is probing whether you can turn a running COUNT(DISTINCT ...) — which window functions cannot do directly — into a running SUM of first-appearances.

1. The source data — event

Two edge cases drive the answer. Users 1 and 2 appear on multiple days — only their first day should add to the count. And 2025-01-04 has no events at all; the curve must skip it (the question asks only for dates that appear).

2. The SQL — running SUM of first-appearance counts

WITH first_seen AS (
  SELECT user_id, MIN(event_date) AS first_date
  FROM event
  GROUP BY user_id
),
new_per_day AS (
  SELECT first_date AS event_date, COUNT(*) AS new_users
  FROM first_seen
  GROUP BY first_date
),
all_days AS (
  SELECT DISTINCT event_date FROM event
)
SELECT d.event_date,
       SUM(COALESCE(n.new_users, 0)) OVER (ORDER BY d.event_date
         ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS cumulative_distinct_users
FROM all_days d
LEFT JOIN new_per_day n ON n.event_date = d.event_date
ORDER BY d.event_date;

first_seen collapses each user to their earliest date. new_per_day counts how many users debut on each date. A running SUM over those daily debuts gives the cumulative distinct count — because every user is counted exactly once, on the day they first appear.

The curve is monotonic and never double-counts: Jan-02 is 4, not 5, because user 1's return contributes nothing. Jan-04 has no row at all — all_days only carries dates that appear in the data.

4. Strategic explanations

Why a running COUNT(DISTINCT) does not exist

The obvious idea — COUNT(DISTINCT user_id) OVER (ORDER BY event_date ...) — is illegal in standard SQL and PostgreSQL: DISTINCT is not permitted inside a windowed aggregate. The fix is to make each user contribute exactly once before the window: reduce every user to their first date, then a plain running SUM of those first-appearances is the cumulative distinct count. Distinctness is enforced by the MIN(event_date) GROUP BY user_id step, not by the window.

The frame clause is doing real work

ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW defines the running total. Critically, ROWS (physical rows) is the right choice over the default RANGE. With ORDER BY event_date and a RANGE frame, all rows sharing a date are treated as peers and folded together — harmless here because all_days has one row per date, but a latent bug if duplicates ever creep in. ROWS is unambiguous: it sums this row and every earlier one.

The LEFT JOIN + COALESCE guard

all_days is the date spine; new_per_day only has rows for dates where someone debuted. If every user on a given date is a returning user, that date has no new_per_day match and the LEFT JOIN yields NULL. SUM would skip the NULL and the running total would still be correct — but COALESCE(n.new_users, 0) makes the intent explicit and protects you if the frame logic changes. Here every appearing date happens to have a debut, but the guard is correct defensive style.

Gaps: dates with no events

2025-01-04 simply does not exist in event, so SELECT DISTINCT event_date FROM event never produces it — matching the question's "for each date that appears". If the requirement were a gap-free daily curve, you would replace all_days with a generated calendar (generate_series) and the running SUM would carry the previous total forward across empty days. Know which version the stakeholder wants.

The naive self-join alternative and why it is worse

You could write SELECT d.event_date, COUNT(DISTINCT e.user_id) FROM all_days d JOIN event e ON e.event_date <= d.event_date GROUP BY d.event_date. It is correct but O(n²): every date re-scans the whole history. The first-seen + running-SUM approach touches each user once and sorts once — O(n log n). On a real event table that difference is decisive.

Senior signal: know that COUNT(DISTINCT) is forbidden inside a window, so collapse each user to their first date and run a plain cumulative SUM of debuts; choose a ROWS frame over RANGE deliberately; and distinguish "dates that appear" from a gap-free calendar spine.
