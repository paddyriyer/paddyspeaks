Private-equity back-office teams ask this daily: "What was the total marked-to-market value of each portfolio on each reporting date?" The answer requires joining position snapshots (holdings) to daily price ticks (prices) on a composite key — both asset_id and date must match. The interviewer is probing whether you understand the point-in-time join pattern, the NULLIF(regexp_replace(...)::numeric) defensive cast for dirty numeric columns, and the aggregation that sums per-position values up to a portfolio total.

2. The SQL — composite-key JOIN to compute portfolio value

SELECT h.portfolio_id, h.as_of_date AS day,
       SUM(NULLIF(regexp_replace(h.shares::text, '[^-.0-9]', '', 'g'), '')::numeric
           * p.close_price) AS portfolio_value
FROM holdings h
JOIN prices p
  ON h.asset_id   = p.asset_id
 AND h.as_of_date = p.price_date
GROUP BY h.portfolio_id, h.as_of_date
ORDER BY h.portfolio_id, day;

4. Strategic explanations

The composite-key JOIN — both asset and date must match

Joining on asset_id alone would pair every price row for AAPL (Jan-31, Feb-28, …) with every holding of AAPL regardless of date, producing a Cartesian explosion and wrong values. Adding AND h.as_of_date = p.price_date makes this a point-in-time join: each holding is valued at exactly the price on the same date it was recorded. This is the canonical pattern for mark-to-market valuation and for slowly-changing-dimension lookups.

Silent data loss — INNER JOIN drops unpriced positions

When P1 holds MSFT on 2025-02-28 but prices has no MSFT row for that date, the INNER JOIN silently omits that holding from the portfolio value calculation. For financial reporting this is usually incorrect — the portfolio value becomes understated. Production systems handle this with either a LEFT JOIN prices p ON ... COALESCE(p.close_price, prior_close) using a fill-forward price, or by guaranteeing a complete prices table before running valuation. At the interview, flag this data-quality risk immediately.

Defensive numeric cast — NULLIF(regexp_replace(...)::numeric)

The shares column may arrive as text with non-numeric characters (unit labels, commas as thousands separators). The regex strips everything except digits, decimal point, and minus sign; NULLIF turns the resulting empty string into NULL; the final ::numeric cast parses the clean string. SUM ignores NULL, so a corrupted shares value contributes zero rather than crashing the query. In Snowflake, prefer TRY_TO_NUMBER(h.shares, 10, 4) which returns NULL on parse failure without any regex.

Aggregation level — SUM collapses per-position rows to portfolio totals

Without the GROUP BY h.portfolio_id, h.as_of_date, the query would return one row per (portfolio, asset, date) triple. The SUM(...) + group-by is what converts individual position market values into a single portfolio-level number. If the downstream consumer needs both the position detail and the portfolio total, use a window function: SUM(shares * close_price) OVER (PARTITION BY portfolio_id, as_of_date) alongside the per-row detail, avoiding a second query.

Dialect portability
  • Snowflake: TRY_TO_NUMBER(h.shares) replaces the regex cast. Snowflake date equality in joins works identically.
  • BigQuery: SAFE_CAST(h.shares AS FLOAT64). Date columns are DATE type; equality join on date works natively.
  • MySQL: The regexp_replace with 'g' flag is not supported; use a different sanitisation approach or ensure column types.
Senior signal: identify this as a point-in-time composite-key JOIN pattern, immediately flag that INNER JOIN silently drops unpriced positions (understated portfolio value), explain the NULLIF(regexp_replace(...)::numeric) sanitisation chain and its Snowflake equivalent (TRY_TO_NUMBER), and offer the window-function variant that delivers per-position detail alongside the portfolio total in a single pass.
