An HR analytics report wants each employee's longest run of consecutive workdays — calendar-adjacent dates with no gap. This is the classic gaps-and-islands problem, and the interviewer is checking whether you know the elegant trick: subtract a row number from the date, and every consecutive run collapses to a constant key you can group on.

The edge cases that decide the interview: emp 1 has two separate runs and we want the longer (3, not 2); emp 2's longest run is not its first appearance; and emp 3 has a single day — a streak of length 1, which the query must still report.

2. The SQL — the date-minus-rownumber islands trick

WITH dt AS (
  SELECT DISTINCT emp_id, work_date FROM attendance
),
grouped AS (
  SELECT emp_id, work_date,
         work_date - (ROW_NUMBER() OVER (PARTITION BY emp_id ORDER BY work_date))::int AS streak_key
  FROM dt
),
streaks AS (
  SELECT emp_id, streak_key, COUNT(*) AS streak_len
  FROM grouped
  GROUP BY emp_id, streak_key
)
SELECT emp_id, MAX(streak_len) AS longest_streak
FROM streaks
GROUP BY emp_id
ORDER BY emp_id;

ROW_NUMBER() numbers each employee's dates 1, 2, 3, ... in date order. Subtracting that counter from the date yields a value that stays constant across a consecutive run and jumps at every gap — that constant is the island key. Grouping by it gives each run's length; the outer MAX picks the longest per employee.

Emp 1 returns 3 — the MAX over its two islands. Emp 3 returns 1: a lone day is a degenerate run of length 1, and the query handles it without special-casing.

4. Strategic explanations

Why subtracting the row number works

Within a consecutive run, the date increases by exactly 1 each step — and so does ROW_NUMBER(). Two quantities that march in lockstep have a constant difference. So work_date - row_number is identical for every date in a run. The instant there is a gap, the date jumps by 2+ while the row number still increments by 1, so the difference changes — a new island begins. The difference value itself is meaningless; only its constancy matters, which is why it makes a perfect GROUP BY key.

Why ::int on the row number

work_date is a DATE. In PostgreSQL date - integer yields a date shifted by that many days — exactly what we want. But ROW_NUMBER() returns bigint, and date - bigint is not a defined operator; the explicit ::int cast makes the subtraction type-check. (The resulting streak_key is itself a date — a perfectly valid grouping value.) Engines vary here: in some you would instead convert the date to an integer day-count. The cast is small but its absence is a hard error.

The SELECT DISTINCT guard against duplicate dates

The first CTE deduplicates (emp_id, work_date). If an employee had the same work_date twice (a double clock-in), ROW_NUMBER() would assign them different counters while the date is identical — breaking the lockstep and splitting or corrupting the island key. Deduplicating first guarantees one row per employee-day, which is the precondition the trick relies on. The demo data has no duplicates, but a robust solution defends against them anyway.

Single-day runs and ties

A lone date forms an island of size 1 — the grouping produces a one-row group, COUNT(*) is 1, and emp 3 is correctly reported. Ties are handled for free: if an employee had two separate runs both of length 4, there would be two streak_len = 4 rows and MAX simply returns 4. The question only asks for the longest length, so no tiebreak is needed; if it asked which run, you would add the run's start date via MIN(work_date) per island.

Dialect notes

The date-minus-rownumber idiom is portable wherever ROW_NUMBER() exists — PostgreSQL, SQL Server, Oracle, MySQL 8+, BigQuery. The only thing that changes is the date arithmetic: SQL Server uses DATEADD(DAY, -ROW_NUMBER()..., work_date); MySQL uses DATE_SUB; BigQuery uses DATE_SUB(work_date, INTERVAL ROW_NUMBER()... DAY). An alternative formulation uses LAG() to flag gaps then a running SUM of the gap-flags as the island id — equivalent, and sometimes preferred when "adjacent" is defined by something other than a fixed +1 day step.

Senior signal: solve consecutive-run problems with the gaps-and-islands trick — subtract ROW_NUMBER() from the date so each run shares a constant key — deduplicate dates first so the row-number stays in lockstep, cast the row number to int for the date subtraction, and know the LAG+running-sum variant for non-+1 adjacency rules.
