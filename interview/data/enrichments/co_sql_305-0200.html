Instacart wants to study reorder behaviour: for each order, what did the same customer order next? The name "Corrected Order Pairing" hints at a story — an earlier attempt paired orders incorrectly, probably by accidentally matching an order with itself or with earlier orders. The reference fixes that with a self-join and a strict >. The interviewer is watching whether you see what this join actually produces, and whether it is the pairing the business asked for.

Customer 101 has three orders, customer 102 has two, and customer 103 has just one. That single-order customer is the edge case — order 6 has no "next" order at all, so it should not appear in any pair.

2. The SQL — self-join on later timestamp

SELECT a.order_id, b.order_id AS next_order
FROM orders a JOIN orders b ON a.customer_id = b.customer_id AND b.ordered_at > a.ordered_at;

Each order a joins to every order b by the same customer with a strictly later timestamp. The strict > is the "correction": it stops an order pairing with itself and stops earlier orders being treated as "next".

Order 6 is correctly absent — customer 103 has no later order, so the join finds no partner. But look at order 1: it pairs with both 2 and 3. This join produces every later order, not the immediately next one. If "next order" means the one right after, this answer over-produces; if it means "any subsequent order", it is exactly right. That ambiguity is the interview.

4. Strategic explanations

"Next order" almost never means "all later orders"

The reference returns the full set of later pairs — useful for "all reorder pairs", but a literal "next order" wants only the immediately following one. Order 1's next order is 2, not also 3. The clean fix is LEAD, which steps exactly one row forward within each customer:

SELECT order_id,
       LEAD(order_id) OVER (PARTITION BY customer_id ORDER BY ordered_at) AS next_order
FROM orders;

This gives 1→2, 2→3, 4→5, and — importantly — 3→NULL, 5→NULL, 6→NULL: the last order of each customer is kept with a NULL next, rather than dropped. Confirm with the interviewer which behaviour the analysis needs before committing.

The inner-join trap — vanishing last orders

The self-join uses an inner join, so any order with no later partner disappears entirely — orders 3, 5, and 6 are gone from the result. For a reorder-rate metric that is a silent bug: you would be dividing by only the orders that had a follow-up, inflating the rate. If every order must appear (with NULL when there is no next), either use the LEAD form above or a LEFT JOIN LATERAL:

SELECT a.order_id, n.order_id AS next_order
FROM orders a
LEFT JOIN LATERAL (
  SELECT order_id FROM orders b
  WHERE b.customer_id = a.customer_id AND b.ordered_at > a.ordered_at
  ORDER BY b.ordered_at LIMIT 1
) n ON true;
The duplicate-timestamp trap

If a customer places two orders at the exact same ordered_at, the strict > excludes both from being each other's "next" — they tie, and neither is strictly later. LEAD with ORDER BY ordered_at alone would also order them arbitrarily. Add a deterministic tiebreaker, e.g. ORDER BY ordered_at, order_id, so the sequence is stable and reproducible.

The performance angle

The self-join is O(n²) per customer in the worst case: every order pairs with every later one, so a customer with 100 orders generates ~5,000 pair rows. The LEAD version sorts once per partition — O(n log n) — and emits exactly one row per order. On a real orders table the window function is not just semantically tighter, it is the only one that scales.

Edge case: the single-order customer

Customer 103 demonstrates the boundary cleanly. With the self-join, order 6 is dropped (no later partner) — fine if you only want pairs, wrong if you want a row per order. With LEAD, order 6 survives as 6 → NULL, which is the more honest representation: "this customer has not reordered yet."

Senior signal: recognise the self-join produces all later pairs rather than the immediate next, reach for LEAD() OVER (PARTITION BY customer ORDER BY time) when "next" means strictly the following order, and flag that the inner join silently drops every customer's last order — a hidden bias in any reorder-rate metric.
