An operations analyst wants a single product report combining units sold from transactions, current stock from the warehouse, and a reorder flag — all in one query across three tables. The interviewer is testing three-table LEFT JOIN aggregation, COALESCE for products with no sales, and the CASE WHEN comparison that drives the reorder decision. The query's use of NULLIF(regexp_replace(…)) is a defensive pattern for dirty numeric data; understanding it is itself a signal.

Product 4 (Cable) has no sales rows — its units_sold should be 0. Product 1 (Widget) has on_hand=12 and reorder_point=20: stock is at or below the threshold, so it should show REORDER. Product 3 (Book) has on_hand = reorder_point = 5, which also triggers REORDER (the CASE uses ≤).

2. The SQL — three-table LEFT JOIN with CASE WHEN reorder logic

SELECT p.product_id, p.name,
       COALESCE(SUM(s.qty), 0)  AS units_sold,
       MAX(i.on_hand)            AS current_stock,
       MAX(i.reorder_point)      AS reorder_point,
       CASE
         WHEN MAX(i.on_hand) <= MAX(i.reorder_point) THEN 'REORDER'
         ELSE 'OK'
       END                       AS stock_status
FROM products  p
LEFT JOIN sales     s ON p.product_id = s.product_id
LEFT JOIN inventory i ON p.product_id = i.product_id
GROUP BY p.product_id, p.name;

Note on the production query: the original solution wraps inventory columns in MAX(NULLIF(regexp_replace(i.on_hand::text, '[^-.0-9]', '', 'g'), '')::numeric). This is a defensive pattern for when the inventory table stores on_hand as TEXT with potential non-numeric characters (e.g., "12 units" or empty strings). The simplified version above assumes clean INTEGER columns — which is the appropriate schema. The defensive regex is worth explaining as a real-world data-quality pattern.

4. Strategic explanations

Why both joins are LEFT JOIN

A product with no sales rows (Cable) would be dropped by an INNER JOIN to sales. Similarly, a product not yet in the inventory table would be dropped by an INNER JOIN to inventory. LEFT JOINs ensure every product appears in the result regardless of whether it has sales history or an inventory record. The COALESCE on SUM(s.qty) converts the NULL aggregate (no sales rows) to 0.

Why MAX instead of plain column reference for inventory

Once LEFT JOIN inventory is added to a query that already has GROUP BY p.product_id, p.name, the inventory columns must either appear in the GROUP BY or be wrapped in an aggregate. Since inventory has exactly one row per product, MAX(i.on_hand) and MIN(i.on_hand) are equivalent — both return the single value. Using MAX is the conventional pattern for a 1:1 join attribute when the GROUP BY is driven by the many-side table (sales). An alternative is to aggregate sales first in a subquery, then JOIN that to products and inventory without a GROUP BY.

The CASE WHEN ≤ boundary

The condition is MAX(i.on_hand) <= MAX(i.reorder_point). Using strict < instead would leave Book (on_hand = reorder_point = 5) as "OK" — but the business intent is typically "trigger reorder when stock is at or below the threshold," which requires ≤. Always clarify the boundary with the stakeholder before coding.

The defensive regex in the original query

NULLIF(regexp_replace(i.on_hand::text, '[^-.0-9]', '', 'g'), '')::numeric strips all non-numeric characters from a text representation, converts empty result to NULL via NULLIF, then casts to numeric. This handles dirty data like "12 units", "$30", or completely empty strings. In Snowflake, TRY_TO_NUMBER(i.on_hand) is a cleaner built-in alternative that returns NULL on parse failure without a regex. The fact that this pattern is in the original query signals the table was inherited with a poorly typed schema.

Dialect notes

The CASE WHEN expression is ANSI SQL. In Snowflake, IFF(on_hand <= reorder_point, 'REORDER', 'OK') is a shorter single-condition alternative. COALESCE is universally supported. The regex defensive cast uses PostgreSQL syntax; in Snowflake use TRY_CAST(i.on_hand AS NUMBER) or TRY_TO_NUMBER(i.on_hand).

Senior signal: explain that both LEFT JOINs are required to preserve zero-sale and no-inventory products, clarify that MAX on a 1:1 join attribute is the idiom for "unwrap a 1:1 join inside a GROUP BY," explicitly state whether the CASE uses < or ≤ and why the business intent determines the answer, and mention Snowflake's TRY_TO_NUMBER as the clean replacement for the defensive regex cast.
