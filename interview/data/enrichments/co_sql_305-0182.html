Infosys runs an NLP pipeline that scores customer reviews from −1 (most negative) to +1 (most positive). The product team wants one summary row: the average sentiment, plus how many reviews are clearly positive and clearly negative. It looks like three independent aggregates — but the thresholds and the NULL handling are where an interviewer probes whether you understand CASE inside an aggregate.

1. The source data — reviews

Seven rows, but two edge cases. Review 3 (0.1) and review 6 (0.4) are neutral — they fall in the −0.5-to-+0.5 dead zone and count as neither positive nor negative. Review 7 has a NULL score: the scorer failed, or the review had no text. How the three aggregates treat that NULL differs.

2. The SQL — AVG + conditional SUM

SELECT
  AVG(sentiment_score) AS avg,
  SUM(CASE WHEN sentiment_score > 0.5 THEN 1 ELSE 0 END) AS positive,
  SUM(CASE WHEN sentiment_score < -0.5 THEN 1 ELSE 0 END) AS negative
FROM reviews;

One scan, three numbers. Each CASE maps a row to 1 or 0, and SUM tallies the 1s — the classic "conditional count" pattern.

3. The resulting analysis

avgpositivenegative
0.0583333…22

The six non-NULL scores sum to 0.35, and 0.35 ÷ 6 = 0.0583 — AVG dropped the NULL row from both sum and count. positive = 2 (only 0.9 and 0.7 clear +0.5; the 0.4 is neutral). negative = 2 (−0.8 and −0.95). The two neutral reviews and the NULL all sit outside both buckets — so positive + negative is 4, not 7.

4. Strategic explanations

The NULL is treated inconsistently across the three columns — and that is fine

AVG ignores the NULL automatically (it divides by COUNT(sentiment_score) = 6). The conditional SUMs also exclude it, but for a different reason: NULL > 0.5 evaluates to unknown, the WHEN is not satisfied, so the ELSE 0 branch fires and contributes 0. Different mechanism, same outcome — the NULL review counts toward nothing. Be able to articulate both paths.

Why SUM(CASE…) and not COUNT(CASE…)

A frequent bug is writing COUNT(CASE WHEN sentiment_score > 0.5 THEN 1 END) — which works — versus COUNT(CASE WHEN sentiment_score > 0.5 THEN 1 ELSE 0 END) — which is wrong. COUNT counts non-NULL values, so with an ELSE 0 it counts every row (the 0s included) and returns 7. The reliable rules: SUM(CASE … THEN 1 ELSE 0 END), or COUNT(CASE … THEN 1 END) with no ELSE. Postgres also offers the cleaner FILTER clause:

SELECT AVG(sentiment_score) AS avg,
       COUNT(*) FILTER (WHERE sentiment_score > 0.5)  AS positive,
       COUNT(*) FILTER (WHERE sentiment_score < -0.5) AS negative
FROM reviews;
The dead zone — is it intentional?

Scores in [−0.5, +0.5] are counted as neither positive nor negative. That is a deliberate "strongly positive / strongly negative" definition, but the boundaries are arbitrary. Confirm with the product owner: should the buckets be > 0 / < 0 (no dead zone), or is a third neutral count wanted? If neutral matters, add SUM(CASE WHEN sentiment_score BETWEEN -0.5 AND 0.5 THEN 1 ELSE 0 END) so the three buckets partition the non-NULL rows.

Boundary values — strict vs inclusive

A score of exactly 0.5 satisfies neither > 0.5 nor < -0.5, so it lands in the neutral zone. If the spec says "0.5 and above is positive", the operator must change to >=. These off-by-one boundary decisions are exactly what interviewers poke at — state your choice rather than letting the > default it.

Edge case: an empty table or all-NULL scores

If reviews is empty, AVG returns NULL while both SUM(CASE…) columns return NULL too (SUM of zero rows is NULL, not 0). The same happens if every score is NULL. A dashboard expecting numbers needs COALESCE(SUM(CASE…), 0). Worth noting because the conditional-sum pattern feels like it should bottom out at 0.

Senior signal: explain that AVG and the conditional SUMs exclude the NULL by two different mechanisms, never pair COUNT with an ELSE 0, surface the arbitrary ±0.5 dead-zone boundaries as a definition question, and flag that SUM over zero rows returns NULL rather than 0.
