A finance team wants total spend per customer for a billing report — and crucially, customers who have never ordered must still appear, showing 0, not a blank. The interviewer is probing two linked ideas: that an inner join would silently drop the zero-order customers, and that SUM over no rows returns NULL, which COALESCE must convert to a real zero.

Cara is the edge case the whole question is built around: a customer with zero matching order rows. A correct answer must still emit a Cara row, and her total must read 0 rather than NULL or nothing.

2. The SQL — LEFT JOIN with COALESCE

SELECT
  c.id,
  c.name,
  COALESCE(SUM(o.amount), 0) AS total
FROM customer c
LEFT JOIN orders o ON o.customer_id = c.id
GROUP BY c.id, c.name
ORDER BY c.id;

The LEFT JOIN keeps every customer row even when no orders row matches — the order columns simply come back NULL for those customers. GROUP BY then collapses each customer's rows, SUM totals their amounts, and COALESCE rewrites the NULL that SUM produces for an all-NULL group into 0.

Cara is the proof point. The LEFT JOIN gives her one row with o.amount = NULL; SUM of a group containing only NULL is NULL; COALESCE(NULL, 0) turns it into 0. With a plain INNER JOIN Cara would have produced no rows at all and vanished from the billing report — a missing customer, not a zero one.

4. Strategic explanations

Why LEFT JOIN and not INNER JOIN

An INNER JOIN emits a row only when both sides match, so a customer with no orders contributes nothing and disappears. LEFT JOIN guarantees every left-side (customer) row survives; unmatched right-side columns are filled with NULL. The requirement "including customers with zero orders" is precisely the definition of a left join — recognising that phrasing as a join-type signal is the core skill being tested.

SUM of nothing is NULL, not 0

This surprises people: SUM ignores NULLs, and a group made entirely of NULLs (or of no rows) has nothing to add, so the result is NULL — not 0. That is why COALESCE(SUM(o.amount), 0) is needed: it supplies the zero the report wants. COUNT behaves differently — COUNT(o.order_id) of an all-NULL group is genuinely 0 — so if the report needed an order count, no COALESCE would be required there.

The filter-placement trap: WHERE vs the ON clause

If the task were "total of completed orders, still keeping zero-order customers", a predicate like o.status = 'completed' must go in the ON clause, not WHERE. A WHERE o.status = 'completed' tests NULL = 'completed' for Cara's unmatched row, which is false — quietly converting the LEFT JOIN back into an inner join and dropping her. Filters on the optional table belong in ON; filters on the preserved table can stay in WHERE.

Grouping by the natural key, and an aggregate-free alternative

GROUP BY c.id, c.name includes name because every non-aggregated SELECT column must be grouped (or functionally dependent on the group key). Grouping by c.id alone would also be valid in PostgreSQL when id is the primary key, since name is then functionally determined — but listing both is the portable habit. A correlated-subquery form — SELECT c.id, c.name, COALESCE((SELECT SUM(amount) FROM orders WHERE customer_id = c.id), 0) — avoids the join and the GROUP BY entirely and reads cleanly for one aggregate.

Senior signal: read "including customers with zero orders" as a mandate for LEFT JOIN, know that SUM over an all-NULL group returns NULL so COALESCE(..., 0) is required to show a true zero, and remember that any filter on the optional table must sit in the ON clause or it silently degrades the left join into an inner join.
