A sales dashboard needs the most active customer — the one with the most orders — and if several customers tie at the top, all of them should show. The interviewer steers you toward a window function specifically because ties must "surface naturally": RANK() gives every co-leader the same rank 1, where a LIMIT 1 would arbitrarily keep just one.

1. The source data — customers and orders

customer_idnameorders(narrative)
1Adobe3 (Jan 1–3)3 orders — tied for the lead…
2Google3 (Feb 1–3)…also 3 orders — tied with Adobe
3Apple2 (Mar 1–2)2 orders — behind the leaders
4Twitter1 (Apr 1)single order — lowest

The deliberate edge case: Adobe and Google tie at 3 orders — the joint maximum. A correct answer must return both. Twitter, with a single order, tests the bottom of the distribution and confirms the query is not accidentally returning everyone.

2. The SQL — count, rank, keep rank 1

WITH counts AS (
  SELECT customer_id, COUNT(*) AS order_count
  FROM orders
  GROUP BY customer_id
),
ranked AS (
  SELECT customer_id, order_count,
         RANK() OVER (ORDER BY order_count DESC) AS rk
  FROM counts
)
SELECT r.customer_id, c.name, r.order_count::int AS order_count
FROM ranked r
JOIN customers c ON c.customer_id = r.customer_id
WHERE rk = 1
ORDER BY r.customer_id;

counts reduces orders to one row per customer with their order tally. ranked assigns a competition rank by descending count. WHERE rk = 1 keeps the leader(s); the join to customers attaches the name.

Both Adobe and Google appear, each with 3 orders — RANK() gave them the same rank 1, so WHERE rk = 1 keeps both. Apple (2) and Twitter (1) get ranks 3 and 4 and are filtered out.

4. Strategic explanations

Why RANK() is the right window function for "the max, ties included"

RANK() OVER (ORDER BY order_count DESC) gives every customer at the joint-highest count the value 1. Filtering rk = 1 therefore returns all co-leaders — precisely "the customer or customers, if tied". ROW_NUMBER() would be the wrong choice: it assigns 1, 2, 3, 4 unconditionally, so the two 3-order customers get distinct numbers and rn = 1 arbitrarily keeps just one of them. When ties must all win, use RANK (or DENSE_RANK); reserve ROW_NUMBER for "exactly one row".

Why this beats ORDER BY … LIMIT 1

The obvious shortcut — SELECT … ORDER BY order_count DESC LIMIT 1 — silently drops a co-leader: it returns one row no matter how many customers tie at the top. The question's "if tied" clause is exactly the case LIMIT mishandles. The RANK() … WHERE rk = 1 pattern is tie-correct and is the idiom to reach for whenever "the maximum" might not be unique. (LIMIT WITH TIES exists in newer Postgres and SQL Server, but is less portable than the window approach.)

Why aggregate first, then rank — the two-CTE shape

The ranking metric is COUNT(*) per customer, which must exist before it can be ordered. So the query aggregates in counts, then ranks the aggregated rows in ranked. You cannot collapse this into one level: RANK() OVER (ORDER BY COUNT(*) DESC) in a GROUP BY query is legal Postgres (the window runs after the group-by), but you still cannot reference the resulting rk in the same query's WHERE — window functions are evaluated after WHERE. A second query level (CTE or subquery) is required to filter on rk.

Customers with zero orders — the join direction trap

counts is built from orders, so a customer who never ordered has no row and cannot be ranked — correct here, since a zero-order customer is never the most active. But note the join: ranked JOIN customers is an inner join used only to fetch names for already-selected leaders. If the task were instead "rank all customers including zero-order ones", you would start from customers with a LEFT JOIN orders and COUNT(o.order_id) (which counts 0 for non-matches, unlike COUNT(*)). Knowing which join direction the question needs is the senior signal.

Senior signal: use RANK() OVER (ORDER BY order_count DESC) filtered to rk = 1 so every tied top customer surfaces — never ROW_NUMBER or LIMIT 1, which keep just one — aggregate before ranking in a separate CTE because window results cannot be filtered in WHERE, and pick the join direction based on whether zero-order customers must be ranked.
