A trust-and-safety service needs the list of users who are not banned, but the banned table has a known data-quality flaw: it sometimes contains a NULL user_id. The interviewer is deliberately setting the classic NOT IN trap — a single NULL in the subquery silently turns the whole anti-join into the empty set. The task asks you to show both the broken naive count and the correct one side by side, so you must understand why three-valued logic breaks NOT IN.

2. The SQL — safe NOT EXISTS beside broken NOT IN

SELECT
  (SELECT COUNT(*) FROM users u
   WHERE NOT EXISTS (SELECT 1 FROM banned b WHERE b.user_id = u.user_id)) AS safe_count,
  (SELECT COUNT(*) FROM users u
   WHERE u.user_id NOT IN (SELECT user_id FROM banned)) AS naive_count;

safe_count uses NOT EXISTS with a correlated equality — immune to NULLs. naive_count uses NOT IN against a subquery that contains a NULL — the textbook bug. Both run in one statement so the two numbers can be compared directly.

4. Strategic explanations

Why one NULL empties a NOT IN

x NOT IN (a, b, NULL) expands to x <> a AND x <> b AND x <> NULL. The comparison x <> NULL is never TRUE — it evaluates to UNKNOWN in SQL's three-valued logic. An AND chain that contains an UNKNOWN can be FALSE (if another term is false) but can never be TRUE. So for every user the predicate is at best UNKNOWN, the WHERE keeps only TRUE rows, and the result is empty. The NULL does not just affect itself — it sabotages the test for every row.

Why NOT EXISTS is immune

NOT EXISTS (SELECT 1 FROM banned b WHERE b.user_id = u.user_id) asks a different question: "does any banned row's user_id equal this user's id?" The correlated predicate b.user_id = u.user_id against the NULL row evaluates to UNKNOWN, which EXISTS treats exactly like FALSE — that row does not satisfy the inner query, so it simply contributes nothing. EXISTS only ever returns a clean TRUE/FALSE; there is no UNKNOWN leaking out. The NULL row is harmlessly ignored, and users 1, 3, 5 correctly pass.

The third option: LEFT JOIN ... IS NULL

The anti-join can also be written FROM users u LEFT JOIN banned b ON b.user_id = u.user_id WHERE b.user_id IS NULL. This is also NULL-safe for the same reason — it matches on equality, and an unmatched user gets a NULL right side. It produces the correct 3. All three NULL-safe forms (NOT EXISTS, anti-JOIN, EXCEPT) are interchangeable; NOT IN against a nullable column is the one to avoid.

How to make NOT IN safe if you must use it

NOT IN is not forbidden — it is only dangerous when the subquery column is nullable. Two fixes: filter the NULLs out (... NOT IN (SELECT user_id FROM banned WHERE user_id IS NOT NULL)), or declare the column NOT NULL so the defect cannot occur. The deeper senior point: this is a data-quality bug. A NULL user_id in a banned table is meaningless — the real fix is a NOT NULL constraint at the schema level so the poisoned row can never be inserted in the first place.

Dialect notes

This three-valued-logic behaviour is SQL-standard — PostgreSQL, MySQL, SQL Server, and Oracle all empty a NOT IN the same way when the subquery yields a NULL. It is one of the most universal SQL footguns. Some optimisers will rewrite NOT IN to an anti-join only when they can prove the column is non-nullable; the moment NULLs are possible, the rewrite is illegal and you are stuck with the slow, wrong semantics.

Senior signal: know that a single NULL in a NOT IN subquery collapses the result to empty because x <> NULL is UNKNOWN, default to NOT EXISTS or an anti-JOIN which match on equality and are NULL-safe, and treat the underlying NULL user_id as a data-quality defect to fix with a NOT NULL constraint.
