A community-platform analyst needs to find pages nobody has liked — candidates for archival or a promotion push. The data hides a deliberate landmine: one page_likes row has a NULL page_id (an orphaned like that matches no page). The interviewer is probing whether you can express an anti-join correctly, and crucially whether you avoid NOT IN, whose interaction with NULL silently breaks this exact query.

1. The source data — pages and page_likes

1CatLoverstwo rows, page_id 1has likes

Separately, page_likes contains a row with page_id = NULL — a like attached to no page. It must not be treated as a match for any page, and it must not poison the "which pages are unliked" logic. Pages 2 and 4 are the expected answer.

2. The SQL — anti-join via NOT EXISTS

SELECT p.page_id
FROM pages p
WHERE NOT EXISTS (
  SELECT 1 FROM page_likes l WHERE l.page_id = p.page_id
)
ORDER BY p.page_id;

NOT EXISTS keeps a page only when the correlated subquery finds no matching like. The correlation l.page_id = p.page_id is an ordinary equality: a NULL page_id in page_likes can never satisfy it, so the orphaned like is ignored exactly as required.

3. The resulting analysis

Pages 1, 3, and 5 each have a matching like and are excluded. The NULL-page_id like contributes nothing: it does not match page 2 or 4 (so they correctly stay), and it does not match 1/3/5 either. The result is clean precisely because NOT EXISTS uses row-by-row correlation rather than set membership.

4. Strategic explanations

Why NOT IN is forbidden here — the NULL-IN gotcha

WHERE p.page_id NOT IN (SELECT page_id FROM page_likes) looks equivalent but is catastrophically wrong when the subquery returns a NULL. NOT IN expands to page_id <> v1 AND page_id <> v2 AND ... AND page_id <> NULL; that last term is UNKNOWN, and anything AND UNKNOWN can never be TRUE. The result: the query returns zero rows for every page, even pages that genuinely have no likes. The single orphaned like would blank the entire report.

Why NOT EXISTS is immune

NOT EXISTS asks a yes/no question — "did the subquery produce any row?" — not a value comparison. The subquery's WHERE l.page_id = p.page_id simply yields no rows for a NULL page_id (the equality is UNKNOWN, which is not TRUE, so the row is not emitted). No row produced means NOT EXISTS is TRUE. The three-valued logic stays contained inside the subquery and never propagates to a query-killing AND UNKNOWN.

LEFT JOIN ... IS NULL — the other safe anti-join

The equivalent portable idiom is SELECT p.page_id FROM pages p LEFT JOIN page_likes l ON l.page_id = p.page_id WHERE l.page_id IS NULL. The left join attaches likes where they exist; unmatched pages get all-NULL page_likes columns, and WHERE l.page_id IS NULL keeps exactly those. It is also NULL-safe, because the orphaned like's NULL page_id never matches any page in the ON clause. Both NOT EXISTS and LEFT JOIN/IS NULL are correct; NOT IN is the one to avoid.

Performance — anti-joins and indexing

With an index on page_likes(page_id), NOT EXISTS can stop at the first matching like per page (a semi-join probe), and most planners optimise NOT EXISTS and LEFT JOIN/IS NULL into the same anti-join plan. NOT IN with a nullable column, by contrast, often cannot be optimised into an anti-join at all because of the NULL semantics — another reason to prefer the alternatives even when correctness happens not to bite.

Dialect notes

NOT EXISTS is fully standard and behaves identically in PostgreSQL, MySQL, SQL Server, Oracle, and SQLite — this query needs no translation. The NOT IN/NULL hazard is universal too: it bites the same way in every engine, which is exactly why the question bakes "without NOT IN" into the requirement.

Senior signal: implement an anti-join with NOT EXISTS or LEFT JOIN ... IS NULL, never NOT IN over a nullable column — a single NULL collapses NOT IN to zero rows because x <> NULL is UNKNOWN — and explain that NOT EXISTS is safe because it tests row existence, keeping three-valued logic contained in the subquery.
