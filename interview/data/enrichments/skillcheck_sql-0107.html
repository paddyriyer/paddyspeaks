An org-chart or category-tree feature needs every node labelled by its structural role: roots have no parent, leaves have no children, and inner nodes have both. The interviewer is checking whether you can classify a node by a property of other rows (does any row name me as its parent?) inside a single self-contained query, without a procedural loop or a recursive walk.

Node 3 is the instructive case: it has a parent (so it is not a Root) yet appears in nobody's p_id column, so it is a Leaf, not an Inner node. Node 1 is the NULL-parent edge case the WHERE p_id IS NOT NULL filter inside the subquery must respect.

2. The SQL — CASE over a self-referencing membership test

SELECT id,
  CASE
    WHEN p_id IS NULL THEN 'Root'
    WHEN id IN (SELECT p_id FROM tree_node WHERE p_id IS NOT NULL) THEN 'Inner'
    ELSE 'Leaf'
  END AS node_type
FROM tree_node
ORDER BY id;

The CASE is evaluated top-down. First branch: a NULL p_id means root. Second branch: the row's id appears somewhere as a p_id — meaning at least one node calls it parent — so it has children. Anything reaching the ELSE has a parent but no children: a leaf.

The three roles are correctly partitioned: one Root, one Inner, three Leaves. The ordering of the CASE branches matters — the root test runs first so a hypothetical root that also had children would still be labelled Root, which is the conventional precedence.

4. Strategic explanations

Why the IN subquery is the natural "has children" test

"Does node X have children?" is equivalent to "does X's id appear in the p_id column of any row?" The set SELECT p_id FROM tree_node is precisely the set of all parents. Testing membership with id IN (...) answers it directly. No join, no aggregation — the structure of the question maps one-to-one onto a set-membership check.

The critical WHERE p_id IS NOT NULL inside the subquery

This filter looks optional but protects correctness with NOT IN and clarifies intent with IN. The root's p_id is NULL; leaving it in the subquery puts a NULL in the candidate set. With IN a NULL in the list is harmless (it just never matches), but if you ever rewrite the second branch as id NOT IN (SELECT p_id FROM tree_node) to find leaves directly, that stray NULL makes the whole NOT IN return unknown for every row — and no row is classified as a leaf. Filtering NULLs out is the habit that makes the rewrite safe.

Branch order encodes the precedence rule

CASE stops at the first matching WHEN. Putting p_id IS NULL first means "root" wins over "inner" if a node were somehow both. In a strict tree the root has no parent and the inner test is independent, so order rarely bites here — but stating the precedence explicitly (root, then inner, then leaf) is what a reviewer wants to hear, because in a malformed graph the order is the tie-breaker.

Dialect notes — this query is fully portable

CASE, scalar IN subqueries, and IS NULL are core SQL and behave identically in PostgreSQL, MySQL, SQL Server, Oracle, and SQLite. No translation needed. The one engine-specific concern is performance: a LEFT JOIN or EXISTS form (below) can plan better than IN on some optimisers, but all three are logically equivalent here.

A cleaner alternative — EXISTS or a parent-count join

A senior might prefer a correlated EXISTS: WHEN EXISTS (SELECT 1 FROM tree_node c WHERE c.p_id = t.id) THEN 'Inner'. It conveys "has at least one child" more literally and short-circuits on the first match. For large trees you could instead LEFT JOIN to an aggregated child-count CTE once, avoiding a per-row subquery scan. Mentioning that the IN form is fine for modest data but you would switch to a single pre-aggregated join at scale is the senior signal.

Senior signal: classify a node by testing whether its id appears in the table's own p_id column, always filter p_id IS NOT NULL in that subquery so a NULL-root never poisons a NOT IN rewrite, order the CASE branches deliberately, and note you would pre-aggregate child counts into one join at scale instead of a per-row subquery.
