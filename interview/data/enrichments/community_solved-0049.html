The classic gaps-and-islands streak detection, dressed up as an HR comp question. "At least 3 consecutive salary increases" is the same shape as "at least 3 consecutive days of activity", "at least 3 consecutive winning trades", "at least 3 consecutive on-time deliveries". Once you have the pattern, every variant of the prompt drops out. The trap: most candidates reach for LAG + a CASE flag and stop there — but you haven't identified the streak, you've just labelled individual rows. The cumulative-flag trick is what stitches the labelled rows into named streaks.

3. The algorithm — five steps the interviewer wants to hear in order

Step 1 — LAG to get each row's previous salary
SELECT emp_id, log_id, salary,
  LAG(salary) OVER (PARTITION BY emp_id ORDER BY log_id) AS prev_salary
FROM salary_logs;

For Aarav, this produces: (prev_salary) = NULL, 50K, 55K, 60K, 65K. The first row's prev is NULL because there's no row before it.

Step 2 — Flag each row as is_increase
CASE WHEN salary > prev_salary THEN 1 ELSE 0 END AS is_increase

For Aarav: 0, 1, 1, 1, 1 (first row is 0 because prev_salary is NULL; NULL > anything = NULL = falsy, which the CASE turns into 0). For Bhavna: 0, 1, 0, 1, 1 — the drop at log 8 lands as a 0.

Stopping here is the junior mistake. You have a per-row label, but you haven't grouped the increase rows into streaks. The next step is the gaps-and-islands trick.

Step 3 — The cumulative-flag group_id
SUM(CASE WHEN is_increase = 0 THEN 1 ELSE 0 END)
  OVER (PARTITION BY emp_id ORDER BY log_id) AS group_id

This is the trick. The running SUM of (1 - is_increase) increments by 1 every time we hit a non-increase row, and stays flat through increase rows. So every consecutive run of increases shares the same group_id, and a non-increase row "starts" a new group.

Bhavna gets two groups. Within group 1, only one row is an increase. Within group 2, only two rows are increases. Neither reaches 3, so Bhavna is filtered out at the HAVING.

Step 4 — Filter to increase rows, group, HAVING ≥ 3
SELECT emp_id, group_id,
  COUNT(*)    AS consecutive_increases,
  MIN(log_id) AS streak_first_log,
  MAX(log_id) AS streak_last_log
FROM grouped
WHERE is_increase = 1
GROUP BY emp_id, group_id
HAVING COUNT(*) >= 3;

The WHERE is_increase = 1 drops the non-increase rows. The GROUP BY emp_id, group_id collapses each streak to one row. HAVING COUNT(*) >= 3 keeps only the streaks that meet the threshold.

Step 5 — Join to employees for the human-readable output

The streaks CTE has emp_id; join to get emp_name + department.

4. Why the shipped "Method 1" and "Method 2" were both wrong

Method 1's bug — partition-wide SUM with no group_id
-- (the original Method 1)
SUM(nw_flg) OVER (PARTITION BY emp_id) AS period
...
GROUP BY t3.emp_id, period, e.emp_name, e.department

Two problems:

  • SUM(nw_flg) OVER (PARTITION BY emp_id) with no ORDER BY is just the net count of increase rows minus decrease rows for the whole employee — a single number per employee. It doesn't identify which rows form streaks.
  • The GROUP BY then de-duplicates by emp_id + that net number, which means employees with even one increase show up. The HAVING/filter to >= 3 is missing too.

Net effect: returns every employee with at least one log, regardless of whether they had a streak.

Method 2's bug — partition by salary, not by streak
-- (the original Method 2)
ROW_NUMBER() OVER (PARTITION BY emp_id, salary ORDER BY log_id) AS grp

This is a well-known gaps-and-islands variant — but it groups rows where the salary is the same, not rows where there's a streak of increases. For Esha (salary stays at $50K for three logs), this would group all three rows together with COUNT=3 → and incorrectly flag Esha as "3 consecutive increases" when her salary never increased at all. The partition axis was wrong.

5. The interviewer's edge-case probes

Probe 1 — "Does your query handle ties (salary stays the same)?"

Yes. CASE WHEN salary > prev_salary THEN 1 ELSE 0 END treats equality as 0 (not an increase). So Esha's $50K → $50K → $50K produces all zeros, no streaks. If the prompt meant "non-decreasing" instead of "strictly increasing", change to salary >= prev_salary and be ready to defend it.

Probe 2 — "What if log_id isn't sequential per employee — what if there are gaps?"

The algorithm doesn't care. It only uses ORDER BY log_id to define the sequence; gaps in log_id values are fine. If the question's "consecutive" means "consecutive in log order" (which is how it reads), gaps are irrelevant. If "consecutive" means "consecutive in time" (no missing months), you'd switch to a date-based check with explicit gap tolerance.

Probe 3 — "What if the first row should count as an 'increase' (e.g. starting salary)?"

Treat NULL prev_salary as 0:

CASE WHEN salary > COALESCE(prev_salary, 0) THEN 1 ELSE 0 END AS is_increase

Surface this in the interview — most users don't intend "starting salary counts as a raise" but it's worth asking.

Probe 4 — "At Netflix scale (millions of employees, billions of salary log rows), does this query work?"

The window functions (LAG, SUM OVER ORDER BY) require sorting within each employee partition. With partition pruning on emp_id this is O(n log n) per employee, but the global sort is the bottleneck. Two production optimisations:

  • Incremental computation — compute the streak length on each new log_id row as it arrives (a streaming job tracks the running streak); only the latest streak per employee needs to live in a hot table.
  • Pre-filter to recent rows — if the question is "anyone with a 3+ streak in the last year", filter to that window before the window functions.
Probe 5 — "Generalise to 'any N consecutive increases' — does the query stay the same?"

Yes. The only change is HAVING COUNT(*) >= N, with N parameterised. Same algorithm scales to "3 increases", "5 increases", "10 increases" without rewriting the CTE chain.

Probe 6 — "What about employees with MULTIPLE qualifying streaks? Does each one get a row?"

Yes. GROUP BY emp_id, group_id means one row per streak per employee. If Aarav had increases at logs 2-5 AND again at logs 10-13, you'd see two rows for Aarav with different streak_first_log / streak_last_log. To collapse to "the longest streak per employee", wrap in another GROUP BY emp_id with MAX(consecutive_increases).

Senior signal: name the algorithm out loud — "this is gaps-and-islands; I'll use LAG to flag each row, then a cumulative running-sum of non-flag rows as the group_id, then filter and HAVING." The five steps in that order is the interview answer. Then pre-empt the ties question (Probe 1), the starting-salary question (Probe 3), and the generalise-to-N question (Probe 5). The shipped methods on this question both got the algorithm wrong — Method 1 used a partition-wide SUM with no group_id, Method 2 partitioned on the wrong axis (salary instead of streak). Knowing why those approaches break is what separates a senior answer from a "looks right" answer.
