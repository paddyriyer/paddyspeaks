DoorDash's data team is reconciling two pipelines that should produce identical orders — the legacy Postgres OLTP and the new Snowflake mart. The ops question: "which order_ids appear in BOTH?" The answer is the canonical use of INTERSECT — the cleanest of the three SQL set operators (UNION, INTERSECT, EXCEPT). One line of SQL, but the dialect support and dedup behaviour are surprising.

4. Strategic explanations

The three set operators — one cheat sheet
operatorreturnsanalogous to
UNIONrows in A or B (deduplicated)OR + DISTINCT
UNION ALLrows in A and B (with duplicates)OR (no dedup)
INTERSECTrows in A and B (deduplicated)AND
EXCEPT (or MINUS in Oracle)rows in A but not B (deduplicated)AND NOT

All three implicitly DISTINCT the result — that's the gotcha that surprises senior engineers used to UNION ALL's no-dedup behaviour.

The "with duplicates preserved" variants

Postgres, BigQuery, Snowflake and SQL Server support:

SELECT order_id FROM orders_a INTERSECT ALL SELECT order_id FROM orders_b;
-- Returns each id MIN(count_in_a, count_in_b) times — multiset semantics

For our data with O_005 duplicated in A but unique in B, INTERSECT ALL returns O_005 once (the min). MySQL doesn't support INTERSECT at all (until 8.0.31) — portability matters.

Equivalent forms — when INTERSECT isn't available
-- Inner join on the matching column
SELECT DISTINCT a.order_id
FROM orders_a a JOIN orders_b b USING (order_id);

-- IN / EXISTS
SELECT DISTINCT order_id FROM orders_a
WHERE order_id IN (SELECT order_id FROM orders_b);

SELECT DISTINCT order_id FROM orders_a a
WHERE EXISTS (SELECT 1 FROM orders_b b WHERE b.order_id = a.order_id);

For a single column the JOIN form is the most readable. For "all columns must match exactly" (full row dedup), INTERSECT is shorter than the equivalent JOIN with N=ColCount equality predicates.

The NULL trap — INTERSECT treats NULLs as equal

Surprising but true: SELECT NULL INTERSECT SELECT NULL returns one row. INTERSECT uses "is not distinct from" semantics, not =. This is the right behaviour for set semantics but trips up engineers who expect NULL = NULL → NULL → FALSE.

-- INTERSECT: NULL behaves like a value (matches another NULL)
-- JOIN with =: NULL never matches NULL — query loses these rows silently

If you're dedup-reconciling rows that may contain NULLs, INTERSECT is safer than the JOIN form.

The reconciliation report — all three set operators in one query

The actual ask is rarely just "the intersection" — it's the full reconciliation matrix:

SELECT 'in_both'    AS bucket, COUNT(*) AS n FROM (SELECT order_id FROM orders_a INTERSECT SELECT order_id FROM orders_b) t
UNION ALL
SELECT 'only_in_a',           COUNT(*)        FROM (SELECT order_id FROM orders_a EXCEPT    SELECT order_id FROM orders_b) t
UNION ALL
SELECT 'only_in_b',           COUNT(*)        FROM (SELECT order_id FROM orders_b EXCEPT    SELECT order_id FROM orders_a) t;

Three counts, one slide. The data-quality team's daily smoke test for any pipeline migration.

Schema-mismatch trap — row shape matters

INTERSECT requires both sides to have the same number of columns and compatible types. The names are taken from the first SELECT. If orders_a has order_id BIGINT and orders_b has order_id VARCHAR, you must cast: SELECT order_id::TEXT FROM orders_a INTERSECT SELECT order_id FROM orders_b.

The "row-level reconciliation" pattern

For "do these two tables match column-for-column?", run EXCEPT in both directions:

(SELECT * FROM orders_a EXCEPT SELECT * FROM orders_b)
UNION ALL
(SELECT * FROM orders_b EXCEPT SELECT * FROM orders_a);

Empty result → the tables are identical. Any rows → precise diff of the mismatch. Standard pattern for golden-vs-candidate ETL validation.
