A merchandising team wants the customers who bought both staple products A and B but never touched product C — a classic "has all of these, none of those" segmentation. The interviewer is checking whether you can express set membership and set absence in a single GROUP BY, rather than stitching together INTERSECT and EXCEPT or three subqueries.

1. The source data — customer_purchases

customer_idproduct(narrative)
1A, B, Chas C → disqualified
2A, BA and B, no C → qualifies
3A, B, Aduplicate A, still A+B no C → qualifies
4A, Chas C, and no B → disqualified
5B, Bonly B, missing A → disqualified
6A, B, A, Bduplicates of both, no C → qualifies

Two edge cases are deliberate. Customer 3 and 6 buy the same product more than once — a correct query must not double-count them out of contention. Customer 5 has rows but is missing A entirely — "having some products" is not "having the required products."

2. The SQL — conditional counts in HAVING

SELECT customer_id
FROM customer_purchases
GROUP BY customer_id
HAVING COUNT(*) FILTER (WHERE product = 'A') >= 1
   AND COUNT(*) FILTER (WHERE product = 'B') >= 1
   AND COUNT(*) FILTER (WHERE product = 'C') = 0
ORDER BY customer_id;

Each COUNT(*) FILTER (WHERE …) tallies, within one customer's group, how many rows match a product. The HAVING clause then asserts: at least one A, at least one B, exactly zero C.

3. The resulting analysis

customer_id(why it passed)
21 A, 1 B, 0 C
32 A, 1 B, 0 C — the duplicate A still counts as ≥ 1
62 A, 2 B, 0 C — duplicates of both, still no C

Customers 1 and 4 are dropped by the C = 0 test; customer 5 is dropped by the A >= 1 test. Using >= 1 rather than = 1 is what lets the repeat-buyers 3 and 6 through — the question asks "bought A," not "bought A exactly once."

4. Strategic explanations

Why one GROUP BY beats INTERSECT/EXCEPT

You could write (SELECT customer_id WHERE product='A') INTERSECT (SELECT … 'B') EXCEPT (SELECT … 'C'). It works, but it scans the table three times and the logic is spread across three set operations. The conditional-count form reads the table once, groups once, and states all three rules side by side — easier to extend (add product D) and easier for a reviewer to verify.

The presence vs absence asymmetry

Presence is >= 1; absence is = 0. A frequent bug is writing the absence test as COUNT(*) FILTER (WHERE product != 'C') = COUNT(*) — subtly different and easy to get wrong with NULLs. FILTER (WHERE product = 'C') = 0 is direct: it counts the forbidden rows and demands there be none.

NULL products would slip past the absence test

If product could be NULL, a row with product IS NULL satisfies neither = 'A' nor = 'C' (both evaluate to unknown). That is harmless for the C-absence check — a NULL is correctly not counted as a C — but if the business rule were "NULL means unknown product, treat as possibly C," you would need COUNT(*) FILTER (WHERE product = 'C' OR product IS NULL) = 0. State the assumption.

Dialect — FILTER vs CASE

COUNT(*) FILTER (WHERE …) is standard SQL and supported by PostgreSQL and SQLite (3.30+). MySQL and older engines lack it; there the portable form is SUM(CASE WHEN product = 'A' THEN 1 ELSE 0 END) >= 1. The two are semantically identical — FILTER is simply the cleaner spelling. Knowing both lets you adapt without changing the logic.

What to say next

Note that the query returns customers who bought A and B among other things — it does not require A and B to be their only products. If the requirement were "bought exactly A and B and nothing else," you would add AND COUNT(*) FILTER (WHERE product NOT IN ('A','B')) = 0. Clarifying "at least" versus "exactly" before writing the query is the senior move.

Senior signal: collapse "has all of X, none of Y" into a single GROUP BY with one COUNT(*) FILTER per rule, use >= 1 so repeat purchases still count as presence, express absence as = 0, and know the SUM(CASE) fallback for engines without FILTER.
