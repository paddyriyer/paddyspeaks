TikTok's "Trending" tab needs more than total volume — the algorithm picks hashtags accelerating. The query computes WoW growth via LAG.

2. The SQL — week-over-week growth

WITH weekly AS (
  SELECT hashtag_id,
    CAST((julianday(used_at) - julianday('2024-01-01')) / 7 AS INTEGER) AS week_n,
    COUNT(*) AS uses
  FROM hashtag_uses GROUP BY hashtag_id, week_n
)
SELECT hashtag_id, week_n, uses,
  LAG(uses) OVER (PARTITION BY hashtag_id ORDER BY week_n) AS prev_week,
  ROUND(100.0 * (uses - LAG(uses) OVER (PARTITION BY hashtag_id ORDER BY week_n))
              / NULLIF(LAG(uses) OVER (PARTITION BY hashtag_id ORDER BY week_n), 0), 1) AS wow_pct
FROM weekly ORDER BY hashtag_id, week_n;

3. Strategic explanations

The week-bucket via integer division

(julianday(date) - julianday(epoch)) / 7 casts to integer = week number from epoch. Cleaner than strftime('%W') which has year-boundary issues.

The double-LAG idiom

Using LAG(...) twice (once for value, once in the formula) is repetitive. SQLite doesn't have a way to alias window expressions in WHERE; production wraps in another CTE.

Trending vs spam

1 → 100 uses (10000% growth) is meaningless if absolute volume is tiny. Filter:

WHERE uses >= 100 AND wow_pct >= 50
Decay detection

Trending hashtags peak then decay. Negative WoW after several positive = peak passed. Production demotes from "Trending" page.

Geographic localization

"Brat summer" trends US. Different hashtags trend in IN, BR, DE. Production runs per-country trend detection.

Senior signal: integer week-bucket via julianday math, filter both volume + growth to avoid noise, detect decay for demoting from Trending, and run per-country localization.
