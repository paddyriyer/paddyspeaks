DoorDash's operations team wants to flag "slow" orders — ones that took more than 48 hours from placement to completion — for a fulfilment audit. The core idea is a timestamp difference; the interview value is in how you compute a duration in hours, whether the threshold comparison is exact, and whether the chosen formula introduces floating-point noise.

Order 2 spans exactly 50 hours — just past the 48-hour line, the boundary case that tests whether the threshold is exclusive. Orders 4 (72h) and 5 (102h) are clearly slow; orders 1, 3, 6 are same-day and fast.

2. The SQL — EPOCH duration over 48h

SELECT order_id,
  ((EXTRACT(EPOCH FROM (completed_at - placed_at)) / 86400.0) * 86400) / 3600 AS hrs
FROM orders
WHERE ((EXTRACT(EPOCH FROM (completed_at - placed_at)) / 86400.0) * 86400) / 3600 > 48;

completed_at - placed_at is an interval; EXTRACT(EPOCH FROM …) turns it into total seconds. The expression then divides by 86400 (seconds per day), multiplies by 86400, and divides by 3600 (seconds per hour). The WHERE repeats the same expression to keep only durations above 48 hours.

Three orders cross the line. But look at order 2: it spans exactly 50 hours, yet hrs reads 49.9999999999999992. The / 86400.0 step promotes the value to floating point, and 50 has no exact binary representation once divided and re-multiplied — the round trip leaves residue. The number is right; its presentation is not.

4. Strategic explanations

The formula is a no-op dressed up — simplify it

((seconds / 86400.0) * 86400) / 3600 divides by 86400 and immediately multiplies by 86400 — algebraically those cancel, leaving seconds / 3600. The detour buys nothing except the floating-point error you can see in order 2's row. Write the duration directly:

SELECT order_id,
  EXTRACT(EPOCH FROM (completed_at - placed_at)) / 3600 AS hrs
FROM orders
WHERE completed_at - placed_at > INTERVAL '48 hours';

Spotting that an expression simplifies — and that the redundant step actively introduces noise — is exactly the senior instinct an interviewer rewards.

Compare intervals directly — skip the arithmetic entirely

The cleanest filter never converts to a number at all. completed_at - placed_at is an interval, and intervals compare directly against an interval literal: WHERE completed_at - placed_at > INTERVAL '48 hours'. This is exact — no epoch, no division, no float — and reads as the business rule verbatim. Compute the hrs column for display if the report needs it, but do the filtering on the interval.

The boundary — is exactly 48 hours slow?

The query uses > 48, so an order taking precisely 48 hours is not flagged. Is that intended? "More than 48 hours" usually means strictly greater, so > is defensible — but confirm it. If 48 hours exactly should count, the operator is >=. And the floating-point residue makes this worse: a value that is "really" 48.0 could compute as 47.9999… and slip under a >= 48 test. Interval comparison sidesteps that fragility completely.

Repeating the expression — DRY it with a CTE or lateral

The solution writes the duration formula twice — once in SELECT, once in WHERE. You cannot reference a SELECT alias in the same query's WHERE (the WHERE is evaluated first), but a CTE or subquery computes it once:

WITH durations AS (
  SELECT order_id,
         EXTRACT(EPOCH FROM (completed_at - placed_at)) / 3600 AS hrs
  FROM orders)
SELECT * FROM durations WHERE hrs > 48;

One definition of hrs means the displayed value and the filtered value can never drift apart — and rounding (ROUND(hrs, 1)) can be applied in one place.

NULL timestamps — orders still open

An order that has been placed but not yet completed has completed_at = NULL. completed_at - placed_at is then NULL, the duration is NULL, and NULL > 48 is unknown — so the order is silently excluded. An order open for a week is arguably the slowest of all, yet this query never flags it. Decide explicitly: filter WHERE completed_at IS NOT NULL to be clear you mean "completed but slow", or measure open orders against now() with COALESCE(completed_at, now()) - placed_at if stuck-open orders must surface.

Senior signal: recognise that (x/86400)*86400 cancels to a no-op that only adds floating-point noise — visible as 49.999… for an exact 50-hour order — prefer a direct interval > INTERVAL '48 hours' comparison that is exact and self-documenting, compute the duration once via a CTE, and handle NULL completed_at deliberately rather than letting the slowest open orders vanish.
