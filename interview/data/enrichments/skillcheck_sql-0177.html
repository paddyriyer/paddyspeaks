Finding the Lowest Common Ancestor (LCA) in an org-chart hierarchy is a classic tree-algorithm question that SQL interviewers use to test recursive CTEs and set intersection. The real-world problem: given two employees, who is the most specific manager (not just "everyone reports to the CEO") that oversees both? The approach — walk each employee's ancestor chain to the root, then intersect and pick the shallowest common ancestor — generalises to any directed tree stored as an adjacency list.

2. The SQL — recursive CTE ancestor chains then intersection

WITH RECURSIVE ancestors AS (
  SELECT emp_id AS start_id, emp_id AS anc_id, 0 AS depth
  FROM employee
  UNION ALL
  SELECT a.start_id, e.manager_id, a.depth + 1
  FROM ancestors a
  JOIN employee e ON e.emp_id = a.anc_id
  WHERE e.manager_id IS NOT NULL
)
SELECT a1.anc_id AS lca_emp_id, e.name AS lca_name
FROM ancestors a1
JOIN ancestors a2 ON a1.anc_id = a2.anc_id
JOIN employee e ON e.emp_id = a1.anc_id
WHERE a1.start_id = 6 AND a2.start_id = 7
ORDER BY a1.depth ASC
LIMIT 1;

The recursive CTE seeds with every employee as their own ancestor at depth 0, then walks up: at each step, it replaces the current node (anc_id) with its manager, incrementing depth. The recursion stops when manager_id IS NULL (the root). The outer query filters to the two target chains, joins on the shared ancestor node, orders by ascending depth (so the deepest common ancestor — the one closest to the targets — appears first), and returns only the first row.

4. Strategic explanations

Recursive CTE structure: seed + recursive step + termination

Every recursive CTE has three parts. The anchor (before UNION ALL) defines the base case: here, every employee is their own ancestor at depth 0. The recursive member extends each row one step up the tree: given a current node, it looks up that node's manager and adds a new row with depth + 1. The termination condition is the WHERE e.manager_id IS NOT NULL guard: when the current node is the root (Alice, manager_id NULL), the join finds no parent, no new rows are added, and recursion ends. Without this guard the CTE would loop indefinitely once it reached the root.

Why depth ASC LIMIT 1 gives the LCA and not the root

Depth measures how many edges separate the ancestor from the starting employee. The LCA is the common ancestor that is closest to the two targets, which means the smallest depth among common ancestors — ORDER BY a1.depth ASC LIMIT 1. It is tempting to think "deepest in the tree" means largest depth number, but depth is measured from the target downward to the root, so depth 2 is closer (deeper in the tree, further from the root) than depth 3.

Edge case: one employee is an ancestor of the other

If emp_id 6 were instead emp_id 4 (Dan), the anchor clause seeds Dan as his own ancestor at depth 0, so the chain for emp_id 4 includes row (4, 4, 0). The chain for emp_id 7 (Gita) passes through node 4 at depth 1 (4 → 2 → 1). The intersection at anc_id = 4 yields depth 0 for the chain starting at 4, which is the minimum. So Dan himself is returned as the LCA — correct, because the question specifies "if one is an ancestor of the other, return that employee."

Performance considerations

This CTE generates ancestor rows for all employees in the table even though only two chains are needed. A more efficient approach filters in the anchor: replace FROM employee with FROM employee WHERE emp_id IN (6, 7). This limits recursion to only two chains, dramatically reducing work on large hierarchies. For very deep trees (corporate hierarchies with 10+ levels), also consider adding a depth-limit guard (AND a.depth < 20) to prevent infinite loops caused by data errors (cycles).

Dialect notes

WITH RECURSIVE is standard SQL:1999 and supported by PostgreSQL, MySQL 8.0+, SQL Server (using WITH … UNION ALL, no RECURSIVE keyword required), Snowflake, BigQuery, and Oracle 11g+. SQLite 3.8.3+ supports it. The termination condition syntax and LIMIT clause are consistent across these engines.

Senior signal: seed the recursive CTE with only the two target employees for efficiency; know that depth 0 = the node itself (enabling ancestor-of-the-other detection); explain that ORDER BY depth ASC LIMIT 1 picks the deepest common ancestor (not the root); and add a depth cap to defend against data cycles in production hierarchies.
