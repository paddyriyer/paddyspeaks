Netflix subscriptions move through a lifecycle — active → paused → cancelled → reactivated — with billing-cycle status events that don't always represent a real change. The query reconstructs the actual state periods from the noisy log: when did each status start, when did the next one begin, and which period is still open.

1. The source data — subscription_events

2. The SQL — LAG to detect change + LEAD for end_date

WITH changes AS (
  SELECT account_id, recorded_at, status,
    LAG(status) OVER (PARTITION BY account_id ORDER BY recorded_at) AS prev_status
  FROM subscription_events
),
filtered AS (
  SELECT account_id, recorded_at, status
  FROM changes
  WHERE status != prev_status OR prev_status IS NULL
)
SELECT account_id, status,
  recorded_at AS period_start,
  LEAD(recorded_at) OVER (PARTITION BY account_id ORDER BY recorded_at) AS period_end
FROM filtered
ORDER BY account_id, period_start;

3. Strategic explanations

Why "ping noise" is the Netflix-specific twist

Most state-machine logs at Netflix are append-only event streams that re-emit on every billing cycle, every region replication, every CDC retry. The same active status appears 12 times a year per account even when nothing changed. Period reconstruction requires deduplicating these pings before any analysis.

The lifecycle question pattern

The output of this query is the input to half of Netflix's growth-accounting questions:

  • "What's our churn-recovery rate?" — count active → cancelled → active sequences
  • "How long does the average pause last?" — AVG(period_end - period_start) WHERE status='paused'
  • "At any given moment, what's our DAU mix?" — find periods where period_start ≤ ts < period_end
The LAD-cousin warning

If the subscription_events stream has late-arriving events (offline mobile uploads), an event timestamped Jan 5 might land in the table on Feb 10. Re-running the query produces different periods than yesterday. Production guards: snapshot the input as of a watermark, OR sort by (recorded_at, ingest_seq) for deterministic tiebreak.

The "still open" period semantic

The most-recent status has period_end IS NULL — that's correct. The user is currently in that state; the period hasn't closed. Downstream queries that compute period duration should use COALESCE(period_end, CURRENT_DATE) for live duration.

The cross-account aggregate

"How many accounts are currently paused?" — filter to NULL period_end + status='paused':

SELECT COUNT(*) FROM (/* this query */) WHERE period_end IS NULL AND status = 'paused';
Senior signal: recognize billing-cycle ping noise as the Netflix-specific data-quality issue, dedup before period computation, treat NULL period_end as "still in this state", and connect the period output to growth-accounting questions (churn-recovery rate, pause duration). The LAD watermark caveat is the over-the-bar pre-empt.
