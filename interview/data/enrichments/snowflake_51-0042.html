Wildlife survey teams need a leaderboard: "At each observation site, which reptile species were spotted most often? Return the top 5." The pattern is classic ranked-aggregation — aggregate first to get per-(location, species) counts, then apply a window-function rank over the count within each location, then filter to the top-5 tier. The interviewer is checking whether the candidate knows that window functions cannot appear in a WHERE clause (hence the subquery), and whether they reach for RANK vs ROW_NUMBER deliberately.

2. The SQL — aggregate then rank, filter in outer query

SELECT location_id, species_id, common_name, observation_count
FROM (
  SELECT o.location_id, o.species_id, s.common_name,
         COUNT(*) AS observation_count,
         RANK() OVER (PARTITION BY o.location_id ORDER BY COUNT(*) DESC) AS rk
  FROM observations o
  JOIN species      s ON o.species_id = s.species_id
  WHERE s.class = 'Reptilia'
  GROUP BY o.location_id, o.species_id, s.common_name
)
WHERE rk <= 5
ORDER BY location_id, rk;

At L1: Nile Monitor has 3 observations (rk=1), Green Iguana has 2 (rk=2), Saltwater Crocodile has 1 (rk=3) — all ≤5, so all appear. At L2: Nile Monitor and Saltwater Crocodile both have 1 observation — they tie, both receive rk=1. All rk≤5 rows pass the outer filter.

4. Strategic explanations

Why the subquery is mandatory — window functions and WHERE

Window functions are computed after GROUP BY but before ORDER BY in the logical processing order. They cannot appear in a WHERE clause because WHERE filters before aggregation. The only standard way to filter on a window-function result is to push the window into a subquery (or CTE) and apply the filter in the outer query's WHERE. Snowflake offers the non-standard QUALIFY clause as a syntactic shortcut: QUALIFY RANK() OVER (...) <= 5 eliminates the subquery entirely, but it is Snowflake / BigQuery-specific and not portable to Postgres or MySQL.

RANK vs ROW_NUMBER vs DENSE_RANK

The choice between ranking functions is a frequent follow-up:

  • RANK: tied rows get the same rank; the next rank after a tie of two skips a number (1, 1, 3). "Top 5" means all tied-for-5th are included, and there may be more than 5 output rows per location.
  • DENSE_RANK: tied rows get the same rank; no gaps (1, 1, 2). "Top 5 tiers" semantics.
  • ROW_NUMBER: no ties — arbitrary tiebreak gives strictly distinct ranks. "Exactly 5 rows per location" but silently breaks ties arbitrarily, which can be misleading in a leaderboard context.

For a leaderboard, RANK is the fairest choice: tied species at position 5 both appear. Using ROW_NUMBER here would silently drop equally-observed species, which is a business-logic bug.

Pre-filter vs post-filter — WHERE class = 'Reptilia' before GROUP BY

The WHERE s.class = 'Reptilia' is in the inner query, applied before GROUP BY. This is correct: it reduces the number of rows the aggregation has to process. Placing it in the outer query's WHERE (after computing ranks for all classes) would produce wrong ranks — a non-reptile that dominated a location would push reptile ranks downward. Always filter the population before ranking it.

Performance considerations

The inner query must aggregate all qualifying observations before the window function can rank them. On large observation tables, an index on observations(location_id, species_id) accelerates the GROUP BY. The join to species for the class filter benefits from an index on species(species_id, class). Snowflake's micro-partition pruning helps when location_id is clustered.

Dialect portability
  • Snowflake: replace the subquery with QUALIFY rk <= 5 at the end of the inner query for cleaner syntax.
  • BigQuery: also supports QUALIFY.
  • Postgres / MySQL / SQL Server: require the subquery pattern shown.
Senior signal: explain why window functions cannot appear in WHERE (logical processing order), choose RANK over ROW_NUMBER deliberately for a leaderboard that must include all tied positions, apply the class filter before ranking to avoid rank displacement, and offer QUALIFY as the Snowflake/BigQuery shortcut that eliminates the subquery wrapper.
