Snap Map shows real-time location density. The query buckets activity by geohash + hour to produce the "heat map" data — the input to the visual map UI.

2. The SQL — geohash × hour density

SELECT geohash,
  CAST(strftime('%H', recorded_at) AS INTEGER) AS hour,
  COUNT(DISTINCT user_id) AS active_users
FROM map_activity GROUP BY geohash, hour
ORDER BY active_users DESC LIMIT 20;

3. Strategic explanations

Geohash — the spatial bucketing

Geohash encodes lat/long into a string of variable precision. 4-char prefix ≈ 20km×20km city. 6-char ≈ 1km. 8-char ≈ 50m. Bucketing at the right precision matches the use case.

The privacy guarantee — k-anonymity

Snap Map shows aggregated heat — never individual location. k-anonymity: don't show data unless ≥ k=20 users in the bucket. Add HAVING:

HAVING COUNT(DISTINCT user_id) >= 20
The temporal grain

Hour-level density vs 15-minute density. Finer = more responsive map; less = better k-anonymity. Production uses 15-minute internally, displays at hour-level.

Daypart-vs-event spike

Routine diurnal pattern: bucket density peaks 6-9pm. Spike density peaks at events (concerts, sports). Compare to baseline to identify "interesting" buckets.

The friend-presence filter

Snap Map respects friendship privacy: viewers see only buckets containing their friends. The query is the population-level view; production adds per-viewer joins.

Senior signal: geohash precision matches use-case grain, k-anonymity gate is non-negotiable, distinguish baseline diurnal from event-spike, and connect to friend-presence filtering for the user-facing version.
