A payroll team is running a quirky promotional payout: a "special bonus" goes only to employees whose ID is odd and whose name does not begin with a capital M. The interviewer is probing whether you can express a compound boolean condition in a single CASE, and — the real trap — whether you respect that the name test is case-sensitive, so a lowercase m does not disqualify anyone.

1. The source data — employees

employee_idnamesalary(narrative)
1Alice100odd id, no M — gets bonus
2Bob200even id — no bonus
3Maria300odd id but capital M — no bonus
4Mike400even id and capital M — no bonus
5Cara500odd id, no M — gets bonus
7Mira700odd id but capital M — no bonus
9mike900odd id, lowercase m — gets bonus

Row 9 is the whole interview: mike starts with a lowercase m. Because the rule is case-sensitive, m is not M, so this employee qualifies — even though "Mike" in row 4 would not. Note also the ID gap: there is no employee 6 or 8, which is fine — IDs need not be contiguous.

2. The SQL — a compound CASE predicate

SELECT employee_id,
       CASE WHEN employee_id % 2 = 1 AND name NOT LIKE 'M%'
            THEN salary ELSE 0 END AS bonus
FROM employees
ORDER BY employee_id;

employee_id % 2 = 1 tests oddness via the modulo remainder. name NOT LIKE 'M%' is true unless the name starts with a capital M — % matches any trailing characters. Joined by AND, both must hold for the THEN salary branch; otherwise ELSE 0 pays nothing.

Employee 9 receives 900: 'mike' LIKE 'M%' is false in case-sensitive PostgreSQL, so NOT LIKE 'M%' is true and the bonus flows. Employee 3 ("Maria"), despite an odd ID, gets 0 because the capital M trips the second condition.

4. Strategic explanations

The whole answer is one CASE with an AND

"If and only if (a) and (b)" maps directly to a single CASE WHEN cond_a AND cond_b THEN ... ELSE .... There is no need for nested CASEs or a join — both predicates are row-local. The ELSE 0 branch is mandatory: drop it and non-qualifying rows return NULL instead of 0, which the spec ("everyone else gets bonus = 0") explicitly forbids. ELSE here is correctness, not cosmetics.

LIKE is case-sensitive in PostgreSQL — that is the trap

PostgreSQL's LIKE is case-sensitive: 'mike' LIKE 'M%' is false. The spec says the name test is case-sensitive, so NOT LIKE 'M%' is exactly right and employee 9 qualifies. The mistake is to "helpfully" make it case-insensitive — e.g. ILIKE, UPPER(name) NOT LIKE 'M%', or LEFT(LOWER(name),1) <> 'm' — which would wrongly strip the bonus from "mike". Read the spec's case-sensitivity clause and honour it; do not normalise reflexively.

Modulo and negative or NULL IDs

employee_id % 2 = 1 cleanly identifies odd positives. Be aware of two edge cases: in PostgreSQL (-3) % 2 = -1, so a negative odd ID would fail = 1 — ABS(employee_id) % 2 = 1 or % 2 <> 0 is safer if negatives are possible. And a NULL employee_id makes the modulo NULL, the AND unknown, and the row falls to ELSE 0. The fixture has only positive non-NULL IDs, so % 2 = 1 is fine — but naming these cases shows depth.

NOT LIKE and NULL names

If name were NULL, name NOT LIKE 'M%' evaluates to unknown, the compound AND is unknown, and the employee lands in ELSE 0 — no bonus. That is a defensible default (an unknown name should not auto-qualify), but it is a silent decision. If a NULL name should still get the bonus when the ID is odd, you would need (name NOT LIKE 'M%' OR name IS NULL). Decide explicitly rather than letting three-valued logic choose for you.

Dialect — case-sensitivity is not universal

This query's correctness hinges on LIKE being case-sensitive. In SQL Server and MySQL the default collations are case-insensitive, so 'mike' LIKE 'M%' would be true there — and employee 9 would lose the bonus, contradicting the spec. To be portable you would force a binary/case-sensitive collation, e.g. SQL Server name NOT LIKE 'M%' COLLATE Latin1_General_BIN. PostgreSQL and Oracle are case-sensitive by default, so the plain form is correct here.

Senior signal: express "(a) and (b)" as one CASE WHEN ... AND ... THEN salary ELSE 0 END with a mandatory ELSE 0, honour the spec's case-sensitivity by using plain LIKE (never ILIKE/UPPER) so lowercase "mike" still qualifies, and flag that modulo on negatives, NULL inputs, and engine collation defaults would each change the result.
