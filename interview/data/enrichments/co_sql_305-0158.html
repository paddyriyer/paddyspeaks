X's payments team needs every order tagged for the operations queue: cancelled orders route to refunds, big-ticket orders get a high-value flag, everything else is standard. It is a single CASE expression, but the interviewer is watching one thing closely — do you understand that CASE stops at the first branch that matches, so the order of the WHENs is itself a business decision?

1. The source data — orders

Order 2 is the crux: it is cancelled and over $1000 — two branches could fire, and only the first one written wins. Order 5 sits exactly on the boundary at $1000 — an off-by-one test for the > operator.

2. The SQL — CASE WHEN … END

SELECT order_id,
  CASE WHEN status='cancelled' THEN 'Refund'
       WHEN amount > 1000 THEN 'High Value'
       ELSE 'Standard' END AS flag
FROM orders;

A searched CASE: each WHEN is an independent boolean test, evaluated top to bottom. The first one that is true supplies the result; if none match, ELSE applies.

Order 2 (cancelled, $2000) is flagged Refund, not High Value — the status='cancelled' branch comes first and short-circuits. Order 4 (cancelled, $50) is also Refund. Order 5 at exactly $1000 is Standard: amount > 1000 is strict, so the boundary value fails the test.

4. Strategic explanations

First-match-wins — clause order is a business rule

CASE is not "pick the best-fitting branch"; it is "take the first true branch and stop". Order 2 satisfies both status='cancelled' and amount > 1000, but because the cancelled test is written first, it never reaches High Value. Swap the two WHENs and order 2 becomes "High Value" instead — a different operational routing from the same data. Always confirm the priority with the interviewer: "if an order is both cancelled and high-value, which queue should it land in?"

The boundary — > vs >=

Order 5 at exactly $1000 is Standard because amount > 1000 excludes the endpoint. If "high value" means $1000 and above, the test must be amount >= 1000. This is a classic off-by-one: pin the definition down. "Is $1000 itself high-value?" is the kind of clarifying question that separates a careful engineer from one who guesses.

The NULL trap — an unmatched NULL falls through to ELSE

If status were NULL, status='cancelled' evaluates to unknown (not true), so that branch is skipped. If amount were NULL, amount > 1000 is also unknown. A row with both columns NULL silently lands in ELSE — "Standard". That may be wrong: a NULL status might mean "not yet processed", which deserves its own flag. Handle it explicitly rather than letting it default:

CASE WHEN status IS NULL THEN 'Needs Review'
     WHEN status='cancelled' THEN 'Refund'
     WHEN amount > 1000 THEN 'High Value'
     ELSE 'Standard' END
Always include ELSE

A CASE with no ELSE returns NULL for any unmatched row — an implicit ELSE NULL. That turns a "no rule matched" situation into a silent NULL that flows downstream. Writing ELSE 'Standard' (or ELSE 'Unclassified') makes the catch-all intentional and visible. An interviewer notices a missing ELSE.

Searched CASE vs simple CASE

This is a searched CASE — each WHEN is a full boolean expression, so the branches can test different columns (status here, amount there). The simple form, CASE status WHEN 'cancelled' THEN …, compares one expression against constants and cannot mix columns or use ranges. Because this problem needs both an equality check and a > comparison, the searched form is the only option. Knowing why is the senior detail.

Senior signal: state that CASE takes the first matching branch and that WHEN order therefore encodes a priority decision, pin down whether the threshold is > or >=, handle NULL inputs explicitly instead of letting them fall through to ELSE, and always write an explicit ELSE.
