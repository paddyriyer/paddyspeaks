A learning-platform analytics team wants each student's "best subject" — the single course where they earned their top grade — for a transcript highlights widget. The twist the interviewer is probing: a student can earn their top grade in two courses at once, so "the best course" is genuinely ambiguous until you commit to a deterministic tie-break. Picking exactly one row per group, predictably, is the whole exercise.

1. The source data — student_courses

Student 1 scored 95 in two courses; student 2 scored 80 in two. Both are ties at the top, and the rule says break them toward the smaller course_id. Student 3 has only one row — the degenerate single-member group that any correct query must still return.

2. The SQL — ROW_NUMBER with a tie-breaking ORDER BY

WITH ranked AS (
  SELECT student_id, course_id, grade,
         ROW_NUMBER() OVER (PARTITION BY student_id ORDER BY grade DESC, course_id ASC) AS rn
  FROM student_courses
)
SELECT student_id, course_id, grade
FROM ranked
WHERE rn = 1
ORDER BY student_id;

PARTITION BY student_id restarts the numbering for every student. The two-key ORDER BY grade DESC, course_id ASC sorts the highest grade first and, within an equal-grade tie, the smallest course_id first — so rn = 1 is always the intended row.

Course 103 and 202 vanish even though they share the top grade: ROW_NUMBER assigns exactly one 1 per partition, and the second sort key decides who gets it.

4. Strategic explanations

Why ROW_NUMBER and not RANK here

The requirement is "the course" — exactly one row per student. RANK() would give both 95-rows rank 1 and WHERE rk = 1 would return both, breaking the one-row contract. ROW_NUMBER is the deduplicating ranking function: it never repeats a number inside a partition, so it forces a single winner. Choose RANK only when the spec explicitly says "return all ties".

The tie-break must be in the ORDER BY, not left to chance

Drop , course_id ASC and the two 95-rows are sorted only by grade — a tie. Which one gets rn = 1 then depends on physical row order or the plan, and can change between runs or after a VACUUM. A deterministic query encodes every tie-break the spec names as an explicit sort key. Non-determinism in a transcript widget is a real bug, not a cosmetic one.

The filter belongs outside the window

You cannot write WHERE ROW_NUMBER() OVER (…) = 1 directly: window functions are evaluated after WHERE, so the alias does not yet exist at filter time. The CTE (or a subquery) materialises rn as an ordinary column, and the outer WHERE rn = 1 then filters it. This "rank in a CTE, filter outside" shape is the canonical top-N-per-group idiom.

NULL grades would corrupt the pick

If a grade were NULL, ORDER BY grade DESC sorts NULLs first in PostgreSQL (NULLs are largest under DESC unless you add NULLS LAST) — so an ungraded course could be crowned the student's "best". With clean integer grades the demo is safe, but on real data add NULLS LAST or filter WHERE grade IS NOT NULL and say so out loud.

Dialect — QUALIFY trims the boilerplate

Snowflake, BigQuery, and DuckDB support QUALIFY ROW_NUMBER() OVER (…) = 1, which filters on a window result without the wrapping CTE. PostgreSQL, MySQL, and SQL Server have no QUALIFY, so the CTE form shown here is the portable answer — mention QUALIFY as the terser equivalent where it exists.

Senior signal: reach for ROW_NUMBER — not RANK — when the spec wants exactly one row per group, encode every tie-break as an explicit secondary ORDER BY key so the result is deterministic, filter the rank in an outer query because windows run after WHERE, and flag that NULL sort positions could otherwise hijack the winner.
