Gmail's auto-categorization (Primary / Social / Promotions / Updates / Forums) is one of Google's most-used ML systems. The query is a confusion-matrix-style aggregation: predicted vs actual label, with row-percentages showing where the model confuses categories.

1. The source data — email_labels

message_idpredicted_labelactual_label
M_001primaryprimary
M_002promotionsupdates

2. The SQL — confusion matrix with row %

SELECT predicted_label, actual_label,
  COUNT(*) AS messages,
  ROUND(100.0 * COUNT(*)
              / SUM(COUNT(*)) OVER (PARTITION BY actual_label), 1) AS pct_of_actual
FROM email_labels
GROUP BY predicted_label, actual_label
ORDER BY actual_label, messages DESC;

3. Strategic explanations

Reading the confusion matrix

For each actual label, what % does the model predict each predicted label?

  • Diagonal cells (predicted = actual) = correct classifications. Want these high.
  • Off-diagonal cells = errors. Want these low.
  • The biggest off-diagonal cell is the model's most-confused pair — the prime improvement target.
Per-actual normalization — why PARTITION BY actual_label

The window-aggregate denominator partitions by actual_label. So each row's % is "of all messages whose true label is X, what % were predicted as Y?". This gives recall per row. Switching to PARTITION BY predicted_label gives precision instead.

Precision vs Recall — both views are needed
viewpartitiondiagonal interpretation
Recallby actual_label"of true X messages, % we caught"
Precisionby predicted_label"of predicted X messages, % were really X"

Production dashboards show both. F1 score (their harmonic mean) is the single-number summary.

The cost-asymmetry — false positives kill

Misclassifying a personal email as Promotions = user misses something important = delete-Gmail event. Misclassifying a Promotions email as Primary = user sees ads in main inbox = annoyance. The two errors have different costs. Production weights the confusion matrix by impact — so the algorithm minimizes cost-weighted error, not raw error.

The "user override" gold-standard

"Actual label" is itself a model output (user moves messages to fix). Two flavors:

  • User-confirmed (user moved to a label) = high-quality label
  • Auto-assigned without override = could be wrong without us knowing

Senior production splits the matrix by label-source for an honest evaluation.

The class-imbalance trap

If 90% of emails are Promotions, a naive "always predict Promotions" model is 90% accurate but useless. The confusion matrix exposes this — diagonal is dominated by one class, others are zeroed. Always weight by class balance.

The retraining cadence

Drift detection: monthly comparison of this matrix vs last month's. Cells that shift > 2pp trigger retraining. Spam senders adapt; the model has to keep up.

Senior signal: read the matrix as recall-per-row, distinguish recall vs precision via partition choice, weight the off-diagonal by cost asymmetry, and pin the class-imbalance gotcha. The label-source split for honest evaluation and retraining-on-drift cadence are the over-the-bar finishes.
