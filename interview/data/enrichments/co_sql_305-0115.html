AMD's product team wants a single number for a catalog-breadth slide: how many distinct categories does the product line span? It is the simplest possible aggregate — one COUNT(DISTINCT ...). The interview is whether you know exactly how DISTINCT inside COUNT treats NULL, duplicates, and case, because those three quietly change the number.

1. The source data — products

product_idcategorynameprice
1periphMouse20
2periphKeyboard50
3displayMonitor300
4audioHeadset80
5NULLMystery10

Five products, but only three category values — periph appears twice, and product 5 has a NULL category (an uncategorised SKU). The duplicate and the NULL are the whole interview: they decide whether the answer is 5, 4, or 3.

2. The SQL — COUNT(DISTINCT ...)

SELECT COUNT(DISTINCT category) AS distinct_categories FROM products;

DISTINCT inside the aggregate deduplicates the category values first; COUNT then counts what survives. No GROUP BY, so one row of output.

3. The resulting analysis

distinct_categories
3

The answer is 3: periph, display, audio. The duplicate periph counts once — that is DISTINCT doing its job. And the NULL row counts zero times: COUNT(column) ignores NULLs, distinct or not. Five rows, three categories.

4. Strategic explanations

The NULL is excluded — and that is the trap

The single most important fact: COUNT(DISTINCT category) does not count NULL. So this query answers "how many named categories exist" — not "how many distinct states does the category field take, including uncategorised". If the business considers "uncategorised" a real bucket, the count is understated by one. Make it explicit:

SELECT COUNT(DISTINCT category)
       + (CASE WHEN COUNT(*) FILTER (WHERE category IS NULL) > 0 THEN 1 ELSE 0 END)
       AS categories_incl_null
FROM products;

Naming this gap — rather than silently shipping 3 — is the senior move.

COUNT(DISTINCT col) vs COUNT(col) vs COUNT(*)

Three counts, three answers on this data: COUNT(*) = 5 (all rows), COUNT(category) = 4 (non-NULL values), COUNT(DISTINCT category) = 3 (distinct non-NULL values). They coincide only when the column has no NULLs and no duplicates. Reaching for the wrong one is a classic silent bug — the query runs, the number is just wrong.

The case-and-whitespace trap

DISTINCT compares values exactly. If the data holds 'Periph', 'periph', and 'periph ' (trailing space), those are three distinct values and the count inflates — the catalog looks broader than it is. Real-world category fields are rarely clean. Normalise before counting: COUNT(DISTINCT lower(trim(category))). Whether to do so depends on whether the source is trusted; flag the assumption either way.

The natural follow-up — products per category

"How many categories" almost always leads to "how big is each one":

SELECT category, COUNT(*) AS product_count
FROM products
GROUP BY category
ORDER BY product_count DESC;

Here GROUP BY category does emit a row for the NULL group (NULL is one grouping value) — a useful contrast: GROUP BY keeps NULL as a bucket, COUNT(DISTINCT) drops it. Same column, opposite NULL treatment.

Performance on large tables

COUNT(DISTINCT) must materialise and deduplicate every value — a sort or hash over the whole column, which is memory-hungry on a huge table. For an exact count there is no cheaper path. If an approximate count is acceptable for a dashboard, Postgres extensions offer HyperLogLog (hll); naming that trade-off — exactness vs speed — signals you have run this at scale.

Senior signal: state up front that COUNT(DISTINCT category) excludes NULL, so the result is "named categories" and may undercount if "uncategorised" is a real bucket; distinguish COUNT(*) / COUNT(col) / COUNT(DISTINCT col) precisely; and normalise case and whitespace before counting so dirty data does not inflate the breadth number.
