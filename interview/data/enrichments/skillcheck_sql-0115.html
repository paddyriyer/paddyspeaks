An HR system stores a manager hierarchy as an adjacency list — each employee row points at its direct manager. The task: list everyone who reports to employee 1, directly or through any chain of managers, however deep. The interviewer is checking whether you reach for a recursive CTE to walk a tree of unknown depth, and whether you correctly exclude a separate org subtree that does not roll up to employee 1.

The deliberate edge case is employees 9 and 10: a second, disconnected org tree. A correct recursive walk anchored at employee 1 must never reach them. Including them — e.g. by walking the whole table — is the classic failure this question is built to expose.

2. The SQL — recursive descent from employee 1

WITH RECURSIVE downstream AS (
  SELECT employee_id, employee_name FROM employees WHERE manager_id = 1
  UNION ALL
  SELECT e.employee_id, e.employee_name
  FROM employees e
  JOIN downstream d ON e.manager_id = d.employee_id
)
SELECT employee_id, employee_name
FROM downstream
ORDER BY employee_id;

The anchor selects employee 1's direct reports (manager_id = 1) — note it omits employee 1 itself, satisfying "excluding employee 1". The recursive arm joins the table back to the rows found so far: anyone whose manager is already in downstream joins in. Iteration repeats until no new rows appear.

4. Strategic explanations

Why a recursive CTE and not a fixed chain of joins

The prompt says "the hierarchy depth is unknown in advance." A self-join handles exactly one level; three self-joins handle three; but you cannot pre-write joins for an unbounded depth. A recursive CTE iterates until a fixed point — it walks however many levels exist. That is precisely the tool for transitive closure over an adjacency list.

How the anchor scopes the result to one subtree

The recursive arm only ever adds employees whose manager is already in downstream. So the entire result is determined by the anchor. Seeding with WHERE manager_id = 1 means the walk can only ever reach descendants of employee 1. Employees 9 and 10 form an island with no edge into that set, so they are unreachable. If the anchor had been WHERE manager_id IS NULL (all roots) the result would wrongly include the Outsider tree — the anchor predicate is the scope.

Excluding employee 1 itself — anchor choice, not a post-filter

The prompt says exclude employee 1. Two correct approaches: seed with manager_id = 1 (employee 1 is never inserted, as done here), or seed with employee_id = 1 and then add WHERE employee_id <> 1 to the final SELECT. The first is cleaner — the exclusion is structural. Picking the anchor to express the requirement directly is the senior move.

Termination and the cycle hazard

This CTE terminates because a valid org tree is acyclic — eventually no employee has an unvisited manager in the set, the recursive arm yields zero rows, and iteration stops. But adjacency-list data can be corrupt: if employee A managed B and B managed A, the walk would loop forever. Production-grade code defends against this — carry a path array and filter WHERE NOT e.employee_id = ANY(path), or use a depth counter with a sane cap. Mentioning the cycle risk unprompted is a strong signal.

Dialect notes

WITH RECURSIVE is standard — PostgreSQL, SQLite, MySQL 8+, and Oracle support it (Oracle also has the older CONNECT BY ... PRIOR syntax for the same job). SQL Server uses WITH without the RECURSIVE keyword and defaults to a 100-level recursion cap (OPTION (MAXRECURSION n) to raise it). For a hierarchy deeper than 100, that limit must be lifted explicitly on SQL Server.

Senior signal: walk an unbounded-depth hierarchy with a recursive CTE, recognise that the anchor predicate scopes the whole result so seeding at manager_id = 1 naturally excludes other org trees and employee 1 itself, and flag the cycle hazard in adjacency-list data — production code carries a visited-path guard.
