A merchandising analyst wants a Pareto (80/20) breakdown: rank products by sales, show the running cumulative percentage, and flag the minimal set of top products that together account for ~80% of revenue. The interviewer is probing window-function fluency — a running SUM over an ordered frame — and a subtle boundary definition: the flag uses each row's previous cumulative percentage, so the row that crosses into the 80% zone is included but nothing past it.

The edge case that decides the interview: product B lands the cumulative at exactly 80.00%. The flag's "previous cumulative strictly less than 80" rule must include B (its predecessor A was at 50%) and exclude C (its predecessor B was at exactly 80, not strictly less). The grand total is 1000.

2. The SQL — running percent + lagged boundary flag

WITH cum AS (
  SELECT product_id, name, sales,
         SUM(sales) OVER () AS grand_total,
         SUM(sales) OVER (ORDER BY sales DESC, product_id
                          ROWS UNBOUNDED PRECEDING) AS running
  FROM product
),
with_pct AS (
  SELECT product_id, name, sales, grand_total, running,
         ROUND((running / grand_total * 100)::numeric, 2) AS cumulative_pct,
         LAG(running, 1, 0) OVER (ORDER BY sales DESC, product_id) AS prev_running
  FROM cum
)
SELECT product_id, name, sales, cumulative_pct,
       (prev_running / grand_total * 100 < 80) AS in_pareto_80
FROM with_pct
ORDER BY sales DESC, product_id;

SUM(sales) OVER () with an empty window gives the grand total on every row. The second window, ordered and framed ROWS UNBOUNDED PRECEDING, gives the running total. LAG(running, 1, 0) fetches the prior row's running total (defaulting to 0 for the first row), and the boolean compares that to the 80% threshold.

4. Strategic explanations

Two windows, two jobs

SUM(sales) OVER () — no ORDER BY, no PARTITION — aggregates the entire result set and broadcasts the single grand total onto every row. SUM(sales) OVER (ORDER BY sales DESC, product_id ROWS UNBOUNDED PRECEDING) is a running sum: it accumulates from the first row up to the current one. Computing both in one CTE pass means dividing one by the other gives the cumulative percentage without a self-join or a correlated subquery.

Why the explicit ROWS UNBOUNDED PRECEDING frame matters

When a window has an ORDER BY but no explicit frame, the default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. RANGE includes all peer rows that tie on the ordering key in the current frame — so two products with identical sales would both jump to the same running total, double-counting within the tie. ROWS counts physical rows, so each row gets a distinct running total. The composite sort key sales DESC, product_id also breaks ties deterministically. Using ROWS (not RANGE) for a running total is a senior-level detail.

The lagged-boundary flag — reading the spec precisely

The naive flag is cumulative_pct <= 80, but that would exclude the row that pushes past 80 — you would never reach 80% coverage. The spec says: include a row if its previous cumulative was strictly below 80. LAG(running, 1, 0) fetches the predecessor's running total; the first row's predecessor defaults to 0, so the top seller is always in. The comparison prev_running / grand_total * 100 < 80 then includes exactly the rows up to and including the one that crosses the line, and nothing beyond. This is the difference between "rows already inside 80%" and "the minimal set that reaches 80%".

NULL and zero traps

If grand_total were 0 (no sales anywhere), every percentage division would raise division by zero. On a real table, guard with NULLIF(grand_total, 0). A NULL sales value would be skipped by SUM but would still produce a product row with a NULL sales and a carried-forward running total — decide whether NULL-sales products belong in the ranking at all, or filter them in a WHERE first. The LAG default of 0 is doing real work: without it the first row's prev_running would be NULL and the flag would evaluate to NULL, not true.

Dialect notes

Window functions, LAG with a default, and explicit ROWS framing are SQL-standard — portable to PostgreSQL, SQL Server, Oracle, BigQuery, Snowflake, and MySQL 8+. The ::numeric cast before ROUND is a Postgres requirement (ROUND(double, int) does not exist there); other engines round doubles directly. MySQL before 8.0 has no window functions at all — the running total would need a correlated subquery or a session-variable trick, and the whole query becomes far uglier.

Senior signal: compute a Pareto curve with two windows — SUM() OVER () for the grand total and a ROWS UNBOUNDED PRECEDING running SUM — use ROWS not the default RANGE so ties do not double-count, and implement a "minimal set reaching 80%" flag with LAG(running,1,0) on the predecessor rather than a naive cumulative threshold.
