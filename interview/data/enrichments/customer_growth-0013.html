The CEP-flavoured (Complex Event Processing) sibling of the velocity rules. Where customer_growth-0010 and customer_growth-0012 look for density of one event type, this question looks for an ordered pair: event A (address_change) followed by event B (high-value order) within a deadline. The senior signal is choosing KeyedProcessFunction + ValueState + event-time timer over a window primitive — and being able to say why: the two events are sparse, asymmetric, and the time bound is a deadline (a timer), not a window boundary.

1. The fraud pattern — what we're catching and what we're trying NOT to flag

This is the textbook Account Takeover (ATO) signature on a crypto-exchange or e-commerce platform. The attacker has obtained the victim's credentials (credential-stuffing, phishing, SIM swap). Step 1: change the shipping/withdrawal address quietly so the victim's normal email alerts go to the attacker's inbox or an email-spoofed lookalike. Step 2: place a large order or initiate a withdrawal before the victim notices.

PatternFraud signatureLegitimate pattern that looks similar
Address change → $5,000+ order within 1 hourAccount takeover — attacker rerouting goods to a drop address before the victim revokes sessionCustomer moved house, then placed their first big order at the new address — common with high-value items intentionally delivered to the new home
Wallet address whitelist change → 5 BTC withdrawal in 30 minCrypto-exchange ATO — attacker bypasses the 24h withdrawal hold by whitelisting a new address and racing the alertCustomer rotating hot wallets after a security audit — added a new cold-storage address, then sent funds
Phone number change → 2FA reset → password reset within 15 minSIM-swap takeover — attacker has hijacked the SMS number and resets credentialsCustomer lost their phone — legitimately resetting recovery after a device upgrade

Coinbase, in particular, enforces a 48-hour withdrawal cooldown after address whitelist changes precisely because this pattern is so common. The detector in this question is what produces the alert that triggers the cooldown — and the manual-review queue.

3. The query — walk through window mechanics

This question uses no window primitive. Window operators (TUMBLE/HOP/SESSION) assume the events being correlated are homogeneous and dense. Here they're heterogeneous and sparse — a customer might have 1 address change and 1 order in a week, with weeks of silence between them. The right primitive is keyed state:

class AddressChangeThenBigOrder(KeyedProcessFunction):

    def open(self, ctx):
        self.state = ctx.get_state(
            ValueStateDescriptor('addr_state', Types.PICKLED_BYTE_ARRAY()))

    def process_element(self, ev, ctx):
        st = self.state.value() or {'last_address': None, 'changed_at': None}

        if ev['type'] == 'address_change':
            # only mark changed_at when the address ACTUALLY differs
            if st['last_address'] and ev['address'] != st['last_address']:
                st['changed_at'] = ev['ts']
            st['last_address'] = ev['address']
            self.state.update(st)

        elif ev['type'] == 'order':
            ca = st.get('changed_at')
            if ca and (ev['ts'] - ca) <= WINDOW_MS \
                  and ev['amount'] >= HIGH_VALUE:
                yield { ... }  # ALERT

The deadline trick — event-time timers (production extension)

The shipped solution checks ev['ts'] - changed_at <= 30 min at order time. That works as long as some event eventually arrives for the customer. Edge case: if no order arrives within 30 min, the state's changed_at field sits there indefinitely, growing stale until the next event for that customer (could be days later). Cleanly handle it with an event-time timer:

def process_element(self, ev, ctx):
    ...
    if ev['type'] == 'address_change' and ... actually changed ...:
        st['changed_at'] = ev['ts']
        self.state.update(st)
        # Register timer at change_ts + 30 min; on fire, clear the alert state
        ctx.timer_service().register_event_time_timer(ev['ts'] + WINDOW_MS)

def on_timer(self, timestamp, ctx):
    st = self.state.value()
    if st and st['changed_at'] and (timestamp - st['changed_at']) >= WINDOW_MS:
        st['changed_at'] = None       # window expired → no longer primed
        self.state.update(st)

Event-time timers fire deterministically against the watermark, not wall-clock. Combined with checkpointing, the detector is fully reprocess-safe: replay yesterday's stream from offset 0 and you get the same alerts, byte-identical.

Why pandas isn't the right tool here

The pandas equivalent would be a self-join: address_changes table left-join orders table on customer_id with a between-ts predicate. That works for bounded backfill (the offline twin) but cannot run live — you'd need an open join over an unbounded stream, which is exactly what keyed state was invented for. The pandas version is fine for "did this alert fire in Tuesday's backfill?" — see the pattern in customer_growth-0010.

4. Why event-time matters — the L5 trap

The 30-minute window is a logical deadline. It has to be measured on event timestamps, not processing time, because the events flow through different paths:

ScenarioEvent-time answerProcessing-time answer
Address change at 14:05, order at 14:10, both arrive on timeAlert fires, gap = 5 minSame
Address change at 14:05 arrives at 14:30 (mobile offline); order at 14:10 arrived at 14:11Alert fires when the change event is processed — the order is already in state, and our handler can re-check. Or with timer: doesn't fire because order arrived before the change. Order matters: see Probe 4.Misses — when the change arrives at 14:30, processing-time says order was 19 min ago, which is fine, but the ordering is wrong relative to event-time
Replay yesterday's stream for backfillDeterministic — same alertsAll events have "now" as processing-time; window logic collapses
Auditor question: "When did the attacker change the address?"14:05 — event-time, defensible in a SAR filing14:30 — when our consumer saw the row; useless for forensics

Production rule: for any compliance-grade alert (SAR, OFAC, Travel Rule), the timestamp on the alert must be event-time. Processing time is a red flag in a regulatory exam.

5. Where this appears in production

  • Coinbase Trade Compliance — Coinbase's documented 48-hour withdrawal hold after address whitelist change. The detector is keyed-state-per-customer; the cooldown is enforced by a separate downstream service that consumes the alert topic.
  • Binance / Kraken — same pattern, different cooldown windows (24h to 7d depending on jurisdiction). All built on Flink + keyed state, with the trigger event being "withdrawal address whitelist change".
  • Stripe Radar — Account Takeover rules — Radar exposes customer.shipping_changed_recently as a rule attribute; merchants can chain it with amount > X in the rule editor.
  • Shopify Fraud Analysis — the "high risk" verdict on the order admin page is driven partly by shipping-address-versus-billing-address pattern detection, with a similar deadline structure.
  • Plaid Identity Match — same primitive (event correlation with a time deadline) used to detect synthetic-identity creation: an SSN-to-bank-link followed by a high-value ACH within 24h.

6. Interviewer probes

Probe 1 — "What if the order arrives before the address change (out-of-order arrival)?"

This is the hardest probe and the L5 separator. In the shipped solution, if the order is processed first, changed_at is still NULL and the alert misses. Production fixes: (a) buffer recent orders in state too — keep last_order_ts and last_order_amount, then when the change event arrives, look backward as well as forward; (b) use Flink's buffering within the bounded-out-of-orderness watermark so the operator sorts events by event-time within the slack window before processing. (a) handles arbitrary lateness; (b) handles only watermark-bounded lateness. Production crypto exchanges do both.

Probe 2 — "How do you handle the customer who legitimately moved house?"

Two layers. (a) Cohort-aware threshold: a long-tenured customer with a stable purchase history gets a higher HIGH_VALUE threshold than a 30-day-old account. Whale-customer baseline from customer_growth-0032. (b) Risk-score augmentation: the alert isn't an auto-decline — it's a feature that feeds a downstream LightGBM model along with device fingerprint, IP geo-distance from prior address, and amount-vs-baseline-spend. The model decides "review queue" vs "auto-block". The detector is a feature, not a decision.

Probe 3 — "Backfill — can you re-run this on yesterday's stream?"

Yes, with event-time timers and savepoints. Take a savepoint at 00:00 yesterday, restart against yesterday's Kafka offsets, run forward. The same address-change events fire the same timers at the same event-time, and the alerts emitted are byte-identical. This is why event-time timers, not processing-time timers — the latter would fire at wall-clock "now" during replay, producing wrong results.

Probe 4 — "What's the right primitive — KeyedProcessFunction or FlinkCEP?"

FlinkCEP's Pattern.begin("A").followedBy("B").within(...) directly expresses this rule. Trade-offs: CEP is more declarative but has higher per-event overhead (the NFA state machine), and debugging "why didn't this alert fire" is harder. KeyedProcessFunction is verbose but transparent — you can printf the state, you can add a 3rd attribute (phone change) by adding a dict field. Production teams choose KeyedProcessFunction for < 3-event patterns and CEP for > 3-event patterns or patterns with quantifiers ("3 failures then 1 success").

Probe 5 — "Hot key — one popular merchant's 'address book' service generates 100K address-change events/sec. What breaks?"

The merchant's customer_id (or shared service account) keys onto one task manager. Its RocksDB grows hot, checkpoint duration blows up. Standard fix: salting — but address changes need per-customer ordering (we care which address came first), so naive salting breaks correctness. The right fix here is data-model surgery: ensure the key is per-end-user customer, not a shared service account. If a single legitimate customer truly does 100K changes/sec, that's not a hot key — that's a bug in upstream and you should drop the duplicate-emit at ingest. See Hot Shards & Skew design page §03 on ordering-sensitive salting.

Probe 6 — "How would you express this in CEP for comparison?"
Pattern<Event, ?> pattern = Pattern.<Event>begin("change")
    .where(SimpleCondition.of(e -> e.getType().equals("address_change")
                                  && e.actuallyChanged()))
    .followedBy("order")
    .where(SimpleCondition.of(e -> e.getType().equals("order")
                                  && e.getAmount() >= 1000.0))
    .within(Time.minutes(30));

CEP.pattern(stream.keyBy(Event::getCustomerId), pattern)
   .select(matches -> new Alert(matches.get("change"), matches.get("order")));

One-liner clearer than the KeyedProcessFunction — but you've lost the explicit state, the deterministic "same address re-saved doesn't reset timer" rule (which would need a custom where with side-context), and ease of incremental extension.

Probe 7 — "What about exactly-once semantics on the alert output?"

Flink's keyed-state operator gives exactly-once on the state side. For the alert sink, use a Kafka producer in transactional mode (read-process-write with the same checkpoint), so an alert is committed iff the state update is committed. If the sink is a downstream service, make the consumer idempotent on (customer_id, order_id) — the alert's natural key — and treat duplicate alerts as no-ops.
