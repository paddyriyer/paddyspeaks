A growth team measures the e-commerce funnel: who actually moved visit → add-to-cart → purchase, in that order? The event table is a raw clickstream — events can arrive in any order, repeat, or be missing — so the question is not "did all three happen" but "did they happen in sequence". The interviewer is probing whether you compare the first timestamp of each step and enforce strict ordering, rather than just checking set membership.

The traps are users 2, 4, and 5. User 2 has every event but purchased before adding to cart. User 4 added to cart before visiting. User 5 visited again after purchasing — a duplicate event that must not break the funnel because we compare the earliest occurrence of each step.

2. The SQL — first-timestamp-per-step, then strict comparison

WITH firsts AS (
  SELECT user_id,
         MIN(ts) FILTER (WHERE event = 'visit')       AS first_visit,
         MIN(ts) FILTER (WHERE event = 'add_to_cart') AS first_cart,
         MIN(ts) FILTER (WHERE event = 'purchase')    AS first_purchase
  FROM event GROUP BY user_id
)
SELECT user_id FROM firsts
WHERE first_visit IS NOT NULL
  AND first_cart IS NOT NULL
  AND first_purchase IS NOT NULL
  AND first_visit < first_cart
  AND first_cart  < first_purchase
ORDER BY user_id;

The CTE pivots each user's events into one row with three timestamps — the earliest visit, cart, and purchase — using MIN(...) FILTER (WHERE ...). The outer query then demands all three exist and that they strictly increase: visit before cart before purchase.

4. Strategic explanations

Why MIN(ts) FILTER — the earliest occurrence is the funnel step

A user can fire visit many times; the funnel step "visit" is defined by when they first did it. MIN(ts) FILTER (WHERE event = 'visit') collapses all visit rows to that earliest moment. This conditional-aggregation pivot turns a tall event log into one wide row per user — exactly the shape needed to compare steps with simple < operators.

Strict < vs <= — the ordering semantics

The spec says each step is "strictly later" than the prior. first_visit < first_cart rejects a user whose first visit and first cart share the exact same timestamp. With <=, simultaneous events would pass — defensible if ties mean "same session", wrong if the funnel requires genuine progression. State which you mean: at second-granularity timestamps, ties are common and the choice is material.

The classic mistake — membership instead of ordering

A naive solution checks only that all three event types exist for the user (COUNT(DISTINCT event) = 3 or three EXISTS clauses). That passes user 2 and user 4 — they have every event — but their events are out of order. The funnel is fundamentally a sequence question; you must compare timestamps, not just confirm presence. This is the trap the question is built around.

NULL-safety — why the explicit IS NOT NULL guards

User 3 never purchased, so first_purchase is NULL. The comparison first_cart < first_purchase would evaluate to NULL (not FALSE), and a WHERE clause drops NULL rows — so user 3 is excluded even without the explicit guard. But stating first_purchase IS NOT NULL makes the "must have completed all steps" requirement visible and intentional, rather than relying on three-valued-logic side effects. Senior code documents its assumptions.

Dialect notes — FILTER is not universal

aggregate(...) FILTER (WHERE ...) is SQL-standard and supported by Postgres and SQLite 3.30+. SQL Server, MySQL, and older engines lack it — use MIN(CASE WHEN event = 'visit' THEN ts END), which is semantically identical (the CASE yields NULL for non-matching rows and MIN ignores NULL). The funnel logic is unchanged; only the conditional-aggregation syntax differs.

Senior signal: treat a funnel as a sequence not a set — pivot to the MIN(ts) of each step with FILTER, then enforce strictly increasing timestamps, choosing < vs <= for simultaneous events deliberately and guarding incomplete funnels with explicit IS NOT NULL checks.
