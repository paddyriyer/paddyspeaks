Funnel analysis is one of the most common product-analytics queries: which users started a journey and completed every step in order? The interviewer is probing whether you understand that "in order" means comparing the first occurrence of each step — not any occurrence — and that a user who viewed after carting, or carted after purchasing, should fail the funnel. The CTE-with-conditional-aggregation approach using MIN(ts) FILTER (WHERE …) is the clean, readable idiom; knowing its edge cases is what separates senior from mid-level candidates.

2. The SQL — MIN-per-event-type then strict timestamp ordering

WITH firsts AS (
  SELECT user_id,
         MIN(ts) FILTER (WHERE event_type = 'view')     AS first_view,
         MIN(ts) FILTER (WHERE event_type = 'cart')     AS first_cart,
         MIN(ts) FILTER (WHERE event_type = 'purchase') AS first_purchase
  FROM event
  GROUP BY user_id
)
SELECT user_id FROM firsts
WHERE first_view IS NOT NULL
  AND first_cart IS NOT NULL
  AND first_purchase IS NOT NULL
  AND first_view < first_cart
  AND first_cart < first_purchase
ORDER BY user_id;

The CTE collapses each user's event history into three timestamps using MIN(ts) FILTER (WHERE …). The outer query then applies three NULL guards (all three steps must have occurred) and two strict inequality checks to enforce the required temporal ordering.

4. Strategic explanations

MIN(ts) FILTER (WHERE event_type = …) — conditional aggregation in one pass

The FILTER clause is a PostgreSQL (SQL:2003) extension that applies a condition inside an aggregate without requiring a separate CASE WHEN. It is equivalent to MIN(CASE WHEN event_type = 'view' THEN ts END), which returns NULL when no matching row exists in the group. The IS NOT NULL guards in the outer query therefore serve double duty: they check that all three steps happened. This conditional-aggregation approach is a single group-by pass over the table, more efficient than joining three subqueries.

Why "earliest occurrence" is the right semantics

Users may view a product page multiple times before adding to cart. Using MIN captures intent from the first touchpoint, which is the most meaningful signal for funnel attribution. Using MAX would be wrong: a user who viewed, purchased, then viewed again would have MAX(view) > MAX(purchase), failing the funnel despite having completed it. The "earliest occurrence" semantics are standard in product analytics.

Strict vs non-strict inequality

The predicate uses strict <, so two steps at the exact same timestamp fail the funnel. Whether ties should pass is a business decision: a user who viewed and added to cart in the same database write (e.g. a synchronised import) might legitimately deserve credit. If same-instant is acceptable, change < to ≤. Most real schemas have sub-second timestamps that make this moot, but raising it in an interview shows thoroughness.

Alternative: CASE WHEN … THEN for non-Postgres engines

The FILTER clause is standard SQL:2003 but not universally implemented. MySQL 5.x and older SQL Server versions require: MIN(CASE WHEN event_type = 'view' THEN ts END). BigQuery and Snowflake both support FILTER in recent versions. The logic is identical; only syntax changes.

Alternative approach: EXISTS / self-join

An alternative is three correlated EXISTS subqueries or self-joins to find users where a view record exists before a cart record before a purchase record. This is more verbose and hits the event table multiple times. The CTE approach's single-pass aggregation is preferable in most engines.

Senior signal: use MIN(ts) FILTER (WHERE event_type = …) for a single-pass conditional aggregation; apply IS NOT NULL guards to handle users missing steps; use strict < and explicitly discuss whether simultaneous timestamps should qualify; call out the CASE WHEN fallback for non-Postgres engines.
