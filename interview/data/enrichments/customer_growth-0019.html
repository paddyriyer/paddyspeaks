Pinterest's growth team has the same problem every catalogue commerce site has: a user landed, browsed, didn't buy. Which of them are worth a remarketing dollar — the customer who deliberated on one product for a week and silently quit at checkout, or the customer who scrolled past 30 pins and never looked back? Weighted-event scoring on the clickstream is how Pinterest, Wayfair, ASOS, and Etsy decide; the bored half goes into a cheap email blast, the high-intent half gets a paid retargeting impression. Doing this with one threshold (e.g. "has add_to_cart") leaves 30 %+ of buyable intent on the table.

1. The data — realistic clickstream

One row per recorded interaction. In production this is a Kafka topic events.web.engagement fanned out to S3+Glue and a real-time Flink job.

The weights aren't pulled from a hat. The standard derivation: take a month of data, regress purchased_within_7d on each event-type count, and use the regression coefficients (or log-odds) as the weights. Re-fit quarterly because user behaviour drifts.

2. The business question — what the consumer of this metric decides on

  • Per-cohort marketing spend. Pinterest pays $0.40 CPM on the "considering" cohort (cheap broad emails) and $4.20 CPM on the "high_intent" cohort (Meta retargeting CPM). Misclassifying a high-intent user as bored costs ~$3.80 in foregone retargeting impressions; the inverse costs $3.80 in wasted CPM. Both errors matter, hence the three-band labelling.
  • On-site personalisation. The "high_intent" cohort sees a banner with a 10 % coupon at session start. The "bored" cohort sees the standard discovery feed. The model output is the routing key.
  • Conversion-rate forecasting. The finance team uses the count of high_intent users today as the leading indicator for purchases in the next 7 days — feeds the revenue forecast model. If high_intent count drops 20 % week-over-week, ring the alarm before revenue does.

3. The pandas implementation — annotated

def passive_intent(events, days=30):
    weights = {
        'view':         0.1,   'detail_view':  0.3,  'review_read': 0.5,
        'wishlist_add': 1.5,   'price_check':  1.5,
        'add_to_cart':  2.0,   'checkout':     4.0,
    }

    # ── Step 1. Window the data on event-time, not processing-time.
    # `events_ts.max() - 30d` gives the same cutoff regardless of WHEN
    # this notebook ran. Determinism matters for replay & A/B testing.
    df = events.copy()
    df['event_ts'] = pd.to_datetime(df['event_ts'])
    cutoff = df['event_ts'].max() - pd.Timedelta(days=days)
    recent = df[df['event_ts'] >= cutoff]

    # ── Step 2. Drop the buyers — they're not "passive" by definition.
    # set() comprehension is O(1) lookup per customer; .isin() with a
    # set is faster than .isin() with a list (pandas optimises this).
    buyers = set(recent.loc[recent['event_type'] == 'purchase',
                            'customer_id'].unique())
    passive = recent[~recent['customer_id'].isin(buyers)].copy()

    # ── Step 3. Vectorised event-type → weight map. .map() is the
    # right tool: dictionary lookup, runs in C, NaN for unknown types
    # which we fillna(0) — safer than KeyError.
    passive['weight'] = passive['event_type'].map(weights).fillna(0)

    # ── Step 4. The base aggregate per customer.
    # 'sessions' = nunique on event_ts (one session ≈ one unique ts
    # in the demo; in production sessionise by 30-min idle gap first).
    base = (passive.groupby('customer_id')
                   .agg(sessions=('event_ts','nunique'),
                        unique_products=('product_id','nunique'),
                        intent_score=('weight','sum'))
                   .reset_index())

    # ── Step 5. Revisit bonus. A customer with 3+ hits on the SAME
    # product is deliberating, not browsing. This is the orthogonal
    # signal that pure event-weight summing misses — a user could rack
    # up score=4.0 by viewing 40 different products (low intent) OR by
    # poring over one product 8 times (high intent). The revisit bonus
    # rewards the second pattern.
    revisits = (passive.groupby(['customer_id','product_id']).size()
                       .reset_index(name='hits'))
    revisit_n = (revisits[revisits['hits'] >= 3]
                 .groupby('customer_id').size()
                 .reset_index(name='revisits'))
    out = base.merge(revisit_n, on='customer_id', how='left')
    out['revisits'] = out['revisits'].fillna(0).astype(int)
    out['intent_score'] = (out['intent_score']
                           + (out['revisits'] > 0).astype(int) * 1.0).round(2)

    # ── Step 6. Three-band label. Thresholds (5.0, 2.0) are tunable
    # hyperparameters — re-fit quarterly via precision-recall analysis
    # against actual purchase outcomes in the following 7 days.
    def _label(s):
        if s >= 5.0: return 'high_intent'
        if s >= 2.0: return 'considering'
        return 'bored'
    out['label'] = out['intent_score'].apply(_label)
    return out.sort_values('intent_score', ascending=False).reset_index(drop=True)

Why .map() instead of .apply() for the weight lookup? .map() on a dict is a single C-level hash lookup per row. .apply(lambda x: weights[x]) is a Python call per row — ~50× slower on a 10 M-row event table. Same for the label assignment at the end: on a small per-customer frame the apply is fine; if it ever ran on raw event-level data it'd need to be re-cast as pd.cut.

5. Production patterns — how this scales

  • pandas → 1-2 GB clickstream before OOM. Pinterest's web event stream is > 1 B events/day. The production pipeline: Kafka → Flink for sessionisation + intent scoring keyed by user_id → DynamoDB serving table → real-time personalisation reads.
  • Wayfair publishes its scoring weights as Tobit-regression coefficients re-fit weekly. The same architecture: events to Kafka, Flink for stateful scoring, output to a feature store consumed by the personalisation team.
  • ASOS uses a near-identical "high intent / browsing / inactive" three-band split feeding email cadence. The exclusion of recent buyers is the same — a one-day post-purchase silence prevents the "you just bought this!" embarrassment.
  • Hot-user skew. Power-users with 50k events/day exist. The score sum is associative — same salt + two-pass aggregation as customer_growth-0027 applies. The revisit count is also associative (sum of per-salt partials). The only piece that doesn't salt cleanly is nunique(product_id); production uses HyperLogLog sketches keyed by user_id, merged across salts.
  • Why the buyer filter must happen before scoring, not after. If you score everyone then filter, you've wasted ~30 % of CPU. If a customer has 5 000 events and the 5 001st is a purchase, you don't want to score the first 5 000. In streaming this is an early-stop predicate on the keyed state.

6. Interviewer probes

Probe 1 — "Where do the weights come from? Are they magic numbers?"

No — they're regression coefficients. Take a labelled training set: X = per-event-type counts in the last 7 days, y = bought-or-not in days 8-14. Fit a logistic regression, take the coefficients (or e^coefficient — the odds ratio) as the weights. Re-fit quarterly because user behaviour drifts (TikTok-style discovery shortens the deliberation phase year-over-year). Treating the weights as static forever is a red flag.

Probe 2 — "What if a customer has 5 000 'view' events from a bot crawl? Score balloons to 500 — false high_intent."

Two defences. (a) Cap per-event-type contribution: weight × min(count, 50) caps a single event's impact. (b) Filter bot traffic upstream — Pinterest has a bot-detection model that tags events with is_bot=True on the way through the pipeline; the scoring function drops those before aggregation. In practice both defences are layered: the bot detector catches the obvious cases, the cap protects against the long tail of low-grade automation.

Probe 3 — "Why include 'sessions' and 'unique_products' if only intent_score drives the label?"

They're diagnostic columns for the analyst. The model only outputs the label; the columns let a human ask why. A customer with intent_score=5.1 from sessions=20 unique_products=20 is suspicious (probably a bot or a comparison-shopper) and merits a manual look. A customer with intent_score=5.1 from sessions=3 unique_products=1 is the classic high-intent signature. Operational visibility is the goal — the senior candidate names it.

Probe 4 — "How would you incrementalise this for streaming?"

Flink keyed state on customer_id. State holds (running_intent_score, per_product_hit_count, has_purchased). Each event updates the state in O(1). The has_purchased flag, once true, evicts the customer from the active set with a TTL-based clean-up. Sessionisation upstream uses Flink's SessionWindowAssigner with 30-min idle gap. The pandas function is the offline twin — same algorithm, batch instead of stream. See customer_growth-0010 for the pandas-twin pattern in detail.

Probe 5 — "The thresholds 5.0 and 2.0 — what's the rationale?"

Empirical. Take the historical score distribution, look at the cumulative purchase rate at each score quantile, pick the inflection points. Typical e-commerce: the top 10 % of passive users convert at 8-15× the bottom 60 % rate — that gap is where the high_intent threshold lives. The middle band is everything between "obviously not buying" and "obviously deliberating". Re-tune monthly via precision-recall trade-off against the marketing team's CAC ceiling.

Probe 6 — "What about unique_products as a divisor? Is 7 events on one product the same as 7 events on 7 products?"

No, and that's exactly what the revisit bonus encodes. Without the bonus, both sum to the same intent_score. With it, the single-product deliberator gets +1. The senior candidate suggests going further — per-product score concentration (Herfindahl index or just max_hits_per_product / total_hits) as a feature. A user whose 80 % of score sits on one product is structurally different from one whose score is uniform across 7 products.

Probe 7 — "How do you A/B test changes to this scoring function?"

Holdout cohort: randomly assign 5 % of new visitors to a control bucket where the OLD model decides their retargeting label. Measure 14-day purchase rate in treatment vs control. If the new model shifts 10 % of the cohort from "considering" to "high_intent" but the high_intent retargeting CTR drops by > 20 %, the new threshold is too loose. Always tune intent-scoring with a downstream metric in the loop, not in isolation — the scoring function's only purpose is to feed a marketing decision.
