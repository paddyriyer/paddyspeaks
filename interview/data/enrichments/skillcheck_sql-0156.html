A product team wants to chop a raw event stream into sessions: a user's clicks belong to the same session until they go quiet for more than an hour, after which the next click starts a fresh session. The interviewer is probing the classic gaps-and-islands pattern — flag the gaps with LAG, turn the flags into a session id with a running sum, then aggregate per session.

The edge cases: the 75-minute gap for user 1 must cut a new session (just over the threshold), user 2's first session has a single event, and the very first event of each user has no predecessor — LAG returns NULL there and that must count as a session start.

2. The SQL — flag gaps with LAG, cumulative-sum into session ids

WITH flagged AS (
  SELECT user_id, ts,
         CASE WHEN ts - LAG(ts) OVER (PARTITION BY user_id ORDER BY ts) > INTERVAL '60 minutes'
              OR LAG(ts) OVER (PARTITION BY user_id ORDER BY ts) IS NULL
              THEN 1 ELSE 0 END AS is_new
  FROM event
),
numbered AS (
  SELECT user_id, ts,
         SUM(is_new) OVER (PARTITION BY user_id ORDER BY ts) AS session_id
  FROM flagged
)
SELECT user_id, session_id,
       MIN(ts) AS session_start,
       MAX(ts) AS session_end,
       COUNT(*) AS event_count
FROM numbered
GROUP BY user_id, session_id
ORDER BY user_id, session_id;

Step 1 (flagged): LAG fetches the previous event's timestamp; the CASE marks is_new = 1 when the gap exceeds 60 minutes or there is no previous event. Step 2 (numbered): a running SUM of those flags becomes the 1-based session id — it increments only at each new-session marker. Step 3 groups by (user_id, session_id) for the per-session stats.

4. Strategic explanations

Gaps and islands — the three-step recipe

This is the canonical sessionization pattern. (1) Use LAG to compare each row to its predecessor and emit a boundary flag. (2) A running SUM of that 0/1 flag produces a group id that stays constant within a session and bumps at each boundary — the flags act as "tick marks". (3) GROUP BY that id to collapse each island to one row. The same recipe handles consecutive-streaks, price-regime changes, and downtime windows — only the boundary condition changes.

Why the running SUM produces a clean session id

is_new is 1 exactly at each session's first event and 0 elsewhere. A cumulative SUM over rows in time order therefore reads 1,1,1,2,2 for user 1: it holds steady through a session and steps up by one at each new-session flag. Because the flag is exactly 1 (never 2+), the id increases by exactly one per session — that is what makes it a clean 1-based ordinal. PARTITION BY user_id resets the sum per user, so session ids restart at 1 for user 2.

The first-event NULL is a genuine session start

LAG returns NULL for each user's earliest event — there is no prior row. The gap test ts - LAG(ts) > INTERVAL '60 minutes' against NULL evaluates to unknown, which is not true, so on its own it would flag is_new = 0 and the first event would wrongly inherit session 0. The explicit OR LAG(ts) IS NULL branch fixes that: the first event is always a session start. Forgetting this clause is the most common sessionization bug — the running sum then starts at 0 and every session id is off by one.

The boundary is strict — "more than 60 minutes"

The spec says a new session begins after a gap of more than 60 minutes, so the test is > INTERVAL '60 minutes', strict. A gap of exactly 60 minutes stays in the same session. User 1's 75-minute gap clearly exceeds it; if a gap were exactly 60, >= would wrongly split. Match the inequality to the wording precisely — "after a gap of more than 60" is strict, "60 or more" would be >=.

Interval arithmetic, ties, and dialect

ts - LAG(ts) on two timestamps yields an interval, compared directly to INTERVAL '60 minutes' — clean in Postgres. Other engines differ: MySQL uses TIMESTAMPDIFF(MINUTE, prev, ts) > 60, SQL Server DATEDIFF(MINUTE, prev, ts) > 60, BigQuery TIMESTAMP_DIFF(ts, prev, MINUTE). One unstated risk: if two events share an identical ts, the window ORDER BY ts is non-deterministic — add a unique tie-break (ORDER BY ts, event_id) so the gap calculation and session assignment are stable.

Senior signal: sessionize with the gaps-and-islands recipe — LAG to flag boundaries, a running SUM of the 0/1 flag to mint the session id, then GROUP BY — and never forget the LAG(ts) IS NULL branch so each user's first event counts as a session start; without it every session id is off by one.
