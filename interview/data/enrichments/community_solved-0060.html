Ruby, a teacher, keeps a SeatArrangement table of students and their seat ids, and wants to swap every pair of adjacent seats — seat 1 with 2, seat 3 with 4, and so on. This is LeetCode's "Exchange Seats" in classroom dress, and the same pattern as any "flip pairs of rows" task. The interviewer is checking whether you can express the swap with arithmetic on the id, and how you handle a class with an odd headcount.

1. The source data — seat_arrangement

idstudent_name(narrative)
1Abbotodd — pairs with seat 2
2Doriseven — pairs with seat 1
3Emersonodd — pairs with seat 4
4Greeneven — pairs with seat 3
5Jeamesodd — no seat 6, stays put

There are five students — an odd count. Seats 1↔2 and 3↔4 swap cleanly, but seat 5 has no partner. The unpaired last student must keep their seat, not vanish or land on a non-existent seat 6.

2. The SQL — parity arithmetic via UNION ALL

with union_cte as(
select *, (id-1) as nw_id from seat_arrangement
where id%2 = 0

union all

select *, case when (id+1) in (select id from seat_arrangement) then (id+1)
        else id end as nw_id
from seat_arrangement
where id%2 <> 0
)
select nw_id as id, student_name
from union_cte;

Even seats move down one (id - 1); odd seats move up one (id + 1) — but only if that seat exists, otherwise the odd seat keeps its own id. The two branches are stacked with UNION ALL.

3. The resulting analysis

Seats 1–4 are cleanly exchanged; Jeames stays in seat 5 because the CASE checked that seat 6 does not exist and fell through to else id. Sort by id for a tidy seating chart.

4. Strategic explanations

Why the parity arithmetic works

Adjacent swapping means: in each pair, the even-id seat takes the odd id and vice versa. An even id n always pairs with n−1; an odd id n with n+1. So computing a new id of id - 1 for even rows and id + 1 for odd rows performs the swap — the student_name rides along unchanged, only the id label is rewritten. UNION ALL recombines the two halves.

Why the odd-count guard is essential

With an odd number of students the highest odd id has no n+1 partner. Blindly setting nw_id = id + 1 would send Jeames to a phantom seat 6. The subquery WHEN (id+1) IN (SELECT id FROM seat_arrangement) tests whether the partner seat actually exists; if not, ELSE id keeps the student in place. This single CASE is the difference between a correct answer and an off-by-one bug.

The single-pass alternative — one CASE, no UNION

The whole swap fits in one expression, avoiding the two-branch scan:

SELECT
  CASE WHEN id % 2 = 1 AND id = (SELECT MAX(id) FROM seat_arrangement) THEN id
       WHEN id % 2 = 1 THEN id + 1
       ELSE id - 1 END AS id,
  student_name
FROM seat_arrangement
ORDER BY id;

It reads the table once and uses MAX(id) to spot the lone trailing odd seat. A modern variant swaps the name with LEAD/LAG instead of relabelling ids — cleaner still when seat ids must stay fixed and only occupants move.

The contiguous-id assumption

The arithmetic assumes id is a gap-free 1, 2, 3, … sequence — the problem states "id is continuous increment", so it holds here. If ids had gaps (a removed student leaving seat 3 empty), id ± 1 would point at the wrong or a missing seat. The robust generalisation pairs rows by position: ROW_NUMBER() to get a dense 1..n rank, swap on that rank, then map back. State this assumption explicitly — interviewers probe what happens when a seat is vacated.

Senior signal: express the swap as parity arithmetic (even → id−1, odd → id+1), and immediately call out the odd-count edge — guard the last unpaired seat with an existence check or MAX(id) test. Note the solution assumes gap-free ids; with gaps, pair on ROW_NUMBER() instead.
