Slack's customer-success team wants to spotlight "power users" — the people sending the most messages this year — for a case-study program. The ask sounds like a leaderboard, and the SQL is short. But "top 10" hides a tie problem, and the date filter hides a timezone problem, and an interviewer asking this wants both said out loud.

Two things to watch. User 3 has a message dated 2023-12-30 — just outside the window — so the filter must drop exactly that row. And users 2, 3, and 5 each land on the same count: with limited leaderboard slots, ties decide who makes the cut.

2. The SQL — COUNT + ORDER BY + LIMIT

SELECT user_id, COUNT(*) AS msgs FROM messages
WHERE sent_at >= '2024-01-01'
GROUP BY user_id ORDER BY msgs DESC LIMIT 10;

Filter to messages from 2024 onward, count per user, sort by count descending, keep the top 10.

User 1 is the clear leader at 4. User 3 counts 2, not 3 — the 2023-12-30 message was filtered out, exactly as intended. Users 2, 3, and 5 are all tied at 2, and the order they appear in (3, 5, 2 here) is arbitrary: nothing in ORDER BY msgs DESC breaks that tie, so the engine returns them in whatever order is convenient. On a table large enough for the tie to straddle the LIMIT 10 boundary, that arbitrariness becomes a correctness bug.

4. Strategic explanations

The tie at the LIMIT boundary — the silent bug

LIMIT 10 with an unstable sort means: if users ranked 10th and 11th have the same count, which one gets the slot is undefined, and can flip between runs or after a planner change. Two fixes. Add a deterministic tiebreaker so the result is at least stable and reproducible:

SELECT user_id, COUNT(*) AS msgs FROM messages
WHERE sent_at >= '2024-01-01'
GROUP BY user_id
ORDER BY msgs DESC, user_id ASC   -- stable: ties broken by user_id
LIMIT 10;

Or, if the business rule is "include everyone tied for 10th", drop LIMIT and use a window function:

SELECT user_id, msgs FROM (
  SELECT user_id, COUNT(*) AS msgs,
         RANK() OVER (ORDER BY COUNT(*) DESC) AS rk
  FROM messages WHERE sent_at >= '2024-01-01'
  GROUP BY user_id
) t WHERE rk <= 10;

The senior move is asking which the team wants: "exactly 10 rows" or "the top-10 ranks, ties included".

The date filter — '2024-01-01' is a timestamp at midnight

sent_at >= '2024-01-01' compares against 2024-01-01 00:00:00. That correctly includes everything on or after New Year. The trap is the upper bound if the spec is "this year": sent_at <= '2024-12-31' silently excludes everything after midnight on the 31st — almost a full day of messages. The robust pattern is a half-open interval: sent_at >= '2024-01-01' AND sent_at < '2025-01-01'. Half-open ranges never miss the last day and never double-count the boundary.

Timezones — whose midnight?

If sent_at is timestamptz, the literal '2024-01-01' is interpreted in the session timezone. A message sent at 11pm Pacific on Dec 31 is already Jan 1 in UTC. For a global product like Slack, "this year" is genuinely ambiguous — confirm whether the boundary is UTC or a business timezone, and if needed convert explicitly with sent_at AT TIME ZONE 'America/Los_Angeles'.

COUNT(*) vs counting distinct days

COUNT(*) counts message rows. "Power user" might instead mean active days — someone who sends one message every day is arguably more engaged than someone who dumps 50 in an hour. That is COUNT(DISTINCT sent_at::date). Different metric, different ranking; worth a clarifying question before committing to raw volume.

Senior signal: spot that ORDER BY msgs DESC LIMIT 10 has an undefined tie-break at the cut line and either add a deterministic key or switch to RANK(), prefer a half-open date range so the last day is never lost, and raise the timezone interpretation of "this year" before writing the filter.
