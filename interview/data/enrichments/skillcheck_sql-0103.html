A trust-and-safety team moderating streaming sessions wants to flag accounts for banning when the same account was logged in from two different IP addresses at the same time — concurrent access from two locations. Each session has a login and logout timestamp. The interviewer hints at the trap: a window function that only looks at the previous row misses interleaved sessions, so this needs a self-join over all pairs.

Every failure mode is represented. Account 1 must be banned (a third-IP session overlaps the others). Account 2 has two different IPs but the sessions are disjoint in time — no ban. Account 3 has overlapping sessions but the same IP — no ban. Account 4 is a clean ban: two IPs, overlapping windows.

2. The SQL — self-join over all session pairs

SELECT DISTINCT l1.account_id
FROM log_info l1
JOIN log_info l2
  ON l1.account_id = l2.account_id
 AND l1.ip_addr <> l2.ip_addr
 AND l1.login_ts BETWEEN l2.login_ts AND l2.logout_ts
ORDER BY l1.account_id;

The table is joined to itself on the same account, requiring a different IP, and the overlap is detected by testing whether one session's login_ts falls inside the other session's window. DISTINCT collapses an account that has many offending pairs to a single flagged row.

4. Strategic explanations

Why a self-join, not LAG — the hinted trap

A window-function approach sorts each account's sessions and compares each to the previous one with LAG. That only checks adjacent sessions. With three interleaved sessions — exactly account 1's situation — the offending pair may not be adjacent in sort order, and LAG would miss it. A self-join compares every pair of sessions for an account, so no overlapping pair can slip through regardless of ordering. When the question is "do any two rows in this group relate?", pairwise self-join is the robust tool.

The overlap test — what BETWEEN is really checking

l1.login_ts BETWEEN l2.login_ts AND l2.logout_ts asks: does session l1 start while session l2 is still active? If so, the two sessions were live simultaneously. The fully general interval-overlap predicate is l1.login_ts < l2.logout_ts AND l2.login_ts < l1.login_ts... more precisely a.start < b.end AND b.start < a.end. The BETWEEN form here catches every overlap because the join is symmetric: each unordered pair is examined in both directions, so if l1 starts inside l2 or l2 starts inside l1, one of the two join rows fires.

The three join conditions, each a distinct rule

l1.account_id = l2.account_id scopes the comparison to one account — you never compare sessions of different users. l1.ip_addr <> l2.ip_addr encodes "different locations" — this is what spares account 3, whose overlapping sessions share an IP. l1.login_ts BETWEEN ... encodes the time overlap. Drop any one and the rule is wrong: without the IP clause you would ban account 3; without the time clause you would ban account 2.

Why a row joins itself — and why it does not matter here

A naive self-join lets a row match itself (l1 = l2). That self-match is harmless here: l1.ip_addr <> l2.ip_addr is false when a row meets itself (an IP equals itself), so the row is filtered out automatically. Had the predicate been only on time, you would need an explicit l1.ctid <> l2.ctid or an id tiebreaker to stop a session "overlapping itself". The <> on IP doubles as the self-match guard.

DISTINCT because an account can offend many times

Account 1's third-IP session overlaps both 10.0.0.1 sessions, so the join yields multiple offending rows for account 1. Without DISTINCT the result would list account 1 twice. The deliverable is the set of flagged accounts, so SELECT DISTINCT account_id — or equivalently GROUP BY account_id — reduces it to one row per account. Reporting accounts, not offending pairs, is the requirement.

Senior signal: when the rule is "do any two rows in a group overlap", use a pairwise self-join — a LAG-style previous-row check misses non-adjacent interleaved sessions; put each business rule (same account, different IP, time overlap) in its own join predicate, and DISTINCT the result because one account can match many offending pairs.
