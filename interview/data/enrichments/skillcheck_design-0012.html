The late-arriving dimension problem occurs when a fact event references an entity that has not yet been registered in the source system — a product mentioned in an order before the product master data arrives. The warehouse must not reject or lose the fact, so it creates a placeholder dimension row. The interviewer is probing whether you can identify the columns needed to track, reconcile, and audit that placeholder: the is_placeholder flag, the timestamps for when the placeholder was created and when it was resolved, and the “Unknown” row for genuinely missing NKs.

1. The requirements — what the design must capture

  • Placeholder mechanics: when a fact references a product_nk that does not exist in dim_product, a minimal row is inserted with is_placeholder = TRUE so the fact’s FK is satisfied immediately.
  • Reconciliation: when the real product master data eventually arrives, the placeholder row’s attributes are backfilled and reconciled_at / reconciled_by are set — full audit trail of resolution.
  • Validity window: SCD2 mechanics (effective_from / effective_to / is_current) for products whose attributes change over time after reconciliation.
  • Reserved unknown row: a row with SK = −1 and NK = ‘__UNKNOWN__’ for facts whose NK is genuinely absent (not late-arriving, truly missing) — avoids NULLable FKs on the fact table.
  • Composite unique constraint: (product_nk, effective_from) prevents duplicate placeholder rows for the same natural key at the same time.

2. The DDL — the CREATE TABLE

CREATE TABLE dim_product (
  product_sk        BIGINT       NOT NULL,                  -- surrogate PK
  product_nk        VARCHAR(64)  NOT NULL,                  -- natural key from source
  product_name      VARCHAR(200),
  category          VARCHAR(100),
  brand             VARCHAR(100),
  price             DECIMAL(10,2),
  -- placeholder mechanics
  is_placeholder    BOOLEAN      NOT NULL DEFAULT FALSE,    -- TRUE when fact arrived before dim
  placeholder_seen_at TIMESTAMP,                            -- first time a fact referenced this NK
  reconciled_at     TIMESTAMP,                              -- when real source row backfilled the attributes
  reconciled_by     VARCHAR(64),                            -- pipeline run id
  -- audit
  effective_from    TIMESTAMP NOT NULL,
  effective_to      TIMESTAMP,                              -- NULL for current
  is_current        BOOLEAN   NOT NULL DEFAULT TRUE,
  ingested_at       TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (product_sk),
  UNIQUE (product_nk, effective_from)
);

-- A reserved 'Unknown' row for facts whose NK is genuinely absent.
INSERT INTO dim_product (product_sk, product_nk, product_name, is_placeholder, effective_from, is_current)
VALUES (-1, '__UNKNOWN__', 'Unknown Product', TRUE, '1970-01-01', TRUE);

4. Strategic explanations

The late-arriving dimension problem and the two options

When a fact row arrives with a product_nk that is not in dim_product, the ETL has two choices: (1) hold the fact in a staging queue and wait for the dimension to arrive — simple but introduces latency and queue management complexity; (2) insert a placeholder dimension row immediately and let the fact flow into the warehouse — complex but preserves fact timeliness. For high-volume streaming pipelines where dimension lag is routine (new products added intra-day), option 2 is standard. The is_placeholder column is the design mechanism that makes option 2 safe for downstream consumers.

SK stability during reconciliation

The placeholder row keeps its original surrogate key when reconciled. Fact rows already loaded are referencing that SK — updating the SK would break all historical fact joins. Reconciliation is an UPDATE to the descriptor columns (product_name, category, brand, price, is_placeholder, reconciled_at, reconciled_by) on the existing row, not an INSERT of a new row. If the reconciliation triggers a meaningful attribute change (e.g., the product name on the source differs from what was guessed), an SCD2 new version may then be created — but that new version gets a new SK, and future facts reference it while historical facts still reference the original placeholder SK.

The “Unknown” row (SK = −1)

The reserved row handles facts whose product_nk is genuinely absent at the source — not late-arriving but missing entirely (corrupt feed, deleted product, referential integrity violation). Rather than NULLing the FK (which breaks INNER JOIN queries) or discarding the fact (losing data), the fact is assigned product_sk = −1. Analysts see “Unknown Product” in reports, which is explicit about data quality rather than silently excluding the fact. The convention of using negative SKs for special members (Unknown, N/A, Not Applicable) is a Kimball best practice.

Monitoring placeholder health

Operational dashboards should track: (1) count of rows where is_placeholder = TRUE AND reconciled_at IS NULL — the current backlog; (2) NOW() - placeholder_seen_at for unreconciled rows — the age of outstanding placeholders; (3) distribution of reconciled_at - placeholder_seen_at — the typical dimension lag. Alerting on backlog size or lag SLA breaches catches cases where master data feeds have silently stopped.

Warehouse dialect notes

In BigQuery, the UNIQUE constraint is not enforced; use a scheduled data-quality assertion to detect duplicates. In Snowflake, the UNIQUE constraint exists but is also not enforced by default — use it as documentation and add a dbt test. In PostgreSQL, the UNIQUE constraint is enforced and enables the UPSERT pattern (ON CONFLICT (product_nk, effective_from) DO UPDATE SET …) for idempotent reconciliation loads.

Senior signal: explain the two options for handling late-arriving dimensions (hold vs. placeholder) and justify the placeholder approach for streaming pipelines; show that the placeholder’s SK must remain stable through reconciliation to preserve historical fact accuracy; defend the “Unknown” row at SK = −1 as the Kimball-standard mechanism for genuinely missing NKs (vs. late-arriving NKs, which get a real placeholder); and describe the operational monitoring strategy using placeholder_seen_at and reconciled_at — these points together show you understand the full lifecycle of a dimension row, not just its birth.
