Salesforce’s market intelligence team wants to filter the startup portfolio to companies that have raised at least $1M in total funding. The query demonstrates the fundamental HAVING vs WHERE distinction: you cannot filter on an aggregate in a WHERE clause because the aggregate does not exist yet at that stage of query execution. The correct tool is HAVING, which filters after grouping and aggregation are complete.

2. The SQL — JOIN + GROUP BY + HAVING threshold filter

SELECT c.company_id, c.name,
       SUM(f.amount) AS total_raised,
       COUNT(*)      AS rounds
FROM companies      c
JOIN funding_rounds f ON c.company_id = f.company_id
GROUP BY c.company_id, c.name
HAVING SUM(f.amount) >= 1000000
ORDER BY total_raised DESC;

4. Strategic explanations

HAVING vs WHERE — the execution order distinction

SQL logical execution order: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. WHERE filters individual rows before grouping — the aggregate SUM(f.amount) does not exist yet. HAVING filters groups after aggregation — SUM(f.amount) is fully computed and can be compared to the threshold. Writing WHERE SUM(f.amount) >= 1000000 is a syntax error in all major engines.

HAVING SUM(...) vs subquery filter

An alternative approach: compute the sum in a subquery and filter in the outer WHERE:

SELECT company_id, name, total_raised, rounds
FROM (
  SELECT c.company_id, c.name,
         SUM(f.amount) AS total_raised,
         COUNT(*) AS rounds
  FROM companies c JOIN funding_rounds f USING (company_id)
  GROUP BY c.company_id, c.name
) agg
WHERE total_raised >= 1000000
ORDER BY total_raised DESC;

This produces identical results. The HAVING form is more concise; the subquery form lets you reference the alias total_raised in the outer WHERE (Snowflake also allows HAVING total_raised >= 1000000 with the alias directly, as does Postgres since the alias is resolved before HAVING in practice — though standard SQL requires repeating the expression).

Why INNER JOIN drops Company 4 correctly

Delta Systems (company 4) does have a funding round (round 107, $200 000). The JOIN matches it, it gets included in aggregation, and HAVING excludes it because 200 000 < 1 000 000. If a company had no funding rounds at all, an INNER JOIN would silently drop it. Whether that is correct depends on the business rule: "only funded companies appear in this report" — here it is correct. If you wanted unfunded companies to show up with total_raised=0, you would use a LEFT JOIN and COALESCE(SUM(f.amount), 0).

COUNT(*) vs COUNT(f.round_id)

COUNT(*) counts all rows in the group, including those where any column might be NULL. Because this is an INNER JOIN, every row in the group has a non-NULL round_id, so COUNT(*) and COUNT(f.round_id) are equivalent here. With a LEFT JOIN, COUNT(*) would count the NULL-padded row for an unfunded company as 1, while COUNT(f.round_id) would correctly return 0 (COUNT ignores NULLs). A senior candidate knows this distinction and picks the right one for the join type.

Dialect notes

HAVING with aggregates is ANSI SQL and universal. The ::numeric cast in the solution (not present in the SELECT here) is Postgres/Snowflake shorthand for CAST(... AS NUMERIC). NUMERIC(14,2) gives two decimal places of precision, matching typical currency requirements. BigQuery, MySQL, and SQL Server handle the HAVING pattern identically.
