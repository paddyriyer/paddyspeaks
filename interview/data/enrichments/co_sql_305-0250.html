The CRO's Monday morning email: "Send me the top seller in each region for last quarter — they're getting a personal call from the CEO." The data team has billions of rows of sales detail. The query needs to roll the rows up per (region × seller), then rank within region, then keep only the top.

Multiple rows per seller (each row = one sale). Need to sum first, then rank.

2. The SQL — aggregate inside, rank outside

SELECT region, seller_id, total
FROM (
  SELECT
    region, seller_id,
    SUM(amount) AS total,
    -- RANK over the GROUPED result, not the raw rows
    RANK() OVER (PARTITION BY region ORDER BY SUM(amount) DESC) AS rk
  FROM sales
  GROUP BY region, seller_id
) t
WHERE rk = 1;

4. Strategic explanations

The "aggregate then rank" composition trap

The order matters: GROUP BY first, RANK second. Reverse it (rank inside an unaggregated query) and you'd be ranking individual sale rows, not seller totals — the answer would be the seller with the single biggest sale, not the biggest cumulative seller. Senior engineers spot this composition error in code review.

Why RANK() over SUM(amount) works

Window functions can use aggregate expressions in their ORDER BY when they live in the same SELECT as the GROUP BY. The engine first computes the aggregate, then ranks the aggregated rows. This is a really clean pattern — rank over the aggregated metric — and worth memorising.

Tie-handling — what if two sellers tie?

If S101 and S102 both totalled $105,000 in North, RANK() = 1 would return both rows. That's usually right (both get the CEO call). If you must return exactly one (say, an engraving on a single trophy), add a deterministic tie-breaker:

RANK() OVER (
  PARTITION BY region
  ORDER BY SUM(amount) DESC, seller_id ASC    -- alphabetical fallback
) AS rk

Or switch to ROW_NUMBER() with the same ORDER BY — guaranteed exactly one row per partition.

"Top 3" instead of "top 1"

The CRO's next email: "Actually, give me the top 3 per region so I can also send notes to the runners-up." Trivial change:

... WHERE rk <= 3

This is the killer case for window functions vs subqueries — top-3-per-group via correlated subquery is awful; via window function it's trivial. Mention this contrast in the interview.

Snowflake / BigQuery shortcut: QUALIFY
-- Snowflake / BigQuery — no outer subquery
SELECT region, seller_id, SUM(amount) AS total
FROM sales
GROUP BY region, seller_id
QUALIFY RANK() OVER (PARTITION BY region ORDER BY SUM(amount) DESC) = 1;
Senior signal: name the pattern (top-N per group via window function over an aggregated query), justify the GROUP-then-RANK ordering, and show the trivial WHERE rk <= N extension for top-N. The contrast with correlated subqueries (which fall apart at top-3) is the senior insight.
