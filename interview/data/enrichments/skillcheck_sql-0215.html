The window-function approach reframes the problem as a roll-up: tag every employee as a Manager or not, then — for each manager — count how many of their reports are Managers and how many are Senior Managers. Aggregate window functions partitioned by manager_id do the counting. The trap is believing a window can also do the tagging.

2. The window-function solution

WITH base AS (
  SELECT e.emp_id, e.emp_name, e.manager_id,
         CASE WHEN EXISTS (SELECT 1 FROM employees s WHERE s.manager_id = e.emp_id)
              THEN 1 ELSE 0 END AS is_mgr,
         CASE WHEN EXISTS (
                SELECT 1 FROM employees s
                JOIN employees gs ON gs.manager_id = s.emp_id
                WHERE s.manager_id = e.emp_id)
              THEN 1 ELSE 0 END AS manages_a_mgr
  FROM employees e
),
boss_stats AS (
  SELECT manager_id AS mgr_id,
         COUNT(*)           OVER (PARTITION BY manager_id) AS direct_reports,
         SUM(is_mgr)        OVER (PARTITION BY manager_id) AS manager_reports,
         SUM(manages_a_mgr) OVER (PARTITION BY manager_id) AS sm_reports,
         ROW_NUMBER()       OVER (PARTITION BY manager_id) AS rn
  FROM base
  WHERE manager_id IS NOT NULL
)
SELECT emp.emp_name AS senior_manager, bs.direct_reports
FROM boss_stats bs
JOIN employees emp ON emp.emp_id = bs.mgr_id
WHERE bs.rn = 1
  AND bs.manager_reports >= 1
  AND bs.sm_reports = 0
ORDER BY bs.direct_reports DESC, senior_manager ASC;

base flags each employee: is_mgr (has a report) and manages_a_mgr (has a report who themselves has a report). boss_stats then partitions by manager_id, so every report row carries its boss’s totals. A manager qualifies when at least one report is a Manager and none is a Senior Manager.

4. Strategic explanations

PARTITION BY manager_id rolls the team up to its boss

Partitioning by manager_id groups all of one manager’s reports together. SUM(is_mgr) OVER (PARTITION BY manager_id) is then “how many of this manager’s reports are Managers” — a fact about the boss, copied onto each report row. ROW_NUMBER() plus WHERE rn = 1 collapses each partition back to one row per manager.

The trap: MAX(...) OVER (PARTITION BY emp_id) cannot detect a Manager

It is tempting to flag managers in “one pass” with MAX(CASE WHEN manager_id IS NOT NULL THEN 1 ELSE 0 END) OVER (PARTITION BY emp_id). It does not work. emp_id is unique, so each partition is a single row — the window sees no one else. The expression collapses to “does this row have a manager,” the opposite of being a manager: on this data it flags Alice (the CEO) as 0 and the ICs as 1. Identifying a Manager requires relating emp_id to the manager_id column — a semi-join (EXISTS / IN), which is exactly why base uses EXISTS.

Window functions annotate; they do not group

SUM(...) OVER (...) leaves the row count unchanged — it writes the aggregate onto every row of the partition. Without ROW_NUMBER() ... WHERE rn = 1 (or DISTINCT) the result would repeat each Senior Manager once per direct report. Forgetting that collapse step is a classic window-function slip.

Two flags, two levels

is_mgr is the first level — a report is a Manager. manages_a_mgr is the second — a report is itself a Senior Manager. Requiring manager_reports >= 1 and sm_reports = 0 encodes both halves of the definition: manage a Manager, but not a manager-of-managers.

Senior signal: window aggregates with PARTITION BY manager_id are ideal for rolling team facts up to a manager — but a window partitioned by a unique key sees only its own row. Detecting “is a manager” needs a semi-join, not a window; and a window annotates rows rather than collapsing them, so you still need a row-picking step.
