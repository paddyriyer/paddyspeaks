You want a report where each top-level root category — Film, Series — gets exactly one row, and the watch hours of individual sub-genres are broken out into their own columns. That row-to-column reshape is a pivot, and in standard SQL it is done with conditional aggregation: SUM(CASE WHEN ... THEN ... END).

2. The query — flatten, then pivot with SUM(CASE WHEN)

WITH category_hierarchy AS (
  SELECT
    c1.category_id   AS root_id,
    c1.category_name AS root_name,
    COALESCE(c3.category_id,   c2.category_id,   c1.category_id)   AS descendant_id,
    COALESCE(c3.category_name, c2.category_name, c1.category_name) AS descendant_name
  FROM categories c1
  LEFT JOIN categories c2 ON c2.parent_id = c1.category_id
  LEFT JOIN categories c3 ON c3.parent_id = c2.category_id
  WHERE c1.parent_id IS NULL
)
SELECT
  h.root_name AS main_category,
  COALESCE(SUM(t.watch_hours), 0) AS total_watch_hours,
  COALESCE(SUM(CASE WHEN h.descendant_name = 'Superhero'    THEN t.watch_hours END), 0) AS superhero_hours,
  COALESCE(SUM(CASE WHEN h.descendant_name = 'Spy'          THEN t.watch_hours END), 0) AS spy_hours,
  COALESCE(SUM(CASE WHEN h.descendant_name = 'Comedy'       THEN t.watch_hours END), 0) AS comedy_hours,
  COALESCE(SUM(CASE WHEN h.descendant_name = 'Drama Series' THEN t.watch_hours END), 0) AS drama_series_hours
FROM category_hierarchy h
LEFT JOIN titles t ON t.category_id = h.descendant_id
GROUP BY h.root_id, h.root_name
ORDER BY total_watch_hours DESC;

4. Strategic explanations

How SUM(CASE WHEN) becomes a column

Each CASE WHEN genre = '...' acts as a filter: it returns watch_hours for the rows of one genre and NULL for the rest. SUM then collapses those values per group, and SUM ignores NULLs — so each CASE expression yields one genre's total as one column. The GROUP BY on the root is what gives one output row per Film / Series.

Why every SUM is wrapped in COALESCE

A CASE with no ELSE returns NULL when nothing matches. For Film, no row is Drama Series, so every input to that SUM is NULL and the aggregate itself returns NULL. COALESCE(..., 0) turns that empty cell into a clean 0 — which is what drama_series_hours = 0 for Film shows.

total_watch_hours is not the sum of the named columns

Series totals 185, but its named columns sum to only 160. The missing 25 hours are Reality — a genre with no pivoted column. The pivot's column list is a deliberate choice; it need not be exhaustive. total_watch_hours counts every title; the named columns count only the genres you chose to break out.

The CASE literals must match exactly

'Drama Series' must match the data character-for-character. A typo or a casing mismatch ('drama series') does not error — it silently produces an all-zero column. Pivot columns are only as trustworthy as their literals.

Static pivot, and the dialect split

Standard SQL has no portable dynamic pivot: the column list is hard-coded, and a new genre means a new CASE line (or application-side dynamic SQL). SQL Server and Oracle offer a dedicated PIVOT operator; PostgreSQL, MySQL and SQLite do not. SUM(CASE WHEN ...) is the one form that runs everywhere.

Senior signal: a pivot is conditional aggregation — SUM(CASE WHEN dimension = literal THEN measure END), one expression per target column, GROUP BY the dimension that survives. Wrap each in COALESCE(..., 0) for clean zeros, and treat the column list as a deliberate, static choice rather than an exhaustive one.
