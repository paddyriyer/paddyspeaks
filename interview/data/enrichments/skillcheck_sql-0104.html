A log-archiving job needs to compress a set of unique-but-gappy log ids into contiguous ranges — 1,2,3 becomes the range 1–3, an isolated id becomes a range of itself. This is the classic "gaps and islands" problem, and the interviewer is checking whether you know the ROW_NUMBER difference trick that collapses a consecutive run into a single constant group key.

1. The source data — logs

log_id(narrative)
1start of run 1–3
2contiguous
3end of run; 4, 5, 6 missing
7start of run 7–8
8end of run; 9 missing
10isolated — start = end (9 and 11–14 missing)
15start of run 15–18
16, 17, 18contiguous to the end

The edge cases: 10 is a singleton island — its range must be 10–10, start equal to end. The run 15–18 sits at the very end of the data, testing that the last island closes cleanly. And there are gaps of varying width (one missing id between 8 and 10, several between 10 and 15).

2. The SQL — the ROW_NUMBER difference trick

WITH numbered AS (
  SELECT log_id, log_id - ROW_NUMBER() OVER (ORDER BY log_id) AS grp
  FROM logs
)
SELECT MIN(log_id) AS start_id, MAX(log_id) AS end_id
FROM numbered
GROUP BY grp
ORDER BY start_id;

ROW_NUMBER() numbers the sorted ids 1, 2, 3, ... With no gaps, log_id and the row number climb in lockstep, so their difference is constant within a run. A gap bumps log_id without bumping the row number, so the difference jumps — a new island. Grouping on that difference and taking MIN/MAX yields each range.

3. The resulting analysis

start_idend_idsignal
13ids 1,2,3 — difference constant at 0
78ids 7,8 — difference constant at 3
1010singleton — MIN = MAX, range 10–10
1518ids 15..18 — difference constant at 8

The singleton 10 emerges as 10–10 with no special handling — a one-row group still has a MIN and MAX, and they coincide. The trailing run 15–18 closes correctly at the end of the data.

4. Strategic explanations

Why log_id - ROW_NUMBER() is constant within a run

This is the heart of the technique. In a consecutive run, both log_id and the dense, gapless ROW_NUMBER increase by exactly 1 from row to row. Subtracting one from the other cancels that shared increment, so the difference holds steady across the whole run. The moment the data skips an id, log_id leaps ahead of ROW_NUMBER and the difference steps up by the size of the gap. The difference is constant if and only if the rows are consecutive — so it is a perfect group key for islands.

The grp value is a label, not a number to interpret

The actual values of grp (0, 3, 8 here) are meaningless — do not read anything into them. All that matters is that rows in the same island share one value and rows in different islands do not. GROUP BY grp then partitions the rows into islands, and MIN/MAX read off each island's endpoints. The trick converts an inherently sequential problem into a plain GROUP BY.

ROW_NUMBER, not RANK or DENSE_RANK

The technique relies on a numbering that increases by exactly 1 with no ties and no gaps. Log ids are unique, so ROW_NUMBER, RANK, and DENSE_RANK happen to agree here. But if the column could contain duplicates, RANK would skip and DENSE_RANK would repeat — both breaking the lockstep. ROW_NUMBER is the safe, intention-revealing choice: it is the only one guaranteed to be a strict 1,2,3,... sequence.

Singletons need no special case

A common worry is the isolated id 10. It needs nothing extra: it forms its own grp value, becomes a group of one row, and MIN(log_id) = MAX(log_id) = 10 — the range 10–10 the spec asks for. A solution built on LAG/LEAD to detect "gap before" and "gap after" boundaries is far more fiddly and easy to get wrong at the singleton and at the first/last row. The difference trick handles all of them uniformly.

The ORDER BY inside OVER is mandatory

ROW_NUMBER() OVER (ORDER BY log_id) must order by log_id — the numbering only tracks the ids if it is assigned in id order. Omit the ORDER BY and ROW_NUMBER assigns numbers in arbitrary physical order, the difference becomes noise, and the islands are garbage. This ORDER BY defines the computation; it is not optional styling. (The trailing ORDER BY start_id separately controls output order.)

Senior signal: recognise "collapse consecutive ids into ranges" as gaps-and-islands, solve it with log_id - ROW_NUMBER() OVER (ORDER BY log_id) — constant within a run, so it groups islands — treat the difference as an opaque label, use ROW_NUMBER (not RANK) for a strict gapless sequence, and note singletons fall out for free as MIN = MAX.
