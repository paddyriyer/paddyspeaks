Google's compensation team builds a report of each employee's total cash compensation — salary plus bonus. The arithmetic is trivial, but there is a catch hiding in the data: not everyone has a bonus, and an employee with no bonus carries a NULL. The interviewer asking this is testing one specific thing: whether you know that salary + NULL is NULL, and that COALESCE is the fix.

1. The source data — employees

emp_idnamesalarybonus
1Alex1000005000
2Bailey90000NULL
3Casey800002000
4Drew70000NULL

Alex and Casey have a bonus; Bailey and Drew have a NULL bonus — they are not on a bonus plan, so the field was never populated. Those two NULL rows are the whole interview: a naive salary + bonus would blank out their total comp entirely.

2. The SQL — COALESCE(bonus, 0)

SELECT emp_id, name, salary + COALESCE(bonus,0) AS total_comp FROM employees;

COALESCE(bonus, 0) returns bonus when it is present and 0 when it is NULL. So the addition always has two real numbers and never produces a NULL.

3. The resulting analysis

emp_idnametotal_comp
1Alex105000
2Bailey90000
3Casey82000
4Drew70000

Alex: 100000 + 5000 = 105000. Bailey: 90000 + COALESCE(NULL, 0) = 90000 — her salary survives intact. Without the COALESCE, Bailey and Drew would show a NULL total_comp and look like they earn nothing. That silent corruption is exactly what the question is about.

4. Strategic explanations

Why salary + NULL is NULL — the core trap

NULL means "unknown", and any arithmetic with an unknown is unknown: 90000 + NULL evaluates to NULL, not 90000. So a single NULL bonus doesn't just lose the bonus — it wipes the entire total. The bug is invisible: no error, just wrong numbers for every employee without a bonus. COALESCE substitutes a concrete 0 so the addition stays well-defined.

NULL is not zero — deciding that on purpose

Mapping a NULL bonus to 0 is a business decision, and here it is the right one: "no bonus" genuinely means a bonus contribution of $0. But the equation isn't always valid — if bonus were NULL because payroll hasn't computed it yet, treating it as 0 would understate comp, and leaving the total NULL ("not yet known") might be the honest answer. Saying that distinction out loud is the senior move.

COALESCE vs ISNULL / IFNULL / NVL

COALESCE is ANSI standard and works on every engine; ISNULL (SQL Server), IFNULL (MySQL) and NVL (Oracle) are vendor-specific two-argument equivalents. Prefer COALESCE — it is portable and, unlike the others, takes any number of arguments and returns the first non-NULL: COALESCE(bonus, signing_bonus, 0).

Coalesce the column, not the sum

A subtle ordering point. COALESCE(salary + bonus, 0) looks similar but is wrong here: if bonus is NULL the inner sum is NULL, COALESCE turns the whole thing into 0, and Bailey's $90,000 salary vanishes. The NULL must be neutralised on the specific operand that can be NULL — salary + COALESCE(bonus, 0) — before it contaminates the expression.

Edge case: a NULL salary too

The query trusts salary to be non-NULL. If salary could itself be NULL, the same trap returns — guard both: COALESCE(salary,0) + COALESCE(bonus,0). And note that SUM(total_comp) over the column would skip NULLs, so an aggregate could mask the bug at the row level — another reason to fix it at the source.

Senior signal: explain that any arithmetic with NULL yields NULL so one missing bonus blanks the whole total, neutralise the NULL on the exact operand (salary + COALESCE(bonus,0), not around the sum), prefer portable COALESCE over vendor functions, and flag that "NULL means 0" is a business decision worth confirming.
