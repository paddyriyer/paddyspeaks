The mixed-language framing of python_175-0008 — the in-memory kernel that every production observability pipeline (Datadog, Prometheus, CloudWatch, Splunk, Kafka Streams) reduces to. A firehose of (timestamp, key, value) tuples folds into a hash map keyed by (time_bucket, key). The interesting interview ground is not the implementation — it's the eviction, the cardinality blow-up, the top-K-at-scale question, and how this class plugs in front of a sharded back-end with the salt pattern from python_175-0002.

1. The problem — concrete numbers

10 web servers, each emitting ~1,000 log lines/min. One host (web-07) has a degraded disk and is the source of ~95% of ERROR-level lines. Cluster total: 10,000 lines/min.

Storage approachPer-minute rows in store"Errors on host X / min" latency
Raw log lines10,000O(10,000) scan — 50–500ms
Aggregator: (minute, (level, host)) → count~30O(1) hash look-up — <1µs

The aggregator turns a 10M-line/day firehose into a 43K-entry/day hash map (3 levels × 10 hosts × 1,440 minutes). Dashboard queries are point look-ups; the offender (web-07) jumps to the top of top() immediately.

2. The source code — with a runnable demo

from collections import defaultdict
import random

class LogAggregator:
    """Folds (ts, key, value) events into fixed-width time buckets.
    add / query / top are the primitives every metrics pipeline exposes."""
    def __init__(self, bucket_seconds=60):
        self.buckets = defaultdict(int)
        self.bs = bucket_seconds

    def add(self, ts, key, val=1):
        self.buckets[(ts // self.bs, key)] += val

    def query(self, bucket_id, key):
        return self.buckets.get((bucket_id, key), 0)

    def top(self, bucket_id, k=10):
        items = [(key, v) for (b, key), v in self.buckets.items() if b == bucket_id]
        return sorted(items, key=lambda x: -x[1])[:k]


if __name__ == "__main__":
    random.seed(11)
    start_ts, BUCKET = 1_700_000_000, 60
    agg = LogAggregator(bucket_seconds=BUCKET)
    hosts = [f"web-{i:02d}" for i in range(10)]
    sick = "web-07"

    # 10 min of logs, 10k lines/min, sick host emits 95% of ERRORs
    for minute in range(10):
        for _ in range(10_000):
            ts = start_ts + minute * 60 + random.randint(0, 59)
            if random.random() < 0.95:
                host = sick if random.random() < 0.95 else random.choice(hosts)
                level = "ERROR" if random.random() < 0.20 else "INFO"
            else:
                host = random.choice(hosts)
                level = "INFO"
            agg.add(ts, key=(level, host))

    m5 = (start_ts + 5 * 60) // BUCKET
    print("ERROR web-07 in minute 5:", agg.query(m5, ("ERROR", "web-07")))
    print("ERROR web-02 in minute 5:", agg.query(m5, ("ERROR", "web-02")))
    print("Top 5 (level, host) in minute 5:")
    for k, v in agg.top(m5, k=5):
        print(" ", k, "->", v)

    total = sum(v for (b, (lvl, h)), v in agg.buckets.items() if lvl == "ERROR")
    sick_n = sum(v for (b, (lvl, h)), v in agg.buckets.items()
                 if lvl == "ERROR" and h == sick)
    print(f"\nTotal ERRORs: {total}; on {sick}: {sick_n} "
          f"({100 * sick_n / total:.1f}%)")

3. The same pattern at every layer

  • Datadog / DogStatsD — wire format is metric:value|c|#tags; agent-side aggregator folds 10K of those/sec into one entry per (metric, tags, bucket). Pre-aggregation at the edge is how Datadog handles trillions of points/day.
  • Prometheus — counters like http_requests_total{method, status} are entries in this hash map, keyed by (scrape_bucket, (method, status)). Histograms add buckets as another dimension.
  • AWS CloudWatch EMF — Lambda emits embedded-metric-format JSON, server aggregates by (namespace, dimensions, minute). ~100× cheaper than PutMetricData per metric.
  • Honeycomb BubbleUp — keeps raw events but runs this aggregator at query time over a filtered set to surface the over-represented dimension value.
  • Splunk / Elasticsearch — stats count by host | bin _time span=1m or date_histogram + terms are this class wrapped in DSL.
  • Kafka Streams / Flink — groupBy(...).windowedBy(...).count() compiles to a RocksDB state store keyed by (window, key). Same algorithm; durable state instead of an in-memory dict.

5. Interviewer probes

  • "How do you bound memory?" — ring buffer of N most recent buckets, or per-bucket TTL eviction, or sketch-based (Count-Min + Misra-Gries) for unbounded cardinality.
  • "What if the key space is unbounded?" — fingerprint-hash the message, or sketch with bounded error, or drop the high-cardinality dimension from the key and surface it via sampled log search.
  • "How do you make top() fast?" — incremental min-heap of size K maintained on add(). For streaming top-K with bounded memory: Misra-Gries or Space-Saving.
  • "What about late events?" — ring buffer determines tolerated lateness; Kafka Streams grace() and Flink allowedLateness are the same knob. Beyond that, write to a "late events" stream for offline correction.
  • "How do you scale past one process?" — shard by key, fan-out at query time. Hot keys then pin shards — apply the salt pattern from python_175-0002.
  • "Why bucket by time at all?" — rates are what SLOs and alerts care about, and a running counter can't tell you "the last 5 minutes". The bucket size is the temporal resolution of every downstream question.
