An analytics team reports the "typical" score per category and someone asks for the median rather than the average, because a couple of outliers are dragging the mean. The interviewer wants to see whether you reach for PERCENTILE_CONT — an ordered-set aggregate — and whether you understand that it interpolates between the two middle values when a group has an even number of rows.

1. The source data — score

categoryvalue(narrative)
A10odd count (3) → exact middle value
A20
A30
B5even count (4) → interpolated middle
B15
B25
B35
C100single-member group → median is the value itself

Three deliberate shapes: an odd-count group (A), an even-count group (B) where the median falls between two stored values, and a single-row group (C) — the boundary case where median, min, max and mean all coincide.

2. The SQL — median via PERCENTILE_CONT

SELECT category,
       ROUND(PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY value)::numeric, 2) AS median
FROM score
GROUP BY category
ORDER BY category;

PERCENTILE_CONT(0.5) is an ordered-set aggregate: the WITHIN GROUP (ORDER BY value) clause tells it how to sort each group, and 0.5 asks for the 50th percentile. The CONT (continuous) variant assumes the underlying scale is continuous and interpolates linearly between bracketing values.

3. The resulting analysis

categorymediansignal
A20.00middle of {10,20,30} — an actual stored value
B20.00halfway between 15 and 25 — not in the table
C100.00single row — median equals that value

Category B is the teaching moment: its median is 20, the interpolated midpoint of the two central values 15 and 25, even though no row holds the value 20. That is exactly what PERCENTILE_CONT promises — and what distinguishes it from its sibling.

4. Strategic explanations

PERCENTILE_CONT vs PERCENTILE_DISC — the choice you must justify

PERCENTILE_DISC (discrete) returns an actual value from the set — for group B it would pick 15 (the first value whose cumulative position reaches 0.5), not 20. PERCENTILE_CONT interpolates. Use CONT for genuinely continuous measures (revenue, latency, temperature) where a midpoint is meaningful; use DISC when the value must be one that truly occurred (a discrete category code, an SLA tier). The question explicitly says "interpolated median", so CONT is correct here.

Why ordered-set aggregates need WITHIN GROUP

An ordinary aggregate like SUM does not care about row order. A percentile fundamentally does — you cannot find a median without sorting. WITHIN GROUP (ORDER BY ...) is the special syntax that supplies that ordering to the aggregate. It is not the same as an ORDER BY in the query's final clause; it lives inside the aggregate call.

NULL handling — quietly ignored

PERCENTILE_CONT skips NULL values in the ordering column, computing the percentile over the non-NULL members only — consistent with how AVG and SUM behave. That is usually what you want, but be explicit: if half a category's values are NULL the "median" describes only the populated half. If a group is entirely NULL (or empty) the function returns NULL.

The cast and the ROUND

PERCENTILE_CONT returns double precision for floating inputs; ROUND(value, 2) in Postgres requires a numeric argument, hence the explicit ::numeric cast before rounding. Skip the cast and you get function round(double precision, integer) does not exist. The interpolation arithmetic is also done in floating point, so the cast-then-round step is what gives you a clean two-decimal presentation value.

Dialect notes

PERCENTILE_CONT ... WITHIN GROUP is SQL-standard and works in PostgreSQL, Oracle, SQL Server, BigQuery and Snowflake. MySQL has no ordered-set aggregate before 8.0 and still lacks PERCENTILE_CONT as an aggregate — there you emulate the median with window functions: number the rows per category, count them, and average the middle one or two. Knowing that fallback signals you can deliver a median on any engine.

Senior signal: compute medians with the ordered-set aggregate PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY value), explain that CONT interpolates while DISC returns a stored value, cast to numeric before ROUND, and know the window-function emulation for engines like MySQL that lack the aggregate.
