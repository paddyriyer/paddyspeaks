Google asks "Add Two Numbers" because it hides a surprising amount of engineering judgement behind grade-school arithmetic. The digits are stored reversed in two linked lists (ones place first), and you must produce a third list. It tests whether you can manage a carry, walk two lists of unequal length in one pass, and — the senior tell — whether you reach for a dummy head node instead of special-casing the first append.

1. The input data

Numbers 342 and 465. Stored least-significant-digit first:

l1:  2 -> 4 -> 3        # represents 342
l2:  5 -> 6 -> 4        # represents 465
                       # 342 + 465 = 807

2. The approach

One pass, simulating column addition. A dummy head (dummy = cur = ListNode()) lets every digit be appended uniformly via cur.next = ... — no "is this the first node?" branch. The loop condition while l1 or l2 or carry is the elegant part: it keeps going while either list has digits left or a final carry still needs a node (e.g. 5 + 5 = 10).

Each iteration sums the two current digits, substituting 0 for an exhausted list (l1.val if l1 else 0), plus the incoming carry. The key idiom is carry, v = divmod(s, 10): one call splits the sum into the digit to store (s % 10) and the carry to forward (s // 10). Both list pointers advance, guarding against None. At the end, dummy.next skips the placeholder and returns the real head.

4. Strategic explanations

Complexity

Time O(max(m, n)) where m, n are the list lengths — one node visited per iteration. Space O(max(m, n)) for the result list (O(1) auxiliary beyond the output). The result can be one node longer than the longer input when a final carry escapes.

The idiomatic alternative — and why it's a trap

The "Pythonic" temptation is to decode each list to an int, add, and re-encode:

def to_int(node):
    n, p = 0, 1
    while node: n += node.val * p; p *= 10; node = node.next
    return n
# add the two ints, then split digits back into a list

It works in Python because ints are arbitrary precision — but it's the wrong answer in an interview. It quietly assumes bignum support (fails in C++/Java for long lists) and signals you missed the point: the question is about streaming digit-by-digit with a carry. State that you know the shortcut exists but reject it for portability.

The edge case

Two tricky ones, both handled. Unequal lengths: the shorter list contributes 0 once exhausted, so 99 + 1 works. Trailing carry: 5 + 5 → the lists are empty but carry == 1 keeps the loop alive for one more node, producing 0 -> 1 (i.e. 10). Without or carry in the guard, that leading digit is silently dropped.

The follow-up

"What if the digits are stored most-significant-first?" Now you can't add left-to-right because the carry flows the other way. Two answers: reverse both lists, add, reverse the result — or push both lists onto stacks and pop in tandem, building the result by prepending. The stack version keeps the inputs immutable.

The trap

Forgetting the dummy node and instead initialising head = None forces an ugly if head is None check inside every iteration and makes returning the head awkward. The dummy-head pattern is the standard linked-list builder idiom — not using it here is a junior tell.

Senior signal: uses a dummy head so the append loop has zero special cases, collapses digit-and-carry extraction into a single divmod, gets the while l1 or l2 or carry condition right so a trailing carry is never lost, and explicitly rejects the "convert to int" shortcut as non-portable rather than reaching for it.
