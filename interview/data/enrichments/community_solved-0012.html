A Community aggregation question: from an events table, find the client (desktop, mobile, tablet…) that generates the most call activity — video and voice calls, sent or received. It sounds like a one-line "group, sort, take the top". The reference solution, however, reaches for a "more than 50% of total" filter, and that is the trap the interviewer is laying.

1. The source data — fact_events

client_idevent_type
desktopvideo call received
desktopvideo call sent
desktopvoice call received
desktopmessage sent
mobilevoice call sent
mobilemessage received
tabletmessage sent

Counting only call events: desktop has 3, mobile has 1, tablet has 0. The non-call rows (message sent, message received) are noise — they must not count. desktop is clearly the most active caller.

2. The SQL — group, then take the top by call count

WITH t1 AS (
    SELECT client_id,
           SUM(CASE WHEN event_type IN ('video call received', 'video call sent',
                                        'voice call received', 'voice call sent')
                    THEN 1 ELSE 0 END) AS specific_event_count
    FROM fact_events
    GROUP BY client_id
)
SELECT client_id FROM t1
ORDER BY specific_event_count DESC
LIMIT 1;

The CASE inside SUM is a conditional count — it adds 1 only for the four call event types, so specific_event_count is each client's call volume. Sorting that descending and taking LIMIT 1 returns the most active client.

3. The resulting analysis

client_id
desktop

desktop wins with 3 call events against mobile's 1 and tablet's 0 — the expected answer.

4. Strategic explanations

Why the original "> 50% of total" solution is wrong

The reference answer did not sort and take the top. It filtered:

SELECT client_id FROM t1
WHERE specific_event_count > 0.5 * (SELECT SUM(specific_event_count) FROM t1)
LIMIT 1;

That keeps a client only if it owns a strict majority — over half — of all call events. "Most popular" means the maximum, not "an absolute majority". The two coincide only when one client happens to dominate. The moment activity is split — say desktop 4, mobile 3, tablet 3 — no client clears 50%, the WHERE matches nothing, and the query returns zero rows. It fails to name a most-popular client precisely when the question is least trivial. The corrected query in section 2 sorts by count and takes the top, which always returns the genuine leader.

Conditional aggregation: SUM(CASE...) vs COUNT FILTER

SUM(CASE WHEN ... THEN 1 ELSE 0 END) is the portable conditional count. Postgres has a cleaner native form:

COUNT(*) FILTER (WHERE event_type IN ('video call received', ...)) AS specific_event_count

Same result, and it reads as exactly what it is. Either way, doing the filtering inside the aggregate — rather than a WHERE before the GROUP BY — keeps clients with zero calls in the result (tablet shows 0), which matters if you later want to rank or report them all.

The tie trap — LIMIT 1 is non-deterministic

If two clients tie for the most calls, ORDER BY count DESC LIMIT 1 returns one of them, arbitrarily — whichever the planner emits first. To return all co-leaders, rank instead:

SELECT client_id FROM (
  SELECT client_id, specific_event_count,
         RANK() OVER (ORDER BY specific_event_count DESC) AS rk
  FROM t1
) WHERE rk = 1;

Confirm with the interviewer: "if there's a tie for most popular, do you want one client or every tied client?"

The category-list maintainability problem

Hard-coding four call types in an IN list is brittle — a new 'video call missed' event ships and the query silently undercounts. If event types follow a pattern, match the pattern: event_type LIKE '%call%', or better, join to a small dimension table that classifies event types into "call" / "message" / etc. A senior flags that the literal list will rot.

Edge case: no call events at all

If the table has only message events, every specific_event_count is 0. The corrected query still returns a row — some client with a count of 0 — which is misleading ("most popular" among zero activity). Guard it if needed: WHERE specific_event_count > 0 before the LIMIT, so an empty result honestly means "nobody made calls".

Senior signal: recognise that "most popular" is a max-and-rank problem, not a "> 50% majority" filter — the majority test returns nothing whenever activity is split — use COUNT FILTER for the conditional count, and handle ties with RANK() rather than a non-deterministic LIMIT 1.
