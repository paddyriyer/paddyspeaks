A meeting-room booking system needs a double-booking report: for each room, which pairs of reservations collide in time? The interviewer is testing whether you know the correct interval-overlap predicate — a formula people routinely get wrong — and whether you can produce each colliding pair exactly once without duplicating it as both (A,B) and (B,A).

The boundary case is the whole point. Booking 1 ends at 10:00 and booking 3 starts at 10:00; bookings 2 and 4 meet at 10:30. With half-open [start, end) intervals these touch but do not overlap — the room is free the instant one ends and the next begins. A correct predicate must exclude touching pairs.

2. The SQL — self-join with the overlap predicate

SELECT a.room_id, a.id AS id_a, b.id AS id_b
FROM booking a
JOIN booking b
  ON a.room_id = b.room_id
 AND a.id < b.id
 AND a.start_ts < b.end_ts
 AND b.start_ts < a.end_ts
ORDER BY a.room_id, id_a, id_b;

Two intervals overlap when each one starts before the other ends: a.start < b.end AND b.start < a.end. The a.id < b.id condition does double duty — it stops a row from joining to itself, and it picks just one of the two mirror-image orderings so every pair appears once.

4. Strategic explanations

Why a.start < b.end AND b.start < a.end is the right formula

It is far easier to define when two intervals don't overlap: one finishes before the other starts (a.end <= b.start OR b.end <= a.start). Negate that with De Morgan's law and you get a.end > b.start AND b.end > a.start — the same predicate, rearranged. Deriving it this way, rather than memorising it, means you can re-derive it under pressure and adjust the inequalities for open vs closed intervals.

Half-open intervals and the choice of < vs <=

The question specifies [start, end) — end-exclusive. A booking occupies the room up to but not including its end time, so a 10:00 end and a 10:00 start do not conflict. That maps to strict <. If intervals were fully closed [start, end] (both endpoints occupied), touching would count as overlap and you would use <=. Getting this inequality wrong is the single most common bug in overlap queries.

Why a.id < b.id beats a.id != b.id

Using != would return every pair twice — once as (1,2) and once as (2,1) — and would also need separate self-match exclusion. The strict < collapses both problems: it is asymmetric, so exactly one of (1,2)/(2,1) survives, and since no id is less than itself, self-joins are impossible. It also guarantees the requested id_a < id_b output ordering for free.

Performance — the self-join is O(n²) without an index

A naive self-join compares every pair of rows in a room. For large booking tables, create an index on (room_id, start_ts) so the planner can range-scan candidate partners instead of doing a full cross product per room. At very large scale, range types plus a GiST exclusion index (tstzrange with &&) let Postgres prune overlaps directly — worth mentioning as the production-grade approach.

Postgres range types — the idiomatic alternative

Postgres can express this natively: tsrange(a.start_ts, a.end_ts) && tsrange(b.start_ts, b.end_ts) using the && "overlaps" operator, and tsrange(..., '[)') makes the half-open semantics explicit in the type itself. Even better, an EXCLUDE USING gist constraint can prevent overlapping bookings from being inserted at all — pushing the rule into the schema rather than a report.

Senior signal: derive the overlap predicate (a.start < b.end AND b.start < a.end) from the simpler non-overlap condition rather than memorising it, match the inequality strictness to the interval's open/closed semantics, use a.id < b.id to dedupe pairs and exclude self-joins in one stroke, and know that Postgres range types with a GiST exclusion constraint are the scalable, schema-level answer.
