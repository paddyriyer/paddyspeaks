An e-commerce growth team defines a "returning active user" as anyone who made a second purchase within 7 days of one of their other purchases — a retention metric that feeds churn dashboards. The interviewer is probing whether you can compute per-user gaps between consecutive events with a window function instead of an expensive self-join, and whether you understand why comparing each purchase to only its immediate successor is sufficient.

User 100 has a 4-day gap then a 46-day gap; user 200 a single 14-day gap; user 300 a lone purchase; user 400 made two purchases on the same day (gap = 0). The single-purchase user and the same-day pair are the edge cases that decide the filter logic.

2. The SQL — LEAD to find the next purchase, then a gap

WITH np AS (
  SELECT id, userid, item, createdat, revenue,
         LEAD(createdat, 1, NULL) OVER (PARTITION BY userid
                                        ORDER BY createdat ASC) AS next_purchase_date
  FROM transactions_ecomm
), fc AS (
  SELECT *,
    CASE WHEN createdat::text ~ '^[0-9]{4}-[0-9]{2}-[0-9]{2}'
          AND next_purchase_date::text ~ '^[0-9]{4}-[0-9]{2}-[0-9]{2}'
         THEN next_purchase_date::date - createdat::date
         ELSE NULL END AS diff_flag
  FROM np
)
SELECT DISTINCT userid FROM fc
WHERE diff_flag IS NOT NULL AND diff_flag <= 7 AND diff_flag <> 0;

4. Strategic explanations

Why LEAD over a sorted partition is enough

The definition says "within 7 days of any other purchase" — which sounds like every pair must be compared. But once a user's purchases are sorted ascending, the smallest possible gap from any purchase is the gap to its immediate next neighbour. If even that consecutive gap exceeds 7 days, no farther purchase can be closer. So LEAD(createdat) OVER (PARTITION BY userid ORDER BY createdat) — comparing each purchase to the very next one — provably catches every qualifying user. This is the key insight that turns an O(n²) self-join into an O(n log n) single sort.

The diff_flag <> 0 clause — a debatable edge case

User 400 bought a phone and a case on the same day: a genuine second purchase, gap 0. The query's <> 0 filter excludes them — an implicit decision that same-day purchases are "one shopping session", not a return visit. That is defensible if the product wants distinct days, but it is a silent assumption. State it explicitly: if a same-day repeat purchase counts as returning (it usually does — the user came back and transacted), drop <> 0 and use diff_flag BETWEEN 0 AND 7. An interviewer notices when you surface a buried business rule instead of copying it.

The regex guard is defensive scaffolding, not core logic

The ~ '^[0-9]{4}-[0-9]{2}-[0-9]{2}' checks exist because createdat might be stored as text with dirty values. If the column is a proper DATE/TIMESTAMP (as it should be), the guard is dead weight and the CTE simplifies to next_purchase_date - createdat AS diff_flag. In an interview, note this: validating date format inside the query is a symptom of a schema problem — the fix belongs at ingestion, with a typed column.

LEAD's default argument handles the last purchase cleanly

LEAD(createdat, 1, NULL) — the third argument is the default when there is no next row. User 300's only purchase, and the last purchase of every user, get next_purchase_date = NULL, so diff_flag is NULL and the IS NOT NULL filter drops them. That is correct: a user with no subsequent purchase cannot be "returning". Without the explicit default the behaviour is the same (LEAD returns NULL past the edge anyway), but spelling it out documents intent.

Common mistake — using createdat > current_date - 7

A frequent wrong turn is interpreting "within 7 days" as "in the last 7 days from today". The metric is about the gap between two of the user's own purchases, anchored to each other — not to the current date. Read retention definitions precisely: the window is relative to the user's events, not the calendar.

Senior signal: articulate why comparing each purchase only to its immediate LEAD neighbour fully satisfies "within 7 days of any purchase" (sorted ⇒ consecutive gap is minimal), turning a quadratic self-join into one sort. Flagging the <> 0 same-day exclusion as a buried, debatable business rule — rather than copying it — is the senior move.
