An e-commerce team wants each customer's most recent order — the canonical "latest record per group" problem. The reference solution uses a correlated subquery that, for every row, finds that customer's maximum order date and keeps the row only if it matches. The interviewer is probing whether you understand why this returns more than one row per customer when dates tie, and whether you know the window-function alternative.

Customer 103 is the trap: two orders on the same date, 2026-04-01. "Latest order" is now ambiguous — the query has to do something with that, and what it does tells the interviewer how well you understand the pattern.

2. The SQL — correlated subquery on MAX(order_date)

SELECT * FROM orders o
WHERE order_date = (SELECT MAX(order_date) FROM orders
                    WHERE customer_id = o.customer_id);

For each outer row o, the subquery computes the maximum order_date among orders with the same customer_id. The row survives only if its own date equals that maximum.

Customer 103 returns two rows. Both orders share the maximum date 2026-04-01, so both satisfy order_date = MAX(...). If the report assumed exactly one row per customer, this duplicate would silently break a downstream join or count.

4. Strategic explanations

Why ties produce multiple rows — and whether that is acceptable

The filter is an equality against a date, and a date is not unique within a customer. Two orders on the same day both equal the max, so both pass. This is not a bug in the SQL — it is the SQL faithfully answering "rows whose date is the latest". If the business needs exactly one row per customer, you must break the tie deterministically, e.g. by order_id or amount. State the tie-breaker explicitly; "the latest order" is underspecified until you do.

The window-function alternative — clearer and tie-controllable

The modern idiom ranks rows per customer and keeps rank 1:

SELECT order_id, customer_id, order_date, status, amount
FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY customer_id
                               ORDER BY order_date DESC, order_id DESC) AS rn
  FROM orders
) t
WHERE rn = 1;

ROW_NUMBER guarantees one row per customer because the ORDER BY order_id DESC tie-breaker makes the ordering total. Swap it for RANK() if you deliberately want all tied rows — then the choice is explicit, not accidental.

Postgres has an even terser tool — DISTINCT ON

PostgreSQL offers SELECT DISTINCT ON (customer_id) * FROM orders ORDER BY customer_id, order_date DESC, order_id DESC. It keeps the first row of each customer_id group as defined by the ORDER BY — exactly one row per customer, tie broken by the trailing key. It is concise and fast, but Postgres-only; ROW_NUMBER is the portable equivalent.

The correlated subquery runs once per row

WHERE customer_id = o.customer_id makes the subquery correlated: it re-executes for every outer row. Without an index on (customer_id, order_date) that is repeated scanning — quadratic in the worst case. With that index each MAX is an index lookup, and the pattern is fine. The window-function version sorts once, O(n log n), and does not depend on the planner caching the subquery.

NULL order dates — the silent dropout

If a customer's only order has a NULL order_date, MAX ignores NULLs and returns NULL, and order_date = NULL is never true — that customer vanishes from the result entirely. If unscheduled orders must still appear, coalesce to a sentinel or handle NULL dates explicitly. The window form has the same issue unless you specify NULLS LAST in the ORDER BY.

Senior signal: recognise that order_date = MAX(order_date) returns every tied row, not one per customer, pick ROW_NUMBER (one row, total order) vs RANK (all ties) deliberately, reach for DISTINCT ON in Postgres, and note that NULL dates silently drop customers from the answer.
