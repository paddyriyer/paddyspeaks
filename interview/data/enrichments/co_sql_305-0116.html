Yahoo's HR analytics team wants every employee tagged into a compensation band — High, Mid, or Low — for a pay-equity dashboard. It is a per-row label, the bread-and-butter use of CASE in a SELECT. The interview is whether your band thresholds are airtight: every employee gets exactly one band, the boundary salaries land where you intend, and a NULL salary does not silently misfile.

1. The source data — employees

Two rows sit exactly on a threshold: Ben at 100000 and Dee at 50000. With ≥ they fall into the upper band; with > they would drop to the lower one. Those boundary employees are the whole interview.

2. The SQL — CASE in the select list

SELECT emp_id, salary,
  CASE WHEN salary >= 100000 THEN 'High'
       WHEN salary >= 50000 THEN 'Mid' ELSE 'Low' END AS band
FROM employees;

CASE evaluates its WHEN arms top to bottom and returns the first match. There is no GROUP BY — this is a per-row transformation, one band label per employee.

4. Strategic explanations

Order matters — first match wins

The arms are correct because they descend. Ann at 120000 satisfies both ≥ 100000 and ≥ 50000, but CASE stops at the first true arm, so she is High. Reorder them — WHEN salary ≥ 50000 THEN 'Mid' first — and everyone earning 50K+ becomes Mid; the High arm becomes unreachable dead code. The thresholds also need not restate their lower bound: reaching the ≥ 50000 arm already implies salary is under 100000.

The boundary is a real decision

"High is 100K and up" vs "above 100K" is a one-character change (≥ vs >) that moves every employee sitting exactly on a band edge. On a real payroll, salaries cluster on round numbers, so this is not a corner case — it is many people. Confirm the intended bands with the interviewer, ideally as a half-open convention (lower bound inclusive, upper exclusive) so adjacent bands never overlap or gap.

The NULL-salary trap

Every WHEN here is a comparison, and NULL ≥ anything is unknown — not true. So an employee with a NULL salary fails both WHEN arms and falls into ELSE, getting silently labelled 'Low'. That is wrong: their pay is unknown, not low. Handle it before the comparisons:

CASE WHEN salary IS NULL THEN 'Unknown'
     WHEN salary >= 100000 THEN 'High'
     WHEN salary >= 50000 THEN 'Mid'
     ELSE 'Low' END

A bare ELSE is a magnet for NULLs and out-of-range values — never let it absorb cases you have not thought about.

The follow-up — count per band

HR will immediately ask "how many in each band". Wrap the CASE and group by it:

SELECT band, COUNT(*) FROM (
  SELECT CASE WHEN salary >= 100000 THEN 'High'
              WHEN salary >= 50000 THEN 'Mid' ELSE 'Low' END AS band
  FROM employees
) t GROUP BY band;

Watch the ordering trap on the result: ORDER BY band sorts High, Low, Mid alphabetically — not the natural Low/Mid/High. Carry a numeric sort key, or order by the band's lower-bound salary.

Maintainability — bands as data, not code

Hard-coded thresholds (100000, 50000) mean a comp-policy change requires editing every query that bands salaries. At scale, store the bands in a small table — salary_bands(band, min_salary, max_salary) — and join with a range condition. The literal CASE is perfect for an interview answer; mentioning the table-driven version shows you have maintained this kind of logic in production.

Senior signal: explain that CASE returns the first matching arm so descending thresholds are essential, pin down ≥ vs > on the band boundaries, and — the real differentiator — handle NULL salary explicitly before the comparisons so the ELSE arm does not silently misfile it as Low.
