A data engineer inherits a user_profiles table and wants a fast health check before trusting it: which columns are mostly or entirely empty? The deliverable is one row per column showing its NULL count and NULL percentage, sorted nulls-first so dead columns surface at the top. The interviewer is probing whether you can compute a per-column profile when SQL aggregates work down rows, not across columns.

The edge case is signup_source: it is NULL in every row — a 100%-NULL column, exactly the kind of dead field this audit must catch. country is the opposite extreme: fully populated, 0% NULL. The check has to span both ends cleanly.

2. The SQL — transpose with UNION ALL over conditional counts

The reference solution returns the null count and the table total, but the question explicitly asks for the null percent too — that column is missing, so the solution does not fully answer the brief. The corrected version below adds it.

-- ORIGINAL (incomplete — no percent column)
WITH totals AS (SELECT COUNT(*) AS n FROM user_profiles)
SELECT 'age' AS col, SUM(CASE WHEN age IS NULL THEN 1 ELSE 0 END) AS nulls,
       (SELECT n FROM totals) AS total FROM user_profiles
UNION ALL SELECT 'gender', SUM(CASE WHEN gender IS NULL THEN 1 ELSE 0 END),
       (SELECT n FROM totals) FROM user_profiles
-- … one branch per column …
ORDER BY nulls DESC;
-- CORRECTED — adds null_pct so the result answers "count and percent"
WITH totals AS (SELECT COUNT(*) AS n FROM user_profiles),
counts AS (
  SELECT 'age' AS col, SUM(CASE WHEN age IS NULL THEN 1 ELSE 0 END) AS nulls FROM user_profiles
  UNION ALL SELECT 'gender',        SUM(CASE WHEN gender        IS NULL THEN 1 ELSE 0 END) FROM user_profiles
  UNION ALL SELECT 'country',       SUM(CASE WHEN country       IS NULL THEN 1 ELSE 0 END) FROM user_profiles
  UNION ALL SELECT 'signup_source', SUM(CASE WHEN signup_source IS NULL THEN 1 ELSE 0 END) FROM user_profiles
  UNION ALL SELECT 'last_login',    SUM(CASE WHEN last_login    IS NULL THEN 1 ELSE 0 END) FROM user_profiles
)
SELECT c.col, c.nulls, t.n AS total,
       ROUND(100.0 * c.nulls / t.n, 1) AS null_pct
FROM counts c CROSS JOIN totals t
ORDER BY c.nulls DESC;

Each SUM(CASE WHEN col IS NULL THEN 1 ELSE 0 END) counts NULLs in one column; UNION ALL stacks those one-number results into rows, transposing columns into rows. The corrected query joins the row total once and divides to get the percentage.

4. Strategic explanations

Why SUM(CASE…) and not COUNT

COUNT(col) counts non-NULL values, so NULLs are COUNT(*) - COUNT(col) — workable but indirect. SUM(CASE WHEN col IS NULL THEN 1 ELSE 0 END) counts the NULLs directly and reads as exactly what it does. The ELSE 0 matters: without it, non-NULL rows contribute NULL, and while SUM skips NULLs the explicit 0 keeps the intent obvious and the result a guaranteed integer.

The float-division trap in the percentage

100 * nulls / total with all-integer operands does integer division in Postgres, SQLite, and MySQL: 2/6 becomes 0, and every non-100% column reports 0. Writing 100.0 (a numeric literal) forces the whole expression to numeric, so 33.3 comes through. This is the single most common bug in any "compute a percentage" query — one of the operands must be non-integer.

Why transpose at all — SQL aggregates go down, not across

SUM, COUNT, AVG collapse rows within a column; there is no built-in "aggregate across columns of one row". To get a per-column profile you must compute each column's metric separately and then UNION ALL them into a row each. That is the structural reason this query is verbose — it is rotating a column-wise question into SQL's row-wise model.

This does not scale by hand — code-generate it

Five columns is fine to type; eighty is not, and the query would be unmaintainable. The senior move is to read column names from information_schema.columns in a host language (or a DO block) and generate the UNION ALL text programmatically, then execute it. The SUM(CASE) + UNION ALL pattern is the right target shape; for wide tables it should be machine-written, not hand-written.

Dialect notes

The pattern is fully portable. Postgres can shortcut the count with count(*) FILTER (WHERE col IS NULL) — cleaner than SUM(CASE) but Postgres/SQLite-only. ROUND(numeric, 1) is standard; some engines need an explicit cast. The CROSS JOIN totals in the corrected query is a one-row broadcast — equivalent to the original's scalar subquery, just expressed once instead of per branch.

Senior signal: profile NULLs per column with SUM(CASE WHEN col IS NULL THEN 1 ELSE 0 END) stacked by UNION ALL, force numeric division with 100.0 so percentages are not silently zeroed, return the percentage alongside the count so results compare across table sizes, and state that for wide tables this query should be code-generated from information_schema rather than hand-typed.
