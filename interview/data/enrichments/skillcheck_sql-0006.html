A product-analytics team wants each user's two most recent events — the classic "latest N per group" feed query. The interviewer is checking whether you reach for ROW_NUMBER partitioned per user and filtered to a small rn, rather than something that fails on ties or only returns one row per group.

1. The source data — event

user_idevent_ts(narrative)
12025-01-05 10:00user 1 — most recent
12025-01-04 09:00user 1 — 2nd most recent
12025-01-03 08:00user 1 — 3rd, should be dropped
22025-01-07 12:00user 2 — most recent
22025-01-06 11:00user 2 — 2nd most recent
32025-01-01 07:00user 3 — only one event

Two edge cases. User 1 has three events, so the query must keep two and discard the oldest; user 3 has only one event, so "top 2" must gracefully return just that single row rather than erroring or padding.

2. The SQL — ROW_NUMBER partitioned per user

WITH ranked AS (
  SELECT
    user_id,
    event_ts,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY event_ts DESC) AS rn
  FROM event
)
SELECT user_id, event_ts
FROM ranked
WHERE rn <= 2
ORDER BY user_id, event_ts DESC;

PARTITION BY user_id restarts the numbering for each user; ORDER BY event_ts DESC puts the newest event first, so it receives rn = 1. The outer query keeps only rn <= 2 — the two newest per user — because a window function cannot be referenced in the same query's WHERE, hence the CTE.

User 1's oldest event (2025-01-03) is correctly absent — it received rn = 3 and failed rn <= 2. User 3 returns a single row: the partition has only one event, so rn never exceeds 1 and the <= 2 filter passes it through unharmed.

4. Strategic explanations

Why ROW_NUMBER is the right tool for "top N per group"

ROW_NUMBER guarantees a strict 1, 2, 3… sequence within each partition, so WHERE rn <= N returns at most N rows per group — never more. That bounded count is usually what a feed wants: exactly two cards per user, no surprises. The pattern — window function in a CTE, integer filter outside — is the canonical "latest N" idiom across every modern SQL engine.

ROW_NUMBER vs RANK/DENSE_RANK on ties

If a user had two events with the identical timestamp competing for the second slot, ROW_NUMBER picks one arbitrarily (some tie-break by physical order) and still returns exactly two rows. RANK would give both tied events rn = 2 and return three rows; DENSE_RANK likewise. So the function encodes a policy: ROW_NUMBER = "exactly 2, break ties arbitrarily", RANK = "2 plus anyone tied". State which the product wants — and if ties must break deterministically, add a tiebreaker to the ORDER BY, e.g. ORDER BY event_ts DESC, event_id DESC.

Why the CTE is necessary — window functions and WHERE

You cannot write WHERE ROW_NUMBER() OVER (...) <= 2 directly: window functions are evaluated after WHERE in the logical processing order, so the alias does not yet exist when WHERE runs. Wrapping the calculation in a CTE (or subquery) materialises rn as an ordinary column the outer WHERE can test. This ordering rule is worth being able to recite.

Edge cases: single-event users and timestamps stored as text

User 3 shows that "top 2" must tolerate groups smaller than 2 — rn <= 2 handles it for free, returning whatever exists. One latent risk: event_ts is declared TEXT. The ordering happens to be correct because the values are zero-padded YYYY-MM-DD HH:MM, which sorts identically as text and as a timestamp. With a non-sortable format the ORDER BY would silently mis-rank; the robust fix is a real TIMESTAMP column or an explicit event_ts::timestamp cast.

Senior signal: solve "latest N per group" with ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY event_ts DESC) filtered to rn <= N in an outer query, explain why the CTE is required (window functions run after WHERE), choose ROW_NUMBER over RANK deliberately for a hard cap on rows, and add an explicit tiebreaker so identical timestamps resolve deterministically.
