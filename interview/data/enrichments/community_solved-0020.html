A Community question on the Flipkart transactions table: find "returning active users" — people who made a repeat purchase within seven days of a previous one. Retention teams use this to size the genuinely sticky cohort. The crux is comparing each purchase to the same user's prior purchase, which is a textbook LAG, and then being precise about how a date difference is measured.

User 3 has three purchases (one row omitted above): a 4-day gap then a long gap, so they qualify on one of the pairs. User 4 has a single purchase — no prior to compare. Four users, four distinct behaviours.

2. The SQL — LAG then 7-day filter

WITH t1 AS (
    SELECT ft.*,
           LAG(created_at) OVER (PARTITION BY user_id ORDER BY created_at ASC) AS previous_purchase_date
    FROM flipkart_transactions ft
),
t2 AS (
    SELECT t1.*,
           EXTRACT(DAY FROM (created_at::timestamp - previous_purchase_date::timestamp))::int AS date_diff
    FROM t1
    WHERE EXTRACT(DAY FROM (created_at::timestamp - previous_purchase_date::timestamp))::int <= 7
)
SELECT user_id
FROM t2
GROUP BY user_id
ORDER BY user_id ASC;

LAG pulls each user's previous purchase date into the same row. t2 keeps only rows where the gap to that prior purchase is 7 days or fewer. The final GROUP BY dedupes to one row per qualifying user.

4. Strategic explanations

Why LAG over a self-join

"Compare each purchase to the same user's previous one" is the canonical LAG use. The window PARTITION BY user_id ORDER BY created_at means the previous row is the previous purchase for that user, never bleeding across users. The self-join alternative — join the table to itself on user_id and "earlier date" — works but matches every earlier purchase, not just the immediately preceding one, so you would need a MAX correlation to recover "the latest before this". LAG says it in one clause.

The EXTRACT(DAY FROM interval) trap — this is the real interview point

created_at::timestamp - previous_purchase_date::timestamp is an interval, and EXTRACT(DAY FROM interval) returns only the days component of that interval — not the total span. For gaps under a month it equals the total days, so it is fine here. But an interval of '1 mon 2 days' yields 2, not ~32. The robust way to get whole elapsed days is plain date subtraction, which returns an integer count directly:

WHERE created_at::date - previous_purchase_date::date <= 7

On this dataset both give the same answer; on month-spanning gaps they diverge badly. Naming this is the senior move.

The boundary: is day 7 in or out?

<= 7 includes a purchase exactly 7 days later; < 7 would exclude it. "Within a week" is genuinely ambiguous — 7 days inclusive, or strictly under 7? Confirm with the interviewer; an off-by-one on the boundary silently shifts the cohort. Also note the difference is always ≥ 0 here because LAG orders ascending, so a same-day repeat (diff 0) also qualifies — intended or not, say so.

Why the final GROUP BY is needed

A user with several quick repeats produces several qualifying rows in t2. The question asks which users, so the answer must be one row per user — GROUP BY user_id (or SELECT DISTINCT user_id) collapses the duplicates. Skip it and a hyper-active user appears multiple times, breaking any downstream count.

Edge cases: single-purchase users and tied timestamps

A one-purchase user has a NULL previous_purchase_date; the subtraction is NULL and fails the WHERE — correctly excluded, no guard required. If two purchases share an identical created_at, LAG's ordering between them is arbitrary (a tie in the ORDER BY); add a tiebreaker like ORDER BY created_at, id for deterministic output.

Senior signal: reach for LAG to fetch the prior purchase per user, and above all flag that EXTRACT(DAY FROM interval) returns only the days field — preferring integer date subtraction for true elapsed days — while pinning the inclusive/exclusive boundary of "within 7 days" before coding.
