Zscaler's customer-success team wants the average order size for VIP customers, to decide whether the VIP tier is pulling its weight on revenue. The data spans two tables: customers carries the is_vip flag, orders carries the amounts. The query joins them and averages. The interviewer is watching the join grain, the is_vip type, and the subtle difference between "average order" and "average per customer".

Customer 1 is a VIP with two orders; customer 3 is a VIP with one. Customer 2 is not a VIP and must be excluded. The uneven order counts (2 vs 1) are the trap: they make "average order" and "average customer spend" diverge.

2. The SQL — JOIN + filtered AVG

SELECT AVG(o.amount) AS vip_avg
FROM customers c JOIN orders o ON c.customer_id = o.customer_id
WHERE c.is_vip::int = 1;

An inner join pairs each order with its customer, the WHERE keeps only VIP customers' orders, and AVG averages the amounts of whatever rows survive.

3. The resulting analysis

vip_avg
400.0000000000000000

Three VIP orders survive the join — 200, 400, 600 — summing to 1200, and 1200 ÷ 3 = 400. This is the average order, not the average customer. Customer 1's two orders each count once, so customer 1 is weighted twice as heavily as customer 3 in the result.

4. Strategic explanations

"Average order" vs "average customer" — the grain question

The query averages over orders: each order is one data point. Customer 1 contributes two points (200, 400), customer 3 contributes one (600). So a frequent-buyer skews the number. If the question means "what does a typical VIP customer spend", that is a two-level aggregation — sum per customer first, then average the per-customer totals:

SELECT AVG(cust_total) AS avg_vip_customer
FROM (
  SELECT o.customer_id, SUM(o.amount) AS cust_total
  FROM customers c JOIN orders o ON c.customer_id = o.customer_id
  WHERE c.is_vip::int = 1
  GROUP BY o.customer_id
) t;
-- AVG(600, 600) = 600, not 400

Customer 1 totals 600, customer 3 totals 600, average 600. Same data, a 50% different headline — purely a grain decision. Naming "average order or average customer?" before writing SQL is the senior move.

is_vip::int = 1 — what the cast is defending against

The ::int cast hints that is_vip is stored inconsistently — perhaps a boolean, perhaps text '1'/'0', perhaps a smallint. Casting normalises it before comparison. But casts hide assumptions: if is_vip were the text 'yes', 'yes'::int would throw. The robust move is to know the column's real type. If it is a genuine boolean, the idiomatic filter is simply WHERE c.is_vip — no cast, no literal. If it is nullable, c.is_vip IS TRUE handles the NULL case cleanly.

The inner join silently excludes VIPs who never ordered

INNER JOIN keeps only customers with at least one order. A VIP who has never placed an order contributes nothing — which is correct for "average order size" (they have no orders to average). But beware the related question "average revenue per VIP": there, a zero-order VIP should count as 0, so you would need a LEFT JOIN and COALESCE. The join type encodes which question you are answering.

Status and the meaning of "order"

The orders table has a status column not used here. Cancelled or pending orders are sitting inside the average. "Average order size" almost always means realised orders — add AND o.status = 'completed'. Otherwise a VIP's abandoned $5,000 cart drags the metric around.

Edge case: no VIP customers, or VIPs with no orders

If no customer is a VIP — or every VIP has zero orders — the join produces no rows and AVG returns NULL, not 0. A scorecard tile expecting a number needs COALESCE(AVG(o.amount), 0). Mention it: an empty aggregate is NULL, and that NULL will surface the first time the VIP segment is empty.

Senior signal: distinguish "average order" (average over order rows) from "average customer spend" (per-customer sum, then average) and confirm which the stakeholder means, treat the ::int cast as a flag to verify is_vip's real type rather than trust it, add the status = 'completed' filter, and note the empty-segment NULL.
