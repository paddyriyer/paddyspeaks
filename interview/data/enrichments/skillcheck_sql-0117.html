A gaming analytics team wants each player's cumulative games-played total — a running sum that, on any given day, shows everything they have played up to and including that date. This is the textbook cumulative-window question. The interviewer is checking whether you use a windowed SUM with the right frame and partition, rather than a self-join that re-aggregates the whole history per row.

1. The source data — activity

player_idevent_dategames_played(narrative)
12025-01-015player 1, day 1
12025-01-023player 1, next day
12025-01-042player 1 — note the gap, Jan 3 has no row
22025-01-0110player 2 — same date as player 1, separate partition
22025-01-034player 2, day 2

Two edge cases. Player 1 has a calendar gap (no Jan 3 row) — the running total must simply skip it, not reset or interpolate. And both players have a Jan 1 row — the running sum must keep their histories independent, which the PARTITION BY guarantees.

2. The SQL — windowed SUM with an explicit running frame

SELECT player_id, event_date,
       SUM(games_played) OVER (PARTITION BY player_id ORDER BY event_date
                                ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)::int AS games_played_so_far
FROM activity
ORDER BY player_id, event_date;

PARTITION BY player_id keeps each player's sum separate. ORDER BY event_date defines the accumulation direction. The frame ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW says "sum every row from the start of this player's partition through the current row" — that span is exactly the running total.

Player 1's total climbs 5 → 8 → 10; the missing Jan 3 causes no reset because the window sums rows, not calendar days. Player 2 starts fresh at 10 — the PARTITION BY prevents player 1's running total from leaking in even though both share the Jan 1 date.

4. Strategic explanations

Why a windowed SUM beats a correlated self-join

The pre-window approach is a correlated subquery: for each row, SELECT SUM(games_played) FROM activity a2 WHERE a2.player_id = a.player_id AND a2.event_date <= a.event_date. It re-scans and re-sums the player's whole history for every row — O(n²) within a partition. The windowed SUM sorts each partition once and accumulates in a single linear pass. Same answer, dramatically better complexity.

The ROWS frame is the heart of "running total"

A window function with an ORDER BY but no explicit frame defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — which is usually a running total too, but with a critical subtlety (next point). ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW states the intent unambiguously: accumulate from partition start to here. Other frames give other shapes — ROWS BETWEEN 2 PRECEDING AND CURRENT ROW is a 3-row moving sum; omitting ORDER BY sums the whole partition into every row.

ROWS vs RANGE — the tie-handling trap

This is the senior detail. With ROWS, the frame is "this physical row and all before it." With RANGE, the frame includes all rows whose ORDER BY value ties the current row. If a player had two rows on the same event_date, RANGE would give both of those rows the same running total (the sum through the end of that date), whereas ROWS would show two stepped values. The data here is one row per player per day so they coincide — but spelling out ROWS is the correct, deliberate choice. Mention this distinction unprompted.

Calendar gaps do not break a row-based running total

Player 1 has no Jan 3 row. Because the frame counts rows, not days, the running total just moves from the Jan 2 row to the Jan 4 row — 8 then 10. The total is "games played so far," and on a day with no play there is nothing to add. If the requirement were a per-calendar-day series with zeros on idle days, you would first generate a complete date spine and LEFT JOIN the activity onto it — a different question. State which is wanted.

Dialect notes

The windowed SUM with a ROWS frame is standard SQL — PostgreSQL, SQL Server, Oracle, MySQL 8+, and SQLite 3.25+ all support it. Pre-8.0 MySQL had no window functions and forced the self-join or a session-variable hack. The only Postgres-specific token is the ::int cast; elsewhere use CAST(... AS INT). The framing clause itself needs no translation.

Senior signal: compute a running total with a windowed SUM partitioned by the entity and ordered by date, write the frame explicitly as ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, know that ROWS vs RANGE diverges the moment the order key has ties, and recognise that a row-based running total skips calendar gaps by design.
