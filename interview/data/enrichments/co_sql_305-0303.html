Snowflake's clinic-scheduling team needs to flag double-booked doctors: any two appointments for the same physician whose time ranges overlap. This is the classic interval-overlap problem, and the elegant solution is a self-join with the deceptively simple overlap predicate. The interview tests whether you know that predicate cold — and whether you build the join so each colliding pair appears exactly once, with both appointment IDs legible in the output.

2. The SQL — self-join on the overlap predicate

SELECT a.appt_id AS appt_id_a, b.appt_id AS appt_id_b
FROM appointments a
JOIN appointments b
  ON a.doctor_id = b.doctor_id
 AND a.appt_id < b.appt_id
 AND a.start_ts < b.end_ts
 AND b.start_ts < a.end_ts;

The table is joined to itself, matched on the same doctor. a.appt_id < b.appt_id keeps each pair once and drops self-matches; the last two conditions are the overlap test. Note: the reference solution selected a.appt_id, b.appt_id — two columns with the same name. Distinct aliases (appt_id_a, appt_id_b) are required so both IDs survive into the result; see the trap below.

4. Strategic explanations

The duplicate-column-name bug — why aliasing is mandatory

The reference query's SELECT a.appt_id, b.appt_id produces a result set with two columns both named appt_id. SQL permits this, but many drivers materialise each row as an object keyed by column name — and a second appt_id key overwrites the first. The output collapses to a single garbled column, silently losing one side of every pair. The query is logically sound but practically broken on output. Always give self-join projections distinct aliases (appt_id_a / appt_id_b); it costs nothing and makes the result unambiguous.

The overlap predicate — memorise this exact form

Two intervals overlap when a.start < b.end AND b.start < a.end. The trick is that it is easier to state when they don't overlap — one ends before the other begins — and negate. People reach for tangled BETWEEN logic and miss the case where one interval fully contains the other. This two-clause form handles partial overlap, containment, and identical ranges uniformly. Note the strict <: appointments that merely touch (one ends 09:30, the next starts 09:30) are treated as not overlapping — usually the right call for back-to-back slots, but confirm it.

a.appt_id < b.appt_id does two jobs

Drop that clause and the self-join returns: every appointment paired with itself (it trivially overlaps itself), and every real pair twice — (1,2) and (2,1). The < inequality eliminates self-matches and de-duplicates the mirrored pairs in one stroke. Using <> instead would still produce both (1,2) and (2,1); the ordering inequality is what makes each collision appear exactly once.

Counting collisions per doctor — the natural follow-up

Schedulers rarely want the raw pair list; they want "which doctors are double-booked, and how badly":

SELECT a.doctor_id, COUNT(*) AS overlapping_pairs
FROM appointments a
JOIN appointments b
  ON a.doctor_id = b.doctor_id
 AND a.appt_id < b.appt_id
 AND a.start_ts < b.end_ts
 AND b.start_ts < a.end_ts
GROUP BY a.doctor_id;

This returns doctor 10 with 1 and doctor 11 with 3. Doctor 12, with no collisions, drops out entirely — correct, since the inner join produces no rows for them.

Scale — the O(n²) shape

A self-join compares appointments pairwise; per doctor with k appointments that is ~k² comparisons. Partitioning by doctor_id already keeps each doctor's quadratic separate — good. For very busy calendars, a window-function approach scales better: sort each doctor's appointments by start_ts and compare each to the running MAX(end_ts) of prior rows with LAG/a frame — one ordered pass instead of a cross product. Mention this trade-off; the self-join is clear and fine for clinic-sized data, but it is not the answer for millions of rows.

Senior signal: give every self-join projection distinct aliases so duplicate column names cannot silently drop a side of the pair, recall the a.start < b.end AND b.start < a.end overlap predicate exactly, explain that a.appt_id < b.appt_id both removes self-matches and de-duplicates mirrored pairs, and name the O(n²) cost with an ordered-scan alternative for large calendars.
