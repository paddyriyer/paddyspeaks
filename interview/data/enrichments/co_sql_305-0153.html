Crypto.com's compensation team wants every employee shown against their department's average salary, with the gap spelled out — the raw input for "who is underpaid relative to peers" reviews. The natural shape is: compute one average per department, then join it back to each employee. A CTE does the first step cleanly. The interview value is in the join grain, what happens to employees with no department, and the tempting-but-wrong window-function shortcut.

Department 10 has three salaries averaging 120K — employee 3 sits exactly on the average (diff 0). Department 20 averages 100K. Department 30 is a one-person department: its average is employee 6's salary, so the diff is necessarily 0. Those two zero-diff rows are the edge cases worth pointing at.

2. The SQL — CTE for dept average, then join

WITH dept_avg AS (
  SELECT dept_id, AVG(salary) AS avg_sal
  FROM employees
  GROUP BY dept_id
)
SELECT e.emp_id, e.salary, d.avg_sal,
       e.salary - d.avg_sal AS diff
FROM employees e
JOIN dept_avg d ON e.dept_id = d.dept_id;

The CTE dept_avg collapses the table to one row per department; the main query joins each employee back to their department's row to pick up avg_sal and compute the gap.

4. Strategic explanations

The window-function shortcut — one pass, no join

The CTE-plus-join works, but it scans employees twice and pays for a join. A windowed AVG computes the department average alongside each employee row in a single pass:

SELECT emp_id, salary,
       AVG(salary) OVER (PARTITION BY dept_id) AS avg_sal,
       salary - AVG(salary) OVER (PARTITION BY dept_id) AS diff
FROM employees;

Same numbers, no GROUP BY, no join, and it naturally keeps every employee — including ones the join might drop. On a large table this is the answer to give; the CTE version is fine but the window version is what signals fluency.

The dead dept_avg column — a denormalisation trap

The schema already carries a dept_avg column on employees. It is tempting to just SELECT salary - dept_avg and skip all computation. Resist it: a stored aggregate is a cached value that goes stale the instant anyone's salary changes — give a raise and every cached dept_avg in that department is now wrong until something recomputes it. Compute the average live from the salaries unless you can prove the column is maintained by a trigger. Flagging this stale-denormalisation risk is the senior observation.

Employees with a NULL dept_id

If an employee has no department (NULL dept_id — a contractor, an unassigned new hire), the INNER JOIN drops them: NULL = NULL is never true, so they match no dept_avg row. They simply vanish from the comp review. Decide deliberately: a LEFT JOIN keeps them with avg_sal NULL (and diff NULL), or filter them out on purpose — but do not let them disappear silently. The window-function version sidesteps this: a NULL-dept employee just forms their own partition.

The grain check — why this join is safe

Joining a detail table to an aggregate is safe only because the CTE produces exactly one row per dept_id — GROUP BY dept_id guarantees it. If dept_avg could emit two rows for a department, each employee would match both and the result would double. Always confirm the join's right side is unique on the join key; here GROUP BY is the guarantee, so each employee matches exactly one average.

The ranking follow-up

Expect "now rank each employee within their department." That is one more window function over the same partition: RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC). Pointing out that the dept-average and the in-dept rank share the identical PARTITION BY dept_id — computable side by side in one query — is the over-the-bar finish.

Senior signal: offer AVG(salary) OVER (PARTITION BY dept_id) as the single-pass alternative to the CTE-and-join, refuse to trust the stored dept_avg column because cached aggregates go stale, handle NULL-dept_id employees deliberately rather than letting the inner join drop them, and confirm the aggregate side is unique per key so the join cannot fan out.
