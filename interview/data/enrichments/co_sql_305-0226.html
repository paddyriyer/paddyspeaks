Databricks' logistics-analytics team wants the average time from order placed to order delivered, in days, for an SLA dashboard. The arithmetic is timestamp subtraction — but the interviewer is watching whether you handle orders not yet delivered, whether you understand that subtracting two timestamps yields an interval (not a number), and whether the convoluted epoch formula in the reference solution can be reasoned about.

1. The source data — orders

Four orders are delivered with clean intervals (2 days, 1.5 days, 5 days, 0.5 days). Order 5 is still in transit — delivered_at is NULL. That undelivered row is the interview: it must not be counted as a zero-day delivery.

2. The SQL — AVG of an epoch-based day count

SELECT AVG(((EXTRACT(EPOCH FROM (delivered_at - ordered_at)) / 86400.0) * 86400)/86400) AS avg_days FROM orders;

delivered_at - ordered_at is an interval; EXTRACT(EPOCH FROM ...) turns it into total seconds; dividing by 86,400 (seconds per day) gives days. The ×86400 ÷86400 in the middle cancels out — more on that below.

3. The resulting analysis

avg_days
2.25000000000000000000

The four delivered orders took 2, 1.5, 5, and 0.5 days — summing to 9, averaged over 4 rows, gives 2.25. Order 5 contributed nothing: delivered_at is NULL, so the subtraction is NULL, and AVG skips NULL values in both the sum and the count. The undelivered order was correctly excluded automatically.

4. Strategic explanations

The formula has dead arithmetic — simplify it

(EXTRACT(EPOCH FROM diff) / 86400.0) * 86400 / 86400 divides by 86400, multiplies by 86400, divides by 86400 again — the middle ×86400 ÷86400 is a no-op. The whole expression is just EXTRACT(EPOCH FROM diff) / 86400. It still produces the right answer (2.25), so this is not a bug — but a senior candidate calls out the redundancy and writes the clear version:

SELECT AVG(EXTRACT(EPOCH FROM (delivered_at - ordered_at)) / 86400.0) AS avg_days
FROM orders;
Why NULL handling here is a feature, not luck

Order 5 is undelivered. delivered_at - ordered_at with a NULL operand is NULL; AVG ignores NULLs, so the row drops out of both numerator and denominator — the average is over 4 delivered orders, which is correct. But say it out loud, and consider making the intent explicit so a reader is not relying on implicit behaviour:

... WHERE delivered_at IS NOT NULL

If instead a NULL delivered_at should count as "delivery time so far" (an aging metric), you would COALESCE(delivered_at, now()) — a different, deliberate choice.

Timestamp subtraction yields an interval, not a number

delivered_at - ordered_at is an interval — you cannot directly AVG a "number of days" out of it without conversion. EXTRACT(EPOCH FROM ...) is the standard bridge to seconds. A cleaner Postgres-native alternative averages the interval itself, then formats:

SELECT AVG(delivered_at - ordered_at) AS avg_interval FROM orders;
-- returns an interval like '2 days 06:00:00'

AVG can average an interval directly — useful when you want a human-readable duration rather than a decimal.

Watch the dialect — EXTRACT(EPOCH ...) is not universal

The epoch trick is Postgres/Redshift idiom. Other engines differ: SQL Server uses DATEDIFF(day, ordered_at, delivered_at), BigQuery TIMESTAMP_DIFF(delivered_at, ordered_at, DAY), MySQL TIMESTAMPDIFF(DAY, ...). Note DATEDIFF-style functions count boundary crossings — DATEDIFF(day, ...) between 11pm and 1am the next day returns 1, while the epoch method returns ~0.083. The epoch approach gives a true fractional duration; pick it when partial days matter.

Status filtering and bad data

Averaging over all rows mixes statuses. Cancelled orders may carry a non-NULL delivered_at that is really a cancellation timestamp — filter to status = 'delivered' to be safe. And guard the impossible: a delivered_at earlier than ordered_at (clock skew, backfill error) yields a negative duration that silently drags the average down — worth a sanity check or a CHECK constraint.

Senior signal: spot that the ×86400 ÷86400 cancels and rewrite the formula as a plain EXTRACT(EPOCH ...) / 86400, explain that the NULL delivered_at is correctly excluded by AVG (and make it explicit with a filter), know that timestamp subtraction returns an interval that AVG can also average directly, and flag that DATEDIFF-style functions count boundary crossings rather than true elapsed time.
