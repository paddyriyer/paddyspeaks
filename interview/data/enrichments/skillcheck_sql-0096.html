A merchandising team has a small reference list of "top items" and wants the users who have bought every single one. Order, duplicates, and purchases of items outside the list are all irrelevant — only full coverage of the reference set counts. This is a relational-division problem, and the interviewer is checking whether you express "owns all of set X" as a count match rather than chaining one INTERSECT per item.

1. The source data — top_items and purchase

The edge cases test both failure modes. User 3 bought item 30 three times — the duplicates must not inflate coverage to "more than 3". Users 1 and 5 bought items (40, 50) not in the list — those extras must not be counted toward the target.

2. The SQL — relational division by COUNT(DISTINCT)

SELECT p.user_id
FROM purchase p
WHERE p.item_id IN (SELECT item_id FROM top_items)
GROUP BY p.user_id
HAVING COUNT(DISTINCT p.item_id) = (SELECT COUNT(*) FROM top_items)
ORDER BY p.user_id;

The WHERE ... IN keeps only purchases of reference items, so items 40 and 50 are dropped first. Then per user, COUNT(DISTINCT p.item_id) counts how many distinct reference items they bought; if that equals the size of top_items, they own the whole set.

4. Strategic explanations

This is relational division — "owns all of X"

"Find users who bought every top item" is the textbook relational-division pattern. The clean SQL idiom: filter to the divisor set, group by the entity, and require the distinct-count of matched items to equal the divisor's cardinality. It scales to any set size without touching the query — add a fourth top item and the subquery SELECT COUNT(*) FROM top_items just returns 4.

Why COUNT(DISTINCT) is non-negotiable

User 3 bought item 30 three times. Plain COUNT(p.item_id) would count 5 purchase rows — 5 = 3 is false, and user 3 would be wrongly excluded. DISTINCT answers the real question ("how many different reference items") rather than "how many purchase events". Whenever duplicates exist in the fact table, a coverage test must count distinct keys.

The WHERE ... IN filter does two jobs

It excludes out-of-list items and makes the count comparison sound. Without it, COUNT(DISTINCT p.item_id) would include items 40 and 50, so user 1 would score 4 distinct items — 4 = 3 is false and a fully-qualified user would be dropped. Restricting to the divisor set first guarantees the distinct-count can never exceed the target. Filter to the set, then count within it.

The alternative: NOT EXISTS double negation

The other canonical form is "no top item exists that this user did not buy": SELECT user_id FROM purchase p1 WHERE NOT EXISTS (SELECT 1 FROM top_items t WHERE NOT EXISTS (SELECT 1 FROM purchase p2 WHERE p2.user_id = p1.user_id AND p2.item_id = t.item_id)). It is provably equivalent and handles an empty divisor differently — with an empty top_items, the double-NOT EXISTS returns all users (vacuously true), while the count form returns none. Know both; pick by which empty-set semantics the business wants.

Edge case: the empty divisor

If top_items were empty, (SELECT COUNT(*) FROM top_items) is 0, and no grouped user has COUNT(DISTINCT) = 0 (a group only exists if it has rows) — so this query returns no one. That is a defensible answer ("nobody has bought all of nothing... in a useful sense"), but it differs from the set-theoretic "everybody". Flagging that divergence unprompted is the senior move.

Senior signal: recognise "bought every top item" as relational division, implement it by filtering to the reference set then requiring COUNT(DISTINCT item_id) = (SELECT COUNT(*) FROM top_items), insist on DISTINCT so repeat purchases do not inflate coverage, and know the NOT EXISTS equivalent diverges only on the empty-divisor edge case.
