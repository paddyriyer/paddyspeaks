Etsy's seller-analytics team reports profit margin by region so the merchandising org can see where the marketplace makes money. Margin looks like a one-line ratio, but it hides two real decisions: whether you average per-row margins or compute one margin from the totals, and what happens when a region has zero revenue. An interviewer is watching whether you aggregate then divide, not divide then aggregate.

1. The source data — sales

regionrevenuecost
North1000600
North500350
South20002200
East8000
West00

Three edge cases are baked in. South sells at a loss (cost > revenue). East has zero cost — a 100% margin. And West has zero revenue, which would divide by zero if nothing guarded it.

2. The SQL — margin from group totals

SELECT region,
  100.0 * (SUM(revenue) - SUM(cost)) / NULLIF(SUM(revenue),0) AS profit_margin_pct
FROM sales GROUP BY region;

For each region, sum revenue and cost separately, take the difference (gross profit), and divide by total revenue. NULLIF(SUM(revenue), 0) converts a zero denominator into NULL so the division is defined.

3. The resulting analysis

regionprofit_margin_pct
East100.0000000000000000
South-10.0000000000000000
WestNULL
North36.6666666666666667

North combines its two rows: revenue 1500, cost 950, profit 550, margin 550/1500 = 36.67%. South returns −10% — a real loss, correctly negative. East is exactly 100%. West, with zero revenue, returns NULL rather than crashing the query. Note the long decimal tail — numeric division does not round on its own.

4. Strategic explanations

Aggregate then divide — the heart of the question

The correct margin is (∑revenue − ∑cost) / ∑revenue. A common mistake is to compute a margin per row and then average those:

-- WRONG: averages percentages, ignores that rows have different sizes
SELECT region, AVG(100.0 * (revenue - cost) / NULLIF(revenue,0))
FROM sales GROUP BY region;

For North that gives AVG(40%, 30%) = 35%, not 36.67%. Averaging ratios weights a $500 sale the same as a $1000 sale. Margin is a weighted quantity — sum the parts, divide once.

Why NULLIF and not CASE

NULLIF(SUM(revenue), 0) is the compact idiom for "guard this denominator". The verbose equivalent is CASE WHEN SUM(revenue) = 0 THEN NULL ELSE SUM(revenue) END — identical behaviour, more typing. Either way the result for a zero-revenue region is NULL. If the dashboard wants 0% there instead, wrap the whole expression: COALESCE(100.0 * … / NULLIF(…,0), 0) — but a region with no sales arguably has undefined margin, so NULL is often the more honest answer.

The integer-division trap

The leading 100.0 is load-bearing. If both revenue and cost were integer columns and you wrote 100 * (SUM(revenue) - SUM(cost)) / SUM(revenue), Postgres does integer division on the final step and a 36.67% margin truncates to 36 — or worse, a small-profit region truncates to 0%. Multiplying by 100.0 (a numeric literal) promotes the whole expression to numeric arithmetic. (SUM(revenue) - SUM(cost))::numeric would do the same job.

Rounding for the report

The 16-digit decimal tail is ugly on a slide. Round at the presentation layer:

SELECT region,
  ROUND(100.0 * (SUM(revenue) - SUM(cost)) / NULLIF(SUM(revenue),0), 2) AS profit_margin_pct
FROM sales GROUP BY region
ORDER BY profit_margin_pct DESC NULLS LAST;

The NULLS LAST keeps the zero-revenue region from sorting to the top as if it were the best performer.

NULLs inside the columns

SUM skips NULL inputs, so a row with NULL cost contributes its revenue but no cost — inflating margin. If a missing cost should mean "cost unknown, exclude the sale", filter it; if it means "cost was zero", use SUM(COALESCE(cost, 0)). Decide which, and say so.

Senior signal: compute one margin from summed totals rather than averaging per-row ratios, explain why margin is weighted, guard the zero-revenue denominator with NULLIF, keep the numeric promotion (100.0) so integer division never truncates the percentage, and decide deliberately whether a no-revenue region is 0% or NULL.
