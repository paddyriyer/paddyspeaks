The ETL function nobody writes for fun but everyone ships eventually. Datadog's signup pipeline ingests millions of trial-signup rows a year from a marketing landing page; that page is the dirtiest data source in the company because real humans type emails with leading spaces, phone numbers with parentheses and dashes, and "two thousand" in the amount field. Garbage in, garbage in the warehouse, garbage in the segmentation model — and the marketing director's quarterly cohort report has to be defensible to the board. This pipeline is where dirty becomes clean.

1. The data — what "raw" actually looks like

Typical landing-page CSV — every column has a failure mode:

The status filter at the end (dropna(subset=['signup_date', 'email_domain'])) is a quarantine pattern: rows that failed to enrich go in a side table, not silently into the report. Production at Datadog routes the bad rows to a signups_quarantine table that the DQ team reviews weekly.

2. The business question — what the consumer of clean data decides on

  • Email-domain segmentation. email_domain tells marketing whether the signup is from a corporate account (datadog.com = internal test, drop; shopify.com = enterprise lead; gmail.com = consumer trial). The whole sales playbook bifurcates here.
  • Tier bucketing for activation. The high-tier signups (amount > $1000) trigger an SDR (sales-development-rep) follow-up within 2 hours. The low-tier signups get a self-serve onboarding email. Misclassifying a $2400 enterprise lead as "low" because the amount field had "not a num" means a missed deal.
  • Cohort analytics. signup_year is the keystone of every retention curve. If 2023 has 30 % more drop-outs than 2022 at day 30, that's a product-fit alarm — but only if the year extraction worked on every row.

3. The pandas implementation — annotated

def etl_pipeline(raw):
    df = raw.copy()

    # ── CLEAN — eliminate or normalise garbage.
    # Drop rows that fail the existence check. id is the immutable
    # entity key; without it the row can't be joined to anything
    # else in the warehouse.
    df = df.dropna(subset=['id'])

    # Emails: lowercase + strip whitespace. RFC 5321: the local-part
    # is technically case-sensitive but no major mail provider treats
    # it that way; the canonical form is lowercase for de-dup.
    df['email'] = df['email'].str.lower().str.strip()

    # Phones: strip every non-digit char. Keep the digits-only form
    # so downstream SMS sends pass via Twilio without rejecting
    # parens / dashes. The regex pattern \D matches "not a digit".
    # .astype(str) coerces any numeric/null up-front so the regex
    # doesn't crash on a NaN value.
    df['phone'] = df['phone'].astype(str).str.replace(r'\D', '', regex=True)

    # Amount: coerce with errors='coerce' → bad values become NaN,
    # then fillna(0). The 0-fill is a policy decision: missing amount
    # means "free tier" in Datadog's data model. If the dataset's
    # semantics treat missing as truly unknown, prefer dropna instead.
    df['amount'] = pd.to_numeric(df['amount'], errors='coerce').fillna(0)

    # Dates: errors='coerce' → bad values become NaT. We don't fill
    # because a fake "today's date" would silently corrupt the cohort
    # analysis. Bad dates ride through, get dropped at the end.
    df['signup_date'] = pd.to_datetime(df['signup_date'], errors='coerce')

    # ── ENRICH — derive analytical columns.
    df['signup_year']  = df['signup_date'].dt.year
    df['email_domain'] = df['email'].str.split('@').str[-1]

    # pd.cut creates a categorical with labelled bins. The bin
    # boundaries (-1, 100, 1000, ∞) cover negative values too, in
    # case a refund slips through. labels= names the bins; the
    # default would be 'left-right interval' strings (unreadable).
    df['tier'] = pd.cut(df['amount'], bins=[-1, 100, 1000, float('inf')],
                        labels=['low', 'mid', 'high'])

    # ── QUARANTINE — drop rows where mandatory enrichment failed.
    # Bad date = no cohort year. Bad email = no domain. These rows
    # are unusable for downstream segmentation.
    df = df.dropna(subset=['signup_date', 'email_domain'])
    return df

Why errors='coerce' and not errors='raise'? The whole point of an ETL is to survive bad data. raise would halt the pipeline on the first "not-a-num", requiring manual intervention. coerce turns the bad value into NaN/NaT, which the rest of the pipeline handles uniformly. The quarantine step (the final dropna) is where the bad rows leave the dataset; production routes them to a side table for DQ review rather than dropping silently.

Why pd.cut and not np.select or nested np.where? pd.cut returns a Categorical, which is memory-efficient (one int per row + a label dictionary) and orderable. np.where returns plain strings, blowing memory on 100 M rows. pd.cut also handles the bin-boundary semantics explicitly (closed-on-the-right by default) so you can defend "exactly $100" deterministically.

5. Production patterns — how this scales

  • pandas → 1 GB before OOM. Datadog's signup volume (~3 M/year) fits comfortably; the production version writes to BigQuery via a Cloud Run job that processes a daily Parquet partition. The function is identical — just wrapped in read_parquet → etl_pipeline → to_parquet.
  • Idempotency. The pipeline must be re-runnable on the same input and produce identical output. The function above is idempotent because every step is deterministic. The only non-deterministic risk is the source CSV itself — Datadog uses content-hash deduplication on the input file path to ensure the same file isn't processed twice.
  • Schema evolution. When marketing adds a new column (say utm_source), the pipeline must not break. Defence: read with explicit dtypes, fall back to object for unknown columns, and log a warning. dbt's on_schema_change='append_new_columns' is the same pattern at the warehouse layer.
  • Hot batch sizes. If a single CSV is 5 GB (one viral marketing campaign), the function needs chunked reading: pd.read_csv(path, chunksize=100_000) and an apply-per-chunk loop. Or move to Polars (lazy, columnar, multi-core) for the same algorithm at 10× speed.
  • Stripe Atlas, Segment, mParticle all expose this exact "clean + enrich + bucket" shape as a managed service. The pandas version is what you write before you've outgrown the team's appetite for a managed CDP — and what you write to understand what the CDP is doing for you.

6. Interviewer probes

Probe 1 — "Amount = 'not-a-num' → 0. Is that correct?"

It's a policy choice, not a universally correct one. Three options: (a) coerce to 0 = "free tier" assumption (what we did, OK for trial signups); (b) coerce to NaN, drop the row (strict — kills the lead entirely); (c) coerce to NaN, route to quarantine for manual review (most defensible). Pick (c) for production. The candidate who says "depends on how the consumer uses the data" is showing senior judgement.

Probe 2 — "International phone numbers — +44 20 7946 0958 becomes 442079460958. Is that right?"

Right for digit-only canonicalisation. Wrong if downstream code assumes US 10-digit phones. The senior fix: keep both columns — phone_raw for display and phone_e164 formatted to E.164 (the international ITU standard: +CCXXXXXXXXX). Libraries like phonenumbers do this properly with country-code inference. Stripping non-digits is the cheap-and-cheerful version; production wants E.164.

Probe 3 — "What's wrong with .apply() for the cleaning steps?"

Standard answer: row-by-row Python is slow. The pipeline uses vectorised string methods (.str.lower, .str.replace) which run in C. A naive df['email'].apply(lambda x: x.lower().strip()) would be 20-50× slower on a 10 M-row signup table. The senior candidate names .str accessor methods explicitly as the vectorised path.

Probe 4 — "Email validation — your function lowercases but doesn't validate. What if email is 'not_an_email'?"

Out of scope for cleaning; in scope for downstream validation. A common production pattern: cleaning is format-normalisation, validation is a separate step (often a regex match against RFC 5322 simplified, or a deliverability check via SendGrid). Mixing them tends to under-clean (skip the lower() because validation will run anyway) or over-validate (regex in the ETL chokes on weird-but-valid emails). Keep them separate; the cleaning function above is correct as a single-responsibility unit.

Probe 5 — "Datadog tier boundaries — $100 and $1000. Where do they come from?"

Empirically. Look at the historical distribution of signup amounts; the bin boundaries are where the cohort behaviour bifurcates. If $1001 customers convert at the same rate as $999 customers, the boundary is in the wrong place. Re-tune quarterly from the cohort analysis. Hard-coded thresholds without a re-tune schedule are a tech-debt smell.

Probe 6 — "How would you incrementalise this for daily delta loads?"

Process today's partition, append to the cleaned table. Idempotency is paramount: if today's partition is re-processed (e.g. due to a backfill), the previous day's rows must be replaced, not duplicated. The dbt pattern: {{ config(materialized='incremental', unique_key='id') }} — same algorithm, warehouse-native execution, MERGE-based upsert.

Probe 7 — "How would you observe DQ in production?"

Four metrics, emitted as Datadog gauges: (1) etl.rows_in — input size, expected to grow steadily; (2) etl.rows_dropped — drop rate, alert at > 10 %; (3) etl.tier_distribution{tier=high} — proportion of high-tier signups, alert on drift; (4) etl.bad_amount_rate — fraction of "not-a-num" coercions, alert if the landing page form broke and started sending strings.
