Cumulative LTV (lifetime value) curves by cohort are the cornerstone metric of subscription and marketplace businesses. The interviewer is probing whether you can (1) identify each user's signup cohort as the week of their first activity, (2) compute a per-cohort-per-offset activity summary, and (3) generate a running cumulative without window functions — instead using a semi-cartesian self-join of the cohort aggregates to accumulate all weeks up to the current offset.

2. The SQL — cohort aggregation + semi-cartesian self-join for running totals

WITH first_week AS (
  SELECT user_id, MIN(week) AS first_week FROM wau GROUP BY user_id
),
wau_d AS (
  SELECT w.week, w.user_id, w.inc_amt, f.first_week
  FROM wau w JOIN first_week f ON w.user_id = f.user_id
),
cohorts AS (
  SELECT first_week, (week - first_week)/7 AS weeks_since_first,
         COUNT(DISTINCT user_id) AS users, SUM(inc_amt) AS inc_amt
  FROM wau_d GROUP BY 1,2
),
cohort_sizes AS (
  SELECT first_week, users FROM cohorts WHERE weeks_since_first = 0
)
SELECT c1.first_week, c1.weeks_since_first, c1.users,
       cs.users AS cohort_num_users,
       ROUND(1.0*c1.users/cs.users, 3) AS retained_pctg,
       SUM(c2.inc_amt) AS cum_amt,
       ROUND(1.0*SUM(c2.inc_amt)/cs.users, 2) AS cum_amt_per_user
FROM cohorts c1
JOIN cohort_sizes cs ON cs.first_week = c1.first_week
JOIN cohorts c2 ON c1.first_week = c2.first_week AND c2.weeks_since_first <= c1.weeks_since_first
GROUP BY 1,2,3,4
ORDER BY 1,2;

The trick is the final JOIN: c2.weeks_since_first <= c1.weeks_since_first pulls in all earlier offsets for the same cohort, and the outer SUM(c2.inc_amt) accumulates them into a running total without any window function.

4. Strategic explanations

The semi-cartesian self-join as a running total

The join condition c2.weeks_since_first <= c1.weeks_since_first within the same cohort creates a triangular pairing: for the row at offset 2, c2 contributes offsets 0, 1, and 2. Summing c2.inc_amt then aggregating gives the cumulative through each offset. This is the same logic as a triangular self-join for running sums, but applied at the cohort-aggregate level rather than the raw event level — far cheaper because cohorts has only one row per (cohort, offset) pair.

Why date arithmetic gives weeks cleanly

(week - first_week) / 7 works because both columns are DATE in PostgreSQL, so subtraction gives an integer number of days; integer division by 7 then gives the zero-based week offset. The schema guarantees each row's week is already a Monday (or consistent weekly boundary), so the division is always exact. If dates were arbitrary timestamps, you would use DATE_TRUNC first. In BigQuery use DATE_DIFF(week, first_week, WEEK).

cohort_sizes at offset 0

The cohort size (denominator for retention and per-user metrics) is the number of distinct users at offset 0. Defining it from the cohorts CTE where weeks_since_first = 0 is clean and consistent. A common mistake is re-counting from the raw table, which can give different numbers if the same user appears more than once in the same week.

retained_pctg at offset 2 for Jan 06 cohort

User B was absent at offset 1 but returned at offset 2, so the users count jumps from 1 back to 2, yielding retained_pctg = 1.000 — which looks surprising but is correct. In product analytics this week-over-week count is "weekly active users in cohort" not "strictly retained since week 0"; if the question wanted monotonically decreasing retention, one would track users who were active at every week up to the current offset, which is a harder query.

Dialect and performance notes

The semi-cartesian self-join scales as O(k²) in the number of offsets per cohort — typically small (tens of weeks). An equivalent using SUM(...) OVER (PARTITION BY first_week ORDER BY weeks_since_first ROWS UNBOUNDED PRECEDING) on the cohorts CTE is often cleaner and more readable in PostgreSQL, BigQuery, and Snowflake, but the self-join form is portable to MySQL 5.x and SQLite which lack window functions. Both approaches are valid; mentioning both earns the senior mark.

Senior signal: build cohort aggregates first (one row per cohort per offset), then use a semi-cartesian self-join on c2.weeks_since_first <= c1.weeks_since_first to accumulate running totals without window functions; anchor cohort size at offset 0; and note the window-function alternative (SUM OVER ROWS UNBOUNDED PRECEDING) as the more readable production version.
