A finance analyst opens the orders ledger and asks the simplest revenue question there is: "on average, how much does an order bring in?" The query is a single AVG over order_total. It looks trivial, but the interviewer is watching whether you know what AVG silently does with NULLs and cancelled orders — the difference between an honest number and a misleading one.

1. The source data — orders

Two edge rows: order 3 has a real 0.00 total, and order 5 has a NULL total. They are not treated the same.

2. The SQL — a single aggregate over the column

SELECT AVG(order_total) AS avg_order_value
FROM orders;

3. The resulting analysis

avg_order_value
159.15

The math: (120 + 340.50 + 0 + 75.25 + 260) / 5 = 795.75 / 5 = 159.15. Six rows exist, but the divisor is 5, not 6 — the NULL row was dropped entirely. The 0.00 row, by contrast, was kept and pulled the average down.

4. Strategic explanations

Why AVG ignores NULL but counts zero

AVG(x) is defined as SUM(x) / COUNT(x) — and COUNT(x) on a column counts only non-NULL values. A NULL contributes to neither the sum nor the count, so it is invisible. A genuine 0.00 contributes 0 to the sum and 1 to the count, so it drags the mean toward zero. Knowing that NULL and zero diverge here is the whole point of the question.

The status trap — should cancelled orders even be in the average?

Order 3 is cancelled with a zero total. Including it answers "average over all order records"; excluding it answers "average value of a real sale." Those are different business questions. A senior answer states the assumption out loud and offers the filtered form:

SELECT AVG(order_total) AS avg_completed_order_value
FROM orders
WHERE status = 'completed';
The NULL trap — is missing the same as nothing?

If a NULL total really means "this order was free" or "amount not yet known," dropping it changes the answer. To treat NULLs as zero instead, average a coalesced expression: AVG(COALESCE(order_total, 0)) — that forces the divisor back to the full row count. Pick deliberately; do not let the default behaviour decide the business meaning.

Mean vs. median — the distribution follow-up

One large order (here, 340.50) skews a mean. Interviewers often follow with "what about a typical order?" — that is the median, which resists outliers. In Postgres: PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY order_total). Reporting AVG alongside the median, plus COUNT(*) for sample size, is the over-the-bar finish.

Common mistakes

SUM(order_total) / COUNT(*) divides by every row including NULLs — a different (smaller) number than AVG. And integer division bites in some dialects: if order_total were an integer column, SUM/COUNT would truncate. AVG over an integer column returns a numeric in Postgres but an integer in older MySQL — cast when in doubt.

Senior signal: state that AVG skips NULLs but counts genuine zeros, surface the cancelled-order assumption before being asked, and offer median via PERCENTILE_CONT as the outlier-resistant companion metric.
