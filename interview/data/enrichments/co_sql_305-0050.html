HR data quality audit. The new HRIS migration finished overnight; now the People-Ops team needs to find every employee whose supervisor_id points to a record that doesn't exist anymore. These are orphan rows — supervisors who were terminated and removed but whose reports still reference them. Performance reviews, org charts, payroll routing all break silently when these orphans linger.

2. The SQL — anti-join via LEFT JOIN + IS NULL

SELECT e.emp_id, e.name
FROM employees e
LEFT JOIN employees s ON e.supervisor_id = s.emp_id
WHERE e.supervisor_id IS NOT NULL    -- skip the CEO (legitimate NULL)
  AND s.emp_id IS NULL;              -- supervisor row didn't exist

4. Strategic explanations

Why this is a "self-join"

The same table appears twice with different aliases (e for the report-side, s for the supervisor-side). The hierarchy is encoded in a single table via the recursive FK supervisor_id → emp_id. Self-joins are the universal answer for any single-table hierarchy.

The "anti-join" pattern — LEFT JOIN ... IS NULL

The trick is doing a LEFT JOIN (which keeps unmatched left rows with NULLs on the right) and then filtering WHERE s.emp_id IS NULL. The result: rows where the join failed = rows where the supervisor doesn't exist. This is the canonical SQL idiom for "find rows in A with no matching row in B".

Equivalent forms (different performance profiles):

-- Anti-join via NOT EXISTS — often the fastest in modern engines
SELECT e.emp_id, e.name
FROM employees e
WHERE e.supervisor_id IS NOT NULL
  AND NOT EXISTS (SELECT 1 FROM employees s WHERE s.emp_id = e.supervisor_id);

-- Anti-join via NOT IN — readable but DANGEROUS with NULLs
SELECT emp_id, name
FROM employees
WHERE supervisor_id IS NOT NULL
  AND supervisor_id NOT IN (SELECT emp_id FROM employees);
-- ↑ if any emp_id is NULL, NOT IN returns NULL → row excluded → SILENT BUG
The CEO trap — handle the legitimate NULL

Without the WHERE e.supervisor_id IS NOT NULL clause, the CEO (with NULL supervisor by design) gets flagged as an orphan. The query treats the CEO's NULL as "missing supervisor" when it should be "no supervisor expected". Always handle the legitimate-NULL boundary case explicitly.

Why NOT IN is unsafe

If any row in the inner subquery has emp_id = NULL, the NOT IN predicate evaluates to NULL (not FALSE) for every outer row → the WHERE filter rejects all of them → the query returns zero rows. This is a real production bug; never use NOT IN against a NULL-able column. Use NOT EXISTS or LEFT JOIN ... IS NULL.

The follow-up: also flag self-references

What if Drew's row says supervisor_id = E_04 (himself)? That's a different bug — circular reference. Add another check:

WHERE e.supervisor_id IS NOT NULL
  AND (s.emp_id IS NULL OR e.supervisor_id = e.emp_id)
The over-the-bar finish: full hierarchy validation in one pass

Production HR data-quality runs check four things at once: (1) orphan supervisor, (2) self-reference, (3) supervisor terminated but record still active, (4) circular cycle in the chain (A → B → A). Each is a join condition; modern HRIS pipelines run all four nightly and PagerDuty if any returns rows.

Senior signal: name the self-join + anti-join patterns explicitly, contrast LEFT JOIN IS NULL vs NOT EXISTS vs NOT IN with the NULL-trap warning, and pre-empt the legitimate-NULL boundary (CEO row). The four-check HR DQ pipeline is the over-the-bar finish.
