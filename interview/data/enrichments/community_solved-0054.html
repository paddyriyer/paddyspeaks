A community dataset stores ranges as pairs — start_range and end_range — and asks you to expand them into the individual numbers they cover. Turning a compact range into one row per value is a routine ETL move: think of unpacking date ranges into days, or seat ranges into seats. The interviewer wants to see a number-generator — either a recursive CTE or a series function — rather than a hardcoded list.

1. The source data — sequence_tab

start_rangeend_range(narrative)
13covers 1,2,3
710covers 7,8,9,10 — note the gap (4,5,6 absent)

The two ranges do not touch — there is a gap between 3 and 7. That gap is the edge case: it exposes whether the query expands each range on its own or, as written, spans one continuous block from the global minimum to the global maximum.

2. The SQL — recursive CTE counting from MIN to MAX

WITH RECURSIVE bounds AS (
  SELECT MIN(CASE WHEN start_range::text ~ '^-?[0-9]+$' THEN start_range::int END) AS lo,
         MAX(CASE WHEN end_range::text   ~ '^-?[0-9]+$' THEN end_range::int   END) AS hi
  FROM sequence_tab
),
rcte(n) AS (
  SELECT lo FROM bounds
  UNION ALL
  SELECT n + 1 FROM rcte, bounds WHERE n < hi
)
SELECT n FROM rcte;

bounds collapses the table to one row holding the smallest start and the largest end. The recursive rcte then walks lo, lo+1, lo+2, … until it reaches hi.

The output is the unbroken sequence 1…10. Note 4, 5, 6 appear even though no stored range contains them — this query bridges MIN(start) to MAX(end), it does not expand each range independently.

4. Strategic explanations

How a recursive CTE generates numbers

A recursive CTE has two arms joined by UNION ALL: an anchor (SELECT lo FROM bounds — the seed row) and a recursive arm (SELECT n+1 ... WHERE n < hi) that feeds on the previous iteration's output. Each pass adds one row; the WHERE is the termination guard. Drop it and the recursion never stops — Postgres would eventually error or hang.

The gap behaviour — one block, not per-range expansion

This solution does not answer "expand each range" — it answers "list every integer from the overall lowest start to the overall highest end". With non-contiguous ranges it invents the in-between values (4, 5, 6). If the spec wants only numbers truly inside some stored range, expand each row instead: SELECT t.start_range, gs FROM sequence_tab t, LATERAL generate_series(t.start_range::int, t.end_range::int) gs — that skips the gap entirely. Knowing which interpretation the question wants is the senior judgement call.

The simpler Postgres idiom — generate_series

For a single span, the recursive CTE is overkill: SELECT generate_series(MIN(start_range::int), MAX(end_range::int)) FROM sequence_tab produces the same 1…10 in one line. Recursive CTEs earn their keep for hierarchies and graph walks; for a plain integer run, generate_series is clearer and faster. Other dialects: SQL Server uses a recursive CTE (or GENERATE_SERIES in 2022+), Oracle CONNECT BY LEVEL, MySQL 8 a recursive CTE.

The TEXT-cast guard and recursion limits

start_range/end_range are TEXT, so the CASE ... ~ '^-?[0-9]+$' regex validates each value is an integer before ::int — a stray non-numeric string would otherwise abort the query. Also watch the blast radius: if hi - lo is huge, the CTE materialises that many rows. Postgres has no default recursion cap, so a bad hi can exhaust memory; production code adds a sanity bound like WHERE n < hi AND n < lo + 1000000.

Senior signal: note that this query bridges MIN(start)…MAX(end) as one block — it invents the gap values — and offer the per-range LATERAL generate_series expansion if the spec actually wants only stored values. For a single contiguous span, prefer plain generate_series over a recursive CTE.
