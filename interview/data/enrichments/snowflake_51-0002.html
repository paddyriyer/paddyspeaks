An e-commerce analytics team wants a single flat result showing each order line alongside the customer name, product name, and the computed line total — a four-table join that forms the backbone of almost every order-analysis report. The interviewer is probing whether you understand how to chain joins correctly, where each ON clause sits, and what happens if any link in the chain uses the wrong key.

2. The SQL — four-table inner join chain

SELECT c.name      AS customer,
       o.order_id,
       p.name      AS product,
       oi.qty,
       oi.qty * p.price AS line_total
FROM customers   c
JOIN orders      o  ON c.customer_id = o.customer_id
JOIN order_items oi ON o.order_id    = oi.order_id
JOIN products    p  ON oi.product_id = p.product_id
ORDER BY o.order_id;

Each JOIN … ON clause connects the preceding derived result to a new table. The computed column oi.qty * p.price folds quantity and unit price into a single line total without requiring a subquery.

4. Strategic explanations

Why inner joins are correct here

All four JOINs are inner joins, meaning a row is included only if it has a match in every table. That is the right choice when the business definition of an "order line" requires that the customer, order, item, and product all exist. An orphaned order_item row with no matching product would silently disappear — which is usually the correct behaviour for an anomalous row. If the requirement were "show all order_items even when the product has been deleted," a LEFT JOIN products at the last step would be needed, with COALESCE(p.name, 'UNKNOWN') for the name and COALESCE(p.price, 0) for the computation.

Join key correctness — the most common interview error

The key mistake candidates make is joining on the wrong column — for instance writing JOIN order_items oi ON c.customer_id = oi.order_id (crossing customer ID with order ID). A production schema with integer surrogate keys makes this easy to get wrong because the types match even when the semantics do not. Always trace the primary-key → foreign-key relationship explicitly: customers.customer_id → orders.customer_id, then orders.order_id → order_items.order_id, then order_items.product_id → products.product_id.

Computed column: arithmetic in the SELECT list

oi.qty * p.price AS line_total is evaluated after all joins are resolved, so both operands are guaranteed to be non-NULL (given inner joins). If price could be NULL (e.g., a price not yet set), wrap it: oi.qty * COALESCE(p.price, 0). In Snowflake and PostgreSQL, multiplying an integer by a NUMERIC promotes the result to NUMERIC; in MySQL the result type depends on the column declarations — explicit CAST can prevent surprise truncation.

Performance: join order and optimizer hints

Modern optimizers (Snowflake, Postgres, BigQuery) choose the join order themselves based on statistics. However, the written order matters for readability and for databases with weaker optimizers (older MySQL). A useful heuristic: join the most restrictive filter first to reduce cardinality early. On large tables in Snowflake, a cluster key on orders(customer_id) and order_items(order_id) enables micro-partition pruning for the join columns.

Dialect notes

The JOIN … ON syntax is fully portable across Snowflake, PostgreSQL, BigQuery, and MySQL. The arithmetic expression in the SELECT list is also universal. BigQuery users working with nested data (where order_items might be an ARRAY inside the orders table) would instead use CROSS JOIN UNNEST(o.items) AS oi — a structurally different but semantically equivalent unnest pattern rather than a separate table join.

Senior signal: trace each join key pair explicitly (PK → FK), explain why inner vs LEFT joins differ in what rows survive, note that a NULL-able price column requires COALESCE before arithmetic, and mention that Snowflake's optimizer picks join order from statistics — but clustering keys on the join columns make micro-partition pruning effective at scale.
