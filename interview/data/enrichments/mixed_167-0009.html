A fulfilment system stores address parts in separate columns — street, city, state, zip — and the shipping label needs one clean line. The query uses CONCAT_WS to join the fields with a comma separator. The interviewer is probing whether you know how CONCAT_WS handles missing fields, because real customer data is full of NULL address components.

1. The source data — customers

customer_idstreetcitystatezip
112 Elm StAustinTX78701
299 Oak AveRenoNV89501
3NULLSeattleWA98101
45 Pine RdNULLNULL10001
57 Birch LnMiamiFLNULL

Rows 3, 4, and 5 each have a different NULL component — the test is whether the output degrades gracefully or shows stray commas.

2. The SQL — join fields with CONCAT_WS

SELECT customer_id,
       CONCAT_WS(', ', street, city, state, zip) AS full_address
FROM customers;

3. The resulting analysis

customer_idfull_address
112 Elm St, Austin, TX, 78701
299 Oak Ave, Reno, NV, 89501
3Seattle, WA, 98101
45 Pine Rd, 10001
57 Birch Ln, Miami, FL

The key result: row 3 has no leading ", ", row 4 reads "5 Pine Rd, 10001" with no double comma, row 5 has no trailing separator. CONCAT_WS simply skipped each NULL and did not emit a separator for it.

4. Strategic explanations

Why CONCAT_WS skips NULLs cleanly

CONCAT_WS(sep, a, b, c, ...) — "concatenate with separator" — joins only the non-NULL arguments and inserts the separator between surviving values, never around them. That is exactly the behaviour an address line needs: a missing field collapses without leaving a gap or a dangling comma. It is purpose-built for assembling delimited strings from optionally-missing parts.

The contrast that makes this question — || and plain CONCAT

The naive approach, street || ', ' || city || ', ' || state || ', ' || zip, breaks badly. In standard SQL, || with a NULL operand makes the entire expression NULL — one missing field blanks the whole address. Plain CONCAT(...) treats NULL as an empty string, so you avoid the total NULL but get stray separators: row 3 would render ", Seattle, WA, 98101" with a leading comma. CONCAT_WS is the only one of the three that handles both the separator and the NULL together.

The trap CONCAT_WS does NOT solve — empty strings

CONCAT_WS skips NULL but not the empty string ''. If state were stored as '' rather than NULL, you would still get a double comma. Real-world data mixes both. Defend with NULLIF(TRIM(col), '') on each field to fold blanks into NULLs first:

CONCAT_WS(', ',
  NULLIF(TRIM(street), ''),
  NULLIF(TRIM(city),   ''),
  NULLIF(TRIM(state),  ''),
  NULLIF(TRIM(zip),    ''))
Dialect notes

CONCAT_WS exists in PostgreSQL, MySQL, SQL Server (2017+), Spark, and BigQuery, with consistent NULL-skipping semantics. SQLite has no CONCAT_WS — you build the equivalent there with || plus explicit COALESCE on every part, or use the newer concat_ws in very recent builds. Always verify the target engine before relying on it.

Common mistakes

Forgetting the separator argument — the first argument is always the delimiter, not data. Passing the separator as a column by accident shifts everything. And note zip here is text; if it were an integer, CONCAT_WS still coerces it to string, but a leading-zero ZIP like 01234 would have lost its zero long before reaching the function — store ZIPs as text.

Senior signal: choose CONCAT_WS precisely because it joins with a separator and skips NULLs without stray commas — then note its blind spot, empty strings, and pre-clean each field with NULLIF(TRIM(...), '').
