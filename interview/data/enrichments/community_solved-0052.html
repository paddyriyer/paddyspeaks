A community dataset stores a column of IDs that should be a continuous sequence, and asks which values are missing. This is the data-quality check behind every "find the gaps in invoice numbers / ticket numbers / page IDs" task. The twist here is to also surface the next five IDs after the maximum, so the report is never empty — even a perfectly contiguous table produces actionable rows. The interviewer wants to see you generate a reference range and anti-join against it.

1. The source data — id_tab

id(narrative)
1start
2…
43 is missing
5…
86 and 7 are missing
109 is missing; MAX(id) = 10

Six rows with three interior gaps (3; 6–7; 9). MAX(id) is 10, so the report must also include 11–15 — the five IDs that would come next.

2. The SQL — generate_series plus an anti-join

WITH bounds AS (
    SELECT MAX(id) + 5 AS upper_bound FROM id_tab
)
SELECT g.id AS missing_id
FROM bounds,
     LATERAL generate_series(1, bounds.upper_bound) AS g(id)
LEFT JOIN id_tab t ON t.id = g.id
WHERE t.id IS NULL
ORDER BY g.id;

generate_series(1, MAX(id)+5) manufactures the complete reference range; the LEFT JOIN ... WHERE t.id IS NULL keeps only generated values with no match in the table — the gaps.

The three interior gaps (3, 6, 7, 9) plus the five-ID tail (11–15). Even if id_tab were perfectly contiguous, rows 11–15 would still appear — that is the point of MAX(id) + 5.

4. Strategic explanations

Why the anti-join finds the gaps

The reference range is the universe of expected IDs. A LEFT JOIN from that range to the real table preserves every generated value; rows that found no partner get NULLs on the table side. Filtering WHERE t.id IS NULL is the standard anti-join idiom — "rows on the left with nothing matching on the right". Equivalent forms are NOT EXISTS and EXCEPT; NOT IN is the one to avoid (see below).

LATERAL and the comma join to bounds

generate_series's upper argument is bounds.upper_bound — a value from another table expression — so the set-returning function must run per row of bounds. That is exactly what LATERAL licenses: a join input that references columns from a sibling to its left. Since bounds yields a single row, the effect is just "generate 1..upper_bound once". In Postgres a set-returning function in FROM that references an earlier table is treated as LATERAL implicitly, but writing it is clearer.

The NOT IN NULL trap

A tempting one-liner is SELECT id FROM generate_series(...) g WHERE g.id NOT IN (SELECT id FROM id_tab). If id_tab ever contains a NULL id, NOT IN against a list containing NULL evaluates to unknown for every row — the query returns zero rows, silently. LEFT JOIN / IS NULL and NOT EXISTS are NULL-safe; NOT IN is not. This is a top-five SQL interview gotcha.

The empty-table edge — MAX of nothing

If id_tab is empty, MAX(id) returns NULL, so upper_bound is NULL and generate_series(1, NULL) produces no rows — the report is empty. If the spec wants a baseline range even on an empty table, wrap it: COALESCE(MAX(id), 0) + 5. Also note generate_series is Postgres-specific; SQL Server uses a recursive CTE or GENERATE_SERIES (2022+), Oracle uses CONNECT BY LEVEL, MySQL 8 uses a recursive CTE.

Senior signal: frame this as "generate the expected range, anti-join the actual table", insist on LEFT JOIN / IS NULL or NOT EXISTS over NOT IN because a single NULL id makes NOT IN return nothing, and guard the empty-table case with COALESCE(MAX(id), 0).
