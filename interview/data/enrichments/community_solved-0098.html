A product team wants its strongest performers: products averaging at least 4 stars, ranked by total sales, with average rating and review count as tie-breakers. The interviewer is probing a clean two-step — aggregate per product in a CTE, then rank and filter — and whether you order your multi-key sort to match the stated priority.

Two edge cases. Product 4 averages 3.0 — below the 4-star bar — so even with the largest single sale ($500) it must be filtered out. Products 2 and 5 both average a perfect 5.0, so the tie-breaker on total sales decides their order.

2. The SQL — aggregate in a CTE, then rank

with t1 as (
  select productid, avg(reviewrating) as avg_review,
      count(reviewid) as review_count,
      sum(saleamount) as total_sales
  from product_reviews
  group by productid
)
select productid,
    avg_review,
    review_count,
    total_sales,
    rank() over (order by total_sales desc, avg_review desc, review_count desc) as product_rank
from t1
where avg_review >= 4;

4. Strategic explanations

Why the CTE comes first

You cannot reference an aggregate alias like avg_review in a WHERE clause on the same query level — WHERE is evaluated before GROUP BY finishes. The t1 CTE materialises the per-product aggregates, and the outer query then treats them as ordinary columns it can both filter (WHERE avg_review >= 4) and rank. This is the standard "aggregate then post-process" shape.

WHERE here is the right tool, not HAVING

Because the filter runs in the outer query against an already-aggregated column, a plain WHERE avg_review >= 4 is correct and idiomatic. HAVING AVG(reviewrating) >= 4 would be the equivalent if you filtered inside the aggregating query itself. Both work; splitting aggregation and filtering across CTE boundaries keeps each step readable.

Filter-then-rank vs rank-then-filter — an important ordering choice

Here the WHERE avg_review >= 4 and the RANK() are in the same SELECT. SQL applies WHERE before window functions, so RANK() only sees the four qualifying products — product 4 never enters the ranking and consumes no rank number. If the spec wanted ranks assigned across all products and then the list trimmed to 4-star ones, you would rank in the CTE and filter outside it. The two readings give different rank numbers; confirm which the business wants.

Multi-key sort must mirror the stated priority

ORDER BY total_sales DESC, avg_review DESC, review_count DESC encodes the priority literally: sales first, rating as the first tie-break, review count as the second. SQL evaluates keys left to right, so the order of the columns is the business rule. Swapping them would silently rank by the wrong primary metric.

RANK and the genuine-tie caveat

If two products tie on all three keys, RANK() gives them the same number and skips the next (e.g. 1, 2, 2, 4). DENSE_RANK() would not skip. With distinct total_sales in this dataset no tie survives, but state which semantics the leaderboard wants. Also note SUM(saleamount) repeats the sale figure across a product's review rows — if SaleAmount is a per-product attribute rather than per-review, summing double-counts it; an interviewer will want you to question that schema.

Senior signal: use a CTE so the aggregate can be filtered and ranked as a plain column, recognise that WHERE before the window means excluded products consume no rank number (vs rank-then-filter), order the multi-key sort to mirror the stated priority exactly, and question whether SUM(saleamount) double-counts a per-product value.
