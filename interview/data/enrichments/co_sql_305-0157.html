Elastic's revenue-ops team wants total sales broken down by region and by category — a two-dimensional grid for the finance pack. The query is a textbook GROUP BY, but grouping on two columns is where candidates reveal whether they understand that the grouping key is the combination of values, not each column separately.

1. The source data — sales

regioncategoryamount
WestElectronics100
WestElectronics200
WestApparel50
EastElectronics300
EastApparelNULL
EastApparel75

Six rows collapse into four groups — one per distinct (region, category) pair. The East/Apparel group has a NULL amount: that row is what tests whether you know how SUM treats missing values.

2. The SQL — GROUP BY region, category

SELECT region, category, SUM(amount) FROM sales GROUP BY region, category;

The grouping key is the pair (region, category). Every row whose pair matches lands in the same bucket; SUM(amount) totals that bucket.

3. The resulting analysis

regioncategorysum
EastElectronics300
WestElectronics300
EastApparel75
WestApparel50

Four groups, as expected. West/Electronics sums 100 + 200 = 300. East/Apparel sums NULL + 75 and reports 75, not NULL — SUM skips the NULL row entirely. Note also there is no ORDER BY, so the row order shown is whatever the planner emitted — it is not guaranteed.

4. Strategic explanations

The grouping key is the tuple, not the columns separately

The most common conceptual error is reading GROUP BY region, category as "group by region, then somehow also by category". It is one operation: rows are bucketed by the distinct combination of values. Two regions × two categories yields up to four groups; with three regions and four categories you would see up to twelve. The number of output rows equals the number of distinct pairs that actually occur in the data — empty combinations simply do not appear.

The select-list rule — every non-aggregate column must be grouped

In standard SQL, any column in the SELECT list that is not inside an aggregate must appear in GROUP BY. Here both region and category are grouped, so they are legal. Adding a bare amount or order_id to the select list would be an error in Postgres:

-- ERROR: column "sales.order_id" must appear in the GROUP BY clause
SELECT region, category, order_id, SUM(amount) FROM sales GROUP BY region, category;

MySQL historically allowed this and returned an arbitrary value for the ungrouped column — a notorious source of silent bugs. Postgres rejects it outright. Knowing this rule, and why MySQL's old behaviour was dangerous, is a senior tell.

SUM and NULL — skipped, but an all-NULL group is NULL

SUM ignores NULL inputs, so East/Apparel correctly reports 75. The subtle trap: if every row in a group had a NULL amount, SUM returns NULL for that group, not 0. If the finance grid must show 0 for an empty-revenue cell, wrap it:

SELECT region, category, COALESCE(SUM(amount), 0) AS total
FROM sales GROUP BY region, category;
Name the aggregate, and order the output

The reference leaves the aggregate unnamed, so the column comes back as sum — fine ad hoc, brittle in a pipeline. Always alias it (AS total_amount). And because GROUP BY does not imply any sort, add an explicit ORDER BY region, category for a stable, readable grid:

SELECT region, category, SUM(amount) AS total_amount
FROM sales GROUP BY region, category
ORDER BY region, category;
When you also want subtotals — ROLLUP / GROUPING SETS

A finance pack usually wants per-region subtotals and a grand total alongside the cells. Do not UNION three queries — use ROLLUP:

SELECT region, category, SUM(amount)
FROM sales GROUP BY ROLLUP (region, category);

This adds a row per region with category = NULL (the region subtotal) and one all-NULL row (the grand total). GROUPING SETS gives finer control over which aggregation levels to emit. Reaching for these shows you know GROUP BY beyond the basic form.

Senior signal: describe the grouping key as the distinct tuple of values, recite the rule that every non-aggregate select column must be grouped (and why MySQL's lax version bred bugs), flag that an all-NULL group sums to NULL, and know that ROLLUP — not a pile of UNIONs — is how you add subtotals.
