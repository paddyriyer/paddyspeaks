An analytics interviewer hands you a deliberate constraint: compute the median per category, but with PERCENTILE_CONT, PERCENTILE_DISC, and every quantile function off the table. This is a classic "do you understand what the function does" probe — the median is just the middle of a sorted list, and the test is whether you can rebuild it from ROW_NUMBER and COUNT, handling both odd and even group sizes.

1. The source data — score

categoryvalue(narrative)
A10, 20, 30, 40, 50odd count (5) — one true middle
B5, 7, 9, 11even count (4) — two middles, must average
C100single-member group — median is the lone value

The three groups are chosen to stress the formula: category A has an odd count (the middle row), B has an even count (the average of two middle rows), and C is a single-row group — the degenerate case where the one value is trivially the median.

2. The SQL — ROW_NUMBER + COUNT, average the middle

WITH ranked AS (
  SELECT category, value,
         ROW_NUMBER() OVER (PARTITION BY category ORDER BY value) AS rn,
         COUNT(*)    OVER (PARTITION BY category)                  AS n
  FROM score
)
SELECT category,
       ROUND(AVG(value)::numeric, 2) AS median
FROM ranked
WHERE rn IN ((n + 1) / 2, (n + 2) / 2)   -- integer arithmetic handles odd & even
GROUP BY category
ORDER BY category;

Each row gets its rank within the category (rn) and the category's size (n). The filter rn IN ((n+1)/2, (n+2)/2) selects the middle — one row when n is odd, two rows when n is even — and AVG returns either the single middle value or the mean of the two.

3. The resulting analysis

categorymediansignal
A30.00odd: middle of 10..50
B8.00even: avg of 7 and 9
C100.00single row: value is its own median

For A (n=5), the formula picks rn IN (3, 3) — the same row twice — so AVG sees only row 3, value 30. For B (n=4), it picks rn IN (2, 3) — values 7 and 9 — averaging to 8. For C (n=1), rn IN (1, 1) selects the lone row 100.

4. Strategic explanations

The integer-division trick — one formula, both parities

The elegance is that (n+1)/2 and (n+2)/2 use integer division, which truncates. For odd n=5: (6)/2=3 and (7)/2=3 — identical, so IN matches one row. For even n=4: (5)/2=2 and (6)/2=3 — distinct, so it matches two adjacent rows. A single predicate covers both cases with no CASE on parity. That is the insight the interviewer wants verbalised.

Why the window functions, and why a CTE

ROW_NUMBER() OVER (PARTITION BY category ORDER BY value) imposes the sort order the median definition needs; COUNT(*) OVER (PARTITION BY category) gives every row the group size without collapsing rows. Both must be computed before the filter can run, which is why they live in a CTE: you cannot reference a window function inside the same query's WHERE clause — window functions are evaluated after WHERE.

NULL handling — the silent rank shift

If value contained NULLs, ORDER BY value places them last (Postgres default) and COUNT(*) still counts them — so n includes NULL rows and the middle index points at the wrong place. The statistically correct median ignores NULLs: filter WHERE value IS NOT NULL before ranking, or use COUNT(value) (which skips NULLs) instead of COUNT(*). Naming this is essential — the demo data has no NULLs, so the bug stays hidden.

Ties in value — ROW_NUMBER vs RANK

ROW_NUMBER is the right choice here even when values tie: the median formula needs positional indices 1..n with no gaps, and duplicate values must still occupy distinct positions. Using RANK would assign tied values the same rank, leaving holes in the sequence and breaking the (n+1)/2 lookup. The median cares about position in the sorted list, not about competition ranking.

Dialect notes — the function you were told not to use

The idiomatic answer in Postgres/Oracle is PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY value); SQL Server spells it PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY value) OVER (PARTITION BY category); MySQL 8+ has no percentile function at all, making this manual technique the only portable option there. Knowing the built-in exists — and that this hand-rolled version is what you fall back to when it is banned or absent — is the senior framing.

Senior signal: rebuild the median from ROW_NUMBER + COUNT using the integer-division pair (n+1)/2, (n+2)/2) so one predicate handles odd and even sizes, compute the window functions in a CTE because WHERE runs before them, and call out that NULLs must be excluded before ranking or the middle index silently shifts.
