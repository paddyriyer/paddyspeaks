A Community crypto-exchange wants a 2020 trading report broken down by mining algorithm and quarter: a wide grid with one row per algorithm and four columns Q1–Q4. The catch is that volume is stored as messy text — thousands separators, stray symbols — and the algorithm lives on a separate coins table. The interviewer is watching whether you join cleanly, pivot quarters with conditional aggregation, and defuse the dirty-text column without crashing.

The coins table maps code → algorithm: BTC→SHA-256, LTC and DOGE→Scrypt, ETH→Ethash. Three edge cases are deliberate: the 2019 row must be excluded by the year filter, the 'N/A' volume must not crash the numeric cast, and LTC + DOGE both being Scrypt means the algorithm-level totals must aggregate across coins.

2. The SQL — join + clean + quarter pivot

SELECT
    C.algorithm,
    SUM(CASE WHEN C.Quarter = 1 THEN NULLIF(regexp_replace(C.volume::text, '[^-.0-9]', '', 'g'), '')::numeric ELSE 0 END) AS transactions_Q1,
    SUM(CASE WHEN C.Quarter = 2 THEN NULLIF(regexp_replace(C.volume::text, '[^-.0-9]', '', 'g'), '')::numeric ELSE 0 END) AS transactions_Q2,
    SUM(CASE WHEN C.Quarter = 3 THEN NULLIF(regexp_replace(C.volume::text, '[^-.0-9]', '', 'g'), '')::numeric ELSE 0 END) AS transactions_Q3,
    SUM(CASE WHEN C.Quarter = 4 THEN NULLIF(regexp_replace(C.volume::text, '[^-.0-9]', '', 'g'), '')::numeric ELSE 0 END) AS transactions_Q4
FROM (
    SELECT b.algorithm, a.volume, a.Quarter
    FROM (
        SELECT coin_code, volume, EXTRACT(QUARTER FROM dt)::int AS Quarter
        FROM transactions
        WHERE EXTRACT(YEAR FROM dt)::int = 2020
    ) A
    JOIN (SELECT algorithm, code FROM coins) B
      ON A.coin_code = B.code
) C
GROUP BY C.algorithm;

Innermost: filter transactions to 2020 and derive the quarter from dt. Next: join to coins to attach the algorithm. Outermost: group by algorithm and pivot, with four SUM(CASE WHEN Quarter = n ...) buckets.

4. Strategic explanations

The quarter pivot — conditional aggregation

There is no portable PIVOT for "turn four quarter values into four columns". The idiom is SUM(CASE WHEN Quarter = n THEN amount ELSE 0 END) once per quarter. The ELSE 0 matters here (unlike a label-pivot, where you want NULL): a quarter with no transactions for an algorithm should read 0, not NULL — see SHA-256's empty Q3/Q4. Each row falls into exactly one quarter bucket; the others receive 0; SUM per group totals them.

Cleaning dirty text — regexp_replace + NULLIF + cast

The chain NULLIF(regexp_replace(volume, '[^-.0-9]', '', 'g'), '')::numeric is three defensive steps stacked:

  • regexp_replace(volume, '[^-.0-9]', '', 'g') — strip every character that is not a digit, minus, or dot. '1,200.50' → '1200.50'; 'N/A' → '' (empty string).
  • NULLIF(..., '') — turn that empty string into NULL. Without this, ''::numeric throws "invalid input syntax for type numeric" and kills the whole query.
  • ::numeric — now safe: a real number casts, a NULL stays NULL.

A NULL inside SUM is silently skipped, so the 'N/A' row neither crashes nor distorts the total — it contributes nothing. This "strip → nullify-empty → cast" pattern is the senior way to survive a text column that should have been numeric.

Aggregating across coins — the grain of the question

The report is per algorithm, not per coin. LTC and DOGE are different coins but the same Scrypt algorithm, so their volumes must merge. GROUP BY C.algorithm does exactly that — the Scrypt row is the sum of LTC and DOGE activity. A common mistake is grouping by coin_code (or by both), which would split Scrypt into two rows and answer a different question. Match the GROUP BY to the requested grain.

Deriving the quarter — and the year filter

EXTRACT(QUARTER FROM dt) returns 1–4 directly — cleaner than a hand-rolled CASE on month ranges. The WHERE EXTRACT(YEAR FROM dt) = 2020 filter is applied in the innermost subquery, before the join and pivot — correct, because it is a per-row predicate and filtering early shrinks the join input. One performance note: wrapping dt in EXTRACT(...) makes the predicate non-sargable — an index on dt cannot be used. On a large table a half-open range dt >= '2020-01-01' AND dt < '2021-01-01' is index-friendly and preferable.

Edge cases: missing coins, negative volume, the schema noise

The query uses an INNER JOIN to coins, so a transaction whose coin_code has no matching coin row is dropped — if unmapped coins should still surface (under an "Unknown" algorithm), switch to a LEFT JOIN with COALESCE(algorithm, 'Unknown'). The regex keeps a leading -, so a genuinely negative volume survives as a negative number — intended if refunds exist, worth flagging if not. Finally, the schema signature lists many columns (qq, yy, transaction_time, txn_type, status…) the solution never touches — ignore the noise and use only coin_code, dt, volume, and the coins mapping.

Senior signal: pivot quarters with SUM(CASE WHEN Quarter = n ... ELSE 0), defuse a dirty text column with the regexp_replace → NULLIF(...,'') → ::numeric chain so junk values become harmless NULLs, group strictly at the requested grain (algorithm, aggregating across coins), and note that wrapping the date in EXTRACT defeats an index where a half-open range would not.
