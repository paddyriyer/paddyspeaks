DeliveryHero's CRM team wants order frequency per customer to feed a loyalty model. "Count orders per customer" is a textbook GROUP BY — but the interviewer is testing whether you count all orders or only the ones that actually happened, whether you know customers with zero orders are invisible here, and whether COUNT(*) is the right counter.

1. The source data — orders

Customer 101 has three orders — but one is cancelled. That cancelled row is the interview: does "order count" mean orders placed or orders fulfilled?

2. The SQL — COUNT(*) per customer

SELECT customer_id, COUNT(*) AS orders FROM orders GROUP BY customer_id;

One group per customer_id; COUNT(*) tallies every row in each group.

Customer 101 shows 3 — the cancelled order is counted alongside the two completed ones. The result row order is arbitrary (no ORDER BY). And note who is missing: any customer who never ordered does not appear at all, because this table only knows about orders.

4. Strategic explanations

The cancelled-order decision

Counting 101 as a 3-order customer overstates loyalty if one order was cancelled. "How many orders" almost always means realised orders for a CRM signal. Surface the choice and filter explicitly:

SELECT customer_id, COUNT(*) AS orders
FROM orders
WHERE status = 'completed'
GROUP BY customer_id;
-- customer 101 now counts 2, not 3

Confirm with the interviewer: "placed or completed?" The filter changes the headline number more than the aggregate does.

Customers with zero orders are invisible — and why

This query is sourced from orders, so a customer who never ordered contributes no rows and is simply absent — not shown as 0. If the loyalty model needs every customer including the dormant ones, you must start from customers and left-join:

SELECT c.customer_id, COUNT(o.order_id) AS orders
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
GROUP BY c.customer_id;

Here the counter must be COUNT(o.order_id), not COUNT(*) — see the next note.

COUNT(*) vs COUNT(column) — the left-join trap

In the single-table query COUNT(*) and COUNT(order_id) agree, because every order row has an order_id. But after a LEFT JOIN, a customer with no orders still produces one row — with all order columns NULL. COUNT(*) would count that phantom row as 1; COUNT(o.order_id) ignores the NULL and correctly reports 0. Pick the column-form counter whenever an outer join is in play.

Distinct orders vs order lines

If orders were actually one row per order line (an order split across products), COUNT(*) counts lines, not orders, and inflates frequency. The honest counter is then COUNT(DISTINCT order_id). Always confirm the grain of the table before trusting COUNT(*).

Presentation: sort and threshold

A frequency report wants order — ORDER BY orders DESC puts your best customers on top. If the ask is "customers with more than 5 orders," that is a HAVING COUNT(*) > 5 filter (post-aggregation), never a WHERE.

Senior signal: raise the cancelled-order question and add a status filter on purpose, recognise that an orders-sourced query silently omits zero-order customers (and switch to a customers LEFT JOIN orders when they matter), and know that COUNT(*) overcounts NULL-extended rows after an outer join where COUNT(order_id) does not.
