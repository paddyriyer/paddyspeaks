A Community question on a content-moderation table: count how many distinct users flagged each video. Trust & Safety wants to know which videos drew complaints from many people versus one persistent complainer. The schema gives no user_id — only user_firstname and user_lastname — so the real test is constructing a person identity from name parts and counting people, not flag rows.

1. The source data — user_flags

flag_iduser_firstnameuser_lastnamevideo_id
f1MaryLeevidA
f2JohnDoevidA
f3MaryLeevidA
f4AnneKimvidB
NULLZoeFoxvidB
f5BobNgvidC

Two edge cases packed in. Mary Lee flagged vidA twice (f1 and f3) — that is one person, not two. And one vidB row has a NULL flag_id — a row that was never a real flag. Both must be handled or the counts lie.

2. The SQL — COUNT(DISTINCT concat(first,last))

select video_id,
        count(distinct (concat(user_firstname,user_lastname))) as user_count
from user_flags
where flag_id is not null
group by video_id

WHERE flag_id IS NOT NULL drops the non-flag row before counting. concat(user_firstname, user_lastname) builds a per-person key, and COUNT(DISTINCT …) counts distinct people per video.

3. The resulting analysis

video_iduser_count
vidA2
vidB1
vidC1

vidA is 2, not 3: Mary Lee's two flags collapse to one person, joining John Doe. vidB is 1: Anne Kim counts, but Zoe Fox's NULL-flag_id row was filtered out before counting. The DISTINCT and the WHERE each remove one inflation; drop either and vidA or vidB reads too high.

4. Strategic explanations

Why count people, not flags

"How many users flagged the video" asks for distinct people. COUNT(*) counts flag rows, so Mary Lee's two flags on vidA would score 2 for her alone — the metric becomes "number of flags", a different question that overstates community concern. COUNT(DISTINCT person) answers the one asked. Always reconcile the count with the noun in the question: "users" ⇒ distinct users.

The name-concatenation trap — a fragile identity

With no user_id, the solution improvises a key from name parts — pragmatic, but fragile two ways. First, collision: concat('Mary','Lee') and concat('Mar','yLee') both yield 'MaryLee'; without a separator, distinct people can merge. Use a delimiter that cannot appear in a name:

COUNT(DISTINCT user_firstname || '|' || user_lastname)

Second, NULL propagation: in standard SQL || with a NULL operand yields NULL, so a person missing a last name vanishes from the count — Postgres's concat() function avoids this by treating NULL as empty, but mixing the two behaviours is a known footgun. And two real different people can share a full name — the only true fix is a stable user id.

Why flag_id IS NOT NULL matters

A NULL flag_id means the row is not an actual flag — perhaps a placeholder or a join artefact. Counting it would credit vidB with a flagger who never flagged. The WHERE removes it pre-aggregation. Confirm the semantics, though: if NULL flag_id instead meant "flag pending", you might keep it. Either way, decide deliberately rather than letting NULLs drift into the count.

The videos-with-zero-flags gap

This query only ever returns videos that have at least one flag — a clean, never-flagged video produces no row at all. If the report should show every video with user_count = 0 for the quiet ones, you need a videos table and a LEFT JOIN:

SELECT v.video_id, COUNT(DISTINCT f.user_firstname || '|' || f.user_lastname) AS user_count
FROM videos v
LEFT JOIN user_flags f ON f.video_id = v.video_id AND f.flag_id IS NOT NULL
GROUP BY v.video_id;

Note the filter moves into the ON clause so the LEFT JOIN still keeps zero-flag videos.

Ordering and case sensitivity

There is no ORDER BY, so row order is undefined — add ORDER BY user_count DESC, video_id to surface the most-flagged videos first. And the name key is case-sensitive: 'mary lee' and 'Mary Lee' would count as two; normalise with lower() if the source data is inconsistent.

Senior signal: count distinct people not flag rows, treat name concatenation as a fragile identity (use a delimiter, beware NULL propagation, prefer a real user id), filter the NULL-flag_id non-flags deliberately, and note that never-flagged videos need a LEFT JOIN from a video table to appear.
