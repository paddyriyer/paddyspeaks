A sales team wants the dormant accounts — customers who exist in the system but have never placed an order — to target a win-back campaign. This is a set-difference (anti-join) problem, and the interviewer specifically wants the LEFT JOIN / IS NULL pattern: keep all customers, attach orders, and surface only the rows where no order attached.

C01 deliberately has two order rows — the duplicate-match edge case that a naive approach can mishandle. C02 and C05 have no matching order at all; they are the two customers the query must return. C04 has exactly one order — the boundary between "ordered" and "never ordered".

2. The SQL — LEFT JOIN then IS NULL anti-join

SELECT c.cust_code, c.cust_name, c.cust_country
FROM customer c
LEFT JOIN orders o ON o.cust_code = c.cust_code
WHERE o.cust_code IS NULL
ORDER BY c.cust_code;

The LEFT JOIN keeps every customer row, attaching matching orders where they exist. For a customer with no order, every right-side column — including o.cust_code — comes back NULL. WHERE o.cust_code IS NULL then keeps exactly those unmatched rows: the customers with no order.

4. Strategic explanations

How the LEFT JOIN / IS NULL anti-join works

A LEFT JOIN guarantees every left row appears at least once. When the join finds no match, it manufactures one output row with all right-side columns set to NULL. Filtering WHERE o.cust_code IS NULL selects precisely those manufactured rows — the left rows that had no partner. The result is set difference: customers minus customers-with-orders. The column you test for NULL must be one that is never legitimately NULL in a matched row — the join key o.cust_code is the safe choice, since a matched order always has a non-NULL cust_code.

The filter belongs in WHERE, not the ON clause

A subtle, common bug: putting AND o.cust_code IS NULL in the ON clause instead of WHERE. With a LEFT JOIN, conditions in ON only affect which rows match — non-matching left rows are still kept with NULLs. The anti-join filter must run after the join, in WHERE, to discard the matched rows. ON decides matching; WHERE decides the final row set. Getting this wrong silently returns every customer.

Why NOT IN was explicitly ruled out — the NULL trap

The spec bars NOT IN, and for a real reason. WHERE c.cust_code NOT IN (SELECT cust_code FROM orders) is dangerous if any row in orders has a NULL cust_code: NOT IN against a list containing NULL evaluates to unknown for every customer, so the query returns zero rows — a silent, total failure. The LEFT JOIN / IS NULL pattern (and NOT EXISTS) are immune to this because they compare keys row-by-row rather than against a set membership that NULL poisons.

Duplicates on the right side do not multiply the result

C01 has two order rows. In an inner join that customer would appear twice; here it does not matter, because the anti-join only keeps the unmatched customers and C01 is matched. The duplicate-match case is exactly why you select only customer columns and never aggregate — matched customers are dropped wholesale, so their fan-out is irrelevant. The result has one row per never-ordering customer, guaranteed.

Dialect — anti-join forms and EXCEPT

The LEFT JOIN / IS NULL pattern is portable to every SQL engine. NOT EXISTS (SELECT 1 FROM orders o WHERE o.cust_code = c.cust_code) is an equally correct, NULL-safe alternative and often optimises to the same anti-join plan — the spec just chose to test the join form here. EXCEPT (PostgreSQL, SQL Server) does set difference too, but operates on whole rows and requires matching column lists, so it is awkward when you want extra customer columns. For "rows in A not matched in B", the LEFT JOIN / IS NULL and NOT EXISTS idioms are the two to know.

Senior signal: implement "never placed an order" as a LEFT JOIN / IS NULL anti-join, test the join key for NULL (never a nullable payload column), keep the IS NULL filter in WHERE not ON, and explain that NOT IN is the trap — a single NULL in the subquery makes it return nothing — while NOT EXISTS is the equally safe alternative.
