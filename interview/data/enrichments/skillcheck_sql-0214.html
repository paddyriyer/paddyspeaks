The definition of a Senior Manager is quietly recursive — “a Manager who manages at least one Manager, but none of whose direct reports is itself a Senior Manager.” The set-operations approach untangles it in layers: first the Managers, then the Managers who manage a Manager, then — by subtraction — remove the ones who sit above the Senior Manager tier.

Four tiers run Alice (CEO) → Bob and Carol → Dave, Eve and Frank → the individual contributors. Bob’s team is deliberately mixed — Dave and Eve are Managers, but Karl is an IC.

2. The set-operations solution

WITH managers AS (
  SELECT DISTINCT manager_id AS mgr FROM employees WHERE manager_id IS NOT NULL
),
manages_a_manager AS (
  SELECT DISTINCT e.manager_id AS mgr
  FROM employees e
  JOIN managers m ON e.emp_id = m.mgr
  WHERE e.manager_id IS NOT NULL
),
senior_managers AS (
  SELECT mgr FROM manages_a_manager
  EXCEPT
  SELECT DISTINCT e.manager_id
  FROM employees e
  JOIN manages_a_manager mm ON e.emp_id = mm.mgr
  WHERE e.manager_id IS NOT NULL
)
SELECT sm_emp.emp_name AS senior_manager,
       COUNT(r.emp_id) AS direct_reports
FROM senior_managers sm
JOIN employees sm_emp ON sm_emp.emp_id = sm.mgr
JOIN employees r ON r.manager_id = sm.mgr
GROUP BY sm_emp.emp_id, sm_emp.emp_name
ORDER BY direct_reports DESC, senior_manager ASC;

managers is every id that appears as someone’s manager_id. manages_a_manager joins each Manager back to the table to collect their bosses — the people who manage a Manager. The final EXCEPT subtracts the bosses of those people: anyone who manages a manager-of-managers sits above the Senior Manager tier and is removed.

4. Strategic explanations

The exclusion is “manages a manager-of-managers”

“None of their reports is a Senior Manager” needs unpacking. A report is a Senior Manager exactly when that report manages a Manager. So the people to exclude are those who manage someone who manages a Manager — the second SELECT in the EXCEPT. Alice manages Bob, and Bob manages Dave (a Manager), so Alice is removed; Bob manages Dave and Eve, but neither of them manages a Manager, so Bob survives.

EXCEPT over NOT IN — and why it matters here

EXCEPT is a true set difference: it deduplicates and compares rows structurally, so a NULL on either side is handled cleanly. The obvious alternative, NOT IN (SELECT manager_id ...), is a trap — manager_id is nullable (Alice has none), and NOT IN against a list containing NULL never returns true. EXCEPT avoids that failure mode entirely.

The rule reaches two levels, not one

A frequent wrong answer stops at “manages a Manager” — one level down. That keeps every executive in the result, because a VP also manages a Manager. The definition is really two levels deep: manage a Manager, but not a manager-of-managers. The layered CTEs make both levels explicit and separately inspectable.

Mixed teams still qualify

Bob manages Dave and Eve (Managers) and Karl (an IC). He is still a Senior Manager: the rule asks for at least one Manager among the reports, not all of them. The final COUNT(r.emp_id) then counts every direct report — all three — not only the managerial ones.

Senior signal: a recursive-sounding hierarchy rule often unfolds into plain set operations — build each tier as a CTE and use EXCEPT to subtract the tier above. Reach for EXCEPT rather than NOT IN whenever the compared column is nullable.
