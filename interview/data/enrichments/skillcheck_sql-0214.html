A category taxonomy is a tree, and the textbook tool for walking a tree is a recursive CTE. But when the tree has a known, shallow maximum depth — here three tiers, root → sub-genre group → leaf genre — recursion is overkill. Two LEFT JOINs and a COALESCE flatten it into a plain (root, descendant) lookup.

Note the uneven depth: the Action branch reaches tier 3, but Comedy stops at tier 2. The flatten must report Comedy as its own deepest node.

2. The query — two LEFT JOINs and COALESCE

SELECT
  c1.category_id   AS root_id,
  c1.category_name AS root_name,
  COALESCE(c3.category_id,   c2.category_id,   c1.category_id)   AS descendant_id,
  COALESCE(c3.category_name, c2.category_name, c1.category_name) AS descendant_name
FROM categories c1
LEFT JOIN categories c2 ON c2.parent_id = c1.category_id
LEFT JOIN categories c3 ON c3.parent_id = c2.category_id
WHERE c1.parent_id IS NULL
ORDER BY root_id, descendant_id;

c1 is pinned to roots by WHERE c1.parent_id IS NULL. Each join steps one tier down. COALESCE then reads the chain right-to-left and keeps the deepest node that actually exists.

4. Strategic explanations

Why LEFT JOIN, not INNER

An INNER JOIN to c3 would drop every branch that has no tier-3 node. Comedy has no children, so its c3 is empty — an inner join would erase the (Film, Comedy) row entirely. LEFT JOIN keeps the row with c3 as NULL, and COALESCE falls back to c2.

What COALESCE is doing

COALESCE(c3, c2, c1) returns its first non-NULL argument. Listed deepest-first, it always yields the lowest node present on the branch: tier 3 if there is one, otherwise tier 2, otherwise the root itself. A root with no children at all would produce a single (root, root) row.

When this beats a recursive CTE — and when it does not

Use the join chain when depth is bounded and small: it is simpler to read, the planner optimizes plain joins well, and it is portable to engines with weak recursion support. Each extra tier costs exactly one more LEFT JOIN. Switch to WITH RECURSIVE the moment depth is unknown or unbounded — an org chart, a threaded comment tree — because you cannot pre-write a join per level you have not counted.

This returns leaves, not every ancestor pair

The query maps each root to its deepest nodes only — it deliberately omits intermediate rows like (Film, Action). That is exactly what a title-rollup needs, since titles attach to leaf genres. If you needed every (ancestor, descendant) pair, you would UNION the tiers or recurse.

Senior signal: when a hierarchy's depth is fixed and shallow, a static chain of LEFT JOINs collapsed by COALESCE(deepest, ..., root) is simpler, faster and more portable than WITH RECURSIVE. Recursion earns its cost only when the depth is genuinely unknown.
