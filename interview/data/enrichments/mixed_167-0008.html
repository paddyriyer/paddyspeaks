A payroll report must show every employee's bonus, and employees with no bonus should read 0, not a blank. The query wraps bonus in COALESCE to substitute a default for NULL. It is a one-function answer, but the interviewer is probing whether you know the difference between a NULL bonus and a genuine zero bonus — and whether you understand what NULLs do to the arithmetic downstream.

1. The source data — employees

emp_idnamebonus(narrative)
1Ava Lin5000has a bonus
2Ben ColeNULLbonus unknown / not set
3Cara Mott0genuine zero bonus
4Dan PopeNULLbonus unknown / not set
5Eli Voss12000has a bonus

The instructive contrast is Ben/Dan (NULL) versus Cara (0). They are different facts; the report will make them look identical.

2. The SQL — substitute a default with COALESCE

SELECT name, COALESCE(bonus, 0) AS bonus_amount
FROM employees
ORDER BY name;

Ben and Dan's NULLs became 0. Note that Ben's row and Cara's row now both read 0 — the report can no longer distinguish "no bonus recorded" from "bonus is genuinely zero."

4. Strategic explanations

Why COALESCE works

COALESCE(a, b, c, ...) returns the first non-NULL argument left to right. COALESCE(bonus, 0) yields bonus when it is present and 0 when it is NULL. It is ANSI-standard and short-circuits — once a non-NULL is found, the rest are not evaluated, which matters if a later argument is an expensive subquery.

The semantic trap — NULL is not zero

NULL means "unknown" or "not applicable"; 0 is a known quantity. Coalescing flattens that distinction. For a display total it is usually fine. But if someone later asks "how many employees have a bonus of exactly zero?", running that count over the coalesced column would wrongly include Ben and Dan. Coalesce at the presentation layer; keep the raw NULL in the stored data so the two cases remain separable.

Why this matters for SUM and AVG

Aggregates already skip NULLs, so SUM(bonus) ignores Ben and Dan, and AVG(bonus) divides by 3, not 5. If the business wants the average bonus per employee (treating "no bonus" as zero), you must coalesce before aggregating: AVG(COALESCE(bonus, 0)) forces the divisor to all 5. Choosing whether to coalesce changes the denominator — state which average you mean.

COALESCE vs. ISNULL vs. IFNULL vs. NVL

COALESCE is the portable, standard choice and takes any number of arguments. Dialect-specific two-argument variants exist — ISNULL (SQL Server), IFNULL (MySQL/SQLite), NVL (Oracle) — and they carry quirks: SQL Server's ISNULL truncates the result to the first argument's type, whereas COALESCE applies standard type precedence. Default to COALESCE.

Common mistakes

Comparing with = NULL instead of IS NULL — bonus = NULL is always unknown, never true. And type mismatch: COALESCE(bonus, 'N/A') mixes a numeric column with a text default and errors (or forces an unwanted cast). The fallback must be type-compatible with the column; 0 for numeric, '' for text.

Senior signal: use COALESCE for display but state out loud that NULL and zero are different facts — and note that coalescing before AVG changes the denominator, so the choice is a business decision, not a formatting one.
