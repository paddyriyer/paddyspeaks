LinkedIn's recruiting-tools team is filtering a candidate pool for a data-analytics role: they need everyone who lists both Python and SQL. The catch is the data model — skills are crammed into a single skills_array text column rather than a proper related table. The interview is whether you make the substring search work and whether you flag that this schema is the real problem.

1. The source data — candidates

candidate_idnameskills_array
1Anapython,sql,excel
2Benjava,sql
3Carapython,r
4Dansql,python,tableau
5Evepython

Only Ana and Dan have both skills. Note Dan lists them in the order sql,python — order must not matter. Ben has SQL but not Python; Cara and Eve have Python but not SQL. The "both required" logic is the whole question.

2. The SQL — two LIKE predicates, ANDed

SELECT candidate_id, name FROM candidates
WHERE skills_array LIKE '%python%'
  AND skills_array LIKE '%sql%';

Each LIKE '%...%' tests whether the substring appears anywhere in the text. Joining them with AND demands that both appear — the conjunction is what enforces "both skills".

3. The resulting analysis

candidate_idname
1Ana
4Dan

Ana and Dan match. Dan's reversed order (sql,python) is irrelevant — two independent substring tests do not care about position. Ben, Cara and Eve each have only one of the two skills and fall out.

4. Strategic explanations

Why AND, not a single pattern

You cannot express "both substrings, any order" in one LIKE pattern — LIKE '%python%sql%' only matches when "python" precedes "sql", so it would miss Dan. Two separate LIKE predicates joined by AND is the correct shape: each is order-independent, and the conjunction requires both. Using OR instead would return anyone with either skill — a different, much larger pool.

The substring trap — partial-word false matches

LIKE '%sql%' matches any text containing "sql" — including mysql, postgresql, nosql. A candidate listing only mysql would be wrongly flagged as having "sql". Likewise %python% would match a hypothetical micropython. The robust fix is to match whole tokens, not loose substrings:

-- normalise to delimiter-padded tokens, then match exactly
SELECT candidate_id, name FROM candidates
WHERE ',' || skills_array || ',' LIKE '%,python,%'
  AND ',' || skills_array || ',' LIKE '%,sql,%';

Wrapping the value in commas turns "is this an exact element" into a substring test that cannot match a partial word.

The real fix — the schema is wrong

The senior observation is that skills_array stuffing a list into one text column is the underlying defect. The correct model is a junction table, candidate_skills(candidate_id, skill), one row per skill. Then the query is exact, indexable, and clean:

SELECT candidate_id
FROM candidate_skills
WHERE skill IN ('python', 'sql')
GROUP BY candidate_id
HAVING COUNT(DISTINCT skill) = 2;

The HAVING COUNT(DISTINCT skill) = 2 idiom is the canonical "has all of these" pattern and scales to any number of required skills. Naming the schema as the root cause — while still solving the question as posed — is the over-the-bar answer.

If it really is a Postgres array

The column is named skills_array; if it is genuinely a text[] rather than a delimited string, LIKE is the wrong tool entirely — use the array-contains operator, which is exact and index-friendly:

SELECT candidate_id, name FROM candidates
WHERE skills_array @> ARRAY['python','sql'];

@> ("contains") tests that the left array includes every element of the right — no false partial matches, and a GIN index makes it fast.

Case sensitivity and whitespace

LIKE is case-sensitive in Postgres, so Python or SQL in mixed case would not match the lowercase patterns. Normalise with ILIKE or lower(skills_array). And free-text lists often carry stray spaces (python, sql) — the comma-padding trick above must account for ", " as well as ",", or trim first.

Senior signal: use two ANDed predicates so skill order does not matter, recognise that bare %sql% false-matches mysql/postgresql and fix it with token-exact matching, and name the real problem — a delimited skills column should be a junction table, where HAVING COUNT(DISTINCT skill) = N answers "has all of these" cleanly.
