A ride-sharing platform ingests customer profile updates nightly from a staging table. The merge rule is: if the customer exists in the target, update their record only when the incoming row is newer (guarded by updated_at); if they are new, insert them. In Snowflake this is a native MERGE INTO statement; the solution shown uses PostgreSQL's INSERT … ON CONFLICT DO UPDATE WHERE idiom, which is semantically equivalent. The interviewer is testing mastery of upsert patterns, timestamp-gated update logic, and dialect awareness.

2. The SQL — INSERT … ON CONFLICT with timestamp guard

INSERT INTO customers (customer_id, name, email, updated_at)
SELECT customer_id, name, email, updated_at
FROM staging_customers
ON CONFLICT (customer_id) DO UPDATE
   SET name       = EXCLUDED.name,
       email      = EXCLUDED.email,
       updated_at = EXCLUDED.updated_at
 WHERE EXCLUDED.updated_at > customers.updated_at;

SELECT customer_id, name, email, updated_at
FROM customers
ORDER BY customer_id;

Snowflake native equivalent: MERGE INTO customers c USING staging_customers s ON c.customer_id = s.customer_id WHEN MATCHED AND s.updated_at > c.updated_at THEN UPDATE SET name=s.name, email=s.email, updated_at=s.updated_at WHEN NOT MATCHED THEN INSERT (customer_id, name, email, updated_at) VALUES (s.customer_id, s.name, s.email, s.updated_at);

4. Strategic explanations

The timestamp guard — why it is critical

Without WHERE EXCLUDED.updated_at > customers.updated_at, an older staging row would overwrite a newer target row — a silent data regression. The guard makes the merge idempotent with respect to ordering: re-running the same staging batch multiple times will never regress data. This is the key correctness property of any CDC (Change Data Capture) or SCD (Slowly Changing Dimension) upsert. Always state this invariant explicitly in an interview.

The EXCLUDED pseudo-table in PostgreSQL

In ON CONFLICT DO UPDATE, EXCLUDED is a virtual table that refers to the row that would have been inserted (i.e., the incoming staging row). So EXCLUDED.updated_at is the staging row's timestamp and customers.updated_at is the existing target row's timestamp. The WHERE clause on the DO UPDATE block makes the update conditional — if it evaluates to false, the row in the target is left untouched, which is exactly what we want for Carol's stale staging record.

Snowflake MERGE INTO vs PostgreSQL ON CONFLICT

Snowflake's native MERGE INTO … USING … ON … WHEN MATCHED THEN UPDATE … WHEN NOT MATCHED THEN INSERT is the ANSI SQL:2003 MERGE statement. It is more readable for complex upsert logic and supports multiple WHEN MATCHED clauses (e.g., a second WHEN MATCHED for deletes). PostgreSQL added a MERGE statement in version 15; prior versions use INSERT … ON CONFLICT. BigQuery supports MERGE. MySQL does not have MERGE — INSERT … ON DUPLICATE KEY UPDATE is the alternative (but lacks the conditional WHERE on the update).

What happens to Alice?

Customer id=1 is not present in staging_customers, so the INSERT finds no row to insert for her and she remains unchanged. This is the correct behavior for a nightly delta load: only touched records are in staging. A full-load strategy (where staging contains all customers) would instead treat absence from staging as a delete signal, requiring a different pattern.

Atomicity and concurrency

The INSERT … ON CONFLICT block executes atomically within a transaction. If two concurrent staging loads try to upsert the same customer_id at the same time, one will win and the other will see a conflict and apply the WHERE guard — no lost update. In Snowflake, MERGE is similarly atomic. Mentioning ACID guarantees in the context of concurrent ETL loads is the senior signal.

Senior signal: name the timestamp guard (WHERE EXCLUDED.updated_at > customers.updated_at) as the idempotency invariant that prevents stale data from overwriting fresh data; explain the EXCLUDED pseudo-table; contrast the PostgreSQL ON CONFLICT pattern with Snowflake's native MERGE INTO … WHEN MATCHED … WHEN NOT MATCHED; and note that the whole statement is atomic — concurrent staging loads are safe.
