A customer-insights query asks for the second-most-recent order per customer per category — the "what did they buy just before their latest" signal often used for repeat-purchase analysis. The orders table can have many orders per (customer, category), or just one. The interviewer is probing the canonical "Nth row per group" pattern: rank within each partition, then filter to the rank you want.

The edge case is the single-order group: customer 1 / category Y and customer 3 / category X each have exactly one order. There is no "second" order, so those groups must produce no row at all — not a NULL row, simply absent from the output.

2. The SQL — ROW_NUMBER per group, filter to rank 2

SELECT customer_id, category, order_id, ts, amount
FROM (
  SELECT customer_id, category, order_id, ts, amount,
         ROW_NUMBER() OVER (PARTITION BY customer_id, category ORDER BY ts DESC) AS rn
  FROM orders
) t
WHERE rn = 2
ORDER BY customer_id, category;

The inner query numbers each (customer, category) group's orders from newest to oldest — ORDER BY ts DESC makes rn = 1 the latest, rn = 2 the one before it. The outer query keeps only rn = 2. Groups with a single order never produce an rn = 2, so they vanish automatically.

4. Strategic explanations

Why a subquery is unavoidable

You cannot write WHERE ROW_NUMBER() OVER (...) = 2 directly: window functions are evaluated after the WHERE clause, so the rank does not exist yet when WHERE runs. The fix is to compute rn in an inner query (or CTE) and filter it in an outer one — a mandatory two-step shape for any "Nth per group" question.

Why ROW_NUMBER and not RANK / DENSE_RANK

"The second-most-recent order" means exactly one order. ROW_NUMBER guarantees distinct 1, 2, 3, ... even if two orders share a timestamp — so rn = 2 always picks a single row. RANK would assign two tied-newest orders both rank = 1, then jump to rank = 3 — there would be no rank = 2 and the query would silently return nothing for that group. For "pick exactly one row", ROW_NUMBER is the correct ranking function.

Single-member groups disappear — for free

The spec's tricky requirement — "if a customer has only one order in a category, that category should NOT appear" — needs no special handling. A one-row group only ever produces rn = 1; the predicate rn = 2 matches nothing and the group is excluded. The pattern handles the edge case structurally. Contrast this with LEAD/OFFSET 1 approaches that might emit a NULL row — the rank-and-filter form is cleaner here.

The tie trap — what if the two newest orders share a ts

ORDER BY ts DESC alone is non-deterministic when timestamps tie: which order becomes rn = 1 vs rn = 2 is arbitrary, and can change between runs. If the data permits same-timestamp orders, add a deterministic tiebreaker — ORDER BY ts DESC, order_id DESC — so "second-most-recent" is well-defined and reproducible. Naming this unprompted is the senior signal; the demo's distinct timestamps hide the ambiguity.

Dialect notes — portability and alternatives

ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...) is standard and identical in Postgres, SQL Server, BigQuery, Snowflake, Oracle, and MySQL 8+/SQLite 3.25+. Postgres also offers a terser non-standard route — SELECT DISTINCT ON (customer_id, category) ... — but that only fetches the first row per group, not the second, so the rank-and-filter form is the right tool for an arbitrary Nth.

Senior signal: solve "Nth per group" with ROW_NUMBER() in a subquery filtered to the target rank in the outer query — window functions cannot live in WHERE — pick ROW_NUMBER over RANK so exactly one row is selected, and add a tiebreaker to the ORDER BY so the result is deterministic under tied timestamps.
