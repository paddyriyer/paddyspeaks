A growth team wants a cohort-retention grid: group customers by the month they signed up, then for each later month count how many of that cohort came back to order. This is a foundational analytics query, and the interviewer is checking two things — whether you join the cohort definition (signup month) to the activity table correctly, and whether you use COUNT(DISTINCT ...) so a customer who orders twice in a month is counted once.

The edge cases: C02 has a gap (active Jan and Mar, silent in Feb), and C03 never orders in its own signup month — its first activity is the month after. The grid must reflect both honestly.

2. The SQL — cohort grid via join + COUNT(DISTINCT)

SELECT c.signup_month AS cohort,
       o.ord_month,
       COUNT(DISTINCT c.cust_code) AS retained_users
FROM customer c
JOIN orders o ON o.cust_code = c.cust_code
GROUP BY c.signup_month, o.ord_month
ORDER BY cohort, o.ord_month;

The join attaches each order to its customer's signup_month (the cohort label). GROUP BY cohort, ord_month forms one cell per (cohort, activity-month) pair, and COUNT(DISTINCT cust_code) counts how many different customers of that cohort were active that month.

The Jan cohort holds steady at 2 retained users each month — but it is a different 2 each time (C02 drops in Feb, C03 drops in Jan and Mar). The grid is sparse: only cells with at least one retained user appear, exactly as the question asks.

4. Strategic explanations

Why COUNT(DISTINCT cust_code) and not COUNT(*)

COUNT(*) counts order rows. If C01 placed three orders in March, COUNT(*) would report 3 retained users for that cell — wildly wrong, since retention measures people, not transactions. COUNT(DISTINCT cust_code) counts unique customers, so a customer who orders ten times in a month still contributes exactly 1. In this demo each customer happens to have one order per month, but the moment real data has repeat buyers, the DISTINCT is the difference between a correct retention number and an inflated one.

The cohort label rides along on the join

The cohort is a property of the customer, not the order. Joining orders to customer pulls signup_month onto every order row, so each transaction is tagged with the cohort it belongs to. Grouping by that tagged signup_month together with ord_month produces the two-dimensional grid. This is why the join is to customer and not, say, a self-join on orders.

Sparse grid vs dense grid — a deliberate choice

This query only emits cells that have activity — the question explicitly says "only emit cells with retained_users >= 1". That is why there is no row for, e.g., (Feb cohort, no March-skip placeholder). A dense retention grid — every cohort crossed with every calendar month, zeros included — needs a generated month spine LEFT JOINed in, because an inner join can never produce a zero. Knowing which shape the consumer wants (sparse for storage, dense for a heatmap) is the senior call. Here, sparse is specified.

Month 0 and the C03 subtlety

Notice the Jan cohort's January cell is 2, not 3 — C03 never ordered in its own signup month. A common cohort-analysis assumption is "everyone is active in month 0 by definition", but that is only true if signup itself counts as activity. Here activity means placing an order, and C03's first order is in February. Whether month-0 retention should be forced to 100% is a definitional question to raise with the stakeholder, not assume.

Dialect notes

This query is fully ANSI-standard — JOIN, GROUP BY, and COUNT(DISTINCT ...) behave identically in PostgreSQL, MySQL, SQL Server, Oracle, and BigQuery. The only caution: on very large fact tables COUNT(DISTINCT ...) can be expensive (it must deduplicate); warehouses like BigQuery offer APPROX_COUNT_DISTINCT for a fast, slightly inexact alternative when the cohort sizes are huge and exactness is not critical.

Senior signal: build cohort retention by joining activity to the customer's signup_month and grouping by (cohort, activity-month), always use COUNT(DISTINCT cust_code) so repeat orders do not inflate the count, and state whether the grid should be sparse (inner join) or dense with zeros (month-spine left join).
