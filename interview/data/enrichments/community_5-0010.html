A Community school keeps grades in a wide table — one column per subject. The analytics team needs the long shape: one row per student-per-subject, so the data can be filtered, grouped, and charted generically. This is an unpivot. The interviewer is watching whether you reach for UNION ALL, why it scales the way it does, and how the column labels stay correct.

1. The source data — student_grades

student_idstudent_namemathscienceenglishhistory

502Bianca Rossi88759184
503Chen Wei95988088
504Daria Petrova62707365

Four students, four subject columns each. The wide layout is hostile to analysis: "average grade across all subjects" needs (math + science + english + history)/4, and adding a fifth subject means rewriting every query. The long shape fixes that. Note all 16 cells are populated — a real dataset would have NULLs (a subject not taken), and those are the edge case to think about.

2. The SQL — UNION ALL per subject

SELECT student_id, student_name, 'math'    AS subject, math    AS grade FROM student_grades
UNION ALL
SELECT student_id, student_name, 'science', science FROM student_grades
UNION ALL
SELECT student_id, student_name, 'english', english FROM student_grades
UNION ALL
SELECT student_id, student_name, 'history', history FROM student_grades
ORDER BY student_id, subject;

One SELECT per subject column. Each branch hard-codes the subject name as a string literal and projects that subject's column under the common alias grade. UNION ALL stacks the four branches; the final ORDER BY gives a stable, readable order.

4. Strategic explanations

Why UNION ALL, never UNION

This is the single most important detail. UNION deduplicates its result — it sorts or hashes every row to discard duplicates. If two students happened to score the same grade in the same subject (e.g. two 70s in history), plain UNION would silently collapse them into one row, losing a student. UNION ALL keeps every row unconditionally, which is what unpivot requires — and it skips the dedup pass, so it is also faster. For unpivot the answer is always UNION ALL.

The cost: one table scan per subject

Each UNION ALL branch is an independent SELECT FROM student_grades — four branches means the table is scanned four times. For four subjects on a small table that is fine. For a wide table with 40 columns it is 40 scans, and the pattern gets clumsy. The single-scan alternative is a CROSS JOIN against an inline list of subjects:

WITH subjects(subject) AS (
  VALUES ('math'), ('science'), ('english'), ('history')
)
SELECT g.student_id, g.student_name, s.subject,
       CASE s.subject
         WHEN 'math'    THEN g.math
         WHEN 'science' THEN g.science
         WHEN 'english' THEN g.english
         WHEN 'history' THEN g.history
       END AS grade
FROM student_grades g CROSS JOIN subjects s
ORDER BY g.student_id, s.subject;

The CROSS JOIN reads student_grades once and multiplies each row by the four-row subjects list; the CASE picks the right column. Same 16-row result, one scan — the better choice when subject count grows.

NULL grades survive — usually what you want

If a student never took history, that cell is NULL. Both the UNION ALL and CROSS JOIN forms still emit a row for them — (504, 'Daria', 'history', NULL). That is normally correct: the long table records that the subject exists for that student even if ungraded. If the consumer instead wants only graded subjects, add WHERE grade IS NOT NULL as an outer filter. Decide on purpose; do not let it be accidental.

The dialect UNPIVOT operator

SQL Server and Oracle have a dedicated UNPIVOT clause:

SELECT student_id, student_name, subject, grade
FROM student_grades
UNPIVOT (grade FOR subject IN (math, science, english, history)) AS u;

Cleaner to write, but two caveats: it is not portable (no UNPIVOT in Postgres, MySQL, or SQLite), and standard UNPIVOT drops NULL-grade rows by default — the opposite of the UNION ALL behaviour. Postgres's idiomatic equivalent is LATERAL (VALUES ...) or jsonb_each. Know the operator exists; default to UNION ALL in an interview for portability.

Type compatibility across branches

UNION ALL matches columns by position and requires compatible types per column. All four grade columns are INTEGER, so grade is clean. If one subject column were NUMERIC and another INTEGER, the union would promote to the wider type — usually harmless, but if one were text the union would fail or coerce unexpectedly. Keep the unpivoted columns type-aligned, and cast explicitly if they are not.

Senior signal: always use UNION ALL (never UNION, which silently dedupes away real rows), know the single-scan CROSS JOIN-against-a-values-list alternative for wide tables, decide deliberately whether NULL-grade rows stay or go, and recognise UNPIVOT as non-portable sugar that drops NULLs by default.
