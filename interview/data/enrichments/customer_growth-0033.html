The L5 trap. The prompt: "List each department's highest and second-highest salary as columns — and if a new department appears tomorrow, the query must auto-add a column." The honest answer is that standard SQL cannot truly do dynamic pivoting. SQL's relational model fixes the column list at parse time — a "pivot" turns row values into columns, so adding "Legal" tomorrow means the schema of the result changes. No database can do that in a single static query. What we have are three escape hatches.

The senior signal isn't picking the most clever workaround. It's recognising that a request for dynamic pivot is usually a request to push presentation logic OUT of SQL.

2. The three approaches — name all three, pick consciously

Approach 1 — Two-step dynamic SQL (works in every DB)

A metadata query discovers the distinct values; a host language (or stored procedure) stitches them into a SQL string; then executes it. MySQL uses GROUP_CONCAT + PREPARE/EXECUTE:

SET @cols = NULL;
SELECT GROUP_CONCAT(DISTINCT
    CONCAT('MAX(CASE WHEN department = ''', department,
           ''' THEN salary END) AS `', department, '`')
) INTO @cols
FROM employees;

SET @sql = CONCAT('
  WITH RankedSalaries AS (
    SELECT department, salary,
      DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rnk
    FROM employees
  )
  SELECT CASE WHEN rnk=1 THEN ''1st Highest''
              WHEN rnk=2 THEN ''2nd Highest'' END AS salary_position,
         ', @cols, '
  FROM RankedSalaries
  WHERE rnk IN (1,2)
  GROUP BY rnk
  ORDER BY rnk');

PREPARE stmt FROM @sql; EXECUTE stmt; DEALLOCATE PREPARE stmt;

PostgreSQL has the same pattern with string_agg + EXECUTE inside a DO block or function. There's also the tablefunc extension's crosstab() — but it still requires you to declare the output column list, so it's not actually dynamic. You'd wrap it in PL/pgSQL that builds the column types first.

Snowflake is the cleanest of the lot — it has native dynamic pivot:

SELECT *
FROM (SELECT department, salary, rnk FROM RankedSalaries WHERE rnk IN (1,2))
PIVOT (MAX(salary) FOR department IN (ANY ORDER BY department));

PIVOT … IN (ANY) resolves the column list at execution time. Databricks SQL has the same. SQL Server has PIVOT but the IN list must be literal — same dynamic-SQL workaround needed.

The tradeoffs to name in the interview:

  • SQL injection if department values come from user input. Always quote-escape (PostgreSQL quote_ident), or use an allowlist join against a trusted dim.
  • Schema-of-result changes between runs, which breaks downstream consumers that expect a stable contract. A BI tool doing SELECT col_engineering FROM ... dies the day Engineering gets renamed.
  • Not portable — the dynamic-SQL syntax is dialect-specific.
Approach 2 — Keep it long-format, pivot in the consumer (the pragmatic answer)

Often the right answer at scale:

SELECT department, salary_position, salary
FROM (
  SELECT department, salary,
    DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rnk
  FROM employees
) r
WHERE rnk IN (1, 2);

Return long-format, let pandas (df.pivot), Tableau, Looker, or whatever consumes it handle presentation. This is what most production warehouses actually do, because:

  • The result schema stays stable as new departments appear.
  • You avoid dynamic-SQL injection surface entirely.
  • BI tools are built for the row-to-column reshape.

The interview signal: you know when not to push work into SQL. Don't be the engineer who insists every problem must be solved in one query.

Approach 3 — JSON aggregation (semi-structured pivot) · the runnable answer

Every modern DB has a JSON object aggregator. The schema stays one column wide; the keys are dynamic. This is the canonical solution in this question's runnable SQL block — it executes on PGlite.

WITH RankedSalaries AS (
  SELECT department, salary,
    DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS rnk
  FROM employees
)
SELECT
  CASE WHEN rnk = 1 THEN '1st Highest'
       WHEN rnk = 2 THEN '2nd Highest' END                  AS salary_position,
  jsonb_object_agg(department, salary ORDER BY department)  AS by_department
FROM RankedSalaries
WHERE rnk IN (1, 2)
GROUP BY rnk
ORDER BY rnk;

Result (the actual PGlite output):

Add "Legal" tomorrow — it just shows up as a new key. No DDL, no schema break, no rewrite. Downstream code reads by_department ->> 'Engineering' for the columns it cares about. This is the pattern Netflix-style event pipelines use most often: stable outer schema, evolving inner payload.

3. How auto-add actually works in production

The "new department appears and the pivot updates itself" requirement is almost never solved at query time. It's solved at pipeline time:

PatternHowWhen to pick
Schema-on-read (JSON column)Approach 3 above — nothing to update; consumers read keys they care aboutFrequently-evolving dimensions, downstream is code (not BI)
Materialised view + scheduled refreshA job re-runs the dynamic-SQL generator nightly. View column list evolves. Downstream must be additive-tolerant (SELECT * breaks, SELECT explicit_cols doesn't)BI dashboards that need wide format, slow-evolving dimensions
dbt + get_column_values macroAt compile time, dbt queries the source for distinct values and templates them into the model SQL. The model re-compiles on the next runThe default in Snowflake / BigQuery / Redshift / Databricks shops
Event-time enrichmentWhen a new dimension value first appears, an upstream Flink job (or DB trigger) writes it to a dim_departments registry; the pivot generator reads from that registryStreaming pipelines, decouples discovery from query execution

The dbt macro pattern, concretely:

{# macros/get_column_values.sql — runs at dbt compile time #}
{% set depts = dbt_utils.get_column_values(
    table=ref('employees'),
    column='department'
) %}

select
  case when rnk = 1 then '1st Highest' when rnk = 2 then '2nd Highest' end as salary_position,
  {% for dept in depts %}
  max(case when department = '{{ dept }}' then salary end) as {{ dept | lower }}
  {%- if not loop.last %},{% endif %}
  {% endfor %}
from {{ ref('ranked_salaries') }}
where rnk in (1, 2)
group by rnk
order by rnk;

Run dbt run; the SQL is re-templated with the current department list. Next run picks up Legal automatically. The contract: additive only — adding a column is safe; renaming or removing one is a breaking change downstream consumers must subscribe to.

4. The interviewer's edge-case probes

Probe 1 — "What's wrong with Approach 1 in production?"

Three things:

  1. SQL injection surface. If department values come from user input or untrusted upstream, dynamic SQL is a pen-test target. Mitigation: quote_ident / parameterised allowlist join.
  2. Result schema is unstable. Every BI tool consuming this needs to subscribe to schema updates. SELECT * in a dashboard breaks the moment a column is renamed.
  3. Dialect-specific. The dynamic-SQL pattern is different on every engine. You write it once for MySQL, again for Postgres, again for SQL Server — and your dbt models can't easily target multiple warehouses.
Probe 2 — "Is `PIVOT ... IN (ANY)` on Snowflake the silver bullet?"

It's the cleanest syntax, but the schema-stability problem doesn't go away — the result still has dynamic columns. Snowflake handles the discovery, but your BI dashboard or pandas reader still has to deal with a result whose shape changes between runs. For downstream code: use it. For BI dashboards expecting stable columns: still need a contract — either explicit column listing or wrap in a JSON aggregator. Don't sell PIVOT … ANY as a complete answer.

Probe 3 — "What if the consumer is a Tableau dashboard, not code?"

Then Approach 2 (long-format) is almost always correct. Tableau, Looker, PowerBI all have native pivot/measure-as-column controls. Push the pivot out of SQL. The wide-format answer is for code consumers; for BI tools, long-format + tool-side pivot is the convention because the tool gracefully handles new dimension values.

Probe 4 — "How do you handle the rename case? 'Engineering' is being renamed to 'Platform Engineering' next quarter."

This is the failure mode that breaks every dynamic-pivot solution at some point. Three guards:

  • Map renames in a dim table: dim_department(canonical_name, display_name, valid_from, valid_to) with SCD2. Queries join through the canonical_name; display_name is the BI-facing label.
  • Pin the column list to the canonical names, not display names. engineering stays engineering even when the display label changes.
  • Emit a deprecation event when a column is going away. Downstream subscribers get N days to migrate.
Probe 5 — "What about the column-explosion failure mode? What if there are 10,000 departments?"

The dynamic-pivot path collapses: SQL engines have per-statement column limits (Postgres 1600, BigQuery 10K, Snowflake 4096). At that grain you've outgrown the wide format. JSON aggregation (Approach 3) still works because the "column count" is just the JSON key count inside one column. Or — more honestly — the question is wrong: nobody reads a 10K-column pivot. The interviewer wants you to surface that and re-frame: "at 10K dims I'd return long-format and let the consumer top-K filter."

5. The L5 close — what the interviewer is actually testing

Senior signal — the 60-second answer:

"True dynamic pivot in standard SQL doesn't exist — columns are fixed at parse time. There are three workarounds: two-step dynamic SQL (works everywhere but injection-prone and schema-unstable), long-format + client pivot (the pragmatic answer for BI), and JSON aggregation (the schema-stable answer for code consumers — jsonb_object_agg in Postgres, OBJECT_AGG in Snowflake). I'd reach for PIVOT … ANY only on Snowflake/Databricks, and only when downstream tools tolerate a changing column list. In production the auto-add requirement is solved at pipeline time — dbt's get_column_values macro re-templates the model on each run, or a Flink job writes new dim values to a registry that the pivot generator reads. The failure mode to name explicitly is schema drift breaking BI."

That paragraph signals: (1) you understand the relational-model constraint, (2) you know three named workarounds, (3) you've got the dialect detail (Snowflake's ANY, Postgres's jsonb_object_agg), (4) you know production solves this at pipeline time, not query time, (5) you can articulate the failure mode. Senior-IC bar cleared.
