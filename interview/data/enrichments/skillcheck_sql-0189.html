A retail analytics team tracks daily revenue per store, but only on days when the store made any sales (sparse data). They want a 7-day trailing average that treats missing days as zero — a "calendar window," not just a "N-rows" window. The interviewer is probing the critical distinction between ROWS BETWEEN (physical row count) and RANGE BETWEEN INTERVAL '6 days' PRECEDING (calendar span), and the correct denominator (always 7, not the count of present rows).

2. The SQL — RANGE frame with date interval

SELECT store_id,
       sale_date,
       revenue,
       ROUND((SUM(revenue) OVER (
          PARTITION BY store_id
          ORDER BY sale_date
          RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW
       ) / 7.0)::numeric, 2) AS rolling_7d_avg
FROM daily_sales
ORDER BY store_id, sale_date;

The RANGE frame includes all rows for the same store where sale_date ≥ current_row_date - 6 days. The denominator is always 7.0 because missing days contribute 0 to the sum — they are absent from the table so they don't appear in the SUM, but the window still covers exactly 7 calendar days.

3. The resulting analysis

Hand-trace for store 1 (dates in Jan 2025):

  • Jan 01: window = [Dec 26, Jan 01] ∩ store 1 rows = {Jan 01 = 100}. Sum = 100. Avg = 100/7 = 14.29.
  • Jan 05: window = [Dec 30, Jan 05] ∩ store 1 rows = {Jan 01 = 100, Jan 05 = 200}. Sum = 300. Avg = 300/7 = 42.86.
  • Jan 07: window = [Jan 01, Jan 07] ∩ store 1 rows = {Jan 01 = 100, Jan 05 = 200, Jan 07 = 300}. Sum = 600. Avg = 600/7 = 85.71.
  • Jan 08: window = [Jan 02, Jan 08] ∩ store 1 rows = {Jan 05 = 200, Jan 07 = 300, Jan 08 = 400}. Jan 01 falls outside (Jan 01 < Jan 02). Sum = 900. Avg = 900/7 = 128.57.

4. Strategic explanations

RANGE vs ROWS — the central distinction

ROWS BETWEEN 6 PRECEDING AND CURRENT ROW counts exactly 7 physical rows (or fewer at the start of the partition). For store 1, Jan 08 would look back to Jan 05 (the 6th prior row), capturing Jan 05, Jan 07, Jan 08 — 3 present rows, but the window would span only 3 days. The denominator would then be 3, not 7, giving a very different (higher) average. RANGE BETWEEN INTERVAL '6 days' PRECEDING uses the date value to bound the window, so it always spans exactly 7 calendar days regardless of how many rows are present in that span.

Dividing by 7.0, not COUNT(*)

The question specifies "missing days count as 0." A missing day contributes 0 to the numerator (it is not in the table) but 1 to the denominator (it is a real calendar day). Dividing the RANGE-windowed SUM by 7.0 implements this exactly. Dividing by COUNT(*) OVER (...same RANGE...) would give the average over present days only, ignoring the missing zeros.

RANGE frame requirements in PostgreSQL

An INTERVAL RANGE frame in PostgreSQL requires the ORDER BY column to be a date or timestamp type — numeric offsets use ROWS or numeric RANGE. The frame boundary INTERVAL '6 days' PRECEDING means the window starts at the greatest date ≤ current_date - INTERVAL '6 days'. Frames are evaluated per partition, so the PARTITION BY store_id ensures store 1 and store 2 windows never mix.

Edge cases

The first row of each store has fewer than 7 calendar days of history — the window simply extends from the first available row to the current row. The sum of fewer rows divided by 7 gives a lower average than if the prior days had data. This is the correct "sparse-data" interpretation. If you instead wanted to exclude rows where the window has fewer than 7 days of history, add a HAVING or WHERE filter on the window start date.

Dialect notes

Interval RANGE frames are supported in PostgreSQL and Snowflake with the same syntax. BigQuery uses RANGE BETWEEN 6 PRECEDING AND CURRENT ROW where the ORDER BY column is a numeric offset (e.g., DATE_DIFF(sale_date, start_date, DAY)); it does not support interval-typed RANGE bounds natively, requiring a workaround with numeric epoch values. SQL Server supports ROWS frames but not interval RANGE bounds — calendar windows require a different approach (e.g., a correlated subquery or a calendar table join). MySQL 8+ supports RANGE with numeric bounds only.

Senior signal: use RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW (not ROWS) to get a calendar-based 7-day window over sparse data, and always divide by 7.0 (the fixed calendar denominator) rather than COUNT(*) over the window — explain that BigQuery and SQL Server lack interval RANGE support and need workarounds.
