A nested comment system is a tree-in-a-table problem that has two classical solutions — materialized path (store each node's full ancestry as a string) and closure table (store all ancestor–descendant pairs) — and each has a different performance profile for the three core operations: fetch all comments under a post, fetch all descendants of a comment, and order by score. The interviewer is probing which technique you choose, why, and how Postgres's ltree extension makes the materialized-path approach first-class.

1. The requirements — what the design must capture

  • Nested hierarchy: comments have a parent comment (nullable for top-level); the depth can be unbounded in theory, though Reddit-style UIs cap display depth at ~6–8 levels.
  • Fast subtree retrieval: "all descendants of comment 42" must be a single indexed query, not a recursive CTE that walks level by level.
  • Fast post-level retrieval: "all comments under post 123" must be a single indexed query regardless of thread depth.
  • Score ordering: comments are ordered by vote score within their level; top-level comments and reply threads each need score-ordered retrieval.
  • Soft-delete: deleted comments are not hard-removed (this would orphan children); they are replaced with "[deleted]" and the row is kept with is_deleted = TRUE.

2. The DDL — the CREATE TABLE

-- Required extension for ltree-based path indexing:
CREATE EXTENSION IF NOT EXISTS ltree;

CREATE TABLE comment (
  comment_id        BIGSERIAL    PRIMARY KEY,
  post_id           BIGINT       NOT NULL,
  parent_comment_id BIGINT       REFERENCES comment(comment_id),
  author_id         BIGINT       NOT NULL,
  body              TEXT         NOT NULL,
  score             INT          NOT NULL DEFAULT 0,             -- upvotes - downvotes (denormalized for ordering)
  depth             SMALLINT     NOT NULL DEFAULT 0,             -- 0 = top-level
  path              LTREE        NOT NULL,                       -- materialized-path: e.g., '1.13.42'
  is_deleted        BOOLEAN      NOT NULL DEFAULT FALSE,
  created_at        TIMESTAMPTZ  NOT NULL DEFAULT now(),
  updated_at        TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE INDEX ix_comment_post_score ON comment (post_id, score DESC);
CREATE INDEX ix_comment_post_created ON comment (post_id, created_at);
CREATE INDEX ix_comment_path_gist ON comment USING gist (path);
CREATE INDEX ix_comment_parent ON comment (parent_comment_id) WHERE parent_comment_id IS NOT NULL;

-- Common queries:
-- All comments under a post, ordered by score:
--   SELECT * FROM comment WHERE post_id = ? ORDER BY score DESC, created_at;
-- All descendants of a specific comment (its subtree):
--   SELECT * FROM comment WHERE path <@ '1.13'::ltree;     -- ltree descendant operator
-- Direct children:
--   SELECT * FROM comment WHERE parent_comment_id = ?;

4. Strategic explanations

Materialized path (ltree) vs. closure table

The two main alternatives: materialized path stores the full ancestry as a string on each row; closure table stores every ancestor–descendant pair in a separate table. Materialized path is chosen here because: (1) Postgres's ltree extension makes it first-class with a GiST index and the <@ operator; (2) subtree reads are one index scan; (3) it requires no join for retrieval. The tradeoff is that moving a comment subtree requires updating the path of every descendant — expensive for deep trees. Closure table is better for frequent subtree moves and for engines without ltree, but it doubles write volume (one row per ancestor for every insert) and requires a JOIN for every read.

The <@ ltree operator and GiST index

SELECT * FROM comment WHERE path <@ '1.13'::ltree retrieves all descendants of comment 13 (which is itself a child of comment 1). The GiST index on path makes this an index scan — Postgres uses the index to find all rows whose path is a descendant of the given prefix. Without the GiST index this degrades to a sequential scan on a LIKE '1.13.%' equivalent. The ~ (match) and @ (ancestor of) operators are also GiST-accelerated.

Denormalized score and vote consistency

score = upvotes - downvotes is stored denormalized on the comment row for fast ORDER BY without a JOIN to a vote-count aggregate. This is correct for read-heavy workloads (Reddit-scale reads dwarf writes). The tradeoff: every vote must atomically update comment.score (UPDATE comment SET score = score + 1 WHERE comment_id = ?), which creates write contention on popular comments. Mitigation: use SELECT ... FOR UPDATE on the vote write, or use Postgres's advisory locks, or accept approximate counts with a reconciliation job.

Adjacency list + materialized path: belt and suspenders

Keeping both parent_comment_id (adjacency list) and path (materialized path) provides redundancy: the adjacency list is the authoritative source of truth for the immediate parent relationship; the path is a derived, performance-oriented artifact. If the path becomes corrupted or needs regeneration, it can always be reconstructed from the adjacency list via a recursive CTE. The partial index on parent_comment_id WHERE parent_comment_id IS NOT NULL excludes top-level comments, keeping the index compact.

Dialect notes

ltree is a Postgres extension (bundled in the postgresql-contrib package). In MySQL, implement materialized path as a VARCHAR with a LIKE '1.13.%' query — performant only with a prefix index and shallow trees. In SQL Server, use the native hierarchyid type. In BigQuery, hierarchical queries use STRUCT arrays or repeated fields; subtree retrieval uses recursive CTEs (available since BigQuery GA 2023). In graph-heavy deployments, export the adjacency list to a graph database (Neo4j, Amazon Neptune) and query depth traversal there.

Senior signal: choose and justify your tree encoding — materialized path with ltree for read-heavy Reddit-style threads, closure table for write-heavy or engine-agnostic contexts; identify score as a semi-additive denormalized measure and name the write-contention risk on popular comments; and explain that parent_comment_id and path are complementary (one is truth, one is performance) and that the path can always be regenerated from the adjacency list.
