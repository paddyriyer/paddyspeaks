Uber's payments team wants the users who have crossed a transaction milestone — people with at least 2 transactions, the threshold that separates a one-off rider from a repeat customer worth retaining. The query is a single GROUP BY with a HAVING count filter. Simple on the surface; the interviewer is watching whether you put the count condition in the right clause and whether you noticed what "transaction" actually means in this messy schema.

1. The source data — transactions

User 1 has 2 transactions, user 3 has 3 — both clear the milestone. User 2 has exactly one — the boundary case that must be excluded. The threshold is ">= 2", so user 2 with a count of 1 falls just short.

2. The SQL — GROUP BY + HAVING COUNT

SELECT user_id FROM transactions
GROUP BY user_id HAVING COUNT(*) >= 2;

One row per user via GROUP BY; HAVING COUNT(*) >= 2 keeps only users whose group has 2 or more rows.

3. The resulting analysis

user_id
1
3

Users 1 and 3 qualify; user 2 is filtered out by HAVING. The result is just the id list — no count column — which answers "who", but a reviewer often wants the count alongside for sanity-checking. Row order is not guaranteed without an ORDER BY.

4. Strategic explanations

WHERE vs HAVING — the core distinction

You cannot write WHERE COUNT(*) >= 2 — WHERE is evaluated row-by-row before grouping, when no aggregate exists yet; Postgres rejects it outright. HAVING is evaluated after grouping, so it can see COUNT(*). The rule: WHERE filters rows going into the groups, HAVING filters the groups coming out. They often appear together — WHERE to drop irrelevant rows first, then HAVING on the aggregate.

COUNT(*) vs COUNT(DISTINCT txn_id) — what is a transaction?

COUNT(*) counts rows. That is correct only if each row is exactly one transaction. This schema is alarming: it carries both txn_id and transaction_id — redundant id columns that smell of a sloppy join or a denormalised table. If a single transaction can appear on multiple rows (one row per line-item, say), COUNT(*) over-counts and a one-transaction user could falsely clear the milestone. The safe milestone count is over distinct transactions:

SELECT user_id FROM transactions
GROUP BY user_id
HAVING COUNT(DISTINCT txn_id) >= 2;

Asking "is one row guaranteed to be one transaction?" before trusting COUNT(*) is the senior move.

The status filter the milestone implies

The table has a status column. A "transaction milestone" almost certainly means successful transactions — failed, declined, or reversed attempts should not count toward someone being a repeat customer. The reference counts every row regardless of status. Confirm the definition and filter the rows before grouping, which is exactly the job of WHERE:

SELECT user_id FROM transactions
WHERE status = 'completed'
GROUP BY user_id
HAVING COUNT(*) >= 2;
user_id vs customer_id — which identity?

The schema also exposes customer_id alongside user_id. They may be the same person under two systems, or genuinely different grains (a customer account vs an app user). Grouping by the wrong one silently answers a different question. Verify which column is the intended unit of "user" rather than guessing.

Edge case: NULL user_id

If some transactions have a NULL user_id (unauthenticated or orphaned), GROUP BY gathers them all into a single NULL group. If that group has >= 2 rows it appears in the output as a NULL user_id — almost never a meaningful "user". Add WHERE user_id IS NOT NULL to keep the milestone list clean.

Senior signal: explain that the count condition must live in HAVING because WHERE runs before aggregation, question whether one row equals one transaction given the duplicate txn_id/transaction_id columns and reach for COUNT(DISTINCT) if not, add the status filter so only successful transactions count toward the milestone, and exclude NULL user_id.
