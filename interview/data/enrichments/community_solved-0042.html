A Community member wants a classic cumulative metric: for each product, a running total of quantity sold as the dates advance. It is the canonical window-function exercise — and the interviewer uses it to check whether you reach for a window function or fall back to a self-join, and whether you understand the default frame you just inherited.

1. The source data — product_sales

product_nameproduct_codequantitysales_date
WidgetP1102024-01-01
WidgetP152024-01-03
WidgetP182024-01-05
GadgetP2202024-01-02
GadgetP2152024-01-04

Two products interleaved by date. The running total must reset at each product_code boundary — P2's accumulation must not pick up where P1 left off, even though P2's first sale falls between two P1 sales.

2. The SQL — SUM() OVER (PARTITION BY ... ORDER BY ...)

SELECT *,
    SUM(quantity) OVER (PARTITION BY product_code
                        ORDER BY sales_date ASC) AS running_total
FROM product_sales;

PARTITION BY product_code restarts the accumulation for each product; ORDER BY sales_date defines the direction it grows. No GROUP BY — every input row is preserved, with the cumulative figure attached.

P1 climbs 10 → 15 → 23; P2 climbs 20 → 35 independently. P2's run starts fresh at 20 even though its first sale (Jan 2) sits chronologically between P1's Jan 1 and Jan 3 — that is PARTITION BY isolating the two series.

4. Strategic explanations

The default frame you silently accepted

An ORDER BY inside an OVER with no explicit frame defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. For a running total that is what you want — sum everything up to and including the current row. Worth stating out loud, because the alternative bites in the next paragraph.

The RANGE-vs-ROWS trap on duplicate dates

The default frame is RANGE, not ROWS. If a product has two sales on the same sales_date, RANGE treats them as peers — both rows get the same running total, the sum through the end of that date. ROWS would give them stepped, distinct totals. For a cumulative quantity that is genuinely a judgement call:

SUM(quantity) OVER (PARTITION BY product_code
                    ORDER BY sales_date
                    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)

If two sales can share a date, decide whether you want them to share a total (RANGE) or step (ROWS), and write the frame explicitly so the next reader knows it was a choice.

The pre-window-function answer — and why it's worse

Before window functions, the running total was a correlated self-join:

SELECT a.product_code, a.sales_date, a.quantity,
       (SELECT SUM(b.quantity) FROM product_sales b
        WHERE b.product_code = a.product_code
          AND b.sales_date <= a.sales_date) AS running_total
FROM product_sales a;

It returns the same numbers but is O(n²): the subquery rescans the table for every row. The window version sorts once and accumulates in a single pass. On a real sales table the self-join is the wrong answer purely on cost.

The tiebreaker the demo hides

ORDER BY sales_date alone is ambiguous when a product sells twice on one day — and under ROWS the running total then depends on arbitrary row order. Add a deterministic tiebreaker, e.g. ORDER BY sales_date, product_name or a surrogate id, so the result is reproducible run to run.

NULL quantities

SUM ignores NULLs, so a row with a NULL quantity leaves the running total flat across that row — the cumulative value simply does not advance, and the row itself still appears (it is not a GROUP BY). If a NULL should count as zero that is already the effect; if it signals missing data you may want to COALESCE or exclude it deliberately.

Senior signal: reach straight for SUM() OVER (PARTITION BY ... ORDER BY ...) over a self-join, know that the default frame is RANGE and switch to ROWS when duplicate dates would otherwise share a total, and pin a deterministic tiebreaker so the running total is reproducible.
