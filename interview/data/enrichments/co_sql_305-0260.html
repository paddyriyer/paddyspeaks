Tesla's auto-financing risk team needs a single headline: what fraction of the loan book, weighted by principal, has defaulted? Not the count of defaulted loans — the dollar exposure. The reference solution wraps the arithmetic in a thick layer of type-coercion armour, and understanding why is the whole interview: it is defending against a schema where the columns are not the types you would hope.

1. The source data — loans

loan_iddefault_flagprincipal
1true50000
2false120000
3false80000
4true30000
5false200000
6true20000

Three of six loans defaulted — but by count that is 50%. The defaulted principals (50k + 30k + 20k = 100k) are small loans; the total book is 500k. The dollar-weighted rate will be far lower than the count-weighted rate. That divergence is the point.

2. The SQL — principal-weighted default rate

SELECT 100.0 * SUM(CASE WHEN default_flag::text IN ('1','true','t','True','TRUE') THEN NULLIF(regexp_replace(principal::text, '[^-.0-9]', '', 'g'), '')::numeric ELSE 0 END)/NULLIF(SUM(NULLIF(regexp_replace(principal::text, '[^-.0-9]', '', 'g'), '')::numeric), 0) AS default_rate_pct
FROM loans;

Stripped down: 100.0 * SUM(defaulted principal) / SUM(all principal). The CASE contributes a loan's principal to the numerator only when it defaulted; the denominator sums every principal. The regexp_replace and NULLIF wrappers are defensive coercion, discussed below.

3. The resulting analysis

default_rate_pct
20.0000000000000000

Defaulted principal is 100,000; total principal is 500,000; the rate is 20%. Contrast that with the count-based rate of 50% (3 of 6 loans). The dollar-weighted number is the one a risk committee cares about — it says the defaults clustered in the small loans, which is a materially better book than "half our loans defaulted" implies.

4. Strategic explanations

Dollar-weighted vs count-weighted — the metric that matters

This is the senior insight: which default rate?

-- count-weighted: 3/6 = 50%
SELECT 100.0 * COUNT(*) FILTER (WHERE default_flag) / COUNT(*) FROM loans;
-- dollar-weighted: 100k/500k = 20%   <-- the reference query

For credit risk, dollar-weighted is almost always the right denominator — losing a $20k loan is not the same event as losing a $200k loan. State which one the question wants and why; answering with the count rate when they wanted exposure is a real miss.

Why the coercion armour? — the schema is defensive

If default_flag were a clean BOOLEAN and principal a clean NUMERIC, the query would be one readable line. The armour exists because the source might not be clean:

  • default_flag::text IN ('1','true','t','True','TRUE') — copes with the flag arriving as a string, an integer, or a boolean, in any casing.
  • regexp_replace(principal::text, '[^-.0-9]', '', 'g') — strips currency symbols and thousands separators if principal is text like "$50,000".
  • NULLIF(…, '') — turns an empty string (post-scrub) into NULL so the ::numeric cast doesn't blow up.

A senior candidate names this: "this is hardened against a dirty, untyped feed; against a typed table I'd drop the regex for clarity."

The division-by-zero guard

The denominator is wrapped in NULLIF(SUM(…), 0). If the loan book is empty, or every principal scrubs to NULL, SUM is 0 (or NULL), and dividing by it would either error or be undefined. NULLIF(x, 0) converts a zero denominator into NULL, so the whole expression returns NULL — "no rate" — instead of crashing. Wrap in COALESCE(…, 0) if the dashboard insists on a number.

The NULL-flag trap

A loan whose default_flag is NULL fails the IN list (NULL matches nothing), so its principal lands in the ELSE 0 branch — counted as not defaulted. But its principal is still in the denominator. So an unknown-status loan quietly dilutes the default rate downward. If status-unknown loans should be excluded entirely, or treated as defaulted (conservative), handle them explicitly rather than letting ELSE absorb them.

The 100.0 — forcing real division

Leading with 100.0 (a numeric literal) rather than 100 guarantees the whole expression is evaluated in numeric, not integer, arithmetic. With integer 100 and integer operands, 100 * 100000 / 500000 could truncate. The .0 is a deliberate, cheap insurance against silent integer division.

Senior signal: distinguish the dollar-weighted default rate (20%) from the count-weighted rate (50%) and explain why exposure is the right lens for credit risk, read the regex/NULLIF layer as defensive coercion for an untyped feed, and flag that NULL default_flag loans silently dilute the rate via the ELSE 0 branch.
