NPM's analytics team wants a one-line customer summary: per customer, how many orders, the total, the average, the biggest and the smallest. It is the bread-and-butter GROUP BY — five aggregates over one grouping. The query is trivial to write; the interview value is knowing precisely which rows each aggregate counts, how NULLs slip past some of them, and why avg, max, and min are dangerous things to name a column.

Customer 1 has three orders spanning a wide range (50 to 200). Customer 2 is a single-order group — the edge case where min, max, and avg all collapse to the same number. Customer 3 has two mid-range orders.

2. The SQL — five aggregates, one GROUP BY

SELECT customer_id,
  COUNT(*) AS orders, SUM(amount) AS total, AVG(amount) AS avg,
  MAX(amount) AS max, MIN(amount) AS min
FROM orders
GROUP BY customer_id;

One row per customer_id; each aggregate folds that customer's order rows into a single value.

4. Strategic explanations

COUNT(*) vs COUNT(amount) — the NULL distinction

They diverge the moment amount can be NULL. COUNT(*) counts rows; COUNT(amount) counts non-NULL values. If a customer placed an order with a NULL amount (priced later, say), COUNT(*) includes it and COUNT(amount) does not. This matters because AVG(amount) is SUM(amount) / COUNT(amount) — it divides by the non-NULL count. So if you ever hand-roll the average as SUM(amount) / COUNT(*), you get a different (wrong) number than AVG whenever a NULL is present. SUM, MAX, and MIN all skip NULLs too; only COUNT(*) sees them.

avg, max, min are reserved-ish — rename the columns

Aliasing columns avg, max, min is asking for trouble: they collide with function names and with reserved words in some engines, forcing awkward quoting ("max") downstream and confusing every reader. Use descriptive, safe names — this is a real code-review note, not pedantry:

SELECT customer_id,
  COUNT(*)     AS order_count,
  SUM(amount)  AS total_spend,
  AVG(amount)  AS avg_order_value,
  MAX(amount)  AS largest_order,
  MIN(amount)  AS smallest_order
FROM orders
GROUP BY customer_id
ORDER BY total_spend DESC;

The ORDER BY is added too — a summary table almost always wants a deterministic, meaningful order.

The integer-division trap on AVG

If amount were an INTEGER column, some engines compute AVG with integer arithmetic and truncate — customer 1's 116.67 would come back as 116. Postgres returns numeric for an integer AVG (so it is safe here), but to be portable and explicit, cast: AVG(amount::numeric) or ROUND(AVG(amount), 2) to control the displayed precision. The long 116.6666666666666667 above is also worth rounding for a real report.

Customers with zero orders are invisible

This query is sourced only from orders, so a customer who has never ordered produces no row at all — not a row of zeros. If the summary must cover the full customer base, drive it from the customers table with a LEFT JOIN, and note the asymmetry: COUNT(o.order_id) correctly gives 0 for a no-order customer (it counts non-NULL join keys), but SUM(o.amount) gives NULL — wrap it in COALESCE(SUM(o.amount), 0).

The grand-total / rollup follow-up

A frequent next ask is "and the totals across all customers." Rather than a separate query, GROUP BY ROLLUP(customer_id) appends a summary row (with customer_id NULL) carrying the overall COUNT/SUM — one pass, one result set.

Senior signal: articulate that COUNT(*) counts rows while COUNT(col)/SUM/AVG/MIN/MAX skip NULLs, refuse to alias columns avg/max/min, pre-empt integer-division truncation with a cast, and switch to a LEFT JOIN + COALESCE when zero-order customers must appear.
