An HR tool needs an org-chart breadcrumb: for each employee, the full reporting chain from the CEO down to them, rendered as "CEO > VP > Manager > You", plus a depth number. The interviewer is probing recursive CTEs — specifically whether you can accumulate a string as the recursion descends, rather than just walk the hierarchy.

1. The source data — employee

The edge case is the root: Alice has manager_id = NULL, so she is the one row with no parent — the recursion's base case. Grace at depth 3 is the deepest chain, which exercises string accumulation across four names.

2. The SQL — recursive CTE accumulating a path string

WITH RECURSIVE tree AS (
  SELECT emp_id, name, manager_id, 0 AS depth, name AS path
  FROM employee WHERE manager_id IS NULL
  UNION ALL
  SELECT e.emp_id, e.name, e.manager_id,
         t.depth + 1, t.path || ' > ' || e.name
  FROM employee e JOIN tree t ON e.manager_id = t.emp_id
)
SELECT emp_id, name, depth, path FROM tree ORDER BY depth, emp_id;

The anchor seeds the CEO with depth = 0 and path = name. The recursive arm joins each employee to their manager's already-computed row in tree, increments depth, and appends ' > ' || name to the manager's path. Because the manager's path is already complete when the child is processed, the breadcrumb builds top-down automatically.

4. Strategic explanations

The accumulator pattern — carry state in a CTE column

A recursive CTE is not limited to walking edges; any column you compute in the recursive arm becomes state passed to the next iteration. Here two accumulators ride along: depth (a counter) and path (a growing string). The child reads the parent's value, transforms it, and hands the new value down. This same trick computes running products, level numbers, or visited-node arrays — the CTE row is your loop variable.

Anchor on the root, join child-to-parent

The anchor WHERE manager_id IS NULL picks exactly the CEO — the only node whose chain is trivially just itself. The recursive join e.manager_id = t.emp_id reads "find employees whose manager is already in the tree". Direction matters: joining the other way (e.emp_id = t.manager_id) would walk up from leaves and you would not have the root's path ready when you need it. Top-down from the root is what makes the breadcrumb read CEO-first.

Why the path is correct top-down, not reversed

Each iteration appends the child's name to the end of the parent's path: t.path || ' > ' || e.name. Since the parent row was finalised in an earlier iteration, its path already runs from the CEO down to the parent — appending the child extends it by one. If you instead built the chain from a leaf upward you would get the names reversed and need a final string-reverse step. Building from the root sidesteps that entirely.

Separator and NULL hygiene

name is assumed non-NULL; if it could be NULL, || in Postgres yields NULL and would blank out the whole path — wrap with COALESCE(name, '?'). The anchor sets path = name with no leading separator, so the CEO's breadcrumb is just "Alice" rather than "> Alice". Picking ' > ' (spaces around the arrow) is cosmetic but consistent — whatever you choose, use the identical literal in both arms.

Termination, ordering, and dialect

A clean org chart is a finite tree, so each step descends one level and the recursion ends when leaves produce no children. A data error creating a cycle (A manages B manages A) would loop forever — defend with a visited-array check or UNION (instead of UNION ALL) plus a depth cap. WITH RECURSIVE is standard in Postgres, SQLite, and MySQL 8+; SQL Server uses WITH without the keyword; Oracle also supports CONNECT BY … PRIOR with SYS_CONNECT_BY_PATH for exactly this breadcrumb.

Senior signal: treat a recursive CTE column as loop state — accumulate both depth and the path string as the recursion descends, anchor on the NULL-parent root and join child-to-parent so the breadcrumb builds top-down, keep the separator literal identical in both arms, and mention cycle defence for dirty hierarchy data.
