Facilities managers asking "Is our ceiling height adequate for warehouse racking on each floor of each building?" need both a numeric average and a categorical label. The interviewer is probing two things: can you aggregate over a composite group key, and can you apply a multi-tier CASE WHEN bucket to an aggregate expression? The twist here is the NULLIF(regexp_replace(...)::numeric) defensive cast — a signal that the column may contain non-numeric noise and that you need to sanitise before averaging.

2. The SQL — per-floor average with categorical classification

SELECT building_id, floor,
       AVG(NULLIF(regexp_replace(ceiling_height_m::text, '[^-.0-9]', '', 'g'), '')::numeric) AS avg_height_m,
       CASE
         WHEN AVG(NULLIF(regexp_replace(ceiling_height_m::text, '[^-.0-9]', '', 'g'), '')::numeric) >= 3   THEN 'High'
         WHEN AVG(NULLIF(regexp_replace(ceiling_height_m::text, '[^-.0-9]', '', 'g'), '')::numeric) >= 2.5 THEN 'Standard'
         ELSE 'Low'
       END AS height_class
FROM rooms
GROUP BY building_id, floor
ORDER BY building_id, floor;

4. Strategic explanations

Composite GROUP BY — why both columns matter

Grouping by building_id, floor creates one row per (building, floor) pair. Grouping by building_id alone would conflate ground-floor low rooms with upper-floor high ones. Grouping by floor alone would average across all buildings — useless if buildings differ structurally. The composite key reflects the real-world unit of analysis: a single floor of a single building.

The defensive NULLIF(regexp_replace(...)::numeric) cast

The ceiling_height_m column is likely typed as TEXT or VARCHAR because upstream data may include unit suffixes like "3.2m", currency characters, or empty strings. The pipeline: strip everything not a digit, minus sign, or decimal point with regexp_replace(val::text, '[^-.0-9]', '', 'g'); then NULLIF(..., '') converts the empty string that remains when the whole value was non-numeric into NULL; finally ::numeric does the actual cast. AVG ignores NULLs, so corrupted rows drop out of the average rather than crashing the query. This pattern is a Postgres idiom — Snowflake's TRY_CAST(val AS FLOAT) is the cleaner equivalent in that dialect.

CASE WHEN on an aggregate — evaluated once or twice?

The CASE WHEN references the AVG(...) expression verbatim rather than the alias avg_height_m. SQL's logical processing order means SELECT aliases are not available inside the same SELECT's expressions (in most engines). To avoid repeating the long expression you can wrap the query in a subquery or CTE and reference the alias in the outer CASE. Snowflake and BigQuery relax this restriction in some positions, but the subquery approach is universally portable. In the query as written the engine computes the same AVG expression up to three times in the worst case — though optimisers typically common-subexpression-eliminate it into one pass.

Threshold boundary semantics — ≥3 is checked first

CASE WHEN evaluates clauses in order and returns on the first match. A room averaging exactly 3.00 m hits the first branch (≥ 3 → High) and never reaches the second. If the branches were reversed — ≥2.5 first — a 3.10 m average would be misclassified as Standard. Always place the highest threshold first in a descending classification like this.

Dialect portability
  • Snowflake: use TRY_CAST(ceiling_height_m AS FLOAT) instead of the regexp_replace / NULLIF / ::numeric triple; it returns NULL on any non-parseable value cleanly.
  • BigQuery: SAFE_CAST(ceiling_height_m AS FLOAT64) is the equivalent safe-cast function.
  • MySQL: no native safe cast; CASE WHEN val REGEXP '^-?[0-9]+\\.?[0-9]*$' THEN CAST(val AS DECIMAL(10,2)) ELSE NULL END is the workaround.
  • regexp_replace with the 'g' (global) flag is Postgres-specific; Snowflake's REGEXP_REPLACE applies globally by default.
