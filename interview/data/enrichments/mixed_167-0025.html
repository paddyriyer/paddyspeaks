An analytics team tracking a noisy daily metric — signups, latency, revenue — wants a smoothed 7-day moving average so a stand-up chart shows trend rather than jitter. The reference solution uses a windowed AVG with an explicit ROWS BETWEEN 6 PRECEDING AND CURRENT ROW frame. The interviewer is probing whether you understand the difference between a row frame and a range frame, what the early rows do, and how a NULL day distorts the average.

Two things to watch. The first six rows do not have six prior rows, so their average is over a partial window. And 2026-05-05 has a NULL value — how the average treats that missing day is the heart of the question.

2. The SQL — trailing 7-day average via a row frame

SELECT metric_date, value,
  AVG(value) OVER (ORDER BY metric_date
                   ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS ma7
FROM metrics;

The OVER clause sorts by date and, for each row, defines a frame of itself plus the six rows before it. AVG is computed over the values inside that sliding frame.

The 2026-05-05 row has a NULL value, yet its ma7 is 105.00 — the average of the four numbers in its frame (100, 120, 90, 110). AVG skipped the NULL: it divides by the count of non-null values, not by the frame size. From 2026-05-07 on the frame holds 7 rows but only 6 have data, so each ma7 is a sum-over-6, not sum-over-7.

4. Strategic explanations

ROWS vs RANGE — the distinction that matters most

ROWS BETWEEN 6 PRECEDING counts six physical rows back. If a calendar day is missing entirely (no row at all), this frame still reaches back exactly six rows — spanning more than seven calendar days. To make the window mean "the last 7 calendar days" regardless of gaps, use a range frame: RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW. Knowing that ROWS counts rows and RANGE counts values along the ordering key is the senior distinction here.

The early rows are partial averages, not nulls

Day 1 has no preceding rows, so its ma7 is just its own value; day 2 averages 2 rows, and so on. The result is not NULL — the frame is simply smaller. Sometimes that ramp-up is undesirable (the first few points look artificially flat). If the chart should only show a true 7-day average, filter to rows where a full window exists, e.g. add COUNT(*) OVER (...) = 7 or start the series later.

NULL values shrink the denominator silently

AVG ignores NULLs: it sums non-null values and divides by their count. So a missing day does not pull the average toward zero — it just makes the average over fewer points. That is often acceptable, but it means a 7-row frame with one NULL is really a 6-point average. If a missing reading should count as zero, use AVG(COALESCE(value, 0)); if it should void the whole window, that needs explicit logic. Decide which, and say so.

The frame default trap — always specify it with ORDER BY

If you write AVG(value) OVER (ORDER BY metric_date) with no frame clause, the default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — a cumulative average from the start, not a 7-day window. Worse, with ties on metric_date the default RANGE includes all peer rows. Explicitly stating ROWS BETWEEN 6 PRECEDING AND CURRENT ROW, as this solution does, is exactly right and avoids that silent cumulative behaviour.

Ties on the ordering column

If two rows share a metric_date, the ROWS frame still treats them as distinct physical rows in some order, so "6 preceding" may split a same-date pair arbitrarily. For a daily metric you normally have one row per day; if you do not, pre-aggregate to one row per date first, or add a tie-breaker to the ORDER BY so the frame is deterministic.

Senior signal: know that ROWS counts physical rows while RANGE counts along the ordering key — gaps in the calendar make them differ — recognise that early rows yield partial averages and that AVG silently drops NULL values from the denominator, and always specify the frame explicitly because the default with ORDER BY is a cumulative average.
