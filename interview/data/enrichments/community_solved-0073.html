A retail analytics team wants the average order value, but only for the customers who matter for that metric — those who have placed more than five orders. The interviewer is checking whether you know that a per-group threshold belongs in HAVING, not WHERE, and whether you choose the right counting expression. The boundary customer sits at exactly five orders: do they get included? "More than five" says no, and the SQL must reflect that strict inequality.

1. The source data — customer_order

customer_idorder countorder_amount values
16100, 150, 200, 120, 180, 90
27250, 75, 300, 110, 220, 60, 95
33500, 400, 350

Customer 1 has exactly 6 orders (just over the line), customer 2 has 7, customer 3 has only 3. Customer 3 is the edge case: a high average (~417) but too few orders — the threshold must exclude them regardless of how large their spend is.

2. The SQL — group, then filter groups with HAVING

SELECT customer_id, AVG(order_amount) AS avg_order_amount
FROM customer_order
GROUP BY customer_id
HAVING COUNT(order_id) > 5;

3. The resulting analysis

customer_idavg_order_amountnote
1140.006 orders — sum 840 / 6
2158.577 orders — sum 1110 / 7

Customers 1 and 2 are returned. Customer 3 is excluded — only 3 orders — even though their average order value (417) is the highest. The HAVING filter is about order count, not order size.

4. Strategic explanations

Why the threshold must be in HAVING, not WHERE

WHERE filters rows before grouping — at that stage there is no per-customer count to test. HAVING filters the groups produced by GROUP BY, after the aggregates exist, so it can evaluate COUNT(order_id) > 5. Writing WHERE COUNT(order_id) > 5 is a hard syntax error in every dialect. The mental model: WHERE is a row gate, HAVING is a group gate — and the SQL clause order (FROM → WHERE → GROUP BY → HAVING → SELECT) tells you exactly when each can see what.

"More than five" is a strict inequality — > 5, not >= 5

This is the precision the interviewer watches for. "More than five" means six or more, so the predicate is > 5. A customer with exactly five orders fails it. If the spec were "at least five" the operator would be >= 5. Off-by-one errors on these boundary words are a classic way to silently get the wrong row set — read the threshold literally and pick the matching operator.

COUNT(order_id) vs COUNT(*) — here equal, but know the difference

order_id is the primary key, so it is never NULL — COUNT(order_id) and COUNT(*) give the identical result. But the habit matters: COUNT(*) counts rows; COUNT(col) counts non-NULL values of that column. If you counted a nullable column instead, rows with NULLs would silently drop and the threshold would misfire. For "number of orders", count the row (COUNT(*)) or a guaranteed-non-null key — never a nullable attribute.

The averaging trap — AVG and NULL amounts

AVG(order_amount) ignores rows where order_amount is NULL — it divides the sum by the count of non-NULL amounts, not by the total order count. So a customer with 6 orders where 2 have NULL amounts gets an average over 4 values, while HAVING COUNT(order_id) > 5 still counted all 6. If a NULL amount should be treated as 0, write AVG(COALESCE(order_amount, 0)). Worth stating: COUNT and AVG can disagree on which rows they "see", and that mismatch is a real reporting bug.

Decimal display — rounding the average

AVG over NUMERIC values can return many trailing digits (customer 2's true average is 158.5714...). For a report, wrap it: ROUND(AVG(order_amount), 2). This is presentation, not correctness — but a senior delivers a clean two-decimal currency figure rather than a 16-digit fraction.

Senior signal: place the count threshold in HAVING (group gate) not WHERE (row gate), and read "more than five" strictly as > 5. The senior nuance is noting that AVG silently skips NULL amounts while COUNT does not — so the two aggregates can disagree on the row set unless you COALESCE.
