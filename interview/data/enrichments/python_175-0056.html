Google's analytics teams profile every incoming dataset before modelling — a one-call summary of every numeric column catches scaling problems, skew, and silent nulls early. "Calculate Descriptive Statistics for Numeric Columns" tests whether you know pandas' aggregation surface well enough to assemble a richer profile than the default df.describe() — and whether you handle column selection and missing data correctly.

1. The input data — a DataFrame

regionpriceunits
East10.05
West20.03
East30.08
West40.0NaN
East50.04

region is non-numeric and must be excluded; units has one null to exercise the count vs nulls distinction.

2. The approach

select_dtypes(include='number') drops region, leaving only price and units. num.agg([...]) runs eight reducers at once and returns a stats-by-column frame; .T transposes it so each row is a column and each column is a statistic — the natural profile layout.

Three extras are bolted on: the 25th and 75th percentiles via quantile, their difference as the iqr (a robust spread measure), and nulls via isna().sum(). The count from agg counts non-null values, so count + nulls equals the row total — the two columns are complementary by design.

3. The output

countmeanmedianstdminmaxp25p75iqrnulls
price530.030.015.8110.050.020.040.020.00
units45.04.52.163.08.03.755.752.01

For price, std uses the sample formula (ddof=1): variance = ((20²+10²+0+10²+20²)/4) = 250, so std = √250 ≈ 15.81. For units, count is 4 (the NaN is skipped) while nulls is 1; mean of {5,3,8,4} = 5.0, median = 4.5. skew and kurt appear too — both are 0 for the perfectly symmetric price column.

4. Strategic explanations

Complexity

O(R · C) for R rows and C numeric columns — each statistic is a linear pass, and the constant factor is the eight-plus reducers. quantile sorts internally, so it is O(R log R) per column; that is the asymptotic bottleneck.

The idiomatic alternative — describe with percentiles
df.describe(percentiles=[.25, .5, .75]).T

describe gives count/mean/std/min/quartiles/max in one call and already auto-selects numeric columns. The hand-rolled version exists to add skew, kurt, iqr, and an explicit nulls column — knowing that describe covers 80% of this for free is the senior framing.

The edge case — no numeric columns, or single-row input

If every column is non-numeric, num is an empty frame and agg returns an empty result — no crash, but a downstream consumer may not expect zero rows. With a single row, std, skew, and kurt are NaN (sample variance needs n>1) — a real gotcha. Boolean columns count as numeric and sneak into the profile; integer columns with nulls get silently upcast to float.

The follow-up — group the profile by a category

"Now give me these stats per region." That is df.groupby('region').agg(...), which the interviewer uses to test whether you can pivot from a global summary to a grouped one without rebuilding the logic.

The trap — running agg on the raw DataFrame

Calling df.agg(['mean', ...]) without select_dtypes either throws on the string column or, worse, silently produces NaN/garbage for it — mean of strings is undefined. Equally, assuming count equals the row total is wrong the moment nulls exist; that is precisely why the separate nulls column has to be computed.

Senior signal: filter to numeric dtypes before aggregating, know that count excludes nulls so a dedicated nulls column is needed, and cite df.describe() as the built-in baseline this function deliberately extends.
