"Most recent per group" is the second-most-asked SQL pattern in interviews — and it shows up across wildly different business problems. Below: the same ROW_NUMBER() shape solving four high-stakes real-time scenarios. Once you see the pattern, you can apply it anywhere.

Scenario 1 — Marketing: the win-back campaign

The goal. Find the last order for customers who haven't shopped in over 6 months, then mail them a targeted discount.

The data — orders

customer_idorder_idorder_date
101A12026-01-10
101A22026-05-01 (recent)
102B12025-08-15 (old)
103C12025-09-20 (old)

The SQL

WITH LatestOrders AS (
  SELECT customer_id, order_id, order_date,
    ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC) AS rn
  FROM orders
)
SELECT customer_id, order_id, order_date
FROM LatestOrders
WHERE rn = 1
  AND order_date < CURRENT_DATE - INTERVAL '6 months';

The logic. PARTITION BY customer_id groups order history per customer. ORDER BY order_date DESC puts the most recent order at rn = 1. The outer WHERE filters: pull only the most-recent order, and only if it's older than 6 months.

Result: customers 102 and 103 are flagged for the win-back. Customer 101 is excluded because their most recent order (May 2026) is too fresh — they don't need a discount yet.

Scenario 2 — Logistics: real-time GPS "ping"

The goal. A delivery truck sends a GPS ping every 30 seconds. The customer-facing map only needs the last coordinate to render the live truck position.

The data — pings

delivery_idlat_longping_time
TRUCK_734.05, -118.2412:00:01
TRUCK_734.06, -118.2512:00:31 (live)
TRUCK_940.71, -74.0011:59:45 (live)

The SQL

SELECT delivery_id, lat_long, ping_time
FROM (
  SELECT *,
    ROW_NUMBER() OVER (PARTITION BY delivery_id ORDER BY ping_time DESC) AS rn
  FROM pings
) t
WHERE rn = 1;

The logic. This handles high-velocity data. In a table with millions of pings, the query "flattens" history so the UI only sees one row per truck. ROW_NUMBER() is safer than MAX(ping_time) here because it pulls the associated lat_long column on the same row — no second JOIN required.

Scenario 3 — Fraud: point of compromise

The goal. A user reports their card stolen at 2:00 PM. The risk team needs the last legitimate transaction before that report to see where the thief's spree started.

The data — orders joined to reports

user_idorder_idorder_date
55X_9910:00 AM
55X_10001:30 PM (last legit)
55X_10102:15 PM (after report — fraud?)

The SQL

SELECT o.user_id, o.order_id, o.order_date
FROM (
  SELECT o.*,
    ROW_NUMBER() OVER (PARTITION BY o.user_id ORDER BY o.order_date DESC) AS rn
  FROM orders o
  JOIN reports r ON o.user_id = r.user_id
  WHERE o.order_date < r.report_time          -- only look at events BEFORE report
) o
WHERE rn = 1;

The logic. This is a point-in-time analysis. By joining the order history to the report timestamp and filtering order_date < report_time, we "cut off" everything after the theft. rn = 1 then identifies the very last safe transaction — the cliff edge before fraud started. That row's location, merchant, and time are the breadcrumbs the fraud team uses to triangulate the compromise point.

Scenario 4 — Growth: identify "power shoppers"

The goal. Find customers whose last 3 orders all happened within a 7-day window — they're on a streak and warrant a VIP-tier upgrade.

The SQL

SELECT customer_id, order_date, two_orders_ago
FROM (
  SELECT customer_id, order_date,
    -- Reach back two rows in the customer's history
    LAG(order_date, 2) OVER (PARTITION BY customer_id ORDER BY order_date ASC) AS two_orders_ago,
    ROW_NUMBER()      OVER (PARTITION BY customer_id ORDER BY order_date DESC) AS rn
  FROM orders
) t
WHERE rn = 1                                            -- only their current state
  AND order_date - two_orders_ago <= INTERVAL '7 days'; -- 3 orders inside a week

The logic. This combines LAG with the ranking pattern:

  1. Look at the latest order (rn = 1).
  2. Reach back two rows via LAG(order_date, 2) — the order from two purchases ago.
  3. Compare the most-recent date to the two-back date.

If those three orders happened within 7 days, the customer is a power shopper. This is materially harder than the first three scenarios because it looks at the relationship between rows, not just a single row's attributes.

The unifying pattern

Every scenario above uses the same shape: ROW_NUMBER() OVER (PARTITION BY entity ORDER BY ts DESC) + WHERE rn = 1. The variation is what you do around it:

  • Marketing adds an outer date filter (... AND order_date < CURRENT_DATE - 6 months).
  • Logistics uses it directly for live-state flattening.
  • Fraud wraps the partition in a JOIN that pre-filters by report time.
  • Growth layers LAG on top to compare the latest row against earlier rows.
Senior signal: the interviewer wants to hear "this is the Latest-N per group pattern" — and that you can recognise it across domains. Bonus points: explain why ROW_NUMBER() beats MAX(date) when you also need the row's other columns (you don't have to re-join), and why ties matter (use RANK if ties should both qualify, deterministic tie-breaker if you want exactly one).
