Wix's commerce ops team runs an inventory aging report: how long has each product been sitting in the warehouse since it was received? The classic answer is a CASE expression that drops each product into a 0-30 / 31-60 / 61-90 / 90+ bucket. It looks trivial, but the boundary conditions, the timestamp-to-date cast, and the dependence on "today" are exactly where this question earns its keep.

Evaluated against today (2026-05-15), the ages are 14, 35, 71, 120 and 176 days. Product 2 at 35 days sits just past the 30-day line, and product 3 at 71 days sits just past the 60-day line — the boundary cases that test whether the < comparisons land where you think.

2. The SQL — CASE aging buckets

SELECT product_id,
  CASE WHEN CURRENT_DATE - received_at::date < 30 THEN '0-30'
       WHEN CURRENT_DATE - received_at::date < 60 THEN '31-60'
       WHEN CURRENT_DATE - received_at::date < 90 THEN '61-90'
       ELSE '90+' END AS aging_bucket
FROM inventory;

CURRENT_DATE - received_at::date yields an integer day count. CASE evaluates branches top to bottom and stops at the first true one, so each later branch implicitly assumes the earlier thresholds were not met.

4. Strategic explanations

The label-vs-logic mismatch — the real trap

The bucket labels say "31-60" and "61-90", but the logic uses < 30, < 60, < 90. A product aged exactly 30 days fails < 30 and falls into the "31-60" bucket — whose label implies it starts at 31. The label is off by one at every boundary. To make label and logic agree, the cuts should be <= 30, <= 60, <= 90:

CASE WHEN age <= 30 THEN '0-30'
     WHEN age <= 60 THEN '31-60'
     WHEN age <= 90 THEN '61-90'
     ELSE '90+' END

Catching this off-by-one and confirming the intended boundary with the interviewer is the senior detail of the whole question.

"As of when?" — CURRENT_DATE makes the report non-reproducible

CURRENT_DATE means every run gives a different answer, and you cannot re-create last month's report. For a snapshot report, pass the as-of date as a parameter (or join to a reporting-date table) so the output is deterministic and auditable. "Aging as of report date X" beats "aging as of whenever this happened to run".

The timestamp cast — why ::date matters

received_at is a timestamp. CURRENT_DATE - received_at (no cast) mixes a date with a timestamp and yields an interval with hours and minutes, which compares awkwardly against the integer 30. The ::date cast truncates the time-of-day so the subtraction is a clean integer day count. Drop the cast and the boundary behaviour gets fuzzy by up to a day.

A set-based aging summary

The per-product list is the raw report; leadership usually wants the distribution. Wrap it and count:

SELECT aging_bucket, COUNT(*) AS n
FROM (
  SELECT CASE WHEN CURRENT_DATE - received_at::date <= 30 THEN '0-30'
              WHEN CURRENT_DATE - received_at::date <= 60 THEN '31-60'
              WHEN CURRENT_DATE - received_at::date <= 90 THEN '61-90'
              ELSE '90+' END AS aging_bucket
  FROM inventory
) t
GROUP BY aging_bucket
ORDER BY aging_bucket;
Edge cases: NULL and future-dated receipts

If received_at is NULL, the subtraction is NULL, every WHEN is unknown, and the row falls through to the ELSE — silently mislabelled "90+" rather than flagged as missing data. And a receipt dated in the future (a data-entry error) yields a negative age, which still satisfies < 30 and lands in "0-30". Add an explicit WHEN received_at IS NULL THEN 'unknown' and consider guarding against negative ages.

Senior signal: spot that the bucket labels are off by one against the < thresholds and align them with <=, replace CURRENT_DATE with an as-of parameter so the report is reproducible, justify the ::date cast on the timestamp, and handle NULL or future-dated receipts explicitly instead of letting them fall through the ELSE.
