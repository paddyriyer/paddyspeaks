BMW's connected-vehicle BI team ingests a daily sales feed from dealerships worldwide. On any given day a few regions simply don't report — a network hiccup, a late close, a regional holiday — and those gaps land in the warehouse as NULL amounts. The dashboard that rolls these up does arithmetic, and arithmetic plus NULL equals NULL. This question looks like one tiny function call, but it is really about whether you know what a missing number means before you decide how to fill it.

1. The source data — daily_sales

dregionamount
2025-03-01North1200
2025-03-01SouthNULL
2025-03-02NorthNULL
2025-03-02South900
2025-03-03North1500
2025-03-03SouthNULL

Three of the six rows have a NULL amount. The question is whether a non-reporting region means "they sold €0 that day" or "we genuinely don't know". The SQL below picks the first interpretation — correct only if that is what the business intends.

2. The SQL — COALESCE(amount, 0)

SELECT d, region, COALESCE(amount,0) AS amount FROM daily_sales;

COALESCE returns its first non-NULL argument. For a row with a real amount it passes the value straight through; for a NULL row it substitutes 0. No GROUP BY — this is a row-by-row projection that simply cleans the column.

3. The resulting analysis

dregionamount
2025-03-01North1200
2025-03-01South0
2025-03-02North0
2025-03-02South900
2025-03-03North1500
2025-03-03South0

Every NULL has become 0; the populated rows are untouched. The output now sums and averages without infecting the result with NULL — SUM over this column gives 3600, a finite number a dashboard can render.

4. Strategic explanations

The interviewer's first follow-up: "is zero the right fill?"

Filling with 0 is a business decision disguised as a syntax choice. If "South didn't report" really means "South sold nothing", 0 is correct and SUM is honest. But if it means "data is in flight", a 0 understates the total and quietly drags down any AVG. The senior answer names both readings and asks which one holds. Sometimes the right move is to leave the NULL alone and let AVG skip it, or carry the last known value forward.

COALESCE vs the gotchas around it

COALESCE is ANSI SQL and portable. Two cousins are not drop-in equivalents:

-- NVL is Oracle-only; ISNULL is SQL Server-only and ISNULL takes exactly 2 args
COALESCE(amount, 0)          -- portable, n-ary
-- NULLIF does the OPPOSITE: turns a value INTO null
NULLIF(amount, 0)            -- returns NULL when amount = 0

Reach for COALESCE by default; it accepts any number of fallbacks — COALESCE(a, b, c, 0).

The type trap — all branches must agree

COALESCE's arguments must share a common type. COALESCE(amount, 0) is fine because 0 coerces to numeric. But COALESCE(amount, 'N/A') would fail — you cannot fall back from a number to a string. If you need a text placeholder, cast the number first: COALESCE(amount::text, 'N/A').

Filling on the way in vs filling on the way out

This query patches the value at read time, leaving the warehouse honest about what is missing — usually the right call, because the next analyst can choose a different fill. The alternative, UPDATE daily_sales SET amount = 0 WHERE amount IS NULL, destroys the distinction between "zero sales" and "no data" permanently. Patch in the SELECT, not in the table.

Edge case: a region that never reports

COALESCE only touches rows that exist. If a region is entirely absent from daily_sales for a date, no row means nothing to coalesce — the gap is invisible. Surfacing those requires a calendar × region grid LEFT JOINed to the feed, and then COALESCE on the join result.

Senior signal: treat the NULL-to-0 substitution as a business decision — confirm that a non-reporting region means zero sales, not unknown — reach for portable COALESCE over NVL/ISNULL, keep the fill in the query rather than mutating the table, and note that COALESCE cannot rescue rows that are missing entirely.
