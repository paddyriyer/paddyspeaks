Customer Success teams synthesize multiple signals into one health score: logins + tickets + NPS + renewal-pct. Single number drives the renewal prediction; breakdowns drive the conversation.

2. The SQL — weighted composite

SELECT customer_id, login_30d_count, support_tickets_open, nps, last_renewal_pct,
  ROUND(
    MIN(login_30d_count * 1.0 / 10, 1.0) * 30
    + (CASE WHEN support_tickets_open <= 1 THEN 25 WHEN support_tickets_open <= 3 THEN 15 ELSE 0 END)
    + (nps + 100) / 200.0 * 25
    + last_renewal_pct * 0.20
  , 1) AS health_score
FROM customer_signals ORDER BY health_score DESC;

3. Strategic explanations

The MIN-cap on login bonus

Logins beyond 10/month don't add value (already engaged). MIN(count/10, 1.0) caps the contribution at 1.0 = 30 points. SQLite uses MIN; Postgres has LEAST.

The 30/25/25/20 weights

Calibrated weights summing to 100. CS teams tune these via regression on observed renewal vs each component. Production weights drift quarterly as the customer base changes.

The NPS scaling

NPS is -100 to +100. (nps + 100) / 200 scales to 0-1. Multiply by 25 for the points. Symmetric: -100 = 0 points, 0 = 12.5, +100 = 25.

Composite vs component view

The number alone hides why. CS reps need: "score 65 — but tickets are high (5 open) AND NPS dropped from 50 to 20". Always emit components alongside.

Predictive vs descriptive

This is a descriptive composite. Predictive (ML model) outperforms — but is opaque. Best-of-both: ML for the prediction, composite for the human-readable "why".

Senior signal: cap engagement contributions to prevent gaming, calibrate weights via regression, expose components alongside the score, and contrast descriptive composite vs ML predictive.
