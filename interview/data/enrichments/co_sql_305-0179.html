Splunk's revenue team wants a single report that shows sales at three levels at once — every region×product cell, each region's subtotal, and each product's subtotal. The reference solution stitches three GROUP BY queries together with UNION ALL. It produces the right numbers, but the interviewer is watching whether you recognise this as a hand-rolled GROUPING SETS and know the trap that emulation hides.

1. The source data — sales

regionproductamount
WestWidget100
WestGadget200
EastWidget150
EastGadget50

Four clean rows — two regions, two products. Deliberately tidy: the interest here is not messy data, it is the shape of the multi-level output and the NULLs that the subtotal rows carry.

2. The SQL — UNION ALL of three groupings

SELECT region, product, SUM(amount) AS total
FROM sales GROUP BY region, product
UNION ALL
SELECT region, NULL, SUM(amount) FROM sales GROUP BY region
UNION ALL
SELECT NULL, product, SUM(amount) FROM sales GROUP BY product
ORDER BY region, product;

Branch 1 gives the leaf cells; branch 2 gives per-region subtotals (with product = NULL); branch 3 gives per-product subtotals (with region = NULL). UNION ALL stacks all three.

3. The resulting analysis

EastWidget150
EastNULL200
WestGadget200
WestWidget100
WestNULL300
NULLGadget250
NULLWidget250

Eight rows at three levels. The product = NULL rows are region subtotals (East = 200, West = 300); the region = NULL rows are product subtotals (Gadget = 250, Widget = 250). A NULL in a dimension column here means "all values of this dimension" — that overloading is the catch.

4. Strategic explanations

This is GROUPING SETS — the one-pass native form

Three UNION ALL branches each scan sales independently — three table scans for one report. Postgres expresses the same intent in a single statement, single scan:

SELECT region, product, SUM(amount) AS total
FROM sales
GROUP BY GROUPING SETS ((region, product), (region), (product))
ORDER BY region, product;

Each parenthesised tuple is one grouping level; () would add the grand total. It is shorter, the optimiser scans once, and it states the intent directly. Naming GROUPING SETS as the right tool — and CUBE/ROLLUP as its relatives — is the senior move.

The trap: a real NULL in the data is indistinguishable from a subtotal

Both the emulation and GROUPING SETS mark subtotal rows by putting NULL in the rolled-up column. But what if region itself contains genuine NULLs — sales with no region recorded? Then a leaf row (NULL, 'Widget', …) and the product-subtotal row (NULL, 'Widget', 250) are visually identical, and any consumer reading the result cannot tell a real NULL from "all regions". The fix is the GROUPING() function, which returns 1 for a column rolled up in that row and 0 otherwise:

SELECT region, product, SUM(amount) AS total,
       GROUPING(region)  AS region_is_total,
       GROUPING(product) AS product_is_total
FROM sales
GROUP BY GROUPING SETS ((region, product), (region), (product));

The UNION ALL emulation has no equivalent — it cannot distinguish the two cases at all. That is the concrete reason to prefer the native syntax beyond mere tidiness.

UNION ALL, not UNION

The branches are designed to produce different rows, but UNION would still pay for a dedup pass — and if a region subtotal coincidentally equalled a leaf total, UNION could collapse two semantically different rows into one. UNION ALL is correct and cheaper. (The reference uses it correctly.)

ORDER BY with NULLs — where do subtotals land?

ORDER BY region, product sorts NULL last by default in Postgres, so each region's subtotal row sorts after its leaf rows, and the product subtotals sink to the bottom — which happens to read naturally here. But the placement is a side effect of NULL-ordering, not a designed layout. If a report needs subtotals in a specific position, sort on GROUPING() flags explicitly rather than relying on where NULLs fall.

Consistency across branches

Each UNION ALL branch must emit the same column count and compatible types in the same order — here (region, product, total). The literal NULL placeholders are untyped; Postgres infers their type from the other branches. If the branches ever drift (a reordered column, a renamed alias), the union silently misaligns data into the wrong column. A single GROUPING SETS statement removes that whole class of maintenance bug.

Senior signal: recognise the three-way UNION ALL as a hand-rolled GROUPING SETS, replace it with the single-scan native syntax, and — the killer detail — use the GROUPING() function to distinguish a subtotal row's NULL from a genuine NULL in the data, something the UNION ALL form cannot do.
