Microsoft's compensation team wants the three highest salary tiers in the org — not the three highest-paid people, the three distinct salary levels. The distinction is the whole question. If five people earn the top number and you forget DISTINCT, "top three salaries" returns that one number three times. An interviewer is watching whether you read "salaries" as values or as rows.

Six employees but only four distinct salaries — 150K and 120K each occur twice. Those duplicates are the interview: they decide whether "top three" means three values or three people.

2. The SQL — DISTINCT + ORDER BY + LIMIT

SELECT DISTINCT salary FROM employees ORDER BY salary DESC LIMIT 3;

DISTINCT collapses the duplicate salaries to one row each, ORDER BY salary DESC sorts highest first, and LIMIT 3 keeps the top three values.

Three rows, three distinct values. Without DISTINCT the query would return 150000, 150000, 120000 — only two real tiers, the third row a duplicate. DISTINCT is doing the load-bearing work; drop it and the answer is silently wrong on any data with salary ties.

4. Strategic explanations

Why DISTINCT is mandatory, not optional

"Top three salaries" is a question about distinct compensation levels. LIMIT 3 alone counts rows; if the top earners are tied, those rows are the same number repeated. The order of operations is DISTINCT (dedupe) → ORDER BY → LIMIT, so the limit is applied to already-unique values. State this explicitly — it is the single thing the question tests.

The window-function equivalent — DENSE_RANK

DENSE_RANK() expresses the same intent and extends cleanly to "top N":

SELECT salary FROM (
  SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rk
  FROM employees
) t WHERE rk <= 3;

DENSE_RANK gives tied salaries the same rank with no gaps (1, 1, 2, 2, 3) — so rk <= 3 picks exactly the top three distinct values. Note this returns one row per matching distinct salary value; if you want the people, select name too and drop the outer dedupe.

The RANK trap — gaps skip tiers

Swapping RANK for DENSE_RANK quietly breaks it. RANK leaves gaps after ties: with two people at 150K and two at 120K, the ranks are 1, 1, 3, 3, 5. Filtering rk <= 3 would then capture 150K and 120K but miss 90K entirely, returning only two tiers. For "Nth distinct value" questions, DENSE_RANK is the safe default; RANK answers a different question (competition placing).

People vs values — clarify before coding

If the recruiter actually wants the three highest-paid employees, ties change the row count and DISTINCT is wrong — you would want all people at the top salaries:

-- everyone in the top 3 salary tiers, names included
SELECT name, salary FROM (
  SELECT name, salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rk
  FROM employees
) t WHERE rk <= 3;
-- returns A, B, C, D, E — five people across three tiers

One sentence to the interviewer — "three salary values, or all employees at those values?" — saves writing the wrong query.

Edge cases: fewer than three tiers, and NULL salaries

If the company has only two distinct salaries, the query returns two rows, not three — LIMIT is a cap, not a guarantee. If a consumer needs exactly three, that is a different (and odd) requirement. NULL salaries are another decision: ORDER BY salary DESC places NULLs first in Postgres by default, so an unpaid contractor row could occupy a top slot — add NULLS LAST or filter WHERE salary IS NOT NULL to keep them out.

Senior signal: recognise that DISTINCT is the entire point of "top three salaries", reach for DENSE_RANK (not RANK) as the window-function form, clarify whether the ask is three values or three people, and pre-empt the NULL-salary sort-order surprise.
