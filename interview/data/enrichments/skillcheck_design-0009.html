The transactional outbox pattern solves the dual-write problem: how does a service atomically update its own database and publish a domain event to a message broker? The answer is that it does not write to Kafka at all — it writes an outbox row in the same local transaction as the business row, then a relay process reads the outbox and publishes to Kafka. The interviewer is probing whether you understand the ordering, idempotency, and relay-observability columns that make this pattern actually reliable at scale.

1. The requirements — what the design must capture

  • Ordered delivery: the relay must publish events in the order they were committed. outbox_id BIGSERIAL provides a monotonic cursor the relay uses to advance its position.
  • Idempotent publishing: if the relay crashes after publishing but before marking the row as sent, it will re-publish. event_id UUID UNIQUE lets the consumer detect and discard duplicates.
  • Kafka key for per-aggregate ordering: Kafka preserves order within a partition; routing all events for the same aggregate_id to the same partition (by using it as the Kafka message key) ensures per-entity ordering.
  • Relay observability: published_at, publish_attempts, last_error expose relay progress and failure detail without requiring a separate monitoring table.
  • Event routing metadata: aggregate_type and event_type allow the relay to route events to the correct Kafka topic without deserialising the payload.
  • Schema evolution: schema_version lets consumers apply the correct deserialiser for each event without coupling the relay to payload internals.

2. The DDL — the CREATE TABLE

CREATE TABLE outbox (
  outbox_id          BIGSERIAL    PRIMARY KEY,                -- monotonic id for ordering & relay cursor
  event_id           UUID         NOT NULL UNIQUE,            -- producer-side idempotency key
  aggregate_type     VARCHAR(64)  NOT NULL,                   -- e.g. 'order', 'invoice'
  aggregate_id       VARCHAR(64)  NOT NULL,                   -- business key (used as Kafka key for ordering)
  event_type         VARCHAR(64)  NOT NULL,                   -- e.g. 'order.placed'
  payload            JSONB        NOT NULL,                   -- the event body
  headers            JSONB,                                   -- correlation / trace ids
  schema_version     SMALLINT     NOT NULL DEFAULT 1,
  created_at         TIMESTAMPTZ  NOT NULL DEFAULT now(),
  -- relay state (or store this in a separate cursor table)
  published_at       TIMESTAMPTZ,
  publish_attempts   INT          NOT NULL DEFAULT 0,
  last_error         TEXT
);

-- Relay index: cheapest scan of unpublished events in insertion order.
CREATE INDEX ix_outbox_unpublished
  ON outbox (outbox_id)
  WHERE published_at IS NULL;

-- Useful for partition-by-aggregate ordering when fan-out matters.
CREATE INDEX ix_outbox_aggregate ON outbox (aggregate_type, aggregate_id, outbox_id);

4. Strategic explanations

Why the outbox solves the dual-write problem

A service that writes to its database and then writes to Kafka in separate operations faces a gap: the database write can succeed while the Kafka write fails (or vice versa), leaving the two systems permanently inconsistent. The outbox pattern eliminates this gap: the outbox row is written in the same database transaction as the business row. Either both commit or both roll back. The relay’s job is to drain the outbox — if it fails, it retries; the events are never lost because they are durably stored in Postgres.

Relay cursor strategy

The relay maintains a cursor: the highest outbox_id it has successfully published. Each polling cycle runs SELECT * FROM outbox WHERE outbox_id > :cursor AND published_at IS NULL ORDER BY outbox_id LIMIT :batch_size. The partial index ix_outbox_unpublished on (outbox_id) WHERE published_at IS NULL makes this scan cheap: once a row is published, it leaves the index and never appears in relay scans again. Without this index, every poll would scan all historical rows.

Idempotency: event_id is the dedup key

If the relay publishes an event and Kafka acknowledges it but the relay crashes before updating published_at, the event will be published again on the next relay restart. The event_id UUID, carried in the Kafka message header, lets consumers implement at-least-once-with-dedup: they record processed event_ids in a short-lived cache or a Kafka Streams state store, and discard events whose event_id has already been processed. This gives effective exactly-once processing without requiring Kafka exactly-once semantics end-to-end.

Aggregate ordering via Kafka partition key

Kafka guarantees order within a partition. By using aggregate_id as the Kafka message key, the relay routes all events for the same entity (e.g., order_id=42) to the same partition, ensuring a consumer sees OrderPlaced before OrderShipped for that order. Using aggregate_type + aggregate_id as the key (e.g., “order:42”) allows cross-entity type routing to different topic partitions while preserving intra-entity order.

Outbox purging and retention

The outbox will grow indefinitely unless published rows are deleted. Options: (1) a scheduled job deletes rows where published_at < NOW() - INTERVAL ’7 days’; (2) partition the outbox by created_at date and DROP old partitions. Option 1 is simpler; option 2 is faster for high-volume tables. Retaining some published history (e.g., 7 days) aids debugging without bloating the table.

Alternative: cursor table

The inline relay-state columns (published_at, publish_attempts, last_error) simplify the schema. An alternative is a separate outbox_relay_cursor table that stores only the relay’s position, leaving the outbox table append-only and update-free. The tradeoff: the separate cursor table requires two queries per relay cycle (read cursor, then query outbox) but eliminates UPDATE contention on the outbox in high-throughput scenarios.

Senior signal: explain the dual-write problem and why the outbox row must be in the same transaction as the business row; justify the partial index on (outbox_id) WHERE published_at IS NULL as the mechanism that keeps relay polls O(unpublished) not O(all-time); describe the event_id UUID as the consumer-side dedup key for at-least-once-with-dedup semantics; and mention aggregate_id as the Kafka partition key for per-entity ordering — these four points show you understand the pattern as a distributed-systems contract, not just a table schema.
