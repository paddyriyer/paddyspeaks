Workforce demographic reports are a staple of HR analytics dashboards. This question probes conditional aggregation — the ability to pivot one column's values into separate computed columns within a single GROUP BY pass — and forces you to handle the asymmetric edge case where a department is entirely one-gender: the missing average must surface as NULL, not zero, and the gap percentage must follow suit.

Key edge cases: Ops has zero female employees (female avg and gap must be NULL); HR has zero male employees (male avg and gap must be NULL). These are the discriminating rows that separate candidates who handle NULL correctly from those who do not.

2. The SQL — conditional aggregation with FILTER and NULL-safe gap

SELECT dept,
       COUNT(*) AS total_headcount,
       COUNT(*) FILTER (WHERE gender = 'F') AS num_female,
       COUNT(*) FILTER (WHERE gender = 'M') AS num_male,
       ROUND((100.0 * COUNT(*) FILTER (WHERE gender = 'F') / COUNT(*))::numeric, 2) AS female_share_pct,
       AVG(salary) FILTER (WHERE gender = 'F')::numeric(10,2) AS female_avg_salary,
       AVG(salary) FILTER (WHERE gender = 'M')::numeric(10,2) AS male_avg_salary,
       CASE
         WHEN AVG(salary) FILTER (WHERE gender = 'M') IS NULL
           OR AVG(salary) FILTER (WHERE gender = 'F') IS NULL
         THEN NULL
         ELSE ROUND(((AVG(salary) FILTER (WHERE gender = 'M') - AVG(salary) FILTER (WHERE gender = 'F'))
               / AVG(salary) FILTER (WHERE gender = 'M') * 100)::numeric, 2)
       END AS salary_gap_pct
FROM employee
GROUP BY dept
ORDER BY dept;

Every gender split is computed inside the same GROUP BY dept sweep using FILTER (WHERE gender = …) clauses. When no rows match the filter, COUNT(*) FILTER returns 0 and AVG returns NULL — the exact behaviour required.

4. Strategic explanations

FILTER (WHERE …) vs CASE inside an aggregate

COUNT(*) FILTER (WHERE gender = 'F') is the SQL:2003 standard idiom and reads clearly. The equivalent CASE-inside-aggregate form — SUM(CASE WHEN gender = 'F' THEN 1 ELSE 0 END) — works in every engine but is more verbose and requires an ELSE 0 branch for counts. For AVG specifically the CASE form needs care: AVG(CASE WHEN gender = 'F' THEN salary END) relies on ELSE NULL being the default (which is correct but easy to forget). The FILTER clause is cleaner and PostgreSQL, DuckDB, and SQLite all support it; MySQL and SQL Server do not — for portability you fall back to CASE.

Why AVG returns NULL when no rows match, and why that is correct

AVG aggregates over zero rows returns NULL per the SQL standard — there is no meaningful average of an empty set. This is the desired behaviour here: a NULL average signals "no data" rather than "zero salary". The COUNT(*) FILTER path returns 0 (not NULL) for count, which is also correct — zero people is a valid count.

The salary_gap_pct NULL guard

The CASE WHEN … IS NULL … THEN NULL wrapper is necessary because arithmetic with NULL propagates to NULL automatically for the subtraction and division, but explicitly checking in a CASE makes the intent unambiguous. The denominator is the male average; dividing by zero is impossible here (NULL is returned instead), but worth noting: if the male count were 0, the male average is NULL and the CASE returns NULL before division ever occurs — no division-by-zero risk.

Ordering and the alphabetical edge

ORDER BY dept yields alphabetical order: Eng, HR, Ops, Sales. If the department values could be NULL (a real-world possibility), you would add NULLS LAST to keep them at the bottom rather than the top (PostgreSQL default for ascending order is NULLS LAST; MySQL sorts NULLs first).

Performance and dialect notes

A single GROUP BY dept scan is optimal — one pass over the table. Avoid the naive approach of joining three subqueries (all, female-only, male-only) because that reads the table three times. In BigQuery the equivalent is COUNTIF(gender = 'F') and AVG(IF(gender = 'F', salary, NULL)). In SQL Server and MySQL you use the CASE-inside-aggregate pattern. Snowflake supports FILTER as of 2023.

Senior signal: collapse all gender-specific metrics into one GROUP BY pass using FILTER (WHERE gender = …); rely on AVG returning NULL over zero rows as the natural signal for a missing gender; guard the gap percentage with an explicit CASE … IS NULL … THEN NULL to document intent; and note the CASE-inside-aggregate fallback for engines that lack FILTER.
