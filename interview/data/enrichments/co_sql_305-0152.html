Salesforce's CRM team wants a customer roster with each customer's orders attached — and crucially, customers who have never ordered must still appear, flagged as having no activity. That single requirement is the whole question: it forces a LEFT JOIN over an INNER JOIN, and an interviewer asking this is testing whether you reach for the right join by reflex and can read the NULLs it produces.

1. The source data — customers and orders

Acme has two orders, Globex has one, and Initech has none. Initech is the entire point: an inner join would erase it; a left join must keep it with NULL order columns.

2. The SQL — LEFT JOIN customers to orders

SELECT c.*, o.order_id, o.amount
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id;

customers is the left (preserved) table; every customer row survives, and orders columns are filled where a match exists and NULL where it does not.

4. Strategic explanations

LEFT JOIN vs INNER JOIN — the one-word difference that drops a customer

Swap LEFT JOIN for JOIN and Initech disappears: an inner join emits a row only where both sides match, and Initech has no order. The result would silently shrink from 4 rows to 3, and "customers with no orders" — often the most important segment for a churn or re-engagement analysis — would be invisible. The choice of join is the answer to this question. Say out loud: "the requirement to keep zero-order customers is precisely why this is a LEFT JOIN."

The NULL-in-the-WHERE trap that turns it back into an inner join

The single most common mistake: filtering on a right-table column in the WHERE clause. WHERE o.amount > 50 evaluates to unknown for Initech's NULL amount, so Initech is dropped — the left join silently degrades into an inner join. If a condition on the right table must coexist with preserving all customers, put it in the ON clause, not the WHERE:

-- keeps Initech; filters which ORDERS attach
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id AND o.amount > 50;

-- drops Initech: the WHERE kills the NULL row
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.amount > 50;

The only legitimate right-table WHERE on a left join is the deliberate anti-join: WHERE o.order_id IS NULL to get exactly the never-ordered customers.

The row-fan-out and one-row-per-customer follow-up

Acme's two rows are correct for a detail listing, but if the consumer wants one row per customer, the join must be followed by aggregation — and you must COUNT/SUM a right-table column so a no-order customer reads as 0, not 1:

SELECT c.customer_id, c.name,
       COUNT(o.order_id)            AS order_count,
       COALESCE(SUM(o.amount), 0)   AS total_spent
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
GROUP BY c.customer_id, c.name;

COUNT(o.order_id) counts non-NULL keys, so Initech gets 0; COUNT(*) would wrongly give it 1. SUM over no rows is NULL, hence the COALESCE.

SELECT c.* is fragile

c.* is fine in an ad-hoc query but a liability in shipped code: add or reorder a column in customers and every downstream consumer's column layout shifts underneath them. For anything persisted — a view, an ETL step — enumerate the columns explicitly. It also makes the two amount columns in the schema (one on each table) less likely to collide confusingly.

Direction matters — which table is preserved

A LEFT JOIN keeps the left table whole. Writing orders LEFT JOIN customers would instead preserve all orders and surface orphan orders (an order whose customer was deleted) — a different and also useful query, but not this one. customers LEFT JOIN orders answers "all customers"; flip it for "all orders". A FULL OUTER JOIN would surface both anomalies at once.

Senior signal: reach for LEFT JOIN because the spec demands zero-order customers, never put a right-table filter in the WHERE (it silently re-creates an inner join — use the ON clause), use COUNT(o.order_id) + COALESCE(SUM(...),0) when collapsing to one row per customer, and know that join direction decides which side is preserved.
