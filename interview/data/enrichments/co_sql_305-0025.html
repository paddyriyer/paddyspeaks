Crypto.com runs a tournament leaderboard and needs every player's rank by score. The reference solution computes rank the pre-window-function way: for each player, count how many players scored strictly higher. It produces the right numbers — but the interviewer is watching whether you recognise this as an O(n²) self-join in disguise, and whether you know which ranking semantics it actually implements.

1. The source data — scores

Players 1 and 3 are tied at 95 — the top score. That tie is the whole interview: it decides whether the player below them is ranked 2 or 3.

2. The SQL — COUNT(higher) + 1

SELECT player_id, score,
  (SELECT COUNT(*)+1 FROM scores s2 WHERE s2.score > s1.score) AS rank
FROM scores s1 ORDER BY rank;

For each player s1, the correlated subquery counts how many rows have a strictly greater score, then adds 1. A player with nobody above them gets rank = 1.

Both 95-scorers get rank 1 — neither has anyone strictly above. Then player 2 gets rank 3, not 2: two players outscore them. The count of strict-greater values is exactly RANK() semantics — ties share a rank and the next rank is skipped. The "2" is gone.

4. Strategic explanations

This is RANK() — the gap-after-ties behaviour is deliberate

"Count how many are strictly higher, plus one" is the textbook definition of competition ranking. Ties get the same rank; the rank after a tie jumps by the size of the tie group (ranks 1, 1, 3). The window-function equivalent is one line and far clearer about intent:

SELECT player_id, score,
       RANK() OVER (ORDER BY score DESC) AS rank
FROM scores
ORDER BY rank;

If the leaderboard should instead read 1, 1, 2 — no gap — that is DENSE_RANK(), and the subquery would have to count distinct higher scores, not rows. Naming which one the product wants is the senior move.

The performance trap — O(n²)

The correlated subquery runs once per player, and each run scans the whole table: with n players that is roughly n² comparisons. Fine for a 5-row demo, quadratic and unacceptable for a million-row leaderboard. RANK() OVER (…) sorts once — O(n log n) — and assigns ranks in a single pass. On any real table the window function is not just cleaner, it is the only viable answer.

The type label is misleading

The question is tagged "SQL / Arrays", but there is no array work here — it is a ranking problem. Do not be lured into array_agg or unnest gymnastics; recognise it for what it is. (Mislabelled categories happen; solve the question, not the tag.)

ROW_NUMBER is the wrong tool — know why

A common mistake is ROW_NUMBER() OVER (ORDER BY score DESC). That assigns 1, 2, 3, 4, 5 — it would give the two tied 95-scorers different numbers (1 and 2), arbitrarily breaking the tie by physical row order. ROW_NUMBER is for "pick exactly one row per group"; for genuine ranking where ties must share a position, it is wrong. RANK / DENSE_RANK respect ties; ROW_NUMBER ignores them.

NULL scores and the strict inequality

s2.score > s1.score is unknown whenever either side is NULL, so a NULL-score player counts nobody as above them and lands at rank 1 — almost certainly wrong. And NULL never counts as "higher" for anyone else either. Decide explicitly: filter WHERE score IS NOT NULL, or use RANK() OVER (ORDER BY score DESC NULLS LAST) so unscored players sink to the bottom.

Senior signal: identify "count strictly higher + 1" as RANK() semantics with skipped ranks after ties, replace the O(n²) correlated self-join with RANK() OVER (ORDER BY score DESC), distinguish RANK/DENSE_RANK/ROW_NUMBER by how each treats ties, and handle NULL scores deliberately rather than letting them default to rank 1.
