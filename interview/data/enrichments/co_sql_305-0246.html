Google's supply-chain team scores suppliers by how fast they deliver and ranks them fastest-first. The query computes a per-supplier average and a RANK() in one pass. The interviewer is testing whether you can mix an aggregate and a window function correctly — and whether you know what RANK does when two suppliers tie on average speed.

1. The source data — shipments

Suppliers 1, 2 and 4 all average exactly 3 days — a three-way tie at the top. That tie is the whole interview: it decides what rank the slow supplier (3) gets, and whether the tied trio share a number or get 1, 2, 3.

2. The SQL — RANK over AVG

SELECT supplier_id, AVG(days_to_deliver) AS avg_days,
  RANK() OVER (ORDER BY AVG(days_to_deliver)) AS rk
FROM shipments GROUP BY supplier_id;

The query groups by supplier, computes AVG(days_to_deliver), and ranks ascending — lower average means faster, so rank 1 is the fastest. The window function runs after the GROUP BY, so it ranks one row per supplier, not raw shipments.

All three fast suppliers get rank 1 — RANK gives tied rows the same rank. Then supplier 3 jumps to rank 4, not 2: RANK leaves a gap equal to the size of the tie group (1, 1, 1, 4). The numbers 2 and 3 simply do not appear.

4. Strategic explanations

You can put an aggregate inside a window's ORDER BY

RANK() OVER (ORDER BY AVG(days_to_deliver)) looks like a layering violation, but it is legal and standard. Logically the GROUP BY runs first, producing one row per supplier with a scalar AVG; the window function then operates on those grouped rows. Window functions are evaluated after grouping, so the aggregate is just an ordinary value to the OVER clause.

RANK vs DENSE_RANK vs ROW_NUMBER — the gap question

With the three-way tie, the choice of function changes the output:

43111

13113
364 (skips!)24

RANK skips ranks after a tie (1,1,1,4). DENSE_RANK does not (1,1,1,2). ROW_NUMBER ignores the tie and assigns 1,2,3,4 arbitrarily — which fast supplier gets "1" is non-deterministic. For a supplier scorecard, RANK or DENSE_RANK is right; ROW_NUMBER would falsely declare one of three equally-fast suppliers the unique winner.

The result is unordered — add an explicit ORDER BY

The window ORDER BY drives the ranking; it does not sort the final output. The demo rows happen to come back fastest-first, but that is not guaranteed. For a presentable scorecard, end with an explicit ORDER BY rk (or ORDER BY avg_days).

NULL delivery times — ascending sort puts them last

If a shipment has a NULL days_to_deliver, AVG ignores it — the supplier is still ranked on their recorded shipments. But a supplier whose deliveries are all NULL gets avg_days = NULL, and under an ascending ORDER BY Postgres sorts NULLs last — so they would rank as the "slowest", which may be misleading. Make the intent explicit with ORDER BY AVG(days_to_deliver) NULLS LAST, or filter unmeasured suppliers out.

Tie-awareness vs sample size

A supplier with one fast shipment and a supplier with fifty fast shipments can tie on average, but the one-shipment average is far less trustworthy. A senior answer mentions weighting by volume or adding COUNT(*) AS shipment_count to the output so a reader can judge confidence — ranking on a mean alone hides sample size.

Senior signal: know that an aggregate is legal inside a window ORDER BY because windows run after grouping, choose RANK (gaps after ties) vs DENSE_RANK (no gaps) vs ROW_NUMBER (arbitrary tiebreak) on purpose, add a final ORDER BY for deterministic output, and surface shipment count so a tiny-sample average is not mistaken for a reliable one.
