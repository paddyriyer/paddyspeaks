A support-ops team logs every status touch on a ticket, but for an SLA report they need the stints: contiguous runs where a ticket sat in one status. The status_log table has one noisy row per touch — the same status may appear back-to-back — so the task is to collapse consecutive identical statuses into single intervals. This is the classic gaps-and-islands problem, and the interviewer wants to see the "difference of running counts" trick that identifies each island.

The trap is entity 1 returning to open on Jan 3 after a spell in in_progress. That is a separate island from the Jan 1 open run — grouping by status alone would wrongly merge them. The Jan 4 closed row also tests the single-row stint.

2. The SQL — gaps-and-islands via LAG and a running change-count

WITH flagged AS (
  SELECT entity_id, ts, status,
         LAG(status) OVER (PARTITION BY entity_id ORDER BY ts) AS prev_status
  FROM status_log
),
changes AS (
  SELECT entity_id, ts, status,
         SUM(CASE WHEN prev_status IS DISTINCT FROM status THEN 1 ELSE 0 END)
           OVER (PARTITION BY entity_id ORDER BY ts
                 ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS grp
  FROM flagged
)
SELECT entity_id,
       MIN(ts) AS stint_start,
       MAX(ts) AS stint_end,
       MIN(status) AS status,
       COUNT(*) AS row_count
FROM changes
GROUP BY entity_id, grp
ORDER BY entity_id, stint_start;

LAG exposes the previous row's status. The SUM(CASE ...) running total ticks up by 1 every time the status differs from the prior row — so every row in one contiguous run shares the same grp value. Grouping by (entity_id, grp) then collapses each island into one stint.

Entity 1's open appears as two separate rows — the Jan 1 run and the Jan 3 run — exactly as required: a status that recurs after an interruption is a new stint. The single-row closed stint shows stint_start = stint_end, which is correct, not a bug.

4. Strategic explanations

Why the running-sum-of-changes assigns one id per island

The core idea: increment a counter only at a boundary (where status differs from the prior row). Every row inside a contiguous run sees zero increments and inherits the same cumulative total; the first row of the next run sees a +1 and jumps to a new value. That cumulative total is the island id. It is monotonic and gap-tolerant — it does not care whether open reappears later, because each fresh run gets its own number.

IS DISTINCT FROM — the NULL-safe comparison that matters here

The very first row of each entity has prev_status = NULL (no prior row). A plain prev_status <> status would evaluate to NULL — not TRUE — so the CASE would fall to ELSE 0 and the first row would not start a new island. IS DISTINCT FROM treats NULL as just another value: NULL IS DISTINCT FROM 'open' is TRUE, so the first row correctly increments to 1. This single operator is what makes the first stint of every entity register.

The classic mistake — grouping by status

The instinctive (wrong) solution is GROUP BY entity_id, status. That merges entity 1's two open runs into one row spanning Jan 1 to Jan 3 — erasing the in_progress spell in between. Gaps-and-islands exists precisely because the same value can recur: you must group by a computed island id, never by the value itself.

MIN(status) is safe only because the group is homogeneous

MIN(status) looks odd — why aggregate a column that is constant within the group? It is a workaround for status not being in the GROUP BY: every row in one island has the same status, so MIN, MAX, or any_value all return that one value. Adding status to the GROUP BY alongside grp would be equally correct and arguably clearer. State that the aggregate is a no-op picking a constant.

Dialect notes — portability of the building blocks

LAG, framed running SUM, and GROUP BY are standard across Postgres, SQL Server, BigQuery, Snowflake, and MySQL 8+. The one portability snag is IS DISTINCT FROM: SQL Server lacks it — use prev_status <> status OR prev_status IS NULL OR status IS NULL, or in newer versions the standard IS DISTINCT FROM predicate. BigQuery and Snowflake support it directly.

Senior signal: solve gaps-and-islands with a running SUM of boundary flags to assign one id per contiguous run, group by that computed id — never by the recurring value — and use IS DISTINCT FROM so the leading NULL from LAG still registers the first stint.
