NPM's platform team needs an org chart flattened from a self-referencing employees table: every person, how deep they sit, and the full chain of command from the CEO down to them. There is no fixed depth — the tree can be three levels or ten — so a plain join cannot do it. This is the canonical recursive-CTE question; the grand title is dressing on a well-known pattern.

1. The source data — employees

emp_idnamemanager_id
1AdaNULL
2Ben1
3Cleo1
4Drew2
5Eve2
6Finn3

Ada has a NULL manager_id — she is the root, the anchor of the recursion. Ben and Cleo report to her; Drew and Eve report to Ben; Finn reports to Cleo. The tree is three levels deep, and nothing in the schema tells you that in advance.

2. The SQL — WITH RECURSIVE

WITH RECURSIVE org AS (
  SELECT emp_id, name, manager_id, 1 AS lvl, name AS path
  FROM employees WHERE manager_id IS NULL
  UNION ALL
  SELECT e.emp_id, e.name, e.manager_id, o.lvl+1, o.path || ' > ' || e.name
  FROM employees e JOIN org o ON e.manager_id = o.emp_id)
SELECT emp_id, name, lvl, path FROM org ORDER BY lvl, name;

The anchor selects the root (level 1). The recursive arm joins each employee to a row already in org via their manager, incrementing lvl and extending path — it keeps firing until no new rows are produced.

3. The resulting analysis

emp_idnamelvlpath
1Ada1Ada
2Ben2Ada > Ben
3Cleo2Ada > Cleo
4Drew3Ada > Ben > Drew
5Eve3Ada > Ben > Eve
6Finn3Ada > Cleo > Finn

Each person carries their depth and an unambiguous chain of command. The recursion ran three times: anchor (Ada), then Ben/Cleo, then Drew/Eve/Finn — and stopped because no employee reports to a level-3 person.

4. Strategic explanations

Anchor + recursive term — the two-part contract

A recursive CTE is always two queries glued by UNION ALL. The anchor is the non-recursive seed — here, the root with manager_id IS NULL. The recursive term references the CTE name (org) and produces the next layer from the previous one. Postgres iterates: run the anchor, then repeatedly run the recursive term against only the rows added last round, until a round adds nothing. Get the anchor predicate wrong — e.g. omit IS NULL — and you either seed nothing or seed everything.

The infinite-loop trap — cycles

UNION ALL never dedupes, so if the data has a cycle (A manages B, B manages A — a bad edit), the recursion never terminates and the query hangs. Real hierarchical code defends against it. Postgres 14+ has native cycle detection:

WITH RECURSIVE org AS ( … )
CYCLE emp_id SET is_cycle USING cycle_path

Pre-14, you carry an array path and stop manually: WHERE NOT e.emp_id = ANY(o.id_path). Mentioning the cycle risk — not just writing the happy path — is the senior tell.

UNION ALL vs UNION — and why it must be ALL

The recursive term must use UNION ALL, not UNION. UNION forces a distinct check every iteration, which is slower and, more subtly, can mask a genuine cycle by silently swallowing the repeated row instead of letting cycle detection catch it. UNION ALL plus an explicit cycle guard is the correct combination.

Building the path string — and its sort gotcha

o.path || ' > ' || e.name accumulates the breadcrumb top-down. It is great for display, but sorting by path gives lexicographic order, not tree order — fine here because the query sorts by lvl, name. For a true depth-first tree listing, sort by an array path instead: carry array_append(o.id_path, e.emp_id) and ORDER BY id_path, which orders children directly under their parent.

Subtree and depth-limit follow-ups

"Everyone under Ben" just changes the anchor to WHERE emp_id = 2. "Only three levels deep" adds WHERE o.lvl < 3 to the recursive term — also a pragmatic safety cap against runaway recursion even when you believe the data is clean.

Senior signal: structure the recursion as anchor (root via manager_id IS NULL) plus recursive term joined to the CTE, use UNION ALL deliberately, and proactively guard against cycles — native CYCLE on PG14+ or an array-path check — rather than assuming the org chart is acyclic.
