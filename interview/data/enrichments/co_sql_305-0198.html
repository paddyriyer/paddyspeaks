The CRO at a marketplace asks: "Which customers have purchased from every product category we sell? They're our highest-value advocates — Amazon Prime members, Costco loyalists. Send them a milestone thank-you and a free upgrade." The query is the canonical relational division pattern — find customers whose purchase set covers the universal set of categories.

2. The SQL — count distinct categories vs the universal count

SELECT customer_id
FROM orders
GROUP BY customer_id
HAVING COUNT(DISTINCT category) = (
  SELECT COUNT(DISTINCT category) FROM orders
);

4. Strategic explanations

The trick — a scalar subquery in HAVING

The right-hand side of HAVING COUNT(DISTINCT category) = (SELECT COUNT(DISTINCT category) FROM orders) is a scalar subquery — it returns exactly one number (the universal category count). The query computes it once and compares each customer's distinct-category count against it. Engines typically run the inner query once and cache the result, so it's not an O(n²) per-customer rescan.

Why COUNT(DISTINCT category) not COUNT(*)

C_004 has 5 orders but only 4 distinct categories (Books appears twice). A naïve COUNT(*) would say C_004 has 5 categories — wrong. DISTINCT deduplicates within the group; this is the bug-or-feature distinction that catches a lot of candidates.

What if a category has zero customers — the "missing dimension" trap

The query computes the universal category count from orders itself. If a category exists in products but has zero orders, the query won't see it — and a customer who hasn't bought from it would still be flagged as "complete". To handle that, derive the universal count from the dimension table:

HAVING COUNT(DISTINCT category) = (SELECT COUNT(DISTINCT category) FROM products)

This is a real production bug; senior candidates ask "where does the universal set come from?" unprompted.

Relational division — the deeper pattern

This question is the textbook relational division: "find R-rows that cover all S-rows under a given relationship." Other instances:

  • Skills coverage: "find candidates who have every required skill"
  • Region presence: "find sellers active in every region we operate"
  • Permission completeness: "find users with every permission needed for an admin role"

The SQL shape — HAVING COUNT(DISTINCT x) = (SELECT COUNT(DISTINCT x) FROM ...) — is universal. Naming the pattern in the interview is a senior signal.

The double-NOT-EXISTS variant — when DISTINCT is too expensive

At billions-of-rows scale, COUNT(DISTINCT) is expensive (no shortcut without a hash). The set-theory equivalent is "find customers for whom no required category is missing":

SELECT c.customer_id
FROM customers c
WHERE NOT EXISTS (
  SELECT 1 FROM products p
  WHERE NOT EXISTS (
    SELECT 1 FROM orders o
    WHERE o.customer_id = c.customer_id AND o.category = p.category
  )
);
-- "Customers for whom there is NO category they did NOT buy"

The double-negation is brain-bending but lets the optimiser use indexes. Production reporting tables sometimes prefer it.

The "X-of-Y" relaxation — most-but-not-all coverage

CRO follow-up: "What about customers who covered 3 of 4 categories? They're almost-complete; one nudge might convert them."

HAVING COUNT(DISTINCT category) >= (SELECT COUNT(DISTINCT category) FROM products) - 1
Senior signal: name the relational division pattern, articulate the DISTINCT-vs-COUNT(*) distinction, identify the "missing dimension" trap (universal count from products vs orders), and pre-empt the X-of-Y relaxation. The double-NOT-EXISTS variant is the over-the-bar performance move.
