A growth team at a social platform wants a "distribution of distributions" view: not how many tweets were posted in 2025, but how many users posted exactly one tweet, exactly two, exactly three, and so on. That is a histogram of a per-user count — a two-stage aggregation. The interviewer is checking whether you can aggregate an aggregate: count per user, then count the users at each count.

1. The source data — tweet

Two edge cases are deliberate. User 5's only tweet falls on 2024-12-30 — outside the year, so user 5 has zero 2025 tweets and must not appear at all. And users 1 and 3 both post exactly 3 times, so the tweet_count = 3 bucket must show 2 users, not 1.

2. The SQL — per-user count, then histogram

WITH per_user AS (
  SELECT user_id, COUNT(*) AS tweet_count
  FROM tweet
  WHERE tweet_date >= DATE '2025-01-01' AND tweet_date < DATE '2026-01-01'
  GROUP BY user_id
)
SELECT tweet_count, COUNT(*) AS users
FROM per_user
GROUP BY tweet_count
ORDER BY tweet_count;

The CTE produces one row per user with their 2025 tweet total. The outer query then groups those rows by tweet_count and counts how many users sit in each bucket — a GROUP BY over the output of a GROUP BY.

The tweet_count = 3 row reads 2: both heavy posters land in the same bucket and COUNT(*) adds them. User 5 is absent everywhere — their single tweet was in 2024, so they never enter per_user, and there is correctly no tweet_count = 0 row.

4. Strategic explanations

Why two stages, not one

The question asks about a derived quantity — "number of tweets a user posted" — that does not exist as a column. You must compute it first (stage one: COUNT(*) ... GROUP BY user_id), then treat that result as a new fact table and aggregate again (stage two: GROUP BY tweet_count). Trying to do it in a single GROUP BY is impossible because the two groupings are at different grains — per-user, then per-count.

The half-open date range avoids a year-boundary bug

>= '2025-01-01' AND < '2026-01-01' is a half-open interval: it captures every instant of 2025 and nothing of 2026. Writing BETWEEN '2025-01-01' AND '2025-12-31' works for a DATE column but silently drops rows if the column is ever widened to a TIMESTAMP — 2025-12-31 14:00 is greater than the date 2025-12-31. The half-open form is correct for both types, which is why seniors default to it.

Zero-tweet users are correctly invisible

The spec explicitly says users with no 2025 tweets should not appear, and this query honours that for free: a user with no qualifying rows never survives into per_user, so there is no tweet_count = 0 bucket. If the requirement flipped — "include a 0 bucket counting dormant users" — you would need a separate users table and a LEFT JOIN, because the tweet table alone cannot know about users who never tweeted.

Common mistake — COUNT(DISTINCT ...) in the wrong place

A frequent error is COUNT(DISTINCT user_id) in the outer query. Inside per_user each user already appears once, so a plain COUNT(*) is exactly the user count — DISTINCT is redundant and only adds a sort. Conversely, if you skipped the CTE and tried to count distinct users grouped by a raw per-row count, you would be grouping by something that does not exist. Knowing where DISTINCT is needed (nowhere here) signals you understand the grain.

Senior signal: identify a "histogram of a per-user metric" as a two-stage aggregation — count per user in a CTE, then count users per count — use a half-open date range so the filter survives a DATE-to-TIMESTAMP change, and explain that zero-activity users are absent by construction unless you LEFT JOIN a user dimension.
