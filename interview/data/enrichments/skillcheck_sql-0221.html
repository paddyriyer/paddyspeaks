A comment replies to a comment, which replies to another comment. Measuring a discussion — how many replies a thread drew, how deep the back-and-forth went — means walking the reply chain down from each root. A WITH RECURSIVE CTE does it, carrying both the thread’s identity and a depth counter.

A parent_comment_id of NULL marks a top-level comment — a thread root, replying to the post itself. Ana’s comment roots one thread; Fay’s roots another.

2. The recursive solution

WITH RECURSIVE thread AS (
  SELECT comment_id, comment_id AS root_id, 1 AS depth
  FROM comments
  WHERE parent_comment_id IS NULL
  UNION ALL
  SELECT c.comment_id, t.root_id, t.depth + 1
  FROM thread t
  JOIN comments c ON c.parent_comment_id = t.comment_id
)
SELECT r.author AS root_author,
       COUNT(*) - 1 AS reply_count,
       MAX(t.depth) AS max_depth
FROM thread t
JOIN comments r ON r.comment_id = t.root_id
GROUP BY t.root_id, r.author
ORDER BY reply_count DESC, root_author ASC;

The anchor selects the root comments and stamps each with its own id as root_id and a depth of 1. The recursive term follows replies, propagating the same root_id and adding 1 to depth.

4. Strategic explanations

Carry the root identity all the way down

The anchor sets root_id = comment_id for each root. The recursive term then selects t.root_id — the parent row’s root, unchanged — not c’s own id. So every comment, however deep, is stamped with the thread it ultimately belongs to, and a single GROUP BY root_id collapses each thread.

Depth is an incrementing accumulator

depth starts at 1 on the root, and the recursive term emits t.depth + 1 for each reply. MAX(depth) per thread is how far its longest reply chain runs. The value lives nowhere in the table — recursion computes it from the shape of the tree.

reply_count is COUNT(*) minus the root

The recursive set includes the root comment itself, so a thread’s row count is replies plus one. Subtracting 1 gives the reply count. The alternative — seeding the anchor with the roots’ direct children to exclude the root — works too, but then a thread with zero replies would produce no rows and disappear from the report entirely.

Roots are defined by a NULL parent

A top-level comment replies to the post, not to another comment, so its parent_comment_id is NULL — and that predicate is exactly what the anchor selects. Recursion terminates because replies form a tree: each comment has one parent, leaves have no children, and the recursive term eventually returns nothing. A reply that somehow pointed back up its own chain would loop forever.

Senior signal: to analyze threaded replies, recurse from the roots (parent_comment_id IS NULL), carrying the root’s id unchanged so every descendant is tagged with its thread, plus a depth accumulator for nesting. Remember the recursive set contains the root itself, so reply counts are COUNT(*) - 1.
