Barclays' merchandising team wants the three best-selling products in every region for a category review. "Top 3 per group" is the canonical window-function interview question — it tests whether you can aggregate and rank in the same query, and whether you pick a ranking function whose tie behaviour matches what "top 3" is supposed to mean.

1. The source data — sales

Each product can have several sale rows — product 1 in West appears twice, so totals must be aggregated first. East has a tie: products 5 and 6 both total 200. With only three East products that tie is harmless here, but it is the seed of the edge case discussed below.

2. The SQL — ROW_NUMBER within region

SELECT region, product_id, total FROM (
  SELECT region, product_id, SUM(amount) AS total,
    ROW_NUMBER() OVER (PARTITION BY region ORDER BY SUM(amount) DESC) rn
  FROM sales GROUP BY region, product_id) t WHERE rn <= 3;

The inner query groups by (region, product_id) to get per-product totals, then ROW_NUMBER() numbers the products 1, 2, 3, … within each region by descending total. The outer query keeps numbers ≤ 3. You cannot filter on a window function in the same SELECT that defines it — hence the subquery.

West's product 1 totals 150 (100 + 50) and tops its region; products 4 (total 30) is correctly dropped as the 4th. In East, products 5 and 6 are tied at 200 — ROW_NUMBER still assigned them distinct numbers (1 and 2), and which one got "1" was decided arbitrarily by the planner. That is fine when both are inside the top 3, but it is a landmine at the boundary.

4. Strategic explanations

The tie-at-the-boundary trap — ROW_NUMBER vs RANK

Imagine three products tied for 3rd place. ROW_NUMBER assigns them rn 3, 4, 5 and keeps exactly one — chosen non-deterministically. Is that what "top 3" means, or should all three co-3rd products survive? If ties must all be kept, use RANK():

SELECT region, product_id, total FROM (
  SELECT region, product_id, SUM(amount) AS total,
    RANK() OVER (PARTITION BY region ORDER BY SUM(amount) DESC) rk
  FROM sales GROUP BY region, product_id) t WHERE rk <= 3;

This can return four or more rows for a region. ROW_NUMBER guarantees exactly three but picks arbitrarily among ties; RANK keeps every tied product but can exceed three. Naming that tradeoff out loud is the senior move — and adding a deterministic tiebreaker (ORDER BY SUM(amount) DESC, product_id) at least makes ROW_NUMBER reproducible.

Why ORDER BY SUM(amount) works inside OVER

It can look odd to call SUM(amount) in a window ORDER BY, but it is legal: the query is already grouped by (region, product_id), so SUM(amount) is a per-group scalar the window can sort on. You could alias it — but the alias total is not visible to the window clause in the same SELECT, so repeating SUM(amount) is the standard workaround.

Top-N with a CTE — the same query, more readable

For review-friendly SQL, lift the inner query into a CTE:

WITH product_totals AS (
  SELECT region, product_id, SUM(amount) AS total
  FROM sales GROUP BY region, product_id),
ranked AS (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY region ORDER BY total DESC) rn
  FROM product_totals)
SELECT region, product_id, total FROM ranked WHERE rn <= 3
ORDER BY region, rn;

Note the CTE form lets the second stage reference the total alias directly. Functionally identical to the subquery; just easier to read and to extend.

Regions with fewer than three products

If a region has only one or two products, the query simply returns one or two rows for it — no error, no padding. That is almost always the desired behaviour, but if a report needs exactly three slots per region (with blanks), that is a presentation concern, not a SQL one. Worth a sentence to the interviewer so the expectation is explicit.

Add an explicit final ORDER BY

The WHERE rn <= 3 filter does not impose output order — rows can come back in any sequence (the demo happens to show East before West). For a deterministic, presentable result, end with ORDER BY region, total DESC (or , rn). The window ORDER BY controls numbering, not the final row order.

Senior signal: aggregate before ranking, know that you must wrap a window function in a subquery or CTE to filter on it, and consciously choose ROW_NUMBER (exactly three, arbitrary among ties) versus RANK (keeps all ties, may exceed three) based on what "top 3" means for the business — then add a deterministic tiebreaker and a final ORDER BY.
