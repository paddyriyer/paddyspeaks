ML platform teams need to benchmark model performance in production: "For each model type, how many predictions were made, what fraction were correct, and what are the average and 95th-percentile latencies?" This question probes the AVG(CASE WHEN ... THEN 1.0 ELSE 0.0 END) accuracy pattern, the PERCENTILE_CONT ordered-set aggregate, and defensive NULLIF(regexp_replace(...)) casting for dirty numeric columns.

2. The SQL — accuracy via conditional average, P95 latency via PERCENTILE_CONT

SELECT model_type,
       COUNT(*) AS calls,
       AVG(CASE WHEN predicted = actual THEN 1.0 ELSE 0.0 END) AS accuracy,
       AVG(NULLIF(regexp_replace(latency_ms::text, '[^-.0-9]', '', 'g'), '')::numeric) AS avg_latency_ms,
       PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY NULLIF(regexp_replace(latency_ms::text, '[^-.0-9]', '', 'g'), '')::numeric) AS p95_latency_ms
FROM model_predictions
GROUP BY model_type
ORDER BY calls DESC;

4. Strategic explanations

The AVG(CASE WHEN ... THEN 1.0 ELSE 0.0 END) accuracy pattern

Dividing "correct count" by "total count" is the definition of accuracy. The CASE WHEN predicted = actual THEN 1.0 ELSE 0.0 END converts each row into a binary indicator (1=correct, 0=wrong). AVG of those indicators equals the fraction of 1s — which is the accuracy rate. Using 1.0 (float) rather than 1 (integer) avoids integer-division truncation in engines that return integers from integer division. This is the canonical SQL accuracy pattern; an alternative is SUM(CASE ...) * 1.0 / COUNT(*), but AVG is more concise.

The NULLIF(regexp_replace(...), '') cast guard

The latency_ms column appears to be stored as text or a variant type with possible non-numeric characters (e.g., "42ms", "120 ms"). The regexp_replace(col::text, '[^-.0-9]', '', 'g') strips all characters that are not digits, decimal points, minus signs, or scientific-notation markers. NULLIF(..., '') then converts any empty result to NULL before the ::numeric cast, preventing a cast error when the column is empty or fully non-numeric after stripping. This defensive pipeline (strip → NULLIF → cast) is a robust pattern for untrustworthy numeric columns.

PERCENTILE_CONT — ordered-set aggregates

PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY value) computes the 95th percentile with linear interpolation between adjacent values. Its sibling PERCENTILE_DISC returns the actual value at or just above the percentile (no interpolation). For latency SLAs, PERCENTILE_CONT is standard; for discrete categories (e.g., rank percentiles), PERCENTILE_DISC is appropriate. Both are ordered-set aggregates — they take an ORDER BY inside the function, which is a different syntax from window functions.

Dialect portability
  • Postgres (solution dialect): PERCENTILE_CONT and PERCENTILE_DISC are built-in ordered-set aggregates. Works as written.
  • Snowflake native: PERCENTILE_CONT is supported. Alternatively: APPROX_PERCENTILE(value, 0.95) for approximate results on large datasets (HyperLogLog-based, much faster).
  • BigQuery: APPROX_QUANTILES(value, 100)[OFFSET(95)] for approximate; PERCENTILE_CONT(value, 0.95) OVER () as a window function (not an aggregate).
  • MySQL: No PERCENTILE_CONT before 8.0. Use ROW_NUMBER + math to approximate.
Snowflake’s IFF alternative for accuracy

Snowflake has the IFF(condition, true_val, false_val) shorthand: AVG(IFF(predicted = actual, 1.0, 0.0)). This is purely stylistic — same semantics as CASE WHEN.

Senior signal: name the AVG(CASE WHEN ...) trick as the canonical accuracy pattern (equivalent to proportion of 1s in a 0/1 indicator), explain the NULLIF(regexp_replace(...)) cast guard for dirty columns, distinguish PERCENTILE_CONT (interpolated) from PERCENTILE_DISC (actual value), and mention Snowflake’s APPROX_PERCENTILE for large-scale latency monitoring where approximate results suffice.
