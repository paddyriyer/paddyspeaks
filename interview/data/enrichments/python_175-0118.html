Merge k Sorted Lists is the algorithm at the heart of every external sort and every LSM-tree compaction. Cassandra, RocksDB, LevelDB, ScyllaDB, BigTable, ClickHouse — every storage engine that maintains sorted-runs-on-disk runs this primitive every few minutes during compaction. Kafka's consumer-group rebalance state-merge uses it. Snowflake's micro-partition pruning uses it. The reason the question survives in interview loops despite being "an old chestnut" is the constraint: k in the thousands but total nodes in the millions, where the difference between O(Nk) and O(N log k) is the difference between a compaction that takes 30 seconds and one that takes 8 hours.

1. The problem in a sentence

Given k sorted singly-linked lists, merge them into one sorted linked list and return the head. The two naive approaches are both wrong at scale: (1) flatten everything into a list and sort — O(N log N) and ignores the input's existing sortedness; (2) pairwise-merge (merge list 0 with list 1, then with list 2, ...) — O(Nk) because the running merged list grows linearly and each element gets compared k times on average. The constraint that rules them out is a 1000-way merge of a billion total rows (external-sort scenario). The trick is a size-k min-heap of (head_value, tiebreak, head_node): each of the N nodes is pushed once and popped once, giving O(N log k).

The heap is always size ≤ k (three lists, so size ≤ 3). Total heap operations: 2N (one push and one pop per node), each O(log k). Total work: O(N log k) where N is the sum of all list lengths.

3. The algorithm — naming the data structure

import heapq

class ListNode:
    def __init__(self, val=0, nxt=None):
        self.val = val
        self.next = nxt

def mergeKLists(lists):
    """Size-k min-heap of (value, tiebreak, node). The tiebreak counter
       is required because two ListNode objects aren't comparable; without
       it, when two values tie, heapq.heappush would try ListNode < ListNode
       and TypeError."""
    h = []
    for i, node in enumerate(lists):
        if node:
            heapq.heappush(h, (node.val, i, node))
    dummy = tail = ListNode()                          # sentinel head
    counter = len(lists)                                # monotonically increasing tiebreaker
    while h:
        val, i, node = heapq.heappop(h)
        tail.next = node                                # splice node in -- preserves identity
        tail = tail.next
        if node.next:
            counter += 1
            heapq.heappush(h, (node.next.val, counter, node.next))
    return dummy.next

Three production-critical details. (1) The tiebreak counter exists because heapq falls back to comparing the third element when the first two tie; raw ListNode objects don't implement __lt__ and Python raises TypeError. The fix is a unique-per-push integer. (2) We splice nodes in (re-use the input nodes) rather than allocate new ones — important for memory-constrained environments. (3) The dummy head simplifies the empty-input case: dummy.next is None by default, which is the correct return for empty input without any branching.

4. Complexity

Time: O(N log k) where N is total nodes across all lists and k is the number of lists. Each node is pushed exactly once and popped exactly once; each heap op is O(log k) because the heap is size ≤ k. Space: O(k) for the heap (plus O(1) auxiliary for the dummy and counter). The trick that gets you there: instead of scanning all k heads for the current minimum (which is the O(Nk) pairwise-merge cost), we maintain the k heads in a heap so the min is the root.

The other valid solution is divide-and-conquer pairwise merge: pair up the k lists, merge each pair, recurse on k/2 merged lists. This is also O(N log k) — same complexity — but with a heavier constant due to recursion and intermediate list allocation. The heap version is what production code ships because it streams output (you can emit the next sorted node as soon as it's popped, without waiting for the full merge), which matters for external sorts where the output is written to disk as it's produced.

5. The interviewer's edge-case probes

  • Empty input list (lists = []) — return None. The initial loop produces an empty heap; the while loop is skipped; dummy.next is None.
  • All lists are None (lists = [None, None]) — return None. The initial loop's if node guard skips empty lists.
  • One list is much longer than the others — the heap drains the short lists first, then the long list is essentially copied through with a size-1 heap. Still O(N log k) but the constant is dominated by the linear pass.
  • Many ties (e.g., k lists of all-zeros) — without the tiebreaker, Python raises TypeError: '<' not supported between instances of 'ListNode'. This is the most common interview bug; mention it before you write the code.
  • k = 1 — return lists[0]. The algorithm works but is wasteful; some implementations short-circuit.
  • Streaming variant — the lists arrive as iterators (k Kafka partitions). The algorithm is unchanged; substitute node.next with next(iter_i, None). This is the form that runs in production Kafka/Pulsar consumers doing time-ordered merging across partitions.
  • External-sort scale (k = 1000 sorted runs on disk, each is GBs) — same algorithm; the "node" is a disk-backed buffered iterator. The heap holds one buffered chunk per run. This is exactly Unix sort -m and Hadoop's reducer merge.

6. Real-world appearance — what runs this in production

LSM-tree compaction — RocksDB, LevelDB, Cassandra, ScyllaDB

Every LSM-tree storage engine periodically compacts overlapping sorted SSTables (sorted-string-tables) by merging them into one. RocksDB's MergingIterator is literally this algorithm with file-backed iterators; Cassandra's compaction strategy (STCS, LCS, TWCS) all sit on top of a k-way merge. ScyllaDB's seastar-based compaction runs the merge with k in the hundreds during major compactions. The choice between this algorithm and tournament-tree-based "loser-tree" variants is benchmarked in the RocksDB literature and the heap wins for moderate k.

Kafka / Pulsar consumer-group time-ordered merge

When a consumer group reads from k partitions and needs global time order (for change-data-capture into a downstream system that requires monotonic timestamps), the consumer applies a k-way merge across the per-partition iterators. LinkedIn's Brooklin uses this for CDC fan-in. Confluent's ksqlDB uses it for stream-stream joins where timestamps must be aligned.

External sort — Unix sort -m, Hadoop reducer, Spark ExternalSorter

Sorting more data than fits in RAM: chunk it into k sorted runs on disk, then merge. Unix sort -m takes pre-sorted files; Hadoop's reduce phase merges k sorted spill files from the map outputs; Spark's ExternalSorter spills sorted runs and merges them with a min-heap. The classic reference is Knuth Vol. 3 §5.4, but every production sort still implements the same algorithm.

Snowflake / BigQuery / ClickHouse — micro-partition merge

Snowflake's micro-partitions are stored sorted by clustering key; queries that scan a clustered table run a k-way merge across the relevant micro-partitions. ClickHouse's MergeTree engine name is literally a reference to this algorithm — the engine continuously merges sorted parts in the background. BigQuery does the same on its capacitor file format.

Median-of-medians / weighted merge — order matching engines

Stock exchanges with multiple order books (e.g., NYSE's order-of-protection rule) merge price-time-priority quotes across venues. The merge is a k-way heap-based merge keyed on (price, timestamp). Refinitiv's TRTH (Tick History) data uses the same primitive offline for cross-venue reconciliation.

LLM agent state — multi-source memory merge

RAG agents that pull from multiple vector stores and rerank by relevance score are doing a k-way merge over (-score, doc_id, snippet) tuples. LangChain's EnsembleRetriever implements this directly.
