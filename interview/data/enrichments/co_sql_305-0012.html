Broadcom's HR-analytics team wants the full org chart flattened with a depth label — the CEO at level 1, their directs at level 2, and so on down. A plain self-join can reach one level; an org of unknown depth needs a recursive CTE that walks the tree until it runs out of employees. The interview question is whether you understand the anchor / recursive-member structure — and what stops it from looping forever.

1. The source data — employees

A four-deep tree. Ada (CEO) has a NULL manager_id — she is the root. Ben and Cara report to her; Dan reports to Ben and Finn to Cara; Eve sits deepest, under Dan. The NULL at the root is what the recursion anchors on.

2. The SQL — WITH RECURSIVE

WITH RECURSIVE hierarchy AS (
  SELECT emp_id, name, manager_id, 1 AS level
  FROM employees WHERE manager_id IS NULL
  UNION ALL
  SELECT e.emp_id, e.name, e.manager_id, h.level + 1
  FROM employees e
  JOIN hierarchy h ON e.manager_id = h.emp_id
)
SELECT * FROM hierarchy ORDER BY level, emp_id;

Two halves joined by UNION ALL. The anchor (above the UNION ALL) seeds the result with the root — the employee whose manager_id IS NULL, at level 1. The recursive member (below) joins employees back to rows already in hierarchy, finding everyone whose manager was found in the previous round, and stamps them one level deeper.

4. Strategic explanations

How a recursive CTE actually executes — the termination question

The recursive member runs repeatedly. Each iteration, hierarchy on the right side refers only to the rows produced by the previous iteration. The engine keeps iterating until an iteration adds zero new rows, then stops. That is the natural terminator: once the deepest employees are found, the next round's join finds nobody whose manager is at that depth, the round is empty, and recursion ends. No explicit loop counter required — the tree's finite depth ends it.

UNION ALL vs UNION — not interchangeable here

Recursive CTEs use UNION ALL deliberately. UNION deduplicates on every iteration, which is extra work and — more importantly — can mask a structural problem instead of surfacing it. In a clean tree every employee is reached exactly once, so there is nothing to dedupe; UNION ALL is both correct and faster. If the data were a graph with diamonds (an employee reachable by two paths), UNION ALL would reveal the duplication so you can deal with it consciously.

The infinite-loop trap — cycles in the data

The termination guarantee depends on the data being an actual tree. If a cycle exists — Ada's manager_id set to Eve, say — the recursive join keeps finding "new" rows forever and the query never terminates (or hits a recursion limit). Defensive recursion carries a visited-path and a depth cap:

WITH RECURSIVE hierarchy AS (
  SELECT emp_id, name, manager_id, 1 AS level,
         ARRAY[emp_id] AS path
  FROM employees WHERE manager_id IS NULL
  UNION ALL
  SELECT e.emp_id, e.name, e.manager_id, h.level + 1,
         h.path || e.emp_id
  FROM employees e
  JOIN hierarchy h ON e.manager_id = h.emp_id
  WHERE e.emp_id <> ALL(h.path)   -- stop if we revisit a node
    AND h.level < 100             -- belt-and-braces depth cap
)
SELECT emp_id, name, manager_id, level FROM hierarchy ORDER BY level, emp_id;

Postgres 14+ also offers CYCLE emp_id SET is_cycle USING path as built-in syntax for the same protection.

Edge cases: multiple roots and orphans

The anchor is WHERE manager_id IS NULL, so it seeds every root, not just one — correct for a company with several independent top-level executives, which simply produces a forest. The risk is the opposite: an employee in a cycle has a non-NULL manager_id but no path back to a root, so the anchor never reaches them and they are silently absent from the output. Reconcile the row count — SELECT COUNT(*) FROM hierarchy against SELECT COUNT(*) FROM employees — and any shortfall is an orphan or a cycle to investigate.

Senior signal: explain the anchor / recursive-member split and that recursion stops when an iteration yields zero new rows, justify UNION ALL over UNION, pre-empt the infinite loop a data cycle would cause with a visited-path guard or the CYCLE clause, and reconcile the output count against the table to catch orphans the anchor never reaches.
