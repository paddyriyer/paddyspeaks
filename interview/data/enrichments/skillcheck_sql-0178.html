Tracking when an employee crosses a compensation tier boundary — moving from the bottom third of their pay history to the middle, or from middle to top — is a novel use of two window functions composed in sequence. The interviewer is probing whether you understand that NTILE ranks by value (salary magnitude) while LAG then looks at the chronological sequence of those tier labels, and that these two orderings are different axes that must be applied in a specific order.

2. The SQL — NTILE then LAG on tier label

WITH t AS (
  SELECT emp_id, effective_date, salary,
         NTILE(3) OVER (PARTITION BY emp_id ORDER BY salary) AS tier
  FROM salary_history
),
lag_t AS (
  SELECT emp_id, effective_date, salary, tier,
         LAG(tier) OVER (PARTITION BY emp_id ORDER BY effective_date) AS prev_tier
  FROM t
)
SELECT emp_id, effective_date, salary, prev_tier, tier AS new_tier
FROM lag_t
WHERE prev_tier IS NOT NULL AND tier > prev_tier
ORDER BY emp_id, effective_date;

CTE t assigns a compensation tier (1=lowest third, 3=highest third) to each row using NTILE(3) ordered by salary magnitude. CTE lag_t then looks at those tier labels in chronological order and fetches the previous row's tier with LAG. The final filter tier > prev_tier isolates upward transitions only.

4. Strategic explanations

The critical insight: two different orderings

NTILE ranks by salary value to assign tiers — a value-based ordering. LAG then examines those tier labels in chronological order to detect transitions — a time-based ordering. These two orderings are independent and must not be conflated. In particular, do not use a single window ORDER BY effective_date for the NTILE: that would number rows by arrival time rather than salary magnitude, turning the tier into a "when did this raise happen" rank rather than "how big is this salary" rank. The two-CTE structure enforces the correct separation.

NTILE(3) bucket sizes with remainder rows

With 4 rows and 3 buckets: 4 ÷ 3 = 1 remainder 1. The first remainder buckets (bucket 1) each get one extra row, making them size 2. Buckets 2 and 3 get size 1. So for employee 1, the two lowest salaries (100 and 110) both land in tier 1; the next (180) in tier 2; the highest (200) in tier 3. This is not a mistake — it is the defined behaviour of NTILE — but it means tier 1 can be wider than the other tiers when the row count is not evenly divisible. In practice with many salary history rows per employee this effect averages out.

Downward transitions are correctly excluded

Employee 2's April entry (pay cut from 60 to 55) creates a tier 2 → tier 1 chronological transition. The filter tier > prev_tier strictly excludes this. The question asks only for upward transitions; lateral moves (tier = prev_tier) are also excluded. If the question were "any tier change," the filter would be tier ≠ prev_tier.

Dialect notes

Both NTILE and LAG are standard SQL window functions available in PostgreSQL, MySQL 8+, SQL Server, Snowflake, BigQuery, and Oracle. The FILTER clause is not used here; the two-CTE approach is fully portable. In MySQL 5.x, neither window function is available; the equivalent requires a user-defined variable trick or an application-layer computation.

Senior signal: articulate that NTILE orders by value (salary) while LAG orders by time (effective_date) — two distinct axes requiring two separate window specifications; know NTILE's remainder-row distribution rule; and confirm that a pay cut creating a downward tier transition is correctly excluded by tier > prev_tier.
