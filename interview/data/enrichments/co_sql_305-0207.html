Accenture's contact-centre team runs a data-quality scorecard, and one line on it is the share of calls that were never tagged with a category — a proxy for how disciplined agents are about wrap-up coding. "Uncategorised" sounds binary, but it hides a decision: is a call uncategorised when category is NULL, or also when it is an empty string? The reference counts both. The interviewer is checking whether you saw that distinction at all.

1. The source data — calls

call_idcategory
1sales
2NULL
3support
4(empty string)
5sales
6NULL
7billing
8support

Eight calls. Two have a NULL category (rows 2, 6) and one has an empty string (row 4). NULL and '' are different values in SQL but mean the same thing to the business — "not categorised". Three uncategorised in total.

2. The SQL — conditional count of NULL-or-blank

SELECT 100.0 * SUM(CASE WHEN category IS NULL OR category='' THEN 1 ELSE 0 END)/COUNT(*) AS uncat_pct
FROM calls;

The CASE flags a call as uncategorised when category is NULL or the empty string, mapping it to 1. SUM counts those; COUNT(*) counts every call. The literal 100.0 forces floating-point division.

3. The resulting analysis

uncat_pct
37.500000

Three uncategorised of eight → 37.5%. Both halves of the OR earned their keep: the two NULLs and the one empty string. Drop OR category='' and the result falls to 25% — the empty-string call would be miscounted as properly tagged. That clause is the entire question.

4. Strategic explanations

Why IS NULL alone misses empty strings

NULL means "unknown / absent"; '' is a known value that happens to be a zero-length string. They are distinct, and category IS NULL matches only the first. A blank-string call is just as uncategorised to the business, so the OR category = '' is required. Equally, do not write category = NULL — any comparison to NULL with = yields unknown, never true; NULL tests must use IS NULL. A compact way to cover both: COALESCE(category, '') = '', or NULLIF(category, '') IS NULL — both fold NULL and blank into one test.

Whitespace-only categories — the next layer of dirty

A category of ' ' (spaces) is neither NULL nor '', so the reference counts it as categorised — almost certainly wrong. If agents can submit blank-looking-but-not-empty values, trim first:

SUM(CASE WHEN category IS NULL OR trim(category) = '' THEN 1 ELSE 0 END)

And a literal 'uncategorised' or 'N/A' string is yet another case — data sometimes encodes "no category" as an actual category. Ask what the dirty values really look like before trusting IS NULL OR category = ''.

The integer-division trap

SUM(...) and COUNT(*) are both integers, so SUM/COUNT alone does integer division and truncates — 3/8 would be 0. The leading 100.0 * promotes the expression to numeric so the fraction survives. Placement matters: 100.0 * SUM / COUNT is correct; 100 * (SUM / COUNT) truncates inside the parentheses first. An explicit ::numeric cast on the numerator is the most readable way to make the intent obvious.

The FILTER alternative

Postgres expresses conditional counts more directly with FILTER:

SELECT 100.0 * COUNT(*) FILTER (WHERE category IS NULL OR category = '')
       / COUNT(*) AS uncat_pct
FROM calls;

Functionally identical to the SUM(CASE ...) form, just easier to read. The CASE version is the portable fallback for engines without FILTER.

Edge case: the empty table

If calls has no rows, COUNT(*) is 0 and the division becomes 0/0, which Postgres raises as a division-by-zero error — not NULL, not 0. Guard the denominator with NULLIF(COUNT(*), 0) so the result is NULL instead of an exception, and wrap in COALESCE(..., 0) if the scorecard must always show a number.

Senior signal: treat NULL and the empty string as distinct values and cover both (a COALESCE/NULLIF fold, or trim to also catch whitespace), never compare to NULL with =, multiply by 100.0 so integer division does not zero out the ratio, and guard the empty-table 0/0 with NULLIF.
