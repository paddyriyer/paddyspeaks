Oracle's monitoring team tracks a daily metric and wants each day shown alongside the day before and the day after — the raw material for deltas, trend arrows, and anomaly flags. LAG and LEAD are the window functions built exactly for this. The query is short; the interview value is in the frame clause you did not write, the ordering you must get exactly right, and what happens at the two endpoints.

1. The source data — metrics

metric_datevalue
2026-01-01100
2026-01-02120
2026-01-0390
2026-01-04150
2026-01-05150

Five consecutive days. Days 4 and 5 hold the same value (150) — useful for showing that LAG/LEAD report the neighbouring value faithfully even when it is a repeat. The first and last rows are the edge cases: one has no predecessor, the other no successor.

2. The SQL — LAG / LEAD over date order

SELECT metric_date, value,
  LAG(value)  OVER (ORDER BY metric_date) AS prev,
  LEAD(value) OVER (ORDER BY metric_date) AS next
FROM metrics;

LAG(value) pulls value from the row one position earlier in the ordering; LEAD(value) pulls it from one position later. The ORDER BY metric_date inside OVER defines what "earlier" and "later" mean.

The first row's prev is NULL — nothing precedes Jan 1. The last row's next is NULL — nothing follows Jan 5. Every interior row sees both neighbours. Day 4 and Day 5 both show 150 in adjacent cells; the function does not collapse repeats.

4. Strategic explanations

The endpoint NULLs and the third argument

LAG at the first row and LEAD at the last row have nothing to point at, so they return NULL. If a downstream day-over-day delta will compute value - prev, that NULL propagates — the first row's delta becomes NULL. Supply a default with the third argument:

LAG(value, 1, 0)  OVER (ORDER BY metric_date) AS prev
-- offset 1, default 0 when there is no prior row

Whether 0 is the right default is a business call (for a delta you might prefer the row's own value, so the first delta is 0). The point is to choose, not to let NULL leak.

The missing-day trap — positional, not calendar

LAG steps one row back, not one day back. If Jan 3 is absent from the table, Jan 4's prev becomes Jan 2's value — a two-day gap silently treated as one step, and any "daily change" computed from it is wrong. LAG cannot know the calendar. For true calendar-adjacency, either generate a complete date spine and left-join the metrics onto it, or use a range-based self-join on metric_date = d - INTERVAL '1 day'. Always ask whether the series can have gaps.

Computing the delta — the actual goal

Nobody wants prev for its own sake; they want the change. Day-over-day movement and percentage:

SELECT metric_date, value,
       value - LAG(value) OVER (ORDER BY metric_date) AS dod_change,
       ROUND( (value - LAG(value) OVER w) * 100.0
              / NULLIF(LAG(value) OVER w, 0), 1) AS dod_pct
FROM metrics
WINDOW w AS (ORDER BY metric_date);

The named WINDOW clause defines the frame once and reuses it — cleaner than repeating OVER (ORDER BY metric_date) three times. NULLIF(..., 0) guards against a divide-by-zero when yesterday's value was 0.

PARTITION BY — the multi-series follow-up

The moment the table has more than one metric (a metric_name column), an un-partitioned LAG bleeds across series — the first row of "latency" would pull its prev from the last row of "throughput". Partition to isolate each series:

LAG(value) OVER (PARTITION BY metric_name ORDER BY metric_date)

LAG resets at every partition boundary, so each series independently gets a NULL on its own first row.

The ORDER BY inside OVER is mandatory and load-bearing

LAG/LEAD are meaningless without an explicit ORDER BY in the window — "previous" needs a defined sequence. The ORDER BY at the end of the SELECT sorts the output; it does not feed the window. They are two separate orderings and confusing them is a classic mistake. Ties in the window's ordering make LAG non-deterministic about which peer counts as "previous" — add a tiebreaker column if dates can repeat.

Senior signal: explain the endpoint NULLs and use LAG's third (default) argument deliberately, warn that LAG is positional so missing days corrupt deltas, add PARTITION BY the instant multiple series exist, guard percentage deltas with NULLIF, and keep the window ORDER BY distinct from the output ORDER BY.
