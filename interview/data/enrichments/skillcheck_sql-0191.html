A compensation analyst wants the third-distinct salary tier in each department — not the third employee when sorted, but the third unique salary value. The interviewer is probing DISTINCT before ranking: you must deduplicate salaries per department first, then rank the unique values, or you will produce incorrect results when there are salary ties at the top.

Eng distinct salaries (descending): 300, 250, 200, 150 → third = 200. Sales has only 2 distinct salaries (400, 350) → excluded. Ops distinct salaries: 120, 100, 90 → third = 90.

2. The SQL — DISTINCT + DENSE_RANK on the deduplicated set

WITH d AS (
  SELECT DISTINCT dept, salary FROM employee
),
r AS (
  SELECT dept, salary,
         DENSE_RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS rk
  FROM d
)
SELECT dept, salary AS third_highest_distinct_salary
FROM r
WHERE rk = 3
ORDER BY dept;

4. Strategic explanations

Why DISTINCT before ranking — the key insight

In Eng, salaries 300, 300, 250, 200, 150 appear for 5 employees. Without the dedup CTE, applying DENSE_RANK to the raw table would still give: 300→rank 1, 300→rank 1, 250→rank 2, 200→rank 3, 150→rank 4. The WHERE rk = 3 filter would return 200 — accidentally correct in this case. However, in Ops, without dedup: 120→1, 120→1, 100→2, 90→3, 90→3. The result 90 would be correct here too. The problem is that DENSE_RANK on the raw table returns the correct value at position rk=3, but only because DENSE_RANK already handles ties within a rank. The DISTINCT CTE is conceptually cleaner and makes the intent explicit: you are ranking salary values, not employees. Without the DISTINCT CTE, applying ROW_NUMBER (not DENSE_RANK) would be incorrect. The solution's approach — DISTINCT then DENSE_RANK — is unambiguous.

DENSE_RANK vs RANK vs ROW_NUMBER on the distinct set

After SELECT DISTINCT dept, salary, each (dept, salary) pair is unique. On this deduplicated result, all three ranking functions produce identical output (no ties remain within the distinct set). DENSE_RANK is semantically the most natural ("the Nth distinct salary level") and is the conventional choice for this question type.

Departments with fewer than 3 distinct salaries

The WHERE rk = 3 filter automatically excludes departments that have fewer than 3 distinct salary values — they simply produce no row with rk = 3. No explicit HAVING clause is needed. Sales has only rk = 1 and rk = 2, so it silently disappears from the result, satisfying the requirement "departments with fewer than 3 distinct salaries should NOT appear."

Performance note

For large tables, the DISTINCT CTE reduces the row count before the window function runs. An index on (dept, salary) allows the executor to satisfy the DISTINCT with an index scan rather than a sort + hash. In practice, salary cardinality per department is low, so the DISTINCT step is cheap regardless.

Dialect notes

The SELECT DISTINCT ... DENSE_RANK() OVER (...) pattern is ANSI SQL and works in PostgreSQL, Snowflake, BigQuery, SQL Server, and MySQL 8+. An alternative in PostgreSQL is to use DISTINCT ON (dept, salary) ordered by salary DESC and then apply a subquery to get the third row per dept — but the CTE approach is cleaner and more portable. BigQuery's QUALIFY clause (analogous to HAVING for window functions) can replace the outer WHERE: QUALIFY rk = 3.

Senior signal: always deduplicate to distinct salary values before ranking when the question asks for the Nth-highest distinct value; then DENSE_RANK on the clean set is unambiguous; and note that the WHERE rk = 3 filter implicitly excludes departments with fewer than 3 distinct values without needing an explicit HAVING guard.
