This is the classic growth accounting question every analytics-heavy company asks: decompose each month's active base into new, resurrected, existing (retained), and churned. The interviewer wants to see whether you can classify a customer-month purely from its neighbours — the previous and next active month — using LAG/LEAD, and whether you correctly project a churn into the month after a customer's last activity.

Customer 2 is the key edge case: an order in January, nothing in February, then an order in March. That February gap means March is not "existing" — it is a resurrection. Customers 3 and 4 are single-month customers whose churn must surface in the following month even though they have no order there.

2. The SQL — LAG/LEAD classification with a churn projection

The reference solution is written in SQLite (strftime, date(...,'+1 month')). Executed here on Postgres, the date functions translate directly — the logic is identical.

WITH user_activity AS (
    SELECT customer_id, TO_CHAR(order_date::date, 'YYYY-MM') AS ym
    FROM orders
    WHERE order_date >= '2024-12-01'   -- one-month buffer for LAG/LEAD
      AND order_date <  '2025-05-01'
    GROUP BY customer_id, TO_CHAR(order_date::date, 'YYYY-MM')
),
edges AS (
    SELECT customer_id, ym,
        (ym || '-01')::date              AS current_m,
        LAG ((ym || '-01')::date) OVER w AS prev_m,
        LEAD((ym || '-01')::date) OVER w AS next_m
    FROM user_activity
    WINDOW w AS (PARTITION BY customer_id ORDER BY ym)
),
classifications AS (
    SELECT ym, customer_id,
        CASE
            WHEN prev_m IS NULL                          THEN 'new'
            WHEN prev_m = current_m - INTERVAL '1 month'  THEN 'existing'
            ELSE                                               'resurrected'
        END AS bucket,
        CASE
            WHEN next_m IS NULL OR next_m > current_m + INTERVAL '1 month'
            THEN TO_CHAR(current_m + INTERVAL '1 month', 'YYYY-MM')
        END AS churn_ym
    FROM edges
)
SELECT ym AS year_month, bucket, COUNT(DISTINCT customer_id) AS customers
FROM (
    SELECT ym, customer_id, bucket FROM classifications
    UNION ALL
    SELECT churn_ym, customer_id, 'churned'
    FROM classifications WHERE churn_ym IS NOT NULL
) z
GROUP BY ym, bucket
ORDER BY ym,
    CASE bucket WHEN 'new' THEN 1 WHEN 'resurrected' THEN 2
                WHEN 'existing' THEN 3 ELSE 4 END;

The accounting balances: every customer that became inactive shows up exactly once as churned in the month after their final order, and customer 2's January → (gap) → March pattern produces both a February churn and a March resurrection — the same person, counted on both sides of the gap, which is exactly how growth accounting should treat a lapse-and-return.

4. Strategic explanations

The whole classification is "look at the neighbours"

Once activity is collapsed to one row per (customer, month), the bucket depends only on the gap to the previous active month: no previous month → new; previous month is exactly one month back → existing; previous month exists but is further back → resurrected. LAG partitioned by customer and ordered by month gives that neighbour in a single pass. LEAD looks forward the same way to decide whether this month is the customer's last.

Why churn needs a UNION ALL, not a fifth CASE branch

A new/resurrected/existing label describes a month the customer was active. Churn describes a month the customer was absent — there is no source row for it. So churn cannot be a branch of the same CASE; it must be a synthesised row. The query computes churn_ym = the month after a last-or-pre-gap activity, then UNION ALLs those rows in with bucket 'churned'. This "emit an extra row for a non-event" pattern is worth naming explicitly.

The one-month buffer in the WHERE clause

The filter starts at 2024-12-01, a month before the real report window. If a customer was active in December and again in January, that January row must be classified as existing — but without the December row in scope, LAG returns NULL and January is mislabelled new. Pulling one extra month of history lets the window functions see across the report boundary. Symmetrically, churn_ym can land in the month just past the window, which is intended.

Date arithmetic is the dialect-portability risk

The original is SQLite: strftime('%Y-%m', d), date(ym||'-01'), and date(d,'+1 month'). The Postgres equivalents shown are TO_CHAR(d,'YYYY-MM'), (ym||'-01')::date, and d + INTERVAL '1 month'. Comparing prev_m = current_m - INTERVAL '1 month' is reliable because both sides are first-of-month dates; never compare raw order_date values for "consecutive months" — a Jan 31 and a Mar 1 order are 29 days apart but two months apart.

What a senior would add

"This is a closed system: new + resurrected + existing for a month minus churned entering the next month reconciles to the month-over-month change in the active base. I'd assert that identity as a data-quality check. I'd also confirm the product's definition of churn — here it's a single inactive month; some teams require two or three consecutive silent months before calling churn, which would change the next_m threshold."

Senior signal: frame growth accounting as a neighbour-comparison via LAG/LEAD on a (customer, month) grid, explain that churned is a synthesised non-event row requiring UNION ALL rather than a CASE branch, justify the one-month WHERE buffer that keeps boundary months from being mislabelled, and offer the new+resurrected+existing−churned reconciliation identity as a built-in sanity check.
