DoorDash's pizza-merchant team wants to surface "build your own" combinations on the menu. Given a list of available toppings, generate all unique 3-topping combos — {Mushroom, Olive, Pepper} should appear once, not six times (one per ordering). This is the classic combinations-without-repetition problem solved cleanly with a self-join and an ordering predicate.

1. The source data — toppings

5 toppings → expected combinations = C(5,3) = 10.

2. The SQL — triple self-join with strict < ordering

SELECT a.topping AS t1, b.topping AS t2, c.topping AS t3
FROM toppings a
JOIN toppings b ON a.topping < b.topping
JOIN toppings c ON b.topping < c.topping;

4. Strategic explanations

Why < not <= — the dedup mechanism

A naive CROSS JOIN of 3 copies of toppings gives 5×5×5 = 125 rows, including duplicates like {Olive, Mushroom, Pepper} and {Pepper, Olive, Mushroom} (the same set in different orders) plus repeats like {Mushroom, Mushroom, Olive}. The trick:

  • a.topping < b.topping < c.topping — strict less-than enforces both distinct values (a ≠ b ≠ c) and a canonical ordering (alphabetical) so each set surfaces exactly once.
  • <= would allow {Mushroom, Mushroom, Olive} — combinations with repetition, not what was asked.

(no predicate)n^3 = 125Cartesian (with repeats and orders)

Generalizing to k-combinations

The pattern extends directly: k self-joins with < chained between adjacent aliases.

-- 4-topping combo
SELECT a.topping, b.topping, c.topping, d.topping
FROM toppings a
JOIN toppings b ON a.topping < b.topping
JOIN toppings c ON b.topping < c.topping
JOIN toppings d ON c.topping < d.topping;

For k unknown at query-time, use a recursive CTE instead of a fixed number of joins.

Recursive CTE — variable-size combinations
WITH RECURSIVE combos AS (
  SELECT topping AS path, topping AS last_topping, 1 AS depth
  FROM toppings
  UNION ALL
  SELECT c.path || ',' || t.topping, t.topping, c.depth + 1
  FROM combos c
  JOIN toppings t ON t.topping > c.last_topping
  WHERE c.depth < 3
)
SELECT path FROM combos WHERE depth = 3;
The performance trap — cardinality explosion

This pattern is O(nk) in the worst case (without the predicate) and O(nk/k!) with the ordering predicate. For 5 toppings × 3 = 10 rows. For 50 toppings × 3 = 19,600. For 500 toppings × 5 = 250 billion. Don't run unbounded combination generation against a real product catalog — bound the depth and pre-filter:

JOIN toppings b ON a.topping < b.topping
                AND b.popularity_rank <= 20  -- top 20 only
The product-sense follow-up — co-occurrence ranking

The PM's actual question is rarely "all combos" — it's "which 3-topping combos are most ordered together?" That requires the orders table and counts, not pure combinatorics:

SELECT a.topping, b.topping, c.topping, COUNT(DISTINCT oa.order_id) AS orders
FROM order_toppings oa
JOIN order_toppings ob USING (order_id)
JOIN order_toppings oc USING (order_id)
JOIN toppings a ON a.topping = oa.topping
JOIN toppings b ON b.topping = ob.topping AND a.topping < b.topping
JOIN toppings c ON c.topping = oc.topping AND b.topping < c.topping
GROUP BY a.topping, b.topping, c.topping
ORDER BY orders DESC LIMIT 10;
The dietary-constraint wrinkle

Real menus exclude impossible combos: pepperoni + cheese-free, anchovy + dessert pizza. Add a compatible(topping_a, topping_b) table or a NOT IN ('forbidden_pair_set') guard.

Senior signal: name the strict-less-than as the dedup-and-distinct mechanism in one breath, contrast it crisply against permutations (≠) and combinations-with-repetition (<=), and pre-empt the cardinality explosion at scale. Pivoting to "what the PM actually wants" (co-occurrence ranking from the orders table) is the over-the-bar finish.
