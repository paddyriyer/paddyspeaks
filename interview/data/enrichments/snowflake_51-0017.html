Geological and scientific databases frequently store sample descriptions as free-text strings: "Approximately 3.8 billion years old" or "45.2 million years old". The challenge: extract the numeric magnitude and convert it to a uniform unit (years) so the data can be sorted, filtered, and compared. This question tests regex-based extraction, ILIKE pattern matching for unit detection, and the CASE WHEN / multiplier pattern for unit normalisation.

1. The source data — rock_samples

RS-005 has no unit keyword — the multiplier defaults to 1. RS-006 has no numeric token — the regex returns NULL, which propagates through the multiplication. These are the edge cases that exercise both branches of the logic.

2. The SQL — regex extraction with unit-multiplier CASE WHEN

SELECT sample_id, description,
       NULLIF(substring(description from '\d+(?:\.\d+)?'), '')::numeric AS numeric_age,
       CASE
         WHEN description ILIKE '%billion%'  THEN 1e9
         WHEN description ILIKE '%million%'  THEN 1e6
         WHEN description ILIKE '%thousand%' THEN 1e3
         ELSE 1 END AS unit_multiplier,
       NULLIF(substring(description from '\d+(?:\.\d+)?'), '')::numeric
         * CASE
             WHEN description ILIKE '%billion%'  THEN 1e9
             WHEN description ILIKE '%million%'  THEN 1e6
             WHEN description ILIKE '%thousand%' THEN 1e3
             ELSE 1 END AS age_in_years
FROM rock_samples;

4. Strategic explanations

The regex pattern \d+(?:\.\d+)?

\d+ matches one or more digits; (?:\.\d+)? is a non-capturing group that optionally matches a decimal point followed by more digits. The ?: prefix suppresses capture — Postgres/Snowflake's substring(text FROM pattern) returns the first full match when the pattern has no capturing group, or the first captured group when one is present. Using ?: ensures the whole pattern is returned, not just the decimal part. This regex handles integers (450) and decimals (3.5, 2.1) but not scientific notation — a stated limitation.

Why NULLIF(..., '') before the cast

When no match is found, substring returns NULL in Postgres but can return an empty string in some contexts. NULLIF(expr, '') converts an empty string to NULL before the ::numeric cast — preventing a cast error on empty strings. It is a defensive pattern when the regex might not match. Without it, a row with no numeric token would throw a runtime error instead of cleanly returning NULL.

ILIKE for case-insensitive unit detection

ILIKE '%billion%' matches "Billion", "BILLION", "billion" without the need for LOWER(description) LIKE '%billion%'. In Snowflake ILIKE is a first-class operator. In MySQL, LIKE is case-insensitive by default for most collations; in SQL Server, use LIKE with a case-insensitive collation or LOWER().

Dialect differences: regex extraction
  • Postgres (solution dialect): substring(text FROM pattern) with POSIX regex.
  • Snowflake native: REGEXP_SUBSTR(description, '\\d+(\\.\\d+)?', 1, 1, 'e', 1) — the 'e' flag enables extraction and the last 1 picks group 1. Or more simply: REGEXP_SUBSTR(description, '\\d+(?:\\.\\d+)?') for the full match.
  • BigQuery: REGEXP_EXTRACT(description, r'\d+(?:\.\d+)?').
  • MySQL: REGEXP_SUBSTR(description, '[0-9]+(\.[0-9]+)?') (MySQL 8.0+).
The CASE WHEN multiplier repetition — and how to avoid it

The query repeats the CASE WHEN block twice: once to expose unit_multiplier as a column, and once inside the multiplication for age_in_years. In practice a CTE or subquery eliminates the duplication:

WITH extracted AS (
  SELECT sample_id, description,
         NULLIF(substring(description from '\d+(?:\.\d+)?'), '')::numeric AS numeric_age,
         CASE WHEN description ILIKE '%billion%'  THEN 1e9
              WHEN description ILIKE '%million%'  THEN 1e6
              WHEN description ILIKE '%thousand%' THEN 1e3
              ELSE 1 END AS unit_multiplier
  FROM rock_samples
)
SELECT *, numeric_age * unit_multiplier AS age_in_years FROM extracted;

This is cleaner and avoids the engine evaluating the CASE twice (though most optimisers would common-subexpression-eliminate it anyway).
