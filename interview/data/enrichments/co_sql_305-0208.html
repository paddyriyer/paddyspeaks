Every Monday morning, the Meta product leads open the Growth Review and the first chart on the slide is MAU by month. Monthly Active Users — distinct users who took any action in the calendar month — is the textbook health metric for a B2C product. The query is a one-liner; the strategic depth is in what counts as "active" and what timezone defines "month".

Three months of data. U_001 is active in all three months, but should count once per month, not three times within April.

2. The SQL — DATE_TRUNC + COUNT(DISTINCT user_id)

SELECT
  DATE_TRUNC('month', action_date) AS month,
  COUNT(DISTINCT user_id) AS mau
FROM activity
GROUP BY DATE_TRUNC('month', action_date)
ORDER BY month;

4. Strategic explanations

Why DATE_TRUNC over EXTRACT(MONTH FROM ...)

EXTRACT(MONTH FROM action_date) returns the integer 1-12 — so April 2025 and April 2026 collapse into the same bucket. Use DATE_TRUNC('month', action_date), which returns a date pinned to the first of the month (e.g., 2026-04-01) — year-distinct and chart-ready. SQLite has no DATE_TRUNC; use strftime('%Y-%m', action_date) or date(action_date, 'start of month').

The "active" definition trap — what counts?

The query above counts any row in activity as a sign of life. Real product orgs draw a sharper line: a passive push-notification open isn't the same as a real session. Production MAU usually filters:

WHERE action_type IN ('feed_view', 'post', 'comment', 'message', 'reaction')
  AND session_duration_seconds >= 5

Always pin the active-event whitelist with the PM in the kickoff — "what counts as active" is the most common silent disagreement in growth analytics.

The timezone trap — UTC vs user-local

action_date is usually stored in UTC. A user in Tokyo posting at 11pm local on March 31 lands in UTC on April 1 — and counts toward April MAU instead of March. For global products this skews month-end heavily. The fix:

DATE_TRUNC('month', action_ts AT TIME ZONE user_timezone) AS month

...assuming a users.timezone column. If you don't have user timezone, at minimum use the company's reporting timezone (PT for Meta, ET for finance teams).

L28 vs calendar-month MAU

Calendar-month MAU has a known bias: a user active on the 1st gets credit for the whole month; a user active on the 30th barely scrapes in. Meta and Snap report L28 (rolling 28-day window, ending on the report date) precisely because it's resampling-stable.

SELECT report_date,
  COUNT(DISTINCT user_id) AS l28_mau
FROM activity
WHERE action_date BETWEEN report_date - INTERVAL '27 days' AND report_date
GROUP BY report_date;

For an interview answer, present calendar-month MAU first (what was asked) then volunteer L28 as the "what we actually use in production" follow-up.

The DAU/MAU stickiness ratio — the senior follow-up

The PM's actual question is usually "is the product getting stickier?" — the DAU/MAU ratio (Facebook's famous >50% benchmark). Run both metrics and divide:

WITH dau AS (
  SELECT action_date AS d, COUNT(DISTINCT user_id) AS dau
  FROM activity GROUP BY action_date
),
mau AS (
  SELECT d, COUNT(DISTINCT user_id) AS mau
  FROM activity, generate_series(...) g(d)  -- date spine
  WHERE action_date BETWEEN d - INTERVAL '29 days' AND d
  GROUP BY d
)
SELECT d, dau, mau, ROUND(100.0 * dau / mau, 1) AS stickiness_pct
FROM dau JOIN mau USING (d);
WAU and the engagement pyramid

Mature product orgs report DAU / WAU / MAU together — the "engagement pyramid". WAU/MAU sniffs out users who log in only once a month (low quality); DAU/WAU sniffs out daily-vs-weekly cadence shifts.

Senior signal: name DATE_TRUNC's year-distinct property over EXTRACT, raise the timezone trap unprompted, and pre-empt the L28 vs calendar-month conversation. The DAU/MAU stickiness ratio with a date-spine'd L28 implementation is the over-the-bar finish.
