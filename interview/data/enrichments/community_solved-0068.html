An education platform stores exam marks in long format — one row per student per subject — and the registrar needs a pass/fail verdict per student against two compound rules: a per-subject floor (≥ 40 in every subject) and an overall threshold (≥ 50% across all four). The interviewer is testing whether you can pivot long rows into a wide per-student shape with conditional aggregation, and then encode a multi-condition gate without missing the "every subject" semantics.

1. The source data — exam_score

studentidsubjectidmarks

Student 2 hits exactly the 40 floor in every subject but a low total; student 3 is a top scorer in three subjects yet flunks one with a 35. These two boundary students — one failing the overall rule, one failing the per-subject rule — are the whole point of the question.

2. The SQL — conditional-aggregation pivot, then a compound CASE

WITH marks_cte AS (
  SELECT studentid,
         COALESCE(SUM(CASE WHEN subjectid::text = '1' THEN marks::int END), 0) AS subject1,
         COALESCE(SUM(CASE WHEN subjectid::text = '2' THEN marks::int END), 0) AS subject2,
         COALESCE(SUM(CASE WHEN subjectid::text = '3' THEN marks::int END), 0) AS subject3,
         COALESCE(SUM(CASE WHEN subjectid::text = '4' THEN marks::int END), 0) AS subject4,
         COALESCE(SUM(marks::int), 0) AS total_marks
  FROM exam_score
  WHERE marks::text ~ '^-?[0-9]+$'
  GROUP BY studentid
)
SELECT studentid, subject1, subject2, subject3, subject4, total_marks,
  CASE WHEN LEAST(subject1, subject2, subject3, subject4) >= 40
        AND (1.0*total_marks/400) >= 0.5 THEN 'pass'
       ELSE 'fail' END AS result
FROM marks_cte;

4. Strategic explanations

The pivot — SUM(CASE WHEN subjectid = n)

Long-to-wide reshaping uses conditional aggregation: SUM(CASE WHEN subjectid = '1' THEN marks END) collapses to a single subject1 column per student because, after GROUP BY studentid, only that student's subject-1 row is non-NULL inside the SUM. This is the universal pivot idiom — portable to every dialect, unlike a vendor PIVOT operator. SUM (rather than MAX) is the safe aggregate here: if a student somehow has duplicate subject rows, SUM exposes the inflated mark instead of silently hiding it.

LEAST(...) is the elegant "every subject" test

The per-subject rule is "≥ 40 in each of the four". Rather than four chained AND comparisons, LEAST(subject1, subject2, subject3, subject4) >= 40 says it once: if the minimum subject clears 40, all of them do. This is the senior shortcut for an all-of-N condition — and its mirror, GREATEST(...) < threshold, encodes a none-of-N condition just as compactly.

The NULL trap that LEAST hides — a missing subject

This is the question's deepest subtlety. LEAST in Postgres ignores NULL arguments — LEAST(80, NULL, 90, 60) returns 60, not NULL. So if a student is missing a subject row entirely, the COALESCE(..., 0) turns that subject into 0 — which correctly fails the floor. But drop the COALESCE and the subject stays NULL; LEAST skips it, and a student who never sat subject 3 could be marked "pass" on three subjects. The COALESCE(..., 0) is not cosmetic — it is what forces an absent subject to fail rather than vanish. Worth saying out loud: a missing row and a genuine zero are treated identically here, which is the right call for a pass gate but should be a deliberate decision.

The percentage arithmetic — why 1.0 * and why 400

total_marks / 400 in pure integer arithmetic would truncate to 0 for everyone, so the CASE would always fail the overall rule. 1.0 * total_marks / 400 forces floating-point division. The divisor is 400 because the maximum is 100 per subject × 4 subjects; "50% overall" then means ≥ 0.5. If the rubric instead meant "average mark ≥ 50", that is the same arithmetic — but confirm the interpretation, because "50% overall" and "average of 50" only coincide when every subject has the same maximum.

Common mistake — checking only the total

A frequent wrong answer tests total_marks >= 200 alone and declares student 3 a pass — missing that one subject is below 40. Compound pass rules need both gates ANDed; the per-subject floor is not implied by a healthy total. Reading "at least 40 in each subject" as a separate, non-negotiable condition is what the question rewards.

Senior signal: pivot with SUM(CASE), then compress the "≥ 40 in every subject" rule into a single LEAST(...) >= 40 — and crucially explain that LEAST silently skips NULLs, so the COALESCE(..., 0) is what forces a missing subject to fail instead of disappearing. Forcing float division with 1.0 * and ANDing both gates separately is the complete answer.
