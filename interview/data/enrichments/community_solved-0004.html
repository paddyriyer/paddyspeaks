A Community genealogy puzzle: given a parent/child table with each child's birth year, compute how old each parent was when that child was born. It looks like simple subtraction, but it hides a quiet trap — the very first generation has no parent in the table, so you cannot subtract anything, and the solution has to decide what to do about it.

1. The source data — tab_parent

parentchildbirthyear
caroldave1975
daveerin2000
erinfrank2025

Each row records a child and the year that child was born. The chain is carol → dave → erin → frank. The catch: carol never appears as a child, so her own birth year is unknown — and "how old was carol when dave was born" is genuinely unanswerable from this data.

2. The SQL — WITH RECURSIVE chaining birth years

WITH RECURSIVE cte AS (
    SELECT parent, child, birthyear, 0 AS parent_age FROM tab_parent
    WHERE parent = 'carol'
    UNION ALL
    SELECT tp.parent, tp.child, tp.birthyear,
           (tp.birthyear::int - cte.birthyear::int) AS parent_age
    FROM cte
    JOIN tab_parent tp ON cte.child = tp.parent
)
SELECT * FROM cte;

The anchor seeds the root generation (carol → dave) with parent_age = 0 — a placeholder, because carol's age is unknowable. Each recursive step joins cte.child = tp.parent: the previous row's child becomes the next row's parent, and tp.birthyear - cte.birthyear is the new parent's age — the new child's birth year minus the parent's own birth year (which the previous row carried).

3. The resulting analysis

parentchildbirthyearparent_age
caroldave19750
daveerin200025
erinfrank202525

dave was born 1975; erin (his child) was born 2000, so dave was 25. erin was born 2000; frank arrived 2025, so erin was 25 too. The carol row reads parent_age = 0 — not because carol was a newborn, but because the anchor literally hardcoded 0. That row is the honest "unknown" of the dataset.

4. Strategic explanations

Why cte.birthyear is the parent's birth year, not the child's

This is the subtle part. In the recursive step, tp is the new row (parent → grandchild) and cte is the previous row. The previous row's birthyear is the birth year of its child — who is exactly the new row's parent. So tp.birthyear - cte.birthyear = (grandchild's birth year) − (parent's birth year) = the parent's age. The join condition cte.child = tp.parent is what makes that alignment hold. Trace one hop slowly in the interview; that is the whole question.

The unanswerable root — name it, do not fake it

carol's parent_age of 0 is a placeholder for "no data". A junior reports 0 as if it were real; a senior says out loud: "the root generation has no parent record, so its age is NULL/unknown — I am seeding it with 0 only as a sentinel." If the consumer should never see a fake 0, anchor with NULL instead:

SELECT parent, child, birthyear, NULL::int AS parent_age
FROM tab_parent WHERE parent = 'carol'

and the root row honestly reads NULL rather than a misleading 0.

Do you even need recursion here?

If every parent's own birth year is recoverable, this is a plain self-join — no recursion required:

SELECT p.parent, p.child, p.birthyear,
       p.birthyear - g.birthyear AS parent_age
FROM tab_parent p
LEFT JOIN tab_parent g ON g.child = p.parent;

Each row's parent is looked up as somebody else's child in g; p.birthyear - g.birthyear is the age. Rows whose parent is a root (carol) get NULL from the LEFT JOIN — the same honest "unknown". Recursion only earns its keep if the question demands cumulative walking (e.g. ancestor depth). Knowing the set-based shortcut is the senior move; reaching for WITH RECURSIVE reflexively is not.

Edge cases: missing links and impossible ages

If a child's parent is absent (a broken link mid-tree), the recursive join finds nothing and that branch silently stops — rows downstream vanish. A LEFT JOIN self-join would keep them with NULL age. Also guard the arithmetic: a parent "born" after their child (dirty data) yields a negative age — surface those with WHERE parent_age < 0 rather than trusting the input.

String birth years — the ::int casts

The solution casts birthyear::int defensively, implying the column may arrive as text ("1975" or even " 1975 "). If the raw data is messy, scrub it first: NULLIF(regexp_replace(birthyear, '\\D', '', 'g'), '')::int. Subtracting two text values would either error or, worse, do something unexpected.

Senior signal: articulate that tp.birthyear - cte.birthyear aligns to the parent's age because of the cte.child = tp.parent join, treat the rootless first generation as an honest NULL rather than a fake 0, and recognise that a plain self-join solves this unless the question genuinely needs cumulative tree-walking.
