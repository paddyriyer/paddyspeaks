An analytics team wants to tier its customer base: "for every customer, what fraction of the book sits at or below their lifetime value?" A customer at the 0.90 mark is in the top 10% of spenders. The interviewer is checking whether you reach for the right ranking window function — specifically one that returns a cumulative fraction, not an ordinal position — and whether you know how it treats customers tied on LTV.

1. The source data — customer

Ten customers in all (ids 1–10), with one detail that decides the interview: ids 3 and 4 are tied at 200. A percentile function must give tied rows the same percentile — and the rule it uses to do so is the whole question.

2. The SQL — CUME_DIST() over LTV

SELECT id, lifetime_value,
       ROUND(CUME_DIST() OVER (ORDER BY lifetime_value)::numeric, 3) AS percentile
FROM customer
ORDER BY lifetime_value DESC, id;

CUME_DIST() returns, for each row, the count of rows with a value less than or equal to the current row divided by the total row count. That definition is exactly "fraction of customers with LTV ≤ theirs" — the question's wording mapped one-to-one onto a built-in.

4. Strategic explanations

Why CUME_DIST and not PERCENT_RANK

The two are easy to confuse. CUME_DIST = (rows ≤ current) / N — it always reaches 1.0 at the top and is a genuine "≤ me" fraction. PERCENT_RANK = (rank − 1) / (N − 1) — it always starts at 0.0 for the minimum and is a "strictly-below me" fraction. The question literally says "less than or equal to", which is the CUME_DIST definition. Picking the function whose definition matches the prose is the senior move.

Tie handling is the trap

Both 200-LTV rows get 0.400, not 0.300 and 0.400. CUME_DIST evaluates the whole peer group at once: every row sharing a value gets the cumulative fraction computed at the last member of that group. If you instead used ROW_NUMBER() / COUNT(*) you would hand the two tied customers different percentiles (0.3 and 0.4) decided arbitrarily by physical order — almost certainly wrong for a percentile.

The ::numeric cast and ROUND

CUME_DIST returns double precision. PostgreSQL's two-argument ROUND(value, scale) only accepts numeric, so ROUND(some_double, 3) raises "function round(double precision, integer) does not exist". The explicit ::numeric cast is not cosmetic — it is what makes the rounding compile. Many candidates omit it and get a confusing error.

NULL LTV and ordering

If lifetime_value can be NULL, the default ORDER BY lifetime_value sorts NULLs last, so a NULL-LTV customer would land at percentile 1.000 — ranked as the top spender, which is nonsense. Decide explicitly: filter WHERE lifetime_value IS NOT NULL, or use ORDER BY lifetime_value NULLS FIRST so missing-value customers sink to the bottom of the distribution.

Dialect notes

CUME_DIST is ANSI SQL and works in PostgreSQL, SQL Server, Oracle, Snowflake, BigQuery, and MySQL 8+. Older MySQL (< 8.0) has no window functions at all — there you would emulate it with a correlated subquery (SELECT COUNT(*) FROM customer c2 WHERE c2.lifetime_value <= c1.lifetime_value)::float / (SELECT COUNT(*) FROM customer), which is O(n²) and far slower.

Senior signal: map "fraction with value ≤ mine" directly to CUME_DIST() (not PERCENT_RANK, which is strictly-below and starts at 0), know that tied rows share the cumulative fraction of the last peer, cast to numeric before two-argument ROUND, and decide NULL-LTV placement deliberately instead of letting it default to the 100th percentile.
