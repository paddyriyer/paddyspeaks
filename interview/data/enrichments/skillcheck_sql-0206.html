This looks like a textbook running total — and that is the trap. The intuitive answer, SUM(amount) OVER (ORDER BY txn_date), does not produce a strict per-row running total when txn_date has duplicate values. The cause is the default window frame: an ORDER BY with no explicit frame uses RANGE, not ROWS — and under RANGE the “current row” silently expands to cover every peer row sharing the same ORDER BY value.

2. The trap — SUM(amount) OVER (ORDER BY txn_date)

SELECT txn_id, txn_date, amount,
       SUM(amount) OVER (ORDER BY txn_date) AS running_total
FROM account_ledger
ORDER BY txn_id;

With no frame clause, the window defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. Under RANGE, CURRENT ROW means “this row and every row tied with it on the ORDER BY key.” Both 2025-02-03 rows therefore get the same figure — the sum through the end of that date — and likewise for 2025-02-05:

3. The correct query

SELECT txn_id, txn_date, amount,
       SUM(amount) OVER (ORDER BY txn_date, txn_id
                         ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_total
FROM account_ledger
ORDER BY txn_id;

Two fixes: switch the frame to ROWS, where CURRENT ROW means literally this one row; and add txn_id as a tiebreaker so the order within a date is deterministic.

4. Strategic explanations

RANGE vs ROWS — the one-line difference that changes the answer

ROWS counts physical rows: ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW is exactly “every row up to and including this one.” RANGE counts logical peers: CURRENT ROW stretches to cover all rows with the same ORDER BY value. The two agree only when the ORDER BY key is unique — the moment a value repeats, RANGE plateaus across the tie group.

The default frame is RANGE, not ROWS

This is the heart of the trap. SUM(x) OVER (ORDER BY k) is shorthand for SUM(x) OVER (ORDER BY k RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW). The popular claim that OVER (ORDER BY k) equals OVER (ORDER BY k ROWS UNBOUNDED PRECEDING) holds only for a unique k. Repeated values break the equivalence.

When RANGE is actually what you want

RANGE is not a defect — it is correct for “total through end of day” reporting, where every transaction on a date should share one cumulative figure. The skill is choosing deliberately: RANGE for peer-aware totals, ROWS for a strict row-by-row ledger.

Dialect notes

The RANGE default is ANSI SQL and holds in PostgreSQL, Oracle, SQL Server, Snowflake, BigQuery, and DuckDB; MySQL 8.0 follows it too. There is no engine where a bare ORDER BY in a window defaults to ROWS, so this tie trap is universal.

Senior signal: know that an OVER (ORDER BY ...) with no frame clause defaults to RANGE, and that RANGE makes CURRENT ROW include all tied peers — so a “running total” over a non-unique key plateaus across ties. Write ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW and add a unique tiebreaker to the ORDER BY for a true per-row running total.
