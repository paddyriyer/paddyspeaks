Cloudflare's payments team is building a checkout dropdown and needs the list of payment methods customers have actually used — one entry per method, no repeats. It is the textbook SELECT DISTINCT, but the interviewer is probing whether you understand how DISTINCT treats NULL and whether DISTINCT is even the right tool versus GROUP BY.

1. The source data — payments

payment_idmethodamountstatus
1credit_card$50completed
2paypal$75completed
3credit_card$20completed
4wire$900completed
5paypal$30completed
6NULL$10failed

credit_card and paypal each appear twice, and row 6 has a NULL method — a failed payment where the method was never recorded. That NULL decides whether the dropdown ends up with a blank entry.

2. The SQL — SELECT DISTINCT

SELECT DISTINCT method FROM payments ORDER BY method;

DISTINCT collapses repeated values to one row each. ORDER BY method sorts the result so the dropdown is alphabetical.

3. The resulting analysis

method
credit_card
paypal
wire
NULL

Four rows from six. The two duplicates collapsed — but notice NULL survived as its own row. DISTINCT keeps exactly one NULL, and Postgres sorts NULLs last by default. A real product would not want a blank dropdown option.

4. Strategic explanations

NULL is a distinct value — filter it

DISTINCT treats all NULLs as equal to each other (so you get one, not several) but as distinct from every real value (so it is not absorbed). The result is a stray blank row. For a UI list you almost always want it gone:

SELECT DISTINCT method FROM payments
WHERE method IS NOT NULL
ORDER BY method;

State the choice explicitly: a NULL method is "method unknown", and an unknown is not a selectable option. The filter is a product decision, not a cosmetic one.

DISTINCT vs GROUP BY — equivalent, until they aren't

For a single column, these two are interchangeable and modern planners generate the same plan:

SELECT method FROM payments
WHERE method IS NOT NULL
GROUP BY method
ORDER BY method;

Prefer GROUP BY the moment you need a per-method aggregate — usage counts make the dropdown smarter (popular methods first). That is the natural follow-up, and DISTINCT cannot do it:

SELECT method, COUNT(*) AS uses
FROM payments
WHERE method IS NOT NULL
GROUP BY method
ORDER BY uses DESC, method;
The DISTINCT applies to the whole row — a classic misread

DISTINCT is not a function you apply to one column; it deduplicates the entire projected row. SELECT DISTINCT method, status FROM payments returns distinct pairs — credit_card could appear several times, once per status. And SELECT DISTINCT(method), status does not mean "distinct on method only" — the parentheses are decorative; it still dedups both columns. To dedup on a subset, use DISTINCT ON (method).

Edge cases: case sensitivity and whitespace

DISTINCT is an exact-match operation. 'PayPal', 'paypal', and 'paypal ' (trailing space) are three different values and all three survive — producing a messy triple-entry dropdown from what is really one method. Dirty source data needs normalising first: SELECT DISTINCT LOWER(TRIM(method)). On an empty table, DISTINCT simply returns zero rows — no error, no NULL.

Senior signal: recognise that DISTINCT keeps NULL as its own row and filter it for a UI list as a deliberate product call, know DISTINCT and single-column GROUP BY are equivalent but switch to GROUP BY when a count is wanted, and remember DISTINCT dedups the whole row — normalising case and whitespace before deduplicating dirty data.
