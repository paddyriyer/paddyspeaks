A Community civic dataset records ballots, but with a twist: a single voter may name several candidates on one ballot, and their vote is meant to be split equally across the names they picked. The task is to total the fractional votes per candidate and rank them. The interviewer is checking whether you compute the per-voter split with a window function and handle the rounding honestly.

1. The source data — election_result

votercandidate
Voter1Alice
Voter1Bob
Voter2Alice
Voter3Bob
Voter3Carol
Voter3Alice
Voter4Carol

Voter1 named two candidates, so each gets ½ a vote. Voter2 named one — a full vote. Voter3 named three, so each gets ⅓. Voter3 is the edge case: a three-way split yields 0.333 repeating, which forces a rounding decision.

2. The SQL — split the vote with a window SUM

with vote_split as(
select voter,
    candidate,
    cast(1.0/sum(1) over(partition by voter) as decimal(10,3)) as num_votes
from election_result
)
select candidate, sum(num_votes) as votes
from vote_split
where candidate is not null
group by candidate
order by votes desc;

sum(1) over (partition by voter) counts how many rows — candidates — that voter has, without collapsing the rows. 1.0 / that count is each candidate's share of that ballot. The outer query sums every fractional share a candidate received.

3. The resulting analysis

candidatevotes
Alice1.833
Carol1.333
Bob0.833

Alice = 0.5 (Voter1) + 1.0 (Voter2) + 0.333 (Voter3) = 1.833. Carol = 0.333 (Voter3) + 1.0 (Voter4) = 1.333. Bob = 0.5 (Voter1) + 0.333 (Voter3) = 0.833. Alice wins. The totals are slightly under what the headcount implies, for a reason worth understanding.

4. Strategic explanations

Why a window SUM instead of GROUP BY

You need two things at once: each candidate's row must survive, and each must know how many candidates shared its voter's ballot. A plain GROUP BY voter would collapse the rows and lose the candidate detail. sum(1) OVER (PARTITION BY voter) — equivalent to COUNT(*) OVER (PARTITION BY voter) — attaches the per-voter headcount to every row while keeping all rows. That is the defining use of a window aggregate.

The rounding leak — why totals do not sum to the voter count

Four voters cast one vote each, so the grand total should be 4.000. But 1.833 + 1.333 + 0.833 = 3.999. The missing 0.001 is Voter3's three-way split: 1/3 rounded to 0.333 three times reconstitutes 0.999, not 1.000. Rounding before aggregating leaks precision. The fix is to round only at the very end:

select candidate,
       round(sum(1.0 / cnt), 3) as votes
from (
  select candidate, count(*) over (partition by voter) as cnt
  from election_result
) s
where candidate is not null
group by candidate;

Keep full precision through the SUM, round once for display. Naming this leak is the senior catch.

Why 1.0 / and not 1 /

1 / 3 in integer arithmetic is 0 — every split candidate would score zero. Writing 1.0 makes the numerator numeric, so the division produces a real fraction. The same integer-division hazard that bites percentage queries bites vote-splitting.

The WHERE candidate IS NOT NULL guard

A ballot row with a NULL candidate — a blank or spoiled entry — would still be counted by sum(1) OVER (PARTITION BY voter), shrinking every other candidate's share on that ballot, and then GROUP BY would create a phantom NULL "candidate" in the results. Filtering NULL candidates in the outer query removes the phantom row — but note it does not retroactively fix the denominator. If spoiled entries should not dilute a ballot at all, filter them in the inner query before the window count runs.

Dialect note — decimal(10,3)

CAST(... AS decimal(10,3)) is standard SQL and runs in Postgres (decimal is a synonym for numeric). Some engines spell precision differently or require numeric explicitly; the construct is portable enough, but on an unfamiliar engine prefer ROUND(x, 3) for display formatting rather than relying on a column type cast.

Senior signal: use COUNT(*) OVER (PARTITION BY voter) to keep rows while attaching the per-ballot headcount, force numeric division with 1.0, and — the over-the-bar move — round only after the final SUM so a three-way split does not silently leak 0.001 from the totals.
