Datadog's commerce analytics team tracks product quality, and one early-warning signal is the return rate per product: of all orders for a SKU, what fraction came back? The data lives in two tables — every order in orders, and a row in returns only when an order was sent back. The interview tension is the join: pick the wrong type and you either lose products with zero returns or double-count.

Product 10 has 4 orders, 2 returned. Product 20 has 2 orders, 1 returned. Product 30 has 1 order and nothing in returns — the edge case. A return rate report that drops product 30 entirely is wrong; "0% returned" is a real, important answer.

2. The SQL — LEFT JOIN + ratio

SELECT o.product_id,
  100.0*COUNT(r.order_id)/COUNT(*) AS return_pct
FROM orders o LEFT JOIN returns r ON o.order_id = r.order_id
GROUP BY o.product_id;

Two counts after a LEFT JOIN: COUNT(*) counts every order row, COUNT(r.order_id) counts only the rows that matched a return (non-NULL). The ratio is the return percentage.

4. Strategic explanations

Why COUNT(*) and COUNT(r.order_id) differ — the whole trick

After a LEFT JOIN, unmatched left rows still appear, with the right-side columns set to NULL. COUNT(*) counts rows regardless of NULLs; COUNT(r.order_id) counts only non-NULL values. So for product 30's single unmatched order, COUNT(*) = 1 but COUNT(r.order_id) = 0. Swap to COUNT(*) in the numerator and every product reads 100% — a classic, silent bug.

The INNER JOIN trap

Change LEFT to INNER and only orders that were returned survive. Product 30 vanishes; product 10 looks like 2 orders, 2 returns = 100%. The report would claim every product has a catastrophic return rate. A return-rate denominator must be all orders, so the join from orders must be LEFT.

The integer-division trap

The solution writes 100.0*COUNT(...), not 100*COUNT(...). COUNT returns bigint; bigint / bigint does integer division and truncates — 1/2 becomes 0, so product 20 would report 0% instead of 50%. The 100.0 literal is numeric, which promotes the whole expression to numeric arithmetic. Equivalent fixes: CAST(... AS numeric) or ... * 1.0.

Duplicate rows in returns — an edge case to raise

The solution assumes at most one return row per order. If a return can be logged twice, the LEFT JOIN fans out and COUNT(r.order_id) exceeds COUNT(*) — you can report over 100%. Defend against it by counting distinct returned orders:

SELECT o.product_id,
  100.0 * COUNT(DISTINCT r.order_id) / COUNT(DISTINCT o.order_id) AS return_pct
FROM orders o LEFT JOIN returns r ON o.order_id = r.order_id
GROUP BY o.product_id;
A clearer aggregate-only alternative

You can skip the join entirely if returns can be expressed as a flag, using a conditional count — or compute it without the join via FILTER:

SELECT o.product_id,
  100.0 * COUNT(*) FILTER (WHERE r.order_id IS NOT NULL) / COUNT(*) AS return_pct
FROM orders o LEFT JOIN returns r ON o.order_id = r.order_id
GROUP BY o.product_id;

FILTER makes the "returned vs all" intent explicit and reads better than relying on COUNT's NULL behaviour. Should cancelled orders count toward the denominator? That is a definition question worth asking — the status column is right there.

Senior signal: reach for LEFT JOIN so zero-return products still appear, articulate exactly why COUNT(r.order_id) (non-NULL) differs from COUNT(*) (rows), force numeric division with a 100.0 literal, and pre-empt the over-100% bug from duplicate return rows with COUNT(DISTINCT …).
