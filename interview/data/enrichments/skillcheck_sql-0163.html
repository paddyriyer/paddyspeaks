A product catalog needs each item's tags rendered as a single comma-separated string, ordered so the heaviest-weighted tag leads. The interviewer is probing whether you know STRING_AGG accepts an ORDER BY inside the aggregate call, and whether you handle the dirty-data wrinkle: some tag rows have a NULL tag value, and one item has only NULL tags — it should drop out of the result entirely.

1. The source data — tag

item_idtagweight(narrative)
1red3item 1 — middle weight
1blue1item 1 — lowest weight, ends up last
1green5item 1 — highest weight, leads
2big10item 2 — highest, leads
2NULL2item 2 — NULL tag, must be skipped
2fast7item 2 — second
3NULL5item 3 — its ONLY row, and tag is NULL

Two edge cases drive the answer. Item 2 has a NULL-tag row in the middle — it must vanish without leaving a stray , ,. Item 3's only tag is NULL, so after skipping it the item has nothing left and should not appear in the output at all.

2. The SQL — STRING_AGG with an in-aggregate ORDER BY

SELECT item_id,
       STRING_AGG(tag, ', ' ORDER BY weight DESC) AS tag_list
FROM tag
WHERE tag IS NOT NULL
GROUP BY item_id
ORDER BY item_id;

STRING_AGG(tag, ', ' ...) concatenates the tag values of each group, joined by the delimiter ', '. The ORDER BY weight DESC sits inside the aggregate's parentheses — it controls the order in which values are stitched together. The WHERE tag IS NOT NULL removes NULL-tag rows before grouping.

3. The resulting analysis

item_idtag_listsignal
1green, red, blueordered by weight 5 > 3 > 1
2big, fastNULL-tag row skipped — no empty slot

Item 1's list is green, red, blue — weight order, not alphabetical, not insertion order. Item 2 is big, fast: the NULL-tag row was filtered, so the string has no dangling delimiter. Item 3 is absent entirely — the WHERE removed its only row, so it formed no group.

4. Strategic explanations

The ORDER BY inside the aggregate is the whole trick

Without an in-aggregate ORDER BY, STRING_AGG concatenates values in an unspecified order — whatever the executor happens to produce, which can vary run to run. The clause STRING_AGG(tag, ', ' ORDER BY weight DESC) makes the output deterministic and meaningful: heaviest tag first. This is a distinct, lesser-known piece of syntax — the ORDER BY lives between the aggregate's arguments and its closing paren, not in the query's outer ORDER BY (which here only sorts the final rows by item_id).

Filtering NULLs in WHERE vs relying on the aggregate

STRING_AGG already skips NULL inputs — a NULL tag contributes neither a value nor a delimiter, so big, fast would come out clean even without the filter. But the explicit WHERE tag IS NOT NULL is still essential: it is what makes item 3 disappear. Without the filter, item 3 forms a group, STRING_AGG aggregates its single NULL row, and returns NULL as the tag_list — the item would appear with an empty list rather than being omitted. The question says skip NULL tags and the natural reading is "no tags, no row", so filter first.

Duplicate tags fold in — or do they?

STRING_AGG keeps every matching row, so a tag listed twice for an item appears twice in the string. If the requirement were a deduplicated tag set you would write STRING_AGG(DISTINCT tag, ', ') — but note a Postgres limitation: DISTINCT and an ORDER BY on a different expression cannot be combined, because the ordering key must then be part of the distinct set. Knowing that constraint, and pre-deduplicating in a CTE when you need both, is a senior detail.

Dialect notes — the function name changes everywhere

This aggregate has a different name in nearly every engine. PostgreSQL: STRING_AGG(expr, sep ORDER BY ...). MySQL: GROUP_CONCAT(expr ORDER BY ... SEPARATOR sep) — beware its default group_concat_max_len truncation. SQL Server (2017+): STRING_AGG(expr, sep) WITHIN GROUP (ORDER BY ...) — the ordering moves to a WITHIN GROUP clause. Oracle: LISTAGG(expr, sep) WITHIN GROUP (ORDER BY ...). SQLite: GROUP_CONCAT, but it cannot order the concatenation at all. The concept is universal; the spelling is not.

The MySQL truncation trap

If this ran on MySQL, GROUP_CONCAT would silently cut the string at 1024 bytes by default — a long tag list would be quietly truncated with no error. On a large catalog you would raise group_concat_max_len first. SQL Server's STRING_AGG caps at 8000 bytes unless the inputs are MAX types. Calling out these silent limits unprompted is the senior move.

Senior signal: use STRING_AGG with its in-aggregate ORDER BY weight DESC for deterministic ordering, filter tag IS NOT NULL in WHERE so an all-NULL item drops out instead of returning an empty list, and know the function's name and ordering syntax differ across MySQL/SQL Server/Oracle plus the silent length-truncation traps.
