Data warehouses that ingest product catalogs from multiple upstream systems routinely accumulate duplicate records for the same SKU — differing only in timestamps or minor field variations. The task is classic deduplication: "For each (sku, product_name) combination, keep only the most recently updated record and discard older duplicates." This tests the ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ... DESC) pattern, which is the industry-standard idiomatic deduplication approach in SQL.

SKU-A / Widget Pro has three versions; the most recent (updated_at = 2025-04-01) should survive. SKU-B / Gadget Lite has two rows with identical timestamps — the edge case where ROW_NUMBER breaks the tie arbitrarily. SKU-C / Mega Bolt has one row and should pass through unchanged.

2. The SQL — ROW_NUMBER deduplication pattern

SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY sku, product_name ORDER BY updated_at DESC) AS rn
  FROM products
) t
WHERE rn = 1;

Three rows survive. Product_id 3 wins for SKU-A because it has the latest updated_at. For SKU-B the tie means either row 4 or row 5 can be rank 1 — the result is non-deterministic without a tiebreaker. SKU-C passes through unchanged. Note: the rn column appears in SELECT * because the outer SELECT is SELECT * from the subquery that includes rn; in practice, callers would list columns explicitly to exclude it.

4. Strategic explanations

The ROW_NUMBER deduplication pattern — and its guarantee

ROW_NUMBER() OVER (PARTITION BY dedup_keys ORDER BY recency DESC) assigns rank 1 to exactly one row per partition regardless of ties. This is the right tool when the business rule is "give me one row, I don't care which when there's a tie." When a tie needs to be resolved (e.g. pick the row with the lowest product_id as a stable tiebreaker), add a secondary ORDER BY: ORDER BY updated_at DESC, product_id ASC. Always ask the interviewer whether tie-breaking matters; the answer changes the ORDER BY.

Partition key choice: why (sku, product_name)

The schema carries both sku and product_name as the deduplication identity. Using only sku could incorrectly collapse rows for the same SKU assigned to different products (a data-quality scenario). Including both columns makes the identity more specific. In practice, the analyst should verify with the data owner whether the natural key is sku alone, product_name alone, or the composite — this question surfaces that intent.

Snowflake's QUALIFY shorthand

Snowflake supports the QUALIFY clause that filters on window function results without a subquery wrapper:

SELECT * FROM products
QUALIFY ROW_NUMBER() OVER (PARTITION BY sku, product_name ORDER BY updated_at DESC) = 1;

This is the idiomatic Snowflake deduplication pattern — one fewer level of nesting, same semantics. Postgres requires the subquery or CTE form.

SELECT * in deduplication queries — the rn column leak

The outer SELECT * includes the synthetic rn column in the result. For an intermediate CTE this is harmless, but inserting into a target table or exposing via a view will include rn. Best practice: list columns explicitly in the outer SELECT, or use SELECT * EXCLUDE (rn) (Snowflake / DuckDB syntax) to drop it cleanly.

Portability notes
  • Snowflake: prefer QUALIFY; SELECT * EXCLUDE (rn) available to drop the synthetic column.
  • Postgres: subquery or CTE; list columns explicitly to omit rn.
  • BigQuery: supports QUALIFY; also supports EXCEPT(rn) in the outer SELECT.
  • MySQL: no QUALIFY; subquery form required.
Senior signal: call out the tie-breaking risk with ROW_NUMBER when updated_at values are identical, add product_id ASC as a stable secondary sort, name the QUALIFY shorthand in Snowflake, and know that SELECT * leaks the synthetic rn column — use explicit column lists or EXCLUDE in Snowflake to avoid it.
