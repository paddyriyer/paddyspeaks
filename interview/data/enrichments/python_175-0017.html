Google's Search Quality team computes a rolling-median latency over the last k requests to monitor tail behaviour without letting one slow query distort the signal the way a mean would. "Sliding Window Median" tests whether you can keep a window sorted as it slides — an O(log k) insert/delete problem dressed up as an array question. The naive re-sort is the trap the interviewer is watching for.

1. The input data

nums = [1, 3, -1, -3, 5, 3, 6, 7]
k    = 3

Seven windows of width 3 slide across eight values. Negative numbers are deliberate — they catch off-by-one errors in any homemade comparison logic.

2. The approach

A SortedList from sortedcontainers holds the current window in sorted order, backed by a list of small lists so add and remove are both O(log k) plus a localized memmove. The window is seeded with nums[:k].

The closure med() reads the middle without scanning: for odd k both k//2 and (k-1)//2 point at the same central index; for even k they straddle the two middle elements and the average is the true median. Each slide does exactly two mutations — add(nums[i]) brings in the new element, remove(nums[i-k]) evicts the one that fell off the left — then records a fresh median. The window size never changes, so the index arithmetic stays valid throughout.

Every result is a float because med() uses /, even when the window length is odd.

4. Strategic explanations

Complexity

O(n log k) time — each of the n−k slides does two O(log k) operations on the structure. O(k) space for the window. The naive "re-sort every window" alternative is O(n·k log k) and is the answer the interviewer wants you to reject out loud.

The idiomatic alternative — two heaps

Without sortedcontainers, the textbook solution is a max-heap (lower half) plus a min-heap (upper half) kept balanced in size. The catch is lazy deletion: heaps cannot remove an arbitrary expiring element, so you stamp it for removal in a dict and discard it only when it surfaces at a heap top. That is far more code — naming SortedList as the pragmatic choice while knowing the two-heap fallback is the senior move.

The edge case — even k and integer overflow of precision

With even k, the median of two large odd numbers is a genuine .5 value — returning int here is wrong. Also (a + b) / 2 can lose precision for very large ints in some languages; Python's arbitrary-precision ints keep the sum exact, so the float division is the only rounding step. A window of k == 1 still works: both indices are 0.

The follow-up — streaming, unbounded window

"What if k is the whole stream so far?" Then it is the classic Find Median from Data Stream: two heaps, no eviction, O(log n) per insert. The interviewer is checking you can drop the fixed-window assumption cleanly.

The trap — list.remove on a plain list

Keeping a plain Python list and calling list.remove(nums[i-k]) is O(k) per slide for the linear scan and the shift — quietly O(n·k). Worse, if the window holds duplicates, list.remove deletes the first match, which is still correct for a value-based eviction but masks the performance bug. SortedList.remove is O(log k) and equally duplicate-safe.

Senior signal: say "this is an ordered-multiset problem" within the first sentence, reach for SortedList for clean O(n log k), and articulate why two heaps need lazy deletion rather than hand-waving the eviction.
