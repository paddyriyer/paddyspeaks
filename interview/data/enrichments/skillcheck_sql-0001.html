A compensation analyst is auditing pay bands and needs the runner-up salary inside each department — the second-best paid figure, by team. It sounds like a trivial "max but not quite", yet the interviewer is really probing one decision: when the top salary is shared by several people, does "second highest" mean the second-distinct value, or the third physical row? The choice of ranking function is the whole answer.

The edge case lives in department 2: Dan and Finn are tied at 80, the highest salary on that team. There is no distinct runner-up below the tie except Eve at 75 — so the question becomes whether 75 surfaces as the "second highest" or whether the tie swallows the slot.

2. The SQL — DENSE_RANK in a CTE

WITH ranked AS (
  SELECT
    dept_id,
    salary,
    DENSE_RANK() OVER (PARTITION BY dept_id ORDER BY salary DESC) AS rk
  FROM employee
)
SELECT DISTINCT dept_id, salary AS second_highest_salary
FROM ranked
WHERE rk = 2
ORDER BY dept_id;

DENSE_RANK assigns a rank per dept_id partition, ordering salary high-to-low. Because it is dense, tied salaries share a rank and the next distinct salary gets the very next integer — no gap. Filtering rk = 2 then picks the second-distinct salary in each department; DISTINCT collapses any duplicate rows that share that second value.

3. The resulting analysis

Department 2 is the payoff. Dan and Finn both sit at rk = 1 because DENSE_RANK gives ties the same rank; Eve's 75 becomes rk = 2, so it surfaces correctly as the second highest. Had the query used RANK, Eve would have been rk = 3 (rank 2 skipped by the two-way tie) and department 2 would have wrongly returned nothing.

4. Strategic explanations

Why DENSE_RANK and not RANK — the question says so for a reason

RANK leaves gaps after ties: two values tied at rank 1 push the next value to rank 3, so WHERE rk = 2 matches no rows and a tied-at-top department silently drops out of the result. DENSE_RANK never skips, so rank 2 always exists if at least two distinct salaries do. The prompt explicitly calls for DENSE_RANK precisely so ties at the top still surface a real 2nd-place figure — reading that requirement and explaining it back is the easy win.

Why ROW_NUMBER would be wrong

ROW_NUMBER hands every row a unique number, so in department 2 the two 80s become rows 1 and 2 — and rk = 2 would return 80, a top salary masquerading as the runner-up. ROW_NUMBER answers "give me one arbitrary row per position"; it cannot express "second distinct value" because it does not know two rows are equal. For a salary-band question, that distinction is the difference between right and wrong.

The PARTITION BY does the per-department work

PARTITION BY dept_id restarts the ranking for each department, so rank 1 means "top within this team", not "top in the company". Drop the partition and you would get a single global ranking — the second-highest salary overall, not per department. The partition clause is what turns one ranking into many independent ones, exactly matching "per department".

Edge cases: single-member and uniform departments

If a department has only one employee, no row ever reaches rk = 2, so that department is absent — correct, there is no second salary. The same happens if every employee in a department earns the identical salary: only rk = 1 exists. If the business wants those departments listed with a NULL second salary, you would need a LEFT JOIN from a distinct-department list onto this result — worth naming, because "missing rows" can otherwise look like a bug.

The DISTINCT is load-bearing

If two people share the second-highest salary, WHERE rk = 2 returns two rows — same dept_id, same salary — and the output would duplicate the department. SELECT DISTINCT folds them into one. An equivalent and arguably cleaner shape is to aggregate (MAX(salary) ... WHERE rk = 2 GROUP BY dept_id), which also makes the intent "one row per department" explicit.

Senior signal: pick DENSE_RANK deliberately so a tie at the top still yields a real rank 2, explain why RANK would drop tied-top departments and ROW_NUMBER would return a top salary as the runner-up, use PARTITION BY dept_id to rank per team, and keep the DISTINCT (or aggregate) so a shared second salary does not duplicate a department.
