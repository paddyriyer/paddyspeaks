The classic Netflix interview phrasing: "find every user who has watched on three or more consecutive calendar days." The answer is the canonical gaps-and-islands trick — view_date - row_number = constant within a streak. This is the per-day binge metric (vs the per-session "3 episodes in 24h" metric); both are real Netflix metrics for different teams.

2. The SQL — the 3-CTE pattern

WITH daily_views AS (
  -- Step 1: dedupe — a user can watch 10 shows in one day
  SELECT user_id, view_date
  FROM user_views
  GROUP BY user_id, view_date
),
islands AS (
  -- Step 2: the trick — date minus row-number is constant within a streak
  SELECT user_id, view_date,
    view_date - (ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY view_date) * INTERVAL '1 day') AS island_group
  FROM daily_views
),
streak_counts AS (
  -- Step 3: aggregate per island
  SELECT user_id,
    MIN(view_date) AS streak_start,
    MAX(view_date) AS streak_end,
    COUNT(*)       AS consecutive_days
  FROM islands
  GROUP BY user_id, island_group
)
SELECT *
FROM streak_counts
WHERE consecutive_days >= 3
ORDER BY consecutive_days DESC;

3. Strategic explanations

The DEDUP step is the silent-bug killer

Without the daily_views CTE, a user who watches 5 shows on May 3 would create 5 rows for May 3, the row_number jumps by 5 in one day, and the island_id math collapses. Always dedupe to one row per (user, day) before applying the streak trick. This is the #1 mistake in interview answers.

The trick — why date - rn equals constant within a streak
view_daterow_numberview_date - rn (days)
May 11Apr 30
May 22Apr 30
May 33Apr 30
May 5 (gap!)4May 1
May 65May 1

Within a streak, both date and row_number tick by 1 each row — their difference is constant. The moment a date skips, the difference shifts. Distinct difference = distinct streak.

The * INTERVAL '1 day' idiom

In Postgres, date - integer works directly: '2026-05-01'::DATE - 3 = '2026-04-28'. But in some dialects you need to multiply by an interval to get type-correct subtraction. The original solution uses the explicit interval form for cross-dialect safety. Equivalent forms:

-- Postgres / Snowflake / DuckDB
view_date - ROW_NUMBER() OVER (...)::INT

-- BigQuery — DATE_SUB
DATE_SUB(view_date, INTERVAL CAST(ROW_NUMBER() OVER (...) AS INT64) DAY)

-- SQLite — julianday math
julianday(view_date) - ROW_NUMBER() OVER (...)

-- MySQL — TO_DAYS
TO_DAYS(view_date) - ROW_NUMBER() OVER (...)
The interpretation question — "any 3-day streak" vs "longest streak ≥ 3"

The query above returns every qualifying streak. A user with two 3-day streaks shows up twice. Three plausible interpretations:

  • "Users who have any 3-day streak": SELECT DISTINCT user_id FROM streak_counts WHERE consecutive_days >= 3
  • "Users whose longest streak is ≥ 3": aggregate first — SELECT user_id, MAX(consecutive_days) AS longest FROM streak_counts GROUP BY user_id HAVING longest >= 3
  • "Each qualifying streak with start/end dates" (the original): for downstream campaign targeting, useful as-is

Always pin the interpretation with the requester before writing.

The lifecycle layer on top — what happens AFTER the streak?

The truly senior version (matching the customer_growth-0020 question) doesn't stop at "they had a streak". It computes the exit pattern from each streak:

WITH /* ... streak_counts ... */
next_activity AS (
  SELECT s.user_id, s.streak_end,
    MIN(d.view_date) AS next_date
  FROM streak_counts s
  LEFT JOIN daily_views d
    ON d.user_id = s.user_id AND d.view_date > s.streak_end
  GROUP BY s.user_id, s.streak_end
)
SELECT user_id, streak_start, streak_end, consecutive_days,
  CASE
    WHEN next_date - streak_end <= 1   THEN 'continuing'
    WHEN next_date - streak_end <= 28  THEN 'engaged'
    WHEN next_date - streak_end <= 90  THEN 'resurrected'
    WHEN next_date IS NULL AND CURRENT_DATE - streak_end > 90 THEN 'churned'
    WHEN next_date IS NULL                 THEN 'open'
    ELSE 'long_resurrect'
  END AS exit_status
FROM streak_counts s LEFT JOIN next_activity USING (user_id, streak_end);

Now the streak isn't an endpoint — it's the input to a retention CASE that drives the winback campaign.

The tolerant-streak follow-up

"3 consecutive days" is strict. Real-world tax-software / fitness-app / news-app analytics often allow a single missed day. Use LAG:

SELECT user_id, view_date,
  CASE WHEN LAG(view_date) OVER (PARTITION BY user_id ORDER BY view_date)
            >= view_date - INTERVAL '2 days' THEN 0 ELSE 1 END AS new_streak
FROM daily_views;

Cumulative-sum the new_streak flag = streak_id. Replace 2 days with the tolerated gap.

Index that makes this query fast

Composite index on (user_id, view_date) lets the planner do an index-only sort within partitions. Without it, the ROW_NUMBER triggers a full table sort — for 100M+ rows the difference is hours vs minutes.

Senior signal: never skip the dedupe step, articulate the date-rn-constant trick from first principles, contrast per-day-streak with per-session-binge as different real metrics, and pre-empt the interpretation ambiguity ("any streak" vs "longest streak"). Layering the lifecycle CASE on top to expose churn / resurrect status is the over-the-bar finish.
