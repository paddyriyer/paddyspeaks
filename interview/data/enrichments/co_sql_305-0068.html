Lyft's program-management office wants a daily alert listing every project that has gone off the rails — over budget, or delivered past its deadline. The schema is messy in a very real way: deadline is stored as free text, not a proper date, and some rows carry junk like "TBD". An interviewer here is testing whether you can write an OR condition across two failure modes and defend a query against a column that cannot be trusted to be a date.

The traps are deliberate. Project 1 is over budget. Project 2 is on budget but shipped late. Project 4 has a non-date deadline ("TBD") that would throw a cast error if you fed it to ::date blindly. Project 5 is wildly over budget but has a NULL end_date.

2. The SQL — regex-guarded date cast

SELECT project_id FROM projects
WHERE actual_spend > budget
   OR (deadline ~ '^\d{4}-\d{2}-\d{2}' AND end_date > deadline::date);

Two failure conditions joined by OR: spend exceeds budget, or the project ended after its deadline. The deadline branch is gated by a regex test — deadline ~ '^\d{4}-\d{2}-\d{2}' — so the ::date cast only runs on strings that actually look like a date.

4. Strategic explanations

Why the regex guard is mandatory

deadline::date on the string "TBD" raises invalid input syntax for type date — and in Postgres that aborts the whole query, not just one row. The regex ~ '^\d{4}-\d{2}-\d{2}' filters to ISO-shaped strings first. Crucially, AND short-circuits: when the left side is false, Postgres never evaluates end_date > deadline::date. Order matters — put the cheap, protective test first.

Short-circuiting is not guaranteed — the subtle version

SQL's AND short-circuits in practice, but the standard does not promise evaluation order, and the planner can reorder predicates. The bulletproof pattern is a safe cast that returns NULL instead of erroring:

-- Postgres 16+: never throws, bad strings become NULL
WHERE actual_spend > budget
   OR end_date > to_date(deadline, 'YYYY-MM-DD')   -- still errors on 'TBD' in older PG
-- Robust everywhere: a CASE that gates the cast
WHERE actual_spend > budget
   OR end_date > (CASE WHEN deadline ~ '^\d{4}-\d{2}-\d{2}$'
                       THEN deadline::date END)

Wrapping the cast in a CASE that yields NULL for non-dates is the most defensible form — a NULL comparison is simply false, so the row is excluded cleanly.

The NULL end_date — in-flight projects

Project 5 has no end_date because it hasn't finished. end_date > deadline::date is NULL (unknown) for that row, so the deadline branch can never flag it — which is correct: a project that hasn't ended cannot be "late" yet. It still gets caught by the budget branch. But note the asymmetry: an unfinished project that is already past its deadline is arguably an alert too. If so, add OR (end_date IS NULL AND deadline::date < CURRENT_DATE).

The OR-defeats-the-index trap

An OR across two different columns usually prevents the planner from using an index on either — it falls back to a full scan, or a bitmap OR of two index scans if both columns are indexed. On a large projects table, a UNION of two single-condition queries can be faster because each arm uses its own index:

SELECT project_id FROM projects WHERE actual_spend > budget
UNION
SELECT project_id FROM projects
WHERE deadline ~ '^\d{4}-\d{2}-\d{2}' AND end_date > deadline::date;
The real fix is the schema

The honest senior comment: deadline should be a date column with a CHECK or NOT NULL constraint, not free text. Regex-guarding every read is a tax paid forever. Flag it — "this query works, but the data model is forcing defensive code; the deadline column wants a type."

Senior signal: recognise that casting an untrusted text column can abort the entire query, gate the cast with a regex or a NULL-returning CASE rather than relying on undefined short-circuit order, reason correctly about the NULL end_date meaning "not finished", and name the schema fix instead of just coding around it.
