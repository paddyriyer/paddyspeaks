Atlassian's education team runs an online course catalog and wants a popularity leaderboard — courses ranked by enrollment count. The query counts enrollments per course and applies a ranking window over that count. It works, but the interviewer is watching for two things: how the rank behaves when two courses are tied, and the subtle fact that you can rank by an aggregate you also computed.

Course 1 has 3 enrollments, courses 2 and 3 have 2 each — a tie — and course 4 has 1. That tie at the second position is the whole interview: it decides what rank course 4 receives.

2. The SQL — RANK() OVER (ORDER BY COUNT(*) DESC)

SELECT course_id, COUNT(*) AS enrolls,
  RANK() OVER (ORDER BY COUNT(*) DESC) AS rk
FROM enrollments
GROUP BY course_id;

The GROUP BY reduces the table to one row per course with its enrollment count. The window function then ranks those grouped rows — and crucially, the window can reference COUNT(*) directly because window functions execute after grouping.

4. Strategic explanations

Window functions can see aggregates — the execution-order point

It surprises many candidates that RANK() OVER (ORDER BY COUNT(*) DESC) is legal. SQL evaluates GROUP BY and aggregates before window functions, so by the time the window runs, COUNT(*) is already a settled per-group value. You do not need a subquery or CTE to "materialise the count first" — though wrapping it in a CTE is fine for readability:

WITH counts AS (
  SELECT course_id, COUNT(*) AS enrolls FROM enrollments GROUP BY course_id
)
SELECT course_id, enrolls, RANK() OVER (ORDER BY enrolls DESC) AS rk
FROM counts;

RANK leaves a gap after a tie (1, 2, 2, 4). DENSE_RANK does not (1, 2, 2, 3). ROW_NUMBER ignores ties entirely and would give courses 2 and 3 different numbers, breaking the tie arbitrarily by physical row order — almost never right for a leaderboard. Name which semantics the product wants.

The "top N courses" follow-up — and the LIMIT trap

"Show the top 3" is the natural next ask. LIMIT 3 after the ranking would cut off mid-tie — with our data it returns courses 1, 2, 3 but arbitrarily excludes nothing problematic here, yet on other data it could keep one tied course and drop its twin. The robust answer filters on the rank:

SELECT * FROM (
  SELECT course_id, COUNT(*) AS enrolls,
         RANK() OVER (ORDER BY COUNT(*) DESC) AS rk
  FROM enrollments GROUP BY course_id
) t
WHERE rk <= 3;

This keeps all courses tied at rank 3, which is the honest meaning of "top 3".

Counting distinct students and ordering determinism

COUNT(*) counts enrollment rows. If a student could enroll in the same course twice (a re-take), COUNT(*) would overstate popularity — use COUNT(DISTINCT student_id) if "popular" means unique students. Also, the outer query has no ORDER BY, so the display order of rows is not guaranteed even though the rk values are; add ORDER BY rk for a stable leaderboard.

Senior signal: know that window functions run after aggregation so RANK() OVER (ORDER BY COUNT(*)) needs no subquery, contrast RANK/DENSE_RANK/ROW_NUMBER by tie behaviour, filter on rk <= N rather than LIMIT so ties aren't split, and use COUNT(DISTINCT student_id) if re-enrollments would inflate the count.
