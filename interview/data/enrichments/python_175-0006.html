Apple's interviewers use this to see whether you understand what a database actually does under a JOIN. Anyone can write SELECT ... JOIN ... in SQL; far fewer can explain why the engine usually picks a hash join over a nested loop, and rebuild it in twenty lines of Python. The question tests data-structure instinct: turning an O(n×m) cross-comparison into O(n+m) by trading memory for a hash index.

1. The input data

Two lists of dicts — orders on the left, customers on the right, joined on customer id:

left  = [{'order': 'A1', 'cust': 7},
         {'order': 'A2', 'cust': 9},
         {'order': 'A3', 'cust': 7}]

right = [{'cust': 7, 'name': 'Maya'},
         {'cust': 8, 'name': 'Devin'}]

hash_join(left, right, 'cust', 'cust')

2. The approach

This is the textbook build / probe hash join. Build phase: iterate the right side once and bucket every row into a defaultdict(list) keyed by its join value — a list, not a single value, because a key can repeat. Probe phase: iterate the left side; for each left row, do an O(1) dictionary lookup (idx.get(l[lkey], [])) to fetch all matching right rows, and emit a merged dict per match using {**l, **r}. The defaultdict avoids "key missing" branching during build, and .get(..., []) makes a left row with no match contribute nothing instead of raising. The result is an inner join: only pairs that match appear.

3. The output

Customer 7 (Maya) matches orders A1 and A3; order A2's customer 9 has no row in right, so it is dropped:

[{'order': 'A1', 'cust': 7, 'name': 'Maya'},
 {'order': 'A3', 'cust': 7, 'name': 'Maya'}]

4. Strategic explanations

Complexity — why this beats the nested loop

Build is O(m), probe is O(n) lookups plus O(matches) emits, so total O(n + m + matches) time and O(m) extra space for the index. The naive double loop is O(n×m). On a 100k × 100k join that is the difference between instant and ten billion comparisons — exactly why query planners build a hash table on the smaller relation.

The idiomatic detail — build the smaller side

The code always indexes right. A senior would note the index should be built on whichever input is smaller, to minimize the hash table's memory footprint, and probe with the larger one — the lookups are O(1) either way.

The edge case — key collision in the merge

{**l, **r} merges left then right; if both dicts share a non-join key (say both have 'name'), the right value silently wins. Real databases prefix columns with the table name to disambiguate. Also note the join keys here happen to share the name 'cust', so the merged dict keeps one cust field cleanly; differing key names would leave both.

The follow-up — make it a LEFT join

"Now keep unmatched orders too." Change the probe: if idx.get(l[lkey]) is empty, still emit the left row, padding the right side's columns with None. The build phase is untouched — left/right/full outer joins are all variations on what the probe does with a miss.

The trap — storing one row per key

Writing idx[r[rkey]] = r instead of appending to a list looks simpler but is wrong: when the right side has duplicate keys, each overwrites the last, so matches are lost. Join keys are not unique in general — the defaultdict(list) is mandatory, not stylistic.

Senior signal: a senior names this as the engine's hash-join strategy, indexes the smaller relation, uses a list-valued bucket because keys repeat, and flags the silent key-collision in {**l, **r} — a junior writes a nested loop or maps each key to a single row.
