A Community member posts a year-over-year growth puzzle: the sales table holds one row per customer per year, and the task is to surface each customer whose sales in some year met or beat the year before. It reads like a simple comparison, but the reference solution hides a quiet bug — what happens to a customer whose sales fell every single year? The interviewer is watching whether you notice that row before it ships.

Four customers, four trajectories. Alice grew then dipped; Bob dipped then recovered; Carol's 2024 exactly ties 2023 (the >= edge); and Dave declines every year — the row that breaks the reference query.

2. The SQL — pivot then compare

WITH pivot_result AS (
  SELECT Customer_Name,
         SUM(CASE WHEN Year = 2022 THEN Sales_Amount END) AS y2022,
         SUM(CASE WHEN Year = 2023 THEN Sales_Amount END) AS y2023,
         SUM(CASE WHEN Year = 2024 THEN Sales_Amount END) AS y2024
  FROM sales
  GROUP BY Customer_Name
)
SELECT Customer_Name,
       CASE WHEN y2023 >= y2022 THEN y2023
            WHEN y2024 >= y2023 THEN y2024 END AS Sales,
       CASE WHEN y2023 >= y2022 THEN '2023'
            WHEN y2024 >= y2023 THEN '2024' END AS Yr,
       CASE WHEN y2023 >= y2022 THEN y2022
            WHEN y2024 >= y2023 THEN y2023 END AS Prev_Yr_Sales
FROM pivot_result
WHERE y2023 >= y2022 OR y2024 >= y2023
ORDER BY Customer_Name;

The CTE pivots the long table into one row per customer with a column per year. The outer CASE ladder picks the first year that beat its predecessor. The WHERE clause — the fix — drops customers who never qualified.

4. Strategic explanations

The bug: the original solution emits a phantom NULL row

Without the WHERE clause, Dave still produces a row — the CASE ladder runs out of branches, so all three CASE expressions return NULL (a CASE with no matching WHEN and no ELSE yields NULL). The query then reports Dave | NULL | NULL | NULL, which means "Dave qualified, with no data" — nonsense. The query should only list customers who actually met the condition, so a WHERE y2023 >= y2022 OR y2024 >= y2023 filter is mandatory.

The CASE ladder picks the FIRST qualifying year, not all of them

A subtle product question hides here. Alice grew in 2023 and her 2024 (900) is below 2023, so she has exactly one qualifying year. But suppose a customer grew in both 2023 and 2024 — the ladder reports only 2023, because CASE stops at the first true WHEN. If the business wants every qualifying year, the pivot model is the wrong shape; you want a self-comparison on the long table instead.

The scalable rewrite — LAG instead of a hand-pivot

Hard-coding y2022/y2023/y2024 means editing the query every January. A window function compares each year to the previous one with no year literals at all:

SELECT Customer_Name, Year, Sales_Amount,
       LAG(Sales_Amount) OVER (PARTITION BY Customer_Name
                               ORDER BY Year) AS prev_yr_sales
FROM sales
QUALIFY Sales_Amount >= prev_yr_sales;   -- Snowflake/BigQuery
-- Postgres: wrap in a CTE and filter
-- WHERE Sales_Amount >= prev_yr_sales

This returns one row per qualifying customer-year, generalises to any number of years, and never breaks when 2025 data arrives. QUALIFY is Snowflake/BigQuery sugar; in Postgres you push the comparison into an outer WHERE over the CTE.

The first-year NULL trap

LAG returns NULL for each customer's earliest year — there is no prior year to compare. Sales_Amount >= NULL evaluates to unknown, so that row is correctly excluded by WHERE. Good. But if someone writes NOT (Sales_Amount < prev_yr_sales) expecting the opposite, they get a different (still unknown) result — three-valued logic, not boolean negation. Be deliberate about which side of the NULL you want.

Gaps in the year sequence

The pivot quietly assumes every customer has all three years. If a customer is missing 2023 entirely, y2023 is NULL, and both y2023 >= y2022 and y2024 >= y2023 become unknown — the customer drops out even if 2024 dwarfed 2022. LAG has the same blind spot: it compares to the previous row, not the previous calendar year. If gap years matter, join against a generated year spine first.

Senior signal: spot that the bare CASE ladder ships a meaningless all-NULL row for never-qualifying customers and add the guarding WHERE; then note that the hard-coded year pivot is unmaintainable and reach for LAG with an explicit year spine when gap years are possible.
