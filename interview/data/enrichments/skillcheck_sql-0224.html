This is the inverse of a pivot. You inherit a wide summary — one row per category, one column per sub-genre — and you need to melt it vertically into a tall ledger, so each category has multiple rows broken out by a generic sub_category label and a watch_hours value. In PostgreSQL the cleanest tool is CROSS JOIN LATERAL over an inline VALUES list.

1. The source data — category_watch_summary

main_categorysuperhero_hoursspy_hourscomedy_hoursdrama_series_hours
Film180601000
Series000160

Two rows, four metric columns — the exact shape a pivot produces. The goal is to turn those four columns back into rows.

2. The query — CROSS JOIN LATERAL over VALUES

SELECT
  w.main_category,
  u.sub_category,
  u.watch_hours
FROM category_watch_summary w
CROSS JOIN LATERAL (
  VALUES
    ('Superhero',    w.superhero_hours),
    ('Spy',          w.spy_hours),
    ('Comedy',       w.comedy_hours),
    ('Drama Series', w.drama_series_hours)
) AS u(sub_category, watch_hours)
WHERE u.watch_hours > 0
ORDER BY w.main_category, u.watch_hours DESC;

4. Strategic explanations

What LATERAL actually grants

A normal subquery in the FROM clause cannot see columns from the other tables in that same FROM. LATERAL lifts that restriction: the VALUES list is re-evaluated for each row of w, and it may reference w.superhero_hours, w.spy_hours and the rest. Without LATERAL, those references are an error.

The VALUES list is the column-to-row map

Each tuple ('Superhero', w.superhero_hours) pairs a label with the column it came from. AS u(sub_category, watch_hours) names the two output columns. The four tuples per source row are exactly the four columns being melted — 2 rows × 4 tuples = 8 candidate rows.

WHERE u.watch_hours > 0 — a reporting choice

Of the 8 candidate rows, four have a zero value (Film has no Drama Series; Series has zero of the other three). The WHERE drops them, leaving 4. Keeping or dropping zero cells is a deliberate decision — a dense ledger drops them; a complete matrix keeps them.

One pass over the table

CROSS JOIN LATERAL scans category_watch_summary exactly once and expands each row in place. The portable UNION ALL form (see the next question) scans the base table once per column. On a small summary that difference is invisible; on a wide or large table it is the reason to prefer LATERAL.

Dialect notes

CROSS JOIN LATERAL is PostgreSQL (and standard SQL). SQL Server expresses the same idea as CROSS APPLY; Oracle has a dedicated UNPIVOT operator; the universally portable fallback is UNION ALL. The VALUES table-constructor used here is itself a PostgreSQL strength.

Senior signal: to unpivot in PostgreSQL, CROSS JOIN LATERAL (VALUES (label, col), ...) — the column-to-row mapping is declared inline and the base table is scanned once. LATERAL is the keyword that lets the VALUES list reference the outer row.
