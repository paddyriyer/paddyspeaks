A double-entry ledger is the accounting invariant that every financial system must enforce: every transaction must produce ledger entries that sum to exactly zero across debits and credits. The interviewer is probing whether you understand the separation of transaction (the intent) from ledger entries (the execution), how to enforce the zero-sum invariant at the database layer rather than relying on application code, and how to handle the reversal pattern correctly (reversals are new transactions, not mutations to existing ones). This is a Hard-difficulty question because the constraint trigger and deferral mechanics are non-trivial.

1. The requirements — what the design must capture

  • Account types: asset, liability, revenue, expense, equity — the classic accounting equation; the type determines the sign convention in reports.
  • Transaction as a unit of work: every business event (a transfer, a fee, a payout) is one transaction; it groups two or more ledger entries that must net to zero.
  • Ledger entry immutability: once posted, entries are never updated or deleted; corrections are new reversing transactions.
  • Zero-sum enforcement: a deferred constraint trigger fires after all entries for a transaction are inserted, then verifies the balance; if not zero, the entire transaction is rolled back.
  • Reversal pattern: a reversed transaction is corrected by a new transaction whose entries are the mirror image (debits become credits) of the original; the original is flagged as reversed via reversed_by_txn_id.
  • Idempotency: transaction.external_ref is a UNIQUE idempotency key from the upstream payment system; re-processing the same event is a no-op.

2. The DDL — the CREATE TABLE

CREATE TABLE account (
  account_id        BIGSERIAL    PRIMARY KEY,
  external_ref      VARCHAR(64)  NOT NULL UNIQUE,             -- e.g., user-wallet ID or system-account name
  account_type      VARCHAR(16)  NOT NULL,                    -- 'asset','liability','revenue','expense','equity'
  currency          CHAR(3)      NOT NULL,
  is_active         BOOLEAN      NOT NULL DEFAULT TRUE,
  created_at        TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE TABLE transaction (
  transaction_id    BIGSERIAL    PRIMARY KEY,
  external_ref      VARCHAR(64)  NOT NULL UNIQUE,             -- idempotency key from upstream
  description       TEXT,
  occurred_at       TIMESTAMPTZ  NOT NULL,                    -- business time
  recorded_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),      -- system time
  status            VARCHAR(16)  NOT NULL DEFAULT 'posted',   -- 'posted','reversed'
  reversed_by_txn_id BIGINT      REFERENCES transaction(transaction_id)
);

CREATE TABLE ledger_entry (
  entry_id          BIGSERIAL    PRIMARY KEY,
  transaction_id    BIGINT       NOT NULL REFERENCES transaction(transaction_id) ON DELETE RESTRICT,
  account_id        BIGINT       NOT NULL REFERENCES account(account_id),
  direction         CHAR(1)      NOT NULL CHECK (direction IN ('D','C')),
  amount_cents      BIGINT       NOT NULL CHECK (amount_cents > 0),
  currency          CHAR(3)      NOT NULL,
  created_at        TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX ix_ledger_entry_txn ON ledger_entry (transaction_id);
CREATE INDEX ix_ledger_entry_account_time ON ledger_entry (account_id, created_at);

-- Enforce zero-sum per transaction using a deferred constraint trigger.
CREATE OR REPLACE FUNCTION assert_ledger_balanced() RETURNS trigger AS $$
DECLARE
  imbalance NUMERIC;
BEGIN
  SELECT
    SUM(CASE WHEN direction = 'D' THEN amount_cents ELSE -amount_cents END)
    INTO imbalance
  FROM ledger_entry
  WHERE transaction_id = NEW.transaction_id;
  IF imbalance IS NULL OR imbalance <> 0 THEN
    RAISE EXCEPTION 'Ledger entries for transaction % do not balance: %', NEW.transaction_id, imbalance;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE CONSTRAINT TRIGGER trg_ledger_balanced
  AFTER INSERT OR UPDATE OR DELETE ON ledger_entry
  DEFERRABLE INITIALLY DEFERRED
  FOR EACH ROW EXECUTE FUNCTION assert_ledger_balanced();

-- Reversal pattern: insert a NEW transaction (status='posted') whose entries are the inverse of the
-- original (debits become credits and vice versa), and set the original.status = 'reversed' and
-- original.reversed_by_txn_id = new_txn_id.

4. Strategic explanations

Deferred constraint trigger — why deferred

A double-entry transaction requires inserting at least two ledger entries. If the trigger fired INITIALLY IMMEDIATE, the first insert would always fail the balance check (one debit with no matching credit yet sums to a non-zero value). DEFERRABLE INITIALLY DEFERRED defers the trigger check to the end of the transaction, by which point all entries have been inserted and the balance can be verified as a whole. The entire transaction is rolled back atomically if the check fails.

Why amounts are always positive with a direction flag

Storing negative amount_cents for credits is a tempting shortcut but causes silent bugs: a developer might pass a negative amount for a debit, creating an entry that appears to be a very large credit. The CHECK (amount_cents > 0) + direction IN ('D','C') pattern makes the sign explicit and computable: SUM(CASE WHEN direction = 'D' THEN amount_cents ELSE -amount_cents END) produces the signed balance. This is the accounting convention.

Reversal, not mutation

Correcting a posted transaction by UPDATEing or DELETEing its ledger entries would destroy the audit trail. The correct pattern is to insert a new reversal transaction whose entries are the mirror of the original (every D becomes a C and vice versa, same amounts). This preserves every historical row intact. The original transaction is marked status = 'reversed' and linked to the reversal transaction via reversed_by_txn_id. Balance queries should exclude reversed transactions (or include both the original and its reversal, which cancel out).

Computing account balances

A point-in-time balance for an account is: SELECT SUM(CASE WHEN direction = 'D' THEN amount_cents ELSE -amount_cents END) FROM ledger_entry JOIN transaction USING (transaction_id) WHERE account_id = ? AND transaction.status = 'posted' AND occurred_at <= ?. For performance, materialize running balances in a account_balance_snapshot table (updated after each transaction) and compute incremental deltas from the snapshot forward. This avoids a full ledger scan for every balance query.

Dialect notes

Constraint triggers (CREATE CONSTRAINT TRIGGER) are Postgres-specific. In MySQL, enforce the invariant in a stored procedure or at the application layer within a transaction. In Oracle, deferred constraints on triggers are supported. In Snowflake, there are no triggers; the zero-sum invariant must be enforced by the ingestion service before committing rows. In all non-Postgres dialects, this is a strong argument for having a dedicated ledger service that owns the invariant.

Senior signal: use a deferred constraint trigger (not an immediate one) to allow multi-entry inserts within a single transaction before the balance check fires; enforce amount_cents > 0 and capture sign in a direction flag to prevent double-sign bugs; and treat every reversal as a new transaction — never mutate posted entries — so the complete audit trail is always reconstructible from the ledger alone.
