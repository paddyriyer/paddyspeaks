Netflix's account model is unusual: one billing account, multiple profiles (Mom, Dad, Kid 1, Kid 2). The household-data team needs to know which profile is the dominant viewer for each account, plus how the watch-time splits across profiles. The output drives the "primary profile" detection used in churn modelling, recommendations bias, and household-vs-sharing classification. The query is a window aggregate of an aggregate.

2. The SQL — per-profile total + share% + rank

SELECT account_id, profile_id,
  SUM(watch_minutes) AS profile_minutes,
  ROUND(100.0 * SUM(watch_minutes) /
    SUM(SUM(watch_minutes)) OVER (PARTITION BY account_id), 1) AS share_pct,
  RANK() OVER (PARTITION BY account_id ORDER BY SUM(watch_minutes) DESC) AS profile_rank
FROM plays
GROUP BY account_id, profile_id
ORDER BY account_id, share_pct DESC;

4. Strategic explanations

The double-SUM(SUM()) pattern

The inner SUM(watch_minutes) is the GROUP BY aggregate — per-profile total. The outer SUM(...) OVER (PARTITION BY account_id) is the window aggregate over the post-grouping rows — account-level total. Both in one query, no subquery needed. This is the same idiom as the Okta percent-contribution question, lifted to the household level.

Why RANK not DENSE_RANK here

For "primary profile" detection, ties are interesting on their own — they signal a 50/50 household where neither user dominates. RANK keeps the gap (1, 1, 3); DENSE_RANK collapses it (1, 1, 2). Both are fine for "is this profile #1?" but RANK preserves the "n-way tie" diagnostic. ROW_NUMBER would arbitrarily break ties, which is the wrong signal.

The single-profile household — the password-sharing red flag

An account where one profile has 100% share and the others have 0% might look like a single-user household — or it might be a shared password where the legit user only ever uses one profile and the squatters never bothered to switch. Cross-tab against device count:

WITH primary_share AS (
  SELECT account_id, MAX(share_pct) AS top_share
  FROM /* the original query */
  GROUP BY account_id
)
SELECT ps.account_id, ps.top_share, COUNT(DISTINCT s.device_id) AS distinct_devices
FROM primary_share ps JOIN streams s USING (account_id)
WHERE ps.top_share > 90 AND COUNT(DISTINCT s.device_id) > 5
-- 90%+ share but 5+ devices = single-profile-with-lots-of-devices = sharing risk
The "kids profile" classification

Netflix has explicit "kids" profiles with content filters. The watch-pattern signature (cartoon-heavy, Saturday morning peak, no after-9pm activity) is diagnostic. Add an is_kids flag to the GROUP BY for parental-controls coverage analysis.

The "bored secondary" flag

Profile 2 with 5% share might be a kid who watches every now and then — or might be a churning user who's losing interest. The diagnostic is recency:

SUM(watch_minutes)                                    AS lifetime_minutes,
SUM(CASE WHEN played_at >= CURRENT_DATE - 30 THEN watch_minutes END) AS l30_minutes,
SUM(CASE WHEN played_at >= CURRENT_DATE - 30 THEN watch_minutes END) * 1.0 /
  NULLIF(SUM(watch_minutes), 0)                       AS recent_share

A profile whose recent_share is much lower than lifetime_share is disengaging.

Account-level metrics — HHI as a household-concentration index

The Herfindahl-Hirschman Index from economics works perfectly for "how concentrated is this household's viewing?":

SUM(POWER(share_pct/100.0, 2)) OVER (PARTITION BY account_id) AS hhi
HHIshapeinterpretation
1.0one profile, 100%single user (or all-on-one-profile)
0.5two profiles 50/50balanced household
0.33three profiles 33/33/33family of three
0.25four profiles 25/25/25/25family of four

HHI is a smoother proxy than profile count for "household-ness".

Number-of-active-profiles — the cleaner top-line
SELECT account_id,
  COUNT(DISTINCT profile_id) AS total_profiles,
  COUNT(DISTINCT CASE WHEN profile_minutes > 0 THEN profile_id END) AS active_profiles,
  COUNT(DISTINCT CASE WHEN share_pct >= 10 THEN profile_id END) AS meaningful_profiles
FROM /* the original query */
GROUP BY account_id;

"Meaningful profiles" with the ≥ 10% share threshold filters out one-off curiosity profiles.

The retention follow-up — multi-profile households churn less

Empirically, accounts with 3+ active profiles churn at half the rate of single-profile accounts — the household network effect. The query above feeds that retention analysis directly.

The genre-overlap follow-up — do family members watch together?

Two profiles with 50/50 share and identical genre mix = probably the same person on two profiles (or two people with shared taste). Two profiles with 50/50 share and disjoint genre mix = a real two-viewer household. Compute Jaccard similarity over genre sets:

The window-aggregate dialect note

The double-SUM-over-window form needs the engine to evaluate the inner aggregate first. All major engines do this correctly, but if a query plan looks weird, fall back to the explicit CTE form — same answer, easier for the planner.

Senior signal: name the aggregate-of-aggregate window (SUM(SUM()) OVER) idiom in one breath, choose RANK over DENSE_RANK to preserve the n-way-tie household signal, and connect the share distribution back to concrete product decisions (password-sharing detection, churn risk, household HHI). The recent-share-vs-lifetime-share dis-engagement diagnostic and Jaccard genre-overlap follow-up are the over-the-bar finish.
