A lifecycle-marketing team wants to know which users confirmed their account by SMS exactly one calendar day after signing up — a clean "next-day activation" cohort. The data spans two tables and timestamps, not dates. The interviewer is probing two things: can you compare across a join on a derived date difference, and do you handle the "a user can get many texts" multiplicity without double-counting or missing a qualifying message?

1. The source data — signups and text_messages

user_idsignup_tstext (signup_ack, sent_ts)(narrative)
12025-05-01 10:00FALSE 05-01 12:00; TRUE 05-02 09:00confirms on day +1 — qualifies

Three traps. User 2 confirms at 23:59 the same day — under an hour-based diff that 23h59m is "less than a day," but the date difference is 0, so it must not qualify. User 3 has a non-confirmation and a confirmation on day +1: only the TRUE row counts. User 5 is a day late.

2. The SQL — join on the date difference

SELECT DISTINCT s.user_id
FROM signups s
JOIN text_messages t ON t.user_id = s.user_id
WHERE t.signup_ack = TRUE
  AND (t.sent_ts::date - s.signup_ts::date) = 1
ORDER BY s.user_id;

Casting both timestamps to ::date drops the time component, and subtracting two date values in PostgreSQL yields an integer number of days. The predicate = 1 keeps only confirmations whose calendar day is exactly one after the signup day. signup_ack = TRUE filters to genuine confirmations.

3. The resulting analysis

user_id(why it qualified)
1TRUE text on 05-02, signup 05-01 → diff = 1
3TRUE text on 05-04, signup 05-03 → diff = 1 (the FALSE row ignored)

Users 1 and 3 qualify. User 2 is excluded despite being only 23h59m later — '2025-05-02'::date - '2025-05-02'::date = 0. User 4 confirmed same-day (diff 0); user 5 confirmed two days later (diff 2). The DISTINCT matters most for user 3, whose row would survive once anyway here, but protects against a user with two qualifying TRUE texts appearing twice.

4. Strategic explanations

Date subtraction vs timestamp subtraction — the core trick

timestamp - timestamp returns an interval (e.g. 23:59:00), and "is this interval one day?" is awkward and ambiguous. date - date returns a plain integer — the count of calendar days between them. By casting first, "exactly the next calendar day" becomes the crisp test = 1. This is why user 2's 23h59m confirmation is correctly rejected: calendar-day arithmetic, not elapsed-time arithmetic, is what the question asks for.

Why DISTINCT — the join multiplies rows

One signup can have many texts, so the join can produce several rows per user. If a user had two TRUE confirmations both landing on day +1, the user_id would appear twice without DISTINCT. The spec says "if ANY TRUE confirmation falls on day +1, the user qualifies" — an existence test — so collapsing duplicates with DISTINCT (or an equivalent EXISTS subquery) is mandatory. An alternative phrasing is WHERE EXISTS (SELECT 1 FROM text_messages t WHERE ... ), which never multiplies rows in the first place.

The filter must sit in WHERE, not be confused with the join

Because this is an INNER JOIN, placing signup_ack = TRUE in the ON clause or the WHERE clause yields the same result — both restrict to confirmation texts. But the instinct should be deliberate: with an inner join, post-join row filters belong in WHERE. If this were a LEFT JOIN (to also list users with no qualifying text), the placement would change the answer drastically — an ON-clause condition preserves unmatched left rows, a WHERE-clause condition discards them.

NULL signup_ack and the boolean test

If signup_ack can be NULL (text sent, acknowledgement unknown), signup_ack = TRUE evaluates to UNKNOWN for those rows and they are correctly excluded — a NULL is not a confirmation. That is the desired behaviour here, but state it: a reviewer wants to see you noticed that = TRUE silently drops NULLs rather than treating them as FALSE.

Dialect notes

date - date = integer is PostgreSQL-specific. In MySQL use DATEDIFF(t.sent_ts, s.signup_ts) = 1 (which itself compares the date parts). In BigQuery, DATE_DIFF(DATE(t.sent_ts), DATE(s.signup_ts), DAY) = 1. SQL Server: DATEDIFF(day, s.signup_ts, t.sent_ts) = 1. All compute calendar-day differences; only the spelling changes.

Senior signal: cast timestamps to date so the difference is an integer day count — making "the next calendar day" the exact test = 1 and correctly rejecting a 23h59m same-day confirmation — and use DISTINCT or EXISTS because the one-to-many join would otherwise duplicate qualifying users.
