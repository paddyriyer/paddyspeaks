AMD's fulfillment operations team wants a scorecard: per warehouse, what percentage of orders actually shipped? It is a conditional count over a total — a ratio. The interviewer is checking two reflexes: that you force floating-point division rather than integer truncation, and that you think about what "fulfilled" excludes and what an empty warehouse does to the denominator.

1. The source data — orders

order_idstatuswarehouse_id
1fulfilled1
2fulfilled1
3pending1
4fulfilled2
5cancelled2
6fulfilled3

Warehouse 1 has a mix — 2 fulfilled, 1 pending; warehouse 2 has a cancelled order alongside a fulfilled one; warehouse 3 has a single, fulfilled order. The cancelled row and the lone-order warehouse are the edge cases the interviewer planted.

2. The SQL — conditional SUM over COUNT

SELECT warehouse_id,
  100.0 * SUM(CASE WHEN status='fulfilled' THEN 1 ELSE 0 END)/COUNT(*)
    AS fulfillment_rate
FROM orders GROUP BY warehouse_id;

The CASE emits 1 for a fulfilled order and 0 otherwise; SUM of those is the fulfilled count; dividing by COUNT(*) gives the fraction; the leading 100.0 scales it to a percentage.

3. The resulting analysis

166.6666666666666667

Warehouse 3 scores 100% (1 of 1), warehouse 2 scores 50% (1 of 2 — the cancelled order counts against it), warehouse 1 scores 66.67% (2 of 3). The result is correct — and 66.67% only because the leading 100.0 is a numeric literal that forced real division. Drop it and integer division would have returned a flat 0.

4. Strategic explanations

The integer-division trap — why 100.0 is essential

If you wrote SUM(...) / COUNT(*) * 100, both operands of the first division are integers, so Postgres does integer division: 2 / 3 evaluates to 0, then 0 × 100 = 0. Every warehouse would report 0% or 100%, nothing between. The leading 100.0 is a numeric literal — it promotes the whole expression to numeric arithmetic before the division, so 100.0 × 2 / 3 = 66.67. An explicit cast, SUM(...)::numeric / COUNT(*), is equally valid; what you must not do is divide two integers and expect a fraction.

SUM(CASE...) vs COUNT(*) FILTER — the modern idiom

The SUM(CASE WHEN ... THEN 1 ELSE 0 END) pattern is portable and universally understood, but Postgres offers a cleaner equivalent:

SELECT warehouse_id,
  100.0 * COUNT(*) FILTER (WHERE status = 'fulfilled') / COUNT(*)
    AS fulfillment_rate
FROM orders
GROUP BY warehouse_id;

COUNT(*) FILTER (WHERE ...) states the intent directly — "count rows matching this predicate". Same result; better readability. Mention both: the CASE form for cross-engine portability, FILTER for Postgres clarity.

What counts as the denominator?

The query divides by COUNT(*) — all orders, including cancelled ones. Is that right? If a customer cancels, the warehouse arguably never had a chance to fulfill, so a fairer "fulfillment rate" excludes cancellations from the denominator: COUNT(*) FILTER (WHERE status <> 'cancelled'). That single decision moves warehouse 2 from 50% to 100%. There is no universally correct answer — but a senior engineer surfaces the choice instead of silently baking one in.

The empty / no-order warehouse

A warehouse with zero orders produces no GROUP BY row at all — it simply vanishes from the scorecard, which may hide an idle facility. And if you ever join from a warehouse dimension table, a warehouse with no orders would hit COUNT(*) = 0 in the denominator and raise division by zero. Guard it: 100.0 * ... / NULLIF(COUNT(*), 0). Within a single orders-only GROUP BY every group has at least one row, so it cannot trigger here — but say why it is safe.

NULL and unexpected status values

status = 'fulfilled' is false for NULL statuses and for any value the data team did not anticipate ('returned', 'fulfilled ' with a stray space). Those rows land in the denominator but never the numerator — correct if they are genuinely not-fulfilled, a bug if a typo'd status should have counted. Validate the status domain, and consider LOWER(TRIM(status)) if the source is dirty.

Senior signal: explain that the leading 100.0 is what averts integer-division truncation, offer COUNT(*) FILTER (WHERE ...) as the readable Postgres idiom, question whether cancelled orders belong in the denominator, and guard divide-by-zero with NULLIF whenever the row set is not guaranteed non-empty.
