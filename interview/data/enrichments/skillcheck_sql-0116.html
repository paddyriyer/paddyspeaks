Two tables record per-day outcomes — one of failures, one of successes — and the task is a "state-changes" report: collapse consecutive same-status days into maximal runs with a start and end date. This is the classic gaps-and-islands problem. The interviewer is checking whether you know the difference-of-row-numbers trick to identify contiguous groups, rather than a procedural scan.

Two edge cases. 2024-12-31 sits outside calendar 2025 and must be dropped before the run logic. And 2025-01-07 is a lone success between failures — a run of length one, which the grouping must still emit as its own period.

2. The SQL — gaps-and-islands via difference of row numbers

WITH unioned AS (
  SELECT fail_date AS d, 'failed' AS state FROM failed
  UNION ALL
  SELECT success_date, 'succeeded' FROM succeeded
),
filtered AS (
  SELECT d, state FROM unioned WHERE d BETWEEN DATE '2025-01-01' AND DATE '2025-12-31'
),
grouped AS (
  SELECT d, state,
         ROW_NUMBER() OVER (ORDER BY d)
           - ROW_NUMBER() OVER (PARTITION BY state ORDER BY d) AS grp
  FROM filtered
)
SELECT state AS period_state, MIN(d) AS start_date, MAX(d) AS end_date
FROM grouped
GROUP BY state, grp
ORDER BY start_date;

The two tables are merged and labelled, then filtered to 2025. The grouped CTE computes a global row number and a per-state row number; their difference stays constant exactly while consecutive days share a state. Grouping by (state, grp) and taking MIN/MAX of the date collapses each run.

4. Strategic explanations

Why the difference of two row numbers identifies islands

This is the trick worth internalising. ROW_NUMBER() OVER (ORDER BY d) numbers all days 1, 2, 3... ROW_NUMBER() OVER (PARTITION BY state ORDER BY d) numbers days within each state. While a run of the same state continues, both counters advance in lockstep, so their difference is constant. The moment the state changes, the global counter keeps climbing but the per-state counter resets — the difference jumps. So (state, grp) is a unique key per contiguous run. Note the difference value itself is meaningless; only its constancy matters.

Why two non-adjacent failure runs do not merge

Failures on Jan 1-2 and Jan 5-6 are the same state, but the per-state row numbers run 1,2 (first run) then 3,4 (second run) — while the global numbers are 1,2 then 5,6. The differences are 0,0 then 2,2: two distinct grp values. Grouping by (state, grp) therefore yields two failure rows, not one merged span. This is exactly why you group by both columns, not by grp alone.

The dataset has no gap days — a real assumption

This solution treats "contiguous" as "consecutive rows in date order," which only equals "consecutive calendar days" if every day in range has a row. Here Jan 1-7 are all present. If a day were missing entirely (neither failed nor succeeded), the row-number difference would still see the surrounding same-state days as one run — bridging a true calendar gap. If gaps must break a run, you need a date-difference test (d - LAG(d) = 1) instead. State this assumption out loud.

The tag says "Recursive CTEs" — but this needs none

The question is tagged recursive CTEs, yet the solution uses only window functions and grouping — no WITH RECURSIVE. Gaps-and-islands can be done recursively (walk day by day, carrying run state), but the row-number-difference method is the standard, far more efficient approach: one sort, no iteration. Solve the problem, not the label — and say why the window-function form is preferable.

Dialect notes

ROW_NUMBER() OVER (...), UNION ALL, and BETWEEN are fully standard — this query runs unchanged on PostgreSQL, SQL Server, MySQL 8+, Oracle, and SQLite. The only Postgres-flavoured token is DATE '2025-01-01'; portable alternatives are CAST('2025-01-01' AS DATE) or, in most engines, the bare string literal which coerces in a date comparison.

Senior signal: solve gaps-and-islands with the constant difference of a global ROW_NUMBER and a per-state ROW_NUMBER, group by (state, grp) so non-adjacent runs of the same state stay separate, filter the date range before the run logic, and call out that the method assumes no missing calendar days unless a LAG-based gap test is added.
