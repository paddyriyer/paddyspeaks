A community dataset gives you a source_tab and a target_tab and asks for a reconciliation report — the bread-and-butter of every data-migration and ETL-validation job. You must classify each key into three buckets: present only on the source, present only on the target, or present on both but with a mismatched value. The interviewer is watching whether you cover all three cases cleanly and whether your NULL handling is sound.

Id 2 is the trap: it exists on both sides, so it is neither "only in source" nor "only in target" — it is a value mismatch and needs its own branch.

2. The SQL — three-way UNION ALL of join results

select s.id, 'Only in Source' as Comment
from source_tab s left join target_tab t
on s.id = t.id
where t.id is null

union all

select t.id,'Only in Target' as Comment
from target_tab t left join source_tab s
on s.id = t.id
where s.id is null

union all

select s.id, 'Mismatch' as Comment
from source_tab s inner join target_tab t
on s.id = t.id and s.letter <> t.letter;

Three independent queries stacked: a left-anti-join for source-only ids, the mirror for target-only ids, and an inner join keyed on equal id but unequal letter for mismatches.

4. Strategic explanations

Why LEFT JOIN ... IS NULL isolates "only in X"

A LEFT JOIN keeps every left-side row; rows with no partner get NULLs on the right. Filtering WHERE t.id IS NULL keeps exactly the unmatched left rows — the anti-join. The second block flips source and target to find target-only ids. Two anti-joins plus one inner join cover all three discrepancy types.

UNION ALL vs UNION — pick the cheap one deliberately

The three branches produce disjoint id sets by construction, so there is nothing to deduplicate. UNION ALL simply concatenates them; plain UNION would add a pointless distinct-sort over the whole result. When you can prove the inputs are disjoint, UNION ALL is both correct and faster — saying why you chose it is the signal.

The FULL OUTER JOIN alternative — one pass, all three cases

The whole report is one statement if you use a full outer join:

SELECT COALESCE(s.id, t.id) AS id,
  CASE WHEN t.id IS NULL                  THEN 'Only in Source'
       WHEN s.id IS NULL                  THEN 'Only in Target'
       WHEN s.letter <> t.letter           THEN 'Mismatch'
       ELSE 'Match' END AS comment
FROM source_tab s FULL OUTER JOIN target_tab t ON s.id = t.id;

One join, one scan of each table, every id classified. Filter out 'Match' if matches should not appear. This scales better than three separate joins and keeps the logic in one readable CASE.

The NULL-letter trap in the mismatch branch

s.letter <> t.letter evaluates to unknown — not true — whenever either letter is NULL. So an id whose letter is 'A' on the source and NULL on the target would not be flagged as a mismatch, even though the values genuinely differ. NULL-safe comparison fixes it: Postgres s.letter IS DISTINCT FROM t.letter, or COALESCE both sides to a sentinel. On a real reconciliation job this silent miss is the bug that ships.

Senior signal: collapse the three-way UNION ALL into a single FULL OUTER JOIN with a CASE, justify UNION ALL over UNION by proving the branches are disjoint, and replace <> with IS DISTINCT FROM so a NULL on one side is still caught as a mismatch.
