Given unsorted positions on a number line, find the smallest distance between any two of them — the "closest pair in 1-D" problem. The naive answer cross-joins every point with every other and is O(n²). The interviewer is explicitly probing whether you know the key insight: once the points are sorted, the closest pair must be adjacent, so a single LAG over the sorted list solves it in O(n log n).

The rows arrive unordered. The edge cases: a negative coordinate (the distance formula must not go negative), and a tie for the minimum — the gap -1→0 and the gap 10→11 are both 1. The answer is a single number, so the tie is invisible in the output but tests that your aggregate picks the true minimum.

2. The SQL — LAG over the sorted list, then MIN

SELECT MIN(x - prev_x)::int AS shortest
FROM (
  SELECT x, LAG(x) OVER (ORDER BY x) AS prev_x
  FROM point
) t
WHERE prev_x IS NOT NULL;

LAG(x) OVER (ORDER BY x) sorts the points and attaches each point's immediate predecessor. Because the list is sorted, x >= prev_x always, so x - prev_x is a non-negative adjacent gap. MIN over those gaps is the shortest distance. The WHERE prev_x IS NOT NULL drops the first row, which has no predecessor.

3. The resulting analysis

The six adjacent gaps from the sorted sequence -1, 0, 2, 5, 10, 11, 20 are 1, 2, 3, 5, 1, 9. MIN returns 1. Note it appears twice; MIN is indifferent to multiplicity, so the tie produces the correct single answer with no special handling.

4. Strategic explanations

Why the closest pair must be adjacent after sorting

This is the load-bearing insight. Suppose the two closest points were not neighbours in the sorted order — then some third point lies strictly between them, and that third point is closer to each of them than they are to each other. Contradiction. Therefore the global minimum distance is always realised by some adjacent pair, and you only need to inspect the n - 1 adjacent gaps, not all n(n-1)/2 pairs.

The complexity argument the interviewer wants spoken aloud

The cross-join SELECT MIN(ABS(a.x - b.x)) FROM point a JOIN point b ON a.x < b.x materialises ~n²/2 pairs — fine for 7 rows, catastrophic for a million. The LAG approach costs one sort (O(n log n)) plus one linear pass. Stating "sorted-adjacency reduces this from quadratic to log-linear" is exactly the senior signal the prompt is fishing for.

Why no ABS is needed — and when it would be

Because the window orders by x ascending, every current x is ≥ its prev_x, so x - prev_x is already non-negative even though the data includes -1. If you instead used LEAD or computed prev_x - x, you would get negatives and MIN would return the wrong (most-negative) value — there you would wrap it in ABS. The direction of the subtraction must match the sort direction.

Edge cases — the first row and duplicate coordinates

The smallest x has LAG(x) = NULL; NULL - prev is NULL and MIN would ignore it anyway, but the explicit WHERE prev_x IS NOT NULL makes intent obvious. The prompt promises all x are distinct; if duplicates were possible, two equal coordinates would produce a gap of 0 — arguably the correct "shortest distance" of zero, but worth confirming the business definition. A table with fewer than two rows yields no adjacent pair and MIN returns NULL.

Dialect notes

LAG with an ORDER BY in the OVER clause is standard SQL — PostgreSQL, SQL Server, Oracle, MySQL 8+, BigQuery, and SQLite all support it. The only non-portable touch is the ::int cast, which is Postgres syntax; elsewhere use CAST(... AS INT). The algorithm itself needs no translation.

Senior signal: recognise closest-pair-in-1-D as a sorted-adjacency problem, replace the O(n²) self-cross-join with one LAG over the ordered list plus MIN for O(n log n), match the subtraction direction to the sort order so no ABS is needed, and articulate the proof that the nearest pair is always adjacent.
