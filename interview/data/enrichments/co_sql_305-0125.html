TCS runs an infrastructure SLA dashboard for a client, and the headline figure is total downtime per server in hours. The reference solution computes it with a chain of EXTRACT and arithmetic that looks deliberately convoluted — the interviewer wants to see whether you can simplify it, and whether you notice the messy floating-point output it produces.

1. The source data — downtime

server_idstart_tsend_ts
12024-01-01 02:00:002024-01-01 05:30:00

22024-01-02 09:15:002024-01-02 09:45:00
32024-01-05 00:00:002024-01-06 00:00:00

Server 1 has two outages that must be summed (3.5h + 3h). Server 2's outage crosses no hour boundary cleanly (30 minutes). Server 3's outage spans a calendar day — a good check that the math gives 24h, not 1.

2. The SQL — SUM of EXTRACT(EPOCH)

SELECT server_id,
       SUM(((EXTRACT(EPOCH FROM (end_ts - start_ts)) / 86400.0) * 86400) / 3600)
         AS total_hours
FROM downtime
GROUP BY server_id;

end_ts - start_ts is an interval; EXTRACT(EPOCH FROM …) turns it into seconds. The / 86400.0 * 86400 pair cancels out (divide by a day, multiply by a day), so the expression is really just seconds / 3600 — downtime in hours — summed per server.

The numbers are right — server 1 is 3.5 + 3 = 6.5h, server 2 is 0.5h, server 3 is a full 24h — but look at the display. The redundant / 86400.0 introduces a fractional intermediate, and the trailing …9992 is floating-point dust. The arithmetic is correct; the presentation is not dashboard-ready.

4. Strategic explanations

Simplify the expression — the day factors cancel

(EXTRACT(EPOCH FROM d) / 86400.0) * 86400 / 3600 reduces algebraically to EXTRACT(EPOCH FROM d) / 3600. The detour through "days" does nothing but inject rounding error. Write what you mean:

SELECT server_id,
       ROUND(SUM(EXTRACT(EPOCH FROM (end_ts - start_ts))) / 3600.0, 2)
         AS total_hours
FROM downtime
GROUP BY server_id
ORDER BY server_id;

Note SUM first, then divide — one division instead of one per row — and a final ROUND(…, 2) so the SLA report reads 6.50, not 6.499…92.

The interval-to-hours follow-up

Interviewers often ask for the answer without EXTRACT at all. Postgres can sum intervals directly:

SELECT server_id, SUM(end_ts - start_ts) AS total_downtime
FROM downtime
GROUP BY server_id;

That returns a native interval like 06:30:00 — perfect if the consumer wants HH:MM:SS. The EXTRACT(EPOCH …) / 3600 form is better when the consumer wants a decimal number to do further math on. Pick based on the output contract.

The trap: end_ts < start_ts and open outages

If a row has end_ts earlier than start_ts (clock skew, a bad import), the interval is negative and silently reduces the total — a server looks healthier than it is. And an ongoing outage often has end_ts IS NULL; NULL - start_ts is NULL, and SUM skips it, so live incidents vanish from the report. Guard both:

SUM(EXTRACT(EPOCH FROM (COALESCE(end_ts, NOW()) - start_ts)) / 3600.0)
WHERE end_ts IS NULL OR end_ts >= start_ts
Integer-division gotcha — why the literal needs a dot

Dividing by 3600.0 (not 3600) forces numeric division. EXTRACT(EPOCH …) already returns numeric so it is safe here, but in engines where the seconds value is an integer, / 3600 would floor 6.5 to 6. The decimal point on the divisor is a cheap insurance policy.

Overlapping outages — the question behind the question

If two rows for the same server overlap in time, a plain SUM double-counts the shared minutes. "Total downtime" usually means wall-clock time the server was down at least once, which needs an interval-merge (a gaps-and-islands pass) before summing. Worth asking: "can outage windows for one server overlap?" If yes, the simple sum is wrong.

Senior signal: recognise that the /86400*86400 pair cancels and simplify to EXTRACT(EPOCH …)/3600, SUM before dividing and ROUND for a clean report, guard against negative and NULL end_ts, and raise overlapping-window double-counting before trusting a naive sum.
