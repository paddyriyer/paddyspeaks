A regional sales lead wants the classic quarterly grid: "one row per region, four columns Q1 through Q4." The source stores sales in long form — one row per region/quarter — so the task is a pivot: rotate the distinct quarter values up into column headers. There is no portable PIVOT keyword, so the interviewer wants the conditional-aggregation idiom and wants to see it survive missing quarters and duplicate rows.

Two details decide the interview. NA has two Q1 rows, and EU has no Q3 row. A correct pivot must fold the duplicates together and supply a value for the missing cell.

2. The SQL — pivot via conditional aggregation

SELECT
  region,
  SUM(CASE WHEN quarter = 'Q1' THEN amount ELSE 0 END) AS q1,
  SUM(CASE WHEN quarter = 'Q2' THEN amount ELSE 0 END) AS q2,
  SUM(CASE WHEN quarter = 'Q3' THEN amount ELSE 0 END) AS q3,
  SUM(CASE WHEN quarter = 'Q4' THEN amount ELSE 0 END) AS q4
FROM sales
GROUP BY region
ORDER BY region;

One CASE per target column lets a row's amount through only when it belongs to that quarter, contributing 0 otherwise. GROUP BY region collapses each region's rows into one output row, and SUM totals whatever every CASE admitted.

4. Strategic explanations

Why this is a pivot, not a plain GROUP BY

SELECT region, quarter, SUM(amount) ... GROUP BY region, quarter returns the same numbers as rows — long format, one row per region/quarter pair. A pivot turns the quarter values into column headers: it is a rotation. The SUM(CASE WHEN ...) idiom — one expression per output column — is the universal way to rotate, because CASE exists in every SQL engine.

ELSE 0 versus no ELSE — a deliberate choice

With ELSE 0, a region with no sales in a quarter shows 0. Drop the ELSE and non-matching rows become NULL; SUM ignores NULLs, so a region that sold nothing all quarter would yield NULL rather than 0. For a financial grid 0 is usually right — the sale genuinely was zero. A coverage report might prefer NULL to keep "no data" distinct from a real zero. EU's Q3 is the case in point: here we want it to read 0.

Duplicate rows fold in — by design

NA's two Q1 rows are not an error to dedupe; SUM over the group adds every matching row, so 100 + 40 = 140. If the requirement were "how many distinct selling quarters" you would switch to COUNT(DISTINCT quarter); if the duplicates were genuine data-quality noise you would dedupe in a CTE first. The aggregate you choose is the business rule — state which one you mean.

The hard limit: the columns are static

The four quarters are hard-coded. A row tagged 'Q5' or a typo'd 'q1' would match no CASE and silently vanish from the report. Pure SQL cannot grow columns from data: a truly dynamic pivot needs a procedural step (build the SQL text from SELECT DISTINCT quarter, then execute it) or PostgreSQL's crosstab() from the tablefunc extension. Naming this limitation unprompted is the senior signal.

Dialect — the dedicated PIVOT operator

Snowflake, Oracle, and SQL Server ship a PIVOT clause — e.g. Snowflake: SELECT * FROM sales PIVOT(SUM(amount) FOR quarter IN ('Q1','Q2','Q3','Q4')). PostgreSQL, MySQL, and SQLite have no PIVOT keyword, so the SUM(CASE) form shown here is the portable answer — and it still makes you enumerate the values, exactly like that explicit IN list.

Senior signal: recognise a pivot as rotating row values into columns, implement it portably with one SUM(CASE WHEN ...) per column, choose ELSE 0 versus NULL deliberately, know that SUM folds duplicate cells together by design, and call out unprompted that the column list is static.
