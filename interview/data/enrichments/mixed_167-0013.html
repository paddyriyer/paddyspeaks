A finance team reviews the annual P&L and asks: "for each year, what was revenue growth versus the prior year, as a percentage?" The query uses the LAG window function to pull last year's revenue onto the current row, then computes the growth ratio. The interviewer is probing whether you reach for a window function rather than a self-join, and whether you defend the division against a zero or NULL prior year.

1. The source data — annual_revenue

yearrevenue(narrative)
2020500000first year — no prior
2021650000up from 2020
2022650000flat — same as 2021
2023520000down from 2022
2024910000sharp recovery

The edge cases: 2020 has no prior year (LAG returns NULL), and 2022 is flat — a 0% growth row that must not be confused with a NULL.

2. The SQL — LAG for prior-year revenue

SELECT year, revenue,
  100.0 * (revenue - LAG(revenue) OVER (ORDER BY year))
        / NULLIF(LAG(revenue) OVER (ORDER BY year), 0) AS yoy_pct
FROM annual_revenue;

3. The resulting analysis

yearrevenueyoy_pctcommentary
2020500000NULLno prior year to compare
202165000030.00(650k − 500k) / 500k
20226500000.00flat — genuine zero, not NULL
2023520000−20.00decline shows as negative
202491000075.00(910k − 520k) / 520k

2020 is NULL — correct, there is nothing before it. 2022 is 0.00 — also correct, revenue was genuinely flat. The query distinguishes "no comparison possible" from "comparison says no change."

4. Strategic explanations

Why LAG is the right tool

LAG(revenue) OVER (ORDER BY year) returns the revenue value from the row one position earlier in the ordering. It places this year's and last year's figures on the same row, so a simple arithmetic expression computes the delta. The pre-window alternative — annual_revenue a JOIN annual_revenue b ON b.year = a.year - 1 — works but is more code, breaks if a year is missing, and reads less clearly. LAG states the intent directly.

The ORDER BY inside OVER is mandatory

LAG has no meaning without an ordering — "the previous row" is undefined otherwise. OVER (ORDER BY year) defines the sequence. If the data spanned multiple business units, you would add PARTITION BY unit so LAG never bleeds the last year of one unit into the first year of another. Here a single series needs only the ORDER BY.

The NULLIF guard — division-by-zero defence

If a prior year's revenue were 0, the denominator would be zero and the query would error. NULLIF(LAG(...), 0) converts a zero divisor to NULL, and dividing by NULL yields NULL instead of crashing — the row degrades to "undefined growth" gracefully. This guard is the senior reflex on any ratio. (Note: a zero base also makes percentage growth mathematically meaningless — NULL is the honest answer.)

The NULL-vs-zero distinction

2020's yoy_pct is NULL because LAG has no prior row — revenue - NULL is NULL, the whole expression is NULL. 2022's is 0.00 because revenue truly did not change. Conflating them is a real reporting bug: a dashboard that renders NULL as 0% would falsely claim 2020 was a flat year. Keep them distinct, or label the first row explicitly as "baseline."

The integer-division trap and dialect notes

Multiplying by 100.0 (not 100) forces floating-point math; with integer revenue and a bare 100, some engines would truncate 30.00 toward an integer or compute 0 for sub-100% changes. LAG, NULLIF, and the window OVER clause are standard and portable across Postgres, MySQL 8+, SQL Server, BigQuery, Snowflake, and Oracle. LAG(revenue, 1, 0) with a third argument would substitute a default for the missing prior year — usually not what you want for a percentage, since it manufactures a misleading number.

Senior signal: use LAG ... OVER (ORDER BY year) instead of a self-join, always wrap a ratio denominator in NULLIF(..., 0), and articulate that the first-year NULL ("no comparison") and a flat-year 0% ("no change") are different facts a dashboard must not merge.
