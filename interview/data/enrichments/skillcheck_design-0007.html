An ad-impression fact table at 500 million rows per day is one of the largest-volume analytical tables a senior engineer will encounter. The interviewer is probing partition and clustering strategy for a BigQuery-native design: choosing the right partition column to enable pruning on time-range queries, choosing clustering columns in the right order to match the most selective predicates, setting a partition expiration to manage cost, and handling privacy-safe user identification. Getting the PARTITION BY / CLUSTER BY ordering and the OPTIONS block right signals hands-on BigQuery experience.

1. The requirements — what the design must capture

  • Volume: ~500M rows/day — partitioning and clustering are not optional; they are the primary cost-control levers.
  • Common predicates: filter by event_date, filter by advertiser_id, aggregate by campaign_id.
  • Partition column: event_date DATE — date-level partitioning is the standard BigQuery pattern; each day is one partition.
  • Cluster columns: advertiser_id, campaign_id, country — in selectivity order (advertiser filters most, country least).
  • Privacy: user identity must be pseudonymous (user_pseudo_id), not a raw PII identifier.
  • Bid/win price: stored in micros (integer) to avoid float arithmetic on monetary values.
  • Partition expiration: 400 days covers a full year plus reporting lag; older partitions are dropped automatically.

2. The DDL — the CREATE TABLE

CREATE TABLE fact_ad_impression (
  impression_id       STRING       NOT NULL,        -- natural key from edge server
  event_ts            TIMESTAMP    NOT NULL,        -- event time (from client/edge)
  event_date          DATE         NOT NULL,        -- partition column
  advertiser_id       INT64        NOT NULL,
  campaign_id         INT64        NOT NULL,
  creative_id         INT64,
  placement_id        INT64,
  user_pseudo_id      STRING,                       -- privacy-safe pseudonymous id
  country             STRING,
  device_type         STRING,
  bid_price_micros    INT64,
  win_price_micros    INT64,
  was_viewable        BOOL,
  ingested_at         TIMESTAMP
)
PARTITION BY event_date
CLUSTER BY advertiser_id, campaign_id, country
OPTIONS (
  partition_expiration_days = 400,
  description = 'One row per ad impression. Partitioned by event_date, clustered for advertiser/campaign filtering.'
);

4. Strategic explanations

Grain declaration

One row per ad impression event. This is a transaction fact — the most granular possible grain. Aggregated tables (hourly or daily rollups) would be materialised views or summary tables derived from this base fact, not this table itself. At 500M rows/day, 365 days without expiration is 182.5B rows; the 400-day expiration cap keeps the table within manageable bounds while preserving over a full year of trend data.

Partitioning: event_date as the primary control knob

BigQuery charges by bytes scanned per query. A query with WHERE event_date = ‘2025-05-01’ scans only that day’s partition — 1/400 of the table. Without partitioning, every query scans all 182.5B rows. The event_date column is a required redundant derivation of event_ts; BigQuery cannot auto-partition on TIMESTAMP, so the DATE column must be materialised. ETL must populate it consistently, ideally in the same pipeline step that extracts event_ts.

Clustering: column order matters

BigQuery clustering sorts data within each partition into blocks. The cluster key order (advertiser_id, campaign_id, country) reflects decreasing selectivity: most queries specify an advertiser, fewer also specify a campaign, fewer still also specify a country. BigQuery’s cost estimator prunes blocks matching any prefix of the cluster key. Putting advertiser_id first maximises the pruning benefit for the most common query type. Reversing the order (e.g., country first) would only help country-level-only queries, which are less common.

Prices in micros — why not FLOAT64 or NUMERIC

INT64 micros avoid two problems: (1) floating-point rounding errors that accumulate when summing billions of prices, and (2) the overhead of NUMERIC (arbitrary precision, slower arithmetic). The micros convention is standard in ad tech (Google’s API surfaces use it). To display as dollars, divide by 1,000,000 in the query layer or in a reporting-layer view.

BigQuery-specific OPTIONS block

partition_expiration_days = 400 causes BigQuery to automatically drop partitions older than 400 days, preventing unbounded table growth without manual DELETE jobs. The description field serves as inline documentation in the BigQuery UI and the INFORMATION_SCHEMA, visible to analysts without access to source-control comments. These are BigQuery-native features absent from Redshift and Snowflake.

Equivalent designs in other engines

In Snowflake, drop the PARTITION BY / CLUSTER BY / OPTIONS syntax and use CLUSTER BY (event_date, advertiser_id, campaign_id); Snowflake handles micro-partition organisation automatically. In Redshift, use DISTKEY(advertiser_id) to co-locate an advertiser’s data across slices and SORTKEY(event_date, advertiser_id, campaign_id) for zone-map pruning. In Redshift, partition management is manual (partition elimination is done via sort-key zone maps, not BigQuery-style date partitions).

Senior signal: justify the event_date DATE column as a mandatory redundancy (“BigQuery cannot partition on TIMESTAMP”); explain cluster key ordering by descending selectivity and why it matters for block pruning; defend micros over FLOAT64 (“rounding errors accumulate at 500M rows/day”); and mention partition expiration as the BigQuery-native cost-control mechanism — these four points demonstrate that you design for query cost and operational manageability, not just schema correctness.
