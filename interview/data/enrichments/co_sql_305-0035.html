IBM's promotions team wants a running total of discount dollars given away — a cumulative tracker so finance can see the giveaway accumulate day by day. The reference solution uses SUM(…) OVER (ORDER BY order_date). It is the right tool, and the interview lives in one subtlety: what the window does when two orders share a date.

1. The source data — orders

order_idorder_datediscount_amount

22024-01-02$25

42024-01-04$40
52024-01-05$15

Orders 2 and 3 fall on the same date, 2024-01-02. Two rows, one date — that tie is exactly what exposes how the window frame works.

2. The SQL — SUM() OVER (ORDER BY order_date)

SELECT order_date, discount_amount,
  SUM(discount_amount) OVER (ORDER BY order_date) AS cumulative_discount
FROM orders;

A windowed SUM ordered by date. No PARTITION BY, so the running total spans the whole table; the ORDER BY defines what "so far" means.

3. The resulting analysis

Both 2024-01-02 rows show 40, not 35 and 40. The window did not add the two same-date orders one at a time — it added them together, so both rows see the full day-end total. That is the default frame behaving exactly as specified, and it surprises people.

4. Strategic explanations

The default frame is RANGE, and that is why the tie shows 40

When you write ORDER BY in a window with no explicit frame, the default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. Crucially, RANGE's "current row" means every row with the same ORDER BY value — they are peers. So for both 2024-01-02 orders the frame includes each other, and both get 10 + 25 + 5 = 40. If you want a strict row-by-row running total — 35 then 40 — switch the frame to ROWS and add a tiebreaker:

SELECT order_date, discount_amount,
  SUM(discount_amount) OVER (
    ORDER BY order_date, order_id
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS cumulative_discount
FROM orders;
-- now the two 2024-01-02 rows show 35 and 40

ROWS counts physical rows; RANGE counts logical peer-groups. For a daily tracker, the RANGE answer (both rows = day-end total) is often more correct — but you must say which you intended.

Per-day totals: do you even want one row per order?

If finance wants one cumulative figure per day, returning two rows for 2024-01-02 is noise. Aggregate to the day first, then run the window over the daily totals:

SELECT order_date, daily_discount,
       SUM(daily_discount) OVER (ORDER BY order_date) AS cumulative_discount
FROM (
  SELECT order_date, SUM(discount_amount) AS daily_discount
  FROM orders
  GROUP BY order_date
) d;

One row per date, cumulative 10, 40, 80, 95 — the gap-day 2024-01-03 simply absent. Clarify the grain the consumer expects before choosing.

The trap: a non-deterministic order with no tiebreaker

ORDER BY order_date alone does not uniquely order the two same-date rows. With the RANGE default it does not matter — peers get the same total either way. But the instant you switch to ROWS, the per-row running values (35 vs 40) depend on which same-date row the engine processes first, and that is not guaranteed. Always add a deterministic tiebreaker like order_id to a ROWS frame.

Gaps, ordering, and a missing window: edge cases

2024-01-03 has no orders, so it never appears — the running total simply has no row for it. If the tracker must show every calendar day (carrying the total flat across gaps), join against a generated date series first. Also note the output is unordered unless you add a top-level ORDER BY order_date — the window's internal ordering does not guarantee the result-set ordering. And if discount_amount can be NULL, SUM skips it, so the running total flatlines across that row rather than breaking.

Dialect note

This is standard SQL and runs as-is in PostgreSQL. Some engines (older MySQL) lack window functions entirely — there the cumulative total needs a correlated subquery or a self-join, which is O(N²). Naming that fallback shows range.

Senior signal: know that an ORDER BY window defaults to a RANGE frame so same-date rows become peers sharing the day-end total, switch to ROWS with a deterministic tiebreaker when a strict row-by-row running total is wanted, and pre-aggregate to the day when the consumer expects one cumulative figure per date.
