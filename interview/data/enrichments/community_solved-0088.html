An engagement team wants customers who bought on back-to-back calendar days — a same-day-then-next-day pattern that signals an active, sticky shopper — restricted to meaningful purchases above $100. For each such customer the query returns the span of their consecutive-purchase activity and the combined value. The interviewer is probing the LEAD-and-date-difference technique for detecting adjacency.

2. The SQL — LEAD to the next purchase, then a date diff

WITH lead_purchase AS (
  SELECT purchase_id, customer_id, purchase_date, amount,
    LEAD(purchase_date, 1, NULL) OVER (PARTITION BY customer_id ORDER BY purchase_date ASC) AS lead_purchase_date,
    LEAD(amount, 1, NULL) OVER (PARTITION BY customer_id ORDER BY purchase_date ASC) AS lead_purchase_amount
  FROM purchases_info
  WHERE amount > 100
), with_diff AS (
  SELECT *,
    (lead_purchase_date::date - purchase_date::date) AS diff_date
  FROM lead_purchase
), base_data AS (
  SELECT DISTINCT customer_id,
    MIN(purchase_date) OVER (PARTITION BY customer_id) AS purchase_start_date,
    FIRST_VALUE(amount) OVER (PARTITION BY customer_id ORDER BY purchase_date ASC) AS purchase_amount,
    MAX(lead_purchase_date) OVER (PARTITION BY customer_id) AS purchase_end_date,
    SUM(lead_purchase_amount) OVER (PARTITION BY customer_id) AS lead_purchase_amount
  FROM with_diff
  WHERE diff_date = 1
)
SELECT customer_id, purchase_start_date, purchase_end_date,
       (purchase_amount + lead_purchase_amount) AS total_purchase_amount
FROM base_data;

4. Strategic explanations

The adjacency test — LEAD plus a date subtraction

LEAD(purchase_date) OVER (PARTITION BY customer_id ORDER BY purchase_date) pulls each row's next purchase date for the same customer. Subtracting dates in Postgres yields an integer day count, so diff_date = 1 means "the next purchase was exactly the following calendar day". Filtering to diff_date = 1 keeps only rows that begin a consecutive-day pair. This LEAD/LAG-then-difference idiom is the standard way to detect adjacency without a self-join.

The filter-before-window subtlety

WHERE amount > 100 runs before LEAD is computed, so the window function operates only on surviving rows. This is deliberate and consequential: customer 2's $80 purchase is removed first, and then LEAD links purchase 5 directly to purchase 7 — which are two days apart. Had the filter run after the window, the chain would look different. Knowing that WHERE precedes window evaluation (window functions belong to the SELECT stage) is exactly the kind of execution-order knowledge an interviewer wants to hear stated.

The first-row / last-row NULL from LEAD

The final purchase in each customer's sequence has no successor, so LEAD returns the supplied default NULL. Its diff_date is then NULL, and NULL = 1 is unknown — so a trailing purchase never falsely registers as a consecutive pair. Customer 3, with only one purchase, has every LEAD value NULL and is correctly dropped. The NULL handling is load-bearing, not incidental.

Where this query is fragile — runs longer than two days

Customer 1 actually has a three-day run (Mar 1-2-3). The query collapses the customer to a single row via DISTINCT + partition-wide MIN/MAX/SUM, so total_purchase_amount = FIRST_VALUE(amount) + SUM(lead_purchase_amount over qualifying rows). For a clean single run that arithmetic happens to total the run's purchases, but the design does not generalise: a customer with two separate consecutive-day runs would have them merged into one span and one sum, which is almost certainly not intended. The robust approach is gaps-and-islands — assign a run id, then aggregate per run, not per customer.

What to say next — name the real shape of the problem

"Consecutive purchases" is a runs problem, and the senior answer says so: tag each row with whether it continues the previous day's run, cumulative-sum the breaks into a run id, then GROUP BY (customer_id, run_id) to get each run's start, end, length and value. That correctly handles multiple runs per customer and runs of any length — the LEAD-and-partition-aggregate shortcut here only survives because the sample has one tidy run per customer.

Senior signal: use LEAD + a date subtraction to test day-adjacency, and know that WHERE filters before the window runs — that is what breaks customer 2's chain. Then flag the real limitation: collapsing per-customer cannot separate multiple runs — a proper gaps-and-islands run id is the correct general solution.
