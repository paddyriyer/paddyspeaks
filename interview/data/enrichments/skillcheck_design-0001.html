A star-schema fact table for ride-share trips is a classic analytical modeling exercise. The interviewer is probing whether you understand grain declaration (one row = one completed or attempted trip), the distinction between surrogate and natural keys, proper FK references to shared dimension tables (including role-playing dimensions like location), and the choice of partition column for large-scale time-series data.

1. The requirements — what the design must capture

  • Grain: one row per trip event (transaction fact).
  • Dimension FKs: rider, driver, vehicle, pickup location, dropoff location, payment method — location is role-played from a single dim_location.
  • Degenerate dimension: the natural trip key from the source system, kept on the fact for drill-through.
  • Measures: distance, duration, fare, surge multiplier, tip, driver payout — mix of additive and non-additive.
  • Status flag: completed / cancelled / no_show — a degenerate attribute rather than a separate dimension.
  • Partition column: a DATE column derived from the event timestamp for efficient time-range pruning.
  • Audit column: ingestion timestamp for pipeline observability.

2. The DDL — the CREATE TABLE

CREATE TABLE fact_trip (
  trip_sk             BIGINT      NOT NULL,        -- surrogate PK
  trip_nk             VARCHAR(64) NOT NULL,        -- natural key from source
  rider_sk            BIGINT      NOT NULL,        -- FK -> dim_rider
  driver_sk           BIGINT      NOT NULL,        -- FK -> dim_driver
  vehicle_sk          BIGINT,                      -- FK -> dim_vehicle
  pickup_location_sk  BIGINT      NOT NULL,        -- FK -> dim_location
  dropoff_location_sk BIGINT      NOT NULL,        -- FK -> dim_location
  payment_method_sk   BIGINT,                      -- FK -> dim_payment
  request_ts          TIMESTAMP   NOT NULL,        -- degenerate dim / event time
  pickup_ts           TIMESTAMP,
  dropoff_ts          TIMESTAMP,
  -- measures
  distance_km         DECIMAL(8,3),
  duration_seconds    INT,
  fare_amount         DECIMAL(10,2),
  surge_multiplier    DECIMAL(4,2),
  tip_amount          DECIMAL(10,2),
  driver_payout       DECIMAL(10,2),
  status              VARCHAR(20),                  -- completed / cancelled / no_show
  -- partition / lineage
  trip_date           DATE NOT NULL,
  ingested_at         TIMESTAMP DEFAULT CURRENT_TIMESTAMP
)
PARTITION BY (trip_date);

4. Strategic explanations

Grain declaration

The grain is one row per trip attempt. This includes cancelled and no-show trips — a deliberate choice. Excluding them would make cancellation-rate metrics impossible without a separate table. Declaring grain explicitly before writing DDL is the single most important step in fact-table design: every column decision follows from it.

Surrogate vs. natural keys

trip_sk is a BIGINT surrogate generated in the warehouse. trip_nk preserves the source system's string identifier. Surrogates are stable (source systems can reassign natural keys), join-efficient (integer equality vs. string comparison), and SCD-friendly (the same surrogate can absorb multiple natural-key versions). The natural key stays on the fact as a degenerate dimension rather than forcing a round-trip join to retrieve it.

Role-playing dimensions

Both pickup_location_sk and dropoff_location_sk are foreign keys into the same dim_location table. In queries the table is aliased twice: JOIN dim_location pickup ON pickup.location_sk = f.pickup_location_sk. This is the canonical role-playing pattern — it avoids denormalising the dimension or duplicating its rows.

Fact-table type: transaction fact

This is a transaction fact: one row per discrete event (the trip), written once, never updated. Contrast with a periodic snapshot (one row per entity per time period, e.g., weekly driver earnings) or an accumulating snapshot (one row per business process lifecycle, updated as milestones occur). Transaction facts are the right choice when event-level analysis (per-trip revenue, cancellation details) matters.

Additive vs. non-additive measures

distance_km, fare_amount, tip_amount, and driver_payout are fully additive — they sum correctly across any dimension. surge_multiplier is non-additive: summing multipliers across trips is meaningless. It should be averaged or used in weighted calculations. duration_seconds is additive for fleet-level totals but must be converted (divide by trip count) to get average trip duration.

Partition and clustering strategy

PARTITION BY (trip_date) ensures that time-range queries (common in ride-share analytics — daily/weekly trends) scan only relevant partitions. In BigQuery, pair this with CLUSTER BY rider_sk, driver_sk to accelerate per-rider or per-driver queries. In Redshift use DISTKEY(driver_sk) if driver-level aggregations dominate and SORTKEY(trip_date, driver_sk) for zone-map pruning. In Snowflake, CLUSTER BY (trip_date, driver_sk) achieves similar micro-partition pruning without PARTITION BY syntax.

Nullable FKs and the unknown member

Nullable surrogate keys (vehicle_sk, payment_method_sk) are valid but require LEFT JOINs and can silently drop rows in INNER JOIN queries. An alternative is to provision an “Unknown” row (SK = −1) in each dimension and default NULLable FKs to −1, making all joins INNER. The tradeoff: cleaner SQL vs. slightly misleading “Unknown” aggregates. Either approach is defensible if stated explicitly.

Senior signal: declare the grain first (“one row per trip attempt, including cancellations”) before touching DDL; identify the role-playing dimension pattern for location; classify each measure as additive, semi-additive, or non-additive; justify the trip_date partition column as a derived redundant column for pruning efficiency; and discuss nullable FK alternatives (unknown member row) — these four moves together demonstrate dimensional modeling fluency that goes well beyond memorising star-schema vocabulary.
