An idempotency key table sits in the hot path of every payment API: it stores the first response to a client-supplied key so that any retry — from a mobile app re-sending after a timeout, or an orchestrator replaying a failed step — receives the identical body and status without re-executing the money movement. The interviewer is probing whether you understand composite key scoping (keys must be namespaced to the endpoint to prevent cross-endpoint collisions), in-flight vs. completed state (a concurrent request with the same key must be held, not executed), request-body hashing (to detect a client re-sending a different body with an existing key, which is a protocol violation), and TTL housekeeping to keep the table bounded.

1. The requirements — what the design must capture

  • Scope per endpoint: the same idempotency key string can legitimately be used by the same client on different API paths; the PK must include api_path.
  • Client-supplied key: the key is chosen by the caller (typically a UUID) and must be stored verbatim.
  • Request body hash: if a client re-uses a key with a different body, the server must detect and reject it; a SHA-256 hex digest satisfies this with a fixed-width CHAR(64).
  • Two-phase status: a row is inserted as in-flight (status 0) the moment a request is received; it is updated to completed (status 1) once the response is ready. A concurrent duplicate reads the in-flight row and must wait or return 409.
  • Stored response: the full HTTP response — status code, body, and relevant headers — must be captured so retries can reproduce it exactly without touching downstream systems.
  • TTL: rows must expire; an expires_at column drives a periodic delete; an index on it makes the delete a fast range scan.

2. The DDL — the CREATE TABLE

CREATE TABLE idempotency_key (
  api_path           VARCHAR(128) NOT NULL,                  -- scope keys per endpoint
  idempotency_key    VARCHAR(128) NOT NULL,                  -- client-supplied
  request_hash       CHAR(64)     NOT NULL,                  -- hash of the request body
  status             SMALLINT     NOT NULL,                  -- 0 = in-flight, 1 = completed
  response_code      SMALLINT,                               -- HTTP status of stored response
  response_body      JSONB,
  response_headers   JSONB,
  created_at         TIMESTAMPTZ  NOT NULL DEFAULT now(),
  completed_at       TIMESTAMPTZ,
  expires_at         TIMESTAMPTZ  NOT NULL,                  -- TTL (e.g. created_at + 24h)
  PRIMARY KEY (api_path, idempotency_key)
);

CREATE INDEX ix_idempotency_expires ON idempotency_key (expires_at);

-- Daily housekeeping:
-- DELETE FROM idempotency_key WHERE expires_at < now();

4. Strategic explanations

Why the composite PK is (api_path, idempotency_key)

A single-column PK on idempotency_key alone would be a silent bug: a client could inadvertently reuse a UUID that happens to match a key from a different endpoint, causing the wrong stored response to be returned. Prefixing with api_path creates independent namespaces at zero cost. The PK also provides the uniqueness enforcement that turns duplicate inserts into a detectable conflict, enabling INSERT ... ON CONFLICT DO NOTHING as the atomic "claim" operation.

Optimistic locking with two-phase status

The canonical algorithm: (1) INSERT with status = 0 (in-flight) — if a conflict is returned, another request owns the key; (2) execute the payment; (3) UPDATE to status = 1 and store the response. A retry that finds an in-flight row should wait briefly and re-poll, or return HTTP 409 with a Retry-After header. This prevents double-execution even under concurrent retry storms without distributed locks.

Request-body hashing and CHAR(64)

A hex-encoded SHA-256 digest is always exactly 64 characters, making CHAR(64) a perfect fit — it wastes no padding bytes and allows an equality check in O(1). On a retry, the server must verify the hash before returning the stored response; mismatches indicate client bugs or misuse and should return HTTP 422, not the stored response.

TTL strategy: expires_at + indexed delete

A common stripe-style convention is 24 hours. The expires_at index enables a daily DELETE FROM idempotency_key WHERE expires_at < now() to run as an index scan — it reads only the pages containing expired rows rather than a full table scan. For very high throughput, partition by day on expires_at so cleanup is a DROP PARTITION rather than a row-by-row delete (zero table bloat).

Dialect notes

In Postgres, INSERT ... ON CONFLICT (api_path, idempotency_key) DO NOTHING is the atomic claim. The application should use READ COMMITTED isolation; the PK uniqueness constraint is sufficient for correctness without serializable transactions. For very high concurrency, a pg_try_advisory_lock on a hash of the key can prevent the thundering-herd of simultaneous retries all reading an in-flight row.

Senior signal: scope idempotency keys per endpoint with a composite PK, use a two-phase in-flight → completed status to prevent concurrent double-execution, hash the request body to detect key reuse with a different payload, and design TTL as an indexed delete — or a partitioned table — so housekeeping never causes a full table scan.
