Amazon's retail-quality team wants the return rate per product category — the share of orders that come back. The shape is a LEFT JOIN from orders to returns plus a ratio, but the interview lives in two details: getting the join direction right so non-returned orders are not dropped, and counting the right thing in numerator and denominator.

1. The source data

orders — one row per order:

categoryorder_idamount
Electronics1100
Electronics2200
Electronics3150
Electronics4300
Apparel560
Apparel680
Books725

returns — one row per returned order: order_id ∈ {1, 3, 5}. So Electronics had 4 orders / 2 returns, Apparel 2 / 1, and Books had 1 order and 0 returns — the category that tests whether the join keeps unmatched rows.

2. The SQL — LEFT JOIN + ratio

SELECT o.category,
  100.0 * COUNT(r.order_id)/COUNT(*) AS return_rate
FROM orders o LEFT JOIN returns r ON o.order_id = r.order_id
GROUP BY o.category;

LEFT JOIN keeps every order; matched ones pick up a non-NULL r.order_id, unmatched ones get NULL there. COUNT(*) counts all orders in the group; COUNT(r.order_id) counts only the rows that matched a return.

3. The resulting analysis

categoryreturn_rate
Electronics50.0000000000000000
Apparel50.0000000000000000
Books0.00000000000000000000

Electronics: 2 returns / 4 orders = 50%. Apparel: 1 / 2 = 50%. Books: 0 / 1 = 0% — and crucially Books still appears in the result with a real rate. An INNER JOIN would have erased Books entirely, silently dropping a 0%-return category from a quality report.

4. Strategic explanations

Why LEFT JOIN, not INNER — the dropped-category trap

An INNER JOIN emits only orders that have a matching return. Books, with zero returns, would produce zero joined rows and vanish from the output — the report would imply Books does not exist, when in fact it has the best return rate. A return-rate analysis must keep the denominator population intact. orders LEFT JOIN returns guarantees every order survives; the absence of a return becomes a NULL, not a missing row. Naming this is the single most important point.

COUNT(*) vs COUNT(column) — numerator and denominator

The ratio works only because the two counts count different things. COUNT(*) counts rows — every order in the group. COUNT(r.order_id) counts only non-NULL values — and after a LEFT JOIN, r.order_id is NULL exactly for orders with no return. So COUNT(r.order_id) is precisely the number of returned orders. Swapping it for COUNT(*) would make every rate 100%; swapping it for COUNT(o.order_id) would too, since o.order_id is never NULL. The numerator must reference a column from the right table.

The integer-division trap — why 100.0 not 100

Both COUNTs return integers. In Postgres integer / integer does integer division: 2/4 would be 0, not 0.5. Multiplying by the numeric literal 100.0 first promotes the whole expression to numeric, so the division is real. Writing 100 * COUNT(...)/COUNT(*) with an integer 100 can still truncate depending on evaluation order — the .0 is load-bearing. Equivalent fixes: COUNT(r.order_id)::numeric or CAST.

The duplicate-returns edge case

The query assumes returns has at most one row per order. If an order can be returned twice (two rows in returns with the same order_id), the LEFT JOIN fans out — that order now contributes two rows, inflating both COUNT(*) and COUNT(r.order_id) and corrupting the rate. Guard against it by counting distinct orders:

SELECT o.category,
  100.0 * COUNT(DISTINCT r.order_id) / COUNT(DISTINCT o.order_id) AS return_rate
FROM orders o LEFT JOIN returns r ON o.order_id = r.order_id
GROUP BY o.category;

Or pre-aggregate returns to one row per order before joining. Asking "can an order be returned more than once?" is the senior clarifying question.

Empty categories and rounding

A category with zero orders simply will not appear — it has no rows in orders to group. There is no divide-by-zero risk here because every group has COUNT(*) ≥ 1. For presentation, wrap the rate in ROUND(…, 2) so the dashboard shows 50.00 rather than the long numeric tail seen above.

Senior signal: justify LEFT JOIN by naming the category that an INNER JOIN would silently drop, explain why the numerator must COUNT a right-table column while the denominator uses COUNT(*), force real division with a numeric literal, and pre-empt join fan-out when returns are not unique per order.
