UBS's retail-analytics team wants a single report that splits each product category into "high-value" and "low-value" orders and shows the average of each side. The reference solution does it with AVG(CASE WHEN …) — conditional aggregation. It is the right pattern, and the interview hinges on understanding what happens to a category that has only high-value orders, or only low.

1. The source data — orders

order_idcategoryorder_total
1books$150
2books$40
3books$90
4toys$200
5toys$300
6garden$60

Deliberately uneven categories. books has orders on both sides of $100. toys has only high-value orders ($200, $300). garden has only one order, and it is low-value. Those one-sided categories are the whole exercise.

2. The SQL — AVG(CASE WHEN …)

SELECT category,
  AVG(CASE WHEN order_total > 100 THEN order_total END) AS avg_high_value,
  AVG(CASE WHEN order_total <= 100 THEN order_total END) AS avg_low_value
FROM orders GROUP BY category;

Each CASE rewrites order_total to itself when the row qualifies, and to NULL otherwise (there is no ELSE). AVG then averages only the non-NULL values — so each column averages just its own side.

3. The resulting analysis

books gets both averages ($150 high, $65 low). toys has no low-value orders so avg_low_value is NULL; garden has no high-value orders so avg_high_value is NULL. The NULL is the correct answer to "what is the average of zero values" — not 0, not an error.

4. Strategic explanations

Why the missing-ELSE NULL is load-bearing

The trick that makes one-row-per-category pivots work is that CASE with no ELSE returns NULL, and AVG ignores NULLs. So AVG(CASE WHEN order_total > 100 THEN order_total END) divides the sum of high-value orders by the count of high-value orders — the non-qualifying rows vanish from both numerator and denominator. Adding ELSE 0 would be a serious bug: the zeros would join the average and drag it down. The absence of ELSE is intentional.

The cleaner Postgres idiom: FILTER

PostgreSQL supports an aggregate FILTER clause that says the same thing more directly:

SELECT category,
       AVG(order_total) FILTER (WHERE order_total > 100)  AS avg_high_value,
       AVG(order_total) FILTER (WHERE order_total <= 100) AS avg_low_value
FROM orders
GROUP BY category;

Identical output. FILTER reads as "average these rows, but only where …" and avoids the easy CASE mistake of an accidental ELSE. The CASE form is the portable one; FILTER is the idiomatic one. Knowing both, and why they match, is the senior signal.

The boundary: where does $100 go?

The predicates are > 100 and <= 100, so an order of exactly $100 is low-value. That is a real product decision, and it must be airtight: the two conditions have to partition the rows — every order in exactly one bucket. If someone wrote >= 100 and <= 100, a $100 order would be counted in both averages; if they wrote > 100 and < 100, it would fall into neither and silently disappear. Confirm which side the boundary belongs to.

The NULL-output trap downstream

The NULLs are correct, but a dashboard may not expect them — a chart could render NULL as a gap, or a later subtraction avg_high_value - avg_low_value would yield NULL for toys and garden. If the consumer needs a number, wrap deliberately: COALESCE(AVG(CASE WHEN …), 0). But pause first — "average of zero high-value orders" is genuinely unknown, and forcing it to 0 can mislead. NULL is often the more honest value.

The unpriced-order edge case and the order_total/amount ambiguity

If order_total can be NULL, that row satisfies neither > 100 nor <= 100 (any comparison with NULL is unknown), so it is excluded from both averages — usually fine, but say it. Also: the schema ships both order_total and amount. They may be pre- vs post-discount values that disagree. Don't guess which one defines "value" — confirm it and note the assumption.

Senior signal: explain that the missing ELSE is what makes conditional AVG correct (and that ELSE 0 would corrupt it), offer FILTER as the idiomatic equivalent, guarantee the two predicates partition the rows so the $100 boundary lands in exactly one bucket, and treat the empty-bucket NULL as a deliberate "unknown" rather than papering over it with a zero.
