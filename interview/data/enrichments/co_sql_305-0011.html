Samsung's payroll team is exporting a compensation file, and the bonus column is sparse — employees without a bonus this cycle have NULL, not 0. A downstream system that adds salary and bonus would produce NULL totals for those people. COALESCE is the one-function fix. The interview question underneath: do you understand the difference between a NULL that means "zero" and a NULL that means "unknown"?

1. The source data — employees

emp_idnamebonus
1Ada$5,000
2BenNULL
3Cara$0
4DanNULL

Four employees. Two have NULL bonuses (Ben, Dan), one has a genuine $0 bonus (Cara). The NULL-vs-zero distinction is the whole interview: Cara already has a value; Ben and Dan have no value — and whether "no value" should become 0 is a business decision, not a default.

2. The SQL — COALESCE(bonus, 0)

SELECT emp_id, name, COALESCE(bonus, 0) AS bonus
FROM employees;

COALESCE returns its first non-NULL argument. For each row, if bonus is non-NULL it passes through unchanged; if bonus is NULL, the function returns 0 instead.

3. The resulting analysis

emp_idnamebonus
1Ada5000
2Ben0
3Cara0
4Dan0

All four rows now carry a number — no NULLs to break downstream arithmetic. But look at Ben and Cara: both show 0, yet they arrived there differently. Cara's 0 was real data; Ben's 0 was manufactured by COALESCE. The output column has lost the ability to tell those two cases apart — a deliberate trade-off worth naming.

4. Strategic explanations

Why NULL breaks arithmetic in the first place

NULL is not a value; it is the absence of one. Any arithmetic touching it yields NULL: 50000 + NULL = NULL, not 50000. So a total-comp column computed as salary + bonus would be NULL for every employee without a bonus — the report would show blank pay for real employees. COALESCE(bonus, 0) substitutes a neutral value so the addition behaves:

SELECT emp_id, name, salary + COALESCE(bonus, 0) AS total_comp
FROM employees;
The semantic question: does NULL bonus mean zero?

Replacing NULL with 0 is correct only if "no bonus recorded" genuinely means "received nothing". For bonuses that is usually true. But consider COALESCE(rating, 0) on a performance score — a missing rating does not mean a score of zero; it means "not yet reviewed", and zeroing it would slander the employee. The senior instinct is to ask: "is a NULL here a real zero, or genuinely unknown?" If unknown, leaving the NULL — or surfacing a label like 'not reviewed' — is more honest than a fake number.

COALESCE vs ISNULL / NVL — portability

COALESCE is ANSI SQL and works in every engine; it also takes any number of arguments and returns the first non-NULL: COALESCE(bonus, dept_default_bonus, 0). The vendor-specific equivalents — SQL Server's ISNULL, Oracle's NVL, MySQL's IFNULL — are two-argument only and not portable. Always reach for COALESCE. One numeric subtlety: COALESCE resolves to a single result type, so COALESCE(int_col, 0.0) may promote the whole column to numeric — keep the fallback's type consistent with the column.

The trap: COALESCE is not NULLIF, and = NULL never works

Two adjacent mistakes. First, NULLIF(a, b) is the opposite tool — it turns a value into NULL when a = b (handy for NULLIF(denominator, 0) to dodge divide-by-zero). Don't confuse the two. Second, you cannot filter for NULLs with WHERE bonus = NULL — that expression is itself NULL and matches nothing. NULL comparisons need the dedicated predicate:

-- find the employees whose bonus was missing
SELECT emp_id, name FROM employees WHERE bonus IS NULL;     -- correct
-- WHERE bonus = NULL  -- returns ZERO rows, always
Senior signal: explain that NULL poisons arithmetic so COALESCE is needed before any + or SUM, but pause on whether a NULL truly means zero or means "unknown" — zeroing an unknown rating is a bug — prefer portable COALESCE over NVL/ISNULL, and never test NULLs with =.
