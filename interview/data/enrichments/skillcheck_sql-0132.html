A growth team is auditing a user table before launching email campaigns and needs to flag addresses that resolve to the same inbox. Email is case-insensitive in the local-part for practical purposes and entirely case-insensitive in the domain, so A@x.com and a@x.com are one person. The interviewer wants to see you normalise before you group — not group raw text and miss the duplicates.

1. The source data — users

user_idemail(narrative)
1a@x.comlowercase
2A@X.comsame inbox as user 1, different casing
3b@x.comunique — appears once
4c@x.compart of a 3-way duplicate
5C@X.COMsame inbox, all caps
6C@x.comsame inbox, mixed

Three casings of c@x.com exercise a group bigger than two, and b@x.com is the single-occurrence row that must be excluded. Group on raw email and you would wrongly report zero duplicates.

2. The SQL — normalise, group, filter with HAVING

SELECT LOWER(email) AS email_lower, COUNT(*)::int AS occurrences
FROM users
GROUP BY LOWER(email)
HAVING COUNT(*) > 1
ORDER BY email_lower;

LOWER(email) folds every casing to one canonical key, GROUP BY on that expression buckets the variants together, COUNT(*) tallies each bucket, and HAVING COUNT(*) > 1 keeps only buckets with a genuine collision.

3. The resulting analysis

email_loweroccurrencessignal
a@x.com2a@x.com + A@X.com folded together
c@x.com3three casings collapse into one key

b@x.com is absent — one occurrence fails HAVING COUNT(*) > 1. The counts (2 and 3) are only correct because the grouping key was lower-cased; on raw text every row would be its own group.

4. Strategic explanations

WHERE vs HAVING — filtering before vs after the aggregate

WHERE filters individual rows before grouping; HAVING filters whole groups after the aggregate is computed. "Appears two or more times" is a property of the group, not of any single row, so it can only live in HAVING. A common mistake — WHERE COUNT(*) > 1 — is a syntax error, because the count does not exist yet at WHERE time.

Group on the expression, and select the same expression

PostgreSQL lets you GROUP BY LOWER(email) and select LOWER(email) — the grouped expression is a legal output. Selecting the raw email instead would fail ("must appear in GROUP BY"), because within one lower-cased bucket the original column holds several distinct values. The canonical key is the only thing that is single-valued per group.

Case folding has limits — collation and Unicode

LOWER() handles ASCII cleanly but is locale-sensitive for accented or non-Latin characters, and it does not trim whitespace or strip Gmail-style +tags / dots. If the real goal is "same deliverable inbox" you may also need TRIM() and provider-specific normalisation. A case-insensitive COLLATE or CITEXT column is the cleaner long-term fix — mention it.

The ::int cast is cosmetic but intentional

COUNT(*) returns bigint in PostgreSQL. Casting to int keeps the API contract tidy and the JSON payload small; it does not change correctness. Know why a cast is there — here it is presentation, not logic — so you can defend or drop it deliberately.

Dialect — case-insensitivity without LOWER()

SQL Server's default collations are case-insensitive, so a raw GROUP BY email would already fold cases there — portable code should still call LOWER() so behaviour does not depend on server collation. MySQL behaves similarly under _ci collations. PostgreSQL is case-sensitive by default, which is why the explicit LOWER() is mandatory here.

Senior signal: normalise the grouping key with LOWER() before GROUP BY so case variants collapse, use HAVING — never WHERE — for the "count > 1" group predicate, select exactly the grouped expression, and call out that true inbox-equality may also need trimming, Unicode-aware collation, or a CITEXT column.
