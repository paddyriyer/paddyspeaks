Meta runs a nightly maintenance job over its ad inventory: any ad that has not been active in the last 30 days should be flipped from active to inactive, so stale campaigns stop showing in advertiser dashboards. It is a routine UPDATE. The interview value is in the predicate — getting the date arithmetic, the status guard, and the idempotency right so the job is safe to run every night.

1. The source data — ads

Sample rows, with last_active_date shown as days before today:

1active5 days ago
2active45 days ago
3active60 days ago
4inactive120 days ago

6active90 days ago

Ad 4 is the edge case: it is stale (120 days) but already inactive — the job should leave it alone, not "re-deactivate" it. Ads 1 and 5 are recent and stay active; ads 2, 3, 6 are the ones that flip.

2. The SQL — conditional UPDATE

UPDATE ads SET status = 'inactive'
WHERE last_active_date < CURRENT_DATE - INTERVAL '30 days'
  AND status = 'active';

Two predicates joined by AND: the row must be older than the 30-day cutoff and currently active. The status check is what keeps the job from touching ads that are already inactive.

Ads 2, 3, and 6 flipped to inactive. Ads 1 and 5 stayed active — inside the 30-day window. Ad 4 stayed inactive and, crucially, was not rewritten: the status = 'active' predicate excluded it from the update entirely.

4. Strategic explanations

Why the status = 'active' guard matters — idempotency

Without AND status = 'active', every stale row gets a physical write each night, including the thousands already inactive. The end-state is identical, but each no-op write produces a dead tuple and WAL traffic, bloating the table and the replication stream. The guard makes the job idempotent in cost: a second run touches zero rows. On a billion-row ad table at Meta scale, that is the difference between a cheap job and a vacuum problem.

The boundary — what does "30 days" mean exactly?

last_active_date < CURRENT_DATE - INTERVAL '30 days' uses a strict <, so an ad last active exactly 30 days ago is not flipped — it is treated as still inside the window. Whether the rule is "more than 30 days" or "30 or more days" is a product decision; switch to <= if the latter. Always pin this down rather than guessing.

DATE vs TIMESTAMP — a subtle correctness issue

Here last_active_date is a DATE, so the comparison is clean. If it were a TIMESTAMP, CURRENT_DATE - INTERVAL '30 days' is midnight 30 days ago, and an ad active at 14:00 that day would compare as older than midnight — an off-by-part-of-a-day error. With timestamps, compare against now() - INTERVAL '30 days' instead, and be explicit about the time component.

Use a relative cutoff, never a hardcoded date

CURRENT_DATE - INTERVAL '30 days' recomputes the window every run, so the job is correct whenever it fires. Hardcoding a literal like '2026-04-15' would silently rot — the window would freeze while the calendar moves on. For a nightly job, the cutoff must be relative to CURRENT_DATE.

Auditability — the follow-up an interviewer expects

A destructive bulk UPDATE with no record of what changed is risky in production. Two senior moves: preview first with the matching SELECT, and use RETURNING to capture exactly which ads were flipped for an audit log:

UPDATE ads SET status = 'inactive'
WHERE last_active_date < CURRENT_DATE - INTERVAL '30 days'
  AND status = 'active'
RETURNING ad_id, last_active_date;

A real pipeline would also stamp a deactivated_at column so the transition is traceable, not just inferred.

Senior signal: keep the status = 'active' guard so the job is idempotent and does not bloat the table with no-op writes, use a relative CURRENT_DATE cutoff rather than a literal, treat the strict < boundary and any DATE/TIMESTAMP mismatch as explicit correctness decisions, and reach for RETURNING to make the bulk change auditable.
