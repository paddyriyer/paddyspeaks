A streaming-service analyst wants a per-month top-3 leaderboard of viewers by total watch time — the kind of feed that powers a "your top fans this month" panel. The interviewer is probing a partitioned top-N: you must sum minutes per user per month, rank within each month, and keep three per month, with a deterministic tie-break when totals collide.

1. The source data — watch

Two engineered ties. In January users 1 and 2 both total 120. In February users 2, 5, 6 all total 100 for the 2nd/3rd/4th slots — only two of them fit in the top 3, and the rule "smaller user_id wins" must decide which. User 7 at 99 sits just outside.

2. The SQL — sum per user-month, rank within month, keep rank ≤ 3

WITH monthly AS (
  SELECT DATE_TRUNC('month', started_at)::date AS month_start,
         user_id,
         SUM(watched_minutes) AS watched_minutes
  FROM watch
  WHERE started_at >= TIMESTAMP '2025-01-01' AND started_at < TIMESTAMP '2026-01-01'
  GROUP BY 1, 2
),
ranked AS (
  SELECT month_start, user_id, watched_minutes,
         ROW_NUMBER() OVER (PARTITION BY month_start ORDER BY watched_minutes DESC, user_id) AS rn
  FROM monthly
)
SELECT month_start, user_id, watched_minutes::int AS watched_minutes
FROM ranked
WHERE rn <= 3
ORDER BY month_start, watched_minutes DESC, user_id;

monthly totals each user's minutes per month. ranked assigns a row number within each month, ordered by minutes descending then user_id ascending. WHERE rn <= 3 keeps the top three of every month.

February's top 3 are users 1, 2, 5. Users 2, 5, 6 all total 100, but the user_id tie-break orders them 2 < 5 < 6, so user 6 is bumped to rank 4 and dropped. User 7's 99 was never in contention. The tie-break is decisive, not cosmetic.

4. Strategic explanations

Partitioned top-N — the canonical pattern

"Top 3 per month" is rank-within-group-then-filter. ROW_NUMBER() OVER (PARTITION BY month_start ORDER BY watched_minutes DESC, user_id) restarts the counter for each month and orders viewers by total; WHERE rn <= 3 slices the top of each partition. The filter must be in an outer query (or the window in a CTE) because you cannot reference a window alias in the same WHERE — window functions are computed after WHERE.

ROW_NUMBER vs RANK — the tie behaviour changes the row count

This is the crux. With February's three-way tie at 100, ROW_NUMBER assigns 2, 3, 4 (broken by user_id) so rn <= 3 yields exactly 3 rows. RANK would assign 2, 2, 2 to all three — all ≤ 3 — producing four rows for February, contradicting "top 3." The spec's "prefer smaller user_id" tie-break explicitly demands a strict ordering, which is precisely what ROW_NUMBER with a tiebreaker column delivers. If the business genuinely wanted "include everyone tied for 3rd," RANK would be right — but then row counts vary by month.

The tie-break is the secondary ORDER BY key

ORDER BY watched_minutes DESC, user_id encodes the rule directly: equal totals are resolved by ascending user_id. Without that second key, the order among tied users is arbitrary and non-reproducible — the query could return user 5 or user 6 on different runs. A top-N with possible ties must always carry a deterministic tiebreaker, or the result is unstable.

The year filter and the redundant-but-honest GROUP BY

started_at >= '2025-01-01' AND started_at < '2026-01-01' is a half-open interval over a TIMESTAMP — correct, because BETWEEN ... '2025-12-31' would drop the final day's evening sessions. GROUP BY 1, 2 references the SELECT positions (month_start, user_id); grouping by the truncated month plus user is what defines the per-user-per-month total. The data is all in 2025, so the filter changes nothing here — but it makes the query correct for a multi-year table.

Dialect notes

DATE_TRUNC('month', ...) is PostgreSQL; MySQL uses DATE_FORMAT(started_at,'%Y-%m-01'), SQL Server DATEFROMPARTS(YEAR(started_at),MONTH(started_at),1), BigQuery DATE_TRUNC(started_at, MONTH). ROW_NUMBER() OVER (...) and the rn <= 3 filter are standard. SQL Server also offers TOP ... WITH TIES, but that applies to a whole result, not per-partition — the window-function form is the portable per-group top-N.

Senior signal: implement per-group top-N with ROW_NUMBER() OVER (PARTITION BY group ORDER BY metric DESC, tiebreak) filtered to rn <= N in an outer query — choose ROW_NUMBER over RANK so a tie at the cutoff yields exactly N rows, and always carry a tie-break column so the leaderboard is deterministic.
