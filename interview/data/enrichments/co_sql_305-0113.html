The Trust & Safety team at a marketplace got a complaint: "this seller keeps changing the price on the same item — bait-and-switch." They want a list of products whose price has changed more than 5 times. The query is one line; the interesting part is what counts as a "change" and what to do once flagged.

2. The SQL — GROUP BY + HAVING on aggregate count

SELECT product_id, COUNT(*) AS change_count
FROM price_history
GROUP BY product_id
HAVING COUNT(*) > 5;

4. Strategic explanations

Why HAVING not WHERE

WHERE COUNT(*) > 5 doesn't work — there's no aggregation at the WHERE phase. HAVING filters AFTER the GROUP BY produces aggregated rows. Standard interview-answer phrasing: "WHERE filters rows; HAVING filters groups."

The "5" is arbitrary — make it data-driven

Why 5? Probably because someone said "more than five feels frequent" in a meeting. The data-driven version compares each product to the population's distribution and flags genuine outliers:

WITH stats AS (
  SELECT AVG(c)    AS mu,
         STDDEV(c) AS sigma
  FROM (SELECT COUNT(*) AS c FROM price_history GROUP BY product_id) t
)
SELECT p.product_id, COUNT(*) AS change_count
FROM price_history p, stats
GROUP BY p.product_id, mu, sigma
HAVING COUNT(*) > mu + 2 * sigma;
-- Flags products whose change count is >2 standard deviations above the mean.

Senior signal: don't hardcode magic numbers when the data can pick them.

Time-bounded version — "5 changes in 7 days"

Most price-manipulation patterns happen in burst windows, not lifetime. The actionable version layers a date filter:

SELECT product_id, COUNT(*) AS changes_last_week
FROM price_history
WHERE change_date >= CURRENT_DATE - INTERVAL '7 days'
GROUP BY product_id
HAVING COUNT(*) > 5;
Distinct-day vs total-change distinction

What if a seller changed the price 7 times in one day but 0 times the rest of the week? That's still suspicious but a different pattern. Use COUNT(DISTINCT change_date) to require changes on different days:

HAVING COUNT(DISTINCT change_date) > 5
The "magnitude" follow-up — flagging price swings, not just changes

5 small price tweaks (±$0.50) is normal merchandising. 5 huge price swings (50% up, 30% down, ...) is bait-and-switch. The audit-grade query joins to the actual prices:

WITH change_swings AS (
  SELECT product_id, change_date, price,
    LAG(price) OVER (PARTITION BY product_id ORDER BY change_date) AS prev_price
  FROM price_history
)
SELECT product_id,
  COUNT(*) AS total_changes,
  ROUND(AVG(ABS(price - prev_price) / NULLIF(prev_price, 0)) * 100, 1) AS avg_pct_swing
FROM change_swings
WHERE prev_price IS NOT NULL
GROUP BY product_id
HAVING COUNT(*) > 5
   AND AVG(ABS(price - prev_price) / NULLIF(prev_price, 0)) > 0.10;
-- Products with 5+ changes AND average swing > 10% — actionable signal.
Senior signal: articulate WHERE-vs-HAVING upfront, refuse the magic number ("5") and propose the σ-based variant, layer a time-window filter, and pre-empt the magnitude question with the LAG-based swing calculation. The σ-based threshold is the over-the-bar move — Trust & Safety teams actually use it.
