An org-chart tool needs to enumerate every ancestor–descendant pair in the management hierarchy along with the number of hops between them — not just immediate manager–report links but transitive ones at any depth. The interviewer is probing whether you can write a recursive CTE that walks the tree upward from every employee, correctly seeds the base case, and accumulates the hop counter without duplicating or losing chains.

1. The source data — employee

Tree structure: Alice (1) → Bob (2) → Dan (4) → Erin (5); Alice (1) → Carla (3). Alice is the root (NULL manager_id). Every non-root employee has exactly one chain of ancestors up to Alice.

2. The SQL — recursive CTE walking upward

WITH RECURSIVE chains AS (
  SELECT manager_id AS ancestor_id, emp_id AS descendant_id, 1 AS hops
  FROM employee
  WHERE manager_id IS NOT NULL
  UNION ALL
  SELECT e.manager_id, c.descendant_id, c.hops + 1
  FROM chains c
  JOIN employee e ON e.emp_id = c.ancestor_id
  WHERE e.manager_id IS NOT NULL
)
SELECT ancestor_id, descendant_id, hops
FROM chains
ORDER BY ancestor_id, descendant_id;

The base case seeds one row per direct manager–report edge. Each recursive step climbs one level: it takes the current ancestor and looks up that employee's manager, extending the chain by 1 hop. Recursion stops when the current ancestor is the root (manager_id IS NULL).

4. Strategic explanations

Base case design — edges, not nodes

The base case starts with every direct manager–report edge (one row per non-NULL manager_id). It does not start with every node — that would require a dummy self-pair (0 hops) to bootstrap the walk, and the question explicitly excludes (X, X) pairs. Starting with edges and hop = 1 is cleaner and directly matches "immediate (hops=1)" in the spec.

Recursive step — climbing the tree

In each iteration, chains holds paths whose upper end is some ancestor. The recursive SELECT joins employee e on e.emp_id = c.ancestor_id to find that ancestor's own manager. If that manager is non-NULL, a new row extends the path: ancestor becomes the grandparent, descendant stays the same, hops increments by 1. When the current ancestor is Alice (emp_id = 1, manager_id NULL), the WHERE clause e.manager_id IS NOT NULL drops that row, terminating the recursion for paths that have reached the root.

Correctness check — no self-pairs, no duplicates

The base case uses direct edges (manager_id ≠ emp_id by construction in a valid org tree). The recursive step always moves upward (toward the root), and the tree is acyclic, so no chain can revisit a node. The result set contains exactly one row per (ancestor, descendant) pair: no duplicate paths exist because each distinct (ancestor_id, descendant_id) combination is produced by exactly one root-to-descendant walk.

Performance — depth and breadth

The recursion depth equals the depth of the deepest chain in the tree (4 hops for Alice→Erin in this example). Total rows produced is O(D × E) where D is tree depth and E is employee count — in balanced trees this is O(N log N). For wide, flat hierarchies (few levels) this is fast. PostgreSQL enforces a default recursion limit of 100 levels; for very deep trees, set SET max_recursive_iterations = .... An index on employee(emp_id) is critical for the join in the recursive step.

Dialect notes

PostgreSQL uses WITH RECURSIVE. SQL Server uses WITH ... AS (... UNION ALL ...) without the RECURSIVE keyword. Oracle and MySQL 8+ use WITH RECURSIVE. BigQuery supports recursive CTEs since 2022 with WITH RECURSIVE. Snowflake supports recursive CTEs. The WHERE termination condition is identical across dialects. An alternative without recursive CTEs (available in all engines) is a stored procedure or application-side BFS, but recursive CTEs are the canonical single-query answer.

Senior signal: seed the recursive CTE with direct edges (hops=1) rather than self-pairs (hops=0) to avoid (X,X) rows naturally; terminate by filtering WHERE e.manager_id IS NOT NULL on the ancestor; and explain that tree acyclicity guarantees no duplicate paths, but that you would add a depth guard or cycle detection (CYCLE emp_id SET is_cycle in PostgreSQL 14+) in production to be safe.
