Distributed traces decompose request latency by service. Per-service span aggregation pinpoints which downstream is slow.

2. The SQL — per-service span stats

SELECT service_name, COUNT(*) AS spans,
  ROUND(AVG(duration_ms), 1) AS avg_ms, MAX(duration_ms) AS max_ms,
  COUNT(DISTINCT trace_id) AS traces_touched,
  ROUND(SUM(duration_ms) * 1.0 / NULLIF(COUNT(DISTINCT trace_id), 0), 1) AS avg_total_per_trace_ms
FROM spans GROUP BY service_name ORDER BY avg_ms DESC;

3. Strategic explanations

Avg vs P95 vs P99

Average hides tail. The metric SREs actually monitor: P99 latency. SQLite-friendly approximation:

WITH ranked AS (SELECT *, NTILE(100) OVER (PARTITION BY service_name ORDER BY duration_ms) AS pct FROM spans)
SELECT service_name, MAX(duration_ms) AS p99 FROM ranked WHERE pct <= 99 GROUP BY service_name;
Total time per trace

Avg span duration × spans-per-trace = wall-clock per request. The customer-facing number. Different from per-span latency.

Critical-path filtering

Many spans run in parallel; only the longest blocking path matters. Production filters to spans_on_critical_path before computing latency.

Service-mesh vs business-logic

50% of latency is often network / proxy / TLS. Splitting by span-kind (server / client / internal) reveals where to optimize.

The error-correlated latency

Failed spans often have very different latency than successful ones (timeouts vs fast errors). Filter or split.

Senior signal: P99 over avg, total-per-trace as the customer-facing number, filter to critical-path for optimization targeting, split by span-kind to isolate network from logic, and separate error-spans from success.
