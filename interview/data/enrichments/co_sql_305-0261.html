BMW's connected-services team runs a subscription business for in-car features and wants a running total of subscribers over time: each period's net change, plus the cumulative count. The reference solution reaches for a window function — SUM() OVER (ORDER BY …) — which is exactly right. The interview is about whether you understand the implicit window frame, and whether the cumulative total is even meaningful without a starting balance.

1. The source data — subscriptions

periodnew_subscancelled_subs
2025-01-01500120
2025-02-01640200
2025-03-01300350
2025-04-01450180

March is the interesting period: 300 new but 350 cancelled, a negative net of −50. A running total has to absorb a down month gracefully — the cumulative line should dip, not break.

2. The SQL — cumulative SUM() OVER

SELECT period, new_subs - cancelled_subs AS net,
  SUM(new_subs - cancelled_subs) OVER (ORDER BY period) AS cumulative_subs
FROM subscriptions;

net is the per-period change. The window function SUM(…) OVER (ORDER BY period) accumulates that net from the first period up to the current row — a running total.

January nets +380. February's +440 lifts the running total to 820. March's −50 correctly drags it down to 770 — the window SUM handles the negative period naturally. April recovers to 1040.

4. Strategic explanations

The implicit window frame — the detail interviewers wait for

SUM(…) OVER (ORDER BY period) with no explicit frame defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. That is what makes it a running total. The trap is RANGE vs ROWS: with RANGE, all rows tied on period are summed together into the same cumulative value. Periods are unique here so it doesn't bite — but if two rows shared a period, you'd want the explicit, unambiguous frame:

SUM(new_subs - cancelled_subs)
  OVER (ORDER BY period ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)

Naming the implicit frame and knowing the RANGE/ROWS distinction is the senior signal here.

The cumulative total assumes the book started at zero

The running total begins at January's 380 — it implicitly assumes there were no subscribers before January. If the business existed before the first row in this table, cumulative_subs is not the true subscriber count; it is the net change since the window opened. To get an absolute count, add the opening balance:

SELECT period,
       :opening_balance + SUM(new_subs - cancelled_subs) OVER (ORDER BY period) AS total_subs
FROM subscriptions;

Confirm whether the data is the full history or a slice — it changes whether the number is a count or a delta.

Window function vs self-join — why OVER wins

The pre-window way to do a running total is a correlated self-join: for each period, sum the net of every period up to it. That is O(n²) and verbose. SUM() OVER (ORDER BY period) sorts once and accumulates in a single pass — O(n log n), and far clearer. On a real subscription history the window function is the only sensible choice.

No ORDER BY on the outer query

The window's ORDER BY period controls accumulation order, but it does not guarantee the output rows come back sorted. The rows happen to be ordered here, but to be safe add an explicit final ORDER BY period so the cumulative column reads top-to-bottom.

Edge cases: gaps in periods and the schema mismatch

The window sums consecutive rows, not consecutive months. If a period is missing from the table, the running total simply skips it — the cumulative value is still correct, but a chart will draw a flat segment across the gap. Separately, the question's schema lists per-subscriber columns (subscription_id, customer_id, plan, start_date, …) alongside the per-period new_subs/cancelled_subs; the solution correctly uses only the period-grain columns. If the table were actually per-subscriber, you'd first aggregate to a period grain before the running total.

Senior signal: name the implicit UNBOUNDED PRECEDING … CURRENT ROW frame and the RANGE/ROWS distinction, recognise that cumulative_subs is a delta-since-window-open unless an opening balance is added, prefer SUM() OVER to an O(n²) self-join, and add a final ORDER BY so the running total renders in sequence.
