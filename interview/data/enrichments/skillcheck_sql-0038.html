An IoT pipeline ingests sensor readings where a downed sensor records NULL. The team wants LOCF — last observation carried forward — so each gap inherits the most recent real reading. The interviewer is probing whether you can do gap-fill without IGNORE NULLS (unsupported in older Postgres, MySQL 8, older BigQuery), using the running-count grouping trick.

1. The source data — sensor_readings

Sensor 2's edge case: its first two rows are NULL with no prior reading at all. Correct LOCF leaves them NULL — there is nothing to carry forward yet.

2. The SQL — running-count groups, then MAX per group

WITH grouped AS (
  SELECT ts, sensor_id, value,
         COUNT(value) OVER (PARTITION BY sensor_id ORDER BY ts
                            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS grp
  FROM sensor_readings
)
SELECT ts, sensor_id, value,
       MAX(value) OVER (PARTITION BY sensor_id, grp) AS filled
FROM grouped
ORDER BY sensor_id, ts;

COUNT(value) ignores NULLs, so as the per-sensor stream advances it increments only on a real reading and stays flat across NULLs. Every NULL therefore inherits the group number of the last non-NULL before it. MAX(value) OVER (PARTITION BY sensor_id, grp) then broadcasts that group's single real value across all its rows.

Sensor 2's leading NULLs stay NULL — they fall in grp = 0, a group with no real value, so its MAX is NULL. That is the correct LOCF behaviour: you cannot carry forward what does not yet exist.

4. Strategic explanations

Why the running COUNT builds the fill groups

COUNT(value) — argument form, not COUNT(*) — counts only non-NULL values. As a running window it ticks up by 1 at each real reading and holds steady over NULLs. The consequence: a real reading and every NULL after it (until the next reading) share one grp number. Each group thus contains exactly one non-NULL value plus the NULLs that should inherit it.

Why MAX broadcasts the value

Within a group there is precisely one non-NULL value. MAX(value) OVER (PARTITION BY sensor_id, grp) finds it — MIN or SUM would work identically since there is only one value to aggregate; MAX is conventional. The window broadcasts that single value to every row in the group, which is the carry-forward. No ORDER BY is needed inside this second window: it covers the whole group as one block.

Why the leading-NULL group is 0 — and why that is correct

Before any real reading, the running COUNT(value) is 0. Sensor 2's first two NULLs both sit in grp = 0, whose only members are NULLs, so MAX returns NULL. LOCF is undefined when there is no prior observation, and the trick yields NULL automatically — no special-casing required. If the business wanted a backward-fill (carry the first future reading), you would reverse the sort or add a second COALESCE pass.

Why not LAG

LAG(value) only reaches back one row, so it fails across consecutive NULLs — sensor 1's 04:00 row would see the 03:00 NULL, not 12. A correlated subquery (MAX(value) WHERE ts < this.ts AND value IS NOT NULL) is correct but O(n²). The count-group pattern sorts once per sensor — O(n log n) — and fills arbitrarily long gaps in a single pass.

Dialect notes

Postgres 16+, BigQuery, Snowflake, Oracle, and DuckDB support the direct form LAST_VALUE(value IGNORE NULLS) OVER (PARTITION BY sensor_id ORDER BY ts ROWS UNBOUNDED PRECEDING) — one line, very readable. The COUNT-group trick shown here is the portable fallback that works on every engine with window functions, including older Postgres and MySQL 8, which lack IGNORE NULLS.

Senior signal: implement LOCF without IGNORE NULLS by using a running COUNT(value) — which skips NULLs — to tag each NULL with the group of its last real reading, then MAX(value) OVER (PARTITION BY key, grp) to broadcast it; the leading-NULL group 0 naturally stays NULL, which is correct.
