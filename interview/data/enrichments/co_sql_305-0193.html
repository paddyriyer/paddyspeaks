Meta's messaging team wants a per-sender activity profile: for each user who sends messages, how many messages they sent and how many distinct people they messaged. The gap between those two numbers — total sent vs unique recipients — tells you whether someone broadcasts to many contacts or hammers one. The query is a GROUP BY with a plain count and a DISTINCT count, and the interviewer is watching whether you understand exactly what each count includes.

1. The source data — messages

Sender 1 sent 3 messages but to only 2 distinct people — they messaged user 2 twice. That repeat is the whole point: it is the one row of data where COUNT(*) and COUNT(DISTINCT receiver_id) diverge.

2. The SQL — COUNT vs COUNT DISTINCT

SELECT sender_id, COUNT(*) AS sent, COUNT(DISTINCT receiver_id) AS unique_recipients
FROM messages GROUP BY sender_id;

One row per sender. COUNT(*) tallies every message they sent; COUNT(DISTINCT receiver_id) tallies how many different people received at least one.

4. Strategic explanations

COUNT(*) vs COUNT(DISTINCT col) — rows vs distinct values

COUNT(*) counts every row in the group, NULLs included. COUNT(DISTINCT receiver_id) counts the number of distinct non-NULL values. Two consequences worth saying aloud: DISTINCT deduplicates (sender 1's repeat receiver counts once), and DISTINCT drops NULLs (a message with no receiver_id would inflate sent but not unique_recipients). The two counts being equal means "every message went to a different person"; the two diverging means "someone got messaged more than once".

The interviewer's follow-up: senders who never receive

This query profiles only people who appear as a sender. A user who only ever receives messages produces no sender_id row and is invisible. To profile every user — including pure receivers with 0 sent — you cannot GROUP BY sender_id alone; you need a user list left-joined to messages:

SELECT u.user_id,
       COUNT(m.sent_at)               AS sent,
       COUNT(DISTINCT m.receiver_id)  AS unique_recipients
FROM users u
LEFT JOIN messages m ON m.sender_id = u.user_id
GROUP BY u.user_id;

Note COUNT(m.sent_at), not COUNT(*): after a LEFT JOIN, a user with no messages still yields one row with NULLs, and COUNT(*) would wrongly score them 1. COUNT(a_nullable_column) correctly returns 0. That swap is a classic outer-join trap.

Receiving activity — the symmetric metric

"Messaging activity" often means both directions. The mirror query groups by receiver_id for messages received and distinct senders. To get a single row per user with all four numbers, aggregate each direction separately and full-join, or UNION ALL the two roles and group:

SELECT user_id,
       SUM(is_sender)   AS sent,
       SUM(is_receiver) AS received
FROM (
  SELECT sender_id   AS user_id, 1 AS is_sender, 0 AS is_receiver FROM messages
  UNION ALL
  SELECT receiver_id AS user_id, 0,             1                FROM messages
) t
GROUP BY user_id;
Self-messages and time scope

Should a message where sender_id = receiver_id (a note-to-self) count as a "unique recipient"? Often not — WHERE sender_id <> receiver_id filters them. And "messaging activity" usually has an implied window (last 30 days, this quarter); the reference counts all history. Pin the time scope down with a WHERE sent_at >= ... rather than silently reporting lifetime totals.

Edge case: NULL receiver_id or NULL sender_id

A message with a NULL sender_id forms its own NULL group — a meaningless "sender". A message with a NULL receiver_id still increments sent (via COUNT(*)) but not unique_recipients (DISTINCT skips NULL), so the two counts drift apart for a reason that has nothing to do with behaviour. If NULLs are possible, filter them or decide explicitly how a "no recipient" message should score.

Senior signal: articulate that COUNT(*) counts rows while COUNT(DISTINCT receiver_id) counts distinct non-NULL values, recognise that grouping by sender_id excludes pure receivers and that the fix needs a LEFT JOIN with COUNT(nullable_col) instead of COUNT(*), and raise self-messages and the implied time window before reporting lifetime totals.
