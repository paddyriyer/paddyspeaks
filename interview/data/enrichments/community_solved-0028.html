A Community finance dashboard tracks revenue month over month, and the headline tile is the percentage change versus the prior month. It looks trivial — aggregate by month, compare to the row above — but the interviewer is watching three things: do you order the months correctly, do you handle the very first month that has nothing to compare against, and do you avoid integer-division truncation on the percentage.

1. The source data — sales_transactions

2023-01-05100
2023-01-2050
2023-02-10300

2023-03-2830

Five raw transactions roll up to three months: January 150, February 300, March 300. January is the edge case — it is the earliest month, so it has no previous month and its percentage diff must be NULL, not zero. March deliberately ties February at 300 so the diff lands on exactly 0.

2. The SQL — LAG over monthly sums

WITH t1 AS (
  SELECT to_char(created_at, 'YYYY-MM') AS yymm,
         SUM(value) AS value_current,
         LAG(SUM(value)) OVER (ORDER BY to_char(created_at, 'YYYY-MM') ASC) AS previous_month
  FROM sales_transactions
  GROUP BY to_char(created_at, 'YYYY-MM')
)
SELECT yymm AS month,
       ROUND(CAST(((1.0*value_current-previous_month)/previous_month)*100 AS numeric), 2) AS percentage_diff
FROM t1;

The CTE groups transactions into YYYY-MM buckets, sums each, and uses LAG to pull the previous bucket's sum onto the current row. The outer query turns the two numbers into a percentage.

3. The resulting analysis

January is NULL — LAG returns NULL on the first row, and any arithmetic touching NULL stays NULL. February jumps from 150 to 300, a clean +100.00%. March holds at 300, so (300 − 300) / 300 = 0.00. The NULL is correct behaviour, not a bug to paper over.

4. Strategic explanations

Why 1.0 * appears in the numerator

If value is an integer column, (value_current - previous_month) / previous_month is integer division: 150/300 truncates to 0, and the whole percentage collapses to zero. Multiplying by 1.0 promotes the expression to numeric before the divide, so fractional changes survive. This is the single most common silent bug in percentage-diff queries — always force the type.

The ordering trap — why to_char is safe here

LAG is only as correct as its ORDER BY. The solution orders by the YYYY-MM string, which sorts identically to chronological order because the format is zero-padded and big-endian. Order by a month name or a non-padded value and "April" would sort before "January". Safer still is to order by a real date so the intent is unambiguous:

... LAG(SUM(value)) OVER (ORDER BY date_trunc('month', created_at)) ...
Missing months break the "previous month" assumption

LAG returns the previous row, not the previous calendar month. If March had no sales, the table jumps from February straight to April, and April's "previous month" would silently be February — a 60-day gap reported as a one-month change. When gaps are possible, generate a complete month spine with generate_series and left-join sales onto it so every month exists before LAG runs.

Division by zero in the prior month

If a previous month summed to exactly 0, / previous_month raises a division-by-zero error. Guard it with NULLIF(previous_month, 0), which turns the zero into NULL and yields a NULL percentage instead of crashing the dashboard refresh.

An alternative without a window function

A self-join expresses the same idea for engines or reviewers who dislike LAG:

SELECT cur.yymm AS month,
       ROUND(((cur.v - prev.v) / NULLIF(prev.v,0)) * 100, 2) AS percentage_diff
FROM monthly cur
LEFT JOIN monthly prev
  ON prev.yymm = to_char((cur.yymm || '-01')::date - INTERVAL '1 month', 'YYYY-MM');

The LEFT JOIN version has the advantage that it compares against the true prior calendar month, so a missing month produces a NULL rather than a wrong neighbour.

Senior signal: force numeric division before the divide, justify why a YYYY-MM string sorts chronologically (and prefer a real date when unsure), recognise that LAG gives the previous row not the previous month so a gap corrupts the comparison, and guard the prior-month denominator with NULLIF.
