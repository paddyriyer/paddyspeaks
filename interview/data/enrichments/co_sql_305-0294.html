The CFO is reviewing org structure ahead of a reorg. She asks: "For each Senior Manager, how many people ultimately report up to them — direct reports, indirect reports, and everyone deeper in their tree combined?" The answer needs the entire reporting chain, not just one level. That's the canonical use case for a recursive CTE — walking down a self-referencing hierarchy.

Two Senior Managers (E02, E03). Each has a tree below them. We want a count per Senior Manager of everyone in their tree — directs and indirects.

2. The SQL — recursive CTE walking the tree

WITH RECURSIVE chain AS (
  -- Anchor: start from each Senior Manager
  SELECT emp_id, manager_id, 1 AS depth
  FROM employees
  WHERE level = 'Senior Manager'

  UNION ALL

  -- Recursive step: find everyone whose manager is already in the chain.
  -- Each iteration walks one level deeper into the tree.
  SELECT e.emp_id, e.manager_id, c.depth + 1
  FROM employees e
  JOIN chain c ON e.manager_id = c.emp_id
)
SELECT
  manager_id,                   -- the Senior Manager (root of each tree)
  COUNT(*) AS reports
FROM chain
WHERE depth > 1                 -- exclude the Senior Manager themselves
GROUP BY manager_id;

(Note: the original solution doesn't filter depth > 1, which would count the Senior Manager in their own tree. Added here for correctness.)

The CFO sees E02 manages a tree of 5 people, E03 manages a tree of 2. That delta is what reorgs are made of.

4. Strategic explanations

The two halves of every recursive CTE
  • Anchor (the SELECT before UNION ALL) — the seed rows. Here: every Senior Manager.
  • Recursive step (the SELECT after) — references the CTE name itself (chain). Each iteration takes the current set of rows in chain, finds employees whose manager_id matches, adds them in. The engine repeats until the recursive step produces zero new rows.
Why UNION ALL not UNION

UNION ALL just appends rows; UNION deduplicates. In a hierarchy walk you don't need dedup (each employee appears at exactly one depth) and the dedup costs an O(n²) sort. Always UNION ALL in recursive CTEs unless you actually need the dedup.

The "infinite loop" gotcha — cycles in the data

If your data has a cycle (E02's manager is E04, but E04's manager is E02 — illegal but possible from a bad migration), the recursion never terminates. Postgres caps recursion at max_recursion_depth (default 100); other engines may run forever. Defensive version uses depth < 20 in the join condition. SQLite tolerates infinite recursion until you hit memory.

The depth column is gold

It tells you how far down the tree each report sits. Useful for:

  • Direct reports only: WHERE depth = 2 (depth 1 is the Senior Manager, depth 2 is direct reports).
  • Skip-level meetings: WHERE depth = 3.
  • Span of control distribution: GROUP BY depth shows tree shape — flat orgs concentrate at low depth, deep orgs spread across many.
Walking up the tree (instead of down)

Same pattern, reverse the JOIN. "Find every manager in the chain above this employee":

WITH RECURSIVE chain_up AS (
  SELECT emp_id, manager_id, 1 AS depth FROM employees WHERE emp_id = 'E08'
  UNION ALL
  SELECT e.emp_id, e.manager_id, c.depth + 1
  FROM employees e
  JOIN chain_up c ON c.manager_id = e.emp_id    -- <-- direction reversed
)
SELECT * FROM chain_up;
-- E08 → E05 → E02 → E01 → (NULL — we hit the top)
The non-recursive alternative — when it works

If the hierarchy depth is bounded and small (e.g. always exactly 4 levels), a series of explicit LEFT JOINs to employees aliased four times will work. But you can't generalise to "any depth". Recursive CTE is the only portable answer once depth is unknown.
