A film database's data-quality team wants to find every movie with unreported box-office revenue — either an explicit NULL or a stored zero, which means "not recorded." The interviewer is testing whether you know that NULL cannot be detected with = NULL (it requires IS NULL), and that a zero value stored in a numeric column is semantically distinct from NULL but should also be treated as missing data in this context.

1. The source data — movies

movie_idtitlerelease_yearbox_office
1The Grand Illusion1937NULL

3Dune: Part Two2024711000000
4Short Film A20220
5Festival Documentary2021NULL

7Blockbuster X2020500000000

Rows 1 and 5 have explicit NULLs; rows 4 and 6 have stored zeros — all four should appear in the result. Rows 2, 3, and 7 have genuine positive revenue and should be excluded.

2. The SQL — IS NULL plus zero-detection

SELECT movie_id, title, release_year
FROM movies
WHERE box_office IS NULL
   OR NULLIF(regexp_replace(box_office::text, '[^-.0-9]', '', 'g'), '')::numeric = 0
ORDER BY release_year DESC;

Note on dialect: the regexp_replace … ::numeric defensive cast is a PostgreSQL pattern for a text-typed column. If box_office is stored as a clean numeric type (INT or DECIMAL), the query simplifies to: WHERE box_office IS NULL OR box_office = 0. In Snowflake with a text column, use TRY_TO_NUMBER(box_office) instead of the regex cast.

5Festival Documentary2021

Four rows qualify, ordered by release_year DESC. Rows 2, 3, and 7 are excluded because box_office IS NULL is false and box_office = 0 is false (all have positive revenue).

4. Strategic explanations

Why = NULL never works

NULL represents an unknown value. Any comparison with NULL using =, !=, <, etc. evaluates to NULL (not TRUE or FALSE) — and a WHERE clause requires TRUE to include the row. So WHERE box_office = NULL never returns any rows, even when box_office is NULL. The correct test is IS NULL (or IS NOT NULL). This is one of the most common SQL beginner errors and the first thing to verify in a NULL-handling question.

NULL vs zero: two different problems, one filter

NULL means "the value was never recorded." Zero means "the value was recorded as zero" — possibly a valid box-office figure for a film that did not release commercially, or possibly a data entry error meaning "unknown." The business decision to treat both as "missing" is encoded in the OR clause. A purist would store missing data as NULL and never use 0 as a sentinel; real-world data pipelines are rarely that clean, so defensive OR box_office = 0 handling is a practical necessity.

The defensive regex cast — when and why

If box_office is typed as TEXT (a schema smell but common with legacy imports), values like "N/A", "$0", or empty string can all appear. The pattern regexp_replace(box_office::text, '[^-.0-9]', '', 'g') strips all non-numeric characters; NULLIF(..., '') converts an empty result (e.g., the input was "N/A") to NULL; the final ::numeric cast converts the cleaned string to a number. This is defensive I/O at the database layer. In Snowflake, TRY_TO_NUMBER(box_office) is the idiomatic equivalent and handles parse failures by returning NULL rather than raising an error.

Simplification for clean schemas

On a properly typed schema where box_office is BIGINT or DECIMAL, the query reduces to a two-predicate OR:

WHERE box_office IS NULL OR box_office = 0

This is what a senior candidate proposes first, then qualifies with "if the column is text-typed, we need the defensive cast pattern." Proposing the clean solution first and adding complexity only when justified is a hallmark of senior SQL reasoning.

ORDER BY release_year DESC

Ordering by year descending surfaces the most recently released films with missing data first — useful for triage because recent films with missing box-office figures are more likely data pipeline failures, whereas 1937 films (The Grand Illusion) genuinely lack commercial revenue records. This is a business context observation that elevates the answer.

Dialect notes
  • Snowflake: TRY_TO_NUMBER(box_office) is null-safe; IS NULL works identically.
  • PostgreSQL: regex cast with ::numeric as shown; or use NULLIF(box_office, 0) if the column is already numeric.
  • BigQuery: SAFE_CAST(box_office AS INT64) returns NULL on failure.
  • MySQL: CAST(box_office AS SIGNED) = 0 returns 0 for non-numeric strings rather than NULL — use IF(box_office REGEXP '^[0-9]+$', CAST(box_office AS SIGNED), NULL) = 0 for safe handling.
Senior signal: open with why = NULL is always wrong (IS NULL is required); distinguish NULL (never recorded) from zero (recorded as zero, possibly sentinel); propose the clean WHERE box_office IS NULL OR box_office = 0 first and escalate to the defensive regex only when the column is text-typed; and call out Snowflake's TRY_TO_NUMBER as the idiomatic null-safe alternative to the regex cast.
