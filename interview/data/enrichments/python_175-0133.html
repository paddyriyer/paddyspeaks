Trie-pruned DFS. The same shape as autocomplete-on-a-grid, OCR word-matching with a fixed lexicon, and mobile keyboard suggestion. The naive approach — for each word, DFS the board — is O(W × m·n·4^L). The trie inverts that: walk the board ONCE, and the trie tells you which board path is still a viable prefix of ANY word. Atlassian asks this version because their Confluence search-as-you-type uses an in-memory trie of project terms to prune the candidate scan over a document's bigram grid. Same algorithm; same trie; just the substrate (grid vs document) differs.

1. The problem in one sentence + the key insight

Given an m×n board of letters and a list of words, return every word that can be spelled by walking adjacent cells (no cell reuse within a single word). Insight: build a trie of the wordlist. DFS each starting cell once, descending both the board AND the trie in lockstep. If the current board letter is not a child of the current trie node, the DFS is pruned — every word in the wordlist that would have used this prefix is eliminated by one comparison. Mark visited cells in-place with a sentinel ('*') and restore on backtrack to avoid a separate seen set per DFS frame.

2. Worked example — 2×3 board, trie-pruned DFS

Board: [['c','a','t'], ['d','o','g']]. Words: ['cat', 'cad', 'cog', 'dog'].

Step 1 — Build the trie:

  root
   ├── 'c'
   │    └── 'a'
   │         ├── 't' (#='cat')
   │         └── 'd' (#='cad')
   │    └── 'o'
   │         └── 'g' (#='cog')
   └── 'd'
        └── 'o'
             └── 'g' (#='dog')

Step 2 — DFS from cell (0,0)='c':
  trie['c'] exists -> descend; mark board[0][0]='*'
    neighbour (0,1)='a': trie['c']['a'] exists -> descend; mark board[0][1]='*'
      neighbour (0,2)='t': trie[..]['t'] exists, '#' present -> FOUND 'cat'
                            continue DFS, no further children -> return
      neighbour (1,1)='o': trie[..]['o'] does NOT exist (only 't','d' under 'a')
                            -> PRUNED (one comparison saved a whole subtree)
      restore (0,1)='a', return
    neighbour (1,0)='d': trie['c']['d'] does NOT exist -> PRUNED
    restore (0,0)='c', return

Step 3 — DFS from cell (0,2)='t':
  trie['t'] does NOT exist -> PRUNED in one step

Step 4 — DFS from cell (1,0)='d':
  trie['d'] exists -> descend; mark board[1][0]='*'
    neighbour (0,0)='c': trie['d']['c'] does NOT exist -> PRUNED
    neighbour (1,1)='o': trie['d']['o'] exists -> descend
      neighbour (0,1)='a': trie[..]['a'] does NOT exist -> PRUNED
      neighbour (1,2)='g': trie[..]['g'] exists, '#' present -> FOUND 'dog'

Result: {'cat', 'dog'}.  'cad' and 'cog' were valid trie paths
but the BOARD didn't support them.

The key win: cell (0,2)='t' as a starting cell is pruned in one trie lookup — without the trie we'd have launched a full 4-direction DFS from it. With a 30-word dictionary, this prune saves ~95% of the work.

3. The algorithm — code with inline comments

def findWords(board, words):
    # Build trie. Use plain dict; sentinel '#' stores the full word at end-nodes
    # so we don't have to re-stringify the path.
    root = {}
    for w in words:
        n = root
        for c in w: n = n.setdefault(c, {})
        n['#'] = w

    R, C = len(board), len(board[0])
    found = set()          # set, not list: same word reachable many ways = one hit

    def dfs(r, c, n):
        ch = board[r][c]
        if ch not in n: return         # trie prune — the heart of the optimisation
        n = n[ch]
        if '#' in n: found.add(n['#']) # word complete, but keep DFSing for longer hits
        board[r][c] = '*'              # in-place visited mark (saves a set per frame)
        for dr, dc in [(1,0),(-1,0),(0,1),(0,-1)]:
            nr, nc = r+dr, c+dc
            if 0 <= nr < R and 0 <= nc < C: dfs(nr, nc, n)
        board[r][c] = ch               # backtrack restore

    for r in range(R):
        for c in range(C): dfs(r, c, root)
    return list(found)

Data structures: a nested-dict trie (no third-party Trie class needed) and a set for deduplicated hits. Invariant: at every DFS frame, n is the trie node corresponding to the prefix already laid down on the board path; the board cells on the path are temporarily '*' and will be restored on return.

4. Complexity

Preprocessing: O(W · L) where W = number of words and L = max word length — building the trie. Search: O(m · n · 4 · 3^(L-1)) — every starting cell launches a DFS bounded by the longest word; each step has 4 neighbours but one is the cell you just came from (so effectively 3 onward branches). Space: O(W · L) for the trie + O(L) recursion depth. The trie prune is what keeps the search exponent in L, not in the total word count.

5. The interviewer's probes

Probe 1 — Empty board / empty wordlist. Empty wordlist → trie has just root = {}, the very first ch not in n check prunes immediately, return []. Empty board → never enters the loop, return []. The algorithm handles both with no special casing.

Probe 2 — Word longer than board area. The DFS naturally bottoms out because all neighbouring cells will be marked '*' before the word can complete. No explicit check needed; this is one of the elegance points.

Probe 3 — Generalise: can cells be reused? If yes (sometimes called "Word Search III" or "Boggle Plus"), drop the '*' marking and add a per-DFS-frame counter to bound recursion. Different complexity: now exponential without bound, must impose max word length.

Probe 4 — Performance trap: dead trie branches. After a word is found, that trie leaf is never pruned, so the DFS keeps walking it on every later traversal. Senior optimisation: delete the '#' entry after first hit AND prune empty trie branches on the way back up the recursion. For a 30k-word dictionary on a 12×12 board, this cuts runtime ~3×.

Probe 5 — Alternative algorithm: per-word DFS. For each word, DFS the board looking for that exact string. O(W · m · n · 4^L) — strictly worse than the trie approach for any W > 1. Useful to mention as the "obvious" baseline.

Probe 6 — Concurrent: parallelise across starting cells. Each starting cell's DFS is independent (after the in-place '*' marker is changed to per-thread visited sets). Embarrassingly parallel — m·n workers, each O(3^L). Used in real OCR scan workers.

6. Where this pattern appears in production

  • Mobile keyboard autocomplete (Apple, SwiftKey, Gboard) — the keyboard maintains a trie of likely words and walks both the touch-event sequence AND the trie to suggest completions; same prune logic.
  • Google Vision OCR / Tesseract — character recognition emits a lattice of candidate characters per position; a trie of valid dictionary words is walked over the lattice to pick the most likely word path.
  • Atlassian Confluence search-as-you-type — trie of project terms applied to bigram tiling of the current document; results filtered live as keystrokes arrive.
  • Aho-Corasick automaton (grep, fail2ban, network IDS) — generalisation of trie DFS to multi-pattern string matching with failure links; same idea, no grid.
  • Crossword / Scrabble solver (Words With Friends server) — the board is the grid, the trie is the SCRABBLE dictionary (TWL or SOWPODS), and the algorithm is identical with the addition of letter-rack constraints.
