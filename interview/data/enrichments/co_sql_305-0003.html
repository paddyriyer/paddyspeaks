RedHat's growth team wants a weekly split of the customer base into two buckets — New customers, who have ordered exactly once, and Returning customers, who have ordered more than once. It is a classic two-level aggregation: first count orders per customer, then count customers per bucket. The trap is that the second level has to group on the same expression it labels with — and a subtle mismatch there silently mislabels every row.

Seven orders across four customers: 101 ordered three times, 102 twice, and 103 and 104 once each. So the correct answer is 2 New (103, 104) and 2 Returning (101, 102). Keep those numbers in mind — the reference query does not produce them.

2. The SQL — two-level aggregation

SELECT
  CASE WHEN cnt = 1 THEN 'New' ELSE 'Returning' END AS customer_type,
  COUNT(*) AS customers
FROM (SELECT customer_id, COUNT(*) AS cnt FROM orders GROUP BY customer_id) t
GROUP BY CASE WHEN cnt = 1 THEN 'New' ELSE 'Returning' END
ORDER BY customer_type;

The inner query counts orders per customer into cnt. The outer query buckets each customer row by whether cnt = 1, and counts how many customers land in each bucket. This is the corrected query — see the trap below for what the original reference got wrong.

4. Strategic explanations

The trap that breaks the reference solution

The original answer labelled the bucket with CASE WHEN COUNT(*) = 1 ... in the SELECT while grouping by CASE WHEN cnt = 1 .... Those are two different expressions. The GROUP BY correctly forms two buckets — but inside the SELECT, COUNT(*) is the number of customer rows in that bucket, which is 2 for both buckets here. So both groups evaluate COUNT(*) = 1 as false and both get labelled Returning:

-- BROKEN: SELECT labels with COUNT(*), GROUP BY uses cnt
SELECT
  CASE WHEN COUNT(*) = 1 THEN 'New' ELSE 'Returning' END AS customer_type,
  COUNT(DISTINCT customer_id) AS customers
FROM (SELECT customer_id, COUNT(*) AS cnt FROM orders GROUP BY customer_id) t
GROUP BY CASE WHEN cnt = 1 THEN 'New' ELSE 'Returning' END;
-- returns: ('Returning', 2), ('Returning', 2)  -- two rows, both mislabelled

The fix is consistency: the label expression in SELECT and the grouping key in GROUP BY must be the same expression on cnt. The customer tally is then COUNT(*) over the grouped customer rows.

A cleaner rewrite — bucket once in a CTE

Repeating the CASE in two places is what invited the bug. Compute the label once and group on its alias:

WITH per_customer AS (
  SELECT customer_id,
         CASE WHEN COUNT(*) = 1 THEN 'New' ELSE 'Returning' END AS customer_type
  FROM orders
  GROUP BY customer_id
)
SELECT customer_type, COUNT(*) AS customers
FROM per_customer
GROUP BY customer_type
ORDER BY customer_type;

One source of truth for the label, impossible to drift. Postgres also lets you GROUP BY customer_type by output-column name, which removes the duplicated expression entirely.

Conditional aggregation — one row instead of two

If the consumer wants New and Returning as two columns on a single row (a tidy dashboard tile), pivot with FILTER:

SELECT
  COUNT(*) FILTER (WHERE cnt = 1) AS new_customers,
  COUNT(*) FILTER (WHERE cnt > 1) AS returning_customers
FROM (SELECT customer_id, COUNT(*) AS cnt FROM orders GROUP BY customer_id) t;
Edge case: what counts as an "order"?

The schema carries a status column. A cancelled or failed order arguably should not promote a customer from New to Returning. If the business defines a customer by completed orders, the inner query needs WHERE status = 'completed' — and a customer whose only orders were all cancelled then disappears from the report entirely, since they produce no inner row. Confirm the definition before writing the filter. Also note: customers with zero orders never appear here at all; if "New" should include never-ordered signups, you need a LEFT JOIN from a customers table instead.

Senior signal: spot that a CASE in SELECT must match the CASE in GROUP BY exactly — and that COUNT(*) in the outer query counts grouped rows, not orders — then refactor the duplicated expression into a single CTE-computed label so the bug cannot recur, and pin down whether cancelled and never-ordered customers belong in the report.
