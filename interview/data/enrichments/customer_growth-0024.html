A streaming service runs a nightly batch that does daily growth accounting: it takes today's active users, diffs them against yesterday's saved state, and labels each user NEW / RESURRECTED / RETAINED / CHURNED / STALE. The interviewer is probing the snapshot-merge pattern — a FULL OUTER JOIN between a tiny single-day partition and the persisted state table — and whether you understand why this scales: history is never re-scanned, only today is.

1. The source data — watch_history and user_state_master

The deliberate edge cases: user 5 exists only in today's activity (NEW — the FULL OUTER JOIN must keep the unmatched right side), and user 7 exists only in the master with no activity (STALE — the unmatched left side). A plain INNER JOIN would silently drop both.

2. The SQL — today vs yesterday liveness state machine

WITH today_l1 AS (
    -- Single-day partition prune: scan TODAY, never history.
    SELECT user_id, MAX(watch_date) AS last_login
    FROM watch_history
    WHERE watch_date = '2026-05-05'
    GROUP BY user_id
),
merged AS (
    -- FULL OUTER JOIN keeps users present on either side only.
    SELECT
        COALESCE(t.user_id, s.user_id) AS user_id,
        t.user_id IS NOT NULL          AS active_today,
        s.user_id IS NOT NULL          AS in_master,
        s.is_active_yesterday          AS active_yesterday,
        s.current_streak               AS prev_streak,
        s.tier                         AS prev_tier
    FROM user_state_master s
    FULL OUTER JOIN today_l1 t ON s.user_id = t.user_id
),
labelled AS (
    SELECT user_id,
        CASE
            WHEN NOT in_master AND active_today                         THEN 'NEW'
            WHEN active_today AND COALESCE(active_yesterday, 0) = 0     THEN 'RESURRECTED'
            WHEN active_today AND COALESCE(active_yesterday, 0) = 1     THEN 'RETAINED'
            WHEN NOT active_today AND COALESCE(active_yesterday, 0) = 1 THEN 'CHURNED'
            ELSE                                                             'STALE'
        END AS growth_status,
        CASE WHEN active_today THEN COALESCE(prev_streak, 0) + 1 ELSE 0 END AS new_streak,
        prev_tier AS tier_carry
    FROM merged
)
SELECT user_id, growth_status, new_streak, tier_carry
FROM labelled
ORDER BY
    CASE growth_status
        WHEN 'NEW' THEN 1 WHEN 'RESURRECTED' THEN 2 WHEN 'RETAINED' THEN 3
        WHEN 'CHURNED' THEN 4 ELSE 5
    END, user_id;

The companion summary reuses the same labelled CTE: SELECT growth_status, COUNT(*) FROM labelled WHERE growth_status IN ('NEW','RESURRECTED','RETAINED','CHURNED') GROUP BY growth_status with the same ordering.

4. Strategic explanations

Why FULL OUTER JOIN is mandatory here

The five labels need both unmatched sides of the join. NEW users appear only in today_l1 (right side, no master row). STALE users appear only in user_state_master (left side, no activity). An INNER JOIN drops both — you would lose every acquisition and every long-dormant account. LEFT JOIN from master would keep STALE but lose NEW; LEFT JOIN from today would do the reverse. Only FULL OUTER JOIN retains everyone, and COALESCE(t.user_id, s.user_id) rebuilds a single non-null key from whichever side matched.

Detecting which side matched — user_id IS NOT NULL

After a FULL OUTER JOIN, an unmatched row has NULLs in every column of the missing side. t.user_id IS NOT NULL is therefore a clean "was this user active today?" boolean, and s.user_id IS NOT NULL is "did this user exist in the master?". This is far more robust than testing a data column — a data column could legitimately be NULL, but the join key cannot be NULL for a matched row.

The COALESCE-on-the-flag trap

is_active_yesterday comes from the master and is NULL for any user with no master row (i.e. NEW users). The CASE wraps it in COALESCE(active_yesterday, 0) so that a NULL never makes a comparison evaluate to unknown and silently skip a branch. Branch order matters too: the NOT in_master test for NEW must come first, because a NEW user has no active_yesterday at all and should never reach the RESURRECTED/RETAINED logic.

Dialect notes — FULL OUTER JOIN availability

PostgreSQL, SQL Server, Oracle, Snowflake, and BigQuery all support FULL OUTER JOIN natively. MySQL does not — the standard workaround there is LEFT JOIN ... UNION ... RIGHT JOIN, or emulating it with two outer joins. SQLite gained FULL OUTER JOIN in 3.39 (2022); older builds need the same UNION trick. The single-day WHERE watch_date = '...' is the partition prune that makes the pattern scale — on a real partitioned table the planner reads exactly one partition regardless of how many years of history exist.

What a senior says next

"This same query is also the writer for tomorrow's user_state_master — project labelled down to (user_id, last_active, is_active_yesterday = active_today, current_streak = new_streak, tier) and that becomes the next batch's input. The whole pipeline is one self-perpetuating shuffle; the report and the next state snapshot fall out of the same DAG node."

Senior signal: justify FULL OUTER JOIN by naming which label needs each unmatched side, use join_key IS NOT NULL (not a data column) to detect membership, COALESCE the master-sourced flag so NULLs from NEW users do not skip a CASE branch, and point out the query doubles as the writer of the next day's state table.
