The single most-watched chart at any subscription business is the cohort retention curve: by signup-week cohort, what fraction is still active at week 1, week 4, week 12? It's the leading indicator for LTV, the input to acquisition-cost ROI, and the quickest way to see if a product change moved the needle. The query pivots three concepts: cohorts (signup time), retention (still active later), and the curve (week-N as a relative axis).

2. The SQL — cohort + week-N matrix

WITH cohorts AS (
  SELECT user_id, DATE_TRUNC('week', signup_date) AS cohort_week
  FROM users
),
dedup AS (
  SELECT DISTINCT user_id, DATE_TRUNC('week', activity_date) AS active_week
  FROM activity
),
joined AS (
  SELECT c.cohort_week, d.active_week,
    EXTRACT(EPOCH FROM (d.active_week - c.cohort_week)) / (7*86400) AS week_n,
    c.user_id
  FROM cohorts c JOIN dedup d USING (user_id)
)
SELECT cohort_week, week_n,
  COUNT(DISTINCT user_id) AS active_users,
  ROUND(100.0 * COUNT(DISTINCT user_id) /
    FIRST_VALUE(COUNT(DISTINCT user_id)) OVER (PARTITION BY cohort_week ORDER BY week_n), 1)
    AS retention_pct
FROM joined
WHERE week_n BETWEEN 0 AND 12
GROUP BY cohort_week, week_n
ORDER BY cohort_week, week_n;

3. Strategic explanations

The 3-CTE structure — what each one does
CTEpurposeoutput shape
cohortslabel each user with their signup-week bucket(user_id, cohort_week)
dedupcollapse multi-day activity into "weeks the user was active"(user_id, active_week)
joinedcompute relative week_n = active_week - cohort_week(cohort_week, active_week, week_n, user_id)

The cleanness of this pattern is why every retention dashboard reaches for these three CTEs.

Why FIRST_VALUE over a self-join

The denominator for retention% is the cohort's week-0 size. Without a window function, you'd self-join the result back to itself filtered to week_n = 0. FIRST_VALUE with ORDER BY week_n grabs the week-0 count and replicates it onto every row of the same cohort — one pass instead of two.

The DATE_TRUNC('week', ...) bucketization choice

Why week, not day or month?

  • Day: way too noisy — weekend/weekday signup demographics differ. Hard to read.
  • Week: dampens day-of-week effects, still granular enough to see campaigns.
  • Month: clean for finance but laggy — takes 6 weeks to detect a change.

Production growth dashboards almost universally use weekly cohorts. Pin DATE_TRUNC's week-start (Monday in Postgres, Sunday in BigQuery default) for cross-dialect parity.

The "in-flight cohorts" trap

If today is May 1, the cohort that signed up the week of April 28 has only had 3 days — their week-12 retention is 0% because week 12 hasn't happened yet. Don't render those cells. Production fix:

WHERE active_week <= CURRENT_DATE
  AND cohort_week + INTERVAL '7 days' * 12 <= CURRENT_DATE   -- only complete 12-week cohorts
The cohort-vs-cumulative retention distinction
variant"week N retention" means
Classic (this query)fraction of cohort active during week N
Cumulative (sometimes called "rolling")fraction of cohort active any week N or earlier = always 100%
"Bracket" or "any time after"fraction of cohort active any week ≥ N

Always confirm the definition. The classic one is what most "retention curve" charts plot.

The "active" definition trap

What counts as active? "Any row in activity" is the loosest. Pinned definitions:

  • Any login — very loose
  • Any play that exceeded 60 seconds — Netflix's standard
  • Any play that completed an episode — tighter
  • Any play AND not the auto-play preview — tightest

All four give different curves. Use the definition the PM actually uses, not the loosest one.

Cohort comparison — A/B-style

The chart's actual question is rarely the curve in isolation — it's "is this week's cohort better or worse than last week's?". Pivot to side-by-side:

SELECT week_n,
  MAX(CASE WHEN cohort_week = '2026-04-22' THEN retention_pct END) AS cohort_a,
  MAX(CASE WHEN cohort_week = '2026-04-15' THEN retention_pct END) AS cohort_b,
  MAX(CASE WHEN cohort_week = '2026-04-22' THEN retention_pct END) -
  MAX(CASE WHEN cohort_week = '2026-04-15' THEN retention_pct END) AS lift_pp
FROM /* the retention CTE */
GROUP BY week_n;
The dialect cheat sheet for week_n calculation
-- Postgres / DuckDB
EXTRACT(EPOCH FROM (active_week - cohort_week)) / (7*86400)

-- BigQuery
DATE_DIFF(active_week, cohort_week, WEEK)

-- Snowflake
DATEDIFF(WEEK, cohort_week, active_week)

-- SQLite
(julianday(active_week) - julianday(cohort_week)) / 7
The unsatisfied-demand follow-up — why did they stop?

The retention curve is a what; the actionable follow-up is the why. Each cohort gets a "last activity type" classification: did they leave during a billing event, after a specific show ended, after a price change? Joining to cancellation_events exposes the policy levers.

Senior signal: structure the query as 3 CTEs with crisp roles, name FIRST_VALUE as the cleanest week-0 denominator (vs self-join), pre-empt the in-flight-cohort artifact, and pin "active" to a stricter definition than "any row". The cohort-vs-cumulative distinction and the side-by-side A/B comparison are the over-the-bar finish.
