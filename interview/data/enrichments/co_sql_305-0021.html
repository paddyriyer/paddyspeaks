Reddit's order-ops team wants an account-history view: orders grouped by customer, and within each customer the most recent and most valuable orders on top. It is a pure ORDER BY question, but the interviewer is checking whether you understand that sort keys are applied left to right as a tiebreak chain, and that mixing ASC and DESC in one clause is both legal and intentional.

1. The source data — orders

order_idcustomer_idorder_dateamountstatus
11012024-03-01$120completed
21012024-03-05$80completed
31012024-03-05$200completed
41022024-02-10$50cancelled
51022024-04-01$300completed

Orders 2 and 3 belong to the same customer and share the same order_date — a deliberate tie on the first two sort keys. Only the third key, amount, can break it.

2. The SQL — ORDER BY a, b DESC, c DESC

SELECT * FROM orders ORDER BY customer_id ASC, order_date DESC, amount DESC;

Three sort keys. Rows are first ordered by customer_id ascending; within equal customers, by order_date descending; and within equal customer-and-date, by amount descending. Each key only acts as a tiebreaker for the keys to its left.

Customer 101 sorts before 102 (ASC). Within 101, the two March-5th orders rise above the March-1st order (order_date DESC), and between those two ties, $200 beats $80 (amount DESC). The third key did real work: without it, orders 2 and 3 would be in arbitrary, planner-dependent order.

4. Strategic explanations

Sort keys are a priority chain, not independent sorts

The single most common misconception: that each key sorts the whole table separately. It does not. ORDER BY a, b, c means "sort by a; for rows where a ties, sort by b; for rows where both tie, sort by c." A later key never reorders rows that an earlier key already separated. So amount DESC here only ever rearranges orders that share both a customer and a date — it does not sort all orders by amount.

ASC and DESC attach per-column

The direction keyword binds to its own column only. Here customer_id is ascending while order_date and amount are descending — oldest customers first, but newest, biggest orders first within each. A frequent bug is writing ORDER BY order_date, amount DESC expecting both descending; order_date there is silently ASC (the default). Spell out every direction when they differ.

The deterministic-ordering trap

Even with three keys, this sort is only fully deterministic if no two rows tie on all three. Two orders by the same customer, same day, same amount would still come back in arbitrary relative order — and that order can change between runs, after a VACUUM, or under parallelism. For a guaranteed-stable sort (pagination, snapshot tests, diffable exports), append a unique tiebreaker:

SELECT * FROM orders
ORDER BY customer_id ASC, order_date DESC, amount DESC, order_id ASC;

The primary key as a final key costs almost nothing and removes all non-determinism.

Where NULLs land

If order_date or amount can be NULL, position matters: Postgres sorts NULLs last for ASC and first for DESC. So a NULL amount under amount DESC floats to the top of its tie group — possibly not what you want. Override explicitly with amount DESC NULLS LAST. (Other engines differ — MySQL sorts NULLs first for ASC — so never assume.)

SELECT * — fine for an answer, risky in production

SELECT * is acceptable in an interview, but flag it: if the table gains a column, the view's shape changes silently, which can break downstream consumers and pull wide unused columns over the wire. Naming columns explicitly is the production-grade habit.

Senior signal: describe multi-key ORDER BY as a left-to-right tiebreak chain rather than independent sorts, attach ASC/DESC per column deliberately, and add a unique final key plus NULLS LAST/FIRST when the result must be reproducibly deterministic.
