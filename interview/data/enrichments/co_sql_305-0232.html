DoorDash's customer-support tooling needs each customer's most recent order — the one an agent should pull up first. The reference solution uses a correlated subquery: keep a row only if its date equals the max date for that same customer. It produces the right rows, but the interviewer wants to see whether you spot what happens when a customer placed two orders on the same day.

1. The source data — orders

Customer 101 has a clear latest order (March beats January). But customer 102 placed two orders on the same date, 2026-02-01 — a tie. Customer 103 has a single order. The tie is the whole interview.

2. The SQL — correlated subquery on MAX(order_date)

SELECT * FROM orders o
WHERE order_date = (
  SELECT MAX(order_date) FROM orders WHERE customer_id = o.customer_id
);

For each row o, the subquery computes the latest order date for that customer. The row survives only if its own date equals that max. One subquery execution per row.

3. The resulting analysis

51032026-04-15completed150

Customer 101 returns one row, as expected. But customer 102 returns two — orders 3 and 4 both match the max date of 2026-02-01. "The latest record" silently became "all records on the latest date". Whether that is a bug depends on whether the support tool expects exactly one row per customer.

4. Strategic explanations

The same-day tie — date granularity matters

Matching on order_date assumes the date alone uniquely identifies "latest". It does not when two orders share a day. If the table has a finer column — a created_at timestamp, or a monotonically increasing order_id — tie-break on that instead. If it genuinely does not, you must decide which same-day order wins (newest order_id? highest amount?) and say so.

The window-function answer — exactly one row, ties broken

ROW_NUMBER guarantees a single row per customer:

SELECT order_id, customer_id, order_date, status, amount
FROM (
  SELECT *,
         ROW_NUMBER() OVER (PARTITION BY customer_id
                            ORDER BY order_date DESC, order_id DESC) AS rn
  FROM orders
) t
WHERE rn = 1;

The order_id DESC secondary sort breaks the 02-01 tie deterministically — customer 102 returns only order 4. Swap ROW_NUMBER for RANK and you get the subquery's "all ties" behaviour back; the choice between them is the answer to "one row or all of them?"

Performance — correlated subquery vs window

The correlated subquery re-runs once per row of orders; on a large table that is a repeated aggregate scan and can degrade toward O(n²) without a good index on (customer_id, order_date). The window-function form sorts once and assigns row numbers in a single pass — O(n log n). A correlated NOT EXISTS ("no later order exists for this customer") is a third idiom and often optimises well, but the window function is the cleanest default.

NULL dates and missing customers

If order_date can be NULL, that order can never equal MAX(order_date) (NULL is not equal to anything), so a customer whose only order has a NULL date returns zero rows — they vanish from the report. Decide whether that is acceptable or whether NULL dates should sort last via ORDER BY order_date DESC NULLS LAST in the window form.

Senior signal: recognise that matching on MAX(order_date) returns every row on the latest day, not one row, switch to ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC, …) with an explicit tie-breaker when exactly one row is required, and note both the O(n²) risk of the correlated subquery and the NULL-date drop-out.
