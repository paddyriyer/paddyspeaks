A merchandising team wants each customer's favourite product: "return every customer's most-purchased product — and if two products tie for the top, return them both." The interviewer is probing the per-group mode with ties: can you find the maximum count within each group and keep all rows that reach it, rather than arbitrarily picking one?

1. The source data — sales

customer_idproducts purchased(narrative)
1A, A, B, CA bought twice — clear single winner
2X, X, Y, YX and Y both twice — a tie for the top
3P, Q, Rall bought once — three-way tie
4Asingle purchase — trivially the mode

Customer 2 is the headline edge case: a genuine tie that must return both X and Y. Customer 3 stretches it to a three-way tie. Customer 4 is a single-row group. A naive "pick one" solution fails 2 and 3.

2. The SQL — count, RANK, keep rank 1

WITH counts AS (
  SELECT customer_id, product, COUNT(*) AS purchase_count
  FROM sales
  GROUP BY customer_id, product
),
ranked AS (
  SELECT *, RANK() OVER (PARTITION BY customer_id ORDER BY purchase_count DESC) AS rk
  FROM counts
)
SELECT customer_id, product, purchase_count
FROM ranked
WHERE rk = 1
ORDER BY customer_id, product;

The first CTE counts purchases per customer/product. The second ranks products within each customer by that count, descending. Keeping rk = 1 returns the top — and because RANK gives tied rows the same rank, every tied product survives.

4. Strategic explanations

Why RANK and not ROW_NUMBER

This is the single decision the question tests. ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY purchase_count DESC) assigns 1, 2, 3… even to tied rows — for customer 2 it would label X as 1 and Y as 2, and WHERE rk = 1 would return only X, dropping a genuine co-winner by physical accident. RANK gives every tied row the same rank, so all tied products carry rk = 1 and survive the filter. "Return all ties" maps exactly to RANK.

RANK vs DENSE_RANK — identical here, and why

For this query both behave the same: we only ever keep rk = 1, and both RANK and DENSE_RANK assign 1 to every top-count row. They diverge only at rank 2+ (RANK skips after a tie, DENSE_RANK does not), which this filter never inspects. Worth knowing so you can explain the choice confidently rather than guessing.

The two-CTE shape: aggregate, then rank

You cannot rank by COUNT(*) and filter in one step — window functions cannot sit in a WHERE clause, and the count must exist before it can be ranked. The first CTE does the GROUP BY aggregation; the second adds the window RANK over those aggregates; the outer query filters on the window result. This "aggregate in a CTE, rank in the next, filter outside" pattern is the standard structure for top-N-per-group and per-group-mode problems.

Edge cases fall out for free

The single-row groups (customer 4) need no special handling — one product, count 1, RANK 1, kept. The three-way tie (customer 3) works because RANK does not care how many rows share the top; all get rank 1. A correct mode-with-ties query should require zero branches for "what if everything ties" or "what if there's only one row" — if yours needs them, the approach is wrong.

Dialect notes

Window functions including RANK are ANSI SQL — PostgreSQL, SQL Server, Oracle, Snowflake, BigQuery, MySQL 8+. Pre-8.0 MySQL lacks them; there the per-group mode with ties needs a correlated subquery (HAVING COUNT(*) = (SELECT MAX(cnt) ...)), which is wordier and slower. PostgreSQL also has a mode() ordered-set aggregate, but mode() returns only one value and silently discards ties — so it is the wrong tool for this exact question.

Senior signal: for per-group mode that must "return all ties", count per group in a CTE then apply RANK() OVER (PARTITION BY group ORDER BY count DESC) and keep rk = 1 — RANK gives tied rows the same rank so co-winners all survive, whereas ROW_NUMBER would arbitrarily keep one. Note PostgreSQL's mode() drops ties and is wrong here.
