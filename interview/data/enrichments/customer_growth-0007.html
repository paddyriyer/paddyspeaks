A growth team wants to spot passive customers — people browsing the site but who have never bought — and rank them by how their engagement is trending. The question (snapshot 2025-04-01) restricts to customers with at least 5 sessions and zero orders, then compares a 7-day per-day session rate against a 28-day per-day rate to classify each as warming, cooling, stable, or cold. The interviewer is checking an anti-join (NOT EXISTS), per-day rate normalisation, and careful zero-handling.

2. The SQL — anti-join with per-day rate normalisation

WITH params AS (SELECT DATE '2025-04-01' AS snapshot),
sess AS (
    SELECT s.session_id, s.customer_id, s.session_date,
           CASE WHEN s.added_to_cart::text ~ '^-?[0-9]+(\.[0-9]+)?$'
                THEN s.added_to_cart::numeric ELSE 0 END AS added_to_cart,
           ((SELECT snapshot FROM params) - s.session_date::date) AS days_back
    FROM sessions s
)
SELECT s.customer_id,
    COUNT(*) AS lifetime_sessions,
    SUM(s.added_to_cart) AS lifetime_cart_adds,
    SUM(CASE WHEN days_back < 7  THEN 1 ELSE 0 END) AS l7_sessions,
    SUM(CASE WHEN days_back < 28 THEN 1 ELSE 0 END) AS l28_sessions,
    SUM(CASE WHEN days_back < 90 THEN 1 ELSE 0 END) AS l90_sessions,
    SUM(CASE WHEN days_back < 7  THEN s.added_to_cart ELSE 0 END) AS l7_cart_adds,
    SUM(CASE WHEN days_back < 28 THEN s.added_to_cart ELSE 0 END) AS l28_cart_adds,
    ROUND(1.0 * SUM(CASE WHEN days_back < 7  THEN 1 ELSE 0 END) / 7.0,  2) AS l7_daily_sessions,
    ROUND(1.0 * SUM(CASE WHEN days_back < 28 THEN 1 ELSE 0 END) / 28.0, 2) AS l28_daily_sessions,
    ROUND(
        (1.0 * SUM(CASE WHEN days_back < 7  THEN 1 ELSE 0 END) / 7.0)
        / NULLIF(1.0 * SUM(CASE WHEN days_back < 28 THEN 1 ELSE 0 END) / 28.0, 0),
    2) AS cart_velocity,
    CASE
        WHEN SUM(CASE WHEN days_back < 28 THEN 1 ELSE 0 END) = 0 THEN 'cold'
        WHEN (1.0*SUM(CASE WHEN days_back<7  THEN 1 ELSE 0 END)/7.0)
           > 1.5 * (1.0*SUM(CASE WHEN days_back<28 THEN 1 ELSE 0 END)/28.0) THEN 'warming'
        WHEN (1.0*SUM(CASE WHEN days_back<7  THEN 1 ELSE 0 END)/7.0)
           < 0.5 * (1.0*SUM(CASE WHEN days_back<28 THEN 1 ELSE 0 END)/28.0) THEN 'cooling'
        ELSE 'stable'
    END AS engagement_trend
FROM sess s
WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = s.customer_id)
GROUP BY s.customer_id
HAVING COUNT(*) >= 5
ORDER BY
    CASE
        WHEN SUM(CASE WHEN days_back < 28 THEN 1 ELSE 0 END) = 0 THEN 4
        WHEN (1.0*SUM(CASE WHEN days_back<7 THEN 1 ELSE 0 END)/7.0)
           > 1.5 * (1.0*SUM(CASE WHEN days_back<28 THEN 1 ELSE 0 END)/28.0) THEN 1
        WHEN (1.0*SUM(CASE WHEN days_back<7 THEN 1 ELSE 0 END)/7.0)
           < 0.5 * (1.0*SUM(CASE WHEN days_back<28 THEN 1 ELSE 0 END)/28.0) THEN 3
        ELSE 2
    END,
    l28_sessions DESC;

4. Strategic explanations

Why per-day rates, not raw counts

Comparing 4 sessions in 7 days against 5 sessions in 28 days directly would be meaningless — the windows are different lengths. Dividing each count by its window length (/7.0, /28.0) normalises both to a "sessions per day" rate, making them comparable. Customer 1's 0.57/day in L7 versus 0.18/day in L28 is an apples-to-apples acceleration signal. This length-normalisation is the conceptual heart of the question.

The NOT EXISTS anti-join and why it beats NOT IN

"Zero orders" is an anti-join. NOT EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = s.customer_id) is the robust form: it short-circuits on the first match and, critically, behaves correctly even if orders.customer_id contains a NULL. NOT IN (SELECT customer_id FROM orders) looks equivalent but if that subquery yields a single NULL, the entire NOT IN evaluates to UNKNOWN for every row and the result is silently empty. Naming this NOT IN/NULL hazard is a senior tell.

The cold branch must be checked first

A customer with zero L28 sessions has an L28 daily rate of 0. The warming and cooling tests both divide-or-compare against that rate — cart_velocity would be x / 0. Putting WHEN l28_sessions = 0 THEN 'cold' as the first CASE branch short-circuits before any division is attempted. The separate NULLIF(..., 0) inside cart_velocity is the belt-and-braces guard so the displayed ratio is NULL, not an error. Order-of-branches is correctness, not cosmetics.

The defensive regex cast on added_to_cart

added_to_cart::text ~ '^-?[0-9]+(\.[0-9]+)?$' checks the value is numeric-looking before casting it. This guards against a dirty column where added_to_cart might hold 'true', 'yes', or an empty string — a blind ::numeric cast would throw and abort the whole query. In a clean schema this is belt-and-braces, but it shows the instinct to defend a cast against bad data rather than assume the column is pristine.

The duplicated CASE in SELECT and ORDER BY

The classification logic appears twice — once to label engagement_trend, once to drive the sort priority (warming=1, stable=2, cooling=3, cold=4). Postgres lets you ORDER BY engagement_trend by alias, but that would sort alphabetically (cold, cooling, stable, warming) — wrong order. The explicit priority CASE in ORDER BY is necessary. A senior would mention that wrapping the query one level deeper (SELECT ... FROM (subquery) ORDER BY priority) removes the duplication and the bug risk.

Senior signal: normalise both windows to per-day rates so a 7-day and 28-day count are comparable, use NOT EXISTS for the zero-orders anti-join and call out the NOT IN/NULL trap, order the CASE so the cold (zero-denominator) branch is evaluated before any division, and note that the duplicated classification logic in SELECT and ORDER BY is a maintenance hazard best fixed with a wrapping subquery.
