A payments processor wants a fraud-adjacent metric: how many duplicate charges slipped through per merchant? A duplicate is a charge that exactly matches an earlier one — same card, same merchant, same amount — landing within a 10-minute window. The interviewer is testing whether you can use LAG to compare each row to its predecessor within a carefully chosen partition, and whether you spot the subtlety of chained duplicates.

Payment 3 is 30 minutes after the original (payment 1) but only 27 minutes after payment 2 — still outside the 10-minute window from its immediate predecessor, so it is not a duplicate. Merchant 2's payments 5, 6, 7 form a chain: 6 is within 10m of 5, and 7 is within 10m of 6, even though 7 is 18m30s after the original.

2. The SQL — LAG within (merchant, card, amount), gap test

WITH lagged AS (
  SELECT merchant_id, credit_card_id, amount, paid_at,
         LAG(paid_at) OVER (PARTITION BY merchant_id, credit_card_id, amount ORDER BY paid_at) AS prev_paid_at
  FROM payment
)
SELECT merchant_id, COUNT(*) AS duplicate_count
FROM lagged
WHERE prev_paid_at IS NOT NULL
  AND paid_at - prev_paid_at <= INTERVAL '10 minutes'
GROUP BY merchant_id
ORDER BY merchant_id;

LAG(paid_at) partitioned by (merchant_id, credit_card_id, amount) gives each charge the timestamp of the previous matching charge. A row is a duplicate when that predecessor exists (prev_paid_at IS NOT NULL, so the first charge of a cluster is skipped) and the gap is ≤ 10 minutes. The outer query counts duplicates per merchant.

4. Strategic explanations

Why the partition key encodes the "same-ness" rule

The definition of a duplicate is "same card, same merchant, same amount". That triple becomes the PARTITION BY merchant_id, credit_card_id, amount clause — LAG then only ever looks back at a charge that already matches on all three attributes. Within a partition, ordering by paid_at means the predecessor is the most recent prior matching charge. Getting the partition key right is getting the duplicate definition right.

The chained-duplicate subtlety

Merchant 2's payment 7 is 18m30s after the original payment 5 — outside 10 minutes from the original. Yet it counts, because LAG compares it to payment 6 (9 minutes earlier), not to 5. The question says "an earlier payment ... within 10 minutes before it", and payment 6 is such an earlier payment. The LAG-to-immediate-predecessor approach happens to capture this correctly: in a tight cluster, every charge after the first has a qualifying predecessor. (If the rule were strictly "within 10 min of the cluster's first charge", you would instead carry the cluster start forward — a different query.)

Why prev_paid_at IS NOT NULL excludes the original

LAG returns NULL for the first row of every partition — the original charge in each cluster has no earlier match. The WHERE prev_paid_at IS NOT NULL filter drops exactly those originals, so the count reflects "extra" charges only. This directly implements the spec's "exclude the first occurrence of each cluster". Omitting this filter would not silently over-count — NULL <= INTERVAL is UNKNOWN so the row is dropped anyway — but stating the filter makes intent explicit and robust.

Interval arithmetic and the boundary

paid_at - prev_paid_at yields an interval, compared directly against INTERVAL '10 minutes'. The spec says "within 10 minutes (600 seconds)", and <= makes a gap of exactly 10 minutes count as a duplicate. If the rule were "strictly less than 10 minutes" you would use <. Comparing intervals avoids the error-prone EXTRACT(EPOCH FROM ...) conversion — though that form (... <= 600) is equivalent and common in dialects without rich interval support.

Only merchants with duplicates appear — by design

The WHERE clause removes all non-duplicate rows before the GROUP BY, so a merchant with zero duplicates contributes no rows and never reaches the output. That satisfies "only include merchants whose count is > 0" for free — no HAVING COUNT(*) > 0 needed, because a group cannot exist with zero members. Merchant 3 disappears precisely this way. If the report needed a zero row per merchant, you would LEFT JOIN from a merchant list instead.

Senior signal: encode a duplicate-detection rule directly in the LAG partition key (here merchant + card + amount), use prev_paid_at IS NOT NULL to exclude each cluster's original, and compare intervals (<= INTERVAL '10 minutes') rather than converting to seconds. Recognise the chained-duplicate case — LAG measures against the immediate predecessor, which is right for "an earlier payment within 10 min" but wrong if the rule anchors on the cluster's first charge.
