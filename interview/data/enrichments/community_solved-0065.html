ABC Bank wants to know, per branch location, which payment mode (Gpay, PhonePe, Paytm, or cash) carried the single highest-value transactions, and how many such transactions there were. The interviewer is testing a per-group top-N pattern: rank transactions within each location, keep the top tier, then attribute it to a mode. The hidden trap — and the reason this is rated Hard — is that "in each location" must survive into the final SELECT and GROUP BY; drop it and the answer silently collapses all branches into one bucket.

Mumbai has a tie at 5000 across two different apps — the edge case that decides whether the answer has one row or two per location. Customer 4 has a NULL app: an offline (cash) transaction.

2. The SQL — DENSE_RANK per location (original, flagged broken)

WITH txn AS (
  SELECT *, DENSE_RANK() OVER (PARTITION BY loc_name ORDER BY amount_paid DESC) AS rnk
  FROM transaction_tbls
)
SELECT COALESCE(app, 'cash') AS mode, COUNT(*) AS num_transaction
FROM txn
INNER JOIN customer c ON txn.cus_id = c.customer_id
WHERE rnk = 1
GROUP BY app;

Broken: the question asks for the result "in each location", but loc_name is absent from both the SELECT and the GROUP BY. The top transactions of every branch are merged into one global tally — the location dimension is lost.

2b. Corrected — keep loc_name through to the output

WITH txn AS (
  SELECT t.*, COALESCE(c.app, 'cash') AS mode,
         DENSE_RANK() OVER (PARTITION BY t.loc_name
                            ORDER BY t.amount_paid DESC) AS rnk
  FROM transaction_tbls t
  INNER JOIN customer c ON t.cus_id = c.customer_id
)
SELECT loc_name, mode, COUNT(*) AS num_transaction
FROM txn
WHERE rnk = 1
GROUP BY loc_name, mode
ORDER BY loc_name, mode;

4. Strategic explanations

Why DENSE_RANK over PARTITION BY loc_name is the core

PARTITION BY loc_name restarts the ranking inside every branch, and ORDER BY amount_paid DESC puts the biggest transaction at rank 1. Filtering rnk = 1 in an outer query (you cannot use a window function in WHERE directly) keeps the top tier per location. This "rank-then-filter" CTE is the canonical top-N-per-group pattern.

The bug is a lost grouping key, not bad logic

The window function in the original is correct — it does rank per location. The defect is downstream: the final SELECT ... GROUP BY app throws loc_name away. A result that claims to be "per location" but has no location column cannot be right. The lesson: every dimension the question names must appear in the final projection and the GROUP BY. Always re-read the ask against your SELECT list.

DENSE_RANK vs RANK vs ROW_NUMBER — the tie decision

Mumbai's 5000 tie makes the choice visible. ROW_NUMBER would arbitrarily pick one of the two apps and hide the other — wrong, because both genuinely transacted at the maximum. RANK and DENSE_RANK both give the tied rows rank 1, so for a rnk = 1 filter they behave identically here; they only diverge for rank 2+. Keeping all tied top transactions is the defensible reading of "highest amount transactions", so DENSE_RANK (or RANK) is right and ROW_NUMBER is a trap.

NULL handling — COALESCE and the join type

An offline transaction has NULL app; COALESCE(app, 'cash') labels it cleanly. Note also the INNER JOIN to customer: if a transaction's cus_id has no matching customer row, that transaction is dropped — including, potentially, a branch's top transaction. If transactions can exist for unknown customers, switch to LEFT JOIN so the ranking still sees every transaction and label the orphan as 'cash'/'unknown'.

Part 2 of the question — most-used mode per location

The prompt's second ask ("mode used maximum for a given location") is a different metric — frequency, not amount. Rank by count instead: RANK() OVER (PARTITION BY loc_name ORDER BY COUNT(*) DESC) over a GROUP BY loc_name, mode pre-aggregation, then keep rank 1. Distinguishing "highest-value" from "most-frequent" out loud signals you parsed a multi-part question carefully.

Senior signal: recognise the rank-then-filter top-N-per-group pattern, then catch the real bug — the original drops loc_name so it cannot answer "per location" despite correct window logic. Naming the ROW_NUMBER-vs-DENSE_RANK tie trap and separating "highest value" from "most frequent" for part 2 is the senior finish.
