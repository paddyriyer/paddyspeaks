A messaging app's growth team wants to know how each age cohort splits its time between sending messages and reading them. For every age bucket, what percentage of engagement time went to send versus open? The interviewer is testing a percentage-of-group pattern: you need a per-bucket-per-activity numerator and a per-bucket denominator, and the denominator must be scoped to exactly the activities in scope — not all activity types.

The trap is user 1's chat row with a deliberately huge time_spent of 999. If the denominator accidentally includes chat, every percentage in the 21-25 bucket collapses. Bucket 31-35 is a single-user control where send and open are equal — the answer must be exactly 50/50.

2. The SQL — per-activity totals, per-bucket totals, divide

WITH tot AS (
  SELECT u.age_bucket,
         a.activity_type,
         SUM(a.time_spent) AS act_time
  FROM activities a
  JOIN user_age u ON u.user_id = a.user_id
  WHERE a.activity_type IN ('send','open')
  GROUP BY u.age_bucket, a.activity_type
),
bk AS (
  SELECT age_bucket, SUM(act_time) AS bucket_time FROM tot GROUP BY age_bucket
)
SELECT t.age_bucket,
       t.activity_type,
       ROUND(100.0 * t.act_time / b.bucket_time, 2) AS percentage
FROM tot t
JOIN bk b USING (age_bucket)
ORDER BY t.age_bucket, t.activity_type;

tot filters to the two activities of interest, joins each activity to its user's bucket, and sums time per (bucket, activity). bk re-aggregates tot to a per-bucket total. The final query divides each activity's time by its bucket total — and because bk is built from the already-filtered tot, the chat time can never leak into the denominator.

4. Strategic explanations

Why the denominator must be derived from the filtered set

The single most important decision is where the WHERE activity_type IN ('send','open') filter lives. It is applied in tot, and bk sums tot — so the per-bucket denominator is the sum of only send and open time. If you instead computed bk directly from activities without the filter, chat's 999 would inflate every denominator and the percentages would no longer sum to 100. The numerator and denominator must both be scoped to the same universe.

100.0 forces floating-point division

time_spent here is NUMERIC, so division is exact — but the literal 100.0 (not 100) is a defensive habit. If the columns were INTEGER, 100 * act_time / bucket_time could do integer division and round each ratio toward zero before scaling, yielding nonsense like 40 becoming 0. Writing 100.0 guarantees the whole expression evaluates in a fractional type. ROUND(..., 2) then trims to the two decimal places the spec asks for.

The two-CTE structure vs a window function

This split into tot then bk is the classic "detail then total" pattern. An elegant alternative collapses it into one level with a window function: SUM(act_time) OVER (PARTITION BY age_bucket) computes the bucket total alongside each detail row, with no second CTE and no self-join. Both are correct; the window version is usually faster (one scan, no extra join) and is the answer a senior engineer would offer as the refinement.

The JOIN can lose rows — know the failure mode

An inner JOIN between activities and user_age silently drops any activity whose user has no age bucket, and any user whose activities are all non-send/open. Here every user is present in user_age, so it is safe — but in real data you would decide consciously: a LEFT JOIN with a COALESCE(age_bucket, 'unknown') bucket keeps orphan activity time visible rather than silently discarding it.

Dialect — JOIN ... USING and rounding

JOIN bk b USING (age_bucket) is standard SQL and works in Postgres, MySQL, and Oracle; SQL Server lacks USING and needs an explicit ON t.age_bucket = b.age_bucket. ROUND(x, 2) is portable, though SQL Server's ROUND returns the input's type so you may need an explicit CAST to decimal(5,2) for clean display. The aggregation and percentage logic are identical everywhere.

Senior signal: in a percentage-of-group query, scope the numerator and the denominator to the exact same filtered universe — deriving the bucket total from the already-filtered detail (or a SUM(...) OVER (PARTITION BY bucket) window) is what stops an out-of-scope category from poisoning the ratio. Use a 100.0 literal to force fractional division, and verify each group's percentages sum to 100.
