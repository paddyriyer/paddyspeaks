IBM's recruiting-ops team wants a single number for the funnel dashboard: the interview success rate — what share of interviews end in an offer? It is a one-line ratio, but the interviewer is probing two things: do you avoid the integer-division trap, and have you thought about what the denominator should actually be when outcomes are messier than offer/reject.

1. The source data — interviews

candidate_idoutcome
1offer
2reject
3offer
4reject
5reject
6no_show
7offer
8reject

Eight interviews: 3 offers, 4 rejects, and 1 no_show — an outcome that is neither a yes nor a no. That stray category is the whole interview: it decides whether the denominator is 8 or 7.

2. The SQL — offers / total × 100

SELECT 100.0 * SUM(CASE WHEN outcome='offer' THEN 1 ELSE 0 END)
              / COUNT(*) AS success_rate
FROM interviews;

SUM(CASE…) counts the offers (1 per offer, 0 otherwise); COUNT(*) counts every interview; 100.0 * scales the ratio to a percentage.

3. The resulting analysis

success_rate
37.500000

3 offers out of 8 total interviews → 37.5%. Note what the query did with the no_show: it counted it in the denominator (COUNT(*) sees all 8 rows) but not the numerator. Whether that is the right call is a definition question, not a SQL one.

4. Strategic explanations

The integer-division trap — why 100.0 leads

Both SUM(CASE…) and COUNT(*) are integers, and integer ÷ integer truncates in Postgres: 3 / 8 alone is 0. Putting the 100.0 numeric literal first forces the whole expression to numeric before the division happens, so you get 37.5 rather than 0. Reorder it carelessly — SUM(…) / COUNT(*) * 100.0 — and the division truncates to 0 before the multiply, yielding a flat 0%. This is the single most common bug in rate queries; the literal placement is doing real work.

The denominator question — the real interview

Is a no_show a failed interview? It is arguably not an interview that happened at all. "Success rate" has at least three defensible denominators:

-- A. offers / all records (the reference) — no_show counts against you
100.0 * SUM(CASE WHEN outcome='offer' THEN 1 ELSE 0 END) / COUNT(*)

-- B. offers / interviews that reached a decision — exclude no_show
100.0 * COUNT(*) FILTER (WHERE outcome='offer')
      / NULLIF(COUNT(*) FILTER (WHERE outcome IN ('offer','reject')), 0)

-- C. offers / rejects — odds, not a rate

For our data, A gives 37.5%, B gives 3/7 = 42.9%. Same table, a 5-point swing in the headline metric — purely from how one ambiguous row is classified. The senior move is to ask which definition recruiting-ops wants before writing anything.

The cleaner idiom: COUNT(*) FILTER (WHERE …) and AVG

SUM(CASE WHEN cond THEN 1 ELSE 0 END) works everywhere, but Postgres offers a more readable conditional count:

SELECT 100.0 * COUNT(*) FILTER (WHERE outcome='offer') / COUNT(*) AS success_rate
FROM interviews;

Even slicker, a boolean averaged is a rate directly: AVG((outcome='offer')::int) * 100, since the mean of 1s and 0s is the proportion. All three give 37.5% — pick the one that reads clearest to your team.

The empty-table edge case

If interviews has no rows, COUNT(*) is 0 and the query raises division by zero. A dashboard query should never crash on no data — guard the denominator with NULLIF(COUNT(*), 0), which yields NULL (shown as "no data") instead of an error. SUM(CASE…) over an empty table is 0, which is fine; it is the denominator that bites.

NULL outcomes and grouping

If outcome can be NULL, outcome='offer' is NULL (not true), so a NULL row correctly fails the offer test — but it still inflates COUNT(*). Decide whether an unrecorded outcome belongs in the denominator. And the natural follow-up is success rate per recruiter, role, or quarter: add GROUP BY and the same ratio becomes a comparison.

Senior signal: lead with the 100.0 literal to dodge integer truncation to 0%, surface the denominator ambiguity (does no_show count?) as a definition question before coding, reach for the readable COUNT(*) FILTER (WHERE …) idiom, and guard the empty-table divide-by-zero with NULLIF.
