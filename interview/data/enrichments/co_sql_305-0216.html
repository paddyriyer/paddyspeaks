X's growth team wants customer counts broken out by city to decide where to run regional campaigns. The query is a one-line GROUP BY — but the interviewer is watching whether you notice the rows with no city, whether you reach for COUNT(*) or COUNT(column) on purpose, and whether you sort and clean the output the way a real dashboard needs.

1. The source data — customers

customer_idcitynamecountry
1LondonAnnUK
2LondonBenUK
3ParisCaraFR
4NULLDanFR
5BerlinEveDE
6LondonFayUK

Three customers are in London. The interesting row is Dan, whose city is NULL — he never filled it in. Whether that row gets its own bucket or vanishes is the whole question.

2. The SQL — GROUP BY city

SELECT city, COUNT(*) AS customers FROM customers GROUP BY city;

One group per distinct city value, and COUNT(*) counts the rows that fell into each group.

3. The resulting analysis

citycustomers
NULL1
Paris1
Berlin1
London3

Four groups. Crucially, GROUP BY treats all NULLs as one group — so Dan lands in a real NULL bucket with a count of 1, rather than disappearing. The row order is arbitrary: no ORDER BY means the engine returns groups in whatever order is convenient.

4. Strategic explanations

GROUP BY keeps NULL — COUNT(column) would have dropped it differently

Two NULL behaviours collide here, and a senior answer separates them. GROUP BY city folds every NULL-city row into a single group (NULL is "equal to" NULL for grouping, unlike in WHERE). Meanwhile COUNT(*) counts rows, so that group shows 1. If you had written COUNT(city), it counts only non-NULL values — the NULL group would still appear as a row but with count 0. For "how many customers," COUNT(*) is right; COUNT(city) would be a subtle bug.

Making the NULL bucket presentable

A "NULL" row on a regional dashboard looks like a defect. Label it explicitly so stakeholders see it is intentional:

SELECT COALESCE(city, 'Unknown') AS city, COUNT(*) AS customers
FROM customers
GROUP BY COALESCE(city, 'Unknown')
ORDER BY customers DESC;

Note the GROUP BY repeats the expression — standard SQL groups by the underlying expression, not the SELECT alias. (Postgres tolerates grouping by alias or position; being explicit is the portable habit.)

Sort the output — never ship an unordered group result

Our result came back NULL, Paris, Berlin, London — no logic to it. A grouped query without ORDER BY has an undefined row order that can change between runs or after a plan change. A campaign-targeting report wants ORDER BY customers DESC (biggest cities first) or ORDER BY city (alphabetical). Always state the sort.

Only counting cities above a threshold — HAVING

If the ask narrows to "cities with at least 3 customers," that is a post-aggregation filter — HAVING, not WHERE:

SELECT city, COUNT(*) AS customers
FROM customers
GROUP BY city
HAVING COUNT(*) >= 3;

WHERE filters rows before grouping and cannot see COUNT(*); HAVING filters groups after. Mixing them up is the classic aggregation mistake.

City names that are not really distinct

'London', 'london', and 'London ' are three different groups to GROUP BY. Dirty input fragments your buckets and undercounts every city. If the data is not trusted, normalise: GROUP BY lower(trim(city)). And "London, UK" vs "London, CA" is a genuine collision — grouping by (country, city) may be the honest key.

Senior signal: note that GROUP BY collapses NULL cities into one real bucket (and label it with COALESCE), choose COUNT(*) over COUNT(city) deliberately, always attach an ORDER BY to a grouped report, and reach for HAVING — not WHERE — when filtering on the count.
