Amazon's Operations team measures one number daily: on-time delivery rate. The query joins each order's promised vs actual delivery date and computes the % within SLA per marketplace.

1. The source data — orders

2. The SQL — julianday compare + on-time conditional aggregation

SELECT marketplace,
  COUNT(*) AS orders,
  SUM(CASE WHEN julianday(delivered_at) <= julianday(promised_at) THEN 1 ELSE 0 END) AS on_time,
  ROUND(100.0 * SUM(CASE WHEN julianday(delivered_at) <= julianday(promised_at) THEN 1 ELSE 0 END)
              / NULLIF(COUNT(*), 0), 1) AS on_time_pct,
  ROUND(AVG((julianday(delivered_at) - julianday(ordered_at)) * 24), 1) AS avg_delivery_hours
FROM orders WHERE delivered_at IS NOT NULL
GROUP BY marketplace ORDER BY on_time_pct DESC;

3. Strategic explanations

The two metrics — on-time pct + avg duration

Pct alone ("95% on-time") doesn't say how on-time. A 95% rate at 30-hour avg vs 95% at 18-hour avg are different products. Always emit both.

The undelivered-still-in-flight trap

WHERE delivered_at IS NOT NULL excludes in-flight orders — without it, you'd treat them as "late" (NULL is not ≤ promised_at). For active-fleet visibility, replace with OR delivered_at IS NULL in a separate "in_flight" bucket.

P95 / tail latency — beyond average

Average masks the long tail. Production SLAs target P95 ("95% of deliveries within 24 hours") not mean. NTILE-based percentile estimation:

WITH ntiled AS (SELECT *, NTILE(20) OVER (PARTITION BY marketplace ORDER BY julianday(delivered_at) - julianday(ordered_at)) AS tile FROM orders)
SELECT marketplace, MAX((julianday(delivered_at) - julianday(ordered_at)) * 24) AS p95_hours
FROM ntiled WHERE tile <= 19 GROUP BY marketplace;
Prime vs non-Prime split

Prime orders have 2-day SLA; non-Prime has 5-7. Mixing them in one number obscures the actual story. Slice:

GROUP BY marketplace, is_prime
The "promised" definition trap

promised_at changes during the order lifecycle: initial estimate at checkout, then revised after carrier pickup, then revised again on day-of-delivery. Which promise are we comparing against? Use the customer-facing initial promise for trust metrics, the final carrier ETA for ops.

Senior signal: emit both rate + duration, surface the in-flight NULL handling, push to P95 over mean, split Prime vs non-Prime, and disambiguate which "promised_at" is being measured.
