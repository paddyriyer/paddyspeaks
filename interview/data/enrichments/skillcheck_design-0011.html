Event sourcing stores state as a sequence of immutable domain events rather than as a mutable current-state row. Replaying the event stream from the beginning (or from a snapshot) reconstructs the aggregate’s state at any point in time. The interviewer is probing whether you can design for all four constraints simultaneously: ordered per-aggregate replay (sequence_no), optimistic concurrency control (the PK prevents duplicate sequence numbers), snapshot pairing (a companion table keyed on the same (aggregate_id, sequence_no)), and producer-side idempotency (event_id UNIQUE prevents double-writes from retry storms).

1. The requirements — what the design must capture

  • Ordered replay: events for a given aggregate must be retrieved in sequence_no order to rebuild state deterministically.
  • Optimistic concurrency: the write path appends an event at the next expected sequence_no; if another writer has already taken that slot, the PK raises a conflict, signalling a concurrency violation that the caller retries.
  • Snapshot pairing: snapshots must be keyed to the exact sequence_no at which state was captured so replay starts from that position, not from event 1.
  • Idempotent writes: retried event appends must not produce duplicate rows; event_id UUID UNIQUE is the dedup guard.
  • Schema evolution: event_version carries the schema version of the payload so consumers can apply the right deserialiser.
  • Observability: occurred_at (domain time) vs. recorded_at (ingest time) separation enables both business-time replay and pipeline-time audit.

2. The DDL — the CREATE TABLE

CREATE TABLE order_events (
  aggregate_id      UUID         NOT NULL,
  sequence_no       BIGINT       NOT NULL,                  -- monotonic per aggregate (1, 2, 3, ...)
  event_id          UUID         NOT NULL,                  -- idempotency key (one row per event_id)
  event_type        VARCHAR(64)  NOT NULL,                  -- e.g. 'OrderPlaced', 'ItemAdded'
  event_version     SMALLINT     NOT NULL DEFAULT 1,        -- event schema version
  payload           JSONB        NOT NULL,                  -- the event body
  metadata          JSONB,                                  -- correlation id, causation id, user id
  occurred_at       TIMESTAMPTZ  NOT NULL,                  -- wall-clock event time
  recorded_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),    -- ingest time
  PRIMARY KEY (aggregate_id, sequence_no),
  UNIQUE (event_id)
);

CREATE INDEX ix_order_events_by_recorded_at
  ON order_events (recorded_at);

-- Companion snapshot table
CREATE TABLE order_snapshots (
  aggregate_id     UUID        NOT NULL,
  sequence_no      BIGINT      NOT NULL,                    -- snapshot taken AT this sequence_no
  state            JSONB       NOT NULL,
  state_version    SMALLINT    NOT NULL,
  created_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (aggregate_id, sequence_no)
);

4. Strategic explanations

Optimistic concurrency via the composite PK

The application code reads the current max sequence_no for an aggregate, computes next_seq = max + 1, and attempts to INSERT the new event at that sequence number. If two concurrent writers both read the same max (say, 5) and both try to INSERT at sequence 6, the PK constraint rejects the second INSERT with a unique violation. The loser retries from scratch: re-reads the aggregate state (now at sequence 6), applies its command, and tries to INSERT at sequence 7. This is optimistic concurrency control without any locking — correct and scalable under low contention.

event_id for idempotent writes

The application generates the event_id UUID before sending the INSERT. If the network times out after the INSERT committed (but before the application received the confirmation), a retry will attempt to INSERT the same event_id again. The UNIQUE constraint rejects it. The application detects the constraint violation, checks whether its event_id already exists (it does), and treats the write as successful. Without this, every network timeout would produce a duplicate event in the stream, breaking state reconstruction.

Snapshot strategy: amortising replay cost

Replaying from event 1 is O(n) in the number of events. For a high-traffic order with 1,000 events, replay cost is linear. Snapshots break this: store the aggregate state at event 500, then replay only events 501–1000. The companion order_snapshots table is keyed on (aggregate_id, sequence_no) — the exact position at which the snapshot was taken. To load an aggregate: (1) SELECT the latest snapshot; (2) SELECT all events WHERE sequence_no > snapshot.sequence_no; (3) apply them. Snapshots are taken periodically (e.g., every 100 events) by a background process, not on every write.

occurred_at vs. recorded_at

occurred_at is the domain time — when the business event occurred according to the application clock. recorded_at is the wall-clock time when the row was physically written to Postgres. These diverge in backfills, clock-skew scenarios, and async event buffering. Projections (read models) should be built using occurred_at for business correctness; pipeline monitoring should use recorded_at to detect ingest lag.

Global event ordering and projections

The ix_order_events_by_recorded_at index enables a projector to scan all events globally in insertion order: SELECT * FROM order_events WHERE recorded_at > :cursor ORDER BY recorded_at LIMIT :batch. This is how a projection (materialised read model, e.g., a reporting table) replays all order events to build its view. The cursor is advanced after each batch. This pattern is analogous to the outbox relay, but operating on domain events rather than integration events.

Postgres-specific design notes

JSONB in Postgres supports GIN indexing on payload fields if specific event types are queried frequently (e.g., WHERE payload->>‘order_id’ = ‘42’). For high-throughput event stores, consider partitioning order_events by recorded_at date range to enable efficient partition pruning and archival. In distributed systems with multiple Postgres shards, the global recorded_at index approach breaks down; event streaming infrastructure (Kafka, EventStoreDB) is the appropriate foundation at that scale.

Senior signal: explain optimistic concurrency as “the composite PK is the lock — a duplicate sequence_no raises a PK violation that the caller handles as a concurrency conflict and retries”; describe the event_id idempotency pattern (“generate before sending, check the unique constraint violation on retry”); and articulate the snapshot strategy as “load latest snapshot, replay only events after it” — these three mechanisms together show you understand event sourcing as a consistency protocol, not just a table shape.
