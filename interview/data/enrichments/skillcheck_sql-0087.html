A rideshare analyst wants each driver's best tipper — the rider who has tipped them the most in total across all trips. This is a "top-1 per group after a sub-aggregation" problem, with an explicit tie-break rule. The interviewer is probing whether you aggregate to the (driver, rider) grain first, then rank, and whether you encode "smallest rider_id wins ties" into the ranking ORDER BY.

1. The source data — trip

trip_iddriver_idrider_idtip_amount(narrative)
11103.00driver 1, rider 10

Driver 1 is the deliberate trap: riders 10, 20, and 30 all total 8.00 in tips. The tie-break rule says the smallest rider_id wins, so rider 10 must be returned. Driver 3 is the single-rider edge case — only one candidate, so no tie-break needed.

2. The SQL — sum per pair, rank per driver, keep rank 1

WITH per_pair AS (
  SELECT driver_id, rider_id, SUM(tip_amount) AS total_tip
  FROM trip
  GROUP BY driver_id, rider_id
),
ranked AS (
  SELECT driver_id, rider_id, total_tip,
         ROW_NUMBER() OVER (PARTITION BY driver_id ORDER BY total_tip DESC, rider_id) AS rn
  FROM per_pair
)
SELECT driver_id, rider_id AS top_rider, ROUND(total_tip, 2) AS total_tip
FROM ranked
WHERE rn = 1
ORDER BY driver_id;

per_pair sums tips for each (driver, rider) combination. ranked numbers each driver's riders by total_tip DESC, breaking ties by rider_id ascending. WHERE rn = 1 keeps the single top rider per driver.

Driver 1 returns rider 10: riders 10, 20, 30 each total 8.00, and the ORDER BY total_tip DESC, rider_id places the smallest id first within the tie group, so rn = 1 lands on rider 10. The tie-break is built into the window's ORDER BY — it is not an afterthought.

4. Strategic explanations

Why aggregate before ranking

"Tipped the most in total" is a sum across each rider's trips, so you cannot rank raw trip rows — rider 10's two trips (3.00 and 5.00) must first combine into 8.00. The per_pair CTE does that. Ranking the un-aggregated trip table would compare individual tip amounts, which answers "the single largest tip," a completely different question. Aggregate to the grain the question asks about, then rank.

The tie-break lives in the window's ORDER BY

ROW_NUMBER() OVER (PARTITION BY driver_id ORDER BY total_tip DESC, rider_id) has a two-key order: primary by descending total, secondary by ascending rider_id. When totals tie, the second key decides which row gets rn = 1. This is the clean way to encode "smallest id wins" — no post-processing, no extra filter. If the rule were "largest id wins" you would write rider_id DESC; if "most recent trip wins" you would add a timestamp. The tie-break rule is the secondary sort key.

ROW_NUMBER vs RANK — why the tie must collapse to one row

ROW_NUMBER is deliberate. For driver 1's three-way tie, RANK would assign all three rn = 1, and WHERE rn = 1 would return three rows for one driver — violating "one row per driver." ROW_NUMBER always produces exactly one rn = 1, with the rider_id tiebreaker deterministically choosing which. When the spec demands a single winner per group, ROW_NUMBER is correct and RANK is a bug.

Rounding placement — after the sum, on the survivor

ROUND(total_tip, 2) sits in the final SELECT, applied to the already-summed value of the surviving rank-1 row. Rounding individual tip_amount values before summing could accumulate per-row rounding error; rounding once at the end is both correct and cheaper (it runs on three rows, not seven). With NUMERIC tip amounts the sums are exact here, but rounding-last is the habit that protects you when FLOAT money sneaks in.

Dialect notes

ROW_NUMBER() OVER (...) is standard (PostgreSQL, SQL Server, Oracle, MySQL 8+, SQLite 3.25+). PostgreSQL also offers a terser DISTINCT ON (driver_id) ... ORDER BY driver_id, total_tip DESC, rider_id after the aggregation — same result, Postgres-only. The window-function form shown is the portable, interview-safe choice; DISTINCT ON is a nice "I also know the Postgres shortcut" addition.

Senior signal: for "top-1 per group," sub-aggregate to the comparison grain first, then rank with ROW_NUMBER() OVER (PARTITION BY group ORDER BY metric DESC, tiebreak) — folding the tie-break rule directly into the secondary sort key — and use ROW_NUMBER not RANK so a tie collapses to exactly one winning row per group.
