A finance team runs a month-end reconciliation: every customer row carries a stored recorded_outstanding snapshot, and the team wants to recompute that balance from first principles — total orders minus total payments — and surface only the customers whose stored figure disagrees with the derived one. The interviewer is probing whether you can fan out two independent aggregations, re-join them with LEFT JOINs so customers with no orders or no payments still appear, and treat the resulting NULLs as zero.

C04 is the all-missing edge case — absent from both orders and payment. C05 has orders but no payment row. Both must survive the joins; their missing sums have to coalesce to 0, or the discrepancy arithmetic produces NULL and the customer silently disappears.

2. The SQL — pre-aggregate, LEFT JOIN, COALESCE

WITH o AS (
  SELECT cust_code, SUM(ord_amount) AS total_orders FROM orders GROUP BY cust_code
),
p AS (
  SELECT cust_code, SUM(paid_amount) AS total_payments FROM payment GROUP BY cust_code
)
SELECT c.cust_code,
       c.recorded_outstanding,
       COALESCE(o.total_orders,0)   AS total_orders,
       COALESCE(p.total_payments,0) AS total_payments,
       COALESCE(o.total_orders,0) - COALESCE(p.total_payments,0) AS calculated_balance,
       c.recorded_outstanding - (COALESCE(o.total_orders,0) - COALESCE(p.total_payments,0)) AS discrepancy
FROM customer c
LEFT JOIN o ON o.cust_code = c.cust_code
LEFT JOIN p ON p.cust_code = c.cust_code
WHERE c.recorded_outstanding - (COALESCE(o.total_orders,0) - COALESCE(p.total_payments,0)) <> 0
ORDER BY ABS(c.recorded_outstanding - (COALESCE(o.total_orders,0) - COALESCE(p.total_payments,0))) DESC, c.cust_code;

The o and p CTEs collapse orders and payments to one row per customer before joining. The customer table drives two LEFT JOINs, so every customer is kept; COALESCE(...,0) converts the NULL a missing match produces into a real zero, and the arithmetic builds calculated_balance and discrepancy.

4. Strategic explanations

Why pre-aggregate in CTEs before joining

If you joined customer to raw orders and raw payment first and then aggregated, the join would multiply rows — a customer with 2 orders and 3 payments yields 6 combined rows, and SUM(ord_amount) would count each order three times. This is the classic fan-out double-counting trap. Collapsing each side to one row per customer first means the joins are strictly one-to-one and the sums stay honest. Aggregate, then join — never join, then aggregate across independent dimensions.

LEFT JOIN + COALESCE is what keeps C04 and C05

An INNER JOIN to orders would drop C04 (no orders) entirely, and an inner join to payment would drop C04 and C05. The customer table must drive the query and both joins must be LEFT, so every customer is preserved. A non-matching LEFT JOIN yields NULL for the right-side columns, so COALESCE(...,0) is mandatory: "no orders" means a balance contribution of 0, not unknown. The spec says exactly this — "NULLs from missing orders/payments treated as 0".

The discrepancy expression is repeated — and why that smells

The same recorded_outstanding - (orders - payments) formula appears in the SELECT, the WHERE, and the ORDER BY — three times. SQL does not let you reference a SELECT alias in WHERE (the select list is evaluated after the filter), hence the duplication. The clean fix is to wrap the computed columns in another CTE or subquery and filter/sort on the alias from the outside — one definition, no drift risk. Mentioning this refactor unprompted signals maintainability awareness.

Filtering "non-zero" — <> 0 and the NULL it cannot see

discrepancy <> 0 keeps genuine mismatches and drops exact matches. Because every input is coalesced to a number, discrepancy is never NULL here, so <> 0 is safe. Without the COALESCEs, a missing match would make the subtraction NULL, NULL <> 0 would be unknown, and that customer would be filtered out — the very customers (C04, C05) the reconciliation most needs to inspect. Coalescing first is what makes the comparison total.

Ordering by ABS(discrepancy) — magnitude, not sign

ORDER BY ABS(discrepancy) DESC ranks the worst mismatches first regardless of direction — a stored balance 5000 too high and one 5000 too low are equally urgent. Sorting by the signed value would bury large negative discrepancies at the bottom. The cust_code tie-break makes the order deterministic when two customers have equal-magnitude gaps. For an exception report, "biggest problem first" is the right default.

Senior signal: pre-aggregate orders and payments in separate CTEs to avoid join fan-out double-counting, drive the query off customer with LEFT JOINs and COALESCE(...,0) so zero-activity customers survive and their NULL sums become real zeros, recognise that the repeated discrepancy formula should be defined once in a wrapping CTE, and sort exception reports by ABS() so the largest mismatch — either direction — surfaces first.
