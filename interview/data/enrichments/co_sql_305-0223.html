Autodesk's billing team exports a sales feed where amounts must show exactly two decimal places — cents, never a stray third digit from floating-point drift. The fix is a one-line CAST to DECIMAL(10,2), but the interviewer is probing whether you know that the cast rounds rather than truncates, what the precision and scale digits actually mean, and where this can overflow or lose money.

1. The source data — sales

amount
123.456
99.9
1000
0.005
42.125

Each row stresses the cast differently: 123.456 has a third decimal that must round, 99.9 needs a trailing zero added, 1000 is a whole number, and 0.005 / 42.125 sit exactly on a rounding half — the values that expose the rounding rule.

2. The SQL — CAST to DECIMAL(10,2)

SELECT CAST(amount AS DECIMAL(10,2)) AS amount_2dp FROM sales;

DECIMAL(10,2) means a fixed-point number with 10 total significant digits, of which 2 are after the decimal point — so up to 8 digits before it.

3. The resulting analysis

amount_2dp
123.46
99.90
1000.00
0.01
42.13

123.456 became 123.46 — the cast rounded, it did not chop. 99.9 gained a trailing zero, 1000 shows full cents. The two halves both rounded up: 0.005 → 0.01 and 42.125 → 42.13 — Postgres rounds halves away from zero for numeric casts.

4. Strategic explanations

The cast rounds — this is not truncation

The single most common misconception: casting to a smaller scale must round, not truncate. 123.456 casts to 123.46, not 123.45. If you genuinely need to drop digits without rounding (rare, but it happens in some fee calculations), CAST is the wrong tool — use TRUNC(amount, 2), which would give 123.45. Know which one the business rule wants.

What the half-rounding rule actually is

0.005 and 42.125 both rounded up because Postgres uses round-half-away-from-zero for numeric/decimal casts. This is not universal — ROUND on double precision in Postgres uses banker's rounding (round-half-to-even), and other engines differ. For money, work in the numeric/decimal domain so rounding is deterministic; never let a value pass through float on the way to a financial report.

Precision vs scale — and the overflow risk

In DECIMAL(p, s), p is total digits and s is digits right of the point — so DECIMAL(10,2) caps the integer part at 10 − 2 = 8 digits (max 99,999,999.99). A sale of 100,000,000.00 would raise a numeric overflow error, not silently wrap. If amounts can be large, size the precision generously: DECIMAL(18,2). Picking p too small is a latent production bug.

Formatting for display vs storing a number

CAST(... AS DECIMAL(10,2)) yields a numeric value with scale 2 — correct for arithmetic and storage. If the requirement is a display string with thousands separators or a currency symbol, that is to_char:

SELECT to_char(amount, 'FM999,999,990.00') AS amount_display FROM sales;
-- 123.456 -> '123.46', 1000 -> '1,000.00'

Keep the distinction sharp: cast for a number you will compute on, to_char for text a human reads. Never store the formatted string and do math on it later.

NULL and the rounding-bias caveat

CAST(NULL AS DECIMAL(10,2)) is just NULL — the cast does not invent a 0. Wrap with COALESCE if the feed needs a value. And a subtler point for finance: rounding every line independently can make the rounded lines fail to sum to the rounded total (the classic penny-rounding problem). If totals must reconcile, round once at the total or apply a largest-remainder allocation — do not assume per-row rounding is free.

Senior signal: state that the cast rounds (not truncates) and name TRUNC as the alternative, explain that DECIMAL(10,2) allows only 8 integer digits and overflows loudly past that, keep money in the numeric domain for deterministic half-rounding, and separate a numeric cast from a to_char display string.
