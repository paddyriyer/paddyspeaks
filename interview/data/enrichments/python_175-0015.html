Google asks Accounts Merge because it is identity resolution — the real problem of deciding when two records describe the same person. A user signs up multiple times with overlapping emails; you must stitch those accounts into one. The interviewer is testing whether you recognize a graph-connectivity problem hiding inside a list of string lists, and whether you reach for Union-Find rather than hand-rolling a graph traversal.

1. The input data

A list of accounts, each [name, email1, email2, ...]:

accounts = [
  ['John', 'j1@x.com', 'j2@x.com'],
  ['John', 'j3@x.com'],
  ['John', 'j1@x.com', 'j3@x.com'],
  ['Mary', 'm@x.com'],
]

The first and third Johns share j1@x.com; the third also has j3@x.com, which bridges to the second John. All three Johns are one person. Mary stands alone. Note: a shared name means nothing — only a shared email merges accounts.

2. The approach

This is Union-Find (disjoint-set union) over emails. Think of every email as a graph node; emails listed in the same account are connected. parent maps each email to its set representative. find(x) climbs to the root and applies path compression — parent[x] = parent[parent[x]] — flattening the tree so future lookups are near-instant. For each account, the code registers every email with parent.setdefault(e, e) (a new email is its own root) and records its owner in email_name, then unions each email with the account's first email via parent[find(e)] = find(acc[1]) — pointing one root at the other. Because emails are shared across accounts, those unions transitively chain separate accounts into one component. Finally, a defaultdict(list) buckets every email under find(e), and each bucket becomes [name] + sorted(emails).

3. The output

The three Johns collapse into one component; emails are sorted within each merged account:

[['John', 'j1@x.com', 'j2@x.com', 'j3@x.com'],
 ['Mary', 'm@x.com']]

Trace: account 0 unions j1–j2. Account 1 leaves j3 in its own set. Account 2 unions j1–j1 (no-op) then j3–j1 — that edge bridges the j3 set into the j1/j2 set. Result: one John with all three emails.

4. Strategic explanations

Complexity

With path compression, find runs in near-constant amortized O(α(n)) — the inverse Ackermann function, effectively ≤ 4 for any realistic input. Processing all emails is O(n · α(n)); the final sort dominates at O(n log n) for n total emails. Space is O(n) for the two maps. The interviewer wants to hear "the sort is the expensive part, the union-find is essentially linear."

The idiomatic alternative — DFS over a graph

You can model each account as a graph (connect acc[1] to every other email), then DFS/BFS to find connected components. It is equally correct and often easier to explain, but Union-Find is more compact, has no recursion-depth risk, and is the canonical answer. Naming both — and why you picked DSU — is the senior move. Note: this solution unions by pointing arbitrarily; adding union by rank is the textbook companion optimization.

The edge case — same name, different people; same email, "different" names

Two distinct people both named "John" with no shared email must stay separate — the algorithm gets this right because it keys on emails, never names. The mirror case: the problem guarantees one email belongs to one person, so email_name overwriting is harmless; if that guarantee were dropped, the merge would be ambiguous — worth flagging to the interviewer.

The follow-up — streaming accounts, or scale beyond memory

"Accounts arrive one at a time and never stop." Union-Find is naturally incremental — each new account just runs more unions, no rebuild needed, which is a genuine advantage over recomputing a graph. "Now it does not fit in memory" pushes toward a distributed connected-components job (e.g. iterative label propagation in Spark/MapReduce).

The trap — merging by name, or one pairwise pass

Grouping accounts by the name string is the headline wrong answer — it both over-merges (two unrelated Johns) and under-merges (it ignores the email bridges). The other trap is a single pairwise sweep "merge account i with account j if they share an email": that misses transitive chains where A–B and B–C share emails but A and C do not. Transitivity is exactly what Union-Find delivers for free.

Senior signal: a senior names this as connected components, picks Union-Find with path compression, keys strictly on emails (never names), and explains that the final sort — not the unions — sets the O(n log n) bound — a junior groups by name or does one non-transitive pairwise pass.
