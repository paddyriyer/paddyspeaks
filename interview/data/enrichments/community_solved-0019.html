A Community question rooted in a Google file-store problem: count how often each word appears across files whose name contains "draft". The original was a SQL Server puzzle leaning on CROSS APPLY STRING_SPLIT — here the real work is tokenising a free-text column into rows, stripping punctuation so "fox." and "fox" are the same word, and only then aggregating.

1. The source data — google_file_store

filenamecontents
draft1.txtthe quick brown fox the fox
draft2.txtlazy dog, the dog.
final.txtthis should be ignored ignored ignored

Three things to notice. draft2.txt has a comma after "dog," and a period after "dog." — without scrubbing, those become three distinct tokens. final.txt does not match %draft% and must contribute nothing. And "the" spans both draft files, so the count has to aggregate across rows, not within one.

2. The SQL — UNNEST(string_to_array(…))

WITH t1 AS (
    SELECT word AS words
    FROM google_file_store, UNNEST(string_to_array(REPLACE(REPLACE(contents, ',', ''), '.', ''), ' ')) AS word
    WHERE filename LIKE '%draft%'
)
SELECT words AS value, COUNT(*) AS No_Of_Occurrences
FROM t1
GROUP BY words;

The nested REPLACE deletes commas and periods; string_to_array(…, ' ') splits on spaces into a text array; UNNEST in the FROM clause explodes that array into one row per word, joined laterally to each file.

3. The resulting analysis

valueno_of_occurrences
brown1
dog2
fox2
lazy1
quick1
the3

"dog" is 2 — "dog," and "dog." both scrubbed to the same token, exactly the point of the punctuation strip. "the" is 3: twice in draft1, once in draft2, aggregated across files. Nothing from final.txt appears. The result has no defined row order — there is no ORDER BY.

4. Strategic explanations

The implicit lateral join — the heart of the trick

FROM google_file_store, UNNEST(…) is comma-join syntax, but because UNNEST references contents from the table on its left, Postgres treats it as an implicit LATERAL join: the set-returning function is evaluated per file row, and each file fans out into as many rows as it has words. This is the Postgres answer to SQL Server's CROSS APPLY STRING_SPLIT. Writing it explicitly makes the intent obvious:

FROM google_file_store f
CROSS JOIN LATERAL UNNEST(string_to_array(...)) AS word
Why scrub punctuation before splitting

Split first and "dog," is a token distinct from "dog"; they would land in separate groups and the count for "dog" would read 1 and 1 instead of 2. The nested REPLACE removes commas and periods up front so tokenisation sees clean words. It is a deliberately minimal cleaner, though — semicolons, parentheses, em-dashes all slip through. A regex-based regexp_split_to_table is more robust:

SELECT lower(w) AS value, COUNT(*)
FROM google_file_store,
     regexp_split_to_table(contents, '[^a-zA-Z0-9]+') AS w
WHERE filename ILIKE '%draft%' AND w <> ''
GROUP BY lower(w);
The empty-token and case traps

Two consecutive spaces, or a leading/trailing space, make string_to_array emit an empty string '' as a "word" — it then becomes its own group with a real count. Defend with WHERE word <> ''. And matching is case-sensitive: "The" and "the" are different tokens. If the brief means words case-insensitively, fold with lower() before grouping — ask which the interviewer wants.

LIKE '%draft%' — what it really catches

filename LIKE '%draft%' matches "draft" anywhere in the name — draft1.txt, but also final_draft.txt or redraft.doc. It is also case-sensitive in Postgres, so Draft1.txt would be missed; use ILIKE for case-insensitive. If the rule is strictly "filename begins with draft", anchor it: LIKE 'draft%'. Confirm the matching rule rather than assuming.

The missing ORDER BY

The query has no ORDER BY, so the row order is whatever the executor produces — not stable across runs or engines. A word-frequency report almost always wants ORDER BY No_Of_Occurrences DESC, value so the most common words surface first. Add it; never rely on incidental ordering.

Senior signal: recognise UNNEST/regexp_split_to_table in the FROM as the lateral-join equivalent of CROSS APPLY STRING_SPLIT, scrub punctuation and filter empty tokens before grouping, decide case-folding deliberately, and add the ORDER BY a frequency report needs.
