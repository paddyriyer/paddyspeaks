Red Hat's revenue-ops team wants to know which January orders did not recur in February — the churn signal hiding between two monthly snapshots. The instinct is a LEFT JOIN ... WHERE IS NULL, but there is a cleaner, more declarative tool: the set operator EXCEPT. The interview is about knowing it exists, knowing it de-duplicates, and knowing how it treats NULLs differently from a join.

1. The source data — orders_jan and orders_feb

orders_jan.order_idorders_feb.order_id
10011003
10021005
10031006
10041007
1005

January holds five order IDs, February four. The overlap is {1003, 1005} — orders present in both months. The IDs unique to January are {1001, 1002, 1004}; those unique to February ({1006, 1007}) are not what this query returns — EXCEPT is directional, and that asymmetry is the whole point.

2. The SQL — EXCEPT

SELECT order_id FROM orders_jan
EXCEPT
SELECT order_id FROM orders_feb;

EXCEPT takes the rows produced by the first SELECT and removes any that also appear in the second. The result is "in January, not in February".

3. The resulting analysis

order_id
1001
1002
1004

Three IDs survive — {1001, 1002, 1004}, the January orders with no February match. (The rows come back in no guaranteed order; EXCEPT does not sort.) The shared orders 1003 and 1005 are removed; the February-only orders 1006 and 1007 never had a chance to appear, because they are not in the left-hand set.

4. Strategic explanations

EXCEPT is directional — flipping the operands flips the question

A EXCEPT B is not B EXCEPT A. The query above answers "lost orders" (in Jan, gone by Feb). Swap the tables and you get "new orders" (in Feb, absent in Jan):

SELECT order_id FROM orders_feb
EXCEPT
SELECT order_id FROM orders_jan;
-- returns {1006, 1007}

State which direction the business asked for. Getting churn and acquisition backwards is a silent, plausible-looking error.

EXCEPT de-duplicates — EXCEPT ALL does not

Plain EXCEPT behaves like DISTINCT: if January listed order 1002 three times, the result still shows 1002 once. That is usually right for "which order IDs". But if you are doing multiset arithmetic — "how many occurrences remain after subtraction" — you want EXCEPT ALL, which subtracts row counts: three copies minus one copy leaves two. Know both forms and pick on purpose.

The LEFT JOIN ... IS NULL equivalent — and where it differs

The classic anti-join writes the same idea procedurally:

SELECT j.order_id
FROM orders_jan j
LEFT JOIN orders_feb f ON j.order_id = f.order_id
WHERE f.order_id IS NULL;

It returns the same IDs here, but it is not identical. The anti-join does not de-duplicate — duplicate January rows come through duplicated. And NOT IN (SELECT order_id FROM orders_feb), a third tempting form, is a genuine trap: if February's list contains a single NULL, NOT IN returns zero rows for everything, because x <> NULL is unknown. EXCEPT sidesteps that NULL landmine entirely — it treats two NULLs as equal and matches them.

How EXCEPT handles NULL rows

For set operators, NULL is not special: EXCEPT uses NOT DISTINCT semantics, so a NULL order_id in January is removed if February also has a NULL order_id. This is the opposite of how = behaves in a JOIN or NOT IN. If order IDs can be NULL, that difference is the deciding factor in which construct is correct — and it is exactly the kind of detail a senior is expected to volunteer.

Column compatibility and ordering

Both SELECTs must return the same number of columns with compatible types — EXCEPT compares whole rows positionally, not by name. And the result has no inherent order; if revenue-ops wants a sorted list, append ORDER BY order_id after the entire EXCEPT expression (not after the first SELECT). For very large snapshots, EXCEPT and the anti-join usually plan similarly — both can hash — so the choice is about correctness and clarity, not speed.

Senior signal: reach for EXCEPT as the declarative tool for set difference, state that it is directional and de-duplicating (with EXCEPT ALL as the multiset variant), and explain that it matches NULLs as equal — the precise behaviour that makes it safer than NOT IN when the compared column is nullable.
