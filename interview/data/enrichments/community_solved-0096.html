A music-streaming service wants its three most-played songs overall, and for each one the Free-vs-Premium split: what percentage of plays came from each tier, and which tier dominates. The interviewer is probing a two-stage pattern — rank to find the top N, then re-aggregate only those N with conditional aggregation — plus careful percentage arithmetic and tie handling.

Song 10 is the edge case: a perfect 2-Premium / 2-Free tie, so "most streamed tier" is genuinely ambiguous — the query must resolve it explicitly rather than picking arbitrarily.

2. The SQL — rank top-3, then conditional aggregation

WITH top_three AS (
  SELECT song_id, COUNT(stream_id) AS stream_count
  FROM song_streams
  GROUP BY song_id
  ORDER BY stream_count DESC
  LIMIT 3
), t2 AS (
  SELECT song_id,
         SUM(CASE WHEN subscription_type = 'Premium' THEN 1 ELSE 0 END) AS pre_sub_count,
         SUM(CASE WHEN subscription_type = 'Free' THEN 1 ELSE 0 END) AS fre_sub_count,
         COUNT(stream_id) AS stream_count
  FROM song_streams
  WHERE song_id IN (SELECT song_id FROM top_three)
  GROUP BY song_id
)
SELECT song_id,
       ROUND((100.0*pre_sub_count/stream_count)::numeric, 2) AS premium_percent,
       ROUND((100.0*fre_sub_count/stream_count)::numeric, 2) AS fre_percent,
       CASE WHEN pre_sub_count > fre_sub_count THEN 'Premium'
            WHEN pre_sub_count < fre_sub_count THEN 'Free'
            ELSE 'Both' END AS most_streamed_sub,
       stream_count
FROM t2
ORDER BY stream_count DESC;

4. Strategic explanations

Why two stages, not one

"Top 3 overall" must be decided on total streams before the Free/Premium breakdown happens. The top_three CTE ranks and clips to three song ids; t2 then re-aggregates the full table but WHERE song_id IN (top_three). Trying to do it in one pass — e.g. LIMIT 3 on a query already split by tier — would cut rows, not songs.

Conditional aggregation for the tier split

SUM(CASE WHEN subscription_type = 'Premium' THEN 1 ELSE 0 END) counts Premium rows in one pass; the Free counterpart counts the rest. This pivots the tier column into two numeric columns without a self-join. COUNT(stream_id) gives the denominator. The FILTER form — COUNT(*) FILTER (WHERE subscription_type = 'Premium') — is the cleaner Postgres equivalent.

The integer-division trap in the percentage

100.0 * pre_sub_count / stream_count deliberately starts with 100.0 (a numeric literal). Had it been 100 * pre_sub_count / stream_count with all integers, 2/4 would floor to 0 and every sub-100% figure would collapse to zero. Forcing one operand to a decimal makes the whole expression numeric. The trailing ::numeric cast then lets ROUND(…, 2) apply — in Postgres two-argument ROUND requires numeric, not double precision.

Handling the tie — the third CASE branch

Song 10's 2–2 split makes "most streamed tier" undefined. The ELSE 'Both' branch names that case instead of letting a two-way CASE silently fall through to NULL or arbitrarily favour one side. Whether the label should be 'Both', 'Tie', or NULL is a product decision — the senior move is to ask, then make the code say it explicitly.

The LIMIT 3 tie ambiguity at the boundary

ORDER BY stream_count DESC LIMIT 3 is non-deterministic if several songs tie for 3rd place — the engine picks an arbitrary three. If the rule is "include all songs tied at the cutoff", switch to RANK() OVER (ORDER BY stream_count DESC) and filter rank <= 3; that keeps every tied song. Mention this even when the sample data has a clean ranking.

Senior signal: structure this as rank-then-re-aggregate, force numeric arithmetic with a 100.0 literal so percentages do not floor to zero, give the Free/Premium tie an explicit 'Both' branch instead of a silent fall-through, and note that LIMIT 3 is non-deterministic at a tied cutoff — RANK() resolves it.
