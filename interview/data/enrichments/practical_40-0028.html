A growth analyst stares at a jagged daily-revenue chart and wants the trend, not the noise: each date annotated with its 7-day trailing average — the day itself plus the six before it. That smoothing is the textbook use of a moving-window aggregate, and the interviewer is checking whether you can frame a window with ROWS BETWEEN, and whether you understand it counts rows, not calendar days.

Two edge cases shape the result. At the start (03-01 to 03-05) there are fewer than 7 prior rows, so the window is short. And there is a 3-day gap before 03-09 — revealing that a ROWS window walks physical rows, so its 7 rows can span far more than 7 calendar days.

2. The SQL — AVG over a ROWS BETWEEN frame

SELECT metric_date,
       value,
       AVG(value) OVER (
           ORDER BY metric_date
           ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
       ) AS avg_7d
FROM daily_metrics
ORDER BY metric_date;

ORDER BY metric_date sequences the rows; ROWS BETWEEN 6 PRECEDING AND CURRENT ROW defines a sliding 7-row frame — the current row plus the six before it. AVG(value) recomputes over that frame for every row.

The opening rows average only what exists — 03-01 equals its own value. The crucial row is 03-10: its "7-day" average covers 03-01 through 03-10, a 10-day calendar span, because three days are missing and ROWS counts rows, not dates.

4. Strategic explanations

Why ROWS is a count of rows, not days — the headline trap

ROWS BETWEEN 6 PRECEDING AND CURRENT ROW is a physical frame: the six rows immediately before, regardless of their dates. On gap-free daily data that is identical to a 7-calendar-day window. The instant a day is missing, the two diverge: at 03-10 the frame reaches back to 03-01. If the metric must be a true calendar week, this is wrong — and quietly wrong, since the output still looks plausible.

The calendar-correct alternative — RANGE with an interval

For a genuine 7-day average use RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW. RANGE frames by the value of the ORDER BY column — here the date — so it includes exactly the rows whose date falls in the trailing 6 days and naturally excludes the gap. The honest senior answer is: choose ROWS when the data is dense and you mean "last 7 observations"; choose RANGE when you mean "last 7 calendar days". State which definition the business wants.

The ramp-up at the start is a feature, not a bug

The first six rows have fewer than seven rows behind them, so their average is computed over a partial frame — 03-01 averages just itself. That is correct windowing behaviour. If a partial average is misleading for the consumer (a "7-day average" of one day is barely a trend), the senior move is to NULL it out: wrap in a CASE on COUNT(*) OVER (same frame) = 7, or filter the warm-up rows. Decide deliberately rather than shipping noisy early values.

The implicit default frame trap

If you write AVG(value) OVER (ORDER BY metric_date) with no frame clause, the default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — a running cumulative average from the start, not a 7-row trailing one. Omitting the explicit ROWS BETWEEN silently gives a completely different metric. Always spell out the frame when you want a fixed trailing window.

Dialect notes

ROWS BETWEEN n PRECEDING AND CURRENT ROW is supported everywhere with window functions — Postgres, MySQL 8+, SQLite 3.28+, Snowflake, SQL Server. The calendar-correct RANGE … INTERVAL form is not universal: Postgres and Snowflake support it, SQLite does not, and SQL Server does not allow interval offsets in RANGE. On those engines a true date window needs a self-join or a pre-densified calendar table.

Senior signal: build a moving average with an explicit AVG(value) OVER (ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW), know that ROWS counts observations not days so gaps distort it, reach for RANGE … INTERVAL when a true calendar window is required, and handle the partial-frame warm-up rows deliberately instead of shipping them as noise.
