Accenture's analytics team publishes a weekly metrics pack, and the headline number every client asks for is average order value — AOV. It looks like a one-line query, and it is. But the one line hides three decisions that change the number, and an interviewer asking this is really probing whether you make those decisions on purpose.

1. The source data — orders

2102$80completed
3101$200completed

Six orders — but one has a NULL order_total (a pending order not yet priced), and one is cancelled. Those two rows are the whole interview.

2. The SQL — AVG(order_total)

SELECT AVG(order_total) AS avg_order_value FROM orders;

One aggregate over every row in the table. No GROUP BY, so the result is a single row.

3. The resulting analysis

avg_order_value
120

The five priced orders — 120, 80, 200, 50, 150 — sum to 600, and 600 ÷ 5 = 120. The pending order never entered the arithmetic. Whether that is correct depends on what the client meant.

4. Strategic explanations

AVG silently skips NULLs

AVG(order_total) is really SUM(order_total) / COUNT(order_total), and COUNT(column) counts only non-NULL values. So the NULL row is dropped from both the numerator and the denominator — the average is over 5 rows, not 6. That is usually what you want, but say it out loud. If a pending order should count as a $0 sale, fold the NULL in explicitly:

SELECT AVG(COALESCE(order_total, 0)) AS avg_order_value FROM orders;
-- 600 / 6 = 100, not 120

Same column, same table, a $20 swing in the headline metric — purely from how NULL is treated.

The AVG vs SUM / COUNT(*) trap

A common "equivalent" rewrite is wrong:

SELECT SUM(order_total) / COUNT(*) AS avg_order_value FROM orders;
-- 600 / 6 = 100

COUNT(*) counts rows (6), while AVG divides by non-NULL values (5). The two agree only when the column has no NULLs. If you ever hand-roll an average, pair SUM(col) with COUNT(col) — never COUNT(*).

The cancelled order — the filter the interviewer is waiting for

Order 4 is cancelled but still priced at $50, so it is sitting inside the average. AOV almost always means realised revenue per order, which excludes cancellations:

SELECT AVG(order_total) AS avg_order_value
FROM orders
WHERE status = 'completed';
-- AVG(120, 80, 200, 150) = 137.5

Confirm the definition before writing SQL: "should this be over all orders, or only completed ones?" Getting the filter right matters more than the aggregate.

Edge cases: empty tables and integer columns

AVG over a table with no matching rows returns NULL, not 0 — wrap it in COALESCE(AVG(…), 0) if the dashboard needs a number. And if order_total were an INTEGER column, some engines do integer division and truncate the average; cast first with AVG(order_total::numeric) to be safe.

Which column is the source of truth?

The schema ships both order_total and amount. They are often near-duplicates that drift apart — one pre-discount, one post-discount; one with tax, one without. Don't guess: ask which column the business treats as the order value, and note the assumption in a comment.

Senior signal: state that AVG ignores NULLs and decide on purpose whether a NULL means "skip" or "$0", never substitute COUNT(*) for COUNT(col) in a hand-rolled average, add the status = 'completed' filter that turns a raw average into a real AOV, and flag the empty-table NULL and the order_total/amount ambiguity before anyone has to ask.
