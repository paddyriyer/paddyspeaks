Broadcom's analytics platform supports an old MySQL 5.7 cluster where window functions don't exist. The PM still needs the top-3 products by revenue per category for a quarterly catalog refresh. The trick: a correlated count subquery — for each row, count how many rows in the same category have higher revenue, and keep only those with fewer than N higher. This is the canonical "top-N per group" pattern without ROW_NUMBER or RANK.

2. The SQL — correlated count subquery

SELECT s1.category, s1.product_id, s1.revenue
FROM sales s1
WHERE (
  SELECT COUNT(*) FROM sales s2
  WHERE s2.category = s1.category
    AND s2.revenue  > s1.revenue
) < 3
ORDER BY s1.category, s1.revenue DESC;

4. Strategic explanations

The clever-but-slow trick — reading the predicate

For each outer row, count how many same-category rows have strictly higher revenue:

rowcount of higher-revenue siblingskept?
P_001 ($50K)0 — nobody beats ityes (rank 1)
P_002 ($45K)1 — just P_001yes (rank 2)
P_003 ($30K)2 — P_001, P_002yes (rank 3)
P_004 ($20K)3 — outside top 3NO

"Fewer than 3 beat me" = "I'm one of the top 3". The pattern generalises: for top-N, use < N.

The tie problem — this is RANK-equivalent, not DENSE_RANK-equivalent

If two rows tie at $30K, both have the same higher-revenue count — both pass the < 3 predicate, you get 4 rows, not 3. That's RANK-style tie-emission. To make it ROW_NUMBER-style (exactly N rows, arbitrary tiebreak), add a tiebreaker column to the inner predicate:

WHERE (
  SELECT COUNT(*) FROM sales s2
  WHERE s2.category = s1.category
    AND (s2.revenue, s2.product_id) > (s1.revenue, s1.product_id)
) < 3

The composite tuple comparison breaks ties deterministically by product_id.

The modern equivalent — window functions, when available
-- Modern: ROW_NUMBER (no ties), RANK (ties allowed, gaps), DENSE_RANK (ties, no gaps)
SELECT category, product_id, revenue FROM (
  SELECT *, DENSE_RANK() OVER (PARTITION BY category ORDER BY revenue DESC) AS rk
  FROM sales
) WHERE rk <= 3;

Same answer, one sort instead of N² predicate evaluations. Use this whenever the dialect supports it (Postgres, BigQuery, Snowflake, MySQL 8+, SQLite 3.25+, SQL Server 2005+).

Performance — why the original is dangerous at scale

The correlated-subquery pattern is conceptually O(n²): for each of N outer rows, scan the table again counting matches. Without an index on (category, revenue) it's a full re-scan per row. With that index, the planner can use it to short-circuit the count. Always check:

EXPLAIN ANALYZE SELECT ... -- look for "Index Scan" not "Seq Scan" on the inner

For tables with millions of rows, the window-function form is typically 10-100x faster.

The "top N or all if fewer than N" follow-up

What if a category has only 2 products? Both forms return all 2 — the predicate is naturally robust. No special handling needed.

Top-N-per-group with secondary aggregations

Real ask: "top 3 products per category, plus their share of category revenue."

WITH ranked AS (
  SELECT category, product_id, revenue,
    DENSE_RANK() OVER (PARTITION BY category ORDER BY revenue DESC) AS rk,
    SUM(revenue) OVER (PARTITION BY category) AS category_total
  FROM sales
)
SELECT category, product_id, revenue,
  ROUND(100.0 * revenue / category_total, 1) AS pct_of_category
FROM ranked WHERE rk <= 3;
The "lateral-join" alternative — Postgres / SQL Server
SELECT s.category, top.product_id, top.revenue
FROM (SELECT DISTINCT category FROM sales) s
CROSS JOIN LATERAL (
  SELECT product_id, revenue FROM sales
  WHERE category = s.category
  ORDER BY revenue DESC LIMIT 3
) top;

One of the few cases where LATERAL is cleaner than window functions — especially when the LIMIT clause uses different sort orders per outer row.

The MySQL 5.7 reality — user-variables hack (avoid in interviews)

Pre-window-function MySQL had a notorious "session variable" trick to fake ROW_NUMBER. Don't reach for it in an interview — mention it exists, then either use the correlated-count form (portable, slow) or upgrade the database. The variable trick has surprising semantics about evaluation order and breaks after MySQL 8 deprecated reading-and-writing the same variable in a SELECT.

Senior signal: trace the "fewer than N beat me" predicate explicitly, name the result as RANK-style (ties emit) and contrast with ROW_NUMBER-style (composite tiebreaker), pre-empt the perf trap with the index requirement, and offer the modern DENSE_RANK form as the upgrade path. The LATERAL-join alternative shows depth.
