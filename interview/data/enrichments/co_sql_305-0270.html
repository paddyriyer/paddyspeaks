GitLab is reconciling two order systems — say a billing pipeline and a fulfilment pipeline — and needs the order IDs that appear in both. The instinct is a join, but the reference solution reaches for INTERSECT. That choice is the interview: INTERSECT and a join answer almost-the-same question, and the difference shows up the instant either table has duplicate IDs.

1. The source data — orders_a and orders_b

orders_a.order_id
1
2
3
4
2
orders_b.order_id
2

3

Order 2 appears twice in A; order 3 appears twice in B. Those duplicates are the whole question — they are what make INTERSECT and a join diverge. Orders 1 and 4 are A-only; order 5 is B-only.

2. The SQL — INTERSECT

SELECT order_id FROM orders_a
INTERSECT
SELECT order_id FROM orders_b;

INTERSECT returns rows present in both result sets — and, like all the set operators, it removes duplicates from its output.

3. The resulting analysis

order_id
2
3

Just orders 2 and 3 — the IDs in both tables. Despite order 2 appearing twice in A and order 3 twice in B, each shows up exactly once. INTERSECT deduplicated for free. Row order is not guaranteed; add an ORDER BY if the consumer needs one.

4. Strategic explanations

INTERSECT vs an inner join — the duplicate trap

The "equivalent" join is not equivalent:

SELECT a.order_id
FROM orders_a a
JOIN orders_b b ON a.order_id = b.order_id;

This is a Cartesian product per matching key. Order 2 (2 rows in A × 1 in B) yields 2 rows; order 3 (1 in A × 2 in B) yields 2 rows — the join returns four rows, not two. To match INTERSECT you must add DISTINCT: SELECT DISTINCT a.order_id FROM …. INTERSECT bakes the de-duplication in; the join makes you remember it. That is the senior point: pick the operator whose default behaviour is the requirement.

INTERSECT vs INTERSECT ALL

If you genuinely want duplicates preserved — matched pairwise — that is INTERSECT ALL. It returns each value min(count_in_a, count_in_b) times. Order 2 is min(2, 1) = 1 row; order 3 is min(1, 2) = 1 row — here it happens to match plain INTERSECT, but if A had order 2 three times and B twice, INTERSECT ALL would emit it twice while INTERSECT emits it once. Know both exist and which one the question wants.

EXISTS — the scalable alternative

On large tables a semi-join via EXISTS is often the planner's friend and reads clearly as "A rows that have a match in B":

SELECT DISTINCT a.order_id
FROM orders_a a
WHERE EXISTS (SELECT 1 FROM orders_b b
              WHERE b.order_id = a.order_id);

EXISTS stops at the first match — it never multiplies rows — so it sidesteps the join's fan-out. You still need DISTINCT because A itself has duplicate 2s.

The NULL gotcha in set operators

Unlike =, set operators treat NULL as a matchable value: NULL is "the same as" NULL for INTERSECT. So if both tables contain a NULL order_id, INTERSECT returns one NULL row — whereas a join on a.order_id = b.order_id would never match those NULLs (NULL = NULL is unknown). This is a real, often-surprising behavioural difference; mention it.

Column count and type alignment

Set operators match by position, not name, and both sides must have the same number of columns with compatible types. SELECT order_id, customer_id … INTERSECT SELECT order_id … fails with a column-count error. With a single ID column that is a non-issue here, but it is the first thing that breaks when someone extends the query.

Senior signal: choose INTERSECT because its built-in de-duplication matches "IDs in both", and explain that a naive join fans out on duplicate keys and needs DISTINCT to agree; know INTERSECT ALL for pairwise multiplicity and that set operators — unlike = — treat NULL as matchable.
