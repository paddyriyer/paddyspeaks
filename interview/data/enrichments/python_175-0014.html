Google asks Continuous Subarray Sum to test whether you know the prefix-sum-with-modular-arithmetic trick. Naively, "is there a contiguous run summing to a multiple of k?" invites checking every subarray. The elegant answer never enumerates subarrays at all — it watches running-sum remainders. Recognizing that reframe is the entire interview.

1. The input data

An array of non-negative integers and a divisor k:

nums = [23, 2, 4, 6, 7]
k    = 6
checkSubarraySum(nums, k)

The question: does some contiguous subarray of length ≥ 2 have a sum that is a multiple of 6?

2. The approach

The key identity: if two prefix sums leave the same remainder mod k, the subarray strictly between those two indices has a sum divisible by k — because their difference is a multiple of k. So the code maintains a running sum s and, at each index, computes m = s % k (falling back to raw s when k is 0). The dict seen maps each remainder to the first index where it appeared, seeded with {0: -1} so a prefix that is itself divisible by k is handled, and so the length math works from the very start. If the current remainder is already in seen, a qualifying subarray exists between seen[m]+1 and i; the code checks i - seen[m] >= 2 to enforce the length-≥-2 rule, returning True if so. Critically it only records a remainder the first time — keeping the earliest index maximizes the achievable subarray length.

3. The output

inumrunning sum ss % 6seenaction
——00{0: -1}seed
023235{0:-1, 5:0}new remainder
12251{0:-1, 5:0, 1:1}new remainder
24295—5 seen at 0; 2−0=2 ≥ 2 → True

Remainder 5 recurs at indices 0 and 2, so the subarray nums[1:3] = [2, 4] sums to 6 — a multiple of 6, length 2. The function returns True.

4. Strategic explanations

Complexity

O(n) time, one pass with O(1) work per element, and O(min(n, k)) space — the dict holds at most k distinct remainders. The brute force of summing every subarray is O(n²) (or O(n³) without a running sum); for the large arrays Google tests, only the linear remainder approach passes.

The idiomatic detail — store the first index, not the last

The single most important line is recording a remainder only when it is not already present. Keeping the earliest index is what lets the length check i - seen[m] >= 2 succeed; overwrite with the latest index and you can shrink a valid subarray below length 2 and wrongly answer False.

The edge case — k is zero, and the length rule

When k == 0 the code skips the modulo (no ZeroDivisionError) and compares raw sums — with non-negative inputs, two equal prefix sums mean a zero-sum run, i.e. consecutive zeros. The >= 2 guard is the other quiet trap: a single element equal to k satisfies "divisible" but fails the length requirement, so [6] alone must return False.

The follow-up — return the subarray, or exact sum k

"Don't just say yes — return the indices." Easy: the bounds are (seen[m]+1, i). The sibling problem "subarray summing to exactly k" (not a multiple) drops the modulo and stores prefix sums directly, looking up s - k — the same prefix-sum-in-a-hashmap skeleton, which is why interviewers often ask them back to back.

The trap — negative numbers and modulo

The clean logic relies on inputs being non-negative. With negative numbers, Python's % still returns a non-negative remainder (unlike C), so the approach mostly survives — but candidates who learned modulo in C-family languages often assume a negative result and "fix" it incorrectly, breaking a solution that was already right.

Senior signal: a senior reframes the problem as "two prefix sums with equal remainder," seeds the map with {0: -1}, stores the first index to preserve length, and pre-empts the k=0 and length-≥-2 edge cases — a junior enumerates every subarray at O(n²).
