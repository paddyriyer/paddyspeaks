Leaderboard queries that truncate at a hard row limit silently drop tied players at the boundary, creating an unfair and confusing user experience. FETCH FIRST … ROWS WITH TIES is the SQL:2008 standard solution: it extends the result to include all rows that are tied with the last returned row. The interviewer is testing whether you know this alternative to the more common (and flawed) LIMIT 2.

1. The source data — score

playerpoints
Ana95
Ben95
Cy88
Dee88
Eli70

The top score is 95, shared by Ana and Ben. If you ask for "top 2 rows" with a plain LIMIT 2, you get exactly Ana and Ben — but only because they happen to sit at the top. If instead the top 2 positions were occupied by three players all at 95, a plain LIMIT would arbitrarily exclude one. The question uses 95 with two players to illustrate the mechanism cleanly.

2. The SQL — FETCH FIRST 2 ROWS WITH TIES

SELECT player, points
FROM score
ORDER BY points DESC
FETCH FIRST 2 ROWS WITH TIES;

The engine sorts by points DESC, then returns the first 2 rows (Ana and Ben, both 95) and any additional rows that tie with the last returned score. Since 88 (Cy and Dee) is below 95, they are excluded. If a third player also had 95, WITH TIES would include all three.

3. The resulting analysis

playerpoints
Ana95
Ben95

Two rows returned. Both have the highest score (95). The question specifies a secondary sort of player ASC — Ana (A) sorts before Ben (B) within the tied score group. Cy and Dee (88) are below the tie threshold and are excluded. Eli (70) is excluded. Note: the solution as written does not include player ASC in the ORDER BY, so the tie-breaking between Ana and Ben is engine-dependent; adding , player ASC would make it deterministic.

4. Strategic explanations

WITH TIES vs LIMIT / OFFSET

LIMIT 2 is a non-standard shorthand that always returns at most 2 rows regardless of ties. FETCH FIRST 2 ROWS ONLY is the ISO SQL:2008 equivalent of LIMIT 2 — also always exactly 2 rows (or fewer). FETCH FIRST 2 ROWS WITH TIES is the inclusive variant: it keeps all rows that compare equal to the Nth row by the ORDER BY key. The word "ONLY" and "WITH TIES" are the two legal options after ROWS; omitting one defaults to ONLY in PostgreSQL.

What "tied with the last returned row" means precisely

The engine conceptually fills its 2-row bucket, notes the ORDER BY key value of the last row placed in the bucket (95 in this case), then scans further and includes all subsequent rows whose ORDER BY key equals that value. This means the actual returned count can exceed the requested N — it is unbounded in the worst case (e.g., if all 1 million rows have the same score). Production code should always pair WITH TIES with a deterministic secondary sort to control which rows are emitted when the primary key value is tied.

Relation to RANK() vs ROW_NUMBER()

FETCH FIRST N ROWS WITH TIES is semantically equivalent to WHERE RANK() OVER (ORDER BY points DESC) <= N. Both return every row whose rank is at or below N. By contrast, WHERE ROW_NUMBER() OVER (ORDER BY points DESC) <= N is the LIMIT 2 equivalent — exactly N rows, ties broken arbitrarily. Knowing this mapping demonstrates deep understanding of window-function semantics.

Dialect support

FETCH FIRST … ROWS WITH TIES is supported in PostgreSQL (since 13), SQL Server (since 2012 via TOP N WITH TIES syntax), Oracle (since 12c), and BigQuery. MySQL and MariaDB do not support it; the workaround is the RANK() window function. Snowflake supports LIMIT … WITH TIES (non-standard spelling). SQLite does not support it. This patchy support is why it remains a less-known feature despite being the correct tool.

SQL Server's alternative syntax

SQL Server uses SELECT TOP 2 WITH TIES player, points FROM score ORDER BY points DESC — the TOP keyword comes before the column list. PostgreSQL and Oracle use the FETCH FIRST … ROWS WITH TIES form in the trailing clause. Both are valid ISO extensions; knowing both syntaxes distinguishes candidates who have used multiple platforms.

Senior signal: use FETCH FIRST 2 ROWS WITH TIES instead of LIMIT 2 whenever ties at the boundary must be preserved; explain the semantic equivalence to RANK() <= N vs ROW_NUMBER() <= N; always add a deterministic secondary sort key to control output when the primary sort has ties; and note the absence of WITH TIES in MySQL and SQLite, where the RANK() workaround is required.
