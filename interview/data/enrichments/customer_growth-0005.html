A trust-and-safety team needs to flag customers whose return behaviour looks fraudulent — serial returners, refund bursts, customers whose return rate is suddenly spiking. The question (snapshot 2025-04-01) layers L7/L28/L90 return rates with a rolling-7-day "burst" detector and a priority-ordered fraud_flag. The interviewer is probing whether you can join two event streams (orders, returns), compute windowed rates with a zero-safe denominator, and find the densest 7-day cluster with a self-join.

2. The SQL — windowed return rates with a self-join burst detector

WITH params AS (SELECT DATE '2025-04-01' AS snapshot),
o_e AS (
    SELECT o.*, ((SELECT snapshot FROM params) - o.order_date::date) AS days_back
    FROM orders o
),
r_e AS (
    SELECT r.return_id, o.customer_id, r.return_date,
           ((SELECT snapshot FROM params) - r.return_date::date) AS days_back
    FROM returns r JOIN orders o ON r.order_id = o.order_id
),
return_pairs AS (
    SELECT a.customer_id, a.return_date AS anchor, COUNT(*) AS burst_count
    FROM r_e a
    JOIN r_e b ON b.customer_id = a.customer_id
              AND b.return_date >= a.return_date
              AND (b.return_date::date - a.return_date::date) < 7
    GROUP BY a.customer_id, a.return_date
),
burst AS (
    SELECT customer_id, MAX(burst_count) AS max_7d_return_burst
    FROM return_pairs GROUP BY customer_id
),
order_w AS (
    SELECT customer_id, COUNT(*) AS lifetime_orders,
        SUM(CASE WHEN days_back < 7  THEN 1 ELSE 0 END) AS l7_orders,
        SUM(CASE WHEN days_back < 28 THEN 1 ELSE 0 END) AS l28_orders,
        SUM(CASE WHEN days_back < 90 THEN 1 ELSE 0 END) AS l90_orders
    FROM o_e GROUP BY customer_id
),
return_w AS (
    SELECT customer_id, COUNT(*) AS lifetime_returns,
        SUM(CASE WHEN days_back < 7  THEN 1 ELSE 0 END) AS l7_returns,
        SUM(CASE WHEN days_back < 28 THEN 1 ELSE 0 END) AS l28_returns,
        SUM(CASE WHEN days_back < 90 THEN 1 ELSE 0 END) AS l90_returns
    FROM r_e GROUP BY customer_id
),
joined AS (
    SELECT o.customer_id, o.lifetime_orders,
        COALESCE(r.lifetime_returns,0) AS lifetime_returns,
        o.l7_orders,  COALESCE(r.l7_returns,0)  AS l7_returns,
        o.l28_orders, COALESCE(r.l28_returns,0) AS l28_returns,
        o.l90_orders, COALESCE(r.l90_returns,0) AS l90_returns,
        COALESCE(b.max_7d_return_burst,0) AS max_7d_return_burst
    FROM order_w o
    LEFT JOIN return_w r ON r.customer_id = o.customer_id
    LEFT JOIN burst    b ON b.customer_id = o.customer_id
)
SELECT customer_id, lifetime_orders, lifetime_returns,
    ROUND(1.0*lifetime_returns/NULLIF(lifetime_orders,0), 2) AS lifetime_return_rate,
    l7_orders,  l7_returns,  ROUND(1.0*l7_returns /NULLIF(l7_orders ,0), 2) AS l7_return_rate,
    l28_orders, l28_returns, ROUND(1.0*l28_returns/NULLIF(l28_orders,0), 2) AS l28_return_rate,
    l90_orders, l90_returns, ROUND(1.0*l90_returns/NULLIF(l90_orders,0), 2) AS l90_return_rate,
    max_7d_return_burst,
    CASE
        WHEN l90_orders < 5 THEN 'low_volume'
        WHEN max_7d_return_burst > 4 THEN 'burst'
        WHEN 1.0*l28_returns/NULLIF(l28_orders,0) > 0.40 THEN 'high_return_rate'
        WHEN 1.0*l28_returns/NULLIF(l28_orders,0)
             > 1.5*COALESCE(1.0*l90_returns/NULLIF(l90_orders,0),0)
             AND l28_returns >= 2 THEN 'rising_returns'
        ELSE 'clean'
    END AS fraud_flag
FROM joined
ORDER BY
    CASE WHEN max_7d_return_burst > 4 THEN 1
         WHEN 1.0*l28_returns/NULLIF(l28_orders,0) > 0.40 THEN 2
         WHEN 1.0*l28_returns/NULLIF(l28_orders,0)
              > 1.5*COALESCE(1.0*l90_returns/NULLIF(l90_orders,0),0)
              AND l28_returns >= 2 THEN 3
         WHEN l90_orders < 5 THEN 4
         ELSE 5 END,
    l28_return_rate DESC;

4. Strategic explanations

The self-join burst detector and the half-open window

max_7d_return_burst asks: "anchored on each return, how many returns fall in the 7 days starting there?" The self-join pairs every return a with every return b of the same customer where b.return_date >= a.return_date and the gap is < 7 days. Grouping by the anchor counts the cluster size; MAX over anchors finds the densest window. The window is one-directional (forward only) so each distinct 7-day span is evaluated exactly once — a symmetric BETWEEN -7 AND 7 would double-count and inflate the burst.

NULLIF is the only thing standing between you and a divide-by-zero

Every rate is returns / NULLIF(orders, 0). A customer with zero L7 orders would otherwise crash the query in Postgres (it raises division by zero, it does not return NULL). NULLIF(orders,0) turns a zero denominator into NULL, and x / NULL = NULL — the rate is reported as "undefined" rather than fabricated. Customer 4's L7 rate is NULL for exactly this reason: no L7 orders means no L7 rate exists.

Orders drive the result; LEFT JOIN keeps zero-return customers

The final joined CTE starts from order_w and LEFT JOINs the return aggregates. This is deliberate: a customer with zero returns has no row in return_w at all. An inner join would drop customer 3 entirely — yet customer 3 must still be flagged low_volume. The COALESCE(..., 0) on every return column converts the missing side to an explicit zero so the rate math and the flag logic both behave.

The CASE priority ladder is the business spec

The flag is a strict hierarchy: low_volume → burst → high_return_rate → rising_returns → clean. low_volume sits first because with fewer than 5 orders any rate is statistically meaningless — you suppress noisy signals before evaluating severity. The exact same ladder is repeated verbatim in the ORDER BY so the output is sorted by the same priority it was labelled with. A senior keeps that single source of truth in mind: if the spec reorders, both copies must move together — a strong argument for factoring the ladder into one expression or a lookup table.

The l28_returns >= 2 floor on rising_returns

rising_returns additionally requires at least 2 L28 returns. Without it, a customer with 1 L28 return out of 2 orders (rate 0.50) and 0 L90 returns would trip "rate > 1.5× the L90 rate" on a single event — statistical noise dressed up as a trend. The floor is a minimum-sample guard, the same defensive instinct as the low_volume flag, applied one rule deeper.

Senior signal: build the burst detector with a forward-only self-join so each 7-day span is counted once, wrap every rate denominator in NULLIF to dodge Postgres divide-by-zero, drive the LEFT JOIN from orders with COALESCEd returns so zero-return customers survive, and recognise the CASE/ORDER BY priority ladder as a duplicated business spec that should be a single source of truth.
