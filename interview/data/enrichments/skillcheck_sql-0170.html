Concurrent-user counting appears in dashboards for streaming services, call centres, and SaaS platforms: "how many sessions were live at this exact moment?" The interviewer is probing the point-in-time interval probe pattern — joining a table of time windows against a spine of probe timestamps using a half-open containment predicate — and whether you use a LEFT JOIN … COUNT(s.id) idiom to correctly return zero for probe times with no active sessions.

Key edge cases: probe 09:00 matches session 1 because the predicate is login_ts ≤ probe (inclusive start); probe 10:30 matches session 3 because its login_ts is exactly 10:30 and the predicate is non-strict on the left side. Probe 13:00 matches nobody — that is the zero-count case the LEFT JOIN must preserve.

2. The SQL — left join probe spine against sessions

SELECT p.ts, COUNT(s.id) AS active_users
FROM probe_ts p
LEFT JOIN session s
  ON s.login_ts <= p.ts AND s.logout_ts > p.ts
GROUP BY p.ts
ORDER BY p.ts;

The driving table is the probe spine p. Every probe row is preserved by LEFT JOIN; for each probe the join condition login_ts ≤ p.ts AND logout_ts > p.ts matches every session whose window contains the instant. COUNT(s.id) — counting a column from the right side, not COUNT(*) — counts only matched rows and returns 0 when all right-side columns are NULL (i.e. no active session).

4. Strategic explanations

The half-open containment predicate: login_ts ≤ p.ts AND logout_ts > p.ts

Sessions are modelled as half-open intervals [login, logout): the login instant belongs to the session, the logout instant does not. The point-in-time test is the standard half-open membership test: a probe instant p is inside the interval when login ≤ p < logout, which SQL spells as login_ts ≤ p.ts AND logout_ts > p.ts. Getting the operator choice wrong — for example using logout_ts ≥ p.ts — would double-count the boundary instant where one session ends and another begins, inflating the concurrent count by 1.

Why COUNT(s.id) rather than COUNT(*)

COUNT(*) counts rows, including rows where all right-side columns are NULL — the NULL placeholder row that a LEFT JOIN inserts when no match exists. That would return 1 instead of 0 for the dead-zone probe. COUNT(s.id) ignores NULL values, so it returns 0 exactly when the left join finds no matching session. This is the canonical idiom for "count matching rows, return zero when none match" in a LEFT JOIN … GROUP BY pattern.

Driving from the probe spine, not the sessions table

The query puts probe_ts on the left so every probe timestamp is guaranteed a row in the output regardless of matches. Reversing the join — driving from session — would drop probe timestamps with zero sessions entirely, silently omitting the trough periods that are often the most operationally important (no active users could mean a system outage). Always drive from the spine when you need complete coverage of an axis.

Performance — the non-equi join problem

The join condition has no equality clause on a column that can use a standard B-tree index lookup; the planner must scan or loop. For small probe sets the query is fast. At scale (millions of sessions, thousands of probe points) consider: (1) an index on session(login_ts, logout_ts) to let the engine prune by start time; (2) the event-timeline approach — union +1 events at login and −1 events at logout into a timeline, use SUM window functions to derive concurrent counts at every change point, then join probe timestamps to that derived timeline; or (3) in PostgreSQL, a range-typed column with a GiST index and the <@ containment operator.

Dialect notes

The LEFT JOIN … COUNT(col) pattern is fully portable: PostgreSQL, MySQL, SQL Server, BigQuery, Snowflake, and SQLite all support it. The ::numeric cast and FILTER (WHERE …) clause used elsewhere in the bank are PostgreSQL-specific — neither appears here, so this query runs unchanged everywhere. BigQuery uses DATETIME instead of TIMESTAMP for timezone-naive instants; the logic is identical.

Senior signal: drive from the probe spine with LEFT JOIN to guarantee zero-count rows; use COUNT(s.id) not COUNT(*) to correctly return 0 when no sessions match; apply the half-open predicate login_ts ≤ p.ts AND logout_ts > p.ts with careful operator choice; and flag the event-timeline window-function approach as the scalable alternative for large datasets.
