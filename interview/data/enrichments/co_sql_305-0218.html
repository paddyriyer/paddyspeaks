The LeetCode classic. The exact prompt: "Find the second-highest salary." Sounds trivial. It isn't. The reason it's been asked in literally tens of thousands of interviews is that the three obvious answers all behave differently on edge cases — and the interviewer's real test is whether you surface those edge cases before writing SQL.

1. The source data — employees

emp_idnamedept_idsalary
E01AlexENG$320,000
E02BaileyENG$320,000
E03CaseySALES$280,000
E04DrewSALES$240,000
E05EllisMKTG$200,000

Alex and Bailey are tied at the top ($320K). Casey is alone in second place at $280K. Ellis is the only person in Marketing — which becomes important the moment someone asks for "second highest per department".

2. The SQL — the three canonical answers

-- A. Correlated subquery (the LeetCode-blessed one)
SELECT MAX(salary) AS second_highest
FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);

-- B. DENSE_RANK — handles ties at the top correctly
SELECT salary AS second_highest
FROM (SELECT DISTINCT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rk
      FROM employees) t
WHERE rk = 2;

-- C. LIMIT / OFFSET with DISTINCT
SELECT DISTINCT salary AS second_highest
FROM employees
ORDER BY salary DESC
LIMIT 1 OFFSET 1;

3. The resulting analysis

ApproachResult on the data aboveResult if NO second salary exists
Correlated subquery (A)$280,000NULL (one row, NULL value)
DENSE_RANK (B)$280,000Zero rows
LIMIT / OFFSET (C)$280,000Zero rows

All three return the right answer on the happy path. They diverge on the "what if there's no second salary?" edge case — and that's exactly what the interviewer will ask next.

4. Strategic explanations

The user's prompt has two unstated assumptions — surface them before writing SQL

"Find the second-highest salary" hides:

  1. Tie semantics: if Alex and Bailey are both at $320K, is the "second highest" $320K (one of them) or $280K (the next distinct value)? Almost always the latter — but ask.
  2. Empty-result semantics: if everybody earns the same salary, or only one row exists, what should the query return? NULL? Zero rows? An error? Each approach above behaves differently.
The "only one salary in the department" probe — the user's classic follow-up

Generalising to "second-highest salary in Engineering" with the user-supplied CTE pattern:

WITH emp AS (
  SELECT name, salary, d.department_name,
    DENSE_RANK() OVER (PARTITION BY e.dept_id ORDER BY salary DESC) AS rnk
  FROM employees e JOIN departments d ON e.dept_id = d.dept_id
)
SELECT * FROM emp WHERE rnk = 2;

What does this return for Marketing, which has only Ellis at $200K?

Nothing. Ellis gets rnk = 1 and there's no rnk = 2 — so Marketing silently disappears from the report. The hiring manager opens the dashboard and sees no entry for Marketing without realising it's a "headcount = 1" problem, not a data-quality issue.

The defensive rewrite — keep every department on the report, surface "N/A" explicitly:

SELECT d.department_name,
  COUNT(*) AS headcount,
  MAX(CASE WHEN rnk = 1 THEN salary END) AS top_salary,
  MAX(CASE WHEN rnk = 2 THEN salary END) AS second_salary  -- NULL if headcount < 2
FROM (
  SELECT e.salary, e.dept_id,
    DENSE_RANK() OVER (PARTITION BY e.dept_id ORDER BY e.salary DESC) AS rnk
  FROM employees e
) r
JOIN departments d ON d.dept_id = r.dept_id
GROUP BY d.department_name;

Now Marketing appears with headcount=1, top=200K, second=NULL — the consumer of the report can spot the issue immediately.

The "do you want all employees tied for second?" probe

Imagine the data was {$320K, $320K, $280K, $280K, $240K}. "Second-highest" by DENSE_RANK is $280K — but two people are tied at that salary. Which ones come back?

If the query is…It returns…When to pick
WHERE rnk = 2 from DENSE_RANK over employeesBoth tied employeesPay-equity audits, "everyone in tier 2"
WHERE rnk = 2 from DENSE_RANK over DISTINCT salaryOne row — just the salary, not the employees"What's the salary at tier 2?" (no people)

WHERE rnk = 2 from RANKNothing if the top is tied (ranks 1, 1, 3 — skips 2)Never — almost always a bug

With RANK, WHERE rk = 2 returns NOTHING because rank 2 was skipped. With DENSE_RANK, WHERE rk = 2 returns Casey. For "Nth-highest" semantics, DENSE_RANK is almost always the right pick.

What if the dept genuinely has duplicates that should both count?

Suppose ENG has {$320K, $320K, $280K}. Should the "second-highest in ENG" be $320K (the other person at the top) or $280K (the next distinct salary)? Both interpretations are defensible:

  • Ordinal interpretation: "the 2nd row when I sort by salary" → ROW_NUMBER = 2 → $320K (whichever person the engine picks).
  • Distinct-value interpretation: "the 2nd-highest distinct salary band" → DENSE_RANK = 2 over DISTINCT salaries → $280K.

The interviewer almost always means the distinct-value version (because the prompt is about compensation analysis, not row-ordering). Confirm explicitly.

Performance — when each approach matters
ApproachComplexityIndex hit?
Correlated subqueryO(n²) without an index on salary, O(n) with oneYes — MAX(salary) uses an index range scan
DENSE_RANK windowO(n log n) — full sort over the partitionPartial — sort can spill to disk on large tables

For 10M-row tables in OLAP warehouses, all three are within ~2× of each other. Pick the approach that communicates intent most clearly. DENSE_RANK wins on readability for the Nth-highest family.

The empty-result wrapper — when the consumer demands a single NULL row

Some downstream consumers (BI tools that expect "one number") break when the query returns zero rows. Wrap the window-function approach to guarantee one row:

SELECT (SELECT salary FROM (
  SELECT DISTINCT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rk
  FROM employees
) t WHERE rk = 2) AS second_highest;
-- Returns NULL if no second salary exists; otherwise the salary.
Senior signal: before writing SQL, surface (1) tie semantics, (2) what to return when there's no second salary, and (3) "all ties or one representative?". Then write the DENSE_RANK answer, contrast it with RANK to show you understand the skip-on-tie trap, and pre-empt the per-department generalisation with the PARTITION BY dept_id + LEFT JOIN defensive pattern.
