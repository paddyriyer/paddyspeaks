Walmart's PMO wants to spot resourcing collisions: pairs of projects whose date ranges overlap, because two concurrent projects compete for the same engineers and budget. There is no overlap column — you have to derive it from start_date and end_date with an interval-intersection test. The interview is whether you get the overlap condition exactly right and avoid reporting each pair twice.

Project 5 runs in autumn and overlaps nobody — the isolated case. Projects 1–4 form a chain of partial overlaps. Project 1 and project 3 do not overlap (1 ends 03-31, 3 starts 04-01), which tests whether the condition is precise rather than loose.

2. The SQL — self-join on interval intersection

SELECT a.project_id, b.project_id
FROM projects a JOIN projects b
  ON a.project_id < b.project_id
 AND a.start_date < b.end_date AND b.start_date < a.end_date;

The table is joined to itself. a.project_id < b.project_id keeps each unordered pair once and rules out a project pairing with itself. The two date inequalities are the overlap test.

4. Strategic explanations

The overlap condition — why this form

Two intervals overlap when each starts before the other ends. Formally: a.start < b.end AND b.start < a.end. It is far easier to define overlap by its negation — two ranges do not overlap only if one finishes entirely before the other begins (a.end < b.start OR b.end < a.start). Negate that with De Morgan and you land exactly on the solution's two inequalities. Many candidates try to enumerate the overlap cases directly and miss one; deriving it from the non-overlap is the reliable method.

The duplicate-pair trap — why a.project_id < b.project_id

A self-join with a.project_id <> b.project_id reports every overlap twice — once as (2,3) and once as (3,2) — and also lets a row pair with itself if you use <=. The strict < does double duty: it removes self-pairs and keeps only one ordering of each pair. Switching <> for < is the single most common bug in this question.

Closed vs half-open intervals — the boundary decision

With strict <, two projects that merely touch — one ends on the same day the next starts — are not counted as overlapping. That is why (1,3) is excluded. If a same-day handover should count as a collision, the test becomes <=:

-- inclusive: touching endpoints count as overlap
AND a.start_date <= b.end_date AND b.start_date <= a.end_date

With <=, the pair (1,3) would appear. Neither is universally right — it depends on whether end_date is the last active day (closed) or an exclusive bound. State the assumption.

NULL dates — open-ended projects

An ongoing project often has a NULL end_date. Any comparison with NULL is unknown, so a.start_date < b.end_date fails and the open-ended project silently overlaps nobody — the opposite of reality, since a project with no end overlaps everything after its start. Treat a NULL end as "infinitely far in the future":

AND a.start_date < COALESCE(b.end_date, DATE '9999-12-31')
AND b.start_date < COALESCE(a.end_date, DATE '9999-12-31')
Scale — the self-join is quadratic

A self-join compares roughly n²/2 pairs. Fine for five projects, but on tens of thousands it is slow. An index on (start_date, end_date) helps the planner prune; for very large interval datasets a range type with a GiST index (daterange and the && overlap operator) turns the scan into an index probe. Mentioning that the elegant self-join has a scaling ceiling is the senior note.

Senior signal: derive the overlap test by negating the simpler non-overlap condition, use strict < on the ids to emit each pair once and exclude self-pairs, make the closed-vs-half-open boundary choice explicit, and handle NULL end dates as open-ended rather than letting them match nothing.
