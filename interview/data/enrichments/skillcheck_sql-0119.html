A sales-ops team wants to find dormant sellers: which sellers had no orders during calendar year 2025? The subtlety the interviewer plants is that a seller with orders in 2024 or 2026 still counts as "no 2025 sales" — so a plain WHERE sale_date NOT BETWEEN … on a joined row is wrong. This is a textbook anti-join, and the trick is putting the date filter in the right place.

1. The source data — seller and orders

seller_idseller_nameorders (sale_date)(narrative)
1Acme2025-03-01has a 2025 order → not dormant
2BetaCorp2024-12-31order exists, but in 2024 → still dormant for 2025
3CelStar2025-06-15, 2026-01-01one 2025 order → not dormant
4Delta(no orders at all)never ordered → dormant

Two edge cases carry the interview. BetaCorp's only order is 2024-12-31 — one day outside the window — and Delta has no rows in orders at all. A correct query must flag both, while letting CelStar through even though one of its two orders (2026-01-01) is also outside 2025.

2. The SQL — LEFT JOIN with the date filter in the ON clause

SELECT s.seller_name
FROM seller s
LEFT JOIN orders o
  ON o.seller_id = s.seller_id
 AND o.sale_date >= DATE '2025-01-01'
 AND o.sale_date <  DATE '2026-01-01'
WHERE o.seller_id IS NULL
ORDER BY s.seller_name;

The two date predicates sit in ON, not WHERE. The join therefore only matches a seller to their 2025 orders; any seller with zero 2025 orders gets all-NULL columns from orders, and WHERE o.seller_id IS NULL keeps exactly those.

3. The resulting analysis

Acme and CelStar are correctly absent: each has at least one order inside 2025. CelStar's 2026 order is irrelevant — its 2025-06-15 order alone makes the join match, so it is not dormant.

4. Strategic explanations

Why the date filter must live in ON, not WHERE

This is the crux. With a LEFT JOIN, predicates in ON decide which rows match; predicates in WHERE filter the result after the join, NULLs included. Move the date test to WHERE — WHERE o.sale_date >= '2025-01-01' AND … — and the NULL sale_date of an unmatched seller fails the comparison (NULL is not >= anything), so Delta is silently dropped. The query would then return only sellers with non-2025 orders and miss the never-ordered ones. Filter in ON to keep the outer side intact.

The anti-join pattern: LEFT JOIN … IS NULL

"Rows in A with no match in B" is an anti-join. The LEFT JOIN attaches matches where they exist and NULLs where they do not; WHERE <join key> IS NULL then isolates the no-match rows. Test the join key (o.seller_id) for NULL, not an arbitrary nullable column — the key is NULL precisely when no row matched. Equivalent formulations are NOT EXISTS and NOT IN; the planner often treats LEFT JOIN / IS NULL and NOT EXISTS identically (a hash anti-join).

Half-open date ranges beat BETWEEN

The window is >= '2025-01-01' AND < '2026-01-01' — half-open, lower bound included, upper bound excluded. BETWEEN '2025-01-01' AND '2025-12-31' looks equivalent but breaks if sale_date is ever a timestamp: 2025-12-31 14:00 is after 2025-12-31 00:00:00 and gets wrongly excluded. The < next-year-start form is correct for both date and timestamp columns and is the habit to show.

Why NOT IN is the risky alternative

You could write WHERE s.seller_id NOT IN (SELECT seller_id FROM orders WHERE sale_date >= …). It works here, but NOT IN has a notorious NULL trap: if the subquery ever yields a NULL seller_id, the whole NOT IN evaluates to NULL/false and returns zero rows. NOT EXISTS and the LEFT JOIN / IS NULL anti-join are NULL-safe by construction. Preferring them, and being able to explain the NOT IN footgun, is the senior signal.

Senior signal: recognise "no orders in a window" as an anti-join, put the date predicate in ON (not WHERE) so never-ordered sellers survive the LEFT JOIN, test the join key for IS NULL, use a half-open date range that is safe for timestamps, and know why NOT IN with a nullable subquery silently returns nothing.
