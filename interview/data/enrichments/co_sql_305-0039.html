Nintendo's regional sales leads each run a territory, and the quarterly review opens with one slide: a ranked board of territories by revenue. The query is a textbook GROUP BY plus a ranking window — but the interesting moment is what happens when two territories finish the quarter at exactly the same total. An interviewer asking this wants to see that you know how RANK() behaves on a tie, and that you can rank an aggregate, not just a raw column.

1. The source data — sales

West2300

East4400
North5200

Each territory has multiple reps, so the rows must be collapsed before ranking. The set-up is deliberate: West sums to 800 and East also sums to 800 — a tie at the top. That tie decides whether North is ranked 2 or 3.

2. The SQL — RANK() OVER (ORDER BY SUM(...))

SELECT territory, SUM(amount) AS total_sales,
  RANK() OVER (ORDER BY SUM(amount) DESC) AS rk
FROM sales GROUP BY territory;

The GROUP BY collapses reps into one row per territory; SUM(amount) is the per-territory total. The window function then ranks those grouped rows — note that SUM(amount) inside OVER(...) is legal and refers to the same aggregate, evaluated after grouping.

3. The resulting analysis

territorytotal_salesrk
East8001
West8001
North2003
South1994

East and West both land at rank 1 — RANK() gives tied rows the same rank. Then North is rank 3, not 2: the rank after a tie skips by the size of the tie group. That gap is the headline behaviour of RANK(), and it is correct unless the business wants a gapless board.

4. Strategic explanations

The ordering of tied rows is not deterministic

East and West both show rk = 1, but which one is printed first is up to the planner — ORDER BY SUM(amount) DESC does not distinguish them. If the slide needs a stable order, add a tiebreaker to the window and to a final ORDER BY:

SELECT territory, SUM(amount) AS total_sales,
  RANK() OVER (ORDER BY SUM(amount) DESC) AS rk
FROM sales GROUP BY territory
ORDER BY rk, territory;
RANK vs DENSE_RANK vs ROW_NUMBER

The interviewer's natural follow-up: "what if you don't want the gap?" Three window functions, three behaviours on the East/West tie:

territorytotalRANKDENSE_RANKROW_NUMBER
East800111
West800112
North200323
South199434

DENSE_RANK closes the gap (1, 1, 2). ROW_NUMBER ignores the tie entirely and forces a 1/2 split — wrong here, because it would arbitrarily declare one of two equal territories the winner.

Why not a correlated subquery?

The pre-window way to rank is to count how many territories outscored each one. It works but is O(n²) and far harder to read:

SELECT territory, total_sales,
  (SELECT COUNT(*)+1 FROM (SELECT SUM(amount) s FROM sales GROUP BY territory) o
   WHERE o.s > t.total_sales) AS rk
FROM (SELECT territory, SUM(amount) AS total_sales FROM sales GROUP BY territory) t;

On a real warehouse the window function sorts once and is the only sensible choice.

Edge case: NULL amounts and empty territories

SUM skips NULL amount values, and a territory whose rows are all NULL sums to NULL — which sorts to the end under DESC by default. A territory with no rows at all simply never appears, since GROUP BY only emits groups that exist. If the board must list every known territory including zero-sellers, you need a territories dimension table and a LEFT JOIN.

Senior signal: rank the aggregate directly with RANK() OVER (ORDER BY SUM(amount) DESC), explain that ties share a rank and the next rank skips, contrast RANK/DENSE_RANK/ROW_NUMBER for the gapless follow-up, and add an explicit tiebreaker so the printed order of tied territories is stable.
