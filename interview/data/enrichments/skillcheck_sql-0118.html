An HR analytics team stores a turnstile feed: every clock-in/clock-out pair is one row, holding minutes-since-midnight. Someone steps out for lunch and badges back in — that is a second row for the same person on the same day. The interviewer asks for total minutes worked per employee per day, and is really checking whether you understand that the grain of the table is the session, not the day, so the answer is a straightforward GROUP BY over a per-row arithmetic expression.

1. The source data — employees

emp_idevent_dayin_timeout_time(narrative)
12025-07-0160120morning session — 60 min

The edge case that decides the interview: employee 1 has two rows on 2025-07-01. A naive query that returns one row per session, or that picks MAX(out_time) - MIN(in_time), gets the wrong number. The grain is a session; the answer must roll sessions up to the day.

2. The SQL — sum per-row durations, grouped by day and employee

SELECT event_day, emp_id, SUM(out_time - in_time)::int AS total_time
FROM employees
GROUP BY event_day, emp_id
ORDER BY event_day, emp_id;

out_time - in_time is evaluated per row — the length of one session. SUM then adds those session lengths within each (event_day, emp_id) group, so employee 1's two Tuesday sessions (60 + 60) collapse into one total.

3. The resulting analysis

event_dayemp_idtotal_timesignal
2025-07-011120two sessions summed: 60 + 60
2025-07-012240single session, 300 − 60
2025-07-02160same employee, separate day → own group

Employee 1 on 2025-07-01 shows 120, not 60: both sessions count. The same employee on 2025-07-02 is a distinct group because event_day is part of the GROUP BY key — the day is not folded away.

4. Strategic explanations

Why the expression goes inside the aggregate, not outside

SUM(out_time - in_time) computes each row's duration first, then totals them — correct. The tempting wrong form is SUM(out_time) - SUM(in_time), which happens to give the same answer here only because every row is a clean pair. The principled reason it is still fine: subtraction distributes over summation. But MAX(out_time) - MIN(in_time) does not — that measures span-of-day including the lunch gap (240 − 60 = 180 for employee 1), not minutes worked. State that you are summing session lengths, deliberately, so the lunch break is excluded.

The grouping key must match the requested grain

The question asks for one row per (day, employee), so both columns belong in GROUP BY. Drop event_day and you would sum across all of an employee's days into one number; drop emp_id and you would merge employees. Every non-aggregated column in the SELECT must appear in GROUP BY — Postgres enforces this; MySQL historically did not and would silently return an arbitrary value.

NULL clock-outs — the trap this data hides

Here every session is closed, but real turnstile data has open sessions: someone clocked in and never clocked out, leaving out_time NULL. out_time - in_time is then NULL, and SUM silently skips it — the day's total quietly understates. Decide explicitly: exclude open sessions with WHERE out_time IS NOT NULL, or treat an open session as running to end-of-shift with COALESCE(out_time, 1440). Mention midnight-spanning shifts too — if out_time < in_time the shift crossed midnight and needs + 1440.

The ::int cast — cosmetic but intentional

SUM over integer columns returns bigint in Postgres (to avoid overflow on huge tables). The ::int downcast just keeps the output type tidy as integer; it is safe here because daily minutes never approach the int ceiling. On a genuinely large sum you would leave it as bigint. Knowing why the cast is there — not blindly copying it — is the senior tell.

Senior signal: recognise the table's grain is the clock-in/out session, sum out_time - in_time per row inside the aggregate (never MAX - MIN, which leaks the lunch gap), group by exactly the requested (day, employee) key, and call out unprompted how NULL clock-outs and midnight-spanning shifts would corrupt the total.
