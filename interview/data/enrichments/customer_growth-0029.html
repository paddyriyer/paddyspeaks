A merchandising team wants each customer's "go-to" products — their top 3 by total spend — to drive personalised recommendations. The interviewer is probing two things: the standard "rank then filter" pattern (a window function cannot be filtered in WHERE, so it must be wrapped in a CTE), and whether you know the modern QUALIFY shortcut and which engines support it. The ranking-function choice — DENSE_RANK here — quietly decides how ties at the boundary behave.

The edge case is customer 1's products 103 ($45) and 105 ($40): close in spend but distinct, so 103 takes rank 3 and 105 lands at rank 4 and is excluded. Product 101 also appears in two separate order rows — the per-product spend must aggregate before ranking, or the same product would be ranked twice.

2. The SQL — top-3 per customer via DENSE_RANK + WHERE

WITH per_product AS (
    SELECT customer_id, product_id,
           ROUND(SUM(qty * unit_price), 2) AS total_spend
    FROM order_items
    GROUP BY customer_id, product_id
),
ranked AS (
    SELECT customer_id, product_id, total_spend,
           DENSE_RANK() OVER (PARTITION BY customer_id
                              ORDER BY total_spend DESC) AS rk
    FROM per_product
)
SELECT customer_id, product_id, total_spend, rk
FROM ranked
WHERE rk <= 3
ORDER BY customer_id ASC, rk ASC;

The QUALIFY equivalent collapses the ranked CTE entirely — on engines that support it (BigQuery, Snowflake, DuckDB, Databricks):

SELECT customer_id, product_id, total_spend,
       DENSE_RANK() OVER (PARTITION BY customer_id
                          ORDER BY total_spend DESC) AS rk
FROM per_product
QUALIFY rk <= 3
ORDER BY customer_id, rk;

4. Strategic explanations

Why the window function must be wrapped in a CTE

Window functions are evaluated after the WHERE clause — that is fixed by SQL's logical processing order (FROM → WHERE → GROUP BY → SELECT/window → ORDER BY). So WHERE rk <= 3 in the same query that computes rk fails: rk does not exist yet when WHERE runs. The fix is to compute the rank in an inner CTE (or subquery), then filter in the outer query where rk is now an ordinary column. This rank-then-filter pattern is one of the most common window-function idioms.

QUALIFY — the purpose-built shortcut, and its reach

QUALIFY is to window functions what HAVING is to aggregates: it filters on a window result without a wrapping layer. QUALIFY rk <= 3 reads cleaner and lets the optimiser see the intent directly. But it is not standard SQL — it exists in BigQuery, Snowflake, DuckDB, Databricks, and Teradata, but not in PostgreSQL, MySQL, SQLite, SQL Server, or Oracle. On those, the CTE-plus-WHERE form is the portable answer. A senior names both and states the support boundary.

DENSE_RANK vs RANK vs ROW_NUMBER — the tie behaviour decides the result

All three would return 1, 2, 3 on this tie-free data, but they diverge the moment two products share a spend. Suppose two products tie for 2nd at $80: DENSE_RANK gives 1, 2, 2, 3 — so rk <= 3 returns four products. RANK gives 1, 2, 2, 4 — rk <= 3 returns three, and the 4th product is excluded by the skipped rank. ROW_NUMBER gives 1, 2, 3, 4 — exactly three, but it breaks the tie arbitrarily by physical row order, so which tied product makes the cut is non-deterministic. The right pick depends on the product question: "top 3 distinct spend levels" wants DENSE_RANK; "at most 3 rows, ties broken deterministically" wants ROW_NUMBER with an explicit tie-breaker like ORDER BY total_spend DESC, product_id.

Aggregate before you rank

Product 101 appears in two order rows. Ranking order_items directly would treat each row as a separate contender and rank product 101 twice. The per_product CTE collapses to one row per (customer_id, product_id) with SUM(qty * unit_price) first — grain correctness before ranking. A common mistake is to skip this step and rank raw line items, producing duplicate products in the "top 3."

What a senior says next

"I would confirm the tie policy with the product owner before shipping. If two products genuinely tie for the 3rd slot, do we show both (a 4-row result with DENSE_RANK), or do we force exactly 3 and break the tie by recency or margin? That is a business decision, and the ranking-function choice is just how we encode the answer."

Senior signal: explain that window functions run after WHERE so rank-then-filter needs a CTE, present QUALIFY as the shortcut while stating it is absent from Postgres/MySQL/SQLite, walk through how DENSE_RANK/RANK/ROW_NUMBER each return a different row count at a boundary tie, and aggregate to per-product grain before ranking.
