Reddit's commerce team wants a single number for the weekly deck: the average basket size — how many items an order contains on average. The catch is that the data lives in order_items, one row per line item, not one row per order. So "average" has a hidden two-step shape: first roll line items up to orders, then average across orders. Getting the grain right is the entire question.

1. The source data — order_items

Seven line-item rows, but only four orders. Order 1 has two lines totalling 5 units; order 3 has three lines totalling 3 units. The mismatch between row count and order count is exactly the trap.

2. The SQL — AVG of per-order SUM

SELECT AVG(items_per_order) AS avg_basket_size
FROM (SELECT order_id, SUM(qty) AS items_per_order
      FROM order_items GROUP BY order_id) t;

The inner query collapses line items to one row per order, summing qty. The outer query averages those four per-order totals. Two aggregations at two different grains.

3. The resulting analysis

avg_basket_size
4.2500000000000000

Per-order totals are 5, 5, 3, 4 — they sum to 17, and 17 ÷ 4 = 4.25. The long decimal tail is just Postgres's exact numeric arithmetic; a dashboard would round it to 4.3.

4. Strategic explanations

Why the subquery is not optional

The instinct is to skip the nesting and write AVG(qty) directly. That answers a different question: the average line-item quantity (17 ÷ 7 = 2.43), not the average basket. The two-step structure exists because the metric is defined at the order grain while the data is stored at the line-item grain. You cannot average across orders until you have one number per order.

Items vs lines — clarify the definition

"Basket size" is ambiguous. It could mean total units (SUM(qty), what the solution uses) or the number of distinct lines — how many products were in the cart. They differ whenever an order has qty > 1:

-- average number of distinct lines per order
SELECT AVG(line_count) AS avg_lines
FROM (SELECT order_id, COUNT(*) AS line_count
      FROM order_items GROUP BY order_id) t;
-- line counts 2,1,3,1 -> 7/4 = 1.75

4.25 units vs 1.75 lines — same data, very different stories. Ask which the product team means before writing SQL.

A flatter rewrite with the same answer

The nested form is the clearest, but the average of per-group sums can also be expressed without a subquery, because the average equals total units divided by distinct orders:

SELECT SUM(qty)::numeric / COUNT(DISTINCT order_id) AS avg_basket_size
FROM order_items;
-- 17 / 4 = 4.25

This is handy to know, but the subquery version is more readable and generalises better (e.g. if you later need a median or a percentile per order). Reach for the explicit two-step unless performance demands otherwise.

Edge cases: zero-qty lines and empty orders

If a line can have qty = 0 (a removed item left in the table), it inflates the line count but not the unit total — another reason units and lines diverge. And an order present in an orders table but with no rows in order_items simply will not appear in the inner GROUP BY, so it is excluded from the average entirely. If empty baskets should count as size 0, you need a LEFT JOIN from orders and a COALESCE(SUM(qty), 0).

Senior signal: recognise the line-item-to-order grain mismatch and explain why the subquery is structurally required, distinguish "average units" from "average distinct lines" and confirm which the metric means, and flag that orders with no line items quietly drop out of the average rather than counting as zero.
