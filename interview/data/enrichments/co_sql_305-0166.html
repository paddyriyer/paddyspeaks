Capital One's procurement team scores suppliers on delivery reliability: what percentage of shipments arrived on or before the scheduled date, then ranks suppliers by that score for a vendor scorecard. The query combines a conditional aggregate with a window RANK() — and the interview tests whether you can express "on-time rate" cleanly and know how the ranking treats ties.

Supplier 1 has one early, one exactly on time (12th = 12th — tests ≤), and one late: 2 of 3 on time. Supplier 2 is two-for-two on time. Supplier 3 is one late, one early: 1 of 2. Supplier 2 will top the ranking.

2. The SQL — conditional aggregate + RANK()

SELECT supplier_id,
  100.0 * SUM(CASE WHEN actual_date <= scheduled_date THEN 1 ELSE 0 END)/COUNT(*) AS on_time_pct,
  RANK() OVER (ORDER BY 100.0 * SUM(CASE WHEN actual_date <= scheduled_date THEN 1 ELSE 0 END)/COUNT(*) DESC) AS rk
FROM shipments GROUP BY supplier_id;

The CASE emits 1 for an on-time shipment and 0 otherwise; SUM of that is the on-time count; dividing by COUNT(*) gives the rate. The RANK() window orders suppliers by that same expression, highest rate first.

4. Strategic explanations

The conditional-aggregate idiom — SUM(CASE WHEN … THEN 1 ELSE 0 END)

This is the canonical way to count rows meeting a condition within a group. The CASE turns each shipment into 1 (on time) or 0 (late); SUM tallies the 1s. Dividing by COUNT(*) — the total shipments — yields the rate. The cleaner Postgres equivalent is the FILTER clause, which many interviewers like to see:

SELECT supplier_id,
  100.0 * COUNT(*) FILTER (WHERE actual_date <= scheduled_date) / COUNT(*) AS on_time_pct
FROM shipments GROUP BY supplier_id;

COUNT(*) FILTER (WHERE …) says exactly what it does. SUM(CASE…) is the portable form that works on every engine; FILTER is standard SQL but newer. Knowing both, and that they are equivalent, is a senior tell.

The boundary — is "on the scheduled date" on time?

Supplier 1's shipment on the 12th, scheduled for the 12th, counts as on-time because the test is actual_date <= scheduled_date. Arriving exactly on the due date is almost always considered on time — ≤ is right. With a strict <, that shipment would be "late" and supplier 1 would drop to 33%. Confirm the definition; the operator silently encodes the SLA.

The integer-division trap — 100.0 is load-bearing

SUM(CASE…) and COUNT(*) are both integers. Postgres integer / integer truncates: 2/3 is 0, not 0.667. Multiplying by the numeric literal 100.0 before the division promotes the expression to numeric so the result is a real percentage. Write 100 (integer) instead and supplier 1's 66.67% collapses to 0. The .0 is not cosmetic.

The dialect note — this would be cleaner with QUALIFY

The reference repeats the entire on-time-rate expression inside RANK() OVER (ORDER BY …) — correct, but verbose and error-prone (two copies must stay in sync). You cannot reference the on_time_pct alias in the same SELECT's window clause. On Snowflake/BigQuery/DuckDB you would compute the rate once in a subquery and filter or rank with QUALIFY; on Postgres, factor the rate into a CTE so it is written once:

WITH rates AS (
  SELECT supplier_id,
    100.0 * SUM(CASE WHEN actual_date <= scheduled_date THEN 1 ELSE 0 END)/COUNT(*) AS on_time_pct
  FROM shipments GROUP BY supplier_id
)
SELECT supplier_id, on_time_pct,
       RANK() OVER (ORDER BY on_time_pct DESC) AS rk
FROM rates;

Same result, the formula appears once. That refactor is the over-the-bar answer.

Ties — RANK vs DENSE_RANK, and NULL dates

Here every supplier has a distinct rate, so ranks are 1, 2, 3. But if two suppliers tied at, say, 100%, RANK() gives them both rank 1 and the next supplier rank 3 (a gap); DENSE_RANK() would give 1, 1, 2. Pick the one the scorecard wants. Also watch NULL dates: if actual_date is NULL (shipment not yet delivered), NULL <= scheduled_date is unknown, the CASE falls to ELSE 0, and the shipment counts as late — which may be wrong if it is simply still in transit. Decide whether undelivered shipments should be excluded from the denominator entirely.

Senior signal: recognise SUM(CASE WHEN … THEN 1 ELSE 0 END) as the conditional-count idiom and know COUNT(*) FILTER is its cleaner equivalent, force numeric division with 100.0, refactor the duplicated rate expression into a CTE (or QUALIFY on engines that have it), and handle boundary dates and NULL actual_date deliberately rather than letting them default to "late".
