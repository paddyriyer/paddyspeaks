A school board applies a "best of five" rule: each student sits six subjects, but only the top five marks count toward the final percentage. The interviewer is testing two things — can you unpivot six wide columns into rows so they can be ranked, and can you take a per-student top-N then average it. The hidden landmine is a sorting bug: if marks are compared as text, a single-digit score like 9 outranks 100, and the wrong five get picked.

1. The source data — exam

idenglishmathssciencegeographyhistorysanskrit
1908570609540
2505050505050
3100303030309

Student 2 has a six-way tie at 50; student 3 mixes a perfect 100 with a single-digit 9. Those two rows expose, respectively, the tie-breaking behaviour and the text-vs-numeric sort bug.

2. The SQL — unpivot, rank, average top 5 (original, flagged broken)

WITH marks_cte AS (
  SELECT id, english::text AS marks, 'English' AS subjects FROM exam
  UNION ALL SELECT id, maths::text,    'Maths'     FROM exam
  UNION ALL SELECT id, science::text,  'Science'   FROM exam
  UNION ALL SELECT id, geography::text,'Geography' FROM exam
  UNION ALL SELECT id, history::text,  'History'   FROM exam
  UNION ALL SELECT id, sanskrit::text, 'Sanskrit'  FROM exam
),
rank_cte AS (
  SELECT id, subjects, marks,
         ROW_NUMBER() OVER (PARTITION BY id ORDER BY marks DESC) AS rnk
  FROM marks_cte
),
percentage_cte AS (
  SELECT id,
    ROUND(AVG(CASE WHEN marks ~ '^-?[0-9]+(\.[0-9]+)?$' THEN marks::numeric END), 2) AS percentage_top5
  FROM rank_cte WHERE rnk <= 5 GROUP BY id
)
SELECT e.*, pc.percentage_top5 AS percentage
FROM exam e JOIN percentage_cte pc ON e.id = pc.id;

Broken: marks is cast to text, so ORDER BY marks DESC sorts lexicographically. The string '9' sorts after '100' (character '9' > '1'), so for student 3 the rule keeps 9 and discards 100 — it averages the wrong five.

2b. Corrected — rank on the numeric value

WITH marks_cte AS (
  SELECT id, english AS marks, 'English' AS subjects FROM exam
  UNION ALL SELECT id, maths,    'Maths'     FROM exam
  UNION ALL SELECT id, science,  'Science'   FROM exam
  UNION ALL SELECT id, geography,'Geography' FROM exam
  UNION ALL SELECT id, history,  'History'   FROM exam
  UNION ALL SELECT id, sanskrit, 'Sanskrit'  FROM exam
),
rank_cte AS (
  SELECT id, subjects, marks,
         ROW_NUMBER() OVER (PARTITION BY id ORDER BY marks DESC) AS rnk
  FROM marks_cte
),
percentage_cte AS (
  SELECT id, ROUND(AVG(marks), 2) AS percentage_top5
  FROM rank_cte WHERE rnk <= 5 GROUP BY id
)
SELECT e.*, pc.percentage_top5 AS percentage
FROM exam e JOIN percentage_cte pc ON e.id = pc.id;

Student 3's correct percentage is 44.00. The broken query keeps 9 and drops 100, producing a wrong, lower figure — a student penalised by a string-sort artefact.

4. Strategic explanations

The bug — lexical sort masquerading as numeric

english::text turns 100 into the string '100' and 9 into '9'. Text comparison is character-by-character: '9' begins with '9', '100' begins with '1', and '9' > '1', so '9' > '100'. The reference data hides this — all marks happened to be two digits, where lexical and numeric order coincide. Introduce one single-digit or three-digit mark and the ranking inverts. The fix is to never cast to text for ordering: rank on the native int, or if a cast is unavoidable, ORDER BY marks::numeric DESC.

Why the UNION ALL unpivot is the right shape

The marks live in six columns; ranking needs them in six rows. UNION ALL stacks each column into a (id, marks, subject) tuple — the portable unpivot idiom. UNION ALL, not UNION: two subjects with the same mark must both survive (a plain UNION would dedupe student 2's six identical 50s down to one row, and the average would be computed over too few rows). Postgres also offers UNNEST(ARRAY[english, maths, ...]) as a terser unpivot.

ROW_NUMBER vs RANK — the tie at 50

Student 2's six identical 50s force the issue. ROW_NUMBER assigns 1..6 arbitrarily among the ties, so "top 5" keeps exactly five rows — here harmless, since every value is 50 and the average is 50 regardless. But consider five marks tied for the 5th slot: ROW_NUMBER picks one arbitrarily, RANK/DENSE_RANK would keep all of them and "top 5" could become "top 8". For "best of five" the spec wants exactly five marks, so ROW_NUMBER is the correct choice — just be ready to say why, and to note the tie at the boundary is decided non-deterministically.

The regex guard — a symptom, removed by typing

marks ~ '^-?[0-9]+(\.[0-9]+)?$' only earns its keep if marks could be dirty text. With proper int columns the guard is unnecessary — the corrected query drops it entirely. Validating numeric format inside a query is a code smell pointing at an untyped schema; fix it at ingestion.

"Percentage" assumes each subject is out of 100

Averaging the top five marks is the percentage only because each subject's maximum is 100. If subjects had different maxima, you would first convert each to a percentage (marks * 100.0 / max_marks), then average the top five of those. Confirm the per-subject maximum before equating "average mark" with "percentage".

Senior signal: spot that casting marks to text makes ORDER BY DESC lexical — '9' outranks '100' — so the "best of five" silently picks the wrong subjects; the reference data only hides it because all marks are two digits. Ranking on the native numeric type, and choosing ROW_NUMBER (not RANK) so "top 5" stays exactly five, is the complete fix.
