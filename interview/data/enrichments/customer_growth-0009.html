Amazon's growth and retention teams obsess over the customers who almost buy — people who load a cart and then leave. They sit between window-shoppers (never add anything) and buyers, and they're the cheapest segment to convert: a free-shipping nudge often closes the gap. This question tests the everyday analytics-engineering loop — window an event stream, reshape long→wide, derive a rate safely, and bucket customers with ordered rules — in idiomatic pandas.

1. The input data

An events DataFrame: one row per customer action. event_type ∈ {view, add_to_cart, checkout, purchase}.

2. The approach

Step 1 — window. pd.to_datetime normalises the timestamps, then the cutoff is anchored to the data itself: events['event_ts'].max() - pd.Timedelta(days=days). Anchoring on the latest event (not datetime.now()) means the same code works for live data and historical replay tests.

Step 2 — long→wide. pivot_table(index='customer_id', columns='event_type', values='n', aggfunc='sum', fill_value=0) turns each event type into a per-customer count column. The defensive for col in (...) loop adds any event type that never appeared, so downstream code never hits a KeyError.

Step 3 — safe rate. Abandonment = 1 - purchase / add_to_cart. The trick is add_to_cart.replace(0, pd.NA) — turning a zero denominator into NA so the division yields NA instead of raising or producing inf; .fillna(0).clip(lower=0).round(2) then tidies it.

Step 4 — labels. _label applied row-wise; order matters. The unsteady rule (cart_adds >= 3 and abandonment >= 0.6) is checked first so it wins over plainer buckets — the cart_adds >= 3 floor filters out one-off noise. Step 5 renames columns and sorts worst-abandonment first.

4. Strategic explanations

Complexity

Time O(n log n) — the date filter and pivot are near-linear in n events; the final sort_values dominates. The .apply(_label, axis=1) is O(rows) but with Python-level per-row overhead. Space O(n) — the events copy plus the per-customer pivot.

The idiomatic alternative — vectorise the labels

apply(axis=1) is the slow part — it's a Python loop. np.select evaluates the same ordered rules as vectorised boolean masks:

import numpy as np
conds = [
    (p['add_to_cart'] >= 3) & (p['abandonment_rate'] >= 0.6),
    p['purchase'] > 0,
    p['checkout'] > 0,
    (p['add_to_cart'] == 0) & (p['purchase'] == 0),
]
choices = ['unsteady', 'buyer', 'checkout_dropout', 'window_shopper']
p['label'] = np.select(conds, choices, default='browser')

np.select picks the first true condition — same precedence as the if-ladder, but C-speed.

The edge case

The replace(0, pd.NA) is the headline edge case: a customer with zero cart-adds would otherwise divide by zero — here it becomes NA → fillna(0) → rate 0. A customer with more purchases than cart-adds (e.g. a purchase recorded without a tracked add) gives a negative rate; clip(lower=0) floors it at 0. An empty events frame would break the .max() anchor — worth an explicit guard.

The follow-up

"Score the cohort by revenue at risk, not just rate" — join an order-value table and weight. "Make the window rolling per-customer" — switch from a global cutoff to a per-customer groupby + time filter. "Why anchor on max() instead of now()?" — reproducibility: tests replay fixed historical data.

The trap

Reordering the _label rules so purchase > 0 is checked before unsteady. A customer who adds to cart 4 times and makes one purchase would then be tagged buyer and slip out of the actionable cohort — even though their abandonment is still high. Rule order encodes business priority; the actionable label must win. The other trap: dividing without the replace(0, pd.NA) guard, which yields inf/NaN rows that silently poison the sort.

Senior signal: anchors the lookback window on the data's own max() timestamp for reproducible replay, neutralises the divide-by-zero with replace(0, pd.NA) before computing the rate, orders the label rules so the actionable cohort wins ties, and replaces row-wise apply with vectorised np.select when the cohort grows past toy size.
