A customer-360 dashboard needs each customer's two most recent activity events for a "recent activity" panel. The interviewer specifically asks for the LATERAL pattern — probing whether you know PostgreSQL's correlated-subquery-in-the-FROM-clause, how it compares to a window-function top-N, and how each join type treats customers with no events.

1. The source data — customers and events

1 AliceJan 1 login, Jan 2 view, Jan 3 purchase3 events — only newest 2 wanted
2 BobJan 4 login1 event — fewer than 2
3 CaraJan 5 login, Jan 6 viewexactly 2 events
4 Dan— none —zero events — must be omitted

Three edge cases in one tiny dataset: Alice has more than 2 events (must trim to 2), Bob has fewer than 2 (return the 1 he has), and Dan has none (drop him entirely). The result must handle all three.

2. The SQL — CROSS JOIN LATERAL with per-customer LIMIT

SELECT c.id AS customer_id, c.name, e.id AS event_id, e.ts, e.kind
FROM customers c
CROSS JOIN LATERAL (
  SELECT id, ts, kind
  FROM events
  WHERE customer_id = c.id
  ORDER BY ts DESC
  LIMIT 2
) e
ORDER BY customer_id, ts DESC;

The LATERAL subquery runs once per customer row and may reference c.id — ordinary subqueries in FROM cannot see the outer row. For each customer it sorts that customer's events newest-first and takes the top 2.

4. Strategic explanations

What LATERAL actually does

Normally a subquery in the FROM clause is evaluated independently and cannot reference other items in the same FROM. LATERAL lifts that restriction: the subquery is re-evaluated for each row of the table to its left and may reference that row's columns — here WHERE customer_id = c.id. It is a correlated subquery that returns a set of rows rather than a single value, which is exactly what "top 2 per customer" needs.

CROSS JOIN LATERAL vs LEFT JOIN LATERAL — this is the key decision

The question says customers with no events should be omitted — so CROSS JOIN LATERAL is correct: when the subquery yields nothing (Dan), the customer produces no output rows. Had the requirement been "list every customer, NULLs if no events", you would write LEFT JOIN LATERAL (...) e ON true, which keeps the customer with NULL event columns. The join type is the entire difference between "drop Dan" and "keep Dan with NULLs" — choose it to match the spec.

LATERAL vs the window-function top-N

The classic alternative is ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY ts DESC) in a CTE, then WHERE rn <= 2. Both are correct. The trade-off: the window approach scans and sorts the entire events table once; LATERAL does a small ordered lookup per customer. With an index on events(customer_id, ts DESC), LATERAL can be dramatically faster — each iteration is an index range scan reading just 2 rows. For few customers against a huge events table, LATERAL wins; for a single pass over everything, the window function is fine.

Ties on the timestamp

ORDER BY ts DESC LIMIT 2 is non-deterministic if a customer has two events at the exact same ts for the 2nd/3rd slot — the engine may pick either. If "most recent" must be reproducible, add a deterministic tiebreaker: ORDER BY ts DESC, id DESC. The demo timestamps are all distinct so it does not bite here, but a real event stream will eventually collide — flag it.

Dialect notes

LATERAL is SQL-standard and supported by PostgreSQL, Oracle (also via CROSS APPLY), and MySQL 8.0.14+. SQL Server spells it CROSS APPLY / OUTER APPLY — OUTER APPLY being the analogue of LEFT JOIN LATERAL. Engines without it (older MySQL, SQLite) must fall back to the ROW_NUMBER window approach. Naming the APPLY equivalence shows cross-dialect fluency.

Senior signal: use CROSS JOIN LATERAL with a per-row ORDER BY ... LIMIT n for top-N-per-group — CROSS drops groups with no matches (Dan), LEFT JOIN LATERAL ... ON true would keep them with NULLs. With an index on (customer_id, ts DESC) LATERAL beats a whole-table ROW_NUMBER scan; add a tiebreaker for timestamp ties, and know SQL Server spells it CROSS/OUTER APPLY.
