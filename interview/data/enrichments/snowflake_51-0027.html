Fintech and mortgage platforms need to compute monthly payment amounts directly in SQL: "For each loan application, look up the interest rate valid at the application date and compute the fixed monthly payment using the standard amortisation formula." This question tests a temporally-bounded JOIN (matching a rate by date range), scalar financial math in SQL (POWER and the amortisation formula), and correct handling of cast types in arithmetic.

2. The SQL — temporal JOIN with amortisation formula

SELECT a.app_id, a.principal, a.term_months, r.rate_pct,
       a.principal::numeric
         * (r.rate_pct::numeric / 100 / 12)
         / (1 - POWER(1 + r.rate_pct::numeric / 100 / 12, -a.term_months::int)) AS monthly_payment
FROM applications a
JOIN rates r
  ON a.product_code = r.product_code
 AND a.applied_on BETWEEN r.valid_from AND r.valid_to;

4. Strategic explanations

The temporal JOIN — the date-range validity pattern

The JOIN condition has two parts: a.product_code = r.product_code (equi-join on product identity) and a.applied_on BETWEEN r.valid_from AND r.valid_to (the temporal filter that picks the rate valid at application time). This is the standard "slowly changing dimension" (SCD Type 2) lookup pattern: a rate table where each product has multiple rows, each valid for a date range, and you must pick exactly the one row whose range contains the target date.

The amortisation formula — derivation

The standard fixed-rate monthly payment formula is:

M = P * r / (1 - (1 + r)^(-n))

Where P = principal, r = monthly interest rate (annual_rate / 12), and n = number of payments (term_months). The formula is derived by setting the present value of n equal payments equal to P. In SQL: principal * (rate/100/12) / (1 - POWER(1 + rate/100/12, -term_months)). The POWER(base, -n) call computes (1+r)²³ using Snowflake’s and Postgres’s built-in POWER function.

Why explicit casts matter

If principal, rate_pct, and term_months are stored as integers, dividing integers produces integer results in some engines (e.g., 6 / 100 = 0 in strict integer division). The ::numeric and ::int casts ensure: (a) the rate arithmetic uses floating-point or fixed-point precision, and (b) POWER receives the correct numeric type. Always cast financial arithmetic columns explicitly to NUMERIC rather than relying on implicit promotion.

Edge case: gap or overlap in rate validity periods

The BETWEEN join assumes the rate table is gapless and non-overlapping for each product code — every application date maps to exactly one rate row. If there is a gap (no rate row covers the application date), the application is silently excluded from the result (INNER JOIN semantics). If two rows overlap for the same date, the application appears twice with two different rates. Always validate the rates table for completeness and non-overlap before running this query in production (a LEAD(valid_from) OVER (PARTITION BY product_code ORDER BY valid_from) check surfaces overlaps).

Dialect notes
  • Postgres (solution dialect): POWER(base, exp) is built-in. BETWEEN is inclusive on both ends ([valid_from, valid_to]). Correct for day-granularity date ranges.
  • Snowflake: identical — POWER and BETWEEN behave the same. POW is also an alias.
  • BigQuery: POW(base, exp). BETWEEN inclusive.
  • MySQL: POW(base, exp). BETWEEN inclusive.
Extension: total interest paid

monthly_payment * term_months - principal AS total_interest gives the total interest cost of the loan. Adding this column to the SELECT costs nothing and is always requested in mortgage analytics dashboards.

Senior signal: name this as the SCD Type 2 temporal lookup pattern (rate-table with valid_from/valid_to, one row per date), explain the amortisation formula derivation (PV of n equal payments = P), flag the gap/overlap risk in the rates table and offer the LEAD-based validation check, and always cast to NUMERIC explicitly for financial arithmetic to avoid integer-division surprises.
