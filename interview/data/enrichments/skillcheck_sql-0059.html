A recommendations team wants the basis of a "frequently bought together" feature: "which unordered pair of products co-occurs in the most orders?" This is market-basket analysis — itemset counting. The interviewer is probing whether you can generate all 2-product combinations within each order using a self-join, canonicalise each pair so {1,2} and {2,1} are the same, and rank by frequency.

The edge case is the tie at the top: pair {1,2} appears in orders 1, 2, 4 and pair {2,3} appears in orders 2, 3, 6 — both three times. The ranking must break that tie deterministically, and 3-item baskets must contribute all three of their pairs.

2. The SQL — self-join on order, p1 < p2

SELECT a.product_id AS p1, b.product_id AS p2, COUNT(*) AS occurrences
FROM order_items a
JOIN order_items b
  ON a.order_id = b.order_id
 AND a.product_id < b.product_id
GROUP BY a.product_id, b.product_id
ORDER BY occurrences DESC, p1, p2
LIMIT 3;

The table is joined to itself on the same order_id, so two rows of one basket meet. The a.product_id < b.product_id condition keeps each unordered pair exactly once and drops the self-match. GROUP BY the pair, COUNT(*) the orders it appeared in.

4. Strategic explanations

Why a.product_id < b.product_id is doing three jobs at once

A self-join on just order_id would pair every product with itself ({1,1}) and produce each real pair twice ({1,2} and {2,1}). The strict-inequality predicate < solves all of it in one stroke: it excludes the self-match (a product is never < itself) and canonicalises orientation (only the (smaller, larger) ordering survives). Using <= instead would let the self-pairs back in; using != would double-count. The operator choice is the de-duplication.

The GROUP BY is over the pair, the COUNT is over orders

After the join, one row exists per (pair, order) co-occurrence. Grouping by (p1, p2) and taking COUNT(*) counts how many orders each pair appeared in. If a single order could list the same product twice (duplicate rows), COUNT(*) would overcount — you would first SELECT DISTINCT order_id, product_id in a CTE, or switch to COUNT(DISTINCT a.order_id). Here the data is clean, but flagging that assumption is the senior move.

Deterministic tie-breaking

Two pairs tie at 3. ORDER BY occurrences DESC alone leaves their relative order undefined — the engine could return either first, and LIMIT 3 would be non-deterministic at the boundary. Adding p1, p2 as secondary sort keys makes the output stable and reproducible. Any time LIMIT meets a tie, an explicit total ordering is mandatory; "the top N" is meaningless without it.

Combinatorial blow-up on large baskets

A basket of k items generates k(k−1)/2 pairs — quadratic in basket size. A 100-item basket produces ~5,000 pair rows; the self-join can explode on wide baskets or huge order tables. For production market-basket work at scale you would cap basket size, pre-aggregate, or move to a dedicated frequent-itemset algorithm (Apriori, FP-growth). Naming this scaling limit unprompted is exactly what an interviewer wants to hear.

The question's loose wording

The prompt mentions "orders containing exactly 2 distinct products" then immediately relaxes it to "both products at least, regardless of size". The solution implements the relaxed reading — it counts a pair's co-occurrence in any basket, including 3-item ones. That is the sensible market-basket definition; if the interviewer truly wanted only 2-item baskets you would add HAVING COUNT(DISTINCT product_id) = 2 in a per-order CTE. Stating which reading you took resolves the ambiguity out loud.

Senior signal: generate co-occurring pairs with a self-join on order_id and let a.product_id < b.product_id simultaneously drop self-matches and canonicalise each unordered pair, break the top-N tie with an explicit secondary sort, and call out the quadratic pair blow-up on wide baskets — real frequent-itemset mining needs Apriori/FP-growth.
