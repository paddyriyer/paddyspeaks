Microsoft's commerce-reporting team wants revenue broken out by order status — how much sits in completed, how much in pending, how much was lost to cancelled. A clean GROUP BY status. The interviewer is checking whether you know what SUM does with a NULL amount, and whether the status column is clean enough to group on directly.

1. The source data — orders

statusamountorder_idcustomer_idorder_date
completed12011012024-01-01
completed20021022024-01-02
cancelled8031032024-01-03
pending5041042024-01-04
completedNULL51052024-01-05

There are three completed orders, but one has a NULL amount (order 5, perhaps captured before pricing finished). That NULL decides what the completed bucket sums to, and whether a status with only NULL amounts would show 0 or a blank.

2. The SQL — GROUP BY status

SELECT status, SUM(amount) AS revenue
FROM orders GROUP BY status;

One group per distinct status value; SUM(amount) totals the revenue within each.

3. The resulting analysis

statusrevenue
completed320
cancelled80
pending50

The completed bucket is 320 (120 + 200) — order 5's NULL amount was simply skipped, not treated as 0. With three completed orders but only two priced, the sum is over the two. The other two statuses each have a single order. Note the rows come back in no guaranteed order — GROUP BY does not sort.

4. Strategic explanations

SUM ignores NULLs — usually right, but state it

SUM(amount) skips NULL amounts, so order 5 contributes nothing to the completed total. That is normally correct — an unpriced order should not be counted as $0 revenue. But the edge case bites when a status group has only NULL amounts: SUM then returns NULL, not 0, and a dashboard tile shows a blank. Guard it when the consumer needs a number:

SELECT status, COALESCE(SUM(amount), 0) AS revenue
FROM orders GROUP BY status;

COALESCE wraps the aggregate. Decide deliberately: skip unpriced orders (current behaviour) or fold them in as $0 with SUM(COALESCE(amount, 0)).

Counting orders alongside revenue — COUNT(*) vs COUNT(amount)

Reports usually want order volume next to revenue. Be precise about which count:

SELECT status,
       COUNT(*)            AS order_count,
       COUNT(amount)       AS priced_orders,
       COALESCE(SUM(amount), 0) AS revenue
FROM orders GROUP BY status;

For completed, COUNT(*) = 3 but COUNT(amount) = 2. The gap is exactly the unpriced orders — surfacing both numbers tells the reader the revenue figure is built on two of three orders, not all three.

Dirty status values — the silent group-fragmenter

GROUP BY status trusts the column to be clean. If the data contains 'Completed', 'completed', and 'completed ' (trailing space), those become three separate groups and the revenue is scattered across them. On real data, normalise: GROUP BY LOWER(TRIM(status)), or fix the source. A senior answer raises this before trusting the raw column.

NULL status itself

If status can be NULL, GROUP BY collects all NULL-status rows into one group — GROUP BY treats NULLs as equal to each other (unlike =). It appears as a row with a blank status. That is usually acceptable, but label it (COALESCE(status, 'unknown')) so the report does not show a mysterious empty bucket.

What "revenue" should mean — the definitional question

The query reports a row for cancelled — $80 of orders that produced no money. Is "cancelled revenue" meaningful, or should the report show only realised revenue? Often the team wants completed only, or a FILTER split: SUM(amount) FILTER (WHERE status = 'completed') as realised revenue with the rest shown as pipeline. Confirm whether each status is a revenue bucket or whether only some count.

Senior signal: state that SUM skips NULL amounts and that an all-NULL status group returns NULL not 0 — coalesce when a number is required — pair the right COUNT with the sum to expose how many orders are unpriced, and flag dirty or NULL status values that would silently fragment the groups before trusting the raw column.
