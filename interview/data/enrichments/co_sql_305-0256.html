Salesforce's growth team reports week-over-week web traffic to the marketing org. The headline is a single percentage per week: did visits go up or down, and by how much? The natural tool is LAG — reach back one row for last week's number. The reference solution wraps that in a defensive layer of regex sanitising and NULLIF, which is worth understanding: it is guarding against a column that should be an integer but might not be.

1. The source data — traffic

week_startvisits
2025-01-061000
2025-01-131200
2025-01-201080
2025-01-271080
2025-02-031350

Two edge cases are baked in. The first week has no prior week, so its growth is undefined. And weeks 3 and 4 are flat at 1080 — a genuine 0% change, which must not be confused with a NULL.

2. The SQL — LAG week-over-week %

SELECT week_start, visits,
  100.0*(NULLIF(regexp_replace(visits::text, '[^-.0-9]', '', 'g'), '')::numeric - LAG(NULLIF(regexp_replace(visits::text, '[^-.0-9]', '', 'g'), '')::numeric) OVER (ORDER BY week_start))/NULLIF(LAG(NULLIF(regexp_replace(visits::text, '[^-.0-9]', '', 'g'), '')::numeric) OVER (ORDER BY week_start), 0) AS wow_pct
FROM traffic;

Strip away the armour and the core is 100.0 * (this - prev) / prev, where prev = LAG(visits) OVER (ORDER BY week_start). The regexp_replace scrubs any stray characters out of visits; the outer NULLIF(…, 0) stops a division-by-zero if a prior week had zero visits.

The first week is correctly NULL — no prior week to compare. Week 2 is +20%, week 3 is −10%, week 4 is exactly 0% (not NULL — the flat week is a real data point), week 5 is +25%. The trend is choppy but recovering.

4. Strategic explanations

Why the regex armour? — the schema is lying

If visits were a clean INTEGER, this query would be a simple 100.0*(visits - LAG(visits) OVER w)/LAG(visits) OVER w. The regexp_replace(…, '[^-.0-9]', '', 'g') strips every character that is not a digit, minus or dot — it exists because the column might actually be TEXT holding values like "1,200" or "1200 visits". A senior candidate calls this out: "this defends against a dirty text column; if the column is truly numeric, drop the regex for readability."

The 0% vs NULL distinction — the trap

Week 4 returns 0, week 1 returns NULL, and they mean completely different things: 0% is "we measured, no change"; NULL is "we could not measure". A dashboard that renders both as a blank, or both as "0%", is wrong. The NULLIF(prev, 0) in the denominator is what keeps a zero prior week from throwing a division error — it converts the divide-by-zero into a clean NULL instead.

The window-frame subtlety — LAG needs a deterministic order

LAG reaches back over ORDER BY week_start. If two rows shared a week_start, "the previous row" would be ambiguous and the percentage non-deterministic. With one row per week that is fine — but for safety on dirtier data, ensure week_start is unique, or add a tiebreaker to the window's ORDER BY.

Computing LAG once with a CTE

The reference calls the full LAG(…) expression three times. The planner usually de-duplicates it, but a CTE makes intent obvious and the SQL legible:

WITH w AS (
  SELECT week_start, visits,
         LAG(visits) OVER (ORDER BY week_start) AS prev_visits
  FROM traffic
)
SELECT week_start, visits,
       100.0 * (visits - prev_visits) / NULLIF(prev_visits, 0) AS wow_pct
FROM w;
Edge case: gaps in the week sequence

LAG fetches the previous row, not the previous calendar week. If a week is missing from the table entirely, LAG happily compares against two weeks ago and labels it week-over-week. If the series can have holes, generate a complete week spine and LEFT JOIN traffic onto it before applying LAG.

Senior signal: recognise the regex/NULLIF layer as defensive plumbing for a possibly-dirty text column and offer the clean version if the column is numeric, articulate why a flat week returns 0% while the first week returns NULL, and note that LAG trusts row order — gaps in the week sequence silently corrupt the comparison.
