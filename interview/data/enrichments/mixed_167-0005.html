An HR dashboard needs a roster: every employee next to the name of their department. The query is the canonical inner join on dept_id. It is the most common SQL operation in existence — so the interviewer is not testing whether you can write it, but whether you notice which employees the inner join silently drops.

Two edge cases: Eve has a NULL dept_id (unassigned), and department 40 (Research) has no employees. An inner join will lose both.

2. The SQL — inner join on the department key

SELECT e.name, d.dept_name
FROM employees e
JOIN departments d ON e.dept_id = d.dept_id;

Four rows out of five employees. Eve is gone — her NULL dept_id matches nothing. And Research never appears: no employee points to it. The inner join keeps only rows that match on both sides.

4. Strategic explanations

Why JOIN means INNER JOIN

Bare JOIN is shorthand for INNER JOIN in every major dialect. It returns the intersection: a row appears only when the ON predicate is true. e.dept_id = d.dept_id is never true when e.dept_id is NULL — NULL never equals anything, not even another NULL — so Eve is excluded by the three-valued logic, not by a bug.

The dropped-rows trap — pick the join that matches the question

If the dashboard must list every employee, including the unassigned, use a left join so employee rows survive even without a department match:

SELECT e.name, d.dept_name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;

Eve then appears with dept_name = NULL. If instead you must list every department including empty ones, flip it (departments LEFT JOIN employees) or use a FULL OUTER JOIN to keep both unmatched sides. Naming which entity must be complete is the senior move.

Ordering is not guaranteed

Notice the result led with Dave, not Alice — without an ORDER BY, row order reflects the join's physical execution and can change between runs or after an index is added. For a real roster, add ORDER BY d.dept_name, e.name. Never rely on incidental ordering.

Ambiguous columns and the schema overlap

Both tables carry dept_name (and name, and manager_id). Selecting dept_name unqualified would raise an "ambiguous column" error. Always table-qualify columns when the schemas overlap — d.dept_name, e.name — both for correctness and so the next reader knows the source.

Common mistakes

Joining on the wrong key (e.g. manager_id) compiles fine and returns plausible nonsense. A missing ON clause yields a cross join — every employee paired with every department. And if dept_id values are non-unique in departments, the join multiplies rows; confirm the join key is a primary or unique key on at least one side.

Senior signal: call out that an inner join silently drops the NULL-department employee and the employee-less department, then offer the correct outer join for whichever side the business needs complete — and always table-qualify columns when schemas overlap.
