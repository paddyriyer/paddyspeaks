PayPal's ledger team needs a current balance per account, derived from a stream of transactions. The query is a tidy conditional sum — credits add, debits subtract. The interview is about the assumptions baked into that one CASE: that every non-credit is a debit, that every transaction counts regardless of status, and that summing a mutable log is the same as a stored balance.

1. The source data — transactions

Account 1 nets positive across three transactions. Account 2 is the edge case: a 200 debit against only a 75 credit leaves a negative balance — an overdraft the query must be willing to report. Account 3 has a single transaction.

2. The SQL — signed conditional SUM

SELECT account_id,
  SUM(CASE WHEN txn_type='credit' THEN amount ELSE -amount END) AS balance
FROM transactions GROUP BY account_id;

Per account, each credit contributes +amount and everything else contributes -amount; SUM folds the signed values into one balance.

Account 1: 100 − 30 + 50 = 120. Account 2: 75 − 200 = −125 — the overdraft surfaces correctly. Account 3: a lone 500 credit. Rows are unordered absent an ORDER BY. The negative balance is the point: the query does not clamp at zero, and for a ledger that is right — a negative balance is real information, not an error.

4. Strategic explanations

The ELSE -amount trap — "everything that isn't a credit is a debit"

The CASE has no explicit debit branch; it assumes every non-credit row is a debit. Real ledgers have more types: refund, fee, reversal, chargeback, hold, adjustment. A refund probably should add, not subtract — but ELSE -amount silently subtracts it, corrupting the balance. Be explicit:

SUM(CASE
      WHEN txn_type IN ('credit','refund','reversal') THEN amount
      WHEN txn_type IN ('debit','fee','chargeback')   THEN -amount
      ELSE 0   -- or RAISE: an unknown type should not be guessed
    END) AS balance

Naming the open-ended ELSE as a risk — and asking for the full list of transaction types — is the senior instinct.

Status is in the schema and the query ignores it

The table has a status column — transactions can be pending, failed, completed, reversed. The solution sums all of them. A failed debit that still subtracts, or a pending credit counted as settled, gives a wrong balance. A "current available balance" almost always filters:

SELECT account_id, SUM(CASE WHEN txn_type='credit' THEN amount
                            ELSE -amount END) AS balance
FROM transactions
WHERE status = 'completed'
GROUP BY account_id;

There is a genuine product distinction here: posted balance (completed only) versus available balance (also nets pending holds). Surfacing that the query conflates them is over-the-bar.

Accounts with no transactions are missing

A newly opened account with zero transactions has no rows, so GROUP BY produces no group — it is absent rather than showing a balance of 0. If the report must list every account, LEFT JOIN from the accounts table and COALESCE(SUM(...), 0) the balance.

Negative amounts — sign applied twice

The CASE assumes amount is always stored positive and the type carries the sign. If the data instead stores debits as negative numbers already, then ELSE -amount flips a negative into a positive — a debit that adds to the balance. Confirm the sign convention: is amount a magnitude, or already signed? The two conventions need opposite queries.

Summing a log vs. trusting a stored balance

Recomputing the balance from the full transaction history is correct and auditable, but O(history) per account — expensive once an account has millions of rows. Production ledgers keep a materialised balance (or periodic snapshots) and only sum transactions since the last snapshot. Mentioning the snapshot pattern shows you have thought past the 6-row demo to a real ledger.

Senior signal: challenge the implicit "non-credit means debit" by enumerating real transaction types, filter on status and distinguish posted vs available balance, confirm whether amount is a magnitude or already signed, and note that summing the whole log does not scale — snapshots do.
