DoorDash orders track two parallel state machines: the restaurant's state (preparing / ready) and the dasher's state (assigned / at_restaurant / en_route / delivered). The order is in a unique combined state at every point in time, and the period changes whenever either dimension changes. The query computes period boundaries on the composite key.

2. The SQL — concat both axes, then standard LAG-detect-change

WITH composite AS (
  SELECT order_id, ping_at,
    restaurant_state || '|' || dasher_state AS combined_state,
    restaurant_state, dasher_state
  FROM order_state_pings
),
changes AS (
  SELECT order_id, ping_at, combined_state, restaurant_state, dasher_state,
    LAG(combined_state) OVER (PARTITION BY order_id ORDER BY ping_at) AS prev_combined
  FROM composite
),
filtered AS (
  SELECT order_id, ping_at, restaurant_state, dasher_state, combined_state
  FROM changes
  WHERE combined_state != prev_combined OR prev_combined IS NULL
)
SELECT order_id, restaurant_state, dasher_state,
  ping_at AS period_start,
  LEAD(ping_at) OVER (PARTITION BY order_id ORDER BY ping_at) AS period_end
FROM filtered
ORDER BY order_id, period_start;

3. Strategic explanations

The string-concat composite key

restaurant_state || '|' || dasher_state creates a single comparable token for the change-detection step. The | separator avoids ambiguity (without it, 'pre' + 'paring' and 'preparing' + '' would collide). Cleaner than nested OR predicates and faster than tuple comparison in most engines.

Why "either dimension changes" → new period

DoorDash analytics needs to attribute time correctly: "the order was in (ready, at_restaurant) for 2 minutes" is a meaningful slice. If you only tracked the restaurant axis, the at_restaurant→en_route transition would disappear; if only the dasher axis, the preparing→ready transition would disappear. Composite tracking captures all transitions.

Production attribution: average duration per composite state, per market, per cuisine. The period with longest duration is the bottleneck to invest in.

The "stuck order" detection

Any composite state period > 30 minutes is alarming. Production fires an alert when the LIVE period (where period_end IS NULL) has period_start > 30 minutes ago — operations team intervenes manually.

Three-axis generalization

Add customer_state (idle / app_open / order_tracking) for the third axis. The pattern generalizes — the only change is adding to the concat string. Three-axis composite is the upper bound; beyond that, prefer storing each axis as its own period table and doing temporal joins for cross-axis questions.

The non-monotonic gotcha

Restaurant state should only go preparing → ready (one-way). If you see ready → preparing, that's data corruption — re-emitted with stale field. Production guard: validate state-machine transitions in the producer, reject illegal sequences, alert on attempts.

Senior signal: composite-state via string-concat for parallel state machines, period-on-EITHER-axis-change for honest latency attribution, decompose periods into ownership-attributed bottleneck buckets, and validate monotonic transitions at the producer (not just at query time). The three-axis generalization sets the upper bound before recommending separate period tables + temporal joins.
