A window Window.orderBy("ts") with no partitionBy looks innocent. It is the most dangerous Spark anti-pattern in the entire DataFrame API. Spark cannot parallelise an unbounded-global window — every row must be collated into a single partition for the cumulative computation. On a 1-billion-row DataFrame this guarantees an OOM. This question is a trap; the senior answer is "I would refuse to write this in production, and here's what I'd do instead."

1. The source data — the transaction stream that breaks this pattern

propertyvalue
tabletransactions(txn_id, account_id, ts, amount)
rows1,200,000,000 (one year of activity)
size~180 GB Parquet
partitions on disk365 (by ds = date(ts))
cluster50 executors × 16 GB RAM
question asked"Add a global running total of amount ordered by ts."

The interviewer's prompt LOOKS like a textbook running total. The data shape says: no single executor can hold 180 GB in memory. The conflict is the whole question.

2. The problem — what Window.orderBy("ts") with no partition actually does

Spark plan when partitionBy is OMITTED:

  Stage N-1:  ┌─────────────────────────────────────────────────────┐
              │ Shuffle ALL rows to ONE executor (single partition) │
              └────────────────────┬────────────────────────────────┘
                                   │
                                   ▼
  Stage N:    ┌─────────────────────────────────────────────────────┐
              │ Sort 1.2B rows on ONE executor                      │ ← O(N log N) on 180 GB
              │ Apply window function row-by-row in order           │ ← spill to disk
              │ Emit running totals                                 │
              └─────────────────────────────────────────────────────┘

  Spark warning in the log:
   "WARN WindowExec: No Partition Defined for Window operation!
    Moving all data to a single partition, this can cause serious
    performance degradation."

  Outcome: OOM at ~30% through, after burning 4 hours of cluster time.

This is not a bug — it's the correct semantics. A running total ordered globally has no parallelisation strategy that produces row-by-row correct cumulative sums without seeing all prior rows. Spark obeys the spec by doing the dumb-but-correct thing: collapse to one task. The fix is to change the semantics, not to optimise the operator.

3. The fix — three escape hatches, ordered by preference

Fix A — Add a partitionBy that matches the analytical question

"Running total" is almost never globally meaningful. It's usually running total per account or per day per account. Adding the partition column makes the window embarrassingly parallel.

from pyspark.sql import SparkSession, Window, functions as F
from pyspark.sql import Row

spark = SparkSession.builder.appName("window-without-partition").getOrCreate()

# ─── Seeded sample so the example is runnable ─────────────────────────────
txns = spark.createDataFrame([
    Row(txn_id=1, account_id="A", ts="2026-01-01 09:00", amount=100.0),
    Row(txn_id=2, account_id="A", ts="2026-01-01 09:05", amount= 50.0),
    Row(txn_id=3, account_id="B", ts="2026-01-01 09:10", amount=200.0),
    Row(txn_id=4, account_id="A", ts="2026-01-01 09:15", amount= 25.0),
    Row(txn_id=5, account_id="B", ts="2026-01-01 09:20", amount= 75.0),
    Row(txn_id=6, account_id="A", ts="2026-01-01 09:25", amount=300.0),
]).withColumn("ts", F.to_timestamp("ts"))

# GOOD: per-account running total. Parallelises across accounts.
# Each account's rows shuffle to ONE task, but accounts spread across the cluster.
w_per_account = Window.partitionBy("account_id").orderBy("ts") \
                       .rowsBetween(Window.unboundedPreceding, Window.currentRow)

per_account = (txns
    .withColumn("rn",            F.row_number().over(w_per_account))
    .withColumn("running_total", F.sum("amount").over(w_per_account))
)
per_account.show()
# A: 100 → 150 → 175 → 475 (cumulative per account, parallel across accounts)
# B: 200 → 275
Fix B — When you truly need a global running total: two-pass aggregation

This is the senior trick. Compute per-partition running totals in parallel, then a small second pass adds the inter-partition offsets.

# Step 1 — bucket the data into K range-partitions on the order column.
#   range-partition ensures each partition holds a contiguous time slice.
K = 200
ranged = txns.repartitionByRange(K, "ts").withColumn(
    "pid", F.spark_partition_id()
)

# Step 2 — local running totals inside each partition (parallel, no shuffle).
w_local = Window.partitionBy("pid").orderBy("ts") \
                 .rowsBetween(Window.unboundedPreceding, Window.currentRow)
locally = ranged.withColumn("local_running", F.sum("amount").over(w_local))

# Step 3 — sum of amounts per partition, used to compute the offset for each partition.
partition_totals = locally.groupBy("pid").agg(F.sum("amount").alias("part_sum"))

# Step 4 — cumulative offset across partitions (this DF has K rows — runs on 1 task, that's fine).
w_offsets = Window.orderBy("pid").rowsBetween(Window.unboundedPreceding, -1)
offsets = partition_totals.withColumn(
    "offset",
    F.coalesce(F.sum("part_sum").over(w_offsets), F.lit(0.0))
).select("pid", "offset")

# Step 5 — global running total = local running total + offset for that partition.
global_running = (locally.join(offsets, "pid")
                  .withColumn("running_total", F.col("local_running") + F.col("offset"))
                  .drop("pid", "local_running"))

Why this works: sum is associative. SUM(p1) + SUM(p2) + ... + SUM(pk-1) + local_running_in_pk equals the true global running total at any row in partition pk. The K=200 second-pass DataFrame is small enough to run on one task without harm.

Fix C — Push it down to SQL with proper ranges

If the question really is "give me one running total column at the end of the day", consider whether you actually need row-level cumulative values or just N checkpoint values:

# Maybe what was REALLY wanted: hourly running totals (24 rows/day, not 1.2B rows).
hourly = (txns
    .groupBy(F.window("ts", "1 hour").alias("hr"))
    .agg(F.sum("amount").alias("hourly_total")))

w_hr = Window.orderBy("hr").rowsBetween(Window.unboundedPreceding, Window.currentRow)
hourly.withColumn("running_total", F.sum("hourly_total").over(w_hr)).show()
# 24 rows × N days — fits in one task naturally, no anti-pattern.

4. Where this appears in production

  • JPMorgan's intraday position monitor had this exact anti-pattern in a job that ran nightly across 800 GB of trade data. The first version used Window.orderBy("trade_ts") for a "running net exposure" column. Failed every Friday (heaviest volume day) at the 4-hour mark. Fixed by partitioning by (book_id, trade_date) — running exposure per book per day was the actual business question.
  • Stripe's daily payout reconciliation deliberately uses Fix B (two-pass) for the global running balance per merchant: repartitionByRange("settlement_ts", numPartitions=400), then local cumsum, then offset join. Production-grade running totals at 10 B rows.
  • Uber's surge multiplier validation initially used a global-window running average over time, which OOMed on the LA region (highest volume). Refactor: partition by (city, hour_bucket) — every analytical question really was per-city anyway.
  • Snowflake / BigQuery handle global windows natively because their execution model can stream-sort across nodes. But even there, a SUM() OVER (ORDER BY ts) on a 1B-row table is a 30-minute query — not OOM, just slow. Range-partitioned two-pass is faster on those engines too.
  • Pandas cumsum() is single-node by design; this whole problem disappears at the Pandas scale (< 50 GB). The Spark-specific issue is that the API looks like pandas but the execution model rejects it.

5. Interviewer probes — the edge cases

Probe 1 — "Why does Spark warn instead of refusing?"

Backward compatibility with small DataFrames. Window.orderBy(...) with no partition is legal SQL — every database supports it — and works fine on a 1 K-row DataFrame. Spark warns once at plan time so dev/test code keeps running, but the warning is in cluster logs where you might miss it. Production teams set spark.sql.parser.quotedRegexColumnNames=true AND turn the warning into an error with a custom QueryExecutionListener.

Probe 2 — "What if I actually need a row-level global rank for the whole dataset?"

Rare, but real (e.g. "label every transaction with its overall position by time, system-wide"). Two paths:

  • zipWithIndex via RDD: df.rdd.zipWithIndex().toDF() — distributed, gives every row a monotonically-increasing global index, but the index isn't sorted unless you sort first. Cost: one full shuffle + scan.
  • The two-pass pattern from Fix B but with row_number() + partition offsets (offset = cumulative row count of preceding partitions).
  • monotonically_increasing_id(): fast but not contiguous (it encodes partition ID in high bits) and not ordered. Use only when you need a unique row identifier, not a sortable rank.
Probe 3 — "Is row_number() vs sum() different here?"

For the OOM behaviour, no — both force the single-partition shuffle. For the parallelisation fix, they differ:

  • sum/min/max/count are associative → Fix B (two-pass) works.
  • row_number/rank/dense_rank/percent_rank need a global sort to assign numbers; you can fake it with partition-count offsets (sum of row counts of preceding partitions) plus local row_numbers.
  • lag/lead at a global level cross partition boundaries; need a special "last row of previous partition" join. Usually a sign the analysis should be per-account.
  • percentile_cont / median are not associative — Fix B doesn't work. Use approx_percentile (HLL-style sketch, mergeable).
Probe 4 — "How does AQE affect this?"

It doesn't help. AQE optimises shuffle partition counts and detects skew, but it cannot rewrite a single-partition window into a parallel one — the semantics genuinely require seeing every prior row in order. AQE is helpful for the Fix B two-pass pattern (it'll auto-coalesce the small offsets DataFrame), but the original anti-pattern remains broken.

Probe 5 — "What about rowsBetween(unboundedPreceding, currentRow) vs rangeBetween?"

Same single-partition trap; the frame type doesn't change the parallelisation. But rangeBetween uses the ORDER BY column's values for the frame boundary (e.g. "all rows within 7 days before"), so it's the right pick for time-windowed running totals. Per-account 7-day running totals: Window.partitionBy("account_id").orderBy(F.col("ts").cast("long")).rangeBetween(-7*86400, 0) — cast to seconds because rangeBetween wants a numeric column.

Probe 6 — "How do you spot this anti-pattern in code review?"

Three grep patterns:

  • Window.orderBy with no partitionBy on the same line or the line above.
  • "Window operation" in Spark driver logs — the WARN message.
  • SinglePartition in df.explain() output where it shouldn't be. Look for an Exchange SinglePartition just before a Window physical operator.

Production teams add a SparkListener that fails the job on Exchange SinglePartition nodes over a row-count threshold.
