A merchandising team runs a category tree and wants a roll-up: for every node — leaf or internal — the total sales of all products anywhere in its subtree. Products only ever hang off leaf categories, so internal nodes have no products of their own; their total has to be summed from descendants. The interviewer is checking whether you can build an ancestor–descendant closure with a recursive CTE and then aggregate against it without double-counting.

The edge case is the internal nodes — All, Electronics, Apparel have zero products attached directly; their totals exist only by rolling up children. The query must still report them, not drop them.

2. The SQL — recursive descendant closure, then LEFT JOIN aggregate

WITH RECURSIVE descendants(ancestor_id, descendant_id) AS (
  SELECT category_id, category_id FROM category          -- every node is its own descendant
  UNION ALL
  SELECT d.ancestor_id, c.category_id
  FROM descendants d JOIN category c ON c.parent_id = d.descendant_id
)
SELECT c.category_id, c.name,
       COALESCE(SUM(p.sales), 0) AS total_sales
FROM category c
LEFT JOIN descendants d ON d.ancestor_id = c.category_id
LEFT JOIN product p ON p.category_id = d.descendant_id
GROUP BY c.category_id, c.name
ORDER BY c.category_id;

The recursive CTE builds the transitive closure: one (ancestor, descendant) pair for every node and each node beneath it, plus the self-pair from the anchor. Joining a category to its descendant set, then to products on those descendants, gathers every product in the subtree; SUM totals them and COALESCE turns "no products" into 0.

4. Strategic explanations

Why a closure, not a one-level join

A naive JOIN category child ON child.parent_id = c.category_id only reaches direct children — All would see Electronics and Apparel but not the leaf products two levels down. The recursive CTE walks every depth, emitting an (ancestor, descendant) pair for each node-under-node relationship. Aggregating against that closure is depth-agnostic: it works for a 2-level or a 20-level tree with no query change.

The anchor's self-pair is load-bearing

The anchor SELECT category_id, category_id emits each node as its own descendant. That is what lets a leaf see its own directly-attached products: without the self-pair, Phones would join only to its (empty) set of children and report 0. "Every node is an ancestor of itself" is the reflexive part of the closure — drop it and leaves silently lose their own sales.

COALESCE + LEFT JOIN keeps empty subtrees visible

An internal node whose subtree happened to contain no products (none here, but possible) would produce no product rows; an inner join would drop the category entirely. LEFT JOIN product keeps the category row with p.sales = NULL, and SUM of all-NULL is NULL, so COALESCE(…, 0) converts it to the required 0. Both LEFT JOINs matter: the first keeps the category, the second keeps it when products are absent.

Why no double-counting here — and when it would happen

Each product hangs off exactly one leaf, and each leaf appears once in any ancestor's descendant set, so every product is counted once per ancestor. If products could be attached to internal nodes too, or a category could have two parents (a DAG, not a tree), a product could reach an ancestor by multiple paths and be summed repeatedly. With a strict tree and leaf-only products, the arithmetic is clean — state that assumption out loud.

Termination and dialect notes

The recursion terminates because a tree is finite and acyclic: each step descends one level, and leaves produce no further rows. A genuine cycle would loop forever — you would then track a visited-path array and filter it. WITH RECURSIVE is standard in Postgres, SQLite, MySQL 8+, and SQL Server (where the keyword is just WITH); Oracle offers both CONNECT BY and recursive CTEs.

Senior signal: model "subtree roll-up" as a recursive ancestor–descendant closure with the reflexive self-pair in the anchor, aggregate via double LEFT JOIN + COALESCE so internal and empty nodes still report, and call out that the no-double-count guarantee depends on a strict tree with leaf-only products.
