Stripe Radar's bread-and-butter rule: 3+ payment attempts within 5 minutes from the same user signals card-testing fraud. The query is a self-join on a sliding window.

1. The source data — payment_attempts

2. The SQL — sliding-window via self-join

SELECT a.user_id, a.attempted_at,
  COUNT(b.attempt_id) AS attempts_in_5min
FROM payment_attempts a
JOIN payment_attempts b
  ON b.user_id = a.user_id
 AND julianday(b.attempted_at) BETWEEN julianday(a.attempted_at) - 1.0/288 AND julianday(a.attempted_at)
GROUP BY a.user_id, a.attempted_at
HAVING COUNT(b.attempt_id) >= 3
ORDER BY a.user_id, a.attempted_at;

3. Strategic explanations

The 1.0/288 trick

5 minutes = 5/(60*24) days = 1/288 days. The julianday subtraction in days; this constant gives a 5-minute window. Memorize: 1/24 = 1 hour, 1/288 = 5 min, 1/1440 = 1 minute.

Sliding window via self-join — the canonical pattern

For each row a, count all rows b for the same user within 5 minutes. SQLite/MySQL can't do RANGE BETWEEN INTERVAL windows; the self-join is the portable workaround. O(n²) without a composite index — production needs (user_id, attempted_at) indexed.

The "card testing" pattern

Fraudsters try stolen card numbers in rapid succession until one works. Signature: 3-10 attempts in 30 seconds, mostly failing, eventually one succeeds. Velocity rule catches the cluster.

False-positive: legitimate retry

A user who fat-fingers their card 3x in 1 minute is NOT a fraudster. Add a refinement: require the attempts span DIFFERENT card_last4 to flag:

HAVING COUNT(*) >= 3 AND COUNT(DISTINCT card_last4) >= 3
The action — block, allow, or step-up?

Three responses to detected velocity:

  • Block — too aggressive, hurts conversion
  • Step-up to 3DS — friction but legitimate
  • Allow + flag — wait for chargeback signal, learn pattern

Stripe Radar's ML balances. The query above is the candidate-pool feeder; the decision logic is downstream.

Cross-account variation

Velocity at the IP level catches networks of fraudsters using many fake accounts. Slice by IP instead of user_id for that.

Senior signal: use self-join sliding-window for non-supporting dialects, distinguish legitimate retry from fraud (different last4 / different cards), articulate the block-vs-step-up-vs-allow action menu, and pivot to IP-level for fraud-network detection.
