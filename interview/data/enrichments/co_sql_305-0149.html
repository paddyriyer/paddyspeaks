GE's finance team keeps sales in a long, tall table — one row per year/quarter/amount — but the board deck wants it wide: one row per year, four quarter columns across. That reshape is a pivot, and the portable way to do it in SQL is conditional aggregation: a CASE inside each SUM. The query is mechanical once you see the trick; the interview value is in what the CASE returns when it does not match, and what happens when a year/quarter pair appears more than once.

1. The source data — sales

yearquarteramount
2024Q1100
2024Q2150
2024Q3120
2024Q4200
2025Q1180
2025Q2210
2025Q120

2024 is complete — one row per quarter. 2025 is the interesting year: it is missing Q3 and Q4 entirely, and it has two Q1 rows (180 and 20). The pivot has to handle both the gap and the duplicate.

2. The SQL — SUM(CASE WHEN quarter = ...)

SELECT year,
  SUM(CASE WHEN quarter='Q1' THEN amount END) AS q1,
  SUM(CASE WHEN quarter='Q2' THEN amount END) AS q2,
  SUM(CASE WHEN quarter='Q3' THEN amount END) AS q3,
  SUM(CASE WHEN quarter='Q4' THEN amount END) AS q4
FROM sales
GROUP BY year;

Each CASE lets a row's amount through only for its own quarter and yields NULL otherwise; SUM then collapses the four sparse columns down to one row per year.

3. The resulting analysis

yearq1q2q3q4
2025200210NULLNULL
2024100150120200

Two facts to read carefully. 2025's q1 is 200, not 180 — the two Q1 rows (180 + 20) were summed, which is exactly what the aggregate is for. And 2025's q3/q4 are NULL, not 0 — no row matched, so every CASE returned NULL and SUM of all-NULL is NULL.

4. Strategic explanations

The implicit ELSE NULL — and why SUM not MAX

A CASE with no ELSE returns NULL on a miss. That is deliberate: SUM ignores NULLs, so non-matching quarters contribute nothing. SUM is the right aggregate because it also adds up duplicates — the 180 and 20 Q1 rows. A common alternative, MAX(CASE WHEN ...), works only when each year/quarter pair is unique; on this data MAX would report q1 = 180 and silently lose the 20. Choose SUM vs MAX based on whether the cell should aggregate or just relocate a single value.

NULL vs 0 for a missing quarter

2025's empty Q3/Q4 come back NULL. On a finance deck a blank cell often needs to read as 0:

COALESCE(SUM(CASE WHEN quarter='Q3' THEN amount END), 0) AS q3

But pause before doing it — NULL ("no data reported yet") and 0 ("reported, sold nothing") are genuinely different statements. For a quarter that has not happened, NULL is arguably more honest. Decide with the consumer; do not reflexively coalesce.

The native FILTER clause — cleaner Postgres

Postgres has first-class syntax for "aggregate a subset" that reads better than CASE:

SELECT year,
  SUM(amount) FILTER (WHERE quarter='Q1') AS q1,
  SUM(amount) FILTER (WHERE quarter='Q2') AS q2,
  SUM(amount) FILTER (WHERE quarter='Q3') AS q3,
  SUM(amount) FILTER (WHERE quarter='Q4') AS q4
FROM sales GROUP BY year;

Identical results, identical NULL behaviour — just clearer intent. There is also the tablefunc extension's crosstab(), but it is awkward and rarely worth it; conditional aggregation is the portable default every interviewer expects.

The fixed-columns limitation

This pivot hard-codes four quarters. That is fine here — quarters are a closed set — but the moment the pivot key is open-ended (one column per product, per region, per month-as-it-arrives), you cannot enumerate columns in static SQL. A truly dynamic pivot needs the application layer to read the distinct keys and assemble the SQL string, or a return type of jsonb/array. Naming this boundary — "pivot columns must be known at query-write time" — is the senior observation.

Add a row total and guard the grain

A deck usually wants an annual total column too — SUM(amount) AS full_year beside the four quarters, free in the same GROUP BY. And confirm the table's grain: if there is genuinely meant to be one row per year/quarter, the duplicate 2025 Q1 is a data-quality bug, and summing it masks the duplicate instead of surfacing it. SUM gives the arithmetically correct total either way, but a HAVING COUNT(*) > 1 check on (year, quarter) is how you catch the dupe rather than hide it.

Senior signal: explain that the absent ELSE yields NULL so SUM skips non-matching quarters, pick SUM over MAX precisely because it folds duplicate rows, treat NULL-vs-0 for missing quarters as a deliberate choice, prefer the native FILTER clause, and name the fixed-column limit of any static pivot.
