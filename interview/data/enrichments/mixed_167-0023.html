A catalogue team auditing a product database wants to find every description that contains a numeral — sizes, pack counts, model numbers, dimensions. The reference solution chains six LIKE '%digit%' predicates with OR. The interviewer is probing whether you see this as a hand-rolled, incomplete substitute for a real pattern match, and whether you know the portable and Postgres-native ways to do it properly.

Two rows decide the interview. Row 6 has a NULL description, and the solution only checks digits 0–5 — a "7", "8", or "9" in a description would be missed entirely.

2. The SQL — numeral search via chained LIKE

SELECT product_id, description
FROM products
WHERE description LIKE '%0%' OR description LIKE '%1%' OR description LIKE '%2%'
   OR description LIKE '%3%' OR description LIKE '%4%' OR description LIKE '%5%';

Each LIKE '%n%' tests whether the literal character n appears anywhere in the string; OR accepts a row if any one of them matches.

Only two rows return. The NULL description (row 6) is silently excluded — NULL LIKE anything is unknown, never true. The word "numerals" in row 5 does not match because no actual digit character is present. And critically, the query would also miss any description containing only 6–9, because those patterns were never written.

4. Strategic explanations

The pattern is incomplete — six predicates is not "any digit"

The solution checks 0 through 5 only. A description like "Model 789" contains a digit but matches none of the six LIKE clauses, so it is wrongly excluded. To genuinely mean "contains any numeral" you need all ten digits — or, far better, a single pattern. This is the headline bug an interviewer wants you to spot: enumerating characters by hand is error-prone exactly because it is easy to stop short.

The portable fix — LIKE with a character class is not enough

Standard LIKE has no character-class syntax, so you cannot write LIKE '%[0-9]%' portably (that bracket syntax is SQL Server's LIKE, not ANSI). The genuinely portable answer is SIMILAR TO '%[0-9]%' (Postgres) or a regex. In PostgreSQL the idiomatic form is the ~ operator: WHERE description ~ '[0-9]' — one short predicate that matches any digit anywhere, with no enumeration to get wrong.

NULL descriptions vanish — decide if that is intended

Row 6's NULL description is dropped because every LIKE against NULL evaluates to unknown, and unknown is not true. That is usually fine for "find descriptions with digits" — a missing description plainly has none. But if the audit needs to flag products with no description at all, add an explicit OR description IS NULL branch; do not let NULL handling be accidental.

Performance — leading-wildcard LIKE cannot use a B-tree index

Every pattern here starts with %, so a standard B-tree index on description is useless — the engine must scan and substring-test every row. A regex (~) has the same cost. On a large catalogue, the real accelerator is a trigram index: CREATE EXTENSION pg_trgm; CREATE INDEX ON products USING gin (description gin_trgm_ops);, which lets both LIKE '%...%' and ~ use the index.

The "regex" tag is the hint

This question is tagged regex, yet the solution uses no regex at all — it is six string literals. The tag is telling you the expected answer is a single regular expression: description ~ '\d' in Postgres. Recognising that the chained LIKE is the naive form, and that one regex replaces all six predicates and fixes the missing-digits bug, is the move the interviewer is waiting for.

Senior signal: spot that six chained LIKE clauses only cover digits 0–5 and silently miss 6–9, replace the whole chain with one regex (description ~ '[0-9]'), note that NULL descriptions drop out by three-valued logic, and know that leading-wildcard matches need a pg_trgm GIN index to scale.
