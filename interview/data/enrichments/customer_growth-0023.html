A marketing team wants to find sustained engagement runs in an activity log, but with a twist: a row exists for some days where the user did something measurable yet the metric came back as zero (user_code = 0). Those zero rows still count as "the user was in the stream" — they keep the streak alive — but they must not count toward the headline activity total. The interviewer is probing gaps-and-islands plus conditional aggregation that treats presence and value as two different signals.

Two edge cases sit side by side: zero-code days inside an island (09-02, 09-09, 09-06) must not break the run, and user2's lone 09-01 day forms an island with result = 1 that the final filter drops.

2. The SQL — gaps-and-islands with conditional start/end/count

WITH filtered AS (
    -- Step 1: clip to the report window. The 08-31 row is excluded so it
    -- cannot contaminate the per-user ROW_NUMBER sequence below.
    SELECT user_name, user_date, user_code,
        CAST(julianday(user_date) AS INTEGER) AS jd
    FROM user_activity_log
    WHERE user_date BETWEEN '2014-09-01' AND '2014-09-11'
),
indexed AS (
    -- Step 2: classic value - ROW_NUMBER() islands. jd and the row
    -- number both step by 1 inside a contiguous run, so their difference
    -- is constant per island. Zero-code rows are still rows, so they
    -- slot into the surrounding island; only a missing calendar day
    -- opens a new one.
    SELECT user_name, user_date, user_code, jd,
        jd - ROW_NUMBER() OVER (PARTITION BY user_name ORDER BY user_date)
            AS island
    FROM filtered
),
islands AS (
    -- Step 3: per island, MIN/MAX dates among non-zero rows and a COUNT
    -- of non-zero rows.
    SELECT user_name, island,
        MIN(CASE WHEN user_code <> 0 THEN user_date END) AS start_date,
        MAX(CASE WHEN user_code <> 0 THEN user_date END) AS end_date,
        SUM(CASE WHEN user_code <> 0 THEN 1 ELSE 0 END)  AS result
    FROM indexed
    GROUP BY user_name, island
)
SELECT user_name, start_date, end_date, result
FROM islands
WHERE result >= 2
ORDER BY user_name ASC, start_date ASC;

The solution uses SQLite's julianday() to turn dates into a monotonic integer. In PostgreSQL the date itself subtracts directly — user_date - (ROW_NUMBER() OVER (...))::INT yields a date-typed island key with no conversion needed. Pure dialect difference; the islands logic is unchanged.

4. Strategic explanations

Why zero-code rows do not break the island

The gaps-and-islands trick keys on row presence, not row value. ROW_NUMBER() increments for every row in the partition; a zero-code day is still a row, so it advances the counter in lock-step with the calendar date, and the jd - row_number difference stays constant. An island only breaks when a calendar day is genuinely missing — then the date jumps but the row number does not. This is precisely why presence and value must be modelled separately: the islands step uses presence, the conditional aggregates use value.

Conditional aggregation: CASE with no ELSE drops the zero days from start/end

MIN(CASE WHEN user_code <> 0 THEN user_date END) — with no ELSE, zero-code rows yield NULL, and MIN/MAX ignore NULLs. So the boundary dates are the first and last non-zero days, even if the island physically begins or ends on a zero day. The result count uses SUM(CASE ... THEN 1 ELSE 0 END) rather than COUNT, because COUNT(*) would tally zero days too — the explicit 1/0 sum counts only what the business cares about.

The filter-window ordering trap

Clipping the date range happens before ROW_NUMBER() is assigned — deliberately. user1's 08-31 row sits one day before 09-01. If it were left in, the row numbering would start a day early and, worse, 08-31→09-01 is a contiguous pair, so 08-31 would merge into island A and shift every subsequent island key. Filtering first guarantees the numbering reflects only in-window rows. A classic mistake is filtering in the outer query after the window function has already run on the full table.

Dialect notes — date-to-integer conversion

SQLite has no date type, so julianday() manufactures the monotonic integer the islands trick needs. PostgreSQL subtracts dates natively, returning an integer day count — user_date - (ROW_NUMBER() OVER (...))::INT is a date, and grouping on it works fine. MySQL uses TO_DAYS(user_date); BigQuery uses DATE_DIFF(user_date, DATE '1970-01-01', DAY). The pattern is universal; only the "make the date an integer" step swaps out.

What a senior says next

"The zero-code rule is a business decision I would surface explicitly — right now a zero day silently bridges a streak, which inflates engagement runs. If marketing instead wants zero days to count as absence, the fix is one line: filter them out in the filtered CTE so they become gaps. I would confirm which semantics they want before shipping."

Senior signal: separate presence from value — the islands trick keys on row presence so zero-code days bridge a streak, while the conditional SUM/MIN/MAX key on value so zero days are excluded from the count and boundary dates. Filter the date window before ROW_NUMBER() runs, and flag the zero-day-bridges-streak rule as a business choice.
