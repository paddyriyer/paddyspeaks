A compensation equity team wants to flag salary compression: situations where junior employees earn as much as or more than some senior employees within the same department. The interviewer is probing self-join aggregation on a structured level ordering: aggregate min/max per (dept, level), join adjacent level pairs, and filter on the compression condition — a pattern that requires correctly mapping "junior" and "senior" to adjacent level labels.

Check compression (junior_max ≥ senior_min) for each adjacent pair per dept: Eng L1–L2: 90 ≥ 95? No. Eng L2–L3: 110 ≥ 150? No. Sales L1–L2: 75 ≥ 70? Yes. Sales L2–L3: 95 ≥ 100? No. Ops L1–L2: 50 ≥ 80? No. Ops L2–L3: 80 ≥ 120? No. Only Sales L1–L2 has compression.

2. The SQL — aggregate by (dept, level), self-join adjacent pairs, filter

WITH agg AS (
  SELECT dept, level, MIN(salary) AS min_s, MAX(salary) AS max_s
  FROM employee
  GROUP BY dept, level
),
pairs AS (
  SELECT j.dept, j.level AS junior_level, j.max_s AS junior_max,
         s.level AS senior_level, s.min_s AS senior_min
  FROM agg j JOIN agg s
    ON j.dept = s.dept
   AND ((j.level = 'L1' AND s.level = 'L2') OR (j.level = 'L2' AND s.level = 'L3'))
)
SELECT dept, junior_level, junior_max, senior_level, senior_min
FROM pairs
WHERE junior_max >= senior_min
ORDER BY dept, junior_level;

4. Strategic explanations

Why max of junior vs min of senior

Compression is defined as any junior earning as much as or more than any senior. The worst-case combination is the highest junior salary against the lowest senior salary. If junior_max < senior_min, then every junior earns less than every senior — no compression. If junior_max ≥ senior_min, at least one junior earns at or above at least one senior — compression exists. Using max–min is the correct boundary check without enumerating all individual pairs.

Encoding the adjacent-level predicate

The OR condition in the JOIN — (L1,L2) OR (L2,L3) — is the most explicit and readable way to specify exactly which pairs to evaluate. Alternatives include a numeric mapping table (L1→1, L2→2, L3→3) joined on j.level_num + 1 = s.level_num. With only three levels, the explicit OR is preferable for clarity; with many levels a mapping table scales better and avoids a large OR chain.

The aggregation is lightweight

The agg CTE computes only two values per (dept, level) group. This is far cheaper than the alternative of joining the raw employee rows and computing max/min in the WHERE clause. With an index on (dept, level, salary), the GROUP BY can be resolved with an index-only scan.

Edge cases — missing levels within a dept

Ops has L1, L2, L3 but only one employee each. The aggregation handles single-employee groups fine: min = max = that employee's salary. If a department were missing an intermediate level (e.g., no L2 employees), the JOIN would produce no row for the L1–L2 pair in that department, and the L1–L3 pair would not be evaluated because the predicate only covers adjacent levels. This is correct behavior — the question defines compression only between adjacent levels.

Dialect notes

The aggregate + self-join pattern is ANSI SQL and portable to all major engines. No window functions are used, so MySQL 5.7 handles it correctly. The OR predicate in the JOIN ON clause is standard. If you use a level-number mapping table instead of the OR, it is even more portable — just a numeric equality join. The output column names (junior_level, junior_max, senior_level, senior_min) match the question specification exactly.

Senior signal: compare MAX(junior_salary) ≥ MIN(senior_salary) rather than joining individual rows, because that boundary check is both sufficient and necessary for compression detection; encode the adjacent-level constraint explicitly in the JOIN predicate; and mention the numeric level-mapping alternative for systems with more than three levels.
