Bloomberg's terminal team wants the top three revenue-generating products in every category for a sales dashboard. "Top N per group" is one of the most-asked SQL patterns precisely because the naive approaches — GROUP BY, LIMIT — cannot do it. The interviewer wants to see you reach for a windowed row number, and wants to hear which ranking function handles ties the way the product needs.

1. The source data — sales

Books has four products — one must be cut. Toys has only three, and two of them are tied at 900. That tie is the interview: with ROW_NUMBER the engine still hands the tied rows distinct numbers, picking a winner arbitrarily.

2. The SQL — ROW_NUMBER() <= 3

SELECT category, product_id, revenue FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY category ORDER BY revenue DESC) rn FROM sales
) t WHERE rn <= 3;

PARTITION BY category restarts the numbering for each category; ORDER BY revenue DESC numbers within it from highest. The outer query keeps rows numbered 1, 2, 3. The window function must live in a subquery because you cannot reference its alias rn in a WHERE at the same level.

But notice: products 5 and 6 are both revenue 900 — ROW_NUMBER assigned them 1 and 2 in some order the planner chose. If Toys had a fourth product, the cut between the two tied 900s would be arbitrary.

4. Strategic explanations

Why GROUP BY and LIMIT both fail here

GROUP BY category collapses each category to one row — it can give you the single max revenue, never the top three rows with their product_ids. A bare ORDER BY revenue DESC LIMIT 3 returns three rows total, not three per category. "Top N per group" structurally needs a per-group counter, and that counter is a window function. Saying why the simple tools fail is half the answer.

The tie trap — ROW_NUMBER vs RANK vs DENSE_RANK

The reference uses ROW_NUMBER, which never produces a tie: equal-revenue rows still get consecutive integers, so it always returns exactly 3 rows per group. That is right when the product wants a hard cap. But it makes the cutoff non-deterministic when rows tie on the boundary. The three functions differ precisely on ties:

revenueROW_NUMBERRANKDENSE_RANK
900111

100332

If a tie at the boundary should mean "keep everyone tied at 3rd place", use RANK() <= 3 (may return more than 3 rows). If "top 3 distinct revenue levels", use DENSE_RANK() <= 3. Confirm which the dashboard wants.

Make the tiebreak deterministic

Even with ROW_NUMBER, the cut should be reproducible. Add a unique column to the window's ORDER BY so equal-revenue rows always order the same way:

ROW_NUMBER() OVER (PARTITION BY category ORDER BY revenue DESC, product_id ASC)

Now if Toys had a fourth product, the lower product_id among the tied 900s wins consistently — the same answer on every run.

The dialect note — QUALIFY

Snowflake, BigQuery and DuckDB let you skip the subquery with QUALIFY:

SELECT category, product_id, revenue,
       ROW_NUMBER() OVER (PARTITION BY category ORDER BY revenue DESC) rn
FROM sales
QUALIFY rn <= 3;

QUALIFY is to window functions what HAVING is to aggregates. Postgres and MySQL do not support it — there the subquery (or a CTE) is mandatory. Mention this; it shows you know the standard and the vendor extensions.

Edge cases: small groups and NULL revenue

A category with fewer than 3 products simply returns all of them — rn <= 3 degrades gracefully, no special case needed. NULL revenue is the subtle one: under ORDER BY revenue DESC, Postgres sorts NULLs first, so a NULL-revenue product would be ranked #1 in its category. Add NULLS LAST if a missing revenue should sink, not float.

Senior signal: explain why GROUP BY/LIMIT cannot express top-N-per-group, pick the ranking function deliberately (ROW_NUMBER for a hard cap, RANK/DENSE_RANK when boundary ties must all survive), make the tiebreak deterministic with a unique sort key, and know that QUALIFY replaces the subquery only on engines that support it.
