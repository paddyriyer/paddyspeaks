A trading-analytics pipeline needs each trade marked with the prevailing quote — the latest quote for the same symbol with a timestamp at or before the trade. This is an "as-of join", the bread and butter of market-data work. The interviewer explicitly asks for a LATERAL subquery and is checking whether you can produce a per-row "top-1 by time" lookup without exploding rows in a self-join.

Two edge cases must be handled. Trade 1 (AAPL at 08:00) and trade 5 (TSLA at 09:00) happen before any qualifying quote — last_price must be NULL, and the trade must still appear in the output. Symbol matching matters too: a TSLA trade must never pick up an AAPL quote.

2. The SQL — LEFT JOIN LATERAL with ORDER BY ... LIMIT 1

SELECT t.trade_id, t.symbol, t.traded_at, q.price AS last_price
FROM trade t
LEFT JOIN LATERAL (
  SELECT price
  FROM quote
  WHERE quote.symbol = t.symbol AND quote.quoted_at <= t.traded_at
  ORDER BY quote.quoted_at DESC
  LIMIT 1
) q ON TRUE
ORDER BY t.trade_id;

The LATERAL subquery runs once per trade and can reference t.symbol and t.traded_at. It filters quotes to the same symbol at or before the trade time, sorts them newest-first, and LIMIT 1 keeps the single most recent — the prevailing quote.

4. Strategic explanations

Why LATERAL — the subquery depends on the outer row

A normal subquery in FROM is evaluated once, independently. LATERAL lifts that restriction: the subquery is re-evaluated per outer row and may reference its columns — here t.symbol and t.traded_at. That is exactly what an as-of join needs: a fresh "find the latest quote for this trade" computation for every trade. It is a correlated subquery promoted to a full table expression, so it can return columns and even multiple rows.

ORDER BY ... LIMIT 1 — clean top-1-per-group

Inside the LATERAL, ORDER BY quoted_at DESC LIMIT 1 selects the single most recent qualifying quote. This is the most readable "greatest-per-group" idiom and it pairs naturally with LATERAL. The alternative — ROW_NUMBER() OVER (PARTITION BY symbol ORDER BY quoted_at DESC) filtered to = 1 — works but ranks every quote for every trade window; the LATERAL + LIMIT 1 stops as soon as it has one row.

LEFT JOIN LATERAL ... ON TRUE is what preserves no-quote trades

Two pieces matter. LEFT JOIN (not inner) keeps trades 1 and 5 even though the LATERAL block yields nothing — with a plain inner join those rows would silently disappear, badly under-reporting trades. And ON TRUE is the required join predicate: a LATERAL subquery already encodes its correlation in its own WHERE, so there is no extra condition to put in the ON clause — ON TRUE means "join on the lateral correlation alone".

The row-explosion trap the question warns against

The naive answer is trade JOIN quote ON quote.symbol = t.symbol AND quote.quoted_at <= t.traded_at, then MAX or a window to pick the latest. That join first produces every qualifying quote for each trade — trade 4 would briefly match all three AAPL... no, the same symbol, but a trade late in the day matches many quotes, and the intermediate result balloons before you collapse it. LATERAL + LIMIT 1 never materialises that explosion: it fetches one quote per trade directly.

The boundary: <= vs <

The spec says "quoted_at ≤ traded_at" — a quote stamped at the exact trade time counts. <= is therefore correct; < would exclude a simultaneous quote. With <= and ties at the same timestamp, LIMIT 1 picks one arbitrarily — if quotes could collide on quoted_at, add a tiebreaker like ORDER BY quoted_at DESC, quote_id DESC for determinism. Naming that tie risk unprompted is the senior detail.

Senior signal: recognise "latest quote at or before each trade" as an as-of join, implement it with LEFT JOIN LATERAL (... ORDER BY quoted_at DESC LIMIT 1) ON TRUE — LEFT keeps trades with no prior quote, LATERAL correlates per row, LIMIT 1 avoids the self-join row explosion — and add a tiebreaker if timestamps can collide.
