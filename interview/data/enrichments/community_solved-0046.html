A Community member has a wide marks table — one row per student with separate English, Maths and Science columns — and needs it reshaped tall: one row per student-subject pair, a single Marks column. This is unpivot, and the interviewer wants to see whether you reach for a portable construct and whether you understand what happens to a NULL mark.

1. The source data — student_info

StudentNameEnglishMathsScience
Asha789285
Ben647088
Chris908176

Three students, three subject columns — nine logical scores. The target shape turns each of those nine cells into its own row. The edge to keep in mind: if a cell were NULL (a student who skipped a subject), an unpivot still emits a row, with a NULL Marks — unless you choose to drop it.

2. The SQL — UNION ALL unpivot

SELECT StudentName, 'English' AS Subject, English AS Marks
FROM student_info
UNION ALL
SELECT StudentName, 'Maths'   AS Subject, Maths   AS Marks
FROM student_info
UNION ALL
SELECT StudentName, 'Science' AS Subject, Science AS Marks
FROM student_info
ORDER BY StudentName, Subject;

One SELECT per source column, each hard-coding the subject name as a string literal and aliasing that column to a common Marks. UNION ALL stacks the three result sets; the final ORDER BY groups each student's rows together.

Three input rows become nine — one per student-subject pair. The data is unchanged; only the shape moved from wide to tall. This long form is what lets you GROUP BY Subject for a class average, or GROUP BY StudentName for a per-student total, neither of which is natural against the wide table.

4. Strategic explanations

Why UNION ALL, not UNION

Plain UNION de-duplicates the combined result. If two students scored the same mark in the same subject — entirely possible — UNION would silently merge those rows and a student would lose a score. UNION ALL keeps every row unconditionally. It is also cheaper: no sort-and-dedup pass. For an unpivot, UNION ALL is correctness and performance.

The dialect map — UNPIVOT vs the portable form

SQL Server and Oracle have a dedicated operator:

SELECT StudentName, Subject, Marks
FROM student_info
UNPIVOT (Marks FOR Subject IN (English, Maths, Science)) u;

It is terser, but it does not run in SQLite, MySQL, or older Postgres. The UNION ALL form runs everywhere. Postgres also offers LATERAL (VALUES ('English', English), ...) — a single table scan instead of three. Naming the trade-off (portability vs one scan) is the senior move.

The NULL divergence — UNPIVOT drops, UNION ALL keeps

A genuine behavioural difference, not a style point. If a student has a NULL Science mark, the UNION ALL form emits (student, 'Science', NULL) — a row with a NULL Marks. The SQL Server UNPIVOT operator excludes NULL-valued rows by default. So the two approaches return different row counts on NULL data. Decide which you want: keep the NULL row and filter later, or add WHERE Marks IS NOT NULL to the UNION ALL form to match UNPIVOT's behaviour.

The single-scan optimisation

UNION ALL of three SELECTs scans student_info three times. On a 3-row demo that is nothing; on a million-row table it triples I/O. The LATERAL VALUES / cross-join-to-a-subject-list rewrite reads the base table once and expands each row in memory:

SELECT s.StudentName, v.Subject, v.Marks
FROM student_info s
CROSS JOIN LATERAL (VALUES
  ('English', s.English),
  ('Maths',   s.Maths),
  ('Science', s.Science)
) AS v(Subject, Marks);
Type consistency across the unioned columns

Every branch's Marks column must be a compatible type — here all three are integers, fine. But if one subject column were stored as TEXT (a grade like 'A') and another as INT, the UNION ALL would fail or implicitly coerce. Unpivot only makes sense when the columns being folded are genuinely the same kind of measure; mixing a numeric score with a letter grade is a modelling smell.

Senior signal: reach for the portable UNION ALL unpivot (or LATERAL VALUES for a single scan) over the non-portable UNPIVOT operator, insist on UNION ALL so equal marks are never merged, and call out that UNPIVOT drops NULL rows while UNION ALL keeps them — a real difference the consumer must decide on.
