A Community question on the HackerRank "Top Competitors" schema: find the students who earned a full score on more than one challenge. A challenge's full score is the score attached to its difficulty level, so a submission is "full" only when submission.score equals the difficulty's score. The interview tests a four-table join through a bridge dimension, then counting only the maximal submissions.

1. The source data

Four tables. students (id→name), difficulty (level→full score: 1→20, 2→40, 3→60, 4→80), challenge (each challenge has a difficulty_level), and submissions:

Alice (1) has 2 full-score subs, Bob (2) has 3, Carol (3) has only 1, Eve (5) scored 10 on a difficulty-2 (full = 40) challenge — not full. The "more than one" rule must keep Alice, Bob, Dan; drop Carol (exactly one) and Eve (none).

2. The SQL — 4-table join, window count, filter > 1

with t1 as
(select h.name, s.submission_id, s.student_id, s.challenge_id, s.score,
    count(1) over(partition by s.student_id) as full_score
from submissions s
inner join challenge c on s.challenge_id = c.challenge_id
inner join difficulty d on c.difficulty_level = d.difficulty_level
inner join students h on s.student_id = h.student_id
where d.score = s.score)

select t1.student_id, t1.name
from t1
where full_score <> 1
group by t1.student_id, t1.name, t1.full_score
order by t1.full_score desc, t1.student_id asc;

The WHERE d.score = s.score keeps only full-score submissions; count(1) OVER (PARTITION BY student_id) — computed after that filter — counts each student's full-score subs; full_score <> 1 keeps students with more than one.

4. Strategic explanations

The join chain — why difficulty is reached through challenge

A submission knows its challenge_id but not its difficulty; only challenge carries difficulty_level, and only difficulty carries the full score. So the path is submissions→challenge (to learn the level)→difficulty (to learn the full score), plus submissions→students for the name. challenge is a bridge table — you cannot jump submissions straight to difficulty. All four joins are INNER, which is deliberate: a submission with no matching challenge or difficulty is unscorable and rightly excluded.

The filter-before-window subtlety

WHERE executes before window functions, so count(1) OVER (PARTITION BY student_id) counts only the rows that survived d.score = s.score — i.e. full-score submissions per student. That is exactly the intended metric. If the count instead needed to include non-full subs, the filter would have to move out of WHERE (into a CASE inside the count, or a later stage). Knowing this evaluation order is the crux of the query.

Why full_score <> 1 and not > 1 — and the zero-row subtlety

After the WHERE, t1 contains only full-score rows, so full_score is always ≥ 1 — a student with zero full scores has no rows at all. Within that universe, <> 1 and > 1 are equivalent. > 1 reads the intent ("more than one") more honestly; <> 1 works only because 0 is impossible here. Eve's exclusion is not done by this filter — it happens earlier, in the join's WHERE.

The GROUP BY + window redundancy

count(1) OVER (…) repeats the same full_score on every one of a student's rows, so the outer GROUP BY student_id, name, full_score exists only to collapse those duplicates to one row per student. A cleaner formulation skips the window entirely and uses a plain grouped aggregate with HAVING:

SELECT s.student_id, h.name
FROM submissions s
JOIN challenge c   ON s.challenge_id = c.challenge_id
JOIN difficulty d  ON c.difficulty_level = d.difficulty_level
JOIN students h    ON s.student_id = h.student_id
WHERE d.score = s.score
GROUP BY s.student_id, h.name
HAVING COUNT(*) > 1
ORDER BY COUNT(*) DESC, s.student_id;

Same result, no window, no dedup step — HAVING is the natural tool for "groups with more than one row".

Edge cases: duplicate submissions and the counting unit

If a student submitted the same challenge twice at full score, both rows count — full_score would be 2 from one challenge, arguably wrong if "more than one" means more than one distinct challenge. Then count DISTINCT challenge_id. The HackerRank original also sorts ties by hacker_id ascending, which the ORDER BY here mirrors.

Senior signal: route the join through the challenge bridge to reach difficulty, recognise that WHERE runs before the window so the count tallies only full-score subs, and prefer a plain GROUP BY … HAVING COUNT(*) > 1 over a window-plus-dedup — while deciding whether the unit is submissions or distinct challenges.
