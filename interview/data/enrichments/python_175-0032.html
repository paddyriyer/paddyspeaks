Netflix's content-tagging pipeline clusters localized title strings that are permutations of the same token set — useful for catching slug collisions and near-duplicate metadata. "Group Anagrams" tests one core idea: can you design a canonical key that is identical for every member of a group, then bucket by it? The whole question lives or dies on the key function.

1. The input data

strs = ["eat", "tea", "tan", "ate", "nat", "bat"]

Three anagram families hide here: {eat, tea, ate}, {tan, nat}, and the singleton {bat}.

2. The approach

For each string, ''.join(sorted(s)) produces a canonical signature — sorting the characters means every permutation maps to the same string ("eat", "tea", "ate" all become "aet"). That signature is the dictionary key.

defaultdict(list) removes the "is this key present yet?" branch: the first time a signature is seen, an empty list is auto-created and the original string is appended; subsequent matches append to the existing list. After one pass, g.values() yields the groups. Insertion order is preserved (Python 3.7+ dicts), so groups appear in first-seen order.

3. The output

stringsorted keybucket
eataet[eat]
teaaet[eat, tea]
tanant[tan]
ateaet[eat, tea, ate]
natant[tan, nat]
batabt[bat]
 
groupAnagrams(strs)  ->  [['eat', 'tea', 'ate'], ['tan', 'nat'], ['bat']]

4. Strategic explanations

Complexity

O(n · k log k) time for n strings of max length k — the sorted() call dominates each iteration. O(n · k) space to store every string in the buckets. The sort is the bottleneck worth naming.

The idiomatic alternative — a count tuple key

Replace the sort with a 26-element character count, eliminating the log k factor:

from collections import defaultdict, Counter
g = defaultdict(list)
for s in strs:
    key = tuple(sorted(Counter(s).items()))   # or a fixed length-26 tuple
    g[key].append(s)

A fixed tuple(count[c] for c in range(26)) key is O(k) per string — strictly better than sorting for long strings, and the senior answer when the alphabet is small and known.

The edge case — empty strings and case sensitivity

Empty strings all share the key "" and correctly cluster together. But "Eat" and "eat" sort to different keys ("Eat" → "Eat"), so mixed case splits a group — if the spec is case-insensitive you must s.lower() before keying. Unicode adds the same hazard: combining accents need normalization (unicodedata.normalize) first.

The follow-up — "return only groups of size ≥ 2" or stream the input

Filtering with [v for v in g.values() if len(v) > 1] is trivial. The harder follow-up is a streaming source too large for memory — then you key into an on-disk store or shard by hash(key) % N across workers, which the interviewer uses to probe distributed thinking.

The trap — hashing the string itself

Using hash(s) or the raw string as the key never groups anything — anagrams have different raw strings. And a key built from set(s) wrongly merges "aab" with "abb" (same letter set, different multiset). The key must encode multiplicity, which is exactly what sorting or counting gives you.

Senior signal: articulate "I need a canonical key invariant under permutation," then proactively offer the O(k) count-tuple key over the O(k log k) sort and flag the case-sensitivity assumption before the interviewer has to ask.
