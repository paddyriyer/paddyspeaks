A finance team wants a year-over-year revenue view: every monthly row annotated with the same month from the prior year and a YoY growth percentage. The interviewer is probing two things at once — can you join a table to itself on a shifted date key, and do you handle the rows that have no prior-year partner (the first year of data, or a region that did not exist last year) without crashing on a divide-by-zero or fabricating a misleading 0%.

The edge cases that decide the interview: the entire 2024 data has no prior year, so those rows must return NULL; and EU has a 2025-07 row but no matching 2024-07 — a gap, not a zero. A correct query must report NULL for both, never 0%.

2. The SQL — self-join on a one-year-shifted date

SELECT m.region, m.month_start, m.revenue,
       p.revenue AS prev_year_revenue,
       CASE WHEN p.revenue IS NULL OR p.revenue = 0 THEN NULL
            ELSE ROUND(((m.revenue / p.revenue - 1) * 100)::numeric, 2)
       END AS yoy_pct
FROM monthly m
LEFT JOIN monthly p
  ON p.region = m.region
 AND p.month_start = (m.month_start - INTERVAL '1 year')::date
ORDER BY m.region, m.month_start;

The table is joined to itself: alias m is the current month, alias p is the prior-year month. The join key shifts m.month_start back exactly one year and looks for a row at that date. LEFT JOIN keeps m's row even when no p exists; the CASE then guards the division.

4. Strategic explanations

Why a self-join, and why on a shifted key

"Same month last year" is a relationship between two rows of the same table, so the table appears twice under two aliases. The join condition encodes the relationship: p.month_start = m.month_start - INTERVAL '1 year'. Because both columns are first-of-month DATEs, the shifted value lands exactly on a stored key when a partner exists — no range logic, no BETWEEN, just equality. The ::date cast trims the interval arithmetic result back to a clean date.

The LEFT JOIN is load-bearing

An inner join would silently delete every row without a prior-year partner — all of 2024 and the EU gap month would disappear, and the report would understate its own row count. LEFT JOIN preserves every current-month row and supplies NULL for p.* when no match exists. The question demands those rows be present with NULL annotations, so the join type is not a stylistic choice — it is the requirement.

The divide-by-zero / divide-by-NULL guard

The growth formula divides by p.revenue. Two things can go wrong: p.revenue is NULL (no partner) — the arithmetic would yield NULL anyway, but explicitly is clearer — or p.revenue is 0, which raises division by zero in Postgres and aborts the whole query. The CASE WHEN p.revenue IS NULL OR p.revenue = 0 THEN NULL catches both before the division is ever attempted. Forgetting the = 0 branch is the classic bug: it works on demo data and explodes the first month a region had zero revenue.

The window-function alternative — and why it is risky here

A tempting one-liner is LAG(revenue, 12) OVER (PARTITION BY region ORDER BY month_start). That only works if every region has a dense, gap-free monthly series — "12 rows back" must literally be 12 months back. EU violates this: its 2024 series is a single June row, so LAG(...,12) would line up the wrong months or nothing at all. The shifted-key self-join matches on the actual date, so gaps and uneven histories are handled correctly. Mention this trade-off: LAG is cleaner only when the data is guaranteed contiguous.

Dialect notes

INTERVAL '1 year' date arithmetic is portable to Oracle and works in MySQL as month_start - INTERVAL 1 YEAR. SQL Server uses DATEADD(YEAR, -1, month_start). The divide-by-zero behaviour differs: Postgres and Oracle raise an error, while SQL Server also errors by default — so the CASE guard (or NULLIF(p.revenue, 0)) is good practice everywhere. NULLIF(p.revenue,0) is a terser alternative to the = 0 branch and is fully portable.

Senior signal: implement period-over-period as a self-join on a date key shifted by exactly one year, use LEFT JOIN so unmatched rows survive with NULL, guard the growth division against both NULL and zero denominators, and explain that LAG(...,12) is only safe when the monthly series is guaranteed gap-free.
