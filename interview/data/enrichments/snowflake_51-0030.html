Vercel’s payment team ingests a raw transactions feed that contains two classes of noise: (1) the same logical transaction appears multiple times because of retries or replication lag — distinguished by the same txn_id but different timestamps, and (2) near-duplicate transactions where a different txn_id was issued for an amount that hit the same account at the same instant (system quirk). The query must also drop any row in 'reversed' or 'failed' status. The solution chains two CTE passes — the first de-dupes by txn_id, the second de-dupes structural duplicates — a disciplined two-stage deduplication pattern.

T01 appears twice (same txn_id, 5-second gap — retry). T02 is reversed — must be excluded. T03 and T04/T05 share (account, amount, timestamp) — structural duplicates. T06 is failed — excluded.

2. The SQL — two-stage CTE deduplication

WITH ranked AS (
  SELECT *,
         ROW_NUMBER() OVER (PARTITION BY txn_id ORDER BY txn_ts DESC) AS rn
  FROM transactions
  WHERE status NOT IN ('reversed', 'failed')
),
deduped AS (
  SELECT txn_id, account_id, amount, txn_ts, status,
         ROW_NUMBER() OVER (PARTITION BY account_id, amount, txn_ts ORDER BY txn_id) AS rn2
  FROM ranked
  WHERE rn = 1
)
SELECT txn_id, account_id, amount, txn_ts, status
FROM deduped
WHERE rn2 = 1;

3. The resulting analysis

Stage 1 (ranked): status filter removes T02 (reversed) and T06 (failed). For T01 which appears twice, ROW_NUMBER() PARTITION BY txn_id ORDER BY txn_ts DESC gives the latest occurrence rn=1; the earlier row rn=2. After WHERE rn = 1 we keep: T01 (10:00:05), T03, T04, T05.

Stage 2 (deduped): T04 and T05 share (account_id=A2, amount=350, txn_ts=11:00:00). PARTITION BY account_id, amount, txn_ts ORDER BY txn_id gives T04 rn2=1, T05 rn2=2. After WHERE rn2 = 1 we keep T04.

4. Strategic explanations

Why two deduplication passes instead of one

The two noise types have different signatures. Same-txn_id duplicates are definitionally the same transaction regardless of timestamp — retain the latest. Structural duplicates have different txn_ids but identical (account, amount, timestamp) — retain one arbitrarily. Combining these into a single partition key would mis-identify legitimate transactions sharing an account or amount but different timestamps. Separating the passes makes the logic inspectable and testable independently.

Stage 1 ordering: why DESC

Within a txn_id group, ORDER BY txn_ts DESC puts the most recent occurrence at rn=1. This is correct for retry scenarios: the most recent attempt is the authoritative state. If the data model guaranteed that the first insertion is always correct, you would use ASC instead. The direction is a business-logic choice, not a SQL technicality — articulate it.

Stage 2 ordering: why ORDER BY txn_id

When (account, amount, timestamp) collide, there is no principled way to say which txn_id is “more correct” — they are both system-generated surrogates. Ordering by txn_id is deterministic (picks the lexicographically smallest ID) and stable across re-runs. The alternative — ordering by txn_ts again — would not break ties because the timestamps are identical by definition of the partition.

Snowflake QUALIFY shortcut

In Snowflake both CTE steps that filter on a window function (WHERE rn = 1, WHERE rn2 = 1) can be collapsed with QUALIFY. The equivalent one-pass version is:

SELECT txn_id, account_id, amount, txn_ts, status
FROM transactions
WHERE status NOT IN ('reversed', 'failed')
QUALIFY ROW_NUMBER() OVER (PARTITION BY txn_id ORDER BY txn_ts DESC) = 1
  -- then wrap or use a second QUALIFY pass for structural dedup

QUALIFY cannot chain two independent window conditions in a single scan, so the two-CTE structure remains cleaner for this specific two-stage problem. But for the first stage alone, QUALIFY eliminates the outer wrapper query entirely.

Dialect notes

ROW_NUMBER() OVER (...) is ANSI SQL and works in all major engines. The CTE structure is portable. NOT IN ('reversed', 'failed') can alternatively be written as status <> 'reversed' AND status <> 'failed' for clarity. In BigQuery, QUALIFY is also supported. MySQL prior to 8.0 has no window functions and requires correlated subqueries — a significantly more complex approach. Snowflake’s QUALIFY is the cleanest single-engine shortcut.

Senior signal: name the two distinct duplicate types (same-ID retry vs structural-duplicate), justify the two-pass architecture as enforcing separate deduplication semantics that cannot be collapsed safely, explain why ORDER BY txn_ts DESC (keep latest) is a business-logic decision not just a SQL detail, and offer Snowflake’s QUALIFY as the dialect-specific shortcut for the first pass.
