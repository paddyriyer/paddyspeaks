An automotive lead-gen platform reports daily volume by manufacturer, and the dashboard must not double-count: the same lead can be logged twice by an integration retry, and partners can submit multiple leads. The interviewer wants to see you reach for COUNT(DISTINCT ...) deliberately — and explain why a plain COUNT(*) would inflate the numbers.

1. The source data — daily_leads

date_idmake_namelead_idpartner_id(narrative)
2025-01-01Toyota110lead 1
2025-01-01Toyota110exact duplicate row — integration retry
2025-01-01Toyota211lead 2
2025-01-01Honda520single-lead group
2025-01-02Toyota112lead 1 reappears next day, new partner

The first two rows are an exact duplicate — same lead, same partner, same day. A naive count would say Toyota had 3 leads on Jan 1; the truth is 2. Honda is a clean single-member group, and lead 1 legitimately recurs on Jan 2 under a different partner.

2. The SQL — COUNT(DISTINCT) per group

SELECT date_id, make_name,
       COUNT(DISTINCT lead_id)::int    AS unique_leads,
       COUNT(DISTINCT partner_id)::int AS unique_partners
FROM daily_leads
GROUP BY date_id, make_name
ORDER BY date_id, make_name;

GROUP BY date_id, make_name forms one bucket per (day, make). Inside each bucket COUNT(DISTINCT lead_id) tallies unique lead identifiers — the retry duplicate collapses — and COUNT(DISTINCT partner_id) does the same for partners.

3. The resulting analysis

date_idmake_nameunique_leadsunique_partnerssignal
2025-01-01Honda11single row, single lead
2025-01-01Toyota223 rows, but the retry duplicate collapses to 2
2025-01-02Toyota11separate day → separate bucket

Toyota on Jan 1 shows 2, not 3 — DISTINCT folded the identical (1, 10) rows. Lead 1's reappearance on Jan 2 is a different group entirely, so it is counted independently.

4. Strategic explanations

COUNT(*) vs COUNT(DISTINCT col) — rows vs entities

COUNT(*) answers "how many rows", which here means "how many lead submissions" — retries included. COUNT(DISTINCT lead_id) answers "how many lead identifiers". The question asks for distinct leads, so the duplicate submission must not inflate the figure. Choosing between the two is choosing between an event count and an entity count — name which one the metric needs.

Two independent DISTINCTs, not one combined

COUNT(DISTINCT lead_id) and COUNT(DISTINCT partner_id) are evaluated separately, each de-duplicating its own column. They are not the same as COUNT(DISTINCT (lead_id, partner_id)), which would count unique pairs. With one lead spread across several partners the three numbers diverge — be precise about which cardinality the business wants.

DISTINCT and NULL

COUNT(DISTINCT col) ignores NULLs entirely — a NULL partner_id contributes nothing and is not counted as a distinct value. If "unknown partner" should register as one bucket, you would need COUNT(DISTINCT COALESCE(partner_id, -1)) or count a sentinel. The fixture has no NULLs, but stating this rule shows you have thought past the happy path.

Performance — DISTINCT aggregation is not free

COUNT(DISTINCT) must materialise and de-duplicate the values per group, typically via a sort or hash — heavier than a plain COUNT(*), and two of them in one query means two such operations. On large tables this is usually fine, but at extreme scale teams pre-aggregate or use approximate distinct counts (HyperLogLog) — know the trade-off exists.

Dialect — approximate and multi-distinct support

Standard COUNT(DISTINCT) works everywhere. MySQL uniquely allows COUNT(DISTINCT a, b) for tuple counts; other engines need COUNT(DISTINCT (a, b)) or a row constructor. For huge data, BigQuery offers APPROX_COUNT_DISTINCT, Snowflake APPROX_COUNT_DISTINCT, and PostgreSQL the postgresql-hll extension — mention these when the interviewer pushes on scale.

Senior signal: use COUNT(DISTINCT col) — not COUNT(*) — when integration retries can duplicate rows, keep separate DISTINCTs for separate columns rather than counting tuples, remember that DISTINCT drops NULLs, and acknowledge that distinct aggregation costs a sort/hash that approximate counters can relieve at scale.
