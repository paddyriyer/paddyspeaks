Apple uses "Copy List with Random Pointer" to test deep-copy reasoning under aliasing. Each node has the usual next plus a random pointer that can target any node in the list (or None). The challenge: when you clone a node, its random may point to a node you haven't created yet. It's really a graph-cloning question wearing a linked-list costume, and it probes whether you can decouple "allocate every node" from "wire up the pointers".

1. The input data

Three nodes. random can point forward, backward, or be None:

node A(val=7)   next->B   random->None
node B(val=13)  next->C   random->A
node C(val=11)  next->None random->B

head = A

2. The approach

The classic two-pass hash-map solution. The dict m maps each original node to its freshly-allocated clone.

Pass 1 walks the list and does nothing but m[cur] = Node(cur.val) — every clone now exists, but all its pointers are still None. This is the trick that defeats the "random points to an uncreated node" problem: after pass 1, nothing is uncreated.

Pass 2 walks again and wires pointers purely by lookup: m[cur].next = m.get(cur.next) and m[cur].random = m.get(cur.random). Using m.get(...) rather than m[...] is deliberate — when cur.next or cur.random is None, get returns None instead of raising KeyError. The guard if not head: return None handles the empty list up front.

3. The output

A wholly independent list — same values and same pointer topology, but every node is a new object:

clonevalnext →random →
A'7B'None
B'13C'A'
C'11NoneB'

Returned head is m[head] = A'. Mutating the original list afterwards cannot touch the copy.

4. Strategic explanations

Complexity

Time O(n) — exactly two linear passes. Space O(n) for the dictionary holding n original→clone entries (plus the O(n) output, which is unavoidable for a copy).

The idiomatic alternative — collapse to one pass

A defaultdict-style trick lets you allocate lazily and finish in a single walk:

def copyRandomList(head):
    clones = {None: None}
    def get(node):
        if node not in clones:
            clones[node] = Node(node.val)
        return clones[node]
    cur = head
    while cur:
        c = get(cur)
        c.next   = get(cur.next)
        c.random = get(cur.random)
        cur = cur.next
    return clones[head]

Seeding {None: None} kills the null-check entirely. Same O(n)/O(n), one pass — a clean senior refinement.

The edge case

Three are worth naming: the empty list (head is None → early return); a self-referential random (node.random is node — works, because the clone already exists in m by pass 2); and multiple nodes' random pointing at the same target — the map guarantees they all resolve to the one shared clone, preserving aliasing rather than duplicating it.

The follow-up

"Do it in O(1) extra space." The interleaving trick: insert each clone immediately after its original (A → A' → B → B' → ...), so clone.random = original.random.next, then unweave the two lists. O(n) time, O(1) auxiliary — the over-the-bar answer.

The trap

Trying to set random in a single naive pass without the map: when you reach node B and want B'.random = clone_of(A), that clone may not exist yet if A is later in the list. You'd either crash or create a second copy of A, breaking aliasing. The two-phase "allocate all, then wire all" split is what makes the random pointer tractable.

Senior signal: separates allocation from wiring so a forward random pointer is never a problem, uses m.get (or a {None: None} seed) to make null handling disappear, preserves aliasing when several pointers share a target, and can produce the O(1)-space interleaving solution when pushed.
