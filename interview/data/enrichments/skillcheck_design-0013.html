Designing a simplified Iceberg-style metastore reveals whether a candidate understands how table formats achieve snapshot isolation and time travel without locking: every write creates a new snapshot that references a set of manifest entries, which in turn reference immutable data files in object storage. The interviewer is probing whether you can separate the three layers (snapshots, manifests, data files), model the parent-snapshot chain for time travel, represent the file-level column statistics that enable predicate pushdown, and enforce referential integrity within the metastore itself.

1. The requirements — what the design must capture

  • Snapshots: a table-level version history. Each snapshot is immutable once committed; reading at a past snapshot_id gives an isolated, consistent view of the table.
  • Manifest entries: the bridge between a snapshot and its constituent data files. Each entry records whether a file was ADDED, is EXISTING (carried over from the parent snapshot), or was DELETED (logically removed) in this snapshot.
  • Data files: the physical Parquet files in object storage. Stored with file-level statistics (row count, size, per-column min/max/null counts) to enable predicate pushdown without reading the files.
  • Time travel: supported by the parent_snapshot_id chain — walking back gives the set of files visible at any historical moment.
  • Schema and partition evolution: tracked via schema_id and partition_spec_id on the snapshot (references to separate versioning tables, implied).

2. The DDL — the CREATE TABLE

CREATE TABLE table_snapshot (
  table_id          BIGINT       NOT NULL,
  snapshot_id       BIGINT       NOT NULL,                  -- monotonic per table
  parent_snapshot_id BIGINT,                                -- nullable for the first snapshot
  committed_at      TIMESTAMPTZ  NOT NULL DEFAULT now(),
  committed_by      VARCHAR(64)  NOT NULL,
  summary           JSONB,                                  -- e.g. files-added, rows-added
  schema_id         INT          NOT NULL,                  -- references a schema-version table
  partition_spec_id INT          NOT NULL,                  -- references a partition-spec-version table
  PRIMARY KEY (table_id, snapshot_id),
  FOREIGN KEY (table_id, parent_snapshot_id) REFERENCES table_snapshot (table_id, snapshot_id)
);

CREATE TABLE table_data_file (
  data_file_id      BIGSERIAL    PRIMARY KEY,
  table_id          BIGINT       NOT NULL,
  file_path         TEXT         NOT NULL,                  -- s3://... or hdfs://...
  file_size_bytes   BIGINT       NOT NULL,
  row_count         BIGINT       NOT NULL,
  partition_values  JSONB        NOT NULL,                  -- e.g. {"event_date":"2025-05-12"}
  column_stats      JSONB        NOT NULL,                  -- per-column min/max/null counts
  created_at        TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE TABLE table_manifest_entry (
  table_id          BIGINT       NOT NULL,
  snapshot_id       BIGINT       NOT NULL,                  -- which snapshot owns this entry
  data_file_id      BIGINT       NOT NULL,                  -- references table_data_file
  status            SMALLINT     NOT NULL,                  -- 0 = EXISTING, 1 = ADDED, 2 = DELETED
  added_in_snapshot BIGINT       NOT NULL,
  deleted_in_snapshot BIGINT,
  PRIMARY KEY (table_id, snapshot_id, data_file_id),
  FOREIGN KEY (table_id, snapshot_id) REFERENCES table_snapshot (table_id, snapshot_id),
  FOREIGN KEY (data_file_id) REFERENCES table_data_file (data_file_id)
);

CREATE INDEX ix_manifest_by_table_snapshot ON table_manifest_entry (table_id, snapshot_id);

A query with WHERE amount > 1000 can skip entire Parquet files whose column_stats.amount.max < 1000 without reading a single row. This is the key mechanism behind Iceberg’s dramatic scan reduction at scale.
ix_manifest_by_table_snapshotQuery-path indexThe most common metastore query is “give me all data files for snapshot X of table Y”; this index makes it a fast index seek rather than a full manifest scan.

4. Strategic explanations

How snapshot isolation works

A reader issues SELECT MAX(snapshot_id) FROM table_snapshot WHERE table_id = :t and pins that value for the duration of its query. It then reads manifest entries for exactly that snapshot_id, following only ADDED and EXISTING (non-DELETED) files. A concurrent writer commits a new snapshot_id; the reader sees none of it — full read isolation without row-level locking. This is the core of optimistic multi-version concurrency: writes never block reads because the data files are immutable and new writes produce new snapshots.

Time travel mechanics

Time travel has two access patterns: (1) by snapshot_id — query the manifest for that exact snapshot_id; (2) by timestamp — SELECT MAX(snapshot_id) FROM table_snapshot WHERE table_id = :t AND committed_at <= :target_ts. The parent_snapshot_id chain is needed for GC (to determine which snapshots are reachable from the current snapshot) but not for the read path, which goes directly to the manifest by snapshot_id.

Column statistics: the predicate pushdown engine

column_stats stores a per-column min/max/null-count map. For a query WHERE revenue > 500000, the metastore scan filters column_stats->>‘revenue’->>‘max’ > 500000 before returning any file paths to the query engine. Files whose max revenue is below the threshold are excluded without ever opening the Parquet file. In real Iceberg these statistics are stored in manifest files (Avro) rather than a Postgres table, but the logical structure is identical: per-column bounds per data file, evaluated at plan time.

Immutability and the garbage collection problem

Data files are never overwritten or deleted by normal write operations. A DELETE or UPDATE in the logical table produces a new snapshot with new data files and marks old files with status 2 (DELETED) in the manifest. The old files remain in object storage until a garbage collection (GC) job identifies files that are: (a) not referenced by any non-expired snapshot, and (b) older than the retention window. The deleted_in_snapshot column enables this: a file is GC-eligible when its deleted_in_snapshot is older than the oldest live snapshot. Without this tracking, GC would require scanning all manifest entries for every file.

Optimistic concurrency for writes

Two concurrent writers each read the current snapshot, produce new data files, and attempt to commit a new snapshot with parent_snapshot_id = current_snapshot_id. The PK (table_id, snapshot_id) does not prevent both from choosing the same next snapshot_id unless sequence generation is centralised. The standard approach: use a CAS (compare-and-swap) operation on a table_current_snapshot pointer table — UPDATE table_current SET snapshot_id = :new WHERE snapshot_id = :expected. If the update affects 0 rows, a concurrent writer has committed; the loser aborts, re-reads the new current snapshot, and retries. This is exactly the optimistic concurrency pattern that Iceberg uses at the metastore level.

Senior signal: explain that snapshot isolation is achieved by readers pinning a snapshot_id and following only that snapshot’s manifest entries — writes never block reads because data files are immutable; describe column_stats as the predicate-pushdown mechanism that lets the engine skip entire Parquet files without opening them; and articulate the GC eligibility condition (“a file is safe to delete when it is not referenced by any snapshot newer than the retention boundary”) using deleted_in_snapshot — these three points reveal that you understand the table format as a distributed systems protocol, not just a metadata schema.
