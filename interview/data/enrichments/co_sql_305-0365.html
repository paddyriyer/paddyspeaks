Airbnb's host-response-time SLA is a Superhost criterion. The query uses LEAD to find each host's first response after a guest message and computes per-listing average response latency.

1. The source data — messages

listing_idhost_idsent_bysent_at
L_01H_01guest10:00
L_01H_01host10:30

2. The SQL — LEAD for next message + filter to host responses

WITH guest_msgs AS (
  SELECT listing_id, sent_at AS guest_at,
    LEAD(sent_at) OVER (PARTITION BY listing_id ORDER BY sent_at) AS host_at,
    LEAD(sent_by) OVER (PARTITION BY listing_id ORDER BY sent_at) AS next_sender
  FROM messages
)
SELECT listing_id,
  COUNT(*) AS guest_messages,
  ROUND(AVG((julianday(host_at) - julianday(guest_at)) * 1440), 1) AS avg_response_minutes
FROM guest_msgs
WHERE next_sender = 'host'
GROUP BY listing_id
HAVING COUNT(*) >= 3
ORDER BY avg_response_minutes;

3. Strategic explanations

LEAD for "next message" pairing

For each guest message, look at the very next message (same listing, by sent_at). If the next message is from the host, the time-diff is response latency. If it's another guest message, no response yet — exclude.

The thread-fragmentation problem

"Same listing" doesn't mean "same conversation". One listing has many guest threads. The query above conflates them — treats every host msg as response to every preceding guest msg. Production keys on (listing_id, conversation_id) for honest pairing.

The 24-hour Superhost gate

Airbnb's Superhost criterion: 90% of inquiries answered within 24 hours. The query measures latency; flag the per-listing breach rate:

SUM(CASE WHEN host_at - guest_at > 1.0 THEN 1 ELSE 0 END) AS breaches_24h
Mean vs median latency

One bad night-time message taking 8 hours skews the mean. Median is more representative. SQLite lacks PERCENTILE_CONT; workaround via ROW_NUMBER:

WITH ranked AS (
  SELECT listing_id, latency,
    ROW_NUMBER() OVER (PARTITION BY listing_id ORDER BY latency) AS rn,
    COUNT(*) OVER (PARTITION BY listing_id) AS n
  FROM /* the diff CTE */
) SELECT listing_id, AVG(latency) AS median FROM ranked WHERE rn IN (n/2, n/2+1) GROUP BY listing_id;
Host-side timezone gap

Inquiries from US guests at 9pm ET arrive at 2am for a Tokyo host. Excluding overnight gaps gives the actual responsiveness:

WHERE NOT (CAST(strftime('%H', guest_at) AS INT) BETWEEN 0 AND 6)
Bot-response proxy

Hosts using Airbnb's auto-reply tool have artificially fast first-response times that don't reflect real engagement. Flag where first response time < 5 seconds = likely auto-reply.

Senior signal: use LEAD for next-message pairing, name the thread-fragmentation problem, surface the 24-hour Superhost gate, prefer median over mean for latency, and exclude overnight gaps for honest responsiveness. Auto-reply detection is the over-the-bar nuance.
