Google's infra teams model a lot of state as linked structures — LRU eviction chains, job-dependency lists, free-block pools. A corrupted pointer that loops a list back on itself turns a simple traversal into an infinite hang. "Detect a cycle in a linked list" looks like a toy question, but it's really probing whether you reach for constant extra space instead of the obvious hash set, and whether you can reason about two pointers moving at different speeds.

1. The input data

A singly linked list. The tail's next points back to node index 1, forming a cycle:

3 -> 2 -> 0 -> -4
     ^___________|     # -4.next == the node holding 2

head = node(3)         # cycle present  -> expect True
# contrast: 1 -> 2 -> None  -> expect False

2. The approach

This is Floyd's cycle detection ("tortoise and hare"). Two pointers start at head. Each iteration slow advances one node and fast advances two: slow, fast = slow.next, fast.next.next. The loop guard while fast and fast.next is doing double duty — it both stops cleanly at the end of an acyclic list and guarantees fast.next.next never dereferences None.

If there is a cycle, fast cannot "escape" it — once both pointers are inside the loop, the gap between them shrinks by exactly one node per step, so they are guaranteed to collide. The check slow is fast uses identity, not ==, because we care whether they are the same node object, not whether two nodes carry equal values.

3. The output

listhasCycle
3 → 2 → 0 → -4 → (back to 2)True
1 → 2 → NoneFalse
None (empty)False

For the cyclic list, slow and fast meet at the node holding value 2 after a few steps; the function returns True on that iteration.

4. Strategic explanations

Complexity

Time O(n), space O(1). The O(1) space is the entire point of the question. Before any pointers meet, fast traverses at most n nodes to enter the loop; once inside, it closes a gap of at most n, so total work is linear.

The idiomatic alternative

The "obvious" solution is a set of visited node ids — walk the list, return True if a node is seen twice:

def hasCycle(head):
    seen = set()
    while head:
        if head in seen: return True
        seen.add(head)
        head = head.next
    return False

It is O(n) time but O(n) space. Floyd's algorithm is the senior answer precisely because it drops the space to O(1). Mention both, then say which you'd ship and why.

The edge case

Empty list (head is None) and single node with no self-loop: while fast and fast.next is false immediately, so the function correctly returns False without entering the loop. A single node whose next points to itself is handled too — fast.next exists, both pointers stay on that node, and they're equal on the first iteration.

The follow-up

"Now return the node where the cycle begins." The classic extension: after the meeting point, reset one pointer to head, then advance both one step at a time — they meet at the cycle entrance (provable from the 2d = d + nk distance algebra). Same O(1) space.

The trap

Writing while fast.next and fast.next.next crashes on an empty list because fast itself is None. The guard must check fast before fast.next — short-circuit order matters. Another trap: comparing values (slow.val == fast.val) instead of identity, which gives false positives on lists with duplicate values.
