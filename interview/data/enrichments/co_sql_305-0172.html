PayPal's billing team sometimes applies a retroactive discount — a credit negotiated after an order was placed, stored in a separate retro table. Finance needs each order's true final total: original price, minus the discount at checkout, minus any retro credit. The interviewer is watching one thing: do you use the right join so that orders without a retro credit are not silently dropped?

Only orders 1 and 3 ever got a retro credit. Order 2's checkout discount is NULL — no discount applied. Orders 2 and 4 have no matching retro row. Those two facts — a NULL discount and a missing retro match — are the interview.

2. The SQL — LEFT JOIN + COALESCE

SELECT o.order_id,
  o.original_total
  - COALESCE(o.discount, 0)
  - COALESCE(r.retro_discount, 0) AS final_total
FROM orders o
LEFT JOIN retro r ON o.order_id = r.order_id;

A LEFT JOIN keeps every order; COALESCE(…, 0) turns both a NULL discount and a NULL (unmatched) retro_discount into 0 so the subtraction always produces a number.

4. Strategic explanations

LEFT JOIN, not INNER JOIN — the silent-revenue-loss trap

This is the heart of the question. An INNER JOIN returns only rows with a match in both tables — so orders 2 and 4, which have no retro entry, would vanish entirely. The finance report would be missing two orders and understate total revenue, with no error to warn you. The retro table is an optional supplement, so the order table must be the preserved (left) side. Picking the join type is the senior decision here, not the arithmetic.

Why COALESCE on retro_discount is mandatory

For an unmatched order, the LEFT JOIN fills r.retro_discount with NULL. And in SQL, any arithmetic involving NULL yields NULL: 100 - 10 - NULL = NULL. Without COALESCE, orders 2 and 4 would not disappear — worse, they would show final_total = NULL, a poisoned number that breaks every downstream SUM. COALESCE(r.retro_discount, 0) makes "no retro credit" mean "subtract zero", which is correct.

The duplicate-retro edge case — row fan-out

The query assumes at most one retro row per order. If retro can hold two credits for the same order_id, the LEFT JOIN fans out: that order appears twice, and a downstream SUM(final_total) double-counts its original_total. The robust pattern pre-aggregates the supplement before joining:

LEFT JOIN (
  SELECT order_id, SUM(retro_discount) AS retro_discount
  FROM retro GROUP BY order_id
) r ON o.order_id = r.order_id

Now each order matches exactly one summed row. Ask the interviewer whether retro is one-row-per-order or a log of credits — the answer changes the query.

The negative-total edge case

Nothing stops the discounts from exceeding original_total — a generous retro credit on a cheap order could push final_total negative. Is that a refund the customer is owed, or a data error to clamp at zero with GREATEST(…, 0)? Surface it; do not let an unexpected negative quietly enter the ledger.

The schema's redundant columns

orders ships both original_total and amount. They are probably pre- and post-discount snapshots that have drifted. Do not assume which is which — confirm whether amount already reflects the checkout discount, because if it does, subtracting discount again double-discounts the order.

Senior signal: choose LEFT JOIN so orders with no retro credit are kept rather than silently dropped by an INNER JOIN, wrap every nullable term in COALESCE(…, 0) because NULL arithmetic poisons the total, pre-aggregate retro if it can have multiple rows per order, and flag negative totals and the original_total/amount ambiguity.
