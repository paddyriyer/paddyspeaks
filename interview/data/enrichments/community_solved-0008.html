A Community data-cleaning question with a twist: a routes table where A→B and B→A are logically the same trip, and the distance column is dirty text. Compute the average distance per undirected route. The interviewer is probing two skills at once — canonicalising a pair so direction does not matter, and scrubbing free-text numbers safely.

1. The source data — locations

sourcedestinationdistance
MumbaiDelhi1400
DelhiMumbai1390
BangaloreChennai350 km
ChennaiBangalore360
HyderabadPune560

Mumbai–Delhi appears twice with slightly different distances (1400 vs 1390) — the same road measured imperfectly. One distance is dirty: '350 km' carries a unit suffix. And Kolkata–Patna appears only once. All three are deliberate edge cases.

2. The SQL — canonical route key + cleaned average

SELECT
    CASE WHEN source > destination THEN concat(source, '-', destination)
         ELSE concat(destination, '-', source) END AS unique_route,
    AVG(NULLIF(regexp_replace(distance::text, '[^-.0-9]', '', 'g'), '')::numeric) AS average_distance
FROM locations
GROUP BY CASE WHEN source > destination THEN concat(source, '-', destination)
              ELSE concat(destination, '-', source) END;

The CASE always puts the alphabetically-larger city first, so Mumbai→Delhi and Delhi→Mumbai both collapse to the key Mumbai-Delhi. regexp_replace(..., '[^-.0-9]', '', 'g') strips every character that is not a digit, dot, or minus — turning '350 km' into '350' — and NULLIF(..., '') guards against a value that cleans down to nothing.

Four routes from six rows — the two-row pairs were folded. Mumbai-Delhi averages 1400 and 1390 to 1395; Chennai-Bangalore averages the clean 360 and the de-suffixed '350 km' → 350 to 355, proving the scrub worked. Single-row routes (Patna–Kolkata, Pune–Hyderabad) just report their lone value. The result is a numeric, hence the long decimal tail — cosmetic only.

4. Strategic explanations

The canonical-key trick — why GREATEST/LEAST is cleaner

Sorting the two endpoints into a fixed order is the heart of "undirected pair" problems. The CASE works but is verbose and repeated in both SELECT and GROUP BY. Postgres has a one-liner:

concat(LEAST(source, destination), '-', GREATEST(source, destination)) AS unique_route

Same result, no duplicated expression. You can also GROUP BY 1 to reference the select-list position and avoid restating it at all.

The "$0 vs NULL" question hiding in regexp_replace

A distance like 'N/A' or 'unknown' strips down to the empty string. Without the NULLIF(..., ''), ''::numeric throws an error and the whole query fails. With it, the value becomes NULL — and AVG skips NULLs, so a missing distance is excluded from both numerator and denominator. That is usually right, but say it: a route with one real reading and one 'N/A' averages to the real reading, not to half of it.

The regex keeps - and . — a double-edged choice

The character class [^-.0-9] preserves minus and decimal point so '-12.5' survives intact. But '12-15 km' (a range) cleans to '12-15', which fails the ::numeric cast. And a stray dot — '1.400 km' in a locale that uses dot as a thousands separator — would be misread as 1.4. Free-text numbers are genuinely ambiguous; flag that the cleaner assumes a single well-formed number per cell.

Interviewer follow-up: flag the inconsistent duplicates

Averaging 1400 and 1390 hides a data-quality problem. A senior surfaces it rather than smoothing it over:

SELECT unique_route, MIN(d) AS lo, MAX(d) AS hi, MAX(d) - MIN(d) AS spread
FROM ( ... cleaned d ... ) x
GROUP BY unique_route
HAVING MIN(d) <> MAX(d);

That isolates routes whose two directions disagree — the rows worth a human's attention.

Self-referential routes and the > tie

If a row has source = destination (a degenerate "trip" from a city to itself), the CASE's > is false, so it falls to the ELSE and keys as City-City — harmless, but probably should be filtered with WHERE source <> destination. Equal city names land in the ELSE branch deterministically, so there is no key ambiguity.

Senior signal: canonicalise the undirected pair with LEAST/GREATEST instead of a repeated CASE, explain that NULLIF(regexp_replace(...), '') both prevents an empty-string cast error and makes AVG skip unparseable distances, and treat disagreeing reverse-pair distances as a data-quality finding rather than silently averaging them.
