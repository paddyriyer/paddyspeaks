Expedia's analytics team needs every order for an Electronics product. The natural instinct is to join orders to products and then filter by category — but where the category test goes (in the ON clause or in a WHERE) is the entire point of the question. For an inner join the two are equivalent; the interviewer wants to hear that you know why, and when they stop being equivalent.

Product 2 is a Book — order 2 must be excluded. Product 1 (a Laptop) is ordered twice (orders 1 and 4), so both should survive. That repeat is a quiet check that the join is a clean one-row-per-order match, not a fan-out.

2. The SQL — JOIN with a condition in ON

SELECT o.* FROM orders o
JOIN products p ON o.product_id = p.product_id AND p.category = 'Electronics';

The ON clause carries two predicates: the join key (o.product_id = p.product_id) and a filter on the product side (p.category = 'Electronics'). Only order rows whose product is both matched and Electronics pass.

4. Strategic explanations

ON vs WHERE — identical for an inner join

For an INNER JOIN, putting p.category = 'Electronics' in the ON clause or in a separate WHERE produces exactly the same result — the planner applies both predicates before any row is emitted:

SELECT o.* FROM orders o
JOIN products p ON o.product_id = p.product_id
WHERE p.category = 'Electronics';

Many teams prefer the WHERE form precisely because it reads as "join key in ON, filters in WHERE" — a discipline that pays off the moment the join becomes outer.

The trap: the same condition under a LEFT JOIN

This is the real lesson. Change JOIN to LEFT JOIN and the placement stops being cosmetic. A category predicate in ON only decides whether the right side matches — unmatched left rows are still kept with NULLs. The same predicate in WHERE filters after the join and discards those NULL rows, silently turning the LEFT JOIN back into an inner join:

-- keeps ALL orders; non-Electronics show NULL product columns
SELECT o.*, p.name FROM orders o
LEFT JOIN products p ON o.product_id = p.product_id AND p.category = 'Electronics';

-- keeps only Electronics orders — the LEFT was pointless
SELECT o.*, p.name FROM orders o
LEFT JOIN products p ON o.product_id = p.product_id
WHERE p.category = 'Electronics';

Knowing this distinction — ON filters the match, WHERE filters the result — is exactly what a "JOIN with multiple conditions" question is probing.

The duplicate-rows risk — is the join key unique?

This works cleanly because product_id is the primary key of products — each order matches at most one product, so orders rows are never multiplied. If you joined on a non-unique column (say products.manufacturer), one order could match many product rows and SELECT o.* would emit duplicate orders. Before any join, confirm the right-side key is unique — or be ready with DISTINCT or EXISTS.

EXISTS — the intent-revealing alternative

Since the query returns only orders columns and never uses products data, a semi-join expresses the intent precisely — "orders whose product is Electronics" — and is immune to the duplication risk:

SELECT * FROM orders o
WHERE EXISTS (SELECT 1 FROM products p
              WHERE p.product_id = o.product_id
                AND p.category = 'Electronics');

EXISTS stops at the first match, so even a non-unique key cannot fan out the result. When you need filtering but not columns from the other table, it is often the cleaner choice.

NULL product_id and case sensitivity

An order with a NULL product_id never matches anything (NULL equality is unknown) and is correctly dropped by the inner join. And category = 'Electronics' is case- and whitespace-sensitive — 'electronics' or ' Electronics' would silently match nothing; normalise the data or use ILIKE/TRIM if the category column is dirty.

Senior signal: state that ON and WHERE are interchangeable for an inner join but diverge sharply under a LEFT JOIN — ON filters the match, WHERE filters the result — confirm the join key is unique before trusting SELECT o.*, and reach for EXISTS when you need filtering without the other table's columns.
