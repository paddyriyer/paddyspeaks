Facebook's "People You May Know" feature is the canonical 2-hop graph problem: friends of my friends, ranked by mutual count. The query is a self-join on the friendships table. The senior conversation: how to handle existing friendships, undirected vs directed edges, and the explosion at scale.

User 1 is friends with 2 and 4. Both 2 and 4 are friends with 3. So 3 is a 2-hop neighbor of 1 with mutual count 2.

2. The SQL — self-join with one-hop intermediate

WITH pairs AS (
  SELECT a.user_id AS me, b.friend_id AS suggested_friend,
    COUNT(*) AS mutual_friends
  FROM friendships a
  JOIN friendships b
    ON a.friend_id = b.user_id
   AND a.user_id  != b.friend_id
  GROUP BY a.user_id, b.friend_id
)
SELECT * FROM pairs
ORDER BY mutual_friends DESC, me, suggested_friend
LIMIT 20;

3. Strategic explanations

The 2-hop join in 3 lines

JOIN friendships b ON a.friend_id = b.user_id says "go from me to my friend a.friend_id, then from there to whoever they're friends with — b.friend_id". The result set is every (me, friend-of-friend) pair, with one row per mutual-friend path. GROUP BY + COUNT gives the mutual count.

The "exclude existing friends" filter

The result includes friends-of-friends who are already direct friends. Production "People You May Know" excludes those:

WHERE NOT EXISTS (
  SELECT 1 FROM friendships f
  WHERE f.user_id = me AND f.friend_id = suggested_friend
)
AND me != suggested_friend  -- exclude self
Undirected vs directed

If friendships is undirected (every friendship has two rows: A→B and B→A), the query works as written. If directed (only one row per pair), you need to OR the join condition: (a.friend_id = b.user_id) OR (a.friend_id = b.friend_id). Get this wrong and you find half the recommendations.

The cardinality explosion

For a graph with avg degree 100 and 3B users, this self-join produces ~30T candidate pairs. Real Meta uses pre-computed materialized 2-hop tables, refreshed nightly via Spark — never the live join. Mention this when asked about scale: "the SQL is correct; in prod we'd batch-precompute".

The ranking signal stack

"Mutual friends" is one signal. Production ranking adds:

  • School / employer overlap — strong predictor
  • Geo proximity — same city/zip
  • Profile-view symmetry — they checked you, you check them
  • Negative signals — blocked / hidden / unfriended

Mutual count alone produces a noisy rank; adding 5-10 features makes the recommendations dramatically better. The ranker is a learned model; this query just produces the candidate pool.

The privacy filter — the must-have

"PYMK" must respect block lists. Filter out anyone the user has explicitly blocked, hidden, or unfriended. This requires a join to a user_blocks table — easy to forget, catastrophic when you do.

Senior signal: articulate the 2-hop join in one breath, raise the directed-vs-undirected gotcha, name the cardinality explosion at scale (precomputed in prod), and pre-empt privacy/block filters. The mutual-count-as-one-feature framing for the ranking pipeline is the over-the-bar finish.
