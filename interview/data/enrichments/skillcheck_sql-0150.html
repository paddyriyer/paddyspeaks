Survey responses arrive in long format — one row per (question, respondent, answer) — but the analytics team wants a wide table: one row per respondent, one column per question. That is a pivot. The interviewer is probing conditional aggregation, and specifically whether you pick an aggregate that works on text answers and whether you handle respondents who skipped a question.

The edge cases are partial responders. Respondent 2 has no q3 row, respondent 3 has no q2 row, and respondent 4 answered only q2. A correct pivot must place a NULL in each missing cell — the absent answer must not disturb the answers that exist.

2. The SQL — pivot via MAX(CASE …)

SELECT respondent_id,
       MAX(CASE WHEN question_id = 'q1' THEN answer END) AS q1,
       MAX(CASE WHEN question_id = 'q2' THEN answer END) AS q2,
       MAX(CASE WHEN question_id = 'q3' THEN answer END) AS q3
FROM survey
GROUP BY respondent_id
ORDER BY respondent_id;

One CASE per target column lets a row's answer through only when it belongs to that question, and yields NULL otherwise (no ELSE). GROUP BY respondent_id collapses each respondent's rows into one; MAX picks out the single non-NULL answer the CASE exposed.

4. Strategic explanations

Why MAX and not SUM — the answers are text

SUM only works on numbers; the answers here are strings, so the aggregate must be one that operates on text. MAX (and MIN) accept text, ordering it lexicographically. Within each respondent/question cell the CASE exposes at most one non-NULL value, so "the max of one value" is just that value — MAX is being used as a NULL-skipping picker, not for any ordering meaning. SUM(CASE …) would simply error on text.

No ELSE → non-matching rows are NULL, and MAX ignores them

CASE WHEN question_id = 'q1' THEN answer END has no ELSE, so every row that is not q1 produces NULL. Aggregate functions skip NULLs, so MAX over a respondent's rows sees only their actual q1 answer (if any) and returns it — or NULL if there was none. That NULL-skipping is the entire mechanism: it both isolates the right answer and produces the blank for a skipped question. The question explicitly wants NULL for skips, so not adding an ELSE is the correct, deliberate choice.

What if a respondent answered the same question twice

The fixture has at most one row per (respondent, question), so the cell is unambiguous. If a duplicate existed — respondent 1 answering q1 both "yes" and "no" — MAX would silently pick the lexicographically larger ("yes") and the other answer would vanish with no warning. That is a real data-quality risk: if duplicates are possible, dedupe upstream or decide a tie-break rule explicitly rather than letting MAX choose arbitrarily.

The column list is static — the title says "dynamic", the data does not

The three questions q1/q2/q3 are hard-coded as three CASE expressions. Add a q4 and it appears in no column — it silently disappears from the report. The question states the question set is "small and known", so static columns are acceptable here. A genuinely dynamic pivot — columns grown from SELECT DISTINCT question_id — needs procedural code to build the SQL text, or Postgres's crosstab() from the tablefunc extension. Naming that limit unprompted is the senior signal.

Dialect — the PIVOT operator

SQL Server, Oracle, and Snowflake ship a dedicated PIVOT clause — e.g. Snowflake … PIVOT(MAX(answer) FOR question_id IN ('q1','q2','q3')) — which is terser but still makes you enumerate the values in the IN list, exactly like the CASE form. Postgres, MySQL, and SQLite have no PIVOT keyword, so MAX(CASE WHEN …) is the portable answer and is worth knowing as the universal idiom.

Senior signal: pivot text data with MAX(CASE WHEN …) — SUM would fail on strings — rely on the absent ELSE so skipped questions naturally become NULL, watch that duplicate (respondent, question) rows would let MAX drop an answer silently, and note the columns are static so a truly dynamic question set needs dynamic SQL or crosstab().
