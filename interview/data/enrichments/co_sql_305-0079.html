EPAM is auditing a client's warehouse-management system and the data-quality lead has a hunch: somewhere in the inventory table, qty_on_hand has gone negative. A negative stock count is physically impossible — you cannot have minus three units on a shelf — so any such row is a symptom of a bug: a double-decrement, an out-of-order event, a sale recorded before a receipt. The query itself is one line; the interview is whether you treat the result as a finding rather than an answer.

Two rows are negative (products 2 and 5). Product 4 sits at exactly zero — that is the edge case the predicate has to get right: zero is a legitimate "sold out" state, not a defect, so it must not be flagged.

2. The SQL — WHERE qty_on_hand < 0

SELECT product_id, qty_on_hand FROM inventory WHERE qty_on_hand < 0;

A strict less-than filter. Using < 0 rather than <= 0 is the deliberate choice: it keeps the zero-stock row out of the result.

Exactly the two impossible rows surface. Product 4 (qty 0) is correctly absent. The output is a defect list — each row is a ticket for someone to investigate.

4. Strategic explanations

The < 0 vs <= 0 trap

The single most common mistake here is writing qty_on_hand <= 0. That folds in every sold-out product — product 4 — and buries the two real bugs in a sea of false positives. Zero is normal; negative is broken. Be explicit about which one you are hunting and why the boundary sits where it does.

The NULL blind spot

If qty_on_hand is nullable, a NULL means "quantity unknown" — arguably a worse data-quality problem than a negative number. But NULL < 0 evaluates to unknown, so a NULL row silently fails the filter and never appears. A thorough audit checks for both:

SELECT product_id, qty_on_hand
FROM inventory
WHERE qty_on_hand < 0 OR qty_on_hand IS NULL;

Mention this even if the column is declared NOT NULL today — schemas drift, and the audit query should be robust to that.

Quantifying the damage

An interviewer will often push past "which rows" to "how bad is it". A summary row turns the finding into something a manager can act on:

SELECT COUNT(*) AS bad_rows,
       SUM(qty_on_hand) AS total_negative_units,
       MIN(qty_on_hand) AS worst
FROM inventory
WHERE qty_on_hand < 0;

For this data: 2 bad rows, −13 total units, worst case −10. That framing — scale and severity — is what separates a query result from a remediation plan.

Finding the cause, not just the symptom

Negative inventory is downstream of a movements/ledger table. The real question is which transaction drove the balance below zero. If a stock_movements(product_id, delta, moved_at) table exists, a running total exposes the exact event:

SELECT product_id, moved_at, delta,
       SUM(delta) OVER (PARTITION BY product_id ORDER BY moved_at) AS running_qty
FROM stock_movements
ORDER BY product_id, moved_at;

The first row where running_qty goes negative is the offending event. Naming this as the follow-up shows you see the table as part of a system, not in isolation.

Senior signal: use strict < 0 so legitimate zero-stock rows are not flagged, remember that NULL quantities slip through the predicate and decide whether to catch them, and frame the output as a defect list — quantifying severity and pointing at the movements ledger as where the root cause actually lives.
