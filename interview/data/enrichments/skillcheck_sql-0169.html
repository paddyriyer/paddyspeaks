A room-booking system needs to detect double-bookings: pairs of reservations for the same room whose time intervals overlap. The interviewer is probing the interval-overlap predicate — the deceptively simple a.start < b.end AND b.start < a.end — and whether you understand half-open [start, end) semantics, where a booking ending at 11:00 does not clash with one starting at 11:00.

2. The SQL — self-join with the overlap predicate

SELECT a.room_id, a.id AS id_a, b.id AS id_b
FROM booking a JOIN booking b
  ON a.room_id = b.room_id
 AND a.id < b.id
 AND a.start_ts < b.end_ts
 AND b.start_ts < a.end_ts
ORDER BY a.room_id, a.id, b.id;

The table joins to itself: every booking a is paired with every booking b. a.room_id = b.room_id keeps pairs in the same room; a.id < b.id ensures each pair is emitted once; and the two strict inequalities are the overlap test — two intervals intersect exactly when each starts before the other ends.

4. Strategic explanations

Why a.start < b.end AND b.start < a.end is the whole test

It is far easier to characterise when intervals do not overlap: either a ends before b begins (a.end ≤ b.start) or b ends before a begins (b.end ≤ a.start). Overlap is the negation of that disjunction, and De Morgan turns it into a conjunction: a.start < b.end AND b.start < a.end. Trying to enumerate overlap cases directly — a inside b, b inside a, partial left, partial right — is error-prone; the "negate the non-overlap" derivation gets it right in two clean clauses every time.

Half-open intervals and the choice of < vs ≤

The question specifies [start, end) — the start instant belongs to the interval, the end instant does not. Under that model a booking ending at 11:00 occupies the room up to but not including 11:00, so a new booking starting at 11:00 is fine. Strict < encodes exactly this: equal boundary timestamps fail the test. If the intervals were closed [start, end] (touching counts as a clash), you would use ≤ instead. This single operator choice is the crux — getting < vs ≤ wrong silently mis-classifies every boundary-touching pair.

a.id < b.id — dedupe and self-pair elimination in one stroke

An unconstrained self-join produces every pair twice (as (1,2) and (2,1)) and pairs every row with itself (1,1) — and a row trivially "overlaps" itself. The predicate a.id < b.id fixes both at once: it is false when a.id = b.id (kills self-pairs) and admits only the ordered version of each pair (kills the mirror duplicate). It also conveniently delivers the requested id_a < id_b output ordering for free.

Performance — the self-join is quadratic

A self-join compares every booking against every other — O(n²) pairs before filtering. Fine for six rows, punishing for a year of bookings. The a.id < b.id predicate roughly halves the work but does not change the complexity class. On a large table, an index on booking(room_id, start_ts) lets the optimiser restrict the inner scan to the same room and a relevant time window. For heavy interval workloads PostgreSQL offers a genuinely better tool: a tstzrange column with a GiST index and the && overlap operator, which can answer overlap queries without the full cross product. Mentioning this scaling path unprompted is the senior signal.

Dialect notes — range types

The </≤ two-inequality predicate is portable to every SQL engine. PostgreSQL additionally has first-class range types: model the booking as tsrange(start_ts, end_ts, '[)') — the '[)' literally declares half-open bounds — and test conflicts with the && operator. That makes the half-open semantics self-documenting and, with a GiST index, fast. It is also the foundation for an EXCLUDE constraint that prevents double-bookings at write time rather than detecting them after the fact — a stronger design than a detection query. Other engines (MySQL, SQL Server) have no range types and rely on the explicit inequalities.

Senior signal: express interval overlap as a.start < b.end AND b.start < a.end (derived by negating non-overlap), use strict < for half-open [start, end) so touching intervals do not clash, add a.id < b.id to kill self-pairs and mirror duplicates at once, and know the self-join is O(n²) — Postgres range types with a GiST index (and an EXCLUDE constraint) are the scalable answer.
