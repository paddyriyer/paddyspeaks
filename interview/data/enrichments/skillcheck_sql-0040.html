A subscription business reviews its plan economics at year end: "for every customer, how many days did they actually spend in each plan up to 2025-12-31?" The subs table is an append-only event log — one row per state change, no end dates — so a plan period only ends when the next row begins. The interviewer is probing whether you can derive intervals from a change-log and clamp them to a reporting cutoff without double-counting or running past the boundary.

The edge case is the last row of each customer: customer 1's churn and customer 2's annual have no following event, so their period must be closed by the cutoff, not by a NULL. Get that wrong and the final plan contributes zero days.

2. The SQL — LEAD to close intervals, clamped to the cutoff

WITH intervals AS (
  SELECT customer_id, plan, start_date,
         COALESCE(
           LEAD(start_date) OVER (PARTITION BY customer_id ORDER BY start_date),
           DATE '2026-01-01'
         ) AS end_exclusive
  FROM subs
)
SELECT customer_id, plan,
       SUM((LEAST(end_exclusive, DATE '2026-01-01') - start_date))::int AS days
FROM intervals
WHERE start_date < DATE '2026-01-01'
GROUP BY customer_id, plan
ORDER BY customer_id, plan;

LEAD(start_date) looks one row forward within each customer's ordered timeline to find when the current plan ends. COALESCE(..., DATE '2026-01-01') supplies the cutoff (one day past 2025-12-31, so the boundary day is included) for the final row. LEAST clamps any interval so it can never extend past the cutoff, and date subtraction in Postgres yields an integer day count.

4. Strategic explanations

Why LEAD turns a change-log into intervals

An append-only log stores when a state began but never when it ended — the end is implicit in the next row. LEAD(start_date) OVER (PARTITION BY customer_id ORDER BY start_date) reaches forward to that next row, converting "valid-from" events into "valid-from / valid-to" intervals. This is the canonical pattern for any temporal log: status histories, price changes, employment records all yield to the same trick.

Half-open intervals avoid the off-by-one

The next plan's start_date is used as an exclusive end. Trial runs [Jan 1, Jan 8) — the day basic begins belongs to basic, not trial. Subtracting dates then gives exactly 7, with no day counted twice and none lost at the seam. Had the query used an inclusive end (next_start - 1) it would still work, but half-open [start, end) arithmetic is the convention precisely because end - start is the duration directly.

The cutoff must be applied in two places

COALESCE handles the last row (no successor), but LEAST(end_exclusive, cutoff) is still needed for any non-final row whose successor starts after the cutoff — without the clamp, a plan that began in 2025 but whose next change is in 2026 would over-count into the future. The WHERE start_date < cutoff guard then drops any row that begins entirely after the reporting window. Cutoff logic belongs at both the open and close of every interval.

Boundary convention: why 2026-01-01, not 2025-12-31

The cutoff date is written as 2026-01-01 even though the report stops at 2025-12-31. Because the end is exclusive, using the day after the last in-scope day makes 2025-12-31 itself count. Using 2025-12-31 as the exclusive bound would silently drop the final day. State the inclusive-vs-exclusive intent out loud — off-by-one at the reporting boundary is the single most common bug in time-in-state questions.

Dialect notes — date subtraction is not universal

date - date => integer is Postgres behaviour. SQL Server needs DATEDIFF(day, start_date, end_exclusive); BigQuery uses DATE_DIFF(end_exclusive, start_date, DAY); MySQL uses DATEDIFF(end_exclusive, start_date). The interval logic (LEAD + COALESCE + LEAST) is portable; only the day-count subtraction needs translation per engine.

Senior signal: recognise an append-only log as implicit intervals, close them with LEAD + COALESCE, use half-open [start, end) arithmetic so durations are exact, and apply the cutoff at both ends — clamping with LEAST and filtering out-of-window rows — while stating the inclusive/exclusive boundary convention explicitly.
