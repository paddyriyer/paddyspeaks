The HackerRank "Symmetric Pairs" problem, posed here from a Community question bank. A table holds (x, y) pairs; a pair is symmetric if its mirror (y, x) also appears in the table. Report each symmetric pair once, with x ≤ y. It is a self-join puzzle, and the interviewer is watching for the deduplication condition and the self-pair edge case.

(1,2)/(2,1), (3,4)/(4,3) and (5,6)/(6,5) are three mirrored pairs. (7,8) has no mirror — the non-symmetric case. (9,9) is its own mirror — the self-pair edge case that the x ≤ y filter must still admit.

2. The SQL — self-join on the mirror condition

SELECT DISTINCT p1.x AS x1, p1.y AS y1
FROM pairs p1
INNER JOIN pairs p2
  ON p1.x = p2.y AND p2.x = p1.y
WHERE p1.x <= p1.y
ORDER BY x1 ASC;

Join the table to itself: a row p1 matches a row p2 when p2 is p1 reversed. The WHERE p1.x <= p1.y keeps only one representative of each mirrored pair so the output is not doubled.

4. Strategic explanations

Why x ≤ y — and why ≤, not <

Without a filter, both (1,2) and (2,1) satisfy the join, so every symmetric pair would print twice. p1.x <= p1.y keeps exactly the orientation where the first value is the smaller, collapsing each mirror down to one row. It must be ≤, not <: a self-pair like (9,9) has x = y, and a strict < would wrongly discard it. The boundary choice is the edge-case handling.

The self-pair: a join partner that is the same row

(9,9) is subtle. The join condition p1.x = p2.y AND p2.x = p1.y is satisfied by p1 matched against itself. So a single (9,9) row is "symmetric" with no second row needed. Some variants of this question require the mirror to be a distinct row — if so, add AND p1.ctid <> p2.ctid (or a primary-key inequality) to forbid self-matching. Clarify which semantics the question wants.

The trap: duplicate rows inflate the join

If (1,2) appeared twice, the self-join would pair each copy of (1,2) with each copy of (2,1) — a Cartesian blow-up. DISTINCT in the SELECT rescues the output here, but on large tables the intermediate explosion is real work. If duplicates are possible, dedupe first or carry a unique key.

A set-based alternative — INTERSECT

"A pair whose mirror also exists" is a set-membership statement, which INTERSECT expresses without a join:

SELECT x, y FROM pairs
INTERSECT
SELECT y, x FROM pairs;

This returns every pair that equals some other pair's reverse — including both orientations, so wrap it with WHERE x <= y to dedupe. INTERSECT also removes duplicates for free, sidestepping the blow-up above. EXISTS is a third route: WHERE EXISTS (SELECT 1 FROM pairs m WHERE m.x = p.y AND m.y = p.x).

NULL coordinates

If any x or y is NULL, the equality join conditions evaluate to unknown and that row matches nothing — a NULL-bearing pair is never reported as symmetric. That is the right behaviour (a NULL has no defined mirror), but say it: NULLs drop out silently rather than erroring.

Senior signal: use ≤ not < in the dedup filter so self-pairs survive, explain that a self-pair joins to its own row, recognise the duplicate-row Cartesian blow-up that DISTINCT only masks, and know that INTERSECT expresses the same problem more cleanly.
