A warehouse-management system records every stock movement as a signed delta — positive for receipts, negative for shipments — and the ops dashboard needs the running on-hand quantity after each event. The inventory_event table is partitioned by warehouse and SKU. The interviewer is probing whether you reach for a windowed cumulative SUM with the correct partitioning and frame, rather than a self-join or a procedural loop.

1. The source data — inventory_event

warehouse_idskutsdelta(narrative)
1A2025-01-01 09:00+100opening receipt
1A2025-01-02 10:00-30shipment — running total drops
1A2025-01-03 11:00-20another shipment
1A2025-01-04 12:00+50restock
1B2025-01-01 09:00+40different SKU — separate balance
1B2025-01-02 09:00-10...
2A2025-01-01 09:00+200same SKU, different warehouse
2A2025-01-03 09:00-50...

The edge case is SKU A existing in both warehouse 1 and warehouse 2. The running balance must track each (warehouse, sku) stream independently — warehouse 2's 200 units must never leak into warehouse 1's on-hand. SKU B confirms the same isolation across SKUs within one warehouse.

2. The SQL — cumulative SUM over a partitioned window

SELECT warehouse_id, sku, ts, delta,
       SUM(delta) OVER (PARTITION BY warehouse_id, sku ORDER BY ts
                        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS on_hand
FROM inventory_event
ORDER BY warehouse_id, sku, ts;

PARTITION BY warehouse_id, sku gives every (warehouse, sku) pair its own independent accumulator. ORDER BY ts sequences the events. ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW sums every event from the first up to and including the current one — the running total.

Warehouse 1 / SKU A walks 100 → 70 → 50 → 100. Warehouse 2 / SKU A starts fresh at 200, not 300 — the partition boundary resets the accumulator, so the two warehouses' stocks of the same SKU never commingle.

4. Strategic explanations

Why a window function beats a self-join

The pre-window approach — SUM(e2.delta) from a correlated subquery joining every prior row — is O(n²): each row re-scans all rows before it. SUM(delta) OVER (... ORDER BY ts) sorts each partition once and accumulates in a single pass, O(n log n). For an event log that grows without bound, the window function is not just cleaner — it is the only approach that scales.

The frame is what makes it "running"

ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW is the explicit running-total frame. As it happens, this is also Postgres' default frame when an ORDER BY is present (the default is the RANGE variant, which here yields the same numbers because timestamps are unique). Writing the frame explicitly removes all ambiguity — and protects you if two events ever share a timestamp, where RANGE and ROWS diverge.

Composite partition key — the isolation that matters

PARTITION BY warehouse_id, sku — both columns — is the heart of correctness. SKU A lives in two warehouses; partitioning by sku alone would sum 100 and 200 into one nonsensical 300-unit balance. The accumulator must reset at every distinct (warehouse, sku) stream. The demo's shared-SKU rows are placed precisely to expose this if the partition key is wrong.

ROWS vs RANGE when timestamps collide

If two events on the same SKU shared an identical ts, RANGE (the default) would treat them as peers and give both rows the same running total — the sum through the end of that timestamp group. ROWS processes them one at a time, producing a distinct intermediate total per row. For an inventory ledger you almost always want ROWS; if simultaneous events are possible, add a deterministic tiebreaker like ORDER BY ts, event_id.

Dialect notes — broadly portable

Windowed cumulative SUM with PARTITION BY, ORDER BY, and an explicit ROWS frame is SQL-standard and behaves identically in Postgres, SQL Server, BigQuery, Snowflake, Oracle, and MySQL 8+/SQLite 3.25+. There is nothing dialect-specific in this query — it ports verbatim, which is part of why the window function is the canonical answer.

Senior signal: compute a running balance with a windowed cumulative SUM — O(n log n), not the O(n²) self-join — partition by the full stream key (warehouse_id, sku) so independent ledgers never merge, and state the explicit ROWS frame plus a tiebreaker so simultaneous events stay deterministic.
