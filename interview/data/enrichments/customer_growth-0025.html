The CFO wants a single daily report that explains where today's revenue came from: not just a total, but a breakdown by growth bucket — new customers, retained, resurrected, churned — each with revenue, ARPU, and L1/L7/L28 activity counts. The interviewer is probing whether you can fuse a growth-accounting FULL OUTER JOIN with conditional aggregation, handle the awkward CHURNED bucket (zero revenue, NULL ARPU), and compute window-based recency counts with FILTER.

1. The source data — orders and user_state_master

tablerow(narrative)
orders501 spent $150 today; was active_l28=1→ RETAINED
orders502 spent $80 today; was active_l28=0→ RESURRECTED
orders503, 504 spent today; active_l28=1→ RETAINED
orders505, 506 spent today; no master row→ NEW
user_state_master507, 508 — active_l28=1, no order today→ CHURNED
user_state_master509 — active_l28=0, no order today→ STALE (excluded)

The edge cases: the CHURNED bucket (507, 508) has no orders today, so its revenue sum is empty and ARPU must be NULL rather than a divide-by-zero. User 509 is STALE — inactive both windows — and the report drops it entirely.

2. The SQL — growth-accounting revenue flow with L-window velocity

WITH today_activity AS (
    SELECT user_id, SUM(amount) AS today_spend, COUNT(order_id) AS today_qty
    FROM orders WHERE order_date = DATE '2026-05-06'
    GROUP BY user_id
),
growth_merge AS (
    SELECT
        COALESCE(t.user_id, s.user_id) AS user_id,
        t.today_spend, t.today_qty,
        s.last_active_date, s.is_active_l28,
        CASE
            WHEN s.user_id IS NULL                                          THEN 'NEW'
            WHEN t.user_id IS NOT NULL AND COALESCE(s.is_active_l28,0) = 1   THEN 'RETAINED'
            WHEN t.user_id IS NOT NULL AND COALESCE(s.is_active_l28,0) = 0   THEN 'RESURRECTED'
            WHEN t.user_id IS NULL     AND COALESCE(s.is_active_l28,0) = 1   THEN 'CHURNED'
            ELSE 'STALE'
        END AS growth_status,
        CASE WHEN t.user_id IS NOT NULL THEN 0
             ELSE (DATE '2026-05-06' - s.last_active_date)
        END AS days_since_active
    FROM user_state_master s
    FULL OUTER JOIN today_activity t ON s.user_id = t.user_id
)
SELECT growth_status,
    COUNT(DISTINCT user_id)                  AS user_count,
    COALESCE(ROUND(SUM(today_spend), 2), 0)  AS total_revenue,
    COALESCE(SUM(today_qty), 0)              AS total_qty,
    CASE WHEN COALESCE(SUM(today_spend), 0) = 0 THEN NULL
         ELSE ROUND(SUM(today_spend) / COUNT(DISTINCT user_id), 2) END AS arpu,
    COUNT(DISTINCT user_id) FILTER (WHERE days_since_active = 0)  AS l1_users,
    COUNT(DISTINCT user_id) FILTER (WHERE days_since_active <= 7) AS l7_users,
    COUNT(DISTINCT user_id) FILTER (WHERE days_since_active <= 28) AS l28_users
FROM growth_merge
WHERE growth_status <> 'STALE'
GROUP BY growth_status
ORDER BY CASE growth_status
    WHEN 'NEW' THEN 1 WHEN 'RETAINED' THEN 2
    WHEN 'RESURRECTED' THEN 3 WHEN 'CHURNED' THEN 4
END;

The CHURNED row is the interesting one: zero revenue (no orders today), arpu = NULL by the explicit rule, and l1_users = 0 — they were not active today — but l7_users = 2 because 507 (gap 2 days) and 508 (gap 3 days) both last logged in within the past week. Churn is recent here, which is a louder alarm than long-dormant churn.

4. Strategic explanations

Why ARPU must be NULL, not zero, for CHURNED

ARPU is revenue per user. For CHURNED there is revenue of zero across two users — arithmetically that is $0.00, but reporting "ARPU = $0" implies the segment was measured and earned nothing, when in fact ARPU is undefined for a segment that did not transact. The query guards with CASE WHEN SUM(today_spend) = 0 THEN NULL, which also doubles as the divide-by-zero protection. A senior distinguishes "zero, a real measurement" from "NULL, not applicable" — conflating them corrupts every downstream average.

The L-window counts: pre-compute the gap, then FILTER

days_since_active is computed once per user in growth_merge — 0 if they ordered today, otherwise today - last_active_date. The outer query then uses COUNT(...) FILTER (WHERE days_since_active <= N) to bucket users into L1/L7/L28 without three separate scans. The windows are nested (every L1 user is also L7 and L28), so the counts are cumulative by design. NEW users get days_since_active = 0 because they ordered today — that satisfies "NEW counts toward L1 by definition."

The COUNT(DISTINCT) inside FILTER — and why it matters

A customer can place several orders in one day. SUM(today_spend) already aggregates spend per user in the today_activity CTE, so each user contributes one row to growth_merge — but COUNT(DISTINCT user_id) is still the safe choice for user_count and the L-counts, because it is robust even if the CTE were later changed to keep order-grain rows. COUNT(DISTINCT ...) FILTER (...) composes cleanly in PostgreSQL: the DISTINCT applies only to rows passing the filter predicate.

Dialect notes — FILTER and date subtraction

FILTER (WHERE ...) is standard SQL, supported by PostgreSQL, SQLite 3.30+, BigQuery, and Snowflake; MySQL has no FILTER and needs COUNT(DISTINCT CASE WHEN ... THEN user_id END) instead. DATE '2026-05-06' - last_active_date returning an integer day count is PostgreSQL-specific — SQLite needs julianday() differences, MySQL DATEDIFF(), BigQuery DATE_DIFF(..., DAY). The FULL OUTER JOIN is the structural heart and is itself unsupported in MySQL (use LEFT ∪ RIGHT there).

What a senior says next

"Net revenue momentum is what the CFO actually wants on top of this: (NEW + RESURRECTED revenue) is fresh inflow, CHURNED is the revenue you stopped earning. I would add a final summary row computing inflow minus the churned segment's prior daily run-rate, so the report shows whether the business grew or shrank in dollar terms, not just headcount."

Senior signal: return ARPU as NULL (not 0) for the non-transacting CHURNED bucket and explain the "undefined vs measured-zero" distinction, pre-compute days_since_active once then bucket with COUNT(DISTINCT ...) FILTER, note that NEW users land in L1 because their gap is 0, and propose a dollar-denominated net-momentum line for the CFO.
