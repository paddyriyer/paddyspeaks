An SCD Type 2 customer dimension is the canonical slowly-changing dimension question. The interviewer is probing whether you can distinguish a surrogate key from a natural key, structure a validity window correctly, use a current-flag efficiently, and add a row-hash to make change detection cheap during ETL. Getting the effective-to sentinel value right (“9999-12-31 or NULL?”) and providing a partial unique index for the current row are the details that separate a competent answer from an excellent one.

1. The requirements — what the design must capture

  • Grain: one row per version of a customer — a customer with three address changes has three rows.
  • Surrogate key: a warehouse-generated integer PK that fact tables reference; must not change when customer attributes change.
  • Natural key: the source system’s customer identifier, stable across versions.
  • Tracked attributes: name, email, country, tier, signup date — any change to these triggers a new version row.
  • Validity window: effective_from / effective_to columns to allow point-in-time joins from fact tables.
  • Current flag: is_current boolean for fast “latest version” lookups without date arithmetic.
  • Row hash: a hash of tracked attributes to make ETL change detection O(1) per row.
  • Audit column: ingested_at for pipeline lineage.
  • Constraint: exactly one current row per natural key, enforced by a partial unique index.

2. The DDL — the CREATE TABLE

CREATE TABLE dim_customer (
  customer_sk     BIGINT      NOT NULL,            -- surrogate PK
  customer_nk     VARCHAR(64) NOT NULL,            -- natural key from source
  full_name       VARCHAR(200),
  email           VARCHAR(200),
  country         VARCHAR(2),
  tier            VARCHAR(20),
  signup_date     DATE,
  effective_from  TIMESTAMP NOT NULL,
  effective_to    TIMESTAMP,                       -- NULL or 9999-12-31 for current row
  is_current      BOOLEAN   NOT NULL DEFAULT TRUE,
  row_hash        CHAR(64),                        -- to detect attribute change
  ingested_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (customer_sk)
);

CREATE UNIQUE INDEX ux_dim_customer_current
  ON dim_customer (customer_nk)
  WHERE is_current = TRUE;

4. Strategic explanations

Grain and versioning mechanics

Every time a tracked attribute changes, the ETL: (1) closes the current row by setting effective_to = new_effective_from and is_current = FALSE; (2) inserts a new row with a fresh customer_sk, effective_from = now(), effective_to = NULL, and is_current = TRUE. Fact tables already joined to the old SK are unaffected — historical accuracy is preserved automatically.

Surrogate vs. natural key — why both are needed

The natural key (customer_nk) is stable and meaningful but not unique across versions. The surrogate key (customer_sk) is unique per version row and is what fact tables reference. Fact rows are pinned to the SK that was current at the time of the event, giving correct historical analysis: “revenue for this customer while they were a Gold member.” Without surrogates, point-in-time accuracy requires every fact query to do an expensive range join on effective_from/effective_to.

effective_to: NULL vs. 9999-12-31 sentinel

Both conventions are widely used. NULL is semantically precise (“no end date yet”) but requires COALESCE in every range predicate. The sentinel 9999-12-31 allows BETWEEN directly but wastes four bytes per row and is an artificial value. A practical rule: use NULL in Postgres (COALESCE is cheap) and the sentinel in Redshift / older engines that handle NULL comparison poorly in sort keys. Whatever you choose, document it and enforce it consistently.

Partial unique index — enforcing the one-current-row invariant

CREATE UNIQUE INDEX ux_dim_customer_current ON dim_customer (customer_nk) WHERE is_current = TRUE prevents two current rows for the same natural key at the database level, not just at the ETL layer. This is a Postgres-specific feature (partial indexes are not universally supported); in data warehouses like Snowflake or BigQuery that do not enforce unique constraints, the invariant must be guarded by ETL logic and tested with data-quality checks.

row_hash — efficient change detection

The ETL pipeline computes a hash (e.g., SHA-256) of all tracked attribute values for each incoming source row and compares it to the stored row_hash. If equal, no action is needed; if different, a new version is inserted. This collapses a multi-column comparison into a single string equality check, making the ETL pattern scale to wide customer records without repeated CASE expressions.

Warehouse dialect notes

In BigQuery there is no partial index; enforce the invariant with a scheduled data-quality assertion. In Snowflake, cluster the dimension on (customer_nk, is_current) to accelerate both point-in-time and current-row lookups. In Redshift, use SORTKEY(customer_nk, effective_from) and DISTKEY(customer_nk) to co-locate all versions of the same customer for merge operations.

Senior signal: explain why a new surrogate key is minted for each version (historical fact accuracy), choose a clear effective_to convention and justify it, add the partial unique index to enforce the one-current-row invariant at the database level, and introduce the row-hash as the ETL change-detection mechanism — together these show you understand not just the table structure but the full load pattern that keeps it correct over time.
