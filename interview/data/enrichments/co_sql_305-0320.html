Uber's marketplace lead opens the morning huddle and the first slide is always driver utilization — the % of online time spent on a paying trip. It's the single best proxy for both driver income (high utilization = more rides per shift) and rider experience (high utilization = drivers are busy → ETAs longer). The query is two timestamp diffs and a ratio; the senior depth is what counts as "online" and how to handle multi-trip overlaps.

2. The SQL — julianday() diffs + ratio

SELECT s.driver_id,
  ROUND(SUM(julianday(s.offline_at) - julianday(s.online_at)) * 24.0, 2) AS hours_online,
  ROUND(SUM(julianday(t.ended_at)   - julianday(t.started_at)) * 24.0, 2) AS hours_on_trip,
  ROUND(100.0 * SUM(julianday(t.ended_at) - julianday(t.started_at))
              / NULLIF(SUM(julianday(s.offline_at) - julianday(s.online_at)), 0), 1) AS utilization_pct
FROM driver_shifts s
LEFT JOIN trips t ON t.driver_id = s.driver_id
GROUP BY s.driver_id
ORDER BY utilization_pct DESC;

3. Strategic explanations

julianday() — the SQLite portable date diff

julianday(date) returns the date as a fractional day count from a 4713 BC epoch. Subtracting two julian days gives a fractional day difference. Multiply by 24 for hours, 1440 for minutes, 86400 for seconds. In Postgres the equivalent is EXTRACT(EPOCH FROM (b - a)) / 86400.

The query above measures P0/Online — the strictest metric. Production dashboards usually show P0, P1, P2 side-by-side. Marketplace ops cares about P1 (accepted but not driving = positioning inefficiency); driver income cares about P0.

The overlap pitfall

If trips ever shows overlapping intervals for the same driver (data error: the app fired two "trip start" events for one ride), summing trip durations double-counts that overlap. Two defenses:

  • Sweep-line dedup before summing — flatten trip intervals to a single non-overlapping list per driver, then sum.
  • Cap utilization at 100% as a sanity gate: LEAST(utilization_pct, 100). If many drivers exceed 100% the data pipeline has a bug.
The "shift" definition

The query treats each shift row as one online window. Real driver-status data is a stream of state changes (online → idle → busy → offline). Reconstructing shifts means pairing consecutive online-online-offline events:

WITH events AS (
  SELECT driver_id, status, status_at,
    LAG(status) OVER (PARTITION BY driver_id ORDER BY status_at) AS prev
  FROM driver_status_events
),
shifts AS (
  SELECT driver_id, status_at AS online_at,
    LEAD(status_at) OVER (PARTITION BY driver_id ORDER BY status_at) AS offline_at
  FROM events
  WHERE status = 'online' AND (prev IS NULL OR prev = 'offline')
)
Per-hour utilization — the daypart cut

Average utilization hides peak-vs-trough variation. Marketplace's actual question: "in what hours is utilization < 30%?" — those are the hours to incentivize driver supply pullback or rider demand push. Slice by hour:

GROUP BY driver_id, CAST(strftime('%H', s.online_at) AS INTEGER)
The driver-income tie-in

Drivers paid per-trip-mile have utilization correlate directly with earnings. Drivers paid per-online-hour (some markets) have utilization decouple. The query result feeds both: per-trip-mile drivers want utilization > 60%; per-hour drivers don't care.

The marketplace-balance follow-up

Pair this with the demand-side question: "during the same hour, what was the unfulfilled-request rate?" If utilization > 80% AND unfulfilled > 5%, you're under-supplied; the playbook is incentive bonuses to bring more drivers online.

Senior signal: name P0 / P1 / P2 utilization tiers, surface the overlap-double-count trap, and connect the metric to both driver income and marketplace ops decisions. The shift-reconstruction LAG/LEAD pattern is the over-the-bar finish.
