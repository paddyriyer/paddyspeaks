A loyalty platform stores each member's earned reward codes packed into a single delimited string — 'MM:CI:CD' — and the business wants total points per club. The interviewer is probing two skills: can you explode a delimited string into rows, and can you map each code to a score with a lookup. The trap is the member whose reward column is NULL or empty — a careless split makes their entire club vanish from the report.

1. The source data — club

club_idmember_idrewards
110MM:CI:CD
111CL:CM
210CO
212MM:MM:XX
313NULL

Member 12 has a repeated code (MM:MM) and an unknown code (XX) — both must be handled. Club 3's only member has NULL rewards: the edge case that decides whether club 3 appears at all. Point scale: MM/CI/CO = 0.5; CD/CL/CM = 1.

2. The SQL — split the string, score each code, sum per club

WITH cte AS (
  SELECT club_id, member_id, UNNEST(string_to_array(rewards, ':')) AS reward
  FROM club
),
point_cte AS (
  SELECT club_id, member_id, reward,
    CASE reward
      WHEN 'MM' THEN 0.5 WHEN 'CI' THEN 0.5 WHEN 'CO' THEN 0.5
      WHEN 'CD' THEN 1   WHEN 'CL' THEN 1   WHEN 'CM' THEN 1
      ELSE 0 END AS points
  FROM cte
)
SELECT club_id, SUM(points) AS total_points
FROM point_cte
GROUP BY club_id;

This solution runs, but it is subtly wrong: the question asks for each club, and club 3 silently disappears.

2b. Corrected — preserve clubs whose members have NULL rewards

WITH cte AS (
  SELECT c.club_id, c.member_id, r.reward
  FROM club c
  LEFT JOIN LATERAL UNNEST(string_to_array(c.rewards, ':')) AS r(reward) ON true
),
point_cte AS (
  SELECT club_id, member_id, reward,
    CASE reward
      WHEN 'MM' THEN 0.5 WHEN 'CI' THEN 0.5 WHEN 'CO' THEN 0.5
      WHEN 'CD' THEN 1   WHEN 'CL' THEN 1   WHEN 'CM' THEN 1
      ELSE 0 END AS points
  FROM cte
)
SELECT club_id, COALESCE(SUM(points), 0) AS total_points
FROM point_cte
GROUP BY club_id;

The original returns only clubs 1 and 2. The corrected version adds club 3 with 0 — which is what "total points by each club" demands.

4. Strategic explanations

How string_to_array + UNNEST explodes a delimited string

string_to_array('MM:CI:CD', ':') builds the Postgres array {MM,CI,CD}; UNNEST turns that array into one row per element. Together they are the Postgres idiom for "split this CSV-in-a-cell into rows". The repeated MM:MM correctly yields two rows — splitting never deduplicates, which is the desired behaviour here since each earned reward counts.

Why the original drops club 3 — the NULL/empty trap

string_to_array(NULL, ':') returns NULL, and UNNEST(NULL) produces zero rows. Because UNNEST in the SELECT list acts like an implicit cross join, a row that explodes to nothing is dropped entirely. Member 13 vanishes, and since they were club 3's only member, club 3 vanishes too. The fix is LEFT JOIN LATERAL UNNEST(...) ON true: a LEFT JOIN keeps the club row even when the set-returning function yields nothing, substituting NULL for reward. COALESCE(SUM(points), 0) then turns club 3's all-NULL group into a clean 0.

The ELSE 0 branch is doing real work

Code XX is not in the point scale. The CASE's ELSE 0 catches it so an unknown code contributes nothing rather than erroring or returning NULL. That is defensive — but a senior would flag it: silently scoring unknown codes as 0 hides data-quality problems. In production you might instead surface SUM(CASE WHEN reward NOT IN (...) THEN 1 ELSE 0 END) as an unknown_code_count and alert if it is ever non-zero.

Dialect notes — this is Postgres-specific

string_to_array/UNNEST are Postgres. The equivalents: MySQL 8 has no native split — use a recursive CTE or JSON_TABLE; SQL Server uses STRING_SPLIT(); Snowflake uses SPLIT_TO_TABLE() or SPLIT() + LATERAL FLATTEN; BigQuery uses UNNEST(SPLIT(rewards, ':')). The deeper point in any interview: storing multiple values in one delimited column violates first normal form — a member_rewards bridge table would make this query a trivial join with no string parsing.

Senior signal: name string_to_array+UNNEST as the split idiom, then immediately spot that UNNEST(NULL) yields zero rows and silently drops the whole club — fixing it with LEFT JOIN LATERAL ... ON true. Closing with "this delimited column breaks 1NF; a bridge table removes the parsing entirely" is the over-the-bar finish.
