Macroeconomic analysts ask: "For each country, show the year-over-year GDP growth rate and a 5-year rolling average — all in one query." This is a multi-window-function problem: LAG fetches the prior year's GDP to compute the growth percentage, while a bounded AVG frame smooths the time series. The interviewer is checking whether you reach for LAG correctly, handle the first-year NULL, guard against division by zero with NULLIF, and can reason about the rolling frame boundary semantics.

2. The SQL — LAG for YoY growth plus rolling 5-year average

SELECT country, year, gdp_value,
       LAG(gdp_value) OVER (PARTITION BY country ORDER BY year) AS prev_gdp,
       100.0 * (gdp_value - LAG(gdp_value) OVER (PARTITION BY country ORDER BY year))
             / NULLIF(LAG(gdp_value) OVER (PARTITION BY country ORDER BY year), 0) AS yoy_growth_pct,
       AVG(gdp_value) OVER (PARTITION BY country ORDER BY year
                            ROWS BETWEEN 4 PRECEDING AND CURRENT ROW) AS rolling_5yr_avg
FROM gdp
ORDER BY country, year;

4. Strategic explanations

LAG for period-over-period deltas

LAG(gdp_value) OVER (PARTITION BY country ORDER BY year) fetches the value from the previous row within the same country partition. On the first year for each country there is no prior row, so LAG returns NULL. The YoY calculation then produces NULL for that row, which is correct — there is no growth rate to compute. An alternative is LAG(gdp_value, 1, 0) where the third argument provides a default, but a default of 0 would produce a division-by-zero or an artificially large growth rate; NULL is the right semantic for "not applicable."

NULLIF — division-by-zero guard

NULLIF(LAG(gdp_value) ..., 0) converts a zero prior GDP to NULL, making the division produce NULL rather than a runtime error or infinity. This matters because historical GDP values can be zero (new countries, data gaps). Without the guard, a single zero prior value crashes the entire query in Postgres and SQL Server, though some engines (BigQuery, MySQL) return NULL or Inf silently. Always guard with NULLIF in financial calculations.

The rolling frame — ROWS BETWEEN 4 PRECEDING AND CURRENT ROW

This frame includes up to 5 rows: the current row plus up to 4 preceding rows within the country partition. When fewer than 5 rows exist (e.g., the first 4 years of a country's data), the frame automatically shrinks to however many rows are available — so the "rolling 5-year average" is effectively a growing average at the start of each country's history. This is the correct behaviour for a standard rolling mean. If you require exactly 5 years of data before producing a result, wrap the whole query and filter WHERE year >= MIN(year) OVER (PARTITION BY country) + 4.

ROWS vs RANGE framing

The query uses ROWS BETWEEN 4 PRECEDING AND CURRENT ROW. The alternative, RANGE BETWEEN, uses logical value distance rather than physical row distance. With RANGE and ORDER BY year, if two rows share the same year they are both treated as the "current" position — duplicate years would expand the frame unexpectedly. ROWS mode counts physical rows and is deterministic when years are unique per country. Always prefer ROWS for time-series rolling calculations where the ordering column is dense and unique per partition.

Dialect portability
  • Snowflake / Postgres / BigQuery / SQL Server: all support LAG, NULLIF, and the ROWS BETWEEN ... PRECEDING frame syntax as written.
  • MySQL: supports window functions from version 8.0; LAG and AVG OVER (ROWS BETWEEN ...) work identically.
  • Repeated LAG expression: the query calls LAG(...) three times for the same window. In Snowflake/Postgres, optimisers typically common-subexpression-eliminate this. For readability and guaranteed de-duplication, refactor with a CTE: compute LAG(gdp_value) AS prev_gdp once, then reference prev_gdp in the growth calculation.
Senior signal: articulate that LAG returns NULL for the first partition row (correct, not a bug), explain why NULLIF(..., 0) is mandatory before division, distinguish ROWS from RANGE frames for rolling time-series calculations, and offer the CTE refactor to avoid repeating the LAG expression three times.
