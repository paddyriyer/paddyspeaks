A fintech spend-insights feature shows each transaction next to a smoothed trend: "the rolling average of your last three transactions in this category." The txn table is partitioned by customer and category, and the metric is a moving window over time. The interviewer is checking whether you can frame a window function precisely — the right PARTITION BY, the right ORDER BY, and crucially the right physical frame ROWS BETWEEN 2 PRECEDING AND CURRENT ROW.

1. The source data — txn

1food2025-01-01 10:0010first food txn — only 1 row available
1food2025-01-02 10:00202 rows available
1food2025-01-03 10:0030first full 3-row window
1food2025-01-04 10:0040window slides — drops the 10
1gas2025-01-02 10:0050different category — separate window
1gas2025-01-05 10:0060only 2 gas rows total
2food2025-01-01 10:005different customer — isolated
2food2025-01-02 10:0015only 2 rows — averages over 2

The edge cases are the warm-up rows: customer 1's first food txn has no prior rows, the second has only one. The spec says "average over what's available", so those rows must not be NULL — they average 1 and 2 values respectively. Customer 1's gas rows confirm the window never bleeds across categories.

2. The SQL — framed window average

SELECT customer_id, category, ts, amount,
       ROUND(AVG(amount) OVER (
         PARTITION BY customer_id, category
         ORDER BY ts
         ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
       )::numeric, 2) AS rolling_avg_3
FROM txn
ORDER BY customer_id, category, ts;

PARTITION BY customer_id, category isolates each customer's per-category timeline. ORDER BY ts sequences it. ROWS BETWEEN 2 PRECEDING AND CURRENT ROW is a physical 3-row window — the current row plus up to two before it — and AVG means over whatever rows fall inside.

The first food row averages just itself (10). The second averages 2 rows ((10+20)/2 = 15). From the third on, the full 3-row window applies: (10+20+30)/3 = 20, then it slides to (20+30+40)/3 = 30 — the 10 has dropped out. No row is NULL: the partial windows degrade gracefully.

4. Strategic explanations

ROWS vs RANGE — the single most important choice here

The frame ROWS BETWEEN 2 PRECEDING AND CURRENT ROW counts physical rows: exactly the last three records regardless of how far apart their timestamps are. RANGE would count by value of the ordering column — and would also lump together all rows sharing the current ts. "Last 3 transactions" is a row-count concept, so ROWS is mandatory. This distinction is the crux of the question.

Why partial windows are a feature, not a bug

For the first row, "2 preceding" rows simply do not exist, so the frame contains one row and AVG averages one value. SQL frames automatically shrink at partition edges — you get a graceful warm-up, not a NULL or an error. This is exactly what "average over what's available" asks for. If the spec instead demanded NULL until three rows accumulate, you would add CASE WHEN COUNT(*) OVER (...) < 3 THEN NULL.

The default frame trap — omitting ROWS BETWEEN

If you write AVG(amount) OVER (PARTITION BY ... ORDER BY ts) with no explicit frame, Postgres applies the default RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — a cumulative average from the start of the partition, not a 3-row rolling one. The explicit ROWS BETWEEN 2 PRECEDING AND CURRENT ROW is what makes it a fixed-width moving average. Forgetting the frame is the most common rolling-average bug.

Partitioning by both keys — category isolation

PARTITION BY customer_id, category (not just customer_id) ensures customer 1's gas transactions never enter a food average. The window resets at every (customer, category) boundary. Drop category from the partition and the gas 50 would contaminate the food trend — a subtle correctness error that the demo's interleaved categories are designed to catch.

Dialect notes — framing and rounding portability

Windowed AVG with ROWS BETWEEN is standard and identical in Postgres, SQL Server, BigQuery, Snowflake, and MySQL 8+/SQLite 3.25+. The ::numeric cast before ROUND is a Postgres nicety — ROUND(double precision, int) is not allowed, so the cast forces the two-argument numeric form. Other engines (ROUND(AVG(...), 2) directly) need no cast; that line is the only piece to adjust when porting.

Senior signal: use ROWS BETWEEN 2 PRECEDING AND CURRENT ROW — never the default RANGE frame — so "last 3 transactions" means three physical rows, partition by both customer and category so windows never cross category, and recognise that partial warm-up windows are the correct graceful behaviour.
