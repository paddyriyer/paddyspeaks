A social graph wants user 1's "friends-of-friends" — people exactly two hops away who are not already direct friends and not user 1 themselves — the suggestion engine behind "people you may know". The interviewer is probing two things: handling an undirected graph stored as unordered pairs, and correctly excluding the 1-hop and 0-hop nodes that a raw 2-hop traversal would otherwise include.

1. The source data — friendship

Two edge cases. User 5 is reachable on two distinct 2-hop paths (1→2→5 and 1→3→5) — it must appear once, not twice. And the traversal naturally tries to walk back to user 1 itself (1→2→1) — that self-return has to be filtered out.

2. The SQL — symmetrise the edges, then a 2-hop self-join

WITH edges AS (
  SELECT a AS u, b AS v FROM friendship
  UNION ALL
  SELECT b, a FROM friendship
),
direct AS (
  SELECT v FROM edges WHERE u = 1
),
fof AS (
  SELECT DISTINCT e2.v AS user_id
  FROM edges e1
  JOIN edges e2 ON e1.v = e2.u
  WHERE e1.u = 1
)
SELECT user_id FROM fof
WHERE user_id <> 1
  AND user_id NOT IN (SELECT v FROM direct)
ORDER BY user_id;

edges makes the graph traversable in both directions — each stored pair is emitted as (a,b) and (b,a). fof joins two edge hops: e1 leaves user 1, e2 continues from wherever e1 landed. The outer query then removes user 1 itself and anyone in direct (the 1-hop friends).

Users 4, 5, 6 are returned. User 5 appears once despite two 2-hop paths — SELECT DISTINCT deduplicates. User 1 is dropped by user_id <> 1 (the 1→2→1 walk-back). Users 2 and 3 are dropped by NOT IN direct — they are 1-hop friends, even though a 2-hop path also reaches them (1→2→... no, but 1→3→5→... etc.). Users 7 and 8 are correctly absent: they sit three hops out.

4. Strategic explanations

Why the graph must be symmetrised first

Friendships are undirected but stored once, as an unordered pair — only (1,2) exists, not also (2,1). A join that walked solely on the stored direction could traverse 1→2 but never 2→1, missing half the graph. The edges CTE fixes this by emitting both orientations of every pair, so a join on e1.v = e2.u can move freely in any direction. Forgetting this symmetrisation is the most common bug on undirected-graph questions.

Why "exactly 2 hops" needs three exclusions

A raw two-hop traversal from user 1 collects more than friends-of-friends. It includes user 1 itself (every friend leads back: 1→2→1), so user_id <> 1 is required. It includes 1-hop friends (a friend of a friend can be a direct friend), so NOT IN (SELECT v FROM direct) strips them — "people you may know" should not re-suggest existing friends. Only after removing the 0-hop and 1-hop nodes is what remains genuinely "exactly 2 hops away, new". Each exclusion maps to a concrete product requirement.

The DISTINCT handles multi-path reachability

In any non-trivial graph a node is reachable by several paths — user 5 here via both friend 2 and friend 3. The join produces one row per path, so without DISTINCT user 5 would appear twice. SELECT DISTINCT collapses paths to nodes, which is what a suggestion list wants. (If you wanted to rank suggestions by how many mutual friends connect them, you would instead GROUP BY ... COUNT(*) — the path count becomes the mutual-friend score.)

NOT IN and the NULL trap; a recursive alternative

NOT IN has a notorious pitfall: if the subquery (direct) ever returns a NULL, NOT IN evaluates to unknown for every row and the whole result goes empty. Here direct derives from edges columns that are never NULL, so it is safe — but NOT EXISTS or a LEFT JOIN ... WHERE v IS NULL is the NULL-robust habit. Note this solution is labelled "Recursive CTEs" yet uses none — a fixed 2-hop reach is a plain double join. A recursive CTE is the right tool only for variable-depth reachability ("anyone within N hops"); for exactly 2, the explicit join is clearer and faster. Solve the question, not the tag.

Senior signal: symmetrise an undirected edge list before traversing, recognise that "exactly 2 hops" requires explicitly excluding the start node (self-return) and the 1-hop friends, DISTINCT away multi-path duplicates (or COUNT them as a mutual-friend score), and note that a fixed-depth reach is a plain join — a recursive CTE is only needed for variable-depth reachability.
