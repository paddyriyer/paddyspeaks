An HR analytics team needs an org-chart breadcrumb for every employee: "show me the chain of command from the CEO down to this person." The employee table is self-referential — each row points at its manager — so the depth of the tree is unknown at query time. The interviewer is testing whether you can write a recursive CTE that walks a hierarchy, accumulating both a depth counter and a human-readable path string as it descends.

1. The source data — employee

idnamemanager_id(narrative)
1AliceNULLthe CEO — recursion's anchor
2Bob1reports to Alice
3Cara1reports to Alice
4Dan2reports to Bob
5Eve2reports to Bob
6Finn3reports to Cara
7Gwen4deepest node — four levels down

The edge case is the root: Alice has manager_id = NULL, the single node with no parent. Gwen sits at the opposite extreme — the deepest leaf — and exercises the recursion to its full depth of three.

2. The SQL — recursive CTE accumulating depth and path

WITH RECURSIVE chain AS (
  SELECT id, name, 0 AS depth, name AS path
  FROM employee
  WHERE manager_id IS NULL
  UNION ALL
  SELECT e.id, e.name, c.depth + 1, c.path || ' > ' || e.name
  FROM employee e
  JOIN chain c ON e.manager_id = c.id
)
SELECT id, name, depth, path
FROM chain
ORDER BY depth, id;

The anchor selects the CEO — depth 0, path is just the name. The recursive member joins every employee onto rows already in chain by matching e.manager_id = c.id, incrementing depth and appending the name to the parent's path. UNION ALL stacks each generation; recursion stops naturally when no employee's manager appears in the latest batch.

Each path is built incrementally: Gwen's Alice > Bob > Dan > Gwen is her manager Dan's path with > Gwen appended. Depth equals the number of edges from the root, so the count of > separators in path always equals depth.

4. Strategic explanations

Why a recursive CTE and not a fixed set of joins

You cannot self-join the table N times because N — the org's depth — is unknown and may change as people are hired. A recursive CTE iterates: the anchor produces level 0, the recursive member consumes the previous level to produce the next, and Postgres repeats until the recursive query returns no rows. It is the only construct that walks an arbitrary-depth tree in standard SQL.

The path string is a running accumulator

Carrying path as a column means each level appends one name to its parent's already-complete string — no post-processing, no array flattening. The same column could just as easily accumulate an array (c.id_path || e.id) for programmatic use; the string form is for display. Either way, the principle is identical: state that must persist down the recursion travels as a column in the CTE.

Cycle safety — the trap recursion can fall into

If the data were corrupt — say Dan's manager pointed back to Gwen — the recursion would loop forever (or until Postgres' default depth limit aborts it). Production hierarchy queries defend against this: carry the visited ids in an array and add WHERE e.id <> ALL(c.id_path) to the recursive member, or use Postgres 14+'s CYCLE id SET is_cycle USING cycle_path clause. Mentioning cycle detection unprompted is the senior signal — clean demo data hides a real production risk.

Ordering and orphan rows

The final ORDER BY depth, id is applied outside the CTE — a recursive CTE produces rows generation by generation, not in any guaranteed order, so the sort must be external. Note also that if a non-CEO row had a manager_id pointing to a non-existent manager, that subtree would simply never be reached — it would vanish from the output. A data-quality check (every non-NULL manager_id exists in employee.id) belongs alongside this query.

Dialect notes — RECURSIVE keyword and string concat

Postgres, SQL Server, and Oracle require WITH RECURSIVE (SQL Server omits the RECURSIVE word but the mechanism is the same). MySQL 8+ supports it identically to Postgres. String concatenation || is standard SQL but SQL Server uses + and MySQL needs CONCAT() unless PIPES_AS_CONCAT is set. Oracle additionally offers the older CONNECT BY ... PRIOR proprietary syntax for the same job.

Senior signal: reach for a recursive CTE when tree depth is unbounded, carry per-path state (depth, path string) as CTE columns so it accumulates one level at a time, sort outside the recursion, and proactively raise cycle detection and orphaned-manager data-quality risks that clean sample data conceals.
