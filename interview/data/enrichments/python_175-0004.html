LFU is the eviction policy that wins when the workload has long-lived hot keys and a noisy cold tail. Redis 4.0+ ships it as maxmemory-policy allkeys-lfu precisely for this case — a session-store hot user gets pinned in RAM while one-off scrapers are flushed out. Cloudflare and Fastly use the same policy inside their CDN edge caches so the most-requested objects stay resident even when a low-traffic crawler hits a million distinct URLs in a minute. The reason it appears in every L5 interview loop: the naive implementation is O(n) per put (scan all keys for the lowest frequency) — getting both get and put to O(1) requires a non-obvious dual-index data structure, and that's the senior signal the interviewer is probing for.

1. The problem in a sentence

Build a fixed-capacity cache where get(key) returns the value (and counts as a hit), put(key, value) inserts or overwrites, and on overflow we evict the least-frequently used key — breaking ties by least-recently used within that frequency. The constraint that rules out the obvious approach: both get and put must be O(1), independent of cache size. That kills any "linear scan for the min-frequency key" implementation the moment your cache holds more than a few thousand entries.

2. Worked example with the seeded data

                                   freq=1 bucket emptied)
put(5,E)      {1@3, 4@2, 5@1}     evict 2        1        {1:[5], 2:[4], 3:[1]}
                                  (min_freq=2 LRU = 2)
get(4)→D      {1@3, 4@3, 5@1}                    1        {1:[5], 3:[1,4]}
get(5)→E      {1@3, 4@3, 5@2}                    2        {2:[5], 3:[1,4]}

The two crucial transitions to read carefully: (1) the put(4) eviction — min_freq was 1 and the LRU within freq-1 was key 3, so 3 dies; (2) the get(4) right after — that bumps 4 from 1→2, the freq-1 bucket goes empty, and min_freq advances to 2. This advance is the entire trick. Without that pointer maintenance, finding the next eviction candidate would require scanning every frequency.

3. The algorithm — naming the data structure

from collections import defaultdict, OrderedDict

class LFUCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.min_freq = 0                         # pointer to smallest live freq
        self.key_to_val  = {}                     # primary store: key -> value
        self.key_to_freq = {}                     # secondary index: key -> freq
        self.freq_to_keys = defaultdict(OrderedDict)
        # ^ tertiary index: freq -> insertion-ordered keys at that freq.
        #   OrderedDict gives O(1) move-to-back AND O(1) pop-from-front,
        #   which is what implements the LRU tie-break within a frequency.

    def _bump(self, key):
        # Promote key from freq f to freq f+1.
        f = self.key_to_freq[key]
        del self.freq_to_keys[f][key]
        if not self.freq_to_keys[f]:              # bucket emptied
            del self.freq_to_keys[f]
            if self.min_freq == f:                # we just emptied the min bucket
                self.min_freq += 1                # advance the pointer
        self.key_to_freq[key] = f + 1
        self.freq_to_keys[f + 1][key] = None      # appended at back (most-recent at freq f+1)

    def get(self, key):
        if key not in self.key_to_val: return -1
        self._bump(key)                           # a hit counts as a use
        return self.key_to_val[key]

    def put(self, key, value):
        if self.cap <= 0: return
        if key in self.key_to_val:                # overwrite path
            self.key_to_val[key] = value
            self._bump(key)
            return
        if len(self.key_to_val) >= self.cap:      # eviction path
            evict, _ = self.freq_to_keys[self.min_freq].popitem(last=False)
            del self.key_to_val[evict]; del self.key_to_freq[evict]
        self.key_to_val[key]  = value             # insert path
        self.key_to_freq[key] = 1
        self.freq_to_keys[1][key] = None
        self.min_freq = 1                         # new key always resets the pointer

Three indexes maintained in lockstep: key→value is the cache itself; key→freq lets us look up "what bucket is this key in" in O(1); freq→OrderedDict[key] gives us both "all keys at freq f" and "LRU within freq f" in O(1). The min_freq pointer is the glue — without it, eviction would require iterating freq_to_keys.keys() to find the smallest, which Python's dict doesn't sort.

4. Complexity

Time: O(1) amortised for both get and put. Each operation does a constant number of dict lookups, one OrderedDict move (which is O(1) thanks to the underlying doubly-linked list), and at most one increment of min_freq. Space: O(capacity) — each live key has exactly one entry in each of the three indexes; the freq_to_keys dict holds at most capacity distinct frequencies. The "trick that gets you there" is the min_freq pointer: it converts an "find argmin over all frequencies" operation into a single integer increment, but only because new keys always reset it to 1 and _bump can only advance it when its own bucket empties.

5. The interviewer's edge-case probes

  • capacity = 0 — every put is a no-op, every get returns -1. The if self.cap <= 0: return guard is non-negotiable; without it, the eviction path will KeyError on the empty freq_to_keys[self.min_freq].
  • Tie-break when two keys share the same freq — LRU within the bucket. The OrderedDict preserves insertion order; popitem(last=False) pops the front (oldest). When _bump reinserts at freq f+1, it goes to the back (newest at that freq).
  • put on a full cache where the key already exists — must NOT evict. This is the trap: a junior implementation checks "len >= cap" before checking "key in cache" and evicts a different key on every overwrite. The order matters: existence-check first, evict-on-overflow second.
  • get on a non-existent key — must NOT bump anything. Return -1 immediately.
  • min_freq after eviction — eviction removes one key from the min_freq bucket but the bucket may still be non-empty (other keys at the same low freq). Don't advance min_freq after eviction; only _bump advances it, and only when its own source bucket becomes empty.
  • Repeated put with the same key/value — still counts as a use. Each call bumps the frequency. This is how the LFU "warms up" hot keys against a flood of cold ones.

6. Real-world appearance — what runs this in production

Redis — allkeys-lfu / volatile-lfu (since 4.0, 2017)

Redis's LFU implementation is an approximation — it uses a logarithmic counter per key (8 bits in the object header) plus a decay rate, rather than the exact-freq-bucket structure above. The approximation is what lets it fit the counter in the existing object header without growing every entry by a pointer. The semantic is identical: hottest keys stay resident, one-off scans don't pollute. Pinterest's session store and Twitter's home-timeline cache both run this policy.

CDN edge caches — Cloudflare, Fastly, Akamai

Cloudflare's edge cache documents an "LFU-with-aging" policy: a 30-second sliding-window decay on the frequency counter prevents long-dead-popular objects from squatting in cache forever. Fastly's VCL exposes obj.cacheable + frequency hints so origin teams can pin known hot objects. The same approximation idea (counter-with-decay) is what makes it tractable at 100M+ object scales — you can't afford an OrderedDict per frequency at that size.

Database buffer pool managers — PostgreSQL clock-sweep, MySQL InnoDB

InnoDB's buffer pool uses a variant called LRU-K (effectively K=2) that approximates LFU by tracking the K-th most recent access timestamp per page. PostgreSQL's shared_buffers uses a clock-sweep that decrements a usage counter each pass — same idea, different mechanism. Both exist because pure LRU got crushed by sequential-scan queries that touched every page once and evicted all the genuinely-hot ones.

GPU shader cache — NVIDIA, AMD driver-level

NVIDIA's PSO (pipeline state object) cache evicts shaders using LFU because game workloads have a steady-state of "always-on" shaders (UI, sky, common materials) and a noisy tail of one-off effects. A pure-LRU shader cache thrashes during cutscenes; an LFU cache keeps the steady-state shaders resident.

The ML feature store — Tecton, Feast, Uber Michelangelo

Feature-store online serving caches use LFU because some features (a user's signup country) are hit on every request while others (last-rare-event timestamp) are hit once. Tecton's docs cite "LFU with bounded TTL" — a blend that prevents stale features from squatting forever even if they're frequently accessed.
