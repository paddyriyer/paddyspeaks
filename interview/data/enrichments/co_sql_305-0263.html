Discord's content team wants a "readers who downloaded this also downloaded…" recommendation for its e-book library. The raw signal is co-occurrence: how often two books were downloaded by the same user. The classic answer is a self-join on downloads — pairing a user's books with each other. The interview is about the join condition: get it wrong and you get duplicate pairs, mirrored pairs, or a book paired with itself.

User 1 downloaded all three books; users 2 and 3 downloaded two each; user 4 downloaded just one. The single-book user 4 is the edge case — they form no pair and must contribute nothing.

2. The SQL — self-join with a.book_id < b.book_id

SELECT a.book_id AS b1, b.book_id AS b2, COUNT(*) AS pair_count
FROM downloads a JOIN downloads b ON a.user_id = b.user_id AND a.book_id < b.book_id
GROUP BY a.book_id, b.book_id;

The table is joined to itself on user_id, so it pairs books that share a downloader. The crucial clause is a.book_id < b.book_id: it keeps each pair exactly once and rules out self-pairs.

4. Strategic explanations

Why a.book_id < b.book_id — the heart of the question

Three things go wrong if the condition is loose:

ON a.user_id = b.user_id                       -- pairs (10,10), (10,20), (20,10) ...
ON a.user_id = b.user_id AND a.book_id != b.book_id   -- (10,20) AND (20,10): mirror dupes
ON a.user_id = b.user_id AND a.book_id < b.book_id    -- (10,20) only: correct

With no book condition, a book joins to itself (self-pair). With !=, every pair appears twice in mirror order — (10,20) and (20,10) — doubling every count. The strict < imposes a canonical order: smaller id is always b1, so each unordered pair is counted exactly once. This is the single most important line in the query.

The interviewer's follow-up: rank the recommendations

A raw pair list isn't a recommendation. Order it, and for "given book X, what to recommend" filter and sort:

SELECT a.book_id AS b1, b.book_id AS b2, COUNT(*) AS pair_count
FROM downloads a JOIN downloads b
  ON a.user_id = b.user_id AND a.book_id < b.book_id
GROUP BY a.book_id, b.book_id
ORDER BY pair_count DESC, b1, b2;
Co-occurrence count vs similarity — the popularity bias trap

Raw pair_count rewards popular books: a bestseller co-occurs with everything simply because everyone downloads it, not because it is genuinely related. Senior answers normalise — e.g. the Jaccard index, shared_users / users_of_either_book, or cosine similarity. The raw count is a fine first pass, but naming its popularity bias shows you understand what a recommendation actually needs.

The duplicate-download trap

The query assumes one row per (user, book). If a user can download the same book twice, those duplicate rows inflate COUNT(*) — a single user could contribute 4 to a pair's count from 2×2 download rows. If downloads is an event log rather than a unique set, de-duplicate first:

WITH d AS (SELECT DISTINCT user_id, book_id FROM downloads)
SELECT a.book_id AS b1, b.book_id AS b2, COUNT(*) AS pair_count
FROM d a JOIN d b ON a.user_id = b.user_id AND a.book_id < b.book_id
GROUP BY a.book_id, b.book_id;
Edge cases: single-book users and scale

A user with one book (user 4) self-joins only to their own row, which a.book_id < b.book_id rejects — they correctly contribute nothing, no special handling needed. At scale, beware: a "power user" with k downloads generates k(k−1)/2 join rows on their own; a user with 1,000 books explodes to ~500,000 pairs. On a real catalogue you'd cap downloads per user, or pre-aggregate, before the self-join.

Senior signal: explain that a.book_id < b.book_id simultaneously kills self-pairs and mirror-duplicates by imposing a canonical pair order, recognise that raw co-occurrence counts are biased toward popular books and name a normalised similarity metric, and guard against duplicate download rows with SELECT DISTINCT before the self-join.
