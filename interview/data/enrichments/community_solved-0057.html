A community retail dataset logs purchases in order and asks which items were bought three or more times consecutively. It is the same streak-detection problem as the "consecutive numbers" classic. The interviewer is probing whether you can isolate genuine runs of length 3+ — and the proposed solution is a textbook example of a query that looks clever but quietly returns the wrong answer.

1. The source data — items_table

iditem(narrative)
1Penlone Pen
2Erasersingle purchase, Pen on both sides

4–6Applerun of 3 — qualifies
7Mangolone Mango
8–10Bananarun of 3 — qualifies

12–14Appleanother run of 3 — Apple again
15Grapelone Grape

Eraser at id 2 is the trap: it is bought once, but its neighbours (Pen at id 1 and Pen at id 3) are identical to each other. Any correct answer returns only Apple and Banana.

2. The SQL — original (flagged broken) and corrected

Original solution — produces a wrong result:

with t1 as(
select id,item,
    lag(item) over(order by id asc) lag_item,
    lead(item) over(order by id asc) lead_item
from items_table
)
select distinct item
from t1
where lag_item = lead_item;

The flaw: lag_item = lead_item tests whether the rows before and after match each other — it never checks that they equal the current row. For Eraser (id 2), lag = Pen and lead = Pen, so Pen = Pen is true and Eraser is wrongly selected despite appearing only once. The condition detects "same item three rows apart", not "three consecutive identical rows".

Corrected solution — gaps-and-islands:

WITH ordered AS (
    SELECT id, item,
           LAG(item) OVER (ORDER BY id) AS prev_item
    FROM items_table
),
flagged AS (
    SELECT id, item,
           CASE WHEN item = prev_item THEN 0 ELSE 1 END AS is_break
    FROM ordered
),
grouped AS (
    SELECT id, item,
           SUM(is_break) OVER (ORDER BY id) AS group_id
    FROM flagged
)
SELECT DISTINCT item
FROM grouped
GROUP BY item, group_id
HAVING COUNT(*) >= 3
ORDER BY item;

3. The resulting analysis

Corrected query output:

itemcommentary
Applerun at id 4–6 (and a second run at 12–14)
Bananarun at id 8–10

Only Apple and Banana — the two items with a true run of 3+. The broken original would have additionally returned Eraser (and missed nothing only by luck on this data), which is plainly wrong: Eraser was bought once.

4. Strategic explanations

Why the original is wrong, precisely

"Three in a row" means rows i, i+1, i+2 all carry the same item. The correct test at the middle row is prev_item = item AND item = next_item. The original drops the item term and keeps only prev_item = next_item — satisfied whenever the item two rows back equals the item two rows forward, even when the middle row is something else entirely. It is a different predicate that coincides with the right answer only by accident.

The gaps-and-islands fix

LAG exposes the previous item; is_break flags 1 wherever the item changes; a running SUM of breaks gives every unbroken run its own group_id. Grouping by (item, group_id) isolates each run and HAVING COUNT(*) >= 3 keeps the long ones. This generalises — change the literal 3 for any streak length — and correctly treats Apple's two separate runs as distinct islands.

The minimal fix to the original idiom

If you must keep the LAG/LEAD shape, the correct middle-of-run test is:

SELECT DISTINCT item FROM t1
WHERE item = lag_item AND item = lead_item;

This flags the centre row of every 3-run, so it answers "appears in a run of at least 3". But it cannot count run length beyond 3 — for "exactly 3" or "5+" you still need the island grouping. Showing both the quick patch and the general solution is the senior move.

Boundary and NULL behaviour

The first and last rows have a NULL prev_item/next_item; in the corrected query the leading NULL makes is_break = 1, which is right — a run always starts at row 1. A NULL item mid-stream also counts as a break since NULL = NULL is unknown; if consecutive NULLs should form a run, use item IS NOT DISTINCT FROM prev_item. The original query is doubly fragile at boundaries: a NULL lag or lead makes the equality unknown, so edge rows are never selected regardless of the data.

Senior signal: spot that lag_item = lead_item compares the neighbours to each other but never to the current row — so a single purchase flanked by a matching pair is falsely flagged. Replace it with gaps-and-islands (LAG + break flag + running sum + HAVING COUNT(*) >= 3), which is correct and generalises to any run length.
