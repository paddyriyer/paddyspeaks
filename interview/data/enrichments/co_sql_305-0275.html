Samsung's catalogue spans phones, books, and shirts in one products table, and every category cares about different attributes — screen size, author, fabric size. The team stores those varying attributes in a single JSONB column so they don't have to alter the schema every time merchandising adds a category. The interview question: can you read a column whose shape depends on a sibling column, without writing a different query per category?

1. The source data — products

product_idcategoryattribute_json
1phone{"screen_size": 6.1, "ram_gb": 8, …}
2phone{"screen_size": 6.7, "ram_gb": 12, …}
3book{"author": "Ursula K. Le Guin", "pages": 254, …}
4book{"author": "N.K. Jemisin", "pages": 449, …}
5shirt{"size": "M", "color": "navy", …}
6shirt{"size": "L", "color": "olive", …}

The interesting part is that no two categories share a key. A phone has screen_size; a book has author; there is no universal "primary attribute" key. The shape of the document is a function of category — that is the "dynamic" in the title.

2. The SQL — CASE over ->>

SELECT
    product_id,
    category,
    CASE category
        WHEN 'phone' THEN attribute_json->>'screen_size'
        WHEN 'book'  THEN attribute_json->>'author'
        WHEN 'shirt' THEN attribute_json->>'size'
    END AS primary_attr,
    CASE category
        WHEN 'phone' THEN attribute_json->>'ram_gb'
        WHEN 'book'  THEN attribute_json->>'pages'
        WHEN 'shirt' THEN attribute_json->>'color'
    END AS secondary_attr
FROM products
ORDER BY category, product_id;

The CASE category picks which JSON key to extract per row, simulating a query whose column list changes by category — all in static SQL.

Every row resolves to its category's relevant pair. Note the column type: primary_attr holds a number for phones (6.1), a string for books, a size code for shirts — it is text for all of them, because that is the only type that can hold all three.

4. Strategic explanations

Why ->> and not ->

Postgres has two JSON accessors. -> returns a jsonb value (a quoted JSON scalar: the string "navy" with the quotes); ->> returns the value as plain text (navy, no quotes). A CASE expression must have one return type for all branches, and ->> gives every branch a clean text result. Mixing -> and ->> across branches, or trying to return raw numbers from one branch and strings from another, is a type-mismatch error.

The missing-key case returns NULL, not an error

attribute_json->>'screen_size' on a book row would return NULL — JSON extraction of an absent key is quietly NULL. That is why the CASE matters: it only ever asks for a key the category is known to have. But a category with no WHEN branch (say a new laptop) falls through to an implicit ELSE NULL. If new categories are expected, add an explicit ELSE or the rows go silently blank.

The schema lists columns that don't exist here

The signature mentions name, price, sku, manufacturer — the broader product table — but the JSON pattern only needs category and attribute_json. Don't be thrown: the question is specifically about reading semi-structured data, so solve that and ignore the unused columns.

A more scalable alternative — an attribute config table

A wall of CASEs grows every time a category appears. The data-driven version stores the mapping in a table and joins to it, so adding a category is an INSERT, not a code change:

-- attr_config(category, slot, json_key)
SELECT p.product_id, p.category,
       p.attribute_json->>ac.json_key AS attr_value, ac.slot
FROM products p
JOIN attr_config ac ON ac.category = p.category;

This returns one row per attribute (long form) rather than columns; pivot with FILTER if you need the wide shape. Mention this trade-off — static CASE is fine for three stable categories, the config table wins once categories churn.

Casting back to a usable type

Because everything comes out as text, sorting primary_attr would sort '12' before '8' lexically. If a downstream consumer needs numeric phone RAM, cast inside that branch: (attribute_json->>'ram_gb')::int — but then that branch no longer type-matches the string branches, so you would split into per-category queries or keep the numeric work outside the unified CASE.

Senior signal: use ->> for a uniform text result so the CASE branches type-check, know that missing JSON keys return NULL (so an unhandled category goes silently blank), and recognise that a CASE ladder is fine for a few fixed categories but a config-table join is the maintainable pattern once categories proliferate.
