An HR data-quality audit asks a pointed question: "which employees report to a supervisor whose ID does not actually exist in the table?" These are dangling references — broken org-chart links. The query self-joins employees to itself and uses an anti-join to surface the orphans. The interviewer is probing whether you can model a hierarchy with a self-join and separate "no supervisor" from "bad supervisor."

1. The source data — employees

Two distinct cases to keep apart: Alice's NULL (legitimately no boss) and Eve/Frank/Hank pointing at IDs 99/88/77 that have no matching row (genuine corruption).

2. The SQL — self anti-join

SELECT e.emp_id, e.name
FROM employees e
LEFT JOIN employees s ON e.supervisor_id = s.emp_id
WHERE e.supervisor_id IS NOT NULL
  AND s.emp_id IS NULL;

Exactly the three broken links. Alice is correctly excluded — her supervisor_id is NULL, which the IS NOT NULL guard filters out before the orphan test. Valid reports (Bob, Carol, Dave, Grace) match a real row and are dropped by the IS NULL condition.

4. Strategic explanations

Why a self-join models the hierarchy

employees holds both the "child" and the "parent" of every reporting edge — the parent is just another row. Joining the table to itself under two aliases (e for the employee, s for the supervisor) lets one query reach from a worker to their manager. ON e.supervisor_id = s.emp_id walks one level up the org chart.

Why LEFT JOIN + IS NULL finds the orphans

A plain inner join would silently drop Eve, Frank, and Hank — the very rows we want — because their supervisor has no match. The LEFT JOIN keeps every e row and fills s.* with NULL when no supervisor matches. WHERE s.emp_id IS NULL then isolates exactly the unmatched rows. This is the anti-join: use a left join to find what is missing, not what is present.

The critical guard — e.supervisor_id IS NOT NULL

Without that condition, Alice would be flagged: her supervisor_id is NULL, the join finds no match, and s.emp_id is NULL — she would look like a broken link. But Alice is correct; the CEO has no boss. The guard separates "no supervisor" (NULL, legitimate) from "supervisor that does not exist" (non-NULL pointing nowhere, the actual defect). Distinguishing those two NULL-producing cases is the senior insight.

The NULL-test column must be a key

The IS NULL check uses s.emp_id — the primary key. A real employee can never have a NULL emp_id, so a NULL there unambiguously means "the join failed." Testing a nullable column (say s.manager_id) would also catch supervisors who simply have no manager, corrupting the result. Always anti-join on the join target's primary key.

Equivalent form and the production fix

NOT EXISTS expresses the same intent: WHERE e.supervisor_id IS NOT NULL AND NOT EXISTS (SELECT 1 FROM employees s WHERE s.emp_id = e.supervisor_id). (Avoid NOT IN — a NULL among the emp_id list would make it return nothing.) Long term, the real cure is a self-referencing foreign key on supervisor_id REFERENCES employees(emp_id), which makes IDs 99/88/77 impossible to insert in the first place. Naming that as the root-cause fix elevates the answer.

Senior signal: model the org chart with an aliased self anti-join, guard with supervisor_id IS NOT NULL so the bossless CEO is not mistaken for corruption, anti-join on the primary key, and recommend a self-referencing foreign key to prevent dangling references entirely.
