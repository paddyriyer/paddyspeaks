The first principle of debugging a distributed system: every event for the same user request must be findable in one place. Atlassian's Jira backend fans a single API call out to ~8 microservices (gateway, auth, issue, search, db, notification, indexer, audit). When the P99 latency dashboard blinks red, the on-call engineer needs to filter the firehose to "show me everything that happened for the slow request" — and the only way that works at 10⁴ requests/second is a propagated request_id (trace ID) that every service stamps into every log line. This question is the offline-twin of that propagation pattern; the production version is OpenTelemetry + Jaeger / Tempo.

1. The data — structured logs from a fan-out call

Three Jira-style traces in the demo, each with the canonical shape:

traceservices touchedshape
REQ-Agateway → auth → issue → db → gatewayHappy path, 120 ms
REQ-Bgateway → auth → search (ERROR) → search (retry) → gatewaySearch service failed, retried, 310 ms total
REQ-Cgateway → issue → db (slow) → gatewayLatency outlier, 1520 ms — db slow query

The exit row at each service has level=INFO or level=ERROR. The order matters: a trace where the gateway logs "out" before the downstream service logs "in" indicates a missing trace propagation. There's also one orphan log line — no request_id — which must be skipped gracefully (a system component, not a user request).

2. The business question — what the SRE / on-call engineer decides on

  • Root-cause for a latency alert. P99 latency = 1500 ms — the dashboard tells you the symptom, the per-trace breakdown tells you it's the db slow query in REQ-C, not a global problem.
  • Error budget burn analysis. "Of all error-containing traces in the last hour, what fraction had the search service in the path?" Filter the per-trace table; aggregate.
  • Deploy regression detection. Compare per-trace duration P95 before/after a deploy. If REQ-B-shaped traces (the search-retry flow) suddenly take 2× as long, the search service's new release broke something.
  • Customer support. "Customer X reports a slow Jira load at 10:32 UTC." Filter traces by customer cookie → find the matching request_id → pull the full trace → identify which service was slow. The whole RCA happens in this table.

3. The pandas implementation — annotated

def correlate_traces(log_records):
    # ── Step 1. Group by request_id. defaultdict(list) avoids the
    # "did we initialise this key yet?" boilerplate. Skip rows without
    # request_id — those are system-component logs (health probes,
    # internal heartbeats), not user-request events.
    traces = defaultdict(list)
    for ev in log_records:
        if 'request_id' not in ev:
            continue
        traces[ev['request_id']].append(ev)

    # ── Step 2. Per-trace summary. Sort each group by ts so the
    # service-call order is canonical (events from different services
    # arrive at the log aggregator in undefined order).
    rows = []
    for req_id, events in traces.items():
        events.sort(key=lambda e: e['ts'])
        start = pd.Timestamp(events[0]['ts'])
        end   = pd.Timestamp(events[-1]['ts'])
        rows.append({
            'request_id':    req_id,
            # Service hops, in canonical order — for the UI tree view.
            'services':      [e['service'] for e in events],
            # Distinct services — denser proxy for "blast radius".
            'service_count': len({e['service'] for e in events}),
            # End-to-end latency. Use the first and last event's
            # timestamps, not first + sum-of-deltas (gaps are real).
            'duration_ms':   (end - start).total_seconds() * 1000,
            # Any-error flag for filtering; error count for severity.
            'errors':        sum(1 for e in events if e.get('level') == 'ERROR'),
            'event_count':   len(events),
            'has_error':     any(e.get('level') == 'ERROR' for e in events),
        })
    return pd.DataFrame(rows)

Why defaultdict(list) and not df.groupby('request_id').agg(...)? Both work. The dict version handles the "list of services" output cleanly — pandas' agg can do list as an aggregator but the syntax is awkward (groupby('request_id')['service'].apply(list)). For 10 M log lines, the groupby form is faster (C kernel for the count/sum aggs); for an interview, the dict form is more readable and matches how production Python parsers actually do it.

Why two ts timestamps (start + end) and not sum(per-hop deltas)? The deltas don't capture wall-clock latency — if service A logged "out" at t=100 and service B logged "in" at t=105 (5 ms RPC), the gap is real, not noise. last_ts - first_ts is the correct end-to-end latency. Sum-of-deltas would understate by all the inter-service gaps.

Why .get('level') not ['level']? Resilient to missing fields. A log line without a level field shouldn't crash the pipeline; treating it as not-an-error is a reasonable default. The .get('level') == 'ERROR' idiom is the cheap defence.

The orphan log line (no request_id) is correctly absent from the table — it would have been a junk row with empty services and zero duration. The skip-on-missing-key defence keeps that out.

5. Production patterns — how this scales

  • pandas → 100k traces comfortable. One hour of Jira backend logs = ~10 M log lines = ~1 M traces. pandas handles a hour-window for forensic analysis but not real-time alerting. The production pattern: structured logs in JSON → Fluentd → Kafka → Spark Streaming aggregating per-request-id state → write to Elasticsearch / OpenSearch for query. The pandas function is the analyst's tool for "give me the breakdown of yesterday's outage between 14:00 and 14:30".
  • OpenTelemetry & Jaeger / Tempo / Honeycomb are the productionised version. Same data model: trace_id = the request_id, span_id per service hop, parent_span_id for the call tree. The OTel collector does the per-trace aggregation that this pandas function does — but on streaming data with sampling.
  • Tail-based sampling. Sampling 1 % of all traces randomly is cheap but misses the rare slow ones. Tail-based sampling: collect all spans for every trace in a short buffer (60s), then keep the trace if it's slow or error-y, else drop. The pandas function shape — group-then-decide — is exactly tail-based sampling's logic, just in batch.
  • Atlassian's own incident tooling (Statuspage, Opsgenie + post-incident review) uses per-trace correlation as the foundational data structure. The on-call engineer searches by request_id, gets the full call tree, can pinpoint the failing service in under a minute. Without trace propagation, the same MTTR is 30+ minutes of grep across N service stdouts.
  • Hot trace. A misbehaving client that hammers the same endpoint with the same request_id (a bug — request_id should be unique per request) creates a "trace" with 10k events. The groupby sends them all to one reducer/dict-key. Defence: cap per-trace event count, alert when exceeded, treat as a separate "abusive client" class.

6. Interviewer probes

Probe 1 — "What if the clocks on different services aren't synchronised?"

Clock skew is real — even with NTP, different hosts can drift by 10-100 ms. The last_ts - first_ts latency is approximate, not exact. Production trace systems use a single clock (the gateway's, propagated as start_time) and span-relative offsets, so the math is consistent within a trace. OpenTelemetry's monotonic-clock spans are the fix; the pandas function inherits whatever clocks the input logs were written with. The senior candidate names the gotcha.

Probe 2 — "What's wrong with .apply() for the per-trace summary?"

The function above doesn't use .apply; it uses a Python loop over traces.items() which is roughly equivalent in cost. The vectorised version uses pandas groupby(request_id).agg({...}) with named aggregations. For 1 M log lines, the groupby version is ~5-10× faster because it stays in C. Trade-off: the list-of-services aggregation needs .apply(list) which re-enters Python — sometimes you keep the hand-rolled loop for clarity even when slightly slower.

Probe 3 — "How do you incrementalise this for streaming?"

Flink with keyed state on request_id, plus a session window (or a fixed timeout: "if no new events for this request_id in 30 s, the trace is closed"). Each event updates state; on window close, emit the per-trace summary row. The closed-trace stream feeds the alerting pipeline. Pitfall: long-running traces (a slow customer query that takes 60 s) get evicted prematurely if the timeout is too short. Use the explicit "trace end" signal (gateway's "out" log line) if available; fall back to timeout otherwise.

Probe 4 — "How do you handle missing intermediate logs — e.g., the issue service crashed before logging its 'out' event?"

The trace is incomplete. last_ts - first_ts may understate the duration (no late event to anchor against). Detection: service_count < expected_path_length flags partial traces; route to a separate "incomplete" cohort for investigation. Production: every service logs both "in" and "out"; if the "out" is missing, the service crashed mid-request. This is one of the highest-value alerting signals.

Probe 5 — "Two services with the same name (e.g., two replicas of 'auth'). How do you tell them apart?"

Include service_instance (host or pod identifier) in the log payload. The function's services list would then show "auth-pod-7", "auth-pod-12" so the ops team can correlate against the specific replica's logs. OpenTelemetry's resource.service.instance.id is the spec-compliant version. The senior candidate names the OTel convention.

Probe 6 — "Hot request_id — bot client reuses the same ID across 10k requests. Will this OOM?"

The defaultdict grows unboundedly for that one key. For 10k events / trace it's ~10 MB — still survivable in pandas. For 10 M events / trace (genuinely abusive), OOM. Defence: cap per-trace events at e.g. 1 000 with a "trace truncated" flag, emit the summary early. Same shape as backpressure in a streaming system: protect the aggregator from the abusive key. Also tag the request_id as "suspected reuse" for the security team.

Probe 7 — "How do you propagate request_id across an async boundary (a queue, a future)?"

Context propagation. The producer attaches request_id to the message headers; the consumer reads the header and stamps it into its own log context (Python's contextvars, Java's MDC, Node's async_hooks). Without explicit propagation, the async work logs without a request_id and the trace is broken. OpenTelemetry's baggage spec is exactly this: a key-value bundle carried with the trace context across all hops.
