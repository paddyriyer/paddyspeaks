The third primitive in the streaming-fraud trilogy: HOP (sliding) windows. Where customer_growth-0012 used TUMBLE for hard-aligned fixed-minute reporting and customer_growth-0013 used KeyedProcessFunction for sparse correlation, this question demands continuous rolling coverage — any 60-minute span should be evaluated, not just the clock-aligned ones. The L5 signal is naming the boundary-straddle failure mode of TUMBLE before the interviewer probes for it, defending HOP's 12× state cost, and explaining how the sink's PRIMARY KEY collapses overlapping-window dedup at the database layer.

1. The fraud pattern — what we're catching and what we're trying NOT to flag

"Serial returner" or refund fraud is one of e-commerce's fastest-growing loss categories — the National Retail Federation puts US 2023 return fraud at $101B. The signature is a single account (or a ring sharing an address) submitting an unnatural burst of return requests, typically within minutes of each other, trying to game the return-window policy before the merchant manually reviews.

PatternFraud signatureLegitimate pattern that looks similar
4+ RETURN_REQUESTED in any rolling 60 minRefund-fraud ring submitting bulk returns to game the merchant's auto-approval before reviewPower-buyer post-Christmas cleanup — household returning 5 mismatched-size apparel items in one Amazon session
3+ returns within 2 weeks, same shipping address, different orders"Drop-house" return ring — multiple stolen accounts shipping to one fraudster's address, all returned for refundSubscription-box customer — Stitch Fix / Trunk Club where 60% return rate is the business model
10+ returns in 24h, all "item not as described"Wardrobing fraud — wearing items once and returning, claiming defectReturning gift-recipient — bulk-return after receiving duplicates at a wedding/birthday

The false-positive cost is real: flagging a legitimate Stitch Fix customer cancels their loyalty discount and triggers a churn-save call. Stripe Radar exposes refund_velocity_* as Radar attributes; the merchant tunes thresholds against their conversion-vs-loss curve. Subscription-box merchants whitelist their customer cohort; fast-fashion merchants set aggressive thresholds.

3. The query — walk through window mechanics

INSERT INTO serial_returner_alerts
SELECT
  user_id,
  HOP_START(proctime, INTERVAL '5' MINUTE, INTERVAL '1' HOUR) AS window_start,
  HOP_END  (proctime, INTERVAL '5' MINUTE, INTERVAL '1' HOUR) AS window_end,
  COUNT(event_id) AS return_count
FROM stream_events
WHERE event_type = 'RETURN_REQUESTED'
GROUP BY user_id,
  HOP(proctime, INTERVAL '5' MINUTE, INTERVAL '1' HOUR)
HAVING COUNT(event_id) > 3
HOP semantics — slide vs size, the two knobs
  • size = 1 HOUR — the lookback width. "How wide is each window I'm evaluating against the threshold?"
  • slide = 5 MINUTES — how often a fresh window opens. "How granularly do I want to scan the timeline?"
  • Each event lives in size / slide = 12 overlapping windows. A single RETURN_REQUESTED is buffered into 12 different aggregate state slots until its 60 minutes expire.
  • Output cardinality: for a user with one burst, expect ~12 alert rows (one per overlapping window that contains all the returns). The PRIMARY KEY on the sink collapses repeat fires; without it, your risk DB would see 12× the rows.
Why HOP, not TUMBLE — the boundary-straddle proof
PrimitiveWhat it doesResult for user 703 (4 returns at 11:55, 12:05, 12:50, 12:55)
TUMBLE(1h)Hard-aligned at clock hour. Counts reset at :00.Window [11:00, 12:00): count = 1. Window [12:00, 13:00): count = 3. Never fires (need > 3).
HOP(slide=5min, size=1h)Continuous rolling. New window every 5 min.Window [11:55, 12:55): count = 4 → fires. Window [12:00, 13:00): count = 3 (no fire). Window [11:50, 12:50): count = 3 (no fire). The right window catches it.
HOP(slide=1min, size=1h)60 overlapping windows. Maximum coverage.~5 overlapping windows fire, all dedupe at sink PK. Higher cost, same correctness.

Rule of thumb: if the business rule says "any rolling N-period span" — HOP. If it says "in the hour of N" — TUMBLE. The serial-returner business spec is the former; HOP wins. The slide:size ratio is a cost knob — smaller slide = more state but tighter alert timing, larger slide = less state but a burst might wait up to slide minutes before any window catches it.

Why the PRIMARY KEY on the sink is the L5 detail
CREATE TABLE serial_returner_alerts (
  user_id      BIGINT,
  window_start TIMESTAMP(3),
  window_end   TIMESTAMP(3),
  return_count BIGINT,
  PRIMARY KEY (user_id, window_start) NOT ENFORCED
) WITH (
  'connector' = 'jdbc',
  ...
)

HOP emits ~12 alerts for a single 60-minute burst. Without the PK, the risk DB would have 12 rows describing the same incident — the analyst sees noise. With the PK, the upsert sink dedupes on (user_id, window_start), so each overlapping window writes its own row but a single "incident" is still represented as multiple ordered rows (one per anchor minute). Analysts query SELECT user_id, MIN(window_start), MAX(window_end), MAX(return_count) FROM serial_returner_alerts GROUP BY user_id, date_trunc('day', window_start) to roll up to "incidents per day per user". The primary-key + post-hoc rollup is the production pattern.

4. Why event-time matters — the L5 trap

The shipped solution uses PROCTIME() for brevity. Production must switch to event-time. Compare:

ScenarioEvent-timeProcessing-time (PROCTIME)
4 returns submitted 11:55 – 12:55, all arrived within secondsFires in the [11:55, 12:55) windowSame
Same 4 returns, but the e9 event was buffered in a mobile retry queue and arrived at 13:30Still fires — event_ts puts it in the 12:05 slotMisses — e9 is processed at 13:30, lands in a window an hour later, count never reaches 4 anywhere
Replay yesterday's streamDeterministic — same alertsAll events have "now" as proctime; all collapse into the current rolling hour
Auditor: "When were these returns actually submitted?"The event_ts column — what the customer saw on their screenThe processing time — meaningless to the customer

The event-time fix:

CREATE TABLE stream_events (
  event_id   STRING,
  user_id    BIGINT,
  event_type STRING,
  event_ts   TIMESTAMP(3),
  WATERMARK FOR event_ts AS event_ts - INTERVAL '30' SECOND
) WITH (
  'connector' = 'kafka',
  ...
)

-- replace HOP(proctime, ...) with HOP(event_ts, ...)

The 30-second watermark slack matches typical e-commerce P95 lateness. Mobile-heavy traffic should use 2-5 minutes. Late events past the watermark are dropped by default; production should add side-output via a TUMBLE-emit-late hint or a separate dead-letter ingestion job.

5. Where this appears in production

  • Stripe Radar — refund velocity — refund_velocity_24h is exactly this primitive, exposed as a merchant-configurable attribute. Stripe runs Flink Table API on AWS, exactly this query shape.
  • Shopify Fraud Analysis — the "refund risk" score on the Shopify admin uses HOP-style sliding refund counts per (shipping_address, customer) cohort.
  • Amazon Return Abuse — Amazon's account-suspension policy for return abuse uses overlapping windows on RETURN_INITIATED events, with thresholds that vary by tenure and prior-year spend.
  • eBay Money-Back Guarantee — buyer-side abuse detection (buyer claims item-not-received N times in 90 days) is a HOP detector with a longer window size and 1-day slide.
  • Klarna / Affirm BNPL — return-window abuse where the buyer returns the item after the BNPL settlement triggers; detected with HOP windows on (settlement_ts, return_ts) overlap.

6. Interviewer probes

Probe 1 — "What happens if events arrive 5 minutes late?"

With the 30-second watermark, anything later than 30 s past max_event_ts is dropped by default. Production answer: configure 'sink.partition-commit.policy.kind' = 'metastore,success-file' on a side-output sink, or use Flink's EmitTriggerTime.LATE on the window to emit retractions/updates when late events arrive. The HOP windows happen to be more forgiving than TUMBLE here — overlapping windows give every late event multiple chances to land in some still-open window.

Probe 2 — "How do you handle subscription-box merchants where 60% returns is normal?"

Per-merchant or per-cohort threshold. Join the stream_events stream against a small merchants table that has expected_return_rate; compute the alert threshold as baseline × 3 instead of hard-coding 3. For per-customer cohorts (Whale/Dolphin/Minnow — see customer_growth-0032), broadcast-join the cohort table and pick threshold per cohort. The same query shape with a parameterized HAVING.

Probe 3 — "Backfill — can you re-run on yesterday's stream?"

Yes, requires event-time. Take a savepoint at midnight yesterday, start the job against yesterday's Kafka offsets, run forward. The HOP windows fire deterministically against the event timestamps and watermarks — alerts emitted are byte-identical to the live run. The PRIMARY KEY on the sink makes the upsert idempotent: replaying produces the same rows.

Probe 4 — "What's the difference between TUMBLE and HOP for 'orders in last 5 min'?"

TUMBLE = hard-aligned, non-overlapping. HOP = continuous, overlapping. Each event lives in size/slide HOP windows simultaneously, so HOP uses size/slide × the state and emits size/slide × the output rows. HOP wins when the business rule says "any rolling N-minute span"; TUMBLE wins for fixed-interval reporting and dashboards. This question (HOP), customer_growth-0012 (TUMBLE), and customer_growth-0013 (no window — keyed state) cover the three primitives in the trilogy.

Probe 5 — "Hot key — one merchant generates 100K return events/sec. What breaks?"

Every event of that merchant routes to one task manager's RocksDB. HOP makes it worse than TUMBLE because each event opens 12 overlapping aggregate state slots. Standard remediation: two-level aggregation with salting — pass 1 groups by (user_id, salt) with salt = RAND_INTEGER(16), pass 2 rolls up by user_id. The Table API expresses this as nested queries; the SQL shape is identical to customer_growth-0027. Flink-specific guidance: Hot Shards & Skew design page §03.

Probe 6 — "Why a 5-minute slide instead of 1-minute?"

Trade-off: 1-minute slide gives 60 overlapping windows per hour, 5× the state and CPU of 5-minute slide. The benefit is that an alert fires within 1 minute of the threshold breach instead of within 5. For fraud detection, the alert SLA is usually "within a few minutes" — 5-minute slide is the sweet spot. If you need real-time alerting (sub-30s), drop slide to 30s or use a continuous-aggregation approach with a custom ProcessFunction. Don't pick 1-minute slide reflexively — quantify the cost.

Probe 7 — "Can you express this with the DataStream API instead of Table API?"

Yes:

events
  .filter(e -> "RETURN_REQUESTED".equals(e.eventType))
  .keyBy(e -> e.userId)
  .window(SlidingEventTimeWindows.of(Time.hours(1), Time.minutes(5)))
  .aggregate(new CountAgg(), new EmitWindowMeta())
  .filter(r -> r.count > 3);

Same primitive (SlidingEventTimeWindows = HOP). Trade-off: Table API is declarative and the optimizer can fuse with other stream joins; DataStream gives explicit control over the aggregator (incremental AggregateFunction vs buffering ProcessWindowFunction). For pure count-and-threshold, both are equivalent in performance.
