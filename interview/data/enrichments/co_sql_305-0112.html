Hulu's catalog team wants to surface its premium-priced titles — everything priced above the catalog average — for a "premium tier" merchandising shelf. The query is the canonical scalar-subquery pattern. The interview is whether you understand when that subquery runs, why the comparison cannot live in a WHERE directly as an aggregate, and what the average does in the presence of NULLs and skew.

The average price is (20 + 50 + 300 + 90 + 40) / 5 = 100. Notice the skew: the $300 monitor drags the mean up so far that only it clears the bar — the $90 webcam, above the median, still falls below the mean. That single outlier is the whole interview.

2. The SQL — scalar subquery in WHERE

SELECT * FROM products WHERE price > (SELECT AVG(price) FROM products);

The inner SELECT AVG(price) collapses the whole table to a single number; the outer query then keeps every row whose price exceeds it. The subquery is uncorrelated — it does not reference the outer row — so it is computed once.

One row clears the bar. The $90 webcam — the second-most-expensive product, above three of the five prices — is below the $100 average and excluded. That is not a bug; it is what "above average" means when one outlier inflates the mean. If a stakeholder expected "the top few", the mean is the wrong threshold — flag it.

4. Strategic explanations

Why the aggregate must be a subquery

You cannot write WHERE price > AVG(price) directly. WHERE is evaluated row-by-row, before aggregation happens — there is no group yet, so AVG is meaningless there and Postgres rejects it. The subquery isolates the aggregation into its own scope, producing a finished scalar that the row-level WHERE can then compare against. This scoping is the entire point of the pattern.

It runs once, not once per row

Because the subquery is uncorrelated, the planner evaluates AVG(price) a single time and reuses the constant for every row — an InitPlan in the query plan. Contrast a correlated subquery (one referencing the outer row), which can re-run per row and turn O(n) into O(n²). Knowing this query is not that — and reading the plan to confirm — is the senior detail.

The "above-average within category" follow-up

Interviewers almost always extend it: "above average for its category." A correlated subquery works but a window function is cleaner and single-pass:

SELECT product_id, price, name, category
FROM (
  SELECT *, AVG(price) OVER (PARTITION BY category) AS cat_avg
  FROM products
) t
WHERE price > cat_avg;

AVG(price) OVER (PARTITION BY category) attaches each category's mean to every row of that category, so the outer filter is a plain row comparison — no nested scan.

The mean-vs-median trap

Our data is the cautionary tale: one $300 monitor pulls the mean to $100, so a product priced higher than 60% of the catalog still misses the cut. If the business wants "expensive products", the median is the robust measure: WHERE price > (SELECT percentile_cont(0.5) WITHIN GROUP (ORDER BY price) FROM products). Naming this distinction — and asking which the stakeholder means — is over the bar.

NULL prices and the strict inequality

Two NULL effects. First, AVG(price) skips NULL prices entirely, so the threshold is the average of priced products only. Second, a product with a NULL price fails price > threshold (the comparison is unknown) and is excluded from the result. Usually fine — an unpriced product is not "premium" — but state it. Also: the strict > means a product priced exactly at the average is excluded; confirm whether the boundary should be ≥.

SELECT * is fine here, risky in production

SELECT * is acceptable for an exploratory answer, but for a merchandising feed it is fragile: an added column silently changes the output shape and can break a downstream consumer. Name the columns the shelf actually needs (product_id, name, price). A small point, but interviewers notice it.

Senior signal: explain that the aggregate must be a subquery because WHERE precedes aggregation, recognise the uncorrelated subquery runs once (not per row), reach for AVG(...) OVER (PARTITION BY ...) for the per-category follow-up, and warn that an outlier-inflated mean is a poor "expensive" threshold — the median is the robust choice.
