Bolt's logistics team wants a cost-efficiency view of shipments — cost per kilogram — ranked cheapest first, so the network team can spot which lanes are bleeding money. It is a division plus a sort, but the interview hinges on one defensive move: a shipment with zero weight would divide by zero, and the query has to survive that.

1. The source data — shipments

shipment_idweight_kgcost

Shipment 4 has weight 0 — a data-quality glitch (an unweighed parcel, or a documentation-only entry). A raw cost / weight_kg would throw division by zero and abort the whole query. That single row is the entire interview.

2. The SQL — cost / NULLIF(weight, 0)

SELECT shipment_id, cost / NULLIF(weight_kg,0) AS cost_per_kg
FROM shipments ORDER BY cost_per_kg;

NULLIF(weight_kg, 0) returns NULL when the weight is exactly 0, otherwise the weight unchanged. Dividing by NULL yields NULL — no error. The result is sorted ascending so the most efficient (lowest cost per kg) shipments surface first.

4. Strategic explanations

The divide-by-zero trap — the whole point of the question

Without the guard, cost / weight_kg hits shipment 4 and raises ERROR: division by zero — and SQL aborts the entire statement, so you get no rows at all, not just a bad row. NULLIF(weight_kg, 0) is the idiomatic fix: it swaps the dangerous 0 for a NULL, and division by NULL is defined to produce NULL. The query degrades gracefully — one unusable row becomes NULL, the other four are fine. Recognising that a real shipments table will contain a zero-weight row, and defending against it unprompted, is the senior instinct.

NULL result vs zero result — choose deliberately

NULLIF makes cost_per_kg NULL for the bad row, which honestly says "undefined" — you cannot compute efficiency for a zero-weight shipment. If the consumer needs a number instead, wrap it: COALESCE(cost / NULLIF(weight_kg,0), 0). But think before you do — a 0 cost-per-kg would sort to the top as if it were the most efficient shipment, which is misleading. NULL-and-sort-last is usually the more honest representation. Either way, make the choice on purpose.

The integer-division trap

If cost and weight_kg are integer columns, cost / weight_kg does integer division: shipment 2 would give 200/5 = 40 (fine), but a 100/8 case would truncate to 12, not 12.5. In this dataset the values divide evenly so the bug stays hidden — which is exactly how it reaches production. Force real division with a cast:

SELECT shipment_id, cost::numeric / NULLIF(weight_kg,0) AS cost_per_kg
FROM shipments ORDER BY cost_per_kg;

Casting one operand to numeric promotes the whole expression. Mention this even if the sample data happens to divide cleanly.

Where NULLs sort — and controlling it

Postgres places NULL last under ORDER BY ... ASC and first under DESC. Here ascending order conveniently buries shipment 4 at the bottom. But this default is engine-specific — MySQL sorts NULL first under ASC. For portable, intentional behaviour, be explicit: ORDER BY cost_per_kg ASC NULLS LAST. If the undefined rows should not appear in an efficiency ranking at all, filter them: WHERE weight_kg > 0.

Negative or NULL weights

NULLIF only neutralises exactly 0. A negative weight (bad data) would still divide and produce a nonsensical negative efficiency; a NULL weight produces NULL via normal division. For a robust pipeline, validate the input range — WHERE weight_kg > 0 — rather than relying on NULLIF to catch every species of garbage. NULLIF handles the zero; it does not handle "implausible".

Senior signal: anticipate the zero-weight row and guard the division with NULLIF so the statement does not abort, decide consciously between a NULL and a COALESCEd 0 (and note 0 mis-sorts as "most efficient"), cast to force real division even when the sample divides evenly, and make NULL ordering explicit rather than trusting the engine default.
