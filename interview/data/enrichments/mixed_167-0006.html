An operations analyst needs Q1 orders: "give me everything placed between January 1 and March 31." The query is a WHERE filter with BETWEEN. It looks like a one-liner, but the interviewer is probing whether you understand that BETWEEN is inclusive on both ends, and the subtle date-vs-timestamp bug that follows from that.

The row to watch is 2025-03-31 — the exact upper bound. Whether it is kept or dropped is the entire test.

2. The SQL — range filter with BETWEEN

SELECT *
FROM orders
WHERE order_date BETWEEN '2025-01-01' AND '2025-03-31';

3. The resulting analysis

order_datecommentary

2025-03-31kept — the upper bound is included

Five rows. 2025-03-31 survived: BETWEEN includes both endpoints. The December and April/June rows fell outside the range.

4. Strategic explanations

Why BETWEEN is inclusive on both sides

x BETWEEN a AND b is exactly equivalent to x >= a AND x <= b — a closed interval. Both endpoints qualify. That is convenient when the column is a pure DATE, as here: 2025-03-31 is one value and it is included.

The timestamp trap — the classic BETWEEN date bug

If order_date were a TIMESTAMP instead of a DATE, the literal '2025-03-31' coerces to 2025-03-31 00:00:00 — midnight. An order placed at 2025-03-31 14:30 is then after the upper bound and silently excluded. You would lose almost an entire day of orders. The robust pattern for timestamps is a half-open interval:

WHERE order_date >= '2025-01-01'
  AND order_date <  '2025-04-01'

Lower bound inclusive, upper bound exclusive on the next day. This is correct regardless of time component and is the form a senior engineer reaches for by default.

Half-open ranges compose cleanly

Beyond correctness, >= start AND < next_start tiles without gaps or overlaps: Q1's < '2025-04-01' is exactly Q2's >= '2025-04-01'. No order is double-counted or missed at a quarter boundary. Inclusive-inclusive ranges (<= '2025-03-31' then >= '2025-04-01') leave the gap of March 31's afternoon ambiguous.

Sargability — keep the index usable

WHERE order_date >= ... AND order_date < ... is sargable: an index on order_date can do a range scan. Wrapping the column in a function — WHERE EXTRACT(QUARTER FROM order_date) = 1 or DATE_TRUNC('quarter', order_date) = ... — defeats the index and forces a full scan. Filter the raw column against literal bounds.

Common mistakes

SELECT * is fine for an ad-hoc check but list explicit columns in production code. Reversed bounds (BETWEEN '2025-03-31' AND '2025-01-01') return zero rows silently — BETWEEN does not auto-swap. And string comparison only behaves like date comparison when dates are zero-padded YYYY-MM-DD; rely on a real DATE type, not lexical luck.

Senior signal: state that BETWEEN is inclusive on both endpoints, then immediately raise the timestamp pitfall — prefer the half-open >= start AND < next_start form, which is correct for timestamps, tiles without gaps, and stays sargable.
