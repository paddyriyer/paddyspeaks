Google's people-analytics team wants a compensation view that ties pay to project load: every employee, how many active projects they are staffed on, and a total comp that adds a $1,000 bonus per active project on top of base salary. It is a multi-join aggregation, and the interview hinges on three things — keeping employees with zero projects in the result, counting projects without double-counting, and putting the "active" filter in the right place.

1. The source data

employees — the people and their base salary:

Ada is on three projects but one (102) is closed — only 100 and 101 count. Ben is on one active project. Cleo is on one active project. Dan has no assignments at all — the edge case that decides whether a LEFT JOIN was used.

2. The SQL — LEFT JOIN chain with COUNT DISTINCT

SELECT e.emp_id, e.name,
  COUNT(DISTINCT p.project_id) AS active_projects,
  e.base_salary + 1000 * COUNT(DISTINCT p.project_id) AS total_comp
FROM employees e
LEFT JOIN project_assignments pa ON e.emp_id = pa.emp_id
LEFT JOIN projects p ON pa.project_id = p.project_id AND p.status='active'
GROUP BY e.emp_id, e.name, e.base_salary;

Two LEFT JOINs keep every employee even with no assignments; the p.status='active' predicate sits in the ON clause, so non-active projects join to NULL instead of filtering rows out. COUNT(DISTINCT p.project_id) counts the active projects per employee.

Ada counts 2 (project 102 closed, excluded) → 120,000 + 2×1,000 = 122,000. Ben and Cleo count 1 each. Dan counts 0 and his comp is exactly his base — he survives the result with a clean zero, which is the whole point of the LEFT JOINs.

4. Strategic explanations

The filter location decides whether Dan disappears — the central trap

p.status='active' belongs in the ON clause, not a WHERE. Move it to WHERE p.status='active' and the LEFT JOIN silently becomes an INNER JOIN: for Dan, p is all NULL, NULL = 'active' is unknown, and his row is filtered away. He vanishes from a comp report — a serious, plausible-looking bug. Rule: a predicate on the nullable side of a LEFT JOIN goes in ON; a predicate on the preserved side can go in WHERE. Be ready to recite this distinction.

Why COUNT(DISTINCT) and not COUNT(*)

For Dan, the join yields one row with all project columns NULL. COUNT(*) would count that row as 1 — wrongly crediting a project-less employee with a project and a $1,000 bonus. COUNT(p.project_id) counts non-NULL values, so Dan correctly scores 0. The DISTINCT adds a second layer of protection: if the join fans out — an employee assigned to the same project twice, or the schema's denormalised columns causing row multiplication — DISTINCT stops the same project from being counted (and bonused) more than once. COUNT(DISTINCT p.project_id) is the correct, defensive choice here.

The schema is a minefield — ambiguous column names

Look at the signature: base_salary appears in all three tables, status in all three, name in all three, emp_id and project_id scattered around. This is a deliberately denormalised, redundant schema. Every column reference must be table-qualified — e.base_salary, p.status — or the query is ambiguous and may error or, worse, silently read the wrong table's status. The solution qualifies everything; that discipline is itself a senior signal. Confirm which table is authoritative for salary and for project status before trusting either.

GROUP BY must list every non-aggregated column

The SELECT projects e.emp_id, e.name, and uses e.base_salary in an expression — so all three appear in GROUP BY. Omitting e.base_salary would be a "must appear in GROUP BY" error in Postgres. Grouping by e.emp_id alone does work if emp_id is the table's primary key (Postgres allows functionally-dependent columns to be selected), but listing all three is the portable, unambiguous habit.

Pre-aggregate to avoid join fan-out — the cleaner pattern

Joining then grouping works, but on large tables the fan-out (one row per assignment) inflates the intermediate result before the COUNT(DISTINCT) deflates it. A tidier, often faster shape aggregates projects first, then joins one clean row per employee:

SELECT e.emp_id, e.name,
       COALESCE(ap.cnt, 0) AS active_projects,
       e.base_salary + 1000 * COALESCE(ap.cnt, 0) AS total_comp
FROM employees e
LEFT JOIN (
  SELECT pa.emp_id, COUNT(DISTINCT pa.project_id) AS cnt
  FROM project_assignments pa
  JOIN projects p ON pa.project_id = p.project_id
  WHERE p.status = 'active'
  GROUP BY pa.emp_id
) ap ON e.emp_id = ap.emp_id;

Here the WHERE p.status='active' is safe — it is inside the subquery, before the outer LEFT JOIN — and COALESCE(ap.cnt, 0) turns Dan's missing row into a clean 0.

Senior signal: put p.status='active' in the ON clause so the LEFT JOIN does not collapse into an inner join and drop project-less employees, choose COUNT(DISTINCT p.project_id) so NULLs score 0 and join fan-out cannot double-count a bonus, table-qualify every column in a redundant multi-table schema, and know the pre-aggregate-then-join pattern as the scalable alternative.
