Palantir's customer-success team wants the list of customers who have ever placed an order — the "activated" accounts, separating real users from sign-ups who never bought. The reference solution uses EXISTS. It is the right instinct, but the interviewer is checking whether you know why EXISTS beats a JOIN or an IN here, and how each behaves around duplicates and NULLs.

1. The source data — customers & orders

customer_idnamehas orders?
1Anayes — 2 orders
2Benno
3Carayes — 1 order
4Danno

Ana has two orders, Cara has one, Ben and Dan have none. Ana's multiple orders are the edge case — the answer must list her exactly once, not twice.

2. The SQL — WHERE EXISTS (…)

SELECT c.customer_id FROM customers c
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id);

EXISTS is a boolean test: for each customer, does the correlated subquery return at least one row? The SELECT 1 is a convention — the projected value is irrelevant; only the presence or absence of rows matters.

3. The resulting analysis

customer_id
1
3

Ana (1) and Cara (3) — the two customers with orders. Ben and Dan are correctly excluded. Critically, Ana appears once despite having two orders: EXISTS stops at the first matching row, so it cannot duplicate the outer customer.

4. Strategic explanations

Why not a JOIN? — the duplication trap

The instinctive answer is an inner join:

SELECT c.customer_id
FROM customers c JOIN orders o ON o.customer_id = c.customer_id;

This returns three rows — Ana twice, once per order — because a join produces one row per match. You would have to bolt on DISTINCT to fix it. EXISTS never multiplies the outer row, so it needs no DISTINCT and signals intent clearly: "I want a filter, not a combination." That is the senior reason to prefer it.

EXISTS vs IN — equivalent here, not always

An IN subquery reads the same:

SELECT customer_id FROM customers
WHERE customer_id IN (SELECT customer_id FROM orders);

For this query they are equivalent and modern planners often compile them identically. The difference surfaces with NULLs: if orders.customer_id contained a NULL, NOT IN would behave catastrophically — it returns no rows at all, because x NOT IN (1, NULL) evaluates to unknown. NOT EXISTS is immune to that. So default to EXISTS/NOT EXISTS; it is the NULL-safe choice.

The inevitable follow-up: customers with no orders

Once you give the activated list, the interviewer flips it — "now the customers who have never ordered." Just negate:

SELECT c.customer_id FROM customers c
WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id);
-- returns Ben (2) and Dan (4)

The LEFT JOIN … WHERE o.customer_id IS NULL "anti-join" gives the same result, but NOT EXISTS is shorter and avoids the NOT IN/NULL landmine entirely. This negation symmetry is the main reason to learn EXISTS well.

SELECT 1 vs SELECT * — and short-circuiting

SELECT 1, SELECT *, SELECT NULL inside EXISTS are all identical — the optimizer discards the select list and only checks row existence. EXISTS also short-circuits: it can stop scanning orders the instant it finds one match for a customer, which makes it efficient when an index on orders.customer_id exists. It never needs to count or read every matching order.

Senior signal: choose EXISTS over a JOIN because it filters without multiplying rows (no DISTINCT needed), prefer EXISTS/NOT EXISTS over IN/NOT IN for NULL-safety, and reach for NOT EXISTS as the clean answer to the "never ordered" anti-join follow-up.
