A Community member needs a directory-style roll-up: for each department, collapse every employee's email into a single delimited string — the kind of output that feeds a "notify the whole team" mail merge. It is a one-line aggregate, but the interviewer is probing whether you know what the delimiter, the ordering, and a NULL email each do to the result.

1. The source data — emp_details

EMPIDGenderEmailIDDeptID
1Mcharlie@co.com10
2Falice@co.com10
3Mbob@co.com10
4Fdana@co.com20
5Meve@co.com20
6Ffrank@co.com30

Department 10 has three members not inserted in alphabetical order (charlie, alice, bob); department 30 has a single member — the edge case that proves the aggregate still emits one clean value, not a string with stray delimiters.

2. The SQL — STRING_AGG

SELECT deptid,
       STRING_AGG(emailid, ';' ORDER BY emailid ASC) AS email_list
FROM emp_details
GROUP BY deptid;

STRING_AGG concatenates the column across each GROUP BY bucket, joining values with the literal ;. The ORDER BY inside the parentheses sorts the elements within each list — independent of any outer ordering.

3. The resulting analysis

deptidemail_list
10alice@co.com;bob@co.com;charlie@co.com
20dana@co.com;eve@co.com
30frank@co.com

Department 10's list is alphabetised even though the rows were inserted out of order — that is the ORDER BY emailid doing its job. Department 30 returns frank@co.com with no leading or trailing semicolon: a delimiter is placed between elements, so a one-element list has zero delimiters.

4. Strategic explanations

Why the inner ORDER BY is not optional

Drop the ORDER BY and the engine concatenates in whatever physical order rows arrive — which is unspecified and can change between runs, after a VACUUM, or under parallel scan. For a value a human reads or a downstream system diffs, that nondeterminism is a bug. Always pin the order inside the aggregate so the output is reproducible.

The dialect map — same idea, four names

STRING_AGG is Postgres and SQL Server. The interviewer may want the others:

-- MySQL
SELECT deptid, GROUP_CONCAT(emailid ORDER BY emailid SEPARATOR ';')
FROM emp_details GROUP BY deptid;

-- Oracle
SELECT deptid, LISTAGG(emailid, ';')
       WITHIN GROUP (ORDER BY emailid)
FROM emp_details GROUP BY deptid;

BigQuery uses STRING_AGG too. Knowing all four — and that they share one mental model: group, order, join — signals breadth.

NULL emails vanish silently

Like every aggregate, STRING_AGG skips NULL inputs. If an employee has a NULL emailid, they simply do not appear in the list — no empty slot, no ;;. That is usually desirable, but it means the list length can be smaller than the headcount. If the consumer needs every employee represented, substitute first: STRING_AGG(COALESCE(emailid, '(no email)'), ';' ORDER BY ...).

Duplicates and the DISTINCT option

If two rows share an email (a shared alias, a data-entry dup), it appears twice in the string. Postgres and MySQL accept DISTINCT inside the aggregate — but note Postgres rejects DISTINCT combined with an ORDER BY on a different expression. Order by the same column you de-duplicate on:

STRING_AGG(DISTINCT emailid, ';' ORDER BY emailid)
The reverse operation — splitting back to rows

A natural follow-up: "now explode that list back into one row per email." That is the inverse, and Postgres has it built in:

SELECT deptid, unnest(string_to_array(email_list, ';')) AS emailid
FROM dept_email_lists;

Knowing both directions — aggregate to a string, and split a string back to rows — shows you treat delimited text as a reversible transform, not a dead end.

Senior signal: always pin an inner ORDER BY so the concatenation is deterministic, name the GROUP_CONCAT/LISTAGG/STRING_AGG dialect equivalents, and call out that NULL emails are dropped silently — deciding on purpose whether that is correct for the mail-merge consumer.
