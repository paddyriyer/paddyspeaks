Every billing team at every B2B SaaS company hits this problem on day one: revenue is recorded in three places and none of them match. Stripe says the customer paid; the webhook stream says they paid (sometimes twice); the internal credit-adjustments table moves the number around for refunds and discounts. The monthly revenue report has to reconcile all three, drop the duplicates, drop the nulls, and produce one number per region per month that the CFO can sign off on. Vercel's billing engineers ship this exact pipeline against Stripe, Tinybird, and Postgres; the data model below is verbatim from the FinOps wiki of half the YC S22 cohort.

1. The data — three feeds, one fact

SourceCadenceWhy it's the source of truth for…Why it's NOT for…
Stripe CSV exportDaily, batchSettled charges with auth + captureReal-time — lags by 24 h
Webhook JSON streamReal-time, at-least-onceLive revenue dashboardReconciliation — duplicates happen
Postgres credit adjustmentsManual entry by FinOpsRefunds, prorations, free-creditsCustomer charges — only adjustments

The shared key is transaction_id, which appears in Stripe and the webhook stream (because the webhook is generated from the same Stripe object). The adjustments table has its own ID space — never collides. That's why drop_duplicates(subset=['transaction_id']) works: it dedupes Stripe-vs-webhook overlap while preserving adjustments.

2. The business question — what the CFO signs off on

  • Investor reporting. "What was Q4 EU revenue?" The number on the slide deck has to match Stripe's official figure to the penny, but also include the adjustments — refunds reduce ARR, prorations smooth it. One pipeline, one query, one truth.
  • Per-region tax compliance. EU revenue is taxed differently from US — the region partition is non-decorative. A missing-region row in the report becomes a 6-figure tax filing problem.
  • Anomaly detection for the on-call. If avg_order drops 40 % week-over-week, either pricing is broken or the dedup logic just started over-dropping rows. The avg_order metric exists for exactly this — sanity check on the aggregation.

3. The pandas implementation — annotated

def unified_report(csv_frames, json_frames, db_frames):
    # ── Step 1. Concatenate. axis=0 (rows) is default; ignore_index
    # re-numbers the rows so we get 0..N instead of overlapping
    # indexes from the source frames.
    parts = list(csv_frames) + list(json_frames) + list(db_frames)
    df = pd.concat(parts, ignore_index=True)
    df['date'] = pd.to_datetime(df['date'])

    # ── Step 2. De-dup on the natural key. Default keep='first' —
    # the Stripe CSV row wins over the webhook duplicate (it's the
    # authoritative settlement record). If you flipped the order of
    # `parts`, the webhook row would win — explicit ordering matters.
    df = df.drop_duplicates(subset=['transaction_id'])

    # Drop null amounts — webhook can fire with amount=None on
    # half-formed payment intents (auth-only, not captured).
    df = df.dropna(subset=['amount'])

    # ── Step 3. The double groupby — region (categorical) × month
    # (time-binned). pd.Grouper(freq='ME') = month-end resampling
    # combined with a regular groupby key. This is the canonical
    # "category × time" aggregation pattern; remember it.
    report = (df.groupby(['region', pd.Grouper(key='date', freq='ME')])
                .agg(revenue=('amount', 'sum'),
                     orders=('transaction_id', 'count'),
                     avg_order=('amount', 'mean'))
                .reset_index())
    report['avg_order'] = report['avg_order'].round(2)
    report['revenue']   = report['revenue'].round(2)
    return report

Why pd.Grouper(key='date', freq='ME') instead of df['date'].dt.to_period('M')? Grouper is the right tool when you want to combine time-binning with another group key (region) in one groupby. The .dt.to_period('M') form makes a new column you then groupby on; same answer, more memory. Pick Grouper for the combined case.

Why drop nulls AFTER dedup, not before? Order doesn't change correctness here (dedup is on transaction_id, null is on amount — orthogonal). But conceptually: a duplicate with amount=null and another duplicate with amount=120 should keep the 120, not drop both. drop_duplicates(keep='first') after sorting by amount-non-null gives that guarantee. The demo data is benign so the default works.

5. Production patterns — how this scales

  • pandas → 1 GB before OOM. For Vercel's transaction volume (~10 M/year) this fits; for Stripe's own internal dashboard (~10 B/year), no. The production rewrite: dbt models on Snowflake / BigQuery with the same SQL shape — UNION ALL in place of pd.concat, QUALIFY ROW_NUMBER() OVER (PARTITION BY transaction_id) = 1 in place of drop_duplicates, DATE_TRUNC('month', date) in place of pd.Grouper. Same algorithm, columnar engine.
  • The "merge-then-aggregate" anti-pattern. Some candidates try stripe.merge(webhook, on='transaction_id', how='outer') — which keeps both copies on the duplicate key and forces you to coalesce columns. Concat-then-dedup is cleaner because the schemas are identical. Merge is for joining different facts (orders + customers); concat is for stacking same facts from multiple sources.
  • Hot region — US has 80 % of rows. The groupby on (region, month) ships all US-month combinations to one shuffle bucket. In pandas this is fine; in Spark, salt the region with a small salt (4-8 buckets is enough — only 5-6 regions exist, so the skew isn't extreme). See customer_growth-0027.
  • Stripe Sigma exposes this exact data model. The charges + balance_transactions tables map to stripe_csv; the events table is the webhook stream. Sigma users write SQL against the same schema; the pandas function above is the same query in DataFrame form.
  • Datadog billing uses a multi-source unified report for cross-cloud customer billing (AWS, GCP, Azure usage all stream in separately and have to reconcile against the customer's Stripe charges). Same algorithm shape: concat + dedup + groupby on (cloud, month) instead of (region, month).

6. Interviewer probes

Probe 1 — "What if the same transaction_id appears in two sources with different amounts?"

Bug. Either the source data is wrong (Stripe and the webhook should agree to the cent — they're generated from the same payment intent), or your transaction_id isn't actually the unique key. Production defence: write an assertion before drop_duplicates — assert df.groupby('transaction_id')['amount'].nunique().max() == 1. If it fails, page the on-call rather than silently dropping the wrong row. Silent dedup of mismatched data is how revenue numbers diverge from the actual cash.

Probe 2 — "What's wrong with using .apply() for the aggregation?"

Same answer as every pandas question: row-by-row Python. The .agg() on the groupby runs the sum/count/mean in C kernels — one pass over the data, no Python overhead. .apply(lambda g: pd.Series({'revenue': g.amount.sum(), ...})) would re-enter Python once per group, ~10-100× slower depending on group count.

Probe 3 — "How would you incrementalise this for daily delta loads?"

Two patterns. (a) Insert-only: only the new day's rows enter the pipeline, you re-run the full month-aggregate on the affected months. dbt's {{ config(materialized='incremental') }} does this. (b) Streaming: Kafka + Flink with a 1-month tumbling window keyed on region. The aggregate state is updated per event. The pandas version is the offline backfill / audit tool; the streaming version is the live dashboard.

Probe 4 — "Webhook is at-least-once and fires duplicates in the same minute. Will your dedup catch them?"

Yes — the duplicates have the same transaction_id, which is the dedup key. The harder case is when the webhook fires a row with transaction_id=null (incomplete payment intent) and a different field — there your dedup misses. Defence: filter out null transaction_id rows explicitly before dedup, or generate a synthetic key from (customer_id, ts, amount).

Probe 5 — "What if a transaction crosses month-end — paid at 23:59 UTC on the 31st but settled at 00:01 on the 1st?"

Two timestamps in play: created_at (when the customer paid) and captured_at (when Stripe settled). The CFO wants revenue recognised against created_at (the customer event), not captured_at (the settlement event). The schema needs to make this explicit. A common bug is to use whichever column the data dump happens to ship with — both are "valid" timestamps, only one is right for revenue reporting. Pick deliberately; document the choice.

Probe 6 — "Refund handling — how does the report show a refund?"

In the demo, refunds enter via the credits table with a negative amount. That nets into the monthly revenue cleanly — October US is $300 - $50 = $250. Alternative: keep refunds in a separate column (gross_revenue, refunds, net_revenue) for transparency. Investor reporting wants the gross/net split. Tax filings sometimes want both. Talk through the trade-off; senior candidates suggest the three-column form for an auditable report.

Probe 7 — "Excel output (.to_excel) — is that production-acceptable?"

For an emailed CFO report, yes. For a real data pipeline, no — write to Parquet on S3, materialise as a dbt model, or push to a BI tool's data layer. Excel is the convenience-output for "I just need to send this to finance". The candidate who calls this out is showing operational maturity.
