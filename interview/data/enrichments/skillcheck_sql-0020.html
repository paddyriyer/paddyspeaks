A growth team wants the classic conversion funnel: of everyone who viewed, how many added to cart, and of those, how many purchased — with the strict rule that a step only counts if every earlier step happened before it in time. The interviewer is probing whether you understand that a funnel is not just COUNT per step: out-of-order events (a cart before a view) and skipped steps must be excluded, or the funnel lies.

1. The source data — event

user_idstepts(narrative)
1view / cart / purchase09:00 / 09:05 / 09:10full funnel, in order
2view / cart09:00 / 09:01reached cart, no purchase
3view09:00viewed only
4cart / view09:00 / 09:05cart before view — out of order
5view / purchase09:00 / 09:05purchase with no cart — step skipped

Users 4 and 5 are the trap. User 4 has both a view and a cart, but the cart came first — they never validly reached cart. User 5 purchased without ever carting — a skipped step. A naive per-step count would wrongly credit both.

2. The SQL — first-timestamp per step, then ordered reachability flags

WITH firsts AS (
  SELECT user_id,
         MIN(CASE WHEN step='view' THEN ts END) AS view_ts,
         MIN(CASE WHEN step='cart' THEN ts END) AS cart_ts,
         MIN(CASE WHEN step='purchase' THEN ts END) AS purchase_ts
  FROM event
  GROUP BY user_id
),
flags AS (
  SELECT
    user_id,
    (view_ts IS NOT NULL)::int AS reached_view,
    (view_ts IS NOT NULL AND cart_ts IS NOT NULL AND cart_ts >= view_ts)::int AS reached_cart,
    (view_ts IS NOT NULL AND cart_ts IS NOT NULL AND purchase_ts IS NOT NULL
      AND cart_ts >= view_ts AND purchase_ts >= cart_ts)::int AS reached_purchase
  FROM firsts
)
SELECT 'view' AS step, SUM(reached_view) AS users FROM flags
UNION ALL
SELECT 'cart', SUM(reached_cart) FROM flags
UNION ALL
SELECT 'purchase', SUM(reached_purchase) FROM flags;

firsts collapses each user to one row holding their earliest timestamp for each step. flags then turns reachability into a 0/1 per step, requiring every prior step to be non-NULL and chronologically ordered. The final union sums the flags into one row per step.

3. The resulting analysis

stepuserssignal
view5all five users have a view event
cart2users 1 and 2 — user 4 excluded (cart before view)
purchase1only user 1 — user 5 excluded (no cart)

The funnel reads 5 → 2 → 1. User 4 has a cart but it predates their view, so cart_ts >= view_ts is false — not counted. User 5 purchased but cart_ts is NULL, so reached_purchase is false. The funnel only credits genuine, ordered progress.

4. Strategic explanations

Why MIN(ts) per step and not just a presence check

A user can hit the same step many times; what matters for ordering is the first time they reached it. MIN(CASE WHEN step=... THEN ts END) extracts that earliest timestamp, returning NULL when the step never happened. Comparing first-timestamps is what lets cart_ts >= view_ts mean "carted no earlier than they first viewed". A plain COUNT(DISTINCT step) would know which steps occurred but never in what order — and order is the entire point of a funnel.

The chronological predicate is what makes it a funnel

reached_cart is not "has a view and has a cart" — it is "has a view, has a cart, and the cart was not before the view". That third clause excludes user 4. Each deeper step ANDs in one more ordering check (purchase_ts >= cart_ts). Drop these comparisons and you get a set-membership count, not a funnel; the count would no longer decrease monotonically in a meaningful way. The ordering is the feature, not decoration.

NULL handling is structural, not incidental

If a step never happened, its first-timestamp is NULL. Every comparison against NULL yields unknown, so any AND chain containing it is false — which is exactly right: user 5's cart_ts is NULL, so reached_purchase collapses to false automatically. The explicit IS NOT NULL guards make the intent legible rather than relying on three-valued logic alone. NULL here is the "step skipped" signal.

::int casting and a tie at the boundary

A boolean cast to int gives 1 for true and 0 for false, so SUM over the column counts the users who reached the step. Note the comparison is >=, not >: if a user's cart and view share the exact same timestamp, that should still count as reaching cart. Using strict > would wrongly drop a same-instant progression — pick the inclusive operator deliberately.

Scaling and an alternative shape

For three steps the explicit columns are clear. For a long or variable funnel, a more scalable pattern computes each user's step sequence with ROW_NUMBER/LAG over (user_id ORDER BY ts) and checks that steps appear in the canonical order, or self-joins each step to the next on user_id with a ts inequality. Mentioning that the hard-coded columns do not scale — and how you would generalise — is the senior signal.

Senior signal: a funnel is ordered reachability, not per-step counting — reduce each user to first-timestamp-per-step, require every prior step to be both present and chronologically earlier (cart_ts >= view_ts), let NULL naturally exclude skipped steps, and note how the hard-coded steps would generalise for a longer funnel.
