iCloud's tier ladder (free → 50GB → 200GB → 2TB) is one of Apple's stickiest revenue moats. The query uses LAG to detect tier transitions and counts each direction (upgrade / downgrade / churn).

1. The source data — icloud_tiers

user_idtierupgraded_at
U_001free2024-01-01
U_00150gb2024-03-15

2. The SQL — LAG-based transition counting

WITH transitions AS (
  SELECT user_id, tier,
    LAG(tier) OVER (PARTITION BY user_id ORDER BY upgraded_at) AS prev_tier
  FROM icloud_tiers
)
SELECT prev_tier, tier AS new_tier, COUNT(*) AS users
FROM transitions
WHERE prev_tier IS NOT NULL
GROUP BY prev_tier, tier
ORDER BY prev_tier, users DESC;

3. Strategic explanations

The LAG trap — first row has NULL

The first row per user has prev_tier IS NULL — that's a brand-new account, not a transition. Filter it out: WHERE prev_tier IS NOT NULL.

Upgrade vs downgrade — directional

The transition matrix has both directions: 50gb → 200gb (upgrade) and 200gb → 50gb (downgrade). Add a tier-rank lookup to classify:

JOIN tier_rank rp ON rp.tier = prev_tier
JOIN tier_rank rn ON rn.tier = new_tier
... CASE WHEN rn.rank > rp.rank THEN 'upgrade' ELSE 'downgrade' END
Time-to-upgrade — the velocity metric

Average time spent on each tier before upgrading is the marketing-intervention signal. Users on free for 90 days unlikely to ever pay; users who upgrade within 30 are the high-intent segment.

Skip-tier upgrades — the cliff

Free → 2TB skipping 50gb and 200gb is rare but high-LTV. Users who skip indicate strong commitment + photo-heavy use case. The query treats free→2TB as one row; you can spot it directly.

Cohort-by-signup-month split

Cohorts that signed up in iPhone-launch months have different upgrade behavior (new device = more photos). Slice by signup cohort to see if marketing campaigns are working.

Senior signal: name the LAG-NULL-first-row gotcha, classify direction (upgrade / downgrade) via tier-rank join, time-to-upgrade as velocity, and skip-tier identification. Cohort split by signup month is the over-the-bar segmentation.
