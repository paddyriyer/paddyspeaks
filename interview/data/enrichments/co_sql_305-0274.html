The growth team is studying purchase cadence: how often each customer buys, and whether they're slowing down (a churn leading indicator). The single most useful column to compute is the days since their previous order — the inter-purchase interval. LAG handles it in one line.

2. The SQL — LAG partitioned per customer

SELECT
  customer_id,
  order_date,
  -- Subtract the previous order_date (per customer) from this one
  order_date - LAG(order_date) OVER (
    PARTITION BY customer_id
    ORDER BY order_date
  ) AS gap_days
FROM orders;

The NULL on each customer's first row is correct — they have no prior order to compare against. Ignore those when computing per-customer averages.

4. Strategic explanations

The PARTITION BY is the entire point

Drop PARTITION BY customer_id and the gap on row 5 (customer 2002's first order) becomes 2026-02-10 − 2026-04-15 = -64 days — the LAG would compare to the previous customer's last order. Wrong, and the dashboard quietly produces nonsense. The partition isolates each customer's history into its own LAG window.

Why order_date - LAG(order_date) returns days (in some dialects)

In Postgres, BigQuery and Snowflake, subtracting two DATE values returns an integer number of days — exactly what you want. In SQLite, you need julianday(order_date) - julianday(LAG(order_date) OVER ...). In MySQL, DATEDIFF(order_date, LAG(...)). State the dialect; the pattern is identical.

The follow-up: average gap per customer (cadence)

Once each row has its gap, the average per customer is the cadence:

WITH gaps AS (
  SELECT customer_id,
    order_date - LAG(order_date) OVER (PARTITION BY customer_id ORDER BY order_date) AS gap_days
  FROM orders
)
SELECT customer_id,
  ROUND(AVG(gap_days), 1) AS avg_cadence_days,
  COUNT(*) - 1            AS gap_observations,    -- exclude NULL first-row
  MAX(gap_days)           AS longest_gap
FROM gaps
WHERE gap_days IS NOT NULL
GROUP BY customer_id;
The "is this customer slowing down?" version — compare last gap to running median

The real growth-team query is: "flag customers whose most recent gap is >2× their median historical gap". That's churn risk. Achievable by extending the LAG pattern with another window function:

SELECT customer_id, order_date, gap_days,
  ROUND(gap_days * 1.0 / median_gap_so_far, 2) AS gap_vs_median
FROM (
  SELECT customer_id, order_date, gap_days,
    -- median of all gaps seen so far for this customer (excluding NULL)
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY gap_days)
      OVER (PARTITION BY customer_id ROWS UNBOUNDED PRECEDING) AS median_gap_so_far
  FROM gaps
)
WHERE gap_vs_median >= 2;
"Days since last order" right now — CURRENT_DATE − MAX(order_date)

For a live "are they still active?" check, you don't need LAG — just CURRENT_DATE - MAX(order_date) per customer. LAG is for historical gap pattern; a single value is for current state. Both have a place.

Senior signal: show the PARTITION BY customer_id as the critical piece (without it the query is silently wrong), volunteer the dialect difference for date subtraction, and pre-empt the cadence-aggregation follow-up. The "gap-vs-median" extension is the over-the-bar answer.
