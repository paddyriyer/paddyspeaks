This is LeetCode's "Consecutive Numbers" in community form: scanning num_log in seq_id order, find every value that appears three or more times in a row. It is the signal-detection pattern — a sustained streak, not a scattered total. The interviewer wants to see the gaps-and-islands technique, and to confirm you do not confuse "appears 3 times consecutively" with "appears 3 times overall".

Value 3 appears three times overall (seq_id 4, 9, 10) — but never three in a row. It must not qualify. That is the trap separating "consecutive" from "total count".

2. The SQL — canonical gaps-and-islands

WITH ordered AS (
    SELECT seq_id, user_id,
           LAG(user_id) OVER (ORDER BY seq_id) AS prev_user_id
    FROM num_log
),
flagged AS (
    SELECT seq_id, user_id,
           CASE WHEN user_id = prev_user_id THEN 0 ELSE 1 END AS is_break
    FROM ordered
),
grouped AS (
    SELECT seq_id, user_id,
           SUM(is_break) OVER (ORDER BY seq_id) AS group_id
    FROM flagged
)
SELECT user_id
FROM grouped
GROUP BY user_id, group_id
HAVING COUNT(*) >= 3
ORDER BY user_id;

LAG looks at the previous row's value; a "break" is flagged whenever the value changes; a running SUM of those breaks gives each unbroken run a distinct group_id.

3. The resulting analysis

user_idcommentary
7run at seq_id 1–3, length 3
9run at seq_id 5–8, length 4

Only 7 and 9 qualify. Value 3 is correctly excluded: its longest consecutive run is just two rows (seq_id 9–10), even though it occurs three times in total.

4. Strategic explanations

How the running-sum builds island IDs

is_break is 1 exactly where the value differs from the row before it — the start of a new run. The cumulative SUM(is_break) OVER (ORDER BY seq_id) therefore increments by 1 at each run boundary and holds steady inside a run. Every row of a single unbroken run shares one group_id. Grouping by (user_id, group_id) isolates each run; HAVING COUNT(*) >= 3 keeps the long ones. Including user_id in the GROUP BY guards against the rare case of two separate runs accidentally landing on the same group_id.

Why "consecutive" is not "total" — the core trap

A naive GROUP BY user_id HAVING COUNT(*) >= 3 counts every occurrence and would wrongly return value 3. Consecutiveness is a property of adjacency in seq_id order, and only an ordered window function (LAG, or a running sum) can see adjacency. Recognising that the ordering column is load-bearing — not decoration — is the senior insight here.

The classic three-self-join alternative

The textbook LeetCode answer joins the table to itself twice:

SELECT DISTINCT a.user_id
FROM num_log a
JOIN num_log b ON b.seq_id = a.seq_id + 1 AND b.user_id = a.user_id
JOIN num_log c ON c.seq_id = a.seq_id + 2 AND c.user_id = a.user_id;

It works, but it is hardwired to a streak of exactly 3 — checking for 5-in-a-row needs four joins. The gaps-and-islands version generalises by changing one literal in HAVING, and it does one ordered pass instead of an n-way join. Note this self-join also assumes seq_id has no gaps; LAG does not.

The gap-in-seq_id and NULL traps

LAG(user_id) OVER (ORDER BY seq_id) reads the previous row by sort order, regardless of whether seq_id values are contiguous — so it tolerates gaps, unlike the seq_id + 1 self-join. Two subtleties: the very first row has prev_user_id = NULL, so user_id = NULL is unknown and is_break is correctly 1 (a run always starts there); and a NULL user_id mid-stream would also break the run, since NULL = NULL is unknown. If consecutive NULLs should count as a run, switch to user_id IS NOT DISTINCT FROM prev_user_id.

Senior signal: name this gaps-and-islands, stress that "consecutive" depends on seq_id ordering so a plain COUNT is wrong, and prefer the LAG + running-sum solution over the triple self-join because it generalises to any streak length and tolerates gaps in seq_id.
