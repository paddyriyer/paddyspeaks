Data Vault 2.0 separates concerns that traditional Kimball star schemas conflate: business keys live in hubs, relationships live in links, and descriptive attributes with their history live in satellites. The interviewer is probing whether you understand those three responsibilities, why hubs are append-only keyed on a hash, why satellites are append-only keyed on (hub_hash, load_ts), and why links capture only relationship structure without carrying descriptive attributes. Getting the hash-key pattern and the record_source lineage column right is the key signal.

1. The requirements — what the design must capture

  • Hub: stores the customer’s business key plus load metadata. No descriptive attributes. Append-only.
  • Satellite: stores descriptive attributes (name, email, country, tier) with full history via (hub_hash, load_ts) PK. Append-only; a hash_diff detects attribute changes without column-by-column comparison.
  • Link: stores the relationship between the customer hub and an account hub. Append-only. Contains only the hub hashes and load metadata — no attributes.
  • All tables: must carry record_source for full lineage to the originating system.

2. The DDL — the CREATE TABLE

-- Hub: business key + load metadata only
CREATE TABLE hub_customer (
  customer_hk        CHAR(32)     NOT NULL,        -- hash of business key
  customer_bk        VARCHAR(64)  NOT NULL,        -- business / natural key
  load_ts            TIMESTAMP    NOT NULL,
  record_source      VARCHAR(50)  NOT NULL,
  PRIMARY KEY (customer_hk),
  UNIQUE (customer_bk)
);

-- Satellite: descriptive attributes, append-only, versioned by load_ts
CREATE TABLE sat_customer (
  customer_hk        CHAR(32)     NOT NULL,
  load_ts            TIMESTAMP    NOT NULL,
  load_end_ts        TIMESTAMP,                    -- NULL for current version
  hash_diff          CHAR(32)     NOT NULL,        -- detect attribute changes
  full_name          VARCHAR(200),
  email              VARCHAR(200),
  country            VARCHAR(2),
  tier               VARCHAR(20),
  record_source      VARCHAR(50)  NOT NULL,
  PRIMARY KEY (customer_hk, load_ts),
  FOREIGN KEY (customer_hk) REFERENCES hub_customer (customer_hk)
);

-- Link: relationship between hubs (customer <-> account)
CREATE TABLE link_customer_account (
  link_hk            CHAR(32)     NOT NULL,        -- hash(customer_hk, account_hk)
  customer_hk        CHAR(32)     NOT NULL,
  account_hk         CHAR(32)     NOT NULL,
  load_ts            TIMESTAMP    NOT NULL,
  record_source      VARCHAR(50)  NOT NULL,
  PRIMARY KEY (link_hk),
  FOREIGN KEY (customer_hk) REFERENCES hub_customer (customer_hk)
);

4. Strategic explanations

The three Data Vault building blocks

Hubs contain exactly one business key per entity. They are insert-only: once a customer_bk is in the hub, it never changes or moves. Satellites contain all descriptive attributes and their history; they are also insert-only (new versions are appended, old ones are closed with load_end_ts). Links capture relationships between hubs; they too are insert-only. This append-only design means no UPDATE or DELETE statements ever touch the raw vault — the entire history of data is preserved by construction.

Hash keys: deterministic surrogates

Unlike integer surrogates generated by a sequence, hash keys are computed: customer_hk = MD5(UPPER(TRIM(customer_bk))). Because any system can compute the same hash from the same business key, multiple source systems can independently generate and load records without a central key-registry lookup. This is the property that makes Data Vault parallelisable across many load processes — a critical scalability advantage over Kimball surrogates, which require a central sequence or SEQUENCE object lookup.

Multiple satellites per hub

A single customer hub can have multiple satellite tables: sat_customer_crm (attributes from the CRM), sat_customer_erp (financial attributes from ERP), sat_customer_contact (contact preferences from a marketing system). Each satellite has its own load cadence and schema; adding a new source system adds a satellite without altering the hub or existing satellites. This source-system isolation is a core Data Vault advantage over monolithic staging tables.

hash_diff: efficient change detection in satellites

The ETL computes hash_diff = MD5(CONCAT(full_name, email, country, tier)) for each incoming row and compares it to the latest satellite row’s hash_diff. If equal, no action; if different, a new satellite row is inserted. This is the same pattern as row_hash in SCD2, but in Data Vault it is called hash_diff by convention. The hash function, field ordering, and null-handling must be consistent across all ETL jobs or spurious new versions will be created.

Links and relationship history

The link captures when a relationship was first observed (load_ts) and from which source (record_source). If the relationship ends (customer closes their account), Data Vault does not delete the link; it adds a link-effectivity satellite (sat_link_customer_account_eff) with an end-date attribute. This keeps the raw vault immutable while still modelling relationship termination. Describing this satellite pattern for link history shows deep Data Vault knowledge.

Data Vault vs. Kimball

Data Vault excels when: source systems are numerous and heterogeneous, auditability is paramount (regulatory environments), and schemas change frequently. Kimball star schemas are simpler to query for business users and perform better in OLAP scenarios with fewer source systems. Data Vault typically has a “business vault” or “information mart” layer on top that transforms vault structures into star-schema dimensions and facts for consumption.

Senior signal: explain that all three vault table types are insert-only — never UPDATE or DELETE — and why (full historical preservation by construction); justify deterministic hash keys as enabling parallel, source-independent loads without a central sequence lookup; describe the multiple-satellite pattern for source-system isolation; and note that Data Vault typically feeds a downstream star-schema information mart rather than being queried directly by analysts.
