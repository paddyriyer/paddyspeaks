A merchandising team wants every product numbered in ascending price order — a simple "rank by price" list for a pricing review. The reference solution applies ROW_NUMBER() OVER (ORDER BY price ASC). The interviewer is probing whether you understand how ROW_NUMBER breaks ties (it does not — it assigns arbitrary distinct numbers), where NULL prices land, and when RANK would be the right tool instead.

Two rows decide the interview. Cable A and Mug are tied at 9.99, and Sensor has a NULL price. How ROW_NUMBER orders the tie, and where the NULL ends up, is the whole question.

2. The SQL — sequential numbering with ROW_NUMBER

SELECT product_id, name, price,
       ROW_NUMBER() OVER (ORDER BY price ASC) AS seq
FROM products;

ROW_NUMBER assigns 1, 2, 3, … to rows in the order defined by the window's ORDER BY — one distinct integer per row, no gaps, no repeats.

The two 9.99 products get 3 and 4 — different numbers for an identical price. Which one got 3 is arbitrary: ROW_NUMBER needs a total order, and when price alone does not provide one the engine picks whatever order it likes. Re-run the query and the 3/4 assignment could flip. Sensor's NULL price sorts last — PostgreSQL places NULLs last for ORDER BY ... ASC by default.

4. Strategic explanations

ROW_NUMBER never shares a number — ties are broken arbitrarily

ROW_NUMBER always produces 1..n with no duplicates, even when the ORDER BY values tie. For the two 9.99 rows it must pick an order, and with no tie-breaker that choice is non-deterministic — not stable across runs, indexes, or parallel plans. If "sequence number" must be reproducible, the fix is a deterministic tie-breaker: ORDER BY price ASC, product_id ASC. Always make the window ORDER BY a total order when the output must be stable.

ROW_NUMBER vs RANK vs DENSE_RANK — pick by tie semantics

If the two 9.99 products should share a position, ROW_NUMBER is the wrong function. RANK() OVER (ORDER BY price) gives 1, 2, 3, 3, 5, 6 — ties share, next rank skips. DENSE_RANK() gives 1, 2, 3, 3, 4, 5 — ties share, no gap. ROW_NUMBER is correct only when you genuinely want one distinct integer per row (paging, "pick exactly one row per group"). Naming which of the three the task wants is the senior move.

NULL prices — where they land and how to control it

Sensor's NULL got seq = 7 because Postgres sorts NULLs last under ASC (and first under DESC). Other engines differ — Oracle also puts NULLs last for ASC, but SQL Server puts them first. Never rely on the default if it matters: write ORDER BY price ASC NULLS LAST (or NULLS FIRST) explicitly. If unpriced products should not be numbered at all, filter WHERE price IS NOT NULL before the window.

Window function vs the old self-join

The pre-window way to number rows was a correlated subquery counting cheaper rows — O(n²) and clumsy with ties. ROW_NUMBER sorts once, O(n log n), and assigns numbers in a single pass. It is both faster and unambiguous about intent. An index on price lets the optimiser skip the sort entirely, making the window function close to free on large tables.

The window has no PARTITION BY — numbering is global

This OVER (ORDER BY price) has no PARTITION BY, so seq runs 1..7 across the whole table. If the requirement were "sequence within each category", you would add PARTITION BY category — numbering would then restart at 1 per category. Knowing that omitting PARTITION BY means "one global partition" — and adding it scopes the numbering — shows you understand the clause, not just the function name.

Senior signal: know that ROW_NUMBER assigns distinct integers and breaks ties arbitrarily — add a tie-breaker like product_id for reproducibility — choose RANK/DENSE_RANK when tied rows must share a position, and control NULL placement explicitly with NULLS LAST/FIRST rather than trusting an engine-specific default.
