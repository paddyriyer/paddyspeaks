Palantir's deployment analytics team needs every order in a mid-tier spend band — between $100 and $1000 — for a cohort study, and only orders with a valid date. It is a two-clause WHERE, but BETWEEN hides a question candidates routinely get wrong: is it inclusive or exclusive at the endpoints? And mixing ANDs tests whether you know operator precedence.

2. The SQL — BETWEEN + IS NOT NULL

SELECT * FROM orders WHERE amount BETWEEN 100 AND 1000 AND order_date IS NOT NULL;

Two conditions joined by AND: the amount must fall in the inclusive range 100–1000, and the date must be present.

4. Strategic explanations

BETWEEN is inclusive — both endpoints count

amount BETWEEN 100 AND 1000 is exactly equivalent to amount >= 100 AND amount <= 1000. Orders 1 and 2 surviving the filter is the proof. Candidates frequently assume it is exclusive (or are unsure) — state it plainly. If the requirement is really "above $100", BETWEEN is wrong and you need a strict amount > 100. Inclusivity is a business decision; do not let the operator make it silently.

The argument-order trap — BETWEEN is not symmetric

BETWEEN requires low AND high in that order. amount BETWEEN 1000 AND 100 does not quietly swap them — it expands to amount >= 1000 AND amount <= 100, which is impossible, so it matches zero rows and throws no error. A reversed BETWEEN is a silent empty result. Always write the smaller bound first.

The NULL clause — why IS NOT NULL, never != NULL

Order 4 is excluded by order_date IS NOT NULL. The trap is writing order_date != NULL or order_date <> NULL — any comparison to NULL with =, !=, < yields unknown, never true, so such a predicate matches nothing and order 4 would not be the only casualty — every row would be dropped. NULL tests must use the IS NULL / IS NOT NULL operators. This is one of the most common SQL mistakes; getting it right is a baseline senior expectation.

BETWEEN already excludes NULL amounts

A subtle point: if amount itself were NULL, amount BETWEEN 100 AND 1000 evaluates to unknown and the row is excluded anyway — WHERE keeps only rows where the whole predicate is true. So NULL amounts never sneak in. The explicit IS NOT NULL in the query is guarding order_date specifically, because the requirement singles out the date column.

Operator precedence — why no parentheses are needed (but help)

The WHERE clause is amount BETWEEN 100 AND 1000 AND order_date IS NOT NULL. The AND inside BETWEEN is part of the BETWEEN syntax, not a logical AND — the parser binds it to the range. The trailing AND is the logical one joining the two predicates. It happens to parse correctly, but it reads ambiguously. In production, parenthesise for the human:

SELECT * FROM orders
WHERE (amount BETWEEN 100 AND 1000)
  AND (order_date IS NOT NULL);

The query is identical to the engine; the parentheses are for the next person to read it. Precedence becomes a genuine bug magnet once OR enters the mix — OR binds looser than AND — so the parenthesising habit is worth forming now.

Senior signal: state that BETWEEN is inclusive on both endpoints and that a reversed low/high silently matches nothing, use IS NOT NULL rather than != NULL (which would drop every row), note that BETWEEN already excludes NULL amounts via three-valued logic, and parenthesise compound predicates for readability before OR turns precedence into a bug.
