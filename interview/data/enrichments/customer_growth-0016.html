A growth-analytics team wants a monthly dashboard that classifies every order's customer as NEW, RESURRECTED, or RETAINED_REPEAT, counts distinct customers per status per month, and shows the month-over-month change for each status line. The interviewer is probing a two-stage window pattern: first LAG over a customer's order history to label each order, then a second LAG over the aggregated monthly grid to compute the MoM trend.

Labelling happens on full history; the window filter is applied only afterward.

2. The SQL — two-stage LAG: per-order label, then per-month trend

WITH labeled AS (
    SELECT customer_id, order_date,
        TO_CHAR(order_date, 'YYYY-MM') AS ym,
        CASE
            WHEN LAG(order_date) OVER w IS NULL THEN 'NEW'
            WHEN (order_date - LAG(order_date) OVER w) > 90 THEN 'RESURRECTED'
            ELSE 'RETAINED_REPEAT'
        END AS growth_status
    FROM orders
    WINDOW w AS (PARTITION BY customer_id ORDER BY order_date)
),
monthly AS (
    SELECT ym AS year_month, growth_status,
           COUNT(DISTINCT customer_id) AS customers
    FROM labeled
    WHERE order_date >= DATE '2024-11-01'
    GROUP BY ym, growth_status
),
trended AS (
    SELECT year_month, growth_status, customers,
           LAG(customers) OVER (PARTITION BY growth_status
                                ORDER BY year_month) AS prev_month_customers
    FROM monthly
)
SELECT year_month, growth_status, customers, prev_month_customers,
    CASE WHEN prev_month_customers > 0
         THEN ROUND(100.0*(customers - prev_month_customers)/prev_month_customers, 1)
         END AS mom_growth_pct
FROM trended
ORDER BY year_month DESC,
    CASE growth_status WHEN 'NEW' THEN 1
                       WHEN 'RESURRECTED' THEN 2
                       ELSE 3 END;

Two structural details show in the output. February has no RESURRECTED row at all — nobody resurrected that month — so the RESURRECTED line for March's LAG reaches back to December, not February. And the earliest months (Nov/Dec 2024) have NULL prev_month_customers because there is no prior month inside the result set to lag from.

4. Strategic explanations

Two distinct window passes — do not conflate them

The query uses LAG twice for two different jobs. The first, in labeled, is PARTITION BY customer_id ORDER BY order_date — it looks at a customer's previous order to classify the current one. The second, in trended, is PARTITION BY growth_status ORDER BY year_month — it looks at the previous month's count for the same status line. Different grain, different partition key. Articulating that you need two separate passes (not one clever window) is the senior framing.

The labelling runs on full history; the window filter comes after

The WHERE order_date >= '2024-11-01' sits in the monthly CTE, after labeled has already classified every order. This ordering is essential: customer 6's first order was April 2024, well before the window. If the filter ran before labelling, LAG would not see that April order and customer 6's March 2025 order would be mislabelled NEW instead of RESURRECTED. Filter the aggregation window, never the classification input.

The MoM LAG compares adjacent rows, not adjacent calendar months

LAG(customers) OVER (PARTITION BY growth_status ORDER BY year_month) returns the previous row within that status partition — which is the previous month that has a row, not necessarily the immediately preceding calendar month. February has no RESURRECTED row, so for the March RESURRECTED line, LAG jumps to December. If the business needs a true "previous calendar month even when zero", you must first build a complete month×status grid (cross join a calendar to the statuses, LEFT JOIN the counts, COALESCE to 0) and lag over that. This gap-in-the-series subtlety is the question's deepest trap.

Why COUNT(DISTINCT customer_id) and not COUNT(*)

A customer could place two orders in the same month, both labelled RETAINED_REPEAT. The dashboard wants how many customers were retained, not how many orders — so COUNT(DISTINCT customer_id) deduplicates within the (month, status) cell. COUNT(*) would over-count multi-order customers and break the headcount semantics.

The NULL-safe MoM division

mom_growth_pct is wrapped in CASE WHEN prev_month_customers > 0. This handles two cases at once: the earliest month where prev_month_customers is NULL (no prior row), and any month where the prior count is genuinely 0. Both correctly yield NULL — "growth is undefined" — rather than a divide-by-zero error or a misleading number. Note the comparison NULL > 0 is itself UNKNOWN, which the CASE treats as not-matched, so the guard covers the NULL case for free.

Senior signal: recognise this as two separate LAG passes — per-customer order history for labelling, per-status month series for the trend — apply the report-window filter only after classification so pre-window history still informs LAG, use COUNT(DISTINCT customer_id) for a headcount, and flag that the MoM LAG compares adjacent rows so a month with zero rows for a status silently makes the comparison skip a month.
