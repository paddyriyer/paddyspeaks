Subscription billing with proration is one of the most nuanced schema design problems in SaaS: a mid-cycle plan change means the billing period must be split into segments, each priced at a different rate. The interviewer is probing whether you understand immutable event sourcing (the subscription_event log is the source of truth; the subscription table is a projection), segment-based proration (each invoice line item covers exactly one time segment at one plan price), and why denormalizing the current state into the subscription table is a useful but secondary concern. Four cooperating tables must be designed coherently.

1. The requirements — what the design must capture

  • Plan catalog: plans have a price, billing cycle length in days, and currency; plan_code is the stable external identifier.
  • Subscription snapshot: the subscription table holds the current live state for fast reads; it is derived from the event log but should not be the only source.
  • Event log: every state transition (created, upgraded, downgraded, paused, resumed, canceled) is appended as an immutable event with the effective timestamp; this is the durable record of truth.
  • Segment-based invoice lines: when a plan change occurs mid-cycle, the billing cycle is split; each sub-period gets its own invoice_line_item row with a prorated amount.
  • Proration formula: amount = plan.unit_price_cents × (segment_days / cycle_days); both segment boundaries and cycle boundaries must be stored on the line item for auditability.

2. The DDL — the CREATE TABLE

CREATE TABLE plan (
  plan_id           BIGSERIAL    PRIMARY KEY,
  plan_code         VARCHAR(64)  NOT NULL UNIQUE,
  name              VARCHAR(128) NOT NULL,
  cycle_days        INT          NOT NULL,                    -- 30 = monthly, 365 = annual
  unit_price_cents  INT          NOT NULL,
  currency          CHAR(3)      NOT NULL,                    -- ISO 4217
  is_active         BOOLEAN      NOT NULL DEFAULT TRUE,
  created_at        TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE TABLE subscription (
  subscription_id   BIGSERIAL    PRIMARY KEY,
  customer_id       BIGINT       NOT NULL,
  current_plan_id   BIGINT       NOT NULL REFERENCES plan(plan_id),
  current_state     VARCHAR(16)  NOT NULL,                    -- 'active','paused','canceled'
  current_cycle_start TIMESTAMPTZ NOT NULL,
  current_cycle_end   TIMESTAMPTZ NOT NULL,
  created_at        TIMESTAMPTZ  NOT NULL DEFAULT now(),
  updated_at        TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX ix_subscription_customer ON subscription (customer_id);

CREATE TABLE subscription_event (
  event_id          BIGSERIAL    PRIMARY KEY,
  subscription_id   BIGINT       NOT NULL REFERENCES subscription(subscription_id) ON DELETE CASCADE,
  event_type        VARCHAR(32)  NOT NULL,                    -- 'created','upgraded','downgraded','paused','resumed','canceled'
  from_plan_id      BIGINT       REFERENCES plan(plan_id),
  to_plan_id        BIGINT       REFERENCES plan(plan_id),
  effective_at      TIMESTAMPTZ  NOT NULL,                    -- when the change takes effect
  created_at        TIMESTAMPTZ  NOT NULL DEFAULT now(),
  reason            TEXT
);
CREATE INDEX ix_subscription_event_sub_effective ON subscription_event (subscription_id, effective_at);

CREATE TABLE invoice_line_item (
  line_item_id      BIGSERIAL    PRIMARY KEY,
  subscription_id   BIGINT       NOT NULL REFERENCES subscription(subscription_id),
  invoice_id        BIGINT       NOT NULL,
  cycle_start       TIMESTAMPTZ  NOT NULL,                    -- the cycle this line item belongs to
  cycle_end         TIMESTAMPTZ  NOT NULL,
  segment_start     TIMESTAMPTZ  NOT NULL,                    -- the sub-period within the cycle
  segment_end       TIMESTAMPTZ  NOT NULL,                    -- segment_end > segment_start
  plan_id           BIGINT       NOT NULL REFERENCES plan(plan_id),
  amount_cents      INT          NOT NULL,                    -- prorated: plan_price * (segment_days / cycle_days)
  currency          CHAR(3)      NOT NULL,
  created_at        TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE INDEX ix_invoice_line_item_invoice ON invoice_line_item (invoice_id);

-- Proration: when a customer upgrades on day D of a 30-day cycle starting S,
-- close the existing line item at D (amount = old_price * (D-S)/30),
-- open a new line item from D to S+30 (amount = new_price * (30-(D-S))/30).
-- All proration logic operates over `subscription_event` between (cycle_start, cycle_end).

4. Strategic explanations

Event sourcing + projection pattern

The subscription_event table is the immutable ledger of truth. The subscription table is a mutable projection (a "current state" cache). At any point in time, replaying all events for a subscription in effective_at order must reproduce the current state. This means: always write an event before updating the subscription row, and update both in the same transaction. The event log also enables time-travel: "what was this subscription's state on 2025-03-15?" is answered by filtering events with effective_at ≤ '2025-03-15'.

Why cycle_days is safer than calendar months

A 30-day billing cycle is unambiguous. A "monthly" cycle is not: February has 28 or 29 days; the proration formula amount = price * (segment_days / cycle_days) requires a fixed denominator. Storing cycle_days = 30 (or 365 for annual) makes arithmetic deterministic and avoids the edge cases of month-end billing (what is "one month" from January 31?). Stripe uses this approach; it is a deliberate product decision.

Proration mechanics in the line-item table

When a customer upgrades on day 10 of a 30-day cycle: (1) the billing job queries subscription_event for all events with effective_at between cycle_start and cycle_end; (2) it produces line items for each continuous same-plan segment: segment A covers days 1–10 at the old plan price × (10/30); segment B covers days 10–30 at the new plan price × (20/30); (3) these are inserted into invoice_line_item and their sum is the invoice total. Downgrades may produce a credit line item (negative amount_cents) rather than a charge.

Dialect notes

In Postgres, use TIMESTAMPTZ for all billing timestamps to avoid DST bugs when customers are in different timezones. Integer cents (INT) is sufficient for most currencies; for high-precision currencies (e.g., currencies with more than 2 decimal places), use BIGINT. In Snowflake or BigQuery, the event log is often landed into a partitioned table (by effective_at month) and the proration logic runs as a SQL job that inserts into an invoice_line_item fact table.

Senior signal: treat subscription_event as the immutable source of truth and subscription as a projection; store cycle_days as an integer not a calendar-month abstraction; and model each prorated segment as its own invoice_line_item row with explicit segment_start/segment_end boundaries so any auditor can verify the arithmetic independently without replaying events.
