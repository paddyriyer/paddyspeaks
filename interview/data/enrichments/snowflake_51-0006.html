A social platform's trust-and-safety team needs to scrub prohibited keywords from user-generated post text before it is surfaced in a feed API. The requirement: replace any occurrence of a banned word — case-insensitively, at a word boundary — with three asterisks, in a single SQL expression with no looping or application code. The interviewer is testing knowledge of REGEXP_REPLACE with alternation, the word-boundary metacharacter \b, and the case-insensitive flag — plus awareness of how these differ between Snowflake and PostgreSQL.

1. The source data — posts

post_idcontent
1This is a spam message with abuse in it
2I hate everything about this. Pure profanity.
3Have a great day! Nothing wrong here.
4SPAM and HATE are capitalized here
5spammy is not the same as spam

Row 3 contains no banned words and should pass through unchanged. Row 4 tests case-insensitivity. Row 5 tests word-boundary correctness: "spammy" must not be replaced because the word does not end at a boundary after "spam".

2. The SQL — word-boundary regex replacement

SELECT post_id,
       REGEXP_REPLACE(content,
                      '\\b(spam|abuse|hate|profanity)\\b',
                      '***', 1, 0, 'i') AS cleaned_content
FROM posts;

Dialect note: this is Snowflake syntax. The arguments are: subject, pattern, replacement, position (start at character 1), occurrence (0 = all), parameters ('i' = case-insensitive). The PostgreSQL equivalent uses a flags argument in a different position: REGEXP_REPLACE(content, '\\m(spam|abuse|hate|profanity)\\M', '***', 'gi') — and the global flag 'g' is required for multi-occurrence replacement. PostgreSQL also uses \m/\M for word boundaries rather than \b.

3. The resulting analysis

post_idcleaned_content
1This is a *** message with *** in it
2I *** everything about this. Pure ***.
3Have a great day! Nothing wrong here.
4*** and *** are capitalized here
5spammy is not the same as ***

Row 1: "spam" and "abuse" both replaced. Row 2: "hate" and "profanity" replaced; the period after "profanity" is not part of the word. Row 3: no match, content unchanged. Row 4: case-insensitive flag matches "SPAM" and "HATE". Row 5: "spammy" is not replaced (word boundary after 'm' in "spammy" is not hit by the pattern ending with \\b), but "spam" at the end of the sentence is replaced correctly.

4. Strategic explanations

The word-boundary metacharacter \b

\b is a zero-width assertion that matches at the transition between a word character ([A-Za-z0-9_]) and a non-word character (space, punctuation, start/end of string). It ensures "spam" matches as a standalone word but not as a prefix inside "spammy" or a suffix inside "instagram". Without \b, every occurrence of the substring fires — including embedded occurrences in innocent words. This single metacharacter is the difference between a production-safe content filter and one that silently damages legitimate text.

Alternation (a|b|c|d) vs multiple passes

The four banned words are joined with | inside a capture group, so the engine makes a single pass through the string and matches any of the four alternatives at each position. An alternative implementation with four nested REGEXP_REPLACE calls would make four passes — quadratic risk when the list grows. Using alternation also makes it easy to add new banned words by extending the pattern string, which in practice is stored in a config table and injected dynamically.

Case-insensitive flag: Snowflake vs PostgreSQL
  • Snowflake: REGEXP_REPLACE(subject, pattern, replacement, 1, 0, 'i') — the sixth argument is a parameter string; 'i' means case-insensitive, 'c' (default) means case-sensitive.
  • PostgreSQL: REGEXP_REPLACE(subject, pattern, replacement, 'gi') — the fourth argument is a flag string; 'g' = global (all occurrences), 'i' = case-insensitive. Without 'g', only the first match is replaced.
  • BigQuery: REGEXP_REPLACE(subject, pattern, replacement) — always replaces all non-overlapping matches; case-insensitive flag uses (?i) inline: '(?i)\\b(spam|abuse|hate|profanity)\\b'.
  • MySQL 8+: REGEXP_REPLACE(subject, pattern, replacement) — global by default; no flag argument; use (?i) inline for case-insensitivity.
Performance and indexing

Regex replacement on a text column always requires a full table scan — no index can accelerate it. For a high-volume feed table, pre-filtering with a full-text index (ts_vector in PostgreSQL, or a SEARCH optimization service) to identify posts containing any banned word before running the regex is the practical production approach. Alternatively, the replacement is done at write time (in the insert/update pipeline) rather than at query time.

The pattern escaping subtlety

The pattern string '\\b(spam|abuse|hate|profanity)\\b' uses double backslashes because one layer of backslash escaping is consumed by the SQL string parser, leaving \b(spam|abuse|hate|profanity)\b as the actual regex string handed to the regex engine. Single backslash in a SQL string literal ('\b') is treated as a literal backslash-b in some engines (a no-op in most regex flavours), not a word boundary. Always double the backslashes in SQL regex patterns.

Senior signal: anchor the pattern with \\b on both sides to prevent substring matches inside longer words; use alternation (a|b|c) for a single-pass replacement; know the Snowflake REGEXP_REPLACE(s, p, r, 1, 0, 'i') signature vs PostgreSQL's 'gi' flag string; and note that regex on a large text column requires a full scan — production content filtering belongs in the write pipeline or a search layer, not a cold SELECT.
