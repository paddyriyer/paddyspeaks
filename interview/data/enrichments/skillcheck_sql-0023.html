A commerce dashboard needs each customer's two most recent orders next to their name — a "recent activity" panel. This is a top-N-per-group problem, the single most common window-function interview question. The interviewer is watching whether you reach for ROW_NUMBER partitioned per user and filter on it, rather than a fragile correlated subquery — and whether you handle a user with fewer than two orders and a timestamp tie.

Two edge cases. Cara (user 3) has just one order — "top 2" must still return it, not require two. And Bob's orders 104 and 105 share the exact same ts — a tie that ROW_NUMBER must break somehow. (Users with no orders, like a hypothetical user 4, would simply not appear.)

2. The SQL — ROW_NUMBER per user, then filter

WITH ranked AS (
  SELECT o.*, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY ts DESC) AS rn
  FROM orders o
)
SELECT r.user_id, u.name, r.id AS order_id, r.ts, r.amount
FROM ranked r
JOIN users u ON u.id = r.user_id
WHERE r.rn <= 2
ORDER BY r.user_id, r.ts DESC;

ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY ts DESC) numbers each user's orders 1, 2, 3… newest first. The outer WHERE rn <= 2 keeps the top two per user; the join attaches the user's name.

4. Strategic explanations

Why filter-on-window beats a correlated subquery

The pre-window approach — "keep an order if fewer than 2 orders for this user are more recent" — runs a correlated subquery once per order, each scan touching that user's rows: roughly O(n²) within a user. ROW_NUMBER sorts each partition once and labels rows in a single pass — O(n log n). You cannot put ROW_NUMBER directly in a WHERE (window functions are evaluated after WHERE), so it goes in a CTE or subquery and the filter wraps it. That two-layer shape is the canonical top-N-per-group answer.

The timestamp tie — ROW_NUMBER needs a tiebreaker

Bob's two 2025-01-09 orders are indistinguishable to ORDER BY ts DESC, so ROW_NUMBER assigns 1 and 2 in an unspecified order — rerun the query and the two could swap. Here both survive the rn <= 2 cut so the result set is stable, but if the cut were "top 1" the answer would be non-deterministic. Always add a unique tiebreaker for reproducibility: ORDER BY ts DESC, id DESC. Naming this unprompted is the senior move.

ROW_NUMBER vs RANK — pick by what ties should do

ROW_NUMBER gives every row a distinct number, so "top 2" returns exactly 2 rows even under a tie. RANK would give Bob's two tied orders the same rank 1, then the next order rank 3 — and rank <= 2 could return more or fewer than 2 rows. If the requirement is strictly "at most 2 rows", ROW_NUMBER is correct; if it is "all orders tied for the 2 most recent", RANK is. The spec here says top 2 orders — ROW_NUMBER.

The single-order user and why no HAVING is needed

Cara has one order; ROW_NUMBER simply labels it rn = 1, which passes <= 2. There is no minimum-count requirement, so nothing filters her out — "top 2" naturally means "up to 2". A user with zero orders never enters ranked at all, so they are absent; if the panel needed to show them with a blank, you would LEFT JOIN users to the ranked set instead of inner-joining.

Dialect — QUALIFY and lateral joins

The CTE-then-WHERE form is fully portable. Snowflake, BigQuery, and DuckDB offer QUALIFY ROW_NUMBER() OVER (...) <= 2, which filters on the window function inline — no wrapper needed. PostgreSQL has no QUALIFY; a slick Postgres alternative is LEFT JOIN LATERAL (SELECT ... FROM orders WHERE user_id = u.id ORDER BY ts DESC LIMIT 2) ON true, which can use an index to fetch just two rows per user. Mention these as the engine-specific upgrades.

Senior signal: solve top-N-per-group with ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...) in a CTE then filter rn <= N, add a unique tiebreaker to the ORDER BY so ties are deterministic, choose ROW_NUMBER over RANK because the result must be exactly N rows, and know QUALIFY / LATERAL as the dialect-specific shortcuts.
