A Community member has a daily bank-balance log and wants to compress it: whenever the balance stays at the same amount across consecutive days, collapse that stretch into a single row with a start date and an end date. This is the classic "gaps and islands" problem — and the reference solution gets it subtly, dangerously wrong.

1. The source data — bank_account

amountdate_stamp
10002024-01-01
10002024-01-02
15002024-01-03
15002024-01-04
15002024-01-05
10002024-01-06

The balance is 1000 for two days, rises to 1500 for three days, then drops back to 1000. That last row is the whole interview: the value 1000 appears in two separate, non-adjacent stretches. A correct answer reports three islands; a naive one merges the two 1000-runs into a single Jan-1-to-Jan-6 span that never actually happened.

2. The SQL — gaps-and-islands by difference of row numbers

WITH grouped AS (
  SELECT amount, date_stamp,
         ROW_NUMBER() OVER (ORDER BY date_stamp)
           - ROW_NUMBER() OVER (PARTITION BY amount ORDER BY date_stamp) AS grp
  FROM bank_account
)
SELECT amount,
       MIN(date_stamp) AS bank_start_date,
       MAX(date_stamp) AS bank_end_date
FROM grouped
GROUP BY amount, grp
ORDER BY bank_start_date;

This is the corrected query. The trick: a global row number minus a per-amount row number stays constant only while rows of the same amount are contiguous. That constant, grp, plus the amount, uniquely tags each island — so grouping on both yields one row per genuine run.

3. The resulting analysis

amountbank_start_datebank_end_date
10002024-01-012024-01-02
15002024-01-032024-01-05
10002024-01-062024-01-06

Three rows — and 1000 correctly appears twice, as two distinct islands (Jan 1–2 and the single-day Jan 6). The 1500 run spans Jan 3–5. A single-day island degrades gracefully: start and end are the same date.

4. Strategic explanations

The bug — the reference merges non-adjacent runs

The original solution partitions only by amount and takes MIN/MAX over that whole partition:

-- ORIGINAL (broken)
MIN(date_stamp) OVER (PARTITION BY amount) AS bank_start_date,
MAX(date_stamp) OVER (PARTITION BY amount) AS bank_end_date

For amount 1000 that partition holds Jan 1, Jan 2 and Jan 6, so it reports the run as Jan 1→Jan 6 — a six-day stretch that never happened, straddling three days at 1500. The question explicitly says continuous; the reference's flag CTE column is computed and then never used, and because it is a window function with no GROUP BY it also emits one row per input day (six rows, duplicated). It answers "first and last time we ever saw this balance", not "contiguous runs".

Why the row-number difference works

Walk amount 1000: global row numbers 1, 2, 6; per-amount row numbers 1, 2, 3. The differences are 0, 0, 3 — the first two share grp = 0, Jan 6 gets grp = 3. The instant a same-amount row is not adjacent, the global counter has advanced faster than the per-amount counter, so the difference jumps. (amount, grp) is therefore a unique island key. This is the canonical "Tabibitosan" / difference-of-row-numbers technique — worth knowing by name.

Date gaps vs row gaps — the deeper assumption

The row-number trick defines "consecutive" as adjacent rows when sorted, not adjacent calendar days. If the log skips Jan 4 entirely (no row at all), Jan 3 and Jan 5 are still adjacent rows and would be treated as one island despite the missing day. If a true calendar gap should break a run, group on the date arithmetic instead:

date_stamp - (ROW_NUMBER() OVER (PARTITION BY amount
              ORDER BY date_stamp))::int AS grp

Subtracting a row counter from the date itself yields a constant only across truly consecutive dates. Pick the variant that matches what the business means by "continuous".

The duplicate-date trap

If two rows share the same date_stamp and amount, ROW_NUMBER still numbers them 1 and 2 — arbitrarily — so the grouping holds, but the daily-log assumption is already violated. Confirm the grain: is it strictly one row per day, or can a day appear twice? If duplicates are possible, de-duplicate first or decide whether DENSE_RANK is the right counter.

The LAG-based alternative

An equivalent island detector compares each row to its predecessor and starts a new group on every change:

SUM(CASE WHEN amount = LAG(amount) OVER (ORDER BY date_stamp)
         THEN 0 ELSE 1 END) OVER (ORDER BY date_stamp) AS grp

This flags a boundary wherever the amount differs from the prior day, then a running SUM turns those flags into group ids. It is often more readable than the row-number difference and generalises cleanly when "same island" depends on more than equality.

Senior signal: recognise that MIN/MAX partitioned by value alone collapses non-adjacent runs — the bug here — and instead tag islands with the difference-of-row-numbers (or a LAG change-flag); then ask whether "continuous" means adjacent rows or adjacent calendar dates, and switch the grouping expression accordingly.
