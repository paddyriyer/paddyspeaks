Messenger's read-receipt latency drives a lot of UX decisions: blue-tick anxiety, "active now" status, last-seen indicators. The query computes per-conversation read latency (sent → read time) and sorts to find the slowest-responding conversations.

2. The SQL — julianday diff in minutes

SELECT conversation_id,
  COUNT(*) AS messages,
  COUNT(read_at) AS read_messages,
  ROUND(AVG((julianday(read_at) - julianday(sent_at)) * 1440), 1) AS avg_read_minutes,
  ROUND(MAX((julianday(read_at) - julianday(sent_at)) * 1440), 1) AS max_read_minutes
FROM messages
WHERE read_at IS NOT NULL
GROUP BY conversation_id
HAVING COUNT(read_at) >= 2
ORDER BY avg_read_minutes DESC;

3. Strategic explanations

The minutes conversion — *1440

julianday() returns days. Multiply by 1440 (24×60) to get minutes. Multiply by 86400 for seconds. Memorize the constants:

unitmultiplier
days1
hours24
minutes1440
seconds86400
The unread-message handling

WHERE read_at IS NOT NULL excludes unread messages from the latency calc — otherwise julianday(NULL) returns NULL, polluting the average. The COUNT(read_at) column counts only-read; COUNT(*) counts all messages — emit both so the reader sees the unread ratio.

The mean vs median trap

Average is sensitive to outliers — one message read 3 days later distorts the average for the whole conversation. Production uses median (or P50/P95) which SQLite doesn't have natively. Workaround:

WITH ranked AS (
  SELECT conversation_id, latency,
    ROW_NUMBER() OVER (PARTITION BY conversation_id ORDER BY latency) AS rn,
    COUNT(*) OVER (PARTITION BY conversation_id) AS n
  FROM /* the diff CTE */
)
SELECT conversation_id, AVG(latency) AS median_latency
FROM ranked WHERE rn IN (n/2, n/2 + 1)
GROUP BY conversation_id;
The "active hours" filter

A message sent at 3am and read at 8am has 5 hours of latency — but the recipient was asleep. To measure actual response speed, exclude overnight gaps:

WHERE NOT (CAST(strftime('%H', sent_at) AS INT) BETWEEN 0 AND 6)
Per-relationship-type cut

1-on-1 chats vs group chats have wildly different read-latency distributions: 1-on-1 is fast (focused), group is slow (everyone reads when they happen by). Always slice by chat type.

The "double tick" signal

WhatsApp's blue ticks (and Messenger's read receipt) are based on this exact metric. Disabled-receipts users skew the data: their read_at is NULL even when they read, depressing the answered-rate. Filter to receipts-on users for a clean denominator.

The product-decisioning use case

Messenger's "active now" status uses read-latency: under 5 min = "active now"; 5-30 min = "active recently"; over 30 = no status shown. The thresholds come from the distribution of this query's output.

Senior signal: memorize the julianday-to-minutes multiplier (1440), guard against NULL read_at, prefer median/P95 over AVG for latencies, and exclude overnight gaps for the actual response-speed metric. The "active now" UX threshold derivation is the over-the-bar tie-back.
