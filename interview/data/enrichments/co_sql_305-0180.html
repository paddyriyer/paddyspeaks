Coinbase's compensation team is modelling next year's payroll: each employee has a base salary and a projected raise percentage, and finance wants the projected new salary. The catch the interviewer planted — projected_pct arrived as messy text ("5%", "7.5"), and some employees have no projection at all.

1. The source data — employees

2800005%Analyst

Three quirks in four rows: employee 2's percent carries a stray % sign, employee 4's is a decimal, and employee 3 has no projection at all (NULL). Those messy and missing values are the entire interview.

2. The SQL — salary × (1 + pct/100)

SELECT emp_id, salary,
  salary * (
    1 + NULLIF(regexp_replace(projected_pct::text, '[^-.0-9]', '', 'g'), '')::numeric
        / 100.0
  ) AS projected
FROM employees;

regexp_replace strips projected_pct down to digits, sign and decimal point; NULLIF(…, '') handles a value that cleans to nothing; the cleaned percent is divided by 100 and applied as a growth factor.

Employee 1: 100000 × 1.10 = 110000. Employee 2's "5%" cleaned to 5 → 84000. Employee 4's 7.5% → 96750. Employee 3 returns NULL: with no projection, 1 + NULL/100 is NULL, and salary × NULL is NULL — the missing input propagates straight to the output.

4. Strategic explanations

The NULL employee — the decision the interviewer is waiting for

Employee 3's NULL projected salary is arithmetically correct — an unknown raise gives an unknown new salary. But a payroll model that drops an employee's number is useless: a SUM(projected) for the budget would be poisoned to NULL by that one row. Decide what "no projection" means and encode it. If it means "no raise", default the percent to 0:

salary * (1 + COALESCE(
  NULLIF(regexp_replace(projected_pct::text, '[^-.0-9]', '', 'g'), '')::numeric, 0
) / 100.0) AS projected
-- employee 3 now projects 120000 (a 0% raise) instead of NULL

NULL-propagation is the default; whether to keep it or COALESCE it to a 0% raise is a business call — make it explicitly.

Why NULLIF(…, '') is not optional

If projected_pct holds a blank string or text with no digits ("TBD", "n/a"), regexp_replace returns the empty string '' — and ''::numeric raises invalid input syntax for type numeric, aborting the whole query. NULLIF(cleaned, '') converts that empty result to NULL before the cast, so a junk value degrades one row to NULL instead of killing the report. It is a guard against the cast, distinct from the missing-value question above.

The text-column code smell — and the dialect note

All the regexp_replace scaffolding exists only because projected_pct is stored as text with inconsistent formatting ("5%" vs "10"). If it were a proper numeric column, the query collapses to salary * (1 + projected_pct / 100.0). Flag the upstream typing as the real bug: defensive parsing in a query is a symptom, not a fix. (The regexp_replace(…, 'g') here is standard Postgres; other engines spell the global-replace flag differently, but the technique ports.)

Integer division — why 100.0 not 100

The divisor is written 100.0 on purpose. Had the cleaned value been cast to integer and divided by an integer 100, 5 / 100 would truncate to 0 and every projection would equal the base salary — a silent, total failure of the model. Casting to numeric and dividing by 100.0 keeps the fractional percent intact. Always make at least one operand non-integer in a ratio.

Negatives, rounding, and the redundant columns

The regex keeps a leading -, so a "-3" projected_pct correctly models a pay cut — confirm with finance whether negative projections are expected or a data error. Also, the raw output carries a long fractional tail; payroll wants currency precision, so wrap it: ROUND(…, 2). And the wide schema ships salary alongside many other columns — verify salary is the base figure the raise should apply to, not a column that already includes a bonus.

Senior signal: recognise the NULL projection propagates to a NULL salary and decide deliberately whether to COALESCE it to a 0% raise, keep NULLIF(…, '') as a guard against the empty-string cast error, divide by 100.0 to dodge integer truncation, and call out the text-typed percentage column as the upstream bug the regex is papering over.
