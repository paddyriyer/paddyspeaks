EpicGames' player-analytics team wants each customer's profile alongside their lifetime spend. The natural shape is to aggregate orders in a subquery, then join it back to customers — but the interviewer is watching whether you notice that an inner join silently drops customers who never ordered, whether SELECT * across a join is safe, and how a derived table differs from a correlated subquery.

1. The source data — customers and orders

Ann has two orders, Ben has one — and Cara has none. That zero-order customer is the whole interview: does she appear in the result or vanish?

2. The SQL — join to an aggregating derived table

SELECT * FROM customers c JOIN
(SELECT customer_id, SUM(amount) AS total FROM orders GROUP BY customer_id) o
ON c.customer_id = o.customer_id;

The subquery o is a derived table: it pre-aggregates orders to one row per customer with a total. The outer query inner-joins customers to that.

Ann's two orders sum to 350; Ben's single order is 75. Cara is gone — she has no row in the orders subquery, so the inner join finds no match and drops her. Also note the result has two amount columns muddled by SELECT *: the customer-table amount (here 0) and the order total.

4. Strategic explanations

The inner-join drop — the central trap

"Each customer's spend" almost always means every customer, including those with zero orders shown as 0. The inner join cannot do that — no order rows, no match, no output. Use a LEFT JOIN and coalesce the NULL:

SELECT c.customer_id, c.name, COALESCE(o.total, 0) AS total
FROM customers c
LEFT JOIN (SELECT customer_id, SUM(amount) AS total FROM orders GROUP BY customer_id) o
  ON c.customer_id = o.customer_id;
-- Cara now appears with total = 0

For a churn or LTV analysis the zero-spend customers are often the most important rows — dropping them biases the whole dataset. Naming this is the senior signal.

SELECT * across a join is a liability

Both tables have an amount column, so the result carries two columns of the same name — ambiguous for any downstream consumer, and a hazard if code refers to amount by name. SELECT * also breaks silently when either table gains or drops a column. Always project an explicit, named column list: SELECT c.customer_id, c.name, o.total.

Derived table vs correlated subquery

The subquery here is a derived table — it runs once, independently, producing a small aggregated relation the planner joins like any table. Contrast a correlated scalar subquery in the SELECT list:

SELECT c.*, (SELECT SUM(amount) FROM orders o WHERE o.customer_id = c.customer_id) AS total
FROM customers c;

This naturally includes Cara (her subquery returns NULL), but conceptually re-evaluates per customer row. Modern planners often optimise both to a similar plan, yet the derived-table-plus-LEFT JOIN form is the clearer, more scalable idiom.

Aggregate before joining, not after

Pre-aggregating in the subquery keeps o at one row per customer, so the join does not fan out. If you instead joined raw orders to customers and aggregated afterward, each customer row would multiply by their order count before the GROUP BY — correct only if you remember to group by the customer key and are careful with any customer-level columns. Aggregate-then-join is the safer pattern.

Status filtering and the spend definition

SUM(amount) over all orders includes cancelled and refunded ones. "Lifetime spend" usually means realised revenue, so a WHERE status = 'completed' inside the subquery is the filter to raise. Confirm the definition before coding.

Senior signal: recognise that the inner join silently drops zero-order customers and switch to LEFT JOIN with COALESCE(total, 0) when every customer must appear, replace SELECT * with an explicit column list to kill the duplicate amount ambiguity, and explain why aggregating inside a derived table before joining avoids row fan-out.
