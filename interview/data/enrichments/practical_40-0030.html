An analytics team runs weekly KPIs by customer segment and wants an automated alert: flag every (segment, week) where the metric swung more than 20% versus the same segment's prior week, showing the old value and the percent change. This is a period-over-period comparison — the interviewer is probing partitioned LAG, and the two arithmetic landmines hiding in the threshold test.

1. The source data — weekly_metrics

week_startsegmentvalue(narrative)
2025-01-06enterprise1000first week — no prior, can't compare
2025-01-13enterprise1100+10% — under threshold
2025-01-20enterprise1500+36% — flagged
2025-01-27enterprise1450−3% — under threshold
2025-01-06smb500first week for smb — no prior
2025-01-13smb300−40% — flagged
2025-01-20smb310+3% — under threshold

Two edge cases. Each segment's first week (2025-01-06) has no prior row, so LAG yields NULL — it must not be flagged. And the change can be negative (smb drops 40%): the test has to catch swings in both directions, not just growth.

2. The SQL — LAG partitioned by segment, threshold in WHERE

WITH lagged AS (
    SELECT week_start, segment, value,
           LAG(value) OVER (PARTITION BY segment ORDER BY week_start) AS prev_value
    FROM weekly_metrics
)
SELECT week_start,
       segment,
       prev_value,
       value,
       ROUND(100.0 * (value - prev_value) / prev_value, 2) AS pct_change
FROM lagged
WHERE prev_value IS NOT NULL
  AND prev_value <> 0
  AND ABS(value - prev_value) * 1.0 / prev_value > 0.20
ORDER BY segment, week_start;

LAG(value) with PARTITION BY segment fetches the previous week's value within the same segment. The WHERE then keeps only rows where the absolute relative change exceeds 0.20 — with two guards in front of that test.

Only two rows survive. Both segments' 2025-01-06 rows are gone — prev_value is NULL, the guard removes them. The +10% and −3% and +3% weeks fall under the 20% bar. The smb 40% fall is flagged because ABS measures magnitude regardless of sign.

4. Strategic explanations

Why PARTITION BY segment is load-bearing

Drop the PARTITION BY and LAG reaches across segment boundaries: smb's first week would be compared to enterprise's last week — a meaningless cross-segment delta. The partition resets LAG at each segment so "prior week" always means the same segment. The ORDER BY week_start inside the window defines what "prior" means; both clauses are mandatory for a correct period-over-period.

The NULL guard — the first row of every partition

LAG has no previous row for the earliest week in each segment, so prev_value is NULL. WHERE prev_value IS NOT NULL drops those rows explicitly. Even without it the arithmetic would be NULL and the > 0.20 test would be unknown (so the row would not pass) — but relying on that is fragile and obscure. The explicit guard documents intent and is the senior habit.

The division-by-zero guard

If a segment's prior week had value = 0, then (value - prev_value) / prev_value divides by zero — an error in Postgres, or a NULL/Inf in laxer engines. WHERE prev_value <> 0 filters those rows before the division runs. Realise the semantic gap too: a jump from 0 to anything is an infinite percent change. Excluding it is one choice; flagging it via a separate "was zero, now non-zero" rule is another — state which the business wants.

ABS — catching drops as well as spikes

The brief says "changed by more than 20%", not "grew by". ABS(value - prev_value) / prev_value > 0.20 measures the magnitude of the swing, so a 40% collapse is flagged exactly like a 40% surge. Without ABS, the test catches only increases and silently ignores every decline — usually the more urgent alert. The pct_change column still shows the sign (-40.00) so a reader sees direction.

The integer-division trap — * 1.0 and 100.0

If value and prev_value are integers, (value - prev_value) / prev_value does integer division in SQLite and MySQL: a 36% change truncates to 0, fails > 0.20, and the alert silently never fires. The * 1.0 in the WHERE and the 100.0 literal in the SELECT force float division. Postgres avoids this only because the column here is NUMERIC; on integer columns the coercion is essential. This is the most dangerous bug in the query because it fails quietly.

Senior signal: compute period-over-period with LAG(value) OVER (PARTITION BY segment ORDER BY week), then defend the threshold test with three guards — prev_value IS NOT NULL for first rows, <> 0 against division-by-zero, and * 1.0/100.0 to force float division — and use ABS so the alert catches drops as well as spikes.
