Robinhood's markets team wants the all-time high and low price for each of the big tech tickers — the kind of "52-week range" figure that sits on every stock detail page. The query is a grouped MAX/MIN with an IN filter. An interviewer asking this is checking whether you put the filter in the right place and whether you understand what "extremes" means once dividends, splits, and time windows enter the picture.

META has three price points; AMZN, NFLX, and GOOG have only one each — for them high equals low. There is also a TSLA row that is not in the FAANG list; the filter must drop it.

2. The SQL — grouped MAX/MIN with IN filter

SELECT ticker, MAX(price) AS hi, MIN(price) AS lo
FROM stocks WHERE ticker IN ('META','AAPL','AMZN','NFLX','GOOG') GROUP BY ticker;

The WHERE keeps only the five named tickers, GROUP BY ticker makes one row per ticker, and MAX/MIN pick the extreme prices within each group.

Five rows — TSLA is correctly gone. META's range is 480–510 across its three days. NFLX, GOOG, and AMZN show hi = lo because each has a single observation: that is correct, not a bug, but worth flagging to a stakeholder who may misread a zero-width range as missing data. The row order is unspecified — the query has no ORDER BY.

4. Strategic explanations

WHERE before GROUP BY — filter then aggregate

WHERE runs before grouping, so non-FAANG rows are discarded before MAX/MIN ever see them — efficient and correct. The mistake is putting the ticker test in HAVING: HAVING filters after aggregation, so the engine would group every ticker (including TSLA), aggregate them all, then throw groups away. Same result, wasted work. Reserve HAVING for conditions on the aggregates themselves, e.g. HAVING MAX(price) > 500.

Five rows guaranteed? Only if every ticker has data

GROUP BY produces a row only for tickers that appear in the table. If GOOG had no rows in the date range, the result would have four rows, not five — the missing ticker is silently absent, not a row of NULLs. If a dashboard must show all five names always, drive the query from a ticker list and LEFT JOIN the prices, or VALUES-list the tickers. Don't assume the cardinality of the IN clause equals the cardinality of the result.

"Extremes" over what window? The split-and-dividend trap

MAX(price) over the whole table is the all-time extreme. A 52-week range needs a date filter: WHERE trade_date >= CURRENT_DATE - INTERVAL '1 year'. More subtly, raw price across a stock split is meaningless — a pre-split $600 and a post-split $150 are the "same" value. If the table stores unadjusted prices, the high/low must be computed on split-adjusted prices, or the range is nonsense. Surfacing that question ("are these prices split-adjusted?") is the senior move.

Want the date of the high, not just the price?

A frequent follow-up: "on which day did META peak?" MAX(price) gives the value but not the row. Use DISTINCT ON or a ranking window:

SELECT DISTINCT ON (ticker) ticker, trade_date, price
FROM stocks
WHERE ticker IN ('META','AAPL','AMZN','NFLX','GOOG')
ORDER BY ticker, price DESC;

This returns the peak row per ticker. If two days tie at the high, DISTINCT ON keeps one arbitrarily — add a tiebreaker to ORDER BY (e.g. , trade_date) for determinism.

Edge cases: NULL prices and single observations

MAX and MIN skip NULL prices, so a ticker whose only rows are NULL-priced yields (NULL, NULL). The single-observation tickers here (hi = lo) are legitimate — a one-day range is genuinely zero-width. Both are fine; just name them so nobody mistakes correct output for a data problem.

Senior signal: put the ticker filter in WHERE (not HAVING) so it runs before aggregation, do not assume the result has one row per IN-list entry, raise the split-adjustment and time-window questions that "extremes" really depends on, and reach for DISTINCT ON when the ask shifts from the price to the day it happened.
