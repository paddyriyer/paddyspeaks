Amazon's marketplace team wants every order ranked by value — biggest basket is rank 1 — for a "top orders" leaderboard. The reference solution reaches for DENSE_RANK(). The interviewer's real question is whether you can explain why it is DENSE_RANK and not RANK or ROW_NUMBER, because the three diverge exactly when two orders tie.

1. The source data — orders

order_idamount
1$500
2$300
3$500
4$200
5$300

There are two ties: orders 1 and 3 both at $500, orders 2 and 5 both at $300. The ties are the whole interview — they decide what number the order below a tie receives.

2. The SQL — DENSE_RANK over amount

SELECT order_id, amount,
       DENSE_RANK() OVER (ORDER BY amount DESC) AS rk
FROM orders;

DENSE_RANK() is a window function: it sorts every row by amount descending and assigns a rank. Tied rows share a rank, and — the defining trait — the next distinct value gets the very next integer, with no gap.

3. The resulting analysis

Both $500 orders are rank 1, both $300 orders are rank 2, and the $200 order is rank 3. The sequence is 1, 1, 2, 2, 3 — no number is skipped. That gap-free behaviour after a tie is precisely what distinguishes DENSE_RANK from RANK.

4. Strategic explanations

DENSE_RANK vs RANK vs ROW_NUMBER — the core comparison

All three sort the same way; they differ only in how they treat ties. On this exact data:

order_idamountROW_NUMBERRANKDENSE_RANK
1500111
3500211
230033 (skips 2)2
5300432
420055 (skips 4)3

ROW_NUMBER gives every row a unique integer and breaks ties arbitrarily by physical order. RANK shares a rank for ties but leaves a gap — after two 1s it jumps to 3. DENSE_RANK shares a rank and leaves no gap. Naming all three in one breath is the senior signal.

Which one does the leaderboard actually want?

"Top 3 distinct order sizes" wants DENSE_RANK — WHERE rk <= 3 reliably returns the three highest amounts however many orders tie. "Olympic-style standings" (two golds, then a bronze, no silver) wants RANK. "Pick exactly one row per tie group" wants ROW_NUMBER. The function encodes a business rule; confirm the rule before choosing.

Ties are resolved arbitrarily unless you add a tiebreaker

With ORDER BY amount DESC alone, orders 1 and 3 are interchangeable — their row order in the output is not defined, even though they share rk = 1. For ROW_NUMBER this is worse: it would hand them 1 and 2 based on planner whim, so the result is non-deterministic between runs. Add a deterministic tiebreaker:

DENSE_RANK() OVER (ORDER BY amount DESC, order_id ASC)

That does not change the dense ranks here, but it makes row order stable — essential when the leaderboard must look identical on every refresh.

NULL amounts and sort placement

In Postgres, ORDER BY amount DESC puts NULLs first by default — so an order with an unpriced NULL amount would rank above the $500 orders, which is absurd for a value leaderboard. Be explicit: ORDER BY amount DESC NULLS LAST, or filter WHERE amount IS NOT NULL before ranking.

The "top N per group" extension

Real leaderboards are usually segmented — top orders per customer, per month. Add PARTITION BY and the ranking restarts within each group:

SELECT order_id, amount, customer_id,
       DENSE_RANK() OVER (PARTITION BY customer_id ORDER BY amount DESC) AS rk
FROM orders;

Then WHERE rk = 1 in an outer query gives each customer's biggest order. This partition-then-rank-then-filter pattern is the workhorse behind most "top N" questions.

Senior signal: explain that DENSE_RANK gives gap-free ranks after ties (1,1,2) while RANK skips (1,1,3) and ROW_NUMBER ignores ties, match the function to the business rule, add a deterministic tiebreaker for stable output, and handle NULL amounts with NULLS LAST.
