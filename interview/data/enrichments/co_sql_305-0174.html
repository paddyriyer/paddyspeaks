Year-over-year revenue growth is the headline number on every annual report. The mechanic is identical to last-quarter / last-month / last-week comparisons — and LAG handles all of them. The trap is the first row (no prior year exists) and zero-prior-year edge cases that crash the percent calculation.

1. The source data — revenue

yearamount (USD)
2021$8,200,000
2022$10,500,000
2023$13,400,000
2024$12,800,000
2025$16,900,000

One row per year. The metric we want: this year's amount vs last year's amount, expressed as percentage change.

2. The SQL — LAG over yearly ordering

SELECT
  year,
  amount,
  100.0 * (amount - LAG(amount) OVER (ORDER BY year))
        / NULLIF(LAG(amount) OVER (ORDER BY year), 0) AS yoy_pct
FROM revenue;

Notice 2024's negative growth — the board pack would highlight this, frame the cause, and contrast 2025's recovery. The query computes the math; the narrative is the analyst's job.

4. Strategic explanations

Why LAG(amount) is called twice

The query references LAG(amount) OVER (ORDER BY year) in both the numerator and the denominator. Most engines (Postgres, Snowflake, BigQuery) evaluate window functions exactly once per row regardless of how many times they're textually referenced — the optimiser de-duplicates. SQLite is the same. So no perf cost; just visual repetition.

The first-row NULL — feature, not bug

2021 has yoy_pct = NULL because there's no prior year to compare against. LAG returns NULL on the first row of the partition; arithmetic with NULL produces NULL. This is the right behaviour — the alternative (defaulting to 0%) would mean "we grew 0% from nothing", which is misleading. Dashboards typically render NULL as "—".

Why NULLIF(..., 0) guards the denominator

If 2020's revenue was $0 (the company hadn't launched yet), LAG = 0, and the division crashes. NULLIF(LAG..., 0) turns 0 into NULL; the result becomes NULL; the query keeps running. Same pattern as the CTR query — guard every denominator on every rate metric.

QoQ, MoM, WoW — same shape, change the window

The query shape (amount - LAG(amount)) / LAG(amount) is the universal period-over-period growth pattern. Just adjust the order column:

-- Monthly (assuming one row per month):
LAG(amount) OVER (ORDER BY month)

-- Quarterly with explicit (year, quarter):
LAG(amount) OVER (ORDER BY year, quarter)

-- Per-region MoM (partition by region so growth is per-region not global):
LAG(amount) OVER (PARTITION BY region ORDER BY month)
The "two-year prior" comparison — LAG(amount, 2)

LAG takes an optional second argument: how many rows back. LAG(amount, 2) reaches back two rows. Useful for “same-quarter-last-year” comparisons in monthly data — to compare Q3-2025 to Q3-2024, the partition is per-month and you reach back 12 rows:

LAG(amount, 12) OVER (ORDER BY month)
Compounding growth (CAGR) — the over-the-bar follow-up

The interviewer's natural next question: "What's the compound annual growth rate over the full period?" CAGR isn't a window function — it's a single number derived from first and last:

SELECT
  POWER(MAX(amount) * 1.0 / MIN(amount), 1.0 / (MAX(year) - MIN(year))) - 1 AS cagr
FROM revenue;
-- 2021→2025 over 4 years: (16.9/8.2)^(1/4) - 1 ≈ 19.8% CAGR
-- (Note: assumes max=last and min=first; for non-monotonic data,
--  use first/last by year, not min/max amount)
Senior signal: name the period-over-period pattern ((curr − LAG(curr)) / LAG(curr) × 100, NULLIF-guarded), explain the first-row NULL as correct, and pre-empt the QoQ / per-partition / CAGR follow-ups. The CAGR finish lands above the bar.
