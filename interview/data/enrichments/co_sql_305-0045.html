The Site Reliability team watches a stream of daily metrics — request count, error rate, p99 latency, signup conversions. Single-day numbers tell you nothing; what matters is day-over-day change. Did errors spike? Did signups drop? LAG over a date-ordered series is the universal answer.

2. The SQL — LAG for the previous-day delta

SELECT
  metric_date,
  value,
  value - LAG(value) OVER (ORDER BY metric_date) AS day_over_day_change
FROM metrics;

4. Strategic explanations

Why LAG beats a self-join

The traditional way is FROM metrics a LEFT JOIN metrics b ON b.metric_date = a.metric_date - 1. That works but does a costly join even when the data is naturally ordered. LAG is a single sorted pass — same answer, far cheaper. On a million-row metrics table, the self-join takes seconds; LAG takes milliseconds.

The first-row NULL — feature, not bug

The earliest date has no prior day → LAG = NULL → delta is NULL. That's correct: you can't compute "change since yesterday" if there is no yesterday. Dashboards render NULL as "—" or skip the row.

Day-over-day raw delta vs percent change

Raw delta is good for absolute counts (1,310 → 1,180 = −130 fewer signups). Percent change is better for things that scale (a +130 change at scale 100 is huge; at scale 100,000 is noise). The percent variant:

SELECT metric_date, value,
  100.0 * (value - LAG(value) OVER (ORDER BY metric_date))
        / NULLIF(LAG(value) OVER (ORDER BY metric_date), 0) AS dod_pct_change
FROM metrics;
Anomaly detection — comparing to a rolling baseline, not just yesterday

"Yesterday vs today" is noisy. Real ops dashboards compare today to a 7-day rolling average — much more robust:

SELECT metric_date, value,
  AVG(value) OVER (ORDER BY metric_date ROWS BETWEEN 7 PRECEDING AND 1 PRECEDING) AS baseline_7d,
  ROUND(100.0 * (value - AVG(value) OVER (...)) / AVG(value) OVER (...), 1) AS pct_vs_baseline
FROM metrics;

Pages fire when pct_vs_baseline exceeds ±20% (tuneable per metric).

Multi-metric ranking — PARTITION BY metric_name

Real systems track 100+ metrics in the same table. Compare each to its own history:

value - LAG(value) OVER (PARTITION BY metric_name ORDER BY metric_date) AS dod_change

Without the PARTITION, LAG would compare today's signup count to yesterday's error rate — nonsense. Always partition by the entity whose history you want.

Day-7 / day-28 comparison via LAG(value, N)

LAG(value, 7) reaches back 7 rows. Useful for "same-day-last-week" comparisons (controlling for day-of-week effects):

value - LAG(value, 7) OVER (ORDER BY metric_date) AS week_over_week_change
Senior signal: name the rolling-baseline upgrade unprompted (raw day-over-day is too noisy for production alerting), articulate the PARTITION BY as the safety net for multi-metric tables, and pre-empt the WoW / MoM follow-ups with LAG(value, 7) / LAG(value, 30).
