An airline operations team wants to flag flights where two confirmed passengers sit in adjacent seats — seat numbers differing by exactly one — perhaps for contact-tracing or seat-swap analysis. The reference solution reaches for window functions and a self-join, but it contains a subtle, classic bug: it compares row-number positions instead of actual seat numbers. The interviewer is watching whether you spot that "adjacent in rank" is not the same as "adjacent in seat".

2. The SQL

Original solution (flagged broken):

with cte1 as (
  select passengername, seatnumber, bookingdate, flightid, status,
    row_number() over (partition by flightid, bookingdate order by seatnumber asc) as flag_one
  from seatreservations
), cte2 as (
  select *,
    row_number() over (partition by flightid, bookingdate order by seatnumber asc) as flag_two
  from seatreservations
  where status = 'Confirmed'
)
select one.flightid as Flight_ID,
    one.passengername as Passenger_One,
    two.passengername as Passenger_Two
from cte1 as one
inner join cte2 as two
  on one.flightid = two.flightid
 and one.bookingdate = two.bookingdate
 and one.passengername <> two.passengername
where one.status = 'Confirmed'
  and (flag_one - flag_two) = 1;

This subtracts two independently computed row_number() values — one over all rows, one over confirmed rows only. A difference of 1 in those rankings has no relationship to seats differing by 1. It returns Ivan/Judy on flight 300 (seats 1 and 3, not adjacent) and misses Dave/Eve on flight 100.

Corrected solution — compare the seat numbers themselves:

SELECT a.flightid AS flight_id,
       a.passengername AS passenger_one,
       b.passengername AS passenger_two
FROM seatreservations AS a
JOIN seatreservations AS b
  ON a.flightid = b.flightid
 AND b.seatnumber = a.seatnumber + 1
WHERE a.status = 'Confirmed'
  AND b.status = 'Confirmed'
ORDER BY a.flightid, a.seatnumber;

4. Strategic explanations

Why the rank-difference approach is wrong

ROW_NUMBER() produces a dense 1,2,3,… sequence regardless of the gaps between the values it ranks. Two passengers ranked consecutively could sit in seats 1 and 9 if seats in between were empty. "Adjacency" is a property of the seat number domain, so the comparison must be on seatnumber itself: b.seatnumber = a.seatnumber + 1.

The self-join is the natural tool

Joining the table to itself on b.seatnumber = a.seatnumber + 1 pairs every confirmed seat with the confirmed seat one higher. Filtering status = 'Confirmed' on both aliases ensures cancelled rows (Carol) never form or block a pair — Dave(4) and Eve(5) still match because the join looks at numbers, not physical adjacency in the table.

The window-function alternative — LEAD

You can avoid the self-join entirely. Restrict to confirmed rows, then within each flight look one row ahead by seat:

WITH conf AS (
  SELECT flightid, passengername, seatnumber,
         LEAD(seatnumber)   OVER (PARTITION BY flightid ORDER BY seatnumber) AS next_seat,
         LEAD(passengername) OVER (PARTITION BY flightid ORDER BY seatnumber) AS next_name
  FROM seatreservations WHERE status = 'Confirmed')
SELECT flightid, passengername, next_name
FROM conf WHERE next_seat = seatnumber + 1;

Crucially the WHERE status = 'Confirmed' must run before LEAD, so the "next" seat is the next confirmed one. LEAD on the full table would let a cancelled seat sit between two confirmed ones and break detection.

Duplicates and the "flight list" follow-up

The join naturally lists every adjacent pair once (ordered low-to-high seat). If the question only wants the flights that have at least one such pair, wrap it: SELECT DISTINCT flight_id FROM (…). If it wants pairs as unordered sets, the + 1 condition already prevents the mirrored duplicate.

Senior signal: catch that subtracting two ROW_NUMBER() sequences measures rank-adjacency, not seat-adjacency, and rebuild the query to compare seatnumber directly — via a self-join on seatnumber + 1 or a LEAD over confirmed-only rows, filtering status before the window so cancelled seats neither create nor mask a pair.
