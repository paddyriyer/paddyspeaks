A multi-tenant SaaS ticket table is a study in three compounding concerns: data isolation (one tenant must never see another's rows), query performance (every lookup is scoped to a single tenant), and retention policy heterogeneity (enterprise customers often contractually require different data-retention windows). The interviewer is probing your choice of partitioning strategy, how you enforce isolation at the database layer rather than relying solely on the application, and how you index efficiently when tenant_id must lead every predicate.

1. The requirements — what the design must capture

  • Tenant isolation: every row is owned by exactly one tenant; the database must enforce this via row-level security, not only application logic.
  • Efficient per-tenant queries: tenant_id must be the leading column of every index so the planner can prune partitions and use index-only scans for tenant-scoped workloads.
  • Per-tenant retention: a retention_until date column carries the computed expiry; a scheduled job or Postgres policy can delete expired rows without scanning all tenants.
  • Status lifecycle: tickets move through open → pending → resolved → closed; the schema must support filtering by open/pending status efficiently.
  • Assignment: tickets can be assigned to a support user; assigned_user_id is nullable (unassigned tickets are valid).
  • Partial index for hot status values: the overwhelming majority of queries filter on open or pending tickets; a partial index on those values avoids indexing the large resolved/closed tail.

2. The DDL — the CREATE TABLE

CREATE TABLE support_ticket (
  tenant_id         BIGINT       NOT NULL,                  -- first column on every index for isolation
  ticket_id         BIGINT       NOT NULL,                  -- unique within tenant
  created_at        TIMESTAMPTZ  NOT NULL,
  updated_at        TIMESTAMPTZ  NOT NULL,
  status            VARCHAR(20)  NOT NULL,                  -- open / pending / resolved / closed
  priority          SMALLINT     NOT NULL,
  assigned_user_id  BIGINT,
  subject           VARCHAR(200) NOT NULL,
  body              TEXT,
  retention_until   DATE,                                   -- per-tenant policy result
  PRIMARY KEY (tenant_id, ticket_id)
)
PARTITION BY HASH (tenant_id);  -- spreads tenants across partitions to avoid one giant partition

CREATE INDEX ix_ticket_open ON support_ticket (tenant_id, status, created_at DESC)
  WHERE status IN ('open', 'pending');

-- Postgres RLS:
ALTER TABLE support_ticket ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON support_ticket
  USING (tenant_id = current_setting('app.tenant_id')::BIGINT);

4. Strategic explanations

Hash partitioning vs. range partitioning for multi-tenancy

Range partitioning on tenant_id creates risk of hotspots if one large tenant gets its own partition while many small tenants share another. Hash partitioning on tenant_id distributes rows uniformly across a fixed number of child tables, regardless of tenant size distribution, which prevents the "whale tenant" problem. The downside is that you cannot drop a hash partition to delete one tenant's data — per-tenant deletion still requires row-level deletes. If data-isolation-by-partition-drop is required, use range partitioning with one partition per tenant range, or a separate schema per tenant (stronger isolation, higher management cost).

Row-level security as the second line of defence

Application-layer filtering (WHERE tenant_id = ?) is the first line; it fails open if a developer forgets the clause. Postgres RLS is the second line: CREATE POLICY tenant_isolation ON support_ticket USING (tenant_id = current_setting('app.tenant_id')::BIGINT) makes every query implicitly add the tenant predicate, so a missing WHERE still cannot leak cross-tenant data. The current_setting value is set at connection time (SET LOCAL app.tenant_id = ?) in the application's request handler. RLS is bypassed by superusers and table owners — the application role must not be a superuser.

Partial index on active statuses

In a mature ticket system, resolved and closed tickets vastly outnumber open and pending ones — perhaps 90%+ of rows. A full index on (tenant_id, status, created_at DESC) would index all of these, but the dashboard queries only ever need the active fraction. The WHERE status IN ('open', 'pending') predicate on the index creates a partial index that is 10x smaller and fits in shared buffer cache much more readily. This pattern is a high-leverage optimization that seniors should name immediately.

Per-tenant retention policy

Storing retention_until as a column allows a single DELETE WHERE retention_until < CURRENT_DATE AND status NOT IN ('open','pending') to handle all tenants in one pass — no per-tenant scheduled job proliferation. An index on (retention_until) (possibly partial, excluding NULLs) makes the purge fast. For stronger compliance guarantees, move expired rows to an immutable archive store before deletion.

Dialect notes

Hash partitioning is Postgres-native (available since Postgres 11). In Snowflake, cluster by (tenant_id, status); micro-partition pruning achieves similar isolation without explicit partitions. In CockroachDB or YugabyteDB (distributed SQL), leading tenant_id on the PK automatically routes all tenant data to the same shard range, keeping cross-shard joins rare. RLS is supported in Postgres and Snowflake but not in all engines.

Senior signal: lead every index with tenant_id, use hash partitioning to avoid whale-tenant hotspots, enforce isolation with Postgres RLS as a second line of defence (not just app-layer filtering), and add a partial index on active statuses only — the inactive tail is the majority of rows and does not need to be in a hot B-tree.
