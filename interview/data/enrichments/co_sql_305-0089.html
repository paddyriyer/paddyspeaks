Meta's content-analytics team is hunting unengaged pages — pages that get traffic but generate zero interaction (no likes, comments, or shares). These are content that draws eyeballs and converts nothing, prime candidates for a redesign. The twist in the reference solution is that it does not trust engagements to be a clean number — it scrubs the value through a regex before comparing.

Pages 1 and 4 are the targets: real traffic, zero engagement. Page 2 is the edge case — zero engagement but also zero views, so it is not "unengaged", it is simply unseen and must be excluded.

2. The SQL — views > 0 and scrubbed engagements = 0

SELECT page_id FROM pages
WHERE views > 0
  AND NULLIF(regexp_replace(engagements::text, '[^-.0-9]', '', 'g'), '')::numeric = 0;

The first predicate keeps only pages with traffic. The second casts engagements to text, strips every character that is not a digit, minus sign, or dot, turns an emptied string into NULL, casts back to numeric, and checks it equals zero.

Pages 1 and 4 surface — both have views and zero engagement. Page 2 is correctly excluded by views > 0 despite its zero engagement. Pages 3 and 5 have non-zero engagement and fall out.

4. Strategic explanations

Why the views > 0 guard is essential

"Unengaged" specifically means traffic without interaction. A page with zero views and zero engagement (page 2) is not under-performing — it has simply never had the chance. Dropping the views > 0 predicate would lump these unseen pages in with the genuinely unengaged ones and pollute the redesign backlog. The guard encodes the actual definition of the metric.

What the regex scrub is defending against

regexp_replace(engagements::text, '[^-.0-9]', '', 'g') deletes every character that is not a digit, minus, or dot. This only matters if engagements is not a clean numeric column — e.g. it is stored as text and may contain values like "1,024", "12 likes", or "~0". The scrub coerces such junk into a parseable number. If engagements is genuinely an INT column (as a sane schema would have it), the whole regex is unnecessary defensive machinery and the predicate collapses to:

SELECT page_id FROM pages
WHERE views > 0 AND engagements = 0;

The over-the-bar move is to say so: "if the column is numeric, this simplifies; the regex only earns its place against dirty text input."

The NULLIF guards an empty-string cast

If engagements is something like "n/a", the regex strips it to an empty string ''. Casting ''::numeric throws an invalid input syntax error and the whole query dies. NULLIF(..., '') turns that empty string into NULL first; NULL::numeric is harmless NULL, and NULL = 0 is unknown — so the row is simply not selected rather than crashing the query. It is a quiet but important piece of defensive plumbing.

NULL engagements — missing vs zero

A page with NULL engagements is debatable: is "engagement unknown" the same as "zero engagement"? The scrubbed expression treats NULL input as NULL output, and NULL = 0 is unknown, so NULL-engagement pages are excluded. If the business wants missing engagement treated as zero, that must be an explicit COALESCE(engagements, 0) = 0 decision, not an accident of the regex.

The likely follow-up — an engagement rate

A binary "zero or not" is coarse. Analysts usually want the engagement rate — engagements per view — and a threshold like "below 1%":

SELECT page_id, engagements::numeric / views AS engagement_rate
FROM pages
WHERE views > 0
ORDER BY engagement_rate ASC;

The views > 0 guard does double duty here — it also prevents a division-by-zero. Anticipating that "unengaged" really wants a rate, not a flag, is the senior instinct.

Senior signal: explain that views > 0 encodes the real definition (traffic without interaction, not merely unseen), recognise the regex+NULLIF as defence against a dirty text column — and that it is redundant if engagements is properly numeric — and propose an engagement-rate threshold as the more useful follow-up metric.
