Plus500's growth team wants the top 10 customers by lifetime spend to seed a VIP outreach list. The reference solution computes the ranking inside a subquery, then the outer query selects only customer_id from it. That extra layer of nesting is doing real work — an interviewer asking this wants to know whether you understand why the LIMIT has to live in the inner query, and whether you noticed the answer drops everything except the id.

Customer 101 has two orders that must be summed before ranking — 500 + 300 = 800. With only six distinct customers and a LIMIT 10, every customer survives; the limit becomes a no-op, which is itself worth flagging.

2. The SQL — subquery + LIMIT

SELECT customer_id FROM (
  SELECT customer_id, SUM(amount) AS total FROM orders GROUP BY customer_id ORDER BY total DESC LIMIT 10
) t;

The inner query aggregates per customer, sorts by total descending, and keeps the top 10. The outer query then projects away total, returning just the ids.

Six rows, ordered by spend: 102 ($900), 101 ($800), 106 ($700), 104 ($400), 105 ($250), 103 ($150). Note what is missing — the totals. The result tells you who the top customers are but not how much they spent, and the outer query's row order is technically not guaranteed.

4. Strategic explanations

Why the LIMIT cannot move to the outer query

You cannot write ... ORDER BY total DESC LIMIT 10 in the outer query, because total is not in its select list — the outer query only sees customer_id. The aggregation, the sort, and the cut all have to happen together in the inner query while total still exists. The subquery is not decoration; it is the only place the ranking can be expressed.

The outer query loses the ordering guarantee

SQL only guarantees row order when the outermost query has an ORDER BY. Here the ORDER BY is buried in the subquery, so the planner is free to return the ten rows in any order. The LIMIT still picks the correct ten, but their sequence is undefined. If the consumer renders a ranked list, lift the sort out:

SELECT customer_id FROM (
  SELECT customer_id, SUM(amount) AS total
  FROM orders GROUP BY customer_id
  ORDER BY total DESC LIMIT 10
) t
ORDER BY total DESC;
-- ERROR: total not visible here

That fails for the same reason — so keep total in the projection if you need to re-sort, or just SELECT customer_id, total and let the consumer use both.

Ties at the boundary — LIMIT is non-deterministic

If the 10th and 11th customers are tied on spend, LIMIT 10 picks one of them arbitrarily. For a VIP list that is unfair. RANK() makes the tie explicit and lets you decide:

SELECT customer_id FROM (
  SELECT customer_id, SUM(amount) AS total,
         RANK() OVER (ORDER BY SUM(amount) DESC) AS rk
  FROM orders GROUP BY customer_id
) t
WHERE rk <= 10;

This returns all customers in the top 10 ranks — possibly 11 or 12 rows if there are ties. Naming that trade-off ("strict top 10, or everyone tied for 10th?") is the senior move.

The status filter the question implies

"Top customers by spend" almost always means realised revenue. The sample data is all completed, but a real orders table carries cancelled and pending rows that inflate the totals. Confirm the definition and filter: WHERE status = 'completed' inside the subquery, before the GROUP BY.

Edge case: customers with zero orders

A customer who never ordered has no rows in orders, so GROUP BY never produces them — correct here, since they cannot be a top spender. But the same query shape is wrong for "all customers and their spend": that needs a LEFT JOIN from a customers table with COALESCE(SUM(amount), 0).

Senior signal: explain that the LIMIT must sit in the subquery because the ranking column is not visible outside it, note that the outer query loses the ordering guarantee, raise the LIMIT-tie ambiguity and offer RANK(), and add the status = 'completed' filter that turns raw spend into lifetime value.
