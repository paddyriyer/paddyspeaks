NULL is not a value — it is the absence of a value — and that distinction makes ordinary equality joins silently wrong when the join key can be NULL. A standard ON a.region = b.region predicate evaluates NULL = NULL as UNKNOWN (not TRUE), causing NULL-keyed rows to be dropped. IS NOT DISTINCT FROM is the SQL-standard null-safe equality predicate that evaluates NULL IS NOT DISTINCT FROM NULL as TRUE. The interviewer is probing whether you recognise this three-valued-logic trap and know the fix.

Match logic: "North" ↔ "North" (match), "South" has no counterpart in snap_b (no match), "East" has no counterpart in snap_a (no match), NULL ↔ NULL (match — the key requirement). A plain equality join would drop the NULL-NULL pair.

2. The SQL — NULL-safe join with IS NOT DISTINCT FROM

SELECT a.region AS region, a.metric AS a_metric, b.metric AS b_metric
FROM snap_a a
JOIN snap_b b ON a.region IS NOT DISTINCT FROM b.region
ORDER BY a.region NULLS LAST;

The predicate a.region IS NOT DISTINCT FROM b.region returns TRUE when both are equal (non-null) or when both are NULL. It is logically equivalent to (a.region = b.region) OR (a.region IS NULL AND b.region IS NULL) but more concise and idiomatic. The NULLS LAST in ORDER BY places the NULL-region row at the end.

4. Strategic explanations

Three-valued logic and why NULL = NULL is UNKNOWN

SQL operates on three-valued logic: TRUE, FALSE, UNKNOWN. Any comparison involving NULL yields UNKNOWN. A JOIN predicate must evaluate to TRUE to admit a row pair — UNKNOWN (which is what NULL = NULL produces) is treated as FALSE for this purpose. This means two rows with NULL in the join key will never match under ordinary equality, even though they represent the "same" unspecified value. IS NOT DISTINCT FROM short-circuits this by handling NULL explicitly outside the three-valued system.

IS NOT DISTINCT FROM vs the manual workaround

The equivalent explicit predicate is (a.region = b.region) OR (a.region IS NULL AND b.region IS NULL). This is verbose and easy to get wrong (e.g., writing AND where you need OR). IS NOT DISTINCT FROM is the ISO SQL:1999 standard form and is preferred because it is atomic, readable, and cannot have an off-by-one logic error. Its complement IS DISTINCT FROM is the null-safe inequality — it returns FALSE when both sides are NULL, unlike <> which returns UNKNOWN.

NULLS LAST in ORDER BY

PostgreSQL's default ORDER BY ASC behaviour places NULLs last (ASC NULLS LAST). However, this is not universal: MySQL and SQL Server sort NULLs first in ascending order. Adding NULLS LAST explicitly is therefore a portability and clarity best practice: it states your intent unambiguously and behaves consistently across engines (where supported). In MySQL the workaround is ORDER BY region IS NULL, region.

Performance: IS NOT DISTINCT FROM and indexes

Most query optimisers cannot use a B-tree index on a column to satisfy IS NOT DISTINCT FROM for the NULL-matching case, because B-tree indexes do not index NULL values in most engines. PostgreSQL can satisfy the non-null branch via an index but must fall back to a seq scan (or an IS NULL index, if explicitly created) for the NULL rows. This is usually fine when NULLs are rare, but worth mentioning. An alternative: replace NULL with a sentinel value (e.g., '__UNSPECIFIED__') at ingestion time to allow a plain equality join with full index support — a design-level trade-off.

Dialect notes

IS NOT DISTINCT FROM is supported in PostgreSQL, SQLite, DuckDB, and (as of 2022) Snowflake. SQL Server uses WHERE EXISTS (SELECT a.region INTERSECT SELECT b.region) as the idiomatic null-safe equality test, or explicitly (a.region = b.region OR (a.region IS NULL AND b.region IS NULL)). MySQL 8.0+ has the <=> operator (null-safe equality) in the WHERE clause but not in ON; BigQuery uses a.region = b.region OR (a.region IS NULL AND b.region IS NULL).

Senior signal: use IS NOT DISTINCT FROM in the ON clause whenever the join key can legitimately be NULL — plain equality silently drops NULL-NULL pairs because NULL = NULL is UNKNOWN in SQL's three-valued logic; add NULLS LAST to ORDER BY to be explicit about NULL sort position; and name the engine-specific equivalents (SQL Server's INTERSECT trick, MySQL's <=>) to show cross-platform fluency.
