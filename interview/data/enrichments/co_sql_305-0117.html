Uber's CFO opens the daily ride-revenue tab: "are we tracking ahead of last quarter's pace?" — the answer is always a cumulative line. Running totals are the bedrock of every finance dashboard. The query is a single window function; the subtleties live in the frame clause and the missing-day trap.

2. The SQL — SUM as a window function

SELECT sale_date, amount,
  SUM(amount) OVER (ORDER BY sale_date ROWS UNBOUNDED PRECEDING) AS cumulative
FROM sales
ORDER BY sale_date;

4. Strategic explanations

ROWS UNBOUNDED PRECEDING — the explicit frame

Without a frame clause, SUM(...) OVER (ORDER BY x) uses the default RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — which sums all rows with the same ORDER BY value as the current row, plus everything before. For unique dates the result is identical to ROWS; the moment you have two rows on the same date, RANGE gives you both rows' totals on each row of that date. Always pin the frame:

-- ROWS = physical row count, fast, what you usually want
SUM(amount) OVER (ORDER BY sale_date ROWS UNBOUNDED PRECEDING)

-- RANGE = include all peers (rows with same ORDER BY key) — surprising on ties
SUM(amount) OVER (ORDER BY sale_date RANGE UNBOUNDED PRECEDING)
The missing-day trap — build a date spine first

If April 5 had zero rides, the cumulative line jumps from $4,430 to $6,080 with no intermediate row. Charts plotted on a continuous axis show a flat line across the gap, not a "$0 day, no progress" point. Production fix: generate a date spine and LEFT JOIN:

WITH dates AS (
  SELECT generate_series(DATE '2026-04-01', DATE '2026-04-30', '1 day'::interval)::DATE AS d
)
SELECT d.d, COALESCE(s.amount, 0) AS amount,
  SUM(COALESCE(s.amount, 0)) OVER (ORDER BY d.d ROWS UNBOUNDED PRECEDING) AS cumulative
FROM dates d
LEFT JOIN sales s ON s.sale_date = d.d
ORDER BY d.d;
Per-region cumulative — PARTITION BY

The CFO's follow-up is always "break it down by region or city." Partition the window:

SUM(amount) OVER (
  PARTITION BY city
  ORDER BY sale_date
  ROWS UNBOUNDED PRECEDING
) AS cumulative_by_city

Each city's accumulator resets at its first row. Without PARTITION BY, all cities share one running sum — useless for comparison.

Year-to-date that resets every January

The actual finance ask is YTD: cumulative from Jan 1 of the current year, then resets. Two PARTITION BYs:

SUM(amount) OVER (
  PARTITION BY EXTRACT(YEAR FROM sale_date)
  ORDER BY sale_date
  ROWS UNBOUNDED PRECEDING
) AS ytd_total
Sliding window vs cumulative window — the senior distinction

Drop UNBOUNDED PRECEDING and put a numeric anchor and you have a trailing window, not a cumulative:

-- Last-7-days running total
SUM(amount) OVER (ORDER BY sale_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS rolling_7d

-- Forward-looking 30-day projection (rare but useful for backlog)
SUM(amount) OVER (ORDER BY sale_date ROWS BETWEEN CURRENT ROW AND 29 FOLLOWING) AS next_30d

Cumulative gives "how far ahead are we?". Trailing gives "is recent demand strong?". They answer different questions; finance dashboards usually need both.

The pace-vs-target follow-up

Cumulative alone isn't the punchline — the punchline is "are we ahead of plan?" Join to a daily target:

SELECT s.sale_date, s.cumulative,
  t.cumulative_target,
  s.cumulative - t.cumulative_target AS variance,
  ROUND(100.0 * s.cumulative / NULLIF(t.cumulative_target,0), 1) AS pct_of_plan
FROM cumulative_sales s
LEFT JOIN daily_targets t USING (sale_date);
The "this month vs last month" overlay — the actual chart

Most finance dashboards plot two cumulatives on the same axis indexed to month-day-1. The trick is to extract the day-of-month and partition by month:

SELECT DATE_TRUNC('month', sale_date) AS month,
  EXTRACT(DAY FROM sale_date) AS dom,
  SUM(amount) OVER (
    PARTITION BY DATE_TRUNC('month', sale_date)
    ORDER BY sale_date
    ROWS UNBOUNDED PRECEDING
  ) AS mtd
FROM sales;

Plot mtd against dom with one line per month — instant pace comparison.

Senior signal: name ROWS vs RANGE (with the ties caveat), pre-empt the missing-day trap with a date spine, and offer the per-region PARTITION BY + YTD reset variants without being asked. The MTD pace overlay (PARTITION BY month, ORDER BY date) is the over-the-bar finish.
