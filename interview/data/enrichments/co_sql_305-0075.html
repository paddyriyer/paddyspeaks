Vercel's people-ops team reports a training-completion rate per employee — what fraction of assigned courses each person has finished. The schema hides a real-world mess: completed is not a clean boolean. The upstream system has written it as 'true', 't', '1', and various casings, all meaning the same thing. An interviewer wants to see whether you can normalise that flag correctly and still get the percentage arithmetic right.

1. The source data — training

emp_idcourse_idcompleted

The trap is the mixed encoding. Employee 1's three rows use 'true', 'false', and '1' — the '1' must count as completed. Employee 2 is fully done (two 't' values). Employee 3 has finished nothing ('0' and 'false'). A query that only recognises one spelling of "true" will undercount.

2. The SQL — normalise then average a flag

SELECT emp_id,
  100.0 * SUM(CASE WHEN completed::text IN ('1','true','t','TRUE','True') THEN 1 ELSE 0 END)/COUNT(*) AS completion_rate
FROM training GROUP BY emp_id;

The CASE maps any recognised "truthy" spelling to 1 and everything else to 0. SUM of those 1s is the count of completed courses; COUNT(*) is the total assigned. The ratio, times 100.0, is the percentage.

3. The resulting analysis

Employee 1 scores 66.67% — 2 of 3 (the 'true' and the '1' both counted, the 'false' did not). Employee 2 is 100% (2 of 2). Employee 3 is 0% (0 of 2). The '1' being counted is the proof the normalisation worked — a single-spelling check would have given employee 1 only 33%.

4. Strategic explanations

The "sum a 0/1 flag, divide by count" idiom

SUM(CASE WHEN <cond> THEN 1 ELSE 0 END) / COUNT(*) is the canonical way to compute "what fraction of rows satisfy a condition". Postgres also lets you write it with a filtered aggregate, which many reviewers find cleaner:

SELECT emp_id,
  100.0 * COUNT(*) FILTER (WHERE completed::text IN ('1','true','t','TRUE','True'))
        / COUNT(*) AS completion_rate
FROM training GROUP BY emp_id;

Identical result; the FILTER clause states intent more directly than a CASE buried in a SUM.

The integer-division trap — why 100.0 leads

SUM(...) and COUNT(*) are both integers. Writing SUM(...) / COUNT(*) first does integer division: employee 1's 2/3 becomes 0, and after * 100 the rate is a flat 0% — catastrophically wrong. Putting 100.0 first promotes the whole chain to numeric, so the division happens in numeric arithmetic. SUM(...)::numeric / COUNT(*) achieves the same. This is the single most common bug in percentage queries.

The fragile allow-list — the real weakness

The IN ('1','true','t','TRUE','True') list is a guess at every spelling the source might emit. It misses 'TRUE ' with a trailing space, 'yes', 'Y', 'T' uppercase. A sturdier normalisation lower-cases and trims first, shrinking the list to the real distinct values:

CASE WHEN lower(trim(completed::text)) IN ('1','true','t','yes','y') THEN 1 ELSE 0 END

Best of all, if the source were a real boolean column the entire CASE collapses to SUM(completed::int) — the cleanup is a tax imposed by a bad data type. Name that.

What happens to unrecognised values

The ELSE 0 means anything not in the allow-list — including NULL, an empty string, or a typo like 'ture' — is silently treated as "not completed". That is a decision: a NULL completed might mean "not done" or "status unknown". If unknown rows should be excluded from the denominator rather than counted as failures, filter them out before aggregating, or the rate is artificially depressed.

The empty-assignment edge case

An employee assigned zero courses would have COUNT(*) = 0 and the division would be x/0. They produce no group here because they have no rows — but if you LEFT JOIN from an employee roster to include everyone, guard it: / NULLIF(COUNT(*), 0), yielding NULL ("no courses assigned") rather than a divide-by-zero error.

Senior signal: normalise the messy completed flag with lower(trim(...)) instead of guessing every casing, keep the numeric promotion (100.0 first) so integer division never zeroes the rate, decide deliberately whether NULL/unknown rows are failures or excluded, and call out that a proper boolean column would erase the whole CASE.
