Cloudflare's logistics team wants stock totals broken down by both warehouse and product category — not one or the other, but every (warehouse, category) combination. It is the most basic multi-column GROUP BY, and precisely because it looks trivial, the interviewer uses it to check whether you understand what a multi-column grouping key actually is, and what it does not give you.

1. The source data — inventory

warehouse_idcategoryqty
1Electronics50
1Electronics30
1Apparel20
2Electronics40
2Apparel60
2Apparel10

Six rows. Warehouse 1 has two Electronics rows (a category split across multiple SKUs or shipments); warehouse 2 has two Apparel rows. The grouping must collapse each (warehouse, category) pair into a single summed row.

2. The SQL — GROUP BY warehouse_id, category

SELECT warehouse_id, category, SUM(qty)
FROM inventory
GROUP BY warehouse_id, category;

The grouping key is the pair of columns. Every distinct combination of warehouse_id and category becomes one output row, with SUM(qty) totalling all input rows in that combination.

3. The resulting analysis

warehouse_idcategorysum
1Apparel20
1Electronics80
2Apparel70
2Electronics40

Four rows — one per (warehouse, category) pair. Warehouse 1's two Electronics rows (50 + 30) collapsed to 80; warehouse 2's two Apparel rows (60 + 10) collapsed to 70. The result is a clean inventory matrix.

4. Strategic explanations

The grouping key is the combination, not two separate groupings

GROUP BY warehouse_id, category does not group by warehouse and then by category — it groups by the tuple (warehouse_id, category). Two rows land in the same group only if they agree on both columns. That is why warehouse 1's Electronics and Apparel rows stay separate even though they share a warehouse. Internalising "the key is the whole column list" is what the question tests.

The select-list rule — the classic error this question baits

Every non-aggregated column in the SELECT must appear in GROUP BY. Add qty to the select list without aggregating it — SELECT warehouse_id, category, qty, SUM(qty) — and Postgres rejects the query: column "inventory.qty" must appear in the GROUP BY clause or be used in an aggregate function. Each output row represents many input rows, so a bare qty has no single value to show. (MySQL historically allowed this and returned an arbitrary row's value — a silent-wrong-answer trap. Postgres is strict, and correctly so.)

This gives a long table, not a cross-tab

The result has one (warehouse, category) per row. If the logistics team wants a matrix — warehouses as rows, categories as columns — that is a pivot, a different query:

SELECT warehouse_id,
  SUM(qty) FILTER (WHERE category='Electronics') AS electronics,
  SUM(qty) FILTER (WHERE category='Apparel')     AS apparel
FROM inventory
GROUP BY warehouse_id;

Knowing that "grouped by two columns" and "pivoted on the second column" are different shapes — and which one the consumer asked for — is the senior clarification.

Subtotals need ROLLUP / GROUPING SETS

This query gives only the leaf cells — no per-warehouse total, no grand total. If the dashboard wants "warehouse 1: 100 total" alongside the breakdown, plain GROUP BY cannot do it in one pass. GROUP BY ROLLUP(warehouse_id, category) adds the warehouse subtotals and the grand total as extra rows (with NULL in the rolled-up columns). Reaching for ROLLUP when subtotals are wanted shows range.

NULL categories and alias the aggregate

If category can be NULL, GROUP BY treats all NULLs as one group — "uncategorised stock" gets its own row. That is usually desirable, but say so. Also: the reference selects SUM(qty) with no alias, so the column comes back named sum — harmless in an interview, but in production always alias it (AS total_qty) so downstream consumers bind to a stable, meaningful name.

Senior signal: explain that the grouping key is the column tuple so rows must match on all of it, recall that every non-aggregated select column must be in GROUP BY (Postgres errors, MySQL silently lied), distinguish this long-format result from a pivot, and reach for ROLLUP when subtotals are needed.
