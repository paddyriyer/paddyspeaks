Growth accounting is the framework popularised by the consumer-product industry to decompose daily active users into four mutually exclusive buckets: new, retained, resurrected, and churned. The interviewer is probing whether you can (1) resolve each user's first-ever day, (2) align today's activity with yesterday's via a FULL OUTER self-join offset by one day, and (3) correctly classify users who appear only on the "yesterday" side as churned — reported as a negative number per the standard convention.

First-day lookup: A first on 2025-01-01, B first on 2025-01-01, C first on 2025-01-02. The challenge: on 2025-01-03, user B returns after being absent on 2025-01-02 (resurrected), and after 2025-01-04 users A and B both go dark (churned on the synthetic trailing row 2025-01-05).

2. The SQL — FULL OUTER self-join with per-user first_dt decoration

WITH first_dt AS (
  SELECT user_id, MIN(dt) AS first_dt FROM dau GROUP BY user_id
),
dau_decorated AS (
  SELECT d.dt, d.user_id, f.first_dt
  FROM dau d JOIN first_dt f ON d.user_id = f.user_id
  WHERE d.inc_amt > 0
)
SELECT
  COALESCE(tm.dt, lm.dt + 1) AS day,
  COUNT(DISTINCT tm.user_id) AS dau,
  COUNT(DISTINCT CASE WHEN tm.first_dt = tm.dt THEN tm.user_id END) AS new_users,
  COUNT(DISTINCT CASE WHEN lm.user_id IS NOT NULL THEN tm.user_id END) AS retained,
  COUNT(DISTINCT CASE WHEN tm.first_dt <> tm.dt AND lm.user_id IS NULL THEN tm.user_id END) AS resurrected,
  -1 * COUNT(DISTINCT CASE WHEN tm.user_id IS NULL THEN lm.user_id END) AS churned
FROM dau_decorated tm
FULL OUTER JOIN dau_decorated lm
  ON tm.user_id = lm.user_id AND tm.dt = lm.dt + 1
GROUP BY 1
ORDER BY 1;

The join pairs each user's today-row (tm) with their yesterday-row (lm). Users who were active yesterday but not today appear only on the lm side (churned). Users active today but not yesterday are on the tm side with lm.user_id IS NULL — further split into new (first_dt = today) or resurrected (returning from dormancy).

4. Strategic explanations

Why FULL OUTER JOIN and not LEFT JOIN

A LEFT JOIN from today to yesterday would miss the churned users entirely — they exist only in yesterday's set, not in today's. A FULL OUTER JOIN keeps both sides: rows in tm with no lm match reveal new/resurrected users; rows in lm with no tm match reveal churned users. The COALESCE(tm.dt, lm.dt + 1) in the SELECT reconstructs the "day" for purely churned rows that appear only on the lm side.

The trailing churn row

The day after the last activity in the table has no tm rows at all. But the FULL OUTER JOIN still produces lm rows for every user who was active on the final day. This is the intentional "synthetic trailing row" the problem specifies — it captures the last day's churn rather than silently dropping it. Interviewers frequently ask about this; knowing it falls out naturally from the FULL OUTER JOIN — without any special UNION or date spine — is the discriminating insight.

Four-bucket completeness — the mutual-exclusivity check

For any day, the four buckets are mutually exclusive and collectively exhaustive over the active base: new + retained + resurrected = dau. The churned count is a separate measure about yesterday's base, so it does not add to today's dau — this is a common interview stumbling block. A senior candidate states this identity explicitly and can verify it on the result table.

inc_amt > 0 filter and the decoration CTE

The WHERE d.inc_amt > 0 guard is a production pattern: raw DAU tables sometimes include zero-value session records that should not count as genuine activity. The dau_decorated CTE attaches first_dt to every active row in one pass so the main query can test tm.first_dt = tm.dt without a separate subquery join inside the window.

Dialect and performance notes

FULL OUTER JOIN is supported by PostgreSQL, SQL Server, Oracle, BigQuery, and Snowflake; MySQL lacks it (workaround: UNION of LEFT and RIGHT JOINs). On large DAU tables materialise dau_decorated first and index on (user_id, dt) before the self-join. BigQuery users should prefer the equivalent pattern using FULL JOIN on date-partitioned tables with clustering on user_id to bound cost.

Senior signal: use a FULL OUTER self-join offset by one day to simultaneously reveal all four growth-accounting buckets in a single GROUP BY; recognise that churned users appear exclusively on the "yesterday" side and that the synthetic trailing row falls out of the join naturally without a date spine; state the identity new + retained + resurrected = dau to show you understand the framework's bookkeeping.
