Spotify's logistics-analytics team tracks how long each order sits between being created and being shipped — a fulfilment-latency metric the ops team watches daily. The data stores two timestamps per order; the report wants the gap in hours. The interview question is whether you can do timestamp arithmetic cleanly — and whether you notice when an "answer" is written in a needlessly tangled way.

1. The source data — orders

Four orders. Order 1 shipped exactly 24 hours later; order 2 took 6 hours; order 4 took two and a bit days. Order 3 is still unshipped — its shipped_at is NULL — and that NULL is the edge case the query must handle.

2. The SQL — EXTRACT(EPOCH ...) to hours

SELECT order_id,
  ((EXTRACT(EPOCH FROM (shipped_at - created_at)) / 86400.0) * 86400) / 3600
    AS hours_to_ship
FROM orders
WHERE shipped_at IS NOT NULL;

shipped_at - created_at is an interval; EXTRACT(EPOCH FROM …) converts that interval to a total number of seconds. The remaining arithmetic turns seconds into hours — though, as the analysis shows, it does so by a scenic route.

Three rows — order 3 is correctly excluded by the WHERE shipped_at IS NOT NULL filter. Order 1 is 24 hours, order 2 is 6, order 4 is 60.5. The numbers are right, but the long decimal tails are a presentation problem, and order 4's 60.4999999999999992 is a small floating-point artefact of 60.5 — both worth cleaning up.

4. Strategic explanations

The arithmetic is correct — but written the long way

Read the expression carefully: it divides the epoch seconds by 86400.0, then multiplies by 86400, then divides by 3600. The / 86400.0 and * 86400 cancel out — algebraically the whole thing is just EXTRACT(EPOCH FROM …) / 3600, seconds to hours. The detour through "days" leaves no trace in the result, but it does make the query harder to review and invites a typo. The clean form:

SELECT order_id,
  ROUND(EXTRACT(EPOCH FROM (shipped_at - created_at)) / 3600.0, 2) AS hours_to_ship
FROM orders
WHERE shipped_at IS NOT NULL;

Same answer, one obvious operation, plus ROUND(…, 2) to kill the decimal tail and the 60.4999… float wobble.

Why filter shipped_at IS NOT NULL — and what the alternative means

Order 3 has no shipped_at. Without the filter, shipped_at - created_at is NULL - timestamp = NULL, and the row would show a NULL latency — not wrong, but it pollutes an AVG later. The WHERE filter scopes the metric to shipped orders, which is the honest definition of "time to ship". An alternative — measuring age of unshipped orders — is a different metric: EXTRACT(EPOCH FROM (now() - created_at)) / 3600 for rows where shipped_at IS NULL. Name which one the ops team wants.

Don't subtract a DATE and a TIMESTAMP expecting the same thing

Postgres timestamp arithmetic has a subtlety. timestamp - timestamp yields an interval (what this query relies on). But date - date yields an integer number of days, not an interval — so EXTRACT(EPOCH FROM (date_col - date_col)) would behave differently. The schema also carries a separate order_date (a DATE) alongside created_at (a TIMESTAMP); make sure latency is computed from the two timestamps, not accidentally from the date column, or the hours will be quantised to whole days.

Edge cases: clock skew and timezones

Two real-world traps. First, if shipped_at is somehow earlier than created_at — clock skew between services, or a backfilled row — hours_to_ship goes negative. A sanity guard such as GREATEST(EXTRACT(EPOCH FROM (shipped_at - created_at)), 0), or surfacing negatives as a data-quality flag, keeps the dashboard honest. Second, if the columns are timestamp without time zone but events were recorded in different zones, the subtraction silently mixes wall clocks; timestamptz is the correct type for cross-zone event data. For a clean per-day rollup, wrap the result: AVG(hours_to_ship) grouped by DATE(created_at).

Senior signal: simplify the tangled /86400 *86400 /3600 expression to a plain EXTRACT(EPOCH ...) / 3600 and wrap it in ROUND, justify the shipped_at IS NOT NULL filter as the definition of the metric, know that date - date gives days while timestamp - timestamp gives an interval, and guard against negative latency from clock skew.
