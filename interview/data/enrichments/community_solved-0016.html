A Community marketing-analytics question: count the users whose second-ever purchase was a product they had never bought before. Marketing wants to size the audience that a campaign genuinely cross-sold — people who came back and tried something new, not people who just re-ordered the same item. The whole problem is encoding "second purchase" and "new product" precisely, and the reference does it with two cooperating RANK() windows.

User 2 only ever re-buys product 30 — never a new product. User 3 has a single purchase. Users 1 and 4 both eventually buy something new on a non-first day. The mix is deliberate: each user exercises a different branch of the logic.

2. The SQL — two RANK windows

with t1 as
(
select user_id,
    created_at,
    product_id,
    rank() over(partition by user_id order by created_at asc) as date_rank,
    rank() over(partition by user_id, product_id order by created_at) as product_rank
from marketing_campaign
)

select count(distinct user_id) as cnt
from t1
where product_rank < 2 and date_rank <> 1

date_rank orders each user's purchases over time; product_rank orders purchases within a user-and-product. A row with product_rank = 1 is the first time that user bought that product; date_rank <> 1 means it was not their very first purchase overall.

3. The resulting analysis

cnt
2

Two users qualify — user 1 (product 20 on Jan 10 is new, and Jan 10 is not their first day) and user 4 (product 60 on Jan 8). User 2's only non-first purchase is a repeat of product 30, so its product_rank is 2, filtered out. User 3 never has a date_rank <> 1 row at all. The answer matches the campaign question.

4. Strategic explanations

How the two windows interlock

The clever part is the different PARTITION BY clauses on the same rows. product_rank < 2 (i.e. = 1) selects only rows that are a first encounter with a product. date_rank <> 1 drops the user's opening purchase — a first purchase is trivially "new" and would count everyone. The intersection is precisely "a new product, bought on some later visit". Two windows, one pass, no self-join.

The RANK-vs-ROW_NUMBER tie hazard on identical timestamps

If a user buys the same product twice at the exact same created_at, RANK() assigns both rows product_rank = 1 — so a pure repeat could slip through as "new". ROW_NUMBER() would force one to 1 and the other to 2, which is what the logic actually wants:

row_number() over(partition by user_id, product_id order by created_at) as product_rank

On distinct timestamps the two agree; on ties they diverge. For "is this the first occurrence" semantics, ROW_NUMBER is the safer primitive. Flag it.

An EXISTS alternative that reads more directly
SELECT COUNT(DISTINCT m.user_id) AS cnt
FROM marketing_campaign m
WHERE m.created_at > (SELECT MIN(created_at)
                      FROM marketing_campaign x WHERE x.user_id = m.user_id)
  AND NOT EXISTS (SELECT 1 FROM marketing_campaign y
                  WHERE y.user_id = m.user_id
                    AND y.product_id = m.product_id
                    AND y.created_at < m.created_at);

This spells out "after the first purchase" and "no earlier buy of this product" literally. It is more readable but fires correlated subqueries per row; the window version is one sorted pass and scales better. Knowing both, and when each wins, is the senior move.

The "second purchase" wording trap

The title says second purchase, but the SQL accepts any non-first purchase (date_rank <> 1 — could be the 3rd, 4th, …). If the campaign strictly means the 2nd order specifically, the filter must be date_rank = 2. These give different counts the moment a user has three or more orders. Pin the definition with the interviewer before coding — "any later purchase" vs "exactly the second" is a real ambiguity.

Edge cases: single-purchase users and NULL products

A user with one row has date_rank = 1 only, so they are correctly excluded — no special handling needed. But if product_id can be NULL, all NULL-product rows for a user share one product_rank partition, and NULLs compare oddly; decide whether a NULL product is a real product or junk to filter.

Senior signal: explain how two differently-partitioned RANK() windows combine to mean "new product on a later visit", flag that RANK mis-handles same-timestamp duplicates where ROW_NUMBER would not, and challenge whether "second purchase" means exactly the second or any non-first before writing a line.
