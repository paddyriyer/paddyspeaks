A digital asset pipeline needs a clean active-artifact listing: "Return the ID, uppercased name, and status of every active artifact that has a non-null name." This looks trivially simple but the interviewer is probing three things in one query: WHERE filter ordering and short-circuit evaluation, the difference between NULL and empty string for name IS NOT NULL, and the behaviour of UPPER() on NULL inputs. The answer is a one-level SELECT with two filter predicates and a string transform.

1. The source data — artifacts

artifact_idnamestatus
1alpha_scanneractive
2beta_probeinactive
3gamma_lensactive
4NULLactive
5delta_readeractive
6epsilon_corearchived
7zeta_chipactive

Row 4 has name = NULL — it should be excluded. Row 2 has status = 'inactive' and row 6 has status = 'archived' — both excluded. Rows 1, 3, 5, 7 are active with non-null names.

2. The SQL — filter, uppercase transform, deterministic sort

SELECT artifact_id, UPPER(name) AS name_upper, status
FROM artifacts
WHERE status = 'active'
  AND name IS NOT NULL
ORDER BY name_upper;

Row 4 (NULL name, active) is excluded by name IS NOT NULL. Rows 2 and 6 are excluded by status = 'active'. The four surviving rows are returned with uppercased names, sorted alphabetically by name_upper: ALPHA, DELTA, GAMMA, ZETA.

4. Strategic explanations

name IS NOT NULL — why != NULL is wrong

In SQL, any comparison to NULL using = or != always evaluates to UNKNOWN (the third truth value), which WHERE treats as false. WHERE name != NULL would silently exclude every row — including rows with actual names — because the comparison is always UNKNOWN. The only correct way to check for NULL is IS NULL or IS NOT NULL. This is one of the most common beginner mistakes and a standard interview filter question.

NULL vs empty string — different things

name IS NOT NULL passes rows where name = '' (empty string) because empty string is not NULL. If the requirement is "name has actual content," add AND name <> '' or AND LENGTH(name) > 0. Snowflake and Postgres treat empty string and NULL as distinct; Oracle historically treats '' as NULL (a notorious incompatibility). MySQL follows the standard: empty string is not NULL.

UPPER(NULL) propagates NULL

In standard SQL, calling any scalar function on NULL returns NULL. UPPER(NULL) = NULL in every major engine. If a NULL slipped through the WHERE name IS NOT NULL filter (e.g., through a lateral join), the name_upper column would be NULL rather than raising an error. The filter ensures this cannot happen here, but knowing the propagation rule matters when UPPER appears in computed columns without a prior NULL guard.

ORDER BY on an alias — when is it allowed?

ORDER BY name_upper references the SELECT alias. The SQL standard requires repeating the expression in ORDER BY, but Postgres, Snowflake, BigQuery, MySQL, and SQL Server all permit alias references in ORDER BY as an extension. The alias reference here is correct and portable to every engine in practice.

Filter predicate ordering — selectivity and short-circuit

The two predicates status = 'active' and name IS NOT NULL are ANDed. Most optimisers apply the most selective filter first. If status has an index, the engine may use it to eliminate most rows before checking name IS NOT NULL. Some engines (Postgres, SQL Server) support short-circuit evaluation for AND in simple cases, but SQL does not guarantee short-circuit order. In practice, placing the highly selective indexed filter first in the source text is a readability signal, not a performance guarantee.

Dialect portability
  • Snowflake / Postgres / BigQuery / MySQL / SQL Server: UPPER(name), IS NOT NULL, and string equality are all standard and identical across these engines.
  • Snowflake case sensitivity: Snowflake column names and string comparisons are case-insensitive by default, but string values are case-sensitive. WHERE status = 'active' would not match 'Active' unless the session uses a case-insensitive collation.
  • Oracle: the empty-string-equals-NULL trap applies here if the source system is Oracle-originated data.
Senior signal: distinguish IS NOT NULL from != NULL (the latter always returns UNKNOWN and silently drops all rows), clarify that empty string passes IS NOT NULL and needs a separate guard if blank names must also be excluded, and note Snowflake's case-sensitive string value comparison for the status filter.
