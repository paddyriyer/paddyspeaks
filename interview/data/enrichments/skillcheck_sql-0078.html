A card issuer wants a launch-performance report: for each card product, how many cards were issued in its very first month on the market? This is a first-row-per-group problem — pick, for each card, the chronologically earliest monthly record and read one column off it. The interviewer is checking whether you know a clean way to do this without an unnecessary join back to the table.

Two cases stress the logic. AmberOne spans a year boundary: its launch month is December 2023, which must sort before January 2024 — a year-then-month ordering, not month alone. NorthLite has only one row, so it is its own launch month. And SapphireGo's launch value (12000) is deliberately not its largest month (20000), so a query that grabbed the max would be wrong.

2. The SQL — DISTINCT ON to pick the earliest row per card

SELECT card_name, launch_issued FROM (
  SELECT DISTINCT ON (card_name)
         card_name,
         issued_cards AS launch_issued,
         issue_year, issue_month
  FROM card_issuance
  ORDER BY card_name, issue_year, issue_month
) t
ORDER BY launch_issued DESC, card_name;

DISTINCT ON (card_name) keeps exactly one row per card — the first row each card produces under the inner ORDER BY card_name, issue_year, issue_month. Sorting by year then month means that first row is the launch month. The outer query just re-sorts the one-row-per-card result for presentation.

4. Strategic explanations

Why DISTINCT ON is the cleanest first-per-group tool

DISTINCT ON (card_name) is a Postgres feature that, given an ORDER BY starting with card_name, keeps the first row of each card_name group and discards the rest. It expresses "one row per card, specifically the earliest" in a single construct — no window function, no self-join, no subquery to find the min and join back. The ORDER BY inside the DISTINCT ON query is not optional decoration: it defines which row survives.

The ORDER BY must lead with the group key

DISTINCT ON requires its expressions to be the leftmost ORDER BY terms. So the inner sort is card_name, issue_year, issue_month: card_name first to satisfy DISTINCT ON, then issue_year, issue_month to make "first row" mean "earliest month". Reverse the year/month terms or drop one and the surviving row would no longer be the launch month. This is also why a separate outer query handles the final launch_issued DESC sort — that ordering is incompatible with what DISTINCT ON needs internally.

Why sort by (year, month) and not a combined value

"Earliest month" across a year boundary means December 2023 precedes January 2024. Ordering by issue_month alone would wrongly rank January (1) before December (12) and pick the wrong launch for AmberOne. Sorting by issue_year first, then issue_month, gives correct chronological order. Equivalently you could sort on issue_year * 12 + issue_month or a constructed date — but the two-column sort is clearest and needs no arithmetic.

The portable alternative — ROW_NUMBER

DISTINCT ON is Postgres-only. The cross-dialect equivalent is a window function: ROW_NUMBER() OVER (PARTITION BY card_name ORDER BY issue_year, issue_month) in a subquery, then WHERE rn = 1. It produces the identical result and works in SQL Server, MySQL 8+, Oracle, and BigQuery. Knowing both — DISTINCT ON for terse Postgres, ROW_NUMBER for portability — and being able to convert between them is the senior signal.

Why not MIN with a GROUP BY

A natural-looking but wrong attempt is SELECT card_name, issued_cards WHERE (year,month) = (MIN per card). The flaw: MIN(issue_year) and MIN(issue_month) taken independently do not identify a single real row — for AmberOne that would be year 2023 and month 1, a (2023, January) combination that does not exist. You need the issued count from the row that has the minimum month, which requires either DISTINCT ON, ROW_NUMBER, or a correlated subquery matching the full (year, month) tuple — not independent column minimums.

Senior signal: for "first row per group", use Postgres DISTINCT ON (group_key) with an ORDER BY that leads with that key and then the tiebreaker columns — or the portable ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...) = 1. Sort by (year, month) so launches across a year boundary land correctly, and avoid independent column MINs, which can synthesise a tuple that matches no real row.
