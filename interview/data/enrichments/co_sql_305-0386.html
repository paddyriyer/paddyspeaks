LinkedIn's "People You May Know" (PYMK) suggestions: friends-of-friends ranked by mutual count. The query is a self-join 2-hop graph aggregation.

2. The SQL — 2-hop self-join

SELECT a.member_id AS me, b.connection_id AS suggested,
  COUNT(*) AS shared_connections
FROM connections a
JOIN connections b
  ON a.connection_id = b.member_id
 AND a.member_id != b.connection_id
GROUP BY a.member_id, b.connection_id
ORDER BY shared_connections DESC, me, suggested
LIMIT 30;

3. Strategic explanations

The 2-hop join in 3 lines

From me → my-connection-A → A's-connection-B = me to B (2 hops). a.connection_id = b.member_id bridges the hop. a.member_id != b.connection_id excludes self.

The "exclude existing direct connection" filter

Production PYMK omits already-connected pairs:

WHERE NOT EXISTS (
  SELECT 1 FROM connections f
  WHERE f.member_id = me AND f.connection_id = suggested
)
Undirected vs directed

LinkedIn connections are bidirectional (every accept produces a row pair). The query assumes undirected; if the schema is directed, OR-condition the join.

Cardinality — the scale problem

Avg 200 connections × 1B users = 40B candidate edges per user. Real LinkedIn precomputes this in a daily Spark job; the query above is the offline-analytics version, not the live UI.

The ranking signal stack

"Mutual count" is one signal. Production adds: school overlap, employer overlap, geo, profile-views, message exchanges. The query emits the candidate pool; the ranker model orders them.

Senior signal: articulate the 2-hop bridge predicate, exclude existing direct connections, name the precomputation requirement at scale, and contrast mutual-count as one of 5-10 signals fed to the ranker.
