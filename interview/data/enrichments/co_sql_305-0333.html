Facebook posts with deep comment threads (5+ levels) are the highest-engagement signals — back-and-forth conversations are vastly more sticky than 1-level comment dumps. The query uses a recursive CTE to walk the parent_id chain and report max depth + total comments per post.

1. The source data — comments

2. The SQL — recursive CTE depth walk

WITH RECURSIVE chain AS (
  SELECT comment_id, parent_id, post_id, 1 AS depth
  FROM comments WHERE parent_id IS NULL
  UNION ALL
  SELECT c.comment_id, c.parent_id, c.post_id, ch.depth + 1
  FROM comments c JOIN chain ch ON c.parent_id = ch.comment_id
)
SELECT post_id,
  MAX(depth) AS max_depth,
  COUNT(*)   AS total_comments
FROM chain
GROUP BY post_id
ORDER BY max_depth DESC;

3. Strategic explanations

The recursive CTE structure

Two parts joined by UNION ALL:

  • Anchor — root comments (parent_id IS NULL) at depth 1
  • Recursive — comments whose parent is already in chain get depth+1

The engine iterates until no new rows are added. Each iteration adds one level of depth.

The cycle-protection requirement

If comments ever has a cycle (data corruption: A → B → A), recursive CTE loops infinitely. SQLite/Postgres will eventually error or memory-bomb. Production guard:

WHERE ch.depth < 100  -- hard cap on recursion depth

100 levels is more than any real comment thread. The cap is a circuit breaker.

Iterative deepening — the alternative

For DBs without recursive CTEs (older MySQL), self-join N times manually:

SELECT c1.post_id, COUNT(DISTINCT c1.comment_id) FROM comments c1
LEFT JOIN comments c2 ON c2.parent_id = c1.comment_id
LEFT JOIN comments c3 ON c3.parent_id = c2.comment_id
... up to N levels

Bounded depth, but ugly. Recursive CTE is strictly better when supported.

The "deepest single thread" follow-up

Max depth alone hides shape. A post with one 10-deep thread vs a post with ten 1-deep threads have very different engagement. Track both:

  • Max depth (the longest single conversation)
  • Median depth (typical conversation length)
  • Total comments / max depth (breadth-vs-depth ratio)
The "conversation participant" count

True engagement signal: "how many distinct users are in this thread?". A 10-deep thread between 2 users (back-and-forth) is different from a 10-deep thread with 8 participants (community discussion). Add user_id and aggregate:

COUNT(DISTINCT c.user_id) AS unique_participants
The Reddit/Hacker News parallel

This pattern is not Meta-specific — every threaded-comment system uses it. Reddit's "show parent" feature, Hacker News' indentation, Twitter's reply-tree all run this query (or its precomputed mat-view).

Performance — the big-table caveat

Recursive CTE on billion-row comments is slow. Production: a denormalized thread_summary table updated incrementally as comments arrive. The query above is for offline analytics; the live UI uses cached values.

Senior signal: articulate the recursive-CTE structure (anchor + recursive + UNION ALL), insist on the cycle-protection cap, and reframe max-depth as one signal among shape/participants/median. The denormalized-mat-view production note is the over-the-bar scale finish.
