§ 01 — The Universal Framework

Five moves, every prompt — on every domain.

The framework doesn't care whether you're modeling Uber trips, Google ads, or Spotify streams. Domain knowledge changes what you say in each move; the moves themselves stay identical. Drill the moves until they're reflex, then layer domain context on top.

  1. Clarify the use cases. Who reads this? What 3–5 questions must it answer? What's the SLA — minutes, hours, days? Don't model anything until you have 3–5 named consumers.
  2. Pin the grain. One sentence per fact table: "One row per X per Y per Z." If you can't say it cleanly, the model isn't ready.
  3. Conceptual → logical → physical. Entities first. Then attributes and types. Then partitioning, clustering, SCD strategy.
  4. SCDs and time. Type 1 (overwrite), Type 2 (versioned rows), Type 6 (hybrid). Justify each dim.
  5. Stress test. Volunteer two failure modes — late data, GDPR, schema drift, hot partitions, time zones — without being asked.
The phrase that lifts you a level, in any domain: "…and the unlock here for the [recommendations / risk / finance / dispatch] team is X." Naming the downstream consumer turns the answer from "tables" into "data product."

The vocabulary that signals seniority — across all domains

TermOne-sentence definition
GrainThe meaning of one row, stated as a sentence.
SCD Type 1 / 2 / 6Overwrite vs versioned-rows vs hybrid (current + history columns).
Conformed dimensionSame dim used across multiple fact tables (dim_date, dim_geography).
Bridge / factless factM:N resolver, or an event-occurrence table with no measure.
Late-arriving dimFact references dim row not yet loaded — placeholder SK -1, backfill.
Slowly-changing factRestating financials. Append corrections; never UPDATE in place.
Idempotent loadRe-running yesterday produces the same result. MERGE on natural key.
Surrogate keySynthetic integer PK. Required for SCD2.
Star vs snowflakeStar — denormalized dims. Snowflake — normalized. Star is default for analytics.
Wide tableOne denormalized fact pre-joined to dims. Read speed vs storage + drift.
Activity schemaSingle-table model where every event is a row. Useful for product analytics.
Data vault3-table pattern (hub/link/satellite) for high-change EDWs. Mention; rarely use.

§ 03 — Twelve Scenarios

The twelve most-asked modeling prompts — full walk-throughs.

Uber / Lyft · ride-sharing

Scenario 1 — Trip lifecycle

Design a model that captures every Uber trip from request → match → pickup → dropoff → payment, supporting Operations (live ETA), Finance (driver payouts), and Risk (fraud).

Move 1 — Clarify

  • "Is one row per trip, or per state transition? Both — trip-level fact + factless transitions table."
  • "UberPool / shared rides — same trip_id with multiple riders, or many trips? Multiple riders → bridge table."
  • "Surge factor — locked at request or at completion? Locked at request, stored as surge_multiplier_at_request."
  • "Cancellations: rider, driver, system — same fact? Same fact, with terminal_state ∈ {completed, cancelled_rider, cancelled_driver, cancelled_system, no_show}."

Move 2 — Grain

fact_trip: one row per trip request, terminal-state captured. fact_trip_state_transition: one row per state change (factless).

Move 3 — Physical

dim_rider          (SCD2 — phone changes, plan tier changes)
dim_driver         (SCD2 — vehicle, rating tier, region)
dim_vehicle        (SCD2 — owner, insurance status)
dim_h3_cell        (SCD1 — hex grid, level 9 default)
dim_payment_method (SCD2)

CREATE TABLE fact_trip (
  trip_sk             BIGINT,
  trip_id             STRING,           -- natural
  rider_sk            BIGINT,
  driver_sk           BIGINT,
  vehicle_sk          BIGINT,
  origin_h3_sk        BIGINT,
  destination_h3_sk   BIGINT,
  request_ts          TIMESTAMP_TZ,
  match_ts            TIMESTAMP_TZ,
  pickup_ts           TIMESTAMP_TZ,
  dropoff_ts          TIMESTAMP_TZ,
  trip_distance_km    DECIMAL(8,3),
  trip_duration_sec   INT,
  base_fare_local     DECIMAL(10,2),
  surge_multiplier    DECIMAL(4,2),
  total_fare_local    DECIMAL(10,2),
  fx_rate_to_usd      DECIMAL(18,8),
  total_fare_usd      DECIMAL(10,2),
  currency_code       CHAR(3),
  product_type        STRING,           -- UberX / Pool / Black
  terminal_state      STRING,
  cancellation_actor  STRING NULL,
  rider_rating        TINYINT NULL,
  driver_rating       TINYINT NULL,
  ingest_ts           TIMESTAMP_TZ
)
PARTITION BY DATE(request_ts)
CLUSTER BY (origin_h3_sk, driver_sk);

Move 4 — SCD strategy

  • dim_driver SCD2 — rating tier and region change; finance needs the historic value.
  • dim_vehicle SCD2 — insurance status changes; risk needs as-of-trip status.
  • dim_h3_cell SCD1 — hex grid is static; geography rarely re-segmented.

Move 5 — Stress tests (volunteer)

  • Pool stitching: shared rides have multiple riders per trip — model with bridge_trip_rider; fares allocated by ratio.
  • Late dropoff signal: driver app loses connectivity, dropoff arrives 30 min late. Watermark on stream side; daily batch backfills 4-hr tail.
  • FX rate: lock to request_ts rate, not today's. dim_fx_rate is itself SCD2.
  • GDPR: tokenize rider_id at ingest; on delete, invalidate the token mapping.
Senior framing. "And the unlock here is — once fact_trip is conformed with fact_payout through dim_driver, finance can run weekly driver payouts without joining raw event tables."
Uber · supply/demand

Scenario 2 — Surge pricing & supply/demand state

Design a model that captures the supply/demand state per geography per minute, supporting the surge engine, post-hoc analysis, and capacity planning.

Move 1 — Clarify

  • "Geo unit: H3 hexes (level 8 = ~0.7km²)? Yes — Uber's standard."
  • "Time bucket — 1-min, 5-min? 1-min for live, 5-min for analytics."
  • "Snapshot or event-driven? Both — events feed a 1-min snapshot."

Move 2 — Grain

fact_geo_state_minute: one row per (h3_cell × minute). fact_surge_event: one row per surge multiplier change.

Move 3 — Physical

CREATE TABLE fact_geo_state_minute (
  h3_sk                 BIGINT,
  minute_bucket_ts      TIMESTAMP_TZ,
  open_requests         INT,            -- riders waiting
  available_drivers     INT,            -- on-app, not on a trip
  driver_to_request_ratio DECIMAL(8,3),
  surge_multiplier      DECIMAL(4,2),   -- live computed
  predicted_eta_sec     INT,
  weather_signal        STRING NULL,
  event_signal          STRING NULL,    -- nearby concert, sports event
  ingest_ts             TIMESTAMP_TZ
)
PARTITION BY DATE(minute_bucket_ts)
CLUSTER BY (h3_sk);

Move 5 — Stress tests

  • Empty cells: not every cell × minute has activity. Don't densify in raw — let consumer queries spine.
  • Skew: downtown SF on Saturday night = 100× the rows of suburb. Sub-partition popular cells if needed.
  • Cold start: new cells (just-launched cities) — bootstrap with neighbor-cell averages.
Senior framing. "The surge model itself is a product of fact_geo_state_minute joined to a learned function of (ratio, weather, event_signal). Versioning the model in dim_surge_model SCD2 lets us replay any historical surge with the model that was active then."
DoorDash / Uber Eats · 3-sided marketplace

Scenario 3 — Order & courier dispatch

Model the DoorDash order from customer placement → restaurant prep → courier pickup → delivery, accounting for batched (stacked) orders where one courier delivers multiple orders.

Move 1 — Clarify

  • "Stacked orders: one courier, two drop-offs — separate orders, shared dispatch_id."
  • "Restaurant prep time — measured how? Estimated at order time vs actual at pickup; store both."
  • "Tips — can edit post-delivery within 24hr — modeled as a slowly-changing fact (append correction)."

Move 2 — Grain

fact_order: one row per order. fact_dispatch: one row per courier dispatch (may cover multiple orders). fact_dispatch_order_link: bridge.

Move 3 — Physical

dim_customer    (SCD1 + history of address changes via SCD2 on dim_address)
dim_restaurant  (SCD2 — menu, hours, status)
dim_courier     (SCD2 — vehicle type, rating tier, on/off platform)
dim_h3_cell     (SCD1)

CREATE TABLE fact_order (
  order_sk           BIGINT,
  order_id           STRING,
  customer_sk        BIGINT,
  restaurant_sk      BIGINT,
  delivery_h3_sk     BIGINT,
  placed_ts          TIMESTAMP_TZ,
  confirmed_ts       TIMESTAMP_TZ,
  ready_ts           TIMESTAMP_TZ,
  picked_up_ts       TIMESTAMP_TZ,
  delivered_ts       TIMESTAMP_TZ,
  subtotal_usd       DECIMAL(10,2),
  tax_usd            DECIMAL(10,2),
  delivery_fee_usd   DECIMAL(10,2),
  service_fee_usd    DECIMAL(10,2),
  tip_usd            DECIMAL(10,2),     -- mutable up to 24h
  tip_locked_ts      TIMESTAMP_TZ,
  promo_applied_usd  DECIMAL(10,2),
  terminal_state     STRING,
  ingest_ts          TIMESTAMP_TZ
)
PARTITION BY DATE(placed_ts);

CREATE TABLE fact_dispatch (
  dispatch_sk         BIGINT,
  courier_sk          BIGINT,
  pickup_h3_sk        BIGINT,
  dispatch_start_ts   TIMESTAMP_TZ,
  dispatch_end_ts     TIMESTAMP_TZ,
  is_batched          BOOLEAN,
  total_orders        INT,
  total_distance_km   DECIMAL(8,3),
  payout_usd          DECIMAL(10,2)
)
PARTITION BY DATE(dispatch_start_ts);

CREATE TABLE fact_dispatch_order_link (
  dispatch_sk BIGINT,
  order_sk    BIGINT,
  leg_index   INT,           -- 1, 2 for stacked
  PRIMARY KEY (dispatch_sk, order_sk)
);

Move 5 — Stress tests

  • Tip mutation: tips edit up to 24h post-delivery — append correction with tip_corrected_at; never UPDATE.
  • Restaurant outage: restaurant offline mid-order — terminal_state = cancelled_restaurant; refund cascade.
  • Courier handoff: rare — first courier abandons, second picks up. Two dispatch rows; first ends as terminal_state = abandoned.
Senior framing. "The 3-sided design lets us answer the three different stakeholders in one model: was the customer's order on time? (joins on fact_order), did the courier earn enough per hour? (aggregates fact_dispatch), did the restaurant lose orders to long prep times? (joins placed_ts → ready_ts by restaurant_sk)."
Google · search ads

Scenario 4 — Search ads auction & quality score

Design a model for Google's search ads — capturing the impression (an auction clearing), click, and conversion, supporting advertiser billing, quality-score recomputation, and incrementality measurement.

Move 1 — Clarify

  • "Three facts: impression, click, conversion. Linked by auction_decision_id."
  • "Loser logs — kept? Yes, separate fact, sampled — required for ranker training."
  • "Conversion attribution window — 30 days last-touch by default; configurable per advertiser."
  • "Multi-currency — bid currency, billing currency, FX locked at impression time."

Move 2 — Grain

  • fact_ad_impression: one row per impression (auction won + ad served).
  • fact_ad_click: one row per click on an impression.
  • fact_ad_conversion: one row per advertiser-reported conversion, attributed.
  • fact_auction_loser: one row per losing candidate (sampled).

Move 3 — Physical

dim_advertiser     (SCD2)
dim_campaign       (SCD2)
dim_ad_group       (SCD2)
dim_creative       (SCD2)
dim_keyword        (SCD1, high cardinality)
dim_user_segment   (SCD2)
dim_device         (SCD1)

CREATE TABLE fact_ad_impression (
  impression_sk         BIGINT,
  auction_decision_id   STRING,         -- joins to losers
  user_id_token         STRING,         -- privacy-tokenized
  device_sk             BIGINT,
  query_text_hash       STRING,         -- not raw
  keyword_sk            BIGINT,
  campaign_sk           BIGINT,
  creative_sk           BIGINT,
  served_ts             TIMESTAMP_TZ,
  bid_amount_local      DECIMAL(10,4),
  predicted_action_rate DECIMAL(8,6),
  quality_score         DECIMAL(4,2),
  ad_rank               DECIMAL(10,6),
  clearing_price_local  DECIMAL(10,4),  -- second-price
  fx_rate_to_usd        DECIMAL(18,8),
  clearing_price_usd    DECIMAL(10,4),
  privacy_path          STRING,         -- standard / atta / sandbox
  is_invalid            BOOLEAN,        -- bot/fraud post-flag
  ingest_ts             TIMESTAMP_TZ
)
PARTITION BY DATE(served_ts)
CLUSTER BY (campaign_sk, keyword_sk);

CREATE TABLE fact_ad_click (
  click_sk          BIGINT,
  impression_sk     BIGINT,
  click_ts          TIMESTAMP_TZ,
  is_invalid        BOOLEAN,
  invalidation_ts   TIMESTAMP_TZ NULL    -- detected later
);

CREATE TABLE fact_ad_conversion (
  conversion_sk        BIGINT,
  advertiser_sk        BIGINT,
  click_sk             BIGINT NULL,     -- attributed (last-touch)
  impression_sk        BIGINT NULL,     -- view-through if no click
  conversion_ts        TIMESTAMP_TZ,
  conversion_type      STRING,          -- purchase, signup, install
  conversion_value_usd DECIMAL(10,2),
  attribution_window_days INT,
  attribution_model    STRING           -- last_touch, position_based, mta
);

Move 4 — SCD

  • dim_creative SCD2 — creative changes affect quality score; need historic creative for ranker debug.
  • dim_campaign SCD2 — budget and targeting drift; advertiser disputes need as-of-impression state.
  • dim_user_segment SCD2 — segment membership changes nightly.

Move 5 — Stress tests

  • Invalid clicks: bots/fraud detected hours-to-days later. Append is_invalid = TRUE with invalidation_ts; advertiser gets credit. Slowly-changing fact.
  • Privacy paths: ATT-opted-out users hit a different code path with no user_id_token; aggregate-only reporting.
  • Loser log volume: ~20× winner volume. Sample to 1% and re-weight for ranker training.
  • Attribution churn: a conversion at T+15d may be re-attributed if a closer click is found. Conversions are append-only with attribution_run_id.
Senior framing. "The auction-decision-id is the join key that turns three independent facts into a coherent advertiser narrative — and re-runnable attribution becomes a question of selecting the right attribution_run_id, not rebuilding the join."
Meta / Facebook · ads attribution

Scenario 5 — Cross-device attribution & identity

Design a model for Facebook ads where a user sees an ad on phone, opens the advertiser's site on laptop later, and converts on tablet a week after. Support 1/7/28-day click + view-through windows, ATT/SKAdNetwork constraints, and incrementality lift.

Move 1 — Clarify

  • "Identity graph: probabilistic + deterministic. dim_identity resolves device → person."
  • "Attribution windows configurable per advertiser. Pre-compute multiple in parallel."
  • "SKAdNetwork (iOS): aggregate-only, postback-based. Different fact table — no user-level."

Move 2 — Grain

  • fact_ad_event: one row per impression OR click (event-typed).
  • fact_conversion: one row per advertiser conversion event.
  • fact_skan_postback: one row per SKAdNetwork postback (iOS, aggregate-only).
  • fact_attribution: one row per (conversion × ad-event × window × model) — derived.

Move 3 — Physical (sketch)

dim_identity (the graph)
  identity_sk, person_id_token,
  -- many-to-many to devices via bridge:
bridge_identity_device
  identity_sk, device_id_token, link_confidence DECIMAL(4,3),
  effective_from, effective_to

CREATE TABLE fact_ad_event (
  event_sk            BIGINT,
  event_type          STRING,         -- 'impression' | 'click'
  identity_sk         BIGINT NULL,    -- null on iOS / opt-out
  device_id_token     STRING,
  campaign_sk         BIGINT,
  creative_sk         BIGINT,
  placement           STRING,         -- feed, story, reel
  event_ts            TIMESTAMP_TZ,
  privacy_path        STRING          -- 'opted_in' | 'att_optout' | 'skan'
)
PARTITION BY DATE(event_ts);

CREATE TABLE fact_attribution (
  attribution_sk        BIGINT,
  conversion_sk         BIGINT,
  ad_event_sk           BIGINT,
  window_type           STRING,       -- 'click_1d' | 'click_7d' | 'view_1d' | 'view_28d'
  attribution_model     STRING,       -- 'last_click' | 'last_touch' | 'data_driven'
  attribution_run_id    STRING,
  is_attributed         BOOLEAN,
  fractional_credit     DECIMAL(8,6)  -- for MTA
)
PARTITION BY attribution_run_id;

Move 5 — Stress tests

  • Identity graph drift: users add/remove devices. Bridge is SCD2 with confidence; never delete history.
  • iOS opt-out: identity_sk is NULL; can only roll up by campaign × placement × day at advertiser level.
  • Late conversions: 28-day view-through means attribution today restates impressions from 28 days ago. attribution_run_id is the immutability boundary.
  • Incrementality: requires a holdout group; modeled as dim_user_holdout with assignment SCD2 per advertiser per period.
Senior framing. "The decisive design choice is to never compute attribution at write time — every conversion is paired with every eligible ad event, and the attribution model is a function evaluated at read time. That's how we re-attribute when an advertiser switches from last-click to data-driven without re-ingesting raw events."
Netflix · CTV ads inventory

Scenario 6 — CTV ad inventory, pacing & frequency capping

Design a model for Netflix's ad-supported tier — capturing every ad slot opportunity, every ad served, frequency caps per user, and pacing guarantees per advertiser ("guaranteed 10M impressions by month-end").

Move 1 — Clarify

  • "Inventory unit: one ad-slot opportunity in a viewing session. Pre-roll, mid-roll, post-roll are different slot_types."
  • "Pacing: per (advertiser × campaign × day). Daily delivery vs daily commitment."
  • "Frequency cap: per (user × campaign), e.g. 'no more than 3 per day, 10 per week'."
  • "Make-good: under-delivered campaigns get bonus inventory next month."

Move 2 — Grain

  • fact_ad_opportunity: one row per slot opportunity (filled or unfilled).
  • fact_ad_impression: one row per ad served (subset of opportunities, with creative_sk).
  • fact_pacing_daily: one row per (campaign × day), aggregated.
  • fact_frequency_state: one row per (user × campaign × day), capped.

Move 3 — Physical (sketch)

CREATE TABLE fact_ad_opportunity (
  opportunity_sk    BIGINT,
  profile_sk        BIGINT,
  title_sk          BIGINT,
  session_sk        BIGINT,
  slot_type         STRING,        -- 'pre_roll' | 'mid_roll' | 'post_roll'
  slot_position_sec INT,
  slot_duration_sec INT,
  predicted_cpm     DECIMAL(8,4),
  served_creative_sk BIGINT NULL,  -- NULL = unfilled
  fill_reason       STRING,        -- 'won_auction' | 'guaranteed_buy' | 'house_ad'
  unfilled_reason   STRING NULL,   -- 'frequency_capped' | 'no_eligible_demand'
  opportunity_ts    TIMESTAMP_TZ
)
PARTITION BY DATE(opportunity_ts);

CREATE TABLE fact_pacing_daily (
  campaign_sk        BIGINT,
  day_sk             INT,
  delivered          INT,
  committed_today    INT,           -- daily plan
  cumulative_to_date INT,
  total_committed    INT,           -- contract total
  pacing_index       DECIMAL(6,3)   -- 1.0 = on track
);

Move 5 — Stress tests

  • Under-delivery: if pacing_index < 0.95 with 5 days left, adserver biases toward this campaign. Make-goods append to next month.
  • Frequency cap conflict: two campaigns both eligible, user at 3/3 cap on one — second wins. Logged in fill_reason.
  • Time zone for daily caps: per-user TZ for caps that say "per day"; per-advertiser TZ for pacing.
Senior framing. "Modeling unfilled opportunities (not just impressions) is the senior move — it's the only way to answer 'why did we leave money on the table this quarter?' which is the exec-level question this data product exists for."
Amazon · e-commerce

Scenario 7 — Orders, returns & multi-warehouse inventory

Design a model for Amazon orders that handles partial returns, replacements, marketplace seller orders, and inventory drawn from multiple warehouses.

Move 1 — Clarify

  • "Grain: one row per order line item, not per order. An order with 3 items = 3 rows."
  • "Returns: separate fact, append-only, allows partial."
  • "Multi-warehouse: a single line item may be split across warehouses if needed."
  • "Seller-of-record: Amazon retail, FBA seller, marketplace seller — different revenue treatment."

Move 2 — Grain

  • fact_order_line: one row per order × product × warehouse-source.
  • fact_return: one row per return event (line-level).
  • fact_inventory_movement: one row per stock change (in/out/transfer/damage).
  • snap_inventory_daily: one row per (product × warehouse × day) snapshot.

Move 3 — Physical

dim_product       (SCD2 — title, category, dimensions can change)
dim_seller        (SCD2 — Amazon Retail / FBA seller / 3P)
dim_warehouse     (SCD1)
dim_customer      (SCD1, address via SCD2 dim_address)
dim_promotion     (SCD2)

CREATE TABLE fact_order_line (
  order_line_sk        BIGINT,
  order_id             STRING,
  line_number          INT,
  customer_sk          BIGINT,
  product_sk           BIGINT,
  seller_sk            BIGINT,
  source_warehouse_sk  BIGINT,
  ordered_ts           TIMESTAMP_TZ,
  shipped_ts           TIMESTAMP_TZ NULL,
  delivered_ts         TIMESTAMP_TZ NULL,
  quantity             INT,
  unit_price_local     DECIMAL(10,2),
  promo_discount_local DECIMAL(10,2),
  tax_local            DECIMAL(10,2),
  shipping_local       DECIMAL(10,2),
  fx_rate_to_usd       DECIMAL(18,8),
  total_revenue_usd    DECIMAL(10,2),
  is_prime             BOOLEAN,
  is_subscribe_save    BOOLEAN,
  ingest_ts            TIMESTAMP_TZ
)
PARTITION BY DATE(ordered_ts)
CLUSTER BY (customer_sk, product_sk);

CREATE TABLE fact_return (
  return_sk        BIGINT,
  order_line_sk    BIGINT,
  return_ts        TIMESTAMP_TZ,
  refund_ts        TIMESTAMP_TZ NULL,
  reason_code      STRING,
  quantity         INT,         -- partial possible
  refund_amount_usd DECIMAL(10,2),
  is_replacement   BOOLEAN
);

Move 5 — Stress tests

  • Returns: never UPDATE fact_order_line. Append fact_return. Net revenue = SUM(orders.revenue) - SUM(returns.refund).
  • Partial returns: quantity on the return may be less than original. Track on the row.
  • Replacement orders: linked via is_replacement = TRUE + replacement_for_order_line_sk. Counts as gross order, nets to zero.
  • Multi-warehouse split: one ordered SKU split = two rows with same order_id, line_number, different source_warehouse_sk.
  • Marketplace seller revenue: Amazon recognizes only the commission, not the GMV. dim_seller.recognized_as ∈ {gmv, commission_only}.
Senior framing. "Append-only on returns is the contract that lets finance restate any historical period without rebuilding orders — and that's the difference between a model that survives quarter-end and one that doesn't."
Meta / Instagram · social feed

Scenario 8 — Feed engagement at scale

Model engagement on the Instagram feed — impressions, dwell time, likes, saves, comments, shares — supporting feed ranker training, creator analytics, and brand safety.

Move 1 — Clarify

  • "Grain: one row per (user × post × event_type × event_ts). High volume — billions per day."
  • "Dwell time — heartbeat-based, not single timestamp. Aggregate up to per-impression."
  • "Negative engagement (hide, report, block) — same fact, different event_type."

Move 2 — Grain

  • fact_impression: one row per (viewer × post × impression_ts).
  • fact_engagement: one row per (viewer × post × event_type × event_ts).
  • fact_dwell_aggregate: one row per impression with total dwell.

Move 3 — Physical

dim_user        (SCD2)
dim_post        (SCD2 — caption edits, audio swap)
dim_creator     (SCD2)
dim_post_audio  (SCD1)

CREATE TABLE fact_impression (
  impression_sk      BIGINT,
  viewer_sk          BIGINT,
  post_sk            BIGINT,
  feed_position      SMALLINT,       -- 1 = top of feed
  impression_ts      TIMESTAMP_TZ,
  device_sk          BIGINT,
  ranker_model_id    STRING,         -- which ranker version
  predicted_score    DECIMAL(8,6)
)
PARTITION BY DATE(impression_ts)
CLUSTER BY (post_sk, viewer_sk);

CREATE TABLE fact_engagement (
  engagement_sk   BIGINT,
  impression_sk   BIGINT,            -- nullable: comments arrive without an impression event
  viewer_sk       BIGINT,
  post_sk         BIGINT,
  event_type      STRING,            -- 'like' | 'save' | 'share' | 'comment' | 'hide' | 'report'

Move 5 — Stress tests

  • Volume: billions of impressions per day. Dedup via event_id at ingest. Use APPROX_COUNT_DISTINCT for analytics.
  • Like/unlike toggle: a like can be undone; track as is_undone + undone_ts rather than deleting.
  • Dwell time: heartbeats every 250ms while post is on screen. Aggregate to per-impression at ETL time, not at query.
  • Brand safety: negative engagement (hide, report) is a high-leverage signal — pre-aggregate per post for ranker.
Senior framing. "Storing ranker_model_id on the impression is what lets the feed team A/B test new models without losing the ability to compute baseline metrics — model-version is a first-class dimension, not a footnote."
Spotify · streaming media

Scenario 9 — Listening history & royalty payouts

Design a model for Spotify's listening history that supports recommendations, the "year in review" feature, and per-stream royalty payouts to multiple rights holders (artist + label + publisher + songwriter).

Move 1 — Clarify

  • "Royalty stream is qualified — typically ≥30s play. Below that = no payout."
  • "Per-stream royalty: pool model (revenue / total streams in the pool) — not per-track price."
  • "Multi-rights: one stream pays artist (recording), label (master), publisher (composition), songwriter (composition). Splits stored in bridge_track_rights."

Move 2 — Grain

  • fact_play_event: one row per play event (≥30s threshold flag).
  • fact_royalty_attribution: one row per (play × rights_holder × period) — derived.

Stripe · payments & ledger

Scenario 10 — Double-entry ledger for payments

Design a model for Stripe's payment system supporting charges, refunds, chargebacks, payouts, multi-currency, and rigorous double-entry accounting (every entry has equal debits and credits).

Move 1 — Clarify

  • "Double-entry: every transaction = at least 2 ledger entries summing to zero per currency."
  • "Multi-currency: charges in EUR, payouts in USD, FX in between — three entries minimum."
  • "Append-only ledger. No UPDATE. Ever. Reversal = new entries with opposite signs."

Move 2 — Grain

  • fact_ledger_entry: one row per debit OR credit on a specific account in a specific currency. This is the source of truth.
  • fact_balance_transaction: one row per business event (charge/refund/payout) — joins entries.
  • snap_account_balance_daily: derived — running sum per (account × currency × day).

Move 3 — Physical

dim_merchant     (SCD2)
dim_customer     (SCD2)
dim_payment_method (SCD2)
dim_account      (SCD1 — chart of accounts: 'merchant_balance', 'fees', 'reserve', 'tax')
dim_currency     (SCD1)
dim_country      (SCD1)

CREATE TABLE fact_ledger_entry (
  entry_sk           BIGINT,
  entry_id           STRING,         -- natural, idempotent
  balance_txn_sk     BIGINT,
  account_sk         BIGINT,         -- which account is being debited/credited
  merchant_sk        BIGINT,
  currency_code      CHAR(3),
  amount             DECIMAL(20,4),  -- signed; debits negative, credits positive (or per convention)
  entry_type         STRING,         -- 'charge' | 'refund' | 'fee' | 'fx_gain' | 'payout' | 'chargeback' | 'reserve'
  effective_ts       TIMESTAMP_TZ,
  posted_ts          TIMESTAMP_TZ,   -- when entered the ledger
  reversal_of_sk     BIGINT NULL,    -- for reversals
  source_event_id    STRING,         -- idempotency
  ingest_ts          TIMESTAMP_TZ,

  -- the invariant: SUM(amount) per (balance_txn_sk, currency_code) = 0
  CONSTRAINT chk_idempotent UNIQUE (source_event_id)
)
PARTITION BY DATE(effective_ts)
CLUSTER BY (merchant_sk, account_sk);

CREATE TABLE fact_balance_transaction (
  balance_txn_sk      BIGINT,
  balance_txn_id      STRING,
  txn_type            STRING,        -- 'charge' | 'refund' | 'payout' | 'chargeback' | 'fee' | 'transfer'
  txn_ts              TIMESTAMP_TZ,
  source_object_id    STRING,        -- charge_id, refund_id, payout_id
  merchant_sk         BIGINT,
  status              STRING
);

Move 5 — Stress tests

  • The invariant test: for any balance_txn_sk, SUM(amount) GROUP BY currency = 0. Run as a continuous data-quality check.
  • FX leg: EUR charge → USD payout creates a third entry in EUR (debit merchant) + entry in USD (credit merchant) + an FX_gain/loss entry in the platform's reporting currency.
  • Chargebacks: 30+ days after charge. Append-only — debit merchant, credit dispute-reserve. Original charge stays untouched.
  • Idempotency: retries from upstream produce same source_event_id; UNIQUE constraint prevents double-write.
  • Reserves: high-risk merchants — N% of balance held as reserve. Modeled as transfer entries between merchant_balance and reserve accounts.
Senior framing. "Append-only is not a style choice in payments — it's a regulatory requirement. The model has to support 'show me the full audit trail for merchant X on date Y as it appeared then' regardless of corrections made since. That's why every reversal is a new pair of entries, never a retro edit."

Scenario 11 — Bookings, calendar & reviews

Design a model for Airbnb that supports search/availability lookup, booking lifecycle (request → accepted → stayed → reviewed), cancellation policies, and host payouts after stay completion.

Move 1 — Clarify

Scenario 12 — Subscription billing & usage metering

Design a model for a SaaS like Snowflake or Datadog with mixed pricing — flat-rate seats + usage-based metering (per-query, per-GB, per-API-call) — handling plan changes mid-cycle, prorations, overages, and trials.

Move 1 — Clarify

  • fact_subscription_state: one row per (subscription × day) — daily snapshot.
  • fact_billing_event: one row per charge/proration/refund/dunning.

§ 04 — Cross-Cutting Concerns

The patterns that show up regardless of domain.

1. SCD Types — the deep dive

TypeMechanismWhen to useDomain example
Type 1Overwrite. No history.Current state only matters; cheap fix-ups.User's display name correction.
Type 2New row per change with effective_from / effective_to.Historical accuracy required — financial, regulatory, ML.Driver rating tier (Uber), creative version (Meta), plan tier (SaaS).
Type 3Add previous_X column. Rarely used.Only the most recent change matters.Job title (current + previous).
Type 4Separate history table.Current dim is small + hot; history rarely queried.Geography reorganizations.
Type 6Hybrid — current_X column + SCD2 history.90% of queries want current, 10% want history.Customer's current address vs every address ever.

2. Late-arriving data — the four cases

CasePattern
Late-arriving factEvent timestamped yesterday, lands today. Watermark on stream side; daily backfill window (typically 24–48h) on batch.
Late-arriving dimensionFact references a dim row not yet loaded. Use placeholder SK -1, mark with needs_dim_lookup = TRUE, backfill nightly.
Slowly-changing factFinancial restatement (refund of last month's invoice). Append correction row; never UPDATE. Net = SUM.
Out-of-order eventsStream-time vs event-time. Watermark = max_seen_ts − lag. Sort within partition.

3. GDPR / privacy — the universal pattern

  1. Tokenize at ingest. Replace user_id with user_id_token; keep mapping in a separate, encrypted vault.
  2. On delete request: invalidate the token mapping. Events become un-rejoinable to the person.
  3. Aggregates survive — DAU, revenue, etc. stay accurate.
  4. Time-bounded: 30-day SLA; track in fact_gdpr_request.
  5. Tax / legal carve-outs: finance facts retained 7 years even after request — anonymize (replace name) but keep transaction.

4. Multi-currency — the lock-at-event-time rule

StepRule
Store bothAlways store amount_local + currency_code AND amount_usd + fx_rate. Never just one.
Lock the rateLock fx_rate_to_usd at event time, not query time. Same charge can't yield different USD amounts on different days.
FX dim is SCD2dim_fx_rate(currency_pair, rate, effective_from, effective_to) — daily rate at minimum.
Reporting currencySome companies report in a third currency (Stripe USD, Adyen EUR). Carry that as a third pair.

5. Time zones — the three rules

  1. Storage in UTC, always. Use TIMESTAMP_TZ when supported.
  2. Dim for the reporting TZ. "Daily" reports are in the reporting entity's TZ (driver's local TZ for Uber payouts; advertiser's TZ for Google ads pacing).
  3. DST-safe partitioning. Partition by DATE(event_ts AT TIME ZONE 'UTC') always. DST shifts cause off-by-one if you partition by local TZ.

6. Hot partitions / skew — the four counter-moves

Counter-moveWhen
Sub-partition by hashOne title/cell/merchant dominates: HASH(key) MOD 16 as a sub-partition.
Salted aggregationTwo-stage GROUP BY: SUM by (key || salt), then SUM by key. Spark/Trino.
Separate hot tableTop-N keys go to a "hot" partition with looser retention; rest to "cold".
Pre-aggregate at edgeSketch the hot keys upstream (HLL, Count-Min) so the warehouse never sees raw skew.

7. Idempotency — the universal contract

Every fact table needs one of:

  • Natural-key MERGE: MERGE INTO target USING source ON natural_key WHEN MATCHED AND source.ingest_ts > target.ingest_ts THEN UPDATE.
  • Append-only with dedup at read: ROW_NUMBER() OVER (PARTITION BY natural_key ORDER BY ingest_ts DESC) = 1.
  • Unique constraint on source_event_id: the producer's idempotency key, enforced.

8. Conformed dimensions — the discipline

One dim_date, one dim_geography, one dim_user — owned by the platform team, not duplicated per consumer. Otherwise:

  • Joins lie (different definitions of "user").
  • Cross-fact analysis (revenue × engagement) is impossible.
  • SCD2 versioning drifts.

Drop the term in interviews: "I'd want this to share dim_user with the existing customer-data platform — otherwise we're rebuilding the join every team."

§ 05 — Traps & Articulation

The eight traps + the script that wins the round.

The eight modeling traps — and the senior counter-move

TrapCounter-move
Writing columns before stating grainSay grain first, every time.
Using natural keys as PKsSurrogate as PK, natural as a column. Mandatory for SCD2.
UPDATE on factsAppend corrections; let MV pick latest.
One status column on a dim ("current_stage")Factless fact for transitions + daily snapshot for current.
Hierarchies via recursive joins for hot queriesDenormalize parent IDs onto leaf, drift-check nightly.
GDPR via cascade DELETEsTokenize at ingest; invalidate token on delete.
"We'll add SCD later"Adding SCD2 to a populated dim is a 6-month migration. Decide upfront.
Per-team conformed dim copiesOne conformed dim, owned by platform. Otherwise joins lie.

The 90-second articulation script — works on every domain

"Before I start — let me confirm: [3 clarifying questions]. Okay, assuming [X], the grain of the primary fact is [one sentence]. I'd model this as a star — [fact] surrounded by [3–4 dims] — with [dim_X] as Type 2 because [reason]. Physically, partition by [date_sk], cluster by [hot key]. Two failure modes I'd plan for: [late data + watermark] and [GDPR + tokenization]. The unlock for the business is [decision the data product enables]."

The three sentences that signal seniority — in any round

  1. "The grain of this fact is one row per X per Y per Z." — names the unit of analysis up front.
  2. "The trade-off here is X vs Y; I'm picking X because [downstream consumer / cost / regulatory]." — names the choice and owns it.
  3. "This breaks when [late data / GDPR / FX / skew / time-zone shift]." — volunteers a failure mode.

Drill those three sentence-shapes until they're reflex. The interviewer cannot give you a senior score if these never come out of your mouth.
