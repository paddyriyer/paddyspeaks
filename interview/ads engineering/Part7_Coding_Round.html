The Coding Round

Tricky SQL and Python problems pulled from real ads data-engineering interview loops. Every problem has a hidden edge case — miss it and the round moves on without you.

← Part 6 · Index

The coding round for an ads DE loop is not leetcode. It is a test of whether you understand the shape of ads data — attribution windows, de-duplication, pacing, frequency, incrementality — and whether you can express that understanding in correct, readable, performant code. Every problem below is one I have either given or been given in a real interview. Each one contains a specific trap that distinguishes someone who has shipped the system from someone who has read about it.

Before reading the reference solution, take a timer. Give yourself 15 minutes per SQL problem, 25 per Python. Do not look at the solution until the timer ends or you have a working answer. The value of this part is in the failed attempts, not the clean ones.

1. SQL — Eight Tricky Problems

SQL 1 — Last-click attribution with mixed click and view windows

Prompt

For each conversion, attribute it to the last touch that satisfies either (a) a click within 7 days before the conversion, or (b) a view within 1 day before the conversion. Click wins over view on ties. Output: conversion_id, attributed_touch_id, attribution_type, campaign_id.

Schema

conversions(conversion_id BIGINT, user_id BIGINT, conversion_ts TIMESTAMP, revenue_usd DECIMAL)
touches(touch_id BIGINT, user_id BIGINT, campaign_id BIGINT, touch_ts TIMESTAMP, touch_type STRING)  -- 'click' or 'view'

The trap

Candidates join once with a combined window and take the last row. This is wrong when a user has a recent view and an older click — last-touch-by-time gives the view, but click should win. You need a preference order inside the window filter.

Reference solution

WITH candidates AS (
  SELECT
    c.conversion_id,
    c.revenue_usd,
    t.touch_id,
    t.campaign_id,
    t.touch_type,
    t.touch_ts,
    ROW_NUMBER() OVER (
      PARTITION BY c.conversion_id
      ORDER BY
        CASE WHEN t.touch_type = 'click' THEN 0 ELSE 1 END,  -- click beats view
        t.touch_ts DESC                                        -- then most recent
    ) AS rn
  FROM conversions c
  JOIN touches t
    ON t.user_id = c.user_id
   AND (
        (t.touch_type = 'click' AND t.touch_ts >= c.conversion_ts - INTERVAL '7 days'
                                 AND t.touch_ts <= c.conversion_ts)
     OR (t.touch_type = 'view'  AND t.touch_ts >= c.conversion_ts - INTERVAL '1 day'
                                 AND t.touch_ts <= c.conversion_ts)
       )
)
SELECT
  conversion_id,
  touch_id AS attributed_touch_id,
  touch_type AS attribution_type,
  campaign_id
FROM candidates
WHERE rn = 1;

Follow-ups the interviewer probes

  • "What if a conversion has no qualifying touch?" → add a LEFT JOIN and COALESCE to 'organic'.
  • "What if click and view from different campaigns?" → the ORDER BY already breaks the tie by type + time; campaign is intentionally not part of the tiebreaker.

SQL 2 — Frequency capping: who saw this creative more than N times in a rolling 24h

Prompt

For each (user_id, creative_id) pair, identify any rolling 24-hour window in the past 30 days during which they received >10 impressions. Output users and creatives with the count of windows that breached the cap.

Schema

impressions(impression_id BIGINT, user_id BIGINT, creative_id BIGINT, imp_ts TIMESTAMP)

The trap

"Rolling 24h" is not the same as "per calendar day." A candidate who groups by DATE(imp_ts) misses windows that span midnight. The correct pattern uses COUNT(*) OVER with a range frame, not a row frame.

Reference solution

WITH windowed AS (
  SELECT
    user_id,
    creative_id,
    imp_ts,
    COUNT(*) OVER (
      PARTITION BY user_id, creative_id
      ORDER BY EXTRACT(EPOCH FROM imp_ts)
      RANGE BETWEEN 86400 PRECEDING AND CURRENT ROW  -- 24h rolling
    ) AS imps_in_24h
  FROM impressions
  WHERE imp_ts >= CURRENT_DATE - INTERVAL '30 days'
)
SELECT
  user_id,
  creative_id,
  SUM(CASE WHEN imps_in_24h > 10 THEN 1 ELSE 0 END) AS windows_over_cap,
  MAX(imps_in_24h)                                  AS peak_count
FROM windowed
GROUP BY user_id, creative_id
HAVING SUM(CASE WHEN imps_in_24h > 10 THEN 1 ELSE 0 END) > 0
ORDER BY peak_count DESC;

Follow-ups

  • "Some engines don't support RANGE with time intervals." → materialize epoch seconds in a CTE and range over the integer, or self-join with BETWEEN imp_ts - 24h AND imp_ts (slower but universal).
  • "What about viewable impressions only?" → filter on is_viewable = TRUE upstream.

SQL 3 — Multi-touch attribution with time-decay weights

Prompt

Attribute each conversion's revenue across the user's touches in the past 30 days using time-decay: a touch N days before conversion gets weight 2^(-N/7) (half-life of 7 days). Normalize weights per conversion so they sum to 1. Output per-campaign attributed revenue.

Schema

conversions(conversion_id BIGINT, user_id BIGINT, conversion_ts TIMESTAMP, revenue_usd DECIMAL)
touches(touch_id BIGINT, user_id BIGINT, campaign_id BIGINT, touch_ts TIMESTAMP)

The trap

Normalizing per touch instead of per conversion — produces wrong totals. And computing EXP(LN(2) * ...) differently per engine. The clean approach uses a windowed SUM for the denominator.

Reference solution

WITH weighted AS (
  SELECT
    c.conversion_id,
    c.revenue_usd,
    t.campaign_id,
    POWER(2.0, -1.0 * DATEDIFF('day', t.touch_ts, c.conversion_ts) / 7.0) AS raw_weight
  FROM conversions c
  JOIN touches t
    ON t.user_id = c.user_id
   AND t.touch_ts BETWEEN c.conversion_ts - INTERVAL '30 days' AND c.conversion_ts
),
normalized AS (
  SELECT
    conversion_id,
    revenue_usd,
    campaign_id,
    raw_weight,
    raw_weight / SUM(raw_weight) OVER (PARTITION BY conversion_id) AS norm_weight
  FROM weighted
)
SELECT
  campaign_id,
  SUM(revenue_usd * norm_weight) AS attributed_revenue
FROM normalized
GROUP BY campaign_id
ORDER BY attributed_revenue DESC;

Follow-up

  • "How does this compare to linear MTA?" → linear uses 1/n_touches for each; time-decay favors recency. Run both in parallel in production; compare.

SQL 4 — Incrementality via holdout group

Prompt

Compute campaign incrementality: for each campaign, the conversion rate in the exposed group vs the holdout group. The assignment table randomly held out 5% of the audience from ads during the campaign. Report per-campaign lift and confidence interval width.

Schema

assignments(user_id BIGINT, campaign_id BIGINT, group_name STRING)  -- 'exposed' or 'holdout'
conversions(user_id BIGINT, campaign_id BIGINT, conversion_ts TIMESTAMP)

The trap

Missing the holdout group entirely and computing a gross "conversion rate" — which shows whether ads were shown to purchasers, not whether ads caused purchases. Also: forgetting the standard-error formula for a rate difference.

Reference solution

WITH rates AS (
  SELECT
    a.campaign_id,
    a.group_name,
    COUNT(DISTINCT a.user_id)                                  AS n_users,
    COUNT(DISTINCT c.user_id)                                  AS n_convs,
    COUNT(DISTINCT c.user_id) * 1.0 / COUNT(DISTINCT a.user_id) AS conv_rate
  FROM assignments a
  LEFT JOIN conversions c
    ON c.user_id = a.user_id AND c.campaign_id = a.campaign_id
  GROUP BY a.campaign_id, a.group_name
),
pivoted AS (
  SELECT
    campaign_id,
    MAX(CASE WHEN group_name = 'exposed' THEN conv_rate END) AS exposed_rate,
    MAX(CASE WHEN group_name = 'exposed' THEN n_users   END) AS exposed_n,
    MAX(CASE WHEN group_name = 'holdout' THEN conv_rate END) AS holdout_rate,
    MAX(CASE WHEN group_name = 'holdout' THEN n_users   END) AS holdout_n
  FROM rates
  GROUP BY campaign_id
)
SELECT
  campaign_id,
  exposed_rate,
  holdout_rate,
  (exposed_rate - holdout_rate) / NULLIF(holdout_rate, 0) AS lift_pct,
  exposed_rate - holdout_rate                              AS absolute_lift,
  -- 95% CI half-width for a difference in proportions
  1.96 * SQRT(
      exposed_rate * (1 - exposed_rate) / exposed_n
    + holdout_rate * (1 - holdout_rate) / holdout_n
  ) AS ci_half_width
FROM pivoted
ORDER BY lift_pct DESC;

Follow-up

  • "Lift is +2% but CI half-width is 3%." → The result is not statistically significant. You cannot claim lift. Senior candidates name this explicitly rather than reporting the +2% as a win.

SQL 5 — Triple-key dedupe with a suspected-dupe sidecar

Prompt

The ingest pipeline sometimes receives the same event twice because of SDK retries. event_id is not a reliable dedupe key (different SDK versions generate it differently). Build a silver table that keeps the first copy per logical event, and a sidecar table that preserves suspected duplicates for audit.

Schema

bronze.impression(event_id BIGINT, user_id BIGINT, event_name STRING, event_ts TIMESTAMP, ingest_ts TIMESTAMP, sdk_version STRING, dt DATE)

The trap

Using event_id alone misses retries with regenerated IDs. Using (user_id, event_ts) without rounding misses retries milliseconds apart that are functionally the same event.

Reference solution

CREATE OR REPLACE TABLE silver.fact_impression AS
WITH keyed AS (
  SELECT
    *,
    CONCAT_WS('|',
      CAST(user_id AS STRING),
      event_name,
      CAST(DATE_TRUNC('second', event_ts) AS STRING)
    ) AS dedupe_key
  FROM bronze.impression
  WHERE dt = :yesterday
)
SELECT * FROM keyed
QUALIFY ROW_NUMBER() OVER (PARTITION BY dedupe_key ORDER BY ingest_ts ASC) = 1;

CREATE OR REPLACE TABLE ops.suspected_duplicates AS
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY
      CONCAT_WS('|', CAST(user_id AS STRING), event_name,
                CAST(DATE_TRUNC('second', event_ts) AS STRING))
      ORDER BY ingest_ts ASC) AS rn
  FROM bronze.impression WHERE dt = :yesterday
) WHERE rn > 1;

Follow-up

  • "How do you alert when one SDK version suddenly drops 10× more than others?" → a per-sdk_version drop-rate monitor from the sidecar table.

SQL 6 — Cohort ROAS matrix (acquisition week × revenue week)

Prompt

For users acquired each week in the past 12 weeks, report their cumulative revenue at week 0, week 1, week 2, week 4, week 8 after acquisition. Each row is a cohort week; columns are week-offset revenue per user.

Schema

acquisitions(user_id BIGINT, acquired_ts TIMESTAMP, cost_usd DECIMAL)
revenue(user_id BIGINT, revenue_ts TIMESTAMP, revenue_usd DECIMAL)

The trap

Computing "week 1 revenue" as a separate WHERE-filtered query and joining four times — expensive and error-prone. The clean pattern uses conditional SUMs over a single join.

Reference solution

WITH cohort AS (
  SELECT
    user_id,
    DATE_TRUNC('week', acquired_ts) AS cohort_week,
    cost_usd
  FROM acquisitions
  WHERE acquired_ts >= CURRENT_DATE - INTERVAL '12 weeks'
),
joined AS (
  SELECT
    c.cohort_week,
    c.user_id,
    c.cost_usd,
    r.revenue_usd,
    DATEDIFF('week', c.cohort_week, DATE_TRUNC('week', r.revenue_ts)) AS week_offset
  FROM cohort c
  LEFT JOIN revenue r USING (user_id)
)
SELECT
  cohort_week,
  COUNT(DISTINCT user_id)                                                       AS n_users,
  SUM(DISTINCT cost_usd)                                                        AS cohort_spend,
  SUM(CASE WHEN week_offset <= 0 THEN revenue_usd ELSE 0 END) / COUNT(DISTINCT user_id) AS rev_per_user_w0,
  SUM(CASE WHEN week_offset <= 1 THEN revenue_usd ELSE 0 END) / COUNT(DISTINCT user_id) AS rev_per_user_w1,
  SUM(CASE WHEN week_offset <= 2 THEN revenue_usd ELSE 0 END) / COUNT(DISTINCT user_id) AS rev_per_user_w2,
  SUM(CASE WHEN week_offset <= 4 THEN revenue_usd ELSE 0 END) / COUNT(DISTINCT user_id) AS rev_per_user_w4,
  SUM(CASE WHEN week_offset <= 8 THEN revenue_usd ELSE 0 END) / COUNT(DISTINCT user_id) AS rev_per_user_w8,
  SUM(CASE WHEN week_offset <= 8 THEN revenue_usd ELSE 0 END) / NULLIF(SUM(DISTINCT cost_usd), 0) AS roas_w8
FROM joined
GROUP BY cohort_week
ORDER BY cohort_week DESC;

SQL 7 — Pacing: detect over-delivery before the end of day

Prompt

A campaign's daily budget is spread evenly across 24 hours. For each campaign, identify any point in today's calendar so far where cumulative spend is more than 110% of the linear-pacing target. Output the first breach timestamp per campaign and how far over it ran.

Schema

impressions(campaign_id BIGINT, imp_ts TIMESTAMP, spend_cents BIGINT)
campaigns(campaign_id BIGINT, daily_budget_usd DECIMAL)

Reference solution

WITH minute_spend AS (
  SELECT
    campaign_id,
    DATE_TRUNC('minute', imp_ts)              AS t,
    SUM(spend_cents) / 100.0                  AS spend_usd
  FROM impressions
  WHERE DATE(imp_ts) = CURRENT_DATE
  GROUP BY campaign_id, DATE_TRUNC('minute', imp_ts)
),
running AS (
  SELECT
    campaign_id,
    t,
    SUM(spend_usd) OVER (PARTITION BY campaign_id ORDER BY t) AS cum_spend,
    (EXTRACT(EPOCH FROM (t - DATE_TRUNC('day', t))) / 86400.0) AS day_fraction
  FROM minute_spend
),
breaches AS (
  SELECT
    r.campaign_id,
    r.t,
    r.cum_spend,
    r.day_fraction * c.daily_budget_usd       AS linear_target,
    r.cum_spend / NULLIF(r.day_fraction * c.daily_budget_usd, 0) AS pace_ratio
  FROM running r
  JOIN campaigns c USING (campaign_id)
  WHERE r.cum_spend > 1.10 * r.day_fraction * c.daily_budget_usd
)
SELECT
  campaign_id,
  MIN(t)                                 AS first_breach_ts,
  MAX(pace_ratio)                        AS peak_pace_ratio,
  MAX(cum_spend - linear_target)         AS peak_overshoot_usd
FROM breaches
GROUP BY campaign_id
ORDER BY peak_pace_ratio DESC;

SQL 8 — Creative fatigue: CTR decay by days since launch

Prompt

For each creative, compute its CTR on each day since launch. Flag any creative whose CTR on day N is below 60% of its own CTR on day 1, where day 1 is the creative's own first day of delivery.

Schema

impressions(creative_id BIGINT, imp_ts TIMESTAMP, did_click BOOLEAN)

The trap

Using a calendar day-of-month instead of days since the creative itself launched. Creatives launched on different dates can't be compared on absolute dates.

Reference solution

WITH daily AS (
  SELECT
    creative_id,
    DATE(imp_ts)                              AS dt,
    COUNT(*)                                  AS imps,
    SUM(CASE WHEN did_click THEN 1 ELSE 0 END) AS clicks,
    SUM(CASE WHEN did_click THEN 1 ELSE 0 END) * 1.0 / COUNT(*) AS ctr
  FROM impressions
  GROUP BY creative_id, DATE(imp_ts)
),
indexed AS (
  SELECT
    d.*,
    MIN(dt) OVER (PARTITION BY creative_id) AS launch_dt,
    DATEDIFF('day', MIN(dt) OVER (PARTITION BY creative_id), dt) + 1 AS day_n
  FROM daily d
),
day1_ctr AS (
  SELECT creative_id, ctr AS ctr_day1
  FROM indexed WHERE day_n = 1
)
SELECT
  i.creative_id,
  i.dt,
  i.day_n,
  i.ctr,
  d1.ctr_day1,
  i.ctr / NULLIF(d1.ctr_day1, 0) AS relative_ctr,
  CASE WHEN i.ctr < 0.60 * d1.ctr_day1 THEN TRUE ELSE FALSE END AS fatigued
FROM indexed i
JOIN day1_ctr d1 USING (creative_id)
WHERE i.day_n > 7                   -- only flag after the first week
  AND i.imps >= 1000                -- only flag once there's volume
ORDER BY i.creative_id, i.dt;

2. Python — Six Coding Problems

Python 1 — Sessionize an event stream with a timeout

Prompt

Given a list of events [(user_id, ts, event_name)] sorted by user_id then ts, partition each user's events into sessions where any gap > 30 minutes starts a new session. Return a list of (user_id, session_id, start_ts, end_ts, event_count). Solve in O(N).

from dataclasses import dataclass
from datetime import datetime, timedelta
from typing import List, Tuple

@dataclass
class Session:
    user_id: int
    session_id: str
    start_ts: datetime
    end_ts: datetime
    event_count: int

GAP = timedelta(minutes=30)

def sessionize(events: List[Tuple[int, datetime, str]]) -> List[Session]:
    out: List[Session] = []
    if not events:
        return out
    cur_user = None
    cur_start = cur_end = None
    cur_count = 0

    def flush():
        if cur_user is not None and cur_count > 0:
            sid = f"{cur_user}_{int(cur_start.timestamp())}"
            out.append(Session(cur_user, sid, cur_start, cur_end, cur_count))

    for user_id, ts, _ in events:
        if user_id != cur_user or (cur_end is not None and ts - cur_end > GAP):
            flush()
            cur_user, cur_start, cur_end, cur_count = user_id, ts, ts, 1
        else:
            cur_end = ts
            cur_count += 1
    flush()
    return out

The trap

Forgetting that user boundaries also start a new session — a candidate who only checks the gap will merge two users' events.

Python 2 — Token-bucket rate limiter for pixel firing

Prompt

Implement a rate limiter that allows at most N pixel events per user per second, with a burst capacity of B. It should be thread-safe and O(1) per call.

import time
import threading
from collections import defaultdict

class TokenBucket:
    def __init__(self, rate_per_sec: float, burst: int):
        self.rate = rate_per_sec
        self.burst = burst
        self._state = defaultdict(lambda: (burst, time.monotonic()))
        self._lock = threading.Lock()

    def allow(self, user_id: int) -> bool:
        with self._lock:
            tokens, last = self._state[user_id]
            now = time.monotonic()
            # Refill based on elapsed time
            tokens = min(self.burst, tokens + (now - last) * self.rate)
            if tokens >= 1.0:
                self._state[user_id] = (tokens - 1.0, now)
                return True
            self._state[user_id] = (tokens, now)
            return False

The trap

Using wall-clock (time.time()) instead of monotonic — NTP adjustments can move the clock backwards and make the limiter too generous or too strict. Forgetting the lock — race conditions between read and write.

Python 3 — Bloom filter for event dedupe at ingest

Prompt

Implement a Bloom filter to dedupe event_ids at ingest time. Capacity 10M items, false-positive rate ≤ 1%. Provide add(id) and maybe_contains(id) methods.

import math
import hashlib
import array

class BloomFilter:
    def __init__(self, capacity: int, fp_rate: float = 0.01):
        # m = -n ln p / (ln 2)^2  ; k = (m/n) ln 2
        self.m = int(-capacity * math.log(fp_rate) / (math.log(2) ** 2))
        self.k = max(1, int((self.m / capacity) * math.log(2)))
        # Bit array: one bit per slot, packed 8 per byte
        self.bits = array.array('B', [0] * ((self.m + 7) // 8))

    def _positions(self, key: bytes):
        h1 = int.from_bytes(hashlib.blake2b(key, digest_size=8).digest(), 'big')
        h2 = int.from_bytes(hashlib.sha256(key).digest()[:8], 'big')
        for i in range(self.k):
            yield (h1 + i * h2) % self.m

    def add(self, key) -> None:
        b = str(key).encode()
        for p in self._positions(b):
            self.bits[p >> 3] |= (1 << (p & 7))

    def maybe_contains(self, key) -> bool:
        b = str(key).encode()
        return all(self.bits[p >> 3] & (1 << (p & 7)) for p in self._positions(b))

The trap

Using Python set() — works in memory but blows through RAM at 10M items (~700 MB). The bloom filter is roughly 12 MB for the same specs. Interviewers ask because they want to see you pick the right structure for scale.

Python 4 — Attribution resolver (streaming)

Prompt

Maintain a per-user rolling buffer of the last 7 days of touches in memory (assume you'll scale this out later). When a conversion event arrives, return the most recent qualifying touch using the same last-click-with-view-fallback logic from SQL 1.

from collections import defaultdict, deque
from dataclasses import dataclass
from datetime import datetime, timedelta
from typing import Optional

@dataclass
class Touch:
    touch_id: int
    campaign_id: int
    ts: datetime
    touch_type: str  # 'click' or 'view'

CLICK_WINDOW = timedelta(days=7)
VIEW_WINDOW  = timedelta(days=1)

class AttributionResolver:
    def __init__(self):
        self.touches: dict[int, deque[Touch]] = defaultdict(deque)

    def record_touch(self, user_id: int, t: Touch) -> None:
        dq = self.touches[user_id]
        dq.append(t)
        # Evict anything older than the longest window
        cutoff = t.ts - CLICK_WINDOW
        while dq and dq[0].ts < cutoff:
            dq.popleft()

    def attribute(self, user_id: int, conversion_ts: datetime) -> Optional[Touch]:
        dq = self.touches.get(user_id) or ()
        best: Optional[Touch] = None
        for t in dq:
            if t.ts > conversion_ts:
                continue
            in_window = (
                (t.touch_type == 'click' and conversion_ts - t.ts <= CLICK_WINDOW)
                or (t.touch_type == 'view' and conversion_ts - t.ts <= VIEW_WINDOW)
            )
            if not in_window:
                continue
            # Prefer click over view, then most-recent
            if best is None:
                best = t
            else:
                if best.touch_type == 'view' and t.touch_type == 'click':
                    best = t
                elif best.touch_type == t.touch_type and t.ts > best.ts:
                    best = t
        return best

Python 5 — SKAN conversion-value encoder

Prompt

Apple's SKAdNetwork gives the advertiser 6 bits of "conversion value" (0–63) to encode an event signal back to the ad network. Build an encoder that maps a sequence of user events (registered, tutorial_done, purchase with revenue tier) to a single integer 0–63.

from dataclasses import dataclass

@dataclass
class UserState:
    registered: bool = False
    tutorial_done: bool = False
    revenue_usd: float = 0.0

# 6 bits: 1 bit registered | 1 bit tutorial | 4 bits revenue tier
# 16 revenue tiers: 0, $1, $2, $5, $10, $20, $50, $100, ...
REVENUE_TIERS = [0, 1, 2, 5, 10, 20, 50, 100, 200, 500, 1000, 2000, 5000, 10000, 25000, 100000]

def encode_conversion_value(state: UserState) -> int:
    reg_bit = 1 if state.registered else 0
    tut_bit = 1 if state.tutorial_done else 0
    tier = 0
    for i, threshold in enumerate(REVENUE_TIERS):
        if state.revenue_usd >= threshold:
            tier = i
    # Pack: [reg:1][tut:1][tier:4]
    return (reg_bit << 5) | (tut_bit << 4) | tier

def decode_conversion_value(v: int) -> dict:
    return {
        'registered': bool((v >> 5) & 1),
        'tutorial_done': bool((v >> 4) & 1),
        'revenue_tier_index': v & 0b1111,
        'revenue_floor_usd': REVENUE_TIERS[v & 0b1111],
    }

The trap

Trying to encode exact revenue in 6 bits — it does not fit. The tiered mapping is the whole point. Senior candidates name the tier-design tradeoff: finer tiers at low revenue (more common) vs coarser tiers at high revenue (rare).

Python 6 — Incrementality lift with confidence interval

Prompt

Given exposed and holdout conversion counts and user counts, compute the lift percentage and a 95% CI for the difference in proportions. Return (lift_pct, ci_half_width_pp, is_significant).

import math

def lift_with_ci(
    exposed_convs: int, exposed_users: int,
    holdout_convs: int, holdout_users: int,
    z: float = 1.96,  # 95% normal approx
) -> tuple[float, float, bool]:
    p1 = exposed_convs / exposed_users
    p2 = holdout_convs / holdout_users
    abs_diff = p1 - p2
    # Standard error of difference in two proportions
    se = math.sqrt(p1 * (1 - p1) / exposed_users + p2 * (1 - p2) / holdout_users)
    ci_half = z * se
    lift_pct = (p1 - p2) / p2 * 100 if p2 > 0 else float('inf')
    # Significant if the CI doesn't cross zero
    significant = (abs_diff - ci_half) > 0
    return lift_pct, ci_half * 100, significant

The trap

Reporting the lift without the CI — a +2% lift with ±3% CI is not a lift. Senior candidates return the significance flag alongside the number so the caller cannot misread it.

3. How to Approach the Ads Coding Round

Four habits that separate strong candidates in ads coding rounds specifically.

  1. Ask for the attribution window before you write a line. Last-click? 7-day? 28-day? Click-only or click-plus-view? Half the SQL problems above change shape depending on this answer. If the interviewer didn't volunteer it, ask.
  2. Name the dedupe strategy out loud. Even if the prompt doesn't mention duplicates, your code should handle them. Say: "I'll assume event_id may repeat across SDK retries; I'll dedupe on (user_id, event_name, second-rounded ts)." If the interviewer says "no, event_id is unique" — great, you asked, and you write a simpler query. If they say "good point" — you just showed senior judgment.
  3. Write schema-aware code. Ads tables are billions of rows. Any code that does SELECT * into Python, or loops a Python function over rows, will be scored junior. Push work into the SQL engine or into vectorized operations. When in doubt, ask: "Can this run as a single SQL query? If not, what's the smallest Python layer on top?"
  4. Return the uncertainty, not just the point estimate. Every ads number has statistical noise. The senior answer to "what's the lift" is never a single number — it's a number plus a confidence range, plus whether the range crosses zero. Build that shape into your functions as default output, not an optional afterthought.
