Part 0 · What Data Engineering Actually Is

Most people meet a data engineer through their absence. The dashboard is stale. The number doesn't match the other number. The ML model trained on yesterday's distribution and now nobody trusts the score. Somewhere upstream, a pipeline shrugged.

Data engineering is the discipline of moving data from where it is born — clicks, payments, sensors, logs, third-party APIs — to where decisions get made, and doing it with enough rigor that the people downstream (analysts, scientists, ML systems, executives, regulators) can build on top without flinching. It sits between software engineering and analytics, and inherits the obligations of both: the correctness of code, the trust of numbers.

A Working Definition

A data engineer designs, builds, and operates the systems that turn raw events into reliable, queryable, governed datasets — at the latency, cost, and quality the business has agreed to.

The five things you actually do

  1. Ingest. Pull data from operational systems (Postgres, Kafka, S3 drops, SaaS APIs) without losing rows, without double-counting, and without melting the source.
  2. Model. Decide what a "user," an "order," a "session," a "subscription" means — and shape the data so a hundred analysts get the same answer.
  3. Transform. Write SQL/Python/Spark that turns raw events into clean facts and dimensions, idempotently, on a schedule, and re-runnable.
  4. Serve. Make those datasets available — warehouse tables, lakehouse files, low-latency caches, feature stores, BI semantic layers.
  5. Operate. Monitor freshness, completeness, schema drift, cost, lineage; respond when something breaks; explain why a number changed.

The skill stack interviewers actually probe

Different companies emphasize different layers, but the canonical L3–L5 stack is well-known:

  • SQL fluency — joins, window functions, CTEs, recursive CTEs, sessionization, gaps-and-islands, dedup patterns, query plans, partition pruning.
  • Python — pandas for medium data, PySpark for big data, plus enough vanilla Python (collections, generators, OOP basics) to write a small ETL utility from scratch.
  • Data modeling — star schemas, fact-vs-dim, slowly changing dimensions (SCD Type 1/2/3), grain decisions, late-arriving facts.
  • Distributed compute — Spark / Trino / BigQuery internals, shuffle, skew, broadcast joins, partitioning, bucketing.
  • Streaming — Kafka semantics, exactly-once vs at-least-once, watermarks, windowing, late data.
  • Lakehouse — Iceberg / Delta / Hudi, table formats, time travel, compaction, schema evolution.
  • Orchestration & reliability — Airflow / Dagster / Prefect, retries, backfills, idempotency, SLAs, DQ checks.
  • Cloud & cost — S3 / GCS / ADLS, IAM, partitioning for cost, query cost analysis.

This article focuses sharply on the two skills that every loop tests: SQL and Python. If you can't ace the SQL screen and the coding round, the design round never happens.

Part 1 · How to Prepare for SQL & Python Rounds

Most candidates over-study the wrong things — memorizing every window function, every pandas method — and under-study the things interviewers actually weight: clarifying the question, picking the right primitive, narrating tradeoffs, and writing code that runs.

The Four Things Interviewers Score

1. Clarity — Did you restate the problem? Did you ask about edge cases (nulls, ties, late data, time zones)?

2. Correctness — Does the query / function actually return the right answer on the obvious test cases?

3. Reasoning — Can you talk through why you picked this join / window / pattern? Can you compare it to one alternative?

4. Production sense — Would this scale? Is it idempotent? What happens at 100x the volume? What happens if the source schema changes?

A 2-week practice plan

  1. Days 1–3 — Foundations. Tier 1 SQL (Q1–Q10) and Tier 1 Python (Q1–Q10). Get fluent with the dataset. No timer.
  2. Days 4–6 — Joins & aggregation. Tier 2 of both. Time-box at 10 minutes per question. Talk out loud (yes, alone).
  3. Days 7–9 — Windows & CTEs. Tier 3. Force yourself to write the answer once without looking, then critique your own.
  4. Days 10–12 — Production patterns. Tier 4. Dedup, sessionize, funnel, retention, date spines.
  5. Days 13–14 — Senior signals. Tier 5. SCD2, recursive CTEs, skew, Spark, idempotency. Know the tradeoffs, not just the syntax.

How to use this question bank

Each question has three pieces: a scenario (what a real PM or analyst would actually ask), a solution (the SQL or Python that answers it), and a brief why-it-matters note (the production angle, edge case, or interviewer follow-up). Read the scenario, write your own answer, then compare. The compare-after-attempt step is where the learning lives.

Part 3 · 50 SQL Questions, Simple to Complex

Open a Postgres / DuckDB / SQLite shell, load the CSVs, and try each question on your own before reading the solution. Solutions use ANSI SQL with a Postgres flavor; Spark / BigQuery / Snowflake variants are noted where they differ meaningfully.

Tier 1 · Foundation (Q1–Q10)

SQL · Q1Tier 1

How many users do we have per country?

A growth PM wants to see where our 200 users are concentrated before deciding which market to localize for next.

SELECT country, COUNT(*) AS user_count
FROM users
GROUP BY country
ORDER BY user_count DESC;

Why it matters: The bread-and-butter GROUP BY. Interviewers watch whether you alias the count, sort the result, and resist SELECT *.

SQL · Q2Tier 1

Show the top 5 highest-paid employees with their department.

HR is auditing comp bands and wants the top earners across the company.

SELECT employee_id, full_name, department, salary
FROM employees
ORDER BY salary DESC
LIMIT 5;

Follow-up: "What if two people tie at #5?" → discuss FETCH FIRST 5 ROWS WITH TIES (Postgres 13+) or RANK() OVER (ORDER BY salary DESC).

SQL · Q3Tier 1

What is the total revenue across all purchases?

Finance needs the simplest possible top-line number for a board slide.

SELECT ROUND(SUM(amount), 2) AS total_revenue
FROM purchases;

Edge case: If amount can be NULL, SUM already ignores nulls — but state that explicitly. Interviewers love when you pre-empt the null question.

SQL · Q4Tier 1

Which products generated more than $1,000 in revenue?

Merchandising wants the "winners shelf" — anything clearing four figures of contribution.

SELECT product_id, product_name, category, revenue
FROM products
WHERE revenue > 1000
ORDER BY revenue DESC;

Why it matters: WHERE + sort. Trivial — but watch candidates who use HAVING here without a GROUP BY.

SQL · Q5Tier 1

What is the average session duration by platform?

Product wants to know if iOS users actually spend longer in the app than Android users.

SELECT platform,
       COUNT(*)                          AS sessions,
       ROUND(AVG(duration_sec)::NUMERIC, 1) AS avg_duration_sec
FROM sessions
GROUP BY platform
ORDER BY avg_duration_sec DESC;

Follow-up: "Why include the count?" → averages without sample size are dangerously seductive.

SQL · Q6Tier 1

How many sessions occur on weekends only?

An ops lead suspects weekend traffic is driving the support backlog and wants to size it.

SELECT COUNT(*) AS weekend_sessions
FROM sessions
WHERE EXTRACT(DOW FROM session_date) IN (0, 6);  -- Sun=0, Sat=6 in Postgres

Portability: MySQL uses DAYOFWEEK() (1=Sun…7=Sat); Spark uses dayofweek() with the same convention as MySQL. Always state your DB.

SQL · Q7Tier 1

List all distinct event types in the events table.

A new analyst is exploring the dataset and wants to know what funnel stages exist before writing anything else.

SELECT DISTINCT event_type
FROM events
ORDER BY event_type;

Production angle: In a real warehouse, prefer querying a documented enum/lookup table instead of DISTINCT on a high-cardinality fact table.

SQL · Q8Tier 1

How many users signed up in Q4 2024?

Marketing is reconciling Q4 acquisition numbers against the quarterly board pack.

SELECT COUNT(*) AS q4_2024_signups
FROM users
WHERE signup_date >= DATE '2024-10-01'
  AND signup_date <  DATE '2025-01-01';

Why half-open ranges: >= start AND < end is timezone-safe, leap-day-safe, and avoids the classic BETWEEN '2024-12-31' off-by-one when timestamps creep in.

SQL · Q9Tier 1

What is the largest single order amount, and on what date?

A fraud analyst wants the top-of-distribution for an outlier review.

SELECT order_id, user_id, order_date, amount
FROM orders
ORDER BY amount DESC
LIMIT 1;

Alternative: A subquery with WHERE amount = (SELECT MAX(amount) FROM orders) — and discuss why ORDER BY ... LIMIT is usually clearer and almost always indexed-friendlier.

SQL · Q10Tier 1

Show the number of sessions per day for the most recent 7 days in the data.

An on-call engineer wants a quick freshness sanity check after a deploy.

SELECT session_date, COUNT(*) AS sessions
FROM sessions
WHERE session_date >= (SELECT MAX(session_date) - INTERVAL '6 days' FROM sessions)
GROUP BY session_date
ORDER BY session_date;

Why MAX() instead of CURRENT_DATE: the data may be stale; you want "last 7 days that exist," not "last 7 calendar days."

Tier 2 · Joins & Multi-Table (Q11–Q20)

SQL · Q11Tier 2

Find users who have never made a purchase.

Lifecycle marketing wants to target dormant users with a "first purchase" coupon.

SELECT u.user_id, u.name, u.country, u.signup_date
FROM users u
LEFT JOIN purchases p ON p.user_id = u.user_id
WHERE p.purchase_id IS NULL;

Why anti-join via LEFT JOIN + IS NULL: often plans better than NOT IN (which silently breaks on NULLs) or NOT EXISTS (correct but sometimes slower depending on optimizer).

SQL · Q12Tier 2

What is the average purchase amount per user country?

Pricing wants to test whether IN, UK, and US baskets differ enough to justify country-specific promotions.

SELECT u.country,
       COUNT(p.purchase_id)        AS purchases,
       ROUND(AVG(p.amount), 2)     AS avg_basket
FROM users u
JOIN purchases p ON p.user_id = u.user_id
GROUP BY u.country
ORDER BY avg_basket DESC;

Trap: A LEFT JOIN here would inflate countries with many no-purchase users — they'd show AVG = NULL, not 0, which is correct but easy to mis-read.

SQL · Q13Tier 2

Which users appear in users but are missing from user_profiles?

A data quality check before a profile-personalization launch — you can't personalize for users with no profile row.

SELECT u.user_id, u.name
FROM users u
WHERE NOT EXISTS (
    SELECT 1 FROM user_profiles up WHERE up.user_id = u.user_id
);

Why NOT EXISTS here: safe with NULLs, semantically clearest, and the optimizer typically transforms it into an anti-join.

SQL · Q14Tier 2

Total spend per user — name and amount, top 10 spenders.

CX is building a "VIP" segment for white-glove support.

SELECT u.user_id, u.name,
       ROUND(SUM(o.amount), 2) AS total_spend,
       COUNT(o.order_id)       AS order_count
FROM users u
JOIN orders o ON o.user_id = u.user_id
GROUP BY u.user_id, u.name
ORDER BY total_spend DESC
LIMIT 10;

Note: orders.user_id is not a strict FK in the dataset — discuss what happens if an order references a missing user (use LEFT JOIN to find them; see Q46).

SQL · Q15Tier 2

List each employee with their manager's name.

HR wants a flat org list for onboarding packets — employee, title, who they report to.

SELECT e.employee_id, e.full_name AS employee, e.department,
       m.full_name AS manager
FROM employees e
LEFT JOIN employees m ON m.employee_id = e.manager_id
ORDER BY e.department, e.full_name;

Why LEFT JOIN: the CEO has no manager (manager_id IS NULL) and would otherwise vanish.

SQL · Q16Tier 2

Salary delta vs. each employee's manager.

A comp review: which managers earn less than one of their direct reports? (A real signal worth flagging.)

SELECT e.full_name AS employee, e.salary AS emp_salary,
       m.full_name AS manager,  m.salary AS mgr_salary,
       (m.salary - e.salary)    AS manager_premium
FROM employees e
JOIN employees m ON m.employee_id = e.manager_id
WHERE m.salary < e.salary;

Follow-up: "What about skip-level comparisons?" — that's the recursive CTE in Q42–Q43.

SQL · Q17Tier 2

Top 3 product categories by total purchase revenue.

Merchandising wants to know where to double down for the holiday push.

SELECT product_category,
       ROUND(SUM(amount), 2) AS revenue,
       COUNT(*)              AS purchases
FROM purchases
GROUP BY product_category
ORDER BY revenue DESC
LIMIT 3;

Subtle: "by revenue" can mean SUM(amount) or COUNT(*) × avg price. Always confirm which revenue definition before writing.

SQL · Q18Tier 2

Find users who appear in both events and logins (active across both surfaces).

Engagement is defining "truly active" as "did a tracked event AND logged in" within the data window.

SELECT DISTINCT e.user_id
FROM events e
WHERE EXISTS (SELECT 1 FROM logins l WHERE l.user_id = e.user_id);

-- Or, equivalently:
SELECT user_id FROM events
INTERSECT
SELECT user_id FROM logins;

Tradeoff: INTERSECT is concise and dedups for free; EXISTS wins when one side is much larger or you want to stream early.

SQL · Q19Tier 2

Country-level signup→purchase conversion rate.

Growth wants a per-country funnel: of all users who signed up, what fraction ever purchased?

SELECT u.country,
       COUNT(DISTINCT u.user_id)                         AS signups,
       COUNT(DISTINCT p.user_id)                         AS purchasers,
       ROUND(100.0 * COUNT(DISTINCT p.user_id)
                   / NULLIF(COUNT(DISTINCT u.user_id), 0), 2) AS conv_pct
FROM users u
LEFT JOIN purchases p ON p.user_id = u.user_id
GROUP BY u.country
ORDER BY conv_pct DESC;

Why NULLIF(..., 0): defends against divide-by-zero if a country has no signups (won't happen here, but it's a habit).

SQL · Q20Tier 2

Join products with the current price from product_dim (SCD2).

A pricing dashboard needs the live price for each catalog product, ignoring expired historical rows.

SELECT p.product_id, p.product_name, p.category,
       pd.price AS current_price, pd.effective_date
FROM products p
LEFT JOIN product_dim pd
       ON pd.product_id = p.product_id
      AND pd.is_current = 'Y';

Setup for Q41: using is_current = 'Y' is fast but only correct for "now" queries. For "as-of-some-historical-date" you must use effective_date / expiry_date windows.

Tier 3 · Window Functions, CASE, Subqueries (Q21–Q30)

SQL · Q21Tier 3

Rank users by total order spend (with ties).

A loyalty program needs ranks, not raw amounts — and people with identical spend should share a rank.

SELECT user_id,
       ROUND(SUM(amount), 2)                          AS total_spend,
       RANK()       OVER (ORDER BY SUM(amount) DESC)  AS spend_rank,
       DENSE_RANK() OVER (ORDER BY SUM(amount) DESC)  AS spend_dense_rank
FROM orders
GROUP BY user_id
ORDER BY spend_rank
LIMIT 20;

RANK vs DENSE_RANK vs ROW_NUMBER: ranks skip after ties (1,2,2,4); dense_rank doesn't (1,2,2,3); row_number breaks ties arbitrarily. Pick deliberately.

SQL · Q22Tier 3

Deduplicate user_profiles — keep the latest row per user.

The CRM team wants one row per user: the most recent updated_at wins. (The table has 334 rows for 200 users.)

WITH ranked AS (
    SELECT up.*,
           ROW_NUMBER() OVER (
               PARTITION BY user_id
               ORDER BY updated_at DESC, city DESC  -- tiebreak deterministic
           ) AS rn
    FROM user_profiles up
)
SELECT user_id, name, city, updated_at
FROM ranked
WHERE rn = 1;

Always tiebreak. Without the secondary ORDER BY, two rows with identical updated_at produce a non-deterministic winner — the kind of bug that only shows up after a backfill.

SQL · Q23Tier 3

WoW growth in WAU using LAG.

The weekly business review wants WAU and "% change vs last week" in one query.

SELECT week_start,
       wau,
       LAG(wau) OVER (ORDER BY week_start)                  AS prev_wau,
       ROUND( 100.0 * (wau - LAG(wau) OVER (ORDER BY week_start))
                    / NULLIF(LAG(wau) OVER (ORDER BY week_start), 0), 2) AS wow_pct
FROM weekly_metrics
ORDER BY week_start;

Readability tip: in a real codebase, wrap the LAG() in a CTE so you don't repeat it three times.

SQL · Q24Tier 3

CASE pivot — sessions per platform per day.

A weekly report wants one row per day with three columns: iOS, Android, Web.

SELECT session_date,
       SUM(CASE WHEN platform = 'ios'     THEN 1 ELSE 0 END) AS ios,
       SUM(CASE WHEN platform = 'android' THEN 1 ELSE 0 END) AS android,
       SUM(CASE WHEN platform = 'web'     THEN 1 ELSE 0 END) AS web,
       COUNT(*)                                              AS total
FROM sessions
GROUP BY session_date
ORDER BY session_date;

vs. native PIVOT: Snowflake / SQL Server have PIVOT; Postgres uses the CASE pattern (or crosstab in tablefunc). The CASE form is the most portable.

SQL · Q25Tier 3

Top 3 highest-revenue products in each category.

Merchandising wants a "shelf" of the three best products per category for a homepage rotation.

WITH ranked AS (
    SELECT product_id, product_name, category, revenue,
           DENSE_RANK() OVER (PARTITION BY category
                              ORDER BY revenue DESC) AS rk
    FROM products
)
SELECT category, product_id, product_name, revenue, rk
FROM ranked
WHERE rk <= 3
ORDER BY category, rk;

Why DENSE_RANK: if two products tie for #2, you keep both and still get a #3.

SQL · Q26Tier 3

Median order amount.

A finance lead is suspicious of AVG(amount) being skewed by a few whale orders and asks for the median.

-- Postgres / Snowflake / Spark: PERCENTILE_CONT
SELECT PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY amount) AS median_amount
FROM orders;

-- Portable fallback (works in MySQL too):
SELECT AVG(amount) AS median_amount
FROM (
    SELECT amount,
           ROW_NUMBER() OVER (ORDER BY amount) AS rn,
           COUNT(*)     OVER ()                AS n
    FROM orders
) t
WHERE rn IN (FLOOR((n+1)/2.0), CEIL((n+1)/2.0));

Why two answers: show both. The first is idiomatic; the second proves you understand what median means.

SQL · Q27Tier 3

For each user, find their largest order and its date.

A re-engagement campaign wants to reference each customer's biggest-ever order in the email subject line.

WITH ranked AS (
    SELECT order_id, user_id, order_date, amount,
           ROW_NUMBER() OVER (PARTITION BY user_id
                              ORDER BY amount DESC, order_date DESC) AS rn
    FROM orders
)
SELECT user_id, order_id, order_date, amount
FROM ranked
WHERE rn = 1;

Window beats correlated subquery: a single pass over orders instead of N subqueries. On large tables this is a 10–100× difference.

SQL · Q28Tier 3

Cumulative distribution of order amounts.

A pricing experiment wants to know "what fraction of orders are at or below $X" for a tier definition.

SELECT order_id, amount,
       ROUND(CUME_DIST() OVER (ORDER BY amount)::NUMERIC, 4) AS cume_pct,
       NTILE(10)         OVER (ORDER BY amount)              AS decile
FROM orders
ORDER BY amount;

CUME_DIST vs PERCENT_RANK: CUME_DIST counts ties as "at or below" (≤); PERCENT_RANK uses (rank-1)/(N-1). Use CUME_DIST for "what % of orders are ≤ this".

SQL · Q29Tier 3

For each order, show the difference vs. that user's average order.

Anomaly detection: which orders are unusually large for that specific user?

SELECT order_id, user_id, order_date, amount,
       ROUND(AVG(amount) OVER (PARTITION BY user_id), 2) AS user_avg,
       ROUND(amount - AVG(amount) OVER (PARTITION BY user_id), 2) AS delta
FROM orders
ORDER BY user_id, order_date;

Window without ORDER BY means the average covers the whole partition (the user) — exactly what we want here.

SQL · Q30Tier 3

Find users who signed up but never reached purchase in the events funnel.

Lifecycle wants the dropped-off cohort: users who got past signup but never completed a purchase event.

SELECT DISTINCT e.user_id
FROM events e
WHERE e.event_type = 'signup'
  AND NOT EXISTS (
      SELECT 1 FROM events e2
      WHERE e2.user_id = e.user_id
        AND e2.event_type = 'purchase'
  );

Cohort-correctness: if "signup" can fire multiple times per user (it shouldn't, but data is data), this still works because DISTINCT + NOT EXISTS are set-based.

Tier 4 · CTEs, Sessionization, Cohorts (Q31–Q40)

SQL · Q31Tier 4

7-day moving average of daily revenue.

Finance wants a smoother trend than the spiky daily series for a board chart.

SELECT ds, daily_revenue,
       ROUND(AVG(daily_revenue) OVER (
           ORDER BY ds
           ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
       ), 2) AS rev_7d_avg
FROM daily_metrics
ORDER BY ds;

ROWS vs RANGE: ROWS counts rows; RANGE matches values. For dense daily data they're equivalent — but if dates are gapped, RANGE with an interval is the safer "true 7-day window."

SQL · Q32Tier 4

Running monthly revenue total, reset each month.

A finance dashboard wants "month-to-date revenue" on every row.

SELECT ds, daily_revenue,
       SUM(daily_revenue) OVER (
           PARTITION BY DATE_TRUNC('month', ds)
           ORDER BY ds
           ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS mtd_revenue
FROM daily_metrics
ORDER BY ds;

The "reset" is the partition. Group on the period you want to reset over; window-order within it.

SQL · Q33Tier 4

Weekly retention by signup cohort (W0, W1, W2, W3).

Growth wants a triangle: rows = signup week, columns = weeks since signup, values = % of cohort logging in.

WITH cohort AS (
    SELECT user_id,
           DATE_TRUNC('week', signup_date) AS cohort_week
    FROM users
),
joined AS (
    SELECT c.cohort_week,
           c.user_id,
           FLOOR(
             EXTRACT(EPOCH FROM (l.login_date - c.cohort_week)) / (7*86400)
           )::INT AS weeks_since
    FROM cohort c
    LEFT JOIN logins l ON l.user_id = c.user_id
)
SELECT cohort_week,
       COUNT(DISTINCT user_id) FILTER (WHERE weeks_since = 0) AS w0,
       COUNT(DISTINCT user_id) FILTER (WHERE weeks_since = 1) AS w1,
       COUNT(DISTINCT user_id) FILTER (WHERE weeks_since = 2) AS w2,
       COUNT(DISTINCT user_id) FILTER (WHERE weeks_since = 3) AS w3
FROM joined
WHERE weeks_since BETWEEN 0 AND 3
GROUP BY cohort_week
ORDER BY cohort_week;

Convert to %: divide each wN by the cohort size (a separate CTE). Most interviewers will follow up by asking for that.

SQL · Q34Tier 4

Funnel conversion: signup → activation → purchase.

Product wants the absolute counts and step-to-step conversion rates from the events table.

WITH first_event AS (
    SELECT user_id, event_type,
           MIN(event_date) AS first_at
    FROM events
    WHERE event_type IN ('signup','activation','purchase')
    GROUP BY user_id, event_type
),
pivoted AS (
    SELECT user_id,
           MAX(CASE WHEN event_type='signup'     THEN first_at END) AS signup_at,
           MAX(CASE WHEN event_type='activation' THEN first_at END) AS activation_at,
           MAX(CASE WHEN event_type='purchase'   THEN first_at END) AS purchase_at
    FROM first_event GROUP BY user_id
)
SELECT
    COUNT(*) FILTER (WHERE signup_at     IS NOT NULL) AS step1_signup,
    COUNT(*) FILTER (WHERE activation_at IS NOT NULL
                     AND activation_at >= signup_at)  AS step2_activation,
    COUNT(*) FILTER (WHERE purchase_at   IS NOT NULL
                     AND purchase_at >= COALESCE(activation_at, signup_at))
                                                       AS step3_purchase
FROM pivoted;

Order matters. A "purchase" before "signup" is dirty data — the >= guards keep the funnel monotonic.

SQL · Q35Tier 4

Date spine: fill missing days in daily_metrics_gapped with zero.

A BI dashboard breaks because 15 days are missing — every chart needs every day, even zeros.

WITH spine AS (
    SELECT generate_series(
        (SELECT MIN(ds) FROM daily_metrics_gapped),
        (SELECT MAX(ds) FROM daily_metrics_gapped),
        INTERVAL '1 day'
    )::DATE AS ds
)
SELECT s.ds,
       COALESCE(d.daily_revenue, 0) AS daily_revenue,
       COALESCE(d.daily_dau,     0) AS daily_dau,
       (d.ds IS NULL)               AS was_missing
FROM spine s
LEFT JOIN daily_metrics_gapped d ON d.ds = s.ds
ORDER BY s.ds;

Spark/BigQuery: use sequence(start, stop, step). Snowflake: GENERATOR. Same pattern, different syntax.

SQL · Q36Tier 4

Sessionize the clickstream — 30 minutes of inactivity = new session.

Product wants per-user "sessions" from raw page events. Industry standard: 30-min idle gap.

WITH gapped AS (
    SELECT user_id, event_time, page,
           LAG(event_time) OVER (PARTITION BY user_id ORDER BY event_time) AS prev_time
    FROM clickstream
),
flagged AS (
    SELECT user_id, event_time, page,
           CASE
             WHEN prev_time IS NULL
               OR event_time - prev_time > INTERVAL '30 minutes'
             THEN 1 ELSE 0
           END AS is_new_session
    FROM gapped
),
sessionized AS (
    SELECT user_id, event_time, page,
           SUM(is_new_session) OVER (PARTITION BY user_id ORDER BY event_time) AS session_id
    FROM flagged
)
SELECT user_id, session_id,
       MIN(event_time) AS session_start,
       MAX(event_time) AS session_end,
       COUNT(*)        AS pageviews
FROM sessionized
GROUP BY user_id, session_id
ORDER BY user_id, session_start;

The cumulative-sum trick is the canonical sessionization pattern. Memorize it.

SQL · Q37Tier 4

Longest consecutive login streak per user (gaps & islands).

Engagement wants to crown each user's "best streak" for a leaderboard.

WITH grp AS (
    SELECT user_id, login_date,
           login_date - (ROW_NUMBER() OVER (
               PARTITION BY user_id ORDER BY login_date
           ))::INT AS group_key
    FROM (SELECT DISTINCT user_id, login_date FROM logins) d
),
streaks AS (
    SELECT user_id, group_key,
           COUNT(*)        AS streak_len,
           MIN(login_date) AS streak_start,
           MAX(login_date) AS streak_end
    FROM grp
    GROUP BY user_id, group_key
)
SELECT user_id, MAX(streak_len) AS longest_streak
FROM streaks
GROUP BY user_id
ORDER BY longest_streak DESC
LIMIT 20;

The trick: date − row_number is constant within consecutive runs. Group by it, you get the islands. Dedup logins first or streaks lie.

SQL · Q38Tier 4

Detect data skew — which join keys hold most of the rows?

A Spark job is mysteriously slow on the skewed_table; you suspect one key is doing all the work.

SELECT join_key,
       COUNT(*)                                     AS rows,
       ROUND(100.0 * COUNT(*)
             / SUM(COUNT(*)) OVER (), 2)            AS pct_of_total
FROM skewed_table
GROUP BY join_key
ORDER BY rows DESC
LIMIT 10;

What you'll find: hot_key holds ~70% of rows. Mitigations: salt the hot key, broadcast the small side, or pre-aggregate before the join.

Tier 5 · Senior Patterns (Q41–Q50)

SQL · Q41Tier 5

SCD2 as-of join: price each purchase at the price that was effective on its date.

Finance wants historical revenue "at the right price," not at today's price — i.e., use product_dim with effective/expiry dates instead of is_current.

SELECT p.purchase_id, p.user_id, p.purchase_date,
       p.product_category, p.amount,
       pd.product_id, pd.price AS price_on_purchase_date
FROM purchases p
LEFT JOIN product_dim pd
       ON pd.product_name = p.product_category   -- (assume mapping for the exercise)
      AND p.purchase_date >= pd.effective_date
      AND (pd.expiry_date IS NULL OR p.purchase_date < pd.expiry_date)
ORDER BY p.purchase_date;

The SCD2 invariant: exactly one row matches per (product, point-in-time). If you ever see duplicates, your effective/expiry windows overlap — that's a data-quality bug, not a query bug.

SQL · Q42Tier 5

Recursive CTE: full org chart with reporting depth.

HR wants every employee with their distance from the CEO and their full reporting chain as a string.

WITH RECURSIVE org AS (
    SELECT employee_id, full_name, manager_id, department,
           0                          AS depth,
           full_name::TEXT            AS chain
    FROM employees
    WHERE manager_id IS NULL          -- the root(s)

    UNION ALL

    SELECT e.employee_id, e.full_name, e.manager_id, e.department,
           o.depth + 1,
           o.chain || ' > ' || e.full_name
    FROM employees e
    JOIN org o ON e.manager_id = o.employee_id
)
SELECT employee_id, full_name, department, depth, chain
FROM org
ORDER BY depth, full_name;

Termination: a recursive CTE stops when the recursive step yields no new rows. Cycles in manager_id would loop forever — guard with a depth < N cap in production.

SQL · Q43Tier 5

Recursive CTE: total org headcount under each manager.

A reorg planning exercise needs "people in your org" (yourself + everyone reporting up to you, transitively).

WITH RECURSIVE descendants AS (
    SELECT employee_id AS root_id, employee_id
    FROM employees

    UNION ALL

    SELECT d.root_id, e.employee_id
    FROM descendants d
    JOIN employees   e ON e.manager_id = d.employee_id
)
SELECT m.employee_id, m.full_name AS manager,
       COUNT(*) - 1 AS reports_below   -- subtract self
FROM descendants d
JOIN employees   m ON m.employee_id = d.root_id
GROUP BY m.employee_id, m.full_name
ORDER BY reports_below DESC;

Pattern: seed with self, recurse downward, count − 1 to exclude the root from their own org count.

SQL · Q44Tier 5

Cohort retention as a percentage matrix.

The same triangle as Q33, but each cell is a percentage of the cohort size — the chart growth teams actually look at.

WITH cohort AS (
    SELECT user_id, DATE_TRUNC('week', signup_date) AS cohort_week
    FROM users
),
cohort_sizes AS (
    SELECT cohort_week, COUNT(*) AS cohort_n FROM cohort GROUP BY cohort_week
),
activity AS (
    SELECT c.cohort_week, c.user_id,
           FLOOR(EXTRACT(EPOCH FROM (l.login_date - c.cohort_week)) / (7*86400))::INT AS w
    FROM cohort c
    JOIN logins l ON l.user_id = c.user_id
    WHERE l.login_date >= c.cohort_week
)
SELECT a.cohort_week, cs.cohort_n,
       ROUND(100.0 * COUNT(DISTINCT CASE WHEN w=0 THEN a.user_id END)/cs.cohort_n, 1) AS w0,
       ROUND(100.0 * COUNT(DISTINCT CASE WHEN w=1 THEN a.user_id END)/cs.cohort_n, 1) AS w1,
       ROUND(100.0 * COUNT(DISTINCT CASE WHEN w=2 THEN a.user_id END)/cs.cohort_n, 1) AS w2,
       ROUND(100.0 * COUNT(DISTINCT CASE WHEN w=3 THEN a.user_id END)/cs.cohort_n, 1) AS w3
FROM activity a
JOIN cohort_sizes cs USING (cohort_week)
GROUP BY a.cohort_week, cs.cohort_n
ORDER BY a.cohort_week;

Watch the denominator. Some teams normalize by W0 (always 100%) instead of cohort_n — both are valid; pick one and label it.

SQL · Q45Tier 5

New vs returning users by day from logins.

A daily report wants two lines: "new users today" (first-ever login) and "returning users today" (logged in before).

WITH first_seen AS (
    SELECT user_id, MIN(login_date) AS first_login
    FROM logins
    GROUP BY user_id
)
SELECT l.login_date,
       COUNT(DISTINCT CASE WHEN l.login_date  = f.first_login THEN l.user_id END) AS new_users,
       COUNT(DISTINCT CASE WHEN l.login_date > f.first_login THEN l.user_id END) AS returning_users
FROM logins     l
JOIN first_seen f USING (user_id)
GROUP BY l.login_date
ORDER BY l.login_date;

Definition discipline: "new" = first-ever-anywhere; "returning" = active today and active before. Some teams want "first-ever-this-month" — clarify before coding.

SQL · Q47Tier 5

Reconcile daily_metrics.daily_revenue against the sum of purchases.amount per day.

A finance audit: the curated metric and the raw fact should match within a tolerance, every day.

WITH src AS (
    SELECT purchase_date AS ds,
           SUM(amount)    AS src_revenue
    FROM purchases
    GROUP BY purchase_date
),
diff AS (
    SELECT d.ds,
           d.daily_revenue                     AS metric_revenue,
           COALESCE(s.src_revenue, 0)          AS source_revenue,
           d.daily_revenue - COALESCE(s.src_revenue, 0) AS delta
    FROM daily_metrics d
    LEFT JOIN src s ON s.ds = d.ds
)
SELECT *
FROM diff
WHERE ABS(delta) > 0.01
ORDER BY ABS(delta) DESC;

Why a tolerance: floating point + currency rounding mean exact equality is the wrong check. Pick a meaningful epsilon (e.g., one cent).

SQL · Q48Tier 5

Per-user outlier orders: more than 3σ above that user's mean.

Fraud / risk wants a flag on orders that are anomalously large for the specific user's history.

WITH stats AS (
    SELECT order_id, user_id, order_date, amount,
           AVG(amount)    OVER (PARTITION BY user_id) AS u_avg,
           STDDEV(amount) OVER (PARTITION BY user_id) AS u_std,
           COUNT(*)       OVER (PARTITION BY user_id) AS u_n
    FROM orders
)
SELECT *
FROM stats
WHERE u_n >= 5                       -- enough history to trust
  AND amount > u_avg + 3 * u_std
ORDER BY amount - u_avg DESC;

Statistical caveat: 3σ assumes roughly normal — order amounts are usually log-normal. For production, log-transform first or use IQR / MAD.

SQL · Q49Tier 5

"Power users" — top 10% by orders and top 10% by sessions.

CX wants the intersection of high-spend and high-engagement users for a VIP launch.

WITH order_rank AS (
    SELECT user_id,
           NTILE(10) OVER (ORDER BY SUM(amount) DESC) AS o_decile
    FROM orders GROUP BY user_id
),
sess_rank AS (
    SELECT user_id,
           NTILE(10) OVER (ORDER BY COUNT(*) DESC)    AS s_decile
    FROM sessions GROUP BY user_id
)
SELECT o.user_id
FROM order_rank o
JOIN sess_rank  s USING (user_id)
WHERE o.o_decile = 1 AND s.s_decile = 1;

NTILE caveat: NTILE distributes ties arbitrarily, so two users with identical spend may land in different deciles. For high-stakes segmentation, prefer PERCENT_RANK <= 0.10.

SQL · Q50Tier 5

Daily ARPU + a sanity check against the raw orders.

A daily ARPU (avg revenue per active user) is being shipped to the exec dashboard. You want both the curated number and a self-computed reconciliation.

WITH curated AS (
    SELECT ds, daily_revenue, daily_dau,
           ROUND(daily_revenue / NULLIF(daily_dau, 0), 4) AS arpu_curated
    FROM daily_metrics
),
raw_arpu AS (
    SELECT order_date AS ds,
           SUM(amount)              AS raw_revenue,
           COUNT(DISTINCT user_id)  AS raw_active_users,
           ROUND(SUM(amount)
                 / NULLIF(COUNT(DISTINCT user_id), 0), 4) AS arpu_raw
    FROM orders
    GROUP BY order_date
)
SELECT c.ds,
       c.arpu_curated, r.arpu_raw,
       ROUND(c.arpu_curated - r.arpu_raw, 4) AS arpu_delta
FROM curated c
LEFT JOIN raw_arpu r USING (ds)
ORDER BY c.ds;

Why ship the delta: a metric is only as trusted as the reconciliation report behind it. Senior interviewers love this answer because it shows you think about trust, not just calculation.

Compute DAU and 7-day rolling WAU from logins.

               .rename("dau").reset_index())

# rolling distinct count is non-trivial — approximate via set logic:

    return df_full.loc[(df_full["login_date"] >= start)
                       & (df_full["login_date"] <= idx),
                       "user_id"].nunique()

Build a tiny DAG executor in pure Python (topological sort).

A 5-line task DAG used to demonstrate orchestration intuition without leaning on Airflow.

from collections import defaultdict, deque

def topo_run(graph: dict, run_fn):
    indeg = defaultdict(int)
    rev   = defaultdict(list)
    for node, deps in graph.items():
        indeg[node]      # ensure key exists
        for d in deps:
            indeg[node] += 1
            rev[d].append(node)
    q = deque([n for n, k in indeg.items() if k == 0])
    order = []
    while q:
        n = q.popleft()
        run_fn(n)
        order.append(n)
        for nxt in rev[n]:
            indeg[nxt] -= 1
            if indeg[nxt] == 0:
                q.append(nxt)
    if len(order) != len(graph):
        raise ValueError("cycle in DAG")
    return order

dag = {"extract":[], "stage":["extract"], "transform":["stage"],
       "publish":["transform"], "audit":["transform"]}
print(topo_run(dag, print))

Why this question shows up: orchestration is half the job. Knowing toposort + cycle detection by heart proves you've built (not just used) a scheduler.

PY · Q47Tier 5

Implement an LRU cache (no functools).

A classic mid-level Python question — proves you know OrderedDict and big-O.

from collections import OrderedDict

class LRUCache:
    def __init__(self, capacity: int):
        self.cap = capacity
        self.d   = OrderedDict()

    def get(self, key):
        if key not in self.d:
            return None
        self.d.move_to_end(key)
        return self.d[key]

    def put(self, key, value):
        if key in self.d:
            self.d.move_to_end(key)
        self.d[key] = value
        if len(self.d) > self.cap:
            self.d.popitem(last=False)

c = LRUCache(2)
c.put("a", 1); c.put("b", 2); c.get("a"); c.put("c", 3)
print(list(c.d.items()))   # 'b' evicted

O(1) get/put. If you write the doubly-linked-list version from scratch you'll impress, but the OrderedDict version is the real-world answer.

PY · Q48Tier 5

Streaming median with two heaps.

Compute the running median of orders.amount streamed one-at-a-time, in O(log n) per insert.

import heapq

class StreamingMedian:
    def __init__(self):
        self.lo = []   # max-heap (invert sign)
        self.hi = []   # min-heap

    def add(self, x: float):
        heapq.heappush(self.lo, -x)
        heapq.heappush(self.hi, -heapq.heappop(self.lo))
        if len(self.hi) > len(self.lo):
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def median(self) -> float:
        if len(self.lo) > len(self.hi):
            return -self.lo[0]
        return (-self.lo[0] + self.hi[0]) / 2

m = StreamingMedian()
for v in orders["amount"].tolist():
    m.add(v)
print("running median:", round(m.median(), 2))

Two-heap pattern shows up in interviews and in real streaming aggregations. Worth memorizing the invariant: len(lo) ∈ {len(hi), len(hi)+1}.

Idempotent ETL: write a daily partition with overwrite-by-date semantics.

from pathlib import Path
import shutil

def publish_daily_fact(df: pd.DataFrame, ds: str, root: str):
    """
    Atomic-ish overwrite for a single date partition.
    df must contain only rows for `ds`.
    """
    assert (df["ds"].astype(str) == ds).all(), "frame contains other dates"
    target = Path(root) / f"ds={ds}"
    tmp    = Path(root) / f".staging_{ds}"
    if tmp.exists(): shutil.rmtree(tmp)
    tmp.mkdir(parents=True)
    df.to_parquet(tmp / "data.parquet", index=False)
    if target.exists(): shutil.rmtree(target)
    tmp.rename(target)        # near-atomic on POSIX FS
