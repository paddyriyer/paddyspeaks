The Stack

The Room · First 90 seconds · scorecard

Four moves. Miss any one and the round drops.

  1. Refused to answer the question as asked. "Which is correct" → "neither; here is what each measures."
  2. Gave a specific time commitment. "45 minutes with a deliverable," not "I need some time."
  3. Ended with a recommendation, not a number. Finance gets a labelled recommendation, not a digit.
  4. Named what you don't know. "Depends on which definition the dashboard is computing" — humility is the signal.

It looks like a system-design round. It's an incident-response round with a design wrapper.

The Landmine · The label is the schema

"Revenue — September" is not one number.

At StreamCo it is four numbers in four systems. Someone labelled a dashboard column Revenue eighteen months ago. A hundred people built mental models on it. The label calcified. The engineering team inherited a political debt nobody spoke about.

The senior fix is not a query. Every dashboard at StreamCo gets a definition footer: which table, which attribution window, which revenue concept, which currency basis. New dashboards without the footer fail code review. Existing dashboards back-filled this quarter.

Rule: if a number has a label, the label is part of the schema.

Where this is going · the spine

Raj's design brief never retracts. Act 2: the reconciliation you just promised. Act 3: the platform that has to prevent this morning from recurring. Act 4: Raj challenges every layer. Act 5: the platform breaks on Black Friday while he watches. Act 6: you debug the system you yourself designed. Act 7: the quiet truth.

Act 2 · What Is Revenue?

Act 2 · Round 1 · 10:03 AM · continuing

Four Numbers, One Word.

Raj is watching. The timer reads 4:42. You have about forty minutes of interview left. He asks for the reconciliation. You have to produce it out loud, in SQL, while narrating the shape of StreamCo's four revenues — because at this scale they do not collapse into one.

You begin. Speaking to Raj, keyboard in your lap, you say:

You: "At StreamCo scale there are four numbers that get called 'revenue' on ad dashboards. They sit in four systems, produced by four different pipelines, and they don't agree because they shouldn't. I'll name all four before I touch the query. Otherwise the query answers the wrong question."

Raj nods. You draw four boxes on the sketch pad you have shared to your screen.

Billed revenue
Owner: Finance
What advertisers have been invoiced for. Produced by the billing system — at StreamCo, the Salesforce ledger. Lags reality by the billing cycle — typically a month. Excludes anything invoicing hasn't swept yet.
Updates: month-end + 3-day settlement.
Spend revenue
Owner: Ads pipeline
Sum of auction-clearing prices × served impressions, in the advertiser's contracted currency, converted to USD at the delivery-time FX. Produced near-real-time by the impression pipeline. Knows about impressions the billing system doesn't yet know about. Doesn't know about refunds or chargebacks.
Updates: streaming, ~5-minute lag.
Attributed revenue
Owner: Ads measurement
Revenue credited to specific campaigns under an attribution model — last-click 7-day, last-touch, MTA linear, whatever the advertiser's contract calls for. Depends on the window, the touch model, and the identity graph. Almost always differs from spend revenue because it excludes unattributable spend and absorbs post-attribution adjustments.
Updates: nightly batch for 28-day; streaming for 24-hour.
Settled revenue
Owner: Finance
The number Finance closes the month on. Post-refund, post-chargeback, post-IVT-recovery credits, post-partner-attribution-lag, post-currency-revaluation. Sourced from the ledger after all adjustments have posted. This is what the 10-Q filing is built from.
Updates: monthly, with quarter-end revisions.

You turn back to Raj and continue:

You: "The CFO's $9.1M is almost certainly settled revenue. The dashboard's $8.2M — without seeing the query, my bet is spend revenue, because that's what the ads pipeline produces natively. Your Turn · SQL · 15 minutes on your timer

Write the reconciliation before reading the answer

The tables you have access to:

  • silver.fact_impression — one row per served impression. Columns: spend_cents, advertiser_id, campaign_id, impression_ts, is_valid (FALSE = flagged as IVT by the pre-bid filter), currency, fx_rate_to_usd.
  • finance.ad_revenue_ledger — one row per (advertiser_id, month). Columns: invoiced_usd, refunds_usd, ivt_credits_usd, partner_attribution_adjustment_usd, fx_revaluation_usd.

Write one query that reconciles September. The output should be one row with all four revenue concepts, the per-category gap, and an unexplained_gap_usd. The last column is the one Raj is actually testing — if it is non-zero, you have a bug in the pipeline, and you need to be able to say so. If it's within rounding, the numbers are all explained by known adjustments, and you close the ticket.

Reference solution · try the prompt above first, then check against this
WITH pipeline AS (
  SELECT
    SUM(spend_cents * fx_rate_to_usd) / 100.0
      AS spend_revenue_usd,
    SUM(CASE WHEN is_valid = FALSE
             THEN spend_cents * fx_rate_to_usd ELSE 0 END) / 100.0
      AS pipeline_ivt_flagged_usd
  FROM silver.fact_impression
  WHERE impression_ts >= '2025-09-01'
    AND impression_ts <  '2025-10-01'
),
ledger AS (
  SELECT
    SUM(invoiced_usd)                        AS billed_revenue_usd,
    SUM(refunds_usd)                         AS finance_refunds_usd,
    SUM(ivt_credits_usd)                     AS finance_ivt_credits_usd,
    SUM(partner_attribution_adjustment_usd)  AS partner_adj_usd,
    SUM(fx_revaluation_usd)                  AS fx_reval_usd,
    SUM(invoiced_usd) - SUM(refunds_usd)
      - SUM(ivt_credits_usd)
      + SUM(partner_attribution_adjustment_usd)
      + SUM(fx_revaluation_usd)              AS settled_revenue_usd
  FROM finance.ad_revenue_ledger
  WHERE month = '2025-09-01'
)
SELECT
  p.spend_revenue_usd,
  l.billed_revenue_usd,
  l.settled_revenue_usd,
  -- per-category components of the gap
  p.pipeline_ivt_flagged_usd,
  l.finance_ivt_credits_usd,
  l.finance_refunds_usd,
  l.partner_adj_usd,
  l.fx_reval_usd,
  -- the unexplained residual
  l.settled_revenue_usd - p.spend_revenue_usd
    + (p.pipeline_ivt_flagged_usd - l.finance_ivt_credits_usd)
    + l.finance_refunds_usd
    - l.partner_adj_usd
    - l.fx_reval_usd                          AS unexplained_gap_usd
FROM pipeline p CROSS JOIN ledger l;

What the query returns today:

  • spend_revenue_usd = $8.21M — matches the dashboard exactly
  • settled_revenue_usd = $9.12M — matches Finance
  • partner_adj_usd = $0.74M — the partner-attribution lag, explained
  • finance_ivt_credits_usd − pipeline_ivt_flagged_usd = $0.16M — Finance's IVT catch exceeds our pre-bid filter, explained
  • fx_reval_usd = $0.02M — end-of-month revaluation, explained
  • unexplained_gap_usd = −$0.01M — within rounding. No pipeline bug.

The $900K gap decomposes cleanly. Neither side is wrong. Finance's number is more complete by construction. Yours is more timely by construction.

The Landmine · Currency math at monthly close

The $20K FX revaluation you almost missed

Your pipeline books revenue at the impression-time FX rate — that is what you charge the advertiser. Finance books the same revenue at the settlement-time FX rate, because that is what actually hits the bank. For most advertisers at most times, the gap is rounding. At quarter-end, when emerging-market currencies move, the gap can be six figures.

If you do not know this and you report unexplained_gap_usd = $20K, Raj assumes the pipeline has a bug. If you do know this and the column you named is fx_reval_usd, Raj knows you have shipped this system before.

Every adjustment line on the ledger exists because a real edge case happened at a real company. A senior engineer's job is to know which lines exist and why — before the reconciliation starts.

Rule: when you reconcile, you name every known adjustment category first. The residual is what's left after all the known ones are accounted for. Never call the gross difference the "gap."

What you say

"The enemy is not latency. It is not cost. It is silent label drift between audiences who each think they are reading the same number. Every layer I draw has one job: make that drift impossible, or make it obvious within five minutes."

Now you draw. Not a wiring diagram — a layer stack. Five layers, top to bottom. Each layer does one thing. Each layer has a contract with the layer above and below. You draw them as stacked rectangles.

5
Ingest · bronze
What ships: Kafka topics per source (impressions, clicks, bids, partner-postbacks, finance-ledger-CDC). Schema registry in front of every producer. Raw append-only landing tables with one row per event, never mutated.
Because of this morning: the affiliate-partner postback stream is a first-class source, not a nightly SFTP. 14-day attribution lag becomes visible at ingest, not at close.
Rejected: single "events" firehose with JSON payloads. Schema-on-read is schema-never.
4
Conform · silver
What ships: canonical fact_impression, fact_click, fact_conversion. Currency normalised to USD at event time. IVT flag materialised here. Identity graph join resolved here, once, with lineage columns.
Because of this morning: IVT is marked at silver with a resolution stamp. When Finance's $0.21M credit arrives two days later we can diff against our $0.05M flag and explain the $160K delta in one query.
Rejected: letting every team join identities themselves. That is how you get four identity graphs and four answers.
3
Metrics · gold
What ships: four revenue tables, named exactly. gold.revenue_spend, gold.revenue_attributed, gold.revenue_billed, gold.revenue_settled. Each has a definition header row, a grain contract, and an owning team.
Because of this morning: there is no table called gold.revenue. That name is reserved and will never be created. Anyone who writes SELECT revenue FROM … gets a query-parser error with a link to the definitions page.
Rejected: a single wide "revenue" table with a revenue_type column. Defaults will be chosen. The default becomes the bug.
2
Semantic · contracts
What ships: a semantic layer (dbt + a metrics API) where every dashboard, notebook, and spreadsheet pulls from named metrics — not from raw tables. Every metric has a YAML spec: source table, grain, definition, owner, audience.
Because of this morning: the CFO's dashboard cannot have a column labelled Revenue without a metric binding. The label is the schema now. Enforced by CI.
Rejected: BI-tool-level definitions (Tableau calculations). They are invisible to engineering, untested, and drift in ninety days.
1
Consume · four audiences
What ships: four purpose-built surfaces. Finance gets settled-basis dashboards + a reconciliation workbench. Marketing gets attributed-basis ROAS. Advertisers get spend-basis self-serve + billing export. Sellers get attributed + pacing.
Because of this morning: no audience ever sees a number unless its audience tag matches the metric's audience tag. Finance cannot accidentally pull a spend-basis number.
Rejected: one unified dashboard with role-based filters. That is how the September bug shipped.

The stack is on the board. Raj has not interrupted. You have four minutes of air, so you spend them on the three contracts that make the stack real. Without them the boxes are furniture.

Contract 1
Append-only attribution
Every conversion row carries two timestamps: event_time (when it happened) and attribution_time (when we credited it). We never UPDATE. Late postbacks arrive as new rows. Historical dashboards are reproducible at any prior attribution_time.
Kills the class of bug where a Tuesday-morning rerun silently rewrites last Thursday's revenue number.
Contract 2
Definition header on every gold table
Row 0 of every gold.* table is a definition row: metric_name, grain, currency_basis, attribution_model, audience, owner_team, ticket_for_changes. Schema migration to change any of these requires a signed change request from the owning team.
Kills the class of bug where a label on a dashboard calcifies into institutional meaning that nobody can challenge.
Contract 3
Identity graph contract test
CI-enforced test: the identity graph version used to produce today's attributed revenue is pinned in the gold table's metadata. If the graph changes, a backfill job runs and emits a new attribution_time cohort — the old numbers are preserved, the new numbers are labelled.
Kills the class of bug where an identity-graph upgrade silently shifts $5M of revenue between campaigns and nobody can explain why a quarter later.

The SQL shape of the append-only attribution table is the single hardest thing to get right, so you write it on the board:

-- gold.revenue_attributed · append-only, bi-temporal
CREATE TABLE gold.revenue_attributed (
  conversion_id         UUID        NOT NULL,
  advertiser_id         BIGINT      NOT NULL,
  campaign_id           BIGINT      NOT NULL,
  event_time            TIMESTAMP   NOT NULL,  -- when the conversion happened
  attribution_time      TIMESTAMP   NOT NULL,  -- when we credited it
  attribution_model     VARCHAR(32) NOT NULL,  -- last_click_7d, mta_linear, ...
  identity_graph_version VARCHAR(16) NOT NULL, -- pinned for reproducibility
  credited_usd          NUMERIC(18,4) NOT NULL,
  is_superseded         BOOLEAN     NOT NULL DEFAULT FALSE,
  PRIMARY KEY (conversion_id, attribution_time)
)
PARTITION BY RANGE (attribution_time);

-- "what did we show Finance on the 10:37 email on 2025-09-30?"
SELECT SUM(credited_usd)
FROM   gold.revenue_attributed
WHERE  event_time >= '2025-09-01' AND event_time < '2025-10-01'
  AND  attribution_time <= '2025-09-30 10:37:00'
  AND  is_superseded = FALSE;

That last query is the one that makes the platform trustworthy. Any number Finance ever saw is reproducible by plugging the attribution_time ceiling into the same query. "What did we tell you, and when?" becomes a SQL answer instead of a Slack archaeology project.

Retention curve

WITH installs AS (
  SELECT install_id, install_ts
  FROM fact_install
  WHERE campaign_id = :c
    AND install_ts BETWEEN :start AND :end
),
opens AS (
  SELECT install_id, DATE(event_ts) AS open_dt
  FROM fact_app_event
  WHERE event_name = 'app_open'
)
SELECT
  DATEDIFF(o.open_dt, DATE(i.install_ts)) AS days_since_install,
  COUNT(DISTINCT o.install_id) * 1.0 /
    (SELECT COUNT(DISTINCT install_id) FROM installs) AS retention_rate
FROM installs i
JOIN opens o USING (install_id)
WHERE o.open_dt BETWEEN DATE(i.install_ts) AND DATE(i.install_ts) + INTERVAL '30' DAY
GROUP BY days_since_install
ORDER BY days_since_install;

Standard curve is D1 ~50%, D7 ~25%, D30 ~15% for a solid app. Less than that, the install was probably a low-quality install from poor audience targeting.

SKAN caveat

SKAdNetwork postbacks are:

  • Delayed (24h minimum, up to 60h).
  • Conversion-value-constrained (6-bit on classic SKAN, 64-bit with SKAN 4).
  • Subject to anonymization thresholds (no postback below cohort-size floor).
  • Non-deterministic at granular grain.

Your SKAN pipeline has to handle all four. Don't mix SKAN-derived metrics with MMP-derived metrics in the same chart without clear labeling.

      GROUP BY campaign_id) s USING (campaign_id)
WHERE f.submitted_ts BETWEEN :start AND :end
GROUP BY campaign_id
ORDER BY ROAS_leadgen DESC;

Key chart: lead funnel + cycle time

Leads submitted:        1,200
Qualified (within 7d):    720 (60%)
Closed won (within 30d):  180 (15%)
Avg cycle time:           18 days
Revenue:              $450,000
Spend:                 $60,000
ROAS:                    7.5x

The "cycle time" dimension matters — you can't declare the campaign a failure after 2 weeks if the average lead closes on day 18.

Section 6: Cross-Objective Benchmark Table

Most senior dashboards include a benchmark row showing where this campaign sits relative to a reference set.

WITH this_campaign AS (
  SELECT SUM(impressions) imps, SUM(clicks) clicks, SUM(spend_usd) spend, SUM(revenue) rev
  FROM metric_campaign_daily
  WHERE campaign_id = :c AND dt BETWEEN :start AND :end
),
advertiser_baseline AS (
  SELECT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY clicks * 1.0 / imps) AS median_ctr,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY rev / spend) AS median_roas
  FROM (
    SELECT campaign_id,
           SUM(impressions) AS imps,
           SUM(clicks) AS clicks,
           SUM(spend_usd) AS spend,
           SUM(revenue) AS rev
    FROM metric_campaign_daily
    WHERE advertiser_id = (SELECT advertiser_id FROM dim.campaign WHERE campaign_id = :c)
      AND dt BETWEEN CURRENT_DATE - 28 AND CURRENT_DATE - 1
    GROUP BY campaign_id
  ) t
  WHERE imps > 10000
),
vertical_baseline AS (
  SELECT
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY clicks * 1.0 / imps) AS median_ctr,
    PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY rev / spend) AS median_roas
  FROM (...vertical-level aggregation...) t
)
SELECT
  tc.clicks * 1.0 / tc.imps AS campaign_ctr,
  ab.median_ctr AS acct_median_ctr,
  (tc.clicks * 1.0 / tc.imps) / NULLIF(ab.median_ctr, 0) - 1.0 AS vs_account_pct,
  vb.median_ctr AS vert_median_ctr,
  (tc.clicks * 1.0 / tc.imps) / NULLIF(vb.median_ctr, 0) - 1.0 AS vs_vertical_pct
FROM this_campaign tc CROSS JOIN advertiser_baseline ab CROSS JOIN vertical_baseline vb;

Benchmarks should always include at least two references — the advertiser's own trailing performance, and a vertical-wide median. One reference is too easy to game.

Section 7: Visualization Anti-Patterns

  • Dual-axis charts. Always confusing. Split into two panels.
  • Mixed-metric stacked bars. "Clicks + impressions" stacked makes no sense.
  • Percentages without denominators. "CVR dropped 30%" — from 3% to 2.1% or from 0.1% to 0.07%? Always show the base.
  • Moving-average without the raw. Hides short-term changes.
  • Color-coded-only diff. 8% of users are color-blind. Always label with a sign or arrow too.
  • Truncated Y-axis. Zoomed axes lie. Start at zero unless there's a reason to not.

Section 8: The One-Pager the Exec Actually Reads

At the most senior level, the scorecard shrinks to one screen with five numbers:

CAMPAIGN HEALTH — <campaign_name>

  Spend:         $54,230 (91% of budget)
  Primary KPI:   ROAS 4.2x  (target 4.0x)  ✓
  Guardrail:     IVT 0.8%   (cap 2.0%)     ✓
  Trajectory:    +12% vs. last week
  Next action:   Increase budget 20%, refresh creative

  🟢 Running well — no intervention needed.

That's it. If the exec wants more, they'll click through. If they don't, you've already told them everything they need. Senior DE mastery is knowing when to stop adding numbers.

Section 9: Measurement Checklist (before a campaign launches)

  • Objective is declared and mapped to exactly one primary KPI.
  • At least two guardrail metrics are declared.
  • Attribution window(s) are pre-committed; no post-hoc changes.
  • Benchmark references (account, vertical) are identified.
  • Dashboard is wired to the metric registry, not a hand-rolled SQL.
  • Incrementality measurement plan exists (holdout, or lift study, or forgo).
  • D+7 lock policy is written down — numbers become immutable on day 7.
  • Advertiser approved the measurement plan in writing.

A campaign without measurement is a spend event. With measurement, it's a learning event. The DE's job is to make sure it's the second one.

Act 7 · The Lesson

If you've been preparing for an ads data-engineering loop by reading about attribution models, you've been preparing wrong. I've watched strong candidates who knew more about MTA than the interviewer lose the loop inside four minutes — not because their knowledge was weak, but because they answered the wrong question with the right words. This part is not a syllabus. It is a catalogue of the specific moments where candidates go silent, and what should have come out of their mouth instead.

Everything below is simulation, not explanation. Read it with a timer. Say the answers out loud. If any of them feel uncomfortable to say, that is the signal that you haven't yet rehearsed them — not that the answer is wrong.

1. The Scar Files

Each of the transcripts below is a specific moment I have seen end a loop. The candidate's answer is not wrong on the surface. The answer is wrong because the interviewer is scoring something the candidate did not know was being tested. I will tell you what was being scored in each case. You will not forget it.

Scar 1 — The "we use Kafka" death

Interviewer: "Walk me through how your current ads pipeline handles impression events."

Candidate: "Sure — we use Kafka for ingest, Flink for real-time, Spark for batch, Iceberg for storage, and Looker for dashboards."

Interviewer (silent for 3 seconds, then): "OK. Anything else?"

Candidate: "Uh… and we have data quality checks in Great Expectations."

Interviewer: "Got it. Let's move on."

Loop score at this point: weak hire, trending no hire. The candidate listed tools. The interviewer was asking them to tell a story. What should have come out of their mouth:

"We carry roughly 800K impressions per second at peak, 300K steady. Kafka is partitioned by advertiser_id to keep hot advertisers' downstream pacing co-located — we learned that the hard way when LinkedIn-scale campaigns knocked over our initial user-id partitioning in April last year. Flink consumes into a pacing KV with 5-second tolerance; batch reconciles it hourly. The painful thing in our stack right now is that Iceberg compaction is bumping against the batch window on quarter-close weeks — I'm six weeks into solving it."

Notice what the second answer does that the first doesn't: it gives numbers, a war story (the LinkedIn-scale campaign), a pain point you've lived with, and a thing you're actively working on. Interviewers score operational ownership, not tool knowledge. This is the single most common way strong candidates look mid-level.

Scar 2 — Attribution, confidently wrong

Interviewer: "How do you think about attribution for a campaign that runs across display and search?"

Candidate: "We use last-click attribution. It's standard."

Interviewer: "And your marketing team is OK with that?"

Candidate: "Yeah, they've always used it."

Interviewer (neutral tone): "I see. Next question."

The candidate doesn't know it, but the loop is over. The interviewer was testing whether the candidate has ever been in the room when this was fought over. The correct answer is not a better attribution model. The correct answer is:

"We default to last-click for operational attribution, but we maintain a parallel MTA lens for budget reallocation decisions. Last-click systematically under-credits upper-funnel — when our marketing director cut display spend 40% last year because last-click showed it was unprofitable, search ROAS fell 30% within six weeks, because search had been catching users that display had primed. We restored display. Now we show both lenses on the same dashboard, labeled by what decision each is fit for. I'd never recommend a team unify on one attribution number, because the number and the decision are coupled."

What's being scored: whether the candidate has been inside the consequence of last-click. The tell is the six-week lag — you cannot fabricate that specific detail unless you lived it.

Scar 3 — The "exactly once" lie

Interviewer: "How do you guarantee impression counts are exact?"

Candidate: "We use Kafka's exactly-once semantics and Flink's checkpointing, so we're exactly-once end to end."

Interviewer: "OK. What about the SDK?"

Candidate: "…it sends to Kafka."

Interviewer: "Right. And when the user's phone loses network, the SDK buffers and retries?"

Candidate: "…yes."

Interviewer: "So you're not exactly-once end to end."

The candidate never recovers. What they should have said:

"Exactly-once is a useful guarantee within the Kafka-to-sink path, but the SDK side is at-least-once by design — retries on network drop create duplicates we have to catch at ingest. We run a triple-key dedupe on (user_id, event_name, event_ts rounded to the second) with a 24-hour reconciliation window against a suspected-duplicates sidecar. We accept a 0.1% false-positive loss rate in exchange for ~0.8% duplicate catch. Naming that budget in the contract with the advertiser team was the hardest conversation of that project."

The signal: senior engineers know where "exactly-once" ends. They know it ends at the SDK. Candidates who claim end-to-end exactly-once reveal they have never been paged at 3 a.m. about a 2% revenue overcount.

Scar 4 — The system design that doesn't scope

Interviewer: "Design a pipeline for a new ads product that reports per-creative performance."

Candidate (starts drawing): "OK so we have Kafka here, events go into bronze, then we have a silver layer…"

Interviewer (four minutes in): "Who consumes this?"

Candidate: "…the advertiser."

Interviewer: "How often?"

Candidate: "Daily?"

Interviewer: "If it's daily, why did we just spend four minutes on a real-time ingestion layer?"

I have watched this exchange kill three candidates this year. The cure is uncomfortably simple: do not draw a single box until you have asked who reads it and how often. The first 60 seconds of a system-design answer should be clarifying questions, not architecture.

Scar 5 — The metric that didn't move

Interviewer: "Tell me about a project you shipped."

Candidate: "I built the new impression deduplication pipeline."

Interviewer: "Great — what did it change?"

Candidate: "It's faster and more accurate."

Interviewer: "By how much? And what downstream behavior changed because it was more accurate?"

Candidate: "…I'd have to check."

If you do not know what your project changed in dollars, hours, or headcount, you shipped a feature, not an outcome. Senior candidates walk in with three numbers for every project: the number that moved, the number that stayed flat (to show you knew what to watch), and the number you'd still want to improve. "I'd have to check" closes the loop immediately.

Scar 6 — The monitoring vacuum

Interviewer: "How do you know your pipeline is healthy?"

Candidate: "We have alerts on job failures."

Interviewer: "What about when the job succeeds but the data is wrong?"

Candidate: "We'd catch that in the downstream dashboards eventually."

Interviewer (written in the scorecard afterwards): "No data-quality discipline. Weak on production ownership."

"Eventually" is the word that killed this loop. The correct answer distinguishes pipeline liveness (job finished) from data correctness (output is what downstream expected), names the specific checks that run on every successful completion (row count floors, null rate ceilings, schema stability, freshness), and describes how violations route — blocking downstream until triage vs warning to a channel the owner watches.

Scar 7 — The "it depends" that wasn't followed up

Interviewer: "Should we denormalize this schema for the new reporting layer?"

Candidate: "It depends."

(Silence.)

Interviewer: "On?"

Candidate: "The tradeoffs."

This candidate had the knowledge. They had built denormalized reporting tables. They lost the loop because they did not name the axes the decision depends on. "It depends" without what-it-depends-on is indistinguishable from "I don't know." The cure, drilled to reflex: "It depends on X, Y, and Z. If X then denormalize because of tradeoff T. If not, keep normalized because of tradeoff U. I'd lean toward denormalize unless [specific condition], because in my experience the reversibility cost of denormalization is low and the query-performance cost of over-normalization is high."

Scar 8 — The candidate who was too junior by three words

Interviewer: "How would you handle a schema change in the event stream?"

Candidate: "I'd ask my manager."

The candidate is a competent senior engineer. But they just signaled to the interviewer that the decision does not feel like theirs to make. Staff+ roles are scored on whether you own the decision, not whether you escalate it. The answer the interviewer needed to hear: "I'd flag it to my manager as context, and I'd bring a proposed path — versioned schema with a 30-day producer notice, backward-compatible field addition, dual-write during transition, consumer migration gated by a coverage metric. If the manager has a better path, great. If not, I execute." Owning the proposal is the signal.

2. This Sounds Smart But Gets Rejected

Every bullet below is an answer I have heard in an interview loop that sounded competent enough that the candidate sat down thinking they did well. In every case the loop was scored down. Read each one with the assumption that your first instinct was probably the answer the candidate gave.

"We'd use Kafka for exactly-once delivery."

Gets rejected because: Kafka exactly-once is a guarantee within Kafka, not across the SDK-to-sink path. The interviewer is probing for the candidate's understanding of where the guarantee breaks. The right answer names the breakpoints (SDK retries, consumer side effects outside the transaction, external API calls from the processor) before it names the feature.

"We run data-quality checks in Great Expectations."

Gets rejected because: it is a tool, not a discipline. The interviewer wants to know which specific checks run on which specific tables, how thresholds are set (learned or hand-configured), who is paged on violation, and what the ratio of false-positive pages is. If you cannot answer those four questions, "we use Great Expectations" reads as "we installed Great Expectations."

"I'd partition by user_id for better parallelism."

Gets rejected because: it reveals the candidate has not thought about cardinality explosion. user_id as a partition key in a file-based store produces tens of millions of files over a retention window, and the metadata overhead kills every engine that reads it. The right instinct is partition by date plus bucketed user_id.

"We use medallion architecture: bronze, silver, gold."

Gets rejected because: it is vocabulary. Any candidate can say this. The interviewer wants the rule for what earns a silver table, the criterion for promotion to gold, and a specific example of a row that lives in silver but not in gold (with reason). Without those specifics, "medallion" is a word, not an answer.

"Let's put a cache in front of it."

Gets rejected because: caching is the lazy answer to read-heavy performance. The interviewer is scoring whether the candidate thought about invalidation. A cache in front of an ads aggregation without invalidation will serve stale bid-eligibility decisions and cost real money. Naming the invalidation policy (TTL, event-driven, versioned key) is the senior signal.

"We do idempotent writes using upserts."

Gets rejected because: idempotent writes are easy; idempotent pipelines are hard. The interviewer probes: what about the side effects — emails sent, counters incremented, webhooks fired? Upserts handle the table. Pipelines need end-to-end idempotency, which often requires an explicit processed-events ledger the pipeline checks before acting.

"Schema-on-read gives us flexibility."

Gets rejected because: flexibility is engineer-speak for "we punted the decision downstream." The interviewer is checking whether the candidate has felt the pain of five downstream consumers each implementing a slightly different schema interpretation of the same raw events. The senior answer says: schema-on-read at bronze; enforced typed contract at bronze-to-silver; if you do it the other way you pay for it for years.

"We'd scale horizontally."

Gets rejected because: it is the answer that means "I haven't thought about where the bottleneck is." The interviewer is looking for whether the candidate knows which specific component is constrained: the producer, the broker, the consumer, the sink, or (most often in ads pipelines) the shuffle in the downstream aggregation. Horizontal scaling the wrong component moves the bottleneck without solving it.

"We use Parquet because it's columnar."

Gets rejected because: columnar is table stakes. The interviewer wants dictionary encoding, predicate pushdown, min-max statistics, bloom-filter usage, and row-group sizing decisions. If the candidate says "Parquet is columnar" and stops, they are a consumer of formats, not an engineer who has tuned one.

"We'd denormalize for query speed."

Gets rejected because: denormalization is a tool, not a default. The interviewer is checking whether the candidate understands the asymmetry: denormalization is easy to add, extremely expensive to remove. Senior candidates default to keeping the grain in silver and denormalizing only in gold, where the denormalization is attached to a specific consumer-facing view.

"We use Airflow."

Gets rejected because: "we use Airflow" says nothing about how your team treats Airflow. Who owns DAG code review? How do you test a DAG before production? How do you handle cross-DAG dependencies? How do you prevent DAG sprawl at 500+ DAGs? If the candidate has answers to those, they are senior. If not, they are downstream of someone who set up Airflow for them.

"I would A/B test it."

Gets rejected because: A/B testing for DE changes (not product changes) is a specific discipline. The interviewer probes: how do you split traffic? Do you backfill historical data through both pipelines for comparison? What's your minimum detectable effect, and how did you calculate sample size? "I'd A/B test" without these answers reads as a phrase the candidate learned without running one.

3. The Ads Landmines — Where the Consequences Live

These are not topics. They are the specific moments when an ads pipeline hurts you financially, loudly, and publicly. Every landmine below has taken someone down in a loop because they could not name the consequence — they could only name the concept.

Landmine 1 — Last-click paid for itself until it didn't

The landmine: leadership sees that paid-search is credited with 68% of last-click conversions and concludes that display is not working. They cut display 40%. Paid-search ROAS drops 30% six weeks later. Revenue craters for one quarter before the team reverses.

The query that shipped the wrong decision:

-- Naive last-click attribution. Correct, and dangerous.
SELECT
  channel,
  COUNT(DISTINCT conversion_id)       AS conversions,
  SUM(revenue_usd)                    AS attributed_revenue
FROM fact_conversion c
JOIN fact_touch t
  ON t.user_id = c.user_id
 AND t.touch_ts <= c.conversion_ts
 AND t.touch_ts >= c.conversion_ts - INTERVAL '7 days'
QUALIFY ROW_NUMBER() OVER (
  PARTITION BY c.conversion_id
  ORDER BY t.touch_ts DESC
) = 1
GROUP BY channel;

Run this, paste the output into a slide, "display is 4% of conversions." Display gets cut. Six weeks later paid-search collapses because display was priming its funnel.

The senior move — ship two lenses, labeled by decision:

-- Parallel MTA lens on the same event log, linear weighting.
-- Same query can swap 'linear' for 'time_decay' or 'position_based'.
WITH touch_counts AS (
  SELECT c.conversion_id, c.revenue_usd, COUNT(*) AS n_touches
  FROM fact_conversion c
  JOIN fact_touch t
    ON t.user_id = c.user_id
   AND t.touch_ts <= c.conversion_ts
   AND t.touch_ts >= c.conversion_ts - INTERVAL '30 days'
  GROUP BY c.conversion_id, c.revenue_usd
)
SELECT
  t.channel,
  SUM(c.revenue_usd / tc.n_touches) AS mta_linear_revenue
FROM fact_conversion c
JOIN fact_touch t
  ON t.user_id = c.user_id
 AND t.touch_ts <= c.conversion_ts
 AND t.touch_ts >= c.conversion_ts - INTERVAL '30 days'
JOIN touch_counts tc USING (conversion_id)
GROUP BY t.channel;

Now the dashboard shows two columns — "last-click revenue" and "MTA linear revenue" — labeled by the decision each is fit for. Budget-reallocation conversations use MTA. Operational-dashboard comparisons use last-click. Never a single number for a multi-channel decision.

What the interviewer is testing: whether you understand that attribution numbers are not neutral data. They are policy decisions wearing a number's clothes.

Landmine 2 — Conversions that arrive tomorrow, attributed to yesterday

The landmine: your pipeline closes yesterday's books at 6 a.m. today. A conversion arrives at 10 a.m. today, clicked yesterday. Rewriting yesterday's numbers silently drifts every dashboard that consumed them. Finance loses confidence within two quarters.

The bad pattern — retroactive UPDATE:

-- DO NOT. Mutates closed books; breaks every downstream that cached yesterday.
UPDATE fact_campaign_daily
SET conversions   = conversions   + 1,
    revenue_usd   = revenue_usd   + :new_revenue
WHERE dt = :click_date
  AND campaign_id = :campaign_id;

The senior pattern — append-only with two clocks:

-- Every conversion carries BOTH timestamps. History never mutates.
CREATE TABLE fact_conversion_attributed (
  conversion_id    BINARY(16),
  campaign_id      BIGINT,
  event_time       TIMESTAMP,   -- when it happened in the world (click date)
  attribution_time TIMESTAMP,   -- when we credited it in our books
  revenue_usd      DECIMAL(18,4),
  PRIMARY KEY (conversion_id)
)
PARTITIONED BY (DATE(attribution_time));

-- Insert-only. Late-arriving conversions land in today's partition
-- with event_time = yesterday, attribution_time = today.
INSERT INTO fact_conversion_attributed
SELECT
  conversion_id,
  campaign_id,
  event_ts                         AS event_time,
  CURRENT_TIMESTAMP                AS attribution_time,
  revenue_usd
FROM stg_conversion_arrivals
WHERE arrival_batch_id = :batch_id;

Two reporting views, one immutable source:

-- "Revenue as reported on each day" — what finance sees at close.
-- Filter on attribution_time = reporting_date. Historical rows NEVER change.
SELECT DATE(attribution_time) AS reporting_dt,
       campaign_id,
       SUM(revenue_usd) AS revenue_as_reported
FROM fact_conversion_attributed
WHERE attribution_time >= :from AND attribution_time < :to
GROUP BY 1, 2;

-- "Revenue as we now understand it" — what BI wants for cohort math.
-- Filter on event_time = cohort date. Rolls up late-arriving conversions.
SELECT DATE(event_time) AS cohort_dt,
       campaign_id,
       SUM(revenue_usd) AS revenue_as_now_known
FROM fact_conversion_attributed
WHERE event_time >= :from AND event_time < :to
GROUP BY 1, 2;

What the interviewer is testing: whether you know that finance defaults to reporting-time and BI defaults to event-time, and that the data layer serves both by never mutating history. Candidates who propose rerunning yesterday reveal they have never been in a finance-close conversation.

Landmine 3 — The duplicate that grew revenue by 3% for two weeks

The landmine: a SDK update inadvertently retries on transient server errors without marking retries as duplicates. Revenue grows 3% for two weeks. Legal calls. The advertiser asks for a refund.

The bad pattern — trust event_id:

-- DO NOT. event_id is unstable across SDK versions and server CAPI.
-- The dedupe misses retries where event_id differs by a field.
SELECT COUNT(DISTINCT event_id) AS events
FROM bronze.impression
WHERE dt = :yesterday;

The senior pattern — triple-key dedupe plus sidecar:

-- Dedupe on a composite key that survives SDK retries.
-- (user_id, event_name, event_ts_rounded_sec) catches retries even when
-- event_id drifts. 0.1% false-positive rate is the negotiated budget.
CREATE OR REPLACE TABLE silver.fact_impression AS
WITH keyed AS (
  SELECT
    *,
    CONCAT_WS('|',
      CAST(user_id AS STRING),
      event_name,
      CAST(DATE_TRUNC('second', event_ts) AS STRING)
    ) AS dedupe_key
  FROM bronze.impression
  WHERE dt = :yesterday
)
SELECT *
FROM keyed
QUALIFY ROW_NUMBER() OVER (
  PARTITION BY dedupe_key
  ORDER BY ingest_ts ASC  -- keep first-seen row; others go to sidecar
) = 1;

-- Everything we dropped goes here, auditable by the analytics team.
CREATE OR REPLACE TABLE ops.suspected_duplicates AS
SELECT *
FROM keyed
QUALIFY ROW_NUMBER() OVER (
  PARTITION BY dedupe_key ORDER BY ingest_ts ASC
) > 1;

The audit query that catches the SDK bug next time:

-- Monitor: dedupe-drop rate per SDK version. If any version spikes
-- above 0.5%, page the producer team — it's usually a retry bug.
SELECT
  sdk_version,
  COUNT(*)                          AS dropped,
  COUNT(*) * 1.0 /
    (SELECT COUNT(*) FROM bronze.impression
     WHERE dt = :yesterday AND sdk_version = s.sdk_version) AS drop_rate
FROM ops.suspected_duplicates s
WHERE dt = :yesterday
GROUP BY sdk_version
HAVING drop_rate > 0.005;

What the interviewer is testing: whether you know that duplicates are an audit problem, not a theoretical one. The senior move is a suspected-duplicate sidecar and a reconciliation budget negotiated with the advertiser team — so the 0.1% false-positive rate is a disclosed acceptable number rather than a surprise.

Landmine 4 — Identity stitching that dropped 12% of conversions overnight

The landmine: the identity team updates its graph with stricter confidence thresholds to address a privacy audit. Your pipeline fails to stitch 12% of cross-device conversions. Advertiser dashboards show a 12% revenue drop that is not a revenue drop — it is a measurement drop.

The query that drops the 12% silently:

-- Inner-joining to the identity graph silently drops any user whose
-- cross-device edges no longer meet the new confidence threshold.
SELECT
  idg.identity_id,
  COUNT(DISTINCT c.conversion_id) AS conversions
FROM fact_conversion c
JOIN dim.identity_graph idg
  ON idg.user_id = c.user_id
 AND idg.confidence >= 0.8   -- threshold moved from 0.6 -> 0.8 overnight
GROUP BY idg.identity_id;

The senior pattern — contract test before consuming the graph:

-- Run BEFORE every production pipeline that joins to the identity graph.
-- If coverage drops >2% vs the 7-day baseline, fail the pipeline and page.
WITH today AS (
  SELECT COUNT(DISTINCT user_id) AS covered_users
  FROM dim.identity_graph
  WHERE graph_version = :today_version
    AND confidence >= 0.8
),
baseline AS (
  SELECT AVG(covered_users) AS avg_covered
  FROM ops.identity_coverage_history
  WHERE dt BETWEEN CURRENT_DATE - 7 AND CURRENT_DATE - 1
),
check_result AS (
  SELECT
    t.covered_users,
    b.avg_covered,
    (t.covered_users - b.avg_covered) / b.avg_covered AS drift
  FROM today t CROSS JOIN baseline b
)
SELECT
  CASE
    WHEN drift < -0.02 THEN 'FAIL: identity coverage down ' || ROUND(drift*100,2) || '%'
    ELSE 'PASS'
  END AS gate_status,
  *
FROM check_result;

Left-join + NULL-aware, so measurement drops are visible, not silent:

-- Conversions that could not be stitched still appear in output with
-- identity_id = NULL. Dashboard can show "12% of conversions unstitched"
-- as a first-class metric, not hide them behind an inner-join.
SELECT
  COALESCE(idg.identity_id, 'UNSTITCHED') AS identity_bucket,
  COUNT(DISTINCT c.conversion_id)         AS conversions
FROM fact_conversion c
LEFT JOIN dim.identity_graph idg
  ON idg.user_id = c.user_id
 AND idg.confidence >= 0.8
GROUP BY 1;

What the interviewer is testing: whether you treat identity as an upstream dependency your pipeline does not own, and whether you have contract tests that catch graph changes before production reads them. Without a contract, every identity update is a potential incident.

Landmine 5 — The pacing counter that ran 90 seconds behind during Black Friday

The landmine: Kafka lag on peak. The real-time pacing counter is 90 seconds stale. The bidder thinks the advertiser has $400 left when they actually have $50 left. 9,000 more auctions get won. The advertiser over-spends by $400. Multiply by every advertiser affected.

The bad pattern — unconditional bidder read:

# DO NOT. Reads pacing counter without checking freshness.
# During lag, happily continues bidding on stale spend estimate.
def should_bid(campaign_id, bid_amount):
    spent_today = redis.get(f"pacing:{campaign_id}:spent_today") or 0
    daily_budget = campaign.daily_budget
    return (spent_today + bid_amount) < daily_budget

The senior pattern — freshness-aware bidder with automatic degradation:

# Bidder reads both the counter AND its last-update timestamp.
# When lag exceeds the budget, it doesn't stop bidding — it derates.

def should_bid(campaign_id, bid_amount):
    spent_today, updated_at = redis.mget(
        f"pacing:{campaign_id}:spent_today",
        f"pacing:{campaign_id}:updated_at",
    )
    spent_today  = float(spent_today or 0)
    lag_seconds  = time.time() - float(updated_at or 0)
    daily_budget = campaign.daily_budget

    # Fresh path: normal bidding against live counter
    if lag_seconds < 30:
        return (spent_today + bid_amount) < daily_budget

    # Degraded path: cap effective budget at 80% to absorb uncertainty
    # This is policy, written in the design doc, not an incident fix.
    if lag_seconds < 300:
        return (spent_today + bid_amount) < (daily_budget * 0.80)

    # Catastrophic path: unknown spend. Refuse rather than risk 10x overspend.
    metrics.increment("pacing.stale.bid_refused", tags=[f"campaign:{campaign_id}"])
    return False

The hourly reconciliation that catches KV drift:

-- Source of truth: impressions joined to spend, aggregated from lake.
-- Runs hourly; writes corrections to Redis if drift exceeds 5%.
WITH truth AS (
  SELECT
    campaign_id,
    SUM(spend_cents) / 100.0 AS truth_spent_today
  FROM silver.fact_impression
  WHERE dt = CURRENT_DATE
    AND is_valid = TRUE
  GROUP BY campaign_id
),
kv_reported AS (
  SELECT campaign_id, kv_spent_today
  FROM ops.pacing_kv_snapshot  -- Redis snapshot exported hourly
  WHERE snapshot_ts = (SELECT MAX(snapshot_ts) FROM ops.pacing_kv_snapshot)
)
SELECT
  t.campaign_id,
  t.truth_spent_today,
  k.kv_spent_today,
  ABS(t.truth_spent_today - k.kv_spent_today) /
    NULLIF(t.truth_spent_today, 0) AS drift_pct
FROM truth t LEFT JOIN kv_reported k USING (campaign_id)
WHERE ABS(t.truth_spent_today - k.kv_spent_today) /
      NULLIF(t.truth_spent_today, 0) > 0.05;
-- For each row, the reconciler writes the corrected value back to Redis.

Landmine 7 — The iOS 14 revenue cliff nobody modeled

First, a plain-English primer — because most candidates fumble this terminology in interviews.

App Tracking Transparency (ATT) is Apple's privacy framework introduced in iOS 14.5 (April 2021). When an app wants to track a user across other apps and websites, Apple requires it to show a system dialog asking for permission. Two buttons: "Allow Tracking" or "Ask App Not to Track." The user picks.

IDFA (Identifier for Advertisers) is the device-level ID that ads pipelines had always joined on to answer "did the person who clicked this ad also buy this thing." Before ATT, apps got the IDFA by default. After ATT, if the user taps "Ask App Not to Track," the IDFA the app reads is all-zeros — useless for joining. Industry-wide opt-in rates settled at roughly 15–30% depending on vertical. Meaning 70–85% of your iOS traffic lost its deterministic join key overnight.

SKAdNetwork (SKAN) is Apple's privacy-preserving replacement. The advertiser doesn't see the user; instead, Apple sends a postback to the ad network 24–72 hours after an attributed install, containing a campaign ID and a 6-bit "conversion value" (64 possible buckets, advertiser-defined). No user identifier. If too few conversions happen in a time bucket, Apple further blanks out the source publisher ID — Apple calls this "privacy thresholding." SKAN and your server-side conversion pipeline measure different populations with different methodologies. Reconciling them produces spurious "missing" conversions. The correct design is dual-reporting, not reconciliation.

What the landmine actually cost teams: iOS attribution numbers fell 60%+ overnight in April 2021 for anyone who had inner-joined on IDFA. Advertisers saw "their iOS campaigns stopped working," paused budget, and called their account managers. Two quarters to rebuild the measurement stack around SKAN and explain the drop wasn't a drop in revenue — it was a drop in measurability.

The bad pattern — single pipeline assuming deterministic IDs:

FROM fact_conversion c

 AND i.ts  <= c.conversion_ts
GROUP BY campaign_id;

4. Thirty-Second Drills — Forced-Choice Tradeoffs

For each drill below, start a timer. Answer out loud. If you pause more than ten seconds before speaking, you have not rehearsed it enough. If you do not say one of the named sentences below, you have said the wrong thing.

Drill 1 — The 4 TB query

Your query is correct but scans 4 TB. The interviewer is watching what you do next.

What the junior says: "I'd add a partition filter." Why it fails: you jumped to optimization before asking what the query is for. A daily analyst query at 4 TB is fine; a 5-minute dashboard query at 4 TB is a six-figure monthly bill. The interviewer is scoring whether you distinguish between the two.

What the senior says: "Two questions. Who runs this, and how often? If it is an analyst's one-off, 4 TB is acceptable. If it is a dashboard hitting it every 5 minutes, we materialize the aggregate in gold, not tune the scan. Which case are we in?"

Drill 2 — SLA or correctness

Your pipeline is late. You can ship stale-by-two-hours at 6 a.m., or you can slip to 8 a.m. and include everything.

What the junior says: picks one. Why it fails: there is no engineer's answer — this is a stakeholder call.

What the senior says: "If it is the CFO's revenue number, I am not making this call — I am paging the VP of Finance with both options and a recommended one in 10 minutes. If it is a team standup dashboard, I ship at 6 a.m. with a banner that says 'incomplete — reconciled at 8 a.m.' The axis is who is paying for the miss."

Drill 3 — Facts as events or aggregated

You are designing a click-stream schema. Thirty seconds: facts as events, or facts as aggregated impressions?

What the senior says: "Events at bronze, always — the grain is irreversible going up and cheap going down. At silver: events unless we have no impression-level consumer, in which case daily aggregates. I ask what the hardest downstream query is; if any of them needs per-impression, the answer is events. Aggregation is an optimization; un-aggregation is a backfill."

Drill 4 — Push back on a staff engineer's "we don't need Iceberg"

What the senior says: "You are right that a new format has real cost. I am not proposing Iceberg because it is new — I am proposing it because of a specific pain we have lived through: we lost a day last quarter rebuilding a silver table after a schema change. If we have two more of those this year at eight engineer-days each, Iceberg's migration cost pays back in a year. If you think we will not have those incidents, I should not propose it. Can we look at the last four quarters of schema-change incidents together?"

Drill 5 — Kafka lag on Black Friday

What the senior says: "None of the three options in isolation. I route pacing consumption to a pre-provisioned stand-by cluster immediately — a stale pacing signal is the only thing costing money right now. I accept dashboard lag up to 5 minutes silently, page the dashboard owner at 10. I start the broker-cluster scale-up in parallel with a 30-minute mitigation horizon. Dropping data is the last resort and requires an executive decision, because the lost rows may be billable."

Drill 6 — The conversion 7 days late

What the senior says: "For which attribution window? Same conversion, four answers. SKAN: out of window, not counted. Paid search 7-day click: counted. MMP 28-day click: counted. Finance billing for a closed month: counted in the current month's revenue, not retroactively. The engineer's job is not to pick — it is to preserve event_time so every consumer applies its own policy."

Drill 7 — The numbers that don't match

What the senior says: "Three possibilities, ranked: timezone, refund treatment, attribution-window drift. I reconcile by asking for the exact filters of both sides, not the queries. Both are usually correct for their purpose. The fix is almost always a labeled definition on each dashboard, not a query change."

Drill 8 — The vendor pitch

What the senior says: "Shadow mode 60 days. Daily divergence check against our pipeline. Kill criteria written before we sign: if divergence exceeds 2% in any week, or operational cost exceeds in-house, we walk. That reframe usually reveals that sales loves the vendor because of ownership anxiety, not the technology."

Drill 9 — The identity merge

What the senior says: "Two-track. Forward-only for attribution; sidecar ledger for historical reconciliation. Billing tables are immutable after monthly close. Identity changes never rewrite published revenue. Anyone proposing retroactive rewrites does not know what quarterly close looks like."

Drill 10 — The benchmark-resistant peer

What the senior says: "I don't know whether Postgres hits 500K/sec for our shape until we measure. Three engineer-days for a benchmark beats six months of migration on a hunch. Same schema, same partition key, same class of hardware. I'll have the numbers Friday."

Drill 11 — Privacy vs fraud

What the senior says: "Hashed IP with rotating salt, 7-day retention aligned to the fraud team's detection window. Privacy team gets minimization. Fraud team keeps 40% of their signal. The design decision to escalate: is there any case we absolutely need raw IP for, and if so, can we run that case in an enclave with tighter governance?"

Drill 12 — The 10x scale question

What the senior says: "At 10x, compaction breaks before Kafka or Spark — our file count per partition goes from 10K to 100K and the rewriter collides with the next day's load. Storage egress costs scale linearly and become the single largest line item. At 100x, the partition scheme itself has to change — hourly plus bucketed sub-partitioning. Kafka and Spark scale with configuration; the scheduling graph and the cloud bill do not."

5. The Interviewer-Pushback Playbook

When the interviewer pushes back, they are not saying you are wrong. They are saying "prove you can hold this under pressure." The weak move is to capitulate. The senior move is to restate with more specificity.

"That's not how we do it here."

"Fair. What I gave you is from my previous stack. What is different in yours, so I can tell you whether my answer changes?"

"What if we had 100x the volume?"

"At 10x, compaction breaks. At 100x, the partition scheme does. But the real decision is whether we plan the migration in advance or emergency-handle it later, because the migration is 3–6 months of work. What is your growth trajectory, so I can tell you whether I am designing for 10x or 100x?"

"Isn't that over-engineered?"

"You might be right. Let me name the specific pain I was solving — [pain]. If we don't have that pain, I simplify to [simpler design]. Which side are we on?"

"Why not just use a vendor?"

"Buy if three conditions hold: accuracy within 2% of ours, total cost under 18 months of engineering, no existential dependency that would make the vendor a single point of failure. If any fails, we build. I've seen teams default in either direction and regret it."

"Convince me."

"Three reasons, ranked. One — [strongest, addresses biggest risk]. Two — [second]. Three — [weakest, named as weakest]. If you kill it, my concern is [specific downside]. If you approve it, my concern is [specific downside]. I'm making a case for one path with eyes on both costs."

"What would you regret?"

"The decision I'd regret most is [specific choice]. It was reversible but had ongoing ops cost I may have under-weighted. Escape hatch: keep schema compatible with [alternative] so we can migrate in 6 months if it bites."

The silence after your answer

Junior candidates fill silence with more content, often undermining a good answer. Senior candidates wait five seconds and then say: "Did that answer what you asked, or did I miss the actual question?" This reframe preserves the good answer and surfaces what was actually being tested. Interviewers almost always tell you.

If you walked into this part thinking the ads interview was a knowledge test, you now know it is not. It is a pressure test where knowledge is the surface layer and judgment is what gets scored. The candidates who pass have rehearsed not concepts but responses — the pause, the question back, the named tradeoff, the admitted regret.

1. SQL — Eight Tricky Problems

SQL 1 — Last-click attribution with mixed click and view windows

Prompt

For each conversion, attribute it to the last touch that satisfies either (a) a click within 7 days before the conversion, or (b) a view within 1 day before the conversion. Click wins over view on ties. Output: conversion_id, attributed_touch_id, attribution_type, campaign_id.

Schema

conversions(conversion_id BIGINT, user_id BIGINT, conversion_ts TIMESTAMP, revenue_usd DECIMAL)
touches(touch_id BIGINT, user_id BIGINT, campaign_id BIGINT, touch_ts TIMESTAMP, touch_type STRING)  -- 'click' or 'view'

The trap

Candidates join once with a combined window and take the last row. This is wrong when a user has a recent view and an older click — last-touch-by-time gives the view, but click should win. You need a preference order inside the window filter.

Reference solution

WITH candidates AS (
  SELECT
    c.conversion_id,
    c.revenue_usd,
    t.touch_id,
    t.campaign_id,
    t.touch_type,
    t.touch_ts,
    ROW_NUMBER() OVER (
      PARTITION BY c.conversion_id
      ORDER BY
        CASE WHEN t.touch_type = 'click' THEN 0 ELSE 1 END,  -- click beats view
        t.touch_ts DESC                                        -- then most recent
    ) AS rn
  FROM conversions c
  JOIN touches t
    ON t.user_id = c.user_id
   AND (
        (t.touch_type = 'click' AND t.touch_ts >= c.conversion_ts - INTERVAL '7 days'
                                 AND t.touch_ts <= c.conversion_ts)
     OR (t.touch_type = 'view'  AND t.touch_ts >= c.conversion_ts - INTERVAL '1 day'
                                 AND t.touch_ts <= c.conversion_ts)
       )
)
SELECT
  conversion_id,
  touch_id AS attributed_touch_id,
  touch_type AS attribution_type,
  campaign_id
FROM candidates
WHERE rn = 1;

Follow-ups the interviewer probes

  • "What if a conversion has no qualifying touch?" → add a LEFT JOIN and COALESCE to 'organic'.
  • "What if click and view from different campaigns?" → the ORDER BY already breaks the tie by type + time; campaign is intentionally not part of the tiebreaker.

SQL 2 — Frequency capping: who saw this creative more than N times in a rolling 24h

Prompt

For each (user_id, creative_id) pair, identify any rolling 24-hour window in the past 30 days during which they received >10 impressions. Output users and creatives with the count of windows that breached the cap.

Schema

impressions(impression_id BIGINT, user_id BIGINT, creative_id BIGINT, imp_ts TIMESTAMP)

The trap

"Rolling 24h" is not the same as "per calendar day." A candidate who groups by DATE(imp_ts) misses windows that span midnight. The correct pattern uses COUNT(*) OVER with a range frame, not a row frame.

Reference solution

WITH windowed AS (
  SELECT
    user_id,
    creative_id,
    imp_ts,
    COUNT(*) OVER (
      PARTITION BY user_id, creative_id
      ORDER BY EXTRACT(EPOCH FROM imp_ts)
      RANGE BETWEEN 86400 PRECEDING AND CURRENT ROW  -- 24h rolling
    ) AS imps_in_24h
  FROM impressions
  WHERE imp_ts >= CURRENT_DATE - INTERVAL '30 days'
)
SELECT
  user_id,
  creative_id,
  SUM(CASE WHEN imps_in_24h > 10 THEN 1 ELSE 0 END) AS windows_over_cap,
  MAX(imps_in_24h)                                  AS peak_count
FROM windowed
GROUP BY user_id, creative_id
HAVING SUM(CASE WHEN imps_in_24h > 10 THEN 1 ELSE 0 END) > 0
ORDER BY peak_count DESC;

Follow-ups

  • "Some engines don't support RANGE with time intervals." → materialize epoch seconds in a CTE and range over the integer, or self-join with BETWEEN imp_ts - 24h AND imp_ts (slower but universal).
  • "What about viewable impressions only?" → filter on is_viewable = TRUE upstream.

SQL 3 — Multi-touch attribution with time-decay weights

Prompt

Attribute each conversion's revenue across the user's touches in the past 30 days using time-decay: a touch N days before conversion gets weight 2^(-N/7) (half-life of 7 days). Normalize weights per conversion so they sum to 1. Output per-campaign attributed revenue.

Schema

conversions(conversion_id BIGINT, user_id BIGINT, conversion_ts TIMESTAMP, revenue_usd DECIMAL)
touches(touch_id BIGINT, user_id BIGINT, campaign_id BIGINT, touch_ts TIMESTAMP)

The trap

Normalizing per touch instead of per conversion — produces wrong totals. And computing EXP(LN(2) * ...) differently per engine. The clean approach uses a windowed SUM for the denominator.

Reference solution

WITH weighted AS (
  SELECT
    c.conversion_id,
    c.revenue_usd,
    t.campaign_id,
    POWER(2.0, -1.0 * DATEDIFF('day', t.touch_ts, c.conversion_ts) / 7.0) AS raw_weight
  FROM conversions c
  JOIN touches t
    ON t.user_id = c.user_id
   AND t.touch_ts BETWEEN c.conversion_ts - INTERVAL '30 days' AND c.conversion_ts
),
normalized AS (
  SELECT
    conversion_id,
    revenue_usd,
    campaign_id,
    raw_weight,
    raw_weight / SUM(raw_weight) OVER (PARTITION BY conversion_id) AS norm_weight
  FROM weighted
)
SELECT
  campaign_id,
  SUM(revenue_usd * norm_weight) AS attributed_revenue
FROM normalized
GROUP BY campaign_id
ORDER BY attributed_revenue DESC;

Follow-up

  • "How does this compare to linear MTA?" → linear uses 1/n_touches for each; time-decay favors recency. Run both in parallel in production; compare.

SQL 4 — Incrementality via holdout group

Prompt

Compute campaign incrementality: for each campaign, the conversion rate in the exposed group vs the holdout group. The assignment table randomly held out 5% of the audience from ads during the campaign. Report per-campaign lift and confidence interval width.

Schema

assignments(user_id BIGINT, campaign_id BIGINT, group_name STRING)  -- 'exposed' or 'holdout'
conversions(user_id BIGINT, campaign_id BIGINT, conversion_ts TIMESTAMP)

The trap

Missing the holdout group entirely and computing a gross "conversion rate" — which shows whether ads were shown to purchasers, not whether ads caused purchases. Also: forgetting the standard-error formula for a rate difference.

Reference solution

WITH rates AS (
  SELECT
    a.campaign_id,
    a.group_name,
    COUNT(DISTINCT a.user_id)                                  AS n_users,
    COUNT(DISTINCT c.user_id)                                  AS n_convs,
    COUNT(DISTINCT c.user_id) * 1.0 / COUNT(DISTINCT a.user_id) AS conv_rate
  FROM assignments a
  LEFT JOIN conversions c
    ON c.user_id = a.user_id AND c.campaign_id = a.campaign_id
  GROUP BY a.campaign_id, a.group_name
),
pivoted AS (
  SELECT
    campaign_id,
    MAX(CASE WHEN group_name = 'exposed' THEN conv_rate END) AS exposed_rate,
    MAX(CASE WHEN group_name = 'exposed' THEN n_users   END) AS exposed_n,
    MAX(CASE WHEN group_name = 'holdout' THEN conv_rate END) AS holdout_rate,
    MAX(CASE WHEN group_name = 'holdout' THEN n_users   END) AS holdout_n
  FROM rates
  GROUP BY campaign_id
)
SELECT
  campaign_id,
  exposed_rate,
  holdout_rate,
  (exposed_rate - holdout_rate) / NULLIF(holdout_rate, 0) AS lift_pct,
  exposed_rate - holdout_rate                              AS absolute_lift,
  -- 95% CI half-width for a difference in proportions
  1.96 * SQRT(
      exposed_rate * (1 - exposed_rate) / exposed_n
    + holdout_rate * (1 - holdout_rate) / holdout_n
  ) AS ci_half_width
FROM pivoted
ORDER BY lift_pct DESC;

Follow-up

  • "Lift is +2% but CI half-width is 3%." → The result is not statistically significant. You cannot claim lift. Senior candidates name this explicitly rather than reporting the +2% as a win.

SQL 5 — Triple-key dedupe with a suspected-dupe sidecar

Prompt

The ingest pipeline sometimes receives the same event twice because of SDK retries. event_id is not a reliable dedupe key (different SDK versions generate it differently). Build a silver table that keeps the first copy per logical event, and a sidecar table that preserves suspected duplicates for audit.

Schema

bronze.impression(event_id BIGINT, user_id BIGINT, event_name STRING, event_ts TIMESTAMP, ingest_ts TIMESTAMP, sdk_version STRING, dt DATE)

The trap

Using event_id alone misses retries with regenerated IDs. Using (user_id, event_ts) without rounding misses retries milliseconds apart that are functionally the same event.

Reference solution

CREATE OR REPLACE TABLE silver.fact_impression AS
WITH keyed AS (
  SELECT
    *,
    CONCAT_WS('|',
      CAST(user_id AS STRING),
      event_name,
      CAST(DATE_TRUNC('second', event_ts) AS STRING)
    ) AS dedupe_key
  FROM bronze.impression
  WHERE dt = :yesterday
)
SELECT * FROM keyed
QUALIFY ROW_NUMBER() OVER (PARTITION BY dedupe_key ORDER BY ingest_ts ASC) = 1;

CREATE OR REPLACE TABLE ops.suspected_duplicates AS
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY
      CONCAT_WS('|', CAST(user_id AS STRING), event_name,
                CAST(DATE_TRUNC('second', event_ts) AS STRING))
      ORDER BY ingest_ts ASC) AS rn
  FROM bronze.impression WHERE dt = :yesterday
) WHERE rn > 1;

Follow-up

  • "How do you alert when one SDK version suddenly drops 10× more than others?" → a per-sdk_version drop-rate monitor from the sidecar table.

SQL 6 — Cohort ROAS matrix (acquisition week × revenue week)

Prompt

For users acquired each week in the past 12 weeks, report their cumulative revenue at week 0, week 1, week 2, week 4, week 8 after acquisition. Each row is a cohort week; columns are week-offset revenue per user.

Schema

acquisitions(user_id BIGINT, acquired_ts TIMESTAMP, cost_usd DECIMAL)
revenue(user_id BIGINT, revenue_ts TIMESTAMP, revenue_usd DECIMAL)

The trap

Computing "week 1 revenue" as a separate WHERE-filtered query and joining four times — expensive and error-prone. The clean pattern uses conditional SUMs over a single join.

Reference solution

WITH cohort AS (
  SELECT
    user_id,
    DATE_TRUNC('week', acquired_ts) AS cohort_week,
    cost_usd
  FROM acquisitions
  WHERE acquired_ts >= CURRENT_DATE - INTERVAL '12 weeks'
),
joined AS (
  SELECT
    c.cohort_week,
    c.user_id,
    c.cost_usd,
    r.revenue_usd,
    DATEDIFF('week', c.cohort_week, DATE_TRUNC('week', r.revenue_ts)) AS week_offset
  FROM cohort c
  LEFT JOIN revenue r USING (user_id)
)
SELECT
  cohort_week,
  COUNT(DISTINCT user_id)                                                       AS n_users,
  SUM(DISTINCT cost_usd)                                                        AS cohort_spend,
  SUM(CASE WHEN week_offset <= 0 THEN revenue_usd ELSE 0 END) / COUNT(DISTINCT user_id) AS rev_per_user_w0,
  SUM(CASE WHEN week_offset <= 1 THEN revenue_usd ELSE 0 END) / COUNT(DISTINCT user_id) AS rev_per_user_w1,
  SUM(CASE WHEN week_offset <= 2 THEN revenue_usd ELSE 0 END) / COUNT(DISTINCT user_id) AS rev_per_user_w2,
  SUM(CASE WHEN week_offset <= 4 THEN revenue_usd ELSE 0 END) / COUNT(DISTINCT user_id) AS rev_per_user_w4,
  SUM(CASE WHEN week_offset <= 8 THEN revenue_usd ELSE 0 END) / COUNT(DISTINCT user_id) AS rev_per_user_w8,
  SUM(CASE WHEN week_offset <= 8 THEN revenue_usd ELSE 0 END) / NULLIF(SUM(DISTINCT cost_usd), 0) AS roas_w8
FROM joined
GROUP BY cohort_week
ORDER BY cohort_week DESC;

SQL 7 — Pacing: detect over-delivery before the end of day

Prompt

A campaign's daily budget is spread evenly across 24 hours. For each campaign, identify any point in today's calendar so far where cumulative spend is more than 110% of the linear-pacing target. Output the first breach timestamp per campaign and how far over it ran.

Schema

impressions(campaign_id BIGINT, imp_ts TIMESTAMP, spend_cents BIGINT)
campaigns(campaign_id BIGINT, daily_budget_usd DECIMAL)

Reference solution

WITH minute_spend AS (
  SELECT
    campaign_id,
    DATE_TRUNC('minute', imp_ts)              AS t,
    SUM(spend_cents) / 100.0                  AS spend_usd
  FROM impressions
  WHERE DATE(imp_ts) = CURRENT_DATE
  GROUP BY campaign_id, DATE_TRUNC('minute', imp_ts)
),
running AS (
  SELECT
    campaign_id,
    t,
    SUM(spend_usd) OVER (PARTITION BY campaign_id ORDER BY t) AS cum_spend,
    (EXTRACT(EPOCH FROM (t - DATE_TRUNC('day', t))) / 86400.0) AS day_fraction
  FROM minute_spend
),
breaches AS (
  SELECT
    r.campaign_id,
    r.t,
    r.cum_spend,
    r.day_fraction * c.daily_budget_usd       AS linear_target,
    r.cum_spend / NULLIF(r.day_fraction * c.daily_budget_usd, 0) AS pace_ratio
  FROM running r
  JOIN campaigns c USING (campaign_id)
  WHERE r.cum_spend > 1.10 * r.day_fraction * c.daily_budget_usd
)
SELECT
  campaign_id,
  MIN(t)                                 AS first_breach_ts,
  MAX(pace_ratio)                        AS peak_pace_ratio,
  MAX(cum_spend - linear_target)         AS peak_overshoot_usd
FROM breaches
GROUP BY campaign_id
ORDER BY peak_pace_ratio DESC;

SQL 8 — Creative fatigue: CTR decay by days since launch

Prompt

For each creative, compute its CTR on each day since launch. Flag any creative whose CTR on day N is below 60% of its own CTR on day 1, where day 1 is the creative's own first day of delivery.

Schema

impressions(creative_id BIGINT, imp_ts TIMESTAMP, did_click BOOLEAN)

The trap

Using a calendar day-of-month instead of days since the creative itself launched. Creatives launched on different dates can't be compared on absolute dates.

Reference solution

WITH daily AS (
  SELECT
    creative_id,
    DATE(imp_ts)                              AS dt,
    COUNT(*)                                  AS imps,
    SUM(CASE WHEN did_click THEN 1 ELSE 0 END) AS clicks,
    SUM(CASE WHEN did_click THEN 1 ELSE 0 END) * 1.0 / COUNT(*) AS ctr
  FROM impressions
  GROUP BY creative_id, DATE(imp_ts)
),
indexed AS (
  SELECT
    d.*,
    MIN(dt) OVER (PARTITION BY creative_id) AS launch_dt,
    DATEDIFF('day', MIN(dt) OVER (PARTITION BY creative_id), dt) + 1 AS day_n
  FROM daily d
),
day1_ctr AS (
  SELECT creative_id, ctr AS ctr_day1
  FROM indexed WHERE day_n = 1
)
SELECT
  i.creative_id,
  i.dt,
  i.day_n,
  i.ctr,
  d1.ctr_day1,
  i.ctr / NULLIF(d1.ctr_day1, 0) AS relative_ctr,
  CASE WHEN i.ctr < 0.60 * d1.ctr_day1 THEN TRUE ELSE FALSE END AS fatigued
FROM indexed i
JOIN day1_ctr d1 USING (creative_id)
WHERE i.day_n > 7                   -- only flag after the first week
  AND i.imps >= 1000                -- only flag once there's volume
ORDER BY i.creative_id, i.dt;

Python 1 — Sessionize an event stream with a timeout

Prompt

Given a list of events [(user_id, ts, event_name)] sorted by user_id then ts, partition each user's events into sessions where any gap > 30 minutes starts a new session. Return a list of (user_id, session_id, start_ts, end_ts, event_count). Solve in O(N).

from dataclasses import dataclass
from datetime import datetime, timedelta
from typing import List, Tuple

@dataclass
class Session:
    user_id: int
    session_id: str
    start_ts: datetime
    end_ts: datetime
    event_count: int

GAP = timedelta(minutes=30)

def sessionize(events: List[Tuple[int, datetime, str]]) -> List[Session]:
    out: List[Session] = []
    if not events:
        return out
    cur_user = None
    cur_start = cur_end = None
    cur_count = 0

    def flush():
        if cur_user is not None and cur_count > 0:
            sid = f"{cur_user}_{int(cur_start.timestamp())}"
            out.append(Session(cur_user, sid, cur_start, cur_end, cur_count))

    for user_id, ts, _ in events:
        if user_id != cur_user or (cur_end is not None and ts - cur_end > GAP):
            flush()
            cur_user, cur_start, cur_end, cur_count = user_id, ts, ts, 1
        else:
            cur_end = ts
            cur_count += 1
    flush()
    return out

The trap

Forgetting that user boundaries also start a new session — a candidate who only checks the gap will merge two users' events.

Python 2 — Token-bucket rate limiter for pixel firing

Prompt

Implement a rate limiter that allows at most N pixel events per user per second, with a burst capacity of B. It should be thread-safe and O(1) per call.

import time
import threading
from collections import defaultdict

class TokenBucket:
    def __init__(self, rate_per_sec: float, burst: int):
        self.rate = rate_per_sec
        self.burst = burst
        self._state = defaultdict(lambda: (burst, time.monotonic()))
        self._lock = threading.Lock()

    def allow(self, user_id: int) -> bool:
        with self._lock:
            tokens, last = self._state[user_id]
            now = time.monotonic()
            # Refill based on elapsed time
            tokens = min(self.burst, tokens + (now - last) * self.rate)
            if tokens >= 1.0:
                self._state[user_id] = (tokens - 1.0, now)
                return True
            self._state[user_id] = (tokens, now)
            return False

The trap

Using wall-clock (time.time()) instead of monotonic — NTP adjustments can move the clock backwards and make the limiter too generous or too strict. Forgetting the lock — race conditions between read and write.

Python 3 — Bloom filter for event dedupe at ingest

Prompt

Implement a Bloom filter to dedupe event_ids at ingest time. Capacity 10M items, false-positive rate ≤ 1%. Provide add(id) and maybe_contains(id) methods.

import math
import hashlib
import array

class BloomFilter:
    def __init__(self, capacity: int, fp_rate: float = 0.01):
        # m = -n ln p / (ln 2)^2  ; k = (m/n) ln 2
        self.m = int(-capacity * math.log(fp_rate) / (math.log(2) ** 2))
        self.k = max(1, int((self.m / capacity) * math.log(2)))
        # Bit array: one bit per slot, packed 8 per byte
        self.bits = array.array('B', [0] * ((self.m + 7) // 8))

    def _positions(self, key: bytes):
        h1 = int.from_bytes(hashlib.blake2b(key, digest_size=8).digest(), 'big')
        h2 = int.from_bytes(hashlib.sha256(key).digest()[:8], 'big')
        for i in range(self.k):
            yield (h1 + i * h2) % self.m

    def add(self, key) -> None:
        b = str(key).encode()
        for p in self._positions(b):
            self.bits[p >> 3] |= (1 << (p & 7))

    def maybe_contains(self, key) -> bool:
        b = str(key).encode()
        return all(self.bits[p >> 3] & (1 << (p & 7)) for p in self._positions(b))

The trap

Using Python set() — works in memory but blows through RAM at 10M items (~700 MB). The bloom filter is roughly 12 MB for the same specs. Interviewers ask because they want to see you pick the right structure for scale.

Python 4 — Attribution resolver (streaming)

Prompt

Maintain a per-user rolling buffer of the last 7 days of touches in memory (assume you'll scale this out later). When a conversion event arrives, return the most recent qualifying touch using the same last-click-with-view-fallback logic from SQL 1.

from collections import defaultdict, deque
from dataclasses import dataclass
from datetime import datetime, timedelta
from typing import Optional

@dataclass
class Touch:
    touch_id: int
    campaign_id: int
    ts: datetime
    touch_type: str  # 'click' or 'view'

CLICK_WINDOW = timedelta(days=7)
VIEW_WINDOW  = timedelta(days=1)

class AttributionResolver:
    def __init__(self):
        self.touches: dict[int, deque[Touch]] = defaultdict(deque)

    def record_touch(self, user_id: int, t: Touch) -> None:
        dq = self.touches[user_id]
        dq.append(t)
        # Evict anything older than the longest window
        cutoff = t.ts - CLICK_WINDOW
        while dq and dq[0].ts < cutoff:
            dq.popleft()

    def attribute(self, user_id: int, conversion_ts: datetime) -> Optional[Touch]:
        dq = self.touches.get(user_id) or ()
        best: Optional[Touch] = None
        for t in dq:
            if t.ts > conversion_ts:
                continue
            in_window = (
                (t.touch_type == 'click' and conversion_ts - t.ts <= CLICK_WINDOW)
                or (t.touch_type == 'view' and conversion_ts - t.ts <= VIEW_WINDOW)
            )
            if not in_window:
                continue
            # Prefer click over view, then most-recent
            if best is None:
                best = t
            else:
                if best.touch_type == 'view' and t.touch_type == 'click':
                    best = t
                elif best.touch_type == t.touch_type and t.ts > best.ts:
                    best = t
        return best

Python 5 — SKAN conversion-value encoder

Prompt

Apple's SKAdNetwork gives the advertiser 6 bits of "conversion value" (0–63) to encode an event signal back to the ad network. Build an encoder that maps a sequence of user events (registered, tutorial_done, purchase with revenue tier) to a single integer 0–63.

from dataclasses import dataclass

@dataclass
class UserState:
    registered: bool = False
    tutorial_done: bool = False
    revenue_usd: float = 0.0

# 6 bits: 1 bit registered | 1 bit tutorial | 4 bits revenue tier
# 16 revenue tiers: 0, $1, $2, $5, $10, $20, $50, $100, ...
REVENUE_TIERS = [0, 1, 2, 5, 10, 20, 50, 100, 200, 500, 1000, 2000, 5000, 10000, 25000, 100000]

def encode_conversion_value(state: UserState) -> int:
    reg_bit = 1 if state.registered else 0
    tut_bit = 1 if state.tutorial_done else 0
    tier = 0
    for i, threshold in enumerate(REVENUE_TIERS):
        if state.revenue_usd >= threshold:
            tier = i
    # Pack: [reg:1][tut:1][tier:4]
    return (reg_bit << 5) | (tut_bit << 4) | tier

def decode_conversion_value(v: int) -> dict:
    return {
        'registered': bool((v >> 5) & 1),
        'tutorial_done': bool((v >> 4) & 1),
        'revenue_tier_index': v & 0b1111,
        'revenue_floor_usd': REVENUE_TIERS[v & 0b1111],
    }

The trap

Trying to encode exact revenue in 6 bits — it does not fit. The tiered mapping is the whole point. Senior candidates name the tier-design tradeoff: finer tiers at low revenue (more common) vs coarser tiers at high revenue (rare).

Python 6 — Incrementality lift with confidence interval

Prompt

Given exposed and holdout conversion counts and user counts, compute the lift percentage and a 95% CI for the difference in proportions. Return (lift_pct, ci_half_width_pp, is_significant).

import math

def lift_with_ci(
    exposed_convs: int, exposed_users: int,
    holdout_convs: int, holdout_users: int,
    z: float = 1.96,  # 95% normal approx
) -> tuple[float, float, bool]:
    p1 = exposed_convs / exposed_users
    p2 = holdout_convs / holdout_users
    abs_diff = p1 - p2
    # Standard error of difference in two proportions
    se = math.sqrt(p1 * (1 - p1) / exposed_users + p2 * (1 - p2) / holdout_users)
    ci_half = z * se
    lift_pct = (p1 - p2) / p2 * 100 if p2 > 0 else float('inf')
    # Significant if the CI doesn't cross zero
    significant = (abs_diff - ci_half) > 0
    return lift_pct, ci_half * 100, significant

The trap

Reporting the lift without the CI — a +2% lift with ±3% CI is not a lift. Senior candidates return the significance flag alongside the number so the caller cannot misread it.

3. How to Approach the Ads Coding Round

Four habits that separate strong candidates in ads coding rounds specifically.

  1. Ask for the attribution window before you write a line. Last-click? 7-day? 28-day? Click-only or click-plus-view? Half the SQL problems above change shape depending on this answer. If the interviewer didn't volunteer it, ask.
  2. Name the dedupe strategy out loud. Even if the prompt doesn't mention duplicates, your code should handle them. Say: "I'll assume event_id may repeat across SDK retries; I'll dedupe on (user_id, event_name, second-rounded ts)." If the interviewer says "no, event_id is unique" — great, you asked, and you write a simpler query. If they say "good point" — you just showed senior judgment.
  3. Write schema-aware code. Ads tables are billions of rows. Any code that does SELECT * into Python, or loops a Python function over rows, will be scored junior. Push work into the SQL engine or into vectorized operations. When in doubt, ask: "Can this run as a single SQL query? If not, what's the smallest Python layer on top?"
  4. Return the uncertainty, not just the point estimate. Every ads number has statistical noise. The senior answer to "what's the lift" is never a single number — it's a number plus a confidence range, plus whether the range crosses zero. Build that shape into your functions as default output, not an optional afterthought.
