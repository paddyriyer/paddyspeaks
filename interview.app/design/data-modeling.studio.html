Trip lifecycle — multi-driver convoys

Uber / Lyft · ride-sharing · Principal
Full write-up ↗
PromptDesign a complete data model for the Trip Lifecycle — including the complex convoy / multi-driver scenario where one group request is split across two or more vehicles. Must support Operations (live ETA), Finance (driver payouts), and Risk (fraud).

Why it's hard

One group request can fan out across multiple drivers, riders enter and exit at different points, and finance, ops and risk each need a different grain from the same events. Collapse them into one table and every aggregate either double-counts or hides the audit trail.

Parent fact · fct_journeys (the group intent)

ColumnTypeDescription
journey_keyPK BIGINTSurrogate.
journey_idUUIDNatural ID; idempotency anchor.
split_typeENUM{EQUAL_SPLIT, DISTANCE_PRORATA, REQUESTER_PAYS, CUSTOM}.
total_group_fare_usdDECIMAL(10,2)Single bill for the group; reconciles to brg_journey_riders.
num_vehicles_dispatchedSMALLINT1 normal, ≥2 convoy.

Vehicle fact · fct_trips (one row per driver/vehicle per journey)

ColumnTypeDescription
trip_key / trip_idPK / UUIDSurrogate + natural.
journey_idFKParent journey — many trips share one journey (convoy).
surge_multiplierDECIMAL(4,2)Locked at request — immutable.
fx_rate_to_usdDECIMAL(18,8)Locked at request_timestamp — revenue can never be restated by FX drift.
terminal_stateENUM{completed, cancelled_rider, cancelled_driver, cancelled_system, expired, no_show}.

Plus · event log, feedback, financial bridge

fct_trip_events (append-only state transitions — the audit trail), fct_trip_feedback (per-rider sentiment), and brg_journey_riders (M:N — which rider rode in which car, and what they actually paid).

Consistency contracts, per field

The same row mixes consistency regimes on purpose: surge_multiplier and fx_rate_to_usd are strong — locked at request, immutable forever; ETAs and rollups are eventual. Write the contract down before the first column — retrofitting strong consistency onto a field built eventual is a rebuild, not a tweak.

One write path, three derived paths

OLTP mutates the live trips row in place; OLAP keeps fct_trips as an append-only snapshot and fct_trip_events as the immutable log. The crossing is CDC governed by a watermark — the same entity is normalized in OLTP and dimensional in OLAP by design.

Boundary violations that disqualifyA column on fct_trips a second service writes directly → two writers, no source of truth. Raw GPS inside fct_trip_events → a 1 Hz firehose welded to an append-only audit log, now unqueryable. A fare written before the terminal state → finance reads an estimate as if it were committed. Each one collapses a boundary the model depends on.
Senior framing"The clean star schema is the steady-state view; production runs a reconciliation engine beneath it. Idempotency keys absorb replays, optimistic locks absorb races, append-only ledgers absorb rollbacks, and a stuck-state sweeper absorbs the events that never arrive. The model isn't only correct — it's correct and self-healing."

The reconciliation invariants

A convoy of 3 riders billed $60, split across 2 vehicles, must reconcile three ways:

-- Invariant 1: the bridge sums to the group bill
SUM(brg_journey_riders.individual_charge_usd) per journey_id
   = fct_journeys.total_group_fare_usd        -- $20 + $20 + $20 = $60 ✓

-- Invariant 2: the vehicle shares sum to the group bill
SUM(fct_trips.trip_fare_share_usd) per journey_id
   = fct_journeys.total_group_fare_usd        -- $30 + $30 = $60 ✓

Riders R1+R2 ride in T-101 while R3 rides in T-102 — the bridge carries trip_id so a rider is always tied to the exact car, and "Pool for 4 / Bus / wedding party of 12" needs no schema migration, just more bridge rows.

Q · What is the grain of fct_trips in the convoy model?

One row per driver/vehicle per journey — not per rider. Rider-level money lives in brg_journey_riders; collapsing them double-counts fares.

Q · Why is surge_multiplier stored on the row instead of derived at query time?

Because the rider's quoted price, the driver's earnings and booked revenue must agree forever. Lock it at request; never recompute.

Order & courier dispatch — stacked deliveries

DoorDash / Uber Eats · 3-sided marketplace · Advanced
Full write-up ↗
PromptModel DoorDash's order lifecycle from customer placement → restaurant prep → courier pickup → delivery, supporting Customers (on-time SLA), Couriers (earnings), Restaurants (operations), with native handling of stacked orders where one courier delivers multiple orders on a single dispatch.

The design tension

A 3-sided marketplace (customer + restaurant + courier) where one fact table can't satisfy all three. Stacked orders make it worse: one courier dispatch covers multiple orders, but each order has its own customer SLA and restaurant relationship. Mutable tips (editable up to 24h) demand append-only economics. The model must read clean for each stakeholder without forcing the others to filter.

The order / dispatch split + bridge

  • fct_orders — customer grain. One row per customer order.
  • fct_dispatches — courier grain. One row per dispatch; covers 1 (single) or N (stacked) orders.
  • brg_dispatch_orders — stacking bridge. Links a dispatch to its 1-N orders with leg sequence + per-order distance.
  • fct_order_state_events — telemetry grain. One row per state transition.

Sample data — dispatch DSP-99 (1 courier, 2 stacked orders)

INSERT INTO fct_dispatches VALUES
  ('DSP-99','COURIER_X','2025-05-01 18:18','2025-05-01 18:42', TRUE, 2, 4.2, 6.50, 11.00, 17.50);

INSERT INTO brg_dispatch_orders VALUES
  ('DSP-99','O-201', 1, 2.0, 8.50),
  ('DSP-99','O-202', 2, 2.2, 9.00);

Q · stage-by-stage latency with whose-fault attribution

SELECT order_id,
  EXTRACT(EPOCH FROM (ready_ts - confirmed_ts))/60      AS prep_min,
  EXTRACT(EPOCH FROM (picked_up_ts - ready_ts))/60      AS pickup_wait_min,
  EXTRACT(EPOCH FROM (delivered_ts - picked_up_ts))/60  AS drive_min,
  CASE
    WHEN (ready_ts - confirmed_ts)     > INTERVAL '20 min' THEN 'restaurant_slow'
    WHEN (picked_up_ts - ready_ts)     > INTERVAL '10 min' THEN 'courier_late_pickup'
    WHEN (delivered_ts - picked_up_ts) > INTERVAL '20 min' THEN 'courier_slow_drive'
    ELSE 'on_track' END                                   AS bottleneck
FROM fct_orders WHERE DATE(placed_ts) = '2025-05-01';

Q · earnings/hr, stacked vs single

SELECT is_batched, COUNT(*) AS dispatches,
  ROUND(SUM(payout_usd) /
        SUM(EXTRACT(EPOCH FROM (dispatch_end_ts - dispatch_start_ts))/3600), 2) AS earnings_per_hr
FROM fct_dispatches
GROUP BY is_batched;
-- If stacked is < 1.3× single payout/hr, batching isn't worth it.

Why this works

  • Three stakeholders, three grains, one model. Customer queries hit fct_orders; courier queries hit fct_dispatches; the bridge ties them only when stacking math matters.
  • Stack uplift is measurable. Compare is_batched=TRUE vs FALSE earnings/hr — if stacking doesn't beat single by ~30%, dispatch should stop offering it.
  • Mutable tips handled cleanly. Append correction rows; tip_locked_ts tracks the immutability boundary; finance restates from it.
Senior framing"The bridge is what makes stacking a first-class concept rather than a hack: was the customer on time? joins fct_orders; did the courier earn enough/hour? aggregates fct_dispatches; did the restaurant lose orders to prep time? joins dim_restaurants."
Collapse the model and three things breakOrder grain only: can't represent "courier did 2 stacked deliveries" — courier earnings double-count. Dispatch grain only: the customer-SLA query has to sum inside the bridge — ugly, error-prone. No bridge (1:1 dispatch:order): forces non-stacking — kills the entire economic case for batching.

Worked example — stacked dispatch DSP-99

  1. Courier X accepts at 18:18 with 2 stacked orders from RST_PIZZA (shared pickup, 2 drops).
  2. O-201 delivered 18:34 (16 min, 2.0 km); O-202 delivered 18:42 (24 min, +2.2 km).
  3. Earnings: base $6.50 + tips $11.00 = $17.50 for 24 min → $43.75/hr vs ~$28/hr single.
  4. Bridge: O-201 share $8.50 + O-202 share $9.00 = $17.50 ✓
  5. SLA trade-off: O-201 on-time; O-202 ran 8 min late waiting for O-201's drop — courier earnings up, second customer's SLA at risk.

Double-entry ledger — replay-safe payments

Stripe · payments · Advanced
Full write-up ↗
PromptStripe processes ~$1T/year. Every cent must reconcile across charges, refunds, fees, payouts, FX, taxes and chargebacks — and survive replay, retries and partial failures.

The design tension

The whole system is an append-only double-entry ledger: nothing is updated, only debited or credited. The same model banks have used for centuries; the schema is just its SQL realization.

TypeTableNotes
Dimdim_accountEvery party (customer, merchant, platform, fee, FX, tax, chargeback) is an "account" with a balance.
Fact (event)fct_journal_entryEvery transaction emits N rows summing to zero — the append-only spine.
Fact (snapshot)fct_account_balanceDerived per-account daily balance, materialized from journal entries.
Bridgebridge_charge_journalOne charge → many entries (revenue + fee + tax).

The double-entry contract — signed amounts sum to zero per txn

-- A $100 charge with a $3 Stripe fee:
INSERT INTO fct_journal_entry (txn_id, account, currency, amount) VALUES
  ('TXN_001', 'customer_xyz',  'USD', -100.00),  -- customer debited $100
  ('TXN_001', 'merchant_abc',  'USD',   97.00),  -- merchant credited $97
  ('TXN_001', 'stripe_revenue','USD',    3.00);  -- Stripe credited $3
-- SUM = 0 ✓

Daily integrity check

SELECT txn_id, SUM(amount) AS net
FROM fct_journal_entry
GROUP BY txn_id
HAVING ABS(SUM(amount)) > 0.001;
-- ANY row returned = data corruption, page on-call

Refunds are NEW entries, not deletes

-- Refund of TXN_001
INSERT INTO fct_journal_entry (txn_id, account, currency, amount, references_txn) VALUES
  ('TXN_002', 'merchant_abc',  'USD',  -97.00, 'TXN_001'),
  ('TXN_002', 'customer_xyz',  'USD',  100.00, 'TXN_001'),
  ('TXN_002', 'stripe_revenue','USD',   -3.00, 'TXN_001');

Idempotency via natural key

Same idempotency key + same body = same result. The schema enforces UNIQUE (idempotency_key, account_id, txn_type): a retry tries to insert a duplicate → unique violation → the app returns the original txn. Replay-safe by construction.

Multi-currency lock

Charge in EUR, book revenue in USD, settle in GBP — three currencies, one transaction, three journal entries with an fx_rate_locked_at_event on each. "What did we earn in March in USD?" sums the USD-denominated entries — never re-does FX.

The junior moveMarking a charge "refunded" with an UPDATE. Wrong — it destroys the Monday close that accounting already locked. The ledger appends; both transactions exist forever, and fct_account_balance is just SUM(amount) GROUP BY account — always derivable, never wrong.
Chargebacks arrive 60-90 days lateThe original transaction long since posted; the chargeback emits a NEW entry with references_txn = 'TXN_001'. Monthly close-of-books locks the past, but new chargeback rows still arrive — they land in a "post-close adjustment" sub-period.
Senior framing"The append-only double-entry ledger is the only model where SUM-equals-zero is an enforceable integrity constraint at the database level. The unique constraint on idempotency_key turns API retries into no-ops."
